---
title: "Entity Mutations"
description: "Direct CRUD operations for creating, updating and deleting entities"
---

Direct entity mutation endpoints provide CRUD operations for creating, updating, and deleting entities in the system. These operations bypass the change request workflow and apply changes immediately.

<Warning>
Entity mutations require elevated permissions and should be used with caution as they directly modify the database without approval workflows.
</Warning>

## Authentication

All entity mutation endpoints require Bearer token authentication:

```
Authorization: Bearer <token>
```

## Common Response Formats

### Success Responses

Successful mutations return the created or updated entity:

```json
{
  "id": 123,
  "nameEn": "Updated Name",
  "nameAr": "الاسم المحدث",
  // ... other entity fields
}
```

### Error Responses

```json
{
  "statusCode": 400,
  "message": "Validation failed",
  "errors": [
    {
      "field": "nameEn",
      "message": "Name is required"
    }
  ]
}
```

## Developer Mutations

### Create Developer

<CodeGroup>
```bash POST
POST /api/developers
Content-Type: application/json
Authorization: Bearer <token>

{
  "nameEn": "New Developer",
  "nameAr": "مطور جديد",
  "developerNumber": "DEV-123",
  "logo": "https://example.com/logo.png",
  "webpage": "https://developer.com"
}
```

```json Response
{
  "id": 456,
  "nameEn": "New Developer",
  "nameAr": "مطور جديد",
  "developerNumber": "DEV-123",
  "sourceDeveloperId": "auto-generated",
  "logo": "https://example.com/logo.png",
  "webpage": "https://developer.com",
  "createdAt": "2024-01-01T00:00:00Z",
  "updatedAt": "2024-01-01T00:00:00Z"
}
```
</CodeGroup>

#### Request Body

| Field                  | Type   | Required | Notes                    |
| ---------------------- | ------ | -------- | ------------------------ |
| `nameEn`               | string | no       | max 500 characters       |
| `nameAr`               | string | no       | max 500 characters       |
| `developerNumber`      | string | no       | max 50 characters        |
| `registrationDate`     | string | no       | ISO date format          |
| `licenseSourceId`      | number | no       | FK to license_sources    |
| `licenseTypeId`        | number | no       | FK to license_types      |
| `licenseNumber`        | string | no       | max 100 characters       |
| `licenseIssueDate`     | string | no       | ISO date format          |
| `licenseExpiryDate`    | string | no       | ISO date format          |
| `chamberOfCommerceNo`  | string | no       | max 100 characters       |
| `legalStatusId`        | number | no       | FK to legal_statuses     |
| `webpage`              | string | no       | max 500 characters, URL  |
| `licenseUrl`           | string | no       | max 500 characters, URL  |
| `phone`                | string | no       | max 100 characters       |
| `fax`                  | string | no       | max 100 characters       |
| `logo`                 | string | no       | max 500 characters, URL  |
| `logoDark`             | string | no       | max 500 characters, URL  |

### Update Developer

<CodeGroup>
```bash PATCH
PATCH /api/developers/:id
Content-Type: application/json
Authorization: Bearer <token>

{
  "nameEn": "Updated Developer Name",
  "webpage": "https://newwebsite.com"
}
```

```json Response
{
  "id": 456,
  "nameEn": "Updated Developer Name",
  "nameAr": "مطور جديد",
  "webpage": "https://newwebsite.com",
  // ... other fields
  "updatedAt": "2024-01-02T00:00:00Z"
}
```
</CodeGroup>

<Note>
Only provided fields are updated. Omitted fields remain unchanged.
</Note>

### Delete Developer

<CodeGroup>
```bash DELETE
DELETE /api/developers/:id
Authorization: Bearer <token>
```

```json Response
{
  "message": "Developer deleted successfully",
  "id": 456
}
```
</CodeGroup>

<Warning>
Deleting a developer will soft-delete the record (sets `deleted_at` timestamp). Related projects and master projects will need to be handled separately.
</Warning>

## Validation Rules

<AccordionGroup>
<Accordion title="General Validation">
- All string fields are trimmed before validation
- URLs must be valid HTTP/HTTPS format
- Dates must be in ISO 8601 format (YYYY-MM-DD)
- Foreign key references must exist in their respective tables
</Accordion>

<Accordion title="Name Validation">
- English names (`nameEn`) should use Latin characters
- Arabic names (`nameAr`) should use Arabic script
- At least one name (English or Arabic) is typically required
- Names are indexed for search performance
</Accordion>

<Accordion title="Unique Constraints">
- `developerNumber` must be unique across developers
- `sourceDeveloperId` must be unique across developers
- `municipalityNumber` must be unique across areas within the same city
</Accordion>
</AccordionGroup>

## Error Handling

<Tabs>
<Tab title="400 Bad Request">
Validation errors or malformed request body:

```json
{
  "statusCode": 400,
  "message": "Validation failed",
  "errors": [
    {
      "field": "nameEn",
      "message": "Name must be at least 1 character"
    }
  ]
}
```
</Tab>

<Tab title="404 Not Found">
Entity not found for update/delete operations:

```json
{
  "statusCode": 404,
  "message": "Developer not found"
}
```
</Tab>

<Tab title="409 Conflict">
Unique constraint violations:

```json
{
  "statusCode": 409,
  "message": "Developer number already exists"
}
```
</Tab>
</Tabs>

## Best Practices

<Steps>
<Step title="Validate before mutation">
Always validate your data client-side before sending mutation requests to minimize server errors.
</Step>

<Step title="Handle soft deletes">
Remember that deletions are soft deletes - records are marked as deleted but not physically removed from the database.
</Step>

<Step title="Check foreign key relationships">
Ensure referenced entities (states, cities, areas) exist before creating dependent entities.
</Step>

<Step title="Use transactions for related operations">
When creating multiple related entities, consider using database transactions to maintain data consistency.
</Step>
</Steps>

<Info>
Entity mutations are immediately applied to the database. For workflows requiring approval processes, use the [Change Requests](/backend/api/change-requests) endpoints instead.
</Info>
