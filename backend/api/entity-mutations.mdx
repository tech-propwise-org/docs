---
title: "Direct Entity Mutations"
description: "Direct create, update, and delete operations on entities"
---

Direct entity mutations allow you to perform immediate create, update, and delete operations on entities without going through the change request system. These operations require appropriate permissions and immediately modify the database.

<Warning>
Direct mutations bypass the change request approval workflow. Use these endpoints only when you have the necessary permissions and want immediate data changes.
</Warning>

## Authentication

All direct mutation endpoints require Bearer token authentication:

```http
Authorization: Bearer <your-token>
```

## Developers

    `POST /api/developers`
  </Tab>
  <Tab title="Request Body">

    ```json
      "nameEn": "Emaar Properties",
      "nameAr": "إعمار العقارية",
      "registrationDate": "2023-01-15",
      "licenseSource": 1,
      "licenseType": 1,
      "licenseNumber": "LIC-123456",
      "licenseIssueDate": "2023-01-01",
      "licenseExpiryDate": "2025-12-31",
      "chamberOfCommerceNo": "CC-789",
      "legalStatus": 1,
    ```

#### Required Fields

<Info>
Only `sourceDeveloperId` is required. All other fields are optional.

| `sourceDeveloperId` | string | Required, max 50 chars, must be unique |

### Update Developer

<Tabs>
  <Tab title="Endpoint">
    `PUT /api/developers/:id`
  </Tab>
  <Tab title="Request Body">
    ```json
    {
      "nameEn": "Updated Developer Name",
      "nameAr": "اسم المطور المحدث",
      "phone": "+971-4-123-4567",
      "webpage": "https://updated-website.com"
    }
    ```
  </Tab>
</Tabs>

<Note>
Only provide the fields you want to update. The request body should contain a partial object with the fields to modify.
</Note>

#### Response `200 OK`

Returns the updated developer object with the same structure as the create response.

### Delete Developer

<Tabs>
  <Tab title="Endpoint">
    `DELETE /api/developers/:id`
  </Tab>
</Tabs>

<Warning>
This performs a soft delete by setting the `deletedAt` timestamp. The developer will no longer appear in list queries but relationships may still reference it.
</Warning>

#### Response `204 No Content`

## Cities

### Create City

<Tabs>
  <Tab title="Endpoint">
    `POST /api/cities`
  </Tab>
  <Tab title="Request Body">
    ```json
    {
      "nameEn": "Dubai",
      "nameAr": "دبي",
      "stateId": 1,
      "sourceCityId": "CITY-001"
    }
    ```
  </Tab>
</Tabs>

#### Required Fields

| Field | Type | Constraints |
|-------|------|-------------|
| `nameEn` | string | Required, max 200 chars |
| `stateId` | number | Required, must reference existing state |

#### Response `201 Created`

```json
{
  "id": 15,
  "nameEn": "Dubai",
  "nameAr": "دبي",
  "stateId": 1,
  "sourceCityId": "CITY-001",
  "createdAt": "2023-12-15T10:30:00Z",
  "updatedAt": "2023-12-15T10:30:00Z"
}
```

<Tabs>

  </Tab>
  <Tab title="Request Body">
    ```json
    {
      "nameEn": "Updated City Name",
    ```

### Delete City

<Tabs>
  <Tab title="Endpoint">
    `DELETE /api/cities/:id`
  </Tab>
</Tabs>

<Warning>
Cities with associated areas cannot be deleted. You must first delete or reassign all areas in this city.
</Warning>

## Error Responses

<AccordionGroup>
  <Accordion title="400 Bad Request">
    Returned when request validation fails:

    ```json
    {
      "statusCode": 400,
      "message": ["nameEn must be a string", "cityId must be a number"],
      "error": "Bad Request"
    }
    ```
  </Accordion>

  <Accordion title="404 Not Found">
    Returned when the entity to update/delete doesn't exist:

    ```json
    {
      "statusCode": 404,
      "message": "Developer not found",
      "error": "Not Found"
    }
    ```
  </Accordion>

  <Accordion title="409 Conflict">
    Returned when a unique constraint is violated:

    ```json
    {
      "statusCode": 409,
      "message": "Developer with this sourceDeveloperId already exists",
      "error": "Conflict"
    }
    ```
  </Accordion>
</AccordionGroup>

## Best Practices

<Steps>
  <Step title="Validate data before sending">
    Always validate required fields and data formats on the client side before making mutation requests.
  </Step>
  
  <Step title="Handle foreign key constraints">
    Ensure referenced entities (like `stateId`, `cityId`, `areaId`) exist before creating dependent entities.
  </Step>
  
  <Step title="Use appropriate HTTP methods">
    - `POST` for creating new entities
    - `PUT` for updating existing entities (partial updates supported)
    - `DELETE` for removing entities
  </Step>
  
  <Step title="Check deletion dependencies">
    Before deleting entities, ensure they don't have dependent relationships that would prevent deletion.
  </Step>
</Steps>

<Tip>
For bulk operations or when you need approval workflows, consider using the [Change Requests](/backend/api/change-requests) system instead of direct mutations.
</Tip>
