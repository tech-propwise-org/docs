description: "Direct entity creation, update, and deletion endpoints"
---

Direct entity mutations provide immediate database operations for creating, updating, and deleting entities. These endpoints require appropriate authentication and bypass the change request workflow.

<Warning>
These endpoints directly modify the database. Use with caution in production environments.
</Warning>

## Authentication

All entity mutation endpoints require a Bearer token:

```bash
Authorization: Bearer <your-token>
```

## Response Format

All mutation endpoints return standardized responses:

- **201 Created**: For successful creation operations
- **200 OK**: For successful update operations  
- **204 No Content**: For successful deletion operations
- **400 Bad Request**: For validation errors
- **404 Not Found**: When the entity doesn't exist
- **409 Conflict**: When constraints are violated

## Developers

### Create Developer

Create a new developer entity.

<CodeGroup>
```bash cURL
curl -X POST /api/developers \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" \
  -d '{
    "sourceDeveloperId": "DEV-001",
    "developerNumber": "DEV-001", 
    "nameEn": "Emaar Properties",
    "nameAr": "إعمار العقارية",
    "registrationDate": "2020-01-15",
    "licenseNumber": "LIC-001",
    "licenseIssueDate": "2020-01-01",
    "licenseExpiryDate": "2025-01-01",
    "chamberOfCommerceNo": "COM-001",
    "webpage": "https://emaar.com",
    "phone": "+971-4-367-3333",
    "logo": "https://example.com/logo.png",
    "logoDark": "https://example.com/logo-dark.png"
  }'
```

```javascript JavaScript
const response = await fetch('/api/developers', {
  method: 'POST',
  headers: {
    'Authorization': `Bearer ${token}`,
    'Content-Type': 'application/json'
  },
  body: JSON.stringify({
    sourceDeveloperId: 'DEV-001',
    nameEn: 'Emaar Properties',
    nameAr: 'إعمار العقارية',
    // ... other fields
  })
});

const developer = await response.json();
```
</CodeGroup>

#### Required Fields

| Field | Type | Description |
|-------|------|-------------|
| `sourceDeveloperId` | string | Unique external identifier (max 50 chars) |

#### Optional Fields

<AccordionGroup>
<Accordion title="Basic Information">
| Field | Type | Constraints |
|-------|------|-------------|
| `developerNumber` | string | max 50 chars |
| `nameEn` | string | max 500 chars, indexed |
| `nameAr` | string | max 500 chars, indexed |
| `registrationDate` | date | ISO format |
</Accordion>

<Accordion title="License Information">
| Field | Type | Constraints |
|-------|------|-------------|
| `licenseSource` | number | Foreign key to license_sources |
| `licenseType` | number | Foreign key to license_types |
| `licenseNumber` | string | max 100 chars |
| `licenseIssueDate` | date | ISO format |
| `licenseExpiryDate` | date | ISO format |
| `licenseUrl` | string | max 500 chars |
</Accordion>

<Accordion title="Contact & Media">
| Field | Type | Constraints |
|-------|------|-------------|
| `chamberOfCommerceNo` | string | max 100 chars |
| `legalStatus` | number | Foreign key to legal_statuses |
| `webpage` | string | max 500 chars |

### Update Developer

Update an existing developer by ID.

```bash
PUT /api/developers/:id
```

<Note>
Only provided fields will be updated. Omitted fields remain unchanged.
</Note>

### Delete Developer

Soft delete a developer (sets `deleted_at` timestamp).

```bash
DELETE /api/developers/:id
```

<Warning>
Deleting a developer with associated projects may cause referential integrity issues.
</Warning>

## Cities

### Create City

Create a new city entity.

```json
POST /api/cities
{
  "nameEn": "Dubai",
  "nameAr": "دبي",
  "stateId": 1
```

| Field | Type | Description |

#### Optional Fields

| Field | Type | Constraints |
|-------|------|-------------|
| `nameAr` | string | Arabic name |

### Update City

```bash
PUT /api/cities/:id
```

## Validation Rules

<Steps>
<Step title="Data Type Validation">
All fields must match their specified data types (string, number, date, etc.)
</Step>

<Step title="Length Constraints">
String fields have maximum length limits as specified in the schema
</Step>

<Step title="Foreign Key Validation">
Referenced entities must exist in their respective tables
</Step>

<Step title="Unique Constraints">
Source IDs and other unique fields cannot be duplicated
</Step>
</Steps>

## Error Handling

### Validation Errors (400)

```json
{
  "statusCode": 400,
  "message": "Validation failed",
  "errors": [
    {
      "field": "nameEn",
      "message": "Name in English is required"
    }
  ]
}
```

### Constraint Violations (409)

```json
{
  "statusCode": 409, 
  "message": "Duplicate source developer ID",
  "error": "Conflict"
}
```

<Tip>
Use the validation errors to provide user feedback in your application forms.
</Tip>

## Best Practices

- Always validate data on the client side before sending requests
- Use batch operations when creating multiple entities
- Implement proper error handling for all mutation operations
- Consider using transactions for related entity creation
- Monitor for constraint violations in production
