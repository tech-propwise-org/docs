## Overview

This document provides a complete reference for the API, covering developer search functionality, entity endpoints with configurable includes, lookup endpoints, direct mutations, and change request workflows.

<Note>
All endpoints require appropriate authentication. Most use Bearer token authentication unless otherwise specified.
</Note>

## Developer Search

Fuzzy search for developers by name using PostgreSQL `pg_trgm` similarity matching.

### Search Developers

<CodeGroup>
```bash curl
GET /developers/search?q=emaar&limit=5
Authorization: X-API-Key YOUR_API_KEY
```

```javascript fetch
const response = await fetch('/developers/search?q=emaar&limit=5', {
  headers: {
    'X-API-Key': 'YOUR_API_KEY'
  }
});
```
</CodeGroup>

**Authentication:** API Key (`ApiAuthGuard`)

#### Query Parameters

| Parameter | Type | Required | Default | Constraints |
|-----------|------|----------|---------|-------------|
| `q` | string | Yes | — | Minimum length: 1 |
| `limit` | integer | No | `10` | Range: 1–50 |

#### Search Algorithm

<Steps>
<Step title="Query Processing">
Trims the search query and runs `similarity()` against both `name_en` and `name_ar` columns
</Step>
<Step title="Similarity Filtering">
Filters rows where either similarity score is greater than 0.15
</Step>
<Step title="Result Ordering">
Orders results by the highest similarity score (descending)
</Step>
<Step title="Limit Application">
Returns up to the specified limit of results
</Step>
</Steps>

<Info>
This search functionality requires the `pg_trgm` PostgreSQL extension to be enabled.
</Info>

#### Response Format

```json Response (200 OK)
[
  {
    "id": 42,
    "nameEn": "Emaar Properties",
    "nameAr": "إعمار العقارية",
    "developerNumber": "DEV-001",
    "logo": "https://example.com/logo.png",
    "logoDark": "https://example.com/logo-dark.png",
    "licenseUrl": "https://example.com/license.pdf",
    "similarity": 0.85
  }
]
```

<Note>
All fields except `id` and `similarity` are nullable and may not be present in the response.
</Note>

## Entity GET Endpoints

All entity endpoints use Bearer token authentication and return paginated results in a standard envelope format.

### Pagination Envelope

```json Standard Response Format
{
  "data": [...],
  "total": 1234,
  "page": 1,
  "limit": 20
}
```

### Configurable Includes

<Tip>
Every entity endpoint supports an optional `include` query parameter to control which relations and computed fields are returned.
</Tip>

| Include Value | Behavior |
|---------------|----------|
| _(omitted)_ | Default relations populated (backward-compatible) |
| `none` | No relations or stats — only scalar fields |
| `all` | All allowed relations and stats |
| `field1,field2` | Only the specified relations/stats |

<Warning>
Invalid include values return a `400` error with the list of allowed options for that endpoint.
</Warning>

### Developers

#### List Developers

```bash
GET /api/developers
Authorization: Bearer <token>
```

| Parameter | Type | Required | Default | Constraints |
|-----------|------|----------|---------|-------------|
| `page` | integer | No | `1` | ≥ 1 |
| `limit` | integer | No | `20` | 1–100 |
| `sortBy` | string | No | `nameEn` | `id`, `nameEn`, `nameAr`, `developerNumber`, `createdAt` |
| `sortOrder` | string | No | `asc` | `asc`, `desc` |
| `nameEn` | string | No | — | Substring filter |
| `nameAr` | string | No | — | Substring filter |
| `search` | string | No | — | Searches `nameEn`, `nameAr` |
| `include` | string | No | — | Allowed: `stats` |

**Default includes (list):** None

#### Response Fields

| Field | Type | Always Present |
|-------|------|----------------|
| `id` | number | Yes |
| `nameEn` | string | No |
| `nameAr` | string | No |
| `developerNumber` | string | No |
| `logo` | string | No |
| `logoDark` | string | No |

#### Get Developer Details

```bash
GET /api/developers/:id
Authorization: Bearer <token>
```

**Default includes (detail):** `stats`  
**Allowed includes:** `stats`

<Tabs>
<Tab title="Base Response">
Extends the list item with:

| Field | Type | Notes |
|-------|------|-------|
| `sourceDeveloperId` | string | Always present |
</Tab>

<Tab title="With Stats">
When `stats` is included:

| Field | Type | Description |
|-------|------|-------------|
| `stats` | object | `{ projectsCount, masterProjectsCount }` |
</Tab>
</Tabs>

## Lookup Endpoints

Lookup endpoints provide reference data for dropdowns and form validation. These endpoints typically return simplified objects optimized for UI components.

<Info>
Lookup endpoints are designed for high-performance scenarios where you need minimal data for selection lists and form validation.
</Info>

## Direct Entity Mutations

Direct mutation endpoints allow immediate updates to entities without going through the change request workflow.

<Warning>
Direct mutations bypass the change request system and should be used carefully, typically only by administrative users or for system-level operations.
</Warning>

## Change Requests

The change request system provides a workflow for proposing, reviewing, and applying changes to entities with proper audit trails and approval processes.

<Note>
Change requests are the recommended way to handle entity modifications in production environments where approval workflows and audit trails are required.
</Note>

### Change Request Lifecycle

<Steps>
<Step title="Creation">
A change request is created with proposed modifications
</Step>
<Step title="Review">
The request enters a review state where it can be examined
</Step>
<Step title="Approval/Rejection">
Authorized users can approve or reject the change request
</Step>
<Step title="Application">
Approved changes are applied to the target entity
</Step>
</Steps>
