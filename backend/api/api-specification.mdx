## Overview

This document provides a complete reference for all API endpoints, including developer search, entity management, lookup operations, mutations, and change request workflows.

<Note>
All endpoints require proper authentication unless otherwise specified. Bearer tokens are used for most operations, while some endpoints use API key authentication.
</Note>

## 1. Developer Search

Fuzzy search for developers by name using PostgreSQL `pg_trgm` similarity matching.

### Search Developers

<CodeGroup>
```http GET /developers/search
GET /developers/search?q=emaar&limit=5
Authorization: X-API-Key <api_key>
```

```javascript
const response = await fetch('/developers/search?q=emaar&limit=5', {
  headers: {
    'X-API-Key': 'your-api-key'
  }
});
```
</CodeGroup>

**Authentication:** API key (`ApiAuthGuard`)

#### Query Parameters

| Parameter | Type   | Required | Default | Constraints  |
|-----------|--------|----------|---------|--------------|
| `q`       | string | yes      | —       | min length 1 |
| `limit`   | int    | no       | `10`    | 1–50         |

#### Search Algorithm

<Steps>
<Step title="Text Processing">
Trims the query string and prepares it for similarity matching
</Step>
<Step title="Similarity Calculation">
Runs `similarity()` against both `name_en` and `name_ar` columns using PostgreSQL's `pg_trgm` extension
</Step>
<Step title="Filtering">
Filters rows where either similarity score > 0.15
</Step>
<Step title="Ranking">
Orders results by the highest similarity score (descending)
</Step>
<Step title="Limiting">
Returns up to `limit` results
</Step>
</Steps>

<Warning>
This endpoint requires the `pg_trgm` PostgreSQL extension to be installed and enabled.
</Warning>

#### Response

```json
[
  {
    "id": 42,
    "nameEn": "Emaar Properties",
    "nameAr": "إعمار العقارية",
    "developerNumber": "DEV-001",
    "logo": "https://example.com/logo.png",
    "logoDark": "https://example.com/logo-dark.png",
    "licenseUrl": "https://example.com/license.pdf",
    "similarity": 0.85
  }
]
```

<Info>
All fields except `id` and `similarity` are nullable.
</Info>

## 2. Entity GET Endpoints

All entity endpoints require Bearer token authentication and return paginated results in a consistent envelope format.

<Note>
All endpoints use the following authentication header:
```
Authorization: Bearer <token>
```
</Note>

### Pagination Response Format

All list endpoints return data in this standardized envelope:

```json
{
  "data": [...],
  "total": 1234,
  "page": 1,
  "limit": 20
}
```

### Configurable Includes

<Tip>
Every entity supports an optional `include` query parameter to control which relations and computed fields are returned, optimizing response size and query performance.
</Tip>

| Include Value    | Behavior                                                          |
|------------------|-------------------------------------------------------------------|
| _(omitted)_      | Default relations populated (backward-compatible)                 |
| `none`           | No relations or stats — only scalar fields                       |
| `all`            | All allowed relations and stats                                   |
| `field1,field2`  | Only the specified relations/stats                                |

<Warning>
Invalid include values return `400` status with a list of allowed options for that endpoint.
</Warning>

### 2.1 Developers

#### List Developers

<CodeGroup>
```http GET /api/developers
GET /api/developers?page=1&limit=20&sortBy=nameEn&include=stats
Authorization: Bearer <token>
```

```javascript
const response = await fetch('/api/developers?include=stats', {
  headers: {
    'Authorization': 'Bearer ' + token
  }
});
```
</CodeGroup>

#### Query Parameters

| Parameter   | Type   | Required | Default  | Constraints                                              |
|-------------|--------|----------|----------|----------------------------------------------------------|
| `page`      | int    | no       | `1`      | ≥ 1                                                      |
| `limit`     | int    | no       | `20`     | 1–100                                                    |
| `sortBy`    | string | no       | `nameEn` | `id`, `nameEn`, `nameAr`, `developerNumber`, `createdAt` |
| `sortOrder` | string | no       | `asc`    | `asc`, `desc`                                            |
| `nameEn`    | string | no       | —        | substring filter                                         |
| `nameAr`    | string | no       | —        | substring filter                                         |
| `search`    | string | no       | —        | searches `nameEn`, `nameAr`                              |
| `include`   | string | no       | —        | allowed: `stats`                                         |

**Default includes (list):** _(none)_

#### Response Item

| Field             | Type   | Always Present |
|-------------------|--------|----------------|
| `id`              | number | yes            |
| `nameEn`          | string | —              |
| `nameAr`          | string | —              |
| `developerNumber` | string | —              |
| `logo`            | string | —              |
| `logoDark`        | string | —              |

#### Get Developer by ID

```http
GET /api/developers/:id?include=stats
Authorization: Bearer <token>
```

**Default includes (detail):** `stats`  
**Allowed includes:** `stats`

**Additional Response Fields:**

| Field               | Type   | Notes                                                               |
|---------------------|--------|---------------------------------------------------------------------|
| `sourceDeveloperId` | string | always present                                                      |
| `stats`             | object | `{ projectsCount, masterProjectsCount }` — only if `stats` included |

## 3. Lookup Endpoints

Lookup endpoints provide reference data for dropdowns, filters, and form selections. These endpoints typically return simpler, lighter-weight responses optimized for UI components.

## 4. Direct Entity Mutations

Direct mutations bypass the change request system and immediately modify entities. Use with caution in production environments.
</Warning>
