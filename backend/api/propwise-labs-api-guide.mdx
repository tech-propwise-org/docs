## Overview

The PropWise Labs API provides programmatic access to real estate data, analytics, and AI-powered valuations for external consumers.

### Base URL

All endpoints are prefixed with `/api`:

```
https://your-domain.com/api/projects
```

### Response Format

<Info>
All list endpoints return a standardized paginated envelope with `data`, `total`, `page`, and `limit` fields.
</Info>

```json
{
  "data": [ ... ],
  "total": 1234,
  "page": 1,
  "limit": 20
}
```

### Standard Pagination Parameters

| Parameter | Type | Default | Constraints |
|-----------|------|---------|-------------|
| `page` | integer | `1` | >= 1 |
| `limit` | integer | `20` | 1-100 |
| `sortBy` | string | entity-specific | allowlisted fields |
| `sortOrder` | string | `asc` | `asc` or `desc` |

### Configurable Includes

Most GET endpoints support an optional `include` query parameter to control which relations and computed fields are returned:

<Tabs>
  <Tab title="Default">
    Omit the parameter for backward-compatible default relations
  </Tab>
  <Tab title="None">
    Use `include=none` to return only scalar fields (no relations or stats)
  </Tab>
  <Tab title="All">
    Use `include=all` to return all allowed relations and computed fields
  </Tab>
  <Tab title="Specific">
    Use `include=field1,field2` to return only specified relations/stats
  </Tab>
</Tabs>

<Warning>
Invalid include values return `400` with a list of allowed options.
</Warning>

### Error Responses

<CodeGroup>

```json Standard Error
{
  "statusCode": 404,
  "message": "Project not found",
  "error": "Not Found"
}
```

```json Validation Error
{
  "statusCode": 400,
  "message": [
    "limit must not be greater than 100",
    "sortBy must be one of the following values: id, name, area, status, createdAt"
  ],
  "error": "Bad Request"
}
```

```json Change Request Validation
{
  "message": "Validation failed",
  "errors": [
    {
      "path": "projects[0]",
      "field": "name",
      "message": "name should not be empty"
    }
  ]
}
```

</CodeGroup>

### HTTP Status Codes

| Status | Meaning |
|--------|---------|
| `200` | Success |
| `400` | Validation error or bad request |
| `401` | Missing or invalid authentication |
| `403` | Insufficient permissions |
| `404` | Entity not found |
| `409` | Conflict (e.g., stale update) |

### Global Request Rules

<Check>
All request bodies must be JSON with `Content-Type: application/json`
</Check>

- Unknown fields in request bodies and query params are rejected
- Query params are auto-coerced to their declared types
- Comma-separated ID lists (e.g., `areaIds=1,2,3`) are parsed into integer arrays
- Soft-deleted records are excluded from all results

## Authentication

The API supports two authentication methods: **API Client Token** (for external consumers) and **JWT** (for internal dashboard users).

### API Client Token

<Steps>
  <Step title="Obtain credentials">
    Request an API key and secret from an administrator
  </Step>
  
  <Step title="Exchange for token">
    Use the credentials to get a short-lived Bearer token via `POST /api/auth/token`
  </Step>
  
  <Step title="Include in requests">
    Add the token to every request: `Authorization: Bearer <token>`
  </Step>
</Steps>

### Access Levels

| Level | Permissions |
|-------|-------------|
| `basic` | Read-only access to all entity, lookup, analytics, and change-request endpoints |
| `super` | All `basic` permissions plus direct entity mutations (PATCH projects/buildings, POST/PATCH/DELETE units) |

### `POST /api/auth/token`

Exchange API client credentials for a Bearer token.

<Note>
This is a public endpoint that does not require authentication.
</Note>

**Request Body:**

```json
{
  "apiKey": "your-api-key",
  "apiSecret": "your-api-secret"
}
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `apiKey` | string | yes | Your API key |
| `apiSecret` | string | yes | Your API secret |

**Response `200 OK`:**

```json
{
  "access_token": "eyJhbGciOiJIUzI1NiIs...",
  "token_type": "Bearer",
  "expires_in": 1800
}
```

| Field | Type | Description |
|-------|------|-------------|
| `access_token` | string | JWT token for Authorization header |
| `token_type` | string | Always `"Bearer"` |
| `expires_in` | number | Token lifetime in seconds (default 1800 = 30 min) |

**Error Responses:**

| Status | Condition |
|--------|-----------|
| `401` | Invalid API key or secret |
| `401` | API client is disabled |

#### `GET /api/developers/search`

Fuzzy search for developers by name using PostgreSQL trigram similarity.

<Tip>
This endpoint uses `pg_trgm` similarity scoring to find developers even with partial or misspelled names.
</Tip>

**Query Parameters:**

| Parameter | Type | Required | Default | Constraints |
|-----------|------|----------|---------|-------------|
| `q` | string | yes | - | Search query (min length 1) |
| `limit` | integer | no | `10` | Max results (1-50) |

**How It Works:**

1. Trims query and runs `similarity()` against both `name_en` and `name_ar` columns
2. Filters rows where either similarity score > 0.15
3. Orders by highest similarity score (descending)
4. Returns up to `limit` results

**Response `200 OK`:**

```json
[
  {
    "id": 42,
    "nameEn": "Emaar Properties",
    "nameAr": "إعمار العقارية",
    "developerNumber": "DEV-001",
    "logo": "https://example.com/logo.png",
    "logoDark": "https://example.com/logo-dark.png",
    "licenseUrl": "https://example.com/license.pdf",
    "similarity": 0.85
  }
]
```

<Note>
All fields except `id` and `similarity` are nullable.
</Note>

## Change Requests

Change requests allow `basic` access-level clients to propose entity modifications for admin approval.

<Info>
Change requests are ideal for external consumers who need to suggest updates but don't have direct write access.
</Info>

### Workflow

<Steps>
  <Step title="Submit request">
    External client submits a change request via `POST /api/change-requests`
  </Step>
  
  <Step title="Admin review">
    Admin reviews the request in the dashboard
  </Step>
  
  <Step title="Approval or rejection">
    Admin approves (applies changes) or rejects (with reason)
  </Step>
  
  <Step title="Notification">
    Client can poll status via `GET /api/change-requests/:id`
  </Step>
</Steps>

### `POST /api/change-requests`

Submit a new change request.

**Request Body:**

```json
{
  "projects": [
    {
      "id": 123,
      "nameEn": "Updated Project Name",
      "description": "New description"
    }
  ],
  "buildings": [
    {
      "id": 456,
      "numberOfFloors": 50
    }
  ],
  "units": [
    {
      "action": "create",
      "buildingId": 456,
      "unitNumber": "301",
      "unitTypeId": 2,
      "bedrooms": 2
    },
    {
      "action": "update",
      "id": 789,
      "builtUpArea": 1500
    },
    {
      "action": "delete",
      "id": 790
    }
  ],
  "reason": "Updating property information based on recent survey"
}
```

<Accordion title="Field Specifications">

**projects** (array, optional):
- Each item must have `id` (existing project)
- Include only fields you want to update
- All project PATCH fields are supported

**buildings** (array, optional):
- Each item must have `id` (existing building)
- Include only fields you want to update
- All building PATCH fields are supported

**units** (array, optional):
- Each item must have `action`: `"create"`, `"update"`, or `"delete"`
- **create**: requires `buildingId`, `unitNumber`, `unitTypeId` + optional fields
- **update**: requires `id` + fields to update
- **delete**: requires only `id`

**reason** (string, optional):
- Explanation for the change request
- Helps admins understand the context

</Accordion>

**Validation:**

<Warning>
At least one of `projects`, `buildings`, or `units` must be non-empty.
</Warning>

All entities are validated against their respective DTOs:
- Field types, formats, and constraints
- Foreign key references (IDs must exist)
- Business rules (e.g., unit numbers unique per building)

**Response `201 Created`:**

```json
{
  "id": 42,
  "apiClientId": 1,
  "status": "pending",
  "reason": "Updating property information based on recent survey",
  "projects": [ ... ],
  "buildings": [ ... ],
  "units": [ ... ],
  "rejectionReason": null,
  "createdAt": "2024-01-15T10:30:00.000Z",
  "updatedAt": "2024-01-15T10:30:00.000Z"
}
```

**Error Responses:**

```json
{
  "message": "Validation failed",
  "errors": [
    {
      "path": "projects[0]",
      "field": "nameEn",
      "message": "nameEn should not be empty"
    },
    {
      "path": "units[2]",
      "field": "buildingId",
      "message": "Building with ID 999 not found"
    }
  ]
}
```

## Appendix B: Entity Relationships

The following diagram illustrates the core entity relationships in the PropWise Labs data model:

```
State
  └── City
       └── Area
            └── Project (has Developer, Status, HandoverStatus, ProjectType, OwnershipType)
                 └── Building (has BuildingType)
                      └── Unit (has UnitType, View, Quality, Furnishing)
                           └── Transaction (sale/rental records)

Developer
  ├── LicenseSource
  ├── LicenseType
  ├── LegalStatus
  └── parentDeveloper / childDevelopers (self-referencing)

Building
  └── BuildingArea (many-to-many with Area, one marked as primary)
```
