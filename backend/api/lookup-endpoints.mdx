This section covers specialized lookup endpoints that provide reference data for dropdowns, search functionality, and entity relationships. These endpoints are optimized for frontend components that need to display lists of options or perform entity lookups.

## Developer search

The developer search endpoint provides fuzzy search capabilities for finding developers by name using PostgreSQL's `pg_trgm` similarity matching.

### Search developers

<CodeGroup>
```bash cURL
curl -X GET "https://api.example.com/developers/search?q=emaar&limit=5" \
  -H "Authorization: ApiKey YOUR_API_KEY"
```

```javascript JavaScript
const response = await fetch('/developers/search?q=emaar&limit=5', {
  headers: {
    'Authorization': 'ApiKey YOUR_API_KEY'
  }
});
const developers = await response.json();
```
</CodeGroup>

`GET /developers/search`

<Note>
This endpoint requires API key authentication using the `ApiAuthGuard`.
</Note>

#### Query parameters

| Parameter | Type | Required | Default | Description |
|-----------|------|----------|---------|-------------|
| `q` | string | Yes | — | Search query (minimum 1 character) |
| `limit` | integer | No | `10` | Maximum results to return (1-50) |

#### How it works

<Steps>
<Step title="Query processing">
The search query is trimmed and then compared against both English (`name_en`) and Arabic (`name_ar`) developer names using PostgreSQL's `similarity()` function.
</Step>

<Step title="Filtering">
Only results with a similarity score greater than 0.15 for either language are returned.
</Step>

<Step title="Sorting">
Results are ordered by the highest similarity score in descending order.
</Step>

<Step title="Limiting">
Returns up to the specified `limit` number of results.
</Step>
</Steps>

<Warning>
This endpoint requires the `pg_trgm` PostgreSQL extension to be installed and enabled on the database.
</Warning>

#### Response

Returns an array of developer objects with similarity scores:

```json
[
  {
    "id": 42,
    "nameEn": "Emaar Properties",
    "nameAr": "إعمار العقارية",
    "developerNumber": "DEV-001",
    "logo": "https://example.com/logo.png",
    "logoDark": "https://example.com/logo-dark.png",
    "licenseUrl": "https://example.com/license.pdf",
    "similarity": 0.85
  }
]
```

<Info>
All fields except `id` and `similarity` are nullable and may not be present in the response.
</Info>

## Entity lookup endpoints

All entity lookup endpoints support configurable includes and return paginated results. These endpoints are designed for populating dropdowns and relationship selections in forms.

### Include parameter

Every entity endpoint supports an optional `include` query parameter:

<AccordionGroup>
<Accordion title="Include options">
| Value | Behavior |
|-------|----------|
| _(omitted)_ | Default relations populated (backward-compatible) |
| `none` | Only scalar fields, no relations or computed data |
| `all` | All allowed relations and statistics |
| `field1,field2` | Only specified relations/statistics |
</Accordion>
</AccordionGroup>

<Tip>
Use `include=none` for simple dropdowns where you only need basic fields like ID and name. Use specific includes when you need related data.
</Tip>

### Developers

#### List developers

`GET /api/developers`

<Tabs>
<Tab title="Parameters">
| Parameter | Type | Default | Description |
|-----------|------|---------|-------------|
| `page` | integer | `1` | Page number (≥1) |
| `limit` | integer | `20` | Items per page (1-100) |
| `sortBy` | string | `nameEn` | Sort field: `id`, `nameEn`, `nameAr`, `developerNumber`, `createdAt` |
| `sortOrder` | string | `asc` | Sort direction: `asc`, `desc` |
| `nameEn` | string | — | Filter by English name (substring) |
| `nameAr` | string | — | Filter by Arabic name (substring) |
| `search` | string | — | Search across `nameEn` and `nameAr` |
| `include` | string | — | Allowed: `stats` |
</Tab>

<Tab title="Response">
```json
{
  "data": [
    {
      "id": 1,
      "nameEn": "Emaar Properties",
      "nameAr": "إعمار العقارية",
      "developerNumber": "DEV-001",
      "logo": "https://example.com/logo.png",
      "logoDark": "https://example.com/logo-dark.png"
    }
  ],
  "total": 150,
  "page": 1,
  "limit": 20
}
```
</Tab>
</Tabs>

#### Get developer details

`GET /api/developers/:id`

**Default includes:** `stats`  
**Available includes:** `stats`

The detail response extends the list item with additional fields:

```json
{
  "id": 1,
  "sourceDeveloperId": "ext_dev_123",
  "nameEn": "Emaar Properties",
  "nameAr": "إعمار العقارية",
  "developerNumber": "DEV-001",
  "logo": "https://example.com/logo.png",
  "logoDark": "https://example.com/logo-dark.png",
  "stats": {
    "projectsCount": 45,
    "masterProjectsCount": 12
  }
}
```

## Authentication

<CardGroup cols={2}>
<Card title="API Key Authentication" icon="key">
Developer search endpoints use API key authentication with the `ApiAuthGuard`.
</Card>

<Card title="Bearer Token Authentication" icon="shield-check">
Entity lookup endpoints require Bearer token authentication in the Authorization header.
</Card>
</CardGroup>

## Error handling

<Warning>
Invalid `include` parameter values return a `400` error with a list of allowed options for that endpoint.
</Warning>

Common error responses:

```json
{
  "statusCode": 400,
  "message": "Invalid include parameter. Allowed values: stats, city, city.state",
  "error": "Bad Request"
}
```

```json
{
  "statusCode": 401,
  "message": "Unauthorized",
  "error": "Unauthorized"
}
```
