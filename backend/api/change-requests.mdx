title: "Change Requests"
description: "Change request system for managing entity modifications with approval workflows"
---

Change requests provide a systematic way to manage modifications to entities through an approval workflow. This system ensures data integrity by routing proposed changes through appropriate approval processes before applying them to the live data.

<Note>
All change request endpoints require Bearer token authentication (`Authorization: Bearer <token>`).
</Note>

## Overview

The change request system allows users to:
- Submit proposed modifications to existing entities
- Track the approval status of submitted changes
- Review and approve/reject pending changes
- Maintain an audit trail of all modifications

## Change Request Status Flow

<Steps>
  <Step title="Pending">
    Change request is submitted and awaiting review
  </Step>
  <Step title="Under Review">
    Change request is being evaluated by approvers
  </Step>
  <Step title="Approved">
    Changes have been approved and applied to the entity
  </Step>
  <Step title="Rejected">
    Changes were rejected with reasoning provided
  </Step>
</Steps>

## API Endpoints

### List Change Requests

```http
GET /api/change-requests
```

#### Query Parameters

| Parameter | Type | Required | Default | Description |
|-----------|------|----------|---------|-------------|
| `page` | int | no | `1` | Page number (≥ 1) |
| `limit` | int | no | `20` | Items per page (1-100) |
| `status` | string | no | — | Filter by status: `pending`, `approved`, `rejected`, `under_review` |
| `entityType` | string | no | — | Filter by entity type: `developer`, `project`, `city`, `area` |
| `entityId` | int | no | — | Filter by specific entity ID |
| `submittedBy` | int | no | — | Filter by submitter user ID |
| `sortBy` | string | no | `createdAt` | Sort field: `id`, `createdAt`, `updatedAt`, `status` |
| `sortOrder` | string | no | `desc` | Sort direction: `asc`, `desc` |

### Submit Change Request

```http
POST /api/change-requests
```

#### Request Body

```json
{
  "entityType": "developer",
  "entityId": 42,
  "proposedChanges": {
    "nameEn": "Updated Developer Name",
    "phone": "+971-4-123-4567",
    "webpage": "https://newwebsite.com"
  },
  "comments": "Updating contact information as per latest company records"
}
```

#### Field Validation

<Tabs>
  <Tab title="Developer Changes">
    - `nameEn`: string (max 500 chars)
    - `nameAr`: string (max 500 chars)
    - `phone`: string (max 100 chars)
    - `webpage`: valid URL
    - `logo`: valid URL
    - `logoDark`: valid URL
  </Tab>
  <Tab title="Project Changes">
    - `nameEn`: string (max 500 chars)
    - `nameAr`: string (max 500 chars)
    - `description`: string
    - `completionDate`: ISO date
    - `website`: valid URL
  </Tab>
</Tabs>

#### Response `201 Created`

```json

  "id": 124,
  "entityType": "developer",
  "entityId": 42,
  "status": "pending",
```

### Review Change Request

```http
PATCH /api/change-requests/:id/review
```

<Warning>
This endpoint requires admin or reviewer permissions.
</Warning>

#### Request Body

```json
{
  "action": "approve",
  "comments": "Changes verified and approved"
}
```

| Field | Type | Required | Values | Description |
|-------|------|----------|--------|-------------|
| `action` | string | yes | `approve`, `reject` | Review decision |
| `comments` | string | no | — | Review notes (required for rejection) |

#### Response `200 OK`

```json
{
  "id": 124,
  "status": "approved",
  "reviewedBy": {
    "id": 3,
    "name": "Admin User",
    "email": "admin@example.com"
  },
  "reviewedAt": "2024-01-15T16:20:00Z",
  "reviewComments": "Changes verified and approved",
  "appliedAt": "2024-01-15T16:20:15Z"
}
```

### Withdraw Change Request

```http
DELETE /api/change-requests/:id
```

<Info>
Only the submitter can withdraw a pending change request.
</Info>

#### Response `204 No Content`

## Change Request Fields

### Core Fields

| Field | Type | Description |
|-------|------|-------------|
| `id` | int | Unique change request identifier |
| `entityType` | string | Type of entity being modified |
| `entityId` | int | ID of the target entity |
| `status` | string | Current approval status |
| `proposedChanges` | object | Fields to be updated with new values |
| `currentValues` | object | Current field values (for comparison) |
| `submittedAt` | timestamp | When the request was submitted |
| `submittedBy` | object | User who submitted the request |

### Review Fields

| Field | Type | Description |
|-------|------|-------------|
| `reviewedAt` | timestamp | When the request was reviewed |
| `reviewedBy` | object | User who reviewed the request |
| `reviewComments` | string | Review notes or rejection reason |
| `appliedAt` | timestamp | When approved changes were applied |

## Best Practices

<Tip>
**Batch Related Changes**: Submit all related modifications in a single change request rather than creating multiple requests for the same entity.
</Tip>

<Check>
**Provide Context**: Always include meaningful comments explaining the reason for the proposed changes.
</Check>

<Note>
**Review Timing**: Change requests are typically reviewed within 24-48 hours during business days.
</Note>

### Data Validation

- All proposed changes undergo the same validation as direct entity updates
- Foreign key references are validated to ensure referential integrity
- File uploads (logos, documents) must be accessible URLs
- Date fields must be in ISO 8601 format

### Audit Trail

The system maintains a complete audit trail including:
- Original field values at the time of request
- Proposed changes with timestamps
- Review decisions and comments
- Application timestamps for approved changes
