The Change Request System provides a workflow-based approach to managing entity modifications with approval processes, audit trails, and bulk operations.

## Overview

Change requests enable controlled modifications to entities through a structured workflow that includes:

- **Approval workflows** with configurable states
- **Audit trails** tracking all changes and decisions
- **Bulk operations** for processing multiple changes
- **Validation** ensuring data integrity before application

<Note>
All change request endpoints require Bearer token authentication (`Authorization: Bearer <token>`).
</Note>

## Core Concepts

### Change Request States

| State | Description |
|-------|-------------|
| `pending` | Awaiting review and approval |
| `approved` | Approved and ready for application |
| `rejected` | Rejected with reason |
| `applied` | Successfully applied to target entity |
| `failed` | Application failed with error details |

### Entity Types

The system supports change requests for the following entity types:

- `developer` - Developer entities
- `city` - City entities  
- `area` - Area entities
- `community` - Community entities
- `project` - Project entities
- `building` - Building entities
- `unit` - Unit entities

## API Endpoints

### List Change Requests

<CodeGroup>

```http GET /api/change-requests
GET /api/change-requests?entityType=developer&status=pending&page=1&limit=20
Authorization: Bearer <token>
```

```json Response

      "requestType": "update",

      "requestedBy": "user@company.com",
      "requestedAt": "2024-01-15T10:30:00Z",
      "reviewedBy": null,
      "reviewedAt": null,
      "appliedAt": null,
      "rejectionReason": null
    }
  ],
  "total": 45,
  "page": 1,
  "limit": 20

```

</CodeGroup>

#### Query Parameters

| Parameter | Type | Required | Default | Description |
|-----------|------|----------|---------|-------------|
| `page` | integer | no | `1` | Page number (≥ 1) |
| `limit` | integer | no | `20` | Items per page (1-100) |
| `entityType` | string | no | - | Filter by entity type |
| `entityId` | integer | no | - | Filter by specific entity ID |
| `status` | string | no | - | Filter by status |
| `requestedBy` | string | no | - | Filter by requester email |
| `sortBy` | string | no | `requestedAt` | Sort field |
| `sortOrder` | string | no | `desc` | Sort direction (`asc`, `desc`) |

### Create Change Request

<Steps>

<Step title="Prepare the request payload">
Include the entity type, ID, proposed changes, and justification reason.
</Step>

<Step title="Submit the change request">
<CodeGroup>

```http POST /api/change-requests
POST /api/change-requests
Authorization: Bearer <token>
Content-Type: application/json

{
  "entityType": "developer",
  "entityId": 42,
  "requestType": "update",
  "proposedChanges": {
    "nameEn": "Updated Developer Name",
    "webpage": "https://newwebsite.com",
    "phone": "+971-4-123-4567"
  },
  "reason": "Company rebranding and contact update"
}
```

```json Response 201 Created
{
  "id": 124,
  "entityType": "developer",
  "entityId": 42,
  "requestType": "update",
  "status": "pending",
  "proposedChanges": {
    "nameEn": "Updated Developer Name",
    "webpage": "https://newwebsite.com",
    "phone": "+971-4-123-4567"
  },
  "reason": "Company rebranding and contact update",
  "requestedBy": "user@company.com",
  "requestedAt": "2024-01-15T14:22:00Z",
  "validationErrors": null
}
```

</CodeGroup>
</Step>

<Step title="Handle validation errors">
If validation fails, the response will include specific error details:

```json Response 400 Bad Request
{
  "error": "Validation failed",
  "details": {
    "validationErrors": {
      "webpage": ["Invalid URL format"],
      "phone": ["Phone number format invalid for UAE"]
    }
  }
}
```
</Step>

</Steps>

#### Request Body

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `entityType` | string | yes | Target entity type |
| `entityId` | integer | yes | Target entity ID |
| `requestType` | string | yes | `create`, `update`, or `delete` |
| `proposedChanges` | object | yes | Fields to change with new values |
| `reason` | string | yes | Justification for the change |

<CodeGroup>

```http POST /api/change-requests/:id/apply

Authorization: Bearer <token>
```

  "id": 123,

  "appliedAt": "2024-01-15T17:30:00Z",

<Warning>
Only approved change requests can be applied. The system will validate the current state of the entity before applying changes to prevent conflicts.
</Warning>

## Entity-Specific Validation

Each entity type has specific validation rules applied to change requests:

<AccordionGroup>

<Accordion title="Developer Validation">
- `nameEn` and `nameAr`: Maximum 500 characters
- `webpage`: Valid URL format if provided
- `phone`, `fax`: Valid phone number format
- `developerNumber`: Must be unique if provided
- `licenseNumber`: Must be unique within the same license source
</Accordion>

<Accordion title="City Validation">
- `nameEn`: Required, maximum 255 characters
- `nameAr`: Maximum 255 characters if provided
- `stateId`: Must reference existing state
- Name uniqueness within the same state
</Accordion>

<Accordion title="Area Validation">
- `nameEn`: Required, maximum 255 characters
- `cityId`: Must reference existing city
- `municipalityNumber`: Must be unique if provided
- Name uniqueness within the same city
</Accordion>

<Accordion title="Community Validation">
- `nameEn`: Required, maximum 255 characters
- `areaId`: Must reference existing area
- Name uniqueness within the same area
</Accordion>

</AccordionGroup>

## Audit Trail

All change requests maintain a complete audit trail:

<Info>
The system automatically tracks:
- **Request creation** with timestamp and requester
- **Status changes** with reviewer and timestamp
- **Applied changes** with before/after values
- **Metadata** including source system and IP address
</Info>

Missing or invalid authentication token
</Card>
