Change requests provide a structured workflow for proposing modifications to entities that require approval before being applied to the system. This ensures data integrity and proper oversight for critical entity changes.

## Overview

The change request system allows users to submit proposed modifications to various entities including projects, buildings, units, and developers. Each request goes through an approval workflow before the changes are applied to the actual entities.

<Note>
Change requests are particularly useful for maintaining data quality in multi-user environments where certain modifications need oversight.
</Note>

## Change Request Workflow

<Steps>
<Step title="Submit Change Request">
Create a new change request with the proposed modifications using the appropriate endpoint for the entity type.
</Step>

<Step title="Review Process">
The change request enters a pending state where it can be reviewed by authorized personnel.
</Step>

<Step title="Approval or Rejection">
Authorized users can approve or reject the change request with optional comments.
</Step>

<Step title="Apply Changes">
Upon approval, the changes are automatically applied to the target entity.
</Step>
</Steps>

## Authentication

All change request endpoints require Bearer token authentication:

```http
Authorization: Bearer <your_token>
```

## API Endpoints

### Create Change Request

Create a new change request for entity modifications.

#### `POST /api/change-requests`

**Request Body:**

```json
{
  "entityType": "developer",
  "entityId": 123,
  "changes": {
    "nameEn": "Updated Company Name",
    "nameAr": "اسم الشركة المحدث",
    "phone": "+971-4-123-4567"
  },
  "reason": "Updating company contact information after rebranding"
}
```

**Response `201 Created`:**

```json
{
  "id": 456,
  "entityType": "developer",
  "entityId": 123,
  "status": "pending",
  "changes": {
    "nameEn": "Updated Company Name",
    "nameAr": "اسم الشركة المحدث",
    "phone": "+971-4-123-4567"
  },
  "reason": "Updating company contact information after rebranding",
  "submittedBy": {
    "id": 789,
    "name": "John Doe",
    "email": "john.doe@example.com"
  },
  "submittedAt": "2024-01-15T10:30:00Z",
  "reviewedBy": null,
  "reviewedAt": null,
  "reviewComments": null
}
```

### List Change Requests

Retrieve a paginated list of change requests with filtering options.

#### `GET /api/change-requests`

**Query Parameters:**

| Param        | Type   | Required | Default    | Constraints                           |
|--------------|--------|----------|------------|---------------------------------------|
| `page`       | int    | no       | `1`        | ≥ 1                                   |
| `limit`      | int    | no       | `20`       | 1–100                                 |
| `status`     | string | no       | —          | `pending`, `approved`, `rejected`     |
| `entityType` | string | no       | —          | `developer`, `city`, `area`, `community` |
| `entityId`   | int    | no       | —          | filter by specific entity             |
| `submittedBy`| int    | no       | —          | filter by submitter user ID           |
| `sortBy`     | string | no       | `submittedAt` | `id`, `submittedAt`, `status`      |
| `sortOrder`  | string | no       | `desc`     | `asc`, `desc`                         |

**Response `200 OK`:**

```json
{
  "data": [
    {
      "id": 456,
      "entityType": "developer",
      "entityId": 123,
      "status": "pending",
      "reason": "Updating company contact information after rebranding",
      "submittedBy": {
        "id": 789,
        "name": "John Doe"
      },
      "submittedAt": "2024-01-15T10:30:00Z"
    }
  ],
  "total": 1,
  "page": 1,
  "limit": 20
}
```

**Response `200 OK`:**

### Approve Change Request

Approve a pending change request, applying the changes to the target entity.

#### `POST /api/change-requests/:id/approve`

**Request Body:**

```json
{
  "comments": "Changes approved after verification of rebranding documents"
}
```

**Response `200 OK`:**

```json
{
  "id": 456,
  "status": "approved",
  "reviewedBy": {
    "id": 101,
    "name": "Jane Smith",
    "email": "jane.smith@example.com"
  },
  "reviewedAt": "2024-01-15T14:45:00Z",
  "reviewComments": "Changes approved after verification of rebranding documents"
}
```

<Info>
When a change request is approved, the changes are automatically applied to the target entity, and the entity's `updatedAt` timestamp is updated.
</Info>

### Reject Change Request

Reject a pending change request with optional comments explaining the decision.

#### `POST /api/change-requests/:id/reject`

**Request Body:**

```json
{
  "comments": "Missing required documentation for name change"
}
```

**Response `200 OK`:**

```json
{
  "id": 456,
  "status": "rejected",
  "reviewedBy": {
    "id": 101,
    "name": "Jane Smith",
    "email": "jane.smith@example.com"
  },
  "reviewedAt": "2024-01-15T14:45:00Z",
  "reviewComments": "Missing required documentation for name change"
}
```

## Status Values

<AccordionGroup>
<Accordion title="pending">
The change request has been submitted and is awaiting review. Changes can still be modified at this stage.
</Accordion>

<Accordion title="approved">
The change request has been approved and the changes have been applied to the target entity.
</Accordion>

<Accordion title="rejected">
The change request has been rejected and no changes were applied to the target entity.
</Accordion>
</AccordionGroup>

## Supported Entity Fields

<Tabs>
<Tab title="Developers">

</Tab>

- `municipalityNumber` - Municipality identification
- `imageUrl` - Area representative image
</Tab>

<Tab title="Communities">
- `nameEn`, `nameAr` - Community names
- Community-specific attributes and classifications
</Tab>
</Tabs>

## Error Responses

<Warning>
All endpoints may return the following error responses:
</Warning>

**`400 Bad Request`** - Invalid request parameters or malformed data
**`401 Unauthorized`** - Missing or invalid authentication token
**`403 Forbidden`** - Insufficient permissions for the requested operation
**`404 Not Found`** - Change request or target entity not found
**`409 Conflict`** - Change request already processed or entity state conflict

```json
{
  "error": "Bad Request",
  "message": "Invalid entity type. Supported types: developer, city, area, community",
  "statusCode": 400
}
```

## Best Practices

<Tip>
**Change Request Guidelines:**
- Provide clear, descriptive reasons for all change requests
- Include supporting documentation references when applicable
- Review changes carefully before submission as approved requests cannot be easily reversed
- Use bulk change requests for multiple related modifications to maintain data consistency
</Tip>

<Check>
**Security Considerations:**
- All change requests are logged with full audit trails
- Only authorized users can approve or reject change requests
- Original values are preserved for rollback capabilities
- All changes are validated against business rules before application
</Check>
