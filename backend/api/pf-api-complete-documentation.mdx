# Propwise Labs API Documentation

Complete API reference for external consumers (non-admin endpoints) with authentication, entities, lookups, mutations, and analytics.

## Overview

### Base URL

All endpoints are prefixed with `/api`. Example:

```
https://your-domain.com/api/projects
```

### Authentication

<Info>
The API supports two authentication methods for different use cases.
</Info>

**API Client Token (external consumers):**

<Steps>
<Step title="Obtain credentials">
Get an API key and secret from an admin.
</Step>
<Step title="Exchange for token">
Exchange them for a short-lived Bearer token via `POST /api/auth/token`.
</Step>
<Step title="Include in requests">
Include the token on every request: `Authorization: Bearer <token>`.
</Step>
</Steps>

**JWT (internal/dashboard users):**
- Obtained via `POST /api/auth/login` (admin dashboard)
- Sent as `Authorization: Bearer <token>` or via the `access_token` httpOnly cookie

Both methods are accepted on all external-facing endpoints. The API tries JWT first, then falls back to API key token.

**Access levels for API clients:**

| Level   | Permissions                                                                                            |
| ------- | ------------------------------------------------------------------------------------------------------ |
| `basic` | Read-only access to all entity, lookup, analytics, and change-request endpoints                        |
| `super` | All of `basic` plus direct entity mutations (PATCH projects, PATCH buildings, POST/PATCH/DELETE units) |

### Paginated Response Envelope

<Note>
All list endpoints return a standardized pagination envelope format.
</Note>

```json
{
  "data": [ ... ],
  "total": 1234,
  "page": 1,
  "limit": 20
}
```

Standard pagination query parameters:

| Param       | Type   | Default        | Constraints        |
| ----------- | ------ | -------------- | ------------------ |
| `page`      | int    | `1`            | >= 1               |
| `limit`     | int    | `20`           | 1--100             |
| `sortBy`    | string | _(per-entity)_ | allowlisted fields |
| `sortOrder` | string | `asc`          | `asc`, `desc`      |

### Configurable Includes

Most entity GET endpoints support an optional `include` query parameter that controls which relations and computed fields are returned:

| Value           | Behavior                                          |
| --------------- | ------------------------------------------------- |
| _(omitted)_     | Default relations populated (backward-compatible) |
| `none`          | No relations or stats -- only scalar fields       |
| `all`           | All allowed relations and stats                   |
| `field1,field2` | Only the specified relations/stats                |

<Warning>
Invalid include values return `400` with the list of allowed options. Virtual includes like `stats` and `buildingAreas` control expensive aggregate queries rather than ORM relations.
</Warning>

### Error Response Format

**Standard errors:**

```json
{
  "statusCode": 404,
  "message": "Project not found",
  "error": "Not Found"
}
```

**Validation errors (400):**

```json
{
  "statusCode": 400,
  "message": [
    "limit must not be greater than 100",
    "sortBy must be one of the following values: id, name, area, status, createdAt"
  ],
  "error": "Bad Request"
}
```

**Change request validation errors (400):**

```json
{
  "message": "Validation failed",
  "errors": [{ "path": "projects[0]", "field": "name", "message": "name should not be empty" }]
}
```

| Status | Meaning                                                              |
| ------ | -------------------------------------------------------------------- |
| `400`  | Validation error, bad request                                        |
| `401`  | Missing or invalid authentication                                    |
| `403`  | Insufficient permissions (e.g. `basic` client attempting a mutation) |
| `404`  | Entity not found                                                     |
| `409`  | Conflict (stale update in change requests)                           |

### Global Request Rules

<Tip>
All request and response handling follows these standardized rules:
</Tip>

- All request bodies are JSON (`Content-Type: application/json`)
- Unknown fields in request bodies and query params are rejected (`forbidNonWhitelisted`)
- Query params are auto-coerced to their declared types (e.g. `"1"` becomes `1` for int fields)
- Comma-separated ID lists (e.g. `areaIds=1,2,3`) are parsed into integer arrays
- Soft-deleted records are excluded from all results

## Authentication

### Exchange API credentials for token

<CodeGroup>

```bash curl
curl -X POST https://your-domain.com/api/auth/token \
  -H "Content-Type: application/json" \
  -d '{
    "apiKey": "your-api-key",
    "apiSecret": "your-api-secret"
  }'
```

```javascript JavaScript
const response = await fetch('/api/auth/token', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
  },
  body: JSON.stringify({
    apiKey: 'your-api-key',
    apiSecret: 'your-api-secret'
  })
});

const data = await response.json();
```

</CodeGroup>

**POST** `/api/auth/token`

Exchange API client credentials for a short-lived Bearer token.

**Auth:** None (public endpoint)

**Request body:**

| Field       | Type   | Required | Constraints |
| ----------- | ------ | -------- | ----------- |
| `apiKey`    | string | yes      | non-empty   |
| `apiSecret` | string | yes      | non-empty   |

**Response `200 OK`:**

```json
{
  "access_token": "eyJhbGciOiJIUzI1NiIs...",
  "token_type": "Bearer",
  "expires_in": 1800
}
```

| Field          | Type   | Description                                        |
| -------------- | ------ | -------------------------------------------------- |
| `access_token` | string | JWT token to use in `Authorization: Bearer` header |
| `token_type`   | string | Always `"Bearer"`                                  |
| `expires_in`   | number | Token lifetime in seconds (default 1800 = 30 min)  |

**Error responses:**

| Status | Condition                 |
| ------ | ------------------------- |
| `401`  | Invalid API key or secret |
| `401`  | API client is disabled    |

#### Search developers

**GET** `/api/developers/search`

Fuzzy search for developers by name using PostgreSQL `pg_trgm` similarity.

| Param   | Type   | Required | Default | Constraints  |
| ------- | ------ | -------- | ------- | ------------ |
| `q`     | string | yes      | --      | min length 1 |
| `limit` | int    | no       | `10`    | 1--50        |

<Info>
The search uses PostgreSQL trigram similarity, filtering rows where either `name_en` or `name_ar` has a similarity score > 0.15, ordered by the highest score.
</Info>

**Response `200 OK`:**

```json
[
  {
    "id": 42,
    "nameEn": "Emaar Properties",
    "nameAr": "إعمار العقارية",
    "developerNumber": "DEV-001",
    "logo": "https://example.com/logo.png",
    "logoDark": "https://example.com/logo-dark.png",
    "licenseUrl": "https://example.com/license.pdf",
    "similarity": 0.85
  }
]
```

All fields except `id` and `similarity` are nullable.

## Change Requests

<Info>
Change requests provide a controlled workflow for proposing entity modifications. All API clients can create change requests regardless of access level.
</Info>

### Create change request

**POST** `/api/change-requests`

<CodeGroup>

```bash curl
curl -X POST https://your-domain.com/api/change-requests \
  -H "Authorization: Bearer YOUR_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "projects": [
      {
        "id": 123,
        "name": "Updated Project Name",
        "status": "COMPLETED"
      }
    ],
    "buildings": [
      {
        "id": 456,
        "floors": 25
      }
    ]
  }'
```

```javascript JavaScript
const response = await fetch('/api/change-requests', {
  method: 'POST',
  headers: {
    'Authorization': 'Bearer YOUR_TOKEN',
    'Content-Type': 'application/json'
  },
  body: JSON.stringify({
    projects: [
      {
        id: 123,
        name: "Updated Project Name",
        status: "COMPLETED"
      }
    ],
    buildings: [
      {
        id: 456,
        floors: 25
      }
    ]
  })
});
```

</CodeGroup>

**Request body:**

| Field       | Type    | Required | Description                    |
| ----------- | ------- | -------- | ------------------------------ |
| `projects`  | array   | no       | Array of project updates       |
| `buildings` | array   | no       | Array of building updates      |
| `units`     | array   | no       | Array of unit updates          |

At least one array must be non-empty. Each entity object must include:
- `id`: number (required) - existing entity ID
- Other fields: same validation as direct mutation endpoints

**Response `201 Created`:**

## Unit Evaluation (AI Valuation)

<Warning>
AI valuation endpoints may have rate limits and require special permissions. Results are estimates only and should not be used as official appraisals.
</Warning>

### Request unit valuation

**POST** `/api/ai-valuation/evaluate`

<CodeGroup>

```bash curl
curl -X POST https://your-domain.com/api/ai-valuation/evaluate \
  -H "Authorization: Bearer YOUR_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "unitId": 12345,
    "features": {
      "view": "SEA_VIEW",
      "floor": 15,
      "furnished": true,
      "parking": 2
    }
  }'
```

```javascript JavaScript
const response = await fetch('/api/ai-valuation/evaluate', {
  method: 'POST',
  headers: {
    'Authorization': 'Bearer YOUR_TOKEN',
    'Content-Type': 'application/json'
  },
  body: JSON.stringify({
    unitId: 12345,
    features: {
      view: "SEA_VIEW",
      floor: 15,
      furnished: true,
      parking: 2
    }
  })
});
```

</CodeGroup>

**Request body:**

| Field      | Type   | Required | Description                          |
| ---------- | ------ | -------- | ------------------------------------ |
| `unitId`   | number | yes      | Existing unit ID for context         |
| `features` | object | no       | Additional features for valuation    |

**Features object (all optional):**

| Field       | Type    | Description                    |

| `floor`     | number  | Floor number                   |
| `furnished` | boolean | Furnished status               |
| `parking`   | number  | Number of parking spaces       |
| `balcony`   | boolean | Has balcony                    |
| `amenities` | array   | List of building amenities     |
