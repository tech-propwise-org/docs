## Developer Search

<Info>
The developer search endpoint uses PostgreSQL `pg_trgm` similarity for fuzzy matching across both English and Arabic developer names.
</Info>

### GET /developers/search

**Authentication:** API key (`ApiAuthGuard`)

#### Query Parameters

| Parameter | Type | Required | Default | Constraints |
|-----------|------|----------|---------|-------------|
| `q` | string | yes | — | min length 1 |
| `limit` | int | no | `10` | 1–50 |

#### How it works

<Steps>
<Step title="Query processing">
The system trims the `q` parameter and runs `similarity()` against both `name_en` and `name_ar` columns.
</Step>
<Step title="Filtering">
Filters rows where either similarity score is greater than 0.15.
</Step>
<Step title="Ordering">
Orders results by the highest of the two similarity scores in descending order.
</Step>
<Step title="Limiting">
Returns up to `limit` results (maximum 50).
</Step>
</Steps>

<Note>
This endpoint requires the `pg_trgm` PostgreSQL extension to be installed and enabled.
</Note>

#### Response (200 OK)

```json
[
  {
    "id": 42,
    "nameEn": "Emaar Properties",
    "nameAr": "إعمار العقارية",
    "developerNumber": "DEV-001",
    "logo": "https://example.com/logo.png",
    "logoDark": "https://example.com/logo-dark.png",
    "licenseUrl": "https://example.com/license.pdf",
    "similarity": 0.85
  }
]
```

<Warning>
All fields except `id` and `similarity` are nullable and may not be present in the response.
</Warning>

## Entity GET Endpoints

<Info>
All endpoints require a Bearer token: `Authorization: Bearer <token>`
</Info>

All list endpoints return a paginated response envelope:

```json
{
  "data": [...],
  "total": 1234,
  "page": 1,
  "limit": 20
}
```

### Configurable Includes

Every entity supports an optional `include` query parameter to control relations and computed fields:

<Tabs>
<Tab title="Include options">
| Value | Behavior |
|-------|----------|
| _(omitted)_ | Default relations populated (backward-compatible) |
| `none` | No relations or stats — only scalar fields |
| `all` | All allowed relations and stats |
| `field1,field2` | Only the specified relations/stats |
</Tab>
<Tab title="Virtual includes">
Virtual includes like `stats` and `buildingAreas` control expensive aggregate queries rather than ORM relations.
</Tab>
</Tabs>

<Warning>
Invalid `include` values return a `400` error with the list of allowed options.
</Warning>

### Developers

#### GET /api/developers

| Parameter | Type | Required | Default | Constraints |
|-----------|------|----------|---------|-------------|
| `page` | int | no | `1` | ≥ 1 |
| `limit` | int | no | `20` | 1–100 |
| `sortBy` | string | no | `nameEn` | `id`, `nameEn`, `nameAr`, `developerNumber`, `createdAt` |
| `sortOrder` | string | no | `asc` | `asc`, `desc` |
| `nameEn` | string | no | — | substring filter |
| `nameAr` | string | no | — | substring filter |
| `search` | string | no | — | searches `nameEn`, `nameAr` |
| `include` | string | no | — | allowed: `stats` |

**Default includes (list):** _(none)_

**Response item:**

```json
{
  "id": 42,
  "nameEn": "Emaar Properties",
  "nameAr": "إعمار العقارية", 
  "developerNumber": "DEV-001",
  "logo": "https://example.com/logo.png",
  "logoDark": "https://example.com/logo-dark.png"
}
```

#### GET /api/developers/:id

| Parameter | Type | Required |
|-----------|------|----------|
| `id` | int (path) | yes |
| `include` | string (query) | no |

**Default includes (detail):** `stats`  
**Allowed includes:** `stats`

**Response (extends list item):**

```json
{
  "id": 42,
  "sourceDeveloperId": "EXT_DEV_001",
  "nameEn": "Emaar Properties",
  "stats": {
    "projectsCount": 25,
    "masterProjectsCount": 5
  }
}
```

<Note>
The `stats` object is only included when the `stats` include is specified.
</Note>

## Lookup Endpoints

<Info>
Lookup endpoints provide reference data for dropdowns and form validation. They typically return simple key-value pairs without pagination.
</Info>

These endpoints serve static or semi-static reference data used throughout the application for forms, filters, and validation.

## Direct Entity Mutations

<Warning>
Direct mutations bypass the change request system and immediately modify data. Use with caution in production environments.
</Warning>

Direct mutation endpoints allow immediate data modifications without going through the change request workflow. These are typically used for administrative operations or in environments where immediate changes are required.

## Change Requests

<Tip>
Change requests provide an audit trail and approval workflow for data modifications, making them ideal for production environments where data integrity is critical.
</Tip>

The change request system allows users to propose modifications to entities that require approval before being applied. This provides better control over data quality and maintains an audit trail of all changes.
