## Developer Search

Fuzzy search for developers by name using PostgreSQL `pg_trgm` similarity.

### `GET /developers/search`

<Note>
**Authentication Required:** API key (`ApiAuthGuard`)
</Note>

#### Query Parameters

| Parameter | Type   | Required | Default | Constraints  |
|-----------|--------|----------|---------|--------------|
| `q`       | string | yes      | —       | min length 1 |
| `limit`   | int    | no       | `10`    | 1–50         |

#### How It Works

<Steps>
<Step title="Query Processing">
Trims the query string and runs `similarity()` against both `name_en` and `name_ar` columns
</Step>
<Step title="Filtering">
Filters rows where either similarity score is greater than 0.15
</Step>
<Step title="Ordering">
Orders results by the highest of the two similarity scores (descending)
</Step>
<Step title="Limiting">
Returns up to the specified limit of results
</Step>
</Steps>

<Warning>
Requires the `pg_trgm` PostgreSQL extension to be installed and enabled.
</Warning>

#### Response `200 OK`

```json
[
  {
    "id": 42,
    "nameEn": "Emaar Properties",
    "nameAr": "إعمار العقارية",
    "developerNumber": "DEV-001",
    "logo": "https://…/logo.png",
    "logoDark": "https://…/logo-dark.png",
    "licenseUrl": "https://…/license.pdf",
    "similarity": 0.85
  }
]
```

<Info>
All fields except `id` and `similarity` are nullable.
</Info>

## Entity GET Endpoints

<Note>
All endpoints require a Bearer token (`Authorization: Bearer <token>`).
</Note>

All list endpoints return a paginated envelope:

```json
{
  "data": [...],
  "total": 1234,
  "page": 1,
  "limit": 20
}
```

### Configurable Includes

Every entity supports an optional `include` query parameter that controls which relations and computed fields are returned:

| Value           | Behavior                                                          |
|-----------------|-------------------------------------------------------------------|
| _(omitted)_     | Default relations populated (backward-compatible, same as before) |
| `none`          | No relations or stats — only scalar fields                        |
| `all`           | All allowed relations and stats                                   |
| `field1,field2` | Only the specified relations/stats                                |

<Warning>
Invalid values return `400` with the list of allowed options. Virtual includes like `stats` and `buildingAreas` control expensive aggregate queries rather than ORM relations.
</Warning>

### Developers

#### `GET /api/developers`

| Parameter   | Type   | Required | Default  | Constraints                                              |
|-------------|--------|----------|----------|----------------------------------------------------------|
| `page`      | int    | no       | `1`      | ≥ 1                                                      |
| `limit`     | int    | no       | `20`     | 1–100                                                    |
| `sortBy`    | string | no       | `nameEn` | `id`, `nameEn`, `nameAr`, `developerNumber`, `createdAt` |
| `sortOrder` | string | no       | `asc`    | `asc`, `desc`                                            |
| `nameEn`    | string | no       | —        | substring filter                                         |
| `nameAr`    | string | no       | —        | substring filter                                         |
| `search`    | string | no       | —        | searches `nameEn`, `nameAr`                              |
| `include`   | string | no       | —        | allowed: `stats`                                         |

**Default includes (list):** _(none)_

**Response item:**

| Field             | Type   | Always present |
|-------------------|--------|----------------|
| `id`              | number | yes            |
| `nameEn`          | string | —              |
| `nameAr`          | string | —              |
| `developerNumber` | string | —              |
| `logo`            | string | —              |
| `logoDark`        | string | —              |

#### `GET /api/developers/:id`

| Parameter | Type           | Required |
|-----------|----------------|----------|
| `id`      | int (path)     | yes      |
| `include` | string (query) | no       |

**Default includes (detail):** `stats`  
**Allowed includes:** `stats`

**Response (extends list item):**

| Field               | Type   | Notes                                                               |
|---------------------|--------|---------------------------------------------------------------------|
| `sourceDeveloperId` | string | always present                                                      |
| `stats`             | object | `{ projectsCount, masterProjectsCount }` — only if `stats` included |

## Lookup Endpoints

<Note>
Lookup endpoints provide reference data for dropdowns and form selections.
</Note>

These endpoints typically return simple lists without pagination, designed for populating select options and other UI components.

## Direct Entity Mutations

<Warning>
Direct mutations bypass the change request workflow and immediately modify entity data. Use with caution in production environments.
</Warning>

Direct mutations are available for immediate updates when change request approval is not required.

## Change Requests

<Info>
Change requests provide an approval workflow for entity modifications, ensuring data quality and governance.
</Info>

The change request system allows for:
- Proposed changes to existing entities
- Approval workflows
- Change history tracking
- Rollback capabilities

<Tip>
Use change requests for production environments where data modifications require review and approval.
</Tip>
