# API Specification - Comprehensive

This comprehensive API specification covers all available endpoints including developer search, entity operations, lookups, mutations, and change request workflows.

<Note>
All endpoints require proper authentication unless otherwise specified. Bearer token authentication is used for most endpoints, while developer search uses API key authentication.
</Note>

## Developer Search

Fuzzy search for developers by name using PostgreSQL `pg_trgm` similarity matching.

### `GET /developers/search`

**Authentication:** API key (`ApiAuthGuard`)

<Info>
This endpoint uses PostgreSQL's trigram similarity for fuzzy matching against both English and Arabic developer names.
</Info>

#### Query Parameters

| Parameter | Type   | Required | Default | Constraints  |
|-----------|--------|----------|---------|--------------|
| `q`       | string | yes      | —       | min length 1 |
| `limit`   | int    | no       | `10`    | 1–50         |

#### How It Works

<Steps>
<Step title="Query Processing">
Trims the query string `q` and runs `similarity()` against both `name_en` and `name_ar` columns
</Step>
<Step title="Filtering">
Filters rows where either similarity score is greater than 0.15
</Step>
<Step title="Ordering">
Orders results by the highest similarity score (descending)
</Step>
<Step title="Limiting">
Returns up to `limit` results (default 10, max 50)
</Step>
</Steps>

<Warning>
Requires the `pg_trgm` PostgreSQL extension to be installed and enabled.
</Warning>

#### Response `200 OK`

```json
[
  {
    "id": 42,
    "nameEn": "Emaar Properties",
    "nameAr": "إعمار العقارية",
    "developerNumber": "DEV-001",
    "logo": "https://example.com/logo.png",
    "logoDark": "https://example.com/logo-dark.png",
    "licenseUrl": "https://example.com/license.pdf",
    "similarity": 0.85
  }
]
```

<Note>
All fields except `id` and `similarity` are nullable.
</Note>

## Entity GET Endpoints

<Warning>
All endpoints require a Bearer token (`Authorization: Bearer <token>`).
</Warning>

All list endpoints return a paginated envelope structure:

```json
{
  "data": [...],
  "total": 1234,
  "page": 1,
  "limit": 20
}
```

### Configurable Includes

Every entity supports an optional `include` query parameter for controlling relations and computed fields:

<Tabs>
<Tab title="Include Options">
| Value | Behavior |
|-------|----------|
| _(omitted)_ | Default relations populated (backward-compatible) |
| `none` | No relations or stats — only scalar fields |
| `all` | All allowed relations and stats |
| `field1,field2` | Only the specified relations/stats |
</Tab>
<Tab title="Virtual Includes">
Virtual includes like `stats` and `buildingAreas` control expensive aggregate queries rather than ORM relations.
</Tab>
</Tabs>

<Info>
Invalid `include` values return `400` with the list of allowed options.
</Info>

### Developers

#### `GET /api/developers`

<CodeGroup>
```bash Basic Request
curl -H "Authorization: Bearer <token>" \
  "https://api.example.com/api/developers"
```

```bash With Filters
curl -H "Authorization: Bearer <token>" \
  "https://api.example.com/api/developers?search=emaar&include=stats&limit=50"
```
</CodeGroup>

#### Query Parameters

| Parameter | Type | Required | Default | Constraints |
|-----------|------|----------|---------|-------------|
| `page` | int | no | `1` | ≥ 1 |
| `limit` | int | no | `20` | 1–100 |
| `sortBy` | string | no | `nameEn` | `id`, `nameEn`, `nameAr`, `developerNumber`, `createdAt` |
| `sortOrder` | string | no | `asc` | `asc`, `desc` |
| `nameEn` | string | no | — | substring filter |
| `nameAr` | string | no | — | substring filter |
| `search` | string | no | — | searches `nameEn`, `nameAr` |
| `include` | string | no | — | allowed: `stats` |

**Default includes (list):** _(none)_

#### Response Structure

| Field | Type | Always Present |
|-------|------|----------------|
| `id` | number | yes |
| `nameEn` | string | — |
| `nameAr` | string | — |
| `developerNumber` | string | — |
| `logo` | string | — |
| `logoDark` | string | — |

#### `GET /api/developers/:id`

| Parameter | Type | Required |
|-----------|------|----------|
| `id` | int (path) | yes |
| `include` | string (query) | no |

**Default includes (detail):** `stats`  
**Allowed includes:** `stats`

**Response (extends list item):**

| Field | Type | Notes |
|-------|------|-------|
| `sourceDeveloperId` | string | always present |
| `stats` | object | `{ projectsCount, masterProjectsCount }` — only if `stats` included |

## Lookup Endpoints

Lookup endpoints provide reference data for dropdowns and form validation.

<Note>
Lookup endpoints typically return simplified objects with `id`, `nameEn`, and `nameAr` fields unless otherwise specified.
</Note>

## Direct Entity Mutations

Direct mutations allow immediate updates to entity data without going through the change request workflow.

<Warning>
Direct mutations bypass approval workflows and should be used with caution in production environments.
</Warning>

## Change Requests

The change request system provides a workflow for proposing, reviewing, and approving modifications to entity data.

<Steps>
<Step title="Submit Change Request">
Create a new change request with proposed modifications
</Step>
<Step title="Review Process">
Authorized users review the proposed changes
</Step>
<Step title="Approval/Rejection">
Changes are either approved and applied, or rejected with feedback
</Step>
<Step title="Audit Trail">
All changes are tracked with timestamps and user attribution
</Step>
</Steps>

<Check>
Change requests provide a complete audit trail for all data modifications, ensuring transparency and accountability.
</Check>
