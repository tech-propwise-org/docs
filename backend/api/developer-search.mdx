# Developer Search

The Developer Search API provides fuzzy search capabilities for finding developers by name using PostgreSQL's `pg_trgm` similarity extension. This endpoint is particularly useful for building search interfaces where users might not know the exact developer name.

## Endpoint

### `GET /developers/search`

Performs a fuzzy search across developer names in both English and Arabic using PostgreSQL's trigram similarity matching.

<Note>
This endpoint requires API key authentication using the `ApiAuthGuard`.
</Note>

## Query Parameters

| Parameter | Type   | Required | Default | Constraints  | Description |
|-----------|--------|----------|---------|--------------|-------------|
| `q`       | string | yes      | —       | min length 1 | Search query string |
| `limit`   | int    | no       | `10`    | 1–50         | Maximum number of results |

## How It Works

<Steps>
<Step title="Query Processing">
The search query `q` is trimmed and then processed using PostgreSQL's `similarity()` function against both `name_en` and `name_ar` columns.
</Step>

<Step title="Similarity Filtering">
Only rows where either English or Arabic name similarity score is greater than 0.15 are included in results.
</Step>

<Step title="Result Ranking">
Results are ordered by the highest similarity score (either English or Arabic) in descending order.
</Step>

<Step title="Result Limiting">
Up to `limit` results are returned (default 10, maximum 50).
</Step>
</Steps>

<Warning>
This endpoint requires the `pg_trgm` PostgreSQL extension to be installed and enabled on your database.
</Warning>

## Response

### Success Response `200 OK`

Returns an array of developer objects matching the search criteria:

```json
[
  {
    "id": 42,
    "nameEn": "Emaar Properties",
    "nameAr": "إعمار العقارية",
    "developerNumber": "DEV-001",
    "logo": "https://example.com/logo.png",
    "logoDark": "https://example.com/logo-dark.png",
    "licenseUrl": "https://example.com/license.pdf",
    "similarity": 0.85
  }
]
```

### Response Fields

| Field             | Type   | Description | Nullable |
|-------------------|--------|-------------|----------|
| `id`              | int    | Unique developer identifier | No |
| `nameEn`          | string | Developer name in English | Yes |
| `nameAr`          | string | Developer name in Arabic | Yes |
| `developerNumber` | string | Developer registration number | Yes |
| `logo`            | string | URL to developer logo | Yes |
| `logoDark`        | string | URL to dark theme logo | Yes |
| `licenseUrl`      | string | URL to license document | Yes |
| `similarity`      | float  | Similarity score (0.0-1.0) | No |

<Info>
All fields except `id` and `similarity` are nullable. The `similarity` field indicates how closely the developer name matches your search query.
</Info>

## Example Usage

<Tabs>
<Tab title="cURL">
```bash
curl -X GET "https://api.example.com/developers/search?q=emaar&limit=5" \
  -H "Authorization: Bearer YOUR_API_KEY"
```
</Tab>

<Tab title="JavaScript">
```javascript
const response = await fetch('/developers/search?q=emaar&limit=5', {
  headers: {
    'Authorization': 'Bearer YOUR_API_KEY'
  }
});

const developers = await response.json();
console.log(developers);
```
</Tab>

<Tab title="Python">
```python
import requests

headers = {
    'Authorization': 'Bearer YOUR_API_KEY'
}

params = {
    'q': 'emaar',
    'limit': 5
}

response = requests.get('/developers/search', headers=headers, params=params)
developers = response.json()
```
</Tab>
</Tabs>

## Developer Entity Schema

The search results are based on the `developers` table with the following structure:

<AccordionGroup>
<Accordion title="Complete Developer Schema">

| Column | Type | Notes |
|--------|------|-------|
| `id` | int PK | Auto-increment primary key |
| `source_developer_id` | varchar(50) | Unique external source ID |
| `developer_number` | varchar(50) | Registration number (nullable) |
| `name_en` | varchar(500) | English name (nullable, indexed) |
| `name_ar` | varchar(500) | Arabic name (nullable, indexed) |
| `registration_date` | date | Registration date (nullable) |
| `license_source` | FK | Foreign key to license_sources |
| `license_type` | FK | Foreign key to license_types |
| `license_number` | varchar(100) | License number (nullable) |
| `license_issue_date` | date | License issue date (nullable) |
| `license_expiry_date` | date | License expiry date (nullable) |
| `chamber_of_commerce_no` | varchar(100) | Chamber of commerce number |
| `legal_status` | FK | Foreign key to legal_statuses |
| `webpage` | varchar(500) | Developer website (nullable) |
| `license_url` | varchar(500) | License document URL (nullable) |
| `phone` | varchar(100) | Contact phone (nullable) |
| `fax` | varchar(100) | Fax number (nullable) |
| `logo` | varchar(500) | Logo URL (nullable) |
| `logo_dark` | varchar(500) | Dark theme logo URL (nullable) |
| `created_at` | timestamptz | Creation timestamp (default now()) |
| `updated_at` | timestamptz | Last update timestamp (auto-updated) |
| `deleted_at` | timestamptz | Soft delete timestamp (nullable) |

</Accordion>
</AccordionGroup>

## Relationships

The developer entity has the following relationships:

- **Projects**: One-to-many relationship via `developer` field
- **Master Projects**: One-to-many relationship via `masterDeveloper` field

<Tip>
For better search performance, consider using this fuzzy search for user-facing search interfaces, while using exact match queries for programmatic lookups when you have the exact developer ID or number.
</Tip>

## Error Responses

### `400 Bad Request`
Returned when query parameters are invalid:

```json
{
  "error": "Validation failed",
  "details": [
    {
      "field": "q",
      "message": "Query string must be at least 1 character long"
    }
  ]
}
```

### `401 Unauthorized`
Returned when API key is missing or invalid:

```json
{
  "error": "Unauthorized",
  "message": "Invalid or missing API key"
}
```
