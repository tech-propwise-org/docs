## 1. Developer Search

Fuzzy search for developers by name using PostgreSQL `pg_trgm` similarity.

### Search Developers

<CodeGroup>
```bash cURL
curl -X GET "https://api.example.com/developers/search?q=Emaar&limit=10" \
  -H "X-API-Key: your-api-key"
```

```javascript JavaScript
const response = await fetch('/developers/search?q=Emaar&limit=10', {
  headers: {
    'X-API-Key': 'your-api-key'
  }
});
```
</CodeGroup>

**Endpoint:** `GET /developers/search`  
**Auth:** API key (`ApiAuthGuard`)

#### Query Parameters

| Parameter | Type | Required | Default | Constraints |
|-----------|------|----------|---------|-------------|
| `q` | string | Yes | — | min length 1 |
| `limit` | integer | No | `10` | 1–50 |

#### How Search Works

<Steps>
<Step title="Query Processing">
The system trims the query string and runs `similarity()` against both `name_en` and `name_ar` columns
</Step>

<Step title="Similarity Filtering">
Filters rows where either similarity score is greater than 0.15
</Step>

<Step title="Result Ranking">
Orders results by the highest of the two similarity scores in descending order
</Step>

<Step title="Limit Application">
Returns up to the specified `limit` number of results
</Step>
</Steps>

<Info>
This search functionality requires the `pg_trgm` PostgreSQL extension to be installed.
</Info>

#### Response Format

```json
[
  {
    "id": 42,
    "nameEn": "Emaar Properties",
    "nameAr": "إعمار العقارية", 
    "developerNumber": "DEV-001",
    "logo": "https://example.com/logo.png",
    "logoDark": "https://example.com/logo-dark.png",
    "licenseUrl": "https://example.com/license.pdf",
    "similarity": 0.85
  }
]
```

<Warning>
All fields except `id` and `similarity` are nullable and may not be present in the response.
</Warning>

## 2. Entity GET Endpoints

All endpoints require Bearer token authentication and return paginated results in this format:

```json
{
  "data": [...],
  "total": 1234,
  "page": 1,
  "limit": 20
}
```

### Configurable Includes

<Tip>
Every entity endpoint supports an optional `include` query parameter to control which relations and computed fields are returned.
</Tip>

| Include Value | Behavior |
|---------------|----------|
| _(omitted)_ | Default relations populated (backward-compatible) |
| `none` | No relations or stats — only scalar fields |
| `all` | All allowed relations and stats |
| `field1,field2` | Only the specified relations/stats |

<Warning>
Invalid include values return `400` status with a list of allowed options.
</Warning>

### 2.1 Developers

#### List Developers

**Endpoint:** `GET /api/developers`

<Tabs>
<Tab title="Query Parameters">
| Parameter | Type | Required | Default | Constraints |
|-----------|------|----------|---------|-------------|
| `page` | integer | No | `1` | ≥ 1 |
| `limit` | integer | No | `20` | 1–100 |
| `sortBy` | string | No | `nameEn` | `id`, `nameEn`, `nameAr`, `developerNumber`, `createdAt` |
| `sortOrder` | string | No | `asc` | `asc`, `desc` |
| `nameEn` | string | No | — | substring filter |
| `nameAr` | string | No | — | substring filter |
| `search` | string | No | — | searches `nameEn`, `nameAr` |
| `include` | string | No | — | allowed: `stats` |
</Tab>

<Tab title="Response Fields">
| Field | Type | Always Present |
|-------|------|----------------|
| `id` | number | Yes |
| `nameEn` | string | — |
| `nameAr` | string | — |
| `developerNumber` | string | — |
| `logo` | string | — |
| `logoDark` | string | — |
</Tab>
</Tabs>

**Default includes (list):** _(none)_

#### Get Developer Details

**Endpoint:** `GET /api/developers/:id`

| Parameter | Type | Required |
|-----------|------|----------|
| `id` | integer (path) | Yes |
| `include` | string (query) | No |

**Default includes (detail):** `stats`  
**Allowed includes:** `stats`

**Additional response fields:**
- `sourceDeveloperId` (string, always present)
- `stats` (object with `projectsCount`, `masterProjectsCount`) — only if `stats` included
