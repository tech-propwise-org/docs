This document provides a comprehensive overview of all API endpoints, including developer search functionality, entity operations, lookup endpoints, direct mutations, and change request workflows.

<Note>
All endpoints require proper authentication. API key authentication uses `ApiAuthGuard`, while entity endpoints require Bearer token authorization.
</Note>

## Developer Search

Fuzzy search for developers by name using PostgreSQL `pg_trgm` similarity.

### Search Developers

<CodeGroup>
```http GET /developers/search
GET /developers/search?q=emaar&limit=5
Authorization: X-API-Key your-api-key
```

```javascript
const response = await fetch('/developers/search?q=emaar&limit=5', {
  headers: {
    'X-API-Key': 'your-api-key'
  }
});
const developers = await response.json();
```
</CodeGroup>

#### Query Parameters

| Parameter | Type   | Required | Default | Constraints  | Description |
|-----------|--------|----------|---------|--------------|-------------|
| `q`       | string | yes      | —       | min length 1 | Search query |
| `limit`   | int    | no       | `10`    | 1–50         | Max results |

<Steps>
<Step title="Query Processing">
The system trims the query string and runs similarity analysis against both `name_en` and `name_ar` columns.
</Step>

<Step title="Similarity Filtering">
Filters rows where either similarity score exceeds 0.15 threshold.
</Step>

<Step title="Result Ranking">
Orders results by the highest similarity score in descending order.
</Step>

<Step title="Response Limiting">
Returns up to the specified limit of results.
</Step>
</Steps>

<Warning>
This endpoint requires the `pg_trgm` PostgreSQL extension to be installed and enabled.
</Warning>

#### Response Format

```json
[
  {
    "id": 42,
    "nameEn": "Emaar Properties",
    "nameAr": "إعمار العقارية",
    "developerNumber": "DEV-001",
    "logo": "https://…/logo.png",
    "logoDark": "https://…/logo-dark.png",
    "licenseUrl": "https://…/license.pdf",
    "similarity": 0.85
  }
]
```

<Info>
All fields except `id` and `similarity` are nullable and may not be present in the response.
</Info>

## Entity GET Endpoints

All entity endpoints require Bearer token authentication and return paginated results in a standardized envelope format.

### Authentication

```http
Authorization: Bearer <your-token>
```

### Pagination Envelope

All list endpoints return results in this format:

```json
{
  "data": [...],
  "total": 1234,
  "page": 1,
  "limit": 20
}
```

### Configurable Includes

<Tabs>
<Tab title="Include Options">
Every entity endpoint supports an optional `include` query parameter:

| Value           | Behavior |
|-----------------|----------|
| _(omitted)_     | Default relations (backward-compatible) |
| `none`          | Only scalar fields, no relations |
| `all`           | All available relations and stats |
| `field1,field2` | Only specified relations/stats |
</Tab>

<Tab title="Virtual Includes">
Some includes like `stats` and `buildingAreas` control expensive aggregate queries rather than ORM relations.
</Tab>
</Tabs>

<Warning>
Invalid include values return `400` status with a list of allowed options for that endpoint.
</Warning>

## Developers

### List Developers

<CodeGroup>
```http GET /api/developers
GET /api/developers?page=1&limit=20&sortBy=nameEn&include=stats
Authorization: Bearer <token>
```

```javascript
const response = await fetch('/api/developers?include=stats', {
  headers: {
    'Authorization': 'Bearer your-token'
  }
});
```
</CodeGroup>

#### Query Parameters

| Parameter   | Type   | Required | Default  | Constraints | Description |
|-------------|--------|----------|----------|-------------|-------------|
| `page`      | int    | no       | `1`      | ≥ 1         | Page number |
| `limit`     | int    | no       | `20`     | 1–100       | Items per page |
| `sortBy`    | string | no       | `nameEn` | See allowed values | Sort field |
| `sortOrder` | string | no       | `asc`    | `asc`, `desc` | Sort direction |
| `nameEn`    | string | no       | —        | — | Filter by English name |
| `nameAr`    | string | no       | —        | — | Filter by Arabic name |
| `search`    | string | no       | —        | — | Search both names |
| `include`   | string | no       | —        | `stats` | Additional data |

**Allowed sortBy values:** `id`, `nameEn`, `nameAr`, `developerNumber`, `createdAt`

#### Response Item

```json
{
  "id": 1,
  "nameEn": "Emaar Properties",
  "nameAr": "إعمار العقارية",
  "developerNumber": "DEV-001",
  "logo": "https://example.com/logo.png",
  "logoDark": "https://example.com/logo-dark.png"
}
```

## Communities

### List Communities

Similar structure to areas with the following differences:

**Allowed sortBy values:** `id`, `nameEn`, `nameAr`, `area`, `createdAt`

**Query Parameters:**
- `areaId` - Filter by area instead of city
- `search` searches `nameEn`, `nameAr`, `area.nameEn`

**Default includes:** `area`  
**Allowed includes:** `area`, `area.city`, `area.city.state`, `stats`

<Tip>
Communities are nested under areas, which are nested under cities, allowing for hierarchical filtering and includes.
</Tip>

## Direct Entity Mutations

Direct mutations provide immediate database updates with proper validation and authentication.

<Warning>
Direct mutations bypass the change request workflow and immediately modify the database. Use with caution in production environments.
</Warning>

### Create Developer

<CodeGroup>
```http POST /api/developers
POST /api/developers
Authorization: Bearer <token>
Content-Type: application/json

{
  "nameEn": "New Developer",
  "nameAr": "مطور جديد",
  "developerNumber": "DEV-123"
}
```

```javascript
const response = await fetch('/api/developers', {
  method: 'POST',
  headers: {
    'Authorization': 'Bearer your-token',
    'Content-Type': 'application/json'
  },
  body: JSON.stringify({
    nameEn: "New Developer",
    nameAr: "مطور جديد",
    developerNumber: "DEV-123"
  })
});
```
</CodeGroup>

### Update Developer

<CodeGroup>

### Delete Developer

<CodeGroup>
```http DELETE /api/developers/:id
DELETE /api/developers/123
Authorization: Bearer <token>
```

```javascript
const response = await fetch('/api/developers/123', {
  method: 'DELETE',
  headers: {
    'Authorization': 'Bearer your-token'

```

<Info>
Delete operations perform soft deletes by setting the `deleted_at` timestamp rather than physically removing records.
</Info>

## Change Requests

The change request system provides a workflow for proposing, reviewing, and applying changes to entities with proper audit trails.

### Submit Change Request

<CodeGroup>
```http POST /api/change-requests
POST /api/change-requests
Authorization: Bearer <token>
Content-Type: application/json

{
  "entityType": "developer",
  "entityId": 123,
  "operation": "update",
  "changes": {
    "nameEn": "Updated Developer Name",
    "logo": "https://example.com/new-logo.png"
  },
  "reason": "Updating company branding"
}
```

```javascript
const response = await fetch('/api/change-requests', {
  method: 'POST',
  headers: {
    'Authorization': 'Bearer your-token',
    'Content-Type': 'application/json'
  },
  body: JSON.stringify({
    entityType: "developer",
    entityId: 123,
    operation: "update",
    changes: {
      nameEn: "Updated Developer Name",
      logo: "https://example.com/new-logo.png"
    },
    reason: "Updating company branding"
  })
});
```
</CodeGroup>

#### Request Body

| Field        | Type   | Required | Description |
|--------------|--------|----------|-------------|
| `entityType` | string | yes      | Target entity type |
| `entityId`   | number | yes      | Target entity ID |
| `operation`  | string | yes      | `create`, `update`, `delete` |
| `changes`    | object | yes      | Proposed changes |
| `reason`     | string | no       | Change justification |

### Review Change Request

<CodeGroup>
```http PUT /api/change-requests/:id/review
PUT /api/change-requests/456/review
Authorization: Bearer <token>
Content-Type: application/json

{
  "action": "approve",
  "comments": "Changes look good, approved for implementation"
}
```

```javascript
const response = await fetch('/api/change-requests/456/review', {
  method: 'PUT',
  headers: {
    'Authorization': 'Bearer your-token',
    'Content-Type': 'application/json'
  },
  body: JSON.stringify({
    action: "approve",
    comments: "Changes look good, approved for implementation"
  })
});
```
</CodeGroup>

#### Review Actions

| Action     | Description |
|------------|-------------|
| `approve`  | Approve and apply changes |
| `reject`   | Reject the change request |
| `pending`  | Mark as pending review |

## Error Handling

All endpoints return standardized error responses:

```json
{
  "error": "Validation failed",
  "message": "The provided data is invalid",
  "details": {
    "field": "nameEn",
    "message": "Name is required"
  },
  "statusCode": 400
}
```

### Common Status Codes

| Code | Description |
|------|-------------|
| 400  | Bad Request - Invalid parameters |
| 401  | Unauthorized - Invalid or missing token |
| 403  | Forbidden - Insufficient permissions |
| 404  | Not Found - Resource doesn't exist |
| 422  | Unprocessable Entity - Validation failed |
| 500  | Internal Server Error |
