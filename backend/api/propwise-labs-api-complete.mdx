## Overview

The PropWise Labs API provides comprehensive access to real estate data, market analytics, and AI-powered valuations for external consumers.

### Base URL

All endpoints are prefixed with `/api`:

```
https://your-domain.com/api/projects
```

### Authentication Methods

<CardGroup cols={2}>
  <Card title="API Client Token" icon="key">
    For external consumers - exchange credentials for short-lived Bearer tokens
  </Card>
  <Card title="JWT Token" icon="shield-halved">
    For internal dashboard users - obtained via admin login
  </Card>
</CardGroup>

### Paginated Response Envelope

All list endpoints return a consistent pagination structure:

```json
{
  "data": [ ... ],
  "total": 1234,
  "page": 1,
  "limit": 20
}
```

**Standard pagination parameters:**

| Parameter | Type | Default | Constraints |
|-----------|------|---------|-------------|
| `page` | integer | `1` | >= 1 |
| `limit` | integer | `20` | 1-100 |
| `sortBy` | string | entity-specific | allowlisted fields |
| `sortOrder` | string | `asc` | `asc` or `desc` |

### Configurable Includes

Control which relations and computed fields are returned using the `include` query parameter:

| Value | Behavior |
|-------|----------|
| _(omitted)_ | Default relations populated (backward-compatible) |
| `none` | No relations or stats - only scalar fields |
| `all` | All allowed relations and stats |
| `field1,field2` | Only specified relations/stats |

<Note>
Invalid `include` values return a `400` error with the list of allowed options.
</Note>

### Error Response Format

<Tabs>
  <Tab title="Standard Error">
    ```json
    {
      "statusCode": 404,
      "message": "Project not found",
      "error": "Not Found"
    }
    ```
  </Tab>
  <Tab title="Validation Error">
    ```json
    {
      "statusCode": 400,
      "message": [
        "limit must not be greater than 100",
        "sortBy must be one of the following values: id, name, area, status, createdAt"
      ],
      "error": "Bad Request"
    }
    ```
  </Tab>
  <Tab title="Change Request Error">
    ```json
    {
      "message": "Validation failed",
      "errors": [
        {
          "path": "projects[0]",
          "field": "name",
          "message": "name should not be empty"
        }
      ]
    }
    ```
  </Tab>
</Tabs>

**HTTP status codes:**

| Status | Meaning |
|--------|---------|
| `400` | Validation error, bad request |
| `401` | Missing or invalid authentication |
| `403` | Insufficient permissions |
| `404` | Entity not found |
| `409` | Conflict (stale update in change requests) |

### Global Request Rules

<Check>All request bodies are JSON with `Content-Type: application/json`</Check>
<Check>Unknown fields in request bodies and query params are rejected</Check>
<Check>Query params are auto-coerced to declared types</Check>
<Check>Comma-separated ID lists are parsed into integer arrays</Check>
<Check>Soft-deleted records are excluded from all results</Check>

## Authentication

### Exchange API credentials for token

<Note>
This endpoint is public and does not require authentication.
</Note>

```http
POST /api/auth/token
```

Exchange API client credentials for a short-lived Bearer token.

**Request body:**

```json
{
  "apiKey": "your-api-key",
  "apiSecret": "your-api-secret"
}
```

| Field | Type | Required | Constraints |
|-------|------|----------|-------------|
| `apiKey` | string | yes | non-empty |
| `apiSecret` | string | yes | non-empty |

**Response `200 OK`:**

```json
{
  "access_token": "eyJhbGciOiJIUzI1NiIs...",
  "token_type": "Bearer",
  "expires_in": 1800
}
```

| Field | Type | Description |
|-------|------|-------------|
| `access_token` | string | JWT token to use in `Authorization: Bearer` header |
| `token_type` | string | Always `"Bearer"` |
| `expires_in` | number | Token lifetime in seconds (default 1800 = 30 min) |

**Error responses:**

| Status | Condition |
|--------|-----------|
| `401` | Invalid API key or secret |
| `401` | API client is disabled |

#### Search developers

```http
GET /api/developers/search
```

Fuzzy search for developers by name using PostgreSQL `pg_trgm` similarity.

**Query parameters:**

| Parameter | Type | Required | Default | Constraints |
|-----------|------|----------|---------|-------------|
| `q` | string | yes | - | min length 1 |
| `limit` | integer | no | `10` | 1-50 |

<Tip>
The search runs `similarity()` against both `name_en` and `name_ar` columns, filtering rows where either similarity score > 0.15, ordered by the highest score.
</Tip>

## Change Requests

Change requests allow clients to propose bulk entity changes that require admin approval before being applied to the database.

<Info>
Available to both `basic` and `super` access levels.
</Info>

### Change Request Workflow

<Steps>
  <Step title="Submit change request">
    Client submits a change request with one or more entity changes
  </Step>
  <Step title="Validation">
    System validates all changes and returns errors if validation fails
  </Step>
  <Step title="Pending approval">
    Change request enters `pending` status, awaiting admin review
  </Step>
  <Step title="Admin review">
    Admin approves or rejects the change request via dashboard
  </Step>
  <Step title="Applied or rejected">
    If approved, changes are applied to entities. If rejected, request is marked as rejected
  </Step>
</Steps>

| Entity | Allowed Actions |
|--------|-----------------|

| Buildings | `update` |

**Validation rules:**

<AccordionGroup>
  <Accordion title="Update actions">
    - Must include `id` field
    - At least one field to update (besides `id` and `action`)
    - Entity must exist and not be soft-deleted
  </Accordion>
  <Accordion title="Create actions">
    - Must include all required fields for entity type
    - Foreign keys must reference existing entities
  </Accordion>
  <Accordion title="Delete actions">
    - Must include `id` field only
    - Entity must exist and not be soft-deleted
  </Accordion>
</AccordionGroup>
