## Overview

The Propwise Labs API provides comprehensive access to real estate data including projects, buildings, units, developers, and market analytics. This documentation covers all external-facing endpoints for non-admin consumers.

<Note>
All endpoints are prefixed with `/api`. Example: `https://your-domain.com/api/projects`
</Note>

## Authentication

The API supports two authentication methods for external consumers.

### API Client Token (Recommended for External Integration)

<Steps>
<Step title="Obtain Credentials">
Request an API key and secret from an administrator.
</Step>

<Step title="Exchange for Bearer Token">
Use the credentials to get a short-lived token via `POST /api/auth/token`.
</Step>

<Step title="Include Token in Requests">
Add the token to every request: `Authorization: Bearer <token>`
</Step>
</Steps>

### Access Levels

<CardGroup cols={2}>
<Card title="Basic Level" icon="eye">
Read-only access to all entity, lookup, analytics, and change-request endpoints
</Card>

<Card title="Super Level" icon="pencil">
All basic permissions plus direct entity mutations (PATCH projects, PATCH buildings, POST/PATCH/DELETE units)
</Card>
</CardGroup>

### `POST /api/auth/token`

Exchange API client credentials for a Bearer token.

<CodeGroup>
```json Request
{
  "apiKey": "your-api-key",
  "apiSecret": "your-api-secret"
}
```

```json Response (200 OK)
{
  "access_token": "eyJhbGciOiJIUzI1NiIs...",
  "token_type": "Bearer",
  "expires_in": 1800
}
```
</CodeGroup>

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `apiKey` | string | yes | Your API key |
| `apiSecret` | string | yes | Your API secret |

<Warning>
Tokens expire after 30 minutes (1800 seconds). Store the `expires_in` value and refresh before expiration.
</Warning>

## Response Format

### Paginated Responses

All list endpoints return data in a standardized envelope:

```json
{
  "data": [...],
  "total": 1234,
  "page": 1,
  "limit": 20
}
```

### Standard Pagination Parameters

| Parameter | Type | Default | Constraints | Description |
|-----------|------|---------|-------------|-------------|
| `page` | int | `1` | >= 1 | Page number |
| `limit` | int | `20` | 1-100 | Items per page |
| `sortBy` | string | varies | allowlisted fields | Sort field |
| `sortOrder` | string | `asc` | `asc`, `desc` | Sort direction |

### Configurable Includes

Most entity endpoints support an `include` parameter to control which relations and computed fields are returned:

<Tabs>
<Tab title="Default Behavior">
When `include` is omitted, default relations are populated for backward compatibility.
</Tab>

<Tab title="None">
Use `include=none` to get only scalar fields without any relations or stats.
</Tab>

<Tab title="All">
Use `include=all` to get all available relations and computed fields.
</Tab>

<Tab title="Selective">
Use `include=field1,field2` to specify exactly which relations/stats to include.
</Tab>
</Tabs>

### Error Responses

<CodeGroup>
```json Standard Error
{
  "statusCode": 404,
  "message": "Project not found",
  "error": "Not Found"
}
```

```json Validation Error
{
  "statusCode": 400,
  "message": [
    "limit must not be greater than 100",
    "sortBy must be one of: id, name, area, status, createdAt"
  ],
  "error": "Bad Request"
}
```

```json Change Request Validation
{
  "message": "Validation failed",
  "errors": [
    {
      "path": "projects[0]",
      "field": "name",
      "message": "name should not be empty"
    }
  ]
}
```
</CodeGroup>

| Status Code | Meaning |
|-------------|---------|
| `400` | Validation error, bad request |
| `401` | Missing or invalid authentication |
| `403` | Insufficient permissions |
| `404` | Entity not found |
| `409` | Conflict (stale update in change requests) |

## Entity Endpoints

### Developers

#### List Developers

`GET /api/developers`

| Parameter | Type | Required | Default | Description |
|-----------|------|----------|---------|-------------|
| `page` | int | no | `1` | Page number |
| `limit` | int | no | `20` | Items per page (1-100) |

| `nameAr` | string | no | -- | Filter by Arabic name (substring) |

| `include` | string | no | -- | Include relations: `licenseSource`, `licenseType`, `legalStatus`, `parentDeveloper`, `childDevelopers`, `stats` |

#### Search Developers

`GET /api/developers/search`

Fuzzy search using PostgreSQL `pg_trgm` similarity.

| Parameter | Type | Required | Default | Description |
|-----------|------|----------|---------|-------------|
| `q` | string | yes | -- | Search query (min length 1) |
| `limit` | int | no | `10` | Max results (1-50) |

<Info>
The search compares against both English and Arabic names, filtering results with similarity > 0.15 and ordering by highest similarity score.
</Info>

<CodeGroup>
```json Response
[
  {
    "id": 42,
    "nameEn": "Emaar Properties",
    "nameAr": "إعمار العقارية",
    "developerNumber": "DEV-001",
    "logo": "https://example.com/logo.png",
    "logoDark": "https://example.com/logo-dark.png",

  }
]
```
</CodeGroup>

## Change Requests

Change requests allow external clients to propose bulk modifications that require approval.

<Info>
Change requests support optimistic locking to prevent conflicts. Always include the current `version` when updating existing entities.
</Info>

### Submit Change Request

`POST /api/change-requests`

<CodeGroup>
```json Request Body
{
  "title": "Q4 2024 Project Updates",
  "description": "Quarterly update to project information",
  "projects": [
    {
      "action": "update",
      "id": 123,
      "version": 5,
      "data": {
        "status": "completed",
        "completionDate": "2024-12-15"
      }
    }
  ],
  "buildings": [
    {
      "action": "create",
      "data": {
        "projectId": 123,
        "nameEn": "New Building A",
        "floors": 25
      }
    }
  ]
}
```

```json Response (201 Created)
{
  "id": 456,
  "title": "Q4 2024 Project Updates", 
  "description": "Quarterly update to project information",
  "status": "pending",
  "submittedAt": "2024-01-15T10:30:00Z",
  "submittedBy": {
    "id": "api-client-123",
    "type": "api_client",
    "name": "External Integration"
  }
}
```
</CodeGroup>

### List Change Requests

`GET /api/change-requests`

| Parameter | Type | Required | Description |

### Get Change Request Details

## Entity Relationships

<Info>
Understanding entity relationships helps optimize your API usage and minimize unnecessary requests.
</Info>

**Core Hierarchy:**
- **Developer** → **Project** → **Building** → **Unit**
- **City** → **Area** (hierarchical, can have sub-areas)
- **Project** can have child projects (master developments)

**Cross-References:**
- Projects and Buildings can span multiple Areas
- Units can have multiple Amenities
- All entities link to PropertyTypes and other lookup data

**Virtual Fields:**
- `stats` objects contain computed aggregations (project count, unit count, etc.)
- `buildingAreas` provides area breakdowns for projects
- These are expensive queries, so use `include` parameter selectively
