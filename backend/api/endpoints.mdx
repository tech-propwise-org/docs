<Note>
All endpoints require proper authentication unless otherwise specified. Bearer tokens are required for entity endpoints, while API keys are used for developer search.
</Note>

## Developer Search

Fuzzy search for developers by name using PostgreSQL `pg_trgm` similarity.

### Search Developers

<CodeGroup>
```http GET /developers/search
GET /developers/search?q=emaar&limit=10
Authorization: X-API-Key <your-api-key>
```

```javascript
const response = await fetch('/developers/search?q=emaar&limit=10', {
  headers: {
    'X-API-Key': 'your-api-key'
  }
});
const developers = await response.json();
```
</CodeGroup>

#### Query Parameters

| Parameter | Type | Required | Default | Description |
|-----------|------|----------|---------|-------------|
| `q` | string | yes | — | Search query (minimum 1 character) |
| `limit` | integer | no | 10 | Maximum results (1-50) |

<Info>
The search uses PostgreSQL's trigram similarity against both English (`name_en`) and Arabic (`name_ar`) names, filtering results with similarity > 0.15.
</Info>

#### Response

```json
[
  {
    "id": 42,
    "nameEn": "Emaar Properties",
    "nameAr": "إعمار العقارية", 
    "developerNumber": "DEV-001",
    "logo": "https://example.com/logo.png",
    "logoDark": "https://example.com/logo-dark.png",
    "licenseUrl": "https://example.com/license.pdf",
    "similarity": 0.85
  }
]
```

<Warning>
Requires the `pg_trgm` PostgreSQL extension to be installed and enabled.
</Warning>

### Configurable Includes

<AccordionGroup>
  <Accordion title="Include Parameter Options">
    Every entity endpoint supports an optional `include` query parameter:

    | Value | Behavior |
    |-------|----------|
    | _(omitted)_ | Default relations (backward-compatible) |
    | `none` | No relations - scalar fields only |
    | `all` | All allowed relations and stats |
    | `field1,field2` | Only specified relations/stats |

    <Tip>
    Use `include=none` for faster responses when you don't need related data.
    </Tip>
  </Accordion>
</AccordionGroup>

### Developers

#### List Developers

<CodeGroup>
```http GET /api/developers
GET /api/developers?page=1&limit=20&sortBy=nameEn&include=stats
Authorization: Bearer <token>
```

```javascript
const response = await fetch('/api/developers?include=stats', {
  headers: {
    'Authorization': 'Bearer ' + token
  }
});
```
</CodeGroup>

##### Query Parameters

| Parameter | Type | Default | Constraints | Description |
|-----------|------|---------|-------------|-------------|
| `page` | integer | 1 | ≥ 1 | Page number |
| `limit` | integer | 20 | 1-100 | Items per page |
| `sortBy` | string | nameEn | `id`, `nameEn`, `nameAr`, `developerNumber`, `createdAt` | Sort field |
| `sortOrder` | string | asc | `asc`, `desc` | Sort direction |
| `nameEn` | string | — | — | Filter by English name |
| `nameAr` | string | — | — | Filter by Arabic name |
| `search` | string | — | — | Search across name fields |
| `include` | string | — | `stats` | Include related data |

#### Get Developer

```http
GET /api/developers/:id?include=stats
Authorization: Bearer <token>
```

**Default includes:** `stats`  
**Response includes:** `sourceDeveloperId`, and `stats` object with `projectsCount` and `masterProjectsCount`

## Direct Entity Mutations

<Warning>
Direct mutations bypass the change request system and immediately modify the database. Use with caution in production environments.
</Warning>

### Create Developer

<CodeGroup>
```http POST /api/developers
POST /api/developers
Authorization: Bearer <token>
Content-Type: application/json

{
  "nameEn": "New Developer",
  "nameAr": "مطور جديد",
  "developerNumber": "DEV-123"
}
```

```json Response
{
  "id": 123,
  "nameEn": "New Developer",
  "nameAr": "مطور جديد",
  "developerNumber": "DEV-123",
  "sourceDeveloperId": "generated-uuid",
  "createdAt": "2023-12-01T10:00:00Z"
}
```
</CodeGroup>

### Update Developer

```http
PATCH /api/developers/:id
Authorization: Bearer <token>
Content-Type: application/json

{
  "nameEn": "Updated Developer Name",
  "logo": "https://example.com/new-logo.png"
}
```

### Delete Developer

```http
DELETE /api/developers/:id
Authorization: Bearer <token>
```

<Info>
Deletion is soft delete - sets `deleted_at` timestamp rather than removing the record.
</Info>

## Change Requests

The change request system provides audit trails and approval workflows for entity modifications.

### Submit Change Request

<Steps>
  <Step title="Create Change Request">
    Submit a change request for review and approval.

    ```http
    POST /api/change-requests
    Authorization: Bearer <token>
    Content-Type: application/json

    {
      "entityType": "developer",
      "entityId": 123,
      "operation": "update",
      "changes": {
        "nameEn": "New Company Name",
        "logo": "https://example.com/new-logo.png"
      },
      "reason": "Company rebranding"
    }
    ```
  </Step>

  <Step title="Review Process">
    The change request enters the approval workflow with status `pending`.
  </Step>

  <Step title="Apply Changes">
    Once approved, changes are automatically applied to the target entity.
  </Step>
</Steps>

### Change Request Status

| Status | Description |
|--------|-------------|
| `pending` | Awaiting review |
| `approved` | Approved and applied |
| `rejected` | Rejected by reviewer |
| `cancelled` | Cancelled by submitter |

### List Change Requests

```http
GET /api/change-requests?status=pending&entityType=developer
Authorization: Bearer <token>
```

##### Query Parameters

| Parameter | Type | Description |
|-----------|------|-------------|
| `status` | string | Filter by status |
| `entityType` | string | Filter by entity type |
| `entityId` | integer | Filter by specific entity |
| `submittedBy` | integer | Filter by submitter |

### Get Change Request

```http
GET /api/change-requests/:id
Authorization: Bearer <token>
```

```json
{
  "id": 456,
  "entityType": "developer",
  "entityId": 123,
  "operation": "update",
  "status": "pending",
  "changes": {
    "nameEn": "New Company Name",
    "logo": "https://example.com/new-logo.png"
  },
  "reason": "Company rebranding",
  "submittedBy": 789,
  "submittedAt": "2023-12-01T10:00:00Z",
  "reviewedBy": null,
  "reviewedAt": null
}
```

### Approve/Reject Change Request

<Tabs>
  <Tab title="Approve">
    ```http
    POST /api/change-requests/:id/approve
    Authorization: Bearer <token>
    Content-Type: application/json

    {
      "reviewComment": "Approved - valid rebranding request"
    }
    ```
  </Tab>
  <Tab title="Reject">
    ```http
    POST /api/change-requests/:id/reject
    Authorization: Bearer <token>
    Content-Type: application/json

    {
      "reviewComment": "Rejected - insufficient documentation"
    }
    ```
  </Tab>
</Tabs>

<Check>
When approved, changes are automatically applied to the target entity and the change request status is updated to `approved`.
</Check>

### Cancel Change Request

```http
POST /api/change-requests/:id/cancel
Authorization: Bearer <token>
Content-Type: application/json

{
  "reason": "No longer needed"
}
```

<Tip>
Only pending change requests can be cancelled. The submitter or an admin can cancel a request.
</Tip>
