The Dashboard module provides a **role-aware** dashboard at `/dashboard` where every authenticated organization member sees a personalized view, with each persona progressively seeing more data based on their permissions. The base controller uses **AuthGuard-only** protection, with only sensitive widgets carrying additional `@CheckAccess` gates.

<Note>
This design fixes the previous issue where users with zero permissions received a `403` error and blank dashboard. Now, BASIC users get a non-empty self-scoped dashboard.
</Note>

## Architecture

### Scope Resolution Model

The `DashboardScopeService.resolveScopes(userId, orgId)` returns per-domain resolved scopes computed from **team-membership-level** data via consolidated loading of user roles and permissions.

<Warning>
The scope service does NOT use the flattened `PermissionService.getUserEffectivePermissions` set, as that would collapse team permissions across all teams and lose team attribution context.
</Warning>

#### Domain Capabilities

Each domain has specific capability requirements defined by the `DomainScope` interface:

| Domain | Org Permission | Team Key | Self Floor | Hidden When |
|--------|---------------|----------|------------|-------------|
| CRM (leads) | `crm.view`/`crm.manage`/admin/owner | `team_crm.view` | Always | Never |
| Sales (deals) | `sales.view`/`sales.manage`/admin/owner | `team_sales.view` | Always | Never |
| Finance (money) | `finance.view`/`finance.manage`/admin/owner | — (no team) | Always | Never |
| Real Estate (listings) | `real_estate.view`/`real_estate.manage`/admin/owner | — | — | No real estate access |
| Analytics (team perf) | `analytics.view`/admin/owner | `team_analytics.view` OR team leader | — | No analytics access |

#### ResolvedScope Contract

```typescript
type DashboardScopeLevel = 'org' | 'team' | 'self';

interface ResolvedScope {
  level: DashboardScopeLevel;
  userId: string;
  orgId: string;
  teamIds: string[];
}
```

<Info>
The `ResolvedScope` interface is defined in `src/common/scope/resolved-scope.ts` to avoid circular dependencies between domain services and the dashboard module.
</Info>

### Scope Switching and Clamping

The UI provides a global My/Team/Org switcher plus a **team picker** that appears when `scope === 'team'`. The requested level is clamped per widget based on domain capabilities:

<Accordion title="Domain-Specific Clamping Rules">
- **CRM/Sales** (self floor): `org`→org else team else self; `team`→team else org else self; `my`→self
- **Finance** (no team): `my`→self; `team`/`org`→org if `finance.view` else **self**
- **Real Estate** (no team, no self): always org if `real_estate.view`, else hidden
- **Analytics** (no self): `org`→org else team else hidden; `team`/`my`→team else org else hidden
</Accordion>

### Team Attribution

Dashboard team scope counts leads/deals where `entity_stakeholder.team_id` matches eligible team IDs, aligned with CRM lead/deal list "Team" tab behavior. This includes unclaimed team-pool rows with `es.user_id IS NULL` but `es.team_id` set.

## API Endpoints

All dashboard endpoints are located at `/dashboard/*` and accept common query parameters:

<CodeGroup>
```typescript Query Parameters
interface DashboardQueryDto {
  scope?: 'my' | 'team' | 'org';
  teamId?: string;
  dateFrom?: string; // ISO date
  dateTo?: string;   // ISO date
}
```
</CodeGroup>

### Date Range Validation

The `DashboardQueryDto` validates date ranges with the following rules:

<Check>
- `dateTo >= dateFrom` (inclusive)
- Span ≤ `DASHBOARD_MAX_RANGE_DAYS` (366 days)
- Invalid ranges return **HTTP 400**
</Check>

## Multi-Domain Widgets

<Warning>
KPIs and Pipeline Summary widgets span multiple domains and apply each sub-metric's own domain `ResolvedScope`. The single requested level is clamped per domain individually.
</Warning>

### Monetary Rule

All monetary aggregates (Total Revenue, Revenue Forecast, pipeline VALUE totals) resolve their amounts via the **Finance** scope, even when deal counts resolve via the **Sales** scope. Users with `sales.view` but without `finance.view` see team/org deal counts but self-scoped money values.

## Activity Feed

The activity feed uses **performer-based** scoping:

<Steps>
<Step title="Self Scope">
Shows activities performed by the current user only
</Step>
<Step title="Team Scope">
Shows activities by the user plus all members of eligible CRM/Sales teams
</Step>
<Step title="Org Scope">
Shows all organization activities (no performer filter)
</Step>
</Steps>

<Info>
The feed merges audit lifecycle events (lead created, stage changed, etc.) with logged activities (notes, calls, emails, meetings) using `UnifiedTimelineService` with cursor pagination.
</Info>

## Configuration Constants

Widget keys, scope enums, and timing constants are defined in `dashboard.constants.ts`:

- **Default range:** 30 days
- **Maximum range:** 366 days (one calendar year including leap day)
- **Lead volume default:** 14 days
- **Stalled lead threshold:** Configurable via `DASHBOARD_STALLED_LEAD_DAYS`

<Tip>
The dashboard module location is `src/modules/crm/dashboard/` and follows standard NestJS module patterns with proper dependency injection and transaction management.
</Tip>
