## Purpose

The Dashboard Module provides a **role-aware**, personalized dashboard at `/dashboard` for every authenticated organization member. Each user sees a progressively scoped view based on their permissionsŌĆöfrom self-only data to team aggregates to full organizational visibility.

<Note>
The base dashboard controller uses **AuthGuard only**. Only the two sensitive widgets (`team-performance` and `listings`) carry `@CheckAccess` gates for specific permission validation.
</Note>

### Problem Solved

The legacy `/dashboard` endpoint called `GET /analytics/permissions` with a hard `@CheckAccess(ANALYTICS_VIEW)` gate, causing **403 Forbidden** responses for users without analytics permissions. This resulted in blank dashboards for basic users.

**Solution:** The dashboard module is un-gated at the base level, so users with minimal permissions now receive a **non-empty, self-scoped dashboard** showing their own data.

## Scope Resolution Model

The `DashboardScopeService.resolveScopes(userId, orgId)` method computes per-domain resolved scopes from **team-membership-level** data through a single consolidated load:

```
UserOrgRole ŌåÆ role ŌåÆ permissions
TeamMembership ŌåÆ teamRoles ŌåÆ role ŌåÆ permissions
```

<Warning>
**Critical Gap #1:** The scope resolver does NOT use the flattened `PermissionService.getUserEffectivePermissions` set. That method collapses all team permissions across teams and loses attribution of which team granted `team_crm.view`. 

A user who leads Team A (with `team_crm.view`) and is a plain member of Team B (no team permissions) resolves `crm.defaultTeamIds === [A]` only.
</Warning>

### Domain Scope Structure

Each domain returns a `DomainScope` with:

- **`hasOrg`**: Organization-level permission present (or `system.admin`/`system.owner`, which grant org-level access on every domain)
- **`defaultTeamIds`**: Membership teams whose roles grant the domain's team permission key
- **`availableTeams`**: Dropdown/validation set = membership teams (+ all org teams when user has org capability)

## Scope Switcher & Clamping

### Global Controls

The UI exposes:
- **One global switcher:** My / Team / Org
- **Team picker:** Appears only when `scope === 'team'`

The team picker lists the union of `availableTeams` across CRM, Sales, and Analytics (deduplicated by ID). When a user has exactly one eligible team, the UI shows a **read-only team label** instead of a dropdown.

### Server-Authoritative Clamping

The `clampDomain` function ensures requested scope levels are valid for each domain:

<AccordionGroup>
  <Accordion title="CRM & Sales (Self Floor)">
    - `org` ŌåÆ org else team else self
    - `team` ŌåÆ team else org else self
    - `my` ŌåÆ self
  </Accordion>
  
  <Accordion title="Finance (No Team)">
    - `my` ŌåÆ self
    - `team` or `org` ŌåÆ org if `finance.view` else **self** (Gap A)
  </Accordion>
  
  <Accordion title="Real Estate (No Team, No Self)">
    - Always org if `real_estate.view`, else hidden
  </Accordion>
  
  <Accordion title="Analytics (No Self)">
    - `org` ŌåÆ org else team else hidden
    - `team` or `my` ŌåÆ team else org else hidden
  </Accordion>
</AccordionGroup>

### Team ID Validation

A passed `teamId` query parameter is:
1. Re-validated against the domain's `availableTeams`
2. Narrows `teamIds` to `[teamId]` when valid
3. **Ignored** when out-of-domain (never trusted), falling back to `defaultTeamIds`

<Tip>
When `team` scope resolves with no specific `teamId`, it aggregates across **ALL eligible teams** via `es.team_id IN (?)` (Knex expands the bound ID array). There is no "must select a team" dead-end.
</Tip>

### Team Attribution Alignment

Dashboard `team` scope counts leads/deals where `entity_stakeholder.team_id` matches eligible team ID(s)ŌĆöthe same rule as `LeadService.applyTeamLeadsFilter`.

**This includes:**
- Ō£ģ Unclaimed team-pool rows (`es.user_id IS NULL` with `es.team_id` set)

**This does NOT:**
- ŌØī Join `team_membership` on the assignee's user ID (old path that silently dropped pool leads)

<Check>
Team-scoped dashboard counts now **align with the CRM lead/deal list "Team" tab** counts.
</Check>

### Monetary Rule (Gap A)

<Warning>
**ALL monetary aggregates** (Total Revenue, Revenue Forecast, pipeline VALUE totals) resolve their amounts via the **Finance scope**, even though the deal SET/counts resolve via the **Sales scope**.

A `sales.view`-without-`finance.view` user sees team/org deal **counts** but **self-scoped money**.
</Warning>

### Frontend Implementation

**Files:** `dashboard-view.tsx` + `dashboard-controls.tsx`

- `mergeGlobalAvailableTeams()`: Builds the global picker list
- `resolveTeamScopeSelection()`: Maintains valid default when switching to Team scope
- `buildDashboardParams()`: Includes `teamId` only while Team scope is active

## API Endpoints

All endpoints accept query parameters: `?scope&teamId&dateFrom&dateTo`

### Date Range Validation

`DashboardQueryDto` validates date ranges with these rules:

<Steps>
  <Step title="Temporal Order">
    `dateTo >= dateFrom` (inclusive)
  </Step>
  
  <Step title="Maximum Span">
    Span Ōēż `DASHBOARD_MAX_RANGE_DAYS` (366 days ŌĆö one calendar year, including leap day)
  </Step>
  
  <Step title="Error Response">
    Invalid ranges return **HTTP 400** with:
    ```
    dateTo must be on or after dateFrom and within 366 days
    ```
  </Step>
</Steps>

### Frontend Normalization

Before widgets call `/dashboard/*`, the CRM app normalizes user picks in `dashboard.utils.ts` ŌåÆ `rangeToDates()`:

1. Order inverted endpoints (swap when `from` is after `to`)
2. Cap `dateTo` at `now` (future end dates are rejected server-side)
3. If window is still inverted after capping, collapse `dateFrom` to `startOfDay(dateTo)`
4. Clamp `dateFrom` to the most recent `DASHBOARD_MAX_RANGE_DAYS` window when span would exceed one year

<Note>
Custom date pickers disable future calendar days (`maxDate={today}`). Empty windows (valid range, zero rows) return **200** with empty seriesŌĆöwidgets show the empty state, not the error shell.
</Note>

## Activity Feed

### Performer-Based Scope (Gap C)

The activity feed uses **performer-based scoping**:

- **`self`:** `[userId]`
- **`team`:** `[userId] Ōł¬ members of eligible CRM/Sales teams`
- **`org`:** `null` (no performer filter)

Member IDs resolved via `TeamMembershipService.getMemberUserIdsInTransaction`.

### Cursor Pagination

Uses base64url-encoded `{ createdAt, id }` keyset for infinite scroll. No "view full log" link.

### Unified Timeline Source

<Info>
The feed is backed by `UnifiedTimelineService.getActivityFeedInTransaction(em, orgId, { actorUserIds, cursor, limit })`, **not** raw `activity` rows.
</Info>

It merges:
- **Audit lifecycle events:** Lead created, stage changed, assigned, transfer, deal won/lost (curated audit tables resolved into human messages by `AuditTimelineResolver`)
- **Logged activities:** Note, call, email, meeting

All attributed to resolved performers (`null` = org).

### Response Structure

```typescript

  source: 'audit' | 'activity';
  activityType: TimelineEventType | ActivityType;
  summary: string;  // pre-resolved message
  accentColor?: string;
  targetEntityType?: string;
  targetEntityId?: string;
  // ... timestamp, actor, etc.
}
```

The frontend widget keys icon/tint off `activityType`.

## Team Performance

### Agent Clamping (Gap #5)

- **`team` level:** Agents clamped to `team_membership` of eligible `teamIds`
- **`org` level:** All org team members

<Note>
A team leader never sees every org agent.
</Note>

### Per-Agent Metrics

Each agent row includes:
- Active leads
- Deals (total / won / open)
- Win rate
- **Stalled leads:** Active leads with no activity in the last `DASHBOARD_STALLED_LEAD_DAYS` days
