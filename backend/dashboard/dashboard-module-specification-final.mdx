## Purpose

A **role-aware** dashboard at `/dashboard` where every authenticated org member sees a personal view, with each persona progressively seeing more data based on their permissions. The base controller uses **AuthGuard-only** protection — only the two sensitive widgets (`team-performance` and `listings`) carry a `@CheckAccess` gate.

<Note>
**Last Updated:** 2026-06-05  
**Status:** ACTIVE ✅  
**Location:** `src/modules/crm/dashboard/`
</Note>

### The Problem This Fixes

The old `/dashboard` called `GET /analytics/permissions`, gated by `@CheckAccess(ANALYTICS_VIEW)`, so zero-permission users got a `403` and a blank dashboard. The dashboard module is un-gated at the base level, so a BASIC user now gets a non-empty self-scoped dashboard.

## Scope Model

`DashboardScopeService.resolveScopes(userId, orgId)` returns a per-domain resolved scope computed from **team-membership-level** data via ONE consolidated load of:
- `UserOrgRole → role → permissions` 
- `TeamMembership → teamRoles → role → permissions`

<Warning>
**CRITICAL Gap #1:** This does NOT use the flattened `PermissionService.getUserEffectivePermissions` set — that collapses all team permissions across all teams and loses which team granted `team_crm.view`. A user who leads Team A (`team_crm.view`) and is a plain member of Team B (no team perms) resolves `crm.defaultTeamIds === [A]` only.
</Warning>

### Domain Scope Capabilities

Per-domain capability (`DomainScope`):

- **`hasOrg`**: org-permission present (or `system.admin` / `system.owner`, which grant org on **every** domain)
- **`defaultTeamIds`**: the membership teams whose roles grant that domain's team key — the default `team`-level aggregate
- **`availableTeams`**: the dropdown/validation set = membership teams (+ all org teams when the user is org-capable, so org users can drill into any team)

| Domain | Org When | Team Key | Self Floor | Hidden When |
|--------|----------|----------|------------|-------------|
| CRM (leads) | `crm.view`/`crm.manage`/admin/owner | `team_crm.view` | always | never |
| Sales (deals) | `sales.view`/`sales.manage`/admin/owner | `team_sales.view` | always | never |
| Finance (money) | `finance.view`/`finance.manage`/admin/owner | — (no team) | always | never |
| Real Estate (listings) | `real_estate.view`/`real_estate.manage`/admin/owner | — | — | no real_estate access |
| Analytics (team perf) | `analytics.view`/admin/owner | `team_analytics.view` OR team leader | — | no analytics access |

### ResolvedScope Contract

```ts
type DashboardScopeLevel = 'org' | 'team' | 'self';

interface ResolvedScope { 
  level: DashboardScopeLevel; 
  userId: string; 
  orgId: string; 
  teamIds: string[]; 
}
```

<Info>
Defined in `src/common/scope/resolved-scope.ts` (neutral location so domain services can depend on it without a cycle back into the dashboard module). Passed to **every** composable.
</Info>

### Switcher and Clamping

The UI exposes one global My/Team/Org switcher plus a **team picker** that appears only when `scope === 'team'`. The picker lists the union of `availableTeams` across CRM, Sales, and Analytics (deduped by id).

<Steps>
<Step title="Team Selection">
When the user has exactly one eligible team, the UI shows a read-only team label instead of a dropdown.
</Step>
<Step title="Level Clamping">
The requested level is clamped per widget to its domain max:
- **CRM/Sales** (self floor): `org`→org else team else self; `team`→team else org else self; `my`→self
- **Finance** (no team): `my`→self; `team`/`org`→org if `finance.view` else **self** (gap A)
- **Real Estate** (no team, no self): always org if `real_estate.view`, else hidden
- **Analytics** (no self): `org`→org else team else hidden; `team`/`my`→team else org else hidden
</Step>
<Step title="TeamId Validation">
A passed `teamId` is re-validated against the domain's `availableTeams` and narrows `teamIds` to `[teamId]`; an out-of-domain `teamId` is **ignored** (never trusted), falling back to `defaultTeamIds`.
</Step>
</Steps>

### Team Attribution

Dashboard `team` scope counts leads/deals where `entity_stakeholder.team_id` matches the eligible team id(s) — the same rule as `LeadService.applyTeamLeadsFilter`. This **includes unclaimed team-pool rows** (`es.user_id IS NULL` with `es.team_id` set).

<Warning>
**Monetary Rule (Gap A):** ALL monetary aggregates (Total Revenue, Revenue Forecast, pipeline VALUE totals) resolve their amount via the **Finance** scope, even though the deal SET/counts resolve via the **Sales** scope. A `sales.view`-without-`finance.view` user therefore sees team/org deal **counts** but **self-scoped money**.
</Warning>

## API Endpoints

All `/dashboard/*` endpoints accept `?scope&teamId&dateFrom&dateTo` query parameters.

### Date Range Validation

`DashboardQueryDto` validates `dateTo` when both bounds are present:

- `dateTo >= dateFrom` (inclusive)
- Span ≤ `DASHBOARD_MAX_RANGE_DAYS` (366 days — one calendar year, incl. leap day)

Invalid ranges return **HTTP 400** with `dateTo must be on or after dateFrom and within 366 days`.

<Tip>
**Frontend Normalization:** Before widgets call `/dashboard/*`, the CRM app normalizes user picks in `dashboard.utils.ts` → `rangeToDates()` by ordering inverted endpoints, capping `dateTo` at `now`, and clamping `dateFrom` to the most recent 366-day window.
</Tip>

### Activity Feed Details

**Performer-based scoping:**
- `self` = `[userId]`
- `team` = `[userId] ∪ members of eligible CRM/Sales teams`  
- `org` = `null` (no performer filter)

The feed uses `UnifiedTimelineService` which merges:
- **Audit lifecycle events** (lead created, stage changed, assigned, deal won/lost, etc.)
- **Logged activities** (note/call/email/meeting, etc.)

<Note>
The `DashboardActivityItemDto` includes `source` (`'audit' | 'activity'`), `activityType`, `summary` (pre-resolved message), optional `accentColor`, and nullable `targetEntityType`/`targetEntityId`.
</Note>

## Configuration Constants

Defined in `dashboard.constants.ts`:

- Widget keys and scope-request enums
- Default date range: 30 days
- Maximum date range: 366 days (one calendar year including leap day)
- Lead volume default: 14 days
- Stalled lead threshold: `DASHBOARD_STALLED_LEAD_DAYS`

<Check>
All constants are centralized in the dashboard module to ensure consistent behavior across all widgets and prevent magic numbers in the codebase.
</Check>
