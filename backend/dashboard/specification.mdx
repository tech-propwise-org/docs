## Purpose

A **role-aware** dashboard at `/dashboard` where every authenticated org member sees a personal view with each persona progressively accessing more data. The base controller uses **AuthGuard-only** protection, with only two sensitive widgets requiring `@CheckAccess` gates.

<Note>
**Bug Fix**: The old `/dashboard` called `GET /analytics/permissions` gated by `@CheckAccess(ANALYTICS_VIEW)`, causing zero-permission users to receive a `403` and blank dashboard. The new dashboard module is un-gated at the base level, ensuring BASIC users receive a non-empty self-scoped dashboard.
</Note>

## Scope Model

The scope system resolves server-side using a "broadest wins" approach. `DashboardScopeService.resolveScopes(userId, orgId)` returns per-domain resolved scopes computed from **team-membership-level** data.

<Warning>
**Critical Gap #1**: The system does NOT use the flattened `PermissionService.getUserEffectivePermissions` set, which would collapse all team permissions and lose team attribution. A user who leads Team A (`team_crm.view`) and is a plain member of Team B resolves `crm.defaultTeamIds === [A]` only.
</Warning>

### Domain Capabilities

Each domain has a `DomainScope` with the following properties:

- `hasOrg`: org-permission present (or `system.admin`/`system.owner`)
- `defaultTeamIds`: membership teams whose roles grant the domain's team key
- `availableTeams`: dropdown/validation set including membership teams

<Tabs>
<Tab title="Domain Configuration">
| Domain | Org Access | Team Key | Self Floor | Hidden When |
|--------|------------|----------|------------|-------------|
| CRM (leads) | `crm.view`/`crm.manage`/admin/owner | `team_crm.view` | always | never |
| Sales (deals) | `sales.view`/`sales.manage`/admin/owner | `team_sales.view` | always | never |
| Finance (money) | `finance.view`/`finance.manage`/admin/owner | — (no team) | always | never |
| Real Estate (listings) | `real_estate.view`/`real_estate.manage`/admin/owner | — | — | no real_estate access |
| Analytics (team perf) | `analytics.view`/admin/owner | `team_analytics.view` OR team leader | — | no analytics access |
</Tab>

```ts
interface ResolvedScope { 
  level: DashboardScopeLevel; 
  userId: string; 
  orgId: string; 
  teamIds: string[]; 
}
```
</Tab>

### Scope Switcher and Clamping

The UI provides a global My/Team/Org switcher plus a **team picker** that appears only when `scope === 'team'`. The requested level is clamped per widget to its domain maximum:

<AccordionGroup>
<Accordion title="CRM/Sales (Self Floor)">
- `org` → org else team else self
- `team` → team else org else self  
- `my` → self
</Accordion>
<Accordion title="Finance (No Team)">
- `my` → self
- `team`/`org` → org if `finance.view` else **self** (Gap A)
</Accordion>
<Accordion title="Real Estate (No Team, No Self)">
- Always org if `real_estate.view`, else hidden
</Accordion>
<Accordion title="Analytics (No Self)">
- `org` → org else team else hidden
- `team`/`my` → team else org else hidden
</Accordion>
</AccordionGroup>

### Team Attribution

<Info>
**Team Scope Alignment**: Dashboard `team` scope counts leads/deals where `entity_stakeholder.team_id` matches eligible team IDs, following the same rule as `LeadService.applyTeamLeadsFilter`. This includes unclaimed team-pool rows (`es.user_id IS NULL` with `es.team_id` set).
</Info>

### Monetary Rule (Gap A)

ALL monetary aggregates (Total Revenue, Revenue Forecast, pipeline VALUE totals) resolve their amount via the **Finance** scope, even though deal counts resolve via the **Sales** scope. Users with `sales.view` but without `finance.view` see team/org deal **counts** but **self-scoped money**.

## API Endpoints

All endpoints accept query parameters: `?scope&teamId&dateFrom&dateTo`

### Date Range Validation

`DashboardQueryDto` validates date ranges with the following rules:

<Steps>
<Step title="Date Order">
`dateTo >= dateFrom` (inclusive)
</Step>
<Step title="Maximum Span">
Span ≤ `DASHBOARD_MAX_RANGE_DAYS` (366 days)
</Step>
<Step title="Error Response">
Invalid ranges return **HTTP 400** with: `dateTo must be on or after dateFrom and within 366 days`
</Step>
</Steps>

## Multi-Domain Widgets

<Warning>
**Gap B**: KPIs and Pipeline Summary widgets apply each sub-metric's own domain `ResolvedScope`. The single requested level is clamped per domain, meaning different metrics within the same widget may operate at different scope levels.
</Warning>

### Activity Feed Implementation

<Tip>
**Performer-Based Scoping**: 
- `self` = `[userId]`
- `team` = `[userId] ∪ members of eligible CRM/Sales teams`  
- `org` = `null` (no performer filter)
</Tip>

The activity feed uses **unified timeline** merging:
- **Audit lifecycle events** (lead created, stage changed, assigned, etc.)
- **Logged activities** (notes, calls, emails, meetings)

Cursor pagination uses base64url `{ createdAt, id }` keyset for infinite scroll.

### Tasks vs Events

<Info>
**Gap G**: Self-only scope by design. "Events" are tasks with `taskType IN (MEETING, SITE_VISIT)` AND `hasDueTime = true`. "Tasks" are everything else.
</Info>
