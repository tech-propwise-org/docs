## Purpose

A **role-aware** dashboard at `/dashboard` where every authenticated org member sees a personal view, with each persona progressively seeing more content based on their permissions. The base controller is **AuthGuard-only** — only the two sensitive widgets carry a `@CheckAccess` gate.

### The Bug This Fixes

<Warning>
The old `/dashboard` called `GET /analytics/permissions`, gated by `@CheckAccess(ANALYTICS_VIEW)`, so zero-permission users got a `403` and a blank dashboard. The dashboard module is un-gated at the base, so a BASIC user now gets a non-empty self-scoped dashboard.
</Warning>

## Scope Model

The scope model is resolved server-side with a "broadest wins" strategy.

### Scope Resolution

`DashboardScopeService.resolveScopes(userId, orgId)` returns a per-domain resolved scope computed from **team-membership-level** data via ONE consolidated load of:
- `UserOrgRole → role → permissions`
- `TeamMembership → teamRoles → role → permissions`

<Warning>
**CRITICAL (gap #1):** The scope resolution does NOT use the flattened `PermissionService.getUserEffectivePermissions` set — that collapses all team permissions across all teams and loses which team granted `team_crm.view`. A user who leads Team A (`team_crm.view`) and is a plain member of Team B (no team perms) resolves `crm.defaultTeamIds === [A]` only.
</Warning>

### Domain Scope Contract

Per-domain capability (`DomainScope`):

- **`hasOrg`**: org-permission present (or `system.admin` / `system.owner`, which grant org on **every** domain)
- **`defaultTeamIds`**: the membership teams whose roles grant that domain's team key — the default `team`-level aggregate
- **`availableTeams`**: the dropdown/validation set = membership teams (+ all org teams when the user is org-capable, so org users can drill into any team)

| Domain | Org When | Team Key | Self Floor | Hidden When |
|--------|----------|----------|------------|-------------|
| CRM (leads) | `crm.view`/`crm.manage`/admin/owner | `team_crm.view` | always | never |
| Sales (deals) | `sales.view`/`sales.manage`/admin/owner | `team_sales.view` | always | never |
| Finance (money) | `finance.view`/`finance.manage`/admin/owner | — (no team) | always | never |
| Real Estate (listings) | `real_estate.view`/`real_estate.manage`/admin/owner | — | — | no real_estate access |
| Analytics (team perf) | `analytics.view`/admin/owner | `team_analytics.view` OR team leader | — | no analytics access |

### ResolvedScope Interface

```typescript
type DashboardScopeLevel = 'org' | 'team' | 'self';

interface ResolvedScope {
  level: DashboardScopeLevel;
  userId: string;
  orgId: string;
  teamIds: string[];
}
```

<Info>
Defined in `src/common/scope/resolved-scope.ts` (neutral location so domain services can depend on it without a cycle back into the dashboard module). Passed to **every** composable.
</Info>

### Scope Switcher and Clamping

The UI exposes one global My/Team/Org switcher plus a **team picker** that appears only when `scope === 'team'`.

<Steps>
  <Step title="Team Picker Display">
    The picker lists the union of `availableTeams` across CRM, Sales, and Analytics (deduped by id). When the user has exactly one eligible team, the UI shows a read-only team label instead of a dropdown.
  </Step>
  
  <Step title="Scope Clamping">
    The requested level is clamped per widget to its domain max using the `clampDomain` function (server-authoritative):
    
    - **CRM/Sales** (self floor): `org`→org else team else self; `team`→team else org else self; `my`→self
    - **Finance** (no team): `my`→self; `team`/`org`→org if `finance.view` else **self** (gap A)
    - **Real Estate** (no team, no self): always org if `real_estate.view`, else hidden
    - **Analytics** (no self): `org`→org else team else hidden; `team`/`my`→team else org else hidden
  </Step>
  
  <Step title="Team ID Validation">
    A passed `teamId` is re-validated against the domain's `availableTeams` and narrows `teamIds` to `[teamId]`. An out-of-domain `teamId` is **ignored** (never trusted), falling back to `defaultTeamIds`.
  </Step>
</Steps>

<Tip>
When `team` resolves with no `teamId`, it aggregates across ALL eligible teams via `es.team_id IN (?)` (Knex expands the bound id array) — there is NO "must select a team" dead-end.
</Tip>

### Team Attribution

<Info>
Dashboard `team` scope counts leads/deals where `entity_stakeholder.team_id` matches the eligible team id(s) — the same rule as `LeadService.applyTeamLeadsFilter`. This **includes unclaimed team-pool rows** (`es.user_id IS NULL` with `es.team_id` set).
</Info>

It does **not** join `team_membership` on the assignee's user id (the old path silently dropped pool leads and produced counts that disagreed with the lead list).

**Frontend Implementation:**
- `dashboard-view.tsx` + `dashboard-controls.tsx`
- `mergeGlobalAvailableTeams()` builds the global picker list
- `resolveTeamScopeSelection()` keeps a valid default when the user switches to Team scope
- `buildDashboardParams()` includes `teamId` only while Team scope is active

### Monetary Rule (Gap A)

<Warning>
ALL monetary aggregates (Total Revenue, Revenue Forecast, pipeline VALUE totals) resolve their amount via the **Finance** scope, even though the deal SET/counts resolve via the **Sales** scope. A `sales.view`-without-`finance.view` user therefore sees team/org deal **counts** but **self-scoped money**.
</Warning>

## API Endpoints

All endpoints under `/dashboard/*` accept query parameters: `?scope&teamId&dateFrom&dateTo`

### Date Range Validation

`DashboardQueryDto` validates `dateTo` when both bounds are present:

<Check>
- `dateTo >= dateFrom` (inclusive)
- Span ≤ `DASHBOARD_MAX_RANGE_DAYS` (366 days — one calendar year, incl. leap day)
</Check>

Invalid ranges return **HTTP 400** with:
```
dateTo must be on or after dateFrom and within 366 days
```

### Frontend Date Normalization

<Steps>
  <Step title="Order Correction">
    Order inverted endpoints (swap when `from` is after `to`)
  </Step>
  
  <Step title="Future Date Capping">
    Cap `dateTo` at `now` (future end dates are rejected server-side)
  </Step>
  
  <Step title="Window Validation">
    If the window is still inverted after capping, collapse `dateFrom` to `startOfDay(dateTo)`
  </Step>
  
  <Step title="Range Clamping">
    Clamp `dateFrom` to the most recent `DASHBOARD_MAX_RANGE_DAYS` window when the preset/custom span would exceed one calendar year
  </Step>
</Steps>

<Info>
`This Year` preset is **not** silently shortened to 90 days. Custom date pickers also disable future calendar days (`maxDate={today}`). Empty windows (valid range, zero rows) return **200** with empty series — widgets show the empty state, not the error shell.
</Info>

| Endpoint | Guard | Scope(s) | Source |
|----------|-------|----------|--------|

| `GET /kpis` | AuthGuard | CRM + Sales + Finance | `AnalyticsService.getAnalyticsForScopeInTransaction` ×2 + `DealService.sumWonDealValueInTransaction` |
| `GET /revenue-forecast` | AuthGuard | Finance | `DealService.getRevenueForecastInTransaction` |
| `GET /pipeline-summary` | AuthGuard | CRM + Sales + Finance | `LeadService.getLeadPipelineSummaryInTransaction` + `DealService.getDealStageCounts/Values` |
| `GET /lead-volume` | AuthGuard | CRM | `LeadService.getLeadVolumeSeriesInTransaction` |
| `GET /lead-source` | AuthGuard | CRM | `AnalyticsService.getAnalyticsForScopeInTransaction` (leads) |
| `GET /activity-feed` | AuthGuard | CRM/Sales (performer-based) | `UnifiedTimelineService.getActivityFeedInTransaction` (cursor) |
| `GET /tasks` | AuthGuard | self only | `TaskService.getUpcomingTasksInTransaction` |

### Multi-Domain Widgets (Gap B)

<Info>
KPIs (Total Leads = CRM; Win-rate = Sales; Revenue = Sales set + Finance money) and Pipeline Summary (lead pipeline = CRM; deal pipeline = Sales; value = Finance) apply EACH sub-metric's own domain `ResolvedScope` — the single requested level is clamped per domain.
</Info>

### Activity Feed Details (Gap C)

**Performer-based scope resolution:**
- `self` = `[userId]`
- `team` = `[userId]` ∪ members of eligible CRM/Sales teams
- `org` = `null` (no performer filter)

Member ids resolved via `TeamMembershipService.getMemberUserIdsInTransaction`.

<Tip>
Cursor pagination uses a base64url `{ createdAt, id }` keyset — infinite scroll, no "view full log" link.
</Tip>

#### Unified Timeline Source

The feed is backed by `UnifiedTimelineService.getActivityFeedInTransaction(em, orgId, { actorUserIds, cursor, limit })`, **not** raw `activity` rows.

<Info>
It merges **audit lifecycle events** (lead created, stage changed, assigned, transfer, deal won/lost, …; curated audit tables resolved into human messages by `AuditTimelineResolver`) with **logged activities** (note/call/email/meeting/…), all attributed to the resolved performers (`null` = org).
</Info>

The `DashboardActivityItemDto` carries:
- `source` (`'audit' | 'activity'`)
- `activityType` = the row `kind` (`TimelineEventType` for audit, `ActivityType` for activity)
- `summary` (pre-resolved message)
- optional `accentColor`
- nullable `targetEntityType` / `targetEntityId`

The frontend widget keys its icon/tint off `activityType`. See `Docs/TIMELINE_MODULE_SPECIFICATION.md` → *Org-Wide Actor Activity Feed*.

### Pipeline Summary Details

<Note>
Terminal **DONE** lead stage is split into **Won** / **Lost** virtual buckets by `lead.isWon`, keyed `${stageId}:won`/`:lost`, green/red colors.
</Note>

### Tasks vs Events (Gap G)

Self-only scope (decision F):
- **Events** = tasks with `taskType IN (MEETING, SITE_VISIT)` AND `hasDueTime = true`
- **Tasks** = the rest

Both come from `TaskService`.

### Team Performance Agent Clamp (Gap #5)

<Info>
- `team` level → agents clamped to `team_membership` of the eligible `teamIds`
- `org` level → all org team members
- A team leader never sees every org agent
</Info>

Per-agent rows include:
- Active leads
- Deals (total/won/open)
- Win rate
- **Stalled** leads (active leads with no activity in the last `DASHBOARD_STALLED_LEAD_DAYS` days)

## Implementation Notes

<Tip>
The dashboard module demonstrates progressive disclosure: all authenticated users get a functional dashboard at their permission level, avoiding the previous "403 blank screen" experience for basic users.
</Tip>

<Note>
Server-side scope resolution ensures security — client requests are always validated and clamped to the user's actual permissions, with no trust placed in client-provided scope/team parameters.
</Note>
