## Purpose

A **role-aware** dashboard at `/dashboard` where every authenticated org member sees a personalized view. Each persona progressively sees more data based on their permissions. The base controller uses **AuthGuard-only** protection, with only the two sensitive widgets carrying a `@CheckAccess` gate.

<Warning>
**The bug this fixes:** The old `/dashboard` called `GET /analytics/permissions`, gated by `@CheckAccess(ANALYTICS_VIEW)`, so zero-permission users got a `403` and a blank dashboard. The new dashboard module is un-gated at the base, so a BASIC user now gets a non-empty self-scoped dashboard.
</Warning>

## Scope Model

**Last Updated:** 2026-06-05  
**Status:** ACTIVE ✅

</Info>

### Scope Resolution

`DashboardScopeService.resolveScopes(userId, orgId)` returns a per-domain resolved scope computed from **team-membership-level** data via ONE consolidated load of:
- `UserOrgRole → role → permissions`
- `TeamMembership → teamRoles → role → permissions`

<Warning>
**CRITICAL (gap #1):** This does NOT use the flattened `PermissionService.getUserEffectivePermissions` set — that collapses all team permissions across all teams and loses which team granted `team_crm.view`. A user who leads Team A (`team_crm.view`) and is a plain member of Team B (no team perms) resolves `crm.defaultTeamIds === [A]` only.
</Warning>

### Domain Capabilities (DomainScope)

Each domain has specific capability requirements:

- `hasOrg`: org-permission present (or `system.admin` / `system.owner`, which grant org on **every** domain)
- `defaultTeamIds`: the membership teams whose roles grant that domain's team key — the default `team`-level aggregate
- `availableTeams`: the dropdown/validation set = membership teams (+ all org teams when the user is org-capable)

| Domain | Org When | Team Key | Self Floor | Hidden When |
| --- | --- | --- | --- | --- |
| CRM (leads) | `crm.view`/`crm.manage`/admin/owner | `team_crm.view` | always | never |
| Sales (deals) | `sales.view`/`sales.manage`/admin/owner | `team_sales.view` | always | never |
| Finance (money) | `finance.view`/`finance.manage`/admin/owner | — (no team) | always | never |
| Real Estate (listings) | `real_estate.view`/`real_estate.manage`/admin/owner | — | — | no real_estate access |
| Analytics (team perf) | `analytics.view`/admin/owner | `team_analytics.view` OR team leader | — | no analytics access |

### ResolvedScope Contract

<CodeGroup>
```typescript ResolvedScope Interface
type DashboardScopeLevel = 'org' | 'team' | 'self';

interface ResolvedScope { 
  level: DashboardScopeLevel; 
  userId: string; 
  orgId: string; 
  teamIds: string[]; 
}
```
</CodeGroup>

<Note>
Defined in `src/common/scope/resolved-scope.ts` (neutral location so domain services can depend on it without a cycle back into the dashboard module). Passed to **every** composable.
</Note>

### Scope Switcher & Clamping

The UI exposes one global My/Team/Org switcher plus a **team picker** that appears only when `scope === 'team'`. The requested level is clamped per widget to its domain max:

<Tabs>
  <Tab title="CRM/Sales (self floor)">
    - `org` → org else team else self
    - `team` → team else org else self  
    - `my` → self
  </Tab>
  <Tab title="Finance (no team)">
    - `my` → self
    - `team`/`org` → org if `finance.view` else **self** (gap A)
  </Tab>
  <Tab title="Real Estate (no team, no self)">
    - Always org if `real_estate.view`, else hidden
  </Tab>
  <Tab title="Analytics (no self)">
    - `org` → org else team else hidden
    - `team`/`my` → team else org else hidden
  </Tab>
</Tabs>

### Team Attribution

<Check>
Dashboard `team` scope counts leads/deals where `entity_stakeholder.team_id` matches the eligible team id(s) — the same rule as `LeadService.applyTeamLeadsFilter`. This **includes unclaimed team-pool rows** (`es.user_id IS NULL` with `es.team_id` set).
</Check>

### Monetary Rule (Gap A)

<Warning>
ALL monetary aggregates (Total Revenue, Revenue Forecast, pipeline VALUE totals) resolve their amount via the **Finance** scope, even though the deal SET/counts resolve via the **Sales** scope. A `sales.view`-without-`finance.view` user therefore sees team/org deal **counts** but **self-scoped money**.
</Warning>

## API Endpoints

All dashboard endpoints follow the pattern `/dashboard/*` and accept these query parameters:

<CodeGroup>
```typescript Query Parameters
interface DashboardQueryDto {
  scope?: 'my' | 'team' | 'org';
  teamId?: string;
  dateFrom?: string; // ISO date
  dateTo?: string;   // ISO date
}
```
</CodeGroup>

### Date Range Validation

`DashboardQueryDto` validates `dateTo` when both bounds are present:

<Steps>
<Step title="Date Order Check">
`dateTo >= dateFrom` (inclusive)
</Step>
<Step title="Span Validation">
Span ≤ `DASHBOARD_MAX_RANGE_DAYS` (366 days — one calendar year, including leap day)
</Step>
<Step title="Error Response">
Invalid ranges return **HTTP 400** with message: `dateTo must be on or after dateFrom and within 366 days`
</Step>
</Steps>

## Multi-Domain Widgets

<Tip>
**Gap B:** KPIs and Pipeline Summary widgets apply EACH sub-metric's own domain `ResolvedScope` — the single requested level is clamped per domain.

Example: Total Leads uses CRM scope, Win-rate uses Sales scope, Revenue uses Sales set + Finance money scope.
</Tip>

## Activity Feed Details

<Note>
**Gap C:** Performer-based scoping where:
- `self` = `[userId]`
- `team` = `[userId] ∪ members of eligible CRM/Sales teams`  
- `org` = `null` (no performer filter)
</Note>

The feed uses **unified timeline** merging:
- **Audit lifecycle events** (lead created, stage changed, assigned, etc.)
- **Logged activities** (note/call/email/meeting)

<CodeGroup>
```typescript Activity Item Structure
interface DashboardActivityItemDto {
  source: 'audit' | 'activity';
  activityType: TimelineEventType | ActivityType;
  summary: string; // pre-resolved message
  accentColor?: string;
  targetEntityType?: string;
  targetEntityId?: string;
}
```
</CodeGroup>

## Tasks vs Events

<Info>
**Gap G:** Self-only scoping (decision F). 

"Events" = tasks with `taskType IN (MEETING, SITE_VISIT)` AND `hasDueTime = true`  
"Tasks" = everything else

Both come from `TaskService`.
</Info>

## Team Performance

<Warning>
**Gap #5:** Agent clamping rules:
- `team` level → agents clamped to `team_membership` of the eligible `teamIds`
- `org` level → all org team members
- A team leader never sees every org agent

Per-agent metrics include: active leads, deals (total/won/open), win rate, and **stalled** leads (active leads with no activity in the last `DASHBOARD_STALLED_LEAD_DAYS` days).
</Warning>

## Constants

Widget keys, scope-request enum, and date range constants are defined in `dashboard.constants.ts`:

- Default range: 30 days
- Maximum range: 366 days (one calendar year including leap day)  
- Lead volume default: 14 days
- Stalled lead threshold: `DASHBOARD_STALLED_LEAD_DAYS`
