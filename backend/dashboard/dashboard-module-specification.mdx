## Purpose

A **role-aware** dashboard at `/dashboard`. Every authenticated org member sees a personal view; each persona progressively sees more. The base controller is **AuthGuard-only** — only the two sensitive widgets carry a `@CheckAccess` gate.

### The Bug This Fixes

The old `/dashboard` called `GET /analytics/permissions`, gated by `@CheckAccess(ANALYTICS_VIEW)`, so zero-permission users got a `403` and a blank dashboard. The dashboard module is un-gated at the base, so a BASIC user now gets a non-empty self-scoped dashboard.

## Scope Model

`DashboardScopeService.resolveScopes(userId, orgId)` returns a per-domain resolved scope computed from **team-membership-level** data via ONE consolidated load of `UserOrgRole → role → permissions` + `TeamMembership → teamRoles → role → permissions`.

<Warning>
**CRITICAL (gap #1):** it does NOT use the flattened `PermissionService.getUserEffectivePermissions` set — that collapses all team permissions across all teams and loses which team granted `team_crm.view`. A user who leads Team A (`team_crm.view`) and is a plain member of Team B (no team perms) resolves `crm.defaultTeamIds === [A]` only.
</Warning>

### Domain Scope Capability

Per-domain capability (`DomainScope`):

- `hasOrg`: org-permission present (or `system.admin` / `system.owner`, which grant org on **every** domain)
- `defaultTeamIds`: the membership teams whose roles grant that domain's team key — the default `team`-level aggregate
- `availableTeams`: the dropdown/validation set = membership teams (+ all org teams when the user is org-capable, so org users can drill into any team)

| Domain | org when | team key | self floor | hidden when |
|--------|----------|----------|------------|-------------|
| CRM (leads) | `crm.view`/`crm.manage`/admin/owner | `team_crm.view` | always | never |
| Sales (deals) | `sales.view`/`sales.manage`/admin/owner | `team_sales.view` | always | never |
| Finance (money) | `finance.view`/`finance.manage`/admin/owner | — (no team) | always | never |
| Real Estate (listings) | `real_estate.view`/`real_estate.manage`/admin/owner | — | — | no real_estate access |
| Analytics (team perf) | `analytics.view`/admin/owner | `team_analytics.view` OR team leader | — | no analytics access |

### ResolvedScope Contract

```ts
type DashboardScopeLevel = 'org' | 'team' | 'self';

interface ResolvedScope { 
  level: DashboardScopeLevel; 
  userId: string; 
  orgId: string; 
  teamIds: string[]; 
}
```

Defined in `src/common/scope/resolved-scope.ts` (neutral location so domain services can depend on it without a cycle back into the dashboard module). Passed to **every** composable.

### Switcher + Clamping

The UI exposes one global My/Team/Org switcher plus a **team picker** that appears only when `scope === 'team'`. The picker lists the union of `availableTeams` across CRM, Sales, and Analytics (deduped by id); the chosen `teamId` is sent on every `/dashboard/*` widget request.

When the user has exactly one eligible team, the UI shows a read-only team label instead of a dropdown. The requested level is clamped per widget to its domain max:

<Tabs>
<Tab title="CRM/Sales (self floor)">
- `org`→org else team else self
- `team`→team else org else self  
- `my`→self
</Tab>
<Tab title="Finance (no team)">
- `my`→self
- `team`/`org`→org if `finance.view` else **self** (gap A)
</Tab>
<Tab title="Real Estate (no team, no self)">
- Always org if `real_estate.view`, else hidden
</Tab>
<Tab title="Analytics (no self)">
- `org`→org else team else hidden
- `team`/`my`→team else org else hidden
</Tab>
</Tabs>

### Team Attribution

<Info>
Dashboard `team` scope counts leads/deals where `entity_stakeholder.team_id` matches the eligible team id(s) — the same rule as `LeadService.applyTeamLeadsFilter`. This **includes unclaimed team-pool rows** (`es.user_id IS NULL` with `es.team_id` set).
</Info>

**Frontend components:**
- `dashboard-view.tsx` + `dashboard-controls.tsx`
- `mergeGlobalAvailableTeams()` builds the global picker list
- `resolveTeamScopeSelection()` keeps a valid default when the user switches to Team scope
- `buildDashboardParams()` includes `teamId` only while Team scope is active

### Monetary Rule (gap A)

<Warning>
ALL monetary aggregates (Total Revenue, Revenue Forecast, pipeline VALUE totals) resolve their amount via the **Finance** scope, even though the deal SET/counts resolve via the **Sales** scope. A `sales.view`-without-`finance.view` user therefore sees team/org deal **counts** but **self-scoped money**.
</Warning>

## Endpoints

All endpoints at `/dashboard/*` accept `?scope&teamId&dateFrom&dateTo` query parameters.

### Date Range Validation

`DashboardQueryDto` validates `dateTo` when both bounds are present:

- `dateTo >= dateFrom` (inclusive)
- Span ≤ `DASHBOARD_MAX_RANGE_DAYS` (366 days — one calendar year, incl. leap day)

Invalid ranges return **HTTP 400** with: `dateTo must be on or after dateFrom and within 366 days`.

<Steps>
<Step title="Frontend normalization (Custom preset)">
Before widgets call `/dashboard/*`, the CRM app normalizes user picks in `dashboard.utils.ts` → `rangeToDates()`:

1. Order inverted endpoints (swap when `from` is after `to`)
2. Cap `dateTo` at `now` (future end dates are rejected server-side)
3. If the window is still inverted after capping, collapse `dateFrom` to `startOfDay(dateTo)`
4. Clamp `dateFrom` to the most recent `DASHBOARD_MAX_RANGE_DAYS` window when the preset/custom span would exceed one calendar year
</Step>
</Steps>

### Config Response Shape

```ts
{
  scopes: {
    crm:        { level: 'org'|'team'|'self',  teamIds: string[] },
    sales:      { level: 'org'|'team'|'self',  teamIds: string[] },
    finance:    { level: 'org'|'self',          teamIds: [] },
    realEstate: { level: 'org'|'none',          teamIds: [] },
    analytics:  { level: 'org'|'team'|'none',   teamIds: string[] },
  },
  availableScopes: ('my'|'team'|'org')[],   // union, most-permissive wins
  availableTeams: { crm: [...], sales: [...], analytics: [...] },  // domain-keyed
  visibleWidgets: string[],
}
```

### Multi-Domain Widgets (gap B)

<Tip>
KPIs (Total Leads = CRM; Win-rate = Sales; Revenue = Sales set + Finance money) and Pipeline Summary (lead pipeline = CRM; deal pipeline = Sales; value = Finance) apply EACH sub-metric's own domain `ResolvedScope` — the single requested level is clamped per domain.
</Tip>

### Activity Feed Performer Scope (gap C)

Performer-based scoping:
- `self` = `[userId]`
- `team` = `[userId] ∪ members of eligible CRM/Sales teams`
- `org` = `null` (no performer filter)

Member ids resolved via `TeamMembershipService.getMemberUserIdsInTransaction`. Cursor pagination uses a base64url `{ createdAt, id }` keyset — infinite scroll, no "view full log" link.

<Info>
**Source = unified timeline (audit + activity).** The feed is backed by `UnifiedTimelineService.getActivityFeedInTransaction(em, orgId, { actorUserIds, cursor, limit })`, **not** raw `activity` rows.
</Info>

It merges **audit lifecycle events** (lead created, stage changed, assigned, transfer, deal won/lost) with **logged activities** (note/call/email/meeting), all attributed to the resolved performers.

The `DashboardActivityItemDto` carries:
- `source` (`'audit' | 'activity'`)
- `activityType` = the row `kind` (`TimelineEventType` for audit, `ActivityType` for activity)
- `summary` (pre-resolved message)
- Optional `accentColor`
- Nullable `targetEntityType` / `targetEntityId`

### Tasks vs Events (gap G)

<Check>
Self-only scope (decision F). "events" = tasks with `taskType IN (MEETING, SITE_VISIT)` AND `hasDueTime = true`; "tasks" = the rest. Both come from `TaskService`.
</Check>

### Team Performance Agent Clamp (gap #5)

- `team` level → agents clamped to `team_membership` of the eligible `teamIds`
- `org` level → all org team members
- A team leader never sees every org agent

Per-agent rows include:
- Active leads
- Deals (total/won/open)
- Win rate
- **Stalled** leads (active leads with no activity in the last `DASHBOARD_STALLED_LEAD_DAYS` days)
