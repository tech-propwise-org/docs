## Purpose

A **role-aware** dashboard at `/dashboard` where every authenticated organization member sees a personalized view. Each persona progressively sees more data based on their permissions. The base controller uses **AuthGuard-only** â€” only two sensitive widgets carry a `@CheckAccess` gate.

### Problem Solved

The old `/dashboard` called `GET /analytics/permissions`, gated by `@CheckAccess(ANALYTICS_VIEW)`, causing zero-permission users to receive a `403` error and see a blank dashboard. The new dashboard module is un-gated at the base level, allowing BASIC users to see a non-empty self-scoped dashboard.

## Scope Model

### Server-Side Resolution ("Broadest Wins")

`DashboardScopeService.resolveScopes(userId, orgId)` returns a per-domain resolved scope computed from **team-membership-level** data via ONE consolidated load of:
- `UserOrgRole â†’ role â†’ permissions`
- `TeamMembership â†’ teamRoles â†’ role â†’ permissions`

<Warning>
**CRITICAL (Gap #1):** The system does NOT use the flattened `PermissionService.getUserEffectivePermissions` set â€” that collapses all team permissions across all teams and loses which team granted `team_crm.view`. A user who leads Team A (`team_crm.view`) and is a plain member of Team B (no team perms) resolves `crm.defaultTeamIds === [A]` only.
</Warning>

### Domain Scope Capabilities

Per-domain capability (`DomainScope`):

- **`hasOrg`**: Organization-level permission present (or `system.admin`/`system.owner`, which grant org on every domain)
- **`defaultTeamIds`**: Membership teams whose roles grant that domain's team key â€” the default `team`-level aggregate
- **`availableTeams`**: The dropdown/validation set = membership teams (+ all org teams when the user is org-capable, so org users can drill into any team)

| Domain | Org When | Team Key | Self Floor | Hidden When |
|--------|----------|----------|------------|-------------|
| CRM (leads) | `crm.view`/`crm.manage`/admin/owner | `team_crm.view` | always | never |
| Sales (deals) | `sales.view`/`sales.manage`/admin/owner | `team_sales.view` | always | never |
| Finance (money) | `finance.view`/`finance.manage`/admin/owner | â€” (no team) | always | never |
| Real Estate (listings) | `real_estate.view`/`real_estate.manage`/admin/owner | â€” | â€” | no real_estate access |
| Analytics (team perf) | `analytics.view`/admin/owner | `team_analytics.view` OR team leader | â€” | no analytics access |

### ResolvedScope Contract

```typescript
type DashboardScopeLevel = 'org' | 'team' | 'self';

interface ResolvedScope {
  level: DashboardScopeLevel;
  userId: string;
  orgId: string;
  teamIds: string[];
}
```

<Note>
Defined in `src/common/scope/resolved-scope.ts` (neutral location so domain services can depend on it without a cycle back into the dashboard module). Passed to **every** composable method.
</Note>

## Scope Switcher and Clamping

### Global Switcher

The UI exposes one global My/Team/Org switcher plus a **team picker** that appears only when `scope === 'team'`. The picker lists the union of `availableTeams` across CRM, Sales, and Analytics (deduped by id).

<Tip>
When the user has exactly one eligible team, the UI shows a read-only team label instead of a dropdown.
</Tip>

### Domain-Specific Clamping

The requested level is clamped per widget to its domain maximum using `clampDomain` (server-authoritative):

<AccordionGroup>
<Accordion title="CRM/Sales (Self Floor)">
- `org` â†’ org else team else self
- `team` â†’ team else org else self
- `my` â†’ self
</Accordion>

<Accordion title="Finance (No Team)">
- `my` â†’ self
- `team`/`org` â†’ org if `finance.view` else **self** (Gap A)
</Accordion>

<Accordion title="Real Estate (No Team, No Self)">
- Always org if `real_estate.view`, else hidden
</Accordion>

<Accordion title="Analytics (No Self)">
- `org` â†’ org else team else hidden
- `team`/`my` â†’ team else org else hidden
</Accordion>
</AccordionGroup>

### Team ID Handling

A passed `teamId` is re-validated against the domain's `availableTeams` and narrows `teamIds` to `[teamId]`. An out-of-domain `teamId` is **ignored** (never trusted), falling back to `defaultTeamIds`.

<Info>
When `team` resolves with no `teamId`, it aggregates across ALL eligible teams via `es.team_id IN (?)` (Knex expands the bound id array) â€” there is NO "must select a team" dead-end.
</Info>

### Team Attribution

**Aligned with CRM lead/deal list "Team" tab:** Dashboard `team` scope counts leads/deals where `entity_stakeholder.team_id` matches the eligible team id(s) â€” the same rule as `LeadService.applyTeamLeadsFilter`.

<Check>
This **includes unclaimed team-pool rows** (`es.user_id IS NULL` with `es.team_id` set).
</Check>

<Warning>
It does **not** join `team_membership` on the assignee's user id (the old path silently dropped pool leads and produced counts that disagreed with the lead list).
</Warning>

### Frontend Implementation

**Files:** `dashboard-view.tsx` + `dashboard-controls.tsx`

- `mergeGlobalAvailableTeams()` builds the global picker list
- `resolveTeamScopeSelection()` keeps a valid default when the user switches to Team scope
- `buildDashboardParams()` includes `teamId` only while Team scope is active

### Monetary Rule (Gap A)

<Warning>
ALL monetary aggregates (Total Revenue, Revenue Forecast, pipeline VALUE totals) resolve their amount via the **Finance** scope, even though the deal SET/counts resolve via the **Sales** scope. A `sales.view`-without-`finance.view` user therefore sees team/org deal **counts** but **self-scoped money**.
</Warning>

## Dashboard Endpoints

All endpoints are under `/dashboard/*` and accept query parameters: `?scope&teamId&dateFrom&dateTo`

### Date Range Validation

`DashboardQueryDto` validates `dateTo` when both bounds are present:

<Steps>
<Step title="Range Order Check">
`dateTo >= dateFrom` (inclusive)
</Step>

<Step title="Maximum Span Check">
Span â‰¤ `DASHBOARD_MAX_RANGE_DAYS` (366 days â€” one calendar year, including leap day)
</Step>

<Step title="Error Response">
Invalid ranges return **HTTP 400** with message:  
`dateTo must be on or after dateFrom and within 366 days`
</Step>
</Steps>

### Frontend Date Normalization

**Custom preset handling** in `dashboard.utils.ts` â†’ `rangeToDates()`:

<Steps>
<Step title="Order Inverted Endpoints">
Swap when `from` is after `to`
</Step>

<Step title="Cap Future Dates">
Cap `dateTo` at `now` (future end dates are rejected server-side)
</Step>

<Step title="Collapse Invalid Windows">
If the window is still inverted after capping, collapse `dateFrom` to `startOfDay(dateTo)`
</Step>

<Step title="Clamp to Maximum Range">
Clamp `dateFrom` to the most recent `DASHBOARD_MAX_RANGE_DAYS` window when the preset/custom span would exceed one calendar year (`This Year` is **not** silently shortened to 90 days)
</Step>
</Steps>

<Note>
Custom date pickers disable future calendar days (`maxDate={today}`). Empty windows (valid range, zero rows) return **200** with empty series â€” widgets show the empty state, not the error shell.
</Note>

## Widget-Specific Details

### Multi-Domain Widgets (Gap B)

<Info>
KPIs (Total Leads = CRM; Win-rate = Sales; Revenue = Sales set + Finance money) and Pipeline Summary (lead pipeline = CRM; deal pipeline = Sales; value = Finance) apply EACH sub-metric's own domain `ResolvedScope` â€” the single requested level is clamped per domain.
</Info>

### Pipeline Summary

Lead pipeline uses `LeadService.getLeadPipelineSummaryInTransaction` with special handling for terminal **DONE** stage:

<Note>
Terminal DONE lead stage is split into **Won** / **Lost** virtual buckets by `lead.isWon`, keyed as `${stageId}:won` / `${stageId}:lost` with green/red colors respectively.
</Note>

### Activity Feed (Gap C)

**Performer-based scope resolution:**
- `self` = `[userId]`
- `team` = `[userId]` âˆª members of eligible CRM/Sales teams
- `org` = `null` (no performer filter)

Member IDs resolved via `TeamMembershipService.getMemberUserIdsInTransaction`.

<Info>
Cursor pagination uses a base64url `{ createdAt, id }` keyset â€” infinite scroll, no "view full log" link.
</Info>

#### Unified Timeline Source

The feed is backed by `UnifiedTimelineService.getActivityFeedInTransaction(em, orgId, { actorUserIds, cursor, limit })`, **not** raw `activity` rows.

It merges:
- **Audit lifecycle events** (lead created, stage changed, assigned, transfer, deal won/lost, etc.)
  - Curated audit tables resolved into human messages by `AuditTimelineResolver`
- **Logged activities** (note/call/email/meeting, etc.)

All attributed to the resolved performers (`null` = org).

#### Activity Item Schema

```typescript
interface DashboardActivityItemDto {
  source: 'audit' | 'activity';
  activityType: TimelineEventType | ActivityType;  // row kind
  summary: string;  // pre-resolved message
  accentColor?: string;
  targetEntityType?: string;
  targetEntityId?: string;
}
```

<Tip>
The frontend widget keys its icon/tint off `activityType`. See `TIMELINE_MODULE_SPECIFICATION.md` â†’ *Org-Wide Actor Activity Feed*.
</Tip>

### Tasks vs Events (Gap G)

<Warning>
Self-only scope (Decision F).
</Warning>

**Classification:**
- **Events** = tasks with `taskType IN (MEETING, SITE_VISIT)` AND `hasDueTime = true`
- **Tasks** = all other tasks

Both come from `TaskService.getUpcomingTasksInTransaction`.

### Team Performance (Gap #5)

**Agent clamping:**
- `team` level â†’ agents clamped to `team_membership` of the eligible `teamIds`
- `org` level â†’ all org team members

<Check>
A team leader never sees every org agent.
</Check>

**Per-agent metrics:**
- Active leads
- Deals (total/won/open)
- Win rate
- **Stalled** leads (active leads with no activity in the last `DASHBOARD_STALLED_LEAD_DAYS` days)

The dashboard module never raw-SQLs another module's tables.

## Known Gaps

<AccordionGroup>
<Accordion title="Gap #1: Team Permission Resolution">
Does not use flattened `PermissionService.getUserEffectivePermissions` â€” preserves team-specific grants for accurate `defaultTeamIds` calculation.
</Accordion>

<Accordion title="Gap #2: Domain-Keyed Teams">
`availableTeams` response is domain-keyed rather than a flat list to support domain-specific validation.
</Accordion>

<Accordion title="Gap A: Finance Scope on Money">
Monetary aggregates always use Finance scope, even when viewing Sales team/org deal counts.
</Accordion>

<Accordion title="Gap B: Multi-Domain Widget Clamping">
KPIs and Pipeline Summary apply per-metric domain clamping within a single widget.
</Accordion>

<Accordion title="Gap C: Performer-Based Activity Feed">
Activity feed uses performer scope (who performed actions) rather than entity ownership scope.
</Accordion>

<Accordion title="Gap F: Self-Only Tasks">
Tasks widget intentionally restricted to self scope regardless of user permissions.
</Accordion>

<Accordion title="Gap G: Events Classification">
Events distinguished from tasks by type and due-time presence for calendar integration.
</Accordion>

<Accordion title="Gap #5: Team Performance Agent Clamp">
Team leaders see only their team members' performance, never all org agents.
</Accordion>

<Accordion title="Gap H: Listings Summary">
Real Estate listings summary aggregation across `InventoryUnitService` and `SharedListingService`.
</Accordion>
</AccordionGroup>
