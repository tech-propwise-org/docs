## Purpose

A **role-aware** dashboard at `/dashboard` where every authenticated organization member sees a personal view, with each persona progressively seeing more data based on their permissions. The base controller uses **AuthGuard-only** protection, with only two sensitive widgets carrying a `@CheckAccess` gate.

<Note>
This module fixes a critical bug where the old `/dashboard` called `GET /analytics/permissions` with `@CheckAccess(ANALYTICS_VIEW)`, causing zero-permission users to receive a `403` error and blank dashboard. The new dashboard module is un-gated at the base level, allowing BASIC users to see a non-empty self-scoped dashboard.
</Note>

**Location:** `src/modules/crm/dashboard/`

## Scope Resolution Model

The dashboard uses a server-side scope resolution system where "broadest wins" across different domains.

### DashboardScopeService

`DashboardScopeService.resolveScopes(userId, orgId)` returns a per-domain resolved scope computed from **team-membership-level** data via one consolidated load:

```
UserOrgRole â†’ role â†’ permissions + TeamMembership â†’ teamRoles â†’ role â†’ permissions
```

<Warning>
**Critical Gap #1:** The scope service does NOT use the flattened `PermissionService.getUserEffectivePermissions` set. That approach collapses all team permissions across teams and loses which specific team granted `team_crm.view`. A user who leads Team A (with `team_crm.view`) and is a plain member of Team B (no team permissions) will resolve `crm.defaultTeamIds === [A]` only.
</Warning>

### Domain Capabilities

Each domain has a `DomainScope` with:

- `hasOrg`: Org-level permission present (or `system.admin`/`system.owner` which grant org access on every domain)
- `defaultTeamIds`: Membership teams whose roles grant that domain's team key
- `availableTeams`: Dropdown/validation set (membership teams + all org teams when user is org-capable)

| Domain | Org Access When | Team Key | Self Floor | Hidden When |
|--------|----------------|----------|------------|-------------|
| CRM (leads) | `crm.view`/`crm.manage`/admin/owner | `team_crm.view` | always | never |
| Sales (deals) | `sales.view`/`sales.manage`/admin/owner | `team_sales.view` | always | never |
| Finance (money) | `finance.view`/`finance.manage`/admin/owner | â€” (no team) | always | never |
| Real Estate (listings) | `real_estate.view`/`real_estate.manage`/admin/owner | â€” | â€” | no real_estate access |
| Analytics (team perf) | `analytics.view`/admin/owner | `team_analytics.view` OR team leader | â€” | no analytics access |

### ResolvedScope Contract

```typescript
type DashboardScopeLevel = 'org' | 'team' | 'self';

interface ResolvedScope {
  level: DashboardScopeLevel;
  userId: string;
  orgId: string;
  teamIds: string[];
}
```

<Info>
Defined in `src/common/scope/resolved-scope.ts` (neutral location so domain services can depend on it without circular dependencies). This scope is passed to every composable method.
</Info>

## Scope Switcher & Clamping

The UI provides:
- One global **My/Team/Org switcher**
- A **team picker** that appears only when `scope === 'team'`

### Clamping Rules

The requested level is clamped per widget to its domain maximum:

<Tabs>
  <Tab title="CRM/Sales (self floor)">
    - `org` â†’ org else team else self
    - `team` â†’ team else org else self  
    - `my` â†’ self
  </Tab>
  <Tab title="Finance (no team)">
    - `my` â†’ self
    - `team`/`org` â†’ org if `finance.view` else **self**
  </Tab>
  <Tab title="Real Estate (no team, no self)">
    - Always org if `real_estate.view`, else hidden
  </Tab>
  <Tab title="Analytics (no self)">
    - `org` â†’ org else team else hidden
    - `team`/`my` â†’ team else org else hidden
  </Tab>
</Tabs>

### Team Attribution

<Check>
Dashboard `team` scope counts leads/deals where `entity_stakeholder.team_id` matches the eligible team ID(s) â€” aligned with CRM lead/deal list "Team" tab. This **includes unclaimed team-pool rows** (`es.user_id IS NULL` with `es.team_id` set).
</Check>

### Monetary Rule

<Warning>
**Gap A:** ALL monetary aggregates (Total Revenue, Revenue Forecast, pipeline VALUE totals) resolve their amount via the **Finance** scope, even though the deal SET/counts resolve via the **Sales** scope. A `sales.view`-without-`finance.view` user sees team/org deal **counts** but **self-scoped money**.
</Warning>

## API Endpoints

All endpoints are under `/dashboard/*` and accept query parameters: `?scope&teamId&dateFrom&dateTo`

### Date Range Validation

`DashboardQueryDto` validates date ranges with these rules:

<Steps>
  <Step title="Date Order">
    `dateTo >= dateFrom` (inclusive)
  </Step>
  <Step title="Maximum Span">
    Span â‰¤ `DASHBOARD_MAX_RANGE_DAYS` (366 days â€” one calendar year including leap day)
  </Step>
  <Step title="Error Response">
    Invalid ranges return **HTTP 400** with message: `dateTo must be on or after dateFrom and within 366 days`
  </Step>
</Steps>

### Frontend Normalization

The frontend normalizes user picks in `dashboard.utils.ts` â†’ `rangeToDates()`:

1. Order inverted endpoints (swap when `from` is after `to`)
2. Cap `dateTo` at `now` (future end dates are rejected server-side)  
3. If window is still inverted after capping, collapse `dateFrom` to `startOfDay(dateTo)`
4. Clamp `dateFrom` to most recent `DASHBOARD_MAX_RANGE_DAYS` window when span exceeds one calendar year

## Widget Implementation Details

### Multi-Domain Widgets

<Info>
**Gap B:** KPIs and Pipeline Summary widgets apply EACH sub-metric's own domain `ResolvedScope` â€” the single requested level is clamped per domain.
</Info>

**KPIs Example:**
- Total Leads = CRM scope
- Win-rate = Sales scope  
- Revenue = Sales set + Finance money

### Activity Feed

<Tip>
**Gap C:** Uses performer-based scoping where `self` = `[userId]`, `team` = `[userId] âˆª members of eligible CRM/Sales teams`, `org` = no performer filter.
</Tip>

The feed merges:
- **Audit lifecycle events** (lead created, stage changed, assigned, deal won/lost)
- **Logged activities** (note/call/email/meeting)

All attributed to resolved performers using cursor pagination with base64url `{ createdAt, id }` keyset.

```typescript
interface DashboardActivityItemDto {
  source: 'audit' | 'activity';
  activityType: string; // TimelineEventType | ActivityType
  summary: string; // pre-resolved message
  accentColor?: string;
  targetEntityType?: string;
  targetEntityId?: string;
}
```

### Tasks vs Events

<Warning>
**Gap G:** Tasks widget is self-only. "Events" = tasks with `taskType IN (MEETING, SITE_VISIT)` AND `hasDueTime = true`; "tasks" = everything else.
</Warning>

### Team Performance

<Info>
**Gap #5:** `team` level clamps agents to `team_membership` of eligible `teamIds`; `org` level shows all org team members. Team leaders never see every org agent.
</Info>

Per-agent metrics include:
- Active leads
- Deals (total/won/open)  
- Win rate
- Stalled leads (active leads with no activity in last `DASHBOARD_STALLED_LEAD_DAYS`)
