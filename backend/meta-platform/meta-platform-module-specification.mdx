# Meta Platform Module Specification

The Meta Platform module provides integration capabilities with Meta's Graph API ecosystem, including Facebook and Instagram messaging, advertising, and webhook handling. This module serves as a foundational layer for interacting with Meta's various APIs while handling authentication, error management, and webhook verification.

<Note>
**Last Updated:** 2026-04-14  
**Status:** Active
</Note>

## Overview

The module abstracts common Meta API operations, implements OAuth flows, and provides secure webhook validation to ensure reliable communication with Meta's services. It's designed to be stateless and relies on dependency injection for configuration management.

### Key Features

- **Graph API Integration**: Core HTTP client for all Meta API interactions
- **OAuth Management**: Complete token lifecycle handling
- **Webhook Security**: HMAC-SHA256 signature verification
- **Error Handling**: Meta-specific error classification and retry logic
- **Circuit Breaker**: Per-account isolation for reliability

## Services

### MetaGraphApiService

Core service for making authenticated requests to Meta's Graph API with automatic retry logic, comprehensive error handling, and type-safe request/response handling.

<Tip>
All methods support optional circuit breaker isolation by passing a `circuitKey` parameter, typically the `channelAccountId`.
</Tip>

#### Methods

<AccordionGroup>
  <Accordion title="GET Request">
    **Signature:** `get<T>(path, params, token, version?, circuitKey?, baseUrl?, timeoutMs?)`
    
    GET request with query parameters and optional timeout configuration.
  </Accordion>
  
  <Accordion title="POST Request">
    **Signature:** `post<T>(path, body, token, version?, circuitKey?, baseUrl?)`
    
    POST request with JSON body for standard API operations.
  </Accordion>
  
  <Accordion title="POST Form Data">
    **Signature:** `postFormData<T>(path, formData, token, version?, circuitKey?, baseUrl?)`
    
    POST request with multipart form data for file uploads.
  </Accordion>
  
  <Accordion title="DELETE Request">
    **Signature:** `delete<T>(path, token, version?, circuitKey?, baseUrl?)`
    
    DELETE request with no body or query params. Used by `ChannelAccountService` for Meta webhook unsubscription on channel account deletion.
    
    <Note>
    Append query params to the `path` parameter if needed.
    </Note>
  </Accordion>
</AccordionGroup>

#### Configuration Dependencies

- `app.meta.graphApiBaseUrl`: Base URL for Graph API (defaults to `https://graph.facebook.com`)
- `app.meta.whatsappApiVersion`: Default API version when callers don't pass an explicit version

### MetaOAuthService

Handles OAuth token operations for Meta platform authentication with support for various token types and exchange flows.

#### Configuration Dependencies

- `app.meta.graphApiBaseUrl`: Base URL for OAuth endpoints
- `app.meta.appId`: Meta application ID
- `app.meta.appSecret`: Meta application secret
- `app.meta.oauthApiVersion`: API version for OAuth dialog and token exchange URLs (default `v22.0`)

#### Methods

<AccordionGroup>
  <Accordion title="Authorization URL Generation">
    **Method:** `getAuthorizationUrl(scopes, redirectUri, state)`
    
    Builds the Facebook OAuth dialog URL with the configured `oauthApiVersion`.
  </Accordion>
  
  <Accordion title="Code Exchange">
    **Method:** `exchangeCodeForToken(code, redirectUri)`
    
    Exchanges an authorization code for a short-lived access token.
  </Accordion>
  
  <Accordion title="Embedded Signup Exchange">
    **Method:** `exchangeEmbeddedSignupCode(code)`
    
    Exchanges a WhatsApp Embedded Signup code for a non-expiring BISUAT.
    
    <Warning>
    Uses `axios` directly (no retries) because the code is single-use with a 30s TTL — retries would fail with Meta error 36007.
    </Warning>
  </Accordion>
  
  <Accordion title="Long-Lived Token Exchange">
    **Method:** `exchangeForLongLivedToken(shortLivedToken)`
    
    Exchanges a short-lived user token for a long-lived token (~60 days).
  </Accordion>
  
  <Accordion title="Token Refresh">
    **Method:** `refreshToken(token)`
    
    Refreshes a token (delegates to `exchangeForLongLivedToken`).
  </Accordion>
</AccordionGroup>

#### Token Response Interface

```typescript
interface TokenResponse {
  access_token: string;
  token_type: string;
  expires_in?: number;
}
```

### MetaWebhookGuard

NestJS guard that validates webhook requests from Meta using HMAC-SHA256 signature verification with enhanced security features.

#### Security Features

- **HMAC-SHA256 signature validation**: Ensures request authenticity
- **Timing-safe comparison**: Prevents timing attack vectors
- **Automatic rejection**: Invalid signatures are rejected immediately

#### Configuration Dependencies

- `app.meta.webhookVerifyToken`: Secret token for webhook verification

<Info>
Single value configuration is correct for one-Meta-App architecture. See MESSAGING_MODULE_SPECIFICATION.md § "Webhook Verify Token" for rationale.
</Info>

## Error Handling

### MetaApiError

Specialized error class for Meta API operations with enhanced context and retry logic classification.

```typescript
class MetaApiError extends Error {
  constructor(
    message: string,
    statusCode: number,
    metaErrorCode?: number,
    metaErrorSubcode?: number,
    isRetryable: boolean = false,
  );
}
```

#### Properties

<AccordionGroup>
  <Accordion title="statusCode">
    HTTP status code from the API response
  </Accordion>
  
  <Accordion title="metaErrorCode">
    Meta-specific error code (if available)
  </Accordion>
  
  <Accordion title="metaErrorSubcode">
    Meta-specific error subcode (if available)
  </Accordion>
  
  <Accordion title="isRetryable">
    Indicates if the operation can be safely retried
  </Accordion>
</AccordionGroup>

<Note>
Both `MetaApiError` and `CircuitOpenError` call `Object.setPrototypeOf(this, <Class>.prototype)` in their constructors to ensure `instanceof` checks work correctly with TypeScript class inheritance from `Error`.
</Note>

### Required Configuration Parameters

```typescript
meta: {
  graphApiBaseUrl: string; // Meta Graph API base URL
  appId: string; // Meta application ID
  appSecret: string; // Meta application secret
  webhookVerifyToken: string; // Webhook verification token
  tokenEncryptionKey: string; // AES-256-GCM key for token encryption at rest

  instagramApiVersion: string; // Instagram Messaging API version (e.g. 'v23.0')
  messengerApiVersion: string; // Messenger Platform API version (e.g. 'v23.0')
  oauthApiVersion: string; // OAuth dialog/token exchange version (e.g. 'v22.0')
}
```

### Configurable API Versions

Meta deprecates Graph API versions on a rolling basis. Each channel's API version is configurable via environment variables so upgrades require only a config change, not code changes.

| Environment Variable | Config Key | Default | Used By |
|---------------------|------------|---------|---------|
| `META_WHATSAPP_API_VERSION` | `app.meta.whatsappApiVersion` | `v22.0` | WhatsAppProviderService, TemplateService, ChannelAccountService |
| `META_INSTAGRAM_API_VERSION` | `app.meta.instagramApiVersion` | `v23.0` | InstagramProviderService, ChannelAccountService |
| `META_MESSENGER_API_VERSION` | `app.meta.messengerApiVersion` | `v23.0` | MessengerProviderService, ChannelAccountService |
| `META_OAUTH_API_VERSION` | `app.meta.oauthApiVersion` | `v22.0` | MetaOAuthService |

<Steps>
  <Step title="Update Environment Variable">
    Modify the appropriate environment variable for the API version you want to upgrade.
  </Step>
  
  <Step title="Deploy Changes">
    Deploy the updated configuration to your environment.
  </Step>
  
  <Step title="Test Integration">
    Verify that the integration works correctly with the new API version.

</Steps>

## Entities

### WebhookEventLog

Raw webhook payload storage used for idempotency and debugging. Persisted before async processing so duplicate webhooks from Meta are detected and skipped.

| Field | Type | Description |
|-------|------|-------------|
| id | UUID | Primary key |
| platform | WebhookPlatform | `META` or `GOOGLE` |
| eventType | WebhookEventType | `MESSAGING`, `STATUS_UPDATE`, or `LEAD_FORM` |
| externalEventId | string | Deterministic ID for idempotency (unique constraint) |
| payload | JSONB | Raw webhook payload |
| status | WebhookEventStatus | `RECEIVED` → `PROCESSING` → `PROCESSED` / `FAILED` |
| organization | Organization? | Resolved after processing (nullable — org unknown at receive time) |
| processedAt | Date? | When processing completed |
| error | string? | Error message if processing failed |
| createdAt | Date | Receive timestamp |

<Warning>
- No RLS — webhook arrives before organization context is known
- No soft delete — rows are hard-deleted by retention cron after 30 days
</Warning>

## Circuit Breaker Integration

`MetaGraphApiService` supports per-account circuit breaker isolation. All Meta Graph API call sites (13 across 4 files in the Messaging module) pass the `channelAccountId` as the `circuitKey` parameter, ensuring one degraded account does not trip the circuit breaker for others.

<Tip>
This design prevents cascading failures where issues with one Meta account affect other accounts using the same service.
</Tip>

## Business Rules

<Check>
**Retry Logic**: Failed requests are retried up to 3 times with exponential backoff
</Check>

<Check>
**Signature Validation**: All webhook requests must include valid `x-hub-signature-256` header
</Check>

<Check>
**Token Management**: OAuth tokens should be refreshed before expiration
</Check>

<Check>
**Error Classification**: API errors are classified as retryable or non-retryable based on status codes
</Check>

## Security Considerations

<Warning>
- All webhook requests are validated using HMAC-SHA256 signatures
- Timing-safe comparison prevents timing attack vectors
- Application secrets are loaded from environment variables
- Raw request bodies are preserved for signature verification
</Warning>

## Usage Patterns

### Webhook Validation

```typescript
@UseGuards(MetaWebhookGuard)
@Post('/webhook')
async handleWebhook(@Body() body: any) {
  // Webhook payload is already validated
  // Process the webhook data
}
```

### API Requests with Circuit Breaker

```typescript
const response = await this.metaGraphApiService.post(
  '/messages',
  messagePayload,
  accessToken,
  'v22.0',
  channelAccountId, // Circuit breaker key
);
```

### OAuth Flow Implementation

```typescript
// Generate authorization URL
const authUrl = this.metaOAuthService.getAuthorizationUrl(
  ['pages_messaging'],
  redirectUri,
  state
);

// Exchange code for token
const tokenResponse = await this.metaOAuthService.exchangeCodeForToken(
  code,
  redirectUri
);
```
