## Overview

The Meta Platform module provides integration capabilities with Meta's Graph API ecosystem, including Facebook and Instagram messaging, advertising, and webhook handling. This module serves as a foundational layer for interacting with Meta's various APIs while handling authentication, error management, and webhook verification.

The module abstracts common Meta API operations, implements OAuth flows, and provides secure webhook validation to ensure reliable communication with Meta's services.

<Info>
**Last Updated:** 2026-06-01  
**Status:** Active
</Info>

## Architecture

The module follows a service-oriented architecture with clear separation of concerns:

- **MetaGraphApiService**: Core HTTP client for Graph API interactions
- **MetaOAuthService**: Handles OAuth token management and exchange
- **MetaWebhookGuard**: Validates incoming webhook requests from Meta
- **MetaApiError**: Specialized error class for Meta API responses

<Note>
The module is designed to be stateless and relies on dependency injection for configuration management.
</Note>

## Services

### MetaGraphApiService

Core service for making authenticated requests to Meta's Graph API.

#### Key Features

- Automatic retry logic with exponential backoff
- Comprehensive error handling with Meta-specific error codes
- Type-safe request/response handling
- Configurable base URL with a fixed 3-total-attempt retry cap
- Per-account circuit breaker isolation

#### Methods

<AccordionGroup>
<Accordion title="get<T>">
**Signature:** `(path, params, token, version?, circuitKey?, baseUrl?, timeoutMs?)`

**Description:** GET request with query parameters

```typescript
const response = await metaGraphApiService.get<UserProfile>(
  '/me',
  { fields: 'id,name,email' },
  accessToken,
  'v22.0'
);
```
</Accordion>

<Accordion title="post<T>">
**Signature:** `(path, body, token, version?, circuitKey?, baseUrl?)`

**Description:** POST request with JSON body

```typescript
const result = await metaGraphApiService.post<MessageResponse>(
  '/me/messages',
  { message: { text: 'Hello World' } },
  accessToken
);
```
</Accordion>

<Accordion title="postFormData<T>">
**Signature:** `(path, formData, token, version?, circuitKey?, baseUrl?)`

**Description:** POST request with multipart form data

```typescript
const formData = new FormData();
formData.append('file', buffer, 'image.jpg');

const response = await metaGraphApiService.postFormData<UploadResponse>(
  '/me/photos',
  formData,
  accessToken
);
```
</Accordion>

<Accordion title="uploadResumableAsset">
**Signature:** `({ fileName, fileType, fileBuffer, token, version?, circuitKey? })`

**Description:** Creates a Meta Resumable Upload session under the configured app id and uploads raw bytes, returning the asset handle (`h`) used for WhatsApp template media header examples

```typescript
const assetHandle = await metaGraphApiService.uploadResumableAsset({
  fileName: 'template-header.jpg',
  fileType: 'image/jpeg',
  fileBuffer: imageBuffer,
  token: accessToken
});
```
</Accordion>

<Accordion title="delete<T>">
**Signature:** `(path, token, version?, circuitKey?, baseUrl?)`

**Description:** DELETE request (no body or query params — append query params to `path` if needed). Used by messaging account cleanup for webhook unsubscription and app-permission revocation.

```typescript
await metaGraphApiService.delete(
  '/webhook-subscription?object=page',
  accessToken
);
```
</Accordion>
</AccordionGroup>

<Note>
All methods accept an optional `version` (defaults to `app.meta.whatsappApiVersion`), `circuitKey` for per-account circuit breaker isolation, and `baseUrl` to override the default `graph.facebook.com` (e.g. `graph.instagram.com` for Instagram Login API calls).
</Note>

#### Configuration Dependencies

- `app.meta.graphApiBaseUrl`: Base URL for Graph API (defaults to `https://graph.facebook.com`)
- `app.meta.whatsappApiVersion`: Default API version when callers don't pass an explicit version

### MetaOAuthService

Handles OAuth token operations for Meta platform authentication.

#### Configuration Dependencies

- `app.meta.graphApiBaseUrl`: Base URL for OAuth endpoints
- `app.meta.appId`: Meta application ID
- `app.meta.appSecret`: Meta application secret
- `app.meta.oauthApiVersion` (`META_OAUTH_API_VERSION` env var): API version for OAuth dialog and token exchange URLs (default `v22.0`)

#### Methods

<AccordionGroup>
<Accordion title="getAuthorizationUrl">
**Signature:** `(scopes, redirectUri, state)`

**Description:** Builds the Facebook OAuth dialog URL with the configured `oauthApiVersion`

```typescript
const authUrl = metaOAuthService.getAuthorizationUrl(
  ['pages_messaging', 'pages_manage_metadata'],
  'https://app.example.com/auth/callback',
  'random-state-string'
);
```
</Accordion>

<Accordion title="exchangeCodeForToken">
**Signature:** `(code, redirectUri)`

**Description:** Exchanges an authorization code for a short-lived access token

```typescript
const tokenResponse = await metaOAuthService.exchangeCodeForToken(
  authorizationCode,
  'https://app.example.com/auth/callback'
);
```
</Accordion>

<Accordion title="exchangeEmbeddedSignupCode">
**Signature:** `(code)`

**Description:** Exchanges a WhatsApp Embedded Signup code for a non-expiring BISUAT. Uses `axios` directly (no retries) because the code is single-use with a 30s TTL — retries would fail with Meta error 36007.

```typescript
const token = await metaOAuthService.exchangeEmbeddedSignupCode(
  embeddedSignupCode
);
```
</Accordion>

<Accordion title="exchangeForLongLivedToken">
**Signature:** `(shortLivedToken)`

**Description:** Exchanges a short-lived user token for a long-lived token (~60 days)

```typescript
const longLivedToken = await metaOAuthService.exchangeForLongLivedToken(
  shortLivedToken.access_token
);
```
</Accordion>

<Accordion title="refreshToken">
**Signature:** `(token)`

**Description:** Refreshes a token (delegates to `exchangeForLongLivedToken`)

```typescript
const refreshedToken = await metaOAuthService.refreshToken(existingToken);
```
</Accordion>
</AccordionGroup>

#### Token Response Interface

```typescript
interface TokenResponse {
  access_token: string;
  token_type: string;
  expires_in?: number;
}
```

### MetaWebhookGuard

NestJS guard that validates webhook requests from Meta using HMAC-SHA256 signature verification.

#### Security Features

- HMAC-SHA256 signature validation
- Timing-safe comparison to prevent timing attacks
- Automatic request rejection for invalid signatures
- Route-level throttling remains enabled on webhook endpoints to protect CPU, database writes, and pg-boss queue capacity

<Warning>
Signature validation proves authenticity, but it does not prevent signed flood/retry storms. Route-level throttling provides additional protection.
</Warning>

#### Configuration Dependencies

- `app.meta.webhookVerifyToken`: Secret token for webhook verification (single value — correct for one-Meta-App architecture; see MESSAGING_MODULE_SPECIFICATION.md § "Webhook Verify Token" for rationale)

## Error Handling

### MetaApiError

Specialized error class for Meta API operations with enhanced context:

```typescript
class MetaApiError extends Error {
  constructor(
    message: string,
    statusCode: number,
    metaErrorCode?: number,
    metaErrorSubcode?: number,
    isRetryable: boolean = false,
  );
}
```

#### Properties

- `statusCode`: HTTP status code from the API response
- `metaErrorCode`: Meta-specific error code (if available)
- `metaErrorSubcode`: Meta-specific error subcode (if available)
- `isRetryable`: Indicates if the operation can be safely retried

<Note>
Both `MetaApiError` and `CircuitOpenError` call `Object.setPrototypeOf(this, <Class>.prototype)` in their constructors to ensure `instanceof` checks work correctly with TypeScript class inheritance from `Error`.
</Note>

### WhatsApp Billing Errors

Meta WhatsApp send/status failures that indicate WABA billing setup problems are handled in the Messaging module, not by changing Propwise subscription billing. The canonical billing/payment error code is `131042` ("Business eligibility payment issue").

<Info>
Messaging classifies this and related payment-method/billing/credit-line text as `MessageFailureCategory.META_BILLING`, preserves the raw `failedReason`, and includes the category on message DTOs and `message-status-updated` socket events so the frontend can guide the user to Meta Business Suite billing settings.
</Info>

## Configuration

The module requires the following configuration parameters in the application config:

```typescript
encryptionKey: string; // AES-256-GCM key for at-rest encryption — top-level, shared across all modules

meta: {
  graphApiBaseUrl: string; // Meta Graph API base URL
  instagramGraphApiBaseUrl: string; // Instagram Login Graph API base URL
  appId: string; // Meta application ID
  appSecret: string; // Meta application secret
  webhookVerifyToken: string; // Webhook verification token
  whatsappApiVersion: string; // WhatsApp Cloud API version (e.g. 'v22.0')
  instagramApiVersion: string; // Instagram Messaging API version (e.g. 'v23.0')
  messengerApiVersion: string; // Messenger Platform API version (e.g. 'v23.0')
  oauthApiVersion: string; // OAuth dialog/token exchange version (e.g. 'v22.0')
  metaSyncPageDelayMs: number; // Delay between historical sync pagination calls
  mediaShareEnrichmentTimeoutMs: number; // Timeout for best-effort shared reel/post preview lookup
}
```

### Configurable API Versions

Meta deprecates Graph API versions on a rolling basis. Each channel's API version is configurable via environment variables so upgrades require only a config change, not code changes:

| Environment Variable         | Config Key                     | Default | Used By                                                                |
| ---------------------------- | ------------------------------ | ------- | ---------------------------------------------------------------------- |
| `META_WHATSAPP_API_VERSION`  | `app.meta.whatsappApiVersion`  | `v22.0` | WhatsAppProviderService, TemplateService, ChannelAccountService        |
| `META_INSTAGRAM_API_VERSION` | `app.meta.instagramApiVersion` | `v23.0` | InstagramProviderService, ChannelAccountService, InstagramMediaService |
| `META_MESSENGER_API_VERSION` | `app.meta.messengerApiVersion` | `v23.0` | MessengerProviderService, ChannelAccountService                        |
| `META_OAUTH_API_VERSION`     | `app.meta.oauthApiVersion`     | `v22.0` | MetaOAuthService                                                       |

## Entities

### WebhookEventLog

Raw webhook payload storage used for idempotency and debugging. Persisted before async processing so duplicate webhooks from Meta are detected and skipped.

| Field           | Type               | Description                                                        |
| --------------- | ------------------ | ------------------------------------------------------------------ |
| id              | UUID               | Primary key                                                        |
| platform        | WebhookPlatform    | `META` or `GOOGLE`                                                 |
| eventType       | WebhookEventType   | `MESSAGING`, `STATUS_UPDATE`, or `LEAD_FORM`                       |
| externalEventId | string             | Deterministic ID for idempotency (unique constraint)               |
| payload         | JSONB              | Raw webhook payload                                                |
| status          | WebhookEventStatus | `RECEIVED` → `PROCESSING` → `PROCESSED` / `FAILED`                 |
| organization    | Organization?      | Resolved after processing (nullable — org unknown at receive time) |
| processedAt     | Date?              | When processing completed                                          |
| error           | string?            | Error message if processing failed                                 |

<Tip>
The `externalEventId` field ensures webhook idempotency by creating a unique constraint that prevents duplicate processing of the same webhook event from Meta.
</Tip>
