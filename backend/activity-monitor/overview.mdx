The Activity Monitor module provides automated monitoring and alerting for lead activity within the PropWise CRM. It tracks leads that become inactive, stale, or stuck in pipeline stages according to configurable rules, and generates alerts to agents and team leads to ensure timely follow-up actions.

<Info>
The Activity Monitor module is fully implemented and actively processing alerts through the shared PgBoss infrastructure.
</Info>

## Key Features

<CardGroup cols={2}>
  <Card title="Rule-Based Monitoring" icon="gear">

  </Card>
  <Card title="Agent Notifications" icon="bell">

  </Card>
  <Card title="Team Lead Escalation" icon="arrow-up">
    Automatic escalation with recurring reminders and optional org-admin notifications
  </Card>

  </Card>
</CardGroup>

## Architecture

The module follows a scanner-based architecture with the following components:

- **ActivityMonitorScanner** - Evaluates leads against active rules every 60 seconds
- **ActivityMonitorFollowUp** - Processes overdue alerts every 15 minutes
- **ActivityMonitorScheduler** - Manages pg-boss job scheduling with per-org distribution
- **ActivityMonitorService** - Core business logic for alert management
- **ActivityMonitorNotificationHandler** - Handles user interactions with alert notifications

<Note>
All monitoring operates within tenant context boundaries using scheduled processing via pg-boss for reliable, distributed job execution.
</Note>

## Entities

### ActivityMonitorRule

Defines monitoring conditions and alert parameters for leads.

| Field | Type | Description |
|-------|------|-------------|
| `id` | UUID | Primary key |
| `organization_id` | UUID | Organization reference |
| `trigger_type` | MonitorTriggerType | Type of condition to monitor |
| `name` | String | Rule display name |
| `description` | String | Rule description |
| `is_active` | Boolean | Whether rule is enabled |
| `apply_to` | MonitorApplyTo | Which leads to monitor |
| `stage_ids` | UUID[] | Specific stages (when apply_to is SPECIFIC_STAGES) |
| `temperature_levels` | String[] | Lead temperatures to monitor |
| `threshold_value` | Integer | Threshold amount |
| `threshold_unit` | ThresholdUnit | Time unit for threshold |
| `follow_up_hours` | Integer | Follow-up value before escalation |
| `follow_up_unit` | ThresholdUnit | Time unit for follow-up |
| `snooze_duration_value` | Integer | Suppress alerts after acknowledge/dismiss |
| `snooze_duration_unit` | ThresholdUnit | Unit for snooze duration |
| `max_follow_ups` | Integer | Max team-lead notifications before org-admin escalation |
| `notify_team_lead` | Boolean | Enable team lead notifications |

<Tabs>
  <Tab title="MonitorTriggerType">
    - `NO_ACTIVITY` - No activities recorded within threshold
    - `STALE_LEAD` - No lead updates within threshold  
    - `STAGE_STUCK` - Lead in same stage too long
  </Tab>
  <Tab title="MonitorApplyTo">
    - `ALL_LEADS` - Monitor all leads
    - `SPECIFIC_STAGES` - Monitor specific pipeline stages
    - `SPECIFIC_TEMPERATURES` - Monitor specific lead temperatures
  </Tab>
  <Tab title="ThresholdUnit">
    - `MINUTES` - Minute-based thresholds
    - `HOURS` - Hour-based thresholds
    - `DAYS` - Day-based thresholds
  </Tab>
</Tabs>

### ActivityMonitorAlert

Represents an alert generated when a lead breaches monitoring thresholds.

| Field | Type | Description |
|-------|------|-------------|
| `id` | UUID | Primary key |
| `organization_id` | UUID | Organization reference |
| `rule_id` | UUID | Monitoring rule reference |
| `lead_id` | UUID | Lead that triggered alert |
| `agent_user_id` | UUID | Lead's assigned agent |
| `team_lead_user_id` | UUID | Agent's team lead |
| `status` | AlertStatus | Current alert status |
| `triggered_at` | DateTime | When alert was created |
| `agent_notified_at` | DateTime | When agent was notified |
| `follow_up_sent_at` | DateTime | When follow-up was sent |
| `acknowledged_at` | DateTime | When alert was acknowledged |
| `escalated_at` | DateTime | When alert was escalated |
| `follow_up_count` | Integer | Count of follow-up notifications |
| `org_admin_notified_at` | DateTime | When org admins were notified |
| `resolved_by_tl_at` | DateTime | When team lead dismissed alert |
| `reassigned_to_user_id` | UUID | Agent selected for reassignment |

<Warning>
A partial unique index ensures at most one open alert (PENDING or ESCALATED) per lead+rule combination.
</Warning>

## API Endpoints

### Activity Monitor Rules

<CodeGroup>
```http GET /activity-monitor/rules
Authorization: org-settings:read

Response:
{
  "rules": [
    {
      "id": "uuid",
      "trigger_type": "NO_ACTIVITY",
      "name": "Inactive Lead Alert",
      "is_active": true,
      "threshold_value": 24,
      "threshold_unit": "HOURS"
    }
  ]
}
```

```http POST /activity-monitor/rules
Authorization: org-settings:write
Content-Type: application/json

{
  "trigger_type": "STALE_LEAD",
  "name": "Weekly Follow-up",
  "description": "Alert for leads with no activity in 7 days",
  "is_active": true,
  "apply_to": "ALL_LEADS",
  "threshold_value": 7,
  "threshold_unit": "DAYS",
  "follow_up_hours": 2,
  "follow_up_unit": "HOURS"
}
```
</CodeGroup>

<Steps>
  <Step title="Create monitoring rule">
    Use POST endpoint with required trigger type, name, and threshold configuration
  </Step>
  <Step title="Configure escalation">
    Set follow-up timing and maximum team lead notifications before org-admin escalation
  </Step>
  <Step title="Activate monitoring">
    Set `is_active: true` to begin monitoring leads against the rule
  </Step>
</Steps>

## Business Rules

### Alert Generation

<Check>
- At most one open alert (PENDING or ESCALATED) per lead per rule
- Alerts created only for leads assigned to active agents
- Rules must be active to generate alerts
- Lead must match rule's apply_to criteria
</Check>

### Alert Lifecycle

The alert follows this progression through its lifecycle:

<Steps>
  <Step title="Alert Creation">
    New alerts start as **PENDING** status when monitoring thresholds are breached
  </Step>
  <Step title="Agent Response">
    Agent can acknowledge alert (requires reason) or escalate for help (optional description)
  </Step>
  <Step title="Follow-up Processing">
    Recurring team lead notifications based on follow-up interval and max_follow_ups setting
  </Step>
  <Step title="Resolution">
    Team lead or org admin can reassign lead or dismiss alert with snooze duration
  </Step>
</Steps>

<Tip>
When agents acknowledge alerts, they must provide a reason which is forwarded to the team lead as a notification.
</Tip>

### Threshold Evaluation

<Tabs>
  <Tab title="NO_ACTIVITY">
    Monitors for leads with no recorded activities within the specified threshold period
  </Tab>
  <Tab title="STALE_LEAD">
    Tracks leads that haven't been updated within the threshold timeframe
  </Tab>
  <Tab title="STAGE_STUCK">
    Identifies leads that remain in the same pipeline stage longer than the threshold
  </Tab>
</Tabs>

## Integration Points

### Notification Module

The Activity Monitor integrates with the notification system through event emission:

- **AGENT_ALERT** - Created when new alerts are generated
- **TEAM_LEAD_ALERTED** - Sent for escalations and recurring follow-ups  
- **AGENT_ACKNOWLEDGED** - Triggered when agent acknowledges with reason
- **ORG_ADMIN_ALERT** - Escalated after max team lead follow-ups

<Note>
All notification events include interactive action buttons for immediate response and resolution.
</Note>

The module handles user interactions through the `ActivityMonitorNotificationHandler`, enabling seamless workflow management directly from notifications.
