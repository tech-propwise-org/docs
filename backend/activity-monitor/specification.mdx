The Activity Monitor module provides automated monitoring and alerting for lead activity within the PropWise CRM. It tracks leads that become inactive, stale, or stuck in pipeline stages according to configurable rules, and generates alerts to agents and team leads to ensure timely follow-up actions.

## Overview

The module operates through a scheduled scanning system that evaluates leads against monitoring rules and creates alerts when thresholds are breached. It integrates with the notification system to deliver actionable alerts and supports escalation workflows when initial alerts are not acknowledged.

### Key Features

- **Rule-based monitoring** with configurable trigger types (NO_ACTIVITY, STALE_LEAD, STAGE_STUCK)
- **Agent notifications** with "I'm on it" and "Need help" action buttons
- **Team lead escalation** when agents don't respond within configured timeframes (recurring reminders, then optional org-admin escalation)
- **Team lead / org admin resolution** via **Reassign to Agent** (stakeholder sync) or **Dismiss** (reason + snooze), not distribution-engine redistribute
- **Snooze duration on rules** (`snooze_duration_value` / `snooze_duration_unit`) for post-acknowledge and post-dismiss suppression
- **Comprehensive audit trail** of all alert activities
- **Scheduled scanning** via pg-boss for reliable, distributed processing

### Architecture

The module follows a scanner-based architecture:
- **ActivityMonitorScanner** - Evaluates leads against active rules every 60 seconds
- **ActivityMonitorFollowUp** - Processes overdue alerts every 15 minutes (recurring team-lead reminders, then optional org-admin escalation)
- **ActivityMonitorScheduler** - Manages pg-boss job scheduling with per-org distribution
- **ActivityMonitorService** - Core business logic for alert management
- **ActivityMonitorNotificationHandler** - Handles user interactions with alert notifications

## Architecture

The module follows a rule-based monitoring architecture with the following key components:

- **Rules Engine**: Configurable rules that define monitoring conditions and thresholds
- **Scanner Service**: Periodic evaluation of leads against active rules  
- **Alert Management**: Creation, tracking, and lifecycle management of alerts
- **Notification Integration**: Delivery of alerts through the notification system
- **Follow-up System**: Recurring reminders to team leads, then one-time org-admin escalation per rule limits

### Design Decisions

<Info>
Key architectural decisions that shape the module's behavior:
</Info>

- **Scheduled Processing**: Uses pg-boss for reliable, distributed job scheduling
- **Tenant Isolation**: All monitoring operates within tenant context boundaries
- **Event-Driven Notifications**: Emits events for notification system integration
- **Unique Alert Constraints**: Partial unique index on open statuses (PENDING, ESCALATED) per lead+rule
- **Stateful Alert Tracking**: Maintains alert status throughout lifecycle

## Entities

### ActivityMonitorRule

Defines monitoring conditions and alert parameters for leads.

| Field | Type | Description |
|-------|------|-------------|
| `id` | UUID | Primary key |
| `organization_id` | UUID | Organization reference |
| `trigger_type` | MonitorTriggerType | Type of condition to monitor |
| `name` | String | Rule display name |
| `description` | String | Rule description |
| `is_active` | Boolean | Whether rule is enabled |
| `apply_to` | MonitorApplyTo | Which leads to monitor |
| `stage_ids` | UUID[] | Specific stages (when apply_to is SPECIFIC_STAGES) |
| `temperature_levels` | String[] | Lead temperatures to monitor |
| `threshold_value` | Integer | Threshold amount |
| `threshold_unit` | ThresholdUnit | Time unit for threshold (MINUTES, HOURS, DAYS) |
| `follow_up_hours` | Integer | Numeric follow-up value before escalation |
| `follow_up_unit` | ThresholdUnit | Time unit for follow-up (MINUTES, HOURS, DAYS); defaults to HOURS |
| `snooze_duration_value` | Integer | After acknowledge or dismiss, suppress new alerts for this duration (default 24) |
| `snooze_duration_unit` | ThresholdUnit | Unit for snooze duration (default HOURS) |
| `max_follow_ups` | Integer | Max team-lead follow-up notifications on the follow-up interval before one-time org-admin escalation (default 3) |
| `notify_team_lead` | Boolean | If false, follow-up job only stamps `follow_up_sent_at` (no TL/org notifications) |
| `created_at` | DateTime | Creation timestamp |
| `updated_at` | DateTime | Last update timestamp |

#### Enums

<Tabs>
<Tab title="MonitorTriggerType">

- `STALE_LEAD`: No updates to lead within threshold period  
- `STAGE_STUCK`: Lead in same stage longer than threshold
</Tab>

<Tab title="MonitorApplyTo">
- `ALL_LEADS`: Monitor all leads
- `SPECIFIC_STAGES`: Monitor only leads in specified stages
- `SPECIFIC_TEMPERATURES`: Monitor only leads with specified temperatures

<Tab title="ThresholdUnit">
- `MINUTES`: Time unit in minutes
- `HOURS`: Time unit in hours
- `DAYS`: Time unit in days

### ActivityMonitorAlert

Represents an alert generated when a lead breaches monitoring thresholds.

| Field | Type | Description |
|-------|------|-------------|
| `id` | UUID | Primary key |
| `organization_id` | UUID | Organization reference |
| `rule_id` | UUID | Monitoring rule reference |
| `lead_id` | UUID | Lead that triggered alert |
| `agent_user_id` | UUID | Lead's assigned agent |
| `team_lead_user_id` | UUID | Agent's team lead |
| `status` | AlertStatus | Current alert status |
| `triggered_at` | DateTime | When alert was created |
| `agent_notified_at` | DateTime | When agent was notified |
| `follow_up_sent_at` | DateTime | When follow-up was sent |
| `acknowledged_at` | DateTime | When alert was acknowledged |
| `escalated_at` | DateTime | When alert was escalated |
| `follow_up_count` | Integer | Count of follow-up notifications sent on the recurring schedule |
| `org_admin_notified_at` | DateTime | When org admins were notified after `max_follow_ups` TL reminders |
| `viewed_at` | DateTime | Optional: agent opened the in-app notification |
| `resolved_by_tl_at` | DateTime | When team lead/org admin dismissed the alert (`RESOLVED_BY_TL`) |
| `reassigned_to_user_id` | UUID | Agent selected when status is `REASSIGNED` |

#### Alert Status Enum

- `PENDING`: Initial status when alert is created
- `ACKNOWLEDGED`: Agent clicked "I'm on it"
- `ESCALATED`: Agent clicked "Need help"
- `IGNORED`: Edge cases such as no team lead when required
- `EXPIRED`: In-app notification expired
- `REDISTRIBUTED`: (deprecated)
- `REASSIGNED`: Lead reassigned to new agent
- `RESOLVED_BY_TL`: Team lead/org admin dismissed the alert

#### Constraints

<Warning>
Partial unique index `uq_activity_monitor_alert_open` on `(lead_id, rule_id)` **WHERE** `status IN ('PENDING','ESCALATED')` â€” at most one "open" alert per lead+rule (scanner insert uses the same predicate)
</Warning>

## API Endpoints

#### GET /activity-monitor/rules

Lists monitoring rules for the organization.

**Authorization:** `org-settings:read`

**Response:**
```typescript

  rules: ActivityMonitorRuleDto[]
```

Creates a new monitoring rule.

**Authorization:** `org-settings:write`

## Business Rules

### Alert Generation

<Steps>
<Step title="Constraint validation">
At most one **open** alert (PENDING or ESCALATED) per lead per rule at a time (partial unique index)
</Step>
<Step title="Agent validation">
Alerts are created only for leads assigned to active agents
</Step>
<Step title="Rule validation">
Rules must be active to generate alerts
</Step>
<Step title="Criteria matching">
Lead must match rule's apply_to criteria (stages, temperature)
</Step>
</Steps>

### Alert Lifecycle

The alert lifecycle follows a structured progression through various states:

<AccordionGroup>
<Accordion title="Initial Creation">
- New alerts start as **PENDING**
- Agent receives notification with action buttons
</Accordion>

<Accordion title="Agent Actions">
- **Acknowledge alert**: Agent must provide a reason via dialog â†’ status becomes **ACKNOWLEDGED**
  - Reason is forwarded to team lead as `ACTIVITY_MONITOR_ACKNOWLEDGE_ALERT` notification
  - `snooze_until` uses rule `snooze_duration_value` / `snooze_duration_unit`
- **Escalate alert**: Agent optionally provides help description â†’ status becomes **ESCALATED**
</Accordion>

<Accordion title="Follow-up Processing">
**ActivityMonitorFollowUp** runs on the follow-up interval:
- If `notify_team_lead` is false: only stamps `follow_up_sent_at`
- Otherwise: sends recurring **ACTIVITY_MONITOR_TEAM_LEAD_ALERT** notifications while `follow_up_count < max_follow_ups`
- Then one-time **ACTIVITY_MONITOR_ORG_ADMIN_ALERT** to org admins when `follow_up_count >= max_follow_ups` and `org_admin_notified_at` is null
</Accordion>

<Accordion title="Team Lead Resolution">
Team lead and org admin resolve via notification actions:
- **Reassign**: Primary agent replaced via `EntityStakeholderService.syncStakeholders` â†’ **REASSIGNED**
- **Dismiss**: Required reason provided â†’ **RESOLVED_BY_TL** with snooze from rule snooze fields
</Accordion>

<Accordion title="Edge Cases">
- **IGNORED**: For edge cases such as no team lead when required
- **EXPIRED**: In-app notification expiry calls `markAlertExpired`
</Accordion>
</AccordionGroup>

### Follow-up Rules

<Tip>
Follow-up processing uses sophisticated timing logic to ensure appropriate escalation:
</Tip>

- Candidates use `agent_notified_at` and the rule's follow-up interval (`follow_up_hours` Ă— `follow_up_unit`)
- Recurring TL notifications increment `follow_up_count`
- Org-admin wave fires once per alert when thresholds are met
- `follow_up_sent_at` is updated when a follow-up wave is processed (schedule advancement)
