The Lead Auto-Distribution & Escalation System automates intelligent lead routing and follow-up management through two interconnected engines: distribution for automatic lead assignment and escalation for handling inactive leads.

## Overview

This system handles two critical sales operations:

1. **Distribution**: Automatically assigns new leads to the most appropriate agents based on configurable rules
2. **Escalation**: Monitors lead activity and triggers notifications or reassignments when leads become stale

## System Architecture

The system consists of five core modules working together:

### Core Modules

<AccordionGroup>
  <Accordion title="Distribution Module">
    - **Path**: `src/modules/distribution/`
    - **Responsibility**: Lead assignment and routing
    - **Features**: Rules engine, agent availability, capacity management
    - **Triggers**: `LEAD_CREATED` events from CRM module
  </Accordion>

  <Accordion title="Escalation Module">
    - **Path**: `src/modules/escalation/`
    - **Responsibility**: Stale lead management and reassignment
    - **Features**: Multi-tier escalation, auto-resolution, comprehensive tracking
    - **Triggers**: Scheduled jobs and lead lifecycle events
  </Accordion>

  <Accordion title="Activity Monitor Module">
    - **Path**: `src/modules/activity-monitor/`
    - **Responsibility**: Agent notifications and team lead alerts
    - **Features**: Non-invasive notifications, snooze functionality, escalation chains
    - **Approach**: Notification-first rather than forced reassignment
  </Accordion>

  <Accordion title="PgBoss Module">
    - **Path**: `src/common/pgboss/`
    - **Responsibility**: Shared job queue and scheduling infrastructure
    - **Features**: Reliable background processing, retry logic, job monitoring
    - **Integration**: Used by all modules for async operations
  </Accordion>

  <Accordion title="Health Module">
    - **Path**: `src/modules/health/`
    - **Responsibility**: System monitoring and health checks
    - **Features**: Database connectivity, memory usage, pg-boss status monitoring
    - **Endpoints**: `/health/live`, `/health/ready` for container orchestration
  </Accordion>
</AccordionGroup>

## Design Principles

<Tabs>
  <Tab title="Async Processing">
    Lead creation emits `LEAD_CREATED` events processed by pg-boss workers. Distribution never blocks the primary lead creation flow.
  </Tab>
  
  <Tab title="Stakeholder Integration">
    Uses existing `EntityStakeholder` system for lead assignments rather than creating separate ownership paradigms.
  </Tab>
  
  <Tab title="Rule Priority">
    Distribution rules evaluated top-to-bottom by priority. First matching rule wins, preventing conflicts.
  </Tab>
  
  <Tab title="Idempotency">
    Distribution engine checks for existing assignments before processing to prevent duplicate stakeholder records.
  </Tab>
</Tabs>

## Key Implementation Details

### Event-Driven Processing

The system leverages comprehensive event handling:

```typescript
// Distribution triggered by lead creation
LEAD_CREATED → Distribution Engine → Agent Assignment

// Escalation triggered by activity monitoring  
LEAD_ACTIVITY_CREATED → Auto-resolution Check
LEAD_STAGE_CHANGED → Escalation Trigger Evaluation
```

### Resolution Tracking

<Info>
The escalation system includes detailed resolution tracking via the `ResolvedBy` enum for comprehensive audit trails.
</Info>

| Resolution Type | Description |
|---|---|
| `MANUAL` | User explicitly resolved via UI/API |
| `AUTO_ACTIVITY` | New activity added to lead |
| `AUTO_STAGE_CHANGE` | Lead moved to different stage |
| `AUTO_REASSIGNMENT` | Lead reassigned to different agent |
| `AUTO_ARCHIVED` | Lead archived (cancels trackers) |
| `REDISTRIBUTED` | Successful redistribution after escalation |

### Activity Monitor Integration

<Warning>
The Activity Monitor provides notification-based alerts as a gentler alternative to forced reassignment, complementing the Escalation Engine's more aggressive intervention strategies.
</Warning>

Key features include:
- **Agent Notifications**: Direct alerts with "I'm on it" and "Need help" actions
- **Team Lead Escalation**: Follow-up when agents don't respond
- **Snooze Functionality**: Prevents notification spam
- **Trigger Types**: `NO_ACTIVITY`, `STALE_LEAD`, `STAGE_STUCK`

## Analytics & Reporting

The system provides comprehensive analytics across three main areas:

### Distribution Analytics
- **Endpoint**: `GET /distribution-analytics`
- **Metrics**: Outcome breakdown, method analysis, rule effectiveness, per-agent performance
- **Constraints**: 90-day maximum date range

### Escalation Analytics  
- **Endpoint**: `GET /escalation-analytics` 
- **Metrics**: Trigger analysis, resolution tracking, rule frequency, timing metrics
- **Performance**: Raw SQL aggregations for optimal query speed

### Distribution Log
- **Endpoint**: `GET /distribution-log`
- **Features**: Paginated audit trail with filtering by rule, agent, lead, and outcome
- **Use Case**: Detailed forensic analysis and troubleshooting

<Tip>
All analytics endpoints require `VIEW_DISTRIBUTION_ANALYTICS` permission and use optimized SQL queries for high-performance data aggregation.
</Tip>
