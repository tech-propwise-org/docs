## Overview

The user-status module manages **business-level availability** for users within organizations. This is distinct from real-time presence (Session + WebSocket heartbeats) and instead drives operational behavior such as lead distribution eligibility, sidebar availability toggles, and staff profile availability display.

Users are **ONLINE** (available for work) or **AWAY** (unavailable), with explicit or automatic away reasons, optional custom notes, scheduled vacation windows, and per-user working-hours overrides.

## Entity Structure

### UserStatus

Per-user, per-organization availability record with the following structure:

<AccordionGroup>
  <Accordion title="Core Fields">
    | Field | Type | Description |
    | ----- | ---- | ----------- |
    | id | UUID | Primary key |
    | user | User | ManyToOne relationship |
    | organization | Organization | ManyToOne relationship |
    | status | UserStatusType | ONLINE or AWAY |
    | awayReason | AwayReason | Nullable; required semantically when AWAY |
    | customReasonText | string | Free text for CUSTOM away or vacation note (max 255 / 2000 for scheduled note) |
    | lastActiveAt | Date | Updated on API activity and explicit status changes |
  </Accordion>
  
  <Accordion title="Scheduled Away Fields">
    | Field | Type | Description |
    | ----- | ---- | ----------- |
    | scheduledAwayFrom | Date | Start of scheduled away window |
    | scheduledAwayUntil | Date | End of scheduled away window |
  </Accordion>
  
  <Accordion title="Working Hours Fields">
    | Field | Type | Description |
    | ----- | ---- | ----------- |
    | workingHoursTimezone | string | IANA timezone override |
    | workingHoursSchedule | JSONB | Weekly schedule entries `{ dayOfWeek, startTime, endTime }` |
  </Accordion>
  
  <Accordion title="Metadata Fields">
    | Field | Type | Description |
    | ----- | ---- | ----------- |
    | createdAt | Date | Record creation timestamp |
    | updatedAt | Date | Last modification timestamp |
  </Accordion>
</AccordionGroup>

<Note>
**Indexes:** unique `(user, organization)`; index on `(organization, status)`.

**Default when no row exists:** Distribution and scoped read APIs treat missing records as **ONLINE** (same as `findOrCreate` default on self endpoints).
</Note>

### Calendar Roster Endpoint

The `GET /user-status/calendar-roster` endpoint provides scheduled vacation windows for organization or team calendar views.

**Parameters:**
- `userIds` (query, required): Comma-separated subject user UUIDs (1–10, **excluding** the authenticated user)
- `teamId` (query, optional): When set, caller must have team calendar access

**Response:**
```typescript
{
  byUserId: Record<string, { scheduledAwayFrom: string; scheduledAwayUntil: string } | null>;
}
```

**Access Control:**
- Without `teamId`: Same gate as `GET /v1/calendar/org` (organization CRM/sales/admin roles only)
- With `teamId`: Same gate as `GET /v1/calendar/teams/:teamId` (org admin/owner or team visibility permissions)

## Enums and Types

### UserStatusType

<Tabs>
  <Tab title="Values">
    - `ONLINE` — Available for assignment / shown as online
    - `AWAY` — Unavailable for work
  </Tab>
</Tabs>

### AwayReason

| Value | Set by | Notes |
| ----- | ------ | ----- |
| `ON_LEAVE` | User / admin | Manual leave status |
| `CUSTOM` | User / admin | Requires `customReasonText` |
| `VACATION` | Scheduler / scheduled-away | Active vacation window |
| `INACTIVITY` | Scheduler | Auto after inactivity threshold |
| `OUTSIDE_HOURS` | Scheduler | Outside working hours |
| `SESSION_EXPIRED` | Session listener | All org sessions revoked/expired |

### UserStatusViewScope

<Note>
Used by `UserStatusService.resolveViewScope()` — not exposed on the wire.
</Note>

- `FULL` — All DTO fields including `customReasonText`
- `BASIC` — Availability fields only; `customReasonText` redacted in `UserStatusDto`

| Method | Path | Access | Description |
| ------ | ---- | ------ | ----------- |
| GET | `/organizations/:orgId/user-status` | `system.admin` | Paginated org list (staff table) |
| PUT | `/organizations/:orgId/users/:userId/status` | `system.admin` | Force status on any org member |

### Scoped Per-User Read

<Steps>
  <Step title="Endpoint">
    `GET /organizations/:orgId/users/:userId/status`
  </Step>
  <Step title="Guards">
    `AuthGuard`, `OrgMembershipGuard` (`:orgId` must match JWT org)
  </Step>
  <Step title="Authorization">
    Enforced in `UserStatusService.getForViewer()` - no `@CheckAccess` decorator
  </Step>
</Steps>

#### View Scope Rules

| Viewer | Scope | `customReasonText` |
| ------ | ----- | ------------------ |
| Self (`viewer.id === targetUserId`) | FULL | visible |
| `system.admin` | FULL | visible |
| `team.admin` on a team shared with target | FULL | visible |
| Any other active org member | BASIC | **omitted** |
| Target not in org / inactive user | 404 | — |
| Viewer not in org | 403 | — |

<Warning>
**Read behavior:** Does **not** call `findOrCreate`. Missing `user_status` row returns synthetic ONLINE via `UserStatusDto.forDefaultOnline()` (no `id` field).
</Warning>

#### Frontend Usage

<Tip>
Staff profile modal and other per-user surfaces should call this endpoint (`UserStatusApi.getByUser`) instead of loading the admin list to avoid over-fetching and to respect note redaction.
</Tip>

**Self-service scheduled away:** Personal settings → **Availability** tab and the sidebar profile menu **On Vacation** shortcut both use scheduled away endpoints. Quick **Set Away** / **Go Online** toggles remain profile-menu-only.

## Business Rules

### Manual vs Automatic Away

<Tabs>
  <Tab title="Manual Reasons">
    - `ON_LEAVE`, `CUSTOM`, `VACATION`
    - Persist across session restore unless user toggles online or admin overrides
  </Tab>
  <Tab title="Automatic Reasons">
    - `INACTIVITY`, `OUTSIDE_HOURS`, `SESSION_EXPIRED`
    - Cleared when user creates a new org session (`restoreOnlineIfAutoAway`)
  </Tab>
</Tabs>

### Distribution Integration

- **`getOnlineAgentIds()`** — Agents without a row or with `ONLINE` pass the pre-filter
- **`filterByWorkingHours()`** — Per-user schedule/timezone overrides fall back to org defaults

### Scheduled Jobs

<Check>
Cron handlers in `UserStatusScheduler`: inactivity marking, scheduled-away activation/expiry, working-hours enforcement.
</Check>

## Integration Points

The user status module integrates with several core systems:

<AccordionGroup>
  <Accordion title="Authentication & Sessions">
    - `UserStatusListener` on session create → `restoreOnlineIfAutoAway`
    - Session expiry → `setAwayOnSessionExpiry`
  </Accordion>
  
  <Accordion title="Activity Tracking">
    - `SessionActivityInterceptor` → `touchLastActive`
  </Accordion>
  
  <Accordion title="Distribution System">
    - `UserStatusService.getOnlineAgentIds`
    - `filterByWorkingHours`
  </Accordion>
  
  <Accordion title="Access Control">
    - `AccessService.hasPermission`
    - `getPermissionedTeamIdsFromUser` for scoped read
  </Accordion>
  
  <Accordion title="Scheduled Operations">
    - `UserStatusScheduler` handles inactivity detection
    - Working-hours enforcement
    - Scheduled-away activation/expiry
  </Accordion>
  
  <Accordion title="Database">
    - Tenant-scoped reads/writes use `TenantContext`
    - Missing status rows remain a soft default rather than hard database requirement
  </Accordion>
</AccordionGroup>
