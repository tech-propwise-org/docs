# User Status Module Specification

<Note>
Last Updated: 2024-12-19 | Status: Active
</Note>

## Overview

The User Status module manages business-level availability indicators for users within organizations. This system is distinct from real-time presence tracking (handled by the Session entity via WebSocket heartbeats) and focuses on deliberate availability states that affect business operations like lead assignment and customer service routing.

<Info>
The module tracks whether users are **ONLINE** (available for work) or **AWAY** (unavailable), along with specific reasons for away status.
</Info>

### Key Features

- **Manual Status Changes**: Users can explicitly set themselves away with reasons like breaks or custom messages
- **Automatic Status Tracking**: System automatically transitions users based on inactivity, working hours, session expiry, and scheduled leave
- **Event-Driven Updates**: Listens to session lifecycle events to automatically restore availability when appropriate
- **Scheduled Jobs**: Cron-based background processes enforce time-based business rules

## Entity Structure

### UserStatus Entity

The core entity representing per-user per-organization availability state.

| Field | Type | Description |
|-------|------|-------------|
| `id` | UUID | Primary key |
| `user` | User | ManyToOne relationship to User entity |
| `organization` | Organization | ManyToOne relationship to Organization entity |
| `status` | UserStatusType | Current status (ONLINE/AWAY) |
| `awayReason` | AwayReason | Reason for away status (nullable) |
| `customReason` | string | Free-text reason when awayReason is CUSTOM (max 500 chars) |
| `awayUntil` | Date | Optional end time for away status |
| `createdAt` | Date | Record creation timestamp |
| `updatedAt` | Date | Last modification timestamp |

<Note>
**Database Indexes:**
- Composite unique index on (user, organization)
- Index on (organization, status) for efficient filtering
</Note>

## Enums

### UserStatusType

```typescript
enum UserStatusType {
  ONLINE = 'ONLINE',  // User is available for work assignments
  AWAY = 'AWAY'       // User is unavailable
}
```

### AwayReason

<Tabs>
  <Tab title="Manual Reasons">
    - `ON_LEAVE`: User explicitly set themselves away
    - `CUSTOM`: User provided custom reason text
  </Tab>
  <Tab title="Automatic Reasons">
    - `VACATION`: Scheduled vacation period is active
    - `INACTIVITY`: No activity for 24+ hours (automatic)
    - `OUTSIDE_HOURS`: Outside configured working hours (automatic)
    - `SESSION_EXPIRED`: All user sessions expired or revoked (automatic)
  </Tab>
</Tabs>

### List Organization User Statuses

<CodeGroup>
```http GET

```

```typescript Response
{
  data: UserStatusResponseDto[],
  total: number,
  page: number,
```

## Business Rules

### Status Transitions

<Steps>
  <Step title="Default State">
    New users start as **ONLINE** when they join an organization
  </Step>
  <Step title="Manual Away">
    Users can set themselves away with any reason except automatic ones (INACTIVITY, OUTSIDE_HOURS, SESSION_EXPIRED)
  </Step>
  <Step title="Automatic Away">
    System sets users away for business rule violations
  </Step>
  <Step title="Restoration Priority">
    Manual reasons persist until manually changed; automatic reasons can be overridden by new sessions or manual updates
  </Step>
</Steps>

### Automatic Status Management

<AccordionGroup>
  <Accordion title="Inactivity Detection">
    Users are marked away after 24 hours of no activity. The system checks every 5 minutes using a scheduled job.
  </Accordion>
  
  <Accordion title="Working Hours Enforcement">
    Users outside configured hours are automatically set to away (if working hours are configured for the organization).
  </Accordion>
  
  <Accordion title="Session Management">
    When all user sessions expire or are revoked, users are automatically marked away with `SESSION_EXPIRED` reason.
  </Accordion>
  
  <Accordion title="Session Restoration">
    When users create new sessions, they're restored to ONLINE if previously away for automatic reasons only.
  </Accordion>
</AccordionGroup>

### Validation Rules

<Warning>
The following validation rules are enforced:

1. `customReason` is **required** when `awayReason` is CUSTOM
2. `customReason` is **forbidden** when `awayReason` is not CUSTOM
3. `awayReason` is **required** when status is AWAY
4. `awayReason` is **forbidden** when status is ONLINE
5. Users **cannot** manually set automatic away reasons (INACTIVITY, OUTSIDE_HOURS, SESSION_EXPIRED)
6. Custom reasons are limited to **500 characters**
</Warning>
