## Overview

The user-status module manages business-level availability indicators for users within organizations. This is distinct from real-time presence tracking (handled by the Session entity via WebSocket heartbeats) and instead focuses on deliberate availability states that affect business operations like lead assignment and customer service routing.

<Info>
The module tracks whether users are **ONLINE** (available for work) or **AWAY** (unavailable), along with specific reasons for away status.
</Info>

The system supports both manual status changes and automatic transitions based on business rules like working hours, inactivity periods, and scheduled leave.

## Architecture

The module follows standard PropWise patterns with clear separation between manual and automatic status management:

## Entity Structure

### UserStatus Entity

Represents per-user per-organization availability state.

| Field | Type | Description |
|-------|------|-------------|
| id | UUID | Primary key |
| user | User | ManyToOne relationship to User entity |
| organization | Organization | ManyToOne relationship to Organization entity |
| status | UserStatusType | Current status (ONLINE/AWAY) |
| awayReason | AwayReason | Reason for away status (nullable) |
| customReason | string | Free-text reason when awayReason is CUSTOM (max 500 chars) |
| awayUntil | Date | Optional end time for away status |
| createdAt | Date | Record creation timestamp |
| updatedAt | Date | Last modification timestamp |

<Note>
**Indexes:**
- Composite unique index on (user, organization)
- Index on (organization, status) for efficient filtering
</Note>

## API Endpoints

### List User Statuses

<CodeGroup>
```http GET
GET /user-status/:orgId
```
</CodeGroup>

List user statuses for an organization.

**Parameters:**
- `orgId` (path): Organization UUID
- `status` (query): Filter by UserStatusType
- `page` (query, default 1): Page number
- `limit` (query, default 50): Items per page

**Response:**
```typescript
{
  data: UserStatusResponseDto[],
  total: number,
  page: number,
  limit: number
}
```

<Warning>
**Access:** Requires MANAGE_TEAM or ASSIGN_LEADS permissions
</Warning>

### Bulk Update User Statuses

<CodeGroup>
```http PUT
PUT /user-status/:orgId/bulk
```
</CodeGroup>

Update multiple user statuses in a single request.

**Request Body:**
```typescript
{
  updates: [
    {
      userId: string,
      status: UserStatusType,
      awayReason?: AwayReason,
      customReason?: string,
      awayUntil?: Date
    }
  ]
}
```

**Response:** Array of updated UserStatusResponseDto objects

<Warning>
**Access:** Requires MANAGE_TEAM permission
</Warning>

### Update Current User Status

<CodeGroup>
```http PUT
PUT /user-status/:orgId/me
```
</CodeGroup>

Update current user's status.

**Request Body:**
```typescript
{
  status: UserStatusType,
  awayReason?: AwayReason,
  customReason?: string,
  awayUntil?: Date
}
```

**Response:** UserStatusResponseDto

<Info>
**Access:** User must be member of the organization
</Info>

### Remove User Status

<CodeGroup>
```http DELETE
DELETE /user-status/:orgId/:userId
```
</CodeGroup>

Remove user status record (sets user back to default ONLINE state).

<Warning>
**Access:** Requires MANAGE_TEAM permission
</Warning>

### UserStatusType

    User is unavailable

### AwayReason

<AccordionGroup>
  <Accordion title="Manual Reasons">
    - `ON_LEAVE`: User explicitly set themselves away
    - `CUSTOM`: User provided custom reason text
    - `VACATION`: Scheduled vacation period is active
  </Accordion>
  <Accordion title="Automatic Reasons">
    - `INACTIVITY`: No activity for 24+ hours (automatic)
    - `OUTSIDE_HOURS`: Outside configured working hours (automatic)
    - `SESSION_EXPIRED`: All user sessions expired or revoked (automatic)
  </Accordion>
</AccordionGroup>

## Business Rules

### Status Transitions

<Steps>
  <Step title="Default State">
    New users start as ONLINE when they join an organization
  </Step>
  <Step title="Manual Away">
    Users can set themselves away with any reason except automatic ones (INACTIVITY, OUTSIDE_HOURS, SESSION_EXPIRED)
  </Step>
  <Step title="Automatic Away">
    System sets users away for business rule violations
  </Step>
  <Step title="Restoration Priority">
    Manual reasons persist until manually changed; automatic reasons can be overridden by new sessions or manual updates
  </Step>
</Steps>

### Automatic Status Management

<CardGroup cols={2}>
  <Card title="Inactivity Detection" icon="clock">
    Users are marked away after 24 hours of no activity (checked every 5 minutes)
  </Card>
  <Card title="Working Hours" icon="calendar">
    Users outside configured hours are automatically set to away (if working hours are configured)
  </Card>
  <Card title="Session Management" icon="key">
    When all sessions expire or are revoked, users are marked away
  </Card>
  <Card title="Session Restoration" icon="refresh">
    When users create new sessions, they're restored to ONLINE if previously away for automatic reasons only
  </Card>
</CardGroup>

### Validation Rules

<Check>
✓ `customReason` is required when `awayReason` is CUSTOM
</Check>

<Check>
✓ `customReason` is forbidden when `awayReason` is not CUSTOM
</Check>

<Check>
✓ `awayReason` is required when status is AWAY
</Check>

<Check>
✓ `awayReason` is forbidden when status is ONLINE
</Check>

<Warning>
Users cannot manually set automatic away reasons (INACTIVITY, OUTSIDE_HOURS, SESSION_EXPIRED)
</Warning>

<Note>
Custom reasons are limited to 500 characters
</Note>
