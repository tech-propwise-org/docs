The User Status module manages business-level availability indicators for users within organizations, providing essential functionality for lead assignment and customer service routing operations.

<Note>
This module is distinct from real-time presence tracking (handled by the Session entity via WebSocket heartbeats) and instead focuses on deliberate availability states that affect business operations.
</Note>

## Overview

The user-status module tracks whether users are ONLINE (available for work) or AWAY (unavailable), along with specific reasons for away status. It supports both manual status changes and automatic transitions based on business rules.

### Key Features

- **Manual Status Changes**: Users can explicitly set themselves away with reasons like breaks or custom messages
- **Automatic Status Tracking**: System automatically transitions users based on inactivity, working hours, session expiry, and scheduled vacation windows
- **Event-Driven Updates**: Listens to session lifecycle events to automatically restore availability
- **Scheduled Jobs**: Cron-based background processes enforce time-based business rules

## Entity Design

### UserStatus

The core entity that represents per-user per-organization availability state.

| Field | Type | Description |
|-------|------|-------------|
| id | UUID | Primary key |
| user | User | ManyToOne relationship to User entity |
| organization | Organization | ManyToOne relationship to Organization entity |
| status | UserStatusType | Current status (ONLINE/AWAY) |
| awayReason | AwayReason | Reason for away status (nullable) |
| customReason | string | Free-text reason when awayReason is CUSTOM (max 500 chars) |
| awayUntil | Date | Optional end time for away status |
| createdAt | Date | Record creation timestamp |
| updatedAt | Date | Last modification timestamp |

<Info>
The entity uses a composite unique index on (user, organization) and an index on (organization, status) for efficient filtering operations.
</Info>

## API Endpoints

### List User Statuses

<CodeGroup>
```http GET
GET /user-status/:orgId
```

```typescript Parameters
{
  orgId: string,     // Organization UUID (path)
  status?: string,   // Filter by UserStatusType (query)
  page?: number,     // Page number, default 1 (query)
  limit?: number     // Items per page, default 50 (query)
}
```

```typescript Response
{
  data: UserStatusResponseDto[],
  total: number,
  page: number,
  limit: number
}
```
</CodeGroup>

<Check>
**Access:** Requires MANAGE_TEAM or ASSIGN_LEADS permissions
</Check>

### Bulk Update Statuses

<CodeGroup>


```typescript Request

  updates: [
    {

      awayReason?: AwayReason,

```

```typescript Response
UserStatusResponseDto[]
```

</CodeGroup>

<Check>

</Check>

### Update Own Status

<CodeGroup>
```http PUT
PUT /user-status/:orgId/me
```

```typescript Request
{
  status: UserStatusType,
  awayReason?: AwayReason,
  customReason?: string,
  awayUntil?: Date
}
```

```typescript Response
UserStatusResponseDto
```
</CodeGroup>

**Access:** User must be member of the organization
</Check>

### Remove User Status

```http
DELETE /user-status/:orgId/:userId
```

Sets user back to default ONLINE state by removing their status record.

<Check>
**Access:** Requires MANAGE_TEAM permission
</Check>

### AwayReason

<AccordionGroup>
  <Accordion title="Manual Reasons">
    - `ON_LEAVE`: User explicitly set themselves away
    - `CUSTOM`: User provided custom reason text
    - `VACATION`: Scheduled vacation period is active
  </Accordion>
  
  <Accordion title="Automatic Reasons">
    - `INACTIVITY`: No activity for 24+ hours (automatic)
    - `OUTSIDE_HOURS`: Outside configured working hours (automatic)
    - `SESSION_EXPIRED`: All user sessions expired or revoked (automatic)
  </Accordion>
</AccordionGroup>

## Business Rules

### Status Transitions

<Steps>
  <Step title="Default State">
    New users start as ONLINE when they join an organization
  </Step>
  
  <Step title="Manual Away">
    Users can set themselves away with any reason except automatic ones (INACTIVITY, OUTSIDE_HOURS, SESSION_EXPIRED)
  </Step>
  
  <Step title="Automatic Away">
    System sets users away for business rule violations
  </Step>
  
  <Step title="Restoration Priority">
    Manual reasons persist until manually changed; automatic reasons can be overridden by new sessions or manual updates
  </Step>
</Steps>

### Automatic Status Management

<CardGroup cols={2}>
  <Card title="Inactivity Detection" icon="clock">
    Users are marked away after 24 hours of no activity (checked every 5 minutes)
  </Card>
  
  <Card title="Working Hours" icon="calendar">
    Users outside configured hours are automatically set to away
  </Card>
  
  <Card title="Session Management" icon="sign-out">
    When all sessions expire or are revoked, users are marked away
  </Card>
  
  <Card title="Session Restoration" icon="sign-in">
    New sessions restore users to ONLINE if previously away for automatic reasons only
  </Card>
</CardGroup>

### Validation Rules

<Warning>
The following validation rules are strictly enforced:
</Warning>

1. `customReason` is required when `awayReason` is CUSTOM
2. `customReason` is forbidden when `awayReason` is not CUSTOM  
3. `awayReason` is required when status is AWAY
4. `awayReason` is forbidden when status is ONLINE
5. Users cannot manually set automatic away reasons
6. Custom reasons are limited to 500 characters

The module uses NestJS scheduler for automated status management:

```typescript
@Cron(CronExpression.EVERY_5_MINUTES)
async processStatusUpdates() {
  // Inactivity detection
  // Working hours enforcement
  // Vacation window management
}
```

<Tip>
Scheduled jobs run every 5 minutes to ensure timely status updates while minimizing system load.
</Tip>
