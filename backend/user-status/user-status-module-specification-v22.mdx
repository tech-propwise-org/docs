## Overview

The user-status module manages **business-level availability** for users within organizations. This is distinct from real-time presence (Session + WebSocket heartbeats) and instead drives operational behavior such as lead distribution eligibility, sidebar availability toggles, and staff profile availability display.

Users are **ONLINE** (available for work) or **AWAY** (unavailable), with explicit or automatic away reasons, optional custom notes, scheduled vacation windows, and per-user working-hours overrides.

## Entity Structure

### UserStatus

Per-user, per-organization availability record with the following fields:

<AccordionGroup>
<Accordion title="Core Fields">
| Field | Type | Description |
|-------|------|-------------|
| id | UUID | Primary key |
| user | User | ManyToOne relationship |
| organization | Organization | ManyToOne relationship |
| status | UserStatusType | ONLINE or AWAY |
| awayReason | AwayReason | Nullable; required semantically when AWAY |
| customReasonText | string | Free text for CUSTOM away or vacation note (max 255 / 2000 for scheduled note) |
| lastActiveAt | Date | Updated on API activity and explicit status changes |
</Accordion>

<Accordion title="Scheduled Away Fields">
| Field | Type | Description |
|-------|------|-------------|
| scheduledAwayFrom | Date | Start of scheduled away window |
| scheduledAwayUntil | Date | End of scheduled away window |
</Accordion>

<Accordion title="Working Hours Fields">
| Field | Type | Description |
|-------|------|-------------|
| workingHoursTimezone | string | IANA timezone override |
| workingHoursSchedule | JSONB | Weekly schedule entries `{ dayOfWeek, startTime, endTime }` |
</Accordion>

<Accordion title="System Fields">
| Field | Type | Description |
|-------|------|-------------|
| createdAt | Date | Record creation timestamp |
| updatedAt | Date | Last modification timestamp |
</Accordion>
</AccordionGroup>

**Database Indexes:**
- Unique constraint on `(user, organization)`
- Index on `(organization, status)`

<Info>
**Default Behavior:** When no record exists, distribution and scoped read APIs treat missing records as **ONLINE** (same as `findOrCreate` default on self endpoints).
</Info>

### Calendar Roster Endpoint

The module provides a special endpoint for calendar integration:

#### GET /user-status/calendar-roster

Returns scheduled vacation windows (`scheduledAwayFrom` / `scheduledAwayUntil`) for users shown on **organization** or **team** stacked calendars (red diagonal stripe overlay in the web app).

**Parameters:**
- `userIds` (query, required): Comma-separated subject user UUIDs (1–10, **excluding** the authenticated user)
- `teamId` (query, optional): When set, caller must have **team calendar** access for that org team and every subject must be a member of that team

**Response:**
```typescript
{
  byUserId: Record<string, { scheduledAwayFrom: string; scheduledAwayUntil: string } | null>;
}
```

**Access Control:**
- Without `teamId`: Same gate as `GET /v1/calendar/org` (organization CRM/sales/admin roles only)
- With `teamId`: Same gate as `GET /v1/calendar/teams/:teamId` (org admin/owner or team visibility permissions)

### UserStatusViewScope

Used by `UserStatusService.resolveViewScope()` for internal service operations (not exposed on the wire).

- `FULL` — All DTO fields including `customReasonText`
- `BASIC` — Availability fields only; `customReasonText` redacted in `UserStatusDto`

### Scoped Per-User Read Access

The scoped read endpoint uses `AuthGuard` and `OrgMembershipGuard` with authorization enforced in `UserStatusService.getForViewer()`.

#### View Scope Rules

<Steps>
<Step title="Determine Viewer Relationship">
The system evaluates the relationship between the viewer and target user.
</Step>

<Step title="Apply Scope Rules">
| Viewer Type | Scope | `customReasonText` Visibility |
|-------------|-------|------------------------------|
| Self (`viewer.id === targetUserId`) | FULL | ✅ Visible |
| `system.admin` | FULL | ✅ Visible |
| `team.admin` on shared team | FULL | ✅ Visible |
| Other active org member | BASIC | ❌ Omitted |
| Target not in org / inactive | 404 | — |
| Viewer not in org | 403 | — |
</Step>

<Step title="Return Response">
Missing `user_status` records return synthetic ONLINE via `UserStatusDto.forDefaultOnline()` (no `id` field).
</Step>
</Steps>

<Warning>
The scoped read endpoint does **not** call `findOrCreate`. Privacy is applied at serialization time via `redactCustomReasonText`.
</Warning>

#### Frontend Integration

<Tip>
Staff profile modals and other per-user surfaces should call the scoped endpoint (`UserStatusApi.getByUser`) instead of loading the admin list to avoid over-fetching and respect note redaction.
</Tip>

**Self-Service Features:**
- Personal settings → **Availability** tab (`AccountScheduledAwaySection`)
- Sidebar profile menu **On Vacation** shortcut
- Both use shared `ScheduledAwayDialog` for scheduled away management
- Quick **Set Away** / **Go Online** toggles in profile menu
- Going online clears active or upcoming scheduled windows

## Business Rules

### Manual vs Automatic Away States

<Tabs>
<Tab title="Manual Reasons">
**Persistent across sessions:**
- `ON_LEAVE`
- `CUSTOM` 
- `VACATION`

These persist across session restore unless the user toggles online or an admin overrides the status.
</Tab>

<Tab title="Automatic Reasons">
**Cleared on session creation:**
- `INACTIVITY`
- `OUTSIDE_HOURS`
- `SESSION_EXPIRED`

These are automatically cleared when the user creates a new org session via `restoreOnlineIfAutoAway`.
</Tab>
</Tabs>

## Integration Points

The User Status module integrates with multiple system components:

<AccordionGroup>

- Updates `lastActiveAt` on API activity
</Accordion>

<Accordion title="Distribution System">
- **`UserStatusService.getOnlineAgentIds`** for agent filtering
- **`filterByWorkingHours`** for schedule-based filtering
</Accordion>

<Accordion title="Access Control">
- **`AccessService.hasPermission`** for permission checks
- **`getPermissionedTeamIdsFromUser`** for scoped read access
</Accordion>

<Accordion title="Scheduled Operations">
- **`UserStatusScheduler`** handles:
  - Inactivity detection
  - Working hours enforcement  
  - Scheduled away activation/expiry
</Accordion>

<Accordion title="Database Operations">
- **Tenant-scoped** reads/writes use `TenantContext`
- **Soft defaults** for missing status rows rather than hard database requirements
</Accordion>
</AccordionGroup>
