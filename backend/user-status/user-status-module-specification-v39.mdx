## Overview

The user-status module manages **business-level availability** for users within organizations. This is distinct from real-time presence (Session + WebSocket heartbeats) and instead drives operational behavior such as lead distribution eligibility, sidebar availability toggles, and staff profile availability display.

Users are **ONLINE** (available for work) or **AWAY** (unavailable), with explicit or automatic away reasons, optional custom notes, scheduled vacation windows, and per-user working-hours overrides.

## Entity

### UserStatus

Per-user, per-organization availability record.

| Field                | Type           | Description                                                                    |
| -------------------- | -------------- | ------------------------------------------------------------------------------ |
| id                   | UUID           | Primary key                                                                    |
| user                 | User           | ManyToOne                                                                      |
| organization         | Organization   | ManyToOne                                                                      |
| status               | UserStatusType | ONLINE or AWAY                                                                 |
| awayReason           | AwayReason     | Nullable; required semantically when AWAY                                      |
| customReasonText     | string         | Free text for CUSTOM away or vacation note (max 255 / 2000 for scheduled note) |
| lastActiveAt         | Date           | Updated on API activity and explicit status changes                            |
| scheduledAwayFrom    | Date           | Start of scheduled away window                                                 |
| scheduledAwayUntil   | Date           | End of scheduled away window                                                   |
| workingHoursTimezone | string         | IANA timezone override                                                         |
| workingHoursSchedule | JSONB          | Weekly schedule entries `{ dayOfWeek, startTime, endTime }`                    |
| createdAt            | Date           | Record creation                                                                |
| updatedAt            | Date           | Last modification                                                              |

<Info>
**Indexes:** unique `(user, organization)`; index on `(organization, status)`.

**Default when no row exists:** Distribution and scoped read APIs treat missing records as **ONLINE** (same as `findOrCreate` default on self endpoints).
</Info>

### GET /user-status/calendar-roster

Scheduled vacation windows (`scheduledAwayFrom` / `scheduledAwayUntil`) for users shown on **organization** or **team** stacked calendars (red diagonal stripe overlay in the web app).

#### Parameters

- `userIds` (query, required): Comma-separated subject user UUIDs (1–10, **excluding** the authenticated user — same cap as `GET /v1/calendar/org` `userIds`)
- `teamId` (query, optional): When set, caller must have **team calendar** access for that org team and every subject must be a member of that team

#### Response

```typescript
{
  byUserId: Record<string, { scheduledAwayFrom: string; scheduledAwayUntil: string } | null>;
}
```

#### Access control

- Without `teamId`: same gate as **`GET /v1/calendar/org`** (`AccessService.assertOrgCalendarViewer` — organization CRM/sales/admin roles only)
- With `teamId`: same gate as **`GET /v1/calendar/teams/:teamId`** (org admin/owner or team visibility permissions on that team)

<Check>
**Implementation:** `UserStatusService.getCalendarRosterScheduledAway`, constant `CALENDAR_ROSTER_SCHEDULED_AWAY_MAX_USER_IDS` (aligned with `ORG_ROSTER_CALENDAR_MAX_USER_IDS`).
</Check>

## Enums

### UserStatusType

- `ONLINE` — available for assignment / shown as online
- `AWAY` — unavailable

### AwayReason

| Value             | Set by                     | Notes                            |
| ----------------- | -------------------------- | -------------------------------- |
| `ON_LEAVE`        | User / admin               | Manual                           |
| `CUSTOM`          | User / admin               | Requires `customReasonText`      |
| `VACATION`        | Scheduler / scheduled-away | Active vacation window           |
| `INACTIVITY`      | Scheduler                  | Auto after inactivity threshold  |
| `OUTSIDE_HOURS`   | Scheduler                  | Outside working hours            |
| `SESSION_EXPIRED` | Session listener           | All org sessions revoked/expired |

### UserStatusViewScope (service-layer)

<Note>
Used by `UserStatusService.resolveViewScope()` — not exposed on the wire.
</Note>

- `FULL` — all DTO fields including `customReasonText`
- `BASIC` — availability fields only; `customReasonText` redacted in `UserStatusDto`

## API Endpoints

All routes are under `/v1` and require JWT auth unless noted.

### Self endpoints (tenant JWT org context)

| Method | Path                          | Description                                     |
| ------ | ----------------------------- | ----------------------------------------------- |
| GET    | `/user-status/me`             | Own status; **creates** row if missing          |
| PUT    | `/user-status/online`         | Set self ONLINE; clears away + scheduled window |
| PUT    | `/user-status/away`           | Set self AWAY with reason                       |
| PUT    | `/user-status/scheduled-away` | Schedule vacation/leave window                  |
| DELETE | `/user-status/scheduled-away` | Clear scheduled window                          |
| PUT    | `/user-status/working-hours`  | Personal timezone + schedule overrides          |

### Admin list / override

| Method | Path                                         | Access                                            |
| ------ | -------------------------------------------- | ------------------------------------------------- |
| GET    | `/organizations/:orgId/user-status`          | `system.admin` — paginated org list (staff table) |
| PUT    | `/organizations/:orgId/users/:userId/status` | `system.admin` — force status on any org member   |

### Scoped per-user read

<Tip>
This is the recommended endpoint for frontend surfaces like staff profile modals, as it respects privacy controls and avoids over-fetching.
</Tip>

| Method | Path                                         | Access                                      |
| ------ | -------------------------------------------- | ------------------------------------------- |
| GET    | `/organizations/:orgId/users/:userId/status` | Org membership + scoped privacy (see below) |

**Guards:** `AuthGuard`, `OrgMembershipGuard` (`:orgId` must match JWT org). No `@CheckAccess` — authorization is enforced in `UserStatusService.getForViewer()`.

#### View scope rules

| Viewer                                    | Scope | `customReasonText` |
| ----------------------------------------- | ----- | ------------------ |
| Self (`viewer.id === targetUserId`)       | FULL  | visible            |
| `system.admin`                            | FULL  | visible            |
| `team.admin` on a team shared with target | FULL  | visible            |
| Any other active org member               | BASIC | **omitted**        |
| Target not in org / inactive user         | 404   | —                  |
| Viewer not in org                         | 403   | —                  |

<Warning>
**Read behavior:** Does **not** call `findOrCreate`. Missing `user_status` row returns synthetic ONLINE via `UserStatusDto.forDefaultOnline()` (no `id` field).
</Warning>

**Response:** `UserStatusDto` — same shape as self/admin endpoints; privacy is applied at serialization time via `redactCustomReasonText`.

#### Frontend usage

Staff profile modal and other per-user surfaces should call this endpoint (`UserStatusApi.getByUser`) instead of loading the admin list (`GET /organizations/:orgId/user-status`) to avoid over-fetching and to respect note redaction.

<Info>
**Self-service scheduled away:** Personal settings → **Availability** tab (`AccountScheduledAwaySection`) and the sidebar profile menu **On Vacation** shortcut both use `PUT /user-status/scheduled-away` and `DELETE /user-status/scheduled-away` via shared `ScheduledAwayDialog`. Quick **Set Away** / **Go Online** toggles remain profile-menu-only (`PUT /user-status/away` / `PUT /user-status/online`); going online clears an active or upcoming scheduled window.
</Info>

## Business Rules

### Manual vs automatic away

<Tabs>
  <Tab title="Manual reasons">
    Manual reasons (`ON_LEAVE`, `CUSTOM`, `VACATION`) persist across session restore unless user toggles online or admin overrides.
  </Tab>
  <Tab title="Automatic reasons">
    Automatic reasons (`INACTIVITY`, `OUTSIDE_HOURS`, `SESSION_EXPIRED`) are cleared when the user creates a new org session (`restoreOnlineIfAutoAway`).
  </Tab>
</Tabs>

### Distribution integration

- `getOnlineAgentIds()` — agents without a row or with `ONLINE` pass the pre-filter.
- `filterByWorkingHours()` — per-user schedule/timezone overrides fall back to org defaults.

### Scheduled jobs

Cron handlers (in `UserStatusScheduler`): inactivity marking, scheduled-away activation/expiry, working-hours enforcement.

## Key Implementation Notes

<Steps>
  <Step title="Status record initialization">
    Self endpoints (`GET /user-status/me`) automatically create a status record if one doesn't exist. Other endpoints return synthetic ONLINE status for missing records.
  </Step>
  
  <Step title="Privacy enforcement">
    The scoped per-user read endpoint applies view scope rules at the service layer, redacting `customReasonText` for non-privileged viewers.
  </Step>
  
  <Step title="Scheduled away windows">
    Calendar roster endpoint provides vacation overlay data with appropriate access controls based on organization or team calendar permissions.
  </Step>
  
  <Step title="Working hours integration">
    Per-user timezone and schedule overrides integrate with distribution eligibility filtering, with fallback to organization defaults.
  </Step>
</Steps>

<Check>
The module maintains a clear separation between business-level availability (user-status) and real-time presence (WebSocket/session heartbeats), ensuring predictable behavior across distribution, UI display, and scheduled automation.
</Check>
