## Overview

The user-status module manages **business-level availability** for users within organizations. This is distinct from real-time presence (Session + WebSocket heartbeats) and instead drives operational behavior such as lead distribution eligibility, sidebar availability toggles, and staff profile availability display.

Users are **ONLINE** (available for work) or **AWAY** (unavailable), with explicit or automatic away reasons, optional custom notes, scheduled vacation windows, and per-user working-hours overrides.

## Entity Structure

### UserStatus

Per-user, per-organization availability record with the following schema:

| Field                | Type           | Description                                                                    |
| -------------------- | -------------- | ------------------------------------------------------------------------------ |
| id                   | UUID           | Primary key                                                                    |
| user                 | User           | ManyToOne                                                                      |
| organization         | Organization   | ManyToOne                                                                      |
| status               | UserStatusType | ONLINE or AWAY                                                                 |
| awayReason           | AwayReason     | Nullable; required semantically when AWAY                                      |
| customReasonText     | string         | Free text for CUSTOM away or vacation note (max 255 / 2000 for scheduled note) |
| lastActiveAt         | Date           | Updated on API activity and explicit status changes                            |
| scheduledAwayFrom    | Date           | Start of scheduled away window                                                 |
| scheduledAwayUntil   | Date           | End of scheduled away window                                                   |
| workingHoursTimezone | string         | IANA timezone override                                                         |
| workingHoursSchedule | JSONB          | Weekly schedule entries `{ dayOfWeek, startTime, endTime }`                    |
| createdAt            | Date           | Record creation                                                                |
| updatedAt            | Date           | Last modification                                                              |

<Note>
**Database Indexes:** Unique constraint on `(user, organization)` and index on `(organization, status)`.

**Default Behavior:** When no row exists, distribution and scoped read APIs treat missing records as **ONLINE** (same as `findOrCreate` default on self endpoints).
</Note>

### Calendar Roster Integration

The calendar roster endpoint provides scheduled vacation windows for organization and team calendar views.

**Endpoint:** `GET /user-status/calendar-roster`

**Parameters:**
- `userIds` (query, required): Comma-separated subject user UUIDs (1‚Äì10, **excluding** the authenticated user)
- `teamId` (query, optional): When set, caller must have team calendar access

**Response:**
```json
{
  "byUserId": {
    "user-uuid-1": {
      "scheduledAwayFrom": "2026-07-01T00:00:00Z",
      "scheduledAwayUntil": "2026-07-15T23:59:59Z"
    },
    "user-uuid-2": null
  }
}
```

**Access Control:**
- Without `teamId`: Organization CRM/sales/admin roles only
- With `teamId`: Organization admin/owner or team visibility permissions

### AwayReason

| Value             | Set by                     | Notes                            |
| ----------------- | -------------------------- | -------------------------------- |
| `ON_LEAVE`        | User / admin               | Manual                           |
| `CUSTOM`          | User / admin               | Requires `customReasonText`      |
| `VACATION`        | Scheduler / scheduled-away | Active vacation window           |
| `INACTIVITY`      | Scheduler                  | Auto after inactivity threshold  |
| `OUTSIDE_HOURS`   | Scheduler                  | Outside working hours            |
| `SESSION_EXPIRED` | Session listener           | All org sessions revoked/expired |

### UserStatusViewScope

Used internally by `UserStatusService.resolveViewScope()` for privacy controls:

- `FULL` ‚Äî All DTO fields including `customReasonText`
- `BASIC` ‚Äî Availability fields only; `customReasonText` redacted

## API Endpoints

All routes are under `/v1` and require JWT authentication unless noted.

### Self-Service Endpoints

<AccordionGroup>
  <Accordion title="GET /user-status/me">
    Get own status. **Creates** row if missing.
    
    **Access:** Tenant JWT org context required
    
    **Response:** Complete `UserStatusDto` with all fields visible
  </Accordion>

  <Accordion title="PUT /user-status/online">
    Set self ONLINE. Clears away reason and scheduled window.
    
    **Body:** None required
    
    **Effect:** Removes any active or upcoming scheduled away periods
  </Accordion>

  <Accordion title="PUT /user-status/away">
    Set self AWAY with reason.
    
    **Body:**
    ```json
    {
      "awayReason": "ON_LEAVE",
      "customReasonText": "Out for training"
    }
    ```
  </Accordion>

  <Accordion title="PUT /user-status/scheduled-away">
    Schedule vacation/leave window.
    
    **Body:**
    ```json
    {
      "scheduledAwayFrom": "2026-07-01T00:00:00Z",
      "scheduledAwayUntil": "2026-07-15T23:59:59Z",
      "customReasonText": "Annual vacation"
    }
    ```
  </Accordion>

  <Accordion title="DELETE /user-status/scheduled-away">
    Clear scheduled window.
    
    **Body:** None
    
    **Effect:** Removes future scheduled away periods
  </Accordion>

  <Accordion title="PUT /user-status/working-hours">
    Personal timezone and schedule overrides.
    
    **Body:**
    ```json
    {
      "workingHoursTimezone": "America/New_York",
      "workingHoursSchedule": [
        {
          "dayOfWeek": 1,
          "startTime": "09:00",
          "endTime": "17:00"
        }
      ]
    }
    ```
  </Accordion>
</AccordionGroup>

### Scoped User Status Read

**Endpoint:** `GET /organizations/:orgId/users/:userId/status`

**Guards:** `AuthGuard`, `OrgMembershipGuard` (`:orgId` must match JWT org)

**Authorization:** Enforced in `UserStatusService.getForViewer()` with privacy scoping

#### View Scope Rules

<Steps>
  <Step title="Self Access">
    When `viewer.id === targetUserId`: **FULL** scope with `customReasonText` visible
  </Step>
  
  <Step title="System Admin">
    Users with `system.admin` role: **FULL** scope with all fields visible
  </Step>
  
  <Step title="Team Admin">
    Users with `team.admin` on shared team with target: **FULL** scope
  </Step>
  
  <Step title="Organization Member">
    Any other active org member: **BASIC** scope with `customReasonText` omitted
  </Step>
  
  <Step title="Access Denied">
    - Target not in org or inactive: **404**
    - Viewer not in org: **403**
  </Step>
</Steps>

<Warning>
**Read Behavior:** Does **not** call `findOrCreate`. Missing `user_status` row returns synthetic ONLINE via `UserStatusDto.forDefaultOnline()` (no `id` field).
</Warning>

## Business Rules

### Manual vs Automatic Away States

<Tabs>
  <Tab title="Manual Reasons">
    **Persist across sessions:**
    - `ON_LEAVE`
    - `CUSTOM` 
    - `VACATION`
    
    These remain until user toggles online or admin overrides.
  </Tab>
  
  <Tab title="Automatic Reasons">
    **Cleared on new session:**
    - `INACTIVITY`
    - `OUTSIDE_HOURS`
    - `SESSION_EXPIRED`
    
    These are cleared when user creates new org session via `restoreOnlineIfAutoAway`.
  </Tab>
</Tabs>

### Distribution Integration

<Check>
**Agent Filtering:** `getOnlineAgentIds()` includes agents without a row or with `ONLINE` status in the pre-filter for lead distribution.
</Check>

<Check>
**Working Hours:** `filterByWorkingHours()` uses per-user schedule/timezone overrides, falling back to organization defaults.
</Check>

### Scheduled Jobs

The `UserStatusScheduler` handles automated status management:

- **Inactivity marking** - Auto-away after threshold
- **Scheduled-away activation/expiry** - Vacation window management  
- **Working-hours enforcement** - Outside hours detection
