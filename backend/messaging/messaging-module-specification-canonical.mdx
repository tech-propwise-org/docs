## Overview

The Messaging module provides a unified, channel-agnostic messaging system for WhatsApp, Instagram, and Facebook Messenger. It replaces the separate per-channel modules with shared entities, a shared queue, and a single WebSocket namespace.

### Problem → Solution

| Problem | Solution |
|---------|----------|
| Duplicated logic across WhatsApp and Instagram modules | Single `MessagingModule` with channel providers |
| No webhook signature validation (security gap) | Shared `MetaWebhookGuard` validates `X-Hub-Signature-256` |
| Inconsistent WebSocket auth (Instagram gateway has no JWT) | Single `/messaging` gateway with JWT auth |
| No Facebook Messenger support | Third channel provider |
| Separate entity schemas per channel | Unified entities: `Conversation`, `Message`, `ChannelAccount` |
| No shared queue infrastructure | Shared `PgBossQueueService` for messaging + notifications |

### Key Design Decisions

<AccordionGroup>
<Accordion title="Queue Infrastructure & Database Design">
**pg-boss over BullMQ** — Project already uses pg-boss for notifications. No new Redis dependency. Interface-based design (`IQueueService`) allows swapping later.

**Direct PersonChannel FK on Conversation** — Conversations link directly to the CRM's `PersonChannel` via FK. Simpler model, no bidirectional sync overhead. The lead FK was moved from Conversation to Lead (`Lead.sourceConversation`).

**Archive as boolean, not status** — `Conversation.isArchived` is orthogonal to `status` (OPEN/CLOSED), following `ARCHIVE_SYSTEM_SPECIFICATION.md`.
</Accordion>

<Accordion title="Assignment & Permissions">
**`ConversationAssignment` entity** — Conversations use a dedicated `conversation_assignment` table instead of the CRM `entity_stakeholder` pattern. Each assignment is one row with nullable `user_id` and `team_id`:
- `user + null` = direct assignment
- `user + team` = agent on behalf of team  
- `null + team` = team pool

**`ResourcePermissionsDto` for conversations** — Conversations return per-resource permissions following the project-wide CRM pattern. `ConversationPermissionService` computes permissions in-memory — no extra DB queries.
</Accordion>

<Accordion title="AI & Templates">
**Per-conversation AI mode with cascade** — Each conversation has an `aiMode` field (OFF, AUTO_REPLY, SUGGEST_ONLY, DRAFT). Default cascades: ChannelAccount.defaultAiMode → Organization default → OFF.

**Three-tier template system** — `MessageTemplate` supports three types: `META_APPROVED` (platform-approved), `QUICK_REPLY` (agent shortcuts with variable resolution), and `AI_PROMPT` (AI system prompts with optional SystemPrompt link).
</Accordion>

<Accordion title="OAuth & Security">
**Personal accounts share org WABA token** — WhatsApp personal accounts reuse the organization's WABA access token. Instagram and Messenger personal accounts use their own Page Access Token obtained via OAuth.

**OAuth state includes `level` for defense-in-depth** — The HMAC-signed OAuth state payload carries a `level` field (`personal` | `organization`) to prevent cross-level misuse.

**`@PublicEndpoint()` + `MetaWebhookGuard`** — Meta webhooks arrive before org context is known, so normal tenant authentication cannot be used. The guard validates `X-Hub-Signature-256` using `META_APP_SECRET`.
</Accordion>
</AccordionGroup>

## Multi-Tenancy Patterns

<Warning>
The messaging module introduces unique multi-tenancy challenges because webhooks arrive without org context.
</Warning>

### Two-Step RLS Bypass

The webhook controller receives events for ALL organizations from a single Meta App. Org context is unknown at arrival time.

<Steps>
<Step title="Find Organization">
Use `executeReadOnlyWithBypass` to find which org owns the account
```typescript
const account = await this.tenantContext.executeReadOnlyWithBypass(async (em) => {
  return em.findOne(ChannelAccount, { externalAccountId: job.data.accountId });
});
```
</Step>

<Step title="Process in Org Context">
Switch to that organization's context for processing
```typescript
await this.tenantContext.executeInOrg(
  account.organization.id,
  async (em) => {
    await this.processMessageInTransaction(em, job.data);
  },
  { userId: undefined }, // system action
);
```
</Step>
</Steps>

### Composable Transaction Pattern

Services that participate in existing transactions expose `*InTransaction` methods:

```typescript
// Public API — wraps TenantContext
async matchOrCreate(channel, identifier, profileData, orgId): Promise<MatchResult>;

// Composable — accepts EntityManager from caller's transaction
async matchOrCreateInTransaction(em, channel, identifier, profileData, orgId): Promise<MatchResult>;
```

<Tip>
The `em` parameter must always be the one provided by the TenantContext callback — never `this.em`.
</Tip>

## Message Flows

### Inbound Message Flow

<Steps>
<Step title="Webhook Receipt">
Meta platform sends webhook to `POST /messaging/webhook`
- Validates `X-Hub-Signature-256` 
- Returns 200 immediately
- Persists to `WebhookEventLog`
- Enqueues to pg-boss
</Step>

<Step title="Queue Processing">
Worker processes webhook event:
- Check idempotency using `externalEventId`
- Find organization via account lookup
- Route to appropriate channel provider
</Step>

<Step title="Entity Resolution">
Channel provider resolves entities:
- Match/create `PersonChannel`
- Match/create `Person` + `Lead`
- Find/create `Conversation`
- Create `Message` entity
</Step>

<Step title="CRM Integration">
Bridge services handle CRM integration:
- Create CRM `Activity` record
- Update `PersonChannel` statistics
- Link conversation to existing contacts
</Step>

<Step title="Event Emission">
Emit real-time events:
- WebSocket events to connected clients
- Notification events for mobile/email
</Step>
</Steps>

### Outbound Message Flow

<Steps>
<Step title="Message Creation">
Agent creates message via API or WebSocket
- Validate permissions and conversation state
- Create `Message` entity with `PENDING` status
- Create `MessageOutbox` entry in same transaction
</Step>

<Step title="Queue Processing">
Message sender worker processes outbox:
- Call Meta Graph API to send message
- Update message status based on response
- Handle rate limiting and retries
</Step>

<Step title="Delivery Updates">
Meta webhooks provide delivery updates:
- Status updates: sent, delivered, read, failed
- Update `Message.status` accordingly
- Emit WebSocket events for real-time UI
</Step>
</Steps>

## Business Rules

<AccordionGroup>
<Accordion title="Conversation Management">
- **Auto-creation**: Conversations are created automatically on first inbound message
- **Status transitions**: Only OPEN ↔ CLOSED allowed, independent of archive state
- **Assignment rules**: Multiple assignments per conversation supported (agent + team pool)
- **Archive behavior**: Archived conversations remain accessible but hidden from default views
</Accordion>

<Accordion title="AI Mode Cascading">
- **Conversation level**: Each conversation has explicit `aiMode` setting
- **Account default**: Falls back to `ChannelAccount.defaultAiMode`
- **Organization default**: Falls back to organization-wide setting
- **Final fallback**: `AiMode.OFF` if no defaults configured
</Accordion>

<Accordion title="Message Templates">
- **META_APPROVED**: Platform-approved templates for marketing/notifications
- **QUICK_REPLY**: Agent shortcuts with variable substitution
- **AI_PROMPT**: System prompts for AI conversation handling
- **Variable resolution**: Templates support `{{variable}}` syntax with context data
</Accordion>

<Accordion title="Personal Account Linking">
- **WhatsApp**: Personal accounts share organization WABA token
- **Instagram/Messenger**: Personal accounts use individual Page Access Tokens
- **Access control**: Personal account owners get conversation access regardless of team assignments
</Accordion>
</AccordionGroup>

### Permission Resolution

<CodeGroup>
```typescript Permission Service
export class ConversationPermissionService {
  computePermissions(conversation: Conversation, user: User): ResourcePermissionsDto {
    // 1. Check org-level permissions
    if (user.hasPermission('MESSAGING_MANAGE')) {
      return this.fullAccess();
    }
    
    // 2. Check personal account ownership
    if (this.isPersonalAccountOwner(conversation.channelAccount, user)) {
      return { canView: true, canReply: true, canAssign: false, ... };
    }
    
    // 3. Check conversation assignments
    const assignment = conversation.assignments.find(a => 
      a.user?.id === user.id || user.teams.includes(a.team?.id)
    );
    
    if (assignment) {
      return { canView: true, canReply: assignment.canReply, ... };
    }
    
    // 4. Default: no access
    return this.noAccess();
  }
}
```

  constructor(

    public type: string,
    public message: string,

## Known Gaps & Technical Debt

<AccordionGroup>
<Accordion title="Rate Limiting">
**Current State**: Basic retry logic in queue workers
**Gap**: No proactive rate limiting or quota management
**Impact**: Risk of hitting Meta API limits during high volume
**Mitigation**: Implement token bucket or sliding window rate limiter
</Accordion>

<Accordion title="Message Threading">
**Current State**: Simple reply-to relationships
**Gap**: No complex threading or conversation branching
**Impact**: Limited UX for complex conversations
**Future**: Consider thread-aware UI and grouping logic
</Accordion>

<Accordion title="Media Optimization">
**Current State**: Media files downloaded and stored as-is
**Gap**: No compression, resizing, or format conversion
**Impact**: Storage costs and performance issues
**Future**: Implement media pipeline with optimization
</Accordion>

<Accordion title="Analytics & Reporting">
**Current State**: Basic message and conversation counts
**Gap**: No detailed analytics, response times, or agent performance metrics
**Impact**: Limited insights for business optimization
**Future**: Dedicated analytics module with time-series data
</Accordion>

<Accordion title="Template Management">
**Current State**: Basic CRUD operations for templates
**Gap**: No template approval workflow, version control, or A/B testing
**Impact**: Risk of using non-compliant templates
**Future**: Template governance and testing framework
</Accordion>
</AccordionGroup>
