## Overview

The Messaging module provides a unified, channel-agnostic messaging system for WhatsApp, Instagram, and Facebook Messenger. It replaces the separate per-channel modules with shared entities, a shared queue, and a single WebSocket namespace.

<Info>
**Last Updated:** 2026-06-01  
**Status:** Active
</Info>

### Problem → Solution

| Problem | Solution |
|---------|----------|
| Duplicated logic across WhatsApp and Instagram modules | Single `MessagingModule` with channel providers |
| No webhook signature validation (security gap) | Shared `MetaWebhookGuard` validates `X-Hub-Signature-256` |
| Inconsistent WebSocket auth (Instagram gateway has no JWT) | Single `/messaging` gateway with JWT auth |
| No Facebook Messenger support | Third channel provider |
| Separate entity schemas per channel | Unified entities: `Conversation`, `Message`, `ChannelAccount` |
| No shared queue infrastructure | Shared `PgBossQueueService` for messaging + notifications |

## Key Design Decisions

<AccordionGroup>
  <Accordion title="pg-boss over BullMQ">
    Project already uses pg-boss for notifications. No new Redis dependency. Interface-based design (`IQueueService`) allows swapping later.
  </Accordion>

  <Accordion title="Direct PersonChannel FK on Conversation">
    Conversations link directly to the CRM's `PersonChannel` via FK. Simpler model, no bidirectional sync overhead. The lead FK was moved from Conversation to Lead (`Lead.sourceConversation`) — conversation detail loads the single active lead for the current conversation through that FK.
  </Accordion>

  <Accordion title="Archive as boolean, not status">
    `Conversation.isArchived` is orthogonal to `status` (OPEN/CLOSED), following `ARCHIVE_SYSTEM_SPECIFICATION.md`. Standard inbox rail reads keep the default `active` filter enabled, so archived conversations are hidden from the live Inbox.
  </Accordion>

  <Accordion title="ConversationAssignment entity">
    Conversations use a dedicated `conversation_assignment` table instead of the CRM `entity_stakeholder` pattern. Each assignment is one row with nullable `user_id` and `team_id`.
  </Accordion>

  <Accordion title="AI Agent Integration">
    The messaging module integrates with the AI module to provide automated responses through AI agents. Conversations have an `aiMode` field that controls AI participation (OFF, AUTO_REPLY, SUGGEST_ONLY, DRAFT).
  </Accordion>

  <Accordion title="Transactional outbox">
    Outbound messages use an outbox table written in the same DB transaction as the Message entity, guaranteeing at-least-once delivery.
  </Accordion>

  <Accordion title="Three-tier template system">
    `MessageTemplate` supports three types: `META_APPROVED` (platform-approved), `QUICK_REPLY` (agent shortcuts with variable resolution), and `AI_PROMPT` (AI system prompts).
  </Accordion>

  <Accordion title="Personal accounts share org WABA token">
    WhatsApp personal accounts reuse the organization's WABA access token (same Business Account). Instagram and Messenger personal accounts use their own Page Access Token obtained via OAuth.
  </Accordion>
</AccordionGroup>

## Multi-Tenancy Patterns

The messaging module follows the organization-scoped multi-tenancy pattern:

<Steps>
  <Step title="Organization Isolation">
    All messaging entities are scoped to an organization through the `organization_id` field
  </Step>
  <Step title="User Context">
    Controllers use `@User()` decorator to access authenticated user context and derive organization scope
  </Step>
  <Step title="Query Filtering">
    All database queries automatically filter by organization ID to prevent cross-tenant data access
  </Step>
</Steps>

## Message Flows

### Inbound Message Flow

<Steps>
  <Step title="Webhook Reception">
    Channel-specific webhook receives message from platform (WhatsApp, Instagram, Messenger)
  </Step>
  <Step title="Signature Validation">
    `MetaWebhookGuard` validates `X-Hub-Signature-256` header for security
  </Step>
  <Step title="Message Processing">
    Channel provider processes platform-specific payload and creates unified `Message` entity
  </Step>
  <Step title="Conversation Resolution">
    System finds or creates `Conversation` based on `channelAccount` + `personChannel` combination
  </Step>
  <Step title="AI Processing">
    If conversation has AI mode enabled, queue AI agent execution for automated response
  </Step>
  <Step title="WebSocket Broadcast">
    Broadcast new message to connected clients via WebSocket room
  </Step>
  <Step title="Notification">
    Send notification to assigned agents/teams based on conversation assignments
  </Step>
</Steps>

### Outbound Message Flow

<Steps>
  <Step title="Message Creation">
    Agent creates outbound message through API or WebSocket
  </Step>
  <Step title="Transactional Write">
    Write both `Message` entity and `MessageOutbox` entry in same database transaction
  </Step>
  <Step title="Queue Processing">
    pg-boss worker processes outbox entry and calls channel provider
  </Step>
  <Step title="Platform API Call">
    Channel provider sends message to platform API (Graph API for Meta channels)
  </Step>
  <Step title="Status Updates">
    Platform webhooks provide delivery confirmations, updating message status
  </Step>
  <Step title="WebSocket Updates">
    Broadcast status changes to connected clients
  </Step>
</Steps>

## Business Rules

<Note>
Key business rules that govern messaging behavior
</Note>

### Conversation Management

- **One active conversation per channel account + person channel pair**
- **Archived conversations remain accessible but hidden from active inbox**
- **Soft-deleted conversations are terminal tombstones**
- **Account disconnect archives and soft-deletes all conversations**

### AI Mode Inheritance

The AI mode cascades with the following priority:
1. Conversation-level `aiMode` (if explicitly set)
2. Channel account `defaultAiMode`
3. Organization default AI mode
4. System default: `OFF`

### Assignment Rules

<Warning>
Assignment rules are critical for message routing and permissions
</Warning>

- **Personal accounts**: Owner assignment is automatically created and protected (`is_owner=true`)
- **Organization accounts**: Manual assignment required
- **Multiple assignments**: Supported (e.g., direct agent + team pool)
- **Team assignments**: Can be `user + team` (agent on behalf) or `null + team` (team pool)

## WebSocket Events & Room Architecture

### Room Structure

<Info>
The messaging module uses a single WebSocket namespace with JWT authentication
</Info>

```typescript
Namespace: /messaging
Rooms:
- conversation:{conversationId} // Per-conversation updates
- inbox:{userId} // Personal inbox updates  
- team-inbox:{teamId} // Team inbox updates
- org-inbox:{orgId} // Organization-wide updates
```

### Event Types

### Room Subscription Logic

<Steps>
  <Step title="Authentication">
    Client connects with JWT token in authentication header
  </Step>
  <Step title="User Rooms">
    Server automatically subscribes authenticated user to:
    - `inbox:{userId}` for personal notifications
    - `team-inbox:{teamId}` for each team the user belongs to
    - `org-inbox:{orgId}` for organization-wide broadcasts
  </Step>
  <Step title="Conversation Rooms">
    Client explicitly joins conversation rooms via `join-conversation` event
  </Step>
  <Step title="Permission Validation">
    Server validates user has access to conversation before allowing room join
  </Step>
</Steps>

## Error Handling & Retry Strategy

### Retry Policies

<Warning>
Different job types have different retry strategies based on their criticality
</Warning>

| Job Type | Max Retries | Backoff Strategy | Timeout |
|----------|-------------|------------------|---------|
| Send Outbound Message | 5 | Exponential (2^n seconds) | 30s |
| Process Webhook | 3 | Linear (30s intervals) | 15s |
| AI Agent Execute | 2 | Exponential (1.5^n minutes) | 60s |
| Sync History | 1 | None | 120s |

### Error Classification
