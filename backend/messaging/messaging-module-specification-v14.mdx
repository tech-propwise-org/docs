## Overview

The Messaging module provides a unified, channel-agnostic messaging system for WhatsApp, Instagram, and Facebook Messenger. It replaces the separate per-channel modules with shared entities, a shared queue, and a single WebSocket namespace.

### Problem → Solution

| Problem | Solution |
| ------- | -------- |
| Duplicated logic across WhatsApp and Instagram modules | Single `MessagingModule` with channel providers |
| No webhook signature validation (security gap) | Shared `MetaWebhookGuard` validates `X-Hub-Signature-256` |
| Inconsistent WebSocket auth (Instagram gateway has no JWT) | Single `/messaging` gateway with JWT auth |
| No Facebook Messenger support | Third channel provider |
| Separate entity schemas per channel | Unified entities: `Conversation`, `Message`, `ChannelAccount` |
| No shared queue infrastructure | Shared `PgBossQueueService` for messaging + notifications |

### Key Design Decisions

<AccordionGroup>
<Accordion title="Queue Infrastructure">
**pg-boss over BullMQ** — Project already uses pg-boss for notifications. No new Redis dependency. Interface-based design (`IQueueService`) allows swapping later.
</Accordion>

<Accordion title="CRM Integration">
**Direct PersonChannel FK on Conversation** — Conversations link directly to the CRM's `PersonChannel` via FK. Simpler model, no bidirectional sync overhead. The lead FK was moved from Conversation to Lead (`Lead.sourceConversation`).
</Accordion>

<Accordion title="Archive Strategy">
**Archive as boolean, not status** — `Conversation.isArchived` is orthogonal to `status` (OPEN/CLOSED), following `ARCHIVE_SYSTEM_SPECIFICATION.md`.
</Accordion>

<Accordion title="Assignment Model">
**`ConversationAssignment` entity** — Dedicated `conversation_assignment` table instead of CRM `entity_stakeholder` pattern. Each assignment is one row with nullable `user_id` and `team_id`.
</Accordion>

<Accordion title="Message Delivery">
**Transactional outbox** — Outbound messages use an outbox table written in the same DB transaction as the Message entity, guaranteeing at-least-once delivery.
</Accordion>

<Accordion title="AI Integration">
**Per-conversation AI mode with cascade** — Each conversation has an `aiMode` field (OFF, AUTO_REPLY, SUGGEST_ONLY, DRAFT). Default cascades: ChannelAccount.defaultAiMode → Organization default → OFF.
</Accordion>

<Accordion title="Template System">
**Three-tier template system** — `MessageTemplate` supports `META_APPROVED`, `QUICK_REPLY`, and `AI_PROMPT` types.
</Accordion>

<Accordion title="Account Authentication">
**Personal accounts share org WABA token** — WhatsApp personal accounts reuse the organization's WABA access token. Instagram and Messenger personal accounts use their own Page Access Token via OAuth.
</Accordion>
</AccordionGroup>

## Architecture & Module Structure

```mermaid
graph TD
    A[Meta Platform Webhooks] --> B[POST /messaging/webhook]
    B --> C[@PublicEndpoint + MetaWebhookGuard]
    C --> D[WebhookEventLog]
    D --> E[pg-boss Queue]
    E --> F[Queue Worker webhook-processor]
    F --> G[Channel Provider]
    G --> H[Message Processing]
    H --> I[WebSocket Events]
```

<Steps>
<Step title="Webhook Reception">
Meta Platform sends webhooks to `POST /messaging/webhook` with `@PublicEndpoint()` and `MetaWebhookGuard` validation of `X-Hub-Signature-256`.
</Step>

<Step title="Queue Processing">
Events are persisted to `WebhookEventLog` and enqueued to pg-boss for processing.
</Step>

<Step title="Message Processing">
Queue worker processes events:
- Routes to channel provider (WhatsApp/Instagram/Messenger)
- Matches/creates PersonChannel and Person
- Creates/updates Conversation and Message
- Emits WebSocket and notification events
</Step>
</Steps>

## Multi-Tenancy Patterns

<Warning>
The messaging module introduces unique multi-tenancy challenges because webhooks arrive without org context.
</Warning>

### Two-Step RLS Bypass

Webhook processing requires a two-step approach since organization context is unknown at webhook arrival:

<CodeGroup>
```typescript Step 1: Find Organization
// Bypass RLS to find which org owns the account
const account = await this.tenantContext.executeReadOnlyWithBypass(async (em) => {
  return em.findOne(ChannelAccount, { externalAccountId: job.data.accountId });
});
```

```typescript Step 2: Process in Context
// Process within the organization's context
await this.tenantContext.executeInOrg(
  account.organization.id,
  async (em) => {
    await this.processMessageInTransaction(em, job.data);
  },
  { userId: undefined }, // system action
);
```
</CodeGroup>

### Composable Transaction Pattern

Services expose `*InTransaction` methods for participation in existing transactions:

```typescript
// Public API — wraps TenantContext
async matchOrCreate(channel, identifier, profileData, orgId): Promise<MatchResult>;

// Composable — accepts EntityManager from caller's transaction
async matchOrCreateInTransaction(em, channel, identifier, profileData, orgId): Promise<MatchResult>;
```

<Note>
The `em` parameter must always be the one provided by the TenantContext callback — never `this.em`.
</Note>

## Message Flows

### Inbound Message Flow

<Steps>
<Step title="Webhook Reception">
Meta platform sends webhook to `/messaging/webhook` endpoint.
</Step>

<Step title="Validation & Queuing">
`MetaWebhookGuard` validates signature and queues event for processing.
</Step>

<Step title="Organization Resolution">
Queue worker finds organization using two-step RLS bypass.
</Step>

<Step title="Channel Processing">
Appropriate channel provider processes the webhook payload.
</Step>

<Step title="Entity Creation">
System creates/updates PersonChannel, Conversation, and Message entities.
</Step>

<Step title="Event Emission">
WebSocket events notify connected clients of new messages.
</Step>
</Steps>

### Outbound Message Flow

<Steps>
<Step title="Message Creation">
Agent creates message through UI, triggering API call.
</Step>

<Step title="Outbox Pattern">
Message saved to both `message` and `message_outbox` tables in same transaction.
</Step>

<Step title="Queue Processing">
`message-sender` queue worker picks up outbox entries.
</Step>

<Step title="Platform Delivery">
Channel provider sends message to appropriate Meta API.
</Step>

<Step title="Status Updates">
Delivery confirmations update message status via webhooks.
</Step>
</Steps>

### Assignment Rules

<CardGroup cols={2}>

- Agent has direct responsibility
- Highest priority for notifications

- Team pool assignment

- Supports team metrics tracking

- Example: Direct agent + team pool
- Flexible escalation patterns

### AI Mode Cascade

AI mode determines automated response behavior with the following cascade:

1. **Conversation.aiMode** (if set)
2. **ChannelAccount.defaultAiMode** 
3. **Organization default**
4. **OFF** (fallback)

<Tabs>
<Tab title="OFF">
No AI assistance. Manual responses only.
</Tab>

<Tab title="AUTO_REPLY">
AI automatically sends responses to inbound messages.
</Tab>

<Tab title="SUGGEST_ONLY">
AI suggests responses but requires agent approval.
</Tab>

<Tab title="DRAFT">
AI creates draft responses for agent review and editing.
</Tab>
</Tabs>

## RBAC Permissions & Access Control

### Module Permissions

| Permission | Description |
| ---------- | ----------- |
| `MESSAGING_MANAGE` | Full messaging management access |
| `MESSAGING_WRITE` | Send messages, basic conversation actions |
| `MESSAGING_READ` | View conversations and messages |

### Resource-Level Permissions

Conversations implement `ResourcePermissionsDto` pattern:

<CodeGroup>
```typescript Permission Computation
export class ConversationPermissionService {
  computePermissions(conversation: Conversation, user: User): ResourcePermissionsDto {
    // MESSAGING_MANAGE → full access
    if (user.hasPermission('MESSAGING_MANAGE')) {
      return this.fullAccess();
    }
    
    // Personal account owner → view + reply
    if (this.isPersonalAccountOwner(conversation, user)) {
      return { canView: true, canReply: true, canAssign: false };
    }
    
    // Assignment-based permissions
    return this.computeAssignmentPermissions(conversation, user);
  }
}
```

```typescript Permission Flags
interface ResourcePermissionsDto {
  canView: boolean;
  canReply: boolean;
  canEdit: boolean;      // false for non-managers
  canTransfer: boolean;  // false for non-managers  
  canArchive: boolean;   // false for non-managers
  canAssign: boolean;    // team managers only
}
```
</CodeGroup>

<Warning>
Operational actions (close, reopen, ai-mode changes) are available to `MESSAGING_WRITE` users and are not gated by `canEdit`.
</Warning>

## Error Handling & Retry Strategy

### Webhook Processing

<Steps>
<Step title="Immediate Response">
Webhooks return 200 immediately after signature validation, regardless of processing outcome.
</Step>

<Step title="Idempotency">
`WebhookEventLog.externalEventId` prevents duplicate processing of the same Meta event.
</Step>

<Step title="Queue Retries">
pg-boss handles retries with exponential backoff for failed webhook processing.
</Step>

<Step title="Dead Letter Queue">
Failed events after max retries go to dead letter queue for manual investigation.
</Step>
</Steps>

### Message Sending

<CodeGroup>
```typescript Outbox Pattern
// Transactional write
await em.transactional(async (em) => {
  const message = em.create(Message, messageData);
  const outboxEntry = em.create(MessageOutbox, { message });
  
  await em.persistAndFlush([message, outboxEntry]);
});

// Queue pickup
await this.queueService.addJob('message-sender', {
  messageId: message.id,
  outboxId: outboxEntry.id
});
```

```typescript Retry Logic  
const maxRetries = 3;
const baseDelay = 1000; // 1 second

for (let attempt = 1; attempt <= maxRetries; attempt++) {
  try {
    await this.metaGraphApi.sendMessage(payload);
    await this.markOutboxEntryProcessed(outboxId);
    return;
  } catch (error) {
    if (attempt === maxRetries) {
      await this.markOutboxEntryFailed(outboxId, error);
      throw error;
    }
    
    await this.delay(baseDelay * Math.pow(2, attempt - 1));
  }
}
```
</CodeGroup>

## Known Gaps & Technical Debt

<Warning>
The following items require future attention:
</Warning>

1. **Media handling** — Current implementation downloads media synchronously. Need async processing with progress tracking.

2. **Rate limiting** — No per-channel rate limiting implemented. Meta APIs have strict limits that could cause delivery failures.

3. **Message encryption** — WhatsApp Business API supports end-to-end encryption but it's not implemented.

4. **Bulk operations** — No bulk message sending or conversation management APIs.

5. **Analytics** — Message delivery metrics and conversation analytics are minimal.

6. **Advanced templates** — Interactive message templates (buttons, quick replies) have limited support.
