## Overview

The Messaging module provides a unified, channel-agnostic messaging system for WhatsApp, Instagram, and Facebook Messenger. It replaces the separate per-channel modules with shared entities, a shared queue, and a single WebSocket namespace.

<Note>
**Last Updated:** 2026-04-15  
**Status:** Active
</Note>

### Problem → Solution

| Problem | Solution |
|---------|----------|
| Duplicated logic across WhatsApp and Instagram modules | Single `MessagingModule` with channel providers |
| No webhook signature validation (security gap) | Shared `MetaWebhookGuard` validates `X-Hub-Signature-256` |
| Inconsistent WebSocket auth (Instagram gateway has no JWT) | Single `/messaging` gateway with JWT auth |
| No Facebook Messenger support | Third channel provider |
| Separate entity schemas per channel | Unified entities: `Conversation`, `Message`, `ChannelAccount` |
| No shared queue infrastructure | Shared `PgBossQueueService` for messaging + notifications |

### Key Design Decisions

<AccordionGroup>
<Accordion title="Queue Selection: pg-boss over BullMQ">
Project already uses pg-boss for notifications. No new Redis dependency. Interface-based design (`IQueueService`) allows swapping later.
</Accordion>

<Accordion title="Direct PersonChannel FK on Conversation">
Conversations link directly to the CRM's `PersonChannel` via FK. Simpler model, no bidirectional sync overhead. The lead FK was moved from Conversation to Lead (`Lead.sourceConversation`).
</Accordion>

<Accordion title="Archive as boolean, not status">
`Conversation.isArchived` is orthogonal to `status` (OPEN/CLOSED), following `ARCHIVE_SYSTEM_SPECIFICATION.md`.
</Accordion>

<Accordion title="ConversationAssignment entity">
Conversations use a dedicated `conversation_assignment` table instead of the CRM `entity_stakeholder` pattern. Each assignment is one row with nullable `user_id` and `team_id`.
</Accordion>

<Accordion title="Transactional outbox">
Outbound messages use an outbox table written in the same DB transaction as the Message entity, guaranteeing at-least-once delivery.
</Accordion>
</AccordionGroup>

## Architecture & Module Structure

```
                    Meta Platform Webhooks
                    (WhatsApp, Instagram, Messenger)
                                |
                    POST /messaging/webhook
                    @PublicEndpoint() + MetaWebhookGuard
                    Validates X-Hub-Signature-256
                    Returns 200 immediately
                    Persists to WebhookEventLog
                    Enqueues to pg-boss queue
                                |
                    Queue Worker (webhook-processor)
                    1. Check idempotency (externalEventId)
                    2. executeReadOnlyWithBypass() -> find org
                    3. executeInOrg(orgId) -> process:
                       a. Route to channel provider (WA/IG/Messenger)
                       b. Match/create PersonChannel
                       c. Match/create Person + Lead
                       d. Find/create Conversation
                       e. Create Message
                       f. Create CRM Activity (via bridge)
                       g. Update PersonChannel stats
                       h. Emit WebSocket events
                       i. Emit notification events
```

## Multi-Tenancy Patterns

<Warning>
The messaging module introduces unique multi-tenancy challenges because webhooks arrive without org context. See `Docs/MULTI_TENANCY.md` for the full RLS reference.
</Warning>

### Two-Step RLS Bypass (Webhook Processing)

The webhook controller receives events for ALL organizations from a single Meta App. Org context is unknown at arrival time.

```typescript
// Step 1: Find which org owns this account (bypass RLS)
const account = await this.tenantContext.executeReadOnlyWithBypass(async (em) => {
  return em.findOne(ChannelAccount, { externalAccountId: job.data.accountId });
});

// Step 2: Process within that org's context
await this.tenantContext.executeInOrg(
  account.organization.id,
  async (em) => {
    await this.processMessageInTransaction(em, job.data);
  },
  { userId: undefined },
); // system action, no user
```

### Composable `*InTransaction` Pattern

Services that participate in existing transactions expose `*InTransaction` methods. The webhook worker calls these within its `executeInOrg` callback:

```typescript
// Public API — wraps TenantContext
async matchOrCreate(channel, identifier, profileData, orgId): Promise<MatchResult>;

// Composable — accepts EntityManager from caller's transaction
async matchOrCreateInTransaction(em, channel, identifier, profileData, orgId): Promise<MatchResult>;
```

<Important>
The `em` parameter must always be the one provided by the TenantContext callback — never `this.em`.
</Important>

## Entities

### Core Entities

<Tabs>
<Tab title="ChannelAccount">
Represents a connected WhatsApp/Instagram/Messenger account.

```typescript
@Entity('channel_account')
export class ChannelAccount {
  @PrimaryGeneratedColumn('uuid')
  id: string;

  @Column({ type: 'enum', enum: Channel })
  channel: Channel;

  @Column()
  externalAccountId: string;

  @Column()
  displayName: string;

  @Column({ nullable: true })
  pageId?: string; // Instagram only

  @Column({ type: 'enum', enum: AiMode, default: AiMode.OFF })
  defaultAiMode: AiMode;

  @Column({ default: true })
  isActive: boolean;

  @ManyToOne(() => Organization)
  organization: Organization;

  @ManyToOne(() => User, { nullable: true })
  connectedBy?: User;
}
```
</Tab>

<Tab title="Conversation">
A messaging thread with a contact.

```typescript
@Entity('conversation')
export class Conversation {
  @PrimaryGeneratedColumn('uuid')
  id: string;

  @ManyToOne(() => ChannelAccount)
  channelAccount: ChannelAccount;

  @ManyToOne(() => PersonChannel)
  personChannel: PersonChannel;

  @ManyToOne(() => Contact, { nullable: true })
  contact?: Contact;

  @Column({ type: 'enum', enum: ConversationStatus })
  status: ConversationStatus;

  @Column({ default: false })
  isArchived: boolean;

  @Column({ type: 'enum', enum: AiMode })
  aiMode: AiMode;

  @Column({ type: 'timestamp', nullable: true })
  lastMessageAt?: Date;

  @OneToMany(() => Message, m => m.conversation)
  messages: Message[];

  @OneToMany(() => ConversationAssignment, a => a.conversation)
  assignments: ConversationAssignment[];
}
```
</Tab>

<Tab title="Message">
Individual message within a conversation.

```typescript
@Entity('message')
export class Message {
  @PrimaryGeneratedColumn('uuid')
  id: string;

  @ManyToOne(() => Conversation)
  conversation: Conversation;

  @Column()
  externalMessageId: string;

  @Column({ type: 'enum', enum: MessageDirection })
  direction: MessageDirection;

  @Column({ type: 'enum', enum: MessageType })
  type: MessageType;

  @Column({ type: 'text', nullable: true })
  text?: string;

  @Column({ type: 'jsonb', nullable: true })
  metadata?: any;

  @Column({ type: 'timestamp' })
  timestamp: Date;

  @ManyToOne(() => User, { nullable: true })
  sentBy?: User;
}
```
</Tab>
</Tabs>

## Message Flows

### Inbound Message Flow

<Steps>
<Step title="Webhook Reception">
Meta platform sends webhook to `/messaging/webhook` endpoint with `X-Hub-Signature-256` header.
</Step>

<Step title="Signature Validation">
`MetaWebhookGuard` validates the signature using `META_APP_SECRET`.
</Step>

<Step title="Event Logging">
Webhook payload is saved to `WebhookEventLog` with status `PENDING`.
</Step>

<Step title="Queue Processing">
`webhook-processor` job processes the event:
- Finds the organization via `ChannelAccount`
- Routes to appropriate channel provider
- Creates/updates `Person`, `PersonChannel`, `Conversation`, `Message`
- Emits WebSocket events
- Triggers notifications
</Step>
</Steps>

### Outbound Message Flow

<Steps>
<Step title="Message Creation">
Agent creates message via API endpoint `/conversations/:id/messages`.
</Step>

<Step title="Transactional Write">
Both `Message` and `MessageOutbox` records are created in the same transaction.
</Step>

<Step title="Queue Processing">
`message-sender` job processes the outbox entry:
- Calls Meta Graph API
- Updates message status
- Removes outbox entry on success
- Implements retry logic on failure
</Step>
</Steps>

## Business Rules

### AI Mode Cascade

AI mode is determined by the following cascade:

<Steps>
<Step title="Conversation Level">
If `Conversation.aiMode` is explicitly set, use that value.
</Step>

<Step title="Channel Account Level">
If conversation has no explicit setting, use `ChannelAccount.defaultAiMode`.
</Step>

<Step title="Organization Level">
If channel account has no setting, use organization default.
</Step>

<Step title="System Default">
If no other setting exists, default to `AiMode.OFF`.
</Step>
</Steps>

### Assignment Rules

<Warning>
Multiple assignments per conversation are supported but should be managed carefully to avoid conflicts.
</Warning>

Assignment types:
- **Direct assignment**: `user_id` set, `team_id` is null
- **Team assignment**: `user_id` is null, `team_id` set
- **Agent on behalf of team**: Both `user_id` and `team_id` set

### Message Deduplication

Inbound messages are deduplicated using `externalMessageId` to prevent processing the same webhook multiple times.

## RBAC Permissions & Access Control

### Resource Permissions

Conversations return `ResourcePermissionsDto` with these flags:

<Tabs>
<Tab title="Permission Flags">
- `canView`: View conversation and messages
- `canReply`: Send messages in the conversation
- `canEdit`: Modify conversation properties
- `canTransfer`: Transfer to another agent/team
- `canArchive`: Archive/unarchive conversation
- `canAssign`: Assign within user's teams
</Tab>

<Tab title="Permission Logic">
```typescript
// MESSAGING_MANAGE → full access
if (hasPermission('MESSAGING_MANAGE')) {
  return fullAccess();
}

// MESSAGING_WRITE → view + reply only
if (hasPermission('MESSAGING_WRITE')) {
  return { canView: true, canReply: true };
}

// Personal account owner
if (isPersonalAccountOwner(user, conversation.channelAccount)) {
  return { canView: true, canReply: true };
}

// Assignment-based access
const assignment = findUserAssignment(user, conversation);
if (assignment) {
  return { 
    canView: true, 
    canReply: assignment.canReply 
  };
}
```
</Tab>
</Tabs>

## WebSocket Events & Room Architecture

### Room Structure

The messaging WebSocket gateway uses a single `/messaging` namespace with these room patterns:

- `org:{orgId}`: Organization-wide events
- `conversation:{conversationId}`: Conversation-specific events
- `user:{userId}`: User-specific notifications

### Event Types

<CodeGroup>

## Error Handling & Retry Strategy

### Queue Retry Logic

<Tabs>
<Tab title="Webhook Processing">
- **Max retries**: 5
- **Backoff**: Exponential (1s, 2s, 4s, 8s, 16s)
- **Dead letter**: Move to `failed_webhooks` table after max retries
</Tab>

<Tab title="Message Sending">
- **Max retries**: 3  
- **Backoff**: Linear (30s, 60s, 120s)
- **Rate limiting**: Respect Meta API rate limits
- **Dead letter**: Mark message as `FAILED` in outbox
</Tab>
</Tabs>

### API Error Responses

Common error scenarios and their HTTP status codes:

| Scenario | Status Code | Error Code |
|----------|-------------|------------|
| Conversation not found | 404 | `CONVERSATION_NOT_FOUND` |
| Insufficient permissions | 403 | `INSUFFICIENT_PERMISSIONS` |
| Invalid message format | 400 | `INVALID_MESSAGE_FORMAT` |
| Channel account disconnected | 422 | `CHANNEL_ACCOUNT_INACTIVE` |
| Rate limit exceeded | 429 | `RATE_LIMIT_EXCEEDED` |

### Database Migrations

<Warning>
Run these migrations in order during deployment:
</Warning>

1. Create messaging tables and enums
2. Migrate existing WhatsApp/Instagram data
3. Create indexes for performance
4. Enable RLS policies

### Performance Considerations

- **Message indexes**: Create compound indexes on `(conversation_id, timestamp)` for efficient message fetching
- **Queue monitoring**: Monitor pg-boss job completion rates and dead letter queues
- **WebSocket scaling**: Consider Redis adapter for horizontal scaling of WebSocket connections
