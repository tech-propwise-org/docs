## Overview

The Messaging module provides a unified, channel-agnostic messaging system for WhatsApp, Instagram, and Facebook Messenger. It replaces the separate per-channel modules with shared entities, a shared queue, and a single WebSocket namespace.

### Problem → Solution

| Problem | Solution |
|---------|----------|
| Duplicated logic across WhatsApp and Instagram modules | Single `MessagingModule` with channel providers |
| No webhook signature validation (security gap) | Shared `MetaWebhookGuard` validates `X-Hub-Signature-256` |
| Inconsistent WebSocket auth (Instagram gateway has no JWT) | Single `/messaging` gateway with JWT auth |
| No Facebook Messenger support | Third channel provider |
| Separate entity schemas per channel | Unified entities: `Conversation`, `Message`, `ChannelAccount` |
| No shared queue infrastructure | Shared `PgBossQueueService` for messaging + notifications |

### Key Design Decisions

<AccordionGroup>
<Accordion title="Queue Infrastructure">
**pg-boss over BullMQ** — Project already uses pg-boss for notifications. No new Redis dependency. Interface-based design (`IQueueService`) allows swapping later.
</Accordion>

<Accordion title="CRM Integration">
**Direct PersonChannel FK on Conversation** — Conversations link directly to the CRM's `PersonChannel` via FK. Simpler model, no bidirectional sync overhead. The lead FK was moved from Conversation to Lead (`Lead.sourceConversation`).
</Accordion>

<Accordion title="Archive System">
**Archive as boolean, not status** — `Conversation.isArchived` is orthogonal to `status` (OPEN/CLOSED), following `ARCHIVE_SYSTEM_SPECIFICATION.md`. Standard inbox rail reads keep the default `active` filter enabled.
</Accordion>

<Accordion title="Assignment Model">
**`ConversationAssignment` entity (not `entity_stakeholder`)** — Conversations use a dedicated `conversation_assignment` table instead of the CRM `entity_stakeholder` pattern.
</Accordion>

<Accordion title="AI Integration">
**Per-conversation AI mode with cascade** — Each conversation has an `aiMode` field (OFF, AUTO_REPLY, SUGGEST_ONLY, DRAFT). Default cascades: ChannelAccount.defaultAiMode → Organization default → OFF.
</Accordion>
</AccordionGroup>

## Message Flows

<Steps>
<Step title="Webhook Reception">
Channel-specific webhook endpoints receive platform notifications and validate signatures using `MetaWebhookGuard`.
</Step>

<Step title="Message Processing">
Webhook processors create `Message` entities and update `Conversation` state within database transactions.
</Step>

<Step title="Queue Dispatch">
Transactional outbox pattern ensures reliable queue job creation for notifications and AI processing.
</Step>

<Step title="WebSocket Broadcast">
Real-time updates are sent to appropriate conversation rooms via the `/messaging` WebSocket namespace.
</Step>

<Step title="AI Processing">
If AI mode is enabled, messages trigger AI agent processing for automated responses or suggestions.
</Step>
</Steps>

## Business Rules

<Warning>
**Critical Rules**

- Conversations are uniquely identified by `channelAccount + personChannel` combination
- Archived conversations can receive new messages but don't appear in active inbox
- Soft-deleted conversations require explicit restoration or create replacement conversations
- Personal WhatsApp accounts share organization WABA tokens
</Warning>

### Assignment Rules

<Info>
Assignment hierarchy follows this priority:
1. Direct user assignment (`user_id + null team_id`)
2. Team agent assignment (`user_id + team_id`)
3. Team pool assignment (`null user_id + team_id`)
</Info>

### AI Mode Cascade

<Tip>
AI mode determination follows this cascade:
`Conversation.aiMode` → `ChannelAccount.defaultAiMode` → Organization default → `OFF`
</Tip>

### Error Handling Patterns

<CodeGroup>
```typescript Webhook Validation
try {
  await this.webhookGuard.validateSignature(request);
  // Process webhook
} catch (error) {
  if (error instanceof UnauthorizedException) {
    // Log security incident
    this.logger.warn('Invalid webhook signature', { error });
    throw new UnauthorizedException('Invalid signature');
  }
  throw error;
}
```

```typescript Message Send Retry
async sendMessage(message: OutboundMessage): Promise<void> {
  const maxRetries = 3;
  let attempt = 0;

  while (attempt < maxRetries) {
    try {
      await this.channelProvider.send(message);
      return;
    } catch (error) {
      attempt++;
      if (attempt >= maxRetries) {
        throw error;
      }
      await this.delay(Math.pow(2, attempt) * 1000);
    }
  }
}
```
</CodeGroup>

## Query Patterns

### Conversation Queries

<CodeGroup>
```sql Active Inbox Query
SELECT c.*, ca.display_name, pc.external_id
FROM conversations c
JOIN channel_accounts ca ON c.channel_account_id = ca.id
JOIN person_channels pc ON c.person_channel_id = pc.id
WHERE c.organization_id = $1
  AND c.is_archived = false
  AND c.is_deleted = false
ORDER BY c.last_message_at DESC
LIMIT $2 OFFSET $3;
```

```sql Assignment Query
SELECT c.*, STRING_AGG(u.full_name, ', ') as assigned_users
FROM conversations c
LEFT JOIN conversation_assignments ca ON c.id = ca.conversation_id
LEFT JOIN users u ON ca.user_id = u.id
WHERE c.organization_id = $1
  AND ca.is_deleted = false
GROUP BY c.id
ORDER BY c.last_message_at DESC;
```
</CodeGroup>

## Error Handling & Retry Strategy

### Webhook Processing

<Steps>
<Step title="Signature Validation">
Validate `X-Hub-Signature-256` header using shared secret
</Step>
<Step title="Idempotency Check">
Check for duplicate webhook processing using `externalId`
</Step>
<Step title="Database Transaction">
Process webhook data within a single database transaction
</Step>
<Step title="Queue Job Creation">
Use transactional outbox for reliable queue job scheduling
</Step>
<Step title="Error Response">
Return appropriate HTTP status codes for webhook retry behavior
</Step>
</Steps>

### Outbound Message Retry

<Info>
Retry strategy uses exponential backoff with jitter:
- Initial delay: 1 second
- Max retries: 3 attempts
- Backoff multiplier: 2x
- Max delay: 30 seconds
</Info>
