## Overview

The Messaging module provides a unified, channel-agnostic messaging system for WhatsApp, Instagram, and Facebook Messenger. It replaces the separate per-channel modules with shared entities, a shared queue, and a single WebSocket namespace.

### Problem → Solution

| Problem | Solution |
|---------|----------|
| Duplicated logic across WhatsApp and Instagram modules | Single `MessagingModule` with channel providers |
| No webhook signature validation (security gap) | Shared `MetaWebhookGuard` validates `X-Hub-Signature-256` |
| Inconsistent WebSocket auth (Instagram gateway has no JWT) | Single `/messaging` gateway with JWT auth |
| No Facebook Messenger support | Third channel provider |
| Separate entity schemas per channel | Unified entities: `Conversation`, `Message`, `ChannelAccount` |
| No shared queue infrastructure | Shared `PgBossQueueService` for messaging + notifications |

### Key Design Decisions

<AccordionGroup>
<Accordion title="pg-boss over BullMQ">
Project already uses pg-boss for notifications. No new Redis dependency. Interface-based design (`IQueueService`) allows swapping later.
</Accordion>

<Accordion title="Direct PersonChannel FK on Conversation">
Conversations link directly to the CRM's `PersonChannel` via FK. Simpler model, no bidirectional sync overhead. The lead FK was moved from Conversation to Lead (`Lead.sourceConversation`) — conversations discover related leads via `personChannel → person → leads`.
</Accordion>

<Accordion title="Archive as boolean, not status">
`Conversation.isArchived` is orthogonal to `status` (OPEN/CLOSED), following `ARCHIVE_SYSTEM_SPECIFICATION.md`.
</Accordion>

<Accordion title="ConversationAssignment entity">
Conversations use a dedicated `conversation_assignment` table instead of the CRM `entity_stakeholder` pattern. Each assignment is one row with nullable `user_id` and `team_id`: `user + null` = direct assignment, `user + team` = agent on behalf of team, `null + team` = team pool.
</Accordion>

<Accordion title="Transactional outbox">
Outbound messages use an outbox table written in the same DB transaction as the Message entity, guaranteeing at-least-once delivery.
</Accordion>

<Accordion title="Per-conversation AI mode with cascade">
Each conversation has an `aiMode` field (OFF, AUTO_REPLY, SUGGEST_ONLY, DRAFT). Default cascades: ChannelAccount.defaultAiMode → Organization default → OFF.
</Accordion>

<Accordion title="Three-tier template system">
`MessageTemplate` supports three types: `META_APPROVED` (platform-approved), `QUICK_REPLY` (agent shortcuts with variable resolution), and `AI_PROMPT` (AI system prompts with optional SystemPrompt link).
</Accordion>

<Accordion title="Personal accounts OAuth and security">
Personal accounts share org WABA token for WhatsApp but use their own Page Access Token for Instagram/Messenger. OAuth state includes `level` for defense-in-depth validation.
</Accordion>
</AccordionGroup>

## Multi-Tenancy Patterns

<Warning>
The messaging module introduces unique multi-tenancy challenges because webhooks arrive without org context.
</Warning>

### Two-Step RLS Bypass (Webhook Processing)

The webhook controller receives events for ALL organizations from a single Meta App. Org context is unknown at arrival time.

<CodeGroup>
```typescript Step 1: Find org
// Step 1: Find which org owns this account (bypass RLS)
const account = await this.tenantContext.executeReadOnlyWithBypass(async (em) => {
  return em.findOne(ChannelAccount, { externalAccountId: job.data.accountId });
});
```

```typescript Step 2: Process in org context
// Step 2: Process within that org's context
await this.tenantContext.executeInOrg(
  account.organization.id,
  async (em) => {
    await this.processMessageInTransaction(em, job.data);
  },
  { userId: undefined }, // system action, no user
);
```
</CodeGroup>

### Composable `*InTransaction` Pattern

Services that participate in existing transactions expose `*InTransaction` methods:

```typescript
// Public API — wraps TenantContext
async matchOrCreate(channel, identifier, profileData, orgId): Promise<MatchResult>;

// Composable — accepts EntityManager from caller's transaction
async matchOrCreateInTransaction(em, channel, identifier, profileData, orgId): Promise<MatchResult>;
```

<Check>
The `em` parameter must always be the one provided by the TenantContext callback — never `this.em`.
</Check>

## Message Flows

<Steps>
<Step title="Inbound Message Processing">
1. **Webhook Receipt**: Meta sends webhook to `/messaging/webhook`
2. **Signature Validation**: `MetaWebhookGuard` validates `X-Hub-Signature-256`
3. **Event Logging**: Persist to `WebhookEventLog` with unknown org
4. **Queue Enqueue**: Add to `webhook-processor` queue
5. **Organization Resolution**: Find org by `externalAccountId` 
6. **Message Processing**: Create/update entities within org context
7. **Event Emission**: Trigger WebSocket and notification events
</Step>

<Step title="Outbound Message Flow">
1. **Message Creation**: Agent/system creates outbound message
2. **Outbox Entry**: Transactional write to `MessageOutbox`
3. **Queue Processing**: `message-sender` queue picks up outbox entry
4. **API Call**: Send via channel provider (WhatsApp/Instagram/Messenger)
5. **Status Update**: Update message status based on API response
6. **Cleanup**: Remove successful outbox entries
</Step>

<Step title="Media Handling">
1. **Inbound Media**: Download and store via `media-downloader` queue
2. **URL Generation**: Generate secure, time-limited access URLs
3. **Outbound Media**: Upload to Meta and reference in message payload
</Step>
</Steps>

## Business Rules

### Conversation Management

<Note>
**Archive vs Status**: `isArchived` is independent of `status`. An archived conversation can be OPEN or CLOSED.
</Note>

- Conversations auto-create on first inbound message
- Archive state persists across status changes
- Last message timestamp drives conversation ordering
- AI mode cascades: Conversation → ChannelAccount → Organization → OFF

### Assignment Rules

<Tabs>
<Tab title="Assignment Types">
- **Direct Assignment**: `user_id` set, `team_id` null
- **Team Pool**: `user_id` null, `team_id` set  
- **Agent on Team**: Both `user_id` and `team_id` set
- **Multiple Assignments**: Conversation can have multiple assignment rows
</Tab>

<Tab title="Permission Logic">
- `MESSAGING_MANAGE`: Full access to all conversations
- `MESSAGING_WRITE`: View + reply to assigned/team conversations
- Personal account owner: View + reply to their account conversations
- Team member: View + reply based on team assignment's `canReply` flag
</Tab>
</Tabs>

### Message Constraints

- Messages are immutable after creation (no updates/deletes)
- External message IDs must be unique per channel account
- Outbound messages require active channel account
- Media messages auto-download for security and availability

### Core Permissions

| Permission | Scope | Description |
|------------|-------|-------------|
| `messaging.manage` | Organization | Full messaging admin access |

## Messaging-Specific Conventions

### Error Handling

<Tabs>
<Tab title="Webhook Processing">
- **Signature validation failures**: Return 401, log security event
- **Processing failures**: Return 200, retry via queue with exponential backoff  
- **Idempotency**: Skip duplicate events based on `externalEventId`
</Tab>

<Tab title="Message Sending">
- **Rate limiting**: Respect Meta platform limits per channel
- **Retry strategy**: Exponential backoff with max 5 attempts
- **Permanent failures**: Mark message as FAILED, notify user
</Tab>

<Tab title="Media Handling">
- **Download timeouts**: 30 second limit, fallback to external URL
- **Size limits**: Enforce platform-specific media size restrictions
- **Format validation**: Check file types against channel capabilities
</Tab>
</Tabs>

### Performance Optimizations

- **Conversation queries**: Eager load last message and assignment data
- **Message pagination**: Cursor-based pagination for large conversations  
- **WebSocket batching**: Batch rapid-fire events to prevent client overload
- **Cache strategies**: Redis cache for template data and AI suggestions

## Query Patterns

<AccordionGroup>
<Accordion title="Conversation Queries">

```sql
-- List conversations with last message
SELECT c.*, m.text as last_message, m.timestamp as last_message_at
FROM conversation c
LEFT JOIN LATERAL (
  SELECT text, timestamp 
  FROM message 
  WHERE conversation_id = c.id 
  ORDER BY timestamp DESC 
  LIMIT 1
) m ON true
WHERE c.organization_id = ? AND c.is_archived = false
ORDER BY COALESCE(m.timestamp, c.created_at) DESC;
```

```sql
-- Find conversations by assignment
SELECT DISTINCT c.*
FROM conversation c
JOIN conversation_assignment ca ON ca.conversation_id = c.id
WHERE (ca.user_id = ? OR ca.team_id IN (?))
  AND c.organization_id = ?
  AND c.status = 'open';
```

</Accordion>

<Accordion title="Message Queries">

```sql
-- Messages with sender info for conversation
SELECT m.*, u.name as sender_name, u.avatar_url
FROM message m
LEFT JOIN "user" u ON u.id = m.sent_by_id
WHERE m.conversation_id = ?
ORDER BY m.timestamp ASC
LIMIT ? OFFSET ?;
```

```sql
-- Unread message counts by conversation
SELECT conversation_id, COUNT(*) as unread_count

  AND direction = 'inbound'
  AND read_at IS NULL
```

## Error Handling & Retry Strategy

### Queue Processing Errors

<Steps>
<Step title="Transient Errors">
**Network timeouts, API rate limits, temporary service unavailability**
- Exponential backoff: 1s, 2s, 4s, 8s, 16s
- Max 5 retry attempts
- Move to dead letter queue after max attempts
</Step>

<Step title="Permanent Errors">
**Invalid webhook signatures, malformed payloads, deleted accounts**  
- No retries
- Log error details
- Mark as permanently failed
</Step>

<Step title="Partial Failures">
**Some messages in batch succeed, others fail**
- Process successful messages
- Retry only failed messages
- Maintain processing order where possible
</Step>
</Steps>

### API Error Responses

```typescript
// Standard error format
{
  "error": "CONVERSATION_NOT_FOUND",
  "message": "Conversation not found or access denied",
  "statusCode": 404,
  "timestamp": "2026-04-15T10:30:00Z",
  "path": "/messaging/conversations/123"

```

Common error codes:
- `CHANNEL_ACCOUNT_INACTIVE`: Account disabled or expired
- `MESSAGE_SEND_FAILED`: External API returned error
- `INVALID_TEMPLATE`: Template not found or variables missing
- `ASSIGNMENT_CONFLICT`: User already assigned to conversation

- Plan DNS/load balancer updates
</Step>

- Route new conversations to unified module
- Maintain legacy webhooks for existing conversations

- Verify all integrations work with new module

## Known Gaps & Technical Debt

<Warning>
**High Priority Items**
</Warning>

- **Message encryption**: No encryption at rest for sensitive content
- **Advanced routing**: No smart assignment based on agent skills/availability  
- **Conversation merging**: No support for duplicate conversation cleanup
- **Bulk operations**: No bulk message sending or conversation management

<Info>
**Medium Priority Items**  
</Info>

- **Message search**: Basic text search, no full-text indexing
- **Analytics**: Limited conversation/agent performance metrics
- **Template variables**: Basic substitution, no complex logic
- **Webhook ordering**: No guaranteed webhook processing order
