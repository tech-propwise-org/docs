## Overview

The Messaging module provides a unified, channel-agnostic messaging system for WhatsApp, Instagram, and Facebook Messenger. It replaces the separate per-channel modules with shared entities, a shared queue, and a single WebSocket namespace.

### Problem → Solution

| Problem | Solution |
|---------|----------|
| Duplicated logic across WhatsApp and Instagram modules | Single `MessagingModule` with channel providers |
| No webhook signature validation (security gap) | Shared `MetaWebhookGuard` validates `X-Hub-Signature-256` |
| Inconsistent WebSocket auth (Instagram gateway has no JWT) | Single `/messaging` gateway with JWT auth |
| No Facebook Messenger support | Third channel provider |
| Separate entity schemas per channel | Unified entities: `Conversation`, `Message`, `ChannelAccount` |
| No shared queue infrastructure | Shared `PgBossQueueService` for messaging + notifications |

### Key Design Decisions

<AccordionGroup>
<Accordion title="Queue Infrastructure">
**pg-boss over BullMQ** — Project already uses pg-boss for notifications. No new Redis dependency. Interface-based design (`IQueueService`) allows swapping later.
</Accordion>

<Accordion title="CRM Integration">
**Direct PersonChannel FK on Conversation** — Conversations link directly to the CRM's `PersonChannel` via FK. Simpler model, no bidirectional sync overhead. The lead FK was moved from Conversation to Lead (`Lead.sourceConversation`).
</Accordion>

<Accordion title="Archive Strategy">
**Archive as boolean, not status** — `Conversation.isArchived` is orthogonal to `status` (OPEN/CLOSED), following `ARCHIVE_SYSTEM_SPECIFICATION.md`. Standard inbox rail reads keep the default `active` filter enabled.
</Accordion>

<Accordion title="Assignment Model">
**ConversationAssignment entity** — Conversations use a dedicated `conversation_assignment` table instead of the CRM `entity_stakeholder` pattern.
</Accordion>

<Accordion title="AI Agent Integration">
**Per-conversation AI mode with cascade** — Each conversation has an `aiMode` field (OFF, AUTO_REPLY, SUGGEST_ONLY, DRAFT). Default cascades: ChannelAccount.defaultAiMode → Organization default → OFF.
</Accordion>

<Accordion title="Message Delivery">
**Transactional outbox** — Outbound messages use an outbox table written in the same DB transaction as the Message entity, guaranteeing at-least-once delivery.
</Accordion>

<Accordion title="Template System">
**Three-tier template system** — `MessageTemplate` supports three types: `META_APPROVED` (platform-approved), `QUICK_REPLY` (agent shortcuts with variable resolution), and `AI_PROMPT` (AI system prompts).
</Accordion>

<Accordion title="Personal Account Authentication">
**Personal accounts share org WABA token** — WhatsApp personal accounts reuse the organization's WABA access token (same Business Account). Instagram and Messenger personal accounts use their own Page Access Token obtained via OAuth.
</Accordion>
</AccordionGroup>

## Message Flows

### Inbound Message Flow

<Steps>
<Step title="Webhook Receipt">
Platform webhook delivers message to Meta webhook endpoint with signature validation
</Step>
<Step title="Channel Provider Processing">
Appropriate channel provider processes platform-specific payload format
</Step>
<Step title="Conversation Resolution">
Find or create conversation based on `channelAccount + personChannel` mapping
</Step>
<Step title="Message Storage">
Store message entity with conversation link and metadata
</Step>
<Step title="AI Processing">
If AI mode is enabled, queue AI agent processing for automated responses
</Step>
<Step title="WebSocket Broadcast">
Broadcast new message to relevant WebSocket rooms (`org:${orgId}`, `conversation:${conversationId}`)
</Step>
<Step title="Notification Dispatch">
Queue notifications for assigned agents and stakeholders
</Step>
</Steps>

### Outbound Message Flow

<Steps>
<Step title="Message Creation">
Create outbound message record with PENDING status
</Step>
<Step title="Outbox Entry">
Write to outbox table in same transaction for guaranteed delivery
</Step>
<Step title="Queue Processing">
pg-boss worker processes outbox entry for platform delivery
</Step>
<Step title="Platform API Call">
Channel provider makes authenticated API call to messaging platform
</Step>
<Step title="Status Update">
Update message status to SENT/FAILED based on platform response
</Step>
<Step title="WebSocket Broadcast">
Broadcast status update to connected clients
</Step>
</Steps>

### Access Control Logic

<CodeGroup>
```typescript Conversation Access
async checkConversationAccess(
  userId: string, 
  conversationId: string, 
  permission: MessagingPermission
): Promise<boolean> {
  // Check if user has global permission
  if (await this.hasPermission(userId, 'messaging:view_all_conversations')) {
    return true;
  }
  
  // Check assignment-based access
  const assignment = await this.findAssignment(conversationId, userId);
  return assignment !== null;
}
```

```typescript Personal Account Access
async checkPersonalAccountAccess(
  userId: string, 
  channelAccountId: string
): Promise<boolean> {
  const account = await this.findChannelAccount(channelAccountId);
  
  if (account.level === AccountLevel.ORGANIZATION) {
    return this.hasPermission(userId, 'messaging:manage_channel_accounts');
  }
  
  // Personal accounts can only be accessed by owner
  return account.ownerId === userId;
}
```
</CodeGroup>

## WebSocket Events & Room Architecture

### Room Structure

The messaging module uses a hierarchical WebSocket room structure:

```typescript
// Organization-wide room for inbox updates
const orgRoom = `org:${organizationId}`;

// Conversation-specific room for message updates  
const conversationRoom = `conversation:${conversationId}`;

// User-specific room for assignments and notifications
const userRoom = `user:${userId}`;
```

## Error Handling & Retry Strategy

### Webhook Processing

<Steps>
<Step title="Signature Validation">
Validate Meta webhook signature using `X-Hub-Signature-256` header
</Step>
<Step title="Idempotency Check">
Check for duplicate webhook delivery using external message ID
</Step>
<Step title="Graceful Degradation">
Log unknown webhook formats but don't fail processing
</Step>
<Step title="Dead Letter Queue">
Failed webhook processing moves to DLQ for manual review
</Step>
</Steps>

### Message Delivery

```typescript
const retryConfig = {
  attempts: 3,
  backoff: {
    type: 'exponential',
    delay: 2000,
    factor: 2,
    maxDelay: 30000
  }
};

// Outbox processing with retry
await this.queueService.add('send-message', messageData, {
  retry: retryConfig,
  removeOnComplete: 10,
  removeOnFail: 50
});
```

<Warning>
Failed messages remain in FAILED status and require manual retry or investigation
</Warning>
