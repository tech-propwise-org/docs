## Architecture Overview

The messaging system follows a conversation-centric architecture with several key design principles:

### Core Design Decisions

<Tabs>
  <Tab title="Channel Abstraction">
    Provides a unified interface across WhatsApp, Instagram, and Facebook Messenger, allowing consistent handling of different messaging platforms.
  </Tab>
  <Tab title="Transactional Outbox">
    Ensures reliable message delivery using the outbox pattern, preventing message loss during system failures.
  </Tab>
  <Tab title="Contact Resolution">
    Conversations link to the CRM's PersonChannel entity for seamless contact management and relationship tracking.
  </Tab>
  <Tab title="AI Mode Control">
    Configurable AI assistance at the conversation level with intelligent cascade from ChannelAccount → Organization → default settings.
  </Tab>
</Tabs>

### Message Flow

<Steps>
  <Step title="Incoming Message">
    Messages arrive via webhooks from messaging platforms and are processed by the appropriate channel handler
  </Step>
  <Step title="Conversation Resolution">
    System identifies or creates conversations based on PersonChannel relationships
  </Step>
  <Step title="AI Processing">
    Based on conversation AI mode, messages may trigger automated responses, suggestions, or drafts
  </Step>
  <Step title="Outbound Delivery">
    Outgoing messages are queued and delivered using the transactional outbox pattern
  </Step>
</Tabs>

## Core Entities

### ChannelAccount

Represents authenticated accounts for messaging channels at organization or personal level.

<AccordionGroup>
  <Accordion title="Core Fields">
    | Field | Type | Description |
    |-------|------|-------------|
    | `id` | UUID | Primary key |
    | `organization` | Organization | Reference to organization |
    | `channel` | Channel | Channel type (WHATSAPP, INSTAGRAM, MESSENGER) |
    | `level` | AccountLevel | ORGANIZATION or PERSONAL |
    | `accountName` | string | Human-readable account name |
    | `externalAccountId` | string | External account identifier |
  </Accordion>

  <Accordion title="Authentication Fields">
    | Field | Type | Description |
    |-------|------|-------------|
    | `authMethod` | AuthMethod | OAuth method: FACEBOOK_LOGIN, INSTAGRAM_LOGIN, or EMBEDDED_SIGNUP |
    | `accessTokenEncrypted` | string? | Channel API access token (encrypted) |
    | `refreshTokenEncrypted` | string? | Long-lived user token (encrypted) |
    | `tokenExpiresAt` | Date? | Expiry time for refreshable tokens |
    | `tokenIssuedAt` | Date? | When current access token was issued |
  </Accordion>

  <Accordion title="Configuration Fields">
    | Field | Type | Description |
    |-------|------|-------------|
    | `defaultAiMode` | AiMode? | Default AI mode for new conversations |
    | `routingStrategy` | RoutingStrategy | POOL (default), ROUND_ROBIN, or LEAST_BUSY |
    | `healthStatus` | HealthStatus | HEALTHY (default), DEGRADED, or UNHEALTHY |
    | `isActive` | boolean | Account activation status |
  </Accordion>
</AccordionGroup>

### Conversation

Represents a conversation thread between contacts and agents on a specific channel.

<AccordionGroup>
  <Accordion title="Relationship Fields">
    | Field | Type | Description |
    |-------|------|-------------|
    | `channelAccount` | ChannelAccount | The channel account handling this conversation |
    | `personChannel` | PersonChannel? | The contact's PersonChannel for this conversation |
    | `person` | Person? | Associated CRM person |
    | `lead` | Lead? | Associated lead |
    | `deal` | Deal? | Associated deal |
  </Accordion>

  <Accordion title="Status Fields">
    | Field | Type | Description |
    |-------|------|-------------|
    | `status` | ConversationStatus | OPEN or CLOSED |
    | `unreadCount` | number | Unread message count |
    | `lastMessageAt` | Date? | Timestamp of last message |
    | `isWindowOpen` | boolean | Whether 24h messaging window is active |
    | `windowExpiresAt` | Date? | When messaging window expires |
  </Accordion>

  <Accordion title="AI Configuration">
    | Field | Type | Description |
    |-------|------|-------------|
    | `aiMode` | AiMode | OFF, AUTO_REPLY, SUGGEST_ONLY, DRAFT |
    | `aiEscalatedAt` | Date? | When AI escalated to human |
    | `aiMessageCount` | number | Count of AI-generated messages |
    | `humanAgentTagUsed` | boolean | Whether HUMAN_AGENT tag was used |
  </Accordion>
</AccordionGroup>

<Note>
Conversations use entity filters for data management:
- `isDeleted` (default=true): Excludes soft-deleted conversations
- `active` (default=true): Excludes archived conversations

When querying archived conversations, disable the `active` filter: `{ filters: { active: false } }`
</Note>

## AI Mode Configuration

The messaging system supports sophisticated AI assistance with four operational modes:

<Tabs>
  <Tab title="OFF">
    AI assistance is completely disabled. All responses must be manual.
  </Tab>
  <Tab title="AUTO_REPLY">
    AI automatically generates and sends responses without human intervention.
  </Tab>
  <Tab title="SUGGEST_ONLY">
    AI provides response suggestions that agents can review and send manually.
  </Tab>
  <Tab title="DRAFT">
    AI creates draft responses that agents can edit before sending.
  </Tab>
</Tabs>

<Info>
AI mode cascades from conversation-specific settings → ChannelAccount defaults → Organization defaults → system default (OFF).
</Info>

## Routing and Assignment

The messaging system provides flexible routing strategies for incoming conversations:

### Routing Strategies

- **POOL**: Default strategy where agents can pick up available conversations
- **ROUND_ROBIN**: Automatically distributes conversations evenly among available agents
- **LEAST_BUSY**: Routes to the agent with the fewest active conversations

### Team-Based Routing

Conversations can be assigned to specific teams based on:
- Channel account configuration
- Content analysis and classification
- Business rules and escalation policies
- Agent availability and expertise

<Note>
Routing configuration is set at the ChannelAccount level and can be customized per messaging channel.
</Note>
