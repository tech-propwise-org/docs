## Overview

The Messaging module provides a unified, channel-agnostic messaging system for WhatsApp, Instagram, and Facebook Messenger. It replaces the separate per-channel modules with shared entities, a shared queue, and a single WebSocket namespace.

### Problem → Solution

| Problem | Solution |
| ------- | -------- |
| Duplicated logic across WhatsApp and Instagram modules | Single `MessagingModule` with channel providers |
| No webhook signature validation (security gap) | Shared `MetaWebhookGuard` validates `X-Hub-Signature-256` |
| Inconsistent WebSocket auth (Instagram gateway has no JWT) | Single `/messaging` gateway with JWT auth |
| No Facebook Messenger support | Third channel provider |
| Separate entity schemas per channel | Unified entities: `Conversation`, `Message`, `ChannelAccount` |
| No shared queue infrastructure | Shared `PgBossQueueService` for messaging + notifications |

## Architecture & Module Structure

<Note>
The unified messaging module follows a channel provider pattern to support multiple platforms while maintaining shared infrastructure.
</Note>

### Key Design Decisions

<AccordionGroup>
<Accordion title="Queue Technology">
**pg-boss over BullMQ** — Project already uses pg-boss for notifications. No new Redis dependency. Interface-based design (`IQueueService`) allows swapping later.
</Accordion>

<Accordion title="CRM Integration">
**Direct PersonChannel FK on Conversation** — Conversations link directly to the CRM's `PersonChannel` via FK. Simpler model, no bidirectional sync overhead.
</Accordion>

<Accordion title="Lifecycle Management">
**Organization lifecycle integration** — MessagingOrganizationLifecycleListener handles organization deletion/restore by disconnecting WebSocket clients organization-wide and pausing/resuming Meta channel accounts non-destructively.
</Accordion>

<Accordion title="Archive Strategy">
**Archive as boolean, not status** — `Conversation.isArchived` is orthogonal to `status` (OPEN/CLOSED), following `ARCHIVE_SYSTEM_SPECIFICATION.md`.
</Accordion>

<Accordion title="Assignment Model">
**`ConversationAssignment` entity (not `entity_stakeholder`)** — Conversations use a dedicated `conversation_assignment` table instead of the CRM `entity_stakeholder` pattern.
</Accordion>

<Accordion title="AI Integration">
**AI Agent Integration** — The messaging module integrates with the AI module to provide automated responses through AI agents with `aiMode` control.
</Accordion>

<Accordion title="Message Delivery">
**Transactional outbox** — Outbound messages use an outbox table written in the same DB transaction as the Message entity, guaranteeing at-least-once delivery.
</Accordion>

<Accordion title="Template System">
**Three-tier template system** — `MessageTemplate` supports `META_APPROVED`, `QUICK_REPLY`, and `AI_PROMPT` types.
</Accordion>

<Accordion title="Account Management">
**Personal accounts share org WABA token** — WhatsApp personal accounts reuse the organization's WABA access token. Instagram and Messenger use separate Page Access Tokens.
</Accordion>
</AccordionGroup>

## Message Flows

### Inbound Message Flow

<Steps>
<Step title="Webhook Receipt">
Meta webhook delivers message to unified endpoint with signature validation
</Step>
<Step title="Channel Processing">
Channel-specific provider processes platform-specific payload
</Step>
<Step title="Entity Creation">
Create/update Conversation, Message, and PersonChannel entities
</Step>
<Step title="Assignment Logic">
Apply auto-assignment rules and stakeholder creation
</Step>
<Step title="AI Processing">
Evaluate AI mode and queue AI agent execution if enabled
</Step>
<Step title="Notifications">
Generate notifications for assigned users/teams
</Step>
<Step title="WebSocket Broadcast">
Broadcast updates to relevant WebSocket rooms
</Step>
</Steps>

### Outbound Message Flow

<Steps>
<Step title="Message Creation">
Create Message entity and MessageOutbox entry in transaction
</Step>
<Step title="Queue Processing">
PgBoss worker processes outbox entries
</Step>
<Step title="Platform Delivery">
Channel provider sends to Meta Graph API
</Step>
<Step title="Status Updates">
Process delivery receipts and update message status
</Step>
<Step title="WebSocket Updates">
Broadcast status changes to connected clients
</Step>
</Steps>

## Business Rules

<Warning>
These rules are critical for maintaining data integrity and proper messaging behavior.
</Warning>

### Conversation Rules

1. **Unique Conversation per PersonChannel + ChannelAccount** — Only one active conversation exists for each combination
2. **Auto-Lead Creation** — Personal channel accounts with `autoCreateLead=true` create leads for new conversations
3. **Archive Independence** — Archived conversations can receive new messages without unarchiving
4. **Deletion Behavior** — Soft-deleted conversations are terminal; new messages create replacement conversations

### Assignment Rules

1. **Owner Protection** — Personal account owners get protected `isOwner=true` assignments
2. **Multiple Assignments** — Conversations support both direct user and team pool assignments
3. **Transfer History** — All assignment changes are tracked via WebSocket and notification events

### AI Mode Rules

1. **Cascading Defaults** — ChannelAccount.defaultAiMode → Organization default → OFF
2. **Inbound Coalescing** — AI processing debounces multiple rapid messages
3. **Credit Gating** — AI operations respect subscription module credit limits

## Query Patterns

### Common Queries

<CodeGroup>
```sql Active Conversations
-- Get active conversations for user
SELECT c.* FROM conversation c
JOIN conversation_assignment ca ON ca.conversation_id = c.id
WHERE c.organization_id = $1
  AND c.is_deleted = false
  AND c.is_archived = false
  AND ca.user_id = $2
  AND ca.is_active = true
ORDER BY c.last_message_at DESC;
```

```sql Conversation History
-- Get conversation with message history
SELECT 
  c.*,
  m.id as message_id,
  m.content as message_content,
  m.direction as message_direction,
  m.timestamp as message_timestamp
FROM conversation c
LEFT JOIN message m ON m.conversation_id = c.id
WHERE c.id = $1
  AND c.organization_id = $2
  AND c.is_deleted = false
ORDER BY m.timestamp ASC;
```

```sql Channel Account Status
-- Get channel accounts with connection status
SELECT 
  ca.*,
  COUNT(CASE WHEN c.is_deleted = false AND c.is_archived = false THEN 1 END) as active_conversations
FROM channel_account ca
LEFT JOIN conversation c ON c.channel_account_id = ca.id
WHERE ca.organization_id = $1
  AND ca.is_active = true
GROUP BY ca.id
ORDER BY ca.created_at DESC;
```
</CodeGroup>

## Error Handling & Retry Strategy

<Warning>
Proper error handling is critical for maintaining message delivery guarantees.
</Warning>

### Retry Patterns

<Tabs>
<Tab title="Outbound Messages">
- **Exponential Backoff**: 1s, 2s, 4s, 8s, 16s, 32s
- **Max Attempts**: 6
- **Dead Letter**: After max attempts, mark as FAILED
- **Manual Retry**: Support admin manual retry for failed messages
</Tab>

<Tab title="Webhook Processing">
- **Immediate Retry**: 3 attempts with 100ms delay
- **Queue Retry**: If immediate fails, queue for background processing
- **Duplicate Detection**: Use webhook ID to prevent duplicate processing
</Tab>

<Tab title="AI Agent Execution">
- **Credit Check**: Validate credits before processing
- **Timeout**: 30s timeout per AI request
- **Fallback**: On failure, optionally create draft for human review
</Tab>
</Tabs>

## Deployment Considerations

<Note>
The messaging module requires careful coordination during deployment due to its real-time nature.
</Note>

### Database Migrations
- Use transactional migrations for schema changes
- Backfill data in separate background jobs
- Validate foreign key constraints before deployment

### Queue Workers
- Deploy workers before API changes
- Use graceful shutdown for in-flight messages
- Monitor queue depth during deployment

### WebSocket Connections
- Implement graceful connection migration
- Use cluster-wide disconnect for emergency maintenance
- Validate JWT tokens after auth service changes
