## Overview

The Messaging module provides a unified, channel-agnostic messaging system for WhatsApp, Instagram, and Facebook Messenger. It replaces the separate per-channel modules with shared entities, a shared queue, and a single WebSocket namespace.

<Info>
Last Updated: 2026-06-01  
Status: Active
</Info>

### Problem → Solution

| Problem | Solution |
|---------|----------|
| Duplicated logic across WhatsApp and Instagram modules | Single `MessagingModule` with channel providers |
| No webhook signature validation (security gap) | Shared `MetaWebhookGuard` validates `X-Hub-Signature-256` |
| Inconsistent WebSocket auth (Instagram gateway has no JWT) | Single `/messaging` gateway with JWT auth |
| No Facebook Messenger support | Third channel provider |
| Separate entity schemas per channel | Unified entities: `Conversation`, `Message`, `ChannelAccount` |
| No shared queue infrastructure | Shared `PgBossQueueService` for messaging + notifications |

### Key Design Decisions

<AccordionGroup>
  <Accordion title="Queue Infrastructure: pg-boss over BullMQ">
    Project already uses pg-boss for notifications. No new Redis dependency. Interface-based design (`IQueueService`) allows swapping later.
  </Accordion>

  <Accordion title="Direct PersonChannel FK on Conversation">
    Conversations link directly to the CRM's `PersonChannel` via FK. Simpler model, no bidirectional sync overhead. The lead FK was moved from Conversation to Lead (`Lead.sourceConversation`).
    
    <Note>
    Conversations may hold a nullable `contact_id`; when duplicate Persons/Contacts are merged, conversations pointing at absorbed Contacts are re-pointed to the surviving Contact.
    </Note>
  </Accordion>

  <Accordion title="Archive as boolean, not status">
    `Conversation.isArchived` is orthogonal to `status` (OPEN/CLOSED), following `ARCHIVE_SYSTEM_SPECIFICATION.md`. Standard inbox rail reads keep the default `active` filter enabled, so archived conversations are hidden from the live Inbox.
  </Accordion>

  <Accordion title="ConversationAssignment entity">
    Conversations use a dedicated `conversation_assignment` table instead of the CRM `entity_stakeholder` pattern. Each assignment is one row with nullable `user_id` and `team_id`.
  </Accordion>

  <Accordion title="AI Agent Integration">
    Comprehensive AI module integration with agent templates, knowledge base, credit management, and bidirectional workflow support. Each conversation has an `aiMode` field (OFF, AUTO_REPLY, SUGGEST_ONLY, DRAFT).
  </Accordion>

  <Accordion title="Three-tier template system">
    `MessageTemplate` supports three types: `META_APPROVED` (platform-approved), `QUICK_REPLY` (agent shortcuts), and `AI_PROMPT` (AI system prompts).
  </Accordion>
</AccordionGroup>

### Conversation Entity

```typescript
@Entity('conversation')
export class Conversation {
  @PrimaryGeneratedColumn('uuid')
  id: string;

  @Column({ name: 'person_channel_id' })
  personChannelId: string;

  @Column({ name: 'channel_account_id' })
  channelAccountId: string;

  @Column({ name: 'contact_id', nullable: true })
  contactId?: string;

  @Column({
    type: 'enum',
    enum: ConversationStatus,
    default: ConversationStatus.OPEN
  })
  status: ConversationStatus;

  @Column({ name: 'is_archived', default: false })
  isArchived: boolean;

  @Column({
    type: 'enum',
    enum: AiMode,
    default: AiMode.OFF
  })
  aiMode: AiMode;

  @Column({ name: 'last_message_at' })
  lastMessageAt: Date;

  // Relations
  @ManyToOne(() => PersonChannel)
  @JoinColumn({ name: 'person_channel_id' })
  personChannel: PersonChannel;

  @ManyToOne(() => ChannelAccount)
  @JoinColumn({ name: 'channel_account_id' })
  channelAccount: ChannelAccount;

  @OneToMany(() => Message, message => message.conversation)
  messages: Message[];

  @OneToMany(() => ConversationAssignment, assignment => assignment.conversation)
  assignments: ConversationAssignment[];
}
```

### Message Entity

```typescript
@Entity('message')
export class Message {
  @PrimaryGeneratedColumn('uuid')
  id: string;

  @Column({ name: 'conversation_id' })
  conversationId: string;

  @Column({
    type: 'enum',
    enum: MessageDirection,
  })
  direction: MessageDirection;

  @Column({
    type: 'enum',
    enum: MessageType,
  })
  type: MessageType;

  @Column({ type: 'text', nullable: true })
  content?: string;

  @Column({ type: 'jsonb', nullable: true })
  metadata?: Record<string, any>;

  @Column({ name: 'external_id', nullable: true })
  externalId?: string;

  @Column({
    type: 'enum',
    enum: MessageStatus,
    default: MessageStatus.PENDING
  })
  status: MessageStatus;

  @Column({ name: 'sent_at', nullable: true })
  sentAt?: Date;

  @Column({ name: 'delivered_at', nullable: true })
  deliveredAt?: Date;

  @Column({ name: 'read_at', nullable: true })
  readAt?: Date;

  // Relations
  @ManyToOne(() => Conversation)
  @JoinColumn({ name: 'conversation_id' })
  conversation: Conversation;
}
```

## Message Flows

### Inbound Message Flow

<Steps>
  <Step title="Webhook Received">
    Platform webhook hits unified endpoint with signature validation
  </Step>
  <Step title="Channel Provider Processing">
    Appropriate provider (WhatsApp/Instagram/Messenger) processes webhook payload
  </Step>
  <Step title="Conversation Resolution">
    Find or create conversation based on `channelAccount + personChannel`
  </Step>
  <Step title="Message Storage">
    Store message with unified schema and metadata
  </Step>
  <Step title="Real-time Notification">
    WebSocket broadcast to relevant rooms and users
  </Step>
  <Step title="AI Processing">
    Queue AI agent execution if conversation has appropriate `aiMode`
  </Step>
</Steps>

### Outbound Message Flow

<Steps>
  <Step title="Message Queued">
    Message stored in outbox table within DB transaction
  </Step>
  <Step title="Queue Processing">
    pg-boss worker picks up outbound message job
  </Step>
  <Step title="Channel Provider Sending">
    Provider sends via platform API (WhatsApp/Instagram/Messenger)
  </Step>
  <Step title="Status Updates">
    Delivery receipts update message status
  </Step>
  <Step title="Real-time Updates">
    WebSocket broadcast of status changes
  </Step>
</Steps>

## Business Rules

### Conversation Lifecycle

<Warning>
**Archive vs Deletion**: Archived conversations remain accessible but are hidden from active inbox. Soft-deleted conversations are terminal tombstones.
</Warning>

<AccordionGroup>
  <Accordion title="Conversation Creation">
    - New conversations created when first message received for `channelAccount + personChannel`
    - Archived conversations are reused without unarchiving
    - Soft-deleted conversations may trigger replacement conversation creation
  </Accordion>

  <Accordion title="Assignment Rules">
    - Personal accounts: Auto-assign to account owner
    - Organization accounts: Follow team/user assignment rules
    - `conversation_assignment.is_owner` protects personal account assignments
  </Accordion>

  <Accordion title="AI Mode Cascade">
    Default AI mode follows hierarchy:
    1. Conversation-specific setting
    2. ChannelAccount.defaultAiMode
    3. Organization default
    4. OFF (fallback)
  </Accordion>
</AccordionGroup>

```typescript POST /api/messaging/accounts/:id/whatsapp/register
// Re-run WhatsApp Cloud API registration on an already-connected account.
// Lets an operator supply the existing two-step PIN so registration succeeds
// WITHOUT disabling two-step verification. Recovery path for Meta error
// 133005 (PIN mismatch) and for retrying a registration that failed during
// embedded signup (the short-lived OAuth `code` is single-use, so the
// connect endpoint cannot be replayed).
//
// Permissions: `messaging.manage` OR channel-account access.
// Throttle: 5 requests / minute (Meta locks out after repeated wrong PINs
// with code 133008).
// Coexistence numbers short-circuit to a no-op success.
POST /api/messaging/accounts/:id/whatsapp/register
{
  // Existing six-digit two-step verification PIN for the number. Supply
  // when the number already has two-step verification enabled; omit to
  // retry registration with the default PIN ('000000').
  twoStepPin?: string // max length 16
}

Response: VerifyAccountResultDto {
  isValid: boolean,
  message?: string
}
```

```typescript POST /api/messaging/accounts/:id/whatsapp/request-code
// Request a phone-ownership verification code (OTP) for a WhatsApp number
// blocked on Meta error 133006 ("Phone number re-verification needed").
// Meta sends a six-digit code to the number via SMS or voice call.
// Internally calls Meta `POST /{phoneNumberId}/request_code?code_method=…&language=…`.
//
// Permissions: `messaging.manage` OR channel-account access.
// Throttle: 5 requests / minute.
// Coexistence numbers are rejected.
POST /api/messaging/accounts/:id/whatsapp/request-code
{
  codeMethod: 'SMS' | 'VOICE', // required — delivery method
  language?: string            // optional locale, e.g. 'en_US' (default), max length 10
}

Response: VerifyAccountResultDto {
  isValid: boolean, // true when the code was requested successfully
  message?: string
}

// Errors:
// - 400 BadRequestException when Meta returns 136024 ("already verified");
//   the operator should use the PIN / register flow instead.
```

```typescript POST /api/messaging/accounts/:id/whatsapp/verify-code
// Submit the verification code (OTP) received via SMS or voice and finish
// Cloud API registration in one call. Internally calls Meta
// `POST /{phoneNumberId}/verify_code?code=…` and, on success, runs the
// registration step (`POST /{phoneNumberId}/register`).
//
// Permissions: `messaging.manage` OR channel-account access.
// Throttle: 5 requests / minute (Meta locks out after repeated wrong codes).
// Coexistence numbers are rejected.
POST /api/messaging/accounts/:id/whatsapp/verify-code
{
  code: string // required, max length 16 — the OTP received from Meta
}

Response: VerifyAccountResultDto {
  isValid: boolean, // false on a failed verify (registration is not attempted)
  message?: string
}
```
</CodeGroup>

<Note>
**WhatsApp Embedded Signup body** — `WhatsAppEmbeddedSignupDto` (sent to the connect endpoint) now accepts an optional `twoStepPin` field. Supply it during initial connect when the number already has two-step verification enabled so Cloud API registration succeeds in one shot. The backend otherwise registers with the default PIN.
</Note>

<Note>
**WhatsApp Cloud API registration health** — A WhatsApp number can be subscribed to a WABA's webhooks (so inbound webhooks arrive) while never being registered on the Cloud API. In that state every outbound send and delivery/read receipt fails with Meta error `133010` (`The account is not registered`). The module tracks registration on `ChannelAccount.metadata.cloudApiRegistered`, `metadata.cloudApiRegistrationIssue`, and `metadata.cloudApiNeedsReverification`, surfaces a **Not registered** badge with an in-app **Register on Cloud API** action, and exposes the three endpoints above as the self-serve recovery path. Coexistence numbers are managed by the WhatsApp Business App and are exempt from these checks.
</Note>

## Query Patterns

### Conversation Filtering

<Note>
Standard inbox queries include `active` filter by default to hide archived conversations.
</Note>

```typescript
// Active inbox (default)
repository.find({
  where: {
    isArchived: false,
    isDeleted: false,
    organizationId: req.user.organizationId
  },
  order: { lastMessageAt: 'DESC' }

// Archived conversations (explicit)
repository.find({
  where: {
    isArchived: true,
    isDeleted: false,
    organizationId: req.user.organizationId

// Assignment filtering
repository.createQueryBuilder('conversation')
  .leftJoin('conversation.assignments', 'assignment')
  .where('assignment.userId = :userId', { userId })
  .andWhere('conversation.isDeleted = false');
```

## Error Handling & Retry Strategy

### Webhook Processing

<Steps>
  <Step title="Immediate Validation">
    Signature validation with `MetaWebhookGuard`
  </Step>
  <Step title="Graceful Degradation">
    Store webhook payload even if processing fails
  </Step>
  <Step title="Exponential Backoff">
    Retry failed webhooks with increasing delays
  </Step>
  <Step title="Dead Letter Queue">
    Move permanently failed webhooks to DLQ for manual review
  </Step>
</Steps>

### Outbound Message Failures

<Warning>
Messages that fail to send are marked with `FAILED` status and require manual retry or investigation.
</Warning>

```typescript
// Retry configuration
const retryConfig = {
  attempts: 3,
  backoff: {
    type: 'exponential',
    delay: 2000
  },
  onFailedAttempt: (error) => {
    logger.warn(`Message send attempt failed`, error);
  }
};
```

## Legacy Module Removal

<Warning>
**Migration Timeline**: Legacy WhatsApp and Instagram modules will be deprecated in phases to ensure smooth transition.
</Warning>

### Phase 1: Parallel Operation
- New messaging module handles new conversations
- Legacy modules continue for existing conversations
- Gradual migration of active conversations

### Phase 2: Legacy Deprecation
- Stop new webhook routing to legacy modules
- Complete conversation migration
- Maintain read-only access to legacy data

### Phase 3: Complete Removal
- Remove legacy module code
- Archive legacy database tables
- Update all references and documentation
