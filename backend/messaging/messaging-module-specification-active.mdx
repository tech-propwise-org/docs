## Overview

The Messaging module provides a unified, channel-agnostic messaging system for WhatsApp, Instagram, and Facebook Messenger. It replaces the separate per-channel modules with shared entities, a shared queue, and a single WebSocket namespace.

### Problem → Solution

| Problem | Solution |
| --- | --- |
| Duplicated logic across WhatsApp and Instagram modules | Single `MessagingModule` with channel providers |
| No webhook signature validation (security gap) | Shared `MetaWebhookGuard` validates `X-Hub-Signature-256` |
| Inconsistent WebSocket auth (Instagram gateway has no JWT) | Single `/messaging` gateway with JWT auth |
| No Facebook Messenger support | Third channel provider |
| Separate entity schemas per channel | Unified entities: `Conversation`, `Message`, `ChannelAccount` |
| No shared queue infrastructure | Shared `PgBossQueueService` for messaging + notifications |

### Key Design Decisions

<AccordionGroup>
  <Accordion title="pg-boss over BullMQ">
    Project already uses pg-boss for notifications. No new Redis dependency. Interface-based design (`IQueueService`) allows swapping later.
  </Accordion>

  <Accordion title="Direct PersonChannel FK on Conversation">
    Conversations link directly to the CRM's `PersonChannel` via FK. Simpler model, no bidirectional sync overhead.
  </Accordion>

  <Accordion title="Organization lifecycle integration">
    MessagingOrganizationLifecycleListener handles organization deletion/restore by disconnecting WebSocket clients organization-wide and pausing/resuming Meta channel accounts non-destructively. The MessagingGateway provides `disconnectOrganization()` for cluster-wide client disconnection via the Postgres IO adapter.
  </Accordion>

  <Accordion title="Archive as boolean, not status">
    `Conversation.isArchived` is orthogonal to `status` (OPEN/CLOSED), following `ARCHIVE_SYSTEM_SPECIFICATION.md`. Standard inbox rail reads keep the default `active` filter enabled, so archived conversations are hidden from the live Inbox.
  </Accordion>

  <Accordion title="ConversationAssignment entity">
    Conversations use a dedicated `conversation_assignment` table instead of the CRM `entity_stakeholder` pattern. Each assignment is one row with nullable `user_id` and `team_id`.
  </Accordion>

  <Accordion title="Transactional outbox">
    Outbound messages use an outbox table written in the same DB transaction as the Message entity, guaranteeing at-least-once delivery.
  </Accordion>

  <Accordion title="AI Agent Integration">
    The messaging module integrates with the AI module to provide automated responses through AI agents. Conversations have an `aiMode` field that controls AI participation (OFF, AUTO_REPLY, SUGGEST_ONLY, DRAFT).
  </Accordion>

  <Accordion title="Three-tier template system">
    `MessageTemplate` supports three types: `META_APPROVED` (platform-approved), `QUICK_REPLY` (agent shortcuts with variable resolution), and `AI_PROMPT` (AI system prompts).
  </Accordion>

  <Accordion title="Personal accounts share org WABA token">
    WhatsApp personal accounts reuse the organization's WABA access token (same Business Account). Instagram and Messenger personal accounts use their own Page Access Token obtained via OAuth.
  </Accordion>
</AccordionGroup>

## Message Flows

### Inbound Message Processing

<Steps>
  <Step title="Webhook Reception">
    Meta webhook endpoints receive and validate incoming messages using `MetaWebhookGuard` for signature verification
  </Step>
  <Step title="Message Parsing">
    Channel-specific providers parse webhook payloads into unified message format
  </Step>
  <Step title="Conversation Resolution">
    System finds or creates conversation based on channel account and person channel
  </Step>
  <Step title="Message Storage">
    Message entity is created and stored in database with proper organization scoping
  </Step>
  <Step title="Real-time Notification">
    WebSocket event broadcasted to organization's messaging room
  </Step>
  <Step title="AI Processing">
    If AI mode is enabled, message is queued for AI agent processing
  </Step>
</Steps>

### Outbound Message Processing

<Steps>
  <Step title="Message Creation">
    User or AI agent creates outbound message through API
  </Step>
  <Step title="Transactional Outbox">
    Message and outbox entry created in same database transaction
  </Step>
  <Step title="Queue Processing">
    Background worker processes outbox entries for delivery
  </Step>
  <Step title="Channel Delivery">
    Channel provider handles platform-specific message delivery
  </Step>
  <Step title="Status Updates">
    Delivery confirmations update message status and broadcast via WebSocket
  </Step>
</Steps>

## Business Rules

<Warning>
These business rules are critical for maintaining data consistency and proper messaging behavior.
</Warning>

### Conversation Management

- **One conversation per channel account + person channel combination**
- **Archived conversations remain accessible but hidden from active inbox**
- **Soft-deleted conversations are terminal tombstones that can spawn replacement conversations**
- **AI mode cascades from ChannelAccount → Organization default → OFF**

### Assignment Rules

- **Personal account owners automatically get protected owner assignments**
- **Lead stakeholder auto-assignment occurs only for personal accounts with `autoCreateLead` enabled**
- **Multiple assignments per conversation supported (direct agent + team pool)**
- **Conversation deletion soft-deletes all related assignments**

### Message Ordering

- **Messages ordered by `sentAt` timestamp for proper conversation flow**
- **Outbound echo messages properly sequenced with inbound messages**
- **Historical message import respects existing conversation context**

## Error Handling & Retry Strategy

### Webhook Processing

<Steps>
  <Step title="Signature Validation">
    Validate `X-Hub-Signature-256` header using shared secret
  </Step>
  <Step title="Idempotency Check">
    Check for duplicate webhook delivery using message IDs
  </Step>
  <Step title="Graceful Degradation">
    Log errors but return 200 to prevent webhook retry storms
  </Step>
  <Step title="Queue Failed Processing">
    Queue failed webhook processing for later retry with exponential backoff
  </Step>
</Steps>

### Outbound Delivery

The outbox pattern ensures reliable message delivery:

```typescript
interface OutboxRetryConfig {
  maxRetries: 3;
  backoffMultiplier: 2;
  baseDelayMs: 1000;
  maxDelayMs: 30000;
}
```
