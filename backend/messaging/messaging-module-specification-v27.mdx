## Overview

The Messaging module provides a unified, channel-agnostic messaging system for WhatsApp, Instagram, and Facebook Messenger. It replaces the separate per-channel modules with shared entities, a shared queue, and a single WebSocket namespace.

<Info>
**Last Updated:** 2026-06-01  
**Status:** Active
</Info>

### Problem → Solution

| Problem | Solution |
|---------|----------|
| Duplicated logic across WhatsApp and Instagram modules | Single `MessagingModule` with channel providers |
| No webhook signature validation (security gap) | Shared `MetaWebhookGuard` validates `X-Hub-Signature-256` |
| Inconsistent WebSocket auth (Instagram gateway has no JWT) | Single `/messaging` gateway with JWT auth |
| No Facebook Messenger support | Third channel provider |
| Separate entity schemas per channel | Unified entities: `Conversation`, `Message`, `ChannelAccount` |
| No shared queue infrastructure | Shared `PgBossQueueService` for messaging + notifications |

## Architecture & Module Structure

### Key Design Decisions

<AccordionGroup>
<Accordion title="Queue Infrastructure">
**pg-boss over BullMQ** — Project already uses pg-boss for notifications. No new Redis dependency. Interface-based design (`IQueueService`) allows swapping later.
</Accordion>

<Accordion title="PersonChannel Integration">
**Direct PersonChannel FK on Conversation** — Conversations link directly to the CRM's `PersonChannel` via FK. Simpler model, no bidirectional sync overhead. The lead FK was moved from Conversation to Lead (`Lead.sourceConversation`).
</Accordion>

<Accordion title="Archive System">
**Archive as boolean, not status** — `Conversation.isArchived` is orthogonal to `status` (OPEN/CLOSED), following `ARCHIVE_SYSTEM_SPECIFICATION.md`. Standard inbox rail reads keep the default `active` filter enabled, so archived conversations are hidden from the live Inbox.
</Accordion>

<Accordion title="Assignment Model">
**`ConversationAssignment` entity** — Conversations use a dedicated `conversation_assignment` table instead of the CRM `entity_stakeholder` pattern for simplified messaging-specific assignment tracking.
</Accordion>
</AccordionGroup>

## Multi-Tenancy Patterns

### Organization vs Personal Accounts

<Tabs>
<Tab title="WhatsApp">
Personal accounts share the organization's WABA token (same Business Account)
</Tab>
<Tab title="Instagram & Messenger">
Personal accounts use their own Page Access Token obtained via OAuth
</Tab>
</Tabs>

<Warning>
OAuth state includes `level` for defense-in-depth. The HMAC-signed OAuth state payload carries a `level` field (`personal` | `organization`) to prevent cross-level misuse.
</Warning>

## Core Entities

### Conversation Entity

```typescript
interface Conversation {
  id: string;
  organizationId: string;
  channelAccountId: string;
  personChannelId: string;
  contactId?: string; // Nullable cached link
  status: ConversationStatus; // OPEN | CLOSED
  isArchived: boolean;
  aiMode: AiMode; // OFF | AUTO_REPLY | SUGGEST_ONLY | DRAFT
  lastMessageAt: Date;
  metadata: Record<string, any>;
}
```

### Message Entity

```typescript
interface Message {
  id: string;
  conversationId: string;
  direction: MessageDirection; // INBOUND | OUTBOUND
  type: MessageType; // TEXT | IMAGE | AUDIO | VIDEO | DOCUMENT | TEMPLATE
  content: string;
  metadata: Record<string, any>;
  status: MessageStatus; // PENDING | SENT | DELIVERED | READ | FAILED
  channelMessageId?: string;
  sentAt?: Date;
  deliveredAt?: Date;
  readAt?: Date;
}
```

interface ChannelAccount {

## Message Flows

<Steps>
<Step title="Inbound Message Processing">
1. Webhook receives message
2. `MetaWebhookGuard` validates signature
3. Channel provider processes webhook
4. Message saved to database
5. WebSocket event emitted
6. AI agent triggered (if enabled)
</Step>

<Step title="Outbound Message Processing">
1. Message created via API/WebSocket
2. Saved to outbox table
3. Queue worker picks up message
4. Channel provider sends to platform
5. Status updates via webhook
6. WebSocket event for status change
</Step>
</Steps>

## Business Rules

### Conversation Management

<Note>
- Conversations are created automatically on first message
- Archive state is independent of open/closed status
- Soft-deleted conversations are terminal tombstones
- AI mode cascades: ChannelAccount → Organization → OFF
</Note>

### Assignment Rules

```typescript
interface ConversationAssignment {
  conversationId: string;
  userId?: string;     // Direct assignment
  teamId?: string;     // Team assignment
  isOwner: boolean;    // Protected personal account owner
}
```

<Tip>
Multiple assignment rows per conversation are supported (e.g., direct agent plus team pool)
</Tip>

## Error Handling & Retry Strategy

### Webhook Processing

<Steps>
<Step title="Signature Validation">
Validate `X-Hub-Signature-256` header using `MetaWebhookGuard`
</Step>
<Step title="Idempotency">
Use `channelMessageId` to prevent duplicate message processing
</Step>
<Step title="Retry Logic">
Exponential backoff for failed webhook processing (max 3 attempts)
</Step>
</Steps>

### Outbound Message Delivery

```typescript
interface RetryStrategy {
  maxAttempts: 5;
  backoffMs: [1000, 2000, 5000, 10000, 30000];
  failureHandling: 'dead-letter-queue';
}
```
