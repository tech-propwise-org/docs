## Overview

The Messaging module provides a unified, channel-agnostic messaging system for WhatsApp, Instagram, and Facebook Messenger. It replaces the separate per-channel modules with shared entities, a shared queue, and a single WebSocket namespace.

### Problem → Solution

| Problem | Solution |
|---------|----------|
| Duplicated logic across WhatsApp and Instagram modules | Single `MessagingModule` with channel providers |
| No webhook signature validation (security gap) | Shared `MetaWebhookGuard` validates `X-Hub-Signature-256` |
| Inconsistent WebSocket auth (Instagram gateway has no JWT) | Single `/messaging` gateway with JWT auth |
| No Facebook Messenger support | Third channel provider |
| Separate entity schemas per channel | Unified entities: `Conversation`, `Message`, `ChannelAccount` |
| No shared queue infrastructure | Shared `PgBossQueueService` for messaging + notifications |

### Key Design Decisions

<AccordionGroup>
<Accordion title="Queue Infrastructure">
**pg-boss over BullMQ** — Project already uses pg-boss for notifications. No new Redis dependency. Interface-based design (`IQueueService`) allows swapping later.
</Accordion>

<Accordion title="CRM Integration">
**Direct PersonChannel FK on Conversation** — Conversations link directly to the CRM's `PersonChannel` via FK. Simpler model, no bidirectional sync overhead. The lead FK was moved from Conversation to Lead (`Lead.sourceConversation`).
</Accordion>

<Accordion title="Archive Pattern">
**Archive as boolean, not status** — `Conversation.isArchived` is orthogonal to `status` (OPEN/CLOSED), following `ARCHIVE_SYSTEM_SPECIFICATION.md`.
</Accordion>

<Accordion title="Assignment System">
**`ConversationAssignment` entity (not `entity_stakeholder`)** — Conversations use a dedicated `conversation_assignment` table instead of the CRM `entity_stakeholder` pattern. Each assignment is one row with nullable `user_id` and `team_id`.
</Accordion>

<Accordion title="Message Delivery">
**Transactional outbox** — Outbound messages use an outbox table written in the same DB transaction as the Message entity, guaranteeing at-least-once delivery.
</Accordion>
</AccordionGroup>

## Multi-Tenancy Patterns

<Warning>
The messaging module introduces unique multi-tenancy challenges because webhooks arrive without org context.
</Warning>

### Two-Step RLS Bypass (Webhook Processing)

The webhook controller receives events for ALL organizations from a single Meta App. Org context is unknown at arrival time.

<Steps>
<Step title="Find Organization">
```typescript
// Step 1: Find which org owns this account (bypass RLS)
const account = await this.tenantContext.executeReadOnlyWithBypass(async (em) => {
  return em.findOne(ChannelAccount, { externalAccountId: job.data.accountId });
});
```
</Step>

<Step title="Process in Context">
```typescript
// Step 2: Process within that org's context
await this.tenantContext.executeInOrg(
  account.organization.id,
  async (em) => {
    await this.processMessageInTransaction(em, job.data);
  },
  { userId: undefined }, // system action, no user
);
```
</Step>
</Steps>

### ChannelAccount Entity

```typescript
@Entity('channel_account')
export class ChannelAccount {
  @PrimaryGeneratedColumn('uuid')
  id: string;

  @Column({ type: 'enum', enum: Channel })
  channel: Channel;

  @Column({ name: 'external_account_id' })
  externalAccountId: string; // WhatsApp phone number ID, IG account ID, FB page ID

  @Column({ nullable: true, name: 'page_id' })
  pageId?: string; // Facebook Page ID for Instagram

  @Column({ name: 'display_name' })
  displayName: string;

  @Column({ name: 'profile_picture_url', nullable: true })
  profilePictureUrl?: string;

  @Column({ name: 'access_token', nullable: true })
  accessToken?: string; // Personal accounts only

  @Column({ name: 'webhook_verification_token', nullable: true })
  webhookVerificationToken?: string;

  @Column({ name: 'is_active', default: true })
  isActive: boolean;

  @Column({ 
    type: 'enum', 
    enum: AiMode, 
    name: 'default_ai_mode',
    default: AiMode.OFF 
  })
  defaultAiMode: AiMode;

  @ManyToOne(() => Organization)
  @JoinColumn({ name: 'organization_id' })
  organization: Organization;

  @ManyToOne(() => User, { nullable: true })
  @JoinColumn({ name: 'personal_account_user_id' })
  personalAccountUser?: User; // null = org account
}
```

### Conversation Entity

```typescript
@Entity('conversation')
export class Conversation {
  @PrimaryGeneratedColumn('uuid')
  id: string;

  @Column({ name: 'external_thread_id', nullable: true })
  externalThreadId?: string;

  @ManyToOne(() => ChannelAccount)
  @JoinColumn({ name: 'channel_account_id' })
  channelAccount: ChannelAccount;

  @ManyToOne(() => PersonChannel)
  @JoinColumn({ name: 'person_channel_id' })
  personChannel: PersonChannel;

  @ManyToOne(() => Contact, { nullable: true })
  @JoinColumn({ name: 'contact_id' })
  contact?: Contact;

  @Column({ 
    type: 'enum', 
    enum: ConversationStatus, 
    default: ConversationStatus.OPEN 
  })
  status: ConversationStatus;

  @Column({ name: 'is_archived', default: false })
  isArchived: boolean;

  @Column({ 
    type: 'enum', 
    enum: AiMode, 
    name: 'ai_mode',
    nullable: true 
  })
  aiMode?: AiMode; // Cascades from account/org defaults

  @Column({ name: 'last_message_at', nullable: true })
  lastMessageAt?: Date;

  @OneToMany(() => ConversationAssignment, assignment => assignment.conversation)
  assignments: ConversationAssignment[];
}
```

## Message Flows

### Inbound Message Flow

<Steps>
<Step title="Webhook Receipt">
Meta platform sends webhook to `POST /messaging/webhook`
- `@PublicEndpoint()` with `MetaWebhookGuard`
- Validates `X-Hub-Signature-256`
- Returns 200 immediately
</Step>

<Step title="Queue Processing">
pg-boss worker processes webhook event
- Check idempotency via `externalEventId`
- Find organization using RLS bypass
- Execute within org context
</Step>

<Step title="Channel Routing">
Route to appropriate channel provider:
- WhatsAppProvider
- InstagramProvider  
- MessengerProvider
</Step>

<Step title="Entity Resolution">
Match or create entities in order:
1. PersonChannel (contact identity)
2. Person + Lead (CRM integration)
3. Conversation (message thread)
4. Message (actual message)
</Step>

<Step title="Event Emission">
Emit events for real-time updates:
- WebSocket events to `/messaging` namespace
- Notification events via queue
</Step>
</Steps>

### Outbound Message Flow

<Steps>
<Step title="API Request">
Client sends message via `POST /messaging/conversations/{id}/messages`
</Step>

<Step title="Transactional Write">
Single transaction creates:
- Message entity
- MessageOutbox entry
</Step>

<Step title="Queue Processing">
message-sender worker:
- Calls Meta Graph API
- Updates message status
- Removes outbox entry on success
</Step>

<Step title="WebSocket Update">
Emit `message-sent` event with updated status
</Step>
</Steps>

## Business Rules

### Conversation Assignment

<Info>
Conversations support multiple assignment types through the `ConversationAssignment` entity.
</Info>

**Assignment Types:**
- `user + null` = Direct assignment to specific agent
- `user + team` = Agent working on behalf of team  
- `null + team` = Team pool (unassigned within team)

**Assignment Rules:**
- Multiple assignments per conversation are allowed
- Team managers can assign within their teams
- Personal account owners have automatic access
- `MESSAGING_MANAGE` permission grants full access

### AI Mode Cascade

AI mode settings cascade through multiple levels:

1. **Conversation level** — `conversation.aiMode` (highest priority)
2. **Account level** — `channelAccount.defaultAiMode`
3. **Organization level** — Organization default setting
4. **System default** — `AiMode.OFF`

### Message Validation

<Warning>
All outbound messages must pass channel-specific validation before sending.
</Warning>

**WhatsApp Rules:**
- Text messages: 4096 character limit
- Media messages require valid MIME types
- Template messages need approved templates

**Instagram Rules:**  
- Text messages: 1000 character limit
- Story replies must reference valid story
- No template messages supported

**Messenger Rules:**
- Text messages: 2000 character limit  
- Rich media support varies by message type
- Persistent menu configuration available

### Resource Permissions

Conversations return `ResourcePermissionsDto` with computed permissions:

```typescript
export interface ConversationPermissions {
  canView: boolean;      // Based on assignment or global perms
  canReply: boolean;     // Can send messages
  canEdit: boolean;      // Can modify conversation (MANAGE only)
  canTransfer: boolean;  // Can reassign (MANAGE only)
  canArchive: boolean;   // Can archive/unarchive (MANAGE only)
  canAssign: boolean;    // Can assign within managed teams
}
```

**Permission Logic:**
- `MESSAGING_MANAGE` → `fullAccess()`
- `MESSAGING_WRITE` → `canView + canReply`
- Personal account owner → `canView + canReply`
- Assigned agent → `canView` + assignment's `canReply`
- Team member → `canView` + team assignment's `canReply`

## Error Handling & Retry Strategy

### Queue Retry Logic

<Steps>
<Step title="Exponential Backoff">
Failed jobs retry with exponential backoff:
- Attempt 1: Immediate
- Attempt 2: 30 seconds  
- Attempt 3: 2 minutes
- Attempt 4: 8 minutes
- Attempt 5: 32 minutes (final)
</Step>

<Step title="Dead Letter Queue">
Jobs failing all retries move to DLQ for manual investigation
</Step>

<Step title="Idempotency">
All webhook processing is idempotent using `externalEventId`
</Step>
</Steps>

### API Error Responses

```typescript
// Standard error format
{
  "error": {
    "code": "CONVERSATION_NOT_FOUND",
    "message": "Conversation not found or access denied",
    "details": {
      "conversationId": "uuid"
    }
  }
}
```

### Common Error Codes

- `WEBHOOK_SIGNATURE_INVALID` - Invalid Meta webhook signature
- `CHANNEL_ACCOUNT_NOT_FOUND` - Referenced account doesn't exist
- `CONVERSATION_ACCESS_DENIED` - User lacks conversation permissions
- `MESSAGE_SEND_FAILED` - Meta API error during send
- `TEMPLATE_NOT_APPROVED` - Attempting to use unapproved template
