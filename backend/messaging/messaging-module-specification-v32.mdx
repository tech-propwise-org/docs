The Messaging module provides a unified, channel-agnostic messaging system for WhatsApp, Instagram, and Facebook Messenger. It replaces the separate per-channel modules with shared entities, a shared queue, and a single WebSocket namespace.

## Overview

### Problem → Solution

| Problem | Solution |
|---------|----------|
| Duplicated logic across WhatsApp and Instagram modules | Single `MessagingModule` with channel providers |
| No webhook signature validation (security gap) | Shared `MetaWebhookGuard` validates `X-Hub-Signature-256` |
| Inconsistent WebSocket auth (Instagram gateway has no JWT) | Single `/messaging` gateway with JWT auth |
| No Facebook Messenger support | Third channel provider |
| Separate entity schemas per channel | Unified entities: `Conversation`, `Message`, `ChannelAccount` |
| No shared queue infrastructure | Shared `PgBossQueueService` for messaging + notifications |

### Key Design Decisions

<AccordionGroup>
<Accordion title="Queue Infrastructure">
**pg-boss over BullMQ** — Project already uses pg-boss for notifications. No new Redis dependency. Interface-based design (`IQueueService`) allows swapping later.
</Accordion>

<Accordion title="CRM Integration">
**Direct PersonChannel FK on Conversation** — Conversations link directly to the CRM's `PersonChannel` via FK. Simpler model, no bidirectional sync overhead.
</Accordion>

<Accordion title="Organization Lifecycle">
**Organization lifecycle integration** — MessagingOrganizationLifecycleListener handles organization deletion/restore by disconnecting WebSocket clients organization-wide and pausing/resuming Meta channel accounts non-destructively. The MessagingGateway provides `disconnectOrganization()` for cluster-wide client disconnection via the Postgres IO adapter.

The lead FK was moved from Conversation to Lead (`Lead.sourceConversation`) — conversation detail loads the single active lead for the current conversation through that FK, while broader CRM screens can still query historical person leads through `personChannel → person → leads`.
</Accordion>

<Accordion title="Archive System">
**Archive as boolean, not status** — `Conversation.isArchived` is orthogonal to `status` (OPEN/CLOSED), following `ARCHIVE_SYSTEM_SPECIFICATION.md`. Standard inbox rail reads keep the default `active` filter enabled, so archived conversations are hidden from the live Inbox; the rail exposes an explicit `isArchived=true` query for the Archived bucket.
</Accordion>

<Accordion title="Assignment Model">
**`ConversationAssignment` entity (not `entity_stakeholder`)** — Conversations use a dedicated `conversation_assignment` table instead of the CRM `entity_stakeholder` pattern.

Each assignment is one row with nullable `user_id` and `team_id`: `user + null` = direct assignment, `user + team` = agent on behalf of team, `null + team` = team pool.
</Accordion>

<Accordion title="AI Integration">
**AI Agent Integration** — The messaging module integrates with the AI module to provide automated responses through AI agents. Conversations have an `aiMode` field that controls AI participation (OFF, AUTO_REPLY, SUGGEST_ONLY, DRAFT).
</Accordion>

<Accordion title="Message Templates">
**Three-tier template system** — `MessageTemplate` supports three types: `META_APPROVED` (platform-approved), `QUICK_REPLY` (agent shortcuts with variable resolution), and `AI_PROMPT` (AI system prompts).
</Accordion>

<Accordion title="Personal Account Authentication">
**Personal accounts share org WABA token** — WhatsApp personal accounts reuse the organization's WABA access token (same Business Account). Instagram and Messenger personal accounts use their own Page Access Token obtained via OAuth.

**OAuth state includes `level` for defense-in-depth** — The HMAC-signed OAuth state payload carries a `level` field (`personal` | `organization`).
</Accordion>
</AccordionGroup>

## Multi-Tenancy Patterns

The messaging module follows the organization-scoped multi-tenancy pattern:

<Steps>
<Step title="Organization Scoping">
All entities are scoped to organizations via `organizationId` foreign keys
</Step>

<Step title="Data Isolation">
Services use `@InjectOrganizationConnection()` for tenant-specific database connections
</Step>

<Step title="Access Control">
RBAC permissions enforce organization boundaries on all operations
</Step>

<Step title="WebSocket Rooms">
Socket rooms are prefixed with organization ID for proper isolation
</Step>
</Steps>

## Message Flows

### Inbound Message Flow

<Steps>
<Step title="Webhook Reception">
Incoming webhook validated via `MetaWebhookGuard` signature verification
</Step>

<Step title="Message Processing">
Channel provider processes platform-specific webhook payload
</Step>

<Step title="Entity Creation">
Message and conversation entities created/updated in database
</Step>

<Step title="WebSocket Broadcast">
Real-time updates sent to connected clients via WebSocket
</Step>

<Step title="AI Processing">
If AI mode enabled, message queued for AI agent processing
</Step>

<Step title="Notifications">
Push notifications sent to assigned agents
</Step>
</Steps>

### Outbound Message Flow

<Steps>
<Step title="Message Creation">
Outbound message created with PENDING status
</Step>

<Step title="Outbox Pattern">
Message written to outbox table in same transaction
</Step>

<Step title="Queue Processing">
Background worker processes outbox entries
</Step>

<Step title="Channel Provider">
Platform-specific API call made to send message
</Step>

<Step title="Status Updates">
Message status updated based on platform response
</Step>

<Step title="WebSocket Sync">
Status updates broadcast to connected clients
</Step>
</Steps>

## Business Rules

### Conversation Management

<Note>
**Archive vs Delete**: Archived conversations remain accessible but hidden from active inbox. Soft-deleted conversations are terminal tombstones that can only be restored during account reconnection.
</Note>

<Warning>
**Lead Attribution**: The lead FK moved from Conversation to Lead (`Lead.sourceConversation`). Only one active lead per conversation is supported.
</Warning>

### Assignment Rules

- **Personal Account Owner**: Automatically assigned with `is_owner=true` flag
- **Team Pool Assignment**: Multiple teams can be assigned to same conversation  
- **Lead Stakeholder Auto-assignment**: Personal account owners become primary stakeholders (100%, FULL) of conversation leads

### AI Mode Cascade

<Steps>
<Step title="Conversation Level">
Check `Conversation.aiMode` field first
</Step>

<Step title="Channel Account Default">
Fall back to `ChannelAccount.defaultAiMode`
</Step>

<Step title="Organization Default">
Use organization-wide AI mode setting
</Step>

<Step title="System Default">
Final fallback to `OFF`
</Step>
</Steps>

## API Endpoints

### Core Messaging APIs

<AccordionGroup>
<Accordion title="GET /messaging/conversations">
**List conversations with filtering and pagination**

Query Parameters:
- `status` - Filter by conversation status
- `isArchived` - Filter archived conversations
- `assignedToMe` - Show only assigned conversations
- `channel` - Filter by messaging channel
- `page`, `limit` - Pagination controls

Response includes conversation metadata, last message preview, and assignment information.
</Accordion>

<Accordion title="GET /messaging/conversations/:id">
**Get conversation details with message history**

Returns full conversation object with:
- Message history (paginated)
- Assignment details
- Channel account information
- AI mode settings
</Accordion>

<Accordion title="POST /messaging/conversations/:id/messages">
**Send outbound message**

Request body includes:
- `type` - Message type (text, media, template)
- `content` - Message content object
- `templateId` - Optional template reference

Message queued via outbox pattern for reliable delivery.
</Accordion>

<Accordion title="PUT /messaging/conversations/:id/assign">
**Assign conversation to user/team**

Request body:
- `userId` - Direct user assignment
- `teamId` - Team pool assignment
- `transferReason` - Optional transfer reason

Creates `ConversationAssignment` record and broadcasts update.
</Accordion>

<Accordion title="POST /messaging/channel-accounts/connect">
**Connect new channel account**

OAuth flow initiation for Instagram/Messenger or WABA token validation for WhatsApp.
</Accordion>
</AccordionGroup>

## Error Handling & Retry Strategy

### Outbound Message Failures

<Steps>
<Step title="Initial Attempt">
Message sent via channel provider API
</Step>

<Step title="Failure Handling">
Failed messages marked with error status and retry count incremented
</Step>

<Step title="Exponential Backoff">
Retry attempts scheduled with exponential backoff (1min, 5min, 15min, 1hr)
</Step>

<Step title="Dead Letter Queue">
Messages exceeding max retries moved to dead letter queue for manual review
</Step>

<Step title="User Notification">
Failed messages shown in UI with retry option
</Step>
</Steps>

### Webhook Processing Failures

<Warning>
Webhook failures are logged but don't block the HTTP response to avoid platform retries. Failed webhooks are queued for retry processing.
</Warning>

## Deployment Considerations

### Database Migrations

<Note>
The messaging module requires careful migration sequencing due to foreign key relationships with CRM entities.
</Note>

Key migration steps:
1. Create messaging-specific tables
2. Migrate data from legacy WhatsApp/Instagram modules  
3. Update foreign key relationships
4. Remove legacy tables

### Queue Worker Scaling

- **Message Processing Workers**: Scale based on webhook volume
- **AI Processing Workers**: Scale based on AI mode usage
- **Notification Workers**: Scale based on user count

### WebSocket Scaling

<Tip>
WebSocket connections are horizontally scalable via the Postgres-backed Socket.IO adapter. Organization disconnection works cluster-wide.
</Tip>
