## Overview

The Messaging module provides a unified, channel-agnostic messaging system for WhatsApp, Instagram, and Facebook Messenger. It replaces the separate per-channel modules with shared entities, a shared queue, and a single WebSocket namespace.

### Problem → Solution

| Problem | Solution |
|---------|----------|
| Duplicated logic across WhatsApp and Instagram modules | Single `MessagingModule` with channel providers |
| No webhook signature validation (security gap) | Shared `MetaWebhookGuard` validates `X-Hub-Signature-256` |
| Inconsistent WebSocket auth (Instagram gateway has no JWT) | Single `/messaging` gateway with JWT auth |
| No Facebook Messenger support | Third channel provider |
| Separate entity schemas per channel | Unified entities: `Conversation`, `Message`, `ChannelAccount` |
| No shared queue infrastructure | Shared `PgBossQueueService` for messaging + notifications |

### Key Design Decisions

<AccordionGroup>
<Accordion title="pg-boss over BullMQ">
Project already uses pg-boss for notifications. No new Redis dependency. Interface-based design (`IQueueService`) allows swapping later.
</Accordion>

<Accordion title="Direct PersonChannel FK on Conversation">
Conversations link directly to the CRM's `PersonChannel` via FK. Simpler model, no bidirectional sync overhead.
</Accordion>

<Accordion title="Organization lifecycle integration">
MessagingOrganizationLifecycleListener handles organization deletion/restore by disconnecting WebSocket clients organization-wide and pausing/resuming Meta channel accounts non-destructively. The MessagingGateway provides `disconnectOrganization()` for cluster-wide client disconnection via the Postgres IO adapter.
</Accordion>

<Accordion title="Archive as boolean, not status">
`Conversation.isArchived` is orthogonal to `status` (OPEN/CLOSED), following `ARCHIVE_SYSTEM_SPECIFICATION.md`. Standard inbox rail reads keep the default `active` filter enabled, so archived conversations are hidden from the live Inbox.
</Accordion>

<Accordion title="ConversationAssignment entity">
Conversations use a dedicated `conversation_assignment` table instead of the CRM `entity_stakeholder` pattern for simplified messaging-specific assignment tracking.
</Accordion>

<Accordion title="AI Agent Integration">
The messaging module integrates with the AI module to provide automated responses through AI agents. Conversations have an `aiMode` field that controls AI participation (OFF, AUTO_REPLY, SUGGEST_ONLY, DRAFT).
</Accordion>

<Accordion title="Transactional outbox">
Outbound messages use an outbox table written in the same DB transaction as the Message entity, guaranteeing at-least-once delivery.
</Accordion>

<Accordion title="Three-tier template system">
`MessageTemplate` supports three types: `META_APPROVED` (platform-approved), `QUICK_REPLY` (agent shortcuts), and `AI_PROMPT` (AI system prompts).
</Accordion>

<Accordion title="Personal accounts share org WABA token">
WhatsApp personal accounts reuse the organization's WABA access token (same Business Account). Instagram and Messenger personal accounts use their own Page Access Token obtained via OAuth.
</Accordion>
</AccordionGroup>

## Multi-Tenancy Patterns

The messaging system implements organization-level isolation with the following patterns:

<Steps>
<Step title="Organization Scoping">
All messaging entities are scoped by `organizationId` to ensure tenant isolation.
</Step>

<Step title="Channel Account Ownership">
Each `ChannelAccount` belongs to a specific organization and cannot be accessed across tenant boundaries.
</Step>

<Step title="WebSocket Room Management">
WebSocket rooms are prefixed with organization ID to prevent cross-tenant message leakage.
</Step>

<Step title="RBAC Integration">
All messaging operations respect organization-level RBAC permissions.
</Step>
</Steps>

## Business Rules

<Warning>
These rules are enforced at the service layer and cannot be bypassed by direct entity manipulation.
</Warning>

### Conversation Rules

1. **One active conversation per channel-person pair** - Only one non-archived conversation can exist for a given `channelAccount + personChannel` combination
2. **Archive orthogonality** - Archived status is independent of conversation status (OPEN/CLOSED)
3. **AI mode inheritance** - Conversation AI mode defaults cascade from ChannelAccount → Organization → OFF
4. **Assignment ownership** - Personal channel account owners get protected assignment status

### Message Rules

1. **Outbound message validation** - All outbound messages must pass platform-specific validation
2. **Template variable resolution** - Quick reply templates resolve variables from conversation context
3. **Media processing** - Media messages processed through AI module for content analysis
4. **Delivery guarantees** - Transactional outbox ensures at-least-once delivery

## Error Handling & Retry Strategy

### Webhook Processing

<Steps>
<Step title="Signature Validation">
All webhooks validated using `MetaWebhookGuard` with HMAC-SHA256
</Step>

<Step title="Idempotency Check">
Duplicate webhook events detected and ignored using platform message IDs
</Step>

<Step title="Graceful Degradation">
Non-critical webhook failures logged but don't block processing
</Step>

<Step title="Dead Letter Queue">
Failed webhook events moved to dead letter queue for investigation
</Step>
</Steps>

## Known Gaps & Technical Debt

<AccordionGroup>
<Accordion title="Message Search Performance">
Full-text search across message content may require optimization for large datasets. Consider implementing search indexing with Elasticsearch or similar.
</Accordion>

<Accordion title="Media Storage Strategy">
Current implementation stores media URLs from platforms. Long-term strategy needed for media archival and CDN optimization.
</Accordion>

<Accordion title="Rate Limiting Granularity">
Rate limiting currently implemented at organization level. May need per-channel-account limits for better control.
</Accordion>

<Accordion title="Webhook Replay Capability">
No current mechanism to replay failed webhooks. Consider implementing webhook replay for debugging and recovery.
</Accordion>

<Accordion title="Message Analytics">
Limited analytics on message patterns, response times, and conversation outcomes. Analytics module integration planned.
</Accordion>
