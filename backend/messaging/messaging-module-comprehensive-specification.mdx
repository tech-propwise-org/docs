## Overview

The Messaging module provides a unified, channel-agnostic messaging system for WhatsApp, Instagram, and Facebook Messenger. It replaces the separate per-channel modules with shared entities, a shared queue, and a single WebSocket namespace.

<Info>
**Last Updated:** 2026-04-15  
**Status:** Active
</Info>

### Problem → Solution

| Problem | Solution |
|---------|----------|
| Duplicated logic across WhatsApp and Instagram modules | Single `MessagingModule` with channel providers |
| No webhook signature validation (security gap) | Shared `MetaWebhookGuard` validates `X-Hub-Signature-256` |
| Inconsistent WebSocket auth (Instagram gateway has no JWT) | Single `/messaging` gateway with JWT auth |
| No Facebook Messenger support | Third channel provider |
| Separate entity schemas per channel | Unified entities: `Conversation`, `Message`, `ChannelAccount` |
| No shared queue infrastructure | Shared `PgBossQueueService` for messaging + notifications |

### Key Design Decisions

<AccordionGroup>
  <Accordion title="Queue Infrastructure">
    **pg-boss over BullMQ** — Project already uses pg-boss for notifications. No new Redis dependency. Interface-based design (`IQueueService`) allows swapping later.
  </Accordion>
  
  <Accordion title="Data Model">
    **Direct PersonChannel FK on Conversation** — Conversations link directly to the CRM's `PersonChannel` via FK. Simpler model, no bidirectional sync overhead. The lead FK was moved from Conversation to Lead (`Lead.sourceConversation`).
  </Accordion>
  
  <Accordion title="Archive System">
    **Archive as boolean, not status** — `Conversation.isArchived` is orthogonal to `status` (OPEN/CLOSED), following `ARCHIVE_SYSTEM_SPECIFICATION.md`.
  </Accordion>
  
  <Accordion title="Assignment Model">
    **`ConversationAssignment` entity** — Uses dedicated `conversation_assignment` table instead of CRM `entity_stakeholder` pattern. Each assignment is one row with nullable `user_id` and `team_id`.
  </Accordion>
  
  <Accordion title="Message Delivery">
    **Transactional outbox** — Outbound messages use an outbox table written in the same DB transaction as the Message entity, guaranteeing at-least-once delivery.
  </Accordion>
</AccordionGroup>

## Architecture & Module Structure

```mermaid
graph TB
    A[Meta Platform Webhooks] --> B[POST /messaging/webhook]
    B --> C[@PublicEndpoint + MetaWebhookGuard]
    C --> D[WebhookEventLog + Queue]
    D --> E[Queue Worker webhook-processor]
    E --> F[Channel Providers]
    F --> G[Entities & Services]
```

<Steps>
  <Step title="Webhook Reception">
    Meta webhooks arrive at `POST /messaging/webhook` with `@PublicEndpoint()` and `MetaWebhookGuard` validation
  </Step>
  <Step title="Immediate Response">
    Returns 200 immediately, persists to `WebhookEventLog`, and enqueues to pg-boss queue
  </Step>
  <Step title="Queue Processing">
    Worker processes events: check idempotency, find organization, route to channel providers
  </Step>
  <Step title="Entity Operations">
    Match/create PersonChannel, Person, Lead, Conversation, and Message entities
  </Step>
  <Step title="Event Emission">
    Emit WebSocket events and notification events for real-time updates
  </Step>
</Steps>

## Multi-Tenancy Patterns

<Warning>
The messaging module introduces unique multi-tenancy challenges because webhooks arrive without org context.
</Warning>

### Two-Step RLS Bypass (Webhook Processing)

The webhook controller receives events for ALL organizations from a single Meta App. Org context is unknown at arrival time.

<CodeGroup>
```typescript Step 1: Find Organization
// Step 1: Find which org owns this account (bypass RLS)
const account = await this.tenantContext.executeReadOnlyWithBypass(async (em) => {
  return em.findOne(ChannelAccount, { externalAccountId: job.data.accountId });
});
```

```typescript Step 2: Process in Context
// Step 2: Process within that org's context
await this.tenantContext.executeInOrg(
  account.organization.id,
  async (em) => {
    await this.processMessageInTransaction(em, job.data);
  },
  { userId: undefined }, // system action, no user
);
```
</CodeGroup>

### Composable `*InTransaction` Pattern

Services that participate in existing transactions expose `*InTransaction` methods:

```typescript
// Public API — wraps TenantContext
async matchOrCreate(channel, identifier, profileData, orgId): Promise<MatchResult>;

// Composable — accepts EntityManager from caller's transaction
async matchOrCreateInTransaction(em, channel, identifier, profileData, orgId): Promise<MatchResult>;
```

<Note>
The `em` parameter must always be the one provided by the TenantContext callback — never `this.em`.
</Note>

### Read-Only vs Mutation Methods

<Tabs>
  <Tab title="Read-Only">
    ```typescript
    // Read-only: queries, lookups, list endpoints
    return this.tenantContext.executeReadOnly(organizationId, async (em) => {
      // Query operations only
    });
    ```
  </Tab>
  <Tab title="Mutation">
    ```typescript
    // Mutation: updates, creates, deletes
    return this.tenantContext.executeInOrg(
      organizationId, 
      async (em) => {
        // Mutation operations
      }, 
      { userId }
    );
    ```
  </Tab>
</Tabs>

### Conversation Entity

```typescript
@Entity('conversation')
export class Conversation extends BaseEntity {
  @Column()
  externalConversationId: string; // Platform-specific ID

  @Column({ type: 'enum', enum: ConversationStatus })
  status: ConversationStatus; // OPEN | CLOSED

  @Column({ default: false })
  isArchived: boolean;

  @Column({ type: 'enum', enum: AiMode, default: AiMode.OFF })
  aiMode: AiMode;

  @ManyToOne(() => ChannelAccount)
  channelAccount: ChannelAccount;

  @ManyToOne(() => PersonChannel)
  personChannel: PersonChannel;

  @OneToMany(() => Message, message => message.conversation)
  messages: Message[];

  @OneToMany(() => ConversationAssignment, assignment => assignment.conversation)
  assignments: ConversationAssignment[];

```

### Message Entity

```typescript
@Entity('message')
export class Message extends BaseEntity {
  @Column()
  externalMessageId: string;

  @Column({ type: 'enum', enum: MessageDirection })
  direction: MessageDirection; // INBOUND | OUTBOUND

  @Column({ type: 'enum', enum: MessageType })
  type: MessageType; // TEXT | IMAGE | AUDIO | VIDEO | DOCUMENT | TEMPLATE

  @Column({ type: 'text', nullable: true })
  content?: string;

  @Column({ type: 'jsonb', nullable: true })
  metadata?: Record<string, any>;

  @Column({ type: 'enum', enum: MessageStatus, default: MessageStatus.PENDING })
  status: MessageStatus; // PENDING | SENT | DELIVERED | READ | FAILED

  @ManyToOne(() => Conversation)
  conversation: Conversation;

  @ManyToOne(() => User, { nullable: true })
  sender?: User; // null for inbound messages
}
```

## Message Flows

### Inbound Message Flow

<Steps>
  <Step title="Webhook Reception">
    Meta webhook arrives at `/messaging/webhook` endpoint
  </Step>
  <Step title="Security Validation">
    `MetaWebhookGuard` validates `X-Hub-Signature-256` header
  </Step>
  <Step title="Event Logging">
    Create `WebhookEventLog` entry and enqueue processing job
  </Step>
  <Step title="Organization Discovery">
    Worker finds organization via `ChannelAccount.externalAccountId`
  </Step>
  <Step title="Entity Matching">
    Match or create `PersonChannel`, `Person`, and `Conversation`
  </Step>
  <Step title="Message Creation">
    Create `Message` entity with inbound direction
  </Step>
  <Step title="CRM Integration">
    Create CRM `Activity` via bridge service
  </Step>
  <Step title="Real-time Updates">
    Emit WebSocket events and notifications
  </Step>
</Steps>

### Outbound Message Flow

<Steps>
  <Step title="API Request">
    Agent/system calls message send endpoint
  </Step>
  <Step title="Validation">
    Validate permissions, conversation status, and message content
  </Step>
  <Step title="Transactional Creation">
    Create `Message` and `MessageOutbox` in same transaction
  </Step>
  <Step title="Queue Processing">
    `message-sender` worker processes outbox entries
  </Step>
  <Step title="Platform API Call">
    Send to Meta Graph API (WhatsApp/Instagram/Messenger)
  </Step>
  <Step title="Status Updates">
    Update message status based on API response
  </Step>
  <Step title="Real-time Updates">
    Emit WebSocket events for status changes
  </Step>
</Steps>

## Business Rules

### Conversation Management

<Note>
**Archive vs Status**: Archive state (`isArchived`) is independent of conversation status (`OPEN`/`CLOSED`). Archived conversations can still receive messages.
</Note>

- **Auto-creation**: Conversations are auto-created when first message arrives from unknown contact
- **Assignment cascade**: New conversations inherit assignment from previous conversations with same `PersonChannel`
- **AI mode inheritance**: `Conversation.aiMode` defaults cascade from `ChannelAccount.defaultAiMode` → Organization default → `OFF`
- **24-hour rule**: WhatsApp template messages required after 24 hours of customer inactivity

### Assignment Rules

<Tabs>
  <Tab title="Direct Assignment">
    ```
    user_id: 123, team_id: null
    → Agent 123 directly assigned
    ```
  </Tab>
  <Tab title="Team Pool">
    ```
    user_id: null, team_id: 456
    → Available to any agent in Team 456
    ```
  </Tab>
  <Tab title="Agent on Team">
    ```
    user_id: 123, team_id: 456  
    → Agent 123 representing Team 456
    ```
  </Tab>
</Tabs>

### Message Validation

- **Character limits**: WhatsApp (4096), Instagram (1000), Messenger (2000)
- **Media size limits**: Varies by platform and media type
- **Template validation**: Meta-approved templates must match exact format
- **Rate limiting**: Per-account sending limits enforced by platforms

### ResourcePermissionsDto

Each conversation returns permissions object:

```typescript
interface ConversationPermissions {
  canView: boolean;      // Can see conversation
  canReply: boolean;     // Can send messages
  canAssign: boolean;    // Can assign to team members
  canTransfer: boolean;  // Can transfer between agents/teams
  canArchive: boolean;   // Can archive/unarchive
  canEdit: boolean;      // Can edit conversation properties
}
```

<Warning>
Permission computation is done in-memory by `ConversationPermissionService` — no additional DB queries required.
</Warning>

### Team Permissions

- **Team messaging managers** (`team_messaging.manage`): Can assign conversations within their team
- **Team members**: Can view and reply to conversations assigned to their team pool
- **Assignment inheritance**: Team assignments allow any team member to pick up conversations

  </Accordion>
  
  <Accordion title="CONVERSATION_TRANSFERRED">
    **Trigger**: Conversation transferred between agents/teams  
    **Recipients**: New assignee, previous assignee (notification of transfer)  
    **Channels**: In-app, email  
    **Conditions**: Skip if same agent transfer
  </Accordion>
  
  <Accordion title="AI_SUGGESTION_READY">
    **Trigger**: AI generates response suggestion  
    **Recipients**: Assigned agent only  
    **Channels**: In-app only

  </Accordion>

## Query Patterns

### Conversation Queries

<CodeGroup>
```sql Inbox Query
-- Optimized for inbox view with filters
SELECT c.*, ca.channel, ca.display_name, pc.external_identifier
FROM conversation c
JOIN channel_account ca ON c.channel_account_id = ca.id  
JOIN person_channel pc ON c.person_channel_id = pc.id
WHERE c.organization_id = $1
  AND ($2::conversation_status IS NULL OR c.status = $2)
  AND ($3::channel IS NULL OR ca.channel = $3)
  AND c.is_archived = COALESCE($4, false)
ORDER BY c.last_message_at DESC NULLS LAST
LIMIT $5 OFFSET $6;
```

```sql Assignment Filter
-- Filter by assignment (agent or team)
SELECT DISTINCT c.*
FROM conversation c
JOIN conversation_assignment ca ON c.id = ca.conversation_id
WHERE c.organization_id = $1
  AND (
    ($2::uuid IS NOT NULL AND ca.user_id = $2) OR
    ($3::uuid IS NOT NULL AND ca.team_id = $3)
  )
  AND ca.is_active = true;
```

```sql Unassigned Conversations
-- Find unassigned conversations
SELECT c.*
FROM conversation c
WHERE c.organization_id = $1
  AND NOT EXISTS (
    SELECT 1 FROM conversation_assignment ca
    WHERE ca.conversation_id = c.id AND ca.is_active = true
  );
```
</CodeGroup>

### Message Queries

<CodeGroup>
```sql Message History
-- Paginated message history for conversation
SELECT m.*, u.first_name, u.last_name
FROM message m
LEFT JOIN "user" u ON m.sender_id = u.id
WHERE m.conversation_id = $1
  AND m.organization_id = $2
ORDER BY m.created_at DESC
LIMIT $3 OFFSET $4;
```

```sql Unread Count
-- Count unread messages per conversation
SELECT conversation_id, COUNT(*) as unread_count
FROM message
WHERE conversation_id = ANY($1)
  AND direction = 'INBOUND'
  AND is_read = false
GROUP BY conversation_id;
```
</CodeGroup>

CREATE INDEX idx_conversation_org_channel ON conversation(organization_id, channel_account_id);

CREATE INDEX idx_conversation_last_message ON conversation(last_message_at DESC NULLS LAST);

-- Message queries  
CREATE INDEX idx_message_conversation_created ON message(conversation_id, created_at DESC);
CREATE INDEX idx_message_external_id ON message(external_message_id);
CREATE INDEX idx_message_status ON message(status) WHERE direction = 'OUTBOUND';

-- Assignment queries
CREATE INDEX idx_conversation_assignment_user ON conversation_assignment(user_id) WHERE is_active = true;
CREATE INDEX idx_conversation_assignment_team ON conversation_assignment(team_id) WHERE is_active = true;

## Error Handling & Retry Strategy

### Webhook Processing

<Steps>
  <Step title="Idempotency Check">
    Check `WebhookEventLog.externalEventId` to prevent duplicate processing
  </Step>
  <Step title="Graceful Degradation">
    Log errors but continue processing other events in queue
  </Step>
  <Step title="Retry Logic">
    pg-boss automatic retry with exponential backoff (max 3 attempts)
  </Step>
  <Step title="Dead Letter Queue">
    Failed events move to `failed_webhook_events` table for manual review
  </Step>
</Steps>

### Outbound Message Sending

<Tabs>
  <Tab title="Transient Failures">
    - Network timeouts: Retry 3x with exponential backoff
    - Rate limiting: Respect platform retry-after headers  
    - Temporary API errors: Queue for retry
  </Tab>
  <Tab title="Permanent Failures">
    - Invalid recipient: Mark message as FAILED, notify sender
    - Template approval required: Return specific error code
    - Account suspended: Disable channel account, alert administrators
  </Tab>
</Tabs>

### Circuit Breaker Pattern

```typescript
// Protect against cascading failures
@Injectable()
export class MetaGraphApiService {
  private circuitBreaker = new CircuitBreaker(this.sendMessage.bind(this), {
    timeout: 10000,      // 10 second timeout
    errorThresholdPercentage: 50, // Open after 50% failure rate
    resetTimeout: 30000, // Reset after 30 seconds
  });
}
```

  </Accordion>
