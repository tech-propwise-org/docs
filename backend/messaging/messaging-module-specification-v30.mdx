## Overview

The Messaging module provides a unified, channel-agnostic messaging system for WhatsApp, Instagram, and Facebook Messenger. It replaces the separate per-channel modules with shared entities, a shared queue, and a single WebSocket namespace.

### Problem → Solution

<CardGroup cols={2}>
  <Card title="Duplicated Logic" icon="copy">
    **Problem:** Duplicated logic across WhatsApp and Instagram modules  
    **Solution:** Single `MessagingModule` with channel providers
  </Card>
  
  <Card title="Security Gap" icon="shield">
    **Problem:** No webhook signature validation  
    **Solution:** Shared `MetaWebhookGuard` validates `X-Hub-Signature-256`
  </Card>
  
  <Card title="Inconsistent Auth" icon="key">
    **Problem:** Inconsistent WebSocket auth (Instagram gateway has no JWT)  
    **Solution:** Single `/messaging` gateway with JWT auth
  </Card>
  
  <Card title="Limited Channels" icon="message">
    **Problem:** No Facebook Messenger support  
    **Solution:** Third channel provider
  </Card>
  
  <Card title="Schema Duplication" icon="database">
    **Problem:** Separate entity schemas per channel  
    **Solution:** Unified entities: `Conversation`, `Message`, `ChannelAccount`
  </Card>
  
  <Card title="Queue Infrastructure" icon="clock">
    **Problem:** No shared queue infrastructure  
    **Solution:** Shared `PgBossQueueService` for messaging + notifications
  </Card>
</CardGroup>

### Key Design Decisions

<AccordionGroup>
  <Accordion title="pg-boss over BullMQ">
    Project already uses pg-boss for notifications. No new Redis dependency. Interface-based design (`IQueueService`) allows swapping later.
  </Accordion>
  
  <Accordion title="Direct PersonChannel FK on Conversation">
    Conversations link directly to the CRM's `PersonChannel` via FK. Simpler model, no bidirectional sync overhead.
  </Accordion>
  
  <Accordion title="Organization lifecycle integration">
    MessagingOrganizationLifecycleListener handles organization deletion/restore by:
    - Disconnecting WebSocket clients organization-wide
    - Pausing/resuming Meta channel accounts non-destructively
    - MessagingGateway provides `disconnectOrganization()` for cluster-wide client disconnection
    - Lead FK moved from Conversation to Lead (`Lead.sourceConversation`)
  </Accordion>
  
  <Accordion title="Archive as boolean, not status">
    `Conversation.isArchived` is orthogonal to `status` (OPEN/CLOSED), following `ARCHIVE_SYSTEM_SPECIFICATION.md`. Standard inbox rail reads keep the default `active` filter enabled, so archived conversations are hidden from the live Inbox.
  </Accordion>
  
  <Accordion title="ConversationAssignment entity">
    Conversations use a dedicated `conversation_assignment` table instead of the CRM `entity_stakeholder` pattern for simplified messaging-specific assignment logic.
  </Accordion>
  
  <Accordion title="Transactional outbox">
    Outbound messages use an outbox table written in the same DB transaction as the Message entity, guaranteeing at-least-once delivery.
  </Accordion>
  
  <Accordion title="Per-conversation AI mode">
    Each conversation has an `aiMode` field (OFF, AUTO_REPLY, SUGGEST_ONLY, DRAFT) with comprehensive AI module integration.
  </Accordion>
  
  <Accordion title="Three-tier template system">
    `MessageTemplate` supports three types: `META_APPROVED` (platform-approved), `QUICK_REPLY` (agent shortcuts), and `AI_PROMPT` (AI system prompts).
  </Accordion>
  
  <Accordion title="Personal accounts share org WABA token">
    WhatsApp personal accounts reuse the organization's WABA access token. Instagram and Messenger personal accounts use their own Page Access Token obtained via OAuth.
  </Accordion>
</AccordionGroup>

## Message Flows

### Inbound Message Flow

<Steps>
  <Step title="Webhook Reception">
    Platform webhooks are received by channel-specific controllers
  </Step>
  
  <Step title="Signature Validation">
    `MetaWebhookGuard` validates `X-Hub-Signature-256` header
  </Step>
  
  <Step title="Message Processing">
    Channel provider processes the webhook and creates/updates entities
  </Step>
  
  <Step title="AI Processing">
    If conversation has AI mode enabled, message is queued for AI processing
  </Step>
  
  <Step title="WebSocket Notification">
    Real-time updates are broadcast to relevant organization rooms
  </Step>
  
  <Step title="Push Notifications">
    Assigned users receive push notifications if applicable
  </Step>
</Steps>

### Outbound Message Flow

<Steps>
  <Step title="Message Creation">
    API endpoint receives outbound message request
  </Step>
  
  <Step title="Validation">
    Message content and channel constraints are validated
  </Step>
  
  <Step title="Database Transaction">
    Message entity and outbox entry are created atomically
  </Step>
  
  <Step title="Queue Processing">
    Outbox worker processes the message for delivery
  </Step>
  
  <Step title="Platform API Call">
    Channel provider makes API call to messaging platform
  </Step>
  
  <Step title="Status Update">
    Message status is updated based on platform response
  </Step>
</Steps>

## Business Rules

### Conversation Management

<Warning>
**Archive vs Delete**: Conversations have separate `isArchived` and `isDeleted` flags. Archived conversations remain accessible but are hidden from active inbox. Deleted conversations are terminal tombstones.
</Warning>

<Tabs>
  <Tab title="Archive Rules">
    - Archived conversations don't appear in default inbox queries
    - New inbound messages to archived conversations don't trigger notifications
    - Messages are still stored and conversation remains accessible via direct access
    - Archive filter can be disabled for conversation detail views
  </Tab>
  
  <Tab title="Delete Rules">
    - Soft-deleted conversations are terminal inbox-thread tombstones
    - Account disconnect soft-deletes and archives all conversations
    - Restoration with `restoreConversations=true` flips back to active
    - New inbound can create replacement conversations for deleted threads
  </Tab>
</Tabs>

### Assignment Rules

<Info>
**Lead Stakeholder Auto-Assignment**: For personal channel accounts with `autoCreateLead` enabled, the account owner is automatically assigned as the primary stakeholder (100%, FULL) of the conversation's lead.
</Info>

Assignment types:
- **Direct Assignment** (`user + null`): Individual agent assignment
- **Team Assignment** (`user + team`): Agent on behalf of team
- **Pool Assignment** (`null + team`): Team pool without specific agent
- **Owner Assignment** (`is_owner=true`): Protected personal account owner

### AI Mode Cascading

Default AI mode follows this hierarchy:
1. `ChannelAccount.defaultAiMode`
2. Organization default AI mode
3. `OFF` (fallback)

## Error Handling & Retry Strategy

### Error Categories

<Tabs>
  <Tab title="Webhook Errors">
    **Signature Validation Failures**
    - Return 401 immediately
    - Log security event
    - No retry attempted

    **Processing Failures**
    - Return 500 for retry by platform
    - Queue for delayed processing
    - Alert on repeated failures
  </Tab>
  
  <Tab title="Outbound Message Errors">
    **Rate Limiting**
    - Exponential backoff retry
    - Respect platform rate limits
    - Queue for later delivery

    **Authentication Failures**
    - Refresh tokens if possible
    - Alert account owner
    - Fail message with clear error
  </Tab>
  
  <Tab title="Database Errors">
    **Connection Issues**
    - Retry with exponential backoff
    - Circuit breaker pattern
    - Graceful degradation

    **Constraint Violations**
    - Return clear error messages
    - No automatic retry
    - Log for debugging
  </Tab>
</Tabs>

INSERT INTO conversations (
