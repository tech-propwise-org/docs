The CRM module is the core of PropWise. It manages the full lifecycle of customer relationships — from initial inquiry (lead) through qualification (contact) to revenue (deal). Every entity links back to a central **Person** identity layer.

## Architecture overview

### Design principles

The CRM module follows six core design principles:

1. **Person + Contact model** — `Person` is the hidden identity layer (single source of truth for personal details). `Contact` is the business relationship layer. `Lead` is the sales opportunity layer. `Deal` links to `Contact`, not `Person` directly.
2. **Unified stakeholder model** — A single `entity_stakeholder` table manages assignment and commission across leads and deals.
3. **Polymorphic patterns** — Notes, tags, and activities use `entity_type`/`entity_id` patterns for flexible targeting.
4. **Channel separation** — The `activity` table indexes the timeline; channel tables (`call_log`, `email_log`, etc.) store full data.
5. **Modular design** — CRM core is independent. Real Estate, Marketing, and Channels are optional modules.
6. **Company via Contact** — Companies associate with `Contact` via `ContactCompanyRole`, not with `Person`.

### Module boundaries

The CRM core contains the foundational entities. External modules connect through well-defined integration points.

```
CRM CORE
  Person, Lead, Contact, Company, Deal, DealContact
  person_email, person_phone, person_address, person_channel
  person_not_duplicate, contact_company_role
  entity_stakeholder, entity_transfer, commission_payment
  activity, note, task, tag

REAL ESTATE MODULE         MARKETING MODULE       CHANNELS MODULE
  off-plan, inventory        campaign               whatsapp
  site_visit                 campaign_lead           instagram
  lead/deal_property_                                (linked via
  interest                                           person_channel)
  unit_ownership → Person
```

### Real estate integration

Real Estate entities link to **Person** (not Contact) for identity:

| Real estate entity | Links to | Rationale |
| --- | --- | --- |
| `unit_ownership` | `person_id` | Ownership is about identity, not CRM status |
| `unit_transaction` | `person_id` | Transaction party is an individual |
| `site_visit` | `person_id` | Who visited the property |
| `lead_property_interest` | `lead_id` | Links to lead for sales context |
| `deal_property_interest` | `deal_id` | Links to deal for transaction context |

<Tip>
Contacts access real estate data through Person: `contact.person.unitOwnerships`. This keeps ownership tied to identity regardless of CRM relationship status.
</Tip>

**Deal property interest workflow:**

```typescript
// Deal created FROM lead:
// Copy primary LeadPropertyInterest → DealPropertyInterest (1:1)
deal.propertyInterest.originatingInterest = leadPropertyInterest;

// Deal created directly (walk-in):
// Create DealPropertyInterest with no originating interest
deal.propertyInterest.originatingInterest = null;
```

## Core entities

### Person (central identity)

Single source of truth for human identity and preferences. Person is the hidden layer — not exposed directly in the frontend.

```
Person
├── Identity: first_name, last_name, avatar_url, title
├── Demographics: date_of_birth
├── Social: website, linkedin_url, twitter_url
├── Preferences: preferred_contact_method, timezone, preferred_language
├── Communication flags: do_not_call, do_not_email
├── Source tracking: original_source, languages[]
├── Merge tracking: merged_into_id, merged_at, merged_by
├── Computed: full_name (getter: first_name + last_name)
└── Related tables:
    ├── person_email (multiple emails, one primary)
    ├── person_phone (multiple phones, one primary)
    ├── person_address (multiple addresses, one primary)
    ├── person_channel (WhatsApp, Instagram, etc.)
    └── person_not_duplicate (deduplication override pairs)
```

<Note>
Every lead and contact must link to a Person. Person preferences apply across all contexts — leads, deals, and contacts. The `original_source` field is set once when the person first enters the system.
</Note>

### PersonChannel (communication channels)

Stores a person's communication channel identities (WhatsApp, Instagram, etc.).

```
person_channel
├── person_id → Person
├── channel_type (whatsapp, instagram, facebook, telegram, sms, webchat)
├── channel_identifier (phone number, username, PSID, etc.)
├── display_name, avatar_url
├── channel_identity_id → WhatsAppUser.id, InstagramUser.id, etc.
├── status (active, inactive, blocked, unsubscribed)
├── is_primary
├── Opt-in: marketing_opt_in, transactional_opt_in
├── Engagement: first_contact_at, last_message_at, message_count
└── Verification: is_verified, verified_at
```

<Info>
Channels belong to Person, not Lead (person-centric architecture). A lead can reference `source_channel_id` for attribution. One person can have multiple channels of the same type.
</Info>

### PersonNotDuplicate (deduplication overrides)

Records pairs of persons manually confirmed as NOT duplicates. Prevents repeated flagging by the deduplication system.

```
person_not_duplicate
├── person1_id → Person
├── person2_id → Person
├── marked_by → User
├── marked_at
├── organization_id → Organization
├── Unique constraint: (person1, person2, organization)
```

The relationship is symmetric — if (A, B) is marked as not-duplicate, (B, A) is treated equivalently. Each organization maintains its own override decisions.

### Person merge system

Consolidates duplicate persons into a single primary record, reassigning all related data.

**API endpoint:** `POST /persons/:primaryPersonId/merge`

<Steps>
<Step title="Validation">
Verify the primary person and all secondary persons exist, are not deleted, and belong to the same organization.
</Step>
<Step title="Field selection">
Accept `fieldSelections: Record<string, string>` (e.g., `{ "firstName": "primary", "lastName": "person-B-id" }`). For each field, pick the value from the specified source person. Fields not listed default to the primary person's values.
</Step>
<Step title="Contact info merge (opt-in per type)">
Three boolean flags control which contact info is reassigned from secondary persons to the primary:

- `mergeAllEmails` — reassign all secondary emails (`isPrimary` reset to `false`)
- `mergeAllPhones` — reassign all secondary phones (`isPrimary` reset to `false`)
- `mergeAllAddresses` — reassign all secondary addresses (`isPrimary` reset to `false`)
</Step>
<Step title="Channel merge (always)">
All channels from secondary persons are reassigned to the primary person (`isPrimary` reset to `false`). This is not optional since channels represent unique identities.
</Step>
<Step title="Lead and ownership reassignment">
All leads pointing to secondary persons are bulk-reassigned to the primary person via `nativeUpdate`. All `InventoryUnitOwnership` records are similarly reassigned.
</Step>
<Step title="Mark secondaries">
Each secondary person is marked with `mergedIntoId`, `mergedAt`, `mergedBy`, and soft-deleted (`isDeleted = true`).
</Step>
</Steps>

<Warning>
Merge is **irreversible** — secondary persons are soft-deleted. Contact entities are NOT directly reassigned; since Contact has a `OneToOne` to Person, the primary person retains its contact if one exists.
</Warning>

### Lead and contact import services

The CRM includes import functionality for bulk data ingestion:

- **`LeadImportService`** — Handles bulk lead import with person matching/creation, deduplication by `referenceId + leadSource`, and stakeholder assignment.
- **`ContactImportService`** — Handles bulk contact import with person matching, uniqueness validation (one contact per person per organization), and activity backfill.

### Lead (sales opportunity)

Represents a sales opportunity or inquiry from a person.

```
Lead
├── person_id (required) → Person
├── stage_id → lead_stage (pipeline position)
├── intent (buyer | seller | renter | landlord | investor) — single enum
├── reference_id (external reference)
├── source_channel_id → PersonChannel (channel attribution)
├── companyName (optional string for display — NO FK)
├── lead_source (enum, nullable), source_detail (text)
├── score (lead scoring), temperature (cold | warm | hot)
├── budget, budget_currency (default: AED)
├── industry, timeline
├── Pipeline dates: converted_at, disqualified_at
├── Disqualification: disqualification_reason (enum), disqualification_notes
└── Notes: use polymorphic 'note' table
```

<Warning>
`intent` is a **single enum value** (buyer, seller, renter, landlord, investor), not an array. `companyName` is display-only with no foreign key to Company — company associations happen at the Contact level.
</Warning>

**Key rules:**

- One person can have multiple leads (each inquiry tracked separately)
- Assignment managed via `entity_stakeholder`, not direct fields
- Stage changes recorded in `lead_stage_history`
- `leadSource` and `sourceDetail` track lead origin separately from Person's `originalSource`
- `temperature` (cold/warm/hot) complements numeric `score`

**Lead creation behavior:**

<AccordionGroup>
<Accordion title="Person resolution">
If `personId` is provided, the system uses that existing person. Otherwise it searches by email/phone via `PersonService.findOrCreatePerson`, or creates a new person (requires at least `firstName`).
</Accordion>
<Accordion title="Deduplication">
If both `referenceId` and `leadSource` are provided, the system checks for an existing lead with the same combination to prevent duplicate imports from external sources (e.g., Bayut, PropertyFinder). A unique DB constraint (error code 23505) catches race conditions.
</Accordion>
<Accordion title="Stage assignment">
New leads are assigned to the "New" stage by default. A custom `stageId` can override this.
</Accordion>
<Accordion title="Stakeholder assignment">
If `assignedUserId` is provided, a primary stakeholder is created with 100% commission. If `teamId` is also provided, the stakeholder is linked to that team. If no `assignedUserId`, the lead goes to the Pool (no stakeholders = unassigned).
</Accordion>
<Accordion title="Property interests">
Optional property interests can be attached at creation. At least one must be marked `isPrimary`. If none is marked primary, the first one is auto-set as primary.
</Accordion>
</AccordionGroup>

### Contact (CRM business record)

Represents the ongoing business relationship with a person. Contact is the **business layer** entity — created when a lead converts.

```
Contact
├── person_id (required) → Person (OneToOne)
├── Business fields: customer_since, customerType, rating, lifetimeValue
├── Reference: reference_number (nullable, indexed)
├── Internal: internal_notes (text, nullable)
├── Company associations: companyRoles → ContactCompanyRole
├── Deals: dealContacts → DealContact[]
└── Notes: use polymorphic 'note' table
```

**Key rules:**

- One Person has ONE Contact (unlike leads)
- Contact visibility is derived from linked lead assignments
- No direct assignment fields — visibility is inherited
- Deals link to Contact, not Person
- Companies link via `ContactCompanyRole`

**Visibility logic:**

```
Person: John Smith
├── Lead #1 → assigned_to: Agent A
├── Lead #2 → assigned_to: Agent B
└── Contact → Visible to: Agent A, Agent B, Managers, Owners
             NOT visible to: Other agents without lead access
```

**Company roles:**

```
Contact: John Smith (qualified customer)
├── companyRoles[]:
│   ├── Company: Acme Corp, Role: Manager, isPrimary: true
│   └── Company: XYZ Ltd, Role: Advisor, isPrimary: false
└── Access via: contact.companyRoles → ContactCompanyRole[]
```

### Company (organization profile)

Represents a business organization.

```
Company
├── name, legal_name, industry, website
├── company_size, number_of_employees, annual_revenue
├── description, logo_url
├── Social: linkedin_url, twitter_url, facebook_url
├── Related tables:
│   ├── company_email (multiple)
│   ├── company_phone (multiple)
│   └── company_address (multiple)
└── contact_company_role → Links contacts to company with roles
```

Companies link to **Contact** via `contact_company_role`. This reflects that company associations are business relationships on the Contact layer.

**`contact_company_role` fields:**

```
├── contact_id → Contact (not Person)
├── company_id → Company
├── role (owner, director, manager, employee, partner, contractor)
├── job_title
├── department
├── is_primary (primary contact for this company)
├── is_decision_maker
├── start_date, end_date
├── is_active (currently active role)
├── notes
```

### Deal (revenue opportunity)

Represents a potential revenue transaction.

```
Deal
├── title (deal name/description)
├── dealContacts[] → DealContact (ALL contacts via junction table)
│   └── isPrimary=true for primary contact
├── originating_lead_id → Lead (optional — source lead for lineage)
├── propertyInterest → DealPropertyInterest (Real Estate, 1:1)
│   └── originatingInterest → LeadPropertyInterest (lineage)
├── stage_id → deal_stage (pipeline position)
├── value, currency, description
├── probability
├── expected_close_date
├── Closure: is_closed, is_won, closed_at, closed_by
├── loss_reason (enum), loss_notes
├── Commission: commission_rate (default 2%), total_commission (calculated on close)
└── Notes: use polymorphic 'note' table
```

**Key rules:**

- All contacts linked via `DealContact` junction table — no direct `contact_id` on Deal
- Exactly one `DealContact` must have `isPrimary=true`
- Access Person via `dealContacts.find(dc => dc.isPrimary).contact.person`
- Access Companies via `dealContacts.find(dc => dc.isPrimary).contact.companyRoles`
- `originating_lead_id` is optional for lineage tracking
- All lead stakeholders are inherited to new deals
- Closure uses `isClosed` (boolean) + `isWon` (boolean) + `closedAt` (date) + `closedBy` (user)
- `commissionRate` stored on deal (default 2%); `totalCommission` calculated on close as `value × (commissionRate / 100)`

**Deal creation behavior:**

- **Contact required** — Deals require a Contact (not a Person directly). Deals are for converted contacts only.
- **Archived entities block creation** — Cannot create a deal from an archived Lead or with an archived Contact.
- **Duplicate prevention** — Checks for an existing deal with the same `lead + title + value` or `contact + title + value`. A unique DB constraint catches race conditions.
- **DealContact** — A primary `DealContact` record is created with `role = BUYER` by default.
- **Stakeholder inheritance** — If the deal originates from a lead, all lead stakeholders are inherited. If no lead, a primary stakeholder is created for the assigned user with 100% commission.

**Deal update behavior:**

- Cannot update an archived deal or a deal in team pool (team-only primary, no user).
- If `assignedToId` is provided and the user is not already a stakeholder, a new primary stakeholder is created with 100% commission.

**Property interest rules (1:1 relationship):**

- One deal = one property = one transaction. If a client buys multiple units, create multiple deals.
- If created from lead: copy `LeadPropertyInterest` → `DealPropertyInterest` using explicit selection (`propertyInterestId`) or auto-selection (**primary** interest first, then **first non-deleted** as fallback).
- If created directly (walk-in): create `DealPropertyInterest` with no originating interest.
- If a lead has multiple property interests, create separate deals (one per interest).

**`DealContact` junction table:**

```
deal_contact
├── deal_id → Deal
├── contact_id → Contact
├── role (buyer, seller, tenant, landlord, investor, guarantor, decision_maker, influencer, other)
├── is_primary
├── notes
```

## Assignment and commission system

### entity_stakeholder (unified assignment)

A single table manages who is assigned to leads and deals and their commission split.

```
entity_stakeholder
├── entity_type (lead | deal)
├── entity_id
├── user_id → User (nullable — null for team-only stakeholders)
├── team_id → Team (nullable)
├── is_primary (only one per entity)
├── access_level (READ | WRITE | FULL, default: READ)
├── Commission:
│   ├── commission_percentage (e.g., 50.00%)
│   └── original_team_commission (nullable — records team allocation at time of claim)
├── Inheritance tracking:
│   ├── inherited_from_stakeholder_id (nullable)
│   └── inherited_from_lead_id (nullable)
├── assigned_at, assigned_by_id
├── notes
```

**Access levels:**

| Level | Description | Typical use |
| --- | --- | --- |
| READ | View only | Observers, auto-synced stakeholders |
| WRITE | View and edit | Collaborating agents |
| FULL | View, edit, delete, transfer | Primary owner, managers |

<Note>
Primary stakeholders default to FULL access unless explicitly specified. Only one stakeholder can have `is_primary = true` per entity.
</Note>

#### Dual-record model

A user can hold **multiple** stakeholder rows on the same entity with different team contexts:

- One row with `team_id = NULL` (personal/global-pool stake)
- One row per team they claim from (e.g., `team_id = Team A`)

The unique index `uq_stakeholder_user_team_entity` enforces `(entity_type, entity_id, user_id, team_id)`. PostgreSQL treats NULLs as distinct, so `(lead-1, ahmed, NULL)` and `(lead-1, ahmed, team-A)` are both allowed.

At commission payment time, aggregate all rows per user: `SUM(commissionPercentage) WHERE userId = X AND entityId = Y`.

**Primary handling with dual records:**

- Claiming from a team pool while holding a personal stake: personal stake demoted to `isPrimary=false`, new team-scoped claim becomes primary.
- Team-scoped stakeholder removed and user has a personal stake: personal stake auto-promoted to `isPrimary=true`.
- Access control resolves the **highest** `accessLevel` across all of a user's stakeholder rows.

#### Bidirectional sync (deal ↔ lead)

When assigning or removing stakeholders on deals, the system automatically syncs with the originating lead:

| Action | Sync behavior | Conditions |
| --- | --- | --- |
| Assign to deal | Auto-assign to originating lead as observer | 0% commission, read access. Skips if same user+team already exists or lead archived. |
| Remove from deal | Auto-remove from originating lead | Matches by user+team context. Only if: (1) user's only deal from that lead, (2) no commission on lead, (3) not primary, (4) lead not archived. |

<Tip>
Sync is **team-context-aware**: assign/remove operations match by `(user_id, team_id)`, not just `user_id`. This prevents confusing a user's personal stake with a team-scoped auto-synced observer.
</Tip>

Duplicate checks in `createInTransaction`, `addStakeholderToEntityInTransaction`, and `getAddPreview` use `(user, team)` instead of just `user`, allowing dual records while preventing true duplicates within the same team context.

### Lead creation assignment authorization

When creating a lead via `POST /leads`, the endpoint has **no `@CheckAccess` decorator** (any authenticated user can create leads), but server-side validation enforces assignment rules based on permissions.

**Authorization matrix:**

| Assignment scenario | Required permission | Notes |
| --- | --- | --- |
| Assign to self | None (always allowed) | Most common case for normal agents |
| Pool lead (global) | `crm.manage` | No `assignedUserId`, no `teamId` |
| Pool lead (team-specific) | `team_crm.manage` in target team | Only `teamId` provided — lead goes to team pool |
| Assign to another user | `crm.manage` | `assignedUserId ≠ userId` |
| Team selection | User must be member OR `crm.manage` | Validates `teamId` if specified |

**Assignment behavior on lead creation:**

<Steps>
<Step title="User + team">
If both `assignedUserId` and `teamId` are provided, a stakeholder record is created with both fields populated. The lead is assigned to the user and associated with the team.
</Step>
<Step title="Team only">
If only `teamId` is provided (no `assignedUserId`), a team-only stakeholder record is created with `user_id = NULL`. The lead appears in the **team pool**.
</Step>
<Step title="User only">
If only `assignedUserId` is provided (no `teamId`), a stakeholder record is created with only the user. The lead is assigned without team association.
</Step>
<Step title="No assignment">
If neither is provided, no stakeholder record is created. The lead appears in the **organization pool**.
</Step>
</Steps>

**Frontend dynamic behavior:**

| User type | Assigned to field | Team selection | Pool lead? |
| --- | --- | --- | --- |
| Normal agent | Self (disabled) | Optional (my teams) | No |
| `team_crm.manage` in Team X | Self (default, unlocked when Team X selected) | Optional (my teams) | Yes (Team X pool only) |
| `crm.manage` | Anyone (enabled) | Any team (all org teams) | Yes (any pool) |

### entity_stakeholder_history (audit trail)

Tracks COMPLETED stakeholder assignment states. History records represent what a stakeholder **was** (completed states), not what they changed from/to. The current state lives on `EntityStakeholder`.

```
entity_stakeholder_history (completed states only)
├── entity_type, entity_id
├── stakeholder_id (the stakeholder record)
├── user_id (nullable), team_id
├── COMPLETED STATE (what they WERE):
│   ├── commission_percentage
│   ├── is_primary
│   └── access_level (READ | WRITE | FULL)
├── Duration tracking:
│   ├── started_at (when this state began)
│   └── duration_seconds (calculated: created_at - started_at)
├── What ended this state:
│   ├── action (commission_changed | removed | transferred | inherited | primary_changed)
│   ├── transfer_id (if from transfer)
│   └── transferred_to_user_id (if transferred)
├── notes
├── performed_by_id, created_at (also = when state ended)
```

**When to record:**

- Commission change → record the OLD commission state, then update stakeholder
- Primary change → record the OLD primary state, then update stakeholder
- Removal → record final state with action: `removed`
- Transfer → record OLD owner state with action: `transferred`, `transfer_id`, `transferred_to_user_id`

### commission_payment (payment tracking)

Tracks actual commission payments when deals close.

```
commission_payment
├── deal_id → Deal
├── stakeholder_id → entity_stakeholder
├── user_id → User (nullable — null for team-only payments)
├── team_id → Team (nullable — set for team-only payments)
├── percentage (commission percentage snapshot at close)
├── base_amount (total commission pool: deal.value × deal.commissionRate / 100)
├── amount (calculated: base_amount × percentage / 100)
├── currency (default: USD)
├── status (pending, approved, paid, cancelled, on_hold)
├── expected_payment_date
├── payment_method, payment_reference
├── paid_at
├── approved_by, approved_at
├── notes
```

<Note>
Either `user_id` OR `team_id` must be set (validated in service). If a team stakeholder has remaining `commissionPercentage > 0` when a deal closes, the commission is paid to the **team**, representing unallocated budget.
</Note>

<Warning>
**Recipient-only approval:** Only the payment recipient (the user whose `user_id` matches the payment) can approve their own `PENDING → APPROVED` transition. Other status transitions are not restricted to the recipient.
</Warning>

**Status transitions:**

| From | Allowed transitions |
| --- | --- |
| PENDING | APPROVED, ON_HOLD, CANCELLED |
| APPROVED | PAID, ON_HOLD, CANCELLED |
| ON_HOLD | PENDING, CANCELLED |
| PAID | (final — cannot change) |
| CANCELLED | (final — cannot change) |

**Creation logic:**

```typescript
// When deal moves to terminal stage with isClosedWon=true:
totalCommission = deal.value * (deal.commissionRate / 100);

// For each stakeholder on the deal:
commissionPayment = {
  percentage: stakeholder.commissionPercentage,
  baseAmount: totalCommission,
  amount: totalCommission * (stakeholder.commissionPercentage / 100),
  userId: stakeholder.user?.id,   // null if team-only
  teamId: stakeholder.team?.id,   // null if user-based
  status: 'pending',
};

// When deal is reopened (terminal → non-terminal stage):
// Cancel all non-PAID payments (PENDING or APPROVED → CANCELLED)
// If any payments already PAID → throw error (cannot reopen)
```

## Transfer system

### entity_transfer (transfer requests)

Manages formal transfer of leads and deals between agents or teams.

```
entity_transfer
├── entity_type (lead | deal)
├── entity_id
├── from_user_id (nullable), from_team_id (nullable)
├── to_user_id (nullable), to_team_id (nullable)
├── Commission handling:
│   ├── from_commission_total (snapshot of sender's commission at request time)
│   └── from_keeps_percentage (e.g., 30%)
├── Workflow:
│   ├── status (pending, approved, rejected, cancelled)
│   ├── reason, reject_reason
│   ├── approved_by, approved_at
│   └── rejected_by, rejected_at
├── requested_by_id
```

**Transfer types supported:**

| From | To | Description |
| --- | --- | --- |
| User | User | Standard agent-to-agent transfer |
| User | Team | Agent transfers to a team pool |
| Team | User | Team pool claim by a user |
| Team | Team | Team-to-team reallocation |

### Transfer workflow

<Steps>
<Step title="Transfer request">
Agent A requests transfer of Lead #1 to Agent B. An `entity_transfer` record is created with `from_commission_total` (snapshot) and `from_keeps_percentage`.
</Step>
<Step title="Pending review">
Agent B (or manager) reviews the request. They can approve, reject, or cancel.

<Warning>
**Auto-cancel:** If while a transfer is pending, any stakeholder on the entity is removed or has their commission reduced, the transfer is automatically cancelled because the commission split is now invalid.
</Warning>
</Step>
<Step title="Transfer approved">
On approval:
- Agent A's stakeholder is updated: `30%`, `isPrimary=false`
- Agent B's stakeholder is created: `isPrimary=true`, `70%`
- History records created for both agents
- Transfer status set to `approved`
</Step>
<Step title="Downstream effects">
When a new deal is created from the lead later, all lead stakeholders are auto-copied to the deal with their commission splits preserved.
</Step>
</Steps>

## Activity and communication system

### Architecture (polymorphic targeting)

The activity system uses a lightweight `activity` table as a timeline index. Full content is stored in separate channel tables.

```
ACTIVITY (lightweight timeline index)
├── target_entity_type → 'lead' | 'deal' | 'contact' | 'company'
├── target_entity_id → UUID of the target entity
├── contact_id (nullable) → for bubble-up aggregation
├── activity_type + reference_id → points to channel table
├── summary (for quick timeline display)
└── is_pinned, is_important
              │
              ▼ reference_id points to:
┌──────────┬──────────┬──────────┬──────────┬──────────┬──────────┐
│ call_log │ email_   │ sms_log  │ meeting  │ site_    │ note     │
│          │ log      │          │          │ visit    │          │
│ Content  │ Content  │ Content  │ Content  │ Content  │ Content  │
│ ONLY     │ ONLY     │ ONLY     │ ONLY     │ ONLY     │ ONLY     │
│ (no FKs) │ (no FKs) │ (no FKs) │ (no FKs) │ (no FKs) │ (no FKs) │
└──────────┴──────────┴──────────┴──────────┴──────────┴──────────┘
```

<Info>

### Activity creation workflow

<Steps>
<Step title="Create channel record">
Create the content record in the appropriate channel table (e.g., `call_log`). This stores the full data — phone number, direction, outcome, duration, notes.
</Step>
<Step title="Create activity record">
Create an `activity` record pointing to the channel record via `reference_id`. The system auto-fills `contact_id` for bubble-up visibility based on the target entity.

```

├── target_entity_type: 'lead'
├── target_entity_id: lead-123
├── contact_id: contact-456 (auto-filled from lead.person.contact)
├── activity_type: 'call'
├── reference_id: call-log-uuid
├── summary: "Called +1234567890 • 5 min • Connected"
```
</Step>
<Step title="Timeline display">
The timeline UI shows `activity.summary` (fast query, ordered by `created_at`). Clicking to expand fetches full channel details via `reference_id` on demand.
</Step>
</Steps>

### Contact ID resolution logic

The system automatically populates `contactId` based on the target entity:

```typescript
// Company → null (no bubble up)
if (targetType === 'company') return null;

// Contact → use contact itself
if (targetType === 'contact') return targetId;

// Lead → get from lead.person.contact (if exists)
if (targetType === 'lead') {
  const lead = await em.findOne(Lead, targetId, { populate: ['person.contact'] });
  return lead?.person?.contact?.id ?? null;
}

// Deal (communication) → require selectedContactId from UI
// Deal (internal note) → null
if (targetType === 'deal') {
  if (isCommunicationActivity(activityType)) {
    if (!selectedContactId) throw new BadRequestException('Contact required');
    return selectedContactId;
  }
  return null;
}
```

```typescript
// In ContactService.create(), after persisting contact:
await activityBackfillService.backfillContactActivities(contact.id, person.id, em);

// 1. Find all leads for this person

```

## Stage history and analytics

### Lead stage system

Lead stages use a `systemType` enum to define programmatic behavior. There are 5 system stages:

| systemType | Default name | Behavior |
| --- | --- | --- |
| `new` | New | Default stage when lead is created |
| `contacted` | Contacted | Auto-transition when first activity is logged |
| `qualified` | Qualified | Lead has been qualified |
| `converted` | Converted | Triggers conversion (sets `lead.convertedAt`) |
| `disqualified` | Disqualified | Triggers disqualification (sets `lead.disqualifiedAt`) |

<Info>
System stages have `systemType` set; custom stages have `systemType = null`. The display name can be customized per organization, but `systemType` determines behavior. `isSystem`, `isConverted`, `isDisqualified`, and `isTerminal` are computed properties derived from `systemType`.
</Info>

```typescript
// Computed properties on LeadStage entity
get isSystem(): boolean { return this.systemType != null; }
get isConverted(): boolean { return this.systemType === SystemLeadStageType.CONVERTED; }
get isDisqualified(): boolean { return this.systemType === SystemLeadStageType.DISQUALIFIED; }
get isTerminal(): boolean { return this.isConverted || this.isDisqualified; }
get isNew(): boolean { return this.systemType === SystemLeadStageType.NEW; }
get isContacted(): boolean { return this.systemType === SystemLeadStageType.CONTACTED; }
```

**Stage order and custom stages:**

- System stages maintain relative order: NEW < CONTACTED < QUALIFIED < CONVERTED/DISQUALIFIED
- Custom stages can only be placed between CONTACTED and QUALIFIED
- Organizations can rename system stages but cannot delete them

#### Terminal stage behaviors

When a lead's stage changes, the service checks the new stage's `systemType` and triggers side effects:

<Tabs>
<Tab title="Conversion">
When a lead moves to the **Converted** stage (`isConverted = true`) and `convertedAt` is not already set:

1. Check if a Contact already exists for `lead.person` in this organization
2. If the Contact exists but is **archived**, auto-unarchive it (clear `isArchived`, `archivedAt`, `archivedBy`)
3. If no Contact exists, create one with `customerSince = NOW()` and `customerType = 'lead_conversion'`
4. Set `lead.convertedAt = NOW()`

<Note>
The idempotency check prevents duplicate contacts when a lead is moved to Converted multiple times or when the person already has a contact from a different lead. If the existing Contact is archived, the system auto-unarchives it rather than blocking conversion.
</Note>
</Tab>
<Tab title="Disqualification">
When a lead moves to the **Disqualified** stage (`isDisqualified = true`) and `disqualifiedAt` is not already set:

- Set `lead.disqualifiedAt = NOW()`
- If `stageChangeReason` is provided, store it in `lead.disqualificationNotes`

Setting `disqualificationReason` directly on the update DTO (without a stage change) also triggers `disqualifiedAt = NOW()` if it was previously null.
</Tab>
<Tab title="Auto-requalify">
When a lead moves **out of** a disqualified stage to any non-disqualified stage, and `disqualifiedAt` is set:

- Clear `lead.disqualifiedAt`
- Clear `lead.disqualificationReason`
- Clear `lead.disqualificationNotes`

This ensures that returning a lead to the pipeline automatically clears all disqualification data. No manual reset is needed.
</Tab>
</Tabs>

#### Stage change reason (`stageChangeReason`)

Both lead and deal update DTOs accept an optional `stageChangeReason` field. When a stage transition occurs:

1. The reason is stored as `notes` in the `lead_stage_history` / `deal_stage_history` record
2. For lead disqualification specifically, the reason is also stored in `lead.disqualificationNotes`

This enables audit trails and analytics on why stage transitions occurred.

### Deal stage system

Deal stages also use a `systemType` enum. There are 5 system stages:

| systemType | Default name | Behavior |
| --- | --- | --- |
| `new` | New | Default stage when deal is created |
| `proposal` | Proposal | Proposal sent |
| `negotiation` | Negotiation | In active negotiation |
| `closed_won` | Closed Won | Triggers deal closure (`deal.isWon = true`, `deal.closedAt`) |
| `closed_lost` | Closed Lost | Triggers deal closure (`deal.isWon = false`, `deal.closedAt`) |

```typescript
// Computed properties on DealStage entity
get isSystem(): boolean { return this.systemType != null; }
get isClosedWon(): boolean { return this.systemType === SystemDealStageType.CLOSED_WON; }
get isClosedLost(): boolean { return this.systemType === SystemDealStageType.CLOSED_LOST; }
get isTerminal(): boolean { return this.isClosedWon || this.isClosedLost; }
```

**Stage order and custom stages:**

- System stages maintain relative order: NEW < PROPOSAL < NEGOTIATION < CLOSED_WON < CLOSED_LOST
- Custom stages can only be placed between NEGOTIATION and CLOSED_WON
- Organizations can rename system stages but cannot delete them

#### Terminal stage behaviors

When a deal's stage changes, the service checks the new stage's `systemType` and triggers side effects:

<Tabs>
<Tab title="Closing">
When a deal moves to a **terminal stage** (`isTerminal = true`):

1. Set `deal.isClosed = true`, `deal.isWon = newStage.isClosedWon`
2. Set `deal.closedAt = NOW()`, `deal.closedBy = currentUser`
3. If `isClosedWon`: calculate `deal.totalCommission = deal.value × (deal.commissionRate / 100)` and create `commission_payment` records for each deal stakeholder
</Tab>
<Tab title="Reopening">
When a deal moves from a **terminal** stage back to a **non-terminal** stage:

1. Cancel all non-PAID commission payments (`PENDING` or `APPROVED` → `CANCELLED`)
2. If any payments are already `PAID`, throw an error — the deal cannot be reopened
3. Clear all closure fields: `isClosed = false`, `isWon = false`, `closedAt`, `closedBy`, `totalCommission`

<Warning>
Reopening a deal fully reverses the closure. If any commission payments have already been marked as PAID, the reopen is blocked because the financial transaction cannot be reversed from within the CRM.
</Warning>
</Tab>
</Tabs>

### Two-tier stage architecture

Both lead and deal stages use a global-plus-override architecture:

1. **Global stages** (`organization = NULL`) — System-defined stages shared across all organizations, seeded on startup.
2. **Organization-specific stages** (`organization = org_id`) — Custom stages created by the org, or overrides of global stages (e.g., changing the color of "New").

**Lookup priority:** Organization-specific → Global. When resolving a stage by `systemType`, the service checks org-specific first, then falls back to global.

<Tabs>
<Tab title="Update rules">
| Stage type | Name | Color | Order |
| --- | --- | --- | --- |
| Global (`org = NULL`) | Cannot modify directly | Cannot modify directly | Cannot modify directly |
| System override (org stage with `systemType`) | Cannot change | Can change | Cannot change |
| Custom stage (no `systemType`) | Can change | Can change | Via reorder API |

<Note>
Updating a global stage auto-creates an org-specific override instead of modifying the global record (lead stages). Deal stages throw an error suggesting the user create an override.
</Note>
</Tab>
<Tab title="Deletion rules">
- **Global system stages** — Cannot be deleted
- **Organization system overrides** — Cannot be deleted
- **Custom stages with active leads/deals** — Cannot be deleted until entities are moved to another stage
- **Custom stages with no active entities** — Can be deleted
</Tab>
<Tab title="Reordering">
- All visible stages must be included in the reorder operation
- System stages act as **anchor points** with canonical order values (1000, 2000, 3000, ...)
- Custom stages receive interpolated order values between surrounding anchors
- All ordering constraints are validated before applying (e.g., NEW before CONTACTED)
</Tab>
</Tabs>

### History design principle

History records track **completed stages only**. The current stage lives on the entity itself.

- `Lead.stage` / `Deal.stage` — current stage
- `Lead.stageEnteredAt` / `Deal.stageEnteredAt` — when the entity entered the current stage
- `lead_stage_history` / `deal_stage_history` — records of stages the entity has **left**

<Warning>
No history record is created when entering a stage. History is created only when **leaving** a stage. The current stage is never in history until you leave it.
</Warning>

### Stage history tables

<Tabs>
<Tab title="lead_stage_history">
```
lead_stage_history (completed stages only)
├── lead_id → Lead
├── stage_id → lead_stage (the stage that was completed)
├── entered_at (when lead entered this stage)
├── duration_seconds (calculated: created_at - entered_at)
├── next_stage_id → lead_stage (optional, the stage moved to)
├── notes
├── changed_by_id
├── created_at (also serves as "exited_at")
```
</Tab>
<Tab title="deal_stage_history">
```
deal_stage_history (completed stages only)
├── deal_id → Deal
├── stage_id → deal_stage (the stage that was completed)
├── entered_at (when deal entered this stage)
├── duration_seconds (calculated: created_at - entered_at)
├── next_stage_id → deal_stage (optional, the stage moved to)
├── notes
├── changed_by_id
├── created_at (also serves as "exited_at")
```
</Tab>
</Tabs>

### Stage change recording

When a lead's `stage_id` changes from A to B:

```
1. CREATE lead_stage_history for stage A (the one being left):
   ├── stage_id: A (the completed stage)
   ├── entered_at: lead.stage_entered_at
   ├── duration_seconds: (NOW() - lead.stage_entered_at) in seconds
   ├── next_stage_id: B (for analytics)
   ├── changed_by_id: current_user
   └── created_at: NOW() (implicitly = exited_at)

2. Update lead:
   ├── stage_id: B
   └── stage_entered_at: NOW()
```

### Analytics queries

```sql
-- Average time in each stage (lead pipeline)
SELECT
  ls.name as stage_name,
  AVG(lsh.duration_seconds) as avg_seconds,
  COUNT(*) as transitions
FROM lead_stage_history lsh
JOIN lead_stage ls ON lsh.stage_id = ls.id
WHERE lsh.organization_id = :org_id
GROUP BY ls.id, ls.name
ORDER BY ls.order;

-- Conversion rate by stage
SELECT
  ls.name,
  COUNT(CASE WHEN l.converted_at IS NOT NULL THEN 1 END) as converted,
  COUNT(*) as total,
  COUNT(CASE WHEN l.converted_at IS NOT NULL THEN 1 END)::float / COUNT(*) as rate
FROM lead l
JOIN lead_stage ls ON l.stage_id = ls.id
WHERE l.organization_id = :org_id
GROUP BY ls.id, ls.name;
```

### Stakeholder queries

```sql
-- Get all stakeholders for a lead/deal
SELECT es.*, u.name as user_name, t.name as team_name
FROM entity_stakeholder es
JOIN user u ON es.user_id = u.id
LEFT JOIN team t ON es.team_id = t.id
WHERE es.entity_type = :type AND es.entity_id = :id AND es.is_deleted = false
ORDER BY es.is_primary DESC, es.commission_percentage DESC;
```

### Permission-aware queries

```sql
-- Leads visible to user (as stakeholder) — uses EXISTS for performance
SELECT l.* FROM lead l
WHERE EXISTS (
  SELECT 1 FROM entity_stakeholder es
  WHERE es.entity_type = 'lead' AND es.entity_id = l.id
    AND es.user_id = :user_id AND es.is_deleted = false
```

## Business rules

### Lead rules

| Rule | Description |
| --- | --- |
| Multiple leads per person | One person can have multiple leads (each inquiry tracked separately) |
| Lead requires person | `person_id` is required |
| Intent is single enum | Lead has one intent (buyer, seller, renter, landlord, investor) |
| Stage changes tracked | Every stage change creates a `lead_stage_history` record |
| Conversion creates deal | When lead converts, create deal with stakeholder copy |

### Deal rules

| Rule | Description |
| --- | --- |
| Unified DealContact | All contacts via `deal_contact` junction table |
| Primary contact required | Exactly one DealContact per deal must have `is_primary=true` |
| Deal from lead | `originating_lead_id` links to source lead (optional) |
| Inherit stakeholders | Copy all lead stakeholders to new deal |
| Inherit property interest | Copy `LeadPropertyInterest` → `DealPropertyInterest` (1:1) |
| Walk-in deals | Deals without leads create `DealPropertyInterest` directly |
| One deal = one property | 1:1 relationship — multiple units = multiple deals |
| Access via DealContact | Person via `dealContact.contact.person`, Companies via `dealContact.contact.companyRoles` |
| Close creates payments | Moving to `closedWon` sets `isClosed=true`, `isWon=true`, creates `commission_payment` records |

### Assignment rules

| Rule | Description |
| --- | --- |
| One primary | Only one stakeholder can have `is_primary=true` |
| Primary required | At least one primary stakeholder must exist at all times |
| Commission total | Must sum to exactly 100% (or 0%); strictly enforced on batch update |
| Team as budget | Team stakeholder's `commissionPercentage` is the allocatable budget |
| Team budget calc | Budget = team remaining % + team members' combined % |
| Team pool freeze | Entity blocked from modification when primary is team-only (no user) |
| Transfer ceiling | `fromKeepsPercentage` cannot exceed primary's current commission |
| Transfer 0% OK | 0% commission transfers allowed to existing stakeholders (primary-only transfer) |
| Transfer requester | Only primary stakeholder or member of primary team can initiate transfer |
| Transfer-back OK | Transferring back to previous owners/teams is permitted |
| Transfer update | Transfer to existing stakeholder updates them instead of creating duplicate |
| Permissions only | Stakeholder management requires explicit RBAC permissions, not entity access |
| AccessLevel on create | `accessLevel` (READ/WRITE/FULL) can be explicitly set on stakeholder creation |
| Primary guard | Cannot remove primary stakeholder while others exist; at least one primary must exist after create |

### Transfer rules

| Rule | Description |
| --- | --- |
| Approval required | Transfer requires target user/team approval |
| Commission preserved | `from_keeps_percentage` sets what transferor keeps |
| Commission snapshot | `from_commission_total` snapshots sender's commission at request time |
| 0% transfer | Allowed only when target is already a stakeholder (transfers primary status without moving commission) |
| Auto-cancel | Pending transfers auto-cancelled when a stakeholder is removed or has commission reduced |
| Team transfers | Supports user→user, user→team, team→user, team→team |
| One pending | Only one pending transfer per entity (enforced by unique partial index) |
| History recorded | All changes logged to `entity_stakeholder_history` |

### Activity rules

| Rule | Description |
| --- | --- |
| Target required | `target_entity_type` + `target_entity_id` are required |
| Contact auto-resolved | `contact_id` auto-filled for bubble-up visibility |
| Reference optional | `reference_id` points to channel table (null for note/task) |
| Summary required | `summary` (max 500 chars) for quick timeline display |
| Channel is truth | Full data (outcome, duration) lives in channel table, not activity |
| Archive block | Cannot create activities for archived entities |

### Note rules

| Rule | Description |
| --- | --- |
| Content only | Note stores title, body, attachments only (no entity references) |
| Entity via activity | Entity targeting handled by Activity table |
| Internal type | Notes are "internal" activities — do not bubble up from Lead/Deal |
| Private option | `is_private=true` limits visibility to creator only |

## Business logic use cases

### Assignment system queries

| Entity | How to get assignment |
| --- | --- |
| **Lead** | `SELECT * FROM entity_stakeholder WHERE entity_type='lead' AND entity_id=:lead_id` |
| **Deal** | `SELECT * FROM entity_stakeholder WHERE entity_type='deal' AND entity_id=:deal_id` |
| **Contact** | Via Person's leads: `JOIN lead ON l.person_id = :person_id` then join `entity_stakeholder` |
| **Person** | No direct assignment — visibility derived from leads |

### Commission logic

**When a deal is created from a lead:**

All lead stakeholders are auto-copied to the deal with inheritance tracking. Commission splits are preserved.

**When a deal closes (won):**

```typescript
function calculateCommissions(deal: Deal) {
  const totalCommission = deal.value * (deal.commissionRate / 100);
  const stakeholders = await getStakeholders('deal', deal.id);

  for (const stakeholder of stakeholders) {
    await createCommissionPayment({
      dealId: deal.id,
      stakeholderId: stakeholder.id,
      userId: stakeholder.user?.id,
      teamId: stakeholder.team?.id,
      percentage: stakeholder.commissionPercentage,
      baseAmount: totalCommission,
      amount: totalCommission * (stakeholder.commissionPercentage / 100),
      status: 'pending',
    });
  }
}
```

<Note>
If a team stakeholder has unallocated budget at close, the commission payment targets the team. This allows organizations to handle team-level commission distribution outside the CRM.
</Note>

### Person vs contact

| Aspect | Person | Contact |
| --- | --- | --- |
| **Purpose** | Identity (who they are) | Relationship (business context) |
| **Created** | When first captured | When lead converts |
| **Assignment** | None | None (derived from leads) |
| **Visibility** | Via leads/contact | Via person's leads |
| **One per org?** | Yes (unique person) | Yes (one contact per person per org) |
| **Holds** | Personal data | Business data |

### Person creation and deduplication

```typescript
async function findOrCreatePerson(data: CreateLeadDto) {
  // 1. Check by email
  const existingByEmail = await personEmailRepo.findOne({
    email: data.email,
    organization: data.organizationId,
  });
  if (existingByEmail) return existingByEmail.person;

  // 2. Check by phone
  const existingByPhone = await personPhoneRepo.findOne({
    phone_number: data.phone,
    organization: data.organizationId,
  });
  if (existingByPhone) return existingByPhone.person;

  // 3. Create new person + contact info
  const person = await personRepo.create({ ... });
  await personEmailRepo.create({ person, email: data.email, is_primary: true });
  await personPhoneRepo.create({ person, phone_number: data.phone, is_primary: true });

  return person;
}
```

### End-to-end lifecycle

<Steps>
<Step title="First contact">
Inquiry comes in (Bayut, website, etc.). Person is created (if not already existing). Lead is created and linked to Person. Stakeholder created (Agent A = primary, 100%).
</Step>
<Step title="Lead worked">
Agent A works the lead. Activities and notes are added. Lead moves through pipeline stages.
</Step>
<Step title="Lead converted">
Lead moves to "Converted" stage. Contact is created, linked to the same Person. Deal is created, linked to the lead. All stakeholders are copied from lead to deal.
</Step>
<Step title="Ongoing relationship">
Person now has: Lead (converted) + Contact + Deal. A new inquiry creates a new Lead for the same Person. Contact accumulates deals, lifetime value, and history.
</Step>
</Steps>

## Data consistency guarantees

The CRM module enforces several invariants across all operations:

| Guarantee | Mechanism | Details |
| --- | --- | --- |
| Transactional writes | `TenantContext.executeInOrg()` | Every write runs in a database transaction. Any failure rolls back the entire operation. |
| Soft-delete everywhere | `isDeleted` flag | No entity is hard-deleted. All deletions set `isDeleted = true`. |
| Unique DB constraints | PostgreSQL unique indexes | Race conditions are caught by database-level constraints (stakeholder uniqueness, transfer uniqueness, lead deduplication). Error code 23505 is handled gracefully. |
| Commission invariant | `Math.abs(total - 100) > 0.01` | Commission percentages must sum to exactly 0% or 100% with 0.01% floating-point tolerance. Enforced on every stakeholder mutation. |
| Primary invariant | Pre-delete + post-create checks | At least one `isPrimary = true` stakeholder must exist when stakeholders are present. Observers (0% commission) cannot be primary. |
| Cascade cleanup | Archive/delete side effects | Archiving cancels pending transfers. Soft-deleting cascades to all child records. Deal `originatingLead` references are set to NULL on lead deletion. |
