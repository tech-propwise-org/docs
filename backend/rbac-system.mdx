This document covers the RBAC (Role-Based Access Control) system architecture: entity relationships, permission resolution, system roles, and multi-organization support.

## Architecture overview

### Design principles

1. **Unified Role & Permission System** — Single `Role` and `Permission` tables with a `scope` field differentiating ORG and TEAM levels.
2. **System vs Custom Roles** — System roles have `organization = NULL` (global/seeded). Custom roles have `organization = specific org ID`.
3. **Explicit Junction Tables** — `UserOrgRole` and `TeamMembershipRole` enable multi-org support with metadata (`assigned_by`, `created_at`, `organization_id` for RLS).
4. **Multi-Tenancy Support** — Users can belong to multiple organizations with different roles per org. RLS enforced at database level.
5. **Permission Inheritance** — Team roles can have both TEAM-scoped and ORG-scoped permissions, controlled via `team_available_permissions`.

### Organization

**Module location:** `/modules/organization/` (separate from RBAC)

Organization serves as the **tenant boundary** for multi-tenancy and is conceptually a peer to User, not a component of RBAC.

```typescript
Organization
├── id: uuid
├── name: string
├── website?: string
├── email?: string
├── vat?: string
├── avatar_url?: string (public URL to org logo, stored as WebP on R2)
├── address?: jsonb (embedded Address object)
├── owner_id: uuid → User (organization owner)
├── Relationships:
│   ├── users → User[] (ManyToMany, owner side, via organization_users)
│   ├── owner → User (ManyToOne)
│   ├── leads → Lead[] (OneToMany)
│   ├── contacts → Contact[] (OneToMany)
│   ├── deals → Deal[] (OneToMany)
│   └── companies → Company[] (OneToMany)
└── Timestamps: created_at, updated_at, is_deleted
```

<Note>
Teams, Roles, and Invitations each reference Organization via their own `@ManyToOne` relationships, but Organization does not declare inverse `@OneToMany` collections for these. CRM entities (Lead, Contact, Deal, Company) do have inverse relationships on Organization.
</Note>

- Each organization has exactly one owner
- Owner is automatically assigned "Owner" role on creation
- System roles (`organization = NULL`) are shared across all orgs
- Soft delete via `isDeleted` flag with `@Filter` decorator

```typescript
├── name: string
├── description?: string
├── color?: string
├── scope: RoleScope (ORG | TEAM)
```

### Permission (unified)

Unified table for all permissions.

```typescript
Permission
├── id: uuid
├── key: string (e.g., 'crm.view', 'team_crm.manage')
├── name: string (display name)
├── description?: string
├── group: string (e.g., 'CRM', 'Sales', 'System', 'TeamCRM')
├── scope: PermissionScope (ORG | TEAM)
├── Flags:
│   ├── is_system_sensitive: boolean (only Owner can use; e.g., system.owner)
│   ├── has_admin_access: boolean (requires admin/leadership access)
│   └── has_write_access: boolean (modify vs read-only)
├── Relationships:
│   └── roles → Role[] (ManyToMany, inverse side)
└── Timestamps: created_at, updated_at, is_deleted

Indexes:
└── (key, scope, is_deleted) [unique, where: is_deleted = false]
```

- Permissions are NOT organization-specific (global registry)
- `key` is the programmatic identifier (used in code via `OrgPermissionKey` / `TeamPermissionKey` enums)
- `key` is unique per scope (the same key cannot appear twice for the same scope)
- No RLS on Permission table (global visibility)
- `is_system_sensitive` permissions are reserved for the Owner role (Admin cannot have them)

## Explicit junction tables

### Why explicit junction tables?

1. **Multi-Org Context** — A user can have different roles in different organizations. Simple pivot tables cannot store organization context.
2. **Metadata Tracking** — Track who assigned the role (`assigned_by_id`), when it was assigned, and when removed.
3. **RLS Compatibility** — Need `organization_id` for database-level filtering via RLS policies.
4. **Audit Trail** — Track role assignment history and identify who made changes.

### UserOrgRole

Manages User ↔ Role assignments in Organization context.

```typescript
UserOrgRole
├── id: uuid (primary key)
├── user_id: uuid → User
├── role_id: uuid → Role (system role with organization = null)
├── organization_id: uuid → Organization (REQUIRED for RLS)
├── assigned_by_id?: uuid → User (who assigned this role)
├── Timestamps:
│   ├── created_at: timestamp
│   ├── updated_at: timestamp
│   └── is_deleted: boolean (soft delete)

Indexes:
├── (user_id, role_id, organization_id) [unique, where: is_deleted = false]
├── (user_id, organization_id)
└── (organization_id, role_id)
```

### Organization field in junction tables

Both junction tables include `organization_id` for RLS compatibility.

| Junction | How `organization_id` is set |
| --- | --- |
| **UserOrgRole** | Explicitly set when assigning role. Allows different roles per org. |
| **TeamMembershipRole** | Inherited from `TeamMembership.organization`. Denormalized for RLS. |

```sql RLS policies
ALTER TABLE "user_org_role" ENABLE ROW LEVEL SECURITY;
CREATE POLICY "organization_isolation_user_org_role" ON "user_org_role"
  USING (organization_id = current_setting('app.organization_id')::uuid);
ALTER TABLE "user_org_role" FORCE ROW LEVEL SECURITY;

ALTER TABLE "team_membership_role" ENABLE ROW LEVEL SECURITY;
CREATE POLICY "organization_isolation_team_membership_role" ON "team_membership_role"
  USING (organization_id = current_setting('app.organization_id')::uuid);
ALTER TABLE "team_membership_role" FORCE ROW LEVEL SECURITY;
```

## Permission resolution

### Resolution flow

```
User requesting action
        │
        ├─── Resolve ORG permissions
        │    └─── Via UserOrgRole → Role → Permission (scope=ORG)
        │
        └─── Resolve TEAM permissions
             └─── Via TeamMembership → TeamMembershipRole → Role → Permission
                  ├─── TEAM-scoped permissions (always allowed)
                  └─── ORG-scoped permissions (must be in team_available_permissions)
```

### Effective permissions calculation

```typescript typescript
interface EffectivePermissions {
  orgPermissions: string[];
  teamPermissions: string[];
  allPermissions: string[];
}

async function getUserEffectivePermissions(
  userId: string,
  organizationId: string,
): Promise<EffectivePermissions> {
  const orgPermissionKeys = new Set<string>();
  const teamPermissionKeys = new Set<string>();

  const userOrgRoles = await em.find(
    UserOrgRole,
    { user: userId, organization: organizationId, isDeleted: false },
    { populate: ['role.permissions'] },
  );

  for (const userOrgRole of userOrgRoles) {
    if (userOrgRole.role.scope === RoleScope.ORG) {
      for (const permission of userOrgRole.role.permissions) {
        orgPermissionKeys.add(permission.key);
      }
    }
  }

  const teamMemberships = await em.find(
    TeamMembership,
    { user: userId, organization: organizationId, isDeleted: false },
    { populate: ['teamMembershipRoles.role.permissions'] },
  );

  for (const membership of teamMemberships) {
    for (const teamMembershipRole of membership.teamMembershipRoles) {
      if (teamMembershipRole.role.scope === RoleScope.TEAM) {
        for (const permission of teamMembershipRole.role.permissions) {
          teamPermissionKeys.add(permission.key);
        }
      } else {
        orgPermissionKeys.add(permission.key);
      }
    }
  }

  return {
    orgPermissions: Array.from(orgPermissionKeys),
    teamPermissions: Array.from(teamPermissionKeys),
    allPermissions: Array.from(new Set([...orgPermissionKeys, ...teamPermissionKeys])),
  };
}
```

```typescript isAdmin
async function isAdmin(userId: string, organizationId: string): Promise<boolean> {
  const userOrgRoles = await em.find(
    UserOrgRole,
    { user: userId, organization: organizationId, isDeleted: false },
    { populate: ['role'] },
  );

  return userOrgRoles.some(

  );
```

```typescript hasTeamPermission
async function hasTeamPermission(
  userId: string, teamId: string, permissionKey: string,
): Promise<boolean> {
  const membership = await em.findOne(
    TeamMembership,
    { user: userId, team: teamId, isDeleted: false },
    { populate: ['teamMembershipRoles.role.permissions'] },
  );
  if (!membership) return false;

  for (const tmr of membership.teamMembershipRoles.getItems()) {
    if (tmr.role.permissions.getItems().some((p) => p.key === permissionKey)) {
      return true;
    }
  }
  return false;
}
```

### Team available permissions

Team roles can have ORG-scoped permissions, but only if those permissions are in the team's `availablePermissions`:

```typescript typescript
async function validateTeamRolePermissions(
  teamId: string, permissionIds: string[]
): Promise<void> {
  const team = await em.findOne(Team, { id: teamId }, { populate: ['availablePermissions'] });
  const availablePermissionIds = team.availablePermissions.getItems().map(p => p.id);

  for (const permissionId of permissionIds) {
    const permission = await em.findOne(Permission, { id: permissionId });
    if (permission.scope === PermissionScope.ORG) {
      if (!availablePermissionIds.includes(permissionId)) {
        throw new Error(`Permission ${permission.key} not available for team roles`);
      }
    }
  }
}
```

## System roles

### Overview

System roles are seeded globally and have `organization = NULL`:

| Role | Scope | Description | Permission strategy |
| --- | --- | --- | --- |
| **Owner** | ORG | Organization owner | `allPermissions: true` (including system-sensitive) |
| **Admin** | ORG | Organization admin | `allNonSensitive: true` (excludes `system.owner`) |
| **Basic** | ORG | Basic user | `permissionKeys: []` (relies on ownership checks) |
| **Team Leader** | TEAM | Team leader | `allPermissions: true` (all team + available org) |
| **Agent** | TEAM | Team agent | `permissionKeys: []` (assigned per-team) |

### Default permissions

The system uses **domain-level coarse-grained permissions** (e.g., `crm.view`, `crm.manage`) rather than fine-grained per-entity permissions.

<Tabs>
<Tab title="Owner">
Gets **all** org-scoped permissions including system-sensitive ones:

`system.owner`, `system.admin`, `crm.view`, `crm.manage`, `sales.view`, `sales.manage`, `real_estate.manage`, `marketing.manage`
</Tab>
<Tab title="Admin">
Gets all org-scoped permissions **except** those marked `isSystemSensitive`:

`system.admin`, `crm.view`, `crm.manage`, `sales.view`, `sales.manage`, `real_estate.manage`, `marketing.manage`

Cannot: `system.owner` (system-sensitive)
</Tab>
<Tab title="Basic">
No org-level permissions by default. Relies on resource ownership checks for access.
</Tab>
<Tab title="Team Leader">
Gets all team-scoped permissions + available org permissions for the team:

`team_crm.view`, `team_crm.manage`, `team_sales.view`, `team_sales.manage`, `team.admin`, `team_transfer.manage`, `team_report.view`
</Tab>
<Tab title="Agent">
No default team permissions. Specific permissions assigned per-team by team leader/admin.
</Tab>
</Tabs>

## Invitation system

### Invitation entity

```typescript
Invitation
├── id: uuid
├── email: string
├── status: InvitationStatus (pending, completed, expired, cancelled)
├── token: string (secure token for acceptance)
├── expires_at?: timestamp (nullable)
├── invited_by_id: uuid → User (ManyToOne)

├── organization_id: uuid → Organization (ManyToOne)
├── org_role_assignments: jsonb ({roleId, roleName}[])
├── team_id?: uuid → Team (ManyToOne, nullable)

├── (invited_by_id)
```

<Note>
RBAC assignments are stored in two separate JSONB fields (`orgRoleAssignments` and `teamRoleAssignments`) rather than a single field. Each includes the role name alongside the ID for display purposes without requiring a join. The `team` ManyToOne relationship stores the primary team assignment separately.
</Note>

### Invitation flow

<Steps>
<Step title="Admin creates invitation">
```json
POST /invitations
{
  "email": "newuser@example.com",
  "orgRoles": ["admin-role-uuid"],
  "teamAssignments": [
    { "teamId": "sales-team-uuid", "roleIds": ["agent-role-uuid"] }
  ],
  "welcomeMessage": "Welcome to the team!"

```

Creates invitation with a secure token, `expires_at = 7 days`, `status = 'pending'`, and RBAC assignment snapshots.
</Step>
<Step title="Email sent with invitation link">
The system sends an email containing a link like `https://app.propwise.com/accept-invitation?token=xxx`.
</Step>
<Step title="User accepts invitation">

**New user** (public endpoint):

`POST /invitations/:token/accept` with `firstName`, `lastName`, `password`, `confirmPassword`. System creates account, adds to org, applies RBAC assignments.

**Existing user** (identity token endpoint):

`POST /invitations/:token/accept-existing` requires `IdentityTokenGuard`. System adds existing user to org and applies RBAC assignments.
</Step>
</Steps>

### RBAC assignment on acceptance

<Warning>
Invitation acceptance runs on public/identity-token endpoints that have **no RLS organization context**. All RBAC tables have RLS policies that deny access when `app.organization_id` is not set. The `applyInvitationRbacAssignments` method **must receive the bypassed EntityManager** from the calling `executeWithBypass` context — never `this.em` or injected repositories.
</Warning>

```typescript typescript
private async applyInvitationRbacAssignments(
  em: EntityManager, user: User, invitation: Invitation,

  if (invitation.orgRoleAssignments.length > 0) {
    const roles = await em.find(Role, { id: { $in: roleIds } });
    const userOrgRoles = rolesToCreate.map((role) =>
      em.create(UserOrgRole, { user, role, organization, assignedBy }),
    );
    em.persist(userOrgRoles);
  } else {
    const basicRole = await em.findOne(Role, { name: 'Basic', isSystem: true });
    if (basicRole) {
      em.persist(em.create(UserOrgRole, { user, role: basicRole, organization, assignedBy }));
    }
  }

  if (invitation.teamRoleAssignments.length > 0) {
    const team = await em.findOne(Team, { id: teamId, organization: organizationId });
    const membership = em.create(TeamMembership, { user, team, organization });
    em.persist(membership);

    const teamRoles = await em.find(Role, { id: { $in: teamRoleIds }, scope: 'TEAM' });
    const teamMembershipRoles = teamRoles.map((role) =>
      em.create(TeamMembershipRole, { teamMembership: membership, role, organization, assignedBy }),
    );
    em.persist(teamMembershipRoles);
  }

  await em.flush();
}
```

## Business rules

<Tabs>
<Tab title="Role assignment">
| Rule | Description |
| --- | --- |
| **One Owner per Org** | Only one user can have Owner role per organization |
| **System Role Protection** | System roles (`organization = NULL`) cannot be deleted or modified |
| **Role Scope Validation** | ORG-scoped roles can only be assigned via UserOrgRole |
| **Team Role Validation** | TEAM-scoped roles can only be assigned via TeamMembershipRole |
| **Organization Consistency** | `UserOrgRole.organization` must match user's organization membership |
| **Team Consistency** | `TeamMembershipRole.organization` must match `TeamMembership.organization` |
</Tab>
<Tab title="Permission assignment">
| Rule | Description |
| --- | --- |
| **Permission Scope** | ORG permissions can be on ORG or TEAM roles |
| **Team Permissions** | TEAM permissions can only be on TEAM roles |
| **Team Available Permissions** | ORG permissions on TEAM roles must be in `team.availablePermissions` |
| **System Permissions** | Permissions with `has_admin_access=true` only for Owner/Admin |
</Tab>
<Tab title="Team membership">
| Rule | Description |
| --- | --- |
| **One Membership per Team** | User can only have one membership per team |
| **Organization Match** | `TeamMembership.organization` must match `Team.organization` |
| **Default Role** | New team members get "Agent" role by default |
| **Team Leader Minimum** | Each team must have at least one Team Leader |
</Tab>
<Tab title="Invitations">
| Rule | Description |
| --- | --- |
| **Expiration** | Invitations expire after 7 days |
| **One-Time Use** | Invitation token can only be used once |
| **Email Uniqueness** | Cannot invite email that already exists in organization |
| **Role Validation** | roleIds in invitation must be valid for organization |
| **Team Validation** | teamId in invitation must exist in organization |
</Tab>

## RLS integration

### RLS-enabled tables

| Table | RLS Policy | Notes |
| --- | --- | --- |
| `user` | `organization_isolation_user` | Users visible only in current org context |
| `organization` | No RLS | Users can see all orgs they belong to |
| `role` | Special handling | System roles (org=NULL) visible to all, custom roles filtered by org |
| `permission` | No RLS | Global permission registry |
| `team` | `organization_isolation_team` | Teams filtered by organization |
| `team_membership` | `organization_isolation_team_membership` | Memberships filtered by organization |
| `user_org_role` | `organization_isolation_user_org_role` | Filtered by organization |
| `team_membership_role` | `organization_isolation_team_membership_role` | Filtered by organization |
| `invitation` | `organization_isolation_invitation` | Invitations filtered by organization |

### RLS policy for junction tables

```sql sql
-- UserOrgRole RLS Policy
ALTER TABLE "user_org_role" ENABLE ROW LEVEL SECURITY;
CREATE POLICY "organization_isolation_user_org_role" ON "user_org_role"
  USING (
    organization_id = current_setting('app.organization_id')::uuid
    OR current_setting('app.bypass_rls', true) = 'true'
  );
ALTER TABLE "user_org_role" FORCE ROW LEVEL SECURITY;

-- TeamMembershipRole RLS Policy
ALTER TABLE "team_membership_role" ENABLE ROW LEVEL SECURITY;
CREATE POLICY "organization_isolation_team_membership_role" ON "team_membership_role"
  USING (
    organization_id = current_setting('app.organization_id')::uuid
    OR current_setting('app.bypass_rls', true) = 'true'
  );
ALTER TABLE "team_membership_role" FORCE ROW LEVEL SECURITY;
```

### Role table special handling

The `Role` table does NOT currently have RLS enabled. System roles (`organization = NULL`) and custom roles are both fetched without RLS filtering. Application logic handles visibility.

### Module dependency graph

```

       ↓
```

- **Tenant boundary** — Organization defines the multi-tenancy boundary; all tenant-scoped entities reference it

### User-centric API design

The system follows a **user-centric API design philosophy**:

1. **Default Scope = User's Data** — All default endpoints return only data the current user has access to
2. **Security by Default** — Users cannot accidentally access data outside their scope
3. **Explicit Admin Access** — Administrative/global endpoints are separate (under `/admin` routes)
4. **Least Privilege** — Users only see what they need, admins explicitly request broader access
