Cross-organization property listing and the **deal-first** negotiation flow. This module code is **retained in the repository but is not imported by `RealEstateModule`**, so its controllers/providers are not mounted by the app. The module remains dormant while the system operates with **six active specialized modules**.

## Module overview

The real estate module is structured as **six active specialized modules** spanning three visibility scopes: **global** (no RLS), **org-private** (standard RLS), with the cross-org shared marketplace module dormant. The cross-org `shared-marketplace` code is retained in the repository but is not imported by `RealEstateModule`, so its controllers/providers are not mounted by the app.

The external **Propwise Labs API** stores and serves all property data — both structural (DLD registry) and enrichment (images, payment plans, documents, amenities, pricing, geocoding where exposed). The CRM database stores only CRM-specific data (developer portal auth, ownership links, property interests, marketplace listings, and org inventory context).

<Info>
Architecture: All modules consume property data exclusively through `PropwiseLabsService`. The CRM no longer opens a second MikroORM connection or maps Propwise Labs entities locally. The reference database configuration has been completely removed and replaced with Propwise Labs API configuration (`app.propwiseLabs.*`) including API credentials, timeout settings, and Bird Verify integration for developer portal contact verification.
</Info>

<Note>
Portal syndication — self-contained `Listing` (2026 build): The `portal-syndication` module's `Listing` is now self-contained: every field any portal (Bayut/dubizzle + Property Finder) needs lives on `Listing`, snapshotted from an `InventoryUnit` at create (linked mode) or entered directly (manual mode). The `inventoryUnit` FK is **optional**. Deleting an inventory unit cascades (event-driven, `inventory-unit.deleted` carrying the user's `removeLinkedListings` choice): **remove** (default) delists the unit's listings from all portals and archives them; **keep** unlinks them (`inventoryUnit = null`) so they survive as self-contained manual listings. Buildings auto-maintain `salesStatus` based on available units (`OUT_OF_STOCK` when no units remain `AVAILABLE`, reverting to `ON_SALE` when available units reappear). New `downpaymentType` enum distinguishes amount vs percentage downpayments.

**Listing approval workflow (Gate A):** a user without `real_estate.listing.publish` / `real_estate.manage` submits a listing for approval (`ListingStatus.PENDING_APPROVAL`). A `real_estate.manage` user approves (auto-publishes to the submitter's selected portals) or rejects (`ListingStatus.REJECTED` + `rejectionReason` persisted on the listing) from a manage-only **Requests** queue (`GET /portal-syndication/listings?type=requests` = pending + rejected). The submitter sees the rejection note and can edit + resubmit (clears the rejection) or delete. The listing **Delete** action soft-deletes (`isDeleted=true`) — there is no user-facing archived state (the `archived` status / `isArchived` columns remain only for the inventory-unit delete cascade). A `real_estate.manage` user may delete **any** listing (including pending requests and rejected ones) directly from the Requests queue.

**Notifications:** submitting emits `LISTING_APPROVAL_REQUESTED` to all `real_estate.manage` approvers; approve / reject / delete emit `LISTING_APPROVED` / `LISTING_REJECTED` / `LISTING_DELETED` to the listing publisher (delete only when the deleter is not the publisher). See `NOTIFICATION_IMPLEMENTATION_GUIDE.md`. Full design: `PORTAL_SYNDICATION_SPECIFICATION.md` §1.5 (As-Built).
</Note>

<Info>
Listing media ingest (WebP) + `/storage/upload` `profile` contract: listing photos and floor plans are uploaded through the shared `POST /storage/upload` endpoint (`StorageApi.uploadFile(file, profile?)`), which takes an optional **`profile`** body field validated against `STORAGE_UPLOAD_PROFILES` (unknown → `400`). The default is **`GENERIC_ATTACHMENT`** (long edge ≤ 1920, WebP q80) so email/workflow/deal-document/tiptap uploads stay lean. Listing image call sites pass **`LISTING`** (≤ 2560, q82) and floor-plan call sites pass **`FLOOR_PLAN`** (≤ 2560, q90) so syndication imagery reaches Property Finder at high resolution and floor plans stay legible. The endpoint optimizes the buffer through `ImageOptimizationService` (raster → WebP; PDF/doc/video → passthrough), writes the object under a profile-specific R2 folder (`STORAGE_UPLOAD_FOLDERS`: `email-attachments` / `listings` / `listings/floor-plans`), and reflects the converted `fileName` (`.webp`) + `mimeType` + `fileSize` in the JSON response. `ListingImageService` (the Property Finder pre-flight) is unchanged: its `compliant` check re-processes a source image only when it exceeds PF's 1920×1080 / format limits — an expected single PF-specific re-encode, not an ingest regression.
</Info>

### External property ID pattern

`InventoryUnit` is the CRM boundary for org-owned secondary inventory. It stores plain integer IDs (`refProjectId`, `refBuildingId`, `refUnitId`, `refDeveloperId`, `refPropertyTypeId`, `refPropertySubTypeId`) plus a denormalized Labs snapshot, including the developer display name used by portal syndication and the canonical `propertyType` (`LayoutType` enum) used by portal syndication adapters. Labs raw property type/sub-type names are retained in `labsPropertyType` and `labsPropertySubType` for traceability. Lead interests can use inventory units as candidate matches while retaining search criteria. Deal property rows are concrete-only: they link either an org `InventoryUnit`, a direct Propwise Labs off-plan unit via `refProjectId` / `refBuildingId` / `refUnitId`, or a pending unit-access request before the inventory link is approved.

## Core principle: deal-first approach

The shared marketplace implements a unique **deal-first** negotiation model that prioritizes commitment over browsing:

<Steps>
<Step title="Unit number protection">
**Unit numbers are hidden** in SharedListing browsing to prevent direct contact bypass
</Step>

<Step title="Deal commitment required">
**Buyer agent must propose a SharedDeal** before seeing sensitive unit information
</Step>

<Step title="Automated deal creation">
On acceptance, the system **auto-creates internal Deal entities** in both organizations
</Step>

<Step title="Unit number revelation">
**Unit numbers are revealed only after** SharedDeal status becomes ACCEPTED
</Step>
</Steps>

This approach ensures all cross-organization transactions go through the proper negotiation and commission tracking workflow.

## Cross-org RLS policies

The shared marketplace uses sophisticated Row Level Security (RLS) policies to enable cross-organization visibility while maintaining security boundaries:

```sql
-- SharedListing: readable by all orgs, writable by owning org only
CREATE POLICY "shared_listing_read" ON "shared_listing" 
FOR SELECT USING (true);

CREATE POLICY "shared_listing_write" ON "shared_listing" 
FOR ALL USING (
  listed_by_organization_id = current_setting('app.organization_id')::uuid
);

-- SharedDeal: visible to both participating organizations
CREATE POLICY "shared_deal_access" ON "shared_deal" 
FOR ALL USING (
  listing_organization_id = current_setting('app.organization_id')::uuid
  OR buyer_organization_id = current_setting('app.organization_id')::uuid
);

-- SharedDealMessage: inherits access from parent SharedDeal
CREATE POLICY "shared_deal_message_access" ON "shared_deal_message" 
FOR ALL USING (
  EXISTS (
    SELECT 1 FROM shared_deal sd 
    WHERE sd.id = shared_deal_id 
    AND (
      sd.listing_organization_id = current_setting('app.organization_id')::uuid
      OR sd.buyer_organization_id = current_setting('app.organization_id')::uuid
    )
  )
);
```

<Info>
The custom RLS policies allow SharedListing to be globally readable (for marketplace browsing) but only writable by the listing organization. SharedDeal entities are visible to both buyer and seller organizations for transparent negotiation.
</Info>

### RLS policy mechanics

<Tabs>
<Tab title="Listing visibility">
- **Global read access**: Any authenticated organization can browse all SharedListings
- **Write restrictions**: Only the listing organization can modify their own listings
- **Unit number protection**: Unit numbers are hidden until deal acceptance, regardless of RLS permissions
</Tab>
<Tab title="Deal negotiation">
- **Dual-org access**: SharedDeal entities are visible to both listing and buyer organizations
- **Message inheritance**: SharedDealMessage access inherits from parent SharedDeal visibility
- **State synchronization**: Both organizations see real-time negotiation updates
</Tab>
<Tab title="Auto-deal creation">
- **Context switching**: Uses `TenantContext.executeInOrg()` to create Deal entities in each organization's context
- **Stakeholder creation**: Automatically creates cross-org stakeholder relationships
- **Commission tracking**: Preserves commission split terms in both organization's Deal records
</Tab>
</Tabs>

## Deal-first negotiation flow

The deal-first approach implements a sophisticated negotiation mechanism that protects sensitive information while enabling transparent deal-making:

<Steps>
<Step title="Listing browsing phase">
```json
// SharedListing browse response - unit number hidden
{
  "id": "uuid",
  "projectName": "Downtown Views",
  "areaName": "Downtown Dubai",
  "communityName": "Business Bay",
  "unitType": "Apartment",
  "bedrooms": 2,
  "areaSqft": 1200,
  "listingType": "SALE",
  "listedPrice": 850000,
  "status": "ACTIVE",
  "unitNumber": "Hidden", // Masked during browsing
  "isUnitNumberVisible": false,
  "listedByOrganization": {
    "id": "uuid",
    "name": "ABC Properties"
  }
}
```
</Step>

<Step title="Deal proposal">
```typescript
// POST /marketplace/listings/:id/propose
{
  "proposedPrice": 820000,
  "message": "Competitive offer with quick close",
  "commissionSplit": {
    "buyerAgent": 50,
    "sellerAgent": 50
  },
  "expiresInDays": 7
}
```

Creates SharedDeal with status PROPOSED and initial SharedDealMessage
</Step>

<Step title="Negotiation rounds">
```typescript
// PATCH /marketplace/deals/:id/counter
{
  "counterPrice": 835000,
  "message": "Counter-offer with adjusted terms",
  "commissionSplit": {
    "buyerAgent": 45,
    "sellerAgent": 55
  }
}
```

Updates SharedDeal status to COUNTER_OFFERED with new terms
</Step>

<Step title="Deal acceptance and auto-creation">
```typescript
// PATCH /marketplace/deals/:id/accept
// Triggers complex multi-org Deal creation flow

// 1. Update SharedDeal
sharedDeal.status = SharedDealStatus.ACCEPTED;
sharedDeal.agreedPrice = latestProposedPrice;
sharedDeal.isUnitNumberRevealed = true;
sharedDeal.unitNumberRevealedAt = new Date();

// 2. Create Deal in listing org context
const listingDeal = await this.tenantContext.executeInOrg(
  sharedDeal.listingOrganization.id, 
  async (em) => {
    const deal = em.create(Deal, {
      name: `Marketplace Sale - ${listing.projectName} ${listing.unitNumber}`,
      value: sharedDeal.agreedPrice,
      stage: DealStage.NEGOTIATION,
      source: DealSource.MARKETPLACE,
      refProjectId: listing.refProjectId,
      refUnitId: listing.refUnitId,
      createdBy: sharedDeal.listingAgent,
      organization: sharedDeal.listingOrganization,
    });

    const buyerStakeholder = em.create(EntityStakeholder, {
      deal,
      stakeholderType: StakeholderType.BUYER,
      organization: sharedDeal.buyerOrganization,
      agent: sharedDeal.buyerAgent,
    });

    await em.persistAndFlush([deal, buyerStakeholder]);
    return deal;
  }
);

// 3. Create Deal in buyer org context
const buyerDeal = await this.tenantContext.executeInOrg(
  sharedDeal.buyerOrganization.id,
  async (em) => {
    const deal = em.create(Deal, {
      name: `Marketplace Purchase - ${listing.projectName} ${listing.unitNumber}`,
      value: sharedDeal.agreedPrice,
      stage: DealStage.NEGOTIATION,
      source: DealSource.MARKETPLACE,
      refProjectId: listing.refProjectId,
      refUnitId: listing.refUnitId,
      createdBy: sharedDeal.buyerAgent,
      organization: sharedDeal.buyerOrganization,
    });

    const sellerStakeholder = em.create(EntityStakeholder, {
      deal,
      stakeholderType: StakeholderType.SELLER,
      organization: sharedDeal.listingOrganization,
      agent: sharedDeal.listingAgent,
    });

    await em.persistAndFlush([deal, sellerStakeholder]);
    return deal;
  }
);

// 4. Link deals back to SharedDeal
sharedDeal.listingDeal = listingDeal;
sharedDeal.buyerDeal = buyerDeal;
```
</Step>

</Step>
</Steps>

## Service architecture

The shared marketplace integrates with the Propwise Labs API system through service dependency injection:

### SharedMarketplaceService

```typescript
@Injectable()
export class SharedMarketplaceService {
  constructor(
    private readonly propwiseLabsService: PropwiseLabsService,
    @InjectEntityManager()
    private readonly em: EntityManager,
    private readonly tenantContext: TenantContext,
    private readonly notificationService: NotificationService,
  ) {}

  async createListing(
    inventoryUnit: InventoryUnit, 
    createDto: CreateSharedListingDto,
    listedBy: User
  ): Promise<SharedListing> {
    // Enrich with Propwise Labs data for denormalization
    const projectData = await this.propwiseLabsService.getProject(
      inventoryUnit.refProjectId
    );
    const unitData = await this.propwiseLabsService.getUnit(
      inventoryUnit.refUnitId
    );

    const listing = this.em.create(SharedListing, {
      ...createDto,
      inventoryUnit,
      listedByOrganization: listedBy.organization,
      listedByAgent: listedBy,
      refProjectId: inventoryUnit.refProjectId,
      refUnitId: inventoryUnit.refUnitId,
      // Denormalized data for fast browsing
      projectName: projectData?.name,
      areaName: projectData?.area?.name,
      communityName: projectData?.community?.name,
      unitType: unitData?.unitType?.name,
      bedrooms: unitData?.bedrooms,
      areaSqft: unitData?.areaSqft,
      unitNumber: unitData?.unitNumber,
    });

    await this.em.persistAndFlush(listing);
    return listing;
  }

  async proposeDeal(
    listing: SharedListing,
    proposeDto: ProposeDealDto,
    buyerAgent: User
  ): Promise<SharedDeal> {
    const expiresAt = proposeDto.expiresInDays 
      ? new Date(Date.now() + proposeDto.expiresInDays * 24 * 60 * 60 * 1000)
      : null;

    const deal = this.em.create(SharedDeal, {
      listing,
      listingOrganization: listing.listedByOrganization,
      listingAgent: listing.listedByAgent,
      buyerOrganization: buyerAgent.organization,
      buyerAgent,
      proposedPrice: proposeDto.proposedPrice,
      commissionSplit: proposeDto.commissionSplit,
      expiresAt,
    });

    // Create initial proposal message
    const message = this.em.create(SharedDealMessage, {
      sharedDeal: deal,
      type: SharedDealMessageType.PROPOSAL,
      content: proposeDto.message || 'Initial proposal',
      terms: { 
        price: proposeDto.proposedPrice,
        commissionSplit: proposeDto.commissionSplit 
      },
      createdBy: buyerAgent,
      organization: buyerAgent.organization,
    });

    await this.em.persistAndFlush([deal, message]);
    
    // Notify listing agent
    await this.notificationService.sendDealProposed(deal);
    
    return deal;
  }

  private async createListingDeal(sharedDeal: SharedDeal): Promise<Deal> {
    return this.tenantContext.executeInOrg(
      sharedDeal.listingOrganization.id,
      async (em) => {
        const deal = em.create(Deal, {
          name: `Marketplace Sale - ${sharedDeal.listing.projectName} ${sharedDeal.listing.unitNumber}`,
          value: sharedDeal.agreedPrice,
          stage: DealStage.NEGOTIATION,
          source: DealSource.MARKETPLACE,
          refProjectId: sharedDeal.listing.refProjectId,
          refUnitId: sharedDeal.listing.refUnitId,
          createdBy: sharedDeal.listingAgent,
          organization: sharedDeal.listingOrganization,
        });

        const buyerStakeholder = em.create(EntityStakeholder, {
          deal,
          stakeholderType: StakeholderType.BUYER,
          organization: sharedDeal.buyerOrganization,
          agent: sharedDeal.buyerAgent,
        });

        await em.persistAndFlush([deal, buyerStakeholder]);
        return deal;
      }
    );
  }

  private async createBuyerDeal(sharedDeal: SharedDeal): Promise<Deal> {
    return this.tenantContext.executeInOrg(
      sharedDeal.buyerOrganization.id,
      async (em) => {
        const deal = em.create(Deal, {
          name: `Marketplace Purchase - ${sharedDeal.listing.projectName} ${sharedDeal.listing.unitNumber}`,
          value: sharedDeal.agreedPrice,
          stage: DealStage.NEGOTIATION,
          source: DealSource.MARKETPLACE,
          refProjectId: sharedDeal.listing.refProjectId,
          refUnitId: sharedDeal.listing.refUnitId,
          createdBy: sharedDeal.buyerAgent,
          organization: sharedDeal.buyerOrganization,
        });

        const sellerStakeholder = em.create(EntityStakeholder, {
          deal,
          stakeholderType: StakeholderType.SELLER,
          organization: sharedDeal.listingOrganization,
          agent: sharedDeal.listingAgent,
        });

        await em.persistAndFlush([deal, sellerStakeholder]);
        return deal;
      }
    );
  }
}
```

### PropwiseLabsService integration

```typescript
@Injectable()
export class PropwiseLabsService {
  private accessToken?: string;
  private tokenExpiresAt = 0;

  constructor(
    @Inject('PROPWISE_LABS_CONFIG')
    private readonly config: PropwiseLabsConfig,
  ) {}

  async getProject(id: number): Promise<Project | null> {
    const token = await this.getAccessToken();
    const response = await fetch(`${this.config.baseUrl}/projects/${id}`, {
      headers: { 
        'Authorization': `Bearer ${token}`,
        'Content-Type': 'application/json'
      },
    });
    
    if (!response.ok) return null;
    return response.json();
  }

  async getUnit(id: number): Promise<Unit | null> {
    const token = await this.getAccessToken();
    const response = await fetch(`${this.config.baseUrl}/units/${id}`, {
      headers: { 
        'Authorization': `Bearer ${token}`,
        'Content-Type': 'application/json'
      },
    });
    
    if (!response.ok) return null;
    return response.json();
  }

  async enrichListingData(
    refProjectId: number, 
    refUnitId: number
  ): Promise<{
    projectName?: string;
    areaName?: string;
    communityName?: string;
    unitType?: string;
    bedrooms?: number;
    areaSqft?: number;
    unitNumber?: string;
  }> {
    const project = await this.getProject(refProjectId);
    const unit = await this.getUnit(refUnitId);

    return {
      projectName: project?.name,
      areaName: project?.area?.name,
      communityName: project?.community?.name,
      unitType: unit?.unitType?.name,
      bedrooms: unit?.bedrooms,
      areaSqft: unit?.areaSqft,
      unitNumber: unit?.unitNumber,
    };
  }

  private async getAccessToken(): Promise<string> {
    if (this.accessToken && this.tokenExpiresAt > Date.now()) {
      return this.accessToken;
    }

    const response = await fetch(`${this.config.baseUrl}/auth/token`, {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({
        apiKey: this.config.apiKey,
        apiSecret: this.config.apiSecret,
      }),
    });

    if (!response.ok) {
      throw new Error('Failed to authenticate with Propwise Labs API');
    }

    const { accessToken, expiresIn } = await response.json();
    this.accessToken = accessToken;
    this.tokenExpiresAt = Date.now() + (expiresIn * 1000) - 30000; // 30s buffer
    
    return this.accessToken;
  }
}
```

### Cross-module data flow

```
Developer Portal → Propwise Labs API → SharedListing → SharedDeal → Internal Deals
     ↓                ↓                      ↓             ↓            ↓
Project/Unit     Property Data          Marketplace   Cross-Org    Org-specific
Management      (via API Service)       Listing      Negotiation   Deal Records
```

### Integration points

- **Inventory module**: SharedListing.inventoryUnit relationship provides org inventory context
- **Propwise Labs module**: `PropwiseLabsService` injection for property data enrichment  
- **Core CRM**: Auto-creation of Deal entities during SharedDeal acceptance flow
- **Organizations**: Cross-org visibility through custom RLS policies
- **MapService**: Shared geocoding utility across all real estate modules
- **Notification system**: Deal proposal, acceptance, and status change notifications
- **TenantContext**: Multi-organization context switching for deal auto-creation

### Deal lifecycle integration

1. **Listing creation**: InventoryUnit (inventory) → SharedListing (marketplace)
2. **Deal proposal**: SharedListing → SharedDeal (cross-org negotiation)
3. **Deal acceptance**: SharedDeal → Deal entities (both organizations)
4. **Unit revelation**: Deal acceptance triggers unit number visibility
5. **Commission tracking**: Commission terms flow from SharedDeal to Deal entities

<Note>
The shared marketplace acts as the bridge between organization-scoped inventory and cross-organization deal negotiation, using the Propwise Labs API system as the single source of truth for all property information. However, this module is currently **dormant and not mounted** in the active real estate architecture.
</Note>
