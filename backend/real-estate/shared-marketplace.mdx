**Path:** `src/modules/real-estate/shared-marketplace/`

Cross-organization property listing and the **deal-first** negotiation flow. One of **6 specialized real estate modules** that enables property listings and deals across organizations with custom RLS policies.

<Info>
The shared marketplace is part of the **cross-org** visibility scope, where CRM entities reference property data from the external reference database via plain integer IDs (`refProjectId`, `refUnitId`). SharedListing and SharedDeal are stored in the CRM database and consume property data through `ReferenceDataService`.
</Info>

## Architecture context

The shared marketplace operates within a **modular real estate architecture**:

```
src/modules/real-estate/
├── reference-data/        # Second DB — ALL property data (structural + enrichment)
├── developer-portal/      # Separate auth + project management
├── map/                   # MapService (2GIS geocoding utility)
├── inventory/             # Org-scoped secondary units (standard RLS)
├── shared-marketplace/    # Cross-org listings + deals (custom RLS) ←
└── property-interest/     # Property interests + matching engine
```

### Cross-database reference pattern

SharedListing references property data via **plain integer IDs** pointing to the reference database. No foreign keys exist across databases — this decouples CRM data from the property data source:

```typescript
// SharedListing entity referencing reference DB data:
@Property({ type: 'int', nullable: true })
@Index()
refProjectId?: number;  // ID in reference DB projects table

@Property({ type: 'int', nullable: true })
@Index()
refUnitId?: number;     // ID in reference DB units table
```

## Core principle: deal-first, access-second

1. **Unit number is protected**: Hidden in SharedListing browsing
2. **Deal commitment first**: Buyer agent proposes a SharedDeal before seeing the unit number
3. **Automated deal creation**: On acceptance, system auto-creates internal Deal in both organizations
4. **Unit number revealed**: Only after SharedDeal is ACCEPTED

## Data ownership

| Entity                               | Database | Access Pattern                    |
| ------------------------------------ | -------- | --------------------------------- |
| SharedListing, SharedDeal (CRM data) | CRM DB   | Direct entity access              |
| Property data (projects, units)      | Ref DB   | `ReferenceDataService` via ref IDs |

## Entities

<AccordionGroup>
<Accordion title="SharedListing">

- UUID PK, **custom RLS** (readable by all, writable by owning org)
- ManyToOne: SecondaryUnit, listedByOrganization, listedByAgent
- **Cross-database references**: `refProjectId`, `refUnitId` (plain integers)
- Denormalized display data: projectName, areaName, communityName, unitType, bedrooms, areaSqft
- Listing details: listingType (Sale/Rent), listedPrice, commissionType, commissionValue
- Status: ListingStatus (ACTIVE, UNDER_OFFER, SOLD, RENTED, WITHDRAWN)

**Relationship to SecondaryUnit**: SharedListing links to SecondaryUnit (org-scoped inventory) which itself references property data via `refProjectId`/`refUnitId`. This creates the cross-database integration pattern.

</Accordion>
<Accordion title="SharedDeal">

- UUID PK, **custom RLS** (visible to both participating orgs)
- Links both sides: listingOrganization/listingAgent, buyerOrganization/buyerAgent
- Auto-created deals: listingDeal (Deal), buyerDeal (Deal) - populated on ACCEPT
- Negotiation: status (SharedDealStatus), proposedPrice, agreedPrice, commissionSplit (JSON)
- Unit reveal: **isUnitNumberRevealed** (false until accepted), unitNumberRevealedAt
- OneToMany: messages (SharedDealMessage[])

</Accordion>
<Accordion title="SharedDealMessage">

- Negotiation audit trail
- type: SharedDealMessageType (PROPOSAL, COUNTER, ACCEPT, REJECT, NOTE, STATUS_UPDATE)
- terms: JSON `{ price?, commissionSplit? }`

</Accordion>
<Accordion title="UnitAccessRequest / UnitAccessGrant">

- **Internal org-level** unit number access (not marketplace — marketplace uses SharedDeal)
- Agent within same org requests unit number from unitManager
- Grant has 30-day expiry, status: Active/Expired/Revoked

</Accordion>
</AccordionGroup>

## SharedDeal accept flow

<Warning>
This is a critical flow that creates Deal entities in both organizations and reveals the unit number. It uses `TenantContext.executeInOrg()` to operate in each organization's context.
</Warning>

<Steps>
<Step title="Update shared deal status">
```typescript
// 1. Update shared deal
sharedDeal.status = SharedDealStatus.ACCEPTED;
sharedDeal.agreedPrice = latestProposedPrice;
sharedDeal.isUnitNumberRevealed = true;
sharedDeal.unitNumberRevealedAt = new Date();
```
</Step>

<Step title="Create Deal in listing organization">
```typescript
// 2. Auto-create Deal in LISTING org
await this.tenantContext.executeInOrg(listingOrgId, async (em) => {
  const deal = em.create(Deal, { ... });
  const stakeholder = em.create(EntityStakeholder, { ... });
});
```
</Step>

<Step title="Create Deal in buyer organization">
```typescript
// 3. Auto-create Deal in BUYER org
await this.tenantContext.executeInOrg(buyerOrgId, async (em) => {
  const deal = em.create(Deal, { ... });
  const stakeholder = em.create(EntityStakeholder, { ... });
});
```
</Step>

<Step title="Link deals to SharedDeal">
```typescript
// 4. Link both deals to SharedDeal
sharedDeal.listingDeal = listingDeal;
sharedDeal.buyerDeal = buyerDeal;
```
</Step>
</Steps>

## RLS for cross-org entities

Custom RLS policies enable cross-organization visibility while maintaining security:

```sql
-- SharedListing: readable by all, writable by owning org
CREATE POLICY "shared_listing_read" ON "shared_listing" FOR SELECT USING (true);
CREATE POLICY "shared_listing_write" ON "shared_listing" FOR ALL USING (
  listed_by_organization_id = current_setting('app.organization_id')::uuid
);

-- SharedDeal: visible to both participating orgs
CREATE POLICY "shared_deal_access" ON "shared_deal" FOR ALL USING (
  listing_organization_id = current_setting('app.organization_id')::uuid
  OR buyer_organization_id = current_setting('app.organization_id')::uuid
);

-- SharedDealMessage: inherits access from parent SharedDeal
CREATE POLICY "shared_deal_message_access" ON "shared_deal_message" FOR ALL USING (
  EXISTS (
    SELECT 1 FROM shared_deal sd 
    WHERE sd.id = shared_deal_id 
    AND (
      sd.listing_organization_id = current_setting('app.organization_id')::uuid
      OR sd.buyer_organization_id = current_setting('app.organization_id')::uuid
    )
  )
);

-- UnitAccessRequest: org-scoped (standard RLS)
CREATE POLICY "unit_access_request_policy" ON "unit_access_request" FOR ALL USING (
  organization_id = current_setting('app.organization_id')::uuid
);

-- UnitAccessGrant: org-scoped (standard RLS)
CREATE POLICY "unit_access_grant_policy" ON "unit_access_grant" FOR ALL USING (
  organization_id = current_setting('app.organization_id')::uuid
);
```

## ReferenceDataService dependency

The shared marketplace consumes property data (projects, units, communities, areas) through `ReferenceDataService`, which abstracts access to the external reference database:

```typescript
// Example: Enriching SharedListing with reference data
const projectData = await this.referenceDataService.getProject(sharedListing.refProjectId);
const unitData = await this.referenceDataService.getUnit(sharedListing.refUnitId);

// Service injection pattern for reference context
@Injectable()
export class SharedMarketplaceService {
  constructor(
    private readonly referenceDataService: ReferenceDataService,
    @InjectEntityManager()
    private readonly em: EntityManager,
  ) {}
}
```

<Tip>
If the reference database is swapped for an external property API, only `ReferenceDataService` needs to change — the shared marketplace module remains unchanged.
</Tip>

## Response DTOs

Constructor-based DTOs handle data serialization:

- `SharedListingDto` - Basic listing information with unit number protection
- `SharedDealDto` - Deal summary information
- `SharedDealDetailDto` - Extends `SharedDealDto` with full message history
- `SharedDealMessageDto` - Individual message serialization
- `RequestUnitAccessDto` - Input DTO for unit access requests with class-validator decorators
- `RespondUnitAccessDto` - Input DTO for access request responses with class-validator decorators

## Access control

<Tabs>
<Tab title="Cross-org endpoints">
**Marketplace listings** (`SharedListingController`): `AuthGuard` only — **no AccessGuard** (globally shared cross-org data)

**Shared deals** (`SharedDealController`): `AuthGuard` only — **no AccessGuard** (cross-org negotiation)
</Tab>
<Tab title="Org-scoped endpoints">

</Tab>
</Tabs>
