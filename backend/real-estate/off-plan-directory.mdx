The real estate module is structured as **six active specialized modules** spanning two visibility scopes: **global** (no RLS) and **org-private** (standard RLS). The cross-org `shared-marketplace` code is retained in the repository but is not imported by `RealEstateModule`, so its controllers/providers are not mounted by the app. The external **Propwise Labs API** stores and serves all property data — both structural (DLD registry) and enrichment (images, payment plans, documents, amenities, pricing, geocoding where exposed). The CRM database stores only CRM-specific data (developer portal auth, ownership links, property interests, marketplace listings, org-scoped inventory unit claims).

<Note>
All modules consume property data exclusively through `PropwiseLabsService`. The CRM no longer opens a second MikroORM connection or maps Propwise Labs entities locally. The reference database configuration has been completely removed and replaced with Propwise Labs API configuration (`app.propwiseLabs.*`) including API credentials, timeout settings, and Bird Verify integration for developer portal contact verification.
</Note>

## Data ownership

The external API architecture separates property data from CRM data:

| Data Type | Source | Access Pattern |
| --- | --- | --- |
| Projects, units, buildings, areas, communities, developers | Propwise Labs API | `PropwiseLabsService` |
| Images, payment plans, documents, amenities | Propwise Labs API | `PropwiseLabsService` |
| Property data mutations | Propwise Labs API | Change requests for create/delete/approval flows; direct wrappers for selected existing-resource edits |
| Developer auth/profile (DeveloperUser) | CRM DB | Direct entity access |
| Off-plan building publication state | CRM DB | `OffPlanBuildingPublication` stores app-owned building visibility by Propwise Labs building ID |
| Inventory units (org-scoped inventory) | CRM DB | `InventoryUnit` stores org-specific claim data and Propwise Labs refs |
| Marketplace listings & deals (SharedListing, SharedDeal) | CRM DB | Direct entity access; module code retained but not mounted in the app |
| Property interests (Lead/Deal) | CRM DB | Lead search criteria and deal concrete property links |
| Property matches | CRM DB | Direct entity access, matched InventoryUnit IDs |
| Portal listings/configuration/sync | CRM DB | Direct entity access through `real-estate/portal-syndication` |

## External property ID pattern

`InventoryUnit` is the CRM boundary for org-owned secondary inventory. It stores plain integer IDs (`refProjectId`, `refBuildingId`, `refUnitId`, `refDeveloperId`, `refPropertyTypeId`, `refPropertySubTypeId`) plus a denormalized Labs snapshot, including the developer display name used by portal syndication and the canonical `propertyType` (`LayoutType` enum) used by portal syndication adapters. Labs raw property type/sub-type names are retained in `labsPropertyType` and `labsPropertySubType` for traceability. Lead interests can use inventory units as candidate matches while retaining search criteria. Deal property rows are concrete-only: they link either an org `InventoryUnit`, a direct Propwise Labs off-plan unit via `refProjectId` / `refBuildingId` / `refUnitId`, or a pending unit-access request before the inventory link is approved.

```typescript
// CRM entity referencing Propwise Labs data:
@Property({ type: 'int', nullable: true })
@Index()
refProjectId?: number;  // ID in Propwise Labs projects

@Property({ type: 'int', nullable: true })
@Index()
refUnitId?: number;     // ID in Propwise Labs units
```

This pattern decouples the CRM from the property data source and enables flexible data architecture. The references are stored as plain integers with indexes for efficient queries. When CRM entities need property data, they call `PropwiseLabsService` with these IDs to fetch live data from the external API. This approach ensures the CRM remains functional even if the property data source changes, and eliminates the need for complex entity mapping or synchronization between systems.

## Propwise Labs module

**Path:** `src/modules/real-estate/propwise-labs/`

### Purpose

Provides the CRM backend facade over the external Propwise Labs API. It handles API authentication, bearer-token caching, read calls for property data, lookup data, change-request submissions for developer portal mutations, and typed wrappers for super-level direct Labs mutations. Local `Ref*` MikroORM entities and the named `reference` EntityManager have been removed.

### API client pattern

`PropwiseLabsService` reads API credentials from `app.propwiseLabs.*`, exchanges `PROPWISE_LABS_API_KEY` and `PROPWISE_LABS_API_SECRET` for a short-lived bearer token, and reuses that token until it is close to expiry. The service includes configurable timeout settings and comprehensive error handling for external API dependencies.

```typescript
@Injectable()
export class PropwiseLabsService {
  private accessToken?: string;
  private tokenExpiresAt = 0;

  async searchProjects(filters: SearchProjectFilters): Promise<PropwiseProject[]> {
    const token = await this.getAccessToken();
    const response = await this.httpService.axiosRef.get(
      `${this.baseUrl}/projects`,
      {
        params: filters,
        headers: { Authorization: `Bearer ${token}` },
        timeout: this.config.get('app.propwiseLabs.timeout'),
      }
    );
    return response.data.projects;
  }

  private async getAccessToken(): Promise<string> {
    if (this.accessToken && Date.now() < this.tokenExpiresAt - 30000) {
      return this.accessToken;
    }

    const response = await this.httpService.axiosRef.post(
      `${this.baseUrl}/auth/token`,
      {
        apiKey: this.config.get('app.propwiseLabs.apiKey'),
        apiSecret: this.config.get('app.propwiseLabs.apiSecret'),
      }
    );

    this.accessToken = response.data.accessToken;
    this.tokenExpiresAt = Date.now() + response.data.expiresIn * 1000;
    return this.accessToken;
  }
}
```

## Off-plan module

**Path:** `src/modules/real-estate/off-plan/`

### Purpose

Provides the main CRM off-plan directory API that applies publication visibility filtering and developer contact enrichment on top of Propwise Labs data. This module determines which buildings are visible in the public off-plan directory through the `OffPlanBuildingPublication` CRM entity.

### Publication state management

The off-plan module manages building visibility through a CRM-owned entity that references Propwise Labs building IDs:

```typescript
@Entity({ tableName: 'off_plan_building_publications' })
export class OffPlanBuildingPublication {
  @PrimaryKey({ type: 'uuid' })
  id = uuid4();

  @Property({ type: 'int' })
  @Index()
  propwiseLabsBuildingId!: number;  // Reference to Propwise Labs building

  @Property({ default: false })
  isPublished = false;

  @Property({ nullable: true })
  publishedAt?: Date;

  @Property({ nullable: true })
  unpublishedAt?: Date;

  @Property()
  createdAt = new Date();

  @Property({ onUpdate: () => new Date() })
  updatedAt = new Date();
}
```

### OffPlanService

The service layer combines Propwise Labs data with CRM-owned publication states:

```typescript
@Injectable()
export class OffPlanService {
  constructor(
    private readonly propwiseLabsService: PropwiseLabsService,
    @InjectEntityManager()
    private readonly em: EntityManager,
  ) {}

  async getPublishedProjects(filters: OffPlanFilters): Promise<OffPlanProjectDto[]> {
    // Get published building IDs from CRM
    const publishedBuildings = await this.em.find(OffPlanBuildingPublication, {
      isPublished: true,
    });
    
    const buildingIds = publishedBuildings.map(pb => pb.propwiseLabsBuildingId);
    
    // Filter Propwise Labs projects to only include projects with published buildings
    const projects = await this.propwiseLabsService.searchProjects({
      ...filters,
      buildingIds,
    });
    
    return projects.map(project => this.enrichWithContacts(project));
  }

  private enrichWithContacts(project: PropwiseProject): OffPlanProjectDto {
    // Add developer contact information and CRM-specific enrichment
    return {
      ...project,
      developerContacts: this.getDeveloperContacts(project.developerId),
      crmMetadata: this.getCrmMetadata(project.id),
    };
  }
}
```

### Off-plan filtering logic

"Off-plan" status is determined by business logic applied to Propwise Labs data:

```typescript
private applyOffPlanFilter(filters: ProjectFilters): ProjectFilters {
  return {
    ...filters,
    // Project must not have ended
    endDateFilter: {
      $or: [
        { $gt: new Date() },  // Future end date
        { $null: true },      // No end date set
      ],
    },
    // Project must not be completed
    completionDate: { $null: true },
    // Project must be in active development phases
    status: {
      $in: ['active', 'pending', 'under_construction', 'announced']
    },
  };
}
```

<Info>
The off-plan module applies both Propwise Labs off-plan business logic AND CRM publication visibility. Only buildings explicitly published through `OffPlanBuildingPublication` appear in the public directory.
</Info>

## Map integration (2GIS)

**MapService** (`src/modules/real-estate/map/map.service.ts`) provides shared geocoding utility across modules:

- **API:** `https://catalog.api.2gis.com/3.0/items/geocode`
- **Environment:** `DGIS_API_KEY`
- **Frontend:** 2GIS MapGL JS API (docs: https://docs.urbi.ae)
- **Map data:** Fetched via `PropwiseLabsService.getMapProjects()`

<Steps>
  <Step title="Developer triggers geocoding">
    POST request to `/developer-portal/projects/:id/geocode`
  </Step>
  <Step title="MapService queries 2GIS API">
    Project address converted to lat/lng coordinates using 2GIS catalog
  </Step>
  <Step title="Coordinates submitted">
    Results sent to Propwise Labs API via change request or direct edit
  </Step>
  <Step title="Map data served">
    Coordinates included in PropwiseLabsService map endpoints for frontend map rendering
  </Step>
</Steps>

```typescript
@Injectable()
export class MapService {
  constructor(private readonly httpService: HttpService) {}

  async geocodeAddress(address: string): Promise<{ lat: number; lng: number }> {
    const response = await this.httpService.axiosRef.get(
      'https://catalog.api.2gis.com/3.0/items/geocode',
      {
        params: {
          key: process.env.DGIS_API_KEY,
          q: address,
          fields: 'items.point',
        },
      }
    );

    const items = response.data?.result?.items;
    if (!items?.length) {
      throw new Error('Address not found');
    }

    const point = items[0].point;
    return {
      lat: point.lat,
      lng: point.lon,
    };
  }
}
```

### Error handling patterns

```typescript
// Propwise Labs API not available doesn't break CRM operations
try {
  const project = await this.propwiseLabsService.findProject(refProjectId);
  return { ...crmData, project };
} catch (error) {
  this.logger.error('Propwise Labs API error:', error);
  // CRM entity remains valid, external data handled gracefully
  return { ...crmData, project: { id: refProjectId, name: 'External Project' } };
```
