The real estate module is structured as **six active specialized modules** spanning two visibility scopes: **global** (no RLS) and **org-private** (standard RLS). The cross-org `shared-marketplace` code is retained in the repository but is not imported by `RealEstateModule`, so its controllers/providers are not mounted by the app. The external **Propwise Labs API** stores and serves all property data — both structural (DLD registry) and enrichment (images, payment plans, documents, amenities, pricing, geocoding where exposed). The CRM database stores only CRM-specific data (developer portal auth, ownership links, property interests, marketplace listings, org-scoped inventory unit claims).

<Note>
All modules consume property data exclusively through `PropwiseLabsService`. The CRM no longer opens a second MikroORM connection or maps Propwise Labs entities locally. The reference database configuration has been completely removed and replaced with Propwise Labs API configuration (`app.propwiseLabs.*`) including API credentials, timeout settings, and Bird Verify integration for developer portal contact verification.
</Note>

<Warning>
**Portal syndication — self-contained `Listing` (2026 build):** The `portal-syndication` module's `Listing` is now self-contained: every field any portal (Bayut/dubizzle + Property Finder) needs lives on `Listing`, snapshotted from an `InventoryUnit` at create (linked mode) or entered directly (manual mode). The `inventoryUnit` FK is **optional**. Deleting an inventory unit cascades (event-driven, `inventory-unit.deleted` carrying the user's `removeLinkedListings` choice): **remove** (default) delists the unit's listings from all portals and archives them; **keep** unlinks them (`inventoryUnit = null`) so they survive as self-contained manual listings. Buildings auto-maintain `salesStatus` based on available units (`OUT_OF_STOCK` when no units remain `AVAILABLE`, reverting to `ON_SALE` when available units reappear). New `downpaymentType` enum distinguishes amount vs percentage downpayments.
</Warning>

## Data ownership

The external API architecture separates property data from CRM data:

| Data Type | Source | Access Pattern |
| --- | --- | --- |
| Projects, units, buildings, areas, communities, developers | Propwise Labs API | `PropwiseLabsService` |
| Images, payment plans, documents, amenities | Propwise Labs API | `PropwiseLabsService` |
| Property data mutations | Propwise Labs API | Change requests for create/delete/approval flows; direct wrappers for selected existing-resource edits |
| News articles, sources, insights | CRM DB + AI Services | `NewsIngestionService`, `NewsSearchService` via semantic search |
| Developer auth/profile (DeveloperUser) | CRM DB | Direct entity access |
| Off-plan building publication state | CRM DB | `OffPlanBuildingPublication` stores app-owned building visibility by Propwise Labs building ID |
| Off-plan villa-project publication state | CRM DB | `OffPlanProjectPublication` stores app-owned project visibility for Villas-type projects (no buildings) |
| Inventory units (org-scoped inventory) | CRM DB | `InventoryUnit` stores org-specific claim data and Propwise Labs refs |
| Marketplace listings & deals (SharedListing, SharedDeal) | CRM DB | Direct entity access; module code retained but not mounted in the app |
| Property interests (Lead/Deal) | CRM DB | Lead search criteria and deal concrete property links |
| Property matches | CRM DB | Direct entity access, matched InventoryUnit IDs |
| Portal listings/configuration/sync | CRM DB | Direct entity access through `real-estate/portal-syndication` |

## Listing approval workflow (Gate A)

The portal syndication module implements a comprehensive listing approval workflow for organizations with proper permissions:

<Steps>
  <Step title="Listing submission">
    A user without `real_estate.listing.publish` or `real_estate.manage` permissions submits a listing for approval (`ListingStatus.PENDING_APPROVAL`)
  </Step>
  <Step title="Approval notification">
    System emits `LISTING_APPROVAL_REQUESTED` notification to all users with `real_estate.manage` permission
  </Step>
  <Step title="Manager review">
    A `real_estate.manage` user reviews the listing from the **Requests** queue (`GET /portal-syndication/listings?type=requests`) containing pending and rejected listings
  </Step>
  <Step title="Approval or rejection">
    Manager can approve (auto-publishes to submitter's selected portals) or reject (`ListingStatus.REJECTED` with `rejectionReason` persisted)
  </Step>
</Steps>

### Delete action behavior

The listing **Delete** action performs a soft-delete by setting `isDeleted=true`. There is no user-facing archived state (the `archived` status and `isArchived` columns remain only for the inventory-unit delete cascade). A `real_estate.manage` user may delete **any** listing, including pending requests and rejected ones, directly from the Requests queue.

### Image optimization workflow

The storage service applies WebP optimization automatically:

<Steps>
  <Step title="Profile validation">
    System validates the `profile` parameter against `STORAGE_UPLOAD_PROFILES` (unknown profiles return `400`)
  </Step>
  <Step title="Image optimization">
    `ImageOptimizationService` processes raster images to WebP format; PDF/doc/video files pass through unchanged
  </Step>
  <Step title="Storage upload">
    Optimized file is written to R2 under the profile-specific folder (`listings/` or `listings/floor-plans/`)
  </Step>
  <Step title="Response metadata">
    Endpoint returns converted `fileName` (`.webp`), `mimeType`, and `fileSize` reflecting the optimization
  </Step>
</Steps>

<Note>
The default profile is `GENERIC_ATTACHMENT` for email/workflow/deal-document/TipTap uploads to maintain lean file sizes. Listing-specific profiles (`LISTING` and `FLOOR_PLAN`) provide higher resolution for Property Finder syndication requirements.
</Note>

### Property Finder compatibility

`ListingImageService` handles Property Finder-specific requirements:

```typescript
@Injectable()
export class ListingImageService {
  async ensureCompliant(imageUrl: string): Promise<ComplianceResult> {
    const image = await this.fetchImage(imageUrl);
    
    // Check if re-processing needed for Property Finder limits
    if (image.width > 1920 || image.height > 1080 || !this.isSupportedFormat(image)) {
      // Single PF-specific re-encode for legacy images
      return this.reprocessForPropertyFinder(image);
    }
    
    return { compliant: true, url: imageUrl };
  }
}
```

<Warning>
`ListingImageService` remains unchanged from previous versions — its `compliant` check re-processes source images only when they exceed Property Finder's 1920×1080 format limits. This represents an expected single PF-specific re-encode, not an ingest regression.
</Warning>

## Propwise Labs module

**Path:** `src/modules/real-estate/propwise-labs/`

Provides the CRM backend facade over the external Propwise Labs API. It handles API authentication, bearer-token caching, read calls for property data, lookup data, change-request submissions for developer portal mutations, and typed wrappers for super-level direct Labs mutations.

### API client pattern

`PropwiseLabsService` reads API credentials from `app.propwiseLabs.*`, exchanges `PROPWISE_LABS_API_KEY` and `PROPWISE_LABS_API_SECRET` for a short-lived bearer token, and reuses that token until it is close to expiry:

```typescript
@Injectable()
export class PropwiseLabsService {
  private accessToken?: string;
  private tokenExpiresAt = 0;

  async searchProjects(filters: SearchProjectFilters): Promise<PropwiseProject[]> {
    const token = await this.getAccessToken();
    const response = await this.httpService.axiosRef.get(
      `${this.baseUrl}/projects`,
      {
        params: filters,
        headers: { Authorization: `Bearer ${token}` },
        timeout: this.config.get('app.propwiseLabs.timeout'),
      }
    );
    return response.data.projects;
  }

  private async getAccessToken(): Promise<string> {
    if (this.accessToken && Date.now() < this.tokenExpiresAt - 30000) {
      return this.accessToken;
    }

    const response = await this.httpService.axiosRef.post(
      `${this.baseUrl}/auth/token`,
      {
        apiKey: this.config.get('app.propwiseLabs.apiKey'),
        apiSecret: this.config.get('app.propwiseLabs.apiSecret'),
      }
    );

    this.accessToken = response.data.accessToken;
    this.tokenExpiresAt = Date.now() + response.data.expiresIn * 1000;
    return this.accessToken;
  }
}
```

```typescript
@Injectable()
export class OffPlanService {
  async getPublishedProjects(filters: OffPlanFilters): Promise<OffPlanProjectDto[]> {
    // Get published building IDs from CRM
    const publishedBuildings = await this.em.find(OffPlanBuildingPublication, {
      isPublished: true,
    });
    
    const buildingIds = publishedBuildings.map(pb => pb.propwiseLabsBuildingId);
    
    // Filter Propwise Labs projects to only include projects with published buildings
    const projects = await this.propwiseLabsService.searchProjects({
      ...filters,
      buildingIds,

    return projects.map(project => this.enrichWithContacts(project));

    // Add developer contact information and CRM-specific enrichment
    return {
      ...project,
      developerContacts: this.getDeveloperContacts(project.developerId),
      crmMetadata: this.getCrmMetadata(project.id),
    };
  }
}
```

## Map integration (2GIS)

**MapService** (`src/modules/real-estate/map/map.service.ts`) provides shared geocoding utility across modules:

```typescript
@Injectable()
export class MapService {
  async geocodeAddress(address: string): Promise<{ lat: number; lng: number }> {
    const response = await this.httpService.axiosRef.get(
      'https://catalog.api.2gis.com/3.0/items/geocode',
      {
        params: {
          key: process.env.DGIS_API_KEY,
          q: address,
          fields: 'items.point',
        },
      }
    );

    const items = response.data?.result?.items;
    if (!items?.length) {
      throw new Error('Address not found');
    }

    const point = items[0].point;
    return {
      lat: point.lat,
      lng: point.lon,
    };
  }
}
```

## External property ID pattern

`InventoryUnit` is the CRM boundary for org-owned secondary inventory. It stores plain integer IDs (`refProjectId`, `refBuildingId`, `refUnitId`, `refDeveloperId`, `refPropertyTypeId`, `refPropertySubTypeId`) plus a denormalized Labs snapshot, including the developer display name used by portal syndication and the canonical `propertyType` (`LayoutType` enum) used by portal syndication adapters:

```typescript
// CRM entity referencing Propwise Labs data:
@Property({ type: 'int', nullable: true })
@Index()
refProjectId?: number;  // ID in Propwise Labs projects

@Property({ type: 'int', nullable: true })
@Index()
refUnitId?: number;     // ID in Propwise Labs units
```

This pattern decouples the CRM from the property data source and enables flexible data architecture. The references are stored as plain integers with indexes for efficient queries.
