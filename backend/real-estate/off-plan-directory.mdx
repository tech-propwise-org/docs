The reference data module connects to a second PostgreSQL database containing **all property data**: DLD registry (structural) plus enrichment (gallery, payment plans, documents, amenities, pricing, geocoding). It provides the foundation for the off-plan directory and developer portal operations.

## Architecture overview

The real estate module spans **six specialized modules** across three visibility scopes:

```
src/modules/real-estate/
├── reference-data/        # Second DB — ALL property data
│   ├── reference-data.controller.ts       # /reference — areas, cities, communities
│   ├── reference-projects.controller.ts   # /reference/projects — search, map, detail
│   └── reference-buildings.controller.ts  # /reference/buildings — search, detail
├── developer-portal/      # Auth + project management
├── map/                   # MapService (2GIS geocoding utility)
├── inventory/             # Org-scoped secondary units
├── shared-marketplace/    # Cross-org listings + deals
└── property-interest/     # Property interests + matching engine
```

<Note>
All modules consume property data exclusively through `ReferenceDataService`. If the reference database is swapped for an external API, only this service needs to change.
</Note>

## Data ownership

The dual-database architecture separates property data from CRM data:

| Data Type | Database | Access Pattern |
| --- | --- | --- |
| Projects, units, buildings, areas, communities | Reference DB | `ReferenceDataService` |
| Gallery, payment plans, documents, amenities | Reference DB | `ReferenceDataService` |
| Geocoding, pricing, sale status, handover | Reference DB | `ReferenceDataService` |
| Developer auth, project ownership links | CRM DB | Direct entity access |
| Secondary units, marketplace listings | CRM DB | Direct entity access |
| Property interests, matches | CRM DB | Plain int references |

## Cross-database reference pattern

CRM entities reference property data using **plain integer IDs** without foreign key constraints:

```typescript
// CRM entity referencing reference DB data
@Property({ type: 'int', nullable: true })
@Index()
refProjectId?: number;  // ID in reference DB projects table

@Property({ type: 'int', nullable: true })
@Index()
refUnitId?: number;     // ID in reference DB units table
```

This pattern decouples the CRM from the property data source and enables flexible data architecture.

## Database configuration

### Second database connection

Configured in `mikro-orm-reference.config.ts` with named context:

```typescript
// app.module.ts
MikroOrmModule.forRoot({
  ...mikroOrmReferenceConfig,
  contextName: 'reference',
});
```

### Entity pattern

Reference entities map to external DB tables with specific characteristics:

```typescript
@Entity({ tableName: 'projects' })
export class RefProject {
  @PrimaryKey()
  id!: number;  // Number PK matching external DB

  @Property({ nullable: true })
  name?: string;

  @ManyToOne(() => RefDeveloper, { fieldName: 'developer_id' })
  developer?: RefDeveloper;

  // Enrichment fields with soft delete
  @Property({ type: 'timestamp', nullable: true })
  deletedAt?: Date;

  @Property({ fieldName: 'is_published', default: false })
  isPublished?: boolean;

  // Collections
  @OneToMany(() => RefProjectGalleryImage, (img) => img.project)
  gallery = new Collection<RefProjectGalleryImage>(this);
}
```

### Service injection

Use named EntityManager for reference context:

```typescript
@Injectable()
export class ReferenceDataService {
  constructor(
    @InjectEntityManager('reference')
    private readonly em: EntityManager,
  ) {}
}
```

## Controllers

The module provides three specialized controllers:

### General reference controller

**Path:** `/reference`

Provides foundational reference data for dropdowns and filters:

```
GET /reference/areas           # Geographic areas
GET /reference/cities          # Cities list
GET /reference/communities     # Communities within areas
GET /reference/developers      # Developer profiles
GET /reference/property-types  # Property type categories
GET /reference/units/:id       # Single unit detail
```

### Projects controller

**Path:** `/reference/projects`

Core off-plan directory functionality:

```
GET /reference/projects                    # Search with filters
GET /reference/projects?type=off-plan      # Off-plan filter applied
GET /reference/projects/map                # Map markers with coordinates
GET /reference/projects/counts             # Aggregated counts by status
GET /reference/projects/:id                # Enriched project detail
GET /reference/projects/:id/buildings      # Project's buildings
GET /reference/projects/:id/units/grouped  # Units grouped by bedrooms
```

### Buildings controller

**Path:** `/reference/buildings`

Building-focused queries for detailed searches:

```
GET /reference/buildings                   # Search buildings
GET /reference/buildings?type=off-plan     # Off-plan buildings only
GET /reference/buildings/:id               # Building detail
GET /reference/buildings/:id/units         # Building's units
```

## Off-plan filtering logic

"Off-plan" is a **derived filter** applied to reference data queries, not a separate module:

```typescript
private applyOffPlanFilter(qb: QueryBuilder<RefProject | RefBuilding>) {
  qb.andWhere({
    $or: [
      { endDate: { $gt: new Date() } },
      { endDate: null },
    ],
  });
  qb.andWhere({ completionDate: null });
  qb.andWhere({
    status: { $in: ['active', 'pending', 'under_construction', 'announced'] },
  });
  qb.andWhere({ isPublished: true });
}
```

<Info>
A project/building is "off-plan" when: end date is future/null, completion date is null, status is active/pending/under_construction/announced, and is published.
</Info>

## Map integration (2GIS)

**MapService** (`src/modules/real-estate/map/map.service.ts`) provides shared geocoding utility:

- **API:** `https://catalog.api.2gis.com/3.0/items/geocode`
- **Environment:** `DGIS_API_KEY`
- **Frontend:** 2GIS MapGL JS API (docs: https://docs.urbi.ae)
- **Map data:** Fetched via `ReferenceDataService.getMapProjects()`

<Steps>
  <Step title="Developer triggers geocoding">
    POST request to `/developer-portal/projects/:id/geocode`
  </Step>
  <Step title="MapService queries 2GIS API">
    Address converted to lat/lng coordinates
  </Step>
  <Step title="Coordinates stored">
    Results saved to `RefProject` entity in reference database
  </Step>
</Steps>

## Developer portal integration

The developer portal writes to the reference database through `ReferenceDataService`:

- **Authentication:** Separate JWT system for developers
- **Project management:** CRUD operations via reference DB
- **Building management:** Nested under projects
- **Unit management:** Nested under buildings
- **Media uploads:** Gallery images, documents, payment plans

<Warning>
Developer portal operations bypass standard RLS since they operate on global property data. Access control is handled through separate developer authentication.
</Warning>
