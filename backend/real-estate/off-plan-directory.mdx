The real estate module is structured as **six specialized modules** spanning three visibility scopes: **global** (no RLS), **org-private** (standard RLS), and **cross-org** (custom RLS). An external **reference database** (Neon "propwise-lab") stores **all property data** — both structural (DLD registry) and enrichment (gallery, payment plans, documents, amenities, pricing, geocoding). The CRM database stores only CRM-specific data (developer portal auth, ownership links, property interests, marketplace listings, secondary unit context).

<Note>
All modules consume property data exclusively through `ReferenceDataService`. If the reference database is swapped for an external property API, only `ReferenceDataService` needs to change.
</Note>

## Data ownership

The dual-database architecture separates property data from CRM data:

| Data Type | Database | Access Pattern |
| --- | --- | --- |
| Projects, units, buildings, areas, communities, developers | Reference DB | `ReferenceDataService` |
| Gallery images, payment plans, documents, amenities | Reference DB | `ReferenceDataService` |
| Geocoding (lat/lng), pricing, sale status, handover | Reference DB | `ReferenceDataService` |
| Developer auth (DeveloperUser) | CRM DB | Direct entity access |
| Developer-to-project ownership (DeveloperProjectLink) | CRM DB | Direct entity access |
| Secondary units (org-scoped inventory) | CRM DB | Direct entity access |
| Marketplace listings & deals (SharedListing, SharedDeal) | CRM DB | Direct entity access |
| Property interests (Lead/Deal) | CRM DB | Direct entity access, ref IDs as plain ints |
| Property matches | CRM DB | Direct entity access, ref IDs as plain ints |

## Cross-database reference pattern

CRM entities reference property data using **plain integer IDs** without foreign key constraints:

```typescript
// CRM entity referencing reference DB data
@Property({ type: 'int', nullable: true })
@Index()
refProjectId?: number;  // ID in reference DB projects table

@Property({ type: 'int', nullable: true })
@Index()
refUnitId?: number;     // ID in reference DB units table
```

This pattern decouples the CRM from the property data source and enables flexible data architecture. No cross-database foreign keys exist — this allows the reference database to be swapped for external APIs without breaking CRM functionality.

## Reference data module

**Path:** `src/modules/real-estate/reference-data/`

### Purpose

Connects to a second PostgreSQL database (Neon "propwise-lab") containing **all property data**: DLD registry data (structural) plus enrichment data (gallery, payment plans, documents, amenities, sales contacts, pricing, geocoding). Provides read endpoints for the off-plan directory and write endpoints for the developer portal.

### Second database connection

Configured in `mikro-orm-reference.config.ts` at project root. Registered in `app.module.ts` with `contextName: 'reference'`:

```typescript
MikroOrmModule.forRoot({
  ...mikroOrmReferenceConfig,
  contextName: 'reference',
});
```

The reference database uses a separate connection pool and configuration from the main CRM database, enabling independent scaling and management of property data infrastructure.

### Entity pattern

All reference entities are mapped to the external DB tables. They have **no UUID primary keys** (use `number` PKs matching the external DB), **no organization FK**, and use **soft delete** (`deletedAt` timestamp) for enrichment entities:

```typescript
@Entity({ tableName: 'projects' })
export class RefProject {
  @PrimaryKey()
  id!: number;

  @Property({ nullable: true })
  name?: string;

  @ManyToOne(() => RefDeveloper, { fieldName: 'developer_id', nullable: true })
  developer?: RefDeveloper;

  @ManyToOne(() => RefArea, { fieldName: 'area_id' })
  area!: RefArea;

  // Enrichment fields
  @Property({ type: 'double', nullable: true })
  latitude?: number;

  @Property({ fieldName: 'price_from', type: 'numeric', nullable: true })
  priceFrom?: number;

  @Property({ fieldName: 'is_published', default: false, nullable: true })
  isPublished?: boolean;

  // Soft delete for enrichment data
  @Property({ fieldName: 'deleted_at', nullable: true })
  deletedAt?: Date;

  // Enrichment collections
  @OneToMany(() => RefProjectGalleryImage, (img) => img.project)
  gallery = new Collection<RefProjectGalleryImage>(this);

  @OneToMany(() => RefPaymentPlan, (p) => p.project)
  paymentPlans = new Collection<RefPaymentPlan>(this);

  @OneToMany(() => RefProjectDocument, (d) => d.project)
  documents = new Collection<RefProjectDocument>(this);

  @OneToMany(() => RefProjectAmenity, (a) => a.project)
  amenities = new Collection<RefProjectAmenity>(this);
}
```

### Service injection pattern

Use the named EntityManager for the reference context:

```typescript
@Injectable()
export class ReferenceDataService {
  constructor(
    @InjectEntityManager('reference')
    private readonly em: EntityManager,
  ) {}

  async findProjects(filters: ProjectSearchFilters): Promise<RefProject[]> {
    const qb = this.em.createQueryBuilder(RefProject, 'p')
      .leftJoinAndSelect('p.developer', 'd')
      .leftJoinAndSelect('p.area', 'a')
      .where({ deletedAt: null });

    if (filters.type === 'off-plan') {
      this.applyOffPlanFilter(qb);
    }

    return qb.getResult();
  }

  private applyOffPlanFilter(qb: QueryBuilder<RefProject>) {
    qb.andWhere({
      $or: [
        { endDate: { $gt: new Date() } },
        { endDate: null },
      ],
    })
    .andWhere({ completionDate: null })
    .andWhere({
      status: { $in: ['active', 'pending', 'under_construction', 'announced'] },
    })
    .andWhere({ isPublished: true });
  }
}
```

### Database schema structure

The reference database contains both **structural data** (DLD registry) and **enrichment data** (CRM-managed content):

#### Core structural entities

- **RefArea** - Geographic areas and neighborhoods
- **RefCity** - Cities and emirate information  
- **RefCommunity** - Community subdivisions within areas
- **RefDeveloper** - Developer profiles and contact information
- **RefPropertyType** - Property type classifications
- **RefProject** - Master project records
- **RefBuilding** - Building records within projects
- **RefUnit** - Individual unit records

#### Enrichment entities

- **RefProjectGalleryImage** - Project media galleries
- **RefPaymentPlan** - Payment plan options
- **RefProjectDocument** - Project documentation
- **RefProjectAmenity** - Project amenities and features
- **RefSalesContact** - Sales team contact information

All enrichment entities use soft delete (`deletedAt` timestamp) to enable safe content management without losing historical data.

### Service layer implementation

The `ReferenceDataService` provides a unified interface for all reference data operations:

```typescript
@Injectable()
export class ReferenceDataService {
  constructor(
    @InjectEntityManager('reference')
    private readonly em: EntityManager,
    private readonly mapService: MapService,
  ) {}

  // Core query methods
  async findProjects(filters: ProjectSearchFilters): Promise<RefProject[]> {
    const qb = this.em.createQueryBuilder(RefProject, 'p');
    
    if (filters.type === 'off-plan') {
      this.applyOffPlanFilter(qb);
    }
    
    // Apply additional filters...
    return qb.getResult();
  }

  // Off-plan business logic
  private applyOffPlanFilter(qb: QueryBuilder<RefProject>) {
    qb.andWhere({
      $or: [
        { endDate: { $gt: new Date() } },
        { endDate: null },
      ],
    })
    .andWhere({ completionDate: null })
    .andWhere({
      status: { $in: ['active', 'pending', 'under_construction', 'announced'] },
    })
    .andWhere({ isPublished: true });
  }

  // Geocoding integration
  async geocodeProject(projectId: number): Promise<void> {
    const project = await this.findProjectById(projectId);
    const coordinates = await this.mapService.geocodeAddress(project.address);
    
    project.latitude = coordinates.lat;
    project.longitude = coordinates.lng;
    await this.em.flush();
  }

  // Enrichment management
  async updateProjectGallery(projectId: number, images: GalleryImageDto[]): Promise<void> {
    // Soft delete existing images
    await this.em.nativeUpdate(
      RefProjectGalleryImage,
      { project: projectId },
      { deletedAt: new Date() }
    );

    // Create new images
    const newImages = images.map(img => 
      this.em.create(RefProjectGalleryImage, { ...img, project: projectId })
    );
    
    await this.em.flush();
  }
}
```

## Off-plan filtering logic

"Off-plan" is a **derived filter** applied to reference data queries, not a separate module. The business logic determines off-plan status based on multiple criteria:

```typescript
private applyOffPlanFilter(qb: QueryBuilder<RefProject | RefBuilding>) {
  // Project must not have ended
  qb.andWhere({
    $or: [
      { endDate: { $gt: new Date() } },  // Future end date
      { endDate: null },                 // No end date set
    ],
  });
  
  // Project must not be completed
  qb.andWhere({ completionDate: null });
  
  // Project must be in active development phases
  qb.andWhere({
    status: { 
      $in: ['active', 'pending', 'under_construction', 'announced'] 
    },
  });
  
  // Project must be published for public viewing
  qb.andWhere({ isPublished: true });
}
```

<Info>
A project/building is "off-plan" when: end date is future/null, completion date is null, status is active/pending/under_construction/announced, and is published. This filter is applied dynamically across all relevant endpoints.
</Info>

## Map integration (2GIS)

**MapService** (`src/modules/real-estate/map/map.service.ts`) provides shared geocoding utility across modules:

- **API:** `https://catalog.api.2gis.com/3.0/items/geocode`
- **Environment:** `DGIS_API_KEY`
- **Frontend:** 2GIS MapGL JS API (docs: https://docs.urbi.ae)
- **Map data:** Fetched via `ReferenceDataService.getMapProjects()`

<Steps>
  <Step title="Developer triggers geocoding">
    POST request to `/developer-portal/projects/:id/geocode`
  </Step>
  <Step title="MapService queries 2GIS API">
    Project address converted to lat/lng coordinates using 2GIS catalog
  </Step>
  <Step title="Coordinates stored">
    Results saved to `RefProject` entity latitude/longitude fields in reference database
  </Step>
  <Step title="Map data served">
    Coordinates included in `/reference/projects/map` endpoint for frontend map rendering
  </Step>
</Steps>

## Implementation details

### Architecture benefits

1. **Decoupled data sources** - CRM remains functional even if reference database changes
2. **Specialized access patterns** - Three controllers optimize for different use cases
3. **Scalable enrichment** - Soft delete enables safe content management
4. **Flexible deployment** - Reference database can be external API in future

### Performance considerations

- **Indexed references** - All `refProjectId` and `refUnitId` fields are indexed
- **Efficient joins** - MikroORM relations optimize database queries
- **Caching strategy** - Reference data suitable for caching layers
- **Map optimization** - Separate map endpoint reduces payload for listing views

### Error handling patterns

```typescript
// Reference data not found doesn't break CRM operations
const project = await this.referenceDataService.findProject(refProjectId);
if (!project) {
  // CRM entity remains valid, reference data handled gracefully
  return { ...crmData, project: null };
}
```

<Note>
The reference data module serves as the foundation for all property-related functionality while maintaining clean separation from organization-specific CRM data. The modular architecture enables each component to evolve independently while sharing common property data infrastructure.
</Note>
