The real estate module is structured as **six specialized modules** spanning three visibility scopes: **global** (no RLS), **org-private** (standard RLS), and **cross-org** (custom RLS). An external **reference database** (Neon "propwise-lab") stores **all property data** â€” both structural (DLD registry) and enrichment (gallery, payment plans, documents, amenities, pricing, geocoding). The CRM database stores only CRM-specific data (developer portal auth, ownership links, property interests, marketplace listings, secondary unit context).

<Note>
All modules consume property data exclusively through `ReferenceDataService`. If the reference database is swapped for an external property API, only `ReferenceDataService` needs to change.
</Note>

## Data ownership

The dual-database architecture separates property data from CRM data:

| Data Type | Database | Access Pattern |
| --- | --- | --- |
| Projects, units, buildings, areas, communities, developers | Reference DB | `ReferenceDataService` |
| Gallery images, payment plans, documents, amenities | Reference DB | `ReferenceDataService` |
| Geocoding (lat/lng), pricing, sale status, handover | Reference DB | `ReferenceDataService` |
| Developer auth (DeveloperUser) | CRM DB | Direct entity access |
| Developer-to-project ownership (DeveloperProjectLink) | CRM DB | Direct entity access |
| Secondary units (org-scoped inventory) | CRM DB | Direct entity access |
| Marketplace listings & deals (SharedListing, SharedDeal) | CRM DB | Direct entity access |
| Property interests (Lead/Deal) | CRM DB | Direct entity access, ref IDs as plain ints |
| Property matches | CRM DB | Direct entity access, ref IDs as plain ints |

## Cross-database reference pattern

CRM entities reference property data using **plain integer IDs** without foreign key constraints:

```typescript
// CRM entity referencing reference DB data
@Property({ type: 'int', nullable: true })
@Index()
refProjectId?: number;  // ID in reference DB projects table

@Property({ type: 'int', nullable: true })
@Index()
refUnitId?: number;     // ID in reference DB units table
```

This pattern decouples the CRM from the property data source and enables flexible data architecture. No cross-database foreign keys exist â€” this allows the reference database to be swapped for external APIs without breaking CRM functionality.

## Reference data module

**Path:** `src/modules/real-estate/reference-data/`

**Purpose:** Connects to a second PostgreSQL database (Neon "propwise-lab") containing **all property data**: DLD registry data (structural) plus enrichment data (gallery, payment plans, documents, amenities, sales contacts, pricing, geocoding). Provides read endpoints for the off-plan directory and write endpoints for the developer portal.

**Scope:** Global - no RLS, read + write access to second database

**Key characteristics:**
- All property data centralized in reference database
- Three specialized controllers for different access patterns
- Serves both public directory and developer portal needs
- Handles enrichment data lifecycle (soft delete pattern)

### Second database connection

Configured in `mikro-orm-reference.config.ts` at project root and registered with named context:

```typescript
// app.module.ts
MikroOrmModule.forRoot({
  ...mikroOrmReferenceConfig,
  contextName: 'reference',
});
```

### Entity pattern

Reference entities map to external DB tables with specific characteristics:

- **No UUID primary keys** (use `number` PKs matching external DB)
- **No organization FK** (global scope)
- **Soft delete** (`deletedAt` timestamp) for enrichment entities

```typescript
@Entity({ tableName: 'projects' })
export class RefProject {
  @PrimaryKey()
  id!: number;  // Number PK matching external DB

  @Property({ nullable: true })
  name?: string;

  @ManyToOne(() => RefDeveloper, { fieldName: 'developer_id', nullable: true })
  developer?: RefDeveloper;

  @ManyToOne(() => RefArea, { fieldName: 'area_id' })
  area!: RefArea;

  // Enrichment fields
  @Property({ type: 'double', nullable: true })
  latitude?: number;

  @Property({ fieldName: 'price_from', type: 'numeric', nullable: true })
  priceFrom?: number;

  @Property({ fieldName: 'is_published', default: false, nullable: true })
  isPublished?: boolean;

  // Soft delete for enrichment
  @Property({ type: 'timestamp', nullable: true })
  deletedAt?: Date;

  // Enrichment collections
  @OneToMany(() => RefProjectGalleryImage, (img) => img.project)
  gallery = new Collection<RefProjectGalleryImage>(this);

  @OneToMany(() => RefPaymentPlan, (p) => p.project)
  paymentPlans = new Collection<RefPaymentPlan>(this);
  // ... documents, amenities
}
```

## Off-plan filtering logic

"Off-plan" is a **derived filter** applied to reference data queries, not a separate module. The business logic determines off-plan status based on multiple criteria:

```typescript
private applyOffPlanFilter(qb: QueryBuilder<RefProject | RefBuilding>) {
  // Project must not have ended
  qb.andWhere({
    $or: [
      { endDate: { $gt: new Date() } },  // Future end date
      { endDate: null },                 // No end date set
    ],
  });
  
  // Project must not be completed
  qb.andWhere({ completionDate: null });
  
  // Project must be in active development phases
  qb.andWhere({
    status: { 
      $in: ['active', 'pending', 'under_construction', 'announced'] 
    },
  });
  
  // Project must be published for public viewing
  qb.andWhere({ isPublished: true });
}
```

<Info>
A project/building is "off-plan" when: end date is future/null, completion date is null, status is active/pending/under_construction/announced, and is published. This filter is applied dynamically across all relevant endpoints.
</Info>

## Map integration (2GIS)

**MapService** (`src/modules/real-estate/map/map.service.ts`) provides shared geocoding utility across modules:

- **API:** `https://catalog.api.2gis.com/3.0/items/geocode`
- **Environment:** `DGIS_API_KEY`
- **Frontend:** 2GIS MapGL JS API (docs: https://docs.urbi.ae)
- **Map data:** Fetched via `ReferenceDataService.getMapProjects()`

<Steps>
  <Step title="Developer triggers geocoding">
    POST request to `/developer-portal/projects/:id/geocode`
  </Step>
  <Step title="MapService queries 2GIS API">
    Project address converted to lat/lng coordinates using 2GIS catalog
  </Step>
  <Step title="Coordinates stored">
    Results saved to `RefProject` entity latitude/longitude fields in reference database
  </Step>
  <Step title="Map data served">
    Coordinates included in `/reference/projects/map` endpoint for frontend map rendering
  </Step>
</Steps>

## Developer portal integration

The developer portal writes to the reference database through `ReferenceDataService`, providing content management for property data:

### Authentication system

The developer portal operates with separate JWT authentication (not organization-based):

**Controllers:**
- `developer-auth.controller.ts` - `/developer-portal/auth` endpoints
  - `POST /signup` - Developer registration with agent card upload requirement
  - `POST /login` - JWT token generation
  - `POST /refresh` - Token refresh
  - `POST /logout` - Session termination
  - `POST /password/reset` - Password reset flow

**Profile management:**
- `developer-profile.controller.ts` - `/developer-portal/me` endpoints
  - `GET /me` - Developer profile retrieval
  - `PATCH /me` - Profile updates
  - `POST /agent-card` - Agent card upload for verification

### Project management

Full CRUD operations on reference database through specialized controllers:

**Project operations:**
- `dev-projects.controller.ts` - `/developer-portal/projects` endpoints
  - Project CRUD with validation
  - Collection management and bulk operations
  - Nested building creation and listing
  - Media upload handling (gallery, documents)

**Building operations:**
- `dev-buildings.controller.ts` - `/developer-portal/buildings` endpoints
  - Building CRUD (flat structure)
  - Collection management
  - Nested unit CRUD operations

### Data flow

```typescript
// Developer portal writes through ReferenceDataService
await this.referenceDataService.updateProject(projectId, updateData);

// Changes immediately available in public directory
const publicProject = await this.referenceDataService.findProjectById(projectId);
```

<Warning>
Developer portal operations bypass standard RLS since they operate on global property data. Access control is handled through separate developer authentication and project ownership verification via DeveloperProjectLink entities.
</Warning>

### Inventory module

**Path:** `src/modules/real-estate/inventory/`
**Scope:** Org-scoped with standard RLS

- References primary property data using `refProjectId`/`refUnitId` pattern
- Organization-specific inventory management

### Shared marketplace module

**Path:** `src/modules/real-estate/shared-marketplace/`
**Scope:** Cross-org with custom RLS

Enables listings and deals across organizations:

- Custom RLS allowing both participating organizations access
- SharedListing and SharedDeal entities
- Cross-organizational transaction management

### Property interest module

**Path:** `src/modules/real-estate/property-interest/`
**Scope:** Org-scoped with standard RLS

Manages lead and deal property interests with matching engine:

- Lead/Deal entities store property references as plain integers
- Matching engine connects interests with available properties
- Uses ReferenceDataService for property data enrichment

## Implementation details

### Architecture benefits

1. **Decoupled data sources** - CRM remains functional even if reference database changes
2. **Specialized access patterns** - Three controllers optimize for different use cases
3. **Scalable enrichment** - Soft delete enables safe content management
4. **Flexible deployment** - Reference database can be external API in future

### Performance considerations

- **Indexed references** - All `refProjectId` and `refUnitId` fields are indexed
- **Efficient joins** - MikroORM relations optimize database queries
- **Caching strategy** - Reference data suitable for caching layers
- **Map optimization** - Separate map endpoint reduces payload for listing views

### Error handling patterns

```typescript
// Reference data not found doesn't break CRM operations
const project = await this.referenceDataService.findProject(refProjectId);
if (!project) {
  // CRM entity remains valid, reference data handled gracefully
  return { ...crmData, project: null };
}
```

<Note>
The reference data module serves as the foundation for all property-related functionality while maintaining clean separation from organization-specific CRM data. The modular architecture enables each component to evolve independently while sharing common property data infrastructure.
</Note>
