## Architecture overview

The real estate module is structured as **six specialized modules** spanning three visibility scopes: **global** (no RLS), **org-private** (standard RLS), and **cross-org** (custom RLS).

The external **Propwise Labs API** stores and serves all property data — both structural (DLD registry) and enrichment (images, payment plans, documents, amenities, pricing, geocoding where exposed). The CRM database stores only CRM-specific data (developer portal auth, ownership links, property interests, marketplace listings, secondary unit context).

<Note>
**Architecture:** All modules consume property data exclusively through `PropwiseLabsService`. The CRM no longer opens a second MikroORM connection or maps Propwise Labs entities locally. The reference database configuration has been completely removed and replaced with Propwise Labs API configuration (`app.propwiseLabs.*`) including API credentials, timeout settings, and Bird Verify integration for developer portal contact verification.
</Note>

### External property ID pattern

<Warning>
CRM entities that reference off-plan projects or units store **plain integer IDs** (`refProjectId`, `refUnitId`) from Propwise Labs. No foreign keys exist — this decouples the CRM from the property data source.
</Warning>

<CodeGroup>
```typescript Property interest with reference IDs
@Entity()
export class PropertyInterest {
  @PrimaryKey({ type: UuidType })
  id: string = v4();

  // Propwise Labs project ID (plain integer)
  @Property({ type: 'int', nullable: true })
  @Index()
  refProjectId?: number;

  // Propwise Labs unit ID (plain integer)
  @Property({ type: 'int', nullable: true })
  @Index()
  refUnitId?: number;

  // Standard CRM fields
  @ManyToOne(() => Organization)
  organization!: Organization;

  @Property({ type: 'boolean', default: false })
  isDeleted: boolean = false;
}
```

```typescript Marketplace listing with reference IDs
@Entity()
export class SharedListing {
  @PrimaryKey({ type: UuidType })

  // Reference to off-plan unit via plain integer
  @Property({ type: 'int', nullable: true })
```

## Propwise Labs module

**Path:** `src/modules/real-estate/propwise-labs/`

### Purpose

Provides the CRM backend facade over the external Propwise Labs API. It handles API authentication, bearer-token caching, read calls for property data, lookup data, change-request submissions for developer portal mutations, and typed wrappers for super-level direct Labs mutations.

### API client pattern

`PropwiseLabsService` reads API credentials from `app.propwiseLabs.*`, exchanges `PROPWISE_LABS_API_KEY` and `PROPWISE_LABS_API_SECRET` for a short-lived bearer token, and reuses that token until it is close to expiry. The service includes configurable timeout settings and comprehensive error handling for external API dependencies.

```typescript
@Injectable()
export class PropwiseLabsService {
  private accessToken?: string;
  private tokenExpiresAt = 0;

  async searchProjects(filters: SearchProjectFilters): Promise<ProjectSearchResult[]> {
    const token = await this.ensureValidToken();
    // External API call with bearer authentication
  }

  async submitChangeRequest(request: ChangeRequestPayload): Promise<ChangeRequestResponse> {
    const token = await this.ensureValidToken();
    // Submit mutation request to Propwise Labs
  }
}
```

## Developer portal module

**Path:** `src/modules/real-estate/developer-portal/`

### Purpose

Separate authentication system for developers with **project management capabilities**. Uses custom JWT tokens (not organization-based) and provides CRUD operations for projects/buildings via `PropwiseLabsService` direct edits and change requests. Includes Bird Verify integration for contact verification.

### Authentication flow

<Steps>
<Step title="Developer signup">
Requires agent card upload verification and Bird Verify contact OTP. Creates `DeveloperUser` entity in CRM DB.
</Step>
<Step title="Project ownership linking">
Managed through Propwise Labs API, no local ownership entities.
</Step>
<Step title="Custom JWT tokens">
Issues developer-specific tokens separate from org-based authentication.
</Step>
</Steps>

## Shared marketplace module

**Path:** `src/modules/real-estate/shared-marketplace/`

### Purpose

Enables **cross-organization** property listings and deal negotiations. Uses custom RLS allowing access to both listing and inquiring organizations.

### Key entities

| Entity | Purpose |
| --- | --- |
| `SharedListing` | Property listings (off-plan or secondary) visible across orgs |
| `SharedDeal` | Deal negotiations between organizations |
| `SharedDealMessage` | Communication thread within deals |

### Custom RLS pattern

<Warning>
Custom RLS grants access when the current organization matches **either** the listing org OR the inquiring org.
</Warning>

```typescript
@Entity()
@Filter({
  name: 'crossOrgAccess',
  default: true,
  cond: {
    $or: [
      { listingOrganization: { id: ':orgId' } },
      { inquiringOrganization: { id: ':orgId' } }
    ]
  }
})
export class SharedDeal {
  @ManyToOne(() => Organization)
  listingOrganization!: Organization;
  
  @ManyToOne(() => Organization, { nullable: true })
  inquiringOrganization?: Organization;
}
```

## Property interest module

**Path:** `src/modules/real-estate/property-interest/`

### Purpose

Manages property interests for leads/deals with a **matching engine** that suggests relevant properties. References off-plan properties via plain integer IDs and uses `PropwiseLabsService` for property data retrieval.

### Matching engine

The matching engine compares client requirements against:
- Off-plan properties (via `PropwiseLabsService`)
- Secondary inventory (direct entity access)
- Marketplace listings (cross-org visibility)

### Interest storage pattern

```typescript
@Entity()
export class PropertyInterest {
  @Property({ type: 'int', nullable: true })
  @Index()

  @Property({ type: 'int', nullable: true })
  @Index() 
  refUnitId?: number;     // Off-plan unit reference
  
  // Criteria for matching
  @Property({ type: 'json', nullable: true })
  searchCriteria?: PropertySearchCriteria;
}
```

## API integration patterns

<Note>
The Propwise Labs API integration uses bearer token authentication with automatic token refresh and configurable timeout settings.
</Note>

<CodeGroup>
```typescript PropwiseLabsService injection
@Injectable()
export class PropertyInterestService {
  constructor(
    private readonly propwiseLabsService: PropwiseLabsService,
    private readonly em: EntityManager,
  ) {}

  async findMatchingProperties(interest: PropertyInterest): Promise<PropertyMatch[]> {
    const offPlanResults = await this.propwiseLabsService.searchProjects({
      areaId: interest.searchCriteria?.areaId,
      priceRange: interest.searchCriteria?.priceRange,
    });
    
    // Combine with secondary inventory matches
    return this.combineMatches(offPlanResults, secondaryMatches);
  }
}
```

```typescript Change request pattern
@Injectable()
export class DeveloperPortalProjectsService {
  constructor(
    private readonly propwiseLabsService: PropwiseLabsService,
  ) {}

  async updateProject(projectId: number, updates: ProjectUpdatePayload): Promise<ChangeRequestResponse> {
    return this.propwiseLabsService.submitChangeRequest({
      type: 'project_update',
      resourceId: projectId,
      changes: updates,
    });
  }
}
```
</CodeGroup>

```typescript Service initialization
@Injectable()
export class PropwiseLabsService {
  constructor(
    @Inject('PROPWISE_LABS_CONFIG')
    private readonly config: PropwiseLabsConfig,
  ) {}

  private async ensureValidToken(): Promise<string> {
    if (!this.accessToken || Date.now() >= this.tokenExpiresAt) {
      await this.refreshToken();
    }
    return this.accessToken!;
  }
}
```

## DTO patterns

All response DTOs follow the **constructor-based transformation** pattern:

<CodeGroup>

```typescript SecondaryUnitDto
export class SecondaryUnitDto {
  id: string;
  constructor(unit: SecondaryUnit, canViewUnitNumber: boolean = true) {
    this.id = unit.id;
    this.unitNumber = canViewUnitNumber ? unit.unitNumber : '***';
  }
}
```

```typescript OffPlanProjectDetailDto
export class OffPlanProjectDetailDto {
  constructor(project: PropwiseLabsProject, units?: PropwiseLabsUnit[]) {
    this.id = project.id;
    this.name = project.name;
    this.gallery = project.gallery?.map(img => new PropwiseLabsImageDto(img)) || [];
  }
}
```
</CodeGroup>

<CodeGroup>
```typescript SharedDealDetailDto
export class SharedDealDetailDto {
  constructor(
    deal: SharedDeal,
    listing: SharedListing,
    refProject?: PropwiseLabsProject,
    refUnit?: PropwiseLabsUnit,
  ) {
    this.id = deal.id;
    this.status = deal.status;
    
    // Handle both off-plan and secondary properties
    if (listing.refUnitId && refProject && refUnit) {
      this.property = {
        type: 'off-plan',
        project: refProject.name,
        unit: refUnit.unitNumber,
      };
    } else if (listing.secondaryUnit) {
      this.property = {
        type: 'secondary',
        unitNumber: listing.secondaryUnit.unitNumber,
      };
    }
  }
}
```

### Naming convention

| Pattern | Example |
| --- | --- |
| `EntityDto` | `SecondaryUnitDto`, `SharedListingDto`, `PropwiseLabsProjectDto` |
| `EntityDetailDto` | `SharedDealDetailDto`, `OffPlanProjectDetailDto` |
| No `ResponseDto` suffix | Use `SecondaryUnitDto`, not `SecondaryUnitResponseDto` |
