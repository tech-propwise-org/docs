## Architecture overview

The real estate module is structured as **six active specialized modules** spanning the global and org-private real-estate surfaces. The cross-org `shared-marketplace` code is retained in the repository but is not imported by `RealEstateModule`, so its controllers/providers are not mounted by the app.

The external **Propwise Labs API** stores and serves all property data — both structural (DLD registry) and enrichment (images, payment plans, documents, amenities, pricing, geocoding where exposed). The CRM database stores only CRM-specific data (developer portal auth, ownership links, property interests, marketplace listings, and org inventory context).

<Note>
**Architecture:** All modules consume property data exclusively through `PropwiseLabsService`. The CRM no longer opens a second MikroORM connection or maps Propwise Labs entities locally. The reference database configuration has been completely removed and replaced with Propwise Labs API configuration (`app.propwiseLabs.*`) including API credentials, timeout settings, and Bird Verify integration for developer portal contact verification.
</Note>

<Info>
**Portal syndication — self-contained `Listing` (2026 build):** The `portal-syndication` module's `Listing` is now self-contained: every field any portal (Bayut/dubizzle + Property Finder) needs lives on `Listing`, snapshotted from an `InventoryUnit` at create (linked mode) or entered directly (manual mode). The `inventoryUnit` FK is **optional**. Deleting an inventory unit cascades (event-driven, `inventory-unit.deleted` carrying the user's `removeLinkedListings` choice): **remove** (default) delists the unit's listings from all portals and archives them; **keep** unlinks them (`inventoryUnit = null`) so they survive as self-contained manual listings. Buildings auto-maintain `salesStatus` based on available units (`OUT_OF_STOCK` when no units remain `AVAILABLE`, reverting to `ON_SALE` when available units reappear). New `downpaymentType` enum distinguishes amount vs percentage downpayments.

**Listing approval workflow (Gate A):** a user without `real_estate.listing.publish` / `real_estate.manage` submits a listing for approval (`ListingStatus.PENDING_APPROVAL`). A `real_estate.manage` user approves (auto-publishes to the submitter's selected portals) or rejects (`ListingStatus.REJECTED` + `rejectionReason` persisted on the listing) from a manage-only **Requests** queue (`GET /portal-syndication/listings?type=requests` = pending + rejected). The submitter sees the rejection note and can edit + resubmit (clears the rejection) or delete. The listing **Delete** action soft-deletes (`isDeleted=true`) — there is no user-facing archived state (the `archived` status / `isArchived` columns remain only for the inventory-unit delete cascade). A `real_estate.manage` user may delete **any** listing (including pending requests and rejected ones) directly from the Requests queue.

**Notifications:** submitting emits `LISTING_APPROVAL_REQUESTED` to all `real_estate.manage` approvers; approve / reject / delete emit `LISTING_APPROVED` / `LISTING_REJECTED` / `LISTING_DELETED` to the listing publisher (delete only when the deleter is not the publisher). See `NOTIFICATION_IMPLEMENTATION_GUIDE.md`. Full design: `PORTAL_SYNDICATION_SPECIFICATION.md` §1.5 (As-Built).

**Listing media ingest (WebP) + `/storage/upload` `profile` contract:** listing photos and floor plans are uploaded through the shared `POST /storage/upload` endpoint (`StorageApi.uploadFile(file, profile?)`), which takes an optional **`profile`** body field validated against `STORAGE_UPLOAD_PROFILES` (unknown → `400`). The default is **`GENERIC_ATTACHMENT`** (long edge ≤ 1920, WebP q80) so email/workflow/deal-document/tiptap uploads stay lean. Listing image call sites pass **`LISTING`** (≤ 2560, q82) and floor-plan call sites pass **`FLOOR_PLAN`** (≤ 2560, q90) so syndication imagery reaches Property Finder at high resolution and floor plans stay legible. The endpoint optimizes the buffer through `ImageOptimizationService` (raster → WebP; PDF/doc/video → passthrough), writes the object under a profile-specific R2 folder (`STORAGE_UPLOAD_FOLDERS`: `email-attachments` / `listings` / `listings/floor-plans`), and reflects the converted `fileName` (`.webp`) + `mimeType` + `fileSize` in the JSON response. `ListingImageService` (the Property Finder pre-flight) is unchanged: its `compliant` check re-processes a source image only when it exceeds PF's 1920×1080 / format limits — an expected single PF-specific re-encode, not an ingest regression.
</Info>

## External property ID pattern

`InventoryUnit` is the CRM boundary for org-owned secondary inventory. It stores plain integer IDs (`refProjectId`, `refBuildingId`, `refUnitId`, `refDeveloperId`, `refPropertyTypeId`, `refPropertySubTypeId`) plus a denormalized Labs snapshot, including the developer display name used by portal syndication and the canonical `propertyType` (`LayoutType` enum) used by portal syndication adapters. Labs raw property type/sub-type names are retained in `labsPropertyType` and `labsPropertySubType` for traceability. Lead interests can use inventory units as candidate matches while retaining search criteria. Deal property rows are concrete-only: they link either an org `InventoryUnit`, a direct Propwise Labs off-plan unit via `refProjectId` / `refBuildingId` / `refUnitId`, or a pending unit-access request before the inventory link is approved.

## Propwise Labs module

**Path:** `src/modules/real-estate/propwise-labs/`

### Purpose

Provides the CRM backend facade over the external Propwise Labs API. It handles API authentication, bearer token caching, read calls for property data, lookup data, change-request submissions for developer portal mutations, and typed wrappers for super-level direct Labs mutations. Local `Ref*` MikroORM entities and the named `reference` EntityManager have been removed.

### API client pattern

`PropwiseLabsService` reads API credentials from `app.propwiseLabs.*`, exchanges `PROPWISE_LABS_API_KEY` and `PROPWISE_LABS_API_SECRET` for a short-lived bearer token, and reuses that token until it is close to expiry. The service includes configurable timeout settings and comprehensive error handling for external API dependencies.

export class PropwiseLabsService {

    const token = await this.ensureValidToken();
    // Submit mutation request to Propwise Labs

## Portal syndication module

**Path:** `src/modules/real-estate/portal-syndication/`

### Purpose

Manages **portal listing, configuration, and sync state** for org-scoped inventory units. Handles marketing listings across different portals and maintains per-portal sync states.

### Portal syndication architecture

The portal syndication module features a comprehensive listing approval workflow and self-contained listing management:

<AccordionGroup>
<Accordion title="Listing workflow">

<Steps>
<Step title="Listing creation">
Users create listings in either linked mode (from `InventoryUnit`) or manual mode (standalone). All portal-required fields are snapshotted to the `Listing` entity.
</Step>
<Step title="Approval process">
Users without `real_estate.listing.publish` permission submit for approval (`ListingStatus.PENDING_APPROVAL`). Users with `real_estate.manage` can approve or reject from the Requests queue.
</Step>
<Step title="Publishing and sync">
Approved listings auto-publish to selected portals. The system maintains per-portal sync states and handles portal-specific formatting.
</Step>
</Steps>

</Accordion>

<Accordion title="Inventory unit cascade behavior">

When an inventory unit is deleted, the system handles linked listings based on the user's choice:
- **Remove (default):** Delists from all portals and archives the listings
- **Keep:** Unlinks listings (`inventoryUnit = null`) so they survive as self-contained manual listings

Buildings automatically maintain `salesStatus` based on available units:
- `OUT_OF_STOCK` when no units remain `AVAILABLE`
- `ON_SALE` when available units reappear

</Accordion>

<Accordion title="Listing media ingest">

The portal syndication module integrates with the shared storage upload system for optimized image handling:

**Upload profiles:**
- **`LISTING`** (≤ 2560px, WebP q82) - For high-resolution syndication imagery
- **`FLOOR_PLAN`** (≤ 2560px, WebP q90) - For legible floor plan documents
- **`GENERIC_ATTACHMENT`** (≤ 1920px, WebP q80) - Default for other uploads

**Image optimization flow:**
1. Upload through `POST /storage/upload` with appropriate profile
2. `ImageOptimizationService` converts raster images to WebP format
3. Files stored in profile-specific R2 folders (`listings`, `listings/floor-plans`)
4. Response includes converted filename and optimized file size

**Property Finder compatibility:**
The existing `ListingImageService` compliant check remains unchanged, providing single PF-specific re-encoding when source images exceed Property Finder's 1920×1080 limits.

</Accordion>
</AccordionGroup>

    private readonly config: PropwiseLabsConfig,
