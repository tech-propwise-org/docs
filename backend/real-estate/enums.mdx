## Architecture overview

The real estate module is structured as **six specialized modules** spanning three visibility scopes: **global** (no RLS), **org-private** (standard RLS), and **cross-org** (custom RLS).

An external **reference database** (Neon "propwise-lab") stores **all property data** — both structural (DLD registry) and enrichment (gallery, payment plans, documents, amenities, pricing, geocoding). The CRM database stores only CRM-specific data (developer portal auth, ownership links, property interests, marketplace listings, secondary unit context).

<Note>
**Architecture:** All modules consume property data exclusively through `ReferenceDataService`. If the reference database is swapped for an external property API, only `ReferenceDataService` needs to change.
</Note>

### Architecture diagram

```
src/modules/real-estate/
├── reference-data/        # Second DB — ALL property data (structural + enrichment)
│   ├── reference-data.controller.ts       # /reference — areas, cities, communities, developers, property types, units/:id
│   ├── reference-projects.controller.ts   # /reference/projects — search (?type=off-plan), map, counts, detail, nested buildings/units
│   └── reference-buildings.controller.ts  # /reference/buildings — search (?type=off-plan), detail, nested units
├── developer-portal/      # Separate auth + project management (writes via ReferenceDataService)
│   ├── auth/developer-auth.controller.ts  # /developer-portal/auth — signup (with agent card), login, refresh, logout, password
│   ├── profile/developer-profile.controller.ts # /developer-portal/me — profile GET/PATCH, agent-card upload
│   └── project-management/
│       ├── dev-projects.controller.ts     # /developer-portal/projects — project CRUD, collections, nested building create/list
│       └── dev-buildings.controller.ts    # /developer-portal/buildings — building CRUD (flat), collections, nested unit CRUD
├── map/                   # MapService (2GIS geocoding utility — shared across modules)
├── inventory/             # Org-scoped secondary units (standard RLS)
├── shared-marketplace/    # Cross-org listings + deals (custom RLS)
└── property-interest/     # Property interests + matching engine (uses ReferenceDataService)
```

## Cross-database reference pattern

<Warning>
CRM entities that reference off-plan projects or units store **plain integer IDs** (`refProjectId`, `refUnitId`) pointing to the reference database. No cross-database foreign keys exist — this decouples the CRM from the property data source.
</Warning>

<CodeGroup>
```typescript Property interest with reference IDs
@Entity()
export class PropertyInterest {
  @PrimaryKey({ type: UuidType })
  id: string = v4();

  // Reference DB project ID (plain integer)
  @Property({ type: 'int', nullable: true })
  @Index()
  refProjectId?: number;

  // Reference DB unit ID (plain integer)
  @Property({ type: 'int', nullable: true })
  @Index()
  refUnitId?: number;

  // Standard CRM fields
  @ManyToOne(() => Organization)
  organization!: Organization;

  @Property({ type: 'boolean', default: false })
  isDeleted: boolean = false;
}
```

```typescript Marketplace listing with reference IDs
@Entity()
export class SharedListing {
  @PrimaryKey({ type: UuidType })
  id: string = v4();

  // Reference to off-plan unit via plain integer
  @Property({ type: 'int', nullable: true })
  @Index()
  refUnitId?: number;

  // OR secondary unit via UUID FK
  @ManyToOne(() => SecondaryUnit, { nullable: true })
  secondaryUnit?: SecondaryUnit;

  @Property()
  listingType!: ListingType;

  @Property()
  status!: ListingStatus;
}
```
</CodeGroup>

## Reference data module

**Path:** `src/modules/real-estate/reference-data/`

### Purpose

Connects to a second PostgreSQL database (Neon "propwise-lab") containing **all property data**: DLD registry data (structural) plus enrichment data (gallery, payment plans, documents, amenities, sales contacts, pricing, geocoding). Provides read endpoints for the off-plan directory and write endpoints for the developer portal.

### Second database connection

Configured in `mikro-orm-reference.config.ts` at project root. Registered in `app.module.ts` with `contextName: 'reference'`:

```typescript
MikroOrmModule.forRoot({
  ...mikroOrmReferenceConfig,
  contextName: 'reference',
});
```

### Entity pattern

All reference entities are mapped to the external DB tables. They have **no UUID primary keys** (use `number` PKs matching the external DB), **no organization FK**, and use **soft delete** (`deletedAt` timestamp) for enrichment entities.

```typescript
@Entity({ tableName: 'projects' })
export class RefProject {
  @PrimaryKey()
  id!: number;

  @Property({ nullable: true })
  name?: string;

  @ManyToOne(() => RefDeveloper, { fieldName: 'developer_id', nullable: true })
  developer?: RefDeveloper;

  @ManyToOne(() => RefArea, { fieldName: 'area_id' })
  area!: RefArea;

  // Enrichment fields
  @Property({ type: 'double', nullable: true })
  latitude?: number;

  @Property({ fieldName: 'price_from', type: 'numeric', nullable: true })
  priceFrom?: number;

  @Property({ fieldName: 'is_published', default: false, nullable: true })
  isPublished?: boolean;

  // Enrichment collections
  @OneToMany(() => RefProjectGalleryImage, (img) => img.project)
  gallery = new Collection<RefProjectGalleryImage>(this);

  @OneToMany(() => RefPaymentPlan, (p) => p.project)
  paymentPlans = new Collection<RefPaymentPlan>(this);
  // ... documents, amenities
}
```

### Service injection pattern

Use the named EntityManager for the reference context:

```typescript
@Injectable()
export class ReferenceDataService {
  constructor(
    @InjectEntityManager('reference')
    private readonly em: EntityManager,
  ) {}
}
```

### Module registration

Feature entities are registered under the `'reference'` context name:

```typescript
MikroOrmModule.forFeature(
  [
    RefArea,
    RefProject,
    RefUnit,
    RefBuilding,
    RefDeveloper,
    RefProjectGalleryImage,
    RefPaymentPlan,
    // ... other reference entities
  ],
  'reference',
)
```

## Database context patterns

<Note>
The reference database uses a separate MikroORM context (`'reference'`) with named EntityManager injection.
</Note>

<CodeGroup>
```typescript Reference service injection
@Injectable()
export class ReferenceDataService {
  constructor(
    @InjectEntityManager('reference')
    private readonly em: EntityManager,
  ) {}

  async findProject(id: number): Promise<RefProject | null> {
    return this.em.findOne(RefProject, { id });
  }
}
```

```typescript CRM service injection (default context)
@Injectable()
export class InventoryService {
  constructor(
    private readonly em: EntityManager, // Default context
  ) {}

  async findSecondaryUnit(id: string): Promise<SecondaryUnit | null> {
    return this.em.findOne(SecondaryUnit, { id });
  }
}
```

```typescript Module registration
// Reference entities
MikroOrmModule.forFeature(
  [RefProject, RefUnit, RefBuilding, RefArea],
  'reference' // Named context
)

// CRM entities
MikroOrmModule.forFeature([
  SecondaryUnit, SharedListing, PropertyInterest
]) // Default context
```
</CodeGroup>

## DTO patterns

All response DTOs follow the **constructor-based transformation** pattern:

<CodeGroup>
```typescript SecondaryUnitDto
export class SecondaryUnitDto {
  id: string;
  constructor(unit: SecondaryUnit, canViewUnitNumber: boolean = true) {
    this.id = unit.id;
    this.unitNumber = canViewUnitNumber ? unit.unitNumber : '***';
  }
}
```

```typescript OffPlanProjectDetailDto
export class OffPlanProjectDetailDto {
  constructor(project: RefProject, units?: RefUnit[]) {
    this.id = project.id;
    this.name = project.name;
    this.gallery = project.gallery?.isInitialized()
      ? project.gallery.getItems()
          .filter((img) => !img.deletedAt)
          .map((img) => new RefGalleryImageDto(img))
      : [];
  }
}
```

```typescript SharedDealDetailDto
export class SharedDealDetailDto {
  constructor(
    deal: SharedDeal,
    listing: SharedListing,
    refProject?: RefProject,
    refUnit?: RefUnit,
  ) {
    this.id = deal.id;
    this.status = deal.status;
    
    // Handle both off-plan and secondary properties
    if (listing.refUnitId && refProject && refUnit) {
      this.property = {
        type: 'off-plan',
        project: refProject.name,
        unit: refUnit.unitNumber,
      };
    } else if (listing.secondaryUnit) {
      this.property = {
        type: 'secondary',
        unitNumber: listing.secondaryUnit.unitNumber,
      };
    }
  }
}
```
</CodeGroup>

### Naming convention

| Pattern | Example |
| --- | --- |
| `EntityDto` | `SecondaryUnitDto`, `SharedListingDto`, `RefAreaDto` |
| `EntityDetailDto` | `SharedDealDetailDto`, `OffPlanProjectDetailDto` |
| No `ResponseDto` suffix | Use `SecondaryUnitDto`, not `SecondaryUnitResponseDto` |
