## Overview

This document defines the **external XML contract** specified by Bayut for syndicating property listings to both Bayut and dubizzle portals through a unified feed.

<Note>
For implementation details on how PropWise *generates* this feed from `Listing` and `InventoryUnit` data—including field mapping, portal selection logic, status rules, and type lookups—refer to **Portal Syndication Specification §6.4 (`BayutDubizzleFeedAdapter`)**.
</Note>

### Integration mechanism

<CardGroup cols={2}>
  <Card title="Feed direction" icon="arrow-right">
    One-way sync from CRM → portals. The XML feed is the source of truth for published listings.
  </Card>
  <Card title="Pull-based sync" icon="cloud-arrow-down">
    Bayut's system pulls from a publicly accessible URL on a scheduled basis.
  </Card>
</CardGroup>

### Unified feed architecture

Bayut and dubizzle now share a **single combined XML feed**. Portal targeting is controlled per-property using the `<Portals>` tag.

<Warning>
Clients with existing integrations must retire separate dubizzle feeds and update the Bayut XML structure to use the `<Portals>` attribute.
</Warning>

---

## XML structure

The feed consists of two main segments:

1. **Header tags** — Document envelope and root structure
2. **Property attributes** — Per-property field definitions

### Root element

The XML document must use `<Properties>` as the root element, with individual `<Property>` elements nested inside:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<Properties>
  <Property>
    <!-- Property 1 attributes -->
  </Property>
  <Property>
    <!-- Property 2 attributes -->
  </Property>
</Properties>
```

<Tip>
Wrap all textual values in `<![CDATA[ ... ]]>` to prevent XML parsing issues with HTML, Arabic text, ampersands, and URLs.
</Tip>

---

## Property attributes

All tags listed below are specified at the property level (inside each `<Property>` element).

### Identity and status

<AccordionGroup>
  <Accordion title="Property_Ref_No">
    The unique identifier for the property as used in the client's CRM.
    
    **Rules:**
    - Must be unique across the feed
    - If duplicates are encountered, only one instance is made live
    - Duplicates are marked **Inactive**
    
    **PropWise implementation:** Generated as `UNIT-{orgShortCode}-{listing.id}`
  </Accordion>

  <Accordion title="Permit_Number">
    The Trakheesi Permit number issued by the DLD, or the Madhmoun Permit number issued by ADREC.
    
    <Check>Required for the listing to go live on portals</Check>
  </Accordion>

  <Accordion title="Property_Status">
    Controls whether the property is live on the portal or deleted.
    
    **Allowed values:**
    - `live` — Property is active and visible
    - `deleted` — Property is removed from portal
    
    <Info>
    Deleted entries should remain in the feed for at least one crawl cycle to ensure prompt delisting.
    </Info>
  </Accordion>

  <Accordion title="Last_Updated">
    The last updated date of the listing per the client's CRM.
    
    **Format:** `YYYY-MM-DD HH:mm:ss`
    
    **Example:** `2024-01-15 14:30:00`
  </Accordion>
</AccordionGroup>

### Classification

<AccordionGroup>
  <Accordion title="Property_purpose">
    The purpose of the advertised property.
    
    **Allowed values:**
    - `Buy` — Property for sale
    - `Rent` — Property for rent
  </Accordion>

  <Accordion title="Property_Type">
    The type of property being advertised.
    
    **Allowed values:**
    - `Villa`
    - `Apartment`
    - `Townhouse`
    - `Pent House`
    - `Hotel Apartment`
    - `Loft Apartment`
    - `Office`
    - `Shop`
    - `Warehouse`
    - `Factory`
    - `Labour Camp`
    - `Other Commercial`
    - `Commercial Building`
    - `Commercial Floor`
    - `Commercial Land`
    - `Residential Building`
    - `Residential Floor`
    - `Residential Land`
    
    **PropWise implementation:** Resolved from `Listing.propertyType` via `LAYOUT_TYPE_TO_BAYUT` mapping
  </Accordion>

  <Accordion title="Off_plan">
    Indicates whether the property is ready or off-plan.
    
    **Allowed values:**
    - `Yes` — Off-plan property
    - `No` — Ready property
    
    <Warning>
    When set to `Yes`, additional off-plan detail tags become required (see Off-plan details section).
    </Warning>
  </Accordion>
</AccordionGroup>

### Portals tag

The `<Portals>` tag controls which portal(s) the listing is published to. Use one `<portal>` child element per target portal.

<Tabs>
  <Tab title="Both portals">
    ```xml
    <Portals>
      <Portal><![CDATA[ Bayut ]]></Portal>
      <Portal><![CDATA[ dubizzle ]]></Portal>
    </Portals>
    ```
  </Tab>
  <Tab title="Bayut only">
    ```xml
    <Portals>
      <Portal><![CDATA[ Bayut ]]></Portal>
    </Portals>
    ```
  </Tab>
  <Tab title="dubizzle only">
    ```xml
    <Portals>
      <Portal><![CDATA[ dubizzle ]]></Portal>
    </Portals>
    ```
  </Tab>
</Tabs>

<Note>
**Implementation note:** The specification shows inconsistent casing across examples. PropWise emits `<Portal>` (capitalized) with CDATA-wrapped values (`Bayut`, `dubizzle`).
</Note>

### Property specifications

| Tag | Description | Values/Rules |
|-----|-------------|--------------|
| `Property_Size` | Numeric value of the property area | Number |
| `Property_Size_Unit` | Unit of measurement for the area | Preferred: `SQFT` |
| `plotArea` | Numeric value of the plot area in square feet | Number |
| `Bedrooms` | Number of bedrooms | `-1` (studio), `1`–`10`, `10+` |
| `Bathrooms` | Number of bathrooms | `1`–`10` |
| `Furnished` | Furnishing status | `Yes`, `No`, `Partly` |

<Info>
**Bedrooms mapping:**
- Use `-1` for studio units
- Use `10+` for properties with more than 10 bedrooms
- Omit the tag if value is null
</Info>

**PropWise furnishing mapping:**
- `FURNISHED` → `Yes`
- `UNFURNISHED` → `No`
- `PARTLY_FURNISHED` → `Partly`
- `null` → Tag omitted

### Features

Property features are specified using a `<Features>` parent tag with individual `<Feature>` child tags:

```xml
<Features>
  <Feature>Swimming Pool</Feature>
  <Feature>Gym</Feature>
  <Feature>Covered Parking</Feature>
  <Feature>Built in Wardrobes</Feature>
</Features>
```

### Pricing

| Tag | Description | Required When | Values |
|-----|-------------|---------------|--------|
| `Price` | Sale price or advertised rent amount | Always | Number |
| `Rent_Frequency` | Frequency of rent payment | `Property_purpose` is `Rent` | `Daily`, `Weekly`, `Monthly`, `Yearly` |

<Warning>
`Rent_Frequency` is **required** when the property purpose is `Rent` and must be omitted for sale properties.
</Warning>

### Off-plan details

These tags are **required only** when `<Off_plan>` is set to `Yes`.

<Steps>
  <Step title="Specify sale type">
    **`offplanDetails_saleType`** — Required for all off-plan properties
    
    **Values:** `New`, `Resale`
  </Step>

  <Step title="Add type-specific fields">
    <Tabs>
      <Tab title="New properties">
        **`offplanDetails_dldWaiver`** — DLD waivers applicable
        
        **Values:** `0` to `100` (percentage)
        
        <Check>Required for off-plan `New` properties</Check>
      </Tab>
      <Tab title="Resale properties">
        **`offplanDetails_originalPrice`** — Original price of the property (Number)
        
        **`offplanDetails_amountPaid`** — Amount paid for the property (Number)
        
        <Check>Both fields required for off-plan `Resale` properties</Check>
      </Tab>
    </Tabs>
  </Step>
</Steps>

### Content

| Tag | Description |
|-----|-------------|
| `Property_Title` | The title displayed on the ad (English) |
| `Property_Description` | The description for the listing (English) |
| `Property_Title_AR` | The Arabic title displayed on the ad |
| `Property_Description_AR` | The Arabic description for the listing |

### Media

All URLs must be publicly accessible. Use child elements for individual media items.

<CodeGroup>
```xml Images
<Images>
  <Image>https://cdn.example.com/property/155352/photos/img-1.jpg</Image>
  <Image>https://cdn.example.com/property/155352/photos/img-2.jpg</Image>
  <Image>https://cdn.example.com/property/155352/photos/img-3.jpg</Image>
</Images>
```

```xml Videos
<Videos>
  <Video>https://www.youtube.com/watch?v=example</Video>
  <Video>https://vimeo.com/123456789</Video>
</Videos>
```

```xml Floor plans
<Floor_Plans>
  <Floor_Plan>https://cdn.example.com/property/155352/floorplan-1.pdf</Floor_Plan>
  <Floor_Plan>https://cdn.example.com/property/155352/floorplan-2.pdf</Floor_Plan>
</Floor_Plans>
```
</CodeGroup>

<Tip>
Provide at least one high-quality image for each listing to improve visibility and engagement on the portals.
</Tip>

### Location

Location is specified using a hierarchical structure from city down to building:

| Tag | Description | Example |
|-----|-------------|---------|
| `City` | The City/Emirate | Dubai |
| `Locality` | The main locality | Jumeirah Lakes Towers |
| `Sub_Locality` | The sub-locality | Cluster C |
| `Tower_Name` | Building name (if applicable) | Pretoni Tower |

**Example** for a property in Pretoni Tower, Cluster C, Jumeirah Lakes Towers, Dubai:

```xml
<City>Dubai</City>
<Locality>Jumeirah Lakes Towers</Locality>
<Sub_Locality>Cluster C</Sub_Locality>
<Tower_Name>Pretoni Tower</Tower_Name>
```

### Contact information

| Tag | Description |
|-----|-------------|
| `Listing_Agent` | Agent name displayed with the property |
| `Listing_Agent_Phone` | Agent contact number |
| `Listing_Agent_Email` | Agent email address |

## Validation rules

### Required fields

<Steps>
  <Step title="Core identity">
    - `Property_Ref_No` (must be unique)
    - `Permit_Number` (required for listing to go live)
    - `Property_Status`
  </Step>

  <Step title="Classification">
    - `Property_purpose`
    - `Property_Type`
    - `Portals` (at least one portal)
  </Step>

  <Step title="Basic specs">
    - `Property_Size`
    - `Property_Size_Unit`
    - `Price`
  </Step>

  <Step title="Media">
    - At least one `Image` URL
  </Step>
</Steps>

### Conditional requirements

<AccordionGroup>
  <Accordion title="Rent-specific fields">
    When `Property_purpose` is `Rent`:
    - `Rent_Frequency` is **required**
  </Accordion>

  <Accordion title="Off-plan properties">
    When `Off_plan` is `Yes`:
    - `offplanDetails_saleType` is **required**
    - If sale type is `New`: `offplanDetails_dldWaiver` is **required**
    - If sale type is `Resale`: Both `offplanDetails_originalPrice` and `offplanDetails_amountPaid` are **required**
  </Accordion>
</AccordionGroup>

### Uniqueness constraints

<Warning>
**Property_Ref_No must be unique** across all properties in the feed. If duplicates are detected:
- Only one instance will be made live
- All duplicate instances will be marked **Inactive**
</Warning>

## PropWise implementation notes

The following describes how PropWise maps internal data models to the external XML contract:

<AccordionGroup>
  <Accordion title="Reference number generation">
    `Property_Ref_No` is generated as: `UNIT-{orgShortCode}-{listing.id}`
    
    This ensures uniqueness per organization.
  </Accordion>

  <Accordion title="Status mapping">
    `Property_Status` is derived from `ListingPortalSync` state:
    - Published/pending → `live`
    - Removed/disabled → `deleted`
    
    <Info>
    Deleted entries remain in the feed for at least one crawl cycle to ensure prompt delisting on portals.
    </Info>
  </Accordion>

  <Accordion title="Property type resolution">
    `Property_Type` is resolved from the canonical `Listing.propertyType` (`LayoutType`) using the `LAYOUT_TYPE_TO_BAYUT` mapping defined in:
    
    `src/modules/shared/property-type-portal-map.ts`
  </Accordion>

  <Accordion title="Portal targeting">
    `<Portal>` children are emitted based on active `ListingPortalSync` records:
    - One `<Portal>Bayut</Portal>` if Bayut sync is active
    - One `<Portal>dubizzle</Portal>` if dubizzle sync is active
  </Accordion>

  <Accordion title="Bedrooms normalization">
    Always emitted as a string:
    - `0` → `-1` (studio)
    - `> 10` → `10+`
    - `null` → Tag omitted
  </Accordion>

  <Accordion title="Furnished mapping">
    Mapped from `InventoryUnit.furnished`:
    - `FURNISHED` → `Yes`
    - `UNFURNISHED` → `No`
    - `PARTLY_FURNISHED` → `Partly`
    - `null` → Tag omitted
  </Accordion>

  <Accordion title="Off-plan handling">
    Currently set to `No` for all listings (v1 inventory represents secondary market).
    
    Off-plan detail tags are reserved for future syndication from `OffPlanUnit` entities.
  </Accordion>

  <Accordion title="Timestamp formatting">
    `Last_Updated` uses the latest of listing/sync `updatedAt` timestamps, formatted as: `YYYY-MM-DD HH:mm:ss`
  </Accordion>
</AccordionGroup>
