<Note>
**Purpose:** This document describes the external XML contract defined by Bayut for unified syndication to both Bayut and dubizzle portals.

**Scope:** For how PropWise produces this feed from internal data models, see the Portal Syndication Specification document.
</Note>

## XML integration overview

An XML integration is required to sync properties with Bayut and/or dubizzle in bulk from an external CRM. It allows clients to maintain their inventory on both portals by providing property information in a unified XML feed.

<Info>
The XML feed must be exposed as a **publicly accessible URL**. The Bayut system pulls this URL on a schedule to sync updates.
</Info>

### Key characteristics

- Listings flow **one direction only**: CRM → portals
- The feed is the source of truth for what is published
- Portals never write back into the feed

### Unified feed structure

Bayut and dubizzle now share a **single combined XML feed**. The portal(s) a property appears on is controlled per-property by the `<Portals>` tag.

<Warning>
Clients with existing integrations must retire separate dubizzle feeds and update their Bayut XML structure to use the `<Portals>` attribute for portal targeting.
</Warning>

## XML structure requirements

The XML feed must follow this exact format with two main segments:

1. **Header tags** — document envelope
2. **Property attributes** — per-property fields

### Root structure

```xml
<Properties>
  <Property> <!-- Property 1 attributes --> </Property>
  <Property> <!-- Property 2 attributes --> </Property>
</Properties>
```

<Tip>
All textual values should be wrapped in `<![CDATA[ ... ]]>` to prevent HTML, Arabic text, ampersands, and URLs from breaking XML parsing.
</Tip>

## Property attributes reference

### Core property details

<AccordionGroup>
<Accordion title="Identity & Status Fields">

| Tag | Description | Allowed Values |
|---|---|---|
| `<Property_Ref_No>` | Unique identifier from client's CRM | Must be unique - duplicates marked **Inactive** |
| `<Permit_Number>` | Trakheesi (DLD) or Madhmoun (ADREC) permit | Required for listing to go live |
| `<Property_Status>` | Portal publication status | `live`, `deleted` |
| `<Last_Updated>` | Last updated date from CRM | `YYYY-MM-DD HH:mm:ss` |

</Accordion>

<Accordion title="Classification Fields">

| Tag | Description | Allowed Values |
|---|---|---|
| `<Property_purpose>` | Sale or rental purpose | `Buy`, `Rent` |
| `<Property_Type>` | Type of property | `Villa`, `Apartment`, `Office`, `Shop`, `Warehouse`, `Factory`, `Labour Camp`, `Other Commercial`, `Commercial Building`, `Residential Floor`, `Commercial Floor`, `Residential Land`, `Commercial Land`, `Townhouse`, `Residential Building`, `Hotel Apartment`, `Loft Apartment`, `Pent House` |
| `<Off_plan>` | Ready vs off-plan status | `Yes`, `No` |

</Accordion>

<Accordion title="Property Specifications">

| Tag | Description | Allowed Values |
|---|---|---|
| `<Property_Size>` | Property area (numeric) | Number |
| `<Property_Size_Unit>` | Area measurement unit | Preferred: `SQFT` |
| `<plotArea>` | Plot area in square feet | Number |
| `<Bedrooms>` | Number of bedrooms | `-1` (studio), `1`-`10`, `10+` |
| `<Bathrooms>` | Number of bathrooms | `1`-`10` |
| `<Furnished>` | Furnishing status | `Yes`, `No`, `Partly` |

</Accordion>

<Accordion title="Pricing Fields">

| Tag | Description | Required When |
|---|---|---|
| `<Price>` | Sale price or rent amount | Always |
| `<Rent_Frequency>` | Payment frequency | Only when `Property_purpose` is `Rent` |

**Rent frequency values:** `Daily`, `Weekly`, `Monthly`, `Yearly`

</Accordion>
</AccordionGroup>

### Portal targeting

The `<Portals>` tag controls which portal(s) receive the listing:

<CodeGroup>
```xml Both Portals
<Portals>
  <Portal><![CDATA[ Bayut ]]></Portal>
  <Portal><![CDATA[ dubizzle ]]></Portal>
</Portals>
```

```xml Bayut Only
<Portals>
  <Portal><![CDATA[ Bayut ]]></Portal>
</Portals>
```

```xml dubizzle Only
<Portals>
  <Portal><![CDATA[ dubizzle ]]></Portal>
</Portals>
```
</CodeGroup>

### Property features

Use nested `<Feature>` tags within `<Features>`:

```xml
<Features>
  <Feature>Swimming Pool</Feature>
  <Feature>Gym</Feature>
  <Feature>Parking</Feature>
</Features>
```

### Off-plan properties

<Warning>
These fields are **required only** when `<Off_plan>` is `Yes`
</Warning>

| Tag | Required For | Description |
|---|---|---|
| `<offplanDetails_saleType>` | All off-plan | `New`, `Resale` |
| `<offplanDetails_dldWaiver>` | Off-plan `New` | DLD waiver percentage (0-100) |
| `<offplanDetails_originalPrice>` | Off-plan `Resale` | Original property price |
| `<offplanDetails_amountPaid>` | Off-plan `Resale` | Amount already paid |

## Field validation rules

<Steps>
<Step title="Unique identifiers">
Ensure `<Property_Ref_No>` values are unique across your feed. Duplicates result in only one active listing.
</Step>

<Step title="Required permits">
Include valid `<Permit_Number>` (Trakheesi/Madhmoun) for listings to go live on portals.
</Step>

<Step title="Conditional fields">
- `<Rent_Frequency>` required only for rental properties
- Off-plan detail tags required only when `<Off_plan>` is `Yes`
</Step>

<Step title="Media requirements">
Provide at least one publicly accessible image URL for optimal listing performance.
</Step>
</Steps>

## PropWise implementation notes

<Info>
PropWise generates this XML format from internal `Listing` and `InventoryUnit` data models with the following transformations:
</Info>

- **Property_Ref_No:** Generated as `UNIT-{orgShortCode}-{listing.id}`
- **Property_Status:** `live` for published/pending syncs, `deleted` for removed listings
- **Property_Type:** Mapped from internal `LayoutType` using portal-specific lookup tables
- **Bedrooms:** String format with `0` → `-1` (studio), `>10` → `10+`
- **Furnished:** Mapped from internal enum values to portal format
- **Off_Plan:** Currently `No` for all listings (secondary market inventory)
