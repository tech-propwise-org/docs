## Overview

An XML integration is required to sync properties with Bayut and/or dubizzle in bulk from an external CRM. This document defines the **external XML contract** that PropWise generates to syndicate listings to both portals.

<Note>
  **Scope:** This reference describes the XML feed format defined by Bayut. For how PropWise *produces* this feed from `Listing` + `InventoryUnit` data (field mapping, portal selection, status logic, type lookups), see the Portal Syndication Specification ┬¦6.4 (`BayutDubizzleFeedAdapter`).
</Note>

### Integration flow

- The XML feed must be exposed as a **publicly accessible URL**
- The **Bayut system pulls** this URL on a schedule to sync updates
- Listings flow **one direction only**: CRM ŌåÆ portals (the feed is the source of truth)
- Portals never write back into the feed

### Unified feed (Bayut + dubizzle)

Bayut and dubizzle now share a **single combined XML feed**. The portal(s) a property appears on is controlled per-property by the `<Portals>` tag.

<Warning>
  Clients with existing integrations must:
  - **Retire** the full & hourly feeds previously created for dubizzle
  - **Update the Bayut XML structure** to use the `<Portals>` attribute to target each portal
</Warning>

## XML structure

The XML feed consists of two segments:

1. **Header tags** ŌĆö the document envelope
2. **Property attributes** ŌĆö per-property fields

### Root element

The root element must be named `<Properties>`. Every individual property is a nested `<Property>` element inside it.

```xml
<Properties>
  <Property>
    <!-- Property 1 attributes -->
  </Property>
  <Property>
    <!-- Property 2 attributes -->
  </Property>
</Properties>
```

<Tip>
  All textual values should be wrapped in `<![CDATA[ ... ]]>` to prevent HTML, Arabic text, ampersands, and URLs from breaking XML parsing.
</Tip>

## Property attributes

### Property details & attributes

<AccordionGroup>
  <Accordion title="Identity & status">
    | Tag | Description | Rules |
    |-----|-------------|-------|
    | `<Property_Ref_No>` | Unique identifier for the property in your CRM | If duplicates are encountered, only one instance is made live. Duplicates are marked **Inactive** |
    | `<Permit_Number>` | Trakheesi Permit (DLD) or Madhmoun Permit (ADREC) | Required for listing to go live |
    | `<Property_Status>` | Whether the property is live or deleted | `live`, `deleted` |
    | `<Last_Updated>` | Last updated date per your CRM | Format: `YYYY-MM-DD HH:mm:ss` |
  </Accordion>

  <Accordion title="Classification">
    | Tag | Description | Allowed Values |
    |-----|-------------|----------------|
    | `<Property_purpose>` | Purpose of the property | `Buy`, `Rent` |
    | `<Property_Type>` | Type of property | `Villa`, `Apartment`, `Office`, `Shop`, `Warehouse`, `Factory`, `Labour Camp`, `Other Commercial`, `Commercial Building`, `Residential Floor`, `Commercial Floor`, `Residential Land`, `Commercial Land`, `Townhouse`, `Residential Building`, `Hotel Apartment`, `Loft Apartment`, `Pent House` |
    | `<Off_plan>` | Whether the property is ready or off-plan | `Yes`, `No` |
  </Accordion>

  <Accordion title="Specifications">
    | Tag | Description | Rules |
    |-----|-------------|-------|
    | `<Property_Size>` | Numeric area value | Number |
    | `<Property_Size_Unit>` | Unit of measurement | Preferred: `SQFT` |
    | `<plotArea>` | Plot area in square feet | Number |
    | `<Bedrooms>` | Number of bedrooms | `-1` (studio), `1`ŌĆō`10`, `10+` |
    | `<Bathrooms>` | Number of bathrooms | `1`ŌĆō`10` |
    | `<Furnished>` | Furnishing status | `Yes`, `No`, `Partly` |
    | `<Features>` | Property features (see below) | Multiple `<Feature>` child tags |
  </Accordion>

  <Accordion title="Pricing">
    | Tag | Description | Rules |
    |-----|-------------|-------|
    | `<Price>` | Sale price or rent amount | Number |
    | `<Rent_Frequency>` | **Required only for rentals** | `Daily`, `Weekly`, `Monthly`, `Yearly` |
  </Accordion>

  <Accordion title="Content">
    | Tag | Description |
    |-----|-------------|
    | `<Property_Title>` | English title displayed on the ad |
    | `<Property_Description>` | English description for the listing |
    | `<Property_Title_AR>` | Arabic title displayed on the ad |
    | `<Property_Description_AR>` | Arabic description for the listing |
  </Accordion>

  <Accordion title="Location">
    | Tag | Description |
    |-----|-------------|
    | `<City>` | City/Emirate where the property is located |
    | `<Locality>` | Main locality of the property |
    | `<Sub_Locality>` | Sub-locality of the property |
    | `<Tower_Name>` | Building name, if applicable |
  </Accordion>

  <Accordion title="Contact information">
    | Tag | Description |
    |-----|-------------|
    | `<Listing_Agent>` | Agent name displayed on the portals |
    | `<Listing_Agent_Phone>` | Agent contact number |
    | `<Listing_Agent_Email>` | Agent email address |
  </Accordion>
</AccordionGroup>

### Features tag

Use nested `<Feature>` tags to list property features:

```xml
<Features>
  <Feature>Swimming Pool</Feature>
  <Feature>Gym</Feature>
  <Feature>Covered Parking</Feature>
</Features>
```

### Portals tag

The `<Portals>` tag controls which portal(s) the listing is published to. Use one `<portal>` child per portal.

<CodeGroup>
```xml Live on both portals
<Portals>
  <portal>bayut</portal>
  <portal>dubizzle</portal>
</Portals>
```

```xml Live on Bayut only
<Portals>
  <portal>bayut</portal>
</Portals>
```

```xml Live on dubizzle only
<Portals>
  <portal>dubizzle</portal>
</Portals>
```
</CodeGroup>

### Off-plan details (conditional)

These tags are **required only** when `<Off_plan>` is `Yes`.

| Tag | Description | Required When |
|-----|-------------|---------------|
| `<offplanDetails_saleType>` | Type of sale | Always for off-plan (`New`, `Resale`) |
| `<offplanDetails_dldWaiver>` | DLD waivers applicable (0ŌĆō100) | Off-plan **New** properties |
| `<offplanDetails_originalPrice>` | Original property price | Off-plan **Resale** properties |
| `<offplanDetails_amountPaid>` | Amount paid for property | Off-plan **Resale** properties |

<Warning>
  - **New** off-plan properties require `offplanDetails_dldWaiver`
  - **Resale** off-plan properties require both `offplanDetails_originalPrice` and `offplanDetails_amountPaid`
</Warning>

### Images & videos

URLs must be publicly accessible. Use child elements for each media item.

```xml
<Images>
  <Image>https://cdn.example.com/property/155352/photos/img-1.jpg</Image>
  <Image>https://cdn.example.com/property/155352/photos/img-2.jpg</Image>
</Images>

<Videos>
  <Video>https://www.youtube.com/watch?v=example</Video>
</Videos>

<Floor_Plans>
  <Floor_Plan>https://cdn.example.com/property/155352/floorplan.pdf</Floor_Plan>
</Floor_Plans>
```

<Info>
  At least one publicly accessible `<Image>` should be supplied for the listing to display properly.
</Info>

### Location example

For a property in Pretoni Tower, Cluster C, Jumeirah Lakes Towers, Dubai:

```xml
<City>Dubai</City>
<Locality>Jumeirah Lakes Towers</Locality>
<Sub_Locality>Cluster C</Sub_Locality>
<Tower_Name>Pretoni Tower</Tower_Name>
```

## Required & conditional rules

<Steps>
  <Step title="Core requirements">
    - `<Property_Ref_No>` must be **unique** across your feed
    - `<Permit_Number>` is **required** for the listing to go live
    - At least one `<Image>` should be provided
    - `<Property_Status>` must be either `live` or `deleted`
  </Step>

  <Step title="Rental properties">
    `<Rent_Frequency>` is **required only** when `<Property_purpose>` is `Rent`
  </Step>

  <Step title="Off-plan properties">
    When `<Off_plan>` is `Yes`:
    - `<offplanDetails_saleType>` is always required (`New` or `Resale`)
    - If `New`: `<offplanDetails_dldWaiver>` is required
    - If `Resale`: Both `<offplanDetails_originalPrice>` and `<offplanDetails_amountPaid>` are required
  </Step>

  <Step title="Duplicate handling">
    If duplicate `<Property_Ref_No>` values are encountered, only one instance is made live. The rest are marked **Inactive**.
  </Step>
</Steps>

## PropWise implementation notes

<Note>
  The values above define the **external contract**. PropWise derives them from internal entities as documented in the Portal Syndication Specification ┬¦6.4.
</Note>

### Key mappings

| XML Field | PropWise Source |
|-----------|-----------------|
| `Property_Ref_No` | Generated as `UNIT-{orgShortCode}-{listing.id}` ŌĆö unique per organization |
| `Property_Status` | `live` when `ListingPortalSync` is published/pending; `deleted` when removed/disabled |
| `Property_Type` | Resolved from `Listing.propertyType` via `LAYOUT_TYPE_TO_BAYUT` mapping |
| `Portals` | Emitted per active `ListingPortalSync` (Bayut and/or dubizzle) |
| `Bedrooms` | String conversion: `0` ŌåÆ `-1` (studio), `>10` ŌåÆ `10+`, `null` ŌåÆ omitted |
| `Furnished` | Maps from `InventoryUnit.furnished`: `FURNISHED` ŌåÆ `Yes`, `UNFURNISHED` ŌåÆ `No`, `PARTLY_FURNISHED` ŌåÆ `Partly` |
| `Off_Plan` | Currently `No` (inventory = secondary market); off-plan syndication reserved for future `OffPlanUnit` integration |
| `Last_Updated` | Latest of listing/sync `updatedAt`, formatted `YYYY-MM-DD HH:mm:ss` |

<Tip>
  Deleted entries remain in the feed for at least one crawl cycle so the portal can delist them promptly.
</Tip>

### Known casing inconsistencies

<Warning>
  The Bayut specification contains minor inconsistencies that PropWise resolves as follows:
  
  - **Off-plan tag:** Specification shows `<Off_plan>`, but examples use `<Off_Plan>` with capital `P`. PropWise emits `<Off_Plan>`.
  - **Portals children:** Some examples use `<portal>bayut</portal>` (lowercase), others use `<Portal><![CDATA[ Bayut ]]></Portal>`. PropWise emits the capitalized form with CDATA.
  - **Bathrooms:** Specification uses `<Bathrooms>` (plural); avoid the singular `<Bathroom>` form. PropWise emits `<Bathrooms>`.
</Warning>
