**Path:** `src/modules/real-estate/inventory/`

The inventory module manages org-scoped secondary units (existing market properties that have been bought and sold) within PropWise's **six-module real estate system**. It provides unit number privacy controls and marketplace listing functionality while integrating with the reference database architecture.

## Architecture context

Inventory is one of six real estate modules spanning three visibility scopes:

<Tabs>
<Tab title="Global modules">
- **reference-data**: All property data (structural + enrichment)
- **developer-portal**: Auth + project management
- **map**: Geocoding utility (shared across modules)
</Tab>
<Tab title="Org-scoped modules">
- **inventory**: Secondary units (this module)
- **property-interest**: Lead/deal property interests + matching
</Tab>
<Tab title="Cross-org modules">
- **shared-marketplace**: Cross-organization listings + deals
</Tab>
</Tabs>

### Data ownership pattern

<Info>
Inventory follows PropWise's **cross-database reference pattern**: CRM entities store plain integer IDs (`refUnitId`, `refProjectId`) pointing to the reference database, with no cross-database foreign keys. All property details come through `ReferenceDataService`.
</Info>

| Data Type | Database | Access Pattern |
|-----------|----------|----------------|
| Property data (projects, units, buildings, areas) | Reference DB | `ReferenceDataService` |
| Gallery, payment plans, documents, amenities | Reference DB | `ReferenceDataService` |
| Geocoding, pricing, sale status, handover | Reference DB | `ReferenceDataService` |
| Secondary units (inventory records) | CRM DB | Direct entity access |
| Unit ownership, transactions, privacy settings | CRM DB | Direct entity access |

### Scope classification

- **Scope**: Org-scoped
- **RLS Behavior**: Standard RLS via TenantContext
- **Description**: Secondary units owned by an organization

## Controller structure

The inventory module implements a single controller with comprehensive endpoint coverage:

<Tab title="InventoryController">

@Controller('inventory')
@UseGuards(AuthGuard, AccessGuard)
export class InventoryController {
  // Core CRUD operations
  @Get('units')

  @Post('units')

  // Marketplace operations
  @Post('units/:id/list-on-marketplace')
  @Post('units/:id/delist-from-marketplace')
}

## Service layer implementation

### InventoryService

Core business logic service handling:

- **CRUD operations**: Create, read, update, delete secondary units with org-scoping
- **Reference data integration**: Fetching property details via `ReferenceDataService`
- **Data denormalization**: Storing property names for performance
- **Marketplace operations**: Enable/disable marketplace listing status
- **Ownership management**: Handle unit ownership records and transactions
- **Filtering and pagination**: Query building with multiple filter criteria

### InventorySecurityService

Specialized service for unit number privacy and access control:

```typescript
canViewUnitNumber(user, unit): boolean
// Returns true if:
// - Unit number is not hidden (isUnitNumberHidden === false)
// - User has 'real_estate.manage' permission
// - User is the unitManager
// - User has active UnitAccessGrant (via shared-marketplace)

getVisibilityLevel(user, unit): 'full' | 'basic'
```

<Warning>
Always pass `canViewUnitNumber` to the DTO constructor. Forgetting this parameter defaults to `true`, which could expose hidden unit numbers.
</Warning>

## Entities

<AccordionGroup>
<Accordion title="SecondaryUnit">

**Primary Entity** - UUID PK, org-scoped with standard RLS

- **Reference links**: `refUnitId`, `refBuildingId`, `refProjectId`, `refAreaId`, `refCommunityId` (plain integers)
- **Denormalized data**: `projectName`, `buildingName`, `areaName`, `communityName` (performance optimization)
- **Unit details**: `unitNumber`, `floor`, `rooms`, `propertySubType`, `areaSqft`, `isFreeHold`
- **Financial data**: `askingPrice`, `rentPrice`, `currency`, `availableForSale`, `availableForRent`
- **Status management**: `SecondaryUnitStatus` enum (VACANT, RENTED, OWNER_OCCUPIED)
- **Privacy controls**: `isUnitNumberHidden` (default: true)
- **Marketplace integration**: `isListedOnMarketplace` (default: false)
- **Relationships**: `owner` (Contact), `unitManager` (User), `organization`, `createdBy`
- **Additional fields**: `description`, `furnishingType`, `parkingSpaces`, `balconyArea`, `viewType`
- **Timestamps**: `createdAt`, `updatedAt`, `deletedAt` (soft delete)

</Accordion>
<Accordion title="InventoryUnitOwnership">

**Ownership tracking** - Custom table name `inventory_unit_ownership`

- **Primary relationship**: Links Person to SecondaryUnit
- **Ownership details**: `ownershipPercentage` (numeric), `ownershipType` enum
- **Date tracking**: `startDate`, `endDate` for ownership periods
- **Documentation**: `documentType`, `documentNumber`, `registrationDate`
- **Status**: `isActive` boolean flag

</Accordion>
<Accordion title="InventoryUnitTransaction">

**Transaction history** - Custom table name `inventory_unit_transaction`

- **Transaction core**: `date`, `value`, `currency`
- **Process details**: `procedureName`, `transactionType` enum
- **Parties**: `buyerName`, `sellerName`, `agentName`
- **Documentation**: `contractNumber`, `registrationNumber`
- **Additional costs**: `commission`, `fees`, `taxes`

</Accordion>

## API endpoints with detailed schemas

### List units
```http
GET /inventory/units?page=1&limit=20&search=...&status=...&propertyType=...
```

**Query parameters:**
- `page`, `limit`: Pagination
- `search`: Text search across unit/project/building names
- `status`: Filter by `SecondaryUnitStatus`
- `propertyType`: Filter by property sub-type
- `availableForSale`, `availableForRent`: Availability filters
- `priceFrom`, `priceTo`: Price range filters
- `areaFrom`, `areaTo`: Area range filters

**Response:** Paginated list with unit visibility applied per user permissions.

### Create unit
```http
POST /inventory/units
```

**Request body:**
```json
{
  "refUnitId": 12345,
  "refProjectId": 678,
  "unitNumber": "A-101",
  "askingPrice": 850000,
  "currency": "AED",
  "availableForSale": true,
  "isUnitNumberHidden": true
}
```

### Unit detail
```http
GET /inventory/units/:id
```

**Response:** Full unit details with reference data populated, unit number visibility applied.

### Marketplace operations
```http
POST /inventory/units/:id/list-on-marketplace
POST /inventory/units/:id/delist-from-marketplace
```

## Reference database integration

Inventory uses the **ReferenceDataService pattern** to consume property data:

```typescript
// CRM entity storing reference to external property data
@Property({ type: 'int', nullable: true })
@Index()
refProjectId?: number;  // ID in reference DB projects table

@Property({ type: 'int', nullable: true })
@Index()
refUnitId?: number;     // ID in reference DB units table
```

<Note>
If the reference database is replaced with an external property API, only `ReferenceDataService` needs to change — inventory module remains unchanged.
</Note>

## Unit number privacy

The `InventorySecurityService` controls unit number visibility:

```typescript
canViewUnitNumber(user, unit): boolean
// Returns true if:
// - Unit number is not hidden (isUnitNumberHidden === false)
// - User has 'real_estate.manage' permission
// - User is the unitManager
// - User has active UnitAccessGrant (via shared-marketplace)

getVisibilityLevel(user, unit): 'full' | 'basic'
```

The `SecondaryUnitDto` constructor accepts `canViewUnitNumber` boolean; if false, `unitNumber` is masked as `'***'`.

## Access control

Inventory is **org-scoped** and uses `@UseGuards(AuthGuard, AccessGuard)` with `@CheckAccess`:

- **List/Create**: `permissions: ['real_estate.view', 'real_estate.manage']` / `['real_estate.manage']`
- **View/Update/Delete**: Permission OR Resource check with `ResourceType.UNIT`
- **Marketplace operations**: Permission OR Resource check with `ResourceAction.MODIFY`

## Transaction and ownership management

### Ownership tracking workflow

<Steps>
<Step title="Create ownership record">
When a secondary unit is created, establish initial ownership via `InventoryUnitOwnership` with ownership percentage and type.
</Step>
<Step title="Record transactions">
All buy/sell activities are tracked in `InventoryUnitTransaction` with complete transaction details including parties, values, and documentation.
</Step>
<Step title="Update ownership">
Ownership transfers automatically update the ownership records with new percentages and parties, maintaining complete audit trail.
</Step>
</Steps>

### Transaction types supported

- **PURCHASE**: Initial acquisition of the unit
- **SALE**: Sale to new owner
- **TRANSFER**: Ownership transfer (gift, inheritance)
- **MORTGAGE**: Mortgage-related transactions
- **REFINANCE**: Refinancing operations

## Usage workflow with reference database

<Steps>
<Step title="Search projects via ReferenceDataService">
Frontend calls `/reference/projects?q=...` to search projects from the reference database.
</Step>
<Step title="Select building">
User selects project, frontend calls `/reference/projects/:id/buildings` to get nested buildings.
</Step>
<Step title="Select unit">
User selects building, frontend calls `/reference/buildings/:id/units` to get available units.
</Step>
<Step title="Create inventory record">
User selects unit, unit details are pre-filled from reference data. Plain integer IDs (`refUnitId`, `refProjectId`, etc.) are stored in the CRM database and property names are denormalized for performance.
</Step>
<Step title="Set privacy and marketplace preferences">
Configure unit number visibility (`isUnitNumberHidden`) and marketplace listing status (`isListedOnMarketplace`) based on organization preferences.
</Step>
<Step title="Manage ownership and transactions">
Record ownership details via `InventoryUnitOwnership` and track all related transactions through `InventoryUnitTransaction` for complete financial history.
</Step>
</Steps>

## Integration examples

<CodeGroup>
```typescript TypeScript - Creating inventory unit
// Service method example
async createSecondaryUnit(data: CreateSecondaryUnitDto, user: User) {
  // Fetch reference data
  const refUnit = await this.referenceDataService.getUnit(data.refUnitId);
  
  // Create with denormalized data
  const unit = this.em.create(SecondaryUnit, {
    ...data,
    projectName: refUnit.project.name,
    buildingName: refUnit.building.name,
    organization: user.organization,
    createdBy: user,
  });
  
  await this.em.flush();
  return unit;
}
```

```typescript TypeScript - Applying unit privacy
// DTO construction with privacy
const dto = new SecondaryUnitDto(
  unit,
  this.inventorySecurityService.canViewUnitNumber(user, unit)
);
// Unit number will be masked if canViewUnitNumber returns false
```

```typescript TypeScript - Marketplace operations
// Enable marketplace listing
async listOnMarketplace(unitId: string, user: User) {
  const unit = await this.findOneOrFail(unitId);
  
  unit.isListedOnMarketplace = true;
  unit.listedAt = new Date();
  
  await this.em.flush();
  
  // Trigger shared marketplace sync
  await this.sharedMarketplaceService.syncListing(unit);
}
```
