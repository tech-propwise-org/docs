**Path:** `src/modules/real-estate/inventory/`

The inventory module manages org-scoped secondary units within PropWise's **six-module real estate system**. It serves as one of the org-scoped modules that uses standard RLS (Row Level Security) via TenantContext.

<Note>
This module is part of PropWise's comprehensive real estate specification that spans six specialized modules. The complete implementation details are being developed as part of the broader real estate architecture.
</Note>

## Architecture context

Inventory is one of six real estate modules spanning three visibility scopes:

<Tabs>
<Tab title="Global modules">
- **reference-data**: All property data (structural + enrichment) — 3 controllers (general, projects, buildings)
- **developer-portal**: Auth + profile + project/building management via ref DB  
- **map**: MapService (2GIS geocoding) — shared across modules
</Tab>
<Tab title="Org-scoped modules">
- **inventory**: Secondary units (this module) — standard RLS
- **property-interest**: Lead/deal property interests + matching engine — standard RLS
</Tab>
<Tab title="Cross-org modules">
- **shared-marketplace**: Cross-organization listings + deals — custom RLS
</Tab>
</Tabs>

### Data ownership pattern

<Info>
Inventory follows PropWise's **cross-database reference pattern**: CRM entities store plain integer IDs (`refUnitId`, `refProjectId`) pointing to the reference database, with no cross-database foreign keys. All property details come through `ReferenceDataService`.
</Info>

| Data Type | Database | Access Pattern |
|-----------|----------|----------------|
| Projects, units, buildings, areas, communities, developers | Reference DB | `ReferenceDataService` |
| Gallery images, payment plans, documents, amenities | Reference DB | `ReferenceDataService` |
| Geocoding (lat/lng), pricing, sale status, handover | Reference DB | `ReferenceDataService` |
| Developer auth (DeveloperUser) | CRM DB | Direct entity access |
| Developer-to-project ownership (DeveloperProjectLink) | CRM DB | Direct entity access |
| **Secondary units (org-scoped inventory)** | **CRM DB** | **Direct entity access** |
| Marketplace listings & deals (SharedListing, SharedDeal) | CRM DB | Direct entity access |
| Property interests (Lead/Deal) | CRM DB | Direct entity access, ref IDs as plain ints |
| Property matches | CRM DB | Direct entity access, ref IDs as plain ints |

## Module entities

### SecondaryUnit entity

The core entity for org-scoped inventory management:

```typescript
@Entity({ tableName: 'secondary_units' })
@UseRequestContext()
export class SecondaryUnit extends BaseUuidEntity {
  @Property()
  name!: string;

  @Property({ nullable: true })
  description?: string;

  // Cross-database references (plain integer IDs)
  @Property({ type: 'int', nullable: true })
  @Index()
  refProjectId?: number;  // ID in reference DB projects table

  @Property({ type: 'int', nullable: true })
  @Index()
  refUnitId?: number;     // ID in reference DB units table

  // Organization scoping (standard RLS)
  @ManyToOne(() => Organization, { fieldName: 'organization_id' })
  @Index()
  organization!: Organization;

  @Property({ fieldName: 'organization_id' })
  organizationId!: string;

  // Unit-specific data
  @Enum(() => UnitStatus)
  status!: UnitStatus;

  @Property({ type: 'numeric', nullable: true })
  price?: number;

  @Property({ type: 'numeric', nullable: true })
  area?: number;

  @Property({ nullable: true })
  bedrooms?: number;

  @Property({ nullable: true })
  bathrooms?: number;

  // Metadata
  @Property({ nullable: true })
  notes?: string;

  @Property({ type: 'datetime', defaultRaw: 'CURRENT_TIMESTAMP' })
  createdAt!: Date;

  @Property({ type: 'datetime', defaultRaw: 'CURRENT_TIMESTAMP', onUpdate: () => new Date() })
  updatedAt!: Date;
}
```

## Cross-database reference pattern

<Note>
The inventory module demonstrates PropWise's architectural pattern for decoupling CRM data from external property databases.
</Note>

CRM entities that reference off-plan projects or units store **plain integer IDs** (`refProjectId`, `refUnitId`) pointing to the reference database. No cross-database foreign keys exist — this decouples the CRM from the property data source.

```typescript
// CRM entity referencing reference DB data:
@Property({ type: 'int', nullable: true })
@Index()
refProjectId?: number;  // ID in reference DB projects table

@Property({ type: 'int', nullable: true })
@Index()
refUnitId?: number;     // ID in reference DB units table
```

<Tip>
If the reference database is replaced with an external property API, only `ReferenceDataService` needs to change — the inventory module remains unchanged.
</Tip>
