**Path:** `src/modules/real-estate/inventory/`

Org-scoped secondary units (existing market properties that have been bought and sold). Supports unit number privacy and marketplace listing.

## Entities

<AccordionGroup>
<Accordion title="SecondaryUnit">

- UUID PK, **org-scoped** (ManyToOne Organization, standard RLS)
- Reference DB links: refUnitId, refBuildingId, refProjectId, refAreaId, refCommunityId
- Denormalized names: projectName, buildingName, areaName, communityName
- Unit details: unitNumber, floor, rooms, propertySubType, areaSqft, isFreeHold
- Pricing: askingPrice, rentPrice, currency, availableForSale, availableForRent
- Status: SecondaryUnitStatus (VACANT, RENTED, OWNER\_OCCUPIED)
- Privacy: **isUnitNumberHidden** (default true)
- Marketplace: **isListedOnMarketplace** (default false)
- Relations: owner (Contact), unitManager (User), organization, createdBy

</Accordion>
<Accordion title="InventoryUnitOwnership">

- Custom table name `inventory_unit_ownership`
- Links Person to SecondaryUnit with ownershipPercentage and type

</Accordion>
<Accordion title="InventoryUnitTransaction">

- Custom table name `inventory_unit_transaction`
- Transaction history: date, value, procedureName, transactionType

</Accordion>
</AccordionGroup>

## Unit number privacy

The `InventorySecurityService` controls unit number visibility:

```typescript
canViewUnitNumber(user, unit): boolean
// Returns true if:
// - Unit number is not hidden (isUnitNumberHidden === false)
// - User has 'real_estate.manage' permission
// - User is the unitManager
// - User has active UnitAccessGrant (via shared-marketplace)

getVisibilityLevel(user, unit): 'full' | 'basic'
```

The `SecondaryUnitDto` constructor accepts `canViewUnitNumber` boolean; if false, `unitNumber` is masked as `'***'`.

<Warning>
Always pass `canViewUnitNumber` to the DTO constructor. Forgetting this parameter defaults to `true`, which could expose hidden unit numbers.
</Warning>

## Access control

Inventory is **org-scoped** and uses `@UseGuards(AuthGuard, AccessGuard)` with `@CheckAccess`:

- **List/Create**: `permissions: ['real_estate.view', 'real_estate.manage']` / `['real_estate.manage']`
- **View/Update/Delete**: Permission OR Resource check with `ResourceType.UNIT`
- **Marketplace operations**: Permission OR Resource check with `ResourceAction.MODIFY`

## API endpoints

```
GET    /inventory/units                       # Paginated list with filters
POST   /inventory/units                       # Create unit
GET    /inventory/units/:id                   # Detail (unit number visibility checked)
PATCH  /inventory/units/:id                   # Update
DELETE /inventory/units/:id                   # Soft delete
POST   /inventory/units/:id/list-on-marketplace    # Enable marketplace listing
POST   /inventory/units/:id/delist-from-marketplace # Disable
```

## Usage with reference DB

<Steps>
<Step title="Search projects">
Frontend calls `/reference/projects?q=...` to search projects.
</Step>
<Step title="Select building">
User selects project, frontend calls `/reference/projects/:id/buildings`.
</Step>
<Step title="Select unit">
User selects building, frontend calls `/reference/buildings/:id/units`.
</Step>
<Step title="Pre-fill and save">
User selects unit, unit details are pre-filled from reference data. `refUnitId`, `refProjectId`, etc. are stored and names are denormalized.
</Step>
</Steps>
