## Overview

The `portal-syndication` module manages the syndication of property listings to external real estate portals like Property Finder, Bayut, and Dubizzle. This module acts as a bridge between the internal property inventory system and external marketing platforms, handling authentication, data transformation, and synchronization status tracking.

<Note>
**Module Path:** `src/modules/real-estate/portal-syndication/`

The module is part of `RealEstateModule` and is imported from `src/modules/real-estate/real-estate.module.ts`.
</Note>

## Architecture

The module follows a layered architecture with clear separation between different concerns:

<CardGroup cols={2}>
  <Card title="Listings" icon="rectangle-list">
    Marketing layer that wraps inventory units with portal-specific metadata
  </Card>
  <Card title="Portal Configuration" icon="gear">
    Per-organization portal credentials and settings management
  </Card>
  <Card title="Sync Tracking" icon="arrows-rotate">
    Per-listing, per-portal synchronization state management
  </Card>
  <Card title="Adapters" icon="plug">
    Portal-specific integration logic (future implementation)
  </Card>
</CardGroup>

### Design decisions

<AccordionGroup>
  <Accordion title="Separation of concerns">
    Listings are separate from inventory units to allow marketing-specific data and metadata that doesn't belong in the core inventory system.
  </Accordion>
  
  <Accordion title="Per-portal tracking">
    Each listing maintains independent sync state per portal, allowing granular control over syndication status and error handling.
  </Accordion>
  
  <Accordion title="Encrypted credentials">
    Portal API keys are stored encrypted using AES-256-GCM encryption to ensure security of sensitive authentication data.
  </Accordion>
  
  <Accordion title="Mapping tables">
    Cached lookups for portal-specific identifiers (agents, locations) to improve performance and reduce external API calls.
  </Accordion>
</AccordionGroup>

Represents a property unit prepared for marketing across portals.

```typescript
interface Listing {
  id: string;
  title: string;
  description: string;
  price: number;
  location: string;
  bedrooms: number;
  bathrooms: number;
  size: number;
  amenities: string[];
  images: string[];
  furnishedStatus: FurnishedStatus;
  rentalPeriod?: RentalPeriod;
  finishingType: FinishingType;

  inventoryUnit: string; // FK to inventory
  createdBy: string; // FK to User
```

### Portal configuration

Stores per-organization portal credentials and settings.

```typescript
interface PortalConfiguration {
  id: string;
  portal: Portal;
  isEnabled: boolean;
  apiKey: string; // Encrypted
  apiSecret?: string; // Encrypted
  organizationId?: string; // Portal-specific org ID
  portalCredits?: number;
  lastCreditsCheckAt?: Date;
  organization: string; // FK to Organization
  createdAt: Date;
  updatedAt: Date;

```

### Listing portal sync

Tracks sync state for each listing on each portal.

```typescript
interface ListingPortalSync {
  id: string;
  portal: Portal;
  isEnabled: boolean;
  status: ListingSyncStatus;
  portalListingId?: string;
  lastSyncedAt?: Date;

  lastPublishCost?: {
    creditCost: number;
    checkedAt: Date;
  };
  listing: string; // FK to Listing
  organization: string; // FK to Organization
  createdAt: Date;
  updatedAt: Date;
}
```

## API Endpoints

### Listing controller

#### List listings

<CodeGroup>
```http GET /listings
GET /listings?page=1&limit=10&status=active&search=villa
```

```typescript Response
{
  data: ListingDto[];
  meta: {
    totalItems: number;
    currentPage: number;
    totalPages: number;
  };
}
```
</CodeGroup>

**Query Parameters:**
- `page?: number` - Page number for pagination
- `limit?: number` - Items per page
- `status?: ListingStatus` - Filter by listing status
- `search?: string` - Search in title and description

#### Create listing

<CodeGroup>
```http POST /listings
POST /listings
Content-Type: application/json
```

```json Request Body
{
  "title": "Luxury Villa in Downtown",
  "description": "Beautiful 3-bedroom villa...",
  "price": 150000,
  "location": "Downtown Dubai",
  "bedrooms": 3,
  "bathrooms": 2,
  "size": 2500,
  "amenities": ["parking", "gym", "pool"],
  "images": ["image1.jpg", "image2.jpg"],
  "furnishedStatus": "furnished",
  "rentalPeriod": "yearly",
  "finishingType": "luxury",
  "inventoryUnitId": "unit-123"
}
```
</CodeGroup>

#### Update listing

```http
PATCH /listings/:id
```

#### Delete listing

```http
DELETE /listings/:id
```

<Warning>
This performs a soft delete by setting the status to `ARCHIVED`. The listing record is preserved for historical tracking.
</Warning>

#### Get portal sync status

```http
GET /listings/:id/portal-sync
```

## Business Rules

### Listing rules

<Steps>
  <Step title="Inventory association">
    Each listing must be associated with a valid inventory unit from the inventory system.
  </Step>
  
  <Step title="Active status requirement">
    Only active listings can be synced to external portals. Draft or archived listings are excluded.
  </Step>
  
  <Step title="Price validation">
    Price must be a positive number greater than zero.
  </Step>
  
  <Step title="Image requirement">
    At least one image is required for portal syndication to meet external platform requirements.
  </Step>
</Steps>

### Portal configuration rules

<Check>
**One configuration per portal** - Each organization can have exactly one configuration per portal type.
</Check>

<Check>
**Encrypted credentials** - API credentials are automatically encrypted before storage using AES-256-GCM.
</Check>

<Check>
**Enable requirement** - Portal must be enabled to sync listings to that platform.
</Check>

<Check>
**Credit monitoring** - Credit balance is checked before Property Finder submissions to prevent failed publishes.
</Check>

### Sync rules

- **Unique sync records** - Each listing can have one sync record per portal
- **Status workflows** - Sync status transitions follow portal-specific workflows
- **Error tracking** - Failed syncs store detailed error messages for debugging
- **Cost tracking** - Sync costs are recorded for billing and analytics purposes

## Patterns and Conventions

### Enums

- **Portal**: Supported external portals (`property_finder`, `bayut`, `dubizzle`)
- **ListingStatus**: Listing lifecycle states (`draft`, `active`, `archived`)
- **ListingSyncStatus**: Per-portal sync states (`pending`, `synced`, `failed`, `disabled`)
- **FurnishedStatus**: Property furnishing states (`furnished`, `semi_furnished`, `unfurnished`)
- **RentalPeriod**: Rental frequency options (`daily`, `weekly`, `monthly`, `yearly`)
- **FinishingType**: Property finishing levels (`basic`, `standard`, `luxury`)

### Data transfer objects

<Tip>
The module uses comprehensive validation with class-validator decorators, separate DTOs for different operations, and nested validation for complex objects to ensure data integrity.
</Tip>

### Security

<Steps>
  <Step title="Authentication">
    All endpoints are protected by `AuthGuard` and `AccessGuard` for proper authentication and authorization.
  </Step>
  
  <Step title="Row-level security">
    Organization filtering ensures users only access data within their organization boundary.
  </Step>
  
  <Step title="Credential encryption">
    Sensitive portal credentials are encrypted at rest using industry-standard encryption.
  </Step>
  
  <Step title="API documentation">
    All endpoints include bearer token authentication requirements in API documentation.
  </Step>
</Steps>

### Error handling

The module implements comprehensive error handling with:

- **Structured error responses** for consistent API behavior
- **Sync failure tracking** stored in the database for debugging
- **Graceful degradation** when external portal connectivity issues occur
- **Detailed logging** for troubleshooting integration problems

<Warning>
Always check portal sync status before assuming a listing is successfully published to external platforms.
</Warning>
