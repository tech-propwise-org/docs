This document provides a comprehensive side-by-side reference for how canonical `Listing` fields map to both Bayut/dubizzle and Property Finder portals, including field name differences, allowed value variations, and portal-specific fields for frontend visibility control.

<Note>
This matrix serves as the cross-portal divergence and frontend visibility companion to the per-portal payload tables in `PORTAL_SYNDICATION_SPECIFICATION.md` sections 6.3 (PF) and 6.4 (Bayut).
</Note>

## Authentication & account linking

None of the three portals use interactive OAuth "Connect with..." redirects. All use **manual credential/URL exchange**, configured per-organization by admin/owner.

<Tabs>
<Tab title="Property Finder">
**Direction:** Push + webhooks  
**Admin provides:** API Key + API Secret + expiry (from PF Expert)  
**PropWise generates:** `webhookSecret`  
**Transport auth:** OAuth2 client-credentials → 30-min Bearer JWT
</Tab>
<Tab title="Bayut">
**Direction:** Feed pull + lead poll  
**Admin provides:** Bayut Pull API Bearer token (inbound leads only)  
**PropWise generates:** `feedSecret` + per-org `feedUrl` (outbound listings)  
**Transport auth:** Org registers PropWise's feed URL; PropWise polls leads with Bearer token
</Tab>
<Tab title="dubizzle">
**Direction:** Shares Bayut  
**Admin provides:** Nothing new — piggybacks Bayut  
**PropWise generates:** Reuses the unified feed  
**Transport auth:** Shares Bayut feed + Bayut lead token
</Tab>
</Tabs>

### Common authentication model

<Steps>
<Step title="Portal configuration">
One `PortalConfiguration` row per `(organization, portal)` with unique constraint
</Step>
<Step title="RBAC requirement">
Requires `REAL_ESTATE_MANAGE` permission (org admin/owner)
</Step>
<Step title="Available endpoints">
- `GET /portal-syndication/config` — list configurations
- `POST /portal-syndication/config` — upsert configuration
- `PATCH /portal-syndication/config/:portal/toggle` — enable/disable
</Step>
</Steps>

<Warning>
API credentials are encrypted at rest (AES-256-GCM) and raw values are never returned in responses or logs.
</Warning>

### Property Finder OAuth2 flow

<Steps>
<Step title="Generate credentials in PF Expert">
Admin opens **Developer Resources → API Credentials**, generates **API Integration** type key, receives API Key + Secret, sets expiry (max 365 days), enables required optional scopes: `listings:full_access`, `leads:read`, `credits:read`
</Step>
<Step title="Configure in PropWise">
Admin enters API Key + Secret + expiry via `POST /config` with `portal=property_finder`
</Step>
<Step title="Runtime token exchange">
PropWise exchanges key+secret at `POST /v1/auth/token` for 30-minute Bearer JWT (no refresh token)
</Step>
<Step title="Webhook setup">
PropWise auto-generates `webhookSecret` and subscribes to PF webhooks with HMAC `X-Signature` verification
</Step>
</Steps>

### Bayut dual-channel setup

<Tabs>
<Tab title="Outbound (listings)">
- PropWise generates per-org HMAC feed URL
- Admin copies `feedUrl` and registers in Bayut XML feed settings
- Bayut pulls feed on schedule
</Tab>
<Tab title="Inbound (leads)">
- Bayut issues Pull API Bearer token
- Admin enters token in PropWise as encrypted `apiKey`
- `BayutLeadPollerService` polls every 15 minutes
</Tab>
</Tabs>

### Feed URL architecture

Each organization gets its own feed URL:

```
GET /portal-syndication/feeds/{orgId}?token={hmac}
```

<Info>
The feed is unified - one endpoint returns ALL org listings with `<Portals>` tag determining visibility per portal based on enabled `ListingPortalSync` rows.
</Info>

## Mapping helper inventory

| Concern | Helper | Status |
|---------|--------|--------|
| Property type → portal value | `LAYOUT_TYPE_TO_BAYUT`, `LAYOUT_TYPE_TO_PF_SLUG` | ✅ Centralized |
| Purpose, furnished, bedrooms, bathrooms, rental period, finishing, emirate→compliance, projectStatus | `portal-value-map.ts` | ✅ Built |

<CodeGroup>
```typescript Purpose mapping
purposeToBayut(p)        // SALE→'Buy',  RENT→'Rent'
purposeToPfPriceType(p, rentalPeriod)  // SALE→'sale', RENT→'yearly'|'monthly'|'weekly'|'daily'
```

```typescript Furnished mapping
furnishedToBayut(f)      // FURNISHED→'Yes', UNFURNISHED→'No', PARTLY_FURNISHED→'Partly'
furnishedToPf(f)         // FURNISHED→'furnished', UNFURNISHED→'unfurnished', PARTLY_FURNISHED→'semi-furnished'
```

```typescript Bedroom mapping
bedroomsToBayut(n)       // 0→'-1', 1..10→'1'..'10', >10→'10+', null→omit
bedroomsToPf(n)          // 0→'studio', 1..30→'1'..'30' (cap 30)
```
</CodeGroup>

## Field name divergence

The same logical data uses different field names across portals:

| Canonical (`Listing`) | Bayut XML tag | PF JSON field | Notes |
|-----------------------|---------------|---------------|-------|
| `id` (+ org short code) | `<Property_Ref_No>` | `reference` | `UNIT-{orgShortCode}-{listing.id}` |
| `permitNumber` | `<Permit_Number>` | `compliance.listingAdvertisementNumber` | PF may be composite |
| — (org license) | — | `compliance.issuingClientLicenseNumber` | PF only |
| `purpose` | `<Property_purpose>` | `price.type` | Value + concept differ |
| `propertyType` | `<Property_Type>` | `type` | Different value maps |
| `price` | `<Price>` | `price.amounts.{sale\|yearly\|monthly\|weekly\|daily}` | PF splits by price type |
| `rentalPeriod` | `<Rent_Frequency>` | folded into `price.type` + `price.amounts` | Bayut separate frequency |
| `bedrooms` | `<Bedrooms>` | `attributes.bedrooms` | Different value handling |
| `bathrooms` | `<Bathrooms>` | `attributes.bathrooms` | Different constraints |
| `size` | `<Size>` | `attributes.area` | — |
| `furnished` | `<Furnished>` | `attributes.furnished` | Different value sets |
| `title` | `<Title_En>` | `title.en` | — |
| `description` | `<Description_En>` | `description.en` | — |
| `lat`, `lng` | `<Latitude>`, `<Longitude>` | `location.coordinates` | Bayut separate, PF array |

## Value space divergence

<AccordionGroup>
<Accordion title="Purpose field mapping">
**Concept difference:** Bayut has transaction type, PF embeds in price structure

| Canonical | Bayut (`Property_purpose`) | PF (`price.type`) | PF impact |
|-----------|---------------------------|-------------------|-----------|
| `SALE` | `Buy` | `sale` | Uses `price.amounts.sale` |
| `RENT` | `Rent` | `yearly`/`monthly`/`weekly`/`daily` | Uses corresponding `price.amounts.*` |

<Note>
PF rental frequency is determined by `rentalPeriod` field, affecting which `price.amounts` field is populated.
</Note>
</Accordion>

<Accordion title="Furnished field mapping">
| Canonical | Bayut | PF |
|-----------|-------|-----|
| `FURNISHED` | `Yes` | `furnished` |
| `UNFURNISHED` | `No` | `unfurnished` |
| `PARTLY_FURNISHED` | `Partly` | `semi-furnished` |
| `null` | omit | omit |
</Accordion>

<Accordion title="Bedroom count mapping">
| Canonical | Bayut | PF |
|-----------|-------|-----|
| `0` | `-1` | `studio` |
| `1` | `1` | `1` |
| `2-10` | `2`-`10` | `2`-`30` |
| `>10` | `10+` | capped at `30` |
| `null` | omit | omit |

<Warning>
Bayut caps at 10+ while PF accepts up to 30 individual values.
</Warning>
</Accordion>
</AccordionGroup>

## Portal-specific fields

These fields only apply to specific portals and should be hidden/shown based on enabled integrations:

<CardGroup cols={2}>
<Card title="Property Finder only" icon="building">
- `compliance.issuingClientLicenseNumber`
- `finishing` (fully-finished, semi-finished, etc.)
- `emirate` → `compliance` mapping (RERA/DTCM/ADREC)
- Flexible rental periods in price structure
</Card>

<Card title="Bayut/dubizzle only" icon="home">
- Separate `Rent_Frequency` field
- `10+` bedroom cap notation
- Simpler furnished values (`Yes`/`No`/`Partly`)
- Single price field regardless of purpose
</Card>
</CardGroup>
