This document provides a comprehensive side-by-side reference for how canonical `Listing` fields map to both Bayut/dubizzle and Property Finder portals, covering field name differences, value transformations, and portal-specific visibility rules.

<Note>
This is the cross-portal divergence and frontend visibility companion to the full portal syndication specification. Field names and allowed values differ significantly between portals for the same logical data.
</Note>

## Authentication & Account Linking

None of the three portals use interactive OAuth "Connect with..." redirects. All require manual credential/URL exchange configured per organization by an admin/owner.

<Tabs>
  <Tab title="Property Finder">
    **Direction:** Push + webhooks  
    **Admin provides:** API Key + API Secret + expiry (from PF Expert)  
    **PropWise generates:** `webhookSecret`  
    **Transport auth:** OAuth2 client-credentials → 30-min Bearer JWT
  </Tab>
  <Tab title="Bayut">
    **Direction:** Feed pull + lead poll  
    **Admin provides:** Bayut Pull API Bearer token (inbound leads only)  
    **PropWise generates:** `feedSecret` + per-org `feedUrl` (outbound listings)  
    **Transport auth:** Org registers PropWise's feed URL; PropWise polls leads with Bearer token
  </Tab>
  <Tab title="dubizzle">
    **Direction:** Shares Bayut  
    **Admin provides:** Nothing new (piggybacks Bayut)  
    **PropWise generates:** Reuses unified feed  
    **Transport auth:** Shares Bayut feed + Bayut lead token
  </Tab>
</Tabs>

### Common Model

<Steps>
  <Step title="Portal Configuration">
    One `PortalConfiguration` row per `(organization, portal)` with unique constraint.
    Requires `REAL_ESTATE_MANAGE` permission (org admin/owner).
  </Step>
  <Step title="API Endpoints">
    - `GET /portal-syndication/config` — list configurations
    - `POST /portal-syndication/config` — upsert configuration
    - `PATCH /portal-syndication/config/:portal/toggle` — enable/disable
  </Step>
  <Step title="Security">
    API credentials encrypted at rest using AES-256-GCM via `EncryptionService`.
    Raw values never returned in responses or logs.
  </Step>
</Tabs>

<Warning>
PropWise-generated secrets (`webhookSecret` for PF; `feedSecret` + `feedUrl` for Bayut/dubizzle) are minted once on first creation and never regenerated on update. Regenerating would break live portal subscriptions.
</Warning>

### Property Finder OAuth2 Flow

<Steps>
  <Step title="Portal Setup">
    Admin opens PF Expert → Developer Resources → API Credentials, generates API Integration key.
    Receives API Key + API Secret, sets expiry (max 365 days), enables required scopes.
  </Step>
  <Step title="Required Scopes">
    Optional scopes: `listings:full_access`, `leads:read`, `credits:read`
    Default scopes (always enabled): `webhooks:full_access`, `compliances:read`, `locations:read`, `projects:read`, `listing_verification:full_access`
  </Step>
  <Step title="PropWise Configuration">
    Admin pastes API Key + API Secret + expiry into `POST /config` with `portal=property_finder`.
    Credentials stored encrypted with `apiKeyExpiresAt` for tracking.
  </Step>
  <Step title="Runtime Authentication">
    `PfTokenService` exchanges key+secret at `POST /v1/auth/token` for 30-minute Bearer JWT.
    No refresh token flow - PropWise re-issues on expiry and caches per org.
  </Step>
</Steps>

### Bayut Dual Channel Setup

<Info>
Bayut uses two separate channels with different credentials for outbound listings and inbound leads.
</Info>

**Outbound Listings (Feed Pull):**
- PropWise generates per-org HMAC feed URL
- Admin copies `feedUrl` from PropWise and registers in Bayut XML feed settings
- Bayut crawls feed on schedule

**Inbound Leads (API Poll):**
- Bayut issues Pull API Bearer token for `www.bayut.com/api-v7/stats/website-client-leads`
- Admin pastes token into PropWise as encrypted `apiKey`
- `BayutLeadPollerService` polls every 15 minutes since `lastLeadPollAt`

### Feed URL Architecture

Each organization gets its own feed URL for Bayut/dubizzle outbound listings:

```
GET /portal-syndication/feeds/{orgId}?token={hmac}
```

<CodeGroup>
```javascript Token Generation
// Generated once when Bayut/dubizzle PortalConfiguration is created
const feedUrl = generateFeedUrl(orgId, feedSecret);
const token = HMAC_SHA256(orgId, feedSecret); // hex format
```

```javascript Token Verification
// Public endpoint with RLS bypass scoped to orgId

  return timingSafeEqual(expectedToken, providedToken);
}
```
</CodeGroup>

<Tip>
The unified feed serves ALL live listings for an organization. The `<Portals>` tag inside each `<Property>` controls visibility on Bayut, dubizzle, or both based on enabled `ListingPortalSync` rows.
</Tip>

## Field Name Divergence

The following table shows how the same logical data uses different field names across portals:

| Canonical Field | Bayut XML Tag | PF JSON Field | Notes |
|---|---|---|---|
| `id` (+ org code) | `<Property_Ref_No>` | `reference` | Format: `UNIT-{orgShortCode}-{listing.id}` |
| `permitNumber` | `<Permit_Number>` | `compliance.listingAdvertisementNumber` | PF may be composite format |
| — (org license) | — | `compliance.issuingClientLicenseNumber` | PF only |
| `purpose` | `<Property_purpose>` | `price.type` | Value + concept differ (see below) |
| `propertyType` | `<Property_Type>` | `type` | Different value maps |
| `price` | `<Price>` | `price.amounts.{sale\|yearly\|monthly\|weekly\|daily}` | PF splits by price type |
| `rentalPeriod` | `<Rent_Frequency>` | Folded into `price.type` + `price.amounts` | Bayut separate, PF integrated |
| `furnished` | `<Furnished>` | `amenities.furnished` | Different value formats |
| `bedrooms` | `<Bedrooms>` | `attributes.bedrooms` | Different value ranges |
| `bathrooms` | `<Bathrooms>` | `attributes.bathrooms` | Different validation rules |
| `area` | `<Size>` + `<Size_unit>` | `attributes.size.value` + `.unit` | Unit handling differs |
| `title` | `<Title_{lang}>` | `translations.{lang}.title` | Structure differs |
| `description` | `<Description_{lang}>` | `translations.{lang}.description` | Structure differs |
| `emirate` | `<City>` | `location.level1` | Value mapping required |
| `location` (area) | `<Location>` | `location.level2` | Value mapping required |
| `subLocation` | `<Sub_Location>` | `location.level3` | May not exist in PF |
| `building` | `<Building>` | `location.level4` | Optional on both |
| `completionStatus` | `<Completion_Status>` | `attributes.completionStatus` | Different value sets |
| `developer` | — | `project.developer` | PF only (if project linked) |
| `projectName` | — | `project.name` | PF only (if project linked) |
| `agentEmail` | `<Email>` | `contact.email` | Same concept |
| `agentPhone` | `<Phone>` | `contact.phone` | Same concept |
| `agentName` | `<Agent_Name>` | `contact.name` | Same concept |

<Note>
Fields marked with "—" are not supported by that portal. Portal-specific fields should be hidden in the frontend when that portal is not enabled for the organization.
</Note>

## Value Space Divergence

Even when field names align conceptually, the allowed values often differ significantly:

### Purpose Field Transformation

<Tabs>
  <Tab title="Bayut">
    **Field:** `<Property_purpose>`  
    **Values:** `Buy`, `Rent`  
    **Source:** Direct mapping from `SALE`/`RENT`
  </Tab>
  <Tab title="Property Finder">
    **Field:** `price.type`  
    **Values:** `sale`, `yearly`, `monthly`, `weekly`, `daily`  
    **Source:** Combined `purpose` + `rentalPeriod` logic
  </Tab>
</Tabs>

### Bedrooms Handling

<Warning>
Bedroom handling is significantly different between portals and requires careful transformation.
</Warning>

**Bayut Logic:**
- `0` bedrooms → `"-1"` (studio representation)
- `1-10` bedrooms → `"1"` to `"10"` (string format)
- `>10` bedrooms → `"10+"` (capped with indicator)
- `null` → omit field entirely

**Property Finder Logic:**
- `0` bedrooms → `"studio"` (explicit studio type)
- `1-30` bedrooms → `"1"` to `"30"` (higher cap, string format)
- `>30` bedrooms → `"30"` (hard cap)
- `null` → validation error (required field)

### Location Hierarchy

<Info>
Both portals use hierarchical location data but with different level naming and validation requirements.
</Info>

| Level | Bayut | Property Finder | Notes |
|---|---|---|---|
| Emirate | `<City>` | `location.level1` | Required on both |
| Area | `<Location>` | `location.level2` | Required on both |
| Sub-area | `<Sub_Location>` | `location.level3` | Optional, may not exist in PF |
| Building | `<Building>` | `location.level4` | Optional on both |

## Frontend Visibility Rules

The frontend should show/hide input fields based on which portals an organization has enabled:

### Portal-Specific Fields

<Steps>
  <Step title="Check Portal Configuration">
    Query organization's enabled portals via `GET /portal-syndication/config`
  </Step>
  <Step title="Apply Visibility Rules">
    Show fields only when relevant portal is enabled:
    - Finishing quality → Property Finder only
    - Project/Developer fields → Property Finder only  
    - Compliance fields → Property Finder only
  </Step>
  <Step title="Validation Rules">
    Apply portal-specific validation:
    - Bedroom range limits differ between portals
    - Bathroom requirements differ by property type on PF
    - Compliance required for certain emirates on PF
  </Step>
</Steps>

### Conditional Field Display

```javascript
// Example frontend logic
const showComplianceFields = enabledPortals.includes('property_finder');
const showFinishingField = enabledPortals.includes('property_finder');
const maxBedrooms = enabledPortals.includes('property_finder') ? 30 : 10;
```

<Tip>
Use the centralized portal value map functions for consistent validation and transformation across frontend and backend components.
</Tip>
