## Authentication & Account Linking

None of the three portals use interactive OAuth "Connect with..." redirects. All require manual credential/URL exchange configured per organization.

<CardGroup cols={2}>
  <Card title="Property Finder" icon="building">
    Push + webhooks with OAuth2 client credentials
  </Card>
  <Card title="Bayut/dubizzle" icon="feed">
    Feed pull + lead polling with static tokens
  </Card>
</CardGroup>

### Portal Configuration Overview

| Portal | Direction | Admin Provides | PropWise Generates | Transport Auth |
|--------|-----------|----------------|-------------------|----------------|
| **Property Finder** | push + webhooks | API Key + Secret + expiry | `webhookSecret` | OAuth2 client-credentials → 30-min Bearer JWT |
| **Bayut** | feed pull + lead poll | Bayut Pull API Bearer token | `feedSecret` + `feedUrl` | Org registers feed URL; Bearer token for leads |
| **dubizzle** | shares Bayut | (nothing new) | reuses unified feed | Shares Bayut feed + token |

### Common Model Structure

<Tabs>
  <Tab title="Configuration">
    - One `PortalConfiguration` row per `(organization, portal)` (unique constraint)
    - RBAC: requires `REAL_ESTATE_MANAGE` (org admin/owner)
    - API credentials encrypted at rest (AES-256-GCM)
    - Generated secrets (`webhookSecret`, `feedSecret`) minted once and never regenerated
  </Tab>
  <Tab title="Endpoints">
    - `GET /portal-syndication/config` — list configurations
    - `POST /portal-syndication/config` — upsert configuration
    - `PATCH /portal-syndication/config/:portal/toggle` — enable/disable
  </Tab>
</Tabs>

### Property Finder Authentication Flow

<Steps>
  <Step title="Portal Setup">
    Admin opens PF Expert → Developer Resources → API Credentials, generates API Integration key with required scopes:
    - Optional: `listings:full_access`, `leads:read`, `credits:read`
    - Default: `webhooks:full_access`, `compliances:read`, `locations:read`, `projects:read`, `listing_verification:full_access`
  </Step>
  <Step title="PropWise Configuration">
    Admin pastes API Key + Secret + expiry into `POST /config` with `portal=property_finder`
  </Step>
  <Step title="Runtime Token Exchange">
    `PfTokenService` exchanges key+secret at `POST /v1/auth/token` → 30-minute Bearer JWT (no refresh token)
  </Step>
  <Step title="Webhook Setup">
    PropWise auto-generates `webhookSecret` and subscribes to PF webhooks with HMAC verification
  </Step>
</Steps>

### Bayut Configuration

<Warning>
Bayut uses two separate channels requiring different credentials for outbound listings and inbound leads.
</Warning>

**Outbound (listings)**: PropWise generates HMAC feed URL; admin registers it in Bayut account
**Inbound (leads)**: Admin provides static Pull API Bearer token for lead polling

### Feed URL Architecture

Each organization gets its own feed URL:

```
GET /portal-syndication/feeds/{orgId}?token={hmac}
```

<Info>
The feed is unified - one endpoint returns all org listings with `<Portals>` tags determining Bayut/dubizzle visibility.
</Info>

## Mapping Helper Inventory

| Concern | Helper | Status |
|---------|--------|---------|
| Property type → portal value | `LAYOUT_TYPE_TO_BAYUT`, `LAYOUT_TYPE_TO_PF_SLUG` | ✅ Centralized |
| Purpose, furnished, bedrooms, etc. | `portal-value-map.ts` | ✅ Built |

<CodeGroup>
```typescript Value Mapping Functions
// Purpose mapping
purposeToBayut(p)        // SALE→'Buy',  RENT→'Rent'
purposeToPfPriceType(p, rentalPeriod)  // SALE→'sale', RENT→'yearly'|'monthly'|'weekly'|'daily'

// Furnished mapping
furnishedToBayut(f)      // FURNISHED→'Yes', UNFURNISHED→'No', PARTLY_FURNISHED→'Partly'
furnishedToPf(f)         // FURNISHED→'furnished', UNFURNISHED→'unfurnished', PARTLY_FURNISHED→'semi-furnished'

// Bedroom mapping
bedroomsToBayut(n)       // 0→'-1', 1..10→'1'..'10', >10→'10+', null→omit
bedroomsToPf(n)          // 0→'studio', 1..30→'1'..'30' (cap 30)
```
</CodeGroup>

## Field Name Divergence

The following table shows how the same logical data maps to different field names across portals:

| Canonical (`Listing`) | Bayut XML Tag | PF JSON Field | Notes |
|----------------------|---------------|---------------|-------|
| `id` (+ org code) | `<Property_Ref_No>` | `reference` | `UNIT-{orgShortCode}-{listing.id}` |
| `permitNumber` | `<Permit_Number>` | `compliance.listingAdvertisementNumber` | PF may be composite |
| `purpose` | `<Property_purpose>` | `price.type` | Value + concept differ |
| `propertyType` | `<Property_Type>` | `type` | Different value maps |
| `price` | `<Price>` | `price.amounts.{sale\|yearly\|monthly\|weekly\|daily}` | PF splits by type |
| `rentalPeriod` | `<Rent_Frequency>` | folded into `price.type` | Different concepts |
| `bedrooms` | `<Bedrooms>` | `bedrooms` | Different value handling |
| `bathrooms` | `<Bathrooms>` | `bathrooms` | Different caps/rules |
| `furnished` | `<Furnished>` | `furnishings.furnished` | Different value sets |
| `builtUpArea` | `<Size>` | `size` | Same concept |
| `plotArea` | `<Plot_Area>` | `landsize` | PF land-only field |
| `amenities` | `<Features>` | `amenities` | Different ID systems |

## Portal-Specific Fields

These fields only apply to certain portals and should be conditionally shown in the frontend:

<Tabs>
  <Tab title="Property Finder Only">
    - `compliance.issuingClientLicenseNumber` (org license)
    - `finishing` (construction status)
    - Land-specific fields for farms/plots
    - Project association fields
    - Verification status fields
  </Tab>
  
  <Tab title="Bayut Only">
    - Certain amenity categories
    - Specific property sub-types
    - Bayut-specific location hierarchies
  </Tab>
  
  <Tab title="Shared">
    - Core property details (bedrooms, bathrooms, size)
    - Basic amenities
    - Price and rental information
    - Media attachments
  </Tab>
</Tabs>

## Frontend Visibility Rules

<Tip>
Use the portal configuration status to conditionally show/hide fields in the frontend based on which portals an organization has enabled.
</Tip>

```javascript
// Example frontend logic
const showFinishingField = organization.portals.includes('property_finder');
const showBayutSpecificAmenities = organization.portals.includes('bayut') || organization.portals.includes('dubizzle');
```

## Validation Considerations

<Warning>
Each portal has different validation rules for the same logical field. Always validate against the target portal's specific requirements.
</Warning>

Key validation differences:
- Bedroom count caps (Bayut: 10+, PF: 30 max)
- Bathroom count limits vary by property type
- Price format and currency handling
- Required vs optional compliance fields by emirate
