This matrix provides a comprehensive reference for how canonical `Listing` fields map to both Bayut/dubizzle and Property Finder portals. It addresses field name differences, allowed value variations, and portal-specific fields for frontend visibility control.

## Portal Authentication & Account Linking

<Warning>
None of the three portals use interactive OAuth "Connect with..." redirects. All require manual credential/URL exchange configured per organization by an admin/owner.
</Warning>

### Authentication Methods by Portal

<CardGroup cols={3}>
  <Card title="Property Finder" icon="key">
    **Push + Webhooks**
    - Admin provides: API Key + Secret + Expiry
    - PropWise generates: `webhookSecret`
    - Auth: OAuth2 client-credentials ŌåÆ 30-min Bearer JWT
  </Card>
  
  <Card title="Bayut" icon="rss">
    **Feed Pull + Lead Poll**
    - Admin provides: Bayut Pull API Bearer token
    - PropWise generates: `feedSecret` + `feedUrl`
    - Auth: Org registers feed URL; PropWise polls leads
  </Card>
  
  <Card title="dubizzle" icon="share">
    **Shares Bayut**
    - Admin provides: Nothing new (piggybacks Bayut)
    - PropWise generates: Reuses unified feed
    - Auth: Shares Bayut feed + lead token
  </Card>
</CardGroup>

### Configuration Model

<Steps>
  <Step title="Portal Configuration Setup">
    One `PortalConfiguration` row per `(organization, portal)` with unique constraint.
    
    **RBAC Requirement:** `REAL_ESTATE_MANAGE` permission (org admin/owner only)
  </Step>
  
  <Step title="API Endpoints (Phase 1 - Implemented)">
    ```http
    GET /portal-syndication/config
    POST /portal-syndication/config  
    PATCH /portal-syndication/config/:portal/toggle
    ```
    
    <Info>
    API credentials are encrypted at rest (AES-256-GCM) and never returned in responses or logs.
    </Info>
  </Step>
  
  <Step title="Secret Generation">
    PropWise-generated secrets (`webhookSecret`, `feedSecret`, `feedUrl`) are minted once on first creation and never regenerated to avoid breaking live portal subscriptions.
  </Step>
</Steps>

### Property Finder OAuth2 Setup

<Tabs>
  <Tab title="Portal Side (PF Expert)">
    1. Open **Developer Resources ŌåÆ API Credentials**
    2. Generate **API Integration** key type
    3. Receive **API Key + Secret**, set expiry (max 365 days)
    4. Enable required optional scopes:
       - `listings:full_access`
       - `leads:read` 
       - `credits:read`
    
    <Note>
    Webhook/compliance/location/project/verification scopes are default and always enabled.
    </Note>
  </Tab>
  
  <Tab title="PropWise Side">
    1. Admin enters API Key + Secret + expiry in `POST /config`
    2. Credentials stored encrypted with `apiKeyExpiresAt` tracking
    3. Runtime: `PfTokenService` exchanges for 30-minute Bearer JWT
    4. Auto-generates `webhookSecret` for callback verification
    5. Daily cron warns before expiry and auto-disables on expiration
  </Tab>
</Tabs>

### Bayut Dual-Channel Setup

<AccordionGroup>
  <Accordion title="Outbound Listings (XML Feed)">
    - PropWise generates per-org HMAC feed URL
    - Admin copies `feedUrl` and registers in Bayut XML feed settings
    - Bayut pulls feed on schedule
    - Format: `GET /portal-syndication/feeds/{orgId}?token={hmac}`
  </Accordion>
  
  <Accordion title="Inbound Leads (Pull API)">
    - Bayut provides static Pull API Bearer token
    - Admin enters token as Bayut config `apiKey`
    - `BayutLeadPollerService` polls every 15 minutes
    - Endpoint: `www.bayut.com/api-v7/stats/website-client-leads`
  </Accordion>
</AccordionGroup>

### dubizzle Integration

dubizzle leverages Bayut's infrastructure:

- **Listings:** Reads same unified feed with `<Portals>` tag filtering
- **Leads:** Shares Bayut endpoint/token with `source` field discrimination  
- **Setup:** Create dubizzle config row + enable (no additional credentials needed)

`Listing field` refers to self-contained Listing columns (snapshotted from unit in linked mode, or entered manually). "ŌĆö" indicates the field is not supported by that portal.
</Note>

|----------------------|---------------|---------------------------|-------|

| ŌĆö (org license) | ŌĆö | `compliance.issuingClientLicenseNumber` | PF only field |

## Value Space Differences

### Purpose Field Mapping

The `purpose` field demonstrates how the same concept maps to completely different structures:

<Tabs>
  <Tab title="Bayut">
    **Field:** `<Property_purpose>`
    
    **Values:**
    - `Buy` (for SALE listings)
    - `Rent` (for RENT listings)
    
    Simple string mapping with capitalized values.
  </Tab>
  
  <Tab title="Property Finder">
    **Field:** `price.type`
    
    **Values:**
    - `sale` (for SALE listings)
    - `yearly`, `monthly`, `weekly`, `daily` (for RENT listings based on rental period)
    
    Combines purpose with rental frequency into structured pricing.
  </Tab>
</Tabs>

### Bedroom Count Variations

<CodeGroup>
```typescript Bayut Transform
bedroomsToBayut(bedrooms: number | null): string | undefined {
  if (bedrooms === null) return undefined;
  if (bedrooms === 0) return '-1';      // Studio represented as -1
  if (bedrooms > 10) return '10+';      // Cap at 10+
  return bedrooms.toString();           // 1-10 as strings
}
```

```typescript Property Finder Transform  
bedroomsToPf(bedrooms: number | null): string | undefined {
  if (bedrooms === null) return undefined;
  if (bedrooms === 0) return 'studio';  // Explicit studio value
  if (bedrooms > 30) return '30';       // Higher cap at 30
  return bedrooms.toString();           // 1-30 as strings
}
```
</CodeGroup>

## Feed URL Architecture

Each organization receives a unique feed URL for Bayut/dubizzle integration:

```
GET /portal-syndication/feeds/{orgId}?token={hmac}
```

<Steps>
  <Step title="URL Generation">
    Generated once when Bayut or dubizzle `PortalConfiguration` is first created.
    
    ```typescript
    token = HMAC-SHA256(orgId, feedSecret)  // hex format
    ```
  </Step>
  
  <Step title="Token Verification">
    Verified using `PortalConfigurationService.verifyFeedToken` with:
    - Constant-time comparison
    - Format validation  
    - RLS-bypassed lookup scoped to `orgId`
  </Step>
  
  <Step title="Unified Feed Response">
    Single endpoint returns all org listings with `<Portals>` tag determining visibility per portal based on enabled `ListingPortalSync` rows.
  </Step>
</Steps>

<Note>
This is a **public endpoint** with no authentication beyond the HMAC token verification, allowing portal crawlers to access the feed without API authentication.
</Note>
