## Authentication and account linking

Unlike OAuth integrations, all three portals use **manual credential/URL exchange** configured by organization admins.

<Tabs>
  <Tab title="Property Finder">
    **Direction**: Push + webhooks  
    **Admin provides**: API Key + API Secret + expiry (from PF Expert)  
    **PropWise generates**: `webhookSecret`  
    **Transport auth**: OAuth2 client-credentials â†’ 30-min Bearer JWT
  </Tab>
  <Tab title="Bayut">
    **Direction**: Feed pull + lead poll  
    **Admin provides**: Bayut Pull API Bearer token (inbound leads only)  
    **PropWise generates**: `feedSecret` + per-org `feedUrl` (outbound listings)  
    **Transport auth**: Org registers PropWise's feed URL; PropWise polls leads with Bearer token
  </Tab>
  <Tab title="dubizzle">
    **Direction**: Shares Bayut infrastructure  
    **Admin provides**: Enable only (piggybacks Bayut)  
    **PropWise generates**: Reuses unified feed  
    **Transport auth**: Shares Bayut feed + Bayut lead token
  </Tab>
</Tabs>

### Portal configuration model

<Info>
One `PortalConfiguration` row per `(organization, portal)` with unique constraint. Requires `REAL_ESTATE_MANAGE` permission (org admin/owner).
</Info>

**Available endpoints** (Phase 1, implemented):
- `GET /portal-syndication/config` â€” List configurations (credentials never returned)
- `POST /portal-syndication/config` â€” Upsert configuration
- `PATCH /portal-syndication/config/:portal/toggle` â€” Enable/disable portal

<Warning>
API credentials are encrypted at rest using AES-256-GCM and never returned in responses or logs. PropWise-generated secrets are minted once and never regenerated to avoid breaking live portal subscriptions.
</Warning>

### Property Finder setup

<Steps>
  <Step title="Generate API credentials">
    In PF Expert, navigate to **Developer Resources â†’ API Credentials** and create an **API Integration** key with:
    - **Required scopes**: `listings:full_access`, `leads:read`, `credits:read`
    - **Expiry**: Maximum 365 days
    - **Default scopes**: Automatically enabled (webhooks, compliance, locations, etc.)
  </Step>
  <Step title="Configure in PropWise">
    Admin enters API Key + API Secret + expiry in `POST /config` with `portal=property_finder`
  </Step>
  <Step title="Runtime token exchange">
    `PfTokenService` exchanges credentials for 30-minute Bearer JWT, cached per organization
  </Step>
  <Step title="Webhook subscription">
    PropWise auto-generates `webhookSecret` and subscribes to PF webhooks with HMAC verification
  </Step>
</Steps>

### Bayut dual-channel setup

**Outbound listings**: PropWise generates HMAC feed URL that admin registers in Bayut XML feed settings.

**Inbound leads**: Admin provides Bayut Pull API Bearer token for lead polling every 15 minutes.

<CodeGroup>
```http Feed URL Format
GET /portal-syndication/feeds/{orgId}?token={hmac}
```

```typescript Token Verification
token = HMAC-SHA256(orgId, feedSecret)
// Verified with timing-safe comparison
```
</CodeGroup>

### Feed URL architecture

<Check>
Each organization gets **one unified feed URL** that serves all listings with per-listing portal targeting via `<Portals>` tags.
</Check>

The feed controller:
- Uses `@PublicEndpoint()` decorator (no auth required)
- Validates HMAC token with constant-time comparison
- Returns XML with RLS bypass scoped to the organization
- Includes recently-deleted listings as `Property_Status = deleted` for one crawl cycle

## Field name divergence

The following table shows how the same logical data maps to different field names across portals:

| Canonical (`Listing`) | Bayut XML tag | PF JSON field | Notes |
|----------------------|---------------|---------------|--------|
| `id` | `<Property_Ref_No>` | `reference` | Format: `UNIT-{orgShortCode}-{listing.id}` |
| `permitNumber` | `<Permit_Number>` | `compliance.listingAdvertisementNumber` | PF may be composite |
| `purpose` | `<Property_purpose>` | `price.type` | **Value + concept differ** |
| `propertyType` | `<Property_Type>` | `type` | Different value maps |
| `price` | `<Price>` | `price.amounts.{sale\|yearly\|monthly\|weekly\|daily}` | PF splits by price type |
| `rentalPeriod` | `<Rent_Frequency>` | Folded into `price.type` + `price.amounts` | Bayut separate, PF embedded |
| `bedrooms` | `<No_of_Bedroom>` | `bedrooms` | Different value handling |
| `bathrooms` | `<No_of_Bathroom>` | `bathrooms` | Different caps and nulls |
| `furnished` | `<Furnished>` | `furnishingStatus` | Different enum values |
| `area` | `<Size>` | `area.value` | PF includes unit separately |
| `areaUnit` | Built into `<Size>` | `area.unit` | Bayut embeds, PF separates |
| `description` | `<Description>` | `description` | Same field name |
| `title` | `<Title>` | `title` | Same field name |

<Tip>
Use the centralized mapping helpers in `portal-value-map.ts` rather than implementing transforms in individual adapters to ensure consistency.
</Tip>

## Portal-specific fields

Some fields only exist on one portal and should be hidden in the frontend when that portal is disabled:

<Tabs>
  <Tab title="Property Finder only">
    - `compliance.issuingClientLicenseNumber` (org license)
    - `finishing` status (no Bayut equivalent)
    - `compliance` requirements vary by emirate
    - Advanced `area` unit specifications
  </Tab>
  <Tab title="Bayut only">
    - Separate `<Rent_Frequency>` field
    - Specific XML-based image handling
    - `<Community_Name>` vs PF's location hierarchy
  </Tab>
  <Tab title="dubizzle shared">
    - Inherits all Bayut field requirements
    - No additional portal-specific fields
    - Same unified feed and lead token
  </Tab>
</Tabs>

## Frontend visibility rules

The frontend should show/hide input fields based on which portals an organization has enabled:

```typescript
// Example visibility logic
const showFinishingField = organization.portals.includes('property_finder');
const showRentalFrequency = organization.portals.some(p => ['bayut', 'dubizzle'].includes(p));
const showComplianceFields = organization.portals.includes('property_finder');
```

<Note>
Use the `PortalConfiguration` enabled status to determine field visibility rather than hardcoding portal lists in frontend components.
</Note>
