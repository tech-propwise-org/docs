</Note>

## Authentication & account linking

None of the portals use interactive OAuth redirects. All three require manual credential exchange configured by organization admins.

<Tabs>
<Tab title="Property Finder">
**Direction:** Push + webhooks  
**Admin provides:** API Key + API Secret + expiry  
**PropWise generates:** `webhookSecret`  
**Authentication:** OAuth2 client-credentials → 30-min Bearer JWT

<Steps>
<Step title="Portal configuration">
Admin opens **Developer Resources → API Credentials** in PF Expert, generates **API Integration** key, receives API Key + Secret with max 365-day expiry.
</Step>
<Step title="PropWise setup">
Admin enters credentials via `POST /portal-syndication/config` with `portal=property_finder`. Credentials stored encrypted.
</Step>
<Step title="Runtime authentication">
PropWise exchanges key+secret at `POST /v1/auth/token` for 30-minute Bearer JWT. Auto-reissues on expiry.
</Step>
<Step title="Webhook subscription">
PropWise auto-generates `webhookSecret` and subscribes to PF webhooks with HMAC `X-Signature` verification.
</Step>
</Tabs>

<Tab title="Bayut">
**Direction:** Feed pull + lead poll  
**Admin provides:** Bayut Pull API Bearer token  
**PropWise generates:** `feedSecret` + per-org `feedUrl`  
**Authentication:** Static Bearer token for leads, HMAC-secured feed URL

<Steps>
<Step title="Feed setup">
PropWise generates unique feed URL: `/portal-syndication/feeds/{orgId}?token={hmac}`
</Step>
<Step title="Portal registration">
Admin copies `feedUrl` and registers it in their Bayut account's XML feed settings.
</Step>
<Step title="Lead polling">
Admin provides Bayut Pull API Bearer token. PropWise polls leads every 15 minutes.
</Step>
</Tabs>

<Tab title="dubizzle">
**Direction:** Shares Bayut infrastructure  
**Admin provides:** Enable only (piggybacks Bayut)  
**PropWise generates:** Reuses unified feed  
**Authentication:** Shares Bayut feed + lead token

dubizzle reads the same unified feed as Bayut, with per-listing `<Portals>` tags controlling visibility.
</Tab>
</Tabs>

### Common model structure

```json
{
  "portal": "property_finder|bayut|dubizzle",
  "isEnabled": true,
  "hasApiKey": true,
  "hasWebhookSecret": false,
  "hasFeedSecret": true
}
```

<Warning>
API credentials are encrypted at rest using AES-256-GCM. Raw values are never returned in responses or logged.
</Warning>

## Field name divergences

The following table shows how identical data maps to different field names across portals:

<AccordionGroup>
<Accordion title="Core property fields">
| Canonical (`Listing`) | Bayut XML tag | PF JSON field | Notes |
|---|---|---|---|
| `id` (+ org short code) | `<Property_Ref_No>` | `reference` | `UNIT-{orgShortCode}-{listing.id}` |
| `permitNumber` | `<Permit_Number>` | `compliance.listingAdvertisementNumber` | PF may be composite |
| `purpose` | `<Property_purpose>` | `price.type` | **Value + concept differ** |
| `propertyType` | `<Property_Type>` | `type` | Different value maps |
| `price` | `<Price>` | `price.amounts.{sale\|yearly\|monthly}` | PF splits by price type |
| `rentalPeriod` | `<Rent_Frequency>` | Folded into `price.type` | Bayut separate frequency |
</Accordion>

## Value space divergences

### Purpose mapping

<CodeGroup>
```typescript Purpose transforms
// Bayut
purposeToBayut('SALE') // → 'Buy'
purposeToBayut('RENT') // → 'Rent'

// Property Finder
purposeToPfPriceType('SALE', null) // → 'sale'
purposeToPfPriceType('RENT', 'YEARLY') // → 'yearly'
purposeToPfPriceType('RENT', 'MONTHLY') // → 'monthly'
```

```json Bayut XML
<Property_purpose>Buy</Property_purpose>
<!-- or -->
<Property_purpose>Rent</Property_purpose>
<Rent_Frequency>Yearly</Rent_Frequency>
```

```json Property Finder API
{
  "price": {
    "type": "sale"
    // or "yearly", "monthly", "weekly", "daily"
  }
}
```
</CodeGroup>

### Furnished status

<Tabs>
<Tab title="Bayut values">
- `FURNISHED` → `"Yes"`
- `UNFURNISHED` → `"No"`  
- `PARTLY_FURNISHED` → `"Partly"`
</Tab>
<Tab title="Property Finder values">
- `FURNISHED` → `"furnished"`
- `UNFURNISHED` → `"unfurnished"`
- `PARTLY_FURNISHED` → `"semi-furnished"`
</Tab>
</Tabs>

### Bedroom mapping

<Info>
Special handling required for studio apartments and caps.
</Info>

| Canonical | Bayut | Property Finder |
|---|---|---|
| `0` | `"-1"` | `"studio"` |
| `1-10` | `"1"-"10"` | `"1"-"30"` (cap 30) |
| `>10` | `"10+"` | Capped at `"30"` |
| `null` | Omitted | Required field |

## Portal-specific fields

### Property Finder exclusive fields

<Check>
These fields only appear when Property Finder is enabled in the organization.
</Check>

- **Compliance fields**: `compliance.issuingClientLicenseNumber`, emirate-based compliance rules
- **Finishing status**: `specifications.finishing` (not available in Bayut)
- **Project status**: For off-plan properties
- **Verification**: Built-in listing verification workflow

### Bayut/dubizzle exclusive fields

- **Built-up area**: `<Built_up_Area>` separate from plot area
- **Plot area**: `<Plot_Area>` for land/villa properties
- **Cheques**: `<No_of_Cheques>` for rental payment terms

## Frontend visibility rules

<Steps>
<Step title="Portal detection">
Frontend checks enabled portals via `GET /portal-syndication/config` to determine which fields to show/hide.
</Step>
<Step title="Conditional rendering">
```typescript
// Show finishing field only for Property Finder
{hasPropertyFinder && (
  <FinishingSelect 
    value={listing.finishing}
    onChange={handleFinishingChange}
  />
)}

// Show cheques field only for Bayut/dubizzle
{(hasBayut || hasDubizzle) && (
  <ChequesInput
    value={listing.cheques}
    onChange={handleChequesChange}
  />
)}
```
</Step>
<Step title="Validation rules">
Apply portal-specific validation using `PortalValidationService` based on enabled portals.
</Step>
</Steps>
