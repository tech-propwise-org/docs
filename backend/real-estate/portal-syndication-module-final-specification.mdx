## Overview

The `portal-syndication` module manages the syndication of property listings to external real estate portals like Property Finder, Bayut, and Dubizzle. This module acts as a bridge between the internal property inventory system and external marketing platforms, handling authentication, data transformation, and synchronization status tracking.

<Note>
**Module path:** `src/modules/real-estate/portal-syndication/`

The module is part of `RealEstateModule` and is imported from `src/modules/real-estate/real-estate.module.ts`.
</Note>

**Last Updated:** January 2024  
**Status:** Development - Phase 1

## Architecture

The module follows a layered architecture with clear separation between:

<CardGroup cols={2}>
  <Card title="Listings" icon="list">
    Marketing layer that wraps inventory units with portal-specific metadata
  </Card>
  <Card title="Portal Configuration" icon="gear">
    Per-organization portal credentials and settings
  </Card>
  <Card title="Sync Tracking" icon="sync">
    Per-listing, per-portal synchronization state management
  </Card>
  <Card title="Adapters" icon="plug">
    Portal-specific integration logic (future implementation)
  </Card>
</CardGroup>

### Design Decisions

1. **Separation of Concerns**: Listings are separate from inventory units to allow marketing-specific data
2. **Per-Portal Tracking**: Each listing maintains independent sync state per portal
3. **Encrypted Credentials**: Portal API keys stored encrypted using AES-256-GCM
4. **Mapping Tables**: Cached lookups for portal-specific identifiers (agents, locations)

## Entities

### Listing

Represents a property unit prepared for marketing across portals.

```typescript
interface Listing {
  id: string;
  title: string;
  description: string;
  price: number;
  location: string;
  bedrooms: number;
  bathrooms: number;
  size: number;
  amenities: string[];
  images: string[];
  furnishedStatus: FurnishedStatus;
  rentalPeriod?: RentalPeriod;
  finishingType: FinishingType;
  status: ListingStatus;
  isActive: boolean;
  inventoryUnit: string; // FK to inventory
  createdBy: string; // FK to User
  organization: string; // FK to Organization
  createdAt: Date;
  updatedAt: Date;
}
```

### PortalConfiguration

Stores per-organization portal credentials and settings.

```typescript
interface PortalConfiguration {
  id: string;
  portal: Portal;
  isEnabled: boolean;
  apiKey: string; // Encrypted
  apiSecret?: string; // Encrypted
  organizationId?: string; // Portal-specific org ID
  portalCredits?: number;
  lastCreditsCheckAt?: Date;
  organization: string; // FK to Organization
  createdAt: Date;
  updatedAt: Date;
}
```

### ListingPortalSync

Tracks sync state for each listing on each portal.

```typescript
interface ListingPortalSync {
  id: string;
  portal: Portal;
  isEnabled: boolean;
  status: ListingSyncStatus;
  portalListingId?: string;
  lastSyncedAt?: Date;
  lastError?: string;
  lastPublishCost?: {
    creditCost: number;
    checkedAt: Date;
  };
  listing: string; // FK to Listing
  organization: string; // FK to Organization
  createdAt: Date;
  updatedAt: Date;
}
```

### PfAgentMapping

Maps CRM users to Property Finder agent profiles.

```typescript
interface PfAgentMapping {
  id: string;
  pfAgentId: string;
  pfPublicProfileId: string;
  lastSyncedAt?: Date;
  user: string; // FK to User
  organization: string; // FK to Organization
}
```

### PfLocationMapping

Caches Property Finder location ID lookups.

```typescript
interface PfLocationMapping {
  id: string;
  locationName: string;
  pfLocationId: string;
  lastSyncedAt?: Date;
  organization: string; // FK to Organization
}
```

## Patterns and Conventions

### Enums

<Info>
The module uses several enums to maintain data consistency:

- **Portal**: Supported external portals
- **ListingStatus**: Listing lifecycle states (draft, active, archived)
- **ListingSyncStatus**: Per-portal sync states
- **FurnishedStatus**: Property furnishing states
- **RentalPeriod**: Rental frequency options
- **FinishingType**: Property finishing levels
</Info>

### Data Transfer Objects

- Comprehensive validation using class-validator decorators
- Separate DTOs for create, update, and response operations
- Nested validation for complex objects

### Security

<Warning>
Security considerations for the portal syndication module:

- All endpoints protected by AuthGuard and AccessGuard
- Row-level security via organization filtering
- Sensitive credentials encrypted at rest
- API documentation with bearer token authentication
</Warning>

### Error Handling

<Tip>
The module implements robust error handling:

- Structured error responses
- Sync failure tracking in database
- Graceful degradation for portal connectivity issues
</Tip>
