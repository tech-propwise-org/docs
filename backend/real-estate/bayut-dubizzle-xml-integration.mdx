## Overview

An XML integration is required to sync properties with Bayut and/or dubizzle in bulk from an external CRM. It allows clients to maintain their inventory on Bayut and/or dubizzle by providing property information in an XML feed.

<Note>
The XML feed must be exposed as a **publicly accessible URL**. The Bayut system pulls (reads) this URL on a schedule to sync updates.
</Note>

### Integration Flow

- Listings flow **one direction only**: CRM → portals
- The feed is the source of truth for what is published
- Portals never write back into the feed

### Unified Feed Structure

Bayut and dubizzle now share a **single combined XML feed**. The portal(s) a property appears on is controlled per-property by the `<Portals>` tag.

<Warning>
Clients already using XML integration must:
- **Retire** the full & hourly feeds previously created for dubizzle
- **Update the Bayut XML structure** to start using the `<Portals>` attribute to target each portal
</Warning>

## XML Structure

The XML feed must follow the format below and use the tags defined by Bayut. The structure has two main segments:

1. **Header tags** — the document envelope
2. **Property attributes** — per-property fields

### Header Tags

The root element must be named `<Properties>`. Every individual property is a nested `<Property>` element inside it.

```xml
<Properties>
  <Property> <!-- Property 1 attributes --> </Property>
  <Property> <!-- Property 2 attributes --> </Property>
</Properties>
```

<Tip>
All textual values should be wrapped in `<![CDATA[ ... ]]>` so HTML, Arabic text, ampersands, and URLs do not break XML parsing.
</Tip>

## Property Attributes

### Property Details & Core Attributes

<AccordionGroup>
<Accordion title="Identity & Status Fields">

| Tag | Description | Values/Rules |
|-----|-------------|--------------|
| `<Property_Ref_No>` | Unique identifier for the property in your CRM | If duplicates are encountered, only one instance is made live |
| `<Permit_Number>` | Trakheesi Permit (DLD) or Madhmoun Permit (ADREC) | Required for listing to go live |
| `<Property_Status>` | Whether property is live or deleted | `live`, `deleted` |
| `<Last_Updated>` | Last updated date from your CRM | Format: `YYYY-MM-DD HH:mm:ss` |

</Accordion>

<Accordion title="Property Classification">

| Tag | Description | Values |
|-----|-------------|---------|
| `<Property_purpose>` | Sale or rent purpose | `Buy`, `Rent` |
| `<Property_Type>` | Type of property | `Villa`, `Apartment`, `Office`, `Shop`, `Warehouse`, `Factory`, `Labour Camp`, `Other Commercial`, `Commercial Building`, `Residential Floor`, `Commercial Floor`, `Residential Land`, `Commercial Land`, `Townhouse`, `Residential Building`, `Hotel Apartment`, `Loft Apartment`, `Pent House` |
| `<Off_plan>` | Ready or off-plan property | `Yes`, `No` |

</Accordion>

<Accordion title="Property Specifications">

| Tag | Description | Values |
|-----|-------------|---------|
| `<Property_Size>` | Numeric area value | Number |
| `<Property_Size_Unit>` | Area measurement unit | Preferred: `SQFT` |
| `<plotArea>` | Plot area in square feet | Number |
| `<Bedrooms>` | Number of bedrooms | `-1` (studio), `1`-`10`, `10+` |
| `<Bathrooms>` | Number of bathrooms | `1`-`10` |
| `<Furnished>` | Furnishing status | `Yes`, `No`, `Partly` |

</Accordion>
</AccordionGroup>

### Portal Selection

The `<Portals>` tag controls which portal(s) the listing is published to. Use one `<portal>` child per portal.

<CodeGroup>
```xml Both Portals
<Portals>
  <portal>bayut</portal>
  <portal>dubizzle</portal>
</Portals>
```

```xml Bayut Only
<Portals>
  <portal>bayut</portal>
</Portals>
```

```xml dubizzle Only
<Portals>
  <portal>dubizzle</portal>
</Portals>
```
</CodeGroup>

### Features

Use a child `<Feature>` tag per feature:

```xml
<Features>
  <Feature>Swimming Pool</Feature>
  <Feature>Gym</Feature>
  <Feature>Parking</Feature>
</Features>
```

### Pricing

| Tag | Description | Required When |
|-----|-------------|---------------|
| `<Price>` | Sale price or rent amount | Always |
| `<Rent_Frequency>` | Rental frequency | Only when `Property_purpose` is `Rent` |

**Rent Frequency Values:** `Daily`, `Weekly`, `Monthly`, `Yearly`

### Off-Plan Details

<Info>
These tags are required **only** when `<Off_plan>` is `Yes`.
</Info>

| Tag | Description | Required For |
|-----|-------------|--------------|
| `<offplanDetails_saleType>` | Sale type | All off-plan: `New`, `Resale` |
| `<offplanDetails_dldWaiver>` | DLD waiver percentage (0-100) | Off-Plan `New` only |
| `<offplanDetails_originalPrice>` | Original property price | Off-Plan `Resale` only |
| `<offplanDetails_amountPaid>` | Amount already paid | Off-Plan `Resale` only |

## Validation Rules

<Steps>
<Step title="Required Fields">
Ensure these critical fields are present:
- `<Property_Ref_No>` (must be unique)
- `<Permit_Number>` (required for listing to go live)
- `<Property_Status>`
- `<Property_purpose>`
- `<Property_Type>`
- At least one `<Image>` URL
</Step>

<Step title="Conditional Requirements">
Check conditional field requirements:
- `<Rent_Frequency>` required when `Property_purpose` is `Rent`
- Off-plan detail tags required when `<Off_plan>` is `Yes`
</Step>

<Step title="Data Format">
Validate data formats:
- Dates: `YYYY-MM-DD HH:mm:ss`
- Numbers: Numeric values only
- URLs: Publicly accessible
</Step>
</Steps>

## PropWise Implementation Notes

<Note>
For PropWise-specific implementation details including field mapping, portal selection logic, and data transformation rules, refer to `PORTAL_SYNDICATION_SPECIFICATION.md` §6.4 (`BayutDubizzleFeedAdapter`).
</Note>

Key implementation points:

- `Property_Ref_No` is generated as `UNIT-{orgShortCode}-{listing.id}`
- `Property_Status` is `live` for published/pending syncs, `deleted` for removed/disabled
- `Property_Type` is resolved from `LayoutType` via portal mapping
- Bedrooms: `0` becomes `-1` (studio), `>10` becomes `10+`
- Off-plan features reserved for future `OffPlanUnit` syndication
