---

The real estate module is structured as **six active specialized modules** spanning the global and org-private real-estate surfaces. The cross-org `shared-marketplace` code is retained in the repository but is not imported by `RealEstateModule`, so its controllers/providers are not mounted by the app.

The external **Propwise Labs API** stores and serves all property data — both structural (DLD registry) and enrichment (images, payment plans, documents, amenities, pricing, geocoding where exposed). The CRM database stores only CRM-specific data (developer portal auth, ownership links, property interests, marketplace listings, and org inventory context).

<Info>
All modules consume property data exclusively through `PropwiseLabsService`. The CRM no longer opens a second MikroORM connection or maps Propwise Labs entities locally. The reference database configuration has been completely removed and replaced with Propwise Labs API configuration (`app.propwiseLabs.*`) including API credentials, timeout settings, and Bird Verify integration for developer portal contact verification.
</Info>

│   └── propwise-labs-change-requests.service.ts # POST/PATCH/DELETE/GET/cancel change-request facade

│   ├── profile/developer-profile.controller.ts # /developer-portal/me — profile GET/PATCH, agent-card upload
│   └── project-management/

│       └── developer-portal-buildings.controller.ts # /developer-portal/buildings — building CRUD (flat), collections, nested unit CRUD

## Data ownership

| Data Type                                                  | Source            | Access Pattern                                                                                         |
| ---------------------------------------------------------- | ----------------- | ------------------------------------------------------------------------------------------------------ |
| Projects, units, buildings, areas, communities, developers | Propwise Labs API | `PropwiseLabsService`                                                                                  |
| Images, payment plans, documents, amenities                | Propwise Labs API | `PropwiseLabsService`                                                                                  |
| Property data mutations                                    | Propwise Labs API | Change requests for create/delete/approval flows; direct wrappers for selected existing-resource edits |
| Developer auth/profile (DeveloperUser)                     | CRM DB            | Direct entity access                                                                                   |
| Off-plan building publication state                        | CRM DB            | `OffPlanBuildingPublication` stores app-owned building visibility by Propwise Labs building ID         |
| Inventory units (org-scoped inventory)                     | CRM DB            | `InventoryUnit` stores org-specific claim data and Propwise Labs refs                                  |
| Marketplace listings & deals (SharedListing, SharedDeal)   | CRM DB            | Direct entity access; module code retained but not mounted in the app                                  |
| Property interests (Lead/Deal)                             | CRM DB            | Direct entity access, links to InventoryUnit                                                           |
| Property matches                                           | CRM DB            | Direct entity access, matched InventoryUnit IDs                                                        |
| Portal listings/configuration/sync                         | CRM DB            | Direct entity access through `real-estate/portal-syndication`                                          |

| Module             | Scope      | RLS Behavior                   | Description                                                                                 |
| ------------------ | ---------- | ------------------------------ | ------------------------------------------------------------------------------------------- |
| propwise-labs      | Global     | External API facade            | Raw property reads, lookups + change request submission; 1 controller                       |

## External property ID pattern

`InventoryUnit` is the CRM boundary for Propwise Labs property references. It stores plain integer IDs (`refProjectId`, `refBuildingId`, `refUnitId`, `refDeveloperId`, `refPropertyTypeId`, `refPropertySubTypeId`) plus a denormalized Labs snapshot, including the developer display name used by portal syndication and the canonical `propertyType` (`LayoutType` enum) used by portal syndication adapters. Labs raw property type/sub-type names are retained in `labsPropertyType` and `labsPropertySubType` for traceability. Lead/deal property interests and site visits link to `InventoryUnit` instead of storing Labs refs directly.

```typescript
// CRM entity referencing Propwise Labs data:
@Property({ type: 'int', nullable: true })
@Index()
refProjectId?: number;  // ID in Propwise Labs projects

@Property({ type: 'int', nullable: true })
@Index()
refUnitId?: number;     // ID in Propwise Labs units
```

## Propwise Labs module

**Path:** `src/modules/real-estate/propwise-labs/`

### Purpose

Provides the CRM backend facade over the external Propwise Labs API. It handles API authentication, bearer-token caching, read calls for property data, lookup data, change-request submissions for developer portal mutations, and typed wrappers for super-level direct Labs mutations. Local `Ref*` MikroORM entities and the named `reference` EntityManager have been removed.

### API client pattern

`PropwiseLabsService` reads API credentials from `app.propwiseLabs.*`, exchanges `PROPWISE_LABS_API_KEY` and `PROPWISE_LABS_API_SECRET` for a short-lived bearer token, and reuses that token until it is close to expiry. The service includes configurable timeout settings and comprehensive error handling for external API dependencies.

```typescript
@Injectable()
export class PropwiseLabsService {
  private accessToken?: string;
  private tokenExpiresAt = 0;

  async searchProjects(filters: SearchProjectFilters): Promise<PropwiseLabsProject[]> {
    const token = await this.getAccessToken();
    // API call with bearer token
  }
}
```

### Change requests pattern

Property data mutations use the "change requests" pattern for developer portal flows:

```typescript
async createProjectChangeRequest(data: CreateProjectRequest): Promise<ChangeRequest> {
  return this.changeRequestsService.submitChangeRequest({
    type: 'create_project',
    payload: data,
  });
}
```

## Off-plan module

**Path:** `src/modules/real-estate/off-plan/`

Main CRM off-plan directory providing published property browsing over Propwise Labs data. Applies publication visibility filtering and developer contact enrichment.

### Publication visibility

`OffPlanBuildingPublication` entity in CRM DB controls which Propwise Labs buildings are visible in the off-plan directory:

```typescript
@Entity()
export class OffPlanBuildingPublication {
  @PrimaryKey()
  id!: string;

  @Property({ type: 'int', unique: true })
  propwiseLabsBuildingId!: number;

  @Property({ default: true })
  isPublished!: boolean;
}
```

## Developer portal module

**Path:** `src/modules/real-estate/developer-portal/`

Separate authentication system for real estate developers with project management capabilities. Uses its own JWT tokens and manages projects/buildings through both Propwise Labs direct edits and change requests.

### Auth controllers

- `developer-auth.controller.ts` - Signup, login, refresh, logout, password, Bird contact OTPs
- `developer-profile.controller.ts` - Profile GET/PATCH, agent-card uploads

### Project management

- `developer-portal-projects.controller.ts` - Project CRUD, collections, nested building creation
- `developer-portal-buildings.controller.ts` - Building CRUD, collections, nested unit management

### Change requests integration

Developer portal uses both direct Propwise Labs API edits and change request flows depending on the operation type and approval requirements.

## Inventory module

**Path:** `src/modules/real-estate/inventory/`

Org-scoped inventory unit claims and unit access requests with standard RLS. Manages organization-owned property inventory with unit number privacy controls.

### Entities

- `InventoryUnit` - Core inventory unit claims with Propwise Labs property references
- `InventoryUnitOwnership` - Ownership tracking
- `InventoryUnitTransaction` - Transaction history

### Access control

Uses `AuthGuard + AccessGuard` with `ResourceType.UNIT` permissions:
- `real_estate.view`
- `real_estate.manage`

## Portal syndication module

**Path:** `src/modules/real-estate/portal-syndication/`

Org-scoped portal listing, configuration, and sync state for marketing inventory units across external property portals.

### Entities

- Portal listing configurations
- Portal credentials and sync state
- Per-portal sync status tracking

### Access control

Uses standard RLS via `TenantContext` for org-scoped portal management.

## Property interest module

**Path:** `src/modules/real-estate/property-interest/`

Lead and deal property interests with matching engine. Connects CRM entities to Propwise Labs properties for lead qualification and deal matching.

### Entities

- `LeadPropertyInterest` - Lead property preferences
- `DealPropertyInterest` - Deal property targets
- `PropertyMatch` - Matching engine results

### Services

- Property matching engine using `PropwiseLabsService`
- Interest management service

## Org-scoped vs global

- **Org-scoped** (standard RLS): Entity has `@ManyToOne(() => Organization) organization!: Organization;`
- **Global** (no RLS): Entity has NO organization FK. Used for DeveloperUser, etc.
- **Cross-org** (custom RLS): Entity references multiple organizations. Needs custom SQL policies.
- **External API** (no RLS): Data lives in Propwise Labs API, accessed via `PropwiseLabsService`.

## Access control summary

### Which modules use AccessGuard

| Module | Guard | @CheckAccess | Reason |
| --- | --- | --- | --- |
| propwise-labs | `EitherAuthGuard` | No | Global data, accepts CRM or developer tokens |
| off-plan | `EitherAuthGuard` | No | Global data, publication visibility applied |
| developer-portal | `DeveloperAuthGuard` | No | Separate auth system, own JWT |
| **inventory** | `AuthGuard + AccessGuard` | **Yes** | Org-scoped, uses `ResourceType.UNIT` |
| **portal-syndication** | `AuthGuard + AccessGuard` | **Yes** | Org-scoped, portal management permissions |
| **property-interest** | `AuthGuard + AccessGuard` | **Yes** | Org-scoped, CRM permission keys |

### Permission keys

| Module | Permissions |
| --- | --- |
| inventory | `real_estate.view`, `real_estate.manage` |
| portal-syndication | `real_estate.view`, `real_estate.manage` |
| property-interest | `crm.view`, `crm.manage` |
