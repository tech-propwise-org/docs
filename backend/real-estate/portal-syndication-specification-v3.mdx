## Module Overview

The Portal Syndication Module allows real estate agents to publish property listings to three UAE property portals directly from PropWise CRM, and automatically receive leads back into the CRM pipeline.

<Info>
This is version 3.0 of the specification, featuring an architectural revision that introduces the `Listing` entity as the marketing layer between `InventoryUnit` (inventory) and `ListingPortalSync` (per-portal state).
</Info>

### Three-Tier Architecture

The module uses a clean separation of concerns across three tiers:

```
InventoryUnit  →  Listing  →  ListingPortalSync
  (inventory)     (marketing)   (per-portal state)
```

<CardGroup cols={3}>
  <Card title="InventoryUnit" icon="building">
    What the unit *is* (rooms, area, price, physical attributes). Unchanged by portal syndication logic.
  </Card>
  <Card title="Listing" icon="megaphone">
    How the unit is *marketed* (title, descriptions, permit number, portal classifications, marketing media). Created by an agent from an `InventoryUnit`.
  </Card>
  <Card title="ListingPortalSync" icon="sync">
    Where the listing is *published* and its current state on each portal.
  </Card>
</CardGroup>

<Note>
This separation ensures `InventoryUnit` stays a clean inventory record and the `Listing` layer can eventually support off-plan units (`refUnitId`) without any structural change to the sync system.
</Note>

### Integration Model Per Portal

| Portal | Listing Syndication | Lead Ingestion | Listing Timing |
|---|---|---|---|
| Property Finder | REST API Push (JSON) | Webhook push (primary) + REST poll fallback (15 min) | Real-time (seconds) |
| Bayut | XML Feed Pull (unified) | Pull API polling — scheduled every 15 min | 30 min – 2 hr delay |
| dubizzle | XML Feed Pull (same as Bayut) | Pull API polling — same API + endpoint as Bayut | 30 min – 2 hr delay |

<Warning>
**Bayut / dubizzle lead ingestion note:** Bayut and dubizzle share one API endpoint and one Bearer token (per agency). The `source` field in each lead response (`"bayut"` or `"dubizzle"`) determines which `LeadSource` enum value is used when the CRM lead is created. The Bearer token is stored encrypted in the existing `apiKey` field of the Bayut `PortalConfiguration` row — no new credential columns are needed.
</Warning>

### Data Flow Rules

<Check>
- Listings flow **one direction only**: CRM → portals (CRM always wins)
- Leads flow **one direction only**: portals → CRM
- Portal data **never** overwrites CRM data
- `Listing` is the **single source of truth** for listing (marketing) content
- `InventoryUnit` is the **single source of truth** for unit inventory data
</Check>

### Listing Sync Status

Nine states covering the full Property Finder async lifecycle and feed-based portal model:

```typescript
export enum ListingSyncStatus {
  DRAFT = 'draft',                           // Not yet publishable (validation failed or not requested)
  PENDING = 'pending',                       // Job queued, adapter call in progress
  AWAITING_CONFIRMATION = 'awaiting_confirmation', // PF: create+publish called, waiting for webhook
  PUBLISHED = 'published',                   // Live on portal
  REJECTED = 'rejected',                     // PF rejected listing (compliance/quality)
  COMPLIANCE_ACTION = 'compliance_action',   // PF flagged listing for action
  ERROR = 'error',                           // Sync failed after retries
  INSUFFICIENT_CREDITS = 'insufficient_credits', // PF: balance < publish cost; agent retries after top-up
  REMOVED = 'removed',                       // Taken down (disabled, archived, deleted)
}
```

<Note>
**Note for Bayut/dubizzle:** Status goes directly `draft → published` on enable (no `awaiting_confirmation`), and `published → removed` on disable. Feed portals have no rejection mechanism.
</Note>

### Architecture Note — Three-Tier Data Model

```
InventoryUnit  ──1:many──▶  Listing  ──1:many──▶  ListingPortalSync
  (inventory)                (marketing)             (per-portal state)
```

<CardGroup cols={3}>
  <Card title="InventoryUnit" icon="database">
    Fields describing **what the unit is** (rooms, area, price, bathrooms, location coords)
  </Card>
  <Card title="Listing" icon="bullhorn">
    Fields describing **how the unit is advertised** (title, descriptions, permit number, portal classifications, marketing media)
  </Card>
  <Card title="ListingPortalSync" icon="arrows-rotate">
    Per-portal state (publish status, portal listing ID, retry count)
  </Card>

<Info>
**Note on portal lead ingestion:** There is no `PortalLead` staging entity. Inbound leads from portal **webhooks (PF) and scheduled Pull API polling (Bayut/dubizzle)** are queued directly via pg-boss (`portal-lead-ingestion` queue) and converted immediately to CRM `Lead` entities by the worker. Deduplication is handled by the existing `(organization, leadSource, referenceId)` unique index on `Lead`.
</Info>

<AccordionGroup>
  <Accordion title="Why a separate Listing entity?">
    1. **Clean separation:** Keeps `InventoryUnit` as a clean inventory record — agents editing unit details are not exposed to portal marketing fields.
    2. **Future-proofs off-plan support:** `Listing` holds an optional `refUnitId` (reference DB integer ID), so off-plan units can be published without changing the sync system.
    3. **Workflow support:** Provides a natural home for any future admin approval workflow before syndication fans out to portals.
  </Accordion>
</AccordionGroup>

### Portal Configuration

Per-organization portal credentials and settings. One row per (org, portal).

```typescript
@Entity()
@Filter({ name: 'isDeleted', default: true, cond: { isDeleted: false } })
@Unique({ properties: ['organization', 'portal'] })
export class PortalConfiguration {
  @PrimaryKey({ type: UuidType })
  id: string = v4();

  @ManyToOne(() => Organization)
  @Index()
  organization!: Organization;

  @Enum(() => Portal)
  @Property({ type: 'text' })
  portal!: Portal;

  @Property({ type: 'boolean', default: false })
  isEnabled: boolean = false;

  // PF credentials (encrypted at application layer)
  @Property({ type: 'text', nullable: true })
  apiKey?: string;

  @Property({ type: 'text', nullable: true })
  apiSecret?: string;

  // Auto-generated on creation for Bayut/dubizzle
  @Property({ nullable: true })
  feedUrl?: string;

  // PF webhook HMAC secret — used only for verifying X-Signature on PF webhook payloads
  @Property({ nullable: true })
  webhookSecret?: string;

  // Feed URL HMAC secret — used only for generating/validating the Bayut/dubizzle feed URL token
  // Separate from webhookSecret so each can be rotated independently
  @Property({ nullable: true })
  feedSecret?: string;

  // PF API key expiry date — visible in PF Expert when the key is generated (max 365 days)
  @Property({ type: 'timestamptz', nullable: true })
  apiKeyExpiresAt?: Date;

  // Tracks the timestamp of the last successful Bayut/dubizzle lead poll
  @Property({ type: 'timestamptz', nullable: true })
  lastLeadPollAt?: Date;

  @Property({ type: 'timestamptz' })
  createdAt: Date = new Date();

  @Property({ type: 'timestamptz', onUpdate: () => new Date() })
  updatedAt: Date = new Date();

  @Property({ type: 'boolean', default: false })
  isDeleted: boolean = false;
}
```

## Core Services

### Listing Service

<Steps>
  <Step title="Create listing from inventory unit">
    Agents create a `Listing` from an existing `InventoryUnit`, copying core data and adding marketing content.
  </Step>
  <Step title="Validate listing data">
    Ensure all required fields are present for the target portals before enabling syndication.
  </Step>
  <Step title="Update marketing content">
    Modify titles, descriptions, classifications, and media URLs without affecting the underlying inventory.
  </Step>
  <Step title="Archive listings">
    Mark listings as archived, triggering removal from all portals.
  </Step>
</Steps>

### Syndication Service

Central orchestrator for all portal syndication operations. Handles the coordination between listings and their portal sync states.

<Note>
The syndication service acts as the main entry point for all portal operations, delegating to specific portal adapters and managing the overall sync lifecycle.
</Note>

### Portal Lead Worker Service

<Warning>
This service consumes the `portal-lead-ingestion` pg-boss queue and converts inbound portal leads directly into CRM `Lead` entities. No intermediate staging entity is used.
</Warning>

Key responsibilities:
- Process webhook payloads from Property Finder
- Handle scheduled lead polling results from Bayut/dubizzle
- Deduplicate leads using existing CRM mechanisms
- Transform portal lead data into CRM format

## Data Flow and Integration Patterns

### Listing Publication Flow

<Steps>
  <Step title="Agent creates listing">
    Agent selects an `InventoryUnit` and creates a `Listing` with marketing content
  </Step>
  <Step title="Portal selection">
    Agent chooses which portals to publish to and enables syndication
  </Step>
  <Step title="Validation">
    System validates listing data against each portal's requirements
  </Step>
  <Step title="Publication">
    For each enabled portal, a `ListingPortalSync` record is created and the appropriate adapter is called
  </Step>
  <Step title="Status tracking">
    Sync status is updated based on portal responses (immediate for feeds, webhook-driven for Property Finder)
  </Step>
</Steps>

### Lead Ingestion Flow

<Tabs>
  <Tab title="Property Finder">
    1. **Webhook received** at `/api/portal-webhook/property-finder`
    2. **HMAC verification** using `webhookSecret`
    3. **Queue job** via pg-boss (`portal-lead-ingestion`)
    4. **Worker processes** and creates CRM `Lead`
    5. **Fallback polling** every 15 minutes for missed webhooks
  </Tab>
  <Tab title="Bayut/dubizzle">
    1. **Scheduled polling** every 15 minutes via `@Cron`
    2. **API call** to shared Bayut/dubizzle endpoint
    3. **Lead parsing** with source detection (`"bayut"` or `"dubizzle"`)
    4. **Queue jobs** via pg-boss for each new lead
    5. **Worker creates** CRM `Lead` with appropriate `LeadSource`
  </Tab>
</Tabs>

### Error Handling and Retry Logic

<Check>
- **Exponential backoff** for failed sync attempts
- **Maximum retry limits** to prevent infinite loops
- **Dead letter queues** for permanently failed jobs
- **Manual retry capabilities** through the admin interface
- **Comprehensive logging** for debugging and monitoring
</Check>

## Security and Authentication

### Credential Storage

All portal credentials are encrypted at the application layer before storage:

```typescript
// Property Finder credentials
apiKey: string;        // JWT token (encrypted)
apiSecret: string;     // Secret for JWT generation (encrypted)

// Bayut/dubizzle credentials  
apiKey: string;        // Bearer token (encrypted, reused field)

// HMAC secrets
webhookSecret: string; // For Property Finder webhook verification
feedSecret: string;    // For Bayut/dubizzle feed URL generation
```

### API Security

<Warning>
- **Webhook endpoints** must verify HMAC signatures
- **Feed URLs** include time-limited tokens
- **API keys** have expiration monitoring and alerts
- **Rate limiting** applies to all portal API calls
</Warning>

## Queue Architecture

The module uses pg-boss for reliable background job processing:

### Job Types

| Queue Name | Purpose | Retry Policy |
|---|---|---|
| `portal-listing-sync` | Publish/unpublish listings to portals | 3 retries, exponential backoff |
| `portal-lead-ingestion` | Process inbound leads from portals | 5 retries, immediate + exponential |
| `portal-image-processing` | Validate and optimize listing images | 2 retries, immediate |
| `portal-reconciliation` | Daily sync status reconciliation | No retries (manual investigation) |

### Worker Configuration

```typescript
// High priority for lead ingestion
@OnWorkerEvent({ name: 'portal-lead-ingestion', options: { 
  teamSize: 5, 
  teamConcurrency: 2 
}})

// Standard priority for listing sync
@OnWorkerEvent({ name: 'portal-listing-sync', options: { 
  teamSize: 3, 
  teamConcurrency: 1 
}})
```
