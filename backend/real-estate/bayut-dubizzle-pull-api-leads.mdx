<Note>
This document describes the **external API contract** defined by Bayut for fetching lead and view data. It covers endpoint configuration, authorization requirements, query parameters, and sample responses for both leads (`is_trulead=1`) and views (`is_trulead=0`).
</Note>

## Overview

The Bayut Pull API enables clients to retrieve lead data from Bayut's platform. This comprehensive API provides:

- **Lead Data**: Actual customer inquiries and interactions (`is_trulead=1`)
- **View Data**: Count-based metrics for tracking engagement (`is_trulead=0`)
- **Multiple Channels**: Support for email, WhatsApp, SMS, and phone interactions
- **Flexible Targeting**: Filter by listing, agent, or agency

## Endpoint

The base endpoint for all Pull API requests:

```
www.bayut.com/api-v7/stats/website-client-leads
```

## Authorization

<Warning>
Each client receives a unique API key that must be included as a bearer token in the Authorization header for all requests.
</Warning>

```http
Authorization: Bearer <API_KEY>
```

## Basic Request Example

<CodeGroup>

```bash cURL
curl --location \
  'www.bayut.com/api-v7/stats/website-client-leads?timestamp=2025-03-10%2000%3A00%3A00&target=listing&type=email&is_trulead=1' \
  --header 'Authorization: Bearer <API_KEY>'
```

</CodeGroup>

<Info>
The `timestamp` parameter must be URL-encoded. For example, `2025-03-10 00:00:00` becomes `2025-03-10%2000%3A00%3A00` where `%20` represents spaces and `%3A` represents colons.
</Info>

## Query Parameters

| Parameter | Description | Possible Values | Required |
|-----------|-------------|-----------------|----------|
| `type` | The type of lead/interaction | `whatsapp`, `sms`, `phone`, `email`, `call_logs`, `story_leads` | Yes |
| `timestamp` | Request date and time | Format: `YYYY-MM-DD HH:MM:SS` | Yes |
| `target` | Entity being targeted | `listing`, `agent`, `agency` | Yes* |
| `is_trulead` | Data type selector | `0` (views), `1` (leads) | Yes |

<Note>
*The `target` parameter is not required for `call_logs` and `story_leads` types.
</Note>

## View Data (`is_trulead=0`)

<Info>
When `is_trulead=0`, the API returns view count metrics instead of detailed lead information. This is useful for tracking engagement and interaction statistics.
</Info>

View data provides aggregated metrics for:

    Measure phone number reveals and call button interactions

</CardGroup>

### SMS and Phone Views

<AccordionGroup>
  <Accordion title="SMS Views">
    SMS views track when users click on SMS contact options:
    
    - **SMS Listing Views**: `target=listing&type=sms&is_trulead=0`
    - **SMS Agent Views**: `target=agent&type=sms&is_trulead=0`
  </Accordion>
  
  <Accordion title="Phone Views">
    Phone views track when users reveal or interact with phone numbers:
    
    - **Phone Listing Views**: `target=listing&type=phone&is_trulead=0`
    - **Phone Agent Views**: `target=agent&type=phone&is_trulead=0`
    - **Phone Agency Views**: `target=agency&type=phone&is_trulead=0`
  </Accordion>
</AccordionGroup>

## Best Practices

<Steps>
  <Step title="Implement proper authentication">
    Always include the Bearer token in your Authorization header and keep your API key secure.
  </Step>
  
  <Step title="Handle URL encoding">
    Ensure timestamp parameters are properly URL-encoded before making requests.
  </Step>
  
  <Step title="Process delivery notifications">
    For WhatsApp leads, track the delivery_notifications array to understand message status.
  </Step>
  
  <Step title="Distinguish leads vs views">
    Use `is_trulead=1` for actionable lead data and `is_trulead=0` for engagement metrics.
  </Step>
</Steps>

<Tip>
Consider implementing retry logic with exponential backoff for handling rate limits and temporary API unavailability.
</Tip>
