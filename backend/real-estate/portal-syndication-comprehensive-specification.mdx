## Overview

The Portal Syndication Module enables real estate agents to publish property listings to three UAE property portals (Property Finder, Bayut, and dubizzle) directly from PropWise CRM, with automated lead ingestion back into the CRM pipeline.

<Note>
This module implements a three-tier architecture that separates inventory data, marketing content, and per-portal syndication state for maximum flexibility and data integrity.
</Note>

## Three-Tier Architecture

The module uses a sophisticated three-tier data model to separate concerns:

```
InventoryUnit  →  Listing  →  ListingPortalSync
  (inventory)     (marketing)   (per-portal state)
```

<Tabs>
  <Tab title="InventoryUnit">
    **What the unit IS** - Physical attributes like rooms, area, price, bathrooms, location coordinates. This layer remains unchanged by portal syndication logic.
  </Tab>
  <Tab title="Listing">
    **How the unit is MARKETED** - Marketing content like titles, descriptions, permit numbers, portal classifications, and marketing media. Created by agents from an InventoryUnit.
  </Tab>
  <Tab title="ListingPortalSync">
    **Where the listing is PUBLISHED** - Per-portal state including publish status, portal listing IDs, and retry counts.
  </Tab>
</Tabs>

<Info>
This separation ensures `InventoryUnit` stays a clean inventory record while the `Listing` layer can eventually support off-plan units without structural changes to the sync system.
</Info>

## Integration Model

### Portal-Specific Integration Patterns

| Portal | Listing Syndication | Lead Ingestion | Timing |
|---|---|---|---|
| Property Finder | REST API Push (JSON) | Webhook push + REST poll fallback | Real-time (seconds) |
| Bayut | XML Feed Pull | Pull API polling (15 min) | 30 min – 2 hr delay |
| dubizzle | XML Feed Pull (unified with Bayut) | Pull API polling (shared with Bayut) | 30 min – 2 hr delay |

<Warning>
Bayut and dubizzle share one API endpoint and Bearer token per agency. The `source` field in lead responses determines which `LeadSource` enum value is used when creating CRM leads.
</Warning>

### Data Flow Rules

<Check>
**Listings flow one direction only**: CRM → portals (CRM always wins)
</Check>

<Check>
**Leads flow one direction only**: portals → CRM
</Check>

<Check>
Portal data **never** overwrites CRM data
</Check>

### Listing Sync Status

Nine states covering the full Property Finder async lifecycle and feed-based portal model:

<AccordionGroup>
  <Accordion title="Status Definitions">
    - **DRAFT** - Not yet publishable (validation failed or not requested)
    - **PENDING** - Job queued, adapter call in progress
    - **AWAITING_CONFIRMATION** - PF: create+publish called, waiting for webhook
    - **PUBLISHED** - Live on portal
    - **REJECTED** - PF rejected listing (compliance/quality)
    - **COMPLIANCE_ACTION** - PF flagged listing for action
    - **ERROR** - Sync failed after retries
    - **INSUFFICIENT_CREDITS** - PF: balance < publish cost
    - **REMOVED** - Taken down (disabled, archived, deleted)
  </Accordion>
</AccordionGroup>

<Note>
For Bayut/dubizzle: Status goes directly `draft → published` on enable (no `awaiting_confirmation`), and `published → removed` on disable. Feed portals have no rejection mechanism.
</Note>

## Entity Architecture

### Portal Configuration

Per-organization portal credentials and settings. One row per (organization, portal).

```typescript
@Entity()
@Filter({ name: 'isDeleted', default: true, cond: { isDeleted: false } })
@Unique({ properties: ['organization', 'portal'] })
export class PortalConfiguration {
  @PrimaryKey({ type: UuidType })
  id: string = v4();

  @ManyToOne(() => Organization)
  @Index()
  organization!: Organization;

  @Enum(() => Portal)
  @Property({ type: 'text' })
  portal!: Portal;

  @Property({ type: 'boolean', default: false })
  isEnabled: boolean = false;

  // Encrypted credentials
  @Property({ type: 'text', nullable: true })
  apiKey?: string;

  @Property({ type: 'text', nullable: true })
  apiSecret?: string;

  // Auto-generated for feed portals
  @Property({ nullable: true })
  feedUrl?: string;

  // Security secrets
  @Property({ nullable: true })
  webhookSecret?: string;

  @Property({ nullable: true })
  feedSecret?: string;

  // API key management
  @Property({ type: 'timestamptz', nullable: true })
  apiKeyExpiresAt?: Date;

  @Property({ type: 'timestamptz', nullable: true })
  lastLeadPollAt?: Date;
}
```

<Warning>
All sensitive data (API keys, secrets) must be encrypted at the application layer before database storage.
</Warning>

  @Property({ type: 'json', nullable: true })

## Lead Ingestion Architecture

### Property Finder Leads

<Steps>
  <Step title="Primary: Webhook Push">
    Real-time lead delivery via HMAC-verified webhook to `/webhooks/portals/property-finder`
  </Step>
  <Step title="Fallback: REST Polling">
    15-minute cron job polls REST API for any missed leads using timestamp-based queries
  </Step>
  <Step title="Immediate Processing">
    Both webhook and polling queue leads for immediate CRM conversion via pg-boss
  </Step>
</Steps>

### Bayut/dubizzle Leads

<Steps>
  <Step title="Unified API Polling">
    Single cron job (every 15 minutes) polls shared Bayut API endpoint
  </Step>
  <Step title="Source Differentiation">
    `source` field in API response (`"bayut"` or `"dubizzle"`) determines CRM `LeadSource`
  </Step>
  <Step title="Lead Type Normalization">
    7 lead types × 3 targets normalized into common CRM lead format
  </Step>
</Steps>

<Info>
No `PortalLead` staging entity exists. Leads are queued directly and converted immediately to CRM `Lead` entities with deduplication via existing `(organization, leadSource, referenceId)` unique index.
</Info>

### Access Control

- **Feed URLs** - Protected by HMAC tokens in query parameters

- **API Credentials** - Encrypted at rest, decrypted only during API calls
- **Portal Access** - Org-scoped with proper authorization checks

</Warning>

## Monitoring and Reconciliation

### Daily Reconciliation Process

<Steps>
  <Step title="Status Verification">
    Compare CRM listing states with portal API responses to detect drift
  </Step>
  <Step title="Missed Lead Detection">
    Query portal APIs for leads that may have been missed by webhooks/polling
  </Step>
  <Step title="Credit Balance Monitoring">
    Check Property Finder credit balances and alert on low funds
  </Step>
  <Step title="Performance Metrics">
    Track sync success rates, lead conversion rates, and error patterns
  </Step>
</Steps>

### Error Handling Strategy

- **Exponential Backoff** - Failed sync jobs retry with increasing delays
- **Circuit Breaker** - Temporarily disable portal sync after consecutive failures
- **Dead Letter Queue** - Store permanently failed jobs for manual investigation
- **Alert Integration** - Notify administrators of critical sync failures

<Note>
All errors are logged with correlation IDs for easy debugging and include sufficient context for troubleshooting portal integration issues.
</Note>
