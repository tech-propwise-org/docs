## Overview

This specification details the implementation of an **Off-Plan** directory feature that replaces the existing Real Estate section with a comprehensive building showcase. The feature displays all published buildings from developer portal users in a modern card grid view with advanced filters, 2GIS map integration, and detailed building profiles.

<Note>
Minimal backend changes are required. Most API endpoints already exist under `/reference/buildings`, `/reference/projects`, and `/reference/units`. The frontend consumes these with the `?type=off-plan` filter parameter.
</Note>

## Architecture Overview

### Buildings as Primary Entity

The off-plan directory uses **buildings** as the primary entity rather than projects:

- Buildings have their own `isPublished`, `priceFrom`, `coverImageUrl`, `status`, `completionDate` properties
- Buildings can override inherited fields from projects (status, area, community, description)
- A project may contain multiple buildings with different statuses and pricing

### Data Flow Architecture

```
Frontend (CRM App)
  └── OffPlanApi → GET /reference/buildings?type=off-plan     (card listing)
  └── OffPlanApi → GET /reference/buildings/:id               (detail view)
  └── OffPlanApi → GET /reference/buildings/:id/units/grouped (units by bedrooms)
  └── OffPlanApi → GET /reference/projects/map                (map markers)
  └── OffPlanApi → GET /reference/developers?q=               (filter dropdowns)
```

<Step title="Implement API Layer">
Create the off-plan API service that wraps existing reference endpoints.

**File: `src/services/api/off-plan.api.ts`**

```typescript
export interface OffPlanBuildingFilters {
  q?: string;
  status?: string;
  areaId?: number;
  communityId?: number;
  developerId?: number;
  propertyTypeId?: number;
  minPrice?: number;
  maxPrice?: number;
  bedrooms?: string;
  completionBefore?: string;
  completionAfter?: string;
  maxPreHandoverPercent?: number;
  page?: number;
  limit?: number;
  sortBy?: string;
  sortOrder?: 'asc' | 'desc';
}

export class OffPlanApi {
  static async searchBuildings(filters: OffPlanBuildingFilters) {
    return apiClient.get('/reference/buildings', {
      params: { ...filters, type: 'off-plan' },
    });
  }

  static async getBuildingDetail(id: number) {
    return apiClient.get(`/reference/buildings/${id}`);
  }

  static async getBuildingUnitsGrouped(buildingId: number) {
    return apiClient.get(`/reference/buildings/${buildingId}/units/grouped`);
  }

  static async getMapMarkers(filters?: MapMarkerFilters) {
    return apiClient.get('/reference/projects/map', { params: filters });
  }
}
```
</Step>

## Filter Implementation

The filters bar supports the following filter types:

<Tabs>
<Tab title="Basic Filters">
- **Search**: Text search across building names and descriptions
- **Developer**: Dropdown populated from `/reference/developers`
- **Area**: Dropdown populated from `/reference/areas`
- **Status**: Building status (EOI, On Sale, Announced, etc.)
</Tab>

<Tab title="Financial Filters">
- **Price Range**: Min/max price sliders
- **Payment Plan**: Max pre-handover percentage filter
- **Unit Type**: Property type from `/reference/property-types`
</Tab>

<Tab title="Property Filters">
- **Bedrooms**: Multi-select for studio, 1BR, 2BR, 3BR+
- **Handover**: Date range for completion dates
- **Unit Count**: Filter by number of available units
</Tab>
</Tabs>

## Backend Requirements

### New Query Parameter

Add `maxPreHandoverPercent` filter to the buildings search endpoint:

```sql
-- Filter buildings by payment plan pre-handover percentage
WHERE pp.on_booking_percentage + pp.construction_percentage <= :maxPreHandoverPercent
```
