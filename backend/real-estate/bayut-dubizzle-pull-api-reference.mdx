## Overview

The Bayut Pull API enables clients to retrieve lead and view data from the Bayut platform. This comprehensive reference covers:

- **Endpoint URL** - The specific web address for API requests
- **Authorization** - Required credentials and bearer token authentication
- **Query Parameters** - Filtering options to refine returned data
- **Sample Responses** - Examples for all lead types and view counts

<Info>
This API supports both **actual leads** (`is_trulead=1`) and **view counts** (`is_trulead=0`) across multiple communication channels including email, WhatsApp, SMS, and phone.
</Info>

## Endpoint

```
www.bayut.com/api-v7/stats/website-client-leads
```

This endpoint allows clients to fetch lead data based on specific query parameters.

## Authorization

Each client receives a unique API key that must be passed as a bearer token in the `Authorization` header.

<CodeGroup>
```bash Header format
Authorization: Bearer <API KEY>
```

```bash Complete CURL example
curl --location \
  'www.bayut.com/api-v7/stats/website-client-leads?timestamp=2025-03-10%2000%3A00%3A00&target=listing&type=email&is_trulead=1' \
  --header 'Authorization: Bearer <API KEY>'
```
</CodeGroup>

<Note>
The `timestamp` value must be URL-encoded: `2025-03-10 00:00:00` becomes `2025-03-10%2000%3A00%3A00` where `%20` = space and `%3A` = colon.
</Note>

## Query parameters

| Parameter | Description | Possible Values |
|-----------|-------------|-----------------|
| `type` | The type of lead | `whatsapp`, `sms`, `phone`, `email`, `call_logs`, `story_leads` |
| `timestamp` | The date and time of the request | Format: `YYYY-MM-DD HH:MM:SS` |
| `target` | The entity for which leads are fetched | `listing`, `agent`, `agency`, `story` |
| `is_trulead` | Whether it is a lead or just a views count | `0` (views), `1` (leads) |

<Tip>
Use `is_trulead=1` to retrieve actual leads with contact details. Use `is_trulead=0` to retrieve view count metrics without personal information.
</Tip>

## Best practices

<Steps>
  <Step title="Use timestamp filtering">
    Always include a `timestamp` parameter to retrieve leads from a specific time period. This prevents duplicate processing and improves performance.
  </Step>

  <Step title="Handle rate limits">
    Implement exponential backoff if you receive rate limit errors. Space out your API calls appropriately based on your data volume.
  </Step>

  <Step title="Store lead IDs">
    The `lead_id` field is unique for each lead. Store these IDs to prevent duplicate processing when polling the API regularly.
  </Step>

  <Step title="Monitor delivery notifications">
    For WhatsApp leads, track the `delivery_notifications` array to understand message delivery status and optimize follow-up timing.
  </Step>

  <Step title="Validate response data">
    Always validate that required fields are present before processing. Some fields may be empty strings or null depending on the lead type.
  </Step>
</Steps>
