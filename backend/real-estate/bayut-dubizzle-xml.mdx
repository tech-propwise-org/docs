<Note>
This document describes the **external XML contract** defined by Bayut for the unified feed format. For how PropWise produces this feed from internal data structures, see the Portal Syndication Specification.
</Note>

## XML Integration Overview

An XML integration is required to sync properties with Bayut and/or dubizzle in bulk from an external CRM. It allows clients to maintain their inventory on both portals by providing property information in a unified XML feed.

### Key Requirements

- The XML feed must be exposed as a **publicly accessible URL**
- The **Bayut system pulls** (reads) this URL on a schedule to sync updates
- Listings flow **one direction only**: CRM → portals (feed is the source of truth)

### Unified Feed Architecture

Bayut and dubizzle now share a **single combined XML feed**. The portal(s) a property appears on is controlled per-property by the `<Portals>` tag.

<Warning>
Clients with existing integrations must retire separate dubizzle feeds and update their Bayut XML structure to use the new `<Portals>` attribute for portal targeting.
</Warning>

## XML Document Structure

The XML feed consists of two main components:

1. **Header tags** — the document envelope
2. **Property attributes** — per-property field data

### Root Structure

```xml
<Properties>
  <Property> <!-- Property 1 attributes --> </Property>
  <Property> <!-- Property 2 attributes --> </Property>
</Properties>
```

<Tip>
All textual values should be wrapped in `<![CDATA[ ... ]]>` to prevent HTML, Arabic text, ampersands, and URLs from breaking XML parsing.
</Tip>

## Property Attributes

### Core Property Details

<AccordionGroup>
<Accordion title="Identity & Status Fields">

| Tag | Description | Allowed Values |
|---|---|---|
| `<Property_Ref_No>` | Unique identifier from client's CRM | Must be unique; duplicates marked **Inactive** |
| `<Permit_Number>` | Trakheesi (DLD) or Madhmoun (ADREC) permit number | Required for listing to go live |
| `<Property_Status>` | Publication status | `live`, `deleted` |
| `<Last_Updated>` | Last modified date from CRM | `YYYY-MM-DD HH:mm:ss` |

</Accordion>

<Accordion title="Classification Fields">

| Tag | Description | Allowed Values |
|---|---|---|
| `<Property_purpose>` | Sale or rental purpose | `Buy`, `Rent` |
| `<Property_Type>` | Property category | `Villa`, `Apartment`, `Office`, `Shop`, `Warehouse`, `Factory`, `Labour Camp`, `Other Commercial`, `Commercial Building`, `Residential Floor`, `Commercial Floor`, `Residential Land`, `Commercial Land`, `Townhouse`, `Residential Building`, `Hotel Apartment`, `Loft Apartment`, `Pent House` |
| `<Off_plan>` | Ready or off-plan status | `Yes`, `No` |

</Accordion>

<Accordion title="Property Specifications">

| Tag | Description | Allowed Values |
|---|---|---|
| `<Property_Size>` | Area of the property | Number |
| `<Property_Size_Unit>` | Area measurement unit | Preferred: `SQFT` |
| `<plotArea>` | Plot area in square feet | Number |
| `<Bedrooms>` | Number of bedrooms | `-1` (studio), `1`-`10`, `10+` |
| `<Bathrooms>` | Number of bathrooms | `1`-`10` |
| `<Furnished>` | Furnishing status | `Yes`, `No`, `Partly` |

</Accordion>

<Accordion title="Pricing Fields">

| Tag | Description | Required When |
|---|---|---|
| `<Price>` | Sale price or rent amount | Always |
| `<Rent_Frequency>` | Rental period | Only when `Property_purpose` is `Rent` |

**Rent Frequency Values:** `Daily`, `Weekly`, `Monthly`, `Yearly`

</Accordion>
</AccordionGroup>

### Portal Targeting

The `<Portals>` tag controls which portal(s) the listing appears on:

<CodeGroup>
```xml Both Portals
<Portals>
  <portal>bayut</portal>
  <portal>dubizzle</portal>
</Portals>
```

```xml Bayut Only
<Portals>
  <portal>bayut</portal>
</Portals>
```

```xml dubizzle Only
<Portals>
  <portal>dubizzle</portal>
</Portals>
```
</CodeGroup>

### Features

Property amenities are listed using nested `<Feature>` tags:

```xml
<Features>
  <Feature>Swimming Pool</Feature>
  <Feature>Gym</Feature>
  <Feature>Parking</Feature>
</Features>
```

### Off-Plan Properties

When `<Off_plan>` is `Yes`, additional fields are required:

| Tag | Description | Required For |
|---|---|---|
| `<offplanDetails_saleType>` | Sale type | All off-plan properties |
| `<offplanDetails_dldWaiver>` | DLD waiver percentage (0-100) | Off-plan `New` properties |
| `<offplanDetails_originalPrice>` | Original property price | Off-plan `Resale` properties |
| `<offplanDetails_amountPaid>` | Amount already paid | Off-plan `Resale` properties |

**Sale Type Values:** `New`, `Resale`

## Location Details

Location hierarchy follows Dubai's administrative structure:

```xml
<City>Dubai</City>
<Locality>Jumeirah Lakes Towers</Locality>
<Sub_Locality>Cluster C</Sub_Locality>
<Tower_Name>Pretoni Tower</Tower_Name>
```

| Tag | Description | Example |
|---|---|---|
| `<City>` | Emirate/City | Dubai, Abu Dhabi |
| `<Locality>` | Main area/district | Dubai Marina, Downtown Dubai |
| `<Sub_Locality>` | Sub-area | Cluster C, Burj Views |
| `<Tower_Name>` | Building name | Marina Tower, Burj Khalifa |

## Validation Rules

<Steps>
<Step title="Required Fields">
- `Property_Ref_No` must be unique across the feed
- `Permit_Number` is required for listing activation
- At least one publicly accessible image should be provided
</Step>

<Step title="Conditional Requirements">
- `Rent_Frequency` required only when `Property_purpose` is `Rent`
- Off-plan detail tags required only when `Off_plan` is `Yes`
- Specific off-plan fields vary by `offplanDetails_saleType`
</Step>

<Step title="Data Integrity">
- Duplicate `Property_Ref_No` values result in only one active listing
- `Property_Status` of `deleted` should remain in feed for one crawl cycle
- All media URLs must be publicly accessible
</Step>
</Steps>
