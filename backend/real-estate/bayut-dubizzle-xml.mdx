## Overview

The Bayut & dubizzle XML feed is a unified syndication format that allows PropWise to publish property listings to both portals through a single XML endpoint. This feed serves as the authoritative source for property data that gets synchronized with the external portals.

<Note>
The XML feed must be exposed as a **publicly accessible URL** that the Bayut system can pull from on a scheduled basis. Data flows one direction only: CRM → portals.
</Note>

## Integration Architecture

### Unified Feed Structure

Bayut and dubizzle now share a **single combined XML feed**. The portal(s) where a property appears is controlled per-property using the `<Portals>` tag, eliminating the need for separate feeds.

<Tabs>
  <Tab title="Both Portals">
    Properties can be published to both Bayut and dubizzle simultaneously by including both portal identifiers.
  </Tab>
  <Tab title="Single Portal">
    Properties can be targeted to only Bayut or only dubizzle by specifying the desired portal.
  </Tab>
</Tabs>

### Feed Requirements

- XML feed must be publicly accessible via HTTP/HTTPS
- Bayut system pulls the feed on a scheduled basis
- All textual content should be wrapped in `<![CDATA[]]>` tags
- Duplicate `Property_Ref_No` values result in only one live instance

## XML Document Structure

The XML document follows a two-level hierarchy:

<Steps>
  <Step title="Root Element">
    The document starts with a `<Properties>` root element that contains all property listings.
  </Step>
  <Step title="Property Elements">
    Each individual property is represented as a nested `<Property>` element with its attributes.
  </Step>
</Steps>

```xml
<Properties>
  <Property>
    <!-- Property 1 attributes -->
  </Property>
  <Property>
    <!-- Property 2 attributes -->
  </Property>
</Properties>
```

## Core Property Attributes

### Identity & Status

<AccordionGroup>
  <Accordion title="Property Reference & Identification">
    | Tag | Description | Rules |
    |---|---|---|
    | `<Property_Ref_No>` | Unique identifier from the CRM system | Must be unique; duplicates marked inactive |
    | `<Permit_Number>` | Trakheesi/Madhmoun permit number | Required for listing to go live |
    | `<Property_Status>` | Listing status on portals | `live`, `deleted` |
    | `<Last_Updated>` | Last modification timestamp | Format: `YYYY-MM-DD HH:mm:ss` |
  </Accordion>
</AccordionGroup>

### Property Classification

```xml
<Property_purpose><![CDATA[ Buy ]]></Property_purpose>
<Property_Type><![CDATA[ Apartment ]]></Property_Type>
<Off_Plan><![CDATA[ No ]]></Off_Plan>
```

**Property Purpose Values:**
- `Buy` - For sale properties
- `Rent` - For rental properties

**Supported Property Types:**
- `Villa`
- `Apartment`
- `Townhouse`
- `Office`
- `Shop`
- `Warehouse`
- `Factory`
- `Hotel Apartment`
- `Loft Apartment`
- `Pent House`
- And others as defined by portal specifications

### Portal Targeting

The `<Portals>` tag controls which platform(s) the listing appears on:

<CodeGroup>
  ```xml Both Portals
  <Portals>
    <Portal><![CDATA[ Bayut ]]></Portal>
    <Portal><![CDATA[ dubizzle ]]></Portal>
  </Portals>
  ```

  ```xml Bayut Only
  <Portals>
    <Portal><![CDATA[ Bayut ]]></Portal>
  </Portals>
  ```

  ```xml dubizzle Only
  <Portals>
    <Portal><![CDATA[ dubizzle ]]></Portal>
  </Portals>
  ```
</CodeGroup>

## Property Specifications

### Size & Layout

| Tag | Description | Values/Rules |
|---|---|---|
| `<Property_Size>` | Property area (numeric) | Number |
| `<Property_Size_Unit>` | Area measurement unit | Preferred: `SQFT` |
| `<plotArea>` | Plot area in square feet | Number |
| `<Bedrooms>` | Number of bedrooms | `-1` (studio), `1`-`10`, `10+` |
| `<Bathrooms>` | Number of bathrooms | `1`-`10` |
| `<Furnished>` | Furnishing status | `Yes`, `No`, `Partly` |

<Warning>
The `<Bedrooms>` value `-1` specifically represents studio apartments. Use `10+` for properties with more than 10 bedrooms.
</Warning>

### Features

Property features are specified using nested `<Feature>` tags:

```xml
<Features>
  <Feature><![CDATA[ Swimming Pool ]]></Feature>
  <Feature><![CDATA[ Gym ]]></Feature>
  <Feature><![CDATA[ Parking ]]></Feature>
  <Feature><![CDATA[ Balcony ]]></Feature>
</Features>
```

## Pricing Information

### Basic Pricing

```xml
<Price><![CDATA[ 1500000 ]]></Price>
```

### Rental Properties

For rental properties, include the rent frequency:

```xml
<Property_purpose><![CDATA[ Rent ]]></Property_purpose>
<Price><![CDATA[ 80000 ]]></Price>
<Rent_Frequency><![CDATA[ Yearly ]]></Rent_Frequency>
```

**Rent Frequency Options:**
- `Daily`
- `Weekly` 
- `Monthly`
- `Yearly`

<Info>
The `<Rent_Frequency>` tag is required only when `<Property_purpose>` is `Rent`.
</Info>

## Off-Plan Properties

When `<Off_Plan>` is set to `Yes`, additional details are required:

### Sale Type Requirements

<Tabs>
  <Tab title="New Off-Plan">
    ```xml
    <Off_Plan><![CDATA[ Yes ]]></Off_Plan>
    <offplanDetails_saleType><![CDATA[ New ]]></offplanDetails_saleType>
    <offplanDetails_dldWaiver><![CDATA[ 5 ]]></offplanDetails_dldWaiver>
    ```
    **Required:** `offplanDetails_dldWaiver` (0-100)
  </Tab>
  
  <Tab title="Resale Off-Plan">
    ```xml
    <Off_Plan><![CDATA[ Yes ]]></Off_Plan>
    <offplanDetails_saleType><![CDATA[ Resale ]]></offplanDetails_saleType>
    <offplanDetails_originalPrice><![CDATA[ 1200000 ]]></offplanDetails_originalPrice>
    <offplanDetails_amountPaid><![CDATA[ 300000 ]]></offplanDetails_amountPaid>
    ```
    **Required:** `offplanDetails_originalPrice` and `offplanDetails_amountPaid`
  </Tab>
</Tabs>

</Tip>

## Location Information

Location data follows a hierarchical structure:

```xml
<City><![CDATA[ Dubai ]]></City>
<Locality><![CDATA[ Jumeirah Lakes Towers ]]></Locality>
<Sub_Locality><![CDATA[ Cluster C ]]></Sub_Locality>
<Tower_Name><![CDATA[ Pretoni Tower ]]></Tower_Name>
```

**Example hierarchy:**
- **City/Emirate:** Dubai
- **Main Locality:** Jumeirah Lakes Towers  
- **Sub-locality:** Cluster C
- **Building:** Pretoni Tower

## Implementation Notes

<Check>
**PropWise Implementation Details:**
- `Property_Ref_No` is generated as `UNIT-{orgShortCode}-{listing.id}`
- `Property_Status` reflects the `ListingPortalSync` status
- `Property_Type` is mapped from internal `LayoutType` enums
- `Last_Updated` uses the latest of listing/sync timestamps
</Check>

For detailed field mapping and internal data transformation logic, see the Portal Syndication Specification document, section 6.4 (BayutDubizzleFeedAdapter).
