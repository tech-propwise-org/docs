## Module overview

The Portal Syndication Module allows real estate agents to publish property listings to three UAE property portals directly from PropWise CRM, and automatically receive leads back into the CRM pipeline.

### Three-tier architecture

```
InventoryUnit  →  Listing  →  ListingPortalSync
  (inventory)     (marketing)   (per-portal state)
```

<AccordionGroup>
<Accordion title="Architecture layer descriptions">
- **`InventoryUnit`** — what the unit *is* (rooms, area, price, physical attributes). Unchanged by portal syndication logic.
- **`Listing`** — how the unit is *marketed* (title, descriptions, permit number, portal classifications, marketing media). Created by an agent from a `InventoryUnit`.
- **`ListingPortalSync`** — where the listing is *published* and its current state on each portal.
</Accordion>
</AccordionGroup>

This separation ensures `InventoryUnit` stays a clean inventory record and the `Listing` layer can eventually support off-plan units (`refUnitId`) without any structural change to the sync system.

### Integration model per portal

| Portal | Listing Syndication | Lead Ingestion | Listing Timing |
|---|---|---|---|
| Property Finder | REST API Push (JSON) | Webhook push (primary) + REST poll fallback (15 min) | Real-time (seconds) |
| Bayut | XML Feed Pull (unified) | Pull API polling — scheduled every 15 min | 30 min – 2 hr delay |
| dubizzle | XML Feed Pull (same as Bayut) | Pull API polling — same API + endpoint as Bayut | 30 min – 2 hr delay |

<Info>
**Bayut / dubizzle lead ingestion note:** Bayut and dubizzle share one API endpoint and one Bearer token (per agency). The `source` field in each lead response (`"bayut"` or `"dubizzle"`) determines which `LeadSource` enum value is used when the CRM lead is created. The Bearer token is stored encrypted in the existing `apiKey` field of the Bayut `PortalConfiguration` row — no new credential columns are needed.
</Info>

### Data flow rules

- Listings flow **one direction only**: CRM → portals (CRM always wins)
- Leads flow **one direction only**: portals → CRM
- Portal data **never** overwrites CRM data
- `Listing` is the **single source of truth** for listing (marketing) content
- `InventoryUnit` is the **single source of truth** for unit inventory data

### Listing sync status

Nine states covering the full PF async lifecycle and feed-based portal model:

```typescript
export enum ListingSyncStatus {
  DRAFT = 'draft',                           // Not yet publishable (validation failed or not requested)
  PENDING = 'pending',                       // Job queued, adapter call in progress
  AWAITING_CONFIRMATION = 'awaiting_confirmation', // PF: create+publish called, waiting for webhook
  PUBLISHED = 'published',                   // Live on portal
  REJECTED = 'rejected',                     // PF rejected listing (compliance/quality)
  COMPLIANCE_ACTION = 'compliance_action',   // PF flagged listing for action
  ERROR = 'error',                           // Sync failed after retries
  INSUFFICIENT_CREDITS = 'insufficient_credits', // PF: balance < publish cost; agent retries after top-up
  REMOVED = 'removed',                       // Taken down (disabled, archived, deleted)
}
```

<Note>
**Note for Bayut/dubizzle:** Status goes directly `draft → published` on enable (no `awaiting_confirmation`), and `published → removed` on disable. Feed portals have no rejection mechanism.
</Note>

## Entity definitions

### Architecture note — three-tier data model

```
InventoryUnit  ──1:many──▶  Listing  ──1:many──▶  ListingPortalSync
  (inventory)                (marketing)             (per-portal state)
```

<AccordionGroup>
<Accordion title="Field distribution across tiers">
- Fields describing **what the unit is** (rooms, area, price, bathrooms, location coords) live on `InventoryUnit`.
- Fields describing **how the unit is advertised** (title, descriptions, permit number, portal classifications, marketing media) live on `Listing`.
- Per-portal state (publish status, portal listing ID, retry count) lives on `ListingPortalSync`.
</Accordion>

<Accordion title="Portal lead ingestion">
There is no `PortalLead` staging entity. Inbound leads from portal **webhooks (PF) and scheduled Pull API polling (Bayut/dubizzle)** are queued directly via pg-boss (`portal-lead-ingestion` queue) and converted immediately to CRM `Lead` entities by the worker. Deduplication is handled by the existing `(organization, leadSource, referenceId)` unique index on `Lead`.
</Accordion>

<Accordion title="Why a separate Listing entity?">
1. Keeps `InventoryUnit` as a clean inventory record — agents editing unit details are not exposed to portal marketing fields.
2. Future-proofs off-plan support: `Listing` holds an optional `refUnitId` (reference DB integer ID), so off-plan units can be published without changing the sync system.
3. Provides a natural home for any future admin approval workflow before syndication fans out to portals.
</Accordion>
</AccordionGroup>

### Portal configuration

Per-organization portal credentials and settings. One row per (org, portal).

```typescript
@Entity()
@Filter({ name: 'isDeleted', default: true, cond: { isDeleted: false } })
@Unique({ properties: ['organization', 'portal'] })
export class PortalConfiguration {
  @PrimaryKey({ type: UuidType })
  id: string = v4();

  @ManyToOne(() => Organization)
  @Index()
  organization!: Organization;

  @Enum(() => Portal)
  @Property({ type: 'text' })
  portal!: Portal;

  @Property({ type: 'boolean', default: false })
  isEnabled: boolean = false;

  // PF credentials (encrypted at application layer)
  @Property({ type: 'text', nullable: true })
  apiKey?: string;

  @Property({ type: 'text', nullable: true })
  apiSecret?: string;

  // Auto-generated on creation for Bayut/dubizzle
  @Property({ nullable: true })
  feedUrl?: string;

  // PF webhook HMAC secret — used only for verifying X-Signature on PF webhook payloads
  @Property({ nullable: true })
  webhookSecret?: string;

  // Feed URL HMAC secret — used only for generating/validating the Bayut/dubizzle feed URL token
  // Separate from webhookSecret so each can be rotated independently
  @Property({ nullable: true })
  feedSecret?: string;

  // PF API key expiry date — visible in PF Expert when the key is generated (max 365 days)
  @Property({ type: 'timestamptz', nullable: true })
  apiKeyExpiresAt?: Date;

  // Tracks the timestamp of the last successful Bayut/dubizzle lead poll
  @Property({ type: 'timestamptz', nullable: true })
  lastLeadPollTimestamp?: Date;

  @Property({ type: 'timestamptz', default: () => new Date() })
  createdAt: Date = new Date();

  @Property({ type: 'timestamptz', onUpdate: () => new Date() })
  updatedAt: Date = new Date();

  @Property({ type: 'boolean', default: false })
  isDeleted: boolean = false;
}
```

```typescript
  furnishedStatus?: FurnishedStatus;

  @Enum(() => FinishingType)

  @Enum(() => RentalPeriod)
  @Property({ type: 'text', nullable: true })
  rentalPeriod?: RentalPeriod;

  // Marketing media
  @Property({ type: 'json', nullable: true })
  imageUrls?: string[];

  @Property({ type: 'json', nullable: true })
  videoUrls?: string[];

  @Property({ type: 'json', nullable: true })
  tourUrls?: string[];

  @Property({ type: 'timestamptz', default: () => new Date() })
```

## Listing lifecycle state machine

<Steps>
<Step title="Draft creation">
Agent creates a `Listing` from an `InventoryUnit`. Status starts as `DRAFT`.
</Step>

<Step title="Content preparation">
Agent fills in marketing content (title, description, images, portal-specific fields).
</Step>

<Step title="Activation">
Agent marks listing as `ACTIVE` when ready for syndication.
</Step>

<Step title="Portal enablement">
Agent enables specific portals for the listing. This creates `ListingPortalSync` records and triggers syndication.
</Step>

<Step title="Portal publishing">
Syndication service publishes to enabled portals based on portal type:
- **Property Finder**: Real-time REST API push
- **Bayut/dubizzle**: XML feed generation, portals pull within 30min-2hr
</Step>

<Step title="Lead ingestion">
Leads flow back from portals automatically:
- **Property Finder**: Webhook push + fallback polling
- **Bayut/dubizzle**: Scheduled API polling every 15 minutes
</Step>
</Steps>

## Security considerations

### Credential encryption

<Warning>
All portal credentials (`apiKey`, `apiSecret`, tokens) are encrypted at the application layer using AES-256-GCM before database storage.
</Warning>

### Feed URL security

<Steps>
<Step title="Token generation">
Bayut/dubizzle feed URLs include HMAC-signed tokens to prevent unauthorized access.
</Step>

<Step title="Token validation">
Each feed request validates the token signature using the organization's `feedSecret`.
</Step>

<Step title="Secret rotation">
Feed secrets can be rotated independently of webhook secrets for security flexibility.
</Step>
</Steps>

### Webhook validation

<Tabs>
<Tab title="Property Finder">
Webhook payloads are validated using HMAC-SHA256 signature verification against the `webhookSecret`.
</Tab>

<Tab title="Rate limiting">
All portal API calls implement exponential backoff and respect portal-specific rate limits.
</Tab>
</Tabs>

## Error handling and monitoring

### Retry logic

<AccordionGroup>
<Accordion title="Exponential backoff">
Failed syndication jobs retry with exponential backoff: 30s, 2min, 8min, 32min, 2hr (max 5 attempts).
</Accordion>

<Accordion title="Status tracking">
Each retry attempt updates `ListingPortalSync.retryCount` and `lastError` for debugging.
</Accordion>

<Accordion title="Circuit breaker">
Organizations with consecutive failures trigger temporary portal disabling to prevent spam.
</Accordion>
</AccordionGroup>

### Monitoring and alerts

### Error recovery

<Steps>
<Step title="Automatic retry">
Failed jobs automatically retry with exponential backoff up to maximum attempt limit.
</Step>

<Step title="Manual intervention">
Agents can manually retry failed syndications through the dashboard interface.
</Step>

<Step title="Bulk reconciliation">
Daily reconciliation service identifies and fixes sync inconsistencies across all portals.
</Step>

<Step title="Fallback polling">
Missed webhook deliveries are caught by fallback polling mechanisms running every 15 minutes.
</Step>
</Steps>

### Caching strategies

<Tabs>
<Tab title="XML feed cache">
**Technology**: Redis  
**TTL**: 30 minutes  
**Strategy**: Generate feeds on-demand, cache by organization + portal combination
</Tab>

<Tab title="PF token cache">
**Technology**: In-memory + Redis backup  
**TTL**: Based on JWT expiration minus 5-minute buffer  
**Strategy**: Proactive renewal before expiration
</Tab>

<Tab title="Location mapping cache">
**Technology**: In-memory LRU cache

**Strategy**: Cache-aside pattern with database fallback
</Tab>
