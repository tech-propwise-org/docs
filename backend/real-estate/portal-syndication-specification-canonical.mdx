## Module Overview

The Portal Syndication Module allows real estate agents to publish property listings to three UAE property portals directly from PropWise CRM, and automatically receive leads back into the CRM pipeline.

### Three-Tier Architecture

The module implements a clean separation of concerns through three distinct layers:

```
InventoryUnit  →  Listing  →  ListingPortalSync
  (inventory)     (marketing)   (per-portal state)
```

<AccordionGroup>
<Accordion title="InventoryUnit — Inventory Layer">
What the unit **is** - rooms, area, price, physical attributes. Remains unchanged by portal syndication logic.
</Accordion>

<Accordion title="Listing — Marketing Layer">
How the unit is **marketed** - title, descriptions, permit number, portal classifications, marketing media. Created by an agent from an `InventoryUnit`.
</Accordion>

<Accordion title="ListingPortalSync — Portal State Layer">
Where the listing is **published** and its current state on each portal.
</Accordion>
</AccordionGroup>

<Info>
This separation ensures `InventoryUnit` stays a clean inventory record and the `Listing` layer can eventually support off-plan units (`refUnitId`) without structural changes to the sync system.
</Info>

### Data Flow Rules

<Check>**Listings flow one direction only**: CRM → portals (CRM always wins)</Check>
<Check>**Leads flow one direction only**: portals → CRM</Check>
<Warning>Portal data **never** overwrites CRM data</Warning>
<Check>`Listing` is the **single source of truth** for listing content</Check>
<Check>`InventoryUnit` is the **single source of truth** for unit inventory data</Check>

#### Self-Contained Listing Entity

<Steps>
<Step title="Listing Independence">

#### XML Feed Generation

The Bayut/dubizzle XML feed is generated with:

- **`BayutDubizzleFeedAdapter`**: Transforms listings to XML format
- **CDATA XML Serializer**: Handles special characters and formatting
- **Property Reference Format**: `UNIT-{orgShortCode}-{listing.id}`

#### Public Feed Endpoint

<Tip>
The XML feed is publicly accessible at `GET /portal-syndication/feeds/:orgId?token=` for portal consumption.
</Tip>

The feed includes:
- Published listings as `Property_Status=live`
- Recently removed listings as `deleted` status for ≥48 hours (ensures prompt delisting)

## Authorization & Approval Workflow

### Publishing Authorization

The module implements a two-gate authorization system:

<Steps>
<Step title="Gate A - Initial Permission Check">
**Permission Required**: `real_estate.listing.publish`

- **With Permission**: Listing moves directly to publication
- **Without Permission**: Listing moves to `PENDING_APPROVAL` status
</Step>

<Step title="Approval Process">
Users with `real_estate.manage` permission can approve or reject pending listings:

- **Approve**: Honors the submitter's publish intent
- **Reject**: Moves listing to `REJECTED` status with rejection reason
</Step>

<Step title="Gate B - Portal-Specific Validation">
- **Property Finder**: Full compliance validation (permits, credits, etc.)
- **Bayut/dubizzle**: Permit validation skipped (post-crawl verification)
</Step>
</Steps>

### Approval Notifications

<Info>
All approval events trigger notifications via `EventEmitter2` and `RealEstateEventListener`:
</Info>

- **Submit for Approval** → `LISTING_APPROVAL_REQUESTED` to all approvers
- **Approve** → `LISTING_APPROVED` to publisher
- **Reject** → `LISTING_REJECTED` to publisher with reason
- **Delete** → `LISTING_DELETED` to publisher (if deleter ≠ publisher)

## Inventory Unit Integration

### Cascade Delete Behavior

When an inventory unit is deleted, users choose how to handle linked listings:

<Tabs>
<Tab title="Remove Linked Listings (Default)">
```typescript
removeLinkedListings = true
```
- Remove listings from all portals
- Archive listings with audit attribution
- Maintain data integrity
</Tab>

<Tab title="Preserve as Manual Listings">
```typescript
removeLinkedListings = false
```
- Sever unit link (`inventoryUnit = null`)
- Convert to self-contained manual listings
- Allow continued editing and publishing
</Tab>
</Tabs>

<Warning>
Inventory units are soft-deleted only and never archived. The `inventory-unit.deleted` event drives the cascade behavior to avoid circular dependencies.
</Warning>

## Error Handling & Recovery

### Sync State Management

The module maintains detailed sync state for each portal:

- **`DRAFT`**: Local listing not yet published
- **`PENDING`**: Awaiting portal API response
- **`PUBLISHED`**: Successfully syndicated
- **`FAILED`**: Syndication error occurred
- **`REMOVED`**: Delisted from portal

### Automatic Recovery

<Steps>
<Step title="Retry Logic">
Failed syndications are automatically retried with exponential backoff
</Step>

<Step title="State Reconciliation">
Periodic reconciliation ensures portal state matches CRM state
</Step>

<Step title="Credit Monitoring">
Property Finder credit levels are monitored to prevent publication failures
</Step>
</Steps>

<Tip>
The `SyncReconciliationService` runs on a scheduled basis to detect and resolve state discrepancies between the CRM and portal systems.
</Tip>
