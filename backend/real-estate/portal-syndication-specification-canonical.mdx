## Module Overview

The Portal Syndication Module allows real estate agents to publish property listings to three UAE property portals directly from PropWise CRM, and automatically receive leads back into the CRM pipeline.

### Three-Tier Architecture

```
InventoryUnit  â†’  Listing  â†’  ListingPortalSync
  (inventory)     (marketing)   (per-portal state)
```

- **`InventoryUnit`** â€” what the unit *is* (rooms, area, price, physical attributes). Unchanged by portal syndication logic.
- **`Listing`** â€” how the unit is *marketed* (title, descriptions, permit number, portal classifications, marketing media). Created by an agent from a `InventoryUnit`.
- **`ListingPortalSync`** â€” where the listing is *published* and its current state on each portal.

<Info>
This separation ensures `InventoryUnit` stays a clean inventory record and the `Listing` layer can eventually support off-plan units (`refUnitId`) without any structural change to the sync system.
</Info>

### Integration Model Per Portal

| Portal | Listing Syndication | Lead Ingestion | Listing Timing |
|---|---|---|---|
| Property Finder | REST API Push (JSON) | Webhook push (primary) + REST poll fallback (15 min) | Real-time (seconds) |
| Bayut | XML Feed Pull (unified) | Pull API polling â€” scheduled every 15 min | 30 min â€“ 2 hr delay |
| dubizzle | XML Feed Pull (same as Bayut) | Pull API polling â€” same API + endpoint as Bayut | 30 min â€“ 2 hr delay |

<Warning>
**Bayut / dubizzle lead ingestion note:** Bayut and dubizzle share one API endpoint and one Bearer token (per agency). The `source` field in each lead response (`"bayut"` or `"dubizzle"`) determines which `LeadSource` enum value is used when the CRM lead is created. Lead ingestion is gated by `PortalConfiguration.leadIngestionEnabled` (per portal).
</Warning>

### Data Flow Rules

- Listings flow **one direction only**: CRM â†’ portals (CRM always wins)
- Leads flow **one direction only**: portals â†’ CRM
- Portal data **never** overwrites CRM data
- `Listing` is the **single source of truth** for listing (marketing) content
- `InventoryUnit` is the **single source of truth** for unit inventory data

### Publish Authorization System

<AccordionGroup>
<Accordion title="Gate A - Permission Check">
`SyndicationService.publish` checks `real_estate.listing.publish` permission:
- **With permission**: Direct publish
- **Without permission**: Listing goes to `ListingStatus.PENDING_APPROVAL`
- Managers use `POST /:listingId/approve|reject` for approval workflow
</Accordion>

<Accordion title="Rejection Workflow">
**Reject** moves listing to `ListingStatus.REJECTED` and persists:
- Approver's rejection reason
- `rejectedAt` and `rejectedBy` timestamps
- Submitter can edit and resubmit (clears rejection status)
</Accordion>

<Accordion title="Approval Workflow">
**Approve** honors submitter's publish intent:
- `publishOnApproval=true`: Auto-publishes to target portals (â†’ ACTIVE)
- `publishOnApproval=false`: Vetted into plain DRAFT
- Sets `Listing.approvedAt` / `approvedBy` (permanent stamp)
</Accordion>

<Accordion title="Owner Self-Manage Bypass">
Post-approval, listing **owner** bypasses Gate A:
- Owner = publisher (`createdBy`) / `agent` / linked-unit `unitManager`
- Once `approvedAt` is set, owner can publish/unpublish/toggle portals directly
- New listings still require approval for first publish
</Accordion>
</AccordionGroup>

### Inventory Cascade Handling

<Tabs>
<Tab title="Remove Listings (Default)">
When `removeLinkedListings = true`:
- Remove unit's listings from all portals via `SyndicationService.removeFromAllPortals`
- Archive listings via `ListingService.archiveByUnit`
- Audit trail preserved with deleting actor attribution
</Tab>

<Tab title="Preserve Listings">
When `removeLinkedListings = false`:
- Keep listings live but sever unit link (`inventoryUnit = null`)
- Convert to self-contained manual listings
- User can continue editing/publishing independently
</Tab>
