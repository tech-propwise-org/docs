# Bayut/Dubizzle Pull API Specification

<Info>
**Purpose:** Authoritative reference for the Bayut **Pull API** used to fetch lead and view data from Bayut.

**Scope:** This document describes the **external API contract** defined by Bayut — endpoint, authorization, query parameters, and sample responses for both *leads* (`is_trulead=1`) and *views* (`is_trulead=0`), plus call logs and story leads.
</Info>

## Overview

This document provides details on the API endpoint used to pull lead data from Bayut. It includes the endpoint URL, authorization requirements, query parameters, and sample responses. The API enables developers to effectively integrate Bayut lead data into their own applications or systems by providing:

- **Endpoint URL:** The specific web address to which API requests should be directed
- **Authorization Requirements:** The necessary credentials or permissions needed to access the API and retrieve data
- **Query Parameters:** Parameters that can be included in the API request to filter or refine the data returned
- **Sample Responses:** Examples of the data structure and format returned by the API

## Endpoint

The base endpoint for all API requests:

```
www.bayut.com/api-v7/stats/website-client-leads
```

This API allows clients to fetch lead data based on specific query parameters.

## Authorization

<Warning>
Each client is provided with a unique API key, which must be passed as a **bearer token** in the `Authorization` header.
</Warning>

```bash
Authorization: Bearer <API KEY>
```

## CURL Request Example

```bash
curl --location \
  'www.bayut.com/api-v7/stats/website-client-leads?timestamp=2025-03-10%2000%3A00%3A00&target=listing&type=email&is_trulead=1' \
  --header 'Authorization: Bearer <API KEY>'
```

<Note>
The `timestamp` value is URL-encoded: `2025-03-10 00:00:00` becomes `2025-03-10%2000%3A00%3A00` (`%20` = space, `%3A` = colon).
</Note>

## Query Parameters

| Parameter | Description | Possible Values |
|-----------|-------------|-----------------|
| `type` | The type of lead | `whatsapp`, `sms`, `phone`, `email` |
| `timestamp` | The date and time of the request | Format: `YYYY-MM-DD HH:MM:SS` |
| `target` | The entity for which leads are fetched | `listing`, `agent`, `agency` |
| `is_trulead` | Whether it is a lead or just a views count | `0`, `1` |

<Tip>
- `is_trulead=1` returns **actual leads** with detailed inquirer information
- `is_trulead=0` returns **view counts** for tracking purposes
</Tip>

## Get Views

<Info>
This API endpoint retrieves the number of views generated from different lead types (`is_trulead=0`). Views represent user interactions that don't qualify as full leads but are still valuable for tracking engagement.
</Info>

The views endpoint uses the same base URL and authentication but with `is_trulead=0` to return view count data instead of detailed lead information. This is useful for analytics and understanding user engagement patterns across different channels (WhatsApp, SMS, phone, email) and targets (listings, agents, agencies).
