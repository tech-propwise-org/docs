## Overview

The Bayut Pull API enables clients to fetch lead and view data from Bayut's platform. This document describes the external API contract defined by Bayut, including endpoint details, authorization requirements, query parameters, and sample responses for both leads (`is_trulead=1`) and views (`is_trulead=0`).

<Note>
This API allows you to retrieve comprehensive lead data including email leads, WhatsApp interactions, phone calls, SMS, and story leads across listings, agents, and agencies.
</Note>

## Endpoint

The base endpoint for the Pull API is:

```
www.bayut.com/api-v7/stats/website-client-leads
```

All requests should be directed to this URL with appropriate query parameters.

## Authorization

Each client receives a unique API key that must be passed as a bearer token in the `Authorization` header.

<CodeGroup>

```bash cURL
curl --location \
  'www.bayut.com/api-v7/stats/website-client-leads?timestamp=2025-03-10%2000%3A00%3A00&target=listing&type=email&is_trulead=1' \
  --header 'Authorization: Bearer <API KEY>'
```

```javascript JavaScript
const response = await fetch(
  'www.bayut.com/api-v7/stats/website-client-leads?timestamp=2025-03-10%2000%3A00%3A00&target=listing&type=email&is_trulead=1',
  {
    headers: {
      'Authorization': 'Bearer <API KEY>'
    }
  }
);
```

```python Python
import requests

response = requests.get(
    'www.bayut.com/api-v7/stats/website-client-leads',
    params={
        'timestamp': '2025-03-10 00:00:00',
        'target': 'listing',
        'type': 'email',
        'is_trulead': '1'
    },
    headers={
        'Authorization': 'Bearer <API KEY>'
    }
)
```

</CodeGroup>

<Warning>
The `timestamp` value must be URL-encoded. For example, `2025-03-10 00:00:00` becomes `2025-03-10%2000%3A00%3A00` (where `%20` = space and `%3A` = colon).
</Warning>

## Query Parameters

The following query parameters are used to filter and refine the data returned by the API:

| Parameter | Description | Possible Values |
|-----------|-------------|-----------------|
| `type` | The type of lead | `whatsapp`, `sms`, `phone`, `email`, `call_logs`, `story_leads` |
| `timestamp` | The date and time of the request | Format: `YYYY-MM-DD HH:MM:SS` |
| `target` | The entity for which leads are fetched | `listing`, `agent`, `agency` |
| `is_trulead` | Whether it is a lead or just a views count | `0` (views), `1` (leads) |

<Info>
Setting `is_trulead=1` returns **actual leads** with detailed contact information. Setting `is_trulead=0` returns **view counts** for tracking engagement metrics.
</Info>

<Note>
The field `is_object_fecthed` is spelled as-is in the Bayut response (a typo for "fetched" preserved from the source contract). Integrators should map it verbatim.
</Note>

<AccordionGroup>
<Accordion title="Call log field descriptions">
- **call_log_id**: Unique identifier for the call log entry
- **call_status**: Status of the call (e.g., `connected`, `missed`, `busy`)
- **call_total_duration**: Total duration of the call including ring time
- **call_connected_duration**: Duration of the connected portion of the call
- **call_recordingurl**: URL to access the call recording (if available)
- **caller_location**: Geographic location of the caller
- **call_type**: Type of call (`in` for incoming, `out` for outgoing)
</Accordion>

<Accordion title="Understanding receiver numbers">
The `receiver_numbers_list` field may contain multiple numbers with priority indicators (e.g., `+971123456789#35`), where the number after `#` represents the priority or order of contact attempts.
</Accordion>
</AccordionGroup>

## Best practices

<Steps>
  <Step title="Secure your API key">
    Store your API key securely and never expose it in client-side code. Use environment variables or secure credential management systems.
  </Step>
  
  <Step title="Implement rate limiting">
    Respect API rate limits to ensure consistent access. Implement exponential backoff for retries if you encounter rate limiting responses.
  </Step>
  
  <Step title="URL encode timestamps">
    Always URL-encode timestamp parameters to avoid request errors. Remember that spaces become `%20` and colons become `%3A`.
  </Step>
  
  <Step title="Handle delivery notifications">
    For WhatsApp leads, parse the `delivery_notifications` array to track message engagement. Sort by `created_at` to get chronological status updates.
  </Step>
  
  <Step title="Parse call recordings carefully">
    Call recording URLs may require additional authentication. Test recording access in a development environment before implementing in production.
  </Step>
</Steps>

## Error handling

<Warning>
The API may return various HTTP status codes. Implement proper error handling for the following scenarios:

- **401 Unauthorized**: Invalid or expired API key
- **400 Bad Request**: Invalid query parameters or malformed timestamp
- **429 Too Many Requests**: Rate limit exceeded
- **500 Internal Server Error**: Temporary server issues (retry with exponential backoff)
</Warning>

```javascript Example error handling
async function fetchBayutLeads(timestamp, target, type) {
  try {
    const response = await fetch(
      `www.bayut.com/api-v7/stats/website-client-leads?timestamp=${encodeURIComponent(timestamp)}&target=${target}&type=${type}&is_trulead=1`,
      {
        headers: {
          'Authorization': `Bearer ${process.env.BAYUT_API_KEY}`
        }
      }
    );

    if (!response.ok) {
      if (response.status === 401) {
        throw new Error('Invalid API key');
      } else if (response.status === 429) {
        throw new Error('Rate limit exceeded. Please retry later.');
      } else {
        throw new Error(`API request failed with status ${response.status}`);
      }
    }

    return await response.json();
  } catch (error) {
    console.error('Failed to fetch Bayut leads:', error);
    throw error;
  }
}
```
