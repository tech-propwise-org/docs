**Path:** `src/modules/real-estate/property-interest/`

The property-interest module is one of **six specialized modules** within the real estate architecture, positioned as an **org-scoped** module that manages lead and deal property interests and the matching engine. It demonstrates the **cross-database reference pattern** by linking to reference DB projects/units via **plain integer IDs** and to CRM DB entities via ManyToOne relations.

## Module context

Property-interest operates within a **three-tier visibility architecture**:

- **Global scope**: reference-data (second DB), developer-portal, map utility
- **Org-private scope**: **property-interest**, inventory (secondary units)  
- **Cross-org scope**: shared-marketplace (listings + deals)

<Info>
Property-interest consumes **all property data exclusively through ReferenceDataService**. This architectural pattern decouples the CRM from the property data source — if the reference database is swapped for an external property API, only ReferenceDataService needs to change.
</Info>

## Architecture overview

The real estate module is structured as **six specialized modules** spanning three visibility scopes: **global** (no RLS), **org-private** (standard RLS), and **cross-org** (custom RLS).

An external **reference database** (Neon "propwise-lab") stores **all property data** — both structural (DLD registry) and enrichment (gallery, payment plans, documents, amenities, pricing, geocoding). The CRM database stores only CRM-specific data (developer portal auth, ownership links, property interests, marketplace listings, secondary unit context).

## Cross-database reference pattern

CRM entities in property-interest store **plain integer IDs** (`refProjectId`, `refUnitId`) pointing to the reference database, with **no cross-database foreign keys**:

```typescript
// LeadPropertyInterest entity
@Property({ type: 'int', nullable: true })
@Index()
refProjectId?: number;  // ID in reference DB projects table

@Property({ type: 'int', nullable: true })
@Index()  
refUnitId?: number;     // ID in reference DB units table
```

This decoupling enables the CRM to reference property data without being tightly coupled to the property data source implementation. The architectural reasoning is:

- **Flexibility**: If the reference database is swapped for an external property API, only `ReferenceDataService` needs to change
- **Performance**: No cross-database joins or foreign key constraints
- **Independence**: CRM can operate even if property data source is temporarily unavailable
- **Scalability**: Each database can be optimized for its specific use case

## Matching engine

The `PropertyMatchService` generates matches when a LeadPropertyInterest is created or updated. It demonstrates the architectural separation by using `ReferenceDataService` for off-plan data (no direct entity access):

<Steps>
<Step title="Query sources in parallel">

- Internal inventory (SecondaryUnit) — org-scoped via TenantContext
- Off-plan directory (RefProject/RefUnit) — via `ReferenceDataService`
- Shared marketplace (SharedListing) — global via bypass

</Step>
<Step title="Score each candidate (0-100)">

| Criterion | Points | Scoring |
| --- | --- | --- |
| Budget match | 0-30 | exact=30, within 20%=20, within 50%=10 |
| Bedroom match | 0-25 | exact=25, +/-1=15, +/-2=5 |
| Area match | 0-20 | preferred area=20, preferred community=10 |
| Size match | 0-15 | within range=15, within 20%=10 |
| Type match | 0-10 | exact=10 |

</Step>
<Step title="Persist matches">
Save matches with score greater than 20, sorted by score descending.
</Step>
</Steps>

<Tip>
The `PropertyMatchService` avoids nested TenantContext calls by running off-plan queries (via `ReferenceDataService`) and shared listing queries (global, `executeWithBypass`) **first**, then entering `executeInOrg` for org-scoped secondary unit queries + match persistence.
</Tip>

## Service layer details

### Key service methods

```typescript

  // Generate matches for a lead property interest
  async generateMatches(leadInterestId: string): Promise<PropertyMatch[]>
```

### Scoring algorithm

The weighted scoring system ensures the most relevant matches appear first:

<Note>
Matches with scores below 20 are automatically filtered out to maintain result quality. This threshold prevents low-relevance matches from cluttering the interface while still allowing for some flexibility in matching criteria.
</Note>

### Data source integration

The matching engine queries three distinct data sources:

1. **Reference database** (via `ReferenceDataService`): Off-plan projects and units
2. **CRM database** (direct): Organization's secondary unit inventory  
3. **Shared marketplace** (bypass RLS): Cross-org listings

This multi-source approach provides comprehensive property matching while respecting data ownership boundaries.

## Match status workflow

Property matches follow a defined status progression:

<Steps>
<Step title="NEW">
Match created automatically by the matching engine
</Step>
<Step title="VIEWED">
User has viewed the match details
</Step>
<Step title="SHORTLISTED">
User marked as potential opportunity
</Step>
<Step title="CONTACTED">
User initiated contact with property owner/agent
</Step>
<Step title="DISMISSED">
User rejected the match (terminal state)
</Step>
</Steps>

<Warning>
Once a match is marked as DISMISSED, it cannot be changed to other states. This prevents accidentally re-engaging with explicitly rejected properties.
</Warning>

## Integration patterns

The property-interest module demonstrates several key architectural patterns:

### Cross-database references
- Uses plain integers for reference DB IDs (no FK constraints)
- Maintains referential integrity through application logic
- Enables flexible property data source swapping

### Multi-tenant data access
- Respects org boundaries for internal inventory and matches
- Bypasses RLS for shared marketplace queries
- Uses TenantContext appropriately to avoid nested calls

### Event-driven matching
- Automatically generates matches on interest creation/update
- Supports manual match regeneration for data refresh scenarios
- Maintains match history for analytics
