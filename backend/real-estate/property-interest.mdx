**Path:** `src/modules/real-estate/property-interest/`

The property-interest module is one of **six specialized modules** within the real estate architecture, positioned as an **org-scoped** module that manages lead and deal property interests and the matching engine. It demonstrates the **external API reference pattern** by linking to Propwise Labs API properties via **plain integer IDs** and to CRM DB entities via ManyToOne relations.

## Real estate module architecture

The real estate module is structured as **six active specialized modules** spanning three visibility scopes: **global** (no RLS), **org-private** (standard RLS), and **cross-org** (custom RLS). The cross-org `shared-marketplace` code is retained in the repository but is not imported by `RealEstateModule`, so its controllers/providers are not mounted by the app.

The external **Propwise Labs API** stores and serves all property data — both structural (DLD registry) and enrichment (images, payment plans, documents, amenities, pricing, geocoding where exposed). The CRM database stores only CRM-specific data (developer portal auth, ownership links, property interests, marketplace listings, and org inventory context).

<Note>
All modules consume property data exclusively through `PropwiseLabsService`. The CRM no longer opens a second MikroORM connection or maps Propwise Labs entities locally. The reference database configuration has been completely removed and replaced with Propwise Labs API configuration (`app.propwiseLabs.*`) including API credentials, timeout settings, and Bird Verify integration for developer portal contact verification.
</Note>

### Listing media ingest

**Listing media ingest (WebP) + `/storage/upload` `profile` contract:** listing photos and floor plans are uploaded through the shared `POST /storage/upload` endpoint (`StorageApi.uploadFile(file, profile?)`), which takes an optional **`profile`** body field validated against `STORAGE_UPLOAD_PROFILES` (unknown → `400`). The default is **`GENERIC_ATTACHMENT`** (long edge ≤ 1920, WebP q80) so email/workflow/deal-document/tiptap uploads stay lean. Listing image call sites pass **`LISTING`** (≤ 2560, q82) and floor-plan call sites pass **`FLOOR_PLAN`** (≤ 2560, q90) so syndication imagery reaches Property Finder at high resolution and floor plans stay legible. The endpoint optimizes the buffer through `ImageOptimizationService` (raster → WebP; PDF/doc/video → passthrough), writes the object under a profile-specific R2 folder (`STORAGE_UPLOAD_FOLDERS`: `email-attachments` / `listings` / `listings/floor-plans`), and reflects the converted `fileName` (`.webp`) + `mimeType` + `fileSize` in the JSON response. `ListingImageService` (the Property Finder pre-flight) is unchanged: its `compliant` check re-processes a source image only when it exceeds PF's 1920×1080 / format limits — an expected single PF-specific re-encode, not an ingest regression.

### Storage upload profiles

The `/storage/upload` endpoint supports three distinct profiles optimized for different use cases:

- **`GENERIC_ATTACHMENT`** (default): Long edge ≤ 1920px, WebP q80 — for email, workflow, deal documents, and Tiptap uploads
- **`LISTING`**: Long edge ≤ 2560px, WebP q82 — for portal syndication listing photos at high resolution
- **`FLOOR_PLAN`**: Long edge ≤ 2560px, WebP q90 — for floor plans requiring maximum legibility

Images are stored in profile-specific R2 folders: `email-attachments`, `listings`, and `listings/floor-plans` respectively.

## External property ID pattern

`InventoryUnit` is the CRM boundary for org-owned secondary inventory. It stores plain integer IDs (`refProjectId`, `refBuildingId`, `refUnitId`, `refDeveloperId`, `refPropertyTypeId`, `refPropertySubTypeId`) plus a denormalized Labs snapshot, including the developer display name used by portal syndication and the canonical `propertyType` (`LayoutType` enum) used by portal syndication adapters. Labs raw property type/sub-type names are retained in `labsPropertyType` and `labsPropertySubType` for traceability. Lead interests can use inventory units as candidate matches while retaining search criteria. Deal property rows are concrete-only: they link either an org `InventoryUnit`, a direct Propwise Labs off-plan unit via `refProjectId` / `refBuildingId` / `refUnitId`, or a pending unit-access request before the inventory link is approved.

```typescript
// CRM entity referencing Propwise Labs data:
@Property({ type: 'int', nullable: true })
@Index()
refProjectId?: number;  // ID in Propwise Labs projects

@Property({ type: 'int', nullable: true })
@Index()
refUnitId?: number;     // ID in Propwise Labs units
```

This decoupling enables the CRM to reference property data without being tightly coupled to the property data source implementation. The architectural reasoning is:

- **Flexibility**: If Propwise Labs API is swapped for another property data source, only `PropwiseLabsService` needs to change
- **Performance**: No cross-database joins or foreign key constraints
- **Independence**: CRM can operate even if property data source is temporarily unavailable
- **Scalability**: Each system can be optimized for its specific use case

## Matching engine

The `PropertyMatchService` generates matches when a LeadPropertyInterest is created or updated. It demonstrates the architectural separation by using `PropwiseLabsService` for off-plan data (no direct entity access):

<Steps>
<Step title="Query sources in parallel">
- Internal inventory (SecondaryUnit) — org-scoped via TenantContext
- Off-plan directory (Propwise Labs projects/units) — via `PropwiseLabsService`
- Shared marketplace (SharedListing) — global via bypass
</Step>

<Step title="Score each candidate (0-100)">

| Criterion | Points | Scoring |
| --- | --- | --- |
| Budget match | 0-30 | exact=30, within 20%=20, within 50%=10 |
| Bedroom match | 0-25 | exact=25, +/-1=15, +/-2=5 |
| Area match | 0-20 | preferred area=20, preferred community=10 |
| Size match | 0-15 | within range=15, within 20%=10 |
| Type match | 0-10 | exact=10 |

</Step>

<Step title="Persist matches">
Save matches with score greater than 20, sorted by score descending.
</Step>
</Steps>

<Tip>
The `PropertyMatchService` avoids nested TenantContext calls by running off-plan queries (via `PropwiseLabsService`) and shared listing queries (global, `executeWithBypass`) **first**, then entering `executeInOrg` for org-scoped secondary unit queries + match persistence.
</Tip>

export class PropertyInterestService {
  constructor(
    private readonly em: EntityManager,
    private readonly tenantContext: TenantContext,
    private readonly propertyMatchService: PropertyMatchService,
  ) {}

  // Lead property interests
  async createLeadInterest(
    leadId: string,
    data: CreateLeadPropertyInterestDto,
  ): Promise<LeadPropertyInterest> {
    const interest = this.em.create(LeadPropertyInterest, {
      ...data,
      lead: leadId,
      orgId: this.tenantContext.currentOrgId,
    });
    
    await this.em.persistAndFlush(interest);
    
    // Auto-generate matches
    await this.propertyMatchService.generateMatches(interest.id);
    
    return interest;
  }

  async getLeadInterests(leadId: string): Promise<LeadPropertyInterest[]> {
    return this.em.find(LeadPropertyInterest, { lead: leadId });
  }

  // Deal property interests
  async createOrUpdateDealInterest(
    dealId: string,
    data: CreateDealPropertyInterestDto,
  ): Promise<DealPropertyInterest> {
    const existing = await this.em.findOne(DealPropertyInterest, { deal: dealId });
    
    if (existing) {
      this.em.assign(existing, data);
      await this.em.flush();
      return existing;
    }
    
    const interest = this.em.create(DealPropertyInterest, {
      ...data,
      deal: dealId,
      orgId: this.tenantContext.currentOrgId,
    });
    
    await this.em.persistAndFlush(interest);
    return interest;
  }
}

<Accordion title="PropertyMatchService">

Core matching engine that orchestrates property matching across data sources:

```typescript
@Injectable()
export class PropertyMatchService {
  constructor(
    private readonly em: EntityManager,
    private readonly tenantContext: TenantContext,
    private readonly propwiseLabsService: PropwiseLabsService,
  ) {}

  async generateMatches(leadInterestId: string): Promise<PropertyMatch[]> {
    const interest = await this.em.findOneOrFail(LeadPropertyInterest, leadInterestId);
    
    // Query all sources in parallel (outside org context)
    const [offPlanProperties, sharedListings] = await Promise.all([
      this.getOffPlanMatches(interest),
      this.getSharedListingMatches(interest),
    ]);
    
    // Enter org context for internal inventory and match persistence
    return this.tenantContext.executeInOrg(interest.orgId, async () => {
      const secondaryUnits = await this.getSecondaryUnitMatches(interest);
      
      const allCandidates = [
        ...offPlanProperties.map(p => ({ type: 'off-plan', data: p })),
        ...secondaryUnits.map(u => ({ type: 'secondary', data: u })),
        ...sharedListings.map(l => ({ type: 'shared', data: l })),
      ];
      
      const matches: PropertyMatch[] = [];
      
      for (const candidate of allCandidates) {
        const score = this.calculateMatchScore(interest, candidate);
        if (score > 20) {
          const match = this.em.create(PropertyMatch, {
            leadInterest: interest,
            matchScore: score,
            matchBreakdown: this.getMatchBreakdown(interest, candidate),
            status: MatchStatus.NEW,
            orgId: interest.orgId,
            ...this.getMatchedEntityIds(candidate),
          });
          matches.push(match);
        }
      }
      
      // Sort by score descending
      matches.sort((a, b) => b.matchScore - a.matchScore);
      
      await this.em.persistAndFlush(matches);
      return matches;
    });
  }

  private async getOffPlanMatches(interest: LeadPropertyInterest): Promise<any[]> {
    const filters = {
      propertyType: interest.propertyType,
      bedrooms: interest.bedrooms,
      priceMin: interest.budgetMin,
      priceMax: interest.budgetMax,
      areas: interest.preferredAreas,
      communities: interest.preferredCommunities,
    };
    
    return this.propwiseLabsService.searchProjects(filters);
  }

  private calculateMatchScore(interest: LeadPropertyInterest, candidate: any): number {
    let score = 0;
    
    // Budget match (0-30 points)
    if (this.isBudgetMatch(interest, candidate)) {
      score += this.getBudgetScore(interest, candidate);
    }
    
    // Bedroom match (0-25 points)
    if (this.isBedroomMatch(interest, candidate)) {
      score += this.getBedroomScore(interest, candidate);
    }
    
    // Area/location match (0-20 points)
    score += this.getLocationScore(interest, candidate);
    
    // Size match (0-15 points)
    if (this.isSizeMatch(interest, candidate)) {
      score += this.getSizeScore(interest, candidate);
    }
    
    // Property type match (0-10 points)
    if (interest.propertyType === candidate.data.propertyType) {
      score += 10;
    }
    
    return Math.min(score, 100);
  }
}
```

    private readonly em: EntityManager,
    private readonly tenantContext: TenantContext,
  ) {}

  async getMatchPerformance(
    orgId: string,
    filters: AnalyticsFilters,
  ): Promise<MatchPerformanceDto> {
    return this.tenantContext.executeInOrg(orgId, async () => {
      const qb = this.em.createQueryBuilder(PropertyMatch, 'm')
        .select([
          'COUNT(*) as total_matches',
          'AVG(m.match_score) as avg_score',
          'COUNT(CASE WHEN m.status = ? THEN 1 END) as contacted',
          'COUNT(CASE WHEN m.status = ? THEN 1 END) as shortlisted',
        ])
        .where('m.created_at >= ? AND m.created_at <= ?')
        .setParameters([
          MatchStatus.CONTACTED,
          MatchStatus.SHORTLISTED,
          filters.from,
          filters.to,
        ]);

      if (filters.leadId) {
        qb.andWhere('m.lead_interest IN (SELECT id FROM lead_property_interests WHERE lead_id = ?)')
          .setParameter(2, filters.leadId);
      }

      const result = await qb.getSingleResult();
      
      return {
        totalMatches: parseInt(result.total_matches),
        averageScore: parseFloat(result.avg_score) || 0,
        contactedCount: parseInt(result.contacted),
        shortlistedCount: parseInt(result.shortlisted),
        conversionRate: result.total_matches > 0 
          ? (result.contacted / result.total_matches) * 100 
          : 0,
      };
    });
  }
}

## Match status workflow

Property matches follow a defined status progression:

<Steps>
<Step title="NEW">
Match created automatically by the matching engine
</Step>
<Step title="VIEWED">
User has viewed the match details
</Step>
<Step title="SHORTLISTED">
User marked as potential opportunity
</Step>
<Step title="CONTACTED">
User initiated contact with property owner/agent
</Step>
<Step title="DISMISSED">
User rejected the match (terminal state)
</Step>
</Steps>

<Warning>
Once a match is marked as DISMISSED, it cannot be changed to other states. This prevents accidentally re-engaging with explicitly rejected properties.
</Warning>

## Scoring algorithm details

The weighted scoring system ensures the most relevant matches appear first:

<Note>
Matches with scores below 20 are automatically filtered out to maintain result quality. This threshold prevents low-relevance matches from cluttering the interface while still allowing for some flexibility in matching criteria.
</Note>

### Detailed scoring breakdown

<AccordionGroup>
<Accordion title="Budget scoring (0-30 points)">

- **Exact match**: 30 points - property price equals client budget
- **Within 20%**: 20 points - property price within ±20% of client budget  
- **Within 50%**: 10 points - property price within ±50% of client budget
- **Outside range**: 0 points

</Accordion>
<Accordion title="Bedroom scoring (0-25 points)">

- **Exact match**: 25 points - exact bedroom count match
- **±1 bedroom**: 15 points - one bedroom difference
- **±2 bedrooms**: 5 points - two bedroom difference
- **>2 difference**: 0 points

</Accordion>
<Accordion title="Location scoring (0-20 points)">

- **Preferred area**: 20 points - property in client's preferred area list
- **Preferred community**: 10 points - property in client's preferred community list
- **Same city/region**: 5 points - property in same general location
- **Different location**: 0 points

</Accordion>
<Accordion title="Size scoring (0-15 points)">

- **Within range**: 15 points - property size within client's min/max range
- **Within 20% of range**: 10 points - close to desired size range
- **Within 50% of range**: 5 points - moderately close to desired size
- **Outside range**: 0 points

</Accordion>
<Accordion title="Property type scoring (0-10 points)">

- **Exact match**: 10 points - exact property type match (Villa, Apartment, etc.)
- **Compatible type**: 5 points - similar property types (Studio → 1BR Apartment)
- **Different type**: 0 points

</Accordion>
</AccordionGroup>

## Integration patterns

The property-interest module demonstrates several key architectural patterns:

### External API references
- Uses plain integers for Propwise Labs API IDs (no FK constraints)
- Maintains referential integrity through application logic
- Enables flexible property data source swapping

### Multi-tenant data access
- Respects org boundaries for internal inventory and matches
- Bypasses RLS for shared marketplace queries
- Uses TenantContext appropriately to avoid nested calls

### Event-driven matching
- Automatically generates matches on interest creation/update
- Supports manual match regeneration for data refresh scenarios
- Maintains match history for analytics

### Data source integration

The matching engine queries three distinct data sources:

1. **Propwise Labs API** (via `PropwiseLabsService`): Off-plan projects and units
2. **CRM database** (direct): Organization's secondary unit inventory  
3. **Shared marketplace** (bypass RLS): Cross-org listings

This multi-source approach provides comprehensive property matching while respecting data ownership boundaries.
