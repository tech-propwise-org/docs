## Module Overview

The Portal Syndication Module allows real estate agents to publish property listings to three UAE property portals directly from PropWise CRM, and automatically receive leads back into the CRM pipeline.

### Three-Tier Architecture

```
InventoryUnit  →  Listing  →  ListingPortalSync
  (inventory)     (marketing)   (per-portal state)
```

- **`InventoryUnit`** — what the unit *is* (rooms, area, price, physical attributes). Unchanged by portal syndication logic.
- **`Listing`** — how the unit is *marketed* (title, descriptions, permit number, portal classifications, marketing media). Created by an agent from a `InventoryUnit`.
- **`ListingPortalSync`** — where the listing is *published* and its current state on each portal.

<Info>
This separation ensures `InventoryUnit` stays a clean inventory record and the `Listing` layer can eventually support off-plan units (`refUnitId`) without any structural change to the sync system.
</Info>

### Integration Model Per Portal

| Portal | Listing Syndication | Lead Ingestion | Listing Timing |
|---|---|---|---|
| Property Finder | REST API Push (JSON) | Webhook push (primary) + REST poll fallback (15 min) | Real-time (seconds) |
| Bayut | XML Feed Pull (unified) | Pull API polling — scheduled every 15 min | 30 min – 2 hr delay |
| dubizzle | XML Feed Pull (same as Bayut) | Pull API polling — same API + endpoint as Bayut | 30 min – 2 hr delay |

<Warning>
**Bayut / dubizzle lead ingestion note:** Bayut and dubizzle share one API endpoint and one Bearer token (per agency). The `source` field in each lead response (`"bayut"` or `"dubizzle"`) determines which `LeadSource` enum value is used when the CRM lead is created. The Bearer token is stored encrypted in the existing `apiKey` field of the Bayut `PortalConfiguration` row — no new credential columns are needed.
</Warning>

### Data Flow Rules

- Listings flow **one direction only**: CRM → portals (CRM always wins)
- Leads flow **one direction only**: portals → CRM
- Portal data **never** overwrites CRM data
- `Listing` is the **single source of truth** for listing (marketing) content
- `InventoryUnit` is the **single source of truth** for unit inventory data

### Listing Sync Status

Nine states covering the full PF async lifecycle and feed-based portal model:

```typescript
export enum ListingSyncStatus {
  DRAFT = 'draft',                           // Not yet publishable (validation failed or not requested)
  PENDING = 'pending',                       // Job queued, adapter call in progress
  AWAITING_CONFIRMATION = 'awaiting_confirmation', // PF: create+publish called, waiting for webhook
  PUBLISHED = 'published',                   // Live on portal
  REJECTED = 'rejected',                     // PF rejected listing (compliance/quality)
  COMPLIANCE_ACTION = 'compliance_action',   // PF flagged listing for action
  ERROR = 'error',                           // Sync failed after retries
  INSUFFICIENT_CREDITS = 'insufficient_credits', // PF: balance < publish cost; agent retries after top-up
  REMOVED = 'removed',                       // Taken down (disabled, archived, deleted)
}
```

<Note>
**For Bayut/dubizzle:** Status goes directly `draft → published` on enable (no `awaiting_confirmation`), and `published → removed` on disable. Feed portals have no rejection mechanism.
</Note>

## Entity Definitions

<Info>
**Architecture Note — Three-Tier Data Model**

```
InventoryUnit  ──1:many──▶  Listing  ──1:many──▶  ListingPortalSync
  (inventory)                (marketing)             (per-portal state)
```

- Fields describing **what the unit is** (rooms, area, price, bathrooms, location coords) live on `InventoryUnit`.
- Fields describing **how the unit is advertised** (title, descriptions, permit number, portal classifications, marketing media) live on `Listing`.
- Per-portal state (publish status, portal listing ID, retry count) lives on `ListingPortalSync`.
</Info>

<Warning>
**Note on portal lead ingestion:** There is no `PortalLead` staging entity. Inbound leads from portal **webhooks (PF) and scheduled Pull API polling (Bayut/dubizzle)** are queued directly via pg-boss (`portal-lead-ingestion` queue) and converted immediately to CRM `Lead` entities by the worker. Deduplication is handled by the existing `(organization, leadSource, referenceId)` unique index on `Lead`.
</Warning>

**Why a separate `Listing` entity?**

<AccordionGroup>
  <Accordion title="Separation Benefits">
    1. Keeps `InventoryUnit` as a clean inventory record — agents editing unit details are not exposed to portal marketing fields.
    2. Future-proofs off-plan support: `Listing` holds an optional `refUnitId` (reference DB integer ID), so off-plan units can be published without changing the sync system.
    3. Provides a natural home for any future admin approval workflow before syndication fans out to portals.
  </Accordion>
</AccordionGroup>

### Portal Configuration

Per-organization portal credentials and settings. One row per (org, portal).

```typescript
@Entity()
@Filter({ name: 'isDeleted', default: true, cond: { isDeleted: false } })
@Unique({ properties: ['organization', 'portal'] })
export class PortalConfiguration {
  @PrimaryKey({ type: UuidType })
  id: string = v4();

  @ManyToOne(() => Organization)
  @Index()
  organization!: Organization;

  @Enum(() => Portal)
  @Property({ type: 'text' })
  portal!: Portal;

  @Property({ type: 'boolean', default: false })
  isEnabled: boolean = false;

  // PF credentials (encrypted at application layer)
  @Property({ type: 'text', nullable: true })
  apiKey?: string;

  @Property({ type: 'text', nullable: true })
  apiSecret?: string;

  // Auto-generated on creation for Bayut/dubizzle
  @Property({ nullable: true })
  feedUrl?: string;

  // PF webhook HMAC secret — used only for verifying X-Signature on PF webhook payloads
  @Property({ nullable: true })
  webhookSecret?: string;

  // Feed URL HMAC secret — used only for generating/validating the Bayut/dubizzle feed URL token
  // Separate from webhookSecret so each can be rotated independently
  @Property({ nullable: true })
  feedSecret?: string;

  // PF API key expiry date — visible in PF Expert when the key is generated (max 365 days)
  // Nullable so existing configs are not broken on migration
  @Property({ type: 'timestamptz', nullable: true })
  apiKeyExpiresAt?: Date;

  // Tracks the timestamp of the last successful Bayut/dubizzle lead poll
  @Property({ type: 'timestamptz', nullable: true })
  lastLeadPollAt?: Date;

  @Property({ type: 'timestamptz' })
  createdAt: Date = new Date();

  @Property({ type: 'timestamptz', onUpdate: () => new Date() })
  updatedAt: Date = new Date();

  @Property({ type: 'boolean', default: false })
  isDeleted: boolean = false;
}
```

### Listing Entity

Marketing layer between `InventoryUnit` and portal syndication. Contains all portal-marketing-specific fields.

```typescript
@Entity()
@Filter({ name: 'isDeleted', default: true, cond: { isDeleted: false } })
export class Listing {
  @PrimaryKey({ type: UuidType })
  id: string = v4();

  @ManyToOne(() => Organization)
  @Index()
  organization!: Organization;

  @ManyToOne(() => User)
  @Index()
  agent!: User; // The agent who created/owns this listing

  // Link to inventory unit (null for off-plan)
  @ManyToOne(() => InventoryUnit, { nullable: true })
  @Index()
  inventoryUnit?: InventoryUnit;

  // Reference unit ID for off-plan (future use)
  @Property({ type: 'integer', nullable: true })
  @Index()
  refUnitId?: number;

  @Enum(() => ListingStatus)
  @Property({ type: 'text', default: ListingStatus.DRAFT })
  status: ListingStatus = ListingStatus.DRAFT;

  // Marketing content
  @Property({ type: 'text' })
  title!: string;

  @Property({ type: 'text', nullable: true })
  description?: string;

  @Property({ type: 'text', nullable: true })
  permitNumber?: string;

  // Portal-specific classifications
  @Enum(() => FurnishedStatus)
  @Property({ type: 'text', nullable: true })
  furnishedStatus?: FurnishedStatus;

  @Enum(() => RentalPeriod)
  @Property({ type: 'text', nullable: true })
  rentalPeriod?: RentalPeriod;

  @Enum(() => FinishingType)
  @Property({ type: 'text', nullable: true })
  finishingType?: FinishingType;

  // Marketing media
  @Property({ type: 'jsonb', nullable: true })
  imageUrls?: string[];

  @Property({ type: 'text', nullable: true })
  virtualTourUrl?: string;

  // Portal-specific features/amenities
  @Property({ type: 'jsonb', nullable: true })
  features?: string[];

  @Property({ type: 'timestamptz' })
  createdAt: Date = new Date();

  @Property({ type: 'timestamptz', onUpdate: () => new Date() })
  updatedAt: Date = new Date();

  @Property({ type: 'boolean', default: false })
  isDeleted: boolean = false;

  // Relations
  @OneToMany(() => ListingPortalSync, sync => sync.listing, { cascade: [Cascade.ALL] })
  portalSyncs = new Collection<ListingPortalSync>(this);
}
```

## Integration Flows

### Property Finder Integration

<Steps>
<Step title="Authentication">
Use JWT tokens generated from API key/secret pairs. Tokens auto-refresh before expiry.
</Step>

<Step title="Listing Publication">
- **Create Listing**: POST to `/listings` with full property data
- **Publish Listing**: POST to `/listings/{id}/publish` 
- **Status**: Async confirmation via webhook or polling
</Step>

<Step title="Lead Ingestion">
- **Primary**: Webhook notifications (instant)
- **Fallback**: REST API polling every 15 minutes
- **Deduplication**: By portal reference ID
</Step>
</Steps>

### Bayut/dubizzle Integration

<Steps>
<Step title="XML Feed Generation">
- **Endpoint**: `GET /portal-feed/{organizationId}`
- **Authentication**: HMAC-signed URL parameters
- **Caching**: Redis-backed with configurable TTL
</Step>

<Step title="Lead Ingestion">
- **Method**: Pull API polling every 15 minutes
- **Authentication**: Bearer token per organization
- **Shared Endpoint**: Same API for both Bayut and dubizzle
- **Source Detection**: `source` field in response (`"bayut"` or `"dubizzle"`)
</Step>
</Steps>

## Queue System Architecture

The module uses pg-boss for async job processing with the following queues:

### Job Queues

| Queue Name | Purpose | Retry Strategy |
|---|---|---|
| `listing-sync` | Portal publish/unpublish operations | Exponential backoff, max 3 retries |
| `portal-lead-ingestion` | Convert portal leads to CRM leads | No retries (idempotent) |

### Worker Services

<CodeGroup>
```typescript Syndication Worker
@Injectable()
export class SyndicationWorkerService {
  async handleListingSync(job: Job<ListingSyncJobData>) {
    const { listingId, portal, action } = job.data;
    
    switch (action) {
      case 'publish':
        return this.syndicationService.publishToPortal(listingId, portal);
      case 'unpublish':
        return this.syndicationService.unpublishFromPortal(listingId, portal);
      case 'retry':
        return this.syndicationService.retrySync(listingId, portal);
    }
  }
}
```

```typescript Lead Ingestion Worker
@Injectable()
export class PortalLeadWorkerService {
  async handlePortalLead(job: Job<PortalLeadData>) {
    const leadData = job.data;
    
    // Convert portal lead to CRM lead
    // Deduplication handled by unique index on Lead entity
    return this.leadService.createFromPortalData(leadData);
  }
}
```
</CodeGroup>

## Security & Compliance

### Credential Management

<Warning>
All portal API credentials are encrypted at the application layer before database storage.
</Warning>

<Steps>
<Step title="Encryption">
- **Algorithm**: AES-256-GCM
- **Key Source**: Environment variable `PORTAL_ENCRYPTION_KEY`
- **Scope**: API keys, secrets, and Bearer tokens
</Step>

<Step title="HMAC Verification">
- **Webhooks**: Property Finder webhook payloads verified via X-Signature header
- **Feed URLs**: Bayut/dubizzle feed URLs signed with organization-specific secrets
</Step>

<Step title="Token Rotation">
- **Property Finder**: JWT tokens auto-refresh before expiry
- **Feed Secrets**: Can be rotated independently per organization
</Step>
</Steps>

### Data Privacy

- **No Portal Data Persistence**: Portal responses are not stored, only status updates
- **Lead Attribution**: Portal leads linked to correct agents via mapping tables
- **Audit Trail**: All sync operations logged with timestamps and outcomes

## Error Handling

### Validation Errors

<AccordionGroup>
<Accordion title="Field Validation">
- **Missing Required Fields**: Portal-specific validation before publish
- **Format Errors**: Image URLs, coordinates, permit numbers
- **Value Constraints**: Price limits, area ranges, room counts
</Accordion>

<Accordion title="Portal-Specific Rules">
- **Property Finder**: DLD permit validation, RERA agent checks
- **Bayut/dubizzle**: Image count limits, description length
</Accordion>
</AccordionGroup>

### Sync Failures

| Error Type | Retry Strategy | User Action |
|---|---|---|
| Network timeout | Exponential backoff (3 attempts) | Automatic retry |
| Invalid credentials | No retry | Update portal configuration |
| Insufficient credits | No retry | Top up Property Finder account |
| Compliance rejection | No retry | Fix listing content |
| Rate limit exceeded | Delayed retry | Wait for rate limit reset |

### Lead Processing Errors

- **Duplicate Leads**: Silently ignored (handled by unique index)
- **Invalid Lead Data**: Logged and skipped
- **CRM Integration Failures**: Retried up to 3 times

<Tip>
All error details are logged with correlation IDs for debugging, and user-friendly messages are shown in the portal dashboard.
</Tip>
