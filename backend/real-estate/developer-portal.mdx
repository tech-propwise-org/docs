The real estate module is structured as **six specialized modules** spanning three visibility scopes: **global** (no RLS), **org-private** (standard RLS), and **cross-org** (custom RLS).

The external **Propwise Labs API** stores and serves all property data — both structural (DLD registry) and enrichment (images, payment plans, documents, amenities, pricing, geocoding where exposed). The CRM database stores only CRM-specific data (developer portal auth, ownership links, property interests, marketplace listings, secondary unit context).

<Note>
**Architecture:** All modules consume property data exclusively through `PropwiseLabsService`. The CRM no longer opens a second MikroORM connection or maps Propwise Labs entities locally. The reference database configuration has been completely removed and replaced with Propwise Labs API configuration (`app.propwiseLabs.*`) including API credentials, timeout settings, and Bird Verify integration for developer portal contact verification.
</Note>

## Propwise Labs module

**Path:** `src/modules/real-estate/propwise-labs/`

### Purpose

Provides the CRM backend facade over the external Propwise Labs API. It handles API authentication, bearer-token caching, read calls for property data, lookup data, change-request submissions for developer portal mutations, and typed wrappers for super-level direct Labs mutations. Local `Ref*` MikroORM entities and the named `reference` EntityManager have been removed.

### API client pattern

`PropwiseLabsService` reads API credentials from `app.propwiseLabs.*`, exchanges `PROPWISE_LABS_API_KEY` and `PROPWISE_LABS_API_SECRET` for a short-lived bearer token, and reuses that token until it is close to expiry. The service includes configurable timeout settings and comprehensive error handling for external API dependencies.

```typescript
@Injectable()
export class PropwiseLabsService {
  private accessToken?: string;
  private tokenExpiresAt = 0;

  async searchProjects(filters: SearchProjectsDto): Promise<PropwiseLabsProject[]> {
    const token = await this.ensureValidToken();
    // API call with bearer token
  }

  async submitChangeRequest(request: ChangeRequestDto): Promise<ChangeRequestResponse> {
    const token = await this.ensureValidToken();
    // Submit mutation request to Propwise Labs
  }

  private async ensureValidToken(): Promise<string> {
    if (this.accessToken && Date.now() < this.tokenExpiresAt - 60000) {
      return this.accessToken;
    }
    // Exchange credentials for new token
  }
}
```

## Developer portal module

**Path:** `src/modules/real-estate/developer-portal/`

Provides a completely separate authentication system for real estate developer sales agents. Developers sign up, get admin approval, then manage their off-plan projects through this portal. All project data is stored via `PropwiseLabsService` using either direct API calls or change requests.

### Authentication (separate from CRM)

- **Separate JWT secrets**:
  - Access token: `JWT_DEVELOPER_SECRET` (configured in `jwt.config.ts` under `developerToken`, expires in 8h)
  - Refresh token: `JWT_DEVELOPER_REFRESH_SECRET` (configured in `jwt.config.ts` under `developerRefreshToken`, expires in 7d)
- **Token payload**: `{ sub, email, type: 'developer', refDeveloperId, tokenVersion }`
- **Refresh token mechanism**: 64-character random token stored in database with expiry timestamp
- **Guard**: `DeveloperAuthGuard` at class-level validates tokens and sets `request.developerUser`
- **Public endpoints**: Use `@PublicEndpoint()` for signup/login/refresh
- **Response DTOs**: `DeveloperLoginResultDto` (includes both tokens), `DeveloperProfileDto` (constructor-based)
- **Token invalidation**: On logout, `tokenVersion` increments and refresh token is cleared — all tokens invalidated. On password reset, token version increments (invalidates existing sessions).
- **Bird Verify integration**: Contact verification using Bird API for OTP validation during signup and profile updates

<Warning>
The developer portal uses its own JWT secret and guard. Never use `AuthGuard` (CRM) on developer portal controllers — use `DeveloperAuthGuard` instead.
</Warning>

### Entities

<Tabs>
<Tab title="DeveloperUser">

- UUID PK, unique email (partial index where `is_deleted = false`)

<Note>

</Note>

### Project management architecture

The developer portal follows a **three-level hierarchy**: Projects contain Buildings, and Buildings contain Units. Projects are lightweight parent containers; Buildings hold all enrichment data (pricing, handover, gallery, etc.); Units are created under Buildings.

<Steps>
<Step title="Project CRUD">
Uses `PropwiseLabsService` for direct API calls or change requests depending on operation type
</Step>
<Step title="Building CRUD">
Verifies project ownership, then delegates to `PropwiseLabsService`. Buildings inherit fields from their parent project by default
</Step>
<Step title="Unit CRUD">
Verifies building→project ownership, then delegates to `PropwiseLabsService`
</Step>
</Steps>

**Data flow with change requests:**

```typescript
// Create project flow (change request):
const changeRequest = await this.propwiseLabsService.submitChangeRequest({
  type: 'CREATE_PROJECT',
  data: projectData,
  developerId: user.refDeveloperId,
});

// Update building flow (direct API):
const building = await this.propwiseLabsService.updateBuilding(buildingId, {
  ...enrichmentFields,
  status: dto.status ?? project.status,  // Inherit if not provided
});

// Create unit flow (change request):
const changeRequest = await this.propwiseLabsService.submitChangeRequest({
  type: 'CREATE_UNIT',
  buildingId,
  data: unitData,
});
```

**Collection entities** have **both** a nullable `project_id` and a nullable `building_id` FK, allowing items to belong to either level.

**Inheritance pattern:**
- Buildings inherit `community`, `area`, `description`, `status`, `percentCompleted`, `startDate`, `endDate` from their parent project by default, but each can be overridden per building
- **Payment plans**: Set at project level; buildings inherit by default but can define their own
- **Documents**: Available at both project and building levels (additive)
- **Amenities**: Set at project level; buildings inherit by default but can define their own
- **Gallery**: Available at both project and building levels

### Approval flow

<Steps>
<Step title="Signup (with agent card)">
Developer calls `POST /developer-portal/auth/signup` with multipart/form-data including agent card file. Backend validates file type/size, processes images (converts to WebP, resizes), stores in R2. Account created with `isApproved = false`, `isEmailVerified = false`. Email verification code sent (6-digit, 30min expiry).
</Step>
<Step title="Contact verification">
Developer verifies email with code and phone with Bird OTP. `isEmailVerified = true`. Account enters pending approval state.
</Step>
<Step title="Admin review">
System admin reviews agent card via `GET /system-admin/developer-users/:id`. Admin approves or rejects.
</Step>
<Step title="On approval">
`isApproved = true`, `approvedAt = now()`. Developer can now login.
</Step>
<Step title="Login">
Receives access token (8h) + refresh token (7d via HTTP-only cookie).
</Step>
<Step title="Project publishing">
Projects start as `isPublished=false`. Developer publishes project, making it visible in the off-plan directory.
</Step>
</Steps>

### Agent card details

- Signup: multipart/form-data, field name: `agentCard`
- Authenticated update: multipart/form-data, field name: `file`
- Max size: 10 MB
- Formats: JPEG, PNG, GIF, WebP (images) or PDF (documents)
- Images auto-converted to WebP: resized to max 1200x1600, WebP 85% quality
- PDFs stored as-is (no processing)
- Storage: `agent-cards/{userId}/{uploadId}.webp` (images) or `.pdf` (documents)

### Matching engine

The property interest module includes an automated matching engine that:

1. **Analyzes property interests**: Reviews lead/deal criteria (budget, location, property type)
2. **Queries Propwise Labs**: Uses `PropwiseLabsService` to find matching properties
3. **Scores matches**: Ranks properties based on criteria alignment
4. **Notifies stakeholders**: Alerts when high-scoring matches are found
5. **Updates interests**: Marks interests when matches are presented

```typescript
@Injectable()
export class PropertyMatchingService {
  constructor(
    private readonly propwiseLabsService: PropwiseLabsService,
    private readonly em: EntityManager,
  ) {}

  async findMatches(interest: PropertyInterest): Promise<PropertyMatch[]> {
    // Build search criteria from interest
    const searchCriteria = {
      budgetMin: interest.budgetMin,
      budgetMax: interest.budgetMax,
      areas: interest.preferredAreas ? JSON.parse(interest.preferredAreas) : [],
      propertyTypes: interest.propertyTypes ? JSON.parse(interest.propertyTypes) : [],
    };

    // Query Propwise Labs via PropwiseLabsService
    const matchingProperties = await this.propwiseLabsService.searchProperties(searchCriteria);

    // Score and create property matches
    const matches: PropertyMatch[] = [];
    for (const property of matchingProperties) {
      const score = this.calculateMatchScore(interest, property);
      if (score >= 70) { // Minimum threshold
        const match = this.em.create(PropertyMatch, {
          propertyInterest: interest,
          refProjectId: property.projectId,
          refUnitId: property.unitId,
          matchScore: score,
        });
        matches.push(match);
      }
    }

    await this.em.persistAndFlush(matches);
    return matches;
  }

  private calculateMatchScore(interest: PropertyInterest, property: any): number {
    let score = 0;
    
    // Budget alignment (40% weight)
    if (property.price >= interest.budgetMin && property.price <= interest.budgetMax) {
      score += 40;
    }

    // Location match (35% weight)
    const preferredAreas = interest.preferredAreas ? JSON.parse(interest.preferredAreas) : [];
    if (preferredAreas.includes(property.areaName)) {
      score += 35;
    }

    // Property type match (25% weight)
    const propertyTypes = interest.propertyTypes ? JSON.parse(interest.propertyTypes) : [];
    if (propertyTypes.includes(property.propertyType)) {
      score += 25;
    }

    return Math.min(score, 100);
  }
}
```
