**Path:** `src/modules/real-estate/developer-portal/`

Provides a completely separate authentication system for real estate developer sales agents. Developers sign up, get admin approval, then manage their off-plan projects through this portal. All project data is stored in the reference database via `ReferenceDataService`.

## Authentication (separate from CRM)

- **Separate JWT secrets**:
  - Access token: `JWT_DEVELOPER_SECRET` (configured in `jwt.config.ts` under `developerToken`, expires in 8h)
  - Refresh token: `JWT_DEVELOPER_REFRESH_SECRET` (configured in `jwt.config.ts` under `developerRefreshToken`, expires in 7d)
- **Token payload**: `{ sub, email, type: 'developer', refDeveloperId, tokenVersion }`
- **Refresh token mechanism**: 64-character random token stored in database with expiry timestamp
- **Guard**: `DeveloperAuthGuard` at class-level validates tokens and sets `request.developerUser`
- **Public endpoints**: Use `@PublicEndpoint()` for signup/login/refresh
- **Response DTOs**: `DeveloperLoginResultDto` (includes both tokens), `DeveloperProfileDto` (constructor-based)
- **Token invalidation**: On logout, `tokenVersion` increments and refresh token is cleared — all tokens invalidated. On password reset, token version increments (invalidates existing sessions).

<Warning>
The developer portal uses its own JWT secret and guard. Never use `AuthGuard` (CRM) on developer portal controllers — use `DeveloperAuthGuard` instead.
</Warning>

## Entities

<Tabs>
<Tab title="DeveloperUser">

- UUID PK, unique email (partial index where `is_deleted = false`)
- passwordHash (bcrypt, saltRounds=10)
- **agentCardUrl**: optional string (max 2000 chars) — agent card image URL uploaded during signup
- **phone**: optional string — contact phone number
- **whatsappNumber**: optional string — WhatsApp contact number
- **languages**: optional JSON array — languages spoken (e.g., `["English", "Arabic"]`)
- **avatarUrl**: optional string — profile avatar URL
- isApproved, isActive, tokenVersion
- **refreshToken**: 64-char token stored in DB (indexed)
- **refreshTokenExpiry**: expiration timestamp for refresh token
- isEmailVerified, emailVerificationCode, emailVerificationExpiry
- passwordResetCode, passwordResetExpiry
- refDeveloperId (integer link to reference DB)
- developerName (denormalized)

<Note>
The developer user serves as the sales contact for projects they publish. When a project is displayed in the off-plan directory, the developer user's contact information (name, email, phone, WhatsApp, languages, avatar) is shown as the sales contact card.
</Note>

</Tab>
<Tab title="DeveloperProjectLink">

- UUID PK, **tracks developer-to-project ownership in CRM**
- ManyToOne: DeveloperUser (indexed)
- `refProjectId: number` — integer ID of the project in the reference database
- `refDeveloperId: number` — denormalized from DeveloperUser (indexed)
- Composite index: `[developerUser, refProjectId]`
- `createdAt` timestamp

```typescript
@Entity()
@Index({ properties: ['developerUser', 'refProjectId'] })
export class DeveloperProjectLink {
  @PrimaryKey({ type: UuidType })
  id: string = v4();

  @ManyToOne(() => DeveloperUser)
  @Index()
  developerUser!: DeveloperUser;

  @Property({ type: 'int' })
  @Index()
  refProjectId!: number;

  @Property({ type: 'int' })
  @Index()
  refDeveloperId!: number;

  @Property({ onCreate: () => new Date() })
  createdAt: Date = new Date();
}
```

## Project management architecture

The developer portal follows a **three-level hierarchy**: Projects contain Buildings, and Buildings contain Units. Projects are lightweight parent containers; Buildings hold all enrichment data (pricing, handover, gallery, etc.); Units are created under Buildings.

<Steps>
<Step title="Project CRUD">
Delegates to `ReferenceDataService` (writes to reference DB). Ownership tracked via `DeveloperProjectLink` (CRM DB).
</Step>
<Step title="Building CRUD">
Verifies project ownership via `DeveloperProjectLink`, then delegates to `ReferenceDataService`. Buildings inherit fields from their parent project by default.
</Step>
<Step title="Unit CRUD">
Verifies building→project ownership, then delegates to `ReferenceDataService`.
</Step>
</Steps>

**Data flow in detail:**

```typescript
// Create project flow:
const project = await this.referenceDataService.createProject(data);  // Ref DB (slim)
const link = this.em.create(DeveloperProjectLink, {                   // CRM DB
  developerUser: user.id,
  refProjectId: project.id,
  refDeveloperId: user.refDeveloperId,
});

// Create building flow:
await this.verifyOwnership(projectId, refDeveloperId);          // Check CRM DB link
const building = await this.referenceDataService.createBuilding({
  project, name, ...enrichmentFields,                           // Ref DB
  status: dto.status ?? project.status,                         // Inherit if not provided
  area: dto.areaId ? ... : project.area,                        // Inherit if not provided
});

// Create unit flow:
await this.verifyBuildingOwnership(buildingId, refDeveloperId); // Check building→project→link
const unit = await this.referenceDataService.createUnit({
  building, project: building.project, ...unitFields,           // Ref DB
});
```

**Collection entities** (`RefProjectGalleryImage`, `RefPaymentPlan`, `RefProjectDocument`, `RefProjectAmenity`) have **both** a nullable `project_id` and a nullable `building_id` FK, allowing items to belong to either level.

**Inheritance pattern:**
- Buildings inherit `community`, `area`, `description`, `status`, `percentCompleted`, `startDate`, `endDate` from their parent project by default, but each can be overridden per building.
- **Payment plans**: Set at project level; buildings inherit by default but can define their own.
- **Documents**: Available at both project and building levels (additive).
- **Amenities**: Set at project level; buildings inherit by default but can define their own.
- **Gallery**: Available at both project and building levels.

## API endpoints

Split across **four controllers** for single responsibility:

<Tabs>
<Tab title="Auth">

```
POST /developer-portal/auth/signup                 # Signup with agent card file (multipart/form-data)
POST /developer-portal/auth/verify-email           # Verify email with 6-digit code
POST /developer-portal/auth/resend-verification    # Resend verification code
POST /developer-portal/auth/login                  # Returns access token; refresh token set as HTTP-only cookie
POST /developer-portal/auth/refresh                # Refresh tokens using HTTP-only cookie
POST /developer-portal/auth/reset-password         # Request password reset code
POST /developer-portal/auth/set-password           # Set new password with code
POST /developer-portal/auth/logout                 # Invalidates all tokens, clears refresh cookie
```

<Tip>
Include `isPublished: true` or `isPublished: false` in the `PATCH` request body for projects and buildings. There is no separate `/publish` toggle endpoint.
</Tip>

<Note>
Building create and list are nested under `projects/:projectId/buildings` (the project is the parent context). Building update, delete, and detail are flat at `/buildings/:id` because the operation targets a specific resource and the project ID is not needed for routing.
</Note>

## Approval flow

<Steps>
<Step title="Signup (with agent card)">
Developer calls `POST /developer-portal/auth/signup` with multipart/form-data including agent card file. Backend validates file type/size, processes images (converts to WebP, resizes), stores in R2. Account created with `isApproved = false`, `isEmailVerified = false`. Email verification code sent (6-digit, 30min expiry).
</Step>
<Step title="Email verification">
Developer verifies email with code. `isEmailVerified = true`. Account enters pending approval state.
</Step>
<Step title="Admin review">
System admin reviews agent card via `GET /system-admin/developer-users/:id`. Admin approves or rejects.
</Step>
<Step title="On approval">
`isApproved = true`, `approvedAt = now()`. Developer can now login.
</Step>
<Step title="Login">
Receives access token (8h) + refresh token (7d via HTTP-only cookie).
</Step>
<Step title="Project publishing">
Projects start as `isPublished=false`. Developer publishes project, making it visible in the off-plan directory.
</Step>
</Steps>

## Agent card details

- Signup: multipart/form-data, field name: `agentCard`
- Authenticated update: multipart/form-data, field name: `file`
- Max size: 10 MB
- Formats: JPEG, PNG, GIF, WebP (images) or PDF (documents)
- Images auto-converted to WebP: resized to max 1200x1600, WebP 85% quality
- PDFs stored as-is (no processing)
- Storage: `agent-cards/{userId}/{uploadId}.webp` (images) or `.pdf` (documents)
