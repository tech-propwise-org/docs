# Off-Plan Directory Implementation

This specification details the implementation of a new **Off-Plan** directory under the Properties section, featuring a card/map split view, 2GIS integration, and comprehensive building lifecycle management.

## Overview

The Off-Plan directory displays all published buildings from developer portal users in a sophisticated interface with:
- Card and map view modes with seamless switching
- Rich filtering and search capabilities  
- 2GIS map integration with custom markers
- Detailed building view with tabbed interface
- Publication system with lifecycle management

<Note>
Off-plan data is served through domain endpoints under `/off-plan/*` that read Propwise Labs catalog data and apply CRM-owned visibility controls.
</Note>

## Architecture Decisions

### Buildings as Primary Entity

**Buildings** serve as the primary enrichment entity over projects because:

- Buildings have dedicated `coverImageUrl`, `status`, `endDate`, `completionDate`, `paymentPlans`, `images`, `documents`, `amenities`
- Buildings can override inherited fields from projects (status, area, community, description)
- Multiple buildings per project can have different lifecycle statuses and pricing

<CodeGroup>
```typescript Primary Endpoints
GET /off-plan/buildings          // List page data
GET /off-plan/buildings/:id      // Detail page data
GET /off-plan/buildings/map      // Map markers
```

```typescript Supporting Endpoints

GET /propwise-labs/developers?q=             // Developer filter options

```
</CodeGroup>

### Publication System

Buildings are published through the developer portal, which manages `off_plan_building_publication.is_published` status:

<Steps>
<Step title="Publish Readiness Validation">
Before publishing, buildings must satisfy the complete building contract:
- Required fields: `name`, `buildingNumber`, `descriptionEn`, `floors`, `googleMapsLink`, `startDate`, `coverImageUrl`
- Area data: `area.id`, `plotSize`, `actualArea`, `parkingCount`, `serviceChargePerSqft`
- Media: At least 1 media item and `salesStatus`
</Step>

<Step title="Villa Project Requirements">
Villa projects have different requirements:
- Basic: `name`, `descriptionEn`, `imageUrl`, `googleMapsLink`, `area.id`
- Location: `latitude`, `longitude`
- Media: At least 1 media item and `salesStatus`
</Step>

<Step title="Auto-Maintained Sales Status">
`salesStatus` is automatically updated when unit availability changes:
- `OUT_OF_STOCK`: No units remain `AVAILABLE`
- `ON_SALE`: At least one `AVAILABLE` unit exists
</Step>
</Steps>

### Frontend Status Mapping

<AccordionGroup>
<Accordion title="Status Color Coding">
Frontend status is derived from `building.status` through `getOffPlanFrontendStatus()`:

| Backend Status | Frontend Display | Color  |
|---------------|------------------|--------|
| `ACTIVE`      | On Sale          | Orange |
| `PENDING`     | EOI              | Purple |
| `FINISHED`    | Out of Stock     | Gray   |
</Accordion>

<Accordion title="Lifecycle Classification">
The off-plan lifecycle helper treats:
- **Off-plan eligible**: `ACTIVE` and `PENDING` statuses
- **Excluded**: `UNKNOWN` status (secondary-only)
- **Auto-filtering**: Only published buildings matching off-plan lifecycle are shown
</Accordion>
</AccordionGroup>

const handleCardHover = (buildingId: string) => {
  map.panTo(getMarkerPosition(buildingId))
  highlightMarker(buildingId)
  showMarkerPreview(buildingId)
}

// Marker hover -> Card interaction  
const handleMarkerHover = (buildingId: string) => {
  scrollToCard(buildingId)
  highlightCard(buildingId)
  showPreviewCard(buildingId)
}
