The real estate module is structured as **six specialized modules** spanning three visibility scopes: **global** (no RLS), **org-private** (standard RLS), and **cross-org** (custom RLS). This page documents how these modules integrate with the CRM through property interests, reference data IDs, and inventory ownership links.

## Architecture overview

An external **reference database** (Neon "propwise-lab") stores **all property data** — both structural (DLD registry) and enrichment (gallery, payment plans, documents, amenities, pricing, geocoding). The CRM database stores only CRM-specific data (developer portal auth, ownership links, property interests, marketplace listings, secondary unit context).

<Note>
All modules consume property data exclusively through `ReferenceDataService`. If the reference database is swapped for an external property API, only `ReferenceDataService` needs to change.
</Note>

### Module structure

```
src/modules/real-estate/
├── reference-data/        # Second DB — ALL property data (structural + enrichment)
│   ├── reference-data.controller.ts       # /reference — areas, cities, communities, developers, property types, units/:id
│   ├── reference-projects.controller.ts   # /reference/projects — search (?type=off-plan), map, counts, detail, nested buildings/units
│   └── reference-buildings.controller.ts  # /reference/buildings — search (?type=off-plan), detail, nested units
├── developer-portal/      # Separate auth + project management (writes via ReferenceDataService)
│   ├── auth/developer-auth.controller.ts  # /developer-portal/auth — signup (with agent card), login, refresh, logout, password
│   ├── profile/developer-profile.controller.ts # /developer-portal/me — profile GET/PATCH, agent-card upload
│   └── project-management/
│       ├── dev-projects.controller.ts     # /developer-portal/projects — project CRUD, collections, nested building create/list
│       └── dev-buildings.controller.ts    # /developer-portal/buildings — building CRUD (flat), collections, nested unit CRUD
├── map/                   # MapService (2GIS geocoding utility — shared across modules)
├── inventory/             # Org-scoped secondary units (standard RLS)
├── shared-marketplace/    # Cross-org listings + deals (custom RLS)
└── property-interest/     # Property interests + matching engine (uses ReferenceDataService)
```

| Module             | Scope      | RLS Behavior                         | Description                                                                    |
| ------------------ | ---------- | ------------------------------------ | ------------------------------------------------------------------------------ |

| Data Type                                                  | Database     | Access Pattern                              |
| ---------------------------------------------------------- | ------------ | ------------------------------------------- |
| Projects, units, buildings, areas, communities, developers | Reference DB | `ReferenceDataService`                      |
| Gallery images, payment plans, documents, amenities        | Reference DB | `ReferenceDataService`                      |

| Developer auth (DeveloperUser)                             | CRM DB       | Direct entity access                        |

| Marketplace listings & deals (SharedListing, SharedDeal)   | CRM DB       | Direct entity access                        |
| Property interests (Lead/Deal)                             | CRM DB       | Direct entity access, ref IDs as plain ints |
| Property matches                                           | CRM DB       | Direct entity access, ref IDs as plain ints |

## Cross-database reference pattern

CRM entities that reference off-plan projects or units store **plain integer IDs** (`refProjectId`, `refUnitId`) pointing to the reference database. No cross-database foreign keys exist — this decouples the CRM from the property data source.

```typescript
// CRM entity referencing reference DB data:
@Property({ type: 'int', nullable: true })
@Index()
refProjectId?: number;  // ID in reference DB projects table

@Property({ type: 'int', nullable: true })
@Index()
refUnitId?: number;     // ID in reference DB units table
```

### Entity pattern

All reference entities are mapped to the external DB tables. They have **no UUID primary keys** (use `number` PKs matching the external DB), **no organization FK**, and use **soft delete** (`deletedAt` timestamp) for enrichment entities.

  @Property({ nullable: true })

  @ManyToOne(() => RefArea, { fieldName: 'area_id' })

@Injectable()
export class SharedMarketplaceService {
  private applySharedRLS(qb: QueryBuilder<any>, entityAlias: string): QueryBuilder<any> {
    const tenantId = this.tenantContext.getTenantId();
    
    return qb.andWhere(
      `(${entityAlias}.listing_org_id = ? OR ${entityAlias}.target_org_id = ? OR ${entityAlias}.buying_org_id = ?)`,
      [tenantId, tenantId, tenantId]
    );
  }
}

### Property matching engine

The matching engine provides automated property recommendations:

```typescript
@Injectable()
export class PropertyMatchingService {
  constructor(
    private readonly referenceDataService: ReferenceDataService,
    private readonly tenantContext: TenantContext,
    @InjectEntityManager() private readonly em: EntityManager,
  ) {}

  async findMatchingProperties(interest: LeadPropertyInterest): Promise<PropertyMatch[]> {
    const matches: PropertyMatch[] = [];

    // Match against reference database projects/units
    if (interest.budgetMin || interest.budgetMax) {
      const refMatches = await this.referenceDataService.searchProjectsByBudget({
        minPrice: interest.budgetMin,
        maxPrice: interest.budgetMax,
        interestType: interest.interestType,
      });
      
      matches.push(...refMatches.map(project => ({
        type: 'reference_project',
        refProjectId: project.id,
        matchScore: this.calculateMatchScore(interest, project),
      })));
    }

    // Match against inventory units
    const inventoryMatches = await this.findInventoryMatches(interest);
    matches.push(...inventoryMatches);

    // Match against shared marketplace listings
    const marketplaceMatches = await this.findMarketplaceMatches(interest);
    matches.push(...marketplaceMatches);

    return matches.sort((a, b) => b.matchScore - a.matchScore);
  }

  private calculateMatchScore(interest: LeadPropertyInterest, property: any): number {
    let score = 0;
    
    // Budget alignment
    if (interest.budgetMin && property.priceFrom >= interest.budgetMin) score += 20;
    if (interest.budgetMax && property.priceTo <= interest.budgetMax) score += 20;
    
    // Location preferences (if available)
    if (interest.preferredAreas?.includes(property.areaId)) score += 30;
    
    // Property type preferences
    if (interest.propertyTypes?.includes(property.propertyTypeId)) score += 20;
    
    // Interest type alignment
    if (interest.interestType === 'buy' && property.availableForSale) score += 10;
    if (interest.interestType === 'rent' && property.availableForRent) score += 10;
    
    return score;
  }
}
```

## Contact and company relations

Project interests create indirect relationships between contacts and properties:

- Access a contact's property interests: `contact.person.leads -> lead.propertyInterests`

## Person entity ownership tracking

The `Person` entity tracks real estate ownership through the inventory module:

```typescript
@Entity()
export class Person extends TenantScopedEntity {
  // ... other person fields

  @OneToMany(() => InventoryUnitOwnership, (ownership) => ownership.person)
  unitOwnerships = new Collection<InventoryUnitOwnership>(this);

  // Computed property for owned units
  get ownedUnits(): SecondaryUnit[] {
    return this.unitOwnerships.getItems().map(ownership => ownership.unit);
  }
}
```

This relationship enables:
- Tracking which persons own units in the internal inventory
- Portfolio management for high-value clients
- Cross-referencing ownership with deal history
- Identifying potential sellers for marketplace listings

## Reference data service pattern

All property data access flows through the `ReferenceDataService`, which maintains the single connection to the external property database:

```typescript
@Injectable()
export class ReferenceDataService {
  constructor(
    @InjectEntityManager('reference')
    private readonly em: EntityManager,
  ) {}

  async getProject(id: number): Promise<RefProject | null> {
    return this.em.findOne(RefProject, { id }, {
      populate: ['developer', 'area', 'gallery', 'paymentPlans', 'buildings', 'buildings.units']
    });
  }

  async searchProjects(criteria: ProjectSearchCriteria): Promise<RefProject[]> {
    const qb = this.em.createQueryBuilder(RefProject, 'p');
    
    if (criteria.areaIds?.length) {
      qb.andWhere({ area: { id: { $in: criteria.areaIds } } });
    }
    
    if (criteria.priceMin) {
      qb.andWhere({ priceFrom: { $gte: criteria.priceMin } });
    }
    
    if (criteria.priceMax) {
      qb.andWhere({ priceTo: { $lte: criteria.priceMax } });
    }
    
    if (criteria.type === 'off-plan') {
      qb.andWhere({ status: 'off-plan' });
    }

    return qb.getResult();
  }

  async getUnit(id: number): Promise<RefUnit | null> {
    return this.em.findOne(RefUnit, { id }, {
      populate: ['building', 'building.project', 'building.project.developer', 'building.project.area']
    });
  }

  async createProject(data: CreateProjectDto): Promise<RefProject> {
    const project = this.em.create(RefProject, data);
    await this.em.persistAndFlush(project);
    return project;
  }

  async updateProject(id: number, data: UpdateProjectDto): Promise<RefProject> {
    const project = await this.em.findOneOrFail(RefProject, { id });
    this.em.assign(project, data);
    await this.em.flush();
    return project;
  }
}
```
