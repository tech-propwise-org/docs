The real estate module is structured as **six specialized modules** spanning three visibility scopes: **global** (no RLS), **org-private** (standard RLS), and **cross-org** (custom RLS). This page documents how these modules integrate with the CRM through property interests, reference data IDs, and inventory ownership links.

## Architecture overview

An external **reference database** (Neon "propwise-lab") stores **all property data** — both structural (DLD registry) and enrichment (gallery, payment plans, documents, amenities, pricing, geocoding). The CRM database stores only CRM-specific data (developer portal auth, ownership links, property interests, marketplace listings, secondary unit context).

<Note>
All modules consume property data exclusively through `ReferenceDataService`. If the reference database is swapped for an external property API, only `ReferenceDataService` needs to change.
</Note>

### Module structure

```
src/modules/real-estate/
├── reference-data/        # Second DB — ALL property data (structural + enrichment)
│   ├── reference-data.controller.ts       # /reference — areas, cities, communities, developers, property types, units/:id
│   ├── reference-projects.controller.ts   # /reference/projects — search (?type=off-plan), map, counts, detail, nested buildings/units
│   └── reference-buildings.controller.ts  # /reference/buildings — search (?type=off-plan), detail, nested units
├── developer-portal/      # Separate auth + project management (writes via ReferenceDataService)
│   ├── auth/developer-auth.controller.ts  # /developer-portal/auth — signup (with agent card), login, refresh, logout, password
│   ├── profile/developer-profile.controller.ts # /developer-portal/me — profile GET/PATCH, agent-card upload
│   └── project-management/
│       ├── dev-projects.controller.ts     # /developer-portal/projects — project CRUD, collections, nested building create/list
│       └── dev-buildings.controller.ts    # /developer-portal/buildings — building CRUD (flat), collections, nested unit CRUD
├── map/                   # MapService (2GIS geocoding utility — shared across modules)
├── inventory/             # Org-scoped secondary units (standard RLS)
├── shared-marketplace/    # Cross-org listings + deals (custom RLS)
└── property-interest/     # Property interests + matching engine (uses ReferenceDataService)
```

## Scope summary

| Module             | Scope      | RLS Behavior                         | Description                                                                    |
| ------------------ | ---------- | ------------------------------------ | ------------------------------------------------------------------------------ |
| reference-data     | Global     | Read + write, second DB              | ALL property data; 3 controllers (general, projects, buildings)                |
| developer-portal   | Global     | No org FK, separate JWT              | Auth + profile + project/building management via ref DB                        |
| map (shared)       | N/A        | No entities, pure utility            | MapService (2GIS geocoding) — used by developer-portal & reference controllers |
| inventory          | Org-scoped | Standard RLS via TenantContext       | Secondary units owned by an organization                                       |
| shared-marketplace | Cross-org  | Custom RLS (both participating orgs) | Listings + deals across organizations                                          |
| property-interest  | Org-scoped | Standard RLS                         | Lead/deal property interests + matching engine                                 |

## Data ownership and access patterns

| Data Type                                                  | Database     | Access Pattern                              |
| ---------------------------------------------------------- | ------------ | ------------------------------------------- |
| Projects, units, buildings, areas, communities, developers | Reference DB | `ReferenceDataService`                      |
| Gallery images, payment plans, documents, amenities        | Reference DB | `ReferenceDataService`                      |
| Geocoding (lat/lng), pricing, sale status, handover        | Reference DB | `ReferenceDataService`                      |
| Developer auth (DeveloperUser)                             | CRM DB       | Direct entity access                        |
| Developer-to-project ownership (DeveloperProjectLink)      | CRM DB       | Direct entity access                        |
| Secondary units (org-scoped inventory)                     | CRM DB       | Direct entity access                        |
| Marketplace listings & deals (SharedListing, SharedDeal)   | CRM DB       | Direct entity access                        |
| Property interests (Lead/Deal)                             | CRM DB       | Direct entity access, ref IDs as plain ints |
| Property matches                                           | CRM DB       | Direct entity access, ref IDs as plain ints |

## Cross-database reference pattern

CRM entities that reference off-plan projects or units store **plain integer IDs** (`refProjectId`, `refUnitId`) pointing to the reference database. No cross-database foreign keys exist — this decouples the CRM from the property data source.

```typescript
// CRM entity referencing reference DB data:
@Property({ type: 'int', nullable: true })
@Index()
refProjectId?: number;  // ID in reference DB projects table

@Property({ type: 'int', nullable: true })
@Index()
refUnitId?: number;     // ID in reference DB units table
```

## Reference data module

**Path:** `src/modules/real-estate/reference-data/`

The reference data module connects to a second PostgreSQL database (Neon "propwise-lab") containing **all property data**: DLD registry data (structural) plus enrichment data (gallery, payment plans, documents, amenities, sales contacts, pricing, geocoding). It provides read endpoints for the off-plan directory and write endpoints for the developer portal.

### Second database connection

Configured in `mikro-orm-reference.config.ts` at project root. Registered in `app.module.ts` with `contextName: 'reference'`:

```typescript
MikroOrmModule.forRoot({
  ...mikroOrmReferenceConfig,
  contextName: 'reference',
});
```

### Entity pattern

All reference entities are mapped to the external DB tables. They have **no UUID primary keys** (use `number` PKs matching the external DB), **no organization FK**, and use **soft delete** (`deletedAt` timestamp) for enrichment entities.

```typescript
@Entity({ tableName: 'projects' })
export class RefProject {
  @PrimaryKey()
  id!: number;

  @Property({ nullable: true })
  name?: string;

  @ManyToOne(() => RefDeveloper, { fieldName: 'developer_id', nullable: true })
  developer?: RefDeveloper;

  @ManyToOne(() => RefArea, { fieldName: 'area_id' })
  area!: RefArea;

  // Enrichment fields
  @Property({ type: 'double', nullable: true })
  latitude?: number;

  @Property({ fieldName: 'price_from', type: 'numeric', nullable: true })
  priceFrom?: number;

  @Property({ fieldName: 'is_published', default: false, nullable: true })
  isPublished?: boolean;

  // Enrichment collections
  @OneToMany(() => RefProjectGalleryImage, (img) => img.project)
  gallery = new Collection<RefProjectGalleryImage>(this);

  @OneToMany(() => RefPaymentPlan, (p) => p.project)
  paymentPlans = new Collection<RefPaymentPlan>(this);
  // ... documents, amenities
}
```

### Service injection pattern

Use the named EntityManager for the reference context:

```typescript
@Injectable()
export class ReferenceDataService {
  constructor(
    @InjectEntityManager('reference')
    private readonly em: EntityManager,
  ) {}
}
```

### Module registration

Feature entities are registered under the `'reference'` context name:

```typescript
MikroOrmModule.forFeature(
  [
    RefArea,
    RefProject,
    RefUnit,
    RefBuilding,
    RefDeveloper,
    // ... all reference entities
  ],
  'reference',
)
```

## Deal creation with property interest

### From a lead (conversion)

When creating a deal from a lead:

<Steps>
<Step title="Copy property interest">
LeadPropertyInterest is copied to DealPropertyInterest
</Step>
<Step title="Transfer reference IDs">
`refProjectId` and `refUnitId` (plain ints) are copied directly
</Step>
<Step title="Resolve CRM entities">
`secondaryUnit` and `sharedListing` (ManyToOne) are resolved from CRM DB
</Step>
</Steps>

### Walk-in (direct deal creation)

When creating a deal directly (walk-in):

<Steps>
<Step title="Accept interest data">
`CreatePropertyInterestDto` accepts: `refProjectId` (int), `refUnitId` (int), `secondaryUnitId` (UUID), `sharedListingId` (UUID)
</Step>
<Step title="Store references">
Int IDs are stored directly; UUID entities are validated via lookup
</Step>
</Steps>

## Deal filtering

Kanban and List queries use:

- `refProjectId` (int) — filters by reference DB project
- `secondaryUnitId` (UUID) — filters by CRM secondary unit

<Note>
These filters appear in the deal view query parameters alongside CRM filters like `stageId`, `assignedTo`, etc.
</Note>

## Contact and company relations

Project interests are managed through `LeadPropertyInterest` and `DealPropertyInterest` entities.

- Access a contact's project interests via: `contact.person.leads -> lead.propertyInterests`

## Site visit entity

The `SiteVisit` entity links to real estate data:

- `SiteVisit.refProjectId: number` (plain int to reference DB)
- `SiteVisit.secondaryUnit` (ManyToOne to SecondaryUnit)

## Person entity

- `Person.unitOwnerships` references `InventoryUnitOwnership`

This allows tracking which persons own units in the internal inventory.

## Reference data service pattern

All property data access flows through the `ReferenceDataService`, which connects to the external Neon database:

```typescript
@Injectable()
export class ReferenceDataService {
  constructor(
    @InjectEntityManager('reference')
    private readonly em: EntityManager,
  ) {}
}
```

<Tip>
The `MapService` provides shared 2GIS geocoding utilities used across both the developer portal and reference data controllers for location enrichment.
</Tip>
