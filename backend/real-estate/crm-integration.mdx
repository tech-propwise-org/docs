The real estate module is structured as **six active specialized modules** spanning the global and org-private real-estate surfaces. The cross-org `shared-marketplace` code is retained in the repository but is not imported by `RealEstateModule`, so its controllers/providers are not mounted by the app.

The external **Propwise Labs API** stores and serves all property data — both structural (DLD registry) and enrichment (images, payment plans, documents, amenities, pricing, geocoding where exposed). The CRM database stores only CRM-specific data (developer portal auth, ownership links, property interests, marketplace listings, and org inventory context).

<Note>
**Architecture:** All modules consume property data exclusively through `PropwiseLabsService`. The CRM no longer opens a second MikroORM connection or maps Propwise Labs entities locally. The reference database configuration has been completely removed and replaced with Propwise Labs API configuration (`app.propwiseLabs.*`) including API credentials, timeout settings, and Bird Verify integration for developer portal contact verification.
</Note>

| Portal listings/configuration/sync                         | CRM DB            | Direct entity access through `real-estate/portal-syndication`                                          |

## External property ID pattern

`InventoryUnit` is the CRM boundary for org-owned secondary inventory. It stores plain integer IDs (`refProjectId`, `refBuildingId`, `refUnitId`, `refDeveloperId`, `refPropertyTypeId`, `refPropertySubTypeId`) plus a denormalized Labs snapshot, including the developer display name used by portal syndication and the canonical `propertyType` (`LayoutType` enum) used by portal syndication adapters. Labs raw property type/sub-type names are retained in `labsPropertyType` and `labsPropertySubType` for traceability. Lead interests can use inventory units as candidate matches while retaining search criteria. Deal property rows are concrete-only: they link either an org `InventoryUnit`, a direct Propwise Labs off-plan unit via `refProjectId` / `refBuildingId` / `refUnitId`, or a pending unit-access request before the inventory link is approved.

### Authentication flow

```typescript
@Injectable()
export class PropwiseLabsService {
  private accessToken?: string;
  private tokenExpiresAt = 0;

  private async getValidAccessToken(): Promise<string> {
    if (this.accessToken && Date.now() < this.tokenExpiresAt - 60000) {
      return this.accessToken;
    }
    
    const authResponse = await this.authenticate();
    this.accessToken = authResponse.access_token;
    this.tokenExpiresAt = Date.now() + (authResponse.expires_in * 1000);
    
    return this.accessToken;
  }

  async getProject(id: number): Promise<Project | null> {
    const token = await this.getValidAccessToken();
    const response = await this.httpService.get(`/projects/${id}`, {
      headers: { Authorization: `Bearer ${token}` },
      params: { 
        include: 'developer,area,gallery,payment_plans,buildings,buildings.units'
      }
    }).toPromise();
    
    return response.data;
  }
}
```

### Matching engine

The property matching engine provides automated property recommendations:

```typescript
@Injectable()
export class PropertyMatchingService {
  async findMatchingProperties(interest: LeadPropertyInterest): Promise<PropertyMatch[]> {
    const matches: PropertyMatch[] = [];

    // Match against Propwise Labs projects/units
    if (interest.budgetMin || interest.budgetMax) {
      const refMatches = await this.propwiseLabsService.searchProjects({
        priceMin: interest.budgetMin,
        priceMax: interest.budgetMax,
        interestType: interest.interestType,
      });
      
      matches.push(...refMatches.map(project => ({
        type: 'propwise_labs_project',
        refProjectId: project.id,
        matchScore: this.calculateMatchScore(interest, project),
      })));
    }

    // Match against inventory units
    const inventoryMatches = await this.findInventoryMatches(interest);
    matches.push(...inventoryMatches);

    // Match against shared marketplace listings
    const marketplaceMatches = await this.findMarketplaceMatches(interest);
    matches.push(...marketplaceMatches);

    return matches.sort((a, b) => b.matchScore - a.matchScore);
  }
}
```

### Person entity ownership tracking

The `Person` entity tracks real estate ownership through the inventory module:

```typescript
@Entity()
export class Person extends TenantScopedEntity {
  // ... other person fields

  @OneToMany(() => InventoryUnitOwnership, (ownership) => ownership.person)
  unitOwnerships = new Collection<InventoryUnitOwnership>(this);

  // Computed property for owned units
  get ownedUnits(): InventoryUnit[] {
    return this.unitOwnerships.getItems().map(ownership => ownership.unit);
  }
}
```

This relationship enables:
- Tracking which persons own units in the internal inventory
- Portfolio management for high-value clients
- Cross-referencing ownership with deal history
- Identifying potential sellers for marketplace listings
