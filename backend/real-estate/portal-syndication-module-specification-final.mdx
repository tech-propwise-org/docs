The portal syndication module manages the distribution of property listings to external real estate portals like Property Finder, Bayut, and Dubizzle. This module acts as a bridge between the internal property inventory system and external marketing platforms.

<Info>
**Module Location:** `src/modules/real-estate/portal-syndication/`  
**Status:** Development - Phase 1  
**Last Updated:** January 2024
</Info>

## Architecture Overview

The module follows a layered architecture with clear separation of concerns:

<CardGroup cols={2}>
  <Card title="Listings Layer" icon="list">
    Marketing layer that wraps inventory units with portal-specific metadata
  </Card>
  <Card title="Portal Configuration" icon="gear">
    Per-organization portal credentials and settings management
  </Card>
  <Card title="Sync Tracking" icon="sync">
    Per-listing, per-portal synchronization state management
  </Card>
  <Card title="Portal Adapters" icon="plug">
    Portal-specific integration logic (future implementation)
  </Card>
</CardGroup>

### Key Design Decisions

<AccordionGroup>
  <Accordion title="Separation of Concerns">
    Listings are separate from inventory units to allow marketing-specific data without affecting the core inventory system.
  </Accordion>
  <Accordion title="Per-Portal Tracking">
    Each listing maintains independent sync state per portal, allowing granular control and error tracking.
  </Accordion>
  <Accordion title="Encrypted Credentials">
    Portal API keys are stored encrypted using AES-256-GCM for enhanced security.
  </Accordion>
  <Accordion title="Mapping Tables">
    Cached lookups for portal-specific identifiers (agents, locations) improve performance and reduce API calls.
  </Accordion>
</AccordionGroup>

Represents a property unit prepared for marketing across portals.

```typescript
interface Listing {
  id: string;
  title: string;
  description: string;
  price: number;
  location: string;

  images: string[];
```

### Portal Configuration Entity

Stores per-organization portal credentials and settings.

```typescript
interface PortalConfiguration {
  id: string;
  portal: Portal;
  isEnabled: boolean;
  apiKey: string; // Encrypted
  apiSecret?: string; // Encrypted
  organizationId?: string; // Portal-specific org ID
  portalCredits?: number;
  lastCreditsCheckAt?: Date;
  organization: string; // FK to Organization
  createdAt: Date;
  updatedAt: Date;

```

<Warning>
API credentials are automatically encrypted before storage using AES-256-GCM encryption for security compliance.
</Warning>

## Business Rules

### Listing Validation Rules

<Check>**Required Fields**: Each listing must have title, description, price, location, and at least one image</Check>

<Check>**Inventory Association**: Each listing must be linked to a valid inventory unit</Check>

<Check>**Active Status**: Only active listings can be synced to external portals</Check>

<Check>**Price Validation**: Price must be a positive number greater than zero</Check>

### Portal Configuration Rules

<Note>
- Each organization can have one configuration per portal
- Portal must be enabled to sync listings  
- Credit balance is checked before Property Finder submissions
- API credentials are automatically encrypted before storage
</Note>

### Synchronization Rules

1. **One Sync Record Per Portal**: Each listing can have one sync record per portal
2. **Status Transitions**: Sync status follows portal-specific workflows
3. **Error Tracking**: Failed syncs store detailed error messages for debugging
4. **Cost Tracking**: Sync costs are recorded for billing and reporting purposes

<Warning>
All API endpoints are protected by AuthGuard and AccessGuard middleware. Row-level security is enforced through organization filtering.

#### Encryption Service Integration

Portal API credentials utilize the shared `EncryptionService`:
- **Algorithm**: AES-256-GCM
- **Key Management**: Environment-based key rotation
- **Decryption**: On-demand for API calls only

## Error Handling Strategy

### Structured Error Responses

All API endpoints return consistent error structures:

```typescript
{
  "statusCode": 400,
  "message": "Validation failed",
  "error": "Bad Request",
  "details": [
    {
      "field": "price",
      "message": "Price must be a positive number"
    }
  ]
}
```

### Portal Sync Error Tracking

<Tip>
Sync failures are automatically logged in the `ListingPortalSync` entity with detailed error messages, enabling effective debugging and monitoring.
</Tip>

### Graceful Degradation

- Portal connectivity issues don't affect core listing management
- Failed syncs are queued for retry with exponential backoff
- Portal-specific errors are isolated from other integrations
