## Overview

The `portal-syndication` module manages the syndication of property listings to external real estate portals like Property Finder, Bayut, and Dubizzle. This module acts as a bridge between the internal property inventory system and external marketing platforms, handling authentication, data transformation, and synchronization status tracking.

<Info>
**Module Path:** `src/modules/real-estate/portal-syndication/`

The module is part of `RealEstateModule` and is imported from `src/modules/real-estate/real-estate.module.ts`.
</Info>

<Note>
**Status:** Development - Phase 1 | **Last Updated:** January 2024
</Note>

## Architecture

The module follows a layered architecture with clear separation between different concerns:

  <Card title="Sync Tracking" icon="arrows-rotate">

### Design Decisions

<AccordionGroup>
  <Accordion title="Separation of Concerns">
    Listings are separate from inventory units to allow marketing-specific data without cluttering the core inventory model. This enables flexible marketing strategies across different portals.
  </Accordion>
  
  <Accordion title="Per-Portal Tracking">
    Each listing maintains independent sync state per portal, allowing granular control over which portals receive which listings and tracking individual portal performance.
  </Accordion>
  
  <Accordion title="Encrypted Credentials">
    Portal API keys are stored encrypted using AES-256-GCM encryption to protect sensitive third-party credentials at rest.
  </Accordion>
  
  <Accordion title="Mapping Tables">
    Cached lookups for portal-specific identifiers (agents, locations) reduce API calls and improve performance when syncing listings.
  </Accordion>
</AccordionGroup>

## Entities

### Listing

Represents a property unit prepared for marketing across portals.

```typescript
interface Listing {
  id: string;
  title: string;
  description: string;
  price: number;
  location: string;
  bedrooms: number;
  bathrooms: number;
  size: number;
  amenities: string[];
  images: string[];
  furnishedStatus: FurnishedStatus;
  rentalPeriod?: RentalPeriod;
  finishingType: FinishingType;
  status: ListingStatus;
  isActive: boolean;
  inventoryUnit: string; // FK to inventory
  createdBy: string; // FK to User
  organization: string; // FK to Organization
  createdAt: Date;
  updatedAt: Date;
}
```

<Tip>
Listings wrap inventory units with marketing-specific metadata, keeping the core inventory system focused on property management while enabling flexible marketing strategies.
</Tip>

### PortalConfiguration

Stores per-organization portal credentials and settings.

```typescript
interface PortalConfiguration {
  id: string;
  portal: Portal;
  isEnabled: boolean;
  apiKey: string; // Encrypted
  apiSecret?: string; // Encrypted
  organizationId?: string; // Portal-specific org ID
  portalCredits?: number;
  lastCreditsCheckAt?: Date;
  organization: string; // FK to Organization
  createdAt: Date;
  updatedAt: Date;
}
```

<Warning>
API credentials (`apiKey` and `apiSecret`) are automatically encrypted before storage and decrypted on retrieval using AES-256-GCM encryption.
</Warning>

### ListingPortalSync

Tracks sync state for each listing on each portal.

```typescript
interface ListingPortalSync {
  id: string;
  portal: Portal;
  isEnabled: boolean;
  status: ListingSyncStatus;
  portalListingId?: string;
  lastSyncedAt?: Date;
  lastError?: string;
  lastPublishCost?: {
    creditCost: number;
    checkedAt: Date;
  };
  listing: string; // FK to Listing
  organization: string; // FK to Organization
  createdAt: Date;
  updatedAt: Date;
}
```

### PfAgentMapping

Maps CRM users to Property Finder agent profiles.

```typescript
interface PfAgentMapping {
  id: string;
  pfAgentId: string;
  pfPublicProfileId: string;
  lastSyncedAt?: Date;
  user: string; // FK to User
  organization: string; // FK to Organization
}
```

### PfLocationMapping

Caches Property Finder location ID lookups.

```typescript
interface PfLocationMapping {
  id: string;
  locationName: string;
  pfLocationId: string;
  lastSyncedAt?: Date;
  organization: string; // FK to Organization
}
```

<Info>
Mapping tables reduce external API calls by caching portal-specific identifiers, improving sync performance and reducing costs.
</Info>

<CodeGroup>


## Business Rules

### Listing Rules

<Check>**Inventory Unit Association** - Each listing must be associated with a valid inventory unit</Check>

<Check>**Active Status Requirement** - Only active listings can be synced to portals</Check>

<Check>**Positive Pricing** - Price must be a positive number greater than zero</Check>

<Check>**Image Requirement** - At least one image is required for portal syndication</Check>

### Portal Configuration Rules

<AccordionGroup>
  <Accordion title="One Configuration Per Portal">
    Each organization can have one configuration per portal. Attempting to create a duplicate configuration will result in an error.
  </Accordion>
  
  <Accordion title="Credential Encryption">
    API credentials are automatically encrypted before storage using AES-256-GCM encryption and decrypted on retrieval.
  </Accordion>
  
  <Accordion title="Portal Enable Requirement">
    Portal must be enabled (`isEnabled: true`) to sync listings. Disabled portals will skip sync operations.
  </Accordion>
  
  <Accordion title="Credit Balance Check">
    For Property Finder, credit balance is checked before submissions to prevent failed publishes due to insufficient credits.
  </Accordion>
</AccordionGroup>

### Sync Rules

<Steps>
  <Step title="One sync record per portal">
    Each listing can have one sync record per portal, preventing duplicate submissions
  </Step>
  <Step title="Status transitions">
    Sync status transitions follow portal-specific workflows (e.g., PENDING → PUBLISHED → ACTIVE)
  </Step>
  <Step title="Error tracking">
    Failed syncs store error messages in `lastError` field for debugging and retry logic
  </Step>
  <Step title="Cost tracking">
    Sync costs are tracked in `lastPublishCost` for billing and analytics purposes
  </Step>
</Steps>

### Data Transfer Objects

<AccordionGroup>
  <Accordion title="Validation Strategy">
    Comprehensive validation using `class-validator` decorators ensures data integrity before database operations:
    
    - Required fields enforced with `@IsNotEmpty()`
    - Type validation with `@IsString()`, `@IsNumber()`, etc.
    - Format validation with `@IsEmail()`, `@IsUrl()`, etc.
    - Custom validators for business rules
  </Accordion>
  
  <Accordion title="DTO Separation">
    Separate DTOs for different operations:
    
    - **Create DTOs:** Validation for new record creation
    - **Update DTOs:** Partial validation with `@IsOptional()`
    - **Response DTOs:** Serialization and data transformation
    - **Query DTOs:** Pagination and filtering parameters
  </Accordion>
  
  <Accordion title="Nested Validation">
    Complex objects support nested validation:
    
    ```typescript
    class CreateListingDto {
      @ValidateNested()
      @Type(() => ListingImageDto)
      images: ListingImageDto[];
    }
    ```
  </Accordion>
</AccordionGroup>

### Error Handling

<Tabs>
  <Tab title="Structured Responses">
    All errors follow consistent structure:
    ```typescript
    {
      "statusCode": 400,
      "message": "Validation failed",
      "errors": [
        {
          "field": "price",
          "message": "Price must be a positive number"
        }
      ]
    }
    ```
  </Tab>
  
  <Tab title="Sync Failure Tracking">
    Sync failures are tracked in database for retry logic:
    ```typescript
    {
      "status": "FAILED",
      "lastError": "Portal API returned 429: Rate limit exceeded",
      "lastSyncedAt": "2024-01-15T14:00:00Z"
    }
    ```
  </Tab>
  
  <Tab title="Graceful Degradation">
    Portal connectivity issues don't block core operations:
    - Sync operations are queued for retry
    - Users can continue managing listings
    - Status clearly indicates sync issues
  </Tab>
</Tabs>
