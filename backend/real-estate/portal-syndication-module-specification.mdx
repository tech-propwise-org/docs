## Overview

The `portal-syndication` module manages the syndication of property listings to external real estate portals like Property Finder, Bayut, and Dubizzle. This module acts as a bridge between the internal property inventory system and external marketing platforms, handling authentication, data transformation, and synchronization status tracking.

<Info>
**Module path:** `src/modules/real-estate/portal-syndication/`

The module is part of `RealEstateModule` and is imported from `src/modules/real-estate/real-estate.module.ts`.
</Info>

**Status:** Development - Phase 1  
**Last Updated:** January 2024

## Architecture

The module follows a layered architecture with clear separation between different concerns:

<CardGroup cols={2}>
  <Card title="Listings" icon="list">
    Marketing layer that wraps inventory units with portal-specific metadata
  </Card>
  <Card title="Portal Configuration" icon="gear">
    Per-organization portal credentials and settings management
  </Card>
  <Card title="Sync Tracking" icon="sync">
    Per-listing, per-portal synchronization state management
  </Card>
  <Card title="Adapters" icon="plug">
    Portal-specific integration logic (future implementation)
  </Card>
</CardGroup>

### Design decisions

<AccordionGroup>
  <Accordion title="Separation of concerns">
    Listings are separate from inventory units to allow marketing-specific data without polluting the core inventory model.
  </Accordion>
  
  <Accordion title="Per-portal tracking">
    Each listing maintains independent sync state per portal, enabling granular control and error handling.
  </Accordion>
  
  <Accordion title="Encrypted credentials">
    Portal API keys stored encrypted using AES-256-GCM for security compliance.
  </Accordion>
  
  <Accordion title="Mapping tables">
    Cached lookups for portal-specific identifiers (agents, locations) to optimize performance.
  </Accordion>
</AccordionGroup>

## Entities

### Listing

Represents a property unit prepared for marketing across portals.

```typescript
interface Listing {
  id: string;
  title: string;
  description: string;
  price: number;
  location: string;
  bedrooms: number;
  bathrooms: number;
  size: number;
  amenities: string[];
  images: string[];
  furnishedStatus: FurnishedStatus;
  rentalPeriod?: RentalPeriod;
  finishingType: FinishingType;
  status: ListingStatus;
  isActive: boolean;
  inventoryUnit: string; // FK to inventory
  createdBy: string; // FK to User
  organization: string; // FK to Organization
  createdAt: Date;
  updatedAt: Date;
}
```

### PortalConfiguration

Stores per-organization portal credentials and settings.

```typescript
interface PortalConfiguration {
  id: string;
  portal: Portal;
  isEnabled: boolean;
  apiKey: string; // Encrypted
  apiSecret?: string; // Encrypted
  organizationId?: string; // Portal-specific org ID
  portalCredits?: number;
  lastCreditsCheckAt?: Date;
  organization: string; // FK to Organization
  createdAt: Date;
  updatedAt: Date;
}
```

### ListingPortalSync

Tracks sync state for each listing on each portal.

```typescript
interface ListingPortalSync {
  id: string;
  portal: Portal;
  isEnabled: boolean;
  status: ListingSyncStatus;
  portalListingId?: string;
  lastSyncedAt?: Date;
  lastError?: string;
  lastPublishCost?: {
    creditCost: number;
    checkedAt: Date;
  };
  listing: string; // FK to Listing
  organization: string; // FK to Organization
  createdAt: Date;
  updatedAt: Date;
}
```

### PfAgentMapping

Maps CRM users to Property Finder agent profiles.

```typescript
interface PfAgentMapping {
```

```typescript
interface PfLocationMapping {
```

## API Endpoints

### Listing controller

<Tabs>
  <Tab title="Query listings">
    **GET** `/listings`
    
    Query listings with filtering and pagination.
    
    **Query parameters:**
    - `page?: number`
    - `limit?: number` 
    - `status?: ListingStatus`
    - `search?: string`
    
    **Response:**
    ```typescript

      meta: {

        currentPage: number;
        totalPages: number;
      };
    }
    ```
  </Tab>
  
  <Tab title="Create listing">
    **POST** `/listings`
    
    Create a new listing.
    
    **Request body:**
    ```typescript
    {
      title: string;
      description: string;
      price: number;
      location: string;
      bedrooms: number;
      bathrooms: number;
      size: number;

      images: string[];
    ```

## Business Rules

### Listing rules

<Check>Each listing must be associated with a valid inventory unit</Check>
<Check>Only active listings can be synced to portals</Check>
<Check>Price must be positive</Check>
<Check>At least one image is required for portal syndication</Check>

### Portal configuration rules

<Warning>
Each organization can have only one configuration per portal to prevent conflicts.
</Warning>

- API credentials are encrypted before storage
- Portal must be enabled to sync listings
- Credit balance checked before Property Finder submissions

### Sync rules

<Note>
Each listing can have one sync record per portal, enabling independent status tracking across different platforms.
</Note>

- Sync status transitions follow portal-specific workflows
- Failed syncs store error messages for debugging
- Sync costs are tracked for billing purposes

### Security

<Warning>
All endpoints are protected by AuthGuard and AccessGuard to ensure proper authentication and authorization.
</Warning>

- Row-level security via organization filtering
- Sensitive credentials encrypted at rest
- API documentation with bearer token authentication
- Rate limiting on external portal API calls

### Error handling

<Steps>
  <Step title="Structured responses">
    All errors return consistent JSON structure with error codes and messages.
  </Step>
  
  <Step title="Sync failure tracking">
    Failed synchronization attempts are logged in the database with detailed error information.
  </Step>
  
  <Step title="Graceful degradation">
    Portal connectivity issues don't affect core listing functionality.
  </Step>
  
  <Step title="Retry mechanisms">
    Failed syncs are automatically retried with exponential backoff.
  </Step>
</Steps>
