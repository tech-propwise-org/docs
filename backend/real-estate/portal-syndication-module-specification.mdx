The Portal Syndication module manages the syndication of property listings to external real estate portals like Property Finder, Bayut, and Dubizzle. This module acts as a bridge between the internal property inventory system and external marketing platforms.

<Note>
**Module Path:** `src/modules/real-estate/portal-syndication/`

This module is part of `RealEstateModule` and is imported from `src/modules/real-estate/real-estate.module.ts`
</Note>

## Architecture

The module follows a layered architecture with clear separation between components:

<CardGroup cols={2}>
  <Card title="Listings" icon="list">
    Marketing layer that wraps inventory units with portal-specific metadata
  </Card>
  <Card title="Portal Configuration" icon="gear">
    Per-organization portal credentials and settings management
  </Card>
  <Card title="Sync Tracking" icon="sync">
    Per-listing, per-portal synchronization state management
  </Card>
  <Card title="Adapters" icon="plug">
    Portal-specific integration logic (future implementation)
  </Card>
</CardGroup>

### Design principles

<AccordionGroup>
  <Accordion title="Separation of concerns">
    Listings are separate from inventory units to allow marketing-specific data
  </Accordion>
  <Accordion title="Per-portal tracking">
    Each listing maintains independent sync state per portal
  </Accordion>
  <Accordion title="Encrypted credentials">
    Portal API keys stored encrypted using AES-256-GCM
  </Accordion>
  <Accordion title="Mapping tables">
    Cached lookups for portal-specific identifiers (agents, locations)
  </Accordion>
</AccordionGroup>

## Data models

### Listing entity

Represents a property unit prepared for marketing across portals.

```typescript
interface Listing {
  id: string;
  title: string;
  description: string;
  price: number;
  location: string;
  bedrooms: number;
  bathrooms: number;
  size: number;
  amenities: string[];
  images: string[];
  furnishedStatus: FurnishedStatus;
  rentalPeriod?: RentalPeriod;
  finishingType: FinishingType;
  status: ListingStatus;
  isActive: boolean;
  inventoryUnit: string; // FK to inventory
  createdBy: string; // FK to User
  organization: string; // FK to Organization
  createdAt: Date;
  updatedAt: Date;
}
```

### Portal configuration entity

Stores per-organization portal credentials and settings.

```typescript
interface PortalConfiguration {
  id: string;
  portal: Portal;
  isEnabled: boolean;
  apiKey: string; // Encrypted
  apiSecret?: string; // Encrypted
  organizationId?: string; // Portal-specific org ID
  portalCredits?: number;
  lastCreditsCheckAt?: Date;
  organization: string; // FK to Organization
  createdAt: Date;
  updatedAt: Date;
}
```

### Listing portal sync entity

Tracks sync state for each listing on each portal.

```typescript
interface ListingPortalSync {
  id: string;
  portal: Portal;
  isEnabled: boolean;
  status: ListingSyncStatus;
  portalListingId?: string;
  lastSyncedAt?: Date;
  lastError?: string;
  lastPublishCost?: {
    creditCost: number;
    checkedAt: Date;
  };
  listing: string; // FK to Listing
  organization: string; // FK to Organization
  createdAt: Date;
  updatedAt: Date;
}
```

## Business rules

### Listing validation

<Steps>
  <Step title="Inventory association">
    Each listing must be associated with a valid inventory unit
  </Step>
  <Step title="Active status">
    Only active listings can be synced to portals
  </Step>
  <Step title="Price validation">
    Price must be positive value
  </Step>
  <Step title="Image requirement">
    At least one image is required for portal syndication
  </Step>
</Steps>

### Portal configuration rules

<Warning>
**Security Requirements**
- Each organization can have one configuration per portal
- API credentials are encrypted before storage using AES-256-GCM
- Portal must be enabled to sync listings
</Warning>

<Info>
Credit balance is checked before Property Finder submissions to prevent failed syncs due to insufficient credits.
</Info>

### Synchronization rules

1. **Per-portal tracking**: Each listing can have one sync record per portal
2. **Status workflow**: Sync status transitions follow portal-specific workflows
3. **Error handling**: Failed syncs store error messages for debugging
4. **Cost tracking**: Sync costs are tracked for billing purposes

## Security and validation

### Authentication and authorization

<Check>
- All endpoints protected by AuthGuard and AccessGuard
- Row-level security via organization filtering
- Bearer token authentication for API access
</Check>

### Data validation

- Comprehensive validation using class-validator decorators
- Separate DTOs for create, update, and response operations
- Nested validation for complex objects

### Error handling

<Steps>
  <Step title="Structured responses">
    All errors return consistent JSON structure with error codes
  </Step>
  <Step title="Sync failure tracking">
    Failed syncs are logged in database with error details
  </Step>
  <Step title="Graceful degradation">
    Portal connectivity issues don't break core functionality
  </Step>
</Steps>
