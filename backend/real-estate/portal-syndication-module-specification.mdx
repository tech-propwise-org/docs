## Overview

The `portal-syndication` module manages the syndication of property listings to external real estate portals like Property Finder, Bayut, and Dubizzle. This module acts as a bridge between the internal property inventory system and external marketing platforms, handling authentication, data transformation, and synchronization status tracking.

<Info>
**Module Location**: `src/modules/real-estate/portal-syndication/`

The module is part of `RealEstateModule` and is imported from `src/modules/real-estate/real-estate.module.ts`.
</Info>

<Note>
**Development Status**: Phase 1 (January 2024)
</Note>

## Architecture

The module follows a layered architecture with clear separation between:

<CardGroup cols={2}>
  <Card title="Listings" icon="rectangle-list">
    Marketing layer that wraps inventory units with portal-specific metadata
  </Card>
  <Card title="Portal Configuration" icon="gear">
    Per-organization portal credentials and settings
  </Card>
  <Card title="Sync Tracking" icon="arrows-rotate">
    Per-listing, per-portal synchronization state management
  </Card>
  <Card title="Adapters" icon="plug">
    Portal-specific integration logic (future implementation)
  </Card>
</CardGroup>

### Design decisions

<AccordionGroup>
  <Accordion title="Separation of concerns">
    Listings are separate from inventory units to allow marketing-specific data without cluttering the core inventory system.
  </Accordion>
  
  <Accordion title="Per-portal tracking">
    Each listing maintains independent sync state per portal, allowing granular control and debugging of syndication issues.
  </Accordion>
  
  <Accordion title="Encrypted credentials">
    Portal API keys stored encrypted using AES-256-GCM to ensure security of sensitive third-party credentials.
  </Accordion>
  
  <Accordion title="Mapping tables">
    Cached lookups for portal-specific identifiers (agents, locations) to reduce API calls and improve performance.
  </Accordion>
</AccordionGroup>

## Entities

### Listing

Represents a property unit prepared for marketing across portals.

```typescript
interface Listing {
  id: string;
  title: string;
  description: string;
  price: number;
  location: string;
  bedrooms: number;
  bathrooms: number;
  size: number;
  amenities: string[];
  images: string[];
  furnishedStatus: FurnishedStatus;
  rentalPeriod?: RentalPeriod;
  finishingType: FinishingType;
  status: ListingStatus;
  isActive: boolean;
  inventoryUnit: string; // FK to inventory
  createdBy: string; // FK to User
  organization: string; // FK to Organization
  createdAt: Date;
  updatedAt: Date;
}
```

<Note>
Each listing is linked to an inventory unit but maintains its own marketing-focused fields like title, description, and amenities.
</Note>

### PortalConfiguration

Stores per-organization portal credentials and settings.

```typescript
interface PortalConfiguration {
  id: string;
  portal: Portal;
  isEnabled: boolean;
  apiKey: string; // Encrypted
  apiSecret?: string; // Encrypted
  organizationId?: string; // Portal-specific org ID
  portalCredits?: number;
  lastCreditsCheckAt?: Date;
  organization: string; // FK to Organization
  createdAt: Date;
  updatedAt: Date;
}
```

<Warning>
The `apiKey` and `apiSecret` fields are encrypted before storage using AES-256-GCM encryption. Never store credentials in plain text.
</Warning>

### ListingPortalSync

Tracks sync state for each listing on each portal.

```typescript
interface ListingPortalSync {
  id: string;
  portal: Portal;
  isEnabled: boolean;
  status: ListingSyncStatus;
  portalListingId?: string;
  lastSyncedAt?: Date;
  lastError?: string;
  lastPublishCost?: {
    creditCost: number;
    checkedAt: Date;
  };
  listing: string; // FK to Listing
  organization: string; // FK to Organization
  createdAt: Date;
  updatedAt: Date;
}
```

<Info>
The `portalListingId` stores the external identifier assigned by the portal after successful publication, enabling updates and deletions.
</Info>

### PfAgentMapping

Maps CRM users to Property Finder agent profiles.

```typescript
interface PfAgentMapping {
  id: string;
  pfAgentId: string;
  pfPublicProfileId: string;
  lastSyncedAt?: Date;
  user: string; // FK to User
  organization: string; // FK to Organization
}
```

### PfLocationMapping

Caches Property Finder location ID lookups.

```typescript
interface PfLocationMapping {
  id: string;
  locationName: string;
  pfLocationId: string;
  lastSyncedAt?: Date;
  organization: string; // FK to Organization
}
```

<Tip>
Location mappings are cached to minimize API calls to Property Finder's location endpoints, improving performance and reducing rate limit concerns.
</Tip>

#### List listings

<CodeGroup>
```bash GET /listings
curl -X GET 'https://api.example.com/listings?page=1&limit=20&status=ACTIVE' \
  -H 'Authorization: Bearer {token}'
```

<CodeGroup>
```bash PATCH /portal-configurations/:portal/toggle
curl -X PATCH 'https://api.example.com/portal-configurations/PROPERTY_FINDER/toggle' \
  -H 'Authorization: Bearer {token}' \
  -H 'Content-Type: application/json' \
  -d '{
    "isEnabled": false
  }'
```

```typescript Request body
{
  isEnabled: boolean;
}
```
</CodeGroup>

<Tip>
Disabling a portal will prevent new syncs but won't affect existing listings on that portal. Re-enable to resume syndication.
</Tip>

## Business rules

### Listing rules

<Steps>
  <Step title="Inventory unit association">
    Each listing must be associated with a valid inventory unit that belongs to the organization
  </Step>
  
  <Step title="Active status requirement">
    Only active listings can be synced to portals. Draft or archived listings are excluded from syndication
  </Step>
  
  <Step title="Price validation">
    Price must be a positive number greater than zero
  </Step>
  
  <Step title="Image requirement">
    At least one image is required for portal syndication to meet portal marketplace standards
  </Step>
</Steps>

### Portal configuration rules

<Check>
**One configuration per portal**: Each organization can have exactly one configuration per portal (e.g., one Property Finder config, one Bayut config)
</Check>

<Check>
**Encrypted storage**: API credentials are automatically encrypted before storage using AES-256-GCM encryption
</Check>

<Check>
**Enable requirement**: Portal must be enabled (`isEnabled: true`) to sync listings
</Check>

<Check>
**Credit checks**: Credit balance is checked before Property Finder submissions to prevent failed publications
</Check>

### Sync rules

The following rules govern listing synchronization across portals:

| Rule | Description |
|------|-------------|
| **One sync per portal** | Each listing can have one sync record per portal |
| **Status transitions** | Sync status transitions follow portal-specific workflows |
| **Error tracking** | Failed syncs store error messages in `lastError` for debugging |
| **Cost tracking** | Sync costs are tracked in `lastPublishCost` for billing purposes |

<Info>
Sync status transitions typically follow: `PENDING` → `SYNCING` → `SYNCED` (or `FAILED` on errors)
</Info>

### Error handling

The module implements structured error handling:

```typescript
// Structured error response format
{
  statusCode: number;
  message: string;
  error?: string;
  details?: any;
}
```

<Info>
**Sync failure tracking**: Portal sync failures are captured in the `lastError` field of `ListingPortalSync` records for debugging and retry logic.
</Info>

<Tip>
**Graceful degradation**: Portal connectivity issues don't block listing creation. Syncs are queued and retried automatically.
</Tip>
