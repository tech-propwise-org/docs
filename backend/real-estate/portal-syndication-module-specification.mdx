## Overview

The `portal-syndication` module manages the syndication of property listings to external real estate portals like Property Finder, Bayut, and Dubizzle. This module acts as a bridge between the internal property inventory system and external marketing platforms, handling authentication, data transformation, and synchronization status tracking.

<Info>
**Module Path:** `src/modules/real-estate/portal-syndication/`

The module is part of `RealEstateModule` and is imported from `src/modules/real-estate/real-estate.module.ts`.
</Info>

<Note>
**Status:** Development - Phase 1  
**Last Updated:** January 2024
</Note>

## Architecture

The module follows a layered architecture with clear separation between different concerns:

<CardGroup cols={2}>
  <Card title="Listings Layer" icon="rectangle-list">
    Marketing layer that wraps inventory units with portal-specific metadata
  </Card>
  <Card title="Portal Configuration" icon="gear">
    Per-organization portal credentials and settings management
  </Card>
  <Card title="Sync Tracking" icon="arrows-rotate">
    Per-listing, per-portal synchronization state management
  </Card>
  <Card title="Adapters" icon="plug">
    Portal-specific integration logic (future implementation)
  </Card>
</CardGroup>

### Design Decisions

<AccordionGroup>
  <Accordion title="Separation of Concerns">
    Listings are separate from inventory units to allow marketing-specific data without polluting the core inventory model. This enables different marketing strategies for the same property unit.
  </Accordion>
  
  <Accordion title="Per-Portal Tracking">
    Each listing maintains independent sync state per portal, allowing granular control over where and how listings are distributed. This supports partial failures and portal-specific retry logic.
  </Accordion>
  
  <Accordion title="Encrypted Credentials">
    Portal API keys are stored encrypted using AES-256-GCM encryption to protect sensitive authentication data at rest. The shared `EncryptionService` handles all cryptographic operations.
  </Accordion>
  
  <Accordion title="Mapping Tables">
    Cached lookups for portal-specific identifiers (agents, locations) reduce API calls and improve performance. These mappings are synced periodically from external portals.
  </Accordion>
</AccordionGroup>

## Entities

### Listing

Represents a property unit prepared for marketing across portals.

```typescript
interface Listing {
  id: string;
  title: string;
  description: string;
  price: number;
  location: string;
  bedrooms: number;
  bathrooms: number;
  size: number;
  amenities: string[];
  images: string[];
  furnishedStatus: FurnishedStatus;
  rentalPeriod?: RentalPeriod;
  finishingType: FinishingType;
  status: ListingStatus;
  isActive: boolean;
  inventoryUnit: string; // FK to inventory
  createdBy: string; // FK to User
  organization: string; // FK to Organization
  createdAt: Date;
  updatedAt: Date;
}
```

<Check>
Each listing is linked to an inventory unit but maintains its own marketing data, allowing multiple marketing strategies for the same property.
</Check>

### PortalConfiguration

Stores per-organization portal credentials and settings.

```typescript
interface PortalConfiguration {
  id: string;
  portal: Portal;
  isEnabled: boolean;
  apiKey: string; // Encrypted
  apiSecret?: string; // Encrypted
  organizationId?: string; // Portal-specific org ID
  portalCredits?: number;
  lastCreditsCheckAt?: Date;
  organization: string; // FK to Organization
  createdAt: Date;
  updatedAt: Date;
}
```

<Warning>
API credentials (`apiKey` and `apiSecret`) are encrypted before storage using AES-256-GCM encryption. Never expose these values in API responses or logs.
</Warning>

### ListingPortalSync

Tracks sync state for each listing on each portal.

```typescript
interface ListingPortalSync {
  id: string;
  portal: Portal;
  isEnabled: boolean;
  status: ListingSyncStatus;
  portalListingId?: string;
  lastSyncedAt?: Date;
  lastError?: string;
  lastPublishCost?: {
    creditCost: number;
    checkedAt: Date;
  };
  listing: string; // FK to Listing
  organization: string; // FK to Organization
  createdAt: Date;
  updatedAt: Date;
}
```

### PfAgentMapping

Maps CRM users to Property Finder agent profiles.

```typescript
interface PfAgentMapping {
  id: string;
  pfAgentId: string;
  pfPublicProfileId: string;
  lastSyncedAt?: Date;
  user: string; // FK to User
  organization: string; // FK to Organization
}
```

### PfLocationMapping

Caches Property Finder location ID lookups.

```typescript
interface PfLocationMapping {
  id: string;
  locationName: string;
  pfLocationId: string;
  lastSyncedAt?: Date;
  organization: string; // FK to Organization
}
```

<Tip>
Location mappings are cached to reduce API calls to Property Finder. Sync these mappings periodically to ensure accuracy.
</Tip>

## Business Rules

### Listing Rules

<Steps>
  <Step title="Inventory Unit Association">
    Each listing must be associated with a valid inventory unit from the inventory module.
  </Step>
  
  <Step title="Active Status Requirement">
    Only active listings can be synced to portals. Draft or archived listings are excluded from syndication.
  </Step>
  
  <Step title="Price Validation">
    Price must be a positive number. Zero or negative prices are rejected.
  </Step>
  
  <Step title="Image Requirement">
    At least one image is required for portal syndication. Most portals require multiple high-quality images.
  </Step>
</Steps>

### Portal Configuration Rules

<AccordionGroup>
  <Accordion title="One Configuration Per Portal">
    Each organization can have only one configuration per portal. Attempting to create a duplicate will result in a conflict error.
  </Accordion>
  
  <Accordion title="Credential Encryption">
    API credentials are automatically encrypted before storage using AES-256-GCM encryption. Decryption only occurs during API calls to external portals.
  </Accordion>
  
  <Accordion title="Portal Enablement">
    Portal must be explicitly enabled (`isEnabled: true`) to sync listings. This allows temporarily pausing syndication without deleting credentials.
  </Accordion>
  
  <Accordion title="Credit Balance Check">
    For Property Finder, credit balance is checked before submissions to prevent failed publishes due to insufficient credits.
  </Accordion>
</AccordionGroup>

### Sync Rules

<Check>
**Sync State Management:**
- Each listing can have one sync record per portal
- Sync status transitions follow portal-specific workflows
- Failed syncs store error messages for debugging
- Sync costs are tracked for billing and reporting purposes
</Check>

### External Services

<Tabs>
  <Tab title="Property Finder API">
    Direct API integration for listing submission and management. Supports real-time updates, status tracking, and credit management.
    
    **Features:**
    - Listing submission and updates
    - Credit balance checking
    - Agent and location mapping
    - Status synchronization
  </Tab>
  
  <Tab title="Bayut Feed">
    XML/CSV feed generation for bulk listing syndication. Feed is generated on-demand and uploaded to Bayut's FTP server.
    
    **Features:**
    - XML feed generation
    - Bulk listing export
    - Scheduled feed updates
    - Image URL management
  </Tab>
  
  <Tab title="Dubizzle Feed">
    XML/CSV feed generation for bulk listing syndication. Similar to Bayut with Dubizzle-specific formatting requirements.
    
    **Features:**
    - XML feed generation
    - Bulk listing export
    - Scheduled feed updates
    - Category mapping
  </Tab>
</Tabs>

### Encryption Service

<Info>
Portal API credentials are encrypted using the shared `EncryptionService` with **AES-256-GCM encryption**. This provides:
- Confidentiality through symmetric encryption
- Integrity verification through authentication tags
- Protection against tampering and replay attacks
</Info>

### Security

<Steps>
  <Step title="Authentication">
    All endpoints protected by `AuthGuard` requiring valid JWT bearer tokens.
  </Step>
  
  <Step title="Authorization">
    `AccessGuard` enforces role-based access control and permissions.
  </Step>
  
  <Step title="Multi-tenancy">
    Row-level security via organization filtering ensures data isolation between tenants.
  </Step>
  
  <Step title="Credential Protection">
    Sensitive credentials encrypted at rest using AES-256-GCM encryption.
  </Step>
  
  <Step title="API Documentation">
    OpenAPI documentation includes bearer token authentication requirements.
  </Step>
</Steps>

### Error Handling

<CardGroup cols={2}>
  <Card title="Structured Responses" icon="code">
    All errors return consistent JSON structure with error codes, messages, and context.
  </Card>
  
  <Card title="Sync Failure Tracking" icon="database">
    Failed sync attempts store error messages in the database for debugging and retry logic.
  </Card>
  
  <Card title="Graceful Degradation" icon="circle-exclamation">

  <Card title="Audit Trail" icon="clock-rotate-left">
