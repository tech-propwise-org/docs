## Overview

The `portal-syndication` module manages the syndication of property listings to external real estate portals like Property Finder, Bayut, and Dubizzle. This module acts as a bridge between the internal property inventory system and external marketing platforms, handling authentication, data transformation, and synchronization status tracking.

<Info>
**Module path:** `src/modules/real-estate/portal-syndication/`  
The module is part of `RealEstateModule` and is imported from `src/modules/real-estate/real-estate.module.ts`.
</Info>

<Note>
**Status:** Development - Phase 1 | **Last Updated:** January 2024
</Note>

## Architecture

The module follows a layered architecture with clear separation between components:

<CardGroup cols={2}>
  <Card title="Listings" icon="house">

  </Card>
  <Card title="Portal Configuration" icon="gear">
    Per-organization portal credentials and settings
  </Card>
  <Card title="Sync Tracking" icon="arrows-rotate">
    Per-listing, per-portal synchronization state management
  </Card>
  <Card title="Adapters" icon="plug">
    Portal-specific integration logic (future implementation)
  </Card>

### Design decisions

<AccordionGroup>
  <Accordion title="Separation of concerns">
    Listings are separate from inventory units to allow marketing-specific data
  </Accordion>
  <Accordion title="Per-portal tracking">
    Each listing maintains independent sync state per portal
  </Accordion>
  <Accordion title="Encrypted credentials">
    Portal API keys stored encrypted using AES-256-GCM
  </Accordion>
  <Accordion title="Mapping tables">
    Cached lookups for portal-specific identifiers (agents, locations)
  </Accordion>
</AccordionGroup>

## Entities

### Listing

Represents a property unit prepared for marketing across portals.

```typescript
interface Listing {
  id: string;
  title: string;
  description: string;
  price: number;
  location: string;
  bedrooms: number;
  bathrooms: number;
  size: number;
  amenities: string[];
  images: string[];
  furnishedStatus: FurnishedStatus;
  rentalPeriod?: RentalPeriod;
  finishingType: FinishingType;
  status: ListingStatus;
  isActive: boolean;
  inventoryUnit: string; // FK to inventory
  createdBy: string; // FK to User
  organization: string; // FK to Organization
  createdAt: Date;
  updatedAt: Date;
}
```

### PortalConfiguration

Stores per-organization portal credentials and settings.

```typescript
interface PortalConfiguration {
  id: string;
  portal: Portal;
  isEnabled: boolean;
  apiKey: string; // Encrypted
  apiSecret?: string; // Encrypted
  organizationId?: string; // Portal-specific org ID
  portalCredits?: number;
  lastCreditsCheckAt?: Date;
  organization: string; // FK to Organization
  createdAt: Date;
  updatedAt: Date;
}
```

### ListingPortalSync

Tracks sync state for each listing on each portal.

```typescript
interface ListingPortalSync {
  id: string;
  portal: Portal;
  isEnabled: boolean;
  status: ListingSyncStatus;
  portalListingId?: string;
  lastSyncedAt?: Date;
  lastError?: string;
  lastPublishCost?: {
    creditCost: number;
    checkedAt: Date;
  };
  listing: string; // FK to Listing
  organization: string; // FK to Organization
  createdAt: Date;
  updatedAt: Date;
}
```

### PfAgentMapping

Maps CRM users to Property Finder agent profiles.

```typescript
interface PfAgentMapping {
  id: string;
  pfAgentId: string;
  pfPublicProfileId: string;
  lastSyncedAt?: Date;
  user: string; // FK to User
  organization: string; // FK to Organization
}
```

### PfLocationMapping

Caches Property Finder location ID lookups.

```typescript
interface PfLocationMapping {
  id: string;
  locationName: string;
  pfLocationId: string;
  lastSyncedAt?: Date;
  organization: string; // FK to Organization
}
```

## API Endpoints

### Listing Controller

<Tabs>
  <Tab title="Query listings">
    **GET** `/listings`

    Query listings with filtering and pagination.

    **Query Parameters:**
    - `page?: number`
    - `limit?: number`
    - `status?: ListingStatus`
    - `search?: string`

    **Response:**
    ```typescript
    {
      data: ListingDto[];
      meta: {
        totalItems: number;
        currentPage: number;
        totalPages: number;
      };
    }
    ```
  </Tab>

  <Tab title="Create listing">
    **POST** `/listings`

    Create a new listing.

    **Request Body:**
    ```typescript
    {
      title: string;
      description: string;
      price: number;
      location: string;
      bedrooms: number;
      bathrooms: number;
      size: number;
      amenities: string[];
      images: string[];
      furnishedStatus: FurnishedStatus;
      rentalPeriod?: RentalPeriod;
      finishingType: FinishingType;
      inventoryUnitId: string;
    }
    ```
  </Tab>

  <Tab title="Update listing">
    **PATCH** `/listings/:id`

    Update an existing listing.
  </Tab>

  <Tab title="Delete listing">
    **DELETE** `/listings/:id`

    Soft delete a listing (sets status to ARCHIVED).
  </Tab>

  <Tab title="Portal sync status">
    **GET** `/listings/:id/portal-sync`

    Get portal sync status for a listing.
  </Tab>
</Tabs>

### Portal Configuration Controller

<Steps>
  <Step title="Get configurations">
    **GET** `/portal-configurations`

    Get all portal configurations for the organization.
  </Step>

  <Step title="Create configuration">
    **POST** `/portal-configurations`

    Create a new portal configuration.

    ```typescript
    {
      portal: Portal;
      apiKey: string;
      apiSecret?: string;
      organizationId?: string;
    }
    ```
  </Step>

  <Step title="Toggle portal">
    **PATCH** `/portal-configurations/:portal/toggle`

    Enable/disable a portal.

    ```typescript
    {
      isEnabled: boolean;
    }
    ```
  </Step>
</Steps>

## Business Rules

### Listing Rules

<Check>Each listing must be associated with a valid inventory unit</Check>
<Check>Only active listings can be synced to portals</Check>
<Check>Price must be positive</Check>
<Check>At least one image is required for portal syndication</Check>

### Portal Configuration Rules

<Check>Each organization can have one configuration per portal</Check>
<Check>API credentials are encrypted before storage</Check>
<Check>Portal must be enabled to sync listings</Check>
<Check>Credit balance checked before Property Finder submissions</Check>

### Sync Rules

<Check>Each listing can have one sync record per portal</Check>
<Check>Sync status transitions follow portal-specific workflows</Check>
<Check>Failed syncs store error messages for debugging</Check>
<Check>Sync costs are tracked for billing purposes</Check>
