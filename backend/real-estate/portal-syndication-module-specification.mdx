## Overview

The `portal-syndication` module manages the syndication of property listings to external real estate portals like Property Finder, Bayut, and Dubizzle. This module acts as a bridge between the internal property inventory system and external marketing platforms, handling authentication, data transformation, and synchronization status tracking.

<Info>
**Module path:** `src/modules/real-estate/portal-syndication/`

The module is part of `RealEstateModule` and is imported from `src/modules/real-estate/real-estate.module.ts`.
</Info>

<Note>
**Status:** Development - Phase 1 (Last Updated: January 2024)
</Note>

## Architecture

The module follows a layered architecture with clear separation between components:

  <Card title="Listings" icon="list">

  </Card>
  <Card title="Portal Configuration" icon="gear">

  </Card>
  <Card title="Sync Tracking" icon="sync">
    Per-listing, per-portal synchronization state management
  </Card>
  <Card title="Adapters" icon="plug">
    Portal-specific integration logic (future implementation)
  </Card>

### Design Decisions

<AccordionGroup>
  <Accordion title="Separation of Concerns">
    Listings are separate from inventory units to allow marketing-specific data and portal customization without affecting core inventory management.
  </Accordion>
  
  <Accordion title="Per-Portal Tracking">
    Each listing maintains independent sync state per portal, enabling selective syndication and granular error tracking.
  </Accordion>
  
  <Accordion title="Encrypted Credentials">
    Portal API keys are stored encrypted using AES-256-GCM for security compliance and data protection.
  </Accordion>
  
  <Accordion title="Mapping Tables">
    Cached lookups for portal-specific identifiers (agents, locations) to improve performance and reduce API calls.
  </Accordion>
</AccordionGroup>

## Entities

### Listing

Represents a property unit prepared for marketing across portals.

```typescript
interface Listing {
  id: string;
  title: string;
  description: string;
  price: number;
  location: string;
  bedrooms: number;
  bathrooms: number;
  size: number;
  amenities: string[];
  images: string[];
  furnishedStatus: FurnishedStatus;
  rentalPeriod?: RentalPeriod;
  finishingType: FinishingType;
  status: ListingStatus;
  isActive: boolean;
  inventoryUnit: string; // FK to inventory
  createdBy: string; // FK to User
  organization: string; // FK to Organization
  createdAt: Date;
  updatedAt: Date;
}
```

### PortalConfiguration

Stores per-organization portal credentials and settings.

```typescript
interface PortalConfiguration {
  id: string;
  portal: Portal;
  isEnabled: boolean;
  apiKey: string; // Encrypted
  apiSecret?: string; // Encrypted
  organizationId?: string; // Portal-specific org ID
  portalCredits?: number;
  lastCreditsCheckAt?: Date;
  organization: string; // FK to Organization
  createdAt: Date;
  updatedAt: Date;
}
```

### ListingPortalSync

Tracks sync state for each listing on each portal.

```typescript
interface ListingPortalSync {
  id: string;
  portal: Portal;
  isEnabled: boolean;
  status: ListingSyncStatus;
  portalListingId?: string;
  lastSyncedAt?: Date;
  lastError?: string;
  lastPublishCost?: {
    creditCost: number;
    checkedAt: Date;
  };
  listing: string; // FK to Listing
  organization: string; // FK to Organization
  createdAt: Date;
  updatedAt: Date;
}
```

### PfAgentMapping

Maps CRM users to Property Finder agent profiles.

```typescript
interface PfAgentMapping {
  id: string;
  pfAgentId: string;
  pfPublicProfileId: string;
  lastSyncedAt?: Date;
  user: string; // FK to User
  organization: string; // FK to Organization
}
```

### PfLocationMapping

Caches Property Finder location ID lookups.

```typescript
interface PfLocationMapping {
  id: string;
  locationName: string;
  pfLocationId: string;
  lastSyncedAt?: Date;
  organization: string; // FK to Organization
}
```

## API Endpoints

### Listing Controller

<Tabs>
  <Tab title="GET /listings">
    Query listings with filtering and pagination.

    **Query Parameters:**
    - `page?: number`
    - `limit?: number`
    - `status?: ListingStatus`
    - `search?: string`

    **Response:**
    ```typescript
    {
      data: ListingDto[];
      meta: {
        totalItems: number;
        currentPage: number;
        totalPages: number;
      };
    }
    ```
  </Tab>

  <Tab title="POST /listings">
    Create a new listing.

    **Request Body:**
    ```typescript
    {
      title: string;
      description: string;
      price: number;
      location: string;
      bedrooms: number;
      bathrooms: number;
      size: number;
      amenities: string[];
      images: string[];
      furnishedStatus: FurnishedStatus;
      rentalPeriod?: RentalPeriod;
      finishingType: FinishingType;
      inventoryUnitId: string;
    }
    ```
  </Tab>

  <Tab title="PATCH /listings/:id">
    Update an existing listing with partial data.
  </Tab>

  <Tab title="DELETE /listings/:id">
    Soft delete a listing (sets status to ARCHIVED).
  </Tab>

  <Tab title="GET /listings/:id/portal-sync">
    Get portal sync status for a specific listing.
  </Tab>
</Tabs>

### Portal Configuration Controller

<Steps>
  <Step title="Get configurations">
    **GET /portal-configurations**
    
    Retrieve all portal configurations for the organization.
  </Step>
  
  <Step title="Create configuration">
    **POST /portal-configurations**
    
    Create a new portal configuration.

    ```typescript
    {
      portal: Portal;
      apiKey: string;
      apiSecret?: string;
      organizationId?: string;
    }
    ```
  </Step>
  
  <Step title="Toggle portal">
    **PATCH /portal-configurations/:portal/toggle**
    
    Enable/disable a specific portal.

    ```typescript
    {
      isEnabled: boolean;
    }
    ```
  </Step>
</Steps>

## Business Rules

### Listing Rules

<Check>Each listing must be associated with a valid inventory unit</Check>
<Check>Only active listings can be synced to portals</Check>
<Check>Price must be positive</Check>
<Check>At least one image is required for portal syndication</Check>

### Portal Configuration Rules

<Warning>
Each organization can have **only one configuration per portal**. Duplicate portal configurations will be rejected.
</Warning>

- API credentials are encrypted before storage using AES-256-GCM
- Portal must be enabled to sync listings
- Credit balance checked before Property Finder submissions

### Sync Rules

<Info>
**Sync State Management:**
- Each listing can have one sync record per portal
- Sync status transitions follow portal-specific workflows
- Failed syncs store error messages for debugging
- Sync costs are tracked for billing purposes
</Info>

### Data Transfer Objects

- **Comprehensive validation** using class-validator decorators

### Security

<Warning>
All endpoints are protected by **AuthGuard** and **AccessGuard** for comprehensive security.
</Warning>

- Row-level security via organization filtering
- Sensitive credentials encrypted at rest
- API documentation with bearer token authentication
- Input validation and sanitization

### Error Handling

<Steps>
  <Step title="Structured responses">
    All errors return consistent JSON structure with error codes and messages
  </Step>
  
  <Step title="Sync failure tracking">
    Failed sync attempts are logged in the database for debugging and retry logic
  </Step>
  
  <Step title="Graceful degradation">
    Portal connectivity issues don't affect core listing functionality
  </Step>
</Steps>

<Note>
The module implements comprehensive error tracking to ensure reliable syndication and easy troubleshooting of portal integration issues.
</Note>
