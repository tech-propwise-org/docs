<Note>
This document describes the **external API contract** defined by Bayut for fetching lead and view data via their Pull API. It covers endpoints, authorization, query parameters, and sample responses for both leads (`is_trulead=1`) and views (`is_trulead=0`).
</Note>

## Overview

The Bayut Pull API enables developers to retrieve lead data and view metrics from the Bayut platform. This comprehensive guide includes:

- **Endpoint URL** - The specific web address for API requests
- **Authorization Requirements** - Credentials needed to access the API
- **Query Parameters** - Parameters to filter and refine returned data
- **Sample Responses** - Examples showing data structure and format for different lead types

By providing this information, developers can effectively integrate Bayut lead data into their applications or systems.

## Endpoint

```
www.bayut.com/api-v7/stats/website-client-leads
```

This API allows clients to fetch lead data based on specific query parameters.

## Authorization

Each client is provided with a unique API key, which must be passed as a **bearer token** in the `Authorization` header.

```http
Authorization: Bearer <API_KEY>
```

<Warning>
Keep your API key secure and never expose it in client-side code or public repositories.
</Warning>

## Making requests

<Tabs>
  <Tab title="cURL">
    ```bash
    curl --location \
      'www.bayut.com/api-v7/stats/website-client-leads?timestamp=2025-03-10%2000%3A00%3A00&target=listing&type=email&is_trulead=1' \
      --header 'Authorization: Bearer <API_KEY>'
    ```
  </Tab>
  <Tab title="JavaScript">
    ```javascript
    const response = await fetch(
      'www.bayut.com/api-v7/stats/website-client-leads?timestamp=2025-03-10%2000%3A00%3A00&target=listing&type=email&is_trulead=1',
      {
        headers: {
          'Authorization': 'Bearer <API_KEY>'
        }
      }
    );
    const data = await response.json();
    ```
  </Tab>
  <Tab title="Python">
    ```python
    import requests

    url = "www.bayut.com/api-v7/stats/website-client-leads"
    params = {
        "timestamp": "2025-03-10 00:00:00",
        "target": "listing",
        "type": "email",
        "is_trulead": "1"
    }
    headers = {
        "Authorization": "Bearer <API_KEY>"
    }

    response = requests.get(url, params=params, headers=headers)
    data = response.json()
    ```
  </Tab>
</Tabs>

<Info>
The `timestamp` value must be URL-encoded: `2025-03-10 00:00:00` becomes `2025-03-10%2000%3A00%3A00` where `%20` represents space and `%3A` represents colon.
</Info>

## Query parameters

<ParamField query="type" type="string" required>
  The type of lead to retrieve.
  
  **Possible values:** `whatsapp`, `sms`, `phone`, `email`, `call_logs`, `story_leads`
</ParamField>

<ParamField query="timestamp" type="string" required>
  The date and time from which to fetch leads.
  
  **Format:** `YYYY-MM-DD HH:MM:SS`
  
  **Example:** `2025-03-10 00:00:00`
</ParamField>

<ParamField query="target" type="string" required>
  The entity for which leads are being fetched.
  
  **Possible values:** `listing`, `agent`, `agency`
</ParamField>

<ParamField query="is_trulead" type="string" required>
  Determines whether to return actual leads or view counts.
  
  **Possible values:**
  - `1` - Returns actual leads (see [Get leads](#get-leads))
  - `0` - Returns view counts (see [Get views](#get-views))
</ParamField>

## Get views

When `is_trulead=0`, the API returns view count metrics instead of actual leads. This is useful for tracking engagement and reach without collecting full inquirer details.

<Warning>
View data (`is_trulead=0`) returns aggregated metrics and does not include individual inquirer information like name, email, or phone numbers.
</Warning>

## Response structure reference

<AccordionGroup>
  <Accordion title="Lead response fields">
    **Common fields across all lead types:**
    
    - `lead_id` (string) - Unique identifier for the lead, format: `{type}|{uuid}`
    - `source` (string) - Always "bayut"
    - `date_time` (string) - Timestamp when lead was created, format: `YYYY-MM-DD HH:MM:SS`
    
    **Listing-specific fields:**
    
    - `listing_details` (object)
      - `listing_id` (number) - Internal listing identifier
      - `listing_reference` (string) - External listing reference code
      - `current_type` (string) - Property type (e.g., "Apartment", "Villa")
    
    **Agent-specific fields:**
    
    - `agent_details` (object)
      - `url` (string) - Agent profile URL
      - `name` (string) - Agent name
      - `email` (string) - Agent email address
    
    **Agency-specific fields:**
    
    - `agency_details` (object)
      - `url` (string) - Agency profile URL
      - `name` (string) - Agency name
    
    **Inquirer details:**
    
    - `inquirer_details` (object)
      - `name` (string) - Inquirer's name
      - `cell` (string) - Inquirer's phone number
      - `email` (string) - Inquirer's email (email leads only)
      - `message` (string) - Inquirer's message
  </Accordion>
  
  <Accordion title="WhatsApp delivery notifications">
    WhatsApp leads include a `delivery_notifications` array tracking message delivery status:
    
    ```json
    "delivery_notifications": [
      {
        "status": "sent|delivered|read",
        "created_at": "YYYY-MM-DD HH:MM:SS"
      }
    ]
    ```
    
    **Status progression:**
    1. `sent` - Message sent to WhatsApp servers
    2. `delivered` - Message delivered to recipient's device
    3. `read` - Message opened by recipient
  </Accordion>
  
  <Accordion title="Call log fields">
    **Call identification:**
    
    - `call_log_id` (number) - Unique call identifier
    - `listing_reference` (string) - Associated listing reference
    - `lead_id` (string) - Associated lead ID
    
    **Call participants:**
    
    - `caller_number` (string) - Phone number of the caller
    - `proxy_number` (string) - Bayut proxy number used
    - `receiver_numbers_list` (string) - List of potential receivers
    - `receiver_number` (string) - Actual receiver number
    
    **Call metrics:**
    
    - `call_status` (string) - Status: "connected", "missed", "failed"
    - `call_total_duration` (string) - Total call time (H:MM:SS)
    - `call_connected_duration` (string) - Connected time (HH:MM:SS)
    - `call_time` (string) - Call initiation timestamp
    - `call_pickup_time` (string) - Call answer timestamp
    - `call_recordingurl` (string) - URL to call recording
  </Accordion>
  
  <Accordion title="Story lead fields">
    Story leads include nested `story_details` with project information:
    
    ```json
    "story_details": {
      "story_id": number,
      "listing_details": { /* standard listing fields */ },
      "project_details": {
        "project_name_en": string,
        "project_name_ar": string|null,
        "developer_name_en": string,
        "developer_name_ar": string,
        "project_property_types_name_en": string,
        "project_property_types_name_ar": string,
        "project_image_path": string (URL),
        "project_lpv_url_en": string|null,
        "project_lpv_url_ar": string|null
      }
    }
    ```
  </Accordion>
</AccordionGroup>

## Best practices

<CardGroup cols={2}>
  <Card title="Polling frequency" icon="clock">
    Poll the API at reasonable intervals (e.g., every 5-15 minutes) to avoid rate limiting. Store the last successful `timestamp` to fetch only new data.
  </Card>
  
  <Card title="Error handling" icon="triangle-exclamation">
    Implement exponential backoff for failed requests. Handle 401 (unauthorized), 429 (rate limit), and 500 (server error) responses appropriately.
  </Card>
  
  <Card title="Data deduplication" icon="clone">
    Use `lead_id` as the unique identifier to prevent duplicate lead processing. Store processed lead IDs in your database.
  </Card>
  
  <Card title="Timestamp management" icon="calendar">
    Always URL-encode timestamp parameters. Start with a historical timestamp on initial sync, then use incremental timestamps for subsequent polls.
  </Card>
</CardGroup>
