## Overview

The `portal-syndication` module manages the syndication of property listings to external real estate portals like Property Finder, Bayut, and Dubizzle. This module acts as a bridge between the internal property inventory system and external marketing platforms, handling authentication, data transformation, and synchronization status tracking.

**Module path:** `src/modules/real-estate/portal-syndication/`

<Info>
The module is part of `RealEstateModule` and is imported from `src/modules/real-estate/real-estate.module.ts`.
</Info>

The module follows a layered architecture with clear separation between:

  <Card title="Sync Tracking" icon="arrows-rotate">

### Design Decisions

<AccordionGroup>
  <Accordion title="Separation of Concerns">
    Listings are separate from inventory units to allow marketing-specific data
  </Accordion>
  <Accordion title="Per-Portal Tracking">
    Each listing maintains independent sync state per portal
  </Accordion>
  <Accordion title="Encrypted Credentials">
    Portal API keys stored encrypted using AES-256-GCM
  </Accordion>
  <Accordion title="Mapping Tables">
    Cached lookups for portal-specific identifiers (agents, locations)
  </Accordion>
</AccordionGroup>

## Entities

### Listing

Represents a property unit prepared for marketing across portals.

```typescript
interface Listing {
  id: string;
  title: string;
  description: string;
  price: number;
  location: string;
  bedrooms: number;
  bathrooms: number;
  size: number;
  amenities: string[];
  images: string[];
  furnishedStatus: FurnishedStatus;
  rentalPeriod?: RentalPeriod;
  finishingType: FinishingType;
  status: ListingStatus;
  isActive: boolean;
  inventoryUnit: string; // FK to inventory
  createdBy: string; // FK to User
  organization: string; // FK to Organization
  createdAt: Date;
  updatedAt: Date;
}
```

### PortalConfiguration

Stores per-organization portal credentials and settings.

```typescript
interface PortalConfiguration {
  id: string;
  portal: Portal;
  isEnabled: boolean;
  apiKey: string; // Encrypted
  apiSecret?: string; // Encrypted
  organizationId?: string; // Portal-specific org ID
  portalCredits?: number;
  lastCreditsCheckAt?: Date;
  organization: string; // FK to Organization
  createdAt: Date;
  updatedAt: Date;
}
```

### ListingPortalSync

Tracks sync state for each listing on each portal.

```typescript
interface ListingPortalSync {
  id: string;
  portal: Portal;
  isEnabled: boolean;
  status: ListingSyncStatus;
  portalListingId?: string;
  lastSyncedAt?: Date;
  lastError?: string;
  lastPublishCost?: {
    creditCost: number;
    checkedAt: Date;
  };
  listing: string; // FK to Listing
  organization: string; // FK to Organization
  createdAt: Date;
  updatedAt: Date;
}
```

### PfAgentMapping

Maps CRM users to Property Finder agent profiles.

```typescript
interface PfAgentMapping {
  id: string;
  pfAgentId: string;
  pfPublicProfileId: string;
  lastSyncedAt?: Date;
  user: string; // FK to User
  organization: string; // FK to Organization
}
```

### PfLocationMapping

Caches Property Finder location ID lookups.

```typescript
interface PfLocationMapping {
  id: string;
  locationName: string;
  pfLocationId: string;
  lastSyncedAt?: Date;
  organization: string; // FK to Organization
}
```

## API Endpoints

### Listing Controller

#### GET /listings

Query listings with filtering and pagination.

<Tabs>
  <Tab title="Query Parameters">
    - `page?: number` - Page number for pagination
    - `limit?: number` - Number of items per page
    - `status?: ListingStatus` - Filter by listing status
    - `search?: string` - Search in title and description
  </Tab>
  <Tab title="Response">
    ```typescript
    {
      data: ListingDto[];
      meta: {
        totalItems: number;
        currentPage: number;
        totalPages: number;
      };
    }
    ```
  </Tab>
</Tabs>

#### POST /listings

Create a new listing.

```typescript
{
  title: string;
  description: string;
  price: number;
  location: string;
  bedrooms: number;
  bathrooms: number;
  size: number;
  amenities: string[];
  images: string[];
  furnishedStatus: FurnishedStatus;
  rentalPeriod?: RentalPeriod;
  finishingType: FinishingType;
  inventoryUnitId: string;
}
```

#### PATCH /listings/:id

Update an existing listing with partial data.

#### DELETE /listings/:id

<Warning>
Soft delete operation - sets status to ARCHIVED instead of removing the record.
</Warning>

#### GET /listings/:id/portal-sync

Get portal sync status for a specific listing across all configured portals.

### Portal Configuration Controller

#### GET /portal-configurations

Get all portal configurations for the organization.

#### POST /portal-configurations

Create a new portal configuration.

```typescript
{
  portal: Portal;
  apiKey: string;
  apiSecret?: string;
  organizationId?: string;
}
```

<Note>
API credentials are automatically encrypted before storage using AES-256-GCM encryption.
</Note>

#### PATCH /portal-configurations/:portal/toggle

Enable or disable a portal configuration.

```typescript
{
  isEnabled: boolean;
}
```

## Business Rules

### Listing Rules

<Steps>
  <Step title="Inventory Association">
    Each listing must be associated with a valid inventory unit
  </Step>
  <Step title="Active Status Required">
    Only active listings can be synced to portals
  </Step>
  <Step title="Price Validation">
    Price must be positive and greater than zero
  </Step>
  <Step title="Image Requirements">
    At least one image is required for portal syndication
  </Step>
</Steps>

### Portal Configuration Rules

<Check>Each organization can have one configuration per portal</Check>
<Check>API credentials are encrypted before storage</Check>

### Error Handling

<Tip>
The module implements graceful error handling with structured responses, sync failure tracking, and graceful degradation for portal connectivity issues.
</Tip>

- **Structured Responses**: Consistent error format across all endpoints
- **Sync Failure Tracking**: Database persistence of sync errors for analysis
- **Graceful Degradation**: Continued operation during portal connectivity issues
- **Retry Logic**: Automatic retry mechanisms for transient failures
