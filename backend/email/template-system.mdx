## Overview

PropWise transactional emails are authored as **React Email components** and uploaded to **Resend** as managed templates. At send-time, the backend calls Resend by template alias with variable values. If a template is accidentally deleted or hasn't been synced yet, the email falls back to in-app rendering (same React component, rendered server-side) so no email is ever silently dropped.

<Note>
The system is **not database-backed**: there is no active `email_template` table in the runtime schema. The source of truth is `src/emails/email-template.registry.tsx` plus the corresponding templates in Resend.
</Note>

---

## Architecture

```
React Email components        Email Template Registry         Resend
(src/emails/templates/)  ──►  (email-template.registry.tsx) ──► pnpm email-templates:sync ──► Resend dashboard
                                                                                               (non-engineers edit content here)
                               at send-time
EmailService.sendTemplateEmail()  ────────────────────────────► Resend API (template by alias)
                                                                      │
                                                         if not_found ▼
                                              descriptor.renderHtml(vars) → sendEmail(html)
```

## Bespoke template payload contracts

<Warning>
Each bespoke sender (`sendDealWonEmail`, `sendTransferEmail`, `sendCommissionPaymentEmail`, `sendEventInviteEmail`) reads specific keys from the stored notification `payload`. The keys below are what listeners must emit; the email channel maps them to template variables.
</Warning>

### Deal won template

| Template variable | Payload key(s) read | Notes |
| --- | --- | --- |
| `DEAL_NAME` | `dealTitle` → `dealName` | Listeners emit `dealTitle` |
| `DEAL_LINK` | `entityId` → `dealId` | Listeners emit `entityId` |
| `DEAL_VALUE` | `dealValue` | Not emitted by any listener yet — renders empty until `DealWonEvent.metadata` is extended with a deal amount field |

### Transfer request template

| Template variable | Payload key(s) read | Notes |
| --- | --- | --- |
| `ENTITY_NAME` | `entityTitle` → `entityName` | All transfer listeners emit `entityTitle` |
| `REQUESTER_NAME` | `requestedByName` → `approverName` → `rejecterName` → `cancelledByName` → `userName` | Coalesced across all five transfer sub-types |

### Commission payment template

| Template variable | Payload key(s) read | Notes |
| --- | --- | --- |
| `PAYMENT_STATUS_LABEL` / `_BG` / `_COLOR` | `newStatus` → type-derived fallback | Status-change listeners emit `newStatus`; `COMMISSION_PAYMENT_CREATED` has no status field, so `'CREATED'` is derived from `event.type` via `COMMISSION_STATUS_FROM_TYPE` |
| `PAYMENT_AMOUNT` | `amount` (number) → `paymentAmount` (string) | Listeners emit `amount` as a number; channel converts to string |

<Info>
The email channel automatically coalesces multiple fallback keys to ensure backward compatibility when payload structures evolve. Always check the source code in `email.channel.ts` for the exact precedence order.
</Info>

## Syncing templates to Resend

<Steps>
  <Step title="Run the sync script">
    The sync script uploads logo assets to R2 and creates/updates templates in Resend.

    ```bash
    # Development (staging CDN)
    pnpm email-templates:sync

    # Production
    pnpm email-templates:sync --prod

    # Force update existing templates
    pnpm email-templates:sync --force
    ```
  </Step>

  <Step title="Verify in Resend dashboard">
    Log in to [Resend](https://resend.com/emails) and confirm all templates appear with their correct aliases (e.g., `propwise-verify-email`, `propwise-deal-won`).
  </Step>

  <Step title="Test send-time fallback">
    Temporarily delete a template in Resend and trigger the corresponding email. The system should fall back to server-side rendering without dropping the email.

    <Tip>
    Check application logs for `[EmailService] Template not found, falling back to render` to confirm fallback behavior.
    </Tip>
  </Step>
</Steps>

## Adding a new template

<Steps>
  <Step title="Create the React Email component">
    Add a new `.tsx` file in `src/emails/templates/`:

    ```tsx src/emails/templates/my-new-template.tsx
    import { EmailLayout } from '../components/email-layout';
    import { Text, Button } from '@react-email/components';

    interface MyNewTemplateProps {
      userName: string;
      actionUrl: string;
    }

    export function MyNewTemplate({ userName, actionUrl }: MyNewTemplateProps) {
      return (
        <EmailLayout variant="app" preview={`Hello ${userName}!`}>
          <Text>Hi {userName},</Text>
          <Text>This is a new template example.</Text>
          <Button href={actionUrl}>Take Action</Button>
        </EmailLayout>
      );
    }
    ```
  </Step>

  <Step title="Register in the template registry">
    Add an entry to `src/emails/email-template.registry.tsx`:

    ```tsx src/emails/email-template.registry.tsx
    import { MyNewTemplate } from './templates/my-new-template';

    export const EMAIL_TEMPLATES = {
      // ... existing templates
      MY_NEW_TEMPLATE: {
        key: 'MY_NEW_TEMPLATE',
        alias: 'propwise-my-new-template',
        subject: 'Take Action Now',
        component: MyNewTemplate,
        variables: ['userName', 'actionUrl'],
        renderHtml: async (vars) => {
          return render(<MyNewTemplate {...vars} />);
        },
      },
    } as const;
    ```
  </Step>

  <Step title="Add contract test coverage">
    Ensure `src/emails/email-template.registry.spec.ts` validates your new template's variables match its component props.
  </Step>

  <Step title="Sync to Resend">
    ```bash
    pnpm email-templates:sync --force
    ```
  </Step>

  <Step title="Use in email service">
    Call `sendTemplateEmail` with your new template key:

    ```typescript
    await emailService.sendTemplateEmail({
      to: 'user@example.com',
      templateKey: 'MY_NEW_TEMPLATE',
      variables: {
        userName: 'Alice',
        actionUrl: 'https://propwise.com/action',
      },
    });
    ```
  </Step>
</Steps>

<Check>
The new template is now live with automatic fallback rendering if the Resend version is unavailable.
</Check>

## Fallback rendering behavior

<Tabs>
  <Tab title="Normal flow">
    1. `sendTemplateEmail()` is called with a template key
    2. Email service looks up the template alias from the registry
    3. Resend API is called with the alias and variables
    4. Email is sent using the managed Resend template
  </Tab>

  <Tab title="Fallback flow">
    1. `sendTemplateEmail()` is called with a template key
    2. Email service looks up the template alias from the registry
    3. Resend API returns `404 Not Found` (template deleted or not synced)
    4. Email service catches the error and calls `descriptor.renderHtml(vars)`
    5. Email is sent using the server-rendered HTML
    6. **No email is lost** — user receives content from the React component
  </Tab>
</Tabs>

<Warning>
While the fallback ensures no emails are dropped, templates should still be kept in sync with Resend. The fallback path bypasses Resend's analytics and A/B testing features.
</Warning>

## Brand variants

PropWise emails support two brand variants that control logo, accent color, and typography:

| Variant | Accent color | Logo | Used for |
| --- | --- | --- | --- |
| `app` | `#2A54CF` (blue) | PropWise CRM logo | Auth, invitations, notifications |
| `developer` | `#316337` (green) | Developer portal logo | Developer portal auth flows |

<CodeGroup>



## Deep link synchronization

<Warning>
**Critical:** `src/modules/notification/utils/entity-route.util.ts` (backend) must be kept in sync with `notification-utils.ts` (frontend). Both resolve entity types and payloads to absolute URLs.
</Warning>

The backend `resolveEntityUrl()` function generates deep links for email action buttons. Changes to entity routing on the frontend require corresponding backend updates.

**Example entity types:**
- `deal` → `/deals/:dealId`
- `contact` → `/contacts/:contactId`
- `commission_payment` → `/commissions/:commissionId`
- `event` → `/calendar/events/:eventId`

<Check>
CI tests in `entity-route.util.spec.ts` validate that all entity types resolve to properly formatted URLs.
</Check>
