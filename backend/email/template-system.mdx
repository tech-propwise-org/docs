## Overview

PropWise transactional emails are authored as **React Email components** and uploaded to **Resend** as managed templates. At send-time, the backend calls Resend by template alias with variable values. If a template is accidentally deleted or hasn't been synced yet, the email falls back to in-app rendering (same React component, rendered server-side) so no email is ever silently dropped.

<Note>
  The system is **not database-backed**: there is no active `email_template` table in the runtime schema. The source of truth is `src/emails/email-template.registry.tsx` plus the corresponding templates in Resend.
</Note>

## Bespoke template payload contracts

Each bespoke sender (`sendDealWonEmail`, `sendTransferEmail`, `sendCommissionPaymentEmail`, `sendEventInviteEmail`) reads specific keys from the stored notification `payload`.

<Warning>
  **The keys below are what listeners must emit; the email channel maps them to template variables.**
</Warning>

### DEAL_WON payload mapping

| Template variable | Payload key(s) read (left-to-right preference) | Notes |
| ----------------- | --------------------------------------------- | ----- |
| `DEAL_NAME` | `dealTitle` â†’ `dealName` | Listeners emit `dealTitle` |
| `DEAL_LINK` | `entityId` â†’ `dealId` | Listeners emit `entityId` |
| `DEAL_VALUE` | `dealValue` | Not emitted by any listener yet â€” renders empty until `DealWonEvent.metadata` is extended with a deal amount field |

### TRANSFER_REQUEST payload mapping

| Template variable | Payload key(s) read (left-to-right preference) | Notes |
| ----------------- | --------------------------------------------- | ----- |
| `ENTITY_NAME` | `entityTitle` â†’ `entityName` | All transfer listeners emit `entityTitle` |
| `REQUESTER_NAME` | `requestedByName` â†’ `approverName` â†’ `rejecterName` â†’ `cancelledByName` â†’ `userName` | Coalesced across all five transfer sub-types |

### COMMISSION_PAYMENT payload mapping

| Template variable | Payload key(s) read (left-to-right preference) | Notes |
| ----------------- | --------------------------------------------- | ----- |
| `PAYMENT_STATUS_LABEL/BG/COLOR` | `newStatus` â†’ type-derived fallback | Status-change listeners emit `newStatus`; `COMMISSION_PAYMENT_CREATED` has no status field, so `'CREATED'` is derived from `event.type` via `COMMISSION_STATUS_FROM_TYPE` |
| `PAYMENT_AMOUNT` | `amount` (number) â†’ `paymentAmount` (string) | Listeners emit `amount` as a number; channel converts to string |

## Syncing templates to Resend

<Steps>
  <Step title="Run the sync script">
    From the backend root:

    ```bash
    pnpm email-templates:sync
    ```

    **Default behavior:** create-only (will not overwrite existing templates in Resend)
  </Step>

  <Step title="Force upsert (optional)">
    To **update** existing templates:

    ```bash
    pnpm email-templates:sync --force
    ```
  </Step>

  <Step title="Target production CDN (optional)">
    By default, logos are uploaded to the **staging** R2 bucket. To target **production**:

    ```bash
    pnpm email-templates:sync --prod
    ```
  </Step>
</Steps>

<Info>
  The sync script uploads logo assets to R2 **and** creates/updates templates in Resend. Non-engineers can edit template content directly in the Resend dashboard after the initial sync.
</Info>

## Fallback rendering

If a template alias is not found in Resend (accidental deletion, sync lag), `EmailService.sendTemplateEmail()` automatically falls back to server-side rendering:

```typescript
descriptor.renderHtml(vars) â†’ sendEmail(html)
```

<Check>
  **No email is ever silently dropped.** The fallback ensures delivery continuity while templates are re-synced.
</Check>

## Entity deep-linking

The `resolveEntityUrl()` utility in `src/modules/notification/utils/entity-route.util.ts` maps entity types and payloads to absolute frontend deep links (e.g., deal detail pages, commission payment modals).

<Warning>
  **Critical:** This backend utility **must be kept in sync** with the frontend counterpart (`notification-utils.ts`). Route changes in either system require updates to both.
</Warning>

### Example entity URL resolution

```typescript
resolveEntityUrl('deal', { entityId: '123' }) 
// â†’ https://app.propwise.com/deals/123

resolveEntityUrl('commission_payment', { paymentId: '456' })
// â†’ https://app.propwise.com/commissions?paymentId=456
```
