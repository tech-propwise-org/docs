---
title: "Template System"
description: "React Email components with Resend integration for transactional emails"
---

PropWise transactional emails are authored as **React Email components** and uploaded to **Resend** as managed templates. At send-time the backend calls Resend by template alias with variable values. If a template is accidentally deleted or hasn't been synced yet, the email falls back to in-app rendering (same React component, rendered server-side) so no email is ever silently dropped.

<Note>
The system is **not database-backed**: there is no active `email_template` table in the runtime schema. The source of truth is `src/emails/email-template.registry.tsx` plus the corresponding templates in Resend.
</Note>

## Architecture

```
React Email components        Email Template Registry         Resend
(src/emails/templates/)  ──►  (email-template.registry.tsx) ──► pnpm email-templates:sync ──► Resend dashboard
                                                                                               (non-engineers edit content here)
                               at send-time
EmailService.sendTemplateEmail()  ────────────────────────────► Resend API (template by alias)
                                                                      │
                                                         if not_found ▼
                                              descriptor.renderHtml(vars) → sendEmail(html)
```

## Bespoke Template Payload Contracts

Each bespoke sender (`sendDealWonEmail`, `sendTransferEmail`, `sendCommissionPaymentEmail`, `sendEventInviteEmail`) reads specific keys from the stored notification `payload`. **The keys below are what listeners must emit; the email channel maps them to template variables.**

<Warning>
These payload key contracts must be maintained between notification listeners and the email channel to ensure proper variable substitution.
</Warning>

### Deal Won Template

| Template Variable | Payload Key(s) Read (left-to-right preference) | Notes |
| --- | --- | --- |
| `DEAL_NAME` | `dealTitle` → `dealName` | Listeners emit `dealTitle` |
| `DEAL_LINK` | `entityId` → `dealId` | Listeners emit `entityId` |
| `DEAL_VALUE` | `dealValue` | Not emitted by any listener yet — renders empty until `DealWonEvent.metadata` is extended with a deal amount field |

### Transfer Request Template

| Template Variable | Payload Key(s) Read (left-to-right preference) | Notes |
| --- | --- | --- |
| `ENTITY_NAME` | `entityTitle` → `entityName` | All transfer listeners emit `entityTitle` |
| `REQUESTER_NAME` | `requestedByName` → `approverName` → `rejecterName` → `cancelledByName` → `userName` | Coalesced across all five transfer sub-types |

### Commission Payment Template

| Template Variable | Payload Key(s) Read (left-to-right preference) | Notes |
| --- | --- | --- |
| `PAYMENT_STATUS_LABEL/BG/COLOR` | `newStatus` → type-derived fallback | Status-change listeners emit `newStatus`; `COMMISSION_PAYMENT_CREATED` has no status field, so `'CREATED'` is derived from `event.type` via `COMMISSION_STATUS_FROM_TYPE` |
| `PAYMENT_AMOUNT` | `amount` (number) → `paymentAmount` (string) | Listeners emit `amount` as a number; channel converts to string |

### Event Invite Template

| Template Variable | Payload Key(s) Read | Notes |
| --- | --- | --- |
| `EVENT_NAME` | `eventName` → `eventTitle` | Event listeners emit event details |
| `EVENT_DATE` | `eventDate` | Date formatting handled in template |
| `EVENT_TIME` | `eventTime` | Time formatting handled in template |
| `EVENT_LOCATION` | `eventLocation` | Optional location field |
| `INVITER_NAME` | `inviterName` → `userName` | Name of the user sending the invite |

## Template Sync Process

<Steps>
<Step title="Development workflow">
Create or modify React Email components in `src/emails/templates/`
</Step>

<Step title="Update registry">
Register templates in `src/emails/email-template.registry.tsx` with stable aliases
</Step>

<Step title="Sync to Resend">
Run the sync command to upload templates to Resend:

```bash
pnpm email-templates:sync          # Development environment
pnpm email-templates:sync --prod   # Production environment
pnpm email-templates:sync --force  # Force update existing templates
```
</Step>

<Step title="Content editing">
Non-engineers can edit template content directly in the Resend dashboard
</Step>
</Steps>

<Info>
The sync script uploads both email logos to R2 and templates to Resend. Use `--force` to update existing templates, and `--prod` to target the production CDN.
</Info>

## Fallback Rendering

If a template is not found in Resend (accidentally deleted or not yet synced), the system automatically falls back to server-side rendering of the same React component. This ensures no emails are ever silently dropped.

```typescript
// Fallback flow in EmailService
try {
  await resend.emails.send({ template: alias, variables });
} catch (error) {
  if (error.code === 'template_not_found') {
    const html = descriptor.renderHtml(variables);
    await this.sendEmail({ html, ...emailOptions });
  }
}
```
