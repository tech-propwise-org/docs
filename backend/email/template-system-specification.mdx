## Overview

PropWise transactional emails are authored as **React Email components** and uploaded to **Resend** as managed templates. At send-time the backend calls Resend by template alias with variable values. If a template is accidentally deleted or hasn't been synced yet, the email falls back to in-app rendering (same React component, rendered server-side) so no email is ever silently dropped.

<Note>
The system is **not database-backed**: there is no active `email_template` table in the runtime schema. The source of truth is `src/emails/email-template.registry.tsx` plus the corresponding templates in Resend.
</Note>

## Architecture

The email template system follows a clear flow from React components to Resend templates:

```
React Email components        Email Template Registry         Resend
(src/emails/templates/)  ──►  (email-template.registry.tsx) ──► pnpm email-templates:sync ──► Resend dashboard
                                                                                               (non-engineers edit content here)
                               at send-time
EmailService.sendTemplateEmail()  ────────────────────────────► Resend API (template by alias)
                                                                      │
                                                         if not_found ▼
                                              descriptor.renderHtml(vars) → sendEmail(html)
```

<Info>
If a template is not found in Resend, the system automatically falls back to server-side rendering of the same React component, ensuring no emails are silently dropped.
</Info>

## Bespoke Template Payload Contracts

Each bespoke template sender reads specific keys from the stored notification `payload`. The email channel maps these payload keys to template variables.

<Warning>
The keys listed below are what event listeners must emit. The email channel is responsible for mapping them to template variables.
</Warning>

### Deal Won Template

<CodeGroup>
```typescript Deal Won Variables
DEAL_NAME    // Maps from: dealTitle → dealName
DEAL_LINK    // Maps from: entityId → dealId  
DEAL_VALUE   // Maps from: dealValue (not yet emitted by listeners)
```
</CodeGroup>

<Note>
`DEAL_VALUE` is not emitted by any listener yet — renders empty until `DealWonEvent.metadata` is extended with a deal amount field.
</Note>

### Transfer Request Template

<CodeGroup>
```typescript Transfer Variables
ENTITY_NAME     // Maps from: entityTitle → entityName
REQUESTER_NAME  // Maps from: requestedByName → approverName → rejecterName → cancelledByName → userName
```
</CodeGroup>

<Info>
`REQUESTER_NAME` is coalesced across all five transfer sub-types using a fallback chain.
</Info>

### Commission Payment Template

<CodeGroup>
```typescript Commission Variables
PAYMENT_STATUS_LABEL/BG/COLOR  // Maps from: newStatus → type-derived fallback
PAYMENT_AMOUNT                 // Maps from: amount (number) → paymentAmount (string)
```
</CodeGroup>

<Tip>
For `COMMISSION_PAYMENT_CREATED` events that have no status field, `'CREATED'` is derived from `event.type` via `COMMISSION_STATUS_FROM_TYPE`.
</Tip>

### Event Invite Template

The event invite template handles three notification types:
- `event_invited`
- `event_invitee_rsvp_changed` 
- `event_reminder`

<CodeGroup>
```typescript Event Variables
EVENT_NAME      // Event title
EVENT_DATE      // Formatted date
EVENT_TIME      // Formatted time
EVENT_LOCATION  // Venue or address
INVITER_NAME    // Person who created/sent the invite
EVENT_LINK      // Deep link to event details
```
</CodeGroup>

## Template Synchronization

<Steps>
  <Step title="Sync Templates to Resend">
    Use the sync script to upload templates to Resend:
    ```bash
    pnpm email-templates:sync
    ```
  </Step>
  
  <Step title="Force Update Existing Templates">
    To update existing templates (upsert mode):
    ```bash
    pnpm email-templates:sync --force
    ```
  </Step>
  
  <Step title="Sync to Production">
    To target production CDN:
    ```bash
    pnpm email-templates:sync --prod
    ```
  </Step>
</Steps>

<Check>
The sync script handles both logo uploads to R2 and template uploads to Resend. By default, it only creates new templates to avoid overwriting manual edits made in the Resend dashboard.
</Check>
