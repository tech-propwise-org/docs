# Testing Specification v13

Architecture, conventions, and run guide for the PropWise CRM backend test suite.

<Info>
  This specification focuses on **RBAC (Role-Based Access Control) E2E testing** — verifying that every API endpoint enforces the correct permissions for every user role.
</Info>

## Overview

The test suite is built with **Jest + supertest + @nestjs/testing** and implements three key design decisions:

- **Singleton app context** — the NestJS app boots once and is shared across all test files
- **Matrix-driven tests** — a declarative permission matrix defines expected HTTP status codes per persona × endpoint
- **API-driven seeding** — test data is created through actual API endpoints so all business logic executes naturally

## Architecture

### Singleton Context Pattern

<Note>
  The app boots once for the entire test run, equivalent to NestJS `TestSuite` pattern but implemented manually since `TestSuite` requires NestJS 11+.
</Note>

The singleton context (`rbac-context.ts`) ensures optimal performance:

```typescript
Jest starts
  → lead-rbac.e2e-spec.ts calls getContext()
    → First call: boots NestJS app (~3-5s), logs in 5 personas
    → Returns { app, personas, teamId }
  → deal-rbac.e2e-spec.ts calls getContext()
    → Returns cached instance instantly (0ms)
  → Last file's afterAll() calls closeContext()
    → Reference count hits 0 → app shuts down
```

**Usage in test files:**

```typescript
import { getContext, closeContext, RbacTestContext } from '../../setup/rbac-context';

let ctx: RbacTestContext;
beforeAll(async () => {
  ctx = await getContext();
}, 60000);
afterAll(async () => {
  await closeContext();
});
```

The context replicates production configuration:
- `ValidationPipe` with whitelist, transform, and implicit conversion
- URI versioning (`/v1/...`)
- Cookie parser

### Matrix-Driven Test Generation

Instead of writing individual test cases, define expectations declaratively:

```typescript
{
  method: 'GET',
  path: '/v1/leads/:leadByBasicId',
  description: 'View lead detail',
  expectations: {
    OWNER: 200,
    ADMIN: 200,
    TEAM_LEADER: 403,
    AGENT: 403,
    BASIC: 200,
  },
}
```

<Tip>
  One matrix entry automatically generates 5 test cases (one per persona) with clear failure messages and actual response bodies.
</Tip>

The runner (`rbac-runner.ts`) processes matrices by:
1. Iterating every endpoint in the matrix
2. For each endpoint, testing all 5 personas
3. Resolving path placeholders (`:leadByBasicId` → actual UUID)
4. Making HTTP requests with persona tenant tokens
5. Asserting response status matches expectations

## RBAC Personas

<CardGroup cols={2}>
  <Card title="Ownership Personas" icon="crown">
    **OWNER** and **ADMIN** bypass resource-level checks via org permissions
  </Card>
  <Card title="Team Personas" icon="users">
    **TEAM_LEADER** and **AGENT** access via team stakeholder relationships
  </Card>
</CardGroup>

### Persona Definitions

| Persona | Email | Org Role | Team Role | Permissions |
|---------|-------|----------|-----------|-------------|
| `OWNER` | owner@propwise.com | Owner | Any | All permissions including `system.owner` |
| `ADMIN` | admin@propwise.com | Admin | Any | All non-sensitive permissions (`system.admin`) |
| `TEAM_LEADER` | teamleader@propwise.com | Basic | Team Leader | Team permissions (`team_crm.*`, `team_sales.*`) |
| `AGENT` | agent@propwise.com | Basic | Agent | No team permissions by default |
| `BASIC` | basic@propwise.com | Basic | None | Stakeholder access only |

### Permission Profiles

<AccordionGroup>
  <Accordion title="OWNER / ADMIN Profile">
    - Bypass resource-level checks via org permissions
    - Can access any entity in the organization
    - OWNER has additional `system.owner` capabilities
    - ADMIN has `system.admin` for most operations
  </Accordion>
  
  <Accordion title="TEAM_LEADER Profile">
    - No org-level permissions
    - Has `team_crm.manage` and `team_sales.manage` in assigned team
    - Can access entities where the team is a stakeholder
    - Demonstrates team-based access control
  </Accordion>
  
  <Accordion title="AGENT Profile">
    - No org or team permissions
    - Can only access entities with personal stakeholder relationships
    - Used to test access level differentiation (READ vs WRITE vs FULL)
    - Tests individual user access patterns
  </Accordion>
  
  <Accordion title="BASIC Profile">
    - Same as AGENT but not a team member
    - Access is purely stakeholder-based
    - Tests non-team member access scenarios
  </Accordion>
</AccordionGroup>

### Authentication Flow

<Steps>
  <Step title="Identity Login">
    `POST /v1/auth/login` → Returns identity token
  </Step>
  <Step title="Organization Selection">
    `POST /v1/auth/select-organization` → Returns tenant token (org-scoped JWT)
  </Step>
  <Step title="API Requests">
    All subsequent calls use tenant token in `Authorization: Bearer` header
  </Step>
</Steps>

### Access Mode Reference

<Note>
  The `@CheckAccess` decorator supports two modes: **OR (default)** and **AND**
</Note>

**OR Mode:** User has required permission **OR** stakeholder access → allowed

**AND Mode:** User must have permission **AND** stakeholder access → allowed

**Resource Action Hierarchy:**
- `ACCESS` — granted by `READ`, `WRITE`, or `FULL` access level
- `MODIFY` — granted by `WRITE` or `FULL`
- `DELETE` — granted by `FULL` only

## Test Data Seeding

### User Setup (One-Time)

<Warning>
  Run this command once before executing tests to create RBAC personas in the database.
</Warning>

```bash
pnpm test:seed-users
```

This command (`test/setup/seed-test-users.ts`):
- Creates RBAC test personas (idempotent)
- Creates "Test Organization" with owner
- Creates "Test Sales Team" with roles
- Assigns org and team permissions
- Uses RLS bypass for direct database writes

### Test Data Creation (Automatic)

<Info>
  Test data is seeded automatically per test run and cached across files for performance.
</Info>

Each test file calls `seedRbacTestData(app, personas, teamId)` which creates entities through the API:

<AccordionGroup>
  <Accordion title="Lead Test Data">
    - Lead by BASIC (auto-assigned as primary stakeholder with FULL access)
    - Lead by OWNER (for admin-only tests)
    - Lead in pool (empty stakeholders)
    - Lead with team as stakeholder
    - Lead with AGENT as READ-only stakeholder
  </Accordion>
  
  <Accordion title="Other Entity Data">
    - **Contacts:** One by BASIC, one by OWNER
    - **Companies:** One by BASIC, one by OWNER  
    - **Deals:** One by OWNER (from lead + contact)
    - **Events:** One linked to OWNER lead, one with BASIC as invitee
    - **Escalation rules:** One OWNER-created rule for RBAC testing
  </Accordion>
</AccordionGroup>

<Tip>
  Seeding through the API ensures all business logic (guards, auto-assignment, stakeholder creation) executes naturally, providing realistic test scenarios.
</Tip>

## How to Run Tests

### Core Test Suites

```bash
# Run all E2E tests (excludes queue tests)
pnpm test:e2e

# Run specific test file
pnpm test:e2e lead-rbac.e2e-spec.ts

# Run with coverage
pnpm test:e2e --coverage
```

### Queue Tests (Live pg-boss)

```bash
# Run queue-specific tests with live pg-boss
pnpm test:e2e:queue

# Run distribution tests specifically
pnpm test:e2e:queue distribution
```

### Test Environment Setup

<Steps>
  <Step title="Database Preparation">
    Ensure test database is running and migrated
  </Step>
  <Step title="Seed Users">
    Run `pnpm test:seed-users` (first time only)
  </Step>
  <Step title="Execute Tests">
    Run desired test command
  </Step>
  <Step title="Cleanup (Optional)">
    Run `pnpm test:cleanup-users` to remove test data
  </Step>
</Steps>

## How to Extend

### Adding New Entity Tests

<Steps>
  <Step title="Create Permission Matrix">
    Add new matrix file in `test/setup/matrices/` following existing patterns
  </Step>
  <Step title="Update Test Data Seeding">
    Extend `seedRbacTestData()` to create required test entities via API
  </Step>
  <Step title="Create Test File">
    Add new E2E test file in `test/e2e/rbac/` using matrix runner pattern
  </Step>
  <Step title="Update Barrel Exports">
    Add exports to `test/setup/matrices/index.ts`
  </Step>
</Steps>

### Adding New Personas

<Warning>
  Adding personas requires updates to matrices, seeding logic, and authentication flow.
</Warning>

1. Update `test/helpers/rbac-personas.ts` with new persona definition
2. Modify `seed-test-users.ts` to create the new user and assign roles
3. Update all permission matrices to include expectations for the new persona
4. Extend authentication logic in `rbac-context.ts`

### Matrix Customization

```typescript
// Example: Custom endpoint with special requirements
{
  method: 'POST',
  path: '/v1/leads/:leadByBasicId/activities',
  description: 'Create lead activity',
  body: { title: 'Test Activity', type: 'call' },
  expectations: {
    OWNER: 201,
    ADMIN: 201,
    TEAM_LEADER: 403,
    AGENT: 403,
    BASIC: 201,
  },
  skipPersonas: ['AGENT'], // Optional: skip specific personas
}
```

## CI/CD Integration

<Note>
  Tests run automatically in GitHub Actions with database seeding and cleanup.
</Note>

The CI pipeline:
1. Spins up test database
2. Runs migrations
3. Seeds RBAC personas
4. Executes test suites
5. Collects coverage reports
6. Cleans up test data

**Environment Variables Required:**
- `DATABASE_URL` - Test database connection
- `JWT_SECRET` - For token generation
- `REDIS_URL` - For caching (if applicable)

## Methodology

<Tip>
  See `TESTING_METHODOLOGY.md` for detailed guidance on analyzing controllers and DTOs before writing tests.
</Tip>

The testing methodology emphasizes:

- **Permission-first design** - Every endpoint must declare its access requirements
- **Realistic scenarios** - Test data reflects actual business workflows
- **Comprehensive coverage** - All persona × endpoint combinations tested
- **Performance optimization** - Shared context and cached data seeding
