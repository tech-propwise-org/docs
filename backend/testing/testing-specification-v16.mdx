# Testing Specification v16

Architecture, conventions, and run guide for the PropWise CRM backend test suite.

## Overview

The test suite focuses on **RBAC (Role-Based Access Control) E2E testing** — verifying that every API endpoint enforces the correct permissions for every user role.

<Info>
**Stack:** Jest + supertest + `@nestjs/testing`
</Info>

### Key design decisions

- **Singleton app context** — the NestJS app boots once and is shared across all test files (equivalent to NestJS `TestSuite` pattern, implemented manually since `TestSuite` requires NestJS 11+)
- **Matrix-driven tests** — a declarative permission matrix defines expected HTTP status codes per persona × endpoint. A runner generates `it()` blocks automatically
- **API-driven seeding** — test data is created through the actual API endpoints so all business logic (guards, auto-assignment, stakeholder creation) executes naturally

## Architecture

### Singleton Context

The app boots once for the entire test run. All test files share the same NestJS app instance, authenticated tokens, and resolved `teamId`.

```typescript
Jest starts
  → lead-rbac.e2e-spec.ts calls getContext()
    → First call: boots NestJS app (~3-5s), logs in 5 personas, resolves teamId
    → Returns { app, personas, teamId }
  → deal-rbac.e2e-spec.ts calls getContext()
    → Returns cached instance instantly (0ms)
  → ... same for contact and company files
  → Last file's afterAll() calls closeContext()
    → Reference count hits 0 → app shuts down
```

<CodeGroup>

```typescript rbac-context.ts
import { getContext, closeContext, RbacTestContext } from '../../setup/rbac-context';

let ctx: RbacTestContext;
beforeAll(async () => {
  ctx = await getContext();
}, 60000);
afterAll(async () => {
  await closeContext();
});
```

</CodeGroup>

The context replicates production config from `main.ts`:

- `ValidationPipe` (whitelist, transform, implicit conversion)
- URI versioning (`/v1/...`)
- Cookie parser

### Matrix Runner

Instead of writing individual `it()` blocks, you define expectations declaratively:

```typescript
{
  method: 'GET',
  path: '/v1/leads/:leadByBasicId',
  description: 'View lead detail',
  expectations: {
    OWNER: 200,
    ADMIN: 200,
    TEAM_LEADER: 403,
    AGENT: 403,
    BASIC: 200,
  },
}
```

The runner calls `runRbacMatrix()` which:

<Steps>
<Step title="Iterate endpoints">
Process every endpoint in the matrix
</Step>

<Step title="Iterate personas">
For each endpoint, test all 5 personas
</Step>

<Step title="Resolve placeholders">
Replace path placeholders (`:leadByBasicId` → actual UUID)
</Step>

<Step title="Make request">
Execute HTTP request with the persona's tenant token
</Step>

<Step title="Assert response">
Verify response status matches expected value and provide clear failure messages
</Step>
</Steps>

<Note>
One matrix entry = 5 generated test cases.
</Note>

## RBAC Personas

Defined in `test/helpers/rbac-personas.ts`. All personas share a single source of truth.

| Key           | Email                   | Org Role | Team                           | Description                                                       |
| ------------- | ----------------------- | -------- | ------------------------------ | ----------------------------------------------------------------- |
| `OWNER`       | owner@propwise.com      | Owner    | Any                            | All permissions including `system.owner`                          |
| `ADMIN`       | admin@propwise.com      | Admin    | Any                            | All non-sensitive permissions (includes `system.admin`)           |
| `TEAM_LEADER` | teamleader@propwise.com | Basic    | Team Leader in Test Sales Team | All team permissions (`team_crm.*`, `team_sales.*`, `team.admin`) |
| `AGENT`       | agent@propwise.com      | Basic    | Agent in Test Sales Team       | No team permissions by default                                    |
| `BASIC`       | basic@propwise.com      | Basic    | None                           | No permissions — relies entirely on stakeholder access            |

### Permission Profiles

<Tabs>
<Tab title="Owner/Admin">
**OWNER / ADMIN** — bypass resource-level checks via org permissions (`crm.view`, `crm.manage`, `sales.view`, `sales.manage`, etc.). Can access any entity in the org.
</Tab>

<Tab title="Team Leader">
**TEAM_LEADER** — no org permissions, but has `team_crm.manage` and `team_sales.manage` in the team. Can access entities where the team is a stakeholder.
</Tab>

<Tab title="Agent">
**AGENT** — no org permissions, no team permissions. Can only access entities where they are personally a stakeholder. Used to test access level differentiation (READ vs WRITE vs FULL).
</Tab>

<Tab title="Basic">
**BASIC** — same as AGENT but not a team member at all. Access is purely stakeholder-based.
</Tab>
</Tabs>

### Login Flow

The context performs a two-step login for each persona:

<Steps>
<Step title="Identity authentication">
`POST /v1/auth/login` → identity token
</Step>

<Step title="Organization selection">
`POST /v1/auth/select-organization` → tenant token (org-scoped JWT)
</Step>
</Steps>

<Info>
All subsequent API calls use the tenant token in the `Authorization: Bearer` header.
</Info>

## Permission Matrix

Matrices are split into per-entity files under `test/setup/matrices/`. The barrel at `test/setup/rbac-matrix.ts` re-exports everything.

### Lead Matrices

<AccordionGroup>
<Accordion title="Lead By Basic Matrix">

<Accordion title="Lead On Team Matrix">

<Accordion title="Lead Agent Read Matrix">
**Entity:** Lead where AGENT has READ access

</Accordion>

<Accordion title="Lead In Pool Matrix">
**Entity:** Lead with no stakeholders  
**Who Can Access:** OWNER, ADMIN only (via `crm.*`)  
**Tests:** View, update
</Accordion>

<Accordion title="Lead Admin Only Matrix">
**Entity:** Lead for admin-only ops  
**Who Can Access:** OWNER, ADMIN (via `system.admin`)

</Accordion>
</AccordionGroup>

### Access Mode Reference

Your `@CheckAccess` decorator uses two modes:

<Tabs>
<Tab title="OR (default)">
User has the required permission **OR** has stakeholder-level access on the resource → allowed.
</Tab>

<Tab title="AND">
User must have the permission **AND** stakeholder access → allowed.
</Tab>
</Tabs>

Resource action hierarchy:

- `ACCESS` — granted by `READ`, `WRITE`, or `FULL` access level
- `MODIFY` — granted by `WRITE` or `FULL`
- `DELETE` — granted by `FULL` only

## Test Data Seeding

### Step 1: Seed Users (one-time, before running tests)

```bash
pnpm test:seed-users
```

This runs `test/setup/seed-test-users.ts` which:

<Steps>
<Step title="Create personas">
Creates the RBAC test personas in the database (idempotent — skips if they exist)
</Step>

<Step title="Create organization">
Creates "Test Organization" with owner
</Step>

<Step title="Setup subscription">
Ensures **ACTIVE BUSINESS `subscription`** and **completed `organization_setup_progress`** on the test org (self-heals stale FREE / `LEAD_PIPELINE` rows from older seeds so Playwright and RBAC E2E skip onboarding and paid-plan gates)
</Step>

<Step title="Create team">
Creates "Test Sales Team" with team leader and agent
</Step>

<Step title="Assign roles">
Assigns org roles (Owner, Admin, Basic) and team roles (Team Leader, Agent)
</Step>

<Step title="Persist data">
Uses RLS bypass to write directly via MikroORM
</Step>
</Steps>

<Warning>
Requires subscription plans to exist (run `pnpm test:init-data` once on a fresh DB before `pnpm test:seed-users`).
</Warning>

### Step 2: Seed Test Data (automatic, per test run)

Each test file calls `seedRbacTestData(app, personas, teamId)` in its `beforeAll`. Results are cached across files. This creates entities through the API:

- **Leads:**
  - One by BASIC (auto-assigned as primary stakeholder with FULL access)
  - One by OWNER (for admin-only tests)
  - One in pool (empty stakeholders)
  - One with team as stakeholder (OWNER creates, assigns team — TEAM_LEADER gets access)
  - One with AGENT as READ-only stakeholder (OWNER creates, adds AGENT with `accessLevel: READ`)

- **Deals:**
  - One by OWNER (admin-only access)

- **Contacts:**
  - One by BASIC (stakeholder access)
  - One by OWNER (admin-only access)

- **Companies:**
  - One by BASIC (stakeholder access)
  - One by OWNER (admin-only access)

## How to Run

### Core RBAC Tests

```bash
# Run all RBAC E2E tests
pnpm test:e2e

# Run specific test file
pnpm test:e2e test/e2e/rbac/lead-rbac.e2e-spec.ts

# Run with verbose output
pnpm test:e2e --verbose
```

### Queue Tests

```bash
# Run queue-specific tests (distribution, escalation)
pnpm test:e2e:queue
```

### Test Database Setup

<Steps>
<Step title="Initialize data">

pnpm test:init-data

</Step>

<Step title="Seed test users">

## Methodology

The test suite follows a systematic approach to RBAC validation:

<CardGroup cols={2}>
<Card title="Permission-first" icon="shield">
Start with the permission requirements defined in your guards and decorators
</Card>

<Card title="Stakeholder-aware" icon="users">
Test both permission-based and stakeholder-based access patterns
</Card>

<Card title="Matrix-driven" icon="table">
Use declarative matrices to ensure comprehensive coverage
</Card>

<Card title="API-realistic" icon="globe">
Create test data through actual API endpoints to match production behavior
</Card>
</CardGroup>

### Test Categories

1. **Permission bypass tests** — OWNER/ADMIN should access everything via org permissions
2. **Team permission tests** — TEAM_LEADER should access team-stakeholder entities
3. **Stakeholder access tests** — Users should access entities where they have stakeholder rights
4. **Access level tests** — READ vs WRITE vs FULL access enforcement
5. **Denial tests** — Users without permissions or stakeholder access should be denied

## CI/CD Integration

### GitHub Actions

The test suite integrates with CI/CD pipelines:

```yaml
- name: Run E2E Tests
  run: |
    pnpm test:init-data
    pnpm test:seed-users
    pnpm test:e2e
  env:
    DATABASE_URL: ${{ secrets.TEST_DATABASE_URL }}
    JWT_SECRET: ${{ secrets.JWT_SECRET }}
```

### Test Environments

<Tabs>
<Tab title="Local Development">
- Uses local PostgreSQL instance
- Test data persists between runs for faster iteration
- Manual cleanup with `pnpm test:cleanup-users`
</Tab>

<Tab title="CI Pipeline">
- Uses fresh database for each run
- Automatic cleanup after test completion
- Parallel test execution where possible
</Tab>

### Performance Considerations

- **Singleton app context** reduces startup overhead from ~15s to ~3-5s for the entire suite
- **Cached test data** prevents redundant API calls across test files
- **Matrix-driven approach** generates hundreds of test cases from minimal configuration

<Tip>
The test suite typically completes in under 30 seconds for the full RBAC matrix, making it suitable for CI/CD integration.
</Tip>
