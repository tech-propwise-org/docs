# Testing Architecture

The PropWise CRM backend test suite focuses on comprehensive **Role-Based Access Control (RBAC) end-to-end testing**, verifying that every API endpoint enforces correct permissions for every user role.

<Note>
The test architecture uses Jest + supertest + `@nestjs/testing` with a singleton app context for optimal performance and reliability.
</Note>

## Key Design Principles

### Singleton App Context
- **Single boot cycle** — NestJS app boots once and is shared across all test files
- **Shared authentication** — All personas authenticate once and reuse tokens
- **Performance optimized** — First test file takes ~3-5s to boot, subsequent files start instantly

### Matrix-Driven Testing
- **Declarative permissions** — Permission expectations defined in matrices rather than individual test cases
- **Automated generation** — Test runner generates parameterized Jest tests from matrix definitions
- **Comprehensive coverage** — Every endpoint × persona combination tested automatically

### API-Driven Seeding
- **Production-like data** — Test data created through actual API endpoints
- **Business logic execution** — Guards, auto-assignment, and stakeholder creation run naturally
- **Realistic scenarios** — Data matches what real users would produce

## RBAC Personas

<Info>
All personas are defined in `test/helpers/rbac-personas.ts` as the single source of truth for test user configurations.
</Info>

### Persona Definitions

| Persona | Email | Org Role | Team Role | Description |
|---------|-------|----------|-----------|-------------|
| **OWNER** | owner@propwise-test.com | Owner | N/A | Full system access including `system.owner` |
| **ADMIN** | admin@propwise-test.com | Admin | N/A | All non-sensitive permissions via `system.admin` |
| **TEAM_LEADER** | teamleader@propwise-test.com | Basic | Team Leader | Team permissions (`team_crm.*`, `team_sales.*`) |
| **AGENT** | agent@propwise-test.com | Basic | Agent | No default permissions, stakeholder-based access |
| **BASIC** | basic@propwise-test.com | Basic | None | Pure stakeholder-based access testing |

### Permission Profiles

<Tabs>
  <Tab title="OWNER / ADMIN">
    - **Bypass resource checks** via organization-level permissions
    - Can access any entity within the organization
    - **OWNER**: Includes sensitive operations (`system.owner`)
    - **ADMIN**: Excludes owner-only operations
  </Tab>
  
  <Tab title="TEAM_LEADER">
    - No organization permissions
    - Has `team_crm.manage` and `team_sales.manage` in assigned team
    - Can access entities where team is a stakeholder
    - Tests team-based permission inheritance
  </Tab>
  
  <Tab title="AGENT / BASIC">
    - No organization or team permissions
    - Pure stakeholder-based access
    - **AGENT**: Team member but no team permissions
    - **BASIC**: Not a team member at all
    - Tests access level differentiation (READ vs WRITE vs FULL)
  </Tab>
</Tabs>

## Singleton Context Architecture

The singleton pattern ensures optimal performance and resource utilization across the test suite.

### Context Lifecycle

```typescript
// First test file
Jest starts → getContext() called
  ↓
App boots (~3-5s) → Personas authenticate → TeamId resolved
  ↓
Context cached and returned

// Subsequent test files  
getContext() called → Cached context returned (0ms)

// Test completion
Last file afterAll() → closeContext() called
  ↓
Reference count = 0 → App shutdown
```

### Usage Pattern

```typescript
import { getContext, closeContext, RbacTestContext } from '../../setup/rbac-context';

let ctx: RbacTestContext;

beforeAll(async () => {
  ctx = await getContext();
}, 60000);

afterAll(async () => {
  await closeContext();
});
```

<Warning>
The context replicates production configuration including ValidationPipe, URI versioning (`/v1/...`), and cookie parser to ensure test fidelity.
</Warning>

## Matrix-Driven Testing

### Permission Matrix Structure

Instead of writing individual test cases, permissions are defined declaratively:

```typescript
{
  method: 'GET',
  path: '/v1/leads/:leadByBasicId',
  description: 'View lead detail',
  expectations: {
    OWNER: 200,        // Full access via crm.view
    ADMIN: 200,        // Full access via crm.view  
    TEAM_LEADER: 403,  // No team stakeholder access
    AGENT: 403,        // No personal stakeholder access
    BASIC: 200,        // Creator with FULL access level
  },
}
```

### Test Generation Process

<Steps>
  <Step title="Matrix Iteration">
    Runner iterates through each endpoint definition in the matrix
  </Step>
  
  <Step title="Persona Testing">
    For each endpoint, tests all 5 personas with their respective tokens
  </Step>
  
  <Step title="Path Resolution">
    Resolves path placeholders (`:leadByBasicId` → actual UUID from test data)
  </Step>
  
  <Step title="Request Execution">
    Makes HTTP request with persona's tenant token in Authorization header
  </Step>
  
  <Step title="Assertion">
    Compares actual response status with expected status from matrix
  </Step>
</Steps>

<Tip>
One matrix entry generates 5 parameterized test cases automatically, providing comprehensive coverage with minimal code.
</Tip>

## Access Control Patterns

### Permission Matrix Categories

<AccordionGroup>
  <Accordion title="Lead Matrices">
    - **LEAD_BY_BASIC_MATRIX**: Creator stakeholder access (BASIC has FULL)
    - **LEAD_ON_TEAM_MATRIX**: Team stakeholder access (TEAM_LEADER via team)
    - **LEAD_AGENT_READ_MATRIX**: Read-only stakeholder access (AGENT READ level)
    - **LEAD_IN_POOL_MATRIX**: No stakeholders (admin-only access)
    - **LEAD_ADMIN_ONLY_MATRIX**: Administrative operations (delete, etc.)
  </Accordion>
  
  <Accordion title="Deal Matrices">
    - **DEAL_BY_OWNER_MATRIX**: Sales pipeline access control
    - Tests `sales.*` permissions vs stakeholder access
  </Accordion>
  
  <Accordion title="Contact Matrices">
    - **CONTACT_BY_BASIC_MATRIX**: Creator access patterns
    - **CONTACT_BY_OWNER_MATRIX**: Admin-level contact management
  </Accordion>
  
  <Accordion title="Company Matrices">
    - **COMPANY_BY_BASIC_MATRIX**: User-created company access
    - **COMPANY_BY_OWNER_MATRIX**: Organization company management
  </Accordion>
</AccordionGroup>

### Access Mode Reference

Your `@CheckAccess` decorator supports two verification modes:

<Tabs>
  <Tab title="OR Mode (Default)">
    **User has required permission OR stakeholder access**
    - Most common pattern for CRUD operations
    - Allows both permission-based and stakeholder-based access
    - Example: `crm.view` permission OR READ stakeholder access
  </Tab>
  
  <Tab title="AND Mode">
    **User must have permission AND stakeholder access**
    - Used for sensitive operations requiring both
    - Ensures proper authorization layering
    - Example: Admin permission AND resource ownership
  </Tab>
</Tabs>

### Access Level Hierarchy

| Level | Grants | Used For |
|-------|--------|----------|
| **READ** | `ACCESS` action | View-only operations |
| **WRITE** | `ACCESS`, `MODIFY` actions | Edit operations |
| **FULL** | `ACCESS`, `MODIFY`, `DELETE` actions | Complete control |

<Check>
The test suite validates this hierarchy by testing that READ-level stakeholders cannot perform MODIFY operations, ensuring proper access control enforcement.
</Check>
