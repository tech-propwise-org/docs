## Overview

The PropWise CRM backend test suite focuses on **RBAC (Role-Based Access Control) E2E testing** — verifying that every API endpoint enforces the correct permissions for every user role.

<Info>
**Stack:** Jest + supertest + `@nestjs/testing`
</Info>

### Key Design Decisions

<CardGroup cols={2}>
  <Card title="Singleton App Context" icon="server">
    The NestJS app boots once and is shared across all test files (equivalent to NestJS `TestSuite` pattern, implemented manually since `TestSuite` requires NestJS 11+)
  </Card>
  <Card title="Matrix-Driven Tests" icon="table">
    A declarative permission matrix defines expected HTTP status codes per persona × endpoint. A runner generates `it()` blocks automatically
  </Card>
  <Card title="API-Driven Seeding" icon="seedling">
    Test data is created through actual API endpoints so all business logic (guards, auto-assignment, stakeholder creation) executes naturally
  </Card>
  <Card title="Cached Test Data" icon="database">
    Test fixtures are created once and shared across test files for faster execution
  </Card>
</CardGroup>

## Singleton Context Pattern

### How It Works

The app boots once for the entire test run. All test files share the same NestJS app instance, authenticated tokens, and resolved `teamId`.

<Steps>
  <Step title="First Test File Initialization">
    When `lead-rbac.e2e-spec.ts` calls `getContext()`:
    - Boots NestJS app (~3-5s)
    - Authenticates all 5 personas
    - Resolves teamId
    - Returns `{ app, personas, teamId }`
  </Step>
  
  <Step title="Subsequent Test Files">
    When `deal-rbac.e2e-spec.ts` calls `getContext()`:
    - Returns cached instance instantly (0ms)
    - No additional initialization required
  </Step>
  
  <Step title="Cleanup">
    Last file's `afterAll()` calls `closeContext()`:
    - Reference count hits 0
    - App shuts down gracefully
  </Step>
</Steps>

### Implementation Example

```typescript test/e2e/rbac/lead-rbac.e2e-spec.ts
import { getContext, closeContext, RbacTestContext } from '../../setup/rbac-context';

let ctx: RbacTestContext;

beforeAll(async () => {
  ctx = await getContext();
}, 60000);

afterAll(async () => {
  await closeContext();
});
```

### Context Configuration

The context replicates production config from `main.ts`:

<AccordionGroup>
  <Accordion title="ValidationPipe">
    - `whitelist: true` — strips unknown properties
    - `transform: true` — transforms payloads to DTO instances
    - `transformOptions: { enableImplicitConversion: true }` — auto-converts types
  </Accordion>
  
  <Accordion title="Versioning">
    - URI versioning enabled (`/v1/...`)
    - Matches production API structure
  </Accordion>
  
  <Accordion title="Middleware">
    - Cookie parser enabled
    - Matches production middleware stack
  </Accordion>
</AccordionGroup>

## Matrix Runner System

### Declarative Permission Definition

Instead of writing individual `it()` blocks, you define expectations declaratively:

```typescript test/setup/matrices/lead-matrix.ts
{
  method: 'GET',
  path: '/v1/leads/:leadByBasicId',
  description: 'View lead detail',
  expectations: {
    OWNER: 200,
    ADMIN: 200,
    TEAM_LEADER: 403,
    AGENT: 403,
    BASIC: 200,
  },
}
```

<Note>
One matrix entry generates **5 test cases** (one per persona). This dramatically reduces boilerplate while improving maintainability.
</Note>

### Runner Execution Flow

The runner calls `runRbacMatrix()` which:

<Steps>
  <Step title="Iterate Endpoints">
    Loop through every endpoint defined in the matrix
  </Step>
  
  <Step title="Iterate Personas">
    For each endpoint, test all 5 personas (OWNER, ADMIN, TEAM_LEADER, AGENT, BASIC)
  </Step>
  
  <Step title="Resolve Placeholders">
    Replace path parameters like `:leadByBasicId` with actual UUIDs from test data
  </Step>
  
  <Step title="Execute Request">
    Make HTTP request with persona's tenant token in Authorization header
  </Step>
  
  <Step title="Assert Status">
    Verify response status matches expected value from matrix
  </Step>
  
  <Step title="Report Failures">
    On mismatch, output clear failure message with actual response body
  </Step>
</Steps>

## RBAC Personas

Defined in `test/helpers/rbac-personas.ts`. All personas share a single source of truth.

### Persona Reference

| Key | Email | Org Role | Team | Description |
|-----|-------|----------|------|-------------|
| `OWNER` | owner@propwise.com | Owner | Any | All permissions including `system.owner` |
| `ADMIN` | admin@propwise.com | Admin | Any | All non-sensitive permissions (includes `system.admin`) |
| `TEAM_LEADER` | teamleader@propwise.com | Basic | Team Leader in Test Sales Team | All team permissions (`team_crm.*`, `team_sales.*`, `team.admin`) |
| `AGENT` | agent@propwise.com | Basic | Agent in Test Sales Team | No team permissions by default |
| `BASIC` | basic@propwise.com | Basic | None | No permissions — relies entirely on stakeholder access |

### Permission Profiles

<Tabs>
  <Tab title="OWNER / ADMIN">
    **Org-Level Access**
    
    - Bypass resource-level checks via org permissions
    - `crm.view`, `crm.manage`, `sales.view`, `sales.manage`
    - Can access any entity in the organization
    
    **Use Case:** Testing administrative operations and org-wide access patterns
  </Tab>
  
  <Tab title="TEAM_LEADER">
    **Team-Level Access**
    
    - No org permissions
    - Has `team_crm.manage` and `team_sales.manage` in team
    - Can access entities where team is a stakeholder
    
    **Use Case:** Testing team-based collaboration and hierarchical access
  </Tab>
  
  <Tab title="AGENT">
    **Individual Access**
    
    - No org permissions, no team permissions
    - Can only access entities where personally a stakeholder
    - Used to test access level differentiation (READ vs WRITE vs FULL)
    
    **Use Case:** Testing granular stakeholder permissions and access levels
  </Tab>
  
  <Tab title="BASIC">
    **Minimal Access**
    
    - Same as AGENT but not a team member
    - Access is purely stakeholder-based
    - No inherited team permissions
    
    **Use Case:** Testing isolated stakeholder access without team context
  </Tab>
</Tabs>

### Authentication Flow

<Steps>
  <Step title="Identity Login">
    `POST /v1/auth/login` → Returns identity token
  </Step>
  
  <Step title="Organization Selection">
    `POST /v1/auth/select-organization` → Returns tenant token (org-scoped JWT)
  </Step>
  
  <Step title="API Requests">
    All subsequent API calls use tenant token in `Authorization: Bearer` header
  </Step>
</Steps>

## Permission Matrices

Matrices are split into per-entity files under `test/setup/matrices/`. The barrel at `test/setup/rbac-matrix.ts` re-exports everything.

### Lead Matrices

<AccordionGroup>
  <Accordion title="LEAD_BY_BASIC_MATRIX">
    **Entity:** Lead created by BASIC persona
    
    **Who Can Access:**
    - OWNER, ADMIN (via `crm.*` permissions)
    - BASIC (via FULL stakeholder access)
    
    **Tests:** View, update, stakeholders, activities, property interests
  </Accordion>
  
  <Accordion title="LEAD_ON_TEAM_MATRIX">
    **Entity:** Lead with team as stakeholder
    
    **Who Can Access:**
    - OWNER, ADMIN (via `crm.*` permissions)
    - TEAM_LEADER (via `team_crm.*` permissions)
    
    **Tests:** View, update, stakeholders, activities
  </Accordion>
  
  <Accordion title="LEAD_AGENT_READ_MATRIX">
    **Entity:** Lead where AGENT has READ-only access
    
    **Who Can Access:**
    - OWNER, ADMIN (full access)
    - AGENT (view only — modify operations denied)
    
    **Tests:** View allowed, update denied, access level enforcement
  </Accordion>
  
  <Accordion title="LEAD_IN_POOL_MATRIX">
    **Entity:** Lead with no stakeholders (unassigned pool)
    
    **Who Can Access:**
    - OWNER, ADMIN only (via `crm.*` permissions)
    
    **Tests:** View, update operations on unassigned leads
  </Accordion>
  
  <Accordion title="LEAD_ADMIN_ONLY_MATRIX">
    **Entity:** Lead for administrative operations
    
    **Who Can Access:**
    - OWNER, ADMIN (via `system.admin` permission)

</AccordionGroup>

## Access Control Modes

Your `@CheckAccess` decorator uses two modes for permission evaluation:

<Tabs>
  <Tab title="OR Mode (Default)">
    **Logic:** User has required permission **OR** has stakeholder-level access
    
    **Example:**
    ```typescript
    @CheckAccess('crm', 'view', 'lead')
    // ✅ User with crm.view permission
    // ✅ User with READ/WRITE/FULL stakeholder access
    ```
  </Tab>
  
  <Tab title="AND Mode">
    **Logic:** User must have permission **AND** stakeholder access
    
    **Example:**
    ```typescript
    @CheckAccess('crm', 'view', 'lead', { mode: 'AND' })
    // ✅ User with crm.view AND stakeholder access
    // ❌ User with only crm.view (no stakeholder)
    // ❌ User with only stakeholder access (no permission)
    ```
  </Tab>
</Tabs>

### Resource Action Hierarchy

<Steps>
  <Step title="ACCESS">
    Granted by `READ`, `WRITE`, or `FULL` access level
    
    - View entity details
    - List related resources
  </Step>
  
  <Step title="MODIFY">
    Granted by `WRITE` or `FULL` access level
    
    - Update entity fields
    - Add/remove related resources
  </Step>
  
  <Step title="DELETE">
    Granted by `FULL` access level only
    
    - Delete entity
    - Irreversible operations
  </Step>
</Steps>

## Test Data Seeding

### Step 1: Seed Users (One-Time Setup)

<Warning>
Run this command **once** before executing tests for the first time.
</Warning>

```bash
pnpm test:seed-users
```

This runs `test/setup/seed-test-users.ts` which performs the following operations:

<Steps>
  <Step title="Create RBAC Personas">
    Creates all 5 test personas in the database (idempotent — skips if they exist)
  </Step>
  
  <Step title="Create Test Organization">
    Creates "Test Organization" with owner assigned
  </Step>
  
  <Step title="Activate Subscription">
    Ensures **ACTIVE BUSINESS subscription** and completed `organization_setup_progress` on test org
    
    <Info>
    Self-heals stale FREE / `LEAD_PIPELINE` rows from older seeds so Playwright and RBAC E2E skip onboarding and paid-plan gates
    </Info>
  </Step>
  
  <Step title="Create Test Team">
    Creates "Test Sales Team" with team leader and agent members
  </Step>
  
  <Step title="Assign Roles">
    Assigns org roles (Owner, Admin, Basic) and team roles (Team Leader, Agent)
  </Step>
  
  <Step title="Bypass RLS">
    Uses RLS bypass to write directly via MikroORM for setup operations
  </Step>
</Steps>

<Note>
**Prerequisites:** Subscription plans must exist. Run `pnpm test:init-data` once on a fresh database before `pnpm test:seed-users`.
</Note>

### Step 2: Seed Test Data (Automatic)

Each test file calls `seedRbacTestData(app, personas, teamId)` in its `beforeAll`. Results are cached across files.

This creates entities through the API:

<Tabs>
  <Tab title="Leads">
    - **Lead by BASIC:** Auto-assigned as primary stakeholder with FULL access
    - **Lead by OWNER:** For admin-only operation tests
    - **Lead in pool:** Empty stakeholders list (unassigned)
    - **Lead with team:** OWNER creates and assigns team (TEAM_LEADER gets access)
    - **Lead with AGENT READ:** OWNER creates, adds AGENT with READ-only access level
  </Tab>
  
  <Tab title="Deals">
    - **Deal by OWNER:** Created by OWNER for sales pipeline tests
  </Tab>
  
  <Tab title="Contacts">
    - **Contact by BASIC:** Created by BASIC persona
    - **Contact by OWNER:** Created by OWNER persona
  </Tab>
  
  <Tab title="Companies">
    - **Company by BASIC:** Created by BASIC persona
    - **Company by OWNER:** Created by OWNER persona
  </Tab>
</Tabs>

<Check>
**Why API-driven seeding?**

Creating entities through actual API endpoints ensures all business logic executes naturally:
- Guards run
- Auto-assignment triggers
- Stakeholder creation happens
- Audit trails are created
- Matches real-world usage patterns
</Check>

## Running Tests

### Core RBAC Test Suite

```bash
pnpm test:e2e
```

Uses `jest-e2e.json` configuration. Excludes distribution and escalation queue tests.

### Queue-Based Tests

```bash
pnpm test:e2e:queue
```

Uses `jest-e2e-queue.json` configuration. Runs distribution, escalation, and queue-e2e specs with live pg-boss integration.

### Test Lifecycle

<Steps>
  <Step title="Global Setup">
    - Singleton context initializes
    - App boots once
    - All personas authenticate
    - TeamId resolves
  </Step>
  
  <Step title="Per-File Setup">
    - Test file calls `getContext()` (instant if cached)
    - Calls `seedRbacTestData()` (instant if cached)
    - Receives fixture IDs for matrix placeholder resolution
  </Step>
  
  <Step title="Test Execution">
    - Matrix runner generates parameterized tests
    - Each test makes HTTP request with persona token
    - Assertions verify expected status codes
  </Step>
  
  <Step title="Teardown">
    - Last file's `afterAll()` calls `closeContext()`
    - Reference count decrements
    - When zero, app shuts down
  </Step>
</Steps>

### Cleanup

Remove test users from database:

```bash
pnpm test:cleanup-users
```

Deletes all `@propwise-test.com` users and related data.

## Extending the Test Suite

### Adding a New Matrix

<Steps>
  <Step title="Create Matrix File">
    Create new file in `test/setup/matrices/` (e.g., `property-matrix.ts`)
    
    ```typescript
    import { EndpointExpectation } from '../rbac-matrix-types';
    
    export const PROPERTY_BY_OWNER_MATRIX: EndpointExpectation[] = [
      {
        method: 'GET',
        path: '/v1/properties/:propertyId',
        description: 'View property detail',
        expectations: {
          OWNER: 200,
          ADMIN: 200,
          TEAM_LEADER: 403,
          AGENT: 403,
          BASIC: 403,
        },
      },
      // ... more endpoints
    ];
    ```
  </Step>
  
  <Step title="Export from Barrel">
    Add to `test/setup/matrices/index.ts`:
    
    ```typescript
    export * from './property-matrix';
    ```
  </Step>
  
  <Step title="Create Test File">
    Create `test/e2e/rbac/property-rbac.e2e-spec.ts`:
    
    ```typescript
    import { getContext, closeContext } from '../../setup/rbac-context';
    import { seedRbacTestData } from '../../setup/rbac-test-data';
    import { runRbacMatrix } from '../../setup/rbac-runner';
    import { PROPERTY_BY_OWNER_MATRIX } from '../../setup/matrices';
    
    describe('Property RBAC', () => {
      let ctx;
      let testData;
      
      beforeAll(async () => {
        ctx = await getContext();
        testData = await seedRbacTestData(ctx.app, ctx.personas, ctx.teamId);
      }, 60000);
      
      afterAll(async () => {
        await closeContext();
      });
      
      runRbacMatrix('Property by OWNER', PROPERTY_BY_OWNER_MATRIX, () => ({
        app: ctx.app,
        personas: ctx.personas,
        placeholders: {
          propertyId: testData.properties.byOwner.id,
        },
      }));
    });
    ```
  </Step>
  
  <Step title="Add Fixture Data">
    Update `test/setup/rbac-test-data.ts` to create test properties via API
  </Step>
</Steps>

### Adding a New Persona

## Troubleshooting

<AccordionGroup>
  <Accordion title="Tests timeout during app bootstrap">
    **Symptom:** `beforeAll` timeout after 60 seconds
    
    **Solutions:**
    - Increase timeout: `beforeAll(async () => { ... }, 120000);`
    - Check database connection
    - Verify all migrations are applied
    - Ensure subscription plans exist (`pnpm test:init-data`)
  </Accordion>
  
  <Accordion title="Unexpected 401 responses">
    **Symptom:** Tests fail with unauthorized errors
    
    **Solutions:**
    - Re-run `pnpm test:seed-users` to refresh test users
    - Check token expiration settings
    - Verify authentication flow in `rbac-context.ts`
    - Ensure organization selection completed successfully
  </Accordion>
  
  <Accordion title="Matrix placeholder not resolved">
    **Symptom:** `Error: Placeholder :entityId not found`
    
    **Solutions:**
    - Verify `seedRbacTestData()` creates the required entity
    - Check placeholder name matches matrix path parameter
    - Ensure placeholder mapping in `runRbacMatrix()` call
  </Accordion>
  
  <Accordion title="Permission tests fail after code changes">
    **Symptom:** Expected 200, received 403 (or vice versa)
    
    **Solutions:**
    - Review recent guard changes
    - Check if permission keys changed
    - Verify stakeholder assignment logic
    - Update matrix expectations if behavior changed intentionally
  </Accordion>
</AccordionGroup>

## Best Practices

<CardGroup cols={2}>
  <Card title="Keep Matrices Synchronized" icon="sync">
    When adding new endpoints, immediately add them to relevant matrices. Don't let test coverage lag behind implementation.
  </Card>
  
  <Card title="Test Negative Cases" icon="shield-halved">
    Verify both allowed (200) and denied (403) access. Security testing requires validating rejections, not just successes.
  </Card>
  
  <Card title="Use Descriptive Names" icon="tag">
    Matrix descriptions should clearly state what's being tested. Future maintainers will thank you.
  </Card>
  
  <Card title="Leverage Caching" icon="bolt">
    The singleton context and cached test data make tests fast. Don't recreate fixtures unnecessarily.
  </Card>
</CardGroup>

<Tip>
When debugging matrix failures, check the actual response body in the test output. The runner includes it in failure messages to help identify guard logic issues.
</Tip>
