# Comprehensive Testing Specification

Architecture, conventions, and run guide for the PropWise CRM backend test suite.

## Overview

The test suite focuses on **RBAC (Role-Based Access Control) E2E testing** вЂ” verifying that every API endpoint enforces the correct permissions for every user role.

<Info>
**Stack:** Jest + supertest + `@nestjs/testing`
</Info>

### Key Design Decisions

<CardGroup cols={2}>
  <Card title="Singleton App Context" icon="server">
    The NestJS app boots once and is shared across all test files (equivalent to NestJS `TestSuite` pattern, implemented manually since `TestSuite` requires NestJS 11+)
  </Card>
  <Card title="Matrix-Driven Tests" icon="grid">
    A declarative permission matrix defines expected HTTP status codes per persona Г— endpoint. A runner generates `it()` blocks automatically
  </Card>
  <Card title="API-Driven Seeding" icon="database">
    Test data is created through the actual API endpoints so all business logic (guards, auto-assignment, stakeholder creation) executes naturally
  </Card>
</CardGroup>

## Architecture

### Singleton Context

<Note>
The app boots once for the entire test run. All test files share the same NestJS app instance, authenticated tokens, and resolved `teamId`.
</Note>

The context flow works as follows:

<Steps>
  <Step title="Jest starts">
    Test files begin execution
  </Step>
  <Step title="First call to getContext()">
    `lead-rbac.e2e-spec.ts` calls `getContext()` - boots NestJS app (~3-5s), logs in 5 personas, resolves teamId, returns `{ app, personas, teamId }`
  </Step>
  <Step title="Subsequent calls">
    `deal-rbac.e2e-spec.ts` calls `getContext()` - returns cached instance instantly (0ms)
  </Step>
  <Step title="Context cleanup">
    Last file's `afterAll()` calls `closeContext()` - reference count hits 0, app shuts down
  </Step>
</Steps>

**Implementation example:**

```typescript
import { getContext, closeContext, RbacTestContext } from '../../setup/rbac-context';

let ctx: RbacTestContext;
beforeAll(async () => { ctx = await getContext(); }, 60000);
afterAll(async () => { await closeContext(); });
```

The context replicates production config from `main.ts`:
- `ValidationPipe` (whitelist, transform, implicit conversion)
- URI versioning (`/v1/...`)
- Cookie parser

### Matrix Runner

<Tip>
Instead of writing individual `it()` blocks, you define expectations declaratively using a matrix system.
</Tip>

```typescript
{
  method: 'GET',
  path: '/v1/leads/:leadByBasicId',
  description: 'View lead detail',
  expectations: {
    OWNER: 200,
    ADMIN: 200,
    TEAM_LEADER: 403,
    AGENT: 403,
    BASIC: 200,
  },
}
```

The runner calls `runRbacMatrix()` which:

1. Iterates every endpoint in the matrix
2. For each endpoint, iterates all 5 personas
3. Resolves path placeholders (`:leadByBasicId` в†’ actual UUID)
4. Makes the HTTP request with the persona's tenant token
5. Asserts the response status matches the expected value
6. Produces clear failure messages with the actual response body

<Check>
One matrix entry = 5 generated test cases automatically
</Check>

## RBAC Personas

Defined in `test/helpers/rbac-personas.ts`. All personas share a single source of truth.

| Key | Email | Org Role | Team | Description |
|-----|-------|----------|------|-------------|
| `OWNER` | owner@propwise.com | Owner | Any | All permissions including `system.owner` |
| `ADMIN` | admin@propwise.com | Admin | Any | All non-sensitive permissions (includes `system.admin`) |
| `TEAM_LEADER` | teamleader@propwise.com | Basic | Team Leader in Test Sales Team | All team permissions (`team_crm.*`, `team_sales.*`, `team.admin`) |
| `AGENT` | agent@propwise.com | Basic | Agent in Test Sales Team | No team permissions by default |
| `BASIC` | basic@propwise.com | Basic | None | No permissions вЂ” relies entirely on stakeholder access |

### Permission Profiles

<AccordionGroup>
  <Accordion title="OWNER / ADMIN">
    Bypass resource-level checks via org permissions (`crm.view`, `crm.manage`, `sales.view`, `sales.manage`, etc.). Can access any entity in the org.
  </Accordion>
  <Accordion title="TEAM_LEADER">
    No org permissions, but has `team_crm.manage` and `team_sales.manage` in the team. Can access entities where the team is a stakeholder.
  </Accordion>
  <Accordion title="AGENT">
    No org permissions, no team permissions. Can only access entities where they are personally a stakeholder. Used to test access level differentiation (READ vs WRITE vs FULL).
  </Accordion>
  <Accordion title="BASIC">
    Same as AGENT but not a team member at all. Access is purely stakeholder-based.
  </Accordion>
</AccordionGroup>

### Login Flow

The context performs a two-step login for each persona:

<Steps>
  <Step title="Identity authentication">
    `POST /v1/auth/login` в†’ identity token
  </Step>
  <Step title="Organization selection">
    `POST /v1/auth/select-organization` в†’ tenant token (org-scoped JWT)
  </Step>
</Steps>

All subsequent API calls use the tenant token in the `Authorization: Bearer` header.

## Permission Matrix

Matrices are split into per-entity files under `test/setup/matrices/`. The barrel at `test/setup/rbac-matrix.ts` re-exports everything.

### Lead Matrices

<Tabs>
  <Tab title="LEAD_BY_BASIC_MATRIX">
    **Entity:** Lead created by BASIC  
    **Who Can Access:** OWNER, ADMIN (via `crm.*`), BASIC (via FULL stakeholder)  
    **Tests:** View, update, stakeholders, activities, property interests
  </Tab>
  <Tab title="LEAD_ON_TEAM_MATRIX">
    **Entity:** Lead with team as stakeholder

  </Tab>
  <Tab title="LEAD_AGENT_READ_MATRIX">
    **Entity:** Lead where AGENT has READ access  
    **Who Can Access:** OWNER, ADMIN, AGENT (view only вЂ” modify denied)

  </Tab>
  <Tab title="LEAD_IN_POOL_MATRIX">
    **Entity:** Lead with no stakeholders  
    **Who Can Access:** OWNER, ADMIN only (via `crm.*`)

  </Tab>
  <Tab title="LEAD_ADMIN_ONLY_MATRIX">
    **Entity:** Lead for admin-only ops  
    **Who Can Access:** OWNER, ADMIN (via `system.admin`)

  </Tab>
</Tabs>

### Access Mode Reference

<Info>
Your `@CheckAccess` decorator uses two modes:

- **OR (default):** User has the required permission **OR** has stakeholder-level access on the resource в†’ allowed.
- **AND:** User must have the permission **AND** stakeholder access в†’ allowed.
</Info>

Resource action hierarchy:
- `ACCESS` вЂ” granted by `READ`, `WRITE`, or `FULL` access level
- `MODIFY` вЂ” granted by `WRITE` or `FULL`
- `DELETE` вЂ” granted by `FULL` only

## Test Data Seeding

### Step 1: Seed Users (one-time, before running tests)

```bash
pnpm test:seed-users
```

This runs `test/setup/seed-test-users.ts` which:

<Steps>
  <Step title="Create test users">
    Creates the 5 test users in the database (idempotent вЂ” skips if they exist)
  </Step>
  <Step title="Create organization">
    Creates "Test Organization" with owner
  </Step>
  <Step title="Create team">
    Creates "Test Sales Team" with team leader and agent
  </Step>
  <Step title="Assign roles">
    Assigns org roles (Owner, Admin, Basic) and team roles (Team Leader, Agent)
  </Step>
  <Step title="Direct database write">
    Uses RLS bypass to write directly via MikroORM
  </Step>
</Steps>

### Step 2: Seed Test Data (automatic, per test run)

Each test file calls `seedRbacTestData(app, personas, teamId)` in its `beforeAll`. Results are cached across files. This creates entities through the API:

<Warning>
Because seeding goes through the real API, all guards, auto-assignment rules, and stakeholder creation run. The test data matches what a real user would produce.
</Warning>

**Created entities:**
- **Leads:**
  - One by BASIC (auto-assigned as primary stakeholder with FULL access)
  - One by OWNER (for admin-only tests)
  - One in pool (empty stakeholders)
  - One with team as stakeholder (OWNER creates, assigns team вЂ” TEAM_LEADER gets access)
  - One with AGENT as READ-only stakeholder (OWNER creates, adds AGENT with `accessLevel: read`)
- **Contacts:** one by BASIC, one by OWNER
- **Companies:** one by BASIC, one by OWNER
- **Deals:** one by OWNER (from lead + contact)

## How to Run

### Prerequisites

<Steps>
  <Step title="Database setup">
    Database must be running with migrations applied
  </Step>
  <Step title="Seed test users">
    Test users must be seeded: `pnpm test:seed-users`
  </Step>
  <Step title="Environment configuration">
    `.env.local` must have `DATABASE_URL`, `JWT_TENANT_SECRET`, `META_TOKEN_ENCRYPTION_KEY`, `META_WEBHOOK_VERIFY_TOKEN`
  </Step>
</Steps>

### Commands

<CodeGroup>
```bash Run all RBAC tests
pnpm test:e2e -- --testPathPattern=rbac
```

```bash Run specific entity
pnpm test:e2e -- --testPathPattern=lead-rbac
pnpm test:e2e -- --testPathPattern=deal-rbac
pnpm test:e2e -- --testPathPattern=contact-rbac
pnpm test:e2e -- --testPathPattern=company-rbac
```

```bash Verbose output
pnpm test:e2e -- --testPathPattern=rbac --verbose
```

```bash With coverage
pnpm test:e2e:cov -- --testPathPattern=rbac
```
</CodeGroup>

## How to Extend

### Adding New Endpoints

<Steps>
  <Step title="Update matrix file">
    Add the endpoint to the relevant matrix in `test/setup/matrices/`
  </Step>
  <Step title="Define expectations">
    Specify expected status codes for each persona
  </Step>
  <Step title="Handle path placeholders">
    If using dynamic IDs, ensure they're resolved in `rbac-test-data.ts`
  </Step>
  <Step title="Run tests">
    The matrix runner will automatically generate test cases
  </Step>
</Steps>

### Adding New Personas

<Steps>
  <Step title="Define persona">
    Add to `test/helpers/rbac-personas.ts`
  </Step>
  <Step title="Update seeder">
    Modify `test/setup/seed-test-users.ts` to create the user
  </Step>
  <Step title="Update context">
    Add authentication logic to `rbac-context.ts`
  </Step>
  <Step title="Update matrices">
    Add expected status codes for the new persona across all matrices
  </Step>
</Steps>

The testing methodology follows a systematic approach:

  </Card>
  <Card title="Scenario Mapping" icon="map">

## CI/CD Integration

The test suite is designed for CI/CD integration:

- **Fast feedback:** Singleton context reduces boot time across test files
- **Reliable cleanup:** Test users are isolated with `@propwise-test.com` domain
- **Coverage reporting:** Supports Jest coverage collection
- **Parallel execution:** Tests can run in parallel at the file level

<Note>
For CI environments, ensure the database is properly seeded and environment variables are configured before running the test suite.
</Note>
