The PropWise CRM testing framework uses API-driven test data creation and comprehensive cleanup strategies to ensure reliable, isolated integration tests. This approach ensures test data reflects real-world usage patterns while maintaining test independence.

## Overview

<Info>
Test data is created through actual API endpoints rather than direct database insertion, ensuring all business logic, guards, and stakeholder assignments execute naturally.
</Info>

The test data management system consists of:
- **User seeding** - One-time creation of test personas
- **Entity seeding** - Per-test-run creation of business entities 
- **Cleanup strategies** - Database cleanup and isolation
- **Caching** - Cross-file data reuse to improve performance

## Entity Test Data Creation

### API-Driven Seeding

Test entities are created through real API endpoints to ensure authentic data:

```typescript
// Example: Creating a lead through the API
const leadResponse = await request(app.getHttpServer())
  .post('/v1/leads')
  .set('Authorization', `Bearer ${personas.BASIC.accessToken}`)
  .send({
    firstName: 'Test',
    lastName: 'Lead',
    email: 'testlead@example.com',
    phone: '+1234567890'
  });
```

<Note>
API-driven creation ensures all guards, validation, auto-assignments, and stakeholder rules execute exactly as they would in production.
</Note>

### Caching Strategy

Test data is cached across test files for performance:

```typescript
// test/setup/rbac-test-data.ts
let cachedTestData: RbacTestData | null = null;

export async function seedRbacTestData(
  app: INestApplication,
  personas: Record<PersonaKey, TestUser>,
  teamId: string
): Promise<RbacTestData> {
  if (cachedTestData) {
    return cachedTestData; // Return cached data instantly
  }
  
  // Create fresh test data on first call
  cachedTestData = await createTestEntities(app, personas, teamId);
  return cachedTestData;
}
```

## Stakeholder Assignment Patterns

### Automatic Assignment

When entities are created through the API, stakeholders are automatically assigned based on business rules:

<CodeGroup>
```typescript Lead Creation
// User creates lead → becomes primary stakeholder with FULL access
POST /v1/leads
{
  "firstName": "Test",
  "lastName": "Lead"
}

// Result: Creator gets FULL access automatically
{
  "id": "uuid",
  "stakeholders": [
    {
      "userId": "creator-id", 
      "accessLevel": "FULL"
    }
  ]
}
```

```typescript Team Assignment  
// Owner creates lead and assigns team
POST /v1/leads/:id/stakeholders
{
  "teamId": "team-uuid",
  "accessLevel": "WRITE"
}

// Result: Team members get appropriate access
// - Team Leader: Can manage via team_crm.manage
// - Agents: No automatic access (team role dependent)
```
</CodeGroup>

### Access Level Testing

Different access levels are tested through stakeholder assignments:

| Access Level | Can View | Can Modify | Can Delete |
|-------------|----------|------------|------------|
| `READ` | ✅ | ❌ | ❌ |
| `WRITE` | ✅ | ✅ | ❌ |
| `FULL` | ✅ | ✅ | ✅ |

## Cleanup Strategies

### User Cleanup

<Warning>
The user cleanup command will delete ALL users with `@propwise-test.com` emails from the database.
</Warning>

```bash
# Clean up test users (use with caution)
pnpm test:cleanup-users
```

### Entity Cleanup

Entities are cleaned up automatically between test runs:

```typescript
// Automatic cleanup in rbac-test-data.ts
afterEach(async () => {
  await cleanupTestEntities();
});
```

### Email Verification Cleanup

The email spy helper provides cleanup for verification codes:

```typescript
// test/helpers/email-spy.ts  
export class EmailSpy {
  async cleanup(): Promise<void> {
    // Clear verification codes from database
    await this.orm.em.nativeDelete(VerificationCode, {});
  }
}
```

## Database Isolation

### Test Database Strategy

<Steps>
<Step title="Separate test database">
Use a dedicated test database to avoid conflicts with development data.

```env
# .env.test
DATABASE_URL=postgresql://user:pass@localhost:5432/propwise_test
```
</Step>

<Step title="Migration sync">
Ensure test database has current schema:

```bash
pnpm migration:up:test
```
</Step>

<Step title="RLS bypass for seeding">
Test user seeding bypasses Row Level Security for initial setup:

```typescript
// Temporarily disable RLS for seeding
await this.orm.em.execute('SET row_security = off');
await this.seedUsers();  
await this.orm.em.execute('SET row_security = on');
```
</Step>
</Steps>

### Transaction Management

Each test runs within isolated transactions:

```typescript
beforeEach(async () => {
  await this.orm.em.begin(); // Start transaction
});

afterEach(async () => {
  await this.orm.em.rollback(); // Rollback changes
});
```

## Performance Optimization

### Singleton Context

The NestJS app boots once and is shared across all test files:

```typescript
// First test file to run
beforeAll(async () => {
  ctx = await getContext(); // Boots app (~3-5s)
}, 60000);

// Subsequent test files  
beforeAll(async () => {
  ctx = await getContext(); // Returns cached instance (0ms)
});
```

### Data Reuse

Test entities are created once per test run and reused across test files:

<Tip>
This approach reduces test execution time from ~30 seconds to ~8 seconds while maintaining test isolation through proper cleanup.
</Tip>
