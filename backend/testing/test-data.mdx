## Overview

The PropWise test suite uses **API-driven seeding** to create realistic test data that respects all business logic, guards, and auto-assignment rules. Test entities are created through actual API endpoints rather than direct database writes, ensuring complete end-to-end validation.

<Note>
  Test data is created once per test run and cached across all test files for performance. The singleton context shares authenticated tokens and resolved references.
</Note>

## Seeding Architecture

### Two-Phase Approach

<Steps>
  <Step title="One-time user seeding">
    Run `pnpm test:seed-users` to create RBAC personas, organization, and team structure in the database
  </Step>
  
  <Step title="Per-run entity seeding">
    Test files automatically call `seedRbacTestData()` in `beforeAll` to create fixture entities via API
  </Step>
</Steps>

### Why API-Driven Seeding?

<CardGroup cols={2}>
  <Card title="Business Logic Execution" icon="gears">
    All guards, interceptors, and auto-assignment logic runs naturally
  </Card>
  
  <Card title="Stakeholder Auto-Creation" icon="users">
    Creating a lead as BASIC automatically assigns them as primary stakeholder with FULL access
  </Card>
  
  <Card title="Real-World Validation" icon="shield-check">
    Tests verify the same code paths that production clients use
  </Card>
  
  <Card title="Permission Enforcement" icon="lock">
    Entity creation respects RBAC rules — failures surface permission gaps early
  </Card>
</CardGroup>

## Step 2: Entity Seeding

### Automatic Invocation

Each test file calls the seeding function in its `beforeAll` hook:

```typescript
import { getContext, closeContext } from '../../setup/rbac-context';
import { seedRbacTestData } from '../../setup/rbac-test-data';

let ctx: RbacTestContext;
let testData: RbacTestData;

beforeAll(async () => {
  ctx = await getContext();
  testData = await seedRbacTestData(ctx.app, ctx.personas, ctx.teamId);
}, 60000);
```

### Caching Strategy

<Info>
  Entity seeding results are cached globally. The first test file triggers API creation (~2-3s), subsequent files receive instant cached references.
</Info>

```typescript
// Singleton cache in rbac-test-data.ts
let cachedTestData: RbacTestData | null = null;

export async function seedRbacTestData(
  app: INestApplication,
  personas: Record<PersonaKey, TestUser>,
  teamId: string,
): Promise<RbacTestData> {
  if (cachedTestData) {
    return cachedTestData;
  }
  
  // Create entities via API...
  cachedTestData = { leads, deals, contacts, companies };
  return cachedTestData;
}
```

    | TEAM_LEADER | ❌ 403 | Not a stakeholder, no org perm |

## Returned Test Data Structure

```typescript
export interface RbacTestData {
  leads: {
    leadByBasicId: string;
    leadByOwnerId: string;
    leadInPoolId: string;
    leadOnTeamId: string;
    leadAgentReadId: string;
  };
  deals: {
    dealByOwnerId: string;
  };
  contacts: {
    contactByBasicId: string;
    contactByOwnerId: string;
  };
  companies: {
    companyByBasicId: string;
    companyByOwnerId: string;
  };
}
```

### Usage in Tests

```typescript
describe('Lead RBAC', () => {
  let ctx: RbacTestContext;
  let testData: RbacTestData;

  beforeAll(async () => {
    ctx = await getContext();
    testData = await seedRbacTestData(ctx.app, ctx.personas, ctx.teamId);
  });

  it('should allow BASIC to view their own lead', async () => {
    const response = await request(ctx.app.getHttpServer())
      .get(`/v1/leads/${testData.leads.leadByBasicId}`)
      .set('Authorization', `Bearer ${ctx.personas.BASIC.tenantToken}`);
    
    expect(response.status).toBe(200);
  });
});
```

## Seeding Best Practices

<CardGroup cols={2}>
  <Card title="Use Descriptive Names" icon="tag">

### When to Create New Fixtures

<Steps>
  <Step title="Identify the permission scenario">
    Does an existing fixture cover this access pattern?
  </Step>
  
  <Step title="Add to rbac-test-data.ts">
    Create the entity via API using the appropriate persona's token
  </Step>
  
  <Step title="Update RbacTestData interface">
    Add the new entity ID to the return type
  </Step>
  
  <Step title="Create corresponding matrix">
    Define expected status codes per persona in `matrices/`
  </Step>
</Steps>

## Cleanup

### Automatic Cleanup

Test users and entities persist across test runs for performance. The singleton context keeps the app alive across files.

### Manual Cleanup

To delete all test users and their data:

```bash
pnpm test:cleanup-users
```

This runs `test/setup/cleanup-test-users.ts` which:
- Deletes all users with `@propwise-test.com` emails
- Cascades to delete their organizations, teams, and entities
- Uses RLS bypass for direct database access

<Warning>
  Cleanup is destructive and cannot be undone. Only run in test/development environments.
</Warning>

## Troubleshooting

<AccordionGroup>
  <Accordion title="ERROR: Subscription plans not found">
    **Cause:** Database is missing required subscription plan seed data
    
    **Fix:**
    ```bash
    pnpm test:init-data  # Run once on fresh DB
    pnpm test:seed-users  # Then seed test users
    ```
  </Accordion>
  
  <Accordion title="ERROR: 403 Forbidden on entity creation">
    **Cause:** Creating persona lacks required permission
    
    **Fix:** Check persona permissions in `rbac-personas.ts`. For pool leads or admin-only entities, create as OWNER.
  </Accordion>
  
  <Accordion title="Timeout on seedRbacTestData()">
    **Cause:** API calls taking longer than 60s (network/DB issue)
    
    **Fix:** Check PostgreSQL connection, increase `beforeAll` timeout, or verify seed-users completed successfully
  </Accordion>
  
  <Accordion title="Entities created but tests still fail">
    **Cause:** Entity IDs cached but test expectations changed
    
    **Fix:**
    ```bash
    pnpm test:cleanup-users  # Clear old data
    pnpm test:seed-users     # Re-seed with current logic
    pnpm test:e2e            # Run tests with fresh fixtures
    ```
  </Accordion>
</AccordionGroup>
