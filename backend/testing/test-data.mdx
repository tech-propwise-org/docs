The PropWise test suite uses **API-driven seeding** to create test fixtures. All test data is created through actual API endpoints, ensuring business logic (guards, auto-assignment, stakeholder creation) executes naturally.

## Overview

Test data seeding happens in two stages:

<Steps>
  <Step title="One-time user setup">
    Seed RBAC personas, organization, team, and subscription data directly to the database
  </Step>
  <Step title="Per-test-run fixtures">
    Create entity fixtures (leads, deals, contacts, companies) through API endpoints with cached results
  </Step>
</Steps>

<Note>
API-driven seeding validates that your business logic works correctly during test setup. If seeding fails, your tests catch real bugs in entity creation flows.
</Note>

## User Seeding (One-Time Setup)

### Running the Seeder

Before running tests for the first time, seed the RBAC personas:

```bash
pnpm test:seed-users
```

<Info>
The seeder is **idempotent** — it skips users that already exist and self-heals stale subscription data.
</Info>

### Self-Healing Behavior

The seeder automatically fixes stale data from previous test runs:

- Upgrades FREE subscriptions to BUSINESS
- Updates incomplete `organization_setup_progress` records
- Ensures all personas have correct org and team memberships

<Warning>
Run `pnpm test:init-data` once on a fresh database to create subscription plans before running `pnpm test:seed-users`.
</Warning>

### Implementation Details

The seeder uses **RLS bypass** to write directly via MikroORM:

```typescript
// Example from seed-test-users.ts
const em = app.get(EntityManager);
await em.transactional(async (em) => {
  const user = em.create(User, {
    email: 'owner@propwise.com',
    password: hashedPassword,
    isVerified: true,
  });
  
  const org = em.create(Organization, {
    name: 'Test Organization',
    owner: user,
  });
  
  // Create BUSINESS subscription
  const subscription = em.create(Subscription, {
    organization: org,
    plan: businessPlan,
    status: 'ACTIVE',
  });
  
  await em.flush();
});
```

## Test Data Seeding (Per-Test-Run)

### Automatic Fixture Creation

Each test file calls `seedRbacTestData()` in its `beforeAll` hook:

```typescript
import { seedRbacTestData } from '../../setup/rbac-test-data';
import { getContext, closeContext } from '../../setup/rbac-context';

let ctx: RbacTestContext;
let testData: any;

beforeAll(async () => {
  ctx = await getContext();
  testData = await seedRbacTestData(ctx.app, ctx.personas, ctx.teamId);
}, 60000);

afterAll(async () => {
  await closeContext();
});
```

<Tip>
Results are **cached across test files**. The first file to call `seedRbacTestData()` creates the entities; subsequent files reuse the same data instantly.
</Tip>

### Seeding Flow

The seeder authenticates each persona and calls API endpoints:

```typescript
// Simplified example from rbac-test-data.ts
export async function seedRbacTestData(
  app: INestApplication,
  personas: Record<PersonaKey, TestUser>,
  teamId: string,
) {
  if (testDataCache) return testDataCache;
  
  const basicToken = personas.BASIC.tenantToken;
  const ownerToken = personas.OWNER.tenantToken;
  
  // Create lead by BASIC (auto-assigned as stakeholder)
  const leadByBasicRes = await request(app.getHttpServer())
    .post('/v1/leads')
    .set('Authorization', `Bearer ${basicToken}`)
    .send({
      firstName: 'Test',
      lastName: 'Lead Basic',
      email: 'lead.basic@test.com',
      phone: '1234567890',
    });
  
  // Create lead in pool (no stakeholders)
  const leadInPoolRes = await request(app.getHttpServer())
    .post('/v1/leads')
    .set('Authorization', `Bearer ${ownerToken}`)
    .send({
      firstName: 'Pool',
      lastName: 'Lead',
      email: 'pool.lead@test.com',
    });
  
  // Create lead with team stakeholder
  const leadWithTeamRes = await request(app.getHttpServer())
    .post('/v1/leads')
    .set('Authorization', `Bearer ${ownerToken}`)
    .send({
      firstName: 'Team',
      lastName: 'Lead',
      email: 'team.lead@test.com',
    });
  
  // Assign team as stakeholder
  await request(app.getHttpServer())
    .post(`/v1/leads/${leadWithTeamRes.body.id}/stakeholders`)
    .set('Authorization', `Bearer ${ownerToken}`)
    .send({
      teamId: teamId,
      accessLevel: 'FULL',
    });
  
  testDataCache = {
    leadByBasic: leadByBasicRes.body,
    leadInPool: leadInPoolRes.body,
    leadWithTeam: leadWithTeamRes.body,
    // ... more entities
  };
  
  return testDataCache;
}
```

<Check>
All entity creation goes through the actual API, including:
- ValidationPipe transformations
- DTO validation
- Auth guards
- Permission checks
- Auto-assignment business logic
</Check>

## Path Placeholder Resolution

The RBAC matrix runner automatically resolves entity IDs:

```typescript
// Matrix definition
{
  method: 'GET',
  path: '/v1/leads/:leadByBasicId',
  description: 'View lead detail',
  expectations: {
    OWNER: 200,
    ADMIN: 200,
    BASIC: 200,
    AGENT: 403,
    TEAM_LEADER: 403,
  },
}

// Runner resolves :leadByBasicId → actual UUID from testData.leadByBasic.id
```

## Cleanup

### During Test Runs

Entity cleanup happens automatically via foreign key cascades when test users are deleted.

### Manual Cleanup

Remove all test personas and their data:

```bash
pnpm test:cleanup-users
```

<Warning>
This permanently deletes:
- All 5 RBAC personas
- Test Organization (cascades to teams, subscriptions, setup progress)
- All entities created by test users (leads, deals, contacts, companies)

Only run this on test databases.
</Warning>

The cleanup script (`test/setup/cleanup-test-users.ts`) uses RLS bypass to delete users by email domain:

```typescript
const testUsers = await em.find(User, {
  email: { $like: '%@propwise.com' }
});

for (const user of testUsers) {
  await em.removeAndFlush(user);
}
```

## Best Practices

<Steps>
  <Step title="Seed users once">
    Run `pnpm test:seed-users` after database migrations. Reuse the same personas across test runs.
  </Step>
  
  <Step title="Let seeding validate business logic">
    If `seedRbacTestData()` fails, you've found a bug in entity creation. Don't bypass failures with direct DB writes.
  </Step>
  
  <Step title="Use cached data">
    Don't create new entities in individual tests unless testing creation specifically. Reuse `testData` fixtures.
  </Step>
  
  <Step title="Clean up on CI">
    Add `pnpm test:cleanup-users` to your CI teardown to prevent test data accumulation.
  </Step>
</Steps>
