# Testing Specification v14

Architecture, conventions, and run guide for the PropWise CRM backend test suite.

## Overview

The test suite focuses on **RBAC (Role-Based Access Control) E2E testing** — verifying that every API endpoint enforces the correct permissions for every user role.

**Stack:** Jest + supertest + `@nestjs/testing`

<Note>
**Key design decisions:**
- **Singleton app context** — the NestJS app boots once and is shared across all test files
- **Matrix-driven tests** — a declarative permission matrix defines expected HTTP status codes per persona × endpoint
- **API-driven seeding** — test data is created through the actual API endpoints so all business logic executes naturally
</Note>

## Architecture

### Singleton Context

The app boots once for the entire test run. All test files share the same NestJS app instance, authenticated tokens, and resolved `teamId`.

<Steps>
<Step title="Jest starts">
`lead-rbac.e2e-spec.ts` calls `getContext()` - first call boots NestJS app (~3-5s), logs in 5 personas, resolves teamId
</Step>
<Step title="Subsequent files">
`deal-rbac.e2e-spec.ts` calls `getContext()` - returns cached instance instantly (0ms)
</Step>
<Step title="Cleanup">
Last file's `afterAll()` calls `closeContext()` - reference count hits 0, app shuts down
</Step>
</Steps>

**Implementation example:**

```typescript
import { getContext, closeContext, RbacTestContext } from '../../setup/rbac-context';

let ctx: RbacTestContext;
beforeAll(async () => {
  ctx = await getContext();
}, 60000);
afterAll(async () => {
  await closeContext();
});
```

<Info>
The context replicates production config from `main.ts`:
- `ValidationPipe` (whitelist, transform, implicit conversion)
- URI versioning (`/v1/...`)
- Cookie parser
</Info>

### Matrix Runner

Instead of writing individual `it()` blocks, you define expectations declaratively:

```typescript
{
  method: 'GET',
  path: '/v1/leads/:leadByBasicId',
  description: 'View lead detail',
  expectations: {
    OWNER: 200,
    ADMIN: 200,
    TEAM_LEADER: 403,
    AGENT: 403,
    BASIC: 200,
  },
}
```

<Tabs>
<Tab title="Matrix Processing">
The runner calls `runRbacMatrix()` which:
1. Iterates every endpoint in the matrix
2. For each endpoint, iterates all 5 personas
3. Resolves path placeholders (`:leadByBasicId` → actual UUID)
4. Makes the HTTP request with the persona's tenant token
5. Asserts the response status matches the expected value
6. Produces clear failure messages with the actual response body
</Tab>
<Tab title="Test Generation">
One matrix entry = 5 generated test cases automatically.
</Tab>
</Tabs>

### Permission Profiles

<AccordionGroup>
<Accordion title="OWNER / ADMIN">
Bypass resource-level checks via org permissions (`crm.view`, `crm.manage`, `sales.view`, `sales.manage`, etc.). Can access any entity in the org.
</Accordion>
<Accordion title="TEAM_LEADER">
No org permissions, but has `team_crm.manage` and `team_sales.manage` in the team. Can access entities where the team is a stakeholder.
</Accordion>
<Accordion title="AGENT">
No org permissions, no team permissions. Can only access entities where they are personally a stakeholder. Used to test access level differentiation (READ vs WRITE vs FULL).
</Accordion>
<Accordion title="BASIC">
Same as AGENT but not a team member at all. Access is purely stakeholder-based.
</Accordion>
</AccordionGroup>

### Login Flow

The context performs a two-step login for each persona:

<Steps>
<Step title="Identity Login">
`POST /v1/auth/login` → identity token
</Step>
<Step title="Organization Selection">
`POST /v1/auth/select-organization` → tenant token (org-scoped JWT)
</Step>
<Step title="API Calls">
All subsequent API calls use the tenant token in the `Authorization: Bearer` header
</Step>
</Steps>

## Permission Matrix

Matrices are split into per-entity files under `test/setup/matrices/`. The barrel at `test/setup/rbac-matrix.ts` re-exports everything.

### Lead Matrices

<AccordionGroup>
<Accordion title="LEAD_BY_BASIC_MATRIX">
**Entity:** Lead created by BASIC
**Who Can Access:** OWNER, ADMIN (via `crm.*`), BASIC (via FULL stakeholder)
**Tests:** View, update, stakeholders, activities, property interests
</Accordion>
<Accordion title="LEAD_ON_TEAM_MATRIX">
**Entity:** Lead with team as stakeholder
**Who Can Access:** OWNER, ADMIN, TEAM_LEADER (via `team_crm.*`)
**Tests:** View, update, stakeholders, activities
</Accordion>
<Accordion title="LEAD_AGENT_READ_MATRIX">
**Entity:** Lead where AGENT has READ access
**Who Can Access:** OWNER, ADMIN, AGENT (view only — modify denied)
**Tests:** View allowed, update denied, access level enforcement
</Accordion>
<Accordion title="LEAD_IN_POOL_MATRIX">
**Entity:** Lead with no stakeholders
**Who Can Access:** OWNER, ADMIN only (via `crm.*`)
**Tests:** View, update
</Accordion>
<Accordion title="LEAD_ADMIN_ONLY_MATRIX">
**Entity:** Lead for admin-only ops
**Who Can Access:** OWNER, ADMIN (via `system.admin`)
**Tests:** Delete
</Accordion>
</AccordionGroup>

### Access Mode Reference

<Note>
Your `@CheckAccess` decorator uses two modes:
- **OR (default):** User has the required permission **OR** has stakeholder-level access on the resource → allowed
- **AND:** User must have the permission **AND** stakeholder access → allowed
</Note>

**Resource action hierarchy:**
- `ACCESS` — granted by `READ`, `WRITE`, or `FULL` access level
- `MODIFY` — granted by `WRITE` or `FULL`
- `DELETE` — granted by `FULL` only

## Test Data Seeding

### Step 1: Seed Users (One-time Setup)

<CodeGroup>
```bash Terminal
pnpm test:seed-users
```
</CodeGroup>

This runs `test/setup/seed-test-users.ts` which:

<Steps>
<Step title="Create Personas">
Creates the RBAC test personas in the database (idempotent — skips if they exist)
</Step>
<Step title="Create Organization">
Creates "Test Organization" with owner
</Step>
<Step title="Create Team">
Creates "Test Sales Team" with team leader and agent
</Step>
<Step title="Assign Roles">
Assigns org roles (Owner, Admin, Basic) and team roles (Team Leader, Agent)
</Step>
<Step title="Database Write">
Uses RLS bypass to write directly via MikroORM
</Step>
</Steps>

### Step 2: Seed Test Data (Automatic)

Each test file calls `seedRbacTestData(app, personas, teamId)` in its `beforeAll`. Results are cached across files.

<Warning>
Because seeding goes through the real API endpoints, all business logic (guards, auto-assignment, stakeholder creation) executes naturally.
</Warning>

**Entities Created:**

<Tabs>
<Tab title="Leads">
- One by BASIC (auto-assigned as primary stakeholder with FULL access)
- One by OWNER (for admin-only tests)
- One in pool (empty stakeholders)
- One with team as stakeholder
- One with AGENT as READ-only stakeholder
</Tab>
<Tab title="Contacts & Companies">
- One contact by BASIC, one by OWNER
- One company by BASIC, one by OWNER
</Tab>
<Tab title="Deals & Events">
- One deal by OWNER (from lead + contact)
- One event linked to the OWNER lead
- One event with BASIC as an invitee
</Tab>
<Tab title="Escalation Rules">
- One OWNER-created rule for escalation-rule RBAC matrices
</Tab>
</Tabs>

## How to Run

<Steps>
<Step title="Seed Test Users">
```bash
pnpm test:seed-users
```
</Step>
<Step title="Run Core E2E Tests">
```bash
pnpm test:e2e
```
</Step>
<Step title="Run Queue Tests (Optional)">
```bash
pnpm test:e2e:queue
```
</Step>
<Step title="Clean Up (Optional)">
```bash
pnpm test:cleanup-users
```
</Step>
</Steps>

<Tip>
The cleanup step is optional since test users are isolated with `@propwise-test.com` emails and don't interfere with production data.
</Tip>

## How to Extend

### Adding New Endpoints

<Steps>
<Step title="Update Matrix">
Add endpoint expectations to the appropriate matrix file in `test/setup/matrices/`
</Step>
<Step title="Path Resolution">
If using new path parameters, update the path resolution logic in `rbac-runner.ts`
</Step>
<Step title="Test Data">
If testing new entities, add creation logic to `rbac-test-data.ts`
</Step>
</Steps>

### Adding New Personas

<Steps>
<Step title="Define Persona">
Add to `test/helpers/rbac-personas.ts`
</Step>
<Step title="Update Seeding">
Modify `seed-test-users.ts` to create the new user
</Step>
<Step title="Update Context">
Add authentication logic to `rbac-context.ts`
</Step>
<Step title="Update Matrices">
Add expectations for the new persona in all relevant matrices
</Step>
</Steps>

## Methodology

<Check>
Before writing tests, analyze controllers and DTOs following the methodology in `test/TESTING_METHODOLOGY.md`
</Check>

Key principles:
- Test permission enforcement, not business logic
- Use matrix-driven approach for consistency
- Verify both positive and negative cases
- Test stakeholder access levels (READ, WRITE, FULL)

## CI/CD Integration

<Info>
The test suite is designed for CI/CD environments with the following considerations:
</Info>

### Database Requirements

- Clean PostgreSQL instance for each test run
- Test users are isolated with `@propwise-test.com` domain
- RLS policies must be configured

### Environment Variables

```env
DATABASE_URL=postgresql://user:pass@localhost:5432/propwise_test
JWT_SECRET=test-secret
BCRYPT_ROUNDS=4  # Faster hashing for tests
```
