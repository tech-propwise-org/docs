The Permission Matrix is a declarative testing framework that automatically generates RBAC (Role-Based Access Control) test cases by defining expected HTTP status codes for each persona-endpoint combination.

## Overview

Instead of manually writing individual test cases, the Permission Matrix allows you to define access expectations declaratively. A single matrix entry generates 5 test cases automatically - one for each persona.

<Info>
The matrix-driven approach ensures comprehensive RBAC coverage while maintaining consistency and reducing test maintenance overhead.
</Info>

## Matrix Structure

Each matrix is defined as an array of `EndpointExpectation` objects:

```typescript
{
  method: 'GET',
  path: '/v1/leads/:leadByBasicId',
  description: 'View lead detail',
  expectations: {
    OWNER: 200,
    ADMIN: 200,
    TEAM_LEADER: 403,
    AGENT: 403,
    BASIC: 200,
  },
}
```

### Matrix Components

<AccordionGroup>
<Accordion title="Method & Path">
- `method`: HTTP method (GET, POST, PATCH, DELETE)
- `path`: API endpoint with placeholders (`:leadByBasicId`, `:dealByOwnerId`)
- Placeholders are resolved to actual UUIDs during test execution
</Accordion>

<Accordion title="Description">
Human-readable test description that appears in Jest output
</Accordion>

<Accordion title="Expectations">
Map of persona keys to expected HTTP status codes:
- `200`: Success (authorized)
- `403`: Forbidden (no access)
- `404`: Not found (common for unauthorized resources)
- `400`: Bad request (validation errors)
</Accordion>
</AccordionGroup>

## Matrix Runner

The `rbac-runner.ts` automatically converts matrix definitions into Jest test cases:

<Steps>
<Step title="Matrix Processing">
Iterates through every endpoint in the matrix
</Step>

<Step title="Persona Testing">
For each endpoint, tests all 5 personas
</Step>

<Step title="Path Resolution">
Resolves placeholders (`:leadByBasicId` → actual UUID)
</Step>

<Step title="HTTP Execution">
Makes authenticated requests with persona's tenant token
</Step>

<Step title="Assertion">
Asserts response status matches expected value
</Step>
</Steps>

### Usage Example

```typescript
import { runRbacMatrix } from '../../setup/rbac-runner';
import { LEAD_BY_BASIC_MATRIX } from '../../setup/matrices/lead-matrix';

describe('Lead RBAC (E2E)', () => {
  describe('Lead owned by BASIC user', () => {
    runRbacMatrix(LEAD_BY_BASIC_MATRIX, () => ctx);
  });
});
```

<Note>
One matrix entry generates 5 individual `it()` blocks - one for each persona's expected behavior.
</Note>

## Access Level Reference

The system uses a hierarchical access level model:

<Tabs>
<Tab title="Resource Actions">
- **ACCESS** — granted by `READ`, `WRITE`, or `FULL` access level
- **MODIFY** — granted by `WRITE` or `FULL` access level  
- **DELETE** — granted by `FULL` access level only
</Tab>

<Tab title="Check Modes">
**OR Mode (default):**
User has required permission **OR** stakeholder access → allowed

**AND Mode:**  
User must have permission **AND** stakeholder access → allowed
</Tab>
</Tabs>
