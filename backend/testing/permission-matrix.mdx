The permission matrix is a declarative testing framework that defines expected HTTP status codes for every API endpoint across all user roles. Instead of manually writing individual test cases, you define access expectations in a structured format, and the test runner automatically generates parameterized Jest tests.

## Overview

The matrix-driven approach provides:

- **Declarative permissions** — define who can access what in a single source of truth
- **Automatic test generation** — one matrix entry produces 5 test cases (one per persona)
- **Clear failure messages** — shows actual response body when expectations don't match
- **Comprehensive coverage** — ensures every endpoint is tested for every role

<Info>
  One matrix entry = 5 generated test cases, covering all RBAC personas (OWNER, ADMIN, TEAM_LEADER, AGENT, BASIC).
</Info>

## Matrix Structure

Each matrix is an array of endpoint definitions. Here's the anatomy of a matrix entry:

```typescript
{
  method: 'GET',
  path: '/v1/leads/:leadByBasicId',
  description: 'View lead detail',
  expectations: {
    OWNER: 200,
    ADMIN: 200,
    TEAM_LEADER: 403,
    AGENT: 403,
    BASIC: 200,
  },
}
```

### Fields

<AccordionGroup>
  <Accordion title="method">
    HTTP method for the endpoint. Supports `GET`, `POST`, `PATCH`, `PUT`, `DELETE`.
  </Accordion>

  <Accordion title="path">
    API endpoint path with URI versioning (`/v1/...`). Supports placeholders like `:leadByBasicId` which are automatically resolved to actual UUIDs from seeded test data.
  </Accordion>

  <Accordion title="description">
    Human-readable description of what the endpoint does. Used in test case names for clear output.
  </Accordion>

  <Accordion title="expectations">
    Object mapping each persona to their expected HTTP status code:
    - `200` — Success (access granted)
    - `403` — Forbidden (no permission)
    - `404` — Not found (resource doesn't exist for this user's scope)
    - `401` — Unauthorized (authentication required)
  </Accordion>
</AccordionGroup>

## Matrix Runner

The `rbac-runner.ts` module processes matrices and generates Jest test cases.

### Usage

```typescript
import { runRbacMatrix } from '../../setup/rbac-runner';
import { LEAD_BY_BASIC_MATRIX } from '../../setup/rbac-matrix';

describe('Lead RBAC - Lead by BASIC', () => {
  runRbacMatrix(
    LEAD_BY_BASIC_MATRIX,
    ctx.app,
    ctx.personas,
    testData,
  );
});
```

### How It Works

<Steps>
  <Step title="Iterate endpoints">
    The runner loops through each endpoint definition in the matrix.
  </Step>

  <Step title="Iterate personas">
    For each endpoint, loops through all 5 RBAC personas (OWNER, ADMIN, TEAM_LEADER, AGENT, BASIC).
  </Step>

  <Step title="Resolve placeholders">
    Replaces path placeholders like `:leadByBasicId` with actual UUIDs from seeded test data.
  </Step>

  <Step title="Make HTTP request">
    Executes the request using supertest with the persona's tenant token in the `Authorization: Bearer` header.
  </Step>

  <Step title="Assert status code">
    Compares the actual response status with the expected status from the matrix. If they don't match, displays the full response body for debugging.
  </Step>
</Steps>

### Generated Test Names

Test names follow the pattern:

```
[METHOD] [PATH] - [DESCRIPTION] - [PERSONA] - should return [STATUS]
```

Example:

```
GET /v1/leads/:leadByBasicId - View lead detail - OWNER - should return 200
GET /v1/leads/:leadByBasicId - View lead detail - AGENT - should return 403
```

## Permission Expectations

The matrix expectations correspond to your RBAC implementation's access control modes.

### Access Control Modes

<Tabs>
  <Tab title="OR Mode (Default)">
    User needs **either** the required permission **OR** stakeholder-level access.

    ```typescript
    @CheckAccess('crm', 'manage', 'lead')
    // Allowed if:
    // - User has crm.manage permission, OR
    // - User has WRITE/FULL stakeholder access on the lead
    ```

    Most endpoints use OR mode for flexibility.
  </Tab>

  <Tab title="AND Mode">
    User needs **both** the required permission **AND** stakeholder access.

    ```typescript
    @CheckAccess('crm', 'manage', 'lead', 'AND')
    // Allowed if:
    // - User has crm.manage permission, AND
    // - User has WRITE/FULL stakeholder access on the lead
    ```

    AND mode is rarely used (e.g., sensitive operations requiring dual authorization).
  </Tab>
</Tabs>

### Access Level Hierarchy

Stakeholder access levels follow a hierarchical model:

<CodeGroup>
```typescript Resource Actions
// Access hierarchy (each level grants previous levels)
ACCESS  → granted by READ, WRITE, or FULL
MODIFY  → granted by WRITE or FULL
DELETE  → granted by FULL only
```

```typescript Example Matrix
{
  method: 'GET',
  path: '/v1/leads/:leadAgentReadId',
  expectations: {
    AGENT: 200,  // Has READ access → can view
  },
},
{
  method: 'PATCH',
  path: '/v1/leads/:leadAgentReadId',
  expectations: {
    AGENT: 403,  // Has READ access → cannot modify
  },
}
```
</CodeGroup>

## Common Patterns

### Pattern 1: Org-Level Access

Endpoints requiring org-level permissions (OWNER/ADMIN only):

```typescript
{
  method: 'DELETE',
  path: '/v1/leads/:leadByBasicId',
  description: 'Delete lead',
  expectations: {
    OWNER: 200,        // Has system.owner
    ADMIN: 200,        // Has system.admin
    TEAM_LEADER: 403,  // No org permission
    AGENT: 403,        // No org permission
    BASIC: 403,        // No org permission
  },
}
```

### Pattern 2: Team-Level Access

Endpoints where team permissions grant access:

```typescript
{
  method: 'GET',
  path: '/v1/leads/:leadOnTeamId',
  description: 'View lead on team',
  expectations: {
    OWNER: 200,        // Has crm.view
    ADMIN: 200,        // Has crm.view
    TEAM_LEADER: 200,  // Has team_crm.view + team is stakeholder
    AGENT: 403,        // In team but no team permission
    BASIC: 403,        // Not in team
  },
}
```

### Pattern 3: Stakeholder-Only Access

Endpoints relying purely on stakeholder relationships:

```typescript
{
  method: 'GET',
  path: '/v1/leads/:leadByBasicId',
  description: 'View own lead',
  expectations: {
    OWNER: 200,        // Has crm.view
    ADMIN: 200,        // Has crm.view
    TEAM_LEADER: 403,  // No org perm, not stakeholder
    AGENT: 403,        // Not stakeholder
    BASIC: 200,        // Is FULL stakeholder
  },
}
```

### Pattern 4: Access Level Differentiation

Testing read vs. write stakeholder access:

```typescript
// View endpoint (READ access sufficient)
{
  method: 'GET',
  path: '/v1/leads/:leadAgentReadId',
  expectations: {
    AGENT: 200,  // Has READ access
  },
},
// Modify endpoint (WRITE access required)
{
  method: 'PATCH',
  path: '/v1/leads/:leadAgentReadId',
  expectations: {
    AGENT: 403,  // READ < WRITE (insufficient)
  },
}
```

## Creating a New Matrix

<Steps>
  <Step title="Create the matrix file">
    Add a new file under `test/setup/matrices/`:

    ```typescript matrices/task-matrix.ts
    import { EndpointExpectation } from '../rbac-matrix-types';

    export const TASK_BY_BASIC_MATRIX: EndpointExpectation[] = [
      {
        method: 'GET',
        path: '/v1/tasks/:taskByBasicId',
        description: 'View task detail',
        expectations: {
          OWNER: 200,
          ADMIN: 200,
          TEAM_LEADER: 403,
          AGENT: 403,
          BASIC: 200,
        },
      },
      // ... more endpoints
    ];
    ```
  </Step>

  <Step title="Export from index">
    Add to `test/setup/matrices/index.ts`:

    ```typescript
    export * from './task-matrix';
    ```
  </Step>

  <Step title="Seed test data">
    Add task creation in `test/setup/rbac-test-data.ts`:

    ```typescript
    // Create task by BASIC
    const taskByBasic = await request(app.getHttpServer())
      .post('/v1/tasks')
      .set('Authorization', `Bearer ${personas.BASIC.tenantToken}`)
      .send({ title: 'Test Task' });

    testData.taskByBasicId = taskByBasic.body.id;
    ```
  </Step>

  <Step title="Create test file">
    Add `test/e2e/rbac/task-rbac.e2e-spec.ts`:

    ```typescript
    import { runRbacMatrix } from '../../setup/rbac-runner';
    import { TASK_BY_BASIC_MATRIX } from '../../setup/rbac-matrix';

    describe('Task RBAC', () => {
      let ctx: RbacTestContext;
      let testData: RbacTestData;

      beforeAll(async () => {
        ctx = await getContext();
        testData = await seedRbacTestData(ctx.app, ctx.personas, ctx.teamId);
      }, 60000);

      afterAll(async () => {
        await closeContext();
      });

      describe('Task by BASIC', () => {
        runRbacMatrix(TASK_BY_BASIC_MATRIX, ctx.app, ctx.personas, testData);
      });
    });
    ```
  </Step>
</Steps>

## Debugging Matrix Failures

When a test fails, the runner displays the full response body:

```bash
Expected: 200
Received: 403

Response body:
{
  "statusCode": 403,
  "message": "You do not have permission to perform this action",
  "error": "Forbidden"
}
```

<Tip>
  Check the `@CheckAccess` decorator on the endpoint to understand which permission it requires. Then verify if the persona has that permission or stakeholder access.
</Tip>

### Common Failure Reasons

<AccordionGroup>
  <Accordion title="Expected 200, got 403">
    The persona lacks both:
    - The required org/team permission
    - Appropriate stakeholder access level

    **Fix:** Either add the permission to the persona's role, or ensure they're a stakeholder on the resource.
  </Accordion>

  <Accordion title="Expected 403, got 200">
    The persona has access they shouldn't have. Common causes:
    - Persona has an org/team permission they shouldn't
    - Persona is unexpectedly a stakeholder
    - Endpoint is missing `@CheckAccess` decorator

    **Fix:** Review the persona's permissions and stakeholder assignments. Add missing guard if needed.
  </Accordion>

  <Accordion title="Expected 200, got 404">
    The resource exists but isn't visible to the persona due to RLS (Row-Level Security) filtering.

    **Fix:** Ensure the resource is in the correct org and the persona's scope includes it.
  </Accordion>

  <Accordion title="Path placeholder not resolved">
    The placeholder (e.g., `:taskByBasicId`) doesn't match any key in the test data object.

    **Fix:** Add the corresponding property to `RbacTestData` interface and seed it in `rbac-test-data.ts`.
  </Accordion>
</AccordionGroup>

## Best Practices

<Check>
  **Group related endpoints** — create separate matrices for different access scenarios (by creator, by team, pool, read-only, etc.)
</Check>

<Check>
  **Test access level granularity** — for stakeholder-based access, create matrices that test READ vs. WRITE vs. FULL differentiation
</Check>

<Check>
  **Use descriptive matrix names** — name matrices after the access pattern they test (e.g., `LEAD_AGENT_READ_MATRIX` clearly indicates it tests READ-only access)
</Check>

<Check>
  **Keep descriptions concise** — endpoint descriptions should fit in one line and clearly state the operation
</Check>

<Warning>
  Don't mix unrelated endpoints in one matrix. If access expectations differ significantly, split into separate matrices for clarity.
</Warning>

## Reference

### Types

```typescript rbac-matrix-types.ts
export interface StatusExpectation {
  OWNER: number;
  ADMIN: number;
  TEAM_LEADER: number;
  AGENT: number;
  BASIC: number;
}

export interface EndpointExpectation {
  method: 'GET' | 'POST' | 'PATCH' | 'PUT' | 'DELETE';
  path: string;
  description: string;
  expectations: StatusExpectation;
  body?: Record<string, any>;  // Optional request body for POST/PATCH/PUT
}
```

### Available Placeholders

Placeholders are resolved from the `RbacTestData` object returned by `seedRbacTestData()`:

| Placeholder | Entity | Creator/Owner |
|-------------|--------|---------------|
| `:leadByBasicId` | Lead | BASIC (FULL stakeholder) |
| `:leadByOwnerId` | Lead | OWNER |
| `:leadInPoolId` | Lead | No stakeholders |
| `:leadOnTeamId` | Lead | Team stakeholder |
| `:leadAgentReadId` | Lead | AGENT (READ stakeholder) |
| `:dealByOwnerId` | Deal | OWNER |
| `:contactByBasicId` | Contact | BASIC |
| `:contactByOwnerId` | Contact | OWNER |
| `:companyByBasicId` | Company | BASIC |
| `:companyByOwnerId` | Company | OWNER |

<Note>
  Add new placeholders by extending the `RbacTestData` interface in `rbac-test-data.ts` and seeding the corresponding entities.
</Note>
