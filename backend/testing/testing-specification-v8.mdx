# Testing Specification v8

Architecture, conventions, and run guide for the PropWise CRM backend test suite.

## Overview

The test suite focuses on **RBAC (Role-Based Access Control) E2E testing** — verifying that every API endpoint enforces the correct permissions for every user role.

<Info>
**Stack:** Jest + supertest + `@nestjs/testing`
</Info>

### Key Design Decisions

- **Singleton app context** — the NestJS app boots once and is shared across all test files (equivalent to NestJS `TestSuite` pattern, implemented manually since `TestSuite` requires NestJS 11+)
- **Matrix-driven tests** — a declarative permission matrix defines expected HTTP status codes per persona × endpoint. A runner generates `it()` blocks automatically
- **API-driven seeding** — test data is created through the actual API endpoints so all business logic (guards, auto-assignment, stakeholder creation) executes naturally

## Architecture

### Singleton Context

The app boots once for the entire test run. All test files share the same NestJS app instance, authenticated tokens, and resolved `teamId`.

<Tip>
This pattern significantly reduces test execution time by avoiding repeated app bootstrapping across test files.
</Tip>

**Flow:**
1. Jest starts
2. `lead-rbac.e2e-spec.ts` calls `getContext()`
   - First call: boots NestJS app (~3-5s), logs in 5 personas, resolves teamId
   - Returns `{ app, personas, teamId }`
3. `deal-rbac.e2e-spec.ts` calls `getContext()`
   - Returns cached instance instantly (0ms)
4. Last file's `afterAll()` calls `closeContext()`
   - Reference count hits 0 → app shuts down

**Implementation:**

```typescript
import { getContext, closeContext, RbacTestContext } from '../../setup/rbac-context';

let ctx: RbacTestContext;
beforeAll(async () => {
  ctx = await getContext();
}, 60000);
afterAll(async () => {
  await closeContext();
});
```

The context replicates production config from `main.ts`:
- `ValidationPipe` (whitelist, transform, implicit conversion)
- URI versioning (`/v1/...`)
- Cookie parser

### Matrix Runner

Instead of writing individual `it()` blocks, you define expectations declaratively:

```typescript
{
  method: 'GET',
  path: '/v1/leads/:leadByBasicId',
  description: 'View lead detail',
  expectations: {
    OWNER: 200,
    ADMIN: 200,
    TEAM_LEADER: 403,
    AGENT: 403,
    BASIC: 200,
  },
}
```

<Steps>
<Step title="Matrix Processing">
The runner calls `runRbacMatrix()` which iterates every endpoint in the matrix
</Step>
<Step title="Persona Iteration">
For each endpoint, iterates all 5 personas
</Step>
<Step title="Path Resolution">
Resolves path placeholders (`:leadByBasicId` → actual UUID)
</Step>
<Step title="HTTP Request">
Makes the HTTP request with the persona's tenant token
</Step>
<Step title="Assertion">
Asserts the response status matches the expected value and produces clear failure messages
</Step>
</Steps>

<Note>
One matrix entry = 5 generated test cases (one per persona)
</Note>

## RBAC Personas

Defined in `test/helpers/rbac-personas.ts`. All personas share a single source of truth.

| Key | Email | Org Role | Team | Description |
|-----|--------|----------|------|-------------|
| `OWNER` | owner@propwise.com | Owner | Any | All permissions including `system.owner` |
| `ADMIN` | admin@propwise.com | Admin | Any | All non-sensitive permissions (includes `system.admin`) |
| `TEAM_LEADER` | teamleader@propwise.com | Basic | Team Leader in Test Sales Team | All team permissions (`team_crm.*`, `team_sales.*`, `team.admin`) |
| `AGENT` | agent@propwise.com | Basic | Agent in Test Sales Team | No team permissions by default |
| `BASIC` | basic@propwise.com | Basic | None | No permissions — relies entirely on stakeholder access |

### Permission Profiles

<Tabs>
<Tab title="OWNER / ADMIN">
Bypass resource-level checks via org permissions (`crm.view`, `crm.manage`, `sales.view`, `sales.manage`, etc.). Can access any entity in the org.
</Tab>
<Tab title="TEAM_LEADER">
No org permissions, but has `team_crm.manage` and `team_sales.manage` in the team. Can access entities where the team is a stakeholder.
</Tab>
<Tab title="AGENT">
No org permissions, no team permissions. Can only access entities where they are personally a stakeholder. Used to test access level differentiation (READ vs WRITE vs FULL).
</Tab>
<Tab title="BASIC">
Same as AGENT but not a team member at all. Access is purely stakeholder-based.
</Tab>
</Tabs>

### Login Flow

The context performs a two-step login for each persona:

1. `POST /v1/auth/login` → identity token
2. `POST /v1/auth/select-organization` → tenant token (org-scoped JWT)

<Warning>
All subsequent API calls use the tenant token in the `Authorization: Bearer` header.
</Warning>

### Access Mode Reference

Your `@CheckAccess` decorator uses two modes:

- **OR (default):** User has the required permission **OR** has stakeholder-level access on the resource → allowed
- **AND:** User must have the permission **AND** stakeholder access → allowed

<Info>
**Resource action hierarchy:**
- `ACCESS` — granted by `READ`, `WRITE`, or `FULL` access level
- `MODIFY` — granted by `WRITE` or `FULL`
- `DELETE` — granted by `FULL` only
</Info>

## Test Data Seeding

### Step 1: Seed Users (One-time Setup)

```bash
pnpm test:seed-users
```

This runs `test/setup/seed-test-users.ts` which:

<Steps>
<Step title="Create Test Users">
Creates the 5 test users in the database (idempotent — skips if they exist)
</Step>
<Step title="Create Organization">
Creates "Test Organization" with owner
</Step>
<Step title="Create Team">
Creates "Test Sales Team" with team leader and agent
</Step>
<Step title="Assign Roles">
Assigns org roles (Owner, Admin, Basic) and team roles (Team Leader, Agent)
</Step>
<Step title="Database Write">
Uses RLS bypass to write directly via MikroORM
</Step>
</Steps>

### Step 2: Seed Test Data (Automatic)

Each test file calls `seedRbacTestData(app, personas, teamId)` in its `beforeAll`. Results are cached across files.

<Tip>
This creates entities through the API to ensure all business logic (guards, auto-assignment, stakeholder creation) executes naturally.
</Tip>

## How to Extend

### Adding New Endpoints to Existing Matrices

1. **Identify the entity and access pattern**
2. **Add to the appropriate matrix file** in `test/setup/matrices/`
3. **Define expected status codes** for each persona

```typescript
// In matrices/lead-matrix.ts
export const LEAD_BY_BASIC_MATRIX: EndpointExpectation[] = [
  // ... existing endpoints
  {
    method: 'POST',
    path: '/v1/leads/:leadByBasicId/new-operation',
    description: 'New lead operation',
    expectations: {
      OWNER: 200,
      ADMIN: 200,
      TEAM_LEADER: 403,
      AGENT: 403,
      BASIC: 200, // Has FULL access as creator
    },
  },
];
```

### Adding New Entity Test Files

<Steps>
<Step title="Create Matrix File">
Create `test/setup/matrices/your-entity-matrix.ts` with permission matrices
</Step>
<Step title="Update Index">
Add export to `test/setup/matrices/index.ts`
</Step>
<Step title="Create Test File">
Create `test/e2e/rbac/your-entity-rbac.e2e-spec.ts` following the pattern:

```typescript
import { getContext, closeContext } from '../../setup/rbac-context';
import { seedRbacTestData } from '../../setup/rbac-test-data';
import { runRbacMatrix } from '../../setup/rbac-runner';
import { YOUR_ENTITY_MATRIX } from '../../setup/rbac-matrix';

describe('Your Entity RBAC', () => {
  let ctx: RbacTestContext;
  let testData: any;

  beforeAll(async () => {
    ctx = await getContext();
    testData = await seedRbacTestData(ctx.app, ctx.personas, ctx.teamId);
  }, 60000);

  afterAll(async () => {
    await closeContext();
  });

  describe('Matrix Tests', () => {
    runRbacMatrix(YOUR_ENTITY_MATRIX, () => ctx, () => testData);
  });
});
```
</Step>
<Step title="Update Test Data">
If needed, extend `seedRbacTestData()` to create test entities for your new resource
</Step>
</Steps>

## Methodology

<Check>
The test suite follows a systematic approach to ensure comprehensive RBAC coverage across all API endpoints.
</Check>

### Test Design Principles

1. **Permission-First Testing** — Every endpoint is tested against every persona to verify proper access control
2. **Real Business Logic** — Test data creation uses actual API endpoints to trigger all guards and business rules
3. **Stakeholder-Aware** — Tests verify both permission-based and stakeholder-based access patterns
4. **Declarative Matrices** — Expected outcomes are defined upfront, making test intentions clear

### Analysis Process

Before writing tests:

1. **Review controller endpoints** and their `@CheckAccess` decorators
2. **Identify required permissions** and stakeholder access patterns
3. **Map persona capabilities** to expected HTTP status codes
4. **Define test entities** that create the necessary access scenarios

<Info>
See `test/TESTING_METHODOLOGY.md` for detailed controller/DTO analysis guidelines.
</Info>
