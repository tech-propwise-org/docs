## Overview

The PropWise CRM backend test suite uses **Jest** with **supertest** and **@nestjs/testing** to verify API endpoints and RBAC (Role-Based Access Control) enforcement.

<Note>
  The test suite focuses on **E2E testing** with a singleton NestJS app context that boots once and is shared across all test files for faster execution.
</Note>

**Key Features:**
- Matrix-driven RBAC tests (declarative permission checks)
- API-driven test data seeding (exercises real business logic)
- Singleton app context (3-5s boot time, then instant reuse)
- Separate Jest configs for core vs. queue-based tests

## Prerequisites

<Steps>
  <Step title="Install Dependencies">
    Ensure all packages are installed:
    ```bash
    pnpm install
    ```
  </Step>

  <Step title="Set Up Test Database">
    Create a dedicated test database (recommended) or use your development database:
    ```bash
    # Example: PostgreSQL
    createdb propwise_test
    ```
    
    Update your `.env.test` or test configuration with the test database connection string.
  </Step>

  <Step title="Initialize Subscription Plans">
    Run the data initialization script once on a fresh database:
    ```bash
    pnpm test:init-data
    ```
    
    This creates required subscription plans and base data.
  </Step>

  <Step title="Seed Test Users">
    Create RBAC test personas (one-time setup):
    ```bash
    pnpm test:seed-users
    ```
    
    <Accordion title="What does test:seed-users create?">
      This script (`test/setup/seed-test-users.ts`) performs the following:
      
      - Creates 5 RBAC personas: `OWNER`, `ADMIN`, `TEAM_LEADER`, `AGENT`, `BASIC`
      - Creates "Test Organization" with an active BUSINESS subscription
      - Ensures completed `organization_setup_progress` (bypasses onboarding)
      - Creates "Test Sales Team" with team leader and agent members
      - Assigns appropriate org roles and team roles
      - **Idempotent:** Safe to run multiple times (skips existing users)
    </Accordion>
  </Step>
</Steps>

## Running Tests Locally

### Core E2E Tests

Run the main test suite (auth lifecycle, RBAC tests for leads, deals, contacts, companies):

```bash
pnpm test:e2e
```

<Info>
  This command uses `jest-e2e.json` configuration and excludes distribution and escalation queue tests.
</Info>

### Queue-Based Tests

Run tests that require live pg-boss queue processing:

```bash
pnpm test:e2e:queue
```

<Info>
  This command uses `jest-e2e-queue.json` configuration for distribution, escalation, and queue-related specs.
</Info>

### Watch Mode

Run tests in watch mode during development:

```bash
pnpm test:e2e -- --watch
```

### Single Test File

Run a specific test file:

```bash
pnpm test:e2e test/e2e/rbac/lead-rbac.e2e-spec.ts
```

### Debug Mode

Run tests with Node.js debugger:

```bash
node --inspect-brk node_modules/.bin/jest --config test/jest-e2e.json --runInBand
```

Then attach your debugger (VS Code, Chrome DevTools, etc.) to `localhost:9229`.

### Auth Tests

Located in `test/e2e/auth/`:

<AccordionGroup>
  <Accordion title="auth-lifecycle.e2e-spec.ts">
    **Tests:**
    - User registration (`POST /v1/auth/signup`)
    - Email verification code retrieval
    - Account verification (`POST /v1/auth/verify`)
    - Login with credentials (`POST /v1/auth/login`)
    - Password reset request (`POST /v1/auth/forgot-password`)
    - Password reset completion (`POST /v1/auth/reset-password`)
    - Organization creation during onboarding
    - Organization selection (`POST /v1/auth/select-organization`)
    
    **Expected Results:**
    - All authentication flows complete successfully
    - Tokens are issued correctly (identity token, tenant token)
    - Email verification codes work as expected
  </Accordion>

  <Accordion title="org-lifecycle.e2e-spec.ts">
    **Tests:**
    - Organization CRUD operations
    - Optional field handling
    - Multi-org support
    - Organization switching
    
    **Expected Results:**
    - Organizations are created with correct data
    - Users can switch between organizations
    - Role permissions apply correctly per organization
  </Accordion>
</AccordionGroup>

## Understanding Test Output

### Successful Test Run

```
PASS test/e2e/rbac/lead-rbac.e2e-spec.ts (12.543 s)
  Lead RBAC Tests
    Lead Created by BASIC (FULL Access)
      ✓ OWNER → GET /v1/leads/:leadByBasicId → 200 (45 ms)
      ✓ ADMIN → GET /v1/leads/:leadByBasicId → 200 (32 ms)
      ✓ TEAM_LEADER → GET /v1/leads/:leadByBasicId → 403 (28 ms)
      ✓ AGENT → GET /v1/leads/:leadByBasicId → 403 (25 ms)
      ✓ BASIC → GET /v1/leads/:leadByBasicId → 200 (30 ms)
```

<Check>
  All personas returned expected HTTP status codes. Access control is working correctly.
</Check>

### Failed Test Example

```
FAIL test/e2e/rbac/lead-rbac.e2e-spec.ts
  Lead RBAC Tests
    Lead Created by BASIC (FULL Access)
      ✕ AGENT → GET /v1/leads/:leadByBasicId → 403 (45 ms)

Expected: 403
Received: 200

Response body:
{
  "id": "123e4567-e89b-12d3-a456-426614174000",
  "firstName": "Test",
  "lastName": "Lead",
  ...
}
```

<Warning>
  This indicates an RBAC guard failure. AGENT should have been denied access but received the resource instead.
</Warning>

**Common Causes:**
- Missing `@CheckAccess()` decorator on controller method
- Incorrect permission requirements in decorator
- Stakeholder guard not executing properly
- Test data seeding issue (AGENT was incorrectly added as stakeholder)

## CI/CD Integration

### GitHub Actions Example

```yaml .github/workflows/test.yml
name: Backend Tests

on:
  pull_request:
    branches: [main, develop]
  push:
    branches: [main, develop]

jobs:
  test:
    runs-on: ubuntu-latest
    
    services:
      postgres:
        image: postgres:15
        env:
          POSTGRES_DB: propwise_test
          POSTGRES_USER: postgres
          POSTGRES_PASSWORD: postgres
        options: >-
          --health-cmd pg_isready
          --health-interval 10s
          --health-timeout 5s
          --health-retries 5
        ports:
          - 5432:5432
    
    steps:
      - uses: actions/checkout@v3
      
      - name: Setup Node.js
        uses: actions/setup-node@v3
        with:
          node-version: '20'
          cache: 'pnpm'
      
      - name: Install pnpm
        run: npm install -g pnpm
      
      - name: Install dependencies
        run: pnpm install --frozen-lockfile
      
      - name: Run database migrations
        run: pnpm migration:run
        env:
          DATABASE_URL: postgresql://postgres:postgres@localhost:5432/propwise_test
      
      - name: Initialize test data
        run: pnpm test:init-data
        env:
          DATABASE_URL: postgresql://postgres:postgres@localhost:5432/propwise_test
      
      - name: Seed test users
        run: pnpm test:seed-users
        env:
          DATABASE_URL: postgresql://postgres:postgres@localhost:5432/propwise_test
      
      - name: Run E2E tests
        run: pnpm test:e2e
        env:
          DATABASE_URL: postgresql://postgres:postgres@localhost:5432/propwise_test
          JWT_SECRET: test-secret-key-change-in-production
          NODE_ENV: test
      
      - name: Run queue tests
        run: pnpm test:e2e:queue
        env:
          DATABASE_URL: postgresql://postgres:postgres@localhost:5432/propwise_test
          JWT_SECRET: test-secret-key-change-in-production
          NODE_ENV: test
      
      - name: Upload coverage
        uses: codecov/codecov-action@v3
        if: always()
        with:
          files: ./coverage/lcov.info
```

## Troubleshooting

<AccordionGroup>
  <Accordion title="Test timeouts during app boot">
    **Symptom:** Tests fail with "Timeout - Async callback was not invoked within the 5000ms timeout"
    
    **Solution:**
    - Increase Jest timeout in `beforeAll`:
      ```typescript
      beforeAll(async () => {
        ctx = await getContext();
      }, 60000); // 60 second timeout
      ```
    - Check database connectivity
    - Ensure no migrations are pending
  </Accordion>

  <Accordion title="'Test organization not found' error">
    **Symptom:** Tests fail with 404 or "Organization not found"
    
    **Solution:**
    - Re-run the seed script:
      ```bash
      pnpm test:seed-users
      ```
    - Verify the test database is correctly configured
    - Check that `organization_setup_progress` is completed for test org
  </Accordion>

  <Accordion title="Permission tests failing unexpectedly">
    **Symptom:** RBAC tests show incorrect access (e.g., 200 when expecting 403)
    
    **Solution:**
    - Check controller decorators (`@CheckAccess()`)
    - Verify guard execution order in `main.ts`
    - Review stakeholder assignments in test data seeding
    - Use debug mode to inspect request flow:
      ```bash
      node --inspect-brk node_modules/.bin/jest --config test/jest-e2e.json --runInBand
      ```
  </Accordion>

  <Accordion title="pg-boss mock errors">
    **Symptom:** "Cannot find module 'pg-boss'" or queue-related errors in core tests
    
    **Solution:**
    - Verify `jest-e2e.json` includes the mock mapping:
      ```json
      {
        "moduleNameMapper": {
          "^pg-boss$": "<rootDir>/mocks/pg-boss.mock.js"
        }
      }
      ```
    - For actual queue tests, use `jest-e2e-queue.json` instead
  </Accordion>

  <Accordion title="Database connection pool exhausted">
    **Symptom:** "too many clients already" or connection timeout errors
    
    **Solution:**
    - Ensure `closeContext()` is called in `afterAll`:
      ```typescript
      afterAll(async () => {
        await closeContext();
      });
      ```
    - Check for unclosed database connections in your code
    - Reduce connection pool size in test config if necessary
  </Accordion>

  <Accordion title="Stale test data affecting results">
    **Symptom:** Inconsistent test results, especially after manual database changes
    
    **Solution:**
    - Clean up test users:
      ```bash
      pnpm test:cleanup-users
      ```
    - Re-seed:
      ```bash
      pnpm test:seed-users
      ```
    - Or reset the entire test database and re-run initialization
  </Accordion>
</AccordionGroup>

## Performance Optimization

<Tip>
  The singleton context pattern reduces test suite runtime from ~60s to ~15s by booting the app once instead of per-file.
</Tip>

### Best Practices

1. **Share the context:** Always use `getContext()` and `closeContext()` from `rbac-context.ts`
2. **Avoid beforeEach app boots:** Never create a new NestJS app in `beforeEach`
3. **Cache test data:** Use `seedRbacTestData()` which caches entities across files
4. **Run tests in parallel:** Jest runs test files in parallel by default (use `--runInBand` only for debugging)
5. **Use the correct Jest config:** Core tests use `jest-e2e.json`, queue tests use `jest-e2e-queue.json`

### Timing Benchmarks

| Operation | Time (First Call) | Time (Cached) |
|-----------|-------------------|---------------|
| Boot NestJS app | 3-5s | 0ms |
| Login 5 personas | 2-3s | 0ms |
| Seed test data | 5-8s | 0ms |
| Single RBAC test | 25-50ms | — |
