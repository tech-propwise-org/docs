The PropWise CRM backend test suite focuses on **RBAC (Role-Based Access Control) E2E testing** — verifying that every API endpoint enforces the correct permissions for every user role.

**Stack:** Jest + supertest + `@nestjs/testing`

## Prerequisites

Before running tests, ensure you have the following setup:

<Steps>
<Step title="Database Setup">
Database must be running with migrations applied:

```bash
pnpm db:migrate
```
</Step>

<Step title="Environment Configuration">
Your `.env.local` must include:

```bash
DATABASE_URL=postgresql://...
JWT_TENANT_SECRET=your-secret
META_TOKEN_ENCRYPTION_KEY=your-key
META_WEBHOOK_VERIFY_TOKEN=your-token
```
</Step>

<Step title="Seed Test Users">
Create the test user data (one-time setup):

```bash
pnpm test:seed-users
```

This creates 5 test users with different roles:
- **OWNER**: owner@propwise-test.com (Organization Owner)
- **ADMIN**: admin@propwise-test.com (Organization Admin) 
- **TEAM_LEADER**: teamleader@propwise-test.com (Team Leader)
- **AGENT**: agent@propwise-test.com (Team Agent)
- **BASIC**: basic@propwise-test.com (Basic User)
</Step>
</Steps>

## Test Architecture

### Singleton Context

The test suite uses a singleton NestJS app instance shared across all test files:

```typescript
import { getContext, closeContext, RbacTestContext } from '../../setup/rbac-context';

let ctx: RbacTestContext;
beforeAll(async () => { ctx = await getContext(); }, 60000);
afterAll(async () => { await closeContext(); });
```

**Benefits:**
- App boots once (~3-5s), not per file
- Authenticated tokens cached
- Consistent state across tests
- Faster overall execution

### Permission Matrix System

Tests are driven by declarative permission matrices:

```typescript
{
  method: 'GET',
  path: '/v1/leads/:leadByBasicId',
  description: 'View lead detail',
  expectations: {
    OWNER: 200,
    ADMIN: 200,
    TEAM_LEADER: 403,
    AGENT: 403,
    BASIC: 200,
  },
}
```

The matrix runner automatically generates parameterized Jest tests for each persona.

## RBAC Personas

<CardGroup cols={2}>
<Card title="Organization Roles" icon="building">
- **OWNER**: All permissions including `system.owner`
- **ADMIN**: All non-sensitive permissions (`system.admin`)
- **BASIC**: No org-level permissions
</Card>

<Card title="Team Roles" icon="users">
- **TEAM_LEADER**: Team admin permissions (`team.admin`, `team_crm.*`, `team_sales.*`)
- **AGENT**: Team member, no special permissions
- **BASIC**: Not a team member
</Card>
</CardGroup>

### Permission Hierarchy

<Tabs>
<Tab title="Organization Permissions">
**OWNER / ADMIN** bypass resource-level checks via org permissions:
- `crm.view`, `crm.manage` - CRM entities
- `sales.view`, `sales.manage` - Sales entities
- Can access any entity in the organization
</Tab>

<Tab title="Team Permissions">
**TEAM_LEADER** has team-scoped permissions:
- `team_crm.manage` - Manage team CRM entities
- `team_sales.manage` - Manage team sales entities
- Can access entities where team is a stakeholder
</Tab>

<Tab title="Stakeholder Access">
**AGENT / BASIC** rely on stakeholder-level access:
- `READ` - View entity details
- `WRITE` - Modify entity (includes READ)
- `FULL` - Delete entity (includes WRITE + READ)
</Tab>
</Tabs>

## Extending Tests

### Adding New Entity Tests

<Steps>
<Step title="Create Permission Matrix">
Create a new matrix file in `test/setup/matrices/`:

```typescript
// test/setup/matrices/property-matrix.ts
export const PROPERTY_BY_BASIC_MATRIX = [
  {
    method: 'GET' as const,
    path: '/v1/properties/:propertyByBasicId',
    description: 'View property detail',
    expectations: {
      OWNER: 200,
      ADMIN: 200,
      TEAM_LEADER: 403,
      AGENT: 403,
      BASIC: 200,
    },
  },
  // Add more endpoints...
];
```
</Step>

<Step title="Update Test Data Seeding">
Add test data creation in `test/setup/rbac-test-data.ts`:

```typescript
// Create properties via API
const propertyByBasic = await request(app.getHttpServer())
  .post('/v1/properties')
  .set('Authorization', `Bearer ${personas.BASIC.tenantToken}`)
  .send({
    address: '123 Test St',
    city: 'Test City',
    // ...
  });
```
</Step>

<Step title="Create Test File">
Create the E2E test file in `test/e2e/rbac/`:

```typescript
// test/e2e/rbac/property-rbac.e2e-spec.ts
import { runRbacMatrix } from '../../setup/rbac-runner';
import { PROPERTY_BY_BASIC_MATRIX } from '../../setup/rbac-matrix';

describe('Property RBAC (E2E)', () => {
  let ctx: RbacTestContext;
  let testData: any;

  beforeAll(async () => {
    ctx = await getContext();
    testData = await seedRbacTestData(ctx.app, ctx.personas, ctx.teamId);
  }, 60000);

  afterAll(async () => {
    await closeContext();
  });

  describe('Property owned by BASIC user', () => {
    runRbacMatrix(PROPERTY_BY_BASIC_MATRIX, () => ctx, () => testData);
  });
});
```
</Step>
</Steps>

### Adding New Endpoints

<Steps>
<Step title="Analyze Controller">
Review the controller and DTOs to understand:
- Required permissions
- Access control mode (OR/AND)
- Resource relationships
- Expected responses

See `test/TESTING_METHODOLOGY.md` for detailed guidance.
</Step>

<Step title="Add to Matrix">
Add the endpoint to the appropriate matrix:

```typescript
{
  method: 'POST',
  path: '/v1/leads/:leadByBasicId/activities',
  description: 'Create lead activity',
  body: {
    type: 'call',
    description: 'Follow up call',
  },
  expectations: {
    OWNER: 201,
    ADMIN: 201,
    TEAM_LEADER: 403,
    AGENT: 403,
    BASIC: 201,
  },
}
```
</Step>

<Step title="Handle Path Parameters">
Update the resolver function if new path parameters are needed:

```typescript
// In rbac-test-data.ts, ensure IDs are available
return {
  // existing data...
  newEntityId: response.body.id,
};

// In the matrix resolver
(data) => ({
  ':newEntityId': data.newEntityId,
})
```
</Step>
</Steps>

<Warning>
Always verify that new test data is created through the actual API endpoints, not direct database insertion. This ensures all business logic, guards, and auto-assignment rules execute.
</Warning>

## CI/CD Integration

### GitHub Actions Setup

<CodeGroup>
```yaml Basic Workflow
name: E2E Tests
on: [push, pull_request]

jobs:
  e2e-tests:
    runs-on: ubuntu-latest
    services:
      postgres:
        image: postgres:15
        env:
          POSTGRES_PASSWORD: postgres
        options: >-
          --health-cmd pg_isready
          --health-interval 10s
          --health-timeout 5s
          --health-retries 5

    steps:
      - uses: actions/checkout@v3
      - uses: actions/setup-node@v3
        with:
          node-version: '18'
          cache: 'npm'
      
      - run: npm ci
      - run: npm run db:migrate
      - run: npm run test:seed-users
      - run: npm run test:e2e -- --testPathPattern=rbac
```

```yaml With Coverage
- run: npm run test:e2e:cov -- --testPathPattern=rbac
- uses: codecov/codecov-action@v3
  with:
    file: ./coverage/lcov.info
```

```yaml Parallel Matrix
strategy:
  matrix:
    test-pattern: [lead-rbac, deal-rbac, contact-rbac, company-rbac]
steps:
  # ... setup steps
  - run: npm run test:e2e -- --testPathPattern=${{ matrix.test-pattern }}
```
</CodeGroup>

### Performance Optimization

<Tips>
- Use database connection pooling
- Run `test:seed-users` only once per workflow
- Consider parallel test execution for independent test files
- Cache `node_modules` and dependencies
- Use faster test database (in-memory or optimized PostgreSQL config)
</Tips>

### Failure Debugging

When tests fail in CI:

<Steps>
<Step title="Check Database State">
```bash
# Add to CI workflow for debugging
- run: npm run db:seed:status
- run: psql $DATABASE_URL -c "SELECT email FROM users WHERE email LIKE '%@propwise-test.com';"
```
</Step>

<Step title="Enable Verbose Logging">
```bash
# Run with verbose output
npm run test:e2e -- --testPathPattern=rbac --verbose

# Or with custom logging
DEBUG=test:* npm run test:e2e
```
</Step>

<Step title="Artifact Collection">
```yaml
- name: Upload test logs
  if: failure()
  uses: actions/upload-artifact@v3
  with:
    name: test-logs
    path: logs/
```
</Step>
</Steps>

<Info>
The singleton context pattern makes tests fast and reliable, but be aware that shared state can occasionally cause issues if not properly cleaned up between test runs.
</Info>
