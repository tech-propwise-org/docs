## Overview

The PropWise CRM backend test suite uses **Jest** with **supertest** and **@nestjs/testing** to verify RBAC (Role-Based Access Control) enforcement across all API endpoints. The architecture uses a singleton app context that boots once and is shared across all test files for optimal performance.

<Info>
The test suite focuses on **E2E RBAC testing** — verifying that every API endpoint enforces the correct permissions for every user role through a declarative permission matrix system.
</Info>

## Prerequisites

Before running tests, ensure you have:

<Steps>
<Step title="Initialize test data">
Run this once on a fresh database to create subscription plans:

```bash
pnpm test:init-data
```
</Step>

<Step title="Seed RBAC test users">
Create the test personas, organization, and team:

```bash
pnpm test:seed-users
```

This script is **idempotent** — it skips existing users and self-heals stale subscription data.
</Step>

<Step title="Verify database connection">
Ensure your `.env.test` file has valid database credentials and the test database exists.
</Step>
</Steps>

<Note>
The `test:seed-users` script creates 5 RBAC personas (OWNER, ADMIN, TEAM_LEADER, AGENT, BASIC), a test organization with BUSINESS subscription, and a test sales team. This data persists between test runs.
</Note>

## Running the Test Suite

### Run All Core E2E Tests

Execute the full RBAC test suite (excludes queue-based tests):

```bash
pnpm test:e2e
```

This runs all tests matching the pattern in `jest-e2e.json` and excludes distribution/escalation queue tests.

<Accordion title="What does this command test?">
- **Auth lifecycle:** signup → verify → login → password reset → org creation → org selection
- **Lead RBAC:** 5 matrices × 8 endpoints = 40 generated test cases
- **Deal RBAC:** Access control for deals with stakeholder validation
- **Contact RBAC:** Contact permissions by creator and stakeholders
- **Company RBAC:** Company access control verification
- **Org lifecycle:** Organization CRUD operations and switching
</Accordion>

### Run Queue-Based Tests

Execute tests that require live `pg-boss` integration:

```bash
pnpm test:e2e:queue
```

This uses `jest-e2e-queue.json` configuration for distribution, escalation, and queue-related specs.

<Warning>
Queue tests require a running PostgreSQL instance with `pg-boss` tables initialized. They are excluded from the default `test:e2e` run to avoid pg-boss mocking conflicts.
</Warning>

### Run Specific Test Suites

<Tabs>
<Tab title="By File Pattern">
Run tests matching a specific file pattern:

```bash
pnpm test:e2e -- lead-rbac
pnpm test:e2e -- auth-lifecycle
pnpm test:e2e -- rbac/
```
</Tab>

<Tab title="By Test Name">
Run tests with specific descriptions:

```bash
pnpm test:e2e -- -t "OWNER"
pnpm test:e2e -- -t "View lead detail"
```
</Tab>

<Tab title="Watch Mode">
Run tests in watch mode for development:

```bash
pnpm test:e2e -- --watch
```
</Tab>
</Tabs>

### Verbose Output

Enable detailed logging to see request/response data:

```bash
pnpm test:e2e -- --verbose
```

## Understanding Test Output

### Matrix-Driven Test Format

The test suite generates parameterized tests from permission matrices. Each matrix entry produces 5 test cases (one per persona):

```
RBAC: Lead by BASIC
  ✓ [OWNER] GET /v1/leads/:leadByBasicId - View lead detail (42ms)
  ✓ [ADMIN] GET /v1/leads/:leadByBasicId - View lead detail (38ms)
  ✗ [TEAM_LEADER] GET /v1/leads/:leadByBasicId - View lead detail (45ms)
  ✗ [AGENT] GET /v1/leads/:leadByBasicId - View lead detail (41ms)
  ✓ [BASIC] GET /v1/leads/:leadByBasicId - View lead detail (39ms)
```

<Tip>
The persona name in brackets shows which user role is being tested. Green checkmarks indicate the response matched the expected HTTP status code.
</Tip>

### Failure Messages

When a test fails, you'll see:

```
Expected status 403 but got 200

Response body:
{
  "id": "uuid-here",
  "firstName": "Test",
  "lastName": "Lead",
  ...
}
```

This detailed output helps debug permission logic issues.

## CI/CD Integration

### GitHub Actions

The test suite integrates with GitHub Actions for continuous testing:

```yaml .github/workflows/test.yml
name: E2E Tests

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main, develop]

jobs:
  test:
    runs-on: ubuntu-latest
    
    services:
      postgres:
        image: postgres:15
        env:
          POSTGRES_DB: propwise_test
          POSTGRES_USER: test
          POSTGRES_PASSWORD: test
        options: >-
          --health-cmd pg_isready
          --health-interval 10s
          --health-timeout 5s
          --health-retries 5
        ports:
          - 5432:5432
    
    steps:
      - uses: actions/checkout@v3
      
      - name: Setup Node.js
        uses: actions/setup-node@v3
        with:
          node-version: '18'
          cache: 'pnpm'
      
      - name: Install dependencies
        run: pnpm install
      
      - name: Run migrations
        run: pnpm migration:run
        env:
          DATABASE_URL: postgresql://test:test@localhost:5432/propwise_test
      
      - name: Initialize test data
        run: pnpm test:init-data
      
      - name: Seed test users
        run: pnpm test:seed-users
      
      - name: Run E2E tests
        run: pnpm test:e2e
        env:
          DATABASE_URL: postgresql://test:test@localhost:5432/propwise_test
          JWT_SECRET: test-secret-key
      
      - name: Run queue tests
        run: pnpm test:e2e:queue
        env:
          DATABASE_URL: postgresql://test:test@localhost:5432/propwise_test
```

<Check>
The CI pipeline ensures all tests pass before merging code. Database setup, migrations, and seeding happen automatically in the workflow.
</Check>

## Test Architecture

### Singleton App Context

The test suite uses a shared NestJS application instance across all test files:

```typescript
import { getContext, closeContext, RbacTestContext } from '../../setup/rbac-context';

let ctx: RbacTestContext;

beforeAll(async () => {
  ctx = await getContext(); // First call boots app, subsequent calls return cached instance
}, 60000);

afterAll(async () => {
  await closeContext(); // Reference counting ensures app closes after last test
});
```

<Info>
**Performance benefit:** The app boots once (~3-5 seconds) instead of once per test file. Subsequent `getContext()` calls return the cached instance in 0ms.
</Info>

### Test Data Lifecycle

<AccordionGroup>
<Accordion title="Persistent Data (Seeded Once)">
Created by `pnpm test:seed-users` and persists between test runs:

- 5 RBAC personas (OWNER, ADMIN, TEAM_LEADER, AGENT, BASIC)
- Test Organization with BUSINESS subscription
- Test Sales Team with team leader and agent
- Org roles and team memberships
- Completed onboarding status

This data is idempotent — re-running the seed script updates existing users.
</Accordion>

<Accordion title="Transient Data (Created Per Run)">
Created by `seedRbacTestData()` at the start of each test run:

- Test leads (by BASIC, by OWNER, in pool, with team stakeholder, with AGENT read-only)
- Test deals
- Test contacts
- Test companies
- Stakeholder relationships
- Activities

This data is cached across test files but recreated for each `pnpm test:e2e` invocation.
</Accordion>
</AccordionGroup>

### Permission Matrix System

Tests are generated from declarative matrices:

```typescript
{
  method: 'GET',
  path: '/v1/leads/:leadByBasicId',
  description: 'View lead detail',
  expectations: {
    OWNER: 200,        // Has crm.view permission
    ADMIN: 200,        // Has crm.view permission
    TEAM_LEADER: 403,  // No permission, not a stakeholder
    AGENT: 403,        // No permission, not a stakeholder
    BASIC: 200,        // Creator with FULL stakeholder access
  },
}
```

The `runRbacMatrix()` function iterates this matrix and generates 5 test cases automatically.

## Cleanup and Maintenance

### Remove Test Data

Remove all test personas from the database:

```bash
pnpm test:cleanup-users
```

<Warning>
This deletes all users with `@propwise-test.com` email addresses. Run this if test data becomes corrupted or you need a fresh start.
</Warning>

### Reset Test Environment

For a complete reset:

<Steps>
<Step title="Clean up test users">
```bash
pnpm test:cleanup-users
```
</Step>

<Step title="Drop and recreate test database">
```bash
dropdb propwise_test
createdb propwise_test
pnpm migration:run
```
</Step>

<Step title="Re-initialize test data">
```bash
pnpm test:init-data
pnpm test:seed-users
```
</Step>

<Step title="Run tests">
```bash
pnpm test:e2e
```
</Step>
</Steps>

## Troubleshooting

<AccordionGroup>
<Accordion title="Tests fail with 'Cannot find test users'">
**Cause:** Test users haven't been seeded or were deleted.

**Solution:**
```bash
pnpm test:seed-users
```
</Accordion>

<Accordion title="Tests timeout during app bootstrap">
**Cause:** Database connection issues or slow migrations.

**Solution:**
1. Verify database is running: `psql propwise_test -c "SELECT 1;"`
2. Check `.env.test` has correct credentials
3. Increase timeout in `beforeAll()` (default is 60 seconds)
</Accordion>

<Accordion title="Permission tests fail unexpectedly">
**Cause:** Stale test data or subscription issues.

**Solution:**
```bash
pnpm test:cleanup-users
pnpm test:seed-users
pnpm test:e2e
```

The seed script self-heals subscription and onboarding status.
</Accordion>

<Accordion title="Queue tests fail with pg-boss errors">
**Cause:** Queue tests require live pg-boss, not mocks.

**Solution:** Use the queue-specific config:
```bash
pnpm test:e2e:queue
```

Don't mix queue tests with core tests — they use different Jest configurations.
</Accordion>

<Accordion title="Tests pass locally but fail in CI">
**Cause:** Environment differences (JWT secrets, database state, timing).

**Solution:**
1. Simulate CI locally (see "Local CI Simulation" above)
2. Check GitHub Actions logs for specific error messages
3. Verify CI database migrations ran successfully
4. Ensure environment variables match CI configuration
</Accordion>
</AccordionGroup>

## Performance Optimization

The test suite is optimized for speed:

<CardGroup cols={2}>
<Card title="Singleton App" icon="server">
App boots once and is shared across all test files (~3-5s overhead total vs ~15-20s per file)
</Card>

<Card title="Cached Test Data" icon="database">
Entities created via API are cached and reused across test files
</Card>

<Card title="Parallel Execution" icon="arrows-split-up-and-left">
Jest runs test files in parallel by default (`--maxWorkers=50%`)
</Card>

<Card title="Smart Seeding" icon="seedling">
Idempotent seed scripts skip existing data and only update what's needed
</Card>
</CardGroup>

<Tip>
On a typical development machine, the full RBAC test suite (200+ tests) completes in **15-25 seconds** including app bootstrap.
</Tip>
