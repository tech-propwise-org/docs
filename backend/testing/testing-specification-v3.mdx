# Testing Specification v3

Architecture, conventions, and run guide for the PropWise CRM backend test suite.

## Overview

The test suite focuses on **RBAC (Role-Based Access Control) E2E testing** — verifying that every API endpoint enforces the correct permissions for every user role.

<Info>
**Stack:** Jest + supertest + `@nestjs/testing`
</Info>

**Key design decisions:**

- **Singleton app context** — the NestJS app boots once and is shared across all test files (equivalent to NestJS `TestSuite` pattern, implemented manually since `TestSuite` requires NestJS 11+)
- **Matrix-driven tests** — a declarative permission matrix defines expected HTTP status codes per persona × endpoint. A runner generates `it()` blocks automatically
- **API-driven seeding** — test data is created through the actual API endpoints so all business logic (guards, auto-assignment, stakeholder creation) executes naturally

## Architecture

### Singleton Context (`rbac-context.ts`)

The app boots once for the entire test run. All test files share the same NestJS app instance, authenticated tokens, and resolved `teamId`.

<Steps>
<Step title="Jest starts">
`lead-rbac.e2e-spec.ts` calls `getContext()`
</Step>
<Step title="First call">
Boots NestJS app (~3-5s), logs in 5 personas, resolves teamId
</Step>
<Step title="Returns context">
Returns `{ app, personas, teamId }`
</Step>
<Step title="Subsequent calls">
`deal-rbac.e2e-spec.ts` calls `getContext()` → returns cached instance instantly (0ms)
</Step>
<Step title="Cleanup">
Last file's `afterAll()` calls `closeContext()` → reference count hits 0 → app shuts down
</Step>
</Steps>

**How it works:**

```typescript
import { getContext, closeContext, RbacTestContext } from '../../setup/rbac-context';

let ctx: RbacTestContext;
beforeAll(async () => { ctx = await getContext(); }, 60000);
afterAll(async () => { await closeContext(); });
```

The context replicates production config from `main.ts`:
- `ValidationPipe` (whitelist, transform, implicit conversion)
- URI versioning (`/v1/...`)
- Cookie parser

### Matrix Runner (`rbac-runner.ts`)

Instead of writing individual `it()` blocks, you define expectations declaratively:

```typescript
{
  method: 'GET',
  path: '/v1/leads/:leadByBasicId',
  description: 'View lead detail',
  expectations: {
    OWNER: 200,
    ADMIN: 200,
    TEAM_LEADER: 403,
    AGENT: 403,
    BASIC: 200,
  },
}
```

The runner calls `runRbacMatrix()` which:
1. Iterates every endpoint in the matrix
2. For each endpoint, iterates all 5 personas
3. Resolves path placeholders (`:leadByBasicId` → actual UUID)
4. Makes the HTTP request with the persona's tenant token
5. Asserts the response status matches the expected value
6. Produces clear failure messages with the actual response body

<Note>
One matrix entry = 5 generated test cases.
</Note>

## RBAC Personas

Defined in `test/helpers/rbac-personas.ts`. All personas share a single source of truth.

| Key | Email | Org Role | Team | Description |
|-----|-------|----------|------|-------------|
| `OWNER` | owner@propwise.com | Owner | Any | All permissions including `system.owner` |
| `ADMIN` | admin@propwise.com | Admin | Any | All non-sensitive permissions (includes `system.admin`) |
| `TEAM_LEADER` | teamleader@propwise.com | Basic | Team Leader in Test Sales Team | All team permissions (`team_crm.*`, `team_sales.*`, `team.admin`) |
| `AGENT` | agent@propwise.com | Basic | Agent in Test Sales Team | No team permissions by default |
| `BASIC` | basic@propwise.com | Basic | None | No permissions — relies entirely on stakeholder access |

### Permission Profiles

<Tabs>
<Tab title="OWNER / ADMIN">
Bypass resource-level checks via org permissions (`crm.view`, `crm.manage`, `sales.view`, `sales.manage`, etc.). Can access any entity in the org.
</Tab>

<Tab title="TEAM_LEADER">
No org permissions, but has `team_crm.manage` and `team_sales.manage` in the team. Can access entities where the team is a stakeholder.
</Tab>

<Tab title="AGENT">
No org permissions, no team permissions. Can only access entities where they are personally a stakeholder. Used to test access level differentiation (READ vs WRITE vs FULL).
</Tab>

<Tab title="BASIC">
Same as AGENT but not a team member at all. Access is purely stakeholder-based.
</Tab>
</Tabs>

### Login Flow

The context performs a two-step login for each persona:

<Steps>
<Step title="Identity login">
`POST /v1/auth/login` → identity token
</Step>
<Step title="Organization selection">
`POST /v1/auth/select-organization` → tenant token (org-scoped JWT)
</Step>
</Steps>

All subsequent API calls use the tenant token in the `Authorization: Bearer` header.

### Access Mode Reference

Your `@CheckAccess` decorator uses two modes:

<Tabs>
<Tab title="OR (default)">
User has the required permission **OR** has stakeholder-level access on the resource → allowed.
</Tab>

<Tab title="AND">
User must have the permission **AND** stakeholder access → allowed.
</Tab>
</Tabs>

**Resource action hierarchy:**
- `ACCESS` — granted by `READ`, `WRITE`, or `FULL` access level
- `MODIFY` — granted by `WRITE` or `FULL`
- `DELETE` — granted by `FULL` only

## Test Data Seeding

### Step 1: Seed Users (one-time, before running tests)

```bash
pnpm test:seed-users
```

This runs `test/setup/seed-test-users.ts` which:
- Creates the 5 test users in the database (idempotent — skips if they exist)
- Creates "Test Organization" with owner
- Creates "Test Sales Team" with team leader and agent
- Assigns org roles (Owner, Admin, Basic) and team roles (Team Leader, Agent)
- Uses RLS bypass to write directly via MikroORM

### Step 2: Seed Test Data (automatic, per test run)

Each test file calls `seedRbacTestData(app, personas, teamId)` in its `beforeAll`. Results are cached across files. This creates entities through the API:

<CardGroup cols={2}>
<Card title="Leads" icon="user-plus">
- One by BASIC (auto-assigned as primary stakeholder with FULL access)
- One by OWNER (for admin-only tests)
- One in pool (empty stakeholders)
- One with team as stakeholder (OWNER creates, assigns team — TEAM_LEADER gets access)
- One with AGENT as READ-only stakeholder (OWNER creates, adds AGENT with `accessLevel: read`)
</Card>

<Card title="Other Entities" icon="database">
- **Contacts:** one by BASIC, one by OWNER
- **Companies:** one by BASIC, one by OWNER
- **Deals:** one by OWNER (from lead + contact)
</Card>
</CardGroup>

<Tip>
Because seeding goes through the real API, all guards, auto-assignment rules, and stakeholder creation run. The test data matches what a real user would produce.
</Tip>

## How to Run

### Prerequisites

<Steps>
<Step title="Database setup">
Database must be running with migrations applied
</Step>

<Step title="Seed test users">
Test users must be seeded: `pnpm test:seed-users`
</Step>

<Step title="Environment config">
`.env.local` must have `DATABASE_URL`, `JWT_TENANT_SECRET`, `META_TOKEN_ENCRYPTION_KEY`, `META_WEBHOOK_VERIFY_TOKEN`
</Step>
</Steps>

### Commands

<CodeGroup>
```bash Run all RBAC tests
pnpm test:e2e -- --testPathPattern=rbac
```

```bash Run specific entity
pnpm test:e2e -- --testPathPattern=lead-rbac
pnpm test:e2e -- --testPathPattern=deal-rbac
pnpm test:e2e -- --testPathPattern=contact-rbac
pnpm test:e2e -- --testPathPattern=company-rbac
```

```bash Verbose output
pnpm test:e2e -- --testPathPattern=rbac --verbose
```

```bash With coverage
pnpm test:e2e:cov -- --testPathPattern=rbac
```
</CodeGroup>

## How to Extend

### Adding New Endpoints

<Steps>
<Step title="Identify the entity">
Determine which matrix file to modify (lead, deal, contact, company)
</Step>

<Step title="Add to appropriate matrix">
Add the endpoint to the relevant matrix with expected status codes for each persona
</Step>

<Step title="Update path resolution">
If using path parameters, ensure the runner can resolve them from test data
</Step>

<Step title="Run tests">
Execute the test suite to verify the new endpoint behaves as expected
</Step>
</Steps>

### Adding New Personas

<Steps>
<Step title="Define persona">
Add new persona to `test/helpers/rbac-personas.ts`
</Step>

<Step title="Update seed users">
Modify `test/setup/seed-test-users.ts` to create the new user
</Step>

<Step title="Update matrices">
Add expectations for the new persona to all relevant matrices
</Step>

<Step title="Update context">
Ensure `rbac-context.ts` handles authentication for the new persona
</Step>
</Steps>

## Methodology

The test suite follows a structured approach to ensure comprehensive coverage:

<Warning>
Read `TESTING_METHODOLOGY.md` for detailed guidance on analyzing controllers and DTOs before writing tests.
</Warning>

1. **Controller Analysis** — Review all endpoints, decorators, and guards
2. **Permission Mapping** — Map required permissions to personas
3. **Matrix Definition** — Define expected outcomes for each persona × endpoint combination
4. **Data Dependencies** — Identify test data requirements and relationships
5. **Validation** — Verify access control works as designed

## CI/CD Integration

The test suite is designed to integrate seamlessly with CI/CD pipelines:

- **Fast startup** — Singleton context minimizes app boot overhead
- **Deterministic** — All test data is created fresh per run
- **Isolated** — Each test file can run independently
- **Comprehensive** — Matrix-driven approach ensures all combinations are tested

All RBAC tests should pass consistently in CI environments with proper database setup and environment variables.
