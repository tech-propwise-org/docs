The test suite focuses on **RBAC (Role-Based Access Control) E2E testing** — verifying that every API endpoint enforces the correct permissions for every user role.

<Info>
**Stack:** Jest + supertest + `@nestjs/testing`
</Info>

## Architecture

### Singleton Context Pattern

<Note>
The app boots once for the entire test run. All test files share the same NestJS app instance, authenticated tokens, and resolved `teamId`.
</Note>

<Steps>
  <Step title="First Test File Calls getContext()">
    Boots NestJS app (~3-5s), logs in 5 personas, resolves teamId
  </Step>
  <Step title="Subsequent Files Get Cached Instance">
    Returns cached instance instantly (0ms)
  </Step>
  <Step title="Last File's afterAll() Cleanup">
    Reference count hits 0 → app shuts down
  </Step>
</Steps>

```typescript
import { getContext, closeContext, RbacTestContext } from '../../setup/rbac-context';

let ctx: RbacTestContext;
beforeAll(async () => {
  ctx = await getContext();
}, 60000);
afterAll(async () => {
  await closeContext();
});
```

The context replicates production config from `main.ts`:
- `ValidationPipe` (whitelist, transform, implicit conversion)
- URI versioning (`/v1/...`)
- Cookie parser

### Matrix Runner System

<Tip>
Instead of writing individual `it()` blocks, you define expectations declaratively and let the runner generate test cases automatically.
</Tip>

```typescript
{
  method: 'GET',
  path: '/v1/leads/:leadByBasicId',
  description: 'View lead detail',
  expectations: {
    OWNER: 200,
    ADMIN: 200,
    TEAM_LEADER: 403,
    AGENT: 403,
    BASIC: 200,
  },
}
```

The runner calls `runRbacMatrix()` which:
1. Iterates every endpoint in the matrix
2. For each endpoint, iterates all 5 personas
3. Resolves path placeholders (`:leadByBasicId` → actual UUID)
4. Makes the HTTP request with the persona's tenant token
5. Asserts the response status matches the expected value
6. Produces clear failure messages with the actual response body

<Check>One matrix entry = 5 generated test cases</Check>

## RBAC Personas

<AccordionGroup>
  <Accordion title="Persona Overview">
    Defined in `test/helpers/rbac-personas.ts`. All personas share a single source of truth.

    | Key           | Email                   | Org Role | Team                           | Description                                                       |
    | ------------- | ----------------------- | -------- | ------------------------------ | ----------------------------------------------------------------- |
    | `OWNER`       | owner@propwise.com      | Owner    | Any                            | All permissions including `system.owner`                          |
    | `ADMIN`       | admin@propwise.com      | Admin    | Any                            | All non-sensitive permissions (includes `system.admin`)           |
    | `TEAM_LEADER` | teamleader@propwise.com | Basic    | Team Leader in Test Sales Team | All team permissions (`team_crm.*`, `team_sales.*`, `team.admin`) |
    | `AGENT`       | agent@propwise.com      | Basic    | Agent in Test Sales Team       | No team permissions by default                                    |
    | `BASIC`       | basic@propwise.com      | Basic    | None                           | No permissions — relies entirely on stakeholder access            |
  </Accordion>

  <Accordion title="Permission Profiles">
    **OWNER / ADMIN** — bypass resource-level checks via org permissions (`crm.view`, `crm.manage`, `sales.view`, `sales.manage`, etc.). Can access any entity in the org.

    **TEAM_LEADER** — no org permissions, but has `team_crm.manage` and `team_sales.manage` in the team. Can access entities where the team is a stakeholder.

    **AGENT** — no org permissions, no team permissions. Can only access entities where they are personally a stakeholder. Used to test access level differentiation (READ vs WRITE vs FULL).

    **BASIC** — same as AGENT but not a team member at all. Access is purely stakeholder-based.
  </Accordion>

  <Accordion title="Login Flow">
    The context performs a two-step login for each persona:

    1. `POST /v1/auth/login` → identity token
    2. `POST /v1/auth/select-organization` → tenant token (org-scoped JWT)

    All subsequent API calls use the tenant token in the `Authorization: Bearer` header.
  </Accordion>
</AccordionGroup>

### Access Mode Reference

<Info>
Your `@CheckAccess` decorator uses two modes:

- **OR (default):** User has the required permission **OR** has stakeholder-level access on the resource → allowed
- **AND:** User must have the permission **AND** stakeholder access → allowed
</Info>

Resource action hierarchy:
- `ACCESS` — granted by `READ`, `WRITE`, or `FULL` access level
- `MODIFY` — granted by `WRITE` or `FULL`
- `DELETE` — granted by `FULL` only

## Test Data Seeding

<Steps>
  <Step title="Seed Users (one-time setup)">
    ```bash
    pnpm test:seed-users
    ```
    
    This runs `test/setup/seed-test-users.ts` which:
    - Creates the RBAC test personas in the database (idempotent)
    - Creates "Test Organization" with owner
    - Creates "Test Sales Team" with team leader and agent
    - Assigns org roles (Owner, Admin, Basic) and team roles
    - Uses RLS bypass to write directly via MikroORM
  </Step>

  <Step title="Seed Test Data (automatic)">
    Each test file calls `seedRbacTestData(app, personas, teamId)` in its `beforeAll`. Results are cached across files. This creates entities through the API:
    
    - **Leads:** One by BASIC, one by OWNER, one in pool, one with team as stakeholder, one with AGENT as READ-only stakeholder
    - **Contacts:** one by BASIC, one by OWNER
    - **Companies:** one by BASIC, one by OWNER
    - **Deals:** one by OWNER (from lead + contact)
    - **Events:** one linked to OWNER lead, one with BASIC as invitee
    - **Escalation rules:** one OWNER-created rule for RBAC matrices
  </Step>
</Steps>

<Warning>
Because seeding goes through the real API endpoints, all business logic (guards, auto-assignment, stakeholder creation) executes naturally, ensuring realistic test conditions.
</Warning>

## How to Run

<CodeGroup>

```bash Core E2E Tests
# Run main RBAC test suite
pnpm test:e2e

# Run with coverage
pnpm test:e2e:cov

# Run specific test file
pnpm test:e2e test/e2e/rbac/lead-rbac.e2e-spec.ts
```

```bash Queue Tests
# Run queue-specific tests with live pg-boss
pnpm test:e2e:queue

# Run distribution tests
pnpm test:e2e test/e2e/distribution/
```

```bash Setup Commands
# Seed test users (run once)
pnpm test:seed-users

# Cleanup test users
pnpm test:cleanup-users
```

</CodeGroup>

<Note>
The main E2E config (`jest-e2e.json`) excludes distribution and escalation queue tests via regex. These require the separate queue config with live pg-boss.
</Note>

## How to Extend

<Steps>
  <Step title="Add New Entity Matrix">
    1. Create `test/setup/matrices/your-entity-matrix.ts`
    2. Export from `test/setup/matrices/index.ts`
    3. Add test data seeding in `rbac-test-data.ts`
    4. Create `test/e2e/rbac/your-entity-rbac.e2e-spec.ts`
  </Step>

  <Step title="Add New Persona">
    1. Update `test/helpers/rbac-personas.ts`
    2. Update `test/setup/seed-test-users.ts`
    3. Update all matrix expectations
    4. Update `rbac-context.ts` login logic
  </Step>

  <Step title="Add New Endpoint">
    Add to the appropriate matrix file with expectations for all 5 personas:
    ```typescript
    {
      method: 'POST',

      description: 'Create your entity',
      expectations: {
        OWNER: 201,
        ADMIN: 201,

      },

    ```
  </Step>
</Steps>

## Methodology

<Tip>
See `test/TESTING_METHODOLOGY.md` for detailed guidance on analyzing controllers and DTOs before writing tests.
</Tip>

The testing methodology follows these principles:

1. **Permission-first design** — Every endpoint must declare its required permissions
2. **Stakeholder access testing** — Verify resource-level access controls work correctly
3. **Access level enforcement** — Test READ vs WRITE vs FULL access differentiation
4. **Negative testing** — Ensure 403s are returned when access should be denied
5. **Business logic integration** — Test through real API to catch guard interactions

## CI/CD Integration

<Warning>
Tests require a live PostgreSQL database. The CI environment must:
- Run database migrations
- Seed test users before running tests
- Use separate test database to avoid conflicts
- Clean up test data between runs
</Warning>

```yaml GitHub Actions Example
- name: Run E2E Tests
  run: |
    pnpm test:seed-users
    pnpm test:e2e
    pnpm test:cleanup-users
  env:
    DATABASE_URL: ${{ secrets.TEST_DATABASE_URL }}
```

The test suite is designed to be deterministic and can run in parallel CI environments with proper database isolation.
