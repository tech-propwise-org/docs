## Overview

The test suite focuses on **RBAC (Role-Based Access Control) E2E testing** — verifying that every API endpoint enforces the correct permissions for every user role.

**Stack:** Jest + supertest + `@nestjs/testing`

<Note>
Key design decisions include singleton app context, matrix-driven tests, and API-driven seeding to ensure comprehensive permission testing.
</Note>

### Key Design Principles

- **Singleton app context** — the NestJS app boots once and is shared across all test files
- **Matrix-driven tests** — a declarative permission matrix defines expected HTTP status codes per persona × endpoint
- **API-driven seeding** — test data is created through actual API endpoints so all business logic executes naturally

## Architecture

### Singleton Context Pattern

The app boots once for the entire test run, with all test files sharing the same NestJS app instance, authenticated tokens, and resolved `teamId`.

<Steps>
<Step title="Initialize Context">
First test file calls `getContext()` which boots the NestJS app (~3-5s), authenticates all 5 personas, and resolves teamId
</Step>
<Step title="Share Context">
Subsequent test files call `getContext()` and receive the cached instance instantly (0ms)
</Step>
<Step title="Cleanup">
Last file's `afterAll()` calls `closeContext()` when reference count hits 0
</Step>
</Steps>

```typescript
import { getContext, closeContext, RbacTestContext } from '../../setup/rbac-context';

let ctx: RbacTestContext;
beforeAll(async () => {
  ctx = await getContext();
}, 60000);
afterAll(async () => {
  await closeContext();
});
```

<Info>
The context replicates production configuration including ValidationPipe, URI versioning, and cookie parser.
</Info>

### Matrix Runner System

Instead of writing individual test blocks, define expectations declaratively:

```typescript
{
  method: 'GET',
  path: '/v1/leads/:leadByBasicId',
  description: 'View lead detail',
  expectations: {
    OWNER: 200,
    ADMIN: 200,
    TEAM_LEADER: 403,
    AGENT: 403,
    BASIC: 200,
  },
}
```

<CodeGroup>
```typescript Matrix Definition
const leadMatrix = [
  {
    method: 'GET',
    path: '/v1/leads/:leadByBasicId',
    description: 'View lead detail',
    expectations: {
      OWNER: 200,
      ADMIN: 200,
      TEAM_LEADER: 403,
      AGENT: 403,
      BASIC: 200,
    },
  },
  // Additional endpoints...
];
```

```typescript Runner Implementation
function runRbacMatrix(matrix, context) {
  matrix.forEach(endpoint => {
    Object.keys(endpoint.expectations).forEach(persona => {
      it(`${endpoint.description} - ${persona}`, async () => {
        const response = await request(app)
          .get(resolvePath(endpoint.path, testData))
          .set('Authorization', `Bearer ${personas[persona].token}`);
        
        expect(response.status).toBe(endpoint.expectations[persona]);
      });
    });
  });
}
```
</CodeGroup>

The runner automatically:
1. Iterates every endpoint in the matrix
2. Tests all 5 personas per endpoint
3. Resolves path placeholders with actual UUIDs
4. Makes HTTP requests with persona tokens
5. Asserts response status matches expectations

### Permission Profiles

<Tabs>
<Tab title="Owner/Admin">
**Bypass resource-level checks** via org permissions (`crm.view`, `crm.manage`, `sales.view`, `sales.manage`). Can access any entity in the organization.
</Tab>
<Tab title="Team Leader">
**Team-based access** with `team_crm.manage` and `team_sales.manage`. Can access entities where the team is a stakeholder.
</Tab>
<Tab title="Agent/Basic">
**Stakeholder-only access**. Can only access entities where they are personally a stakeholder. Used to test access level differentiation (READ vs WRITE vs FULL).
</Tab>
</Tabs>

### Authentication Flow

<Steps>
<Step title="Identity Login">
`POST /v1/auth/login` → receives identity token
</Step>
<Step title="Organization Selection">
`POST /v1/auth/select-organization` → receives tenant token (org-scoped JWT)
</Step>
<Step title="API Authorization">
All subsequent API calls use tenant token in `Authorization: Bearer` header
</Step>
</Steps>

### Access Mode Reference

<Note>
The `@CheckAccess` decorator supports two modes for permission evaluation.
</Note>

**OR Mode (default):** User has required permission **OR** stakeholder-level access → allowed

**AND Mode:** User must have permission **AND** stakeholder access → allowed

**Resource Action Hierarchy:**
- `ACCESS` — granted by `READ`, `WRITE`, or `FULL` access level
- `MODIFY` — granted by `WRITE` or `FULL`
- `DELETE` — granted by `FULL` only

## Test Data Seeding

### User Seeding (One-time Setup)

<Warning>
Run this command before executing tests for the first time.
</Warning>

```bash
pnpm test:seed-users
```

This command executes `test/setup/seed-test-users.ts` which:

<Steps>
<Step title="Create Personas">
Creates all RBAC test personas in the database (idempotent operation)
</Step>
<Step title="Setup Organization">
Creates "Test Organization" with owner assignment
</Step>
<Step title="Setup Team">
Creates "Test Sales Team" with team leader and agent
</Step>
<Step title="Assign Roles">
Assigns org roles (Owner, Admin, Basic) and team roles (Team Leader, Agent)
</Step>
</Steps>

### Test Data Seeding (Automatic)

Each test file calls `seedRbacTestData(app, personas, teamId)` in its `beforeAll` block. Results are cached across files.

<Tabs>
<Tab title="Leads">
- Lead by BASIC (auto-assigned as primary stakeholder with FULL access)
- Lead by OWNER (for admin-only tests)
- Lead in pool (empty stakeholders)
- Lead with team as stakeholder
- Lead with AGENT as READ-only stakeholder
</Tab>
<Tab title="Other Entities">
- **Contacts:** one by BASIC, one by OWNER
- **Companies:** one by BASIC, one by OWNER  
- **Deals:** one by OWNER (from lead + contact)
- **Events:** one linked to OWNER lead, one with BASIC as invitee
- **Escalation rules:** one OWNER-created rule
</Tab>
</Tabs>

<Info>
Seeding uses real API endpoints, ensuring all business logic (guards, auto-assignment, stakeholder creation) executes naturally.

## Methodology

### Test-Driven Permission Design

<Tip>
Follow this methodology when implementing new features with RBAC requirements.
</Tip>

<Steps>
<Step title="Analyze Controller">
Review controller methods and identify permission requirements
</Step>
<Step title="Define Matrix">
Create permission matrix based on business rules and user roles
</Step>
<Step title="Implement Guards">
Add `@CheckAccess` decorators with appropriate permissions
</Step>
<Step title="Write Tests">
Use matrix runner to generate comprehensive test coverage
</Step>
<Step title="Validate Results">
Run tests to ensure permissions work as expected
</Step>
</Steps>

### Permission Testing Checklist

<Check>Org-level permissions bypass resource checks correctly</Check>
<Check>Team-level permissions respect team stakeholder relationships</Check>
<Check>Stakeholder access levels (READ/WRITE/FULL) enforce properly</Check>
<Check>Admin-only operations require `system.admin` permission</Check>
<Check>Endpoint returns appropriate HTTP status codes</Check>
