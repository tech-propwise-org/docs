The test suite focuses on **RBAC (Role-Based Access Control) E2E testing** — verifying that every API endpoint enforces the correct permissions for every user role.

**Stack:** Jest + supertest + `@nestjs/testing`

<Info>
**Key design decisions:**
- **Singleton app context** — the NestJS app boots once and is shared across all test files
- **Matrix-driven tests** — a declarative permission matrix defines expected HTTP status codes per persona × endpoint
- **API-driven seeding** — test data is created through actual API endpoints so all business logic executes naturally
</Info>

## Architecture

### Singleton Context

The app boots once for the entire test run. All test files share the same NestJS app instance, authenticated tokens, and resolved `teamId`.

<Steps>
<Step title="Jest starts">
`lead-rbac.e2e-spec.ts` calls `getContext()`
</Step>
<Step title="First call">
Boots NestJS app (~3-5s), logs in 5 personas, resolves teamId
</Step>
<Step title="Subsequent calls">
Returns cached instance instantly (0ms)
</Step>
<Step title="Cleanup">
Last file's `afterAll()` calls `closeContext()` → app shuts down
</Step>
</Steps>

```typescript
import { getContext, closeContext, RbacTestContext } from '../../setup/rbac-context';

let ctx: RbacTestContext;
beforeAll(async () => {
  ctx = await getContext();
}, 60000);
afterAll(async () => {
  await closeContext();
});
```

<Note>
The context replicates production config from `main.ts`: ValidationPipe, URI versioning (`/v1/...`), and cookie parser.
</Note>

### Matrix Runner

Instead of writing individual `it()` blocks, you define expectations declaratively:

```typescript
{
  method: 'GET',
  path: '/v1/leads/:leadByBasicId',
  description: 'View lead detail',
  expectations: {
    OWNER: 200,
    ADMIN: 200,
    TEAM_LEADER: 403,
    AGENT: 403,
    BASIC: 200,
  },
}
```

<Steps>
<Step title="Iterate endpoints">
The runner processes every endpoint in the matrix
</Step>
<Step title="Test all personas">
For each endpoint, iterates all 5 personas
</Step>
<Step title="Resolve placeholders">
Path placeholders (`:leadByBasicId`) are resolved to actual UUIDs
</Step>
<Step title="Make requests">
HTTP requests are made with the persona's tenant token
</Step>
<Step title="Assert results">
Response status is compared to expected value with clear failure messages
</Step>
</Steps>

## RBAC Personas

All personas are defined in `test/helpers/rbac-personas.ts` as a single source of truth.

| Key | Email | Org Role | Team | Description |
|-----|-------|----------|------|-------------|
| `OWNER` | owner@propwise.com | Owner | Any | All permissions including `system.owner` |
| `ADMIN` | admin@propwise.com | Admin | Any | All non-sensitive permissions (includes `system.admin`) |
| `TEAM_LEADER` | teamleader@propwise.com | Basic | Team Leader in Test Sales Team | All team permissions (`team_crm.*`, `team_sales.*`, `team.admin`) |
| `AGENT` | agent@propwise.com | Basic | Agent in Test Sales Team | No team permissions by default |
| `BASIC` | basic@propwise.com | Basic | None | No permissions — relies entirely on stakeholder access |

### Permission Profiles

<Tabs>
<Tab title="OWNER / ADMIN">
Bypass resource-level checks via org permissions (`crm.view`, `crm.manage`, `sales.view`, `sales.manage`, etc.). Can access any entity in the org.
</Tab>
<Tab title="TEAM_LEADER">
No org permissions, but has `team_crm.manage` and `team_sales.manage` in the team. Can access entities where the team is a stakeholder.
</Tab>
<Tab title="AGENT">
No org permissions, no team permissions. Can only access entities where they are personally a stakeholder. Used to test access level differentiation (READ vs WRITE vs FULL).
</Tab>
<Tab title="BASIC">
Same as AGENT but not a team member at all. Access is purely stakeholder-based.
</Tab>
</Tabs>

### Login Flow

The context performs a two-step login for each persona:

<Steps>
<Step title="Identity login">
`POST /v1/auth/login` → identity token
</Step>
<Step title="Organization selection">
`POST /v1/auth/select-organization` → tenant token (org-scoped JWT)
</Step>
</Steps>

<Warning>
All subsequent API calls use the tenant token in the `Authorization: Bearer` header.
</Warning>

### Access Mode Reference

Your `@CheckAccess` decorator uses two modes:

- **OR (default):** User has the required permission **OR** has stakeholder-level access on the resource → allowed
- **AND:** User must have the permission **AND** stakeholder access → allowed

<Info>
**Resource action hierarchy:**
- `ACCESS` — granted by `READ`, `WRITE`, or `FULL` access level
- `MODIFY` — granted by `WRITE` or `FULL`
- `DELETE` — granted by `FULL` only
</Info>

## Test Data Seeding

### Step 1: Seed Users (One-time Setup)

<CodeGroup>
```bash Terminal
pnpm test:seed-users
```
</CodeGroup>

This runs `test/setup/seed-test-users.ts` which:

<Check>Creates the RBAC test personas in the database (idempotent)</Check>
<Check>Creates "Test Organization" with owner</Check>
<Check>Creates "Test Sales Team" with team leader and agent</Check>
<Check>Assigns org roles (Owner, Admin, Basic) and team roles</Check>
<Check>Uses RLS bypass to write directly via MikroORM</Check>

### Step 2: Seed Test Data (Automatic)

Each test file calls `seedRbacTestData(app, personas, teamId)` in its `beforeAll`. Results are cached across files.

<Tabs>
<Tab title="Leads">
- One by BASIC (auto-assigned as primary stakeholder with FULL access)
- One by OWNER (for admin-only tests)  
- One in pool (empty stakeholders)
- One with team as stakeholder
- One with AGENT as READ-only stakeholder
</Tab>
<Tab title="Other Entities">
- **Contacts:** one by BASIC, one by OWNER
- **Companies:** one by BASIC, one by OWNER
- **Deals:** one by OWNER (from lead + contact)
- **Events:** one linked to OWNER lead, one with BASIC as invitee
- **Escalation rules:** one OWNER-created rule
</Tab>
</Tabs>

<Warning>
Seeding goes through the real API endpoints, so all business logic (guards, auto-assignment, stakeholder creation) executes naturally.
</Warning>

## Methodology

<Tip>
See `test/TESTING_METHODOLOGY.md` for detailed guidance on analyzing controllers and DTOs before writing tests.
</Tip>

The testing approach follows these principles:

1. **Permission-driven:** Tests verify authorization before functionality
2. **Declarative:** Matrix definitions are easier to maintain than imperative test code
3. **Realistic:** Test data creation mirrors real user workflows
4. **Isolated:** Each test file can run independently
5. **Fast:** Singleton context minimizes bootstrap overhead

## CI/CD Integration

<Warning>
Tests require a live PostgreSQL database and Redis instance. Ensure your CI environment provides these dependencies.
</Warning>

The test suite integrates with CI/CD pipelines by:

- Using environment variables for database connections
- Providing clear exit codes for pass/fail status
- Generating JUnit XML reports (when configured)
- Supporting parallel execution across multiple workers

<Note>
The queue-specific tests (`test:e2e:queue`) require pg-boss and should run separately from core RBAC tests to avoid interference.
</Note>
