# Testing Specification v11

Architecture, conventions, and run guide for the PropWise CRM backend test suite.

## Overview

The test suite focuses on **RBAC (Role-Based Access Control) E2E testing** — verifying that every API endpoint enforces the correct permissions for every user role.

<Info>
**Stack:** Jest + supertest + `@nestjs/testing`
</Info>

### Key Design Decisions

<CardGroup cols={2}>
  <Card title="Singleton App Context" icon="server">
    The NestJS app boots once and is shared across all test files (equivalent to NestJS `TestSuite` pattern, implemented manually since `TestSuite` requires NestJS 11+)
  </Card>
  <Card title="Matrix-Driven Tests" icon="grid">
    A declarative permission matrix defines expected HTTP status codes per persona × endpoint. A runner generates `it()` blocks automatically
  </Card>
  <Card title="API-Driven Seeding" icon="seedling">
    Test data is created through actual API endpoints so all business logic (guards, auto-assignment, stakeholder creation) executes naturally
  </Card>
  <Card title="Live Queue Testing" icon="clock">
    Separate Jest config for distribution, escalation, and queue-e2e misc specs with live pg-boss integration
  </Card>
</CardGroup>

## Architecture

### Singleton Context

<Note>
The app boots once for the entire test run. All test files share the same NestJS app instance, authenticated tokens, and resolved `teamId`.
</Note>

```typescript
import { getContext, closeContext, RbacTestContext } from '../../setup/rbac-context';

let ctx: RbacTestContext;
beforeAll(async () => {
  ctx = await getContext();
}, 60000);
afterAll(async () => {
  await closeContext();
});
```

<Steps>
  <Step title="Jest Starts">
    First test file calls `getContext()`
  </Step>
  <Step title="App Bootstrap">
    NestJS app boots (~3-5s), logs in 5 personas, resolves teamId
  </Step>
  <Step title="Context Sharing">
    Subsequent test files get cached instance instantly (0ms)
  </Step>
  <Step title="Cleanup">
    Last file's `afterAll()` shuts down app when reference count hits 0
  </Step>
</Steps>

The context replicates production config from `main.ts`:

- `ValidationPipe` (whitelist, transform, implicit conversion)
- URI versioning (`/v1/...`)
- Cookie parser

### Matrix Runner

<Tip>
Instead of writing individual `it()` blocks, define expectations declaratively
</Tip>

```typescript
{
  method: 'GET',
  path: '/v1/leads/:leadByBasicId',
  description: 'View lead detail',
  expectations: {
    OWNER: 200,
    ADMIN: 200,
    TEAM_LEADER: 403,
    AGENT: 403,
    BASIC: 200,
  },
}
```

The runner calls `runRbacMatrix()` which:

1. Iterates every endpoint in the matrix
2. For each endpoint, iterates all 5 personas
3. Resolves path placeholders (`:leadByBasicId` → actual UUID)
4. Makes the HTTP request with the persona's tenant token
5. Asserts the response status matches the expected value
6. Produces clear failure messages with the actual response body

<Check>
One matrix entry = 5 generated test cases
</Check>

## RBAC Personas

All personas are defined in `test/helpers/rbac-personas.ts` and share a single source of truth.

| Key           | Email                   | Org Role | Team                           | Description                                                       |
| ------------- | ----------------------- | -------- | ------------------------------ | ----------------------------------------------------------------- |
| `OWNER`       | owner@propwise.com      | Owner    | Any                            | All permissions including `system.owner`                          |
| `ADMIN`       | admin@propwise.com      | Admin    | Any                            | All non-sensitive permissions (includes `system.admin`)           |
| `TEAM_LEADER` | teamleader@propwise.com | Basic    | Team Leader in Test Sales Team | All team permissions (`team_crm.*`, `team_sales.*`, `team.admin`) |
| `AGENT`       | agent@propwise.com      | Basic    | Agent in Test Sales Team       | No team permissions by default                                    |
| `BASIC`       | basic@propwise.com      | Basic    | None                           | No permissions — relies entirely on stakeholder access            |

### Permission Profiles

<Tabs>
  <Tab title="OWNER / ADMIN">
    Bypass resource-level checks via org permissions (`crm.view`, `crm.manage`, `sales.view`, `sales.manage`). Can access any entity in the org.
  </Tab>
  <Tab title="TEAM_LEADER">
    No org permissions, but has `team_crm.manage` and `team_sales.manage` in the team. Can access entities where the team is a stakeholder.
  </Tab>
  <Tab title="AGENT">
    No org permissions, no team permissions. Can only access entities where they are personally a stakeholder. Used to test access level differentiation (READ vs WRITE vs FULL).
  </Tab>
  <Tab title="BASIC">
    Same as AGENT but not a team member at all. Access is purely stakeholder-based.
  </Tab>
</Tabs>

### Login Flow

The context performs a two-step login for each persona:

<Steps>
  <Step title="Identity Login">
    `POST /v1/auth/login` → identity token
  </Step>
  <Step title="Organization Selection">
    `POST /v1/auth/select-organization` → tenant token (org-scoped JWT)
  </Step>
</Steps>

All subsequent API calls use the tenant token in the `Authorization: Bearer` header.

### Access Mode Reference

<Info>
Your `@CheckAccess` decorator uses two modes:

- **OR (default):** User has the required permission **OR** has stakeholder-level access on the resource → allowed
- **AND:** User must have the permission **AND** stakeholder access → allowed
</Info>

Resource action hierarchy:
- `ACCESS` — granted by `READ`, `WRITE`, or `FULL` access level
- `MODIFY` — granted by `WRITE` or `FULL`
- `DELETE` — granted by `FULL` only

## Test Data Seeding

### Step 1: Seed Users (One-Time)

<Warning>
Run this before executing tests for the first time
</Warning>

```bash
pnpm test:seed-users
```

This runs `test/setup/seed-test-users.ts` which:

- Creates the RBAC test personas in the database (idempotent — skips if they exist)
- Creates "Test Organization" with owner
- Creates "Test Sales Team" with team leader and agent
- Assigns org roles (Owner, Admin, Basic) and team roles (Team Leader, Agent)
- Uses RLS bypass to write directly via MikroORM

### Step 2: Seed Test Data (Automatic)

Each test file calls `seedRbacTestData(app, personas, teamId)` in its `beforeAll`. Results are cached across files.

<Note>
Test data is created through the API to ensure all business logic executes naturally
</Note>

This creates entities through the API:

<Tabs>
  <Tab title="Leads">
    - One by BASIC (auto-assigned as primary stakeholder with FULL access)
    - One by OWNER (for admin-only tests)
    - One in pool (empty stakeholders)
    - One with team as stakeholder (OWNER creates, assigns team)
    - One with AGENT as READ-only stakeholder
  </Tab>
  
  <Tab title="Other Entities">
    - **Contacts:** one by BASIC, one by OWNER
    - **Companies:** one by BASIC, one by OWNER
    - **Deals:** one by OWNER (from lead + contact)
    - **Events:** one linked to the OWNER lead, one with BASIC as invitee
    - **Escalation rules:** one OWNER-created rule
  </Tab>
</Tabs>

## How to Extend

### Adding New Entity Tests

<Steps>
  <Step title="Create Matrix File">
    Create `test/setup/matrices/new-entity-matrix.ts`:
    
    ```typescript
    import { EndpointExpectation } from '../rbac-matrix-types';
    
    export const NEW_ENTITY_MATRIX: EndpointExpectation[] = [
      {
        method: 'GET',
        path: '/v1/new-entities/:entityId',
        description: 'View entity detail',
        expectations: {
          OWNER: 200,
          ADMIN: 200,
          TEAM_LEADER: 403,
          AGENT: 403,
          BASIC: 200,
        },
      },
    ];
    ```
  </Step>
  
  <Step title="Update Barrel Export">
    Add to `test/setup/matrices/index.ts`:
    
    ```typescript
    export * from './new-entity-matrix';
    ```
  </Step>
  
  <Step title="Create Test File">
    Create `test/e2e/rbac/new-entity-rbac.e2e-spec.ts`:
    
    ```typescript
    import { getContext, closeContext } from '../../setup/rbac-context';
    import { seedRbacTestData } from '../../setup/rbac-test-data';
    import { runRbacMatrix } from '../../setup/rbac-runner';
    import { NEW_ENTITY_MATRIX } from '../../setup/matrices';
    
    describe('New Entity RBAC', () => {
      let ctx;
      
      beforeAll(async () => {
        ctx = await getContext();
        await seedRbacTestData(ctx.app, ctx.personas, ctx.teamId);
      }, 60000);
      
      afterAll(async () => {
        await closeContext();
      });
      
      runRbacMatrix('New Entity Access Control', NEW_ENTITY_MATRIX, () => ctx);
    });
    ```
  </Step>
  
  <Step title="Update Test Data Seeder">
    Add entity creation to `test/setup/rbac-test-data.ts` if needed
  </Step>
</Steps>

## Methodology

<Tip>
See `test/TESTING_METHODOLOGY.md` for detailed guidance on analyzing controllers and DTOs before writing tests
</Tip>

The testing approach follows these principles:

1. **Permission-First Design** — Start with the permission matrix, then implement endpoints
2. **Stakeholder Access Patterns** — Test both org-level permissions and stakeholder-based access
3. **Access Level Differentiation** — Verify READ vs WRITE vs FULL access levels work correctly
4. **Realistic Data Flow** — Use API-driven seeding to replicate production behavior
5. **Comprehensive Coverage** — Test every persona × endpoint combination
