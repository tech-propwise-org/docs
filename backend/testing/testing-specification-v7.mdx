## Architecture

### Singleton Context

<Note>
The app boots once for the entire test run. All test files share the same NestJS app instance, authenticated tokens, and resolved `teamId`.
</Note>

```typescript
import { getContext, closeContext, RbacTestContext } from '../../setup/rbac-context';

let ctx: RbacTestContext;
beforeAll(async () => {
  ctx = await getContext();
}, 60000);
afterAll(async () => {
  await closeContext();
});
```

<Steps>
  <Step title="Jest starts">
    First test file calls `getContext()`
  </Step>
  <Step title="App boots">
    NestJS app initializes (~3-5s), logs in 5 personas, resolves teamId
  </Step>
  <Step title="Context cached">
    Subsequent files get cached instance instantly (0ms)
  </Step>
  <Step title="Cleanup">
    Last file's `afterAll()` shuts down app when reference count hits 0
  </Step>
</Steps>

The context replicates production config from `main.ts`:
- `ValidationPipe` (whitelist, transform, implicit conversion)
- URI versioning (`/v1/...`)
- Cookie parser

### Matrix Runner

Instead of writing individual `it()` blocks, you define expectations declaratively:

```typescript
{
  method: 'GET',
  path: '/v1/leads/:leadByBasicId',
  description: 'View lead detail',
  expectations: {
    OWNER: 200,
    ADMIN: 200,
    TEAM_LEADER: 403,
    AGENT: 403,
    BASIC: 200,
  },
}
```

<Steps>
  <Step title="Matrix iteration">
    Runner calls `runRbacMatrix()` and iterates every endpoint
  </Step>
  <Step title="Persona testing">
    For each endpoint, tests all 5 personas
  </Step>
  <Step title="Path resolution">
    Resolves placeholders (`:leadByBasicId` в†’ actual UUID)
  </Step>
  <Step title="Request execution">
    Makes HTTP request with persona's tenant token
  </Step>
  <Step title="Assertion">
    Verifies response status matches expected value
  </Step>
</Steps>

<Tip>
One matrix entry generates 5 test cases automatically with clear failure messages including response body.
</Tip>

## RBAC Personas

All personas are defined in `test/helpers/rbac-personas.ts` with a single source of truth.

| Key           | Email                   | Org Role | Team                           | Description                                                       |
| ------------- | ----------------------- | -------- | ------------------------------ | ----------------------------------------------------------------- |
| `OWNER`       | owner@propwise.com      | Owner    | Any                            | All permissions including `system.owner`                          |
| `ADMIN`       | admin@propwise.com      | Admin    | Any                            | All non-sensitive permissions (includes `system.admin`)           |
| `TEAM_LEADER` | teamleader@propwise.com | Basic    | Team Leader in Test Sales Team | All team permissions (`team_crm.*`, `team_sales.*`, `team.admin`) |
| `AGENT`       | agent@propwise.com      | Basic    | Agent in Test Sales Team       | No team permissions by default                                    |
| `BASIC`       | basic@propwise.com      | Basic    | None                           | No permissions вЂ” relies entirely on stakeholder access            |

### Permission Profiles

<AccordionGroup>
  <Accordion title="OWNER / ADMIN">
    Bypass resource-level checks via org permissions (`crm.view`, `crm.manage`, `sales.view`, `sales.manage`, etc.). Can access any entity in the org.
  </Accordion>
  
  <Accordion title="TEAM_LEADER">
    No org permissions, but has `team_crm.manage` and `team_sales.manage` in the team. Can access entities where the team is a stakeholder.
  </Accordion>
  
  <Accordion title="AGENT">
    No org permissions, no team permissions. Can only access entities where they are personally a stakeholder. Used to test access level differentiation (READ vs WRITE vs FULL).
  </Accordion>
  
  <Accordion title="BASIC">
    Same as AGENT but not a team member at all. Access is purely stakeholder-based.
  </Accordion>
</AccordionGroup>

### Login Flow

The context performs a two-step login for each persona:

<Steps>
  <Step title="Identity authentication">
    `POST /v1/auth/login` в†’ identity token
  </Step>
  <Step title="Organization selection">
    `POST /v1/auth/select-organization` в†’ tenant token (org-scoped JWT)
  </Step>
  <Step title="API calls">
    All subsequent API calls use the tenant token in `Authorization: Bearer` header
  </Step>
</Steps>

### Access Mode Reference

Your `@CheckAccess` decorator uses two modes:

<Note>
- **OR (default):** User has the required permission **OR** has stakeholder-level access on the resource в†’ allowed
- **AND:** User must have the permission **AND** stakeholder access в†’ allowed
</Note>

**Resource action hierarchy:**
- `ACCESS` вЂ” granted by `READ`, `WRITE`, or `FULL` access level
- `MODIFY` вЂ” granted by `WRITE` or `FULL`
- `DELETE` вЂ” granted by `FULL` only

## Test Data Seeding

### Step 1: Seed Users (One-time Setup)

<CodeGroup>
```bash Terminal
pnpm test:seed-users
```
</CodeGroup>

This runs `test/setup/seed-test-users.ts` which:

<Check>Creates the 5 test users in the database (idempotent вЂ” skips if they exist)</Check>
<Check>Creates "Test Organization" with owner</Check>
<Check>Creates "Test Sales Team" with team leader and agent</Check>
<Check>Assigns org roles (Owner, Admin, Basic) and team roles (Team Leader, Agent)</Check>
<Check>Uses RLS bypass to write directly via MikroORM</Check>

### Step 2: Seed Test Data (Automatic)

Each test file calls `seedRbacTestData(app, personas, teamId)` in its `beforeAll`. Results are cached across files.

<Warning>
Because seeding goes through the real API, all guards, auto-assignment rules, and stakeholder creation run. The test data matches what a real user would produce.
</Warning>

## Methodology

<Tip>
See `test/TESTING_METHODOLOGY.md` for detailed guidance on analyzing controllers and DTOs before writing tests.
</Tip>

Key principles:
- **API-first testing** вЂ” Use real HTTP requests through the full application stack
- **Permission-driven** вЂ” Every test validates authorization, not just functionality  
- **Stakeholder-aware** вЂ” Tests cover both org permissions and stakeholder-based access
- **Realistic data** вЂ” Entities created through business logic, not direct DB writes
