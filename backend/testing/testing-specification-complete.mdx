# Testing Specification (Complete)

Architecture, conventions, and run guide for the PropWise CRM backend test suite.

<Note>
The test suite focuses on **RBAC (Role-Based Access Control) E2E testing** — verifying that every API endpoint enforces the correct permissions for every user role.
</Note>

## Overview

**Stack:** Jest + supertest + `@nestjs/testing`

**Key design decisions:**

- **Singleton app context** — the NestJS app boots once and is shared across all test files (equivalent to NestJS `TestSuite` pattern, implemented manually since `TestSuite` requires NestJS 11+)
- **Matrix-driven tests** — a declarative permission matrix defines expected HTTP status codes per persona × endpoint. A runner generates `it()` blocks automatically
- **API-driven seeding** — test data is created through the actual API endpoints so all business logic (guards, auto-assignment, stakeholder creation) executes naturally

## Architecture

### Singleton Context

The app boots once for the entire test run. All test files share the same NestJS app instance, authenticated tokens, and resolved `teamId`.

```
Jest starts
  → lead-rbac.e2e-spec.ts calls getContext()
    → First call: boots NestJS app (~3-5s), logs in 5 personas, resolves teamId
    → Returns { app, personas, teamId }
  → deal-rbac.e2e-spec.ts calls getContext()
    → Returns cached instance instantly (0ms)
  → ... same for contact and company files
  → Last file's afterAll() calls closeContext()
    → Reference count hits 0 → app shuts down
```

<Info>
The context replicates production config from `main.ts` including `ValidationPipe`, URI versioning (`/v1/...`), and cookie parser.
</Info>

### Usage Pattern

```typescript
import { getContext, closeContext, RbacTestContext } from '../../setup/rbac-context';

let ctx: RbacTestContext;
beforeAll(async () => { ctx = await getContext(); }, 60000);
afterAll(async () => { await closeContext(); });
```

### Matrix Runner

Instead of writing individual `it()` blocks, you define expectations declaratively:

```typescript
{
  method: 'GET',
  path: '/v1/leads/:leadByBasicId',
  description: 'View lead detail',
  expectations: {
    OWNER: 200,
    ADMIN: 200,
    TEAM_LEADER: 403,
    AGENT: 403,
    BASIC: 200,
  },
}
```

The runner calls `runRbacMatrix()` which:

<Steps>
<Step title="Iterate endpoints">
For each endpoint in the matrix
</Step>
<Step title="Iterate personas">
For each endpoint, test all 5 personas
</Step>
<Step title="Resolve placeholders">
Replace path placeholders (`:leadByBasicId` → actual UUID)
</Step>
<Step title="Make requests">
Execute HTTP request with persona's tenant token
</Step>
<Step title="Assert responses">
Verify response status matches expected value with clear failure messages
</Step>
</Steps>

<Tip>
One matrix entry generates 5 test cases automatically - one for each persona.
</Tip>

| Key | Email | Org Role | Team | Description |
|-----|-------|----------|------|-------------|
| `OWNER` | owner@propwise.com | Owner | Any | All permissions including `system.owner` |
| `ADMIN` | admin@propwise.com | Admin | Any | All non-sensitive permissions (includes `system.admin`) |
| `TEAM_LEADER` | teamleader@propwise.com | Basic | Team Leader in Test Sales Team | All team permissions (`team_crm.*`, `team_sales.*`, `team.admin`) |
| `AGENT` | agent@propwise.com | Basic | Agent in Test Sales Team | No team permissions by default |

### Permission Profiles

<Tabs>
<Tab title="OWNER / ADMIN">
Bypass resource-level checks via org permissions (`crm.view`, `crm.manage`, `sales.view`, `sales.manage`, etc.). Can access any entity in the org.
</Tab>
<Tab title="TEAM_LEADER">
No org permissions, but has `team_crm.manage` and `team_sales.manage` in the team. Can access entities where the team is a stakeholder.
</Tab>
<Tab title="AGENT">
No org permissions, no team permissions. Can only access entities where they are personally a stakeholder. Used to test access level differentiation (READ vs WRITE vs FULL).
</Tab>
<Tab title="BASIC">
Same as AGENT but not a team member at all. Access is purely stakeholder-based.
</Tab>
</Tabs>

### Login Flow

The context performs a two-step login for each persona:

<Steps>
<Step title="Identity Login">
`POST /v1/auth/login` → identity token
</Step>
<Step title="Organization Selection">
`POST /v1/auth/select-organization` → tenant token (org-scoped JWT)
</Step>
</Steps>

<Note>
All subsequent API calls use the tenant token in the `Authorization: Bearer` header.
</Note>

### Access Mode Reference

<Warning>
Your `@CheckAccess` decorator uses two modes that affect authorization logic.
</Warning>

- **OR (default):** User has the required permission **OR** has stakeholder-level access on the resource → allowed
- **AND:** User must have the permission **AND** stakeholder access → allowed

Resource action hierarchy:
- `ACCESS` — granted by `READ`, `WRITE`, or `FULL` access level
- `MODIFY` — granted by `WRITE` or `FULL`
- `DELETE` — granted by `FULL` only

## Test Data Seeding

### Step 1: Seed Users (One-time Setup)

<CodeGroup>
```bash Terminal
pnpm test:seed-users
```
</CodeGroup>

This runs `test/setup/seed-test-users.ts` which:

<Check>Creates the 5 test users in the database (idempotent — skips if they exist)</Check>
<Check>Creates "Test Organization" with owner</Check>
<Check>Creates "Test Sales Team" with team leader and agent</Check>
<Check>Assigns org roles (Owner, Admin, Basic) and team roles (Team Leader, Agent)</Check>
<Check>Uses RLS bypass to write directly via MikroORM</Check>

### Step 2: Seed Test Data (Automatic)

Each test file calls `seedRbacTestData(app, personas, teamId)` in its `beforeAll`. Results are cached across files. This creates entities through the API:

**Leads:**
- One by BASIC (auto-assigned as primary stakeholder with FULL access)
- One by OWNER (for admin-only tests)
- One in pool (empty stakeholders)
- One with team as stakeholder (OWNER creates, assigns team — TEAM_LEADER gets access)
- One with AGENT as READ-only stakeholder (OWNER creates, adds AGENT with `accessLevel: read`)

**Other Entities:**
- **Contacts:** one by BASIC, one by OWNER
- **Companies:** one by BASIC, one by OWNER
- **Deals:** one by OWNER (from lead + contact)

<Info>
Because seeding goes through the real API, all guards, auto-assignment rules, and stakeholder creation run. The test data matches what a real user would produce.
</Info>

## Methodology

<Tip>
The test suite follows a systematic approach to ensure comprehensive coverage of RBAC scenarios.
</Tip>

### Controller Analysis

Before writing tests:

1. **Review `@CheckAccess` decorators** - understand required permissions and access modes
2. **Identify stakeholder patterns** - determine how entities become accessible to users
3. **Map permission hierarchy** - understand org vs team vs stakeholder access paths
4. **Document edge cases** - note special behaviors or business rules

### Test Coverage Strategy

- **Permission-based access** - org and team level permissions
- **Stakeholder-based access** - entity-specific access levels (READ, WRITE, FULL)
- **Access level enforcement** - verify that READ access blocks modifications
- **Admin-only operations** - test system-level permissions
- **Pool scenarios** - entities with no stakeholders

## CI/CD Integration

The test suite is designed for CI/CD environments:

- **Database isolation** - tests use separate test database
- **Deterministic seeding** - predictable test data creation
- **Singleton pattern** - optimized for parallel execution
- **Clear failure messages** - detailed error reporting for debugging

<Warning>
Ensure your CI environment has access to the required environment variables and a properly configured test database.
</Warning>
