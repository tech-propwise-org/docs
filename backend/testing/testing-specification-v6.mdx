# Testing Specification v6

Architecture, conventions, and run guide for the PropWise CRM backend test suite.

<Note>
The test suite focuses on **RBAC (Role-Based Access Control) E2E testing** — verifying that every API endpoint enforces the correct permissions for every user role.
</Note>

**Stack:** Jest + supertest + `@nestjs/testing`

**Key design decisions:**
- **Singleton app context** — the NestJS app boots once and is shared across all test files
- **Matrix-driven tests** — a declarative permission matrix defines expected HTTP status codes per persona × endpoint
- **API-driven seeding** — test data is created through actual API endpoints so all business logic executes naturally

## Architecture

### Singleton Context

<Info>
The app boots once for the entire test run. All test files share the same NestJS app instance, authenticated tokens, and resolved `teamId`.
</Info>

The singleton context pattern provides significant performance improvements:

```
Jest starts
  → lead-rbac.e2e-spec.ts calls getContext()
    → First call: boots NestJS app (~3-5s), logs in 5 personas, resolves teamId
    → Returns { app, personas, teamId }
  → deal-rbac.e2e-spec.ts calls getContext()
    → Returns cached instance instantly (0ms)
  → ... same for contact and company files
  → Last file's afterAll() calls closeContext()
    → Reference count hits 0 → app shuts down
```

**Implementation:**

```typescript
import { getContext, closeContext, RbacTestContext } from '../../setup/rbac-context';

let ctx: RbacTestContext;
beforeAll(async () => { ctx = await getContext(); }, 60000);
afterAll(async () => { await closeContext(); });
```

The context replicates production config from `main.ts`:
- `ValidationPipe` (whitelist, transform, implicit conversion)
- URI versioning (`/v1/...`)
- Cookie parser

### Matrix Runner

<Tip>
Instead of writing individual `it()` blocks, you define expectations declaratively using permission matrices.
</Tip>

Define expectations as a matrix:

```typescript
{
  method: 'GET',
  path: '/v1/leads/:leadByBasicId',
  description: 'View lead detail',
  expectations: {
    OWNER: 200,
    ADMIN: 200,
    TEAM_LEADER: 403,
    AGENT: 403,
    BASIC: 200,
  },
}
```

The runner calls `runRbacMatrix()` which:

<Steps>
<Step title="Iterate endpoints">
Loop through every endpoint defined in the matrix
</Step>

<Step title="Test personas">
For each endpoint, test all 5 personas (OWNER, ADMIN, TEAM_LEADER, AGENT, BASIC)
</Step>

<Step title="Resolve placeholders">
Replace path placeholders (`:leadByBasicId` → actual UUID)
</Step>

<Step title="Execute requests">
Make HTTP requests with the persona's tenant token
</Step>

<Step title="Assert responses">
Verify response status matches expected value with clear failure messages
</Step>
</Steps>

## RBAC Personas

All personas are defined in `test/helpers/rbac-personas.ts` as a single source of truth:

| Persona | Email | Org Role | Team | Description |
|---------|-------|----------|------|-------------|
| `OWNER` | owner@propwise.com | Owner | Any | All permissions including `system.owner` |
| `ADMIN` | admin@propwise.com | Admin | Any | All non-sensitive permissions (includes `system.admin`) |
| `TEAM_LEADER` | teamleader@propwise.com | Basic | Team Leader in Test Sales Team | All team permissions (`team_crm.*`, `team_sales.*`, `team.admin`) |
| `AGENT` | agent@propwise.com | Basic | Agent in Test Sales Team | No team permissions by default |
| `BASIC` | basic@propwise.com | Basic | None | No permissions — relies entirely on stakeholder access |

### Permission Profiles

<Tabs>
<Tab title="OWNER / ADMIN">
Bypass resource-level checks via org permissions (`crm.view`, `crm.manage`, `sales.view`, `sales.manage`, etc.). Can access any entity in the org.
</Tab>

<Tab title="TEAM_LEADER">
No org permissions, but has `team_crm.manage` and `team_sales.manage` in the team. Can access entities where the team is a stakeholder.
</Tab>

<Tab title="AGENT">
No org permissions, no team permissions. Can only access entities where they are personally a stakeholder. Used to test access level differentiation (READ vs WRITE vs FULL).
</Tab>

<Tab title="BASIC">
Same as AGENT but not a team member at all. Access is purely stakeholder-based.
</Tab>
</Tabs>

### Login Flow

<Steps>
<Step title="Identity authentication">
`POST /v1/auth/login` → identity token
</Step>

<Step title="Organization selection">
`POST /v1/auth/select-organization` → tenant token (org-scoped JWT)
</Step>

<Step title="API requests">
All subsequent API calls use the tenant token in the `Authorization: Bearer` header
</Step>
</Steps>

### Access Mode Reference

<Note>
Your `@CheckAccess` decorator uses two modes:

- **OR (default):** User has the required permission **OR** has stakeholder-level access on the resource → allowed
- **AND:** User must have the permission **AND** stakeholder access → allowed
</Note>

Resource action hierarchy:
- `ACCESS` — granted by `READ`, `WRITE`, or `FULL` access level
- `MODIFY` — granted by `WRITE` or `FULL`
- `DELETE` — granted by `FULL` only

## Test Data Seeding

### Step 1: Seed Users (One-Time Setup)

<Steps>
<Step title="Run the seed command">
```bash
pnpm test:seed-users
```
</Step>

<Step title="What gets created">
This runs `test/setup/seed-test-users.ts` which:
- Creates the 5 test users in the database (idempotent — skips if they exist)
- Creates "Test Organization" with owner
- Creates "Test Sales Team" with team leader and agent
- Assigns org roles (Owner, Admin, Basic) and team roles (Team Leader, Agent)
- Uses RLS bypass to write directly via MikroORM
</Step>
</Steps>

### Step 2: Seed Test Data (Automatic)

Each test file calls `seedRbacTestData(app, personas, teamId)` in its `beforeAll`. Results are cached across files.

<Warning>
Test data creation goes through the real API, so all guards, auto-assignment rules, and stakeholder creation run. The test data matches what a real user would produce.
</Warning>

**Entities created:**
- **Leads:**
  - One by BASIC (auto-assigned as primary stakeholder with FULL access)
  - One by OWNER (for admin-only tests)
  - One in pool (empty stakeholders)
  - One with team as stakeholder (OWNER creates, assigns team)
  - One with AGENT as READ-only stakeholder
- **Contacts:** one by BASIC, one by OWNER
- **Companies:** one by BASIC, one by OWNER
- **Deals:** one by OWNER (from lead + contact)

## How to Run

### Prerequisites

<Check>
Ensure these requirements are met before running tests:
</Check>

1. Database must be running with migrations applied
2. Test users must be seeded: `pnpm test:seed-users`
3. `.env.local` must have required environment variables:
   - `DATABASE_URL`
   - `JWT_TENANT_SECRET`
   - `META_TOKEN_ENCRYPTION_KEY`
   - `META_WEBHOOK_VERIFY_TOKEN`

### Commands

<CodeGroup>
```bash Run all RBAC tests
pnpm test:e2e -- --testPathPattern=rbac
```

```bash Run specific entity
pnpm test:e2e -- --testPathPattern=lead-rbac
pnpm test:e2e -- --testPathPattern=deal-rbac
pnpm test:e2e -- --testPathPattern=contact-rbac
pnpm test:e2e -- --testPathPattern=company-rbac
```

```bash Verbose output
pnpm test:e2e -- --testPathPattern=rbac --verbose
```

```bash With coverage
pnpm test:e2e:cov -- --testPathPattern=rbac
```
</CodeGroup>

## How to Extend

### Adding New Endpoints

<Steps>
<Step title="Update the matrix">
Add the new endpoint to the appropriate matrix file in `test/setup/matrices/`:

```typescript
{
  method: 'POST',
  path: '/v1/leads/:leadId/notes',
  description: 'Add note to lead',
  expectations: {
    OWNER: 201,
    ADMIN: 201,
    TEAM_LEADER: 403,
    AGENT: 403,
    BASIC: 201,
  },
}
```
</Step>

<Step title="Handle path placeholders">
If your endpoint uses path parameters, ensure they're resolved in `rbac-test-data.ts`
</Step>

<Step title="Run tests">
The matrix runner will automatically generate test cases for your new endpoint
</Step>
</Steps>

### Adding New Personas

<Steps>
<Step title="Define the persona">
Add to `test/helpers/rbac-personas.ts`:

```typescript
export const RBAC_PERSONAS = {
  // ... existing personas
  NEW_ROLE: {
    email: 'newrole@propwise-test.com',
    password: 'TestPass123!',
    orgRole: 'Basic' as const,
    teamRole: 'Agent' as const,
  },
};
```
</Step>

<Step title="Update seed script">
Modify `test/setup/seed-test-users.ts` to create the new user
</Step>

<Step title="Update matrices">
Add expectations for the new persona in all relevant matrices
</Step>
</Steps>

## CI/CD Integration

<Tip>
The test suite is designed for CI/CD environments with proper database isolation and cleanup.
</Tip>

**Best practices for CI integration:**

1. **Database isolation:** Use separate test databases per CI job
2. **Parallel execution:** Tests can run in parallel since they use the singleton context
3. **Cleanup:** The `cleanup-test-users.ts` script removes test data after runs
4. **Environment variables:** Ensure all required secrets are available in CI

**Recommended CI workflow:**

```yaml
- name: Setup test database
  run: pnpm db:migrate
  
- name: Seed test users
  run: pnpm test:seed-users
  
- name: Run RBAC tests
  run: pnpm test:e2e -- --testPathPattern=rbac
  
- name: Cleanup test data
  run: pnpm test:cleanup-users
  if: always()
