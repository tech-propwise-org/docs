## Overview

The PropWise CRM backend test suite focuses on **RBAC (Role-Based Access Control) E2E testing**, verifying that every API endpoint enforces the correct permissions for every user role.

<Info>
**Tech Stack:** Jest + supertest + `@nestjs/testing`
</Info>

### Key Design Decisions

<CardGroup cols={2}>
  <Card title="Singleton App Context" icon="server">
    The NestJS app boots once and is shared across all test files for performance
  </Card>
  <Card title="Matrix-Driven Tests" icon="grid">
    Declarative permission matrices define expected HTTP status codes per persona × endpoint
  </Card>
  <Card title="API-Driven Seeding" icon="seedling">
    Test data is created through actual API endpoints so all business logic executes naturally
  </Card>
  <Card title="Automated Test Generation" icon="magic-wand">
    A runner generates `it()` blocks automatically from the permission matrix
  </Card>
</CardGroup>

## Architecture

### Singleton Context Pattern

The app boots once for the entire test run. All test files share the same NestJS app instance, authenticated tokens, and resolved `teamId`.

<Steps>
  <Step title="Jest starts">
    First test file calls `getContext()`
  </Step>
  <Step title="App initialization">
    Boots NestJS app (~3-5s), logs in 5 personas, resolves teamId
  </Step>
  <Step title="Context sharing">
    Subsequent test files get cached instance instantly (0ms)
  </Step>
  <Step title="Cleanup">
    Last file's `afterAll()` calls `closeContext()` to shut down app
  </Step>
</Steps>

```typescript
import { getContext, closeContext, RbacTestContext } from '../../setup/rbac-context';

let ctx: RbacTestContext;
beforeAll(async () => { ctx = await getContext(); }, 60000);
afterAll(async () => { await closeContext(); });
```

<Note>
The context replicates production config from `main.ts`: ValidationPipe (whitelist, transform, implicit conversion), URI versioning (`/v1/...`), and cookie parser.
</Note>

### Matrix-Driven Test Runner

Instead of writing individual `it()` blocks, you define expectations declaratively:

```typescript
{
  method: 'GET',
  path: '/v1/leads/:leadByBasicId',
  description: 'View lead detail',
  expectations: {
    OWNER: 200,
    ADMIN: 200,
    TEAM_LEADER: 403,
    AGENT: 403,
    BASIC: 200,
  },
}
```

<Info>
One matrix entry generates 5 test cases automatically - one for each persona.
</Info>

The runner calls `runRbacMatrix()` which:
1. Iterates every endpoint in the matrix
2. For each endpoint, iterates all 5 personas
3. Resolves path placeholders (`:leadByBasicId` → actual UUID)
4. Makes the HTTP request with the persona's tenant token
5. Asserts the response status matches the expected value
6. Produces clear failure messages with the actual response body

## RBAC Personas

<AccordionGroup>
  <Accordion title="Persona Overview">
    All personas are defined in `test/helpers/rbac-personas.ts` and share a single source of truth.

    | Key | Email | Org Role | Team | Description |
    |-----|-------|----------|------|-------------|
    | `OWNER` | owner@propwise.com | Owner | Any | All permissions including `system.owner` |
    | `ADMIN` | admin@propwise.com | Admin | Any | All non-sensitive permissions (includes `system.admin`) |
    | `TEAM_LEADER` | teamleader@propwise.com | Basic | Team Leader in Test Sales Team | All team permissions (`team_crm.*`, `team_sales.*`, `team.admin`) |
    | `AGENT` | agent@propwise.com | Basic | Agent in Test Sales Team | No team permissions by default |
    | `BASIC` | basic@propwise.com | Basic | None | No permissions — relies entirely on stakeholder access |
  </Accordion>

  <Accordion title="Permission Profiles">
    **OWNER / ADMIN**
    - Bypass resource-level checks via org permissions (`crm.view`, `crm.manage`, `sales.view`, `sales.manage`, etc.)
    - Can access any entity in the org

    **TEAM_LEADER**
    - No org permissions, but has `team_crm.manage` and `team_sales.manage` in the team
    - Can access entities where the team is a stakeholder

    **AGENT**
    - No org permissions, no team permissions
    - Can only access entities where they are personally a stakeholder
    - Used to test access level differentiation (READ vs WRITE vs FULL)

    **BASIC**
    - Same as AGENT but not a team member at all
    - Access is purely stakeholder-based
  </Accordion>

  <Accordion title="Login Flow">
    The context performs a two-step login for each persona:

    <Steps>
      <Step title="Identity Login">
        `POST /v1/auth/login` → identity token
      </Step>
      <Step title="Organization Selection">
        `POST /v1/auth/select-organization` → tenant token (org-scoped JWT)
      </Step>
      <Step title="API Authorization">
        All subsequent API calls use the tenant token in the `Authorization: Bearer` header
      </Step>
    </Steps>
  </Accordion>
</AccordionGroup>

### Access Mode Reference

Your `@CheckAccess` decorator uses two modes:

- **OR (default):** User has the required permission **OR** has stakeholder-level access on the resource → allowed
- **AND:** User must have the permission **AND** stakeholder access → allowed

<Warning>
Resource action hierarchy:
- `ACCESS` — granted by `READ`, `WRITE`, or `FULL` access level
- `MODIFY` — granted by `WRITE` or `FULL`
- `DELETE` — granted by `FULL` only
</Warning>

## Test Data Seeding

<Steps>
  <Step title="Seed Users (One-time Setup)">
    ```bash
    pnpm test:seed-users
    ```

    This runs `test/setup/seed-test-users.ts` which:
    - Creates the 5 test users in the database (idempotent — skips if they exist)
    - Creates "Test Organization" with owner
    - Creates "Test Sales Team" with team leader and agent
    - Assigns org roles (Owner, Admin, Basic) and team roles (Team Leader, Agent)
    - Uses RLS bypass to write directly via MikroORM
  </Step>

  <Step title="Seed Test Data (Automatic)">
    Each test file calls `seedRbacTestData(app, personas, teamId)` in its `beforeAll`. Results are cached across files.

    <Note>
    Because seeding goes through the real API, all guards, auto-assignment rules, and stakeholder creation run. The test data matches what a real user would produce.
    </Note>

    This creates entities through the API:
    - **Leads:** One by BASIC, one by OWNER, one in pool, one with team stakeholder, one with AGENT as READ-only
    - **Contacts:** One by BASIC, one by OWNER
    - **Companies:** One by BASIC, one by OWNER
    - **Deals:** One by OWNER (from lead + contact)
  </Step>
</Steps>

## How to Extend

### Adding New Matrices

<Steps>
  <Step title="Create Matrix File">
    Add a new matrix file under `test/setup/matrices/`:

    ```typescript
    // test/setup/matrices/property-matrix.ts
    export const PROPERTY_BY_BASIC_MATRIX: EndpointExpectation[] = [
      {
        method: 'GET',
        path: '/v1/properties/:propertyByBasicId',
        description: 'View property detail',
        expectations: {
          OWNER: 200,
          ADMIN: 200,
          TEAM_LEADER: 403,
          AGENT: 403,
          BASIC: 200,
        },
      },
    ];
    ```
  </Step>

  <Step title="Export from Index">
    Add to `test/setup/matrices/index.ts`:

    ```typescript
    export * from './property-matrix';
    ```
  </Step>

  <Step title="Create Test File">
    Create `test/e2e/rbac/property-rbac.e2e-spec.ts`:

    ```typescript
    import { runRbacMatrix } from '../../setup/rbac-runner';
    import { PROPERTY_BY_BASIC_MATRIX } from '../../setup/rbac-matrix';

    describe('Property RBAC (E2E)', () => {
      runRbacMatrix('Property owned by BASIC user', PROPERTY_BY_BASIC_MATRIX);
    });
    ```
  </Step>
</Steps>

## Methodology

<Tip>
See `test/TESTING_METHODOLOGY.md` for detailed guidance on analyzing controllers and DTOs before writing tests.
</Tip>

### Test-First Development

1. **Analyze Controller:** Review endpoints, guards, and permission requirements
2. **Define Matrix:** Create expectation matrix based on business rules
3. **Run Tests:** Execute matrix to identify permission gaps
4. **Fix Implementation:** Adjust guards, permissions, or business logic
5. **Verify:** Re-run tests to confirm fixes

### Permission Debugging

When tests fail, check:
- User has correct org/team roles in database
- `@CheckAccess` decorator configuration on endpoint
- Stakeholder relationships in test data
- Permission hierarchy (org > team > stakeholder)

## CI/CD Integration

<Warning>
Ensure test database is isolated from development/production data. The test suite creates and deletes `@propwise-test.com` users.
</Warning>

### GitHub Actions Example

```yaml
- name: Seed test users

- name: Run RBAC tests
  run: pnpm test:e2e -- --testPathPattern=rbac --maxWorkers=1
```

<Note>
Use `--maxWorkers=1` to prevent race conditions in the singleton context pattern.
