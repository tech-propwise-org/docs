The test suite focuses on **RBAC (Role-Based Access Control) E2E testing** — verifying that every API endpoint enforces the correct permissions for every user role.

<Note>
**Stack:** Jest + supertest + `@nestjs/testing`
</Note>

## Key Design Decisions

<CardGroup cols={3}>
  <Card title="Singleton App Context" icon="server">
    The NestJS app boots once and is shared across all test files
  </Card>
  <Card title="Matrix-Driven Tests" icon="table">
    A declarative permission matrix defines expected HTTP status codes per persona × endpoint
  </Card>
  <Card title="API-Driven Seeding" icon="seedling">
    Test data is created through actual API endpoints so all business logic executes naturally
  </Card>
</CardGroup>

## Architecture

### Singleton Context

The app boots once for the entire test run. All test files share the same NestJS app instance, authenticated tokens, and resolved `teamId`.

<Steps>
  <Step title="Jest starts">
    First test file calls `getContext()`
  </Step>
  <Step title="App bootstrap">
    NestJS app boots (~3-5s), logs in 5 personas, resolves teamId
  </Step>
  <Step title="Context sharing">
    Subsequent files get cached instance instantly (0ms)
  </Step>
  <Step title="Cleanup">
    Last file's afterAll() shuts down when reference count hits 0
  </Step>
</Steps>

```typescript
import { getContext, closeContext, RbacTestContext } from '../../setup/rbac-context';

let ctx: RbacTestContext;
beforeAll(async () => {
  ctx = await getContext();
}, 60000);
afterAll(async () => {
  await closeContext();
});
```

<Info>
The context replicates production config from `main.ts` including ValidationPipe, URI versioning (`/v1/...`), and cookie parser.
</Info>

### Matrix Runner

Instead of writing individual `it()` blocks, you define expectations declaratively:

```typescript
{
  method: 'GET',
  path: '/v1/leads/:leadByBasicId',
  description: 'View lead detail',
  expectations: {
    OWNER: 200,
    ADMIN: 200,
    TEAM_LEADER: 403,
    AGENT: 403,
    BASIC: 200,
  },
}
```

The runner calls `runRbacMatrix()` which:

1. **Iterates** every endpoint in the matrix
2. **Tests** all 5 personas per endpoint  
3. **Resolves** path placeholders (`:leadByBasicId` → actual UUID)
4. **Makes** HTTP request with persona's tenant token
5. **Asserts** response status matches expected value
6. **Reports** clear failure messages with actual response body

<Tip>
One matrix entry = 5 generated test cases automatically
</Tip>

## RBAC Personas

All personas are defined in `test/helpers/rbac-personas.ts` as a single source of truth.

| Key           | Email                   | Org Role | Team                           | Description                                                       |
| ------------- | ----------------------- | -------- | ------------------------------ | ----------------------------------------------------------------- |
| `OWNER`       | owner@propwise.com      | Owner    | Any                            | All permissions including `system.owner`                          |
| `ADMIN`       | admin@propwise.com      | Admin    | Any                            | All non-sensitive permissions (includes `system.admin`)           |
| `TEAM_LEADER` | teamleader@propwise.com | Basic    | Team Leader in Test Sales Team | All team permissions (`team_crm.*`, `team_sales.*`, `team.admin`) |
| `AGENT`       | agent@propwise.com      | Basic    | Agent in Test Sales Team       | No team permissions by default                                    |
| `BASIC`       | basic@propwise.com      | Basic    | None                           | No permissions — relies entirely on stakeholder access            |

### Permission Profiles

<Tabs>
  <Tab title="OWNER / ADMIN">
    Bypass resource-level checks via org permissions (`crm.view`, `crm.manage`, `sales.view`, `sales.manage`, etc.). Can access any entity in the org.
  </Tab>
  <Tab title="TEAM_LEADER">
    No org permissions, but has `team_crm.manage` and `team_sales.manage` in the team. Can access entities where the team is a stakeholder.
  </Tab>
  <Tab title="AGENT">
    No org permissions, no team permissions. Can only access entities where they are personally a stakeholder. Used to test access level differentiation (READ vs WRITE vs FULL).
  </Tab>
  <Tab title="BASIC">
    Same as AGENT but not a team member at all. Access is purely stakeholder-based.
  </Tab>
</Tabs>

### Login Flow

The context performs a two-step login for each persona:

<Steps>
  <Step title="Identity login">
    `POST /v1/auth/login` → identity token
  </Step>
  <Step title="Organization selection">
    `POST /v1/auth/select-organization` → tenant token (org-scoped JWT)
  </Step>
  <Step title="API calls">
    All subsequent API calls use the tenant token in the `Authorization: Bearer` header
  </Step>
</Steps>

### Access Mode Reference

Your `@CheckAccess` decorator uses two modes:

<Tabs>
  <Tab title="OR Mode (default)">
    User has the required permission **OR** has stakeholder-level access on the resource → allowed
  </Tab>
  <Tab title="AND Mode">
    User must have the permission **AND** stakeholder access → allowed
  </Tab>
</Tabs>

**Resource action hierarchy:**
- `ACCESS` — granted by `READ`, `WRITE`, or `FULL` access level
- `MODIFY` — granted by `WRITE` or `FULL`
- `DELETE` — granted by `FULL` only

## Test Data Seeding

### Step 1: Seed Users (One-time Setup)

<Warning>
Run this once before running tests for the first time or after database resets.
</Warning>

```bash
pnpm test:seed-users
```

This runs `test/setup/seed-test-users.ts` which:

- Creates the RBAC test personas in the database (idempotent)
- Creates "Test Organization" with owner
- Ensures **ACTIVE BUSINESS subscription** and **completed organization_setup_progress**
- Creates "Test Sales Team" with team leader and agent
- Assigns org roles (Owner, Admin, Basic) and team roles (Team Leader, Agent)
- Uses RLS bypass to write directly via MikroORM

<Note>
Requires subscription plans to exist. Run `pnpm test:init-data` once on a fresh DB before `pnpm test:seed-users`.
</Note>

### Step 2: Seed Test Data (Automatic)

Each test file calls `seedRbacTestData(app, personas, teamId)` in its `beforeAll`. Results are cached across files.

<Tabs>
  <Tab title="Leads">
    - One by BASIC (auto-assigned as primary stakeholder with FULL access)
    - One by OWNER (for admin-only tests)  
    - One in pool (empty stakeholders)
    - One with team as stakeholder
    - One with AGENT as READ-only stakeholder
  </Tab>
  <Tab title="Deals">
    - One by OWNER (for sales permission tests)
  </Tab>
  <Tab title="Contacts & Companies">
    - One by BASIC (stakeholder access tests)
    - One by OWNER (admin-only tests)
  </Tab>
</Tabs>

## How to Run

### Core RBAC Tests

```bash
# All RBAC tests
pnpm test:e2e

# Specific entity
pnpm test:e2e -- lead-rbac
pnpm test:e2e -- deal-rbac
pnpm test:e2e -- contact-rbac
pnpm test:e2e -- company-rbac

# Auth lifecycle tests
pnpm test:e2e -- auth-lifecycle
pnpm test:e2e -- org-lifecycle
```

### Queue Tests (Live pg-boss)

<Warning>
Queue tests use live pg-boss and require a separate Jest config.
</Warning>

```bash
# Distribution and escalation queue tests
pnpm test:e2e:queue
```

### Test Utilities

```bash
# Seed RBAC test users (one-time setup)
pnpm test:seed-users

# Clean up test users
pnpm test:cleanup-users

# Initialize subscription plans (fresh DB only)
pnpm test:init-data
```

## How to Extend

### Adding New Endpoints

<Steps>
  <Step title="Add to matrix">
    Add endpoint definition to the appropriate matrix file in `test/setup/matrices/`
  </Step>
  <Step title="Define expectations">
    Specify expected HTTP status codes for each persona
  </Step>
  <Step title="Handle placeholders">
    If your endpoint has path parameters, ensure they're resolved in `rbac-test-data.ts`
  </Step>
  <Step title="Run tests">
    The matrix runner will automatically generate test cases
  </Step>
</Steps>

Example matrix entry:

```typescript
{
  method: 'POST',
  path: '/v1/leads/:leadId/activities',
  description: 'Create lead activity',
  expectations: {
    OWNER: 201,
    ADMIN: 201,
    TEAM_LEADER: 403,
    AGENT: 403,
    BASIC: 201,
  },
  body: { type: 'NOTE', content: 'Test activity' }
}
```

### Adding New Personas

<Steps>
  <Step title="Update persona definitions">
    Add new persona to `test/helpers/rbac-personas.ts`
  </Step>
  <Step title="Update seeding">
    Modify `test/setup/seed-test-users.ts` to create the new persona
  </Step>
  <Step title="Update matrices">
    Add expectations for the new persona across all relevant matrices
  </Step>
  <Step title="Update types">
    Add the new persona key to TypeScript types in `rbac-matrix-types.ts`
  </Step>
</Steps>

### Adding New Entity Types

<Steps>
  <Step title="Create matrix file">
    Add new matrix file under `test/setup/matrices/`
  </Step>
  <Step title="Export from barrel">
    Add export to `test/setup/matrices/index.ts`
  </Step>
  <Step title="Create test file">
    Add new RBAC test file under `test/e2e/rbac/`
  </Step>
  <Step title="Seed test data">
    Update `rbac-test-data.ts` to create fixture entities for your new type
  </Step>
</Steps>

## Testing Methodology

### Analysis Process

<Steps>
  <Step title="Controller analysis">
    Identify all endpoints, HTTP methods, and required permissions
  </Step>
  <Step title="DTO validation">
    Map request/response DTOs to understand data flow
  </Step>
  <Step title="Guard analysis">
    Understand which guards apply (`@CheckAccess`, `@RequirePermissions`, etc.)
  </Step>
  <Step title="Matrix design">
    Design permission matrix based on business rules and guard requirements
  </Step>
</Steps>

### Best Practices

<CardGroup cols={2}>
  <Card title="Test Business Logic" icon="brain">
    Use API-driven seeding to test actual business logic execution
  </Card>
  <Card title="Declarative Matrices" icon="table-list">
    Define expectations declaratively rather than writing imperative test code
  </Card>
  <Card title="Stakeholder Testing" icon="users">
    Test both permission-based and stakeholder-based access patterns
  </Card>
  <Card title="Edge Cases" icon="triangle-exclamation">

  </Card>
</CardGroup>

## CI/CD Integration

The test suite is designed for CI/CD environments:

- **Fast startup** via singleton context pattern
- **Deterministic seeding** with idempotent user creation
- **Isolated test data** using unique entity names and cleanup
- **Clear failure reporting** with actual response bodies in assertions

<Tip>
Use `jest-e2e.json` for core RBAC tests and `jest-e2e-queue.json` for queue-specific tests to optimize CI pipeline execution.
</Tip>
