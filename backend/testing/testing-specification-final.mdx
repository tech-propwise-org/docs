## Overview

The test suite focuses on **RBAC (Role-Based Access Control) E2E testing** ‚Äî verifying that every API endpoint enforces the correct permissions for every user role.

<Note>
**Stack:** Jest + supertest + `@nestjs/testing`
</Note>

### Key Design Decisions

<CardGroup cols={2}>
  <Card title="Singleton App Context" icon="server">
    The NestJS app boots once and is shared across all test files (equivalent to NestJS `TestSuite` pattern)
  </Card>
  <Card title="Matrix-Driven Tests" icon="grid">
    A declarative permission matrix defines expected HTTP status codes per persona √ó endpoint
  </Card>
  <Card title="API-Driven Seeding" icon="seedling">
    Test data is created through actual API endpoints so all business logic executes naturally
  </Card>
  <Card title="Automated Test Generation" icon="robot">
    A runner generates `it()` blocks automatically from the permission matrix
  </Card>
</CardGroup>

## Architecture

### Singleton Context

The app boots once for the entire test run. All test files share the same NestJS app instance, authenticated tokens, and resolved `teamId`.

<Steps>
  <Step title="Jest starts">
    First test file calls `getContext()`
  </Step>
  <Step title="App boots">
    NestJS app initializes (~3-5s), logs in 5 personas, resolves teamId
  </Step>
  <Step title="Shared context">
    Subsequent test files get cached instance instantly (0ms)
  </Step>
  <Step title="Cleanup">
    Last file's `afterAll()` calls `closeContext()` when reference count hits 0
  </Step>
</Steps>

```typescript
import { getContext, closeContext, RbacTestContext } from '../../setup/rbac-context';

let ctx: RbacTestContext;
beforeAll(async () => { ctx = await getContext(); }, 60000);
afterAll(async () => { await closeContext(); });
```

<Note>
The context replicates production config from `main.ts`: ValidationPipe, URI versioning (`/v1/...`), and cookie parser.
</Note>

### Matrix Runner

Instead of writing individual `it()` blocks, you define expectations declaratively:

```typescript
{
  method: 'GET',
  path: '/v1/leads/:leadByBasicId',
  description: 'View lead detail',
  expectations: {
    OWNER: 200,
    ADMIN: 200,
    TEAM_LEADER: 403,
    AGENT: 403,
    BASIC: 200,
  },
}
```

<Info>
One matrix entry generates 5 test cases automatically. The runner resolves path placeholders, makes HTTP requests, and produces clear failure messages.
</Info>

## RBAC Personas

<AccordionGroup>
  <Accordion title="Persona Overview">
    All personas are defined in `test/helpers/rbac-personas.ts` and share a single source of truth.

    | Key | Email | Org Role | Team | Description |
    |-----|-------|----------|------|-------------|
    | `OWNER` | owner@propwise.com | Owner | Any | All permissions including `system.owner` |
    | `ADMIN` | admin@propwise.com | Admin | Any | All non-sensitive permissions (includes `system.admin`) |
    | `TEAM_LEADER` | teamleader@propwise.com | Basic | Team Leader in Test Sales Team | All team permissions (`team_crm.*`, `team_sales.*`, `team.admin`) |
    | `AGENT` | agent@propwise.com | Basic | Agent in Test Sales Team | No team permissions by default |
    | `BASIC` | basic@propwise.com | Basic | None | No permissions ‚Äî relies entirely on stakeholder access |
  </Accordion>

  <Accordion title="Permission Profiles">
    **OWNER / ADMIN** ‚Äî bypass resource-level checks via org permissions (`crm.view`, `crm.manage`, `sales.view`, `sales.manage`, etc.). Can access any entity in the org.

    **TEAM_LEADER** ‚Äî no org permissions, but has `team_crm.manage` and `team_sales.manage` in the team. Can access entities where the team is a stakeholder.

    **AGENT** ‚Äî no org permissions, no team permissions. Can only access entities where they are personally a stakeholder. Used to test access level differentiation (READ vs WRITE vs FULL).

    **BASIC** ‚Äî same as AGENT but not a team member at all. Access is purely stakeholder-based.
  </Accordion>

  <Accordion title="Login Flow">
    The context performs a two-step login for each persona:

    1. `POST /v1/auth/login` ‚Üí identity token
    2. `POST /v1/auth/select-organization` ‚Üí tenant token (org-scoped JWT)

    All subsequent API calls use the tenant token in the `Authorization: Bearer` header.
  </Accordion>
</AccordionGroup>

### Access Mode Reference

<Note>
Your `@CheckAccess` decorator uses two modes:

- **OR (default):** User has the required permission **OR** has stakeholder-level access on the resource ‚Üí allowed.
- **AND:** User must have the permission **AND** stakeholder access ‚Üí allowed.
</Note>

**Resource action hierarchy:**
- `ACCESS` ‚Äî granted by `READ`, `WRITE`, or `FULL` access level
- `MODIFY` ‚Äî granted by `WRITE` or `FULL`
- `DELETE` ‚Äî granted by `FULL` only

## Test Data Seeding

<Steps>
  <Step title="Seed Users (one-time)">
    Run before executing tests:
    ```bash
    pnpm test:seed-users
    ```
    
    This creates:
    - 5 test users in the database (idempotent)
    - "Test Organization" with owner
    - "Test Sales Team" with team leader and agent
    - Org roles (Owner, Admin, Basic) and team roles
  </Step>

  <Step title="Seed Test Data (automatic)">
    Each test file calls `seedRbacTestData(app, personas, teamId)` in `beforeAll`. Results are cached across files.
    
    Creates entities through the API:
    - **Leads:** by BASIC, by OWNER, in pool, with team stakeholder, with AGENT read-only
    - **Contacts:** one by BASIC, one by OWNER
    - **Companies:** one by BASIC, one by OWNER
    - **Deals:** one by OWNER (from lead + contact)
  </Step>
</Steps>

<Tip>
Because seeding goes through the real API, all guards, auto-assignment rules, and stakeholder creation run. The test data matches what a real user would produce.
</Tip>

## How to Extend

### Adding New Entity Tests

<Steps>
  <Step title="Create matrix file">
    Add `matrices/your-entity-matrix.ts` with permission expectations
  </Step>
  <Step title="Update barrel exports">
    Add to `matrices/index.ts` and `rbac-matrix.ts`
  </Step>
  <Step title="Extend test data seeding">
    Add entity creation to `rbac-test-data.ts`
  </Step>
  <Step title="Create test file">
    Add `e2e/rbac/your-entity-rbac.e2e-spec.ts` using the matrix runner
  </Step>
</Steps>

### Adding New Endpoints

<Steps>
  <Step title="Analyze the controller">
    Review permissions, stakeholder requirements, and access modes
  </Step>
  <Step title="Add to appropriate matrix">
    Define expected status codes for each persona
  </Step>
  <Step title="Test automatically generates">
    The matrix runner creates test cases for all personas
  </Step>
</Steps>

<Warning>
Always verify that new endpoints have proper `@CheckAccess` decorators and stakeholder validation before adding to matrices.
</Warning>

## Methodology

### Test-First Development

1. **Analyze controllers/DTOs** ‚Äî understand permissions and business rules
2. **Define matrix expectations** ‚Äî declare what each persona should be able to do
3. **Run tests** ‚Äî verify actual behavior matches expectations
4. **Fix implementation** ‚Äî adjust guards, permissions, or stakeholder logic as needed

### Permission Testing Strategy

<CardGroup cols={2}>
  <Card title="Positive Cases" icon="check">
    Verify authorized users can perform expected actions
  </Card>
  <Card title="Negative Cases" icon="x">

  <Card title="Access Level Enforcement" icon="shield">
    Test READ vs WRITE vs FULL access differentiation

  <Card title="Stakeholder Validation" icon="users">
    Verify stakeholder-based access works correctly

## CI/CD Integration

The test suite is designed for automated CI/CD pipelines:

- **Fast startup** ‚Äî singleton context reduces boot time
- **Deterministic** ‚Äî API-driven seeding ensures consistent test data
- **Comprehensive** ‚Äî matrix-driven approach covers all permission combinations
- **Clear failures** ‚Äî descriptive error messages with actual response bodies

<Tip>
Consider running RBAC tests on a schedule (nightly) in addition to PR validation due to their comprehensive nature.
</Tip>
