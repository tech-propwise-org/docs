The PropWise CRM backend test suite provides comprehensive **RBAC (Role-Based Access Control) E2E testing** that verifies API endpoint permissions across all user roles using a matrix-driven approach.

<Info>
**Stack:** Jest + supertest + `@nestjs/testing` with singleton app context and declarative permission matrices
</Info>

## Overview

The test suite focuses on end-to-end verification that every API endpoint enforces the correct permissions for every user role. Key design decisions include:

- **Singleton app context** ‚Äî NestJS app boots once and is shared across all test files
- **Matrix-driven tests** ‚Äî declarative permission matrix defines expected HTTP status codes per persona √ó endpoint
- **API-driven seeding** ‚Äî test data created through actual API endpoints to execute all business logic

## Architecture

### Singleton Context

The app boots once for the entire test run, with all test files sharing the same NestJS app instance, authenticated tokens, and resolved `teamId`.

<Note>
This approach is equivalent to NestJS `TestSuite` pattern but implemented manually since `TestSuite` requires NestJS 11+.
</Note>

```typescript
import { getContext, closeContext, RbacTestContext } from '../../setup/rbac-context';

let ctx: RbacTestContext;
beforeAll(async () => { ctx = await getContext(); }, 60000);
afterAll(async () => { await closeContext(); });
```

**Context flow:**
<Steps>
<Step title="First test file calls getContext()">
Boots NestJS app (~3-5s), logs in 5 personas, resolves teamId
</Step>
<Step title="Subsequent test files get cached instance">
Returns immediately (0ms) with shared context
</Step>
<Step title="Last file's afterAll() calls closeContext()">
Reference count hits 0 ‚Üí app shuts down
</Step>
</Steps>

The context replicates production config from `main.ts`:
- `ValidationPipe` (whitelist, transform, implicit conversion)
- URI versioning (`/v1/...`)
- Cookie parser

### Matrix Runner

Instead of writing individual `it()` blocks, define expectations declaratively:

<CodeGroup>
```typescript Matrix Definition
{
  method: 'GET',
  path: '/v1/leads/:leadByBasicId',
  description: 'View lead detail',
  expectations: {
    OWNER: 200,
    ADMIN: 200,
    TEAM_LEADER: 403,
    AGENT: 403,
    BASIC: 200,
  },
}
```

```typescript Runner Process
// One matrix entry = 5 generated test cases
runRbacMatrix()
  ‚Üí Iterates every endpoint in matrix
  ‚Üí For each endpoint, iterates all 5 personas
  ‚Üí Resolves path placeholders (:leadByBasicId ‚Üí actual UUID)
  ‚Üí Makes HTTP request with persona's tenant token
  ‚Üí Asserts response status matches expected value
  ‚Üí Produces clear failure messages with actual response body
```
</CodeGroup>

## RBAC Personas

All personas are defined in `test/helpers/rbac-personas.ts` and share a single organization.

| Persona | Email | Org Role | Team | Description |
|---------|-------|----------|------|-------------|
| `OWNER` | owner@propwise.com | Owner | Any | All permissions including `system.owner` |
| `ADMIN` | admin@propwise.com | Admin | Any | All non-sensitive permissions (includes `system.admin`) |
| `TEAM_LEADER` | teamleader@propwise.com | Basic | Team Leader in Test Sales Team | All team permissions (`team_crm.*`, `team_sales.*`, `team.admin`) |
| `AGENT` | agent@propwise.com | Basic | Agent in Test Sales Team | No team permissions by default |
| `BASIC` | basic@propwise.com | Basic | None | No permissions ‚Äî relies entirely on stakeholder access |

### Permission Profiles

<Tabs>
<Tab title="OWNER / ADMIN">
- Bypass resource-level checks via org permissions
- Can access any entity in the org
- OWNER has additional `system.owner` permissions
- ADMIN has `system.admin` permissions
</Tab>
<Tab title="TEAM_LEADER">
- No org permissions
- Has `team_crm.manage` and `team_sales.manage` in team
- Can access entities where the team is a stakeholder
</Tab>
<Tab title="AGENT / BASIC">
- No org permissions, no team permissions (BASIC not even in team)
- Can only access entities where they are personally a stakeholder
- Used to test access level differentiation (READ vs WRITE vs FULL)
</Tab>
</Tabs>

### Login Flow

<Steps>
<Step title="Identity login">
`POST /v1/auth/login` ‚Üí identity token
</Step>
<Step title="Organization selection">
`POST /v1/auth/select-organization` ‚Üí tenant token (org-scoped JWT)
</Step>
<Step title="API requests">
All subsequent API calls use tenant token in `Authorization: Bearer` header
</Step>
</Steps>

Matrices are organized by entity in `test/setup/matrices/` with barrel exports from `test/setup/rbac-matrix.ts`.

### Lead Matrices

<CardGroup cols={2}>
<Card title="LEAD_BY_BASIC_MATRIX" icon="user">

</Card>
<Card title="LEAD_ON_TEAM_MATRIX" icon="users">

</Card>

</Warning>

### Access Mode Reference

<Tip>
Your `@CheckAccess` decorator uses two modes:

- **OR (default):** User has the required permission **OR** has stakeholder-level access ‚Üí allowed
- **AND:** User must have the permission **AND** stakeholder access ‚Üí allowed
</Tip>

Resource action hierarchy:
- `ACCESS` ‚Äî granted by `READ`, `WRITE`, or `FULL` access level
- `MODIFY` ‚Äî granted by `WRITE` or `FULL`
- `DELETE` ‚Äî granted by `FULL` only

## Test Data Seeding

### Step 1: Seed Users (One-time Setup)

<Steps>
<Step title="Run the seeder">
```bash
pnpm test:seed-users
```
</Step>
<Step title="Creates test infrastructure">
- 5 test users in database (idempotent ‚Äî skips if exist)
- "Test Organization" with owner
- "Test Sales Team" with team leader and agent
- Assigns org roles (Owner, Admin, Basic) and team roles
- Uses RLS bypass to write directly via MikroORM
</Step>
</Steps>

### Step 2: Seed Test Data (Automatic)

Each test file calls `seedRbacTestData(app, personas, teamId)` in `beforeAll`. Results are cached across files.

<AccordionGroup>
<Accordion title="Test data created through API">
**Leads:**
- One by BASIC (auto-assigned as primary stakeholder with FULL access)
- One by OWNER (for admin-only tests)
- One in pool (empty stakeholders)
- One with team as stakeholder (OWNER creates, assigns team ‚Äî TEAM_LEADER gets access)
- One with AGENT as READ-only stakeholder (OWNER creates, adds AGENT with `accessLevel: read`)

**Contacts:** one by BASIC, one by OWNER

**Companies:** one by BASIC, one by OWNER

**Deals:** one by OWNER (from lead + contact)
</Accordion>
</AccordionGroup>

<Check>
Because seeding goes through the real API, all guards, auto-assignment rules, and stakeholder creation run. The test data matches what a real user would produce.
</Check>

## How to Run

### Prerequisites

<Steps>
<Step title="Database setup">
Database must be running with migrations applied
</Step>
<Step title="Seed test users">
```bash
pnpm test:seed-users
```
</Step>
<Step title="Environment configuration">
`.env.local` must have:
- `DATABASE_URL`
- `JWT_TENANT_SECRET`
- `META_TOKEN_ENCRYPTION_KEY`
- `META_WEBHOOK_VERIFY_TOKEN`
</Step>
</Steps>

### Commands

<CodeGroup>
```bash All RBAC Tests
# Run all RBAC tests
pnpm test:e2e -- --testPathPattern=rbac
```

```bash Specific Entity Tests
# Run specific entity
pnpm test:e2e -- --testPathPattern=lead-rbac
pnpm test:e2e -- --testPathPattern=deal-rbac
pnpm test:e2e -- --testPathPattern=contact-rbac
pnpm test:e2e -- --testPathPattern=company-rbac
```

```bash Verbose and Coverage
# Verbose output (see each test name)
pnpm test:e2e -- --testPathPattern=rbac --verbose

# With coverage
pnpm test:e2e:cov -- --testPathPattern=rbac
```
</CodeGroup>

## How to Extend

### Adding New Endpoints

<Steps>
<Step title="Update the matrix">
Add new endpoint to appropriate matrix file in `test/setup/matrices/`
</Step>
<Step title="Define expectations">
```typescript
{
  method: 'POST',
  path: '/v1/leads/:leadId/actions/convert',
  description: 'Convert lead to deal',
  expectations: {
    OWNER: 201,
    ADMIN: 201,
    TEAM_LEADER: 403,
    AGENT: 403,
    BASIC: 201, // if they own the lead
  },
}
```
</Step>
<Step title="Run tests">
The runner automatically generates test cases for all personas
</Step>
</Steps>

### Adding New Personas

<Steps>
<Step title="Update rbac-personas.ts">
Add new persona definition with email and permissions
</Step>
<Step title="Update seed-test-users.ts">
Add user creation and role assignment
</Step>
<Step title="Update all matrices">
Add expectations for new persona across all endpoint definitions
</Step>
<Step title="Re-seed users">
```bash

```
</Step>
</Steps>

<Steps>
<Step title="Create matrix file">

<Step title="Update barrel exports">
Export from `test/setup/matrices/index.ts` and `test/setup/rbac-matrix.ts`

<Step title="Update test data seeding">
Extend `rbac-test-data.ts` if new test entities are needed
</Step>
</Steps>

## Methodology

The test suite follows these principles:

<CardGroup cols={2}>
<Card title="Permission-first testing" icon="shield">
Every endpoint is tested against every persona to ensure proper access control
</Card>
<Card title="Real API usage" icon="api">
Test data is created through actual API endpoints, not direct database manipulation
</Card>
<Card title="Declarative matrices" icon="table">
Permission expectations are defined as data structures, not imperative test code
</Card>
<Card title="Stakeholder verification" icon="users">
Tests verify both org-level permissions and resource-specific stakeholder access
</Card>
</CardGroup>

## CI/CD Integration

The test suite is designed for continuous integration:

- **Fast feedback:** Singleton context minimizes app boot time
- **Reliable cleanup:** Test data is isolated to `@propwise-test.com` users
- **Deterministic:** Matrix-driven approach ensures consistent test coverage
- **Parallelizable:** Each test file can run independently after context initialization

<Warning>
Always run `pnpm test:cleanup-users` before seeding in CI environments to ensure clean state.
</Warning>
