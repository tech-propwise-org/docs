The PropWise CRM backend test suite focuses on **RBAC (Role-Based Access Control) E2E testing**, verifying that every API endpoint enforces the correct permissions for every user role.

<Info>
**Stack:** Jest + supertest + `@nestjs/testing`
</Info>

## Key Design Decisions

<CardGroup cols={2}>
  <Card title="Singleton App Context" icon="server">
    NestJS app boots once and is shared across all test files for optimal performance
  </Card>
  <Card title="Matrix-Driven Tests" icon="grid">
    Declarative permission matrix defines expected HTTP status codes per persona √ó endpoint
  </Card>
  <Card title="API-Driven Seeding" icon="seedling">
    Test data created through actual API endpoints so all business logic executes naturally
  </Card>
  <Card title="Auto-Generated Tests" icon="magic">
    Runner generates `it()` blocks automatically from permission matrices
  </Card>
</CardGroup>

## Architecture

### Singleton Context

The app boots once for the entire test run. All test files share the same NestJS app instance, authenticated tokens, and resolved `teamId`.

<Steps>
  <Step title="Jest starts">
    First test file calls `getContext()` ‚Üí boots NestJS app (~3-5s), logs in 5 personas, resolves teamId
  </Step>
  <Step title="Subsequent calls">
    Other test files call `getContext()` ‚Üí returns cached instance instantly (0ms)
  </Step>
  <Step title="Cleanup">
    Last file's `afterAll()` calls `closeContext()` ‚Üí reference count hits 0 ‚Üí app shuts down
  </Step>
</Steps>

```typescript
import { getContext, closeContext, RbacTestContext } from '../../setup/rbac-context';

let ctx: RbacTestContext;
beforeAll(async () => { ctx = await getContext(); }, 60000);
afterAll(async () => { await closeContext(); });
```

### Matrix Runner

Instead of writing individual `it()` blocks, you define expectations declaratively:

```typescript
{
  method: 'GET',
  path: '/v1/leads/:leadByBasicId',
  description: 'View lead detail',
  expectations: {
    OWNER: 200,
    ADMIN: 200,
    TEAM_LEADER: 403,
    AGENT: 403,
    BASIC: 200,
  },
}
```

<Note>
One matrix entry = 5 generated test cases (one per persona)
</Note>

The runner:
1. Iterates every endpoint in the matrix
2. For each endpoint, iterates all 5 personas
3. Resolves path placeholders (`:leadByBasicId` ‚Üí actual UUID)
4. Makes the HTTP request with the persona's tenant token
5. Asserts the response status matches the expected value

## RBAC Personas

All personas are defined in `test/helpers/rbac-personas.ts` and share a single source of truth.

| Persona | Email | Org Role | Team | Description |
|---------|-------|----------|------|-------------|
| `OWNER` | owner@propwise-test.com | Owner | Any | All permissions including `system.owner` |
| `ADMIN` | admin@propwise-test.com | Admin | Any | All non-sensitive permissions (includes `system.admin`) |
| `TEAM_LEADER` | teamleader@propwise-test.com | Basic | Team Leader in Test Sales Team | All team permissions (`team_crm.*`, `team_sales.*`, `team.admin`) |
| `AGENT` | agent@propwise-test.com | Basic | Agent in Test Sales Team | No team permissions by default |
| `BASIC` | basic@propwise-test.com | Basic | None | No permissions ‚Äî relies entirely on stakeholder access |

### Permission Profiles

<AccordionGroup>
  <Accordion title="OWNER / ADMIN">
    Bypass resource-level checks via org permissions (`crm.view`, `crm.manage`, `sales.view`, `sales.manage`, etc.). Can access any entity in the org.
  </Accordion>
  <Accordion title="TEAM_LEADER">
    No org permissions, but has `team_crm.manage` and `team_sales.manage` in the team. Can access entities where the team is a stakeholder.
  </Accordion>
  <Accordion title="AGENT">
    No org permissions, no team permissions. Can only access entities where they are personally a stakeholder. Used to test access level differentiation (READ vs WRITE vs FULL).
  </Accordion>
  <Accordion title="BASIC">
    Same as AGENT but not a team member at all. Access is purely stakeholder-based.
  </Accordion>
</AccordionGroup>

### Login Flow

<Steps>
  <Step title="Identity Token">
    `POST /v1/auth/login` ‚Üí identity token
  </Step>
  <Step title="Tenant Token">
    `POST /v1/auth/select-organization` ‚Üí tenant token (org-scoped JWT)
  </Step>
  <Step title="API Calls">
    All subsequent API calls use the tenant token in the `Authorization: Bearer` header
  </Step>
</Steps>

## Permission Matrix

Matrices are split into per-entity files under `test/setup/matrices/`. The barrel at `test/setup/rbac-matrix.ts` re-exports everything.

### Lead Matrices

| Matrix | Entity | Who Can Access | Tests |
|--------|--------|----------------|-------|
| `LEAD_BY_BASIC_MATRIX` | Lead created by BASIC | OWNER, ADMIN (via `crm.*`), BASIC (via FULL stakeholder) | View, update, stakeholders, activities, property interests |
| `LEAD_ON_TEAM_MATRIX` | Lead with team as stakeholder | OWNER, ADMIN, TEAM_LEADER (via `team_crm.*`) | View, update, stakeholders, activities |
| `LEAD_AGENT_READ_MATRIX` | Lead where AGENT has READ access | OWNER, ADMIN, AGENT (view only ‚Äî modify denied) | View allowed, update denied, access level enforcement |
| `LEAD_IN_POOL_MATRIX` | Lead with no stakeholders | OWNER, ADMIN only (via `crm.*`) | View, update |
| `LEAD_ADMIN_ONLY_MATRIX` | Lead for admin-only ops | OWNER, ADMIN (via `system.admin`) | Delete |

### Deal Matrices

| Matrix | Entity | Who Can Access |
|--------|--------|----------------|
| `DEAL_BY_OWNER_MATRIX` | Deal created by OWNER | OWNER, ADMIN only (via `sales.*`) |

### Contact Matrices

| Matrix | Entity | Who Can Access |
|--------|--------|----------------|
| `CONTACT_BY_BASIC_MATRIX` | Contact created by BASIC | OWNER, ADMIN, BASIC |
| `CONTACT_BY_OWNER_MATRIX` | Contact created by OWNER | OWNER, ADMIN only |

### Company Matrices

| Matrix | Entity | Who Can Access |
|--------|--------|----------------|
| `COMPANY_BY_BASIC_MATRIX` | Company created by BASIC | OWNER, ADMIN, BASIC |
| `COMPANY_BY_OWNER_MATRIX` | Company created by OWNER | OWNER, ADMIN only |

### Access Mode Reference

<Tabs>
  <Tab title="OR Mode (Default)">
    User has the required permission **OR** has stakeholder-level access on the resource ‚Üí allowed.
  </Tab>
  <Tab title="AND Mode">
    User must have the permission **AND** stakeholder access ‚Üí allowed.
  </Tab>
</Tabs>

**Resource action hierarchy:**
- `ACCESS` ‚Äî granted by `READ`, `WRITE`, or `FULL` access level
- `MODIFY` ‚Äî granted by `WRITE` or `FULL`
- `DELETE` ‚Äî granted by `FULL` only

## Test Data Seeding

### Step 1: Seed Users (One-Time Setup)

<Warning>
This must be run before executing any tests
</Warning>

```bash
pnpm test:seed-users
```

This runs `test/setup/seed-test-users.ts` which:

<Steps>
  <Step title="Create Users">
    Creates the 5 test users in the database (idempotent ‚Äî skips if they exist)
  </Step>
  <Step title="Create Organization">
    Creates "Test Organization" with owner
  </Step>
  <Step title="Create Team">
    Creates "Test Sales Team" with team leader and agent
  </Step>
  <Step title="Assign Roles">
    Assigns org roles (Owner, Admin, Basic) and team roles (Team Leader, Agent)
  </Step>
</Steps>

### Step 2: Seed Test Data (Automatic)

Each test file calls `seedRbacTestData(app, personas, teamId)` in its `beforeAll`. Results are cached across files.

<Note>
Because seeding goes through the real API, all guards, auto-assignment rules, and stakeholder creation run. The test data matches what a real user would produce.
</Note>

**Created entities:**
- **Leads:** One by BASIC, one by OWNER, one in pool, one with team as stakeholder, one with AGENT as READ-only stakeholder
- **Contacts:** One by BASIC, one by OWNER  
- **Companies:** One by BASIC, one by OWNER
- **Deals:** One by OWNER (from lead + contact)

## How to Extend

### Adding New Endpoints

<Steps>
  <Step title="Update Matrix">
    Add endpoint expectations to the appropriate matrix file in `test/setup/matrices/`
  </Step>
  <Step title="Path Placeholders">
    Use descriptive placeholder names (`:leadByBasicId`, `:dealByOwnerId`) that match your test data
  </Step>
  <Step title="Expected Status Codes">
    Define expected HTTP status for each persona based on their permissions
  </Step>
</Steps>

### Adding New Entities

<Steps>
  <Step title="Create Matrix File">
    Add new matrix file in `test/setup/matrices/`
  </Step>
  <Step title="Export Matrix">
    Add barrel export to `test/setup/matrices/index.ts`
  </Step>
  <Step title="Create Test File">
    Add new test file in `test/e2e/rbac/`
  </Step>
  <Step title="Seed Test Data">
    Update `rbac-test-data.ts` to create entities for the new resource type
  </Step>
</Steps>

### Adding New Personas

<Steps>
  <Step title="Update Personas Definition">
    Add persona to `test/helpers/rbac-personas.ts`
  </Step>
  <Step title="Update Seeder">
    Modify `seed-test-users.ts` to create the new persona
  </Step>
  <Step title="Update Context">
    Add authentication flow to `rbac-context.ts`
  </Step>
  <Step title="Update All Matrices">
    Add expected status codes for the new persona across all matrices
  </Step>
</Steps>

## Methodology

<Tip>
See `test/TESTING_METHODOLOGY.md` for detailed guidance on analyzing controllers and DTOs before writing tests.
</Tip>

The testing approach focuses on:

1. **Permission Verification** ‚Äî Every endpoint enforces correct RBAC rules
2. **Stakeholder Access** ‚Äî Resource-level permissions work correctly
3. **Access Level Enforcement** ‚Äî READ vs WRITE vs FULL access levels are respected
4. **Edge Cases** ‚Äî Pool entities, team assignments, and admin-only operations

## CI/CD Integration

The test suite is designed to run in CI/CD pipelines:

- **Fast startup** ‚Äî Singleton context minimizes app boot time
- **Deterministic** ‚Äî Test data is seeded consistently
- **Isolated** ‚Äî Each test run creates fresh test data
- **Comprehensive** ‚Äî Matrix approach ensures complete permission coverage

<Warning>
Ensure test database is isolated from production and development databases
</Warning>
