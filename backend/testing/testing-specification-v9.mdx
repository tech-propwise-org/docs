## Overview

The test suite focuses on **RBAC (Role-Based Access Control) E2E testing** ‚Äî verifying that every API endpoint enforces the correct permissions for every user role.

<Info>
**Stack:** Jest + supertest + `@nestjs/testing`
</Info>

### Key Design Decisions

<CardGroup cols={2}>
  <Card title="Singleton App Context" icon="server">
    The NestJS app boots once and is shared across all test files (equivalent to NestJS `TestSuite` pattern, implemented manually since `TestSuite` requires NestJS 11+)
  </Card>
  <Card title="Matrix-Driven Tests" icon="grid">
    A declarative permission matrix defines expected HTTP status codes per persona √ó endpoint. A runner generates `it()` blocks automatically
  </Card>
  <Card title="API-Driven Seeding" icon="seedling">
    Test data is created through the actual API endpoints so all business logic (guards, auto-assignment, stakeholder creation) executes naturally
  </Card>
</CardGroup>

## Architecture

### Singleton Context Pattern

<Note>
The app boots once for the entire test run. All test files share the same NestJS app instance, authenticated tokens, and resolved `teamId`.
</Note>

```typescript
import { getContext, closeContext, RbacTestContext } from '../../setup/rbac-context';

let ctx: RbacTestContext;
beforeAll(async () => {
  ctx = await getContext();
}, 60000);
afterAll(async () => {
  await closeContext();
});
```

<Steps>
  <Step title="Jest Starts">
    First test file calls `getContext()` which boots NestJS app (~3-5s), logs in 5 personas, resolves teamId
  </Step>
  <Step title="Subsequent Files">
    Additional test files call `getContext()` and receive cached instance instantly (0ms)
  </Step>
  <Step title="Cleanup">
    Last file's `afterAll()` calls `closeContext()` when reference count hits 0, shutting down the app
  </Step>
</Steps>

The context replicates production config from `main.ts`:

- `ValidationPipe` (whitelist, transform, implicit conversion)
- URI versioning (`/v1/...`)
- Cookie parser

### Matrix Runner System

<Tip>
Instead of writing individual `it()` blocks, you define expectations declaratively using permission matrices.
</Tip>

```typescript
{
  method: 'GET',
  path: '/v1/leads/:leadByBasicId',
  description: 'View lead detail',
  expectations: {
    OWNER: 200,
    ADMIN: 200,
    TEAM_LEADER: 403,
    AGENT: 403,
    BASIC: 200,
  },
}
```

The runner calls `runRbacMatrix()` which:

<Steps>
  <Step title="Iterate Endpoints">
    Process every endpoint defined in the matrix
  </Step>
  <Step title="Test All Personas">
    For each endpoint, test all 5 personas (OWNER, ADMIN, TEAM_LEADER, AGENT, BASIC)
  </Step>
  <Step title="Resolve Placeholders">
    Replace path placeholders (`:leadByBasicId` ‚Üí actual UUID)
  </Step>
  <Step title="Execute Requests">
    Make HTTP requests with persona's tenant token and assert response status
  </Step>
  <Step title="Generate Tests">
    One matrix entry = 5 generated test cases with clear failure messages
  </Step>
</Steps>

## RBAC Personas

<Warning>
All personas are defined in `test/helpers/rbac-personas.ts` as a single source of truth.
</Warning>

| Key | Email | Org Role | Team | Description |
|-----|-------|----------|------|-------------|
| `OWNER` | owner@propwise.com | Owner | Any | All permissions including `system.owner` |
| `ADMIN` | admin@propwise.com | Admin | Any | All non-sensitive permissions (includes `system.admin`) |
| `TEAM_LEADER` | teamleader@propwise.com | Basic | Team Leader in Test Sales Team | All team permissions (`team_crm.*`, `team_sales.*`, `team.admin`) |
| `AGENT` | agent@propwise.com | Basic | Agent in Test Sales Team | No team permissions by default |
| `BASIC` | basic@propwise.com | Basic | None | No permissions ‚Äî relies entirely on stakeholder access |

### Permission Profiles

<Tabs>
  <Tab title="OWNER / ADMIN">
    Bypass resource-level checks via org permissions (`crm.view`, `crm.manage`, `sales.view`, `sales.manage`, etc.). Can access any entity in the org.
  </Tab>
  <Tab title="TEAM_LEADER">
    No org permissions, but has `team_crm.manage` and `team_sales.manage` in the team. Can access entities where the team is a stakeholder.
  </Tab>
  <Tab title="AGENT">
    No org permissions, no team permissions. Can only access entities where they are personally a stakeholder. Used to test access level differentiation (READ vs WRITE vs FULL).
  </Tab>
  <Tab title="BASIC">
    Same as AGENT but not a team member at all. Access is purely stakeholder-based.
  </Tab>
</Tabs>

### Login Flow

<Steps>
  <Step title="Identity Authentication">
    `POST /v1/auth/login` ‚Üí identity token
  </Step>
  <Step title="Organization Selection">
    `POST /v1/auth/select-organization` ‚Üí tenant token (org-scoped JWT)
  </Step>
  <Step title="API Requests">
    All subsequent API calls use the tenant token in the `Authorization: Bearer` header
  </Step>
</Steps>

### Access Mode Reference

<Note>
Your `@CheckAccess` decorator uses two modes:

- **OR (default):** User has the required permission **OR** has stakeholder-level access on the resource ‚Üí allowed
- **AND:** User must have the permission **AND** stakeholder access ‚Üí allowed
</Note>

**Resource Action Hierarchy:**

- `ACCESS` ‚Äî granted by `READ`, `WRITE`, or `FULL` access level
- `MODIFY` ‚Äî granted by `WRITE` or `FULL`
- `DELETE` ‚Äî granted by `FULL` only

## Test Data Seeding

### Step 1: Seed Users (One-Time Setup)

```bash
pnpm test:seed-users
```

<Check>
This runs `test/setup/seed-test-users.ts` which creates the foundation test data (idempotent operation).
</Check>

<Steps>
  <Step title="Create Test Users">
    Creates the 5 test users in the database (skips if they already exist)
  </Step>
  <Step title="Create Organization">
    Creates "Test Organization" with owner
  </Step>
  <Step title="Create Team">
    Creates "Test Sales Team" with team leader and agent
  </Step>
  <Step title="Assign Roles">
    Assigns org roles (Owner, Admin, Basic) and team roles (Team Leader, Agent)
  </Step>
  <Step title="RLS Bypass">
    Uses RLS bypass to write directly via MikroORM
  </Step>
</Steps>

### Step 2: Seed Test Data (Automatic Per Test Run)

Each test file calls `seedRbacTestData(app, personas, teamId)` in its `beforeAll`. Results are cached across files.

<Warning>
This creates entities through the API to ensure all business logic executes naturally.
</Warning>

**Created Entities:**

<CodeGroup>

```

```typescript Other Entities

## How to Run

### Core Test Suites

```bash
# Run all core RBAC tests (excludes distribution + escalation queue)
pnpm test:e2e

# Run specific test file
pnpm test:e2e -- lead-rbac.e2e-spec.ts

# Run with verbose output
pnpm test:e2e -- --verbose
```

### Distribution & Escalation Queue Tests

<Info>
These tests require live pg-boss and are run separately.
</Info>

```bash
# Run queue-dependent tests
pnpm test:e2e:queue

# Run specific queue test
pnpm test:e2e:queue -- distribution
```

### Cleanup Commands

```bash
# Clean up test users from database
pnpm test:cleanup-users

# Reset test environment
pnpm test:seed-users && pnpm test:e2e
```

### Adding New Personas

<Warning>
Adding personas requires updates in multiple files and careful consideration of permission combinations.
</Warning>

<Steps>
  <Step title="Update Personas Definition">
    Add new persona to `test/helpers/rbac-personas.ts`
  </Step>
  <Step title="Update Seeding">

## Methodology

<Tip>
See `test/TESTING_METHODOLOGY.md` for detailed guidance on analyzing controllers and DTOs before writing tests.
</Tip>

### Test-First Approach

1. **Analyze Controller** ‚Äî identify all endpoints, required permissions, and stakeholder checks
2. **Map Permission Requirements** ‚Äî document which personas should have access
3. **Create Matrix** ‚Äî define expected status codes for each persona √ó endpoint combination
4. **Seed Test Data** ‚Äî create entities with appropriate stakeholder relationships
5. **Run Tests** ‚Äî execute matrix-driven tests and verify behavior
6. **Document Edge Cases** ‚Äî capture any special permission logic or exceptions

### Integration Test Scenarios

Manual scenario testing is documented in `test/INTEGRATION_TEST_SCENARIOS.md` with prefixed identifiers:

- **LC-*** ‚Äî Lead conversion scenarios
- **LOP-*** ‚Äî Lead operation scenarios  
- **DS-*** ‚Äî Deal stage scenarios
- **ES-*** ‚Äî Escalation scenarios

## CI/CD Integration

<Info>
The test suite is designed to run in CI/CD environments with proper database isolation and cleanup.
</Info>

### Environment Requirements

- PostgreSQL test database
- Redis instance (for session management)
- Environment variables for test configuration
- pg-boss disabled for core tests (mocked)

### Performance Optimizations

- **Singleton app context** ‚Äî eliminates repeated NestJS bootstrap overhead
- **Cached test data** ‚Äî entities created once and reused across test files
- **Parallel execution** ‚Äî Jest runs test files in parallel where possible
- **Smart seeding** ‚Äî API-driven seeding ensures realistic test conditions

### Monitoring & Reporting

Tests produce detailed failure reports including:

- Expected vs actual HTTP status codes
- Full response bodies for debugging
- Persona and endpoint context
- Matrix position for easy location of failures

```bash

Matrix: LEAD_BY_BASIC_MATRIX[2] - View lead detail
