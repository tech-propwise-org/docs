Architecture, conventions, and run guide for the PropWise CRM backend test suite.
</Info>

## Overview

The test suite focuses on **RBAC (Role-Based Access Control) E2E testing** ‚Äî verifying that every API endpoint enforces the correct permissions for every user role.

**Stack:** Jest + supertest + `@nestjs/testing`

### Key Design Decisions

<CardGroup cols={2}>
  <Card title="Singleton App Context" icon="server">
    The NestJS app boots once and is shared across all test files (equivalent to NestJS `TestSuite` pattern, implemented manually since `TestSuite` requires NestJS 11+)
  </Card>
  <Card title="Matrix-Driven Tests" icon="table">
    A declarative permission matrix defines expected HTTP status codes per persona √ó endpoint. A runner generates `it()` blocks automatically
  </Card>
  <Card title="API-Driven Seeding" icon="seedling">
    Test data is created through the actual API endpoints so all business logic (guards, auto-assignment, stakeholder creation) executes naturally
  </Card>
</CardGroup>

## Architecture

### Singleton Context

The app boots once for the entire test run. All test files share the same NestJS app instance, authenticated tokens, and resolved `teamId`.

<Steps>
  <Step title="Jest Starts">
    The first test file calls `getContext()` which boots the NestJS app (~3-5s), logs in 5 personas, and resolves teamId
  </Step>
  <Step title="Context Sharing">
    Subsequent test files call `getContext()` and receive the cached instance instantly (0ms)
  </Step>
  <Step title="Cleanup">
    The last file's `afterAll()` calls `closeContext()` when the reference count hits 0, shutting down the app
  </Step>
</Steps>

#### Implementation Example

```typescript
import { getContext, closeContext, RbacTestContext } from '../../setup/rbac-context';

let ctx: RbacTestContext;
beforeAll(async () => {
  ctx = await getContext();
}, 60000);
afterAll(async () => {
  await closeContext();
});
```

<Note>
The context replicates production config from `main.ts` including `ValidationPipe` (whitelist, transform, implicit conversion), URI versioning (`/v1/...`), and cookie parser.
</Note>

### Matrix Runner

Instead of writing individual `it()` blocks, you define expectations declaratively:

```typescript
{
  method: 'GET',
  path: '/v1/leads/:leadByBasicId',
  description: 'View lead detail',
  expectations: {
    OWNER: 200,
    ADMIN: 200,
    TEAM_LEADER: 403,
    AGENT: 403,
    BASIC: 200,
  },
}
```

The runner calls `runRbacMatrix()` which:

<Steps>
  <Step title="Iterate Endpoints">
    Iterates every endpoint in the matrix
  </Step>
  <Step title="Test All Personas">
    For each endpoint, iterates all 5 personas
  </Step>
  <Step title="Resolve Placeholders">
    Resolves path placeholders (`:leadByBasicId` ‚Üí actual UUID)
  </Step>
  <Step title="Make Requests">
    Makes the HTTP request with the persona's tenant token
  </Step>
  <Step title="Assert Results">
    Asserts the response status matches the expected value and produces clear failure messages with the actual response body
  </Step>
</Steps>

<Tip>
One matrix entry = 5 generated test cases automatically.
</Tip>

## RBAC Personas

Defined in `test/helpers/rbac-personas.ts`. All personas share a single source of truth.

| Key | Email | Org Role | Team | Description |
|-----|-------|----------|------|-------------|
| `OWNER` | owner@propwise.com | Owner | Any | All permissions including `system.owner` |
| `ADMIN` | admin@propwise.com | Admin | Any | All non-sensitive permissions (includes `system.admin`) |
| `TEAM_LEADER` | teamleader@propwise.com | Basic | Team Leader in Test Sales Team | All team permissions (`team_crm.*`, `team_sales.*`, `team.admin`) |
| `AGENT` | agent@propwise.com | Basic | Agent in Test Sales Team | No team permissions by default |
| `BASIC` | basic@propwise.com | Basic | None | No permissions ‚Äî relies entirely on stakeholder access |

### Permission Profiles

<AccordionGroup>
  <Accordion title="OWNER / ADMIN">
    Bypass resource-level checks via org permissions (`crm.view`, `crm.manage`, `sales.view`, `sales.manage`, etc.). Can access any entity in the org.
  </Accordion>
  <Accordion title="TEAM_LEADER">
    No org permissions, but has `team_crm.manage` and `team_sales.manage` in the team. Can access entities where the team is a stakeholder.
  </Accordion>
  <Accordion title="AGENT">
    No org permissions, no team permissions. Can only access entities where they are personally a stakeholder. Used to test access level differentiation (READ vs WRITE vs FULL).
  </Accordion>
  <Accordion title="BASIC">
    Same as AGENT but not a team member at all. Access is purely stakeholder-based.
  </Accordion>
</AccordionGroup>

### Login Flow

The context performs a two-step login for each persona:

<Steps>
  <Step title="Identity Authentication">
    `POST /v1/auth/login` ‚Üí identity token
  </Step>
  <Step title="Organization Selection">
    `POST /v1/auth/select-organization` ‚Üí tenant token (org-scoped JWT)
  </Step>
  <Step title="API Requests">
    All subsequent API calls use the tenant token in the `Authorization: Bearer` header
  </Step>
</Steps>

## Permission Matrix

Matrices are split into per-entity files under `test/setup/matrices/`. The barrel at `test/setup/rbac-matrix.ts` re-exports everything.

### Lead Matrices

<Tabs>
  <Tab title="Lead by Basic">
    **Matrix:** `LEAD_BY_BASIC_MATRIX`
    **Entity:** Lead created by BASIC
    **Who Can Access:** OWNER, ADMIN (via `crm.*`), BASIC (via FULL stakeholder)
    **Tests:** View, update, stakeholders, activities, property interests
  </Tab>
  <Tab title="Lead on Team">
    **Matrix:** `LEAD_ON_TEAM_MATRIX`
    **Entity:** Lead with team as stakeholder
    **Who Can Access:** OWNER, ADMIN, TEAM_LEADER (via `team_crm.*`)
    **Tests:** View, update, stakeholders, activities
  </Tab>
  <Tab title="Lead Agent Read">
    **Matrix:** `LEAD_AGENT_READ_MATRIX`
    **Entity:** Lead where AGENT has READ access
    **Who Can Access:** OWNER, ADMIN, AGENT (view only ‚Äî modify denied)
    **Tests:** View allowed, update denied, access level enforcement
  </Tab>
  <Tab title="Lead in Pool">
    **Matrix:** `LEAD_IN_POOL_MATRIX`
    **Entity:** Lead with no stakeholders
    **Who Can Access:** OWNER, ADMIN only (via `crm.*`)
    **Tests:** View, update
  </Tab>
  <Tab title="Lead Admin Only">
    **Matrix:** `LEAD_ADMIN_ONLY_MATRIX`
    **Entity:** Lead for admin-only ops
    **Who Can Access:** OWNER, ADMIN (via `system.admin`)
    **Tests:** Delete
  </Tab>
</Tabs>

### Other Entity Matrices

| Entity | Matrix | Who Can Access |
|--------|--------|----------------|
| Deal created by OWNER | `DEAL_BY_OWNER_MATRIX` | OWNER, ADMIN only (via `sales.*`) |
| Contact created by BASIC | `CONTACT_BY_BASIC_MATRIX` | OWNER, ADMIN, BASIC |
| Contact created by OWNER | `CONTACT_BY_OWNER_MATRIX` | OWNER, ADMIN only |
| Company created by BASIC | `COMPANY_BY_BASIC_MATRIX` | OWNER, ADMIN, BASIC |
| Company created by OWNER | `COMPANY_BY_OWNER_MATRIX` | OWNER, ADMIN only |

### Access Mode Reference

Your `@CheckAccess` decorator uses two modes:

<Warning>
- **OR (default):** User has the required permission **OR** has stakeholder-level access on the resource ‚Üí allowed.
- **AND:** User must have the permission **AND** stakeholder access ‚Üí allowed.
</Warning>

Resource action hierarchy:

- `ACCESS` ‚Äî granted by `READ`, `WRITE`, or `FULL` access level
- `MODIFY` ‚Äî granted by `WRITE` or `FULL`
- `DELETE` ‚Äî granted by `FULL` only

## Test Data Seeding

### Step 1: Seed Users

Run this one-time setup before running tests:

```bash
pnpm test:seed-users
```

This runs `test/setup/seed-test-users.ts` which:

<Steps>
  <Step title="Create Personas">
    Creates the RBAC test personas in the database (idempotent ‚Äî skips if they exist)
  </Step>
  <Step title="Create Organization">
    Creates "Test Organization" with owner
  </Step>
  <Step title="Setup Subscription">
    Ensures **ACTIVE BUSINESS `subscription`** and **completed `organization_setup_progress`** on the test org (self-heals stale FREE / `LEAD_PIPELINE` rows from older seeds so Playwright and RBAC E2E skip onboarding and paid-plan gates)
  </Step>
  <Step title="Create Team">
    Creates "Test Sales Team" with team leader and agent
  </Step>
  <Step title="Assign Roles">
    Assigns org roles (Owner, Admin, Basic) and team roles (Team Leader, Agent)
  </Step>
  <Step title="Database Write">
    Uses RLS bypass to write directly via MikroORM
  </Step>
</Steps>

<Note>
Requires subscription plans to exist (run `pnpm test:init-data` once on a fresh DB before `pnpm test:seed-users`).
</Note>

### Step 2: Seed Test Data

Each test file calls `seedRbacTestData(app, personas, teamId)` in its `beforeAll`. Results are cached across files. This creates entities through the API:

#### Leads
- One by BASIC (auto-assigned as primary stakeholder with FULL access)
- One by OWNER (for admin-only tests)
- One in pool (empty stakeholders)
- One with team as stakeholder (OWNER creates, assigns team ‚Äî TEAM_LEADER gets access)
- One with AGENT as READ-only stakeholder (OWNER creates, adds AGENT with access level)

#### Other Entities
- **Deals:** One by OWNER (sales permissions required)
- **Contacts:** One by BASIC, one by OWNER
- **Companies:** One by BASIC, one by OWNER

## How to Run

### Prerequisites

<Steps>
  <Step title="Initialize Database">
    ```bash
    pnpm test:init-data  # One-time: creates subscription plans
    ```
  </Step>
  <Step title="Seed Test Users">
    ```bash
    pnpm test:seed-users  # One-time: creates RBAC personas
    ```
  </Step>
</Steps>

### Running Tests

<CodeGroup>
```bash Core E2E Tests
# Run all RBAC + auth tests (excludes queue tests)
pnpm test:e2e

# Run specific test files
pnpm test:e2e -- --testPathPattern=lead-rbac
pnpm test:e2e -- --testPathPattern=auth-lifecycle
```

```bash Queue Tests
# Run queue tests with live pg-boss
pnpm test:e2e:queue

# Run specific queue test
pnpm test:e2e:queue -- --testPathPattern=distribution
```

```bash Development
# Run in watch mode
pnpm test:e2e -- --watch

# Run with verbose output
pnpm test:e2e -- --verbose

# Run single test case
pnpm test:e2e -- --testNamePattern="OWNER should receive 200"
```
</CodeGroup>

### Cleanup

```bash
# Remove test users (if needed)
pnpm test:cleanup-users
```

## How to Extend

### Adding New Endpoints

<Steps>
  <Step title="Update Matrix">
    Add the endpoint to the appropriate matrix file in `test/setup/matrices/`:
    
    ```typescript
    {
      method: 'POST',
      path: '/v1/leads/:leadByBasicId/notes',
      description: 'Create lead note',
      expectations: {
        OWNER: 201,
        ADMIN: 201,
        TEAM_LEADER: 403,
        AGENT: 403,
        BASIC: 201,
      },
      payload: { content: 'Test note' },
    }
    ```
  </Step>
  <Step title="Run Tests">
    The runner will automatically generate test cases for all 5 personas
  </Step>
</Steps>

### Adding New Entities

<Steps>
  <Step title="Create Matrix File">
    Create `test/setup/matrices/new-entity-matrix.ts`
  </Step>
  <Step title="Update Barrel Export">
    Add to `test/setup/matrices/index.ts`
  </Step>
  <Step title="Create Test File">
    Create `test/e2e/rbac/new-entity-rbac.e2e-spec.ts`
  </Step>
  <Step title="Update Seeding">
    Add entity creation logic to `test/setup/rbac-test-data.ts`
  </Step>
</Steps>

### Adding New Personas

<Steps>
  <Step title="Update Personas File">
    Add to `test/helpers/rbac-personas.ts`
  </Step>
  <Step title="Update Seeder">
    Add user creation logic to `test/setup/seed-test-users.ts`
  </Step>
  <Step title="Update Context">
    Add authentication logic to `test/setup/rbac-context.ts`
  </Step>
  <Step title="Update Matrices">
    Add expectations for the new persona in all matrices
  </Step>
</Steps>

## Methodology

### Test-Driven Development

<Check>
1. Analyze controller endpoints and DTOs (see `TESTING_METHODOLOGY.md`)
2. Define expected permissions per persona in the matrix
3. Run tests to see failures
4. Implement/fix guards and permissions
5. Tests pass ‚Üí endpoint is properly secured
</Check>

### Matrix Design Principles

<CardGroup cols={2}>
  <Card title="Declarative" icon="list">
    Define what should happen, not how to test it
  </Card>
  <Card title="Comprehensive" icon="shield">
    Every persona √ó endpoint combination is tested
  </Card>
  <Card title="Maintainable" icon="wrench">
    Changes to permissions only require matrix updates
  </Card>
  <Card title="Clear" icon="eye">
    Failures show exact persona, endpoint, expected vs actual status
  </Card>
</CardGroup>

## CI/CD Integration

### GitHub Actions

The test suite integrates with CI/CD pipelines:

```yaml
- name: Run E2E Tests
  run: |
    pnpm test:init-data
    pnpm test:seed-users  
    pnpm test:e2e
    pnpm test:e2e:queue
```

### Database Requirements

<Warning>
Tests require a clean PostgreSQL database with:
- All migrations applied
- Subscription plans seeded (`test:init-data`)
- Test users seeded (`test:seed-users`)
</Warning>

### Environment Variables

Required environment variables for testing:

```bash
DATABASE_URL=postgresql://...
JWT_SECRET=test-secret
EMAIL_SERVICE_ENABLED=false  # Disable email sending in tests
QUEUE_ENABLED=true          # Enable for queue tests
```

### Performance Considerations

- **Cold start:** First test file takes ~5 seconds (app boot)
- **Warm runs:** Subsequent files start instantly (shared context)
- **Parallel execution:** Not recommended (shared database state)
- **Memory usage:** ~200MB for full test suite

<Tip>
Use `--testPathPattern` to run specific test files during development to speed up feedback loops.
</Tip>
