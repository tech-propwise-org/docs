## Overview

The test suite focuses on **RBAC (Role-Based Access Control) E2E testing** — verifying that every API endpoint enforces the correct permissions for every user role.

<Info>
**Stack:** Jest + supertest + `@nestjs/testing`
</Info>

### Key Design Decisions

<CardGroup cols={2}>
  <Card title="Singleton App Context" icon="server">
    The NestJS app boots once and is shared across all test files (equivalent to NestJS `TestSuite` pattern, implemented manually since `TestSuite` requires NestJS 11+)
  </Card>
  <Card title="Matrix-driven Tests" icon="table">
    A declarative permission matrix defines expected HTTP status codes per persona × endpoint. A runner generates `it()` blocks automatically
  </Card>
  <Card title="API-driven Seeding" icon="seedling">
    Test data is created through the actual API endpoints so all business logic (guards, auto-assignment, stakeholder creation) executes naturally
  </Card>
</CardGroup>

## Architecture

### Singleton Context (`rbac-context.ts`)

The app boots once for the entire test run. All test files share the same NestJS app instance, authenticated tokens, and resolved `teamId`.

<Steps>
  <Step title="Jest Starts">
    First test file calls `getContext()` which boots NestJS app (~3-5s), logs in 5 personas, resolves teamId
  </Step>
  <Step title="Context Sharing">
    Subsequent test files call `getContext()` and receive cached instance instantly (0ms)
  </Step>
  <Step title="Cleanup">
    Last file's `afterAll()` calls `closeContext()` when reference count hits 0, shutting down the app
  </Step>
</Steps>

**Usage Pattern:**

```typescript
import { getContext, closeContext, RbacTestContext } from '../../setup/rbac-context';

let ctx: RbacTestContext;
beforeAll(async () => {
  ctx = await getContext();
}, 60000);
afterAll(async () => {
  await closeContext();
});
```

<Note>
The context replicates production config from `main.ts` including `ValidationPipe`, URI versioning (`/v1/...`), and cookie parser.
</Note>

### Matrix Runner (`rbac-runner.ts`)

Instead of writing individual `it()` blocks, you define expectations declaratively:

```typescript
{
  method: 'GET',
  path: '/v1/leads/:leadByBasicId',
  description: 'View lead detail',
  expectations: {
    OWNER: 200,
    ADMIN: 200,
    TEAM_LEADER: 403,
    AGENT: 403,
    BASIC: 200,
  },
}
```

<Steps>
  <Step title="Matrix Iteration">
    Runner calls `runRbacMatrix()` and iterates every endpoint in the matrix
  </Step>
  <Step title="Persona Testing">
    For each endpoint, iterates all 5 personas
  </Step>
  <Step title="Path Resolution">
    Resolves path placeholders (`:leadByBasicId` → actual UUID)
  </Step>
  <Step title="Request Execution">
    Makes HTTP request with the persona's tenant token
  </Step>
  <Step title="Assertion">
    Asserts response status matches expected value with clear failure messages
  </Step>
</Steps>

<Tip>
One matrix entry = 5 generated test cases automatically.
</Tip>

## RBAC Personas

Defined in `test/helpers/rbac-personas.ts`. All personas share a single source of truth.

| Key | Email | Org Role | Team | Description |
|-----|-------|----------|------|-------------|
| `OWNER` | owner@propwise.com | Owner | Any | All permissions including `system.owner` |
| `ADMIN` | admin@propwise.com | Admin | Any | All non-sensitive permissions (includes `system.admin`) |
| `TEAM_LEADER` | teamleader@propwise.com | Basic | Team Leader in Test Sales Team | All team permissions (`team_crm.*`, `team_sales.*`, `team.admin`) |
| `AGENT` | agent@propwise.com | Basic | Agent in Test Sales Team | No team permissions by default |
| `BASIC` | basic@propwise.com | Basic | None | No permissions — relies entirely on stakeholder access |

### Permission Profiles

<AccordionGroup>
  <Accordion title="OWNER / ADMIN">
    Bypass resource-level checks via org permissions (`crm.view`, `crm.manage`, `sales.view`, `sales.manage`, etc.). Can access any entity in the org.
  </Accordion>
  
  <Accordion title="TEAM_LEADER">
    No org permissions, but has `team_crm.manage` and `team_sales.manage` in the team. Can access entities where the team is a stakeholder.
  </Accordion>
  
  <Accordion title="AGENT">
    No org permissions, no team permissions. Can only access entities where they are personally a stakeholder. Used to test access level differentiation (READ vs WRITE vs FULL).
  </Accordion>
  
  <Accordion title="BASIC">
    Same as AGENT but not a team member at all. Access is purely stakeholder-based.
  </Accordion>
</AccordionGroup>

### Login Flow

The context performs a two-step login for each persona:

<Steps>
  <Step title="Identity Authentication">
    `POST /v1/auth/login` → identity token
  </Step>
  <Step title="Organization Selection">
    `POST /v1/auth/select-organization` → tenant token (org-scoped JWT)
  </Step>
</Steps>

<Note>
All subsequent API calls use the tenant token in the `Authorization: Bearer` header.
</Note>

## Permission Matrix

Matrices are split into per-entity files under `test/setup/matrices/`. The barrel at `test/setup/rbac-matrix.ts` re-exports everything.

### Lead Matrices (`matrices/lead-matrix.ts`)

| Matrix | Entity | Who Can Access | Tests |
|--------|--------|----------------|-------|
| `LEAD_BY_BASIC_MATRIX` | Lead created by BASIC | OWNER, ADMIN (via `crm.*`), BASIC (via FULL stakeholder) | View, update, stakeholders, activities, property interests |
| `LEAD_ON_TEAM_MATRIX` | Lead with team as stakeholder | OWNER, ADMIN, TEAM_LEADER (via `team_crm.*`) | View, update, stakeholders, activities |
| `LEAD_AGENT_READ_MATRIX` | Lead where AGENT has READ access | OWNER, ADMIN, AGENT (view only — modify denied) | View allowed, update denied, access level enforcement |
| `LEAD_IN_POOL_MATRIX` | Lead with no stakeholders | OWNER, ADMIN only (via `crm.*`) | View, update |
| `LEAD_ADMIN_ONLY_MATRIX` | Lead for admin-only ops | OWNER, ADMIN (via `system.admin`) | Delete |

### Deal Matrices (`matrices/deal-matrix.ts`)

| Matrix | Entity | Who Can Access |
|--------|--------|----------------|
| `DEAL_BY_OWNER_MATRIX` | Deal created by OWNER | OWNER, ADMIN only (via `sales.*`) |

### Contact Matrices (`matrices/contact-matrix.ts`)

| Matrix | Entity | Who Can Access |
|--------|--------|----------------|
| `CONTACT_BY_BASIC_MATRIX` | Contact created by BASIC | OWNER, ADMIN, BASIC |
| `CONTACT_BY_OWNER_MATRIX` | Contact created by OWNER | OWNER, ADMIN only |

### Company Matrices (`matrices/company-matrix.ts`)

| Matrix | Entity | Who Can Access |
|--------|--------|----------------|
| `COMPANY_BY_BASIC_MATRIX` | Company created by BASIC | OWNER, ADMIN, BASIC |
| `COMPANY_BY_OWNER_MATRIX` | Company created by OWNER | OWNER, ADMIN only |

### Access Mode Reference

Your `@CheckAccess` decorator uses two modes:

<Tabs>
  <Tab title="OR Mode (Default)">
    User has the required permission **OR** has stakeholder-level access on the resource → allowed.
  </Tab>
  <Tab title="AND Mode">
    User must have the permission **AND** stakeholder access → allowed.
  </Tab>
</Tabs>

**Resource Action Hierarchy:**

- `ACCESS` — granted by `READ`, `WRITE`, or `FULL` access level
- `MODIFY` — granted by `WRITE` or `FULL`
- `DELETE` — granted by `FULL` only

## Test Data Seeding

### Step 1: Seed Users (One-time Setup)

<Warning>
Run this command before executing tests for the first time.
</Warning>

```bash
pnpm test:seed-users
```

This runs `test/setup/seed-test-users.ts` which:

<Steps>
  <Step title="User Creation">
    Creates the 5 test users in the database (idempotent — skips if they exist)
  </Step>
  <Step title="Organization Setup">
    Creates "Test Organization" with owner
  </Step>
  <Step title="Team Creation">
    Creates "Test Sales Team" with team leader and agent
  </Step>
  <Step title="Role Assignment">
    Assigns org roles (Owner, Admin, Basic) and team roles (Team Leader, Agent)
  </Step>
</Steps>

<Note>
Uses RLS bypass to write directly via MikroORM.
</Note>

### Step 2: Seed Test Data (Automatic)

Each test file calls `seedRbacTestData(app, personas, teamId)` in its `beforeAll`. Results are cached across files. This creates entities through the API:

<AccordionGroup>
  <Accordion title="Leads">
    - One by BASIC (auto-assigned as primary stakeholder with FULL access)
    - One by OWNER (for admin-only tests)
    - One in pool (empty stakeholders)
    - One with team as stakeholder (OWNER creates, assigns team — TEAM_LEADER gets access)
    - One with AGENT as READ-only stakeholder (OWNER creates, adds AGENT with `accessLevel: read`)
  </Accordion>
  
  <Accordion title="Contacts & Companies">
    - Contacts: one by BASIC, one by OWNER
    - Companies: one by BASIC, one by OWNER
  </Accordion>
  
  <Accordion title="Deals & Events">
    - Deals: one by OWNER (from lead + contact)
    - Events: one linked to the OWNER lead, one with BASIC as an invitee
  </Accordion>
  
  <Accordion title="Escalation Rules">
    - One OWNER-created rule for escalation-rule RBAC update/delete matrices
  </Accordion>
</AccordionGroup>

## How to Run

<Tabs>
  <Tab title="Core E2E Tests">
    ```bash
    # Run auth + RBAC tests (excludes distribution/escalation queue)
    pnpm test:e2e
    
    # Run specific test file
    pnpm test:e2e test/e2e/rbac/lead-rbac.e2e-spec.ts
    
    # Run with verbose output
    pnpm test:e2e --verbose
    ```
  </Tab>
  
  <Tab title="Queue Tests">
    ```bash
    # Run distribution + escalation queue tests (requires live pg-boss)
    pnpm test:e2e:queue
    
    # Run specific queue test
    pnpm test:e2e:queue test/e2e/distribution/lead-distribution.e2e-spec.ts
    ```
  </Tab>
  
  <Tab title="All Tests">
    ```bash
    # Run everything
    pnpm test:e2e:all
    ```
  </Tab>
</Tabs>

<Warning>
**Prerequisites:**
1. Run `pnpm test:seed-users` once before first test run
2. Ensure test database is running and accessible
3. For queue tests, ensure pg-boss is running
</Warning>

## How to Extend

### Adding New Entities

<Steps>
  <Step title="Create Matrix File">
    Add a new matrix file under `test/setup/matrices/` (e.g., `property-matrix.ts`)
  </Step>
  <Step title="Define Personas">
    Define test data creation and access expectations for all 5 personas
  </Step>
  <Step title="Export Matrix">
    Add to `test/setup/matrices/index.ts` barrel export
  </Step>
  <Step title="Create Test File">
    Create corresponding test file under `test/e2e/rbac/` (e.g., `property-rbac.e2e-spec.ts`)
  </Step>
  <Step title="Update Seeding">
    Add entity creation logic to `test/setup/rbac-test-data.ts` if needed
  </Step>
</Steps>

### Adding New Endpoints

To add a new endpoint to an existing entity matrix:

```typescript
{
  method: 'POST',
  path: '/v1/leads/:leadByBasicId/notes',
  description: 'Add note to lead',
  expectations: {
    OWNER: 201,
    ADMIN: 201,
    TEAM_LEADER: 403,
    AGENT: 403,
    BASIC: 201,
  },
  body: { content: 'Test note' }
}
```

### Adding New Personas

<Steps>
  <Step title="Update Personas Config">
    Add new persona to `test/helpers/rbac-personas.ts`
  </Step>
  <Step title="Update Seeder">
    Modify `test/setup/seed-test-users.ts` to create the new user
  </Step>
  <Step title="Update Context">
    Update `rbac-context.ts` to authenticate the new persona
  </Step>
  <Step title="Update All Matrices">
    Add expectations for the new persona across all matrix files
  </Step>
</Steps>

## Methodology

<Check>
**Before writing tests, analyze the controller/DTO structure** as documented in `TESTING_METHODOLOGY.md`
</Check>

### Key Principles

1. **Test through the API** — Don't mock business logic, test the actual endpoints
2. **Use realistic data** — Create entities through proper flows to test all guards
3. **Matrix-driven coverage** — Ensure every persona × endpoint combination is tested
4. **Cache expensive operations** — Share app context and test data across files
5. **Clear failure messages** — Include response bodies in assertions for debugging

## CI/CD Integration

The test suite is designed for CI/CD environments:

- **Fast startup** — Singleton context reduces boot time from ~20s to ~5s for full suite
- **Parallel execution** — Each test file can run independently after seeding
- **Clean state** — Tests use isolated data and proper cleanup
- **Clear reporting** — Matrix runner provides detailed failure context

<Tip>
Configure your CI to run `pnpm test:seed-users` once before the test suite in the pipeline setup phase.
</Tip>
