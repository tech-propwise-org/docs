## Overview

The test suite focuses on **RBAC (Role-Based Access Control) E2E testing** — verifying that every API endpoint enforces the correct permissions for every user role.

**Stack:** Jest + supertest + `@nestjs/testing`

<Info>
**Key design decisions:**
- **Singleton app context** — the NestJS app boots once and is shared across all test files (equivalent to NestJS `TestSuite` pattern, implemented manually since `TestSuite` requires NestJS 11+)
- **Matrix-driven tests** — a declarative permission matrix defines expected HTTP status codes per persona × endpoint. A runner generates `it()` blocks automatically
- **API-driven seeding** — test data is created through the actual API endpoints so all business logic (guards, auto-assignment, stakeholder creation) executes naturally
</Info>

## Architecture

### Singleton Context

The app boots once for the entire test run. All test files share the same NestJS app instance, authenticated tokens, and resolved `teamId`.

```typescript
Jest starts
  → lead-rbac.e2e-spec.ts calls getContext()
    → First call: boots NestJS app (~3-5s), logs in 5 personas, resolves teamId
    → Returns { app, personas, teamId }
  → deal-rbac.e2e-spec.ts calls getContext()
    → Returns cached instance instantly (0ms)
  → ... same for contact and company files
  → Last file's afterAll() calls closeContext()
    → Reference count hits 0 → app shuts down
```

<CodeGroup>

```typescript rbac-context.ts
import { getContext, closeContext, RbacTestContext } from '../../setup/rbac-context';

let ctx: RbacTestContext;

beforeAll(async () => {
  ctx = await getContext();
}, 60000);

afterAll(async () => {
  await closeContext();
});
```

</CodeGroup>

<Note>
The context replicates production config from `main.ts`:
- `ValidationPipe` (whitelist, transform, implicit conversion)
- URI versioning (`/v1/...`)
- Cookie parser
</Note>

### Matrix Runner

Instead of writing individual `it()` blocks, you define expectations declaratively:

<CodeGroup>

```typescript matrix-definition.ts
{
  method: 'GET',
  path: '/v1/leads/:leadByBasicId',
  description: 'View lead detail',
  expectations: {
    OWNER: 200,
    ADMIN: 200,
    TEAM_LEADER: 403,
    AGENT: 403,
    BASIC: 200,
  },
}
```

</CodeGroup>

<Steps>

<Step title="Iterate endpoints">
The runner calls `runRbacMatrix()` which iterates every endpoint in the matrix
</Step>

<Step title="Iterate personas">
For each endpoint, iterates all 5 personas
</Step>

<Step title="Resolve placeholders">
Resolves path placeholders (`:leadByBasicId` → actual UUID)
</Step>

<Step title="Make request">
Makes the HTTP request with the persona's tenant token
</Step>

<Step title="Assert status">
Asserts the response status matches the expected value and produces clear failure messages with the actual response body
</Step>

</Steps>

<Tip>
One matrix entry = 5 generated test cases
</Tip>

## RBAC Personas

Defined in `test/helpers/rbac-personas.ts`. All personas share a single source of truth.

| Key | Email | Org Role | Team | Description |
|-----|-------|----------|------|-------------|
| `OWNER` | owner@propwise.com | Owner | Any | All permissions including `system.owner` |
| `ADMIN` | admin@propwise.com | Admin | Any | All non-sensitive permissions (includes `system.admin`) |
| `TEAM_LEADER` | teamleader@propwise.com | Basic | Team Leader in Test Sales Team | All team permissions (`team_crm.*`, `team_sales.*`, `team.admin`) |
| `AGENT` | agent@propwise.com | Basic | Agent in Test Sales Team | No team permissions by default |
| `BASIC` | basic@propwise.com | Basic | None | No permissions — relies entirely on stakeholder access |

### Permission Profiles

<AccordionGroup>

<Accordion title="OWNER / ADMIN">
Bypass resource-level checks via org permissions (`crm.view`, `crm.manage`, `sales.view`, `sales.manage`, etc.). Can access any entity in the org.
</Accordion>

<Accordion title="TEAM_LEADER">
No org permissions, but has `team_crm.manage` and `team_sales.manage` in the team. Can access entities where the team is a stakeholder.
</Accordion>

<Accordion title="AGENT">
No org permissions, no team permissions. Can only access entities where they are personally a stakeholder. Used to test access level differentiation (READ vs WRITE vs FULL).
</Accordion>

<Accordion title="BASIC">
Same as AGENT but not a team member at all. Access is purely stakeholder-based.
</Accordion>

</AccordionGroup>

### Login Flow

<Steps>

<Step title="Request identity token">
`POST /v1/auth/login` → identity token
</Step>

<Step title="Select organization">
`POST /v1/auth/select-organization` → tenant token (org-scoped JWT)
</Step>

<Step title="Make authenticated requests">
All subsequent API calls use the tenant token in the `Authorization: Bearer` header
</Step>

</Steps>

## Permission Matrix

Matrices are split into per-entity files under `test/setup/matrices/`. The barrel at `test/setup/rbac-matrix.ts` re-exports everything.

### Lead Matrices

| Matrix | Entity | Who Can Access | Tests |
|--------|--------|----------------|-------|
| `LEAD_BY_BASIC_MATRIX` | Lead created by BASIC | OWNER, ADMIN (via `crm.*`), BASIC (via FULL stakeholder) | View, update, stakeholders, activities, property interests |
| `LEAD_ON_TEAM_MATRIX` | Lead with team as stakeholder | OWNER, ADMIN, TEAM_LEADER (via `team_crm.*`) | View, update, stakeholders, activities |
| `LEAD_AGENT_READ_MATRIX` | Lead where AGENT has READ access | OWNER, ADMIN, AGENT (view only — modify denied) | View allowed, update denied, access level enforcement |
| `LEAD_IN_POOL_MATRIX` | Lead with no stakeholders | OWNER, ADMIN only (via `crm.*`) | View, update |
| `LEAD_ADMIN_ONLY_MATRIX` | Lead for admin-only ops | OWNER, ADMIN (via `system.admin`) | Delete |

### Deal Matrices

| Matrix | Entity | Who Can Access |
|--------|--------|----------------|
| `DEAL_BY_OWNER_MATRIX` | Deal created by OWNER | OWNER, ADMIN only (via `sales.*`) |

### Contact Matrices

| Matrix | Entity | Who Can Access |
|--------|--------|----------------|
| `CONTACT_BY_BASIC_MATRIX` | Contact created by BASIC | OWNER, ADMIN, BASIC |
| `CONTACT_BY_OWNER_MATRIX` | Contact created by OWNER | OWNER, ADMIN only |

### Company Matrices

| Matrix | Entity | Who Can Access |
|--------|--------|----------------|
| `COMPANY_BY_BASIC_MATRIX` | Company created by BASIC | OWNER, ADMIN, BASIC |
| `COMPANY_BY_OWNER_MATRIX` | Company created by OWNER | OWNER, ADMIN only |

### Access Mode Reference

Your `@CheckAccess` decorator uses two modes:

<Tabs>
  <Tab title="OR (default)">
    User has the required permission **OR** has stakeholder-level access on the resource → allowed
  </Tab>
  <Tab title="AND">
    User must have the permission **AND** stakeholder access → allowed
  </Tab>
</Tabs>

**Resource action hierarchy:**

- `ACCESS` — granted by `READ`, `WRITE`, or `FULL` access level
- `MODIFY` — granted by `WRITE` or `FULL`
- `DELETE` — granted by `FULL` only

## Test Data Seeding

### Step 1: Seed Users (one-time)

<CodeGroup>

```bash terminal
pnpm test:seed-users
```

</CodeGroup>

This runs `test/setup/seed-test-users.ts` which:

<Check>Creates the RBAC test personas in the database (idempotent — skips if they exist)</Check>
<Check>Creates "Test Organization" with owner</Check>
<Check>Ensures **ACTIVE BUSINESS `subscription`** and **completed `organization_setup_progress`** on the test org</Check>
<Check>Creates "Test Sales Team" with team leader and agent</Check>
<Check>Assigns org roles (Owner, Admin, Basic) and team roles (Team Leader, Agent)</Check>
<Check>Uses RLS bypass to write directly via MikroORM</Check>

<Warning>
Requires subscription plans to exist. Run `pnpm test:init-data` once on a fresh DB before `pnpm test:seed-users`.
</Warning>

### Step 2: Seed Test Data (automatic)

Each test file calls `seedRbacTestData(app, personas, teamId)` in its `beforeAll`. Results are cached across files.

**This creates entities through the API:**

<CardGroup cols={2}>

<Card title="Leads" icon="bullseye-arrow">
- One by BASIC (auto-assigned as primary stakeholder with FULL access)
- One by OWNER (for admin-only tests)
- One in pool (empty stakeholders)
- One with team as stakeholder
- One with AGENT as READ-only stakeholder
</Card>

<Card title="Other Entities" icon="database">
- Contacts by BASIC and OWNER
- Companies by BASIC and OWNER
- Deals by OWNER
- Property listings for testing property interests
</Card>

</CardGroup>

<Note>
All seeded data uses the actual API endpoints, so guards, validators, and business logic execute naturally. This ensures tests reflect real-world behavior.
</Note>

## How to Run

### Run RBAC E2E Tests

<CodeGroup>

```bash core-tests
pnpm test:e2e
```

```bash with-queue
pnpm test:e2e-queue
```

```bash specific-file
pnpm test:e2e test/e2e/rbac/lead-rbac.e2e-spec.ts
```

</CodeGroup>

<Steps>

<Step title="Ensure database is ready">
Your `.env.test` should point to a test database
</Step>

<Step title="Seed users (first time only)">
```bash
pnpm test:seed-users
```
</Step>

<Step title="Run tests">
```bash
pnpm test:e2e
```
</Step>

</Steps>

### Run Auth Lifecycle Tests

<CodeGroup>

```bash auth-tests
pnpm test:e2e test/e2e/auth/
```

</CodeGroup>

### Cleanup Test Users

<CodeGroup>

```bash cleanup
pnpm test:cleanup-users
```

</CodeGroup>

<Warning>
This deletes all `@propwise-test.com` users from the database. Use with caution.
</Warning>

## How to Extend

### Adding a New Persona

<Steps>

<Step title="Define in rbac-personas.ts">
Add the new persona to the `RBAC_PERSONAS` object with email, role, and permissions
</Step>

<Step title="Update seed-test-users.ts">
Add logic to create the user, assign roles, and add to teams if needed
</Step>

<Step title="Update rbac-context.ts">
Add login logic for the new persona
</Step>

<Step title="Update matrices">
Add expectations for the new persona in relevant matrices
</Step>

</Steps>

### Adding a New Matrix

<Steps>

<Step title="Create matrix file">
Create a new file in `test/setup/matrices/` (e.g., `property-matrix.ts`)
</Step>

<Step title="Define expectations">
```typescript
export const PROPERTY_BY_BASIC_MATRIX: EndpointExpectation[] = [
  {
    method: 'GET',
    path: '/v1/properties/:propertyId',
    description: 'View property detail',
    expectations: {
      OWNER: 200,
      ADMIN: 200,
      TEAM_LEADER: 403,
      AGENT: 403,
      BASIC: 200,
    },
  },
];
```
</Step>

<Step title="Export from index">
Add to `test/setup/matrices/index.ts`
</Step>

<Step title="Create test file">
Create `test/e2e/rbac/property-rbac.e2e-spec.ts` and call `runRbacMatrix()`
</Step>

</Steps>

### Adding Test Data

<Steps>

<Step title="Update rbac-test-data.ts">
Add API calls to create the new entity type
</Step>

<Step title="Update interface">
Add the new entity IDs to the `RbacTestData` interface
</Step>

<Step title="Cache results">
The seeding function caches results, so new entities are created once per test run
</Step>

</Steps>

## Methodology

<Tip>
See `test/TESTING_METHODOLOGY.md` for detailed guidance on analyzing controllers and DTOs before writing tests.
</Tip>

**General approach:**

1. **Analyze the controller** — identify all endpoints, DTOs, guards, and decorators
2. **Map permissions** — determine which personas should have access to each endpoint
3. **Define matrix** — create declarative expectations for each persona × endpoint combination
4. **Seed data** — create test entities that cover all access scenarios (owner, team, stakeholder, pool)
5. **Run matrix** — let the runner generate parameterized tests automatically

**Access scenarios to cover:**

<CardGroup cols={2}>

<Card title="Org-level access" icon="building">
OWNER and ADMIN can access via `crm.*` or `sales.*` permissions
</Card>

<Card title="Team-level access" icon="users">
TEAM_LEADER can access via `team_crm.*` or `team_sales.*` permissions when team is a stakeholder
</Card>

<Card title="User-level access" icon="user">
AGENT and BASIC can access via personal stakeholder records
</Card>

<Card title="Access level enforcement" icon="shield-halved">
Test READ vs WRITE vs FULL access levels for modify and delete operations
</Card>

</CardGroup>

## CI/CD Integration

### GitHub Actions Workflow

<CodeGroup>

```yaml .github/workflows/test.yml
name: E2E Tests

on: [push, pull_request]

jobs:
  test:
    runs-on: ubuntu-latest
    
    services:
      postgres:
        image: postgres:15
        env:
          POSTGRES_DB: propwise_test
          POSTGRES_USER: postgres
          POSTGRES_PASSWORD: postgres
        options: >-
          --health-cmd pg_isready
          --health-interval 10s
          --health-timeout 5s
          --health-retries 5
    
    steps:
      - uses: actions/checkout@v3
      
      - name: Setup Node.js
        uses: actions/setup-node@v3
        with:
          node-version: '18'
          
      - name: Install dependencies
        run: pnpm install
        
      - name: Run migrations
        run: pnpm migration:run
        env:
          DATABASE_URL: postgresql://postgres:postgres@localhost:5432/propwise_test
          
      - name: Seed test data
        run: pnpm test:init-data && pnpm test:seed-users
        
      - name: Run E2E tests
        run: pnpm test:e2e
```

</CodeGroup>

<Info>
The test suite is designed to run in CI environments with minimal configuration. The singleton context pattern ensures fast execution even with multiple test files.
</Info>

### Test Execution Time

- **First test file:** ~3-5 seconds (app boot + authentication)
- **Subsequent files:** ~0ms context overhead (cached)
- **Total suite:** Depends on number of endpoints tested, typically < 30 seconds for full RBAC matrix
