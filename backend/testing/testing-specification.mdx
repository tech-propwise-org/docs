The test suite focuses on **RBAC (Role-Based Access Control) E2E testing** ‚Äî verifying that every API endpoint enforces the correct permissions for every user role.

**Stack:** Jest + supertest + `@nestjs/testing`

## Key Design Decisions

<CardGroup cols={2}>
  <Card title="Singleton App Context" icon="server">
    The NestJS app boots once and is shared across all test files (equivalent to NestJS `TestSuite` pattern, implemented manually since `TestSuite` requires NestJS 11+)
  </Card>
  <Card title="Matrix-Driven Tests" icon="table">
    A declarative permission matrix defines expected HTTP status codes per persona √ó endpoint. A runner generates `it()` blocks automatically
  </Card>
  <Card title="API-Driven Seeding" icon="seedling">
    Test data is created through the actual API endpoints so all business logic (guards, auto-assignment, stakeholder creation) executes naturally
  </Card>
</CardGroup>

## Architecture

### Singleton Context

The app boots once for the entire test run. All test files share the same NestJS app instance, authenticated tokens, and resolved `teamId`.

<Info>
Jest starts ‚Üí lead-rbac.e2e-spec.ts calls getContext() ‚Üí First call: boots NestJS app (~3-5s), logs in 5 personas, resolves teamId ‚Üí Returns { app, personas, teamId } ‚Üí deal-rbac.e2e-spec.ts calls getContext() ‚Üí Returns cached instance instantly (0ms) ‚Üí Last file's afterAll() calls closeContext() ‚Üí Reference count hits 0 ‚Üí app shuts down
</Info>

```typescript
import { getContext, closeContext, RbacTestContext } from '../../setup/rbac-context';

let ctx: RbacTestContext;
beforeAll(async () => {
  ctx = await getContext();
}, 60000);
afterAll(async () => {
  await closeContext();
});
```

The context replicates production config from `main.ts`:
- `ValidationPipe` (whitelist, transform, implicit conversion)
- URI versioning (`/v1/...`)
- Cookie parser

### Matrix Runner

Instead of writing individual `it()` blocks, you define expectations declaratively:

```typescript
{
  method: 'GET',
  path: '/v1/leads/:leadByBasicId',
  description: 'View lead detail',
  expectations: {
    OWNER: 200,
    ADMIN: 200,
    TEAM_LEADER: 403,
    AGENT: 403,
    BASIC: 200,
  },
}
```

<Steps>
  <Step title="Iterate endpoints">
    The runner calls `runRbacMatrix()` which iterates every endpoint in the matrix
  </Step>
  <Step title="Test personas">
    For each endpoint, iterates all 5 personas
  </Step>
  <Step title="Resolve placeholders">
    Resolves path placeholders (`:leadByBasicId` ‚Üí actual UUID)
  </Step>
  <Step title="Make requests">
    Makes the HTTP request with the persona's tenant token
  </Step>
  <Step title="Assert responses">
    Asserts the response status matches the expected value and produces clear failure messages with the actual response body
  </Step>
</Steps>

<Note>One matrix entry = 5 generated test cases</Note>

## RBAC Personas

Defined in `test/helpers/rbac-personas.ts`. All personas share a single source of truth.

| Key           | Email                   | Org Role | Team                           | Description                                                       |
| ------------- | ----------------------- | -------- | ------------------------------ | ----------------------------------------------------------------- |
| `OWNER`       | owner@propwise.com      | Owner    | Any                            | All permissions including `system.owner`                          |
| `ADMIN`       | admin@propwise.com      | Admin    | Any                            | All non-sensitive permissions (includes `system.admin`)           |
| `TEAM_LEADER` | teamleader@propwise.com | Basic    | Team Leader in Test Sales Team | All team permissions (`team_crm.*`, `team_sales.*`, `team.admin`) |
| `AGENT`       | agent@propwise.com      | Basic    | Agent in Test Sales Team       | No team permissions by default                                    |
| `BASIC`       | basic@propwise.com      | Basic    | None                           | No permissions ‚Äî relies entirely on stakeholder access            |

### Permission Profiles

<AccordionGroup>
  <Accordion title="OWNER / ADMIN">
    Bypass resource-level checks via org permissions (`crm.view`, `crm.manage`, `sales.view`, `sales.manage`, etc.). Can access any entity in the org.
  </Accordion>
  
  <Accordion title="TEAM_LEADER">
    No org permissions, but has `team_crm.manage` and `team_sales.manage` in the team. Can access entities where the team is a stakeholder.
  </Accordion>
  
  <Accordion title="AGENT">
    No org permissions, no team permissions. Can only access entities where they are personally a stakeholder. Used to test access level differentiation (READ vs WRITE vs FULL).
  </Accordion>
  
  <Accordion title="BASIC">
    Same as AGENT but not a team member at all. Access is purely stakeholder-based.
  </Accordion>
</AccordionGroup>

### Login Flow

The context performs a two-step login for each persona:

<Steps>
  <Step title="Identity token">
    `POST /v1/auth/login` ‚Üí identity token
  </Step>
  <Step title="Tenant token">
    `POST /v1/auth/select-organization` ‚Üí tenant token (org-scoped JWT)
  </Step>
</Steps>

<Note>All subsequent API calls use the tenant token in the `Authorization: Bearer` header</Note>

### Access Mode Reference

Your `@CheckAccess` decorator uses two modes:

- **OR (default):** User has the required permission **OR** has stakeholder-level access on the resource ‚Üí allowed
- **AND:** User must have the permission **AND** stakeholder access ‚Üí allowed

<Warning>
Resource action hierarchy:
- `ACCESS` ‚Äî granted by `READ`, `WRITE`, or `FULL` access level
- `MODIFY` ‚Äî granted by `WRITE` or `FULL`
- `DELETE` ‚Äî granted by `FULL` only
</Warning>

## Test Data Seeding

### Step 1: Seed Users (One-time Setup)

<CodeGroup>
```bash npm
npm run test:seed-users
```

```bash pnpm
pnpm test:seed-users
```

```bash yarn
yarn test:seed-users
```
</CodeGroup>

This runs `test/setup/seed-test-users.ts` which:

<Check>Creates the RBAC test personas in the database (idempotent ‚Äî skips if they exist)</Check>
<Check>Creates "Test Organization" with owner</Check>
<Check>Ensures **ACTIVE BUSINESS `subscription`** and **completed `organization_setup_progress`** on the test org</Check>
<Check>Creates "Test Sales Team" with team leader and agent</Check>
<Check>Assigns org roles (Owner, Admin, Basic) and team roles (Team Leader, Agent)</Check>
<Check>Uses RLS bypass to write directly via MikroORM</Check>

<Note>Requires subscription plans to exist (run `pnpm test:init-data` once on a fresh DB before `pnpm test:seed-users`)</Note>

### Step 2: Seed Test Data (Automatic)

Each test file calls `seedRbacTestData(app, personas, teamId)` in its `beforeAll`. Results are cached across files. This creates entities through the API:

**Leads:**
- One by BASIC (auto-assigned as primary stakeholder with FULL access)
- One by OWNER (for admin-only tests)
- One in pool (empty stakeholders)
- One with team as stakeholder (OWNER creates, assigns team ‚Äî TEAM_LEADER gets access)
- One with AGENT as READ-only stakeholder (OWNER creates, adds AGENT with `accessLevel: 'READ'`)

**Deals:**
- One by OWNER (no stakeholders ‚Äî admin-only access via `sales.*`)

**Contacts & Companies:**
- One by BASIC (creator access)
- One by OWNER (admin-only access)

<Tip>All seeding goes through the actual API endpoints, so guards, auto-assignment logic, and stakeholder creation happen naturally</Tip>

## How to Run

<Steps>
  <Step title="Initialize data (first time only)">
    ```bash
    pnpm test:init-data
    ```
  </Step>
  
  <Step title="Seed test users (first time only)">
    ```bash
    pnpm test:seed-users
    ```
  </Step>
  
  <Step title="Run core E2E tests">
    ```bash
    pnpm test:e2e
    ```
    Uses `jest-e2e.json` config. Excludes distribution + escalation queue folders' tests.
  </Step>
  
  <Step title="Run queue-specific tests (optional)">
    ```bash
    pnpm test:e2e:queue
    ```
    Uses `jest-e2e-queue.json` config for live pg-boss queue testing.
  </Step>
</Steps>

<Warning>
The test suite expects a clean database state. If tests fail due to stale data, clean up with:
```bash
pnpm test:cleanup-users
```
Then re-run the seed step.
</Warning>

## How to Extend

### Adding New Endpoints to Existing Matrices

1. **Analyze the controller** ‚Äî identify the `@CheckAccess` decorator parameters and required permissions
2. **Add to the appropriate matrix file** ‚Äî e.g., `test/setup/matrices/lead-matrix.ts`
3. **Define expected status codes** for each persona based on their permission profile

```typescript
{
  method: 'POST',
  path: '/v1/leads/:leadByBasicId/activities',
  description: 'Create activity on lead',
  expectations: {
    OWNER: 201,        // Has crm.manage
    ADMIN: 201,        // Has crm.manage  
    TEAM_LEADER: 403,  // No org perms, no stakeholder access
    AGENT: 403,        // No org perms, no stakeholder access
    BASIC: 201,        // FULL stakeholder access allows MODIFY
  },
}
```

### Adding New Entity Matrices

<Steps>
  <Step title="Create matrix file">
    Create `test/setup/matrices/new-entity-matrix.ts`
  </Step>
  
  <Step title="Define test entities">
    Add entity creation logic to `test/setup/rbac-test-data.ts`
  </Step>
  
  <Step title="Export from barrel">
    Add to `test/setup/matrices/index.ts` and `test/setup/rbac-matrix.ts`
  </Step>
  
  <Step title="Create test spec">
    Create `test/e2e/rbac/new-entity-rbac.e2e-spec.ts`
  </Step>
</Steps>

### Adding New Personas

1. **Update `rbac-personas.ts`** with the new persona definition
2. **Update `seed-test-users.ts`** to create the persona in the database
3. **Update all matrices** to include expected status codes for the new persona
4. **Update `rbac-context.ts`** to authenticate the new persona

## Methodology

The test suite follows a systematic approach:

<CardGroup cols={2}>
  <Card title="Controller Analysis" icon="magnifying-glass">
    Analyze `@CheckAccess` decorators, required permissions, and resource access patterns before writing tests
  </Card>
  <Card title="Matrix-First Design" icon="grid">
    Define permission expectations declaratively in matrices rather than writing individual test cases
  </Card>
  <Card title="API-Driven Seeding" icon="database">
    Create test data through actual API endpoints to ensure business logic executes naturally
  </Card>
  <Card title="Stakeholder Testing" icon="users">
    Test both permission-based access and stakeholder-based access patterns
  </Card>
</CardGroup>
