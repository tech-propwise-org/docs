Architecture, conventions, and run guide for the PropWise CRM backend test suite.

## Overview

The test suite focuses on **RBAC (Role-Based Access Control) E2E testing** — verifying that every API endpoint enforces the correct permissions for every user role.

**Stack:** Jest + supertest + `@nestjs/testing`

<Note>
**Key design decisions:**
- **Singleton app context** — the NestJS app boots once and is shared across all test files
- **Matrix-driven tests** — a declarative permission matrix defines expected HTTP status codes per persona × endpoint
- **API-driven seeding** — test data is created through actual API endpoints so all business logic executes naturally
</Note>

## Architecture

### Singleton Context

The app boots once for the entire test run. All test files share the same NestJS app instance, authenticated tokens, and resolved `teamId`.

<Steps>
<Step title="Jest starts">
First test file calls `getContext()` which boots NestJS app (~3-5s), logs in 5 personas, resolves teamId
</Step>
<Step title="Subsequent files">
Return cached instance instantly (0ms)
</Step>
<Step title="Cleanup">
Last file's `afterAll()` calls `closeContext()` when reference count hits 0
</Step>
</Steps>

**Usage pattern:**

```typescript
import { getContext, closeContext, RbacTestContext } from '../../setup/rbac-context';

let ctx: RbacTestContext;
beforeAll(async () => {
  ctx = await getContext();
}, 60000);
afterAll(async () => {
  await closeContext();
});
```

The context replicates production config from `main.ts`:

- `ValidationPipe` (whitelist, transform, implicit conversion)
- URI versioning (`/v1/...`)
- Cookie parser

### Matrix Runner

Instead of writing individual `it()` blocks, you define expectations declaratively:

```typescript
{
  method: 'GET',
  path: '/v1/leads/:leadByBasicId',
  description: 'View lead detail',
  expectations: {
    OWNER: 200,
    ADMIN: 200,
    TEAM_LEADER: 403,
    AGENT: 403,
    BASIC: 200,
  },
}
```

<Info>
The runner calls `runRbacMatrix()` which:
1. Iterates every endpoint in the matrix
2. For each endpoint, iterates all 5 personas
3. Resolves path placeholders (`:leadByBasicId` → actual UUID)
4. Makes HTTP request with persona's tenant token
5. Asserts response status matches expected value

One matrix entry = 5 generated test cases.
</Info>

## RBAC Personas

Defined in `test/helpers/rbac-personas.ts`. All personas share a single source of truth.

| Key           | Email                   | Org Role | Team                           | Description                                                       |
| ------------- | ----------------------- | -------- | ------------------------------ | ----------------------------------------------------------------- |
| `OWNER`       | owner@propwise.com      | Owner    | Any                            | All permissions including `system.owner`                          |
| `ADMIN`       | admin@propwise.com      | Admin    | Any                            | All non-sensitive permissions (includes `system.admin`)           |
| `TEAM_LEADER` | teamleader@propwise.com | Basic    | Team Leader in Test Sales Team | All team permissions (`team_crm.*`, `team_sales.*`, `team.admin`) |
| `AGENT`       | agent@propwise.com      | Basic    | Agent in Test Sales Team       | No team permissions by default                                    |
| `BASIC`       | basic@propwise.com      | Basic    | None                           | No permissions — relies entirely on stakeholder access            |

### Permission Profiles

<Tabs>
<Tab title="OWNER / ADMIN">
Bypass resource-level checks via org permissions (`crm.view`, `crm.manage`, `sales.view`, `sales.manage`, etc.). Can access any entity in the org.
</Tab>
<Tab title="TEAM_LEADER">
No org permissions, but has `team_crm.manage` and `team_sales.manage` in the team. Can access entities where the team is a stakeholder.
</Tab>
<Tab title="AGENT">
No org permissions, no team permissions. Can only access entities where they are personally a stakeholder. Used to test access level differentiation (READ vs WRITE vs FULL).
</Tab>
<Tab title="BASIC">
Same as AGENT but not a team member at all. Access is purely stakeholder-based.
</Tab>
</Tabs>

### Login Flow

The context performs a two-step login for each persona:

<Steps>
<Step title="Identity login">
`POST /v1/auth/login` → identity token
</Step>
<Step title="Organization selection">
`POST /v1/auth/select-organization` → tenant token (org-scoped JWT)
</Step>
</Steps>

<Note>
All subsequent API calls use the tenant token in the `Authorization: Bearer` header.
</Note>

## Permission Matrix

Matrices are split into per-entity files under `test/setup/matrices/`. The barrel at `test/setup/rbac-matrix.ts` re-exports everything.

### Lead Matrices

<AccordionGroup>
<Accordion title="LEAD_BY_BASIC_MATRIX">
**Entity:** Lead created by BASIC  
**Who Can Access:** OWNER, ADMIN (via `crm.*`), BASIC (via FULL stakeholder)  
**Tests:** View, update, stakeholders, activities, property interests
</Accordion>

<Accordion title="LEAD_ON_TEAM_MATRIX">
**Entity:** Lead with team as stakeholder  
**Who Can Access:** OWNER, ADMIN, TEAM_LEADER (via `team_crm.*`)  
**Tests:** View, update, stakeholders, activities
</Accordion>

<Accordion title="LEAD_AGENT_READ_MATRIX">
**Entity:** Lead where AGENT has READ access  
**Who Can Access:** OWNER, ADMIN, AGENT (view only — modify denied)  
**Tests:** View allowed, update denied, access level enforcement
</Accordion>

<Accordion title="LEAD_IN_POOL_MATRIX">
**Entity:** Lead with no stakeholders  
**Who Can Access:** OWNER, ADMIN only (via `crm.*`)  
**Tests:** View, update
</Accordion>

<Accordion title="LEAD_ADMIN_ONLY_MATRIX">
**Entity:** Lead for admin-only ops  
**Who Can Access:** OWNER, ADMIN (via `system.admin`)  
**Tests:** Delete
</Accordion>
</AccordionGroup>

### Access Mode Reference

Your `@CheckAccess` decorator uses two modes:

<Tabs>
<Tab title="OR (default)">
User has the required permission **OR** has stakeholder-level access on the resource → allowed.
</Tab>
<Tab title="AND">
User must have the permission **AND** stakeholder access → allowed.
</Tab>
</Tabs>

**Resource action hierarchy:**
- `ACCESS` — granted by `READ`, `WRITE`, or `FULL` access level
- `MODIFY` — granted by `WRITE` or `FULL`
- `DELETE` — granted by `FULL` only

## Test Data Seeding

### Step 1: Seed Users (one-time setup)

<Warning>
Run this once before executing tests:
</Warning>

```bash
pnpm test:seed-users
```

This runs `test/setup/seed-test-users.ts` which:

<Steps>
<Step title="Create personas">
Creates the RBAC test personas in the database (idempotent — skips if they exist)
</Step>
<Step title="Setup organization">
Creates "Test Organization" with owner
</Step>
<Step title="Configure subscription">
Ensures **ACTIVE BUSINESS subscription** and **completed organization_setup_progress** on the test org
</Step>
<Step title="Setup team">
Creates "Test Sales Team" with team leader and agent
</Step>
<Step title="Assign roles">
Assigns org roles (Owner, Admin, Basic) and team roles (Team Leader, Agent)
</Step>
</Steps>

<Note>
Requires subscription plans to exist (run `pnpm test:init-data` once on a fresh DB before `pnpm test:seed-users`).
</Note>

### Step 2: Seed Test Data (automatic)

Each test file calls `seedRbacTestData(app, personas, teamId)` in its `beforeAll`. Results are cached across files. This creates entities through the API:

**Leads:**
- One by BASIC (auto-assigned as primary stakeholder with FULL access)
- One by OWNER (for admin-only tests)
- One in pool (empty stakeholders)
- One with team as stakeholder (OWNER creates, assigns team — TEAM_LEADER gets access)
- One with AGENT as READ-only stakeholder (OWNER creates, adds AGENT with access level)

## How to Run

<CodeGroup>
```bash Core RBAC E2E Tests
# Run all RBAC tests (excludes queue tests)
pnpm test:e2e

# Run specific entity tests
pnpm test:e2e --testNamePattern="Lead RBAC"
pnpm test:e2e --testNamePattern="Deal RBAC"
```

```bash Queue Tests
# Run queue-dependent tests (requires live pg-boss)
pnpm test:e2e-queue
```

```bash Full Test Suite
# Run everything
pnpm test:e2e-all
```
</CodeGroup>

<Tip>
Use `--testNamePattern` to run specific test suites during development.
</Tip>

## How to Extend

### Adding New Endpoints

<Steps>
<Step title="Define matrix entry">
Add endpoint to appropriate matrix file in `test/setup/matrices/`:

```typescript
{
  method: 'POST',
  path: '/v1/leads/:leadId/notes',
  description: 'Add note to lead',
  expectations: {
    OWNER: 201,
    ADMIN: 201,
    TEAM_LEADER: 403,
    AGENT: 403,
    BASIC: 201,
  },
}
```
</Step>

<Step title="Handle path resolution">
If using new placeholders, update `rbac-runner.ts` resolver logic
</Step>

<Step title="Run tests">
The matrix runner automatically generates test cases
</Step>
</Steps>

### Adding New Personas

<Steps>
<Step title="Update persona definitions">
Add to `test/helpers/rbac-personas.ts`
</Step>

<Step title="Update seeding">
Modify `test/setup/seed-test-users.ts` to create the new user
</Step>

<Step title="Update context">
Add authentication logic to `test/setup/rbac-context.ts`
</Step>

<Step title="Update matrices">
Add expectations for the new persona in all relevant matrices
</Step>
</Steps>

### Adding New Entity Tests

<Steps>
<Step title="Create matrix file">
Add new matrix file under `test/setup/matrices/`
</Step>

<Step title="Export matrix">
Add to `test/setup/matrices/index.ts` barrel
</Step>

<Step title="Create test file">
Add new test file under `test/e2e/rbac/` following existing patterns
</Step>

<Step title="Update seeding">
Add fixture data creation to `test/setup/rbac-test-data.ts`
</Step>
</Steps>

## CI/CD Integration

The test suite integrates with CI/CD pipelines through standard Jest commands:

```bash
# In CI environment
npm run test:seed-users  # One-time setup
npm run test:e2e         # Core tests
npm run test:e2e-queue   # Queue tests (if pg-boss available)
```

<Warning>
Ensure test database is properly seeded with subscription plans before running persona seeding.
</Warning>

<Info>
The singleton app context pattern significantly reduces CI test execution time by avoiding repeated NestJS bootstrapping.
</Info>
