The test suite focuses on **RBAC (Role-Based Access Control) E2E testing** — verifying that every API endpoint enforces the correct permissions for every user role.

**Stack:** Jest + supertest + `@nestjs/testing`

## Key Design Decisions

<CardGroup cols={2}>
  <Card title="Singleton App Context" icon="server">
    The NestJS app boots once and is shared across all test files for optimal performance
  </Card>
  <Card title="Matrix-Driven Tests" icon="table">
    A declarative permission matrix defines expected HTTP status codes per persona × endpoint
  </Card>
  <Card title="API-Driven Seeding" icon="seedling">
    Test data is created through actual API endpoints so all business logic executes naturally
  </Card>
  <Card title="Auto-Generated Tests" icon="wand-magic-sparkles">

## Architecture

### Singleton Context (`rbac-context.ts`)

<Info>
The app boots once for the entire test run. All test files share the same NestJS app instance, authenticated tokens, and resolved `teamId`.
</Info>

```typescript
Jest starts
  → lead-rbac.e2e-spec.ts calls getContext()
    → First call: boots NestJS app (~3-5s), logs in 5 personas, resolves teamId
    → Returns { app, personas, teamId }
  → deal-rbac.e2e-spec.ts calls getContext()
    → Returns cached instance instantly (0ms)
  → ... same for contact and company files
  → Last file's afterAll() calls closeContext()
    → Reference count hits 0 → app shuts down
```

**How it works:**

```typescript
import { getContext, closeContext, RbacTestContext } from '../../setup/rbac-context';

let ctx: RbacTestContext;
beforeAll(async () => { ctx = await getContext(); }, 60000);
afterAll(async () => { await closeContext(); });
```

The context replicates production config from `main.ts`:
- `ValidationPipe` (whitelist, transform, implicit conversion)
- URI versioning (`/v1/...`)
- Cookie parser

### Matrix Runner (`rbac-runner.ts`)

Instead of writing individual `it()` blocks, you define expectations declaratively:

```typescript
{
  method: 'GET',
  path: '/v1/leads/:leadByBasicId',
  description: 'View lead detail',
  expectations: {
    OWNER: 200,
    ADMIN: 200,
    TEAM_LEADER: 403,
    AGENT: 403,
    BASIC: 200,
  },
}
```

<Tip>
One matrix entry = 5 generated test cases. The runner automatically creates tests for each persona.
</Tip>

The runner calls `runRbacMatrix()` which:
1. Iterates every endpoint in the matrix
2. For each endpoint, iterates all 5 personas
3. Resolves path placeholders (`:leadByBasicId` → actual UUID)
4. Makes the HTTP request with the persona's tenant token
5. Asserts the response status matches the expected value
6. Produces clear failure messages with the actual response body

## RBAC Personas

<Note>
All personas are defined in `test/helpers/rbac-personas.ts` and share a single source of truth.
</Note>

| Key | Email | Org Role | Team | Description |
|-----|-------|----------|------|-------------|
| `OWNER` | owner@propwise-test.com | Owner | Any | All permissions including `system.owner` |
| `ADMIN` | admin@propwise-test.com | Admin | Any | All non-sensitive permissions (includes `system.admin`) |
| `TEAM_LEADER` | teamleader@propwise-test.com | Basic | Team Leader in Test Sales Team | All team permissions (`team_crm.*`, `team_sales.*`, `team.admin`) |
| `AGENT` | agent@propwise-test.com | Basic | Agent in Test Sales Team | No team permissions by default |
| `BASIC` | basic@propwise-test.com | Basic | None | No permissions — relies entirely on stakeholder access |

### Permission Profiles

<Tabs>
  <Tab title="OWNER / ADMIN">
    Bypass resource-level checks via org permissions (`crm.view`, `crm.manage`, `sales.view`, `sales.manage`, etc.). Can access any entity in the org.
  </Tab>
  <Tab title="TEAM_LEADER">
    No org permissions, but has `team_crm.manage` and `team_sales.manage` in the team. Can access entities where the team is a stakeholder.
  </Tab>
  <Tab title="AGENT">
    No org permissions, no team permissions. Can only access entities where they are personally a stakeholder. Used to test access level differentiation (READ vs WRITE vs FULL).
  </Tab>
  <Tab title="BASIC">
    Same as AGENT but not a team member at all. Access is purely stakeholder-based.
  </Tab>
</Tabs>

### Login Flow

The context performs a two-step login for each persona:

<Steps>
  <Step title="Identity Authentication">
    `POST /v1/auth/login` → identity token
  </Step>
  <Step title="Organization Selection">
    `POST /v1/auth/select-organization` → tenant token (org-scoped JWT)
  </Step>
</Steps>

All subsequent API calls use the tenant token in the `Authorization: Bearer` header.

### Access Mode Reference

<Warning>
Your `@CheckAccess` decorator uses two modes that affect permission evaluation.
</Warning>

**Access Modes:**
- **OR (default):** User has the required permission **OR** has stakeholder-level access on the resource → allowed.
- **AND:** User must have the permission **AND** stakeholder access → allowed.

**Resource Action Hierarchy:**
- `ACCESS` — granted by `READ`, `WRITE`, or `FULL` access level
- `MODIFY` — granted by `WRITE` or `FULL`
- `DELETE` — granted by `FULL` only

## Test Data Seeding

### Step 1: Seed Users (one-time, before running tests)

<CodeGroup>

```bash Terminal
pnpm test:seed-users
```

</CodeGroup>

This runs `test/setup/seed-test-users.ts` which:

<Check>Idempotent operation — skips if users already exist</Check>

- Creates the 5 test users in the database
- Creates "Test Organization" with owner
- Creates "Test Sales Team" with team leader and agent
- Assigns org roles (Owner, Admin, Basic) and team roles (Team Leader, Agent)
- Uses RLS bypass to write directly via MikroORM

### Step 2: Seed Test Data (automatic, per test run)

Each test file calls `seedRbacTestData(app, personas, teamId)` in its `beforeAll`. Results are cached across files.

<Info>
Because seeding goes through the real API, all guards, auto-assignment rules, and stakeholder creation run. The test data matches what a real user would produce.
</Info>

This creates entities through the API:

<AccordionGroup>
  <Accordion title="Leads">
    - One by BASIC (auto-assigned as primary stakeholder with FULL access)
    - One by OWNER (for admin-only tests)
    - One in pool (empty stakeholders)
    - One with team as stakeholder (OWNER creates, assigns team — TEAM_LEADER gets access)
    - One with AGENT as READ-only stakeholder (OWNER creates, adds AGENT with `accessLevel: read`)
  </Accordion>
  <Accordion title="Contacts">
    - One by BASIC
    - One by OWNER
  </Accordion>
  <Accordion title="Companies">
    - One by BASIC
    - One by OWNER
  </Accordion>
  <Accordion title="Deals">
    - One by OWNER (from lead + contact)
  </Accordion>
</AccordionGroup>

## How to Extend

### Adding New Endpoints

<Steps>
  <Step title="Analyze Controller">
    Review the controller and DTOs using `TESTING_METHODOLOGY.md` to understand required permissions and stakeholder access patterns.
  </Step>
  <Step title="Update Matrix">
    Add the new endpoint to the appropriate matrix file in `test/setup/matrices/` with expected status codes for each persona.
  </Step>
  <Step title="Test Data Requirements">
    If the endpoint needs specific test data, update `rbac-test-data.ts` to create the necessary entities.
  </Step>
  <Step title="Path Placeholders">
    Update the path resolver in `rbac-runner.ts` if you introduce new placeholder patterns.
  </Step>
</Steps>

### Adding New Entities

<Steps>
  <Step title="Create Matrix File">
    Create a new matrix file in `test/setup/matrices/` following the naming pattern `{entity}-matrix.ts`.
  </Step>
  <Step title="Update Barrel Exports">
    Add the new matrix to `test/setup/matrices/index.ts` and `test/setup/rbac-matrix.ts`.
  </Step>
  <Step title="Create Test File">
    Create a new test file in `test/e2e/rbac/` that imports the matrix and calls `runRbacMatrix()`.
  </Step>
  <Step title="Update Test Data Seeding">
    Add entity creation logic to `rbac-test-data.ts` if the entity requires test data.
  </Step>
</Steps>

## Methodology

<Tip>
Refer to `test/TESTING_METHODOLOGY.md` for detailed guidance on analyzing controllers and DTOs before writing tests.
</Tip>

The testing methodology follows these principles:

1. **Permission-First Design** — Every endpoint is evaluated through the lens of "who should be able to access this?"
2. **Stakeholder-Aware Testing** — Tests verify both org-level permissions and resource-level stakeholder access
3. **Access Level Differentiation** — Tests ensure READ/WRITE/FULL access levels are properly enforced
4. **Real-World Scenarios** — Test data is created through actual API calls to match production behavior

## CI/CD Integration

<Warning>
Ensure the test database is properly isolated from production data in CI/CD pipelines.
</Warning>

The test suite is designed to run in CI/CD environments with these considerations:

- **Database Isolation** — Uses separate test database with `@propwise-test.com` email domain
- **Parallel Safety** — Singleton context prevents conflicts when running multiple test files
- **Cleanup Strategy** — Test users can be cleaned up with `pnpm test:cleanup-users`
- **Performance Optimization** — App boots once per test run, not per test file

For CI/CD integration, ensure:
1. Test database is available and migrations are applied
2. Required environment variables are set
3. Test user seeding runs before the test suite
4. Cleanup runs after tests complete (optional, for database hygiene)
