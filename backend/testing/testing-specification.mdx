## Overview

The test suite focuses on **RBAC (Role-Based Access Control) E2E testing** ‚Äî verifying that every API endpoint enforces the correct permissions for every user role.

<Info>
**Stack:** Jest + supertest + `@nestjs/testing`
</Info>

### Key Design Decisions

- **Singleton app context** ‚Äî the NestJS app boots once and is shared across all test files (equivalent to NestJS `TestSuite` pattern, implemented manually since `TestSuite` requires NestJS 11+)
- **Matrix-driven tests** ‚Äî a declarative permission matrix defines expected HTTP status codes per persona √ó endpoint. A runner generates `it()` blocks automatically
- **API-driven seeding** ‚Äî test data is created through the actual API endpoints so all business logic (guards, auto-assignment, stakeholder creation) executes naturally

## Architecture

### Singleton Context

The app boots once for the entire test run. All test files share the same NestJS app instance, authenticated tokens, and resolved `teamId`.

<Tabs>
  <Tab title="Test Flow">
```
Jest starts
  ‚Üí lead-rbac.e2e-spec.ts calls getContext()
    ‚Üí First call: boots NestJS app (~3-5s), logs in 5 personas, resolves teamId
    ‚Üí Returns { app, personas, teamId }
  ‚Üí deal-rbac.e2e-spec.ts calls getContext()
    ‚Üí Returns cached instance instantly (0ms)
  ‚Üí ... same for contact and company files
  ‚Üí Last file's afterAll() calls closeContext()
    ‚Üí Reference count hits 0 ‚Üí app shuts down
```
  </Tab>
  <Tab title="Implementation">
```typescript
import { getContext, closeContext, RbacTestContext } from '../../setup/rbac-context';

let ctx: RbacTestContext;
beforeAll(async () => { ctx = await getContext(); }, 60000);
afterAll(async () => { await closeContext(); });
```
  </Tab>
</Tabs>

<Note>
The context replicates production config from `main.ts`: ValidationPipe (whitelist, transform, implicit conversion), URI versioning (`/v1/...`), and Cookie parser.
</Note>

### Matrix Runner

Instead of writing individual `it()` blocks, you define expectations declaratively:

```typescript
{
  method: 'GET',
  path: '/v1/leads/:leadByBasicId',
  description: 'View lead detail',
  expectations: {
    OWNER: 200,
    ADMIN: 200,
    TEAM_LEADER: 403,
    AGENT: 403,
    BASIC: 200,
  },
}
```

The runner calls `runRbacMatrix()` which:

<Steps>
  <Step title="Iterate endpoints">
    For each endpoint in the matrix
  </Step>
  <Step title="Test all personas">
    For each endpoint, iterates all 5 personas
  </Step>
  <Step title="Resolve placeholders">
    Resolves path placeholders (`:leadByBasicId` ‚Üí actual UUID)
  </Step>
  <Step title="Make HTTP request">
    Makes the HTTP request with the persona's tenant token
  </Step>
  <Step title="Assert response">
    Asserts the response status matches the expected value and produces clear failure messages
  </Step>
</Steps>

<Tip>
One matrix entry = 5 generated test cases.
</Tip>

## RBAC Personas

<Info>
All personas are defined in `test/helpers/rbac-personas.ts` and share a single source of truth.
</Info>

| Key | Email | Org Role | Team | Description |
|-----|-------|----------|------|-------------|
| `OWNER` | owner@propwise.com | Owner | Any | All permissions including `system.owner` |
| `ADMIN` | admin@propwise.com | Admin | Any | All non-sensitive permissions (includes `system.admin`) |
| `TEAM_LEADER` | teamleader@propwise.com | Basic | Team Leader in Test Sales Team | All team permissions (`team_crm.*`, `team_sales.*`, `team.admin`) |
| `AGENT` | agent@propwise.com | Basic | Agent in Test Sales Team | No team permissions by default |
| `BASIC` | basic@propwise.com | Basic | None | No permissions ‚Äî relies entirely on stakeholder access |

### Permission Profiles

<AccordionGroup>
  <Accordion title="OWNER / ADMIN">
    Bypass resource-level checks via org permissions (`crm.view`, `crm.manage`, `sales.view`, `sales.manage`, etc.). Can access any entity in the org.
  </Accordion>
  <Accordion title="TEAM_LEADER">
    No org permissions, but has `team_crm.manage` and `team_sales.manage` in the team. Can access entities where the team is a stakeholder.
  </Accordion>
  <Accordion title="AGENT">
    No org permissions, no team permissions. Can only access entities where they are personally a stakeholder. Used to test access level differentiation (READ vs WRITE vs FULL).
  </Accordion>
  <Accordion title="BASIC">
    Same as AGENT but not a team member at all. Access is purely stakeholder-based.
  </Accordion>
</AccordionGroup>

### Login Flow

The context performs a two-step login for each persona:

<Steps>
  <Step title="Identity token">
    `POST /v1/auth/login` ‚Üí identity token
  </Step>
  <Step title="Tenant token">
    `POST /v1/auth/select-organization` ‚Üí tenant token (org-scoped JWT)
  </Step>
</Steps>

<Note>
All subsequent API calls use the tenant token in the `Authorization: Bearer` header.
</Note>

## Permission Matrix

Matrices are split into per-entity files under `test/setup/matrices/`. The barrel at `test/setup/rbac-matrix.ts` re-exports everything.

### Lead Matrices

| Matrix | Entity | Who Can Access | Tests |
|--------|--------|----------------|-------|
| `LEAD_BY_BASIC_MATRIX` | Lead created by BASIC | OWNER, ADMIN (via `crm.*`), BASIC (via FULL stakeholder) | View, update, stakeholders, activities, property interests |
| `LEAD_ON_TEAM_MATRIX` | Lead with team as stakeholder | OWNER, ADMIN, TEAM_LEADER (via `team_crm.*`) | View, update, stakeholders, activities |
| `LEAD_AGENT_READ_MATRIX` | Lead where AGENT has READ access | OWNER, ADMIN, AGENT (view only ‚Äî modify denied) | View allowed, update denied, access level enforcement |
| `LEAD_IN_POOL_MATRIX` | Lead with no stakeholders | OWNER, ADMIN only (via `crm.*`) | View, update |
| `LEAD_ADMIN_ONLY_MATRIX` | Lead for admin-only ops | OWNER, ADMIN (via `system.admin`) | Delete |

### Deal Matrices

| Matrix | Entity | Who Can Access |
|--------|--------|----------------|
| `DEAL_BY_OWNER_MATRIX` | Deal created by OWNER | OWNER, ADMIN only (via `sales.*`) |

### Contact Matrices

| Matrix | Entity | Who Can Access |
|--------|--------|----------------|
| `CONTACT_BY_BASIC_MATRIX` | Contact created by BASIC | OWNER, ADMIN, BASIC |
| `CONTACT_BY_OWNER_MATRIX` | Contact created by OWNER | OWNER, ADMIN only |

### Company Matrices

| Matrix | Entity | Who Can Access |
|--------|--------|----------------|
| `COMPANY_BY_BASIC_MATRIX` | Company created by BASIC | OWNER, ADMIN, BASIC |
| `COMPANY_BY_OWNER_MATRIX` | Company created by OWNER | OWNER, ADMIN only |

### Access Mode Reference

Your `@CheckAccess` decorator uses two modes:

- **OR (default):** User has the required permission **OR** has stakeholder-level access on the resource ‚Üí allowed
- **AND:** User must have the permission **AND** stakeholder access ‚Üí allowed

<Note>
Resource action hierarchy:
- `ACCESS` ‚Äî granted by `READ`, `WRITE`, or `FULL` access level
- `MODIFY` ‚Äî granted by `WRITE` or `FULL`
- `DELETE` ‚Äî granted by `FULL` only
</Note>

## Test Data Seeding

### Step 1: Seed Users (one-time setup)

<CodeGroup>
```bash Terminal
pnpm test:seed-users
```
</CodeGroup>

This runs `test/setup/seed-test-users.ts` which:

- Creates the 5 test users in the database (idempotent ‚Äî skips if they exist)
- Creates "Test Organization" with owner
- Creates "Test Sales Team" with team leader and agent
- Assigns org roles (Owner, Admin, Basic) and team roles (Team Leader, Agent)
- Uses RLS bypass to write directly via MikroORM

### Step 2: Seed Test Data (automatic)

Each test file calls `seedRbacTestData(app, personas, teamId)` in its `beforeAll`. Results are cached across files. This creates entities through the API:

<Tabs>
  <Tab title="Leads">
    - One by BASIC (auto-assigned as primary stakeholder with FULL access)
    - One by OWNER (for admin-only tests)
    - One in pool (empty stakeholders)
    - One with team as stakeholder (OWNER creates, assigns team ‚Äî TEAM_LEADER gets access)
    - One with AGENT as READ-only stakeholder (OWNER creates, adds AGENT with `accessLevel: read`)
  </Tab>
  <Tab title="Other Entities">
    - **Contacts:** one by BASIC, one by OWNER
    - **Companies:** one by BASIC, one by OWNER
    - **Deals:** one by OWNER (from lead + contact)
  </Tab>
</Tabs>

<Check>
Because seeding goes through the real API, all guards, auto-assignment rules, and stakeholder creation run. The test data matches what a real user would produce.
</Check>

## How to Run

### Prerequisites

<Steps>
  <Step title="Database setup">
    Database must be running with migrations applied
  </Step>
  <Step title="Seed test users">
    Run `pnpm test:seed-users`
  </Step>
  <Step title="Environment configuration">

## How to Extend

When adding new endpoints or entities, follow these steps:

<Steps>
  <Step title="Create matrix file">
    Create a new matrix file in `test/setup/matrices/` (e.g., `property-matrix.ts`)
  </Step>
  <Step title="Define expectations">
    Define endpoint expectations for each persona using the matrix format
  </Step>
  <Step title="Add to barrel export">
    Export your matrix from `test/setup/matrices/index.ts`
  </Step>
  <Step title="Create test file">
    Create corresponding E2E test file in `test/e2e/rbac/`
  </Step>
  <Step title="Extend test data">
    Add entity creation to `rbac-test-data.ts` if needed
  </Step>
</Steps>

## Methodology

<Info>
See `test/TESTING_METHODOLOGY.md` for detailed guidance on analyzing controllers and DTOs before writing tests.
</Info>

The testing methodology follows these principles:

1. **Permission-first design** ‚Äî Every endpoint must explicitly define its expected behavior for each persona
2. **Real API usage** ‚Äî Test data creation goes through actual API endpoints to ensure business logic executes
3. **Stakeholder-driven access** ‚Äî Test scenarios cover org permissions, team permissions, and individual stakeholder access
4. **Access level enforcement** ‚Äî Verify READ/WRITE/FULL access levels are properly enforced

## CI/CD Integration

<Warning>
Ensure test users are seeded in your CI environment before running the test suite. The seeding step must complete successfully for RBAC tests to function.
</Warning>

The test suite is designed for CI/CD integration with these considerations:

- **Database isolation** ‚Äî Tests use `@propwise-test.com` emails for easy cleanup
- **Deterministic seeding** ‚Äî Idempotent user creation prevents conflicts in parallel runs
- **Singleton context** ‚Äî Fast test execution after initial app boot
- **Clear failure reporting** ‚Äî Matrix runner provides detailed error messages with actual response bodies
