## Overview

The test suite focuses on **RBAC (Role-Based Access Control) E2E testing** — verifying that every API endpoint enforces the correct permissions for every user role.

**Stack:** Jest + supertest + `@nestjs/testing`

**Key design decisions:**

- **Singleton app context** — the NestJS app boots once and is shared across all test files (equivalent to NestJS `TestSuite` pattern, implemented manually since `TestSuite` requires NestJS 11+)
- **Matrix-driven tests** — a declarative permission matrix defines expected HTTP status codes per persona × endpoint. A runner generates `it()` blocks automatically
- **API-driven seeding** — test data is created through the actual API endpoints so all business logic (guards, auto-assignment, stakeholder creation) executes naturally

## Architecture

### Singleton Context

<Info>
The app boots once for the entire test run. All test files share the same NestJS app instance, authenticated tokens, and resolved `teamId`.
</Info>

The singleton context flow:

<Steps>
<Step title="Jest starts execution">
First test file calls `getContext()` which boots the NestJS app (~3-5s), logs in 5 personas, and resolves teamId
</Step>
<Step title="Subsequent files use cached context">
Other test files call `getContext()` and receive the cached instance instantly (0ms)
</Step>
<Step title="Cleanup on completion">
Last file's `afterAll()` calls `closeContext()` when reference count hits 0, shutting down the app
</Step>
</Steps>

```typescript
import { getContext, closeContext, RbacTestContext } from '../../setup/rbac-context';

let ctx: RbacTestContext;
beforeAll(async () => {
  ctx = await getContext();
}, 60000);
afterAll(async () => {
  await closeContext();
});
```

The context replicates production config from `main.ts`:
- `ValidationPipe` (whitelist, transform, implicit conversion)
- URI versioning (`/v1/...`)
- Cookie parser

### Matrix Runner

<Tip>
Instead of writing individual `it()` blocks, you define expectations declaratively and let the runner generate parameterized tests automatically.
</Tip>

Example matrix definition:

```typescript
{
  method: 'GET',
  path: '/v1/leads/:leadByBasicId',
  description: 'View lead detail',
  expectations: {
    OWNER: 200,
    ADMIN: 200,
    TEAM_LEADER: 403,
    AGENT: 403,
    BASIC: 200,
  },
}
```

The runner calls `runRbacMatrix()` which:

1. Iterates every endpoint in the matrix
2. For each endpoint, iterates all 5 personas
3. Resolves path placeholders (`:leadByBasicId` → actual UUID)
4. Makes the HTTP request with the persona's tenant token
5. Asserts the response status matches the expected value
6. Produces clear failure messages with the actual response body

<Check>
One matrix entry = 5 generated test cases automatically
</Check>

## RBAC Personas

<CardGroup cols={2}>
<Card title="Permission Profiles" icon="shield-check">
Defined in `test/helpers/rbac-personas.ts` with shared source of truth
</Card>
<Card title="Login Flow" icon="key">
Two-step authentication: identity token → tenant token

| Key           | Email                   | Org Role | Team                           | Description                                                       |
| ------------- | ----------------------- | -------- | ------------------------------ | ----------------------------------------------------------------- |
| `OWNER`       | owner@propwise.com      | Owner    | Any                            | All permissions including `system.owner`                          |
| `ADMIN`       | admin@propwise.com      | Admin    | Any                            | All non-sensitive permissions (includes `system.admin`)           |
| `TEAM_LEADER` | teamleader@propwise.com | Basic    | Team Leader in Test Sales Team | All team permissions (`team_crm.*`, `team_sales.*`, `team.admin`) |
| `AGENT`       | agent@propwise.com      | Basic    | Agent in Test Sales Team       | No team permissions by default                                    |
| `BASIC`       | basic@propwise.com      | Basic    | None                           | No permissions — relies entirely on stakeholder access            |

### Permission Profiles Details

<Tabs>
<Tab title="OWNER / ADMIN">
Bypass resource-level checks via org permissions (`crm.view`, `crm.manage`, `sales.view`, `sales.manage`, etc.). Can access any entity in the org.
</Tab>
<Tab title="TEAM_LEADER">
No org permissions, but has `team_crm.manage` and `team_sales.manage` in the team. Can access entities where the team is a stakeholder.
</Tab>
<Tab title="AGENT">
No org permissions, no team permissions. Can only access entities where they are personally a stakeholder. Used to test access level differentiation (READ vs WRITE vs FULL).
</Tab>
<Tab title="BASIC">
Same as AGENT but not a team member at all. Access is purely stakeholder-based.
</Tab>
</Tabs>

### Login Flow

The context performs a two-step login for each persona:

1. `POST /v1/auth/login` → identity token
2. `POST /v1/auth/select-organization` → tenant token (org-scoped JWT)

All subsequent API calls use the tenant token in the `Authorization: Bearer` header.

## Permission Matrix

Matrices are split into per-entity files under `test/setup/matrices/`. The barrel at `test/setup/rbac-matrix.ts` re-exports everything.

### Lead Matrices

| Matrix                   | Entity                           | Who Can Access                                           | Tests                                                      |
| ------------------------ | -------------------------------- | -------------------------------------------------------- | ---------------------------------------------------------- |
| `LEAD_BY_BASIC_MATRIX`   | Lead created by BASIC            | OWNER, ADMIN (via `crm.*`), BASIC (via FULL stakeholder) | View, update, stakeholders, activities, property interests |
| `LEAD_ON_TEAM_MATRIX`    | Lead with team as stakeholder    | OWNER, ADMIN, TEAM_LEADER (via `team_crm.*`)             | View, update, stakeholders, activities                     |
| `LEAD_AGENT_READ_MATRIX` | Lead where AGENT has READ access | OWNER, ADMIN, AGENT (view only — modify denied)          | View allowed, update denied, access level enforcement      |
| `LEAD_IN_POOL_MATRIX`    | Lead with no stakeholders        | OWNER, ADMIN only (via `crm.*`)                          | View, update                                               |
| `LEAD_ADMIN_ONLY_MATRIX` | Lead for admin-only ops          | OWNER, ADMIN (via `system.admin`)                        | Delete                                                     |

### Access Mode Reference

<Note>
Your `@CheckAccess` decorator uses two modes:
- **OR (default):** User has the required permission **OR** has stakeholder-level access on the resource → allowed
- **AND:** User must have the permission **AND** stakeholder access → allowed
</Note>

Resource action hierarchy:
- `ACCESS` — granted by `READ`, `WRITE`, or `FULL` access level
- `MODIFY` — granted by `WRITE` or `FULL`
- `DELETE` — granted by `FULL` only

## Test Data Seeding

### Step 1: Seed Users (One-time Setup)

<Warning>
Run this command once before executing tests to create the required RBAC personas in the database.
</Warning>

```bash
pnpm test:seed-users
```

This runs `test/setup/seed-test-users.ts` which:

<Steps>
<Step title="Create RBAC personas">
Creates the test personas in the database (idempotent — skips if they exist)
</Step>
<Step title="Set up test organization">
Creates "Test Organization" with owner and ensures **ACTIVE BUSINESS subscription**
</Step>
<Step title="Complete organization setup">
Ensures completed `organization_setup_progress` on the test org (self-heals stale FREE / `LEAD_PIPELINE` rows)
</Step>
<Step title="Create test team">
Creates "Test Sales Team" with team leader and agent assignments
</Step>
<Step title="Assign roles">
Assigns org roles (Owner, Admin, Basic) and team roles (Team Leader, Agent)
</Step>
</Steps>

<Info>
Requires subscription plans to exist. Run `pnpm test:init-data` once on a fresh DB before `pnpm test:seed-users`.
</Info>

### Step 2: Seed Test Data (Automatic)

Each test file calls `seedRbacTestData(app, personas, teamId)` in its `beforeAll`. Results are cached across files. This creates entities through the API:

**Leads:**
- One by BASIC (auto-assigned as primary stakeholder with FULL access)
- One by OWNER (for admin-only tests)
- One in pool (empty stakeholders)
- One with team as stakeholder (OWNER creates, assigns team — TEAM_LEADER gets access)
- One with AGENT as READ-only stakeholder (OWNER creates, adds AGENT with access level)

**Other entities:**
- Contacts and companies created by different personas
- Deals for sales permission testing
- Property interests and activities for relationship testing

## How to Extend

### Adding New Endpoints

<Steps>
<Step title="Define matrix expectations">
Add endpoint definition to appropriate matrix file in `test/setup/matrices/`
</Step>
<Step title="Specify expected status codes">
Define expected HTTP status codes for each persona (OWNER, ADMIN, TEAM_LEADER, AGENT, BASIC)
</Step>
<Step title="Add test data if needed">
Update `seedRbacTestData()` if new entity types or relationships are required
</Step>
<Step title="Run tests">
The matrix runner will automatically generate test cases for the new endpoint
</Step>
</Steps>

### Adding New Personas

<Steps>
<Step title="Update persona definitions">
Add new persona to `test/helpers/rbac-personas.ts`
</Step>
<Step title="Update seeding">
Modify `test/setup/seed-test-users.ts` to create the new user and assign appropriate roles
</Step>
<Step title="Update matrices">
Add expectations for the new persona in all relevant matrices
</Step>
<Step title="Update context">
Ensure `rbac-context.ts` authenticates the new persona
</Step>
</Steps>

### Adding New Entity Types

<Steps>
<Step title="Create matrix file">
Create new matrix file in `test/setup/matrices/` following existing patterns
</Step>
<Step title="Export from barrel">
Add export to `test/setup/matrices/index.ts`
</Step>
<Step title="Create test file">
Create corresponding test file in `test/e2e/rbac/`
</Step>
<Step title="Update test data seeding">
Add entity creation logic to `rbac-test-data.ts`
</Step>
</Steps>

## Methodology

### Controller Analysis

Before writing tests, analyze the controller following `test/TESTING_METHODOLOGY.md`:

1. **Identify endpoints** — list all routes with their HTTP methods
2. **Check guards** — verify `@CheckAccess` decorators and required permissions
3. **Analyze DTOs** — understand request/response structures
4. **Map access patterns** — determine which personas should have access
