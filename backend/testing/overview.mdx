The test suite focuses on **RBAC (Role-Based Access Control) E2E testing** — verifying that every API endpoint enforces the correct permissions for every user role.

## Stack and Design

<CardGroup cols={2}>
  <Card title="Technology Stack" icon="layer-group">
    Jest + supertest + `@nestjs/testing`
  </Card>
  <Card title="Test Pattern" icon="matrix">
    Matrix-driven parameterized tests
  </Card>
</CardGroup>

### Key Design Decisions

<AccordionGroup>
  <Accordion title="Singleton App Context">
    The NestJS app boots once and is shared across all test files (equivalent to NestJS `TestSuite` pattern, implemented manually since `TestSuite` requires NestJS 11+).
  </Accordion>
  
  <Accordion title="Matrix-Driven Tests">
    A declarative permission matrix defines expected HTTP status codes per persona × endpoint. A runner generates `it()` blocks automatically.
  </Accordion>
  
  <Accordion title="API-Driven Seeding">
    Test data is created through the actual API endpoints so all business logic (guards, auto-assignment, stakeholder creation) executes naturally.
  </Accordion>
</AccordionGroup>

## Architecture

### Singleton Context

The app boots once for the entire test run. All test files share the same NestJS app instance, authenticated tokens, and resolved `teamId`.

<Steps>
  <Step title="Jest starts">
    First test file calls `getContext()`
  </Step>
  
  <Step title="Boot sequence">
    Boots NestJS app (~3-5s), logs in 5 personas, resolves teamId
  </Step>
  
  <Step title="Context sharing">
    Subsequent files receive cached instance instantly (0ms)
  </Step>
  
  <Step title="Cleanup">
    Last file's `afterAll()` calls `closeContext()` → app shuts down
  </Step>
</Steps>

**Implementation:**

```typescript
import { getContext, closeContext, RbacTestContext } from '../../setup/rbac-context';

let ctx: RbacTestContext;

beforeAll(async () => {
  ctx = await getContext();
}, 60000);

afterAll(async () => {
  await closeContext();
});
```

<Note>
The context replicates production config from `main.ts`: ValidationPipe (whitelist, transform), URI versioning (`/v1/...`), and cookie parser.
</Note>

### Matrix Runner

Instead of writing individual `it()` blocks, you define expectations declaratively:

```typescript
{
  method: 'GET',
  path: '/v1/leads/:leadByBasicId',
  description: 'View lead detail',
  expectations: {
    OWNER: 200,
    ADMIN: 200,
    TEAM_LEADER: 403,
    AGENT: 403,
    BASIC: 200,
  },
}
```

<Info>
One matrix entry generates 5 test cases automatically (one per persona).
</Info>

The runner (`runRbacMatrix()`) performs these operations:

<Steps>
  <Step title="Iterate endpoints">
    Loops through every endpoint definition in the matrix
  </Step>
  
  <Step title="Test each persona">
    For each endpoint, iterates all 5 personas
  </Step>
  
  <Step title="Resolve placeholders">
    Replaces path variables (`:leadByBasicId` → actual UUID)
  </Step>
  
  <Step title="Execute request">
    Makes HTTP request with persona's tenant token
  </Step>
  
  <Step title="Assert status">
    Verifies response status matches expected value
  </Step>
  
  <Step title="Report failures">
    Produces clear failure messages with actual response body
  </Step>
</Steps>

## RBAC Personas

All personas are defined in `test/helpers/rbac-personas.ts` with a single source of truth.

<Tabs>
  <Tab title="Persona Table">
    | Key | Email | Org Role | Team | Description |
    |-----|-------|----------|------|-------------|
    | `OWNER` | owner@propwise.com | Owner | Any | All permissions including `system.owner` |
    | `ADMIN` | admin@propwise.com | Admin | Any | All non-sensitive permissions (includes `system.admin`) |
    | `TEAM_LEADER` | teamleader@propwise.com | Basic | Team Leader in Test Sales Team | All team permissions (`team_crm.*`, `team_sales.*`, `team.admin`) |
    | `AGENT` | agent@propwise.com | Basic | Agent in Test Sales Team | No team permissions by default |
    | `BASIC` | basic@propwise.com | Basic | None | No permissions — relies entirely on stakeholder access |
  </Tab>
  
  <Tab title="Permission Profiles">
    <AccordionGroup>
      <Accordion title="OWNER / ADMIN">
        Bypass resource-level checks via org permissions (`crm.view`, `crm.manage`, `sales.view`, `sales.manage`, etc.). Can access any entity in the org.
      </Accordion>
      
      <Accordion title="TEAM_LEADER">
        No org permissions, but has `team_crm.manage` and `team_sales.manage` in the team. Can access entities where the team is a stakeholder.
      </Accordion>
      
      <Accordion title="AGENT">
        No org permissions, no team permissions. Can only access entities where they are personally a stakeholder. Used to test access level differentiation (READ vs WRITE vs FULL).
      </Accordion>
      
      <Accordion title="BASIC">
        Same as AGENT but not a team member at all. Access is purely stakeholder-based.
      </Accordion>
    </AccordionGroup>
  </Tab>
</Tabs>

### Login Flow

<Steps>
  <Step title="Identity authentication">
    `POST /v1/auth/login` → identity token
  </Step>
  
  <Step title="Organization selection">
    `POST /v1/auth/select-organization` → tenant token (org-scoped JWT)
  </Step>
  
  <Step title="Subsequent requests">
    All API calls use tenant token in `Authorization: Bearer` header
  </Step>
</Steps>

### Access Mode Reference

<Tip>
Your `@CheckAccess` decorator uses two modes:

- **OR (default):** User has the required permission **OR** has stakeholder-level access → allowed
- **AND:** User must have the permission **AND** stakeholder access → allowed
</Tip>

**Resource action hierarchy:**

- `ACCESS` — granted by `READ`, `WRITE`, or `FULL` access level
- `MODIFY` — granted by `WRITE` or `FULL`
- `DELETE` — granted by `FULL` only

## Test Data Seeding

### Step 1: Seed Users

<Warning>
Run this once before executing tests. This is a one-time setup operation.
</Warning>

```bash
pnpm test:seed-users
```

This runs `test/setup/seed-test-users.ts` which:

<Steps>
  <Step title="Create personas">
    Creates RBAC test personas in the database (idempotent — skips if they exist)
  </Step>
  
  <Step title="Create organization">
    Creates "Test Organization" with owner
  </Step>
  
  <Step title="Setup subscription">
    Ensures **ACTIVE BUSINESS subscription** and **completed organization_setup_progress** on the test org
  </Step>
  
  <Step title="Create team">
    Creates "Test Sales Team" with team leader and agent
  </Step>
  
  <Step title="Assign roles">
    Assigns org roles (Owner, Admin, Basic) and team roles (Team Leader, Agent)
  </Step>
  
  <Step title="Persist to database">
    Uses RLS bypass to write directly via MikroORM
  </Step>
</Steps>

<Info>
The seeder self-heals stale FREE / `LEAD_PIPELINE` rows from older seeds so Playwright and RBAC E2E skip onboarding and paid-plan gates.
</Info>

<Check>
**Prerequisite:** Subscription plans must exist. Run `pnpm test:init-data` once on a fresh DB before `pnpm test:seed-users`.
</Check>

### Step 2: Seed Test Data

Each test file calls `seedRbacTestData(app, personas, teamId)` in its `beforeAll`. Results are cached across files.

**Entities created through the API:**

<Tabs>
  <Tab title="Leads">
    - One by BASIC (auto-assigned as primary stakeholder with FULL access)
    - One by OWNER (for admin-only tests)
    - One in pool (empty stakeholders)
    - One with team as stakeholder (OWNER creates, assigns team — TEAM_LEADER gets access)
    - One with AGENT as READ-only stakeholder (OWNER creates, adds AGENT with `accessLevel: READ`)
  </Tab>
  
  <Tab title="Deals">
    - One by OWNER (for sales permission tests)
  </Tab>
  
  <Tab title="Contacts">
    - One by BASIC (auto-assigned as stakeholder)
    - One by OWNER (for admin-only tests)
  </Tab>
  
  <Tab title="Companies">
    - One by BASIC (auto-assigned as stakeholder)
    - One by OWNER (for admin-only tests)
  </Tab>
</Tabs>

<Note>
All entities are created through authenticated API requests, ensuring business logic (guards, auto-assignment, stakeholder creation) executes naturally.
</Note>

## Running Tests

### Core RBAC Test Suite

```bash
# Run all E2E tests (excludes queue tests)
pnpm test:e2e

# Run specific test file
pnpm test:e2e -- lead-rbac.e2e-spec.ts

# Watch mode
pnpm test:e2e -- --watch
```

### Queue Integration Tests

```bash
# Run queue tests with live pg-boss
pnpm test:e2e:queue
```

### Cleanup

```bash
# Remove all test users from database
pnpm test:cleanup-users
```

<Warning>
Always run `pnpm test:seed-users` after cleanup to restore test data.
</Warning>

## Extending Tests

### Adding a New Matrix Entry

<Steps>
  <Step title="Open matrix file">
    Edit the appropriate matrix file in `test/setup/matrices/`
  </Step>
  
  <Step title="Add endpoint definition">
    ```typescript
    {
      method: 'POST',
      path: '/v1/leads/:leadByBasicId/convert',
      description: 'Convert lead to deal',
      expectations: {
        OWNER: 201,
        ADMIN: 201,
        TEAM_LEADER: 403,
        AGENT: 403,
        BASIC: 201,
      },
    }
    ```
  </Step>
  
  <Step title="Run tests">
    The runner will automatically generate 5 test cases
  </Step>
</Steps>

### Adding a New Persona

<Steps>
  <Step title="Update personas.ts">
    Add new persona definition in `test/helpers/rbac-personas.ts`
  </Step>
  
  <Step title="Update seeder">
    Add creation logic in `test/setup/seed-test-users.ts`
  </Step>
  
  <Step title="Update context">
    Add authentication in `test/setup/rbac-context.ts`
  </Step>
  
  <Step title="Update all matrices">
    Add expectations for new persona in all matrix files
  </Step>
</Steps>

### Adding a New Entity Type

<Steps>
  <Step title="Create matrix file">
    Create `test/setup/matrices/[entity]-matrix.ts`
  </Step>
  
  <Step title="Define matrices">
    Define permission matrices for different scenarios
  </Step>
  
  <Step title="Export from barrel">
    Add export in `test/setup/matrices/index.ts`
  </Step>
  
  <Step title="Update test data seeder">
    Add entity creation in `test/setup/rbac-test-data.ts`
  </Step>
  
  <Step title="Create test file">
    Create `test/e2e/rbac/[entity]-rbac.e2e-spec.ts`
  </Step>
  
  <Step title="Wire up runner">
    Call `runRbacMatrix()` with your new matrices
  </Step>
</Steps>

## CI/CD Integration

<Tip>
The singleton pattern ensures fast test execution in CI pipelines by eliminating redundant app bootstrapping.
</Tip>

**Typical CI workflow:**

```yaml
- name: Setup database
  run: pnpm test:init-data

- name: Seed test users
  run: pnpm test:seed-users

- name: Run E2E tests
  run: pnpm test:e2e

- name: Run queue tests
  run: pnpm test:e2e:queue
```

<Check>
Total test suite runtime: ~30-45 seconds for core RBAC tests (5 personas × 50+ endpoints).
</Check>
