The PropWise CRM backend test suite focuses on **RBAC (Role-Based Access Control) E2E testing** — verifying that every API endpoint enforces the correct permissions for every user role.

<Info>
**Stack:** Jest + supertest + `@nestjs/testing`
</Info>

## Key Design Decisions

<CardGroup cols={2}>
  <Card title="Singleton App Context" icon="server">
    The NestJS app boots once and is shared across all test files (equivalent to NestJS `TestSuite` pattern, implemented manually since `TestSuite` requires NestJS 11+)
  </Card>
  <Card title="Matrix-Driven Tests" icon="table">
    A declarative permission matrix defines expected HTTP status codes per persona × endpoint. A runner generates `it()` blocks automatically
  </Card>
  <Card title="API-Driven Seeding" icon="seedling">
    Test data is created through actual API endpoints so all business logic (guards, auto-assignment, stakeholder creation) executes naturally
  </Card>
  <Card title="Persona-Based Testing" icon="users">
    Five distinct RBAC personas cover all permission profiles from owner to basic user
  </Card>
</CardGroup>

## Architecture

### Singleton Context

The app boots once for the entire test run. All test files share the same NestJS app instance, authenticated tokens, and resolved `teamId`.

<Steps>
  <Step title="Jest Starts">
    First test file calls `getContext()`
  </Step>
  <Step title="Boot & Authenticate">
    - Boots NestJS app (~3-5s)
    - Logs in 5 personas
    - Resolves teamId
    - Returns `{ app, personas, teamId }`
  </Step>
  <Step title="Reuse Context">
    Subsequent test files call `getContext()` and receive cached instance instantly (0ms)
  </Step>
  <Step title="Shutdown">
    Last file's `afterAll()` calls `closeContext()` when reference count hits 0
  </Step>
</Steps>

**Implementation:**

```typescript
import { getContext, closeContext, RbacTestContext } from '../../setup/rbac-context';

let ctx: RbacTestContext;

beforeAll(async () => {
  ctx = await getContext();
}, 60000);

afterAll(async () => {
  await closeContext();
});
```

<Note>
The context replicates production config from `main.ts`: `ValidationPipe` (whitelist, transform, implicit conversion), URI versioning (`/v1/...`), and cookie parser.
</Note>

### Matrix Runner

Instead of writing individual `it()` blocks, you define expectations declaratively:

```typescript
{
  method: 'GET',
  path: '/v1/leads/:leadByBasicId',
  description: 'View lead detail',
  expectations: {
    OWNER: 200,
    ADMIN: 200,
    TEAM_LEADER: 403,
    AGENT: 403,
    BASIC: 200,
  },
}
```

The runner calls `runRbacMatrix()` which:

1. Iterates every endpoint in the matrix
2. For each endpoint, iterates all 5 personas
3. Resolves path placeholders (`:leadByBasicId` → actual UUID)
4. Makes the HTTP request with the persona's tenant token
5. Asserts the response status matches the expected value
6. Produces clear failure messages with the actual response body

<Tip>
One matrix entry = 5 generated test cases
</Tip>

## RBAC Personas

All personas are defined in `test/helpers/rbac-personas.ts` and share a single source of truth.

| Key | Email | Org Role | Team | Description |
|-----|-------|----------|------|-------------|
| `OWNER` | owner@propwise.com | Owner | Any | All permissions including `system.owner` |
| `ADMIN` | admin@propwise.com | Admin | Any | All non-sensitive permissions (includes `system.admin`) |
| `TEAM_LEADER` | teamleader@propwise.com | Basic | Team Leader in Test Sales Team | All team permissions (`team_crm.*`, `team_sales.*`, `team.admin`) |
| `AGENT` | agent@propwise.com | Basic | Agent in Test Sales Team | No team permissions by default |
| `BASIC` | basic@propwise.com | Basic | None | No permissions — relies entirely on stakeholder access |

### Permission Profiles

<AccordionGroup>
  <Accordion title="OWNER / ADMIN">
    Bypass resource-level checks via org permissions (`crm.view`, `crm.manage`, `sales.view`, `sales.manage`, etc.). Can access any entity in the org.
  </Accordion>
  <Accordion title="TEAM_LEADER">
    No org permissions, but has `team_crm.manage` and `team_sales.manage` in the team. Can access entities where the team is a stakeholder.
  </Accordion>
  <Accordion title="AGENT">
    No org permissions, no team permissions. Can only access entities where they are personally a stakeholder. Used to test access level differentiation (READ vs WRITE vs FULL).
  </Accordion>
  <Accordion title="BASIC">
    Same as AGENT but not a team member at all. Access is purely stakeholder-based.
  </Accordion>
</AccordionGroup>

### Login Flow

The context performs a two-step login for each persona:

<Steps>
  <Step title="Identity Token">
    `POST /v1/auth/login` → identity token
  </Step>
  <Step title="Tenant Token">
    `POST /v1/auth/select-organization` → tenant token (org-scoped JWT)
  </Step>
  <Step title="API Calls">
    All subsequent API calls use the tenant token in the `Authorization: Bearer` header
  </Step>
</Steps>

### Access Mode Reference

Your `@CheckAccess` decorator uses two modes:

- **OR (default):** User has the required permission **OR** has stakeholder-level access on the resource → allowed
- **AND:** User must have the permission **AND** stakeholder access → allowed

<Info>
**Resource action hierarchy:**
- `ACCESS` — granted by `READ`, `WRITE`, or `FULL` access level
- `MODIFY` — granted by `WRITE` or `FULL`
- `DELETE` — granted by `FULL` only
</Info>

## Test Data Seeding

### Step 1: Seed Users (One-Time Setup)

<Warning>
Run this once before running tests for the first time or after database resets.
</Warning>

```bash
pnpm test:seed-users
```

This runs `test/setup/seed-test-users.ts` which:

<Check>Creates the RBAC test personas in the database (idempotent — skips if they exist)</Check>
<Check>Creates "Test Organization" with owner</Check>
<Check>Ensures **ACTIVE BUSINESS `subscription`** and **completed `organization_setup_progress`**</Check>
<Check>Creates "Test Sales Team" with team leader and agent</Check>
<Check>Assigns org roles (Owner, Admin, Basic) and team roles (Team Leader, Agent)</Check>
<Check>Uses RLS bypass to write directly via MikroORM</Check>

<Note>
Requires subscription plans to exist. Run `pnpm test:init-data` once on a fresh DB before `pnpm test:seed-users`.
</Note>

### Step 2: Seed Test Data (Automatic)

Each test file calls `seedRbacTestData(app, personas, teamId)` in its `beforeAll`. Results are cached across files.

This creates entities through the API:

**Leads:**
- One by BASIC (auto-assigned as primary stakeholder with FULL access)
- One by OWNER (for admin-only tests)
- One in pool (empty stakeholders)
- One with team as stakeholder (OWNER creates, assigns team — TEAM_LEADER gets access)
- One with AGENT as READ-only stakeholder (OWNER creates, adds AGENT with `accessLevel: 'READ'`)

**Deals:**
- One by OWNER (for sales permission tests)

**Contacts:**
- One by BASIC
- One by OWNER

**Companies:**
- One by BASIC
- One by OWNER

<Tip>
The seeding uses actual API endpoints, so all business logic (guards, auto-assignment, stakeholder creation) executes naturally.
</Tip>

## How to Extend

### Adding a New Persona

<Steps>
  <Step title="Update rbac-personas.ts">
    Add the new persona definition to `RBAC_PERSONAS` constant
  </Step>
  <Step title="Update seed-test-users.ts">
    Add logic to create the user, assign roles, and configure team membership
  </Step>
  <Step title="Update rbac-context.ts">
    Add authentication logic for the new persona in `initializeContext()`
  </Step>
  <Step title="Update Matrices">
    Add the new persona to all relevant permission matrices with expected status codes
  </Step>
  <Step title="Re-seed">
    Run `pnpm test:seed-users` to create the new test user
  </Step>
</Steps>

### Adding a New Matrix

<Steps>
  <Step title="Create Matrix File">
    Create a new file in `test/setup/matrices/` (e.g., `property-matrix.ts`)
  </Step>
  <Step title="Define Expectations">
    ```typescript
    import { EndpointExpectation } from '../rbac-matrix-types';

    export const PROPERTY_BY_BASIC_MATRIX: EndpointExpectation[] = [
      {
        method: 'GET',
        path: '/v1/properties/:propertyId',
        description: 'View property detail',
        expectations: {
          OWNER: 200,
          ADMIN: 200,
          TEAM_LEADER: 403,
          AGENT: 403,
          BASIC: 200,
        },
      },
      // ... more endpoints
    ];
    ```
  </Step>
  <Step title="Export from Barrel">
    Add export to `test/setup/matrices/index.ts`
  </Step>
  <Step title="Create Test File">
    Create `test/e2e/rbac/property-rbac.e2e-spec.ts` and call `runRbacMatrix()`
  </Step>
  <Step title="Seed Test Data">
    Update `rbac-test-data.ts` to create fixture properties via API
  </Step>
</Steps>

### Adding Test Coverage for a New Endpoint

<Steps>
  <Step title="Identify the Controller">
    Review the controller and its DTOs (see `TESTING_METHODOLOGY.md`)
  </Step>
  <Step title="Determine Required Permissions">
    Check `@CheckAccess()` decorators and guard logic
  </Step>
  <Step title="Add to Matrix">
    Add a new entry to the appropriate matrix file with expected status codes for each persona
  </Step>
  <Step title="Run Tests">
    The matrix runner will automatically generate test cases
  </Step>
</Steps>

## Methodology

### Analyzing Controllers Before Writing Tests

<Info>
See `test/TESTING_METHODOLOGY.md` for the complete controller analysis workflow.
</Info>

Key steps:

1. **Identify endpoints** — list all routes, methods, and DTOs
2. **Map permissions** — document `@CheckAccess()` decorators and required permissions
3. **Analyze DTOs** — identify optional vs required fields, nested objects, validation rules
4. **Determine test scenarios** — create minimal valid payloads, edge cases, and validation tests
5. **Plan matrix entries** — define expected status codes for each persona × endpoint combination

### Access Level Testing Pattern

When testing stakeholder access levels (READ, WRITE, FULL):

```typescript
{
  method: 'GET',
  path: '/v1/leads/:leadAgentReadId',
  description: 'AGENT with READ access can view',
  expectations: {
    OWNER: 200,
    ADMIN: 200,
    TEAM_LEADER: 403,
    AGENT: 200,  // Has READ access
    BASIC: 403,
  },
},
{
  method: 'PATCH',
  path: '/v1/leads/:leadAgentReadId',
  description: 'AGENT with READ access cannot modify',
  expectations: {
    OWNER: 200,
    ADMIN: 200,
    TEAM_LEADER: 403,
    AGENT: 403,  // READ access insufficient for MODIFY
    BASIC: 403,
  },
}
```

## CI/CD Integration

### GitHub Actions Workflow

The test suite runs automatically on:
- Pull requests to `main` or `develop`
- Push to `main` or `develop`
- Manual workflow dispatch

<Note>
Ensure the CI environment has:
- PostgreSQL database
- Environment variables configured (see `.env.example`)
- Subscription plans seeded (`pnpm test:init-data`)
- Test users seeded (`pnpm test:seed-users`)
</Note>

### Test Isolation

<Check>Each test file uses the shared app context but operates on distinct entities</Check>
<Check>Entity IDs are resolved from the seeded data cache (no cross-test contamination)</Check>
<Check>Database state is established once per run, not cleaned between tests</Check>
<Check>Tests are idempotent and can run in any order</Check>

<Warning>
Do not modify shared fixture entities during tests. Create new entities if testing mutations.
</Warning>

### Performance Considerations

<Tip>
**Typical run times:**
- First file: ~3-5s (app boot + authentication)
- Subsequent files: ~0-100ms overhead (cached context)
- Total suite: ~30-60s for all RBAC tests
</Tip>

The singleton pattern reduces total test time by 70-80% compared to booting the app per file.
