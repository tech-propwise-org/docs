# Comprehensive Testing Guide

Architecture, conventions, and run guide for the PropWise CRM backend test suite.

---

## Overview

The test suite focuses on **RBAC (Role-Based Access Control) E2E testing** — verifying that every API endpoint enforces the correct permissions for every user role.

**Stack:** Jest + supertest + `@nestjs/testing`

<Note>
**Key design decisions:**
- **Singleton app context** — the NestJS app boots once and is shared across all test files
- **Matrix-driven tests** — a declarative permission matrix defines expected HTTP status codes per persona × endpoint
- **API-driven seeding** — test data is created through the actual API endpoints so all business logic executes naturally
</Note>

## Architecture

### Singleton Context (`rbac-context.ts`)

The app boots once for the entire test run. All test files share the same NestJS app instance, authenticated tokens, and resolved `teamId`.

<CodeGroup>

```typescript Test Flow
Jest starts
  → lead-rbac.e2e-spec.ts calls getContext()
    → First call: boots NestJS app (~3-5s), logs in 5 personas, resolves teamId
    → Returns { app, personas, teamId }
  → deal-rbac.e2e-spec.ts calls getContext()
    → Returns cached instance instantly (0ms)
  → ... same for contact and company files
  → Last file's afterAll() calls closeContext()
    → Reference count hits 0 → app shuts down
```

```typescript Usage Pattern
import { getContext, closeContext, RbacTestContext } from '../../setup/rbac-context';

let ctx: RbacTestContext;
beforeAll(async () => {
  ctx = await getContext();
}, 60000);
afterAll(async () => {
  await closeContext();
});
```

</CodeGroup>

The context replicates production config from `main.ts`:
- `ValidationPipe` (whitelist, transform, implicit conversion)
- URI versioning (`/v1/...`)
- Cookie parser

### Matrix Runner (`rbac-runner.ts`)

Instead of writing individual `it()` blocks, you define expectations declaratively:

```typescript
{
  method: 'GET',
  path: '/v1/leads/:leadByBasicId',
  description: 'View lead detail',
  expectations: {
    OWNER: 200,
    ADMIN: 200,
    TEAM_LEADER: 403,
    AGENT: 403,
    BASIC: 200,
  },
}
```

<Info>
The runner calls `runRbacMatrix()` which:
1. Iterates every endpoint in the matrix
2. For each endpoint, iterates all 5 personas
3. Resolves path placeholders (`:leadByBasicId` → actual UUID)
4. Makes the HTTP request with the persona's tenant token
5. Asserts the response status matches the expected value
6. Produces clear failure messages with the actual response body

One matrix entry = 5 generated test cases.
</Info>

## RBAC Personas

Defined in `test/helpers/rbac-personas.ts`. All personas share a single source of truth.

| Key           | Email                   | Org Role | Team                           | Description                                                       |
| ------------- | ----------------------- | -------- | ------------------------------ | ----------------------------------------------------------------- |
| `OWNER`       | owner@propwise.com      | Owner    | Any                            | All permissions including `system.owner`                          |
| `ADMIN`       | admin@propwise.com      | Admin    | Any                            | All non-sensitive permissions (includes `system.admin`)           |
| `TEAM_LEADER` | teamleader@propwise.com | Basic    | Team Leader in Test Sales Team | All team permissions (`team_crm.*`, `team_sales.*`, `team.admin`) |
| `AGENT`       | agent@propwise.com      | Basic    | Agent in Test Sales Team       | No team permissions by default                                    |
| `BASIC`       | basic@propwise.com      | Basic    | None                           | No permissions — relies entirely on stakeholder access            |

### Permission Profiles

<AccordionGroup>

<Accordion title="OWNER / ADMIN">
Bypass resource-level checks via org permissions (`crm.view`, `crm.manage`, `sales.view`, `sales.manage`, etc.). Can access any entity in the org.
</Accordion>

<Accordion title="TEAM_LEADER">
No org permissions, but has `team_crm.manage` and `team_sales.manage` in the team. Can access entities where the team is a stakeholder.
</Accordion>

<Accordion title="AGENT">
No org permissions, no team permissions. Can only access entities where they are personally a stakeholder. Used to test access level differentiation (READ vs WRITE vs FULL).
</Accordion>

<Accordion title="BASIC">
Same as AGENT but not a team member at all. Access is purely stakeholder-based.
</Accordion>

</AccordionGroup>

### Login Flow

<Steps>
<Step title="Identity Authentication">
`POST /v1/auth/login` → identity token
</Step>
<Step title="Organization Selection">
`POST /v1/auth/select-organization` → tenant token (org-scoped JWT)
</Step>
<Step title="API Access">
All subsequent API calls use the tenant token in the `Authorization: Bearer` header
</Step>
</Steps>

### Access Mode Reference

<Note>
Your `@CheckAccess` decorator uses two modes:
- **OR (default):** User has the required permission **OR** has stakeholder-level access on the resource → allowed
- **AND:** User must have the permission **AND** stakeholder access → allowed
</Note>

Resource action hierarchy:
- `ACCESS` — granted by `READ`, `WRITE`, or `FULL` access level
- `MODIFY` — granted by `WRITE` or `FULL`
- `DELETE` — granted by `FULL` only

## Test Data Seeding

### Step 1: Seed Users (One-time Setup)

<Steps>
<Step title="Run the seeder">
```bash
pnpm test:seed-users
```
</Step>

<Step title="What it creates">
This runs `test/setup/seed-test-users.ts` which:
- Creates the RBAC test personas in the database (idempotent)
- Creates "Test Organization" with owner
- Ensures **ACTIVE BUSINESS subscription** and **completed organization_setup_progress**
- Creates "Test Sales Team" with team leader and agent
- Assigns org roles (Owner, Admin, Basic) and team roles (Team Leader, Agent)
- Uses RLS bypass to write directly via MikroORM
</Step>
</Steps>

<Warning>
Requires subscription plans to exist. Run `pnpm test:init-data` once on a fresh DB before `pnpm test:seed-users`.
</Warning>

### Step 2: Seed Test Data (Automatic)

Each test file calls `seedRbacTestData(app, personas, teamId)` in its `beforeAll`. Results are cached across files. This creates entities through the API:

<Tabs>
<Tab title="Leads">
- One by BASIC (auto-assigned as primary stakeholder with FULL access)
- One by OWNER (for admin-only tests)  
- One in pool (empty stakeholders)
- One with team as stakeholder (OWNER creates, assigns team)
- One with AGENT as READ-only stakeholder
</Tab>

<Tab title="Deals">
- One by OWNER (admin-only access via `sales.*`)
</Tab>

<Tab title="Contacts">
- One by BASIC (BASIC has FULL access)
- One by OWNER (admin-only access)
</Tab>

<Tab title="Companies">
- One by BASIC (BASIC has FULL access)
- One by OWNER (admin-only access)
</Tab>
</Tabs>

## How to Extend

### Adding New Endpoints

<Steps>
<Step title="Add to existing matrix">
Add new endpoint expectations to the appropriate matrix file in `test/setup/matrices/`:

```typescript
{
  method: 'POST',
  path: '/v1/leads/:leadByBasicId/notes',
  description: 'Create lead note',
  expectations: {
    OWNER: 201,
    ADMIN: 201,
    TEAM_LEADER: 403,
    AGENT: 403,
    BASIC: 201,
  },
}
```
</Step>

<Step title="Handle path parameters">
If your endpoint uses path parameters, add resolution logic in `rbac-runner.ts` or the seeding functions.
</Step>

<Step title="Run tests">
The matrix runner will automatically generate test cases for all personas.
</Step>
</Steps>

### Adding New Entities

<Steps>
<Step title="Create matrix file">
Create a new matrix file in `test/setup/matrices/new-entity-matrix.ts`
</Step>

<Step title="Export from index">
Add to `test/setup/matrices/index.ts`
</Step>

<Step title="Update test data seeding">
Add entity creation logic to `test/setup/rbac-test-data.ts`
</Step>

<Step title="Create test spec">
Create `test/e2e/rbac/new-entity-rbac.e2e-spec.ts` following the existing pattern
</Step>
</Steps>

### Adding New Personas

<Steps>
<Step title="Update personas definition">
Add to `test/helpers/rbac-personas.ts`
</Step>

<Step title="Update seeder">
Modify `test/setup/seed-test-users.ts` to create the new persona
</Step>

<Step title="Update matrices">
Add expectations for the new persona in all relevant matrices
</Step>

<Step title="Update context">
Ensure `rbac-context.ts` authenticates the new persona
</Step>
</Steps>

### Access Mode Testing

<Note>
**OR Mode Testing:** Verify users can access resources through either org permissions OR stakeholder access

**AND Mode Testing:** Verify users need both org permissions AND stakeholder access

</Note>

### Performance Considerations

<Tip>
**Singleton Context:** App boots once (~3-5s) then all tests run against the shared instance

**Cached Test Data:** Entity fixtures are created once per test run and reused across files

**Parallel Execution:** Tests run in parallel within files but files run sequentially to avoid database conflicts
</Tip>

### Test Environment

<Warning>
Always run tests against a dedicated test database. The seeder creates test users with `@propwise-test.com` domains that can be safely cleaned up.
</Warning>

Required environment variables:
- `DATABASE_URL` — Test database connection
- `JWT_SECRET` — JWT signing secret
- `REDIS_URL` — Redis connection for queue tests (optional)
