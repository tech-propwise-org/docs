The test suite focuses on **RBAC (Role-Based Access Control) E2E testing** — verifying that every API endpoint enforces the correct permissions for every user role.

**Stack:** Jest + supertest + `@nestjs/testing`

<Info>
**Key design decisions:**
- **Singleton app context** — the NestJS app boots once and is shared across all test files
- **Matrix-driven tests** — a declarative permission matrix defines expected HTTP status codes per persona × endpoint
- **API-driven seeding** — test data is created through actual API endpoints so all business logic executes naturally
</Info>

## Architecture

### Singleton Context

The app boots once for the entire test run. All test files share the same NestJS app instance, authenticated tokens, and resolved `teamId`.

<Steps>
<Step title="First test file calls getContext()">
Boots NestJS app (~3-5s), logs in 5 personas, resolves teamId
</Step>
<Step title="Subsequent files reuse cached context">
Returns cached instance instantly (0ms)
</Step>
<Step title="Last file's afterAll() shuts down">
Reference count hits 0 → app closes cleanly
</Step>
</Steps>

```typescript
import { getContext, closeContext, RbacTestContext } from '../../setup/rbac-context';

let ctx: RbacTestContext;
beforeAll(async () => {
  ctx = await getContext();
}, 60000);
afterAll(async () => {
  await closeContext();
});
```

<Note>
The context replicates production config from `main.ts`:
- `ValidationPipe` (whitelist, transform, implicit conversion)
- URI versioning (`/v1/...`)
- Cookie parser
</Note>

### Matrix Runner

Instead of writing individual test cases, define expectations declaratively:

```typescript
{
  method: 'GET',
  path: '/v1/leads/:leadByBasicId',
  description: 'View lead detail',
  expectations: {
    OWNER: 200,
    ADMIN: 200,
    TEAM_LEADER: 403,
    AGENT: 403,
    BASIC: 200,
  },
}
```

<Info>
One matrix entry generates 5 test cases automatically. The runner:
1. Iterates every endpoint in the matrix
2. For each endpoint, tests all 5 personas
3. Resolves path placeholders (`:leadByBasicId` → actual UUID)
4. Makes HTTP requests with persona's tenant token
5. Asserts response status matches expected value
</Info>

### Permission Profiles

<Tabs>
<Tab title="OWNER / ADMIN">
Bypass resource-level checks via org permissions (`crm.view`, `crm.manage`, `sales.view`, `sales.manage`). Can access any entity in the org.
</Tab>
<Tab title="TEAM_LEADER">
No org permissions, but has `team_crm.manage` and `team_sales.manage` in the team. Can access entities where the team is a stakeholder.
</Tab>
<Tab title="AGENT">
No org permissions, no team permissions. Can only access entities where they are personally a stakeholder. Used to test access level differentiation.
</Tab>
<Tab title="BASIC">
Same as AGENT but not a team member. Access is purely stakeholder-based.
</Tab>
</Tabs>

### Login Flow

The context performs a two-step login for each persona:

1. `POST /v1/auth/login` → identity token
2. `POST /v1/auth/select-organization` → tenant token (org-scoped JWT)

<Warning>
All subsequent API calls use the tenant token in the `Authorization: Bearer` header.
</Warning>

### Access Mode Reference

<Note>
Your `@CheckAccess` decorator uses two modes:
- **OR (default):** User has required permission **OR** stakeholder-level access → allowed
- **AND:** User must have permission **AND** stakeholder access → allowed
</Note>

Resource action hierarchy:
- `ACCESS` — granted by `READ`, `WRITE`, or `FULL` access level
- `MODIFY` — granted by `WRITE` or `FULL`
- `DELETE` — granted by `FULL` only

## Test Data Seeding

### Step 1: Seed Users (one-time setup)

<Steps>
<Step title="Run the seeding command">
```bash
pnpm test:seed-users
```
</Step>
<Step title="Creates RBAC test personas">
- Idempotent operation (skips if users exist)
- Creates "Test Organization" with owner
- Ensures **ACTIVE BUSINESS subscription**
- Creates "Test Sales Team" with roles
</Step>
<Step title="Assigns permissions">
- Org roles (Owner, Admin, Basic)
- Team roles (Team Leader, Agent)
- Uses RLS bypass via MikroORM
</Step>
</Steps>

<Warning>
Requires subscription plans to exist. Run `pnpm test:init-data` once on a fresh DB before `pnpm test:seed-users`.
</Warning>

### Step 2: Seed Test Data (automatic)

Each test file calls `seedRbacTestData(app, personas, teamId)` in its `beforeAll`. Results are cached across files.

<Tabs>
<Tab title="Leads">
- One by BASIC (auto-assigned as primary stakeholder with FULL access)
- One by OWNER (for admin-only tests)
- One in pool (empty stakeholders)
- One with team as stakeholder
- One with AGENT as READ-only stakeholder
</Tab>
<Tab title="Deals">
- One by OWNER (sales pipeline testing)
</Tab>
<Tab title="Contacts">
- One by BASIC (stakeholder access)
- One by OWNER (admin-only access)
</Tab>
<Tab title="Companies">
- One by BASIC (stakeholder access)
- One by OWNER (admin-only access)
</Tab>
</Tabs>

## How to Run

<CodeGroup>
```bash Core RBAC Tests
# Run all RBAC tests (excludes queue tests)
pnpm test:e2e

# Run specific test file
pnpm test:e2e lead-rbac

# Run with coverage
pnpm test:e2e:cov
```

```bash Queue Tests
# Run queue-specific tests (requires live pg-boss)
pnpm test:e2e:queue

# Run distribution tests
pnpm test:e2e:queue distribution

# Run escalation tests
pnpm test:e2e:queue escalation
```

```bash Setup Commands
# Seed test users (one-time)
pnpm test:seed-users

# Clean up test users
pnpm test:cleanup-users

# Initialize subscription data
pnpm test:init-data
```
</CodeGroup>

<Warning>
Queue tests require a live PostgreSQL connection and pg-boss instance. Core RBAC tests use mocked pg-boss.
</Warning>

## How to Extend

### Adding New Endpoints

<Steps>
<Step title="Update the appropriate matrix">
Add your endpoint to the relevant matrix file in `test/setup/matrices/`
</Step>
<Step title="Define expectations">
```typescript
{
  method: 'POST',
  path: '/v1/leads/:leadByBasicId/activities',
  description: 'Create activity',
  expectations: {
    OWNER: 201,
    ADMIN: 201,
    TEAM_LEADER: 403,
    AGENT: 403,
    BASIC: 201,
  },
}
```
</Step>
<Step title="Handle path parameters">
Ensure your test data includes the necessary entities for path resolution
</Step>
</Steps>

### Adding New Personas

<Steps>
<Step title="Define persona in rbac-personas.ts">
Add email, org role, and team configuration
</Step>
<Step title="Update seed-test-users.ts">
Add user creation and role assignment logic
</Step>
<Step title="Update rbac-context.ts">
Add login flow for the new persona
</Step>
<Step title="Update all matrices">
Add expectations for the new persona across all endpoint matrices
</Step>
</Steps>

### Adding New Entities

<Steps>
<Step title="Create entity matrix file">
Add `test/setup/matrices/{entity}-matrix.ts`
</Step>
<Step title="Update rbac-test-data.ts">
Add seeding logic for the new entity
</Step>
<Step title="Create RBAC test file">
Add `test/e2e/rbac/{entity}-rbac.e2e-spec.ts`
</Step>
<Step title="Export from barrel files">
Update `test/setup/matrices/index.ts` and `test/setup/rbac-matrix.ts`
</Step>
</Steps>

## Methodology

<Tip>
See `test/TESTING_METHODOLOGY.md` for detailed guidance on analyzing controllers and DTOs before writing tests.
</Tip>

The methodology focuses on:

1. **Controller Analysis** — Identify all endpoints, decorators, and permission requirements
2. **DTO Validation** — Test input validation and transformation rules
3. **Permission Mapping** — Map each endpoint to expected persona access patterns
4. **Edge Case Identification** — Find boundary conditions and error scenarios

## CI/CD Integration

<Note>
Tests run automatically on PR creation and merge to main branch.
</Note>

**Pipeline Steps:**
1. Database setup with test data
2. User seeding (`pnpm test:seed-users`)
3. Core RBAC test execution (`pnpm test:e2e`)
4. Queue test execution (`pnpm test:e2e:queue`)
5. Cleanup and teardown

**Environment Requirements:**
- PostgreSQL database
- Redis instance (for queue tests)
- Environment variables matching test configuration

<Check>
All tests must pass before code can be merged to main branch.
</Check>
