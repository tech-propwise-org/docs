## Overview

The test suite focuses on **RBAC (Role-Based Access Control) E2E testing** — verifying that every API endpoint enforces the correct permissions for every user role.

**Stack:** Jest + supertest + `@nestjs/testing`

### Key Design Decisions

<CardGroup cols={2}>
  <Card title="Singleton App Context" icon="server">
    The NestJS app boots once and is shared across all test files (equivalent to NestJS `TestSuite` pattern, implemented manually since `TestSuite` requires NestJS 11+)
  </Card>
  <Card title="Matrix-Driven Tests" icon="table-cells">
    A declarative permission matrix defines expected HTTP status codes per persona × endpoint. A runner generates `it()` blocks automatically
  </Card>
  <Card title="API-Driven Seeding" icon="seedling">
    Test data is created through the actual API endpoints so all business logic (guards, auto-assignment, stakeholder creation) executes naturally
  </Card>
</CardGroup>

## Architecture

### Singleton Context

The app boots once for the entire test run. All test files share the same NestJS app instance, authenticated tokens, and resolved `teamId`.

<Info>
The singleton pattern significantly reduces test execution time by avoiding repeated app initialization (~3-5s savings per test file).
</Info>

```typescript
import { getContext, closeContext, RbacTestContext } from '../../setup/rbac-context';

let ctx: RbacTestContext;
beforeAll(async () => {
  ctx = await getContext();
}, 60000);
afterAll(async () => {
  await closeContext();
});
```

### Matrix Runner

Instead of writing individual `it()` blocks, you define expectations declaratively:

```typescript
{
  method: 'GET',
  path: '/v1/leads/:leadByBasicId',
  description: 'View lead detail',
  expectations: {
    OWNER: 200,
    ADMIN: 200,
    TEAM_LEADER: 403,
    AGENT: 403,
    BASIC: 200,
  },
}
```

<Steps>
  <Step title="Matrix Definition">
    Define endpoint expectations with HTTP methods, paths, and expected status codes per persona
  </Step>
  <Step title="Runner Execution">
    The runner iterates every endpoint in the matrix and generates parameterized Jest tests
  </Step>
  <Step title="Path Resolution">
    Resolves path placeholders (`:leadByBasicId` → actual UUID) dynamically
  </Step>
  <Step title="Request Execution">
    Makes HTTP requests with each persona's tenant token and asserts response status
  </Step>
</Steps>

## RBAC Personas

<Warning>
All personas use `@propwise-test.com` email addresses and share a single test organization and team structure.
</Warning>

| Key | Email | Org Role | Team | Description |
|-----|-------|----------|------|-------------|
| `OWNER` | owner@propwise-test.com | Owner | Any | All permissions including `system.owner` |
| `ADMIN` | admin@propwise-test.com | Admin | Any | All non-sensitive permissions (includes `system.admin`) |
| `TEAM_LEADER` | teamleader@propwise-test.com | Basic | Team Leader in Test Sales Team | All team permissions (`team_crm.*`, `team_sales.*`, `team.admin`) |
| `AGENT` | agent@propwise-test.com | Basic | Agent in Test Sales Team | No team permissions by default |
| `BASIC` | basic@propwise-test.com | Basic | None | No permissions — relies entirely on stakeholder access |

### Permission Profiles

<Tabs>
  <Tab title="OWNER / ADMIN">
    Bypass resource-level checks via org permissions (`crm.view`, `crm.manage`, `sales.view`, `sales.manage`, etc.). Can access any entity in the org.
  </Tab>
  <Tab title="TEAM_LEADER">
    No org permissions, but has `team_crm.manage` and `team_sales.manage` in the team. Can access entities where the team is a stakeholder.
  </Tab>
  <Tab title="AGENT">
    No org permissions, no team permissions. Can only access entities where they are personally a stakeholder. Used to test access level differentiation (READ vs WRITE vs FULL).
  </Tab>
  <Tab title="BASIC">
    Same as AGENT but not a team member at all. Access is purely stakeholder-based.
  </Tab>
</Tabs>

### Login Flow

The context performs a two-step login for each persona:

1. `POST /v1/auth/login` → identity token
2. `POST /v1/auth/select-organization` → tenant token (org-scoped JWT)

<Note>
All subsequent API calls use the tenant token in the `Authorization: Bearer` header.
</Note>

## Permission Matrix

Matrices are split into per-entity files under `test/setup/matrices/`. The barrel at `test/setup/rbac-matrix.ts` re-exports everything.

<AccordionGroup>

    |--------|--------|----------------|-------|

### Access Mode Reference

Your `@CheckAccess` decorator uses two modes:

- **OR (default):** User has the required permission **OR** has stakeholder-level access on the resource → allowed
- **AND:** User must have the permission **AND** stakeholder access → allowed

<Tip>
Resource action hierarchy: `ACCESS` (granted by READ, WRITE, or FULL) → `MODIFY` (granted by WRITE or FULL) → `DELETE` (granted by FULL only)
</Tip>

## Test Data Seeding

### Step 1: Seed Users (one-time)

<CodeGroup>
```bash Terminal
pnpm test:seed-users
```
</CodeGroup>

This runs `test/setup/seed-test-users.ts` which:

<Steps>
  <Step title="Create Personas">
    Creates the RBAC test personas in the database (idempotent — skips if they exist)
  </Step>
  <Step title="Create Organization">
    Creates "Test Organization" with owner
  </Step>
  <Step title="Setup Subscription">
    Ensures **ACTIVE BUSINESS `subscription`** and **completed `organization_setup_progress`** on the test org
  </Step>
  <Step title="Create Team">
    Creates "Test Sales Team" with team leader and agent
  </Step>
  <Step title="Assign Roles">
    Assigns org roles (Owner, Admin, Basic) and team roles (Team Leader, Agent)
  </Step>
</Steps>

<Warning>
Requires subscription plans to exist. Run `pnpm test:init-data` once on a fresh DB before `pnpm test:seed-users`.
</Warning>

### Step 2: Seed Test Data (automatic)

Each test file calls `seedRbacTestData(app, personas, teamId)` in its `beforeAll`. Results are cached across files. This creates entities through the API:

- **Leads:**
  - One by BASIC (auto-assigned as primary stakeholder with FULL access)
  - One by OWNER (for admin-only tests)
  - One in pool (empty stakeholders)
  - One with team as stakeholder
  - One with AGENT as READ-only stakeholder

## How to Extend

### Adding New Entities

<Steps>
  <Step title="Create Matrix File">
    Create `test/setup/matrices/new-entity-matrix.ts` with permission expectations
  </Step>
  <Step title="Add to Barrel Export">
    Update `test/setup/matrices/index.ts` to export your new matrix
  </Step>
  <Step title="Extend Seeding">
    Add entity creation logic to `test/setup/rbac-test-data.ts`
  </Step>
  <Step title="Create Test File">
    Create `test/e2e/rbac/new-entity-rbac.e2e-spec.ts` using the matrix runner pattern
  </Step>
</Steps>

### Adding New Personas

<Steps>
  <Step title="Update Personas Definition">
    Add new persona to `test/helpers/rbac-personas.ts`
  </Step>
  <Step title="Update Seeding">
    Modify `test/setup/seed-test-users.ts` to create the new user
  </Step>
  <Step title="Update Matrices">
    Add expectations for the new persona across all existing matrices
  </Step>
  <Step title="Update Context">
    Modify `test/setup/rbac-context.ts` to authenticate the new persona
  </Step>
</Steps>

## Methodology

<Info>
Refer to `test/TESTING_METHODOLOGY.md` for detailed guidance on analyzing controllers and DTOs before writing tests.
</Info>

The testing methodology follows these principles:

1. **Permission-First Design** — Define what each persona should be able to access before writing implementation
2. **Declarative Matrices** — Express expectations as data, not code
3. **Real API Flow** — Use actual endpoints for seeding to ensure business logic executes
4. **Isolation** — Each test file can run independently with shared context

## CI/CD Integration

The test suite integrates with CI/CD pipelines through:

- **Parallel Execution** — Test files can run in parallel due to singleton context
- **Database Isolation** — Uses dedicated test database with cleanup procedures
- **Artifact Generation** — Produces JUnit XML reports for CI dashboard integration
- **Failure Reporting** — Detailed error messages include actual response bodies for debugging

<Check>
Tests are designed to be deterministic and can run reliably in containerized CI environments.
</Check>
