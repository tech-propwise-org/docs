The test suite focuses on **RBAC (Role-Based Access Control) E2E testing** — verifying that every API endpoint enforces the correct permissions for every user role.

**Stack:** Jest + supertest + `@nestjs/testing`

<Note>
Key design decisions include singleton app context, matrix-driven tests, and API-driven seeding to ensure all business logic executes naturally.
</Note>

## Architecture

### Singleton Context

The NestJS app boots once and is shared across all test files for optimal performance.

<Tabs>
  <Tab title="Boot Flow">
    ```
    Jest starts
      → lead-rbac.e2e-spec.ts calls getContext()
        → First call: boots NestJS app (~3-5s), logs in 5 personas
        → Returns { app, personas, teamId }
      → deal-rbac.e2e-spec.ts calls getContext()
        → Returns cached instance instantly (0ms)
      → Last file's afterAll() calls closeContext()
        → App shuts down
    ```
  </Tab>
  <Tab title="Implementation">
    ```typescript
    import { getContext, closeContext, RbacTestContext } from '../../setup/rbac-context';

    let ctx: RbacTestContext;
    beforeAll(async () => {
      ctx = await getContext();
    }, 60000);
    afterAll(async () => {
      await closeContext();
    });
    ```
  </Tab>
</Tabs>

The context replicates production configuration:
- `ValidationPipe` with whitelist and transform
- URI versioning (`/v1/...`)
- Cookie parser

### Matrix Runner

<Info>
Instead of writing individual test cases, define expectations declaratively using permission matrices.
</Info>

```typescript
{
  method: 'GET',
  path: '/v1/leads/:leadByBasicId',
  description: 'View lead detail',
  expectations: {
    OWNER: 200,
    ADMIN: 200,
    TEAM_LEADER: 403,
    AGENT: 403,
    BASIC: 200,
  },
}
```

The runner automatically:
1. Iterates every endpoint in the matrix
2. Tests all 5 personas for each endpoint
3. Resolves path placeholders with actual UUIDs
4. Makes HTTP requests with persona tokens
5. Asserts response status matches expectations

<Check>One matrix entry generates 5 parameterized test cases automatically</Check>

### Permission Profiles

<AccordionGroup>
  <Accordion title="OWNER / ADMIN">
    Bypass resource-level checks via organization permissions (`crm.view`, `crm.manage`, `sales.view`, `sales.manage`). Can access any entity in the organization.
  </Accordion>
  
  <Accordion title="TEAM_LEADER">
    No org permissions, but has `team_crm.manage` and `team_sales.manage` in the team. Can access entities where the team is a stakeholder.
  </Accordion>
  
  <Accordion title="AGENT">
    No org or team permissions. Can only access entities where they are personally a stakeholder. Used to test access level differentiation (READ vs WRITE vs FULL).
  </Accordion>
  
  <Accordion title="BASIC">
    Same as AGENT but not a team member. Access is purely stakeholder-based for testing isolation.
  </Accordion>
</AccordionGroup>

### Login Flow

Each persona follows a two-step authentication process:

<Steps>
  <Step title="Identity Login">
    `POST /v1/auth/login` → Returns identity token
  </Step>
  <Step title="Organization Selection">
    `POST /v1/auth/select-organization` → Returns tenant token (org-scoped JWT)
  </Step>
</Steps>

<Warning>
All subsequent API calls must use the tenant token in the `Authorization: Bearer` header.
</Warning>

## Permission Matrix

Matrices are organized by entity type in separate files under `test/setup/matrices/`.

### Lead Matrices

| Matrix | Entity | Access Pattern | Test Coverage |
|--------|--------|----------------|---------------|
| `LEAD_BY_BASIC_MATRIX` | Lead created by BASIC | OWNER, ADMIN (via `crm.*`), BASIC (via stakeholder) | View, update, stakeholders, activities |
| `LEAD_ON_TEAM_MATRIX` | Lead with team stakeholder | OWNER, ADMIN, TEAM_LEADER (via `team_crm.*`) | View, update, stakeholders |
| `LEAD_AGENT_READ_MATRIX` | Lead with AGENT read access | OWNER, ADMIN, AGENT (view only) | Access level enforcement |
| `LEAD_IN_POOL_MATRIX` | Lead with no stakeholders | OWNER, ADMIN only (via `crm.*`) | Pool access control |
| `LEAD_ADMIN_ONLY_MATRIX` | Admin operations | OWNER, ADMIN (via `system.admin`) | Delete operations |

### Deal Matrices

| Matrix | Entity | Access Pattern |
|--------|--------|----------------|
| `DEAL_BY_OWNER_MATRIX` | Deal created by OWNER | OWNER, ADMIN only (via `sales.*`) |

### Contact Matrices

| Matrix | Entity | Access Pattern |
|--------|--------|----------------|
| `CONTACT_BY_BASIC_MATRIX` | Contact created by BASIC | OWNER, ADMIN, BASIC |
| `CONTACT_BY_OWNER_MATRIX` | Contact created by OWNER | OWNER, ADMIN only |

### Company Matrices

| Matrix | Entity | Access Pattern |
|--------|--------|----------------|
| `COMPANY_BY_BASIC_MATRIX` | Company created by BASIC | OWNER, ADMIN, BASIC |
| `COMPANY_BY_OWNER_MATRIX` | Company created by OWNER | OWNER, ADMIN only |

### Access Modes

The `@CheckAccess` decorator supports two validation modes:

<Tabs>
  <Tab title="OR Mode (Default)">
    User has the required permission **OR** stakeholder-level access → allowed
  </Tab>
  <Tab title="AND Mode">
    User must have the permission **AND** stakeholder access → allowed
  </Tab>
</Tabs>

**Resource Action Hierarchy:**
- `ACCESS` — granted by READ, WRITE, or FULL access level
- `MODIFY` — granted by WRITE or FULL access level
- `DELETE` — granted by FULL access level only

## Test Data Seeding

### Step 1: Seed Users (One-time Setup)

<Steps>
  <Step title="Run User Seeder">
    ```bash
    pnpm test:seed-users
    ```
    
    This executes `test/setup/seed-test-users.ts` which:
    - Creates RBAC personas (idempotent)
    - Creates "Test Organization" with owner
    - Ensures ACTIVE BUSINESS subscription
    - Creates "Test Sales Team" with roles
    - Uses RLS bypass for direct database writes
  </Step>
  
  <Step title="Verify Prerequisites">
    <Warning>
    Requires subscription plans to exist. Run `pnpm test:init-data` once on fresh databases before seeding users.
    </Warning>
  </Step>
</Steps>

### Step 2: Seed Test Data (Automatic)

Each test file calls `seedRbacTestData(app, personas, teamId)` in `beforeAll`. Results are cached across files.

<Tip>
Test data is created through actual API endpoints, ensuring all business logic (guards, auto-assignment, stakeholder creation) executes naturally.
</Tip>

**Entities Created:**
- Leads (by BASIC, by OWNER, in pool, with team stakeholder, with AGENT read access)
- Deals (by OWNER for admin-only testing)
- Contacts and Companies (by various personas for access testing)

## How to Run

### Available Commands

<CodeGroup>
```bash Core E2E Tests
# Run all RBAC tests (excludes queue tests)
pnpm test:e2e

# Run with coverage
pnpm test:e2e:cov

# Run specific test file
pnpm test:e2e --testNamePattern="lead-rbac"
```

```bash Queue Tests
# Run distribution and escalation tests with live pg-boss
pnpm test:queue

# Single queue test
pnpm test:queue --testNamePattern="distribution"
```

```bash Maintenance
# Seed test users (one-time setup)
pnpm test:seed-users

# Clean up test users
pnpm test:cleanup-users

# Initialize subscription data
pnpm test:init-data
```
</CodeGroup>

### Test Execution Flow

<Steps>
  <Step title="Setup Phase">
    - Boot NestJS application
    - Authenticate all personas
    - Resolve team ID
    - Seed test entities via API
  </Step>
  
  <Step title="Test Phase">
    - Matrix runner generates test cases
    - Each persona attempts each endpoint
    - Assert response status matches expectations
  </Step>
  
  <Step title="Cleanup Phase">
    - Close application context
    - Test data persists for debugging
  </Step>
</Steps>

## How to Extend

### Adding New Endpoints

<Steps>
  <Step title="Define Matrix Entry">
    ```typescript
    {
      method: 'POST',
      path: '/v1/new-entity',
      description: 'Create new entity',
      expectations: {
        OWNER: 201,
        ADMIN: 201,
        TEAM_LEADER: 403,
        AGENT: 403,
        BASIC: 403,
      },
    }
    ```
  </Step>
  
  <Step title="Add Test Data">
    Extend `seedRbacTestData()` to create entities needed for path resolution
  </Step>
  
  <Step title="Update Matrix File">
    Add the new matrix to appropriate entity file in `matrices/` directory
  </Step>
</Steps>

### Adding New Personas

<Steps>
  <Step title="Define Persona">
    Add to `test/helpers/rbac-personas.ts`:
    ```typescript
    export const NEW_PERSONA: TestUser = {
      email: 'new@propwise.com',
      password: 'Password123!',
      persona: 'NEW_PERSONA',
      orgRole: 'Basic',
      teamRole: 'Custom',
    };
    ```
  </Step>
  
  <Step title="Update Seeder">
    Modify `seed-test-users.ts` to create the new persona
  </Step>
  
  <Step title="Update Matrices">
    Add expectations for the new persona in all relevant matrices
  </Step>
</Steps>

### Test Design Principles

  <Card title="Comprehensive Coverage" icon="check-circle">

  </Card>
  <Card title="Real Business Logic" icon="cog">

  </Card>
  <Card title="Performance Optimized" icon="rocket">

  <Card title="Declarative Testing" icon="code">
    Matrix-driven approach reduces boilerplate and improves maintainability
  </Card>

### Analyzing New Controllers

Before writing tests for new endpoints:

1. **Review controller decorators** — identify required permissions
2. **Examine DTO validations** — understand request/response shapes
3. **Check service dependencies** — identify stakeholder requirements
4. **Map access patterns** — determine which personas should have access

### Database Management

<Warning>
Test databases should be isolated per environment and regularly refreshed to prevent data accumulation.
</Warning>

- Automated cleanup of `@propwise-test.com` users
- Test data persists for debugging but doesn't interfere
- Database migrations tested in separate pipeline stage
