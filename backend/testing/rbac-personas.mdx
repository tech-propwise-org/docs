The RBAC testing framework uses five predefined personas to comprehensively test role-based access control across all API endpoints. Each persona represents a different permission level and access pattern in the PropWise CRM system.

## Test Personas Overview

<CardGroup cols={2}>
  <Card title="Administrative Roles" icon="crown">
    **OWNER** and **ADMIN** personas with organization-wide permissions
  </Card>
  <Card title="Team-Based Roles" icon="users">
    **TEAM_LEADER** and **AGENT** personas with team-specific access
  </Card>
</CardGroup>

All personas are defined in `test/helpers/rbac-personas.ts` and share a single source of truth across the test suite.

## Persona Definitions

### OWNER Persona
- **Email:** `owner@propwise-test.com`
- **Org Role:** Owner
- **Team:** Any
- **Permissions:** All permissions including `system.owner`
- **Access Pattern:** Bypasses resource-level checks via org permissions

<Note>
The OWNER persona can access any entity in the organization regardless of stakeholder assignments.
</Note>

### ADMIN Persona
- **Email:** `admin@propwise-test.com`
- **Org Role:** Admin
- **Team:** Any
- **Permissions:** All non-sensitive permissions (includes `system.admin`)
- **Access Pattern:** Has `crm.*` and `sales.*` org permissions

### TEAM_LEADER Persona
- **Email:** `teamleader@propwise-test.com`
- **Org Role:** Basic
- **Team:** Team Leader in Test Sales Team
- **Permissions:** `team_crm.*`, `team_sales.*`, `team.admin`
- **Access Pattern:** No org permissions, but can access entities where the team is a stakeholder

### AGENT Persona
- **Email:** `agent@propwise-test.com`
- **Org Role:** Basic
- **Team:** Agent in Test Sales Team
- **Permissions:** No team permissions by default
- **Access Pattern:** Can only access entities where they are personally a stakeholder

### BASIC Persona
- **Email:** `basic@propwise-test.com`
- **Org Role:** Basic
- **Team:** None
- **Permissions:** No permissions
- **Access Pattern:** Relies entirely on stakeholder access, not a team member

<Warning>
The AGENT and BASIC personas are specifically designed to test access level differentiation (READ vs WRITE vs FULL) and stakeholder-based permissions.
</Warning>

## Permission Profiles

### Organization-Level Permissions

<Tabs>
  <Tab title="OWNER / ADMIN">
    These personas bypass resource-level checks through org permissions:
    
    - `crm.view` and `crm.manage` for lead and contact access
    - `sales.view` and `sales.manage` for deal access
    - Can access any entity in the organization
    - No stakeholder assignment required
  </Tab>
  
  <Tab title="Team Roles">
    **TEAM_LEADER** has team-specific permissions:
    
    - `team_crm.manage` for team lead/contact management
    - `team_sales.manage` for team deal management
    - `team.admin` for team administration
    - Access limited to entities where team is stakeholder
  </Tab>
</Tabs>

### Stakeholder-Based Access

<AccordionGroup>
  <Accordion title="Access Levels">
    The system supports three stakeholder access levels:
    
    - **READ:** View-only access to the entity
    - **WRITE:** View and modify access
    - **FULL:** Complete access including delete operations
  </Accordion>
  
  <Accordion title="Resource Actions">
    Access level hierarchy for resource actions:
    
    - `ACCESS` — granted by READ, WRITE, or FULL
    - `MODIFY` — granted by WRITE or FULL
    - `DELETE` — granted by FULL only
  </Accordion>
</AccordionGroup>

## Authentication Flow

The test framework performs a two-step authentication process for each persona:

<Steps>
  <Step title="Identity Authentication">
    ```bash
    POST /v1/auth/login
    ```
    
    Returns an identity token for the user
  </Step>
  
  <Step title="Organization Selection">
    ```bash
    POST /v1/auth/select-organization
    ```
    
    Returns a tenant token (org-scoped JWT)
  </Step>
  
  <Step title="API Authorization">
    All subsequent API calls use the tenant token:
    
    ```
    Authorization: Bearer <tenant-token>
    ```
  </Step>
</Steps>

## Test Data Structure

### Entity Creation Patterns

Each persona creates specific test entities to verify different access scenarios:

<Tabs>
  <Tab title="Leads">
    - **BASIC-created lead:** Tests stakeholder-based access
    - **OWNER-created lead:** Tests admin-only scenarios
    - **Team-assigned lead:** Tests team permission inheritance
    - **Agent READ lead:** Tests access level enforcement
    - **Pool lead:** Tests entities without stakeholders
  </Tab>
  
  <Tab title="Other Entities">
    - **Contacts:** Created by BASIC and OWNER for different access patterns
    - **Companies:** Created by BASIC and OWNER for permission testing
    - **Deals:** Created by OWNER to test sales permissions
  </Tab>
</Tabs>

<Info>
Test data is created through actual API endpoints, ensuring all business logic (guards, auto-assignment, stakeholder creation) executes naturally.
</Info>

## Access Mode Testing

The framework tests both access modes supported by the `@CheckAccess` decorator:

### OR Mode (Default)
User has the required permission **OR** stakeholder-level access → allowed

### AND Mode
User must have the permission **AND** stakeholder access → allowed

<Tip>
The matrix-driven test approach automatically generates test cases for all persona and endpoint combinations, ensuring comprehensive coverage of permission scenarios.
</Tip>

## Usage in Tests

To use the RBAC personas in your test files:

```typescript
import { getContext, closeContext, RbacTestContext } from '../../setup/rbac-context';

let ctx: RbacTestContext;

beforeAll(async () => {
  ctx = await getContext();
}, 60000);

afterAll(async () => {
  await closeContext();
});

// Access personas in tests
const { personas, teamId } = ctx;
const ownerToken = personas.OWNER.token;
```

The singleton context ensures the NestJS app boots once and authentication tokens are shared across all test files for optimal performance.
