## Overview

The RBAC testing framework uses five predefined personas to verify that every API endpoint enforces the correct permissions for each user role. All personas exist within a single test organization and share standardized credentials and role assignments.

<Note>
All test personas use the `@propwise.com` domain and are seeded once before running the test suite.
</Note>

## Persona Definitions

Test personas are defined in `test/helpers/rbac-personas.ts` and represent the complete spectrum of organizational roles and permission levels.

<CardGroup cols={2}>
  <Card title="OWNER" icon="crown">
    Full system access with all permissions including `system.owner`
  </Card>
  <Card title="ADMIN" icon="user-shield">
    Administrative access with all non-sensitive permissions
  </Card>
  <Card title="TEAM_LEADER" icon="users">
    Team-level management with team CRM and sales permissions
  </Card>
  <Card title="AGENT" icon="user">
    Basic team member with no default permissions
  </Card>
  <Card title="BASIC" icon="user-circle">
    Minimal user relying entirely on stakeholder access
  </Card>
</CardGroup>

## Permission Profiles

### OWNER

<Accordion title="View full profile">
**Email:** `owner@propwise.com`

**Organization Role:** Owner

**Team Membership:** Any

**Permissions:**
- `system.owner` — Full system control
- All organization-level permissions (`crm.*`, `sales.*`, `admin.*`)
- Bypasses resource-level checks via org permissions
- Can access any entity in the organization

**Use Cases:**
- Testing top-level administrative operations
- Verifying owner-only sensitive actions
- Baseline for "full access" scenarios
</Accordion>

### ADMIN

<Accordion title="View full profile">
**Email:** `admin@propwise.com`

**Organization Role:** Admin

**Team Membership:** Any

**Permissions:**
- `system.admin` — System administration (non-sensitive)
- All CRM and sales permissions (`crm.view`, `crm.manage`, `sales.view`, `sales.manage`)
- Bypasses resource-level checks for standard operations
- Cannot perform owner-only sensitive operations

**Use Cases:**
- Testing administrative workflows
- Verifying permission boundaries below owner level
- Standard "admin can do everything" scenarios
</Accordion>

### TEAM_LEADER

<Accordion title="View full profile">
**Email:** `teamleader@propwise.com`

**Organization Role:** Basic

**Team Membership:** Team Leader in "Test Sales Team"

**Permissions:**
- No organization-level permissions
- `team_crm.manage` — Full CRM access for team entities
- `team_sales.manage` — Full sales access for team entities
- `team.admin` — Team administration
- Can access entities where the team is a stakeholder

**Use Cases:**
- Testing team-scoped permission enforcement
- Verifying team stakeholder access
- Team management operations
</Accordion>

### AGENT

<Accordion title="View full profile">
**Email:** `agent@propwise.com`

**Organization Role:** Basic

**Team Membership:** Agent in "Test Sales Team"

**Permissions:**
- No organization-level permissions
- No team-level permissions by default
- Can only access entities where personally added as stakeholder
- Access level (READ/WRITE/FULL) determines allowed operations

**Use Cases:**
- Testing access level differentiation (READ vs WRITE vs FULL)
- Verifying personal stakeholder enforcement
- Zero-permission baseline scenarios
</Accordion>

### BASIC

<Accordion title="View full profile">
**Email:** `basic@propwise.com`

**Organization Role:** Basic

**Team Membership:** None

**Permissions:**
- No organization-level permissions
- No team membership
- Pure stakeholder-based access only
- Must be explicitly added to entities to gain access

**Use Cases:**
- Testing non-team-member scenarios
- Verifying stakeholder-only access patterns
- Minimal permission baseline
</Accordion>

## Authentication Flow

<Steps>
  <Step title="Initial login">
    Each persona authenticates via `POST /v1/auth/login` to obtain an identity token.
    
    ```typescript
    const loginResponse = await request(app.getHttpServer())
      .post('/v1/auth/login')
      .send({
        email: 'owner@propwise.com',
        password: 'TestPassword123!',
      });
    
    const identityToken = loginResponse.body.access_token;
    ```
  </Step>

  <Step title="Organization selection">
    The identity token is exchanged for a tenant token via `POST /v1/auth/select-organization`.
    
    ```typescript
    const selectOrgResponse = await request(app.getHttpServer())
      .post('/v1/auth/select-organization')
      .set('Authorization', `Bearer ${identityToken}`)
      .send({
        organizationId: testOrgId,
      });
    
    const tenantToken = selectOrgResponse.body.access_token;
    ```
  </Step>

  <Step title="Authenticated requests">
    All subsequent API calls use the tenant token (org-scoped JWT) in the Authorization header.
    
    ```typescript
    await request(app.getHttpServer())
      .get('/v1/leads')
      .set('Authorization', `Bearer ${tenantToken}`)
      .expect(200);
    ```
  </Step>
</Steps>

<Info>
The RBAC context (`rbac-context.ts`) performs this two-step login automatically for all five personas during test setup.
</Info>

## Test Organization Structure

All personas belong to **"Test Organization"** with the following configuration:

<AccordionGroup>
  <Accordion title="Organization setup">
    - **Name:** Test Organization
    - **Owner:** OWNER persona
    - **Subscription:** Active BUSINESS plan
    - **Setup Status:** Completed (bypasses onboarding gates)
    - **Members:** All 5 personas with appropriate org roles
  </Accordion>

  <Accordion title="Team structure">
    **Test Sales Team:**
    - **Team Leader:** TEAM_LEADER persona
    - **Agent:** AGENT persona

## Access Level Reference

When personas are added as stakeholders to entities, their access is governed by three levels:

<CardGroup cols={3}>
  <Card title="READ" icon="eye">
    **Grants:** View entity details
    
    **Denies:** Modifications, deletions
  </Card>
  <Card title="WRITE" icon="pen">
    **Grants:** View and modify entity
    
    **Denies:** Deletion
  </Card>
  <Card title="FULL" icon="crown">
    **Grants:** View, modify, delete, manage stakeholders
    
    **Denies:** Nothing
  </Card>
</CardGroup>

<Tip>
AGENT and BASIC personas are commonly used to test access level differentiation since they have no fallback org or team permissions.
</Tip>

## Seeding Test Users

Before running tests for the first time, seed the personas into your database:

<Steps>
  <Step title="Initialize subscription data">
    ```bash
    pnpm test:init-data
    ```
    
    Creates the required subscription plans (FREE, BUSINESS, etc.).
  </Step>

  <Step title="Seed RBAC personas">
    ```bash
    pnpm test:seed-users
    ```
    
    This script (`test/setup/seed-test-users.ts`):
    - Creates all 5 test personas (idempotent)
    - Creates Test Organization with OWNER
    - Activates BUSINESS subscription
    - Completes organization setup progress
    - Creates Test Sales Team with TEAM_LEADER and AGENT
    - Assigns all org and team roles
  </Step>

  <Step title="Verify seeding">
    Check your database for users with `@propwise.com` emails and verify the Test Organization exists.
  </Step>
</Steps>

<Warning>
Seeding uses RLS bypass to write directly via MikroORM. Run `pnpm test:cleanup-users` to remove test personas if needed.
</Warning>

## Using Personas in Tests

The singleton RBAC context provides authenticated tokens for all personas:

```typescript
import { getContext, closeContext, RbacTestContext } from '../../setup/rbac-context';

let ctx: RbacTestContext;

beforeAll(async () => {
  ctx = await getContext();
}, 60000);

afterAll(async () => {
  await closeContext();
});

it('allows OWNER to view lead', async () => {
  const response = await request(ctx.app.getHttpServer())
    .get('/v1/leads')
    .set('Authorization', `Bearer ${ctx.personas.OWNER.token}`)
    .expect(200);
});

it('denies BASIC access to admin endpoint', async () => {
  await request(ctx.app.getHttpServer())
    .delete(`/v1/leads/${leadId}`)
    .set('Authorization', `Bearer ${ctx.personas.BASIC.token}`)
    .expect(403);
});
```
