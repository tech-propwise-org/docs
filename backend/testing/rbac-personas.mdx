## Overview

The PropWise CRM backend test suite uses **five predefined personas** to verify Role-Based Access Control (RBAC) enforcement across all API endpoints. Each persona represents a different permission profile, from system owner to basic user with no inherited permissions.

<Info>
All personas share a single test organization ("Test Organization") and are seeded once before running the test suite. They authenticate via the standard login flow and use tenant tokens for all API requests.
</Info>

---

## Persona Definitions

All personas are defined in `test/helpers/rbac-personas.ts` as a single source of truth.

## Persona Details

### OWNER

```typescript
{
  email: 'owner@propwise-test.com',
  orgRole: 'Owner',
  teamRole: null,
  permissions: ['system.owner', 'crm.*', 'sales.*', ...]
}
```

**Access Profile:**
- Has `system.owner` permission
- Bypasses resource-level checks via org-level permissions
- Can access any entity in the organization
- Can perform admin-only operations (delete, bulk operations)

**Use Case:** Tests that verify owner-specific functionality and ensure admin operations are properly gated.

---

### ADMIN

```typescript
{
  email: 'admin@propwise-test.com',
  orgRole: 'Admin',
  teamRole: null,
  permissions: ['system.admin', 'crm.*', 'sales.*', ...]
}
```

**Access Profile:**
- Has `system.admin` permission
- All org-level CRM and sales permissions
- **Cannot** perform sensitive operations requiring `system.owner`
- Can access any entity in the organization

**Use Case:** Tests that verify admin vs owner permission boundaries and general organization-wide access.

---

### TEAM_LEADER

```typescript
{
  email: 'teamleader@propwise-test.com',
  orgRole: 'Basic',
  teamRole: 'Team Leader',
  teamName: 'Test Sales Team',
  permissions: ['team_crm.*', 'team_sales.*', 'team.admin']
}
```

**Access Profile:**
- **No** org-level permissions
- Has team-level permissions via "Test Sales Team" role
- Can access entities where the team is a stakeholder
- Can manage team members and team settings

**Use Case:** Tests that verify team-scoped permission enforcement and team stakeholder access.

<Warning>
TEAM_LEADER cannot access entities owned by other teams or entities in the pool unless explicitly added as a stakeholder.
</Warning>

---

### AGENT

```typescript
{
  email: 'agent@propwise-test.com',
  orgRole: 'Basic',
  teamRole: 'Agent',
  teamName: 'Test Sales Team',
  permissions: []
}
```

**Access Profile:**
- **No** org-level permissions
- **No** team-level permissions (role exists but grants no permissions by default)
- Can **only** access entities where personally added as a stakeholder
- Access depends on stakeholder access level (READ, WRITE, FULL)

**Use Case:** Tests that verify access level differentiation:
- READ → can view but not modify
- WRITE → can view and modify but not delete
- FULL → can view, modify, and delete

---

### BASIC

```typescript
{
  email: 'basic@propwise-test.com',
  orgRole: 'Basic',
  teamRole: null,
  permissions: []
}
```

**Access Profile:**
- **No** org-level permissions
- **Not** a member of any team
- Can **only** access entities where personally added as a stakeholder
- Access depends on stakeholder access level (READ, WRITE, FULL)

**Use Case:** Tests that verify pure stakeholder-based access without any team membership benefits.

<Note>
BASIC is functionally similar to AGENT but useful for testing scenarios where team membership itself should not grant any implicit access.
</Note>

## Permission Profiles

### Org-Level Permissions (OWNER, ADMIN)

These personas bypass resource-level checks via organization-wide permissions:

| Permission    | Description                              | OWNER | ADMIN |
|---------------|------------------------------------------|-------|-------|
| `system.owner` | Sensitive operations (delete, etc.)     | ✅    | ❌    |
| `system.admin` | Admin operations                        | ✅    | ✅    |
| `crm.view`     | View all CRM entities in org           | ✅    | ✅    |
| `crm.manage`   | Modify all CRM entities in org         | ✅    | ✅    |
| `sales.view`   | View all sales entities in org         | ✅    | ✅    |
| `sales.manage` | Modify all sales entities in org       | ✅    | ✅    |

<Check>
When a user has `crm.view` or `crm.manage`, the `@CheckAccess` guard allows access **without** checking stakeholder relationships.
</Check>

---

### Team-Level Permissions (TEAM_LEADER)

Team Leader has team-scoped permissions but **no** org-level access:

| Permission         | Description                              |
|--------------------|------------------------------------------|
| `team_crm.view`    | View CRM entities where team is stakeholder |
| `team_crm.manage`  | Modify CRM entities where team is stakeholder |
| `team_sales.view`  | View sales entities where team is stakeholder |
| `team_sales.manage`| Modify sales entities where team is stakeholder |
| `team.admin`       | Manage team settings and members         |

<Info>
Team permissions work in OR mode with stakeholder access. If the team is a stakeholder on a lead, TEAM_LEADER can access it via `team_crm.*` permissions.
</Info>

---

### Stakeholder-Based Access (AGENT, BASIC)

These personas have **no inherited permissions**. Access is granted **only** through explicit stakeholder relationships:

<Steps>
  <Step title="Entity is created">
    Creator is automatically assigned as primary stakeholder with FULL access level.
  </Step>
  
  <Step title="Stakeholder is added">
    Another user/team is added as a stakeholder with an access level: READ, WRITE, or FULL.
  </Step>
  
  <Step title="Access is evaluated">
    The `@CheckAccess` guard checks if the user has:
    - Required permission (e.g., `crm.view`) **OR**
    - Required stakeholder access level on the resource
  </Step>
</Steps>

#### Access Level Hierarchy

| Access Level | Can View | Can Modify | Can Delete | Resource Actions Granted |
|--------------|----------|------------|------------|--------------------------|
| **READ**     | ✅       | ❌         | ❌         | `ACCESS`                 |
| **WRITE**    | ✅       | ✅         | ❌         | `ACCESS`, `MODIFY`       |
| **FULL**     | ✅       | ✅         | ✅         | `ACCESS`, `MODIFY`, `DELETE` |

<Tip>
The guard maps these levels to resource actions:
- `ACCESS` → granted by READ, WRITE, or FULL
- `MODIFY` → granted by WRITE or FULL
- `DELETE` → granted by FULL only
</Tip>

## Authentication Flow

All personas follow the same two-step login process:

<Steps>
  <Step title="Identity login">
    ```bash
    POST /v1/auth/login
    Body: { email, password }
    Returns: { accessToken: "identity-jwt" }
    ```
    The identity token contains user ID and email but no org/team context.
  </Step>
  
  <Step title="Organization selection">
    ```bash
    POST /v1/auth/select-organization
    Headers: { Authorization: "Bearer identity-jwt" }
    Body: { organizationId }
    Returns: { accessToken: "tenant-jwt" }
    ```
    The tenant token contains org, team, role, and permission information.
  </Step>
  
  <Step title="API requests">
    All subsequent API calls use the tenant token:
    ```bash
    GET /v1/leads
    Headers: { Authorization: "Bearer tenant-jwt" }
    ```
  </Step>
</Steps>

<Warning>
Tests use the **tenant token** for all API requests. The identity token is discarded after org selection.
</Warning>

## Seeding Personas

### One-Time Setup

Before running RBAC tests for the first time, seed the personas:

```bash
pnpm test:seed-users
```

This script (`test/setup/seed-test-users.ts`):

<AccordionGroup>
  <Accordion title="Creates test organization">
    - Name: "Test Organization"
    - Owner: `owner@propwise-test.com`
    - Subscription: **ACTIVE BUSINESS** plan
    - Setup progress: **COMPLETED** (skips onboarding)
  </Accordion>
  
  <Accordion title="Creates all personas">
    - Uses RLS bypass to write directly via MikroORM
    - Assigns org roles (Owner, Admin, Basic)
    - Sets verified status and initial passwords
  </Accordion>
  
  <Accordion title="Creates test team">
    - Name: "Test Sales Team"
    - Team Leader: `teamleader@propwise-test.com`
    - Agent: `agent@propwise-test.com`
    - Assigns team roles and permissions
  </Accordion>
  
  <Accordion title="Ensures idempotency">
    - Skips creation if personas already exist
    - Self-heals stale subscription data from older seeds
    - Safe to run multiple times
  </Accordion>
</AccordionGroup>

<Check>
Requires subscription plans to exist. Run `pnpm test:init-data` once on a fresh database before seeding users.
</Check>

## Test Data Creation

Each test suite creates fixture entities through the API (not direct DB writes) so all business logic executes naturally:

```typescript
import { seedRbacTestData } from '../../setup/rbac-test-data';

beforeAll(async () => {
  const { leads, deals, contacts, companies } = 
    await seedRbacTestData(app, personas, teamId);
});
```

### Fixture Entities Created

<Tabs>
  <Tab title="Leads">
    - **By BASIC** — auto-assigned as primary stakeholder with FULL access
    - **By OWNER** — for admin-only operation tests
    - **In pool** — no stakeholders (only OWNER/ADMIN can access)
    - **With team** — team assigned as stakeholder (TEAM_LEADER gets access)
    - **With AGENT (READ)** — AGENT added with READ access level
  </Tab>
  
  <Tab title="Deals">
    - **By OWNER** — OWNER is primary stakeholder
    - Used to test sales permissions (`sales.*` vs `crm.*`)
  </Tab>
  
  <Tab title="Contacts">
    - **By BASIC** — BASIC is creator
    - **By OWNER** — OWNER is creator
    - Tests stakeholder access inheritance
  </Tab>
  
  <Tab title="Companies">
    - **By BASIC** — BASIC is creator
    - **By OWNER** — OWNER is creator
    - Tests stakeholder access inheritance
  </Tab>
</Tabs>

<Info>
Test data is **cached** across test files. The first test suite to call `seedRbacTestData()` creates the entities; subsequent calls return the cached IDs instantly.
</Info>

## Usage in Tests

### Basic Pattern

```typescript
import { getContext, closeContext, RbacTestContext } from '../../setup/rbac-context';
import { seedRbacTestData } from '../../setup/rbac-test-data';

let ctx: RbacTestContext;
let testData: RbacTestData;

beforeAll(async () => {
  ctx = await getContext();
  testData = await seedRbacTestData(ctx.app, ctx.personas, ctx.teamId);
}, 60000);

afterAll(async () => {
  await closeContext();
});

it('OWNER can view lead by BASIC', async () => {
  const response = await request(ctx.app.getHttpServer())
    .get(`/v1/leads/${testData.leads.byBasic}`)
    .set('Authorization', `Bearer ${ctx.personas.OWNER.tenantToken}`);
  
  expect(response.status).toBe(200);
});
```

### Matrix-Driven Pattern

Instead of writing individual tests, define expectations declaratively:

```typescript
import { runRbacMatrix } from '../../setup/rbac-runner';
import { LEAD_BY_BASIC_MATRIX } from '../../setup/matrices/lead-matrix';

describe('Lead RBAC - Lead by BASIC', () => {
  runRbacMatrix(LEAD_BY_BASIC_MATRIX, () => ctx, () => testData);
});
```

The runner automatically generates 5 tests per endpoint (one for each persona) and produces clear failure messages with actual response bodies.

## Cleanup

To remove all test personas from the database:

```bash
pnpm test:cleanup-users
```

This script (`test/setup/cleanup-test-users.ts`):
- Deletes all users with `@propwise-test.com` emails
- Cascades to delete their org memberships, team memberships, and created entities
- Useful for resetting test state or cleaning up after failed runs

<Warning>
This is a destructive operation. Only run against test databases, never production.
</Warning>
