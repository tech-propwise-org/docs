The test suite focuses on **RBAC (Role-Based Access Control) E2E testing** — verifying that every API endpoint enforces the correct permissions for every user role.

**Stack:** Jest + supertest + `@nestjs/testing`

**Key design decisions:**
- **Singleton app context** — the NestJS app boots once and is shared across all test files
- **Matrix-driven tests** — a declarative permission matrix defines expected HTTP status codes per persona × endpoint
- **API-driven seeding** — test data is created through actual API endpoints so all business logic executes naturally

## Architecture

### Singleton Context

The app boots once for the entire test run. All test files share the same NestJS app instance, authenticated tokens, and resolved `teamId`.

<Info>
The singleton pattern significantly reduces test execution time by avoiding repeated application bootstrapping (3-5 seconds per file).
</Info>

**Execution Flow:**

<Steps>
<Step title="Jest starts">
First test file calls `getContext()` which boots the NestJS app and authenticates all personas
</Step>
<Step title="Subsequent files">
Other test files call `getContext()` and receive the cached instance instantly
</Step>
<Step title="Cleanup">
Last file's `afterAll()` calls `closeContext()` when reference count hits 0
</Step>
</Steps>

**Implementation:**

```typescript
import { getContext, closeContext, RbacTestContext } from '../../setup/rbac-context';

let ctx: RbacTestContext;
beforeAll(async () => {
  ctx = await getContext();
}, 60000);
afterAll(async () => {
  await closeContext();
});
```

The context replicates production config from `main.ts`:
- `ValidationPipe` (whitelist, transform, implicit conversion)
- URI versioning (`/v1/...`)
- Cookie parser

### Matrix Runner

Instead of writing individual `it()` blocks, you define expectations declaratively:

```typescript
{
  method: 'GET',
  path: '/v1/leads/:leadByBasicId',
  description: 'View lead detail',
  expectations: {
    OWNER: 200,
    ADMIN: 200,
    TEAM_LEADER: 403,
    AGENT: 403,
    BASIC: 200,
  },
}
```

<Note>
One matrix entry generates 5 test cases automatically - one for each persona.
</Note>

The runner calls `runRbacMatrix()` which:

1. Iterates every endpoint in the matrix
2. For each endpoint, iterates all 5 personas
3. Resolves path placeholders (`:leadByBasicId` → actual UUID)
4. Makes the HTTP request with the persona's tenant token
5. Asserts the response status matches the expected value
6. Produces clear failure messages with the actual response body

## RBAC Personas

<CardGroup cols={2}>
<Card title="Administrative Roles" icon="crown">
**OWNER** - All permissions including `system.owner`
**ADMIN** - All non-sensitive permissions (`system.admin`)
</Card>
<Card title="Team-Based Roles" icon="users">
**TEAM_LEADER** - Team permissions in Test Sales Team
**AGENT** - Team member with no special permissions
**BASIC** - No team membership, stakeholder-only access
</Card>
</CardGroup>

### Persona Details

| Key | Email | Org Role | Team | Description |
|-----|-------|----------|------|-------------|
| `OWNER` | owner@propwise.com | Owner | Any | All permissions including `system.owner` |
| `ADMIN` | admin@propwise.com | Admin | Any | All non-sensitive permissions |
| `TEAM_LEADER` | teamleader@propwise.com | Basic | Team Leader | Team permissions (`team_crm.*`, `team_sales.*`) |
| `AGENT` | agent@propwise.com | Basic | Agent | No team permissions by default |
| `BASIC` | basic@propwise.com | Basic | None | Stakeholder access only |

### Permission Profiles

<AccordionGroup>
<Accordion title="OWNER / ADMIN">
Bypass resource-level checks via org permissions (`crm.view`, `crm.manage`, `sales.view`, `sales.manage`, etc.). Can access any entity in the org.
</Accordion>
<Accordion title="TEAM_LEADER">
No org permissions, but has `team_crm.manage` and `team_sales.manage` in the team. Can access entities where the team is a stakeholder.
</Accordion>
<Accordion title="AGENT">
No org permissions, no team permissions. Can only access entities where they are personally a stakeholder. Used to test access level differentiation.
</Accordion>
<Accordion title="BASIC">
Same as AGENT but not a team member at all. Access is purely stakeholder-based.
</Accordion>
</AccordionGroup>

### Login Flow

The context performs a two-step login for each persona:

<Steps>
<Step title="Identity Login">
`POST /v1/auth/login` → identity token
</Step>
<Step title="Organization Selection">
`POST /v1/auth/select-organization` → tenant token (org-scoped JWT)
</Step>
</Steps>

<Warning>
All subsequent API calls must use the tenant token in the `Authorization: Bearer` header.
</Warning>

## Permission Matrix

Matrices are split into per-entity files under `test/setup/matrices/`. The barrel at `test/setup/rbac-matrix.ts` re-exports everything.

### Lead Matrices

| Matrix | Entity | Who Can Access | Tests |
|--------|--------|----------------|-------|
| `LEAD_BY_BASIC_MATRIX` | Lead created by BASIC | OWNER, ADMIN (via `crm.*`), BASIC (via FULL stakeholder) | View, update, stakeholders, activities, property interests |
| `LEAD_ON_TEAM_MATRIX` | Lead with team as stakeholder | OWNER, ADMIN, TEAM_LEADER (via `team_crm.*`) | View, update, stakeholders, activities |
| `LEAD_AGENT_READ_MATRIX` | Lead where AGENT has READ access | OWNER, ADMIN, AGENT (view only — modify denied) | View allowed, update denied, access level enforcement |
| `LEAD_IN_POOL_MATRIX` | Lead with no stakeholders | OWNER, ADMIN only (via `crm.*`) | View, update |
| `LEAD_ADMIN_ONLY_MATRIX` | Lead for admin-only ops | OWNER, ADMIN (via `system.admin`) | Delete |

### Deal Matrices

| Matrix | Entity | Who Can Access |
|--------|--------|----------------|
| `DEAL_BY_OWNER_MATRIX` | Deal created by OWNER | OWNER, ADMIN only (via `sales.*`) |

### Contact Matrices

| Matrix | Entity | Who Can Access |
|--------|--------|----------------|
| `CONTACT_BY_BASIC_MATRIX` | Contact created by BASIC | OWNER, ADMIN, BASIC |
| `CONTACT_BY_OWNER_MATRIX` | Contact created by OWNER | OWNER, ADMIN only |

### Company Matrices

| Matrix | Entity | Who Can Access |
|--------|--------|----------------|
| `COMPANY_BY_BASIC_MATRIX` | Company created by BASIC | OWNER, ADMIN, BASIC |
| `COMPANY_BY_OWNER_MATRIX` | Company created by OWNER | OWNER, ADMIN only |

### Access Mode Reference

Your `@CheckAccess` decorator uses two modes:

<Tabs>
<Tab title="OR Mode (Default)">
User has the required permission **OR** has stakeholder-level access on the resource → allowed.
</Tab>
<Tab title="AND Mode">
User must have the permission **AND** stakeholder access → allowed.
</Tab>
</Tabs>

**Resource Action Hierarchy:**
- `ACCESS` — granted by `READ`, `WRITE`, or `FULL` access level
- `MODIFY` — granted by `WRITE` or `FULL`
- `DELETE` — granted by `FULL` only

### Step 2: Seed Test Data (Automatic)

Each test file calls `seedRbacTestData(app, personas, teamId)` in its `beforeAll`. Results are cached across files.

**Created Entities:**

<AccordionGroup>
<Accordion title="Leads">
- One by BASIC (auto-assigned as primary stakeholder with FULL access)
- One by OWNER (for admin-only tests)
- One in pool (empty stakeholders)
- One with team as stakeholder (OWNER creates, assigns team)
- One with AGENT as READ-only stakeholder
</Accordion>
<Accordion title="Contacts & Companies">
- One contact by BASIC, one by OWNER
- One company by BASIC, one by OWNER
</Accordion>
<Accordion title="Deals & Events">
- One deal by OWNER (from lead + contact)
- One event linked to OWNER lead, one with BASIC as invitee
</Accordion>
<Accordion title="Escalation Rules">
- One OWNER-created rule for escalation-rule RBAC matrices
</Accordion>
</AccordionGroup>

<Note>
Because seeding goes through the real API endpoints, all business logic (guards, auto-assignment, stakeholder creation) executes naturally.
</Note>

## How to Run

<Steps>
<Step title="One-Time Setup">
Seed the test users in your database:
```bash
pnpm test:seed-users
```
</Step>

<Step title="Run Core Test Suite">
Execute the main RBAC tests (excludes queue-related tests):
```bash
pnpm test:e2e
```
</Step>

<Step title="Run Queue Tests (Optional)">
Execute tests that require live pg-boss queues:
```bash
pnpm test:e2e-queue
```
</Step>

<Step title="Cleanup (Optional)">
Remove test users from database:
```bash
pnpm test:cleanup-users
```
</Step>
</Steps>

<Warning>
Make sure your test database is properly configured before running the test suite.
</Warning>

## How to Extend

### Adding New Endpoints

<Steps>
<Step title="Define Matrix">
Add endpoint expectations to the appropriate matrix file:

```typescript
{
  method: 'POST',
  path: '/v1/leads/:leadByBasicId/notes',
  description: 'Create lead note',
  expectations: {
    OWNER: 201,
    ADMIN: 201,
    TEAM_LEADER: 403,
    AGENT: 403,
    BASIC: 201,
  },
}
```
</Step>

<Step title="Update Test File">
Import the updated matrix and run it in your test file:

```typescript
import { LEAD_BY_BASIC_MATRIX } from '../../setup/matrices/lead-matrix';
import { runRbacMatrix } from '../../setup/rbac-runner';

describe('Lead RBAC', () => {
  // ... existing setup

  describe('Lead by BASIC matrices', () => {
    runRbacMatrix(LEAD_BY_BASIC_MATRIX, ctx);
  });
});
```
</Step>
</Steps>

### Adding New Personas

<Steps>

</Steps>

## Methodology

<Tip>
Refer to `TESTING_METHODOLOGY.md` for detailed guidance on analyzing controllers and DTOs before writing tests.
</Tip>

The testing methodology follows these principles:

1. **Permission-First Design** — Start by understanding the required permissions for each endpoint
2. **Stakeholder Access Patterns** — Map out how different access levels (READ/WRITE/FULL) affect operations
3. **Matrix Completeness** — Ensure every persona × endpoint combination is explicitly tested
4. **Real API Usage** — Use actual API endpoints for seeding to exercise all business logic

## CI/CD Integration

<Info>
The test suite is designed to run in CI/CD pipelines with proper database isolation and cleanup.
</Info>

**Pipeline Considerations:**

- **Database State** — Tests assume a clean database with only seeded test data
- **Timing** — Initial app bootstrap takes 3-5 seconds; subsequent tests are fast
- **Isolation** — Each test run should use a separate database or proper cleanup between runs
- **Queue Tests** — Queue-related tests require additional setup and should run separately

**Recommended CI Configuration:**

```yaml
test-e2e:
  steps:
    - name: Setup test database
      run: pnpm db:test:setup
    
    - name: Seed test users
      run: pnpm test:seed-users
    
    - name: Run core E2E tests
      run: pnpm test:e2e
    
    - name: Run queue E2E tests
      run: pnpm test:e2e-queue
    
    - name: Cleanup
      run: pnpm test:cleanup-users
```
