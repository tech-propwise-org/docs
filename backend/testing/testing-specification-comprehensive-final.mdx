# Testing Specification - Comprehensive Final

Architecture, conventions, and run guide for the PropWise CRM backend test suite.

<Note>
This test suite focuses on **RBAC (Role-Based Access Control) E2E testing** — verifying that every API endpoint enforces the correct permissions for every user role.
</Note>

## Overview

The test suite uses Jest + supertest + `@nestjs/testing` with three key design decisions:

- **Singleton app context** — the NestJS app boots once and is shared across all test files
- **Matrix-driven tests** — a declarative permission matrix defines expected HTTP status codes per persona × endpoint
- **API-driven seeding** — test data is created through actual API endpoints so all business logic executes naturally

## Architecture

### Singleton Context

<Info>
The app boots once for the entire test run, equivalent to NestJS `TestSuite` pattern but implemented manually since `TestSuite` requires NestJS 11+.
</Info>

```typescript
import { getContext, closeContext, RbacTestContext } from '../../setup/rbac-context';

let ctx: RbacTestContext;
beforeAll(async () => {
  ctx = await getContext();
}, 60000);
afterAll(async () => {
  await closeContext();
});
```

The context flow works as follows:

<Steps>
<Step title="First test file calls getContext()">
Boots NestJS app (~3-5s), logs in 5 personas, resolves teamId
</Step>
<Step title="Subsequent test files call getContext()">
Returns cached instance instantly (0ms)
</Step>
<Step title="Last file's afterAll() calls closeContext()">
Reference count hits 0 → app shuts down
</Step>
</Steps>

### Matrix Runner

Instead of writing individual `it()` blocks, you define expectations declaratively:

```typescript
{
  method: 'GET',
  path: '/v1/leads/:leadByBasicId',
  description: 'View lead detail',
  expectations: {
    OWNER: 200,
    ADMIN: 200,
    TEAM_LEADER: 403,
    AGENT: 403,
    BASIC: 200,
  },
}
```

The runner:
1. Iterates every endpoint in the matrix
2. For each endpoint, iterates all 5 personas
3. Resolves path placeholders (`:leadByBasicId` → actual UUID)
4. Makes the HTTP request with the persona's tenant token
5. Asserts the response status matches the expected value

<Tip>
One matrix entry generates 5 test cases automatically.
</Tip>

### Permission Profiles

<AccordionGroup>
<Accordion title="OWNER / ADMIN">
Bypass resource-level checks via org permissions (`crm.view`, `crm.manage`, `sales.view`, `sales.manage`, etc.). Can access any entity in the org.
</Accordion>
<Accordion title="TEAM_LEADER">
No org permissions, but has `team_crm.manage` and `team_sales.manage` in the team. Can access entities where the team is a stakeholder.
</Accordion>
<Accordion title="AGENT">
No org permissions, no team permissions. Can only access entities where they are personally a stakeholder. Used to test access level differentiation (READ vs WRITE vs FULL).
</Accordion>
<Accordion title="BASIC">
Same as AGENT but not a team member at all. Access is purely stakeholder-based.
</Accordion>
</AccordionGroup>

### Login Flow

<Steps>
<Step title="Identity Login">
`POST /v1/auth/login` → identity token
</Step>
<Step title="Organization Selection">
`POST /v1/auth/select-organization` → tenant token (org-scoped JWT)
</Step>
<Step title="API Requests">
All subsequent API calls use the tenant token in the `Authorization: Bearer` header
</Step>
</Steps>

## Permission Matrix

Matrices are split into per-entity files under `test/setup/matrices/`.

### Lead Matrices

| Matrix | Entity | Who Can Access | Tests |
|--------|--------|----------------|-------|
| `LEAD_BY_BASIC_MATRIX` | Lead created by BASIC | OWNER, ADMIN (via `crm.*`), BASIC (via FULL stakeholder) | View, update, stakeholders, activities, property interests |
| `LEAD_ON_TEAM_MATRIX` | Lead with team as stakeholder | OWNER, ADMIN, TEAM_LEADER (via `team_crm.*`) | View, update, stakeholders, activities |
| `LEAD_AGENT_READ_MATRIX` | Lead where AGENT has READ access | OWNER, ADMIN, AGENT (view only — modify denied) | View allowed, update denied, access level enforcement |
| `LEAD_IN_POOL_MATRIX` | Lead with no stakeholders | OWNER, ADMIN only (via `crm.*`) | View, update |
| `LEAD_ADMIN_ONLY_MATRIX` | Lead for admin-only ops | OWNER, ADMIN (via `system.admin`) | Delete |

### Access Mode Reference

<Info>
Your `@CheckAccess` decorator uses two modes:
- **OR (default):** User has the required permission **OR** has stakeholder-level access
- **AND:** User must have the permission **AND** stakeholder access
</Info>

Resource action hierarchy:
- `ACCESS` — granted by `READ`, `WRITE`, or `FULL` access level
- `MODIFY` — granted by `WRITE` or `FULL`
- `DELETE` — granted by `FULL` only

## Test Data Seeding

### Step 1: Seed Users (One-time Setup)

<Warning>
Run this once before running tests. Requires subscription plans to exist.
</Warning>

```bash
pnpm test:seed-users
```

This creates:
- RBAC test personas in the database (idempotent)
- "Test Organization" with owner
- **ACTIVE BUSINESS subscription** and completed setup progress
- "Test Sales Team" with team leader and agent
- Proper org and team role assignments

### Step 2: Seed Test Data (Automatic)

Each test file calls `seedRbacTestData(app, personas, teamId)` in `beforeAll`. Results are cached across files.

Creates entities through the API:
- **Leads:** Various ownership scenarios (by BASIC, by OWNER, in pool, with team, with AGENT read-only)
- **Deals:** Basic deal scenarios
- **Contacts/Companies:** Different creator scenarios

<Note>
All seeding uses actual API endpoints so business logic (guards, auto-assignment, stakeholder creation) executes naturally.
</Note>

## How to Extend

### Adding New Endpoints

<Steps>
<Step title="Update the Matrix">
Add new endpoint expectations to the appropriate matrix file:

```typescript
{
  method: 'POST',
  path: '/v1/leads/:leadId/convert',
  description: 'Convert lead to deal',
  expectations: {
    OWNER: 201,
    ADMIN: 201,
    TEAM_LEADER: 403,
    AGENT: 403,
    BASIC: 201,
  },
}
```
</Step>
<Step title="Add Path Resolution">
If using dynamic paths, update the resolver in `rbac-runner.ts`:

```typescript
if (path.includes(':leadId')) {
  return path.replace(':leadId', testData.leadByBasic.id);
}
```
</Step>
<Step title="Run Tests">
The matrix runner automatically generates test cases for all personas.
</Step>
</Steps>

<Steps>

## Methodology

<Check>
Follow these principles when writing RBAC tests:
</Check>

1. **Test real business scenarios** — use actual API endpoints, not mocked guards
2. **Matrix-driven coverage** — every persona × endpoint combination should be tested
3. **Declarative expectations** — define what should happen, let the runner verify it
4. **Stakeholder-based access** — test both permission-based and stakeholder-based access patterns
5. **Idempotent seeding** — tests should be runnable multiple times without conflicts
