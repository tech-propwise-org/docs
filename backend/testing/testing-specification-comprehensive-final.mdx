## Overview

The test suite focuses on **RBAC (Role-Based Access Control) E2E testing** — verifying that every API endpoint enforces the correct permissions for every user role.

<Info>
**Stack:** Jest + supertest + `@nestjs/testing`
</Info>

### Key Design Decisions

<CardGroup cols={2}>
  <Card title="Singleton App Context" icon="server">
    The NestJS app boots once and is shared across all test files (equivalent to NestJS `TestSuite` pattern, implemented manually since `TestSuite` requires NestJS 11+)
  </Card>
  <Card title="Matrix-Driven Tests" icon="table">
    A declarative permission matrix defines expected HTTP status codes per persona × endpoint. A runner generates `it()` blocks automatically
  </Card>
  <Card title="API-Driven Seeding" icon="database">
    Test data is created through the actual API endpoints so all business logic (guards, auto-assignment, stakeholder creation) executes naturally
  </Card>
  <Card title="Persona-Based Testing" icon="users">
    Five distinct RBAC personas cover all permission scenarios from owner to basic user
  </Card>
</CardGroup>

## Architecture

### Singleton Context

The app boots once for the entire test run. All test files share the same NestJS app instance, authenticated tokens, and resolved `teamId`.

<Steps>
  <Step title="Jest Starts">
    The first test file calls `getContext()`
  </Step>
  
  <Step title="App Bootstrap">
    First call boots NestJS app (~3-5s), logs in 5 personas, resolves teamId
  </Step>
  
  <Step title="Context Sharing">
    Subsequent test files receive the cached instance instantly (0ms)
  </Step>
  
  <Step title="Cleanup">
    Last file's `afterAll()` calls `closeContext()` → reference count hits 0 → app shuts down
  </Step>
</Steps>

#### Implementation

```typescript
import { getContext, closeContext, RbacTestContext } from '../../setup/rbac-context';

let ctx: RbacTestContext;

beforeAll(async () => {
  ctx = await getContext();
}, 60000);

afterAll(async () => {
  await closeContext();
});
```

<Tip>
The context replicates production config from `main.ts` including `ValidationPipe`, URI versioning (`/v1/...`), and cookie parser.
</Tip>

### Matrix Runner

Instead of writing individual `it()` blocks, you define expectations declaratively:

```typescript
{
  method: 'GET',
  path: '/v1/leads/:leadByBasicId',
  description: 'View lead detail',
  expectations: {
    OWNER: 200,
    ADMIN: 200,
    TEAM_LEADER: 403,
    AGENT: 403,
    BASIC: 200,
  },
}
```

#### Runner Process

The runner calls `runRbacMatrix()` which:

1. **Iterates** every endpoint in the matrix
2. **Loops** through all 5 personas for each endpoint
3. **Resolves** path placeholders (`:leadByBasicId` → actual UUID)
4. **Makes** the HTTP request with the persona's tenant token
5. **Asserts** the response status matches the expected value
6. **Produces** clear failure messages with the actual response body

<Check>
One matrix entry = 5 generated test cases (one per persona)
</Check>

## RBAC Personas

Defined in `test/helpers/rbac-personas.ts`. All personas share a single source of truth.

### Persona Reference

| Key | Email | Org Role | Team | Description |
|-----|-------|----------|------|-------------|
| `OWNER` | owner@propwise.com | Owner | Any | All permissions including `system.owner` |
| `ADMIN` | admin@propwise.com | Admin | Any | All non-sensitive permissions (includes `system.admin`) |
| `TEAM_LEADER` | teamleader@propwise.com | Basic | Team Leader in Test Sales Team | All team permissions (`team_crm.*`, `team_sales.*`, `team.admin`) |
| `AGENT` | agent@propwise.com | Basic | Agent in Test Sales Team | No team permissions by default |
| `BASIC` | basic@propwise.com | Basic | None | No permissions — relies entirely on stakeholder access |

### Permission Profiles

<AccordionGroup>
  <Accordion title="OWNER / ADMIN">
    Bypass resource-level checks via org permissions (`crm.view`, `crm.manage`, `sales.view`, `sales.manage`, etc.). Can access any entity in the org.
  </Accordion>
  
  <Accordion title="TEAM_LEADER">
    No org permissions, but has `team_crm.manage` and `team_sales.manage` in the team. Can access entities where the team is a stakeholder.
  </Accordion>
  
  <Accordion title="AGENT">
    No org permissions, no team permissions. Can only access entities where they are personally a stakeholder. Used to test access level differentiation (READ vs WRITE vs FULL).
  </Accordion>
  
  <Accordion title="BASIC">
    Same as AGENT but not a team member at all. Access is purely stakeholder-based.
  </Accordion>
</AccordionGroup>

### Login Flow

<Steps>
  <Step title="Identity Login">
    `POST /v1/auth/login` → identity token
  </Step>
  
  <Step title="Organization Selection">
    `POST /v1/auth/select-organization` → tenant token (org-scoped JWT)
  </Step>
  
  <Step title="API Requests">
    All subsequent API calls use the tenant token in the `Authorization: Bearer` header
  </Step>
</Steps>

## Permission Matrix

Matrices are split into per-entity files under `test/setup/matrices/`. The barrel at `test/setup/rbac-matrix.ts` re-exports everything.

### Lead Matrices

<Tabs>
  <Tab title="By Basic">
    **Matrix:** `LEAD_BY_BASIC_MATRIX`
    
    **Entity:** Lead created by BASIC
    
    **Who Can Access:**
    - OWNER, ADMIN (via `crm.*`)
    - BASIC (via FULL stakeholder)
    
    **Tests:** View, update, stakeholders, activities, property interests
  </Tab>
  
  <Tab title="On Team">
    **Matrix:** `LEAD_ON_TEAM_MATRIX`
    
    **Entity:** Lead with team as stakeholder
    
    **Who Can Access:**
    - OWNER, ADMIN
    - TEAM_LEADER (via `team_crm.*`)
    
    **Tests:** View, update, stakeholders, activities
  </Tab>
  
  <Tab title="Agent Read">
    **Matrix:** `LEAD_AGENT_READ_MATRIX`
    
    **Entity:** Lead where AGENT has READ access
    
    **Who Can Access:**
    - OWNER, ADMIN
    - AGENT (view only — modify denied)
    
    **Tests:** View allowed, update denied, access level enforcement
  </Tab>
  
  <Tab title="In Pool">
    **Matrix:** `LEAD_IN_POOL_MATRIX`
    
    **Entity:** Lead with no stakeholders
    
    **Who Can Access:**
    - OWNER, ADMIN only (via `crm.*`)
    
    **Tests:** View, update
  </Tab>
  
  <Tab title="Admin Only">
    **Matrix:** `LEAD_ADMIN_ONLY_MATRIX`
    
    **Entity:** Lead for admin-only ops
    
    **Who Can Access:**
    - OWNER, ADMIN (via `system.admin`)
    
    **Tests:** Delete
  </Tab>
</Tabs>

### Deal Matrices

| Matrix | Entity | Who Can Access |
|--------|--------|----------------|
| `DEAL_BY_OWNER_MATRIX` | Deal created by OWNER | OWNER, ADMIN only (via `sales.*`) |

### Contact Matrices

| Matrix | Entity | Who Can Access |
|--------|--------|----------------|
| `CONTACT_BY_BASIC_MATRIX` | Contact created by BASIC | OWNER, ADMIN, BASIC |
| `CONTACT_BY_OWNER_MATRIX` | Contact created by OWNER | OWNER, ADMIN only |

### Company Matrices

| Matrix | Entity | Who Can Access |
|--------|--------|----------------|
| `COMPANY_BY_BASIC_MATRIX` | Company created by BASIC | OWNER, ADMIN, BASIC |
| `COMPANY_BY_OWNER_MATRIX` | Company created by OWNER | OWNER, ADMIN only |

### Access Mode Reference

<Info>
Your `@CheckAccess` decorator uses two modes:
</Info>

**OR (default):** User has the required permission **OR** has stakeholder-level access on the resource → allowed.

**AND:** User must have the permission **AND** stakeholder access → allowed.

#### Resource Action Hierarchy

- `ACCESS` — granted by `READ`, `WRITE`, or `FULL` access level
- `MODIFY` — granted by `WRITE` or `FULL`
- `DELETE` — granted by `FULL` only

## Test Data Seeding

### Step 1: Seed Users

<Warning>
Run this one-time setup before running tests:
</Warning>

```bash
pnpm test:seed-users
```

This runs `test/setup/seed-test-users.ts` which:

<Steps>
  <Step title="Create Personas">
    Creates the RBAC test personas in the database (idempotent — skips if they exist)
  </Step>
  
  <Step title="Create Organization">
    Creates "Test Organization" with owner
  </Step>
  
  <Step title="Activate Subscription">
    Ensures **ACTIVE BUSINESS `subscription`** and **completed `organization_setup_progress`** on the test org (self-heals stale FREE / `LEAD_PIPELINE` rows from older seeds so Playwright and RBAC E2E skip onboarding and paid-plan gates)
  </Step>
  
  <Step title="Create Team">
    Creates "Test Sales Team" with team leader and agent
  </Step>
  
  <Step title="Assign Roles">
    Assigns org roles (Owner, Admin, Basic) and team roles (Team Leader, Agent)
  </Step>
</Steps>

<Note>
Requires subscription plans to exist. Run `pnpm test:init-data` once on a fresh DB before `pnpm test:seed-users`.
</Note>

### Step 2: Seed Test Data

Each test file calls `seedRbacTestData(app, personas, teamId)` in its `beforeAll`. Results are cached across files.

#### Created Entities

**Leads:**
- One by BASIC (auto-assigned as primary stakeholder with FULL access)
- One by OWNER (for admin-only tests)
- One in pool (empty stakeholders)
- One with team as stakeholder (OWNER creates, assigns team — TEAM_LEADER gets access)
- One with AGENT as READ-only stakeholder (OWNER creates, adds AGENT with `accessLevel: 'READ'`)

**Deals:**
- One by OWNER (for sales pipeline tests)

**Contacts:**
- One by BASIC
- One by OWNER

**Companies:**
- One by BASIC
- One by OWNER

<Tip>
Test data creation uses the actual API endpoints, ensuring all business logic (guards, auto-assignment, stakeholder creation) executes naturally.
</Tip>

## How to Run

### Core E2E Tests

```bash
pnpm test:e2e
```

Uses `jest-e2e.json` config. Runs auth lifecycle, org lifecycle, and all RBAC tests.

<Note>
Excludes distribution + escalation queue tests (those require live pg-boss).
</Note>

### Queue Tests

```bash
pnpm test:e2e:queue
```

Uses `jest-e2e-queue.json` config. Runs distribution, escalation, and queue-specific tests with live pg-boss integration.

### Individual Test Files

```bash
pnpm test:e2e test/e2e/rbac/lead-rbac.e2e-spec.ts
```

### Debug Mode

```bash
node --inspect-brk node_modules/.bin/jest --config test/jest-e2e.json --runInBand
```

<Tip>
Add breakpoints in your IDE and attach to the debugger.
</Tip>

### Cleanup Test Users

```bash
pnpm test:cleanup-users
```

Deletes all `@propwise-test.com` users from the database. Useful after test runs or before re-seeding.

## How to Extend

### Adding a New Endpoint to an Existing Matrix

<Steps>
  <Step title="Locate the Matrix">
    Open the relevant matrix file (e.g., `test/setup/matrices/lead-matrix.ts`)
  </Step>
  
  <Step title="Add Entry">
    Add a new object to the `endpoints` array:
    
    ```typescript
    {
      method: 'POST',
      path: '/v1/leads/:leadByBasicId/notes',
      description: 'Add note to lead',
      expectations: {
        OWNER: 201,
        ADMIN: 201,
        TEAM_LEADER: 403,
        AGENT: 403,
        BASIC: 201,
      },
    }
    ```
  </Step>
  
  <Step title="Run Tests">
    The runner will automatically generate 5 test cases (one per persona)
  </Step>
</Steps>

### Creating a New Matrix for a New Entity

<Steps>
  <Step title="Create Matrix File">
    Add `test/setup/matrices/property-matrix.ts`:
    
    ```typescript
    import { RbacMatrix } from '../rbac-matrix-types';
    
    export const PROPERTY_BY_BASIC_MATRIX: RbacMatrix = {
      name: 'Property (created by BASIC)',
      endpoints: [
        {
          method: 'GET',
          path: '/v1/properties/:propertyByBasicId',
          description: 'View property detail',
          expectations: {
            OWNER: 200,
            ADMIN: 200,
            TEAM_LEADER: 403,
            AGENT: 403,
            BASIC: 200,
          },
        },
      ],
    };
    ```
  </Step>
  
  <Step title="Export from Barrel">
    Add to `test/setup/matrices/index.ts`:
    
    ```typescript
    export * from './property-matrix';
    ```
  </Step>
  
  <Step title="Update Test Data Seeder">
    Add property creation logic to `test/setup/rbac-test-data.ts`:
    
    ```typescript
    const propertyByBasic = await request(app.getHttpServer())
      .post('/v1/properties')
      .set('Authorization', `Bearer ${personas.BASIC.tenantToken}`)
      .send({ name: 'Test Property', ... })
      .expect(201);
    ```
  </Step>
  
  <Step title="Create Test File">
    Add `test/e2e/rbac/property-rbac.e2e-spec.ts`:
    
    ```typescript
    import { runRbacMatrix } from '../../setup/rbac-runner';
    import { PROPERTY_BY_BASIC_MATRIX } from '../../setup/rbac-matrix';
    
    describe('Property RBAC', () => {
      // ... getContext() setup ...
      
      runRbacMatrix(
        () => ctx.app,
        () => ctx.personas,
        PROPERTY_BY_BASIC_MATRIX,
        () => ({
          propertyByBasicId: ctx.testData.propertyByBasic.id,
        }),
      );
    });
    ```
  </Step>
</Steps>

### Adding a New Persona

<Steps>
  <Step title="Update Persona Definitions">
    Add to `test/helpers/rbac-personas.ts`:
    
    ```typescript
    export const RBAC_PERSONAS = {
      // ... existing personas ...
      VIEWER: {
        email: 'viewer@propwise-test.com',
        password: 'Test1234!',
        firstName: 'Test',
        lastName: 'Viewer',
        role: 'Basic' as const,
      },
    };
    ```
  </Step>
  
  <Step title="Update Seeder">
    Add user creation logic to `test/setup/seed-test-users.ts`
  </Step>
  
  <Step title="Update Context">
    Add authentication logic to `test/setup/rbac-context.ts`
  </Step>
  
  <Step title="Update Matrices">
    Add the new persona to all `expectations` objects:
    
    ```typescript
    expectations: {
      OWNER: 200,
      ADMIN: 200,
      TEAM_LEADER: 403,
      AGENT: 403,
      BASIC: 200,
      VIEWER: 200, // New persona
    }
    ```
  </Step>
</Steps>

## Methodology

### Before Writing Tests

<Steps>
  <Step title="Analyze Controller">
    Review the controller file. Identify all endpoints, DTOs, and decorators (`@CheckAccess`, `@RequirePermissions`, etc.)
  </Step>
  
  <Step title="Review Guards">
    Understand which guards apply (`JwtAuthGuard`, `TenantAuthGuard`, `PermissionsGuard`, `AccessCheckGuard`)
  </Step>
  
  <Step title="Map Permissions">
    List all required permissions and stakeholder access levels for each endpoint
  </Step>
  
  <Step title="Design Matrices">
    Create test matrices that cover all permission combinations and edge cases
  </Step>
</Steps>

<Info>
See `test/TESTING_METHODOLOGY.md` for detailed analysis guidelines.
</Info>

### Test Coverage Goals

<CardGroup cols={2}>

  <Card title="Edge Cases" icon="triangle-exclamation">

## CI/CD Integration

### GitHub Actions Workflow

```yaml
name: E2E Tests

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main, develop]

jobs:
  test:
    runs-on: ubuntu-latest
    
    services:
      postgres:
        image: postgres:15
        env:
          POSTGRES_PASSWORD: postgres
          POSTGRES_DB: propwise_test
        options: >-
          --health-cmd pg_isready
          --health-interval 10s
          --health-timeout 5s
          --health-retries 5
    
    steps:
      - uses: actions/checkout@v3
      
      - name: Setup Node.js
        uses: actions/setup-node@v3
        with:
          node-version: '18'
          
      - name: Install dependencies
        run: pnpm install
        
      - name: Run migrations
        run: pnpm migration:run
        
      - name: Initialize test data
        run: pnpm test:init-data
        
      - name: Seed test users
        run: pnpm test:seed-users
        
      - name: Run E2E tests
        run: pnpm test:e2e
        
      - name: Run queue tests
        run: pnpm test:e2e:queue
```

```bash

  echo "E2E tests failed. Push aborted."
  exit 1
fi
```

## Best Practices

<CardGroup cols={2}>
  <Card title="Isolation" icon="box">
    Each test should be independent. Don't rely on test execution order.
  </Card>
  
  <Card title="Cleanup" icon="broom">
    Use `afterAll()` hooks to clean up resources and close connections.
  </Card>
  
  <Card title="Descriptive Names" icon="tag">
    Matrix descriptions should clearly state what's being tested.
  </Card>
  
  <Card title="Status Codes" icon="hashtag">
    Always specify expected HTTP status codes explicitly.
  </Card>
</CardGroup>

### Common Patterns

**Testing forbidden access:**
```typescript
expectations: {
  OWNER: 200,
  ADMIN: 200,
  TEAM_LEADER: 403,
  AGENT: 403,
  BASIC: 403,
}
```

**Testing read-only access:**
```typescript
// GET endpoint
expectations: {
  OWNER: 200,
  ADMIN: 200,
  AGENT: 200, // READ access
  BASIC: 403,
}

// PATCH endpoint (same resource)
expectations: {
  OWNER: 200,
  ADMIN: 200,
  AGENT: 403, // READ doesn't allow MODIFY
  BASIC: 403,
}
```

**Testing admin-only operations:**
```typescript
expectations: {
  OWNER: 200,
  ADMIN: 200,
  TEAM_LEADER: 403,
  AGENT: 403,
  BASIC: 403,
}
```

## Troubleshooting

<AccordionGroup>
  <Accordion title="App won't boot">
    **Symptoms:** Timeout errors in `beforeAll()`, context initialization fails
    
    **Solutions:**
    - Check database connection in `.env.test`
    - Ensure migrations are up to date
    - Verify test users are seeded
    - Increase Jest timeout (`jest.setTimeout(60000)`)
  </Accordion>
  
  <Accordion title="Permission tests failing unexpectedly">
    **Symptoms:** Expected 200, got 403 (or vice versa)
    
    **Solutions:**
    - Verify persona has correct org/team roles in database
    - Check stakeholder assignments on test entities
    - Review guard logic in the controller
    - Inspect actual permissions in `user_permissions` table
  </Accordion>
  
  <Accordion title="Test data conflicts">
    **Symptoms:** Unique constraint violations, FK errors
    
    **Solutions:**
    - Run `pnpm test:cleanup-users` to clear old data
    - Re-seed users with `pnpm test:seed-users`
    - Check for orphaned records in the database
  </Accordion>
  
  <Accordion title="Token authentication issues">
    **Symptoms:** 401 Unauthorized on all requests
    
    **Solutions:**
    - Verify JWT secret matches `.env.test`
    - Check token expiration (default 24h)
    - Ensure tenant token is used (not identity token)
    - Inspect `Authorization` header in failed requests
  </Accordion>
</AccordionGroup>

<Tip>
Enable debug logging with `DEBUG=* pnpm test:e2e` for verbose output.
</Tip>
