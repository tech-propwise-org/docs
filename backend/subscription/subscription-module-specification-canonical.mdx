The Subscription Module implements a **freemium SaaS billing system** for PropWise CRM. Every organization has a subscription tied to one of **three plan tiers** (Free / Pro / Business — Starter was removed). The module handles plan-based feature gating, resource limits, unified AI-credit wallet, seat management, and Stripe integration.

<Note>
**§18 (Subscription Packaging Rollout)** is the authoritative description of the current Free/Pro/Business AED model. Where earlier sections conflict with §18, **§18 wins**.
</Note>

## Architecture Overview

The subscription module follows a service-layer architecture with clear separation between billing logic, Stripe integration, and data persistence.

### Design Principles

<AccordionGroup>
<Accordion title="Core Design Principles">
| Principle | Decision |
|-----------|----------|
| Freemium model | Free plan with limited features; paid tiers unlock progressively |
| Per-org billing | Billing is per organization; developer portal is free |
| Feature flags over tier checks | Gating uses `@RequiresFeature('flag')` on plan JSONB |
| Service-layer limit enforcement | Resource limits checked in service methods, not guards |
| Stripe as source of truth | Webhook-driven lifecycle: app reacts to Stripe events |
| Idempotent webhooks | Every Stripe event logged in `BillingEvent` with unique `stripeEventId` |
</Accordion>
</AccordionGroup>

## Seat Management

The subscription module implements a **single per-agent seat model** where every user consumes exactly one seat regardless of their role.

### Seat Allocation Rules

<Steps>
<Step title="Pro Plan Seats">
- 5-10 seats included
- 11th seat triggers automatic upgrade to Business plan
</Step>
<Step title="Business Plan Seats">
- 10+ seats with volume pricing
- Additional seats billed monthly
</Step>
<Step title="Seat Type Derivation">
- Seat type automatically determined by user's RBAC role
- No explicit seat assignment required
</Step>
</Steps>

<Warning>
When changing plans, the system validates that current users don't exceed the new plan's seat capacity. Plan changes are blocked if seat overflow would occur.
</Warning>

## Stripe Integration

The module integrates with Stripe for payment processing, subscription management, and billing automation.

### Stripe Configuration

<CodeGroup>

```typescript Environment Variables
STRIPE_SECRET_KEY=sk_live_...
STRIPE_WEBHOOK_SECRET=whsec_...
STRIPE_PUBLISHABLE_KEY=pk_live_...
```

```typescript Stripe Service
@Injectable()
export class StripeService {
  private stripe: Stripe;

  constructor(private configService: ConfigService) {
    const secretKey = this.configService.get('STRIPE_SECRET_KEY');
    if (!secretKey) {
      this.logger.warn('Stripe secret key not configured - billing features disabled');
      return;
    }
    
    this.stripe = new Stripe(secretKey, {
      apiVersion: '2023-10-16',
    });
  }

  async createCheckoutSession(params: CheckoutParams) {
    return this.stripe.checkout.sessions.create({
      mode: 'subscription',
      payment_method_types: ['card'],
      line_items: params.lineItems,
      success_url: params.successUrl,
      cancel_url: params.cancelUrl,
      customer: params.customerId,
      metadata: params.metadata,
    });
  }
}
```

</CodeGroup>

## Subscription Lifecycle

The subscription lifecycle is managed through Stripe webhooks and local state synchronization.

### Lifecycle States

<Steps>
<Step title="Free Organization">
- Default state for new organizations
- No Stripe subscription required
- Limited features and resources
</Step>

<Step title="Trial Period">
- 90-day evergreen trial for Pro & Business
- Full plan features enabled
- Credit card required upfront
</Step>

<Step title="Active Subscription">
- Payment successful
- All plan features enabled
- Regular billing cycle active
</Step>

<Step title="Past Due">
- Payment failed, 2-day grace period
- Features remain enabled
- Stripe retries payment
</Step>

<Step title="Suspended">
- All payment retries failed
- Organization becomes read-only
- Data preserved but no writes allowed
</Step>
</Steps>

## Plan Changes & Upgrades

The module supports immediate tier changes and deferred billing cycle changes with proper proration.

### Change Types

<Info>
**Tier Changes** (Free → Pro, Pro → Business) are immediate and prorated.
**Billing Cycle Changes** (Monthly ↔ Annual) are deferred to period end via Stripe Subscription Schedules.
**Combined Changes** (tier + cycle) are immediate and prorated with all line items re-priced atomically.
</Info>

### Implementation Flow

<CodeGroup>

```typescript Plan Change Endpoint
@Post('change-plan')
async changePlan(
  @Body() changePlanDto: ChangePlanDto,
  @CurrentUser() user: User,
) {
  const result = await this.subscriptionService.changePlan(
    user.organizationId,
    changePlanDto.planId,
    changePlanDto.billingCycle,
  );
  
  return { success: true, subscription: result };
}
```

```typescript Service Implementation
async changePlan(
  organizationId: string,
  newPlanId: string,
  billingCycle: 'monthly' | 'annual',
) {
  // 1. Validate seat capacity
  await this.validateSeatCapacity(organizationId, newPlanId);
  
  // 2. Update Stripe subscription
  const stripeSubscription = await this.stripeService.updateSubscription({
    subscriptionId: currentSubscription.stripeSubscriptionId,
    newPriceId: newPlan.getStripePrice(billingCycle),
    prorationBehavior: 'create_prorations',
  });
  
  // 3. Update local subscription
  return this.updateLocalSubscription(stripeSubscription);
}
```

</CodeGroup>

    const request = context.switchToHttp().getRequest();
    const user = request.user;

## Enforcement Points

Resource limits and credit consumption are enforced at the service layer to ensure accurate entity counting and business logic validation.

### Service-Layer Enforcement

<Tabs>
<Tab title="Lead Creation">
```typescript
async createLead(organizationId: string, leadData: CreateLeadDto) {
  // Check resource limits
  const usage = await this.getResourceUsage(organizationId);
  const plan = await this.getCurrentPlan(organizationId);
  
  if (usage.leads >= plan.limits.leads) {
    throw new ForbiddenException('Lead limit exceeded for current plan');
  }
  
  // Create lead
  return this.leadRepository.create(leadData);
}
```
</Tab>
<Tab title="AI Credit Consumption">
```typescript
async consumeCredits(organizationId: string, action: string, amount: number) {
  const balance = await this.getCreditBalance(organizationId);
  
  if (balance.availableCredits < amount) {
    throw new ForbiddenException('Insufficient credits');
  }
  
  // Consume credits with FIFO order
  await this.creditService.consumeCredits(organizationId, amount);
}
```
</Tab>
