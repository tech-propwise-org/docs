# Subscription Module Specification (Canonical)

<Info>
**Status:** Active — fully implemented  
**Module Path:** `src/modules/subscription/`  
**Payment Gateway:** Stripe
</Info>

## Overview

The Subscription Module implements a **freemium SaaS billing system** for PropWise CRM. Every organization has a subscription tied to one of four plan tiers. The module handles:

- **Plan-based feature gating** — binary feature flags per tier
- **Resource limits** — caps on leads, contacts, deals, companies, and storage
- **Credit-based metering** — monthly AI and messaging allowances with purchasable top-ups
- **Dual seat types** — manager seats and agent seats with per-tier pricing; every user consumes a seat
- **Stripe integration** — checkout, subscription management, mid-cycle plan changes, webhooks, billing portal
- **Proration** — mid-cycle upgrades, downgrades, and seat changes are prorated to the day
- **Suspension flow** — 2-day grace period on payment failure, then org goes read-only

### Design Principles

<AccordionGroup>
<Accordion title="Core Billing Principles">
| Principle | Decision |
|---|---|
| Freemium model | Free plan with limited features; paid tiers unlock progressively |
| Per-org billing | Billing is per organization; developer portal is free |
| Dual seat types | Manager seats (Owner, Admin) and agent seats (Basic, custom roles); every user consumes a seat |
| Seat type derived from role | No explicit seat assignment — seat type is automatically determined by the user's RBAC role |
</Accordion>

<Accordion title="Technical Enforcement">
| Principle | Decision |
|---|---|
| Feature flags over tier checks | Gating uses `@RequiresFeature('flag')` on plan JSONB — changing what a tier includes requires only a seeder update, not code changes |
| Service-layer limit enforcement | Resource limits and credit consumption are checked in service methods, not guards, because they need entity counts |
| Stripe as source of truth for payments | Webhook-driven lifecycle: the app reacts to Stripe events rather than polling |
| Prorated plan changes | All mid-cycle changes (upgrade, downgrade, add/remove seats) use `proration_behavior: 'create_prorations'` — charges are fair to the day |
</Accordion>

<Accordion title="Integration & Reliability">
| Principle | Decision |
|---|---|
| Checkout vs. change-plan separation | `POST /checkout` is for first-time subscription (Free → Paid); `POST /change-plan` is for switching between paid tiers |
| Idempotent webhooks | Every Stripe event is logged in `BillingEvent` with a unique `stripeEventId` to prevent duplicate processing |
| Graceful degradation | If `app.stripe.secretKey` (`STRIPE_SECRET_KEY`) is not set, billing features are unavailable but the app still starts |
</Accordion>
</AccordionGroup>

### Core Data Flows

<Tabs>
<Tab title="First-time Checkout">
**First-time checkout flow (Free → Paid):**

```
Frontend "Upgrade" button
  → POST /v1/subscriptions/checkout
    → Rejects if org already has a Stripe subscription (use change-plan instead)
    → SubscriptionService.createCheckoutSession()
      → StripeService.createCheckoutSession()
        → Returns Stripe Checkout URL
          → User pays on Stripe's hosted page
            → Stripe fires checkout.session.completed webhook
              → StripeWebhookController receives + verifies signature
                → SubscriptionService.activateSubscription()
                  → Subscription entity updated to ACTIVE
```
</Tab>

<Tab title="Plan Changes">
**Mid-cycle plan change flow (Paid → different Paid tier):**

```
Frontend "Change Plan" button
  → POST /v1/subscriptions/change-plan
    → SubscriptionService.changePlan()
      → Validates seat overflow (blocks if current users exceed new plan capacity)
      → StripeService.swapSubscriptionPrice() — prorated
      → Reconciles seat line items (old tier price → new tier price)
      → Updates local Subscription entity
      → Returns updated subscription immediately
```
</Tab>

<Tab title="Payment Failures">
**Renewal / payment failure flow:**

```
Stripe charges renewal invoice
  ├─ invoice.paid → handleInvoicePaid() → status stays ACTIVE, period updated
  └─ invoice.payment_failed → handleInvoicePaymentFailed() → status → PAST_DUE
       └─ Stripe retries for 2 days
            ├─ Payment succeeds → invoice.paid → back to ACTIVE
            └─ All retries fail → customer.subscription.updated (status: unpaid)
                 → handleSubscriptionUpdated() → status → SUSPENDED
                      → Org is read-only (SubscriptionActiveGuard blocks writes)
```
</Tab>
</Tabs>

## Seat Management

### Seat Types

Every user in an organization consumes exactly one seat. The seat type is **derived from the user's RBAC role** — there is no separate seat assignment.

| Seat type | Roles that consume it | Price varies by tier |
|---|---|---|
| **Manager** | Owner, Admin | Yes |
| **Agent** | Basic, custom org roles | Yes |

The mapping is defined in `subscription.service.ts`:

```typescript
const ROLE_SEAT_MAP: Record<string, SeatType> = {
  Owner: SeatType.MANAGER,
  Admin: SeatType.MANAGER,
};
// Any other role → SeatType.AGENT
```

### Seat Counting

<Warning>
Seats are **derived from RBAC roles**, not tracked via a separate assignment table. The count is computed on-demand from active `UserOrgRole` records.
</Warning>

```
managerSeatsUsed = count of active users with Owner or Admin org role
agentSeatsUsed   = count of active users with any other org role
```

A seat is **not occupied** by a pending invitation — it only counts when the user has accepted and has an active `UserOrgRole`. This means:

| Step | Seat occupied? |
|---|---|
| Admin sends invitation with role "Admin" | No — seat availability is checked but not reserved |
| User accepts → `UserOrgRole` created | Yes — now counted |
| User removed (role soft-deleted) | No — freed |
| User's role changed (Basic → Admin) | Swaps: frees one agent seat, occupies one manager seat |

### Enforcement Points

Seat availability is checked at two integration points:

<Steps>
<Step title="Invitation Creation">
`invitation.service.ts` — before creating an invitation, the role determines the seat type and availability is checked
</Step>
<Step title="Role Assignment">
`role-assignment-validation.service.ts` — when changing a user's role (e.g. promoting Basic → Admin), checks that the target seat type has room; the old seat type is freed simultaneously
</Step>
</Steps>

### Proration on Seat Changes

Adding or removing seats mid-cycle uses `proration_behavior: 'create_prorations'`:

- **Adding a seat on April 15** (30-day month): prorated charge for 15 remaining days, billed on the next invoice
- **Removing a seat on April 15**: prorated credit for 15 remaining days, applied to the next invoice
- **Adding on April 4, removing on April 6**: net charge for 2 days only (charge for 26 days minus credit for 24 days)

### Stripe Billing

Extra seats are billed as subscription line items with `per_unit` pricing. A subscription for a Professional org with 7 managers and 20 agents would have:

| Line Item | Qty | Price |
|---|---|---|
| PropWise Professional | 1 | $149/mo |
| Extra Manager Seat (Pro) | 2 | $40/mo |
| Extra Agent Seat (Pro) | 5 | $50/mo |

## Credit System

### Consumption Flow

```typescript
SubscriptionService.consumeCredits(orgId, 'ai', 1)
  → CreditService.consumeCredits(subscription, AI, 1)
      1. Check monthly allowance: usage.aiCreditsUsed < plan.aiCreditsPerMonth
      2. If allowance available: increment usage.aiCreditsUsed, return SUCCESS
      3. If allowance exhausted: find oldest CreditPurchase with remaining balance
      4. Deduct from purchase, update remaining balance
      5. If no credits available anywhere: return INSUFFICIENT_CREDITS
```

<Check>
Credits are consumed in **FIFO order** (oldest purchased packs first) to ensure fair usage of time-limited credits.
</Check>

## Stripe Integration

### Webhook Events

<Warning>
All Stripe events are processed idempotently using `BillingEvent` to track processed webhook IDs.
</Warning>

| Event | Handler | Action |
|---|---|---|
| `checkout.session.completed` | `handleCheckoutCompleted()` | Activate subscription, link Stripe IDs |
| `invoice.paid` | `handleInvoicePaid()` | Update billing period, ensure ACTIVE status |
| `invoice.payment_failed` | `handleInvoicePaymentFailed()` | Set status to PAST_DUE |
| `customer.subscription.updated` | `handleSubscriptionUpdated()` | Sync status changes (e.g., unpaid → SUSPENDED) |
| `customer.subscription.deleted` | `handleSubscriptionDeleted()` | Set status to CANCELLED |

### Checkout Session Creation

<CodeGroup>
```typescript SubscriptionService.createCheckoutSession()
async createCheckoutSession(
  orgId: string, 
  planId: string, 
  billingCycle: BillingCycle
): Promise<{ checkoutUrl: string }> {
  const organization = await this.orgRepo.findOneOrFail(orgId);
  const plan = await this.planRepo.findOneOrFail(planId);
  
  // Reject if org already has Stripe subscription
  if (organization.stripeCustomerId) {
    throw new ConflictException('Use change-plan endpoint for existing subscriptions');
  }

  const session = await this.stripeService.createCheckoutSession({
    customer_email: organization.primaryEmail,
    client_reference_id: orgId,
    line_items: [{
      price: billingCycle === 'MONTHLY' ? plan.stripeMonthlyPriceId : plan.stripeAnnualPriceId,
      quantity: 1
    }],
    mode: 'subscription'
  });

  return { checkoutUrl: session.url };
}
```

```typescript StripeService.createCheckoutSession()
async createCheckoutSession(params: Stripe.Checkout.SessionCreateParams) {
  return this.stripe.checkout.sessions.create({
    ...params,
    success_url: `${this.configService.get('app.frontendUrl')}/billing/success`,
    cancel_url: `${this.configService.get('app.frontendUrl')}/billing/cancel`,
    billing_address_collection: 'required',
    customer_creation: 'always'
  });
}
```
</CodeGroup>

<Steps>
<Step title="Initial State">

</Step>
<Step title="Upgrade to Paid">

</Step>
<Step title="Payment Failure">
Invoice payment failure → PAST_DUE (2-day grace period)
</Step>

### Grace Period Behavior

<Note>
During PAST_DUE status (2-day grace period), organizations retain full access. Only SUSPENDED status triggers read-only mode.
</Note>

## Plan Changes (Upgrade / Downgrade)

### Seat Overflow Validation

<Warning>
Plan changes are blocked if the target plan cannot accommodate existing users.
</Warning>

```typescript
// Example: Professional → Starter downgrade
// Professional: 5 manager + 15 agent seats
// Starter: 2 manager + 3 agent seats
// Current usage: 4 managers, 8 agents
// Result: BLOCKED (4 managers > 2 allowed, 8 agents > 3 allowed)

async validateSeatOverflow(subscription: Subscription, targetPlan: SubscriptionPlan) {
  const { managerSeatsUsed, agentSeatsUsed } = subscription;
  
  if (managerSeatsUsed > targetPlan.managerSeatsIncluded) {
    throw new ConflictException(
      `Cannot downgrade: ${managerSeatsUsed} managers exceed ${targetPlan.managerSeatsIncluded} included`
    );
  }
  
  if (agentSeatsUsed > targetPlan.agentSeatsIncluded) {
    throw new ConflictException(
      `Cannot downgrade: ${agentSeatsUsed} agents exceed ${targetPlan.agentSeatsIncluded} included`
    );
  }
}
```

### Proration Logic

All mid-cycle changes use Stripe's `proration_behavior: 'create_prorations'`:

<Tabs>
<Tab title="Upgrade">
- Higher base plan price prorated for remaining days
- Seat reconciliation: remove old seat line items, add new ones
- Net charge applied to next invoice
</Tab>

<Tab title="Downgrade">
- Lower base plan price prorated for remaining days  
- Seat reconciliation: remove old seat line items, add new ones
- Net credit applied to next invoice
</Tab>

<Tab title="Same Tier Seat Change">
- No base plan change
- Only seat line items modified
- Immediate proration
</Tab>
</Tabs>

## Integration with Other Modules

### Cross-Module Dependencies

<Tabs>
<Tab title="User & Organization">
- **Organization creation** → Auto-creates Free subscription
- **User role changes** → Updates seat usage counts
- **User removal** → Frees up seats automatically
</Tab>

<Tab title="Lead Management">
- **Lead creation** → Checks resource limits, increments usage
- **Lead import** → Batch limit validation
- **Lead distribution** → Requires `distributionEngine` feature
</Tab>

<Tab title="AI & Messaging">
- **AI queries** → Consumes AI credits
- **SMS/Email sending** → Consumes messaging credits  
- **Credit depletion** → Graceful degradation or blocking
</Tab>

<Tab title="Analytics & Reporting">
- **Advanced reports** → Requires `advancedAnalytics` feature
- **Custom dashboards** → Feature-gated functionality
- **API access** → Requires `apiAccess` feature
</Tab>
</Tabs>

### Event Integration

The subscription module publishes events that other modules can subscribe to:

```typescript
// Events published by subscription module
export enum SubscriptionEvent {
  SUBSCRIPTION_ACTIVATED = 'subscription.activated',
  SUBSCRIPTION_SUSPENDED = 'subscription.suspended', 
  PLAN_CHANGED = 'subscription.plan_changed',
  CREDITS_DEPLETED = 'subscription.credits_depleted'
}

// Example subscriber in lead module
@OnEvent(SubscriptionEvent.SUBSCRIPTION_SUSPENDED)
async handleSubscriptionSuspended(payload: { orgId: string }) {
  await this.leadService.pauseAutomations(payload.orgId);
}
```
