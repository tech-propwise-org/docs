This specification covers Gmail and Outlook email OAuth integration flows, token storage, refresh mechanisms, and API endpoints. Calendar OAuth is documented separately in the Calendar System Deep Dive.

## OAuth Flow Overview

The email integration OAuth flow supports both personal and organization-level mailbox connections with comprehensive security checks and token management.

<Steps>
<Step title="User Authentication">
User selects an organization and the SPA holds a tenant JWT
</Step>

<Step title="OAuth Initialization">
Choose between personal or organization-level integration:
- **Personal**: `POST /v1/email-integrations/oauth/:provider/authorize`
- **Organization**: `POST /v1/email-integrations/organization/oauth/:provider/authorize`
</Step>

<Step title="Authorization URL Generation">
Backend returns `{ authorizationUrl }` with HMAC-signed state containing user, organization, and token version data
</Step>

<Step title="Provider Authentication">
Browser opens authorization URL in popup, provider redirects to callback endpoints
</Step>

<Step title="Token Exchange and Verification">
Backend verifies state, exchanges authorization code, and performs access verification
</Step>

<Step title="Integration Persistence">
Tokens are encrypted and stored with comprehensive access and conflict checks
</Step>
</Steps>

## OAuth Endpoints

### Authorization Endpoints

<CodeGroup>
```http Personal OAuth
POST /v1/email-integrations/oauth/:provider/authorize
```

```http Organization OAuth  
POST /v1/email-integrations/organization/oauth/:provider/authorize
```
</CodeGroup>

<Note>
The `provider` parameter accepts `gmail` or `outlook`. Organization OAuth requires `SYSTEM_ADMIN` or `SYSTEM_OWNER` permissions.
</Note>

### Callback Endpoints

Public callback endpoints (no `/v1` prefix):
- `GET /auth/gmail/callback`
- `GET /auth/outlook/callback`

## Integration Levels

<Tabs>
<Tab title="Personal">
- **Scope**: Individual user mailbox
- **Database**: `user_id` NOT NULL, `connected_by_id` NULL
- **Access**: Only the mailbox owner can use the integration
- **Uniqueness**: `(user_id, organization_id, provider)` where `is_deleted = false`
</Tab>

<Tab title="Organization">
- **Scope**: Shared workspace mailbox  
- **Database**: `user_id` NULL, `connected_by_id` NOT NULL
- **Access**: `SYSTEM_ADMIN` or `SYSTEM_OWNER` required
- **Uniqueness**: `(organization_id, provider, lower(email))` where `is_deleted = false`
- **Default**: Optional `is_default` flag with one default per organization
</Tab>
</Tabs>

## Gmail Access Verification

<Warning>
Gmail integrations undergo additional access verification to prevent saving broken integrations for accounts that authenticate but don't expose Gmail API access.
</Warning>

The verification process:

<Steps>
<Step title="Token Exchange">
Complete the OAuth token exchange with Google
</Step>

<Step title="API Access Test">
Call `GET .../gmail/v1/users/me/profile` with the new access token
</Step>

<Step title="Response Handling">
- **Success**: Continue with integration persistence
- **400/403 Error**: Reject integration and return `GMAIL_ACCESS_BLOCKED` error
</Step>
</Steps>

<Info>
Outlook integrations skip this verification step as they don't require the same API access pattern.
</Info>

## Security and Access Controls

### OAuth State Security

The OAuth state parameter includes:
- `userId` and `organizationId`
- `tokenVersion` for invalidating stale OAuth flows
- `provider` (gmail/outlook)
- `level` (PERSONAL/ORGANIZATION)

<Note>
State TTL is configured via `EMAIL_OAUTH_STATE_TTL_MS` (default: 15 minutes)
</Note>

### Callback Access Re-verification

Before persisting tokens, the system re-checks:

<Check>User is active with matching token version</Check>
<Check>Active user-organization role exists</Check>  
<Check>For organization level: user still has admin/owner permissions</Check>

### Cross-Level Mailbox Conflicts

<Warning>
A mailbox address cannot exist at both PERSONAL and ORGANIZATION levels within the same organization.
</Warning>

Conflict prevention:
- Database trigger with advisory locking
- Runtime checks during OAuth flow
- Existing personal-personal duplicates remain valid

## API Endpoints

### List Integrations

<Tabs>
<Tab title="Personal Integrations">
```http
GET /v1/email-integrations
```

Returns personal mailboxes for the current user only. Includes pagination with `nextCursor`.

**Query Parameters:**
- `limit`: 1-50 (default: 20)
- `cursor`: Opaque pagination cursor
</Tab>

<Tab title="Organization Integrations">
```http  
GET /v1/email-integrations/organization
```

Returns organization-level mailboxes. Requires admin/owner permissions.

**Response includes:**
- Integration details
- `connectedBy` user information
- Pagination support
</Tab>

<Tab title="Send Accounts">
```http
GET /v1/email-integrations/send-accounts  
```

Returns mailboxes available for composing emails:
- `personal`: User's personal integrations
- `organization`: Organization integrations (if admin/owner)

**Query Parameters:**
- `cursorPersonal`: Personal stream cursor
- `cursorOrganization`: Organization stream cursor
- `limit`: Page size (1-50)
</Tab>
</Tabs>

### Individual Integration Operations

<CodeGroup>
```http Get Integration
GET /v1/email-integrations/:id
```

```http Update Integration  
PUT /v1/email-integrations/:id
```

```http Delete Integration
DELETE /v1/email-integrations/:id
```

```http Send Email
POST /v1/email-integrations/:id/send
```
</CodeGroup>

<Note>
Access control ensures personal integrations are only accessible to their owners, while organization integrations require admin/owner permissions.
</Note>

## Data Model

### Integration Entity

<AccordionGroup>
<Accordion title="Core Fields">
- `id`: UUID primary key
- `level`: PERSONAL | ORGANIZATION  
- `organization_id`: Foreign key to organization
- `provider`: gmail | outlook
- `email`: Normalized mailbox address (lowercase)
- `is_deleted`: Soft delete flag
</Accordion>

<Accordion title="User Relationships">
- `user_id`: Owner (PERSONAL only)
- `connected_by_id`: OAuth actor (ORGANIZATION only)
</Accordion>

<Accordion title="Token Storage">
- `encrypted_access_token`: AES-256-GCM encrypted
- `encrypted_refresh_token`: AES-256-GCM encrypted  
- `token_expires_at`: Access token expiration
- `provider_account_id`: Provider-specific account identifier
</Accordion>

<Accordion title="Organization Features">
- `is_default`: Default mailbox flag (one per org)
- `metadata`: JSON field for provider-specific data
</Accordion>

<Accordion title="Sync Status">
- `last_sync_at`: Last successful sync timestamp
- `last_sync_error_at`: Last sync error timestamp
- `sync_cursor`: Provider sync cursor/token
</Accordion>
</AccordionGroup>

### Uniqueness Constraints

<CardGroup cols={2}>
<Card title="Personal Level" icon="user">
Unique constraint on `(user_id, organization_id, provider)` where `is_deleted = false` and `level = 'PERSONAL'`
</Card>

<Card title="Organization Level" icon="building">  
Unique constraint on `(organization_id, provider, lower(email))` where `is_deleted = false` and `level = 'ORGANIZATION'`
</Card>
</CardGroup>

## Error Handling

### OAuth Callback Errors

Common error scenarios and their handling:

<AccordionGroup>
<Accordion title="Gmail Access Blocked">
**Error Code**: `GMAIL_ACCESS_BLOCKED`

**Cause**: Google account authenticates but doesn't provide Gmail API access

**Response**: AlertDialog in SPA with user-friendly message
</Accordion>

<Accordion title="Tenant Access Denied">
**Error Code**: `OAUTH_CALLBACK_TENANT_ACCESS_DENIED_MESSAGE`

**Cause**: User lost permissions or org access during OAuth flow

**Response**: Generic access denied message
</Accordion>

<Accordion title="Redundant Reconnect">
**Cause**: Attempting to reconnect a healthy integration with the same email address

**Response**: OAuth error instead of success to prevent unnecessary token updates
</Accordion>
</AccordionGroup>

### PostMessage Security

<Check>Uses `postMessage` with specific target origin from `FRONTEND_URL`</Check>
<Check>Never uses wildcard (`*`) target origin</Check>  
<Check>SPA validates message source is the OAuth popup window</Check>
<Check>Generic error messages prevent information leakage</Check>

## Frontend Integration

### SPA Response Handling

```javascript
// Invalidate queries after successful OAuth
queryKeys.emailIntegrations

// Handle specific Gmail access errors  
if (error === 'GMAIL_ACCESS_BLOCKED') {
  showAlertDialog();
}

// Validate postMessage source
if (event.source === oauthPopupWindow) {
  handleOAuthResult(event.data);
}
```

<Tip>
Always refetch integration lists when OAuth popups close to handle dropped messages due to URL mismatches.
</Tip>
