## Overview

This specification covers Gmail and Outlook **email** OAuth integration, including token storage, refresh mechanisms, and API surfaces. Calendar OAuth is documented separately in the Calendar System Deep Dive.

<Note>
  Email integrations support two levels: **PERSONAL** (individual user mailboxes) and **ORGANIZATION** (shared workspace mailboxes for admins/owners).
</Note>

## OAuth Flow

The OAuth flow operates within a tenant session context, with the SPA holding a tenant JWT after organization selection.

### Authorization Flow

<Steps>
  <Step title="Initiate OAuth">
    User initiates OAuth by calling one of two endpoints:
    
    - **Personal mailbox**: `POST /v1/email-integrations/oauth/:provider/authorize`
      - Level: `EmailIntegrationLevel.PERSONAL`
      - Available to all authenticated users
      
    - **Organization mailbox**: `POST /v1/email-integrations/organization/oauth/:provider/authorize`
      - Level: `EmailIntegrationLevel.ORGANIZATION`
      - Requires `SYSTEM_ADMIN` or `SYSTEM_OWNER` role (`@CheckAccess`)
    
    Provider parameter accepts: `gmail` | `outlook`
  </Step>

  <Step title="Receive authorization URL">
    Backend returns `{ authorizationUrl }` containing:
    
    - Google/Microsoft authorize URL
    - HMAC-signed `state` parameter (via `SignedPayloadService`)
    - Purpose: `email.oauth.state`
    - Payload includes:
      - `userId`
      - `organizationId`
      - `tokenVersion` (invalidates popups after password reset/logout)
      - `provider`
      - `level` (optional; omitted = `PERSONAL` for legacy tokens)
  </Step>

  <Step title="Open OAuth popup">
    Browser opens `authorizationUrl` in a popup window.
  </Step>

  <Step title="Provider redirect">
    Provider redirects to public callback endpoint (no `/v1` prefix):
    
    - Gmail: `GET /auth/gmail/callback`
    - Outlook: `GET /auth/outlook/callback`
  </Step>

  <Step title="Backend verification">
    Backend performs multiple verification steps:
    
    1. Verifies signed `state` parameter
    2. Exchanges authorization `code` via provider driver
    3. **Gmail only**: Calls `GET .../gmail/v1/users/me/profile` with new access token
       - If returns 400/403: integration NOT persisted
       - Popup receives `oauth-error` with `EMAIL_OAUTH_GMAIL_ACCESS_BLOCKED_CODE`
    4. Re-checks OAuth actor access (see Callback Access Re-check below)
    5. Runs `connectInTransaction` if all checks pass
  </Step>

  <Step title="postMessage response">
    Callback HTML uses `postMessage` with `app.postMessageTargetOrigin` (from `FRONTEND_URL`):
    
    - Never uses wildcard `*` origin
    - Payload may include `level` for SPA
    - On success: `oauth-success`
    - On error: `oauth-error` with error code and message
  </Step>

  <Step title="SPA refresh">
    SPA invalidates `queryKeys.emailIntegrations` to refresh:
    
    - `GET /v1/email-integrations/send-accounts` (composer)
    - Integration lists
    
    Validates `postMessage` only when `event.source` matches OAuth popup.
    
    Refetches when popup closes (handles dropped messages).
  </Step>
</Steps>

### Gmail Access Verification

<Warning>
  Gmail OAuth includes additional access verification to prevent saving integrations for accounts that authenticate but don't expose Gmail API access.
</Warning>

**When**: Immediately after `exchangeCode`, before `connectInTransaction`

**How**: Backend calls `users.getProfile` with the new access token

**Result**: 
- 400/403 response → Integration NOT persisted
- Popup receives `oauth-error` with `GMAIL_ACCESS_BLOCKED` code

**Required scopes**:
- `gmail.send` - Outbound email sending
- `gmail.metadata` - Profile access probe
- `userinfo.email` - Read mailbox address

**Implementation constant**: `EMAIL_OAUTH_GMAIL_ACCESS_BLOCKED_CODE` in `src/common/constants/email-oauth.constants.ts` (must match frontend constant in `email-integration.api.ts`)

### Callback Access Re-check

Before persisting tokens, the backend re-validates OAuth actor permissions in the same database transaction:

<Tabs>
  <Tab title="Common Checks">
    Applied to both PERSONAL and ORGANIZATION levels:
    
    - `User.isActive` must be true
    - `User.tokenVersion` must match signed state
    - Active (non-deleted) `user_org_roles` row for `(userId, organizationId)`
  </Tab>
  
  <Tab title="Organization-Level Checks">
    Additional checks for ORGANIZATION level only:
    
    - Actor must hold `system.admin` or `system.owner` role
    - Verified via role/permission join (same as `@CheckAccess`)
  </Tab>
</Tabs>

**On failure**: Popup receives `oauth-error` with `OAUTH_CALLBACK_TENANT_ACCESS_DENIED_MESSAGE`, tokens not written

**On success**: `connectInTransaction` executes (encrypts tokens, upserts row)

**Service method**: `AccessService.assertOAuthCallbackOrgAccessInTransaction`

### Mailbox Uniqueness Rules

<AccordionGroup>
  <Accordion title="Organization-Level Uniqueness">
    - `user_id = null`
    - `connected_by_id` = OAuth actor
    - `email` normalized to lowercase
    - Unique constraint: `(organization_id, provider, lower(email))` among active org rows
  </Accordion>

  <Accordion title="Personal-Level Uniqueness">
    - Multiple users may connect the **same** mailbox address as PERSONAL
    - Uniqueness per: user + org + provider
  </Accordion>

  <Accordion title="Cross-Level Conflict Prevention">
    A mailbox address cannot exist at both PERSONAL and ORGANIZATION levels in the same organization:
    
    - `connectInTransaction` rejects PERSONAL OAuth when active ORGANIZATION row has same normalized `email`
    - Rejects ORGANIZATION OAuth when any active PERSONAL row has same normalized `email`
    - Provider intentionally ignored for cross-level rule
    
    **Database enforcement**: Migration `Migration20260521171000_PreventEmailIntegrationCrossLevelMailboxConflicts` installs trigger `email_integrations_cross_level_mailbox_conflict_trigger` with transaction advisory lock per org/mailbox.
  </Accordion>

  <Accordion title="Redundant Reconnect Prevention">
    OAuth rejected when all conditions met:
    
    - Profile `email` matches existing row's stored `email`
    - Row is **healthy**:
      - Active (not deleted)
      - No `last_sync_error_at`
      - Access token present
      - `token_expires_at` beyond configured buffer (see `EMAIL_RECONNECT_ACCESS_TOKEN_BUFFER_MINUTES`)
    
    Result: Popup receives `oauth-error` instead of `oauth-success`
    
    **Allowed scenarios**:
    - Switching to different address (if no cross-level conflict)
    - Unhealthy rows (inactive, sync error, missing/expired token)
  </Accordion>
</AccordionGroup>


<Warning>
  Register these exact redirect URIs in your Google Cloud OAuth client and Azure app configuration (`GMAIL_*` / `OUTLOOK_*` credentials).
</Warning>

**Implementation**: `EmailOAuthCallbackController` serves both callback endpoints matching `app.*.emailRedirectUri` defaults.

### Callback Error Handling

Token exchange failures and IdP `error` query params are logged server-side with:
- Provider HTTP status
- Truncated body preview

The popup `postMessage` payload uses short, generic `error` strings for users—**not** raw Axios/`response.data` bodies.

### TTL Configuration

<Info>
  Signed OAuth state TTL: **15 minutes** (`EMAIL_OAUTH_STATE_TTL_MS`), aligned with calendar UX.
</Info>

## API Endpoints

### List Personal Integrations

```http
GET /v1/email-integrations
```

Returns **PERSONAL** mailboxes for the current user (Account settings tab).

**Access**: Never lists another member's personal mailboxes, even for `SYSTEM_OWNER`/`SYSTEM_ADMIN`

**Query parameters**:
- `cursor` (optional) - Opaque pagination cursor
- `limit` (optional) - Default: 20, Max: 50

**Response**:
```json
{
  "items": [...],
  "nextCursor": "opaque_string"
}
```

**Ordering**: `connected_at DESC`, `id ASC` (keyset pagination)

**Constants**: 
- `EMAIL_INTEGRATION_PAGE_DEFAULT` (20)
- `EMAIL_INTEGRATION_PAGE_MAX` (50)
- Location: `email-integration-list.constants.ts`

### List Send Accounts

```http
GET /v1/email-integrations/send-accounts
```

Returns mailboxes allowed as **From** address in composer:

<Tabs>
  <Tab title="Personal Stream">
    Caller's PERSONAL rows in `personal` array
    
    - Query param: `cursorPersonal`
    - Same pagination limits as main list endpoint
  </Tab>
  
  <Tab title="Organization Stream">
    ORGANIZATION rows in `organization` array (admin/owner only)
    
    - Query param: `cursorOrganization`
    - Ordering: `is_default DESC`, `connected_at DESC`, `id ASC`
    - Uses offset pagination
  </Tab>
</Tabs>

**Response structure**:
```json
{
  "personal": {
    "items": [...],
    "nextCursor": "..."
  },
  "organization": {
    "items": [...],
    "nextCursor": "..."
  }
}
```

**Exhausted stream handling**:

When one stream completes before the other, client must pass literal `__exhausted__` (`EMAIL_INTEGRATION_MAILBOX_STREAM_EXHAUSTED_QUERY`) for that stream's cursor on subsequent requests to prevent restart.

### List Organization Integrations

```http
GET /v1/email-integrations/organization
```

Returns **ORGANIZATION** mailboxes only.

**Access**: Requires `@CheckAccess` (admin/owner)

**Query parameters**: Same as personal list endpoint

**Populated relations**: Includes `connectedBy` to show who ran OAuth

**Service method**: `findOrganizationMailboxesPage` with `populate(['connectedBy'])`

### Get Integration by ID

```http
GET /v1/email-integrations/:id
```

**Access control**: Uses `EmailIntegrationService.getAccessibleIntegrationFindWhere`:
- **PERSONAL**: Matches only when `user_id` equals JWT user
- **ORGANIZATION**: Requires `SYSTEM_ADMIN` or `SYSTEM_OWNER`
- Missing access returns **404** (same as unknown ID)

Secondary check: `assertCanAccessIntegration`

### Update Integration

```http
PUT /v1/email-integrations/:id
```

**Access control**: Same rules as GET by ID

**Request body**: `UpdateEmailIntegrationDto` (includes `metadata` field)

### Delete Integration

```http
DELETE /v1/email-integrations/:id
```

**Access control**: Same rules as GET by ID

Soft delete: Sets `is_deleted = true`

### Send Email

```http
POST /v1/email-integrations/:id/send
```

**Access control**:

<CardGroup cols={2}>
  <Card title="Personal Send" icon="user">
    Only mailbox owner may send
    
    - `user_id` must match JWT user
  </Card>
  
  <Card title="Organization Send" icon="building">
    Admin/owner only
    
    - Requires `SYSTEM_ADMIN` or `SYSTEM_OWNER`
  </Card>
</CardGroup>

**Service methods**: 
- `assertCanAccessIntegration`
- `sendWithIntegration`

### Start Organization OAuth

```http
POST /v1/email-integrations/organization/oauth/:provider/authorize
```

Initiates OAuth for workspace mailbox.

**Access**: Requires `@CheckAccess` (admin/owner)

**Provider**: `gmail` | `outlook`

## Data Transfer Objects

### EmailIntegrationDto

```typescript
interface EmailIntegrationDto {
  id: string;
  level: 'PERSONAL' | 'ORGANIZATION'; // Required
  provider: 'gmail' | 'outlook';
  email: string;
  isDefault?: boolean; // Org rows only
  user?: UserDto; // When relation populated
  connectedBy?: UserDto; // When relation populated
  // ... other fields
}
```

<Note>
  `EmailIntegrationDto.user` is included only when the `user` relation is **populated**. `GET /v1/email-integrations` typically returns integrations without nested `user`—never deserialize `UserDto` from an uninitialized MikroORM reference.
</Note>

**SPA types**: Mirror these DTOs in `email-integration.api.ts`, including `UpdateEmailIntegrationDto.metadata` for `PUT` requests.

## Data Model

### Level Configuration

```sql
level: 'PERSONAL' | 'ORGANIZATION'
```

Enforced by Postgres CHECK constraint + application validation.

### Personal Level Schema

<Tabs>
  <Tab title="Fields">
    - `user_id`: NOT NULL
    - `connected_by_id`: NULL
    - `email`: Normalized to lowercase
    - `level`: 'PERSONAL'
  </Tab>
  
  <Tab title="Uniqueness">
    Partial unique constraint:
    
    ```sql
    (user_id, organization_id, provider)
    WHERE is_deleted = false AND level = 'PERSONAL'
    ```
  </Tab>
</Tabs>

### Organization Level Schema

<Tabs>
  <Tab title="Fields">
    - `user_id`: NULL
    - `connected_by_id`: NOT NULL (OAuth actor)
    - `email`: Normalized to lowercase
    - `level`: 'ORGANIZATION'
    - `is_default`: Optional boolean
  </Tab>
  
  <Tab title="Uniqueness">
    Partial unique constraint:
    
    ```sql
    (organization_id, provider, lower(email))
    WHERE is_deleted = false AND level = 'ORGANIZATION'
    ```
    
    Additional constraint for default:
    ```sql
    (organization_id)
    WHERE is_deleted = false AND is_default = true
    ```
    
    At most one default row per organization.
  </Tab>
</Tabs>

### Cross-Level Uniqueness

<Warning>
  Active rows may not mix PERSONAL and ORGANIZATION for the same `organization_id + lower(email)`.
</Warning>

**Enforcement**: Migration `Migration20260521171000_PreventEmailIntegrationCrossLevelMailboxConflicts` installs trigger `email_integrations_cross_level_mailbox_conflict_trigger`.

**Mechanism**: Transaction advisory lock per organization/mailbox before checking opposite level prevents concurrent split personal/workspace connections.

**Existing duplicates**: Personal-personal duplicates remain valid.

### Token Storage

All tokens encrypted with AES-256-GCM:

```typescript
{
  encrypted_access_token: string;
  encrypted_refresh_token: string;
}
```

**Encryption service**: Shared `EncryptionService`

**Configuration**: 
- Environment variable: `ENCRYPTION_KEY`
- App config: `app.encryption.key`
- Format: 64 hexadecimal characters

### Display Fields

```sql
email: VARCHAR -- Provider mailbox/profile email (lowercase on write)
```

## Integration Health

An integration is considered **healthy** when all conditions are met:

<Check>Active (not deleted)</Check>
<Check>No `last_sync_error_at` value</Check>
<Check>Access token present</Check>
<Check>`token_expires_at` beyond configured buffer</Check>

**Buffer configuration**: `EMAIL_RECONNECT_ACCESS_TOKEN_BUFFER_MINUTES`

**Health check method**: `integrationNeedsRepairOrTokenRefresh`

Unhealthy integrations allow re-OAuth to repair or refresh tokens, but cannot bypass cross-level mailbox rules.
