</Note>

## Overview

The email integration system supports two levels of mailbox connections:

- **Personal**: Individual user mailboxes connected by and accessible only to that user
- **Organization**: Workspace-wide mailboxes connected by admins/owners and usable by authorized members

Both levels use OAuth 2.0 flows with secure token storage and automatic refresh capabilities.

## OAuth Flow

<Steps>
  <Step title="Tenant Session Setup">
    User selects an organization and the SPA holds a tenant JWT for authenticated API calls.
  </Step>

  <Step title="Initiate OAuth">
    Depending on the integration level, call one of these endpoints:

    **Personal Mailbox** (`EmailIntegrationLevel.PERSONAL`):
    ```http
    POST /v1/email-integrations/oauth/:provider/authorize
    ```

    **Organization Mailbox** (`EmailIntegrationLevel.ORGANIZATION`):
    ```http
    POST /v1/email-integrations/organization/oauth/:provider/authorize
    ```

    <Info>
      Organization-level OAuth requires `SYSTEM_ADMIN` or `SYSTEM_OWNER` role.
    </Info>

    The `:provider` parameter accepts `gmail` or `outlook`.
  </Step>

  <Step title="Receive Authorization URL">
    Backend returns:
    ```json
    {
      "authorizationUrl": "https://accounts.google.com/o/oauth2/v2/auth?..."
    }
    ```

    The `state` parameter in this URL is HMAC-signed and includes:
    - `userId`
    - `organizationId`
    - `tokenVersion` (invalidates OAuth popups after password reset/logout)
    - `provider`
    - `level` (PERSONAL or ORGANIZATION)

    <Warning>
      The `tokenVersion` ensures that already-open OAuth popups become invalid after authentication epoch changes.
    </Warning>
  </Step>

  <Step title="User Authorization">
    Open the `authorizationUrl` in a popup window where the user grants permissions to the provider (Google/Microsoft).
  </Step>

  <Step title="Provider Callback">
    Provider redirects to public callback endpoints (no `/v1` prefix):
    - Gmail: `GET /auth/gmail/callback`
    - Outlook: `GET /auth/outlook/callback`

    Backend verifies the signed `state` and exchanges the authorization code for tokens.
  </Step>

  <Step title="Gmail Access Verification">
    <Warning>
      Gmail-specific verification step to prevent broken integrations.
    </Warning>

    Before saving the integration, backend calls:
    ```http
    GET https://gmail.googleapis.com/gmail/v1/users/me/profile
    ```

    If Google returns 400 or 403, the integration is **not persisted** and the callback returns:
    ```javascript
    {
      error: "GMAIL_ACCESS_BLOCKED",
      message: "This Google account does not provide Gmail API access"
    }
    ```

    This prevents saving integrations for Google accounts that authenticate but don't expose Gmail mailbox access. Outlook skips this verification.

    <Info>
      Gmail OAuth must include both `gmail.send` and `gmail.metadata` scopes. The `userinfo.email` scope reads the profile mailbox address.
    </Info>
  </Step>

  <Step title="Callback Access Re-check">
    Before persisting tokens, backend re-validates in the same database transaction:

    - `User.isActive` status
    - Matching `User.tokenVersion` from signed state
    - Active (non-deleted) `user_org_roles` row for `(userId, organizationId)`
    - For **ORGANIZATION** level only: caller still holds `system.admin` or `system.owner` role

    <Note>
      This is handled by `AccessService.assertOAuthCallbackOrgAccessInTransaction`.
    </Note>

    If validation fails, the popup receives an `oauth-error` with `OAUTH_CALLBACK_TENANT_ACCESS_DENIED_MESSAGE` and tokens are not written.
  </Step>

  <Step title="Token Persistence">
    If all checks pass, `connectInTransaction` runs to:
    - Encrypt access and refresh tokens (AES-256-GCM)
    - Upsert the `email_integrations` row
    - Apply uniqueness constraints (see Data Model section)
  </Step>

  <Step title="Popup Communication">
    Response HTML uses `postMessage` with `app.postMessageTargetOrigin` (from `FRONTEND_URL`) to communicate results to the SPA.

    <Warning>
      Never uses wildcard (`*`) as the target origin for security.
    </Warning>

    Payload includes `level` for proper SPA handling.
  </Step>

  <Step title="SPA Refresh">
    On success, invalidate `queryKeys.emailIntegrations` to refresh:
    - `GET /v1/email-integrations/send-accounts` (composer)
    - Integration lists

    <Tip>
      Validate `postMessage` only when `event.source` matches the OAuth popup. Refetch when popup closes to handle dropped messages.
    </Tip>

    If error is `GMAIL_ACCESS_BLOCKED`, display an `AlertDialog` explaining the limitation.
  </Step>
</Steps>

## Mailbox Uniqueness Rules

<CardGroup cols={2}>
  <Card title="Organization Level" icon="building">
    - `user_id = null`
    - `connected_by_id` = OAuth actor
    - Unique: `(organization_id, provider, lower(email))`
    - Only one default mailbox per organization
  </Card>

  <Card title="Personal Level" icon="user">
    - `user_id` NOT NULL
    - `connected_by_id = null`
    - Unique: `(user_id, organization_id, provider)`
    - Multiple users can connect the same address
  </Card>
</CardGroup>

### Cross-Level Conflict Prevention

A mailbox address cannot exist at both levels simultaneously in the same organization:

- **PERSONAL** OAuth is rejected when an active **ORGANIZATION** row already has the same normalized `email`
- **ORGANIZATION** OAuth is rejected when any active **PERSONAL** row already has that normalized `email`
- Provider is intentionally ignored for this cross-level rule

<Info>
  The migration `Migration20260521171000_PreventEmailIntegrationCrossLevelMailboxConflicts` installs a database trigger using advisory locks to prevent concurrent insert races.
</Info>

### Redundant Reconnect Protection

If OAuth returns the same email address for an existing, healthy integration, `connectInTransaction` throws an error:

**Healthy integration criteria:**
- Row is active (not deleted)
- No `last_sync_error_at` timestamp
- Access token present
- `token_expires_at` beyond the configured buffer (`EMAIL_RECONNECT_ACCESS_TOKEN_BUFFER_MINUTES`)

<Tip>
  Unhealthy rows (inactive, sync error, missing token, or expired/near-expiry) allow re-OAuth to repair or refresh.
</Tip>

## API Endpoints

### List Personal Integrations

<CodeGroup>
```http Request
GET /v1/email-integrations
```

```json Response
{
  "items": [
    {
      "id": "uuid",
      "level": "PERSONAL",
      "provider": "gmail",
      "email": "user@example.com",
      "connectedAt": "2024-01-15T10:30:00Z"
    }
  ],
  "nextCursor": "opaque_cursor_string"
}
```
</CodeGroup>

Returns **PERSONAL** mailboxes for the current user only. Admins/owners cannot view other members' personal integrations.

**Query Parameters:**
- `cursor` (optional): Opaque pagination cursor from previous response
- `limit` (optional): Results per page (default 20, max 50)

**Ordering:** `connected_at DESC`, `id ASC` (keyset pagination)

<Note>
  This endpoint never lists another member's personal mailboxes, regardless of the caller's role.
</Note>

### List Send Accounts

<CodeGroup>
```http Request
GET /v1/email-integrations/send-accounts?cursorPersonal=...&cursorOrganization=...
```

```json Response
{
  "personal": {
    "items": [...],
    "nextCursor": "cursor_or_null"
  },
  "organization": {
    "items": [...],
    "nextCursor": "cursor_or_null"
  }
}
```
</CodeGroup>

Returns mailboxes allowed as **From** addresses in the composer:
- `personal`: Caller's **PERSONAL** rows
- `organization`: **ORGANIZATION** rows (requires `SYSTEM_ADMIN` or `SYSTEM_OWNER`)

**Query Parameters:**
- `cursorPersonal` (optional): Cursor for personal stream
- `cursorOrganization` (optional): Cursor for organization stream
- `limit` (optional): Results per stream (default 20, max 50)

<Tip>
  When one stream is exhausted, pass `__exhausted__` as its cursor on subsequent requests to prevent restarting that stream.
</Tip>

### List Organization Integrations

<CodeGroup>
```http Request
GET /v1/email-integrations/organization
```

```json Response
{
  "items": [
    {
      "id": "uuid",
      "level": "ORGANIZATION",
      "provider": "outlook",
      "email": "support@company.com",
      "isDefault": true,
      "connectedBy": {
        "id": "user_uuid",
        "email": "admin@company.com",
        "name": "Admin User"
      },
      "connectedAt": "2024-01-15T10:30:00Z"
    }
  ],
  "nextCursor": "opaque_cursor_string"
}
```
</CodeGroup>

Lists **ORGANIZATION** mailboxes only. Requires `SYSTEM_ADMIN` or `SYSTEM_OWNER` role.

**Ordering:** `is_default DESC`, `connected_at DESC`, `id ASC` (offset pagination)

<Info>
  Uses `findOrganizationMailboxesPage` with `populate(['connectedBy'])` to show who ran OAuth.
</Info>

### Get Integration by ID

```http
GET /v1/email-integrations/:id
```

Returns a single integration if the caller has access:
- **PERSONAL**: Only when `user_id` matches JWT user
- **ORGANIZATION**: Only when caller has `SYSTEM_ADMIN` or `SYSTEM_OWNER`

<Warning>
  Returns 404 for both missing IDs and insufficient access (no distinction for security).
</Warning>

### Update Integration

```http
PUT /v1/email-integrations/:id
```

Supports updating metadata fields. Same access control as GET by ID.

### Delete Integration

```http
DELETE /v1/email-integrations/:id
```

Soft-deletes the integration. Same access control as GET by ID.

### Send Email

```http
POST /v1/email-integrations/:id/send
```

Sends an email using the specified integration.

**Access Control:**
- **Personal**: Only the mailbox owner (`user_id` matches JWT user)
- **Organization**: `SYSTEM_ADMIN` or `SYSTEM_OWNER` only

### Unique Constraints

<Tabs>
  <Tab title="Personal Level">
    ```sql
    UNIQUE (user_id, organization_id, provider)
    WHERE is_deleted = false AND level = 'PERSONAL'
    ```
    
    Allows one integration per user, organization, and provider combination.
  </Tab>

  <Tab title="Organization Level">
    ```sql
    UNIQUE (organization_id, provider, lower(email))
    WHERE is_deleted = false AND level = 'ORGANIZATION'
    ```
    
    Prevents duplicate organization mailboxes (case-insensitive email).
  </Tab>

  <Tab title="Default Mailbox">
    ```sql
    UNIQUE (organization_id)
    WHERE is_deleted = false AND is_default = true
    ```
    
    Ensures only one default mailbox per organization.
  </Tab>

  <Tab title="Cross-Level">
    Database trigger `email_integrations_cross_level_mailbox_conflict_trigger` prevents the same normalized email address from existing at both PERSONAL and ORGANIZATION levels within an organization.
    
    Uses transaction advisory locks to prevent concurrent insert races.
  </Tab>
</Tabs>

### Level-Specific Rules

<CardGroup cols={2}>
  <Card title="PERSONAL Requirements" icon="user">
    - `user_id` NOT NULL
    - `connected_by_id` NULL
    - Multiple users can connect same address
    - Only owner can access/use
  </Card>

  <Card title="ORGANIZATION Requirements" icon="building">
    - `user_id` NULL
    - `connected_by_id` NOT NULL (OAuth actor)
    - One address per org/provider combo
    - Admin/owner required for access
  </Card>
</CardGroup>

## Error Handling

### OAuth Callback Errors

Token exchange failures and IdP errors are logged server-side with provider HTTP status and truncated body previews. The popup `postMessage` uses generic error strings for users.

**Common Error Codes:**

<AccordionGroup>
  <Accordion title="GMAIL_ACCESS_BLOCKED">
    **Constant:** `EMAIL_OAUTH_GMAIL_ACCESS_BLOCKED_CODE`
    
    **When:** Gmail `users.getProfile` returns 400/403 after token exchange
    
    **Action:** Display AlertDialog explaining that this Google account doesn't expose Gmail API access
    
    **Note:** Must stay in sync between backend constants and frontend `email-integration.api.ts`
  </Accordion>

  <Accordion title="OAUTH_CALLBACK_TENANT_ACCESS_DENIED">
    **When:** Callback access re-check fails (user inactive, token version mismatch, missing org role, or insufficient permissions for org-level OAuth)
    
    **Action:** Generic access denied message, tokens not persisted
  </Accordion>

  <Accordion title="Cross-Level Conflict">
    **When:** Attempting to connect a mailbox that already exists at the opposite level
    
    **Action:** Reject OAuth with conflict message
  </Accordion>

  <Accordion title="Redundant Reconnect">
    **When:** OAuth returns the same email for an existing healthy integration
    
    **Action:** Throw error in `connectInTransaction`, show error in popup
  </Accordion>
</AccordionGroup>

## Token Management

### Encryption

All access and refresh tokens are encrypted using AES-256-GCM via `EncryptionService`:

```typescript
// Requires 64 hex character key
ENCRYPTION_KEY=your_64_character_hex_key_here
```

<Warning>
  Never log or expose decrypted tokens. The encryption key must be securely managed and rotated according to your security policy.
</Warning>

### Token Refresh

Tokens are automatically refreshed when:
- `token_expires_at` is within the configured buffer (`EMAIL_RECONNECT_ACCESS_TOKEN_BUFFER_MINUTES`)
- Integration is used for send operations
- Sync operations detect expiration

### Health Checks

An integration is considered **unhealthy** if:
- Row is soft-deleted (`is_deleted = true`)
- `last_sync_error_at` is set
- `encrypted_access_token` is missing
- `token_expires_at` is unknown
- Token is expired or within expiration buffer

<Tip>
  Use `integrationNeedsRepairOrTokenRefresh` to check health status.
</Tip>

## Access Control Summary

<Steps>
  <Step title="OAuth Authorization">
    - **Personal**: Any authenticated user
    - **Organization**: `SYSTEM_ADMIN` or `SYSTEM_OWNER` only
  </Step>

  <Step title="List Operations">
    - **Personal list**: Current user's integrations only
    - **Organization list**: `SYSTEM_ADMIN` or `SYSTEM_OWNER` only
    - **Send accounts**: Personal (own) + Organization (if admin/owner)
  </Step>

  <Step title="Individual Access">
    Resolved via `getAccessibleIntegrationFindWhere`:
    - **Personal**: Only when `user_id` matches JWT user
    - **Organization**: Only when `SYSTEM_ADMIN` or `SYSTEM_OWNER`
    - Returns 404 for both missing IDs and access denial
  </Step>

  <Step title="Send Operations">
    - **Personal**: Only the mailbox owner
    - **Organization**: `SYSTEM_ADMIN` or `SYSTEM_OWNER` only
  </Step>
</Steps>

## Frontend Integration

### Query Keys

Invalidate after OAuth success:

```typescript
queryKeys.emailIntegrations
```

This refreshes:
- `GET /v1/email-integrations` (account settings)
- `GET /v1/email-integrations/send-accounts` (composer)
- `GET /v1/email-integrations/organization` (admin panel)

### PostMessage Security

<Warning>
  Always validate `postMessage` events:
</Warning>

```typescript
window.addEventListener('message', (event) => {
  // Verify origin
  if (event.origin !== expectedOrigin) return;
  
  // Verify source is the OAuth popup
  if (event.source !== oauthPopupWindow) return;
  
  // Process event.data
});
```

### Type Definitions

Ensure `email-integration.api.ts` types mirror `EmailIntegrationDto`:
- Required `level` field (`PERSONAL` | `ORGANIZATION`)
- Optional `isDefault` for organization rows
- Optional `user` / `connectedBy` when relations populated
- `UpdateEmailIntegrationDto.metadata` support

<Check>
  Never deserialize `UserDto` from an uninitialized MikroORM reference when `user` relation is not populated.
