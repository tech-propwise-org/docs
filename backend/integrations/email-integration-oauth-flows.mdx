This specification covers Gmail and Outlook email OAuth integration flows, token storage, refresh mechanisms, and API surfaces for both personal and organization-level email integrations.

<Note>
This documentation covers email OAuth only. Calendar OAuth is documented separately in the Calendar System Deep Dive.
</Note>

## OAuth Flow Overview

The email integration OAuth flow supports two levels of access:
- **Personal**: Individual user mailboxes (`EmailIntegrationLevel.PERSONAL`)
- **Organization**: Workspace-wide mailboxes (`EmailIntegrationLevel.ORGANIZATION`)

<Steps>
<Step title="Initialize OAuth Flow">
User selects an organization and the SPA holds a tenant JWT. Choose the appropriate endpoint:
- `POST /v1/email-integrations/oauth/:provider/authorize` for personal mailboxes
- `POST /v1/email-integrations/organization/oauth/:provider/authorize` for organization mailboxes
</Step>

<Step title="Authorization URL Generation">
Backend returns `{ authorizationUrl }` containing the Google/Microsoft authorize URL with HMAC-signed state including:
- `userId`
- `organizationId` 
- `tokenVersion`
- `provider` (gmail | outlook)
- `level` (PERSONAL | ORGANIZATION, optional for legacy compatibility)
</Step>

<Step title="Provider Authentication">
Browser opens the authorization URL in a popup window for user authentication with the email provider.
</Step>

<Step title="OAuth Callback Processing">
Provider redirects to public callback endpoints:
- `GET /auth/gmail/callback`
- `GET /auth/outlook/callback`

Backend verifies the signed state and exchanges the authorization code.
</Step>

<Step title="Gmail Access Verification">
For Gmail integrations only, backend calls `GET .../gmail/v1/users/me/profile` to verify API access before saving the integration.

<Warning>
If Google returns 400 or 403, the integration is not persisted and the popup receives an `oauth-error` with `EMAIL_OAUTH_GMAIL_ACCESS_BLOCKED_CODE`.
</Warning>
</Step>

<Step title="Access Control Re-verification">
Before token persistence, backend re-checks:
- User is active (`User.isActive`)
- Token version matches signed state
- Active user-organization role exists
- For organization level: user has `system.admin` or `system.owner` permissions
</Step>

<Step title="Token Storage and Response">
If all checks pass, tokens are encrypted and stored. Response HTML uses `postMessage` to communicate results to the SPA.
</Step>
</Steps>

## Access Control Rules

### Personal vs Organization Level

<Tabs>
<Tab title="Personal Integration">
- **Authorization**: Standard `/v1` API authentication
- **Storage**: `user_id` NOT NULL, `connected_by_id` NULL
- **Access**: Only the mailbox owner can access their personal integrations
- **Uniqueness**: `(user_id, organization_id, provider)` for active rows
</Tab>

<Tab title="Organization Integration">
- **Authorization**: Requires `SYSTEM_ADMIN` or `SYSTEM_OWNER` role
- **Storage**: `user_id` NULL, `connected_by_id` NOT NULL (OAuth actor)
- **Access**: Admin/owner users can access organization mailboxes
- **Uniqueness**: `(organization_id, provider, lower(email))` for active rows
</Tab>
</Tabs>

### Cross-Level Mailbox Conflicts

<Warning>
A mailbox email address cannot exist at both personal and organization levels within the same organization. The system prevents:
- Personal OAuth when an active organization row exists for the same email
- Organization OAuth when any active personal row exists for the same email
</Warning>

### Redundant Reconnect Prevention

The system rejects OAuth attempts for healthy existing integrations where:
- OAuth profile email matches existing stored email
- Row is active with no sync errors
- Access token is present and not expired (considers `EMAIL_RECONNECT_ACCESS_TOKEN_BUFFER_MINUTES`)

```bash Gmail
GMAIL_EMAIL_REDIRECT_URI=https://your-backend.com/auth/gmail/callback
```

```bash Outlook
# Default: ${BACKEND_URL}/auth/outlook/callback  
OUTLOOK_REDIRECT_URI=https://your-backend.com/auth/outlook/callback
# Optional override
OUTLOOK_EMAIL_REDIRECT_URI=https://your-backend.com/auth/outlook/callback
```
</CodeGroup>

<Note>
Register these exact redirect URIs in your Google Cloud OAuth client and Azure app configuration.
</Note>

### OAuth State TTL

The signed OAuth state has a 15-minute TTL (`EMAIL_OAUTH_STATE_TTL_MS`) to align with calendar integration UX.

## API Endpoints

### List Email Integrations

<AccordionGroup>
<Accordion title="GET /v1/email-integrations">
Returns personal mailboxes for the current user only.

**Query Parameters:**
- `cursor` (optional): Opaque pagination cursor
- `limit` (optional): Items per page (default 20, max 50)

**Response:**
```json
{
  "items": [
    {
      "id": "uuid",
      "level": "PERSONAL",
      "email": "user@example.com",
      "provider": "gmail",
      "connectedAt": "2024-01-01T00:00:00Z"
    }
  ],
  "nextCursor": "opaque-cursor-string"
}
```

**Ordering:** `connected_at DESC, id ASC` (keyset pagination)
</Accordion>

<Accordion title="GET /v1/email-integrations/send-accounts">
Returns mailboxes available as "From" addresses in the composer.

**Query Parameters:**
- `cursorPersonal` (optional): Pagination cursor for personal mailboxes
- `cursorOrganization` (optional): Pagination cursor for organization mailboxes  
- `limit` (optional): Items per page (default 20, max 50)

**Response:**
```json
{
  "personal": {
    "items": [...],
    "nextCursor": "cursor-or-null"
  },
  "organization": {
    "items": [...],
    "nextCursor": "cursor-or-null"
  }
}
```

<Note>
When a stream is exhausted, pass `__exhausted__` as the cursor for subsequent requests to prevent stream restart.
</Note>
</Accordion>

<Accordion title="GET /v1/email-integrations/organization">
Returns organization-level mailboxes (admin/owner access required).

**Access Control:** `@CheckAccess` - requires `system.admin` or `system.owner`

**Response:** Includes `connectedBy` relation when populated to show who performed the OAuth connection.
</Accordion>
</AccordionGroup>

### Individual Integration Management

Access control for individual integration operations (`GET`, `PUT`, `DELETE`, `POST /send`) uses `getAccessibleIntegrationFindWhere`:

<Tabs>
<Tab title="Personal Access">
- User can only access their own personal integrations
- `user_id` must match JWT user
- Admins/owners cannot access other members' personal integrations
</Tab>

<Tab title="Organization Access">  
- Requires `SYSTEM_ADMIN` or `SYSTEM_OWNER` role
- All qualifying users can access any organization integration
</Tab>
</Tabs>

## Data Model

### Database Schema

<CodeGroup>
```sql Email Integrations Table
-- Core fields
level VARCHAR NOT NULL CHECK (level IN ('PERSONAL', 'ORGANIZATION'))
user_id UUID NULL -- NOT NULL for PERSONAL, NULL for ORGANIZATION  
connected_by_id UUID NULL -- NULL for PERSONAL, NOT NULL for ORGANIZATION
organization_id UUID NOT NULL
provider VARCHAR NOT NULL -- 'gmail' | 'outlook'
email VARCHAR NOT NULL -- normalized to lowercase

-- Token storage (encrypted)
encrypted_access_token TEXT
encrypted_refresh_token TEXT
token_expires_at TIMESTAMP

-- Metadata
is_deleted BOOLEAN DEFAULT FALSE
is_default BOOLEAN -- ORGANIZATION level only
connected_at TIMESTAMP DEFAULT NOW()
last_sync_error_at TIMESTAMP
```

```sql Unique Constraints
-- Personal level uniqueness
UNIQUE (user_id, organization_id, provider) 
WHERE is_deleted = FALSE AND level = 'PERSONAL'

-- Organization level uniqueness  
UNIQUE (organization_id, provider, LOWER(email))
WHERE is_deleted = FALSE AND level = 'ORGANIZATION'

-- Default organization mailbox
UNIQUE (organization_id)
WHERE is_deleted = FALSE AND level = 'ORGANIZATION' AND is_default = TRUE
```
</CodeGroup>

### Cross-Level Conflict Prevention

The system uses a database trigger (`email_integrations_cross_level_mailbox_conflict_trigger`) that:
- Takes a transaction advisory lock per organization/mailbox
- Checks for existing integrations at the opposite level
- Prevents concurrent inserts that would create conflicts

<Info>
Existing personal-personal duplicates (multiple users with same mailbox) remain valid as they don't violate the cross-level rule.
</Info>

## Token Management

### Encryption

All tokens are encrypted using AES-256-GCM via the shared `EncryptionService`:
- Uses `ENCRYPTION_KEY` environment variable (64 hex characters)
- Stores only encrypted tokens in database
- Decrypts tokens when needed for API calls

### Token Refresh

Integration health is determined by:
- Active status (`is_deleted = FALSE`)
- No recent sync errors (`last_sync_error_at`)
- Valid access token present
- Token not expired (considers `EMAIL_RECONNECT_ACCESS_TOKEN_BUFFER_MINUTES`)

<Check>
Unhealthy integrations allow re-OAuth to repair tokens while still respecting cross-level mailbox rules.
</Check>

## Error Handling

### OAuth Callback Errors

<Warning>
Token exchange failures and provider errors are logged server-side with HTTP status and truncated response previews. User-facing errors use generic messages via `postMessage`.
</Warning>

### Gmail Access Verification

Special handling for Gmail accounts that authenticate but don't expose Gmail API access:

```typescript
// Constants must stay synchronized between backend and frontend
EMAIL_OAUTH_GMAIL_ACCESS_BLOCKED_CODE = 'GMAIL_ACCESS_BLOCKED'
```

Required scopes for Gmail:
- `gmail.send` - for outbound email sending
- `gmail.metadata` - for profile access verification
- `userinfo.email` - for reading profile mailbox address

### Frontend Integration

<Steps>
<Step title="PostMessage Handling">
SPA validates `postMessage` events:
- Verify `event.source` is the OAuth popup
- Check `postMessageTargetOrigin` matches `FRONTEND_URL`
- Handle success/error payloads appropriately
</Step>

<Step title="Query Invalidation">
After successful OAuth, invalidate `queryKeys.emailIntegrations` to refresh:
- Composer send account lists
- Integration management interfaces
</Step>

<Step title="Error Display">
Show `AlertDialog` for `GMAIL_ACCESS_BLOCKED` errors with user-friendly messaging about Gmail API access requirements.
</Step>
</Steps>
