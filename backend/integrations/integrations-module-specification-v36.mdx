## Overview

The integrations module provides connectivity to external services and platforms within PropWise CRM. It currently supports email provider integrations (Gmail and Outlook) via OAuth2 authentication, and Tiptap AI for rich text editing capabilities. The module handles OAuth flows, token management, email sending, and secure access to third-party services.

<Info>
**Last Updated:** 2024-01-15  
**Status:** Active
</Info>

## Architecture

The integrations module is organized into three main sub-modules:

<CardGroup cols={3}>
  <Card title="Email Integration" icon="envelope">
    Manages email provider connections and email sending functionality
  </Card>
  <Card title="OAuth" icon="key">
    Calendar OAuth controllers only (`google-calendar`, `outlook-calendar`)
  </Card>
  <Card title="Tiptap AI" icon="wand-magic-sparkles">
    Provides JWT token generation for Tiptap AI services
  </Card>
</CardGroup>

<Note>
**Mail OAuth** lives in the **email-integration** module (`EmailOAuthFlowService`, `EmailOAuthCallbackController`), not in the oauth sub-module.
</Note>

### Key Design Decisions

<AccordionGroup>
  <Accordion title="Provider-agnostic email integration">
    Uses enum-based provider differentiation to support multiple email services (Gmail and Outlook)
  </Accordion>

  <Accordion title="Token refresh automation">
    Implements scheduled token refresh to maintain continuous integration connectivity
  </Accordion>

  <Accordion title="Tenant-aware integrations">
    All integrations are scoped to organizations using tenant context
  </Accordion>

  <Accordion title="Secure token storage">
    OAuth access/refresh tokens are **AES-GCM encrypted** at rest (`encryptedAccessToken` / `encryptedRefreshToken`)
  </Accordion>

  <Accordion title="Personal vs workspace mailboxes">
    **`EmailIntegrationLevel`** distinguishes:
    - **PERSONAL**: Owned by `user_id`
    - **ORGANIZATION**: `user_id` null, `connected_by_id` stores OAuth actor
    
    Partial unique indexes enforce one active personal row per `(user, organization, provider)` and one org mailbox per `(organization, provider, lower(email))`. A cross-level trigger rejects the same normalized mailbox address from being connected as both personal and workspace in the same organization.
  </Accordion>
</AccordionGroup>

## Entities

### EmailIntegration

Represents a connection to Gmail or Outlook for sending mail. Either a **personal** mailbox (`level=PERSONAL`, `user` required) or a **workspace** mailbox (`level=ORGANIZATION`, `user` null, `connectedBy` set).

```typescript
interface EmailIntegration {
  id: string;
  email: string;
  provider: EmailProvider;
  level: 'PERSONAL' | 'ORGANIZATION';
  isDefault?: boolean; // ORGANIZATION: optional default sender for admins
  encryptedAccessToken?: string;
  encryptedRefreshToken?: string;
  scopes?: string;
  externalAccountId?: string;
  lastSyncErrorAt?: Date;
  lastSyncErrorMessage?: string;
  tokenExpiresAt?: Date;
  metadata?: Record<string, any>;
  isActive: boolean;
  lastUsedAt?: Date;
  user?: User | null; // PERSONAL owner; null for ORGANIZATION
  connectedBy?: User | null; // OAuth actor for ORGANIZATION
  organization: Organization;
  isDeleted: boolean;
  createdAt: Date;
  updatedAt: Date;
}
```

#### Unique Constraints

<Tabs>
  <Tab title="Personal">
    **Constraint**: `(user_id, organization_id, provider)`
    
    Applies only where `is_deleted = false` and `level = PERSONAL`
  </Tab>
  <Tab title="Organization">
    **Constraint**: `(organization_id, provider, lower(email))`
    
    At most one `is_default` org mailbox per organization. Applies where `is_deleted = false` and `level = ORGANIZATION`
  </Tab>
  <Tab title="Cross-level Rule">
    The same `organization_id + lower(email)` cannot appear in both **PERSONAL** and **ORGANIZATION** rows, regardless of provider.
    
    <Note>The same mailbox may still be connected by multiple personal users</Note>
  </Tab>
</Tabs>

### EmailProvider Enum

```typescript

  GMAIL = 'gmail',
```

### OAuth Authentication

<Note>
See **`Docs/EMAIL_INTEGRATION_SPECIFICATION.md`** for the canonical mail OAuth flow details including:
- Tenant `POST /v1/email-integrations/oauth/:provider/authorize`
- Signed state bound to `userId` / `organizationId` / `tokenVersion`
- Public `GET /auth/gmail/callback` / `GET /auth/outlook/callback`
- Encrypted token storage
</Note>

<Steps>
  <Step title="Initialize OAuth Flow">
    User initiates OAuth by calling the authorize endpoint for their provider
  </Step>

  <Step title="Redirect to Provider">
    User is redirected to Gmail or Outlook with a signed state parameter
  </Step>

  <Step title="User Grants Access">
    User authenticates with provider and grants requested permissions
  </Step>

  <Step title="Callback Processing">
    Provider redirects to callback URL with authorization code
  </Step>

  <Step title="Token Exchange">
    System exchanges authorization code for access and refresh tokens
  </Step>

  <Step title="Store Integration">
    Tokens are encrypted and stored in the database
  </Step>
</Steps>

## Business Rules

### Email Integrations

<AccordionGroup>
  <Accordion title="One Integration Per Provider">
    At most one active row per `(user, organization, provider)`. Reconnecting updates tokens on the same row.
  </Accordion>

  <Accordion title="Token Expiration">
    Expired tokens are automatically refreshed via scheduled jobs
  </Accordion>

  <Accordion title="Provider Validation">
    Only supported email providers (Gmail, Outlook) are allowed
  </Accordion>

  <Accordion title="Organization Scoping">
    All integrations are scoped to the user's current organization
  </Accordion>

  <Accordion title="Soft Deletion">
    Integrations are soft-deleted to maintain audit history
  </Accordion>

  <Accordion title="Mailbox Address Uniqueness">
    A normalized mailbox address may be connected by multiple **PERSONAL** users, but cannot be connected at **PERSONAL** and **ORGANIZATION** levels in the same organization at the same time
  </Accordion>
</AccordionGroup>

### OAuth Flows

<Check>**State Validation**: OAuth state parameter must contain valid user, organization, provider, and token-version context</Check>

<Check>**Secure Redirects**: Only whitelisted redirect URLs are allowed</Check>

<Check>**Token Exchange**: Authorization codes must be exchanged for tokens immediately</Check>

<Check>**Error Handling**: OAuth errors redirect to frontend with appropriate error messages</Check>

### Calendar Conferencing (Google Meet / Teams)

Calendar drivers can request a conferencing link when a CRM Event meeting channel carries `conferenceType`. See `Docs/CALENDAR_SYSTEM_DEEP_DIVE.md` → "Create event" for the full write-back flow.

<Info>
The driver `CalendarEventPayload` gained:
- `conferenceType`: `google_meet` | `teams`
- `conferenceRequestId`: Stable CRM event id
</Info>

<Tabs>
  <Tab title="Google Meet">
    **Provider**: `google_meet`
    
    - `createEvent` adds `conferenceDataVersion: 1` to request params
    - `toGoogleEvent` emits `conferenceData.createRequest`:
      - `requestId = conferenceRequestId`
      - `conferenceSolutionKey.type = 'hangoutsMeet'`
    - Existing `calendar.events` scope is sufficient — **no re-consent**
    - Mapper already reads `hangoutLink → meetingLink`
  </Tab>
  
  <Tab title="Microsoft Teams">
    **Provider**: `teams`
    
    - `toGraphEvent` sets:
      - `isOnlineMeeting: true`
      - `onlineMeetingProvider: 'teamsForBusiness'`
    - Skips manual link-in-body append
    - Existing `Calendars.ReadWrite` scope is sufficient
    - Mapper already reads `onlineMeeting.joinUrl → meetingLink`
    
    <Warning>
    **Consumer `outlook.com` accounts cannot create Teams meetings**. Graph errors are caught per-integration and surfaced as a sync error without failing event creation.
    </Warning>
  </Tab>
</Tabs>

<Note>
Only the **matching** provider receives the conference request. Other connected providers get the minted link as description text. Drivers remain DB-free and pure.
</Note>
