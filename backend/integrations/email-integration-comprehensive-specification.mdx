# Email Integration Comprehensive Specification

This document provides comprehensive specification for Gmail and Outlook **email** OAuth integration, token management, and API surfaces. Calendar OAuth is documented separately in the calendar system deep dive.

<Note>
This specification covers **email** OAuth only. Calendar OAuth flows are documented in `CALENDAR_SYSTEM_DEEP_DIVE.md`.
</Note>

## OAuth Flow (Tenant Session)

The email OAuth flow operates within a tenant session context where users have already selected an organization and the SPA holds a tenant JWT.

<Steps>
<Step title="User authentication context">
User has selected an organization and the SPA holds a **tenant JWT**.
</Step>

<Step title="Initiate OAuth flow">
Choose between personal or organization-level OAuth:

<Tabs>
<Tab title="Personal OAuth">
**`POST /v1/email-integrations/oauth/:provider/authorize`**
- `provider` = `gmail` | `outlook`
- **Personal** mailbox for the signed-in user (`EmailIntegrationLevel.PERSONAL`)
- Uses same authentication as other `/v1` APIs
</Tab>

<Tab title="Organization OAuth">
**`POST /v1/email-integrations/organization/oauth/:provider/authorize`**
- **Workspace** mailbox (`EmailIntegrationLevel.ORGANIZATION`)
- Requires **`SYSTEM_ADMIN`** or **`SYSTEM_OWNER`** permissions (`@CheckAccess`)
- Uses same callback URLs as personal OAuth
</Tab>
</Tabs>
</Step>

<Step title="Authorization URL generation">
Backend returns `{ authorizationUrl }` containing:
- Google/Microsoft authorize URL with **HMAC-signed** `state`
- `SignedPayloadService` with purpose `email.oauth.state`
- Payload includes: `userId`, `organizationId`, **`tokenVersion`**, `provider`, optional **`level`**
- Omitted `level` is treated as **`PERSONAL`** for legacy signed tokens
- **`tokenVersion`** invalidates OAuth popups on password reset/logout-all/auth epoch bump
</Step>

<Step title="Browser OAuth popup">
Browser opens `authorizationUrl` in a popup window.
</Step>

<Step title="Provider callback">
Provider redirects to **public** callback endpoints:
- **`GET /auth/gmail/callback`**
- **`GET /auth/outlook/callback`**
</Step>

<Step title="State verification and token exchange">
Backend verifies `state` and exchanges `code` via the provider driver.
</Step>

<Step title="Gmail access verification">
<Warning>
Gmail-specific access verification occurs before saving integration.
</Warning>

Before saving to `email_integrations`, backend calls **`GET .../gmail/v1/users/me/profile`** with the new access token.

**Failure handling:**
- If Google returns **400** or **403**, integration is **not persisted**
- Callback HTML posts **`oauth-error`** with:
  - `error: EMAIL_OAUTH_GMAIL_ACCESS_BLOCKED_CODE` (`GMAIL_ACCESS_BLOCKED`)
  - Descriptive `message` for the SPA
- Prevents saving integrations for accounts that authenticate but don't expose Gmail API access

**Outlook:** Skips this verification step.
</Step>

<Step title="Callback access re-check">
Before provider token exchange and again in the same DB transaction as persistence, backend re-checks OAuth actor against **current** org/auth state:

**Validation criteria:**
- **`User.isActive`** status
- Matching **`User.tokenVersion`** from signed state
- Active (non-deleted) **`user_org_roles`** row for `(userId, organizationId)`
- For **`ORGANIZATION`** level only: actor still holds **`system.admin`** or **`system.owner`**

**Implementation:** `AccessService.assertOAuthCallbackOrgAccessInTransaction`

**Failure:** Popup receives **`oauth-error`** with **`OAUTH_CALLBACK_TENANT_ACCESS_DENIED_MESSAGE`** and tokens are **not** written.

**Success:** `connectInTransaction` runs (encrypt tokens, upsert row).
</Step>

<Step title="Data persistence rules">
<Tabs>
<Tab title="Organization level">
- **`user_id = null`**
- **`connected_by_id`** = OAuth actor
- Mailbox **`email`** normalized to lowercase
- Uniqueness: **`(organization_id, provider, lower(email))`** among active org rows
</Tab>

<Tab title="Personal level">
- **Multiple users** may each connect the **same** mailbox address as **PERSONAL**
- DB uniqueness is per user + org + provider
</Tab>

<Tab title="Cross-level conflicts">
A mailbox address cannot exist at both levels in the same organization:
- **`connectInTransaction`** rejects **PERSONAL** OAuth when active **ORGANIZATION** row has same normalized **`email`**
- Rejects **ORGANIZATION** OAuth when any active **PERSONAL** row has same normalized **`email`**
- Provider is intentionally ignored for this cross-level rule
</Tab>

<Tab title="Redundant reconnect">
If OAuth profile **email** matches existing row's stored **`email`** and row is **healthy**:
- Active, no **`last_sync_error_at`**
- Access token present with **`token_expires_at`** beyond buffer
- **`connectInTransaction`** throws and popup gets **`oauth-error`**

**Healthy criteria:** See **`EMAIL_RECONNECT_ACCESS_TOKEN_BUFFER_MINUTES`** / **`integrationNeedsRepairOrTokenRefresh`**
</Tab>
</Tabs>
</Step>

<Step title="Response handling">
Response HTML uses **`postMessage`** with **`app.postMessageTargetOrigin`** (from `FRONTEND_URL`) - never `*`.

Payload may include **`level`** for the SPA.
</Step>

<Step title="SPA integration">
**SPA responsibilities:**
- Invalidate **`queryKeys.emailIntegrations`** after success
- Refresh **`GET /v1/email-integrations/send-accounts`** (composer) and lists
- Treat **`postMessage`** as valid only when **`event.source` is the OAuth popup**
- If **`error === GMAIL_ACCESS_BLOCKED`**, show **`AlertDialog`**
- Refetch when popup closes for UI updates if message was dropped
</Step>
</Steps>

### Gmail Access Verification Details

<Info>
Gmail access verification prevents saving broken integrations for accounts that authenticate but don't expose Gmail API access.
</Info>

**Implementation details:**
- **Constant:** `EMAIL_OAUTH_GMAIL_ACCESS_BLOCKED_CODE` in `src/common/constants/email-oauth.constants.ts`
- **Timing:** Immediately after `exchangeCode`, before `connectInTransaction`
- **Required scopes:**
  - `gmail.send` for outbound send
  - `gmail.metadata` for `users.getProfile` access probe
  - `userinfo.email` for reading profile mailbox address

<Warning>
The constant must stay in sync with frontend `EMAIL_OAUTH_GMAIL_ACCESS_BLOCKED_CODE` in `email-integration.api.ts`.
</Warning>

```bash Gmail Configuration

GMAIL_REDIRECT_URI=

# Optional override
GMAIL_EMAIL_REDIRECT_URI=
```

```bash Outlook Configuration  
# Default: ${BACKEND_URL or http://localhost:$PORT}/auth/outlook/callback
OUTLOOK_REDIRECT_URI=

# Optional override
OUTLOOK_EMAIL_REDIRECT_URI=
```
</CodeGroup>

**Configuration notes:**
- `PORT` defaults to `3002` in app config / `.env.example`
- Set `BACKEND_URL` when API public origin differs from `localhost:$PORT`
- Register exact redirect URIs on Google Cloud OAuth client and Azure app

**Implementation:** `EmailOAuthCallbackController` serves callback endpoints that must match `app.*.emailRedirectUri` defaults.

### Error Handling

<Tabs>
<Tab title="Callback errors">
Token exchange failures and IdP `error` query params are:
- Logged server-side with provider HTTP status and truncated body preview
- Returned to popup via **`postMessage`** with short, generic **`error`** strings
- **Never** expose raw Axios/`response.data` bodies to users
</Tab>

<Tab title="TTL configuration">
**Signed OAuth state TTL:** `EMAIL_OAUTH_STATE_TTL_MS` (15 minutes)
- Aligned with calendar UX expectations
</Tab>
</Tabs>

## API Endpoints & DTOs

### Core Integration APIs

<AccordionGroup>
<Accordion title="GET /v1/email-integrations">
**Purpose:** List **PERSONAL** mailboxes for current user (Account settings)

**Access:** Never lists another member's personal mailboxes, including for `SYSTEM_OWNER`/`SYSTEM_ADMIN`

**Response format:**
```json
{
  "items": [...],
  "nextCursor": "opaque_string_or_null"
}
```

**Pagination:**
- Ordering: `connected_at DESC`, `id ASC` (keyset)
- Query param: `?cursor=` for next page  
- Optional `?limit=` (default **20**, max **50**)
- Limits defined by `EMAIL_INTEGRATION_PAGE_DEFAULT`/`EMAIL_INTEGRATION_PAGE_MAX`
</Accordion>

<Accordion title="GET /v1/email-integrations/send-accounts">
**Purpose:** Mailboxes allowed as **From** in composer

**Response streams:**
- `personal`: Caller's **PERSONAL** rows
- `organization`: **ORGANIZATION** rows (requires `SYSTEM_ADMIN` or `SYSTEM_OWNER`)

**Each stream returns:**
```json
{
  "items": [...],
  "nextCursor": "opaque_string_or_null"  
}
```

**Pagination query params:**
- `cursorPersonal`: Personal stream cursor
- `cursorOrganization`: Organization stream cursor
- Use literal `__exhausted__` (`EMAIL_INTEGRATION_MAILBOX_STREAM_EXHAUSTED_QUERY`) when stream is exhausted

**Ordering:**
- **Personal:** Keyset pagination
- **Organization:** Offset ordering `is_default DESC`, `connected_at DESC`, `id ASC`
</Accordion>

<Accordion title="GET /v1/email-integrations/organization">
**Purpose:** List **ORGANIZATION** mailboxes only

**Access:** `@CheckAccess` (admin/owner required)

**Implementation:** Uses `findOrganizationMailboxesPage` with `populate(['connectedBy'])`

**Response:** Same `{ items, nextCursor }` format with offset pagination and same `limit` bounds
</Accordion>

<Accordion title="POST /v1/email-integrations/organization/oauth/:provider/authorize">
**Purpose:** Start OAuth for workspace mailbox

**Access:** `@CheckAccess` (admin/owner required)

**Parameters:** `provider` = `gmail` | `outlook`
</Accordion>
</AccordionGroup>

### Access Control by ID

**Endpoint pattern:** `GET|PUT|DELETE /v1/email-integrations/:id` and `POST /v1/email-integrations/:id/send`

**Access resolution:** Uses `EmailIntegrationService.getAccessibleIntegrationFindWhere`:

<Tabs>
<Tab title="Personal rows">
- Match only when `user_id` is the JWT user
- Admins/owners **cannot** load another member's integration by UUID
</Tab>

<Tab title="Organization rows">  
- Match only for callers with `SYSTEM_ADMIN` or `SYSTEM_OWNER`
</Tab>

<Tab title="Missing access">
- Returns **404** (same as unknown id)
- `assertCanAccessIntegration` provides secondary check after load
</Tab>
</Tabs>

### Send Access Control

<Warning>
Send permissions are strictly enforced based on integration level.
</Warning>

**Personal send:** Only mailbox owner may use `POST …/:id/send` for `level=PERSONAL` rows (`user_id` must match JWT user)

**Organization send:** `SYSTEM_ADMIN` or `SYSTEM_OWNER` only via `EmailIntegrationService.assertCanAccessIntegration` + `sendWithIntegration`

### DTO Structure

**EmailIntegrationDto includes:**
- **Required:** `level` (`PERSONAL` | `ORGANIZATION`)  
- **Optional:** `isDefault` (org rows only)
- **Optional:** `user`/`connectedBy` when relations are populated

<Note>
`EmailIntegrationDto.user` is included only when the `user` relation is **populated**. `GET /v1/email-integrations` usually returns integrations **without** nested `user` - never deserialize `UserDto` from an uninitialized MikroORM reference.
</Note>

**SPA types:** Mirror DTO structure in `email-integration.api.ts`, including `UpdateEmailIntegrationDto.metadata` for `PUT` operations.

## Data Model

### Level-Based Structure

<Tabs>
<Tab title="Personal level">
**Constraints:**
- `level = 'PERSONAL'`
- `user_id` NOT NULL  
- `connected_by_id` NULL
- Partial unique: `(user_id, organization_id, provider)` where `is_deleted = false`
</Tab>

<Tab title="Organization level">
**Constraints:**
- `level = 'ORGANIZATION'`  
- `user_id` NULL
- `connected_by_id` NOT NULL (OAuth actor)
- Partial unique: `(organization_id, provider, lower(email))` where `is_deleted = false`
- Optional `is_default` with at most one default row per org
- Partial unique on `organization_id` where org + default
</Tab>
</Tabs>

### Cross-Level Uniqueness

<Warning>
Active rows may not mix **PERSONAL** and **ORGANIZATION** for the same `organization_id + lower(email)`.
</Warning>

**Implementation:**
- Migration `Migration20260521171000_PreventEmailIntegrationCrossLevelMailboxConflicts`
- Installs `email_integrations_cross_level_mailbox_conflict_trigger`  
- Takes transaction advisory lock per organization/mailbox
- Checks opposite level before insert to prevent concurrent conflicts
- Existing personal-personal duplicates remain valid

### Token Storage

**Encryption:** All tokens use AES-256-GCM via shared `EncryptionService`
- `encrypted_access_token`
- `encrypted_refresh_token`  
- Key: `ENCRYPTION_KEY` / `app.encryption.key` (64 hex chars)

**Display data:** `email` holds provider mailbox/profile email (normalized to lowercase on write)

### Database Migrations

**Key migration:** `Migration20260508150000_EmailIntegrationOrganizationLevel`
- Introduces `level` column
- Makes `user_id` nullable  
- Adds `connected_by_id` column

<Check>
The data model enforces both application-level and database-level constraints to maintain data integrity across personal and organization email integrations.
</Check>
