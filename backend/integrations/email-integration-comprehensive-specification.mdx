## Overview

This specification covers Gmail and Outlook **mail** OAuth integration, token storage, refresh mechanisms, and API surfaces. Calendar OAuth is documented separately in the Calendar System Deep Dive.

<Info>
This document focuses specifically on email/mail functionality. For calendar integration details, refer to `CALENDAR_SYSTEM_DEEP_DIVE.md`.
</Info>

## OAuth Flow (Tenant Session)

The email integration OAuth flow operates within a tenant session context where users can connect both personal and organization-level mailboxes.

<Steps>
<Step title="Initial Setup">
User selects an organization and the SPA holds a **tenant JWT** for authentication.
</Step>

<Step title="Choose Integration Level">
Two OAuth endpoints are available:
- **Personal mailbox**: `POST /v1/email-integrations/oauth/:provider/authorize`
- **Workspace mailbox**: `POST /v1/email-integrations/organization/oauth/:provider/authorize`

Where `provider` can be `gmail` or `outlook`.
</Step>

<Step title="Authorization Requirements">
- **Personal OAuth**: Uses standard `/v1` API authentication
- **Organization OAuth**: Requires `SYSTEM_ADMIN` or `SYSTEM_OWNER` role via `@CheckAccess`
</Step>

<Step title="Authorization URL Generation">
Backend returns `{ authorizationUrl }` containing:
- Google/Microsoft authorize URL
- HMAC-signed `state` parameter via `SignedPayloadService`
- State payload includes: `userId`, `organizationId`, `tokenVersion`, `provider`, optional `level`
- Omitted `level` defaults to `PERSONAL` for legacy compatibility
</Step>

<Step title="Browser OAuth Flow">
Browser opens `authorizationUrl` in popup window, provider redirects to public callback endpoints:
- Gmail: `GET /auth/gmail/callback`
- Outlook: `GET /auth/outlook/callback`
</Step>

<Step title="Backend Processing">
Backend verifies state, exchanges authorization code via provider driver, and performs additional validation steps.
</Step>
</Steps>

### Gmail Access Verification

<Warning>
Gmail integrations undergo additional verification to prevent saving broken connections.
</Warning>

Before persisting Gmail integrations, the backend calls `GET .../gmail/v1/users/me/profile` with the new access token. If Google returns 400 or 403, the integration is **not persisted** and the callback returns an OAuth error.

**Required Scopes:**
- `gmail.send` - For outbound email sending
- `gmail.metadata` - For profile access verification
- `userinfo.email` - For reading profile mailbox address

**Error Handling:**
```typescript
// Constant synchronization required
EMAIL_OAUTH_GMAIL_ACCESS_BLOCKED_CODE = 'GMAIL_ACCESS_BLOCKED'
```

### Callback Access Verification

<Check>
All OAuth callbacks undergo comprehensive access re-verification before token persistence.
</Check>

The system performs the following checks:
- User account is active (`User.isActive`)
- Token version matches signed state (`User.tokenVersion`)
- Active `user_org_roles` relationship exists
- For `ORGANIZATION` level: user maintains admin/owner permissions

If verification fails, the popup receives `oauth-error` with `OAUTH_CALLBACK_TENANT_ACCESS_DENIED_MESSAGE`.

### Data Persistence Rules

<Tabs>
<Tab title="Organization Level">
- `user_id = null`
- `connected_by_id` = OAuth actor
- Mailbox `email` normalized to lowercase
- Uniqueness: `(organization_id, provider, lower(email))` among active rows
</Tab>

<Tab title="Personal Level">
- Multiple users may connect the same mailbox address
- DB uniqueness: per user + org + provider
- Cross-level conflict prevention applies
</Tab>
</Tabs>

### Cross-Level Mailbox Rules

<Warning>
A mailbox address cannot exist at both PERSONAL and ORGANIZATION levels within the same organization.
</Warning>

**Conflict Prevention:**
- PERSONAL OAuth rejected if active ORGANIZATION row exists for same normalized email
- ORGANIZATION OAuth rejected if any active PERSONAL row exists for same normalized email
- Provider intentionally ignored for cross-level conflict checking

**Redundant Reconnect Prevention:**
OAuth is rejected when the target integration is still "healthy":
- Row is active
- No `last_sync_error_at`
- Access token present with valid expiry beyond buffer period
- Buffer configured via `EMAIL_RECONNECT_ACCESS_TOKEN_BUFFER_MINUTES`

### PostMessage Communication

<CodeGroup>
```javascript Response Payload
{
  type: 'oauth-success',
  level?: 'PERSONAL' | 'ORGANIZATION'
}
```

```javascript Error Payload
{
  type: 'oauth-error',
  error: 'GMAIL_ACCESS_BLOCKED' | 'OAUTH_CALLBACK_TENANT_ACCESS_DENIED_MESSAGE',
  message?: string
}
```
</CodeGroup>

**Security:**
- Uses `app.postMessageTargetOrigin` from `FRONTEND_URL`
- Never uses wildcard (`*`) origin
- SPA validates `event.source` matches OAuth popup

### Environment Configuration

</Accordion>

</Accordion>
</AccordionGroup>

<Note>
Register exact redirect URIs in Google Cloud OAuth client and Azure app configurations used by `GMAIL_*` and `OUTLOOK_*` environment variables.
</Note>

### OAuth State TTL

Signed OAuth state expires after **15 minutes** (`EMAIL_OAUTH_STATE_TTL_MS`), aligned with calendar integration UX patterns.

## API Endpoints & DTOs

### Core Endpoints

<CardGroup cols={2}>
<Card title="Personal Integrations" icon="user">
`GET /v1/email-integrations`

Lists PERSONAL mailboxes for current user only. Never exposes other users' personal mailboxes.
</Card>

<Card title="Send Accounts" icon="paper-plane">
`GET /v1/email-integrations/send-accounts`

Returns mailboxes available as "From" addresses in composer.
</Card>

<Card title="Organization Integrations" icon="building">
`GET /v1/email-integrations/organization`

Lists ORGANIZATION mailboxes (admin/owner access required).
</Card>

<Card title="Organization OAuth" icon="key">
`POST /v1/email-integrations/organization/oauth/:provider/authorize`

Initiates workspace mailbox OAuth (admin/owner access required).
</Card>
</CardGroup>

### Response Structure

All list endpoints return paginated responses:

```typescript
{
  items: EmailIntegrationDto[],
  nextCursor: string | null
}
```

**Pagination Parameters:**
- `limit`: Default 20, max 50 (via `EMAIL_INTEGRATION_PAGE_DEFAULT`/`EMAIL_INTEGRATION_PAGE_MAX`)
- `cursor`: Opaque string for next page
- Ordering: `connected_at DESC, id ASC` (keyset pagination)

### Send Accounts Endpoint

The `/send-accounts` endpoint provides specialized pagination for composer integration:

<Tabs>
<Tab title="Personal Stream">
- Returns caller's PERSONAL integrations
- Uses keyset pagination
- Query param: `cursorPersonal`
</Tab>

<Tab title="Organization Stream">
- Returns ORGANIZATION integrations (admin/owner only)
- Uses offset pagination with ordering: `is_default DESC, connected_at DESC, id ASC`
- Query param: `cursorOrganization`
</Tab>
</Tabs>

<Warning>
When one stream is exhausted, clients must pass `__exhausted__` (`EMAIL_INTEGRATION_MAILBOX_STREAM_EXHAUSTED_QUERY`) for that stream's cursor to prevent restart.
</Warning>

### EmailIntegrationDto Structure

```typescript
interface EmailIntegrationDto {
  id: string;
  level: 'PERSONAL' | 'ORGANIZATION';
  provider: 'gmail' | 'outlook';
  email: string;
  isDefault?: boolean; // Organization rows only
  user?: UserDto; // When relation populated
  connectedBy?: UserDto; // When relation populated
  // ... other fields
}
```

<Info>
The `user` field is included only when the `user` relation is populated. Never deserialize `UserDto` from uninitialized MikroORM references.
</Info>

## Access Control

### Read/Mutate by ID

Individual integration operations (`GET`, `PUT`, `DELETE`, `POST /:id/send`) use `EmailIntegrationService.getAccessibleIntegrationFindWhere`:

<Tabs>
<Tab title="Personal Access">
- PERSONAL rows: Only when `user_id` matches JWT user
- Admins/owners **cannot** access other users' personal integrations
</Tab>

<Tab title="Organization Access">
- ORGANIZATION rows: Only for `SYSTEM_ADMIN` or `SYSTEM_OWNER`
- Returns 404 for missing access (same as unknown ID)
</Tab>
</Tabs>

### Send Operation Access

<Check>
Send operations have strict access controls based on integration level.
</Check>

**Personal Send:**
- Only mailbox owner can use `POST /:id/send` for `level=PERSONAL`
- `user_id` must match JWT user

**Organization Send:**
- Requires `SYSTEM_ADMIN` or `SYSTEM_OWNER` role
- Enforced via `EmailIntegrationService.assertCanAccessIntegration`

## Data Model

### Core Schema

```sql
-- Level constraint
level: 'PERSONAL' | 'ORGANIZATION' -- Postgres check + application constraint

-- Personal integrations
user_id: NOT NULL (for PERSONAL)
connected_by_id: NULL (for PERSONAL)

-- Organization integrations  
user_id: NULL (for ORGANIZATION)
connected_by_id: NOT NULL (OAuth actor, for ORGANIZATION)
```

### Uniqueness Constraints

<AccordionGroup>
<Accordion title="Personal Level">
Partial unique constraint: `(user_id, organization_id, provider)` where `is_deleted = false` and `level = 'PERSONAL'`
</Accordion>

<Accordion title="Organization Level">
Partial unique constraint: `(organization_id, provider, lower(email))` where `is_deleted = false` and `level = 'ORGANIZATION'`

Optional `is_default` field with at most one default per organization.
</Accordion>

<Accordion title="Cross-Level Prevention">
Database trigger `email_integrations_cross_level_mailbox_conflict_trigger` prevents mixing PERSONAL and ORGANIZATION levels for same `organization_id + lower(email)`.

Uses transaction advisory locks to prevent race conditions during concurrent inserts.
</Accordion>
</AccordionGroup>

### Token Storage

<Warning>
All tokens are encrypted at rest using AES-256-GCM encryption.
</Warning>

**Encrypted Fields:**
- `encrypted_access_token`
- `encrypted_refresh_token`

**Encryption Service:**
- Uses shared `EncryptionService`
- Key: `ENCRYPTION_KEY` / `app.encryption.key` (64 hex characters)
- Algorithm: AES-256-GCM

### Display Fields

- `email`: Provider mailbox/profile email (normalized to lowercase on write)
- `provider`: 'gmail' | 'outlook'
- `level`: 'PERSONAL' | 'ORGANIZATION'
- `is_default`: Boolean (organization integrations only)

## Integration Patterns

### Frontend Integration

<Steps>
<Step title="Query Invalidation">
After successful OAuth, invalidate `queryKeys.emailIntegrations` to refresh:
- `GET /v1/email-integrations/send-accounts` (composer)
- Integration lists
</Step>

<Step title="PostMessage Validation">
```javascript
// Validate message source
if (event.source !== oauthPopup) return;

// Handle success/error
if (data.type === 'oauth-success') {
  // Refresh integrations
} else if (data.error === 'GMAIL_ACCESS_BLOCKED') {
  // Show AlertDialog
}
```
</Step>

<Step title="Fallback Refresh">
Always refetch when popup closes in case postMessage was dropped due to URL mismatches (LAN vs localhost).
</Step>
</Steps>

### Error Handling

**Server-side Logging:**
- Token exchange failures logged with provider HTTP status
- Truncated response body previews for debugging
- Never expose raw Axios/response data to users

**Client-side Errors:**
- Generic, user-friendly error strings in postMessage
- Specific error codes for programmatic handling
- AlertDialog integration for Gmail access blocks

<Note>
Maintain constant synchronization between backend `EMAIL_OAUTH_GMAIL_ACCESS_BLOCKED_CODE` and frontend error handling constants.
</Note>
