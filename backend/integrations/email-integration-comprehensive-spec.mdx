# Email Integration Comprehensive Specification

This specification covers Gmail and Outlook **email** OAuth integration, token storage, refresh mechanisms, and API surfaces. 

<Note>
Calendar OAuth is documented separately in `CALENDAR_SYSTEM_DEEP_DIVE.md`.
</Note>

## OAuth Flow Overview

The email integration supports two levels of access:
- **Personal mailbox** (`EmailIntegrationLevel.PERSONAL`) - Individual user's email account
- **Workspace mailbox** (`EmailIntegrationLevel.ORGANIZATION`) - Shared organization email account

<Steps>
<Step title="Initialize OAuth Session">
User selects an organization and the SPA holds a **tenant JWT**.
</Step>

<Step title="Start Authorization">
Choose the appropriate endpoint based on integration level:

**Personal Mailbox:**
```http
POST /v1/email-integrations/oauth/:provider/authorize
```

**Workspace Mailbox:**
```http
POST /v1/email-integrations/organization/oauth/:provider/authorize
```

<Warning>
Workspace mailbox OAuth requires `SYSTEM_ADMIN` or `SYSTEM_OWNER` permissions via `@CheckAccess`.
</Warning>
</Step>

<Step title="Receive Authorization URL">
Backend returns `{ authorizationUrl }` with HMAC-signed `state` parameter containing:
- `userId`
- `organizationId` 
- `tokenVersion` (for invalidating OAuth popups after auth changes)
- `provider` (gmail | outlook)
- `level` (PERSONAL | ORGANIZATION, defaults to PERSONAL for legacy tokens)
</Step>

<Step title="Handle Provider Callback">
Provider redirects to public callback endpoints:
- `GET /auth/gmail/callback`
- `GET /auth/outlook/callback`

Backend verifies `state` and exchanges authorization code via provider driver.
</Step>

<Step title="Gmail Access Verification">
<Warning>
Gmail-specific step: Before saving integration, backend calls `GET .../gmail/v1/users/me/profile` with the new access token. If Google returns 400 or 403, the integration is **not persisted** and callback returns `oauth-error` with `GMAIL_ACCESS_BLOCKED` error code.
</Warning>

This prevents saving integrations for Google accounts that authenticate but don't expose Gmail API access.
</Step>

<Step title="Final Access Validation">
Before token exchange and during persistence transaction, backend re-checks:
- `User.isActive`
- Matching `User.tokenVersion` from signed state
- Active `user_org_roles` row for `(userId, organizationId)`
- For ORGANIZATION level: current `system.admin` or `system.owner` permissions

If validation fails, popup receives `oauth-error` with `OAUTH_CALLBACK_TENANT_ACCESS_DENIED_MESSAGE`.
</Step>

<Step title="Token Storage">
Successful OAuth saves encrypted tokens and updates integration record with conflict resolution rules.
</Step>

<Step title="Frontend Notification">
Response HTML uses `postMessage` with `app.postMessageTargetOrigin` (never `*`) to notify the SPA of success or failure.
</Step>
</Steps>

## Data Model & Constraints

### Integration Levels

<Tabs>
<Tab title="Personal Level">
- `user_id` NOT NULL
- `connected_by_id` NULL
- Unique constraint: `(user_id, organization_id, provider)` where `is_deleted = false` and `level = 'PERSONAL'`
- Multiple users may connect the same mailbox address as personal integrations
</Tab>

<Tab title="Organization Level">
- `user_id` NULL  
- `connected_by_id` NOT NULL (OAuth actor)
- Unique constraint: `(organization_id, provider, lower(email))` where `is_deleted = false` and `level = 'ORGANIZATION'`
- Optional `is_default` flag with at most one default per organization
- Stores mailbox `email` normalized to lowercase
</Tab>
</Tabs>

### Cross-Level Uniqueness

<Warning>
A mailbox address cannot exist at both PERSONAL and ORGANIZATION levels within the same organization. The system enforces this via:

1. Database trigger `email_integrations_cross_level_mailbox_conflict_trigger`
2. Transaction advisory locks per organization/mailbox  
3. Application-level validation in `connectInTransaction`
</Warning>

### Token Security

All tokens are encrypted using AES-256-GCM via `EncryptionService`:
- `encrypted_access_token`
- `encrypted_refresh_token`
- Encryption key from `ENCRYPTION_KEY` environment variable (64 hex characters)

## API Endpoints

### Personal Integrations

<CodeGroup>
```http GET /v1/email-integrations
GET /v1/email-integrations?limit=20&cursor=eyJpZCI6MTIz...

Response:
{
  "items": [
    {
      "id": "uuid",
      "level": "PERSONAL",
      "provider": "gmail",
      "email": "user@example.com",
      "connectedAt": "2024-01-01T00:00:00Z"
    }
  ],
  "nextCursor": "eyJpZCI6MTI0..."
}
```

```http POST /v1/email-integrations/oauth/:provider/authorize
POST /v1/email-integrations/oauth/gmail/authorize

Response:
{
  "authorizationUrl": "https://accounts.google.com/oauth/authorize?..."
}
```
</Tabs>

Lists **PERSONAL** mailboxes for the current user only. Admins cannot view other users' personal integrations.

**Pagination:**
- Default limit: 20, max limit: 50
- Ordering: `connected_at DESC, id ASC` (keyset pagination)
- Use `nextCursor` for subsequent pages

### Organization Integrations  

<CodeGroup>
```http GET /v1/email-integrations/organization
GET /v1/email-integrations/organization?limit=20&cursor=10

Response:
{
  "items": [
    {
      "id": "uuid", 
      "level": "ORGANIZATION",
      "provider": "outlook",
      "email": "support@company.com",
      "isDefault": true,
      "connectedBy": {
        "id": "user-uuid",
        "name": "Admin User"
      }
    }
  ],
  "nextCursor": "20"
}
```

```http POST /v1/email-integrations/organization/oauth/:provider/authorize
POST /v1/email-integrations/organization/oauth/outlook/authorize

Response: 
{
  "authorizationUrl": "https://login.microsoftonline.com/..."
}
```
</CodeGroup>

<Info>
Organization endpoints require `SYSTEM_ADMIN` or `SYSTEM_OWNER` permissions via `@CheckAccess`.
</Info>

**Pagination:**
- Offset-based pagination
- Ordering: `is_default DESC, connected_at DESC, id ASC`
- Includes `connectedBy` relation showing who performed OAuth

### Send Accounts

```http
GET /v1/email-integrations/send-accounts?cursorPersonal=abc&cursorOrganization=def&limit=20
```

Returns mailboxes available as **From** addresses in email composer:
- `personal`: Caller's PERSONAL integrations  
- `organization`: ORGANIZATION integrations (admin/owner only)

<Tip>
When a stream is exhausted, pass `__exhausted__` as the cursor parameter to prevent restarting that stream.
</Tip>

<AccordionGroup>

```

</Accordion>

```

**Access Control:**
- Personal: Only mailbox owner can send
- Organization: `SYSTEM_ADMIN` or `SYSTEM_OWNER` required
</Accordion>
</AccordionGroup>

## Error Handling

### OAuth Callback Errors

The system handles various OAuth failure scenarios with user-friendly error messages:

<Tabs>
<Tab title="Gmail Access Blocked">
**Error Code:** `GMAIL_ACCESS_BLOCKED`

**Cause:** Google account authenticates but doesn't expose Gmail API access

**Frontend:** Show AlertDialog explaining the limitation
</Tab>

<Tab title="Tenant Access Denied">  
**Message:** `OAUTH_CALLBACK_TENANT_ACCESS_DENIED_MESSAGE`

**Cause:** User lost organization access or required permissions during OAuth flow

**Action:** Token exchange is blocked and integration not saved
</Tab>

<Tab title="Cross-Level Conflict">
**Cause:** Attempting to connect mailbox that exists at opposite integration level

**Action:** OAuth rejected, existing integration remains unchanged
</Tab>

<Tab title="Redundant Reconnect">
**Cause:** OAuth for healthy integration with same email address  

**Action:** OAuth blocked to prevent unnecessary token refresh
</Tab>
</Tabs>

### Token Refresh & Health

The system automatically manages token lifecycle:

<Check>
**Healthy Integration:** Active status, no sync errors, valid access token with sufficient time before expiry
</Check>

**Unhealthy Integration Indicators:**
- Inactive status (`is_deleted = true`)
- Recent sync error (`last_sync_error_at` set)  
- Missing access token
- Token expired or within buffer window (`EMAIL_RECONNECT_ACCESS_TOKEN_BUFFER_MINUTES`)

<Info>
Unhealthy integrations allow re-OAuth for repair, but still enforce cross-level mailbox uniqueness rules.
</Info>

## Security Considerations

### State Parameter Security

OAuth state parameters use HMAC signing via `SignedPayloadService`:
- **Purpose:** `email.oauth.state` 
- **TTL:** 15 minutes (`EMAIL_OAUTH_STATE_TTL_MS`)
- **Payload:** Includes user context and token version for security validation

### Access Control Matrix

| Operation | Personal Level | Organization Level |
|-----------|----------------|-------------------|
| **OAuth Setup** | Any authenticated user | `SYSTEM_ADMIN` or `SYSTEM_OWNER` |
| **View Integration** | Owner only | Admin/Owner only |
| **Send Email** | Owner only | Admin/Owner only |
| **Update/Delete** | Owner only | Admin/Owner only |

### Token Storage

<Warning>
All OAuth tokens are encrypted at rest using AES-256-GCM. The encryption key must be securely managed and consistent across application instances.
</Warning>

### Audit Trail

Organization-level integrations track the connecting user via `connected_by_id` for accountability and audit purposes.
