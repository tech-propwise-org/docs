This specification covers the Gmail and Outlook email OAuth integration system, including authentication flows, token storage, refresh mechanisms, and API endpoints. Calendar OAuth is documented separately.

<Note>
Email integration OAuth is tenant-specific and requires users to have selected an organization context.
</Note>

## OAuth Flow (Tenant Session)

The email integration OAuth flow operates within a tenant context and supports both personal and organization-level mailbox connections.

<Steps>
<Step title="User Authentication">
User has selected an organization and the SPA holds a tenant JWT for authentication.
</Step>

<Step title="Initiate OAuth">
Choose the appropriate endpoint:
- **Personal mailbox**: `POST /v1/email-integrations/oauth/:provider/authorize` 
- **Workspace mailbox**: `POST /v1/email-integrations/organization/oauth/:provider/authorize`

Where `provider` = `gmail` | `outlook`
</Step>

<Step title="Authorization URL Generation">
Backend returns `{ authorizationUrl }` containing a Google/Microsoft authorize URL with HMAC-signed state token.
</Step>

<Step title="OAuth Popup">
Browser opens the authorization URL in a popup window for user consent.
</Step>

<Step title="Provider Callback">
Provider redirects to public callback endpoints:
- `GET /auth/gmail/callback`
- `GET /auth/outlook/callback`
</Step>

<Step title="Gmail Access Verification">
For Gmail integrations only, backend verifies access by calling `GET .../gmail/v1/users/me/profile` with the new access token to ensure Gmail API access is available.
</Step>

<Step title="Access Validation">
Backend re-checks OAuth actor permissions and saves encrypted tokens to database.
</Step>

<Step title="Response Handling">
Callback HTML uses `postMessage` to communicate results back to the SPA, which then refreshes integration lists.
</Step>
</Steps>

### Access Levels

<CardGroup cols={2}>
<Card title="Personal Level" icon="user">
`EmailIntegrationLevel.PERSONAL` - Individual user mailbox connection
</Card>
<Card title="Organization Level" icon="building">
`EmailIntegrationLevel.ORGANIZATION` - Shared workspace mailbox (requires SYSTEM_ADMIN or SYSTEM_OWNER)
</Card>

### State Token Security

The OAuth state parameter includes:
- `userId` and `organizationId` 
- `tokenVersion` for invalidation on logout/password reset
- `provider` (gmail/outlook)
- `level` (PERSONAL/ORGANIZATION)

<Warning>
State tokens have a 15-minute TTL (`EMAIL_OAUTH_STATE_TTL_MS`) and are HMAC-signed using `SignedPayloadService`.
</Warning>

### Gmail Access Verification

<Info>
Gmail OAuth includes additional verification to prevent saving broken integrations for accounts that authenticate but don't expose Gmail API access.
</Info>

The verification process:
1. Occurs immediately after `exchangeCode`, before `connectInTransaction`
2. Calls Gmail's `users.getProfile` endpoint with the new access token
3. If the call returns 400 or 403, the integration is rejected
4. Required scopes: `gmail.send` (outbound), `gmail.metadata` (verification), `userinfo.email` (profile)

### Cross-Level Mailbox Rules

<Tabs>
<Tab title="Personal Connections">
- Multiple users may connect the same mailbox address as PERSONAL
- Database uniqueness: per user + organization + provider
- Cannot exist when same address already connected at ORGANIZATION level
</Tab>
<Tab title="Organization Connections">
- Stored with `user_id = null`, `connected_by_id` = OAuth actor
- Uniqueness: `(organization_id, provider, lower(email))` among active rows
- Cannot exist when same address already connected at PERSONAL level
</Tab>
</Tabs>

## API Endpoints

### Personal Mailboxes

<AccordionGroup>
<Accordion title="GET /v1/email-integrations">
**Description**: List personal mailboxes for the current user

**Access**: User's own PERSONAL integrations only

**Parameters**:
- `?limit=` (default 20, max 50)
- `?cursor=` (opaque pagination token)

**Response**:
```json
{
  "items": [...],
  "nextCursor": "opaque_string_or_null"
}
```

**Ordering**: `connected_at DESC, id ASC` (keyset pagination)
</Accordion>

<Accordion title="GET /v1/email-integrations/send-accounts">
**Description**: Get mailboxes available as "From" addresses in composer

**Access**: Current user's PERSONAL + accessible ORGANIZATION mailboxes

**Parameters**:
- `?limit=` (default 20, max 50)
- `?cursorPersonal=` 
- `?cursorOrganization=`

**Response**:
```json
{
  "personal": { "items": [...], "nextCursor": "..." },
  "organization": { "items": [...], "nextCursor": "..." }
}
```

<Note>
Use literal `__exhausted__` for exhausted stream cursors in subsequent requests.
</Note>
</Accordion>
</AccordionGroup>

### Organization Mailboxes

<AccordionGroup>
<Accordion title="GET /v1/email-integrations/organization">
**Description**: List organization-level mailboxes

**Access**: `@CheckAccess` - SYSTEM_ADMIN or SYSTEM_OWNER required

**Response**: Includes `connectedBy` relation showing who performed OAuth
</Accordion>

<Accordion title="POST /v1/email-integrations/organization/oauth/:provider/authorize">
**Description**: Start OAuth flow for workspace mailbox

**Access**: `@CheckAccess` - SYSTEM_ADMIN or SYSTEM_OWNER required

**Parameters**: 
- `provider`: `gmail` | `outlook`
</Accordion>
</AccordionGroup>

### Individual Operations

<AccordionGroup>
<Accordion title="GET /v1/email-integrations/:id">
**Access Control**: 
- PERSONAL: Only when `user_id` matches JWT user
- ORGANIZATION: SYSTEM_ADMIN or SYSTEM_OWNER only
- Missing access returns 404
</Accordion>

<Accordion title="PUT /v1/email-integrations/:id">
**Description**: Update integration metadata

**Access Control**: Same as GET
</Accordion>

<Accordion title="DELETE /v1/email-integrations/:id">
**Description**: Remove integration

**Access Control**: Same as GET
</Accordion>

<Accordion title="POST /v1/email-integrations/:id/send">
**Description**: Send email through integration

**Access Control**:
- Personal: Only mailbox owner
- Organization: SYSTEM_ADMIN or SYSTEM_OWNER
</Accordion>
</AccordionGroup>

## Data Model

### Schema Structure

The `email_integrations` table supports both personal and organization-level connections:

<CodeGroup>
```sql Personal Level (PERSONAL)
-- user_id NOT NULL, connected_by_id NULL
-- Unique constraint: (user_id, organization_id, provider) 
-- where is_deleted = false AND level = 'PERSONAL'
```

```sql Organization Level (ORGANIZATION)  
-- user_id NULL, connected_by_id NOT NULL (OAuth actor)
-- Unique constraint: (organization_id, provider, lower(email))
-- where is_deleted = false AND level = 'ORGANIZATION'
-- Optional is_default flag (max one per org)
```
</CodeGroup>

### Key Fields

- **`level`**: `PERSONAL` | `ORGANIZATION` (Postgres check constraint)
- **`email`**: Provider mailbox address (normalized to lowercase)
- **`encrypted_access_token`** / **`encrypted_refresh_token`**: AES-256-GCM encrypted tokens
- **`user_id`**: Owner for PERSONAL, NULL for ORGANIZATION
- **`connected_by_id`**: OAuth actor for ORGANIZATION, NULL for PERSONAL
- **`is_default`**: Optional flag for ORGANIZATION level (one per org)

### Cross-Level Conflict Prevention

<Warning>
A mailbox address cannot exist at both PERSONAL and ORGANIZATION levels within the same organization.
</Warning>

This is enforced by:
- Database trigger `email_integrations_cross_level_mailbox_conflict_trigger`
- Transaction advisory locks during concurrent inserts
- Application logic in `connectInTransaction`

## Error Handling

### OAuth Callback Errors

<AccordionGroup>
<Accordion title="Gmail Access Blocked">
**Error Code**: `GMAIL_ACCESS_BLOCKED`

**Cause**: Google account authenticates but doesn't expose Gmail API access

**Response**: AlertDialog shown to user with explanation
</Accordion>

<Accordion title="Tenant Access Denied">
**Message**: `OAUTH_CALLBACK_TENANT_ACCESS_DENIED_MESSAGE`

**Causes**:
- User no longer active
- Token version mismatch (logout/password reset)
- Organization access revoked
- Missing required permissions for ORGANIZATION level
</Accordion>

<Accordion title="Redundant Reconnect">
**Cause**: Attempting to re-OAuth a healthy existing integration

**Prevention**: Blocks reconnection when existing token is valid and not near expiry
</Accordion>
</AccordionGroup>

### PostMessage Security

<Warning>
Callback responses use `postMessage` with `app.postMessageTargetOrigin` from `FRONTEND_URL` - never `*` wildcard.
</Warning>

The SPA should:
- Validate `event.source` matches the OAuth popup
- Invalidate relevant query keys on success
- Handle dropped messages by refetching when popup closes
