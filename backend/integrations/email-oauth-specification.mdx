This specification covers Gmail and Outlook **mail** OAuth integration, token storage, refresh mechanisms, and API surfaces. Calendar OAuth is documented separately in the Calendar System Deep Dive.

## OAuth Flow Overview

The email OAuth flow operates within a tenant session context, supporting both personal and organization-level integrations.

### Flow Steps

<Steps>
<Step title="Initial Setup">
User selects an organization and the SPA holds a **tenant JWT** for authentication.
</Step>

<Step title="OAuth Authorization Request">
Choose between two endpoints based on integration level:

- **Personal mailbox:** `POST /v1/email-integrations/oauth/:provider/authorize` 
  - `provider` = `gmail` | `outlook`
  - Creates `EmailIntegrationLevel.PERSONAL` integration
  - Uses standard `/v1` API authentication

- **Workspace mailbox:** `POST /v1/email-integrations/organization/oauth/:provider/authorize`
  - Creates `EmailIntegrationLevel.ORGANIZATION` integration  
  - Requires `SYSTEM_ADMIN` or `SYSTEM_OWNER` permissions (`@CheckAccess`)
  - Uses same callback URLs as personal OAuth
</Step>

<Step title="Authorization URL Generation">
Backend returns `{ authorizationUrl }` containing:
- Google/Microsoft authorize URL
- HMAC-signed `state` parameter using `SignedPayloadService`
- State payload includes: `userId`, `organizationId`, `tokenVersion`, `provider`, optional `level`
- Missing `level` defaults to `PERSONAL` for legacy compatibility
</Step>

<Step title="Provider Authorization">
Browser opens `authorizationUrl` in popup window for user consent.
</Step>

<Step title="OAuth Callback">
Provider redirects to public callback endpoints:
- `GET /auth/gmail/callback`
- `GET /auth/outlook/callback`

Backend verifies `state` and exchanges authorization `code` via provider driver.
</Step>
</Steps>

### Gmail Access Verification

<Warning>
Gmail integrations require additional access verification before persistence.
</Warning>

Before saving Gmail integrations, the backend calls `GET .../gmail/v1/users/me/profile` with the new access token. If Google returns **400** or **403**, the integration is **not persisted** and the callback returns an error.

**Why this matters:** Some Google accounts authenticate via OAuth but don't expose Gmail API access. This verification prevents saving broken integrations.

**Required scopes:**
- `gmail.send` - for outbound email
- `gmail.metadata` - for profile access verification
- `userinfo.email` - for profile mailbox address

**Error handling:** Failed verification triggers `oauth-error` with `EMAIL_OAUTH_GMAIL_ACCESS_BLOCKED_CODE` (`GMAIL_ACCESS_BLOCKED`).

### Callback Access Control

The backend performs comprehensive access validation during the callback:

<Check>
**Pre-persistence validation includes:**
- User account is active (`User.isActive`)
- Token version matches signed state (`User.tokenVersion`)
- Active `user_org_roles` record exists
- For `ORGANIZATION` level: actor maintains `system.admin` or `system.owner` permissions
</Check>

This validation uses `AccessService.assertOAuthCallbackOrgAccessInTransaction`. Failures result in `oauth-error` with `OAUTH_CALLBACK_TENANT_ACCESS_DENIED_MESSAGE`.

### Integration Persistence Rules

#### Organization vs Personal Levels

<Tabs>
<Tab title="Organization Level">
- `user_id = null`
- `connected_by_id` = OAuth actor ID  
- `email` normalized to lowercase
- Uniqueness: `(organization_id, provider, lower(email))` among active org rows
- Multiple users can connect same address as **PERSONAL**
</Tab>

<Tab title="Personal Level">
- `user_id` = integration owner
- `connected_by_id = null` 
- Uniqueness: per user + org + provider
- Same address can exist across different users
</Tab>
</Tabs>

#### Cross-Level Conflict Prevention

<Warning>
A mailbox address cannot exist at both PERSONAL and ORGANIZATION levels within the same organization.
</Warning>

The system enforces this through:
- Database-level advisory locks during insertion
- Application-level validation in `connectInTransaction`
- Migration `Migration20260521171000_PreventEmailIntegrationCrossLevelMailboxConflicts`

#### Redundant Connection Handling

If OAuth attempts to reconnect a healthy existing integration (same email, active status, valid tokens), the system throws an error and returns `oauth-error` instead of `oauth-success`.

**Healthy integration criteria:**
- Active status (not deleted)
- No `last_sync_error_at` timestamp
- Valid access token present
- Token expiry beyond configured buffer (`EMAIL_RECONNECT_ACCESS_TOKEN_BUFFER_MINUTES`)

## Environment Configuration

### Redirect URIs

<AccordionGroup>
<Accordion title="Gmail Configuration">
**Environment variable:** `GMAIL_REDIRECT_URI`  
**Default:** `${BACKEND_URL or http://localhost:$PORT}/auth/gmail/callback`  
**Override:** `GMAIL_EMAIL_REDIRECT_URI` (optional)

Register exact URI in Google Cloud OAuth client configuration.
</Accordion>

<Accordion title="Outlook Configuration">
**Environment variable:** `OUTLOOK_REDIRECT_URI`  
**Default:** `${BACKEND_URL or http://localhost:$PORT}/auth/outlook/callback`  
**Override:** `OUTLOOK_EMAIL_REDIRECT_URI` (optional)

Register exact URI in Azure app configuration.
</Accordion>
</AccordionGroup>

### Default Values

- `PORT` defaults to `3002`
- Set `BACKEND_URL` when API origin differs from `localhost:$PORT`
- OAuth state TTL: `EMAIL_OAUTH_STATE_TTL_MS` (15 minutes)

## API Reference

### Personal Email Integrations

<CodeGroup>
```http GET /v1/email-integrations
Authorization: Bearer <tenant-jwt>

# Query parameters
?limit=20          # default 20, max 50
?cursor=<opaque>   # for pagination
```

```json Response
{
  "items": [
    {
      "id": "uuid",
      "level": "PERSONAL",
      "provider": "gmail",
      "email": "user@example.com",
      "isHealthy": true,
      "connectedAt": "2024-01-01T00:00:00Z"
    }
  ],
  "nextCursor": "opaque-string-or-null"
}
```
</CodeGroup>

**Access control:** Only returns current user's personal integrations. Admins cannot access other users' personal mailboxes.

**Ordering:** `connected_at DESC`, `id ASC` (keyset pagination)

### Send Accounts (Composer)

```http
GET /v1/email-integrations/send-accounts
Authorization: Bearer <tenant-jwt>

# Pagination parameters
?cursorPersonal=<cursor>
?cursorOrganization=<cursor>
?limit=20
```

<Info>
Returns mailboxes available as "From" addresses in the email composer. Includes user's personal integrations plus organization integrations when user has admin/owner permissions.
</Info>

**Stream exhaustion:** When one stream completes before the other, pass `__exhausted__` as the cursor value to prevent stream restart.

### Organization Email Integrations

<CodeGroup>
```http GET /v1/email-integrations/organization
Authorization: Bearer <tenant-jwt>
# Requires SYSTEM_ADMIN or SYSTEM_OWNER
```

```http POST /v1/email-integrations/organization/oauth/:provider/authorize
Authorization: Bearer <tenant-jwt>
# Requires SYSTEM_ADMIN or SYSTEM_OWNER

{
  "provider": "gmail" // or "outlook"
}
```
</CodeGroup>

**Access control:** Organization endpoints require `SYSTEM_ADMIN` or `SYSTEM_OWNER` permissions via `@CheckAccess`.

### Individual Integration Operations

All individual integration operations (`GET`, `PUT`, `DELETE`, `POST /:id/send`) use `EmailIntegrationService.getAccessibleIntegrationFindWhere` for access control:

<Tabs>
<Tab title="Personal Integrations">
- Only accessible to the integration owner
- `user_id` must match JWT user
- Admins cannot access other users' personal integrations
</Tab>

<Tab title="Organization Integrations">
- Requires `SYSTEM_ADMIN` or `SYSTEM_OWNER` 
- Any qualifying admin can access any org integration
- Used for workspace email management
</Tab>
</Tabs>

## Data Model

### Database Schema

<CodeGroup>
```sql Core Fields
level               VARCHAR NOT NULL CHECK (level IN ('PERSONAL', 'ORGANIZATION'))
user_id            UUID             -- NOT NULL for PERSONAL, NULL for ORGANIZATION  
connected_by_id    UUID             -- NULL for PERSONAL, NOT NULL for ORGANIZATION
organization_id    UUID NOT NULL
provider           VARCHAR NOT NULL -- 'gmail' | 'outlook'
email              VARCHAR NOT NULL -- normalized to lowercase
is_default         BOOLEAN          -- only for ORGANIZATION level
is_deleted         BOOLEAN DEFAULT FALSE
```

```sql Token Storage  
encrypted_access_token   TEXT     -- AES-256-GCM encrypted
encrypted_refresh_token  TEXT     -- AES-256-GCM encrypted  
token_expires_at        TIMESTAMP
last_sync_error_at      TIMESTAMP
```
</CodeGroup>

### Uniqueness Constraints

<AccordionGroup>
<Accordion title="Personal Level Constraints">
**Partial unique:** `(user_id, organization_id, provider)` where `is_deleted = false` AND `level = 'PERSONAL'`

Allows multiple users to connect the same email address as personal integrations.
</Accordion>

<Accordion title="Organization Level Constraints">
**Partial unique:** `(organization_id, provider, lower(email))` where `is_deleted = false` AND `level = 'ORGANIZATION'`

**Default mailbox:** At most one `is_default = true` per organization (partial unique on `organization_id` where org + default).
</Accordion>

<Accordion title="Cross-Level Prevention">
**Rule:** Active rows cannot mix PERSONAL and ORGANIZATION for the same `organization_id + lower(email)`.

**Implementation:** Database trigger `email_integrations_cross_level_mailbox_conflict_trigger` with transaction advisory locks.
</Accordion>
</AccordionGroup>

### Encryption

<Note>
All tokens are encrypted using AES-256-GCM via shared `EncryptionService`. Requires `ENCRYPTION_KEY` environment variable (64 hex characters).
</Note>

## Error Handling

### OAuth Callback Errors

<CardGroup cols={2}>
<Card title="Gmail Access Blocked" icon="envelope-circle-check">
**Code:** `GMAIL_ACCESS_BLOCKED`  
**Cause:** Google account authenticates but lacks Gmail API access  
**Action:** Show alert dialog explaining Gmail access requirements
</Card>

<Card title="Tenant Access Denied" icon="shield-exclamation">
**Code:** `OAUTH_CALLBACK_TENANT_ACCESS_DENIED_MESSAGE`  
**Cause:** User permissions changed during OAuth flow  
**Action:** Restart OAuth process or check permissions
</Card>
</CardGroup>

### Token Exchange Failures

Provider HTTP failures are logged server-side with status codes and truncated response bodies. User-facing errors use generic messages to avoid exposing sensitive information.

**Callback TTL:** OAuth state expires after `EMAIL_OAUTH_STATE_TTL_MS` (15 minutes) to prevent stale callback attempts.
