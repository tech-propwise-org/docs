This document covers the complete specification for Gmail and Outlook email OAuth integration, including token management, API surfaces, and data models. Calendar OAuth is documented separately in the Calendar System Deep Dive.

<Note>
This specification covers **email** OAuth only. Calendar OAuth flows are documented in the Calendar System Deep Dive.
</Note>

## OAuth Flow Overview

The email OAuth flow operates within a tenant session where users can connect personal or organization-level mailboxes.

### Prerequisites

- User has selected an organization
- SPA holds a valid tenant JWT
- Proper provider credentials configured (Gmail/Outlook)

## OAuth Implementation

<Steps>
<Step title="Initiate OAuth Flow">
Choose between personal or organization-level mailbox connection:

**Personal Mailbox:**
```http
POST /v1/email-integrations/oauth/:provider/authorize
```
- `provider`: `gmail` | `outlook`
- Level: `EmailIntegrationLevel.PERSONAL`
- Access: Standard `/v1` API authentication

**Organization Mailbox:**
```http
POST /v1/email-integrations/organization/oauth/:provider/authorize
```
- Level: `EmailIntegrationLevel.ORGANIZATION`
- Access: Requires `SYSTEM_ADMIN` or `SYSTEM_OWNER` via `@CheckAccess`
</Step>

<Step title="Backend Authorization Response">
Backend returns authorization URL with signed state:

```json
{
  "authorizationUrl": "https://accounts.google.com/oauth/authorize?..."
}
```

The `state` parameter includes:
- `userId` and `organizationId`
- `tokenVersion` for security validation
- `provider` (gmail/outlook)
- `level` (PERSONAL/ORGANIZATION, omitted = PERSONAL for legacy)
</Step>

<Step title="Provider Authentication">
Browser opens the authorization URL in a popup window where the user completes OAuth with the provider.
</Step>

<Step title="OAuth Callback">
Provider redirects to public callback endpoints:
- `GET /auth/gmail/callback`
- `GET /auth/outlook/callback`

The backend:
1. Verifies the signed `state` parameter
2. Exchanges authorization `code` for tokens
3. Performs provider-specific validation
</Step>

<Step title="Gmail Access Verification">
<Warning>
Gmail integrations require additional access verification before persistence.
</Warning>

Before saving Gmail integrations, the backend calls:
```http
GET https://gmail.googleapis.com/gmail/v1/users/me/profile
```

If Google returns 400 or 403:
- Integration is **not** persisted
- Callback returns `oauth-error` with `GMAIL_ACCESS_BLOCKED`
- SPA displays appropriate error dialog

This prevents saving integrations for accounts that authenticate but lack Gmail API access.
</Step>

<Step title="Access Re-verification">
Before token persistence, backend re-checks OAuth actor permissions:
- `User.isActive` status
- Matching `User.tokenVersion` from signed state
- Active `user_org_roles` for `(userId, organizationId)`
- For ORGANIZATION level: current `system.admin` or `system.owner` role

Implemented via `AccessService.assertOAuthCallbackOrgAccessInTransaction`.
</Step>

<Step title="Token Storage">
If all validations pass, `connectInTransaction` executes:
- Encrypts access and refresh tokens
- Upserts email integration row
- Handles uniqueness constraints and conflict resolution
</Step>

<Step title="Popup Response">
Response HTML uses `postMessage` with configured target origin:
- Target: `app.postMessageTargetOrigin` (from `FRONTEND_URL`)
- Never uses wildcard `*`
- Payload may include `level` for SPA handling
</Step>

<Step title="SPA Integration">
SPA handles the OAuth completion:
- Validates `event.source` matches OAuth popup
- Invalidates `queryKeys.emailIntegrations` cache
- Refreshes composer and integration lists
- Shows error dialog for `GMAIL_ACCESS_BLOCKED`
- Refetches data when popup closes
</Step>
</Steps>

## Data Model and Constraints

### Integration Levels

<Tabs>
<Tab title="Personal Integration">
**Level:** `PERSONAL`
- `user_id`: NOT NULL (mailbox owner)
- `connected_by_id`: NULL
- **Uniqueness:** `(user_id, organization_id, provider)` where `is_deleted = false`
- Multiple users can connect the same mailbox address personally
</Tab>

<Tab title="Organization Integration">
**Level:** `ORGANIZATION`
- `user_id`: NULL
- `connected_by_id`: NOT NULL (OAuth actor)
- `email`: Normalized to lowercase
- **Uniqueness:** `(organization_id, provider, lower(email))` where `is_deleted = false`
- Optional `is_default` flag (max one per organization)
</Tab>
</Tabs>

### Cross-Level Uniqueness Rules

<Warning>
A mailbox address cannot exist at both PERSONAL and ORGANIZATION levels within the same organization.
</Warning>

The system enforces cross-level uniqueness through:
- Database trigger: `email_integrations_cross_level_mailbox_conflict_trigger`
- Transaction advisory locks per organization/mailbox
- Rejection during `connectInTransaction` for conflicting levels

### Redundant Reconnect Prevention

OAuth is rejected when connecting to an existing, healthy integration:
- Same provider and email address
- Integration is active (no `last_sync_error_at`)
- Valid access token beyond expiry buffer
- Buffer configured via `EMAIL_RECONNECT_ACCESS_TOKEN_BUFFER_MINUTES`

Unhealthy integrations allow re-OAuth for repair.

## API Endpoints

### Integration Management

<AccordionGroup>
<Accordion title="GET /v1/email-integrations">
**Purpose:** List personal mailboxes for current user

**Access:** Current user's PERSONAL integrations only

**Response:**
```json
{
  "items": [EmailIntegrationDto],
  "nextCursor": "opaque-pagination-string"
}
```

**Pagination:**
- Default limit: 20, max: 50
- Ordering: `connected_at DESC, id ASC` (keyset)
- Query parameter: `?cursor=nextCursor`
</Accordion>

<Accordion title="GET /v1/email-integrations/send-accounts">
**Purpose:** Get mailboxes available as From addresses in composer

**Access:** Current user's PERSONAL + ORGANIZATION (if admin/owner)

**Response:**
```json
{
  "personal": {
    "items": [EmailIntegrationDto],
    "nextCursor": "string"
  },
  "organization": {
    "items": [EmailIntegrationDto],
    "nextCursor": "string"
  }
}
```

**Pagination:**
- Separate cursors: `cursorPersonal`, `cursorOrganization`
- Exhausted streams use `__exhausted__` literal
- Organization ordering: `is_default DESC, connected_at DESC, id ASC`
</Accordion>

<Accordion title="GET /v1/email-integrations/organization">
**Purpose:** List organization-level mailboxes

**Access:** `SYSTEM_ADMIN` or `SYSTEM_OWNER` via `@CheckAccess`

**Features:**
- Populates `connectedBy` relation
- Offset pagination
- Same limit bounds as other list endpoints
</Accordion>
</AccordionGroup>

### Individual Integration Operations

<CodeGroup>
```http GET /v1/email-integrations/:id
# Get specific integration
# Access controlled via getAccessibleIntegrationFindWhere
```

```http PUT /v1/email-integrations/:id
# Update integration metadata
# Same access rules as GET
```

```http DELETE /v1/email-integrations/:id
# Soft delete integration
# Same access rules as GET
```

```http POST /v1/email-integrations/:id/send
# Send email via integration
# Personal: owner only
# Organization: admin/owner only
```
</CodeGroup>

## Environment Configuration

### Redirect URI Configuration

<Tabs>
<Tab title="Gmail">
**Primary:** `GMAIL_REDIRECT_URI`
**Override:** `GMAIL_EMAIL_REDIRECT_URI`
**Default:** `${BACKEND_URL}/auth/gmail/callback`

If `BACKEND_URL` is not set, defaults to `http://localhost:${PORT}` where `PORT` defaults to `3002`.
</Tab>

<Tab title="Outlook">
**Primary:** `OUTLOOK_REDIRECT_URI`
**Override:** `OUTLOOK_EMAIL_REDIRECT_URI`
**Default:** `${BACKEND_URL}/auth/outlook/callback`

Same fallback logic as Gmail configuration.
</Tab>
</Tabs>

<Info>
These exact redirect URIs must be registered in your Google Cloud OAuth client and Azure app registration.
</Info>

## Security and Token Management

### Token Encryption

- **Algorithm:** AES-256-GCM
- **Service:** Shared `EncryptionService`
- **Key:** `ENCRYPTION_KEY` (64 hex characters)
- **Storage:** `encrypted_access_token`, `encrypted_refresh_token`

### OAuth State Security

- **Signing:** HMAC via `SignedPayloadService`
- **Purpose:** `email.oauth.state`
- **TTL:** 15 minutes (`EMAIL_OAUTH_STATE_TTL_MS`)
- **Token Version:** Invalidates open popups on password reset/logout

### Access Control Patterns

<CardGroup cols={2}>
<Card title="Personal Integrations">
- Owner-only access via `user_id` match
- Admins cannot access other users' personal mailboxes
- 404 response for missing access
</Card>

<Card title="Organization Integrations">
- Admin/owner access via `@CheckAccess`
- Separate connected_by tracking
- Cross-level conflict prevention
</Card>
</CardGroup>

## Error Handling

### OAuth Callback Errors

<Warning>
Provider errors are logged server-side but sanitized for user-facing messages.
</Warning>

Common error scenarios:
- **Token exchange failures:** Logged with HTTP status and truncated body
- **Gmail access blocked:** Specific error code for blocked API access
- **Tenant access denied:** User lost permissions during OAuth flow
- **Cross-level conflicts:** Mailbox already connected at different level

### Error Constants

```typescript
// Must stay synchronized between backend and frontend
EMAIL_OAUTH_GMAIL_ACCESS_BLOCKED_CODE = 'GMAIL_ACCESS_BLOCKED'
EMAIL_INTEGRATION_MAILBOX_STREAM_EXHAUSTED_QUERY = '__exhausted__'
OAUTH_CALLBACK_TENANT_ACCESS_DENIED_MESSAGE = 'Access denied'
```

## Required OAuth Scopes

### Gmail Integration

- `gmail.send` - Required for outbound email sending
- `gmail.metadata` - Required for access verification (`users.getProfile`)
- `userinfo.email` - Required to read profile mailbox address

### Outlook Integration

- Standard email send and read permissions
- No additional access verification step required

<Tip>
Ensure all required scopes are configured in your OAuth provider settings to prevent authentication issues.
</Tip>
