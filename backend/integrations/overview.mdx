The integrations module provides connectivity to external services and platforms within PropWise CRM. It enables seamless connections to email providers (Gmail and Outlook) via OAuth2 authentication and integrates with Tiptap AI for enhanced rich text editing capabilities.

## Architecture overview

The integrations module is structured into three specialized sub-modules that work together to provide comprehensive external service connectivity:

<CardGroup cols={3}>
  <Card title="Email Integration" icon="envelope">
    Manages connections to Gmail and Outlook for sending emails
  </Card>
  <Card title="OAuth Authentication" icon="shield-check">
    Handles secure OAuth2 flows for external service authorization
  </Card>
  <Card title="Tiptap AI" icon="brain">
    Provides JWT token generation for AI-powered text editing
  </Card>
</CardGroup>

### Key design principles

<AccordionGroup>
  <Accordion title="Provider-agnostic architecture">
    Uses enum-based provider differentiation to support multiple email services without code duplication
  </Accordion>
  
  <Accordion title="Automated token management">
    Implements scheduled token refresh to maintain continuous connectivity without manual intervention
  </Accordion>
  
  <Accordion title="Organization-scoped integrations">
    All integrations are tenant-aware and scoped to specific organizations
  </Accordion>
  
  <Accordion title="Security-first approach">
    Email access tokens are encrypted at rest and OAuth flows include CSRF protection
  </Accordion>
</AccordionGroup>

## Core entities

### EmailIntegration entity

The `EmailIntegration` entity represents a connection between a user and an external email provider within an organization context.

```typescript
interface EmailIntegration {
  id: string;                    // UUID primary key
  email: string;                 // Email address being integrated
  provider: EmailProvider;       // OUTLOOK | GMAIL
  accessToken: string;          // Encrypted OAuth access token
  refreshToken?: string;        // Encrypted OAuth refresh token
  tokenExpiresAt?: Date;        // Token expiration timestamp
  metadata?: Record<string, any>; // Provider-specific data
  isActive: boolean;            // Integration status
  lastUsedAt?: Date;           // Last activity timestamp
  user: User;                   // Owner of the integration
  organization: Organization;   // Organization scope
  isDeleted: boolean;          // Soft delete flag
  createdAt: Date;
  updatedAt: Date;
}
```

<Note>
The system enforces a unique constraint on `(user, organization, email)` where `isDeleted = false`, ensuring one integration per email per organization.
</Note>

### EmailProvider enum

```typescript
enum EmailProvider {
  OUTLOOK = 'outlook',
  GMAIL = 'gmail'
}
```

## API reference

### Email integration endpoints

#### List integrations

<CodeGroup>
```typescript Request
GET /email-integrations
```

```typescript Response
{
  data: EmailIntegrationDto[];
  meta: PaginationMeta;
}
```
</CodeGroup>

Returns all email integrations for the authenticated user within their organization context.

#### Create integration

<CodeGroup>
```typescript Request
POST /email-integrations
{
  email: string;
  provider: EmailProvider;
  accessToken: string;
  refreshToken?: string;
  tokenExpiresAt?: Date;
  metadata?: Record<string, any>;
}
```

```typescript Response
{
  id: string;
  email: string;
  provider: EmailProvider;
  isActive: boolean;
  createdAt: Date;
}
```
</CodeGroup>

Creates a new email integration after validating the provider and ensuring uniqueness constraints.

#### Send email

<CodeGroup>
```typescript Request
POST /email-integrations/:id/send
{
  to: string[];
  subject: string;
  body: string;
  isHtml?: boolean;
  cc?: string[];
  bcc?: string[];
  attachments?: Array<{
    filename: string;
    content: string; // base64
    contentType: string;
  }>;
}
```

```typescript Response
{
  success: boolean;
  messageId: string;
}
```
</CodeGroup>

Sends an email through the specified integration using the configured email provider.

## Business rules and constraints

### Email integration rules

<Check>**One integration per email**: Users can maintain only one active integration per email address per organization</Check>

<Check>**Automatic token refresh**: Expired tokens are refreshed automatically via scheduled jobs</Check>

<Check>**Provider validation**: Only supported providers (Gmail, Outlook) are accepted</Check>

<Check>**Organization scoping**: All integrations are isolated by organization context</Check>

<Check>**Audit preservation**: Integrations use soft deletion to maintain history</Check>

### OAuth security requirements

<Warning>
OAuth state parameters must contain valid user and organization context to prevent CSRF attacks
</Warning>

<Info>
Only whitelisted redirect URLs are permitted during OAuth flows
</Info>

### Token management policies

<Tabs>
  <Tab title="Refresh schedule">
    - Tokens refresh automatically 5 minutes before expiration
    - Refresh job runs every 30 minutes
    - Failed attempts use exponential backoff retry logic
  </Tab>
  
  <Tab title="Security measures">
    - All tokens encrypted at rest using application-level encryption
    - Tiptap AI tokens have 1-hour maximum lifetime
    - Expired tokens trigger automatic re-authentication
  </Tab>
</Tabs>

## Integration dependencies

### Internal module dependencies

The integrations module relies on several core modules:

- **TenantContextModule**: Organization scoping and isolation
- **ConfigModule**: OAuth credentials and API configuration
- **ScheduleModule**: Cron-based token refresh automation
- **MikroOrmModule**: Database entity and relationship management

## Monitoring and operations

### Automated tasks

The module includes several scheduled operations:

<Steps>
  <Step title="Token refresh (every 30 minutes)">
    Proactively refreshes tokens nearing expiration
  </Step>
  
  <Step title="Integration cleanup (daily)">
    Removes expired or abandoned integrations
  </Step>
  
  <Step title="Health checks (hourly)">
    Validates integration connectivity and updates status
  </Step>
</Steps>

### Logging and metrics

<Tip>
All integration operations are logged with structured data including:
- Token refresh success/failure rates
- Email sending attempt outcomes
- OAuth flow completion metrics
- Integration health status changes
</Tip>

<Warning>
Failed authentication attempts are logged for security analysis and potential threat detection
</Warning>
