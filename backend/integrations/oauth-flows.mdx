The OAuth module handles OAuth2 authentication flows for external email providers within PropWise CRM. It supports both Gmail and Outlook integrations, managing the complete authentication lifecycle from initial authorization to token refresh.

## Overview

OAuth2 authentication enables secure access to external email services without storing user passwords. The system manages authorization codes, access tokens, and refresh tokens while maintaining security through state validation and encrypted token storage.

<Info>
All OAuth integrations are scoped to organizations and users, ensuring proper tenant isolation and access control.
</Info>

## Authentication Flow

The OAuth authentication process follows the standard OAuth2 authorization code flow:

<Steps>
  <Step title="Initialize OAuth Flow">
    User clicks connect button in the frontend, triggering the OAuth initialization endpoint
  </Step>
  <Step title="Redirect to Provider">
    User is redirected to the email provider's authorization server with encoded state
  </Step>
  <Step title="User Authorization">
    User grants permissions to PropWise CRM on the provider's consent screen
  </Step>
  <Step title="Authorization Code Exchange">
    Provider redirects back to PropWise with authorization code
  </Step>
  <Step title="Token Exchange">
    Backend exchanges authorization code for access and refresh tokens
  </Step>
  <Step title="Integration Creation">
    Email integration record is created with encrypted tokens
  </Step>
</Steps>

## API Endpoints

### Gmail OAuth

#### Initiate Gmail Authentication

```http
GET /oauth/gmail/auth?state={encoded_context}
```

**Query Parameters:**
- `state` (string, required): Base64-encoded JSON containing user and organization context

**Response:**
Redirects to Google OAuth authorization URL

<CodeGroup>
```typescript Example State Parameter
const state = {
  userId: "user-uuid",
  organizationId: "org-uuid",
  email: "user@example.com"
};
const encodedState = btoa(JSON.stringify(state));
```

```javascript Frontend Integration
window.location.href = `/api/oauth/gmail/auth?state=${encodedState}`;
```
</CodeGroup>

#### Gmail OAuth Callback

```http
GET /oauth/gmail/callback?code={auth_code}&state={encoded_context}
```

**Query Parameters:**
- `code` (string, required): Authorization code from Google
- `state` (string, required): Original state parameter for validation

**Response:**
Redirects to frontend with success or error parameters

## State Parameter Security

<Warning>
The state parameter is critical for security and prevents CSRF attacks. It must contain valid user and organization context.
</Warning>

### State Structure

```typescript
interface OAuthState {
  userId: string;           // User UUID
  organizationId: string;   // Organization UUID
  email: string;           // Email being integrated
  timestamp?: number;      // Optional timestamp for expiration
}
```

### State Validation

The OAuth callback validates the state parameter to ensure:
- State can be decoded from Base64
- Contains required user and organization IDs
- User and organization exist in the database
- User has permission to create integrations in the organization

## Token Management

### Token Storage

All OAuth tokens are encrypted before storage in the database:

```typescript
interface TokenStorage {
  accessToken: string;      // Encrypted OAuth access token
  refreshToken?: string;    // Encrypted OAuth refresh token (optional)
  tokenExpiresAt?: Date;   // Token expiration timestamp
  metadata?: Record<string, any>; // Provider-specific data
}
```

### Automatic Token Refresh

<Note>
Tokens are automatically refreshed 5 minutes before expiration to ensure continuous connectivity.
</Note>

The system implements automated token refresh with:
- **Scheduled Jobs**: Run every 30 minutes to check expiring tokens
- **Retry Logic**: Exponential backoff for failed refresh attempts
- **Error Handling**: Re-authentication triggers for refresh failures

<Tabs>
  <Tab title="Refresh Schedule">
    ```typescript
    // Token refresh cron job configuration
    @Cron('0 */30 * * * *') // Every 30 minutes
    async refreshExpiringTokens() {
      const expiringTokens = await this.findTokensExpiringIn(5); // 5 minutes
      
      for (const integration of expiringTokens) {
        await this.refreshToken(integration);
      }
    }
    ```
  </Tab>
  
  <Tab title="Retry Logic">
    ```typescript
    // Exponential backoff for failed refreshes
    async refreshTokenWithRetry(integration: EmailIntegration, attempt = 1) {
      try {
        return await this.refreshToken(integration);
      } catch (error) {
        if (attempt < 3) {
          const delay = Math.pow(2, attempt) * 1000; // 2s, 4s, 8s
          await this.delay(delay);
          return this.refreshTokenWithRetry(integration, attempt + 1);
        }
        throw error;
      }
    }
    ```
  </Tab>
</Tabs>

## Error Handling

### OAuth Errors

OAuth errors are handled gracefully with redirects to the frontend:

<AccordionGroup>
  <Accordion title="Authorization Denied">
    **Scenario**: User denies permission on provider consent screen
    
    **Response**: Redirect to frontend with `error=access_denied`
    
    **Frontend Handling**: Display user-friendly message about required permissions
  </Accordion>
  
  <Accordion title="Invalid State Parameter">
    **Scenario**: State parameter is missing, invalid, or tampered with
    
    **Response**: HTTP 400 Bad Request with error details
    
    **Security**: Potential CSRF attack attempt is logged
  </Accordion>
  
  <Accordion title="Token Exchange Failure">
    **Scenario**: Authorization code cannot be exchanged for tokens
    
    **Response**: Redirect to frontend with `error=token_exchange_failed`
    
    **Logging**: Full error details logged for debugging
  </Accordion>
  
  <Accordion title="Integration Creation Failure">
    **Scenario**: Database error when creating email integration
    
    **Response**: Redirect to frontend with `error=integration_failed`
    
    **Recovery**: User can retry the OAuth flow
  </Accordion>
</AccordionGroup>

### Token Refresh Errors

```typescript
// Token refresh error handling
try {
  const newTokens = await this.refreshAccessToken(refreshToken);
  await this.updateIntegration(integration.id, newTokens);
} catch (error) {
  if (error.code === 'invalid_grant') {
    // Refresh token is invalid, trigger re-authentication
    await this.markIntegrationForReauth(integration.id);
    await this.notifyUserReauth(integration.userId);
  } else {
    // Temporary error, will retry on next cycle
    this.logger.warn('Token refresh failed, will retry', { 
      integrationId: integration.id,
      error: error.message 
    });
  }
}
```

## Security Considerations

<Check>
**Security Best Practices Implemented:**
</Check>

- **Encrypted Token Storage**: All OAuth tokens encrypted using application-level encryption
- **State Parameter Validation**: Prevents CSRF attacks through state verification
- **Secure Redirects**: Only whitelisted redirect URLs accepted
- **Organization Scoping**: All integrations isolated by organization
- **Token Expiration**: Short-lived tokens with automatic refresh
- **Audit Logging**: All OAuth events logged for security monitoring

### Rate Limiting

OAuth endpoints implement rate limiting to prevent abuse:

```typescript
@Throttle(5, 60) // 5 requests per minute
@Get('gmail/auth')
async initiateGmailAuth(@Query('state') state: string) {
  // OAuth initialization logic
}
```

## Integration with Email Module

Once OAuth authentication is complete, the email integration can be used for sending emails:

```typescript
// Using OAuth integration for email sending
const integration = await this.emailIntegrationService
  .findActiveIntegration(userId, organizationId, email);

if (integration) {
  await this.emailService.sendViaIntegration(integration, emailData);
} else {
  throw new Error('No active email integration found');
}
```

<Tip>
Email integrations created through OAuth are automatically marked as active and ready for use in email sending operations.
</Tip>
