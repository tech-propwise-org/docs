## Overview

The integrations module provides connectivity to external services and platforms within PropWise CRM. It currently supports email provider integrations (Gmail and Outlook) via OAuth2 authentication, and Tiptap AI for rich text editing capabilities. The module handles OAuth flows, token management, email sending, and secure access to third-party services.

<Note>
Last updated: 2024-01-15 | Status: Active
</Note>

### Key Design Decisions

<AccordionGroup>
  <Accordion title="Provider-agnostic email integration">
    Uses enum-based provider differentiation to support multiple email services
  </Accordion>
  <Accordion title="Token refresh automation">
    Implements scheduled token refresh to maintain continuous integration connectivity
  </Accordion>
  <Accordion title="Tenant-aware integrations">
    All integrations are scoped to organizations using tenant context
  </Accordion>
  <Accordion title="Secure token storage">
    OAuth access/refresh tokens are AES-GCM encrypted at rest (encryptedAccessToken / encryptedRefreshToken)
  </Accordion>
  <Accordion title="Personal vs workspace mailboxes">
    EmailIntegrationLevel distinguishes PERSONAL (owned by user_id) and ORGANIZATION (user_id null, connected_by_id stores OAuth actor)
  </Accordion>
</AccordionGroup>

<Warning>
**Mail** OAuth lives in **email-integration** (EmailOAuthFlowService, EmailOAuthCallbackController), not in the oauth sub-module.
</Warning>

## Entities

### EmailIntegration

Represents a connection to Gmail or Outlook for sending mail. Either a **personal** mailbox (`level=PERSONAL`, `user` required) or a **workspace** mailbox (`level=ORGANIZATION`, `user` null, `connectedBy` set).

```typescript
interface EmailIntegration {
  id: string;
  email: string;
  provider: EmailProvider;
  level: 'PERSONAL' | 'ORGANIZATION';
  isDefault?: boolean; // ORGANIZATION: optional default sender for admins
  encryptedAccessToken?: string;
  encryptedRefreshToken?: string;
  scopes?: string;
  externalAccountId?: string;
  lastSyncErrorAt?: Date;
  lastSyncErrorMessage?: string;
  tokenExpiresAt?: Date;
  metadata?: Record<string, any>;
  isActive: boolean;
  lastUsedAt?: Date;
  user?: User | null; // PERSONAL owner; null for ORGANIZATION
  connectedBy?: User | null; // OAuth actor for ORGANIZATION
  organization: Organization;
  isDeleted: boolean;
  createdAt: Date;
  updatedAt: Date;
}
```

#### Unique Constraints

<Tabs>
  <Tab title="PERSONAL">
    - `(user_id, organization_id, provider)` - One integration per user per provider
  </Tab>
  <Tab title="ORGANIZATION">
    - `(organization_id, provider, lower(email))` - One org mailbox per provider per email
    - At most one `is_default` org mailbox per organization
  </Tab>
</Tabs>

<Warning>
**Cross-level mailbox rule**: The same `organization_id + lower(email)` cannot appear in both PERSONAL and ORGANIZATION rows, regardless of provider. The same mailbox may still be connected by multiple personal users.
</Warning>

### EmailProvider Enum

```typescript
enum EmailProvider {
  OUTLOOK = 'outlook',
  GMAIL = 'gmail',
}
```

## API Endpoints

### Email Integration Management

#### GET /email-integrations

Lists **PERSONAL** email integrations for the current user.

#### GET /email-integrations/send-accounts

Returns mailboxes usable as **From** in the composer:
- Caller's personal integrations
- **ORGANIZATION** rows when the caller has `SYSTEM_ADMIN` or `SYSTEM_OWNER` role

#### GET /email-integrations/organization

<Check>
Lists **ORGANIZATION** mailboxes (requires `SYSTEM_ADMIN` / `SYSTEM_OWNER`)
</Check>

#### POST /v1/email-integrations/organization/oauth/:provider/authorize

Starts OAuth for a workspace mailbox (same access control as GET organization endpoint).

#### POST /v1/email-integrations/oauth/:provider/authorize

Starts OAuth for **personal** mail (`EmailIntegrationLevel.PERSONAL`).

<Warning>
Creating integrations by POSTing plaintext OAuth tokens is **not supported**. Use the authorize endpoint and provider callback only.
</Warning>

#### PUT /email-integrations/:id

Updates an existing email integration.

#### DELETE /email-integrations/:id

Soft deletes an email integration.

#### POST /email-integrations/:id/send

Sends an email through the specified integration.

**Request:**
```typescript
{
  to: string[];
  subject: string;
  body: string;
  isHtml?: boolean;
  cc?: string[];
  bcc?: string[];
  attachments?: Array<{
    filename: string;
    content: string; // base64
    contentType: string;
  }>;
}
```

### OAuth Authentication

<Steps>
  <Step title="Initiate OAuth Flow">
    Use tenant `POST /v1/email-integrations/oauth/:provider/authorize` with signed state bound to `userId` / `organizationId` / `tokenVersion`
  </Step>
  <Step title="Handle Callback">
    Public `GET /auth/gmail/callback` / `GET /auth/outlook/callback` processes the OAuth response
  </Step>
  <Step title="Store Encrypted Tokens">
    Tokens are encrypted and stored securely in the database
  </Step>
</Steps>

<Tip>
See `Docs/EMAIL_INTEGRATION_SPECIFICATION.md` for the canonical mail OAuth flow documentation.
</Tip>

### Tiptap AI Integration

#### GET /tiptap-ai/token

Generates a short-lived JWT token for Tiptap AI services.

**Response:**
```typescript
{
  token: string; // JWT valid for 1 hour
}
```

## Business Rules

### Email Integrations

<Info>
Key business rules that govern email integration behavior:
</Info>

1. **One Integration Per Provider**: At most one active row per `(user, organization, provider)` — reconnect updates tokens on the same row
2. **Token Expiration**: Expired tokens are automatically refreshed via scheduled jobs
3. **Provider Validation**: Only supported email providers (Gmail, Outlook) are allowed
4. **Organization Scoping**: All integrations are scoped to the user's current organization
5. **Soft Deletion**: Integrations are soft-deleted to maintain audit history
6. **No Personal/Workspace Split**: A normalized mailbox address may be connected by multiple **PERSONAL** users, but cannot be connected at **PERSONAL** and **ORGANIZATION** levels in the same organization simultaneously

### OAuth Flows

1. **State Validation**: OAuth state parameter must contain valid user, organization, provider, and token-version context
2. **Secure Redirects**: Only whitelisted redirect URLs are allowed
3. **Token Exchange**: Authorization codes must be exchanged for tokens immediately
4. **Error Handling**: OAuth errors redirect to frontend with appropriate error messages

### Token Management

1. **Automatic Refresh**: Tokens are refreshed 5 minutes before expiration
2. **Retry Logic**: Failed refresh attempts are retried with exponential backoff
3. **Encryption**: All stored tokens are encrypted at rest
4. **Expiration Handling**: Expired tokens trigger re-authentication flows

### Calendar Conferencing (Google Meet / Teams)

The calendar drivers can request a conferencing link when a CRM Event meeting channel carries `conferenceType`.

<CodeGroup>
```typescript Google Driver
// Google driver (google_meet)
createEvent: {
  conferenceDataVersion: 1,
  // ... other params
}

toGoogleEvent: {
  conferenceData: {
    createRequest: {
      requestId: conferenceRequestId,
      conferenceSolutionKey: {
        type: 'hangoutsMeet'
      }
    }
  }
}
```

```typescript Outlook Driver
// Outlook driver (teams)
toGraphEvent: {
  isOnlineMeeting: true,
  onlineMeetingProvider: 'teamsForBusiness'
}
```
</CodeGroup>

<Note>
- **Google**: Existing `calendar.events` scope is sufficient — **no re-consent** required
- **Outlook**: Existing `Calendars.ReadWrite` scope is sufficient
- **Consumer `outlook.com` accounts cannot create Teams meetings** — errors are caught and surfaced as sync errors
</Note>

## Patterns and Conventions

### Error Handling

- OAuth errors redirect to frontend with error parameters
- API errors return standard HTTP status codes
- Token refresh failures trigger re-authentication
- Rate limiting applied to token refresh operations

### Security

- All OAuth tokens stored encrypted using application-level encryption
- JWT tokens for Tiptap AI have short expiration (1 hour)
- OAuth state parameters prevent CSRF attacks
- Integration endpoints require authentication

### Monitoring

- Token refresh operations are logged with structured data
- Email sending attempts tracked with success/failure metrics
- Integration health monitored via last usage timestamps
- Failed authentication attempts logged for security analysis
