## Overview

This specification covers Gmail and Outlook **email** OAuth integration, token storage, refresh, and API surfaces. Calendar OAuth is documented separately in the Calendar System Deep Dive.

<Note>
This specification focuses specifically on **email** functionality. Calendar OAuth flows are handled by a separate system.
</Note>

## OAuth Flow (Tenant Session)

The OAuth flow enables users to connect personal mailboxes or workspace administrators to connect organization-wide mailboxes.

<Steps>
<Step title="User Authentication">
User selects an organization and the SPA holds a **tenant JWT**.
</Step>

<Step title="OAuth Authorization Request">
Choose the appropriate endpoint based on integration level:

<Tabs>
<Tab title="Personal Mailbox">
```http
POST /v1/email-integrations/oauth/{provider}/authorize
```
- `provider`: `gmail` | `outlook`
- Level: `EmailIntegrationLevel.PERSONAL`
- Uses standard `/v1` API authentication
</Tab>

<Tab title="Organization Mailbox">
```http
POST /v1/email-integrations/organization/oauth/{provider}/authorize
```
- Requires `SYSTEM_ADMIN` or `SYSTEM_OWNER` permissions (`@CheckAccess`)
- Level: `EmailIntegrationLevel.ORGANIZATION`
- Same callback URLs as personal OAuth
</Tab>
</Tabs>
</Step>

<Step title="Authorization URL Generation">
Backend returns `{ authorizationUrl }` containing:
- Google/Microsoft authorize URL
- HMAC-signed `state` parameter using `SignedPayloadService`
- State payload includes: `userId`, `organizationId`, `tokenVersion`, `provider`, optional `level`
- Purpose: `email.oauth.state`
- Omitted `level` defaults to `PERSONAL` for legacy compatibility
</Step>

<Step title="Provider Authorization">
Browser opens `authorizationUrl` in popup for user consent.
</Step>

<Step title="Provider Callback">
Provider redirects to public callback endpoints (no `/v1` prefix):
- `GET /auth/gmail/callback`
- `GET /auth/outlook/callback`
</Step>

<Step title="State Verification and Token Exchange">
Backend verifies `state` parameter and exchanges authorization `code` via provider driver.
</Step>

<Step title="Gmail Access Verification">
<Warning>
Gmail-specific verification step to prevent broken integrations.
</Warning>

Before saving Gmail integrations:
1. Call `GET .../gmail/v1/users/me/profile` with new access token
2. If Google returns **400** or **403**, integration is **not persisted**
3. Callback HTML posts `oauth-error` with `error: EMAIL_OAUTH_GMAIL_ACCESS_BLOCKED_CODE`
4. Prevents saving integrations for accounts that authenticate but don't expose Gmail API access

Outlook integrations skip this verification step.
</Step>

<Step title="Callback Access Re-verification">
Before token exchange and during database transaction, backend re-checks OAuth actor against current state:
- `User.isActive` status
- Matching `User.tokenVersion` from signed state
- Active `user_org_roles` row for `(userId, organizationId)`
- For `ORGANIZATION` level: actor still holds `system.admin` or `system.owner`

Uses `AccessService.assertOAuthCallbackOrgAccessInTransaction`. Failure results in `oauth-error` with `OAUTH_CALLBACK_TENANT_ACCESS_DENIED_MESSAGE`.
</Step>

<Step title="Integration Persistence">
On successful verification, `connectInTransaction` runs to encrypt tokens and upsert database row.

<Tabs>
<Tab title="Organization Level">
- `user_id = null`
- `connected_by_id` = OAuth actor
- Mailbox `email` normalized to lowercase
- Uniqueness: `(organization_id, provider, lower(email))` among active org rows
</Tab>

<Tab title="Personal Level">
- Multiple users may connect the same mailbox address as personal
- DB uniqueness per user + org + provider
</Tab>
</Tabs>

**Cross-level Conflict Prevention:**
- Mailbox address cannot exist at both levels in same organization
- `connectInTransaction` rejects conflicts between personal and organization integrations
</Step>

<Step title="Response Handling">
Response HTML uses `postMessage` with `app.postMessageTargetOrigin` (from `FRONTEND_URL`):
- Never uses wildcard `*` origin
- Payload may include `level` for SPA
- SPA invalidates `queryKeys.emailIntegrations` after success
- Validates `event.source` is the OAuth popup
</Step>
</Steps>

### Gmail Access Verification Details

<Info>
Gmail OAuth requires specific scopes for proper functionality verification.
</Info>

**Required Configuration:**
- Constant: `EMAIL_OAUTH_GMAIL_ACCESS_BLOCKED_CODE` in `src/common/constants/email-oauth.constants.ts`
- Must sync with frontend `EMAIL_OAUTH_GMAIL_ACCESS_BLOCKED_CODE` in `email-integration.api.ts`

**Required Scopes:**
- `gmail.send` - for outbound email sending
- `gmail.metadata` - for `users.getProfile` access verification
- `userinfo.email` - for profile mailbox address reading

### Environment Configuration

<CodeGroup>
```bash Gmail Configuration
# Default: ${BACKEND_URL or http://localhost:$PORT}/auth/gmail/callback
GMAIL_REDIRECT_URI=your_gmail_redirect_uri

# Optional override
GMAIL_EMAIL_REDIRECT_URI=your_gmail_email_redirect_uri
```

```bash Outlook Configuration  
# Default: ${BACKEND_URL or http://localhost:$PORT}/auth/outlook/callback
OUTLOOK_REDIRECT_URI=your_outlook_redirect_uri

# Optional override
OUTLOOK_EMAIL_REDIRECT_URI=your_outlook_email_redirect_uri
```
</CodeGroup>

<Warning>
Register these exact redirect URIs in your Google Cloud OAuth client and Azure app configuration.
</Warning>

### Error Handling

**Callback Errors:**
- Token exchange failures logged server-side with provider HTTP status
- Popup `postMessage` uses generic error strings for user display
- Raw provider error bodies are not exposed to frontend

**OAuth State TTL:**
- `EMAIL_OAUTH_STATE_TTL_MS`: 15 minutes (aligned with calendar UX)

## API Endpoints

### Email Integration Lists

<AccordionGroup>
<Accordion title="GET /v1/email-integrations">
**Personal mailboxes** for current user (Account settings).

**Response Format:**
```json
{
  "items": [...],
  "nextCursor": "opaque_string_for_pagination"
}
```

**Pagination:**
- Ordering: `connected_at DESC, id ASC` (keyset)
- Default limit: 20
- Max limit: 50
- Query param: `?cursor=` for next page
</Accordion>

<Accordion title="GET /v1/email-integrations/send-accounts">
**Mailboxes for composer** From field:

```json
{
  "personal": {
    "items": [...],
    "nextCursor": "..."
  },
  "organization": {
    "items": [...], 
    "nextCursor": "..."
  }
}
```

**Access Rules:**
- `personal`: caller's personal mailboxes
- `organization`: org mailboxes when caller has `SYSTEM_ADMIN` or `SYSTEM_OWNER`

**Pagination Query Params:**
- `cursorPersonal`
- `cursorOrganization` 
- Use literal `__exhausted__` when stream is complete
</Accordion>

<Accordion title="GET /v1/email-integrations/organization">
**Organization mailboxes only** (`@CheckAccess` - admin/owner required).

Uses `findOrganizationMailboxesPage` with `populate(['connectedBy'])` for OAuth actor info.

**Ordering:** `is_default DESC, connected_at DESC, id ASC` (offset pagination)
</Accordion>
</AccordionGroup>

### Access Control

**Resolution Logic:**
Uses `EmailIntegrationService.getAccessibleIntegrationFindWhere`:
- **Personal rows:** Only accessible when `user_id` matches JWT user
- **Organization rows:** Only accessible with `SYSTEM_ADMIN` or `SYSTEM_OWNER`
- Missing access returns **404** (same as unknown ID)

**Send Permissions:**
- **Personal send:** Only mailbox owner (`user_id` must match JWT user)
- **Organization send:** `SYSTEM_ADMIN` or `SYSTEM_OWNER` only

## Data Model

### Integration Levels

<Tabs>
<Tab title="Personal Level">
- `user_id` NOT NULL
- `connected_by_id` NULL
- Unique constraint: `(user_id, organization_id, provider)` where `is_deleted = false` and `level = 'PERSONAL'`
</Tab>

<Tab title="Organization Level">  
- `user_id` NULL
- `connected_by_id` NOT NULL (OAuth actor)
- Unique constraint: `(organization_id, provider, lower(email))` where `is_deleted = false` and `level = 'ORGANIZATION'`
- Optional `is_default` with at most one default per organization
</Tab>
</Tabs>

### Cross-Level Uniqueness

<Warning>
Active rows cannot mix personal and organization levels for the same `organization_id + lower(email)`.
</Warning>

**Implementation:**
- Migration: `Migration20260521171000_PreventEmailIntegrationCrossLevelMailboxConflicts`
- Trigger: `email_integrations_cross_level_mailbox_conflict_trigger`
- Uses transaction advisory locks per organization/mailbox
- Prevents concurrent inserts creating split personal/workspace connections

### Token Storage

**Encryption:**
- `encrypted_access_token`
- `encrypted_refresh_token`
- AES-256-GCM via shared `EncryptionService`
- Key: `ENCRYPTION_KEY` / `app.encryption.key` (64 hex characters)

**Display Fields:**
- `email`: Provider mailbox/profile email (normalized to lowercase on write)
