## Overview

The integrations module provides connectivity to external services and platforms within PropWise CRM. It currently supports email provider integrations (Gmail and Outlook) via OAuth2 authentication, and Tiptap AI for rich text editing capabilities. The module handles OAuth flows, token management, email sending, and secure access to third-party services.

<Info>
**Last Updated:** 2024-01-15  
**Status:** Active
</Info>

### Key Design Decisions

<AccordionGroup>
  <Accordion title="Provider-agnostic Email Integration">
    Uses enum-based provider differentiation to support multiple email services
  </Accordion>
  <Accordion title="Token Refresh Automation">
    Implements scheduled token refresh to maintain continuous integration connectivity
  </Accordion>
  <Accordion title="Tenant-aware Integrations">
    All integrations are scoped to organizations using tenant context
  </Accordion>
  <Accordion title="Secure Token Storage">
    Email access tokens are stored encrypted in the database
  </Accordion>
  <Accordion title="One Integration Per Email">
    Business rule enforced via unique index per organization
  </Accordion>
</AccordionGroup>

## Entities

### EmailIntegration

Represents a connection to an external email provider for a specific user within an organization.

```typescript
interface EmailIntegration {
  id: string;                    // UUID primary key
  email: string;                 // Email address being integrated
  provider: EmailProvider;       // OUTLOOK | GMAIL
  accessToken: string;          // Encrypted OAuth access token
  refreshToken?: string;        // Encrypted OAuth refresh token
  tokenExpiresAt?: Date;        // Token expiration timestamp
  metadata?: Record<string, any>; // Provider-specific data
  isActive: boolean;            // Integration status
  lastUsedAt?: Date;           // Last activity timestamp
  user: User;                   // Owner of the integration
  organization: Organization;   // Organization scope
  isDeleted: boolean;          // Soft delete flag
  createdAt: Date;
  updatedAt: Date;
}
```

<Note>
**Unique Constraint**: `(user, organization, email)` where `isDeleted = false`
</Note>

### EmailProvider Enum

```typescript
enum EmailProvider {
  OUTLOOK = 'outlook',
  GMAIL = 'gmail'
}
```

## API Endpoints

### Email Integration Management

<Tabs>
  <Tab title="List Integrations">
    #### GET /email-integrations
    Lists all email integrations for the current user and organization.

    **Response:**
    ```json
    {
      "data": [
        {
          "id": "uuid",
          "email": "user@example.com",
          "provider": "gmail",
          "isActive": true,
          "lastUsedAt": "2024-01-15T10:00:00Z",
          "createdAt": "2024-01-10T10:00:00Z"
        }
      ],
      "meta": {
        "total": 1,
        "page": 1,
        "limit": 10
      }
    }
    ```
  </Tab>

  <Tab title="Create Integration">
    #### POST /email-integrations
    Creates a new email integration.

    **Request:**
    ```json
    {
      "email": "user@example.com",
      "provider": "gmail",
      "accessToken": "encrypted_token",
      "refreshToken": "encrypted_refresh_token",
      "tokenExpiresAt": "2024-01-15T11:00:00Z",
      "metadata": {}
    }
    ```
  </Tab>

  <Tab title="Update Integration">
    #### PUT /email-integrations/:id
    Updates an existing email integration.

    **Request Body:** Same as create, all fields optional
  </Tab>

  <Tab title="Delete Integration">
    #### DELETE /email-integrations/:id
    Soft deletes an email integration.

    **Response:** `204 No Content`
  </Tab>
</Tabs>

#### POST /email-integrations/:id/send

Sends an email through the specified integration.

**Request:**
```json
{
  "to": ["recipient@example.com"],
  "subject": "Email Subject",
  "body": "Email content",
  "isHtml": true,
  "cc": ["cc@example.com"],
  "bcc": ["bcc@example.com"],
  "attachments": [
    {
      "filename": "document.pdf",
      "content": "base64_encoded_content",
      "contentType": "application/pdf"
    }
  ]
}
```

### OAuth Authentication

<Steps>
  <Step title="Initiate OAuth Flow">
    #### GET /oauth/gmail/auth
    Initiates Gmail OAuth flow.

    **Query Parameters:**
    - `state`: Encoded user and organization context

    #### GET /oauth/outlook/auth
    Initiates Outlook OAuth flow.

    **Query Parameters:**
    - `state`: Encoded user and organization context
  </Step>

  <Step title="Handle Callback">
    #### GET /oauth/gmail/callback
    Handles Gmail OAuth callback.

    **Query Parameters:**
    - `code`: Authorization code from Google
    - `state`: Original state parameter

    #### GET /oauth/outlook/callback
    Handles Outlook OAuth callback.

    **Query Parameters:**
    - `code`: Authorization code from Microsoft
    - `state`: Original state parameter
  </Step>
</Steps>

### Tiptap AI Integration

#### GET /tiptap-ai/token

Generates a short-lived JWT token for Tiptap AI services.

**Response:**
```json
{
  "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..." // JWT valid for 1 hour
}
```

<Warning>
The JWT token expires after 1 hour and must be refreshed for continued access.
</Warning>

## Business Rules

### Email Integrations

<Check>
**One Integration Per Email**: Users can have only one active integration per email address per organization
</Check>

<Check>
**Token Expiration**: Expired tokens are automatically refreshed via scheduled jobs
</Check>

<Check>
**Provider Validation**: Only supported email providers (Gmail, Outlook) are allowed
</Check>

<Check>
**Organization Scoping**: All integrations are scoped to the user's current organization
</Check>

<Check>
**Soft Deletion**: Integrations are soft-deleted to maintain audit history
</Check>

### OAuth Flows

1. **State Validation**: OAuth state parameter must contain valid user and organization context
2. **Secure Redirects**: Only whitelisted redirect URLs are allowed
3. **Token Exchange**: Authorization codes must be exchanged for tokens immediately
4. **Error Handling**: OAuth errors redirect to frontend with appropriate error messages

### Token Management

<Tip>
Tokens are refreshed 5 minutes before expiration to ensure uninterrupted service.
</Tip>

1. **Automatic Refresh**: Tokens are refreshed 5 minutes before expiration
2. **Retry Logic**: Failed refresh attempts are retried with exponential backoff
3. **Encryption**: All stored tokens are encrypted at rest
4. **Expiration Handling**: Expired tokens trigger re-authentication flows

## Patterns and Conventions

### Error Handling

<AccordionGroup>
  <Accordion title="OAuth Errors">
    OAuth errors redirect to frontend with error parameters
  </Accordion>
  <Accordion title="API Errors">
    API errors return standard HTTP status codes
  </Accordion>
  <Accordion title="Token Refresh Failures">
    Token refresh failures trigger re-authentication
  </Accordion>
  <Accordion title="Rate Limiting">
    Rate limiting applied to token refresh operations
  </Accordion>
</AccordionGroup>

### Security

<Warning>
All security measures are critical for protecting user data and maintaining integration integrity.
</Warning>

- All OAuth tokens stored encrypted using application-level encryption
- JWT tokens for Tiptap AI have short expiration (1 hour)
- OAuth state parameters prevent CSRF attacks
- Integration endpoints require authentication

### Monitoring

The system tracks various metrics for operational visibility:

- Token refresh operations are logged with structured data
- Email sending attempts tracked with success/failure metrics
- Integration health monitored via last usage timestamps
- Failed authentication attempts logged for security analysis
