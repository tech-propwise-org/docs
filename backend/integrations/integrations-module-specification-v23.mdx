The integrations module provides connectivity to external services and platforms within PropWise CRM. It currently supports email provider integrations (Gmail and Outlook) via OAuth2 authentication, and Tiptap AI for rich text editing capabilities.

<Note>
**Last Updated:** 2024-01-15 | **Status:** Active
</Note>

## Architecture overview

The integrations module is organized into three main sub-modules:

- **email-integration**: Manages email provider connections and email sending functionality
- **oauth**: Calendar OAuth controllers only (`google-calendar`, `outlook-calendar`). **Mail** OAuth lives in **email-integration** (`EmailOAuthFlowService`, `EmailOAuthCallbackController`)
- **tiptap-ai**: Provides JWT token generation for Tiptap AI services

### Key design decisions

<CardGroup cols={2}>
  <Card title="Provider-agnostic integration" icon="plug">
    Uses enum-based provider differentiation to support multiple email services
  </Card>
  <Card title="Token refresh automation" icon="arrows-rotate">
    Implements scheduled token refresh to maintain continuous integration connectivity
  </Card>
  <Card title="Tenant-aware integrations" icon="building">
    All integrations are scoped to organizations using tenant context
  </Card>
  <Card title="Secure token storage" icon="shield">
    OAuth access/refresh tokens are AES-GCM encrypted at rest
  </Card>
</CardGroup>

<Info>
**Personal vs workspace mailboxes**: `EmailIntegrationLevel` distinguishes **PERSONAL** (owned by `user_id`) and **ORGANIZATION** (`user_id` null, `connected_by_id` stores OAuth actor). Partial unique indexes enforce one active personal row per `(user, organization, provider)` and one org mailbox per `(organization, provider, lower(email))`.
</Info>

## Data entities

### EmailIntegration

Represents a connection to Gmail or Outlook for sending mail. Either a **personal** mailbox (`level=PERSONAL`, `user` required) or a **workspace** mailbox (`level=ORGANIZATION`, `user` null, `connectedBy` set).

```typescript
interface EmailIntegration {
  id: string;
  email: string;
  provider: EmailProvider;
  level: 'PERSONAL' | 'ORGANIZATION';
  isDefault?: boolean; // ORGANIZATION: optional default sender for admins
  encryptedAccessToken?: string;
  encryptedRefreshToken?: string;
  scopes?: string;
  externalAccountId?: string;
  lastSyncErrorAt?: Date;
  lastSyncErrorMessage?: string;
  tokenExpiresAt?: Date;
  metadata?: Record<string, any>;
  isActive: boolean;
  lastUsedAt?: Date;
  user?: User | null; // PERSONAL owner; null for ORGANIZATION
  connectedBy?: User | null; // OAuth actor for ORGANIZATION
  organization: Organization;
  isDeleted: boolean;
  createdAt: Date;
  updatedAt: Date;
}
```

<AccordionGroup>
  <Accordion title="Unique constraints (partial, where is_deleted = false)">
    - **PERSONAL:** `(user_id, organization_id, provider)`
    - **ORGANIZATION:** `(organization_id, provider, lower(email))`; at most one `is_default` org mailbox per organization
    - **Cross-level mailbox rule:** the same `organization_id + lower(email)` cannot appear in both **PERSONAL** and **ORGANIZATION** rows, regardless of provider. The same mailbox may still be connected by multiple personal users.
  </Accordion>
</AccordionGroup>

### EmailProvider enum

```typescript
enum EmailProvider {
  OUTLOOK = 'outlook',
  GMAIL = 'gmail',
}
```

## API endpoints

### Email integration management

#### GET /email-integrations

Lists **PERSONAL** email integrations for the current user.

```bash
GET /email-integrations
Authorization: Bearer {token}
```

#### GET /email-integrations/send-accounts

Returns mailboxes usable as **From** in the composer: caller's personal integrations plus **ORGANIZATION** rows when the caller has `SYSTEM_ADMIN` or `SYSTEM_OWNER` permissions.

```bash
GET /email-integrations/send-accounts
Authorization: Bearer {token}
```

#### GET /email-integrations/organization

Lists **ORGANIZATION** mailboxes (requires `SYSTEM_ADMIN` or `SYSTEM_OWNER` permissions).

```bash
GET /email-integrations/organization
Authorization: Bearer {token}
```

#### OAuth authorization endpoints

<Tabs>
  <Tab title="Workspace mailbox OAuth">
    ```bash
    POST /v1/email-integrations/organization/oauth/:provider/authorize
    Authorization: Bearer {token}
    ```
    
    Starts OAuth for a workspace mailbox (requires same permissions as `GET .../organization`).
  </Tab>
  
  <Tab title="Personal mailbox OAuth">
    ```bash
    POST /v1/email-integrations/oauth/:provider/authorize
    Authorization: Bearer {token}
    ```
    
    Starts OAuth for **personal** mail (`EmailIntegrationLevel.PERSONAL`).
  </Tab>
</Tabs>

<Warning>
Creating integrations by POSTing plaintext OAuth tokens is **not supported**. Use the authorize endpoint and provider callback only.
</Warning>

#### PUT /email-integrations/:id

Updates an existing email integration.

```bash
PUT /email-integrations/{id}
Authorization: Bearer {token}
Content-Type: application/json

{
  "isActive": true,
  "isDefault": false
}
```

#### DELETE /email-integrations/:id

Soft deletes an email integration.

```bash
DELETE /email-integrations/{id}
Authorization: Bearer {token}
```

#### POST /email-integrations/:id/send

Sends an email through the specified integration.

```bash
POST /email-integrations/{id}/send
Authorization: Bearer {token}
Content-Type: application/json
```

**Request body:**

```typescript
{
  to: string[];
  subject: string;
  body: string;
  isHtml?: boolean;
  cc?: string[];
  bcc?: string[];
  attachments?: Array<{
    filename: string;
    content: string; // base64
    contentType: string;
  }>;
}
```

### Tiptap AI integration

#### GET /tiptap-ai/token

Generates a short-lived JWT token for Tiptap AI services.

```bash
GET /tiptap-ai/token
Authorization: Bearer {token}
```

**Response:**

```typescript
{
  token: string; // JWT valid for 1 hour
}
```

## OAuth authentication flow

<Steps>
  <Step title="Initialize OAuth">
    Call the appropriate authorize endpoint (`/oauth/:provider/authorize` or `/organization/oauth/:provider/authorize`) to start the OAuth flow.
  </Step>
  
  <Step title="User authorization">
    User is redirected to the provider's authorization page (Gmail or Outlook) to grant permissions.
  </Step>
  
  <Step title="Callback processing">
    Provider redirects to public callback endpoints:
    - `GET /auth/gmail/callback`
    - `GET /auth/outlook/callback`
  </Step>
  
  <Step title="Token exchange">
    Authorization code is exchanged for access and refresh tokens, which are encrypted and stored.
  </Step>
</Steps>

<Info>
See `Docs/EMAIL_INTEGRATION_SPECIFICATION.md` for the canonical mail OAuth flow including tenant-bound state parameters and token encryption details.
</Info>

## Business rules

### Email integrations

<Check>
**One integration per provider**: At most one active row per `(user, organization, provider)` — reconnect updates tokens on the same row
</Check>

<Check>
**Token expiration**: Expired tokens are automatically refreshed via scheduled jobs
</Check>

<Check>
**Provider validation**: Only supported email providers (Gmail, Outlook) are allowed
</Check>

<Check>
**Organization scoping**: All integrations are scoped to the user's current organization
</Check>

<Check>
**Soft deletion**: Integrations are soft-deleted to maintain audit history
</Check>

<Warning>
**No personal/workspace split**: A normalized mailbox address may be connected by multiple **PERSONAL** users, but cannot be connected at **PERSONAL** and **ORGANIZATION** levels in the same organization simultaneously.
</Warning>

### OAuth flows

1. **State validation**: OAuth state parameter must contain valid user, organization, provider, and token-version context
2. **Secure redirects**: Only whitelisted redirect URLs are allowed
3. **Token exchange**: Authorization codes must be exchanged for tokens immediately
4. **Error handling**: OAuth errors redirect to frontend with appropriate error messages

### Token management

1. **Automatic refresh**: Tokens are refreshed 5 minutes before expiration
2. **Retry logic**: Failed refresh attempts are retried with exponential backoff
3. **Encryption**: All stored tokens are encrypted at rest
4. **Expiration handling**: Expired tokens trigger re-authentication flows

## Patterns and conventions

### Error handling

- OAuth errors redirect to frontend with error parameters
- API errors return standard HTTP status codes
- Token refresh failures trigger re-authentication
- Rate limiting applied to token refresh operations

### Security

<Tip>
All OAuth tokens are stored encrypted using application-level AES-GCM encryption. JWT tokens for Tiptap AI have short expiration (1 hour), and OAuth state parameters prevent CSRF attacks.
</Tip>

### Monitoring

- Token refresh operations are logged with structured data
- Email sending attempts tracked with success/failure metrics
- Integration health monitored via last usage timestamps
- Failed authentication attempts logged for security analysis

### Scheduled tasks

- **Token refresh**: Runs every 30 minutes to refresh expiring tokens
- **Cleanup**: Periodic cleanup of expired or unused integrations
- **Health checks**: Regular validation of integration connectivity
