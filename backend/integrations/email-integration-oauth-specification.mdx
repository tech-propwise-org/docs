This document covers Gmail and Outlook **email** OAuth flows, token storage, refresh mechanisms, and API surfaces. Calendar OAuth is documented separately in the Calendar System Deep Dive.

## OAuth Flow Overview

The email integration OAuth process operates within a tenant session context, supporting both personal and organization-level mailbox connections.

<Steps>
<Step title="Initialize tenant session">
User selects an organization and the SPA holds a **tenant JWT**.
</Step>

<Step title="Start OAuth authorization">
Choose between personal or organization-level integration:

<Tabs>
<Tab title="Personal Mailbox">
```http
POST /v1/email-integrations/oauth/:provider/authorize
```
- `provider`: `gmail` | `outlook`
- Level: `EmailIntegrationLevel.PERSONAL`
- Uses standard `/v1` API authentication
</Tab>

<Tab title="Organization Mailbox">
```http
POST /v1/email-integrations/organization/oauth/:provider/authorize
```
- Requires `SYSTEM_ADMIN` or `SYSTEM_OWNER` permissions
- Level: `EmailIntegrationLevel.ORGANIZATION`
- Uses `@CheckAccess` authorization
</Tab>
</Tabs>
</Step>

<Step title="Receive authorization URL">
Backend returns `{ authorizationUrl }` containing:
- Google/Microsoft authorize URL
- HMAC-signed `state` parameter via `SignedPayloadService`
- Purpose: `email.oauth.state`
- Payload includes: `userId`, `organizationId`, `tokenVersion`, `provider`, optional `level`
</Step>

<Step title="Handle provider callback">
Provider redirects to public callback endpoints:
- `GET /auth/gmail/callback`
- `GET /auth/outlook/callback`

Backend verifies `state` and exchanges `code` via provider driver.
</Step>
</Steps>

## Gmail Access Verification

<Warning>
Gmail integrations require additional verification before persistence to prevent saving broken integrations.
</Warning>

Before saving Gmail integrations, the backend performs access verification:

1. **API Call**: `GET .../gmail/v1/users/me/profile` with new access token
2. **Failure Handling**: If Google returns 400 or 403:
   - Integration is **not persisted**
   - Callback HTML posts `oauth-error` with `error: EMAIL_OAUTH_GMAIL_ACCESS_BLOCKED_CODE`
   - Error constant: `GMAIL_ACCESS_BLOCKED`

**Required Scopes:**
- `gmail.send` - for outbound email
- `gmail.metadata` - for profile access verification
- `userinfo.email` - for profile mailbox address

<Note>
Outlook integrations skip this verification step as they don't require similar access validation.
</Note>

## Security and Access Control

### Callback Access Re-check

Before token exchange and during database persistence, the system re-validates OAuth actor permissions:

<Check>
**Validation Checklist:**
- `User.isActive` status
- Matching `User.tokenVersion` from signed state
- Active `user_org_roles` for `(userId, organizationId)`
- For `ORGANIZATION` level: current `system.admin` or `system.owner` permissions
</Check>

Implemented via `AccessService.assertOAuthCallbackOrgAccessInTransaction`.

### Cross-Level Mailbox Rules

<Warning>
A mailbox address cannot exist at both PERSONAL and ORGANIZATION levels within the same organization.
</Warning>

**Conflict Resolution:**
- **PERSONAL** OAuth rejected if active **ORGANIZATION** row exists for same normalized email
- **ORGANIZATION** OAuth rejected if any active **PERSONAL** row exists for same normalized email
- Provider type intentionally ignored for cross-level conflicts

**Redundant Reconnect Prevention:**
Healthy integrations (active, no sync errors, valid tokens) reject re-OAuth attempts with `oauth-error` instead of `oauth-success`.

## Data Model Structure

### Integration Levels

<Tabs>
<Tab title="PERSONAL">
```typescript
{
  level: "PERSONAL",
  user_id: "uuid",           // NOT NULL
  connected_by_id: null,
  // Unique: (user_id, organization_id, provider) 
  // where is_deleted = false
}
```
</Tab>

<Tab title="ORGANIZATION">
```typescript
{
  level: "ORGANIZATION", 
  user_id: null,
  connected_by_id: "uuid",   // OAuth actor, NOT NULL
  is_default: boolean,       // Optional, max one per org
  // Unique: (organization_id, provider, lower(email))
  // where is_deleted = false
}
```
</Tab>
</Tabs>

### Token Storage

All OAuth tokens are encrypted using AES-256-GCM:

```typescript
{
  encrypted_access_token: string,
  encrypted_refresh_token: string,
  token_expires_at: DateTime,
  // Encrypted via shared EncryptionService
  // Key: ENCRYPTION_KEY (64 hex chars)
}
```

## API Endpoints

### List Integrations

<CodeGroup>
```http GET /v1/email-integrations
// Personal mailboxes for current user only
// Response: { items, nextCursor }
// Pagination: ?cursor=&limit= (default 20, max 50)
```

```http GET /v1/email-integrations/organization  
// Organization mailboxes (@CheckAccess - admin/owner)
// Includes connectedBy relation when populated
// Uses offset pagination with same limits
```

```http GET /v1/email-integrations/send-accounts
// Mailboxes available as "From" in composer
// Returns both personal and organization streams
// Query params: cursorPersonal, cursorOrganization
```
</CodeGroup>

### Individual Integration Operations

<AccordionGroup>
<Accordion title="Get Integration">
```http
GET /v1/email-integrations/:id
```
Access controlled via `getAccessibleIntegrationFindWhere`:
- **PERSONAL**: Only when `user_id` matches JWT user
- **ORGANIZATION**: Requires `SYSTEM_ADMIN` or `SYSTEM_OWNER`
</Accordion>

<Accordion title="Update Integration">
```http
PUT /v1/email-integrations/:id
```
Same access control as GET. Supports metadata updates via `UpdateEmailIntegrationDto.metadata`.
</Accordion>

<Accordion title="Delete Integration">
```http
DELETE /v1/email-integrations/:id
```
Soft delete with same access control patterns.
</Accordion>

<Accordion title="Send Email">
```http
POST /v1/email-integrations/:id/send
```
**Send Access Control:**
- **Personal**: Only mailbox owner (`user_id` = JWT user)
- **Organization**: `SYSTEM_ADMIN` or `SYSTEM_OWNER` only
</Accordion>
</AccordionGroup>

## Error Handling

### Callback Errors

The OAuth callback uses `postMessage` with secure origin validation:

```typescript
// Target origin from FRONTEND_URL, never '*'
window.postMessage({
  type: 'oauth-error',
  error: 'GMAIL_ACCESS_BLOCKED', // or other error codes
  message: 'User-friendly error description'
}, app.postMessageTargetOrigin);
```

**Common Error Types:**
- `GMAIL_ACCESS_BLOCKED` - Gmail API access denied
- `OAUTH_CALLBACK_TENANT_ACCESS_DENIED` - Permission validation failed
- Provider-specific token exchange failures (logged server-side)
