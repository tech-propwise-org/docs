The email integration system provides seamless connectivity to external email providers within PropWise CRM. Currently supporting Gmail and Outlook through OAuth2 authentication, this module enables users to send emails directly through their connected accounts while maintaining secure token management and automated refresh capabilities.

## Overview

The email integration module handles the complete lifecycle of email provider connections, from initial OAuth authentication to ongoing email operations. All integrations are organization-scoped and enforce a one-integration-per-email-per-organization business rule to maintain consistency.

<Info>
Each user can have only one active integration per email address within their organization, ensuring clear ownership and preventing conflicts.
</Info>

## EmailIntegration Entity

The core entity that represents a connection to an external email provider:

```typescript
interface EmailIntegration {
  id: string;                    // UUID primary key
  email: string;                 // Email address being integrated
  provider: EmailProvider;       // OUTLOOK | GMAIL
  accessToken: string;          // Encrypted OAuth access token
  refreshToken?: string;        // Encrypted OAuth refresh token
  tokenExpiresAt?: Date;        // Token expiration timestamp
  metadata?: Record<string, any>; // Provider-specific data
  isActive: boolean;            // Integration status
  lastUsedAt?: Date;           // Last activity timestamp
  user: User;                   // Owner of the integration
  organization: Organization;   // Organization scope
  isDeleted: boolean;          // Soft delete flag
  createdAt: Date;
  updatedAt: Date;
}
```

<Note>
All OAuth tokens are automatically encrypted before storage and include automatic refresh capabilities to maintain continuous connectivity.
</Note>

## OAuth Authentication Flow

### Gmail Integration

<Steps>
  <Step title="Initiate OAuth Flow">
    **GET** `/oauth/gmail/auth?state={encodedContext}`
    
    Redirects users to Google's OAuth consent screen. The state parameter contains encoded user and organization context for security.
  </Step>

  <Step title="Handle Callback">
    **GET** `/oauth/gmail/callback?code={authCode}&state={encodedContext}`
    
    Processes the authorization code from Google and exchanges it for access tokens.
  </Step>

  <Step title="Create Integration">
    Upon successful token exchange, an EmailIntegration record is automatically created and tokens are securely stored.
  </Step>
</Steps>

### Outlook Integration

<Steps>
  <Step title="Initiate OAuth Flow">
    **GET** `/oauth/outlook/auth?state={encodedContext}`
    
    Redirects users to Microsoft's OAuth consent screen for Office 365 or Outlook.com access.
  </Step>

  <Step title="Handle Callback">
    **GET** `/oauth/outlook/callback?code={authCode}&state={encodedContext}`
    
    Exchanges the authorization code for Microsoft Graph API tokens.
  </Step>

  <Step title="Create Integration">
    Creates the integration record with encrypted Microsoft Graph tokens for ongoing email access.
  </Step>
</Steps>

## Token Management

### Automatic Refresh

The system implements robust token management to ensure continuous email connectivity:

<AccordionGroup>
  <Accordion title="Refresh Schedule">
    - Tokens are automatically refreshed 5 minutes before expiration
    - Refresh job runs every 30 minutes to check for expiring tokens
    - Failed refresh attempts use exponential backoff retry logic
  </Accordion>

  <Accordion title="Security Measures">
    - All tokens are encrypted at rest using application-level encryption
    - OAuth state parameters prevent CSRF attacks
    - Only whitelisted redirect URLs are permitted
    - Integration endpoints require valid authentication
  </Accordion>

  <Accordion title="Error Handling">
    - Expired tokens automatically trigger re-authentication flows
    - Rate limiting prevents abuse of token refresh operations
    - Failed authentication attempts are logged for security analysis
  </Accordion>
</AccordionGroup>

## Integration Health Monitoring

The system continuously monitors integration health through several mechanisms:

- **Usage Tracking**: `lastUsedAt` timestamps track integration activity
- **Token Validation**: Regular validation of token connectivity
- **Cleanup Operations**: Periodic removal of expired or unused integrations
- **Structured Logging**: Detailed logs for token refresh and email sending operations

<Tip>
Monitor the `isActive` status and `lastUsedAt` timestamps to identify integrations that may need attention or re-authentication.
</Tip>

## Error Handling

Common error scenarios and their handling:

| Error Type | Handling Strategy |
|------------|------------------|
| Token Expiration | Automatic refresh with fallback to re-authentication |
| OAuth Errors | Redirect to frontend with descriptive error parameters |
| Rate Limiting | Exponential backoff with retry scheduling |
| Provider Outages | Graceful degradation with user notification |

<Warning>
When OAuth flows fail, users are redirected to the frontend with error parameters that should be displayed to guide re-authentication.
</Warning>
