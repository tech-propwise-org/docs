## Overview

The integrations module provides connectivity to external services and platforms within PropWise CRM. It currently supports email provider integrations (Gmail and Outlook) via OAuth2 authentication, and Tiptap AI for rich text editing capabilities. The module handles OAuth flows, token management, email sending, and secure access to third-party services.

<Info>
**Last Updated:** 2024-01-15  
**Status:** Active
</Info>

## Architecture

The integrations module is organized into three main sub-modules:

<CardGroup cols={3}>
  <Card title="Email Integration" icon="envelope">
    Manages email provider connections and email sending functionality
  </Card>
  <Card title="OAuth" icon="lock">
    Calendar OAuth controllers only (google-calendar, outlook-calendar)
  </Card>
  <Card title="Tiptap AI" icon="sparkles">
    Provides JWT token generation for Tiptap AI services
  </Card>
</CardGroup>

<Note>
**Mail OAuth** lives in the `email-integration` sub-module (`EmailOAuthFlowService`, `EmailOAuthCallbackController`), not in the `oauth` sub-module.
</Note>

### Key design decisions

<AccordionGroup>
  <Accordion title="Provider-agnostic email integration">
    Uses enum-based provider differentiation to support multiple email services
  </Accordion>
  
  <Accordion title="Token refresh automation">
    Implements scheduled token refresh to maintain continuous integration connectivity
  </Accordion>
  
  <Accordion title="Tenant-aware integrations">
    All integrations are scoped to organizations using tenant context
  </Accordion>
  
  <Accordion title="Secure token storage">
    OAuth access/refresh tokens are **AES-GCM encrypted** at rest (`encryptedAccessToken` / `encryptedRefreshToken`)
  </Accordion>
  
  <Accordion title="Personal vs workspace mailboxes">
    **`EmailIntegrationLevel`** distinguishes **PERSONAL** (owned by **`user_id`**) and **ORGANIZATION** (`user_id` null, **`connected_by_id`** stores OAuth actor). Partial unique indexes enforce one active personal row per `(user, organization, provider)` and one org mailbox per `(organization, provider, lower(email))`; a cross-level trigger rejects the same normalized mailbox address from being connected as both personal and workspace in the same organization.
  </Accordion>
</AccordionGroup>

## Entities

### EmailIntegration

Represents a connection to Gmail or Outlook for sending mail. Either a **personal** mailbox (**`level=PERSONAL`**, **`user`** required) or a **workspace** mailbox (**`level=ORGANIZATION`**, **`user`** null, **`connectedBy`** set).

```typescript
interface EmailIntegration {
  id: string;
  email: string;
  provider: EmailProvider;
  level: 'PERSONAL' | 'ORGANIZATION';
  isDefault?: boolean; // ORGANIZATION: optional default sender for admins
  encryptedAccessToken?: string;
  encryptedRefreshToken?: string;
  scopes?: string;
  externalAccountId?: string;
  lastSyncErrorAt?: Date;
  lastSyncErrorMessage?: string;
  tokenExpiresAt?: Date;
  metadata?: Record<string, any>;
  isActive: boolean;
  lastUsedAt?: Date;
  user?: User | null; // PERSONAL owner; null for ORGANIZATION
  connectedBy?: User | null; // OAuth actor for ORGANIZATION
  organization: Organization;
  isDeleted: boolean;
  createdAt: Date;
  updatedAt: Date;
}
```

#### Unique constraints

<Warning>
Partial constraints apply only where `is_deleted = false`
</Warning>

- **PERSONAL:** `(user_id, organization_id, provider)`
- **ORGANIZATION:** `(organization_id, provider, lower(email))`; at most one **`is_default`** org mailbox per organization
- **Cross-level mailbox rule:** the same `organization_id + lower(email)` cannot appear in both **PERSONAL** and **ORGANIZATION** rows, regardless of provider. The same mailbox may still be connected by multiple personal users.

### EmailProvider enum

```typescript
enum EmailProvider {
  OUTLOOK = 'outlook',
  GMAIL = 'gmail',
}
```

### OAuth authentication

<Info>
See **`Docs/EMAIL_INTEGRATION_SPECIFICATION.md`** for the canonical mail OAuth flow (tenant `POST /v1/email-integrations/oauth/:provider/authorize`, signed state bound to `userId` / `organizationId` / `tokenVersion`, public **`GET /auth/gmail/callback`** / **`GET /auth/outlook/callback`**, encrypted tokens).
</Info>

<Steps>
  <Step title="Initialize OAuth flow">
    Client calls `POST /v1/email-integrations/oauth/:provider/authorize` with redirect URI
  </Step>
  
  <Step title="User authentication">
    User authenticates with the email provider (Gmail or Outlook)
  </Step>
  
  <Step title="Provider callback">
    Provider redirects to `GET /auth/:provider/callback` with authorization code
  </Step>
  
  <Step title="Token exchange">
    Backend exchanges code for access/refresh tokens and encrypts them
  </Step>
  
  <Step title="Return to application">
    User redirected back to the specified redirect URI
  </Step>
</Steps>

## Business Rules

### Email integrations

<AccordionGroup>
  <Accordion title="One integration per provider">
    At most one active row per `(user, organization, provider)` — reconnect updates tokens on the same row
  </Accordion>
  
  <Accordion title="Token expiration">
    Expired tokens are automatically refreshed via scheduled jobs
  </Accordion>
  
  <Accordion title="Provider validation">
    Only supported email providers (Gmail, Outlook) are allowed
  </Accordion>
  
  <Accordion title="Organization scoping">
    All integrations are scoped to the user's current organization
  </Accordion>
  
  <Accordion title="Soft deletion">
    Integrations are soft-deleted to maintain audit history
  </Accordion>
  
  <Accordion title="Mailbox address constraints">
    A normalized mailbox address may be connected by multiple **PERSONAL** users, but cannot be connected at **PERSONAL** and **ORGANIZATION** levels in the same organization at the same time
  </Accordion>
</AccordionGroup>

### OAuth flows

<Check>
**State validation:** OAuth state parameter must contain valid user, organization, provider, and token-version context
</Check>

<Check>
**Secure redirects:** Only whitelisted redirect URLs are allowed
</Check>

<Check>
**Token exchange:** Authorization codes must be exchanged for tokens immediately
</Check>

<Check>
**Error handling:** OAuth errors redirect to frontend with appropriate error messages
</Check>

### Token management

1. **Automatic refresh**: Tokens are refreshed 5 minutes before expiration
2. **Retry logic**: Failed refresh attempts are retried with exponential backoff
3. **Encryption**: All stored tokens are encrypted at rest
4. **Expiration handling**: Expired tokens trigger re-authentication flows

### Calendar conferencing (Google Meet / Teams)

The calendar drivers can request a conferencing link when a CRM Event meeting channel carries `conferenceType`.

<Info>
See `Docs/CALENDAR_SYSTEM_DEEP_DIVE.md` → "Create event" for the full write-back flow.
</Info>

The driver `CalendarEventPayload` gained `conferenceType` (`google_meet` | `teams`) and `conferenceRequestId` (stable CRM event id):

<Tabs>
  <Tab title="Google Meet">
    **Google driver** (`google_meet`): `createEvent` adds `conferenceDataVersion: 1` to the request params and `toGoogleEvent` emits `conferenceData.createRequest` (`requestId = conferenceRequestId`, `conferenceSolutionKey.type = 'hangoutsMeet'`). 
    
    The existing `calendar.events` scope is sufficient — **no re-consent**. The mapper already reads `hangoutLink → meetingLink`.
  </Tab>
  
  <Tab title="Microsoft Teams">
    **Outlook driver** (`teams`): `toGraphEvent` sets `isOnlineMeeting: true` + `onlineMeetingProvider: 'teamsForBusiness'` (skipping the manual link-in-body append). 
    
    The existing `Calendars.ReadWrite` scope is sufficient. The mapper already reads `onlineMeeting.joinUrl → meetingLink`.
    
    <Warning>
    **Consumer `outlook.com` accounts cannot create Teams meetings** — Graph errors are caught per-integration and surfaced as a sync error without failing event creation.
    </Warning>
  </Tab>
</Tabs>

<Note>
Only the **matching** provider receives the conference request; other connected providers get the minted link as description text. Drivers remain DB-free and pure.
</Note>

## Patterns and Conventions

### Error handling

<AccordionGroup>
  <Accordion title="OAuth errors">
    OAuth errors redirect to frontend with error parameters
  </Accordion>
  
  <Accordion title="API errors">
    API errors return standard HTTP status codes
  </Accordion>
  
  <Accordion title="Token refresh failures">
    Token refresh failures trigger re-authentication
  </Accordion>
  
  <Accordion title="Rate limiting">
    Rate limiting applied to token refresh operations
  </Accordion>
</AccordionGroup>

### Security

<CardGroup cols={2}>
  <Card title="Token encryption" icon="lock">
    All OAuth tokens stored encrypted using application-level encryption
  </Card>
  <Card title="Short-lived JWT" icon="clock">
    JWT tokens for Tiptap AI have short expiration (1 hour)
  </Card>
  <Card title="CSRF protection" icon="shield">
    OAuth state parameters prevent CSRF attacks
  </Card>
  <Card title="Authentication required" icon="key">
    Integration endpoints require authentication
  </Card>
</CardGroup>

### Monitoring

- Token refresh operations are logged with structured data
- Email sending attempts tracked with success/failure metrics
- Integration health monitored via last usage timestamps
- Failed authentication attempts logged for security analysis

  <Step title="Health checks">
    Regular validation of integration connectivity
  </Step>
