The integrations module provides connectivity to external services and platforms within PropWise CRM. It supports email provider integrations (Gmail and Outlook) via OAuth2 authentication, calendar services, and Tiptap AI for rich text editing capabilities.

## Architecture Overview

The integrations module is organized into three main sub-modules:

<CardGroup cols={3}>
  <Card title="Email Integration" icon="envelope">
    Manages email provider connections and email sending functionality
  </Card>
  <Card title="OAuth" icon="key">
    Calendar OAuth controllers only (google-calendar, outlook-calendar)
  </Card>
  <Card title="Tiptap AI" icon="robot">
    Provides JWT token generation for AI writing services
  </Card>
</CardGroup>

<Note>
Mail OAuth functionality lives in the **email-integration** module via `EmailOAuthFlowService` and `EmailOAuthCallbackController`, not in the oauth sub-module.
</Note>

### Key Design Decisions

<AccordionGroup>
  <Accordion title="Provider-agnostic email integration">
    Uses enum-based provider differentiation to support multiple email services seamlessly
  </Accordion>
  
  <Accordion title="Token refresh automation">
    Implements scheduled token refresh to maintain continuous integration connectivity
  </Accordion>
  
  <Accordion title="Tenant-aware integrations">
    All integrations are scoped to organizations using tenant context
  </Accordion>
  
  <Accordion title="Secure token storage">
    OAuth access/refresh tokens are AES-GCM encrypted at rest using `encryptedAccessToken` and `encryptedRefreshToken`
  </Accordion>
  
  <Accordion title="Personal vs workspace mailboxes">
    `EmailIntegrationLevel` distinguishes **PERSONAL** (owned by `user_id`) and **ORGANIZATION** (`user_id` null, `connected_by_id` stores OAuth actor)
  </Accordion>
</AccordionGroup>

## Entities

### EmailIntegration

Represents a connection to Gmail or Outlook for sending mail. Can be either a **personal** mailbox or a **workspace** mailbox.

```typescript
interface EmailIntegration {
  id: string;
  email: string;
  provider: EmailProvider;
  level: 'PERSONAL' | 'ORGANIZATION';
  isDefault?: boolean; // ORGANIZATION: optional default sender for admins
  encryptedAccessToken?: string;
  encryptedRefreshToken?: string;
  scopes?: string;
  externalAccountId?: string;
  lastSyncErrorAt?: Date;
  lastSyncErrorMessage?: string;
  tokenExpiresAt?: Date;
  metadata?: Record<string, any>;
  isActive: boolean;
  lastUsedAt?: Date;
  user?: User | null; // PERSONAL owner; null for ORGANIZATION
  connectedBy?: User | null; // OAuth actor for ORGANIZATION
  organization: Organization;
  isDeleted: boolean;
  createdAt: Date;
  updatedAt: Date;
}
```

### Unique Constraints

<Warning>
Partial unique indexes enforce strict mailbox ownership rules where `is_deleted = false`:
</Warning>

- **PERSONAL:** `(user_id, organization_id, provider)`
- **ORGANIZATION:** `(organization_id, provider, lower(email))`
- **Cross-level mailbox rule:** The same `organization_id + lower(email)` cannot appear in both **PERSONAL** and **ORGANIZATION** rows, regardless of provider

### EmailProvider Enum

```typescript
enum EmailProvider {
  OUTLOOK = 'outlook',
  GMAIL = 'gmail',
}
```

## Business Rules

### Email Integration Rules

<Steps>
  <Step title="Provider limits">
    At most one active integration per `(user, organization, provider)` combination
  </Step>
  
  <Step title="Token management">
    Expired tokens are automatically refreshed via scheduled jobs
  </Step>
  
  <Step title="Provider validation">
    Only supported email providers (Gmail, Outlook) are allowed
  </Step>
  
  <Step title="Organization scoping">
    All integrations are scoped to the user's current organization
  </Step>
  
  <Step title="Mailbox restrictions">
    A normalized mailbox address cannot be connected at both PERSONAL and ORGANIZATION levels in the same organization simultaneously
  </Step>
</Steps>

### OAuth Flow Rules

1. **State Validation**: OAuth state parameter must contain valid user, organization, provider, and token-version context
2. **Secure Redirects**: Only whitelisted redirect URLs are allowed
3. **Token Exchange**: Authorization codes must be exchanged for tokens immediately
4. **Error Handling**: OAuth errors redirect to frontend with appropriate error messages

### Token Management

<Tip>
Tokens are refreshed automatically 5 minutes before expiration to ensure uninterrupted service
</Tip>

- **Automatic Refresh**: Scheduled every 30 minutes
- **Retry Logic**: Failed refresh attempts use exponential backoff
- **Encryption**: All stored tokens are AES-GCM encrypted at rest
- **Expiration Handling**: Expired tokens trigger re-authentication flows

## Calendar Conferencing Integration

The system supports Google Meet and Microsoft Teams integration for calendar events:

<Tabs>
  <Tab title="Google Meet">
    - Uses existing `calendar.events` scope (no re-consent required)
    - Creates conference via `conferenceData.createRequest`
    - Maps `hangoutLink` to `meetingLink`
  </Tab>
  
  <Tab title="Microsoft Teams">
    - Uses existing `Calendars.ReadWrite` scope
    - Sets `isOnlineMeeting: true` and `onlineMeetingProvider: 'teamsForBusiness'`
    - Maps `onlineMeeting.joinUrl` to `meetingLink`
    - **Note**: Consumer `outlook.com` accounts cannot create Teams meetings
  </Tab>
</Tabs>

<Warning>
Only the matching provider receives the conference request. Other connected providers get the minted link as description text.
</Warning>

## Integration Dependencies

### Internal Dependencies

- **AuthModule**: Authentication guards and user context
- **TenantContextModule**: Organization scoping
- **ConfigModule**: OAuth client credentials and API keys
- **ScheduleModule**: Cron-based token refresh
- **MikroOrmModule**: Database entity management

### External Services

- **Google APIs**: Gmail integration via OAuth2 and REST APIs
- **Microsoft Graph**: Outlook integration via OAuth2 and Graph API
- **Tiptap AI**: JWT-based authentication for AI writing features

## Security & Monitoring

### Security Measures

<Check>
- All OAuth tokens stored encrypted using application-level AES-GCM encryption
- JWT tokens for Tiptap AI have short expiration (1 hour)
- OAuth state parameters prevent CSRF attacks
- All integration endpoints require authentication
</Check>

### Monitoring & Logging

- Token refresh operations logged with structured data
- Email sending attempts tracked with success/failure metrics
- Integration health monitored via last usage timestamps
- Failed authentication attempts logged for security analysis
