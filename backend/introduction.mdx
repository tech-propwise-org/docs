The PropWise backend is a modular **NestJS** application with feature-based architecture. Every tenant's data is isolated at the database level using PostgreSQL Row-Level Security, and access is governed by a unified RBAC system.

## Tech stack

| Layer      | Technology                                |
| ---------- | ----------------------------------------- |
| Framework  | NestJS 10 + TypeScript                    |
| ORM        | MikroORM 6 (unit-of-work, identity map)   |
| Database   | PostgreSQL 16 (Neon)                      |
| Auth       | JWT + Passport, per-organization sessions |
| Real-time  | Socket.io                                 |
| Storage    | Cloudflare R2                             |
| Validation | class-validator + class-transformer       |

## Module map

```
propwise-backend/
├── modules/
│   ├── user/              # User accounts, profiles
│   ├── organization/      # Tenant boundary (Organization entity)
│   ├── rbac/              # Roles, permissions, teams, invitations
│   ├── crm/               # Lead, Contact, Company, Deal, Person
│   ├── reference-data/    # Read-only second DB (areas, projects, units)
│   ├── developer-portal/  # Separate developer auth + project management
│   ├── off-plan/          # Global off-plan directory
│   ├── inventory/         # Org-scoped secondary units
│   ├── shared-marketplace/# Cross-org listings + deal negotiation
│   └── property-interest/ # Property interests + matching engine
├── common/
│   ├── guards/            # Auth, RLS, developer-auth guards
│   ├── decorators/        # @CheckAccess, @PublicEndpoint, @BypassRLS
│   ├── interceptors/      # Serialization, tenant context
│   └── filters/           # Exception filters
└── config/                # Database, JWT, storage config
```

## How the layers connect

```
Request
  │
  ▼
AuthGuard (JWT validation)
  │
  ▼
TenantInterceptor (SET LOCAL org_id in transaction)
  │
  ▼
@CheckAccess (permission + resource ownership check)
  │
  ▼
Controller → Service → Repository
  │                        │
  │                        ▼
  │                   PostgreSQL (RLS filters rows automatically)
  ▼
Response (DTO serialized with @Expose groups)
```

## Key architectural decisions

<AccordionGroup>
  <Accordion title="Transaction-first multi-tenancy">
    Every request runs inside an explicit transaction. Tenant context is set via `SET LOCAL` — automatically cleaned up on commit/rollback. No middleware-level RLS, no connection-per-tenant.
  </Accordion>

<Accordion title="Unified stakeholder model">
  A single `entity_stakeholder` table manages assignment and commission for both
  leads and deals. No separate assignment tables per entity type.
</Accordion>

<Accordion title="Person-centric identity">
  `Person` is the shared identity layer. `Lead` and `Contact` are business
  relationship views of the same person. `Deal` connects to `Contact`, not
  `Person`.
</Accordion>

<Accordion title="Three-scope real estate">
  Real estate data spans three visibility scopes: **global** (reference data, no
  RLS), **org-scoped** (inventory, standard RLS), and **cross-org** (shared
  marketplace, custom RLS).
</Accordion>

  <Accordion title="DTO serialization groups">
    One DTO per entity with `@Expose()` groups (`LIST`, `DETAIL`, `FULL`) instead of separate `BasicDto` / `DetailDto` classes. Transformation happens in the service layer via `plainToInstance()`.
  </Accordion>
</AccordionGroup>
