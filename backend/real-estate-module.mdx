The real estate module is structured as **six specialized modules** spanning three visibility scopes: **global** (no RLS), **org-private** (standard RLS), and **cross-org** (custom RLS).

An external **reference database** (Neon "propwise-lab") stores **all property data** — both structural (DLD registry) and enrichment (gallery, payment plans, documents, amenities, pricing, geocoding). The CRM database stores only CRM-specific data (developer portal auth, ownership links, property interests, marketplace listings, secondary unit context).

<Info>
All modules consume property data exclusively through `ReferenceDataService`. If the reference database is swapped for an external property API, only `ReferenceDataService` needs to change.
</Info>

## Architecture

```
src/modules/real-estate/
├── reference-data/        # Second DB — ALL property data (structural + enrichment)
│   ├── reference-data.controller.ts       # /reference — areas, cities, communities, developers, property types, units/:id
│   ├── reference-projects.controller.ts   # /reference/projects — search (?type=off-plan), map, counts, detail, nested buildings/units
│   └── reference-buildings.controller.ts  # /reference/buildings — search (?type=off-plan), detail, nested units
├── developer-portal/      # Separate auth + project management (writes via ReferenceDataService)
│   ├── auth/developer-auth.controller.ts  # /developer-portal/auth — signup (with agent card), login, refresh, logout, password
│   ├── profile/developer-profile.controller.ts # /developer-portal/me — profile GET/PATCH, agent-card upload
│   └── project-management/
│       ├── dev-projects.controller.ts     # /developer-portal/projects — project CRUD, collections, nested building create/list
│       └── dev-buildings.controller.ts    # /developer-portal/buildings — building CRUD (flat), collections, nested unit CRUD
├── map/                   # MapService (2GIS geocoding utility — shared across modules)
├── inventory/             # Org-scoped secondary units (standard RLS)
├── shared-marketplace/    # Cross-org listings + deals (custom RLS)
└── property-interest/     # Property interests + matching engine (uses ReferenceDataService)
```

### Cross-database reference pattern

CRM entities that reference off-plan projects or units store **plain integer IDs** (`refProjectId`, `refUnitId`) pointing to the reference database. No cross-database foreign keys exist — this decouples the CRM from the property data source.

```typescript
@Property({ type: 'int', nullable: true })
@Index()
refProjectId?: number;  // ID in reference DB projects table

@Property({ type: 'int', nullable: true })
@Index()
refUnitId?: number;     // ID in reference DB units table
```

## Reference data module

**Path:** `src/modules/real-estate/reference-data/`

Connects to a second PostgreSQL database (Neon "propwise-lab") containing **all property data**: DLD registry data (structural) plus enrichment data (gallery, payment plans, documents, amenities, sales contacts, pricing, geocoding). Provides read endpoints for the off-plan directory and write endpoints for the developer portal.

### Second database connection

Configured in `mikro-orm-reference.config.ts` at project root. Registered in `app.module.ts` with `contextName: 'reference'`:

```typescript
MikroOrmModule.forRoot({
  ...mikroOrmReferenceConfig,
  contextName: 'reference',
});
```

### Entity pattern

All reference entities are mapped to the external DB tables. They have **no UUID primary keys** (use `number` PKs matching the external DB), **no organization FK**, and use **soft delete** (`deletedAt` timestamp) for enrichment entities.

```typescript
@Entity({ tableName: 'projects' })
export class RefProject {
  @PrimaryKey()
  id!: number;

  @Property({ nullable: true })
  name?: string;

  @ManyToOne(() => RefDeveloper, { fieldName: 'developer_id', nullable: true })
  developer?: RefDeveloper;

  @ManyToOne(() => RefArea, { fieldName: 'area_id' })
  area!: RefArea;

  @Property({ type: 'double', nullable: true })
  latitude?: number;

  @Property({ fieldName: 'price_from', type: 'numeric', nullable: true })
  priceFrom?: number;

  @Property({ fieldName: 'is_published', default: false, nullable: true })
  isPublished?: boolean;

  @OneToMany(() => RefProjectGalleryImage, (img) => img.project)
  gallery = new Collection<RefProjectGalleryImage>(this);

  @OneToMany(() => RefPaymentPlan, (p) => p.project)
  paymentPlans = new Collection<RefPaymentPlan>(this);
}
```

## Developer portal module

**Path:** `src/modules/real-estate/developer-portal/`

Provides a completely separate authentication system for real estate developer sales agents. Developers sign up, get admin approval, then manage their off-plan projects through this portal. All project data is stored in the reference database via `ReferenceDataService`.

### Authentication (separate from CRM)

- **Separate JWT secrets**:
  - Access token: `JWT_DEVELOPER_SECRET` (configured in `jwt.config.ts` under `developerToken`, expires in 8h)
  - Refresh token: `JWT_DEVELOPER_REFRESH_SECRET` (configured in `jwt.config.ts` under `developerRefreshToken`, expires in 7d)
- **Token payload**: `{ sub, email, type: 'developer', refDeveloperId, tokenVersion }`
- **Refresh token mechanism**: 64-character random token stored in database with expiry timestamp
- **Guard**: `DeveloperAuthGuard` at class-level validates tokens and sets `request.developerUser`
- **Public endpoints**: Use `@PublicEndpoint()` for signup/login/refresh

### Project management architecture

The developer portal follows a **three-level hierarchy**: Projects contain Buildings, and Buildings contain Units. Projects are lightweight parent containers; Buildings hold all enrichment data (pricing, handover, gallery, etc.); Units are created under Buildings.

<Steps>
<Step title="Project CRUD">
Delegates to `ReferenceDataService` (writes to reference DB). Ownership tracked via `DeveloperProjectLink` (CRM DB).
</Step>
<Step title="Building CRUD">
Verifies project ownership via `DeveloperProjectLink`, then delegates to `ReferenceDataService`. Buildings inherit fields from their parent project by default.
</Step>
<Step title="Unit CRUD">
Verifies building→project ownership, then delegates to `ReferenceDataService`.
</Step>
</Steps>

**Inheritance pattern:**
- Buildings inherit `community`, `area`, `description`, `status`, `percentCompleted`, `startDate`, `endDate` from their parent project by default, but each can be overridden per building.
- **Payment plans**: Set at project level; buildings inherit by default but can define their own.
- **Documents**: Available at both project and building levels (additive).
- **Amenities**: Set at project level; buildings inherit by default but can define their own.
- **Gallery**: Available at both project and building levels.

### Approval flow

<Steps>
<Step title="Signup (with agent card)">
Developer calls `POST /developer-portal/auth/signup` with multipart/form-data including agent card file. Backend validates file type/size, processes images (converts to WebP, resizes), stores in R2. Account created with `isApproved = false`, `isEmailVerified = false`. Email verification code sent (6-digit, 30min expiry).
</Step>
<Step title="Email verification">
Developer verifies email with code. `isEmailVerified = true`. Account enters pending approval state.
</Step>
<Step title="Admin review">
System admin reviews agent card via `GET /system-admin/developer-users/:id`. Admin approves or rejects.
</Step>
<Step title="On approval">
`isApproved = true`, `approvedAt = now()`. Developer can now login.
</Step>
<Step title="Login">
Receives access token (8h) + refresh token (7d via HTTP-only cookie).
</Step>
<Step title="Project publishing">
Projects start as `isPublished=false`. Developer publishes project, making it visible in the off-plan directory.
</Step>
</Steps>

**Agent card details:**
- Signup: multipart/form-data, field name: `agentCard`
- Authenticated update: multipart/form-data, field name: `file`
- Max size: 10 MB
- Formats: JPEG, PNG, GIF, WebP (images) or PDF (documents)
- Images auto-converted to WebP: resized to max 1200x1600, WebP 85% quality
- PDFs stored as-is (no processing)
- Storage: `agent-cards/{userId}/{uploadId}.webp` (images) or `.pdf` (documents)

## Off-plan filtering

"Off-plan" is a **filter** applied to reference data queries, not a separate module. The `MapService` (2GIS geocoding) lives in `src/modules/real-estate/map/` as a shared utility.

### Filtering logic

Off-plan status is **derived** from the project/building's dates and status fields. Pass `?type=off-plan` to `/reference/projects` or `/reference/buildings` search endpoints:

```typescript
private applyOffPlanFilter(qb: QueryBuilder<RefProject | RefBuilding>) {
  qb.andWhere({
    $or: [
      { endDate: { $gt: new Date() } },
      { endDate: null },
    ],
  });
  qb.andWhere({ completionDate: null });
  qb.andWhere({
    status: { $in: ['active', 'pending', 'under_construction', 'announced'] },
  });
  qb.andWhere({ isPublished: true });
}
```

A project/building is considered "off-plan" when:
1. `endDate` is in the future OR null (not yet completed)
2. `completionDate` is null (not yet handed over)
3. `status` is one of: active, pending, under_construction, announced
4. `isPublished` is true

### Off-plan API endpoints

GET /reference/projects?type=off-plan          # Off-plan project search

### Unit number privacy

The `InventorySecurityService` controls unit number visibility:

```typescript
canViewUnitNumber(user, unit): boolean
// Returns true if:
// - Unit number is not hidden (isUnitNumberHidden === false)
// - User has 'real_estate.manage' permission
// - User is the unitManager
// - User has active UnitAccessGrant (via shared-marketplace)

getVisibilityLevel(user, unit): 'full' | 'basic'
```

The `SecondaryUnitDto` constructor accepts `canViewUnitNumber` boolean; if false, `unitNumber` is masked as `'***'`.

### Access control

Inventory is **org-scoped** and uses `@UseGuards(AuthGuard, AccessGuard)` with `@CheckAccess`:

## Shared marketplace module

**Path:** `src/modules/real-estate/shared-marketplace/`

Cross-organization property listing and the **deal-first** negotiation flow. Unit numbers are only revealed upon deal acceptance.

### Core principle: deal-first, access-second

1. **Unit number is protected**: Hidden in SharedListing browsing
2. **Deal commitment first**: Buyer agent proposes a SharedDeal before seeing the unit number
3. **Automated deal creation**: On acceptance, system auto-creates internal Deal in both organizations
4. **Unit number revealed**: Only after SharedDeal is ACCEPTED

### Entities

<AccordionGroup>
<Accordion title="SharedListing">

- UUID PK, **no standard org RLS** (readable by all, writable by owning org)
- ManyToOne: SecondaryUnit, listedByOrganization, listedByAgent
- Denormalized display data: projectName, areaName, communityName, unitType, bedrooms, areaSqft
- Listing details: listingType (Sale/Rent), listedPrice, commissionType, commissionValue
- Status: ListingStatus (ACTIVE, UNDER\_OFFER, SOLD, RENTED, WITHDRAWN)

</Accordion>
<Accordion title="SharedDeal">

- UUID PK, **custom RLS** (visible to both participating orgs)
- Links both sides: listingOrganization/listingAgent, buyerOrganization/buyerAgent
- Auto-created deals: listingDeal (Deal), buyerDeal (Deal) - populated on ACCEPT
- Negotiation: status (SharedDealStatus), proposedPrice, agreedPrice, commissionSplit (JSON)
- Unit reveal: **isUnitNumberRevealed** (false until accepted), unitNumberRevealedAt
- OneToMany: messages (SharedDealMessage[])

</Accordion>
<Accordion title="SharedDealMessage">

- Negotiation audit trail
- type: SharedDealMessageType (PROPOSAL, COUNTER, ACCEPT, REJECT, NOTE, STATUS\_UPDATE)
- terms: JSON `{ price?, commissionSplit? }`

</Accordion>
<Accordion title="UnitAccessRequest / UnitAccessGrant">

- **Internal org-level** unit number access (not marketplace — marketplace uses SharedDeal)
- Agent within same org requests unit number from unitManager
- Grant has 30-day expiry, status: Active/Expired/Revoked

</Accordion>
</AccordionGroup>

### SharedDeal accept flow

<Warning>
This is a critical flow that creates Deal entities in both organizations and reveals the unit number. It uses `TenantContext.executeInOrg()` to operate in each organization's context.
</Warning>

```typescript
// 1. Update shared deal
sharedDeal.status = SharedDealStatus.ACCEPTED;
sharedDeal.agreedPrice = latestProposedPrice;
sharedDeal.isUnitNumberRevealed = true;
sharedDeal.unitNumberRevealedAt = new Date();

// 2. Auto-create Deal in LISTING org
await this.tenantContext.executeInOrg(listingOrgId, async (em) => {
  const deal = em.create(Deal, { ... });
  const stakeholder = em.create(EntityStakeholder, { ... });
});

// 3. Auto-create Deal in BUYER org
await this.tenantContext.executeInOrg(buyerOrgId, async (em) => {
  const deal = em.create(Deal, { ... });
  const stakeholder = em.create(EntityStakeholder, { ... });
});

// 4. Link both deals to SharedDeal
sharedDeal.listingDeal = listingDeal;
sharedDeal.buyerDeal = buyerDeal;
```

## Property interest module

**Path:** `src/modules/real-estate/property-interest/`

Manages lead and deal property interests and the matching engine. Links to reference DB projects/units via **plain integer IDs** (`refProjectId`, `refUnitId`) and to CRM DB entities (`SecondaryUnit`, `SharedListing`) via ManyToOne relations.

### Entities

<AccordionGroup>
<Accordion title="LeadPropertyInterest">

- UUID PK, org-scoped
- ManyToOne: Lead (required)
- **Reference DB links (plain ints):** `refProjectId?: number`, `refUnitId?: number`
- **CRM DB links (ManyToOne):** secondaryUnit, sharedListing
- Matching criteria: propertyType, layoutType, bedrooms, budgetMin/Max, sizeMin/Max
- Preferred locations: preferredAreas[], preferredCommunities[]
- Metadata: priority, isPrimary, notes

</Accordion>
<Accordion title="DealPropertyInterest">

- UUID PK, org-scoped, **OneToOne with Deal** (owner: true)
- **Reference DB links (plain ints):** `refProjectId?: number`, `refUnitId?: number`
- **CRM DB links (ManyToOne):** secondaryUnit, sharedListing
- originatingInterest: link to LeadPropertyInterest (for lineage)
- interestType: Buy, Sell, Rent, Landlord
- status: Active, Confirmed, Cancelled, Completed
- budgetMin/Max, agreedPrice, confirmedAt

</Accordion>
<Accordion title="PropertyMatch">

- UUID PK, org-scoped
- ManyToOne: LeadPropertyInterest
- **Matched entity IDs:**
  - `matchedRefProjectId?: number` (reference DB integer)
  - `matchedRefUnitId?: number` (reference DB integer)
  - `matchedSecondaryUnitId?: string` (CRM DB UUID)
  - `matchedSharedListingId?: string` (CRM DB UUID)
- matchScore (0-100), matchBreakdown (JSON)
- status: MatchStatus (NEW, VIEWED, SHORTLISTED, DISMISSED, CONTACTED)

</Accordion>
</AccordionGroup>

### Property reference types

| Reference | Type | Notes |
| --- | --- | --- |
| refProjectId | int | Plain int ID to reference DB projects table |
| refUnitId | int | Plain int ID to reference DB units table |
| secondaryUnit | SecondaryUnit (ManyToOne) | Org-scoped, CRM DB entity |
| sharedListing | SharedListing (ManyToOne) | Cross-org marketplace listing |

### Matching engine

The `PropertyMatchService` generates matches when a LeadPropertyInterest is created/updated. It uses `ReferenceDataService` for off-plan data (no direct entity access):

1. **Query sources in parallel:**
   - Internal inventory (SecondaryUnit) — org-scoped via TenantContext
   - Off-plan directory (RefProject/RefUnit) — via `ReferenceDataService`
   - Shared marketplace (SharedListing) — global via bypass

2. **Score each candidate (0-100):**
   - Budget match: 0-30 points (exact=30, within 20%=20, within 50%=10)
   - Bedroom match: 0-25 points (exact=25, +/-1=15, +/-2=5)
   - Area match: 0-20 points (preferred area=20, preferred community=10)
   - Size match: 0-15 points (within range=15, within 20%=10)
   - Type match: 0-10 points (exact=10)

3. **Persist matches** with score > 20, sorted by score descending

<Tip>
The `PropertyMatchService` avoids nested TenantContext calls by running off-plan queries (via `ReferenceDataService`) and shared listing queries (global, `executeWithBypass`) **first**, then entering `executeInOrg` for org-scoped secondary unit queries + match persistence.
</Tip>

### Org-scoped vs global

- **Org-scoped** (standard RLS): Entity has `@ManyToOne(() => Organization) organization!: Organization;`
- **Global** (no RLS): Entity has NO organization FK. Used for DeveloperUser, DeveloperProjectLink, etc.
- **Cross-org** (custom RLS): Entity references multiple organizations. Needs custom SQL policies.
- **Reference DB** (no RLS): Entities live in separate database, no org context needed.
