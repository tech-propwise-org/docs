# Audit Module Specification v11

<Info>
**Last Updated:** 2024-01-15  
**Status:** Active
</Info>

The audit module provides comprehensive activity tracking and timeline functionality for the PropWise CRM system. It captures all data changes across entities through PostgreSQL triggers and presents them as both raw audit logs and human-readable timeline events.

## Overview

The audit module supports both tenant-scoped and system-wide audit queries, ensuring complete traceability of all system changes while maintaining proper access controls and data isolation.

### Key Features

- **Database-driven auditing** with PostgreSQL triggers
- **Read-only entity design** for data integrity
- **Timeline abstraction** for user-friendly event display
- **Dual access patterns** for tenant and system admin queries

## Architecture

### Design Decisions

<AccordionGroup>
<Accordion title="Database-Driven Auditing">
Uses PostgreSQL triggers (`audit_trigger_func()`) to capture changes at the database level, ensuring consistency and completeness across all data modifications.
</Accordion>

<Accordion title="Read-Only Entity">
The `AuditLog` entity is strictly read-only from application code - all writes happen via database triggers to prevent tampering.
</Accordion>

<Accordion title="Timeline Abstraction">
Provides a higher-level timeline view that translates raw audit data into user-friendly messages with semantic event types.
</Accordion>

<Accordion title="Dual Access Patterns">
Supports both tenant-scoped queries (for regular users) and system-wide queries (for system administrators) with appropriate access controls.
</Accordion>
</AccordionGroup>

## Entity Descriptions

### AuditLog

<Note>
The AuditLog entity is read-only and populated exclusively through database triggers.
</Note>

```typescript
interface AuditLog {
  id: string;                    // UUID primary key
  tableName: string;             // Name of the table that changed
  action: AuditAction;           // INSERT, UPDATE, DELETE
  rowData: Record<string, any>;  // Full row data after change
  changedFields: Record<string, any>; // Only changed fields (UPDATE only)
  recordId: string;              // ID of the changed record
  tenantId: string;              // Tenant context
  userId?: string;               // User who made the change
  createdAt: Date;               // When the change occurred
}
```

### AuditAction

```typescript
enum AuditAction {
```

## API Endpoints

### Tenant-Scoped Endpoints

#### GET /audit/logs

<Warning>
This endpoint is automatically scoped to the current tenant. Users cannot access audit logs from other tenants.
</Warning>

Retrieve paginated audit logs for the current tenant.

**Query Parameters:**

```typescript
interface AuditLogQueryDto {
  page?: number;        // Page number (default: 1)
  limit?: number;       // Items per page (1-100, default: 20)
  tableName?: string;   // Filter by table name
  recordId?: string;    // Filter by record ID
  userId?: string;      // Filter by user ID
  action?: AuditAction; // Filter by action type
  startDate?: string;   // ISO date string
  endDate?: string;     // ISO date string
}
```

**Response:**

```typescript
interface PaginatedResponse<AuditLogDto> {
  data: AuditLogDto[];
  pagination: {
    page: number;
    limit: number;
    total: number;
    totalPages: number;
  };
}
```

#### GET /audit/timeline/:entityType/:entityId

Get resolved timeline events for a specific entity with human-readable messages.

**Parameters:**
- `entityType`: The type of entity (e.g., 'lead', 'deal')
- `entityId`: UUID of the specific entity

**Response:**

```typescript
interface ResolvedTimeline {
  eventType: TimelineEventType;
  message: string;
  createdAt: Date;
  user?: AuditUserInfo;
}
```

### System Admin Endpoints

#### GET /system-admin/audit/logs

<Check>
Requires SystemAdmin role for cross-tenant audit access.
</Check>

System-wide audit log access with additional tenant filtering capabilities.

**Query Parameters:**

```typescript
interface SystemAdminAuditQueryDto extends AuditLogQueryDto {
  tenantId?: string;    // Filter by specific tenant
}
```

## Business Rules and Invariants

### Data Integrity

<Steps>
<Step title="Immutable Audit Logs">
Audit logs are immutable once created by the database trigger to ensure data integrity.
</Step>

<Step title="Automatic Data Filtering">
Sensitive fields (passwords, refresh tokens) are automatically stripped by the trigger before persistence.
</Step>

<Step title="Transaction Consistency">
All audit operations must occur within the same transaction as the data change to maintain consistency.
</Step>
</Steps>

### Access Control

- **Regular users** can only access audit logs for their own tenant
- **System administrators** can access audit logs across all tenants
- **Timeline resolution** includes user information only if accessible to the requester

### Temporal Constraints

<Note>
All timestamps are stored in UTC for consistency across time zones.
</Note>

- Date range queries validate that `endDate >= startDate`
- Date filters use inclusive ranges
- Pagination is limited to 100 items per page maximum

## Patterns and Conventions

### Query Patterns

<Tip>
The module uses standardized pagination and filtering patterns consistent with other PropWise CRM modules.
</Tip>

- Implements standard pagination using `buildPaginatedResponse`
- Supports flexible filtering with optional query parameters
- Uses MikroORM's FilterQuery for type-safe database queries

### Error Handling

- Returns 404 for non-existent entities in timeline queries
- Validates UUID format for ID parameters
- Enforces pagination limits with clear error messages

### Caching Considerations

<Warning>
No caching is implemented due to the real-time nature of audit requirements and write-heavy usage patterns.
</Warning>

- Audit data is write-heavy, read-light
- Timeline resolution performed on-demand
- Consider future caching for frequently accessed timeline data

### Security

- **No direct entity manipulation** - all writes via database triggers
- **Multi-layer tenant isolation** enforced at service and database levels
- **Sensitive data filtering** at database level before persistence
