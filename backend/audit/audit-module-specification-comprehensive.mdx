The audit module provides comprehensive activity tracking and timeline functionality for the PropWise CRM system. It captures all data changes across entities through PostgreSQL triggers and presents them as both raw audit logs and human-readable timeline events.

## Architecture Overview

<Info>
The audit module uses a database-driven approach to ensure complete and consistent tracking of all data changes across the system.
</Info>

### Design Principles

<CardGroup cols={2}>
  <Card title="Database-Driven Auditing" icon="database">
    Uses PostgreSQL triggers to capture changes at the database level, ensuring consistency and completeness
  </Card>
  <Card title="Read-Only Entity" icon="lock">
    The AuditLog entity is strictly read-only from application code - all writes happen via database triggers
  </Card>
  <Card title="Timeline Abstraction" icon="timeline">
    Provides a higher-level timeline view that translates raw audit data into user-friendly messages
  </Card>
  <Card title="Dual Access Patterns" icon="users">
    Supports both tenant-scoped queries and system-wide queries for administrators
  </Card>
</CardGroup>

### Key Components

- **AuditLog Entity**: Maps to the `audit_log` table (read-only)
- **AuditService**: Core business logic for querying and formatting audit data
- **AuditTimelineResolver**: Translates audit logs into human-readable timeline events
- **Dual Controllers**: Separate endpoints for tenant and system admin access

## Entity Specifications

### AuditLog Entity

<CodeGroup>
```typescript Interface
interface AuditLog {
  id: string;                    // UUID primary key
  tableName: string;             // Name of the table that changed
  action: AuditAction;           // INSERT, UPDATE, DELETE
  rowData: Record<string, any>;  // Full row data after change
  changedFields: Record<string, any>; // Only changed fields (UPDATE only)
  recordId: string;              // ID of the changed record
  tenantId: string;              // Tenant context
  userId?: string;               // User who made the change
  createdAt: Date;               // When the change occurred
}
```

```typescript Enum
enum AuditAction {
  INSERT = 'INSERT',
  UPDATE = 'UPDATE', 
  DELETE = 'DELETE'
}
```
</CodeGroup>

## API Endpoints

### Tenant-Scoped Endpoints

#### Retrieve Audit Logs

<CodeGroup>
```http Request
GET /audit/logs
```

```typescript Query Parameters
interface AuditLogQueryDto {
  page?: number;        // Page number (default: 1)
  limit?: number;       // Items per page (1-100, default: 20)
  tableName?: string;   // Filter by table name
  recordId?: string;    // Filter by record ID
  userId?: string;      // Filter by user ID
  action?: AuditAction; // Filter by action type
  startDate?: string;   // ISO date string
  endDate?: string;     // ISO date string
}
```

```typescript Response
interface PaginatedResponse<AuditLogDto> {
  data: AuditLogDto[];
  pagination: {
    page: number;
    limit: number;
    total: number;
    totalPages: number;
  };
}
```
</CodeGroup>

#### Get Entity Timeline

<CodeGroup>
```http Request
GET /audit/timeline/{entityType}/{entityId}
```

```typescript Response
interface ResolvedTimeline {
  eventType: TimelineEventType;
  message: string;
  createdAt: Date;
  user?: AuditUserInfo;
}
```
</CodeGroup>

**Parameters:**
- `entityType`: The type of entity (e.g., 'lead', 'deal')
- `entityId`: UUID of the specific entity

## Business Rules

### Data Integrity Rules

<Steps>
  <Step title="Immutable Audit Logs">
    Audit logs are immutable once created by the database trigger and cannot be modified by application code.
  </Step>
  
  <Step title="Sensitive Data Protection">
    Sensitive fields (passwords, refresh tokens) are automatically stripped by the database trigger before persistence.
  </Step>
  
  <Step title="Transactional Consistency">
    All audit operations must occur within the same transaction as the data change to ensure consistency.
  </Step>
</Steps>

### Access Control Rules

<Tabs>
  <Tab title="Regular Users">
    - Can only access audit logs for their own tenant
    - Timeline resolution includes user information only if accessible to the requester
    - Protected by `@AuthGuard()` decorator
  </Tab>
  
  <Tab title="System Administrators">
    - Can access audit logs across all tenants
    - Requires `SystemAdmin` role
    - Can filter by specific tenant or view system-wide data
  </Tab>
</Tabs>

### Temporal Constraints

<Check>
- Date range queries validate that `endDate >= startDate`
- Date filters use inclusive ranges
- All timestamps are stored in UTC
</Check>

## Integration Patterns

### Authentication Integration

<CodeGroup>
```typescript Guard Usage
@AuthGuard()
@Get('logs')
async getAuditLogs(@GetCurrentUser() user: User) {
  // Automatically scoped to user's tenant
}
```

```typescript System Admin
@SystemAdmin()
@Get('system-admin/audit/logs')
async getSystemAuditLogs() {
  // Access across all tenants
}
```
</CodeGroup>

### Tenant Context Integration

<Info>
All queries are automatically scoped to the current tenant context, except for system administrator access. Tenant isolation is enforced at the service layer.
</Info>

### User Module Integration

The audit module integrates with the user module to:
- Resolve user information for timeline display
- Link audit entries to user accounts via `userId` field
- Provide user context in audit trail

### Entity Module Integration

<Note>
The audit module monitors changes across all major entities (leads, deals, contacts, etc.) and uses entity type resolution for timeline formatting.
</Note>

## Implementation Patterns

### Query Patterns

<Steps>
  <Step title="Pagination">
    Implements standard pagination using `buildPaginatedResponse` utility
  </Step>
  
  <Step title="Flexible Filtering">
    Supports optional query parameters for fine-grained filtering
  </Step>
  
  <Step title="Type Safety">
    Uses MikroORM's FilterQuery for type-safe database queries
  </Step>
</Steps>

### Error Handling

<AccordionGroup>
  <Accordion title="Timeline Queries">
    Returns 404 for non-existent entities in timeline queries
  </Accordion>
  
  <Accordion title="Validation">
    - Validates UUID format for ID parameters
    - Enforces pagination limits (max 100 items per page)
    - Validates date range parameters
  </Accordion>
  
  <Accordion title="Security">
    Enforces tenant isolation and access control at multiple layers
  </Accordion>
</AccordionGroup>

### Performance Considerations

<Tip>
The audit module is designed for write-heavy, read-light workloads. No caching is implemented due to real-time requirements, and timeline resolution is performed on-demand.
</Tip>

### Security Features

<CardGroup cols={2}>
  <Card title="Trigger-Only Writes" icon="shield-check">

  </Card>
  <Card title="Multi-Layer Isolation" icon="layers">

  </Card>
  <Card title="Sensitive Data Filtering" icon="eye-slash">

  </Card>
  <Card title="Role-Based Access" icon="user-shield">

  </Card>
</CardGroup>

## Usage Examples

### Basic Audit Log Query

<CodeGroup>
```typescript Service Usage
const auditLogs = await auditService.getAuditLogs(tenantId, {
  tableName: 'leads',
  action: AuditAction.UPDATE,
  page: 1,
  limit: 20
});
```

```typescript Timeline Resolution
const timeline = await auditService.getEntityTimeline(
  tenantId,
  'lead',
  leadId
);
```
</CodeGroup>

### Filtering Examples

<CodeGroup>
```typescript Date Range Filter
const logs = await auditService.getAuditLogs(tenantId, {
  startDate: '2024-01-01T00:00:00Z',
  endDate: '2024-01-31T23:59:59Z',
  userId: 'user-uuid'
});
```

```typescript System Admin Query
const systemLogs = await auditService.getSystemAuditLogs({
  tenantId: 'specific-tenant-id',
  tableName: 'deals',
  action: AuditAction.INSERT
});
```
</CodeGroup>
