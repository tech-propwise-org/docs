The audit module provides comprehensive activity tracking and timeline functionality for the PropWise CRM system. It captures all data changes across entities through PostgreSQL triggers and presents them as both raw audit logs and human-readable timeline events.

<Info>
**Last Updated:** 2024-01-15  
**Status:** Active
</Info>

## Architecture

### Design decisions

The audit module follows several key architectural principles to ensure reliability and performance:

<CardGroup cols={2}>
  <Card title="Database-driven auditing" icon="database">
    Uses PostgreSQL triggers to capture changes at the database level, ensuring consistency and completeness
  </Card>
  <Card title="Read-only entity" icon="lock">
    The AuditLog entity is strictly read-only from application code - all writes happen via database triggers
  </Card>
  <Card title="Timeline abstraction" icon="timeline">
    Provides a higher-level timeline view that translates raw audit data into user-friendly messages
  </Card>
  <Card title="Dual access patterns" icon="users">
    Supports both tenant-scoped queries and system-wide queries for administrators
  </Card>
</CardGroup>

### Key components

- **AuditLog Entity**: Maps to the `audit_log` table (read-only)
- **AuditService**: Core business logic for querying and formatting audit data
- **AuditTimelineResolver**: Translates audit logs into human-readable timeline events
- **Dual Controllers**: Separate endpoints for tenant and system admin access

## Entity descriptions

### AuditLog

The core audit log entity that captures all data changes across the system:

```typescript
interface AuditLog {
  id: string;                    // UUID primary key
  tableName: string;             // Name of the table that changed
  action: AuditAction;           // INSERT, UPDATE, DELETE
  rowData: Record<string, any>;  // Full row data after change
  changedFields: Record<string, any>; // Only changed fields (UPDATE only)
  recordId: string;              // ID of the changed record
  tenantId: string;              // Tenant context
  userId?: string;               // User who made the change
  createdAt: Date;               // When the change occurred
}
```

<Note>
The `AuditLog` entity is read-only from application code. All writes are performed automatically by PostgreSQL triggers.
</Note>

### AuditAction

Defines the type of database operation that triggered the audit log:

```typescript
enum AuditAction {
  INSERT = 'INSERT',
  UPDATE = 'UPDATE', 
  DELETE = 'DELETE'
}
```

## API endpoints

<Tabs>
  <Tab title="Tenant-scoped">
    ### GET /audit/logs
    
    Retrieve paginated audit logs for the current tenant.
    
    **Query parameters:**
    
    ```typescript
    interface AuditLogQueryDto {
      page?: number;        // Page number (default: 1)
      limit?: number;       // Items per page (1-100, default: 20)
      tableName?: string;   // Filter by table name
      recordId?: string;    // Filter by record ID
      userId?: string;      // Filter by user ID
      action?: AuditAction; // Filter by action type
      startDate?: string;   // ISO date string
      endDate?: string;     // ISO date string
    }
    ```
    
    **Response:**
    
    ```typescript
    interface PaginatedResponse<AuditLogDto> {
      data: AuditLogDto[];
      pagination: {
        page: number;
        limit: number;
        total: number;
        totalPages: number;
      };
    }
    ```
    
    ### GET /audit/timeline/:entityType/:entityId
    
    Get resolved timeline events for a specific entity.
    
    **Parameters:**
    - `entityType`: The type of entity (e.g., 'lead', 'deal')
    - `entityId`: UUID of the specific entity
    
    **Response:**
    
    ```typescript
    interface ResolvedTimeline {
      eventType: TimelineEventType;
      message: string;
      createdAt: Date;
      user?: AuditUserInfo;
    }
    ```
  </Tab>
  
  <Tab title="System admin">
    ### GET /system-admin/audit/logs
    
    System-wide audit log access (requires SystemAdmin role).
    
    **Query parameters:**
    
    ```typescript
    interface SystemAdminAuditQueryDto extends AuditLogQueryDto {
      tenantId?: string;    // Filter by specific tenant
    }
    ```
    
    <Warning>
    This endpoint requires SystemAdmin role and provides access to audit logs across all tenants.
    </Warning>
  </Tab>
</Tabs>

## Business rules and invariants

### Data integrity

<Steps>
  <Step title="Immutable audit logs">
    Audit logs are immutable once created by the database trigger
  </Step>
  <Step title="Sensitive data filtering">
    Sensitive fields (passwords, refresh tokens) are automatically stripped by the trigger
  </Step>
  <Step title="Transaction consistency">
    All audit operations must occur within the same transaction as the data change
  </Step>
</Steps>

### Access control

- **Regular users**: Can only access audit logs for their own tenant
- **System administrators**: Can access audit logs across all tenants
- **Timeline resolution**: Includes user information only if accessible to the requester

### Temporal constraints

<Check>Date range queries validate that `endDate >= startDate`</Check>
<Check>Date filters use inclusive ranges</Check>
<Check>Timestamps are stored in UTC</Check>

## Integration points

### With authentication module

The audit module integrates closely with the authentication system:

- Uses `@AuthGuard()` for endpoint protection
- Leverages `@GetCurrentUser()` for tenant context
- Integrates with `@SystemAdmin()` decorator for admin access

### With tenant context

- All queries are automatically scoped to the current tenant (except system admin)
- Tenant isolation is enforced at the service layer

### With user module

- Resolves user information for timeline display
- Links audit entries to user accounts via `userId` field

### With entity modules

- Monitors changes across all major entities (leads, deals, contacts, etc.)
- Uses entity type resolution for timeline formatting

### Query patterns

<CodeGroup>
```typescript Pagination
// Standard pagination implementation
const response = buildPaginatedResponse(auditLogs, query.page, query.limit, total);
```

```typescript Filtering
// Flexible filtering with optional parameters
const filterQuery: FilterQuery<AuditLog> = {
  tenantId: currentUser.tenantId,
  ...(query.tableName && { tableName: query.tableName }),
  ...(query.recordId && { recordId: query.recordId }),
  ...(query.userId && { userId: query.userId }),
  ...(query.action && { action: query.action }),
};
```

```typescript Date Ranges
// Date range filtering
if (query.startDate || query.endDate) {
  filterQuery.createdAt = {};
  if (query.startDate) {
    filterQuery.createdAt.$gte = new Date(query.startDate);
  }
  if (query.endDate) {
    filterQuery.createdAt.$lte = new Date(query.endDate);
  }
}
```
</CodeGroup>

### Error handling

- Returns 404 for non-existent entities in timeline queries
- Validates UUID format for ID parameters
- Enforces pagination limits (max 100 items per page)

### Caching considerations

<Tip>
Audit data is write-heavy, read-light with real-time requirements. No caching is implemented to ensure data freshness, and timeline resolution is performed on-demand.
</Tip>
