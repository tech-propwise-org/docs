## Overview

The audit module provides comprehensive activity tracking and timeline functionality for the PropWise CRM system. It captures all data changes across entities through PostgreSQL triggers and presents them as both raw audit logs and human-readable timeline events. The module supports both tenant-scoped and system-wide audit queries.

<Note>
The audit module uses database-level triggers to ensure complete and consistent tracking of all data changes, making it impossible to bypass auditing from application code.
</Note>

## Architecture

### Design decisions

The audit module follows several key architectural principles:

<AccordionGroup>
<Accordion title="Database-driven auditing">
Uses PostgreSQL triggers (`audit_trigger_func()`) to capture changes at the database level, ensuring consistency and completeness across all data modifications.
</Accordion>

<Accordion title="Read-only entity pattern">
The `AuditLog` entity is strictly read-only from application code - all writes happen via database triggers, preventing tampering or inconsistent audit trails.
</Accordion>

<Accordion title="Timeline abstraction">
Provides a higher-level timeline view that translates raw audit data into user-friendly messages for better user experience.
</Accordion>

<Accordion title="Dual access patterns">
Supports both tenant-scoped queries (for regular users) and system-wide queries (for system administrators).
</Accordion>
</AccordionGroup>

### Key components

The audit module consists of several core components working together:

- **AuditLog Entity**: Maps to the `audit_log` table (read-only)
- **AuditService**: Core business logic for querying and formatting audit data
- **AuditTimelineResolver**: Translates audit logs into human-readable timeline events
- **Dual Controllers**: Separate endpoints for tenant and system admin access

## Entity specifications

### AuditLog entity

The core audit entity that captures all data changes:

```typescript
interface AuditLog {
  id: string;                    // UUID primary key
  tableName: string;             // Name of the table that changed
  action: AuditAction;           // INSERT, UPDATE, DELETE
  rowData: Record<string, any>;  // Full row data after change
  changedFields: Record<string, any>; // Only changed fields (UPDATE only)
  recordId: string;              // ID of the changed record
  tenantId: string;              // Tenant context
  userId?: string;               // User who made the change
  createdAt: Date;               // When the change occurred
}
```

<Info>
The `changedFields` property is only populated for UPDATE operations, containing only the fields that were modified in that specific change.
</Info>

### AuditAction enumeration

Defines the types of database operations tracked:

```typescript
enum AuditAction {
  INSERT = 'INSERT',
  UPDATE = 'UPDATE', 
  DELETE = 'DELETE'
}
```

## API endpoints

### Tenant-scoped endpoints

#### GET /audit/logs

Retrieve paginated audit logs for the current tenant.

**Query parameters:**

```typescript
interface AuditLogQueryDto {
  page?: number;        // Page number (default: 1)
  limit?: number;       // Items per page (1-100, default: 20)
  tableName?: string;   // Filter by table name
  recordId?: string;    // Filter by record ID
  userId?: string;      // Filter by user ID
  action?: AuditAction; // Filter by action type
  startDate?: string;   // ISO date string
  endDate?: string;     // ISO date string
}
```

**Response format:**

```typescript
interface PaginatedResponse<AuditLogDto> {
  data: AuditLogDto[];
  pagination: {
    page: number;
    limit: number;
    total: number;
    totalPages: number;
  };
}
```

#### GET /audit/timeline/:entityType/:entityId

Get resolved timeline events for a specific entity.

**Parameters:**
- `entityType`: The type of entity (e.g., 'lead', 'deal')
- `entityId`: UUID of the specific entity

**Response format:**

```typescript
interface ResolvedTimeline {
  eventType: TimelineEventType;
  message: string;
  createdAt: Date;
  user?: AuditUserInfo;
}
```

<Tip>
Timeline endpoints automatically resolve raw audit data into human-readable messages, making it easier to display activity feeds in the UI.
</Tip>

### System admin endpoints

#### GET /system-admin/audit/logs

System-wide audit log access (requires SystemAdmin role).

**Query parameters:**

```typescript
interface SystemAdminAuditQueryDto extends AuditLogQueryDto {
  tenantId?: string;    // Filter by specific tenant
}
```

<Warning>
System admin endpoints bypass tenant isolation and can access audit logs across all tenants. Use with appropriate caution and access controls.
</Warning>

## Business rules and constraints

### Data integrity rules

<Steps>
<Step title="Immutability guarantee">
Audit logs are immutable once created by the database trigger and cannot be modified or deleted through application code.
</Step>

<Step title="Sensitive data protection">
Sensitive fields (passwords, refresh tokens) are automatically stripped by the database trigger before audit log creation.
</Step>

<Step title="Transactional consistency">
All audit operations must occur within the same transaction as the data change to ensure consistency.
</Step>
</Steps>

### Access control policies

- **Regular users**: Can only access audit logs for their own tenant
- **System administrators**: Can access audit logs across all tenants
- **Timeline resolution**: Includes user information only if accessible to the requester

### Temporal constraints

<Note>
All timestamp operations follow these rules:
- Date range queries validate that `endDate >= startDate`
- Date filters use inclusive ranges
- Timestamps are stored and processed in UTC
</Note>

## Implementation patterns

### Query patterns

<AccordionGroup>
<Accordion title="Pagination implementation">
Implements standard pagination using `buildPaginatedResponse` utility function for consistent response formatting across all endpoints.
</Accordion>

<Accordion title="Flexible filtering">
Supports optional query parameters that can be combined for precise audit log filtering without requiring complex query builders.
</Accordion>

<Accordion title="Type-safe queries">
Uses MikroORM's FilterQuery type system for compile-time query validation and better IDE support.
</Accordion>
</AccordionGroup>

### Error handling strategies

- **404 responses**: Returns appropriate errors for non-existent entities in timeline queries
- **UUID validation**: Validates UUID format for ID parameters before processing
- **Pagination limits**: Enforces maximum limit of 100 items per page to prevent performance issues

### Security considerations

<Check>
Security measures implemented:
- No direct entity manipulation - all writes via database triggers
- Tenant isolation enforced at multiple architectural layers
- Sensitive data filtered at database level before persistence
- Role-based access control for system-wide queries
</Check>

### Performance characteristics

- **Write-heavy pattern**: Audit data is primarily written during normal operations
- **Read-light pattern**: Audit queries are less frequent than data modifications
- **No caching**: Real-time requirements prevent caching of audit data
- **On-demand resolution**: Timeline formatting performed only when requested

<Warning>
Due to the real-time nature of audit requirements, caching strategies should be carefully evaluated to ensure audit data freshness and accuracy.
</Warning>
