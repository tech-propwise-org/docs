## Overview

The audit module provides comprehensive activity tracking and timeline functionality for the PropWise CRM system. It captures all data changes across entities through PostgreSQL triggers and presents them as both raw audit logs and human-readable timeline events. The module supports both tenant-scoped and system-wide audit queries.

<Info>
The audit system operates at the database level using PostgreSQL triggers, ensuring complete and consistent change tracking across all entities.
</Info>

## Architecture

### Design decisions

The audit module follows several key architectural patterns:

<CardGroup cols={2}>
  <Card title="Database-driven auditing" icon="database">
    Uses PostgreSQL triggers (`audit_trigger_func()`) to capture changes at the database level, ensuring consistency and completeness
  </Card>
  <Card title="Read-only entity" icon="lock">
    The `AuditLog` entity is strictly read-only from application code - all writes happen via database triggers
  </Card>
  <Card title="Timeline abstraction" icon="timeline">
    Provides a higher-level timeline view that translates raw audit data into user-friendly messages
  </Card>
  <Card title="Dual access patterns" icon="users">
    Supports both tenant-scoped queries (for regular users) and system-wide queries (for system administrators)
  </Card>
</CardGroup>

### Key components

- **AuditLog Entity**: Maps to the `audit_log` table (read-only)
- **AuditService**: Core business logic for querying and formatting audit data
- **AuditTimelineResolver**: Translates audit logs into human-readable timeline events
- **Dual Controllers**: Separate endpoints for tenant and system admin access

## Entity descriptions

### AuditLog

<CodeGroup>
```typescript AuditLog Entity
interface AuditLog {
  id: string;                    // UUID primary key
  tableName: string;             // Name of the table that changed
  action: AuditAction;           // INSERT, UPDATE, DELETE
  rowData: Record<string, any>;  // Full row data after change
  changedFields: Record<string, any>; // Only changed fields (UPDATE only)
  recordId: string;              // ID of the changed record
  tenantId: string;              // Tenant context
  userId?: string;               // User who made the change
  createdAt: Date;               // When the change occurred
}
```
</CodeGroup>

<Note>
The `changedFields` property only contains data for UPDATE operations, providing a clear view of what specifically changed in the record.
</Note>

## API endpoints

### Tenant-scoped endpoints

#### GET /audit/logs

Retrieve paginated audit logs for the current tenant.

<CodeGroup>
```typescript Query Parameters
interface AuditLogQueryDto {
  page?: number;        // Page number (default: 1)
  limit?: number;       // Items per page (1-100, default: 20)
  tableName?: string;   // Filter by table name
  recordId?: string;    // Filter by record ID
  userId?: string;      // Filter by user ID
  action?: AuditAction; // Filter by action type
  startDate?: string;   // ISO date string
  endDate?: string;     // ISO date string
}
```

```typescript Response
interface PaginatedResponse<AuditLogDto> {
  data: AuditLogDto[];
  pagination: {
    page: number;
    limit: number;
    total: number;
    totalPages: number;
  };
}
```
</CodeGroup>

<Warning>
The maximum limit per page is 100 items. Requests exceeding this limit will be rejected.
</Warning>

#### GET /audit/timeline/:entityType/:entityId

Get resolved timeline events for a specific entity.

<Steps>
  <Step title="Specify entity type">
    Provide the entity type (e.g., 'lead', 'deal') in the URL path
  </Step>
  <Step title="Provide entity ID">
    Include the UUID of the specific entity you want timeline data for
  </Step>
  <Step title="Receive timeline events">
    Get back human-readable timeline events with user context
  </Step>
</Steps>

<CodeGroup>
```typescript Response
interface ResolvedTimeline {
  eventType: TimelineEventType;
  message: string;
  createdAt: Date;
  user?: AuditUserInfo;
}
```
</CodeGroup>

## Business rules and invariants

### Data integrity

<CardGroup cols={1}>
  <Card title="Immutable audit logs" icon="shield">
    Audit logs are immutable once created by the database trigger
  </Card>
  <Card title="Automatic data filtering" icon="filter">
    Sensitive fields (passwords, refresh tokens) are automatically stripped by the trigger
  </Card>
  <Card title="Transactional consistency" icon="arrows-rotate">
    All audit operations must occur within the same transaction as the data change
  </Card>
</CardGroup>

### Access control

<Tabs>
  <Tab title="Regular users">
    - Can only access audit logs for their own tenant
    - Timeline resolution includes user information only if accessible
    - All queries automatically scoped to tenant context
  </Tab>
  
  <Tab title="System administrators">
    - Can access audit logs across all tenants
    - Have additional query parameters for cross-tenant filtering
    - Require special SystemAdmin role authorization
  </Tab>
</Tabs>

### Temporal constraints

<Note>
All temporal operations follow these rules:
- Date range queries validate that `endDate >= startDate`
- Date filters use inclusive ranges
- Timestamps are stored in UTC
</Note>

## Integration points

### With authentication module

<Steps>
  <Step title="Endpoint protection">
    Uses `@AuthGuard()` for securing all audit endpoints
  </Step>
  <Step title="User context">
    Leverages `@GetCurrentUser()` for tenant context extraction
  </Step>
  <Step title="Admin authorization">
    Integrates with `@SystemAdmin()` decorator for admin-only access
  </Step>
</Steps>

### With tenant context

The audit module maintains strict tenant isolation:

- All queries are automatically scoped to the current tenant (except system admin)
- Tenant isolation is enforced at the service layer
- Cross-tenant data access is prevented at multiple levels

The audit system monitors changes across all major entities:

- Automatically captures changes via database triggers

## Patterns and conventions

### Query patterns

<Tip>
The audit module follows standard PropWise patterns:
- Implements pagination using `buildPaginatedResponse`
- Supports flexible filtering with optional query parameters
- Uses MikroORM's FilterQuery for type-safe database queries
</Tip>

### Error handling

<AccordionGroup>
  <Accordion title="Common error responses">
    - **404**: Non-existent entities in timeline queries
    - **400**: Invalid UUID format for ID parameters
    - **400**: Pagination limits exceeded (max 100 items per page)
  </Accordion>
  
  <Accordion title="Validation rules">
    - All entity IDs must be valid UUIDs
    - Date ranges must be logically consistent
    - Pagination parameters must be within allowed bounds
  </Accordion>
</AccordionGroup>

### Caching considerations

<Warning>
The audit module does not implement caching due to:
- Write-heavy, read-light usage patterns
- Real-time requirements for audit data
- Timeline resolution performed on-demand
</Warning>

### Security

<CardGroup cols={1}>
  <Card title="Read-only application access" icon="eye">
    No direct entity manipulation - all writes via database triggers
  </Card>
  <Card title="Multi-layer tenant isolation" icon="layers">
    Tenant isolation enforced at service, controller, and database levels
  </Card>
  <Card title="Automatic data sanitization" icon="broom">
    Sensitive data filtered at database level before persistence
  </Card>
</CardGroup>

<Info>
The audit system is designed with security-first principles, ensuring that sensitive information is never stored in audit logs and that tenant data remains strictly isolated.
