# Audit Module Specification

The audit module provides comprehensive activity tracking and timeline functionality for the PropWise CRM system. It captures all data changes across entities through PostgreSQL triggers and presents them as both raw audit logs and human-readable timeline events.

<Note>
The audit module supports both tenant-scoped and system-wide audit queries, ensuring proper data isolation while enabling administrative oversight.
</Note>

## Architecture Overview

### Design Decisions

The audit module follows several key architectural principles:

<Steps>
<Step title="Database-driven auditing">
Uses PostgreSQL triggers (`audit_trigger_func()`) to capture changes at the database level, ensuring consistency and completeness
</Step>
<Step title="Read-only application layer">
The `AuditLog` entity is strictly read-only from application code - all writes happen via database triggers
</Step>
<Step title="Timeline abstraction">
Provides a higher-level timeline view that translates raw audit data into user-friendly messages
</Step>
<Step title="Dual access patterns">
Supports both tenant-scoped queries (for regular users) and system-wide queries (for system administrators)
</Step>
</Steps>

## Entity Descriptions

### AuditLog Entity

The core audit log entity captures all database changes:

```typescript
interface AuditLog {
  id: string;                    // UUID primary key
  tableName: string;             // Name of the table that changed
  action: AuditAction;           // INSERT, UPDATE, DELETE
  rowData: Record<string, any>;  // Full row data after change
  changedFields: Record<string, any>; // Only changed fields (UPDATE only)
  recordId: string;              // ID of the changed record
  tenantId: string;              // Tenant context
  userId?: string;               // User who made the change
  createdAt: Date;               // When the change occurred
}
```

## API Endpoints

### Tenant-Scoped Endpoints

#### GET /audit/logs

Retrieve paginated audit logs for the current tenant.

<CodeGroup>
```typescript Query Parameters
interface AuditLogQueryDto {
  page?: number;        // Page number (default: 1)
  limit?: number;       // Items per page (1-100, default: 20)
  tableName?: string;   // Filter by table name
  recordId?: string;    // Filter by record ID
  userId?: string;      // Filter by user ID
  action?: AuditAction; // Filter by action type
  startDate?: string;   // ISO date string
  endDate?: string;     // ISO date string
}
```

```typescript Response
interface PaginatedResponse<AuditLogDto> {
  data: AuditLogDto[];
  pagination: {
    page: number;
    limit: number;
    total: number;
    totalPages: number;
  };
}
```
</CodeGroup>

<Warning>
Pagination limits are enforced with a maximum of 100 items per page to prevent performance issues.
</Warning>

#### GET /audit/timeline/:entityType/:entityId

Get resolved timeline events for a specific entity.

**Parameters:**
- `entityType`: The type of entity (e.g., 'lead', 'deal')
- `entityId`: UUID of the specific entity

```typescript Response Format
interface ResolvedTimeline {
  eventType: TimelineEventType;
  message: string;
  createdAt: Date;
  user?: AuditUserInfo;
}
```

### System Admin Endpoints

#### GET /system-admin/audit/logs

<Info>
System-wide audit log access requires SystemAdmin role and provides cross-tenant visibility.
</Info>

interface SystemAdminAuditQueryDto extends AuditLogQueryDto {

## Business Rules and Invariants

### Data Integrity

<Check>
**Immutable audit logs** - Audit logs are immutable once created by the database trigger
</Check>

<Check>
**Sensitive data filtering** - Sensitive fields (passwords, refresh tokens) are automatically stripped by the trigger
</Check>

<Check>
**Transactional consistency** - All audit operations must occur within the same transaction as the data change
</Check>

### Access Control

The module implements strict access control patterns:

<Steps>
<Step title="Tenant isolation">
Regular users can only access audit logs for their own tenant
</Step>
<Step title="Admin privileges">
System administrators can access audit logs across all tenants
</Step>
<Step title="User information">
Timeline resolution includes user information only if accessible to the requester
</Step>
</Steps>

### Temporal Constraints

<Warning>
Date range queries validate that `endDate >= startDate` and use inclusive ranges. All timestamps are stored in UTC.
</Warning>

## Integration Points

### Authentication Module Integration

- Uses `@AuthGuard()` for endpoint protection
- Leverages `@GetCurrentUser()` for tenant context
- Integrates with `@SystemAdmin()` decorator for admin access

### Tenant Context Integration

```typescript Example Integration
// Service method showing tenant scoping
async findAuditLogs(tenantId: string, query: AuditLogQueryDto) {
  const filterQuery: FilterQuery<AuditLog> = {
    tenantId, // Automatic tenant scoping
    ...query // Additional filters
  };
  
  return this.auditRepository.findAndCount(filterQuery, {
    orderBy: { createdAt: 'desc' }
  });
}
```

### User Module Integration

- Resolves user information for timeline display
- Links audit entries to user accounts via `userId` field

### Entity Module Integration

<Tip>
The audit module monitors changes across all major entities (leads, deals, contacts, etc.) and uses entity type resolution for timeline formatting.
</Tip>

## Patterns and Conventions

### Query Patterns

The module follows consistent query patterns:

<CodeGroup>
```typescript Pagination
// Standard pagination implementation
const result = await buildPaginatedResponse(
  query,
  this.auditRepository,
  filterQuery,
  { orderBy: { createdAt: 'desc' } }
);
```

```typescript Filtering
// Type-safe database queries
const filterQuery: FilterQuery<AuditLog> = {
  tenantId,
  ...(tableName && { tableName }),
  ...(recordId && { recordId }),
  ...(userId && { userId }),
  ...(action && { action })
};
```
</CodeGroup>

### Error Handling

- Returns 404 for non-existent entities in timeline queries
- Validates UUID format for ID parameters
- Enforces pagination limits (max 100 items per page)

### Caching Considerations

<Note>
Audit data is write-heavy, read-light with no caching implemented due to real-time requirements. Timeline resolution is performed on-demand.
</Note>

### Security

<Warning>
The module enforces security through multiple layers:
- No direct entity manipulation - all writes via database triggers
- Tenant isolation enforced at multiple layers
- Sensitive data filtered at database level before persistence
</Warning>
