## Overview

The audit module provides comprehensive activity tracking and timeline functionality for the PropWise CRM system. It captures all data changes across entities through PostgreSQL triggers and presents them as both raw audit logs and human-readable timeline events.

<Info>
**Last Updated:** 2024-01-15  
**Status:** Active
</Info>

### Key Features

- Database-driven auditing using PostgreSQL triggers
- Comprehensive activity tracking across all entities
- Human-readable timeline events
- Support for both tenant-scoped and system-wide queries
- Immutable audit trail with automatic sensitive data filtering

## Entity Descriptions

### AuditLog

The core entity that captures all data changes across the system.

```typescript
interface AuditLog {
  id: string;                    // UUID primary key
  tableName: string;             // Name of the table that changed
  action: AuditAction;           // INSERT, UPDATE, DELETE
  rowData: Record<string, any>;  // Full row data after change
  changedFields: Record<string, any>; // Only changed fields (UPDATE only)
  recordId: string;              // ID of the changed record
  tenantId: string;              // Tenant context
  userId?: string;               // User who made the change
  createdAt: Date;               // When the change occurred
}
```

<Note>
The `AuditLog` entity is **strictly read-only** from application code. All writes are performed automatically by database triggers.
</Note>

### AuditAction

Enum defining the types of database operations that can be audited.

```typescript
enum AuditAction {
  INSERT = 'INSERT',
  UPDATE = 'UPDATE', 
  DELETE = 'DELETE'
}
```

### TimelineEventType

Semantic classification of audit events for human-readable timeline presentation.

<Tabs>
  <Tab title="Lead Events">
    - `LEAD_CREATED` - New lead added to the system
    - `LEAD_STAGE_CHANGED` - Lead moved to a different stage
    - `LEAD_QUALIFIED` - Lead marked as qualified
    - `LEAD_DONE` - Lead process completed
    - `LEAD_DISQUALIFIED` - Lead marked as disqualified
    - `LEAD_SCORE_CHANGED` - Lead scoring updated
  </Tab>
  
  <Tab title="Deal Events">
    - `DEAL_CREATED` - New deal created
    - `DEAL_STAGE_CHANGED` - Deal progressed to new stage
    - `DEAL_WON` - Deal successfully closed
    - `DEAL_LOST` - Deal marked as lost
  </Tab>
  
  <Tab title="Contact Events">
    - `CONTACT_CREATED` - New contact added
    - `CONTACT_UPDATED` - Contact information modified
  </Tab>
  
  <Tab title="Generic Events">
    - `RECORD_CREATED` - Generic record creation
    - `RECORD_UPDATED` - Generic record update
    - `RECORD_DELETED` - Generic record deletion
  </Tab>
</Tabs>

## API Endpoints

### Tenant-Scoped Endpoints

#### GET /audit/logs

Retrieve paginated audit logs for the current tenant.

<CodeGroup>
```typescript Query Parameters
interface AuditLogQueryDto {
  page?: number;        // Page number (default: 1)
  limit?: number;       // Items per page (1-100, default: 20)
  tableName?: string;   // Filter by table name
  recordId?: string;    // Filter by record ID
  userId?: string;      // Filter by user ID
  action?: AuditAction; // Filter by action type
  startDate?: string;   // ISO date string
  endDate?: string;     // ISO date string
}
```

```typescript Response
interface PaginatedResponse<AuditLogDto> {
  data: AuditLogDto[];
  pagination: {
    page: number;
    limit: number;
    total: number;
    totalPages: number;
  };
}
```
</CodeGroup>

<Info>
**Pagination Limits:** Maximum 100 items per page. Date ranges must have `endDate >= startDate`.
</Info>

#### GET /audit/timeline/:entityType/:entityId

Get resolved timeline events for a specific entity.

**Path Parameters:**
- `entityType` (string) - The type of entity (e.g., 'lead', 'deal')
- `entityId` (UUID) - Unique identifier of the specific entity

```typescript Response
interface ResolvedTimeline {
  eventType: TimelineEventType;
  message: string;
  createdAt: Date;
  user?: AuditUserInfo;
}
```

<Warning>
Returns **404 Not Found** if the entity does not exist or is not accessible to the current user.
</Warning>

### System Admin Endpoints

#### GET /system-admin/audit/logs

System-wide audit log access requiring SystemAdmin role.

```typescript Query Parameters
interface SystemAdminAuditQueryDto extends AuditLogQueryDto {
  tenantId?: string;    // Filter by specific tenant
}
```

<Check>
**Authorization Required:** This endpoint requires the `SystemAdmin` role and bypasses tenant isolation.
</Check>

## Business Rules and Invariants

### Data Integrity

<Steps>
  <Step title="Immutable Audit Trail">
    Audit logs are immutable once created by the database trigger. No application code can modify or delete audit records.
  </Step>
  
  <Step title="Automatic Sensitive Data Filtering">
    Sensitive fields (passwords, refresh tokens, API keys) are automatically stripped by the trigger before persistence.
  </Step>
  
  <Step title="Transactional Consistency">
    All audit operations must occur within the same transaction as the data change, ensuring atomicity.
  </Step>
</Steps>

### Access Control

<AccordionGroup>
  <Accordion title="Regular User Access">
    Regular users can only access audit logs for their own tenant. All queries are automatically scoped using tenant context.
  </Accordion>
  
  <Accordion title="System Administrator Access">
    System administrators can access audit logs across all tenants using dedicated endpoints with the `@SystemAdmin()` decorator.
  </Accordion>
  
  <Accordion title="Timeline Resolution">
    User information in timeline events is only included if the requesting user has access to view that user's details.
  </Accordion>
</AccordionGroup>

### Temporal Constraints

- Date range queries validate that `endDate >= startDate`
- Date filters use **inclusive ranges** for both start and end dates
- All timestamps are stored and returned in **UTC**

<Tip>
When querying by date range, use ISO 8601 formatted strings (e.g., `2024-01-15T00:00:00Z`) for consistent results.
</Tip>

## Integration Points

### With Authentication Module

The audit module integrates with the authentication system for access control:

- Uses `@AuthGuard()` for endpoint protection
- Leverages `@GetCurrentUser()` decorator for tenant context
- Integrates with `@SystemAdmin()` decorator for admin-only access

```typescript
@Get('logs')
@AuthGuard()
async getAuditLogs(
  @GetCurrentUser() user: User,
  @Query() query: AuditLogQueryDto
) {
  // Automatically scoped to user's tenant
}
```

### With Tenant Context

All audit queries respect multi-tenancy:

- Queries are automatically scoped to the current tenant (except system admin)
- Tenant isolation is enforced at the service layer
- Cross-tenant data access is prevented by default

### With User Module

User information enrichment for timeline display:

- Resolves user details for audit log entries via `userId` field
- Links audit entries to user accounts for attribution
- Provides user context in timeline events when accessible

### With Entity Modules

Monitors changes across all major entities:

- Captures changes to leads, deals, contacts, and other entities
- Uses entity type resolution for appropriate timeline formatting
- Supports custom timeline messages per entity type

### Caching Considerations

<Warning>
**No Caching Implemented** - The audit module does not implement caching due to:
- Write-heavy, read-light access patterns
- Real-time audit trail requirements
- Timeline resolution performed on-demand for accuracy
</Warning>

Consider implementing caching for:
- Frequently accessed timeline views
- System-wide audit dashboards
- Historical audit analytics

### Security

<Steps>
  <Step title="No Direct Entity Manipulation">
    Application code cannot directly create, update, or delete audit logs. All writes occur through database triggers.
  </Step>
  
  <Step title="Multi-Layer Tenant Isolation">
    Tenant isolation is enforced at controller, service, and database query layers for defense in depth.
  </Step>
  
  <Step title="Database-Level Data Filtering">
    Sensitive data is filtered at the database trigger level before persistence, preventing exposure even in case of application bugs.
  </Step>
  
  <Step title="Role-Based Access Control">
    System admin endpoints are protected by dedicated guards preventing unauthorized cross-tenant access.
  </Step>
</Steps>

## Best Practices

<AccordionGroup>
  <Accordion title="Use Appropriate Pagination">
    Always specify reasonable `limit` values (20-50) for list queries to balance performance and usability. Avoid requesting maximum limits unless specifically needed.
  </Accordion>
  
  <Accordion title="Filter by Entity When Possible">
    Use `tableName` and `recordId` filters to narrow queries to specific entities, improving performance and reducing data transfer.
  </Accordion>
  
  <Accordion title="Prefer Timeline Endpoints for UI">
    Use the `/timeline` endpoints for user-facing features as they provide human-readable messages. Use `/logs` for detailed audit trails and exports.
  </Accordion>
  
  <Accordion title="Implement Date Range Limits">
    When building user interfaces, implement reasonable default date ranges (e.g., last 90 days) to prevent expensive full-table scans.
  </Accordion>
  
  <Accordion title="Handle Missing Users Gracefully">
    Timeline events may have `user: undefined` if the user has been deleted or is not accessible. Always handle this case in UI rendering.
  </Accordion>
</AccordionGroup>
