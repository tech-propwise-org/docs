## Overview

The audit module provides comprehensive activity tracking and timeline functionality for the PropWise CRM system. It captures all data changes across entities through PostgreSQL triggers and presents them as both raw audit logs and human-readable timeline events. The module supports both tenant-scoped and system-wide audit queries.

<Info>
The audit module operates on a read-only principle from application code - all writes are handled automatically by database triggers to ensure data integrity and consistency.
</Info>

## Architecture

### Design decisions

The audit module follows several key architectural principles:

<AccordionGroup>
  <Accordion title="Database-driven auditing">
    Uses PostgreSQL triggers (`audit_trigger_func()`) to capture changes at the database level, ensuring consistency and completeness across all data modifications.
  </Accordion>
  
  <Accordion title="Read-only entity pattern">
    The `AuditLog` entity is strictly read-only from application code - all writes happen via database triggers to maintain data integrity.
  </Accordion>
  
  <Accordion title="Timeline abstraction">
    Provides a higher-level timeline view that translates raw audit data into user-friendly messages for better user experience.
  </Accordion>
  
  <Accordion title="Dual access patterns">
    Supports both tenant-scoped queries (for regular users) and system-wide queries (for system administrators).
  </Accordion>
</AccordionGroup>

### Key components

- **AuditLog Entity**: Maps to the `audit_log` table (read-only)
- **AuditService**: Core business logic for querying and formatting audit data
- **AuditTimelineResolver**: Translates audit logs into human-readable timeline events
- **Dual Controllers**: Separate endpoints for tenant and system admin access

## Entity descriptions

### AuditLog

The primary entity for storing audit trail information:

```typescript
interface AuditLog {
  id: string;                    // UUID primary key
  tableName: string;             // Name of the table that changed
  action: AuditAction;           // INSERT, UPDATE, DELETE
  rowData: Record<string, any>;  // Full row data after change
  changedFields: Record<string, any>; // Only changed fields (UPDATE only)
  recordId: string;              // ID of the changed record
  tenantId: string;              // Tenant context
  userId?: string;               // User who made the change
  createdAt: Date;               // When the change occurred
}
```

<Note>
The `changedFields` property is only populated for UPDATE actions and contains only the fields that were modified during the operation.
</Note>

### AuditAction

Enumeration of possible audit actions:

```typescript
enum AuditAction {
  INSERT = 'INSERT',
  UPDATE = 'UPDATE', 
  DELETE = 'DELETE'
}
```

## API endpoints

### Tenant-scoped endpoints

#### GET /audit/logs

Retrieve paginated audit logs for the current tenant.

<CodeGroup>
```typescript Query Parameters
interface AuditLogQueryDto {
  page?: number;        // Page number (default: 1)
  limit?: number;       // Items per page (1-100, default: 20)
  tableName?: string;   // Filter by table name
  recordId?: string;    // Filter by record ID
  userId?: string;      // Filter by user ID
  action?: AuditAction; // Filter by action type
  startDate?: string;   // ISO date string
  endDate?: string;     // ISO date string
}
```

```typescript Response
interface PaginatedResponse<AuditLogDto> {
  data: AuditLogDto[];
  pagination: {
    page: number;
    limit: number;
    total: number;
    totalPages: number;
  };
}
```
</CodeGroup>

<Warning>
The limit parameter is capped at 100 items per page to prevent performance issues with large result sets.
</Warning>

#### GET /audit/timeline/:entityType/:entityId

Get resolved timeline events for a specific entity.

**Parameters:**
- `entityType`: The type of entity (e.g., 'lead', 'deal')
- `entityId`: UUID of the specific entity

```typescript Response
interface ResolvedTimeline {
  eventType: TimelineEventType;
  message: string;
  createdAt: Date;
  user?: AuditUserInfo;
}
```

### System admin endpoints

#### GET /system-admin/audit/logs

System-wide audit log access (requires SystemAdmin role).

```typescript Query Parameters
interface SystemAdminAuditQueryDto extends AuditLogQueryDto {
  tenantId?: string;    // Filter by specific tenant
}
```

<Check>
System admin endpoints provide cross-tenant visibility for compliance and debugging purposes.
</Check>

## Business rules and invariants

### Data integrity

<Steps>
  <Step title="Immutable audit logs">
    Audit logs are immutable once created by the database trigger and cannot be modified through application code.
  </Step>
  
  <Step title="Sensitive data filtering">
    Sensitive fields (passwords, refresh tokens) are automatically stripped by the trigger before persistence.
  </Step>
  
  <Step title="Transactional consistency">
    All audit operations must occur within the same transaction as the data change to ensure consistency.
  </Step>
</Steps>

### Access control

- Regular users can only access audit logs for their own tenant
- System administrators can access audit logs across all tenants
- Timeline resolution includes user information only if accessible to the requester

### Temporal constraints

<Note>
All timestamps are stored in UTC and date range queries validate that `endDate >= startDate` with inclusive ranges.
</Note>

## Patterns and conventions

### Query patterns

The audit module implements several standard patterns:

```typescript
// Standard pagination
buildPaginatedResponse(data, page, limit, total)

// Flexible filtering
const filterQuery: FilterQuery<AuditLog> = {
  tenantId,
  ...(tableName && { tableName }),
  ...(recordId && { recordId })
};

// Type-safe database queries
await this.auditLogRepository.findAndCount(filterQuery, {
  orderBy: { createdAt: 'DESC' },
  limit,
  offset: (page - 1) * limit
});
```

### Error handling

<Tabs>

### Caching considerations

<Warning>
No caching is implemented due to real-time audit requirements. The audit data is write-heavy, read-light, and timeline resolution is performed on-demand.
</Warning>

### Security

<Steps>
  <Step title="No direct manipulation">
    No direct entity manipulation - all writes via database triggers only
  </Step>
  
  <Step title="Multi-layer tenant isolation">
    Tenant isolation enforced at multiple layers (controller, service, repository)
  </Step>
  
  <Step title="Database-level filtering">
    Sensitive data filtered at database level before persistence
  </Step>
</Steps>

<Tip>
The audit module's security model relies on database triggers and multiple layers of validation to ensure data integrity and proper access control.
</Tip>
