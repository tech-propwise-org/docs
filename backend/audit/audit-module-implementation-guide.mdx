## Overview

The audit module provides comprehensive activity tracking and timeline functionality for the PropWise CRM system. It captures all data changes across entities through PostgreSQL triggers and presents them as both raw audit logs and human-readable timeline events.

<Note>
The audit module supports both tenant-scoped queries for regular users and system-wide queries for administrators, ensuring proper data isolation and access control.
</Note>

## Architecture

### Design Principles

The audit module is built on four key architectural decisions:

<AccordionGroup>
<Accordion title="Database-Driven Auditing">
Uses PostgreSQL triggers (`audit_trigger_func()`) to capture changes at the database level, ensuring consistency and completeness across all data modifications.
</Accordion>

<Accordion title="Read-Only Entity Pattern">
The `AuditLog` entity is strictly read-only from application code - all writes happen exclusively via database triggers to maintain data integrity.
</Accordion>

<Accordion title="Timeline Abstraction">
Provides a higher-level timeline view that translates raw audit data into user-friendly messages for better user experience.
</Accordion>

<Accordion title="Dual Access Patterns">
Supports both tenant-scoped queries for regular users and system-wide queries for system administrators with appropriate role-based access control.
</Accordion>
</AccordionGroup>

### Core Components

- **AuditLog Entity**: Maps to the `audit_log` table (read-only)
- **AuditService**: Core business logic for querying and formatting audit data
- **AuditTimelineResolver**: Translates audit logs into human-readable timeline events
- **Dual Controllers**: Separate endpoints for tenant and system admin access

## Entity Models

### AuditLog Entity

The core audit log structure captures all change events:

```typescript
interface AuditLog {
  id: string;                    // UUID primary key
  tableName: string;             // Name of the table that changed
  action: AuditAction;           // INSERT, UPDATE, DELETE
  rowData: Record<string, any>;  // Full row data after change
  changedFields: Record<string, any>; // Only changed fields (UPDATE only)
  recordId: string;              // ID of the changed record
  tenantId: string;              // Tenant context
  userId?: string;               // User who made the change
  createdAt: Date;               // When the change occurred
}
```

### AuditAction Enumeration

```typescript
enum AuditAction {
  INSERT = 'INSERT',
  UPDATE = 'UPDATE', 
  DELETE = 'DELETE'
}
```

## API Implementation

### Tenant-Scoped Endpoints

#### Retrieve Audit Logs

<CodeGroup>
```typescript GET /audit/logs
interface AuditLogQueryDto {
  page?: number;        // Page number (default: 1)
  limit?: number;       // Items per page (1-100, default: 20)
  tableName?: string;   // Filter by table name
  recordId?: string;    // Filter by record ID
  userId?: string;      // Filter by user ID
  action?: AuditAction; // Filter by action type
  startDate?: string;   // ISO date string
  endDate?: string;     // ISO date string
}
```

```typescript Response
interface PaginatedResponse<AuditLogDto> {
  data: AuditLogDto[];
  pagination: {
    page: number;
    limit: number;
    total: number;
    totalPages: number;
  };
}
```
</CodeGroup>

<CodeGroup>
```typescript GET /audit/timeline/:entityType/:entityId
// Parameters:
// - entityType: The type of entity (e.g., 'lead', 'deal')

```

```typescript Response
interface ResolvedTimeline {
```

</CodeGroup>

## Implementation Steps

<Steps>
<Step title="Set up database triggers">
Configure PostgreSQL triggers to capture all data changes at the database level. The `audit_trigger_func()` automatically strips sensitive fields and creates audit log entries.
</Step>

<Step title="Implement AuditLog entity">
Create the read-only entity that maps to the audit_log table. Ensure no direct write operations are allowed from application code.
</Step>

<Step title="Build AuditService">
Implement core business logic for querying audit data with proper tenant isolation and filtering capabilities.
</Step>

<Step title="Create timeline resolver">
Develop the AuditTimelineResolver to translate raw audit logs into human-readable timeline events with appropriate semantic classification.
</Step>

<Step title="Implement dual controllers">
Set up separate controller endpoints for tenant-scoped and system admin access with proper authentication and authorization guards.
</Step>
</Steps>

## Business Rules

### Data Integrity

<Check>
**Immutable Audit Logs**: Once created by database triggers, audit logs cannot be modified or deleted through application code.
</Check>

<Check>
**Sensitive Data Protection**: Passwords, refresh tokens, and other sensitive fields are automatically stripped by the database trigger before persistence.
</Check>

<Check>
**Transactional Consistency**: All audit operations occur within the same transaction as the data change to ensure consistency.
</Check>

### Access Control

- **Tenant Isolation**: Regular users can only access audit logs for their own tenant
- **System Admin Access**: Administrators can access audit logs across all tenants
- **User Information**: Timeline resolution includes user information only if accessible to the requester

### Temporal Constraints

- Date range queries validate that `endDate >= startDate`
- Date filters use inclusive ranges
- All timestamps are stored in UTC

## Performance Considerations

### Query Optimization

<Tip>
The audit module uses MikroORM's FilterQuery for type-safe database queries with proper indexing on frequently queried fields like `tenantId`, `recordId`, and `createdAt`.
</Tip>

### Caching Strategy

- **Write-Heavy Pattern**: Audit data is write-heavy, read-light
- **No Caching**: No caching implemented due to real-time requirements
- **On-Demand Resolution**: Timeline resolution performed on-demand for optimal resource usage

### Pagination

Standard pagination is implemented with:
- Default page size: 20 items
- Maximum page size: 100 items
- Efficient offset-based pagination with total count

## Security Best Practices

<Warning>
The audit module implements multiple layers of security to protect sensitive audit data and ensure proper access control.
</Warning>

- **No Direct Manipulation**: All writes occur via database triggers only
- **Multi-Layer Tenant Isolation**: Enforced at service, controller, and database levels
- **Sensitive Data Filtering**: Performed at database level before persistence
- **Role-Based Access**: Strict separation between tenant and system admin access
