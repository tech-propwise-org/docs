## Overview

The audit module provides comprehensive activity tracking and timeline functionality for the PropWise CRM system. It captures all data changes across entities through PostgreSQL triggers and presents them as both raw audit logs and human-readable timeline events. The module supports both tenant-scoped and system-wide audit queries.

<Note>
  **Last Updated:** 2024-01-15  
  **Status:** Active
</Note>

## Architecture

### Design Decisions

<CardGroup cols={2}>
  <Card title="Database-Driven Auditing" icon="database">
    Uses PostgreSQL triggers (`audit_trigger_func()`) to capture changes at the database level, ensuring consistency and completeness
  </Card>
  <Card title="Read-Only Entity" icon="lock">
    The `AuditLog` entity is strictly read-only from application code - all writes happen via database triggers
  </Card>
  <Card title="Timeline Abstraction" icon="timeline">
    Provides a higher-level timeline view that translates raw audit data into user-friendly messages
  </Card>
  <Card title="Dual Access Patterns" icon="users">
    Supports both tenant-scoped queries (for regular users) and system-wide queries (for system administrators)
  </Card>
</CardGroup>

### Key Components

- **AuditLog Entity**: Maps to the `audit_log` table (read-only)
- **AuditService**: Core business logic for querying and formatting audit data
- **AuditTimelineResolver**: Translates audit logs into human-readable timeline events
- **Dual Controllers**: Separate endpoints for tenant and system admin access

## Entity Descriptions

### AuditLog

The core entity that represents a single audit log entry captured by the database trigger.

```typescript
interface AuditLog {
  id: string;                    // UUID primary key
  tableName: string;             // Name of the table that changed
  action: AuditAction;           // INSERT, UPDATE, DELETE
  rowData: Record<string, any>;  // Full row data after change
  changedFields: Record<string, any>; // Only changed fields (UPDATE only)
  recordId: string;              // ID of the changed record
  tenantId: string;              // Tenant context
  userId?: string;               // User who made the change
  createdAt: Date;               // When the change occurred
}
```

<Warning>
  The `AuditLog` entity is **read-only** from application code. All writes are performed automatically by database triggers.
</Warning>

### AuditAction

Enumeration of possible audit actions:

```typescript
enum AuditAction {
  INSERT = 'INSERT',
  UPDATE = 'UPDATE', 
  DELETE = 'DELETE'
}
```

## API Endpoints

### Tenant-scoped endpoints

#### GET /audit/logs

Retrieve paginated audit logs for the current tenant.

**Query Parameters:**

```typescript
interface AuditLogQueryDto {
  page?: number;        // Page number (default: 1)
  limit?: number;       // Items per page (1-100, default: 20)
  tableName?: string;   // Filter by table name
  recordId?: string;    // Filter by record ID
  userId?: string;      // Filter by user ID
  action?: AuditAction; // Filter by action type
  startDate?: string;   // ISO date string
  endDate?: string;     // ISO date string
}
```

**Response:**

```typescript
interface PaginatedResponse<AuditLogDto> {
  data: AuditLogDto[];
  pagination: {
    page: number;
    limit: number;
    total: number;
    totalPages: number;
  };
}
```

<Tip>
  Use the `tableName` and `recordId` parameters together to get a complete audit trail for a specific record.
</Tip>

**Example Request:**

```bash
GET /audit/logs?tableName=leads&recordId=123e4567-e89b-12d3-a456-426614174000&page=1&limit=20
```

**Example Response:**

```json
{
```

#### GET /audit/timeline/:entityType/:entityId

Get resolved timeline events for a specific entity with human-readable messages.

**Parameters:**
- `entityType`: The type of entity (e.g., 'lead', 'deal', 'contact')
- `entityId`: UUID of the specific entity

**Response:**

```typescript
interface ResolvedTimeline {
  eventType: TimelineEventType;
  message: string;
  createdAt: Date;
  user?: AuditUserInfo;
}
```

**Example Request:**

```bash
GET /audit/timeline/lead/123e4567-e89b-12d3-a456-426614174000
```

**Example Response:**

### System admin endpoints

#### GET /system-admin/audit/logs

System-wide audit log access with cross-tenant querying capabilities.

<Warning>
  This endpoint requires the `SystemAdmin` role and should be used with caution as it provides access to audit data across all tenants.
</Warning>

**Query Parameters:**

```typescript
interface SystemAdminAuditQueryDto extends AuditLogQueryDto {
  tenantId?: string;    // Filter by specific tenant
}
```

**Example Request:**

```bash
GET /system-admin/audit/logs?tenantId=tenant-uuid&tableName=users&page=1&limit=50
```

## Business Rules and Invariants

### Data integrity

<Steps>
  <Step title="Immutable audit logs">
    Audit logs are immutable once created by the database trigger. No updates or deletions are permitted from application code.
  </Step>
  <Step title="Automatic sensitive data filtering">
    Sensitive fields (passwords, refresh tokens) are automatically stripped by the trigger before persistence.
  </Step>
  <Step title="Transactional consistency">
    All audit operations must occur within the same transaction as the data change to ensure consistency.
  </Step>
</Steps>

### Access control

<Check>**Tenant isolation**: Regular users can only access audit logs for their own tenant</Check>
<Check>**System admin access**: System administrators can access audit logs across all tenants</Check>
<Check>**User information privacy**: Timeline resolution includes user information only if accessible to the requester</Check>

### Temporal constraints

- Date range queries validate that `endDate >= startDate`
- Date filters use inclusive ranges
- Timestamps are stored in UTC

<Tip>
  Always provide dates in ISO 8601 format (e.g., `2024-01-15T10:30:00Z`) to ensure consistent timezone handling.
</Tip>

### With tenant context

- All queries are automatically scoped to the current tenant (except system admin endpoints)
- Tenant isolation is enforced at the service layer
- Cross-tenant data access is prevented at multiple layers

- Resolves user information for timeline display
- Links audit entries to user accounts via `userId` field
- Provides user details in timeline events when authorized

### With entity modules

- Monitors changes across all major entities (leads, deals, contacts, etc.)
- Uses entity type resolution for timeline formatting
- Automatically captures changes through database triggers

## Patterns and Conventions

### Query patterns

<AccordionGroup>
  <Accordion title="Pagination">
    Implements standard pagination using `buildPaginatedResponse` utility. All list endpoints support `page` and `limit` query parameters with sensible defaults.
  </Accordion>
  
  <Accordion title="Flexible Filtering">
    Supports optional query parameters for filtering audit logs by table name, record ID, user ID, action type, and date range.
  </Accordion>

    Uses MikroORM's `FilterQuery` for type-safe database queries with compile-time validation.
  </Accordion>
</AccordionGroup>

### Caching considerations

<Warning>
  **No caching implemented** due to real-time requirements. Audit data is write-heavy and read-light, making caching ineffective for most use cases.
</Warning>

- Audit data changes frequently with every entity modification
- Timeline resolution is performed on-demand to ensure accuracy
- Consider implementing short-lived cache for frequently accessed audit trails if performance becomes an issue

### Security

<Steps>
  <Step title="Database-level writes">
    No direct entity manipulation from application code - all writes occur via database triggers
  </Step>
  <Step title="Multi-layer tenant isolation">
    Tenant isolation enforced at multiple layers: database triggers, service layer, and controller guards
  </Step>
  <Step title="Automatic sensitive data filtering">
    Sensitive data (passwords, tokens) filtered at database level before persistence
  </Step>
  <Step title="Role-based access">
    System admin endpoints protected with dedicated role checks
  </Step>
</Steps>

## Implementation Examples

### Querying audit logs for a specific record

```typescript
import { AuditService } from './audit.service';

@Injectable()
export class LeadService {
  constructor(private auditService: AuditService) {}

  async getLeadHistory(leadId: string, tenantId: string) {
    return this.auditService.getAuditLogs({
      tableName: 'leads',
      recordId: leadId,
      tenantId,
      page: 1,
      limit: 50
    });
  }
}
```

### Retrieving timeline events

```typescript
import { AuditService } from './audit.service';

@Get('leads/:id/timeline')
async getLeadTimeline(@Param('id') leadId: string) {
  return this.auditService.getTimeline('lead', leadId);
}
```

### System admin audit query

```typescript
import { SystemAdminAuditService } from './system-admin-audit.service';

@Get('system-admin/audit/suspicious-activity')
@SystemAdmin()
async getSuspiciousActivity() {
  return this.systemAdminAuditService.getAuditLogs({
    action: AuditAction.DELETE,
    startDate: new Date(Date.now() - 24 * 60 * 60 * 1000), // Last 24 hours
    limit: 100
  });
}
```
