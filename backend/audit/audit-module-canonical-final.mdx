# Audit Module - Canonical Specification

**Last Updated:** 2024-01-15  
**Status:** Active

## Overview

The audit module provides comprehensive activity tracking and timeline functionality for the PropWise CRM system. It captures all data changes across entities through PostgreSQL triggers and presents them as both raw audit logs and human-readable timeline events. The module supports both tenant-scoped and system-wide audit queries.

<Note>
The audit module ensures complete data change tracking by using database-level triggers, providing a reliable audit trail for compliance and debugging purposes.
</Note>

## Architecture

### Design Decisions

The audit module is built on four key architectural principles:

<Steps>
<Step title="Database-Driven Auditing">
Uses PostgreSQL triggers (`audit_trigger_func()`) to capture changes at the database level, ensuring consistency and completeness across all data modifications.
</Step>

<Step title="Read-Only Entity Pattern">
The `AuditLog` entity is strictly read-only from application code - all writes happen exclusively via database triggers to maintain data integrity.
</Step>

<Step title="Timeline Abstraction Layer">
Provides a higher-level timeline view that translates raw audit data into user-friendly messages for better user experience.
</Step>

<Step title="Dual Access Control">
Supports both tenant-scoped queries (for regular users) and system-wide queries (for system administrators) with appropriate security boundaries.
</Step>
</Steps>

### Key Components

- **AuditLog Entity**: Maps to the `audit_log` table (read-only)
- **AuditService**: Core business logic for querying and formatting audit data
- **AuditTimelineResolver**: Translates audit logs into human-readable timeline events
- **Dual Controllers**: Separate endpoints for tenant and system admin access

## Entity Descriptions

### AuditLog

<CodeGroup>
```typescript AuditLog Interface
interface AuditLog {
  id: string;                    // UUID primary key
  tableName: string;             // Name of the table that changed
  action: AuditAction;           // INSERT, UPDATE, DELETE
  rowData: Record<string, any>;  // Full row data after change
  changedFields: Record<string, any>; // Only changed fields (UPDATE only)
  recordId: string;              // ID of the changed record
  tenantId: string;              // Tenant context
  userId?: string;               // User who made the change
  createdAt: Date;               // When the change occurred
}
```
</CodeGroup>

<Warning>
The `AuditLog` entity is read-only from application code. All modifications must happen through database triggers to ensure data integrity.
</Warning>

```typescript AuditAction Enum
enum AuditAction {
```

### TimelineEventType

The timeline system provides semantic classification of audit events for better user understanding:

<AccordionGroup>
<Accordion title="Lead Events">
- `LEAD_CREATED`: New lead added to the system
- `LEAD_STAGE_CHANGED`: Lead moved between pipeline stages
- `LEAD_QUALIFIED`: Lead marked as qualified
- `LEAD_DONE`: Lead completed successfully
- `LEAD_DISQUALIFIED`: Lead marked as disqualified
- `LEAD_SCORE_CHANGED`: Lead scoring updated
</Accordion>

<Accordion title="Deal Events">
- `DEAL_CREATED`: New deal created
- `DEAL_STAGE_CHANGED`: Deal progressed through pipeline
- `DEAL_WON`: Deal closed successfully
- `DEAL_LOST`: Deal marked as lost
</Accordion>

<Accordion title="Contact Events">
- `CONTACT_CREATED`: New contact added
- `CONTACT_UPDATED`: Contact information modified
</Accordion>

<Accordion title="Generic Events">
- `RECORD_CREATED`: Generic record creation
- `RECORD_UPDATED`: Generic record modification
- `RECORD_DELETED`: Generic record deletion
</Accordion>
</AccordionGroup>

## API Endpoints

### Tenant-Scoped Endpoints

<Tabs>
<Tab title="Get Audit Logs">

#### GET /audit/logs

Retrieve paginated audit logs for the current tenant.

**Query Parameters:**

<CodeGroup>
```typescript Query Interface
interface AuditLogQueryDto {
  page?: number;        // Page number (default: 1)
  limit?: number;       // Items per page (1-100, default: 20)
  tableName?: string;   // Filter by table name
  recordId?: string;    // Filter by record ID
  userId?: string;      // Filter by user ID
  action?: AuditAction; // Filter by action type
  startDate?: string;   // ISO date string
  endDate?: string;     // ISO date string
}
```
</CodeGroup>

**Response:**

<CodeGroup>
```typescript Response Format
interface PaginatedResponse<AuditLogDto> {
  data: AuditLogDto[];
  pagination: {
    page: number;
    limit: number;
    total: number;
    totalPages: number;
  };
}
```
</CodeGroup>

<Info>
Pagination limits are enforced with a maximum of 100 items per page to ensure optimal performance.
</Info>

</Tab>

<Tab title="Get Timeline">

#### GET /audit/timeline/:entityType/:entityId

Get resolved timeline events for a specific entity.

**Parameters:**
- `entityType`: The type of entity (e.g., 'lead', 'deal')

**Response:**

<CodeGroup>

```typescript Timeline Response
interface ResolvedTimeline {

  message: string;
  createdAt: Date;
  user?: AuditUserInfo;

```
</CodeGroup>

<Tip>
Timeline events provide human-readable messages that make audit data accessible to non-technical users.
</Tip>

</Tab>
</Tabs>

## Business Rules and Invariants

### Data Integrity

<Check>Audit logs are immutable once created by the database trigger</Check>
<Check>Sensitive fields (passwords, refresh tokens) are automatically stripped by the trigger</Check>
<Check>All audit operations must occur within the same transaction as the data change</Check>

### Access Control

The audit module enforces strict access control patterns:

- **Regular users** can only access audit logs for their own tenant
- **System administrators** can access audit logs across all tenants
- **Timeline resolution** includes user information only if accessible to the requester

### Temporal Constraints

<Steps>
<Step title="Date Validation">
Date range queries validate that `endDate >= startDate` to ensure logical consistency.
</Step>

<Step title="Inclusive Ranges">
Date filters use inclusive ranges for both start and end dates.
</Step>

<Step title="UTC Storage">
All timestamps are stored in UTC to avoid timezone-related issues.
</Step>
</Steps>

</Tab>

### Error Handling

The module implements comprehensive error handling:

- Returns **404** for non-existent entities in timeline queries
- Validates **UUID format** for ID parameters
- Enforces **pagination limits** (max 100 items per page)
- Provides **meaningful error messages** for debugging

### Caching Considerations

<Note>
The audit module is designed for write-heavy, read-light patterns with real-time requirements. No caching is implemented to ensure data freshness and accuracy.
</Note>

- **Audit data** is write-heavy, read-light by nature
- **No caching** implemented due to real-time audit requirements
- **Timeline resolution** performed on-demand for accuracy

### Security

<Steps>
<Step title="Write Protection">
No direct entity manipulation allowed - all writes occur exclusively via database triggers
</Step>

<Step title="Tenant Isolation">
Multi-layered tenant isolation enforced at database, service, and controller levels
</Step>

<Step title="Data Filtering">
Sensitive data automatically filtered at database level before persistence
</Step>

<Step title="Access Control">
Role-based access control with separate endpoints for different privilege levels
</Step>
</Steps>
