## Overview

The audit module provides comprehensive activity tracking and timeline functionality for the PropWise CRM system. It captures all data changes across entities through PostgreSQL triggers and presents them as both raw audit logs and human-readable timeline events.

<Info>
The module supports both tenant-scoped and system-wide audit queries, ensuring complete traceability while maintaining proper access controls.
</Info>

## Architecture

### Design decisions

The audit module follows several key architectural principles:

<AccordionGroup>
<Accordion title="Database-driven auditing">
Uses PostgreSQL triggers (`audit_trigger_func()`) to capture changes at the database level, ensuring consistency and completeness across all data modifications.
</Accordion>

<Accordion title="Read-only entity pattern">
The `AuditLog` entity is strictly read-only from application code - all writes happen via database triggers to maintain data integrity.
</Accordion>

<Accordion title="Timeline abstraction layer">
Provides a higher-level timeline view that translates raw audit data into user-friendly messages for better user experience.
</Accordion>

<Accordion title="Dual access patterns">
Supports both tenant-scoped queries (for regular users) and system-wide queries (for system administrators).
</Accordion>
</AccordionGroup>

### Key components

- **AuditLog Entity**: Maps to the `audit_log` table (read-only)
- **AuditService**: Core business logic for querying and formatting audit data
- **AuditTimelineResolver**: Translates audit logs into human-readable timeline events
- **Dual Controllers**: Separate endpoints for tenant and system admin access

## Entity specifications

### AuditLog entity

<CodeGroup>
```typescript TypeScript Interface
interface AuditLog {
  id: string;                    // UUID primary key
  tableName: string;             // Name of the table that changed
  action: AuditAction;           // INSERT, UPDATE, DELETE
  rowData: Record<string, any>;  // Full row data after change
  changedFields: Record<string, any>; // Only changed fields (UPDATE only)
  recordId: string;              // ID of the changed record
  tenantId: string;              // Tenant context
  userId?: string;               // User who made the change
  createdAt: Date;               // When the change occurred
}
```

```typescript Audit Action Enum
enum AuditAction {
  INSERT = 'INSERT',
  UPDATE = 'UPDATE', 
  DELETE = 'DELETE'
}
```
</CodeGroup>

## API endpoints

### Tenant-scoped endpoints

#### GET /audit/logs

Retrieve paginated audit logs for the current tenant.

<CodeGroup>
```typescript Query Parameters
interface AuditLogQueryDto {
  page?: number;        // Page number (default: 1)
  limit?: number;       // Items per page (1-100, default: 20)
  tableName?: string;   // Filter by table name
  recordId?: string;    // Filter by record ID
  userId?: string;      // Filter by user ID
  action?: AuditAction; // Filter by action type
  startDate?: string;   // ISO date string
  endDate?: string;     // ISO date string
}
```

```typescript Response
interface PaginatedResponse<AuditLogDto> {
  data: AuditLogDto[];
  pagination: {
    page: number;
    limit: number;
    total: number;
    totalPages: number;
  };
}
```
</CodeGroup>

#### GET /audit/timeline/:entityType/:entityId

Get resolved timeline events for a specific entity.

**Parameters:**
- `entityType`: The type of entity (e.g., 'lead', 'deal')
- `entityId`: UUID of the specific entity

```typescript Response
interface ResolvedTimeline {
  eventType: TimelineEventType;
  message: string;
  createdAt: Date;
  user?: AuditUserInfo;
}
```

## Business rules and invariants

### Data integrity

<Steps>
<Step title="Immutable audit logs">
Audit logs are immutable once created by the database trigger and cannot be modified through application code.
</Step>

<Step title="Sensitive field filtering">
Sensitive fields (passwords, refresh tokens) are automatically stripped by the trigger before persistence.
</Step>

<Step title="Transactional consistency">
All audit operations must occur within the same transaction as the data change to ensure consistency.
</Step>
</Steps>

### Access control

<Check>
Regular users can only access audit logs for their own tenant
</Check>

<Check>
System administrators can access audit logs across all tenants
</Check>

<Check>
Timeline resolution includes user information only if accessible to the requester
</Check>

### Temporal constraints

- Date range queries validate that `endDate >= startDate`
- Date filters use inclusive ranges
- Timestamps are stored in UTC

## Integration points

### Authentication module integration

<Tabs>
<Tab title="Endpoint Protection">
Uses `@AuthGuard()` decorator for securing all audit endpoints against unauthorized access.
</Tab>

<Tab title="User Context">
Leverages `@GetCurrentUser()` decorator to automatically extract tenant context from the authenticated user.
</Tab>

<Tab title="Admin Access">
Integrates with `@SystemAdmin()` decorator to restrict system-wide access to authorized administrators.
</Tab>
</Tabs>

### Tenant context integration

<Note>
All queries are automatically scoped to the current tenant (except system admin endpoints). Tenant isolation is enforced at the service layer to prevent cross-tenant data access.
</Note>

### User module integration

- Resolves user information for timeline display
- Links audit entries to user accounts via `userId` field
- Handles user lookup for timeline event formatting

### Entity module integration

- Monitors changes across all major entities (leads, deals, contacts, etc.)
- Uses entity type resolution for timeline formatting
- Provides universal change tracking for all auditable entities

## Patterns and conventions

### Query patterns

The audit module implements several standard patterns:

<AccordionGroup>
<Accordion title="Standard pagination">
Implements pagination using `buildPaginatedResponse` helper with configurable page size and limits.
</Accordion>

<Accordion title="Flexible filtering">
Supports optional query parameters for filtering by table, user, action type, and date ranges.
</Accordion>

<Accordion title="Type-safe queries">
Uses MikroORM's FilterQuery for compile-time type safety in database operations.
</Accordion>
</AccordionGroup>

### Error handling

<Steps>
<Step title="Entity validation">
Returns 404 for non-existent entities in timeline queries with descriptive error messages.
</Step>

<Step title="Parameter validation">
Validates UUID format for ID parameters and rejects malformed requests.
</Step>

<Step title="Pagination limits">
Enforces maximum pagination limits (100 items per page) to prevent performance issues.
</Step>
</Steps>

### Caching considerations

<Warning>
No caching is implemented due to real-time audit requirements. Audit data is write-heavy and read-light, making caching potentially counterproductive.
</Warning>

Timeline resolution is performed on-demand to ensure the most current information is always presented to users.

### Security measures

<Tip>
The audit module implements defense-in-depth security:

- No direct entity manipulation - all writes via database triggers
- Tenant isolation enforced at multiple layers  
- Sensitive data filtered at database level before persistence
- Role-based access controls for system-wide queries
</Tip>
