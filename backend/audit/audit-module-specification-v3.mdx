## Overview

The audit module provides comprehensive activity tracking and timeline functionality for the PropWise CRM system. It captures all data changes across entities through PostgreSQL triggers and presents them as both raw audit logs and human-readable timeline events. The module supports both tenant-scoped and system-wide audit queries.

<Note>
The audit module uses database-level triggers to ensure complete and consistent tracking of all data changes, making it impossible to bypass audit logging through application-level modifications.
</Note>

## Architecture

### Design decisions

The audit module follows several key architectural principles:

<AccordionGroup>
<Accordion title="Database-driven auditing">
Uses PostgreSQL triggers (`audit_trigger_func()`) to capture changes at the database level, ensuring consistency and completeness
</Accordion>

<Accordion title="Read-only entity">
The `AuditLog` entity is strictly read-only from application code - all writes happen via database triggers
</Accordion>

<Accordion title="Timeline abstraction">
Provides a higher-level timeline view that translates raw audit data into user-friendly messages
</Accordion>

<Accordion title="Dual access patterns">
Supports both tenant-scoped queries (for regular users) and system-wide queries (for system administrators)
</Accordion>
</AccordionGroup>

### Key components

- **AuditLog Entity**: Maps to the `audit_log` table (read-only)
- **AuditService**: Core business logic for querying and formatting audit data
- **AuditTimelineResolver**: Translates audit logs into human-readable timeline events
- **Dual Controllers**: Separate endpoints for tenant and system admin access

## Entity descriptions

### AuditLog

The core audit log entity that captures all data changes:

```typescript
interface AuditLog {
  id: string;                    // UUID primary key
  tableName: string;             // Name of the table that changed
  action: AuditAction;           // INSERT, UPDATE, DELETE
  rowData: Record<string, any>;  // Full row data after change
  changedFields: Record<string, any>; // Only changed fields (UPDATE only)
  recordId: string;              // ID of the changed record
  tenantId: string;              // Tenant context
  userId?: string;               // User who made the change
  createdAt: Date;               // When the change occurred
}
```

### AuditAction

Enumeration of possible database operations:

```typescript
enum AuditAction {
  INSERT = 'INSERT',
  UPDATE = 'UPDATE', 
  DELETE = 'DELETE'
}
```

## API endpoints

### Tenant-scoped endpoints

These endpoints are available to regular users within their tenant context.

#### GET /audit/logs

Retrieve paginated audit logs for the current tenant.

**Query Parameters:**

```typescript
interface AuditLogQueryDto {
  page?: number;        // Page number (default: 1)
  limit?: number;       // Items per page (1-100, default: 20)
  tableName?: string;   // Filter by table name
  recordId?: string;    // Filter by record ID
  userId?: string;      // Filter by user ID
  action?: AuditAction; // Filter by action type
  startDate?: string;   // ISO date string
  endDate?: string;     // ISO date string
}
```

**Response:**

```typescript
interface PaginatedResponse<AuditLogDto> {
  data: AuditLogDto[];
  pagination: {
    page: number;
    limit: number;
    total: number;
    totalPages: number;
  };
}
```

<Warning>
The maximum limit is 100 items per page to prevent performance issues with large audit datasets.
</Warning>

#### GET /audit/timeline/:entityType/:entityId

Get resolved timeline events for a specific entity.

**Parameters:**
- `entityType`: The type of entity (e.g., 'lead', 'deal')
- `entityId`: UUID of the specific entity

**Response:**

```typescript
interface ResolvedTimeline {
  eventType: TimelineEventType;
  message: string;
  createdAt: Date;
  user?: AuditUserInfo;
}
```

### System admin endpoints

These endpoints require SystemAdmin role and provide cross-tenant access.

#### GET /system-admin/audit/logs

System-wide audit log access with additional tenant filtering capabilities.

**Query Parameters:**

```typescript
interface SystemAdminAuditQueryDto extends AuditLogQueryDto {
  tenantId?: string;    // Filter by specific tenant
}
```

<Info>
System admin endpoints include all the same filtering capabilities as tenant-scoped endpoints, plus the ability to filter by tenant ID.
</Info>

## Business rules and invariants

### Data integrity

<Steps>
<Step title="Immutable audit logs">
Audit logs are immutable once created by the database trigger
</Step>

<Step title="Sensitive data protection">
Sensitive fields (passwords, refresh tokens) are automatically stripped by the trigger
</Step>

<Step title="Transaction consistency">
All audit operations must occur within the same transaction as the data change
</Step>
</Steps>

### Access control

- **Regular users**: Can only access audit logs for their own tenant
- **System administrators**: Can access audit logs across all tenants
- **Timeline resolution**: Includes user information only if accessible to the requester

### Temporal constraints

<Check>Date range queries validate that `endDate >= startDate`</Check>
<Check>Date filters use inclusive ranges</Check>
<Check>Timestamps are stored in UTC</Check>

## Integration points

### With authentication module

The audit module integrates closely with the authentication system:

- Uses `@AuthGuard()` for endpoint protection
- Leverages `@GetCurrentUser()` for tenant context
- Integrates with `@SystemAdmin()` decorator for admin access

### With tenant context

<Note>
All queries are automatically scoped to the current tenant (except system admin endpoints). Tenant isolation is enforced at the service layer.
</Note>

### With user module

- Resolves user information for timeline display
- Links audit entries to user accounts via `userId` field

### With entity modules

- Monitors changes across all major entities (leads, deals, contacts, etc.)
- Uses entity type resolution for timeline formatting

## Patterns and conventions

### Query patterns

The audit module follows standard PropWise CRM query patterns:

- Implements standard pagination using `buildPaginatedResponse`
- Supports flexible filtering with optional query parameters
- Uses MikroORM's FilterQuery for type-safe database queries

Validates UUID format for ID parameters
</Card>

<Card title="Pagination limits" icon="list">
Enforces pagination limits (max 100 items per page)
</Card>

### Caching considerations

<Tip>
Audit data is write-heavy, read-light by nature. No caching is implemented due to real-time requirements, and timeline resolution is performed on-demand.
</Tip>

### Security

The audit module maintains strict security controls:

- **No direct entity manipulation**: All writes occur via database triggers
- **Multi-layer tenant isolation**: Enforced at database, service, and controller layers
- **Sensitive data filtering**: Performed at database level before persistence

<Warning>
The audit system is designed to be tamper-proof. Any attempt to modify audit logs directly through the application will fail, as the AuditLog entity is read-only.
</Warning>
