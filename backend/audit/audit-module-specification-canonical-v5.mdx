## Overview

The audit module provides comprehensive activity tracking and timeline functionality for the PropWise CRM system. It captures all data changes across entities through PostgreSQL triggers and presents them as both raw audit logs and human-readable timeline events. The module supports both tenant-scoped and system-wide audit queries.

<Info>
**Last Updated:** 2024-01-15  
**Status:** Active
</Info>

## Architecture

### Design Decisions

<CardGroup cols={2}>
  <Card title="Database-Driven Auditing" icon="database">
    Uses PostgreSQL triggers (`audit_trigger_func()`) to capture changes at the database level, ensuring consistency and completeness
  </Card>
  <Card title="Read-Only Entity" icon="lock">
    The `AuditLog` entity is strictly read-only from application code - all writes happen via database triggers
  </Card>
  <Card title="Timeline Abstraction" icon="timeline">
    Provides a higher-level timeline view that translates raw audit data into user-friendly messages
  </Card>
  <Card title="Dual Access Patterns" icon="users">
    Supports both tenant-scoped queries (for regular users) and system-wide queries (for system administrators)
  </Card>
</CardGroup>

### Key Components

- **AuditLog Entity**: Maps to the `audit_log` table (read-only)
- **AuditService**: Core business logic for querying and formatting audit data
- **AuditTimelineResolver**: Translates audit logs into human-readable timeline events
- **Dual Controllers**: Separate endpoints for tenant and system admin access

## Entity Descriptions

### AuditLog

The core entity for tracking all data changes in the system.

```typescript
interface AuditLog {
  id: string;                    // UUID primary key
  tableName: string;             // Name of the table that changed
  action: AuditAction;           // INSERT, UPDATE, DELETE
  rowData: Record<string, any>;  // Full row data after change
  changedFields: Record<string, any>; // Only changed fields (UPDATE only)
  recordId: string;              // ID of the changed record
  tenantId: string;              // Tenant context
  userId?: string;               // User who made the change
  createdAt: Date;               // When the change occurred
}
```

<Note>
The `changedFields` property is only populated for UPDATE operations, containing just the fields that were modified.
</Note>

### AuditAction

```typescript
enum AuditAction {
  INSERT = 'INSERT',
  UPDATE = 'UPDATE', 
  DELETE = 'DELETE'
}
```

### Tenant-Scoped Endpoints

<Note>
These endpoints automatically scope all queries to the authenticated user's tenant context.

#### GET /audit/logs

Retrieve paginated audit logs for the current tenant.

**Query Parameters:**

```typescript
interface AuditLogQueryDto {
  page?: number;        // Page number (default: 1)
  limit?: number;       // Items per page (1-100, default: 20)
  tableName?: string;   // Filter by table name
  recordId?: string;    // Filter by record ID
  userId?: string;      // Filter by user ID
  action?: AuditAction; // Filter by action type
  startDate?: string;   // ISO date string
  endDate?: string;     // ISO date string
}
```

**Response:**

```typescript
interface PaginatedResponse<AuditLogDto> {
  data: AuditLogDto[];
  pagination: {
    page: number;

```

## Business Rules and Invariants

### Data Integrity

<Steps>
  <Step title="Immutable Audit Records">
    Audit logs are immutable once created by the database trigger. No application code can modify or delete audit records.
  </Step>
  
  <Step title="Automatic Sensitive Data Filtering">
    Sensitive fields (passwords, refresh tokens) are automatically stripped by the trigger before persistence.
  </Step>
  
  <Step title="Transactional Consistency">
    All audit operations must occur within the same transaction as the data change to ensure consistency.
  </Step>
</Steps>

### Access Control

<CardGroup cols={2}>
  <Card title="Tenant Isolation" icon="shield">
    Regular users can only access audit logs for their own tenant
  </Card>
  <Card title="System Admin Access" icon="user-shield">
    System administrators can access audit logs across all tenants
  </Card>
  <Card title="User Context" icon="user">
    Timeline resolution includes user information only if accessible to the requester
  </Card>
</CardGroup>

### Temporal Constraints

- Date range queries validate that `endDate >= startDate`
- Date filters use inclusive ranges
- Timestamps are stored in UTC

<Note>
When querying with date ranges, both the start and end dates are inclusive. All dates must be provided in ISO 8601 format.
</Note>

### Caching Considerations

<Warning>
Audit data is write-heavy, read-light with real-time requirements. No caching is implemented to ensure data freshness.
</Warning>

**Rationale:**
- Audit logs must reflect the most current state for compliance
- Timeline resolution is performed on-demand for accuracy
- Caching could introduce staleness issues in audit trails

### Security

<Steps>
  <Step title="No Direct Manipulation">
    No direct entity manipulation allowed - all writes occur via database triggers
  </Step>
  
  <Step title="Multi-Layer Tenant Isolation">
    Tenant isolation enforced at multiple layers:
    - Controller level (authentication)
    - Service level (query filtering)
    - Database level (row-level security)
  </Step>
  
  <Step title="Database-Level Filtering">
    Sensitive data filtered at database level before persistence:
    
    ```sql
    CREATE OR REPLACE FUNCTION audit_trigger_func()
    RETURNS TRIGGER AS $$
    BEGIN
      -- Strip sensitive fields
      NEW.row_data = NEW.row_data - 'password' - 'refresh_token';
      RETURN NEW;
    END;
    $$ LANGUAGE plpgsql;
    ```
  </Step>
</Steps>

## Usage Examples

### Querying Recent Changes

```typescript
// Get all changes in the last 24 hours
const yesterday = new Date();
yesterday.setDate(yesterday.getDate() - 1);

const recentChanges = await fetch('/audit/logs?' + new URLSearchParams({
  startDate: yesterday.toISOString(),
  limit: '50'
}));
```

### Viewing Entity Timeline

```typescript
// Get complete timeline for a lead
const leadTimeline = await fetch(
  `/audit/timeline/lead/123e4567-e89b-12d3-a456-426614174000`
);

// Display timeline events
leadTimeline.forEach(event => {
  console.log(`${event.eventType}: ${event.message} by ${event.user?.name}`);
});
```

### System Admin Cross-Tenant Query

```typescript
// Query audit logs across all tenants (system admin only)
const allTenantLogs = await fetch('/system-admin/audit/logs?' + new URLSearchParams({
  tableName: 'deals',
  action: 'DELETE',
  limit: '100'
}));

// Or filter to specific tenant
const tenantLogs = await fetch('/system-admin/audit/logs?' + new URLSearchParams({
  tenantId: 'tenant-123',
  limit: '50'
}));
```
