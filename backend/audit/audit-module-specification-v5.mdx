The audit module provides comprehensive activity tracking and timeline functionality for the PropWise CRM system. It captures all data changes across entities through PostgreSQL triggers and presents them as both raw audit logs and human-readable timeline events.

<Info>
The module supports both tenant-scoped and system-wide audit queries, with strict tenant isolation for security.
</Info>

## Architecture Overview

### Design Decisions

The audit module follows several key architectural principles:

<Steps>
<Step title="Database-Driven Auditing">
Uses PostgreSQL triggers (`audit_trigger_func()`) to capture changes at the database level, ensuring consistency and completeness
</Step>

<Step title="Read-Only Entity">
The `AuditLog` entity is strictly read-only from application code - all writes happen via database triggers
</Step>

<Step title="Timeline Abstraction">
Provides a higher-level timeline view that translates raw audit data into user-friendly messages
</Step>

<Step title="Dual Access Patterns">
Supports both tenant-scoped queries (for regular users) and system-wide queries (for system administrators)
</Step>
</Steps>

## Entity Specifications

### AuditLog Entity

The core audit log entity captures all data changes:

```typescript
interface AuditLog {
  id: string;                    // UUID primary key
  tableName: string;             // Name of the table that changed
  action: AuditAction;           // INSERT, UPDATE, DELETE
  rowData: Record<string, any>;  // Full row data after change
  changedFields: Record<string, any>; // Only changed fields (UPDATE only)
  recordId: string;              // ID of the changed record
  tenantId: string;              // Tenant context
  userId?: string;               // User who made the change
  createdAt: Date;               // When the change occurred
}
```

### AuditAction Enum

<Tabs>
<Tab title="Actions">
```typescript
enum AuditAction {
  INSERT = 'INSERT',
  UPDATE = 'UPDATE', 
  DELETE = 'DELETE'
}
```
</Tab>
<Tab title="Usage">
- **INSERT**: New record creation
- **UPDATE**: Field modifications
- **DELETE**: Record removal
</Tab>
</Tabs>

### Timeline Event Types

Timeline events provide semantic classification of audit data:

<AccordionGroup>
<Accordion title="Lead Events">
- `LEAD_CREATED`: New lead registered
- `LEAD_STAGE_CHANGED`: Stage progression
- `LEAD_QUALIFIED`: Lead marked as qualified
- `LEAD_DONE`: Lead completed
- `LEAD_DISQUALIFIED`: Lead rejected
- `LEAD_SCORE_CHANGED`: Score modification
</Accordion>

<Accordion title="Deal Events">
- `DEAL_CREATED`: New deal initiated
- `DEAL_STAGE_CHANGED`: Deal progression
- `DEAL_WON`: Successful deal closure
- `DEAL_LOST`: Deal marked as lost
</Accordion>

<Accordion title="Contact Events">
- `CONTACT_CREATED`: New contact added
- `CONTACT_UPDATED`: Contact information changed
</Accordion>

<Accordion title="Generic Events">
- `RECORD_CREATED`: Generic record creation
- `RECORD_UPDATED`: Generic record modification
- `RECORD_DELETED`: Generic record removal
</Accordion>
</AccordionGroup>

## API Endpoints

### Tenant-Scoped Endpoints

#### GET /audit/logs

Retrieve paginated audit logs for the current tenant.

<CodeGroup>
```typescript Query Parameters
interface AuditLogQueryDto {
  page?: number;        // Page number (default: 1)
  limit?: number;       // Items per page (1-100, default: 20)
  tableName?: string;   // Filter by table name
  recordId?: string;    // Filter by record ID
  userId?: string;      // Filter by user ID
  action?: AuditAction; // Filter by action type
  startDate?: string;   // ISO date string
  endDate?: string;     // ISO date string
}
```

```typescript Response
interface PaginatedResponse<AuditLogDto> {
  data: AuditLogDto[];
  pagination: {
    page: number;
    limit: number;
    total: number;
    totalPages: number;
  };
}
```
</CodeGroup>

#### GET /audit/timeline/:entityType/:entityId

Get resolved timeline events for a specific entity.

<Note>
Parameters include `entityType` (e.g., 'lead', 'deal') and `entityId` (UUID of the specific entity).
</Note>

```typescript
interface ResolvedTimeline {
  eventType: TimelineEventType;
  message: string;
  createdAt: Date;
  user?: AuditUserInfo;
}
```

### System Admin Endpoints

#### GET /system-admin/audit/logs

interface SystemAdminAuditQueryDto extends AuditLogQueryDto {

## Business Rules and Invariants

### Data Integrity

<Check>Audit logs are immutable once created by the database trigger</Check>
<Check>Sensitive fields (passwords, refresh tokens) are automatically stripped</Check>
<Check>All audit operations occur within the same transaction as the data change</Check>

### Access Control

The module enforces strict access controls:

- **Regular users**: Can only access audit logs for their own tenant
- **System administrators**: Can access audit logs across all tenants
- **Timeline resolution**: Includes user information only if accessible to the requester

### Temporal Constraints

<Tip>
All timestamps are stored in UTC and date range queries validate that `endDate >= startDate`.
</Tip>

- Date filters use inclusive ranges
- Date range validation ensures logical ordering

## Implementation Patterns

### Query Patterns

The module follows consistent query patterns:

- **Pagination**: Uses `buildPaginatedResponse` for standard pagination
- **Filtering**: Supports flexible filtering with optional query parameters
- **Type Safety**: Uses MikroORM's FilterQuery for type-safe database queries

### Error Handling

<Tabs>
<Tab title="Common Errors">
- Returns 404 for non-existent entities in timeline queries
- Validates UUID format for ID parameters
- Enforces pagination limits (max 100 items per page)
</Tab>
<Tab title="Validation">
- Date range validation
- Parameter type checking
- Access permission verification
</Tab>
</Tabs>

### Security Considerations

<Warning>
The audit module implements multiple security layers to protect sensitive data.
</Warning>

- **No direct manipulation**: All writes via database triggers only
- **Multi-layer isolation**: Tenant isolation enforced at multiple layers
- **Data filtering**: Sensitive data filtered at database level before persistence
- **Access controls**: Role-based access with strict tenant boundaries

### Performance Considerations

<Info>
The audit module is optimized for write-heavy, read-light workloads typical of audit systems.
</Info>

- **No caching**: Real-time requirements prevent caching of audit data
- **On-demand resolution**: Timeline resolution performed when requested
- **Database optimization**: Proper indexing on frequently queried fields

## Best Practices

<Steps>
<Step title="Query Optimization">
Use specific filters (tableName, recordId) to reduce query scope and improve performance
</Step>

<Step title="Date Range Queries">
Always specify reasonable date ranges to avoid scanning large datasets
</Step>

<Step title="Timeline Usage">
Prefer timeline endpoints for user-facing features, raw audit logs for debugging
</Step>

<Step title="System Admin Access">
Use system admin endpoints sparingly and only for legitimate administrative needs
</Step>
</Steps>
