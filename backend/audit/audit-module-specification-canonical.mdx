## Overview

The audit module provides comprehensive activity tracking and timeline functionality for the PropWise CRM system. It captures all data changes across entities through PostgreSQL triggers and presents them as both raw audit logs and human-readable timeline events. The module supports both tenant-scoped and system-wide audit queries.

<Info>
The audit module uses database-level triggers to ensure complete and consistent change tracking across all entities.
</Info>

## Architecture

### Design decisions

The audit module follows several key architectural principles:

<AccordionGroup>
  <Accordion title="Database-driven auditing">
    Uses PostgreSQL triggers (`audit_trigger_func()`) to capture changes at the database level, ensuring consistency and completeness regardless of how data is modified.
  </Accordion>
  
  <Accordion title="Read-only entity pattern">
    The `AuditLog` entity is strictly read-only from application code - all writes happen via database triggers, preventing accidental data corruption.
  </Accordion>
  
  <Accordion title="Timeline abstraction">
    Provides a higher-level timeline view that translates raw audit data into user-friendly messages for improved user experience.
  </Accordion>
  
  <Accordion title="Dual access patterns">
    Supports both tenant-scoped queries (for regular users) and system-wide queries (for system administrators).
  </Accordion>
</AccordionGroup>

## Entity descriptions

### AuditLog

The core entity that represents all database changes captured by the audit system.

```typescript
interface AuditLog {
  id: string;                    // UUID primary key
  tableName: string;             // Name of the table that changed
  action: AuditAction;           // INSERT, UPDATE, DELETE
  rowData: Record<string, any>;  // Full row data after change
  changedFields: Record<string, any>; // Only changed fields (UPDATE only)
  recordId: string;              // ID of the changed record
  tenantId: string;              // Tenant context
  userId?: string;               // User who made the change
  createdAt: Date;               // When the change occurred
}
```

<Note>
The `changedFields` property is only populated for UPDATE operations and contains only the fields that actually changed.
</Note>

### AuditAction

Enumeration of possible database operations:

```typescript
enum AuditAction {
  INSERT = 'INSERT',
  UPDATE = 'UPDATE', 
  DELETE = 'DELETE'
}
```

## API endpoints

### Tenant-scoped endpoints

#### GET /audit/logs

Retrieve paginated audit logs for the current tenant.

<CodeGroup>
```typescript Query Parameters
interface AuditLogQueryDto {
  page?: number;        // Page number (default: 1)
  limit?: number;       // Items per page (1-100, default: 20)
  tableName?: string;   // Filter by table name
  recordId?: string;    // Filter by record ID
  userId?: string;      // Filter by user ID
  action?: AuditAction; // Filter by action type
  startDate?: string;   // ISO date string
  endDate?: string;     // ISO date string
}
```

```typescript Response
interface PaginatedResponse<AuditLogDto> {
  data: AuditLogDto[];
  pagination: {
    page: number;
    limit: number;
    total: number;
    totalPages: number;
  };
}
```
</CodeGroup>

#### GET /audit/timeline/:entityType/:entityId

Get resolved timeline events for a specific entity.

**Parameters:**
- `entityType`: The type of entity (e.g., 'lead', 'deal')
- `entityId`: UUID of the specific entity

```typescript Response Format
interface ResolvedTimeline {
  eventType: TimelineEventType;
  message: string;
  createdAt: Date;
  user?: AuditUserInfo;
}
```

### System admin endpoints

#### GET /system-admin/audit/logs

System-wide audit log access (requires SystemAdmin role).

<Warning>
This endpoint provides access to audit logs across all tenants and requires elevated system administrator privileges.
</Warning>

```typescript Query Parameters
interface SystemAdminAuditQueryDto extends AuditLogQueryDto {
  tenantId?: string;    // Filter by specific tenant
}
```

## Business rules and invariants

### Data integrity

<Steps>
  <Step title="Immutable audit logs">
    Audit logs are immutable once created by the database trigger and cannot be modified by application code.
  </Step>
  
  <Step title="Automatic data filtering">
    Sensitive fields (passwords, refresh tokens) are automatically stripped by the trigger before persistence.
  </Step>
  
  <Step title="Transactional consistency">
    All audit operations must occur within the same transaction as the data change to maintain consistency.
  </Step>
</Steps>

### Access control

<Check>
Regular users can only access audit logs for their own tenant
</Check>

<Check>
System administrators can access audit logs across all tenants
</Check>

<Check>
Timeline resolution includes user information only if accessible to the requester
</Check>

### Temporal constraints

- Date range queries validate that `endDate >= startDate`
- Date filters use inclusive ranges
- Timestamps are stored in UTC

<Tip>
When querying by date range, use ISO date strings in UTC format for consistent results.
</Tip>

## Integration points

### With authentication module

The audit module integrates closely with the authentication system:

- Uses `@AuthGuard()` for endpoint protection
- Leverages `@GetCurrentUser()` for tenant context
- Integrates with `@SystemAdmin()` decorator for admin access

### With tenant context

- All queries are automatically scoped to the current tenant (except system admin)
- Tenant isolation is enforced at the service layer

### With user module

- Resolves user information for timeline display
- Links audit entries to user accounts via `userId` field

### With entity modules

- Monitors changes across all major entities (leads, deals, contacts, etc.)
- Uses entity type resolution for timeline formatting

## Patterns and conventions

### Query patterns

The audit module follows consistent patterns for data access:

<AccordionGroup>
  <Accordion title="Standard pagination">
    Implements standard pagination using `buildPaginatedResponse` helper function.
  </Accordion>
  
  <Accordion title="Flexible filtering">
    Supports flexible filtering with optional query parameters for various use cases.
  </Accordion>
  
  <Accordion title="Type-safe queries">
    Uses MikroORM's FilterQuery for type-safe database queries with compile-time validation.
  </Accordion>
</AccordionGroup>

### Error handling

- Returns 404 for non-existent entities in timeline queries
- Validates UUID format for ID parameters
- Enforces pagination limits (max 100 items per page)

### Caching considerations

<Note>
The audit module does not implement caching due to its real-time requirements and write-heavy nature. Timeline resolution is performed on-demand for accuracy.
</Note>

### Security

<Steps>
  <Step title="Database-level security">
    No direct entity manipulation - all writes occur via database triggers to prevent tampering.
  </Step>
  
  <Step title="Multi-layer tenant isolation">
    Tenant isolation is enforced at multiple layers (controller, service, database).
  </Step>
  
  <Step title="Sensitive data protection">
    Sensitive data is filtered at the database level before persistence to prevent exposure.
  </Step>
</Steps>
