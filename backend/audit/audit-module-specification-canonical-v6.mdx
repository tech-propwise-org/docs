## Overview

The audit module provides comprehensive activity tracking and timeline functionality for the PropWise CRM system. It captures all data changes across entities through PostgreSQL triggers and presents them as both raw audit logs and human-readable timeline events.

<Info>
**Last Updated:** 2024-01-15  
**Status:** Active
</Info>

The module supports both tenant-scoped and system-wide audit queries, ensuring complete visibility into system activities while maintaining proper access controls.

## Architecture

### Design Decisions

<CardGroup cols={2}>
  <Card title="Database-Driven Auditing" icon="database">
    Uses PostgreSQL triggers (`audit_trigger_func()`) to capture changes at the database level, ensuring consistency and completeness
  </Card>
  <Card title="Read-Only Entity" icon="lock">
    The `AuditLog` entity is strictly read-only from application code - all writes happen via database triggers
  </Card>
  <Card title="Timeline Abstraction" icon="timeline">
    Provides a higher-level timeline view that translates raw audit data into user-friendly messages
  </Card>
  <Card title="Dual Access Patterns" icon="users">
    Supports both tenant-scoped queries (for regular users) and system-wide queries (for system administrators)
  </Card>
</CardGroup>

### Key Components

- **AuditLog Entity**: Maps to the `audit_log` table (read-only)
- **AuditService**: Core business logic for querying and formatting audit data
- **AuditTimelineResolver**: Translates audit logs into human-readable timeline events
- **Dual Controllers**: Separate endpoints for tenant and system admin access

## Entity Descriptions

### AuditLog

The core entity that represents all audit trail records in the system.

```typescript
interface AuditLog {
  id: string;                    // UUID primary key
  tableName: string;             // Name of the table that changed
  action: AuditAction;           // INSERT, UPDATE, DELETE
  rowData: Record<string, any>;  // Full row data after change
  changedFields: Record<string, any>; // Only changed fields (UPDATE only)
  recordId: string;              // ID of the changed record
  tenantId: string;              // Tenant context
  userId?: string;               // User who made the change
  createdAt: Date;               // When the change occurred
}
```

<Note>
The `rowData` field contains the complete state of the record after the change, while `changedFields` (available only for UPDATE operations) shows just the fields that were modified.
</Note>

### AuditAction

Enumeration of possible audit actions:

```typescript
enum AuditAction {
  INSERT = 'INSERT',
  UPDATE = 'UPDATE', 
  DELETE = 'DELETE'
}
```

## API Endpoints

### Tenant-Scoped Endpoints

#### GET /audit/logs

Retrieve paginated audit logs for the current tenant.

<Warning>
This endpoint automatically filters results to the authenticated user's tenant. Users cannot access audit logs from other tenants.
</Warning>

**Query Parameters:**

```typescript
interface AuditLogQueryDto {
  page?: number;        // Page number (default: 1)
  limit?: number;       // Items per page (1-100, default: 20)
  tableName?: string;   // Filter by table name
  recordId?: string;    // Filter by record ID
  userId?: string;      // Filter by user ID
  action?: AuditAction; // Filter by action type
  startDate?: string;   // ISO date string
  endDate?: string;     // ISO date string
}
```

**Response:**

```typescript
interface PaginatedResponse<AuditLogDto> {
  data: AuditLogDto[];
  pagination: {
    page: number;
    limit: number;
    total: number;
    totalPages: number;
  };
}
```

**Example Request:**

```bash
GET /audit/logs?tableName=leads&page=1&limit=20&startDate=2024-01-01
```

**Example Response:**

```json
{
  "data": [
    {
      "id": "123e4567-e89b-12d3-a456-426614174000",
      "tableName": "leads",
      "action": "UPDATE",
      "rowData": {
        "id": "lead-123",
        "stage": "qualified",
        "score": 85
      },
      "changedFields": {
        "stage": "qualified",
        "score": 85
      },
      "recordId": "lead-123",
      "tenantId": "tenant-456",
      "userId": "user-789",
      "createdAt": "2024-01-15T10:30:00Z"
    }
  ],
  "pagination": {
    "page": 1,
    "limit": 20,
    "total": 150,
    "totalPages": 8
  }
}
```

#### GET /audit/timeline/:entityType/:entityId

Get resolved timeline events for a specific entity with human-readable messages.

**Parameters:**
- `entityType`: The type of entity (e.g., 'lead', 'deal', 'contact')
- `entityId`: UUID of the specific entity

**Response:**

```typescript
interface ResolvedTimeline {
  eventType: TimelineEventType;
  message: string;
  createdAt: Date;
  user?: AuditUserInfo;
}
```

**Example Request:**

```bash
GET /audit/timeline/lead/123e4567-e89b-12d3-a456-426614174000
```

**Example Response:**

```json
[
  {
    "eventType": "LEAD_STAGE_CHANGED",
    "message": "Lead moved to Qualified stage",
    "createdAt": "2024-01-15T10:30:00Z",
    "user": {
      "id": "user-789",
      "name": "John Doe",
      "email": "john@example.com"
    }
  },
  {
    "eventType": "LEAD_SCORE_CHANGED",
    "message": "Lead score updated to 85",
    "createdAt": "2024-01-15T10:30:00Z",
    "user": {
      "id": "user-789",
      "name": "John Doe",

    }

```

<Tip>
The timeline endpoint automatically resolves user information and formats messages for better readability. Use this endpoint for displaying activity feeds in the UI.
</Tip>

### System Admin Endpoints

#### GET /system-admin/audit/logs

System-wide audit log access for administrators.

<Check>
Requires SystemAdmin role. This endpoint bypasses tenant isolation and can access audit logs across all tenants.
</Check>

**Query Parameters:**

```typescript
interface SystemAdminAuditQueryDto extends AuditLogQueryDto {
  tenantId?: string;    // Filter by specific tenant
}
```

**Additional Features:**
- Access to all tenant data
- Optional tenant filtering
- Same pagination and filtering capabilities as tenant-scoped endpoint

**Example Request:**

```bash
GET /system-admin/audit/logs?tenantId=tenant-456&tableName=leads&page=1
```

## Business Rules and Invariants

### Data Integrity

<Steps>
  <Step title="Immutable Audit Logs">
    Audit logs are immutable once created by the database trigger. No application code can modify or delete audit records.
  </Step>
  <Step title="Sensitive Data Protection">
    Sensitive fields (passwords, refresh tokens) are automatically stripped by the trigger before persisting to the audit log.
  </Step>
  <Step title="Transactional Consistency">
    All audit operations must occur within the same transaction as the data change to ensure consistency.
  </Step>
</Steps>

### Access Control

<AccordionGroup>
  <Accordion title="Regular User Access">
    - Can only access audit logs for their own tenant
    - Cannot view system-wide audit data
    - User information resolution respects tenant boundaries
  </Accordion>
  <Accordion title="System Administrator Access">
    - Can access audit logs across all tenants
    - Can filter by specific tenant or view all
    - Requires SystemAdmin role decorator
  </Accordion>
  <Accordion title="Timeline Resolution">
    - Includes user information only if accessible to the requester
    - Respects tenant isolation for user data lookup
    - Gracefully handles missing user information
  </Accordion>
</AccordionGroup>

### Temporal Constraints

<Note>
All date operations follow strict validation and formatting rules:
</Note>

- Date range queries validate that `endDate >= startDate`
- Date filters use inclusive ranges (both start and end dates included)
- Timestamps are stored in UTC format
- Date parameters must be valid ISO 8601 strings

## Integration Points

### With Authentication Module

The audit module integrates tightly with the authentication system:

```typescript
// Uses @AuthGuard() for endpoint protection
@Get('logs')
@AuthGuard()
async getLogs(@GetCurrentUser() user: User) {
  // Automatically scoped to user's tenant
}

// Uses @SystemAdmin() decorator for admin access
@Get('logs')
@SystemAdmin()
async getSystemLogs() {
  // Bypasses tenant isolation
}
```

### With Tenant Context

<Info>
All queries are automatically scoped to the current tenant context (except system admin endpoints).
</Info>

- Tenant isolation is enforced at the service layer
- Uses the authenticated user's tenant ID for filtering
- Prevents cross-tenant data leakage
- System admin endpoints explicitly bypass tenant filtering

### With User Module

The audit module resolves user information for timeline display:

- Links audit entries to user accounts via `userId` field
- Fetches user details for timeline event attribution
- Handles cases where users may have been deleted
- Respects tenant boundaries when resolving user data

## Patterns and Conventions

### Query Patterns

The module follows consistent query patterns across all endpoints:

<Tabs>
  <Tab title="Pagination">
    ```typescript
    // Standard pagination using buildPaginatedResponse
    const result = await buildPaginatedResponse(
      query,
      page,
      limit,
      total
    );
    
    // Enforces max 100 items per page
    // Default page size: 20
    // Page numbers start at 1
    ```
  </Tab>
  <Tab title="Filtering">
    ```typescript
    // Flexible filtering with optional query parameters
    interface FilterOptions {
      tableName?: string;
      recordId?: string;
      userId?: string;
      action?: AuditAction;
      startDate?: string;
      endDate?: string;
    }
    
    // All filters are optional and can be combined
    ```
  </Tab>
  <Tab title="Type Safety">
    ```typescript
    // Uses MikroORM's FilterQuery for type-safe queries
    const filters: FilterQuery<AuditLog> = {
      tenantId: currentTenant.id,
      tableName: { $eq: tableName },
      createdAt: { 
        $gte: startDate,
        $lte: endDate 
      }
    };
    ```
  </Tab>
</Tabs>

### Error Handling

<Warning>
The module implements consistent error handling across all endpoints:
</Warning>

- Returns **404 Not Found** for non-existent entities in timeline queries
- Validates UUID format for ID parameters (returns 400 for invalid UUIDs)
- Enforces pagination limits with descriptive error messages
- Returns **403 Forbidden** for unauthorized access attempts
- Validates date ranges (returns 400 if endDate < startDate)

### Caching Considerations

<Note>
The audit module does **not** implement caching due to its specific characteristics:
</Note>

- **Write-heavy workload**: Audit logs are created on every data change
- **Real-time requirements**: Users expect immediate visibility of recent changes
- **Timeline resolution**: Performed on-demand to ensure fresh user information
- **Low read frequency**: Audit logs are queried less frequently than created

If caching becomes necessary, consider:
- Short TTL (< 60 seconds) for recent audit logs
- Cache invalidation on new audit entries for specific entities
- Separate caching strategy for timeline vs. raw audit logs

### Security

<Check>
The audit module implements defense-in-depth security:
</Check>

<Steps>
  <Step title="Database-Level Security">
    No direct entity manipulation - all writes occur via database triggers that run with elevated privileges
  </Step>
  <Step title="Application-Level Security">
    Tenant isolation enforced at multiple layers (controller, service, repository)
  </Step>
  <Step title="Data Protection">
    Sensitive data filtered at database level before persistence (passwords, tokens, etc.)
  </Step>
  <Step title="Access Control">
    Role-based access control with separate endpoints for regular users and system administrators
  </Step>
</Steps>

## Best Practices

### For Developers

<Tabs>
  <Tab title="Querying Audit Logs">
    ```typescript
    // ✅ DO: Use specific filters to limit result set
    const logs = await auditService.findAll({
      tableName: 'leads',
      recordId: leadId,
      startDate: lastWeek,
      limit: 50
    });
    
    // ❌ DON'T: Query without filters (performance impact)
    const allLogs = await auditService.findAll({});
    ```
  </Tab>
  <Tab title="Timeline Display">
    ```typescript
    // ✅ DO: Use timeline endpoint for UI display
    const timeline = await auditService.getTimeline('lead', leadId);
    
    // ❌ DON'T: Parse raw audit logs in the frontend
    const rawLogs = await auditService.findAll({ recordId: leadId });
    // Then manually format in UI
    ```
  </Tab>
  <Tab title="Date Handling">
    ```typescript
    // ✅ DO: Use ISO 8601 format for dates
    const logs = await auditService.findAll({
      startDate: '2024-01-01T00:00:00Z',
      endDate: '2024-01-31T23:59:59Z'
    });
    
    // ❌ DON'T: Use ambiguous date formats
    const logs = await auditService.findAll({
      startDate: '01/01/2024', // Ambiguous
      endDate: '1/31/24'       // Ambiguous
    });
    ```
  </Tab>
</Tabs>
