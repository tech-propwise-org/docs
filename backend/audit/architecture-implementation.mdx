## Overview

The audit module provides comprehensive activity tracking and timeline functionality for the PropWise CRM system. It captures all data changes across entities through PostgreSQL triggers and presents them as both raw audit logs and human-readable timeline events. The module supports both tenant-scoped and system-wide audit queries.

<Note>
The audit module uses database-level triggers to ensure all data changes are captured consistently, regardless of the application layer that initiates them.
</Note>

## Architecture

### Design Decisions

<AccordionGroup>
<Accordion title="Database-Driven Auditing">
Uses PostgreSQL triggers (`audit_trigger_func()`) to capture changes at the database level, ensuring consistency and completeness across all data modifications.
</Accordion>

<Accordion title="Read-Only Entity Pattern">
The `AuditLog` entity is strictly read-only from application code - all writes happen exclusively via database triggers to maintain data integrity.
</Accordion>

<Accordion title="Timeline Abstraction">
Provides a higher-level timeline view that translates raw audit data into user-friendly messages for better user experience.
</Accordion>

<Accordion title="Dual Access Patterns">
Supports both tenant-scoped queries (for regular users) and system-wide queries (for system administrators) with appropriate security controls.
</Accordion>
</AccordionGroup>

## Entity Specifications

### AuditLog Entity

The core audit tracking entity with the following structure:

```typescript
interface AuditLog {
  id: string;                    // UUID primary key
  tableName: string;             // Name of the table that changed
  action: AuditAction;           // INSERT, UPDATE, DELETE
  rowData: Record<string, any>;  // Full row data after change
  changedFields: Record<string, any>; // Only changed fields (UPDATE only)
  recordId: string;              // ID of the changed record
  tenantId: string;              // Tenant context
  userId?: string;               // User who made the change
  createdAt: Date;               // When the change occurred
}
```

<Warning>
The `AuditLog` entity is strictly read-only from application code. All writes occur through database triggers.
</Warning>

### Action Types

```typescript
enum AuditAction {
  INSERT = 'INSERT',
  UPDATE = 'UPDATE', 
  DELETE = 'DELETE'
}
```

## API Endpoints

### Tenant-Scoped Access

#### GET /audit/logs

Retrieve paginated audit logs for the current tenant.

<CodeGroup>
```typescript Query Parameters
interface AuditLogQueryDto {
  page?: number;        // Page number (default: 1)
  limit?: number;       // Items per page (1-100, default: 20)
  tableName?: string;   // Filter by table name
  recordId?: string;    // Filter by record ID
  userId?: string;      // Filter by user ID
  action?: AuditAction; // Filter by action type
  startDate?: string;   // ISO date string
  endDate?: string;     // ISO date string
}
```

```typescript Response
interface PaginatedResponse<AuditLogDto> {
  data: AuditLogDto[];
  pagination: {
    page: number;
    limit: number;
    total: number;
    totalPages: number;
  };
}
```
</CodeGroup>

#### GET /audit/timeline/:entityType/:entityId

Get resolved timeline events for a specific entity.

<Steps>
<Step title="Specify Entity">
Provide the `entityType` (e.g., 'lead', 'deal') and `entityId` (UUID) in the path parameters.
</Step>

<Step title="Receive Timeline">
Returns a chronologically ordered list of resolved timeline events for the specified entity.
</Step>
</Steps>

```typescript Response Format
interface ResolvedTimeline {
  eventType: TimelineEventType;
  message: string;
  createdAt: Date;
  user?: AuditUserInfo;
}
```

### System Admin Access

#### GET /system-admin/audit/logs

System-wide audit log access requiring SystemAdmin role.

<Info>
This endpoint extends the tenant-scoped query with an additional `tenantId` parameter for cross-tenant access.
</Info>

```typescript Extended Query Parameters
interface SystemAdminAuditQueryDto extends AuditLogQueryDto {
  tenantId?: string;    // Filter by specific tenant
}
```

## Business Rules & Invariants

### Data Integrity

<Check>
Audit logs are immutable once created by the database trigger
</Check>

<Check>
Sensitive fields (passwords, refresh tokens) are automatically stripped by the trigger
</Check>

<Check>
All audit operations must occur within the same transaction as the data change
</Check>

### Access Control Policies

<Steps>
<Step title="Tenant Isolation">
Regular users can only access audit logs for their own tenant through automatic scoping.
</Step>

<Step title="System Admin Access">
System administrators can access audit logs across all tenants with appropriate role validation.
</Step>

<Step title="User Information">
Timeline resolution includes user information only if accessible to the requester.
</Step>
</Steps>

### Temporal Constraints

- Date range queries validate that `endDate >= startDate`
- Date filters use inclusive ranges
- All timestamps are stored and queried in UTC

<Tip>
Use ISO 8601 format for date parameters: `YYYY-MM-DDTHH:mm:ss.sssZ`
</Tip>

## Integration Patterns

### Authentication Integration

The audit module integrates seamlessly with the authentication system:

```typescript
// Endpoint protection
@AuthGuard()

// Tenant context extraction
@GetCurrentUser()

// Admin role validation
@SystemAdmin()
```

### Tenant Context Management

<Warning>
All queries are automatically scoped to the current tenant except for system admin endpoints. Tenant isolation is enforced at the service layer.
</Warning>

### User Module Integration

- Resolves user information for timeline display
- Links audit entries to user accounts via `userId` field
- Handles user lookup failures gracefully

### Entity Module Monitoring

The audit system monitors changes across all major entities:

- **Leads**: Stage changes, qualification status, scoring
- **Deals**: Stage progression, win/loss status  
- **Contacts**: Creation and profile updates
- **Custom Entities**: Generic change tracking

### Error Handling Strategy

<AccordionGroup>
<Accordion title="Entity Not Found">
Returns HTTP 404 for non-existent entities in timeline queries with descriptive error messages.
</Accordion>

<Accordion title="Validation Errors">
Validates UUID format for ID parameters and enforces pagination limits (max 100 items per page).
</Accordion>

<Accordion title="Permission Errors">
Returns HTTP 403 for unauthorized access attempts with tenant isolation violations.
</Accordion>
</AccordionGroup>

### Performance Considerations

<Note>
The audit module is optimized for write-heavy, read-light workloads with the following characteristics:
</Note>

- **No Caching**: Due to real-time requirements and data sensitivity
- **On-Demand Resolution**: Timeline resolution performed when requested
- **Database Indexing**: Optimized indexes on `tenantId`, `recordId`, and `createdAt`

### Security Implementation

<Steps>
<Step title="Write Protection">
No direct entity manipulation allowed - all writes occur exclusively via database triggers.
</Step>

<Step title="Multi-Layer Isolation">
Tenant isolation enforced at database, service, and controller layers.
</Step>

<Step title="Data Sanitization">
Sensitive data automatically filtered at database level before persistence.
</Step>
</Steps>
