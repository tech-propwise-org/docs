## Overview

The audit module provides comprehensive activity tracking and timeline functionality for the PropWise CRM system. It captures all data changes across entities through PostgreSQL triggers and presents them as both raw audit logs and human-readable timeline events. The module supports both tenant-scoped and system-wide audit queries.

<Info>
The audit module is designed to be completely transparent to application code - all data capture happens automatically at the database level through triggers.
</Info>

## Architecture

### Design Decisions

<CardGroup cols={2}>
  <Card title="Database-Driven Auditing" icon="database">
    Uses PostgreSQL triggers (`audit_trigger_func()`) to capture changes at the database level, ensuring consistency and completeness
  </Card>
  <Card title="Read-Only Entity" icon="lock">
    The `AuditLog` entity is strictly read-only from application code - all writes happen via database triggers
  </Card>
  <Card title="Timeline Abstraction" icon="timeline">
    Provides a higher-level timeline view that translates raw audit data into user-friendly messages
  </Card>
  <Card title="Dual Access Patterns" icon="users">
    Supports both tenant-scoped queries (for regular users) and system-wide queries (for system administrators)
  </Card>
</CardGroup>

### Key Components

- **AuditLog Entity**: Maps to the `audit_log` table (read-only)
- **AuditService**: Core business logic for querying and formatting audit data
- **AuditTimelineResolver**: Translates audit logs into human-readable timeline events
- **Dual Controllers**: Separate endpoints for tenant and system admin access

## Entity Specifications

### AuditLog Entity

<CodeGroup>
```typescript AuditLog Interface
interface AuditLog {
  id: string;                    // UUID primary key
  tableName: string;             // Name of the table that changed
  action: AuditAction;           // INSERT, UPDATE, DELETE
  rowData: Record<string, any>;  // Full row data after change
  changedFields: Record<string, any>; // Only changed fields (UPDATE only)
  recordId: string;              // ID of the changed record
  tenantId: string;              // Tenant context
  userId?: string;               // User who made the change
  createdAt: Date;               // When the change occurred
}
```

```typescript AuditAction Enum
enum AuditAction {
  INSERT = 'INSERT',
  UPDATE = 'UPDATE', 
  DELETE = 'DELETE'
}
```
</CodeGroup>

<Warning>
The AuditLog entity is strictly read-only from application code. All writes are performed automatically by PostgreSQL triggers.
</Warning>

## API Endpoints

### Tenant-Scoped Endpoints

#### GET /audit/logs

Retrieve paginated audit logs for the current tenant.

<CodeGroup>
```typescript Query Parameters
interface AuditLogQueryDto {
  page?: number;        // Page number (default: 1)
  limit?: number;       // Items per page (1-100, default: 20)
  tableName?: string;   // Filter by table name
  recordId?: string;    // Filter by record ID
  userId?: string;      // Filter by user ID
  action?: AuditAction; // Filter by action type
  startDate?: string;   // ISO date string
  endDate?: string;     // ISO date string
}
```

```typescript Response
interface PaginatedResponse<AuditLogDto> {
  data: AuditLogDto[];
  pagination: {
    page: number;
    limit: number;
    total: number;
    totalPages: number;
  };
}
```

```bash Example Request
curl -X GET "/audit/logs?tableName=leads&action=UPDATE&page=1&limit=20" \
  -H "Authorization: Bearer {token}"
```
</CodeGroup>

<Note>
All audit log queries are automatically scoped to the authenticated user's tenant context.
</Note>

#### GET /audit/timeline/:entityType/:entityId

Get resolved timeline events for a specific entity.

<CodeGroup>
```typescript Parameters
// URL Parameters
entityType: string; // The type of entity (e.g., 'lead', 'deal')
entityId: string;   // UUID of the specific entity
```

```typescript Response
interface ResolvedTimeline {
  eventType: TimelineEventType;
  message: string;
  createdAt: Date;
  user?: AuditUserInfo;
}
```

```bash Example Request
curl -X GET "/audit/timeline/lead/123e4567-e89b-12d3-a456-426614174000" \
  -H "Authorization: Bearer {token}"
```
</CodeGroup>

## Business Rules and Invariants

### Data Integrity

<Steps>
  <Step title="Immutable Audit Logs">
    Audit logs are immutable once created by the database trigger
  </Step>
  <Step title="Sensitive Data Protection">
    Sensitive fields (passwords, refresh tokens) are automatically stripped by the trigger
  </Step>
  <Step title="Transactional Consistency">
    All audit operations must occur within the same transaction as the data change
  </Step>
</Steps>

### Access Control

<Tabs>
  <Tab title="Regular Users">
    - Can only access audit logs for their own tenant
    - Timeline resolution includes accessible user information
    - All queries automatically tenant-scoped
  </Tab>
  
  <Tab title="System Administrators">
    - Can access audit logs across all tenants
    - Requires SystemAdmin role
    - Can filter by specific tenant or view system-wide
  </Tab>
</Tabs>

### Temporal Constraints

<Check>Date range queries validate that `endDate >= startDate`</Check>
<Check>Date filters use inclusive ranges</Check>
<Check>Timestamps are stored in UTC</Check>

## Patterns and Conventions

### Query Patterns

<Tip>
The module implements standard pagination using `buildPaginatedResponse` for consistent API responses across the system.
</Tip>

- Supports flexible filtering with optional query parameters
- Uses MikroORM's FilterQuery for type-safe database queries
- Enforces pagination limits (max 100 items per page)

### Error Handling

<AccordionGroup>
  <Accordion title="Timeline Queries">
    Returns 404 for non-existent entities in timeline queries
  </Accordion>
  
  <Accordion title="Parameter Validation">
    Validates UUID format for ID parameters
  </Accordion>
  
  <Accordion title="Pagination Limits">
    Enforces pagination limits (max 100 items per page)
  </Accordion>
</AccordionGroup>

### Caching Considerations

<Info>
No caching is implemented for audit data due to real-time requirements and the write-heavy nature of audit logs.
</Info>

- Audit data is write-heavy, read-light
- Timeline resolution performed on-demand
- Database-level optimizations handle performance

### Security

<CardGroup cols={2}>
  <Card title="No Direct Writes" icon="ban">
    No direct entity manipulation - all writes via database triggers
  </Card>
  <Card title="Multi-Layer Isolation" icon="layers">
    Tenant isolation enforced at multiple layers
  </Card>
  <Card title="Database-Level Filtering" icon="filter">
    Sensitive data filtered at database level before persistence
  </Card>
</CardGroup>

<Warning>
The audit system is designed to be tamper-proof. All audit entries are created automatically by database triggers and cannot be modified or deleted through application code.
