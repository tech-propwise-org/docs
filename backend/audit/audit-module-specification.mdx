The audit module provides comprehensive activity tracking and timeline functionality for the PropWise CRM system. It captures all data changes across entities through PostgreSQL triggers and presents them as both raw audit logs and human-readable timeline events.

<Info>
**Last Updated:** 2024-01-15  
**Status:** Active
</Info>

## Architecture Overview

The audit module follows a database-driven approach to ensure comprehensive and consistent activity tracking across all system entities.

### Design Decisions

<CardGroup cols={2}>
  <Card title="Database-Driven Auditing" icon="database">
    Uses PostgreSQL triggers (`audit_trigger_func()`) to capture changes at the database level, ensuring consistency and completeness
  </Card>
  <Card title="Read-Only Entity" icon="lock">
    The `AuditLog` entity is strictly read-only from application code - all writes happen via database triggers
  </Card>
  <Card title="Timeline Abstraction" icon="timeline">
    Provides a higher-level timeline view that translates raw audit data into user-friendly messages
  </Card>
  <Card title="Dual Access Patterns" icon="users">
    Supports both tenant-scoped queries (for regular users) and system-wide queries (for system administrators)
  </Card>
</CardGroup>

### Key Components

- **AuditLog Entity**: Maps to the `audit_log` table (read-only)
- **AuditService**: Core business logic for querying and formatting audit data
- **AuditTimelineResolver**: Translates audit logs into human-readable timeline events
- **Dual Controllers**: Separate endpoints for tenant and system admin access

## Entity Specifications

### AuditLog Entity

The core entity that captures all data changes across the system:

```typescript
interface AuditLog {
  id: string;                    // UUID primary key
  tableName: string;             // Name of the table that changed
  action: AuditAction;           // INSERT, UPDATE, DELETE
  rowData: Record<string, any>;  // Full row data after change
  changedFields: Record<string, any>; // Only changed fields (UPDATE only)
  recordId: string;              // ID of the changed record
  tenantId: string;              // Tenant context
  userId?: string;               // User who made the change
  createdAt: Date;               // When the change occurred
}
```

<Note>
The `AuditLog` entity is read-only from application code. All writes are performed automatically by database triggers.
</Note>

### AuditAction Enum

Defines the types of database operations that can be audited:

```typescript
enum AuditAction {
  INSERT = 'INSERT',
  UPDATE = 'UPDATE', 
  DELETE = 'DELETE'
}
```

## API Endpoints

### Tenant-Scoped Endpoints

These endpoints provide access to audit data within the current tenant's scope.

#### GET /audit/logs

Retrieve paginated audit logs for the current tenant.

**Query Parameters:**

```typescript
interface AuditLogQueryDto {
  page?: number;        // Page number (default: 1)
  limit?: number;       // Items per page (1-100, default: 20)
  tableName?: string;   // Filter by table name
  recordId?: string;    // Filter by record ID
  userId?: string;      // Filter by user ID
  action?: AuditAction; // Filter by action type
  startDate?: string;   // ISO date string
  endDate?: string;     // ISO date string
}
```

**Response:**

```typescript
interface PaginatedResponse<AuditLogDto> {
  data: AuditLogDto[];
  pagination: {
    page: number;
    limit: number;
    total: number;
    totalPages: number;
  };
}
```

<Tip>
Use the `tableName` and `recordId` parameters together to get all audit history for a specific entity.
</Tip>

#### GET /audit/timeline/:entityType/:entityId

Get resolved timeline events for a specific entity with human-readable messages.

**Parameters:**
- `entityType`: The type of entity (e.g., 'lead', 'deal')
- `entityId`: UUID of the specific entity

**Response:**

```typescript
interface ResolvedTimeline {
  eventType: TimelineEventType;
  message: string;
  createdAt: Date;
  user?: AuditUserInfo;
}
```

### System Admin Endpoints

<Warning>
System admin endpoints require `SystemAdmin` role and provide access to audit data across all tenants.
</Warning>

#### GET /system-admin/audit/logs

System-wide audit log access for administrators.

**Query Parameters:**

```typescript
interface SystemAdminAuditQueryDto extends AuditLogQueryDto {
  tenantId?: string;    // Filter by specific tenant
}
```

The response format is identical to the tenant-scoped endpoint, but can include data from multiple tenants.

## Business Rules and Invariants

### Data Integrity

<Steps>
  <Step title="Immutable Audit Logs">
    Audit logs are immutable once created by the database trigger and cannot be modified by application code.
  </Step>
  
  <Step title="Sensitive Data Filtering">
    Sensitive fields (passwords, refresh tokens) are automatically stripped by the trigger before persistence.
  </Step>
  
  <Step title="Transactional Consistency">
    All audit operations must occur within the same transaction as the data change to ensure consistency.
  </Step>
</Steps>

### Access Control

- **Regular users** can only access audit logs for their own tenant
- **System administrators** can access audit logs across all tenants
- Timeline resolution includes user information only if accessible to the requester

### Temporal Constraints

<Check>
Date range queries validate that `endDate >= startDate`
</Check>

<Check>
Date filters use inclusive ranges for precise querying
</Check>

<Check>
All timestamps are stored in UTC for consistency
</Check>

## Implementation Patterns

### Query Patterns

The module implements several standard patterns for data access:

```typescript
// Standard pagination
buildPaginatedResponse(auditLogs, total, page, limit)

// Flexible filtering with optional parameters
const filterQuery: FilterQuery<AuditLog> = {
  tenantId: user.tenantId,
  ...(tableName && { tableName }),
  ...(recordId && { recordId }),
  ...(userId && { userId }),
  ...(action && { action })
};

// Type-safe database queries with MikroORM
await this.auditLogRepository.findAndCount(filterQuery, {
  orderBy: { createdAt: QueryOrder.DESC },
  limit,
  offset: (page - 1) * limit
});
```

### Error Handling

<Warning>
The module implements comprehensive error handling for common scenarios:
</Warning>

- Returns **404** for non-existent entities in timeline queries
- Validates **UUID format** for ID parameters
- Enforces **pagination limits** (max 100 items per page)
- Validates **date range parameters** for temporal queries

### Performance Considerations

- **Write-Heavy, Read-Light**: Audit data is optimized for frequent writes with occasional reads
- **No Caching**: Real-time requirements prevent aggressive caching strategies
- **On-Demand Resolution**: Timeline resolution performed only when requested
- **Indexed Queries**: Database indexes on common filter fields (tenantId, recordId, createdAt)

<Info>
The audit module is designed to provide comprehensive tracking without impacting system performance through careful architectural decisions and optimized query patterns.
