## Overview

The audit module provides comprehensive activity tracking and timeline functionality for the PropWise CRM system. It captures all data changes across entities through PostgreSQL triggers and presents them as both raw audit logs and human-readable timeline events.

<Info>
The module supports both tenant-scoped and system-wide audit queries, ensuring proper data isolation while enabling administrative oversight.
</Info>

**Status:** Active  
**Last Updated:** January 15, 2024

## Architecture

### Design Decisions

The audit module is built on several key architectural principles:

<CardGroup cols={2}>
  <Card title="Database-Driven Auditing" icon="database">
    Uses PostgreSQL triggers (`audit_trigger_func()`) to capture changes at the database level, ensuring consistency and completeness
  </Card>
  <Card title="Read-Only Entity" icon="lock">
    The `AuditLog` entity is strictly read-only from application code - all writes happen via database triggers
  </Card>
  <Card title="Timeline Abstraction" icon="timeline">
    Provides a higher-level timeline view that translates raw audit data into user-friendly messages
  </Card>
  <Card title="Dual Access Patterns" icon="users">
    Supports both tenant-scoped queries (for regular users) and system-wide queries (for system administrators)
  </Card>
</CardGroup>

### Key Components

- **AuditLog Entity**: Maps to the `audit_log` table (read-only)
- **AuditService**: Core business logic for querying and formatting audit data
- **AuditTimelineResolver**: Translates audit logs into human-readable timeline events
- **Dual Controllers**: Separate endpoints for tenant and system admin access

## Entity Specifications

### AuditLog Entity

<CodeGroup>
```typescript AuditLog Interface
interface AuditLog {
  id: string;                    // UUID primary key
  tableName: string;             // Name of the table that changed
  action: AuditAction;           // INSERT, UPDATE, DELETE
  rowData: Record<string, any>;  // Full row data after change
  changedFields: Record<string, any>; // Only changed fields (UPDATE only)
  recordId: string;              // ID of the changed record
  tenantId: string;              // Tenant context
  userId?: string;               // User who made the change
  createdAt: Date;               // When the change occurred
}
```

```typescript AuditAction Enum
enum AuditAction {
  INSERT = 'INSERT',
  UPDATE = 'UPDATE', 
  DELETE = 'DELETE'
}
```
</CodeGroup>

### Timeline Event Types

Timeline events provide semantic classification of audit activities:

<AccordionGroup>
  <Accordion title="Lead Events">
    - `LEAD_CREATED`: New lead record created
    - `LEAD_STAGE_CHANGED`: Lead moved between pipeline stages
    - `LEAD_QUALIFIED`: Lead marked as qualified
    - `LEAD_DONE`: Lead marked as complete
    - `LEAD_DISQUALIFIED`: Lead marked as disqualified
    - `LEAD_SCORE_CHANGED`: Lead scoring updated
  </Accordion>
  
  <Accordion title="Deal Events">
    - `DEAL_CREATED`: New deal record created
    - `DEAL_STAGE_CHANGED`: Deal moved between pipeline stages
    - `DEAL_WON`: Deal marked as won
    - `DEAL_LOST`: Deal marked as lost
  </Accordion>
  
  <Accordion title="Contact Events">
    - `CONTACT_CREATED`: New contact record created
    - `CONTACT_UPDATED`: Contact information modified
  </Accordion>
  
  <Accordion title="Generic Events">
    - `RECORD_CREATED`: Generic record creation
    - `RECORD_UPDATED`: Generic record modification
    - `RECORD_DELETED`: Generic record deletion
  </Accordion>
</AccordionGroup>

## API Endpoints

### Tenant-Scoped Endpoints

#### GET /audit/logs

Retrieve paginated audit logs for the current tenant.

<CodeGroup>
```typescript Query Parameters
interface AuditLogQueryDto {
  page?: number;        // Page number (default: 1)
  limit?: number;       // Items per page (1-100, default: 20)
  tableName?: string;   // Filter by table name
  recordId?: string;    // Filter by record ID
  userId?: string;      // Filter by user ID
  action?: AuditAction; // Filter by action type
  startDate?: string;   // ISO date string
  endDate?: string;     // ISO date string
}
```

```typescript Response Format
interface PaginatedResponse<AuditLogDto> {
  data: AuditLogDto[];
  pagination: {
    page: number;
    limit: number;
    total: number;
    totalPages: number;
  };
}
```
</CodeGroup>

#### GET /audit/timeline/:entityType/:entityId

Get resolved timeline events for a specific entity.

**Parameters:**
- `entityType`: The type of entity (e.g., 'lead', 'deal')
- `entityId`: UUID of the specific entity

<CodeGroup>
```typescript Response Format
interface ResolvedTimeline {
  eventType: TimelineEventType;
  message: string;
  createdAt: Date;
  user?: AuditUserInfo;
}
```

```typescript Example Response
{
  "eventType": "LEAD_STAGE_CHANGED",

  "createdAt": "2024-01-15T10:30:00Z",

    "id": "123e4567-e89b-12d3-a456-426614174000",
    "name": "John Smith",
    "email": "john.smith@company.com"

}
```
</CodeGroup>

### System Admin Endpoints

#### GET /system-admin/audit/logs

System-wide audit log access (requires SystemAdmin role).

## Business Rules and Invariants

### Data Integrity

<Check>
**Immutable Audit Logs**: Audit logs are immutable once created by the database trigger
</Check>

<Check>
**Automatic Data Sanitization**: Sensitive fields (passwords, refresh tokens) are automatically stripped by the trigger
</Check>

<Check>
**Transactional Consistency**: All audit operations must occur within the same transaction as the data change
</Check>

### Access Control

<Steps>
  <Step title="Tenant Isolation">
    Regular users can only access audit logs for their own tenant
  </Step>
  <Step title="System Admin Access">
    System administrators can access audit logs across all tenants
  </Step>
  <Step title="User Information Resolution">
    Timeline resolution includes user information only if accessible to the requester
  </Step>
</Steps>

### Temporal Constraints

<Warning>
Date range queries validate that `endDate >= startDate` and use inclusive ranges. All timestamps are stored in UTC.
</Warning>

### Security Considerations

<Warning>
The audit module implements multiple security layers to protect sensitive information and ensure data integrity.
</Warning>

- **No Direct Manipulation**: All writes occur via database triggers, preventing application-level tampering
- **Multi-Layer Tenant Isolation**: Enforced at database, service, and controller layers
- **Sensitive Data Filtering**: Automatic removal of sensitive data at the database level before persistence
- **Role-Based Access**: Strict separation between tenant-scoped and system admin access

### Performance and Caching

<Tip>
The audit module is optimized for write-heavy, read-light workloads with real-time requirements.
</Tip>

- **No Caching**: Timeline resolution performed on-demand due to real-time requirements
- **Optimized Queries**: Database indexes on frequently queried fields (`tenantId`, `tableName`, `recordId`)
- **Pagination**: Enforced limits prevent excessive memory usage on large datasets
