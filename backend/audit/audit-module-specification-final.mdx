## Overview

The audit module provides comprehensive activity tracking and timeline functionality for the PropWise CRM system. It captures all data changes across entities through PostgreSQL triggers and presents them as both raw audit logs and human-readable timeline events. The module supports both tenant-scoped and system-wide audit queries.

<Note>
The audit module uses database-level triggers to ensure complete and consistent tracking of all data changes, making it impossible to bypass audit logging through application-level modifications.
</Note>

## Architecture

### Design Decisions

The audit module is built on several key architectural principles:

<CardGroup cols={2}>
  <Card title="Database-Driven Auditing" icon="database">
    Uses PostgreSQL triggers (`audit_trigger_func()`) to capture changes at the database level, ensuring consistency and completeness
  </Card>
  <Card title="Read-Only Entity" icon="lock">
    The `AuditLog` entity is strictly read-only from application code - all writes happen via database triggers
  </Card>
  <Card title="Timeline Abstraction" icon="timeline">
    Provides a higher-level timeline view that translates raw audit data into user-friendly messages
  </Card>
  <Card title="Dual Access Patterns" icon="users">
    Supports both tenant-scoped queries (for regular users) and system-wide queries (for system administrators)
  </Card>
</CardGroup>

### Key Components

- **AuditLog Entity**: Maps to the `audit_log` table (read-only)
- **AuditService**: Core business logic for querying and formatting audit data
- **AuditTimelineResolver**: Translates audit logs into human-readable timeline events
- **Dual Controllers**: Separate endpoints for tenant and system admin access

## Entity Descriptions

### AuditLog

The core audit entity that captures all data changes in the system:

```typescript
interface AuditLog {
  id: string;                    // UUID primary key
  tableName: string;             // Name of the table that changed
  action: AuditAction;           // INSERT, UPDATE, DELETE
  rowData: Record<string, any>;  // Full row data after change
  changedFields: Record<string, any>; // Only changed fields (UPDATE only)
  recordId: string;              // ID of the changed record
  tenantId: string;              // Tenant context
  userId?: string;               // User who made the change
  createdAt: Date;               // When the change occurred
}
```

<Warning>
The `AuditLog` entity is strictly read-only from application code. All audit entries are created automatically by database triggers.
</Warning>

### AuditAction

Enumeration of possible audit actions:

```typescript
enum AuditAction {
  INSERT = 'INSERT',
  UPDATE = 'UPDATE', 
  DELETE = 'DELETE'
}
```

### TimelineEventType

Semantic classification of audit events for user-friendly display:

## API Endpoints

### Tenant-Scoped Endpoints

#### GET /audit/logs

Retrieve paginated audit logs for the current tenant.

<Tabs>
  <Tab title="Parameters">
    ```typescript
    interface AuditLogQueryDto {
      page?: number;        // Page number (default: 1)
      limit?: number;       // Items per page (1-100, default: 20)
      tableName?: string;   // Filter by table name
      recordId?: string;    // Filter by record ID
      userId?: string;      // Filter by user ID
      action?: AuditAction; // Filter by action type
      startDate?: string;   // ISO date string
      endDate?: string;     // ISO date string
    }
    ```
  </Tab>
  
  <Tab title="Response">
    ```typescript
    interface PaginatedResponse<AuditLogDto> {
      data: AuditLogDto[];
      pagination: {
        page: number;
        limit: number;
        total: number;
        totalPages: number;
      };
    }
    ```
  </Tab>
  
  <Tab title="Example">
    ```bash
    GET /audit/logs?tableName=leads&action=UPDATE&page=1&limit=10
    ```
  </Tab>
</Tabs>

#### GET /audit/timeline/:entityType/:entityId

Get resolved timeline events for a specific entity.

**Parameters:**
- `entityType`: The type of entity (e.g., 'lead', 'deal')
- `entityId`: UUID of the specific entity

**Response:**
```typescript
interface ResolvedTimeline {
  eventType: TimelineEventType;
  message: string;
  createdAt: Date;
  user?: AuditUserInfo;
}
```

<Tip>
Timeline events provide human-readable messages that are perfect for displaying activity feeds in the user interface.
</Tip>

### System Admin Endpoints

#### GET /system-admin/audit/logs

System-wide audit log access (requires SystemAdmin role).

```typescript
interface SystemAdminAuditQueryDto extends AuditLogQueryDto {
  tenantId?: string;    // Filter by specific tenant
}
```

<Warning>
System admin endpoints require the `SystemAdmin` role and provide access to audit data across all tenants. Use with caution.
</Warning>

## Business Rules and Invariants

### Data Integrity

<Steps>
  <Step title="Immutable Records">
    Audit logs are immutable once created by the database trigger
  </Step>
  <Step title="Sensitive Data Filtering">
    Sensitive fields (passwords, refresh tokens) are automatically stripped by the trigger
  </Step>
  <Step title="Transactional Consistency">
    All audit operations must occur within the same transaction as the data change
  </Step>
</Steps>

### Access Control

- **Regular Users**: Can only access audit logs for their own tenant
- **System Administrators**: Can access audit logs across all tenants
- **Timeline Resolution**: Includes user information only if accessible to the requester

### Temporal Constraints

<Info>
All timestamps are stored in UTC for consistency across time zones.
</Info>

- Date range queries validate that `endDate >= startDate`
- Date filters use inclusive ranges
- Maximum pagination limit is 100 items per page

  @GetCurrentUser() user: UserContext,

  // Implementation automatically scoped to user's tenant

### With Tenant Context

<Note>
All queries are automatically scoped to the current tenant except for system admin endpoints, ensuring proper data isolation.
</Note>

### With User Module

- Resolves user information for timeline display
- Links audit entries to user accounts via `userId` field
- Handles user anonymization for deleted accounts

### With Entity Modules

The audit module monitors changes across all major entities:

- **Leads**: Complete lifecycle tracking from creation to closure
- **Deals**: Pipeline progression and outcome tracking  
- **Contacts**: Contact information changes and relationship updates
- **Properties**: Property details and status changes

### Error Handling

- Returns **404** for non-existent entities in timeline queries
- Validates UUID format for ID parameters
- Enforces pagination limits with clear error messages

### Security

<Check>
**Security Features**
- No direct entity manipulation - all writes via database triggers
- Tenant isolation enforced at multiple layers
- Sensitive data filtered at database level before persistence
- Role-based access control for system admin features
</Check>

### Performance Considerations

- **Write-Heavy, Read-Light**: Optimized for high-volume audit writing
- **No Caching**: Real-time requirements prevent aggressive caching
- **On-Demand Resolution**: Timeline formatting performed when requested
- **Indexed Queries**: Database indexes on common filter fields
