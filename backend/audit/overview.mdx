The audit module provides comprehensive activity tracking and timeline functionality for the PropWise CRM system. It captures all data changes across entities through PostgreSQL triggers and presents them as both raw audit logs and human-readable timeline events.

<Note>
All audit data is captured automatically at the database level using PostgreSQL triggers, ensuring complete consistency and preventing data loss.
</Note>

## Architecture

### Design Philosophy

The audit module follows these key architectural principles:

<AccordionGroup>
  <Accordion title="Database-Driven Auditing">
    Uses PostgreSQL triggers (`audit_trigger_func()`) to capture changes at the database level, ensuring consistency and completeness across all operations.
  </Accordion>
  
  <Accordion title="Read-Only Entity Pattern">
    The `AuditLog` entity is strictly read-only from application code - all writes happen via database triggers, preventing accidental data corruption.
  </Accordion>
  
  <Accordion title="Timeline Abstraction">
    Provides a higher-level timeline view that translates raw audit data into user-friendly messages for better user experience.
  </Accordion>
  
  <Accordion title="Dual Access Patterns">
    Supports both tenant-scoped queries (for regular users) and system-wide queries (for system administrators).
  </Accordion>
</AccordionGroup>

## Core Entities

### AuditLog Entity

```typescript
interface AuditLog {
  id: string;                    // UUID primary key
  tableName: string;             // Name of the table that changed
  action: AuditAction;           // INSERT, UPDATE, DELETE
  rowData: Record<string, any>;  // Full row data after change
  changedFields: Record<string, any>; // Only changed fields (UPDATE only)
  recordId: string;              // ID of the changed record
  tenantId: string;              // Tenant context
  userId?: string;               // User who made the change
  createdAt: Date;               // When the change occurred
}
```

## Implementation Guide

### Setting Up Audit Tracking

<Steps>
  <Step title="Enable Database Triggers">
    Ensure PostgreSQL triggers are properly configured for all audited tables:
    ```sql
    CREATE TRIGGER audit_trigger_update
    AFTER INSERT OR UPDATE OR DELETE ON your_table
    FOR EACH ROW EXECUTE FUNCTION audit_trigger_func();
    ```
  </Step>
  
  <Step title="Configure Entity Access">
    Import and inject the AuditService:
    ```typescript
    import { AuditService } from './audit.service';
    
    @Controller()
    export class YourController {
      constructor(private readonly auditService: AuditService) {}
    }
    ```
  </Step>
  
  <Step title="Query Audit Data">
    Use the service methods to retrieve audit information:
    ```typescript
    const logs = await this.auditService.findLogs(tenantId, queryDto);
    const timeline = await this.auditService.getTimeline(entityType, entityId, tenantId);
    ```
  </Step>
</Steps>

### Timeline Resolution

The timeline resolver automatically translates raw audit data into human-readable messages:

```typescript
// Raw audit log
{
  tableName: "leads",
  action: "UPDATE", 
  changedFields: { stage: "qualified" },
  userId: "user123"
}

// Becomes timeline event
{
  eventType: "LEAD_STAGE_CHANGED",
  message: "Lead stage changed from 'new' to 'qualified'",
  user: { name: "John Doe", email: "john@example.com" }
}
```

## Business Rules

### Data Integrity

<Check>All audit logs are immutable once created by the database trigger</Check>
<Check>Sensitive fields (passwords, refresh tokens) are automatically stripped</Check>
<Check>All audit operations occur within the same transaction as the data change</Check>

### Access Control

- **Regular users**: Can only access audit logs for their own tenant
- **System administrators**: Can access audit logs across all tenants  
- **Timeline resolution**: Includes user information only if accessible to the requester

### Temporal Constraints

<Info>
Date range queries validate that `endDate >= startDate` and use inclusive ranges. All timestamps are stored in UTC.
</Info>

## Integration Points

### Authentication & Authorization
- Uses `@AuthGuard()` for endpoint protection
- Leverages `@GetCurrentUser()` for tenant context
- Integrates with `@SystemAdmin()` decorator for admin access

### Multi-Tenancy
- All queries automatically scoped to current tenant (except system admin)
- Tenant isolation enforced at the service layer

### User Resolution
- Resolves user information for timeline display
- Links audit entries to user accounts via `userId` field

### Entity Monitoring
- Monitors changes across all major entities (leads, deals, contacts, etc.)
- Uses entity type resolution for timeline formatting

## Performance Considerations

<Tip>
The audit module is optimized for write-heavy, read-light patterns. Timeline resolution is performed on-demand without caching due to real-time requirements.
</Tip>

### Query Optimization
- Implements standard pagination with a maximum of 100 items per page
- Supports flexible filtering to reduce data transfer
- Uses indexed queries for efficient database access

### Security Features
- No direct entity manipulation - all writes via database triggers
- Tenant isolation enforced at multiple layers
- Sensitive data filtered at database level before persistence
