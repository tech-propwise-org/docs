## Overview

The audit module provides comprehensive activity tracking and timeline functionality for the PropWise CRM system. It captures all data changes across entities through PostgreSQL triggers and presents them as both raw audit logs and human-readable timeline events. The module supports both tenant-scoped and system-wide audit queries.

<Note>
The audit module is designed to be completely read-only from application code. All data writes occur through database triggers, ensuring consistency and preventing accidental modifications.
</Note>

## Architecture

### Design Decisions

The audit module follows several key architectural principles to ensure reliability and performance:

<Steps>
<Step title="Database-Driven Auditing">
Uses PostgreSQL triggers (`audit_trigger_func()`) to capture changes at the database level, ensuring consistency and completeness
</Step>

<Step title="Read-Only Entity Pattern">
The `AuditLog` entity is strictly read-only from application code - all writes happen via database triggers
</Step>

<Step title="Timeline Abstraction Layer">
Provides a higher-level timeline view that translates raw audit data into user-friendly messages
</Step>

<Step title="Dual Access Patterns">
Supports both tenant-scoped queries (for regular users) and system-wide queries (for system administrators)
</Step>
</Steps>

## Entity Specifications

### AuditLog Entity

The core audit log entity captures all database changes:

```typescript
interface AuditLog {
  id: string;                    // UUID primary key
  tableName: string;             // Name of the table that changed
  action: AuditAction;           // INSERT, UPDATE, DELETE
  rowData: Record<string, any>;  // Full row data after change
  changedFields: Record<string, any>; // Only changed fields (UPDATE only)
  recordId: string;              // ID of the changed record
  tenantId: string;              // Tenant context
  userId?: string;               // User who made the change
  createdAt: Date;               // When the change occurred
}
```

<Warning>
The `AuditLog` entity is strictly read-only. Any attempts to modify audit logs through the application layer will fail.
</Warning>

### AuditAction Enum

```typescript
enum AuditAction {
  INSERT = 'INSERT',
  UPDATE = 'UPDATE', 
  DELETE = 'DELETE'
}
```

## API Endpoints

<Tabs>
<Tab title="Tenant-Scoped">

### GET /audit/logs

Retrieve paginated audit logs for the current tenant.

**Query Parameters:**

```typescript
interface AuditLogQueryDto {
  page?: number;        // Page number (default: 1)
  limit?: number;       // Items per page (1-100, default: 20)
  tableName?: string;   // Filter by table name
  recordId?: string;    // Filter by record ID
  userId?: string;      // Filter by user ID
  action?: AuditAction; // Filter by action type
  startDate?: string;   // ISO date string
  endDate?: string;     // ISO date string
}
```

**Response:**

```typescript
interface PaginatedResponse<AuditLogDto> {
  data: AuditLogDto[];
  pagination: {
    page: number;
    limit: number;
    total: number;
    totalPages: number;
  };
}
```

### GET /audit/timeline/:entityType/:entityId

Get resolved timeline events for a specific entity.

**Parameters:**
- `entityType`: The type of entity (e.g., 'lead', 'deal')
- `entityId`: UUID of the specific entity

**Response:**

```typescript
interface ResolvedTimeline {
  eventType: TimelineEventType;
  message: string;
  createdAt: Date;
  user?: AuditUserInfo;
}
```

</Tab>
<Tab title="System Admin">

### GET /system-admin/audit/logs

System-wide audit log access (requires SystemAdmin role).

<Warning>
This endpoint is restricted to system administrators and provides cross-tenant access to audit data.
</Warning>

**Query Parameters:**

```typescript
interface SystemAdminAuditQueryDto extends AuditLogQueryDto {
  tenantId?: string;    // Filter by specific tenant
}
```

**Authentication Requirements:**
- Must have `SystemAdmin` role
- Uses `@SystemAdmin()` decorator for authorization

</Tab>
</Tabs>

## Business Rules and Invariants

### Data Integrity

<Check>
Audit logs are immutable once created by the database trigger
</Check>

<Check>
Sensitive fields (passwords, refresh tokens) are automatically stripped by the trigger
</Check>

<Check>
All audit operations must occur within the same transaction as the data change
</Check>

### Access Control

<Info>
Regular users can only access audit logs for their own tenant, while system administrators can access audit logs across all tenants.
</Info>

- **Tenant Users**: Access limited to their own tenant's audit data
- **System Administrators**: Cross-tenant access with appropriate filtering
- **Timeline Resolution**: User information included only if accessible to requester

### Temporal Constraints

<Steps>
<Step title="Date Range Validation">
Date range queries validate that `endDate >= startDate`
</Step>

<Step title="Inclusive Ranges">
Date filters use inclusive ranges for both start and end dates
</Step>

<Step title="UTC Storage">
All timestamps are stored and returned in UTC format
</Step>
</Steps>

### Security Considerations

<Warning>
The audit module implements multiple layers of security to prevent unauthorized access and data manipulation.
</Warning>

- **No Direct Manipulation**: All writes occur via database triggers only
- **Multi-Layer Tenant Isolation**: Enforced at service and database levels
- **Sensitive Data Filtering**: Database-level filtering before persistence
- **Role-Based Access**: Separate endpoints for different authorization levels

### Performance and Caching

<Info>
Audit data follows a write-heavy, read-light pattern with real-time requirements that preclude traditional caching strategies.
</Info>

- **No Application Caching**: Due to real-time audit requirements
- **Database Optimization**: Indexes on tenant, table, and timestamp fields
- **On-Demand Resolution**: Timeline formatting performed at query time
- **Pagination**: Mandatory for all list operations to manage memory usage
