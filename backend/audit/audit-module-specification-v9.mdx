## Overview

The audit module provides comprehensive activity tracking and timeline functionality for the PropWise CRM system. It captures all data changes across entities through PostgreSQL triggers and presents them as both raw audit logs and human-readable timeline events. The module supports both tenant-scoped and system-wide audit queries.

Last updated: January 15, 2024  
Status: Active
</Info>

## Architecture

### Design Decisions

The audit module is built around several key architectural decisions that ensure data integrity and performance:

<CardGroup cols={2}>
  <Card title="Database-Driven Auditing" icon="database">
    Uses PostgreSQL triggers (`audit_trigger_func()`) to capture changes at the database level, ensuring consistency and completeness
  </Card>
  <Card title="Read-Only Entity" icon="lock">
    The `AuditLog` entity is strictly read-only from application code - all writes happen via database triggers
  </Card>
  <Card title="Timeline Abstraction" icon="timeline">
    Provides a higher-level timeline view that translates raw audit data into user-friendly messages
  </Card>
  <Card title="Dual Access Patterns" icon="users">
    Supports both tenant-scoped queries (for regular users) and system-wide queries (for system administrators)
  </Card>
</CardGroup>

### Key Components

- **AuditLog Entity**: Maps to the `audit_log` table (read-only)
- **AuditService**: Core business logic for querying and formatting audit data
- **AuditTimelineResolver**: Translates audit logs into human-readable timeline events
- **Dual Controllers**: Separate endpoints for tenant and system admin access

## Entity Specifications

### AuditLog Entity

The core audit log entity captures all data changes with full context:

```typescript
interface AuditLog {
  id: string;                    // UUID primary key
  tableName: string;             // Name of the table that changed
  action: AuditAction;           // INSERT, UPDATE, DELETE
  rowData: Record<string, any>;  // Full row data after change
  changedFields: Record<string, any>; // Only changed fields (UPDATE only)
  recordId: string;              // ID of the changed record
  tenantId: string;              // Tenant context
  userId?: string;               // User who made the change
  createdAt: Date;               // When the change occurred
}
```

### AuditAction Enum

<CodeGroup>

### Timeline Event Types

Timeline events provide semantic classification of audit activities:

<AccordionGroup>
  <Accordion title="Lead Events">

    - `LEAD_STAGE_CHANGED`: Lead moved between pipeline stages
    - `LEAD_QUALIFIED`: Lead marked as qualified
    - `LEAD_DONE`: Lead completed
    - `LEAD_DISQUALIFIED`: Lead marked as disqualified
    - `LEAD_SCORE_CHANGED`: Lead scoring updated
  </Accordion>
  
  <Accordion title="Deal Events">
    - `DEAL_CREATED`: New deal created
    - `DEAL_STAGE_CHANGED`: Deal moved between pipeline stages
    - `DEAL_WON`: Deal marked as won
    - `DEAL_LOST`: Deal marked as lost
  </Accordion>
  
  <Accordion title="Contact Events">
    - `CONTACT_CREATED`: New contact added
    - `CONTACT_UPDATED`: Contact information modified
  </Accordion>
  
  <Accordion title="Generic Events">
    - `RECORD_CREATED`: Generic record creation
    - `RECORD_UPDATED`: Generic record modification
    - `RECORD_DELETED`: Generic record deletion
  </Accordion>

## API Endpoints

### Tenant-Scoped Endpoints

#### Retrieve Audit Logs

<Steps>
  <Step title="Endpoint">
    `GET /audit/logs` - Retrieve paginated audit logs for the current tenant
  </Step>
  
  <Step title="Query Parameters">
    ```typescript
    interface AuditLogQueryDto {
      page?: number;        // Page number (default: 1)
      limit?: number;       // Items per page (1-100, default: 20)
      tableName?: string;   // Filter by table name
      recordId?: string;    // Filter by record ID
      userId?: string;      // Filter by user ID
      action?: AuditAction; // Filter by action type
      startDate?: string;   // ISO date string
      endDate?: string;     // ISO date string
    }
    ```
  </Step>
  
  <Step title="Response Format">
    ```typescript
    interface PaginatedResponse<AuditLogDto> {
      data: AuditLogDto[];
      pagination: {
        page: number;
        limit: number;
        total: number;
        totalPages: number;
      };
    }
    ```
  </Step>
</Steps>

#### Entity Timeline

<Steps>
  <Step title="Endpoint">
    `GET /audit/timeline/:entityType/:entityId` - Get resolved timeline events for a specific entity
  </Step>
  
  <Step title="Parameters">
    - `entityType`: The type of entity (e.g., 'lead', 'deal')
    - `entityId`: UUID of the specific entity
  </Step>
  
  <Step title="Response Format">
    ```typescript
    interface ResolvedTimeline {
      eventType: TimelineEventType;
      message: string;
      createdAt: Date;
      user?: AuditUserInfo;
    }
    ```
  </Step>
</Steps>

## Business Rules and Invariants

### Data Integrity

<Check>All audit operations must occur within the same transaction as the data change</Check>

- Audit logs are immutable once created by the database trigger
- Sensitive fields (passwords, refresh tokens) are automatically stripped by the trigger
- Database-level consistency ensures no audit data is lost

### Access Control

<Tabs>
  <Tab title="Regular Users">
    - Can only access audit logs for their own tenant
    - Timeline resolution includes user information only if accessible
    - All queries automatically scoped to tenant context
  </Tab>
  
  <Tab title="System Administrators">
    - Can access audit logs across all tenants
    - Requires explicit SystemAdmin role
    - Can filter by specific tenant or view system-wide data
  </Tab>
</Tabs>

### Temporal Constraints

<Note>
All timestamps are stored in UTC format for consistency across time zones.
</Note>

- Date range queries validate that `endDate >= startDate`
- Date filters use inclusive ranges
- Support for ISO date string format in queries

## Integration Points

### Authentication Module Integration

The audit module integrates closely with the authentication system:

- Uses `@AuthGuard()` for endpoint protection
- Leverages `@GetCurrentUser()` for tenant context extraction
- Integrates with `@SystemAdmin()` decorator for admin access control

### Tenant Context Integration

<Info>
Tenant isolation is enforced at multiple layers to ensure data security.
</Info>

- All queries automatically scoped to current tenant (except system admin)
- Tenant isolation enforced at the service layer
- Database triggers include tenant context in all audit records

### User Module Integration

- Resolves user information for timeline display
- Links audit entries to user accounts via `userId` field
- Handles user deletion scenarios gracefully

### Entity Module Integration

- Monitors changes across all major entities (leads, deals, contacts, etc.)
- Uses entity type resolution for timeline formatting
- Maintains referential integrity with source entities

### Error Handling

<Warning>
Always validate UUID format for ID parameters to prevent injection attacks.
</Warning>

- Returns 404 for non-existent entities in timeline queries
- Enforces pagination limits (maximum 100 items per page)
- Provides meaningful error messages for validation failures

### Caching Considerations

<Tip>
Due to real-time requirements, audit data is not cached. Timeline resolution is performed on-demand for accurate, up-to-date information.
</Tip>

- Audit data is write-heavy, read-light by nature
- No caching implemented to ensure real-time accuracy
- Consider read replicas for high-volume audit queries

### Security Measures

- No direct entity manipulation - all writes via database triggers
- Tenant isolation enforced at multiple layers
- Sensitive data filtered at database level before persistence
- All endpoints protected by authentication guards
