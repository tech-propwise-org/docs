> **Reference Implementations:** `src/modules/user/user.dto.ts`, `src/modules/organization/organization.dto.ts`

This document defines the standard approach for DTOs (Data Transfer Objects) and API response serialization across the PropWise CRM backend.

## Overview

### Key principles

1. **Separation of concerns** — Entities handle database mapping, DTOs handle API contracts
2. **Constructor-based transformation** — Response DTOs use constructors to transform entities
3. **Explicit field assignment** — Manually assign fields in constructor for clarity and control
4. **DTO composition over duplication** — Import and reuse existing DTOs; never duplicate definitions
5. **Collection safety** — Always check `isInitialized()` before accessing MikroORM collections

### Technology stack

- **class-validator**: For input validation (`@IsString()`, `@IsUUID()`, etc.)
- **class-transformer**: For input transformation (`@Transform()`, `@Type()`)
- **Constructor pattern**: For response DTO transformation (no `plainToInstance()`)

## Architecture

```
┌─────────────────────────────────────────────────────────────────────────┐
│                              API Request                                │
└─────────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
┌─────────────────────────────────────────────────────────────────────────┐
│                            CONTROLLER                                   │
│  • Receives request                                                     │
│  • Validates input using Input DTOs (CreateXxxDto, UpdateXxxDto)        │
│  • Returns Response DTOs from service                                   │
└─────────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
┌─────────────────────────────────────────────────────────────────────────┐
│                              SERVICE                                    │
│  • Business logic                                                       │
│  • Database queries (returns Entity)                                    │
│  • Transforms Entity → ResponseDto using new XxxDto(entity)             │
│  • Controls what data to include via constructor parameters             │
└─────────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
┌─────────────────────────────────────────────────────────────────────────┐
│                              ENTITY                                     │
│  • MikroORM decorators for database mapping                             │
│  • NO serialization logic                                               │
└─────────────────────────────────────────────────────────────────────────┘
```

## DTO categories

Each entity module should have **two types of DTOs**:

### 1. Input DTOs (validation)

Used for validating incoming request data. Use `class-validator` decorators.

```typescript
// Create DTO - for POST requests
export class CreateUserDto {
  @IsString()
  @MinLength(2)
  firstName!: string;

  @IsString()
  @MinLength(2)
  lastName!: string;

  @IsEmail()
  @Transform(({ value }) => value?.toLowerCase())
  email!: string;

  @IsString()
  @MinLength(8)
  password!: string;

  @IsOptional()
  @IsString()
  phoneNumber?: string;
}

// Update DTO - for PUT/PATCH requests
export class UpdateUserDto {
  @IsOptional()
  @IsString()
  @MinLength(2)
  firstName?: string;

  @IsOptional()
  @IsString()
  phoneNumber?: string;
}
```

**Common decorators:**

| Decorator                                               | Purpose                                     |
| ------------------------------------------------------- | ------------------------------------------- |
| `@IsString()`, `@IsUUID()`, `@IsNumber()`, `@IsEmail()` | Type validation                             |
| `@IsOptional()`                                         | Field is optional                           |
| `@IsNotEmpty()`                                         | Field must have a value                     |
| `@ValidateIf()`                                         | Conditional validation                      |
| `@Transform()`                                          | Input normalization (e.g., lowercase email) |
| `@Type()`                                               | Nested object type for validation           |
| `@ValidateNested()`                                     | Validate nested objects                     |
| `@IsArray()`                                            | Validate arrays                             |
| `@MinLength()`, `@MaxLength()`                          | String length validation                    |

### 2. Response DTOs (serialization)

Used for serializing outgoing response data. Use **constructor-based transformation**.

```typescript
export class UserDto {
  id: string;
  firstName: string;
  lastName: string;
  email: string;
  phoneNumber?: string;
  isActive: boolean;
  createdAt: Date;
  updatedAt: Date;

  constructor(user: User) {
    this.id = user.id;
    this.firstName = user.firstName;
    this.lastName = user.lastName;
    this.email = user.email;
    this.phoneNumber = user.phoneNumber;
    this.isActive = user.isActive ?? true;
    this.createdAt = user.createdAt;
    this.updatedAt = user.updatedAt;
  }
}
```

**Directory / picker DTOs** (`LeadDirectoryDto`, `DealDirectoryDto`, `ContactDirectoryDto`, `CompanyDirectoryDto`): Minimal `{ id, displayName, subtitle? }` rows from `GET /{entity}/directory` (see `Docs/CRM_MODULE_SPECIFICATION.md`). `ContactDirectoryDto` additionally supports optional `{ isPrimary?, role? }` when `GET /contacts/directory` is scoped with `dealId` or `companyId` (junction metadata). Same constructor pattern as `UserDirectoryDto` / `TeamDirectoryDto` — services map query rows or partial shapes in `new XxxDirectoryDto({ ... })`. No `@Expose()` or serialization groups; not used with `plainToInstance()` for full entity hydration.

## Constructor-based response DTOs

### Why constructor pattern?

1. **Explicit control** — Every field assignment is visible and intentional
2. **Type safety** — TypeScript ensures entity properties exist
3. **Flexibility** — Easy to add conditional logic, default values, computed fields
4. **No magic** — No hidden transformation rules or decorator ordering issues
5. **Easy debugging** — Stack traces point directly to assignment code

### Basic structure

```typescript
export class EntityDto {
  // Declare all response fields with types
  id: string;
  name: string;
  createdAt: Date;
  
  // Constructor accepts entity and optional parameters
  constructor(entity: Entity, optionalParam?: string) {
    // Assign each field explicitly
    this.id = entity.id;
    this.name = entity.name;
    this.createdAt = entity.createdAt;
  }
}
```

### Default values

Use nullish coalescing for default values:

```typescript
constructor(user: User) {
  this.isActive = user.isActive ?? true;
  this.createdAt = user.createdAt || new Date();
  this.role = user.role ?? 'member';
}
```

## Nested DTOs & composition

### The composition rule

<Warning>
  **CRITICAL: Never duplicate DTO definitions.**

  Each entity has **ONE canonical ResponseDto** in its own module. Other modules must **import and reuse** that DTO.
</Warning>

| Entity       | Canonical DTO location             | DTO class         |
| ------------ | ---------------------------------- | ----------------- |
| User         | `user/user.dto.ts`                 | `UserDto`         |
| Organization | `organization/organization.dto.ts` | `OrganizationDto` |
| Person       | `person/person.dto.ts`             | `PersonDto`       |
| Company      | `company/company.dto.ts`           | `CompanyDto`      |
| Lead         | `lead/lead.dto.ts`                 | `LeadDto`         |
| Deal         | `deal/deal.dto.ts`                 | `DealDto`         |

### Composing nested DTOs

```typescript
// user.dto.ts
import { OrganizationDto } from "@/modules/organization/organization.dto";
import { OrgRoleDto } from "../rbac/user-org-role/user-org-role.dto";

export class MyUserDto extends UserDto {
  organizations: OrganizationDto[];
  orgRoles: OrgRoleDto[];

  constructor(user: User) {
    super(user);

    this.organizations = user.organizations?.isInitialized()
      ? user.organizations.getItems().map((org) => new OrganizationDto(org))
      : [];

    this.orgRoles = user.orgRoles?.isInitialized()
      ? user.orgRoles.getItems().map((role) => new OrgRoleDto(role))
      : [];
  }
}
```

```typescript
  // ... same fields as OrganizationDto

```
</Tab>

```typescript
  id: string;
  createdBy?: UserDto;
  organization: OrganizationDto;

    this.id = lead.id;

    this.organization = new OrganizationDto(lead.organization);
```

## Handling MikroORM collections

<Warning>
MikroORM collections must be checked with `isInitialized()` before access. Accessing an uninitialized collection throws a runtime error.
</Warning>

### Ternary pattern (preferred)

```typescript
constructor(user: User) {
  this.organizations = user.organizations?.isInitialized()
    ? user.organizations.getItems().map((org) => new OrganizationDto(org))
    : [];

  this.teamMemberships = user.teamMemberships?.isInitialized()
    ? user.teamMemberships.getItems().map((tm) => new TeamMembershipDto(tm))
    : [];
}
```

### If-else pattern

```typescript
constructor(user: User) {
  if (user.orgRoles?.isInitialized()) {
    this.orgRoles = user.orgRoles.getItems().map((role) => new OrgRoleDto(role));
  } else {
    this.orgRoles = [];
  }
}
```

### With filtering

Apply filters during transformation to exclude soft-deleted records:

```typescript
constructor(person: Person) {
  this.emails = person.emails?.isInitialized()
    ? person.emails
        .getItems()
        .filter((e) => !e.isDeleted)
        .map((e) => new PersonEmailDto(e))
    : [];
}
```

## DTO inheritance

Use inheritance for DTOs that share a common base but add additional fields:

```typescript
// Base DTO
export class UserDto {
  id: string;
  firstName: string;
  lastName: string;
  email: string;
  isActive: boolean;
  orgRoles: OrgRoleDto[];

  constructor(user: User) {
    this.id = user.id;
    this.firstName = user.firstName;
    this.lastName = user.lastName;
    this.email = user.email;
    this.isActive = user.isActive ?? true;

    this.orgRoles = user.orgRoles?.isInitialized()
      ? user.orgRoles.getItems().map((role) => new OrgRoleDto(role))
      : [];
  }
}

// Extended DTO with additional fields
export class MyUserDto extends UserDto {
  organizations: OrganizationDto[];
  selectedOrganization: OrganizationDto | null;
  isVerified: boolean;
  accessToken?: string;

  constructor(user: User, accessToken?: string) {
    super(user);

    this.organizations = user.organizations?.isInitialized()
      ? user.organizations.getItems().map((org) => new OrganizationDto(org))
      : [];

    this.selectedOrganization = user.selectedOrganization
      ? new OrganizationDto(user.selectedOrganization)
      : null;

    this.isVerified = user.isVerified ?? false;
    this.accessToken = accessToken;
  }
}
```

### When to use inheritance

| Use case                         | Pattern                   | Example                                |
| -------------------------------- | ------------------------- | -------------------------------------- |
| Adding fields to base DTO        | Extend base class         | `MyUserDto extends UserDto`            |
| Detail view with deep relations  | Extend base class         | `EntityDetailDto extends EntityDto`    |
| Different field visibility       | Separate DTO classes      | `UserDto`, `PublicUserDto`             |
| Context-specific computed fields | Simple constructor params | `constructor(user, currentUserId?)`    |
| Same entity, different audiences | Multiple DTO classes      | `UserDto`, `LoginUserDto`, `MyUserDto` |

## Conditional & optional fields

### Basic optional field handling

```typescript
constructor(user: User) {
  this.phoneNumber = user.phoneNumber; // Optional property
  this.middleName = user.middleName || undefined;
  this.bio = user.profile?.bio; // Optional chaining
}
```

### Conditional field inclusion

For fields that depend on context or permissions:

```typescript
constructor(user: User, currentUserId?: string, includePrivate = false) {
  this.id = user.id;
  this.firstName = user.firstName;
  this.lastName = user.lastName;
  
  // Conditionally include sensitive fields
  if (includePrivate || user.id === currentUserId) {
    this.email = user.email;
    this.phoneNumber = user.phoneNumber;
  }
  
  // Context-dependent computed fields
  if (currentUserId) {
    this.isOwner = user.id === currentUserId;
    this.canEdit = this.checkEditPermission(user, currentUserId);
  }
}
```

### Computed fields

Add derived properties based on entity state:

```typescript
constructor(deal: Deal) {
  this.id = deal.id;
  this.amount = deal.amount;
  this.stage = deal.stage;
  
  // Computed fields
  this.isActive = deal.stage?.isActive ?? true;
  this.daysInStage = this.calculateDaysInStage(deal.stageUpdatedAt);
  this.probability = deal.stage?.defaultProbability ?? 0;
}

private calculateDaysInStage(stageUpdatedAt?: Date): number {
  if (!stageUpdatedAt) return 0;
  return Math.floor((Date.now() - stageUpdatedAt.getTime()) / (1000 * 60 * 60 * 24));
}
```

## Service layer pattern

### Creating response DTOs

Services transform entities to DTOs before returning:

```typescript
@Injectable()
export class UserService {
  async getMe(userId: string): Promise<MyUserDto> {
    const user = await this.em.findOne(
      User,
      { id: userId },
      {
        populate: ["organizations", "orgRoles", "selectedOrganization"],
      },
    );

    if (!user) {
      throw new NotFoundException("User not found");
    }

    return new MyUserDto(user);
  }

  async getUsers(organizationId: string): Promise<UserDto[]> {
    const users = await this.em.find(
      User,
      { organizations: { id: organizationId } },
      {
        populate: ["orgRoles"],
      },
    );

    return users.map((user) => new UserDto(user));
  }
}
```

### Passing extra data to DTOs

```typescript
async login(credentials: LoginDto): Promise<LoginResponse> {
  const user = await this.validateCredentials(credentials);
  const accessToken = await this.generateToken(user);

  return {
    user: new LoginUserDto(user, accessToken),
    refreshToken: await this.generateRefreshToken(user),
  };
}

async getOrganization(orgId: string, currentUserId: string): Promise<OrganizationDto> {
  const org = await this.em.findOne(Organization, { id: orgId });
  
  if (!org) {
    throw new NotFoundException('Organization not found');
  }
  
  return new OrganizationDto(org, currentUserId);
}
```

### Error handling in services

```typescript
async getUserById(id: string, currentUserId?: string): Promise<UserDto> {
  try {
    const user = await this.em.findOne(
      User,
      { id },
      { populate: ['orgRoles', 'organizations'] }
    );

    if (!user) {
      throw new NotFoundException(`User with ID ${id} not found`);
    }

    return new UserDto(user);
  } catch (error) {
    this.logger.error(`Failed to get user ${id}:`, error);
    throw error;
  }
}
```

```typescript
import { UserDto } from "@/modules/user/user.dto";

export class CreateOrganizationDto {

  memberCount?: number;

    this.name = data.name;
    this.website = data.website;
    this.email = data.email;
    this.vat = data.vat;
    this.address = data.address;
    this.createdAt = data.createdAt;
```

## Migration checklist

When updating existing DTOs to follow this standard:

<Steps>
<Step title="Audit existing DTOs">
- [ ] Identify all response DTOs using `plainToInstance()` or `@Exclude()` decorators
- [ ] List all DTOs with duplicate field definitions across modules
- [ ] Find DTOs with boolean inclusion flags (e.g., `includeRelations?: boolean`)
</Step>

<Step title="Create canonical DTOs">
- [ ] Ensure each entity has one primary DTO in its own module (`entity/entity.dto.ts`)
- [ ] Convert to constructor-based transformation pattern
- [ ] Add proper TypeScript field declarations
</Step>

<Step title="Update service methods">

## Quick reference

| Aspect                      | Standard                                                              |
| --------------------------- | --------------------------------------------------------------------- |
| **Naming convention**       | Response: `EntityDto` — e.g., `UserDto`, `LeadDto`                    |
| **Input validation**        | `class-validator` decorators in `CreateXxxDto`, `UpdateXxxDto`        |
| **Output serialization**    | Constructor-based transformation in `XxxDto`                          |
| **Transformation location** | Service layer using `new XxxDto(entity)`                              |
| **Nested objects**          | Import and compose DTOs — never duplicate                             |
| **Collections**             | Check `isInitialized()` before `getItems().map()`                     |
| **Deep relations**          | Create `EntityDetailDto extends EntityDto` — no boolean options       |
| **Contextual fields**       | Simple constructor params (e.g., `currentUserId`) for computed fields |
| **DTO variations**          | Inheritance (`MyUserDto extends UserDto`) or separate classes         |
| **DTO ownership**           | Each entity has one canonical DTO in its module                       |
