Authoritative description of the CRM data-export endpoints (`GET /{entity}/export`) for **Lead**, **Deal**, **Contact**, and **Company**: what they return, who is allowed to call them, and exactly how the result set is scoped.

<Note>
**Scope of this document:** This covers user-facing CSV/XLSX exports of CRM entities. It does **not** cover import (see `CRM_MODULE_SPECIFICATION.md`), audit-log exports, or analytics report downloads.
</Note>

## Endpoints at a glance

| Entity  | Endpoint                   | Service method                   | Scoping model        |
| ------- | -------------------------- | -------------------------------- | -------------------- |
| Lead    | `GET /v1/leads/export`     | `LeadService.exportLeads`        | **Accessible union** |
| Deal    | `GET /v1/deals/export`     | `DealService.exportDeals`        | **Accessible union** |
| Contact | `GET /v1/contacts/export`  | `ContactService.exportContacts`  | **Accessible union** |
| Company | `GET /v1/companies/export` | `CompanyService.exportCompanies` | **Accessible union** |

<Info>
All four exports use the same **permission-free accessible-union model**: no `crm.view`, `sales.view`, or `team_crm.view` gate at the export endpoint, only the **data-portability policy** and the caller's own per-entity visibility.
</Info>

All four exports share:
- The **same two hard gates** (see Data-portability policy)
- The **same RLS backstop** (see RLS backstop)
- The **same filter/`selectedIds` narrowing semantics** (see Filter and selectedIds narrowing)
- The **same file-format rules** (see File format rules)

## The two-layer mental model

Every export answer is the intersection of two independent questions:

<Steps>
<Step title="Are you allowed to export at all?">
The **data-portability policy** gate (`assertDataPortabilityPolicy`). This is an org-level governance switch, not a CRM-visibility check.
</Step>

<Step title="Which rows are you allowed to see?">
The **row-scoping** model, which differs per entity (see Lead row-scoping and Deal / Contact / Company row-scoping).
</Step>
</Steps>

<Warning>
These gates are orthogonal. Passing gate 1 never widens gate 2, and vice versa. A user who clears the portability policy still only ever receives rows their CRM visibility allows.
</Warning>

## Data-portability policy (all four exports)

Enforced in the controller **before** the service runs:

```typescript
await this.accessService.assertDataPortabilityPolicy(user, 'export');
```

### Policy enforcement logic

The `AccessService.assertDataPortabilityPolicy(user, 'export')` method (`src/modules/auth/services/access.service.ts`) executes the following logic:

<Steps>
<Step title="System bypass check">
System admin / system owner (`hasSystemBypass`) → always allowed, short-circuit.
</Step>

<Step title="Resolve org policy">
Resolve the org's `settings.dataExportPolicy`, defaulting to `DEFAULT_DATA_PORTABILITY_POLICY` when unset.
</Step>

<Step title="Apply policy rules">
- If policy is `'all'` → **any authenticated member** of the org may export
- If policy is `'admins_only'` → caller **must** hold `OrgPermissionKey.SYSTEM_ADMIN`, otherwise `403 ForbiddenException`: `"Your organisation requires Admin or Owner role to export data."`
</Step>
</Steps>

### Policy values

| `dataExportPolicy`          | Who can export                                                 |
| --------------------------- | -------------------------------------------------------------- |
| `all`                       | Any authenticated org member (subject to row-scoping)          |
| `admins_only` **(default)** | Only Admin / Owner (`system.admin`) + system bypass            |

<Warning>
**Important default:** `DEFAULT_DATA_PORTABILITY_POLICY = 'admins_only'`. Out of the box, **regular agents cannot export anything** until an Owner/Admin flips the org's export policy to `all`. The "any agent can export what they see" behaviour is only true once the policy is `all`.
</Warning>

<Info>
The policy is a per-org setting (`Organization.settings.dataExportPolicy`), editable by org admins via the organization settings API. The frontend mirrors the same rule in `src/lib/data-portability-policy.ts` and matches the 403 message via `isDataPortabilityPolicyError`.
</Info>

## Lead row-scoping (accessible-union model)

The lead export has **no `crm.view` gate** and **never calls `validateTypeAccess`**. After the portability gate, the rows returned are scoped purely by the caller's own permissions and memberships.

### Lead export scopes

| `type` value                                               | Rows returned                                                                  | Filename prefix |
| ---------------------------------------------------------- | ------------------------------------------------------------------------------ | --------------- |
| `my`                                                       | Only leads the user **created** or is a **direct (user-level) stakeholder** on | `leads-my-`     |
| anything else — omitted, `all`, `pool`, `team`, `teamPool` | **Accessible union** of every tab the user can see                             | `leads-all-`    |

### How the accessible union is built

The `exportLeads` method in `src/modules/crm/lead/lead.service.ts`:

```typescript
// type !== MY → resolve the access predicate first (async), then build QB synchronously
const accessFragment =
  type !== LeadViewType.MY ? await this.resolveAccessibleLeadsFragment(em, user) : null;

if (type === LeadViewType.MY) {
  qb = this.buildLeadQueryForType(em, user, LeadViewType.MY); // created_by OR stakeholder
} else {
  qb = this.buildLeadQueryForType(em, user, LeadViewType.ALL); // base query, no access filter
  if (accessFragment) {
    qb.andWhere(accessFragment.sql, accessFragment.params); // OR-combined predicate
  }
}
```

### Access fragment resolution

`resolveAccessibleLeadsFragment(em, user)`:

<Tabs>
<Tab title="Org-wide CRM view">
**Returns `null`** when the user has org-wide CRM view (`crm.view` → `canViewAll`). A `null` fragment means **no access predicate is added**, so the user sees every (non-deleted) lead in the org — equivalent to the `ALL` tab. This is the **only** path that returns `null`.
</Tab>

<Tab title="Limited access">
**Otherwise** it ORs together one fragment per tab the user can see:

- **MY** — always (`myLeadsFragment`: `created_by_id = user` OR a direct `entity_stakeholder` row for the user)
- **POOL** — always (`poolLeadsFragment`: `NOT EXISTS` any stakeholder row → unassigned leads)
- **TEAM** — one `teamLeadsFragment` per team returned by `getAvailableTeams(em, user, orgId, false)` (teams where the user holds `team_crm.view`), **only if** the `team` tab is available
- **TEAM_POOL** — one `teamPoolLeadsFragment` per team returned by `getAllMemberTeams(em, user.id, false)` (teams the user is a member of), **only if** the `teamPool` tab is available

The fragments are combined into a single `(... OR ... OR ...)` SQL predicate, so deduplication is automatic — a lead visible via multiple tabs appears once.
</Tab>
</Tabs>

### Why this matches "only what the user can see"

The export reuses the **exact same** building blocks as the lead list/kanban view (the canonical definition of "what the user sees"):

- The list view's per-tab filters delegate to the **same** fragment methods the export ORs together:
  - `applyMyLeadsFilter → myLeadsFragment`
  - `applyPoolLeadsFilter → poolLeadsFragment`
  - `applyTeamLeadsFilter → teamLeadsFragment`
  - `applyTeamPoolLeadsFilter → teamPoolLeadsFragment`
- The list/kanban view resolves tabs and teams with the **same** helpers: `getAvailableTabs`, `getAvailableTeams`, `getAllMemberTeams`

<Check>
Because the export is a literal OR of the same per-tab predicates, it can never return a lead the list view wouldn't show for that user.
</Check>

### Tab availability rules

The `getAvailableTabs` method determines which tabs a user can access:

| Tab        | Available when                                |
| ---------- | --------------------------------------------- |
| `all`      | `crm.view` (`canViewAll`)                     |
| `my`       | always                                        |
| `pool`     | always — **anyone can see unassigned leads**  |
| `team`     | `crm.view` **or** `team_crm.view`             |
| `teamPool` | `crm.view` **or** the user belongs to ≥1 team |

<Note>
**Product note on POOL:** Because `pool` is unconditionally available, a user with zero permissions and zero teams can still export **every unassigned lead in the org**. This is consistent with the list view (which also shows them the pool), so the export is faithful — but it is a deliberate visibility-model decision, not an export quirk. If pool should be narrower, fix it in `getAvailableTabs`/`poolLeadsFragment` and the export inherits the fix automatically.
</Note>

### teamId parameter handling

<Warning>
The lead export accepts a `teamId` query param for backward compatibility but **does not use it**. The union is always computed server-side from the caller's own memberships/permissions, so a caller-supplied `teamId` cannot be used to read another team's leads. There is no cross-team access risk.
</Warning>

### Implementation footgun (maintainers read this)

<Warning>
The union path builds the **unfiltered** `ALL` base query and then _conditionally_ adds the predicate. The safety of this branch rests entirely on the invariant:

**`resolveAccessibleLeadsFragment` returns `null` if and only if the user has `canViewAll`.**

If a future refactor ever makes that method return `null`/empty on any other path (e.g. an early `return null` when the fragment list is empty), **every non-privileged caller would silently export the entire org's leads**, and the RLS backstop would not catch it (same org). When touching `resolveAccessibleLeadsFragment`, preserve the invariant and keep the regression test that asserts a no-permission user gets a predicate containing both `created_by_id` (MY) and `NOT EXISTS` (POOL).
</Warning>

## Deal / Contact / Company row-scoping (accessible-union model)

Deal, Contact, and Company exports use the **same permission-free accessible-union model as leads**. `validateTypeAccess` is **never called** by any of these export endpoints.

### Export scopes

| Frontend scope | Sent `type`             | Rows returned                                                                        |
| -------------- | ----------------------- | ------------------------------------------------------------------------------------ |
| `my`           | `MY` (entity-specific)  | Only deals/contacts/companies the user **created** or is a **direct stakeholder** on |
| `all`          | `ALL` (entity-specific) | **Accessible union** — every row the user can see across their tabs                  |
| `selected`     | `ALL` + `selectedIds`   | Same as `all` but AND-intersected with the supplied ID list                          |

### How the accessible union is built per entity

Each entity has a `resolveAccessible<Entity>Fragment(em, user)` method that mirrors the lead pattern:

<Tabs>
<Tab title="Org-wide view permission">
**Returns `null`** when the user has the org-wide view permission for that entity:
- `crm.view` for contacts/companies
- `sales.view` for deals

They see every row in the organization.
</Tab>

<Tab title="Limited access">
**Otherwise** returns an OR-combined SQL predicate covering:

- **MY**: rows the user created or is a direct stakeholder on
- **TEAM**: rows for each team where the user holds:
  - `team_sales.view` (deals)
  - `team_crm.view` (contacts/companies)
  
  <Note>
  _Non-privileged team members_ (members without `team_crm.view`): can still see rows where they are a personal stakeholder within that team's scope.
  </Note>
  
- **DEAL-LINKED contacts** (contact export only): contacts linked to active deals, **if** the user holds `sales.view`
</Tab>
</Tabs>

<Info>
The `teamId` query param is **ignored** for all three — the union is always computed server-side from the caller's own memberships/permissions.
</Info>

### Why the same invariant applies

<Warning>
The same footgun from Lead row-scoping applies to all three entities. Each `resolveAccessible<Entity>Fragment` must return `null` **if and only if** the user has `canViewAll` for that entity. The same regression tests cover this invariant.
</Warning>

## RLS backstop (all four exports)

Every export runs inside:

```typescript
return this.tenantContext.executeReadOnly(organizationId, async (em) => { ... });
```

<Info>
This sets the tenant (organization) context, so **PostgreSQL row-level security physically scopes every query to the caller's organization**, preventing cross-org data leakage at the database level.
</Info>

The RLS backstop ensures that even if application-level logic fails, the database will never return rows from a different organization.

## Filter and selectedIds narrowing

Both query parameters narrow the accessible-union result set:

<CardGroup cols={2}>
<Card title="filters" icon="filter">
Standard list-view filters (status, stage, owner, date ranges, etc.) are applied **after** the access predicate is established. Filters can only narrow, never widen the result set.
</Card>

<Card title="selectedIds" icon="check-square">
When provided, exports **only** the rows that appear in both:
1. The accessible union for the user
2. The supplied ID list

This is used for "export selected" functionality in the UI.
</Card>
</CardGroup>

<Warning>
A user cannot use `selectedIds` to export rows they don't have access to. The accessible-union predicate is always applied first, then intersected with the ID list.
</Warning>

## File format rules

All four exports support two output formats:

<Tabs>
<Tab title="CSV (default)">
```
Content-Type: text/csv
Content-Disposition: attachment; filename="{entity}-{scope}-{timestamp}.csv"
```

- UTF-8 encoding with BOM for Excel compatibility
- Standard CSV escaping (double-quote wrapping for fields containing commas, quotes, or newlines)
- Column headers in first row
</Tab>

<Tab title="XLSX">
```
Content-Type: application/vnd.openxmlformats-officedocument.spreadsheetml.sheet
Content-Disposition: attachment; filename="{entity}-{scope}-{timestamp}.xlsx"
```

- Single worksheet named after the entity type
- Column headers in first row with bold formatting
- Auto-sized columns
- Date/datetime fields formatted appropriately
</Tab>
</Tabs>

### Format selection

The export format is determined by the `Accept` header or `format` query parameter:

```typescript
// CSV (default)
GET /v1/leads/export

// XLSX
GET /v1/leads/export?format=xlsx
// or
GET /v1/leads/export
Accept: application/vnd.openxmlformats-officedocument.spreadsheetml.sheet
```

### Filename patterns

| Entity  | Scope | Example filename                         |
| ------- | ----- | ---------------------------------------- |
| Lead    | my    | `leads-my-2024-01-15T10-30-00.csv`       |
| Lead    | all   | `leads-all-2024-01-15T10-30-00.csv`      |
| Deal    | my    | `deals-my-2024-01-15T10-30-00.xlsx`      |
| Deal    | all   | `deals-all-2024-01-15T10-30-00.xlsx`     |
| Contact | my    | `contacts-my-2024-01-15T10-30-00.csv`    |
| Contact | all   | `contacts-all-2024-01-15T10-30-00.csv`   |
| Company | my    | `companies-my-2024-01-15T10-30-00.csv`   |
| Company | all   | `companies-all-2024-01-15T10-30-00.csv`  |

## Common export patterns

<AccordionGroup>
<Accordion title="Export all accessible leads">
```typescript
GET /v1/leads/export?type=all
```

Returns every lead the user can see across all their tabs (MY + POOL + TEAM + TEAM_POOL), subject to:
1. Data-portability policy allowing export
2. User's accessible-union fragment
</Accordion>

<Accordion title="Export only my leads">
```typescript
GET /v1/leads/export?type=my
```

Returns only leads the user created or is a direct stakeholder on.
</Accordion>

<Accordion title="Export selected deals">
```typescript
GET /v1/deals/export?type=all&selectedIds=123,456,789
```

Returns only deals with IDs 123, 456, 789 that the user has access to. If any IDs are outside the user's accessible union, they are silently omitted.
</Accordion>

<Accordion title="Export with filters">
```typescript
GET /v1/contacts/export?type=all&status=active&createdAfter=2024-01-01
```

Returns all accessible contacts filtered by status and creation date. Filters are applied after access scoping.
</Accordion>
</AccordionGroup>

## Security guarantees

<Steps>
<Step title="Org isolation">
RLS backstop ensures queries cannot return data from other organizations, even with application logic bugs.
</Step>

<Step title="Data portability governance">
Organization-level policy controls who can export at all, independent of what they can see.
</Step>

<Step title="Visibility preservation">
Exports never return rows the user couldn't see in the list/kanban view. The accessible-union model guarantees this by reusing the same fragment logic.
</Step>

<Step title="No privilege escalation">
User-supplied parameters (`teamId`, `selectedIds`) cannot be used to access rows outside the user's accessible union.
</Step>

<Step title="Soft-delete respect">
Deleted entities are excluded from all exports unless explicitly included in filters (not currently supported).
</Step>
</Steps>

## Testing recommendations

When testing export functionality, verify:

<CardGroup cols={2}>
<Card title="Policy enforcement" icon="shield-check">
- Admin-only policy blocks non-admins
- All policy allows any org member
- System bypass works regardless of policy
</Card>

<Card title="Accessible-union correctness" icon="users">
- Non-privileged user only sees MY + POOL
- Team member sees team rows
- Org-wide permission sees everything
</Card>

<Card title="Filter narrowing" icon="filter">
- Filters reduce but never expand results
- Filtered count ≤ unfiltered accessible count
</Card>

<Card title="Cross-entity consistency" icon="link">
- Same user gets same scope logic across Lead/Deal/Contact/Company
- Contact export includes deal-linked contacts when user has sales.view
</Card>
</CardGroup>

<Warning>
**Regression test critical path:** Always verify that a user with no permissions still gets an access predicate (not `null`) containing both `created_by_id` (MY) and `NOT EXISTS` (POOL) fragments. This guards against the §4.5 footgun where returning `null` inappropriately would expose the entire org's data.
</Warning>
