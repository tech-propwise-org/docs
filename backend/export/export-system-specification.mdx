Authoritative description of the CRM data-export endpoints (`GET /{entity}/export`) for **Lead**, **Deal**, **Contact**, and **Company**: what they return, who is allowed to call them, and exactly how the result set is scoped.

<Note>
**Scope of this document:** This covers user-facing CSV/XLSX exports of CRM entities. It does **not** cover import (see `CRM_MODULE_SPECIFICATION.md`), audit-log exports, or analytics report downloads.
</Note>

## Endpoints at a glance

| Entity  | Endpoint                   | Service method                   | Scoping model        |
| ------- | -------------------------- | -------------------------------- | -------------------- |
| Lead    | `GET /v1/leads/export`     | `LeadService.exportLeads`        | **Accessible union** |
| Deal    | `GET /v1/deals/export`     | `DealService.exportDeals`        | **Accessible union** |
| Contact | `GET /v1/contacts/export`  | `ContactService.exportContacts`  | **Accessible union** |
| Company | `GET /v1/companies/export` | `CompanyService.exportCompanies` | **Accessible union** |

All four exports use the same **permission-free accessible-union model**: no `crm.view`, `sales.view`, or `team_crm.view` gate at the export endpoint, only the **data-portability policy** (see section 3) and the caller's own per-entity visibility (see sections 4 and 5).

<Info>
All four exports share the **same two hard gates**, the **same RLS backstop**, the **same filter/`selectedIds` narrowing semantics**, and the **same file-format rules**.
</Info>

## The two-layer mental model

Every export answer is the intersection of two independent questions:

<Steps>
<Step title="Are you allowed to export at all?">
The **data-portability policy** gate (`assertDataPortabilityPolicy`). This is an org-level governance switch, not a CRM-visibility check. (See section 3)
</Step>

<Step title="Which rows are you allowed to see?">
The **row-scoping** model, which differs per entity (see sections 4 and 5).
</Step>
</Steps>

<Warning>
These are orthogonal. Passing gate 1 never widens gate 2, and vice versa. A user who clears the portability policy still only ever receives rows their CRM visibility allows.
</Warning>

## Gate 1: Data-portability policy (all four exports)

Enforced in the controller **before** the service runs:

```ts
await this.accessService.assertDataPortabilityPolicy(user, 'export');
```

### Policy enforcement logic

`AccessService.assertDataPortabilityPolicy(user, 'export')` logic from `src/modules/auth/services/access.service.ts`:

<Steps>
<Step title="System bypass check">
System admin / system owner (`hasSystemBypass`) → always allowed, short-circuit.
</Step>

<Step title="Resolve organization policy">
Resolve the org's `settings.dataExportPolicy`, defaulting to `DEFAULT_DATA_PORTABILITY_POLICY` when unset.
</Step>

<Step title="Apply policy rules">
- If policy is `'all'` → **any authenticated member** of the org may export
- If policy is `'admins_only'` → caller **must** hold `OrgPermissionKey.SYSTEM_ADMIN`, otherwise `403 ForbiddenException`: `"Your organisation requires Admin or Owner role to export data."`
</Step>
</Steps>

### Policy values

| `dataExportPolicy`          | Who can export                                                 |
| --------------------------- | -------------------------------------------------------------- |
| `all`                       | Any authenticated org member (subject to row-scoping)          |
| `admins_only` **(default)** | Only Admin / Owner (`system.admin`) + system bypass            |

<Warning>
**Important default:** `DEFAULT_DATA_PORTABILITY_POLICY = 'admins_only'`. Out of the box, **regular agents cannot export anything** until an Owner/Admin flips the org's export policy to `all`. The "any agent can export what they see" behaviour is only true once the policy is `all`.
</Warning>

<Info>
The policy is a per-org setting (`Organization.settings.dataExportPolicy`), editable by org admins via the organization settings API. The frontend mirrors the same rule in `src/lib/data-portability-policy.ts` and matches the 403 message via `isDataPortabilityPolicyError`.
</Info>

## Gate 2: Lead row-scoping (accessible-union model)

The lead export has **no `crm.view` gate** and **never calls `validateTypeAccess`**. After the portability gate, the rows returned are scoped purely by the caller's own permissions and memberships. There are exactly **two scopes**:

| `type` value                                               | Rows returned                                                                  | Filename prefix |
| ---------------------------------------------------------- | ------------------------------------------------------------------------------ | --------------- |
| `my`                                                       | Only leads the user **created** or is a **direct (user-level) stakeholder** on | `leads-my-`     |
| anything else — omitted, `all`, `pool`, `team`, `teamPool` | **Accessible union** of every tab the user can see                             | `leads-all-`    |

### How the accessible union is built

`exportLeads` from `src/modules/crm/lead/lead.service.ts`:

```ts
// type !== MY → resolve the access predicate first (async), then build QB synchronously
const accessFragment =
  type !== LeadViewType.MY ? await this.resolveAccessibleLeadsFragment(em, user) : null;

if (type === LeadViewType.MY) {
  qb = this.buildLeadQueryForType(em, user, LeadViewType.MY); // created_by OR stakeholder
} else {
  qb = this.buildLeadQueryForType(em, user, LeadViewType.ALL); // base query, no access filter
  if (accessFragment) {
    qb.andWhere(accessFragment.sql, accessFragment.params); // OR-combined predicate
  }
}
```

#### `resolveAccessibleLeadsFragment(em, user)` behavior

<Tabs>
<Tab title="Org-wide CRM view">
**Returns `null`** when the user has org-wide CRM view (`crm.view` → `canViewAll`). A `null` fragment means **no access predicate is added**, so the user sees every (non-deleted) lead in the org — equivalent to the `ALL` tab. This is the **only** path that returns `null`.
</Tab>

<Tab title="Limited visibility">
**Otherwise** it ORs together one fragment per tab the user can see:
- **MY** — always (`myLeadsFragment`: `created_by_id = user` OR a direct `entity_stakeholder` row for the user)
- **POOL** — always (`poolLeadsFragment`: `NOT EXISTS` any stakeholder row → unassigned leads)
- **TEAM** — one `teamLeadsFragment` per team returned by `getAvailableTeams(em, user, orgId, false)` (teams where the user holds `team_crm.view`), **only if** the `team` tab is available
- **TEAM_POOL** — one `teamPoolLeadsFragment` per team returned by `getAllMemberTeams(em, user.id, false)` (teams the user is a member of), **only if** the `teamPool` tab is available
</Tab>
</Tabs>

The fragments are combined into a single `(... OR ... OR ...)` SQL predicate, so deduplication is automatic — a lead visible via multiple tabs appears once.

### Why this matches "only what the user can see"

The export reuses the **exact same** building blocks as the lead list/kanban view (the canonical definition of "what the user sees"):

<CardGroup cols={2}>
<Card title="List view filters" icon="filter">
The list view's per-tab filters delegate to the **same** fragment methods the export ORs together:
- `applyMyLeadsFilter → myLeadsFragment`
- `applyPoolLeadsFilter → poolLeadsFragment`
- `applyTeamLeadsFilter → teamLeadsFragment`
- `applyTeamPoolLeadsFilter → teamPoolLeadsFragment`
</Card>

<Card title="Tab resolution" icon="table-layout">
The list/kanban view resolves tabs and teams with the **same** helpers: `getAvailableTabs`, `getAvailableTeams`, `getAllMemberTeams`.
</Card>
</CardGroup>

<Check>
Because the export is a literal OR of the same per-tab predicates, it can never return a lead the list view wouldn't show for that user.
</Check>

### Tab availability rules

`getAvailableTabs` determines which tabs are available:

| Tab        | Available when                                |
| ---------- | --------------------------------------------- |
| `all`      | `crm.view` (`canViewAll`)                     |
| `my`       | always                                        |
| `pool`     | always — **anyone can see unassigned leads**  |
| `team`     | `crm.view` **or** `team_crm.view`             |
| `teamPool` | `crm.view` **or** the user belongs to ≥1 team |

<Note>
**Product note on POOL:** Because `pool` is unconditionally available, a user with zero permissions and zero teams can still export **every unassigned lead in the org**. This is consistent with the list view (which also shows them the pool), so the export is faithful — but it is a deliberate visibility-model decision, not an export quirk. If pool should be narrower, fix it in `getAvailableTabs`/`poolLeadsFragment` and the export inherits the fix automatically.
</Note>

### The `teamId` parameter is ignored

<Warning>
The lead export accepts a `teamId` query param for backward compatibility but **does not use it**. The union is always computed server-side from the caller's own memberships/permissions, so a caller-supplied `teamId` cannot be used to read another team's leads. There is no cross-team access risk.
</Warning>

### Implementation footgun (maintainers read this)

<Warning>
The union path builds the **unfiltered** `ALL` base query and then _conditionally_ adds the predicate. The safety of this branch rests entirely on the invariant:

> `resolveAccessibleLeadsFragment` returns `null` **if and only if** the user has `canViewAll`.

If a future refactor ever makes that method return `null`/empty on any other path (e.g. an early `return null` when the fragment list is empty), **every non-privileged caller would silently export the entire org's leads**, and the RLS backstop would not catch it (same org). When touching `resolveAccessibleLeadsFragment`, preserve the invariant and keep the regression test that asserts a no-permission user gets a predicate containing both `created_by_id` (MY) and `NOT EXISTS` (POOL).
</Warning>

## Gate 2: Deal / Contact / Company row-scoping (accessible-union model)

Deal, Contact, and Company exports use the **same permission-free accessible-union model as leads**. `validateTypeAccess` is **never called** by any of these export endpoints. Two scopes are offered by the frontend:

| Frontend scope | Sent `type`             | Rows returned                                                                        |
| -------------- | ----------------------- | ------------------------------------------------------------------------------------ |
| `my`           | `MY` (entity-specific)  | Only deals/contacts/companies the user **created** or is a **direct stakeholder** on |
| `all`          | `ALL` (entity-specific) | **Accessible union** — every row the user can see across their tabs                  |
| `selected`     | `ALL` + `selectedIds`   | Same as `all` but AND-intersected with the supplied ID list                          |

### How the accessible union is built per entity

Each entity has a `resolveAccessible<Entity>Fragment(em, user)` method that mirrors the lead pattern:

<Tabs>
<Tab title="Org-wide view permission">
**Returns `null`** when the user has the org-wide view permission for that entity:
- `crm.view` for contacts/companies
- `sales.view` for deals

They see every row.
</Tab>

<Tab title="Limited visibility">
**Otherwise** returns an OR-combined SQL predicate covering:

- **MY**: rows the user created or is a direct stakeholder on
- **TEAM**: rows for each team where the user holds `team_sales.view` (deals) or `team_crm.view` (contacts/companies)
  - _Non-privileged team members_ (members without `team_crm.view`): can still see rows where they are a personal stakeholder within that team's scope
- **DEAL-LINKED contacts** (contact export only): contacts linked to active deals, **if** the user holds `sales.view`
</Tab>
</Tabs>

<Info>
The `teamId` query param is **ignored** for all three entities — the union is always computed server-side from the caller's own memberships/permissions.
</Info>

### Why the same invariant applies

<Warning>
The same footgun from the lead section applies to all three entities. Each `resolveAccessible<Entity>Fragment` must return `null` **if and only if** the user has `canViewAll` for that entity. The same regression tests cover this invariant.
</Warning>

## RLS backstop (all four exports)

Every export runs inside:

```ts
return this.tenantContext.executeReadOnly(organizationId, async (em) => { ... });
```

This sets the tenant (organization) context, so **PostgreSQL row-level security physically scopes every query to the caller's organization**.

<Check>
Even if the export logic has a bug that fails to apply the accessible-union predicate, RLS ensures queries cannot leak data across organization boundaries.
</Check>

<Note>
RLS does **not** enforce per-user visibility within the same org — that's the job of the accessible-union predicate. RLS only prevents cross-org leaks.
</Note>

## Filter and `selectedIds` narrowing semantics

### Query parameters

All four exports accept:

```typescript
{
  type?: string;           // 'my' | 'all' | entity-specific types
  selectedIds?: string[];  // Array of entity IDs to export
  teamId?: string;         // Ignored (kept for backward compatibility)
  filters?: FilterGroup;   // Standard CRM filter structure
}
```

### Application order

<Steps>
<Step title="Base scoping">
The accessible-union predicate (or MY predicate) is applied first, establishing the baseline set of rows the user can see.
</Step>

<Step title="Apply selectedIds">
If `selectedIds` is provided and non-empty, the result set is AND-intersected with those IDs.

```ts
if (selectedIds?.length) {
  qb.andWhere({ id: { $in: selectedIds } });
}
```

This means `selectedIds` can only **narrow** the result set, never widen it. A user cannot export an entity they don't have access to by including it in `selectedIds`.
</Step>

<Step title="Apply filters">
If `filters` is provided, standard CRM filter logic is applied:

```ts
if (filters) {
  this.applyFilters(qb, filters, user);
}
```

Filters are additive constraints that further narrow the result set.
</Step>
</Steps>

<Warning>
**Security guarantee:** Neither `selectedIds` nor `filters` can expand visibility beyond what the accessible-union predicate allows. They can only narrow the result set further.
</Warning>

## File format rules

### Format selection

All exports support two output formats:

| Format | MIME type                                                                  | Extension |
| ------ | -------------------------------------------------------------------------- | --------- |
| CSV    | `text/csv; charset=utf-8`                                                  | `.csv`    |
| XLSX   | `application/vnd.openxmlformats-officedocument.spreadsheetml.sheet`       | `.xlsx`   |

Format is determined by the `Accept` header or a `format` query parameter.

### Filename conventions

Generated filenames follow the pattern:

```
{entity}-{scope}-{timestamp}.{extension}
```

Examples:
- `leads-my-2024-01-15T10-30-00Z.csv`
- `deals-all-2024-01-15T10-30-00Z.xlsx`
- `contacts-all-2024-01-15T10-30-00Z.csv`
- `companies-my-2024-01-15T10-30-00Z.xlsx`

### Column structure

<Tabs>
<Tab title="Lead export">
Includes:
- Standard lead fields (name, email, phone, status, etc.)
- Custom field values
- Owner/creator information
- Stakeholder lists
- Associated company name
- Timestamps (created, updated)
</Tab>

<Tab title="Deal export">
Includes:
- Standard deal fields (title, value, stage, probability, etc.)
- Custom field values
- Owner/creator information
- Stakeholder lists
- Associated company and contact names
- Timestamps (created, updated, closed)
</Tab>

<Tab title="Contact export">
Includes:
- Standard contact fields (name, email, phone, title, etc.)
- Custom field values
- Owner/creator information
- Stakeholder lists
- Associated company name
- Timestamps (created, updated)
</Tab>

<Tab title="Company export">
Includes:
- Standard company fields (name, domain, industry, size, etc.)
- Custom field values
- Owner/creator information
- Stakeholder lists
- Contact count
- Timestamps (created, updated)
</Tab>
</Tabs>

### Encoding and special characters

<AccordionGroup>
<Accordion title="CSV encoding">
- UTF-8 with BOM for Excel compatibility
- Double-quote escaping for fields containing commas, quotes, or newlines
- CRLF line endings (`\r\n`)
</Accordion>

<Accordion title="XLSX encoding">
- Native Excel format with proper type hints
- Date/datetime fields formatted as Excel date cells
- Numeric fields formatted with appropriate precision
- Text fields preserve leading zeros and special characters
</Accordion>

<Accordion title="Null and empty values">
- `null` values: empty cell/field
- Empty strings: empty cell/field
- `0` and `false`: rendered as `0` and `false` respectively
</Accordion>
</AccordionGroup>

## Performance considerations

### Row limits

<Warning>
Exports are **not paginated**. The entire accessible result set is returned in a single file. Very large exports (>100k rows) may experience:
- Longer response times
- Higher memory usage
- Potential timeout risks
</Warning>

<Tip>
For large datasets, encourage users to:
1. Use filters to narrow the result set
2. Use `selectedIds` to export specific records
3. Break exports into smaller chunks by date range or other criteria
</Tip>

### Query optimization

All export queries:
- Use database indexes on `organization_id`, `created_by_id`, and `deleted_at`
- Leverage indexed columns for stakeholder lookups
- Apply soft-delete filtering (`deleted_at IS NULL`) early in the query plan
- Use `executeReadOnly` for read-replica routing where available

### Streaming response

<Info>
Exports use streaming response generation to minimize memory footprint:
- CSV rows are generated and written incrementally
- XLSX files are built using streaming workbook writers
- Response headers are sent immediately with `Transfer-Encoding: chunked`
</Info>

## Error handling

### Common error scenarios

<AccordionGroup>
<Accordion title="403 Forbidden: Data portability policy">
**Message:** `"Your organisation requires Admin or Owner role to export data."`

**Cause:** User lacks required permissions under the org's `dataExportPolicy = 'admins_only'`.

**Resolution:** Either:
- Grant user `system.admin` permission, or
- Change org's `dataExportPolicy` to `'all'`
</Accordion>

<Accordion title="403 Forbidden: No accessible rows">
**Message:** `"You don't have permission to export any {entity} records."`

**Cause:** After applying the accessible-union predicate, the user has zero visible rows (extremely rare — even users with no permissions can typically see `MY` and `POOL`).

**Resolution:** Grant appropriate permissions or team memberships.
</Accordion>

<Accordion title="400 Bad Request: Invalid selectedIds">
**Message:** `"selectedIds must be an array of valid UUIDs"`

**Cause:** Malformed `selectedIds` parameter.

**Resolution:** Ensure `selectedIds` is an array of valid UUID strings.
</Accordion>

<Accordion title="400 Bad Request: Invalid filters">
**Message:** `"Invalid filter structure"`

**Cause:** Malformed `filters` parameter that doesn't match the expected FilterGroup schema.

**Resolution:** Validate filter structure against the CRM filter specification.
</Accordion>

<Accordion title="500 Internal Server Error: Query timeout">
**Cause:** Export query exceeded database timeout limit (typically on very large result sets with complex filters).

**Resolution:** Narrow the result set using filters or date ranges.
</Accordion>
</AccordionGroup>

## Testing guidelines

### Required test coverage

<CardGroup cols={2}>
<Card title="Permission tests" icon="shield-check">
- Data portability policy enforcement (`all` vs `admins_only`)
- System bypass behavior
- Accessible-union correctness per entity
- `selectedIds` cannot widen visibility
</Card>

<Card title="Row-scoping tests" icon="filter">
- MY scope returns only user's created/staked rows
- ALL scope returns correct union for each permission level
- Team visibility respects `team_crm.view` / `team_sales.view`
- Pool rows visible to all users
</Card>

<Card title="Format tests" icon="file">
- CSV encoding (UTF-8, quote escaping, line endings)
- XLSX structure (sheets, columns, types)
- Filename generation
- Empty result sets
</Card>

<Card title="Regression tests" icon="bug">
- `resolveAccessible*Fragment` null-return invariant
- Cross-org RLS enforcement
- `teamId` parameter ignored
- Large result set streaming
</Card>
</CardGroup>

### Critical invariant tests

<Warning>
**Must maintain regression coverage for:**

1. **The null-fragment invariant:** Verify that `resolveAccessible*Fragment(user)` returns `null` **if and only if** the user has `canViewAll` for that entity.

2. **No accidental ALL exposure:** Ensure a user with zero permissions cannot export the entire org's data (should get MY + POOL for leads; MY only for deals/contacts/companies).

3. **RLS backstop:** Cross-org export attempts must fail at the database level even if application logic is bypassed.
</Warning>
