Authoritative description of the CRM data-export endpoints (`GET /{entity}/export`) for **Lead**, **Deal**, **Contact**, and **Company**: what they return, who is allowed to call them, and exactly how the result set is scoped.

<Note>
**Scope of this document.** This covers user-facing CSV/XLSX exports of CRM entities. It does **not** cover import (see `CRM_MODULE_SPECIFICATION.md`), audit-log exports, or analytics report downloads.
</Note>

## Endpoints at a glance

| Entity  | Endpoint                   | Service method                   | Scoping model        |
| ------- | -------------------------- | -------------------------------- | -------------------- |
| Lead    | `GET /v1/leads/export`     | `LeadService.exportLeads`        | **Accessible union** |
| Deal    | `GET /v1/deals/export`     | `DealService.exportDeals`        | **Accessible union** |
| Contact | `GET /v1/contacts/export`  | `ContactService.exportContacts`  | **Accessible union** |
| Company | `GET /v1/companies/export` | `CompanyService.exportCompanies` | **Accessible union** |

<Info>
All four exports use the same **permission-free accessible-union model**: no `crm.view`, `sales.view`, or `team_crm.view` gate at the export endpoint, only the **data-portability policy** (Â§3) and the caller's own per-entity visibility (Â§4/Â§5).
</Info>

All four exports share the **same two hard gates** (Â§3), the **same RLS backstop** (Â§6), the **same filter/`selectedIds` narrowing semantics** (Â§7), and the **same file-format rules** (Â§8).

## The two-layer mental model

Every export answer is the intersection of two independent questions:

<Steps>
<Step title="Are you allowed to export at all?">
The **data-portability policy** gate (`assertDataPortabilityPolicy`). This is an org-level governance switch, not a CRM-visibility check. See [Gate 1 â€” Data-portability policy](#gate-1-data-portability-policy-all-four-exports).
</Step>

<Step title="Which rows are you allowed to see?">
The **row-scoping** model, which differs per entity. See [Gate 2 â€” Lead row-scoping](#gate-2-lead-row-scoping-accessible-union-model) and [Gate 2 â€” Deal/Contact/Company row-scoping](#gate-2-deal-contact-company-row-scoping-accessible-union-model).
</Step>
</Steps>

<Warning>
These gates are orthogonal. Passing gate 1 never widens gate 2, and vice versa. A user who clears the portability policy still only ever receives rows their CRM visibility allows.
</Warning>

## Gate 1 â€” Data-portability policy (all four exports)

Enforced in the controller **before** the service runs:

```ts
await this.accessService.assertDataPortabilityPolicy(user, 'export');
```

### Policy enforcement logic

`AccessService.assertDataPortabilityPolicy(user, 'export')` logic (`src/modules/auth/services/access.service.ts`):

<Steps>
<Step title="System bypass check">
System admin / system owner (`hasSystemBypass`) â†’ always allowed, short-circuit.
</Step>

<Step title="Resolve org policy">
Resolve the org's `settings.dataExportPolicy`, defaulting to `DEFAULT_DATA_PORTABILITY_POLICY` when unset.
</Step>

<Step title="Apply policy rules">
- If policy is `'all'` â†’ **any authenticated member** of the org may export.
- If policy is `'admins_only'` â†’ caller **must** hold `OrgPermissionKey.SYSTEM_ADMIN`, otherwise `403 ForbiddenException`: `"Your organisation requires Admin or Owner role to export data."`
</Step>
</Steps>

### Policy values

| `dataExportPolicy`          | Who can export                                                 |
| --------------------------- | -------------------------------------------------------------- |
| `all`                       | Any authenticated org member (subject to row-scoping in Â§4/Â§5) |
| `admins_only` **(default)** | Only Admin / Owner (`system.admin`) + system bypass            |

<Warning>
**Important default.** `DEFAULT_DATA_PORTABILITY_POLICY = 'admins_only'`. Out of the box, **regular agents cannot export anything** until an Owner/Admin flips the org's export policy to `all`. The "any agent can export what they see" behaviour is only true once the policy is `all`.
</Warning>

The policy is a per-org setting (`Organization.settings.dataExportPolicy`), editable by org admins via the organization settings API. The frontend mirrors the same rule in `src/lib/data-portability-policy.ts` and matches the 403 message via `isDataPortabilityPolicyError`.

## Gate 2 â€” Lead row-scoping (accessible-union model)

The lead export has **no `crm.view` gate** and **never calls `validateTypeAccess`**. After the portability gate, the rows returned are scoped purely by the caller's own permissions and memberships. There are exactly **two scopes**:

| `type` value                                               | Rows returned                                                                  | Filename prefix |
| ---------------------------------------------------------- | ------------------------------------------------------------------------------ | --------------- |
| `my`                                                       | Only leads the user **created** or is a **direct (user-level) stakeholder** on | `leads-my-`     |
| anything else â€” omitted, `all`, `pool`, `team`, `teamPool` | **Accessible union** of every tab the user can see (see below)                 | `leads-all-`    |

### How the accessible union is built

`exportLeads` (`src/modules/crm/lead/lead.service.ts`):

```ts
// type !== MY â†’ resolve the access predicate first (async), then build QB synchronously
const accessFragment =
  type !== LeadViewType.MY ? await this.resolveAccessibleLeadsFragment(em, user) : null;

if (type === LeadViewType.MY) {
  qb = this.buildLeadQueryForType(em, user, LeadViewType.MY); // created_by OR stakeholder
} else {
  qb = this.buildLeadQueryForType(em, user, LeadViewType.ALL); // base query, no access filter
  if (accessFragment) {
    qb.andWhere(accessFragment.sql, accessFragment.params); // OR-combined predicate
  }
}
```

`resolveAccessibleLeadsFragment(em, user)`:

- **Returns `null`** when the user has org-wide CRM view (`crm.view` â†’ `canViewAll`). A `null` fragment means **no access predicate is added**, so the user sees every (non-deleted) lead in the org â€” equivalent to the `ALL` tab. This is the **only** path that returns `null`.
- **Otherwise** it ORs together one fragment per tab the user can see:
  - **MY** â€” always (`myLeadsFragment`: `created_by_id = user` OR a direct `entity_stakeholder` row for the user).
  - **POOL** â€” always (`poolLeadsFragment`: `NOT EXISTS` any stakeholder row â†’ unassigned leads).
  - **TEAM** â€” one `teamLeadsFragment` per team returned by `getAvailableTeams(em, user, orgId, false)` (teams where the user holds `team_crm.view`), **only if** the `team` tab is available.
  - **TEAM_POOL** â€” one `teamPoolLeadsFragment` per team returned by `getAllMemberTeams(em, user.id, false)` (teams the user is a member of), **only if** the `teamPool` tab is available.

The fragments are combined into a single `(... OR ... OR ...)` SQL predicate, so deduplication is automatic â€” a lead visible via multiple tabs appears once.

### Why this matches "only what the user can see"

The export reuses the **exact same** building blocks as the lead list/kanban view (the canonical definition of "what the user sees"):

<Check>
- The list view's per-tab filters delegate to the **same** fragment methods the export ORs together:
  - `applyMyLeadsFilter â†’ myLeadsFragment`
  - `applyPoolLeadsFilter â†’ poolLeadsFragment`
  - `applyTeamLeadsFilter â†’ teamLeadsFragment`
  - `applyTeamPoolLeadsFilter â†’ teamPoolLeadsFragment`
- The list/kanban view resolves tabs and teams with the **same** helpers: `getAvailableTabs`, `getAvailableTeams`, `getAllMemberTeams`.
</Check>

Because the export is a literal OR of the same per-tab predicates, it can never return a lead the list view wouldn't show for that user.

### Tab availability rules

`getAvailableTabs` determines which tabs are visible:

| Tab        | Available when                                |
| ---------- | --------------------------------------------- |
| `all`      | `crm.view` (`canViewAll`)                     |
| `my`       | always                                        |
| `pool`     | always â€” **anyone can see unassigned leads**  |
| `team`     | `crm.view` **or** `team_crm.view`             |
| `teamPool` | `crm.view` **or** the user belongs to â‰¥1 team |

<Info>
**Product note on POOL.** Because `pool` is unconditionally available, a user with zero permissions and zero teams can still export **every unassigned lead in the org**. This is consistent with the list view (which also shows them the pool), so the export is faithful â€” but it is a deliberate visibility-model decision, not an export quirk. If pool should be narrower, fix it in `getAvailableTabs`/`poolLeadsFragment` and the export inherits the fix automatically.
</Info>

### The `teamId` parameter

<Warning>
The lead export accepts a `teamId` query param for backward compatibility but **does not use it**. The union is always computed server-side from the caller's own memberships/permissions, so a caller-supplied `teamId` cannot be used to read another team's leads. There is no cross-team access risk.
</Warning>

### Implementation footgun (maintainers read this)

<Warning>
The union path builds the **unfiltered** `ALL` base query and then _conditionally_ adds the predicate. The safety of this branch rests entirely on the invariant:

> `resolveAccessibleLeadsFragment` returns `null` **if and only if** the user has `canViewAll`.

If a future refactor ever makes that method return `null`/empty on any other path (e.g. an early `return null` when the fragment list is empty), **every non-privileged caller would silently export the entire org's leads**, and the RLS backstop (Â§6) would not catch it (same org). When touching `resolveAccessibleLeadsFragment`, preserve the invariant and keep the regression test that asserts a no-permission user gets a predicate containing both `created_by_id` (MY) and `NOT EXISTS` (POOL).
</Warning>

## Gate 2 â€” Deal / Contact / Company row-scoping (accessible-union model)

Deal, Contact, and Company exports use the **same permission-free accessible-union model as leads**. `validateTypeAccess` is **never called** by any of these export endpoints. Two scopes are offered by the frontend:

| Frontend scope | Sent `type`             | Rows returned                                                                        |
| -------------- | ----------------------- | ------------------------------------------------------------------------------------ |
| `my`           | `MY` (entity-specific)  | Only deals/contacts/companies the user **created** or is a **direct stakeholder** on |
| `all`          | `ALL` (entity-specific) | **Accessible union** â€” every row the user can see across their tabs                  |
| `selected`     | `ALL` + `selectedIds`   | Same as `all` but AND-intersected with the supplied ID list                          |

### How the accessible union is built per entity

Each entity has a `resolveAccessible<Entity>Fragment(em, user)` method that mirrors the lead pattern:

<Accordion title="Returns null for org-wide view">
**Returns `null`** when the user has the org-wide view permission for that entity (`crm.view` for contacts/companies; `sales.view` for deals) â€” they see every row.
</Accordion>

<Accordion title="Otherwise returns OR-combined predicate">
**Otherwise** returns an OR-combined SQL predicate covering:
- **MY**: rows the user created or is a direct stakeholder on.
- **TEAM**: rows for each team where the user holds `team_sales.view` (deals) or `team_crm.view` (contacts/companies).
  - _Non-privileged team members_ (members without `team_crm.view`): can still see rows where they are a personal stakeholder within that team's scope.
- **DEAL-LINKED contacts** (contact export only): contacts linked to active deals, **if** the user holds `sales.view`.
</Accordion>

<Info>
The `teamId` query param is **ignored** for all three â€” the union is always computed server-side from the caller's own memberships/permissions.
</Info>

### Why the same invariant applies

<Warning>
The same footgun from [Lead row-scoping implementation](#implementation-footgun-maintainers-read-this) applies to all three entities. Each `resolveAccessible<Entity>Fragment` must return `null` **if and only if** the user has `canViewAll` for that entity. The same regression tests cover this invariant.
</Warning>

## RLS backstop (all four exports)

Every export runs inside:

```ts
return this.tenantContext.executeReadOnly(organizationId, async (em) => { ... });
```

<Check>
This sets the tenant (organization) context, so **PostgreSQL row-level security physically scopes every query to the caller's organization**. This provides a defense-in-depth layer that prevents accidental cross-org data leaks even if the application-level accessible-union logic has a bug.
</Check>

The RLS backstop ensures that:
- No query can return rows from a different organization
- All database operations are automatically scoped to the current tenant
- Cross-organization data access is physically impossible at the database level

<Note>
While RLS provides critical protection against cross-org leaks, it does **not** enforce intra-org row visibility (e.g., team boundaries, stakeholder relationships). The accessible-union logic (Â§4, Â§5) is responsible for those finer-grained controls.
</Note>

## Filter and `selectedIds` narrowing semantics

All four exports support additional narrowing via query parameters:

### Standard filters

Each export endpoint accepts entity-specific filters that further narrow the accessible-union result set:

<Tabs>
<Tab title="Leads">
- `status`: Filter by lead status(es)
- `source`: Filter by lead source(s)
- `assignedTo`: Filter by assigned user(s)
- `tags`: Filter by tag(s)
- Custom field filters
</Tab>

<Tab title="Deals">
- `stage`: Filter by pipeline stage(s)
- `status`: Filter by deal status
- `assignedTo`: Filter by assigned user(s)
- `tags`: Filter by tag(s)
- Custom field filters
</Tab>

<Tab title="Contacts">
- `tags`: Filter by tag(s)
- `assignedTo`: Filter by assigned user(s)
- Custom field filters
</Tab>

<Tab title="Companies">
- `tags`: Filter by tag(s)
- `industry`: Filter by industry
- Custom field filters
</Tab>
</Tabs>

### The `selectedIds` parameter

The `selectedIds` query parameter accepts a comma-separated list of entity IDs:

```http
GET /v1/leads/export?type=all&selectedIds=123,456,789
```

<Steps>
<Step title="Compute accessible union">
The system first computes the full accessible-union result set for the user (based on their permissions and memberships).
</Step>

<Step title="Intersect with selected IDs">
The result set is then AND-intersected with the supplied ID list. Only IDs that appear in **both** the accessible union and the `selectedIds` list are exported.
</Step>

<Step title="Return filtered results">
The final export contains only the rows that passed both filters.
</Step>
</Steps>

<Warning>
**Security note:** `selectedIds` can only **narrow** the export, never widen it. A user cannot use `selectedIds` to export rows they don't have access to. If an ID in `selectedIds` is not in the accessible union, it is silently ignored.
</Warning>

### Filter combination rules

When multiple filters are provided:
- All filters are AND-combined
- `selectedIds` (if provided) is the final AND filter applied after all other predicates
- Empty filter values are ignored (treated as "no filter for this field")

## File format rules

All four exports support two output formats: CSV and XLSX.

### Format selection

The format is determined by the `Accept` header or a `format` query parameter:

| Accept header / format param | Output format |
| ---------------------------- | ------------- |
| `text/csv` / `csv`           | CSV           |
| `application/vnd.openxmlformats-officedocument.spreadsheetml.sheet` / `xlsx` | XLSX |
| Not specified                | CSV (default) |

### CSV format specifications

<CardGroup cols={2}>
<Card title="Encoding" icon="file-code">
UTF-8 with BOM for Excel compatibility
</Card>

<Card title="Delimiter" icon="comma">
Comma (`,`) separator
</Card>

<Card title="Quoting" icon="quote-left">
Double-quote fields containing commas, newlines, or quotes
</Card>

<Card title="Line endings" icon="arrow-turn-down-right">
CRLF (`\r\n`) for Windows compatibility
</Card>
</CardGroup>

### XLSX format specifications

<CardGroup cols={2}>
<Card title="Sheet name" icon="table">
Entity name (e.g., "Leads", "Deals")
</Card>

<Card title="Header row" icon="heading">
Bold, frozen first row
</Card>

<Card title="Column widths" icon="arrows-left-right">
Auto-sized to content
</Card>

<Card title="Data types" icon="hashtag">
Preserved (numbers, dates, booleans)
</Card>
</CardGroup>

### Common field mapping

Both formats include:
- All standard fields for the entity
- All custom fields (if any)
- Related entity names (not IDs) for foreign keys
- Formatted timestamps in the org's timezone
- Tag lists as comma-separated values
- Multi-select custom fields as comma-separated values

### Filename conventions

Export files follow this naming pattern:

```
{entity}-{scope}-{timestamp}.{ext}
```

Examples:
- `leads-my-2024-01-15T14-30-45.csv`
- `deals-all-2024-01-15T14-30-45.xlsx`
- `contacts-my-2024-01-15T14-30-45.csv`

<Info>
The `{scope}` segment is either `my` (for MY type exports) or `all` (for accessible-union exports), matching the scoping model described in Â§4 and Â§5.
</Info>

## Performance considerations

<Warning>
**Large exports.** Exports are generated synchronously and can time out for very large datasets. Consider implementing:
- Pagination or chunking for exports over 10,000 rows
- Background job processing for exports over 50,000 rows
- Client-side timeout handling (recommend 60-120 second timeout)
</Warning>

<Tip>
**Query optimization.** The accessible-union queries use indexed fields (`created_by_id`, stakeholder relationships) and should perform well up to ~100K rows per entity. Monitor slow query logs if export times exceed acceptable thresholds.
</Tip>

## Testing checklist

When modifying export logic, verify:

<Steps>
<Step title="Data portability policy enforcement">
- [ ] System admin/owner bypass works
- [ ] `admins_only` policy blocks regular users
- [ ] `all` policy allows any org member
- [ ] Correct 403 error message
</Step>

<Step title="Row-scoping accuracy">
- [ ] MY scope returns only user's created/stakeholder rows
- [ ] ALL scope returns exact accessible-union (no more, no less)
- [ ] No cross-org data leaks (RLS backstop test)
- [ ] `selectedIds` only narrows, never widens
</Step>

<Step title="Fragment null invariant">
- [ ] `resolveAccessible*Fragment` returns null **only** for `canViewAll`
- [ ] Non-privileged user gets MY + POOL predicate (leads)
- [ ] Team visibility correctly reflected in union
</Step>

<Step title="File format compliance">
- [ ] CSV: UTF-8 BOM, proper quoting, CRLF line endings
- [ ] XLSX: frozen header, auto-sized columns, preserved types
- [ ] Filename matches `{entity}-{scope}-{timestamp}.{ext}` pattern
</Step>
</Steps>
