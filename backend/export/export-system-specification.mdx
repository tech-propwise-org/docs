Authoritative description of the CRM data-export endpoints (`GET /{entity}/export`) for **Lead**, **Deal**, **Contact**, and **Company**: what they return, who is allowed to call them, and exactly how the result set is scoped.

<Note>
**Scope of this document:** This covers user-facing CSV/XLSX exports of CRM entities. It does **not** cover import operations, audit-log exports, or analytics report downloads.
</Note>

## Endpoints at a glance

| Entity  | Endpoint                   | Service method                   | Scoping model        |
| ------- | -------------------------- | -------------------------------- | -------------------- |
| Lead    | `GET /v1/leads/export`     | `LeadService.exportLeads`        | **Accessible union** |
| Deal    | `GET /v1/deals/export`     | `DealService.exportDeals`        | **Accessible union** |
| Contact | `GET /v1/contacts/export`  | `ContactService.exportContacts`  | **Accessible union** |
| Company | `GET /v1/companies/export` | `CompanyService.exportCompanies` | **Accessible union** |

<Info>
All four exports use the same **permission-free accessible-union model**: no `crm.view`, `sales.view`, or `team_crm.view` gate at the export endpoint, only the **data-portability policy** and the caller's own per-entity visibility.
</Info>

All four exports share:
- The **same two hard gates** (data portability policy)
- The **same RLS backstop**
- The **same filter/`selectedIds` narrowing semantics**
- The **same file-format rules**

## The two-layer mental model

Every export answer is the intersection of two independent questions:

<Steps>
<Step title="Are you allowed to export at all?">
The **data-portability policy** gate (`assertDataPortabilityPolicy`). This is an org-level governance switch, not a CRM-visibility check.
</Step>

<Step title="Which rows are you allowed to see?">
The **row-scoping** model, which differs per entity based on the user's permissions and memberships.
</Step>
</Steps>

<Warning>
These gates are orthogonal. Passing gate 1 never widens gate 2, and vice versa. A user who clears the portability policy still only ever receives rows their CRM visibility allows.
</Warning>

## Gate 1 — Data-portability policy (all four exports)

Enforced in the controller **before** the service runs:

```ts
await this.accessService.assertDataPortabilityPolicy(user, 'export');
```

### Policy enforcement logic

`AccessService.assertDataPortabilityPolicy(user, 'export')` in `src/modules/auth/services/access.service.ts`:

<Steps>
<Step title="System bypass check">
System admin / system owner (`hasSystemBypass`) → always allowed, short-circuit.
</Step>

<Step title="Resolve org policy">
Resolve the org's `settings.dataExportPolicy`, defaulting to `DEFAULT_DATA_PORTABILITY_POLICY` when unset.
</Step>

<Step title="Apply policy rules">
- If policy is `'all'` → **any authenticated member** of the org may export
- If policy is `'admins_only'` → caller **must** hold `OrgPermissionKey.SYSTEM_ADMIN`, otherwise `403 ForbiddenException`: `"Your organisation requires Admin or Owner role to export data."`
</Step>
</Steps>

### Policy values

| `dataExportPolicy`          | Who can export                                                 |
| --------------------------- | -------------------------------------------------------------- |
| `all`                       | Any authenticated org member (subject to row-scoping)          |
| `admins_only` **(default)** | Only Admin / Owner (`system.admin`) + system bypass            |

<Warning>
**Important default:** `DEFAULT_DATA_PORTABILITY_POLICY = 'admins_only'`. Out of the box, **regular agents cannot export anything** until an Owner/Admin flips the org's export policy to `all`. The "any agent can export what they see" behaviour is only true once the policy is `all`.
</Warning>

The policy is a per-org setting (`Organization.settings.dataExportPolicy`), editable by org admins via the organization settings API. The frontend mirrors the same rule in `src/lib/data-portability-policy.ts` and matches the 403 message via `isDataPortabilityPolicyError`.

## Gate 2 — Lead row-scoping (accessible-union model)

The lead export has **no `crm.view` gate** and **never calls `validateTypeAccess`**. After the portability gate, the rows returned are scoped purely by the caller's own permissions and memberships.

### Two scopes available

| `type` value                                               | Rows returned                                                                  | Filename prefix |
| ---------------------------------------------------------- | ------------------------------------------------------------------------------ | --------------- |
| `my`                                                       | Only leads the user **created** or is a **direct (user-level) stakeholder** on | `leads-my-`     |
| anything else — omitted, `all`, `pool`, `team`, `teamPool` | **Accessible union** of every tab the user can see                             | `leads-all-`    |

### How the accessible union is built

`exportLeads` in `src/modules/crm/lead/lead.service.ts`:

```ts
// type !== MY → resolve the access predicate first (async), then build QB synchronously
const accessFragment =
  type !== LeadViewType.MY ? await this.resolveAccessibleLeadsFragment(em, user) : null;

if (type === LeadViewType.MY) {
  qb = this.buildLeadQueryForType(em, user, LeadViewType.MY); // created_by OR stakeholder
} else {
  qb = this.buildLeadQueryForType(em, user, LeadViewType.ALL); // base query, no access filter
  if (accessFragment) {
    qb.andWhere(accessFragment.sql, accessFragment.params); // OR-combined predicate
  }
}
```

### Access fragment resolution

`resolveAccessibleLeadsFragment(em, user)`:

- **Returns `null`** when the user has org-wide CRM view (`crm.view` → `canViewAll`). A `null` fragment means **no access predicate is added**, so the user sees every (non-deleted) lead in the org — equivalent to the `ALL` tab. This is the **only** path that returns `null`.

- **Otherwise** it ORs together one fragment per tab the user can see:
  - **MY** — always (`myLeadsFragment`: `created_by_id = user` OR a direct `entity_stakeholder` row for the user)
  - **POOL** — always (`poolLeadsFragment`: `NOT EXISTS` any stakeholder row → unassigned leads)
  - **TEAM** — one `teamLeadsFragment` per team returned by `getAvailableTeams(em, user, orgId, false)` (teams where the user holds `team_crm.view`), **only if** the `team` tab is available
  - **TEAM_POOL** — one `teamPoolLeadsFragment` per team returned by `getAllMemberTeams(em, user.id, false)` (teams the user is a member of), **only if** the `teamPool` tab is available

<Info>
The fragments are combined into a single `(... OR ... OR ...)` SQL predicate, so deduplication is automatic — a lead visible via multiple tabs appears once.
</Info>

### Why this matches "only what the user can see"

The export reuses the **exact same** building blocks as the lead list/kanban view (the canonical definition of "what the user sees"):

<Accordion title="Fragment method mapping">
The list view's per-tab filters delegate to the **same** fragment methods the export ORs together:
- `applyMyLeadsFilter → myLeadsFragment`
- `applyPoolLeadsFilter → poolLeadsFragment`
- `applyTeamLeadsFilter → teamLeadsFragment`
- `applyTeamPoolLeadsFilter → teamPoolLeadsFragment`

The list/kanban view resolves tabs and teams with the **same** helpers: `getAvailableTabs`, `getAvailableTeams`, `getAllMemberTeams`.
</Accordion>

Because the export is a literal OR of the same per-tab predicates, it can never return a lead the list view wouldn't show for that user.

### Tab availability rules

`getAvailableTabs` determines which tabs are available:

| Tab        | Available when                                |
| ---------- | --------------------------------------------- |
| `all`      | `crm.view` (`canViewAll`)                     |
| `my`       | always                                        |
| `pool`     | always — **anyone can see unassigned leads**  |
| `team`     | `crm.view` **or** `team_crm.view`             |
| `teamPool` | `crm.view` **or** the user belongs to ≥1 team |

<Note>
**Product note on POOL:** Because `pool` is unconditionally available, a user with zero permissions and zero teams can still export **every unassigned lead in the org**. This is consistent with the list view (which also shows them the pool), so the export is faithful — but it is a deliberate visibility-model decision, not an export quirk.
</Note>

### `teamId` parameter behavior

<Warning>
The lead export accepts a `teamId` query param for backward compatibility but **does not use it**. The union is always computed server-side from the caller's own memberships/permissions, so a caller-supplied `teamId` cannot be used to read another team's leads. There is no cross-team access risk.
</Warning>

### Implementation footgun (maintainers)

<Warning>
The union path builds the **unfiltered** `ALL` base query and then _conditionally_ adds the predicate. The safety of this branch rests entirely on the invariant:

> `resolveAccessibleLeadsFragment` returns `null` **if and only if** the user has `canViewAll`.

If a future refactor ever makes that method return `null`/empty on any other path (e.g. an early `return null` when the fragment list is empty), **every non-privileged caller would silently export the entire org's leads**, and the RLS backstop would not catch it (same org). 

When touching `resolveAccessibleLeadsFragment`, preserve the invariant and keep the regression test that asserts a no-permission user gets a predicate containing both `created_by_id` (MY) and `NOT EXISTS` (POOL).
</Warning>

## Gate 2 — Deal / Contact / Company row-scoping

Deal, Contact, and Company exports use the **same permission-free accessible-union model as leads**. `validateTypeAccess` is **never called** by any of these export endpoints.

### Frontend scope options

| Frontend scope | Sent `type`             | Rows returned                                                                        |
| -------------- | ----------------------- | ------------------------------------------------------------------------------------ |
| `my`           | `MY` (entity-specific)  | Only deals/contacts/companies the user **created** or is a **direct stakeholder** on |
| `all`          | `ALL` (entity-specific) | **Accessible union** — every row the user can see across their tabs                  |
| `selected`     | `ALL` + `selectedIds`   | Same as `all` but AND-intersected with the supplied ID list                          |

### How the accessible union is built per entity

Each entity has a `resolveAccessible<Entity>Fragment(em, user)` method that mirrors the lead pattern:

<Steps>
<Step title="Check for org-wide view permission">
**Returns `null`** when the user has the org-wide view permission for that entity:
- `crm.view` for contacts/companies
- `sales.view` for deals

When `null` is returned, they see every row.
</Step>

<Step title="Build OR-combined predicate">
**Otherwise** returns an OR-combined SQL predicate covering:

- **MY**: rows the user created or is a direct stakeholder on
- **TEAM**: rows for each team where the user holds:
  - `team_sales.view` (deals)
  - `team_crm.view` (contacts/companies)
  - _Non-privileged team members_ (members without `team_crm.view`): can still see rows where they are a personal stakeholder within that team's scope
- **DEAL-LINKED contacts** (contact export only): contacts linked to active deals, **if** the user holds `sales.view`
</Step>
</Steps>

<Info>
The `teamId` query param is **ignored** for all three entities — the union is always computed server-side from the caller's own memberships/permissions.
</Info>

### Same invariant applies

<Warning>
The same footgun from lead exports applies to all three entities. Each `resolveAccessible<Entity>Fragment` must return `null` **if and only if** the user has `canViewAll` for that entity. The same regression tests cover this invariant.
</Warning>

## RLS backstop (all four exports)

Every export runs inside:

```ts
return this.tenantContext.executeReadOnly(organizationId, async (em) => { ... });
```

<Check>
This sets the tenant (organization) context, so **PostgreSQL row-level security physically scopes every query to the caller's organization**. Even if the access-fragment logic had a bug that widened the predicate, the database would still enforce org isolation at the row level.
</Check>

The RLS backstop provides defense-in-depth:
- Application-level access fragments control which rows within the org are visible
- Database-level RLS ensures cross-org data never leaks, regardless of application bugs

## Filter and `selectedIds` narrowing

All four exports support optional narrowing of the result set:

### Filter parameters

Standard entity filters can be applied to narrow the export:
- Status filters
- Date ranges
- Custom field values
- Search terms

Filters are applied **after** the accessible-union predicate, so they can only narrow (never widen) the result set.

### `selectedIds` parameter

When `selectedIds` is provided:

<Steps>
<Step title="Build accessible union">
The normal accessible-union predicate is computed based on the user's permissions and memberships.
</Step>

<Step title="Intersect with selectedIds">
The query is AND-intersected with `id IN (selectedIds)`, so only rows that are both:
- In the user's accessible union
- In the provided ID list

will be exported.
</Step>
</Steps>

<Warning>
A caller cannot use `selectedIds` to read rows outside their accessible union. The accessible-union predicate is always applied, and `selectedIds` can only narrow further.
</Warning>

## File format rules

All four exports follow the same file generation rules:

### Format options

<Tabs>
  <Tab title="CSV">
    - UTF-8 encoding with BOM
    - Comma-separated values
    - Quoted fields when containing commas/newlines
    - Extension: `.csv`
  </Tab>
  <Tab title="XLSX">
    - Excel 2007+ format
    - Single sheet named after the entity
    - First row contains headers
    - Extension: `.xlsx`
  </Tab>
</Tabs>

### Filename convention

Format: `{entity}-{scope}-{timestamp}.{ext}`

Examples:
- `leads-my-2024-01-15T10-30-45Z.csv`
- `deals-all-2024-01-15T10-30-45Z.xlsx`
- `contacts-my-2024-01-15T10-30-45Z.csv`
- `companies-all-2024-01-15T10-30-45Z.xlsx`

### Column inclusion

<AccordionGroup>
  <Accordion title="Standard fields">
    All standard entity fields are included:
    - ID
    - Name/title
    - Status
    - Created/updated timestamps
    - Owner/creator
    - Entity-specific fields (e.g., deal value, contact email)
  </Accordion>

  <Accordion title="Custom fields">
    All custom fields defined for the entity type are included with their display names as column headers.
  </Accordion>

  <Accordion title="Relationships">
    Related entity references are included as:
    - Related entity name (for display)
    - Related entity ID (for re-import/reference)
  </Accordion>

  <Accordion title="System fields">
    System metadata is included:
    - Organization ID
    - Soft-delete status
    - Audit timestamps
  </Accordion>
</AccordionGroup>

### Sorting

Exports are sorted by:
1. Created date (descending) — newest first
2. ID (ascending) — for stable ordering of same-second creates

This ensures consistent export ordering across multiple runs.

  <Card title="Data portability gate" icon="shield-check">
    Org-level policy controls who can export (admins-only by default)
  </Card>
  <Card title="Row-level scoping" icon="filter">
    Users only see rows their permissions allow — accessible union model
  </Card>
  <Card title="RLS backstop" icon="database">

  </Card>
  <Card title="No privilege escalation" icon="lock">
    Export never grants wider access than list view — same predicates
  </Card>

### Key security properties

<Check>
**Principle of least privilege:** Export cannot be used to read data the user couldn't already see in the list view.
</Check>

<Check>
**Org isolation:** RLS ensures cross-org data never leaks, even if application logic has bugs.
</Check>

<Check>
**Governance control:** Orgs can restrict export to admins-only via data portability policy.
</Check>

<Check>
**No team-hopping:** `teamId` parameter is ignored; users cannot specify another team's ID to read their data.
</Check>

## Testing guidelines

Regression tests must verify:

<Steps>
<Step title="Data portability policy enforcement">
- System admin/owner always passes
- `admins_only` policy blocks non-admins
- `all` policy allows any org member
- Correct 403 message when blocked
</Step>

<Step title="Accessible-union correctness">
- `resolveAccessible<Entity>Fragment` returns `null` if and only if user has `canViewAll`
- No-permission user gets predicate containing MY + POOL fragments (leads)
- Team member without `team_crm.view` only sees personal stakeholder rows
- `canViewAll` user gets no predicate (sees all rows)
</Step>

<Step title="Row-level scoping matches list view">
- Export returns exact same row IDs as list view for same user
- Filters narrow result set correctly
- `selectedIds` cannot widen beyond accessible union
</Step>

<Step title="RLS backstop">
- Cross-org queries return zero rows (even with bad predicate)
- Tenant context is correctly set for all export queries
</Step>
</Steps>
