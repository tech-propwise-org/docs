This document provides the authoritative description of CRM data-export endpoints (`GET /{entity}/export`) for **Lead**, **Deal**, **Contact**, and **Company**: what they return, who is allowed to call them, and exactly how the result set is scoped.

<Note>
This covers user-facing CSV/XLSX exports of CRM entities. It does **not** cover import functionality, audit-log exports, or analytics report downloads.
</Note>

## Two-layer security model

Every export request must pass through two independent validation layers:

<Steps>
  <Step title="Data portability policy check">
    **"Are you allowed to export at all?"** 
    
    This is an org-level governance switch enforced by `assertDataPortabilityPolicy`. It's not a CRM-visibility check but a data governance control.
  </Step>
  <Step title="Row-level scoping">
    **"Which rows are you allowed to see?"**
    
    This determines which specific records the user can access based on their permissions and memberships.
  </Step>
</Steps>

<Warning>
These layers are orthogonal. Passing the portability policy never widens row access, and vice versa. A user who clears the portability policy still only receives rows their CRM visibility allows.
</Warning>

## Data portability policy (Layer 1)

The data portability policy is enforced in the controller **before** any service runs:

```ts
await this.accessService.assertDataPortabilityPolicy(user, 'export');
```

### Policy enforcement logic

<Accordion title="AccessService.assertDataPortabilityPolicy implementation">
Located in `src/modules/auth/services/access.service.ts`:

1. **System bypass**: System admin / system owner (`hasSystemBypass`) → always allowed, short-circuit
2. Resolve the org's `settings.dataExportPolicy`, defaulting to `DEFAULT_DATA_PORTABILITY_POLICY` when unset
3. If policy is `'all'` → **any authenticated member** of the org may export
4. If policy is `'admins_only'` → caller **must** hold `OrgPermissionKey.SYSTEM_ADMIN`, otherwise `403 ForbiddenException`
</Accordion>

### Policy values

| `dataExportPolicy`          | Who can export                                                 |
| --------------------------- | -------------------------------------------------------------- |
| `all`                       | Any authenticated org member (subject to row-scoping) |
| `admins_only` **(default)** | Only Admin / Owner (`system.admin`) + system bypass            |

<Warning>
**Important default**: `DEFAULT_DATA_PORTABILITY_POLICY = 'admins_only'`. Out of the box, **regular agents cannot export anything** until an Owner/Admin changes the org's export policy to `all`.
</Warning>

The policy is configured per-organization (`Organization.settings.dataExportPolicy`) and can be modified by org admins via the organization settings API.

## Lead row scoping (Layer 2)

Lead exports use an **accessible-union model** with no `crm.view` gate and never call `validateTypeAccess`. After passing the portability gate, rows are scoped by the caller's permissions and memberships.

### Scoping types

| `type` value | Rows returned | Filename prefix |
| ------------ | ------------- | --------------- |
| `my` | Only leads the user **created** or is a **direct stakeholder** on | `leads-my-` |
| anything else (`all`, `pool`, `team`, `teamPool`) | **Accessible union** of every tab the user can see | `leads-all-` |

### Accessible union construction

<CodeGroup>
```ts Lead Export Service Logic
// type !== MY → resolve access predicate first, then build QB
const accessFragment =
  type !== LeadViewType.MY ? await this.resolveAccessibleLeadsFragment(em, user) : null;

if (type === LeadViewType.MY) {
  qb = this.buildLeadQueryForType(em, user, LeadViewType.MY); // created_by OR stakeholder
} else {
  qb = this.buildLeadQueryForType(em, user, LeadViewType.ALL); // base query
  if (accessFragment) {
    qb.andWhere(accessFragment.sql, accessFragment.params); // OR-combined predicate
  }
}
```
</CodeGroup>

### Access fragment resolution

`resolveAccessibleLeadsFragment(em, user)` behavior:

<Tabs>
  <Tab title="Org-wide access">
    **Returns `null`** when the user has `crm.view` permission (`canViewAll`). 
    
    A `null` fragment means no access predicate is added, so the user sees every non-deleted lead in the org.
  </Tab>
  <Tab title="Limited access">
    **Returns OR-combined fragments** for each accessible tab:
    
    - **MY**: `created_by_id = user` OR direct `entity_stakeholder` row
    - **POOL**: `NOT EXISTS` any stakeholder row (unassigned leads)  
    - **TEAM**: One fragment per team with `team_crm.view` permission
    - **TEAM_POOL**: One fragment per team the user is a member of
  </Tab>
</Tabs>

### Tab availability rules

| Tab        | Available when                                |
| ---------- | --------------------------------------------- |
| `all`      | `crm.view` (`canViewAll`)                     |
| `my`       | Always available                              |
| `pool`     | Always available                              |
| `team`     | `crm.view` **or** `team_crm.view`             |
| `teamPool` | `crm.view` **or** user belongs to ≥1 team    |

<Note>
The `pool` tab is unconditionally available, meaning any user can export unassigned leads. This matches the list view behavior and is a deliberate product decision.
</Note>

### Implementation considerations

<Warning>
**Critical invariant for maintainers**: `resolveAccessibleLeadsFragment` returns `null` **if and only if** the user has `canViewAll`. 

Breaking this invariant could result in non-privileged users silently exporting entire org datasets.
</Warning>

The `teamId` query parameter is accepted for backward compatibility but **ignored**. The union is always computed server-side from the caller's memberships/permissions.

## Deal, Contact, and Company row scoping

These entities use the **same accessible-union model as leads** with no `validateTypeAccess` calls.

### Frontend scope mapping

| Frontend scope | Sent `type` | Rows returned |
| -------------- | ----------- | ------------- |
| `my` | `MY` (entity-specific) | Records the user created or is a direct stakeholder on |
| `all` | `ALL` (entity-specific) | Accessible union of all visible rows |
| `selected` | `ALL` + `selectedIds` | Accessible union intersected with supplied ID list |

### Per-entity accessible union logic

Each entity implements `resolveAccessible<Entity>Fragment(em, user)`:

<AccordionGroup>
  <Accordion title="Deal export scoping">
    - **Returns `null`** when user has `sales.view` (sees all deals)
    - **Otherwise** combines:
      - **MY**: deals created by user or where user is stakeholder
      - **TEAM**: deals for teams where user has `team_sales.view`
  </Accordion>
  <Accordion title="Contact export scoping">
    - **Returns `null`** when user has `crm.view` (sees all contacts)
    - **Otherwise** combines:
      - **MY**: contacts created by user or where user is stakeholder  
      - **TEAM**: contacts for teams where user has `team_crm.view`
      - **DEAL-LINKED**: contacts linked to active deals (if user has `sales.view`)
  </Accordion>
  <Accordion title="Company export scoping">
    - **Returns `null`** when user has `crm.view` (sees all companies)
    - **Otherwise** combines:
      - **MY**: companies created by user or where user is stakeholder
      - **TEAM**: companies for teams where user has `team_crm.view`
  </Accordion>
</AccordionGroup>

<Info>
The `teamId` query parameter is **ignored** for all three entities. The union is always computed server-side from caller permissions.
</Info>

## Row-level security backstop

All exports execute within a tenant-scoped context:

```ts
return this.tenantContext.executeReadOnly(organizationId, async (em) => { 
  // Export logic here
});
```

<Check>
This sets the tenant (organization) context, ensuring PostgreSQL row-level security physically scopes every query to the caller's organization, preventing cross-org data access.
</Check>

## Filter and ID narrowing semantics

Export endpoints support additional filtering through query parameters:

<Tabs>
  <Tab title="Filter parameters">
    Standard CRM filters can be applied to narrow the export result set:
    - Date ranges
    - Status filters  
    - Custom field filters
    
    Filters are applied **after** access scoping but **before** selectedIds intersection.
  </Tab>
  <Tab title="Selected IDs">
    When `selectedIds` is provided:
    - The accessible union is computed first
    - Filters are applied to the union
    - Result is intersected with the selectedIds list
    - Only IDs the user can actually access will be included
  </Tab>
</Tabs>

## File format specifications

<CardGroup cols={2}>
  <Card title="CSV Export" icon="file-csv">
    Standard comma-separated values format
    UTF-8 encoding with BOM
  </Card>
  <Card title="XLSX Export" icon="file-excel">
    Microsoft Excel format
    Single worksheet per entity
  </Card>
</CardGroup>

Export files follow consistent naming conventions:
- `{entity}-{scope}-{timestamp}.{format}`
- Examples: `leads-my-20231201-143022.csv`, `deals-all-20231201-143022.xlsx`

<Tip>
All exports include a timestamp in the filename to prevent conflicts and provide audit trails for data access.
</Tip>
