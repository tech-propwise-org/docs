## Overview

The audit module provides comprehensive activity tracking and timeline functionality for the PropWise CRM system. It captures all data changes across entities through PostgreSQL triggers and presents them as both raw audit logs and human-readable timeline events.

<Note>
The module supports both tenant-scoped and system-wide audit queries with strict access controls and data isolation.
</Note>

## Architecture

### Design decisions

The audit module follows several key architectural principles:

<AccordionGroup>
  <Accordion title="Database-driven auditing">
    Uses PostgreSQL triggers (`audit_trigger_func()`) to capture changes at the database level, ensuring consistency and completeness across all data modifications.
  </Accordion>

  <Accordion title="Read-only entity pattern">
    The `AuditLog` entity is strictly read-only from application code - all writes happen automatically via database triggers.
  </Accordion>

  <Accordion title="Timeline abstraction">
    Provides a higher-level timeline view that translates raw audit data into user-friendly messages for better UX.
  </Accordion>

  <Accordion title="Dual access patterns">
    Supports both tenant-scoped queries (for regular users) and system-wide queries (for system administrators).
  </Accordion>
</AccordionGroup>

### Key components

- **AuditLog Entity**: Maps to the `audit_log` table (read-only)
- **AuditService**: Core business logic for querying and formatting audit data
- **AuditTimelineResolver**: Translates audit logs into human-readable timeline events
- **Dual Controllers**: Separate endpoints for tenant and system admin access

## Entity specifications

### AuditLog entity

<CodeGroup>
```typescript AuditLog Interface
interface AuditLog {
  id: string;                    // UUID primary key
  tableName: string;             // Name of the table that changed
  action: AuditAction;           // INSERT, UPDATE, DELETE
  rowData: Record<string, any>;  // Full row data after change
  changedFields: Record<string, any>; // Only changed fields (UPDATE only)
  recordId: string;              // ID of the changed record
  tenantId: string;              // Tenant context
  userId?: string;               // User who made the change
  createdAt: Date;               // When the change occurred
}
```

```typescript AuditAction Enum
enum AuditAction {
  INSERT = 'INSERT',
  UPDATE = 'UPDATE', 
  DELETE = 'DELETE'
}
```
</CodeGroup>

## API endpoints

### Tenant-scoped endpoints

#### GET /audit/logs

Retrieve paginated audit logs for the current tenant.

<CodeGroup>
```typescript Query Parameters
interface AuditLogQueryDto {
  page?: number;        // Page number (default: 1)
  limit?: number;       // Items per page (1-100, default: 20)
  tableName?: string;   // Filter by table name
  recordId?: string;    // Filter by record ID
  userId?: string;      // Filter by user ID
  action?: AuditAction; // Filter by action type
  startDate?: string;   // ISO date string
  endDate?: string;     // ISO date string
}
```

```typescript Response Format
interface PaginatedResponse<AuditLogDto> {
  data: AuditLogDto[];
  pagination: {
    page: number;
    limit: number;
    total: number;
    totalPages: number;
  };
}
```
</CodeGroup>

#### GET /audit/timeline/:entityType/:entityId

Get resolved timeline events for a specific entity.

**Parameters:**
- `entityType`: The type of entity (e.g., 'lead', 'deal')
- `entityId`: UUID of the specific entity

```typescript Response
interface ResolvedTimeline {
  eventType: TimelineEventType;
  message: string;
  createdAt: Date;
  user?: AuditUserInfo;
}
```

## Business rules and invariants

### Data integrity

<Check>
**Immutable audit logs** - Once created by the database trigger, audit logs cannot be modified
</Check>

<Check>
**Automatic data sanitization** - Sensitive fields (passwords, refresh tokens) are automatically stripped by the trigger
</Check>

<Check>
**Transactional consistency** - All audit operations occur within the same transaction as the data change
</Check>

### Access control

<Steps>
  <Step title="Tenant isolation">
    Regular users can only access audit logs for their own tenant
  </Step>
  
  <Step title="System admin access">
    System administrators can access audit logs across all tenants
  </Step>
  
  <Step title="User information resolution">
    Timeline resolution includes user information only if accessible to the requester
  </Step>
</Steps>

### Temporal constraints

- Date range queries validate that `endDate >= startDate`
- Date filters use inclusive ranges
- Timestamps are stored in UTC

### Query patterns

<Tabs>
  <Tab title="Pagination">

  </Tab>
  
  <Tab title="Filtering">
    Supports flexible filtering with optional query parameters
  </Tab>
  
  <Tab title="Type safety">
    Uses MikroORM's FilterQuery for type-safe database queries
  </Tab>
</Tabs>

### Error handling

- Returns 404 for non-existent entities in timeline queries
- Validates UUID format for ID parameters
- Enforces pagination limits (max 100 items per page)

### Caching considerations

<Note>
Audit data is write-heavy, read-light with no caching implemented due to real-time requirements. Timeline resolution is performed on-demand.
</Note>

### Security measures

<Warning>
The audit module implements multiple security layers:
- No direct entity manipulation - all writes via database triggers
- Tenant isolation enforced at multiple layers
- Sensitive data filtered at database level before persistence
</Warning>

## Implementation guidelines

### Database triggers

<Steps>
  <Step title="Trigger installation">
    Ensure the `audit_trigger_func()` is installed on all audited tables
  </Step>
  
  <Step title="Sensitive data handling">
    Configure trigger to exclude sensitive fields from audit logs
  </Step>
  
  <Step title="Performance monitoring">
    Monitor trigger performance impact on write operations
  </Step>
</Steps>

### Timeline resolution

<Tip>
When implementing timeline resolvers, consider using a strategy pattern for different entity types to maintain clean, extensible code.
</Tip>

### Query optimization

- Use database indexes on frequently queried fields (`tenantId`, `recordId`, `createdAt`)
- Implement proper pagination to handle large audit datasets
- Consider partitioning audit tables by date for better performance
