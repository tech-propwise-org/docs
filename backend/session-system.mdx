## Overview

The session management system has been completely rebuilt with a **two-level architecture**:

1. **AuthSession** (device-level) — Created when the user logs in or verifies their email. Represents a device/browser login. One AuthSession = one device. Stores device info, IP geolocation, and the refresh token hash.
2. **OrgSession** (org-level) — Lightweight child record created when the user selects an organization. Tracks per-org access without duplicating device/IP/token data.

This design ensures:
- **No security gap after login** — AuthSession exists immediately, before org selection
- **"5 sessions" = 5 devices globally** — matches user expectations (updated default limit)
- **Lightweight org switching** — no device registration or geolocation on org selection
- **Clean separation** — device concerns (tokens, device info, IP, geolocation) on AuthSession; org concerns (per-org access, audit) on OrgSession
- **Block-on-limit enforcement** — configurable global session and device caps block login when limits are reached (no auto-revocation)
- **Per-session token storage** — each AuthSession stores its own refresh token hash, enabling true multi-device support
- **Immediate revocation enforcement** — SessionActivityInterceptor validates sessions on every authenticated request
- **Real-time presence tracking** — works from login, before org selection

The system also includes:
- Global device registry with ThumbmarkJS browser fingerprinting (max registered devices configurable via policy)
- Device removal cooldown (configurable via policy, starts when a device is removed — blocks subsequent removals, not additions)
- Device/IP/User blocking by system admins with immediate enforcement across all endpoints
- Real-time presence tracking via WebSocket heartbeats (works from login, before org selection)
- **Global session policy** — seeded at server startup from SESSION_DEFAULTS constants, fully configurable by system admins, with optional per-user overrides and in-memory caching
- Pre-request security validation via SessionActivityInterceptor (inactivity timeout, device blocking, IP blocking, session revocation, refresh token validation via cookie or `X-Refresh-Token` header)
- O(1) session lookup via tokenIdentifier (SHA-256 prefix) with indexed queries
- Pino structured logging with user/org context and device parser for IP extraction
- Rate limiting via AppThrottlerGuard with per-user/API-key/IP isolation

## Key Design Decisions

1. **Two-level session model**: AuthSession (device) + OrgSession (org context)
2. **Sessions created at login and email verification, not org selection**: `AuthService.login()` and `AuthService.verifyEmail()` both create the AuthSession with device info and geolocation. Both use the lazy-link pattern: the session is created without a refresh token (to check caps first), then linked after tokens are generated. Email verification acts as a full login — the user gets an AuthSession, refresh token cookie, and is sent straight to org selection.
3. **OrgSession is lightweight**: No device info, no geolocation, no tokens — just a FK to AuthSession + Organization
4. **AuthSession has no organization FK**: It's device-level, not org-scoped
5. **Revoking an AuthSession automatically invalidates all its OrgSessions**: Validation always checks the parent first
6. **O(1) session lookup**: Via `tokenIdentifier` (SHA-256 prefix of refresh token)
7. **Lazy token linking**: Sessions can be created without the refresh token cookie; linked on first tenant-token refresh
8. **Session limits apply globally**: `MAX_CONCURRENT_AUTH_SESSIONS` = max devices per user globally (default: 5, not per org)
9. **Session policy is global**: Policies are managed via global defaults from SESSION_DEFAULTS constants, configurable by system admins
10. **Tenant token embeds both IDs**: `{ authSessionId, orgSessionId }` for direct validation
11. **RLS bypass for session operations**: AuthSession and OrgSession queries use `executeWithBypass()` since they're auth infrastructure
12. **Immediate revocation enforcement**: The `SessionActivityInterceptor` validates sessions on every authenticated request, throwing 401 for revoked/expired/inactive sessions
13. **Per-session refresh token storage (multi-device safe)**: Each AuthSession stores its own `refreshTokenHash` and `tokenIdentifier`. No user-level refresh token field exists — validation is per session only. **Token issuance and refresh** (`refresh-access-token`, tenant refresh, org selection) use `sessionService.validateSession()`: O(1) lookup by `tokenIdentifier` plus bcrypt against `refreshTokenHash`. **Tenant API requests** use `findSessionByRefreshTokenIdentifier()`: the same indexed `tokenIdentifier` lookup without bcrypt validation. Logging in on Device B never invalidates Device A's ability to refresh tokens.
14. **One active session per device (hard rule)**: Not configurable. Re-login on the same device replaces the existing session. Backed by a partial unique index.
15. **Serialized session creation**: All session creation is protected by PostgreSQL advisory locks to prevent race conditions
16. **Block-on-limit, not auto-revoke**: When session or device caps are reached, login is denied with a clear error code. The user must manually free a slot. This prevents account sharing.
17. **Logout revokes only the current session**: Other device sessions are not affected. Org switching is handled via `selectOrganization`, not logout.
18. **Password reset revokes every session**: Completing `set-password` after a reset is treated as an account-security event. The service hashes the new password, clears the reset code, increments `User.tokenVersion`, and revokes all active AuthSessions plus their child OrgSessions with reason `password_reset`. Old refresh tokens fail immediately, old tenant tokens fail through AuthGuard tokenVersion/session liveness checks, and old identity tokens fail through IdentityTokenGuard tokenVersion checks.

## Global session policy system

The system supports a **global session policy** seeded from SESSION_DEFAULTS constants at server startup, fully configurable by system admins, with optional per-user overrides and in-memory caching for optimal performance.

<Tabs>
<Tab title="Policy resolution">
**Resolution order**: User override > System default

Unlike the previous per-org system, policies are now managed globally by system admins with optional per-user overrides.

**Policy storage**:
- System defaults: Seeded from SESSION_DEFAULTS constants
- User overrides: Stored in database with in-memory caching
- Real-time updates: Policy changes take effect immediately
- Performance: In-memory cache reduces database lookups

**Policy management features**:
- Global defaults configurable by system admins via `/system-admin/session-policy`
- Per-user overrides via `/system-admin/users/:userId/session-policy`
- Automatic cache invalidation when policies are updated
- Fallback to system defaults when user overrides are removed
- Policy validation ensures limits are within reasonable bounds
</Tab>

<Tab title="Configurable parameters">
| Parameter | Default | Description |
|-----------|---------|-------------|
| `maxConcurrentAuthSessions` | 5 | Max devices per user globally |
| `maxRegisteredDevices` | 5 | Max registered devices per user |
| `sessionExpiryDays` | 14 | AuthSession lifetime |
| `inactivityTimeoutHours` | 24 | Inactivity timeout |
| `deviceRemovalCooldownDays` | 30 | Device replacement cooldown |
| `absoluteSessionLifetimeHours` | 336 | Hard session limit (14 days) |

**Policy enforcement**:
- Applied at session creation time for immediate effect
- User override policies take precedence over system defaults
- Policy changes affect new sessions, existing sessions use their creation-time policies
- System admins can view effective policies per user
- Validation prevents unreasonable limits (e.g., negative values, excessive limits)
</Tab>
</Tabs>

## Architecture

### Two-level session model

<Tabs>
<Tab title="AuthSession (device-level)">
**Created**: At login or email verification  
**Purpose**: Represents a device/browser login  
**Contains**:
- Device information (fingerprint, type, browser, OS)
- IP geolocation data (country, city, coordinates)
- Refresh token hash and token identifier
- Activity tracking (last active, activity count)
- Presence status (online/idle/offline)

**Key fields**:
```typescript
{
  id: UUID;
  userId: UUID;          // FK to User
  refreshTokenHash?: string;
  tokenIdentifier?: string;
  deviceFingerprint?: string;
  deviceType?: string;
  ipAddress: string;
  country?: string;
  city?: string;
  latitude?: decimal;
  longitude?: decimal;
  lastActiveAt?: Date;
  presenceStatus: 'online' | 'idle' | 'offline';
  isRevoked: boolean;
  revokedReason?: string;
  createdAt: Date;
  expiresAt: Date;
}
```
</Tab>

<Tab title="OrgSession (org-level)">
**Created**: When user selects an organization  
**Purpose**: Tracks per-org access without duplicating device data  
**Contains**:
- Link to parent AuthSession
- Organization context
- Minimal audit fields

**Key fields**:
```typescript
{
  id: UUID;
  authSessionId: UUID;   // FK to AuthSession
  organizationId: UUID;  // FK to Organization
  isRevoked: boolean;
  revokedReason?: string;
  createdAt: Date;
}
```
</Tab>
</Tabs>

## Features

### Session creation flow

<Steps>
<Step title="Login/Email verification">
`AuthService.login()` or `AuthService.verifyEmail()` creates an AuthSession with device info and geolocation. Uses lazy-link pattern: session created without refresh token, then linked after token generation.
</Step>

<Step title="Organization selection">
User selects organization via `POST /auth/select-organization`. Creates lightweight OrgSession linking to the AuthSession. No device registration or geolocation lookup needed.
</Step>

<Step title="Token validation">
Tenant token embeds `{ authSessionId, orgSessionId }`. Validation checks both levels: AuthSession for device/token validity, OrgSession for org access.
</Step>
</Steps>

### Global device registry

- Users can register up to 5 devices globally (configurable via policy)
- ThumbmarkJS browser fingerprinting for device identification
- Device removal cooldown: configurable period before device replacement allowed
- Enforced at AuthSession creation time

### Block-on-limit enforcement

<Warning>
When session or device caps are reached, login is **denied** with a clear error code. No automatic revocation occurs. Users must manually free a slot to prevent account sharing.
</Warning>

### Real-time presence tracking

- Online/idle/offline status updated via WebSocket heartbeats
- Works from login, before org selection (attached to AuthSession)
- Sessions without heartbeat for 90+ seconds marked offline
- Managed by `PresenceService` with WebSocket integration

### Immediate revocation enforcement

The `SessionActivityInterceptor` validates sessions on **every authenticated request**:

1. Extracts refresh token from cookies
2. Validates AuthSession (checks revocation, expiry, inactivity)
3. Validates OrgSession if in org context
4. Throws 401 for invalid sessions
5. Updates activity tracking asynchronously

## Session validation

<Tabs>
<Tab title="AuthSession validation">
**Method**: `validateAuthSession(refreshToken)`
- O(1) lookup by `tokenIdentifier`
- bcrypt verification against `refreshTokenHash`
- Checks revocation status and expiry
- Used for token refresh and identity operations
</Tab>

<Tab title="Two-level validation">
**Method**: `validateSessionPair(authSessionId, orgSessionId)`
- Validates both AuthSession and OrgSession
- Ensures OrgSession belongs to the AuthSession
- Used for tenant token validation
- Throws 401 if either level is invalid
</Tab>

<Tab title="Multi-device safety">
Each AuthSession stores its own refresh token hash. Logging in on Device B never invalidates Device A's ability to refresh tokens. True multi-device support without interference.
</Tab>
</Tabs>

### Session revocation

<Tabs>
<Tab title="AuthSession revocation">
Revoking an AuthSession automatically invalidates all its child OrgSessions. Used for:
- Device-level logout
- Global user logout  
- Device blocking enforcement
- Session limit enforcement
</Tab>

<Tab title="OrgSession revocation">
Lightweight revocation for org-specific access removal:
- User removed from organization
- Org-level access restrictions
- Admin org session management
</Tab>

<Tab title="Revocation reasons">
```typescript
{
  MANUAL: 'manual',
  LOGOUT: 'logout',
  EXPIRED: 'expired',
  INACTIVITY_TIMEOUT: 'inactivity_timeout',
  DEVICE_BLOCKED: 'device_blocked',
  SESSION_LIMIT: 'session_limit',
  ORG_ACCESS_REVOKED: 'org_access_revoked',
  ADMIN_REVOKED: 'admin_revoked',
  PASSWORD_RESET: 'password_reset'
}
```
</Tab>
</Tabs>

### Device and IP blocking

- System admins can block devices (by fingerprint) or IP addresses
- Blocking enforced at multiple layers:
  - Public endpoints (signup, login, verify, etc.)
  - SessionActivityInterceptor (every authenticated request)
  - AppThrottlerGuard (rate limiting layer)
- In-memory cache with 60-second TTL prevents database hits
- Immediate enforcement across all endpoints

## RLS & multi-tenancy pattern

Both AuthSession and OrgSession are authentication infrastructure requiring cross-org visibility:

| Operation | RLS method | Reason |
|-----------|-----------|--------|
| `createAuthSession` | `tenantContext.executeWithBypass()` | Device-level, no org context |
| `createOrgSession` | `tenantContext.executeInOrg(orgId)` | Sets org context for child session |
| All validation methods | `tenantContext.executeWithBypass()` | Called during auth before org context exists |
| All revocation methods | `tenantContext.executeWithBypass()` | Need cross-org visibility |
| Session listing | `tenantContext.executeWithBypass()` | May span multiple orgs |
| Policy operations | `tenantContext.executeWithBypass()` | Global policy management |
| Device registry | `tenantContext.executeWithBypass()` | Global across orgs |
| Blocking operations | `tenantContext.executeWithBypass()` | Global enforcement |

<Tip>
See [multi-tenancy & RLS](/backend/multi-tenancy/tenant-context) for the full transaction-first design and `TenantContext` patterns used across the codebase.
</Tip>

## Security considerations

1. **Two-level validation** — Both AuthSession and OrgSession must be valid for tenant access
2. **Per-session token storage** — Each AuthSession has its own refresh token hash, enabling true multi-device support
3. **O(1) token lookup** — `tokenIdentifier` provides fast session lookup without bcrypt scanning
4. **Block-on-limit** — Login denied when caps reached, prevents account sharing
5. **Immediate revocation** — SessionActivityInterceptor enforces revocation on every request
6. **Global device caps** — Prevents unlimited device registration per user
7. **Device fingerprinting** — ThumbmarkJS provides reliable device identification
8. **Multi-layer IP blocking** — Blocked IPs rejected at all endpoints and layers
9. **Advisory locks** — Session creation serialized to prevent race conditions
10. **Lazy token linking** — Sessions can exist before token generation for cap checking

## Organization deletion cascade

When a system admin deletes an organization:

<Steps>
<Step title="Revoke OrgSessions">
All OrgSessions for that organization are bulk-revoked with reason `organization_deleted`.
</Step>

<Step title="Preserve AuthSessions">
AuthSessions remain valid for other organizations the user belongs to.
</Step>

<Step title="Soft-delete organization">
The organization is soft-deleted (`isDeleted = true`).
</Step>
</Steps>
