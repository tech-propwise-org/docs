## Overview

The session management system tracks user logins with comprehensive device, IP, and location information. Sessions are per-organization — when a user selects an organization, a new session is created for that specific org context. This provides better security and audit capabilities in a multi-tenant environment.

## Key design decisions

1. **Sessions are per-organization** — each org selection creates a new session
2. **Sessions are created on org selection, not login** — login establishes identity via `auth.service.login()`; org selection via `auth.controller.getTenantToken()` creates the session
3. **Automatic session revocation on org removal** — when a user is removed from an organization, all their sessions for that org are automatically revoked via `revokeAllOrgSessions()`
4. **Session limits are per-org** — three tiers of enforcement — per-org total (default 5), per-device per-org (default 1), per-IP per-org (default 3)
5. **RLS bypass for session operations** — all `SessionService` methods use `tenantContext.executeWithBypass()` because sessions are authentication infrastructure that needs cross-org visibility. Session creation is the exception — it uses `tenantContext.executeInOrg()` to set the org context
6. **Three-scope logout** — logout supports session-level, organization-level, and global scopes with different side effects

## Features

### Session creation

- Automatically created when user **selects an organization** via `POST /auth/select-organization` (not on login)
- `auth.service.login()` explicitly does NOT create a session — device info is passed through to `issueTenantToken()`
- Each organization has separate sessions
- Stores device fingerprint (browser, OS, user agent, device type/name)
- Fetches geolocation from IP address via `GeolocationService`
- Session created inside `tenantContext.executeInOrg()` to set the organization RLS context

### Session validation

- **General validation** (`validateSession`) — loads all non-revoked, non-expired sessions and compares refresh token via bcrypt against each hash
- **Organization-scoped validation** (`validateSessionForOrganization`) — narrows search to specific user+org combination, used during tenant token refresh

<Note>
Both validation methods use RLS bypass since validation occurs during authentication before any org context exists.
</Note>

### Session limit enforcement

Three tiers of enforcement, all invoked after session creation:

| Tier | Method | Default limit | Behavior |
|------|--------|---------------|----------|
| Per-org total | `enforceSessionLimit()` | 5 per user per org | Revokes oldest sessions beyond limit |
| Per-device per-org | `enforceDeviceLimit()` | 1 per browser+OS combo per org | Revokes oldest same-device sessions |
| Per-IP per-org | `enforceIpLimit()` | 3 per IP address per org | Revokes oldest same-IP sessions |

All enforcement methods sort sessions by `createdAt ASC` and revoke the oldest ones first with reason `session_limit`.

### Session revocation

<Tabs>
<Tab title="User-initiated">
| Scope | Method | Triggered by |
|-------|--------|-------------|
| Individual | `revokeSession(sessionId, userId, reason)` | User revokes own session; single-session logout |
| Per-org bulk | `revokeAllOrgSessions(userId, orgId, reason)` | User removed from org; org-level logout |
| Cross-org bulk | `revokeAllUserSessions(userId, exceptSessionId?, reason)` | Global logout |
| Flexible | `revokeAllSessions(userId, orgId?, exceptCurrentId?)` | "Revoke all" endpoint; uses `MANUAL` reason |
</Tab>

<Tab title="System admin">
| Method | Triggered by | Reason |
|--------|-------------|--------|
| `SystemAdminService.revokeSession(sessionId)` | System admin | `admin_revoked` |
| `SystemAdminService.revokeUserSessions(userId)` | System admin | `admin_revoked_all` |
| `SystemAdminService.revokeOrganizationSessions(orgId)` | System admin | `admin_revoked_org` |
</Tab>

<Tab title="Automatic">
| Method | Triggered by | Reason |
|--------|-------------|--------|
| `SystemAdminService.deleteOrganization()` | Org deletion | `organization_deleted` |
| `enforceSessionLimit/DeviceLimit/IpLimit` | Session creation | `session_limit` |
| `cleanupExpiredSessions()` | Periodic cleanup | `expired` |
</Tab>
</Tabs>

### Logout system (three scopes)

The `POST /auth/logout` endpoint supports three scopes via request body:

| Scope | Body parameter | Side effects | Cookie cleared |
|-------|---------------|--------------|----------------|
| **Session** | `sessionId` (or auto-detected from cookie) | Revokes single session only | No |
| **Organization** | `organizationId` | Revokes all sessions in that org | No |
| **Global** | `logoutEverywhere: true` | Revokes all sessions, increments `tokenVersion`, clears `refreshToken`, invalidates permission cache | Yes |

Default behavior (no body params): auto-detects current session from refresh token cookie and revokes it.

```typescript Logout response
{
  message: 'Logged out successfully',
  scope: 'session' | 'organization' | 'global',
  revokedCount: number,
}
```

### Suspicious login detection

Flags logins as suspicious when any of these conditions are met:

1. **Different country** — `countryCode` of new session differs from last session in same org
2. **Impossible travel** — distance > 500km (configurable) within 2 hours, calculated via Haversine formula using stored `latitude`/`longitude`
3. **New device** — browser+OS fingerprint not found in the last 50 sessions for this user+org

Returns `{ isSuspicious: boolean; reasons: string[] }`.

### Session activity tracking

- Global `SessionActivityInterceptor` tracks every API call
- Reads refresh token from `req.cookies.refresh_token` to identify session
- Calls `getCurrentSession()` to find session, then fires `updateSessionActivity()` as fire-and-forget
- Updates `lastActiveAt`, `lastActivityType` (set to `'api_call'`), and increments `activityCount`
- Runs asynchronously after response (fire-and-forget via `tap()` in RxJS pipe)

<Warning>
The interceptor silently fails — session activity tracking never blocks or breaks API responses.
</Warning>

### Expired session cleanup

`cleanupExpiredSessions()` finds all sessions where `isRevoked = false` and `expiresAt < now`, marks them as revoked with reason `expired`. Can be called periodically.

**Guard:** `AuthGuard`

**Query parameters:**
- `page` (default: 1)
- `limit` (default: 20)
- `allOrgs` (optional, boolean) — if `true`, returns history across all organizations.

  sessions: SessionListItem[];
  total: number;
  page: number;
  limit: number;

</Tab>

**Guard:** `AuthGuard`

**Response:** Session details with organization info. Throws `UnauthorizedException` if no active session.

<Tab title="DELETE /auth/sessions/:id">
Revoke a specific session by ID. User can only revoke their own sessions.

**Guard:** `AuthGuard`
</Tab>

<Tab title="POST /auth/sessions/revoke-all">
Revoke all sessions except the current one.

**Guard:** `AuthGuard`

**Query parameters:**
- `allOrgs` (optional, boolean) — if `true`, revokes across all organizations. Default revokes only current org sessions.
</Tab>

<Tab title="POST /auth/logout">
Logout with different scopes.

**Guard:** `AuthGuard`

**Body (all optional):**

```typescript Logout body

  organizationId?: string;   // Revoke all sessions in this org
  logoutEverywhere?: boolean; // Revoke everything, clear tokens

```

If no body params provided, auto-detects current session from cookie and revokes it.
</Tab>
</Tabs>

## RLS & multi-tenancy pattern

Sessions are authentication infrastructure that needs cross-org visibility. The RLS strategy is:

| Operation | RLS method | Reason |
|-----------|-----------|--------|
| `createSession` | `tenantContext.executeInOrg(orgId)` | Sets org context so session row gets the correct `organization_id` for RLS |
| `validateSession` | `tenantContext.executeWithBypass()` | Called during auth before org context exists |
| `validateSessionForOrganization` | `tenantContext.executeWithBypass()` | Called during tenant token refresh before org context |
| All revocation methods | `tenantContext.executeWithBypass()` | Need cross-org visibility for user sessions |
| `getActiveSessions` | `tenantContext.executeWithBypass()` | May return sessions across multiple orgs (`allOrgs` mode) |
| `getSessionHistory` | `tenantContext.executeWithBypass()` | May return sessions across multiple orgs |
| All enforcement methods | `tenantContext.executeWithBypass()` | Need to see all sessions for user in org |
| System admin methods | `tenantContext.executeWithBypass()` | Admin operations are cross-org by nature |

<Tip>
See [multi-tenancy & RLS](/backend/multi-tenancy/tenant-context) for the full transaction-first design and `TenantContext` patterns used across the codebase.
</Tip>

## Security considerations

1. **Refresh token hashing** — tokens are bcrypt-hashed (cost 10) before storage, never stored in plaintext
2. **Unique constraint** — `idx_session_token` unique index prevents token reuse across sessions
3. **Expiration** — sessions auto-expire after 14 days (`SESSION_EXPIRY_DAYS`)
4. **Activity tracking** — detects dormant sessions via `lastActiveAt` and `activityCount`
5. **Suspicious detection** — alerts on unusual login patterns (different country, impossible travel, new device)
6. **Three-tier limits** — per-org (5), per-device (1), and per-IP (3) limits prevent session accumulation
7. **Cookie security** — refresh tokens in HTTP-only cookies with `secure` (production) and `sameSite` (`none` in production, `lax` in dev)
8. **Global logout** — increments `tokenVersion` to invalidate all JWTs, clears `refreshToken`, and invalidates permission cache

## Organization deletion cascade

When a system admin deletes an organization via `DELETE /system-admin/organizations/:id`:

<Steps>
<Step title="Revoke sessions">
All active sessions for that organization are bulk-revoked via `em.nativeUpdate()` with reason `'organization_deleted'`.
</Step>

<Step title="Soft-delete organization">
The organization is soft-deleted (`isDeleted = true`).
</Step>
</Steps>
