## Overview

The session management system tracks user logins with comprehensive device, IP, and location information. Sessions are per-organization — when a user selects an organization, a new session is created for that specific org context. This provides better security and audit capabilities in a multi-tenant environment.

The system includes:
- Per-organization session management with configurable limits
- **Global device registry** — users can register up to 5 devices across all organizations
- **Concurrent device limit** — at most 2 devices can be active (online) simultaneously
- **Monthly device replacement** — users can replace one registered device every 30 days
- **Device/IP/User blocking** — system admins can block devices, IP addresses, or user accounts
- **Real-time presence tracking** — online/idle/offline status via WebSocket heartbeats
- **Session policy system** — configurable limits per organization and user
- **Enhanced security layers** — pre-request validation, blocking enforcement, activity monitoring

## Key design decisions

1. **Sessions are per-organization** — each org selection creates a new session
2. **Sessions are created on org selection, not login** — login establishes identity via `auth.service.login()`; org selection via `auth.controller.getTenantToken()` creates the session
3. **Automatic session revocation on org removal** — when a user is removed from an organization, all their sessions for that org are automatically revoked via `revokeAllOrgSessions()`
4. **Session limits are per-org** — three tiers of enforcement — per-org total (configurable via policy), per-device per-org (configurable via policy), per-IP per-org (configurable via policy)
5. **Global device registry** — users can register up to 5 devices across all organizations with enforced fingerprinting (configurable via policy)
6. **Concurrent device limits** — at most 2 devices can have active sessions simultaneously (global enforcement, from system constants)
7. **Device replacement cooldown** — users can replace one device every 30 days when the device limit is reached (configurable via policy)
8. **Device and IP blocking** — system admins can block devices or IP addresses, enforced at AuthGuard level with 60-second cache
9. **RLS bypass for session operations** — all `SessionService` methods use `tenantContext.executeWithBypass()` because sessions are authentication infrastructure that needs cross-org visibility. Session creation is the exception — it uses `tenantContext.executeInOrg()` to set the org context
10. **Three-scope logout** — logout supports session-level, organization-level, and global scopes with different side effects
11. **IP blocking uses an in-memory cache** — a 60-second TTL cache prevents a DB hit on every API request for IP block checks
12. **Presence is heartbeat-driven** — online/idle/offline status is updated only via WebSocket heartbeats at 30s intervals; a cron job marks stale sessions (no heartbeat for 90s) as offline
13. **Sessions are created without requiring the refresh-token cookie** — `refreshTokenHash` and `tokenIdentifier` are nullable. Sessions created without the cookie are still fully valid and are lazily linked to the refresh token on the first call to `refreshTenantToken` that includes the cookie. **Both fields are always written atomically** — a session will never have `tokenIdentifier` set without `refreshTokenHash`. Any session found by `tokenIdentifier` lookup that has no `refreshTokenHash` is treated as inconsistent state and rejected, not bypassed
14. **O(1) session lookup** — sessions use a derived `tokenIdentifier` for fast lookup instead of scanning all sessions with bcrypt
15. **Session policy system** — configurable session limits and timeouts per organization and per user with three-tier resolution (user override > org default > system default)
16. **Inactivity timeout enforcement** — sessions are checked for inactivity on every API request via SessionActivityInterceptor and revoked if timeout exceeded
17. **Pre-request security checks** — SessionActivityInterceptor performs inactivity timeout and device blocking checks before processing API requests
18. **Shared CORS validation** — both HTTP server and WebSocket gateways use the same origin validation logic from `cors.util.ts`

## Session policy system

The system supports three-tier session policy configuration:
- **User overrides** — Per-user session policies for specific organizations
- **Organization defaults** — Default policies for all users in an organization  
- **System defaults** — Hard-coded fallback values

Resolution order: User override > Org default > System default

**Configurable Parameters:**
- `maxConcurrentSessions` — Max sessions per user per org (default: 5)
- `maxRegisteredDevices` — Max registered devices per user globally (default: 5)
- `maxSessionsPerDevice` — Max sessions per device per org (default: 1)
- `maxSessionsPerIp` — Max sessions per IP per org (default: 5)
- `sessionExpiryDays` — Session lifetime in days (default: 14)
- `inactivityTimeoutHours` — Inactivity timeout in hours (default: 24)
- `deviceRemovalCooldownDays` — Device replacement cooldown (default: 30)
- `absoluteSessionLifetimeHours` — Hard session limit in hours (default: 336)

**API Endpoints:**
- `GET /organizations/:orgId/session-policy` — Get effective session policy for the current user
- `PUT /organizations/:orgId/session-policy` — Update organization-level policy (admin required)
- `PUT /organizations/:orgId/users/:userId/session-policy` — Set user-specific policy override (admin required)

## Features

### Global device registry

- Users can register up to 5 devices across all organizations (configurable via policy)
- Device fingerprinting based on browser + OS combination
- Concurrent device limit: maximum 2 devices can be active simultaneously
- Monthly device replacement: users can replace one registered device every 30 days
- Enforced at session creation and validated via `DeviceRegistryService`

### Device and IP blocking

- System admins can block specific devices or IP addresses
- Blocking enforced at `AuthGuard` level with 60-second in-memory cache
- Blocked devices/IPs are denied access to all API endpoints
- Managed via `DeviceBlockingService` with cache invalidation

### Real-time presence tracking

- Online/idle/offline status updated via WebSocket heartbeats every 30 seconds
- Sessions without heartbeat for 90+ seconds marked offline by cron job
- Presence status stored in `presenceStatus` and `lastHeartbeatAt` fields
- Managed by `PresenceService` with WebSocket gateway integration

### Session creation

- Automatically created when user **selects an organization** via `POST /auth/select-organization` (not on login)
- `auth.service.login()` explicitly does NOT create a session — device info is passed through to `issueTenantToken()`
- Each organization has separate sessions
- Stores device fingerprint (browser, OS, user agent, device type/name)
- Fetches geolocation from IP address via `GeolocationService`
- Enforces device registry limits and concurrent device limits
- Session created inside `tenantContext.executeInOrg()` to set the organization RLS context
- Supports creation without refresh token cookie (lazily linked on first refresh)

### Session validation

- **General validation** (`validateSession`) — O(1) lookup by `tokenIdentifier` then bcrypt verification
- **Organization-scoped validation** (`validateSessionForOrganization`) — narrows search to specific user+org combination, used during tenant token refresh
- **Legacy fallback** — if `tokenIdentifier` lookup fails, falls back to scanning all sessions with bcrypt

<Note>
Both validation methods use RLS bypass since validation occurs during authentication before any org context exists.
</Note>

### Session limit enforcement

Three tiers of enforcement via session policy system:

| Tier | Method | Configurable limit | Behavior |
|------|--------|-------------------|----------|
| Per-org total | `enforceSessionLimit()` | `maxConcurrentSessions` (default: 5) | Revokes oldest sessions beyond limit |
| Per-device per-org | `enforceDeviceLimit()` | `maxSessionsPerDevice` (default: 1) | Revokes oldest same-device sessions |
| Per-IP per-org | `enforceIpLimit()` | `maxSessionsPerIp` (default: 5) | Revokes oldest same-IP sessions |

All enforcement methods sort sessions by `createdAt ASC` and revoke the oldest ones first with reason `session_limit`.

### Logout system (three scopes)

The `POST /auth/logout` endpoint supports three scopes via request body:

| Scope | Body parameter | Side effects | Cookie cleared |
|-------|---------------|--------------|----------------|
| **Session** | `sessionId` (or auto-detected from cookie) | Revokes single session only | No |
| **Organization** | `organizationId` | Revokes all sessions in that org | No |
| **Global** | `logoutEverywhere: true` | Revokes all sessions, increments `tokenVersion`, clears `refreshToken`, invalidates permission cache | Yes |

Default behavior (no body params): auto-detects current session from refresh token cookie and revokes it.

```typescript Logout response
{
  message: 'Logged out successfully',
  scope: 'session' | 'organization' | 'global',
  revokedCount: number,
}
```

### Suspicious login detection

Flags logins as suspicious when any of these conditions are met:

1. **Different country** — `countryCode` of new session differs from last session in same org
2. **Impossible travel** — distance > 500km (configurable) within 2 hours, calculated via Haversine formula using stored `latitude`/`longitude`
3. **New device** — browser+OS fingerprint not found in the last 50 sessions for this user+org

Returns `{ isSuspicious: boolean; reasons: string[] }`.

### Session activity tracking

- Global `SessionActivityInterceptor` tracks every API call and enforces security policies
- Reads refresh token from `req.cookies.refresh_token` to identify session
- Performs pre-request security checks:
  - Inactivity timeout validation (revokes session if exceeded)
  - Device blocking enforcement (denies access if device blocked)
- Calls `getCurrentSession()` to find session, then fires `updateSessionActivity()` as fire-and-forget
- Updates `lastActiveAt`, `lastActivityType` (set to `'api_call'`), and increments `activityCount`
- Runs asynchronously after response (fire-and-forget via `tap()` in RxJS pipe)

<Warning>
The interceptor silently fails — session activity tracking never blocks or breaks API responses, but security checks (timeout/blocking) will deny access.
</Warning>

### Expired session cleanup

`cleanupExpiredSessions()` finds all sessions where `isRevoked = false` and `expiresAt < now`, marks them as revoked with reason `expired`. Can be called periodically.

## RLS & multi-tenancy pattern

Sessions are authentication infrastructure that needs cross-org visibility. The RLS strategy is:

| Operation | RLS method | Reason |
|-----------|-----------|--------|
| `createSession` | `tenantContext.executeInOrg(orgId)` | Sets org context so session row gets the correct `organization_id` for RLS |
| `validateSession` | `tenantContext.executeWithBypass()` | Called during auth before org context exists |
| `validateSessionForOrganization` | `tenantContext.executeWithBypass()` | Called during tenant token refresh before org context |
| All revocation methods | `tenantContext.executeWithBypass()` | Need cross-org visibility for user sessions |
| `getActiveSessions` | `tenantContext.executeWithBypass()` | May return sessions across multiple orgs (`allOrgs` mode) |
| `getSessionHistory` | `tenantContext.executeWithBypass()` | May return sessions across multiple orgs |
| All enforcement methods | `tenantContext.executeWithBypass()` | Need to see all sessions for user in org |
| System admin methods | `tenantContext.executeWithBypass()` | Admin operations are cross-org by nature |
| Policy resolution | `tenantContext.executeWithBypass()` | Policy lookup spans user and org tables |
| Device registry operations | `tenantContext.executeWithBypass()` | Device registry is global across orgs |
| Blocking operations | `tenantContext.executeWithBypass()` | Blocking enforcement is global |
| Presence updates | `tenantContext.executeWithBypass()` | Presence tracking spans all user sessions |

<Tip>
See [multi-tenancy & RLS](/backend/multi-tenancy/tenant-context) for the full transaction-first design and `TenantContext` patterns used across the codebase.
</Tip>

## Security considerations

1. **Refresh token hashing** — tokens are bcrypt-hashed (cost 10) before storage, never stored in plaintext
2. **O(1) token lookup** — `tokenIdentifier` provides fast session lookup without bcrypt scanning
3. **Atomic token fields** — `tokenIdentifier` and `refreshTokenHash` are written together or not at all
4. **Unique constraints** — `idx_session_token` and `idx_session_token_identifier` prevent token reuse
5. **Expiration** — sessions auto-expire based on configurable policy (`sessionExpiryDays`)
6. **Inactivity timeout** — sessions revoked after configurable inactivity period (`inactivityTimeoutHours`)
7. **Activity tracking** — detects dormant sessions via `lastActiveAt` and `activityCount`
8. **Suspicious detection** — alerts on unusual login patterns (different country, impossible travel, new device)
9. **Three-tier limits** — configurable session limits via policy system prevent session accumulation
10. **Global device registry** — enforces device limits and prevents unlimited device registration
11. **Device and IP blocking** — system admin controls with cached enforcement
12. **Real-time presence** — heartbeat-driven online/offline status prevents stale presence
13. **Pre-request security** — SessionActivityInterceptor enforces blocking and timeouts before API processing
14. **Cookie security** — refresh tokens in HTTP-only cookies with `secure` (production) and `sameSite` (`none` in production, `lax` in dev)
15. **Global logout** — increments `tokenVersion` to invalidate all JWTs, clears `refreshToken`, and invalidates permission cache

## Organization deletion cascade

When a system admin deletes an organization via `DELETE /system-admin/organizations/:id`:

<Steps>
<Step title="Revoke sessions">
All active sessions for that organization are bulk-revoked via `em.nativeUpdate()` with reason `'organization_deleted'`.
</Step>

<Step title="Soft-delete organization">
The organization is soft-deleted (`isDeleted = true`).
</Step>
</Steps>
