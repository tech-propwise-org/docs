## Overview

The audit module provides comprehensive tracking and querying capabilities for all data changes within the PropWise CRM system. It captures create, update, and delete operations across all entities through PostgreSQL triggers, storing them in an immutable audit log.

<Note>
The module includes timeline resolution functionality to convert raw audit entries into human-readable timeline events for frontend consumption.
</Note>

## Architecture

The audit system follows a trigger-based approach where all data changes are captured automatically at the database level:

- **Database Triggers**: PostgreSQL `audit_trigger_func()` captures all CUD operations
- **Read-Only Entity**: `AuditLog` entity is strictly for querying, never persisted by application code  
- **Timeline Resolution**: Converts raw audit data into semantic timeline events
- **Multi-tenant**: Audit logs are scoped by tenant context
- **Performance**: Indexed by tenant, timestamp, and table for efficient querying

<Warning>
The AuditLog entity is read-only. Application code should never attempt to persist audit entries directly - they are automatically generated by database triggers.
</Warning>

## Entities

### AuditLog

The core audit log entity that maps to the `audit_log` database table:

```typescript
class AuditLog {
  id: string;                    // Primary key (UUID)
  tenantId: string;             // Tenant isolation
  tableName: string;            // Source table name
  recordId: string;             // ID of the affected record
  action: AuditAction;          // CREATE | UPDATE | DELETE
  oldValues?: Record<string, unknown>;  // Previous values (UPDATE/DELETE)
  newValues?: Record<string, unknown>;  // New values (CREATE/UPDATE)
  changedFields?: string[];     // List of changed field names
  userId?: string;              // User who made the change
  timestamp: Date;              // When the change occurred
}
```

### AuditAction

Enumeration of possible audit actions:

```typescript
enum AuditAction {
  CREATE = 'CREATE',
  UPDATE = 'UPDATE', 
  DELETE = 'DELETE'
}
```

## API endpoints

### User audit endpoints

<AccordionGroup>
<Accordion title="GET /audit">
Query audit logs for the current tenant with filtering and pagination support.

**Query parameters**
```typescript
class AuditLogQueryDto {
  page?: number = 1;
  limit?: number = 20;
  tableName?: string;
  action?: AuditAction;
  recordId?: string;
  startDate?: string;  // ISO 8601
  endDate?: string;    // ISO 8601
}
```

**Response**
```typescript
interface PaginatedResponse<AuditLogDto> {
  data: AuditLogDto[];
  pagination: {
    page: number;
    limit: number;
    totalPages: number;
    totalItems: number;
  };
}
```

<Tip>
Use date range filtering with `startDate` and `endDate` to improve query performance for large audit logs.
</Tip>
</Accordion>

<Accordion title="GET /audit/:recordId/timeline">
Get timeline events for a specific record with resolved timeline messages for frontend display.

**Path parameters**
- `recordId` (string): UUID of the record to get timeline for

**Response**
Returns an array of timeline events with human-readable messages and user information.
</Accordion>
</AccordionGroup>

### System admin endpoints

<AccordionGroup>

## Business rules

### Data capture rules

<Steps>
<Step title="Automatic capture">
All database changes are captured automatically via triggers without application intervention.
</Step>

<Step title="Sensitive field filtering">
Sensitive fields (passwords, tokens) are automatically filtered out at the database trigger level.
</Step>

<Step title="Immutable entries">
Audit entries are immutable once created and cannot be modified or deleted.
</Step>

<Step title="Transaction scope">
Failed transactions do not generate audit entries as they operate within the same transaction scope.
</Step>
</Steps>

### Timeline resolution rules

<Check>Each audit log entry maps to exactly one `TimelineEventType`</Check>
<Check>Timeline messages include user information and change context</Check>
<Check>Stage changes include both old and new stage names</Check>
<Check>User information is resolved at query time, not stored in audit log</Check>

### Access control

<Info>
Users can only access audit logs for their tenant. System administrators can access cross-tenant audit data with proper authentication.
</Info>

- Record-level access follows the same permissions as the source entity
- All queries are automatically scoped by tenant context
- Cross-tenant access is restricted to system administrators only

## Performance considerations

<Note>
The audit module is optimized for high-performance querying through strategic database indexing and query patterns.
</Note>

- **Database indexes** on `tenant_id`, `timestamp`, `table_name` for efficient querying
- **Pagination** prevents large result sets from impacting performance
- **Timeline resolution** done at query time, not storage time to reduce storage overhead
- **Tenant isolation** enforced at query level for security and performance

## Security

<Steps>
<Step title="Sensitive field filtering">
Sensitive fields are filtered at the database trigger level before audit entries are created.
</Step>

<Step title="Tenant isolation">
All audit queries are automatically scoped by tenant context to prevent cross-tenant data access.
</Step>

<Step title="Read-only design">
No direct audit log manipulation is possible through application code - all entries are generated by database triggers.
</Step>
</Steps>

<Warning>
Never attempt to manipulate audit logs directly through application code. The audit system integrity depends on database-level triggers for all data capture.
</Warning>
