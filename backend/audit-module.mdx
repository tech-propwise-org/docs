## Overview

The audit module provides comprehensive activity tracking and timeline functionality for the PropWise CRM system. It captures all data changes across entities through PostgreSQL triggers and presents them as both raw audit logs and human-readable timeline events.

<Info>
The module supports both tenant-scoped and system-wide audit queries, ensuring proper data isolation while providing flexibility for system administrators.
</Info>

## Architecture

### Design Philosophy

The audit module follows several key architectural principles:

<CardGroup cols={2}>
  <Card title="Database-Driven Auditing" icon="database">
    Uses PostgreSQL triggers to capture changes at the database level, ensuring consistency and completeness
  </Card>
  <Card title="Read-Only Entity" icon="lock">
    The AuditLog entity is strictly read-only from application code - all writes happen via database triggers
  </Card>
  <Card title="Timeline Abstraction" icon="timeline">
    Provides a higher-level timeline view that translates raw audit data into user-friendly messages
  </Card>
  <Card title="Dual Access Patterns" icon="users">
    Supports both tenant-scoped queries and system-wide queries for administrators
  </Card>
</CardGroup>

### Key Components

- **AuditLog Entity**: Maps to the `audit_log` table (read-only)
- **AuditService**: Core business logic for querying and formatting audit data
- **AuditTimelineResolver**: Translates audit logs into human-readable timeline events
- **Dual Controllers**: Separate endpoints for tenant and system admin access

## Data Models

### AuditLog Entity

```typescript
interface AuditLog {
  id: string;                    // UUID primary key
  tableName: string;             // Name of the table that changed
  action: AuditAction;           // INSERT, UPDATE, DELETE
  rowData: Record<string, any>;  // Full row data after change
  changedFields: Record<string, any>; // Only changed fields (UPDATE only)
  recordId: string;              // ID of the changed record
  tenantId: string;              // Tenant context
  userId?: string;               // User who made the change
  createdAt: Date;               // When the change occurred
}
```

### AuditAction Enum

```typescript
enum AuditAction {
  INSERT = 'INSERT',
  UPDATE = 'UPDATE', 
  DELETE = 'DELETE'
}
```

## Business Rules

### Data Integrity

<Steps>
  <Step title="Immutable Audit Logs">
    Once created by the database trigger, audit logs cannot be modified or deleted through application code
  </Step>
  <Step title="Sensitive Data Protection">
    Passwords, refresh tokens, and other sensitive fields are automatically stripped by the trigger before persistence
  </Step>
  <Step title="Transactional Consistency">
    All audit operations occur within the same transaction as the data change to ensure consistency
  </Step>
</Steps>

### Access Control

<Tabs>
  <Tab title="Regular Users">
    - Can only access audit logs for their own tenant
    - Timeline resolution includes user information only if accessible
    - Limited to tenant-scoped endpoints
  </Tab>
  <Tab title="System Administrators">
    - Can access audit logs across all tenants
    - Have access to system-wide endpoints
    - Can filter by specific tenant ID
  </Tab>
</Tabs>

### Temporal Constraints

<Note>
- Date range queries validate that `endDate >= startDate`
- Date filters use inclusive ranges
- All timestamps are stored and returned in UTC
</Note>

## Integration Points

### Authentication & Authorization

The audit module integrates with the authentication system through:

- `@AuthGuard()` decorators for endpoint protection
- `@GetCurrentUser()` for tenant context extraction
- `@SystemAdmin()` decorator for admin-only access

### Tenant Isolation

<Check>
All queries are automatically scoped to the current tenant (except system admin endpoints), ensuring proper data isolation.
</Check>

### User Information Resolution

The module resolves user information for timeline display by:

- Linking audit entries to user accounts via `userId` field
- Providing user context in timeline events
- Respecting access permissions when displaying user data

### Error Handling

<Tip>
The module implements comprehensive error handling:
- Returns 404 for non-existent entities in timeline queries
- Validates UUID format for ID parameters
- Enforces pagination limits (max 100 items per page)
</Tip>

### Security Considerations

<Warning>
- No direct entity manipulation allowed - all writes occur via database triggers
- Tenant isolation enforced at multiple layers
- Sensitive data filtered at database level before persistence
</Warning>

## Performance Considerations

### Caching Strategy

The audit module does not implement caching due to:

- Write-heavy, read-light access patterns
- Real-time requirements for audit data
- Timeline resolution performed on-demand

### Database Optimization

<Steps>
  <Step title="Indexing">
    Proper indexes on `tenantId`, `tableName`, `recordId`, and `createdAt` for efficient querying
  </Step>
  <Step title="Partitioning">
    Consider table partitioning for large audit datasets based on date ranges
  </Step>
  <Step title="Archival">
    Implement archival strategies for old audit data to maintain performance
  </Step>
</Steps>
