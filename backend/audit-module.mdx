## Overview

The audit log system provides a tamper-proof, automatic record of every data change across the CRM. Every `INSERT`, `UPDATE`, and `DELETE` on ~35 tables is captured atomically at the database level, including what changed, who changed it, from which session, and from which IP address.

<Info>
The system is built on PostgreSQL triggers making it impossible to bypass from application code — even raw SQL queries or bulk imports are captured automatically.
</Info>

## Architecture

### Design Philosophy

The audit module follows several key architectural principles:

<CardGroup cols={2}>
  <Card title="Database-Driven Auditing" icon="database">
    Uses PostgreSQL triggers to capture changes at the database level, ensuring consistency and completeness
  </Card>
  <Card title="Tamper-Proof Records" icon="lock">
    Row-Level Security policies make audit logs append-only and immutable from application code
  </Card>
  <Card title="Atomic Capture" icon="gear">
    Trigger fires inside the same transaction as the data change — either both succeed or both roll back
  </Card>
  <Card title="Tenant-Scoped Security" icon="shield">
    RLS policies enforce organization isolation while supporting system admin access
  </Card>
</CardGroup>

### Key Design Principles

| Principle                   | Implementation                                                                                             |
| --------------------------- | ---------------------------------------------------------------------------------------------------------- |
| **Atomic capture**          | Trigger fires inside the same transaction as the data change — either both succeed or both roll back       |
| **Bypass-proof**            | Writes happen at the DB level; application code has no `em.persist()` path to the table                    |
| **Immutable records**       | RLS policies deny UPDATE and DELETE; INSERT is denied to the application role                              |
| **Tenant-scoped reads**     | RLS SELECT policy enforces `organization_id` isolation                                                     |
| **Sensitive data stripped** | Trigger removes `password`, `refresh_token`, `hashed_token`, `refresh_token_hash` before storing snapshots |

### Core Architecture

```
┌─────────────────────────┐       ┌──────────────────────────┐
│   Application Code      │       │   PostgreSQL Trigger     │
│   (NestJS / ORM)        │──────►│   audit_trigger_func()   │
│                         │       │   AFTER INSERT/UPDATE/   │
│  executeInOrg()         │       │   DELETE ON each table   │
│    SET LOCAL            │       │                          │
│      app.organization_id│       │  Reads SET LOCAL vars:   │
│      app.user_id        │       │    app.organization_id   │
│      app.session_id     │       │    app.user_id           │
│      app.ip_address     │       │    app.session_id        │
└─────────────────────────┘       │    app.ip_address        │
                                  │                          │
                                  │  Writes to audit_log     │
                                  └──────────┬───────────────┘
                                             │
                                  ┌──────────▼───────────────┐
                                  │   audit_log table        │
                                  │   (append-only via RLS)  │
                                  └──────────────────────────┘
```

## Database Schema

### Audit Log Table

```sql
CREATE TABLE "audit_log" (
  "id"               UUID        NOT NULL DEFAULT gen_random_uuid() PRIMARY KEY,
  "occurred_at"      TIMESTAMPTZ NOT NULL DEFAULT clock_timestamp(),
  "user_id"          UUID        NULL,       -- NULL for system/cron actions
  "organization_id"  UUID        NOT NULL,
  "session_id"       UUID        NULL,       -- NULL when not in a user session
  "action"           TEXT        NOT NULL CHECK ("action" IN ('INSERT', 'UPDATE', 'DELETE')),
  "table_name"       TEXT        NOT NULL,
  "record_id"        UUID        NOT NULL,
  "old_data"         JSONB       NULL,       -- NULL for INSERT
  "new_data"         JSONB       NULL,       -- NULL for DELETE
  "changed_fields"   TEXT[]      NULL,       -- Populated for UPDATE only
  "ip_address"       TEXT        NULL,
  "is_system_action" BOOLEAN     NOT NULL DEFAULT false
);
```

### Indexes

| Index                      | Columns                                                  | Purpose                              |
| -------------------------- | -------------------------------------------------------- | ------------------------------------ |
| `idx_audit_org_time`       | `organization_id, occurred_at DESC`                      | Default list view — org-scoped feed  |
| `idx_audit_record`         | `table_name, record_id, occurred_at DESC`                | Change history for a specific record |
| `idx_audit_user_time`      | `user_id, occurred_at DESC`                              | Activity feed for a specific user    |
| `idx_audit_table_time`     | `organization_id, table_name, occurred_at DESC`          | Filter by entity type within an org  |
| `idx_audit_log_session_id` | `session_id, occurred_at DESC` (partial, WHERE NOT NULL) | Per-session activity trace           |

## Row-Level Security (RLS)

Four policies enforce the append-only, tenant-isolated contract:

| Policy                | Operation | Rule                                                                                   |
| --------------------- | --------- | -------------------------------------------------------------------------------------- |
| `audit_log_select`    | SELECT    | Allow if `organization_id` matches `app.organization_id`, or `app.bypass_rls = 'true'` |
| `audit_log_no_insert` | INSERT    | `WITH CHECK (false)` — always denied to application role                               |
| `audit_log_no_update` | UPDATE    | `USING (false)` — always denied                                                        |
| `audit_log_no_delete` | DELETE    | `USING (false)` — always denied                                                        |

<Note>
The trigger functions use `SECURITY DEFINER` so they run with elevated privileges and can insert despite the INSERT policy. Application code cannot directly write to audit_log.
</Note>

## Trigger Functions

### `audit_trigger_func()` — General Purpose

Attached to all ~35 audited tables. Reads context variables from the transaction, builds old/new snapshots, strips sensitive fields, and inserts into `audit_log`.

**Tables audited:**
`lead`, `deal`, `contact`, `person`, `company`, `deal_contact`, `contact_company_role`,
`commission_payment`, `org_commission_payment`, `user_org_commission`, `entity_stakeholder`, `entity_transfer`, `lead_stage`, `deal_stage`,
`lead_stage_history`, `deal_stage_history`, `entity_stakeholder_history`, `user`,
`user_org_roles`, `organization`, `role`, `permission`, `invitation`, `team`,
`team_membership`, `team_membership_role`, `secondary_unit`, `inventory_unit_ownership`,
`inventory_unit_transaction`, `shared_deal`, `shared_listing`, `unit_access_grant`,
`unit_access_request`, `developer_project_link`, `system_config`, `email_integrations`

**Logic:**

```
DELETE → record old_data, record_id from OLD row
INSERT → record new_data, record_id from NEW row
UPDATE → record both, compute changed_fields (array of column names that differ)
```

**Sensitive field stripping (applied after snapshot):**

```sql
v_old_data := v_old_data - 'password' - 'refresh_token' - 'hashed_token' - 'refresh_token_hash';
v_new_data := v_new_data - 'password' - 'refresh_token' - 'hashed_token' - 'refresh_token_hash';
```

### `audit_session_trigger_func()` — Session Table

Dedicated trigger for the `session` table. Reads `organization_id` and `user_id` directly from the row (not from `SET LOCAL`) since session operations often run outside a normal tenant transaction.

**Heartbeat filtering:** UPDATE events where only `last_active_at`, `activity_count`, or `last_activity_type` changed are silently discarded to avoid high-frequency noise:

```sql
IF v_changed_fields <@ ARRAY['last_active_at', 'activity_count', 'last_activity_type'] THEN
  RETURN NULL;
END IF;
```

## Context Propagation

Every write transaction sets PostgreSQL session-local variables so triggers can identify who made the change:

```typescript
// In TenantContext.executeInOrg()
SET LOCAL app.organization_id = '<uuid>';
SET LOCAL app.bypass_rls      = 'false';
SET LOCAL app.user_id         = '<uuid>';      // from CLS (auth guard)
SET LOCAL app.session_id      = '<uuid>';      // from CLS (session)
SET LOCAL app.ip_address      = '<ip>';        // from request
```

<Note>
The `executeInOrg()` pattern ensures all database operations within an organization context automatically include the proper audit metadata.
</Note>

## Data Models

### AuditLog Entity

```typescript
interface AuditLog {
  id: string;                    // UUID primary key
  occurred_at: Date;             // When the change occurred
  user_id?: string;              // User who made the change (nullable)
  organization_id: string;       // Organization context
  session_id?: string;           // Session context (nullable)
  action: AuditAction;           // INSERT, UPDATE, DELETE
  table_name: string;            // Name of the table that changed
  record_id: string;             // ID of the changed record
  old_data?: Record<string, any>; // Data before change (NULL for INSERT)
  new_data?: Record<string, any>; // Data after change (NULL for DELETE)
  changed_fields?: string[];      // Array of changed field names (UPDATE only)
  ip_address?: string;           // Client IP address (nullable)
  is_system_action: boolean;     // Whether this was a system/cron action
}
```

### AuditAction Enum

```typescript
enum AuditAction {
  INSERT = 'INSERT',
  UPDATE = 'UPDATE', 
  DELETE = 'DELETE'
}
```

## Sensitive Data Handling

The trigger functions automatically strip sensitive fields before storing snapshots:

```sql
-- Applied to both old_data and new_data
v_old_data := v_old_data - 'password' - 'refresh_token' - 'hashed_token' - 'refresh_token_hash';
v_new_data := v_new_data - 'password' - 'refresh_token' - 'hashed_token' - 'refresh_token_hash';
```

<Check>
This ensures that sensitive authentication data never appears in audit logs, even if the original change included such fields.
</Check>

## Business Rules

### Data Integrity

<Steps>
  <Step title="Immutable Audit Logs">
    Once created by the database trigger, audit logs cannot be modified or deleted through application code
  </Step>
  <Step title="Sensitive Data Protection">
    Passwords, refresh tokens, and other sensitive fields are automatically stripped by the trigger before persistence
  </Step>
  <Step title="Transactional Consistency">
    All audit operations occur within the same transaction as the data change to ensure consistency
  </Step>
</Steps>

### Access Control

<Tabs>
  <Tab title="Regular Users">
    - Can only access audit logs for their own organization
    - Timeline resolution includes user information only if accessible
    - Limited to tenant-scoped endpoints
  </Tab>
  <Tab title="System Administrators">
    - Can access audit logs across all organizations
    - Have access to system-wide endpoints
    - Can filter by specific organization_id
  </Tab>
</Tabs>

### Temporal Constraints

<Note>
- Date range queries validate that `endDate >= startDate`
- Date filters use inclusive ranges
- All timestamps are stored and returned in UTC using `clock_timestamp()` for precision
</Note>

## Integration Points

### Authentication & Authorization

The audit module integrates with the authentication system through:

- `@AuthGuard()` decorators for endpoint protection
- `@GetCurrentUser()` for tenant context extraction
- `@SystemAdmin()` decorator for admin-only access

### Tenant Isolation

<Check>
All queries are automatically scoped to the current organization (except system admin endpoints), ensuring proper data isolation through RLS policies.
</Check>

### Session Tracking

The module tracks user sessions with special handling for session table updates:

- Heartbeat filtering: UPDATE events where only `last_active_at`, `activity_count`, or `last_activity_type` changed are silently discarded
- Session context captured in `session_id` field for activity correlation

### Error Handling

<Tip>
The module implements comprehensive error handling:
- Returns 404 for non-existent entities in timeline queries
- Validates UUID format for ID parameters
- Enforces pagination limits (max 100 items per page)
</Tip>

### Security Considerations

<Warning>
- No direct entity manipulation allowed - all writes occur via database triggers with SECURITY DEFINER
- Multi-layered tenant isolation through RLS policies and application-level checks
- Sensitive data filtered at database level before persistence
- System actions bypass user context but maintain organization context
</Warning>

## Performance Considerations

### Database Optimization

<Steps>
  <Step title="Strategic Indexing">
    Indexes on `organization_id`, `table_name`, `record_id`, and `occurred_at` for efficient querying patterns
  </Step>
  <Step title="Session Filtering">
    Heartbeat filtering reduces noise from high-frequency session updates
  </Step>
  <Step title="Partitioning Strategy">
    Consider table partitioning for large audit datasets based on date ranges or organization
  </Step>
  <Step title="Archival Planning">
    Implement archival strategies for old audit data to maintain query performance
  </Step>
</Steps>

### Caching Strategy

The audit module does not implement caching due to:

- Write-heavy, read-light access patterns
- Real-time requirements for audit data
- Timeline resolution performed on-demand for accuracy
