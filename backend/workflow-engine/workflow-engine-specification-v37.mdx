## Overview

The Workflow Engine provides rule-based automation for the messaging module. It evaluates configurable automation rules against messaging events (new messages, new conversations, AI escalations) and executes actions such as assigning agents, sending templates, creating tasks, and updating lead stages.

<Note>
Rules are evaluated in **Phase B** of the webhook processor pipeline, isolated in a try-catch block so failures never affect message persistence or delivery.
</Note>

---

## Core Concepts

### Visual Workflow

The current workflow builder persists visual automations in the `workflow` table. Each workflow has a `triggerType`, optional `triggerConfig`, and a JSONB `graph` containing ordered step definitions and their connections.

<Info>
**Last Updated:** 2026-06-05  
**Status:** Active
</Info>

#### Frontend Builder Model

The frontend builder uses an explicit-save model:

<Steps>
<Step title="Save (Draft)">
`PATCH /v1/workflows/:id` without lifecycle changes - updates draft content only
</Step>

<Step title="Publish (New Workflow)">
`POST /v1/workflows` with `status: PUBLISHED` - atomic create + publish for a new unsaved draft
</Step>

<Step title="Republish (Existing Workflow)">
`PATCH /v1/workflows/:id` with editor payload plus `status: PUBLISHED` - merge, publish-grade validation, version bump when structural fields change
</Step>

<Step title="Unpublish or Archive">
`PATCH /v1/workflows/:id` with `status: DRAFT` or `status: ARCHIVED` - lifecycle-only rules apply (no structural fields on these transitions)
</Step>
</Steps>

#### Starting Workflows

<Tabs>
<Tab title="Start from Scratch">
Opens a local draft with an empty graph and no trigger selected. The user must choose a trigger before saving.
</Tab>

<Tab title="From Template">
Opens a local draft from a system-level `WorkflowTemplate` record, copying its trigger type and graph into an editable local workflow. The database row is created only on explicit save.
</Tab>
</Tabs>

## Workflow Execution Tracking

Each workflow execution is tracked via `WorkflowExecution` entity with comprehensive monitoring:

### Execution Sources

| Source | Description |
|--------|-------------|
| `TRIGGER` | Automatically triggered by matching event |
| `MANUAL_TEST` | Triggered via builder test function |
| `MANUAL_RUN` | Manually invoked for a conversation |

### Step Execution Tracking

Step executions are tracked in `WorkflowStepExecution` with:

- Status (running, completed, failed, timeout)
- Timestamps (start/end)
- Input/output data
- Error details

<Note>
The `workflow-timeout` queue monitors pending responses (for `ask_question` steps) and times them out after the configured duration.
</Note>

## Builder Action Gating

<Warning>
A non-empty **name is required** before any action. **Save**, **Test**, and **Publish** are all disabled while `name` is empty or whitespace-only.
</Warning>

### Action Requirements

<AccordionGroup>
<Accordion title="Name Requirement (All Actions)">
The "Untitled workflow" placeholder does not satisfy the name requirement. The missing-name reason surfaces in each button's tooltip.
</Accordion>

<Accordion title="Test Requirements">
- Valid name
- Selected trigger
</Accordion>

<Accordion title="Save Requirements">
- Valid name
- Selected trigger (except for metadata-only edits on already-published workflows)
</Accordion>

<Accordion title="Publish Requirements">
- Valid name
- Selected trigger
- At least one step
- Zero validation issues from `getWorkflowValidationIssues`
- Valid per-step configuration
</Accordion>
</AccordionGroup>

<Info>
The name check is enforced for Publish as the first entry in `getWorkflowValidationIssues` and for Save/Test via a shared `hasName` flag. The backend applies its own create/publish validation independently.
</Info>

## Workflow Templates

Workflow templates are stored in the database, not hardcoded in the frontend.

### Template Management

<Steps>
<Step title="Canonical Catalog">
The catalog and its idempotent slug-based upsert live in `WorkflowTemplateSeederService` (in the workflow module)
</Step>

<Step title="Automatic Seeding">
`AppInitializerService.onModuleInit()` runs the seeder on **every server boot** inside a `TenantContext.executeWithBypass()` transaction
</Step>

<Step title="Global Availability">
The global `workflow_template` catalog is always present without a manual seed step
</Step>
</Steps>

### Template Constraints

<Warning>
Seeded templates must be ready-to-publish in any organization. They may use generic message, question, branch, date/time, task, and contact-update steps, but **must not** preselect:

- Organization-specific AI agents
- Users or teams
- Lead stages
- Notification recipients
</Warning>

### Template API

Templates are exposed through `GET /v1/workflows/templates` as global system catalog records with no organization ownership.

<Tip>
Templates are upserted by stable slug and reactivated if a previously soft-deleted slug is seeded again, so catalog edits replace stale template graphs instead of leaving invalid versions in the browser.
</Tip>

## Workflow Actor Resolution

The engine resolves the workflow actor once per execution and threads it onto every step's `StepContext.actorUserId`.

### Resolution Logic

<CodeGroup>
```typescript ORGANIZATION Scope
// Resolves to the per-org AI service user
// via shared `resolveAiServiceUserId` utility
const actorUserId = await resolveAiServiceUserId(organizationId);
```

```typescript PERSONAL Scope
// Resolves to the workflow owner
const actorUserId = workflow.createdBy;
```
</CodeGroup>

### Actor Usage

The resolved actor is used for:

1. **Domain Attribution**
   - Task creation
   - Lead updates
   - Conversation assignments

2. **Audit Row Attribution**
   - Via `auditContextForActor`

<Warning>
When the actor cannot be resolved (missing AI service user or deleted owner), handlers fail with appropriate error codes rather than writing unattributed records.
</Warning>

<Info>
Step handlers access the resolved actor through `context.actorUserId` and no longer load `workflow.createdBy` themselves—the resolved actor is the single source of truth for workflow attribution.
</Info>

## Workflow Triggers

The visual builder trigger picker exposes eight trigger types:

### Available Triggers

| Trigger Type | Display Name | Description |
|--------------|--------------|-------------|
| `FIRST_MESSAGE_RECEIVED` | First Message Received | Triggers when a contact sends their first message |
| `MESSAGING_WINDOW_REOPENED` | Messaging Window Reopened | Triggers when messaging window reopens |
| `CONTACT_FIELD_UPDATED` | Contact Field Updated | Triggers when contact fields change |
| `ASSIGNMENT_CHANGED` | Assignment Changed | Triggers when conversation assignment changes |
| `LEAD_CREATED_FROM_CAMPAIGN` | Lead Created from Campaign | Triggers when campaign creates a lead |
| `LIFECYCLE_UPDATED` | Lead Moved to Stage | Triggers when lead stage changes |
| `INCOMING_WEBHOOK` | Webhook / Integration Event | Triggers from external webhooks |
| `MANUAL` | Manual Trigger | Triggers only when explicitly invoked |

<Note>
The backend `WorkflowTriggerType` enum, frontend API enum, and database CHECK constraints are aligned to this same eight-trigger set.
</Note>

### Manual Trigger Behavior

<Warning>
`MANUAL` has **no automatic event listener**. A manual workflow runs only when explicitly invoked.
</Warning>

#### Invocation Methods

<CardGroup cols={2}>
<Card title="Chat Composer" icon="comment">
"Trigger workflow" button calls `POST /v1/workflows/:id/trigger`
</Card>

<Card title="AI Agent Action" icon="robot">
`trigger_workflow` action invokes via `WorkflowEngineService.startWorkflow()`
</Card>
</CardGroup>

#### Manual Workflow Rules

<Steps>
<Step title="Trigger Validation">
Both invocation methods call `WorkflowEngineService.startWorkflow()` with `requireTriggerType: MANUAL`, which rejects non-manual workflows
</Step>

<Step title="Full Context">
Manual workflows carry full conversation context (conversational steps are allowed)
</Step>

<Step title="Scope Permissions">
Permitted in both organization and personal scope
</Step>

<Step title="Unique Names">
Multiple manual workflows may be published in the same scope, but must have **unique published names** within their scope
</Step>

<Step title="Single Active Execution">
Only **one** manual workflow may be active (`RUNNING`/`WAITING`) per conversation at a time
</Step>
</Steps>

#### Concurrency Control

<Info>
The partial unique index `we_conv_active_manual_unique` on `workflow_execution` enforces race-safe single-active-execution per conversation for manual workflows.
</Info>

<Warning>
A second manual start on a busy conversation:
- Returns the running execution if it is the same workflow
- Otherwise fails with `409 Conflict`
</Warning>

<Note>
`startWorkflow` also **proactively** rejects a manual trigger (`409 Conflict`) whenever *any* workflow execution—auto-triggered or manual—is already active on the conversation. This blocks a manual run from racing an active auto workflow's Ask Question wait.
</Note>

## Workflow Scope

Workflows can be organization-scoped or personal:

### Scope Permissions

<Tabs>
<Tab title="Users with MESSAGING_MANAGE">
- Can create organization workflows by default
- May choose personal scope in the builder
- Have full access to organization workflows
</Tab>

<Tab title="Users without MESSAGING_MANAGE">
- Always create personal workflows owned by the authenticated user
- Cannot create or access organization workflows
- Can only view and manage their own personal workflows
</Tab>
</Tabs>

### Publishing Rules

<AccordionGroup>
<Accordion title="Organization Scope">
One published organization workflow per `(organization_id, trigger_type)`

**Database constraint:** `workflow_org_published_trigger_unique`
</Accordion>

<Accordion title="Personal Scope">
One published personal workflow per `(organization_id, owner_user_id, trigger_type)`

**Database constraint:** `workflow_personal_published_trigger_unique`
</Accordion>
</AccordionGroup>

<Info>
Organization and personal workflows with the same `triggerType` may coexist because they target different audiences.
</Info>

### Access Control

Workflow list requests accept optional `scope` parameter:

<Steps>
<Step title="Users with MESSAGING_VIEW or MESSAGING_MANAGE">
List both organization and personal workflows according to filters
</Step>

<Step title="Users without messaging permissions">
Receive **only their own personal workflows**. `scope=ORGANIZATION` returns **no rows** for them.
</Step>
</Steps>

#### Service-Level Enforcement

Uses `AccessService.assertWorkflowAccessInTransaction`:

| Workflow Type | Read Access | Modify/Delete Access |
|---------------|-------------|---------------------|
| Organization | `MESSAGING_VIEW` or `MESSAGING_MANAGE` | **`MESSAGING_MANAGE`** |
| Personal (Owner) | Full access | Full access |
| Personal (Non-owner) | `MESSAGING_VIEW` or `MESSAGING_MANAGE` | `MESSAGING_MANAGE` |

### Scope Chaining

<Warning>
Workflow-to-workflow chains enforce strict scope matches:

- **Organization workflows** can trigger only organization workflows
- **Personal workflows** can trigger only personal workflows owned by the same user
- `MESSAGING_MANAGE` does **not** allow a personal workflow to trigger organization workflows
</Warning>

#### AI-Triggered Child Workflows

<Info>
- Personal agents can trigger only their owner's personal workflows
- Organization agents can trigger only organization workflows
</Info>

## Visual Builder Operations

### Adding Steps

In the current vertical builder, adding steps wires the selected output connection to the new node automatically.

### Deleting Steps

<Warning>
Deleting a step is graph-aware. If the selected step has downstream targets through:
- `connections.default`
- `onFailure`
- `onTimeout`
- Named branch outputs

The builder prompts for confirmation and removes:
1. The selected step
2. The reachable downstream path
3. While preserving unrelated sibling branches
</Warning>

<Note>
Deleting a step also removes references to deleted nodes from named output connections and branch targets so the saved graph does not retain broken references.
</Note>

## Branch Conditions

Visual workflow branches use the structured backend `BranchCondition` schema.

### Schema Structure

```typescript
interface BranchCondition {
  category: BranchCategory;
  field: string;
  operator: BranchOperator;
  value?: any;
  stepId?: string; // Only when category is 'previous_step_output'
}
```

<Warning>
Legacy elements are **rejected at publish time**:
- Raw field paths
- Wrapped variables
- Deprecated categories: `trigger`, `contact`, `lead`, `conversation`, `previous_step`, `webhook_payload`
- Any serialized `source` property on a condition object
</Warning>

### Supported Categories

<AccordionGroup>
<Accordion title="contact_field">
Values resolve through `PersonService` workflow branch helpers.

**Trigger compatibility:** Checked via catalog
</Accordion>

<Accordion title="previous_step_output">
Reads from `execution.variables.steps[stepId]`

**Validation:** Publish validation requires the selected `stepId` to be the Branch node's single direct previous step
</Accordion>

<Accordion title="assignee_status">
Resolves through `WorkflowMessageStatusReaderService` with explicit organization filters inside the workflow transaction
</Accordion>

<Accordion title="last_incoming_message">
Resolves through `WorkflowMessageStatusReaderService`

**Ordering:** `createdAt DESC, id DESC`

**Fields:** Only Branch catalog fields exposed (outbound `senderType` is not branchable)
</Accordion>

<Accordion title="last_outgoing_message">
Resolves through `WorkflowMessageStatusReaderService`

**Ordering:** `createdAt DESC, id DESC`

**Fields:** Only Branch catalog fields exposed
</Accordion>

<Accordion title="time_since_last_incoming_message">
Time-based comparison against most recent incoming message timestamp
</Accordion>

<Accordion title="time_since_last_outgoing_message">
Time-based comparison against most recent outgoing message timestamp
</Accordion>

<Accordion title="lead_stage">
Compares against organization's effective lead stages (including global stages without org overrides)

**Validation:** Publish validation rejects unknown stage IDs

**Field:** `lead_stage.stageId`
</Accordion>
</AccordionGroup>

### Catalog Management

<Info>
Backend validation and runtime use the backend branch catalog constants as the final authority. The editor intentionally keeps a matching local static catalog so Branch condition controls are available without a metadata API call.
</Info>

<Tip>
Organization-scoped lead stage options are the only dynamic exception and are loaded through the existing lead stage API, filtered to active stages.
</Tip>

<Warning>
Any Branch catalog change must update both:
1. Backend constants
2. Frontend local catalog
</Warning>

## Validation

### Publish Validation

`getWorkflowValidationIssues` checks:

<Check>Valid, non-empty name</Check>
<Check>Selected trigger type</Check>
<Check>At least one step in graph</Check>
<Check>Valid per-step configuration</Check>
<Check>No broken connections or references</Check>
<Check>Branch conditions use supported categories and fields</Check>
<Check>Lead stage IDs exist in organization</Check>
<Check>Previous step output references valid preceding steps</Check>

### Runtime Validation

<Warning>
Technical runtime tracking fields (`messageId`, `pendingResponseId`, `outboundPurpose`) are not exposed in branch conditions to maintain clean separation between workflow logic and internal execution state.
</Warning>
