## Overview

The Workflow Engine provides rule-based automation for the messaging module. It evaluates configurable automation rules against messaging events (new messages, new conversations, AI escalations) and executes actions such as assigning agents, sending templates, creating tasks, and updating lead stages.

<Info>
Rules are evaluated in **Phase B** of the webhook processor pipeline, isolated in a try-catch block so failures never affect message persistence or delivery.
</Info>

## Core Concepts

### Automation Rule

An automation rule consists of:

- **Trigger** â€” the event that initiates evaluation
- **Conditions** â€” JSONB criteria that must be met for the rule to fire
- **Actions** â€” JSONB definitions of what to do when the rule fires
- **Priority** â€” evaluation order (lower number = higher priority)

### Entity: AutomationRule

| Field          | Type              | Description                                    |
|----------------|-------------------|-----------------------------------------------|
| id             | UUID              | Primary key                                    |
| organization   | Organization      | Tenant scope                                   |
| name           | string            | Human-readable rule name                       |
| description    | string?           | Optional description of what the rule does     |
| trigger        | AutomationTrigger | Event that activates this rule                 |
| conditions     | JSONB             | Array of condition objects to evaluate         |
| actions        | JSONB             | Array of action objects to execute             |

| updatedAt      | Date              | Last modification timestamp                    |

## Trigger Types

| Trigger                       | Fires When                                    | Context Available                                    |
|------------------------------|-----------------------------------------------|------------------------------------------------------|
| `NEW_MESSAGE`                 | Any inbound message is received               | message, conversation, personChannel, channelAccount |
| `CONVERSATION_CREATED`        | A new conversation is created                 | conversation, personChannel, channelAccount          |
| `LEAD_CREATED_FROM_MESSAGING` | Contact matching creates a new lead           | lead, person, conversation, channelAccount           |
| `AI_ESCALATION`               | AI mode switches to OFF via escalation        | conversation, escalationReason                       |
| `KEYWORD_DETECTED`            | Inbound message contains a configured keyword | message, conversation, matchedKeyword                |

### Trigger Context

Each trigger provides a context object to the condition evaluator:

```typescript
interface RuleContext {
  trigger: AutomationTrigger;
  organizationId: string;
  conversation: Conversation;
  channelAccount: ChannelAccount;
  personChannel?: PersonChannel;
  message?: Message;
  person?: Person;
  lead?: Lead;
  escalationReason?: string;
  matchedKeyword?: string;
}
```

## Condition Schema

<Note>
Conditions are stored as a JSONB array. All conditions in the array must be true for the rule to fire (AND logic). For OR logic, create separate rules.
</Note>

### Condition Object

```typescript
interface RuleCondition {
  field: string; // dot-notation path into RuleContext
  operator: ConditionOperator;
  value: any; // comparison value
}

type ConditionOperator =
  | 'equals' // field === value
  | 'not_equals' // field !== value
  | 'contains' // string contains (case-insensitive)
  | 'not_contains' // string does not contain
  | 'in' // field value is in array
  | 'not_in' // field value is not in array
  | 'gt' // field > value
  | 'gte' // field >= value
  | 'lt' // field < value
  | 'lte' // field <= value
  | 'is_empty' // field is null/undefined/empty string
  | 'is_not_empty' // field has a value
  | 'matches'; // regex match
```

### Condition Examples

```json
[
  { "field": "channelAccount.channel", "operator": "equals", "value": "WHATSAPP" },
  { "field": "message.content", "operator": "contains", "value": "urgent" },
  { "field": "conversation.assignedAgent", "operator": "is_empty", "value": null }
]
```

## Action Schema

Actions are stored as a JSONB array. All actions in the array are executed sequentially when the rule fires.

### Action Object

```typescript
interface RuleAction {
  type: ActionType;
  params: Record<string, any>;
}

type ActionType =
  | 'assign_agent' // Assign conversation to a specific agent
  | 'assign_team' // Assign conversation to a team
  | 'send_template' // Send a message template
  | 'add_tag' // Add tag(s) to the conversation
  | 'remove_tag' // Remove tag(s) from the conversation
  | 'update_lead_stage' // Move the linked lead to a stage
  | 'set_ai_mode' // Set conversation AI mode
  | 'create_task' // Create a CRM task
  | 'notify_agent' // Send notification to agent/team
  | 'close_conversation'; // Close the conversation
```

### Action Parameters

| Action Type          | Parameters                               | Description                                  |
|---------------------|------------------------------------------|----------------------------------------------|
| `assign_agent`       | `{ agentId: UUID }`                      | Assign to specific user                      |
| `assign_team`        | `{ teamId: UUID }`                       | Assign to team (round-robin or direct)       |
| `send_template`      | `{ templateId: UUID }`                   | Send a QUICK_REPLY or META_APPROVED template |
| `add_tag`            | `{ tags: string[] }`                     | Add tags to conversation                     |
| `remove_tag`         | `{ tags: string[] }`                     | Remove tags from conversation                |
| `update_lead_stage`  | `{ stageId: UUID }`                      | Move linked lead to stage                    |
| `set_ai_mode`        | `{ aiMode: AiMode }`                     | Set conversation AI mode                     |
| `create_task`        | `{ subject: string, assigneeId?: UUID }` | Create CRM task linked to conversation       |
| `notify_agent`       | `{ message: string, agentId?: UUID }`    | Send push/WebSocket notification             |
| `close_conversation` | `{}`                                     | Close the conversation                       |

### Action Examples

```json
[
  { "type": "assign_team", "params": { "teamId": "uuid-of-sales-team" } },
  { "type": "add_tag", "params": { "tags": ["urgent", "vip"] } },
  { "type": "send_template", "params": { "templateId": "uuid-of-greeting-template" } }
]
```

## Evaluation Pipeline

### Phase B Integration

Rule evaluation runs as an isolated block in Phase B of the webhook processor:

```typescript
// Phase B block 3: Automation rule evaluation
try {
  await automationRuleService.evaluateRules(trigger, context);
} catch (e) {
  logger.error('Automation rule evaluation failed', e);
}
```

### Evaluation Flow

<Steps>
<Step title="Load active rules">
Load active rules for organization matching trigger type, ordered by priority ASC, created_at ASC
</Step>

<Step title="Evaluate each rule">
For each rule:
- Evaluate all conditions against context (all must pass - AND logic)
- If conditions pass: execute each action sequentially
- If conditions fail: skip and continue to next rule
</Step>

<Step title="Update rule metrics">
- Increment rule.executionCount
- Set rule.lastExecutedAt
- Log execution details
</Step>

<Step title="Flush changes">
Commit all changes to database
</Step>
</Steps>

### Execution Constraints

<Warning>
The following constraints prevent runaway loops and ensure system stability:
</Warning>

- **Max rules per evaluation:** 50 (prevent runaway loops)
- **Max actions per rule:** 10
- **Action timeout:** 5 seconds per action
- **No cascading:** Actions that create new messages do NOT re-trigger rule evaluation (prevents infinite loops)
- **Idempotency:** `assign_agent` on an already-assigned conversation is a no-op. `add_tag` skips existing tags.

## Example Rules

### 1. Auto-assign WhatsApp messages to Sales team

```json
{
  "name": "Route WhatsApp to Sales",
  "trigger": "NEW_MESSAGE",
  "conditions": [
    { "field": "channelAccount.channel", "operator": "equals", "value": "WHATSAPP" },
    { "field": "conversation.assignedAgent", "operator": "is_empty", "value": null }
  ],
  "actions": [{ "type": "assign_team", "params": { "teamId": "sales-team-uuid" } }],
  "priority": 10
}
```

### 2. VIP keyword detection with greeting

```json
{
  "name": "VIP Customer Greeting",
  "trigger": "KEYWORD_DETECTED",
  "conditions": [
    { "field": "matchedKeyword", "operator": "equals", "value": "VIP" }
  ],
  "actions": [
    { "type": "add_tag", "params": { "tags": ["vip"] } },
    { "type": "assign_agent", "params": { "agentId": "senior-agent-uuid" } },
    { "type": "send_template", "params": { "templateId": "vip-greeting-template" } }
  ],
  "priority": 5
}
```

### 3. AI escalation handling

```json
{
  "name": "Handle AI Escalations",
  "trigger": "AI_ESCALATION",
  "conditions": [
    { "field": "escalationReason", "operator": "equals", "value": "COMPLEX_QUERY" }
  ],
  "actions": [
    { "type": "assign_team", "params": { "teamId": "specialist-team-uuid" } },
    { "type": "add_tag", "params": { "tags": ["ai-escalated", "complex"] } },
    { "type": "create_task", "params": { "subject": "AI escalation requires attention" } }
  ],
  "priority": 1
}
```

<Tip>
Use lower priority numbers for more critical rules that should be evaluated first. Rules with the same priority are processed in creation order.
</Tip>
