## Overview

The Workflow Engine provides rule-based automation for the messaging module. It evaluates configurable automation rules against messaging events (new messages, new conversations, AI escalations) and executes actions such as assigning agents, sending templates, creating tasks, and updating lead stages.

<Note>
Rules are evaluated in **Phase B** of the webhook processor pipeline, isolated in a try-catch block so failures never affect message persistence or delivery.
</Note>

## Core Concepts

### Visual Workflow

The current workflow builder persists visual automations in the `workflow` table. Each workflow has a `triggerType`, optional `triggerConfig`, and a JSONB `graph` containing ordered step definitions and their connections.

<Tabs>
  <Tab title="Frontend Builder Model">
    The frontend builder uses an explicit-save model:
    
    - **Save**: `PATCH /v1/workflows/:id` without lifecycle
    - **Publish/Republish**: 
      - New unsaved draft: `POST /v1/workflows` with `status: PUBLISHED` (atomic create + publish)
      - Existing row: `PATCH /v1/workflows/:id` with editor payload plus `status: PUBLISHED`
  </Tab>
  
  <Tab title="Workflow Creation">
    - **Start from scratch**: Opens a local draft with empty graph and no trigger selected
    - **Template**: Opens a local draft from system-level `WorkflowTemplate` record, copying trigger type and graph
  </Tab>
</Tabs>

### Workflow Templates

<Steps>
  <Step title="Template Storage">
    Workflow templates are stored in the database, not hardcoded in the frontend
  </Step>
  
  <Step title="Seeding">
    Templates are seeded by `WorkflowTemplateSeeder` and exposed through `GET /v1/workflows/templates`
  </Step>
  
  <Step title="System Catalog">
    Templates are global system catalog records with no organization ownership
  </Step>
</Steps>

<Warning>
Seeded templates must be ready-to-publish in any organization. They may use generic steps but must not preselect organization-specific AI agents, users, teams, lead stages, or notification recipients.
</Warning>

### Workflow Scope

Workflows can be organization-scoped or personal:

<AccordionGroup>
  <Accordion title="Organization Workflows">
    - Users with `MESSAGING_MANAGE` can create organization workflows by default
    - May choose personal scope in the builder
    - Require messaging view/manage for read and `MESSAGING_MANAGE` for MODIFY/DELETE
  </Accordion>
  
  <Accordion title="Personal Workflows">
    - Users without `MESSAGING_MANAGE` always create personal workflows
    - Owned by the authenticated user
    - Fully accessible to the owner
    - Non-owners need messaging permissions consistent with access control system
  </Accordion>
</AccordionGroup>

<Check>
Publishing is unique by scope: one published organization workflow per `(organization_id, trigger_type)`, and one published personal workflow per `(organization_id, owner_user_id, trigger_type)`.
</Check>

## Branch Conditions

Visual workflow branches use the structured backend `BranchCondition` schema with the following components:

### Schema Structure

```json
{
  "category": "contact_field",
  "field": "email", 
  "operator": "exists",
  "value": null,
  "stepId": null
}
```

### Supported Categories

<Tabs>
  <Tab title="Contact & Lead">
    - `contact_field`: Contact field values resolved through `PersonService`
    - `lead_stage`: Lead stage comparisons against organization's effective stages
  </Tab>
  
  <Tab title="Messages & Timing">
    - `last_incoming_message`: Latest incoming message fields
    - `last_outgoing_message`: Latest outgoing message fields  
    - `time_since_last_incoming_message`: Time elapsed since last incoming
    - `time_since_last_outgoing_message`: Time elapsed since last outgoing
  </Tab>
  
  <Tab title="Assignment & Steps">
    - `assignee_status`: Current assignment status
    - `previous_step_output`: Output from previous workflow step
  </Tab>
</Tabs>

### Operators

<CodeGroup>

```javascript Basic Operators
// Equality
"eq", "neq"

// Comparison  
"gt", "gte", "lt", "lte"

// String Operations
"contains", "not_contains"
```

```javascript Advanced Operators
// Array Operations
"in", "has_any_of", "has_all_of", "has_none_of"

// Existence
"exists", "not_exists"

// Range
"between"
```

</CodeGroup>

<Warning>
Legacy raw field paths, wrapped variables, and deprecated categories (`trigger`, `contact`, `lead`, `conversation`, `previous_step`, `webhook_payload`) are rejected at publish time.
</Warning>

## Graph Validation

Publish-grade graph validation is server-side and uses `node.connections` as the executable source of truth:

<Steps>
  <Step title="Entry Point Validation">
    Requires exactly one entry node
  </Step>
  
  <Step title="Connection Validation">
    Verifies every connection target references an existing node
  </Step>
  
  <Step title="Reachability Check">
    Rejects unreachable nodes from the entry point
  </Step>
  
  <Step title="Cycle Detection">
    Rejects regular graph cycles, permits intentional loops through `JUMP_TO`
  </Step>
</Steps>

### Connection Types

Different step types store different connection patterns:

<Tabs>
  <Tab title="Branch Steps">
    - `BRANCH`: `connections.branches[].nextStepId`
    - `DATE_TIME`: `connections.inWindowBranch` and `connections.outOfWindowBranch`
    - `CHECK_TEAM_AVAILABILITY`: `connections.availableBranch` and `connections.unavailableBranch`
  </Tab>
  
  <Tab title="Failure Handling">
    - Advanced failure branches: `connections.onFailure`
    - Ask Question send-failure: `connections.onMessageFailure`
    - Ask Question timeout: `connections.onTimeout`
  </Tab>
</Tabs>

<Note>
`END` nodes cannot have any outgoing connections.
</Note>

## Step Execution

### Step Handler Results

Step handlers can return the following states:

<CardGroup cols={2}>
  <Card title="COMPLETED" icon="check">
    Step executed successfully, continue to next
  </Card>
  <Card title="FAILED" icon="x">
    Step failed, follow failure path if available
  </Card>
  <Card title="WAITING" icon="clock">
    Step is waiting for external event or delay
  </Card>
  <Card title="SKIPPED" icon="forward">
    Step was skipped, continue through default connection
  </Card>
</CardGroup>

### Execution Locking

<Steps>
  <Step title="Pessimistic Lock">
    `runStep()` takes a pessimistic lock on the `WorkflowExecution` row
  </Step>
  
  <Step title="Attempt Creation">
    Creates or reuses a `WorkflowStepExecution` attempt
  </Step>
  
  <Step title="Handler Execution">
    Most handlers execute inside the transaction for atomicity
  </Step>
  
  <Step title="Long-Running Handlers">
    `SEND_MESSAGE`, `ASK_QUESTION`, and `TRIGGER_ANOTHER_WORKFLOW` are claimed in short transaction, run outside lock, then re-lock to persist results
  </Step>
</Steps>

### Wait Step Behavior

The Wait step stores duration as `durationMs` and follows this pattern:

<Tabs>
  <Tab title="Duration Constraints">
    ```json
    {
      "durationMs": 300000, // 5 minutes in milliseconds
      "min": 1000,          // 1 second minimum
      "max": 604800000      // 7 days maximum
    }
    ```
  </Tab>
  
  <Tab title="Execution Flow">
    1. Handler schedules delayed resume job for the same wait node
    2. Engine completes the waiting step after delay
    3. Continues to `connections.default` if available
    4. Otherwise completes as terminal successful path
  </Tab>
</Tabs>

## Send Message Step

The Send Message step supports different message types:

<AccordionGroup>
  <Accordion title="TEXT Messages">
    - New visual builder steps start as `TEXT` with empty `content`
    - Require non-empty `content`
    - Limited to 2,000 characters
    - Builder enforces character limits in real-time
  </Accordion>
  
  <Accordion title="Template Messages">
    - Support for predefined message templates
    - Variable substitution capabilities
    - Organization-scoped template selection
  </Accordion>
</AccordionGroup>

<Tip>
Frontend controls may display duration in seconds, minutes, hours, or days, but saved workflow JSON must use milliseconds for queue worker compatibility.
</Tip>

## Security and Access Control

### Service-Level Enforcement

Access control uses `AccessService.assertWorkflowAccessInTransaction`:

- **Organization workflows**: Require messaging view/manage for read and `MESSAGING_MANAGE` for modify/delete
- **Personal workflows**: Fully accessible to owner, non-owners need messaging permissions
- **AI-triggered workflows**: Personal agents trigger only owner's personal workflows, organization agents trigger only organization workflows
- **Workflow chains**: Strict scope matches - organization workflows can only trigger organization workflows, personal workflows can only trigger personal workflows owned by the same user

<Warning>
`MESSAGING_MANAGE` does not allow a personal workflow to trigger organization workflows.
</Warning>
