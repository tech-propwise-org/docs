# Final Specification

The Workflow Engine provides rule-based automation for the messaging module. It evaluates configurable automation rules against messaging events (new messages, new conversations, AI escalations) and executes actions such as assigning agents, sending templates, creating tasks, and updating lead stages.

<Note>
Rules are evaluated in **Phase B** of the webhook processor pipeline, isolated in a try-catch block so failures never affect message persistence or delivery.
</Note>

## Core Concepts

### Automation Rule

An automation rule consists of:

- **Trigger** — the event that initiates evaluation
- **Conditions** — JSONB criteria that must be met for the rule to fire
- **Actions** — JSONB definitions of what to do when the rule fires
- **Priority** — evaluation order (lower number = higher priority)

### Entity: AutomationRule

| Field          | Type              | Description                                    |
| -------------- | ----------------- | ---------------------------------------------- |
| id             | UUID              | Primary key                                    |
| organization   | Organization      | Tenant scope                                   |
| name           | string            | Human-readable rule name                       |
| description    | string?           | Optional description of what the rule does     |
| trigger        | AutomationTrigger | Event that activates this rule                 |
| conditions     | JSONB             | Array of condition objects to evaluate         |
| actions        | JSONB             | Array of action objects to execute             |
| isActive       | boolean           | Whether the rule is enabled (default true)     |
| priority       | int               | Evaluation order, lower = first (default 100)  |
| executionCount | int               | How many times this rule has fired (default 0) |
| lastExecutedAt | Date?             | When this rule last fired                      |
| createdBy      | User?             | Who created the rule                           |
| updatedBy      | User?             | Who last modified the rule                     |
| isDeleted      | boolean           | Soft deletion flag                             |
| createdAt      | Date              | Creation timestamp                             |
| updatedAt      | Date              | Last modification timestamp                    |

## Trigger Types

| Trigger                       | Fires When                                    | Context Available                                    |
| ----------------------------- | --------------------------------------------- | ---------------------------------------------------- |
| `NEW_MESSAGE`                 | Any inbound message is received               | message, conversation, personChannel, channelAccount |
| `CONVERSATION_CREATED`        | A new conversation is created                 | conversation, personChannel, channelAccount          |
| `LEAD_CREATED_FROM_MESSAGING` | Contact matching creates a new lead           | lead, person, conversation, channelAccount           |
| `AI_ESCALATION`               | AI mode switches to OFF via escalation        | conversation, escalationReason                       |
| `KEYWORD_DETECTED`            | Inbound message contains a configured keyword | message, conversation, matchedKeyword                |

### Trigger Context

Each trigger provides a context object to the condition evaluator:

```typescript
interface RuleContext {
  trigger: AutomationTrigger;
  organizationId: string;
  conversation: Conversation;
  channelAccount: ChannelAccount;
  personChannel?: PersonChannel;
  message?: Message;
  person?: Person;
  lead?: Lead;
  escalationReason?: string;
  matchedKeyword?: string;
}
```

## Condition Schema

Conditions are stored as a JSONB array. All conditions in the array must be true for the rule to fire (AND logic). For OR logic, create separate rules.

### Condition Object

```typescript
interface RuleCondition {
  field: string; // dot-notation path into RuleContext
  operator: ConditionOperator;
  value: any; // comparison value
}

type ConditionOperator =
  | 'equals' // field === value
  | 'not_equals' // field !== value
  | 'contains' // string contains (case-insensitive)
  | 'not_contains' // string does not contain
  | 'in' // field value is in array
  | 'not_in' // field value is not in array
  | 'gt' // field > value
  | 'gte' // field >= value
  | 'lt' // field < value
  | 'lte' // field <= value
  | 'is_empty' // field is null/undefined/empty string
  | 'is_not_empty' // field has a value
  | 'matches'; // regex match
```

### Condition Examples

```json
[
  { "field": "channelAccount.channel", "operator": "equals", "value": "WHATSAPP" },
  { "field": "message.content", "operator": "contains", "value": "urgent" },
  { "field": "conversation.assignedAgent", "operator": "is_empty", "value": null }
]
```

### Field Paths

| Path                         | Description                                        |
| ---------------------------- | -------------------------------------------------- |
| `channelAccount.channel`     | Channel type (WHATSAPP, INSTAGRAM, MESSENGER)      |
| `message.content`            | Inbound message text                               |
| `message.messageType`        | Message type (TEXT, IMAGE, etc.)                   |
| `conversation.status`        | Conversation status (OPEN, CLOSED)                 |
| `conversation.assignedAgent` | Assigned agent (null if unassigned)                |
| `conversation.assignedTeam`  | Assigned team                                      |
| `conversation.tags`          | Conversation tags array                            |
| `conversation.unreadCount`   | Unread message count                               |
| `personChannel.channelType`  | PersonChannel type (whatsapp, instagram, facebook) |
| `personChannel.displayName`  | Contact display name                               |
| `person.firstName`           | Person first name                                  |
| `lead.intent`                | Lead intent (buyer, seller, renter, etc.)          |
| `lead.temperature`           | Lead temperature (cold, warm, hot)                 |
| `trigger`                    | The trigger type itself                            |

## Action Schema

Actions are stored as a JSONB array. All actions in the array are executed sequentially when the rule fires.

### Action Object

```typescript
interface RuleAction {
  type: ActionType;
  params: Record<string, any>;
}

type ActionType =
  | 'assign_agent' // Assign conversation to a specific agent
  | 'assign_team' // Assign conversation to a team
  | 'send_template' // Send a message template
  | 'add_tag' // Add tag(s) to the conversation
  | 'remove_tag' // Remove tag(s) from the conversation
  | 'update_lead_stage' // Move the linked lead to a stage
  | 'set_ai_mode' // Set conversation AI mode
  | 'create_task' // Create a CRM task
  | 'notify_agent' // Send notification to agent/team
  | 'close_conversation'; // Close the conversation
```

### Action Parameters

<AccordionGroup>
  <Accordion title="assign_agent">
    **Parameters:** `{ agentId: UUID }`
    
    Assigns the conversation to a specific user. If the conversation is already assigned to the same agent, this is a no-op.
  </Accordion>
  
  <Accordion title="assign_team">
    **Parameters:** `{ teamId: UUID }`
    
    Assigns the conversation to a team using round-robin or direct assignment based on team configuration.
  </Accordion>
  
  <Accordion title="send_template">
    **Parameters:** `{ templateId: UUID }`
    
    Sends a QUICK_REPLY or META_APPROVED template to the conversation.
  </Accordion>
  
  <Accordion title="add_tag">
    **Parameters:** `{ tags: string[] }`
    
    Adds specified tags to the conversation. Existing tags are skipped.
  </Accordion>
  
  <Accordion title="remove_tag">
    **Parameters:** `{ tags: string[] }`
    
    Removes specified tags from the conversation.
  </Accordion>
  
  <Accordion title="update_lead_stage">
    **Parameters:** `{ stageId: UUID }`
    
    Moves the linked lead to the specified stage.
  </Accordion>
  
  <Accordion title="set_ai_mode">
    **Parameters:** `{ aiMode: AiMode }`
    
    Sets the conversation AI mode (ON, OFF, HYBRID).
  </Accordion>
  
  <Accordion title="create_task">
    **Parameters:** `{ subject: string, assigneeId?: UUID }`
    
    Creates a CRM task linked to the conversation.
  </Accordion>
  
  <Accordion title="notify_agent">
    **Parameters:** `{ message: string, agentId?: UUID }`
    
    Sends push/WebSocket notification to specified agent or conversation assignee.
  </Accordion>
  
  <Accordion title="close_conversation">
    **Parameters:** `{}`
    
    Closes the conversation.
  </Accordion>
</AccordionGroup>

## Evaluation Pipeline

### Phase B Integration

Rule evaluation runs as an isolated block in Phase B of the webhook processor:

```typescript
// Phase B block 3: Automation rule evaluation
try {
  await automationRuleService.evaluateRules(trigger, context);
} catch (e) {
  logger.error('Automation rule evaluation failed', e);
}
```

### Evaluation Flow

<Steps>
  <Step title="Load active rules">
    Load active rules for organization matching trigger type, ordered by priority ASC, then created_at ASC.
  </Step>
  
  <Step title="Evaluate conditions">
    For each rule, evaluate all conditions against the context. All conditions must pass (AND logic).
  </Step>
  
  <Step title="Execute actions">
    If conditions pass:
    - Execute each action sequentially
    - Increment rule.executionCount
    - Set rule.lastExecutedAt
    - Log execution
    
    If conditions fail, skip to the next rule.
  </Step>
  
  <Step title="Flush changes">
    Persist all changes to the database.
  </Step>
</Steps>

### Execution Constraints

<Warning>
The following constraints prevent runaway automation and ensure system stability:

- **Max rules per evaluation:** 50 (prevent runaway loops)
- **Max actions per rule:** 10
- **Action timeout:** 5 seconds per action
- **No cascading:** Actions that create new messages do NOT re-trigger rule evaluation (prevents infinite loops)
- **Idempotency:** `assign_agent` on an already-assigned conversation is a no-op. `add_tag` skips existing tags.
</Warning>

## API Endpoints

All endpoints require `MESSAGING_MANAGE` permission.

| Method   | Path                              | Description                 |
| -------- | --------------------------------- | --------------------------- |
| `GET`    | `/messaging/automation-rules`     | List rules for organization |
| `GET`    | `/messaging/automation-rules/:id` | Get rule details            |
| `POST`   | `/messaging/automation-rules`     | Create a new rule           |
| `PATCH`  | `/messaging/automation-rules/:id` | Update a rule               |

### 1. Auto-assign WhatsApp messages to Sales team

```json
{
  "name": "Route WhatsApp to Sales",
  "trigger": "NEW_MESSAGE",
  "conditions": [
    { "field": "channelAccount.channel", "operator": "equals", "value": "WHATSAPP" },
    { "field": "conversation.assignedAgent", "operator": "is_empty", "value": null }
  ],
  "actions": [{ "type": "assign_team", "params": { "teamId": "sales-team-uuid" } }],
  "priority": 10
}
```

### 2. VIP keyword detection with greeting

```json
{
  "name": "VIP Customer Greeting",
  "trigger": "KEYWORD_DETECTED",
  "conditions": [
    { "field": "matchedKeyword", "operator": "in", "value": ["vip", "premium", "priority"] }
  ],
  "actions": [
    { "type": "add_tag", "params": { "tags": ["vip"] } },
    { "type": "assign_team", "params": { "teamId": "vip-support-team-uuid" } },
    { "type": "send_template", "params": { "templateId": "vip-greeting-template-uuid" } }
  ],
  "priority": 5
}
```

### 3. AI escalation handling

```json
{
  "name": "AI Escalation to Human Agent",
  "trigger": "AI_ESCALATION",
  "conditions": [
    { "field": "conversation.assignedAgent", "operator": "is_empty", "value": null }
  ],
  "actions": [
    { "type": "assign_team", "params": { "teamId": "support-team-uuid" } },
    { "type": "add_tag", "params": { "tags": ["escalated"] } },
    { "type": "notify_agent", "params": { "message": "AI escalation: Customer needs human assistance" } }
  ],
  "priority": 1
}
```

<Tip>
Use lower priority numbers (1-10) for critical automation rules that should execute first, such as escalations or VIP handling. Use higher numbers (50-100) for general routing and tagging rules.
</Tip>
