</Note>

## Overview

The Workflow Engine provides rule-based automation for the messaging module. It evaluates configurable automation rules against messaging events (new messages, new conversations, AI escalations) and executes actions such as assigning agents, sending templates, creating tasks, and updating lead stages.

<Info>
Rules are evaluated in **Phase B** of the webhook processor pipeline, isolated in a try-catch block so failures never affect message persistence or delivery.
</Info>

## Core Concepts

### Visual Workflow Architecture

The current workflow builder persists visual automations in the `workflow` table. Each workflow contains:

- **triggerType**: The event that initiates the workflow
- **triggerConfig**: Optional configuration for the trigger
- **graph**: JSONB containing ordered step definitions and connections

<Tabs>
  <Tab title="Save Operation">
    `PATCH /v1/workflows/:id` without lifecycle parameter - saves draft changes
  </Tab>
  <Tab title="Publish Operation">
    - **New workflow**: `POST /v1/workflows` with `status: PUBLISHED` (atomic create + publish)
    - **Existing workflow**: `PATCH /v1/workflows/:id` with editor payload plus `status: PUBLISHED`
  </Tab>
  <Tab title="Lifecycle Operations">
    - **Unpublish**: `PATCH` with `status: DRAFT`
    - **Archive**: `PATCH` with `status: ARCHIVED`
  </Tab>
</Tabs>

### Workflow Creation Methods

<CardGroup cols={2}>
  <Card title="Start from Scratch" icon="plus">
    Opens a local draft with empty graph and no trigger selected. User must choose trigger before saving.
  </Card>
  <Card title="Template-Based" icon="template">
    Opens local draft from system-level `WorkflowTemplate` record, copying trigger type and graph. Database row created only on explicit save.
  </Card>
</CardGroup>

### Builder Action Gating

<Warning>
All builder actions require a non-empty **name** before execution. The "Untitled workflow" placeholder does not satisfy this requirement.
</Warning>

<Steps>
  <Step title="Name Validation">
    **Save**, **Test**, and **Publish** are disabled while name is empty/whitespace-only
  </Step>
  <Step title="Trigger Requirements">
    - **Test**: Always requires selected trigger
    - **Save**: Requires trigger except for metadata-only edits on published workflows
    - **Publish**: Requires zero validation issues via `getWorkflowValidationIssues`
  </Step>
  <Step title="Validation Requirements">
    Publish requires: name, trigger, at least one step, and valid per-step configuration
  </Step>
</Steps>

## Workflow Templates

<Info>
Workflow templates are stored in the database and managed through `WorkflowTemplateSeederService`, which runs on every server boot to ensure the global catalog is always present.
</Info>

### Template Management

- **Storage**: Database-based, not hardcoded in frontend
- **Seeding**: Idempotent slug-based upsert via `WorkflowTemplateSeederService`
- **Execution**: Runs in `AppInitializerService.onModuleInit()` on every server boot
- **Scope**: Global system catalog with no organization ownership

<CodeGroup>
```typescript Template Structure
interface WorkflowTemplate {
  slug: string;
  name: string;
  description: string;
  triggerType: WorkflowTriggerType;
  graph: WorkflowGraph;
  isActive: boolean;
}
```

```typescript API Endpoint
GET /v1/workflows/templates
// Returns global template catalog
```
</CodeGroup>

### Template Constraints

<Check>
Seeded templates must be ready-to-publish in any organization and may only use generic steps:
- Message steps
- Question steps  
- Branch steps
- Date/time steps
- Task steps
- Contact update steps
- Unassign steps
</Check>

<Warning>
Templates must not preselect organization-specific resources:
- AI agents
- Users or teams
- Lead stages
- Notification recipients
</Warning>

## Actor Resolution

The Workflow Engine resolves the workflow actor once per execution and threads it through all step contexts.

<Tabs>
  <Tab title="Organization Scope">
    Uses the per-organization AI service user via shared `resolveAiServiceUserId` utility
  </Tab>
  <Tab title="Personal Scope">
    Uses the workflow owner as the actor
  </Tab>
</Tabs>

### Actor Usage

- **Domain Attribution**: Task creation, lead updates, conversation assignments
- **Audit Attribution**: Via `auditContextForActor`
- **Access Control**: Available through `context.actorUserId` in step handlers

<Note>
When actor cannot be resolved (missing AI service user or deleted owner), handlers fail with appropriate error codes rather than writing unattributed records.
</Note>

## Workflow Scope & Permissions

<Tabs>
  <Tab title="Organization Workflows">
    - Created by users with `MESSAGING_MANAGE` permission
    - Can be accessed by users with `MESSAGING_VIEW` or `MESSAGING_MANAGE`
    - Require `MESSAGING_MANAGE` for modification/deletion
    - Unique publishing constraint per `(organization_id, trigger_type)`
  </Tab>
  <Tab title="Personal Workflows">
    - Created by users without `MESSAGING_MANAGE` (always personal)
    - Users with `MESSAGING_MANAGE` can choose personal scope
    - Fully accessible to the owner
    - Non-owners need messaging permissions for access
    - Unique publishing constraint per `(organization_id, owner_user_id, trigger_type)`
  </Tab>
</Tabs>

### Access Control Rules

<Steps>
  <Step title="Workflow Listing">
    - Users with `MESSAGING_VIEW`/`MESSAGING_MANAGE`: See both org and personal workflows
    - Users without permissions: Only their own personal workflows
    - `scope=ORGANIZATION` returns no rows for users without messaging permissions
  </Step>
  <Step title="Cross-Scope Triggers">
    - Personal AI agents → Only owner's personal workflows
    - Organization AI agents → Only organization workflows
    - Workflow chains must maintain scope consistency
  </Step>
  <Step title="Service Enforcement">
    Uses `AccessService.assertWorkflowAccessInTransaction` for permission validation
  </Step>
</Steps>

## Branch Conditions

Visual workflow branches use structured `BranchCondition` schema with strict validation.

### Condition Structure

<CodeGroup>
```typescript BranchCondition Schema
interface BranchCondition {
  category: string;
  field: string;
  operator: string;
  value?: any;
  stepId?: string; // Only when category is 'previous_step_output'
}
```

```typescript Supported Categories
const categories = [
  'contact_field',
  'previous_step_output', 
  'assignee_status',
  'last_incoming_message',
  'last_outgoing_message',
  'time_since_last_incoming_message',
  'time_since_last_outgoing_message',
  'lead_stage'
];
```
</CodeGroup>

### Operators

<AccordionGroup>
  <Accordion title="Comparison Operators">
    - `eq`, `neq` - Equality/inequality
    - `gt`, `gte`, `lt`, `lte` - Numeric/date comparisons
  </Accordion>
  <Accordion title="String Operators">
    - `contains`, `not_contains` - Substring matching
  </Accordion>
  <Accordion title="Array Operators">
    - `in` - Value in array (requires non-empty array)
    - `has_any_of`, `has_all_of`, `has_none_of` - Array intersection
  </Accordion>
  <Accordion title="Existence Operators">
    - `exists`, `not_exists` - Field presence (no value required)
  </Accordion>
  <Accordion title="Range Operators">
    - `between` - Range check (requires two-element bound)
  </Accordion>
</AccordionGroup>

<Warning>
Legacy formats are rejected at publish time:
- Raw field paths
- Wrapped variables  
- Deprecated categories (`trigger`, `contact`, `lead`, `conversation`, `previous_step`, `webhook_payload`)
- Serialized `source` properties
</Warning>

## Graph Validation

<Steps>
  <Step title="Entry Point Validation">
    Requires exactly one entry node in the workflow graph
  </Step>
  <Step title="Connection Validation">
    Every connection target must reference an existing node
  </Step>
  <Step title="Reachability Check">
    Rejects unreachable nodes that cannot be reached from entry point
  </Step>
  <Step title="Cycle Detection">
    - Rejects regular graph cycles
    - Permits intentional loops through `JUMP_TO` with bounded `maxJumps`
  </Step>
  <Step title="Terminal Validation">
    `END` nodes cannot have any outgoing connections
  </Step>
</Steps>

## Wait Step Implementation

<Info>
Wait steps store duration as `durationMs` in milliseconds. Frontend controls may display other units, but saved JSON must use milliseconds for queue worker compatibility.
</Info>

### Duration Constraints

- **Minimum**: 1 second (1000ms)
- **Maximum**: 7 days (604,800,000ms)
- **Storage**: Always in milliseconds regardless of UI display

<CodeGroup>
```typescript Wait Step Config
interface WaitStepConfig {
  durationMs: number; // 1000 to 604800000
}
```

```typescript Handler Implementation
// Wait handler schedules delayed resume job
const resumeJob = {
  stepId: waitNode.id,
  executionId: context.executionId,
  scheduledFor: new Date(Date.now() + config.durationMs)
};
```
</CodeGroup>

## Step Management

### Adding Steps

In the vertical builder, adding steps automatically wires the selected output connection to the new node, maintaining graph connectivity.

### Deleting Steps

<Warning>
Step deletion is graph-aware and requires user confirmation when downstream targets exist.
</Warning>

<Steps>
  <Step title="Impact Analysis">
    Check if selected step has downstream targets through:
    - `connections.default`
    - `connections.onFailure` 
    - `connections.onTimeout`
    - Named branch outputs
  </Step>
  <Step title="Confirmation Prompt">
    User must confirm deletion when downstream impacts are detected
  </Step>
  <Step title="Cleanup Process">
    - Remove selected step and reachable downstream path
    - Preserve unrelated sibling branches
    - Clean up broken references from named output connections
    - Remove references from branch targets
  </Step>
</Steps>

## Runtime Execution

<Note>
The engine uses `node.connections` as the executable source of truth for step progression, while the `edges` array is retained for visual/editor compatibility.
</Note>

### Execution Flow

1. **Actor Resolution**: Resolve workflow actor once per execution
2. **Context Threading**: Thread actor through all step contexts via `StepContext.actorUserId`
3. **Step Progression**: Follow connections and handler overrides for navigation
4. **Error Isolation**: Failures contained within try-catch blocks
5. **Audit Tracking**: All actions attributed via `auditContextForActor`

### Variable Management

Runtime tracking uses `execution.variables.steps[stepId]` for step output storage, with technical fields (`messageId`, `pendingResponseId`, `outboundPurpose`) remaining internal and not exposed as branchable outputs.
