## Overview

The Workflow Engine provides rule-based automation for the messaging module. It evaluates configurable automation rules against messaging events (new messages, new conversations, AI escalations) and executes actions such as assigning agents, sending templates, creating tasks, and updating lead stages.

<Note>
Rules are evaluated in **Phase B** of the webhook processor pipeline, isolated in a try-catch block so failures never affect message persistence or delivery.
</Note>

## Core Concepts

### Visual Workflow

The current workflow builder persists visual automations in the `workflow` table. Each workflow has a `triggerType`, optional `triggerConfig`, and a JSONB `graph` containing ordered step definitions and their connections.

<Tabs>
  <Tab title="Frontend Builder">
    The frontend builder uses an explicit-save model:
    
    - **Save**: `PATCH /v1/workflows/:id` without lifecycle
    - **Publish/Republish**: 
      - New unsaved draft: `POST /v1/workflows` with `status: PUBLISHED` (atomic create + publish)
      - Existing row: `PATCH /v1/workflows/:id` with editor payload plus `status: PUBLISHED`
    - **Unpublish/Archive**: `PATCH` with `status: DRAFT` or `status: ARCHIVED`
  </Tab>
  
  <Tab title="Workflow Creation">
    - **Start from scratch**: Opens a local draft with empty graph and no trigger selected
    - **Template**: Opens a local draft from a system-level `WorkflowTemplate` record, copying trigger type and graph
  </Tab>
</Tabs>

### Builder Action Gating

<Warning>
A non-empty **name is required** before any action. Save, Test, and Publish are all disabled while name is empty/whitespace-only.
</Warning>

<Steps>
  <Step title="Name Validation">
    The "Untitled workflow" placeholder does not count as a valid name
  </Step>
  
  <Step title="Action Requirements">
    - **Test**: Always requires a selected trigger
    - **Save**: Requires a trigger except for metadata-only edits on published workflows
    - **Publish**: Requires zero `getWorkflowValidationIssues` (name, trigger, at least one step, valid per-step config)
  </Step>
</Steps>

### Workflow Templates

<Info>
Workflow templates are stored in the database, not hardcoded in the frontend.
</Info>

Templates are managed through:
- `WorkflowTemplateSeederService` for canonical catalog and idempotent slug-based upsert
- `AppInitializerService.onModuleInit()` runs on every server boot
- Global `workflow_template` catalog is always present without manual seed step
- Templates exposed through `GET /v1/workflows/templates`

<Note>
Seeded templates must be ready-to-publish in any organization. They may use generic steps but must not preselect organization-specific AI agents, users, teams, lead stages, or notification recipients.
</Note>

## Workflow Scope and Permissions

### Organization vs Personal Workflows

<Tabs>
  <Tab title="Organization Workflows">
    - Users with `MESSAGING_MANAGE` can create organization workflows by default
    - May choose personal scope in the builder
    - Require messaging view/manage for read and `MESSAGING_MANAGE` for modify/delete
  </Tab>
  
  <Tab title="Personal Workflows">
    - Users without `MESSAGING_MANAGE` always create personal workflows
    - Owned by the authenticated user
    - Fully accessible to the owner
    - Non-owners need messaging permissions per access control system
  </Tab>
</Tabs>

### Publishing Constraints

<Check>
Publishing is unique by scope:
- One published organization workflow per `(organization_id, trigger_type)`
- One published personal workflow per `(organization_id, owner_user_id, trigger_type)`
</Check>

Database enforces this with partial unique indexes:
- `workflow_org_published_trigger_unique`
- `workflow_personal_published_trigger_unique`

## Branch Conditions

Visual workflow branches use the structured backend `BranchCondition` schema:

```json
{
  "category": "contact_field",
  "field": "email",
  "operator": "exists",
  "value": null,
  "stepId": null
}
```

### Supported Categories

<AccordionGroup>
  <Accordion title="contact_field">
    Values resolve through `PersonService` workflow branch helpers
  </Accordion>
  
  <Accordion title="previous_step_output">
    Reads from `execution.variables.steps[stepId]`. Requires selected `stepId` to be the Branch node's single direct previous step.
  </Accordion>
  
  <Accordion title="assignee_status">
    Resolves through `WorkflowMessageStatusReaderService` with explicit organization filters
  </Accordion>
  
  <Accordion title="last_incoming_message / last_outgoing_message">
    Latest message reads order by `createdAt DESC, id DESC` and expose only Branch catalog fields
  </Accordion>
  
  <Accordion title="time_since_last_incoming_message / time_since_last_outgoing_message">
    Time-based comparisons for message activity
  </Accordion>
  
  <Accordion title="lead_stage">
    Compares against organization's effective lead stages. Publish validation rejects unknown stage IDs.
  </Accordion>
</AccordionGroup>

### Operators

<Tabs>
  <Tab title="Comparison">
    - `eq` - equals
    - `neq` - not equals
    - `gt` - greater than
    - `gte` - greater than or equal
    - `lt` - less than
    - `lte` - less than or equal
  </Tab>
  
  <Tab title="String">
    - `contains`
    - `not_contains`
  </Tab>
  
  <Tab title="Array">
    - `in` - requires non-empty array
    - `has_any_of`
    - `has_all_of`
    - `has_none_of`
  </Tab>
  
  <Tab title="Existence">
    - `exists` - requires no value
    - `not_exists` - requires no value
  </Tab>
  
  <Tab title="Range">
    - `between` - requires two-element bound
  </Tab>
</Tabs>

## Step Types and Connections

### Connection Types

Branching steps must persist explicit output targets:

- **BRANCH**: `connections.branches[].nextStepId`
- **DATE_TIME**: `connections.inWindowBranch` and `connections.outOfWindowBranch`
- **CHECK_TEAM_AVAILABILITY**: `connections.availableBranch` and `connections.unavailableBranch`

### Advanced Connections

- **Failure branches**: `connections.onFailure`
- **Ask Question send-failure**: `connections.onMessageFailure`
- **Ask Question timeout**: `connections.onTimeout`

### Wait Step Configuration

<Warning>
Wait durations must be stored as `durationMs` in milliseconds, between 1 second and 7 days.
</Warning>

The Wait step:
1. Stores duration as `durationMs`
2. Schedules a delayed resume job for the same wait node
3. Completes the waiting step and continues to `connections.default`
4. If no default target exists, completes as terminal successful path after delay

## Graph Validation

Publish-grade graph validation is server-side and uses `node.connections` as the executable source of truth.

<Steps>
  <Step title="Entry Point">
    Requires exactly one entry node
  </Step>
  
  <Step title="Connection Verification">
    Verifies every connection target references an existing node
  </Step>
  
  <Step title="Reachability">
    Rejects unreachable nodes
  </Step>
  
  <Step title="Cycle Detection">
    Rejects regular graph cycles, permits intentional loops through `JUMP_TO` with bounded `maxJumps`
  </Step>
  
  <Step title="Terminal Nodes">
    `END` nodes cannot have any outgoing connections
  </Step>
</Steps>

<Note>
The `edges` array is retained for visual/editor parity, but runtime step progression follows connections and handler overrides.
</Note>

## Step Execution

### Handler Return Values

Step handlers can return:

- `COMPLETED` - Step finished successfully
- `FAILED` - Step encountered an error
- `WAITING` - Step is waiting for external event
- `SKIPPED` - Step was skipped, continues to default connection

### Execution Locking

<Tip>
`runStep()` takes a pessimistic lock on the `WorkflowExecution` row while creating or reusing a `WorkflowStepExecution` attempt.
</Tip>

**Short Transaction Handlers:**
- Most handlers execute inside the workflow transaction
- Workflow state and domain mutations remain atomic

**Long Transaction Handlers:**
- `SEND_MESSAGE`, `ASK_QUESTION`, and `TRIGGER_ANOTHER_WORKFLOW`
- Claimed in short transaction, run outside workflow row lock
- Re-lock execution to persist handler result
- Serializes duplicate queue jobs

## API Endpoints

### Workflow Management

```http
GET /v1/workflows/templates
POST /v1/workflows
PATCH /v1/workflows/:id
```

### Access Control

- **Workflow list requests**: Accept optional `scope` parameter
- **Users with MESSAGING_VIEW/MANAGE**: List both organization and personal workflows
- **Users without messaging permissions**: Receive only their own personal workflows
- **scope=ORGANIZATION**: Returns no rows for users without messaging permissions

### Service-Level Enforcement

Uses `AccessService.assertWorkflowAccessInTransaction`:
- Organization workflows require messaging view/manage for read
- Organization workflows require `MESSAGING_MANAGE` for MODIFY/DELETE
- Personal workflows fully accessible to owner
- Non-owners need messaging permissions per access control system
