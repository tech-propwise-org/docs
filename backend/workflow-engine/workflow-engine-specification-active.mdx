## Overview

The Workflow Engine provides rule-based automation for the messaging module. It evaluates configurable automation rules against messaging events (new messages, new conversations, AI escalations) and executes actions such as assigning agents, sending templates, creating tasks, and updating lead stages.

<Note>
Rules are evaluated in **Phase B** of the webhook processor pipeline, isolated in a try-catch block so failures never affect message persistence or delivery.
</Note>

## Core Concepts

### Visual Workflow Architecture

The workflow builder persists visual automations in the `workflow` table. Each workflow contains:

- **triggerType** - The event that initiates the workflow
- **triggerConfig** - Optional configuration for the trigger
- **graph** - JSONB containing ordered step definitions and connections

<Tabs>
  <Tab title="Workflow Creation">
    The frontend builder uses an explicit-save model:
    
    - **Save**: `PATCH /v1/workflows/:id` without lifecycle changes
    - **Publish/Republish**: 
      - New draft: `POST /v1/workflows` with `status: PUBLISHED` (atomic create + publish)
      - Existing workflow: `PATCH /v1/workflows/:id` with editor payload plus `status: PUBLISHED`
    - **Unpublish/Archive**: `PATCH` with `status: DRAFT` or `status: ARCHIVED`
  </Tab>
  
  <Tab title="Builder Modes">
    - **Start from scratch** - Opens local draft with empty graph, no trigger selected
    - **Template** - Opens local draft from system-level `WorkflowTemplate` record
  </Tab>
</Tabs>

### Builder Action Gating

<Warning>
A non-empty **name is required** before any action. Save, Test, and Publish are disabled while name is empty or whitespace-only.
</Warning>

Action requirements:
- **Test**: Always requires selected trigger
- **Save**: Requires trigger (except metadata-only edits on published workflows)
- **Publish**: Requires zero validation issues (name, trigger, at least one step, valid configuration)

### Workflow Templates

Templates are stored in the database and managed by `WorkflowTemplateSeederService`. The service runs on every server boot to ensure the global catalog is always present.

<Steps>
  <Step title="Template Seeding">
    Templates are upserted by stable slug on each boot via `TenantContext.executeWithBypass()`
  </Step>
  
  <Step title="Template Access">
    Templates are exposed through `GET /v1/workflows/templates` as global system records
  </Step>
  
  <Step title="Template Usage">
    Seeded templates must be ready-to-publish in any organization with generic steps only
  </Step>
</Steps>

### Workflow Actor Resolution

<Info>
The engine resolves the workflow actor once per execution and threads it to every step's `StepContext.actorUserId`.
</Info>

Actor resolution by scope:
- **ORGANIZATION**: Per-org AI service user via `resolveAiServiceUserId`
- **PERSONAL**: The workflow owner

The resolved actor is used for:
- Domain attribution (tasks, lead updates, assignments)
- Audit row attribution via `auditContextForActor`

<Warning>
When the actor cannot be resolved (missing AI service user or deleted owner), handlers fail with appropriate error codes rather than writing unattributed records.
</Warning>

## Workflow Scopes

Workflows can be **organization-scoped** or **personal**:

### Organization Workflows
- Created by users with `MESSAGING_MANAGE` permission
- One published workflow per `(organization_id, trigger_type)`
- Enforced by `workflow_org_published_trigger_unique` index

### Personal Workflows  
- Created by workflow owner (users without `MESSAGING_MANAGE` create personal by default)
- One published workflow per `(organization_id, owner_user_id, trigger_type)`
- Enforced by `workflow_personal_published_trigger_unique` index

### Access Control

<Tabs>
  <Tab title="Organization Workflows">
    - **Read**: Requires `MESSAGING_VIEW` or `MESSAGING_MANAGE`
    - **Modify/Delete**: Requires `MESSAGING_MANAGE`
  </Tab>
  
  <Tab title="Personal Workflows">
    - **Owner**: Full access to their workflows
    - **Non-owners**: Need messaging permissions per `ACCESS_CONTROL_SYSTEM.md`
  </Tab>
</Tabs>

<Note>
Workflow-to-workflow chains require strict scope matches. Personal workflows can only trigger personal workflows owned by the same user.
</Note>

## Branch Conditions

Visual workflow branches use structured `BranchCondition` schema with:
- `category` - The data category to evaluate
- `field` - Specific field within the category  
- `operator` - Comparison operator
- `value` - Optional comparison value
- `stepId` - Required only for `previous_step_output` category

<Warning>
Legacy raw field paths, wrapped variables, and deprecated categories are rejected at publish time.
</Warning>

### Supported Categories

<AccordionGroup>
  <Accordion title="contact_field">
    Values resolve through `PersonService` workflow branch helpers
  </Accordion>
  
  <Accordion title="previous_step_output">
    Reads from `execution.variables.steps[stepId]`. Requires `stepId` to be the Branch node's single direct previous step.
  </Accordion>
  
  <Accordion title="assignee_status">
    Resolves through `WorkflowMessageStatusReaderService` with organization filters
  </Accordion>
  
  <Accordion title="Message Fields">
    - `last_incoming_message`
    - `last_outgoing_message` 
    - `time_since_last_incoming_message`
    - `time_since_last_outgoing_message`
    
    Latest message reads order by `createdAt DESC, id DESC`
  </Accordion>
  
  <Accordion title="lead_stage">
    Compares against organization's effective lead stages. Publish validation rejects unknown stage IDs.
  </Accordion>
</AccordionGroup>

### Operators

<CodeGroup>
```json Basic Comparison
{
  "operators": [
    "eq", "neq", "gt", "gte", "lt", "lte"
  ]
}
```

```json String Operations
{
  "operators": [
    "contains", "not_contains"
  ]
}
```

```json Set Operations  
{
  "operators": [
    "in", "has_any_of", "has_all_of", "has_none_of"
  ]
}
```

```json Existence Checks
{
  "operators": [
    "exists", "not_exists", "between"
  ]
}
```
</CodeGroup>

<Info>
Operators have per-field allowlists and validation is type-aware. Branch evaluation preserves order with `AND`/`OR` semantics.
</Info>

## Graph Validation

Publish-grade validation is server-side and uses `node.connections` as the executable source of truth.

<Steps>
  <Step title="Entry Point">
    Requires exactly one entry node
  </Step>
  
  <Step title="Connection Integrity">
    Verifies every connection target references an existing node
  </Step>
  
  <Step title="Reachability">
    Rejects unreachable nodes from the entry point
  </Step>
  
  <Step title="Cycle Detection">
    Rejects regular graph cycles, permits intentional loops through `JUMP_TO` with bounded `maxJumps`
  </Step>
  
  <Step title="End Nodes">
    `END` nodes cannot have outgoing connections
  </Step>
</Steps>

### Connection Types

Different step types store connections in specific formats:

    ```json
    }
    ```
  </Tab>
</Tabs>

## Wait Step Configuration

<Warning>
Wait durations must be stored as `durationMs` in milliseconds, between 1 second and 7 days.
</Warning>

The Wait step:
1. Stores duration as `durationMs` in the workflow JSON
2. Frontend controls may display seconds/minutes/hours/days  
3. Handler schedules a delayed resume job for the same wait node
4. Queue worker processes delayed continuation correctly

<Check>
This ensures consistent timing across the workflow execution pipeline regardless of frontend display preferences.
</Check>
