## Overview

The Workflow Engine provides rule-based automation for the messaging module. It evaluates configurable automation rules against messaging events (new messages, new conversations, AI escalations) and executes actions such as assigning agents, sending templates, creating tasks, and updating lead stages.

<Note>
Rules are evaluated in **Phase B** of the webhook processor pipeline, isolated in a try-catch block so failures never affect message persistence or delivery.
</Note>

## Core Concepts

### Visual Workflow

The current workflow builder persists visual automations in the `workflow` table. Each workflow has a `triggerType`, optional `triggerConfig`, and a JSONB `graph` containing ordered step definitions and their connections.

<Info>
The frontend builder uses an explicit-save model for **Save** and **Publish** operations with different behaviors for new vs existing workflows.
</Info>

#### Workflow Creation Models

<Tabs>
<Tab title="Start from Scratch">
Opens a local draft with an empty graph and no trigger selected. The user must choose a trigger before saving.
</Tab>
<Tab title="Template-based">
Opens a local draft from a system-level `WorkflowTemplate` record, copying its trigger type and graph into an editable local workflow. The database row is created only on explicit save.
</Tab>
</Tabs>

#### Save vs Publish Operations

<Steps>
<Step title="Save Operation">
`PATCH /v1/workflows/:id` without lifecycle - creates or updates a draft workflow
</Step>
<Step title="Publish New Workflow">
`POST /v1/workflows` with `status: PUBLISHED` - atomic create + publish operation
</Step>
<Step title="Republish Existing">
`PATCH /v1/workflows/:id` with editor payload plus `status: PUBLISHED` - merge, publish-grade validation, version bump when structural fields change
</Step>
<Step title="Lifecycle Changes">
Unpublish and archive use `PATCH` with `status: DRAFT` or `status: ARCHIVED` (lifecycle-only rules apply)
</Step>
</Steps>

### Supported Trigger Types

The visual builder trigger picker exposes this reduced set:

- `FIRST_MESSAGE_RECEIVED` — First Message Received
- `MESSAGING_WINDOW_REOPENED` — Messaging Window Reopened  
- `CONTACT_FIELD_UPDATED` — Contact Field Updated
- `ASSIGNMENT_CHANGED` — Assignment Changed
- `LEAD_CREATED_FROM_CAMPAIGN` — Lead Created from Campaign
- `LIFECYCLE_UPDATED` — Lead Moved to Stage
- `INCOMING_WEBHOOK` — Webhook / Integration Event

<Check>
The backend `WorkflowTriggerType` enum, frontend API enum, and database CHECK constraints are aligned to this same seven-trigger set.
</Check>

## Workflow Scoping and Access Control

### Organization vs Personal Workflows

<AccordionGroup>
<Accordion title="Permission-based Creation">
- Users with `MESSAGING_MANAGE` can create organization workflows by default and may choose personal scope
- Users without `MESSAGING_MANAGE` always create personal workflows owned by the authenticated user
</Accordion>

<Accordion title="Coexistence Rules">
- Organization and personal workflows with the same `triggerType` may coexist
- Publishing is unique by scope: one published organization workflow per `(organization_id, trigger_type)`
- One published personal workflow per `(organization_id, owner_user_id, trigger_type)`
</Accordion>

<Accordion title="Database Constraints">
- `workflow_org_published_trigger_unique` - enforces organization workflow uniqueness
- `workflow_personal_published_trigger_unique` - enforces personal workflow uniqueness
</Accordion>
</AccordionGroup>

### Access Control Rules

<Tabs>
<Tab title="List Access">
- Users with `MESSAGING_VIEW` or `MESSAGING_MANAGE` list both organization and personal workflows
- Users without those permissions receive only their own personal workflows
- `scope=ORGANIZATION` returns no rows for users without messaging permissions
</Tab>
<Tab title="Modification Access">
- Organization workflows require `MESSAGING_MANAGE` for MODIFY/DELETE
- Personal workflows are fully accessible to the owner
- Non-owners need messaging permissions per `ACCESS_CONTROL_SYSTEM.md`
</Tab>
</AccordionGroup>

### Workflow Trigger Chains

<Warning>
Workflow-to-workflow chains enforce strict scope matching:
- Organization workflows can trigger only organization workflows
- Personal workflows can trigger only personal workflows owned by the same user
- `MESSAGING_MANAGE` does not allow personal workflows to trigger organization workflows
</Warning>

## Branch Conditions and Logic

### Branch Condition Schema

Visual workflow branches use the structured backend `BranchCondition` schema:

```json
{
  "category": "contact_field",
  "field": "email", 
  "operator": "exists",
  "value": null,
  "stepId": null
}
```

<Note>
Legacy raw field paths, wrapped variables, and deprecated categories are rejected at publish time.
</Note>

### Supported Categories

<CodeGroup>
```yaml Contact Fields
category: contact_field
fields: [email, firstName, lastName, phone, etc.]
operators: [eq, neq, contains, exists, etc.]
```

```yaml Previous Step Output  
category: previous_step_output
fields: [depends on step type]
operators: [eq, neq, gt, gte, etc.]
stepId: [required - must be direct previous step]
```

```yaml Assignee Status
category: assignee_status
fields: [isAssigned, assigneeType]  
operators: [eq, neq]
```

```yaml Message Timing
category: time_since_last_incoming_message
fields: [minutes, hours, days]
operators: [gt, gte, lt, lte, between]
```
</CodeGroup>

### Branch Operators

<AccordionGroup>
<Accordion title="Comparison Operators">
`eq`, `neq`, `gt`, `gte`, `lt`, `lte` - Standard comparison with type-aware validation
</Accordion>

<Accordion title="Text Operators">
`contains`, `not_contains` - String matching operations
</Accordion>

<Accordion title="Set Operators">
`in`, `has_any_of`, `has_all_of`, `has_none_of` - Array and set operations
</Accordion>

<Accordion title="Existence Operators">
`exists`, `not_exists` - Null/undefined checks (require no value)
</Accordion>

<Accordion title="Range Operators">
`between` - Requires two-element bound array
</Accordion>
</AccordionGroup>

## Step Execution and Flow Control

### Step Handler Results

Step handlers can return four different statuses:

<Tabs>
<Tab title="COMPLETED">
Step executed successfully, continue to next step via default connection
</Tab>
<Tab title="FAILED">
Step failed, continue to failure branch if configured, otherwise terminate workflow
</Tab>
<Tab title="WAITING">
Step is waiting for external event (user response, delay, etc.), pause execution
</Tab>
<Tab title="SKIPPED">
Step was skipped due to conditions, continue to default connection if exists
</Tab>
</Tabs>

### Execution Locking Strategy

<Steps>
<Step title="Pessimistic Lock">
`runStep()` takes a pessimistic lock on the `WorkflowExecution` row
</Step>
<Step title="Attempt Tracking">
Creates or reuses a `WorkflowStepExecution` attempt with incrementing counter
</Step>
<Step title="Transaction Scope">
Most handlers execute inside the workflow transaction for atomicity
</Step>
<Step title="Long Operations">
`SEND_MESSAGE`, `ASK_QUESTION`, and `TRIGGER_ANOTHER_WORKFLOW` are claimed in short transaction, run outside lock, then re-lock to persist results
</Step>
</Steps>

### Wait Step Implementation

<Info>
The Wait step stores duration as `durationMs` in the workflow configuration.
</Info>

```json
{
  "type": "WAIT",
  "config": {
    "durationMs": 300000
  },
  "connections": {
    "default": "next-step-id"
  }
}
```

<Note>
Wait durations must be between 1 second and 7 days. The handler schedules a delayed resume job, then completes the waiting step after the delay.
</Note>

### Graph Validation

<AccordionGroup>
<Accordion title="Entry Point Validation">
- Requires exactly one entry node
- Verifies every connection target references an existing node
</Accordion>

<Accordion title="Reachability Checks">
- Rejects unreachable nodes  
- Rejects regular graph cycles
- Permits intentional loops through `JUMP_TO` with bounded `maxJumps`
</Accordion>

<Accordion title="Terminal Node Rules">
- `END` nodes cannot have any outgoing connections
- Uses `node.connections` as executable source of truth
</Accordion>
</AccordionGroup>

## Message and Communication Steps

### Send Message Step

<Tabs>
<Tab title="Text Messages">
```json
{
  "type": "SEND_MESSAGE",
  "config": {
    "messageType": "TEXT",
    "content": "Hello! How can I help you today?"
  }
}
```
- Require non-empty `content`
- Limited to 2,000 characters
</Tab>
<Tab title="Template Messages">
```json
{
  "type": "SEND_MESSAGE", 
  "config": {
    "messageType": "TEMPLATE",
    "templateId": "welcome-template-123"
  }
}
```
- Reference pre-configured message templates
- Support dynamic variable substitution
</Tab>
</Tabs>

### Ask Question Step

The Ask Question step sends a message and waits for user response with configurable timeout and failure handling.

<Warning>
Ask Question steps support multiple output branches:
- `connections.default` - Normal response received
- `connections.onTimeout` - No response within timeout period  
- `connections.onMessageFailure` - Message delivery failed
</Warning>

<CodeGroup>

## Error Handling and Recovery

### Failure Isolation

<Check>
Workflow execution failures are isolated and never affect message persistence or delivery since they run in Phase B of the webhook processor.
</Check>

### Retry Mechanisms

- Failed steps can be configured with automatic retry policies
- Delayed retries preserve attempt history without violating uniqueness constraints
- Stale job protection prevents duplicate execution of completed steps

### Maximum Iteration Safety

<Warning>
`MAX_STEP_ITERATIONS` provides a safety bound to prevent runaway workflows from consuming excessive resources.
</Warning>
