## Overview

The Workflow Engine provides rule-based automation for the messaging module. It evaluates configurable automation rules against messaging events (new messages, new conversations, AI escalations) and executes actions such as assigning agents, sending templates, creating tasks, and updating lead stages.

<Note>
Rules are evaluated in **Phase B** of the webhook processor pipeline, isolated in a try-catch block so failures never affect message persistence or delivery.
</Note>

## Core concepts

### Visual workflow

The current workflow builder persists visual automations in the `workflow` table. Each workflow has a `triggerType`, optional `triggerConfig`, and a JSONB `graph` containing ordered step definitions and their connections.

<Tabs>
  <Tab title="Save & Publish">
    The frontend builder uses an explicit-save model:
    
    - **Save**: `PATCH /v1/workflows/:id` without lifecycle
    - **Publish/Republish**: 
      - New unsaved draft: `POST /v1/workflows` with `status: PUBLISHED` (atomic create + publish)
      - Existing row: `PATCH /v1/workflows/:id` with editor payload plus `status: PUBLISHED` (merge, publish-grade validation, version bump when structural fields change)
    - **Unpublish/Archive**: `PATCH` with `status: DRAFT` or `status: ARCHIVED`
  </Tab>
  
  <Tab title="Starting Points">
    - **Start from scratch**: Opens a local draft with an empty graph and no trigger selected. User must choose a trigger before saving.
    - **Template**: Opens a local draft from a system-level `WorkflowTemplate` record, copying its trigger type and graph into an editable local workflow. Database row is created only on explicit save.
  </Tab>
</Tabs>

### Builder action gating

<Warning>
A non-empty **name is required** before any action. Save, Test, and Publish are all disabled while `name` is empty/whitespace-only. The "Untitled workflow" placeholder does not count.
</Warning>

Action requirements:
- **Test**: Always requires a selected trigger
- **Save**: Requires a trigger except for metadata-only edits on already-published workflows
- **Publish**: Requires zero `getWorkflowValidationIssues` (name, trigger, at least one step, and valid per-step config)

### Workflow templates

Workflow templates are stored in the database and managed by `WorkflowTemplateSeederService`:

<Steps>
  <Step title="Template seeding">
    The service runs on **every server boot** inside `AppInitializerService.onModuleInit()` within a `TenantContext.executeWithBypass()` transaction.
  </Step>
  
  <Step title="Template access">
    Templates are exposed through `GET /v1/workflows/templates` as global system catalog records with no organization ownership.
  </Step>
  
  <Step title="Template constraints">
    Seeded templates must be ready-to-publish in any organization. They may use generic steps but must not preselect organization-specific resources.
  </Step>
</Steps>

<Info>
Templates are upserted by stable slug and reactivated if a previously soft-deleted slug is seeded again, so catalog edits replace stale template graphs.
</Info>

## Workflow scoping

Workflows can be organization-scoped or personal with specific access controls:

### User permissions

<AccordionGroup>
  <Accordion title="Users with MESSAGING_MANAGE">
    - Can create organization workflows by default
    - May choose personal scope in the builder
    - Can view and manage both organization and personal workflows
  </Accordion>
  
  <Accordion title="Users without MESSAGING_MANAGE">
    - Always create personal workflows owned by the authenticated user
    - Only see their own personal workflows
    - `scope=ORGANIZATION` returns no rows for them
  </Accordion>
</AccordionGroup>

### Publishing constraints

<Note>
Publishing is unique by scope:
- One published organization workflow per `(organization_id, trigger_type)`
- One published personal workflow per `(organization_id, owner_user_id, trigger_type)`
</Note>

The database enforces these constraints with partial unique indexes:
- `workflow_org_published_trigger_unique`
- `workflow_personal_published_trigger_unique`

### Access control

Service-level enforcement uses `AccessService.assertWorkflowAccessInTransaction`:

- **Organization workflows**: Require messaging view/manage for read and `MESSAGING_MANAGE` for MODIFY/DELETE
- **Personal workflows**: Fully accessible to the owner; non-owners need messaging permissions

<Warning>
AI-triggered child workflows are scope-checked. Personal agents can trigger only their owner's personal workflows, and organization agents can trigger only organization workflows.
</Warning>

## Branch conditions

Visual workflow branches use the structured backend `BranchCondition` schema with `category`, `field`, `operator`, optional `value`, and optional `stepId` for `previous_step_output` category.

### Supported categories

<CodeGroup>
```json Contact Field
{
  "category": "contact_field",
  "field": "email",
  "operator": "exists"
}
```

```json Previous Step Output
{
  "category": "previous_step_output",
  "field": "response",
  "operator": "contains",
  "value": "yes",
  "stepId": "step_123"
}
```

```json Assignee Status
{
  "category": "assignee_status",
  "field": "status",
  "operator": "eq",
  "value": "online"
}
```

```json Lead Stage
{
  "category": "lead_stage",
  "field": "stageId",
  "operator": "eq",
  "value": "stage_456"
}
```
</CodeGroup>

### Operators

Available operators with type-aware validation:

- **Equality**: `eq`, `neq`
- **Comparison**: `gt`, `gte`, `lt`, `lte`
- **Text**: `contains`, `not_contains`
- **Arrays**: `in`, `has_any_of`, `has_all_of`, `has_none_of`
- **Range**: `between`
- **Existence**: `exists`, `not_exists`

<Tip>
`exists` and `not_exists` require no value; `in` requires a non-empty array; `between` requires a two-element bound.
</Tip>

### Legacy restrictions

<Warning>
Legacy raw field paths, wrapped variables, deprecated categories (`trigger`, `contact`, `lead`, `conversation`, `previous_step`, `webhook_payload`), and any serialized `source` property are rejected at publish time.
</Warning>

## Step execution

### Handler results

Step handlers can return:

- **`COMPLETED`**: Step finished successfully, continue to next step
- **`FAILED`**: Step failed, follow failure path if configured
- **`WAITING`**: Step is waiting (e.g., for user response), execution pauses
- **`SKIPPED`**: Step was skipped, continue through default connection

### Wait step behavior

The Wait step stores duration as `durationMs` with the following constraints:

<Check>
- Frontend controls may display seconds, minutes, hours, or days
- Saved workflow JSON must use milliseconds for queue worker compatibility
- Wait durations must be between 1 second and 7 days
</Check>

<Steps>
  <Step title="Schedule delay">
    Handler schedules a delayed resume job for the same wait node
  </Step>
  
  <Step title="Complete wait">
    Engine completes the waiting step after the delay
  </Step>
  
  <Step title="Continue execution">
    Continues to `connections.default` or completes as successful terminal path
  </Step>
</Steps>

### Execution locking

`runStep()` uses pessimistic locking on the `WorkflowExecution` row:

<Tabs>
  <Tab title="Standard Handlers">
    Most handlers execute inside the workflow transaction for atomic state and domain mutations.
  </Tab>
  
  <Tab title="Long-running Handlers">
    `SEND_MESSAGE`, `ASK_QUESTION`, and `TRIGGER_ANOTHER_WORKFLOW` are:
    1. Claimed in a short transaction
    2. Run outside the workflow row lock
    3. Re-lock execution to persist handler result
    
    This serializes duplicate queue jobs while preventing deadlocks.
  </Tab>
</Tabs>

## Graph validation

Publish-grade graph validation is server-side and uses `node.connections` as the executable source of truth:

### Requirements

<AccordionGroup>
  <Accordion title="Graph Structure">
    - Exactly one entry node required
    - Every connection target must reference an existing node
    - No unreachable nodes allowed
    - Regular graph cycles rejected
  </Accordion>
  
  <Accordion title="Intentional Loops">
    Only permitted through `JUMP_TO` with:
    - `config.targetStepId` specified
    - Bounded `maxJumps` to prevent infinite loops
  </Accordion>
  
  <Accordion title="Terminal Nodes">
    `END` nodes cannot have any outgoing connections
  </Accordion>
</AccordionGroup>

### Connection types

Branching steps must persist explicit output targets:

- **`BRANCH`**: Stores `connections.branches[].nextStepId`
- **`DATE_TIME`**: Stores `connections.inWindowBranch` and `connections.outOfWindowBranch`
- **`CHECK_TEAM_AVAILABILITY`**: Stores `connections.availableBranch` and `connections.unavailableBranch`
- **Advanced failure branches**: Store `connections.onFailure`
- **Ask Question branches**: Store `connections.onMessageFailure` and `connections.onTimeout`

<Note>
The `edges` array is retained for visual/editor parity and legacy checks, but runtime step progression follows connections and handler overrides.
</Note>
