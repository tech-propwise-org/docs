## Overview

The Workflow Engine provides rule-based automation for the messaging module. It evaluates configurable automation rules against messaging events (new messages, new conversations, AI escalations) and executes actions such as assigning agents, sending templates, creating tasks, and updating lead stages.

Rules are evaluated in **Phase B** of the webhook processor pipeline, isolated in a try-catch block so failures never affect message persistence or delivery.

## Core Concepts

### Visual Workflow Architecture

The current workflow builder persists visual automations in the `workflow` table. Each workflow has a `triggerType`, optional `triggerConfig`, and a JSONB `graph` containing ordered step definitions and their connections.

<Info>
The frontend builder uses an explicit-save model for workflow management with distinct Save and Publish operations.
</Info>

#### Save and Publish Operations

<Steps>
<Step title="Save Operation">
**Save** uses `PATCH /v1/workflows/:id` without lifecycle parameters to create or update a draft workflow.
</Step>

<Step title="Publish Operation">
- **New workflow**: `POST /v1/workflows` with `status: PUBLISHED` (atomic create + publish)
- **Existing workflow**: `PATCH /v1/workflows/:id` with editor payload plus `status: PUBLISHED` (merge, publish-grade validation, version bump when structural fields change)
</Step>

<Step title="Lifecycle Management">
- **Unpublish**: `PATCH` with `status: DRAFT`
- **Archive**: `PATCH` with `status: ARCHIVED`
</Step>
</Steps>

#### Workflow Creation Methods

<Tabs>
<Tab title="Start from Scratch">
Opens a local draft with an empty graph and no trigger selected. The user must choose a trigger before saving.
</Tab>

<Tab title="Template">
Opens a local draft from a system-level `WorkflowTemplate` record, copying its trigger type and graph into an editable local workflow. The database row is created only on explicit save.
</Tab>
</Tabs>

### Builder Action Gating

<Warning>
A non-empty **name is required** before any action. Save, Test, and Publish are all disabled while `name` is empty/whitespace-only.
</Warning>

Action requirements:
- **Test**: Always requires a selected trigger
- **Save**: Requires a trigger except for metadata-only edits on already-published workflows
- **Publish**: Requires zero `getWorkflowValidationIssues` (name, trigger, at least one step, and valid per-step config)

### Workflow Templates

Workflow templates are stored in the database and managed by `WorkflowTemplateSeederService`. Templates are:

- Upserted on every server boot via `AppInitializerService.onModuleInit()`
- Stored as global catalog records with no organization ownership
- Exposed through `GET /v1/workflows/templates`
- Must be ready-to-publish in any organization

<Note>
Templates may use generic steps but must not preselect organization-specific AI agents, users, teams, lead stages, or notification recipients.
</Note>

### Workflow Actor Resolution

The engine resolves the workflow actor once per execution and threads it through every step's `StepContext.actorUserId`:

- **ORGANIZATION scope**: Uses the per-org AI service user via shared `resolveAiServiceUserId` utility
- **PERSONAL scope**: Uses the workflow owner

<Check>
The resolved actor is used for both domain attribution (task creation, lead updates, conversation assignments) and audit row attribution.
</Check>

## Workflow Scoping

### Organization vs Personal Workflows

<CardGroup cols={2}>
<Card title="Organization Workflows" icon="building">
- Require `MESSAGING_MANAGE` permission to create
- Available to all users with messaging permissions
- One published workflow per `(organization_id, trigger_type)`
</Card>

<Card title="Personal Workflows" icon="user">
- Created by users without `MESSAGING_MANAGE` 
- Owned by the authenticated user
- One published workflow per `(organization_id, owner_user_id, trigger_type)`
</Card>
</CardGroup>

### Access Control

Service-level enforcement uses `AccessService.assertWorkflowAccessInTransaction`:

- **Organization workflows**: Require messaging view/manage for read and `MESSAGING_MANAGE` for MODIFY/DELETE
- **Personal workflows**: Fully accessible to the owner; non-owners need messaging permissions
- **AI-triggered workflows**: Personal agents can trigger only their owner's personal workflows; organization agents can trigger only organization workflows

<Warning>
Workflow-to-workflow chains are strict scope matches. Personal workflows cannot trigger organization workflows, even with `MESSAGING_MANAGE` permissions.
</Warning>

## Branch Conditions

Visual workflow branches use the structured backend `BranchCondition` schema with `category`, `field`, `operator`, optional `value`, and optional `stepId`.

### Supported Categories

<Tabs>
<Tab title="Contact & Lead">
- `contact_field`: Contact field values via `PersonService`
- `lead_stage`: Lead stage comparisons against organization's effective stages
</Tab>

<Tab title="Assignment & Messages">
- `assignee_status`: Current assignment status
- `last_incoming_message`: Latest inbound message fields
- `last_outgoing_message`: Latest outbound message fields
- `time_since_last_incoming_message`: Time elapsed since last inbound message
- `time_since_last_outgoing_message`: Time elapsed since last outbound message
</Tab>

<Tab title="Step Outputs">
- `previous_step_output`: Reads from `execution.variables.steps[stepId]`
</Tab>
</Tabs>

### Operators

Available operators with type-aware validation:

- **Equality**: `eq`, `neq`
- **Comparison**: `gt`, `gte`, `lt`, `lte`
- **Text**: `contains`, `not_contains`
- **Sets**: `in`, `has_any_of`, `has_all_of`, `has_none_of`
- **Range**: `between`
- **Existence**: `exists`, `not_exists`

<Info>
Branch evaluation preserves branch order with `AND` / `OR` semantics per branch. Exactly one else branch (empty `conditions`) is required per Branch step.
</Info>

## Step Types and Connections

### Connection Storage

Different step types store explicit output targets:

- **BRANCH**: `connections.branches[].nextStepId`
- **DATE_TIME**: `connections.inWindowBranch` and `connections.outOfWindowBranch`
- **CHECK_TEAM_AVAILABILITY**: `connections.availableBranch` and `connections.unavailableBranch`
- **Advanced failure branches**: `connections.onFailure`
- **Ask Question branches**: `connections.onMessageFailure` and `connections.onTimeout`

### Graph Validation

<Steps>
<Step title="Entry Point Validation">
Requires exactly one entry node in the workflow graph
</Step>

<Step title="Connection Validation">
Verifies every connection target references an existing node
</Step>

<Step title="Reachability Check">
Rejects unreachable nodes that cannot be reached from the entry point
</Step>

<Step title="Cycle Detection">
Rejects regular graph cycles but permits intentional loops through `JUMP_TO` steps with bounded `maxJumps`
</Step>

<Step title="End Node Validation">
`END` nodes cannot have any outgoing connections
</Step>
</Steps>

<Note>
Runtime step progression follows `connections` and handler overrides. The `edges` array is retained for visual/editor compatibility.
</Note>

### Wait Step Configuration

The Wait step stores duration as `durationMs` for precise scheduling:

- Frontend controls may display seconds, minutes, hours, or days
- Saved workflow JSON must use milliseconds for queue worker compatibility
- Duration must be between 1 second and 7 days
- Handler schedules a delayed resume job for the same wait node

<Tip>
Always convert display units to milliseconds before saving to ensure consistent queue scheduling behavior.
</Tip>

## Step Management

### Adding and Deleting Steps

- **Adding steps**: Wires the selected output connection to the new node
- **Deleting steps**: Graph-aware deletion that handles downstream connections

<Warning>
When deleting a step with downstream targets through `connections.default`, `onFailure`, `onTimeout`, or named branch outputs, the builder prompts for confirmation and removes the selected step plus the reachable downstream path while preserving unrelated sibling branches.
</Warning>

The deletion process also removes references to deleted nodes from named output connections and branch targets to prevent broken references in the saved graph.
