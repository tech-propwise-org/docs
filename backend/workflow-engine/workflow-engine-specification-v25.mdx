## Overview

The Workflow Engine provides rule-based automation for the messaging module. It evaluates configurable automation rules against messaging events (new messages, new conversations, AI escalations) and executes actions such as assigning agents, sending templates, creating tasks, and updating lead stages.

<Note>
Rules are evaluated in **Phase B** of the webhook processor pipeline, isolated in a try-catch block so failures never affect message persistence or delivery.
</Note>

## Core Concepts

### Visual Workflow

The current workflow builder persists visual automations in the `workflow` table. Each workflow has a `triggerType`, optional `triggerConfig`, and a JSONB `graph` containing ordered step definitions and their connections.

#### Frontend Builder Model

The frontend builder uses an explicit-save model with two primary actions:

<Tabs>
  <Tab title="Save">
    **`PATCH /v1/workflows/:id`** without lifecycle - Creates or updates a draft
  </Tab>
  <Tab title="Publish/Republish">
    - **New unsaved draft**: **`POST /v1/workflows`** with `status: PUBLISHED` (atomic create + publish)
    - **Existing row**: **`PATCH /v1/workflows/:id`** with editor payload plus `status: PUBLISHED` (merge, publish-grade validation, version bump when structural fields change)
  </Tab>
</Tabs>

<Steps>
  <Step title="Unpublish">
    **`PATCH`** with `status: DRAFT` (lifecycle-only rules apply)
  </Step>
  <Step title="Archive">
    **`PATCH`** with `status: ARCHIVED` (no structural fields on these transitions)
  </Step>
</Steps>

#### Workflow Creation Options

<CardGroup cols={2}>
  <Card title="Start from Scratch" icon="plus">
    Opens a local draft with an empty graph and no trigger selected. User must choose a trigger before saving.
  </Card>
  <Card title="Template" icon="template">
    Opens a local draft from a system-level `WorkflowTemplate` record, copying its trigger type and graph into an editable local workflow.
  </Card>
</CardGroup>

### Workflow Templates

Workflow templates are stored in the database, not hardcoded in the frontend. They are:

- Seeded by `WorkflowTemplateSeeder`
- Exposed through `GET /v1/workflows/templates`
- Global system catalog records with no organization ownership

<Warning>
Seeded templates must be ready-to-publish in any organization. They may use generic steps but must not preselect organization-specific AI agents, users, teams, lead stages, or notification recipients.
</Warning>

The frontend builder stores scratch workflows and template selections locally until **Save** or **Publish**:
- **Save**: Creates/updates a draft
- **Publish** on new local draft: Uses atomic create with `status: PUBLISHED`

#### Template Management

Templates are upserted by stable slug and reactivated if a previously soft-deleted slug is seeded again, ensuring catalog edits replace stale template graphs instead of leaving invalid versions in the browser.

## Workflow Scoping

Workflows can be organization-scoped or personal:

### Permission Model

<Tabs>
  <Tab title="Users with MESSAGING_MANAGE">
    - Can create organization workflows by default
    - May choose personal scope in the builder
    - Can modify/delete organization workflows
  </Tab>
  <Tab title="Users without MESSAGING_MANAGE">
    - Always create personal workflows owned by the authenticated user
    - Can only access their own personal workflows
  </Tab>
</Tabs>

### Publishing Constraints

<Note>
Publishing is unique by scope:
- One published organization workflow per `(organization_id, trigger_type)`
- One published personal workflow per `(organization_id, owner_user_id, trigger_type)`
</Note>

The database enforces this with partial unique indexes:
- `workflow_org_published_trigger_unique`
- `workflow_personal_published_trigger_unique`

### Access Control

Service-level enforcement uses `AccessService.assertWorkflowAccessInTransaction`:

- **Organization workflows**: Require messaging view/manage for read and `MESSAGING_MANAGE` for modify/delete
- **Personal workflows**: Fully accessible to the owner; non-owners need messaging permissions

<Warning>
AI-triggered child workflows are scope-checked:
- Personal agents can trigger only their owner's personal workflows
- Organization agents can trigger only organization workflows
</Warning>

## Branch Conditions

Visual workflow branches use the structured backend `BranchCondition` schema with:
- `category`
- `field` 
- `operator`
- Optional `value`
- Optional `stepId` (only when `category` is `previous_step_output`)

<Warning>
Legacy raw field paths, wrapped variables, and deprecated categories (`trigger`, `contact`, `lead`, `conversation`, `previous_step`, `webhook_payload`) are rejected at publish time.
</Warning>

### Supported Categories

<AccordionGroup>
  <Accordion title="contact_field">
    Values resolve through `PersonService` workflow branch helpers
  </Accordion>
  <Accordion title="previous_step_output">
    Reads from `execution.variables.steps[stepId]`. Publish validation requires the selected `stepId` to be the Branch node's single direct previous step.
  </Accordion>
  <Accordion title="assignee_status">
    Resolves through `WorkflowMessageStatusReaderService` with explicit organization filters
  </Accordion>
  <Accordion title="last_incoming_message / last_outgoing_message">
    Latest message reads order by `createdAt DESC, id DESC` and expose only Branch catalog fields
  </Accordion>
  <Accordion title="time_since_last_incoming_message / time_since_last_outgoing_message">
    Time-based conditions for message timing
  </Accordion>
  <Accordion title="lead_stage">
    `lead_stage.stageId` compares against organization's effective lead stages. Publish validation rejects unknown stage IDs.
  </Accordion>
</AccordionGroup>

### Operators

Supported operators include:

<CodeGroup>
```javascript Equality
eq        // equals
neq       // not equals
```

```javascript Comparison  
gt        // greater than
gte       // greater than or equal
lt        // less than
lte       // less than or equal
```

```javascript String
contains      // contains substring
not_contains  // does not contain substring
```

```javascript Array/Set
in           // value in array
between      // value between bounds
has_any_of   // has any of values
has_all_of   // has all of values  
has_none_of  // has none of values
```

```javascript Existence

not_exists   // field does not exist (no value required)
```
</CodeGroup>

<Info>
Validation and runtime comparison are type-aware. Each field has operator allowlists, and validation rejects incompatible values before publish.
</Info>

## Graph Structure and Validation

### Connection Storage

Branching steps must persist explicit output targets:

- **`BRANCH`**: Stores `connections.branches[].nextStepId`
- **`DATE_TIME`**: Stores `connections.inWindowBranch` and `connections.outOfWindowBranch`  
- **`CHECK_TEAM_AVAILABILITY`**: Stores `connections.availableBranch` and `connections.unavailableBranch`
- **Advanced failure branches**: Store `connections.onFailure`
- **Ask Question branches**: Store `connections.onMessageFailure` and `connections.onTimeout`

### Publish-Grade Validation

Server-side validation uses `node.connections` as the executable source of truth and requires:

<Check>Exactly one entry node</Check>
<Check>Every connection target references an existing node</Check>
<Check>No unreachable nodes</Check>
<Check>No regular graph cycles (only intentional loops through `JUMP_TO`)</Check>
<Check>`END` nodes cannot have outgoing connections</Check>

<Note>
The `edges` array is retained for visual/editor parity and legacy checks, but runtime step progression follows connections and handler overrides.
</Note>

### Wait Step Configuration

The Wait step stores duration as `durationMs`:

- Frontend controls may display seconds, minutes, hours, or days
- Saved workflow JSON must use milliseconds for queue worker scheduling
- Duration must be between 1 second and 7 days

<Steps>
  <Step title="Schedule Delay">
    Handler schedules a delayed resume job for the same wait node
  </Step>
  <Step title="Complete Wait">
    Engine completes the waiting step and continues to `connections.default`
  </Step>
  <Step title="Terminal Path">
    If no default target exists, wait completes as terminal successful path after delay
  </Step>
</Steps>

## Step Execution

### Handler Return Values

Step handlers can return:

- **`COMPLETED`**: Step finished successfully
- **`FAILED`**: Step failed
- **`WAITING`**: Step is waiting (e.g., for user response)
- **`SKIPPED`**: Step was skipped

<Info>
`SKIPPED` records the step as skipped and continues through the step's default connection when one exists; otherwise execution completes successfully.
</Info>

### Execution Locking

`runStep()` takes a pessimistic lock on the `WorkflowExecution` row while creating or reusing a `WorkflowStepExecution` attempt:

<Tabs>
  <Tab title="Standard Handlers">
    Execute inside the transaction so workflow state and domain mutations remain atomic
  </Tab>
  <Tab title="Long/Independent Handlers">
    `SEND_MESSAGE`, `ASK_QUESTION`, and `TRIGGER_ANOTHER_WORKFLOW` are:
    1. Claimed in a short transaction
    2. Run outside the workflow row lock
    3. Re-lock execution to persist handler result
  </Tab>
</Tabs>

### Retry Mechanism

Repeated executions of the same graph node use incrementing `WorkflowStepExecution.attempt` values:

- Delayed retries preserve each attempt without violating `(execution, stepId, attempt)` uniqueness
- When delayed retry resumes a still-waiting step, existing `WAITING` step execution row is reused
- `WorkflowExecution.stepIterations` still increments for `MAX_STEP_ITERATIONS` safety bound

<Warning>
Stale jobs or stale handler results whose `stepId` no longer matches `currentStepId` are ignored.
</Warning>

## Message Types

The Send Message step supports `messageType`. New visual builder steps start as `TEXT` with empty `content`:

- **Text messages**: Require non-empty `content` and are limited to 2,000 characters
- Additional message types supported through the builder interface

<Tip>
The builder enforces content validation before allowing workflow publication.
</Tip>
