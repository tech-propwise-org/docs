## Overview

The Workflow Engine provides rule-based automation for the messaging module. It evaluates configurable automation rules against messaging events (new messages, new conversations, AI escalations) and executes actions such as assigning agents, sending templates, creating tasks, and updating lead stages.

<Info>
Rules are evaluated in **Phase B** of the webhook processor pipeline, isolated in a try-catch block so failures never affect message persistence or delivery.
</Info>

## Core Concepts

### Visual Workflow

The current workflow builder persists visual automations in the `workflow` table. Each workflow has a `triggerType`, optional `triggerConfig`, and a JSONB `graph` containing ordered step definitions and their connections.

<Note>
The frontend builder uses an explicit-save model for **Save** and **Publish** operations with different behaviors based on workflow state.
</Note>

#### Save and Publish Operations

<Tabs>
<Tab title="Save Operation">
**`PATCH /v1/workflows/:id`** without lifecycle - updates existing draft workflows
</Tab>
<Tab title="Publish Operation">
- **New unsaved draft**: **`POST /v1/workflows`** with `status: PUBLISHED` (atomic create + publish)
- **Existing row**: **`PATCH /v1/workflows/:id`** with editor payload plus `status: PUBLISHED` (merge, publish-grade validation, version bump when structural fields change)
</Tab>
<Tab title="Unpublish/Archive">
**`PATCH`** with `status: DRAFT` or `status: ARCHIVED` (lifecycle-only rules apply)
</Tab>
</Tabs>

#### Workflow Creation Methods

<Steps>
<Step title="Start from Scratch">
Opens a local draft with an empty graph and no trigger selected. The user must choose a trigger before saving.
</Step>
<Step title="From Template">
Opens a local draft from a system-level `WorkflowTemplate` record, copying its trigger type and graph into an editable local workflow. The database row is created only on explicit save.
</Step>
</Steps>

### Workflow Templates

<CardGroup cols={2}>
<Card title="Template Storage" icon="database">
Templates are stored in the database, not hardcoded in the frontend
</Card>
<Card title="Global Access" icon="globe">
System catalog records with no organization ownership
</Card>
</CardGroup>

Workflow templates are:
- Seeded by `WorkflowTemplateSeeder`
- Exposed through `GET /v1/workflows/templates`
- Must be ready-to-publish in any organization
- Cannot preselect organization-specific resources (AI agents, users, teams, lead stages, notification recipients)

<Warning>
Templates are upserted by stable slug and reactivated if a previously soft-deleted slug is seeded again, so catalog edits replace stale template graphs.
</Warning>

## Workflow Scope and Access Control

### Organization vs Personal Workflows

<Tabs>
<Tab title="Organization Workflows">
- Created by users with `MESSAGING_MANAGE` permission
- Visible to all users with messaging permissions
- One published workflow per `(organization_id, trigger_type)`
</Tab>
<Tab title="Personal Workflows">
- Created by individual users (with or without `MESSAGING_MANAGE`)
- Owned by the authenticated user
- One published workflow per `(organization_id, owner_user_id, trigger_type)`
</Tab>
</Tabs>

### Access Control Rules

<Steps>
<Step title="Create Permissions">
- Users with `MESSAGING_MANAGE` can create organization workflows by default and may choose personal scope
- Users without `MESSAGING_MANAGE` always create personal workflows
</Step>
<Step title="List Permissions">
- Users with `MESSAGING_VIEW` or `MESSAGING_MANAGE` list both organization and personal workflows
- Users without those permissions receive only their own personal workflows
- `scope=ORGANIZATION` returns no rows for users without messaging permissions
</Step>
<Step title="Modify Permissions">
- Organization workflows require `MESSAGING_MANAGE` for MODIFY/DELETE
- Personal workflows are fully accessible to the owner
- Non-owners need messaging permissions for personal workflow access
</Step>
</Tabs>

<Info>
Service-level enforcement uses `AccessService.assertWorkflowAccessInTransaction` for permission validation.
</Info>

### AI-Triggered Workflows

<Note>
- Personal agents can trigger only their owner's personal workflows
- Organization agents can trigger only organization workflows
- Workflow chains must maintain strict scope matches
</Note>

## Branch Conditions and Logic

### Branch Condition Schema

Visual workflow branches use the structured backend `BranchCondition` schema:

```json
{
  "category": "contact_field",
  "field": "email",
  "operator": "exists", 
  "value": null,
  "stepId": null
}
```

<Warning>
Legacy raw field paths, wrapped variables, and deprecated categories are rejected at publish time. The backend branch catalog constants are the final authority.
</Warning>

### Supported Categories

<AccordionGroup>
<Accordion title="contact_field">
Values resolve through `PersonService` workflow branch helpers
</Accordion>
<Accordion title="previous_step_output">
Reads from `execution.variables.steps[stepId]` - requires selected `stepId` to be the Branch node's direct previous step
</Accordion>
<Accordion title="assignee_status">
Resolves through `WorkflowMessageStatusReaderService` with organization filters
</Accordion>
<Accordion title="Message Fields">
- `last_incoming_message`
- `last_outgoing_message` 
- `time_since_last_incoming_message`
- `time_since_last_outgoing_message`

Latest message reads order by `createdAt DESC, id DESC` and expose only Branch catalog fields
</Accordion>
<Accordion title="lead_stage">
`lead_stage.stageId` compares against organization's effective lead stages (including global stages without org overrides)
</Accordion>
</AccordionGroup>

### Operators

<Tabs>
<Tab title="Comparison">
- `eq` - equals
- `neq` - not equals  
- `gt` - greater than
- `gte` - greater than or equal
- `lt` - less than
- `lte` - less than or equal
</Tab>
<Tab title="String">
- `contains` - string contains
- `not_contains` - string does not contain
</Tab>
<Tab title="Set">
- `in` - value in array
- `has_any_of` - has any of values
- `has_all_of` - has all values
- `has_none_of` - has none of values
</Tab>
<Tab title="Existence">
- `exists` - field has value
- `not_exists` - field is null/empty
- `between` - value between bounds
</Tab>
</Tabs>

<Check>
Validation and runtime comparison are type-aware with per-field operator allowlists.
</Check>

## Step Execution and Flow Control

### Step Handler Results

Step handlers can return:

<CardGroup cols={2}>
<Card title="COMPLETED" icon="check">
Step executed successfully, continue to next step
</Card>
<Card title="FAILED" icon="x">
Step failed, follow failure path if configured
</Card>
<Card title="WAITING" icon="clock">
Step is waiting for external input or delay
</Card>
<Card title="SKIPPED" icon="forward">
Step was skipped, continue through default connection
</Card>
</CardGroup>

### Execution Locks and Transactions

<Steps>
<Step title="Pessimistic Locking">
`runStep()` takes a pessimistic lock on the `WorkflowExecution` row while creating or reusing a `WorkflowStepExecution` attempt
</Step>
<Step title="Atomic Handlers">
Most handlers execute inside the transaction so workflow state and domain mutations remain atomic
</Step>
<Step title="Long Handlers">
Long or independently idempotent handlers (`SEND_MESSAGE`, `ASK_QUESTION`, `TRIGGER_ANOTHER_WORKFLOW`) are:
- Claimed in a short transaction
- Run outside the workflow row lock  
- Re-lock execution to persist the handler result
</Step>
</Tabs>

### Step Attempts and Retries

<Note>
Repeated executions of the same graph node use incrementing `WorkflowStepExecution.attempt` values so delayed retries can preserve each attempt without violating the `(execution, stepId, attempt)` uniqueness constraint.
</Note>

When delayed retries resume:
- Still-waiting steps reuse the existing `WAITING` step execution row
- `WorkflowExecution.stepIterations` still increments 
- `MAX_STEP_ITERATIONS` remains an effective safety bound

### Graph Validation

Publish-grade graph validation is server-side and requires:

<Steps>
<Step title="Entry Point">
Exactly one entry node must exist
</Step>
<Step title="Valid Connections">
Every connection target must reference an existing node
</Step>
<Step title="Reachability">
No unreachable nodes are allowed
</Step>
<Step title="Cycle Prevention">
Regular graph cycles are rejected; only intentional loops through `JUMP_TO` are permitted
</Step>
<Step title="Terminal Nodes">
`END` nodes cannot have any outgoing connections
</Step>
</Steps>

<Warning>
The `edges` array is retained for visual/editor parity, but runtime step progression follows `node.connections` as the executable source of truth.
</Warning>

## Specialized Step Types

### Wait Step

The Wait step stores duration as `durationMs`:

```json
{
  "type": "WAIT",
  "config": {
    "durationMs": 300000
  }
}
```

<Tip>
Frontend controls may display seconds, minutes, hours, or days, but saved workflow JSON must use milliseconds for queue worker compatibility.
</Tip>

**Wait Duration Constraints:**
- Minimum: 1 second (1,000ms)
- Maximum: 7 days (604,800,000ms)

**Wait Step Execution:**
1. Handler schedules a delayed resume job for the same wait node
2. Engine completes the waiting step after delay
3. Continues to `connections.default` if target exists
4. Otherwise completes as terminal successful path

### Branch Step Output Targets

Different step types store explicit output targets:

<AccordionGroup>
<Accordion title="BRANCH">
Stores `connections.branches[].nextStepId` for each conditional branch
</Accordion>
<Accordion title="DATE_TIME">
Stores `connections.inWindowBranch` and `connections.outOfWindowBranch`
</Accordion>
<Accordion title="CHECK_TEAM_AVAILABILITY">
Stores `connections.availableBranch` and `connections.unavailableBranch`
</Accordion>
<Accordion title="Advanced Failure Branches">
Stores `connections.onFailure` for error handling
</Accordion>
<Accordion title="Ask Question Branches">
- `connections.onMessageFailure` for send failures
- `connections.onTimeout` for response timeouts
</Accordion>
</AccordionGroup>

The Send Message step supports `messageType` configuration:

<Tabs>
<Tab title="TEXT Messages">
