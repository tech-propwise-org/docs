## Background

When the `dev` branch cherry-picked PCD-64's shared knowledge-base infrastructure (`KnowledgeBase`, `KnowledgeChunk`, `OrganizationLlmKey`, `EmbeddingService`, `KnowledgeBaseService`, `AiAgentKnowledgeBaseController`, BYOK provisioning), it could not yet adopt PCD-64's `AiAgentKnowledgeBaseService` because the full AI Agent runtime was out of scope for that PR.

To keep the internal AI assistant's `searchKnowledge` tool working in the interim, dev shipped a thin, dev-only glue layer:

<AccordionGroup>
  <Accordion title="Glue Layer Components">
    - **`src/modules/ai/services/ai-knowledge.service.ts`** — Slim search-only service that wraps `EmbeddingService.generateEmbedding(...)` + raw pgvector SQL against `knowledge_chunks JOIN knowledge_base`
    - **`src/modules/ai/ai-knowledge-tools.provider.ts`** — Wraps the above into the `searchKnowledge` AI tool consumed by `ai-assistant/`
    - **Three lines in `src/modules/ai/ai.module.ts`** — `providers` + `exports` wiring the glue + the `provideAiTool(...) as never` registration
  </Accordion>
</AccordionGroup>

On the `ai-assistant` branch, the PCD-64 AI Agent runtime is **not** present — only the cherry-picked KB infrastructure is. `AiAgentKnowledgeBaseService` (the canonical search service) does **not** exist on this branch yet; it arrives as a clean file-add when `PCD-64-ai-agents-and-workflows` merges.

<Info>
Both the glue service and the canonical service call the same `EmbeddingService.generateEmbedding(query, apiKey)` and `OrganizationLlmKeyService.getActiveKey(orgId)` against the same `knowledge_chunks` table.
</Info>

### Interim Alignment (Completed)

<Check>
**Completed:** 2026-05-30 on `ai-assistant` branch
</Check>

To keep the eventual deletion + PCD-64 merge low-diff, `AIKnowledgeService` was refactored **in place** to mirror the canonical `AiAgentKnowledgeBaseService.searchAgentKnowledge` structure:

- `tenantContext.executeInOrg` + `em.createQueryBuilder(KnowledgeChunk)`
- `raw('(kc.embedding <=> ?::vector)')` distance calculation
- Returns flat `KnowledgeChunkHit[]` with fields: `chunkId`, `content`, `sourceId`, `sourceTitle`, `sourceType`, `distance`
- Adopted canonical `status = READY` filter
- Implemented owner-scoping (`owner_user_id IS NULL OR = :requestingUserId`)
- Skips `aiUsed` write (assistant is `accessMode: 'read'`)

<Tip>
Class name (`AIKnowledgeService`), method name (`searchKnowledge`), and `ai.module.ts` wiring were intentionally left unchanged so this branch's edits stay conflict-free against PCD-64's `ai.module.ts` rewrite.
</Tip>

## Goal

Migrate `AiKnowledgeToolsProvider` to call `AiAgentKnowledgeBaseService` directly, then delete `AIKnowledgeService` and remove the temporary registration lines from `ai.module.ts`.

<Warning>
The `searchKnowledge` tool's external contract (name, schema, accessMode, contextScopes, audit category, untrusted-content treatment) must remain identical so no eval suite or AI assistant test breaks.
</Warning>

## Why This Matters

<Warning>
**Technical Debt Impact**

Two services doing the same thing is a code smell, especially because the glue duplicates the raw pgvector SQL inline and would drift from the canonical service over time.
</Warning>

### Benefits

1. **Eliminates Duplication** — The glue service unnecessarily duplicates logic from `AiAgentKnowledgeBaseService`
2. **Reduces Coupling** — `ai-knowledge-tools.provider.ts` currently ties the AI assistant's lifetime to a dev-period workaround
3. **Simplifies Module** — Shrinks `ai.module.ts` providers list by 1 service and cleans up the `exports` array
4. **Prevents Drift** — Prevents the two implementations from diverging over time

- **Ingestion Flows** — `KnowledgeBaseService` ingestion flows remain unchanged
