## Background

When the `dev` branch cherry-picked PCD-64's shared knowledge-base infrastructure (`KnowledgeBase`, `KnowledgeChunk`, `OrganizationLlmKey`, `EmbeddingService`, `KnowledgeBaseService`, `AiAgentKnowledgeBaseController`, BYOK provisioning), it could not yet adopt PCD-64's `AiAgentKnowledgeBaseService` because the full AI Agent runtime was out of scope for that PR.

To keep the internal AI assistant's `searchKnowledge` tool working in the interim, dev shipped a thin, dev-only glue layer:

- `src/modules/ai/services/ai-knowledge.service.ts` — Slim search-only service that wraps `EmbeddingService.generateEmbedding(...)` + raw pgvector SQL against `knowledge_chunks JOIN knowledge_base`
- `src/modules/ai/ai-knowledge-tools.provider.ts` — Wraps the above into the `searchKnowledge` AI tool consumed by `ai-assistant/`
- Three lines at the bottom of `src/modules/ai/ai.module.ts` (`providers` + `exports`) wiring the glue + the `provideAiTool(...) as never` registration

<Info>
On the `ai-assistant` branch, the PCD-64 AI Agent runtime is **not** present — only the cherry-picked KB infrastructure exists. `AiAgentKnowledgeBaseService` (the canonical search service) does **not** exist on this branch yet; it arrives as a clean file-add when `PCD-64-ai-agents-and-workflows` merges.
</Info>

### Interim Alignment (Completed)

<Check>
**Completed:** 2026-05-30, on `ai-assistant` branch
</Check>

To keep the eventual deletion + PCD-64 merge low-diff, `AIKnowledgeService` was refactored **in place** to mirror the canonical `AiAgentKnowledgeBaseService.searchAgentKnowledge` structure:

- `tenantContext.executeInOrg` + `em.createQueryBuilder(KnowledgeChunk)`
- `raw('(kc.embedding <=> ?::vector)')` distance calculation
- Returns flat `KnowledgeChunkHit[]` (`chunkId`, `content`, `sourceId`, `sourceTitle`, `sourceType`, `distance`)
- Adopted canonical `status = READY` filter
- Added owner-scoping (`owner_user_id IS NULL OR = :requestingUserId`)
- Skips `aiUsed` write (assistant is `accessMode: 'read'`)

<Note>
Class name (`AIKnowledgeService`), method name (`searchKnowledge`), and `ai.module.ts` wiring were intentionally left unchanged so this branch's edits stay conflict-free against PCD-64's `ai.module.ts` rewrite.

**Still pending:** The deletion below, once PCD-64 actually merges.
</Note>

## Goal

Migrate `AiKnowledgeToolsProvider` to call `AiAgentKnowledgeBaseService` directly, then delete `AIKnowledgeService` and remove the temporary registration lines from `ai.module.ts`.

The `searchKnowledge` tool's external contract must remain identical:
- Tool name
- Input schema
- Output projection
- `accessMode`
- `contextScopes`
- Audit category
- Untrusted-content treatment

No eval suite or AI assistant test should break.

## Why This Is Worth Doing

<Tip>
**Primary Benefits:**

1. **Eliminates Duplication** — Two services doing the same thing is a code smell, especially because the glue duplicates raw pgvector SQL inline and would drift from the canonical service over time

2. **Removes Technical Debt** — `ai-knowledge-tools.provider.ts` exporting `AIKnowledgeService` ties the AI assistant's lifetime to a dev-period workaround that has outlived its purpose

3. **Cleaner Module Structure** — Deletion shrinks `ai.module.ts` providers list by 1 service and cleans up the `exports` array

4. **Maintainability** — Single source of truth for knowledge base search logic
</Tip>

## Out of Scope

<Note>
The following items are **NOT** part of this cleanup:

- **`searchKnowledge` tool name** — Stays unchanged. PCD-64 runtime exposes knowledge search to its agents through its own tool-registry layer (`AiAgentToolRegistryService`); the assistant's tool registry (`AiToolRegistryService`) is independent. Both can coexist long-term.

- **HTTP API surface** — The `/v1/ai-knowledge-bases` endpoints remain unchanged

- **Entity models** — `KnowledgeBase`, `KnowledgeChunk`, `OrganizationLlmKey` entities are not modified

- **Core services** — `KnowledgeBaseService` ingestion flows and BYOK provisioning are unchanged
</Note>
