## Background

When the `dev` branch cherry-picked PCD-64's shared knowledge-base infrastructure (`KnowledgeBase`, `KnowledgeChunk`, `OrganizationLlmKey`, `EmbeddingService`, `KnowledgeBaseService`, `AiAgentKnowledgeBaseController`, BYOK provisioning), it could not yet adopt PCD-64's `AiAgentKnowledgeBaseService` because the full AI Agent runtime was out of scope for that PR.

To keep the internal AI assistant's `searchKnowledge` tool working in the interim, dev shipped a thin, dev-only glue layer:

<AccordionGroup>
  <Accordion title="Temporary glue components">
    - **`src/modules/ai/services/ai-knowledge.service.ts`** — Slim search-only service that wraps `EmbeddingService.generateEmbedding(...)` + raw pgvector SQL against `knowledge_chunks JOIN knowledge_base`
    - **`src/modules/ai/ai-knowledge-tools.provider.ts`** — Wraps the above into the `searchKnowledge` AI tool consumed by `ai-assistant/`
    - **Three lines in `src/modules/ai/ai.module.ts`** — `providers` + `exports` wiring the glue + the `provideAiTool(...) as never` registration
  </Accordion>
</AccordionGroup>

On the `ai-assistant` branch, the PCD-64 AI Agent runtime is **not** present — only the cherry-picked KB infrastructure exists. `AiAgentKnowledgeBaseService` (the canonical search service) does **not** exist on this branch yet; it arrives as a clean file-add when `PCD-64-ai-agents-and-workflows` merges.

### Interim alignment (completed)

<Check>
**Completed:** 2025-05-30 on `ai-assistant` branch
</Check>

To keep the eventual deletion + PCD-64 merge low-diff, `AIKnowledgeService` was refactored **in place** to mirror the canonical `AiAgentKnowledgeBaseService.searchAgentKnowledge` structure:

- Uses `tenantContext.executeInOrg` + `em.createQueryBuilder(KnowledgeChunk)`
- Implements `raw('(kc.embedding <=> ?::vector)')` distance calculation
- Returns flat `KnowledgeChunkHit[]` (`chunkId`, `content`, `sourceId`, `sourceTitle`, `sourceType`, `distance`)
- Adopted canonical `status = READY` filter and owner-scoping (`owner_user_id IS NULL OR = :requestingUserId`)
- Skips `aiUsed` write (assistant is `accessMode: 'read'`)

<Info>
Class name (`AIKnowledgeService`), method name (`searchKnowledge`), and `ai.module.ts` wiring were intentionally left unchanged so this branch's edits stay conflict-free against PCD-64's `ai.module.ts` rewrite.
</Info>

## Goal

<Note>

</Note>

## Why this is safe

<CardGroup cols={2}>
  <Card title="Contract preservation" icon="shield-check">
    The tool's external contract (name, input schema, output projection fields `chunkId` / `sourceId` / `sourceTitle` / `sourceType` / `snippet`) is preserved.
  </Card>
  
  <Card title="Identical RLS posture" icon="lock">
    `AiAgentKnowledgeBaseService.searchAgentKnowledge(...)` already runs inside `tenantContext.executeInOrg(orgId, ...)` — same RLS posture as the glue, which (post-alignment) uses the identical `executeInOrg` + `createQueryBuilder` + `raw()` pgvector pattern.
  </Card>
  
  <Card title="Unchanged vector space" icon="cube">
    The vector space (OpenAI `text-embedding-3-small`, 768 dims) and the pgvector cosine distance operator (`<=>`) are unchanged.
  </Card>
  
  <Card title="Consistent BYOK enforcement" icon="key">
    BYOK enforcement (`BadRequestException` when no `OrganizationLlmKey` row exists) is consistent between the two services because both ultimately resolve through `OrganizationLlmKeyService.getActiveKey`.
  </Card>
</CardGroup>

## Why this is worth doing

<Tip>
This cleanup addresses several technical debt concerns:
</Tip>

1. **Eliminates duplication** — Two services doing the same thing is a code smell, especially because the glue duplicates the raw pgvector SQL inline and would drift from the canonical service over time

2. **Removes coupling to workaround** — `ai-knowledge-tools.provider.ts` exporting `AIKnowledgeService` ties the AI assistant's lifetime to a dev-period workaround that has now outlived its purpose

3. **Simplifies module structure** — The eventual deletion shrinks `ai.module.ts` providers list by 1 service + cleans up the `exports` array
