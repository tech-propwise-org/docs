## Background

When `dev` cherry-picked PCD-64's shared knowledge-base infrastructure (`KnowledgeBase`, `KnowledgeChunk`, `OrganizationLlmKey`, `EmbeddingService`, `KnowledgeBaseService`, `AiAgentKnowledgeBaseController`, BYOK provisioning), it could not yet adopt PCD-64's `AiAgentKnowledgeBaseService` because the full AI Agent runtime was out of scope for that PR.

To keep the internal AI assistant's `searchKnowledge` tool working in the interim, dev shipped a thin, dev-only glue layer:

- `src/modules/ai/services/ai-knowledge.service.ts` ŌĆö slim search-only service that wraps `EmbeddingService.generateEmbedding(...)` + raw pgvector SQL against `knowledge_chunks JOIN knowledge_base`
- `src/modules/ai/ai-knowledge-tools.provider.ts` ŌĆö wraps the above into the `searchKnowledge` AI tool consumed by `ai-assistant/`
- Three lines at the bottom of `src/modules/ai/ai.module.ts` (`providers` + `exports`) wiring the glue + the `provideAiTool(...) as never` registration

On the `ai-assistant` branch, the PCD-64 AI Agent runtime is **not** present ŌĆö only the cherry-picked KB infrastructure is. `AiAgentKnowledgeBaseService` (the canonical search service) does **not** exist on this branch yet; it arrives as a clean file-add when `PCD-64-ai-agents-and-workflows` merges.

### Interim alignment (completed)

<Check>
**Completed on 2026-05-30 (`ai-assistant` branch)**

To keep the eventual deletion + PCD-64 merge low-diff, `AIKnowledgeService` was refactored **in place** to mirror the canonical `AiAgentKnowledgeBaseService.searchAgentKnowledge` structure.
</Check>

The service now uses:
- `tenantContext.executeInOrg` + `em.createQueryBuilder(KnowledgeChunk)`
- `raw('(kc.embedding <=> ?::vector)')` distance
- Returns flat `KnowledgeChunkHit[]` (`chunkId`, `content`, `sourceId`, `sourceTitle`, `sourceType`, `distance`)
- Adopted canonical `status = READY` filter and owner-scoping (`owner_user_id IS NULL OR = :requestingUserId`)
- Skips the `aiUsed` write (assistant is `accessMode: 'read'`)

Class name (`AIKnowledgeService`), method name (`searchKnowledge`), and `ai.module.ts` wiring were intentionally left unchanged so this branch's edits stay conflict-free against PCD-64's `ai.module.ts` rewrite.

## Goal

Migrate `AiKnowledgeToolsProvider` to call `AiAgentKnowledgeBaseService` directly, then delete `AIKnowledgeService` and remove the temporary registration lines from `ai.module.ts`. 

<Warning>
The `searchKnowledge` tool's external contract (name, schema, accessMode, contextScopes, audit category, untrusted-content treatment) must remain identical so no eval suite or AI assistant test breaks.
</Warning>

## Why this is worth doing

<Info>
**Technical debt reduction**
- Two services doing the same thing is a smell, especially because the glue duplicates the raw pgvector SQL inline and would drift from the canonical service over time
- `ai-knowledge-tools.provider.ts` exporting `AIKnowledgeService` ties the AI assistant's lifetime to a dev-period workaround that has now outlived its purpose
- The eventual deletion shrinks `ai.module.ts` providers list by 1 service + cleans up the `exports` array
</Info>

## Out of scope

The following components remain unchanged:

- The `searchKnowledge` tool name itself stays. PCD-64 runtime exposes knowledge search to its agents through its own tool-registry layer (`AiAgentToolRegistryService`); the assistant's tool registry (`AiToolRegistryService`) is independent. Both can coexist long-term.
- The `/v1/ai-knowledge-bases` HTTP surface, the `KnowledgeBase` / `KnowledgeChunk` / `OrganizationLlmKey` entities, the `KnowledgeBaseService` ingestion flows, the BYOK provisioning ŌĆö none of these change.
