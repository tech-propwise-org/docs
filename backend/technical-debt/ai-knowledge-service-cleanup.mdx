## Background

When `dev` cherry-picked PCD-64's shared knowledge-base infrastructure (`KnowledgeBase`, `KnowledgeChunk`, `OrganizationLlmKey`, `EmbeddingService`, `KnowledgeBaseService`, `AiAgentKnowledgeBaseController`, BYOK provisioning), it could not yet adopt PCD-64's `AiAgentKnowledgeBaseService` because the full AI Agent runtime was out of scope for that PR.

To keep the internal AI assistant's `searchKnowledge` tool working in the interim, dev shipped a thin, dev-only glue layer:

- `src/modules/ai/services/ai-knowledge.service.ts` — slim search-only service that wraps `EmbeddingService.generateEmbedding(...)` + raw pgvector SQL against `knowledge_chunks JOIN knowledge_base`
- `src/modules/ai/ai-knowledge-tools.provider.ts` — wraps the above into the `searchKnowledge` AI tool consumed by `ai-assistant/`
- Three lines at the bottom of `src/modules/ai/ai.module.ts` (`providers` + `exports`) wiring the glue + the `provideAiTool(...) as never` registration

### Interim Alignment (Completed)

<Check>
Completed on 2026-05-30 on the `ai-assistant` branch
</Check>

To keep the eventual deletion + PCD-64 merge low-diff, `AIKnowledgeService` was refactored **in place** to mirror the canonical `AiAgentKnowledgeBaseService.searchAgentKnowledge` structure:

- Uses `tenantContext.executeInOrg` + `em.createQueryBuilder(KnowledgeChunk)`
- Implements `raw('(kc.embedding <=> ?::vector)')` distance calculation
- Returns flat `KnowledgeChunkHit[]` structure
- Adopts canonical `status = READY` filter and owner-scoping
- Skips the `aiUsed` write (assistant is `accessMode: 'read'`)

<Info>
Class name (`AIKnowledgeService`), method name (`searchKnowledge`), and `ai.module.ts` wiring were intentionally left unchanged to stay conflict-free against PCD-64's `ai.module.ts` rewrite.
</Info>

## Goal

Migrate `AiKnowledgeToolsProvider` to call `AiAgentKnowledgeBaseService` directly, then delete `AIKnowledgeService` and remove the temporary registration lines from `ai.module.ts`. 

<Warning>
The `searchKnowledge` tool's external contract (name, schema, accessMode, contextScopes, audit category, untrusted-content treatment) must remain identical so no eval suite or AI assistant test breaks.
</Warning>

## Implementation Steps

<Steps>
<Step title="Delete AIKnowledgeService">
Remove `src/modules/ai/services/ai-knowledge.service.ts` — the slim search-only glue service. No external module references it after the migration.
</Step>

<Step title="Update AiKnowledgeToolsProvider">
Modify `src/modules/ai/ai-knowledge-tools.provider.ts`:

- Change constructor to inject `AiAgentKnowledgeBaseService` instead of `AIKnowledgeService`
- Call `searchAgentKnowledge(query, organizationId, agent, topK, apiKey, skipAiUsedUpdate)` instead of `searchKnowledge`

<Note>
The canonical method requires an `AiAgent` and explicit `apiKey`, and performs an `aiUsed` write unless `skipAiUsedUpdate=true`. The agent-less assistant must obtain/construct a suitable `AiAgent`, resolve the BYOK key, and pass `skipAiUsedUpdate=true`.
</Note>
</Step>

<Step title="Update AI Module Registration">
Modify `src/modules/ai/ai.module.ts`:

- Remove the `AIKnowledgeService` import
- Remove `AIKnowledgeService` from `providers`
- Remove `AIKnowledgeService` from `exports`
- Keep `AiKnowledgeToolsProvider` and `provideAiTool(AiKnowledgeToolsProvider) as never`
- Keep `AI_TOOL_PROVIDER_TOKEN` in `exports`
- Update module-level JSDoc to remove "dev-only glue" paragraph
</Step>

<Step title="Update Documentation">
Update `Docs/AI_MODULE_SPECIFICATION.md`:

- Remove the `AIKnowledgeService` section under "## Services"
- Update any reference to "dev-only glue" in the overview
- Add changelog note about assistant now consuming `AiAgentKnowledgeBaseService` directly
</Step>
</Steps>

## Why This is Safe

<AccordionGroup>
<Accordion title="Contract Preservation">
The tool's external contract (name, input schema, output projection fields `chunkId` / `sourceId` / `sourceTitle` / `sourceType` / `snippet`) is preserved.
</Accordion>

<Accordion title="Security Posture">
`AiAgentKnowledgeBaseService.searchAgentKnowledge(...)` already runs inside `tenantContext.executeInOrg(orgId, ...)` — same RLS posture as the glue, which uses identical `executeInOrg` + `createQueryBuilder` + `raw()` pgvector pattern.
</Accordion>

<Accordion title="Vector Space Consistency">
The vector space (OpenAI `text-embedding-3-small`, 768 dims) and the pgvector cosine distance operator (`<=>`) are unchanged.
</Accordion>

<Accordion title="BYOK Enforcement">
BYOK enforcement (`BadRequestException` when no `OrganizationLlmKey` row exists) is consistent between the two services because both ultimately resolve through `OrganizationLlmKeyService.getActiveKey`.
</Accordion>
</AccordionGroup>

## Out of Scope

<Tip>
The following items are intentionally **not** changed in this cleanup:

- The `searchKnowledge` tool name itself stays
- The `/v1/ai-knowledge-bases` HTTP surface
- The `KnowledgeBase` / `KnowledgeChunk` / `OrganizationLlmKey` entities
- The `KnowledgeBaseService` ingestion flows
- The BYOK provisioning system

PCD-64 runtime exposes knowledge search to its agents through its own tool-registry layer (`AiAgentToolRegistryService`); the assistant's tool registry (`AiToolRegistryService`) is independent. Both can coexist long-term.
</Tip>
