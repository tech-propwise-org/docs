## Background

When `dev` cherry-picked PCD-64's shared knowledge-base infrastructure (`KnowledgeBase`, `KnowledgeChunk`, `OrganizationLlmKey`, `EmbeddingService`, `KnowledgeBaseService`, `AiAgentKnowledgeBaseController`, BYOK provisioning), it could not yet adopt PCD-64's `AiAgentKnowledgeBaseService` because the full AI Agent runtime was out of scope for that PR.

To keep the internal AI assistant's `searchKnowledge` tool working in the interim, dev shipped a thin, dev-only glue layer:

<CardGroup cols={2}>
  <Card title="AIKnowledgeService" icon="layer-group">
    Slim search-only service wrapping `EmbeddingService.generateEmbedding(...)` + raw pgvector SQL against `knowledge_chunks JOIN knowledge_base`
  </Card>
  <Card title="AiKnowledgeToolsProvider" icon="toolbox">
    Wraps the service into the `searchKnowledge` AI tool consumed by `ai-assistant/`
  </Card>
</CardGroup>

Three lines at the bottom of `src/modules/ai/ai.module.ts` (`providers` + `exports`) wire the glue + the `provideAiTool(...) as never` registration.

### Interim alignment (done — 2026-05-30, `ai-assistant`)

<Check>
To keep the eventual deletion + PCD-64 merge low-diff, `AIKnowledgeService` was refactored **in place** to mirror the canonical `AiAgentKnowledgeBaseService.searchAgentKnowledge` structure.
</Check>

The refactored service now uses:

- `tenantContext.executeInOrg` + `em.createQueryBuilder(KnowledgeChunk)`
- `raw('(kc.embedding <=> ?::vector)')` distance
- Returns flat `KnowledgeChunkHit[]` (`chunkId`, `content`, `sourceId`, `sourceTitle`, `sourceType`, `distance`)
- Adopted canonical `status = READY` filter and owner-scoping (`owner_user_id IS NULL OR = :requestingUserId`)
- Skips the `aiUsed` write (assistant is `accessMode: 'read'`)

<Info>
Class name (`AIKnowledgeService`), method name (`searchKnowledge`), and `ai.module.ts` wiring were intentionally left unchanged so this branch's edits stay conflict-free against PCD-64's `ai.module.ts` rewrite.
</Info>

**Still pending:** The deletion below, once PCD-64 actually merges.

## Why this is safe

<Tabs>
  <Tab title="Contract preservation">
    The tool's external contract is preserved:
    
    - Tool name
    - Input schema
    - Output projection fields (`chunkId` / `sourceId` / `sourceTitle` / `sourceType` / `snippet`)
  </Tab>
  
  <Tab title="RLS posture">
    `AiAgentKnowledgeBaseService.searchAgentKnowledge(...)` already runs inside `tenantContext.executeInOrg(orgId, ...)` — same RLS posture as the glue, which (post-alignment) uses the identical `executeInOrg` + `createQueryBuilder` + `raw()` pgvector pattern.
  </Tab>
  
  <Tab title="Vector space">
    The vector space (OpenAI `text-embedding-3-small`, 768 dims) and the pgvector cosine distance operator (`<=>`) are unchanged.
  </Tab>
  
  <Tab title="BYOK enforcement">
    BYOK enforcement (`BadRequestException` when no `OrganizationLlmKey` row exists) is consistent between the two services because both ultimately resolve through `OrganizationLlmKeyService.getActiveKey`.
  </Tab>
</Tabs>

## Why this is worth doing

<CardGroup cols={2}>
  <Card title="Code smell elimination" icon="broom">
    Two services doing the same thing is a smell, especially because the glue duplicates the raw pgvector SQL inline and would drift from the canonical service over time.
  </Card>
  
  <Card title="Dependency cleanup" icon="link-slash">
    `ai-knowledge-tools.provider.ts` exporting `AIKnowledgeService` ties the AI assistant's lifetime to a dev-period workaround that has now outlived its purpose.
  </Card>
  
  <Card title="Module simplification" icon="minimize">
    The eventual deletion shrinks `ai.module.ts` providers list by 1 service + cleans up the `exports` array.
  </Card>
  
  <Card title="Maintainability" icon="wrench">
    Removes duplicate logic that would require parallel maintenance going forward.
  </Card>
</CardGroup>

## Out of scope

<Warning>
The following items are **NOT** changing as part of this cleanup:
</Warning>

- **Tool name**: The `searchKnowledge` tool name itself stays. PCD-64 runtime exposes knowledge search to its agents through its own tool-registry layer (`AiAgentToolRegistryService`); the assistant's tool registry (`AiToolRegistryService`) is independent. Both can coexist long-term.

- **HTTP surface**: The `/v1/ai-knowledge-bases` HTTP endpoints remain unchanged.

- **Entities**: `KnowledgeBase`, `KnowledgeChunk`, `OrganizationLlmKey` entities are not modified.

- **Services**: `KnowledgeBaseService` ingestion flows and BYOK provisioning remain as-is.

<Tip>
This is purely a refactor to eliminate duplicate internal implementation while preserving all external contracts.
</Tip>
