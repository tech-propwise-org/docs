## Background

When `dev` cherry-picked PCD-64's shared knowledge-base infrastructure (`KnowledgeBase`, `KnowledgeChunk`, `OrganizationLlmKey`, `EmbeddingService`, `KnowledgeBaseService`, `AiAgentKnowledgeBaseController`, BYOK provisioning), it could not yet adopt PCD-64's `AiAgentKnowledgeBaseService` because the full AI Agent runtime was out of scope for that PR.

To keep the internal AI assistant's `searchKnowledge` tool working in the interim, dev shipped a thin, dev-only glue layer:

- `src/modules/ai/services/ai-knowledge.service.ts` — slim search-only service that wraps `EmbeddingService.generateEmbedding(...)` + raw pgvector SQL against `knowledge_chunks JOIN knowledge_base`
- `src/modules/ai/ai-knowledge-tools.provider.ts` — wraps the above into the `searchKnowledge` AI tool consumed by `ai-assistant/`
- Three lines at the bottom of `src/modules/ai/ai.module.ts` (`providers` + `exports`) wiring the glue + the `provideAiTool(...) as never` registration

### Interim Alignment (Completed)

<Check>
**Completed on 2026-05-30 (`ai-assistant` branch)**

To keep the eventual deletion + PCD-64 merge low-diff, `AIKnowledgeService` was refactored in place to mirror the canonical `AiAgentKnowledgeBaseService.searchAgentKnowledge` structure.
</Check>

The refactor adopted:
- `tenantContext.executeInOrg` + `em.createQueryBuilder(KnowledgeChunk)` + `raw('(kc.embedding <=> ?::vector)')` distance
- Flat `KnowledgeChunkHit[]` return format (`chunkId`, `content`, `sourceId`, `sourceTitle`, `sourceType`, `distance`)
- Canonical `status = READY` filter and owner-scoping (`owner_user_id IS NULL OR = :requestingUserId`)
- Skips `aiUsed` write (assistant is `accessMode: 'read'`)

Class name (`AIKnowledgeService`), method name (`searchKnowledge`), and `ai.module.ts` wiring were intentionally left unchanged to avoid conflicts with PCD-64's `ai.module.ts` rewrite.

## Goal

Migrate `AiKnowledgeToolsProvider` to call `AiAgentKnowledgeBaseService` directly, then delete `AIKnowledgeService` and remove the temporary registration lines from `ai.module.ts`.

<Warning>
The `searchKnowledge` tool's external contract (name, schema, accessMode, contextScopes, audit category, untrusted-content treatment) must remain identical so no eval suite or AI assistant test breaks.
</Warning>

## Why This is Safe

<Info>
**Key safety guarantees:**

- Tool's external contract preserved (name, input schema, output projection fields)
- `AiAgentKnowledgeBaseService.searchAgentKnowledge(...)` runs inside `tenantContext.executeInOrg(orgId, ...)` — same RLS posture
- Vector space (OpenAI `text-embedding-3-small`, 768 dims) and pgvector cosine distance operator (`<=>`) unchanged
- BYOK enforcement consistent between services via `OrganizationLlmKeyService.getActiveKey`
</Info>

## Why This is Worth Doing

- **Eliminates duplication:** Two services doing the same thing, with glue duplicating raw pgvector SQL that could drift over time
- **Removes technical debt:** `ai-knowledge-tools.provider.ts` dependency on dev-period workaround that has outlived its purpose  
- **Cleans up module:** Shrinks `ai.module.ts` providers list and exports array

## Out of Scope

<Note>
The following components remain unchanged:

- `searchKnowledge` tool name itself
- `/v1/ai-knowledge-bases` HTTP surface
- `KnowledgeBase` / `KnowledgeChunk` / `OrganizationLlmKey` entities
- `KnowledgeBaseService` ingestion flows
- BYOK provisioning

PCD-64 runtime exposes knowledge search through its own tool-registry layer (`AiAgentToolRegistryService`); the assistant's tool registry (`AiToolRegistryService`) remains independent.
</Note>
