## Background

When `dev` cherry-picked PCD-64's shared knowledge-base infrastructure (`KnowledgeBase`, `KnowledgeChunk`, `OrganizationLlmKey`, `EmbeddingService`, `KnowledgeBaseService`, `AiAgentKnowledgeBaseController`, BYOK provisioning), it could not yet adopt PCD-64's `AiAgentKnowledgeBaseService` because the full AI Agent runtime was out of scope for that PR.

To keep the internal AI assistant's `searchKnowledge` tool working in the interim, dev shipped a thin, dev-only glue layer:

- `src/modules/ai/services/ai-knowledge.service.ts` — slim search-only service that wraps `EmbeddingService.generateEmbedding(...)` + raw pgvector SQL against `knowledge_chunks JOIN knowledge_base`
- `src/modules/ai/ai-knowledge-tools.provider.ts` — wraps the above into the `searchKnowledge` AI tool consumed by `ai-assistant/`
- Three lines at the bottom of `src/modules/ai/ai.module.ts` (`providers` + `exports`) wiring the glue + the `provideAiTool(...) as never` registration

To keep the eventual deletion + PCD-64 merge low-diff, `AIKnowledgeService` was refactored **in place** to mirror the canonical `AiAgentKnowledgeBaseService.searchAgentKnowledge` structure:

- Uses `tenantContext.executeInOrg` + `em.createQueryBuilder(KnowledgeChunk)` + `raw('(kc.embedding <=> ?::vector)')` distance
- Returns flat `KnowledgeChunkHit[]` (`chunkId`, `content`, `sourceId`, `sourceTitle`, `sourceType`, `distance`)
- Adopted canonical `status = READY` filter and owner-scoping (`owner_user_id IS NULL OR = :requestingUserId`)
- Skips the `aiUsed` write (assistant is `accessMode: 'read'`)

Class name (`AIKnowledgeService`), method name (`searchKnowledge`), and `ai.module.ts` wiring were intentionally left unchanged to keep this branch's edits conflict-free against PCD-64's `ai.module.ts` rewrite.

## Goal

Migrate `AiKnowledgeToolsProvider` to call `AiAgentKnowledgeBaseService` directly, then delete `AIKnowledgeService` and remove the temporary registration lines from `ai.module.ts`. 

<Note>
The `searchKnowledge` tool's external contract (name, schema, accessMode, contextScopes, audit category, untrusted-content treatment) must remain identical so no eval suite or AI assistant test breaks.
</Note>

## Why this is safe

<CardGroup cols={2}>
<Card title="External contract preserved" icon="shield">
Tool's name, input schema, output projection fields (`chunkId`, `sourceId`, `sourceTitle`, `sourceType`, `snippet`) remain unchanged.
</Card>

<Card title="Same security posture" icon="lock">
`AiAgentKnowledgeBaseService.searchAgentKnowledge(...)` already runs inside `tenantContext.executeInOrg(orgId, ...)` — identical RLS as the glue.
</Card>

<Card title="Identical vector operations" icon="search">
Same vector space (OpenAI `text-embedding-3-small`, 768 dims) and pgvector cosine distance operator (`<=>`).
</Card>

<Card title="Consistent BYOK enforcement" icon="key">

</Card>

## Why this is worth doing

<Tip>
**Code quality benefits:**
- Eliminates duplicate service logic that would drift over time
- Removes temporary workaround that has outlived its purpose
- Shrinks `ai.module.ts` providers list and cleans up exports array
- Aligns assistant with canonical knowledge search implementation
</Tip>

## Out of scope

The following items are intentionally **not** changed in this refactor:

- The `searchKnowledge` tool name itself stays. PCD-64 runtime exposes knowledge search to its agents through its own tool-registry layer (`AiAgentToolRegistryService`); the assistant's tool registry (`AiToolRegistryService`) is independent. Both can coexist long-term.
- The `/v1/ai-knowledge-bases` HTTP surface, the `KnowledgeBase`/`KnowledgeChunk`/`OrganizationLlmKey` entities, the `KnowledgeBaseService` ingestion flows, the BYOK provisioning — none of these change.
