## Overview

We use **PostgreSQL Row-Level Security (RLS)** for tenant isolation. The **Organization entity** serves as the **tenant boundary**. The architecture uses **explicit transactions** with **`SET LOCAL`** to guarantee tenant isolation with connection pooling.

<Note>
Organization is a top-level module (peer to User), not part of RBAC. It represents the tenant boundary for multi-tenancy and is referenced by all tenant-scoped entities.
</Note>

## Architecture principles

### Transaction-first design

Every tenant-scoped operation runs inside an explicit transaction:

```typescript
await tenantContext.executeInOrg(organizationId, async (em) => {
  const lead = await em.findOne(Lead, id);
  lead.status = 'qualified';
  await em.flush();
});
```

### SET LOCAL only

Only `SET LOCAL` inside transactions — context is automatically cleared on COMMIT/ROLLBACK:

```sql
BEGIN;
SET LOCAL app.organization_id = 'uuid';
SET LOCAL app.user_id = 'uuid';
SET LOCAL app.bypass_rls = 'false';
-- All queries here see this context
-- Context automatically cleared on COMMIT/ROLLBACK
COMMIT;
```

### Database-level audit triggers

Audit logging happens via PostgreSQL triggers, not application code:

- **Atomic**: Same transaction as the data change
- **Automatic**: Can't forget to audit
- **Complete**: Captures all changes, even raw SQL
- **Performant**: No extra round-trips

### No middleware/interceptor RLS logic

Context is managed explicitly via the `TenantContext` service. There are no middleware or interceptors involved in RLS context management.

## Two-token authentication system

The system uses two distinct JWT token types to separate user identity from organization context:

### Identity token

Issued on login/register. Contains user identity only (no organization context):

```typescript
{
  sub: userId,
  email: user.email,
  tokenVersion: user.tokenVersion,
}
```

- Verified using `jwt.accessToken.secret`
- Used for pre-org-selection endpoints (list orgs, select org, refresh tokens)
- Validated by `IdentityTokenGuard`

### Tenant token

Issued when the user selects an organization via `POST /auth/tenant-token`. Contains organization context:

```typescript
{
  sub: userId,
  org: organizationId,
  orgName: organization.name,
  type: 'tenant',
  tokenVersion: user.tokenVersion,
}
```

- Verified using `jwt.tenantToken.secret`
- Used for all org-scoped data API endpoints
- Validated by `AuthGuard`
- A per-organization session is created when this token is issued (see SESSION_SYSTEM_DOCUMENTATION.md)

### Token flow

<Steps>
<Step title="User logs in">
Receives identity token + refresh token
</Step>

<Step title="Get organizations list">
User calls `GET /me/organizations` (identity token) → gets list of orgs
</Step>

<Step title="Select organization">
User calls `POST /auth/tenant-token` (identity token + orgId) → receives tenant token. Session created for this user+org combination
</Step>

<Step title="Access org-scoped data">
User calls data APIs (tenant token) → org-scoped data returned
</Step>

<Step title="Refresh tenant token">
Tenant token expires → refresh via `POST /auth/tenant-token/refresh`. Session validated on refresh
</Step>
</Steps>

## Components

### TenantContext service

The core service for executing tenant-scoped operations. Provided globally via `TenantContextModule`:

```typescript
@Injectable()
export class TenantContext {
  async executeInOrg<T>(
    orgId: string,
    operation: (em: EntityManager) => Promise<T>,
    options?: { userId?: string; isolationLevel?: IsolationLevel },
  ): Promise<T>;

  async executeWithBypass<T>(operation: (em: EntityManager) => Promise<T>): Promise<T>;

  async executeReadOnly<T>(orgId: string, operation: (em: EntityManager) => Promise<T>): Promise<T>;

  async executeReadOnlyWithBypass<T>(operation: (em: EntityManager) => Promise<T>): Promise<T>;

  async executeForEachOrg<T>(
    organizationIds: string[],
    operation: (em: EntityManager, orgId: string) => Promise<T>,
    options?: { continueOnError?: boolean },
  ): Promise<{ orgId: string; result?: T; error?: Error }[]>;

  async getCurrentContext(em: EntityManager): Promise<TenantContextInfo | null>;
}
```

### Auth guards

Three guards handle different authentication scenarios:

| Guard                | Token Type                | Use Case                                       |
| -------------------- | ------------------------- | ---------------------------------------------- |
| `AuthGuard`          | Tenant token              | Org-scoped data API endpoints (default)        |
| `IdentityTokenGuard` | Identity token            | Pre-org-selection endpoints (list/select orgs) |
| `EitherAuthGuard`    | Tenant OR Developer token | Shared data endpoints (reference data)         |

#### AuthGuard (default, applied globally)

Validates tenant tokens for all org-scoped data endpoints. The guard:

1. Checks `@PublicEndpoint()` metadata — skips if public
2. Checks `@IdentityTokenOnly()` metadata — skips if identity-token-only
3. Verifies JWT is a tenant token (rejects identity tokens)
4. Checks token version against the user entity for logout invalidation
5. Loads RBAC data using `tenantContext.executeInOrg()` within the guard
6. Loads organizations list using `tenantContext.executeWithBypass()`
7. Caches RBAC data via `PermissionCacheService`
8. Stores user info in CLS and builds `RequestUser` object

#### IdentityTokenGuard

Validates identity tokens for endpoints that don't need org context:

```typescript
@Get('organizations')
@IdentityTokenOnly()
@UseGuards(IdentityTokenGuard)
async getMyOrganizations() { ... }
```

This guard rejects tenant tokens and stores minimal user info (`id`, `email`, `tokenVersion`) in the request.

#### EitherAuthGuard

Accepts either CRM tenant tokens or developer portal tokens. Used for shared reference data endpoints:

```typescript
@UseGuards(EitherAuthGuard)
@Controller('reference-data')
export class ReferenceDataController { ... }
```

This guard only validates the JWT signature and type — it does NOT load user entities or RBAC data. It's specifically designed for endpoints that provide shared reference data that both CRM tenants and developer portal applications need to access.

### RLS policies

CASE-based policies that handle all scenarios:

```sql
CREATE POLICY "tenant_isolation" ON leads FOR ALL USING (
  CASE
    WHEN current_setting('app.bypass_rls', true) = 'true' THEN true
    WHEN current_setting('app.organization_id', true) = '' THEN false
    WHEN current_setting('app.organization_id', true) IS NULL THEN false
    ELSE organization_id = current_setting('app.organization_id', true)::uuid
  END
);
```

#### Messaging tables — CASE-based policy

The messaging tables (`message`, `conversation`, `channel_account`, `message_template`, `message_outbox`) were migrated from an older `OR`-based policy to the standard **CASE-based** policy in `Migration20260420200003_FixMessagingRlsPolicies`. They now use the same pattern as other tenant-scoped tables. The former `ai_chat_history` table was dropped when the external AI automation integration was removed.

```sql
CREATE POLICY "<table>_tenant_isolation" ON "<table>" FOR ALL USING (
  CASE
    WHEN current_setting('app.bypass_rls', true) = 'true' THEN true
    WHEN current_setting('app.organization_id', true) = '' THEN false
    WHEN current_setting('app.organization_id', true) IS NULL THEN false
    ELSE organization_id = current_setting('app.organization_id', true)::uuid
  END
);
```

The remaining messaging tables (`person_channel`, `message_reaction`, `message_status`, `webhook_history_chunk`) still use the simplified `OR`-based policy, which is functionally equivalent:

```sql
CREATE POLICY "<table>_tenant_isolation" ON "<table>" FOR ALL USING (
  current_setting('app.bypass_rls', true) = 'true'
  OR organization_id = NULLIF(current_setting('app.organization_id', true), '')::uuid
);
```

The `NULLIF` prevents a `::uuid` cast error when the setting is an empty string (returns `NULL` instead, which makes the `OR` branch evaluate to `NULL`/false). When `bypass_rls = 'true'`, the first branch short-circuits. This produces identical behavior to the `CASE` pattern with fewer branches.

The `org_commission_payment` and `user_org_commission` tables use the simplified OR-based policy with bypass branch for tenant isolation, created in `Migration20260409100000_AddOrgCommissionFields`.

### RLS for tables with global rows (organization_id IS NULL)

Some tables contain both **org-scoped rows** (with a specific `organization_id`) and **global/system rows** (with `organization_id = NULL`). Examples include `lead_stage` and `deal_stage`, which ship with system-default stages that have no organization.

The standard RLS policy `organization_id = current_setting('app.organization_id')::uuid` silently excludes global rows because `NULL = <uuid>` evaluates to `NULL` (not `TRUE`) in PostgreSQL. To make global rows visible to all tenants, add an explicit `OR organization_id IS NULL` clause:

```sql
CREATE POLICY "lead_stage_organization_isolation" ON "lead_stage"
  FOR ALL USING (
    organization_id = current_setting('app.organization_id', true)::uuid
    OR organization_id IS NULL
    OR current_setting('app.bypass_rls', true) = 'true'
  );
```

**When to use this pattern:**

- The table has system/default rows with `organization_id = NULL`
- These global rows are intentionally shared across all organizations
- Examples: `lead_stage`, `deal_stage` (system-default pipeline stages)

**When NOT to use this pattern:**

- The table is fully tenant-scoped (all rows have an `organization_id`)
- NULL `organization_id` would indicate a data integrity issue

## Request flow

### Authenticated request (tenant token)

<Steps>
<Step title="Request arrives">
Request arrives with tenant token in Authorization header
</Step>

<Step title="AuthGuard verification">
AuthGuard verifies JWT signature and checks type === 'tenant'
</Step>

<Step title="Token validation">
AuthGuard validates token version against user entity
</Step>

<Step title="RBAC data loading">
AuthGuard loads RBAC data via tenantContext.executeInOrg()
</Step>

<Step title="Organization list loading">
AuthGuard loads organizations list using tenantContext.executeWithBypass()
</Step>

<Step title="RBAC caching">
AuthGuard caches RBAC data via PermissionCacheService and stores user+org in CLS
</Step>

<Step title="Service execution">
Controller extracts orgId from request.user.organizationId and service calls tenantContext.executeInOrg(orgId, async (em) => { ... })
</Step>

<Step title="Database context">
Transaction starts, SET LOCAL sets context, all queries in operation use same connection with RLS filtering
</Step>

<Step title="Audit and commit">
Audit trigger fires on any changes, transaction commits, context auto-cleared, response sent
</Step>
</Steps>

### Identity token request (pre-org-selection)

<Steps>
<Step title="Request with identity token">
Request arrives with identity token in Authorization header
</Step>

<Step title="Guard check">
AuthGuard sees @IdentityTokenOnly() metadata and skips
</Step>

<Step title="Identity verification">
IdentityTokenGuard verifies JWT with identity token secret and stores minimal user data
</Step>
</Steps>

## How it works in practice

### For authenticated endpoints

Pass `organizationId` to services and use `TenantContext`:

```typescript
// Controller
@Get(':id')
async findOne(@Param('id') id: string, @GetCurrentUser() user: RequestUser) {
  return this.leadService.findOne(id, user.organizationId);
}

// Service
async findOne(id: string, organizationId: string): Promise<LeadDto> {
  return this.tenantContext.executeInOrg(organizationId, async (em) => {
    const lead = await em.findOne(Lead, { id });
    if (!lead) throw new NotFoundException();
    return this.toDto(lead);
  });
}
```

### For public endpoints

Add the decorator and use bypass:

```typescript
import { PublicEndpoint } from '@/decorators/rls.decorator';

@Post('login')
@PublicEndpoint()
async login(@Body() dto: LoginDto) {
  return this.authService.login(dto);
}

// In service
async login(dto: LoginDto) {
  return this.tenantContext.executeWithBypass(async (em) => {
    const user = await em.findOne(User, { email: dto.email });
    // ... authentication logic
  });
}
```

### For identity-token-only endpoints

Use `@IdentityTokenOnly()` and `@UseGuards(IdentityTokenGuard)`:

```typescript
import { IdentityTokenOnly } from '@/decorators/rls.decorator';
import { IdentityTokenGuard } from '@/modules/auth/guards/identity-token.guard';

@Get('organizations')
@IdentityTokenOnly()
@UseGuards(IdentityTokenGuard)
async getMyOrganizations(@GetCurrentUser() user: { id: string }) {
  return this.organizationService.findUserOrganizations(user.id);
}
```

## Method reference

### When to use each method

| Method | Use case | Transaction | RLS context |
|--------|----------|-------------|-------------|
| `executeInOrg` | Read/write operations with org context | Yes | Sets org_id |
| `executeReadOnly` | Read-only operations with org context | Yes (READ ONLY, READ COMMITTED) | Sets org_id |
| `executeWithBypass` | Cross-org or system operations | Yes | Bypasses RLS |
| `executeReadOnlyWithBypass` | Read-only cross-org queries | Yes (READ ONLY, READ COMMITTED) | Bypasses RLS |
| `executeForEachOrg` | Process multiple orgs sequentially | Yes (per org) | Sets org_id per iteration |
| `getCurrentContext` | Debug current tenant context | N/A (requires active txn) | Reads current |

<AccordionGroup>
<Accordion title="executeInOrg — for org-scoped operations">
Use for operations that read or modify data within an organization. Accepts optional `userId` (for audit triggers) and `isolationLevel`:

```typescript
async updateLead(id: string, dto: UpdateLeadDto, organizationId: string, userId: string) {
  return this.tenantContext.executeInOrg(organizationId, async (em) => {
    const lead = await em.findOne(Lead, { id });
    // ... update logic
    await em.flush();
  }, { userId });
}
```
</Accordion>

<Accordion title="executeReadOnly — for read-only org-scoped queries">
Uses `READ COMMITTED` isolation and `SET TRANSACTION READ ONLY` for performance hints:

```typescript
async findLeads(organizationId: string, filters: FilterDto) {
  return this.tenantContext.executeReadOnly(organizationId, async (em) => {
    return em.find(Lead, { status: filters.status });
  });
}
```

Benefits:
- Optimized for reads (READ COMMITTED + read-only hints)
- Still sets RLS context
- Prevents accidental writes
</Accordion>

<Accordion title="executeWithBypass — for cross-org operations">
Use when you need to query across organizations or access global data:

```typescript
async findUserOrganizations(userId: string) {
  return this.tenantContext.executeWithBypass(async (em) => {
    return em.find(Organization, { users: userId });
  });
}
```
</Accordion>

<Accordion title="executeReadOnlyWithBypass — for read-only cross-org queries">
Combines RLS bypass with read-only optimization. Used by webhook services (WhatsApp, Instagram) that need to read across tenants:

```typescript
async findOrgByPhoneNumber(phoneNumber: string) {
  return this.tenantContext.executeReadOnlyWithBypass(async (em) => {
    return em.findOne(WhatsAppAccount, { phoneNumber });
  });
}
```
</Accordion>

<Accordion title="executeForEachOrg — for multi-tenant batch processing">
Processes each organization in its own isolated transaction. Designed for cron jobs:

```typescript
await this.tenantContext.executeForEachOrg(orgIds, async (em, orgId) => {
  const expiredLeads = await em.find(Lead, { expiresAt: { $lt: new Date() } });
  // Process expired leads...
}, { continueOnError: true });
```

Returns an array of `{ orgId, result?, error? }` for each org. With `continueOnError: true` (default), failures in one org don't stop processing of others.
</Accordion>

<Accordion title="getCurrentContext — for debugging">
Reads the current PostgreSQL session variables. Only useful within an active transaction:

```typescript
const context = await this.tenantContext.getCurrentContext(em);
// { organizationId: '...', userId: '...' } or null if bypassed
```
</Accordion>
</AccordionGroup>

## Code organization patterns

### Pattern 1: simple method (most common)

For standalone operations, inline the implementation directly:

```typescript
async createTeam(dto: CreateTeamDto, organizationId: string) {
  return this.tenantContext.executeInOrg(organizationId, async (em) => {
    const existingTeam = await em.findOne(Team, { name: dto.name });
    if (existingTeam) throw new BadRequestException('Team exists');
    
    const team = em.create(Team, { ...dto, organization: organizationId });
    await em.persistAndFlush(team);
    
    return { success: true };
  });
}
```

### Pattern 2: composable method (when needed by other methods)

For methods that need to be called from within existing transactions:

```typescript
// Public API - wraps in TenantContext
async batchAssignRole(roleId: string, userIds: string[], orgId: string) {
  return this.tenantContext.executeInOrg(orgId, async (em) => {
    return this.batchAssignRoleInTransaction(em, roleId, userIds, orgId);
  });
}

// For composition - accepts EntityManager
async batchAssignRoleInTransaction(
  em: EntityManager,
  roleId: string,
  userIds: string[],
  organizationId: string,
) {
  const role = await em.findOne(Role, { id: roleId });
  // ... validation and assignment logic
}

// Another method that composes
async setRoleUsers(roleId: string, userIds: string[], orgId: string) {
  return this.tenantContext.executeInOrg(orgId, async (em) => {
    await this.batchAssignRoleInTransaction(em, roleId, usersToAdd, orgId);
    await this.batchRemoveRoleInTransaction(em, roleId, usersToRemove, orgId);
  });
}
```

**When to use `*InTransaction` methods:**
- Method is called by other service methods within the same transaction
- Method performs a distinct operation that's part of a larger workflow
- You need transaction composition (multiple operations must succeed/fail together)

**When NOT to use `*InTransaction` methods:**
- Method is only called from controllers (use Pattern 1)
- Method is a one-off operation (use Pattern 1)
- Adding unnecessary verbosity

### Method naming convention

| Pattern | Method name | Purpose |
|---------|-------------|---------|
| Simple | `createTeam` | Public API, contains implementation |
| Composable (public) | `batchAssignRole` | Public API, wraps InTransaction version |
| Composable (internal) | `batchAssignRoleInTransaction` | For composition, contains implementation |

### Decision tree

```
Is the method called by other service methods within a transaction?
├─ YES → Use Pattern 2 (Composable with *InTransaction)
│  └─ Create public method that wraps tenantContext
│  └─ Create *InTransaction method that accepts EntityManager
│
└─ NO → Use Pattern 1 (Simple inline)
   └─ Implement directly in tenantContext callback
   └─ Use private helper methods if needed (NOT *Impl!)
```

### Private helper methods

If you need private helper methods for code organization, that's fine — but do not use `Impl` suffix:

```typescript
async batchAssign(...) {
  return this.tenantContext.executeInOrg(orgId, async (em) => {
    const role = await this.validateRole(em, roleId);
    const users = await this.loadUsers(em, userIds);
    // ... assignment logic
  });
}

private async validateRole(em: EntityManager, roleId: string) {
  // Validation logic
}

private async loadUsers(em: EntityManager, userIds: string[]) {
  // User loading logic
}
```

### Rules summary

| Rule | Status | Notes |
|------|--------|-------|
| Use `tenantContext.executeInOrg` for writes | Required | Always wrap in TenantContext |
| Use `tenantContext.executeReadOnly` for reads | Recommended | Optimized for queries |
| Use `*InTransaction` for composition | Allowed | When method called from transactions |
| Use `*Impl` methods | Forbidden | Creates unnecessary abstraction |
| Nested TenantContext calls | Forbidden | Breaks transaction composition |
| Private helper methods | Allowed | For code organization, not `*Impl` |

## Special cases

<Tabs>
<Tab title="OAuth callbacks">
OAuth callbacks are public but need to set context from the state parameter:

```typescript
@Get('callback')
@PublicEndpoint()
async callback(@Query('state') state: string) {
  const { organizationId } = JSON.parse(decodeState(state));

  return this.tenantContext.executeInOrg(organizationId, async (em) => {
    // Now queries work within that org
  });
}
```
</Tab>

<Tab title="WebSocket handlers">
WebSocket gateways are **singletons**. Always use `TenantContext`:

```typescript
@Injectable()
@WebSocketGateway({ namespace: '/my-namespace' })
export class MyGateway {
  constructor(private readonly tenantContext: TenantContext) {}

  @SubscribeMessage('some-event')
  async handleEvent(@ConnectedSocket() client: AuthenticatedSocket) {
    return this.tenantContext.executeInOrg(client.organizationId, async (em) => {
      return em.find(SomeEntity, {});
    });
  }
}
```
</Tab>

<Tab title="Cron jobs">
```typescript
@Injectable()
export class MyScheduler {
  constructor(private readonly tenantContext: TenantContext) {}

  @Cron('0 * * * *')
  async hourlyTask() {
    // Option 1: Bypass for cross-org operations
    await this.tenantContext.executeWithBypass(async (em) => {
      const orgs = await em.find(Organization, { isDeleted: false });
      // Process each org...
    });

    // Option 2: Per-org processing with executeForEachOrg
    await this.tenantContext.executeForEachOrg(orgIds, async (em, orgId) => {
      // Process data for this org
    });
  }
}
```
</Tab>

<Tab title="Cross-org operations">
When an authenticated user needs to access a different org (e.g., accepting invitation):

```typescript
async acceptInvitation(token: string, currentUserId: string) {
  return this.tenantContext.executeWithBypass(async (em) => {
    const invitation = await em.findOne(Invitation, { token });

    // IMPORTANT: Pass the bypassed `em` to any helper methods that
    // query RLS-protected tables (team, user_org_roles, team_membership, etc.).
    // Using `this.em` or injected repositories inside bypass callbacks
    // would use the request-scoped EM which has NO RLS context,
    // causing silent query failures on RLS-protected tables.
    await this.applyInvitationRbacAssignments(em, user, invitation);
  });
}
```
</Tab>

<Tab title="Webhook with org lookup">
For external webhooks (WhatsApp, Instagram) that need to look up the target org first:

```typescript
@Post('webhook')
@PublicEndpoint()
async handleWebhook(@Body() body: WebhookPayload) {
  // Step 1: Find the org (read-only, bypass RLS)
  const account = await this.tenantContext.executeReadOnlyWithBypass(async (em) => {
    return em.findOne(WhatsAppAccount, { phoneNumber: body.from });
  });

  // Step 2: Process within the org
  return this.tenantContext.executeInOrg(account.organization.id, async (em) => {
    // Process webhook data within org context
  });
}
```
</Tab>
</Tabs>

## Quick reference

| Scenario | Approach |
| --- | --- |
| Normal authenticated endpoint | `tenantContext.executeInOrg(orgId, ...)` |
| Pre-org-selection endpoint | `@IdentityTokenOnly()` + `@UseGuards(IdentityTokenGuard)` |
| Shared data endpoint (CRM + developer) | `@UseGuards(EitherAuthGuard)` |
| Public + no org context | `@PublicEndpoint()` + `tenantContext.executeWithBypass()` |
| OAuth callback with org | `@PublicEndpoint()` + `tenantContext.executeInOrg(orgId)` |
| Webhook with org in payload | `@PublicEndpoint()` + `tenantContext.executeInOrg(orgId)` |
| Webhook with org lookup first | `@PublicEndpoint()` + `executeReadOnlyWithBypass()` then `executeInOrg()` |
| Cron job (cross-org) | `tenantContext.executeWithBypass()` |
| Cron job (per-org) | `tenantContext.executeForEachOrg(orgIds, ...)` |
| WebSocket handler | `tenantContext.executeInOrg(client.organizationId)` |

## Common mistakes

<Warning>
**Don't use the injected `this.em` inside TenantContext callbacks.** Use the provided transactional `em` parameter.

```typescript
// WRONG - this.em has no RLS context inside the callback
return this.tenantContext.executeInOrg(orgId, async (em) => {
  return this.em.findOne(Lead, { id }); // Uses wrong EM!
});

// CORRECT - use the provided transactional em
return this.tenantContext.executeInOrg(orgId, async (em) => {
  return em.findOne(Lead, { id });
});
```
</Warning>

<Warning>
**Don't nest TenantContext calls.** Use the `*InTransaction` pattern for composition.

```typescript
// WRONG - creates nested transactions
return this.tenantContext.executeInOrg(orgId, async (em) => {
  await this.someMethod(); // This calls executeInOrg again!
});

// CORRECT - use *InTransaction pattern
return this.tenantContext.executeInOrg(orgId, async (em) => {
  await this.someMethodInTransaction(em, ...);
});
```
</Warning>

<Warning>
**Don't forget `@PublicEndpoint` on public endpoints.** AuthGuard will reject the request otherwise.

```typescript
// WRONG - AuthGuard will reject the request
@Post('login')
async login() { }

// CORRECT
@Post('login')
@PublicEndpoint()
async login() { }
```
</Warning>

<Warning>
**Don't use `executeWithBypass` when you have an org context.** Use `executeInOrg` for org-scoped operations.

```typescript
// WRONG - bypasses all tenant isolation unnecessarily
return this.tenantContext.executeWithBypass(async (em) => {
  return em.find(Lead, { organization: orgId });
});

// CORRECT - use executeInOrg for org-scoped operations
return this.tenantContext.executeInOrg(orgId, async (em) => {
  return em.find(Lead, {});
});
```
</Warning>

<Warning>
**Never create `*Impl` methods.** They add unnecessary verbosity and confusion. Use either the simple inline pattern or the `*InTransaction` composable pattern.
</Warning>
