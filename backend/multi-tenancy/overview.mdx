We use **PostgreSQL Row-Level Security (RLS)** for tenant data isolation. The **Organization entity** (located at `/modules/organization/`) serves as the **tenant boundary**. The architecture uses **explicit transactions** with **`SET LOCAL`** to guarantee tenant isolation with connection pooling.

<Note>
Organization is a top-level module (peer to User), not part of RBAC. It represents the tenant boundary for multi-tenancy and is referenced by all tenant-scoped entities.
</Note>

## Architecture principles

### 1. Transaction-first design

Every tenant-scoped operation runs inside an explicit transaction:

```typescript
await tenantContext.executeInOrg(organizationId, async (em) => {
  const lead = await em.findOne(Lead, id);
  lead.status = 'qualified';
  await em.flush();
});
```

### 2. SET LOCAL only

Only `SET LOCAL` inside transactions â€” context is automatically cleared on COMMIT/ROLLBACK:

```sql
BEGIN;
SET LOCAL app.organization_id = 'uuid';
SET LOCAL app.user_id = 'uuid';
SET LOCAL app.bypass_rls = 'false';
-- All queries here see this context
-- Context automatically cleared on COMMIT/ROLLBACK
COMMIT;
```

<Note>
**Transaction-pooling compatibility (PgBouncer / Neon `-pooler`):** Because RLS context is set ONLY with `SET LOCAL` inside an explicit transaction (never a session-level `SET`), the app-query connection is safe to route through a **transaction-mode** pooler. Each transaction re-applies its own `SET LOCAL`, and `applyTransactionAuditContext()` clears stale actor fields at the start of every transaction so a reused pooled connection never leaks a prior transaction's `app.user_id`. **Never** introduce a session-level `SET` (non-`LOCAL`) in the request path or a connection-init `SET` in `driverOptions.connection` â€” that would silently corrupt RLS context under transaction pooling. (The non-`LOCAL` `SET app.bypass_rls` calls in `src/database/*Seeder.ts` are CLI-only and run on the direct `DATABASE_URL`, not the pooler.)
</Note>

### 3. Database-level audit triggers

Audit logging happens via PostgreSQL triggers, not application code:

- **Atomic**: Same transaction as the data change
- **Automatic**: Can't forget to audit
- **Complete**: Captures all changes, even raw SQL
- **Performant**: No extra round-trips

### 4. No middleware/interceptor RLS logic

Context is managed explicitly via the `TenantContext` service. There are no middleware or interceptors involved in RLS context management.

<Note>
**TimezoneMiddleware** runs globally and sets **`req.timezone`** from the optional **`Time-Zone`** HTTP header (validated IANA zone). This is **viewer calendar metadata** for task list/kanban stats only â€” it does **not** establish tenant context and does **not** interact with RLS.
</Note>

## Two-token authentication system

The system uses two distinct JWT token types to separate user identity from organization context:

### Identity token

Issued on login/register. Contains user identity only (no organization context):

```typescript
{
  sub: userId,
  email: user.email,
  tokenVersion: user.tokenVersion,
}
```

- Verified using `jwt.accessToken.secret`
- Used for pre-org-selection endpoints (list orgs, select org, refresh tokens)
- Validated by `IdentityTokenGuard`

### Tenant token

Issued when the user selects an organization via `POST /auth/tenant-token`. Contains organization context:

```typescript
{
  sub: userId,
  org: organizationId,
  orgName: organization.name,
  type: 'tenant',
  tokenVersion: user.tokenVersion,
}
```

- Verified using `jwt.tenantToken.secret`
- Used for all org-scoped data API endpoints
- Validated by `AuthGuard`
- A per-organization session is created when this token is issued (see SESSION_SYSTEM_DOCUMENTATION.md)

### Token flow

<Steps>
<Step title="User login">
User logs in â†’ receives identity token + refresh token
</Step>
<Step title="Organization selection">
User calls GET /me/organizations (identity token) â†’ gets list of orgs
</Step>
<Step title="Tenant token request">
User calls POST /auth/tenant-token (identity token + orgId) â†’ receives tenant token
Session created for this user+org combination
</Step>
<Step title="Data API access">
User calls data APIs (tenant token) â†’ org-scoped data returned
</Step>
<Step title="Token refresh">
Tenant token expires â†’ refresh via POST /auth/tenant-token/refresh
Session validated on refresh
</Step>
</Steps>

## Components

### TenantContext service

The core service for executing tenant-scoped operations. Provided globally via `TenantContextModule`:

```typescript
@Injectable()
export class TenantContext {
  async executeInOrg<T>(
    orgId: string,
    operation: (em: EntityManager) => Promise<T>,
    options?: { userId?: string; isolationLevel?: IsolationLevel },
  ): Promise<T>;

  async executeWithBypass<T>(operation: (em: EntityManager) => Promise<T>): Promise<T>;

  async executeReadOnly<T>(orgId: string, operation: (em: EntityManager) => Promise<T>): Promise<T>;

  async executeReadOnlyWithBypass<T>(operation: (em: EntityManager) => Promise<T>): Promise<T>;

  async executeForEachOrg<T>(
    organizationIds: string[],
    operation: (em: EntityManager, orgId: string) => Promise<T>,
    options?: { continueOnError?: boolean },
  ): Promise<{ orgId: string; result?: T; error?: Error }[]>;

  async getCurrentContext(em: EntityManager): Promise<TenantContextInfo | null>;
}
```

### Auth guards

<Warning>
**Raw SQL inside `executeInOrg` / `executeReadOnly`:** Prefer `em.execute(...)` or MikroORM `em.count` / `createQueryBuilder` on the **same** transactional `EntityManager` so queries run on the connection that received `SET LOCAL app.organization_id`. Avoid `em.getKnex()` **and `em.getConnection().execute(...)`** for RLS-scoped aggregates â€” both can use another pool connection without tenant session variables, so the RLS policy sees no `app.organization_id` and silently returns zero rows, which makes row counts and sums appear as zero. (Concrete regression: `SubscriptionService.countSeatedUsers` used `em.getConnection().execute()` and reported every org as `0/N` seats used until switched to `em.execute()`.)
</Warning>

### OAuth callback security

Public OAuth callbacks (email and calendar integrations) use **signed state tokens** via `SignedPayloadService` to prevent CSRF attacks and ensure callbacks are processed within the correct tenant context. The HMAC-signed state includes `userId`, `organizationId`, and integration-specific metadata. Calendar OAuth starts additionally use tenant-authenticated `calendar_oauth_intent` rows: `/auth/{provider}-calendar/intent` writes the org-scoped intent under normal RLS, while public `/authorize` uses bypass only to atomically consume the one-time intent before provider redirect.

<Note>
**Re-authorization at callback time:** OAuth callbacks re-verify the actor's current organization access within the same transaction as token persistence using `AccessService.assertOAuthCallbackOrgAccessInTransaction()`. This prevents privilege escalation if the user was removed from the organization while on the provider's consent screen.
</Note>

Four guards handle different authentication scenarios:

| Guard                | Token Type                | Use Case                                       |
| -------------------- | ------------------------- | ---------------------------------------------- |
| `AuthGuard`          | Tenant token              | Org-scoped data API endpoints (default)        |
| `IdentityTokenGuard` | Identity token            | Pre-org-selection endpoints (list/select orgs) |
| `EitherAuthGuard`    | Tenant OR Developer token | Shared data endpoints (reference data)         |
| `DeveloperAuthGuard` | Developer token           | Developer portal endpoints                     |

#### AuthGuard (default, applied globally)

Validates tenant tokens for all org-scoped data endpoints. The guard:

1. Checks `@PublicEndpoint()` metadata â€” skips if public
2. Checks `@IdentityTokenOnly()` metadata â€” skips if identity-token-only
3. Verifies JWT is a tenant token (rejects identity tokens)
4. Checks token version against the user entity for logout invalidation
5. OrgSession liveness (`orgSessionId`); on cache miss the bypass read also rejects soft-deleted users and expired users
6. Loads RBAC data using `tenantContext.executeInOrg()` within the guard
7. Loads organizations list using `tenantContext.executeWithBypass()`
8. Caches RBAC data via `PermissionCacheService`
9. Stores user info in CLS and builds `RequestUser` object

#### IdentityTokenGuard

Validates identity tokens for endpoints that don't need org context:

```typescript
@Get('organizations')
@IdentityTokenOnly()
@UseGuards(IdentityTokenGuard)
async getMyOrganizations() { ... }
```

This guard rejects tenant tokens and stores minimal user info (`id`, `email`, `tokenVersion`) in the request.

#### EitherAuthGuard

Accepts either CRM tenant tokens or developer portal tokens. Used for shared Propwise Labs endpoints:

```typescript
@UseGuards(EitherAuthGuard)
@Controller('propwise-labs')
export class PropwiseLabsController { ... }
```

This guard only validates the JWT signature and type â€” it does NOT load user entities or RBAC data. When it accepts a developer token, it marks `request.skipSessionCheck=true` so shared Propwise Labs lookup endpoints can be called from the developer portal without the CRM `refresh_token` cookie. CRM tenant-token requests remain on the normal session-validation path.

#### DeveloperAuthGuard

Developer portal controllers authenticate outside the CRM tenant-token/session model. Use `DeveloperPortalAuth()` on developer portal controller classes; it composes `DeveloperAuthGuard` with `SkipSessionCheck()` so the global `SessionActivityInterceptor` does not require the CRM `refresh_token` cookie for `/developer-portal/me`, `/developer-portal/projects`, `/developer-portal/buildings`, or developer portal logout.

```typescript
@DeveloperPortalAuth()
@Controller('developer-portal/me')
export class DeveloperProfileController {}
```

`DeveloperAuthGuard` validates the developer JWT, enforces approval/active/company-link checks, validates `tokenVersion`, and stores `request.developerUser`.

### RLS policies

CASE-based policies that handle all scenarios:

```sql
CREATE POLICY "tenant_isolation" ON leads FOR ALL USING (
  CASE
    WHEN current_setting('app.bypass_rls', true) = 'true' THEN true
    WHEN current_setting('app.organization_id', true) = '' THEN false
    WHEN current_setting('app.organization_id', true) IS NULL THEN false
    ELSE organization_id = current_setting('app.organization_id', true)::uuid
  END
);
```

#### Messaging tables â€” CASE-based policy

The messaging tables (`message`, `conversation`, `channel_account`, `message_template`, `message_outbox`) were migrated from an older `OR`-based policy to the standard **CASE-based** policy. They now use the same pattern as other tenant-scoped tables:

```sql
CREATE POLICY "<table>_tenant_isolation" ON "<table>" FOR ALL USING (
  CASE
    WHEN current_setting('app.bypass_rls', true) = 'true' THEN true
    WHEN current_setting('app.organization_id', true) = '' THEN false
    WHEN current_setting('app.organization_id', true) IS NULL THEN false
    ELSE organization_id = current_setting('app.organization_id', true)::uuid
  END
);
```

The remaining messaging tables (`person_channel`, `message_reaction`, `message_status`, `webhook_history_chunk`) still use the simplified `OR`-based policy, which is functionally equivalent:

```sql
CREATE POLICY "<table>_tenant_isolation" ON "<table>" FOR ALL USING (
  current_setting('app.bypass_rls', true) = 'true'
  OR organization_id = NULLIF(current_setting('app.organization_id', true), '')::uuid
);
```

The `NULLIF` prevents a `::uuid` cast error when the setting is an empty string (returns `NULL` instead, which makes the `OR` branch evaluate to `NULL`/false). When `bypass_rls = 'true'`, the first branch short-circuits.

## Request flow

### Authenticated request (tenant token)

<Steps>
<Step title="Token verification">
Request arrives with tenant token in Authorization header
AuthGuard verifies JWT signature and checks type === 'tenant'
</Step>
<Step title="Token validation">
AuthGuard validates token version against user entity
</Step>
<Step title="RBAC loading">
AuthGuard loads RBAC data via tenantContext.executeInOrg()
AuthGuard caches RBAC data and stores user+org in CLS
</Step>
<Step title="Service execution">
Controller extracts orgId from request.user.organizationId
Service calls `tenantContext.executeInOrg(orgId, async (em) => { ... })`
</Step>
<Step title="Transaction processing">
Transaction starts, SET LOCAL sets context
All queries in operation use same connection with RLS filtering
Audit trigger fires on any changes
</Step>
<Step title="Response">
Transaction commits, context auto-cleared
Response sent
</Step>
</Steps>

### Identity token request (pre-org-selection)

<Steps>
<Step title="Guard selection">
Request arrives with identity token in Authorization header
AuthGuard sees @IdentityTokenOnly() metadata and skips
</Step>
<Step title="Identity validation">
IdentityTokenGuard verifies JWT with accessToken secret
IdentityTokenGuard rejects tenant tokens
</Step>
<Step title="Minimal context">
Stores minimal user info (id, email, tokenVersion) in request
No RBAC or organization loading
</Step>
<Step title="Service execution">
Service can call tenantContext.executeWithBypass() for cross-org queries
Or executeInOrg() for specific org operations
</Step>
</Steps>
