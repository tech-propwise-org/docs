We use **PostgreSQL Row-Level Security (RLS)** for tenant data isolation. The **Organization entity** (located at `/modules/organization/`) serves as the **tenant boundary**. The architecture uses **explicit transactions** with **`SET LOCAL`** to guarantee tenant isolation with connection pooling.

<Note>
Organization is a top-level module (peer to User), not part of RBAC. It represents the tenant boundary for multi-tenancy and is referenced by all tenant-scoped entities.
</Note>

## Architecture principles

### 1. Transaction-first design

Every tenant-scoped operation runs inside an explicit transaction:

```typescript typescript
await tenantContext.executeInOrg(organizationId, async (em) => {
  const lead = await em.findOne(Lead, id);
  lead.status = 'qualified';
  await em.flush();
});
```

### 2. SET LOCAL only

Only `SET LOCAL` inside transactions — context is automatically cleared on COMMIT/ROLLBACK:

```sql sql
BEGIN;
SET LOCAL app.organization_id = 'uuid';
SET LOCAL app.user_id = 'uuid';
SET LOCAL app.bypass_rls = 'false';
-- All queries here see this context
-- Context automatically cleared on COMMIT/ROLLBACK
COMMIT;
```

### 3. Database-level audit triggers

Audit logging happens via PostgreSQL triggers, not application code:

- **Atomic**: Same transaction as the data change
- **Automatic**: Can't forget to audit
- **Complete**: Captures all changes, even raw SQL
- **Performant**: No extra round-trips

### 4. No middleware/interceptor RLS logic

Context is managed explicitly via the `TenantContext` service. There are no middleware or interceptors involved in RLS context management.

## Two-token authentication system

The system uses two distinct JWT token types to separate user identity from organization context:

### Identity token

Issued on login/register. Contains user identity only (no organization context):

```typescript typescript
{
  sub: userId,
  email: user.email,
  tokenVersion: user.tokenVersion,
}
```

- Verified using `jwt.accessToken.secret`
- Used for pre-org-selection endpoints (list orgs, select org, refresh tokens)
- Validated by `IdentityTokenGuard`

### Tenant token

Issued when the user selects an organization via `POST /auth/tenant-token`. Contains organization context:

```typescript typescript
{
  sub: userId,
  org: organizationId,
  orgName: organization.name,
  type: 'tenant',
  tokenVersion: user.tokenVersion,
}
```

- Verified using `jwt.tenantToken.secret`
- Used for all org-scoped data API endpoints
- Validated by `AuthGuard`
- A per-organization session is created when this token is issued (see [session system](/backend/session-system))

### Token flow

<Steps>
<Step title="User login">
User logs in → receives identity token + refresh token
</Step>
<Step title="Organization selection">
User calls GET /me/organizations (identity token) → gets list of orgs
</Step>
<Step title="Tenant token request">
User calls POST /auth/tenant-token (identity token + orgId) → receives tenant token
Session created for this user+org combination
</Step>
<Step title="Data API access">
User calls data APIs (tenant token) → org-scoped data returned
</Step>
<Step title="Token refresh">
Tenant token expires → refresh via POST /auth/tenant-token/refresh
Session validated on refresh
</Step>
</Steps>

## Components

### TenantContext service

The core service for executing tenant-scoped operations. Provided globally via `TenantContextModule`:

```typescript typescript
@Injectable()
export class TenantContext {
  async executeInOrg<T>(
    orgId: string,
    operation: (em: EntityManager) => Promise<T>,
    options?: { userId?: string; isolationLevel?: IsolationLevel },
  ): Promise<T>;

  async executeWithBypass<T>(operation: (em: EntityManager) => Promise<T>): Promise<T>;

  async executeReadOnly<T>(orgId: string, operation: (em: EntityManager) => Promise<T>): Promise<T>;

  async executeReadOnlyWithBypass<T>(operation: (em: EntityManager) => Promise<T>): Promise<T>;

  async executeForEachOrg<T>(
    organizationIds: string[],
    operation: (em: EntityManager, orgId: string) => Promise<T>,
    options?: { continueOnError?: boolean },
  ): Promise<{ orgId: string; result?: T; error?: Error }[]>;

  async getCurrentContext(em: EntityManager): Promise<TenantContextInfo | null>;
}
```

### Auth guards

Three guards handle different authentication scenarios:

| Guard | Token Type | Use Case |
| --- | --- | --- |
| `AuthGuard` | Tenant token | Org-scoped data API endpoints (default) |
| `IdentityTokenGuard` | Identity token | Pre-org-selection endpoints (list/select orgs) |
| `EitherAuthGuard` | Tenant OR Developer token | Shared data endpoints (reference data) |

#### AuthGuard (default, applied globally)

Validates tenant tokens for all org-scoped data endpoints. The guard:

1. Checks `@PublicEndpoint()` metadata — skips if public
2. Checks `@IdentityTokenOnly()` metadata — skips if identity-token-only
3. Verifies JWT is a tenant token (rejects identity tokens)
4. Checks token version against the user entity for logout invalidation
5. Loads RBAC data using `tenantContext.executeInOrg()` within the guard
6. Loads organizations list using `tenantContext.executeWithBypass()`
7. Caches RBAC data via `PermissionCacheService`
8. Stores user info in CLS and builds `RequestUser` object

#### IdentityTokenGuard

Validates identity tokens for endpoints that don't need org context:

```typescript typescript
@Get('organizations')
@IdentityTokenOnly()
@UseGuards(IdentityTokenGuard)
async getMyOrganizations() { ... }
```

This guard rejects tenant tokens and stores minimal user info (`id`, `email`, `tokenVersion`) in the request.

#### EitherAuthGuard

Accepts either CRM tenant tokens or developer portal tokens. Used for shared reference data endpoints:

```typescript typescript
@UseGuards(EitherAuthGuard)
@Controller('reference-data')
export class ReferenceDataController { ... }
```

This guard only validates the JWT signature and type — it does NOT load user entities or RBAC data.

## RLS policies

CASE-based policies that handle all scenarios:

```sql sql
CREATE POLICY "tenant_isolation" ON leads FOR ALL USING (
  CASE
    WHEN current_setting('app.bypass_rls', true) = 'true' THEN true
    WHEN current_setting('app.organization_id', true) = '' THEN false
    WHEN current_setting('app.organization_id', true) IS NULL THEN false
    ELSE organization_id = current_setting('app.organization_id', true)::uuid
  END
);
```

## Table configuration

### No RLS (global/cross-tenant)

| Table | Reason |
| --- | --- |
| user | Users belong to multiple orgs |
| permission | Global permission registry |
| role | System roles are global (org-specific have org_id but filtered in app) |
| organization | Tenant table itself |

### RLS enabled (tenant-scoped)

All tables with `organization_id NOT NULL`:

- CRM: person, lead, contact, company, deal, etc.
- RBAC: team, team_membership, user_org_roles, team_membership_roles, etc.
- Communication: activity, call_log, email_log, etc.
- Real Estate: area, development, unit, etc.
- Audit: audit_log

## Audit triggers

Database triggers that automatically log changes:

```sql sql
CREATE TRIGGER audit_lead_changes
AFTER INSERT OR UPDATE OR DELETE ON lead
FOR EACH ROW EXECUTE FUNCTION audit_trigger_func();
```

The trigger function is `SECURITY DEFINER` so it bypasses RLS for audit writes while reads remain tenant-isolated.

### Audit log special handling

The audit_log table has RLS but audit triggers bypass it:

```sql sql
CREATE OR REPLACE FUNCTION audit_trigger_func()
RETURNS TRIGGER
SECURITY DEFINER  -- Runs as table owner, bypasses RLS
AS $$ ... $$;
```

This ensures:

- Audit writes always succeed
- Audit reads respect tenant isolation
- No application code needed for audit writes

## Request flow

### Authenticated request (tenant token)

<Steps>
<Step title="Token verification">
Request arrives with tenant token in Authorization header
AuthGuard verifies JWT signature and checks type === 'tenant'
</Step>
<Step title="Token validation">
AuthGuard validates token version against user entity
</Step>
<Step title="RBAC loading">
AuthGuard loads RBAC data via tenantContext.executeInOrg()
AuthGuard caches RBAC data and stores user+org in CLS
</Step>
<Step title="Service execution">
Controller extracts orgId from request.user.organizationId
Service calls tenantContext.executeInOrg(orgId, async (em) => { ... })
</Step>
<Step title="Transaction processing">
Transaction starts, SET LOCAL sets context
All queries in operation use same connection with RLS filtering
Audit trigger fires on any changes
</Step>
<Step title="Response">
Transaction commits, context auto-cleared
Response sent
</Step>
</Steps>

### Identity token request (pre-org-selection)

<Steps>
<Step title="Guard selection">
Request arrives with identity token in Authorization header
AuthGuard sees @IdentityTokenOnly() metadata and skips
</Step>
<Step title="Identity validation">
IdentityTokenGuard verifies JWT with accessToken secret
IdentityTokenGuard rejects tenant tokens
</Step>
<Step title="Minimal context">
Stores minimal user info (id, email, tokenVersion) in request
No RBAC or organization loading
</Step>
<Step title="Service execution">
Service can call tenantContext.executeWithBypass() for cross-org queries
Or executeInOrg() for specific org operations
</Step>
</Steps>

## Error handling

### Transaction rollback

If any operation fails, the entire transaction rolls back:

```typescript typescript
await tenantContext.executeInOrg(orgId, async (em) => {
  await em.findOne(Lead, id); // Works
  throw new Error('Something failed'); // Transaction rolls back
  // SET LOCAL context is automatically cleared
});
```

### No connection pool pollution

Because we use `SET LOCAL`:

- Context only exists within the transaction
- On commit/rollback, context is automatically cleared
- Connection returns to pool in clean state
- No cleanup interceptor needed

## Organization as the tenant boundary

### Module structure

The **Organization module** is at `/modules/organization/` as a top-level module, separate from RBAC:

```
src/modules/
├── organization/           ← Tenant Boundary
│   ├── organization.entity.ts
│   ├── organization.service.ts
│   ├── organization.controller.ts
│   └── organization.module.ts
├── user/                   ← User Identity
├── rbac/                   ← Authorization
└── crm/                    ← Business Logic
```

### Organization's role in multi-tenancy

1. **Tenant isolation**: All tenant-scoped entities have `organization_id` foreign key
2. **RLS context**: The `organization_id` is used to set PostgreSQL RLS context
3. **Access control**: Combined with RBAC to determine what users can see/do
4. **Data partitioning**: Organization boundaries enforce strict data separation

### Tenant-scoped entity pattern

All entities that belong to an organization should:

1. Have `organization` field as ManyToOne relationship
2. Include `organization_id` in RLS policies
3. Filter by organization in queries (automatic with RLS)

```typescript typescript
import { Organization } from '@modules/organization/organization.entity';

@Entity()
export class Lead {
  @ManyToOne(() => Organization)
  organization!: Organization;

  // ... other fields
}
```

## Performance considerations

### Transaction overhead

There is overhead for explicit transactions. Mitigations:

1. **Read-only shortcut**: Use `executeReadOnly` for queries that don't need writes
2. **Batch operations**: Multiple operations in one transaction
3. **Connection pooling**: Still fully utilized

### Audit trigger performance

Triggers add ~1-2ms per write. Acceptable for:

- Data integrity guarantee
- Compliance requirements
- Debugging capability

### Connection pooling optimization

With `SET LOCAL` and transaction-based isolation:

- No connection state bleeding between requests
- Connection pool remains efficient
- Database resources properly managed
- Consistent performance across tenant operations
