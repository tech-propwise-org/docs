The RBAC (Role-Based Access Control) system provides comprehensive access control across organizations and teams with a unified architecture supporting multi-tenancy, system roles, and permission inheritance.

## Architecture overview

### Design principles

The RBAC system follows these core principles:

<Tabs>
<Tab title="Unified System">
- Single `Role` table for both org-level and team-level roles
- Single `Permission` table for all permissions
- `scope` field differentiates between ORG and TEAM levels
</Tab>
<Tab title="System vs Custom">
- System roles have `organization = NULL` (global/seeded)
- Custom roles have `organization = specific org ID`
- System roles: Owner, Admin, Basic, Team Leader, Agent
</Tab>
<Tab title="Explicit Junctions">
- `UserOrgRole` - User ↔ Role assignments in Organization context
- `TeamMembershipRole` - TeamMembership ↔ Role assignments
- Required for multi-org support (user can have different roles in different orgs)
- Includes metadata: `assigned_by`, `created_at`, `organization_id` (for RLS)
</Tab>
<Tab title="Multi-Tenancy">
- Users can belong to multiple organizations
- Same user can have different roles in different organizations
- RLS (Row-Level Security) enforced at database level
</Tab>
<Tab title="Permission Inheritance">
- Team roles can have both TEAM-scoped and ORG-scoped permissions
- Team-available permissions controlled via `team_available_permissions`
</Tab>
</Tabs>

## Core entities

### User

**Purpose**: Represents a user account in the system.

```typescript
User
├── id: uuid
├── first_name, last_name: string
├── email: string (globally unique among active users)
├── password: string (hashed)
├── phone_number?: string
├── avatar_url?: string (public URL to avatar image, stored as WebP on R2)
├── languages?: string[] (array of language codes)
├── preferred_language?: string
├── is_verified: boolean
├── is_active: boolean
├── is_system_admin: boolean (platform-level admin, distinct from RBAC roles)
├── is_deleted: boolean (soft delete)
├── selected_organization_id?: uuid → Organization
├── Auth Fields:
│   ├── refresh_token?: string
│   ├── token_version: number (default: 1, incremented on token invalidation)
│   ├── verification_code?: string
│   ├── verification_code_expires?: timestamp
│   ├── verification_email_attempts: number (default: 0)
│   ├── last_verification_email_sent_at?: timestamp
│   ├── reset_password_code?: string
│   └── reset_password_code_expires?: timestamp
├── Relationships:
│   ├── organizations → Organization[] (via organization_users)
│   ├── orgRoles → UserOrgRole[] (roles in orgs)
│   └── teamMemberships → TeamMembership[]
└── Timestamps: created_at, updated_at

Indexes:
├── (first_name, last_name) [composite]
└── (email, is_deleted) [unique, where: is_deleted = false]
```

<Note>
User can belong to multiple organizations. Email is globally unique among non-deleted users. `selected_organization_id` tracks current org context. Access roles via `orgRoles` collection (not direct `roles`). `is_system_admin` is a platform-level flag independent of per-org RBAC roles.
</Note>

### Organization

**Module Location:** `/modules/organization/` (separate from RBAC)

**Purpose**: Represents a tenant organization serving as the **tenant boundary** for multi-tenancy.

```typescript
Organization
├── id: uuid
├── name: string
├── website?: string
├── email?: string
├── vat?: string
├── avatar_url?: string (public URL to org logo, stored as WebP on R2)
├── address?: jsonb (embedded Address object)
├── owner_id: uuid → User (organization owner)
├── Relationships:
│   ├── users → User[] (ManyToMany, owner side, via organization_users)
│   ├── owner → User (ManyToOne)
│   ├── leads → Lead[] (OneToMany)
│   ├── contacts → Contact[] (OneToMany)
│   ├── deals → Deal[] (OneToMany)
│   └── companies → Company[] (OneToMany)
└── Timestamps: created_at, updated_at, is_deleted
```

<Note>
Teams, Roles, and Invitations each reference Organization via their own `@ManyToOne` relationships, but Organization does not declare inverse `@OneToMany` collections for these. CRM entities (Lead, Contact, Deal, Company) do have inverse relationships on Organization.
</Note>

### Role (Unified)

**Purpose**: Represents a role that can be assigned to users. Unified table for both org-level and team-level roles.

```typescript
Role
├── id: uuid
├── name: string
├── description?: string
├── color?: string
├── scope: RoleScope (ORG | TEAM)
├── is_system: boolean (true for seeded roles)
├── organization_id?: uuid → Organization (NULL for system roles)
├── team_id?: uuid → Team (NULL for org roles)
├── Relationships:
│   ├── permissions → Permission[] (via role_permissions)
│   ├── userOrgRoles → UserOrgRole[] (users with this role)
│   └── teamMembershipRoles → TeamMembershipRole[] (team members with this role)
└── Timestamps: created_at, updated_at, is_deleted

Indexes:
├── (name, scope, is_deleted) [unique, where: is_deleted = false AND organization_id IS NULL AND team_id IS NULL]
│   └── Ensures system role names are unique per scope
└── (name, scope, organization_id, team_id, is_deleted) [unique, where: is_deleted = false AND organization_id IS NOT NULL]
    └── Ensures custom role names are unique per scope within an org/team
```

<Tabs>
<Tab title="Scope rules">
- `scope = ORG`: Organization-level role (e.g., Admin, Basic)
- `scope = TEAM`: Team-level role (e.g., Team Leader, Agent)
</Tab>
<Tab title="System vs custom">
- System roles: `is_system = true`, `organization = NULL`
- Custom roles: `is_system = false`, `organization = specific org`
</Tab>
<Tab title="Team roles">
- Team roles have `scope = TEAM` and `team_id` set
- Can have both TEAM-scoped and ORG-scoped permissions
- ORG permissions must be in `team_available_permissions`
</Tab>
</Tabs>

### Permission

**Purpose**: Represents an individual permission that can be assigned to roles.

```typescript
Permission
├── id: uuid
├── key: string (unique identifier, e.g., 'crm.view')
├── name: string (display name)
├── description?: string
├── category: string (e.g., 'CRM', 'Sales', 'System')
├── scope: PermissionScope (ORG | TEAM)
├── is_system_sensitive: boolean (only for Owner role)
├── Relationships:
│   ├── roles → Role[] (via role_permissions)
│   └── teamAvailablePermissions → TeamAvailablePermission[] (org perms available to teams)
└── Timestamps: created_at, updated_at

Indexes:
└── key [unique]
```

<Note>
Permissions are globally defined and seeded. The `is_system_sensitive` flag restricts certain permissions to only the Owner role.
</Note>

## Explicit junction tables

### UserOrgRole

**Purpose**: Junction table for User-Role assignments within Organization context.

```typescript
UserOrgRole
├── id: uuid
├── user_id: uuid → User
├── role_id: uuid → Role
├── organization_id: uuid → Organization (for RLS)
├── assigned_by_id: uuid → User
├── Relationships:
│   ├── user → User
│   ├── role → Role
│   ├── organization → Organization
│   └── assignedBy → User
└── Timestamps: created_at, updated_at

Indexes:
├── (user_id, role_id, organization_id) [unique]
└── organization_id [for RLS]
```

<Warning>
Required for multi-org support. Allows the same user to have different roles in different organizations. The `organization_id` field enables Row-Level Security (RLS) filtering.
</Warning>

### TeamMembershipRole

**Purpose**: Junction table for TeamMembership-Role assignments (team-scoped roles).

```typescript
TeamMembershipRole
├── id: uuid
├── team_membership_id: uuid → TeamMembership
├── role_id: uuid → Role
├── organization_id: uuid → Organization (for RLS)
├── assigned_by_id: uuid → User
├── Relationships:
│   ├── teamMembership → TeamMembership
│   ├── role → Role
│   ├── organization → Organization
│   └── assignedBy → User
└── Timestamps: created_at, updated_at

Indexes:
├── (team_membership_id, role_id) [unique]
└── organization_id [for RLS]
```

### TeamAvailablePermission

**Purpose**: Defines which ORG-scoped permissions are available to team roles within a team.

```typescript
TeamAvailablePermission
├── id: uuid
├── team_id: uuid → Team
├── permission_id: uuid → Permission
├── organization_id: uuid → Organization (for RLS)
├── Relationships:
│   ├── team → Team
│   ├── permission → Permission
│   └── organization → Organization
└── Timestamps: created_at, updated_at

Indexes:
├── (team_id, permission_id) [unique]
└── organization_id [for RLS]
```

<Tip>
This table enables team roles to have ORG-scoped permissions while maintaining control over which org permissions are available to each team.
</Tip>

## Permission resolution

### Permission inheritance

Team roles can have both TEAM-scoped and ORG-scoped permissions:

1. **TEAM-scoped permissions**: Directly assigned to team roles
2. **ORG-scoped permissions**: Must be in `team_available_permissions` to be usable by team roles

### Resolution algorithm

<Steps>
<Step title="Get user's org-level permissions">
Query `UserOrgRole` to find all org roles for the user, then resolve their permissions.
</Step>
<Step title="Get user's team-level permissions">
For each team membership:
- Query `TeamMembershipRole` to find team roles
- Resolve TEAM-scoped permissions directly
- Resolve ORG-scoped permissions only if they exist in `TeamAvailablePermission`
</Step>
<Step title="Combine and deduplicate">
Merge org and team permissions, removing duplicates by permission key.
</Step>
</Steps>

### Query patterns

<Tabs>
<Tab title="User's effective permissions">
```sql
-- Get all effective permissions for a user in an organization
WITH org_permissions AS (
  SELECT DISTINCT p.key, p.scope
  FROM user_org_roles uor
  JOIN role_permissions rp ON rp.role_id = uor.role_id
  JOIN permissions p ON p.id = rp.permission_id
  WHERE uor.user_id = $1 AND uor.organization_id = $2
),
team_permissions AS (
  SELECT DISTINCT p.key, p.scope
  FROM team_memberships tm
  JOIN team_membership_roles tmr ON tmr.team_membership_id = tm.id
  JOIN role_permissions rp ON rp.role_id = tmr.role_id
  JOIN permissions p ON p.id = rp.permission_id
  LEFT JOIN team_available_permissions tap ON (
    tap.team_id = tm.team_id AND 
    tap.permission_id = p.id AND 
    p.scope = 'ORG'
  )
  WHERE tm.user_id = $1 AND tm.organization_id = $2
  AND (p.scope = 'TEAM' OR tap.id IS NOT NULL)
)
SELECT key, scope FROM org_permissions
UNION
SELECT key, scope FROM team_permissions;
```
</Tab>
<Tab title="Role permission check">
```sql
-- Check if user has specific permission
SELECT EXISTS (
  SELECT 1 FROM user_org_roles uor
  JOIN role_permissions rp ON rp.role_id = uor.role_id
  JOIN permissions p ON p.id = rp.permission_id
  WHERE uor.user_id = $1 
    AND uor.organization_id = $2 
    AND p.key = $3
  
  UNION
  
  SELECT 1 FROM team_memberships tm
  JOIN team_membership_roles tmr ON tmr.team_membership_id = tm.id
  JOIN role_permissions rp ON rp.role_id = tmr.role_id
  JOIN permissions p ON p.id = rp.permission_id
  LEFT JOIN team_available_permissions tap ON (
    tap.team_id = tm.team_id AND 
    tap.permission_id = p.id AND 
    p.scope = 'ORG'
  )
  WHERE tm.user_id = $1 
    AND tm.organization_id = $2 
    AND p.key = $3
    AND (p.scope = 'TEAM' OR tap.id IS NOT NULL)
) AS has_permission;
```
</Tab>
</Tabs>

## System roles

System roles are seeded globally and have `organization = NULL`:

| Role | Scope | Description | Permission strategy |
| --- | --- | --- | --- |
| **Owner** | ORG | Organization owner | `allPermissions: true` (including system-sensitive) |
| **Admin** | ORG | Organization admin | `allNonSensitive: true` (excludes `system.owner`) |
| **Basic** | ORG | Basic user | `permissionKeys: []` (relies on ownership checks) |
| **Team Leader** | TEAM | Team leader | `allPermissions: true` (all team + available org) |
| **Agent** | TEAM | Team agent | `permissionKeys: []` (assigned per-team) |

### Default permissions

<Tabs>
<Tab title="Owner">
Gets **all** org-scoped permissions including system-sensitive ones:

`system.owner`, `system.admin`, `crm.view`, `crm.manage`, `sales.view`, `sales.manage`, `real_estate.manage`, `marketing.manage`
</Tab>
<Tab title="Admin">
Gets all org-scoped permissions **except** those marked `isSystemSensitive`:

`system.admin`, `crm.view`, `crm.manage`, `sales.view`, `sales.manage`, `real_estate.manage`, `marketing.manage`

Cannot: `system.owner` (system-sensitive)
</Tab>
<Tab title="Basic">
No org-level permissions by default. Relies on resource ownership checks for access.
</Tab>
<Tab title="Team Leader">
Gets all team-scoped permissions + available org permissions for the team:

`team_crm.view`, `team_crm.manage`, `team_sales.view`, `team_sales.manage`, `team.admin`, `team_transfer.manage`, `team_report.view`
</Tab>
<Tab title="Agent">
No default team permissions. Specific permissions assigned per-team by team leader/admin.
</Tab>
</Tabs>

When a new organization is created:

```typescript
async function provisionOrganization(data: CreateOrganizationDto) {
```

## Invitation system

### Invitation entity

```typescript
Invitation
├── id: uuid
├── email: string
├── status: InvitationStatus (PENDING | ACCEPTED | EXPIRED | REVOKED)
├── organization_id: uuid → Organization
├── team_id?: uuid → Team (for team invitations)
├── invited_by_id: uuid → User
├── accepted_by_id?: uuid → User
├── token: string (unique, for invitation links)
├── expires_at: timestamp
├── roles: Role[] (roles to assign upon acceptance)
├── Relationships:
│   ├── organization → Organization
│   ├── team → Team (optional)
│   ├── invitedBy → User
│   └── acceptedBy → User (optional)
└── Timestamps: created_at, updated_at, accepted_at

Indexes:
├── token [unique]
├── (email, organization_id, status)
└── organization_id [for RLS]
```

### Invitation flow

<Steps>
<Step title="Create invitation">
Admin/Owner creates invitation with email and roles. System generates unique token and expiration time.
</Step>
<Step title="Send invitation email">
Email sent with invitation link containing token. Link directs to registration/login page.
</Step>
<Step title="Accept invitation">
User registers/logs in and accepts invitation. System validates token and assigns specified roles.
</Step>
<Step title="Role assignment">
Upon acceptance, system creates appropriate UserOrgRole or TeamMembershipRole entries based on invitation type.
</Step>
</Steps>

### Business rules

<Warning>
- Only users with `system.admin` or higher can send org invitations
- Only team leaders or org admins can send team invitations  
- Invitations expire after 7 days by default
- Cannot invite existing organization members
- Accepting expired/revoked invitations fails gracefully
</Warning>

```typescript
// Get user's effective permissions
async getUserPermissions(userId: string, organizationId: string): Promise<string[]> {
  const query = `
    WITH org_perms AS (
      SELECT DISTINCT p.key
      FROM user_org_roles uor
      JOIN role_permissions rp ON rp.role_id = uor.role_id
      JOIN permissions p ON p.id = rp.permission_id

      AND (p.scope = 'TEAM' OR tap.id IS NOT NULL)
    )
    SELECT key FROM org_perms UNION SELECT key FROM team_perms;
  `;
  
  return this.em.getConnection().execute(query, [userId, organizationId]);
}
```

```typescript
// Check if user has specific role
async hasRole(userId: string, organizationId: string, roleName: string): Promise<boolean> {
  const orgRole = await this.userOrgRoleRepo.findOne({
    user: userId,
    organization: organizationId,
    role: { name: roleName, scope: RoleScope.ORG }
  });
  
  if (orgRole) return true;
  
  const teamRole = await this.teamMembershipRoleRepo.findOne({
    teamMembership: {
      user: userId,
      organization: organizationId
    },
    role: { name: roleName, scope: RoleScope.TEAM }
  });
  
  return !!teamRole;
}
```
</Tab>
<Tab title="Resource ownership">
```typescript
// Check resource ownership with role fallback
async canAccessResource(
  userId: string, 
  organizationId: string, 
  resourceOwnerId: string,
  requiredPermission: string
): Promise<boolean> {
  // 1. Check direct ownership
  if (userId === resourceOwnerId) {
    return true;
  }
  
  // 2. Check role permissions
  const hasPermission = await this.hasPermission(userId, organizationId, requiredPermission);
  return hasPermission;
}
```

## Business rules

| Rule | Description |
| --- | --- |
| **One system org-role per user** | A user can only have one system org-scoped role (Owner, Admin, or Basic) at a time. Adding a new system role auto-removes the previous one. |
| **Only Owner can assign Owner/Admin** | Only users with the Owner role or the `organization.owner_id` can assign Owner or Admin roles. |
| **Owner role removal protection** | Only the `organization.owner_id` can remove the Owner role from any user. A user cannot remove Owner from themselves. |
| **Admin role removal protection** | Only users with the Owner role or `organization.owner_id` can remove the Admin role. Admin users cannot modify users who have the Owner role. |
| **System role protection** | System roles (`organization = NULL`) cannot be deleted or modified |
| **Role scope validation** | ORG-scoped roles can only be assigned via UserOrgRole |
| **Team role validation** | TEAM-scoped roles can only be assigned via TeamMembershipRole |
| **Organization consistency** | `UserOrgRole.organization` must match user's organization membership |
| **Team consistency** | `TeamMembershipRole.organization` must match `TeamMembership.organization` |
| **Permission inheritance** | Team roles with ORG permissions must have those permissions in `team_available_permissions` |
| **Invitation expiry** | Invitations automatically expire after 7 days |
| **Unique role assignment** | Cannot assign the same role to the same user/team context twice |

## RLS integration

### Row-Level Security patterns

The RBAC system integrates with PostgreSQL Row-Level Security (RLS) to enforce tenant isolation:

<Tabs>
<Tab title="RLS Policies">
```sql
-- Enable RLS on all RBAC tables
ALTER TABLE user_org_roles ENABLE ROW LEVEL SECURITY;
ALTER TABLE team_membership_roles ENABLE ROW LEVEL SECURITY;
ALTER TABLE invitations ENABLE ROW LEVEL SECURITY;

-- Policy for user_org_roles
CREATE POLICY user_org_roles_tenant_isolation ON user_org_roles
    FOR ALL TO authenticated_user
    USING (organization_id = current_setting('app.current_organization_id')::uuid);

-- Policy for team_membership_roles
CREATE POLICY team_membership_roles_tenant_isolation ON team_membership_roles
    FOR ALL TO authenticated_user
    USING (organization_id = current_setting('app.current_organization_id')::uuid);

-- Policy for invitations
CREATE POLICY invitations_tenant_isolation ON invitations
    FOR ALL TO authenticated_user
    USING (organization_id = current_setting('app.current_organization_id')::uuid);
```
</Tab>
<Tab title="Context Setting">
```typescript
// Set organization context for RLS
async setOrganizationContext(organizationId: string) {
  await this.em.getConnection().execute(
    'SET app.current_organization_id = $1',
    [organizationId]
  );
}

// Middleware to set context from JWT
@Injectable()
export class RLSMiddleware implements NestMiddleware {
  use(req: any, res: any, next: () => void) {
    if (req.user?.organizationId) {
      req.organizationId = req.user.organizationId;
      // Context will be set by interceptor
    }
    next();
  }
}
```
</Tab>
</Tabs>

<Warning>
All RBAC junction tables include `organization_id` for RLS enforcement. This ensures users can only access roles, permissions, and invitations within their current organization context.
</Warning>
