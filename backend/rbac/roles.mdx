The RBAC (Role-Based Access Control) system provides comprehensive access control across organizations and teams with a unified architecture supporting multi-tenancy, system roles, and permission inheritance.

## Architecture overview

### Design principles

The RBAC system follows these core principles:

1. **Unified Role & Permission System**:
   - Single `Role` table for both org-level and team-level roles
   - Single `Permission` table for all permissions
   - `scope` field differentiates between ORG and TEAM levels

2. **System vs Custom Roles**:
   - System roles have `organization = NULL` (global/seeded)
   - Custom roles have `organization = specific org ID`
   - System roles: Owner, Admin, Basic, Team Leader, Agent

3. **Explicit Junction Tables**:
   - `UserOrgRole` - User ‚Üî Role assignments in Organization context
   - `TeamMembershipRole` - TeamMembership ‚Üî Role assignments
   - Required for multi-org support (user can have different roles in different orgs)
   - Includes metadata: `assigned_by`, `created_at`, `organization_id` (for RLS)

4. **Multi-Tenancy Support**:
   - Users can belong to multiple organizations
   - Same user can have different roles in different organizations
   - RLS (Row-Level Security) enforced at database level

5. **Permission Inheritance**:
   - Team roles can have both TEAM-scoped and ORG-scoped permissions
   - Team-available permissions controlled via `team_available_permissions`

**Module Location:** `/modules/organization/` (separate from RBAC)

## Permission resolution

### Permission inheritance

Team roles can have both TEAM-scoped and ORG-scoped permissions:

1. **TEAM-scoped permissions**: Directly assigned to team roles
2. **ORG-scoped permissions**: Must be in `team_available_permissions` to be usable by team roles

### Resolution algorithm

<Steps>
<Step title="Get user's org-level permissions">
Query `UserOrgRole` to find all org roles for the user, then resolve their permissions.
</Step>
<Step title="Get user's team-level permissions">
For each team membership:
- Query `TeamMembershipRole` to find team roles
- Resolve TEAM-scoped permissions directly
- Resolve ORG-scoped permissions only if they exist in `TeamAvailablePermission`
</Step>
<Step title="Combine and deduplicate">
Merge org and team permissions, removing duplicates by permission key.
</Step>
</Steps>

### Query patterns

<Tabs>
<Tab title="User's effective permissions">
```sql
-- Get all effective permissions for a user in an organization
WITH org_permissions AS (
  SELECT DISTINCT p.key, p.scope
  FROM user_org_roles uor
  JOIN role_permissions rp ON rp.role_id = uor.role_id
  JOIN permissions p ON p.id = rp.permission_id
  WHERE uor.user_id = $1 AND uor.organization_id = $2
),
team_permissions AS (
  SELECT DISTINCT p.key, p.scope
  FROM team_memberships tm
  JOIN team_membership_roles tmr ON tmr.team_membership_id = tm.id
  JOIN role_permissions rp ON rp.role_id = tmr.role_id
  JOIN permissions p ON p.id = rp.permission_id
  LEFT JOIN team_available_permissions tap ON (
    tap.team_id = tm.team_id AND 
    tap.permission_id = p.id AND 
    p.scope = 'ORG'
  )
  WHERE tm.user_id = $1 AND tm.organization_id = $2
  AND (p.scope = 'TEAM' OR tap.id IS NOT NULL)
)
SELECT key, scope FROM org_permissions
UNION
SELECT key, scope FROM team_permissions;
```

```sql
    p.scope = 'ORG'
  )
  WHERE tm.user_id = $1 
    AND tm.organization_id = $2 
    AND p.key = $3
    AND (p.scope = 'TEAM' OR tap.id IS NOT NULL)
) AS has_permission;
```

## System roles

System roles are seeded globally and have `organization = NULL`:

| Role | Scope | Description | Permission strategy |
| --- | --- | --- | --- |

| **Basic** | ORG | Basic user | `permissionKeys: []` (relies on ownership checks) |

### Invitation flow

<Steps>
<Step title="Create invitation">
Admin/Owner creates invitation with email and roles. System generates unique token and expiration time.
</Step>
<Step title="Send invitation email">
Email sent with invitation link containing token. Link directs to registration/login page.
</Step>
<Step title="Accept invitation">
User registers/logs in and accepts invitation. System validates token and assigns specified roles.
</Step>
<Step title="Role assignment">
Upon acceptance, system creates appropriate UserOrgRole or TeamMembershipRole entries based on invitation type.
</Step>
</Steps>

### Business rules

<Warning>
- Only users with `system.admin` or higher can send org invitations
- Only team leaders or org admins can send team invitations  
- Invitations expire after 7 days by default
- Cannot invite existing organization members
- Accepting expired/revoked invitations fails gracefully
</Warning>

```typescript
// Get user's effective permissions
async getUserPermissions(userId: string, organizationId: string): Promise<string[]> {
  const query = `
    WITH org_perms AS (
      SELECT DISTINCT p.key
      FROM user_org_roles uor
      JOIN role_permissions rp ON rp.role_id = uor.role_id
      JOIN permissions p ON p.id = rp.permission_id
      WHERE uor.user_id = $1 AND uor.organization_id = $2
    ),
    team_perms AS (
      SELECT DISTINCT p.key
      FROM team_memberships tm
      JOIN team_membership_roles tmr ON tmr.team_membership_id = tm.id
      JOIN role_permissions rp ON rp.role_id = tmr.role_id
      JOIN permissions p ON p.id = rp.permission_id
      LEFT JOIN team_available_permissions tap ON (
        tap.team_id = tm.team_id AND tap.permission_id = p.id AND p.scope = 'ORG'
      )
      WHERE tm.user_id = $1 AND tm.organization_id = $2
      AND (p.scope = 'TEAM' OR tap.id IS NOT NULL)
    )
    SELECT key FROM org_perms UNION SELECT key FROM team_perms;
  `;
  
  return this.em.getConnection().execute(query, [userId, organizationId]);
}
```

```typescript
// Check if user has specific role
async hasRole(userId: string, organizationId: string, roleName: string): Promise<boolean> {
  const orgRole = await this.userOrgRoleRepo.findOne({
    user: userId,
    organization: organizationId,
    role: { name: roleName, scope: RoleScope.ORG }
  });
  
  if (orgRole) return true;
  
  const teamRole = await this.teamMembershipRoleRepo.findOne({
    teamMembership: {
      user: userId,
      organization: organizationId
    },
    role: { name: roleName, scope: RoleScope.TEAM }
  });
  
  return !!teamRole;
}
```

```typescript
async canAccessResource(
  userId: string, 
  organizationId: string, 
  resourceOwnerId: string,
  requiredPermission: string
): Promise<boolean> {
  // 1. Check direct ownership
  if (userId === resourceOwnerId) {
    return true;
  }
  
  // 2. Check role permissions
  const hasPermission = await this.hasPermission(userId, organizationId, requiredPermission);
  return hasPermission;
}
```

## Business rules

| Rule | Description |
| --- | --- |
| **One system org-role per user** | A user can only have one system org-scoped role (Owner, Admin, or Basic) at a time. Adding a new system role auto-removes the previous one. |
| **Only Owner can assign Owner/Admin** | Only users with the Owner role or the `organization.owner_id` can assign Owner or Admin roles. |
| **Owner role removal protection** | Only the `organization.owner_id` can remove the Owner role from any user. A user cannot remove Owner from themselves. |
| **Admin role removal protection** | Only users with the Owner role or `organization.owner_id` can remove the Admin role. Admin users cannot modify users who have the Owner role. |
| **System role protection** | System roles (`organization = NULL`) cannot be deleted or modified |
| **Role scope validation** | ORG-scoped roles can only be assigned via UserOrgRole |
| **Team role validation** | TEAM-scoped roles can only be assigned via TeamMembershipRole |
| **Organization consistency** | `UserOrgRole.organization` must match user's organization membership |
| **Team consistency** | `TeamMembershipRole.organization` must match `TeamMembership.organization` |
| **Permission inheritance** | Team roles with ORG permissions must have those permissions in `team_available_permissions` |
| **Invitation expiry** | Invitations automatically expire after 7 days |
| **Unique role assignment** | Cannot assign the same role to the same user/team context twice |

## RLS integration

### Row-Level Security patterns

The RBAC system integrates with PostgreSQL Row-Level Security (RLS) to enforce tenant isolation:

<Tabs>
<Tab title="RLS Policies">
```sql
-- Enable RLS on all RBAC tables
ALTER TABLE user_org_roles ENABLE ROW LEVEL SECURITY;
ALTER TABLE team_membership_roles ENABLE ROW LEVEL SECURITY;
ALTER TABLE invitations ENABLE ROW LEVEL SECURITY;

-- Policy for user_org_roles
CREATE POLICY user_org_roles_tenant_isolation ON user_org_roles
    FOR ALL TO authenticated_user
    USING (organization_id = current_setting('app.current_organization_id')::uuid);

-- Policy for team_membership_roles
CREATE POLICY team_membership_roles_tenant_isolation ON team_membership_roles
    FOR ALL TO authenticated_user
    USING (organization_id = current_setting('app.current_organization_id')::uuid);

-- Policy for invitations
CREATE POLICY invitations_tenant_isolation ON invitations
    FOR ALL TO authenticated_user
    USING (organization_id = current_setting('app.current_organization_id')::uuid);
```
</Tab>
<Tab title="Context Setting">
```typescript

// Middleware to set context from JWT
@Injectable()
export class RLSMiddleware implements NestMiddleware {
  use(req: any, res: any, next: () => void) {
    if (req.user?.organizationId) {
      req.organizationId = req.user.organizationId;
      // Context will be set by interceptor

    next();
  }
```
