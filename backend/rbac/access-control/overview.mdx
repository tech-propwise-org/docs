The access control and permissions system has **two layers** that work together:

| Layer                    | Purpose                                 | When It Runs                              | Result                                |
| ------------------------ | --------------------------------------- | ----------------------------------------- | ------------------------------------- |
| **Endpoint Protection**  | Decides if a request is allowed         | Before controller handler (NestJS guard)  | 403 Forbidden or allow through        |
| **Response Permissions** | Tells the frontend what the user can do | Inside the service, after data is fetched | `permissions` object on each resource |

**Flow of a request:**

```
Request → AuthGuard → AccessGuard (Layer 1: allow or 403)
  → Controller → Service → PermissionComputeService (Layer 2: attach permissions)
    → Response with { data, permissions }
```

# Layer 1: Endpoint Protection

This layer decides **whether a request is allowed to proceed**. It runs as a NestJS guard before the controller handler. If the user lacks access, the request is rejected with a 403.

## Core Concepts

### Access Check Types

| Type                    | Description                                 | Use Case                          |
| ----------------------- | ------------------------------------------- | --------------------------------- |
| **Permission Check**    | RBAC permission (e.g., `crm.manage`)        | List endpoints, create operations |
| **Resource Check**      | Stakeholder-based access to specific entity | Single-entity operations          |
| **Bulk Resource Check** | Batch access check for multiple entities    | Bulk operations                   |

### Access Modes

| Mode    | Behavior                                               | Default |
| ------- | ------------------------------------------------------ | ------- |
| **OR**  | User passes if they have permission OR resource access | ✅ Yes  |
| **AND** | User must have BOTH permission AND resource access     | No      |

### Resource Types

```typescript
  CONTACT = 'CONTACT',

  TEAM = 'TEAM',

}
```

### Resource Actions

```typescript
enum ResourceAction {
  ACCESS = 'ACCESS', // Can view
  MODIFY = 'MODIFY', // Can update
  DELETE = 'DELETE', // Can delete
}
```

**Messaging Resources:**

- `CHANNEL_ACCOUNT` - Channel accounts (WhatsApp, Instagram, Messenger)
- `CONVERSATION` - Individual messaging conversations with contacts

**Messaging Permissions:**

- `messaging.view` - View conversations and messages
- `messaging.write` - Send messages, update conversations, and basic conversation operations  
- `messaging.manage` - Full conversation management including assignment and transfer, channel account management
- `messaging.admin` - Channel account setup and template management

**Channel Account Access Control:**

Channel account resource checks use ownership-based logic:

- **Personal accounts** (`level = PERSONAL`): The account owner can ACCESS, MODIFY, and DELETE their own accounts without needing `messaging.manage` permission.
- **Organization accounts** (`level = ORGANIZATION`): No resource-level access — requires `messaging.manage` permission via RBAC.

The delete endpoint and sync controller both use **OR mode**: users with `messaging.manage` can operate on any account, OR personal account owners can operate on their own accounts.

```typescript
// Delete account — OR mode: messaging.manage OR personal account owner
@Delete(':id')
@CheckAccess({
  permissions: [OrgPermissionKey.MESSAGING_MANAGE],
  resource: { type: ResourceType.CHANNEL_ACCOUNT, action: ResourceAction.DELETE },
})
async delete(@Param('id') id: string) {}

// Sync controller — OR mode: messaging.manage OR personal account owner
@Controller('messaging/sync/:channelAccountId')
@CheckAccess({
  permissions: [OrgPermissionKey.MESSAGING_MANAGE],
  resource: { type: ResourceType.CHANNEL_ACCOUNT, action: ResourceAction.ACCESS, paramName: 'channelAccountId' },
})
```

**Conversation Access Control:**

Conversation access follows a multi-tier system:

- **Tier 1a (Broad permissions)**: Users with any org messaging permission (`MESSAGING_VIEW`, `MESSAGING_WRITE`, or `MESSAGING_MANAGE`) can access all org conversations
- **Tier 1b (Reply permissions)**: Only agents with `canReply=true` assignments may send messages, react to messages, or toggle AI mode
- **Tier 2 (Personal account ownership)**: Users who own the personal channel account
- **Tier 3 (Assignment-based)**: Users with active (non-deleted) assignments
- **Tier 3b (Team membership)**: Users who are members of a team that has an active assignment on the conversation
- **Tier 4 (CRM stakeholder)**: Users who are stakeholders on linked leads/deals
- **Tier 5 (Team leader scope)**: Team leaders can only assign and remove assignments within their own team scope

All conversation-scoped endpoints use `@CheckAccess` with `ResourceType.CONVERSATION` and `ResourceAction.ACCESS`. In **OR mode** (default), users with any org messaging permission bypass the resource check; users without org permissions fall through to the resource check, which uses a single SQL query testing all 5 access paths (personal owner, direct assignment, team assignment, lead stakeholder, deal stakeholder). This eliminates the need for manual `assertConversationAccess` calls in controller methods.

Message search (`GET /messaging/messages/search`) requires `conversationId` as a mandatory query parameter and uses `@CheckAccess` with `ResourceType.CONVERSATION` to enforce access at the guard level.

## Single-Resource Endpoints

### @CheckAccess Decorator

The `@CheckAccess` decorator combines permission and resource checks:

```typescript
@CheckAccess({
  permissions: string[],           // RBAC permissions (any match = pass)
  resource?: {
    type: ResourceType,            // Entity type
    action: ResourceAction,        // ACCESS, MODIFY, DELETE
    paramName?: string,            // Route param name (default: 'id')
  },
  mode?: AccessMode,               // OR (default) or AND
})
```

### Usage Patterns

#### Permission Only (List/Create)

```typescript
@Get()
@CheckAccess({ permissions: ['crm.view', 'crm.manage'] })
async findAll() {}

@Post()
@CheckAccess({ permissions: ['crm.create', 'crm.manage'] })
async create(@Body() dto: CreateLeadDto) {}
```

#### Permission OR Resource (View/Update/Delete)

```typescript
@Get(':id')
@CheckAccess({
  permissions: ['crm.view', 'crm.manage'],
  resource: { type: ResourceType.LEAD, action: ResourceAction.ACCESS },
})
async findOne(@Param('id') id: string) {}

@Put(':id')
@CheckAccess({
  permissions: ['crm.manage'],
  resource: { type: ResourceType.LEAD, action: ResourceAction.MODIFY },
})
async update(@Param('id') id: string, @Body() dto: UpdateLeadDto) {}

@Delete(':id')
@CheckAccess({
  permissions: ['crm.manage'],
  resource: { type: ResourceType.LEAD, action: ResourceAction.DELETE },
})
async remove(@Param('id') id: string) {}
```

#### Permission AND Resource (Sensitive Operations)

```typescript
@Put(':id/transfer')
@CheckAccess({
  permissions: ['crm.transfer'],
  resource: { type: ResourceType.LEAD, action: ResourceAction.MODIFY },
  mode: AccessMode.AND,
})
async transfer(@Param('id') id: string, @Body() dto: TransferDto) {}
```

#### Resource Only (No Permission Check)

```typescript
@Get(':id')
@CheckAccess({
  resource: { type: ResourceType.LEAD, action: ResourceAction.ACCESS },
})
async getLeadResourceOnly(@Param('id') id: string) {}
```

#### Custom Parameter Name

```typescript
@Put('transfer/:leadId')
@CheckAccess({
  permissions: ['crm.manage'],
  resource: {
    type: ResourceType.LEAD,
    action: ResourceAction.MODIFY,
    paramName: 'leadId',
  },
})
async transfer(@Param('leadId') leadId: string) {}
```

### Permission-Only with Team Permissions (Stakeholder Management)

For endpoints where resource-level access should NOT grant permission (e.g., managing stakeholders on an entity), use permission-only checks that include both org-level and team-level management permissions:

```typescript
// Stakeholder sync on leads - permission-only, NO resource check
@Put(':id/stakeholders')
@CheckAccess({
  permissions: [OrgPermissionKey.CRM_MANAGE, TeamPermissionKey.TEAM_CRM_MANAGE],
})
async syncStakeholders() {}

// Stakeholder sync on deals - permission-only, NO resource check
@Put(':id/stakeholders')
@CheckAccess({
  permissions: [OrgPermissionKey.SALES_MANAGE, TeamPermissionKey.TEAM_SALES_MANAGE],
})
async syncStakeholders() {}

// Legacy individual endpoints (deprecated, same permissions):
// POST :id/stakeholders, PATCH :id/stakeholders/batch, DELETE :id/stakeholders/:id
```

<Warning>
**Why no resource check?** Having `accessLevel: MODIFY` on an entity (via being a stakeholder) should not grant permission to add, remove, or modify other stakeholders. Stakeholder management is an administrative function that requires explicit RBAC permissions.
</Warning>

**Two-tier access for team managers:**

1. **Guard level:** `TEAM_CRM_MANAGE` / `TEAM_SALES_MANAGE` passes the guard (user has this permission on at least one team).
2. **Controller level:** Business logic further validates:
   - Team has a stakeholder on the specific entity (enforced on `createStakeholder`, `batchUpdate`, `removeStakeholder`)
   - Updates/adjustments only target the team's own stakeholders or team member stakeholders (enforced on `createStakeholder`, `batchUpdate`)
   - Removals only target the team's own stakeholders

## Bulk-Resource Endpoints

For operations on multiple resources, use `bulkResource` configuration:

```typescript
@Post('bulk-archive')
@CheckAccess({
  permissions: [OrgPermissionKey.CRM_MANAGE],
  bulkResource: {
    type: ResourceType.LEAD,
    action: ResourceAction.MODIFY,
    paramName: 'leadIds',
    strategy: 'partial', // Allow partial success
  },
})
async bulkArchive(
  @Req() request: any,
  @Body() body: { leadIds: string[] },
  @GetCurrentUser() user: RequestUser,
) {
  const bulkAccessContext = request.bulkAccessContext;
  return this.leadService.bulkArchiveLeads(
    body.leadIds,
    user.id,
    user.organizationId!,
    bulkAccessContext,
  );
}
```

## Access Rules by Resource Type

<Note>
  **Dual-record note:** A user may have multiple stakeholder rows on the same
  entity (personal + team-scoped). The access system resolves the **highest**
  `accessLevel` across all rows. For example, a personal stake with READ and a
  team-scoped stake with FULL resolves to FULL. Single-resource checks use
  `em.find()` + highest-level resolution; bulk checks already iterate all rows.
</Note>

## Lead Creation Assignment Authorization

When creating leads, assignment authorization follows specific rules:

- **Self-assignment:** Always allowed
- **Team member assignment:** Requires team leadership or `team_crm.manage` permission
- **Cross-team assignment:** Requires org-level `crm.manage` permission

```typescript
@Post()
@CheckAccess({ permissions: ['crm.create', 'crm.manage'] })
async createLead(@Body() dto: CreateLeadDto, @GetCurrentUser() user: RequestUser) {
  // Assignment validation happens in service layer
  return this.leadService.createLead(dto, user);
}
```

## Claiming Leads from Team Pool

Team members can claim unassigned leads that belong to their team's pool:

```typescript
@Post(':id/claim')
@CheckAccess({
  permissions: ['crm.create', 'crm.manage'],
  resource: { type: ResourceType.LEAD, action: ResourceAction.ACCESS },
})
async claimLead(@Param('id') id: string, @GetCurrentUser() user: RequestUser) {
  return this.leadService.claimLead(id, user);
}
```

## Frontend Stakeholder UI Permission Gating

The frontend uses response permissions to control UI elements. For stakeholder management specifically:

- **Add stakeholder button:** Shown if user has `crm.manage` or `team_crm.manage`
- **Remove stakeholder button:** Shown if user has management permissions AND stakeholder is removable
- **Edit access level:** Shown if user has management permissions AND can modify the specific stakeholder

# Layer 2: Response Permissions

This layer tells the frontend **what actions the user can perform** on each resource. It runs inside services after data is fetched and attaches a `permissions` object to each resource.

## Which Entities Have Response Permissions

Response permissions are computed for core CRM entities:

- **Leads** - Primary CRM entity
- **Deals** - Sales pipeline entity  
- **Contacts** - Customer contact records
- **Companies** - Business entities
- **Persons** - Individual people
- **Units** - Property/inventory units
- **Teams** - User teams (limited permissions)
- **Conversations** - Messaging conversations

The `PermissionComputeService` centralizes permission computation logic:

```typescript
    organizationId: string,

    // Compute individual permission flags

      canModify,
      canDelete,
      canTransfer,
      canManageStakeholders,
    });
  }

  async bulkComputeLeadPermissions(
    leads: Lead[],
    userId: string, 
    organizationId: string,
  ): Promise<Map<string, LeadPermissionsDto>> {
    // Optimized bulk computation
    const permissions = new Map<string, LeadPermissionsDto>();
    // ... bulk logic
    return permissions;
  }

  async computeConversationPermissions(
    conversationId: string,
    userId: string,
    organizationId: string,
  ): Promise<ConversationPermissionsDto> {
    // Compute conversation-specific permissions
    const canAccess = await this.accessService.hasResourceAccess({
      resource: { type: ResourceType.CONVERSATION, action: ResourceAction.ACCESS },
      userId,
      organizationId,
      resourceId: conversationId,
    });
    
    return new ConversationPermissionsDto({
      canAccess,
      canSendMessage: await this.hasPermission(userId, 'messaging.write'),
      canAssign: await this.hasPermission(userId, 'messaging.manage'),
      canTransfer: await this.hasPermission(userId, 'messaging.manage'),
      canManageSettings: await this.hasPermission(userId, 'messaging.admin'),
    });
  }
}
```

## Integrating Response Permissions Into Services

Services integrate permissions using the `PermissionComputeService`:

```typescript
@Injectable()
export class LeadService {
  constructor(
    private readonly permissionComputeService: PermissionComputeService,
  ) {}

  async findOne(id: string, userId: string, organizationId: string) {
    const lead = await this.em.findOneOrFail(Lead, { id });
    
    const permissions = await this.permissionComputeService.computeLeadPermissions(
      id,
      userId,
      organizationId,
    );

    return {
      data: lead,
      permissions,
    };
  }

  async findAll(userId: string, organizationId: string) {
    const leads = await this.em.find(Lead, {/* query */});
    
    const permissionsMap = await this.permissionComputeService.bulkComputeLeadPermissions(
      leads,
      userId,
      organizationId,
    );

    return {
      data: leads.map(lead => ({
        ...lead,
        permissions: permissionsMap.get(lead.id),
      })),
    };
  }
}
```

```typescript
async bulkArchive(
  ids: string[],
  userId: string,
  organizationId: string,
  bulkAccessContext: BulkAccessContext,
) {
  const allowedIds = bulkAccessContext.allowedResources;
  const deniedIds = bulkAccessContext.deniedResources;
  
  // Process only allowed IDs
  const results = await this.processBulkOperation(allowedIds);
  
  return {
    succeeded: results,
    failed: deniedIds.map(id => ({ id, reason: 'Access denied' })),
  };
}
```

## Complete Controller Example

```typescript
import {
  Controller,
  Get,
  Post,
  Put,
  Delete,
  Param,
  Body,
  Req,
  UseGuards,
} from "@nestjs/common";
import { AuthGuard } from "@/modules/auth/auth.guard";
import { AccessGuard } from "@/modules/auth/guards/access.guard";
import {
  CheckAccess,
  AccessMode,
  ResourceType,
  ResourceAction,
} from "@/modules/auth/decorators";
import { GetCurrentUser } from "@/modules/auth/decorators/get-current-user.decorator";
import { OrgPermissionKey, TeamPermissionKey } from "@/modules/rbac/permission/permission.constants";

@Controller("leads")
@UseGuards(AuthGuard, AccessGuard)
export class LeadController {
  // === List & Create (Permission Only) ===

  @Get()
  @CheckAccess({ permissions: ["crm.view", "crm.manage"] })
  async findAll(@GetCurrentUser() user: RequestUser) {
    return this.leadService.findAll(user);
  }

  @Post()
  @CheckAccess({ permissions: ["crm.create", "crm.manage"] })
  async create(@Body() dto: CreateLeadDto, @GetCurrentUser() user: RequestUser) {
    return this.leadService.create(dto, user);
  }

  // === Single Resource (Permission OR Resource) ===

  @Get(":id")
  @CheckAccess({
    permissions: ["crm.view", "crm.manage"],
    resource: { type: ResourceType.LEAD, action: ResourceAction.ACCESS },
  })
  async findOne(@Param("id") id: string, @GetCurrentUser() user: RequestUser) {
    return this.leadService.findOne(id, user);
  }

  @Put(":id")
  @CheckAccess({
    permissions: ["crm.manage"],
    resource: { type: ResourceType.LEAD, action: ResourceAction.MODIFY },
  })
  async update(
    @Param("id") id: string, 
    @Body() dto: UpdateLeadDto,
    @GetCurrentUser() user: RequestUser,
  ) {
    return this.leadService.update(id, dto, user);
  }

  @CheckAccess({

    return this.leadService.remove(id, user);
  }

  // === Sensitive Operation (Permission AND Resource) ===

  @Put(":id/transfer")
  @CheckAccess({
    permissions: ["crm.transfer"],
    resource: { type: ResourceType.LEAD, action: ResourceAction.MODIFY },
    mode: AccessMode.AND,

  async transfer(
    @Param("id") id: string, 
    @Body() dto: TransferDto,
    @GetCurrentUser() user: RequestUser,
  ) {

  }

  // === Stakeholder Management (Permission Only) ===

  @Post(":id/stakeholders")
  @CheckAccess({

  })
  async createStakeholder(

    @Param("stakeholderId") stakeholderId: string,

  }

  @Delete(":leadId/stakeholders/:stakeholderId")

  })
  async removeStakeholder(
    @Param("leadId") leadId: string,

  // === Bulk Operations ===

  @Post("bulk-archive")
  @CheckAccess({
    permissions: [OrgPermissionKey.CRM_MANAGE],
    bulkResource: {
      type: ResourceType.LEAD,
      action: ResourceAction.MODIFY,
      paramName: "leadIds",
      strategy: "partial",
    },
  })
  async bulkArchive(
    @Req() request: any,
```

## Performance Considerations

<Warning>
**Bulk operations:** Always use bulk permission computation methods rather than individual lookups in loops. The `PermissionComputeService` provides optimized bulk methods that minimize database queries.
</Warning>

<Tip>
**Caching:** Permission computations can be expensive. Consider implementing Redis caching for frequently accessed permissions, especially for list endpoints.
</Tip>

<Info>
**Query optimization:** The access system uses optimized queries with proper indexing. Avoid circumventing the service methods with direct database access.
</Info>

## Implementation Checklists

### Adding access control to a new endpoint

<AccordionGroup>
<Accordion title="Single resource endpoint checklist">
- [ ] Add `@UseGuards(AuthGuard, AccessGuard)` to controller
- [ ] Add `@CheckAccess()` decorator with appropriate permissions
- [ ] Determine if resource check is needed (`resource` config)
- [ ] Choose correct `AccessMode` (OR vs AND)
- [ ] Verify parameter name matches route param
- [ ] Test with different user permission levels
- [ ] Add response permissions computation in service
</Accordion>

<Accordion title="Bulk operation endpoint checklist">
- [ ] Configure `bulkResource` in `@CheckAccess()`
- [ ] Set appropriate `strategy` (partial vs all)
- [ ] Extract `bulkAccessContext` from request
- [ ] Pass context to service method
- [ ] Handle partial success/failure responses
- [ ] Implement proper error reporting for denied resources
- [ ] Test with mixed accessible/inaccessible resources
</Accordion>

<Accordion title="Permission-only endpoint checklist">
- [ ] Use permission-only `@CheckAccess()` (no resource config)
- [ ] Include both org-level and team-level permissions where appropriate
- [ ] Validate business logic constraints in service layer
- [ ] Ensure team-scoped permissions are properly validated
- [ ] Test with users having different permission combinations
</Accordion>
</AccordionGroup>

### Service integration checklist

<AccordionGroup>
<Accordion title="Response permissions integration">
- [ ] Inject `PermissionComputeService` in constructor
- [ ] Use single computation for `findOne` methods
- [ ] Use bulk computation for `findAll` methods  
- [ ] Return standardized `{ data, permissions }` format
- [ ] Handle edge cases (deleted entities, etc.)
- [ ] Test permissions accuracy across different scenarios
</Accordion>

<Accordion title="Bulk operation service integration">
- [ ] Accept `BulkAccessContext` parameter
- [ ] Use `allowedResources` for processing
- [ ] Report `deniedResources` with reasons
- [ ] Return structured success/failure response
- [ ] Log appropriate audit trail information
- [ ] Handle database transaction boundaries properly
</Accordion>
</AccordionGroup>
