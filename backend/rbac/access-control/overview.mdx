description: "Unified two-layer system: endpoint protection (guards) and resource permissions (computed per-resource actions returned in API responses)."
---

The access control and permissions system has **two layers** that work together:

| Layer                    | Purpose                                 | When It Runs                              | Result                                |
| ------------------------ | --------------------------------------- | ----------------------------------------- | ------------------------------------- |
| **Endpoint Protection**  | Decides if a request is allowed         | Before controller handler (NestJS guard)  | 403 Forbidden or allow through        |
| **Response Permissions** | Tells the frontend what the user can do | Inside the service, after data is fetched | `permissions` object on each resource |

**Flow of a request:**

```
Request → AuthGuard (incl. owner-only suspended gate) → AccessGuard (Layer 1: allow or 403)
  → SuspendedWriteInterceptor (org-wide read-only when suspended)
  → Controller → Service → PermissionComputeService (Layer 2: attach permissions)
    → Response with { data, permissions }
```

---

## Layer 0: Suspended-org (billing) access control

Before the RBAC layers below, two billing-state gates run when an org's
subscription is **suspended/canceled** (`isOrgInSuspendedState(status)` =
`SUSPENDED || CANCELED`; see `SUBSCRIPTION_MODULE_SPECIFICATION.md` §9). Both read
status from the `@Global()` `OrgBillingStateCache` (RLS-bypass single-row read — see
`MULTI_TENANCY.md`).

| Gate                   | Where                                                | Effect                                                                                                                                                                |
| ---------------------- | ---------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Owner-only access**  | `AuthGuard` (+ `selectOrganization`)                 | Non-owners (no `system.owner`) get `403 { code: 'ORG_SUSPENDED_NO_ACCESS' }` on EVERY tenant-scoped request; they cannot even select the org. `isSystemAdmin` exempt. |
| **Org-wide read-only** | Global `SuspendedWriteInterceptor` (APP_INTERCEPTOR) | Owners may read but ALL mutating verbs (`POST/PUT/PATCH/DELETE`) return `403 { code: 'SUBSCRIPTION_SUSPENDED' }` UNLESS the handler is `@AllowedWhenSuspended()`.     |

### `@AllowedWhenSuspended()` allowlist

The recovery surface the owner must still reach while suspended. Mutating handlers
NOT on this list are blocked. Public / identity-token / webhook routes (no
`request.user.organizationId`) and all read verbs are naturally exempt and need no
decorator.

| Area               | Endpoints                                                                                                         |
| ------------------ | ----------------------------------------------------------------------------------------------------------------- |
| Billing / recovery | `POST /subscriptions/checkout`, `/checkout/confirm`, `/change-plan`, `/billing-portal`, `/cancel`, `/resume`      |
| Org lifecycle      | `DELETE /organizations/:id` (owner must be able to delete a suspended org)                                        |
| Low-risk UX        | Notification mark-read / mark-all-read / mark-multiple-read, dismiss/delete, and `PUT /notifications/preferences` |

<Info>
Identity-token flows (logout, token refresh, `select-organization`) carry no `organizationId` on `request.user`, so they are exempt automatically. The Stripe `webhook` controller is `@PublicEndpoint` (no user) and is also unaffected.
</Info>

The legacy per-route `@RequiresActiveSubscription()` guard (`SubscriptionActiveGuard`)
throws the same `SUBSCRIPTION_SUSPENDED` code and is left in place but superseded by
this global interceptor for the org-wide read-only goal.

# Layer 1: Endpoint Protection

This layer decides **whether a request is allowed to proceed**. It runs as a NestJS guard before the controller handler. If the user lacks access, the request is rejected with a 403.

---

## Core Concepts

### Access Check Types

| Type                    | Description                                 | Use Case                          |
| ----------------------- | ------------------------------------------- | --------------------------------- |
| **Permission Check**    | RBAC permission (e.g., `crm.manage`)        | List endpoints, create operations |
| **Resource Check**      | Stakeholder-based access to specific entity | Single-entity operations          |
| **Bulk Resource Check** | Batch access check for multiple entities    | Bulk operations                   |

### CRM Directory Visibility

CRM directory endpoints (`/leads/directory`, `/deals/directory`, `/contacts/directory`, `/companies/directory`) are lightweight picker APIs. They use `AuthGuard` + `AccessGuard`, but record-level visibility is enforced inside each service query instead of route-level `@CheckAccess`, because directories must return a filtered page rather than fail the whole request for inaccessible rows.

Lead directory supports `requiredAction=modify` for flows that need editable leads, such as contact creation from an existing lead. That filter mirrors lead response permissions: org/system manage bypass, direct `WRITE`/`FULL` stakeholder access, or `team_crm.manage` on a `WRITE`/`FULL` **team-only** stakeholder row. See `Docs/CRM_DIRECTORY_ACCESS_MATRIX.md` for the full role/permission matrix.

### Nested CRM Related Lists

Contact panel related-list endpoints (`GET /contacts/:id/leads` and `GET /contacts/:id/deals`) are endpoint-gated by Contact access, then service-filtered by each related Lead/Deal's own `canView` policy before pagination and serialization. Inaccessible related rows are excluded from the response body, `total`, contact detail tab counts, and the linked deal value aggregate, so browser network payloads cannot disclose lead/deal IDs or counts that the actor cannot view directly.

The contact panel Units tab (`GET /contacts/:id/units`) is also endpoint-gated by Contact access (`crm.view` OR contact `ACCESS`), but it uses **response-level masking** instead of per-row exclusion: every inventory unit the contact actively owns is returned, and each row is masked individually through the standard inventory `canViewSensitiveDetails` rule (delegated to `InventoryUnitService.findUnitsOwnedByContact`). A caller who can open the contact therefore always sees how many units it owns, but sensitive details may be masked per unit.

### Access Modes

| Mode    | Behavior                                               | Default |
| ------- | ------------------------------------------------------ | ------- |
| **OR**  | User passes if they have permission OR resource access | ✅ Yes  |
| **AND** | User must have BOTH permission AND resource access     | No      |

### System Superuser Bypass

`system.admin` and `system.owner` are treated as organization superuser permissions. These roles have elevated access to system configuration including deal stage management, which requires `system.admin` permission for create, save, update, reorder, and delete operations. `AccessGuard` uses `AccessService.hasSystemBypass()` to check for these roles and allows those users before permission, resource, bulk-resource, or team checks run. `AccessService.getPermissionsAsSet()` expands system bypass users to all org/team permission keys via `addSystemBypassPermissions()` for service-level checks. Programmatic `checkResourceAccess*` and `checkBulkResourceAccess*` calls return `true` immediately for system users when `hasSystemBypass()` is true, ensuring consistency between controller guards and service-level authorization.

**OAuth callback tenant access validation:** `AccessService.assertOAuthCallbackOrgAccessInTransaction()` validates current organization access during OAuth callbacks, supporting optional required permission keys for admin-level flows.

### Permission Implications

RBAC permission checks use the user's **expanded effective permission set**. Higher capability permissions imply same-domain view permissions through the explicit map in `src/modules/shared/permission-implications.ts`; guards and service-level checks do not depend on every controller listing both `*_VIEW` and `*_MANAGE`.

Controller-level `@CheckAccess` metadata and pure service-layer allow/view checks that use expanded permission sets should list the weakest permission that grants the intended operation. For example, a read endpoint should require `crm.view` only; users with `crm.manage` still pass because the guard and expanded-set helpers evaluate implied permissions.

Service helper calls should follow the same weakest-capability pattern when they are pure allow/view or recipient lookups. For example, request `[TeamPermissionKey.TEAM_CRM_VIEW]` from `getPermissionedTeamIdsFromUser()`, `getUserTeamsWithPermissions()`, or `getTeamUsersByPermissions()` when the caller needs team CRM visibility; users with only `team_crm.manage` still match via helper-level implication expansion. DB-backed filters use `expandPermissionQueryKeys()` so raw stored stronger permissions are included while querying for weaker requested permissions. Keep explicit manage checks where manage changes the resulting access level, such as `team_crm.manage -> WRITE` versus `team_crm.view -> READ`.

**Per-team enforcement (no "has it somewhere" shortcuts).** When an endpoint accepts a caller-supplied `teamId`, validate the caller against **that** team, never against "holds the permission/membership on any team".

Examples:

- `crm.manage` satisfies a guard requiring `crm.view`
- `sales.manage` satisfies a guard requiring `sales.view`
- `real_estate.manage` satisfies a guard requiring `real_estate.view`
- `analytics.manage` satisfies a guard requiring `analytics.view`
- `messaging.write` satisfies guards requiring `messaging.view`
- `messaging.manage` satisfies guards requiring `messaging.write` or `messaging.view`
- Team equivalents follow the same explicit map, e.g. `team_crm.manage` satisfies `team_crm.view`
- `system.admin` / `system.owner` satisfy every permission and resource check

This expansion is explicit, not suffix-based. `marketing.manage` and `team.admin` do not create synthetic `*.view` permissions; `system.admin` and `system.owner` are special superuser bypasses rather than ordinary implication-map entries.

**Import Job Access Control:**

Import jobs are tenant-scoped resources created by `POST /leads/import/jobs` and `POST /contacts/import/jobs`. The generic read endpoints use OR-mode `@CheckAccess`:

```typescript
@CheckAccess({
  permissions: [OrgPermissionKey.SYSTEM_ADMIN],
  resource: { type: ResourceType.IMPORT_JOB, action: ResourceAction.ACCESS, paramName: 'id' },
})
```

`system.admin` can inspect any import job in the current organization through the permission branch. Users without `system.admin` must pass the resource branch, which allows `ACCESS` only when `import_job.created_by_id` matches the requesting user.

Import workers must not trust persisted request snapshots. `ImportJobQueueHandler` resolves the actor from `import_job.options.requestUserId` (falling back to legacy `options.requestUser.id` or `created_by_id`), then `ImportJobService.loadCurrentRequestUserForProcessing()` rebuilds the current `RequestUser` from active organization membership and live RBAC data before row processing. If the user was removed from the organization, deactivated, or no longer has permission required by the queued operation, the job is marked failed and no rows are processed.

Contact import is permission-gated twice: `POST /contacts/import/jobs` requires `crm.manage` at enqueue time, and the worker re-checks current `crm.manage` before executing the job. Lead import remains available to authenticated CRM users, but all finance, assignment, and duplicate-update decisions are evaluated from the freshly loaded `RequestUser` at row-processing time.

Lead import duplicate updates are authorized at row-processing time rather than by blocking the job creation endpoint. When a lead import row matches an existing lead and requests `duplicateAction=update`, `LeadImportService` applies the same lead `MODIFY` policy used by normal lead updates: org `crm.manage` bypasses, otherwise the importer needs direct `WRITE`/`FULL` stakeholder access.

**Messaging Resources:**

- `CHANNEL_ACCOUNT` - Channel accounts (WhatsApp, Instagram, Messenger)
- `CONVERSATION` - Individual messaging conversations with contacts

**Messaging Permissions:**

- `messaging.view` - View conversations and messages
- `messaging.write` - Send messages, update conversations, and basic conversation operations  
- `messaging.manage` - Full conversation management including assignment and transfer, channel account management
- `messaging.admin` - Channel account setup and template management

**Channel Account Access Control:**

Channel account resource checks use ownership-based logic:

- **Personal accounts** (`level = PERSONAL`): The account owner can ACCESS, MODIFY, and DELETE their own accounts without needing `messaging.manage` permission.
- **Organization accounts** (`level = ORGANIZATION`): No resource-level access — requires `messaging.manage` permission via RBAC.

The delete endpoint and sync controller both use **OR mode**: users with `messaging.manage` can operate on any account, OR personal account owners can operate on their own accounts.

```typescript
// Delete account — OR mode: messaging.manage OR personal account owner
@Delete(':id')
@CheckAccess({
  permissions: [OrgPermissionKey.MESSAGING_MANAGE],
  resource: { type: ResourceType.CHANNEL_ACCOUNT, action: ResourceAction.DELETE },
})
async delete(@Param('id') id: string) {}

// Sync controller — OR mode: messaging.manage OR personal account owner
@Controller('messaging/sync/:channelAccountId')
@CheckAccess({
  permissions: [OrgPermissionKey.MESSAGING_MANAGE],
  resource: { type: ResourceType.CHANNEL_ACCOUNT, action: ResourceAction.ACCESS, paramName: 'channelAccountId' },
})
```

**Conversation Access Control:**

Conversation access follows a multi-tier system:

- **Tier 1a (Broad permissions)**: Users with any org messaging permission (`MESSAGING_VIEW`, `MESSAGING_WRITE`, or `MESSAGING_MANAGE`) can access all org conversations
- **Tier 1b (Reply permissions)**: Only agents with `canReply=true` assignments may send messages, react to messages, or toggle AI mode
- **Tier 2 (Personal account ownership)**: Users who own the personal channel account
- **Tier 3 (Assignment-based)**: Users with active (non-deleted) assignments
- **Tier 3b (Team membership)**: Users who are members of a team that has an active assignment on the conversation
- **Tier 4 (CRM stakeholder)**: Users who are stakeholders on linked leads/deals
- **Tier 5 (Team leader scope)**: Team leaders can only assign and remove assignments within their own team scope

All conversation-scoped endpoints use `@CheckAccess` with `ResourceType.CONVERSATION` and `ResourceAction.ACCESS`. In **OR mode** (default), users with any org messaging permission bypass the resource check; users without org permissions fall through to the resource check, which uses a single SQL query testing all 5 access paths (personal owner, direct assignment, team assignment, lead stakeholder, deal stakeholder). This eliminates the need for manual `assertConversationAccess` calls in controller methods.

Message search (`GET /messaging/messages/search`) requires `conversationId` as a mandatory query parameter and uses `@CheckAccess` with `ResourceType.CONVERSATION` to enforce access at the guard level.

### Task Calendar Assignee Filtering

The system includes specialized permissions for filtering task calendars by assignee user:

**Policy:** Users may filter task calendars by assignee (`GET /tasks/calendar?userId=`) under these conditions:

- **Self**: Always allowed when `userId` matches the viewer
- **Org-wide managers**: `system.admin` / `system.owner` / `crm.manage` can filter by any active org member
- **Team administrators**: `team.admin` can filter by members of their managed teams
- **Otherwise**: 403 Forbidden

This policy is enforced by `AccessService.canViewerFilterTasksByAssigneeUserInTransaction()` and bulk-resolved via `listTaskCalendarFilterTargetUserIdsInTransaction()`.

### Event Access Control

**Resource Type:** `EVENT`

**Access Model:** Events use a creator-based access model where only the creator of an event can ACCESS, MODIFY, or DELETE that event.

## Single-Resource Endpoints

### @CheckAccess Decorator

The `@CheckAccess` decorator combines permission and resource checks:

```typescript
@CheckAccess({
  permissions: string[],           // RBAC permissions (any match = pass)
  resource?: {
    type: ResourceType,            // Entity type
    action: ResourceAction,        // ACCESS, MODIFY, DELETE
    paramName?: string,            // Route param name (default: 'id')
  },
  mode?: AccessMode,               // OR (default) or AND
})
```

### Usage Patterns

#### Permission Only (List/Create)

```typescript
@Get()
@CheckAccess({ permissions: ['crm.view', 'crm.manage'] })
async findAll() {}

@Post()
@CheckAccess({ permissions: ['crm.create', 'crm.manage'] })
async create(@Body() dto: CreateLeadDto) {}
```

#### Permission OR Resource (View/Update/Delete)

```typescript
@Get(':id')
@CheckAccess({
  permissions: ['crm.view', 'crm.manage'],
  resource: { type: ResourceType.LEAD, action: ResourceAction.ACCESS },
})
async findOne(@Param('id') id: string) {}

@Put(':id')
@CheckAccess({
  permissions: ['crm.manage'],
  resource: { type: ResourceType.LEAD, action: ResourceAction.MODIFY },
})
async update(@Param('id') id: string, @Body() dto: UpdateLeadDto) {}

@Delete(':id')
@CheckAccess({
  permissions: ['crm.manage'],
  resource: { type: ResourceType.LEAD, action: ResourceAction.DELETE },
})
async remove(@Param('id') id: string) {}
```

#### Permission AND Resource (Sensitive Operations)

```typescript
@Put(':id/transfer')
@CheckAccess({
  permissions: ['crm.transfer'],
  resource: { type: ResourceType.LEAD, action: ResourceAction.MODIFY },
  mode: AccessMode.AND,
})
async transfer(@Param('id') id: string, @Body() dto: TransferDto) {}
```

#### Resource Only (No Permission Check)

```typescript
@Get(':id')
@CheckAccess({
  resource: { type: ResourceType.LEAD, action: ResourceAction.ACCESS },
})
async getLeadResourceOnly(@Param('id') id: string) {}
```

#### Custom Parameter Name

```typescript
@Put('transfer/:leadId')
@CheckAccess({
  permissions: ['crm.manage'],
  resource: {
    type: ResourceType.LEAD,
    action: ResourceAction.MODIFY,
    paramName: 'leadId',
  },
})
async transfer(@Param('leadId') leadId: string) {}
```

### Permission-Only with Team Permissions (Stakeholder Management)

For endpoints where resource-level access should NOT grant permission (e.g., managing stakeholders on an entity), use permission-only checks that include both org-level and team-level management permissions:

```typescript
// Stakeholder sync on leads - permission-only, NO resource check
@Put(':id/stakeholders')
@CheckAccess({
  permissions: [OrgPermissionKey.CRM_MANAGE, TeamPermissionKey.TEAM_CRM_MANAGE],
})
async syncStakeholders() {}

// Stakeholder sync on deals - permission-only, NO resource check
@Put(':id/stakeholders')
@CheckAccess({
  permissions: [OrgPermissionKey.SALES_MANAGE, TeamPermissionKey.TEAM_SALES_MANAGE],
})
async syncStakeholders() {}

// Legacy individual endpoints (deprecated, same permissions):
// POST :id/stakeholders, PATCH :id/stakeholders/batch, DELETE :id/stakeholders/:id
```

<Warning>
**Why no resource check?** Having `accessLevel: MODIFY` on an entity (via being a stakeholder) should not grant permission to add, remove, or modify other stakeholders. Stakeholder management is an administrative function that requires explicit RBAC permissions.
</Warning>

**Two-tier access for team managers:**

1. **Guard level:** `TEAM_CRM_MANAGE` / `TEAM_SALES_MANAGE` passes the guard (user has this permission on at least one team).
2. **Controller level:** Business logic further validates:
   - Team has a stakeholder on the specific entity (enforced on `createStakeholder`, `batchUpdate`, `removeStakeholder`)
   - Updates/adjustments only target the team's own stakeholders or team member stakeholders (enforced on `createStakeholder`, `batchUpdate`)
   - Removals only target the team's own stakeholders

## Bulk-Resource Endpoints

For operations on multiple resources, use `bulkResource` configuration:

```typescript
@Post('bulk-archive')
@CheckAccess({
  permissions: [OrgPermissionKey.CRM_MANAGE],
  bulkResource: {
    type: ResourceType.LEAD,
    action: ResourceAction.MODIFY,
    paramName: 'leadIds',
    strategy: 'partial', // Allow partial success
  },
})
async bulkArchive(
  @Req() request: any,
  @Body() body: { leadIds: string[] },
  @GetCurrentUser() user: RequestUser,
) {
  const bulkAccessContext = request.bulkAccessContext;
  return this.leadService.bulkArchiveLeads(
    body.leadIds,
    user.id,
    user.organizationId!,
    bulkAccessContext,
  );
}
```

## Lead Creation Assignment Authorization

When creating leads, assignment authorization follows specific rules:

- **Self-assignment:** Always allowed
- **Team member assignment:** Requires team leadership or `team_crm.manage` permission
- **Cross-team assignment:** Requires org-level `crm.manage` permission

```typescript
@Post()
@CheckAccess({ permissions: ['crm.create', 'crm.manage'] })
async createLead(@Body() dto: CreateLeadDto, @GetCurrentUser() user: RequestUser) {
  // Assignment validation happens in service layer
  return this.leadService.createLead(dto, user);
}
```

## Claiming Leads from Global Pool

Team members can claim unassigned leads that belong to their team's pool:

```typescript
@Post(':id/claim')
@CheckAccess({
  permissions: ['crm.create', 'crm.manage'],
  resource: { type: ResourceType.LEAD, action: ResourceAction.ACCESS },
})
async claimLead(@Param('id') id: string, @GetCurrentUser() user: RequestUser) {
  return this.leadService.claimLead(id, user);
}
```

## Frontend Stakeholder UI Permission Gating

The frontend uses response permissions to control UI elements. For stakeholder management specifically:

- **Add stakeholder button:** Shown if user has `crm.manage` or `team_crm.manage`
- **Remove stakeholder button:** Shown if user has management permissions AND stakeholder is removable
- **Edit access level:** Shown if user has management permissions AND can modify the specific stakeholder

# Layer 2: Response Permissions

This layer tells the frontend **what actions the user can perform** on each resource. It runs inside services after data is fetched and attaches a `permissions` object to each resource.

## Which Entities Have Response Permissions

Response permissions are computed for core CRM entities:

- **Leads** - Primary CRM entity
- **Deals** - Sales pipeline entity  
- **Contacts** - Customer contact records
- **Companies** - Business entities
- **Persons** - Individual people
- **Units** - Property/inventory units
- **Teams** - User teams (limited permissions)
- **Conversations** - Messaging conversations

```typescript
@Injectable()
export class LeadService {

    private readonly permissionComputeService: PermissionComputeService,
  ) {}

  async findOne(id: string, userId: string, organizationId: string) {

      id,
      userId,
      organizationId,
    );

    return {
      data: lead,
      permissions,
    };
  }

  async findAll(userId: string, organizationId: string) {
    const leads = await this.em.find(Lead, {/* query */});
    
    const permissionsMap = await this.permissionComputeService.bulkComputeLeadPermissions(
      leads,
      userId,
      organizationId,
    );

    return {
      data: leads.map(lead => ({
        ...lead,
        permissions: permissionsMap.get(lead.id),
      })),
    };
  }
}
```

# Org-Level Policy Checks

## Data Portability Policy

The Data Portability Policy ensures that data exports comply with data protection regulations and organizational policies. This policy is enforced at the organization level and affects data export capabilities across all entities.

### Policy Implementation

```typescript
export class DataPortabilityPolicyService {
  async checkDataPortabilityPermissions(
    userId: string,
    organizationId: string,
    entityTypes: string[]
  ): Promise<DataPortabilityPermissionsDto> {
    // Check if user has data export permissions
    const canExportPersonalData = await this.hasPermission(userId, 'data.export.personal');
    const canExportOrganizationalData = await this.hasPermission(userId, 'data.export.organizational');
    
    // Apply organization-level restrictions
    const orgPolicy = await this.getOrganizationDataPolicy(organizationId);
    
    return new DataPortabilityPermissionsDto({
      canExportPersonalData: canExportPersonalData && orgPolicy.allowPersonalDataExport,
      canExportOrganizationalData: canExportOrganizationalData && orgPolicy.allowOrganizationalDataExport,
      restrictedEntityTypes: orgPolicy.restrictedEntityTypes,
      retentionPeriod: orgPolicy.retentionPeriod,
    });
  }
}
```

### Usage in Controllers

```typescript
@Get('export/data')
@CheckAccess({
  permissions: ['data.export.personal', 'data.export.organizational'],
})
async exportData(
  @Query() query: DataExportQueryDto,
  @GetCurrentUser() user: RequestUser,
) {
  const permissions = await this.dataPortabilityService.checkDataPortabilityPermissions(
    user.id,
    user.organizationId!,
    query.entityTypes,
  );
  
  if (!permissions.canExportPersonalData && query.includesPersonalData) {
    throw new ForbiddenException('Personal data export not permitted');
  }
  
  return this.dataExportService.exportData(query, user, permissions);
}
```

## Performance Considerations

<Warning>
**Bulk operations:** Always use bulk permission computation methods rather than individual lookups in loops. The `PermissionComputeService` provides optimized bulk methods that minimize database queries.
</Warning>

<Tip>
**Caching:** Permission computations can be expensive. Consider implementing Redis caching for frequently accessed permissions, especially for list endpoints.
</Tip>

<Info>
**Query optimization:** The access system uses optimized queries with proper indexing. Avoid circumventing the service methods with direct database access.
</Info>
