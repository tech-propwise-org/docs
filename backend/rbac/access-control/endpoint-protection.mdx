## Overview

The access control and permissions system has **two layers** that work together:

| Layer | Purpose | When It Runs | Result |
|-------|---------|-------------|--------|
| **Endpoint Protection** | Decides if a request is allowed | Before controller handler (NestJS guard) | 403 Forbidden or allow through |
| **Response Permissions** | Tells the frontend what the user can do | Inside the service, after data is fetched | `permissions` object on each resource |

**Flow of a request:**

```
Request ŌåÆ AuthGuard ŌåÆ AccessGuard (Layer 1: allow or 403)
  ŌåÆ Controller ŌåÆ Service ŌåÆ PermissionComputeService (Layer 2: attach permissions)
    ŌåÆ Response with { data, permissions }
```

This layer decides **whether a request is allowed to proceed**. It runs as a NestJS guard before the controller handler. If the user lacks access, the request is rejected with a 403.

## Core concepts

### Access check types

| Type                    | Description                                 | Use case                          |
| ----------------------- | ------------------------------------------- | --------------------------------- |
| **Permission check**    | RBAC permission (e.g., `crm.manage`)        | List endpoints, create operations |
| **Resource check**      | Stakeholder-based access to specific entity | Single-entity operations          |
| **Bulk resource check** | Batch access check for multiple entities    | Bulk operations                   |

### Access modes

| Mode    | Behavior                                               | Default |
| ------- | ------------------------------------------------------ | ------- |
| **OR**  | User passes if they have permission OR resource access | Ō£ģ Yes  |
| **AND** | User must have BOTH permission AND resource access     | No      |

### Messaging resources and permissions

**Messaging Resources:**
- `CHANNEL_ACCOUNT` - Channel accounts (WhatsApp, Instagram, Messenger)
- `CONVERSATION` - Individual messaging conversations with contacts

**Messaging Permissions:**
- `messaging.view` - View conversations and messages
- `messaging.write` - Send messages, update conversations, and basic conversation operations
- `messaging.manage` - Full conversation management including assignment and transfer, channel account management
- `messaging.admin` - Channel account setup and template management

### Channel account access control

Channel account resource checks use ownership-based logic:

- **Personal accounts** (`level = PERSONAL`): The account owner can ACCESS, MODIFY, and DELETE their own accounts without needing `messaging.manage` permission.
- **Organization accounts** (`level = ORGANIZATION`): No resource-level access ŌĆö requires `messaging.manage` permission via RBAC.

The delete endpoint and sync controller both use **OR mode**: users with `messaging.manage` can operate on any account, OR personal account owners can operate on their own accounts.

```typescript
// Delete account ŌĆö OR mode: messaging.manage OR personal account owner
@Delete(':id')
@CheckAccess({
  permissions: [OrgPermissionKey.MESSAGING_MANAGE],
  resource: { type: ResourceType.CHANNEL_ACCOUNT, action: ResourceAction.DELETE },
})
async delete(@Param('id') id: string) {}

// Sync controller ŌĆö OR mode: messaging.manage OR personal account owner
@Controller('messaging/sync/:channelAccountId')
@CheckAccess({
  permissions: [OrgPermissionKey.MESSAGING_MANAGE],
  resource: { type: ResourceType.CHANNEL_ACCOUNT, action: ResourceAction.ACCESS, paramName: 'channelAccountId' },
})
```

### Conversation access control

Conversation access follows a multi-tier system:
- **Tier 1a (Broad permissions)**: Users with any org messaging permission (`MESSAGING_VIEW`, `MESSAGING_WRITE`, or `MESSAGING_MANAGE`) can access all org conversations
- **Tier 1b (Reply permissions)**: Only agents with `canReply=true` assignments may send messages, react to messages, or toggle AI mode
- **Tier 2 (Personal account ownership)**: Users who own the personal channel account
- **Tier 3 (Assignment-based)**: Users with active (non-deleted) assignments
- **Tier 3b (Team membership)**: Users who are members of a team that has an active assignment on the conversation
- **Tier 4 (CRM stakeholder)**: Users who are stakeholders on linked leads/deals
- **Tier 5 (Team leader scope)**: Team leaders can only assign and remove assignments within their own team scope

All conversation-scoped endpoints use `@CheckAccess` with `ResourceType.CONVERSATION` and `ResourceAction.ACCESS`. In **OR mode** (default), users with any org messaging permission bypass the resource check; users without org permissions fall through to the resource check, which uses a single SQL query testing all 5 access paths (personal owner, direct assignment, team assignment, lead stakeholder, deal stakeholder). This eliminates the need for manual `assertConversationAccess` calls in controller methods.

Message search (`GET /messaging/messages/search`) requires `conversationId` as a mandatory query parameter and uses `@CheckAccess` with `ResourceType.CONVERSATION` to enforce access at the guard level.

## Single-resource endpoints

### @CheckAccess decorator

The `@CheckAccess` decorator combines permission and resource checks:

```typescript
@CheckAccess({
  permissions: string[],           // RBAC permissions (any match = pass)
  resource?: {
    type: ResourceType,            // Entity type
    action: ResourceAction,        // ACCESS, MODIFY, DELETE
    paramName?: string,            // Route param name (default: 'id')
  },
  mode?: AccessMode,               // OR (default) or AND
})
```

### Permission-only with team permissions

For endpoints where resource-level access should NOT grant permission (e.g., managing stakeholders on an entity), use permission-only checks that include both org-level and team-level management permissions:

```typescript
// Stakeholder sync on leads - permission-only, NO resource check
@Put(':id/stakeholders')
@CheckAccess({
  permissions: [OrgPermissionKey.CRM_MANAGE, TeamPermissionKey.TEAM_CRM_MANAGE],
})
async syncStakeholders() {}

// Stakeholder sync on deals - permission-only, NO resource check
@Put(':id/stakeholders')
@CheckAccess({
  permissions: [OrgPermissionKey.SALES_MANAGE, TeamPermissionKey.TEAM_SALES_MANAGE],
})
async syncStakeholders() {}

// Legacy individual endpoints (deprecated, same permissions):
// POST :id/stakeholders, PATCH :id/stakeholders/batch, DELETE :id/stakeholders/:id
```

<Note>
Having `accessLevel: MODIFY` on an entity (via being a stakeholder) should not grant permission to add, remove, or modify other stakeholders. Stakeholder management is an administrative function that requires explicit RBAC permissions.
</Note>

**Two-tier access for team managers:**

1. **Guard level:** `TEAM_CRM_MANAGE` / `TEAM_SALES_MANAGE` passes the guard (user has this permission on at least one team).
2. **Controller level:** Business logic further validates:
   - Team has a stakeholder on the specific entity (enforced on `createStakeholder`, `batchUpdate`, `removeStakeholder`)
   - Updates/adjustments only target the team's own stakeholders or team member stakeholders (enforced on `createStakeholder`, `batchUpdate`)
   - Removals only target the team's own stakeholders (enforced on `removeStakeholder`, `batchUpdate` via `removeStakeholderId`)
   - Budget constraints for commission allocation (`createStakeholder`)
   - Primary restriction: cannot set primary unless a team stakeholder already holds primary (`createStakeholder`, `batchUpdate`)

### Team-scoped permission check (`teamCheck`)

For endpoints that should be accessible to team admins/members within a specific team context, use `teamCheck`. The `permissions` field acts as an org-level bypass:

```typescript
@Get()
@CheckAccess({
  permissions: [OrgPermissionKey.SYSTEM_ADMIN, OrgPermissionKey.CRM_VIEW, OrgPermissionKey.CRM_MANAGE],
  teamCheck: {
    teamPermissions: [TeamPermissionKey.TEAM_ADMIN, TeamPermissionKey.TEAM_CRM_MANAGE],
    teamIdParam: 'teamId',
  },
})
async findAll(@Query('teamId') teamId?: string) {}
```

**Flow:**

1. Check org-level bypass permissions ŌĆö if user has any bypass permission (e.g. `system.admin`, `crm.view`, `crm.manage`), access is granted immediately
2. Extract `teamId` from the request using `teamIdParam`:
   - **If `teamId` present** ŌĆö check if user has any of `teamPermissions` in that specific team
   - **If `teamId` absent** ŌĆö check if user has any of `teamPermissions` in ANY of their teams (any-team fallback). This allows team leaders to access endpoints like user selection dropdowns without specifying a team.
3. If neither bypass nor team check passes ŌĆö throw 403

<Warning>
When `teamCheck` grants access via team permissions (not org bypass), the **controller must scope the returned data to the team**. The guard only authorizes the request ŌĆö it does NOT filter results.
</Warning>

```typescript
const effectiveTeamIds = teamIds
  ? teamIds.split(',').filter(Boolean)
  : teamId
    ? [teamId]
    : undefined;

return this.userService.findAllPaginated({ teamIds: effectiveTeamIds, ... }, orgId);
```

## Bulk-resource endpoints

Bulk endpoints check per-item access in a batch-optimized manner with partial success support. Users with org-level bypass permissions (e.g., `CRM_MANAGE`) can operate on all resources; others are checked per-item via stakeholder access.

### Bulk resource configuration

```typescript
@CheckAccess({
  permissions: [OrgPermissionKey.CRM_MANAGE],
  bulkResource: {
    type: ResourceType.LEAD,
    action: ResourceAction.MODIFY,
    paramName: 'leadIds',
    strategy: 'partial',
  },
})
```

### Strategy options

| Strategy  | Behavior                                                       | Use case                         |
| --------- | -------------------------------------------------------------- | -------------------------------- |
| `partial` | Allow operation, filter inaccessible items                     | Archive, unarchive (recommended) |
| `all`     | Require access to ALL items, reject entire request if any fail | Critical operations              |

### Controller implementation

```typescript
@Post('bulk-archive')
@CheckAccess({
  permissions: [OrgPermissionKey.CRM_MANAGE],
  bulkResource: {
    type: ResourceType.LEAD,
    action: ResourceAction.MODIFY,
    paramName: 'leadIds',

async bulkArchive(
  @Req() request: any,
  @Body() body: { leadIds: string[] },
  @GetCurrentUser() user: RequestUser,
) {
  const bulkAccessContext = request.bulkAccessContext;
  return this.service.bulkArchiveLeads(
    body.leadIds,
    user.id,
    user.organizationId!,
    bulkAccessContext,
  );
}
```

### Service implementation

```typescript
async bulkArchiveLeads(
  leadIds: string[],
  userId: string,
  organizationId: string,
  bulkAccessContext?: BulkAccessContext,
): Promise<{
  success: boolean;
  archivedCount: number;
  failedCount: number;
  failures: Array<{ leadId: string; reason: string }>;
}> {
  const results = {
    success: true,
    archivedCount: 0,
    failedCount: 0,
    failures: [] as Array<{ leadId: string; reason: string }>,
  };

  // Pre-filter inaccessible items
  if (bulkAccessContext?.accessMap && !bulkAccessContext.hasBypassPermission) {
    for (const leadId of leadIds) {
      if (bulkAccessContext.accessMap.get(leadId) === false) {
        results.failures.push({
          leadId,
          reason: 'Access denied - insufficient permissions',
        });
        results.failedCount++;
      }
    }
  }

  // Process only accessible items
  for (const leadId of leadIds) {
    if (results.failures.some(f => f.leadId === leadId)) continue;

    try {
      await this.archiveLead(leadId, userId, organizationId);
      results.archivedCount++;
    } catch (error) {
      results.failures.push({ leadId, reason: error.message });
      results.failedCount++;
    }
  }

  return results;
}
```

### BulkAccessContext type

```typescript
interface BulkAccessContext {
  hasBypassPermission: boolean; // User has org-level permission
  resourceIds: string[]; // Original array of resource IDs from request
  accessMap: Map<string, boolean> | null; // resourceId ŌåÆ hasAccess (null if bypass)
}
```

## Lead creation assignment authorization

The `POST /leads` endpoint has **no `@CheckAccess` decorator** ŌĆö any authenticated user can create leads. However, **server-side validation** in `LeadService.validateLeadAssignmentAuthorization()` enforces assignment rules based on who the lead is assigned to and which team it's assigned to.

### Authorization matrix

| Scenario                                          | Required permission                         | Notes                                                 |
| ------------------------------------------------- | ------------------------------------------- | ----------------------------------------------------- |
| **Self-assignment** (`assignedUserId === userId`) | None                                        | Always allowed for any user                           |
| **Pool lead** (no `assignedUserId`) ŌĆö Global Pool | `crm.manage`                                | Only org admins can create global pool leads          |
| **Pool lead** (no `assignedUserId`) ŌĆö Team Pool   | `team_crm.manage` in target team            | Team managers can create pool leads for their team    |
| **Assign to another user**                        | `crm.manage`                                | Only org admins can assign to others                  |
| **Team validation** (if `teamId` provided)        | User must be member of team OR `crm.manage` | Users can only assign to their own teams unless admin |

### Rules in detail

<AccordionGroup>
<Accordion title="Rule 1 ŌĆö Self-assignment (always allowed)">

## Claiming leads from team pool

**Endpoint:** `POST /leads/:id/stakeholders`

The stakeholder creation endpoint allows users to claim leads from team pools based on their permissions and team membership.

### Authorization matrix for claiming

| User type                                              | Can claim? | Restrictions                                                         |
| ------------------------------------------------------ | ---------- | -------------------------------------------------------------------- |
| **Normal agent** (team member, no special permissions) | Yes        | Can only claim for **themselves** from **their team's pool**         |
| **Team manager** (`team_crm.manage`)                   | Yes        | Can assign to **any team member** from **their managed team's pool** |
| **Org admin** (`crm.manage`)                           | Yes        | Can assign to **anyone** from **any pool** (global or team)          |

## Frontend stakeholder UI permission gating

The frontend enforces stakeholder management permissions to ensure UI elements are only visible to authorized users. Backend-computed `lead.permissions` is the primary source of truth; client-side role checks are used as fallback when `permissions` is not available.

# Layer 2: Response Permissions

This layer tells the frontend **what actions are available** after the request has succeeded. It runs inside the service after data is fetched and attaches a `permissions` object to each resource in the response.

## Which entities have response permissions

The following entities include computed permissions in their API responses:

| Entity | DTO | Permissions Computed |
|--------|-----|-------------------|
| **Leads** | `LeadListDto`, `LeadDetailDto` | `canView`, `canEdit`, `canDelete`, `canAssign`, `canTransfer` |

```typescript
  const canTransfer = canAssign && canEdit;

    canView,

    canDelete,
```

### Batch optimization

For list endpoints, the service provides batch methods to avoid N+1 query problems:

```typescript
async computeLeadPermissionsBatch(
  leads: Lead[],
  user: RequestUser,
): Promise<Map<string, LeadPermissionsDto>> {
  // Get user permissions once
  const userPermissions = await this.getUserPermissionContext(user);
  
  // Batch load stakeholders
  const leadIds = leads.map(l => l.id);
  const stakeholderMap = await this.accessService.findStakeholderMap(
    leadIds,
    user.id,
    ResourceType.LEAD,
  );

  // Compute permissions for each lead
  const permissionsMap = new Map<string, LeadPermissionsDto>();
  
  for (const lead of leads) {
    const stakeholder = stakeholderMap.get(lead.id);
    const permissions = this.computePermissionsFromContext(
      userPermissions,
      stakeholder,
      'LEAD',
    );
    permissionsMap.set(lead.id, permissions);
  }

  return permissionsMap;
}
```

```typescript
@Injectable()
export class LeadService {
  constructor(
    private readonly leadRepository: LeadRepository,
    private readonly permissionComputeService: PermissionComputeService,
  ) {}

  async findOne(id: string, user: RequestUser): Promise<LeadDetailDto> {
    const lead = await this.leadRepository.findOne({
      where: { id },
      relations: ['assignedUser', 'company', 'contact', 'stakeholders'],
    });

    if (!lead) {
      throw new NotFoundException('Lead not found');
    }

    const permissions = await this.permissionComputeService.computeLeadPermissions(
      lead,
      user,
    );

    return {
      id: lead.id,
      title: lead.title,
      status: lead.status,
      assignedUser: lead.assignedUser ? {
        id: lead.assignedUser.id,
        firstName: lead.assignedUser.firstName,
        lastName: lead.assignedUser.lastName,
      } : null,
      company: lead.company ? {
        id: lead.company.id,
        name: lead.company.name,
      } : null,
      stakeholders: lead.stakeholders.map(s => ({
        id: s.id,
        userId: s.userId,
        accessLevel: s.accessLevel,
        commission: s.commission,
        isPrimary: s.isPrimary,
      })),
      permissions,
    };
  }

  async findAllPaginated(
    filters: LeadFiltersDto,
    user: RequestUser,
  ): Promise<PaginatedResponse<LeadListDto>> {
    const queryBuilder = this.leadRepository.createQueryBuilder('lead')
      .leftJoinAndSelect('lead.assignedUser', 'assignedUser')
      .leftJoinAndSelect('lead.company', 'company');

    // Apply filters
    if (filters.status) {
      queryBuilder.andWhere('lead.status = :status', { status: filters.status });
    }

    if (filters.assignedUserId) {
      queryBuilder.andWhere('lead.assignedUserId = :assignedUserId', {
        assignedUserId: filters.assignedUserId,
      });
    }

    // Get paginated results
    const [leads, total] = await queryBuilder
      .orderBy('lead.createdAt', 'DESC')
      .skip((filters.page - 1) * filters.limit)
      .take(filters.limit)
      .getManyAndCount();

    // Batch compute permissions
    const permissionsMap = await this.permissionComputeService.computeLeadPermissionsBatch(
      leads,
      user,
    );

    const leadsWithPermissions: LeadListDto[] = leads.map(lead => ({
      id: lead.id,
      title: lead.title,
      status: lead.status,
      assignedUser: lead.assignedUser ? {
        id: lead.assignedUser.id,
        firstName: lead.assignedUser.firstName,
        lastName: lead.assignedUser.lastName,
      } : null,
      company: lead.company ? {
        id: lead.company.id,
        name: lead.company.name,
      } : null,
      createdAt: lead.createdAt.toISOString(),
      permissions: permissionsMap.get(lead.id)!,
    }));

    return {
      data: leadsWithPermissions,
      pagination: {
        page: filters.page,
        limit: filters.limit,
        total,
        totalPages: Math.ceil(total / filters.limit),
      },
    };
  }
}
```

### Caching strategies

```typescript
// Cache user permissions for the duration of a request
@Injectable()
export class PermissionCacheService {
  private cache = new Map<string, any>();

  async getUserPermissionContext(userId: string, orgId: string) {
    const key = `user:${userId}:org:${orgId}`;
    
    if (!this.cache.has(key)) {
      const context = await this.computeUserPermissionContext(userId, orgId);
      this.cache.set(key, context);
    }
    
    return this.cache.get(key);
  }

  clearCache() {
    this.cache.clear();
  }
}
```

```typescript
// Efficient stakeholder batch loading
async findStakeholderMap(
  resourceIds: string[],
  userId: string,
  resourceType: ResourceType,
): Promise<Map<string, Stakeholder | null>> {
  const stakeholders = await this.stakeholderRepository
    .createQueryBuilder('s')
    .where('s.resourceId IN (:...resourceIds)', { resourceIds })
    .andWhere('s.userId = :userId', { userId })
    .andWhere('s.resourceType = :resourceType', { resourceType })
    .getMany();

  const map = new Map<string, Stakeholder | null>();
  
  // Initialize all resource IDs with null
  resourceIds.forEach(id => map.set(id, null));
  
  // Set actual stakeholder relationships
  stakeholders.forEach(s => map.set(s.resourceId, s));
  
  return map;
}
```
