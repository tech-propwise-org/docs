## Overview

The access control and permissions system has **two layers** that work together:

| Layer | Purpose | When It Runs | Result |
|-------|---------|-------------|--------|
| **Endpoint Protection** | Decides if a request is allowed | Before controller handler (NestJS guard) | 403 Forbidden or allow through |
| **Response Permissions** | Tells the frontend what the user can do | Inside the service, after data is fetched | `permissions` object on each resource |

**Flow of a request:**

```
Request → AuthGuard → AccessGuard (Layer 1: allow or 403)
  → Controller → Service → PermissionComputeService (Layer 2: attach permissions)
    → Response with { data, permissions }
```

This layer decides **whether a request is allowed to proceed**. It runs as a NestJS guard before the controller handler. If the user lacks access, the request is rejected with a 403.

## Core concepts

### Access check types

| Type                    | Description                                 | Use case                          |
| ----------------------- | ------------------------------------------- | --------------------------------- |
| **Permission check**    | RBAC permission (e.g., `crm.manage`)        | List endpoints, create operations |
| **Resource check**      | Stakeholder-based access to specific entity | Single-entity operations          |
| **Bulk resource check** | Batch access check for multiple entities    | Bulk operations                   |

### CRM directory visibility

CRM directory endpoints (`/leads/directory`, `/deals/directory`, `/contacts/directory`, `/companies/directory`) are lightweight picker APIs. They use `AuthGuard` + `AccessGuard`, but record-level visibility is enforced inside each service query instead of route-level `@CheckAccess`, because directories must return a filtered page rather than fail the whole request for inaccessible rows.

Lead directory supports `requiredAction=modify` for flows that need editable leads, such as contact creation from an existing lead. That filter mirrors lead response permissions: org/system manage bypass, direct `WRITE`/`FULL` stakeholder access, or `team_crm.manage` on a `WRITE`/`FULL` team stakeholder row. See `Docs/CRM_DIRECTORY_ACCESS_MATRIX.md` for the full role/permission matrix.

### Nested CRM related lists

Contact panel related-list endpoints (`GET /contacts/:id/leads` and `GET /contacts/:id/deals`) are endpoint-gated by Contact access, then service-filtered by each related Lead/Deal's own `canView` policy before pagination and serialization. Inaccessible related rows are excluded from the response body, `total`, contact detail tab counts, and the linked deal value aggregate, so browser network payloads cannot disclose lead/deal IDs or counts that the actor cannot view directly.

Lead panel deal history (`GET /leads/:id/deals`) follows the same nested-list model. The route is endpoint-gated by Lead access, then service-filtered by each linked Deal's own `canView` policy before pagination. A user with org `sales.view` can see all visible-org linked deals; otherwise the deal must be viewable through creator, deal stakeholder, originating-lead stakeholder, or team-sales stakeholder access. Inaccessible deals are excluded from `data`, `total`, and `totalValue`.

### Access modes

| Mode    | Behavior                                               | Default |
| ------- | ------------------------------------------------------ | ------- |
| **OR**  | User passes if they have permission OR resource access | ✅ Yes  |
| **AND** | User must have BOTH permission AND resource access     | No      |

### System superuser bypass

`system.admin` and `system.owner` are treated as organization superuser permissions. These roles have elevated access to system configuration including deal stage management, which requires `system.admin` permission for create, save, update, reorder, and delete operations. `AccessGuard` uses `AccessService.hasSystemBypass()` to check for these roles and allows those users before permission, resource, bulk-resource, or team checks run. `AccessService.getPermissionsAsSet()` expands system bypass users to all org/team permission keys via `addSystemBypassPermissions()` for service-level checks. Programmatic `checkResourceAccess*` and `checkBulkResourceAccess*` calls return `true` immediately for system users when `hasSystemBypass()` is true, ensuring consistency between controller guards and service-level authorization.

**OAuth callback tenant access validation:** `AccessService.assertOAuthCallbackOrgAccessInTransaction()` validates current organization access during OAuth callbacks, supporting optional required permission keys for admin-level flows.

### Permission implications

RBAC permission checks use the user's **expanded effective permission set**. Higher capability permissions imply same-domain view permissions through the explicit map in `src/modules/shared/permission-implications.ts`; guards and service-level checks do not depend on every controller listing both `*_VIEW` and `*_MANAGE`.

Controller-level `@CheckAccess` metadata and pure service-layer allow/view checks that use expanded permission sets should list the weakest permission that grants the intended operation. For example, a read endpoint should require `crm.view` only; users with `crm.manage` still pass because the guard and expanded-set helpers evaluate implied permissions.

Service helper calls should follow the same weakest-capability pattern when they are pure allow/view or recipient lookups. For example, request `[TeamPermissionKey.TEAM_CRM_VIEW]` from `getPermissionedTeamIdsFromUser()`, `getUserTeamsWithPermissions()`, or `getTeamUsersByPermissions()` when the caller needs team CRM visibility; users with only `team_crm.manage` still match via helper-level implication expansion. DB-backed filters use `expandPermissionQueryKeys()` so raw stored stronger permissions are included while querying for weaker requested permissions. Keep explicit manage checks where manage changes the resulting access level, such as `team_crm.manage -> WRITE` versus `team_crm.view -> READ`.

Examples:

- `crm.manage` satisfies a guard requiring `crm.view`
- `sales.manage` satisfies a guard requiring `sales.view`
- `real_estate.manage` satisfies a guard requiring `real_estate.view`
- `analytics.manage` satisfies a guard requiring `analytics.view`
- `messaging.write` satisfies guards requiring `messaging.view`
- `messaging.manage` satisfies guards requiring `messaging.write` or `messaging.view`
- Team equivalents follow the same explicit map, e.g. `team_crm.manage` satisfies `team_crm.view`
- `system.admin` / `system.owner` satisfy every permission and resource check

This expansion is explicit, not suffix-based. `marketing.manage` and `team.admin` do not create synthetic `*.view` permissions; `system.admin` and `system.owner` are special superuser bypasses rather than ordinary implication-map entries.

**Import Job Access Control:**

Import jobs are tenant-scoped resources created by `POST /leads/import/jobs` and `POST /contacts/import/jobs`. The generic read endpoints use OR-mode `@CheckAccess`:

```typescript
@CheckAccess({
  permissions: [OrgPermissionKey.SYSTEM_ADMIN],
  resource: { type: ResourceType.IMPORT_JOB, action: ResourceAction.ACCESS, paramName: 'id' },
})
```

`system.admin` can inspect any import job in the current organization through the permission branch. Users without `system.admin` must pass the resource branch, which allows `ACCESS` only when `import_job.created_by_id` matches the requesting user.

Import workers must not trust persisted request snapshots. `ImportJobQueueHandler` resolves the actor from `import_job.options.requestUserId` (falling back to legacy `options.requestUser.id` or `created_by_id`), then `ImportJobService.loadCurrentRequestUserForProcessing()` rebuilds the current `RequestUser` from active organization membership and live RBAC data before row processing. If the user was removed from the organization, deactivated, or no longer has permission required by the queued operation, the job is marked failed and no rows are processed.

Contact import is permission-gated twice: `POST /contacts/import/jobs` requires `crm.manage` at enqueue time, and the worker re-checks current `crm.manage` before executing the job. Lead import remains available to authenticated CRM users, but all finance, assignment, and duplicate-update decisions are evaluated from the freshly loaded `RequestUser` at row-processing time.

Lead import duplicate updates are authorized at row-processing time rather than by blocking the job creation endpoint. When a lead import row matches an existing lead and requests `duplicateAction=update`, `LeadImportService` applies the same lead `MODIFY` policy used by normal lead updates: org `crm.manage` bypasses, otherwise the importer needs direct `WRITE`/`FULL` stakeholder access.

## Single-resource endpoints

### @CheckAccess decorator

The `@CheckAccess` decorator combines permission and resource checks:

```typescript
@CheckAccess({
  permissions: string[],           // RBAC permissions (any match = pass)
  resource?: {
    type: ResourceType,            // Entity type
    action: ResourceAction,        // ACCESS, MODIFY, DELETE
    paramName?: string,            // Route param name (default: 'id')
  },
  mode?: AccessMode,               // OR (default) or AND
})
```

### Usage patterns

<Tabs>
<Tab title="Permission only">
Used for list and create endpoints where no specific resource ID is involved.

```typescript

@CheckAccess({ permissions: ['crm.create', 'crm.manage'] })

```

</Tab>

<Tab title="Permission OR resource">

### Permission-only with team permissions

For endpoints where resource-level access should NOT grant permission (e.g., managing stakeholders on an entity), use permission-only checks that include both org-level and team-level management permissions:

```typescript
// Stakeholder sync on leads - permission-only, NO resource check
@Put(':id/stakeholders')
@CheckAccess({
  permissions: [OrgPermissionKey.CRM_MANAGE, TeamPermissionKey.TEAM_CRM_MANAGE],
})
async syncStakeholders() {}

// Stakeholder sync on deals - permission-only, NO resource check
@Put(':id/stakeholders')
@CheckAccess({
  permissions: [OrgPermissionKey.SALES_MANAGE, TeamPermissionKey.TEAM_SALES_MANAGE],
})
async syncStakeholders() {}

// Legacy individual endpoints (deprecated, same permissions):
// POST :id/stakeholders, PATCH :id/stakeholders/batch, DELETE :id/stakeholders/:id
```

<Note>
Having `accessLevel: MODIFY` on an entity (via being a stakeholder) should not grant permission to add, remove, or modify other stakeholders. Stakeholder management is an administrative function that requires explicit RBAC permissions.
</Note>

**Two-tier access for team managers:**

1. **Guard level:** `TEAM_CRM_MANAGE` / `TEAM_SALES_MANAGE` passes the guard (user has this permission on at least one team).
2. **Controller level:** Business logic further validates:
   - Team has a stakeholder on the specific entity (enforced on `createStakeholder`, `batchUpdate`, `removeStakeholder`)
   - Updates/adjustments only target the team's own stakeholders or team member stakeholders (enforced on `createStakeholder`, `batchUpdate`)
   - Removals only target the team's own stakeholders (enforced on `removeStakeholder`, `batchUpdate` via `removeStakeholderId`)
   - Budget constraints for commission allocation (`createStakeholder`)
   - Primary restriction: cannot set primary unless a team stakeholder already holds primary (`createStakeholder`, `batchUpdate`)

### Team-scoped permission check (`teamCheck`)

For endpoints that should be accessible to team admins/members within a specific team context, use `teamCheck`. The `permissions` field acts as an org-level bypass:

```typescript
@Get()
@CheckAccess({
  permissions: [OrgPermissionKey.SYSTEM_ADMIN, OrgPermissionKey.CRM_VIEW, OrgPermissionKey.CRM_MANAGE],
  teamCheck: {
    teamPermissions: [TeamPermissionKey.TEAM_ADMIN, TeamPermissionKey.TEAM_CRM_MANAGE],
    teamIdParam: 'teamId',
  },
})
async findAll(@Query('teamId') teamId?: string) {}
```

**Flow:**

1. Check org-level bypass permissions — if user has any bypass permission (e.g. `system.admin`, `crm.view`, `crm.manage`), access is granted immediately
2. Extract `teamId` from the request using `teamIdParam`:
   - **If `teamId` present** — check if user has any of `teamPermissions` in that specific team
   - **If `teamId` absent** — check if user has any of `teamPermissions` in ANY of their teams (any-team fallback). This allows team leaders to access endpoints like user selection dropdowns without specifying a team.
3. If neither bypass nor team check passes — throw 403

<Warning>
When `teamCheck` grants access via team permissions (not org bypass), the **controller must scope the returned data to the team**. The guard only authorizes the request — it does NOT filter results.
</Warning>

```typescript
const effectiveTeamIds = teamIds
  ? teamIds.split(',').filter(Boolean)
  : teamId
    ? [teamId]
    : undefined;

return this.userService.findAllPaginated({ teamIds: effectiveTeamIds, ... }, orgId);
```

### Open directory endpoint (`GET /users/directory`)

For UI surfaces where any authenticated org member needs to pick a user (e.g. transfer modal, mentions), use the dedicated directory endpoint instead of `GET /users`. It returns only `id`, `firstName`, `lastName`, and `avatarUrl` — no sensitive data — and requires no additional permission beyond being logged in.

```typescript

  return this.userService.findDirectory(user.organizationId, ...);

```

Use `UserApi.getDirectory()` on the frontend whenever you need a permission-free user picker.

## Bulk-resource endpoints

Bulk endpoints check per-item access in a batch-optimized manner with partial success support. Users with org-level bypass permissions (e.g., `CRM_MANAGE`) can operate on all resources; others are checked per-item via stakeholder access.

### Bulk resource configuration

```typescript
@CheckAccess({
  permissions: [OrgPermissionKey.CRM_MANAGE],
  bulkResource: {
    type: ResourceType.LEAD,
    action: ResourceAction.MODIFY,
    paramName: 'leadIds',
    strategy: 'partial',
  },
})
```

### Strategy options

| Strategy  | Behavior                                                       | Use case                         |
| --------- | -------------------------------------------------------------- | -------------------------------- |
| `partial` | Allow operation, filter inaccessible items                     | Archive, unarchive (recommended) |
| `all`     | Require access to ALL items, reject entire request if any fail | Critical operations              |

### Bulk endpoint flow

```
User Request → AccessGuard
                    ↓
        Check bypass permissions?
                    ↓
         ┌──────────┴──────────┐
         │                     │
      YES: Allow all        NO: Batch check
         │                     │
         └─────────┬───────────┘
                   ↓
        Set bulkAccessContext on request
                   ↓
           Controller extracts context
                   ↓
           Service filters items
                   ↓
        Return partial results
```

### Controller implementation

```typescript
@Post('bulk-archive')
@CheckAccess({
  permissions: [OrgPermissionKey.CRM_MANAGE],
  bulkResource: {
    type: ResourceType.LEAD,
    action: ResourceAction.MODIFY,
    paramName: 'leadIds',
    strategy: 'partial',
  },
})
async bulkArchive(
  @Req() request: any,
  @Body() body: { leadIds: string[] },
  @GetCurrentUser() user: RequestUser,
) {
  const bulkAccessContext = request.bulkAccessContext;
  return this.service.bulkArchiveLeads(
    body.leadIds,
    user.id,
    user.organizationId!,
    bulkAccessContext,
  );
}
```

### Service implementation

```typescript
async bulkArchiveLeads(
  leadIds: string[],
  userId: string,
  organizationId: string,
  bulkAccessContext?: BulkAccessContext,
): Promise<{
  success: boolean;
  archivedCount: number;
  failedCount: number;
  failures: Array<{ leadId: string; reason: string }>;
}> {
  const results = {
    success: true,
    archivedCount: 0,
    failedCount: 0,
    failures: [] as Array<{ leadId: string; reason: string }>,
  };

  // Pre-filter inaccessible items
  if (bulkAccessContext?.accessMap && !bulkAccessContext.hasBypassPermission) {
    for (const leadId of leadIds) {
      if (bulkAccessContext.accessMap.get(leadId) === false) {
        results.failures.push({
          leadId,
          reason: 'Access denied - insufficient permissions',
        });
        results.failedCount++;
      }
    }
  }

  // Process only accessible items
  for (const leadId of leadIds) {
    if (results.failures.some(f => f.leadId === leadId)) continue;

    try {
      await this.archiveLead(leadId, userId, organizationId);
      results.archivedCount++;
    } catch (error) {
      results.failures.push({ leadId, reason: error.message });
      results.failedCount++;
    }
  }

  return results;
}
```

### BulkAccessContext type

```typescript
interface BulkAccessContext {
  hasBypassPermission: boolean; // User has org-level permission
  resourceIds: string[]; // Original array of resource IDs from request
  accessMap: Map<string, boolean> | null; // resourceId → hasAccess (null if bypass)
}
```

## Lead creation assignment authorization

The `POST /leads` endpoint has **no `@CheckAccess` decorator** — any authenticated user can create leads. However, **server-side validation** in `LeadService.validateLeadAssignmentAuthorization()` enforces assignment rules based on who the lead is assigned to and which team it's assigned to.

### Authorization matrix

| Scenario                                          | Required permission                         | Notes                                                 |
| ------------------------------------------------- | ------------------------------------------- | ----------------------------------------------------- |
| **Self-assignment** (`assignedUserId === userId`) | None                                        | Always allowed for any user                           |
| **Pool lead** (no `assignedUserId`) — Global Pool | `crm.manage`                                | Only org admins can create global pool leads          |
| **Pool lead** (no `assignedUserId`) — Team Pool   | `team_crm.manage` in target team            | Team managers can create pool leads for their team    |
| **Assign to another user**                        | `crm.manage`                                | Only org admins can assign to others                  |
| **Team validation** (if `teamId` provided)        | User must be member of team OR `crm.manage` | Users can only assign to their own teams unless admin |

### Rules in detail

<AccordionGroup>
<Accordion title="Rule 1 — Self-assignment (always allowed)">
```typescript
if (assignedUserId === user.id) {
  // If team specified, validate user is member of that team (unless crm.manage)
  if (teamId && !hasCrmManage) {
    checkTeamMembership(user.id, teamId); // Must be member
  }
  return; // Authorized
}
```
</Accordion>

<Accordion title="Rule 2 — Pool lead (no assignedUserId)">
```typescript
if (!assignedUserId) {
  if (hasCrmManage) return; // crm.manage → global or team pool allowed

  if (teamId) {
    // Check team_crm.manage in specific team
    if (hasTeamPermission(user.id, teamId, 'team_crm.manage')) {
      return; // Authorized for team pool
    }
    throw ForbiddenException('Need team_crm.manage in this team');
  }

  // No team → trying to create global pool lead without crm.manage
  throw ForbiddenException('Need crm.manage for global pool leads');
}
```
</Accordion>

<Accordion title="Rule 3 — Assign to another user">
```typescript
if (assignedUserId !== user.id) {
  if (!hasCrmManage) {
    throw ForbiddenException('Only crm.manage users can assign to others');
  }
  // crm.manage can assign to anyone, validate team exists if specified
  return; // Authorized
}
```
</Accordion>
</AccordionGroup>

### Frontend assignment behavior

The frontend enforces these rules with dynamic UI behavior:

| User type                                 | Assigned To field                             | Team selection           | Pool lead?             |
| ----------------------------------------- | --------------------------------------------- | ------------------------ | ---------------------- |

## Claiming leads from global pool

**Endpoint:** `POST /leads/:id/stakeholders`

The stakeholder creation endpoint allows users to claim leads from team pools based on their permissions and team membership.

### Authorization matrix for claiming

| User type                                              | Can claim? | Restrictions                                                         |
| ------------------------------------------------------ | ---------- | -------------------------------------------------------------------- |
| **Normal agent** (team member, no special permissions) | Yes        | Can only claim for **themselves** from **their team's pool**         |
| **Team manager** (`team_crm.manage`)                   | Yes        | Can assign to **any team member** from **their managed team's pool** |
| **Org admin** (`crm.manage`)                           | Yes        | Can assign to **anyone** from **any pool** (global or team)          |

## Frontend stakeholder UI permission gating

The frontend enforces stakeholder management permissions to ensure UI elements are only visible to authorized users. Backend-computed `lead.permissions` is the primary source of truth; client-side role checks are used as fallback when `permissions` is not available.

### LeadStakeholdersCard

**File:** `src/components/pages/leads/lead-stakeholders-card.tsx`

The card accepts a `canManageStakeholders` boolean prop (mapped from `lead.permissions?.canAssign`). All management actions are hidden when this is `false`.

| UI element                                            | Gated by                | Who sees it                                  |
| ----------------------------------------------------- | ----------------------- | -------------------------------------------- |
| **Add Stakeholder** button                           | `canManageStakeholders` | `CRM_MANAGE` or `TEAM_CRM_MANAGE` users only |
| **Manage All** button                                | `canManageStakeholders` | `CRM_MANAGE` or `TEAM_CRM_MANAGE` users only |
| **Edit (pencil)** per stakeholder row                | `canManageStakeholders` | `CRM_MANAGE` or `TEAM_CRM_MANAGE` users only |
| **Remove (trash)** per non-primary stakeholder row   | `canManageStakeholders` | `CRM_MANAGE` or `TEAM_CRM_MANAGE` users only |
| Stakeholder list (read-only view)                    | None                    | All users with lead access                   |

**Prop propagation path:**

```

// Single query to check user permissions across all teams
const userTeamPermissions = await this.teamPermissionRepository.find({
  where: { userId: user.id, permission: In(['TEAM_CRM_MANAGE']) },

const stakeholders = await this.stakeholderRepository.find({

    userId: user.id 
  },
});
```
</Accordion>

<Accordion title="Permission caching">
Consider caching user permissions for frequently accessed data:

```typescript
@Injectable()
export class CachedPermissionService {
  constructor(private cacheService: CacheService) {}

  async getUserOrgPermissions(userId: string): Promise<string[]> {
    const cacheKey = `user-org-permissions:${userId}`;
    let permissions = await this.cacheService.get<string[]>(cacheKey);
    
    if (!permissions) {
      permissions = await this.loadUserOrgPermissions(userId);
      await this.cacheService.set(cacheKey, permissions, 300); // 5 min TTL
    }
    
    return permissions;
  }
}
```
</Accordion>

<Accordion title="Selective loading">
Only load relations needed for permission computation:

```typescript
// ✅ Load only what's needed
const leads = await this.leadRepository.find({
  relations: ['stakeholders'], // Only load stakeholders for permission check
  select: ['id', 'title', 'status'], // Only select needed fields
});

// ❌ Avoid loading unnecessary relations
const leads = await this.leadRepository.find({
  relations: ['stakeholders', 'activities', 'notes', 'attachments'], // Too much
});
```
</Accordion>
</AccordionGroup>

## Implementation checklists

### Adding @CheckAccess to a new endpoint

<Steps>
<Step title="Determine access pattern">
- **List/create endpoint?** → Use permission-only check
- **Single resource endpoint?** → Use permission OR resource check
- **Bulk operation?** → Use bulk resource check
- **Administrative function?** → Use permission-only check
- **Team-scoped?** → Add teamCheck configuration
</Step>

<Step title="Choose permissions">
- Identify relevant org permissions (e.g., `CRM_MANAGE`, `SALES_VIEW`)
- Include team permissions if applicable (e.g., `TEAM_CRM_MANAGE`)
- Consider read vs write vs admin permission levels
</Step>

<Step title="Configure resource check">
- Determine resource type (`LEAD`, `DEAL`, `CONVERSATION`, etc.)
- Choose action (`ACCESS`, `MODIFY`)
- Set correct paramName if not `:id`
- Choose access mode (`OR` default, `AND` for sensitive operations)
</Step>

<Step title="Test access scenarios">
- Test with org admin (should have access via permissions)
- Test with team manager (should have access via team permissions)
- Test with stakeholder (should have access via resource check)
- Test with unauthorized user (should get 403)
</Step>
</Steps>

## Complete controller example

Here's a complete controller showing all access control patterns:

```typescript
import { Controller, Get, Post, Put, Delete, Param, Body, Query, UseGuards, Req } from '@nestjs/common';
import { AuthGuard } from '../auth/guards/auth.guard';
import { AccessGuard } from '../auth/guards/access.guard';
import { CheckAccess } from '../auth/decorators/check-access.decorator';
import { GetCurrentUser } from '../auth/decorators/get-current-user.decorator';
import { ResourceType, ResourceAction, OrgPermissionKey, TeamPermissionKey } from '../auth/enums';
import { RequestUser } from '../auth/types';

@Controller('leads')
@UseGuards(AuthGuard, AccessGuard)
export class LeadController {
  constructor(private readonly leadService: LeadService) {}

  // List endpoint - permission only
  @Get()
  @CheckAccess({ permissions: [OrgPermissionKey.CRM_VIEW, OrgPermissionKey.CRM_MANAGE] })
  async findAll(@Query() filters: LeadFilters, @GetCurrentUser() user: RequestUser) {
    return this.leadService.findAllPaginated(filters, user);
  }

  // Create endpoint - permission only (no specific resource)
  @Post()
  @CheckAccess({ permissions: [OrgPermissionKey.CRM_CREATE, OrgPermissionKey.CRM_MANAGE] })
  async create(@Body() dto: CreateLeadDto, @GetCurrentUser() user: RequestUser) {
    return this.leadService.create(dto, user);
  }

  // View endpoint - permission OR resource access
  @Get(':id')
  @CheckAccess({
    permissions: [OrgPermissionKey.CRM_VIEW, OrgPermissionKey.CRM_MANAGE],
    resource: { type: ResourceType.LEAD, action: ResourceAction.ACCESS },
  })
  async findOne(@Param('id') id: string, @GetCurrentUser() user: RequestUser) {
    return this.leadService.findById(id, user);
  }

  // Update endpoint - permission OR resource access
  @Put(':id')
  @CheckAccess({
    permissions: [OrgPermissionKey.CRM_MANAGE],
    resource: { type: ResourceType.LEAD, action: ResourceAction.MODIFY },
  })
  async update(
    @Param('id') id: string,
    @Body() dto: UpdateLeadDto,
    @GetCurrentUser() user: RequestUser,
  ) {
    return this.leadService.update(id, dto, user);
  }

  // Delete endpoint - permission OR resource access
  @Delete(':id')
  @CheckAccess({
    permissions: [OrgPermissionKey.CRM_MANAGE],
    resource: { type: ResourceType.LEAD, action: ResourceAction.DELETE },
  })
  async remove(@Param('id') id: string, @GetCurrentUser() user: RequestUser) {
    return this.leadService.remove(id, user);
  }

  // Stakeholder management - permission only (no resource access)
  @Put(':id/stakeholders')
  @CheckAccess({
    permissions: [OrgPermissionKey.CRM_MANAGE, TeamPermissionKey.TEAM_CRM_MANAGE],
  })
  async syncStakeholders(
    @Param('id') id: string,
    @Body() dto: SyncStakeholdersDto,
    @GetCurrentUser() user: RequestUser,
  ) {
    return this.leadService.syncStakeholders(id, dto, user);
  }

  // Bulk operation - partial strategy
  @Post('bulk-archive')
  @CheckAccess({
    permissions: [OrgPermissionKey.CRM_MANAGE],
    bulkResource: {
      type: ResourceType.LEAD,
      action: ResourceAction.MODIFY,
      paramName: 'leadIds',
      strategy: 'partial',
    },
  })
  async bulkArchive(
    @Body() dto: { leadIds: string[] },
    @GetCurrentUser() user: RequestUser,
    @Req() request: any,
  ) {
    const bulkAccessContext = request.bulkAccessContext;
    return this.leadService.bulkArchive(dto.leadIds, user, bulkAccessContext);
  }

  // Team-scoped endpoint
  @Get('team-leads')
  @CheckAccess({
    permissions: [OrgPermissionKey.CRM_VIEW, OrgPermissionKey.CRM_MANAGE],
    teamCheck: {
      teamPermissions: [TeamPermissionKey.TEAM_CRM_MANAGE],
      teamIdParam: 'teamId',
    },
  })
  async findTeamLeads(
    @Query('teamId') teamId: string,
    @Query() filters: LeadFilters,
    @GetCurrentUser() user: RequestUser,
  ) {
    return this.leadService.findTeamLeads(teamId, filters, user);
  }

  // Sensitive operation requiring both permission AND resource access
  @Put(':id/transfer-ownership')
  @CheckAccess({
    permissions: [OrgPermissionKey.CRM_TRANSFER],
    resource: { type: ResourceType.LEAD, action: ResourceAction.MODIFY },
    mode: AccessMode.AND,
  })
  async transferOwnership(
    @Param('id') id: string,
    @Body() dto: TransferOwnershipDto,
    @GetCurrentUser() user: RequestUser,
  ) {
    return this.leadService.transferOwnership(id, dto, user);
  }
}
```
