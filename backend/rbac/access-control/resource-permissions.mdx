The access control and permissions system has **two layers** that work together:

| Layer | Purpose | When It Runs | Result |
|-------|---------|-------------|--------|
| **Endpoint Protection** | Decides if a request is allowed | Before controller handler (NestJS guard) | 403 Forbidden or allow through |
| **Response Permissions** | Tells the frontend what the user can do | Inside the service, after data is fetched | `permissions` object on each resource |

**Flow of a request:**

```
Request → AuthGuard → AccessGuard (Layer 1: allow or 403)
  → Controller → Service → PermissionComputeService (Layer 2: attach permissions)
    → Response with { data, permissions }
```

# Layer 1: Endpoint Protection

This layer decides **whether a request is allowed to proceed**. It runs as a NestJS guard before the controller handler. If the user lacks access, the request is rejected with a 403.

## Core concepts

### Access check types

| Type                    | Description                                 | Use Case                          |
| ----------------------- | ------------------------------------------- | --------------------------------- |
| **Permission Check**    | RBAC permission (e.g., `crm.manage`)        | List endpoints, create operations |
| **Resource Check**      | Stakeholder-based access to specific entity | Single-entity operations          |
| **Bulk Resource Check** | Batch access check for multiple entities    | Bulk operations                   |

### Access modes

| Mode    | Behavior                                               | Default |
| ------- | ------------------------------------------------------ | ------- |
| **OR**  | User passes if they have permission OR resource access | ✅ Yes  |
| **AND** | User must have BOTH permission AND resource access     | No      |

**Messaging Resources:**

- `CHANNEL_ACCOUNT` - Channel accounts (WhatsApp, Instagram, Messenger)
- `CONVERSATION` - Individual messaging conversations with contacts

**Messaging Permissions:**

- `messaging.view` - View conversations and messages
- `messaging.write` - Send messages, update conversations, and basic conversation operations  
- `messaging.manage` - Full conversation management including assignment and transfer, channel account management
- `messaging.admin` - Channel account setup and template management

**Channel Account Access Control:**

Channel account resource checks use ownership-based logic:

- **Personal accounts** (`level = PERSONAL`): The account owner can ACCESS, MODIFY, and DELETE their own accounts without needing `messaging.manage` permission.
- **Organization accounts** (`level = ORGANIZATION`): No resource-level access — requires `messaging.manage` permission via RBAC.

The delete endpoint and sync controller both use **OR mode**: users with `messaging.manage` can operate on any account, OR personal account owners can operate on their own accounts.

```typescript
// Delete account — OR mode: messaging.manage OR personal account owner
@Delete(':id')
@CheckAccess({
  permissions: [OrgPermissionKey.MESSAGING_MANAGE],
  resource: { type: ResourceType.CHANNEL_ACCOUNT, action: ResourceAction.DELETE },
})
async delete(@Param('id') id: string) {}

// Sync controller — OR mode: messaging.manage OR personal account owner
@Controller('messaging/sync/:channelAccountId')
@CheckAccess({
  permissions: [OrgPermissionKey.MESSAGING_MANAGE],
  resource: { type: ResourceType.CHANNEL_ACCOUNT, action: ResourceAction.ACCESS, paramName: 'channelAccountId' },
})
```

**Conversation Access Control:**

Conversation access follows a multi-tier system:
- **Tier 1a (Broad permissions)**: Users with any org messaging permission (`MESSAGING_VIEW`, `MESSAGING_WRITE`, or `MESSAGING_MANAGE`) can access all org conversations
- **Tier 1b (Reply permissions)**: Only agents with `canReply=true` assignments may send messages, react to messages, or toggle AI mode
- **Tier 2 (Personal account ownership)**: Users who own the personal channel account
- **Tier 3 (Assignment-based)**: Users with active (non-deleted) assignments
- **Tier 3b (Team membership)**: Users who are members of a team that has an active assignment on the conversation
- **Tier 4 (CRM stakeholder)**: Users who are stakeholders on linked leads/deals
- **Tier 5 (Team leader scope)**: Team leaders can only assign and remove assignments within their own team scope

All conversation-scoped endpoints use `@CheckAccess` with `ResourceType.CONVERSATION` and `ResourceAction.ACCESS`. In **OR mode** (default), users with any org messaging permission bypass the resource check; users without org permissions fall through to the resource check, which uses a single SQL query testing all 5 access paths (personal owner, direct assignment, team assignment, lead stakeholder, deal stakeholder). This eliminates the need for manual `assertConversationAccess` calls in controller methods.

Message search (`GET /messaging/messages/search`) requires `conversationId` as a mandatory query parameter and uses `@CheckAccess` with `ResourceType.CONVERSATION` to enforce access at the guard level.

## Single-resource endpoints

### @CheckAccess decorator

The `@CheckAccess` decorator combines permission and resource checks:

```typescript
@CheckAccess({
  permissions: string[],           // RBAC permissions (any match = pass)
  resource?: {
    type: ResourceType,            // Entity type
    action: ResourceAction,        // ACCESS, MODIFY, DELETE
    paramName?: string,            // Route param name (default: 'id')
  },
  mode?: AccessMode,               // OR (default) or AND
})
```

### Usage patterns

#### Permission only (list/create)

```typescript
@Get()
@CheckAccess({ permissions: ['crm.view', 'crm.manage'] })
async findAll() {}

@Post()
@CheckAccess({ permissions: ['crm.create', 'crm.manage'] })
async create(@Body() dto: CreateLeadDto) {}
```

#### Permission OR resource (view/update/delete)


### Permission-only with team permissions (stakeholder management)

For endpoints where resource-level access should NOT grant permission (e.g., managing stakeholders on an entity), use permission-only checks that include both org-level and team-level management permissions:

```typescript
// Stakeholder sync on leads - permission-only, NO resource check
@Put(':id/stakeholders')
@CheckAccess({
  permissions: [OrgPermissionKey.CRM_MANAGE, TeamPermissionKey.TEAM_CRM_MANAGE],
})
async syncStakeholders() {}

// Stakeholder sync on deals - permission-only, NO resource check
@Put(':id/stakeholders')
@CheckAccess({
  permissions: [OrgPermissionKey.SALES_MANAGE, TeamPermissionKey.TEAM_SALES_MANAGE],
})
async syncStakeholders() {}

// Legacy individual endpoints (deprecated, same permissions):
// POST :id/stakeholders, PATCH :id/stakeholders/batch, DELETE :id/stakeholders/:id
```

<Info>
**Why no resource check?** Having `accessLevel: MODIFY` on an entity (via being a stakeholder) should not grant permission to add, remove, or modify other stakeholders. Stakeholder management is an administrative function that requires explicit RBAC permissions.
</Info>

**Two-tier access for team managers:**

1. **Guard level:** `TEAM_CRM_MANAGE` / `TEAM_SALES_MANAGE` passes the guard (user has this permission on at least one team).
2. **Controller level:** Business logic further validates:
   - Team has a stakeholder on the specific entity (enforced on `createStakeholder`, `batchUpdate`, `removeStakeholder`)
   - Updates/adjustments only target the team's own stakeholders or team member stakeholders (enforced on `createStakeholder`, `batchUpdate`)
   - Removals only target the team's stakeholders

## Bulk-resource endpoints

For bulk operations (batch updates, bulk transfers), use the `BulkAccessContext` pattern:

```typescript
@Put('bulk')
@CheckAccess({ permissions: ['crm.manage'] })
async bulkUpdate(
  @Body() dto: BulkUpdateLeadsDto,
  @CurrentUser() user: RequestUser,
) {
  // Controller validates bulk resource access before proceeding
  const context = new BulkAccessContext({
    user,
    resourceType: ResourceType.LEAD,
    action: ResourceAction.MODIFY,
    resourceIds: dto.leadIds,
  });

  return this.leadService.bulkUpdate(context, dto);
}
```

### BulkAccessContext type definition

```typescript
export class BulkAccessContext {
  user: RequestUser;
  resourceType: ResourceType;
  action: ResourceAction;
  resourceIds: string[];

  constructor(data: {
    user: RequestUser;
    resourceType: ResourceType;
    action: ResourceAction;
    resourceIds: string[];
  }) {
    this.user = data.user;
    this.resourceType = data.resourceType;
    this.action = data.action;
    this.resourceIds = data.resourceIds;
  }
}
```

### Bulk access validation patterns

<Steps>
<Step title="Guard-level permission check">
The `@CheckAccess` decorator validates the user has the required permission to perform bulk operations.
</Step>

<Step title="Service-level resource validation">
The service uses `AccessService.validateBulkAccess()` to check each resource:

```typescript
async bulkUpdate(context: BulkAccessContext, dto: BulkUpdateDto) {
  return this.tenantContext.executeReadWrite(context.user.organizationId, async (em) => {
    // Validate bulk resource access
    await this.accessService.validateBulkAccess(em, context);
    
    // Proceed with bulk operation
    return this.performBulkUpdate(em, dto);
  });
}
```
</Step>

<Step title="Batch access optimization">
Bulk access checks are optimized to minimize database queries:
- Single query to load all required resources
- Batch stakeholder access checks
- Early termination on first access failure
</Step>
</Steps>

## Access rules by resource type

### Lead access rules

| Action | Permission Required | Resource Access | Notes |
|--------|-------------------|-----------------|-------|
| **List** | `crm.view` OR `crm.manage` | N/A | Shows user's accessible leads only |
| **Create** | `crm.create` OR `crm.manage` | N/A | Creates with user as primary stakeholder |
| **View** | `crm.view` OR `crm.manage` | Stakeholder OR creator | OR mode (either passes) |
| **Update** | `crm.manage` | Stakeholder with MODIFY+ | OR mode |
| **Delete** | `crm.manage` | Stakeholder with DELETE | OR mode |
| **Transfer** | `crm.transfer` | Stakeholder with MODIFY+ | AND mode required |
| **Assign Stakeholders** | `crm.manage` OR `team_crm.manage` | None | Permission-only check |

### Deal access rules

| Action | Permission Required | Resource Access | Notes |
|--------|-------------------|-----------------|-------|
| **List** | `sales.view` OR `sales.manage` | N/A | Shows user's accessible deals only |
| **Create** | `sales.create` OR `sales.manage` | N/A | Creates with user as primary stakeholder |
| **View** | `sales.view` OR `sales.manage` | Stakeholder OR creator | OR mode |
| **Update** | `sales.manage` | Stakeholder with MODIFY+ | OR mode |
| **Delete** | `sales.manage` | Stakeholder with DELETE | OR mode |
| **Transfer** | `sales.transfer` | Stakeholder with MODIFY+ | AND mode required |
| **Assign Stakeholders** | `sales.manage` OR `team_sales.manage` | None | Permission-only check |

These entities have **derived access** from their related Leads/Deals:

| Action | Permission Required | Resource Access | Notes |
|--------|-------------------|-----------------|-------|
| **List** | `crm.view` OR `crm.manage` | N/A | Shows related to accessible leads/deals |
| **View** | `crm.view` OR `crm.manage` | Related lead/deal access | Computed from relationships |
| **Update** | `crm.manage` | Related lead/deal MODIFY access | Computed from relationships |
| **Delete** | `crm.manage` | Related lead/deal DELETE access | Computed from relationships |

### Person, Unit & Team access rules

| Resource | Access Pattern | Notes |
|----------|---------------|-------|
| **Person** | Creator-based + admin permissions | Users can view/edit their own person record |
| **Unit** | Admin-only (`system.admin`, `system.owner`) | Organizational structure management |
| **Team** | Membership-based + admin permissions | Users can view teams they belong to |

## Lead creation assignment authorization

When creating leads, the system follows specific assignment rules:

<Steps>
<Step title="Direct assignment validation">
```typescript
// User can assign to themselves (always allowed)
if (assignToUserId === user.id) return true;

// User can assign to team members if they have team management permission
if (await this.hasTeamManagementPermission(user, targetTeam)) {
  return this.isUserTeamMember(assignToUserId, targetTeam);
}
```
</Step>

<Step title="Team pool assignment">
```typescript
// User can assign to team pool if they manage the team or have org-level permissions
const canAssignToTeam = 
  await this.hasOrgPermission(user, 'crm.manage') ||
  await this.hasTeamPermission(user, targetTeam, 'team_crm.manage');

if (canAssignToTeam && dto.assignToTeam === targetTeam.id) return true;
```
</Step>

<Step title="Default assignment fallback">
```typescript
// If no explicit assignment, assign to user as primary stakeholder
if (!dto.assignToUserId && !dto.assignToTeam) {
  dto.assignToUserId = user.id;
}
```
</Step>
</Steps>

### Assignment authorization matrix

| User Role | Can Assign To Self | Can Assign To Team Members | Can Assign To Team Pool |
|-----------|-------------------|----------------------------|------------------------|
| **Team Manager** | ✅ Yes | ✅ Yes (own team only) | ✅ Yes (managed teams) |
| **Org CRM Manager** | ✅ Yes | ✅ Yes (any team) | ✅ Yes (any team) |
| **Team Member** | ✅ Yes | ❌ No | ❌ No |

## Claiming leads from team pool

Team pool claiming follows a specific authorization pattern:

```typescript
@Put(':id/claim')
@CheckAccess({
  permissions: ['crm.view', 'crm.manage'],
  resource: { type: ResourceType.LEAD, action: ResourceAction.ACCESS },
})
async claimLead(@Param('id') id: string, @CurrentUser() user: RequestUser) {
  // Additional business logic validation in controller:
  // 1. Lead must be in team pool (no primary stakeholder)
  // 2. User must be member of the assigned team
  // 3. Lead must not be archived
  return this.leadService.claimFromTeamPool(user, id);
}
```

<Note>
The guard validates basic access, but the service performs additional business rule validation for team membership and lead status.
</Note>

### Claiming validation rules

<Steps>
<Step title="Guard validation">
User must have `crm.view` or `crm.manage` permission AND access to the lead (as team member of assigned team).
</Step>

<Step title="Business rule validation">
```typescript
async claimFromTeamPool(user: RequestUser, leadId: string) {
  const lead = await this.findLeadById(leadId);
  
  // Must be in team pool (no primary stakeholder)
  if (lead.primaryStakeholder) {
    throw new ConflictException('Lead is already assigned');
  }
  
  // User must be team member
  if (!lead.assignedTeam || !user.teams.includes(lead.assignedTeam.id)) {
    throw new ForbiddenException('Not authorized to claim this lead');
  }
  
  // Must not be archived
  if (lead.archived) {
    throw new ConflictException('Cannot claim archived lead');
  }
}
```
</Step>

<Step title="Assignment creation">
Creates primary stakeholder record with user as assignee and full access level.
</Step>
</Steps>

# Layer 2: Response Permissions

This layer computes **what actions the current user can perform** on each resource and attaches that information to the API response. The frontend reads these permissions directly for UI decisions (show/hide buttons, enable/disable actions) without duplicating access logic.

## Which entities have response permissions

### Entities WITH resource permissions

These are **CRM business entities** with complex access rules involving stakeholders, teams, and creators:

| Entity | Has `permissions`? | Has `assignedTeam`? | Reason |
|--------|-------------------|---------------------|--------|
| **Lead** | Yes | Yes | Has EntityStakeholder, team assignment, creator access |
| **Deal** | Yes | Yes | Has EntityStakeholder, team assignment, creator access |
| **Contact** | Yes | No | Derived access from related Leads/Deals |
| **Company** | Yes | No | Derived access from related Contacts/Deals |

### Entities WITHOUT resource permissions

These are **configuration/admin entities** with simple permission-based access:

| Entity | Why no resource permissions |
|--------|----------------------------|
| User | Admin-only access (`system.admin`) |
| Team | Admin or team membership access |
| Role | Admin-only access (`system.admin`, `system.owner`) |
| Commission Payment | Recipient-based or `sales.manage` permission |
| Entity Transfer | Sender/recipient-based access |
| Invitation | Admin-only access |
| Activity | Read-only timeline data |
| Stage History | Read-only audit data |

## PermissionComputeService

**Location:** `src/modules/auth/services/permission-compute.service.ts`

This service computes permissions using the existing `AccessService` methods:

```typescript
@Injectable()
export class PermissionComputeService {
  constructor(private readonly accessService: AccessService) {}

  // Single resource methods
  async computeLeadPermissionsInTransaction(
    em: EntityManager, 
    user: RequestUser, 
    leadId: string
  ): Promise<ResourcePermissionsDto> {
    const [canAccess, canModify, canDelete] = await Promise.all([
      this.accessService.checkResourceAccess(em, user, ResourceType.LEAD, ResourceAction.ACCESS, leadId),
      this.accessService.checkResourceAccess(em, user, ResourceType.LEAD, ResourceAction.MODIFY, leadId),
      this.accessService.checkResourceAccess(em, user, ResourceType.LEAD, ResourceAction.DELETE, leadId),
    ]);

    const canAssign = canAccess && this.canManageStakeholders(user);

    return new ResourcePermissionsDto({
      canView: canAccess,
      canEdit: canModify,
      canDelete: canDelete,
      canAssign: canAssign,
      canTransfer: canModify,
      canArchive: canDelete,
      canUnarchive: canDelete,
    });
  }

    em: EntityManager, 
    user: RequestUser, 
    dealId: string
  ): Promise<ResourcePermissionsDto>;

    em: EntityManager, 
    leadIds: string[]
  ): Promise<Map<string, TeamBasicDto>>;

  async getDealAssignedTeamInTransaction(
    em: EntityManager, 
    dealId: string
  ): Promise<TeamBasicDto | null>;

  async getBulkDealAssignedTeamsInTransaction(
    em: EntityManager, 
    dealIds: string[]
  ): Promise<Map<string, TeamBasicDto>>;

  // Helper methods
  private canManageStakeholders(user: RequestUser): boolean {
    return user.hasAnyPermission([
      'crm.manage', 
      'sales.manage', 
      'system.admin', 
      'system.owner',
      'team_crm.manage',
      'team_sales.manage'
    ]);
  }
}
```

**Permission derivation rules:**

- `canAssign` = `canView && canManageStakeholders(user)` where `canManageStakeholders` checks for `crm.manage`, `system.admin`, `system.owner`, or `team_crm.manage`
- `canTransfer` = `canEdit`
- `canArchive` = `canDelete`
- `canUnarchive` = `canDelete`
- For Contacts/Companies: no `canAssign` or `canTransfer` (these are derived-access entities)

### `enrichLeadsWithPermissions()`

The `LeadService` helper computes permissions and fetches primary stakeholders in an optimized pass:

```typescript
private async enrichLeadsWithPermissions(
  em: EntityManager,
  user: any,
  leads: Lead[],
): Promise<LeadDto[]> {
  if (!leads || leads.length === 0) return [];

  const leadIds = leads.map((lead) => lead.id);

  const { permissions: permissionsMap } =
    await this.permissionComputeService.computeAllBulkLeadPermissionsWithTeamsInTransaction(
      em, user, leadIds, leads,
    );

  const primaryStakeholdersMap =
    await this.stakeholderService.getBulkPrimaryStakeholdersInTransaction(
      em, StakeholderEntityType.LEAD, leadIds,
    );

  return leads.map((lead) =>
    new LeadDto(lead, {
      permissions: permissionsMap.get(lead.id),
      primaryStakeholder: primaryStakeholdersMap.has(lead.id)
        ? new EntityStakeholderDto(primaryStakeholdersMap.get(lead.id)!)
        : undefined,
    }),
  );
}
```

## Complete controller example

Here's a complete controller showing both layers in action:

```typescript
@Controller('leads')
@UseGuards(AuthGuard, AccessGuard)
export class LeadController {
  constructor(private readonly leadService: LeadService) {}

  // Layer 1: Permission check only (list)
  @Get()
  @CheckAccess({ permissions: ['crm.view', 'crm.manage'] })
  async findAll(@Query() query: FindLeadsDto, @CurrentUser() user: RequestUser) {
    // Layer 2: Service attaches permissions to each lead
    return this.leadService.findAll(user, query);
  }

  // Layer 1: Permission check only (create)
  @Post()
  @CheckAccess({ permissions: ['crm.create', 'crm.manage'] })
  async create(@Body() dto: CreateLeadDto, @CurrentUser() user: RequestUser) {
    return this.leadService.create(user, dto);
  }

  // Layer 1: Permission OR resource check (view)
  @Get(':id')
  @CheckAccess({
    permissions: ['crm.view', 'crm.manage'],
    resource: { type: ResourceType.LEAD, action: ResourceAction.ACCESS },
  })
  async findOne(@Param('id') id: string, @CurrentUser() user: RequestUser) {
    // Layer 2: Service attaches permissions to lead
    return this.leadService.findOne(user, id);
  }

  // Layer 1: Permission OR resource check (update)
  @Put(':id')
  @CheckAccess({
    permissions: ['crm.manage'],
    resource: { type: ResourceType.LEAD, action: ResourceAction.MODIFY },
  })
  async update(
    @Param('id') id: string,
    @Body() dto: UpdateLeadDto,
    @CurrentUser() user: RequestUser,
  ) {
    return this.leadService.update(user, id, dto);
  }

  // Layer 1: Permission AND resource check (transfer - sensitive)
  @Put(':id/transfer')
  @CheckAccess({
    permissions: ['crm.transfer'],
    resource: { type: ResourceType.LEAD, action: ResourceAction.MODIFY },
    mode: AccessMode.AND,
  })
  async transfer(
    @Param('id') id: string,
    @Body() dto: TransferLeadDto,
    @CurrentUser() user: RequestUser,
  ) {
    return this.leadService.transfer(user, id, dto);
  }

  // Layer 1: Permission only (stakeholder management)
  @Post(':id/stakeholders')
  @CheckAccess({
    permissions: ['crm.manage', 'team_crm.manage'],
  })
  async addStakeholder(
    @Param('id') id: string,
    @Body() dto: CreateStakeholderDto,
    @CurrentUser() user: RequestUser,
  ) {
    return this.leadService.addStakeholder(user, id, dto);
  }

  // Layer 1: Bulk operation with runtime validation
  @Put('bulk')
  @CheckAccess({ permissions: ['crm.manage'] })
  async bulkUpdate(
    @Body() dto: BulkUpdateLeadsDto,
    @CurrentUser() user: RequestUser,
  ) {
    const context = new BulkAccessContext({
      user,
      resourceType: ResourceType.LEAD,
      action: ResourceAction.MODIFY,
      resourceIds: dto.leadIds,
    });

    return this.leadService.bulkUpdate(context, dto);
  }

  // Layer 1: Claiming from team pool
  @Put(':id/claim')
  @CheckAccess({
    permissions: ['crm.view', 'crm.manage'],
    resource: { type: ResourceType.LEAD, action: ResourceAction.ACCESS },
  })
  async claimLead(@Param('id') id: string, @CurrentUser() user: RequestUser) {
    return this.leadService.claimFromTeamPool(user, id);
  }
}
```

## Performance considerations

<Tip>
Always use the optimized bulk methods for list views. They compute all permission types in a single pass with 2-4 queries total.
</Tip>

```typescript
// 2-4 queries total (optimized single-pass)
const { permissions } = await this.permissionComputeService
  .computeAllBulkLeadPermissionsWithTeamsInTransaction(em, user, leadIds, leads);

const { permissions } = await this.permissionComputeService
  .computeAllBulkDealPermissionsWithTeamsInTransaction(em, user, dealIds, deals);
```

<Warning>
Never loop over individual permission computations. This causes N+1 queries.

```typescript
// 300-500 queries for 100 items (BAD)
for (const lead of leads) {
  const permissions = await this.permissionComputeService
    .computeLeadPermissionsInTransaction(em, user, lead.id);
}
```
</Warning>

**Query count comparison:**

| Operation | Method | Database queries |
|-----------|--------|-----------------|
| 1 lead/deal | Single method | 3-5 queries |
| 100 leads/deals | Loop (N+1) | 300-500 queries |
| 100 leads/deals | Optimized bulk (`computeAll*`) | 2-4 queries |

All permission checks happen within the same EntityManager transaction, maintaining RLS context.

### Performance optimization strategies

<Steps>
<Step title="Use preloaded entities">
Pass already-loaded entities to bulk methods to avoid re-querying:

```typescript
const { permissions } = await this.permissionComputeService
  .computeAllBulkLeadPermissionsWithTeamsInTransaction(
    em, user, leadIds, leads  // ← Pass preloaded leads
  );
```
</Step>

<Step title="Batch database operations">
Bulk methods use `WHERE id IN (...)` queries instead of individual lookups:

```typescript
// Single query for all stakeholder checks
const stakeholders = await em.find(EntityStakeholder, {
  entityId: { $in: leadIds },
  entityType: StakeholderEntityType.LEAD,
  user: user.id
});
```
</Step>

<Step title="Minimize permission computation">
Only compute permissions for entities the user can actually see:

```typescript
// Apply RLS filtering first, then compute permissions
const accessibleLeads = await this.applyUserAccessFilters(em, user, query);
const permissions = await this.computeBulkPermissions(em, user, accessibleLeads);
```
</Step>
</Steps>
