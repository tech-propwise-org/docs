This section covers the RBAC (Role-Based Access Control) system: how permissions, roles, teams, and multi-organization support fit together.

## Design principles

1. **Unified Role & Permission system** — Single `Role` and `Permission` tables with a `scope` field differentiating ORG and TEAM levels.
2. **System vs custom roles** — System roles have `organization = NULL` (global/seeded). Custom roles belong to a specific organization.
3. **Explicit junction tables** — `UserOrgRole` and `TeamMembershipRole` enable multi-org support with metadata (`assigned_by`, `created_at`, `organization_id` for RLS).
4. **Multi-tenancy support** — Users can belong to multiple organizations with different roles per org. RLS enforced at database level.
5. **Permission inheritance** — Team roles can have both TEAM-scoped and ORG-scoped permissions, controlled via `team_available_permissions`.

## Core entities

These entities form the foundation of the RBAC system.

### User

```typescript
User
├── id: uuid
├── first_name, last_name: string
├── email: string (globally unique among active users)
├── password: string (hashed)
├── phone_number?: string
├── avatar_url?: string (public URL to avatar image, stored as WebP on R2)
├── languages?: string[] (array of language codes)
├── preferred_language?: string
├── is_verified: boolean
├── is_active: boolean
├── is_system_admin: boolean (platform-level admin, distinct from RBAC roles)
├── is_deleted: boolean (soft delete)
├── selected_organization_id?: uuid → Organization
├── Auth Fields:
│   ├── refresh_token?: string
│   ├── token_version: number (default: 1, incremented on token invalidation)
│   ├── verification_code?: string
│   ├── verification_code_expires?: timestamp
│   ├── verification_email_attempts: number (default: 0)
│   ├── last_verification_email_sent_at?: timestamp
│   ├── reset_password_code?: string
│   └── reset_password_code_expires?: timestamp
├── Relationships:
│   ├── organizations → Organization[] (via organization_users)
│   ├── orgRoles → UserOrgRole[] (roles in orgs)
│   └── teamMemberships → TeamMembership[]
└── Timestamps: created_at, updated_at

Indexes:
├── (first_name, last_name) [composite]
└── (email, is_deleted) [unique, where: is_deleted = false]
```

**Key Rules**:
- User can belong to multiple organizations
- Email is globally unique among non-deleted users (enforced by unique index on `[email, is_deleted]` where `is_deleted = false`)
- `selected_organization_id` tracks current org context
- Access roles via `orgRoles` collection (not direct `roles`)
- `is_system_admin` is a platform-level flag independent of per-org RBAC roles

### Organization

**Module location:** `/modules/organization/` (separate from RBAC)

Organization serves as the **tenant boundary** for multi-tenancy and is conceptually a peer to User, not a component of RBAC.

```typescript
Organization
├── id: uuid
├── name: string
├── website?: string
├── email?: string
├── vat?: string
├── avatar_url?: string (public URL to org logo, stored as WebP on R2)
├── address?: jsonb (embedded Address object)
├── owner_id: uuid → User (organization owner)
├── Relationships:
│   ├── users → User[] (ManyToMany, owner side, via organization_users)
│   ├── owner → User (ManyToOne)
│   ├── leads → Lead[] (OneToMany)
│   ├── contacts → Contact[] (OneToMany)
│   ├── deals → Deal[] (OneToMany)
│   └── companies → Company[] (OneToMany)
└── Timestamps: created_at, updated_at, is_deleted
```

<Note>
Teams, Roles, and Invitations each reference Organization via their own `@ManyToOne` relationships, but Organization does not declare inverse `@OneToMany` collections for these. CRM entities (Lead, Contact, Deal, Company) do have inverse relationships on Organization.
</Note>

**Key Rules**:
- Each organization has exactly one owner
- Owner is automatically assigned "Owner" role on creation
- System roles (`organization = NULL`) are shared across all orgs
- Soft delete via `isDeleted` flag with `@Filter` decorator

### Role (Unified)

**Purpose**: Represents a role that can be assigned to users. Unified table for both org-level and team-level roles.

```typescript
Role
├── id: uuid
├── name: string
├── description?: string
├── color?: string
├── scope: RoleScope (ORG | TEAM)
├── is_system: boolean (true for seeded roles)
├── organization_id?: uuid → Organization (NULL for system roles)
├── team_id?: uuid → Team (NULL for org roles)
├── Relationships:
│   ├── permissions → Permission[] (via role_permissions)
│   ├── userOrgRoles → UserOrgRole[] (org assignments)
│   └── teamMembershipRoles → TeamMembershipRole[] (team assignments)
└── Timestamps: created_at, updated_at, is_deleted

Constraints:
├── team_id NOT NULL only when scope = TEAM
└── organization_id NOT NULL for custom roles (NULL for system roles)
```

**Key Rules**:
- System roles: `is_system = true`, `organization_id = NULL`
- Custom roles: `is_system = false`, `organization_id = specific org`
- Team roles must have `team_id` populated
- Role assignments handled via explicit junction tables

### Permission (Unified)

**Purpose**: Represents a permission that can be granted to roles. Unified table for all permission types.

```typescript
Permission
├── id: uuid
├── key: string (unique identifier, e.g., "users.create", "leads.view")
├── name: string (human-readable name)
├── description?: string
├── scope: PermissionScope (ORG | TEAM)
├── group: string (logical grouping, e.g., "users", "leads")
├── Flags:
│   ├── is_system: boolean (true for seeded permissions)
│   ├── is_resource: boolean (applies to specific resources)
│   ├── is_sensitive: boolean (requires additional verification)
│   └── is_deprecated: boolean (marked for removal)
├── Relationships:
│   └── roles → Role[] (via role_permissions)
└── Timestamps: created_at, updated_at

Indexes:
└── key [unique]
```

**Key Rules**:
- Permissions are seeded from `permission-definitions.ts`
- `key` field is unique across all permissions
- Scope determines if permission applies to org or team context
- Resource permissions apply to specific entities (e.g., specific lead)

### Team

**Purpose**: Represents a team within an organization for collaboration and permission scoping.

```typescript
Team
├── id: uuid
├── name: string
├── description?: string
├── color?: string
├── organization_id: uuid → Organization
├── created_by_id: uuid → User
├── Relationships:
│   ├── organization → Organization (ManyToOne)
│   ├── createdBy → User (ManyToOne)
│   ├── memberships → TeamMembership[] (OneToMany)
│   └── teamAvailablePermissions → Permission[] (via team_available_permissions)
└── Timestamps: created_at, updated_at, is_deleted

Constraints:
└── (organization_id, name) [unique, where: is_deleted = false]
```

**Key Rules**:
- Team names must be unique within organization
- Creator automatically becomes team member
- `team_available_permissions` controls which org-scoped permissions can be assigned at team level

### TeamMembership

**Purpose**: Represents a user's membership in a team, separate from role assignments.

```typescript
TeamMembership
├── id: uuid
├── user_id: uuid → User
├── team_id: uuid → Team
├── organization_id: uuid → Organization (for RLS)
├── joined_at: timestamp
├── Relationships:
│   ├── user → User (ManyToOne)
│   ├── team → Team (ManyToOne)
│   ├── organization → Organization (ManyToOne)
│   └── teamMembershipRoles → TeamMembershipRole[] (OneToMany)
└── Timestamps: created_at, updated_at, is_deleted

Constraints:
└── (user_id, team_id, organization_id) [unique, where: is_deleted = false]
```

**Key Rules**:
- User can only have one membership per team
- Membership is separate from role assignments
- `organization_id` included for RLS enforcement

## Explicit junction tables

These tables handle the many-to-many relationships between users, roles, and teams with metadata and RLS support.

### UserOrgRole

**Purpose**: Assigns organization-level roles to users with metadata for auditing and RLS.

```typescript
UserOrgRole
├── id: uuid
├── user_id: uuid → User
├── role_id: uuid → Role
├── organization_id: uuid → Organization (for RLS)
├── assigned_by_id?: uuid → User (who assigned this role)
├── Relationships:
│   ├── user → User (ManyToOne)
│   ├── role → Role (ManyToOne)
│   ├── organization → Organization (ManyToOne)
│   └── assignedBy → User (ManyToOne, nullable)
└── Timestamps: created_at, updated_at, is_deleted

Constraints:
└── (user_id, role_id, organization_id) [unique, where: is_deleted = false]
```

### TeamMembershipRole

**Purpose**: Assigns team-level roles to team memberships with metadata and RLS support.

```typescript
TeamMembershipRole
├── id: uuid
├── team_membership_id: uuid → TeamMembership
├── role_id: uuid → Role
├── organization_id: uuid → Organization (for RLS)
├── assigned_by_id?: uuid → User (who assigned this role)
├── Relationships:
│   ├── teamMembership → TeamMembership (ManyToOne)
│   ├── role → Role (ManyToOne)
│   ├── organization → Organization (ManyToOne)
│   └── assignedBy → User (ManyToOne, nullable)
└── Timestamps: created_at, updated_at, is_deleted

Constraints:
└── (team_membership_id, role_id, organization_id) [unique, where: is_deleted = false]
```

**Key Rules for Junction Tables**:
- Both include `organization_id` for RLS enforcement
- Both track `assigned_by` for audit trails
- Unique constraints prevent duplicate role assignments
- Support soft deletion with `is_deleted` flag

## Permission resolution

This is the core mechanism that determines what actions a user can perform.

```
User requesting action
        │
        ├─── Resolve ORG permissions
        │    └─── Via UserOrgRole → Role → Permission (scope=ORG)
        │
        └─── Resolve TEAM permissions
             └─── Via TeamMembership → TeamMembershipRole → Role → Permission
                  ├─── TEAM-scoped permissions (always allowed)
                  └─── ORG-scoped permissions (must be in team_available_permissions)
```

### Effective permissions calculation

```typescript
interface EffectivePermissions {
  orgPermissions: string[];
  teamPermissions: string[];
  allPermissions: string[];
}

async function getUserEffectivePermissions(
  userId: string,
  organizationId: string,
): Promise<EffectivePermissions> {
  const orgPermissionKeys = new Set<string>();
  const teamPermissionKeys = new Set<string>();

  // Get organization-level permissions
  const userOrgRoles = await em.find(
    UserOrgRole,
    { user: userId, organization: organizationId, isDeleted: false },
    { populate: ['role.permissions'] },
  );

  for (const userOrgRole of userOrgRoles) {
    if (userOrgRole.role.scope === RoleScope.ORG) {
      for (const permission of userOrgRole.role.permissions) {
        orgPermissionKeys.add(permission.key);
      }
    }
  }

  // Get team-level permissions
  const teamMemberships = await em.find(
    TeamMembership,
    { user: userId, organization: organizationId, isDeleted: false },
    { populate: ['teamMembershipRoles.role.permissions'] },
  );

  for (const membership of teamMemberships) {
    for (const teamMembershipRole of membership.teamMembershipRoles) {
      if (teamMembershipRole.role.scope === RoleScope.TEAM) {
        for (const permission of teamMembershipRole.role.permissions) {
          teamPermissionKeys.add(permission.key);
        }
      } else {
        // ORG-scoped permission via team role
        orgPermissionKeys.add(permission.key);
      }
    }
  }

  return {
    orgPermissions: Array.from(orgPermissionKeys),
    teamPermissions: Array.from(teamPermissionKeys),
    allPermissions: Array.from(new Set([...orgPermissionKeys, ...teamPermissionKeys])),
  };
}
```

## System roles

Predefined roles that are seeded into every organization.

### Organization-level system roles

- **Owner**: Complete control over organization, including deletion
- **Admin**: Full administrative access except organization deletion
- **Basic**: Standard user with basic permissions

### Team-level system roles

- **Team Leader**: Manages team and assigns team roles
- **Agent**: Standard team member with work permissions

<Note>
System roles have `is_system = true` and `organization_id = NULL`. They are cloned into each organization during provisioning with `is_system = false` and the specific `organization_id`.
</Note>

```typescript
interface TeamInvitation {

  role_ids: string[];
```

**Key Rules**:
- One pending invitation per email per organization
- Invitations expire after configurable period
- Role assignments are pre-configured and applied upon acceptance
- Both existing and new users can accept invitations

## Business rules

Key business rules enforced by the RBAC system.

### Role assignment validation

<Steps>
<Step title="Owner restrictions">
Each organization must have exactly one Owner role. Owner role cannot be removed unless ownership is transferred first.
</Step>

<Step title="System role integrity">
System roles cannot be modified or deleted. Only their permission assignments can be updated during reseeding.
</Step>

<Step title="Team role constraints">
Team-level roles can only be assigned to team members. ORG-scoped permissions in team roles must be in the team's `available_permissions`.
</Step>

<Step title="Permission inheritance">
Team roles inherit available permissions from their team's `team_available_permissions` configuration.
</Step>
</Steps>

### Cascading deletion rules

- **Organization deletion**: Soft-deletes all roles, teams, memberships, and invitations
- **Team deletion**: Soft-deletes all memberships and role assignments
- **User deactivation**: Maintains role assignments but user cannot authenticate
- **Role deletion**: Removes all assignments but preserves audit trail

## RLS integration

Row-Level Security (RLS) policies enforce tenant isolation at the database level.

### RLS policy pattern

```sql
-- Example: Users can only access their organization's teams
CREATE POLICY teams_tenant_isolation ON teams
    FOR ALL TO authenticated
    USING (organization_id = current_setting('app.current_organization_id')::uuid);

-- Example: Junction table isolation
CREATE POLICY user_org_roles_tenant_isolation ON user_org_roles
    FOR ALL TO authenticated  
    USING (organization_id = current_setting('app.current_organization_id')::uuid);
```

### Organization context setting

```typescript
// Set organization context for RLS
await this.em.getConnection().execute(`
  SET app.current_organization_id = '${organizationId}';
`);

// All subsequent queries are automatically filtered by RLS
const teams = await this.em.find(Team, {});
```

### RLS-enabled entities

- All RBAC entities include `organization_id` field
- Junction tables (`UserOrgRole`, `TeamMembershipRole`) include `organization_id`
- RLS policies automatically filter queries to current organization
- Organization context set during authentication middleware

### Module dependency graph

```
OrganizationModule (independent tenant boundary)
       ↓
   UserModule (depends on Organization)
       ↓
   AuthModule (depends on User + Organization)
       ↓
   RbacModule (depends on Auth + Organization)
       ↓
   CrmModule (depends on RBAC + Organization)
```

**Key Principles**:
- **Organization is a peer, not a child** — conceptually at the same level as User, not nested within RBAC
- **RBAC depends on Organization** — RBAC imports `OrganizationModule`, not vice versa
- **Tenant boundary** — Organization defines the multi-tenancy boundary; all tenant-scoped entities reference it

### User-centric API design

The system follows a **user-centric API design philosophy**:

<Steps>
<Step title="Default scope = user's data">
All default endpoints return only data the current user has access to
</Step>

<Step title="Security by default">
Users cannot accidentally access data outside their scope
</Step>

<Step title="Explicit admin access">
Administrative/global endpoints are separate (under `/admin` routes)
</Step>

<Step title="Least privilege">
Users only see what they need, admins explicitly request broader access
</Step>
</Steps>

```typescript
// ✅ User-centric default
GET /organizations      → User's organizations only
GET /leads             → User's leads only (or team's based on permissions)
GET /deals             → User's deals only

// ✅ Admin endpoints (separate controller)
GET /admin/organizations → All organizations (requires permission)
GET /admin/leads         → All leads (requires permission)
GET /admin/deals         → All deals (requires permission)
```

**Benefits**:
- **Security** — Reduces risk of data leakage
- **Performance** — Smaller result sets by default
- **UX** — Users see only relevant data
- **RBAC alignment** — Matches RLS and permission model
- **API clarity** — Clear distinction between user and admin access
