The invitation system enables secure user onboarding with predefined access controls, supporting both new user registration and existing user integration across multiple organizations within the complete RBAC framework.

<Info>
The invitation system provides secure token-based user onboarding with comprehensive role and team assignment capabilities, supporting both new user registration and existing user integration across multiple organizations.
</Info>

## Architecture overview

The RBAC system follows these core principles that directly impact invitation workflows:

1. **Unified role & permission system**: Single `Role` table for both org-level and team-level roles with `scope` differentiation
2. **System vs custom roles**: System roles (organization = NULL) are globally shared, custom roles are org-specific
3. **Explicit junction tables**: `UserOrgRole` and `TeamMembershipRole` enable multi-org support with metadata tracking
4. **Multi-tenancy support**: Users can belong to multiple organizations with different roles
5. **Permission inheritance**: Team roles can have both TEAM-scoped and ORG-scoped permissions

<Warning>
Invitation acceptance runs on public/identity-token endpoints that have **no RLS organization context**. All RBAC tables have RLS policies that deny access when `app.organization_id` is not set. The invitation system **must use RLS bypass context** for all RBAC assignments.
</Warning>

## Invitation entity

The core entity that manages the invitation lifecycle and role assignments:

```typescript
Invitation
├── id: uuid
├── email: string
├── status: InvitationStatus (pending, completed, expired, cancelled)
├── token: string (secure, 64-character hex string)
├── expires_at?: timestamp (default: 7 days from creation)
├── invited_by_id: uuid → User (ManyToOne)
├── accepted_by_id?: uuid → User (ManyToOne, nullable)
├── accepted_at?: timestamp
├── organization_id: uuid → Organization (ManyToOne, required)
├── org_role_assignments: jsonb ({roleId, roleName}[])
├── team_id?: uuid → Team (ManyToOne, nullable, primary team assignment)
├── team_role_assignments: jsonb ({roleId, roleName, teamId, teamName, roleIds}[])
├── welcome_message?: string
├── Relationships:
│   ├── organization → Organization (ManyToOne)
│   ├── invitedBy → User (ManyToOne)
│   ├── acceptedBy? → User (ManyToOne, nullable)
│   └── team? → Team (ManyToOne, nullable)
└── Timestamps: created_at, updated_at

Indexes:
├── (invited_by_id)
├── (organization_id)
├── (token) [unique]
├── (status)
└── (email, organization_id) [unique, where: status = 'pending']
```

<Tabs>
<Tab title="pending">
**Initial state** when invitation is created and email sent:
- **Transitions to**: completed, expired, cancelled
- **Actions available**: Accept invitation, cancel invitation, extend expiry

<Tab title="completed">
**Final state** when user successfully accepts invitation:
- **Transitions from**: pending
- **Actions available**: View completion details only

```typescript
invitation.acceptedAt = new Date();
```

<Tab title="expired">
**Final state** when invitation expires without acceptance:
- **Transitions from**: pending (automatic after expires_at)

```typescript
async acceptInvitation(
  @Param('token') token: string,

    accessToken: tokens.accessToken,
    refreshToken: tokens.refreshToken
  };
}
```

### RBAC assignment application

```typescript
private async applyInvitationRbacAssignments(
  bypassEm: EntityManager,
  user: User,
  invitation: Invitation
): Promise<void> {
  // 1. Add user to organization
  await this.addUserToOrganization(bypassEm, user, invitation);
  
  // 2. Apply organizational role assignments
  for (const roleAssignment of invitation.orgRoleAssignments) {
    const userOrgRole = bypassEm.create(UserOrgRole, {
      user,
      role: { id: roleAssignment.roleId },
      organization: invitation.organization,
      assignedBy: invitation.invitedBy
    });
    await bypassEm.persist(userOrgRole);
  }
  
  // 3. Apply team role assignments
  for (const teamAssignment of invitation.teamRoleAssignments) {
    // Create team membership
    const teamMembership = bypassEm.create(TeamMembership, {
      user,
      team: { id: teamAssignment.teamId },
      organization: invitation.organization,
      addedBy: invitation.invitedBy,
      joinedAt: new Date()
    });
    await bypassEm.persist(teamMembership);

    // Apply team roles
    for (const roleId of teamAssignment.roleIds) {
      const teamMembershipRole = bypassEm.create(TeamMembershipRole, {
        teamMembership,
        role: { id: roleId },
        organization: invitation.organization,
        assignedBy: invitation.invitedBy
      });
      await bypassEm.persist(teamMembershipRole);
    }
  }
  
  // 4. Complete invitation
  invitation.status = InvitationStatus.COMPLETED;
  invitation.acceptedBy = user;
  invitation.acceptedAt = new Date();
  
  await bypassEm.flush();
}
```

### Token security and validation

```typescript
private async validateInvitationToken(token: string): Promise<Invitation> {
  // Find invitation by token
  const invitation = await this.em.findOne(
    Invitation,
    { token },
    { populate: ['organization', 'invitedBy', 'team'] }
  );

  if (!invitation) {
    throw new BadRequestException('Invalid invitation token');
  }

  // Check status
  if (invitation.status !== InvitationStatus.PENDING) {
    throw new BadRequestException('Invitation is no longer valid');
  }

  // Check expiry
  if (invitation.expiresAt && new Date() > invitation.expiresAt) {
    invitation.status = InvitationStatus.EXPIRED;
    await this.em.persistAndFlush(invitation);
    throw new BadRequestException('Invitation has expired');
  }

  return invitation;
}

private generateSecureToken(): string {
  // 64-character hex string (256 bits of entropy)
  return crypto.randomBytes(32).toString('hex');
}
```

## Permission resolution

Permission resolution during invitation acceptance follows the unified RBAC permission model:

<AccordionGroup>
<Accordion title="User organizational permissions">
```sql
SELECT DISTINCT p.key, p.scope, p.category
FROM permissions p
INNER JOIN role_permissions rp ON p.id = rp.permission_id
INNER JOIN user_org_roles uor ON rp.role_id = uor.role_id
WHERE uor.user_id = $1 
  AND uor.organization_id = $2
  AND p.scope = 'ORG';
```
</Accordion>

<Accordion title="User team permissions">
```sql
SELECT DISTINCT p.key, p.scope, p.category, t.id as team_id, t.name as team_name
FROM permissions p
INNER JOIN role_permissions rp ON p.id = rp.permission_id
INNER JOIN team_membership_roles tmr ON rp.role_id = tmr.role_id
INNER JOIN team_memberships tm ON tmr.team_membership_id = tm.id
INNER JOIN teams t ON tm.team_id = t.id
WHERE tm.user_id = $1 
  AND tm.organization_id = $2
  AND (p.scope = 'TEAM' OR (p.scope = 'ORG' AND p.team_available = true));
```
</Accordion>

<Accordion title="Team available permissions">
```sql
SELECT p.key, p.description, p.category
FROM permissions p
WHERE p.scope = 'ORG' 
  AND p.team_available = true
ORDER BY p.category, p.key;
```
</Accordion>

## System roles

The invitation system integrates with seeded system roles that are shared across all organizations:

<Tabs>
<Tab title="Owner">
**Organization owner** with full administrative privileges:
- `organization_id`: NULL (system role)
- **Permissions**: All ORG-scoped permissions
- **Assignment**: Only one owner per organization
- **Invitation rules**: Cannot be assigned via invitation (requires ownership transfer)
</Tab>

<Tab title="Admin">
**Administrative role** with broad organizational permissions:
- `organization_id`: NULL (system role)
- **Key permissions**: users:manage, teams:manage, roles:manage
- **Assignment**: Can be assigned to multiple users
- **Invitation rules**: Can be assigned via invitation by Owner or Admin
</Tab>

<Tab title="Basic">
**Standard user role** with essential CRM permissions:
- `organization_id`: NULL (system role)
- **Key permissions**: leads:read, contacts:read, deals:read
- **Assignment**: Default role for new team members
- **Invitation rules**: Automatically assigned if no specific roles provided
</Tab>

<Tab title="Team Leader">
**Team management role** with team-scoped permissions:
- `organization_id`: NULL (system role)
- `scope`: TEAM
- **Key permissions**: teams:manage, team_members:manage
- **Assignment**: Can be assigned to team members
- **Invitation rules**: Requires team context in invitation
</Tab>

<Tab title="Agent">
**Team member role** with operational permissions:
- `organization_id`: NULL (system role)
- `scope`: TEAM
- **Key permissions**: leads:create, contacts:create, deals:update
- **Assignment**: Standard team role
- **Invitation rules**: Can be assigned to team members
</Tab>
</Tabs>

## Business rules

The invitation system operates under comprehensive business rules ensuring security and consistency:

<AccordionGroup>
<Accordion title="Email uniqueness">
- Only one pending invitation per email per organization
- Users can have multiple pending invitations across different organizations
- Completed or expired invitations don't block new invitations
- Enforced via unique index on `(email, organization_id)` where `status = 'pending'`
</Accordion>

<Accordion title="Expiration policy">
- Invitations expire after 7 days by default (configurable)
- Expired invitations cannot be accepted or modified
- System provides cleanup job for expired invitations
- Admins can extend invitation expiry before expiration
</Accordion>

<Accordion title="Role assignment validation">
- All role assignments validated at invitation creation
- Roles must exist and be appropriate for their context (ORG/TEAM scope)
- System roles (organization = NULL) available to all organizations
- Custom roles only available within their organization
- Team role assignments require valid team membership creation
</Accordion>

<Accordion title="Permission inheritance">
- Team members inherit team-scoped permissions from team roles
- ORG-scoped permissions in team roles require team availability
- Organizational roles provide organization-wide permissions
- Permission conflicts resolved through union (most permissive)
</Accordion>

<Accordion title="Multi-organization support">
- Users can belong to multiple organizations with different roles
- Same email can have different role assignments per organization
- Cross-organization role assignments handled via separate junction records
- Organization switching updates `selected_organization_id`
</Accordion>

<Accordion title="Token security">
- 64-character hex tokens provide 256 bits of entropy
- Tokens are globally unique across all invitations
- Token validation includes expiry and status checks
- Secure token generation using crypto.randomBytes(32)
</Accordion>

<Accordion title="Owner role restrictions">
- Only one owner per organization enforced at invitation level
- Owner role transfers require special handling (not via invitations)
- Owner invitations require existing owner approval
- Organization deletion requires owner role
</Accordion>

<Accordion title="Team membership rules">
- Users must have organizational membership before team membership
- Team memberships automatically created during invitation acceptance
- Team role assignments require corresponding team membership
- Team dissolution removes all team memberships and roles
</Accordion>
</AccordionGroup>

## Services & controllers

### InvitationService

Core business logic with comprehensive RBAC integration:

```typescript
@Injectable()
export class InvitationService {
  constructor(
    @InjectRepository(Invitation) private readonly em: EntityManager,
    private readonly authService: AuthService,
    private readonly emailService: EmailService,
    private readonly tenantContext: TenantContext
  ) {}

  async createInvitation(dto: CreateInvitationDto): Promise<Invitation> {
    await this.validateRoleAssignments(
      dto.organizationId, 
      dto.orgRoleAssignments, 
      dto.teamRoleAssignments
    );

    const token = this.generateSecureToken();
    const invitation = this.em.create(Invitation, {
      ...dto,
      token,
      status: InvitationStatus.PENDING,
      expiresAt: dto.expiresAt || new Date(Date.now() + 7 * 24 * 60 * 60 * 1000)
    });

    await this.em.persistAndFlush(invitation);
    await this.sendInvitationEmail(invitation);
    
    return invitation;
  }

  async acceptInvitation(token: string, userData: AcceptInvitationDto): Promise<User> {
    const invitation = await this.validateInvitationToken(token);
    
    const user = await this.authService.register({
      email: invitation.email,
      ...userData
    });

    await this.tenantContext.executeWithBypass(async (bypassEm) => {
      await this.applyInvitationRbacAssignments(bypassEm, user, invitation);
    });

    return user;
  }

  async acceptInvitationExisting(token: string, user: User): Promise<Invitation> {
    const invitation = await this.validateInvitationToken(token);
    
    // Validate email matches
    if (invitation.email !== user.email) {
      throw new BadRequestException('Email mismatch');
    }

    await this.tenantContext.executeWithBypass(async (bypassEm) => {
      await this.applyInvitationRbacAssignments(bypassEm, user, invitation);
    });

    return invitation;
  }

  private async validateRoleAssignments(
    organizationId: string,
    orgRoleAssignments: OrgRoleAssignment[],
    teamRoleAssignments: TeamRoleAssignment[]
  ): Promise<void> {
    // Validate org roles
    for (const roleAssignment of orgRoleAssignments) {
      const role = await this.em.findOne(Role, {
        id: roleAssignment.roleId,
        scope: RoleScope.ORG,
        $or: [
          { organization: organizationId }, // Custom role
          { organization: null }           // System role
        ]
      });
      
      if (!role) {
        throw new BadRequestException(`Invalid org role: ${roleAssignment.roleId}`);
      }
    }
    
    // Validate team roles and teams
    for (const teamAssignment of teamRoleAssignments) {
      const team = await this.em.findOne(Team, {
        id: teamAssignment.teamId,
        organization: organizationId
      });
      
      if (!team) {
        throw new BadRequestException(`Invalid team: ${teamAssignment.teamId}`);
      }
      
      for (const roleId of teamAssignment.roleIds) {
        const role = await this.em.findOne(Role, {
          id: roleId,
          scope: RoleScope.TEAM,
          $or: [
            { organization: organizationId },
            { organization: null }
          ]
        });
        
        if (!role) {
          throw new BadRequestException(`Invalid team role: ${roleId}`);
        }
      }
    }
  }

  private generateSecureToken(): string {
    return crypto.randomBytes(32).toString('hex');
  }
}
```

## RLS integration

The invitation system operates within Row-Level Security constraints and requires bypass patterns:

### RLS bypass for RBAC assignments

```typescript
class InvitationService {
  async acceptInvitation(token: string, userData: AcceptInvitationDto): Promise<User> {
    // Public endpoint - no RLS context
    const invitation = await this.validateInvitationToken(token);
    
    const user = await this.authService.register({
      email: invitation.email,
      ...userData
    });
    
    // Apply RBAC assignments with RLS bypass
    await this.tenantContext.executeWithBypass(async (bypassEm) => {
      await this.applyInvitationRbacAssignments(bypassEm, user, invitation);
    });
    
    return user;
  }
}
```

<Warning>
**Critical RLS Considerations**:

1. **Public endpoint context**: Invitation acceptance endpoints have no organization context
2. **RLS policy blocking**: All RBAC tables deny access when `app.organization_id` is NULL
3. **Bypass requirement**: RBAC assignments **must** use bypass context during invitation acceptance
4. **Security implications**: Bypass context removes multi-tenant isolation temporarily
</Warning>

### Tenant context patterns

```typescript
export class TenantContext {
  async executeWithBypass<T>(operation: (bypassEm: EntityManager) => Promise<T>): Promise<T> {
    // Create entity manager with RLS bypass
    const bypassEm = this.createBypassEntityManager();
    
    try {
      const result = await operation(bypassEm);
      return result;
    } catch (error) {
      await bypassEm.rollback();
      throw error;
    } finally {
      await bypassEm.close();
    }
  }

  private createBypassEntityManager(): EntityManager {
    return this.em.fork({
      // Configuration to bypass RLS policies
      connectionOptions: {
        application_name: 'rbac_bypass',
        search_path: 'public'
      }
    });
  }
}
```
