This document contains the complete specification for the RBAC (Role-Based Access Control) system, including entity relationships, permission resolution, system roles, multi-organization support, and the invitation workflow for user onboarding.

<Info>
The RBAC system provides unified role and permission management across organizations with multi-tenant support, explicit junction tables for complex relationships, and comprehensive audit trails.
</Info>

## Architecture Overview

### Design Principles

The RBAC system follows these core principles:

<AccordionGroup>
<Accordion title="Unified Role & Permission System">
- Single `Role` table for both org-level and team-level roles
- Single `Permission` table for all permissions
- `scope` field differentiates between ORG and TEAM levels
- Consistent permission resolution across all contexts
</Accordion>

<Accordion title="System vs Custom Roles">
- System roles have `organization = NULL` (global/seeded)
- Custom roles have `organization = specific org ID`
- System roles: Owner, Admin, Basic, Team Leader, Agent
- Custom roles can only be assigned within their parent organization
</Accordion>

<Accordion title="Explicit Junction Tables">
- `UserOrgRole` - User ↔ Role assignments in Organization context
- `TeamMembershipRole` - TeamMembership ↔ Role assignments
- Required for multi-org support (user can have different roles in different orgs)
- Includes metadata: `assigned_by`, `created_at`, `organization_id` (for RLS)
</Accordion>

<Accordion title="Multi-Tenancy Support">
- Users can belong to multiple organizations
- Same user can have different roles in different organizations
- RLS (Row-Level Security) enforced at database level
- Strict isolation between organization contexts
</Accordion>

<Accordion title="Permission Inheritance">
- Team roles can have both TEAM-scoped and ORG-scoped permissions
- Team-available permissions controlled via `team_available_permissions`
- Permission resolution follows hierarchical inheritance patterns
</Accordion>
</AccordionGroup>

## Core Entities

### User

**Purpose**: Represents a user account in the system.

```typescript
User
├── id: uuid
├── first_name, last_name: string
├── email: string (globally unique among active users)
├── password: string (hashed)
├── phone_number?: string
├── avatar_url?: string (public URL to avatar image, stored as WebP on R2)
├── languages?: string[] (array of language codes)
├── preferred_language?: string
├── is_verified: boolean
├── is_active: boolean
├── is_system_admin: boolean (platform-level admin, distinct from RBAC roles)
├── is_deleted: boolean (soft delete)
├── selected_organization_id?: uuid → Organization
├── Auth Fields:
│   ├── refresh_token?: string
│   ├── token_version: number (default: 1, incremented on token invalidation)
│   ├── verification_code?: string
│   ├── verification_code_expires?: timestamp
│   ├── verification_email_attempts: number (default: 0)
│   ├── last_verification_email_sent_at?: timestamp
│   ├── reset_password_code?: string
│   └── reset_password_code_expires?: timestamp
├── Relationships:
│   ├── organizations → Organization[] (via organization_users)
│   ├── orgRoles → UserOrgRole[] (roles in orgs)
│   └── teamMemberships → TeamMembership[]
└── Timestamps: created_at, updated_at

Indexes:
├── (first_name, last_name) [composite]
└── (email, is_deleted) [unique, where: is_deleted = false]
```

**Key Rules**:
- User can belong to multiple organizations
- Email is globally unique among non-deleted users (enforced by unique index on `[email, is_deleted]` where `is_deleted = false`)
- `selected_organization_id` tracks current org context
- Access roles via `orgRoles` collection (not direct `roles`)
- `is_system_admin` is platform-level, independent of per-org RBAC roles

### Organization

**Module Location:** `/modules/organization/` (separate from RBAC)

**Purpose**: Represents a tenant organization serving as the tenant boundary for multi-tenancy.

```typescript
Organization
├── id: uuid
├── name: string
├── website?: string
├── email?: string
├── vat?: string
├── avatar_url?: string (public URL to org logo, stored as WebP on R2)
├── address?: jsonb (embedded Address object)
├── owner_id: uuid → User (organization owner)
├── Relationships:
│   ├── users → User[] (ManyToMany, owner side, via organization_users)
│   ├── owner → User (ManyToOne)
│   ├── leads → Lead[] (OneToMany)
│   ├── contacts → Contact[] (OneToMany)
│   ├── deals → Deal[] (OneToMany)
│   └── companies → Company[] (OneToMany)
└── Timestamps: created_at, updated_at, is_deleted
```

<Note>
Teams, Roles, and Invitations each reference Organization via their own `@ManyToOne` relationships, but Organization does not declare inverse `@OneToMany` collections for these. CRM entities (Lead, Contact, Deal, Company) do have inverse relationships on Organization.
</Note>

**Key Rules**:
- Each organization has exactly one owner
- Owner is automatically assigned "Owner" role on creation
- System roles (organization = NULL) are shared across all orgs
- Soft delete via `isDeleted` flag with `@Filter` decorator

### Role (Unified)

**Purpose**: Unified role table for both org-level and team-level roles.

```typescript
Role
├── id: uuid
├── name: string
├── description?: string
├── color?: string
├── scope: RoleScope (ORG | TEAM)
├── is_system: boolean (true for seeded roles)
├── organization_id?: uuid → Organization (NULL for system roles)
├── team_id?: uuid → Team (NULL for org roles)
├── Relationships:
│   ├── permissions → Permission[] (via role_permissions)
│   ├── userOrgRoles → UserOrgRole[] (when assigned to users in orgs)
│   ├── teamMembershipRoles → TeamMembershipRole[] (when assigned to team members)
│   └── organization? → Organization (ManyToOne, nullable)
└── Timestamps: created_at, updated_at
```

**Key Rules**:
- System roles: `organization = NULL`, available to all orgs
- Custom roles: `organization = specific org`, restricted to that org
- Team roles: `scope = TEAM`, can have both TEAM and ORG permissions
- Org roles: `scope = ORG`, typically have ORG-scoped permissions

```typescript
└── Timestamps: created_at, updated_at

Indexes:
├── (key) [unique]
├── (scope)
└── (category)
```

## Permission Resolution

### Hierarchical Permission Resolution Algorithm

The system resolves permissions for a user in a specific organization through hierarchical inheritance:

<Steps>
<Step title="Organizational Role Permissions">
Collect all permissions from roles assigned to the user in the organization context via `UserOrgRole`:

```typescript
const orgPermissions = await em.createQueryBuilder(Permission, 'p')
  .innerJoin('p.roles', 'r')
  .innerJoin('r.userOrgRoles', 'uor')
  .where({
    'uor.user': userId,
    'uor.organization': organizationId
  })
  .getResultList();
```
</Step>

<Step title="Team Role Permissions">
For each team membership, collect permissions from assigned team roles via `TeamMembershipRole`:

```typescript
const teamPermissions = await em.createQueryBuilder(Permission, 'p')
  .innerJoin('p.roles', 'r')
  .innerJoin('r.teamMembershipRoles', 'tmr')
  .innerJoin('tmr.teamMembership', 'tm')
  .where({
    'tm.user': userId,
    'tm.organization': organizationId
  })
  .getResultList();
```
</Step>

<Step title="Permission Scope Validation">
Validate team role permissions against team available permissions:

```typescript
// For ORG-scoped permissions in team roles, check team_available_permissions
const validTeamPermissions = teamPermissions.filter(permission => {
  if (permission.scope === PermissionScope.TEAM) {
    return true; // TEAM permissions always valid in team context
  }
  
  // For ORG permissions, check if team allows this permission
  return teamAvailablePermissions.some(tap => 
    tap.permission.id === permission.id
  );
});
```
</Step>

<Step title="Permission Aggregation">
Combine and deduplicate all valid permissions:

```typescript
const allPermissions = [
  ...orgPermissions,
  ...validTeamPermissions
].reduce((unique, permission) => {
  if (!unique.some(p => p.id === permission.id)) {
    unique.push(permission);
  }
  return unique;
}, []);
```
</Step>
</Steps>

## System Roles

### Predefined System Roles

<Tabs>
<Tab title="Owner">
**Scope**: ORG  
**System Role**: Yes  
**Default Permissions**: All organizational permissions  
**Special Rules**: 
- One per organization
- Assigned during org creation
- Can assign any role including other Owner roles
- Cannot be removed while org exists
</Tab>

<Tab title="Admin">
**Scope**: ORG  
**System Role**: Yes  
**Default Permissions**: User management, team management, most org features  
**Special Rules**:
- Can invite users and assign Admin, Basic, Team Leader, Agent roles
- Cannot assign Owner role
- Multiple admins allowed per organization
</Tab>

<Tab title="Basic">
**Scope**: ORG  
**System Role**: Yes  
**Default Permissions**: Read access to org data, basic CRM operations  
**Special Rules**:
- Default role when no explicit org roles specified
- Cannot send invitations
- Standard organization member role
</Tab>

<Tab title="Team Leader">
**Scope**: TEAM  
**System Role**: Yes  
**Default Permissions**: Team management, team-scoped operations  
**Special Rules**:
- Can invite users to specific teams
- Can assign Agent or lower team roles
- Team-specific leadership capabilities
</Tab>

<Tab title="Agent">
**Scope**: TEAM  
**System Role**: Yes  
**Default Permissions**: CRM operations within assigned teams  
**Special Rules**:
- Cannot send invitations
- Frontline operational role
- Primarily used in team contexts
</Tab>
</Tabs>

## Invitation System

The invitation system enables secure user onboarding with predefined access controls, supporting both new user registration and existing user integration across multiple organizations.

### Invitation Entity

```typescript
Invitation
├── id: uuid
├── email: string
├── status: InvitationStatus (pending, completed, expired, cancelled)
├── token: string (secure, 64-character hex string)
├── expires_at?: timestamp (default: 7 days from creation)
├── invited_by_id: uuid → User (ManyToOne)
├── accepted_by_id?: uuid → User (ManyToOne, nullable)
├── accepted_at?: timestamp
├── organization_id: uuid → Organization (ManyToOne, required)
├── org_role_assignments: jsonb ({roleId, roleName}[])
├── team_id?: uuid → Team (ManyToOne, nullable, primary team assignment)
├── team_role_assignments: jsonb ({roleId, roleName, teamId, teamName, roleIds}[])
├── welcome_message?: string
├── Relationships:
│   ├── organization → Organization (ManyToOne)
│   ├── invitedBy → User (ManyToOne)
│   ├── acceptedBy? → User (ManyToOne, nullable)
│   └── team? → Team (ManyToOne, nullable)
└── Timestamps: created_at, updated_at

Indexes:
├── (invited_by_id)
├── (organization_id)
├── (token) [unique]
├── (status)
└── (email, organization_id) [unique, where: status = 'pending']
```

### Invitation Status Flow

<Tabs>
<Tab title="pending">
**Initial State**: Invitation created and email sent  
**Transitions to**: completed, expired, cancelled  
**Actions Available**: Accept invitation, cancel invitation, extend expiry
</Tab>

<Tab title="completed">
**Final State**: User successfully accepted invitation  
**Transitions from**: pending  
**Actions Available**: View completion details only
</Tab>

<Tab title="expired">
**Final State**: Invitation expired without acceptance  
**Transitions from**: pending (automatic after expires_at)  
**Actions Available**: Create new invitation if needed
</Tab>

<Tab title="cancelled">
**Final State**: Invitation manually cancelled by admin  
**Transitions from**: pending  
**Actions Available**: Create new invitation if needed
</Tab>
</Tabs>

### Invitation Flow

<Steps>
<Step title="Admin creates invitation">
Administrator creates invitation specifying email, organizational roles, and team assignments with comprehensive validation of all role assignments and team memberships.

```typescript
const invitation = await invitationService.createInvitation({
  email: 'user@example.com',
  organizationId: 'org-123',
  orgRoleAssignments: [
    { roleId: 'basic-role-id', roleName: 'Basic' }
  ],
  teamRoleAssignments: [
    {
      teamId: 'team-123',
      teamName: 'Sales Team',
      roleIds: ['agent-role-id'],
      roleNames: ['Agent']
    }
  ],
  welcomeMessage: 'Welcome to our organization!'
});
```
</Step>

<Step title="Email sent with secure link">
System automatically sends email with secure 64-character token link, different templates for new vs. existing users, and comprehensive invitation context.

```typescript
// Email contains secure invitation link
const invitationUrl = `${baseUrl}/accept-invitation/${invitation.token}`;

// Email template varies based on user existence
const template = await userExists ? 'invitation-existing-user' : 'invitation-new-user';
```
</Step>

<Step title="User accepts invitation">
Two acceptance flows:
- **New user**: Public endpoint creates account and applies RBAC
- **Existing user**: Identity token endpoint adds user to organization

```typescript
// New user acceptance (public endpoint)
const newUser = await invitationService.acceptInvitation(token, {
  firstName: 'John',
  lastName: 'Doe',
  password: 'securePassword'
});

// Existing user acceptance (authenticated endpoint)
await invitationService.acceptInvitationExisting(token, currentUser);
```
</Step>

<Step title="RBAC assignments applied">
System applies pre-configured roles and team assignments using RLS bypass context with comprehensive validation and audit logging.

```typescript
// Applied within RLS bypass context
await this.tenantContext.executeWithBypass(async (bypassEm) => {
  await this.applyInvitationRbacAssignments(bypassEm, user, invitation);
});
```
</Step>
</Steps>

<Warning>
Invitation acceptance runs on public/identity-token endpoints that have **no RLS organization context**. All RBAC tables have RLS policies that deny access when `app.organization_id` is not set. The `applyInvitationRbacAssignments` method **must receive the bypassed EntityManager** from the calling `executeWithBypass` context.
</Warning>

### RBAC Assignment Implementation

The invitation acceptance process applies RBAC assignments through the following steps:

<Steps>
<Step title="Organization Membership">
Add user to organization if not already a member:

```typescript
// Check existing membership
const existingMembership = await bypassEm.findOne(OrganizationUser, {
  user: user.id,
  organization: invitation.organization.id
});

if (!existingMembership) {
  const orgUser = bypassEm.create(OrganizationUser, {
    user,
    organization: invitation.organization,
    addedBy: invitation.invitedBy,
    joinedAt: new Date()
  });
  await bypassEm.persist(orgUser);
}
```
</Step>

<Step title="Organizational Role Assignments">
Apply organization-level roles from invitation:

```typescript
for (const roleAssignment of invitation.orgRoleAssignments) {
  const userOrgRole = bypassEm.create(UserOrgRole, {
    user,
    role: roleAssignment.roleId,
    organization: invitation.organization,
    assignedBy: invitation.invitedBy
  });
  await bypassEm.persist(userOrgRole);
}
```
</Step>

<Step title="Team Membership & Role Assignments">
Apply team memberships and team-level roles:

```typescript
for (const teamAssignment of invitation.teamRoleAssignments) {
  // Create team membership
  const teamMembership = bypassEm.create(TeamMembership, {
    user,
    team: teamAssignment.teamId,
    organization: invitation.organization,
    addedBy: invitation.invitedBy
  });
  await bypassEm.persist(teamMembership);
  
  // Assign team roles
  for (const roleId of teamAssignment.roleIds) {
    const teamMembershipRole = bypassEm.create(TeamMembershipRole, {
      teamMembership,
      role: roleId,
      organization: invitation.organization,
      assignedBy: invitation.invitedBy
    });
    await bypassEm.persist(teamMembershipRole);
  }
}
```
</Step>

<Step title="Invitation Completion">
Mark invitation as completed and set acceptance metadata:

```typescript
invitation.status = InvitationStatus.COMPLETED;
invitation.acceptedBy = user;
invitation.acceptedAt = new Date();
await bypassEm.persistAndFlush(invitation);
```
</Step>
</Steps>

## Query Patterns

### Common RBAC Queries

<CodeGroup>

```typescript Get user permissions in organization
async getUserPermissions(
  userId: string, 
  organizationId: string
): Promise<Permission[]> {
  const qb = this.em.createQueryBuilder(Permission, 'p');
  
  // Org-level permissions
  qb.leftJoin('p.roles', 'orgRole')
    .leftJoin('orgRole.userOrgRoles', 'uor')
    .leftJoin('uor.user', 'orgUser')
    .leftJoin('uor.organization', 'orgOrg');
    
  // Team-level permissions
  qb.leftJoin('p.roles', 'teamRole')
    .leftJoin('teamRole.teamMembershipRoles', 'tmr')
    .leftJoin('tmr.teamMembership', 'tm')
    .leftJoin('tm.user', 'teamUser')
    .leftJoin('tm.organization', 'teamOrg')
    .leftJoin('tm.team', 't')
    .leftJoin('t.teamAvailablePermissions', 'tap')
    .leftJoin('tap.permission', 'availablePerm');
  
  qb.where({
    $or: [
      // Org permissions
      { 
        'orgUser.id': userId, 
        'orgOrg.id': organizationId 
      },
      // Team permissions (with availability check for ORG-scoped)
      { 
        'teamUser.id': userId, 
        'teamOrg.id': organizationId,
        $or: [
          { scope: PermissionScope.TEAM },
          { 
            scope: PermissionScope.ORG,
            'availablePerm.id': { $eq: 'p.id' }
          }
        ]
      }
    ]
  });
  
  return qb.getResultList();
}
```

```typescript Get user roles across organizations
async getUserRolesAcrossOrgs(userId: string): Promise<UserOrgRole[]> {
  return this.em.find(UserOrgRole, 
    { user: userId },
```

```typescript Get team permissions with role context
async getTeamPermissionsForUser(
  userId: string, 
  teamId: string
): Promise<Array<{ permission: Permission, role: Role }>> {
  const qb = this.em.createQueryBuilder(Permission, 'p');
  
  qb.innerJoin('p.roles', 'r')
    .innerJoin('r.teamMembershipRoles', 'tmr')
    .innerJoin('tmr.teamMembership', 'tm')
    .leftJoin('tm.team', 't')
    .leftJoin('t.teamAvailablePermissions', 'tap')
    .leftJoin('tap.permission', 'availablePerm')
    .where({
      'tm.user': userId,
      'tm.team': teamId,
      $or: [
        { 'p.scope': PermissionScope.TEAM },
        { 
          'p.scope': PermissionScope.ORG,
          'availablePerm.id': { $eq: 'p.id' }
        }
      ]
    })
    .select(['p.*', 'r.name as roleName', 'r.id as roleId']);
  
  return qb.getResult();
}
```

## Business Rules

### Core Business Rules

<AccordionGroup>
<Accordion title="Organization Ownership">
- Each organization must have exactly one Owner
- Owner role cannot be removed while organization exists
- Owner can assign any role including other Owner roles
- Owner assignment requires special validation
</Accordion>

<Accordion title="Role Assignment Constraints">
- System roles can be assigned across any organization
- Custom roles restricted to their parent organization
- Team roles require valid team membership
- ORG permissions in team roles must be in `team_available_permissions`
</Accordion>

<Accordion title="Multi-Organization Isolation">
- Users can have different roles in different organizations
- RLS policies enforce organization-level data isolation
- Cross-organization data access strictly prohibited
- All operations scoped to current organization context
</Accordion>

<Accordion title="Permission Inheritance">
- Team members inherit team-scoped permissions from team roles
- ORG-scoped permissions in team roles require team availability
- Organizational roles provide organization-wide permissions
- Permission conflicts resolved through union (most permissive)
</Accordion>

<Accordion title="Invitation Business Rules">
- Only one pending invitation per email per organization
- Invitations expire after 7 days by default
- Role assignments validated at invitation creation and acceptance
- Team role assignments require team membership creation
- Expired invitations cannot be accepted or modified
</Accordion>
</AccordionGroup>

## RLS Integration

### Row-Level Security Policies

The RBAC system leverages PostgreSQL RLS for multi-tenant data isolation:

<Steps>
<Step title="Organization Context Setting">
All authenticated requests set the organization context:
```sql
SELECT set_config('app.organization_id', $1, true);
```
</Step>

<Step title="Table-Level RLS Policies">
Each RBAC table has RLS policies that filter by organization:
```sql
-- UserOrgRole RLS Policy
CREATE POLICY user_org_role_tenant_isolation ON user_org_role
  USING (organization_id = current_setting('app.organization_id')::uuid);

-- TeamMembershipRole RLS Policy  
CREATE POLICY team_membership_role_tenant_isolation ON team_membership_role
  USING (organization_id = current_setting('app.organization_id')::uuid);

-- Invitation RLS Policy
CREATE POLICY invitation_tenant_isolation ON invitation
  USING (organization_id = current_setting('app.organization_id')::uuid);
```
</Step>

<Step title="Bypass Context for System Operations">
System operations (like invitation acceptance) use bypass context:
```typescript
await this.tenantContext.executeWithBypass(async (bypassEm) => {
  // Operations here bypass RLS policies
  await this.applyRbacAssignments(bypassEm, user, roles);
});
```
</Step>

<Step title="Cross-Organization Operations">
Cross-organization queries explicitly bypass RLS with proper authorization:
```typescript
// System admin operations across organizations
await this.tenantContext.executeWithBypass(async (bypassEm) => {
  const users = await bypassEm.find(User, { isSystemAdmin: true });
  // Process system-wide operations
});
```
</Step>
</Steps>

<Warning>
Always use `TenantContext.executeWithBypass()` for invitation acceptance and other cross-organization operations. Direct use of `this.em` will fail due to RLS policies when no organization context is set.
</Warning>
