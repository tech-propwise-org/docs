This document contains the complete specification for the RBAC (Role-Based Access Control) system, including entity relationships, permission resolution, system roles, and multi-organization support.

## Architecture overview

### Design principles

The RBAC system follows these core principles:

<AccordionGroup>
<Accordion title="Unified Role & Permission System">
- Single `Role` table for both org-level and team-level roles
- Single `Permission` table for all permissions
- `scope` field differentiates between ORG and TEAM levels
</Accordion>

<Accordion title="System vs Custom Roles">
- System roles have `organization = NULL` (global/seeded)
- Custom roles have `organization = specific org ID`
- System roles: Owner, Admin, Basic, Team Leader, Agent
</Accordion>

<Accordion title="Explicit Junction Tables">
- `UserOrgRole` - User ↔ Role assignments in Organization context
- `TeamMembershipRole` - TeamMembership ↔ Role assignments
- Required for multi-org support (user can have different roles in different orgs)
- Includes metadata: `assigned_by`, `created_at`, `organization_id` (for RLS)
</Accordion>

<Accordion title="Multi-Tenancy Support">
- Users can belong to multiple organizations
- Same user can have different roles in different organizations
- RLS (Row-Level Security) enforced at database level
</Accordion>

<Accordion title="Permission Inheritance">
- Team roles can have both TEAM-scoped and ORG-scoped permissions
- Team-available permissions controlled via `team_available_permissions`
</Accordion>
</AccordionGroup>

## Core entities

### User

**Purpose**: Represents a user account in the system.

```typescript
User
├── id: uuid
├── first_name, last_name: string
├── email: string (globally unique among active users)
├── password: string (hashed)
├── phone_number?: string
├── avatar_url?: string (public URL to avatar image, stored as WebP on R2)
├── languages?: string[] (array of language codes)
├── preferred_language?: string
├── is_verified: boolean
├── is_active: boolean
├── is_system_admin: boolean (platform-level admin, distinct from RBAC roles)
├── is_deleted: boolean (soft delete)
├── selected_organization_id?: uuid → Organization
├── Auth Fields:
│   ├── refresh_token?: string
│   ├── token_version: number (default: 1, incremented on token invalidation)
│   ├── verification_code?: string
│   ├── verification_code_expires?: timestamp
│   ├── verification_email_attempts: number (default: 0)
│   ├── last_verification_email_sent_at?: timestamp
│   ├── reset_password_code?: string
│   └── reset_password_code_expires?: timestamp
├── Relationships:
│   ├── organizations → Organization[] (via organization_users)
│   ├── orgRoles → UserOrgRole[] (roles in orgs)
│   └── teamMemberships → TeamMembership[]
└── Timestamps: created_at, updated_at

Indexes:
├── (first_name, last_name) [composite]
└── (email, is_deleted) [unique, where: is_deleted = false]
```

<Note>
- User can belong to multiple organizations
- Email is globally unique among non-deleted users (enforced by unique index on `[email, is_deleted]` where `is_deleted = false`)
- `selected_organization_id` tracks current org context
- Access roles via `orgRoles` collection (not direct `roles`)
- `is_system_admin` is a platform-level flag independent of per-org RBAC roles
</Note>

### Organization

**Module Location:** `/modules/organization/` (separate from RBAC)

**Purpose**: Represents a tenant organization. Organization serves as the **tenant boundary** for multi-tenancy and is conceptually a peer to User, not a component of RBAC.

```typescript
Organization
├── id: uuid
├── name: string
├── website?: string
├── email?: string
├── vat?: string
├── avatar_url?: string (public URL to org logo, stored as WebP on R2)
├── address?: jsonb (embedded Address object)
├── owner_id: uuid → User (organization owner)
├── Relationships:
│   ├── users → User[] (ManyToMany, owner side, via organization_users)
│   ├── owner → User (ManyToOne)
│   ├── leads → Lead[] (OneToMany)
│   ├── contacts → Contact[] (OneToMany)
│   ├── deals → Deal[] (OneToMany)
│   └── companies → Company[] (OneToMany)
└── Timestamps: created_at, updated_at, is_deleted
```

<Info>
Teams, Roles, and Invitations each reference Organization via their own `@ManyToOne` relationships, but Organization does not declare inverse `@OneToMany` collections for these. CRM entities (Lead, Contact, Deal, Company) do have inverse relationships on Organization.
</Info>

**Key Rules**:
- Each organization has exactly one owner
- Owner is automatically assigned "Owner" role on creation
- System roles (organization = NULL) are shared across all orgs
- Soft delete via `isDeleted` flag with `@Filter` decorator

### Role (Unified)

**Purpose**: Represents a role that can be assigned to users. Unified table for both org-level and team-level roles.

```typescript
Role
├── id: uuid
├── name: string
├── description?: string
├── color?: string
├── scope: RoleScope (ORG | TEAM)
├── is_system: boolean (true for seeded roles)
├── organization_id?: uuid → Organization (NULL for system roles)
├── team_id?: uuid → Team (NULL for org roles)
├── Relationships:
│   ├── permissions → Permission[] (via role_permissions)
│   ├── userOrgRoles → UserOrgRole[]
│   └── teamMembershipRoles → TeamMembershipRole[]
└── Timestamps: created_at, updated_at

Indexes:
├── (name, organization_id, scope) [unique]
└── (organization_id)
```

### Permission (Unified)

**Purpose**: Represents a permission that can be granted to roles.

```typescript
Permission
├── id: uuid
├── key: string (unique identifier like 'leads.create')
├── name: string (display name)
├── description?: string
├── scope: PermissionScope (ORG | TEAM)
├── category: string (grouping for UI)
├── Relationships:
│   └── roles → Role[] (via role_permissions)
└── Timestamps: created_at, updated_at

Indexes:
└── (key) [unique]
```

## Explicit junction tables

### UserOrgRole

**Purpose**: Junction table for User ↔ Role assignments within Organization context.

```typescript
UserOrgRole
├── id: uuid
├── user_id: uuid → User
├── role_id: uuid → Role
├── organization_id: uuid → Organization (for RLS)
├── assigned_by_id: uuid → User
├── assigned_at: timestamp
└── Timestamps: created_at, updated_at

Indexes:
├── (user_id, organization_id)
├── (role_id)
└── (user_id, role_id, organization_id) [unique]
```

### TeamMembershipRole

**Purpose**: Junction table for TeamMembership ↔ Role assignments.

```typescript
TeamMembershipRole
├── id: uuid
├── team_membership_id: uuid → TeamMembership
├── role_id: uuid → Role
├── organization_id: uuid → Organization (for RLS)
├── assigned_by_id: uuid → User
├── assigned_at: timestamp
└── Timestamps: created_at, updated_at

Indexes:
├── (team_membership_id)
├── (role_id)
└── (team_membership_id, role_id) [unique]
```

## Permission resolution

Permission resolution follows a hierarchical structure:

<Steps>
<Step title="Collect org-level permissions">
Gather all permissions from user's org-level roles within the current organization.
</Step>
<Step title="Collect team-level permissions">
For each team the user belongs to, gather permissions from their team-level roles.
</Step>
<Step title="Apply scope filtering">
- ORG-scoped permissions apply organization-wide
- TEAM-scoped permissions apply only within the specific team context
</Step>
<Step title="Merge and deduplicate">
Combine all collected permissions, removing duplicates.
</Step>
</Steps>

```typescript
async resolveUserPermissions(userId: string, organizationId: string): Promise<Permission[]> {
  // Get org-level permissions
  const orgPermissions = await this.em.createQueryBuilder(Permission)
    .select('p.*')
    .from(Permission, 'p')
    .innerJoin(RolePermission, 'rp', 'p.id = rp.permission_id')
    .innerJoin(Role, 'r', 'rp.role_id = r.id')
    .innerJoin(UserOrgRole, 'uor', 'r.id = uor.role_id')
    .where('uor.user_id = ? AND uor.organization_id = ?', [userId, organizationId])
    .getMany();

  // Get team-level permissions
  const teamPermissions = await this.em.createQueryBuilder(Permission)
    .select('p.*')
    .from(Permission, 'p')
    .innerJoin(RolePermission, 'rp', 'p.id = rp.permission_id')
    .innerJoin(Role, 'r', 'rp.role_id = r.id')
    .innerJoin(TeamMembershipRole, 'tmr', 'r.id = tmr.role_id')
    .innerJoin(TeamMembership, 'tm', 'tmr.team_membership_id = tm.id')
    .where('tm.user_id = ? AND tm.organization_id = ?', [userId, organizationId])
    .getMany();

  // Merge and deduplicate
  const allPermissions = [...orgPermissions, ...teamPermissions];
  return Array.from(new Map(allPermissions.map(p => [p.id, p])).values());
}
```

## System roles

System roles are seeded at application startup and shared across all organizations:

<Tabs>
<Tab title="Owner">
**Scope**: ORG  
**Description**: Organization owner with full administrative access
- All ORG-scoped permissions
- Cannot be deleted or modified
- Automatically assigned to organization creator
</Tab>
<Tab title="Admin">
**Scope**: ORG  
**Description**: Administrative role with most permissions except ownership transfer
- Most ORG-scoped permissions (excluding sensitive owner actions)
- Can manage users and roles
</Tab>
<Tab title="Basic">
**Scope**: ORG  
**Description**: Default role for organization members
- Basic ORG-scoped permissions
- View access to most resources
</Tab>
<Tab title="Team Leader">
**Scope**: TEAM  
**Description**: Team management role
- TEAM-scoped management permissions
- Some ORG-scoped view permissions
</Tab>
<Tab title="Agent">
**Scope**: TEAM  
**Description**: Standard team member role
- TEAM-scoped operational permissions
- Limited ORG-scoped view permissions
</Tab>
</Tabs>

## Invitation system

The invitation system allows admins to onboard new or existing users into an organization with pre-configured org roles and team assignments.

### Invitation entity

```typescript
Invitation
├── id: uuid
├── email: string
├── status: InvitationStatus (pending, completed, expired, cancelled)
├── token: string (secure token for acceptance)
├── expires_at?: timestamp (nullable)
├── invited_by_id: uuid → User (ManyToOne)
├── accepted_by_id?: uuid → User (ManyToOne, nullable)
├── accepted_at?: timestamp
├── organization_id: uuid → Organization (ManyToOne)
├── org_role_assignments: jsonb ({roleId, roleName}[])
├── team_id?: uuid → Team (ManyToOne, nullable)
├── team_role_assignments: jsonb ({roleId, roleName, teamId, teamName}[])
├── welcome_message?: string
└── Timestamps: created_at, updated_at

Indexes:
├── (invited_by_id)
└── (email, organization_id) [unique, where: status = 'pending']
```

<Note>
RBAC assignments are stored in two separate JSONB fields (`orgRoleAssignments` and `teamRoleAssignments`) rather than a single field. Each includes the role name alongside the ID for display purposes without requiring a join. The `team` ManyToOne relationship stores the primary team assignment separately.
</Note>

### Invitation flow

<Steps>
<Step title="Admin creates invitation">
```json
POST /invitations
{
  "email": "newuser@example.com",
  "orgRoles": ["admin-role-uuid"],
  "teamAssignments": [
    { "teamId": "sales-team-uuid", "roleIds": ["agent-role-uuid"] }
  ],
  "welcomeMessage": "Welcome to the team!"
}
```

Creates invitation with a secure token, `expires_at = 7 days`, `status = 'pending'`, and RBAC assignment snapshots.
</Step>
<Step title="Email sent with invitation link">
The system sends an email containing a link like `https://app.propwise.com/accept-invitation?token=xxx`.
</Step>
<Step title="User accepts invitation">

**New user** (public endpoint):

`POST /invitations/:token/accept` with `firstName`, `lastName`, `password`, `confirmPassword`. System creates account, adds to org, applies RBAC assignments.

**Existing user** (identity token endpoint):

`POST /invitations/:token/accept-existing` requires `IdentityTokenGuard`. System adds existing user to org and applies RBAC assignments.
</Step>
</Steps>

### RBAC assignment on acceptance

<Warning>
Invitation acceptance runs on public/identity-token endpoints that have **no RLS organization context**. All RBAC tables have RLS policies that deny access when `app.organization_id` is not set. The `applyInvitationRbacAssignments` method **must receive the bypassed EntityManager** from the calling `executeWithBypass` context — never `this.em` or injected repositories.
</Warning>

```typescript
private async applyInvitationRbacAssignments(
  em: EntityManager, user: User, invitation: Invitation,
): Promise<void> {
  if (invitation.orgRoleAssignments.length > 0) {
    const roles = await em.find(Role, { id: { $in: roleIds } });
    const userOrgRoles = rolesToCreate.map((role) =>
      em.create(UserOrgRole, { user, role, organization, assignedBy }),
    );
    em.persist(userOrgRoles);
  } else {
    const basicRole = await em.findOne(Role, { name: 'Basic', isSystem: true });
    if (basicRole) {
      em.persist(em.create(UserOrgRole, { user, role: basicRole, organization, assignedBy }));
    }
  }

  if (invitation.teamRoleAssignments.length > 0) {
    const team = await em.findOne(Team, { id: teamId, organization: organizationId });
    const membership = em.create(TeamMembership, { user, team, organization });
    em.persist(membership);

    const teamRoles = await em.find(Role, { id: { $in: teamRoleIds }, scope: 'TEAM' });
    const teamMembershipRoles = teamRoles.map((role) =>
      em.create(TeamMembershipRole, { teamMembership: membership, role, organization, assignedBy }),
    );
    em.persist(teamMembershipRoles);
  }

  await em.flush();
}
```

```typescript User's roles in org
async getUserRolesInOrganization(userId: string, organizationId: string): Promise<Role[]> {
  return this.em.createQueryBuilder(Role)
    .select('r.*')
    .from(Role, 'r')
    .innerJoin(UserOrgRole, 'uor', 'r.id = uor.role_id')
    .where('uor.user_id = ? AND uor.organization_id = ?', [userId, organizationId])
    .getMany();

```

```typescript Team members with roles
async getTeamMembersWithRoles(teamId: string): Promise<any[]> {
  return this.em.createQueryBuilder()
    .select(['u.id', 'u.first_name', 'u.last_name', 'r.name as role_name'])
    .from(User, 'u')
    .innerJoin(TeamMembership, 'tm', 'u.id = tm.user_id')

    .innerJoin(Role, 'r', 'tmr.role_id = r.id')
    .where('tm.team_id = ?', [teamId])
    .getRawMany();
```

## Business rules

### Invitation rules

| Rule | Description |
| --- | --- |
| **Expiration** | Invitations expire after 7 days |
| **One-time use** | Invitation token can only be used once |
| **Email uniqueness** | Cannot invite email that already exists in organization |
| **Role validation** | roleIds in invitation must be valid for organization |
| **Team validation** | teamId in invitation must exist in organization |

### RBAC rules

| Rule | Description |
| --- | --- |
| **System role immutability** | System roles cannot be modified or deleted |
| **Owner uniqueness** | Only one Owner role per organization |
| **Permission scope validation** | TEAM roles cannot have ORG-scoped permissions unless explicitly allowed |
| **Multi-org isolation** | User roles in one org don't affect permissions in another |

## RLS integration

All RBAC tables implement Row-Level Security (RLS) policies:

<Tabs>
<Tab title="Organization context">
RLS policies check `app.organization_id` session variable to ensure users only access data from their current organization context.
</Tab>
<Tab title="Bypass mechanism">
Some operations (like invitation acceptance) require bypassing RLS. Use `TenantContext.executeWithBypass()` for these cases.
</Tab>
<Tab title="Policy examples">
```sql
-- UserOrgRole RLS policy
CREATE POLICY user_org_role_org_isolation ON user_org_roles
FOR ALL TO app_user
USING (organization_id = current_setting('app.organization_id')::uuid);

-- Role RLS policy
CREATE POLICY role_org_isolation ON roles
FOR ALL TO app_user
USING (organization_id IS NULL OR organization_id = current_setting('app.organization_id')::uuid);
```
</Tab>
</Tabs>
