The invitation system enables secure user onboarding with predefined access controls, supporting both new user registration and existing user integration across multiple organizations within the complete RBAC framework.

<Info>
The invitation system provides secure token-based user onboarding with comprehensive role and team assignment capabilities, supporting both new user registration and existing user integration across multiple organizations.
</Info>

<Warning>
Invitation acceptance runs on public/identity-token endpoints that have **no RLS organization context**. All RBAC tables have RLS policies that deny access when `app.organization_id` is not set. The invitation system **must use RLS bypass context** for all RBAC assignments.
</Warning>

## System overview

The invitation system is a critical component of the RBAC architecture, providing a secure and scalable method for onboarding new users with appropriate access controls. It integrates seamlessly with the broader RBAC system to ensure proper permission resolution and organizational context.

Key capabilities include:
- Secure token-based invitation delivery
- Role and team assignments during invitation creation
- Support for both new user registration and existing user integration
- Multi-organization invitation management
- Comprehensive audit trails and status tracking
- Integration with system roles and custom organizational roles

## Core invitation entity

The invitation system manages the complete user onboarding lifecycle through the core `Invitation` entity:

```typescript
Invitation
├── id: uuid
├── email: string
├── status: InvitationStatus (pending, completed, expired, cancelled)
├── token: string (secure, 64-character hex string)
├── expires_at?: timestamp (default: 7 days from creation)
├── invited_by_id: uuid → User (ManyToOne)
├── accepted_by_id?: uuid → User (ManyToOne, nullable)
├── accepted_at?: timestamp
├── cancelled_by_id?: uuid → User (ManyToOne, nullable)
├── cancelled_at?: timestamp
├── resent_by_id?: uuid → User (ManyToOne, nullable)
├── resent_at?: timestamp
├── email_sent: boolean (default: true)
├── organization_id: uuid → Organization (ManyToOne, required)
├── org_role_assignments: jsonb ({roleId, roleName}[])
├── team_id?: uuid → Team (ManyToOne, nullable, primary team assignment)
├── team_role_assignments: jsonb ({roleId, roleName, teamId, teamName, roleIds}[])
├── welcome_message?: string
├── Relationships:
│   ├── organization → Organization (ManyToOne)
│   ├── invitedBy → User (ManyToOne)
│   ├── acceptedBy? → User (ManyToOne, nullable)
│   ├── cancelledBy? → User (ManyToOne, nullable)
│   ├── resentBy? → User (ManyToOne, nullable)
│   └── team? → Team (ManyToOne, nullable)
└── Timestamps: created_at, updated_at

Indexes:
├── (invited_by_id)
├── (organization_id)
├── (token) [unique]
├── (status)
└── (email, organization_id) [unique, where: status = 'pending']
```

## Invitation status flow

The invitation lifecycle is managed through a well-defined status progression:

<Tabs>
<Tab title="pending">
**Initial state** when invitation is created and email sent:
- **Transitions to**: completed, expired, cancelled
- **Actions available**: Accept invitation, cancel invitation, extend expiry

```typescript
if (invitation.expiresAt && new Date() > invitation.expiresAt) {
  invitation.status = InvitationStatus.EXPIRED;
}
```
</Tab>

<Tab title="completed">
**Final state** when user successfully accepts invitation:
- **Transitions from**: pending
- **Actions available**: View completion details only

```typescript
invitation.status = InvitationStatus.COMPLETED;
invitation.acceptedBy = user;
invitation.acceptedAt = new Date();
```
</Tab>

<Tab title="expired">
**Final state** when invitation expires without acceptance:
- **Transitions from**: pending (automatic after expires_at)
- **Actions available**: Create new invitation if needed

<Warning>
Expired invitations cannot be accepted. A new invitation must be created.
</Warning>
</Tab>

<Tab title="cancelled">
**Final state** when invitation is manually cancelled:
- **Transitions from**: pending
- **Actions available**: Create new invitation if needed

```typescript
invitation.status = InvitationStatus.CANCELLED;
invitation.cancelledBy = { id: cancelledById } as User;
invitation.cancelledAt = new Date();
await this.em.persistAndFlush(invitation);
```
</Tab>
</Tabs>

## RBAC assignment application

The core logic for applying invitation-based role assignments with proper RBAC integration:

```typescript
private async applyInvitationRbacAssignments(
  bypassEm: EntityManager,
  user: User,
  invitation: Invitation
): Promise<void> {
  // 1. Add user to organization
  await this.addUserToOrganization(bypassEm, user, invitation);
  
  // 2. Apply organizational role assignments
  for (const roleAssignment of invitation.orgRoleAssignments) {
    const userOrgRole = bypassEm.create(UserOrgRole, {
      user,
      role: { id: roleAssignment.roleId },
      organization: invitation.organization,
      assignedBy: invitation.invitedBy
    });
    await bypassEm.persist(userOrgRole);
  }
  
  // 3. Apply team role assignments
  for (const teamAssignment of invitation.teamRoleAssignments) {
    // Create team membership
    const teamMembership = bypassEm.create(TeamMembership, {
      user,
      team: { id: teamAssignment.teamId },
      organization: invitation.organization,
      addedBy: invitation.invitedBy,
      joinedAt: new Date()
    });
    await bypassEm.persist(teamMembership);

    // Apply team roles
    for (const roleId of teamAssignment.roleIds) {
      const teamMembershipRole = bypassEm.create(TeamMembershipRole, {
        teamMembership,
        role: { id: roleId },
        organization: invitation.organization,
        assignedBy: invitation.invitedBy
      });
      await bypassEm.persist(teamMembershipRole);
    }
  }
  
  // 4. Complete invitation
  invitation.status = InvitationStatus.COMPLETED;
  invitation.acceptedBy = user;
  invitation.acceptedAt = new Date();
  
  await bypassEm.flush();
}
```

## RLS bypass requirements

The invitation system requires special handling of Row-Level Security (RLS) policies:

<Warning>
**Critical RLS Considerations**:

1. **Public endpoint context**: Invitation acceptance endpoints have no organization context
2. **RLS policy blocking**: All RBAC tables deny access when `app.organization_id` is NULL
3. **Bypass requirement**: RBAC assignments **must** use bypass context during invitation acceptance
4. **Security implications**: Bypass context removes multi-tenant isolation temporarily
</Warning>

### RLS bypass implementation

```typescript
class InvitationService {
  async acceptInvitation(token: string, userData: AcceptInvitationDto): Promise<User> {
    // Public endpoint - no RLS context
    const invitation = await this.validateInvitationToken(token);
    
    const user = await this.authService.register({
      email: invitation.email,
      ...userData
    });
    
    // Apply RBAC assignments with RLS bypass
    await this.tenantContext.executeWithBypass(async (bypassEm) => {
      await this.applyInvitationRbacAssignments(bypassEm, user, invitation);
    });
    
    return user;
  }
}
```

## Token security and validation

The invitation system implements comprehensive token security measures:

```typescript
private async validateInvitationToken(token: string): Promise<Invitation> {
  // Find invitation by token
  const invitation = await this.em.findOne(
    Invitation,
    { token },
    { populate: ['organization', 'invitedBy', 'team'] }
  );

  if (!invitation) {
    throw new BadRequestException('Invalid invitation token');
  }

  // Check status
  if (invitation.status !== InvitationStatus.PENDING) {
    throw new BadRequestException('Invitation is no longer valid');
  }

  // Check expiry
  if (invitation.expiresAt && new Date() > invitation.expiresAt) {
    invitation.status = InvitationStatus.EXPIRED;
    await this.em.persistAndFlush(invitation);
    throw new BadRequestException('Invitation has expired');
  }

  return invitation;
}

private async generateUniqueToken(): Promise<string> {
  let token: string;
  let attempts = 0;
  const maxAttempts = 10;

  do {
    token = crypto.randomBytes(32).toString('hex'); // 256 bits entropy
    attempts++;
    
    if (attempts > maxAttempts) {
      throw new Error('Failed to generate unique token after maximum attempts');
    }
  } while (!(await this.isTokenUnique(token)));

  return token;
}
```

<Note>
**Token Security Features**:
- 256 bits of entropy using crypto.randomBytes(32)
- Globally unique tokens enforced by database constraint
- Automatic expiry validation during token checks
- Single-use tokens that cannot be reused after acceptance
- Retry mechanism for token generation to handle collisions
</Note>

## Business rules and validation

The invitation system operates under comprehensive business rules ensuring security and consistency:

<AccordionGroup>
<Accordion title="Email uniqueness constraints">
- Only one pending invitation per email per organization
- Users can have multiple pending invitations across different organizations
- Completed or expired invitations don't block new invitations
- Enforced via unique index on `(email, organization_id)` where `status = 'pending'`

```sql
-- Unique constraint implementation
CREATE UNIQUE INDEX idx_invitations_email_org_pending 
ON invitations(email, organization_id) 
WHERE status = 'pending';
```
</Accordion>

<Accordion title="Expiration policy">
- Invitations expire after 7 days by default (configurable)
- Expired invitations cannot be accepted or modified
- System provides cleanup job for expired invitations
- Admins can extend invitation expiry before expiration

```typescript
// Auto-expiry check during validation
if (invitation.expiresAt && new Date() > invitation.expiresAt) {
  invitation.status = InvitationStatus.EXPIRED;
  await this.em.persistAndFlush(invitation);
  throw new BadRequestException('Invitation has expired');
}
```
</Accordion>

<Accordion title="Role assignment validation">
All role assignments are validated at invitation creation:

- Roles must exist and be appropriate for their context (ORG/TEAM scope)
- System roles (organization = NULL) available to all organizations
- Custom roles only available within their organization
- Team role assignments require valid team membership creation

```typescript
// Role validation rules
const role = await this.em.findOne(Role, {
  id: roleAssignment.roleId,
  scope: RoleScope.ORG,
  $or: [
    { organization: organizationId }, // Custom role
    { organization: null }           // System role
  ],
  isDeleted: false
});

if (!role) {
  throw new BadRequestException(`Invalid role assignment: ${roleAssignment.roleId}`);
}
```
</Accordion>

<Accordion title="Multi-organization support">
- Users can belong to multiple organizations with different roles
- Same email can have different role assignments per organization
- Cross-organization role assignments handled via separate junction records
- Organization switching updates `selected_organization_id`


## Email delivery system

The invitation system includes a comprehensive email delivery mechanism:

```typescript
class InvitationEmailService {
  private async sendInvitationEmail(invitation: Invitation): Promise<void> {
    const emailData = {
      to: invitation.email,
      subject: `Invitation to join ${invitation.organization.name}`,
      template: 'invitation',
      context: {
        organizationName: invitation.organization.name,
        invitedByName: `${invitation.invitedBy.firstName} ${invitation.invitedBy.lastName}`,
        welcomeMessage: invitation.welcomeMessage,
        acceptUrl: `${process.env.FRONTEND_URL}/invitations/accept/${invitation.token}`,
        expiresAt: invitation.expiresAt,
        orgRoles: invitation.orgRoleAssignments.map(ra => ra.roleName),
        teamAssignments: invitation.teamRoleAssignments.map(ta => ({
          teamName: ta.teamName,
          roleNames: ta.roleNames
        }))
      }
    };

    try {
      await this.emailService.send(emailData);
      invitation.emailSent = true;
    } catch (error) {
      this.logger.error('Failed to send invitation email', {
        invitationId: invitation.id,
        email: invitation.email,
        error: error.message
      });
      
      invitation.emailSent = false;
      // Don't throw - allow invitation creation to succeed
    }
  }
}
```

## API endpoints

Complete HTTP endpoints for invitation management:

```typescript
@Controller('invitations')
@ApiTags('Invitations')
export class InvitationController {
  @Post()
  @UseGuards(JwtAuthGuard, PermissionGuard)
  @RequirePermissions('invitations:create')
  @ApiOperation({ summary: 'Create a new invitation' })
  async createInvitation(
    @Body() dto: CreateInvitationDto,
    @CurrentUser() user: User,
    @CurrentOrganization() organizationId: string
  ): Promise<Invitation> {
    return this.invitationService.createInvitation({ 
      ...dto, 
      invitedById: user.id,
      organizationId 
    });
  }

  @Get(':token')
  @ApiOperation({ summary: 'Get invitation details by token (public endpoint)' })
  async getInvitationByToken(@Param('token') token: string): Promise<InvitationDetailsDto> {
    const invitation = await this.invitationService.getInvitationByToken(token);
    
    return {
      id: invitation.id,
      email: invitation.email,
      organizationName: invitation.organization.name,
      invitedByName: `${invitation.invitedBy.firstName} ${invitation.invitedBy.lastName}`,
      welcomeMessage: invitation.welcomeMessage,
      expiresAt: invitation.expiresAt,
      orgRoles: invitation.orgRoleAssignments,
      teamAssignments: invitation.teamRoleAssignments
    };
  }

  @Post('accept/:token')
  @ApiOperation({ summary: 'Accept invitation - new user registration' })
  async acceptInvitation(
    @Param('token') token: string,
    @Body() userData: AcceptInvitationDto
  ): Promise<AuthResponse> {
    const user = await this.invitationService.acceptInvitation(token, userData);
    const authTokens = await this.authService.generateTokens(user);
    
    return { 
      user: {
        id: user.id,
        email: user.email,
        firstName: user.firstName,
        lastName: user.lastName
      },
      accessToken: authTokens.accessToken,
      refreshToken: authTokens.refreshToken
    };
  }

  @Post('accept-existing/:token')
  @UseGuards(JwtAuthGuard)
  @ApiOperation({ summary: 'Accept invitation - existing user' })
  async acceptInvitationExisting(
    @Param('token') token: string,
    @CurrentUser() user: User
  ): Promise<InvitationAcceptanceResponse> {
    const invitation = await this.invitationService.acceptInvitationExisting(token, user);
    
    return { 
      success: true, 
      invitation: {
        id: invitation.id,
        organizationName: invitation.organization.name,
        completedAt: invitation.acceptedAt
      }
    };
  }

  @Patch(':id/cancel')
  @UseGuards(JwtAuthGuard, PermissionGuard)
  @RequirePermissions('invitations:cancel')
  @ApiOperation({ summary: 'Cancel a pending invitation' })

    @Param('id') id: string,
    @CurrentUser() user: User
  ): Promise<{ success: boolean }> {
    await this.invitationService.cancelInvitation(id, user.id);
    return { success: true };
  }

  @Post(':id/resend')
  @UseGuards(JwtAuthGuard, PermissionGuard)
  @RequirePermissions('invitations:resend')

  }
```
