The invitation system allows administrators to onboard new or existing users into an organization with pre-configured organizational roles and team assignments. This document covers the complete invitation workflow, RBAC integration, and technical implementation details.

## Overview

The invitation system is a core component of the RBAC system that enables secure user onboarding with predefined access controls. It supports both new user registration and existing user integration across multiple organizations.

<Info>
Invitations are designed to work seamlessly with the multi-tenant RBAC system, ensuring proper role assignments and team memberships upon acceptance.
</Info>

## Invitation entity

The `Invitation` entity stores all necessary information for user onboarding and RBAC assignment:

```typescript
Invitation
├── id: uuid
├── email: string
├── status: InvitationStatus (pending, completed, expired, cancelled)
├── token: string (secure token for acceptance)
├── expires_at?: timestamp (nullable)
├── invited_by_id: uuid → User (ManyToOne)
├── accepted_by_id?: uuid → User (ManyToOne, nullable)
├── accepted_at?: timestamp
├── organization_id: uuid → Organization (ManyToOne)
├── org_role_assignments: jsonb ({roleId, roleName}[])
├── team_id?: uuid → Team (ManyToOne, nullable)
├── team_role_assignments: jsonb ({roleId, roleName, teamId, teamName}[])
├── welcome_message?: string
└── Timestamps: created_at, updated_at

Indexes:
├── (invited_by_id)
└── (email, organization_id) [unique, where: status = 'pending']
```

<Note>
RBAC assignments are stored in two separate JSONB fields (`org_role_assignments` and `team_role_assignments`) rather than a single field. Each includes the role name alongside the ID for display purposes without requiring a join. The `team` ManyToOne relationship stores the primary team assignment separately.
</Note>

## Invitation flow

<Steps>
<Step title="Admin creates invitation">
An administrator creates an invitation by specifying the target email, organizational roles, and optional team assignments:

```json
POST /invitations
{
  "email": "newuser@example.com",
  "orgRoles": ["admin-role-uuid"],
  "teamAssignments": [
    { "teamId": "sales-team-uuid", "roleIds": ["agent-role-uuid"] }
  ],
  "welcomeMessage": "Welcome to the team!"
}
```

This creates an invitation with:
- A secure token for acceptance
- `expires_at = 7 days` from creation
- `status = 'pending'`
- RBAC assignment snapshots stored in JSONB fields
</Step>

<Step title="Email sent with invitation link">
The system automatically sends an email containing a secure invitation link:
```
https://app.propwise.com/accept-invitation?token=xxx
```
The email includes the welcome message and organization details.
</Step>

<Step title="User accepts invitation">
The invitation can be accepted through two different flows:

**New user acceptance** (public endpoint):
- `POST /invitations/:token/accept` 
- Requires: `firstName`, `lastName`, `password`, `confirmPassword`
- Creates new user account, adds to organization, applies RBAC assignments

**Existing user acceptance** (identity token endpoint):
- `POST /invitations/:token/accept-existing`
- Requires: `IdentityTokenGuard` authentication
- Adds existing user to organization and applies RBAC assignments
</Step>
</Steps>

## Invitation status management

The invitation system tracks the lifecycle through these statuses:

<Tabs>
<Tab title="pending">
**Initial State**
- Created when invitation is sent
- User can accept or reject the invitation
- Can be cancelled by admin before acceptance
</Tab>
<Tab title="completed">
**Terminal State**
- User successfully accepted the invitation
- RBAC assignments have been applied
- User is now a member of the organization
</Tab>
<Tab title="expired">
**Terminal State**
- Invitation expired after 7 days
- Can be extended by admin (creates new invitation)
- Token is no longer valid for acceptance
</Tab>
<Tab title="cancelled">
**Terminal State**
- Admin cancelled the invitation before acceptance
- Token is immediately invalidated
- User cannot accept the invitation
</Tab>
</Tabs>

## RBAC assignment on acceptance

<Warning>
Invitation acceptance runs on public/identity-token endpoints that have **no RLS organization context**. All RBAC tables have RLS policies that deny access when `app.organization_id` is not set. The `applyInvitationRbacAssignments` method **must receive the bypassed EntityManager** from the calling `executeWithBypass` context — never `this.em` or injected repositories.
</Warning>

When an invitation is accepted, the system applies the pre-configured RBAC assignments:

```typescript
private async applyInvitationRbacAssignments(
  em: EntityManager, 
  user: User, 
  invitation: Invitation,
): Promise<void> {
  // Apply organizational role assignments
  if (invitation.orgRoleAssignments.length > 0) {
    const roleIds = invitation.orgRoleAssignments.map(ra => ra.roleId);
    const roles = await em.find(Role, { id: { $in: roleIds } });
    const userOrgRoles = roles.map((role) =>
      em.create(UserOrgRole, { 
        user, 
        role, 
        organization: invitation.organization, 
        assignedBy: invitation.invitedBy 
      }),
    );
    em.persist(userOrgRoles);
  } else {
    // Default to Basic role if no roles specified
    const basicRole = await em.findOne(Role, { 
      name: 'Basic', 
      isSystem: true 
    });
    if (basicRole) {
      em.persist(em.create(UserOrgRole, { 
        user, 
        role: basicRole, 
        organization: invitation.organization, 
        assignedBy: invitation.invitedBy 
      }));
    }
  }

  // Apply team role assignments
  if (invitation.teamRoleAssignments.length > 0) {
    for (const teamAssignment of invitation.teamRoleAssignments) {
      const team = await em.findOne(Team, { 
        id: teamAssignment.teamId, 
        organization: invitation.organization.id 
      });
      
      if (team) {
        const membership = em.create(TeamMembership, { 
          user, 
          team, 
          organization: invitation.organization 
        });
        em.persist(membership);

        const teamRoleIds = teamAssignment.roleIds || [];
        if (teamRoleIds.length > 0) {
          const teamRoles = await em.find(Role, { 
            id: { $in: teamRoleIds }, 
            scope: 'TEAM' 
          });
          
          const teamMembershipRoles = teamRoles.map((role) =>
            em.create(TeamMembershipRole, { 
              teamMembership: membership, 
              role, 
              organization: invitation.organization, 
              assignedBy: invitation.invitedBy 
            }),
          );
          em.persist(teamMembershipRoles);
        }
      }
    }
  }

  await em.flush();
}
```

## Business rules

### Invitation validation rules

<AccordionGroup>
<Accordion title="Expiration Policy">
- Invitations expire after 7 days by default
- Expired invitations cannot be accepted
- Admins can extend expiration by creating new invitations
</Accordion>

<Accordion title="Email Uniqueness">
- Cannot invite an email that already has a pending invitation for the same organization
- Can invite emails that are already organization members (for role updates)
- Unique constraint: `(email, organization_id)` where `status = 'pending'`
</Accordion>

<Accordion title="Token Security">
- Invitation tokens are cryptographically secure
- One-time use only - token is invalidated after acceptance/rejection
- Tokens are long enough to prevent brute force attacks
</Accordion>

<Accordion title="Role Validation">
- All `roleIds` in invitation must exist and be valid for the organization
- System roles can be assigned to any organization
- Custom roles can only be assigned within their parent organization
</Accordion>
</AccordionGroup>

### RBAC assignment rules

| Rule | Description | Enforcement |
| --- | --- | --- |
| **Default role assignment** | Users without explicit role assignments get "Basic" role | Service layer |
| **Team validation** | `teamId` in invitation must exist in target organization | Database constraint |
| **Permission scope validation** | Team roles cannot have ORG-scoped permissions unless explicitly allowed | Service layer |
| **Multi-org isolation** | Role assignments are scoped to the invitation's organization | RLS policies |

```typescript
// Generate secure invitation token
private generateInvitationToken(): string {
  return crypto.randomBytes(32).toString('hex');
}

// Validate token and check expiration
private validateInvitationToken(invitation: Invitation): void {
  if (invitation.status !== InvitationStatus.PENDING) {
    throw new BadRequestException('Invitation is no longer valid');
  }
  
  if (invitation.expiresAt && invitation.expiresAt < new Date()) {
    throw new BadRequestException('Invitation has expired');
  }
}
```
</Tab>

<Tab title="Email Integration">
```typescript
// Send invitation email
private async sendInvitationEmail(invitation: Invitation): Promise<void> {
  const invitationLink = `${this.configService.get('FRONTEND_URL')}/accept-invitation?token=${invitation.token}`;
  
  await this.emailService.sendInvitationEmail({
    to: invitation.email,
    organizationName: invitation.organization.name,
    inviterName: `${invitation.invitedBy.firstName} ${invitation.invitedBy.lastName}`,
    invitationLink,
    welcomeMessage: invitation.welcomeMessage,
    expiresAt: invitation.expiresAt,
  });
}
```

## Query patterns

Common query patterns for invitation management:

```typescript Get pending invitations

  return this.em.find(Invitation, {

    status: InvitationStatus.PENDING,
    expiresAt: { $gte: new Date() }

    populate: ['invitedBy', 'organization', 'team']
  });
}
```

```typescript Get invitation with full details
async getInvitationByToken(token: string): Promise<Invitation | null> {
  return this.em.findOne(Invitation, 
    { token },
    { 
      populate: [
        'organization', 
        'invitedBy', 
        'team', 
        'acceptedBy'
      ] 
    }
  );
}
```

## Integration with RBAC system

The invitation system is tightly integrated with the broader RBAC system:

<Steps>
<Step title="Role validation">
When creating invitations, the system validates that all specified roles exist and are appropriate for the target organization and teams.
</Step>
<Step title="Permission preview">
The invitation can optionally include a preview of permissions that will be granted, helping users understand their future access levels.
</Step>
<Step title="Audit trail">
All invitation activities (creation, acceptance, cancellation) are logged for security and compliance purposes.
</Step>
<Step title="RLS bypass">
Invitation acceptance requires bypassing Row-Level Security policies since the user doesn't yet have organization context.
</Step>
</Steps>

<Warning>
Always use `TenantContext.executeWithBypass()` when applying RBAC assignments during invitation acceptance. Direct use of `this.em` or injected repositories will fail due to RLS policies.
</Warning>

## Security considerations

<AccordionGroup>
<Accordion title="Token Security">
- Invitation tokens use cryptographically secure random generation
- Tokens are long enough (32 bytes hex-encoded) to prevent brute force attacks
- One-time use policy prevents token reuse
- Tokens are invalidated immediately upon use or cancellation
</Accordion>

<Accordion title="Email Verification">
- Invitations are sent to the specified email address
- Email ownership is implicitly verified through invitation acceptance
- No additional email verification required for invited users
</Accordion>

<Accordion title="Organization Isolation">
- RLS policies ensure invitations are isolated by organization
- Users can only see/manage invitations for their current organization
- Cross-organization invitation access is prevented
</Accordion>

<Accordion title="Role Assignment Validation">
- All role assignments are validated against organization membership
- System roles can be assigned across organizations
- Custom roles are restricted to their parent organization
- Team roles require valid team membership
</Accordion>
</AccordionGroup>
