The Role-Based Access Control (RBAC) system provides comprehensive access management with unified roles, permissions, multi-organization support, and secure invitation workflows. This specification covers the complete architecture from core entities to invitation acceptance flows.

<Info>
The RBAC system follows a unified design with explicit junction tables, system vs custom roles, and comprehensive permission resolution supporting multi-tenancy across organizations and teams.
</Info>

<Warning>
All RBAC tables use Row-Level Security (RLS) policies. Invitation acceptance endpoints require **RLS bypass context** since they run on public endpoints without organization context.
</Warning>

## Architecture overview

The RBAC system is built on five core principles that ensure scalability, security, and flexibility:

<Tabs>
<Tab title="Unified roles & permissions">
**Single source of truth** for access control:
- Single `Role` table for both org-level and team-level roles
- Single `Permission` table for all permissions
- `scope` field differentiates between ORG and TEAM levels
- Eliminates duplicate role definitions across contexts
</Tab>

<Tab title="System vs custom roles">
**Flexible role management** with global and organization-specific roles:
- System roles: `organization = NULL` (global/seeded)
- Custom roles: `organization = specific org ID`
- System roles: Owner, Admin, Basic, Team Leader, Agent, AI Service, System Service
- Custom roles defined per organization for specific needs
</Tab>

<Tab title="Explicit junction tables">
**Multi-organization support** through dedicated relationship tables:
- `UserOrgRole`: User ↔ Role assignments in Organization context
- `TeamMembershipRole`: TeamMembership ↔ Role assignments
- Includes metadata: `assigned_by`, `created_at`, `organization_id`
- Enables users to have different roles across organizations
</Tab>

<Tab title="Multi-tenancy support">
**Complete organization isolation** with secure access controls:
- Users can belong to multiple organizations
- Same user can have different roles in different organizations
- RLS (Row-Level Security) enforced at database level
- Organization context switching via `selected_organization_id`
</Tab>

<Tab title="Permission inheritance">
**Hierarchical permission resolution** across organization and team scopes:
- Team roles can have both TEAM-scoped and ORG-scoped permissions
- Team-available permissions controlled via `team_available_permissions`
- Permission resolution follows inheritance chain
</Tab>
</Tabs>

## Core entities

### User entity

The central entity representing user accounts with comprehensive authentication and organization management:

```typescript
User
├── id: uuid
├── first_name, last_name: string
├── email: string (globally unique among active users)
├── password: string (hashed)
├── phone_number?: string
├── avatar_url?: string (public URL to avatar image, stored as WebP on R2)
├── languages?: { code: string; proficiency: 'BASIC' | 'CONVERSATIONAL' | 'FLUENT' | 'NATIVE' }[] (JSONB)
├── is_verified: boolean
├── is_active: boolean
├── is_system_admin: boolean (platform-level admin, distinct from RBAC roles)
├── is_ai_service_user: boolean (per-org AI agent service account; never logs in)
├── is_system_service_user: boolean (per-org System automation actor; never logs in)
├── is_deleted: boolean (soft delete)
├── selected_organization_id?: uuid → Organization
├── Auth Fields:
│   ├── refresh_token?: string
│   ├── token_version: number (default: 1, incremented on token invalidation)
│   ├── verification_code?: string
│   ├── verification_code_expires?: timestamp
│   ├── verification_email_attempts: number (default: 0)
│   ├── last_verification_email_sent_at?: timestamp
│   ├── reset_password_code?: string
│   └── reset_password_code_expires?: timestamp
├── Relationships:
│   ├── organizations → Organization[] (via organization_users; each membership has public_id + avatar colors, e.g. USER-ADNS-001)
│   ├── orgRoles → UserOrgRole[] (roles in orgs)
│   └── teamMemberships → TeamMembership[]
└── Timestamps: created_at, updated_at

Indexes:
├── (first_name, last_name) [composite]
└── (email, is_deleted) [unique, where: is_deleted = false]
```

**Key rules**:
- User can belong to multiple organizations
- User public IDs and fallback avatar colors are org-scoped membership metadata stored on `organization_users`, not on the global `user` row
- Email is globally unique among non-deleted users (enforced by unique index on `[email, is_deleted]` where `is_deleted = false`)
- `selected_organization_id` tracks current org context
- Access roles via `orgRoles` collection (not direct `roles`)
- `is_system_admin` is a platform-level flag independent of per-org RBAC roles

#### Service accounts

**Service accounts** are real `user` rows that never log in and are excluded from user listings:

## Permission resolution

The permission resolution engine determines user access through a comprehensive inheritance and aggregation system:

<Steps>
<Step title="Collect organization roles">
Gather all roles assigned to the user in the organization context:

```typescript
// Get user's organization roles
const orgRoles = await this.em.find(UserOrgRole, {
  user: userId,
  organization: organizationId
}, {
  populate: ['role', 'role.permissions']
});
```
</Step>

<Step title="Collect team roles">
Aggregate roles from all team memberships within the organization:

```typescript
// Get user's team roles across all teams
const teamRoles = await this.em.createQueryBuilder(TeamMembershipRole, 'tmr')
  .innerJoin('tmr.teamMembership', 'tm')
  .innerJoin('tmr.role', 'r')
  .leftJoinAndSelect('r.permissions', 'p')
  .where('tm.user = ?', [userId])
  .andWhere('tm.organization = ?', [organizationId])
  .andWhere('tm.is_active = true')
  .getResultList();
```
</Step>

<Step title="Permission aggregation">
Combine permissions from organization and team roles with scope inheritance:

```typescript
private aggregatePermissions(orgRoles: UserOrgRole[], teamRoles: TeamMembershipRole[]): Set<string> {
  const permissions = new Set<string>();
  
  // Add organization-level permissions
  orgRoles.forEach(orgRole => {
    orgRole.role.permissions.forEach(permission => {
      permissions.add(permission.key);
    });
  });
  
  // Add team-level permissions (both TEAM and ORG scoped)
  teamRoles.forEach(teamRole => {
    teamRole.role.permissions.forEach(permission => {
      permissions.add(permission.key);
    });
  });
  
  return permissions;
}
```
</Step>

<Step title="Permission caching">
Cache resolved permissions for optimal performance:

```typescript
// Cache permissions with TTL
await this.cacheService.set(
  `permissions:${userId}:${organizationId}`, 
  Array.from(permissions),
  300 // 5 minutes TTL
);
```
</Step>
</Steps>

### Permission resolution algorithm

The complete resolution flow follows this algorithm:

```typescript
async function resolveUserPermissions(
  userId: string, 
  organizationId: string,
  teamId?: string
): Promise<Set<string>> {
  const permissions = new Set<string>();
  
  // Step 1: Resolve organization-level permissions
  const orgRoles = await getOrgRoles(userId, organizationId);
  orgRoles.forEach(role => {
    role.permissions.forEach(p => permissions.add(p.key));
  });
  
  // Step 2: Resolve team-level permissions (if team context)
  if (teamId) {
    const teamRoles = await getTeamRoles(userId, teamId);
    teamRoles.forEach(role => {
      role.permissions.forEach(p => {
        // Team roles can have both TEAM and ORG scoped permissions
        if (p.scope === 'TEAM' || p.scope === 'ORG') {
          permissions.add(p.key);
        }
      });
    });
  } else {
    // If no team context, aggregate from all teams
    const allTeamRoles = await getAllTeamRoles(userId, organizationId);
    allTeamRoles.forEach(role => {
      role.permissions.forEach(p => permissions.add(p.key));
    });
  }
  
  return permissions;
}
```

## Invitation system

### Invitation creation workflow

<Steps>
<Step title="Permission validation">
Verify the inviting user has the required permissions:

```typescript
const hasPermission = await this.rbacService.hasPermission(

  organizationId,
```

<Step title="Role validation">
Validate all assigned roles exist and are appropriate:

```typescript
await this.validateRoleAssignments(
  organizationId, 
  dto.orgRoleAssignments, 
  dto.teamRoleAssignments

// Ensure Owner role is not assigned via invitation
const ownerRole = await this.em.findOne(Role, { 
  name: 'Owner', 
  organization: null
```

### RBAC assignment application

Core logic for applying invitation-based role assignments:

```typescript
private async applyInvitationRbacAssignments(
  bypassEm: EntityManager,
  user: User,
  invitation: Invitation
): Promise<void> {
  // 1. Add user to organization
  await this.addUserToOrganization(bypassEm, user, invitation);
  
  // 2. Apply organizational role assignments
  for (const roleAssignment of invitation.orgRoleAssignments) {
    const userOrgRole = bypassEm.create(UserOrgRole, {
      user,
      role: { id: roleAssignment.roleId },
      organization: invitation.organization,
      assignedBy: invitation.invitedBy
    });
    await bypassEm.persist(userOrgRole);
  }
  
  // 3. Apply team role assignments
  for (const teamAssignment of invitation.teamRoleAssignments) {
    // Create team membership
    const teamMembership = bypassEm.create(TeamMembership, {
      user,
      team: { id: teamAssignment.teamId },
      organization: invitation.organization,
      addedBy: invitation.invitedBy,
      joinedAt: new Date()
    });
    await bypassEm.persist(teamMembership);

    // Apply team roles
    for (const roleId of teamAssignment.roleIds) {
      const teamMembershipRole = bypassEm.create(TeamMembershipRole, {
        teamMembership,
        role: { id: roleId },
        organization: invitation.organization,
        assignedBy: invitation.invitedBy
      });
      await bypassEm.persist(teamMembershipRole);
    }
  }
  
  // 4. Complete invitation
  invitation.status = InvitationStatus.COMPLETED;
  invitation.acceptedBy = user;
  invitation.acceptedAt = new Date();
  
  await bypassEm.flush();
}
```

```typescript
@Injectable()
export class PermissionResolutionService {
  async resolvePermissions(
    userId: string,
    organizationId: string
  ): Promise<Set<string>> {
    // Check cache first
    const cached = await this.cacheService.get(
      `permissions:${userId}:${organizationId}`
    );
    if (cached) {
      return new Set(cached);
    }
    
    // Resolve from database
    const permissions = await this.resolveFromDatabase(userId, organizationId);
    
    // Cache for 5 minutes
    await this.cacheService.set(
      `permissions:${userId}:${organizationId}`,
      Array.from(permissions),
      300
    );
    
    return permissions;
  }
  
  private async resolveFromDatabase(
    userId: string,
    organizationId: string
  ): Promise<Set<string>> {
    const permissions = new Set<string>();
    
    // Get org roles
    const orgRoles = await this.em.find(UserOrgRole, {
      user: userId,
      organization: organizationId
    }, {
      populate: ['role', 'role.permissions']
    });
    
    orgRoles.forEach(uor => {
      uor.role.permissions.forEach(p => permissions.add(p.key));
    });
    
    // Get team roles
    const teamRoles = await this.em.createQueryBuilder(TeamMembershipRole, 'tmr')
      .innerJoin('tmr.teamMembership', 'tm')
      .innerJoin('tmr.role', 'r')
      .leftJoinAndSelect('r.permissions', 'p')
      .where('tm.user = ?', [userId])
      .andWhere('tm.organization = ?', [organizationId])
      .andWhere('tm.is_active = true')
      .getResultList();
    
    teamRoles.forEach(tmr => {
      tmr.role.permissions.forEach(p => permissions.add(p.key));
    });
    
    return permissions;
  }
  
  async invalidateCache(userId: string, organizationId: string): Promise<void> {
    await this.cacheService.del(`permissions:${userId}:${organizationId}`);
  }
}
```
