This specification defines the implementation for removing users from organizations while preserving CRM data and maintaining system integrity across all subsystems.

<Note>
This is an org-scoped deactivation feature, not global account deletion. Users retain their global account and CRM history while losing access to the specific organization.
</Note>

## Executive Summary

**Feature:** Remove user from organization (org-scoped deactivation)  
**User-facing label:** "Remove from organization" (not "Delete user account")  
**Endpoint:** `DELETE /v1/users/:id`  
**Core principle:** Deactivate membership and access in one org; retain CRM history

<Warning>
Do not set global `User.isDeleted` for org removal. This would remove the user everywhere and affect unique email constraints. Removal is expressed via junction soft-deletes and collection removal.
</Warning>

### Architecture Flow

```mermaid
flowchart TD
  Admin["Admin: DELETE /users/:id"]
  Guard["AuthGuard + CheckAccess system.admin"]
  Tx["UserService.remove transaction"]
  OrgLink["organizations.remove"]
  UOR["UserOrgRole isDeleted=true"]
  TM["TeamMembership + TeamMembershipRole isDeleted=true"]
  Sess["revokeAllOrgSessions"]
  Cache["permissionCache.invalidate"]
  Notify["REMOVED_FROM_ORGANIZATION"]
  TeamEvt["TEAM_MEMBER_REMOVED per team"]
  CRM["EntityStakeholder + Lead/Deal rows UNCHANGED"]
  Admin --> Guard --> Tx
  Tx --> OrgLink --> UOR --> TM --> Sess --> Cache
  Tx --> CRM
  Cache --> Notify
  Cache --> TeamEvt
```

## Terminology

| Term | Definition |
|------|------------|
| **Org removal** | User loses access to one organization; CRM rows stay |
| **Global user delete** | `User.isDeleted = true` — out of scope for this feature |
| **Active org member** | Has at least one non-deleted `UserOrgRole` for the org |
| **Removed org member** | No active `UserOrgRole` for org; may appear on historical CRM data |

## Goals and Non-Goals

### Goals

<Check>Access cut-off: Removed user cannot use org-scoped APIs or refresh org sessions</Check>
<Check>RBAC cleanup: All org roles and team roles soft-deleted consistently</Check>
<Check>Realtime/messaging cleanup: Proper event propagation to messaging listeners</Check>
<Check>CRM preservation: No auto-unassign from leads/deals; commission data remains</Check>
<Check>Discoverability: Historical UI shows "Removed from org" badge</Check>
<Check>Re-invite path: Invitation accept can restore membership</Check>

### Non-Goals (v1)

- Global account deletion (`User.isDeleted`)
- Auto-redistribution of leads, deals, or commissions
- Bulk remove users API
- Admin "reassignment worklist" endpoint
- Blocking removal for pending `EntityTransfer`
- Anonymizing PII on `User` row

## API Contract

### Endpoint

<CodeGroup>

```typescript Request
DELETE /v1/users/:id
```

```typescript Response - Success
{
  "success": true
}
```

```typescript Response - Error
{
  "statusCode": 403,
  "message": "Cannot remove yourself from organization"
}
```

</CodeGroup>

### Authorization Requirements

- **Authentication:** Valid JWT with org tenant context
- **Permission:** `OrgPermissionKey.SYSTEM_ADMIN`
- **Path Parameter:** `id` = target user UUID

## Authorization Matrix

| Actor Role | Target Role | Allowed? | Notes |
|------------|-------------|----------|-------|
| Any user | Self | ❌ No | `ForbiddenException` |
| Admin (not Owner) | Admin or Owner | ❌ No | Hierarchy protection |
| Admin | Non-admin | ✅ Yes | Standard removal |
| Owner role | Another Owner | ❌ No | Owner protection |
| `organization.owner_id` | Anyone except self | ✅ Yes | Subject to team leader rule |
| Non-admin | Anyone | ❌ No | Missing permission |

<Warning>
**Team Leader Rule:** If the target user is the last admin on any team, removal is blocked with `BadRequestException`.
</Warning>

## Transaction Flow

All operations execute within a single MikroORM transaction to ensure consistency.

<Steps>

<Step title="Load and Validate">
Load target user with organizations, roles, and team memberships. Validate removal permissions and constraints.
</Step>

<Step title="Remove Organization Link">
Execute `user.organizations.remove(organizationToRemove)` to break M:N relationship.
</Step>

<Step title="Soft-Delete RBAC Records">
Mark all `UserOrgRole`, `TeamMembership`, and `TeamMembershipRole` records as deleted for this organization.
</Step>

<Step title="Clear Session Data">
Clear `selectedOrganization` if it matches the removed org and revoke all org-scoped sessions.
</Step>

<Step title="Invalidate Caches">
Flush transaction changes and invalidate permission cache for the user-org combination.
</Step>

<Step title="Emit Events">
Post-transaction: Send `REMOVED_FROM_ORGANIZATION` and `TEAM_MEMBER_REMOVED` events.
</Step>

</Steps>

## Data Retention

<AccordionGroup>

<Accordion title="Preserved Data">
- `User` row (unchanged, `isDeleted` stays false)
- `EntityStakeholder` records (manual reassignment required)
- Lead/Deal/Contact/Company records (business history)
- `organization_users` profile row (for display maps)
- Pending `entity_transfer` records (v1 - manual resolution)
- Pending `Invitation` records (separate workflow)
- Audit log entries (new rows via triggers)
</Accordion>

<Accordion title="Removed/Modified Data">
- `organizations` M:N relationship (removed)
- `UserOrgRole` records (soft-deleted)
- `TeamMembership` records (soft-deleted)
- `TeamMembershipRole` records (soft-deleted)
- Org-scoped sessions (revoked)
- `selectedOrganization` (cleared if matching)
</Accordion>

</AccordionGroup>

## Subsystem Side Effects

### Session Management

<Info>
See [SESSION_SYSTEM_DOCUMENTATION.md](./SESSION_SYSTEM_DOCUMENTATION.md) for complete session handling details.
</Info>

- All organization sessions for target user are revoked immediately
- If the removed organization was selected, `selectedOrganization` is cleared
- Next login forces organization picker (excluding the removed org)

### Notifications and Events

**Primary Events:**
- `REMOVED_FROM_ORGANIZATION` - Notifies the removed user
- `TEAM_MEMBER_REMOVED` - Emitted per team for messaging cleanup

**Event Flow:**
```typescript
// Per team the user was removed from
{
  organizationId: string,
  userId: string,
  teamId: string,
  teamName: string,
  removedByName: string
}
```

### Messaging System Integration

<Info>
See [MESSAGING_MODULE_SPECIFICATION.md](./MESSAGING_MODULE_SPECIFICATION.md) for messaging subsystem details.
</Info>

The messaging cleanup listener (`messaging-cleanup.listener.ts`) handles:
- Organization-level room removal on `REMOVED_FROM_ORGANIZATION`
- Team-specific room cleanup on `TEAM_MEMBER_REMOVED` events

### Stale Assignment Handling

<Tip>
Use `GET /users/:id/stale-assignments` before removal to identify leads and deals that may need reassignment.
</Tip>

The system provides visibility into potential assignment conflicts:
- Counts leads and deals where user is primary stakeholder
- Informational only - does not block removal
- Follows manual reassignment principle
- Tracked in data integrity audits post-removal

## Error Handling

<Tabs>

<Tab title="Common Errors">

| Error Code | Scenario | Message |
|------------|----------|---------|
| 403 | Self-removal | "Cannot remove yourself from organization" |
| 403 | Insufficient privileges | "Admin cannot remove Owner" |
| 400 | Last team leader | "Cannot remove last admin from team: {teamName}" |
| 404 | User not in org | User not found in organization context |

</Tab>

<Tab title="Recovery Scenarios">

**Re-invitation Process:**
- Removed users can be re-invited to the organization
- Invitation acceptance restores basic membership
- Previous role assignments require manual reconfiguration

**Data Recovery:**
- CRM assignments remain intact and searchable
- Historical audit trails preserved
- Commission calculations unaffected

</Tab>

</Tabs>
