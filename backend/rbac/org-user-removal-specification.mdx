## Executive Summary

<Info>
**Feature:** Remove user from organization (org-scoped deactivation)

**User-facing label:** "Remove from organization" (not "Delete user account")

**Endpoint:** `DELETE /v1/users/:id` → `UserService.remove()`
</Info>

The organization user removal feature enables administrators to revoke a user's access to a specific organization while preserving their CRM history and data integrity. This operation:

- **Deactivates membership and access** in one organization
- **Retains CRM history** (leads, deals, stakeholders, commissions, activities)
- Requires **manual reassignment** of stale assignments by managers
- Shows a **badge** in the UI for users who are no longer active org members

### Architecture Overview

```mermaid
flowchart TD
  Admin["Admin: DELETE /users/:id"]
  Guard["AuthGuard + CheckAccess system.admin"]
  Tx["UserService.remove transaction"]
  OrgLink["organizations.remove"]
  UOR["UserOrgRole isDeleted=true"]
  TM["TeamMembership + TeamMembershipRole isDeleted=true"]
  Sess["revokeAllOrgSessions"]
  Cache["permissionCache.invalidate"]
  Notify["REMOVED_FROM_ORGANIZATION"]
  TeamEvt["TEAM_MEMBER_REMOVED per team"]
  CRM["EntityStakeholder + Lead/Deal rows UNCHANGED"]
  Admin --> Guard --> Tx
  Tx --> OrgLink --> UOR --> TM --> Sess --> Cache
  Tx --> CRM
  Cache --> Notify
  Cache --> TeamEvt
```

### Critical Architectural Decision

<Warning>
**Do NOT set global `User.isDeleted` for org removal.**

The `User` entity is global across organizations. Setting `isDeleted` would:
- Remove the user from ALL organizations
- Affect unique email constraints
- Break user account integrity

Instead, removal is expressed via:
- Junction soft-deletes (`UserOrgRole`, `TeamMembership`, `TeamMembershipRole`)
- Removing the user from the `organizations` M:N collection
</Warning>

<AccordionGroup>
  <Accordion title="Org removal">

  </Accordion>

  </Accordion>

  </Accordion>

  <Accordion title="Removed org member">
    No active `UserOrgRole` for org; may still appear on historical CRM data
  </Accordion>

### Non-Goals (v1)

<Note>
The following features are explicitly **out of scope** for the initial implementation:

- Global account deletion (`User.isDeleted`)
- Auto-redistribution of leads, deals, distribution pools, or commission
- Bulk remove users API
- Admin "reassignment worklist" endpoint (optional Phase 4)
- Blocking removal when user has pending `EntityTransfer` (document as future policy; v1 allows removal)
- Anonymizing PII on `User` row
</Note>

## API Contract

### Request Specification

<CodeGroup>
```bash cURL
curl -X DELETE "https://api.example.com/v1/users/{userId}" \
  -H "Authorization: Bearer {jwt_token}" \
  -H "Content-Type: application/json"
```

```typescript TypeScript
await fetch(`/v1/users/${userId}`, {
  method: 'DELETE',
  headers: {
    'Authorization': `Bearer ${token}`,
    'Content-Type': 'application/json'
  }
});
```
</CodeGroup>

**Endpoint:** `DELETE /v1/users/:id`

**Authentication:** JWT + org tenant context (`organizationId` on token)

**Authorization:** `OrgPermissionKey.SYSTEM_ADMIN` via `@CheckAccess` decorator

**Path Parameters:**
- `id` (string, UUID, required): Target user ID to remove from organization

**Request Body:** None

  </Tab>

  <Tab title="Forbidden (403)">
    ```json
    {
      "statusCode": 403,
      "message": "Cannot remove yourself from organization",
      "error": "Forbidden"
    }
    ```

    - User attempts self-removal

    ```json
      "message": "Cannot remove the last team admin from team: Engineering",
    ```

  <Tab title="Not Found (404)">

    ```json
    {

      "error": "Not Found"
    }
    ```
    Target user not found or not a member of the organization
  </Tab>
</Tabs>

## Authorization Matrix

The following matrix defines which actors can remove which targets:

| Actor | Target | Allowed? | Notes |
|-------|--------|----------|-------|
| Any user | Self | ❌ No | `ForbiddenException` |
| Admin (not Owner, not `org.owner`) | Admin or Owner | ❌ No | Insufficient hierarchy |
| Admin | Non-admin, non-owner | ✅ Yes | Standard removal |
| Owner role (not `org.owner`) | Another Owner | ❌ No | Cannot remove peers |
| `organization.owner_id` | Anyone except self | ✅ Yes | Subject to team-leader rule |
| User without `system.admin` | Anyone | ❌ No | Blocked by guard |

<Warning>
**Team Leader Rule**

If the target user holds `team.admin` on a team and no other active member on that team holds `team.admin`, removal is **blocked** with `BadRequestException`.

This validation uses loaded `teamMemberships` to ensure teams always have at least one administrator.
</Warning>

## Transaction Specification

<Info>
All steps run in **one** MikroORM transaction using the existing `executeInOrg` pattern. Order matters for validations before mutations.
</Info>

### Step-by-Step Process

<Steps>
  <Step title="Load actors">
    Load `deletedByUser` and target `user` with relations:
    - `organizations`
    - `orgRoles`
    - `selectedOrganization`
  </Step>

  <Step title="Validate removal">
    Check:
    - Self-removal prevention
    - Role hierarchy rules
    - Last team leader constraint (on **active** memberships only)
  </Step>

  <Step title="Load team memberships">
    Query `TeamMembership` where:
    - `user` = target user
    - `organization` = current org
    - `isDeleted: false`
    
    Populate: `team`, `teamRoles`, `teamRoles.role.permissions`
  </Step>

  <Step title="Remove organization link">
    Execute `user.organizations.remove(organizationToRemove)`
  </Step>

  <Step title="Soft-delete organization roles">
    For all `UserOrgRole` with `isDeleted: false` for user+org:
    - Set `isDeleted = true`
  </Step>

  <Step title="Soft-delete team memberships">
    For each membership from step 3:
    - For each `TeamMembershipRole`: set `isDeleted = true`
    - Set `membership.isDeleted = true`
    - Collect `{ teamId, teamName }` for post-commit events
  </Step>

  <Step title="Clear selected organization">
    If `user.selectedOrganization.id === organizationId`:
    - Unset `selectedOrganization`
  </Step>

  <Step title="Revoke sessions">
    Call `sessionService.revokeAllOrgSessions(id, organizationId)` inside transaction
  </Step>

  <Step title="Flush changes">
    Execute `em.flush()` to persist all changes
  </Step>

  <Step title="Invalidate cache">
    Call `permissionCache.invalidate(id, organizationId)` (existing implementation)
  </Step>

  <Step title="Post-commit events">
    After transaction commits:
    - Emit `REMOVED_FROM_ORGANIZATION` event
    - Emit `TEAM_MEMBER_REMOVED` event for each team
  </Step>
</Steps>

## CRM History Preservation

<Check>
**Core Principle: Manual Reassignment**

Following industry best practices, the system does NOT automatically reassign or remove CRM data when a user is removed from an organization.
</Check>

### What Gets Preserved

The following data remains **unchanged** after user removal:

- **Leads:** Primary and secondary stakeholder assignments
- **Deals:** Stakeholder assignments and commission percentages
- **Activities:** Historical activity records and ownership
- **Stakeholder Records:** All `EntityStakeholder` rows remain intact
- **Commission Plans:** Commission percentages and splits
- **Distribution Pools:** Pool membership (if applicable)
- **Audit Logs:** Complete audit trail

### Manager Responsibilities

<Warning>
**Manual Reassignment Required**

Managers and administrators are responsible for:

1. Identifying stale assignments before or after removal
2. Manually reassigning leads and deals to active team members
3. Updating commission structures if needed
4. Reviewing distribution pool memberships
</Warning>

## Re-invitation Process

<Tip>
Users removed from an organization can be re-invited and will regain access through the standard invitation flow.
</Tip>

### Re-invitation Steps

<Steps>
  <Step title="Admin sends invitation">
    Organization admin sends a new invitation to the previously removed user's email
  </Step>

  <Step title="User accepts invitation">
    User receives email and clicks invitation link
  </Step>

  <Step title="Membership restoration">
    The invitation acceptance process:
    - Creates new `UserOrgRole` entries
    - Re-adds user to `organizations` collection
    - Does NOT restore previous team memberships automatically
  </Step>

  <Step title="Manual role assignment">
    Admin must manually:
    - Assign organization roles
    - Add user to teams
    - Configure permissions
  </Step>
</Steps>

<Warning>
**Previous CRM Data**

Re-invited users will regain visibility of their previous CRM assignments (leads, deals, stakeholders) as these were never deleted. Administrators may need to review and update stale assignments.
</Warning>

## Security Considerations

<Warning>
**Session Revocation**

All active sessions for the removed user in the affected organization are immediately revoked. The user cannot:
- Refresh existing org-scoped tokens
- Access org APIs even with cached tokens
- View org data through any API endpoint
</Warning>

<Check>
**Permission Cache**

The permission cache is invalidated for the user+organization combination, ensuring:
- No stale permission checks
- Immediate enforcement of access removal
- Consistent authorization behavior across all services
</Check>

## Error Handling

### Common Error Scenarios

<AccordionGroup>
  <Accordion title="Self-removal attempt">
    **Error:** `ForbiddenException`
    
    **Message:** "Cannot remove yourself from organization"
    
    **Resolution:** User must have another admin remove them, or use a proper account closure flow
  </Accordion>

  <Accordion title="Insufficient hierarchy">
    **Error:** `ForbiddenException`
    
    **Message:** "Insufficient permissions to remove this user"
    
    **Resolution:** Only organization owners or higher-level admins can remove other admins
  </Accordion>

  <Accordion title="Last team admin">
    **Error:** `BadRequestException`
    
    **Message:** "Cannot remove the last team admin from team: {teamName}"
    
    **Resolution:** Assign another team admin before removing this user
  </Accordion>

  <Accordion title="Organization owner">
    **Error:** `ForbiddenException`
    
    **Message:** "Cannot remove the organization owner account"
    
    **Resolution:** Transfer organization ownership before attempting removal
  </Accordion>

  <Accordion title="User not found">
    **Error:** `NotFoundException`
    
    **Message:** "User not found in organization"
    
    **Resolution:** Verify the user ID and ensure they are a member of the current organization
  </Accordion>
</AccordionGroup>

## Best Practices

<Tip>
**Before Removing a User**

1. Check stale assignments using the `/stale-assignments` endpoint
2. Identify and reassign critical leads and deals
3. Review team leadership positions
4. Notify relevant team members
5. Document the reason for removal (audit purposes)
</Tip>

<Tip>
**After Removing a User**

1. Review and reassign stale assignments within 24-48 hours
2. Update distribution pool rules if applicable
3. Review commission plans for affected deals
4. Monitor for any system errors related to the removed user
5. Update team documentation if user held key roles
</Tip>

<Warning>
**DO NOT** use user removal as:
- A temporary "disable" mechanism (use separate deactivation if needed)
- A bulk cleanup tool (implement dedicated bulk operations)
- A data deletion strategy (implement separate data retention policies)
</Warning>
