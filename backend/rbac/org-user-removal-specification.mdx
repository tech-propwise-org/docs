## Overview

The organization user removal feature allows administrators to deactivate a user's membership and access within a specific organization while preserving their CRM history and global account. This follows industry-standard practices of maintaining data integrity for business continuity.

<Note>
This feature performs **organization-scoped deactivation**, not global user deletion. The user's global account and CRM history remain intact.
</Note>

## Core Architecture

The removal process is handled through the existing `DELETE /v1/users/:id` endpoint, which calls `UserService.remove()`. The key architectural principle is to **avoid setting global `User.isDeleted`** since users can belong to multiple organizations.

```mermaid
flowchart TD
  Admin["Admin: DELETE /users/:id"]
  Guard["AuthGuard + CheckAccess system.admin"]
  Tx["UserService.remove transaction"]
  OrgLink["organizations.remove"]
  UOR["UserOrgRole isDeleted=true"]
  TM["TeamMembership + TeamMembershipRole isDeleted=true"]
  Sess["revokeAllOrgSessions"]
  Cache["permissionCache.invalidate"]
  Notify["REMOVED_FROM_ORGANIZATION"]
  TeamEvt["TEAM_MEMBER_REMOVED per team"]
  CRM["EntityStakeholder + Lead/Deal rows UNCHANGED"]
  
  Admin --> Guard --> Tx
  Tx --> OrgLink --> UOR --> TM --> Sess --> Cache
  Tx --> CRM
  Cache --> Notify
  Cache --> TeamEvt
```

## Key Concepts

<AccordionGroup>
<Accordion title="Terminology">

| Term | Definition |
|------|------------|
| **Org removal** | User loses access to one organization; CRM data preserved |
| **Global user delete** | `User.isDeleted = true` — **not applicable** for this feature |
| **Active org member** | Has at least one non-deleted `UserOrgRole` for the organization |
| **Removed org member** | No active `UserOrgRole` for org; appears on historical CRM data with badge |

</Accordion>

<Accordion title="Goals">

1. **Access cut-off**: Removed user cannot use org-scoped APIs or refresh org sessions
2. **RBAC cleanup**: All org roles and team roles are consistently soft-deleted
3. **Realtime cleanup**: Messaging listeners receive team removal events
4. **CRM preservation**: No auto-unassign from leads/deals; commission data remains
5. **Discoverability**: Historical UI shows name with "Removed from org" badge
6. **Re-invite support**: Invitation acceptance can restore membership

</Accordion>

<Accordion title="Non-goals">

- Global account deletion (`User.isDeleted`)
- Auto-redistribution of leads, deals, or commissions
- Bulk user removal API
- Blocking removal when user has pending transfers
- PII anonymization on User records

</Accordion>
</AccordionGroup>

## API Specification

### Endpoint

```http
DELETE /v1/users/:id
```

### Authentication & Authorization

<Steps>
<Step title="JWT Authentication">
Requires valid JWT token with organization tenant context (`organizationId`)
</Step>

<Step title="Permission Check">
User must have `OrgPermissionKey.SYSTEM_ADMIN` permission via `@CheckAccess` decorator
</Step>

<Step title="Role Hierarchy Validation">
System enforces role hierarchy rules (see Authorization Matrix below)
</Step>
</Steps>

### Request Parameters

| Parameter | Type | Location | Required | Description |
|-----------|------|----------|----------|-------------|
| `id` | UUID | Path | Yes | Target user's unique identifier |

## Authorization Matrix

<Warning>
The system enforces strict role hierarchy rules to prevent unauthorized access escalation.
</Warning>

| Actor Role | Target Role | Allowed? | Notes |
|------------|-------------|----------|--------|
| Any user | Self | ❌ | Self-removal forbidden |
| Admin | Admin or Owner | ❌ | Cannot remove equal/higher roles |
| Admin | Non-admin, non-owner | ✅ | Standard admin privilege |
| Owner role | Another Owner | ❌ | Owner role protection |
| `organization.owner_id` | Anyone except self | ✅ | Subject to team leader rule |
| Non-admin | Anyone | ❌ | Requires system admin permission |

### Special Rules

<Info>
**Team Leader Protection**: If the target user is the last admin on any team, removal is blocked with a detailed error message including the team name.
</Info>

## Transaction Flow

All operations execute within a single MikroORM transaction to ensure data consistency.

<Steps>
<Step title="Load and Validate">
- Load actor and target users with relationships
- Validate self-removal, role hierarchy, and team leader constraints
- Load active team memberships for the target user
</Step>

<Step title="Remove Organization Link">
- Remove user from `organizations` many-to-many collection
- This is the primary indicator of org membership
</Step>

<Step title="Soft Delete RBAC Records">
- Set `isDeleted = true` on all `UserOrgRole` records for the user+org
- Set `isDeleted = true` on all `TeamMembership` records for the user+org
- Set `isDeleted = true` on all associated `TeamMembershipRole` records
</Step>

<Step title="Clear Session State">
- Unset `selectedOrganization` if it matches the removed org
- Revoke all org-scoped sessions for the user
- Invalidate permission cache entries
</Step>

<Step title="Persist Changes">
- Execute `em.flush()` to commit all changes
- Transaction completes successfully
</Step>
</Steps>

### Post-Transaction Events

After the transaction commits successfully:

1. **`REMOVED_FROM_ORGANIZATION`** event sent to the removed user
2. **`TEAM_MEMBER_REMOVED`** events sent for each team (drives notifications and WebSocket updates)

## Side Effects by System

### Session Management

- All organization-scoped sessions are immediately revoked
- If the removed org was the user's selected organization, they'll see the org picker on next login
- User cannot refresh tokens for the removed organization

### Validation Features

The system includes a stale assignment validation endpoint:

```http
GET /v1/users/:id/stale-assignments
```

This provides visibility into leads and deals where the user is the primary stakeholder, helping administrators identify potential reassignment needs before removal.

<Warning>
While stale assignments are reported, they do **not** block user removal. The system follows a manual reassignment principle where managers handle assignment redistribution separately.
</Warning>

## Best Practices

### Before Removing Users

1. **Check stale assignments** using the stale assignments endpoint
2. **Reassign critical leads and deals** to active team members  
3. **Verify team leadership** to avoid leaving teams without admins
4. **Communicate with stakeholders** about the upcoming access changes

### After Removal

1. **Monitor audit logs** for the removal transaction
2. **Verify session revocation** if security is a concern
3. **Update team documentation** reflecting membership changes
4. **Handle any reported issues** from historical data access

### Re-invitation Process

If a removed user needs to rejoin the organization:

1. Send a new invitation through the standard invitation flow
2. User acceptance will restore basic organization membership
3. **Manually reassign** roles and team memberships as needed
4. Previous CRM associations remain intact and accessible

<Note>
Re-invitation does not automatically restore previous roles or team memberships. These must be manually configured by administrators.
</Note>
