## Overview

The organization user removal feature provides a safe, audit-friendly way to revoke a user's access to an organization while preserving historical CRM data and ensuring proper cleanup of roles, permissions, and sessions.

<Note>
  **Key Principle**: Removing a user from an organization is a **membership deactivation**, not a global account deletion. The user's account remains active globally, and all historical CRM records (leads, deals, stakeholders, commissions) are preserved.
</Note>

## Architecture

### Core Endpoint

- **Method**: `DELETE /v1/users/:id`
- **Controller**: `src/modules/user/user.controller.ts`
- **Service**: `src/modules/user/user.service.ts` (`remove()` method)
- **Authorization**: Requires `system.admin` permission via `@CheckAccess`

### Data Flow

```mermaid
flowchart TD
  Admin["Admin: DELETE /users/:id"]
  Guard["AuthGuard + CheckAccess system.admin"]
  Tx["UserService.remove transaction"]
  OrgLink["organizations.remove"]
  UOR["UserOrgRole isDeleted=true"]
  TM["TeamMembership + TeamMembershipRole isDeleted=true"]
  Sess["revokeAllOrgSessions"]
  Cache["permissionCache.invalidate"]
  Notify["REMOVED_FROM_ORGANIZATION"]
  TeamEvt["TEAM_MEMBER_REMOVED per team"]
  CRM["EntityStakeholder + Lead/Deal rows UNCHANGED"]
  Admin --> Guard --> Tx
  Tx --> OrgLink --> UOR --> TM --> Sess --> Cache
  Tx --> CRM
  Cache --> Notify
  Cache --> TeamEvt
```

<Warning>
  **Critical Design Decision**: Do NOT set `User.isDeleted` for organization removal. The `User` entity is global across organizations. Setting `isDeleted` would remove the person everywhere and break unique email constraints. Removal is expressed through junction table soft-deletes and M:N collection updates.
</Warning>

## Terminology

<AccordionGroup>
  <Accordion title="Organization removal">
    User loses access to one specific organization. All CRM records remain intact and visible to other organization members.
  </Accordion>

  <Accordion title="Global user delete">
    Sets `User.isDeleted = true`, removing the user from all organizations. **Out of scope** for this feature.
  </Accordion>

  <Accordion title="Active org member">
    A user with at least one non-deleted `UserOrgRole` for the organization. This is the authoritative source of truth, not just presence in `User.organizations`.
  </Accordion>

  <Accordion title="Removed org member">
    A user with no active `UserOrgRole` records for the organization. May still appear on historical CRM data with a "Removed from org" badge.
  </Accordion>
</AccordionGroup>

### Core Capabilities

  </Card>

  </Card>

### What Gets Cleaned Up

<Steps>
  <Step title="Organization membership">
    User is removed from the `User.organizations` many-to-many collection
  </Step>
  
  <Step title="Organization roles">
    All `UserOrgRole` records are soft-deleted (`isDeleted = true`)
  </Step>
  
  <Step title="Team memberships">
    All `TeamMembership` and `TeamMembershipRole` records for teams in this organization are soft-deleted
  </Step>
  
  <Step title="Active sessions">
    All JWT sessions for this user-organization combination are revoked
  </Step>
  
  <Step title="Permission cache">
    Cached permissions for this user-organization pair are invalidated
  </Step>
  
  <Step title="Selected organization">
    If this was the user's selected organization, the field is cleared
  </Step>
</Steps>

### What Is Preserved

<Check>
  **Preserved Records**: Leads, deals, entity stakeholders, commission percentages, activities, notes, and all other CRM data remain unchanged. Administrators must manually reassign stale assignments as needed.
</Check>

## Authorization Rules

| Actor Role | Target User | Allowed |
|------------|-------------|---------|
| Any user | Self | ❌ Forbidden |
| Admin (not Owner) | Admin or Owner | ❌ Forbidden |
| Admin | Regular user | ✅ Allowed |
| Owner role | Another Owner | ❌ Forbidden |
| Organization owner (`org.owner_id`) | Any other user | ✅ Allowed* |
| Non-admin user | Anyone | ❌ Forbidden (guard) |

<Info>
  *Subject to team leader validation rule
</Info>

### Special Validations

<Tabs>
  <Tab title="Team Leader Protection">
    If the target user is the last team leader (`team.admin` role) on any team, removal is blocked with a `BadRequestException`. The error message includes the team name.
    
    This prevents teams from becoming leaderless and unable to manage themselves.
  </Tab>
  
  <Tab title="Organization Owner Protection">
    Users who are set as the `organization.owner_id` (the account linkage owner) cannot be removed. This is validated early in the transaction and returns a `ForbiddenException`.
  </Tab>
  
  <Tab title="Self-Removal Prevention">
    Users cannot remove themselves from an organization, even if they have `system.admin` permissions. Returns `ForbiddenException`.
  </Tab>
</Tabs>

<CodeGroup>

```typescript Request
DELETE /v1/users/:id
Authorization: Bearer <jwt_token>
X-Organization-Id: <org_uuid>
```

```typescript Success Response
{
  "success": true
}
```

```typescript Error Responses
// 403 Forbidden - Self-removal or hierarchy violation
{
  "statusCode": 403,
  "message": "Cannot remove yourself from organization",
  "error": "Forbidden"
}

// 400 Bad Request - Last team leader
{
  "statusCode": 400,
  "message": "Cannot remove user: last team leader on team 'Sales Team'",
  "error": "Bad Request"
}

// 404 Not Found - User not in organization
{
  "statusCode": 404,
  "message": "User not found in organization",
  "error": "Not Found"
}
```
</CodeGroup>

### Request Parameters

| Parameter | Type | Location | Required | Description |
|-----------|------|----------|----------|-------------|
| `id` | UUID | Path | Yes | Target user's unique identifier |
| `organizationId` | UUID | JWT context | Yes | Organization from which to remove the user |

### Authorization Requirements

- Valid JWT token with organization context
- Actor must have `system.admin` permission (`OrgPermissionKey.SYSTEM_ADMIN`)
- Subject to role hierarchy and validation rules (see Authorization Rules)

## Transaction Sequence

All operations occur within a single MikroORM transaction (`executeInOrg`). The order is critical for data consistency.

<Steps>
  <Step title="Load entities">
    Load the deleting user, target user with `organizations`, `orgRoles`, and `selectedOrganization` relations
  </Step>
  
  <Step title="Validate permissions">
    Check self-removal, role hierarchy, and team leader constraints using **active** memberships only
  </Step>
  
  <Step title="Load team memberships">
    Query all active `TeamMembership` records for user+organization, including `team`, `teamRoles`, and `role.permissions`
  </Step>
  
  <Step title="Remove organization link">
    Execute `user.organizations.remove(organizationToRemove)` to update the many-to-many relationship
  </Step>
  
  <Step title="Soft-delete organization roles">
    Set `isDeleted = true` on all active `UserOrgRole` records for this user-organization pair
  </Step>
  
  <Step title="Soft-delete team memberships">
    For each team membership:
    - Set `isDeleted = true` on all `TeamMembershipRole` records
    - Set `isDeleted = true` on the `TeamMembership` record
    - Collect team IDs and names for event emission
  </Step>
  
  <Step title="Clear selected organization">
    If `user.selectedOrganization.id` matches the removal target, set to null
  </Step>
  
  <Step title="Revoke sessions">
    Call `sessionService.revokeAllOrgSessions(userId, organizationId)` within transaction
  </Step>
  
  <Step title="Flush changes">
    Execute `em.flush()` to persist all mutations to the database
  </Step>
  
  <Step title="Invalidate permission cache">
    Clear `permissionCache.invalidate(userId, organizationId)` after successful flush
  </Step>
  
  <Step title="Emit events (post-commit)">
    - Emit `REMOVED_FROM_ORGANIZATION` event
    - Emit `TEAM_MEMBER_REMOVED` event for each team
    - Both trigger messaging cleanup listeners
  </Step>
</Steps>

## Event System

### Organization-Level Event

```typescript
{
  type: 'REMOVED_FROM_ORGANIZATION',
  payload: {
    userId: string,
    organizationId: string,
    deletedBy: string,
    timestamp: Date
  }
}
```

**Consumed by**: `messaging-cleanup.listener.ts` (organization-scoped conversation cleanup)

### Team-Level Events

```typescript
{
  type: 'TEAM_MEMBER_REMOVED',
  payload: {
    userId: string,
    teamId: string,
    teamName: string,
    organizationId: string,
    removedBy: string,
    timestamp: Date
  }
}
```

**Consumed by**: `team-membership-removal.listener.ts` (team-scoped conversation room eviction)

<Tip>
  Multiple `TEAM_MEMBER_REMOVED` events are emitted—one for each team the user belonged to. This ensures messaging cleanup listeners can properly evict the user from all team conversation rooms.
</Tip>

### Pre-Removal Check Endpoint

<CodeGroup>
```typescript Request
GET /v1/users/:id/stale-assignments
Authorization: Bearer <jwt_token>
X-Organization-Id: <org_uuid>
```

```typescript Response
{
  "userId": "uuid",
  "organizationId": "uuid",
  "stalePrimaryAssignments": {
    "leads": 12,
    "deals": 5
  },
  "total": 17
}
```
</CodeGroup>

<Info>
  This endpoint is **informational only** and does not block user removal. It helps administrators identify leads and deals that will need reassignment after the user is removed.
</Info>

### Post-Removal Tracking

After removal, the data integrity audit tracks stale assignments via `stakeholdersWithoutActiveUserOrgRoleCount`. This metric identifies historical stakeholder records pointing to users who are no longer active organization members.

## Frontend Integration

### User Display Badge

Removed users should display a **"Removed from org"** badge in historical contexts (stakeholder lists, activity feeds, deal history).

<CodeGroup>
```typescript UserDto
interface UserDto {
  id: string;
  firstName: string;
  lastName: string;
  email: string;
  isActiveOrgMember: boolean; // false for removed users
  // ... other fields
}
```

```tsx Component Example
{user.isActiveOrgMember ? (
  <UserAvatar user={user} />
) : (
  <div>
    <UserAvatar user={user} />
    <Badge variant="secondary">Removed from org</Badge>
  </div>
)}
```
</CodeGroup>

### User Picker Behavior

User selection components (assignment pickers, stakeholder forms) should **exclude** users where `isActiveOrgMember: false`. This prevents assignment of leads/deals to users who no longer have access.

```typescript
const activeUsers = users.filter(u => u.isActiveOrgMember);
```

## Re-Invitation Path

<Warning>
  Re-invitation support is **partially implemented**. When a removed user accepts a new invitation to the same organization, some restoration logic exists but is not fully specified. Full re-invitation behavior is a future enhancement.
</Warning>

When re-inviting a previously removed user:

1. `InvitationService.accept()` will create new `UserOrgRole` records
2. Historical CRM assignments remain unchanged and become accessible again
3. Team memberships must be manually re-created
4. Previous soft-deleted junction records remain deleted (no restoration)

## Limitations & Future Work

<AccordionGroup>
  <Accordion title="Manual reassignment required">
    The system does not automatically reassign leads, deals, or adjust commission percentages when a user is removed. Administrators must manually handle these reassignments.
    
    A future "reassignment worklist" API could provide guidance on which records need attention.
  </Accordion>
  
  <Accordion title="No bulk removal">
    Only single-user removal is supported. Bulk removal would require careful transaction design and event batching.
  </Accordion>
  
  <Accordion title="Entity transfer handling">
    Currently, removal is allowed even if the user has pending `EntityTransfer` records. Future policy may block removal or auto-cancel transfers.
  </Accordion>
  
  <Accordion title="PII anonymization">
    The `User` record is not anonymized or modified during removal. Global account deletion (separate feature) would handle PII requirements.
  </Accordion>
  
  <Accordion title="Re-invitation restoration">
    Full specification and testing of re-invitation scenarios (permission restoration, team membership recreation) is pending.
  </Accordion>
</AccordionGroup>
