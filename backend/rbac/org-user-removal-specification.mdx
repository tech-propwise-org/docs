## Executive Summary

**Feature name:** Remove user from organization (org-scoped deactivation)

**User-facing label:** "Remove from organization" (not "Delete user account")

**Endpoint:** `DELETE /v1/users/:id` → `UserService.remove()`

- **Controller:** `src/modules/user/user.controller.ts`
- **Service:** `src/modules/user/user.service.ts` (`remove()`, lines 864–1103)

<Note>
**Core principle:** Deactivate membership and access in one organization while retaining CRM history (leads, deals, stakeholders, commissions, activities). Managers must manually reassign stale assignments. UI shows a badge on users who are no longer active org members.
</Note>

### Critical Architectural Decision

<Warning>
Do **not** set global `User.isDeleted` for org removal. The `User` entity is global across organizations; setting `isDeleted` would remove the person everywhere and affect unique email constraints. 

Removal is expressed via:
- Junction soft-deletes (`UserOrgRole`, `TeamMembership`, `TeamMembershipRole`)
- Removing the user from the `organizations` M:N collection
</Warning>

```mermaid
flowchart TD
  Admin["Admin: DELETE /users/:id"]
  Guard["AuthGuard + CheckAccess system.admin"]
  Tx["UserService.remove transaction"]
  OrgLink["organizations.remove"]
  UOR["UserOrgRole isDeleted=true"]
  TM["TeamMembership + TeamMembershipRole isDeleted=true"]
  Sess["revokeAllOrgSessions"]
  Cache["permissionCache.invalidate"]
  Notify["REMOVED_FROM_ORGANIZATION"]
  TeamEvt["TEAM_MEMBER_REMOVED per team"]
  CRM["EntityStakeholder + Lead/Deal rows UNCHANGED"]
  Admin --> Guard --> Tx
  Tx --> OrgLink --> UOR --> TM --> Sess --> Cache
  Tx --> CRM
  Cache --> Notify
  Cache --> TeamEvt
```

## Terminology

<AccordionGroup>
  <Accordion title="Org removal">
    User loses access to one organization; CRM rows remain unchanged
  </Accordion>
  
  <Accordion title="Global user delete">
    `User.isDeleted = true` — **out of scope** for this feature
  </Accordion>
  
  <Accordion title="Active org member">
    Has at least one non-deleted `UserOrgRole` for the organization (authoritative; source of truth is active `UserOrgRole`, not `User.organizations` alone)
  </Accordion>
  
  <Accordion title="Removed org member">
    No active `UserOrgRole` for organization; may still appear on historical CRM data
  </Accordion>
</AccordionGroup>

## Non-goals (v1)

<Warning>
The following features are explicitly **not** included in version 1:

- Global account deletion (`User.isDeleted`)
- Auto-redistribution of leads, deals, distribution pools, or commission
- Bulk remove users API
- Admin "reassignment worklist" endpoint (optional Phase 4)
- Blocking removal when user has pending `EntityTransfer` (document as future policy; v1 allows removal)
- Anonymizing PII on `User` row
</Warning>

## API Contract

### Request Specification

<CodeGroup>
```http HTTP Request
DELETE /v1/users/:id HTTP/1.1
Authorization: Bearer <jwt_token>
X-Organization-Id: <organization_uuid>
```

```typescript TypeScript
const response = await fetch(`/v1/users/${userId}`, {
  method: 'DELETE',
  headers: {
    'Authorization': `Bearer ${token}`,
    'X-Organization-Id': organizationId
  }
});
```
</CodeGroup>

**Authentication requirements:**
- JWT token with org tenant context (`organizationId` on token)
- Permission: `OrgPermissionKey.SYSTEM_ADMIN` via `@CheckAccess` decorator
- Path parameter: `id` = target user UUID

### Response Specification

<Tabs>
  <Tab title="200 Success">
    ```json
    {
      "success": true
    }
    ```
    User successfully removed from organization
  </Tab>
  
  <Tab title="400 Bad Request">
    ```json
    {
      "statusCode": 400,
      "message": "Cannot remove the last team leader from team: Engineering"
    }
    ```
    Occurs when user is the last team leader in one or more teams
  </Tab>
  
  <Tab title="403 Forbidden">
    ```json
    {
      "statusCode": 403,
      "message": "Cannot remove yourself from organization"
    }
    ```
    Self-removal, Admin removing Admin/Owner, or Owner removing Owner
  </Tab>
  
  <Tab title="404 Not Found">
    ```json
    {
      "statusCode": 404,
      "message": "User not found in organization"
    }
    ```
    Removing actor not in org, or target not found in org
    
    <Info>
    Currently service returns `{ success: false }` when target user not in org. Consider normalizing to 404 in a follow-up.
    </Info>
  </Tab>
</Tabs>

## Authorization Matrix

<Note>
All authorization checks are enforced through the RBAC system and guard layers
</Note>

| Actor | Target | Allowed? | Notes |
|-------|--------|----------|-------|
| Any user | Self | ❌ **No** | `ForbiddenException` thrown |
| Admin (not Owner, not `org.owner`) | Admin or Owner | ❌ **No** | Hierarchy protection |
| Admin | Non-admin, non-owner | ✅ **Yes** | Standard removal path |
| Owner role (not `org.owner`) | Another Owner | ❌ **No** | Owner peer protection |
| `organization.owner_id` | Anyone except self | ✅ **Yes** | Subject to team-leader rule |
| User without `system.admin` | Anyone | ❌ **No** | Blocked by guard |

### Team Leader Protection

<Warning>
If the target user holds `team.admin` on a team and no other active member on that team holds `team.admin`, removal is **blocked** with `BadRequestException`.

This check uses loaded `teamMemberships` to ensure at least one team leader remains per team.
</Warning>

## Transaction Specification

<Info>
All steps run in **one** MikroORM transaction using the existing `executeInOrg` pattern. Order matters for validations before mutations.
</Info>

### Transaction Steps

<Steps>
  <Step title="Load actors">
    Load `deletedByUser` and target `user` with:
    - `organizations` collection
    - `orgRoles` collection
    - `selectedOrganization` reference
  </Step>
  
  <Step title="Validate operation">
    Perform authorization checks:
    - Self-removal prevention
    - Role hierarchy validation
    - Last team leader check (on **active** memberships only)
  </Step>
  
  <Step title="Load team memberships">
    Query `TeamMembership` where:
    - `user` = target user
    - `organization` = current org
    - `isDeleted: false`
    
    Populate: `team`, `teamRoles`, `teamRoles.role.permissions`
  </Step>
  
  <Step title="Mutate org link">
    Execute: `user.organizations.remove(organizationToRemove)`
    
    This removes the M:N relationship between user and organization
  </Step>
  
  <Step title="Soft-delete org roles">
    For all `UserOrgRole` where:
    - User = target user
    - Organization = current org
    - `isDeleted: false`
    
    Set: `isDeleted = true`
  </Step>
  
  <Step title="Soft-delete team memberships">
    For each membership from step 3:
    
    1. For each `TeamMembershipRole` on membership:
       - Set `isDeleted = true`
    2. Set `membership.isDeleted = true`
    3. Collect `{ teamId, teamName }` for post-commit events
  </Step>
  
  <Step title="Clear selected organization">
    If `user.selectedOrganization.id === organizationId`:
    - Unset `user.selectedOrganization`
  </Step>
  
  <Step title="Revoke sessions">
    Inside transaction: `sessionService.revokeAllOrgSessions(id, organizationId)`
    
    Ensures user cannot continue using existing sessions
  </Step>
  
  <Step title="Flush changes">
    Execute: `em.flush()`
    
    Commits all entity changes to database
  </Step>
  
  <Step title="Invalidate cache">
    Post-flush: `permissionCache.invalidate(id, organizationId)`
    
    Clears cached permission data for the user in this org
  </Step>
  
  <Step title="Emit events">
    Post-commit event emission:
    
    1. `REMOVED_FROM_ORGANIZATION` event (org-level)
    2. `TEAM_MEMBER_REMOVED` event per team (from step 6 collection)
    
    <Info>
    These events trigger messaging cleanup and other async workflows
    </Info>
  </Step>
</Steps>

## CRM Data Preservation

<Check>
All CRM data is preserved when a user is removed from an organization
</Check>

### Preserved Entities

| Entity | Behavior | Rationale |
|--------|----------|-----------|
| **EntityStakeholder** | Unchanged | Historical ownership and stakeholder relationships preserved |
| **Lead assignments** | Not auto-reassigned | Manual reassignment required by managers |
| **Deal assignments** | Not auto-reassigned | Commission percentages remain until manual update |
| **Activity history** | Unchanged | Audit trail and historical context preserved |
| **Commission records** | Unchanged | Financial accuracy requires historical data |
| **Distribution pools** | Not auto-removed | Manual pool management by admins |

<Warning>
**Manual Reassignment Required:**

Managers must manually reassign:
- Stale lead assignments
- Deal ownership
- Distribution pool memberships
- Commission splits

Use `GET /users/:id/stale-assignments` to identify impacted records before removal.
</Warning>

## Session and Permission Management

### Session Revocation

<Steps>
  <Step title="In-transaction revocation">
    `sessionService.revokeAllOrgSessions(userId, organizationId)` is called **inside** the transaction to ensure atomicity
  </Step>
  
  <Step title="Session filter impact">
    After transaction commit:
    - User cannot refresh org sessions for this organization
    - Existing JWT tokens with this org context become invalid
    - User must re-authenticate to access other organizations
  </Step>
</Steps>

### Permission Cache Invalidation

```typescript
// Post-flush, pre-event emission
permissionCache.invalidate(userId, organizationId);
```

<Note>
Permission cache invalidation ensures that any cached RBAC checks for this user-org combination are cleared, preventing stale permission data.
</Note>

## Re-invitation Path

<Info>
**Current Status:** Partially implemented
</Info>

### Re-invitation Flow

<Steps>
  <Step title="Admin sends invitation">
    New invitation created via `InvitationService` for previously removed user's email
  </Step>
  
  <Step title="User accepts invitation">
    Invitation acceptance can restore membership:
    - Creates new `UserOrgRole` records
    - Adds organization back to `user.organizations`
    - Does **not** automatically restore previous team memberships
  </Step>
  
  <Step title="Manual role restoration">
    Admin must manually:
    - Re-add user to teams
    - Reassign appropriate roles
    - Update CRM assignments if needed
  </Step>
</Steps>

<Warning>
Historical CRM data (stakeholder records, activities) will show the user's name continuously, including during the removed period. The `isActiveOrgMember` badge helps distinguish current vs. historical access.
</Warning>

## Error Handling

### Common Error Scenarios

<AccordionGroup>
  <Accordion title="Self-removal attempt">
    **Status Code:** 403 Forbidden
    
    **Message:** "Cannot remove yourself from organization"
    
    **Resolution:** User must have another admin remove them, or transfer ownership first
  </Accordion>
  
  <Accordion title="Last team leader">
    **Status Code:** 400 Bad Request
    
    **Message:** "Cannot remove the last team leader from team: {teamName}"
    
    **Resolution:** Assign another team member as leader before removal
  </Accordion>
  
  <Accordion title="Organization owner removal">
    **Status Code:** 403 Forbidden
    
    **Message:** "Cannot remove organization owner"
    
    **Resolution:** Transfer organization ownership before attempting removal
  </Accordion>
  
  <Accordion title="Hierarchy violation">
    **Status Code:** 403 Forbidden
    
    **Message:** "Insufficient permissions to remove this user"
    
    **Resolution:** Only users with appropriate role hierarchy can remove others
  </Accordion>
</AccordionGroup>
