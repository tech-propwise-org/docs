## Overview

The organization user removal feature allows administrators to remove users from their organization while maintaining CRM history and ensuring data integrity. This is an **organization-scoped deactivation**, not a global user account deletion.

<Note>
This feature deactivates membership and access in one organization while retaining all CRM history including leads, deals, stakeholders, commissions, and activities.
</Note>

## Core Principles

### Industry-Aligned Approach
- **Deactivate** membership and access in one organization
- **Retain** CRM history (leads, deals, stakeholders, commissions, activities)
- **Manual reassignment** of stale assignments by managers
- **Badge UI** shows users who are no longer active organization members

### Critical Architectural Decision

<Warning>
Do **not** set global `User.isDeleted` for organization removal. The `User` entity is global across organizations; setting `isDeleted` would remove the person everywhere and affect unique email constraints.
</Warning>

Removal is expressed via:
- **Junction soft-deletes** (`UserOrgRole`, `TeamMembership`, `TeamMembershipRole`)
- Removing the user from the `organizations` M:N collection

    | **Global user delete** | `User.isDeleted = true` — **out of scope** for this feature |

## API Specification

### Endpoint Details

- **Method:** `DELETE /v1/users/:id`
- **Controller:** `src/modules/user/user.controller.ts`
- **Service:** `src/modules/user/user.service.ts` (`remove()`, lines 864–1103)
- **Authentication:** JWT with organization tenant context
- **Authorization:** `OrgPermissionKey.SYSTEM_ADMIN` via `@CheckAccess`

### Request Format

```typescript
// Path parameter
id: string // Target user UUID

// Headers
Authorization: Bearer <jwt_token>
// organizationId extracted from token context
```

### Response Format

<CodeGroup>
```json Success (200)
{
  "success": true
}
```

```json Forbidden (403)
{
  "message": "Cannot remove yourself from organization",
  "statusCode": 403
}
```

```json Bad Request (400)
{
  "message": "Cannot remove last team leader from team: Engineering",
  "statusCode": 400
}
```

```json Not Found (404)
{
  "message": "User not found in organization",
  "statusCode": 404
}
```
</CodeGroup>

## Authorization Matrix

| Actor Role | Target User | Allowed? | Notes |
|------------|-------------|----------|--------|
| Any user | Self | ❌ | `ForbiddenException` |
| Admin (not Owner) | Admin or Owner | ❌ | Insufficient permissions |
| Admin | Non-admin, non-owner | ✅ | Standard case |
| Owner role | Another Owner | ❌ | Owner-to-owner restriction |
| `organization.owner_id` | Anyone except self | ✅ | Subject to team leader rule |
| Non-admin user | Anyone | ❌ | Guard blocks request |

<Info>
**Team Leader Rule:** If the target user is the last team leader (`team.admin` role) on any team, removal is blocked with a `BadRequestException` that includes the team name.
</Info>

## Transaction Implementation

### Database Operations

All operations execute within a single MikroORM transaction to ensure consistency:

<Steps>
  <Step title="Load and Validate">
    - Load `deletedByUser` and target `user` with relationships
    - Validate self-removal, role hierarchy, and team leader constraints
  </Step>
  
  <Step title="Load Team Memberships">
    - Query active `TeamMembership` records for user in organization
    - Populate `team`, `teamRoles`, and `teamRoles.role.permissions`
  </Step>
  
  <Step title="Remove Organization Link">
    - Execute `user.organizations.remove(organizationToRemove)`
  </Step>
  
  <Step title="Soft-Delete Organization Roles">
    - Set `isDeleted = true` on all active `UserOrgRole` records
  </Step>
  
  <Step title="Soft-Delete Team Memberships">
    - Set `isDeleted = true` on all `TeamMembershipRole` records
    - Set `isDeleted = true` on all `TeamMembership` records
    - Collect team information for post-commit events
  </Step>
  
  <Step title="Clear Selected Organization">
    - Unset `user.selectedOrganization` if it matches the removed organization
  </Step>
  
  <Step title="Revoke Sessions and Cache">
    - Call `sessionService.revokeAllOrgSessions(id, organizationId)`
    - Execute `em.flush()` to persist changes
    - Invalidate `permissionCache.invalidate(id, organizationId)`
  </Step>
</Steps>

### Post-Transaction Events

After successful transaction commit:

1. **`REMOVED_FROM_ORGANIZATION`** event sent to the removed user
2. **`TEAM_MEMBER_REMOVED`** events emitted for each affected team

```typescript
// Event payload structure
{
  organizationId: string,
  userId: string,
  teamId: string,
  teamName: string,
  removedByName: string
}
```

## Data Retention Policy

<Tabs>
  <Tab title="Preserved Data">
    | Entity | Action | Rationale |
    |--------|--------|-----------|
    | `User` row | **Unchanged** | Global identity across organizations |
    | `EntityStakeholder` | **Unchanged** | Manual reassignment required |
    | `Lead` / `Deal` / `Contact` / `Company` | **Unchanged** | Business history preservation |
    | `organization_users` profile | **Retained** | Display maps and historical references |
    | `entity_transfer` pending | **Unchanged** | Managers resolve manually |
    | `Invitation` pending | **Unchanged** | Separate workflow |
    | `audit_log` | **New rows added** | Automatic via database triggers |
  </Tab>
  
  <Tab title="Modified Data">
    | Entity | Action | Rationale |
    |--------|--------|-----------|
    | `organizations` M:N | **Removed** | User no longer in organization |
    | `UserOrgRole` | **Soft-deleted** | Authoritative membership record |
    | `TeamMembership` | **Soft-deleted** | Consistency with team removal |
    | `TeamMembershipRole` | **Soft-deleted** | Cleanup team permissions |
    | `Session` (org-scoped) | **Revoked** | Security requirement |
  </Tab>
</Tabs>

## System Integration Effects

### Session Management
- All organization-scoped sessions are immediately revoked
- If the removed organization was selected, `selectedOrganization` is cleared
- Next login forces organization picker (without the removed organization)

### Notifications
- **User notification:** `REMOVED_FROM_ORGANIZATION` sent to the removed user
- **Team notifications:** `TEAM_MEMBER_REMOVED` events trigger WebSocket updates via `rbac-event.listener.ts`

### Messaging System
- Organization removal triggers cleanup via `messaging-cleanup.listener.ts`
- Team-specific conversation room evictions handled per team event

## Stale Assignment Validation

<Tip>
The system provides visibility into potential assignment conflicts before removal through the `GET /users/:id/stale-assignments` endpoint.
</Tip>

- Counts leads and deals where the user is the primary stakeholder
- **Informational only** - does not block removal
- Follows the manual reassignment principle
- Post-removal tracking via `stakeholdersWithoutActiveUserOrgRoleCount` in data integrity audits

## Error Handling

<Warning>
Common error scenarios and their handling:

- **Self-removal attempts:** Blocked with clear error message
- **Insufficient permissions:** Standard 403 Forbidden response  
- **Last team leader:** Specific error with affected team name
- **Organization owner removal:** Special validation for `organizations.owner_id`
</Warning>

The system prioritizes data integrity and clear error communication to prevent accidental or unauthorized removals while maintaining operational flexibility for legitimate administrative actions.
