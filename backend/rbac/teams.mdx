## Architecture overview

### Design principles

The RBAC system follows these core principles:

1. **Unified Role & Permission System**:
   - Single `Role` table for both org-level and team-level roles
   - Single `Permission` table for all permissions
   - `scope` field differentiates between ORG and TEAM levels

2. **System vs Custom Roles**:
   - System roles have `organization = NULL` (global/seeded)
   - Custom roles have `organization = specific org ID`
   - System roles: Owner, Admin, Basic, Team Leader, Agent

3. **Explicit Junction Tables**:
   - `UserOrgRole` - User ↔ Role assignments in Organization context
   - `TeamMembershipRole` - TeamMembership ↔ Role assignments
   - Required for multi-org support (user can have different roles in different orgs)
   - Includes metadata: `assigned_by`, `created_at`, `organization_id` (for RLS)

4. **Multi-Tenancy Support**:
   - Users can belong to multiple organizations
   - Same user can have different roles in different organizations
   - RLS (Row-Level Security) enforced at database level

5. **Permission Inheritance**:
   - Team roles can have both TEAM-scoped and ORG-scoped permissions
   - Team-available permissions controlled via `team_available_permissions`

## Core entities

### User

**Purpose**: Represents a user account in the system.

```typescript
User
├── id: uuid
├── first_name, last_name: string
├── email: string (globally unique among active users)
├── password: string (hashed)
├── phone_number?: string
├── avatar_url?: string (public URL to avatar image, stored as WebP on R2)
├── languages?: string[] (array of language codes)
├── preferred_language?: string
├── is_verified: boolean
├── is_active: boolean
├── is_system_admin: boolean (platform-level admin, distinct from RBAC roles)
├── is_deleted: boolean (soft delete)
├── selected_organization_id?: uuid → Organization
├── Auth Fields:
│   ├── refresh_token?: string
│   ├── token_version: number (default: 1, incremented on token invalidation)
│   ├── verification_code?: string
│   ├── verification_code_expires?: timestamp
│   ├── verification_email_attempts: number (default: 0)
│   ├── last_verification_email_sent_at?: timestamp
│   ├── reset_password_code?: string
│   └── reset_password_code_expires?: timestamp
├── Relationships:
│   ├── organizations → Organization[] (via organization_users)
│   ├── orgRoles → UserOrgRole[] (roles in orgs)
│   └── teamMemberships → TeamMembership[]
└── Timestamps: created_at, updated_at

Indexes:
├── (first_name, last_name) [composite]
└── (email, is_deleted) [unique, where: is_deleted = false]
```

**Key Rules**:
- User can belong to multiple organizations
- Email is globally unique among non-deleted users (enforced by unique index on `[email, is_deleted]` where `is_deleted = false`)
- `selected_organization_id` tracks current org context
- Access roles via `orgRoles` collection (not direct `roles`)
- `is_system_admin` is a platform-level flag independent of per-org RBAC roles

### Organization

**Module Location:** `/modules/organization/` (separate from RBAC)

**Purpose**: Represents a tenant organization. Organization serves as the **tenant boundary** for multi-tenancy and is conceptually a peer to User, not a component of RBAC.

```typescript
Organization
├── id: uuid
├── name: string
├── website?: string
├── email?: string
├── vat?: string
├── avatar_url?: string (public URL to org logo, stored as WebP on R2)
├── address?: jsonb (embedded Address object)
├── owner_id: uuid → User (organization owner)
├── Relationships:
│   ├── users → User[] (ManyToMany, owner side, via organization_users)
│   ├── owner → User (ManyToOne)
│   ├── leads → Lead[] (OneToMany)
│   ├── contacts → Contact[] (OneToMany)
│   ├── deals → Deal[] (OneToMany)
│   └── companies → Company[] (OneToMany)
└── Timestamps: created_at, updated_at, is_deleted
```

<Note>
Teams, Roles, and Invitations each reference Organization via their own `@ManyToOne` relationships, but Organization does not declare inverse `@OneToMany` collections for these. CRM entities (Lead, Contact, Deal, Company) do have inverse relationships on Organization.
</Note>

**Key Rules**:
- Each organization has exactly one owner
- Owner is automatically assigned "Owner" role on creation
- System roles (organization = NULL) are shared across all orgs
- Soft delete via `isDeleted` flag with `@Filter` decorator

### Team

```typescript
Team
├── id: uuid
├── name: string
├── description?: string
├── organization_id: uuid → Organization
├── is_deleted: boolean (soft delete)
├── Relationships:
│   ├── memberships → TeamMembership[]
│   ├── roles → Role[] (where scope = TEAM, team_id = this.id)
│   └── availablePermissions → Permission[] (via team_available_permissions)
└── Timestamps: created_at, updated_at

Indexes:
└── (name, organization_id, is_deleted) [unique, where: is_deleted = false]
```

**Key Rules**:
- Teams belong to exactly one organization
- Team names must be unique within organization (among non-deleted teams)
- `availablePermissions` defines which ORG-scoped permissions team roles can inherit
- Soft delete via `is_deleted` flag with proper indexing

### Role (Unified)

**Purpose**: Represents a role that can be assigned to users. Unified table for both org-level and team-level roles.

```typescript
Role
├── id: uuid
├── name: string
├── description?: string
├── color?: string
├── scope: RoleScope (ORG | TEAM)
├── is_system: boolean (true for seeded roles)
├── organization_id?: uuid → Organization (NULL for system roles)
├── team_id?: uuid → Team (NULL for org roles)
├── Relationships:
│   ├── permissions → Permission[] (via role_permissions)
│   ├── userOrgRoles → UserOrgRole[] (for ORG scope)
│   └── teamMembershipRoles → TeamMembershipRole[] (for TEAM scope)
└── Timestamps: created_at, updated_at

Indexes:
├── (name, organization_id, team_id) [unique]
└── (scope, team_id) [for team role queries]
```

**Key Rules**:
- System roles: `is_system = true`, `organization_id = NULL`
- Org roles: `scope = ORG`, `team_id = NULL`
- Team roles: `scope = TEAM`, `team_id != NULL`
- Role names must be unique within their scope (org or team)

### Permission (Unified)

**Purpose**: Represents a permission that can be granted to roles.

```typescript
Permission
├── id: uuid
├── key: string (unique, dot notation: "system.admin", "crm.leads.view")
├── name: string (display name)
├── description?: string
├── scope: PermissionScope (ORG | TEAM)
├── category: string (grouping: "System", "CRM", "Sales")
├── is_system: boolean (true for seeded permissions)
├── Relationships:
│   ├── roles → Role[] (via role_permissions)
│   └── teamAvailablePermissions → TeamAvailablePermission[]
└── Timestamps: created_at, updated_at

Indexes:
├── key [unique]
└── (scope, category) [for filtering]
```

**Permission Hierarchy**:
- **System Level**: `system.*` - Platform administration
- **Organization Level**: `crm.*`, `sales.*` - Org-wide permissions
- **Team Level**: `team.*` - Team-specific permissions

### TeamMembership

```typescript
TeamMembership
├── id: uuid
├── user_id: uuid → User
├── team_id: uuid → Team
├── organization_id: uuid → Organization
├── joined_at: timestamp
├── is_deleted: boolean (soft delete)
├── Relationships:
│   ├── user → User
│   ├── team → Team
│   └── teamMembershipRoles → TeamMembershipRole[] (roles in this team)
└── Timestamps: created_at, updated_at

Indexes:
└── (user_id, team_id, is_deleted) [unique, where: is_deleted = false]
```

## Explicit junction tables

### UserOrgRole

**Purpose**: Junction table for User-Role assignments in Organization context.

```typescript
UserOrgRole
├── id: uuid
├── user_id: uuid → User
├── role_id: uuid → Role
├── organization_id: uuid → Organization
├── assigned_by: uuid → User
├── Relationships:
│   ├── user → User
│   ├── role → Role
│   └── assignedBy → User
└── Timestamps: created_at, updated_at

Indexes:
└── (user_id, role_id, organization_id) [unique]
```

**Key Rules**:
- Required for multi-org support (user can have different roles in different orgs)
- `organization_id` enables RLS and prevents cross-org role assignments
- `assigned_by` tracks who granted the role for audit purposes

### TeamMembershipRole

**Purpose**: Junction table for TeamMembership-Role assignments.

```typescript
TeamMembershipRole
├── id: uuid
├── team_membership_id: uuid → TeamMembership
├── role_id: uuid → Role
├── organization_id: uuid → Organization (for RLS)
├── assigned_by: uuid → User
├── assigned_at: timestamp
└── Timestamps: created_at, updated_at

Indexes:
└── (team_membership_id, role_id) [unique]
```

## Permission resolution

When resolving permissions for a user, the system combines:

1. **Organization roles** (via `UserOrgRole`)
2. **Team roles** (via `TeamMembershipRole`)
3. **Available permission filtering** for team ORG-scoped permissions

```typescript
async function resolveUserPermissions(
  userId: string,
  organizationId: string,
  teamId?: string
): Promise<Permission[]> {
  // Get user's org roles
  const orgRoles = await getUserOrgRoles(userId, organizationId);
  
  // Get user's team roles if team context provided
  let teamRoles: Role[] = [];
  if (teamId) {
    teamRoles = await getUserTeamRoles(userId, teamId);
  }
  
  // Combine permissions from both contexts
  const allPermissions = [
    ...getPermissionsFromRoles(orgRoles),
    ...getFilteredTeamPermissions(teamRoles, teamId)
  ];
  
  return deduplicatePermissions(allPermissions);
}
```

## System roles

### Role definitions

| Role | Scope | Description | Key Permissions |
| --- | --- | --- | --- |
| **Owner** | ORG | Organization owner | `system.owner`, `system.admin`, all permissions |
| **Admin** | ORG | Organization administrator | `system.admin`, most management permissions |
| **Basic** | ORG | Basic organization member | `crm.view`, basic read permissions |
| **Team Leader** | TEAM | Team management role | `team.admin`, `team.manage`, team permissions |
| **Agent** | TEAM | Team member role | `team.view`, basic team permissions |

### Invitation acceptance flow

<Steps>
<Step title="User clicks invitation link">
User receives email with invitation token and clicks the link to accept.
</Step>
<Step title="System validates invitation">
Check invitation status, expiration, and organization/team context.
</Step>
<Step title="User account handling">
If user exists: add to organization/team. If new: create account first.
</Step>
<Step title="Role assignment">
Assign specified roles via UserOrgRole or TeamMembershipRole junctions.
</Step>
<Step title="Default role fallback">
If no roles specified, assign default roles (Basic for org, Agent for team).
</Step>
</Steps>

## Business rules

### Core business rules

<AccordionGroup>
<Accordion title="User and Organization Rules">
- **Email uniqueness**: User email must be globally unique among active users
- **Organization ownership**: Each organization has exactly one owner
- **Multi-org membership**: Users can belong to multiple organizations with different roles
- **Selected organization**: Users must have a selected organization for context
</Accordion>

<Accordion title="Role Assignment Rules">
- **System role sharing**: System roles (organization = NULL) are available across all organizations
- **Role scope validation**: ORG roles can only be assigned at org level, TEAM roles at team level
- **Permission inheritance**: Team roles can have ORG permissions only if in team's available permissions
- **Default assignments**: New org members get "Basic" role, new team members get "Agent" role
</Accordion>

<Accordion title="Team Management Rules">
- **Unique membership**: User can only have one membership per team
- **Organization alignment**: Team membership must match team's organization
- **Team leader requirement**: Each team must have at least one Team Leader
- **Available permission validation**: Team roles can only use ORG permissions if available
</Accordion>

<Accordion title="Permission and Access Rules">
- **Permission scope enforcement**: TEAM permissions only work in team context
- **RLS compliance**: All operations respect organization boundaries
- **Soft delete consistency**: Deleting entities cascades soft delete to related records
- **Audit trail**: Role assignments track who assigned them and when
</Accordion>
</AccordionGroup>

## RLS integration

### Row Level Security policies

All RBAC tables implement comprehensive RLS policies for multi-tenant isolation:

```sql
-- Users see only their own organization's data
CREATE POLICY user_isolation ON users FOR ALL TO authenticated
USING (selected_organization_id = get_current_organization_id());

-- Organizations are accessible based on membership
CREATE POLICY organization_access ON organizations FOR ALL TO authenticated
USING (id IN (SELECT organization_id FROM organization_users WHERE user_id = auth.uid()));

-- Teams are isolated by organization
CREATE POLICY team_isolation ON teams FOR ALL TO authenticated
USING (organization_id = get_current_organization_id());

-- Team memberships inherit organization isolation
CREATE POLICY team_membership_isolation ON team_memberships FOR ALL TO authenticated
USING (organization_id = get_current_organization_id());

-- Roles respect organization and system role boundaries
CREATE POLICY role_isolation ON roles FOR ALL TO authenticated
USING (organization_id = get_current_organization_id() OR organization_id IS NULL);

-- Junction tables enforce organization boundaries
CREATE POLICY user_org_role_isolation ON user_org_roles FOR ALL TO authenticated
USING (organization_id = get_current_organization_id());

CREATE POLICY team_membership_role_isolation ON team_membership_roles FOR ALL TO authenticated
USING (organization_id = get_current_organization_id());

-- Invitations are organization-scoped
CREATE POLICY invitation_isolation ON organization_invitations FOR ALL TO authenticated
USING (organization_id = get_current_organization_id());

CREATE POLICY team_invitation_isolation ON team_invitations FOR ALL TO authenticated
USING (organization_id = get_current_organization_id());
```

```sql
-- Get current organization from user context
CREATE OR REPLACE FUNCTION get_current_organization_id()
```


### Module dependencies


**Key Principles**:
- **Organization**: Independent tenant module, minimal dependencies
- **Auth**: Depends on Organization for context, integrates with RBAC for permissions  
- **RBAC**: Depends on both Organization (for tenant isolation) and Auth (for current user)
- **Shared/Core**: Common utilities, decorators, and base entities used by all modules

## Endpoints

### Role management

**`/roles`** — `rbac/role/role.controller.ts`

| Method | Endpoint | Access | Description |
| --- | --- | --- | --- |
| GET | `/roles` | `system.admin` | Get all roles in organization |
| GET | `/roles/:id` | `system.admin` | Get role detail with permissions |
| POST | `/roles` | `system.owner` | Create custom role |
| PATCH | `/roles/:id` | `system.owner` | Update role properties |
| DELETE | `/roles/:id` | `system.owner` | Delete custom role |
| POST | `/roles/:roleId/permissions` | `system.owner` | Assign permissions to role |
| DELETE | `/roles/:roleId/permissions` | `system.owner` | Remove permissions from role |
| GET | `/roles/:roleId/users` | `system.admin` | Get users with specific role |
| POST | `/roles/:roleId/users` | `system.admin` | Batch assign role to users |
| DELETE | `/roles/:roleId/users` | `system.admin` | Batch remove role from users |

### Team management

**`/teams`** — `rbac/team/team.controller.ts`

| Method | Endpoint | Access | Description |
| --- | --- | --- | --- |
| POST | `/teams` | `system.admin` | Create team with organization validation |
| GET | `/teams` | `system.admin` OR `crm.manage` + teamCheck | Get all teams in organization |
| GET | `/teams/:id` | `system.admin` OR resource access | Get team detail with memberships |
| PATCH | `/teams/:id` | `system.admin` | Update team properties |
| DELETE | `/teams/:id` | `system.admin` | Soft delete team and memberships |
| GET | `/teams/:teamId/members` | `system.admin` OR resource access | Get team members with roles |
| POST | `/teams/:teamId/members` | `system.admin` + teamCheck | Add user to team with default role |
| PATCH | `/teams/:teamId/members/:userId` | `system.admin` + teamCheck | Update member roles in team |
| DELETE | `/teams/:teamId/members/:userId` | `system.admin` + teamCheck | Remove user from team |
| POST | `/teams/:teamId/available-permissions` | `system.owner` + teamCheck | Set available permissions |

### Invitation management

**`/invitations`** — `rbac/invitation/invitation.controller.ts`

| Method | Endpoint | Access | Description |
| --- | --- | --- | --- |
| POST | `/invitations/organization` | `system.admin` | Invite user to organization |
| POST | `/invitations/team/:teamId` | `system.admin` + teamCheck | Invite user to team |
| GET | `/invitations` | `system.admin` | Get all pending invitations |
| GET | `/invitations/:id` | `system.admin` | Get invitation detail |
| DELETE | `/invitations/:id` | `system.admin` | Cancel invitation |
| POST | `/invitations/:token/accept` | Public | Accept invitation (with auth) |

<Note>
The `teamCheck` configuration in `@CheckAccess` allows team leaders (users with `team.admin` permission in the specific team) to manage their own teams without requiring organization-level `system.admin` permissions. This enables delegated team management while maintaining security boundaries.
</Note>
