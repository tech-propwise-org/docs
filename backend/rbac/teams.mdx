Teams organize users within an organization and serve as a key component of the unified RBAC system. Teams can have their own team-scoped roles and can restrict which organization-scoped permissions are available to team members through the available permissions system.

## Architecture overview

Teams integrate with the unified RBAC system through:

- **Unified roles**: Teams use the same `Role` table as organizations, with `scope: TEAM` and `team_id` set
- **Permission inheritance**: Team roles can have both TEAM-scoped and ORG-scoped permissions
- **Available permissions**: Controls which ORG-scoped permissions team roles can use
- **Multi-tenancy**: Full RLS support with organization-level isolation
- **Explicit junctions**: `TeamMembershipRole` table manages role assignments

## Team entity

```typescript
Team
├── id: uuid
├── name: string
├── description?: string
├── organization_id: uuid → Organization
├── is_deleted: boolean (soft delete)
├── Relationships:
│   ├── memberships → TeamMembership[]
│   ├── roles → Role[] (where scope = TEAM, team_id = this.id)
│   └── availablePermissions → Permission[] (via team_available_permissions)
└── Timestamps: created_at, updated_at

Indexes:
└── (name, organization_id, is_deleted) [unique, where: is_deleted = false]
```

**Key Rules**:
- Teams belong to exactly one organization
- Team names must be unique within organization (among non-deleted teams)
- `availablePermissions` defines which ORG-scoped permissions team roles can inherit
- Soft delete via `is_deleted` flag with proper indexing

## Team membership

```typescript
TeamMembership
├── id: uuid
├── user_id: uuid → User
├── team_id: uuid → Team
├── organization_id: uuid → Organization
├── joined_at: timestamp
├── is_deleted: boolean (soft delete)
├── Relationships:
│   ├── user → User
│   ├── team → Team
│   └── teamMembershipRoles → TeamMembershipRole[] (roles in this team)
└── Timestamps: created_at, updated_at

Indexes:
└── (user_id, team_id, is_deleted) [unique, where: is_deleted = false]
```

**Key Rules**:
- User can be member of multiple teams within the same organization
- One membership per user per team (enforced only among non-deleted records)
- `organization_id` inherited from team (required for RLS)
- Access team roles via `teamMembershipRoles` collection (explicit junction pattern)

## Available permissions system

Team roles can have ORG-scoped permissions, but only if those permissions are in the team's `availablePermissions`. This provides granular control over permission inheritance:

```typescript
// team_available_permissions junction table
Table team_available_permissions {
  id: uuid (primary key)
  team_id: uuid [ref: > team.id]
  permission_id: uuid [ref: > permission.id]
  organization_id: uuid [ref: > organization.id] // for RLS
  created_at: timestamp
}

// Permission inheritance validation
async function validateTeamRolePermissions(
  teamId: string, 
  permissionIds: string[]
): Promise<void> {
  const team = await em.findOne(Team, { id: teamId }, { 
    populate: ['availablePermissions'] 
  });
  
  const availablePermissionIds = team.availablePermissions
    .getItems()
    .map(p => p.id);

  for (const permissionId of permissionIds) {
    const permission = await em.findOne(Permission, { id: permissionId });
    
    if (permission.scope === PermissionScope.ORG) {
      if (!availablePermissionIds.includes(permissionId)) {
        throw new ValidationError(
          `Permission ${permission.key} not available for team roles`
        );
      }
    }
    // TEAM-scoped permissions are always allowed for team roles
  }
}
```

**Permission Scoping Rules**:
- **TEAM-scoped permissions**: Always allowed on team roles
- **ORG-scoped permissions**: Only allowed if in team's `availablePermissions`
- **System roles on teams**: Can use any permission based on their defined scope

## Team role management

Teams create their own roles using the unified `Role` table:

```typescript
// Creating a team role
const teamRole = new Role({
  name: 'Senior Sales Agent',
  description: 'Advanced sales team member',
  scope: RoleScope.TEAM,
  is_system: false,
  organization_id: team.organization_id,
  team_id: team.id
});

// Team roles inherit organization context but are scoped to specific team
// They can have both TEAM and ORG permissions (if ORG perms are available)
```

**Team Role Characteristics**:
- `scope: TEAM` distinguishes from org-level roles
- `team_id` associates role with specific team
- `organization_id` maintains multi-tenant isolation
- Can combine TEAM-scoped and available ORG-scoped permissions

## Permission resolution

When resolving permissions for a team member, the system combines:

1. **Organization roles** (via `UserOrgRole`)
2. **Team roles** (via `TeamMembershipRole`)
3. **Available permission filtering** for team ORG-scoped permissions

```typescript
async function resolveUserTeamPermissions(
  userId: string, 
  teamId: string
): Promise<Permission[]> {
  // Get user's org roles
  const orgRoles = await getUserOrgRoles(userId);
  
  // Get user's team roles in this specific team
  const teamRoles = await getUserTeamRoles(userId, teamId);
  
  // Get team's available permissions for ORG-scoped filtering
  const team = await em.findOne(Team, { id: teamId }, {
    populate: ['availablePermissions']
  });
  
  // Combine and filter permissions
  const allPermissions = [
    ...getPermissionsFromRoles(orgRoles),
    ...getFilteredTeamPermissions(teamRoles, team.availablePermissions)
  ];
  
  return deduplicatePermissions(allPermissions);
}
```

## Business rules

| Rule | Description | Enforcement |
| --- | --- | --- |
| **Unique membership** | User can only have one membership per team | Database constraint on `(user_id, team_id, is_deleted)` |
| **Organization alignment** | Team membership must match team's organization | Validated in service layer |
| **Default role assignment** | New team members get "Agent" role by default | Automated in `TeamMembershipService.addUserToTeam()` |
| **Team leader requirement** | Each team must have at least one Team Leader | Validated before removing last Team Leader |
| **Available permission validation** | Team roles can only use ORG permissions if available | Enforced in role permission assignment |
| **Soft delete cascade** | Deleting team soft-deletes all memberships | Handled in `TeamService.deleteTeam()` |
| **RLS compliance** | All operations respect organization boundaries | Database-level Row Level Security |

## System integration

<Info>
Teams integrate deeply with other RBAC components:
- **Invitations**: Can invite users directly to teams with specific roles
- **Permission resolution**: Team permissions combine with org permissions
- **RLS**: All team data is isolated by organization
- **System roles**: Team Leader, Agent roles are available across all teams
</Info>

### Row Level Security

All team-related tables include RLS policies:

```sql
-- Teams are isolated by organization
CREATE POLICY team_isolation ON teams FOR ALL TO authenticated
USING (organization_id = get_current_organization_id());

-- Team memberships inherit organization isolation
CREATE POLICY team_membership_isolation ON team_memberships FOR ALL TO authenticated
USING (organization_id = get_current_organization_id());

-- Available permissions are organization-scoped
CREATE POLICY team_available_permissions_isolation ON team_available_permissions FOR ALL TO authenticated
USING (organization_id = get_current_organization_id());
```
