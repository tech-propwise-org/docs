Permissions are the atomic units of access control. Each permission represents a specific capability within a functional domain.

## Permission entity

```typescript
Permission
├── id: uuid
├── key: string (e.g., 'crm.view', 'team_crm.manage')
├── name: string (display name)
├── description?: string
├── group: string (e.g., 'CRM', 'Sales', 'System', 'TeamCRM')
├── scope: PermissionScope (ORG | TEAM)
├── Flags:
│   ├── is_system_sensitive: boolean (only Owner can use; e.g., system.owner)
│   ├── has_admin_access: boolean (requires admin/leadership access)
│   └── has_write_access: boolean (modify vs read-only)
├── Relationships:
│   └── roles → Role[] (ManyToMany, inverse side)
└── Timestamps: created_at, updated_at, is_deleted

Indexes:
└── (key, scope, is_deleted) [unique, where: is_deleted = false]
```

- Permissions are NOT organization-specific (global registry)
- `key` is the programmatic identifier (used in code via `OrgPermissionKey` / `TeamPermissionKey` enums)
- `key` is unique per scope (the same key cannot appear twice for the same scope)
- No RLS on Permission table (global visibility)
- `is_system_sensitive` permissions are reserved for the Owner role (Admin cannot have them)

**Permission scopes:**

- `scope = ORG`: Organization-wide permission (e.g., `crm.view`, `system.admin`)
- `scope = TEAM`: Team-specific permission (e.g., `team_crm.view`, `team.admin`)

## Permission registry

The system uses **domain-level coarse-grained permissions** rather than fine-grained per-entity permissions. Each permission covers a functional domain.

```typescript isAdmin
async function isAdmin(userId: string, organizationId: string): Promise<boolean> {
  const userOrgRoles = await em.find(
    UserOrgRole,
    { user: userId, organization: organizationId, isDeleted: false },
    { populate: ['role'] },
  );

  return userOrgRoles.some(
    (uor) => ['Owner', 'Admin'].includes(uor.role.name) && uor.role.scope === RoleScope.ORG,
  );
}
```

```typescript hasTeamPermission
async function hasTeamPermission(
  userId: string, teamId: string, permissionKey: string,
): Promise<boolean> {
  const membership = await em.findOne(
    TeamMembership,
    { user: userId, team: teamId, isDeleted: false },
    { populate: ['teamMembershipRoles.role.permissions'] },
  );
  if (!membership) return false;

  for (const tmr of membership.teamMembershipRoles.getItems()) {
    if (tmr.role.permissions.getItems().some((p) => p.key === permissionKey)) {
      return true;
    }
  }
  return false;
}
```
</CodeGroup>

## Business rules

| Rule | Description |
| --- | --- |
| **Permission scope** | ORG permissions can be on ORG or TEAM roles |
| **Team permissions** | TEAM permissions can only be on TEAM roles |
| **Team available permissions** | ORG permissions on TEAM roles must be in `team.availablePermissions` |
| **System permissions** | Permissions with `has_admin_access=true` only for Owner/Admin |

## Services

<AccordionGroup>
<Accordion title="PermissionService">
**Location:** `rbac/permission/permission.service.ts`

Handles permission checks and effective permission resolution.

| Method | Signature | Description |
| --- | --- | --- |
| `hasPermission` | `(userId, permissionKey, resourceOwnerId?, resourceCreatorId?, organizationId?)` → `PermissionCheckResult` | Checks permission with resource ownership context |
| `getUserEffectivePermissions` | `(userId, organizationId?)` → `EffectivePermissions` | Resolves all org + team permissions for a user |
</Accordion>
<Accordion title="PermissionSeederService">
**Location:** `rbac/permission/permission-seeder.service.ts`

Manages permission and role seeding lifecycle. Reads from `permission-definitions.ts` to seed/sync the global permission registry.

| Method | Description |
| --- | --- |
| `seedOrganizationPermissions()` | Initial seed of permissions for an organization |
| `syncGlobalPermissions(em)` | Adds new permissions, removes obsolete ones, cleans affected roles, reassigns users with deleted custom roles to fallback roles |
| `syncOrganizationPermissions(organization)` | Syncs permissions for a specific organization |
| `updateGlobalPermissions()` | Triggers global permission update (called from reseed endpoint) |
</Accordion>
</AccordionGroup>
