## Architecture overview

The RBAC system follows a **unified role & permission architecture** with explicit junction tables to support multi-organization tenancy and comprehensive audit trails.

### Design principles

<Tabs>
<Tab title="Unified system">
- **Single Role table** for both org-level and team-level roles
- **Single Permission table** for all permissions  
- `scope` field differentiates between ORG and TEAM levels
- System roles have `organization = NULL` (global/seeded)
- Custom roles have `organization = specific org ID`
</Tab>
<Tab title="Multi-tenancy">
- Users can belong to multiple organizations
- Same user can have different roles in different organizations
- Explicit junction tables store organization context
- RLS (Row-Level Security) enforced at database level
</Tab>
<Tab title="Junction tables">
- **UserOrgRole** - User ↔ Role assignments in Organization context
- **TeamMembershipRole** - TeamMembership ↔ Role assignments  
- Include metadata: `assigned_by`, `created_at`, `organization_id` (for RLS)
- Required for multi-org support (simple pivot tables cannot store org context)
</Tab>
</Tabs>

### System vs custom roles

The unified Role system distinguishes between system and custom roles:

| Role Type | Organization Field | Visibility | Examples |
| --- | --- | --- | --- |
| **System roles** | `organization = NULL` | Available to all orgs | Owner, Admin, Basic, Team Leader, Agent |
| **Custom roles** | `organization = specific org ID` | Only visible within that org | "Sales Manager", "Support Specialist" |

## Explicit junction tables

### Why explicit junction tables?

<Steps>
<Step title="Multi-org context support">
A user can have different roles in different organizations. Simple pivot tables cannot store organization context required for multi-tenancy.
</Step>
<Step title="Metadata tracking">
Track who assigned the role (`assigned_by_id`), when it was assigned, and when removed for comprehensive audit trails.
</Step>
<Step title="RLS compatibility">
Need `organization_id` field for database-level filtering via Row-Level Security policies.
</Step>
<Step title="Permission resolution">
Explicit entities enable complex permission resolution logic that combines org-level and team-level roles with inheritance rules.
</Step>
</Steps>

### UserOrgRole junction table

Manages User ↔ Role assignments in Organization context.

```typescript
UserOrgRole
├── id: uuid (primary key)
├── user_id: uuid → User
├── role_id: uuid → Role (system role with organization = null)
├── organization_id: uuid → Organization (REQUIRED for RLS)
├── assigned_by_id?: uuid → User (who assigned this role)
├── Timestamps:
│   ├── created_at: timestamp
│   ├── updated_at: timestamp
│   └── is_deleted: boolean (soft delete)

Indexes:
├── (user_id, role_id, organization_id) [unique, where: is_deleted = false]
├── (user_id, organization_id)
└── (organization_id, role_id)
```

<Tip>
The unique constraint prevents duplicate role assignments but allows re-assignment after soft deletion.
</Tip>

#### Usage patterns

```typescript
// Find user's org roles
const userOrgRoles = await em.find(
  UserOrgRole,
  { user: userId, organization: organizationId, isDeleted: false },
  { populate: ['role', 'role.permissions'] },
);
const roles = userOrgRoles.map((uor) => uor.role);

// Assign role to user
const userOrgRole = em.create(UserOrgRole, {
  user, role, organization, assignedBy: currentUser,
});
await em.persistAndFlush(userOrgRole);
```

### TeamMembershipRole junction table

Manages TeamMembership ↔ Role assignments within team context.

```typescript
TeamMembershipRole
├── id: uuid (primary key)
├── team_membership_id: uuid → TeamMembership
├── role_id: uuid → Role
├── organization_id: uuid → Organization (REQUIRED for RLS, denormalized from TeamMembership)
├── assigned_by_id?: uuid → User (who assigned this role)
├── Timestamps:
│   ├── created_at: timestamp
│   ├── updated_at: timestamp
│   └── is_deleted: boolean (soft delete)

Indexes:
└── (team_membership_id, role_id) [unique, where: is_deleted = false]
```

<Note>
`organization_id` is denormalized from `TeamMembership.organization` for RLS compatibility. This ensures team membership roles are properly filtered by organization context.
</Note>

#### Business rules

- Team membership must exist before assigning roles
- Only TEAM-scoped or ORG-scoped roles can be assigned to team memberships
- Role assignment requires appropriate permissions in the organization
- Soft deletion preserves audit history

#### Usage patterns

```typescript
// Find team member's roles
const membership = await em.findOne(
  TeamMembership,
  { user: userId, team: teamId },
  { populate: ['teamMembershipRoles.role', 'teamMembershipRoles.role.permissions'] },
);
const roles = membership.teamMembershipRoles.getItems().map((tmr) => tmr.role);

// Assign role to team member
const teamMembershipRole = em.create(TeamMembershipRole, {
  teamMembership: membership, 
  role,
  organization: membership.organization, 
  assignedBy: currentUser,
});
await em.persistAndFlush(teamMembershipRole);
```

### Organization field requirements

Both junction tables include `organization_id` for RLS compatibility and tenant isolation.

| Junction | How `organization_id` is set | Purpose |
| --- | --- | --- |
| **UserOrgRole** | Explicitly set when assigning role. Allows different roles per org. | Multi-org role management |
| **TeamMembershipRole** | Inherited from `TeamMembership.organization`. Denormalized for RLS. | Team-level role filtering |

<Warning>
The `organization_id` field is critical for RLS policies. Never create junction records without properly setting this field.
</Warning>

## Permission resolution

The RBAC system uses a hierarchical permission resolution system:

<Steps>
<Step title="Collect org-level roles">
Query `UserOrgRole` to get all roles assigned to the user in the organization context.
</Step>
<Step title="Collect team-level roles">
Query `TeamMembershipRole` through `TeamMembership` to get all team roles for the user.
</Step>
<Step title="Combine permissions">
Union all permissions from org roles and team roles. Team roles can have both TEAM-scoped and ORG-scoped permissions.
</Step>
<Step title="Apply permission hierarchy">
Owner > Admin > Basic. Team Leader and Agent permissions are additive within team context.
</Step>
<Step title="Filter by available permissions">
For team operations, filter permissions through `team_available_permissions` to ensure the team has access to the permission.
</Step>
</Steps>

### Permission resolution service

```typescript
@Injectable()
export class PermissionService {
  async getUserPermissions(userId: string, organizationId: string): Promise<Permission[]> {
    // Get org-level permissions
    const orgRoles = await this.em.find(
      UserOrgRole,
      { user: userId, organization: organizationId, isDeleted: false },
      { populate: ['role.permissions'] },
    );
    const orgPermissions = orgRoles.flatMap(uor => uor.role.permissions.getItems());

    // Get team-level permissions
    const teamMemberships = await this.em.find(
      TeamMembership,
      { user: userId, organization: organizationId, isDeleted: false },
      { populate: ['teamMembershipRoles.role.permissions'] },
    );
    const teamPermissions = teamMemberships.flatMap(tm =>
      tm.teamMembershipRoles.getItems().flatMap(tmr =>
        tmr.role.permissions.getItems()
      )
    );

    // Combine and deduplicate
    const allPermissions = [...orgPermissions, ...teamPermissions];
    return Array.from(new Set(allPermissions));
  }

  async hasPermission(
    userId: string, organizationId: string, permissionKey: string, teamId?: string
  ): Promise<boolean> {
    const permissions = await this.getUserPermissions(userId, organizationId);
    const hasPermission = permissions.some(p => p.key === permissionKey);
    
    if (!hasPermission) return false;
    
    // For team-scoped operations, check team available permissions
    if (teamId) {
      const team = await this.em.findOne(
        Team,
        { id: teamId },
        { populate: ['availablePermissions'] }
      );
      return team?.availablePermissions.getItems().some(p => p.key === permissionKey) ?? false;
    }
    
    return true;
  }
}
```

## System roles

The RBAC system includes five predefined system roles:

<Tabs>
<Tab title="Organization roles">
- **Owner**: Full control over organization, including deletion and billing
- **Admin**: Manage users, teams, and most organization settings (cannot delete org)
- **Basic**: Access to CRM features based on assigned permissions
</Tab>
<Tab title="Team roles">
- **Team Leader**: Manage team members, assign team roles, and access team-specific features
- **Agent**: Standard team member with basic permissions
</Tab>
</Tabs>

### System role characteristics

- `organization_id = NULL` (shared across all organizations)
- `is_system = true` (cannot be modified or deleted)
- Seeded during application bootstrap
- Available for assignment in any organization

```typescript
async function assignOrgRole(
  userId: string, roleId: string, organizationId: string, assignedById: string,
): Promise<void> {
  const user = await em.findOneOrFail(User, { id: userId });
  const role = await em.findOneOrFail(Role, { id: roleId });
  const organization = await em.findOneOrFail(Organization, { id: organizationId });
  const assignedBy = await em.findOneOrFail(User, { id: assignedById });

  const existing = await em.findOne(UserOrgRole, {
    user: userId, role: roleId, organization: organizationId, isDeleted: false,
  });
  if (existing) throw new BadRequestException('Role already assigned to user');

  const userOrgRole = em.create(UserOrgRole, { user, role, organization, assignedBy });
  await em.persistAndFlush(userOrgRole);
}
```

## Business rules

### Core business rules

1. **Role assignment constraints**:
   - Users can only be assigned system roles or custom roles from their organization
   - Team roles can only be assigned to team members
   - Owner role is automatically assigned to organization creator

2. **Permission inheritance**:
   - Team roles can have both TEAM-scoped and ORG-scoped permissions
   - Team operations require permissions to be in team's available permissions
   - Higher-level roles (Owner > Admin > Basic) include lower-level permissions

3. **Multi-tenancy isolation**:
   - All role assignments are scoped to organization
   - RLS ensures users only see data from their current organization context
   - System roles are shared across organizations but assignments are per-org

### Validation rules

<AccordionGroup>
<Accordion title="UserOrgRole validation">
- User must be member of the organization
- Role must be system role or belong to the same organization
- Cannot assign duplicate active roles to same user
- Owner role can only be assigned once per organization
</Accordion>
<Accordion title="TeamMembershipRole validation">
- TeamMembership must exist and be active
- Role scope must be TEAM or ORG
- Cannot assign duplicate active roles to same team membership
- Team must have the required permissions in availablePermissions
</Accordion>
</AccordionGroup>

## RLS integration

### RLS-enabled tables

| Table | RLS Policy | Notes |
| --- | --- | --- |
| `user` | `organization_isolation_user` | Users visible only in current org context |
| `organization` | No RLS | Users can see all orgs they belong to |
| `role` | Special handling | System roles (org=NULL) visible to all, custom roles filtered by org |
| `permission` | No RLS | Global permission registry |
| `team` | `organization_isolation_team` | Teams filtered by organization |
| `team_membership` | `organization_isolation_team_membership` | Memberships filtered by organization |
| `user_org_role` | `organization_isolation_user_org_role` | Filtered by organization |
| `team_membership_role` | `organization_isolation_team_membership_role` | Filtered by organization |
| `invitation` | `organization_isolation_invitation` | Invitations filtered by organization |

### RLS policies for junction tables

```sql
ALTER TABLE "user_org_role" ENABLE ROW LEVEL SECURITY;
CREATE POLICY "organization_isolation_user_org_role" ON "user_org_role"
  USING (
    organization_id = current_setting('app.organization_id')::uuid
    OR current_setting('app.bypass_rls', true) = 'true'
  );
ALTER TABLE "user_org_role" FORCE ROW LEVEL SECURITY;

ALTER TABLE "team_membership_role" ENABLE ROW LEVEL SECURITY;
CREATE POLICY "organization_isolation_team_membership_role" ON "team_membership_role"
  USING (
    organization_id = current_setting('app.organization_id')::uuid
    OR current_setting('app.bypass_rls', true) = 'true'
  );
ALTER TABLE "team_membership_role" FORCE ROW LEVEL SECURITY;
```

### Role table special handling

The `Role` table requires special handling because system roles have `organization = NULL`:

```sql
-- Role RLS Policy (if needed)
-- Note: Currently Role table does NOT have RLS enabled
-- If we enable it, we need this logic:

DROP POLICY IF EXISTS "organization_isolation_role" ON "role";
CREATE POLICY "organization_isolation_role" ON "role"
  USING (
    organization_id = current_setting('app.organization_id')::uuid
    OR organization_id IS NULL  -- System roles visible to all
    OR current_setting('app.bypass_rls', true) = 'true'
  );
```

<Warning>
Role table does NOT have RLS enabled. System roles (`organization = NULL`) and custom roles are both fetched without RLS filtering. Application logic handles visibility.
</Warning>

### Bypassing RLS

```typescript
import { BypassRLS } from '@decorators/rls.decorator';

@Injectable()
export class SystemService {
  @BypassRLS()

    const allUserOrgRoles = await this.em.find(UserOrgRole, {});
```

### RLS context setting

The organization context is set automatically by `TenantMiddleware`:

```typescript
@Injectable()
export class TenantMiddleware implements NestMiddleware {
  async use(req: Request, res: Response, next: NextFunction) {
    const organizationId = req.user?.organizationId;
    if (organizationId) {
      await this.em.execute(`SET LOCAL app.organization_id = '${organizationId}'`);
    }
    next();
  }
}
```
