## Overview

The Queue module provides a centralized job queue system for PropWise CRM using PostgreSQL-based job processing via the pg-boss library. This module enables asynchronous processing of background tasks such as notifications, data processing, and other long-running operations that should be decoupled from HTTP request/response cycles.

The module implements a queue service abstraction that allows other modules to enqueue jobs for background processing while providing features like job retries, priorities, batching, and worker management.

## Architecture

### Design Decisions

The Queue module follows these key architectural principles:

- **PostgreSQL-based queuing**: Uses pg-boss library for reliability and ACID compliance
- **Interface-driven design**: Implements `IQueueService` interface for potential future queue provider swapping
- **Global module**: Available throughout the application without explicit imports
- **Worker abstraction**: Provides flexible worker registration with configurable options
- **Lifecycle management**: Properly handles startup and shutdown of queue connections

### Queue Provider Selection

The module uses pg-boss as the underlying queue implementation because:

## Interface Definitions

### QueueJob Interface

Represents a job in the queue system with metadata and payload:

```typescript
interface QueueJob<T> {
  id: string; // Unique job identifier
  data: T; // Job payload data
  retryCount: number; // Current retry attempt count
  retryLimit: number; // Maximum retry attempts allowed
}
```

### QueueJobOptions Interface

Configuration options for job enqueueing with flexible retry and deduplication settings:

```typescript
interface QueueJobOptions {
  retryLimit?: number; // Max retry attempts (default: 3)
  retryDelay?: number; // Delay between retries in seconds
  retryBackoff?: boolean; // Use exponential backoff for retries
  expireInSeconds?: number; // Job expiration time
  singletonKey?: string; // Ensure only one job with this key exists
  priority?: number; // Job priority (higher = more priority)
}
```

### QueuePolicy Types

<Warning>
The queue policy is critical for proper deduplication behavior. The default `'standard'` policy does **not** enforce unique constraints.
</Warning>

Controls how `singletonKey` deduplication is enforced at the database level via pg-boss policy-specific unique indexes:

```typescript
type QueuePolicy =
  | 'standard' // No dedup â€” singletonKey stored but no unique constraint
  | 'short' // â‰¤ 1 job per singletonKey in `created` state
  | 'singleton' // â‰¤ 1 job per singletonKey in `active` state
  | 'stately' // â‰¤ 1 job per singletonKey in created | retry | active
  | 'exclusive' // Same uniqueness as stately, different fetch semantics
  | 'key_strict_fifo';
```

<Note>
**Critical:** The default `'standard'` policy applies **no unique index** on `singleton_key`. Any queue that uses `singletonKey` for deduplication **must** be created with `policy: 'stately'` (or `'exclusive'`), otherwise `ON CONFLICT DO NOTHING` never fires and every insert creates a new row regardless of pending jobs for the same key.
</Note>

## IQueueService Interface

The core queue service contract provides comprehensive job management capabilities:

| Method | Description |
|--------|-------------|
| `enqueue<T>(queueName, data, options?)` | Add single job to queue |
| `enqueueBatch<T>(queueName, data[], options?)` | Add multiple jobs to queue |
| `registerWorker<T>(queueName, handler, options?)` | Register job processor (accepts `policy`) |
| `schedule(queueName, cron, data?)` | Register recurring cron (pg-boss `schedule`) |
| `unschedule(queueName, key?)` | Remove recurring schedule (pg-boss `unschedule`) |
| `purgeQueuedJobs(queueName)` | Delete pending jobs (`created` + `retry`); leaves `active` |
| `checkHealth(timeoutMs?)` | Active liveness probe |
| `start()` | Start queue processing |
| `stop()` | Stop queue processing |

### Health Monitoring

The `checkHealth(timeoutMs?)` method provides active liveness probing:

<Steps>
  <Step title="Health Check Process">
    Returns `Promise<QueueHealthResult>` with `{ healthy, reason?, latencyMs? }` and never throws
  </Step>
  <Step title="Connection Validation">
    Round-trips `SELECT 1` through pg-boss's own connection, bounded by `QUEUE_HEALTH_PROBE_TIMEOUT_MS`
  </Step>
  <Step title="Status Reporting">
    - Not started â†’ `{ healthy: false, reason: 'boss.start() has not completed' }`
    - Query rejects â†’ `{ healthy: false, reason: <driver error> }`  
    - Query timeout â†’ `{ healthy: false, reason: 'queue health probe timed out after <n>ms' }`
    - Success â†’ `{ healthy: true, latencyMs }`
  </Step>
</Steps>

<Info>
The health probe uses pg-boss's connection (superuser `DATABASE_URL` pool), covering a gap that the readiness endpoint's ORM-based database ping (`RUNTIME_DATABASE_URL` pool) does not check.
</Info>

## Service Implementation

### PgBossQueueService

The main service implementation using pg-boss with comprehensive lifecycle management:

```typescript
@Injectable()
export class PgBossQueueService implements IQueueService, OnModuleInit, OnModuleDestroy
```

  <Accordion title="Key Features">

  <Accordion title="Connection Management">

    - **Primary connection**: `app.database.url` (superuser/owner pool)

    - **Certificate validation**: Against Node's trust store for managed databases
    - **Graceful shutdown**: Via `app.enableShutdownHooks()` in `src/main.ts`
  </Accordion>
  
  <Accordion title="Worker Processing">
    - **Parallel per-job processing**: `batchSize` maps to pg-boss `localConcurrency`

## Business Rules

### Job Processing Rules

<Steps>
  <Step title="Retry Logic">
    Jobs can be configured with retry limits and exponential backoff strategies
  </Step>
  <Step title="Job Uniqueness">
    `singletonKey` deduplication requires `policy: 'stately'` or `'exclusive'` - default `'standard'` policy stores keys without DB constraints
  </Step>
  <Step title="Priority Processing">
    Higher priority jobs are processed before lower priority ones
  </Step>
  <Step title="Job Expiration">
    Jobs automatically expire if not processed within specified timeframe
  </Step>
  <Step title="Batch Processing">
    Workers can process multiple jobs simultaneously with configurable concurrency
  </Step>
  <Step title="Authorization Handling">
    Queue payloads should carry stable identifiers, not trusted authorization snapshots. Workers performing user-initiated writes must reload current permissions before execution.
  </Step>
</Steps>
