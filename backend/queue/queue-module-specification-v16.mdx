## Overview

The Queue module provides a centralized job queue system for PropWise CRM using PostgreSQL-based job processing via the pg-boss library. This module enables asynchronous processing of background tasks such as notifications, data processing, and other long-running operations that should be decoupled from HTTP request/response cycles.

<Note>
The module implements a queue service abstraction that allows other modules to enqueue jobs for background processing while providing features like job retries, priorities, batching, and worker management.
</Note>

## Architecture

### Design Decisions

The Queue module is built on several key architectural principles:

- **PostgreSQL-based queuing**: Uses pg-boss library for reliability and ACID compliance
- **Interface-driven design**: Implements `IQueueService` interface for potential future queue provider swapping
- **Global module**: Available throughout the application without explicit imports
- **Worker abstraction**: Provides flexible worker registration with configurable options
- **Lifecycle management**: Properly handles startup and shutdown of queue connections

### Queue Provider Selection

<Info>
The module uses pg-boss as the underlying queue implementation because it leverages existing PostgreSQL infrastructure, provides ACID guarantees for job persistence, supports advanced features like job scheduling and retries, and requires no additional infrastructure dependencies.
</Info>

## Core Interfaces

### QueueJob

Represents a job in the queue system:

```typescript
interface QueueJob<T> {
  id: string; // Unique job identifier
  data: T; // Job payload data
  retryCount: number; // Current retry attempt count
  retryLimit: number; // Maximum retry attempts allowed
}
```

### QueueJobOptions

Configuration options for job enqueueing:

```typescript
interface QueueJobOptions {
  retryLimit?: number; // Max retry attempts (default: 3)
  retryDelay?: number; // Delay between retries in seconds
  retryBackoff?: boolean; // Use exponential backoff for retries
  expireInSeconds?: number; // Job expiration time
  singletonKey?: string; // Ensure only one job with this key exists
  priority?: number; // Job priority (higher = more priority)
}
```

### QueuePolicy

<Warning>
The default `'standard'` policy applies **no unique index** on `singleton_key`. Any queue that uses `singletonKey` for deduplication **must** be created with `policy: 'stately'` (or `'exclusive'`), otherwise `ON CONFLICT DO NOTHING` never fires and every insert creates a new row regardless of how many pending jobs already exist for the same key.
</Warning>

Controls how `singletonKey` deduplication is enforced at the database level:

```typescript
type QueuePolicy =
  | 'standard' // No dedup — singletonKey stored but no unique constraint
  | 'short' // ≤ 1 job per singletonKey in `created` state
  | 'singleton' // ≤ 1 job per singletonKey in `active` state
  | 'stately' // ≤ 1 job per singletonKey in created | retry | active
  | 'exclusive' // Same uniqueness as stately, different fetch semantics
  | 'key_strict_fifo';
```

### Worker Configuration

<Tabs>
  <Tab title="WorkerOptions">
    Configuration for single-job workers:

    ```typescript
    interface WorkerOptions {
      concurrency?: number; // Independent worker invocations (default: 1)
      pollingIntervalSeconds?: number; // How often to poll for new jobs
    }
    ```
  </Tab>

  <Tab title="BatchWorkerOptions">
    Configuration for batch workers:

    ```typescript
    interface BatchWorkerOptions {
      batchSize?: number; // Jobs per handler invocation (default: 1)
      pollingIntervalSeconds?: number; // How often to poll for new jobs
      policy?: QueuePolicy; // Queue policy — REQUIRED for singletonKey dedup
    }
    ```

    <Note>
    `policy` is enforced by `ensureQueue` inside `registerBatchWorker`. Use `'stately'` whenever jobs are submitted with `singletonKey` and you need "at most one pending/active job per key".
    </Note>
  </Tab>
</Tabs>

## Business Rules

### Job Processing Rules

<Tip>
Queue payloads should carry stable identifiers, not trusted authorization snapshots. Workers that perform user-initiated writes must reload current organization membership and RBAC permissions before execution.
</Tip>

1. **Retry Logic**: Jobs can be configured with retry limits and backoff strategies
2. **Job Uniqueness**: `singletonKey` deduplication only enforced with `policy: 'stately'` or `'exclusive'`
3. **Priority Processing**: Higher priority jobs are processed first
4. **Expiration**: Jobs can expire if not processed within specified time
5. **Batch Processing**: Workers can process multiple jobs simultaneously
6. **Current Authorization**: Workers must reload permissions for user-initiated operations

### Queue Management

1. **Queue Creation**: Queues are created automatically when first used
2. **Worker Registration**: Each queue can have multiple workers with different configurations
3. **Connection Management**: Single pg-boss instance per service lifetime
4. **Signal Handling**: Graceful shutdown via Nest lifecycle hooks

### Auto-Creation Pattern

`PgBossQueueService` lazily creates queues on first use via a private `ensureQueue(queueName, policy?)` method:

```typescript
private async ensureQueue(queueName: string, policy?: QueuePolicy): Promise<void> {
  if (this.createdQueues.has(queueName)) {
    if (policy) {
      await this.enforceQueuePolicy(queueName, policy);
    }
    return;
  }

  await this.boss.createQueue(queueName, policy ? { policy } : undefined);
  if (policy) {
    await this.enforceQueuePolicy(queueName, policy);
  }
  this.createdQueues.add(queueName);
}
```

<Note>
This method is called internally by `enqueue()`, `enqueueBatch()`, `registerWorker()`, and `registerBatchWorker()` — callers never need to create queues manually.
</Note>

## Testing Framework

### E2E Testing

The queue module includes comprehensive E2E testing support:

- **jest-e2e-queue.json**: Runs distribution, escalation, and misc queue E2E against live pg-boss
- **Matrix Strategy**: PR CI uses queue-domain matrix for distribution, escalation, and misc queue specs
- **Ephemeral Branches**: Each matrix entry creates isolated Neon branch with domain-specific suffix
- **Worker Control**: `ESCALATION_SKIP_PGBOSS_WORKERS=true` prevents cron dispatcher registration during tests

<Warning>
The `EscalationSchedulerService.onModuleInit()` skips registering the cron dispatcher during queue E2E tests, but the per-org `escalation-org` worker stays alive for end-to-end testing.
</Warning>

### Test Harness

The `EscalationQueueHarness` provides shared abstraction for flush + dispatch + await patterns in queue testing scenarios.
