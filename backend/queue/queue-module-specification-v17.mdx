## Overview

The Queue module provides a centralized job queue system for PropWise CRM using PostgreSQL-based job processing via the pg-boss library. This module enables asynchronous processing of background tasks such as notifications, data processing, and other long-running operations that should be decoupled from HTTP request/response cycles.

The module implements a queue service abstraction that allows other modules to enqueue jobs for background processing while providing features like job retries, priorities, batching, and worker management.

## Core Interfaces

### QueueJob

Represents a job in the queue system.

```typescript
interface QueueJob<T> {
  id: string; // Unique job identifier
  data: T; // Job payload data
  retryCount: number; // Current retry attempt count
  retryLimit: number; // Maximum retry attempts allowed
}
```

### QueueJobOptions

Configuration options for job enqueueing.

```typescript
interface QueueJobOptions {
  retryLimit?: number; // Max retry attempts (default: 3)
  retryDelay?: number; // Delay between retries in seconds
  retryBackoff?: boolean; // Use exponential backoff for retries
  expireInSeconds?: number; // Job expiration time
  singletonKey?: string; // Ensure only one job with this key exists
  priority?: number; // Job priority (higher = more priority)
}
```

### QueuePolicy

<Warning>
The default `'standard'` policy applies **no unique index** on `singleton_key`. Any queue that uses `singletonKey` for deduplication **must** be created with `policy: 'stately'` (or `'exclusive'`), otherwise `ON CONFLICT DO NOTHING` never fires and every insert creates a new row regardless of how many pending jobs already exist for the same key.
</Warning>

Controls how `singletonKey` deduplication is enforced at the database level via pg-boss policy-specific unique indexes.

```typescript
type QueuePolicy =
  | 'standard' // No dedup — singletonKey stored but no unique constraint
  | 'short' // ≤ 1 job per singletonKey in `created` state
  | 'singleton' // ≤ 1 job per singletonKey in `active` state
  | 'stately' // ≤ 1 job per singletonKey in created | retry | active
  | 'exclusive' // Same uniqueness as stately, different fetch semantics
  | 'key_strict_fifo';
```

### Worker Configuration

<Tabs>
  <Tab title="WorkerOptions">
    Configuration for single-job workers.

    ```typescript
    interface WorkerOptions {
      concurrency?: number; // Independent worker invocations (default: 1)
      pollingIntervalSeconds?: number; // How often to poll for new jobs
    }
    ```
  </Tab>
  <Tab title="BatchWorkerOptions">
    Configuration for batch workers.

    ```typescript
    interface BatchWorkerOptions {
      batchSize?: number; // Jobs per handler invocation (default: 1)
      pollingIntervalSeconds?: number; // How often to poll for new jobs
      policy?: QueuePolicy; // Queue policy — REQUIRED for singletonKey dedup
    }
    ```

    <Note>
    `policy` is enforced by `ensureQueue` inside `registerBatchWorker`. Use `'stately'` whenever jobs are submitted with `singletonKey` and you need "at most one pending/active job per key".
    </Note>
  </Tab>
</Tabs>

## IQueueService Interface

Core queue service contract with the following methods:

| Method | Description |
|--------|-------------|
| `enqueue<T>(queueName, data, options?)` | Add single job to queue |
| `enqueueBatch<T>(queueName, data[], options?)` | Add multiple jobs to queue |
| `registerWorker<T>(queueName, handler, options?)` | Register job processor |
| `schedule(queueName, cron, data?)` | Register recurring cron (pg-boss `schedule`) |
| `unschedule(queueName, key?)` | Remove recurring schedule (pg-boss `unschedule`) |
| `start()` | Start queue processing |
| `stop()` | Stop queue processing |

## Service Implementation

### PgBossQueueService

The main service implementation using pg-boss:

```typescript
@Injectable()
export class PgBossQueueService implements IQueueService, OnModuleInit, OnModuleDestroy
```

<AccordionGroup>
  <Accordion title="Key Features">
    - **Automatic queue creation** tracking via `createdQueues` Set
    - **Database connection** configured via `ConfigService` from centralized `app.database.*` config
    - **Lifecycle hooks** for proper startup/shutdown
    - **Worker registration** with type-safe job handlers
    - **Parallel per-job processing** via `localConcurrency`
    - **Test-only utilities** for E2E testing scenarios
  </Accordion>
  <Accordion title="Parallel Processing Details">
    When `batchSize` is specified in `WorkerOptions`, it is mapped to pg-boss's `localConcurrency` option (with `batchSize: 1` at the pg-boss level). This spawns N independent workers that each process one job at a time. pg-boss auto-completes on resolve or auto-fails on reject per handler invocation, avoiding the batch auto-completion pitfall where pg-boss `batchSize > 1` would complete/fail the entire batch as a unit.
  </Accordion>
  <Accordion title="Test Utilities">
    **Test-only `__test__offWork(queueName)`** (throws unless `NODE_ENV=test`): wraps pg-boss `offWork()`, detaching all workers subscribed to that queue so jobs remain queued without being fetched. Used by distribution E2E tests together with `DistributionJobHandler.__test__resumeDistributionWorker()` to build a backlog and verify priority ordering.
  </Accordion>
</AccordionGroup>

## Business Rules

### Job Processing Rules

<Steps>
  <Step title="Retry Logic">
    Jobs can be configured with retry limits and backoff strategies
  </Step>
  <Step title="Job Uniqueness">
    `singletonKey` deduplication is only enforced when the queue is created with `policy: 'stately'` (or `'exclusive'`)
  </Step>
  <Step title="Priority Processing">
    Higher priority jobs are processed first
  </Step>
  <Step title="Expiration">
    Jobs can expire if not processed within specified time
  </Step>
  <Step title="Batch Processing">
    Workers can process multiple jobs simultaneously
  </Step>
</Steps>

<Warning>
**Current Authorization for User-Initiated Jobs**: Queue payloads should carry stable identifiers, not trusted authorization snapshots. Workers that perform user-initiated writes must reload current organization membership and RBAC permissions before execution, then fail the domain job if the actor was removed, deactivated, or lost the permission needed for that queued operation.
</Warning>

### Queue Management

1. **Queue Creation**: Queues are created automatically when first used
2. **Worker Registration**: Each queue can have multiple workers with different configurations
3. **Connection Management**: Single pg-boss instance per service lifetime
4. **Signal Handling**: The application bootstrap enables Nest shutdown hooks for graceful shutdown

### Auto-Creation Pattern

<Tip>
`PgBossQueueService` lazily creates queues on first use via a private `ensureQueue(queueName, policy?)` method. A `createdQueues` Set tracks which queues have been created in the current process lifetime so `createQueue()` is called at most once per queue name.
</Tip>

```typescript
private async ensureQueue(queueName: string, policy?: QueuePolicy): Promise<void> {
  if (this.createdQueues.has(queueName)) {
    if (policy) {
      await this.enforceQueuePolicy(queueName, policy);
    }
    return;
  }

  await this.boss.createQueue(queueName, policy ? { policy } : undefined);
  if (policy) {
    await this.enforceQueuePolicy(queueName, policy);
  }
  this.createdQueues.add(queueName);
}
```

This is called internally by `enqueue()`, `enqueueBatch()`, `registerWorker()`, and `registerBatchWorker()` — callers never need to create queues manually.
