## Overview

The Queue module provides a centralized job queue system for PropWise CRM using PostgreSQL-based job processing via the pg-boss library. This module enables asynchronous processing of background tasks such as notifications, data processing, and other long-running operations that should be decoupled from HTTP request/response cycles.

<Note>
The module implements a queue service abstraction that allows other modules to enqueue jobs for background processing while providing features like job retries, priorities, batching, and worker management.
</Note>

## Architecture

### Design Decisions

The Queue module follows these key architectural principles:

- **PostgreSQL-based queuing**: Uses pg-boss library for reliability and ACID compliance
- **Interface-driven design**: Implements `IQueueService` interface for potential future queue provider swapping
- **Global module**: Available throughout the application without explicit imports
- **Worker abstraction**: Provides flexible worker registration with configurable options
- **Lifecycle management**: Properly handles startup and shutdown of queue connections

### Queue Provider Selection

<Info>
The module uses pg-boss as the underlying queue implementation because it leverages existing PostgreSQL infrastructure, provides ACID guarantees for job persistence, supports advanced features like job scheduling, retries, and priorities, and requires no additional infrastructure dependencies.
</Info>

## Interfaces

### QueueJob

Represents a job in the queue system:

```typescript
interface QueueJob<T> {
  id: string; // Unique job identifier
  data: T; // Job payload data
  retryCount: number; // Current retry attempt count
  retryLimit: number; // Maximum retry attempts allowed
}
```

### QueueJobOptions

Configuration options for job enqueueing:

```typescript
interface QueueJobOptions {
  retryLimit?: number; // Max retry attempts (default: 3)
  retryDelay?: number; // Delay between retries in seconds
  retryBackoff?: boolean; // Use exponential backoff for retries
  expireInSeconds?: number; // Job expiration time
  singletonKey?: string; // Ensure only one job with this key exists
  priority?: number; // Job priority (higher = more priority)
}
```

### WorkerOptions

Configuration for job workers:

```typescript
interface WorkerOptions {
  batchSize?: number; // Number of jobs to process in batch
  pollingIntervalSeconds?: number; // How often to poll for new jobs
}
```

### IQueueService Interface

Core queue service contract with the following methods:

| Method | Description |
|--------|-------------|
| `enqueue<T>(queueName, data, options?)` | Add single job to queue |
| `enqueueBatch<T>(queueName, data[], options?)` | Add multiple jobs to queue |
| `registerWorker<T>(queueName, handler, options?)` | Register job processor |
| `start()` | Start queue processing |
| `stop()` | Stop queue processing |

## Service Implementation

### PgBossQueueService

The main service implementation using pg-boss:

```typescript
@Injectable()
export class PgBossQueueService implements IQueueService, OnModuleInit, OnModuleDestroy
```

<Tip>
**Parallel per-job processing via `localConcurrency`**: When `batchSize` is specified in `WorkerOptions`, it is mapped to pg-boss's `localConcurrency` option (with `batchSize: 1` at the pg-boss level). This spawns N independent workers that each process one job at a time.
</Tip>

**Key Features:**

- Automatic queue creation tracking via `createdQueues` Set
- Database connection configured via `ConfigService`
- Lifecycle hooks for proper startup/shutdown
- Worker registration with type-safe job handlers
- pg-boss auto-completes on resolve or auto-fails on reject per handler invocation

### Auto-Creation Pattern

<Note>
`PgBossQueueService` lazily creates queues on first use via a private `ensureQueue()` method. A `createdQueues` Set tracks which queues have been created in the current process lifetime so `createQueue()` is called at most once per queue name.
</Note>

```typescript
private async ensureQueue(queueName: string): Promise<void> {
  if (this.createdQueues.has(queueName)) return;
  await this.boss.createQueue(queueName);
  this.createdQueues.add(queueName);
}
```

This is called internally by `enqueue()`, `enqueueBatch()`, and `registerWorker()` — callers never need to create queues manually.

## Adapter Pattern

Modules that want a domain-specific queue API can wrap `IQueueService` in a thin adapter:

```typescript
@Injectable()
export class PgBossNotificationQueue implements NotificationQueue {
  constructor(@Inject(QUEUE_SERVICE) private readonly queue: IQueueService) {}

  async enqueue(job: NotificationQueueJob): Promise<void> {
    await this.queue.enqueue('notification-delivery', job, DEFAULT_JOB_OPTIONS);
  }
}
```

<Info>
This allows modules to define their own enqueue/worker signatures without coupling to pg-boss internals directly.
</Info>

## Patterns and Conventions

### Dependency Injection Pattern

- Service is registered with `QUEUE_SERVICE` token for injection
- Global module makes queue service available everywhere
- Interface-based design allows for provider swapping

### Job Handler Pattern

Workers register handlers that receive strongly-typed job data:

```typescript
await queueService.registerWorker<EmailData>(
  'email-queue',
  async (job: QueueJob<EmailData>) => {
    // Process a single email job — pg-boss handles per-job completion/failure
  },
  { batchSize: 10 }, // batchSize maps to localConcurrency for parallel processing
);
```

### pg-boss v12+ Compatibility

<Warning>
pg-boss v12+ asserts that `priority` must be an integer and rejects `undefined`. When calling `enqueue()`, only pass `QueueJobOptions` fields that have defined values. The `PgBossQueueService` implementation guards each option with an `!== undefined` check before forwarding to pg-boss.
</Warning>

### Error Handling

- Jobs automatically retry based on configuration
- Failed jobs are logged with appropriate error context
- Service lifecycle ensures graceful shutdown
