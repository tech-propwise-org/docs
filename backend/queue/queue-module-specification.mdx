## Overview

The Queue module provides a centralized job queue system for PropWise CRM using PostgreSQL-based job processing via the pg-boss library. This module enables asynchronous processing of background tasks such as notifications, data processing, and other long-running operations that should be decoupled from HTTP request/response cycles.

The module implements a queue service abstraction that allows other modules to enqueue jobs for background processing while providing features like job retries, priorities, batching, and worker management.

## Interfaces

### QueueJob

Represents a job in the queue system.

```typescript
interface QueueJob<T> {
  id: string; // Unique job identifier
  data: T; // Job payload data
  retryCount: number; // Current retry attempt count
  retryLimit: number; // Maximum retry attempts allowed
}
```

### QueueJobOptions

Configuration options for job enqueueing.

```typescript
interface QueueJobOptions {
  retryLimit?: number; // Max retry attempts (default: 3)
  retryDelay?: number; // Delay between retries in seconds
  retryBackoff?: boolean; // Use exponential backoff for retries
  expireInSeconds?: number; // Job expiration time
  singletonKey?: string; // Ensure only one job with this key exists
  priority?: number; // Job priority (higher = more priority)
}
```

<Warning>
pg-boss v12+ asserts that `priority` must be an integer and rejects `undefined`. Only pass `QueueJobOptions` fields that have defined values to avoid explicit `priority: undefined` values.
</Warning>

### WorkerOptions

Configuration for job workers.

```typescript
interface WorkerOptions {
  batchSize?: number; // Number of jobs to process in batch
  pollingIntervalSeconds?: number; // How often to poll for new jobs
}
```

### IQueueService interface

Core queue service contract with the following methods:

| Method | Description |
| --- | --- |
| `enqueue<T>(queueName, data, options?)` | Add single job to queue |
| `enqueueBatch<T>(queueName, data[], options?)` | Add multiple jobs to queue |
| `registerWorker<T>(queueName, handler, options?)` | Register job processor |
| `start()` | Start queue processing |
| `stop()` | Stop queue processing |

## Service Implementation

### PgBossQueueService

The main service implementation using pg-boss:

```typescript
@Injectable()
export class PgBossQueueService implements IQueueService, OnModuleInit, OnModuleDestroy
```

#### Key features

<AccordionGroup>
<Accordion title="Automatic queue creation tracking">
Uses `createdQueues` Set to track which queues have been created in the current process lifetime.
</Accordion>

<Accordion title="Database connection configuration">
Configured via `ConfigService` reading `DATABASE_URL`, `NODE_ENV`, and `DATABASE_SSL` through NestJS config.
</Accordion>

<Accordion title="Lifecycle hooks">
Implements proper startup/shutdown through `OnModuleInit` and `OnModuleDestroy`.
</Accordion>

<Accordion title="Worker registration">
Provides type-safe job handlers with configurable options.
</Accordion>

<Accordion title="Parallel per-job processing">
When `batchSize` is specified in `WorkerOptions`, it is mapped to pg-boss's `localConcurrency` option with `batchSize: 1` at the pg-boss level. This spawns N independent workers that each process one job at a time, avoiding the batch auto-completion pitfall.
</Accordion>
</AccordionGroup>

<Tip>
pg-boss auto-completes on resolve or auto-fails on reject per handler invocation, avoiding the batch auto-completion pitfall where pg-boss `batchSize > 1` would complete/fail the entire batch as a unit.
</Tip>

## Business Rules

### Job processing rules

<Steps>
<Step title="Retry logic">
Jobs can be configured with retry limits and backoff strategies for failed processing attempts.
</Step>

<Step title="Job uniqueness">
Singleton keys prevent duplicate job enqueueing for operations that should only run once.
</Step>

<Step title="Priority processing">
Higher priority jobs are processed before lower priority ones in the queue.
</Step>

<Step title="Expiration">
Jobs can expire if not processed within the specified time limit.
</Step>

<Step title="Batch processing">
Workers can process multiple jobs simultaneously based on configuration.
</Step>
</Steps>

### Queue management

1. **Queue creation**: Queues are created automatically when first used
2. **Worker registration**: Each queue can have multiple workers with different configurations
3. **Connection management**: Single pg-boss instance per service lifetime

### Global registration

The `QueueModule` is decorated with `@Global()` so all application modules can inject the queue service without explicitly importing `QueueModule`.

<CodeGroup>
```typescript Module Registration
@Global()
@Module({
  providers: [
    PgBossQueueService,
    {
      provide: QUEUE_SERVICE,
      useExisting: PgBossQueueService,
    },
  ],
  exports: [QUEUE_SERVICE],
})
export class QueueModule {}
```

```typescript Consumer Injection
// Any module can inject without importing QueueModule
constructor(@Inject(QUEUE_SERVICE) private readonly queueService: IQueueService) {}
```
</CodeGroup>

### Auto-creation pattern (ensureQueue)

`PgBossQueueService` lazily creates queues on first use via a private `ensureQueue()` method. A `createdQueues` Set tracks which queues have been created in the current process lifetime so `createQueue()` is called at most once per queue name:

```typescript
private async ensureQueue(queueName: string): Promise<void> {
  if (this.createdQueues.has(queueName)) return;
  await this.boss.createQueue(queueName);
  this.createdQueues.add(queueName);
}
```

<Info>
This method is called internally by `enqueue()`, `enqueueBatch()`, and `registerWorker()` — callers never need to create queues manually.
</Info>

## Adapter Pattern

Modules that want a domain-specific queue API can wrap `IQueueService` in a thin adapter. For example, the Notification module uses `PgBossNotificationQueue`:

```typescript
@Injectable()
export class PgBossNotificationQueue implements NotificationQueue {
  constructor(@Inject(QUEUE_SERVICE) private readonly queue: IQueueService) {}

  async enqueue(job: NotificationQueueJob): Promise<void> {
    await this.queue.enqueue('notification-delivery', job, DEFAULT_JOB_OPTIONS);
  }
}
```

<Check>
This allows modules to define their own enqueue/worker signatures without coupling to pg-boss internals directly.
</Check>

## Patterns and Conventions

### Dependency injection pattern

- Service is registered with `QUEUE_SERVICE` token for injection
- Global module makes queue service available everywhere  
- Interface-based design allows for provider swapping

### Job handler pattern

Workers register handlers that receive strongly-typed job data:

```typescript
await queueService.registerWorker<EmailData>(
  'email-queue',
  async (job: QueueJob<EmailData>) => {
    // Process a single email job — pg-boss handles per-job completion/failure
  },
  { batchSize: 10 }, // batchSize maps to localConcurrency for parallel processing
);
```

### Environment configuration

<AccordionGroup>
<Accordion title="DATABASE_URL">
PostgreSQL connection string (required). pg-boss requires DDL privileges to create its schema/tables, so it uses `DATABASE_URL` (superuser) instead of `RUNTIME_DATABASE_URL` (RLS-enabled user).
</Accordion>

<Accordion title="DATABASE_SSL">
Set to `'false'` to disable SSL. Default: SSL enabled with `rejectUnauthorized` set to `true` in production (`NODE_ENV=production`) and `false` otherwise.
</Accordion>

<Accordion title="Queue-specific settings">
Handled through job options rather than global configuration for flexibility.
</Accordion>
</AccordionGroup>
