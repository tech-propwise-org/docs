## Overview

The Queue module provides a centralized job queue system for PropWise CRM using PostgreSQL-based job processing via the pg-boss library. This module enables asynchronous processing of background tasks such as notifications, data processing, and other long-running operations that should be decoupled from HTTP request/response cycles.

<Info>
**Last Updated:** 2026-03-08  
**Status:** Active
</Info>

The module implements a queue service abstraction that allows other modules to enqueue jobs for background processing while providing features like job retries, priorities, batching, and worker management.

## Architecture

### Design Decisions

The Queue module is built on the following architectural principles:

<CardGroup cols={2}>
  <Card title="PostgreSQL-based queuing" icon="database">
    Uses pg-boss library for reliability and ACID compliance
  </Card>
  <Card title="Interface-driven design" icon="layer-group">
    Implements `IQueueService` interface for potential future queue provider swapping
  </Card>
  <Card title="Global module" icon="globe">
    Available throughout the application without explicit imports
  </Card>
  <Card title="Worker abstraction" icon="cogs">
    Provides flexible worker registration with configurable options
  </Card>

<Note>
The module uses pg-boss as the underlying queue implementation because it leverages existing PostgreSQL infrastructure, provides ACID guarantees for job persistence, supports advanced features like job scheduling and retries, and requires no additional infrastructure dependencies.
</Note>

### Lifecycle Management

The service properly handles startup and shutdown of queue connections through NestJS lifecycle hooks:

- **OnModuleInit**: Starts the pg-boss instance and begins job processing
- **OnModuleDestroy**: Gracefully shuts down workers and closes database connections

## Core Interfaces

### QueueJob Interface

Represents a job in the queue system with strongly-typed data payload:

```typescript
interface QueueJob<T> {
  id: string; // Unique job identifier
  data: T; // Job payload data
  retryCount: number; // Current retry attempt count
  retryLimit: number; // Maximum retry attempts allowed
}
```

### QueueJobOptions Interface

Configuration options for job enqueueing with comprehensive control over job behavior:

```typescript
interface QueueJobOptions {
  retryLimit?: number; // Max retry attempts (default: 3)
  retryDelay?: number; // Delay between retries in seconds
  retryBackoff?: boolean; // Use exponential backoff for retries
  expireInSeconds?: number; // Job expiration time
  singletonKey?: string; // Ensure only one job with this key exists
  priority?: number; // Job priority (higher = more priority)
}
```

### WorkerOptions Interface

Configuration for job workers enabling parallel processing:

```typescript
interface WorkerOptions {
  batchSize?: number; // Number of jobs to process in batch
  pollingIntervalSeconds?: number; // How often to poll for new jobs
}
```

<Tip>
When `batchSize` is specified, it maps to pg-boss's `localConcurrency` option, spawning N independent workers that each process one job at a time for true parallel processing.
</Tip>

### IQueueService Interface

Core queue service contract providing the main API:

| Method | Description | Parameters |
|--------|-------------|------------|
| `enqueue<T>(queueName, data, options?)` | Add single job to queue | Queue name, job data, optional configuration |
| `enqueueBatch<T>(queueName, data[], options?)` | Add multiple jobs to queue | Queue name, array of job data, optional configuration |
| `registerWorker<T>(queueName, handler, options?)` | Register job processor | Queue name, handler function, optional worker config |
| `start()` | Start queue processing | None |
| `stop()` | Stop queue processing | None |

## Service Implementation

### PgBossQueueService

The main service implementation using pg-boss with full lifecycle management:

```typescript
@Injectable()
export class PgBossQueueService implements IQueueService, OnModuleInit, OnModuleDestroy
```

<AccordionGroup>
  <Accordion title="Key Features">
    - **Automatic queue creation**: Tracks created queues via `createdQueues` Set
    - **Database connection**: Configured via `ConfigService` reading environment variables
    - **Lifecycle hooks**: Proper startup/shutdown handling
    - **Worker registration**: Type-safe job handlers with configurable options
    - **Parallel processing**: Maps `batchSize` to `localConcurrency` for true parallelism
  </Accordion>

  <Accordion title="Auto-Creation Pattern">
    The service implements lazy queue creation through a private `ensureQueue()` method:

    ```typescript
    private async ensureQueue(queueName: string): Promise<void> {
      if (this.createdQueues.has(queueName)) return;
      await this.boss.createQueue(queueName);
      this.createdQueues.add(queueName);
    }
    ```

    This ensures queues are created automatically on first use and only once per process lifetime.
  </Accordion>
</AccordionGroup>

## Business Rules

### Job Processing Rules

<Steps>
  <Step title="Retry Logic">
    Jobs can be configured with retry limits and exponential backoff strategies for handling transient failures.
  </Step>

  <Step title="Job Uniqueness">
    Singleton keys prevent duplicate job enqueueing, ensuring only one instance of a specific job exists at a time.
  </Step>

  <Step title="Priority Processing">
    Higher priority jobs are processed before lower priority ones, enabling critical task prioritization.
  </Step>

  <Step title="Expiration Handling">
    Jobs can expire if not processed within a specified time, preventing stale job accumulation.
  </Step>

  <Step title="Batch Processing">
    Workers can process multiple jobs simultaneously through configurable parallelism.
  </Step>
</Steps>

### Queue Management Rules

1. **Automatic Creation**: Queues are created automatically when first used
2. **Multiple Workers**: Each queue can have multiple workers with different configurations
3. **Single Instance**: One pg-boss instance per service lifetime for connection efficiency

## Implementation Patterns

### Adapter Pattern

Modules can implement domain-specific queue APIs by wrapping the shared `IQueueService`:

```typescript
@Injectable()
export class PgBossNotificationQueue implements NotificationQueue {
  constructor(@Inject(QUEUE_SERVICE) private readonly queue: IQueueService) {}

  async enqueue(job: NotificationQueueJob): Promise<void> {
    await this.queue.enqueue('notification-delivery', job, DEFAULT_JOB_OPTIONS);
  }
}
```

<Check>
This pattern allows modules to define their own enqueue/worker signatures without coupling to pg-boss internals directly.
</Check>

### Job Handler Pattern

Workers register strongly-typed handlers for processing jobs:

```typescript
await queueService.registerWorker<EmailData>(
  'email-queue',
  async (job: QueueJob<EmailData>) => {
    // Process a single email job
    // pg-boss handles per-job completion/failure automatically
  },
  { batchSize: 10 }, // Maps to localConcurrency for parallel processing
);
```

### Error Handling Pattern

The service implements comprehensive error handling:

- **Automatic Retries**: Jobs retry based on configuration with exponential backoff
- **Error Logging**: Failed jobs are logged with appropriate error context
- **Graceful Shutdown**: Service lifecycle ensures proper cleanup on application shutdown

## Environment Configuration

<Tabs>
  <Tab title="Required Variables">
    ```bash

    ```
  </Tab>

  <Tab title="Optional Variables">
    ```bash
    # Disable SSL for local development
    DATABASE_SSL=false
    
    # Environment affects SSL validation
    NODE_ENV=development|production
    ```
  </Tab>
</Tabs>

<Note>
pg-boss requires DDL privileges to create its schema and tables, so it uses `DATABASE_URL` (superuser connection) instead of `RUNTIME_DATABASE_URL` (RLS-enabled user connection).
</Note>

### SSL Configuration

- **Default**: SSL enabled with automatic certificate validation
- **Production**: `rejectUnauthorized` set to `true` for security
- **Development**: `rejectUnauthorized` set to `false` for flexibility
- **Override**: Set `DATABASE_SSL=false` to disable SSL entirely

## pg-boss Compatibility

<Warning>
pg-boss v12+ requires that `priority` must be an integer and rejects `undefined` values. The service implementation guards against this by checking `!== undefined` before forwarding options to pg-boss.
</Warning>

Queue-specific settings are handled through job options rather than global configuration, providing fine-grained control over individual job behavior while maintaining a simple global service interface.
