## Overview

The Queue module provides a centralized job queue system for PropWise CRM using PostgreSQL-based job processing via the pg-boss library. This module enables asynchronous processing of background tasks such as notifications, data processing, and other long-running operations that should be decoupled from HTTP request/response cycles.

The module implements a queue service abstraction that allows other modules to enqueue jobs for background processing while providing features like job retries, priorities, batching, and worker management.

### Queue Provider Selection

The module uses pg-boss as the underlying queue implementation because:

- Leverages existing PostgreSQL infrastructure
- Provides ACID guarantees for job persistence
- Supports advanced features like job scheduling, retries, and priorities
- No additional infrastructure dependencies

### QueueJobOptions

```typescript
interface QueueJobOptions {

```

| Option | Type | Default | Description |
|--------|------|---------|-------------|

| `retryDelay` | `number` | - | Delay between retries in seconds |
| `retryBackoff` | `boolean` | `false` | Use exponential backoff for retries |
| `expireInSeconds` | `number` | - | Job expiration time |
| `singletonKey` | `string` | - | Ensure only one job with this key exists |
| `priority` | `number` | - | Job priority (higher = more priority) |

### QueuePolicy

Controls how `singletonKey` deduplication is enforced at the database level via pg-boss policy-specific unique indexes.

```typescript
type QueuePolicy =
  | 'standard' // No dedup â€” singletonKey stored but no unique constraint
  | 'short' // â‰¤ 1 job per singletonKey in `created` state
  | 'singleton' // â‰¤ 1 job per singletonKey in `active` state
  | 'stately' // â‰¤ 1 job per singletonKey in created | retry | active
  | 'exclusive' // Same uniqueness as stately, different fetch semantics
  | 'key_strict_fifo';
```

<Warning>
**Critical:** The default `'standard'` policy applies **no unique index** on `singleton_key`. Any queue that uses `singletonKey` for deduplication **must** be created with `policy: 'stately'` (or `'exclusive'`), otherwise `ON CONFLICT DO NOTHING` never fires and every insert creates a new row regardless of how many pending jobs already exist for the same key.
</Warning>

<AccordionGroup>
  <Accordion title="Policy Details">
    | Policy | Deduplication Scope | Use Case |
    |--------|---------------------|----------|
    | `standard` | None (key stored only) | Default - no dedup enforcement |
    | `short` | `created` state only | Prevent duplicate enqueues before processing starts |
    | `singleton` | `active` state only | Ensure only one job running at a time |
    | `stately` | `created` \| `retry` \| `active` | Most common - prevent duplicates across all pending states |
    | `exclusive` | Same as `stately` | Different fetch semantics for advanced use cases |
    | `key_strict_fifo` | FIFO ordering per key | Strict ordering guarantee |
  </Accordion>
</AccordionGroup>

### WorkerOptions

Configuration for single-job workers.

```typescript
interface WorkerOptions {
  concurrency?: number; // Independent worker invocations (default: 1)
  pollingIntervalSeconds?: number; // How often to poll for new jobs
  policy?: QueuePolicy; // Queue policy â€” REQUIRED for singletonKey dedup on the worker's queue
}
```

| Option | Type | Default | Description |
|--------|------|---------|-------------|
| `concurrency` | `number` | `1` | Independent worker invocations |
| `pollingIntervalSeconds` | `number` | - | How often to poll for new jobs |
| `policy` | `QueuePolicy` | `'standard'` | Queue policy â€” REQUIRED for singletonKey dedup |

### BatchWorkerOptions

Configuration for batch workers.

```typescript
interface BatchWorkerOptions {
  batchSize?: number; // Jobs per handler invocation (default: 1)
  pollingIntervalSeconds?: number; // How often to poll for new jobs
  policy?: QueuePolicy; // Queue policy â€” REQUIRED for singletonKey dedup
}
```

| Option | Type | Default | Description |
|--------|------|---------|-------------|
| `batchSize` | `number` | `1` | Jobs per handler invocation |
| `pollingIntervalSeconds` | `number` | - | How often to poll for new jobs |
| `policy` | `QueuePolicy` | `'standard'` | Queue policy â€” REQUIRED for singletonKey dedup |

<Info>
`policy` is enforced by `ensureQueue` inside **both** `registerWorker` and `registerBatchWorker`. Use `'stately'` whenever jobs are submitted with `singletonKey` and you need "at most one pending/active job per key". `ensureQueue` also directly updates `pgboss.queue.policy` when a policy is supplied, so a queue first touched without a policy in the same process still converges when the policy-aware worker registers.
</Info>

<Note>
pg-boss caches each queue's policy per process for `queueCacheIntervalSeconds` (default 60s), so an existing queue's policy change is eventually-consistent within that window after a deploy.
</Note>

<Tip>
The single-job `messaging-outbox-poll` queue is the canonical `registerWorker` + `policy: 'stately'` consumer: its self-enqueuing poll trigger uses `singletonKey: 'outbox-poll'`, which only dedups under `stately`. Under the previous default `standard` policy the trigger accumulated across restarts and fired continuously (see `MessageSenderWorker` in `Docs/MESSAGING_MODULE_SPECIFICATION.md`).
</Tip>

### IQueueService Interface

Core queue service contract with the following methods:

| Method | Description |
|--------|-------------|
| `enqueue<T>(queueName, data, options?)` | Add single job to queue |
| `enqueueBatch<T>(queueName, data[], options?)` | Add multiple jobs to queue |
| `registerWorker<T>(queueName, handler, options?)` | Register job processor (accepts `policy`) |
| `schedule(queueName, cron, data?)` | Register recurring cron (pg-boss `schedule`) |
| `unschedule(queueName, key?)` | Remove recurring schedule (pg-boss `unschedule`) |
| `purgeQueuedJobs(queueName)` | Delete pending jobs (`created` + `retry`); leaves `active` (pg-boss `deleteQueuedJobs`) |
| `checkHealth(timeoutMs?)` | Active liveness probe (see below) |
| `start()` | Start queue processing |
| `stop()` | Stop queue processing |

#### checkHealth() â€” Active Liveness Probe

Returns `Promise<QueueHealthResult>` (`{ healthy, reason?, latencyMs? }`) and never throws. Consumed by `PgBossHealthIndicator` for the `/health/ready` probe.

<Warning>
The `isStarted` boolean is **not** a sufficient health signal: it is set once after `boss.start()` resolves and only flips back on `stop()`, so a backend that loses its connection or whose supervisor loop stalls after startup keeps `isStarted === true` and would report healthy forever.
</Warning>

`checkHealth()` instead round-trips `SELECT 1` through pg-boss's **own** connection (`boss.getDb().executeSql(...)`), bounded by `QUEUE_HEALTH_PROBE_TIMEOUT_MS` (`queue.constants.ts`, default 2000ms):

<Steps>
  <Step title="Check if started">
    Not started â†’ `{ healthy: false, reason: 'boss.start() has not completed' }` (no query issued)
  </Step>
  <Step title="Execute health query">
    Query rejects â†’ `{ healthy: false, reason: <driver error> }`
  </Step>
  <Step title="Check timeout">
    Query exceeds the timeout â†’ `{ healthy: false, reason: 'queue health probe timed out after <n>ms' }`
  </Step>
  <Step title="Return success">
    Otherwise â†’ `{ healthy: true, latencyMs }`
  </Step>
</Steps>

<Info>
Because the probe uses pg-boss's connection (superuser `DATABASE_URL` pool), it covers a gap the readiness endpoint's ORM-based `database` ping (`RUNTIME_DATABASE_URL` pool) does not â€” the two checks hit different pools. See `Docs/HEALTH_MODULE_SPECIFICATION.md` â†’ PgBossHealthIndicator.
</Info>

#### Database Connection

Database connection configured via `ConfigService` from centralized `app.database.*` config (`app.database.url`, `app.database.ssl`). TLS options are resolved by the shared `resolveDatabaseSsl()` helper (`src/common/utils/database-ssl.util.ts`), which validates the server certificate against Node's trust store for managed/Neon databases â€” applied to both the pg-boss connection and the `rawPool` used for queue-policy enforcement.

<Note>
`src/main.ts` must call `app.enableShutdownHooks()` so SIGTERM/SIGINT trigger `onModuleDestroy()` and allow pg-boss to stop gracefully.
</Note>

#### Parallel Per-Job Processing

When `batchSize` is specified in `WorkerOptions`, it is mapped to pg-boss's `localConcurrency` option (with `batchSize: 1` at the pg-boss level). This spawns N independent workers that each process one job at a time.

<Tip>
pg-boss auto-completes on resolve or auto-fails on reject per handler invocation, avoiding the batch auto-completion pitfall where pg-boss `batchSize > 1` would complete/fail the entire batch as a unit.
</Tip>

#### Test-Only Methods

**`__test__offWork(queueName)`** (throws unless `NODE_ENV=test`):

Wraps pg-boss `offWork()`, detaching all workers subscribed to that queue so jobs remain queued without being fetched.

<Info>
Used by distribution E2E tests together with `DistributionJobHandler.__test__resumeDistributionWorker()` to build a backlog and verify priority ordering before restoring the production worker (`batchSize: 10`).
</Info>

## Use Cases

### Knowledge Base Indexing Queue

The `knowledge-base-indexing` queue (`KNOWLEDGE_BASE_INDEXING_QUEUE` constant) handles asynchronous crawling and embedding of PAGE (website) knowledge base sources.

<Tabs>
  <Tab title="Job Configuration">
    Jobs are enqueued with:
    - `singletonKey: 'kb-index:<id>'`
    - `retryLimit: 0` (deliberate â€” no retry loops on crash)
    
    The worker (`KnowledgeBaseIndexingWorker`) is registered with:
    - `policy: 'stately'`
    - `concurrency: 2`
    
    A re-trigger while crawling is pending/active is dropped at the DB level rather than double-crawling.
  </Tab>
  <Tab title="Job Payload">
    ```typescript
    {
      knowledgeBaseId: string,
      orgId: string
    }
    ```
  </Tab>
</Tabs>

#### Processing Workflow

The worker delegates to `KnowledgeBaseService.runPageIngestion()`:

<Steps>
  <Step title="Claim source">
    Claims the source as INDEXING (short txn)
  </Step>
  <Step title="Scrape pages">
    Maps the host + scrapes each page (no txn, isolated per-page failures)
  </Step>
  <Step title="Persist pages">
    Persists `KnowledgeBasePage` rows (short txn)
  </Step>
  <Step title="Generate embeddings">
    Embeds outside any txn (slow OpenAI call)
  </Step>
  <Step title="Persist final state">
    Persists chunks + final status (short txn)
  </Step>
</Steps>

<Info>
A 10-minute recovery cron (`sweepStaleIngestion`) flips sources stuck INDEXING/PENDING beyond their stale thresholds to FAILED so users can reindex.
</Info>

## Business Rules

### Job Processing Rules

<AccordionGroup>
  <Accordion title="1. Retry Logic">
    Jobs can be configured with retry limits and backoff strategies
  </Accordion>
  <Accordion title="2. Job Uniqueness">
    `singletonKey` deduplication is only enforced when the queue is created with `policy: 'stately'` (or `'exclusive'`). With the default `'standard'` policy the key is stored but no DB constraint prevents duplicate inserts.
  </Accordion>
  <Accordion title="3. Priority Processing">
    Higher priority jobs are processed first
  </Accordion>
  <Accordion title="4. Expiration">
    Jobs can expire if not processed within specified time
  </Accordion>
  <Accordion title="5. Batch Processing">
    Workers can process multiple jobs simultaneously
  </Accordion>
  <Accordion title="6. Current Authorization for User-Initiated Jobs">
    Queue payloads should carry stable identifiers, not trusted authorization snapshots. Workers that perform user-initiated writes must reload current organization membership and RBAC permissions before execution, then fail the domain job if the actor was removed, deactivated, or lost the permission needed for that queued operation. The `import-job` worker uses this pattern for lead/contact imports.
  </Accordion>
</AccordionGroup>

### Queue Management

<Steps>
  <Step title="Queue Creation">

  <Step title="Worker Registration">

  </Step>

## Example Usage

### Enqueueing a Job

```typescript
await queueService.enqueue('email-notification', {
  userId: '123',
  templateId: 'welcome-email'
}, {
  retryLimit: 3,
  priority: 10,
  singletonKey: 'welcome-email-123'
});
```

### Registering a Worker

```typescript
await queueService.registerWorker(
  'email-notification',
  async (job) => {
    const { userId, templateId } = job.data;
    await emailService.send(userId, templateId);
  },
  {
    concurrency: 5,
    policy: 'stately' // Required for singletonKey dedup
  }
);
```

### Scheduling a Recurring Job

```typescript
await queueService.schedule(
  'daily-cleanup',
  '0 0 * * *', // Every day at midnight
  { action: 'cleanup' }
);
```

### Batch Processing

```typescript
await queueService.registerWorker(
  'bulk-import',
  async (jobs) => {
    for (const job of jobs) {
      await processImport(job.data);
    }
  },
  {
    batchSize: 10,
    policy: 'stately'
  }
);
```
