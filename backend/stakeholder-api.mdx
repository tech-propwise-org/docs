## Stakeholder sync feature (v2 — Full bidirectional)

### Overview

Leads and their active deals share **identical stakeholder state**. Any change to either side's stakeholders is automatically propagated to the linked entity via `syncToLinkedEntity`. This replaces the previous observer-only sync model.

### Design principles

1. **Full bidirectional sync** — Lead ↔ active Deal stakeholder state is always mirrored (same users, same commission, same primary)
2. **1:1 active deal per lead** — A lead can have at most one active deal (not archived, not closed-lost). A new deal can only be created when no active deal exists
3. **Last write wins** — Whichever entity (lead or deal) is changed most recently becomes the current truth for both
4. **100% commission floor** — Once a lead-deal link exists, both sides must maintain 100% total commission. A lead without an active deal can have 0% (not yet configured)
5. **Inactive deals excluded** — Archived or closed-lost deals are excluded from sync — the lead is effectively "unlinked"
6. **Deal creation gate** — A deal can only be created from a lead that is in the **QUALIFIED** stage. Leads in any other stage are not available for deal creation
7. **DONE stage auto-transitions** (v3.5) — When a deal connected to a lead reaches a terminal stage (CLOSED_WON or CLOSED_LOST), `DealService.update()` emits a `DEAL_TERMINAL_REACHED` event. `LeadService` listens with `@OnEvent` and automatically moves the originating lead to the `DONE` stage (unless the lead is already terminal or archived). Conversely, if the deal is reopened (moved from a terminal to a non-terminal stage), a `DEAL_REOPENED` event is emitted and the lead is automatically moved back to the `QUALIFIED` stage. Users cannot manually move leads to or from the `DONE` stage.

### How sync works

When `PUT /leads/:id/stakeholders` or `PUT /deals/:id/stakeholders` is called:

1. The sync endpoint applies all add/update/remove changes to the target entity
2. After changes are committed, `syncToLinkedEntity` is called automatically
3. It finds the linked entity (lead's active deal, or deal's originating lead)
4. It maps the source stakeholders to the target's desired state (keyed by `userId:teamId` — no row IDs needed)
5. It calls `syncStakeholdersInTransaction` on the target with `skipSync: true` (recursion guard)

```
Lead stakeholders changed via PUT
  → Apply changes to lead
  → syncToLinkedEntity(LEAD, leadId)
    → Find active deal for this lead
    → Call syncStakeholdersInTransaction(DEAL, dealId, ..., { skipSync: true })
```

### Deal creation with stakeholders

When creating a deal from a lead, the frontend pre-fills the stakeholder list from the lead and lets the user modify it. The backend applies stakeholders via one of three branches (priority order):

**Branch 1 — Explicit stakeholders provided (`dto.stakeholders[]` non-empty):**

1. Validates that commission totals 100% before creating
2. Creates all deal stakeholder rows via `createStakeholderBatchInTransaction()`
3. After all rows are persisted, automatically calls `syncToLinkedEntity(DEAL, dealId)`, which mirrors the deal's final stakeholder state to the originating lead — same users, same commissions, same primary flag. This ensures the lead reflects the user's deliberate modifications from the create modal

**Branch 2 — No explicit stakeholders, originating lead exists:**

1. Copies all lead stakeholder rows to the deal via `inheritFromLeadToDealInTransaction()`
2. Validates that the inherited total equals 100%
3. After flush, automatically calls `syncToLinkedEntity(DEAL, dealId)`. Since the deal's stakeholders are an exact copy of the lead's, this is a zero-diff no-op (no mutations, no notifications emitted)

**Branch 3 — No stakeholders, no lead (walk-in deal):**

1. Creates a single primary stakeholder (creator, 100% commission)
2. No sync — there is no originating lead

**`skipSync` opt-out:** Both `createStakeholderBatchInTransaction()` and `inheritFromLeadToDealInTransaction()` accept `options?: { skipSync?: boolean }`. Pass `skipSync: true` to suppress sync (e.g. seeding scripts, migrations). Default is `false` — sync always runs.

### Skip conditions (no sync)

| Condition | When Source Is Lead | When Source Is Deal |
| --- | --- | --- |
| No linked entity | No active deal exists | No originating lead |
| Linked entity deleted | Deal is soft-deleted | Lead is soft-deleted |
| Linked entity archived | Deal is archived | Lead is archived |
| Deal is closed-lost | Deal has `isClosed=true AND isWon=false` | N/A |

### Commission rules with sync

| Scenario | Commission Rule |
| --- | --- |
| Deal (always) | Must be exactly 100% |
| Lead with active deal | Must be exactly 100% (synced with deal) |
| Lead without active deal | 0% or 100% (0% = not yet configured) |
| Deal transitions to closed-lost or archived | Lead becomes "unlinked" — can go back to 0% |

## Team commission budget

A team's `commissionPercentage` on an entity stakeholder acts as a **live budget** that can be sub-allocated to individual team members. This enables team leaders to manage how commission is distributed within their team without needing org-level permissions.

### Design principles

1. **Team as budget holder** — The team stakeholder's `commissionPercentage` is the total pool available for team member allocation
2. **Sub-allocation** — Team managers assign commission from the team budget to individual members. The team's percentage decreases as members receive allocations
3. **Unallocated budget at close** — If a deal closes with remaining team budget (team stakeholder has `commissionPercentage > 0`), that amount is paid to the **team** directly, not to any individual
4. **Team pool freeze** — When a team-only stakeholder (has `team`, no `user`) holds primary status, the entity is blocked from modification until a team member claims or is assigned to it

### Team budget lifecycle

```
1. Lead assigned to Team A (100% commission)
   └── Team A stakeholder: { team: "Team A", user: null, commission: 100%, isPrimary: true,
                              originalTeamCommission: 100 }

2. Team manager assigns Agent X (40% from team budget)
   ├── Team A stakeholder: { team: "Team A", user: null, commission: 60%, isPrimary: false,
   │                          originalTeamCommission: 100 }
   └── Agent X stakeholder: { user: "Agent X", team: "Team A", commission: 40%, isPrimary: true,
                               originalTeamCommission: 100 }

3. Team manager assigns Agent Y (30% from team budget)
   ├── Team A stakeholder: { commission: 30% }   ← remaining budget
   ├── Agent X stakeholder: { commission: 40%, originalTeamCommission: 100 }
   └── Agent Y stakeholder: { commission: 30%, originalTeamCommission: 100 }

4. Deal closes → Commission payments:
   ├── Agent X: 40% → paid to user
   ├── Agent Y: 30% → paid to user
   └── Team A:  30% → paid to TEAM (unallocated budget)
```

`originalTeamCommission` on each member's stakeholder records the team's total allocation (100% in this case), allowing the system to validate that the sum of team member commissions never exceeds the original team budget.

### Claim from team pool

When a team member claims a lead from the team pool, the team stakeholder is **preserved** as a budget placeholder (set to `isPrimary: false`, `commissionPercentage: 0`) rather than being deleted. This allows the team to retain its position on the entity for future budget allocation.

The claiming user's stakeholder is created with `originalTeamCommission` set to the team's commission at the time of claim, enabling team budget enforcement even after subsequent reassignments.

```
BEFORE claim:
└── Team A: { commission: 100%, isPrimary: true, originalTeamCommission: 100 }

AFTER claim by Agent X (same team):
├── Team A: { commission: 0%, isPrimary: false, originalTeamCommission: 100 }
└── Agent X: { commission: 100%, isPrimary: true, originalTeamCommission: 100 }
```

### Commission aggregation with dual-record model

With the dual-record model, a single user can hold **multiple** stakeholder rows on the same entity — one personal (`team_id=NULL`) and one per team they claimed from. Each row carries its own `commissionPercentage`.

**Aggregation Rule for Payment/Reporting:**

```sql
-- Total commission for a user on an entity (across all contexts)
SELECT user_id, SUM(commission_percentage) AS total_commission
FROM entity_stakeholder
WHERE entity_type = :entityType
  AND entity_id = :entityId
  AND user_id = :userId
  AND is_deleted = false
GROUP BY user_id;
```

<Warning>
  When implementing the payment module, always use the aggregation query (`SUM`
  by `user_id`) rather than assuming one row per user.
</Warning>

### Commission payment feedback system

Commission payments now support recipient feedback on terminal-state payments (PAID or CANCELLED):

**Feedback rules:**

- Only the payment recipient can submit feedback
- Feedback is write-once - cannot be overwritten once submitted
- `confirmed=true` means recipient acknowledges receipt/cancellation
- `confirmed=false` means recipient disputes it (note required)
- Feedback triggers notifications to managers via `CommissionFeedbackSubmittedEvent`

**Payment history:**

- All status transitions create immutable audit records in `CommissionPaymentHistory`
- Records include actor, timestamp, reason, and payment method snapshots
- History is accessible via `GET /commission-payments/:id/history`

**Notification integration:**

- `CommissionNotificationHandler` processes confirm/dispute actions
- Co-located with domain service for direct business logic access
- Auto-discovered via `@NotificationActionHandler` decorator

## Entity changes required

To support the stakeholder system features, the following entities require updates:

### Lead entity
- Add `originalTeamCommission` field for tracking team budget allocations
- Add indexes for stakeholder query performance
- Update state machine to handle team pool freeze conditions

### Deal entity  
- Add `originatingLeadId` field for sync service
- Add `originalTeamCommission` field for team budget tracking
- Update creation logic to inherit stakeholders from leads

### User entity
- Add team membership tracking for claim validation
- Update permission computation for team-based access

### Team entity
- Add commission budget tracking
- Add team pool management capabilities

### Deal APIs

Deal stakeholder endpoints mirror lead endpoints with `entityType: "deal"`. Key differences:

- Deal stakeholder create/remove uses **automatic lead sync** via `syncToLinkedEntity`
- Deals do **not** have a `/claim` endpoint — deals are created from leads (inheriting stakeholders) or assigned directly
- Deal `assignToTeam` blocks if _any_ stakeholders exist (unlike lead which only blocks the target team)

## Business rules

### Commission auto-adjustment formula

When removing a stakeholder and redistributing commission:

```
Formula: newPercentage = oldPercentage × (100 / currentTotalExcludingRemoved)

Example:
BEFORE removal:
- Agent A: 50%
- Agent B: 0% ← Removing
- Agent C: 20%
Total: 70%

AFTER (auto-adjusted to 100%):
- Agent A: 71.43% (50 × 100/70)
- Agent C: 28.57% (20 × 100/70)
Total: 100% ✓
```

### Team budget enforcement

- Team managers can only allocate commission up to `originalTeamCommission`
- Sum of individual team member commissions cannot exceed the original team budget
- Team budget decreases as members are assigned commission
- Remaining team budget at deal close is paid to the team directly

### Transfer validation rules

| Rule                             | Description                                                                               |
| -------------------------------- | ----------------------------------------------------------------------------------------- |
| `fromKeepsPercentage` ceiling    | Cannot exceed the primary stakeholder's current commission                                |
| Transfer amount ≥ 0              | 0% transfers allowed only to existing stakeholders (primary-only, no commission movement) |
| Requester must be primary        | Only the primary stakeholder (or member of primary team) can initiate                     |
| Transfer-back allowed            | Transferring back to a previous owner or team is permitted                                |
| Existing stakeholder update      | If target already has a stakeholder, it is updated (commission added) not duplicated      |
| Auto-cancel on removal/reduction | Pending transfers are cancelled when any stakeholder is removed or has commission reduced |

### Team pool freeze rules

| Rule               | Description                                                                                  |
| ------------------ | -------------------------------------------------------------------------------------------- |
| Modification block | Leads/deals cannot be modified when primary stakeholder is team-only (`team` set, no `user`) |
| Resolution         | Assign or claim the entity to a team member to unfreeze                                      |
| Exception          | Users with stakeholder access (`accessLevel: MODIFY` + `user` set) can still modify          |
| Exception          | Primary stakeholders (user-based, `isPrimary: true`) can still modify                        |

## Access control

### Permission-only access for stakeholder management

<Warning>
  Stakeholder mutation endpoints use **permission-only** access checks.
  Resource-level access (e.g., `accessLevel: MODIFY` via stakeholder access)
  does **not** grant permission to manage stakeholders.
</Warning>

```typescript
// Lead stakeholder endpoints
@CheckAccess({
  permissions: [OrgPermissionKey.CRM_MANAGE, TeamPermissionKey.TEAM_CRM_MANAGE],
})

// Deal stakeholder endpoints
@CheckAccess({
  permissions: [OrgPermissionKey.SALES_MANAGE, TeamPermissionKey.TEAM_SALES_MANAGE],
})
```

**Two-tier access:**

1. **Guard level (AccessGuard)** — Checks if user has `CRM_MANAGE` (org) OR `TEAM_CRM_MANAGE` (any team). Coarse gate.
2. **Controller level (business logic)** — For team managers, validates: team has a stakeholder on the entity, commission doesn't exceed budget, adjustments/removals target their own team, primary can only be set if their team holds primary, and assigned user is a team member.

## State transitions

| Scenario                       | Before                               | After                                                                  |
| ------------------------------ | ------------------------------------ | ---------------------------------------------------------------------- |
| Assign to team                 | No team stakeholder with `team_id=X` | Team (100%, primary, `originalTeamCommission: 100`)                    |
| Claim from lead pool           | No stakeholders                      | User (100%, primary)                                                   |
| Claim from team pool           | Team X (100%)                        | User (100%, primary, `team_id=X`), Team X (0%, non-primary, preserved) |
| Update team commission (admin) | Team-only row exists                 | Team commission updated; `originalTeamCommission` synced               |
| Unassign                       | User(s) with commission              | No stakeholders (back to pool)                                         |
| Transfer (no keep)             | A (100%)                             | B (100%)                                                               |
| Transfer (with keep)           | A (100%)                             | A (30%), B (70%)                                                       |
| Transfer 0% to existing        | A (50%), B (50%)                     | A (50%, non-primary), B (50%, primary)                                 |
| Transfer to existing user      | A (60%), B (40%)                     | A (keeps%), B (40% + transfer%, primary)                               |
| Transfer to existing team      | Team (20%), B (80%)                  | Team (20% + transfer%), B (updated)                                    |
| Deal terminal reached          | Lead (QUALIFIED), Deal (CLOSED_WON)  | Lead (DONE), Deal (CLOSED_WON)                                         |
| Deal reopened                  | Lead (DONE), Deal (IN_PROGRESS)      | Lead (QUALIFIED), Deal (IN_PROGRESS)                                   |

## Quick reference

### Validation rules by operation

| Operation             | Validation                                                                                                |
| --------------------- | --------------------------------------------------------------------------------------------------------- |
| Create (simple)       | `new + existing = 100` OR `all = 0`; at least one primary after creation                                  |
| CreateWithAdjustments | `new + adjusted = 100`; at least one primary; auto-cancels pending transfers if commission reduced        |
| Update                | `new total ≤ 100`; cannot remove last primary; auto-cancels pending transfers when commission reduced     |
| Remove                | Auto-redistributes proportionally; cannot remove primary while others exist; auto-cancels transfers       |
| Transfer              | `fromKeepsPercentage ≤ primary's commission`; 0% only to existing stakeholders; requester must be primary |
| Sync (PUT)            | `sum(all) = 100`; at least one primary; full replacement with bidirectional sync                         |
| Team Assign           | Same as create; `originalTeamCommission` set to 100                                                       |
| Team Claim            | User gets team's commission, team → 0%; `originalTeamCommission` recorded; team preserved                 |
| Unassign              | Removes all stakeholders; auto-cancels all pending transfers                                              |

### Team manager constraints

| Rule                | Description                                                                              |
| ------------------- | ---------------------------------------------------------------------------------------- |
| Budget ceiling      | Team budget = team remaining % + team members' combined %. Assignment cannot exceed this |
| Adjustments scope   | Can only adjust own team's stakeholder and team member commissions                       |
| Primary restriction | Cannot set primary if no team stakeholder currently holds primary                        |
| Entity visibility   | Must have a team-associated stakeholder on the entity                                    |
| Assignee membership | Can only assign users who are members of their own team                                  |
| Removal scope       | Can only remove stakeholders belonging to their own team                                 |
