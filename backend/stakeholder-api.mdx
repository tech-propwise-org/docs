This page documents the API endpoints and service functions for managing lead/deal/contact stakeholders, assignment, transfer, and commission handling. For table schemas, data model concepts, and invariants, see the [stakeholder data model](/backend/crm/stakeholder-model).

## Vocabulary

<Note>
**User-facing and AI-facing vocabulary is "Assignment" and "Assignee".** "Stakeholder" is the legacy internal term. It is retained as:

- the entity name (`EntityStakeholder`, `EntityStakeholderHistory`),
- the database table name (`entity_stakeholder`),
- the module folder (`src/modules/crm/entity-stakeholder/`),
- the internal service class names (`EntityStakeholderService`, `StakeholderSyncService`),
- the REST API DTO key on lead/deal/contact responses (`stakeholders` / `primaryStakeholder`),
- the `NotificationType` enum values (`LEAD_STAKEHOLDER_ADDED`, etc. — a rename would require a notification-type DB migration per `.cursor/rules/notification-type-migration.mdc`),
- the AI tool name (`getLeadStakeholders`).

Surfaces that face users or the AI assistant use the new vocabulary:

- **Frontend** modal titles, headings, toasts, notification copy, validation errors, audit log labels, and timeline labels render as "Assignee" / "Assignment" / "assigned to" / "unassigned".
- **Backend notification listeners** (`crm-event.listener.ts`) emit user-visible `title` and `message` fields using assignee vocabulary even though the underlying event names and `NotificationType` enum values keep the legacy "stakeholder" word internally.
- **Audit timeline** (`audit-timeline.resolver.ts`) maps the `entity_stakeholder` table name to the user-facing label "Assignment" via `USER_FACING_TABLE_LABELS`; the timeline event constants (`STAKEHOLDER_ASSIGNED` etc.) keep their legacy names internally but render as "Assigned" / "Unassigned" / "Assignment updated" to users.
- **AI assistant** system prompt + tool descriptions + schema glossary instruct the model to say "assignee" / "assignment" / "assigned to". The AI tool boundary (in `getLeadStakeholders.execute()` and the pre-hydrated `compactContext.lead.assignees` + `compactContext.lead.primaryAssignee`) projects the rows under `assignees` / `primaryAssignee` keys so the model never sees the legacy word in tool-call traces.

The terms in the rest of this document are kept as "stakeholder" because this document specifies the engineering subsystem, not the user-visible surface. Translate to "assignee" / "assignment" when referencing this functionality in user-facing copy.
</Note>

## Stakeholder sync feature (v2 — Full bidirectional)

### Overview

Leads and deals no longer share stakeholder state after deal creation. Source leads can prefill a new deal's stakeholder list, but the persisted deal stakeholders are deal-owned and later lead/deal stakeholder edits do not automatically propagate.

### Design principles

1. **Deal-owned stakeholders** — Deal stakeholders are managed independently from lead stakeholders after creation. Leads provide creation-time suggestions only
2. **1:1 active deal per lead** — A lead can be referenced by at most one active deal contact (deal not archived, not closed-lost). A new active deal participant can only use that lead when no active deal already references it
3. **No bidirectional stakeholder sync** — Lead and deal stakeholder changes do not mirror automatically. This prevents buyer-side and seller-side leads from receiving each other's deal stakeholders
4. **100% commission floor** — Deals must maintain 100% total commission. A lead without an active deal can have 0% (not yet configured)
5. **Inactive deals excluded** — Archived or closed-lost deals are excluded from active-lead conflict checks
6. **Deal creation gate** — A deal can be created from an active source lead/contact only when the requester can use that source at write level. `crm.manage` can use any active org source record; otherwise the requester needs `WRITE`/`FULL` stakeholder access directly or through a `team_crm.manage` team context. `crm.view` / `team_crm.view` are visibility-only and do not allow deal creation from that source
7. **Deal stakeholder invariant** — A deal must always have at least one stakeholder. Deal stakeholder sync rejects `stakeholders: []`; deals do not have an unassign-to-pool operation
8. **Transfer permission parity** — Lead and deal `canTransfer` response permissions follow the same actor model: the primary user stakeholder can initiate transfer, and a team manager can initiate transfer when their managed team holds the primary team-only row
9. **Transfer timeline visibility** — Lead and deal timelines include related transfer request audit events. Request, approval, rejection, and cancellation rows from `entity_transfer` are surfaced on the owning lead/deal timeline with the transfer reason or rejection reason when present
10. **DONE stage auto-transitions** (v3.5) — When a deal reaches a terminal stage (CLOSED_WON or CLOSED_LOST), every lead referenced by `deal_contact.originating_lead_id` moves to the `DONE` stage (unless already terminal or archived). Conversely, reopening the deal moves those participant-originating leads back to `QUALIFIED`
11. **Source-linked deal contacts are preserved** — Deal contact management may edit role, side, side-scoped primary status, and notes. It may add/remove ordinary non-source contacts, but it must not remove or replace a contact row that carries `deal_contact.originating_lead_id`; changing source leads requires a dedicated source-change flow because stage auto-transitions and active-deal conflict checks depend on that lineage

### How stakeholder updates work

When `PUT /leads/:id/stakeholders` or `PUT /deals/:id/stakeholders` is called:

1. The sync endpoint applies all add/update/remove changes to the target entity
2. It emits notifications based on the final transition:
   - `0 → N` lead stakeholders emits `LEAD_ASSIGNED` and suppresses per-row "stakeholder added" notifications
   - `N → 0` lead stakeholders emits `LEAD_UNASSIGNED`, cancels pending transfers, and suppresses per-row "stakeholder removed" notifications
   - `N → 0` is invalid for deals because deals must retain at least one stakeholder
   - Other diffs emit fine-grained stakeholder added/removed/commission-changed notifications
3. No linked lead/deal entity is automatically mutated

Deal contact edits follow the same independence model: contact metadata edits do not mutate source leads or deal stakeholders. Source-linked contact rows are intentionally protected from generic removal/replacement so lead lineage remains stable for downstream deal lifecycle behavior.

### Deal creation with stakeholders

When creating a deal from one or more participant leads, the frontend can pre-fill the stakeholder list from all selected leads and lets the user review the merged list. The backend persists only the final deal-owned stakeholder payload.

Before a participant source can be selected, the frontend calls deal-owned availability endpoints (`GET /deals/available-leads`, `GET /deals/available-contacts`). These endpoints are mutation-scoped and return only source records the requester can use for deal creation. `POST /deals` re-checks the same rule for submitted lead/contact IDs so callers cannot bypass the picker.

**Branch 1 — Explicit stakeholders provided (`dto.stakeholders[]` non-empty):**

1. Validates that commission totals 100% before creating
2. Creates all deal stakeholder rows via `createStakeholderBatchInTransaction()`
3. Does not mutate any source lead stakeholders

**Branch 2 — No explicit stakeholders:**

1. Creates a single primary stakeholder (creator, 100% commission)
2. No source lead stakeholders are copied server-side

**No bidirectional sync:** `syncToLinkedEntity()` is retained as an internal no-op guard for older call paths, but lead/deal stakeholder mutations are intentionally independent.

### Deal creation with a pending unit access request

When a deal is created with a hidden `InventoryUnit` managed by another agent (`unit.unitManager.id !== currentUser.id` and `isUnitNumberHidden !== false`), `DealService.create()` defers the unit link and issues a `Pending` `UnitAccessRequest` in the same transaction unless the submitted deal stakeholders already include that unit manager. Public units and already-included unit managers link directly with no request-owned observer row and no automatic commission transfer. See `Docs/REAL_ESTATE_MODULE_SPECIFICATION.md` §Unit Access — Pending Deal Link for the full flow.

Stakeholder-side rules:

- The deal's requested stakeholders are created exactly as in the normal path (explicit `dto.stakeholders[]` or creator-as-100%-primary). The pending-link path then adds one extra request-owned unit-manager observer row
- If `dto.stakeholders[]` already includes the unit manager, the deal links the inventory unit directly and keeps that stakeholder's submitted commission/access unchanged; the pending access commission-transfer mechanism is skipped
- The deal creator (current user) **must** end up as a primary user-stakeholder of the new deal — otherwise the access request has no requester to attribute and `DealService.create()` rejects the payload
- The proposed `unitManagerCommissionPercentage` must be `<=` the requesting primary stakeholder's commission. The check runs at deal-create time (so the deal never lands in a state that approval cannot service) **and** again at approval time on the **live** stakeholder row, in case stakeholders were re-synced between request and approval
- Request creation adds the unit manager as a request-owned `DEAL` stakeholder with `isPrimary=false`, `commissionPercentage=0`, and `accessLevel=READ`. `UnitAccessRequest.unitManagerStakeholder` points at this row

## Team commission budget

A team's `commissionPercentage` on an entity stakeholder acts as a **live budget** that can be sub-allocated to individual team members. This enables team leaders to manage how commission is distributed within their team without needing org-level permissions.

### Design principles

1. **Team as budget holder** — The team stakeholder's `commissionPercentage` is the total pool available for team member allocation
2. **Sub-allocation** — Team managers assign commission from the team budget to individual members. The team's percentage decreases as members receive allocations
3. **Unallocated budget at close** — If a deal closes with remaining team budget (team stakeholder has `commissionPercentage > 0`), that amount is paid to the **team** directly, not to any individual
4. **Team pool freeze** — When a team-only stakeholder (has `team`, no `user`) holds primary status, the entity is blocked from modification until a team member claims or is assigned to it

### Team budget lifecycle

```
1. Lead assigned to Team A (100% commission)
   └── Team A stakeholder: { team: "Team A", user: null, commission: 100%, isPrimary: true,
                              originalTeamCommission: 100 }

2. Team manager assigns Agent X (40% from team budget)
   ├── Team A stakeholder: { team: "Team A", user: null, commission: 60%, isPrimary: false,
   │                          originalTeamCommission: 100 }
   └── Agent X stakeholder: { user: "Agent X", team: "Team A", commission: 40%, isPrimary: true,
                               originalTeamCommission: 100 }

3. Team manager assigns Agent Y (30% from team budget)
   ├── Team A stakeholder: { commission: 30% }   ← remaining budget
   ├── Agent X stakeholder: { commission: 40%, originalTeamCommission: 100 }
   └── Agent Y stakeholder: { commission: 30%, originalTeamCommission: 100 }

4. Deal closes → Commission payments:
   ├── Agent X: 40% → paid to user
   ├── Agent Y: 30% → paid to user
   └── Team A:  30% → paid to TEAM (unallocated budget)
```

`originalTeamCommission` on each member's stakeholder records the team's total allocation (100% in this case), allowing the system to validate that the sum of team member commissions never exceeds the original team budget.

## Entity changes required

To support the stakeholder system features, the following entities require updates:

### Lead entity
- Add `originalTeamCommission` field for tracking team budget allocations
- Add indexes for stakeholder query performance
- Update state machine to handle team pool freeze conditions

### Deal entity  
- Add `originatingLeadId` field for sync service
- Add `originalTeamCommission` field for team budget tracking
- Update creation logic to inherit stakeholders from leads

### User entity
- Add team membership tracking for claim validation
- Update permission computation for team-based access

### Team entity
- Add commission budget tracking
- Add team pool management capabilities

## Business rules

### Commission auto-adjustment formula

When removing a stakeholder and redistributing commission:

```
Formula: newPercentage = oldPercentage × (100 / currentTotalExcludingRemoved)

Example:
BEFORE removal:
- Agent A: 50%
- Agent B: 0% ← Removing
- Agent C: 20%
Total: 70%

AFTER (auto-adjusted to 100%):
- Agent A: 71.43% (50 × 100/70)
- Agent C: 28.57% (20 × 100/70)
Total: 100% ✓
```

### Team budget enforcement

- Team managers can only allocate commission up to `originalTeamCommission`
- Sum of individual team member commissions cannot exceed the original team budget
- Team budget decreases as members are assigned commission
- Remaining team budget at deal close is paid to the team directly

### Transfer validation rules

| Rule                             | Description                                                                               |
| -------------------------------- | ----------------------------------------------------------------------------------------- |
| `fromKeepsPercentage` ceiling    | Cannot exceed the primary stakeholder's current commission                                |
| Transfer amount ≥ 0              | 0% transfers allowed only to existing stakeholders (primary-only, no commission movement) |
| Requester must be primary        | Only the primary stakeholder (or member of primary team) can initiate                     |
| Transfer-back allowed            | Transferring back to a previous owner or team is permitted                                |
| Existing stakeholder update      | If target already has a stakeholder, it is updated (commission added) not duplicated      |
| Auto-cancel on removal/reduction | Pending transfers are cancelled when any stakeholder is removed or has commission reduced |

### Team pool freeze rules

| Rule               | Description                                                                                  |
| ------------------ | -------------------------------------------------------------------------------------------- |
| Modification block | Leads/deals cannot be modified when primary stakeholder is team-only (`team` set, no `user`) |
| Resolution         | Assign or claim the entity to a team member to unfreeze                                      |
| Exception          | Users with stakeholder access (`accessLevel: MODIFY` + `user` set) can still modify          |
| Exception          | Primary stakeholders (user-based, `isPrimary: true`) can still modify                        |

## Access control

### Permission-only access for stakeholder management

<Warning>
  Stakeholder mutation endpoints use **permission-only** access checks.
  Resource-level access (e.g., `accessLevel: MODIFY` via stakeholder access)
  does **not** grant permission to manage stakeholders.
</Warning>

```typescript
// Lead stakeholder endpoints
@CheckAccess({
  permissions: [OrgPermissionKey.CRM_MANAGE, TeamPermissionKey.TEAM_CRM_MANAGE],
})

// Deal stakeholder endpoints
@CheckAccess({
  permissions: [OrgPermissionKey.SALES_MANAGE, TeamPermissionKey.TEAM_SALES_MANAGE],
})
```

**Two-tier access:**

1. **Guard level (AccessGuard)** — Checks if user has `CRM_MANAGE` (org) OR `TEAM_CRM_MANAGE` (any team). Coarse gate.
2. **Controller level (business logic)** — For team managers, validates: team has a stakeholder on the entity, commission doesn't exceed budget, adjustments/removals target their own team, primary can only be set if their team holds primary, and assigned user is a team member.

## Quick reference

### Validation rules by operation

| Operation             | Validation                                                                                                |
| --------------------- | --------------------------------------------------------------------------------------------------------- |
| Create (simple)       | `new + existing = 100` OR `all = 0`; at least one primary after creation                                  |
| CreateWithAdjustments | `new + adjusted = 100`; at least one primary; auto-cancels pending transfers if commission reduced        |
| Update                | `new total ≤ 100`; cannot remove last primary; auto-cancels pending transfers when commission reduced     |
| Remove                | Auto-redistributes proportionally; cannot remove primary while others exist; auto-cancels transfers       |
| Transfer              | `fromKeepsPercentage ≤ primary's commission`; 0% only to existing stakeholders; requester must be primary |
| Sync (PUT)            | `sum(all) = 100`; at least one primary; full replacement (no automatic sync to linked entities)         |
| Team Assign           | Same as create; `originalTeamCommission` set to 100                                                       |
| Team Claim            | User gets team's commission, team → 0%; `originalTeamCommission` recorded; team preserved                 |
| Unassign              | Removes all stakeholders; auto-cancels all pending transfers                                              |

### Team manager constraints

| Rule                | Description                                                                              |
| ------------------- | ---------------------------------------------------------------------------------------- |
| Budget ceiling      | Team budget = team remaining % + team members' combined %. Assignment cannot exceed this |
| Adjustments scope   | Can only adjust own team's stakeholder and team member commissions                       |
| Primary restriction | Cannot set primary if no team stakeholder currently holds primary                        |
| Entity visibility   | Must have a team-associated stakeholder on the entity                                    |
| Assignee membership | Can only assign users who are members of their own team                                  |
| Removal scope       | Can only remove stakeholders belonging to their own team                                 |

### Performance considerations

With stakeholder sync disabled, performance is improved:

<Tip>
  Stakeholder operations no longer need to check and sync linked entities.
  Use `*InTransaction` methods for batch operations.
</Tip>

| Operation          | Without sync | Performance gain |
| ------------------ | ------------ | ---------------- |
| Assign stakeholder | ~50ms        | No sync overhead |
| Remove stakeholder | ~50ms        | No sync overhead |
| Create deal        | ~80ms        | No sync to lead  |

Operations are now faster since they don't require linked entity checks or sync operations.
