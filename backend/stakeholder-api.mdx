# Stakeholder API Specification

> **Version:** 3.16  
> **Last Updated:** 2026-02-23  
> **Status:** Finalized

This document specifies the API endpoints and service functions for managing lead/deal stakeholders, assignment, transfer, and commission handling.

**Version 3.16 Changes:**

- 🔒 **`claimLead` now requires explicit `teamId` for team pool claims:** The `POST /leads/:leadId/claim` endpoint now accepts an optional `teamId` in the request body. When claiming from a team pool, the `teamId` is **required** and targets the specific team-only stakeholder. When claiming from the global pool (no stakeholders), `teamId` must be omitted. This enables a user who is a member of multiple teams to claim the same lead from each team's pool independently, resulting in one stakeholder row per team.
- 🔒 **Claim guards scoped to specific team:** All claim validation guards (`alreadyClaimedByUser`, `claimedByOtherMember`, team pool existence) now check against the specific `teamId` provided, not against all of the user's teams. This prevents false blocks when a user has claimed from Team A but wants to claim from Team B.
- 🔒 **`createStakeholder` normal agent path scoped to `dto.teamId`:** The `POST /leads/:leadId/stakeholders` endpoint for normal agents (without `crm.manage` or `team_crm.manage`) now requires `dto.teamId` and validates team pool existence and "already claimed" checks against that specific team only.
- 🔒 **Global pool claim blocks when stakeholders exist:** `claimLead` without `teamId` now explicitly rejects if the lead has any existing stakeholders, ensuring global pool claims only work on truly unassigned leads.

**Version 3.15 Changes:**

- 🎨 **Frontend dual-record context labels (`LeadStakeholdersCard`, `ManageStakeholdersModal`):**
  - Both components now detect dual-record users — users appearing in more than one stakeholder row (e.g., a personal row with `team=null` plus a team-scoped row with `team=TeamA`).
  - The personal row (`team=null`) for a dual-record user displays an amber **"Personal"** badge with a `UserCircle` icon, visually distinguishing it from regular team-scoped rows.
  - A **"Total: X%"** inline label appears next to the user's email in the card and in the manage table, showing their combined commission across all rows. A tooltip on this label explains the dual-record situation.
  - In `ManageStakeholdersModal`, the Stakeholder column also shows the total commission tooltip on the "Personal" badge.
  - Labels only appear when dual records actually exist — single-record users are unaffected.

**Version 3.14 Changes:**

- 🔒 **Fixed deal stakeholder sync for dual records (`StakeholderSyncService`):** Assignment sync now checks for an existing lead stakeholder matching the same **team context** (`team_id`), not just `user_id`. This prevents skipping the auto-sync when a user has a personal lead stake but the deal assignment is team-scoped. Removal sync now populates the deal stakeholder's `team` and matches the same team context on lead, ensuring the correct auto-synced observer record is found (not the user's personal primary stake).
- 🔒 **Fixed duplicate checks for dual records (`EntityStakeholderService`):** `createInTransaction`, `addStakeholderToEntityInTransaction`, and `getAddPreview` now check for existing stakeholders by `(user, team)` instead of just `user`. This allows creating a second stakeholder row with a different team context while still preventing true duplicates.
- 📝 **Commission Aggregation with Dual Records (v3.14):** New documentation section under "Team Commission Budget" specifying the SQL aggregation rule (`SUM(commission_percentage) WHERE user_id = X`) for payment/reporting, with examples and rules for per-row independence and team budget enforcement.

**Version 3.13 Changes:**

- 🔒 **Fixed `checkStakeholderAccessInTransaction` for dual records:** Previously used `findOne`, returning an arbitrary single record. Now uses `find` and takes the **highest** `accessLevel` across all matching stakeholder rows for a user. This ensures that a user with both a personal stake (READ) and a team-scoped stake (FULL) correctly resolves to FULL access. Bulk access methods (`checkBulkLeadAccessInTransaction`, `checkBulkDealAccessInTransaction`) already iterate all records and were unaffected.
- ✨ **Primary demotion on dual-record claim:** When a user already holds a personal stake (`team_id=NULL`) and claims from a team pool, the personal stake is automatically demoted to `isPrimary=false` so the new team-scoped claim becomes the primary stakeholder.
- ✨ **Auto-promote personal stake on team-scoped removal:** When a team-scoped user stakeholder is removed and the same user has a personal stake on the same entity, the personal stake is automatically promoted to `isPrimary=true`. This applies whether the removed stakeholder was primary (bypasses the "cannot remove primary" validation) or non-primary (post-removal check).

**Version 3.12 Changes:**

- 🐛 **Fixed team pool visibility filter:** The `NOT EXISTS` clause in `applyTeamPoolLeadsFilter` was incorrectly checking for _any_ user stakeholder on the lead. It now scopes the check to only `team_id = ?` (the specific team), so users with a global-pool stake (`team_id=NULL`) or a different-team stake do not hide the lead from Team X's pool.
- 🐛 **Fixed `claimLead` to allow global-pool stakeholders to claim from team pool:** The old guard blocked claiming if any user stakeholder existed. It now only blocks if a stakeholder with `user_id` AND `team_id=X` (same team) already exists. A user who holds a global-pool stake can now additionally create a team-scoped stake by claiming from the pool, resulting in two stakeholder rows.
- 🐛 **Fixed `assignToTeam` guard:** Previously blocked if any stakeholder existed at all. Now only blocks if the target team already has any stakeholder presence (`team_id=X`), whether a team-only row or a member row. Stakeholders from other teams or global-pool users (`team_id=NULL`) no longer prevent team assignment.
- ✨ **New endpoint `PATCH /leads/:id/team-commission`:** Org admin only. Allows increasing or adjusting the commission on an existing team-only stakeholder row (the team's budget) without re-assigning the team. The team must already have a team-only row on the lead. Validates that the new total does not exceed 100%.
- 🔧 **`UpdateEntityStakeholderDto` now includes `originalTeamCommission`:** The update DTO and `updateInTransaction` now persist `originalTeamCommission` when provided, enabling the `updateTeamCommission` service method to correctly track budget after admin adjustments.
- 🗄️ **Dual-record unique index:** The unique index on `entity_stakeholder` changed from `(entityType, entityId, user)` to `(entityType, entityId, user, team)`. This allows a user to hold both a personal row (`team=NULL`) and a team-scoped row on the same entity. Migration: `Migration20260220000000_UpdateStakeholderUniqueIndexForDualRecords`.

**Version 3.11 Changes:**

- 🐛 **Fixed team presence detection via team membership lookup:** v3.10 still failed when stakeholders were assigned as individual users (by an org admin) without any `team_id` on the stakeholder record. Even though the users were members of the team manager's team, the code only checked the stakeholder's `team_id` FK — which was null. The fix now performs a two-layer check: (1) stakeholders with an explicit `team_id` matching the manager's teams, and (2) stakeholders with no `team_id` whose `user_id` is verified as a member of the manager's teams via `AccessService.filterUserIdsInTeams()`. This correctly identifies all team-associated stakeholders regardless of how they were originally assigned.

**Version 3.10 Changes:**

- 🐛 **Fixed team presence detection for batch update and remove stakeholder:** The v3.9 team-scoping validation incorrectly required a **team-only** stakeholder (team budget placeholder with no user) to exist on the entity. This caused 403 errors for team managers whose team had members assigned to the entity but no team-only placeholder (e.g., when members were assigned directly by an org admin). The validation now checks for **any** stakeholder belonging to the team (team-only or user+team members), allowing team managers to manage their team members' commissions regardless of whether a team budget placeholder exists.

**Version 3.9 Changes:**

- 🐛 **Fixed team-scoping on batch update and remove stakeholder:** Team managers (users with `TEAM_CRM_MANAGE` or `TEAM_SALES_MANAGE` but without org-level `CRM_MANAGE`/`SALES_MANAGE`) were previously able to adjust commission percentages and remove stakeholders that did not belong to their team via the `PATCH /stakeholders/batch` and `DELETE /stakeholders/:id` endpoints. These endpoints now enforce the same team-scoping validation as `POST /stakeholders`: all updates and removals must target the team's own stakeholder or team member stakeholders. Additionally, team managers without a team stake on the entity are now rejected with 403 on both endpoints (previously only enforced on the create endpoint).

**Version 3.8 Changes:**

- 🐛 **Fixed transfer commission display bug:** `EntityTransfer` now stores `fromCommissionTotal` (the sender's total commission percentage at transfer creation time, snapshotted as `numeric(5,2)`). This is returned in `EntityTransferDto` and allows the frontend to correctly compute the receiver's share as `fromCommissionTotal - fromKeepsPercentage` instead of the incorrect `100 - fromKeepsPercentage`. Migration: `Migration20260219074759_AddFromCommissionTotalToEntityTransfer`.
- 🎨 **Transfer card label fix:** The Sent tab on `/home/transfers` now shows "You keep / They receive" labels (instead of the generic "Sender keeps / You receive" which was confusing from the sender's perspective).

**Version 3.7 Changes:**

- 🎨 **Transfer Requests page (`/home/transfers`):** A dedicated frontend page was added so any user can review and act on transfer requests without relying solely on the notification bell. The page has two tabs:
  - **Incoming** — requests where the user is the recipient (`toUser`). Shows Accept/Reject actions for pending requests, history for resolved ones.
  - **Sent** — requests the user initiated. Shows Cancel action for pending requests, history for resolved ones.
  - Backed by `GET /entity-transfers/received?userId=…` and `GET /entity-transfers/sent?userId=…`.
  - Pending counts are shown as badge numbers on each tab.
  - Link visible in the sidebar under "Core Modules" for all authenticated users.

**Version 3.6 Changes:**

- 🎨 **Frontend stakeholder UI permission gating:** The `LeadStakeholdersCard` component now gates all management actions (Add Stakeholder, Manage All, Edit, Remove) behind `lead.permissions?.canAssign`. Normal agents without `CRM_MANAGE` or `TEAM_CRM_MANAGE` see a read-only view of stakeholders. See [ACCESS_CONTROL_SYSTEM.md](./ACCESS_CONTROL_SYSTEM.md#frontend-stakeholder-ui-permission-gating) for implementation details.

- 🐛 **Fixed All Leads tab assign visibility:** The `canAssignLeads()` function for the `all` tab previously returned `true` for all authenticated users. It now correctly requires `crm.manage` or team leader role before showing the Assign action in the dropdown.

**Version 3.5 Changes:**

Normal agents can only assign to themselves, team managers can create team pool leads, org admins can assign to anyone. See [ACCESS_CONTROL_SYSTEM.md](./ACCESS_CONTROL_SYSTEM.md#lead-creation-assignment-authorization) for details.

## Stakeholder Sync Feature (v2.0)

### Overview

When working with deals that originate from leads, stakeholder assignments should maintain consistency. The `StakeholderSyncService` automatically syncs stakeholder assignments between deals and their originating leads.

### Design Principles

1. **One-Way Primary Flow**: Leads inherit to deals at deal creation (existing behavior)
2. **Bidirectional Sync**: Deal stakeholder changes now sync back to lead
3. **Safe by Default**: Multiple safety checks prevent unwanted removals
4. **Non-Blocking**: Sync failures log errors but don't fail the primary operation

---

### Use Case 1: Assign to Deal → Auto-Assign to Lead

**Scenario:** When you assign a stakeholder to a deal, they should also be able to see the originating lead.

**API Call:**

```bash
POST /deals/{dealId}/stakeholders
Content-Type: application/json

{
  "userId": "agent-b-uuid",
  "commissionPercentage": 30,
  "accessLevel": "WRITE"
}
```

**What Happens:**

1. Agent B is assigned to Deal #1 (30% commission, WRITE access)
2. System checks: Does Deal #1 have an originating lead?
3. If yes AND lead exists AND lead not archived:
   - Agent B is also assigned to Lead #1 (0% commission, WRITE access, observer)
   - Returns: `{ success: true, syncedToLead: true, leadId: "lead-uuid" }`

**Sync Settings:**

| Property               | Deal Value                 | Lead Value                |
| ---------------------- | -------------------------- | ------------------------- |
| `commissionPercentage` | As specified (e.g., 30%)   | **0%** (observer)         |
| `accessLevel`          | As specified (e.g., WRITE) | **Same as deal**          |
| `isPrimary`            | As specified               | **false** (never primary) |

**Skip Conditions:**

| Condition            | Action    | Reason                         |
| -------------------- | --------- | ------------------------------ |
| No originating lead  | Skip sync | Nothing to sync to             |
| Lead is archived     | Skip sync | Can't modify archived entities |
| User already on lead | Skip sync | No duplication needed          |

---

### Use Case 2: Remove from Deal → Auto-Remove from Lead

**Scenario:** When you remove a stakeholder from a deal, they should also be removed from the lead if they no longer need access.

**API Call:**

```bash
DELETE /deals/{dealId}/stakeholders/{stakeholderId}
```

**What Happens:**

1. Stakeholder removed from Deal #1
2. System checks: Should we also remove from originating lead?
3. Multiple safety checks applied (see below)
4. If all checks pass:
   - Stakeholder removed from Lead #1
   - Other lead stakeholder commissions auto-adjusted to 100%
   - Returns: `{ success: true, syncedToLead: true, leadId: "lead-uuid" }`

**Safety Checks (ALL must pass):**

| #   | Check                                       | Reason                           |
| --- | ------------------------------------------- | -------------------------------- |
| 1   | Lead exists and not archived                | Can't modify archived leads      |
| 2   | User has **0% commission** on lead          | Protect earned commission rights |
| 3   | User is **NOT primary** on lead             | Protect primary ownership        |
| 4   | This is user's **ONLY deal** from that lead | Keep access if other deals exist |

**If ANY check fails:** Stakeholder removed from deal only, lead unchanged

**Commission Auto-Adjustment:**

If removing would break 100% total on lead, the system auto-adjusts:

```
BEFORE:
- Agent A: 50%
- Agent B: 0% ← Removing
- Agent C: 20%
Total: 70%

AFTER (auto-adjusted to 100%):
- Agent A: 71.43% (50 × 100/70)
- Agent C: 28.57% (20 × 100/70)
Total: 100% ✓

Formula: new% = old% × (100 / current_total)
```

---

### Sync Response Format

Both create and remove operations return:

```typescript
interface StakeholderSyncResultDto {
  success: boolean; // Operation succeeded
  syncedToLead: boolean; // Whether sync occurred
  leadId?: string; // ID of lead (if applicable)
  message?: string; // Optional status message
}
```

## Team Commission Budget (v3.0)

### Overview

A team's `commissionPercentage` on an entity stakeholder acts as a **live budget** that can be sub-allocated to individual team members. This enables team leaders to manage how commission is distributed within their team without needing org-level permissions.

### Design Principles

1. **Team as Budget Holder**: The team stakeholder's `commissionPercentage` is the total pool available for team member allocation.
2. **Sub-Allocation**: Team managers assign commission from the team budget to individual members. The team's percentage decreases as members receive allocations.
3. **Unallocated Budget at Close**: If a deal closes with remaining team budget (team stakeholder has `commissionPercentage > 0`), that amount is paid to the **team** directly, not to any individual.
4. **Team Pool Freeze**: When a team-only stakeholder (has `team`, no `user`) holds primary status, the entity is blocked from modification until a team member claims or is assigned to it.

### Team Budget Lifecycle

```
1. Lead assigned to Team A (100% commission)
   └── Team A stakeholder: { team: "Team A", user: null, commission: 100%, isPrimary: true,
                              originalTeamCommission: 100 }

2. Team manager assigns Agent X (40% from team budget)
   ├── Team A stakeholder: { team: "Team A", user: null, commission: 60%, isPrimary: false,
   │                          originalTeamCommission: 100 }
   └── Agent X stakeholder: { user: "Agent X", team: "Team A", commission: 40%, isPrimary: true,
                               originalTeamCommission: 100 }

3. Team manager assigns Agent Y (30% from team budget)
   ├── Team A stakeholder: { commission: 30% }   ← remaining budget
   ├── Agent X stakeholder: { commission: 40%, originalTeamCommission: 100 }
   └── Agent Y stakeholder: { commission: 30%, originalTeamCommission: 100 }

4. Deal closes → Commission payments:
   ├── Agent X: 40% → paid to user
   ├── Agent Y: 30% → paid to user
   └── Team A:  30% → paid to TEAM (unallocated budget)
```

> **v3.2 Note:** `originalTeamCommission` on each member's stakeholder records the team's total allocation (100% in this case), allowing the system to validate that the sum of team member commissions never exceeds the original team budget.

### Claim from Team Pool (Updated v3.2)

When a team member claims a lead from the team pool, the team stakeholder is **preserved** as a budget placeholder (set to `isPrimary: false`, `commissionPercentage: 0`) rather than being deleted. This allows the team to retain its position on the entity for future budget allocation.

The claiming user's stakeholder is created with `originalTeamCommission` set to the team's commission at the time of claim, enabling team budget enforcement even after subsequent reassignments.

```
BEFORE claim:
└── Team A: { commission: 100%, isPrimary: true, originalTeamCommission: 100 }

AFTER claim by Agent X (same team):
├── Team A: { commission: 0%, isPrimary: false, originalTeamCommission: 100 }
└── Agent X: { commission: 100%, isPrimary: true, originalTeamCommission: 100 }
```

### Commission Aggregation with Dual Records (v3.13)

With the dual-record model, a single user can hold **multiple** stakeholder rows on the same entity — one personal (`team_id=NULL`) and one per team they claimed from. Each row carries its own `commissionPercentage`.

**Aggregation Rule for Payment/Reporting:**

```sql
-- Total commission for a user on an entity (across all contexts)
SELECT user_id, SUM(commission_percentage) AS total_commission
FROM entity_stakeholder
WHERE entity_type = :entityType
  AND entity_id = :entityId
  AND user_id = :userId
  AND is_deleted = false
GROUP BY user_id;
```

**Example:**

```
Lead L has stakeholders:
├── Agent X (personal): { user: "X", team: null,    commission: 30% }  ← from global pool
├── Agent X (team):     { user: "X", team: "TeamA", commission: 20% }  ← claimed from team pool
├── Team A (budget):    { team: "TeamA", user: null, commission: 10% }  ← remaining team budget
└── Agent Y:            { user: "Y", team: "TeamB", commission: 40% }

At payment time:
├── Agent X total: 30% + 20% = 50% → paid to user
├── Agent Y total: 40%             → paid to user
└── Team A budget: 10%             → paid to team
```

**Rules:**

1. **Per-row independence**: Each stakeholder row's `commissionPercentage` is managed independently. Adjustments to a team-scoped row do not affect the personal row.
2. **100% total**: The sum across ALL rows on an entity (all users + all teams) must equal 100%. This is enforced at creation and redistribution time.
3. **Team budget enforcement**: The sum of team member commissions (rows where `team_id=X AND user_id IS NOT NULL`) must not exceed the team's `originalTeamCommission`.
4. **Future payment module**: When implementing the payment module, always use the aggregation query (SUM by user_id) rather than assuming one row per user.

### Transfer Execution with Team Stakeholders (Updated v3.0)

When a transfer moves primary ownership to a team that already has a stakeholder on the entity, the existing team stakeholder is **updated** (commission added) rather than creating a duplicate. This ensures team stakeholders act as persistent budget holders.

## Entity Changes Required

### EntityStakeholder Entity

Make `user` nullable to support team-only stakeholders:

```typescript
@ManyToOne(() => User, { nullable: true })  // Changed from required to optional
user?: User;

@ManyToOne(() => Team, { nullable: true })
team?: Team;

// Business rule: Either user OR team must be set (validated in service)
```

**New field (v3.2): `originalTeamCommission`**

```typescript
/**
 * Original team commission percentage when a team member was assigned.
 * Records the team's allocation at the time a user claimed/was assigned from the team pool.
 * Used to enforce that total team member commissions cannot exceed the original team allocation.
 */
@Property({ type: 'decimal', precision: 5, scale: 2, nullable: true })
originalTeamCommission?: number;
```

This field is populated:

- On the **team-only stakeholder** when a team is assigned to a lead/deal (set to 100 for fresh assignments)
- On the **user stakeholder** when a team member claims from the team pool (set to the team's commission at time of claim)

This enables accurate team budget tracking even if external actors (admins) modify individual member commissions later.

### CommissionPayment Entity (v3.0)

Updated to support team-level payments when team budget is unallocated at deal closure:

```typescript
@ManyToOne(() => User, { nullable: true })  // Changed: nullable for team-only payments
user?: User;

@ManyToOne(() => Team, { nullable: true })  // New: team recipient
team?: Team;

// Business rule: Either user OR team must be set (validated in service)
// When stakeholder has user → payment.user = user
// When stakeholder is team-only → payment.team = team, payment.user = null
```

**DTO Changes:**

- `CommissionPaymentDto.user` is now nullable (`user?: UserDto`)
- `CommissionPaymentDto.team` added (`team?: CommissionPaymentTeamDto`)
- `CreateCommissionPaymentDto.userId` is now optional
- `CreateCommissionPaymentDto.teamId` added (optional)

**Purpose:** Remove a stakeholder with automatic lead sync

**Response (v2.0):**

```json
{
  "success": true,
  "syncedToLead": true,
  "leadId": "lead-uuid-123"
}
```

**Sync Behavior:**

- Removes stakeholder from deal first
- If deal has `originatingLead` AND all safety checks pass:
  - Removes stakeholder from lead
  - Auto-adjusts other lead stakeholder commissions to 100%
- Safety checks:
  1. Lead not archived
  2. User has 0% commission on lead
  3. User not primary on lead
  4. This is user's ONLY deal from that lead

**Response when sync skipped:**

```json
{
  "success": true,
  "syncedToLead": false,
  "leadId": "lead-uuid-123"
}
```

## Business Rules

### Commission Rules

| Rule       | Description                                                                                                                        |
| ---------- | ---------------------------------------------------------------------------------------------------------------------------------- |
| Sum = 100% | If any stakeholder has commission > 0, all must sum to 100%                                                                        |
| Sum = 0%   | Valid state when commission not configured yet                                                                                     |
| Min 0%     | 0% transfer allowed when target is an existing stakeholder (primary-only transfer, v3.3). Otherwise, transfer amount must be > 0%. |
| Primary    | At least one primary stakeholder must exist at all times (v3.0). Enforced on create, update, remove, and batch update (v3.3).      |

### Validation Rules by Operation

| Operation             | Validation                                                                                                                                                                                                           |
| --------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Create (simple)       | `new + existing = 100` OR `all = 0` (v3.2: strictly enforced to be exactly 0% or 100%); at least one primary must exist after creation (v3.3)                                                                        |
| CreateWithAdjustments | `new + adjusted = 100`; at least one primary after operation (v3.2); explicit `accessLevel` supported (v3.3); **auto-cancels pending transfers if any adjusted stakeholder has commission reduced (v3.4)**           |
| Update                | `new total ≤ 100`; cannot remove last primary (v3.0); **auto-cancels pending transfers when commission is reduced (v3.4)**                                                                                           |
| Remove                | Auto-redistributes commission proportionally among remaining stakeholders; cannot remove primary while other stakeholders exist (v3.3, checked pre-delete); **auto-cancels pending transfers for the entity (v3.4)** |
| Transfer              | `fromKeepsPercentage ≤ primary's commission`; `transferAmount ≥ 0` (0% allowed only to existing stakeholders, v3.3); requester must be primary or member of primary team (v3.1)                                      |
| Batch                 | `sum(all) = 100` (enforced, v3.1); at least one primary (v3.0); **auto-cancels pending transfers if any stakeholder has commission reduced (v3.4)**                                                                  |
| Team Assign           | Same as create; `originalTeamCommission` set to 100 (v3.2)                                                                                                                                                           |
| Team Claim            | `userGets = teamHad`, team → 0%; `originalTeamCommission` recorded (v3.2); team stakeholder preserved (v3.0)                                                                                                         |
| Unassign              | Removes all stakeholders; **auto-cancels all pending transfers for the entity (v3.4)**                                                                                                                               |

## Access Control (v3.0)

### Permission-Only Access for Stakeholder Management

**Important change in v3.0:** Stakeholder mutation endpoints (add, update, remove) now use **permission-only** access checks. Resource-level access (e.g., `accessLevel: MODIFY` on the entity via stakeholder access) does **NOT** grant permission to manage stakeholders.

**Rationale:** Stakeholder management is an administrative function that should be controlled by explicit RBAC permissions, not by the entity-level access that comes from being a stakeholder. A user who can modify a lead's data should not automatically be able to add, remove, or reassign stakeholders on that lead.

**Implementation:** The `@CheckAccess` decorator on stakeholder mutation endpoints uses only the `permissions` array (no `resource` configuration):

```typescript
// Lead stakeholder endpoints
@CheckAccess({
  permissions: [OrgPermissionKey.CRM_MANAGE, TeamPermissionKey.TEAM_CRM_MANAGE],
})

// Deal stakeholder endpoints
@CheckAccess({
  permissions: [OrgPermissionKey.SALES_MANAGE, TeamPermissionKey.TEAM_SALES_MANAGE],
})
```

**Two-Tier Access:**

1. **Guard Level (AccessGuard):** Checks if user has `CRM_MANAGE` (org) OR `TEAM_CRM_MANAGE` (any team). This is a coarse gate -- team managers pass even if their team isn't on this specific entity.
2. **Controller Level (Business Logic):** For team managers, further validates:
   - Their team has a stakeholder on the entity (enforced on `createStakeholder`, `batchUpdate`, `removeStakeholder`, v3.9)
   - Commission assignments don't exceed team budget (`createStakeholder` only)
   - Adjustments/updates only touch their own team's stakeholders (enforced on `createStakeholder` and `batchUpdate`, v3.9)
   - Removals only target their own team's stakeholders (enforced on `removeStakeholder` and `batchUpdate` via `removeStakeholderId`, v3.9)
   - Primary can only be set if any stakeholder belonging to their team (team-only or user member) currently holds primary (v3.2)
   - Assigned user must be a member of the manager's team (v3.2)

## Performance Considerations (v2.0)

### Query Optimization

The sync service follows patterns from `QUERY_OPTIMIZATION_PATTERNS.md`:

1. **EXISTS subqueries** for counting other deals (no loading IDs)
2. **Transaction composition** with `*InTransaction` methods
3. **Batch operations** where possible
4. **Non-blocking errors** - logs but doesn't fail primary operation

### Impact on Response Time

| Operation          | Without Sync | With Sync | Overhead |
| ------------------ | ------------ | --------- | -------- |
| Assign stakeholder | ~50ms        | ~80ms     | +30ms    |
| Remove stakeholder | ~50ms        | ~100ms    | +50ms    |

The additional queries are:

- Check if deal has originating lead (1 query)
- Check if lead exists and archived (1 query)
- Check if user already on lead (1 query)
- Count user's other deals (1 query - EXISTS pattern)
- Assign/remove on lead (1-2 queries)

Total: 4-6 additional queries, all optimized with indexes.
