title: "Directory Access Matrix"
description: "Comprehensive access control matrix for CRM directory endpoints, defining visibility rules for leads, deals, contacts, companies, and teams based on user permissions and stakeholder relationships."
---

This document describes what lightweight CRM directory endpoints return for each role/permission tier. Directory endpoints are picker APIs, not full list/kanban views: they return minimal rows and enforce visibility inside the service query.

## Shared Rules

<Info>
Directory rows are active only unless a specific endpoint states otherwise:
- `is_deleted = false`
- `is_archived = false`
- `page`, `limit`, and `search` are supported on all CRM entity directories
</Info>

<Note>
**Visibility Levels:**
- **Org-wide visibility**: All active records in the current organization
- **Team visibility**: Records linked through `entity_stakeholder` rows for teams where the caller holds the listed team permission
- **MY visibility**: Direct user stakeholder paths and entity-specific self paths documented below
</Note>

## Lead Directory

**Endpoint:** `GET /v1/leads/directory`

| Permission / relationship | Visible records |
| --- | --- |
| `system.admin`, `system.owner`, `crm.view`, `crm.manage` | All active organization leads |
| `team_crm.view` | MY leads plus leads with a stakeholder row for a team where the user has `team_crm.view` |
| `team_crm.manage` | MY leads plus leads with a stakeholder row for a team where the user has `team_crm.manage` |
| Direct lead stakeholder | Leads where the user is a non-deleted stakeholder |
| Lead creator | Leads created by the user |
| No matching permission or relationship | No additional records |

### Lead Directory Filters

<AccordionGroup>
<Accordion title="hasContact Filter">
- `hasContact=true` returns only leads whose person has a non-deleted contact
- `hasContact=false` returns only leads whose person has no non-deleted contact, including archived contacts
- This is the correct filter for lead-to-contact conversion pickers
</Accordion>

<Accordion title="requiredAction Filter">
`requiredAction=modify` narrows rows to leads the caller can edit:

| Permission / relationship | Editable directory records |
| --- | --- |
| `system.admin`, `system.owner`, `crm.manage` | All active organization leads after other filters |
| Direct lead stakeholder with `WRITE` or `FULL` | That lead |
| `team_crm.manage` on a team with a lead stakeholder row at `WRITE` or `FULL` | That lead |
| `crm.view`, `team_crm.view`, creator-only, direct `READ` stakeholder | Not editable by this filter |
</Accordion>
</AccordionGroup>

<Tip>
The create-contact modal uses `GET /v1/leads/directory?hasContact=false&requiredAction=modify`, then fetches `GET /v1/leads/:id` after selection to hydrate the full person profile.
</Tip>

## Deal Directory

**Endpoint:** `GET /v1/deals/directory`

| Permission / relationship | Visible records |
| --- | --- |
| `system.admin`, `system.owner`, `sales.view`, `sales.manage` | All active organization deals |
| `team_sales.manage` | MY deals plus deals with a stakeholder row for a team where the user has `team_sales.manage` |
| Direct deal stakeholder | Deals where the user is a non-deleted stakeholder |
| Deal creator | Deals created by the user |
| No matching permission or relationship | No additional records |

## Contact Directory

**Endpoint:** `GET /v1/contacts/directory`

| Permission / relationship | Visible records |
| --- | --- |
| `system.admin`, `system.owner`, `crm.view`, `crm.manage` | All active organization contacts |
| `sales.view`, `sales.manage` | Active contacts linked to at least one non-deleted deal via `deal_contact` |
| `team_crm.view` or `team_crm.manage` | MY contacts plus contacts connected to leads, deals, or contacts where those teams have stakeholder access |
| Direct contact stakeholder paths | Contacts linked through lead/person, deal/contact, or direct contact stakeholder paths |
| Creator-only | Contacts created by the current user |
| No matching permission or relationship | No additional records |

### Scoped Contact Directory Filters

<AccordionGroup>
<Accordion title="dealId Filter">
- Requires `sales.view`, `sales.manage`, or `DEAL ACCESS`
- Queries from `deal_contact` joined to active contacts (same visibility anchor as `GET /deals/:id/contacts`)
- Not a contact-first `EXISTS` filter
- Rows include junction `isPrimary` and `role` for that deal
- Directory mapping must preserve the selected raw `contact_id` alias for `id` so the response never exposes the `deal_contact.id` junction primary key
</Accordion>

<Accordion title="companyId Filter">
- Requires `crm.view`, `crm.manage`, or `COMPANY ACCESS`
- Returns contacts linked to that company
</Accordion>

<Accordion title="excludeCompanyId Filter">
- Unscoped only
- Applies normal unscoped visibility
- Requires company access
- Removes contacts already linked to that company
</Accordion>
</AccordionGroup>

## Company Directory

**Endpoint:** `GET /v1/companies/directory`

| Permission / relationship | Visible records |
| --- | --- |
| `system.admin`, `system.owner`, `crm.view`, `crm.manage` | All active organization companies |
| `team_crm.manage` | MY companies plus companies connected through lead, deal, or contact stakeholder paths for managed teams |
| Direct company stakeholder paths | Companies visible through direct company access or linked CRM relationships |
| No matching permission or relationship | No additional records |

## Team Directory

**Endpoint:** `GET /v1/teams/directory`

Returns minimal `TeamDirectoryDto` rows: `id`, `publicId`, `name`, `description`, `category`

| Query | Access | Visible records |
| --- | --- | --- |
| _(unscoped)_ | Any authenticated org member (no `@CheckAccess`) | All active org teams |
| `userId` | Any authenticated org member (no `@CheckAccess`) | Active teams where the given user has a non-deleted `team_membership` in the org |

<Warning>
Use `GET /v1/teams/directory?userId=…` for user-scoped team pickers (e.g. CRM transfer modal when a recipient user is selected). Do **not** use `GET /v1/users/:userId/teams` for pickers — that route requires `system.admin`.
</Warning>

## User Directories

Minimal user-row pickers (`UserDirectoryDto`: `id`, `firstName`, `lastName`, `email`, `phoneNumber`, optional avatar fields).

<Note>
`email` and `phoneNumber` are included so org-internal surfaces (Cc/Bcc, listing portal-contact prefill) can resolve contact details without the full `GET /users` list.
</Note>

| Endpoint | Access | Visible records |
| --- | --- | --- |
| `GET /v1/users/directory` | Any authenticated org member (no `@CheckAccess`) | All active org members |
| `GET /v1/users/task-calendar-filter` | Any authenticated org member | Self; org managers → all active members; `team.admin` → self + led-team members; otherwise self only |
| `GET /v1/users/listing-agent-directory` | Any authenticated org member | Self; org managers → all active members; `team.admin` → self + led-team members; otherwise self only |
| `GET /v1/teams/:teamId/members/directory` | Any authenticated org member (no `@CheckAccess`) | Active users with a non-deleted membership on the given team |

### User Directory Permission Tiers

<Tabs>
<Tab title="Task Calendar Filter">
- **Self**: Always included
- **Org managers** (`system.admin`, `system.owner`, `crm.manage`): All active members
- **Team admin**: Self + led-team members
- **Otherwise**: Self only
</Tab>

<Tab title="Listing Agent Directory">
- **Self**: Always included
- **Org managers** (`system.admin`, `system.owner`, `crm.manage`, `real_estate.manage`): All active members
- **Team admin**: Self + led-team members
- **Otherwise**: Self only

<Note>
Tiering is implemented by `AccessService.listListingAgentTargetUserIdsInTransaction`. The listing modal also surfaces two **always-available** special options:
- **Unit Manager** (the linked unit's manager)
- **Listing Publisher (You)** (the current user)
</Note>
</Tab>
</Tabs>

<Warning>
Use `GET /v1/teams/:teamId/members/directory` for team-scoped user pickers (e.g. CRM transfer modal when a team is selected). Do **not** use `GET /v1/teams/:teamId/members` for pickers — that route returns full membership payloads and requires `system.admin` or team `ResourceAction.ACCESS`.
</Warning>

### Listing Lead-Routing Team Picker

**Endpoint:** `GET /v1/teams/listing-routing-directory`

Returns minimal `TeamDirectoryDto` rows for teams selectable in listing lead-routing configuration.

<Check>
The same endpoint backs the listing "Lead routing" → **Specific user** picker, so the routing user picker has identical selection permissions to the agent picker.
</Check>
