Directory endpoints are lightweight picker APIs that return minimal rows and enforce visibility inside the service query. They are not full list or kanban views.

## Shared Rules

<Info>
All directory endpoints return **active records only** unless explicitly stated otherwise:
- `is_deleted = false`
- `is_archived = false`
</Info>

**Standard Query Parameters:**
- `page` – Pagination offset
- `limit` – Results per page
- `search` – Text search filter

All CRM entity directories support these parameters.

### Visibility Tiers

<CardGroup cols={3}>
  <Card title="Org-Wide" icon="building">
    All active records in the current organization
  </Card>
  <Card title="Team" icon="users">
    Records linked through `entity_stakeholder` rows for teams where the caller holds the required team permission
  </Card>
  <Card title="MY" icon="user">
    Direct user stakeholder paths and entity-specific self paths
  </Card>
</CardGroup>

## Lead Directory

**Endpoint:** `GET /v1/leads/directory`

### Visibility Rules

| Permission / Relationship | Visible Records |
|--------------------------|-----------------|
| `system.admin`, `system.owner`, `crm.view`, `crm.manage` | All active organization leads |
| `team_crm.view` | MY leads plus leads with a stakeholder row for a team where the user has `team_crm.view` |
| `team_crm.manage` | MY leads plus leads with a stakeholder row for a team where the user has `team_crm.manage` |
| Direct lead stakeholder | Leads where the user is a non-deleted stakeholder |
| Lead creator | Leads created by the user |
| No matching permission or relationship | No additional records |

### Query Parameters

<AccordionGroup>
  <Accordion title="hasContact (boolean)">
    - `hasContact=true` – Returns only leads whose person has a non-deleted contact
    - `hasContact=false` – Returns only leads whose person has no non-deleted contact (including archived contacts)
    
    <Tip>
    Use `hasContact=false` for lead-to-contact conversion pickers
    </Tip>
  </Accordion>

  <Accordion title="requiredAction=modify">
    Narrows rows to leads the caller can edit:

    | Permission / Relationship | Editable Directory Records |
    |--------------------------|---------------------------|
    | `system.admin`, `system.owner`, `crm.manage` | All active organization leads after other filters |
    | Direct lead stakeholder with `WRITE` or `FULL` | That lead |
    | `team_crm.manage` on a team with a lead stakeholder row at `WRITE` or `FULL` | That lead |
    | `crm.view`, `team_crm.view`, creator-only, direct `READ` stakeholder | Not editable by this filter |
  </Accordion>
</AccordionGroup>

### Usage Example

<CodeGroup>
```typescript Create Contact Modal
// Fetch leads that can be converted to contacts
GET /v1/leads/directory?hasContact=false&requiredAction=modify

// After selection, hydrate full person profile
GET /v1/leads/:id
```
</CodeGroup>

## Contact Directory

**Endpoint:** `GET /v1/contacts/directory`

### Visibility Rules

| Permission / Relationship | Visible Records |
|--------------------------|-----------------|
| `system.admin`, `system.owner`, `crm.view`, `crm.manage` | All active organization contacts |
| `sales.view`, `sales.manage` | Active contacts linked to at least one non-deleted deal via `deal_contact` |
| `team_crm.view` or `team_crm.manage` | MY contacts plus contacts connected to leads, deals, or contacts where those teams have stakeholder access |
| Direct contact stakeholder paths | Contacts linked through lead/person, deal/contact, or direct contact stakeholder paths |
| Creator-only | Contacts created by the current user |
| No matching permission or relationship | No additional records |

### Scoped Filters

<Tabs>
  <Tab title="dealId">
    **Requirements:** `sales.view`, `sales.manage`, or `DEAL ACCESS`

    Queries from `deal_contact` joined to active contacts (same visibility anchor as `GET /deals/:id/contacts`), not a contact-first `EXISTS` filter.

    **Response includes:**
    - Junction `isPrimary` for that deal
    - Junction `role` for that deal

    <Warning>
    Directory mapping must preserve the selected raw `contact_id` alias for `id` so the response never exposes the `deal_contact.id` junction primary key as the directory row id.
    </Warning>
  </Tab>

  <Tab title="companyId">
    **Requirements:** `crm.view`, `crm.manage`, or `COMPANY ACCESS`

    Returns contacts linked to the specified company.
  </Tab>

  <Tab title="excludeCompanyId">
    **Unscoped only.** Applies normal unscoped visibility, requires company access, then removes contacts already linked to that company.
  </Tab>
</Tabs>

## Team Directory

**Endpoint:** `GET /v1/teams/directory`

Returns minimal `TeamDirectoryDto` rows: `id`, `publicId`, `name`, `description`, `category`

### Visibility Rules

| Query | Access | Visible Records |
|-------|--------|-----------------|
| _(unscoped)_ | Any authenticated org member (no `@CheckAccess`) | All active org teams |
| `userId` | Any authenticated org member (no `@CheckAccess`) | Active teams where the given user has a non-deleted `team_membership` in the org |

<Note>
Use `GET /v1/teams/directory?userId=…` for user-scoped team pickers (e.g. CRM transfer modal when a recipient user is selected).

**Do not** use `GET /v1/users/:userId/teams` for pickers — that route requires `system.admin`.
</Note>

## User Directories

Returns minimal `UserDirectoryDto` rows: `id`, `firstName`, `lastName`, `email`, `phoneNumber`, optional avatar fields.

<Info>
`email` and `phoneNumber` are included so org-internal surfaces (Cc/Bcc, listing portal-contact prefill) can resolve contact details without the full `GET /users` list.
</Info>

### Endpoints

| Endpoint | Access | Visible Records |
|----------|--------|-----------------|
| `GET /v1/users/directory` | Any authenticated org member (no `@CheckAccess`) | All active org members |
| `GET /v1/users/task-calendar-filter` | Any authenticated org member | Self; org managers (`system.admin` / `system.owner` / `crm.manage`) → all active members; `team.admin` → self + led-team members; otherwise self only |
| `GET /v1/users/listing-agent-directory` | Any authenticated org member | Self; org managers (`system.admin` / `system.owner` / `crm.manage` / `real_estate.manage`) → all active members; `team.admin` → self + led-team members; otherwise self only |
| `GET /v1/teams/:teamId/members/directory` | Any authenticated org member (no `@CheckAccess`) | Active users with a non-deleted membership on the given team (same `UserDirectoryDto` shape as `/users/directory`; optional `search` on name/email) |

<Warning>
Use `GET /v1/teams/:teamId/members/directory` for team-scoped user pickers (e.g. CRM transfer modal when a team is selected).

**Do not** use `GET /v1/teams/:teamId/members` for pickers — that route returns full membership payloads and requires `system.admin` or team `ResourceAction.ACCESS`.
</Warning>

### Listing Agent Directory

`GET /v1/users/listing-agent-directory` is the permission-scoped picker for **who a user may assign as a listing's agent / portal contact** (create/edit-listing Assignment step).

**Tiering:** Implemented by `AccessService.listListingAgentTargetUserIdsInTransaction` — the same three-tier model as the task-calendar filter, but with `real_estate.manage` added to the org-wide manager set.

**Always-Available Special Options:**

The listing modal surfaces two special options **independent of this scoped list**:
1. **Unit Manager** – The linked unit's manager
2. **Listing Publisher (You)** – The current user

<Tip>
The same endpoint backs the listing "Lead routing" → **Specific user** picker, so the routing user picker has identical selection permissions to the agent picker.
</Tip>

### Listing Lead-Routing Team Picker

**Endpoint:** `GET /v1/teams/listing-routing-directory`

Returns minimal `TeamDirectoryDto` rows.

| Permission / Relationship | Selectable Teams |
|--------------------------|------------------|
| `system.admin`, `system.owner`, `real_estate.manage` | All active org teams |
| `team.admin` | Teams where the user holds `team.admin` |
| Otherwise | No teams |

<Check>
This endpoint enforces the same three-tier model as listing-agent selection for team-based routing options.
</Check>
