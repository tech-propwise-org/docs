---
title: "Directory Access Matrix"
description: "Access control rules for CRM directory endpoints including leads, deals, contacts, companies, and teams"
---

Directory endpoints are lightweight picker APIs that return minimal records with enforced visibility rules. Unlike full list or kanban views, these endpoints are designed for selection components and dropdowns throughout the CRM system.

## Shared directory rules

<Note>
All directory endpoints follow these common patterns unless explicitly stated otherwise.
</Note>

### Active records only
Directory endpoints return only active records by default:
- `is_deleted = false`
- `is_archived = false`

### Standard query parameters
All CRM directory endpoints support:
- `page` - Pagination offset
- `limit` - Results per page  
- `search` - Text search across relevant fields

### Visibility tiers
- **Org-wide**: All active records in the current organization
- **Team**: Records linked through `entity_stakeholder` rows for teams where the caller has the specified permission
- **MY**: Records accessible through direct user stakeholder relationships or entity-specific ownership paths

## Lead directory

<CodeGroup>
```http GET
GET /v1/leads/directory
```
</CodeGroup>

### Access matrix

| Permission / Relationship | Visible Records |
|---------------------------|----------------|
| `system.admin`, `system.owner`, `crm.view`, `crm.manage` | All active organization leads |
| `team_crm.view` | MY leads plus leads with stakeholder rows for teams where user has `team_crm.view` |
| `team_crm.manage` | MY leads plus leads with stakeholder rows for teams where user has `team_crm.manage` |
| Direct lead stakeholder | Leads where the user is a non-deleted stakeholder |
| Lead creator | Leads created by the user |
| No matching permission | No additional records |

### Special filters

<Tabs>
<Tab title="Contact status">
- `hasContact=true` - Returns only leads whose person has a non-deleted contact
- `hasContact=false` - Returns only leads whose person has no non-deleted contact (including archived contacts)

<Tip>
Use `hasContact=false` for lead-to-contact conversion pickers to avoid duplicate contacts.
</Tip>
</Tab>

<Tab title="Edit permissions">
`requiredAction=modify` narrows results to leads the caller can edit:

| Permission / Relationship | Editable Records |
|---------------------------|-----------------|
| `system.admin`, `system.owner`, `crm.manage` | All active organization leads after other filters |
| Direct lead stakeholder with `WRITE` or `FULL` | That specific lead |
| `team_crm.manage` on team with lead stakeholder at `WRITE` or `FULL` | That specific lead |
| `crm.view`, `team_crm.view`, creator-only, direct `READ` stakeholder | Not editable by this filter |
</Tab>
</Tabs>

<Info>
The create-contact modal uses `GET /v1/leads/directory?hasContact=false&requiredAction=modify`, then fetches the full lead record after selection.
</Info>

## Contact directory

<CodeGroup>
```http GET
GET /v1/contacts/directory
```
</CodeGroup>

### Access matrix

| Permission / Relationship | Visible Records |
|---------------------------|----------------|
| `system.admin`, `system.owner`, `crm.view`, `crm.manage` | All active organization contacts |
| `sales.view`, `sales.manage` | Active contacts linked to at least one non-deleted deal via `deal_contact` |
| `team_crm.view` or `team_crm.manage` | MY contacts plus contacts connected to leads, deals, or contacts where those teams have stakeholder access |
| Direct contact stakeholder paths | Contacts linked through lead/person, deal/contact, or direct contact stakeholder paths |
| Creator-only | Contacts created by the current user |
| No matching permission | No additional records |

### Scoped filters

<AccordionGroup>
<Accordion title="dealId filter">
**Requirements**: `sales.view`, `sales.manage`, or deal access

Queries from `deal_contact` joined to active contacts. Returns the same visibility as `GET /deals/:id/contacts`.

<Warning>
Response rows include junction `isPrimary` and `role` for the specified deal. Directory mapping must preserve the selected raw `contact_id` as the `id` field to avoid exposing the `deal_contact.id` junction primary key.
</Warning>
</Accordion>

<Accordion title="companyId filter">
**Requirements**: `crm.view`, `crm.manage`, or company access

Returns contacts linked to the specified company.
</Accordion>

<Accordion title="excludeCompanyId filter">
**Requirements**: Company access (unscoped only)

Applies normal unscoped visibility, then removes contacts already linked to the specified company.
</Accordion>
</AccordionGroup>

## Team directory

<CodeGroup>
```http GET
GET /v1/teams/directory
```
</CodeGroup>

Returns minimal `TeamDirectoryDto` rows: `id`, `publicId`, `name`, `description`, `category`.

| Query Parameters | Access Requirements | Visible Records |
|-----------------|-------------------|----------------|
| _(unscoped)_ | Any authenticated org member | All active org teams |
| `userId` | Any authenticated org member | Active teams where the specified user has a non-deleted `team_membership` |

<Tip>
Use `GET /v1/teams/directory?userId=…` for user-scoped team pickers in CRM transfer modals. Do **not** use `GET /v1/users/:userId/teams` for pickers as it requires `system.admin`.
</Tip>

## User directories

User directory endpoints return minimal `UserDirectoryDto` objects containing:
- `id`, `firstName`, `lastName`, `email`, `phoneNumber`
- Optional avatar fields

<Note>
`email` and `phoneNumber` are included so org-internal surfaces can resolve contact details without requiring full user list access.
</Note>

### General user directory

<CodeGroup>
```http GET
GET /v1/users/directory
```
</CodeGroup>

**Access**: Any authenticated org member  
**Returns**: All active org members

### Task calendar filter

<CodeGroup>
```http GET
GET /v1/users/task-calendar-filter
```
</CodeGroup>

| User Role | Visible Users |
|-----------|--------------|
| Self | Always included |
| Org managers (`system.admin`, `system.owner`, `crm.manage`) | All active members |
| Team admins (`team.admin`) | Self + led-team members |
| Other users | Self only |

### Listing agent directory

<CodeGroup>
```http GET
GET /v1/users/listing-agent-directory
```
</CodeGroup>

Permission-scoped picker for assigning listing agents and portal contacts.

| User Role | Selectable Users |
|-----------|-----------------|
| Self | Always included |
| Org managers (`system.admin`, `system.owner`, `crm.manage`, `real_estate.manage`) | All active members |
| Team admins (`team.admin`) | Self + led-team members |
| Other users | Self only |

<Info>
The listing modal also provides two special options independent of this scoped list:
- **Unit Manager** (the linked unit's manager)
- **Listing Publisher (You)** (the current user)
</Info>

### Team member directory

<CodeGroup>
```http GET
GET /v1/teams/:teamId/members/directory
```
</CodeGroup>

**Access**: Any authenticated org member  
**Returns**: Active users with non-deleted membership on the specified team  
**Search**: Optional `search` parameter on name/email

<Warning>
Use this endpoint for team-scoped user pickers. Do **not** use `GET /v1/teams/:teamId/members` for pickers as it returns full membership payloads and requires `system.admin` or team `ResourceAction.ACCESS`.
</Warning>

### Listing routing team picker

<CodeGroup>
```http GET
GET /v1/teams/listing-routing-directory
```
</CodeGroup>

Returns minimal `TeamDirectoryDto` rows for listing lead routing configuration.

<Check>
This endpoint uses the same three-tier permission model as other user directories, with `real_estate.manage` included in the org-wide manager set.
</Check>
