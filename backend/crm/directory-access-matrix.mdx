---
title: "Directory Access Matrix"
description: "CRM directory endpoint visibility rules and access control for different user roles and permissions"
---

This document describes what lightweight CRM directory endpoints return for each role and permission tier. Directory endpoints are picker APIs, not full list or kanban views - they return minimal rows and enforce visibility inside the service query.

<Note>
Directory endpoints are designed for selection workflows like dropdowns and pickers, returning only essential data for display and selection purposes.
</Note>

## Shared Rules

Directory rows are active only unless a specific endpoint states otherwise:

- `is_deleted = false`
- `is_archived = false`
- `page`, `limit`, and `search` are supported on all CRM entity directories

<Info>
**Visibility Scope Definitions:**
- **Org-wide visibility**: All active records in the current organization
- **Team visibility**: Records linked through `entity_stakeholder` rows for teams where the caller holds the listed team permission
- **MY visibility**: Direct user stakeholder paths and entity-specific self paths
</Info>

## Lead Directory

<CodeGroup>
```http Endpoint
GET /v1/leads/directory
```
</CodeGroup>

### Basic Visibility Rules

| Permission / Relationship | Visible Records |
|---------------------------|-----------------|
| `system.admin`, `system.owner`, `crm.view`, `crm.manage` | All active organization leads |
| `team_crm.view` | MY leads plus leads with a stakeholder row for a team where the user has `team_crm.view` |
| `team_crm.manage` | MY leads plus leads with a stakeholder row for a team where the user has `team_crm.manage` |
| Direct lead stakeholder | Leads where the user is a non-deleted stakeholder |
| Lead creator | Leads created by the user |
| No matching permission or relationship | No additional records |

### Contact Filter

<Tabs>
  <Tab title="Has Contact">
    `hasContact=true` returns only leads whose person has a non-deleted contact.
  </Tab>
  <Tab title="No Contact">
    `hasContact=false` returns only leads whose person has no non-deleted contact, including archived contacts. This is the correct filter for lead-to-contact conversion pickers.
  </Tab>
</Tabs>

### Modification Access

`requiredAction=modify` narrows rows to leads the caller can edit:

| Permission / Relationship | Editable Directory Records |
|---------------------------|----------------------------|
| `system.admin`, `system.owner`, `crm.manage` | All active organization leads after other filters |
| Direct lead stakeholder with `WRITE` or `FULL` | That lead |
| `team_crm.manage` on a team with a lead stakeholder row at `WRITE` or `FULL` | That lead |
| `crm.view`, `team_crm.view`, creator-only, direct `READ` stakeholder | Not editable by this filter |

<Tip>
The create-contact modal uses `GET /v1/leads/directory?hasContact=false&requiredAction=modify`, then fetches `GET /v1/leads/:id` after selection to hydrate the full person profile.
</Tip>

## Contact Directory

<CodeGroup>
```http Endpoint
GET /v1/contacts/directory
```
</CodeGroup>

### Basic Visibility Rules

| Permission / Relationship | Visible Records |
|---------------------------|-----------------|
| `system.admin`, `system.owner`, `crm.view`, `crm.manage` | All active organization contacts |
| `sales.view`, `sales.manage` | Active contacts linked to at least one non-deleted deal via `deal_contact` |
| `team_crm.view` or `team_crm.manage` | MY contacts plus contacts connected to leads, deals, or contacts where those teams have stakeholder access |
| Direct contact stakeholder paths | Contacts linked through lead/person, deal/contact, or direct contact stakeholder paths |
| Creator-only | Contacts created by the current user |
| No matching permission or relationship | No additional records |

### Scoped Filters

<AccordionGroup>
  <Accordion title="dealId Filter">
    - **Requirements**: `sales.view`, `sales.manage`, or `DEAL ACCESS`
    - **Behavior**: Queries from `deal_contact` joined to active contacts (same visibility anchor as `GET /deals/:id/contacts`)
    - **Additional Data**: Rows include junction `isPrimary` and `role` for that deal
    - **Important**: Directory mapping must preserve the selected raw `contact_id` alias for `id` so the response never exposes the `deal_contact.id` junction primary key
  </Accordion>
  
  <Accordion title="companyId Filter">
    - **Requirements**: `crm.view`, `crm.manage`, or `COMPANY ACCESS`
    - **Behavior**: Returns contacts linked to that company
  </Accordion>
  
  <Accordion title="excludeCompanyId Filter">
    - **Requirements**: Company access (unscoped only)
    - **Behavior**: Applies normal unscoped visibility, then removes contacts already linked to that company
  </Accordion>
</AccordionGroup>

## Company Directory

<CodeGroup>
```http Endpoint
GET /v1/companies/directory
```
</CodeGroup>

| Permission / Relationship | Visible Records |
|---------------------------|-----------------|
| `system.admin`, `system.owner`, `crm.view`, `crm.manage` | All active organization companies |
| `team_crm.manage` | MY companies plus companies connected through lead, deal, or contact stakeholder paths for managed teams |
| Direct company stakeholder paths | Companies visible through direct company access or linked CRM relationships |
| No matching permission or relationship | No additional records |

## Important Distinction

<Warning>
Directory visibility is not a substitute for detail endpoint access. Any flow that needs full entity data must still call the detail endpoint after a directory selection, so the standard `@CheckAccess` and service-level validation run against the selected record.
</Warning>

<Steps>
  <Step title="User makes directory selection">
    Directory endpoint returns minimal data for picker/dropdown display
  </Step>
  
  <Step title="Application fetches full details">
    After selection, call the appropriate detail endpoint (e.g., `GET /v1/leads/:id`)
  </Step>
  
  <Step title="Access validation occurs">
    Detail endpoint enforces full `@CheckAccess` and service-level validation
  </Step>
</Steps>
