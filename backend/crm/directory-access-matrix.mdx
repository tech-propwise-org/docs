## Overview

Directory endpoints are lightweight picker APIs designed for dropdowns, modals, and autocomplete fields. Unlike full list or kanban views, they:

- Return minimal row data optimized for selection UI
- Enforce visibility rules inside the service query
- Support pagination and search on all endpoints
- Return only active (non-deleted, non-archived) records by default

<Note>
All directory endpoints support `page`, `limit`, and `search` query parameters for pagination and filtering.
</Note>

## Lead Directory

<CodeGroup>
```http GET /v1/leads/directory
GET /v1/leads/directory
GET /v1/leads/directory?hasContact=true
GET /v1/leads/directory?hasContact=false&requiredAction=modify
```
</CodeGroup>

### View Permissions

| Permission / Relationship | Visible Records |
| ------------------------- | --------------- |
| `system.admin`, `system.owner`, `crm.view`, `crm.manage` | All active organization leads |
| `team_crm.view` | MY leads plus leads with a stakeholder row for a team where the user has `team_crm.view` |
| `team_crm.manage` | MY leads plus leads with a stakeholder row for a team where the user has `team_crm.manage` |
| Direct lead stakeholder | Leads where the user is a non-deleted stakeholder |
| Lead creator | Leads created by the user |
| No matching permission | No additional records |

### Query Parameters

<AccordionGroup>
  <Accordion title="hasContact">
    - `hasContact=true` – Returns only leads whose person has a non-deleted contact
    - `hasContact=false` – Returns only leads whose person has no non-deleted contact (including archived contacts)
    
    <Info>
    Use `hasContact=false` for lead-to-contact conversion pickers.
    </Info>
  </Accordion>

  <Accordion title="requiredAction=modify">
    Narrows results to leads the caller can edit:

    | Permission / Relationship | Editable Directory Records |
    | ------------------------- | -------------------------- |
    | `system.admin`, `system.owner`, `crm.manage` | All active organization leads after other filters |
    | Direct lead stakeholder with `WRITE` or `FULL` | That lead |
    | `team_crm.manage` on a team with a lead stakeholder row at `WRITE` or `FULL` | That lead |
    | `crm.view`, `team_crm.view`, creator-only, direct `READ` stakeholder | Not editable by this filter |
  </Accordion>
</AccordionGroup>

<Tip>
The create-contact modal uses `GET /v1/leads/directory?hasContact=false&requiredAction=modify`, then fetches `GET /v1/leads/:id` after selection to hydrate the full person profile.
</Tip>

## Contact Directory

<CodeGroup>
```http GET /v1/contacts/directory
GET /v1/contacts/directory
GET /v1/contacts/directory?dealId=123
GET /v1/contacts/directory?companyId=456
GET /v1/contacts/directory?excludeCompanyId=789
```
</CodeGroup>

### View Permissions

| Permission / Relationship | Visible Records |
| ------------------------- | --------------- |
| `system.admin`, `system.owner`, `crm.view`, `crm.manage` | All active organization contacts |
| `sales.view`, `sales.manage` | Active contacts linked to at least one non-deleted deal via `deal_contact` |
| `team_crm.view` or `team_crm.manage` | MY contacts plus contacts connected to leads, deals, or contacts where those teams have stakeholder access |
| Direct contact stakeholder paths | Contacts linked through lead/person, deal/contact, or direct contact stakeholder paths |
| Creator-only | Contacts created by the current user |
| No matching permission | No additional records |

### Scoped Filters

<AccordionGroup>
  <Accordion title="dealId">
    Requires `sales.view`, `sales.manage`, or `DEAL ACCESS`.
    
    Queries from `deal_contact` joined to active contacts (same visibility anchor as `GET /deals/:id/contacts`). Rows include junction `isPrimary` and `role` for that deal.

    <Warning>
    Directory mapping must preserve the selected raw `contact_id` alias for `id` so the response never exposes the `deal_contact.id` junction primary key as the directory row id.
    </Warning>
  </Accordion>

  <Accordion title="companyId">
    Requires `crm.view`, `crm.manage`, or `COMPANY ACCESS`.
    
    Returns contacts linked to the specified company.
  </Accordion>

  <Accordion title="excludeCompanyId">
    Unscoped only. Applies normal unscoped visibility, requires company access, then removes contacts already linked to that company.
  </Accordion>
</AccordionGroup>

## Team Directory

<CodeGroup>
```http GET /v1/teams/directory
GET /v1/teams/directory
GET /v1/teams/directory?userId=123
```
</CodeGroup>

Returns minimal `TeamDirectoryDto` rows: `id`, `publicId`, `name`, `description`, `category`.

| Query | Access | Visible Records |
| ----- | ------ | --------------- |
| _(unscoped)_ | Any authenticated org member (no `@CheckAccess`) | All active org teams |
| `userId` | Any authenticated org member (no `@CheckAccess`) | Active teams where the given user has a non-deleted `team_membership` in the org |

<Check>
Use `GET /v1/teams/directory?userId=…` for user-scoped team pickers (e.g., CRM transfer modal when a recipient user is selected).
</Check>

<Warning>
Do **not** use `GET /v1/users/:userId/teams` for pickers — that route requires `system.admin`.
</Warning>

## User Directories

All user directory endpoints return minimal `UserDirectoryDto` rows containing:
- `id`
- `firstName`
- `lastName`
- `email`
- `phoneNumber`
- Optional avatar fields

<Info>
`email` and `phoneNumber` are included so org-internal surfaces (Cc/Bcc, listing portal-contact prefill) can resolve contact details without the full `GET /users` list.
</Info>

### General User Directory

<CodeGroup>
```http GET /v1/users/directory
GET /v1/users/directory
GET /v1/users/directory?search=john
```
</CodeGroup>

**Access:** Any authenticated org member (no `@CheckAccess`)  
**Returns:** All active org members

### Task Calendar Filter

<CodeGroup>
```http GET /v1/users/task-calendar-filter
GET /v1/users/task-calendar-filter
```
</CodeGroup>

**Access:** Any authenticated org member

| User Type | Visible Users |
| --------- | ------------- |
| Self | Self only |
| Org managers (`system.admin`, `system.owner`, `crm.manage`) | All active members |
| `team.admin` | Self + led-team members |
| Otherwise | Self only |

### Listing Agent Directory

<CodeGroup>
```http GET /v1/users/listing-agent-directory
GET /v1/users/listing-agent-directory
```
</CodeGroup>

This is the permission-scoped picker for **who a user may assign as a listing's agent / portal contact** (create/edit-listing Assignment step).

**Access:** Any authenticated org member

| User Type | Visible Users |
| --------- | ------------- |
| Self | Self only |
| Org managers (`system.admin`, `system.owner`, `crm.manage`, `real_estate.manage`) | All active members |
| `team.admin` | Self + led-team members |
| Otherwise | Self only |

<Note>
Tiering is implemented by `AccessService.listListingAgentTargetUserIdsInTransaction` — the same three-tier model as the task-calendar filter, but with `real_estate.manage` added to the org-wide manager set.
</Note>

<Tip>
The listing modal also surfaces two **always-available** special options independent of this scoped list:
- **Unit Manager** (the linked unit's manager)
- **Listing Publisher (You)** (the current user)

The same endpoint backs the listing "Lead routing" → **Specific user** picker, so the routing user picker has identical selection permissions to the agent picker.
</Tip>

### Team Members Directory

<CodeGroup>
```http GET /v1/teams/:teamId/members/directory
GET /v1/teams/abc123/members/directory
GET /v1/teams/abc123/members/directory?search=jane
```
</CodeGroup>

**Access:** Any authenticated org member (no `@CheckAccess`)  
**Returns:** Active users with a non-deleted membership on the given team

Uses the same `UserDirectoryDto` shape as `/users/directory` with optional `search` on name/email.

<Check>
Use `GET /v1/teams/:teamId/members/directory` for team-scoped user pickers (e.g., CRM transfer modal when a team is selected).
</Check>

<Warning>
Do **not** use `GET /v1/teams/:teamId/members` for pickers — that route returns full membership payloads and requires `system.admin` or team `ResourceAction.ACCESS`.
</Warning>
