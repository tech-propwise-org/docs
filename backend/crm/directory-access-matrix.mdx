## Overview

Directory endpoints are lightweight picker APIs designed for selection interfaces. They return minimal rows and enforce visibility inside the service query, rather than providing full list or kanban views.

<Note>
All directory endpoints support `page`, `limit`, and `search` query parameters for pagination and filtering.
</Note>

## Shared Rules

<Info>
Directory rows return **active records only** unless explicitly documented otherwise for a specific endpoint.
</Info>

All directory queries filter out:
- `is_deleted = true`
- `is_archived = true`

### Visibility Tiers

<AccordionGroup>
  <Accordion title="Org-wide visibility">
    All active records in the current organization, regardless of stakeholder relationships.
  </Accordion>
  
  <Accordion title="Team visibility">
    Records linked through `entity_stakeholder` rows for teams where the caller holds the specified team permission.
  </Accordion>
  
  <Accordion title="MY visibility">
    Direct user stakeholder paths and entity-specific self paths (creator, direct stakeholder).
  </Accordion>
</AccordionGroup>

## Lead Directory

**Endpoint:** `GET /v1/leads/directory`

### Base Visibility

| Permission / Relationship | Visible Records |
|--------------------------|-----------------|
| `system.admin`, `system.owner`, `crm.view`, `crm.manage` | All active organization leads |
| `team_crm.view` | MY leads plus leads with a stakeholder row for a team where the user has `team_crm.view` |
| `team_crm.manage` | MY leads plus leads with a stakeholder row for a team where the user has `team_crm.manage` |
| Direct lead stakeholder | Leads where the user is a non-deleted stakeholder |
| Lead creator | Leads created by the user |
| No matching permission or relationship | No additional records |

### Query Parameters

<Tabs>
  <Tab title="hasContact">
    Filters leads based on associated contact status:
    
    - `hasContact=true` — Returns only leads whose person has a non-deleted contact
    - `hasContact=false` — Returns only leads whose person has no non-deleted contact (including archived contacts)
    
    <Tip>
    Use `hasContact=false` for lead-to-contact conversion pickers.
    </Tip>
  </Tab>
  
  <Tab title="requiredAction">
    `requiredAction=modify` narrows rows to leads the caller can edit:
    
    | Permission / Relationship | Editable Directory Records |
    |--------------------------|---------------------------|
    | `system.admin`, `system.owner`, `crm.manage` | All active organization leads after other filters |
    | Direct lead stakeholder with `WRITE` or `FULL` | That lead |
    | `team_crm.manage` on a team with a lead stakeholder row at `WRITE` or `FULL` | That lead |
    | `crm.view`, `team_crm.view`, creator-only, direct `READ` stakeholder | Not editable by this filter |
  </Tab>
</Tabs>

### Usage Example

<CodeGroup>
```typescript Create Contact Modal
// Used in create-contact modal
GET /v1/leads/directory?hasContact=false&requiredAction=modify

// After selection, hydrate full person profile
GET /v1/leads/:id
```

```bash cURL
curl -X GET "https://api.example.com/v1/leads/directory?hasContact=false&requiredAction=modify" \
  -H "Authorization: Bearer YOUR_TOKEN"
```
</CodeGroup>

## Contact Directory

**Endpoint:** `GET /v1/contacts/directory`

### Base Visibility

| Permission / Relationship | Visible Records |
|--------------------------|-----------------|
| `system.admin`, `system.owner`, `crm.view`, `crm.manage` | All active organization contacts |
| `sales.view`, `sales.manage` | Active contacts linked to at least one non-deleted deal via `deal_contact` |
| `team_crm.view` or `team_crm.manage` | MY contacts plus contacts connected to leads, deals, or contacts where those teams have stakeholder access |
| Direct contact stakeholder paths | Contacts linked through lead/person, deal/contact, or direct contact stakeholder paths |
| Creator-only | Contacts created by the current user |
| No matching permission or relationship | No additional records |

### Scoped Filters

<Warning>
The `dealId` filter changes the query structure — it queries from `deal_contact` joined to active contacts, not a contact-first `EXISTS` filter.
</Warning>

<AccordionGroup>
  <Accordion title="dealId">
    **Requirements:** `sales.view`, `sales.manage`, or `DEAL ACCESS`
    
    Queries from `deal_contact` joined to active contacts (same visibility anchor as `GET /deals/:id/contacts`). Rows include junction `isPrimary` and `role` for that deal.
    
    <Note>
    Directory mapping must preserve the selected raw `contact_id` alias for `id` so the response never exposes the `deal_contact.id` junction primary key as the directory row id.
    </Note>
  </Accordion>
  
  <Accordion title="companyId">
    **Requirements:** `crm.view`, `crm.manage`, or `COMPANY ACCESS`
    
    Returns contacts linked to the specified company.
  </Accordion>
  
  <Accordion title="excludeCompanyId">
    **Scope:** Unscoped only
    
    **Requirements:** Company access
    
    Applies normal unscoped visibility, then removes contacts already linked to that company.
  </Accordion>
</AccordionGroup>

## Team Directory

**Endpoint:** `GET /v1/teams/directory`

Returns minimal `TeamDirectoryDto` rows: `id`, `publicId`, `name`, `description`, `category`.

| Query | Access | Visible Records |
|-------|--------|-----------------|
| _(unscoped)_ | Any authenticated org member (no `@CheckAccess`) | All active org teams |
| `userId` | Any authenticated org member (no `@CheckAccess`) | Active teams where the given user has a non-deleted `team_membership` in the org |

<Tip>
Use `GET /v1/teams/directory?userId=…` for user-scoped team pickers (e.g., CRM transfer modal when a recipient user is selected).

Do **not** use `GET /v1/users/:userId/teams` for pickers — that route requires `system.admin`.
</Tip>

## User Directories

All user directory endpoints return minimal `UserDirectoryDto` rows containing:
- `id`
- `firstName`
- `lastName`
- `email`
- `phoneNumber`
- Optional avatar fields

<Info>
`email` and `phoneNumber` are included so org-internal surfaces (Cc/Bcc, listing portal-contact prefill) can resolve contact details without the full `GET /users` list.
</Info>

### Permission Tiers

<Tabs>
  <Tab title="Task Calendar Filter">
    `GET /v1/users/task-calendar-filter` visibility:
    
    | User Role | Visible Users |
    |-----------|---------------|
    | Self | Always included |
    | Org managers (`system.admin`, `system.owner`, `crm.manage`) | All active members |
    | `team.admin` | Self + led-team members |
    | Other users | Self only |
  </Tab>
  
  <Tab title="Listing Agent Directory">
    `GET /v1/users/listing-agent-directory` visibility:
    
    | User Role | Visible Users |
    |-----------|---------------|
    | Self | Always included |
    | Org managers (`system.admin`, `system.owner`, `crm.manage`, `real_estate.manage`) | All active members |
    | `team.admin` | Self + led-team members |
    | Other users | Self only |
    
    <Note>
    Tiering is implemented by `AccessService.listListingAgentTargetUserIdsInTransaction`.
    </Note>
    
    **Special Options (Always Available):**
    - **Unit Manager** — The linked unit's manager
    - **Listing Publisher (You)** — The current user
    
    <Info>
    This endpoint also backs the listing "Lead routing" → **Specific user** picker, so the routing user picker has identical selection permissions to the agent picker.
    </Info>
  </Tab>
  
  <Tab title="Team Members">
    `GET /v1/teams/:teamId/members/directory` returns active users with a non-deleted membership on the given team.
    
    Returns the same `UserDirectoryDto` shape as `/users/directory`. Optional `search` parameter filters on name/email.
    
    <Tip>
    Use this endpoint for team-scoped user pickers (e.g., CRM transfer modal when a team is selected).
    
    Do **not** use `GET /v1/teams/:teamId/members` for pickers — that route returns full membership payloads and requires `system.admin` or team `ResourceAction.ACCESS`.
    </Tip>
  </Tab>
</Tabs>

## Listing Lead-Routing Team Picker

**Endpoint:** `GET /v1/teams/listing-routing-directory`

Returns minimal `TeamDirectoryDto` rows for listing lead-routing configuration.

<Check>
This endpoint provides teams selectable for automated lead routing in listing configurations.
</Check>

<Note>
Permission and visibility rules for this endpoint determine which teams can be selected as routing targets for listing-generated leads.
</Note>

## Best Practices

<Steps>
  <Step title="Choose the right endpoint">
    Use directory endpoints for pickers and selection interfaces. Avoid using full list endpoints (e.g., `GET /users/:userId/teams`) which require elevated permissions.
  </Step>
  
  <Step title="Apply appropriate filters">
    Use scoped filters (`dealId`, `companyId`, `hasContact`) to narrow results to the specific context of your picker.
  </Step>
  
  <Step title="Handle permissions gracefully">
    Check permissions client-side before making requests. Directory endpoints return only what the user can access — empty results don't indicate an error.
  </Step>
  
  <Step title="Preserve ID mapping">
    When working with junction tables (e.g., `deal_contact`), ensure your directory mapping preserves the correct entity ID, not the junction table's primary key.
  </Step>
</Steps>
