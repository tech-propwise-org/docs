## Overview

The Distribution Module automates lead assignment within organizations. When a new lead is created, the system evaluates org-defined rules to automatically assign the lead to the most appropriate agent — based on lead attributes, agent availability, language compatibility, and capacity.

<Info>
Module Path: `src/modules/crm/distribution/`
Status: Active — fully implemented
</Info>

### Design Principles

| Principle | Decision |
|-----------|----------|
| Async distribution | `createLead()` emits `LEAD_CREATED`; a pg-boss worker handles distribution — lead creation is never blocked |
| Stakeholder system reuse | Distribution creates `EntityStakeholder` records via `EntityStakeholderService`, not a new paradigm |
| First-match-wins rules | Rules are evaluated top-to-bottom by priority; the first matching rule wins |
| Idempotency | Distribution engine checks for existing stakeholders or pending offers before running |
| No retroactive distribution | Existing leads are unaffected when rules are created; only new leads trigger distribution |
| pg-boss scheduling | Distribution queue uses pg-boss for reliability and retry guarantees |
| RLS compliance | All entities carry `organization_id` for row-level security |

### Distribution Paths

The engine supports two execution paths:

<Tabs>
<Tab title="Path A - Org-level">
**Org-level distribution** (`runDistribution`): triggered when a lead enters the org with no team context. Evaluates org-scoped rules, applies the org default method, and can bridge to Path B if a rule or default method routes to a team that has `distributionEnabled = true`.
</Tab>
<Tab title="Path B - Team-level">
**Team-level distribution** (`runTeamDistribution`): triggered directly when:
- A lead is created with a `teamId` in the event payload (team pool assignment)
- Path A determines the lead belongs to an auto-distributing team
- Idempotency check finds a single team-only stakeholder with auto-distribute enabled

Path B evaluates team-scoped rules, uses team settings (with org fallback for capacity), and logs the team FK on the resulting `DistributionLog` record.
</Tab>
</Tabs>

### Component Responsibilities

| Component | Responsibility |
|-----------|----------------|
| **DistributionEngine** | Orchestrator: receives a lead, evaluates rules, selects agent, creates assignment. Supports Path A (org) and Path B (team). |
| **RuleEvaluator** | Evaluates rule conditions against lead data; returns first matching rule |
| **LanguageMatcher** | Filters and ranks agents by language compatibility with the lead's person |
| **AgentSelector** | Applies the distribution method (round-robin, weighted, weighted-round-robin, direct) to the filtered agent pool |
| **AgentAvailabilityService** | Checks agent capacity, business hours, leave status. Two-phase capacity enforcement with advisory locks. |
| **UserStatusService** | Pre-filters candidate agents to only those with ONLINE status |
| **DistributionListener** | Listens for `LEAD_CREATED` events and enqueues pg-boss jobs |
| **DistributionJobHandler** | pg-boss worker that processes distribution jobs |

<Warning>
**Master toggle behavior:**
- `distributionEnabled = false` (new-org default): Engine is off. `DistributionListener` and `LeadImportService` skip enqueue entirely — leads go to pool, no pg-boss jobs created.
- `distributionEnabled = true`: Engine is active. When toggled from `false` → `true` in `DistributionSettingsService.update()`, if `defaultMethod` is still `POOL` it is auto-upgraded to `ROUND_ROBIN` for a smooth first-run experience.
</Warning>

### Rule Conditions - Supported Fields

| Field | Operator(s) | Example Value |
|-------|-------------|---------------|
| `leadSource` | `eq`, `in` | `'WEBSITE'`, `['WEBSITE', 'REFERRAL']` |
| `temperature` | `eq`, `in` | `'HOT'` |
| `language` | `eq` | `'ar'` (matched against `person.preferredLanguage`) |
| `budget` | `gte`, `lte`, `between` | `500000` |
| `tags` | `contains` | `['vip']` |
| `sourceChannel` | `eq`, `in` | `'WHATSAPP'` |
| `intent` | `eq` | `'BUY'` |
| `area` | `eq`, `in`, `contains` | `'Dubai Marina'`, `['JBR', 'Downtown Dubai']` |

<Info>
All string-based condition fields use **case-insensitive matching**. The `area` field requires data from `LeadPropertyInterest.preferredAreas[]` — the engine pre-loads the lead's property interests before calling the evaluator.
</Info>

### Scope & Team Rules

- **Org-level rules** (`scope = ORGANIZATION`): `team` is null. Evaluated during Path A.
- **Team-scoped rules** (`scope = TEAM`): `team` is set. Only evaluated during Path B for the matching team.

## Distribution Engine

### Core Engine Flow

<Steps>
<Step title="Lead Event Processing">
The `DistributionListener` receives `LEAD_CREATED` events and enqueues pg-boss jobs for async processing.
</Step>

<Step title="Path Selection">
The engine determines whether to use Path A (org-level) or Path B (team-level) based on lead context and existing stakeholders.
</Step>

<Step title="Rule Evaluation">
Rules are evaluated in priority order. The first matching rule determines the distribution method and recipients.
</Step>

<Step title="Agent Filtering">
Available agents are filtered by status, capacity, business hours, and language compatibility.
</Step>

<Step title="Agent Selection">
The configured distribution method (round-robin, weighted, etc.) selects the final agent from the filtered pool.
</Step>

<Step title="Assignment Creation">
An `EntityStakeholder` record is created to assign the lead to the selected agent.
</Step>
</Steps>

### Path A - Organization Level Distribution

```typescript
async runDistribution(payload: DistributionJobPayload): Promise<DistributionResult> {
  // 1. Load lead and check for existing assignments
  // 2. Evaluate org-scoped rules
  // 3. Apply matching rule or fall back to org default method
  // 4. If result points to auto-distributing team, bridge to Path B
  // 5. Otherwise assign directly or send to pool
}
```

### Path B - Team Level Distribution

```typescript
async runTeamDistribution(payload: TeamDistributionJobPayload): Promise<DistributionResult> {
  // 1. Load lead and team context
  // 2. Evaluate team-scoped rules
  // 3. Apply matching rule or team default method
  // 4. Use team settings for capacity/business hours
  // 5. Create assignment with team context logged
}
```

<Warning>
Jobs are retried up to 3 times with exponential backoff. After final failure, the lead remains in the pool and an alert is logged.
</Warning>

### Row Level Security (RLS)

All distribution entities implement RLS policies based on `organization_id`:

<CodeGroup>
```sql Distribution Settings RLS
CREATE POLICY distribution_settings_access ON distribution_settings
  FOR ALL TO authenticated
  USING (organization_id = get_user_organization_id());
```

```sql Team Distribution RLS
CREATE POLICY team_distribution_access ON team_distribution_settings
  FOR ALL TO authenticated
  USING (
    organization_id = get_user_organization_id()
    AND (
      -- Can access if user is member of the team
      team_id IN (SELECT team_id FROM team_members WHERE user_id = auth.uid())
      -- Or has org-level distribution permissions
      OR has_permission('DISTRIBUTION_SETTINGS_VIEW')
    )
  );
```

```sql Distribution Rules RLS
CREATE POLICY distribution_rules_access ON distribution_rules
  FOR ALL TO authenticated
  USING (
    organization_id = get_user_organization_id()
    AND is_deleted = false
  );
```
</CodeGroup>

## Edge Case Handling

### Common Scenarios

<AccordionGroup>
<Accordion title="No Available Agents">
**Scenario:** All agents are at capacity or offline during distribution

**Handling:**
1. If `respectBusinessHours` is enabled and outside hours, apply `outsideHoursAction`
2. Otherwise, route to pool with `reason: NO_AVAILABLE_AGENTS`
3. Trigger pool alert if threshold exceeded
4. Log failure reason for analytics
</Accordion>

<Accordion title="Rule Configuration Errors">
**Scenario:** Rule references deleted agents or invalid team

**Handling:**
1. Skip invalid recipients during agent selection
2. Fall back to next rule or default method
3. Log warning with rule ID and issue details
4. Continue with reduced recipient pool
</Accordion>

<Accordion title="Business Hours Edge Cases">
**Scenario:** Lead arrives exactly at business hours boundary

**Handling:**
1. Use timezone-aware comparison with 1-minute tolerance
2. If within tolerance, treat as business hours
3. Log boundary conditions for analysis
4. Respect `outsideHoursAction` for clear violations
</Accordion>

<Accordion title="Concurrent Distribution Attempts">
**Scenario:** Multiple distribution jobs for the same lead

**Handling:**
1. Check for existing assignments at start of distribution
2. Skip distribution if lead already has active stakeholder
3. Use advisory locks for critical capacity updates
4. Return early success if assignment already exists
</Accordion>
</AccordionGroup>

## Performance & Scaling

### Optimization Strategies

<Steps>
<Step title="Database Indexes">
Key indexes for performance:
- `idx_distribution_rules_org_priority` on `(organization_id, priority, is_active)`
- `idx_agent_availability_org_user` on `(organization_id, user_id)`
- `idx_distribution_log_lead_created` on `(lead_id, created_at)`
</Step>

<Step title="Caching Strategy">
- Distribution settings cached in Redis with 5-minute TTL
- Agent availability cached with 30-second TTL
- Business hours cached with 1-hour TTL
</Step>

<Step title="Queue Management">
- pg-boss team concurrency set to 2 for distribution queue
- Team size of 5 for parallel processing
- Dead letter queue for failed jobs after retries
</Step>

<Step title="Connection Pooling">
- Dedicated read replica for analytics queries
- Connection pooling for high-frequency availability checks
- Prepared statements for common queries
</Step>
</Steps>

### Scaling Considerations

<Warning>
**High Volume Scenarios:**
- Consider increasing pg-boss team size for organizations with >1000 leads/day
- Implement lead batching for bulk imports
- Use read replicas for analytics and availability queries
- Monitor database connection pool utilization
</Warning>
