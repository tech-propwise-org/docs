## Stakeholder Sync Feature

### Overview

When working with deals that originate from leads, stakeholder assignments should maintain consistency. The `StakeholderSyncService` automatically syncs stakeholder assignments between deals and their originating leads.

### Design Principles

1. **One-Way Primary Flow**: Leads inherit to deals at deal creation (existing behavior)
2. **Bidirectional Sync**: Deal stakeholder changes now sync back to lead
3. **Safe by Default**: Multiple safety checks prevent unwanted removals
4. **Non-Blocking**: Sync failures log errors but don't fail the primary operation

### Use Case 1: Assign to Deal → Auto-Assign to Lead

**Scenario:** When you assign a stakeholder to a deal, they should also be able to see the originating lead.

**API Call:**

```bash
POST /deals/{dealId}/stakeholders
Content-Type: application/json

{
  "userId": "agent-b-uuid",
  "commissionPercentage": 30,
  "accessLevel": "WRITE"
}
```

**What Happens:**

<Steps>
<Step title="Primary assignment">
Agent B is assigned to Deal #1 (30% commission, WRITE access)
</Step>

<Step title="Lead sync check">
System checks: Does Deal #1 have an originating lead?
</Step>

<Step title="Auto-sync to lead">
If yes AND lead exists AND lead not archived:
- Agent B is also assigned to Lead #1 (0% commission, WRITE access, observer)
- Returns: `{ success: true, syncedToLead: true, leadId: "lead-uuid" }`
</Step>
</Steps>

<Note>
When assigning to a deal, the system automatically creates a corresponding lead stakeholder with 0% commission (observer status) and the same access level as the deal assignment.
</Note>

**Sync Settings:**

| Property               | Deal Value                 | Lead Value                |
| ---------------------- | -------------------------- | ------------------------- |
| `commissionPercentage` | As specified (e.g., 30%)   | **0%** (observer)         |
| `accessLevel`          | As specified (e.g., WRITE) | **Same as deal**          |
| `isPrimary`            | As specified               | **false** (never primary) |

**Skip Conditions:**

| Condition            | Action    | Reason                         |
| -------------------- | --------- | ------------------------------ |
| No originating lead  | Skip sync | Nothing to sync to             |
| Lead is archived     | Skip sync | Can't modify archived entities |
| User already on lead | Skip sync | No duplication needed          |

### Use Case 2: Remove from Deal → Auto-Remove from Lead

**Scenario:** When you remove a stakeholder from a deal, they should also be removed from the lead if they no longer need access.

**API Call:**

```bash
DELETE /deals/{dealId}/stakeholders/{stakeholderId}
```

**What Happens:**

<Steps>
<Step title="Remove from deal">
Stakeholder removed from Deal #1
</Step>

<Step title="Safety check evaluation">
System checks: Should we also remove from originating lead?
</Step>

<Step title="Lead removal (if safe)">
If all safety checks pass:
- Stakeholder removed from Lead #1
- Other lead stakeholder commissions auto-adjusted to 100%
- Returns: `{ success: true, syncedToLead: true, leadId: "lead-uuid" }`
</Step>
</Steps>

**Safety Checks (ALL must pass):**

| #   | Check                                       | Reason                           |
| --- | ------------------------------------------- | -------------------------------- |
| 1   | Lead exists and not archived                | Can't modify archived leads      |
| 2   | User has **0% commission** on lead          | Protect earned commission rights |
| 3   | User is **NOT primary** on lead             | Protect primary ownership        |
| 4   | This is user's **ONLY deal** from that lead | Keep access if other deals exist |

**If ANY check fails:** Stakeholder removed from deal only, lead unchanged

**Commission Auto-Adjustment:**

If removing would break 100% total on lead, the system auto-adjusts:

```
BEFORE:
- Agent A: 50%
- Agent B: 0% ← Removing
- Agent C: 20%
Total: 70%

AFTER (auto-adjusted to 100%):
- Agent A: 71.43% (50 × 100/70)
- Agent C: 28.57% (20 × 100/70)
Total: 100% ✓

Formula: new% = old% × (100 / current_total)
```

### Sync Response Format

Both create and remove operations return:

```typescript
interface StakeholderSyncResultDto {
  success: boolean; // Operation succeeded
  syncedToLead: boolean; // Whether sync occurred
  leadId?: string; // ID of lead (if applicable)
  message?: string; // Optional status message
}
```

## Team Commission Budget

### Overview

A team's `commissionPercentage` on an entity stakeholder acts as a **live budget** that can be sub-allocated to individual team members. This enables team leaders to manage how commission is distributed within their team without needing org-level permissions.

### Design Principles

1. **Team as Budget Holder**: The team stakeholder's `commissionPercentage` is the total pool available for team member allocation
2. **Sub-Allocation**: Team managers assign commission from the team budget to individual members. The team's percentage decreases as members receive allocations
3. **Unallocated Budget at Close**: If a deal closes with remaining team budget (team stakeholder has `commissionPercentage > 0`), that amount is paid to the **team** directly
4. **Team Pool Freeze**: When a team-only stakeholder (has `team`, no `user`) holds primary status, the entity is blocked from modification until a team member claims or is assigned to it

### Team Budget Lifecycle

<Steps>
<Step title="Lead assigned to team">
```
Team A stakeholder: { 
  team: "Team A", 
  user: null, 
  commission: 100%, 
  isPrimary: true,
  originalTeamCommission: 100 
}
```
</Step>

<Step title="Team manager assigns first member">
```
Team A stakeholder: { 
  commission: 60%, 
  isPrimary: false,
  originalTeamCommission: 100 
}
Agent X stakeholder: { 
  user: "Agent X", 
  team: "Team A", 
  commission: 40%, 
  isPrimary: true,
  originalTeamCommission: 100 
}
```
</Step>

<Step title="Team manager assigns second member">
```
Team A stakeholder: { commission: 30% }   ← remaining budget
Agent X stakeholder: { commission: 40% }
Agent Y stakeholder: { commission: 30% }
```
</Step>

<Step title="Deal closes - commission payments">
- Agent X: 40% → paid to user
- Agent Y: 30% → paid to user  
- Team A: 30% → paid to TEAM (unallocated budget)
</Step>
</Steps>

<Info>
`originalTeamCommission` on each member's stakeholder records the team's total allocation (100% in this case), allowing the system to validate that the sum of team member commissions never exceeds the original team budget.
</Info>

### Claim from Team Pool

When a team member claims a lead from the team pool, the team stakeholder is **preserved** as a budget placeholder (set to `isPrimary: false`, `commissionPercentage: 0`) rather than being deleted. This allows the team to retain its position on the entity for future budget allocation.

The claiming user's stakeholder is created with `originalTeamCommission` set to the team's commission at the time of claim, enabling team budget enforcement even after subsequent reassignments.

```
BEFORE claim:
└── Team A: { commission: 100%, isPrimary: true, originalTeamCommission: 100 }

AFTER claim by Agent X (same team):
├── Team A: { commission: 0%, isPrimary: false, originalTeamCommission: 100 }
└── Agent X: { commission: 100%, isPrimary: true, originalTeamCommission: 100 }
```

### Commission Aggregation

Current users can hold **multiple** stakeholder records on the same entity (personal + team stakes). At commission payment time, the system aggregates all records per user:

```sql
SELECT user_id, SUM(commission_percentage) as total_commission
FROM entity_stakeholder 
WHERE entity_type = 'deal' AND entity_id = :deal_id 
GROUP BY user_id
```

## Business Rules

### Core Invariants

<AccordionGroup>
<Accordion title="Stakeholder assignment rules">
- Only one stakeholder can have `is_primary=true` per entity
- At least one primary stakeholder must exist at all times
- Commission total must sum to exactly 100% (or 0% for all observers)
- Stakeholders with 0% commission cannot be set as `isPrimary=true`
- Cannot remove primary stakeholder while others exist
- Remaining commissions proportionally scaled on removal
</Accordion>

<Accordion title="Team budget constraints">
- Sum of team member commissions cannot exceed `originalTeamCommission`
- Entities with team-only primary stakeholder are read-only until claimed
- Remaining team budget is paid to team at deal close
- Team stakeholder preserved as budget placeholder after claim
</Accordion>

<Accordion title="Sync safety rules">
- Only sync to non-archived leads with existing originating relationship
- Only remove from lead if user has 0% commission and is not primary
- Preserve access levels across sync operations
- Only remove from lead if user has no other deals from same lead
</Accordion>
</AccordionGroup>

### Validation Rules

| Rule | Enforcement | Error Code |
| --- | --- | --- |
| Commission total = 100% | On batch update | `INVALID_COMMISSION_TOTAL` |
| One primary per entity | On assignment | `MULTIPLE_PRIMARY_STAKEHOLDERS` |
| Team budget limit | On allocation | `TEAM_BUDGET_EXCEEDED` |
| Primary guard | On removal | `CANNOT_REMOVE_PRIMARY` |

## Access Control

### Permission Requirements

| Action | Required Permission | Notes |
| --- | --- | --- |
| View stakeholders | `stakeholders.read` | Independent of entity access |
| Assign stakeholder | `stakeholders.create` | Can assign to any accessible entity |
| Modify stakeholder | `stakeholders.update` | Includes commission and access changes |
| Remove stakeholder | `stakeholders.delete` | Cannot remove last stakeholder |
| Claim from team | `stakeholders.create` + team membership | Must be member of claiming team |
| Team budget allocation | `stakeholders.update` + team manager role | Team manager or above |

### Access Level Resolution

For users with multiple stakeholder records on the same entity, the system resolves to the **highest** access level:

```
User has two stakeholder records:
├── Personal stake: READ access
└── Team stake: WRITE access
Result: User has WRITE access to entity
```

## entity_stakeholder

```
entity_stakeholder
├── entity_type (lead | deal)
├── entity_id
├── user_id → User (nullable — null for team-only stakeholders)
├── team_id → Team (nullable)
├── is_primary (only one per entity)
├── access_level (READ | WRITE | FULL, default: READ)
├── Commission:
│   ├── commission_percentage (e.g., 50.00%)
│   └── original_team_commission (nullable — records team allocation at time of claim)
├── Inheritance tracking:
│   ├── inherited_from_stakeholder_id (nullable)
│   └── inherited_from_lead_id (nullable)
├── assigned_at, assigned_by_id
├── notes
```

### Access levels

| Level | Description                  | Typical use                         |
| ----- | ---------------------------- | ----------------------------------- |
| READ  | View only                    | Observers, auto-synced stakeholders |
| WRITE | View and edit                | Collaborating agents                |
| FULL  | View, edit, delete, transfer | Primary owner, managers             |

<Note>
  Primary stakeholders default to FULL access unless explicitly specified. Only
  one stakeholder can have `is_primary = true` per entity.
</Note>

## Dual-record model

A user can hold **multiple** stakeholder rows on the same entity with different team contexts:

- One row with `team_id = NULL` (personal/global-pool stake)
- One row per team they claim from (e.g., `team_id = Team A`)

The unique index `uq_stakeholder_user_team_entity` enforces `(entity_type, entity_id, user_id, team_id)`. PostgreSQL treats NULLs as distinct, so `(lead-1, ahmed, NULL)` and `(lead-1, ahmed, team-A)` are both allowed.

At commission payment time, aggregate all rows per user: `SUM(commissionPercentage) WHERE userId = X AND entityId = Y`.

**Primary handling with dual records:**

- Claiming from a team pool while holding a personal stake: personal stake demoted to `isPrimary=false`, new team-scoped claim becomes primary.
- Team-scoped stakeholder removed and user has a personal stake: personal stake auto-promoted to `isPrimary=true`.
- Access control resolves the **highest** `accessLevel` across all of a user's stakeholder rows.

## entity_stakeholder_history

Tracks COMPLETED stakeholder assignment states. History records represent what a stakeholder **was** (completed states), not what they changed from/to. The current state lives on `EntityStakeholder`.

```
entity_stakeholder_history (completed states only)
├── entity_type, entity_id
├── stakeholder_id (the stakeholder record)
├── user_id (nullable), team_id
├── COMPLETED STATE (what they WERE):
│   ├── commission_percentage
│   ├── is_primary
│   └── access_level (READ | WRITE | FULL)
├── Duration tracking:
│   ├── started_at (when this state began)
│   └── duration_seconds (calculated: created_at - started_at)
├── What ended this state:
│   ├── action (commission_changed | removed | transferred | inherited | primary_changed)
│   ├── transfer_id (if from transfer)
│   └── transferred_to_user_id (if transferred)
├── notes
├── performed_by_id, created_at (also = when state ended)
```

**When to record:**

- Commission change → record the OLD commission state, then update stakeholder
- Primary change → record the OLD primary state, then update stakeholder
- Removal → record final state with action: `removed`
- Transfer → record OLD owner state with action: `transferred`, `transfer_id`, `transferred_to_user_id`

<Accordion title="Worked example: stakeholder history flow">
```
User A assigned as primary (50%) at T1:
  - EntityStakeholder: { user: A, commission: 50%, isPrimary: true, accessLevel: FULL, assignedAt: T1 }
  - History: (empty — no history for new assignments)

User A's commission changes to 30% at T2:

1. CREATE history: { user: A, commission: 50%, isPrimary: true, accessLevel: FULL,
   startedAt: T1, durationSeconds: T2-T1, action: COMMISSION_CHANGED }
2. UPDATE stakeholder: { commission: 30% }

User A is removed at T3:

1. CREATE history: { user: A, commission: 30%, isPrimary: false, accessLevel: FULL,
   startedAt: T2, durationSeconds: T3-T2, action: REMOVED }
2. Soft-delete stakeholder

```
</Accordion>

## commission_payment

Tracks actual commission payments when deals close.

```
commission_payment
├── deal_id → Deal
├── stakeholder_id → entity_stakeholder
├── user_id → User (nullable — null for team-only payments)
├── team_id → Team (nullable — set for team-only payments)
├── percentage (commission percentage snapshot at close)
├── base_amount (total commission pool: deal.value × deal.commissionRate / 100)
├── amount (calculated: base_amount × percentage / 100)
├── currency (default: USD)
├── status (pending, approved, paid, cancelled, on_hold)
├── expected_payment_date
├── payment_method, payment_reference
├── paid_at
├── approved_by, approved_at
├── notes
```

<Note>
Either `user_id` OR `team_id` must be set (validated in service). If a team stakeholder has remaining `commissionPercentage > 0` when a deal closes, the commission is paid to the **team**, representing unallocated budget.
</Note>

<Warning>
**Recipient-only approval:** Only the payment recipient (the user whose `user_id` matches the payment) can approve their own `PENDING → APPROVED` transition. Other status transitions are not restricted to the recipient.
</Warning>

### Status transitions

| From | Allowed transitions |
| --- | --- |
| PENDING | APPROVED, ON_HOLD, CANCELLED |
| APPROVED | PAID, ON_HOLD, CANCELLED |
| ON_HOLD | PENDING, CANCELLED |
| PAID | (final — cannot change) |
| CANCELLED | (final — cannot change) |

### Commission calculation

```typescript
function calculateCommissions(deal: Deal) {
  const totalCommission = deal.value * (deal.commissionRate / 100);

  const stakeholders = await getStakeholders('deal', deal.id);

  for (const stakeholder of stakeholders) {
    await createCommissionPayment({
      dealId: deal.id,
      stakeholderId: stakeholder.id,
      user: stakeholder.user?.id,
      team: stakeholder.team?.id,
      percentage: stakeholder.commissionPercentage,
      baseAmount: totalCommission,
      amount: totalCommission * (stakeholder.commissionPercentage / 100),
      status: 'pending',
    });
  }
}
```

If a team stakeholder has unallocated budget (e.g., team has 30% remaining), the commission payment targets the team, not any individual user. This allows organizations to handle team-level commission distribution outside the CRM.

<Accordion title="Worked example: deal closure with $1,000,000 deal">
```
Deal #1 moves to closedWon stage:
├── isClosed = true, isWon = true, closedAt = NOW()
├── value: $1,000,000
├── commissionRate: 2% (stored on deal)
└── totalCommission: $20,000 (calculated)

Stakeholders:
├── Agent A: 30% → $6,000
└── Agent B: 70% → $14,000

INSERT INTO commission_payment:

Record 1:
├── stakeholder_id: Agent A's stakeholder
├── percentage: 30%
├── base_amount: $20,000 (the commission pool)
├── amount: $6,000
└── status: 'pending'

Record 2:
├── stakeholder_id: Agent B's stakeholder
├── percentage: 70%
├── base_amount: $20,000
├── amount: $14,000
└── status: 'pending'

```
</Accordion>

<Warning>
When a deal is reopened (terminal → non-terminal stage), all non-PAID commission payments are cancelled (`PENDING` or `APPROVED` → `CANCELLED`). If any payments are already `PAID`, the reopen is blocked.
</Warning>

## Query patterns

### Assignment queries

| Entity | How to get assignment |
| --- | --- |
| **Lead** | `SELECT * FROM entity_stakeholder WHERE entity_type='lead' AND entity_id=:lead_id` |
| **Deal** | `SELECT * FROM entity_stakeholder WHERE entity_type='deal' AND entity_id=:deal_id` |
| **Contact** | Via Person's leads: `JOIN lead ON l.person_id = :person_id` then join `entity_stakeholder` |
| **Person** | No direct assignment — visibility derived from leads |

### Full stakeholder query

```sql
SELECT es.*, u.name as user_name, t.name as team_name
FROM entity_stakeholder es
JOIN user u ON es.user_id = u.id
LEFT JOIN team t ON es.team_id = t.id
WHERE es.entity_type = :type AND es.entity_id = :id AND es.is_deleted = false
ORDER BY es.is_primary DESC, es.commission_percentage DESC;
```

### Permission-aware lead visibility

```sql
SELECT l.* FROM lead l
WHERE EXISTS (
  SELECT 1 FROM entity_stakeholder es
  WHERE es.entity_type = 'lead' AND es.entity_id = l.id
    AND es.user_id = :user_id AND es.is_deleted = false
) AND l.is_deleted = false AND l.is_archived = false;
```

### Team budget tracking

```sql
SELECT 
  es.team_id,
  SUM(CASE WHEN es.user_id IS NULL THEN es.commission_percentage ELSE 0 END) as unallocated_budget,
  SUM(CASE WHEN es.user_id IS NOT NULL THEN es.commission_percentage ELSE 0 END) as allocated_budget,
  MAX(es.original_team_commission) as original_budget
FROM entity_stakeholder es
WHERE es.entity_type = :type AND es.entity_id = :id AND es.team_id IS NOT NULL
GROUP BY es.team_id;
```

## Quick Reference

### Commission Calculation Formula

```typescript
// Individual commission amount
amount = dealValue × (dealCommissionRate / 100) × (stakeholderPercentage / 100)

// Auto-adjustment on removal
newPercentage = oldPercentage × (100 / sumOfRemainingPercentages)

// Team budget validation
totalMemberCommission ≤ originalTeamCommission
```

### Key Validation Rules

<AccordionGroup>
<Accordion title="Stakeholder assignment">
- Exactly one primary per entity
- Commission total must equal 100% (or 0% for all observers)
- Cannot set observer (0% commission) as primary
- Team members can only claim from their own team pools
</Accordion>

<Accordion title="Team budget">
- Team member allocations cannot exceed original team budget
- Unallocated budget is paid to team at deal close
- Team stakeholder preserved as placeholder after claim
- Team pool freeze prevents entity modification
</Accordion>

<Accordion title="Stakeholder sync">
- Only sync to non-archived leads with originating relationship
- Auto-sync creates observers (0% commission) on leads
- Safety checks protect existing commission and primary status
- Sync failures are logged but don't block operations
</Accordion>
</AccordionGroup>
