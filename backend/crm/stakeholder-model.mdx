## Stakeholder Sync Feature (v2 — Entity Independence)

### Overview

Leads and deals **no longer share stakeholder state** after deal creation. Source leads can prefill a new deal's stakeholder list, but the persisted deal stakeholders are deal-owned and later lead/deal stakeholder edits do not automatically propagate.

### Design Principles

1. **Deal-Owned Stakeholders**: Deal stakeholders are managed independently from lead stakeholders after creation. Leads provide creation-time suggestions only.
2. **1:1 Active Deal per Lead**: A lead can be referenced by at most one active deal contact (deal not archived, not closed-lost). A new active deal participant can only use that lead when no active deal already references it.
3. **No Bidirectional Stakeholder Sync**: Lead and deal stakeholder changes do not mirror automatically. This prevents buyer-side and seller-side leads from receiving each other's deal stakeholders.
4. **100% Commission Floor**: Deals must maintain 100% total commission. A lead without an active deal can have 0% (not yet configured).
5. **Inactive Deals Excluded**: Archived or closed-lost deals are excluded from active-lead conflict checks.
6. **Deal Creation Gate**: A deal can be created from an active source lead/contact only when the requester can use that source at write level. `crm.manage` can use any active org source record; otherwise the requester needs `WRITE`/`FULL` stakeholder access directly or through a `team_crm.manage` team context. `crm.view` / `team_crm.view` are visibility-only and do not allow deal creation from that source.
7. **Deal Stakeholder Invariant**: A deal must always have at least one stakeholder. Deal stakeholder sync rejects `stakeholders: []`; deals do not have an unassign-to-pool operation.
8. **Transfer Permission Parity**: Lead and deal `canTransfer` response permissions follow the same actor model: the primary user stakeholder can initiate transfer, and a team manager can initiate transfer when their managed team holds the primary team-only row.
9. **DONE Stage Auto-Transitions** (v3.5): When a deal reaches a terminal stage (CLOSED_WON or CLOSED_LOST), every lead referenced by `deal_contact.originating_lead_id` moves to the `DONE` stage (unless already terminal or archived). Conversely, reopening the deal moves those participant-originating leads back to `QUALIFIED`.

### How Stakeholder Updates Work

When `PUT /leads/:id/stakeholders` or `PUT /deals/:id/stakeholders` is called:

<Steps>
<Step title="Apply changes and emit notifications">
The sync endpoint applies all add/update/remove changes to the target entity and emits notifications based on the final transition:
- `0 → N` lead stakeholders emits `LEAD_ASSIGNED` and suppresses per-row "stakeholder added" notifications
- `N → 0` lead stakeholders emits `LEAD_UNASSIGNED`, cancels pending transfers, and suppresses per-row "stakeholder removed" notifications
- `N → 0` is invalid for deals because deals must retain at least one stakeholder
- Other diffs emit fine-grained stakeholder added/removed/commission-changed notifications
</Step>

<Step title="No automatic sync">
After changes are committed, no linked lead/deal entity is automatically mutated. `syncToLinkedEntity()` is retained as an internal no-op guard for older call paths, but lead/deal stakeholder mutations are intentionally independent.
</Step>
</Steps>

### Deal Creation with Stakeholders

When creating a deal from one or more participant leads, the frontend can pre-fill the stakeholder list from all selected leads and lets the user review the merged list. The backend persists only the final deal-owned stakeholder payload.

Before a participant source can be selected, the frontend calls deal-owned availability endpoints (`GET /deals/available-leads`, `GET /deals/available-contacts`). These endpoints are mutation-scoped and return only source records the requester can use for deal creation. `POST /deals` re-checks the same rule for submitted lead/contact IDs so callers cannot bypass the picker.

**Branch 1 — Explicit stakeholders provided (`dto.stakeholders[]` non-empty):**

<Steps>
<Step title="Validate and create">
Validates that commission totals 100% before creating.
</Step>

<Step title="Persist stakeholders">
Creates all deal stakeholder rows via `createStakeholderBatchInTransaction()`.
</Step>

<Step title="No automatic sync">
Does not mutate any source lead stakeholders. The deal's stakeholder state is independent.
</Step>
</Steps>

**Branch 2 — No explicit stakeholders:**

<Steps>
<Step title="Create default stakeholder">
Creates a single primary stakeholder (creator, 100% commission).
</Step>

<Step title="No source sync">
No source lead stakeholders are copied server-side.
</Step>
</Steps>

### Deal Creation with a Pending Unit Access Request

When a deal is created with an `InventoryUnit` managed by another agent (`unit.unitManager.id !== currentUser.id`), `DealService.create()` defers the unit link and issues a `Pending` `UnitAccessRequest` in the same transaction.

**Stakeholder-side rules:**

- The deal's requested stakeholders are created exactly as in the normal path (explicit `dto.stakeholders[]` or creator-as-100%-primary). The pending-link path then adds one extra request-owned unit-manager observer row.
- The deal creator (current user) **must** end up as a primary user-stakeholder of the new deal — otherwise the access request has no requester to attribute and `DealService.create()` rejects the payload.
- The proposed `unitManagerCommissionPercentage` must be `<=` the requesting primary stakeholder's commission. The check runs at deal-create time (so the deal never lands in a state that approval cannot service) **and** again at approval time on the **live** stakeholder row, in case stakeholders were re-synced between request and approval.
- Request creation adds the unit manager as a request-owned `DEAL` stakeholder with `isPrimary=false`, `commissionPercentage=0`, and `accessLevel=READ`. `UnitAccessRequest.unitManagerStakeholder` points at this row. Each pending request gets its own observer row, even if the same unit manager has another pending request on the same deal for a different inventory unit. While the request is `Pending`, no user may edit, change access level, change commission, or remove this row through stakeholder management; the request must be approved, rejected, or cancelled.
- If the inventory unit manager changes while the request is still `Pending`, the request is reassigned to the replacement manager and `unitManagerStakeholder` is rotated from the old manager's observer row to a new 0% READ observer row for the replacement manager. Approval must use that current request-owned observer row, never the previous manager's stakeholder.
- Deal stakeholder DTOs may attach `unitAccess` metadata to the request-owned observer row. Its `unitLabel` is intentionally privacy-safe and never includes raw `InventoryUnit.unitNumber`; it uses project name, building name, or inventory unit ID fallback because stakeholder endpoints are not the authority for unit-number visibility.
- Rejecting or cancelling the request soft-deletes the request-owned unit-manager observer row. Auto-cancellation paths do the same cleanup while preserving the `UnitAccessRequest` row for audit.
- On approval, `UnitAccessService.approveRequest()` updates the request-owned observer row to transfer the proposed commission from the requesting primary stakeholder to the unit manager.

### Organization Commission Split

The total commission pool for a deal (value × commissionRate) is split between the **organization (agency)** and **agents** before individual agent percentages are applied.

**Lazy resolution cascade (evaluated server-side at DTO construction and deal close):**

```
effectiveOrgCommission =
    lead/deal.orgCommissionPercentage              // explicit per-entity override (user set it)
    ?? UserOrgCommission.commissionOverride        // primary stakeholder's per-org rate (user_org_commission table)
    ?? organization.defaultOrgCommissionPercentage  // org-wide default
    ?? 0                                            // ultimate fallback
```

The cascade is implemented in `src/modules/crm/shared/org-commission-resolver.ts` (`resolveOrgCommission()`) and used by:
- **Lead/Deal DTO construction** — provides `effectiveOrgCommissionPercentage` and `orgCommissionSource` on every response
- **Deal close** — determines the actual org/agent split for commission payment creation

**Key rules:**

1. The `orgCommissionPercentage` on a lead/deal is an **explicit override only**. A `null` value means "inherited" — the cascade resolves lazily at read/close time.
2. **Setting `orgCommissionPercentage` on a lead or deal requires `FINANCE_MANAGE` permission.** The lead and deal controllers strip this field from the DTO before calling the service if the user lacks `FINANCE_MANAGE`. Users without this permission can still create/edit leads and deals — they just cannot set the agency commission override.
3. No automatic seeding occurs at creation, assignment, or pool claim. Only a user with `FINANCE_MANAGE` explicitly setting the field writes to this property.

### Commission Payment Feedback System

Commission payments now support recipient feedback on terminal-state payments (PAID or CANCELLED):

**Feedback Rules:**

- Only the payment recipient can submit feedback
- Feedback is write-once - cannot be overwritten once submitted
- `confirmed=true` means recipient acknowledges receipt/cancellation
- `confirmed=false` means recipient disputes it (note required)
- Feedback triggers notifications to managers via `CommissionFeedbackSubmittedEvent`

**Payment History:**

- All status transitions create immutable audit records in `CommissionPaymentHistory`
- Records include actor, timestamp, reason, and payment method snapshots
- History is accessible via `GET /commission-payments/:id/history`

**Notification Integration:**

- `CommissionNotificationHandler` processes confirm/dispute actions
- Co-located with domain service for direct business logic access
- Auto-discovered via `@NotificationActionHandler` decorator

## Team Commission Budget

### Overview

A team's `commissionPercentage` on an entity stakeholder acts as a **live budget** that can be sub-allocated to individual team members. This enables team leaders to manage how commission is distributed within their team without needing org-level permissions.

### Design Principles

1. **Team as Budget Holder**: The team stakeholder's `commissionPercentage` is the total pool available for team member allocation
2. **Sub-Allocation**: Team managers assign commission from the team budget to individual members. The team's percentage decreases as members receive allocations
3. **Unallocated Budget at Close**: If a deal closes with remaining team budget (team stakeholder has `commissionPercentage > 0`), that amount is paid to the **team** directly, not to any individual
4. **Team Pool Freeze**: When a team-only stakeholder (has `team`, no `user`) holds primary status, the entity is blocked from modification until a team member claims or is assigned to it

### Team Budget Lifecycle

<Steps>
<Step title="Lead assigned to team">
```
Team A stakeholder: { 
  team: "Team A", 
  user: null, 
  commission: 100%, 
  isPrimary: true,
  originalTeamCommission: 100 
}
```
</Step>

<Step title="Team manager assigns first member">
```
Team A stakeholder: { 
  team: "Team A", 
  user: null, 
  commission: 60%, 
  isPrimary: false,
  originalTeamCommission: 100 
}
Agent X stakeholder: { 
  user: "Agent X", 
  team: "Team A", 
  commission: 40%, 
  isPrimary: true,
  originalTeamCommission: 100 
}
```
</Step>

<Step title="Team manager assigns second member">
```
Team A stakeholder: { commission: 30% }   ← remaining budget
Agent X stakeholder: { commission: 40% }
Agent Y stakeholder: { commission: 30% }
```
</Step>

<Step title="Deal closes - commission payments">
- Agent X: 40% → paid to user
- Agent Y: 30% → paid to user  
- Team A: 30% → paid to TEAM (unallocated budget)
</Step>
</Steps>

<Info>
`originalTeamCommission` on each member's stakeholder records the team's total allocation (100% in this case), allowing the system to validate that the sum of team member commissions never exceeds the original team budget.
</Info>

### Claim from Team Pool

When a team member claims a lead from the team pool, the team stakeholder is **preserved** as a budget placeholder (set to `isPrimary: false`, `commissionPercentage: 0`) rather than being deleted. This allows the team to retain its position on the entity for future budget allocation.

The claiming user's stakeholder is created with `originalTeamCommission` set to the team's commission at the time of claim, enabling team budget enforcement even after subsequent reassignments.

```
BEFORE claim:
└── Team A: { commission: 100%, isPrimary: true, originalTeamCommission: 100 }

AFTER claim by Agent X (same team):
├── Team A: { commission: 0%, isPrimary: false, originalTeamCommission: 100 }
└── Agent X: { commission: 100%, isPrimary: true, originalTeamCommission: 100 }
```

### Commission Aggregation with Dual Records

With the dual-record model, a single user can hold **multiple** stakeholder rows on the same entity — one personal (`team_id=NULL`) and one per team they claimed from. Each row carries its own `commissionPercentage`.

**Aggregation Rule for Payment/Reporting:**

```sql
-- Total commission for a user on an entity (across all contexts)
SELECT user_id, SUM(commission_percentage) AS total_commission
FROM entity_stakeholder
WHERE entity_type = :entityType
  AND entity_id = :entityId
  AND user_id = :userId
  AND is_deleted = false
GROUP BY user_id;
```

**Example:**

```
Lead L has stakeholders:
├── Agent X (personal): { user: "X", team: null,    commission: 30% }  ← from global pool
├── Agent X (team):     { user: "X", team: "TeamA", commission: 20% }  ← claimed from team pool
├── Team A (budget):    { team: "TeamA", user: null, commission: 10% }  ← remaining team budget
└── Agent Y:            { user: "Y", team: "TeamB", commission: 40% }
Total Agent X commission: 50% (30% + 20%)
```

## Business Rules

### Core Invariants

<AccordionGroup>
<Accordion title="Stakeholder assignment rules">
- Only one stakeholder can have `is_primary=true` per entity
- At least one primary stakeholder must exist at all times
- Commission total must sum to exactly 100% (deals) or 0%/100% (leads)
- Stakeholders with 0% commission cannot be set as `isPrimary=true`
- Cannot remove primary stakeholder while others exist
- Remaining commissions proportionally scaled on removal
- Deal creation requires write access to source records
</Accordion>

<Accordion title="Team budget constraints">
- Sum of team member commissions cannot exceed `originalTeamCommission`
- Entities with team-only primary stakeholder are read-only until claimed
- Remaining team budget is paid to team at deal close
- Team stakeholder preserved as budget placeholder after claim
</Accordion>

<Accordion title="Entity independence rules">
- Lead and deal stakeholder changes do not propagate automatically
- Deal stakeholders are deal-owned after creation
- Source leads provide creation-time suggestions only
- 1:1 active deal per lead constraint enforced
- DONE stage auto-transitions when deals reach terminal states
</Accordion>
</AccordionGroup>

### Validation Rules

| Rule | Enforcement | Error Code |
| --- | --- | --- |
| Commission total = 100% (deals) | On batch update | `INVALID_COMMISSION_TOTAL` |
| One primary per entity | On assignment | `MULTIPLE_PRIMARY_STAKEHOLDERS` |
| Team budget limit | On allocation | `TEAM_BUDGET_EXCEEDED` |
| Primary guard | On removal | `CANNOT_REMOVE_PRIMARY` |
| 1:1 active deal per lead | On deal creation | `MULTIPLE_ACTIVE_DEALS` |
| Source write access | On deal creation | `INSUFFICIENT_SOURCE_ACCESS` |

## Access Control

### Permission Requirements

| Action | Required Permission | Notes |
| --- | --- | --- |
| View stakeholders | `stakeholders.read` | Independent of entity access |
| Assign stakeholder | `stakeholders.create` | Can assign to any accessible entity |
| Modify stakeholder | `stakeholders.update` | Includes commission and access changes |
| Remove stakeholder | `stakeholders.delete` | Cannot remove last stakeholder |
| Claim from team | `stakeholders.create` + team membership | Must be member of claiming team |
| Team budget allocation | `stakeholders.update` + team manager role | Team manager or above |
| Create deal from source | `WRITE`/`FULL` on source or `crm.manage` | Write access to source entity required |

### Deal Creation Permission Model

For deal creation from source leads/contacts, the system enforces write access:

- **`crm.manage`**: Can use any active org source record
- **Team context**: `team_crm.manage` allows use of team-accessible sources at write level
- **Direct access**: User needs `WRITE` or `FULL` stakeholder access on the source
- **View-only access**: `crm.view` and `team_crm.view` provide visibility but not deal creation rights

### Access Level Resolution

For users with multiple stakeholder records on the same entity, the system resolves to the **highest** access level:

```
User has two stakeholder records:
├── Personal stake: READ access
└── Team stake: WRITE access
Result: User has WRITE access to entity
```

## Quick Reference

### Commission Calculation Formula

```typescript
// Individual commission amount
amount = dealValue × (dealCommissionRate / 100) × (stakeholderPercentage / 100)

// Auto-adjustment on removal
newPercentage = oldPercentage × (100 / sumOfRemainingPercentages)

// Team budget validation
totalMemberCommission ≤ originalTeamCommission
```
