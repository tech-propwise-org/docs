## Vocabulary

<Info>
**User-facing and AI-facing vocabulary is "Assignment" and "Assignee".** "Stakeholder" is the legacy internal term.
</Info>

"Stakeholder" is retained as:
- the entity name (`EntityStakeholder`, `EntityStakeholderHistory`)
- the database table name (`entity_stakeholder`)
- the module folder (`src/modules/crm/entity-stakeholder/`)
- the internal service class names (`EntityStakeholderService`)
- the REST API DTO key on lead/deal/contact responses (`stakeholders` / `primaryStakeholder`)
- the `NotificationType` enum values (`LEAD_STAKEHOLDER_ADDED`, etc. — a rename would require a notification-type DB migration per `.cursor/rules/notification-type-migration.mdc`)
- the AI tool name (`getLeadStakeholders`)

Surfaces that face users or the AI assistant use the new vocabulary:

- **Frontend** modal titles, headings, toasts, notification copy, validation errors, audit log labels, and timeline labels render as "Assignee" / "Assignment" / "assigned to" / "unassigned"
- **Backend notification listeners** (`crm-event.listener.ts`) emit user-visible `title` and `message` fields using assignee vocabulary even though the underlying event names and `NotificationType` enum values keep the legacy "stakeholder" word internally
- **Audit timeline** (`audit-timeline.resolver.ts`) maps the `entity_stakeholder` table name to the user-facing label "Assignment" via `USER_FACING_TABLE_LABELS`; the timeline event constants (`STAKEHOLDER_ASSIGNED` etc.) keep their legacy names internally but render as "Assigned" / "Unassigned" / "Assignment updated" to users
- **AI assistant** system prompt + tool descriptions + schema glossary instruct the model to say "assignee" / "assignment" / "assigned to". The AI tool boundary (in `getLeadStakeholders.execute()` and the pre-hydrated `compactContext.lead.assignees` + `compactContext.lead.primaryAssignee`) projects the rows under `assignees` / `primaryAssignee` keys so the model never sees the legacy word in tool-call traces

<Warning>
The terms in the rest of this document are kept as "stakeholder" because this document specifies the engineering subsystem, not the user-visible surface. Translate to "assignee" / "assignment" when referencing this functionality in user-facing copy.
</Warning>

## Lead/Deal Stakeholder Independence

### Overview

Leads and deals no longer share stakeholder state after deal creation. Source leads can prefill a new deal's stakeholder list, but the persisted deal stakeholders are deal-owned and later lead/deal stakeholder edits do not automatically propagate.

### Design principles

1. **Deal-Owned Stakeholders**: Deal stakeholders are managed independently from lead stakeholders after creation. Leads provide creation-time suggestions only.
2. **1:1 Active Deal per Lead**: A lead can be referenced by at most one active deal contact (deal not archived, not closed-lost). A new active deal participant can only use that lead when no active deal already references it.
3. **No Bidirectional Stakeholder Sync**: Lead and deal stakeholder changes do not mirror automatically. This prevents buyer-side and seller-side leads from receiving each other's deal stakeholders.
4. **100% Commission Floor**: Deals must maintain 100% total commission. A lead without an active deal can have 0% (not yet configured).
5. **Inactive Deals Excluded**: Archived or closed-lost deals are excluded from active-lead conflict checks.
6. **Deal Creation Gate**: A deal can be created from an active source lead/contact only when the requester can use that source at write level. `crm.manage` can use any active org source record; otherwise the requester needs `WRITE`/`FULL` stakeholder access directly or through a `team_crm.manage` team context. `crm.view` / `team_crm.view` are visibility-only and do not allow deal creation from that source.
7. **Deal Stakeholder Invariant**: A deal must always have at least one stakeholder. Deal stakeholder sync rejects `stakeholders: []` with `400 "At least one stakeholder is required for a deal."`; deals do not have an unassign-to-pool operation. Enforced up-front in both `syncStakeholdersInTransaction` (the `PUT /stakeholders` engine) and `replaceStakeholdersInTransaction` — it cannot be left to the post-apply commission check because `validateFinalCommissionTotal` treats an empty final list as valid (the lead return-to-pool short-circuit).
8. **Transfer Permission Parity**: Lead and deal `canTransfer` response permissions follow the same actor model: the primary user stakeholder can initiate transfer, and a team manager can initiate transfer when their managed team holds the primary team-only row.
9. **Transfer Timeline Visibility**: Lead and deal timelines include related transfer request audit events. Request, approval, rejection, and cancellation rows from `entity_transfer` are surfaced on the owning lead/deal timeline with the transfer reason or rejection reason when present.
10. **DONE Stage Auto-Transitions** (v3.5): When a deal reaches a terminal stage (CLOSED_WON or CLOSED_LOST), every lead referenced by `deal_contact.originating_lead_id` moves to the `DONE` stage (unless already terminal or archived). Conversely, reopening the deal moves those participant-originating leads back to `QUALIFIED`.
11. **Source-Linked Deal Contacts Are Preserved**: Deal contact management may edit role, side, side-scoped primary status, and notes. It may add/remove ordinary non-source contacts, but it must not remove or replace a contact row that carries `deal_contact.originating_lead_id`; changing source leads requires a dedicated source-change flow because stage auto-transitions and active-deal conflict checks depend on that lineage.

### How stakeholder updates work

When `PUT /leads/:id/stakeholders` or `PUT /deals/:id/stakeholders` is called:

<Steps>
<Step title="Apply changes">
The sync endpoint applies all add/update/remove changes to the target entity.
</Step>

<Step title="Emit notifications">
It emits notifications based on the final transition:
- `0 → N` lead stakeholders emits `LEAD_ASSIGNED` and suppresses per-row "stakeholder added" notifications.
- `N → 0` lead stakeholders emits `LEAD_UNASSIGNED`, cancels pending transfers, and suppresses per-row "stakeholder removed" notifications.
- `N → 0` is invalid for deals because deals must retain at least one stakeholder.
- Other diffs emit fine-grained stakeholder added/removed/commission-changed notifications, plus primary assignee assigned/unassigned/row-changed notifications when the entity-level primary user changes or when primary moves between two rows of the same user (personal ↔ team).
</Step>

<Step title="No automatic sync">
No linked lead/deal entity is automatically mutated.
</Step>
</Steps>

Deal contact edits follow the same independence model: contact metadata edits do not mutate source leads or deal stakeholders. Source-linked contact rows are intentionally protected from generic removal/replacement so lead lineage remains stable for downstream deal lifecycle behavior.

## Full Stakeholder Rules

This section is the canonical rule set for lead/deal/contact stakeholder behavior. If a later section gives an example, the rules below win.

### Row types

| Row shape | Meaning |
|-----------|---------|
| `{ user: X, team: null }` | Personal stakeholder — user X working outside any team context |
| `{ user: X, team: Y }` | Team-affiliated stakeholder — user X working in team Y's context |
| `{ user: null, team: Y }` | Team-only stakeholder — team Y's unallocated pool |

<Info>
The same user can hold multiple stakeholder rows on one entity (personal + team-affiliated) representing different commission sources.
</Info>

### Commission rules

<AccordionGroup>
<Accordion title="Deal commission requirements">
- Deals must maintain exactly 100% total commission at all times
- Cannot remove stakeholders if it would violate 100% requirement
- Commission auto-adjusts proportionally when stakeholders are removed
- Minimum one stakeholder required (cannot unassign deals completely)
</Accordion>

<Accordion title="Lead commission flexibility">
- Leads can have 0% total (unassigned) or 100% total (assigned)
- Partial commission totals (1-99%) are invalid for leads
- Can remove all stakeholders (unassign to pool)
- Auto-adjusts to 100% when first stakeholder is added
</Accordion>

<Accordion title="Primary stakeholder rules">
- Exactly one stakeholder per entity can have `isPrimary: true`
- Primary stakeholder must have `commissionPercentage > 0`
- Observer stakeholders (0% commission) cannot be primary
- Primary gets `FULL` access level automatically
- Removing primary auto-promotes highest-commission stakeholder
</Accordion>
</AccordionGroup>

### Access level hierarchy

| Level | Permissions |
|-------|------------|
| `READ` | View entity and participate in activities |
| `WRITE` | READ + edit entity details and manage activities |
| `FULL` | WRITE + transfer entity and manage stakeholders |

<Note>
Users with multiple stakeholder records resolve to their highest access level across all records.
</Note>

## Team Commission Budget

### Overview

A team's `commissionPercentage` on an entity stakeholder acts as a **live budget** that can be sub-allocated to individual team members. This enables team leaders to manage how commission is distributed within their team without needing org-level permissions.

### Design principles

1. **Team as Budget Holder**: The team stakeholder's `commissionPercentage` is the total pool available for team member allocation
2. **Sub-Allocation**: Team managers assign commission from the team budget to individual members. The team's percentage decreases as members receive allocations
3. **Unallocated Budget at Close**: If a deal closes with remaining team budget (team stakeholder has `commissionPercentage > 0`), that amount is paid to the **team** directly, not to any individual
4. **Team Pool Freeze**: When a team-only stakeholder (has `team`, no `user`) holds primary status, the entity is blocked from modification until a team member claims or is assigned to it

### Team budget lifecycle

<Steps>
<Step title="Lead assigned to team">
```json
Team A stakeholder: { 
  "team": "Team A", 
  "user": null, 
  "commission": "100%", 
  "isPrimary": true,
  "originalTeamCommission": 100 
}
```
</Step>

<Step title="Team manager assigns first member">
```json
Team A stakeholder: { 
  "team": "Team A", 
  "user": null, 
  "commission": "60%", 
  "isPrimary": false,
  "originalTeamCommission": 100 
}
Agent X stakeholder: { 
  "user": "Agent X", 
  "team": "Team A", 
  "commission": "40%", 
  "isPrimary": true,
  "originalTeamCommission": 100 
}
```
</Step>

<Step title="Team manager assigns second member">
```json
Team A stakeholder: { "commission": "30%" }   // ← remaining budget
Agent X stakeholder: { "commission": "40%" }
Agent Y stakeholder: { "commission": "30%" }
```
</Step>

<Step title="Deal closes - commission payments">
- Agent X: 40% → paid to user
- Agent Y: 30% → paid to user  
- Team A: 30% → paid to TEAM (unallocated budget)
</Step>
</Steps>

<Info>
`originalTeamCommission` on each member's stakeholder records the team's total allocation (100% in this case), allowing the system to validate that the sum of team member commissions never exceeds the original team budget.
</Info>

### Commission aggregation with dual records

With the dual-record model, a single user can hold **multiple** stakeholder rows on the same entity — one personal (`team_id=NULL`) and one per team they claimed from. Each row carries its own `commissionPercentage`.

**Aggregation rule for payment/reporting:**

```sql
-- Total commission for a user on an entity (across all contexts)
SELECT user_id, SUM(commission_percentage) AS total_commission
FROM entity_stakeholder
WHERE entity_type = :entityType
  AND entity_id = :entityId
  AND user_id = :userId
  AND is_deleted = false
GROUP BY user_id;
```

**Example:**

```
Lead L has stakeholders:
├── Agent X (personal): { user: "X", team: null,    commission: 30% }  ← from global pool
├── Agent X (team):     { user: "X", team: "TeamA", commission: 20% }  ← claimed from team pool
├── Team A (budget):    { team: "TeamA", user: null, commission: 10% }  ← remaining team budget
└── Agent Y:            { user: "Y", team: "TeamB", commission: 40% }
Total Agent X commission: 50% (30% + 20%)
```

### Deal creation with a pending unit access request

When a deal is created with a hidden `InventoryUnit` managed by another agent (`unit.unitManager.id !== currentUser.id` and `isUnitNumberHidden !== false`), `DealService.create()` defers the unit link and issues a `Pending` `UnitAccessRequest` in the same transaction unless the submitted deal stakeholders already include that unit manager. Public units and already-included unit managers link directly with no request-owned observer row and no automatic commission transfer.

**Stakeholder-side rules:**

- The deal's requested stakeholders are created exactly as in the normal path (explicit `dto.stakeholders[]` or creator-as-100%-primary). The pending-link path then adds one extra request-owned unit-manager observer row.
- If `dto.stakeholders[]` already includes the unit manager, the deal links the inventory unit directly and keeps that stakeholder's submitted commission/access unchanged; the pending access commission-transfer mechanism is skipped.
- The deal creator (current user) **must** end up as a primary user-stakeholder of the new deal — otherwise the access request has no requester to attribute and `DealService.create()` rejects the payload.
- The proposed `unitManagerCommissionPercentage` must be `<=` the requesting primary stakeholder's commission. The check runs at deal-create time (so the deal never lands in a state that approval cannot service) **and** again at approval time on the **live** stakeholder row, in case stakeholders were re-synced between request and approval.
- Request creation adds the unit manager as a request-owned `DEAL` stakeholder with `isPrimary=false`, `commissionPercentage=0`, and `accessLevel=READ`. `UnitAccessRequest.unitManagerStakeholder` points at this row. Each pending request gets its own observer row, even if the same unit manager has another pending request on the same deal for a different inventory unit. While the request is `Pending`, no user may edit, change access level, change commission, or remove this row through stakeholder management; the request must be approved, rejected, or cancelled.

## Business Rules

### Core invariants

<AccordionGroup>
<Accordion title="Stakeholder assignment rules">
- Only one stakeholder can have `is_primary=true` per entity
- At least one primary stakeholder must exist at all times
- Commission total must sum to exactly 100% (deals) or 0%/100% (leads)
- Stakeholders with 0% commission cannot be set as `isPrimary=true`
- Cannot remove primary stakeholder while others exist
- Remaining commissions proportionally scaled on removal
- Deal creation requires write access to source records
</Accordion>

<Accordion title="Team budget constraints">
- Sum of team member commissions cannot exceed `originalTeamCommission`
- Entities with team-only primary stakeholder are read-only until claimed
- Remaining team budget is paid to team at deal close
- Team stakeholder preserved as budget placeholder after claim
</Accordion>

<Accordion title="Entity independence rules">
- Lead and deal stakeholder changes do not propagate automatically
- Deal stakeholders are deal-owned after creation
- Source leads provide creation-time suggestions only
- 1:1 active deal per lead constraint enforced
- DONE stage auto-transitions when deals reach terminal states
</Accordion>
</AccordionGroup>

### Validation rules

| Rule | Enforcement | Error Code |
|------|-------------|------------|
| Commission total = 100% (deals) | On batch update | `INVALID_COMMISSION_TOTAL` |
| One primary per entity | On assignment | `MULTIPLE_PRIMARY_STAKEHOLDERS` |
| Team budget limit | On allocation | `TEAM_BUDGET_EXCEEDED` |
| Primary guard | On removal | `CANNOT_REMOVE_PRIMARY` |
| 1:1 active deal per lead | On deal creation | `MULTIPLE_ACTIVE_DEALS` |
| Source write access | On deal creation | `INSUFFICIENT_SOURCE_ACCESS` |

### Organization commission split

The total commission pool for a deal (value × commissionRate) is split between the **organization (agency)** and **agents** before individual agent percentages are applied.

**Lazy resolution cascade (evaluated server-side at DTO construction and deal close):**

```
effectiveOrgCommission =
    lead/deal.orgCommissionPercentage              // explicit per-entity override (user set it)
    ?? UserOrgCommission.commissionOverride        // primary stakeholder's per-org rate (user_org_commission table)
    ?? organization.defaultOrgCommissionPercentage  // org-wide default
    ?? 0                                            // ultimate fallback
```

<Note>
Setting `orgCommissionPercentage` on a lead or deal requires `FINANCE_MANAGE` permission. Users without this permission can still create/edit leads and deals — they just cannot set the agency commission override.
</Note>

### Commission payment feedback system

Commission payments now support recipient feedback on terminal-state payments (PAID or CANCELLED):

**Feedback rules:**

- Only the payment recipient can submit feedback
- Feedback is write-once - cannot be overwritten once submitted
- `confirmed=true` means recipient acknowledges receipt/cancellation
- `confirmed=false` means recipient disputes it (note required)
- Feedback triggers notifications to managers via `CommissionFeedbackSubmittedEvent`

## Access Control

### Permission requirements

| Action | Required Permission | Notes |
|--------|-------------------|-------|
| View stakeholders | `stakeholders.read` | Independent of entity access |
| Assign stakeholder | `stakeholders.create` | Can assign to any accessible entity |
| Modify stakeholder | `stakeholders.update` | Includes commission and access changes |
| Remove stakeholder | `stakeholders.delete` | Cannot remove last stakeholder |
| Claim from team | `stakeholders.create` + team membership | Must be member of claiming team |
| Team budget allocation | `stakeholders.update` + team manager role | Team manager or above |
| Create deal from source | `WRITE`/`FULL` on source or `crm.manage` | Write access to source entity required |

### Deal creation permission model

For deal creation from source leads/contacts, the system enforces write access:

- **`crm.manage`**: Can use any active org source record
- **Team context**: `team_crm.manage` allows use of team-accessible sources at write level
- **Direct access**: User needs `WRITE` or `FULL` stakeholder access on the source
- **View-only access**: `crm.view` and `team_crm.view` provide visibility but not deal creation rights

### Access level resolution

For users with multiple stakeholder records on the same entity, the system resolves to the **highest** access level:

```
User has two stakeholder records:
├── Personal stake: READ access
└── Team stake: WRITE access
Result: User has WRITE access to entity
```

## State Transitions

### Stakeholder lifecycle

<Tabs>
<Tab title="Assignment">
```
NEW → ASSIGNED
├── Create stakeholder record
├── Auto-assign commission if primary (100%)
├── Default to READ access unless specified
├── No automatic sync to linked entities
└── Record assignment in audit log
```
</Tab>

<Tab title="Commission Change">
```
ASSIGNED → UPDATED
├── Validate new commission total = 100% (deals)
├── Record old state in history
├── Update stakeholder record
├── No automatic sync to linked entities
└── Redistribute if removing stakeholder
```
</Tab>

<Tab title="Primary Transfer">
```
PRIMARY → NON_PRIMARY (old)
NON_PRIMARY → PRIMARY (new)
├── Validate only one primary exists
├── Update access level (new primary gets FULL)
├── Record state changes in history
├── Update both stakeholder records atomically
└── No automatic sync to linked entities
```
</Tab>

<Tab title="Removal">
```
ASSIGNED → REMOVED
├── Check removal safety rules
├── Record final state in history
├── Redistribute remaining commission
├── Soft-delete stakeholder record
├── Auto-promote new primary if needed
└── No automatic sync to linked entities
```
</Tab>
</Tabs>

### Team pool states

<Tabs>
<Tab title="Team Assignment">
```
UNASSIGNED → TEAM_POOL
├── Create team-only stakeholder (user_id = null)
├── Set as primary with 100% commission
├── Entity becomes "frozen" for modifications
└── Notify team members of new assignment
```
</Tab>

<Tab title="Claim from Pool">
```
TEAM_POOL → CLAIMED
├── Validate user is team member
├── Preserve team stakeholder as budget placeholder
├── Create user stakeholder as new primary
├── Entity becomes "unfrozen"
└── Set originalTeamCommission for budget tracking
```
</Tab>

<Tab title="Budget Allocation">
```
CLAIMED → ALLOCATED
├── Validate allocation within team budget
├── Create additional team member stakeholder
├── Reduce team stakeholder commission
├── Maintain budget tracking consistency
└── Update team allocation records
```
</Tab>
</Tabs>

## Quick Reference

### Commission calculation formula

```typescript
// Individual commission amount
amount = dealValue × (dealCommissionRate / 100) × (stakeholderPercentage / 100)

// Auto-adjustment on removal
newPercentage = oldPercentage × (100 / sumOfRemainingPercentages)

// Team budget validation
totalMemberCommission ≤ originalTeamCommission
```

### Key validation rules

<AccordionGroup>
<Accordion title="Stakeholder assignment">
- Exactly one primary per entity
- Commission total must equal 100% for deals, 0% or 100% for leads
- Cannot set observer (0% commission) as primary
- Team members can only claim from their own team pools
- 1:1 active deal per lead constraint enforced
- Deal creation requires write access to source records
</Accordion>

<Accordion title="Team budget">
- Team member allocations cannot exceed original team budget
- Unallocated budget is paid to team at deal close
- Team stakeholder preserved as placeholder after claim
- Team pool freeze prevents entity modification
</Accordion>

<Accordion title="Entity independence">
- Lead and deal stakeholders managed independently after creation
- No automatic bidirectional sync between entities
- Deal creation validates source write access
- DONE stage auto-transitions on deal terminal states
</Accordion>
</AccordionGroup>

| Code | Scenario |
|------|----------|

| `422` | Business rule violation (cannot remove last stakeholder, insufficient source access) |
