## Stakeholder Sync Feature (v2 — Full Bidirectional)

### Overview

Leads and their active deals share **identical stakeholder state**. Any change to either side's stakeholders is automatically propagated to the linked entity via `syncToLinkedEntity`. This replaces the previous observer-only sync model.

### Design Principles

1. **Full Bidirectional Sync**: Lead ↔ active Deal stakeholder state is always mirrored (same users, same commission, same primary)
2. **1:1 Active Deal per Lead**: A lead can have at most one active deal (not archived, not closed-lost). A new deal can only be created when no active deal exists.
3. **Last Write Wins**: Whichever entity (lead or deal) is changed most recently becomes the current truth for both.
4. **100% Commission Floor**: Once a lead-deal link exists, both sides must maintain 100% total commission. A lead without an active deal can have 0% (not yet configured).
5. **Inactive Deals Excluded**: Archived or closed-lost deals are excluded from sync — the lead is effectively "unlinked."
6. **Deal Creation Gate**: A deal can only be created from a lead that is in the **QUALIFIED** stage. Leads in any other stage are not available for deal creation.
7. **DONE Stage Auto-Transitions** (v3.5): When a deal connected to a lead reaches a terminal stage (CLOSED_WON or CLOSED_LOST), `DealService.update()` emits a `DEAL_TERMINAL_REACHED` event. `LeadService` listens with `@OnEvent` and automatically moves the originating lead to the `DONE` stage (unless the lead is already terminal or archived). Conversely, if the deal is reopened (moved from a terminal to a non-terminal stage), a `DEAL_REOPENED` event is emitted and the lead is automatically moved back to the `QUALIFIED` stage. Users cannot manually move leads to or from the `DONE` stage.

### How Sync Works

When `PUT /leads/:id/stakeholders` or `PUT /deals/:id/stakeholders` is called:

<Steps>
<Step title="Apply changes to target entity">
The sync endpoint applies all add/update/remove changes to the target entity.
</Step>

<Step title="Automatic sync initiation">
After changes are committed, `syncToLinkedEntity` is called automatically.
</Step>

<Step title="Find linked entity">
It finds the linked entity (lead's active deal, or deal's originating lead).
</Step>

<Step title="Map and sync stakeholders">
It maps the source stakeholders to the target's desired state (keyed by `userId:teamId` — no row IDs needed).
</Step>

<Step title="Execute sync with guard">
It calls `syncStakeholdersInTransaction` on the target with `skipSync: true` (recursion guard).
</Step>
</Steps>

```
Lead stakeholders changed via PUT
  → Apply changes to lead
  → syncToLinkedEntity(LEAD, leadId)
    → Find active deal for this lead
    → Call syncStakeholdersInTransaction(DEAL, dealId, ..., { skipSync: true })
```

### Deal Creation with Stakeholders

When creating a deal from a lead, the frontend pre-fills the stakeholder list from the lead and lets the user modify it. The backend applies stakeholders via one of three branches (priority order):

**Branch 1 — Explicit stakeholders provided (`dto.stakeholders[]` non-empty):**

<Steps>
<Step title="Validate and create">
Validates that commission totals 100% before creating.
</Step>

<Step title="Persist stakeholders">
Creates all deal stakeholder rows via `createStakeholderBatchInTransaction()`.
</Step>

<Step title="Auto-sync to lead">
After all rows are persisted, automatically calls `syncToLinkedEntity(DEAL, dealId)`, which mirrors the deal's final stakeholder state to the originating lead — same users, same commissions, same primary flag. This ensures the lead reflects the user's deliberate modifications from the create modal.
</Step>
</Steps>

**Branch 2 — No explicit stakeholders, originating lead exists:**

<Steps>
<Step title="Inherit from lead">
Copies all lead stakeholder rows to the deal via `inheritFromLeadToDealInTransaction()`.
</Step>

<Step title="Validate inherited total">
Validates that the inherited total equals 100%.
</Step>

<Step title="Auto-sync (no-op)">
After flush, automatically calls `syncToLinkedEntity(DEAL, dealId)`. Since the deal's stakeholders are an exact copy of the lead's, this is a zero-diff no-op (no mutations, no notifications emitted).
</Step>
</Steps>

**Branch 3 — No stakeholders, no lead (walk-in deal):**

<Steps>
<Step title="Create default stakeholder">
Creates a single primary stakeholder (creator, 100% commission).
</Step>

<Step title="No sync needed">
No sync — there is no originating lead.
</Step>
</Steps>

<Info>
**`skipSync` opt-out:** Both `createStakeholderBatchInTransaction()` and `inheritFromLeadToDealInTransaction()` accept `options?: { skipSync?: boolean }`. Pass `skipSync: true` to suppress sync (e.g. seeding scripts, migrations). Default is `false` — sync always runs.
</Info>

### Skip Conditions (No Sync)

| Condition | When Source Is Lead | When Source Is Deal |
| --- | --- | --- |
| No linked entity | No active deal exists | No originating lead |
| Linked entity deleted | Deal is soft-deleted | Lead is soft-deleted |
| Linked entity archived | Deal is archived | Lead is archived |
| Deal is closed-lost | Deal has `isClosed=true AND isWon=false` | N/A |

### Commission Rules with Sync

| Scenario | Commission Rule |
| --- | --- |
| Deal (always) | Must be exactly 100% |
| Lead with active deal | Must be exactly 100% (synced with deal) |
| Lead without active deal | 0% or 100% (0% = not yet configured) |
| Deal transitions to closed-lost or archived | Lead becomes "unlinked" — can go back to 0% |

## Team Commission Budget

### Overview

A team's `commissionPercentage` on an entity stakeholder acts as a **live budget** that can be sub-allocated to individual team members. This enables team leaders to manage how commission is distributed within their team without needing org-level permissions.

### Design Principles

1. **Team as Budget Holder**: The team stakeholder's `commissionPercentage` is the total pool available for team member allocation
2. **Sub-Allocation**: Team managers assign commission from the team budget to individual members. The team's percentage decreases as members receive allocations
3. **Unallocated Budget at Close**: If a deal closes with remaining team budget (team stakeholder has `commissionPercentage > 0`), that amount is paid to the **team** directly, not to any individual
4. **Team Pool Freeze**: When a team-only stakeholder (has `team`, no `user`) holds primary status, the entity is blocked from modification until a team member claims or is assigned to it

### Team Budget Lifecycle

<Steps>
<Step title="Lead assigned to team">
```
Team A stakeholder: { 
  team: "Team A", 
  user: null, 
  commission: 100%, 
  isPrimary: true,
  originalTeamCommission: 100 
}
```
</Step>

<Step title="Team manager assigns first member">
```
Team A stakeholder: { 
  team: "Team A", 
  user: null, 
  commission: 60%, 
  isPrimary: false,
  originalTeamCommission: 100 
}
Agent X stakeholder: { 
  user: "Agent X", 
  team: "Team A", 
  commission: 40%, 
  isPrimary: true,
  originalTeamCommission: 100 
}
```
</Step>

<Step title="Team manager assigns second member">
```
Team A stakeholder: { commission: 30% }   ← remaining budget
Agent X stakeholder: { commission: 40% }
Agent Y stakeholder: { commission: 30% }
```
</Step>

<Step title="Deal closes - commission payments">
- Agent X: 40% → paid to user
- Agent Y: 30% → paid to user  
- Team A: 30% → paid to TEAM (unallocated budget)
</Step>
</Steps>

<Info>
`originalTeamCommission` on each member's stakeholder records the team's total allocation (100% in this case), allowing the system to validate that the sum of team member commissions never exceeds the original team budget.
</Info>

### Claim from Team Pool

When a team member claims a lead from the team pool, the team stakeholder is **preserved** as a budget placeholder (set to `isPrimary: false`, `commissionPercentage: 0`) rather than being deleted. This allows the team to retain its position on the entity for future budget allocation.

The claiming user's stakeholder is created with `originalTeamCommission` set to the team's commission at the time of claim, enabling team budget enforcement even after subsequent reassignments.

```
BEFORE claim:
└── Team A: { commission: 100%, isPrimary: true, originalTeamCommission: 100 }

AFTER claim by Agent X (same team):
├── Team A: { commission: 0%, isPrimary: false, originalTeamCommission: 100 }
└── Agent X: { commission: 100%, isPrimary: true, originalTeamCommission: 100 }
```

### Commission Aggregation with Dual Records

With the dual-record model, a single user can hold **multiple** stakeholder rows on the same entity — one personal (`team_id=NULL`) and one per team they claimed from. Each row carries its own `commissionPercentage`.

**Aggregation Rule for Payment/Reporting:**

```sql
-- Total commission for a user on an entity (across all contexts)
SELECT user_id, SUM(commission_percentage) AS total_commission
FROM entity_stakeholder
WHERE entity_type = :entityType
  AND entity_id = :entityId
  AND user_id = :userId
  AND is_deleted = false
GROUP BY user_id;
```

**Example:**

```
Lead L has stakeholders:
├── Agent X (personal): { user: "X", team: null,    commission: 30% }  ← from global pool
├── Agent X (team):     { user: "X", team: "TeamA", commission: 20% }  ← claimed from team pool
├── Team A (budget):    { team: "TeamA", user: null, commission: 10% }  ← remaining team budget
└── Agent Y:            { user: "Y", team: "TeamB", commission: 40% }
Total Agent X commission: 50% (30% + 20%)
```

### Commission Payment Feedback System

Commission payments now support recipient feedback on terminal-state payments (PAID or CANCELLED):

**Feedback Rules:**

- Only the payment recipient can submit feedback
- Feedback is write-once - cannot be overwritten once submitted
- `confirmed=true` means recipient acknowledges receipt/cancellation
- `confirmed=false` means recipient disputes it (note required)
- Feedback triggers notifications to managers via `CommissionFeedbackSubmittedEvent`

**Payment History:**

- All status transitions create immutable audit records in `CommissionPaymentHistory`
- Records include actor, timestamp, reason, and payment method snapshots
- History is accessible via `GET /commission-payments/:id/history`

**Notification Integration:**

- `CommissionNotificationHandler` processes confirm/dispute actions
- Co-located with domain service for direct business logic access
- Auto-discovered via `@NotificationActionHandler` decorator

## Business Rules

### Core Invariants

<AccordionGroup>
<Accordion title="Stakeholder assignment rules">
- Only one stakeholder can have `is_primary=true` per entity
- At least one primary stakeholder must exist at all times
- Commission total must sum to exactly 100% (deals) or 0%/100% (leads)
- Stakeholders with 0% commission cannot be set as `isPrimary=true`
- Cannot remove primary stakeholder while others exist
- Remaining commissions proportionally scaled on removal
- Deal creation gate: only QUALIFIED leads can create deals
</Accordion>

<Accordion title="Team budget constraints">
- Sum of team member commissions cannot exceed `originalTeamCommission`
- Entities with team-only primary stakeholder are read-only until claimed
- Remaining team budget is paid to team at deal close
- Team stakeholder preserved as budget placeholder after claim
</Accordion>

<Accordion title="Sync safety rules">
- Full bidirectional sync maintains identical stakeholder state
- Lead-deal links require 100% commission on both sides
- 1:1 active deal per lead constraint enforced
- Last write wins principle for conflict resolution
- DONE stage auto-transitions when deals reach terminal states
</Accordion>
</AccordionGroup>

### Validation Rules

| Rule | Enforcement | Error Code |
| --- | --- | --- |
| Commission total = 100% (deals) | On batch update | `INVALID_COMMISSION_TOTAL` |
| One primary per entity | On assignment | `MULTIPLE_PRIMARY_STAKEHOLDERS` |
| Team budget limit | On allocation | `TEAM_BUDGET_EXCEEDED` |
| Primary guard | On removal | `CANNOT_REMOVE_PRIMARY` |
| 1:1 active deal per lead | On deal creation | `MULTIPLE_ACTIVE_DEALS` |
| QUALIFIED stage gate | On deal creation | `LEAD_NOT_QUALIFIED` |

## Access Control

### Permission Requirements

| Action | Required Permission | Notes |
| --- | --- | --- |
| View stakeholders | `stakeholders.read` | Independent of entity access |
| Assign stakeholder | `stakeholders.create` | Can assign to any accessible entity |
| Modify stakeholder | `stakeholders.update` | Includes commission and access changes |
| Remove stakeholder | `stakeholders.delete` | Cannot remove last stakeholder |
| Claim from team | `stakeholders.create` + team membership | Must be member of claiming team |
| Team budget allocation | `stakeholders.update` + team manager role | Team manager or above |

### Access Level Resolution

For users with multiple stakeholder records on the same entity, the system resolves to the **highest** access level:

```
User has two stakeholder records:
├── Personal stake: READ access
└── Team stake: WRITE access
Result: User has WRITE access to entity
```

## State Transitions

### Stakeholder Lifecycle

<Tabs>
<Tab title="Assignment">
```
NEW → ASSIGNED
├── Create stakeholder record
├── Auto-assign commission if primary (100%)
├── Default to READ access unless specified
├── Trigger sync to linked entity (bidirectional)
└── Record assignment in audit log
```
</Tab>

<Tab title="Commission Change">
```
ASSIGNED → UPDATED
├── Validate new commission total = 100% (deals)
├── Record old state in history
├── Update stakeholder record
├── Trigger sync to linked entity
└── Redistribute if removing stakeholder
```
</Tab>

<Tab title="Primary Transfer">
```
PRIMARY → NON_PRIMARY (old)
NON_PRIMARY → PRIMARY (new)
├── Validate only one primary exists
├── Update access level (new primary gets FULL)
├── Record state changes in history
├── Update both stakeholder records atomically
└── Trigger sync to linked entity
```
</Tab>

<Tab title="Removal">
```
ASSIGNED → REMOVED
├── Check removal safety rules
├── Record final state in history
├── Redistribute remaining commission
├── Soft-delete stakeholder record
├── Auto-promote new primary if needed
└── Trigger sync to linked entity
```
</Tab>
</Tabs>

### Team Pool States

<Tabs>
<Tab title="Team Assignment">
```
UNASSIGNED → TEAM_POOL
├── Create team-only stakeholder (user_id = null)
├── Set as primary with 100% commission
├── Entity becomes "frozen" for modifications
└── Notify team members of new assignment
```
</Tab>

<Tab title="Claim from Pool">
```
TEAM_POOL → CLAIMED
├── Validate user is team member
├── Preserve team stakeholder as budget placeholder
├── Create user stakeholder as new primary
├── Entity becomes "unfrozen"
└── Set originalTeamCommission for budget tracking
```
</Tab>

<Tab title="Budget Allocation">
```
CLAIMED → ALLOCATED
├── Validate allocation within team budget
├── Create additional team member stakeholder
├── Reduce team stakeholder commission
├── Maintain budget tracking consistency
└── Update team allocation records
```
</Tab>
</Tabs>

## Quick Reference

### Commission Calculation Formula

```typescript
// Individual commission amount
amount = dealValue × (dealCommissionRate / 100) × (stakeholderPercentage / 100)

// Auto-adjustment on removal
newPercentage = oldPercentage × (100 / sumOfRemainingPercentages)

// Team budget validation
totalMemberCommission ≤ originalTeamCommission
```

### Key Validation Rules

<AccordionGroup>
<Accordion title="Stakeholder assignment">
- Exactly one primary per entity
- Commission total must equal 100% for deals, 0% or 100% for leads
- Cannot set observer (0% commission) as primary
- Team members can only claim from their own team pools
- 1:1 active deal per lead constraint enforced
- Deal creation gate: only QUALIFIED leads can create deals
</Accordion>

<Accordion title="Team budget">
- Team member allocations cannot exceed original team budget
- Unallocated budget is paid to team at deal close
- Team stakeholder preserved as placeholder after claim
- Team pool freeze prevents entity modification
</Accordion>

<Accordion title="Bidirectional sync">
- Lead and active deal maintain identical stakeholder state
- Last write wins for conflict resolution
- Sync skipped for archived/closed-lost entities
- 100% commission floor when entities are linked
- DONE stage auto-transitions on deal terminal states
</Accordion>
</AccordionGroup>
