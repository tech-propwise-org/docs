The CRM module is the core of PropWise. It manages the full lifecycle of customer relationships тАФ from initial inquiry (lead) through qualification (contact) to revenue (deal). Every entity links back to a central **Person** identity layer.

## Architecture overview

### Design principles

The CRM module follows six core design principles:

1. **Person + Contact model** тАФ `Person` is the hidden identity layer (single source of truth for personal details). `Contact` is the business relationship layer. `Lead` is the sales opportunity layer. `Deal` links to `Contact`, not `Person` directly.
2. **Unified stakeholder model** тАФ A single `entity_stakeholder` table manages assignment and commission across leads and deals.
3. **Polymorphic patterns** тАФ Notes, tags, and activities use `entity_type`/`entity_id` patterns for flexible targeting.
4. **Channel separation** тАФ The `activity` table indexes the timeline; channel tables (`call_log`, `email_log`, etc.) store full data.
5. **Modular design** тАФ CRM core is independent. Real Estate, Marketing, and Channels are optional modules.
6. **Company via Contact** тАФ Companies associate with `Contact` via `ContactCompanyRole`, not with `Person`.

### Real estate integration

Real Estate entities link to **Person** (not Contact) for identity:

| Real estate entity       | Links to    | Rationale                                   |
| ------------------------ | ----------- | ------------------------------------------- |
| `unit_ownership`         | `person_id` | Ownership is about identity, not CRM status |
| `unit_transaction`       | `person_id` | Transaction party is an individual          |
| `site_visit`             | `person_id` | Who visited the property                    |
| `lead_property_interest` | `lead_id`   | Links to lead for sales context             |
| `deal_property_interest` | `deal_id`   | Links to deal for transaction context       |

<Tip>
  Contacts access real estate data through Person:
  `contact.person.unitOwnerships`. This keeps ownership tied to identity
  regardless of CRM relationship status.
</Tip>

**Deal property interest workflow:**

```typescript
// Deal created FROM lead:
// Copy primary LeadPropertyInterest тЖТ DealPropertyInterest (1:1)
deal.propertyInterest.originatingInterest = leadPropertyInterest;

// Deal created directly (walk-in):
// Create DealPropertyInterest with no originating interest
deal.propertyInterest.originatingInterest = null;
```

## Core entities

### Person (central identity)

**Purpose**: Single source of truth for human identity and preferences.

```
Person
тФЬтФАтФА Identity: first_name, last_name, avatar_url, title
тФЬтФАтФА Demographics: date_of_birth, nationality
тФЬтФАтФА Links: links jsonb [{ type, url }] where type is website/facebook/instagram/linkedIn/twitter/tiktok
тФЬтФАтФА Preferences: timezone
тФЬтФАтФА Languages: languages (unified array with code and proficiency per entry)
тФЬтФАтФА Communication Flags: do_not_call, do_not_email
тФЬтФАтФА Source Tracking: original_source
тФЬтФАтФА Merge Tracking: merged_into_id, merged_at, merged_by
тФЬтФАтФА Computed: full_name (getter: first_name + last_name)
тФФтФАтФА Related Tables:
    тФЬтФАтФА person_email (multiple emails, one primary)
    тФЬтФАтФА person_phone (multiple phones, one primary)
    тФЬтФАтФА person_address (multiple addresses, one primary, with is_primary and address_type)
    тФЬтФАтФА person_channel (WhatsApp, Instagram, etc. identities)
    тФФтФАтФА person_not_duplicate (deduplication override pairs)
```

**Key Rules**:
- Every Lead, Contact must link to a Person
- Person preferences apply across all contexts (leads, deals, contacts)
- `nationality` is stored on Person so leads and contacts share the same identity profile value
- `original_source` is set once when person first enters system
- Languages array uses unified `UserLanguageEntry` format with code and proficiency per entry
- `links` stores web and social profile links as JSONB entries with `type` and `url`; deprecated `website`, `linkedin_url`, and `twitter_url` columns are no longer used
- `person_address` rows use `is_primary` for the primary address and `address_type` for classification (`Home`, `Work`, etc.)

### PersonChannel (communication channels)

**Purpose**: Stores a person's communication channel identities (WhatsApp, Instagram, etc.).

```
person_channel
тФЬтФАтФА person_id тЖТ Person
тФЬтФАтФА channel_type (whatsapp, instagram, facebook, telegram, sms, webchat)
тФЬтФАтФА channel_identifier (phone number, username, PSID, etc.)
тФЬтФАтФА display_name, avatar_url
тФЬтФАтФА channel_identity_id тЖТ WhatsAppUser.id, InstagramUser.id, etc.
тФЬтФАтФА status (active, inactive, blocked, unsubscribed)
тФЬтФАтФА is_primary
тФЬтФАтФА Opt-in: marketing_opt_in, transactional_opt_in
тФЬтФАтФА Engagement: first_contact_at, last_message_at, message_count
тФФтФАтФА Verification: is_verified, verified_at
```

**Key Rules**:
- Similar pattern to `person_email`, `person_phone`, `person_address`
- Channel belongs to Person, not Lead (Person-centric architecture)
- Lead can reference `source_channel_id` for attribution (which channel it came through)
- `channel_identity_id` links to detailed channel entities (WhatsAppUser, InstagramUser)
- One Person can have multiple channels of same type (e.g., multiple WhatsApp numbers)

### PersonNotDuplicate (deduplication overrides)

**Purpose**: Records pairs of persons that have been manually confirmed as NOT duplicates. Prevents the deduplication system from repeatedly flagging the same pair.

```
person_not_duplicate
тФЬтФАтФА person1_id тЖТ Person
тФЬтФАтФА person2_id тЖТ Person
тФЬтФАтФА marked_by тЖТ User (who made the decision)
тФЬтФАтФА marked_at (when the decision was made)
тФЬтФАтФА organization_id тЖТ Organization
тФЬтФАтФА Unique constraint: (person1, person2, organization)
```

**Key Rules**:
- Symmetric: if (A, B) is marked as not-duplicate, the system treats (B, A) equivalently
- Organization-scoped: each org maintains its own override decisions
- Used by `PersonNotDuplicateService` to exclude pairs from duplicate detection

### Person merge system

**Purpose**: Consolidates duplicate persons into a single primary record, reassigning all related data.

**API Endpoint**: `POST /persons/:primaryPersonId/merge`

**Merge Workflow**:

<Steps>
<Step title="Validation">
- Verify primary person exists and is not deleted
- Verify all secondary persons exist and are not deleted
- All persons must be in the same organization
</Step>

<Step title="Field selection">
- Accept fieldSelections: Record<string, string>
- For each field, pick the value from the specified source person
- Fields not listed default to primary person's values
</Step>

<Step title="Contact info merge">
- mergeAllEmails: boolean тАФ reassign all secondary emails to primary
- mergeAllPhones: boolean тАФ reassign all secondary phones to primary  
- mergeAllAddresses: boolean тАФ reassign all secondary addresses to primary
- isPrimary flags reset to false for transferred records
</Step>

<Step title="Channel and entity reassignment">
- All person_channels reassigned to primary person
- All leads, contacts, and related CRM entities updated
- Secondary persons marked with merged_into_id and merged_at
</Step>
</Steps>

- Contact visibility derived from Person's leads (via stakeholders)

### Deal (revenue opportunities)

**Purpose**: Represents qualified sales opportunities with revenue potential.

```

тФЬтФАтФА title, value, currency, commission_rate
```

**Purpose**: Manages many-to-many relationship between deals and contacts.

```

тФФтФАтФА organization_id
```

**Key Rules**:
- Exactly one primary contact per deal (is_primary = true)
- Primary contact cannot be removed until another is designated primary
- Role describes the contact's involvement in the deal

## Assignment & commission system

The CRM uses a unified stakeholder model where both leads and deals share the same assignment and commission structure.

### EntityStakeholder (unified assignment)

**Purpose**: Single table for assignment and commission across leads/deals.

```
entity_stakeholder
тФЬтФАтФА entity_type (lead | deal)
тФЬтФАтФА entity_id (UUID pointing to lead or deal)
тФЬтФАтФА user_id (nullable), team_id (nullable)
тФЬтФАтФА is_primary, access_level (READ | WRITE | FULL)
тФЬтФАтФА commission_percentage, original_team_commission
тФЬтФАтФА inherited_from_stakeholder_id, inherited_from_lead_id
тФЬтФАтФА created_at, updated_at
тФФтФАтФА Unique constraint: (entity_type, entity_id, user_id, team_id)
```

**Business Rules**:
- Commission percentages must sum to exactly 0% or 100% (0.01% tolerance)
- At least one `isPrimary = true` stakeholder must exist when stakeholders are present
- Observers (0% commission) cannot be primary
- Stakeholders inherit from lead to deal during conversion

### Commission payments

**Purpose**: Tracks actual commission payouts when deals close.

```
commission_payment
тФЬтФАтФА deal_id тЖТ Deal
тФЬтФАтФА stakeholder_id тЖТ EntityStakeholder
тФЬтФАтФА amount (calculated from deal value ├Ч commission rate ├Ч percentage)
тФЬтФАтФА currency
тФЬтФАтФА status (pending | paid | on_hold | cancelled)
тФЬтФАтФА paid_at, payment_reference
тФЬтФАтФА hold_reason, hold_until
тФФтФАтФА organization_id
```

**Business Rules**:
- Commission payments are created when deal closes with `is_won = true`
- Amount calculated as: `deal.value * deal.commission_rate * stakeholder.commission_percentage / 10000`
- Payments can be held for approval or legal reasons
- Payment status tracked for accounting integration

## Transfer system

The transfer system allows stakeholders to move leads or deals between agents with configurable commission retention.

### EntityTransfer

**Purpose**: Manages transfer requests with approval workflow.

```
entity_transfer
тФЬтФАтФА entity_type (lead | deal)
тФЬтФАтФА entity_id тЖТ Lead or Deal
тФЬтФАтФА from_user_id тЖТ User (current owner)
тФЬтФАтФА to_user_id тЖТ User (intended new owner)
тФЬтФАтФА retain_commission_percentage (0-100)
тФЬтФАтФА reason, notes
тФЬтФАтФА status (pending | approved | rejected | cancelled)
тФЬтФАтФА requested_at, approved_at, approved_by
тФЬтФАтФА completed_at
тФФтФАтФА organization_id
```

### Transfer workflow

<Steps>
<Step title="Request transfer">
Current primary stakeholder creates transfer request specifying commission retention (0-100%).
</Step>

<Step title="Approval">
Target user or manager approves/rejects the transfer request.
</Step>

<Step title="Execute transfer">
System updates stakeholders:
- Original owner: reduced to retained percentage, `is_primary = false`
- New owner: gets remaining percentage, `is_primary = true`
- All stakeholders get inheritance tracking
</Step>

<Step title="Auto-cancel">
Pending transfers are automatically cancelled if the entity is archived or deleted.
</Step>
</Steps>

## Activity & communication system

The activity system provides a unified timeline across all CRM entities while keeping detailed communication data in specialized channel tables.

### Activity (timeline index)

**Purpose**: Lightweight timeline index for all entity interactions.

```
activity
тФЬтФАтФА target_entity_type (lead | deal | contact | company)
тФЬтФАтФА target_entity_id (UUID)
тФЬтФАтФА contact_id (nullable тАФ for bubble-up visibility)
тФЬтФАтФА activity_type (call | email | meeting | site_visit | note | task)

тФЬтФАтФА created_at, created_by_id
тФФтФАтФА organization_id
```

### Channel tables

Detailed communication data is stored in specialized tables:

- `call_log` тАФ Phone call records
- `email_log` тАФ Email communications  
- `sms_log` тАФ SMS messages
- `meeting` тАФ Meeting records
- `site_visit` тАФ Property visits
- `note` тАФ Notes and comments

**Key Rules**:
- Activity table indexes the timeline; channel tables store full data
- `reference_id` links activity to channel table record
- Channel tables can exist without Activity records (internal tracking)
- Activity records always have corresponding channel records

## Notes system

### Note entity

**Purpose**: Pure content storage with polymorphic targeting via Activity table.

```
note
тФЬтФАтФА title (optional), body
тФЬтФАтФА attachments[] (JSONB: {url, filename, size, mimeType})

тФЬтФАтФА created_at, updated_at
```

### Entity targeting

Notes target entities through the Activity system:
1. Create `note` record
2. Create `activity` record with `activity_type = 'note'` and `reference_id = note.id`
3. Set `target_entity_type` and `target_entity_id` on activity

**Benefits**:
- Consistent timeline ordering with other activities
- Unified permissions through activity targeting
- Notes can be pinned/flagged independently of activity metadata

## Stage history & analytics

The CRM tracks pipeline progression for both leads and deals with detailed stage history.

### Stage configuration

```sql
lead_stage / deal_stage
тФЬтФАтФА name, color, order
тФЬтФАтФА system_type (enum: new | qualified | converted | closed | disqualified)
тФЬтФАтФА next_stage_id (nullable)
тФЬтФАтФА organization_id
тФФтФАтФА is_active
```

### Stage history

**Purpose**: Records completed stages only (not current stage).

```sql
lead_stage_history / deal_stage_history
тФЬтФАтФА lead_id / deal_id
тФЬтФАтФА stage_id (the stage we LEFT)
тФЬтФАтФА entered_at, duration_seconds
тФЬтФАтФА next_stage_id
тФЬтФАтФА notes
тФФтФАтФА created_by_id
```

**Key Rules**:
- Current stage is stored on `lead.stage_id` / `deal.stage_id` with `stage_entered_at`
- History only contains stages we've LEFT, never the current stage
- Duration is calculated when leaving a stage
- Analytics aggregate history records for pipeline reporting

### Analytics queries

Common stage analytics patterns:

```sql
-- Average time in each stage
SELECT 
  s.name,
  AVG(h.duration_seconds / 86400.0) as avg_days
FROM lead_stage_history h
JOIN lead_stage s ON h.stage_id = s.id
GROUP BY s.id, s.name;

-- Conversion rates between stages  
SELECT 
  from_stage.name as from_stage,
  to_stage.name as to_stage,
  COUNT(*) as transitions
FROM lead_stage_history h
JOIN lead_stage from_stage ON h.stage_id = from_stage.id
JOIN lead_stage to_stage ON h.next_stage_id = to_stage.id
GROUP BY from_stage.id, to_stage.id;
```

## Query patterns

### Person-centric queries

```sql
-- All CRM entities for a person
SELECT p.*, 
  l.id as lead_id, l.stage_id as lead_stage,
  c.id as contact_id, c.rating as contact_rating,
  d.id as deal_id, d.value as deal_value
FROM person p
LEFT JOIN lead l ON l.person_id = p.id AND l.is_deleted = false
LEFT JOIN contact c ON c.person_id = p.id AND c.is_deleted = false
LEFT JOIN deal_contact dc ON dc.contact_id = c.id
LEFT JOIN deal d ON d.id = dc.deal_id AND d.is_deleted = false
WHERE p.id = :person_id;
```

### Stakeholder visibility

```sql
-- Contacts visible to current user
SELECT DISTINCT c.* 
FROM contact c
JOIN person p ON c.person_id = p.id
JOIN lead l ON l.person_id = p.id
JOIN entity_stakeholder es ON es.entity_type = 'lead' AND es.entity_id = l.id
WHERE es.user_id = :current_user_id
  AND es.access_level IN ('WRITE', 'FULL');
```

### Commission calculations

```sql
-- Commission breakdown for closed deals
SELECT 
  d.id as deal_id,
  d.value * d.commission_rate / 100 as total_commission,
  u.first_name + ' ' + u.last_name as stakeholder,
  es.commission_percentage,
  (d.value * d.commission_rate * es.commission_percentage / 10000) as amount
FROM deal d
JOIN entity_stakeholder es ON es.entity_type = 'deal' AND es.entity_id = d.id
JOIN user u ON es.user_id = u.id
WHERE d.is_closed = true AND d.is_won = true;
```

## Business rules

### Data integrity rules

1. **Unique constraints**: Person emails, phone numbers, and channel identifiers must be unique within organization
2. **Soft deletes**: All entities use `is_deleted` flag; hard deletes are prohibited
3. **Cascade behavior**: Deleting person cascades to all leads/contacts; deleting lead nullifies deal references
4. **Commission validation**: Stakeholder percentages must sum to 0% or 100% (┬▒0.01% tolerance)
5. **Primary stakeholder**: At least one `is_primary = true` stakeholder required when stakeholders exist

### Business logic rules

1. **Lead conversion**: Converting lead creates contact + deal; stakeholders inherit to deal
2. **Person merge**: Secondary persons are marked with `merged_into_id`; all related data reassigns to primary
3. **Transfer approval**: Only primary stakeholders or managers can initiate transfers
4. **Commission inheritance**: Deal stakeholders inherit from originating lead with audit trail
5. **Access control**: Users see entities based on stakeholder relationships and team hierarchies

### Validation rules

1. **Email format**: All email addresses validated against RFC 5322 standard
2. **Phone format**: Phone numbers normalized to E.164 international format
3. **Commission range**: Individual percentages must be 0-100%; total must be 0% or 100%
4. **Stage progression**: Stages must follow configured `next_stage_id` relationships
5. **Entity references**: All foreign keys must reference existing, non-deleted records

## Data consistency guarantees

The CRM module enforces several invariants across all operations:

| Guarantee | Mechanism | Details |
| --- | --- | --- |
| Transactional writes | `TenantContext.executeInOrg()` | Every write runs in a database transaction. Any failure rolls back the entire operation. |
| Soft-delete everywhere | `isDeleted` flag | No entity is hard-deleted. All deletions set `isDeleted = true`. |
| Unique DB constraints | PostgreSQL unique indexes | Race conditions are caught by database-level constraints (stakeholder uniqueness, transfer uniqueness, lead deduplication). Error code 23505 is handled gracefully. |
| Commission invariant | `Math.abs(total - 100) > 0.01` | Commission percentages must sum to exactly 0% or 100% with 0.01% floating-point tolerance. Enforced on every stakeholder mutation. |
| Primary invariant | Pre-delete + post-create checks | At least one `isPrimary = true` stakeholder must exist when stakeholders are present. Observers (0% commission) cannot be primary. |
| Cascade cleanup | Archive/delete side effects | Archiving cancels pending transfers. Soft-deleting cascades to all child records. Deal `originatingLead` references are set to NULL on lead deletion. |

### End-to-end lifecycle

<Steps>
<Step title="First contact">
Inquiry comes in (Bayut, website, etc.). Person is created (if not already existing). Lead is created and linked to Person. Stakeholder created (Agent A = primary, 100%).
</Step>
<Step title="Lead worked">
Agent A works the lead. Activities and notes are added. Lead moves through pipeline stages.
</Step>
<Step title="Lead converted">
Lead moves to "Converted" stage. Contact is created, linked to the same Person. Deal is created, linked to the lead. All stakeholders are copied from lead to deal.
</Step>
<Step title="Ongoing relationship">
Person now has: Lead (converted) + Contact + Deal. A new inquiry creates a new Lead for the same Person. Contact accumulates deals, lifetime value, and history.
</Step>
</Steps>
