</Warning>

</Info>

## Vocabulary

| Term | Definition |
|------|------------|
| **Personal row** | `userId` set, `teamId` empty. Grants one user access directly. |
| **Team pool / team-only row** | `teamId` set, `userId` empty. Represents team-level access and, for leads/deals, the team commission budget / leftover row. |
| **Team-member row** | Both `userId` and `teamId` set. Represents a user assigned under a specific team context. |
| **Primary row** | Exactly one active lead/deal stakeholder row marked `isPrimary=true` whenever stakeholders exist. Contacts always use `isPrimary=false`. |
| **Access level** | `READ`, `WRITE`, or `FULL`. Determines what a stakeholder can do to the entity. |
| **Access-only (contact)** | Contact stakeholders grant permanent contact access only. No commission percentages and no primary assignee in UI or API. |
| **Agency cut** | `orgCommissionPercentage`. Only users with `finance.manage` can set it. Frontend displays the remaining agent pool as the allocatable percentage. |
| **Management scope** | Backend-computed stakeholder editing scope: `org`, `team`, `self`, or `none`. |

## Structural Validation Rules

<Note>
These rules are mirrored in frontend `stakeholder-validation.ts` and backend `stakeholder-validation.utils.ts`.
</Note>

### Core Rules

<Steps>
  <Step title="Entry Requirements">
    - At least one entry required (unless lead moved to pool by allowed sync path)
    - Every entry must identify someone (`userId`, `teamId`, or both)
  </Step>

  <Step title="Primary Assignment">
    - Exactly one primary when stakeholders exist
    - Primary cannot be READ access level
    - Frontend auto-promotes READ to WRITE when row becomes primary
  </Step>

  <Step title="Identity & Structure">
    - No duplicate row identity (same `userId, teamId` combination)
    - `originalTeamCommission` only on team pool rows
    - Team pool row requires commission budget
    - Team member requires matching pool
  </Step>
</Steps>

### Access & Commission Rules

<Tabs>
  <Tab title="Access Levels">
    - Member access cannot exceed team access
    - Team-member row cannot have higher access than parent team pool row
  </Tab>

  <Tab title="Commission Validation">
    - No negative commissions (`commissionPercentage >= 0`)
    - DTOs cap commissions at `<= 100`
    - Total commission must match target
    - Top-level allocation must match target
    - Team members cannot exceed budget
    - Team leftover must be exact: `originalTeamCommission - sum(member commissions)`
  </Tab>
</Tabs>

### Lead-Specific Exceptions

<Warning>
Leads differ from deals in commission handling:
</Warning>

- **Leads** allow a `0%` commission total (supports early assignment before deal exists)
- **Deals** require `100%` commission total
- **Contacts** have no commission system (access-only)

## Implementation Guidelines

<Check>
**Frontend Implementation**
- Use shared components for consistent validation
- Mirror backend authorization scopes
- Fail closed when permissions are absent
- Gate contact-specific behavior with `isAccessOnly`
</Check>

<Check>
**Backend Implementation**
- Enforce structural validation at service level
- Apply authorization at both create and sync time
- Maintain referential integrity between team pools and members
- Compute accurate permission scopes for frontend
</Check>
