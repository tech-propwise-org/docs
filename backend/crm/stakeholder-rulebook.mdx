The backend is the authority. Frontend checks are UX mirrors only. If frontend and backend disagree, the backend wins.
</Warning>

## Entity Applicability

| Entity  | Commission / Primary | Mutation API                       | Sections in this doc                                     |
| ------- | -------------------- | ---------------------------------- | -------------------------------------------------------- |
| Lead    | Yes                  | `PUT /leads/:id/stakeholders` sync | Sections 3–20 (primary focus)                            |
| Deal    | Yes (100% floor)     | `PUT /deals/:id/stakeholders` sync | Same structural/scope rules as lead unless noted in §3   |
| Contact | **No** (access-only) | Granular add / update / remove     | **§21** and §3 contact exceptions — not lead self-assign |

<Info>
Deal stakeholder rules mirror lead rules for commission, primary, and sync authorization unless `STAKEHOLDER_SYSTEM.md` states a deal-only invariant (for example deals cannot have zero stakeholders).
</Info>

## Vocabulary

<CardGroup cols={2}>
  <Card title="Row Types">
    **Personal row:** `userId` set, `teamId` empty. Direct user access.
    
    **Team pool row:** `teamId` set, `userId` empty. Team-level access and commission budget.
    
    **Team-member row:** Both `userId` and `teamId` set. User assigned under team context.
  </Card>
  
  <Card title="Special Attributes">
    **Primary row:** Exactly one active lead/deal stakeholder marked `isPrimary=true`.
    
    **Access level:** `READ`, `WRITE`, or `FULL` permissions.
    
    **Access-only:** Contact stakeholders grant permanent access only (no commission).
  </Card>
</CardGroup>

## Structural Validation Rules

These rules are mirrored in frontend `stakeholder-validation.ts` and backend `stakeholder-validation.utils.ts`.

### Basic Structure Requirements

<Check>**At least one entry** - Stakeholder payload must contain entries unless lead is moved to pool by allowed sync path</Check>

<Check>**Every entry identifies someone** - Each row must have `userId`, `teamId`, or both</Check>

<Check>**Exactly one primary** - When stakeholders exist, exactly one row must be primary</Check>

<Check>**Primary cannot be READ** - Primary row must be `WRITE` or `FULL` access level</Check>

### Identity and Commission Rules

<Tabs>
  <Tab title="Identity Rules">
    - **No duplicate row identity** - Same `(userId, teamId)` combination cannot appear twice
    - **Team member requires pool** - `userId + teamId` row requires matching `teamId`-only row
    - **Member access cannot exceed team access** - Team-member row cannot have higher access than parent team pool
  </Tab>
  
  <Tab title="Commission Rules">
    - **No negative commissions** - `commissionPercentage` must be `>= 0` and `<= 100`
    - **Total commission must match target** - Leads allow `0%` or configured target; deals require `100%`
    - **Top-level allocation must match target** - Standalone users + team budgets must equal target
    - **Team members cannot exceed budget** - Sum of member rows ≤ team budget
    - **Team leftover must be exact** - Team pool percentage = `originalTeamCommission - sum(member commissions)`
  </Tab>
</Tabs>

### Team-Specific Requirements

<Warning>
- **`originalTeamCommission` only on team pool rows** - Only rows with `teamId` and no `userId`
- **Team pool row requires budget** - Team-only row must have `originalTeamCommission`
</Warning>

### Lead vs Deal vs Contact Exceptions

<CodeGroup>
```typescript Lead Rules
// Leads allow 0% commission total
// Supports early assignment before deal exists
totalCommission: 0 | configuredTarget
```

```typescript Deal Rules  
// Deals require 100% commission allocation
totalCommission: 100
// Cannot have zero stakeholders
stakeholders.length >= 1
```

```typescript Contact Rules
// Access-only mode - no commission
isPrimary: false (always)
commissionPercentage: undefined
// Cannot self-assign
userId !== requestingUserId
```
</CodeGroup>

## Authorization Scopes

The backend computes management scope based on user permissions and entity relationships:

<Steps>
  <Step title="Org Scope">
    User has `crm.manage` permission. Can modify any stakeholder row.
  </Step>
  
  <Step title="Team Scope">
    User can manage stakeholders within their team(s). Limited to team-member and team-pool rows.
  </Step>
  
  <Step title="Self Scope">
    User can only modify their own personal stakeholder row.
  </Step>
  
  <Step title="None Scope">
    User cannot modify stakeholders. Read-only access.
  </Step>
</Steps>

## Frontend Permission Mapping

The frontend uses `lead.permissions` from backend to determine UI capabilities:

```typescript
interface LeadPermissions {
  canModify: boolean;
  canDelete: boolean;
  canTransfer: boolean;
  canManageStakeholders: boolean;
  stakeholderManagementScope: 'org' | 'team' | 'self' | 'none';
  managedTeamIds?: number[];
}
```

<Tip>
Frontend permissions fail closed when backend permissions are absent. Always check for permission existence before allowing actions.
</Tip>

## Access Level Hierarchy

Access levels determine stakeholder capabilities on the entity:

| Level | Capabilities | Can be Primary |
|-------|-------------|----------------|
| `READ` | View entity only | ❌ No |
| `WRITE` | View + edit entity | ✅ Yes |
| `FULL` | View + edit + manage stakeholders | ✅ Yes |

<Note>
When a READ-level stakeholder becomes primary, the frontend automatically promotes them to WRITE level.
</Note>

## Team Commission Allocation

Team commission allocation follows a budget-based model:

<Steps>
  <Step title="Set Team Budget">
    Create team pool row with `originalTeamCommission` (e.g., 50%)
  </Step>
  
  <Step title="Assign Team Members">
    Add team-member rows with individual commission percentages
  </Step>
  
  <Step title="Calculate Leftover">
    Team pool `commissionPercentage` = `originalTeamCommission - sum(member commissions)`
  </Step>
  
  <Step title="Validate Constraints">
    - Member sum ≤ team budget
    - Team leftover ≥ 0%
    - Total allocation matches target
  </Step>
</Steps>

## Agency Cut Handling

<Warning>
Only users with `finance.manage` permission can set the agency cut (`orgCommissionPercentage`).
</Warning>

The frontend displays the remaining agent pool as the allocatable percentage:

```typescript
const allocatablePercentage = 100 - (orgCommissionPercentage || 0);
```

## Contact-Specific Rules

Contacts use access-only stakeholder mode with unique restrictions:

<AccordionGroup>
  <Accordion title="Access-Only Behavior">
    - No commission percentages
    - No primary assignee concept
    - Permanent access grants only
    - Uses granular add/update/remove APIs instead of sync
  </Accordion>
  
  <Accordion title="Self-Assignment Restrictions">
    - Users cannot assign themselves to contacts
    - Prevents `userId === requestingUserId` scenarios
    - Team assignments still allowed for contact access
  </Accordion>
  
  <Accordion title="API Differences">
    - `ContactApi.addStakeholder()`
    - `ContactApi.updateStakeholder()`  
    - `ContactApi.removeStakeholder()`
    - No bulk sync endpoint like leads/deals
  </Accordion>
</AccordionGroup>

## Error Handling

Common validation errors and their meanings:

| Error | Cause | Resolution |
|-------|-------|------------|
| `DUPLICATE_STAKEHOLDER_IDENTITY` | Same `(userId, teamId)` appears twice | Remove duplicate entries |
| `MISSING_PRIMARY_STAKEHOLDER` | No row marked as primary | Set exactly one row as primary |
| `PRIMARY_CANNOT_BE_READ_ONLY` | Primary stakeholder has READ access | Promote to WRITE or FULL |
| `COMMISSION_TOTAL_MISMATCH` | Commission total doesn't match target | Adjust percentages to match target |
| `TEAM_BUDGET_EXCEEDED` | Team members exceed team budget | Reduce member allocations or increase budget |
| `MISSING_TEAM_POOL` | Team-member row without parent team | Add team pool row with budget |

<Tip>
The frontend stakeholder assignment section provides real-time validation feedback to prevent these errors before submission.
</Tip>
