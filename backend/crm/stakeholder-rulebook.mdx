  **Status:** Reference document

</Note>

This document is the comprehensive practical map of CRM stakeholder rules. In user-facing copy, "stakeholder" is rendered as **assignee** / **assignment**, but the internal API, entity, DTO, and service names still use "stakeholder".

<Warning>
  The backend is the authority. Frontend checks are UX mirrors only. If frontend and backend disagree, the backend wins.
</Warning>

## Entity applicability

| Entity  | Commission / primary | Mutation API                       | Coverage in this doc                                     |
| ------- | -------------------- | ---------------------------------- | -------------------------------------------------------- |
| Lead    | Yes                  | `PUT /leads/:id/stakeholders` sync | Sections 3–20 (primary focus)                            |
| Deal    | Yes (100% floor)     | `PUT /deals/:id/stakeholders` sync | Same structural/scope rules as lead unless noted         |
| Contact | **No** (access-only) | Granular add / update / remove     | Section 21 and contact-specific exceptions               |

<Info>
  Deal stakeholder rules mirror lead rules for commission, primary, and sync authorization unless the canonical engineering spec states a deal-only invariant (for example, deals cannot have zero stakeholders).
</Info>

## Vocabulary

| Term                      | Meaning                                                                                                                                           |
| ------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------- |
| Personal row              | `userId` set, `teamId` empty. Grants one user access directly.                                                                                    |
| Team pool / team-only row | `teamId` set, `userId` empty. Represents team-level access and, for leads/deals, the team commission budget / leftover row.                       |
| Team-member row           | Both `userId` and `teamId` set. Represents a user assigned under a specific team context.                                                         |
| Primary row               | Exactly one active lead/deal stakeholder row marked `isPrimary=true` whenever stakeholders exist. Contacts always use `isPrimary=false`.          |
| Access level              | `READ`, `WRITE`, or `FULL`. Determines what a stakeholder can do to the entity.                                                                   |
| Access-only (contact)     | Contact stakeholders grant permanent contact access only. No commission percentages and no primary assignee in UI or API.                         |
| Agency cut                | `orgCommissionPercentage`. Only users with `finance.manage` can set it. Frontend displays the remaining agent pool as the allocatable percentage. |
| Management scope          | Backend-computed stakeholder editing scope: `org`, `team`, `self`, or `none`.                                                                     |

## Structural validation rules

These rules are mirrored in frontend `stakeholder-validation.ts` and backend `stakeholder-validation.utils.ts`.

<Warning>
  All validation rules must pass before a stakeholder payload is accepted by the backend.
</Warning>

### Core validation rules

<AccordionGroup>
  <Accordion title="At least one entry">
    A stakeholder payload must contain entries unless the lead is intentionally moved to pool by an allowed sync path.
  </Accordion>
  
  <Accordion title="Every entry identifies someone">
    Each row must have `userId`, `teamId`, or both.
  </Accordion>
  
  <Accordion title="Exactly one primary">
    Whenever stakeholders exist, exactly one row must be primary.
  </Accordion>
  
  <Accordion title="Primary cannot be READ">
    A primary row must be `WRITE` or `FULL`. The frontend auto-promotes READ to WRITE when a row becomes primary.
  </Accordion>
  
  <Accordion title="No duplicate row identity">
    The same `(userId, teamId)` combination cannot appear twice.
  </Accordion>
  
  <Accordion title="originalTeamCommission only on team pool rows">
    Only rows with `teamId` and no `userId` may carry `originalTeamCommission`.
  </Accordion>
  
  <Accordion title="Team pool row requires budget">
    A team-only row must have `originalTeamCommission`.
  </Accordion>
  
  <Accordion title="Team member requires pool">
    A `userId + teamId` row requires a matching `teamId`-only row.
  </Accordion>
  
  <Accordion title="Member access cannot exceed team access">
    A team-member row cannot have higher access than its parent team pool row.
  </Accordion>
  
  <Accordion title="No negative commissions">
    `commissionPercentage` must be `>= 0`. DTOs also cap it at `<= 100`.
  </Accordion>
  
  <Accordion title="Total commission must match target">
    Leads allow `0%` or the configured target. Deals require `100%`.
  </Accordion>
  
  <Accordion title="Top-level allocation must match target">
    Standalone users plus team budgets must equal the target, or `0%` when the lead-specific zero-total exception applies.
  </Accordion>
  
  <Accordion title="Team members cannot exceed budget">
    Sum of member rows under a team must be less than or equal to the team budget.
  </Accordion>
  
  <Accordion title="Team leftover must be exact">
    Team pool `commissionPercentage` must equal `originalTeamCommission - sum(member commissions)`, clamped at `0`.
  </Accordion>
</AccordionGroup>

### Lead-specific exceptions

<Info>
  Leads differ from deals in commission requirements:
  
  - **Leads** allow a `0%` commission total. This supports early assignment before a deal exists.
  - **Deals** require exactly `100%` commission allocation at all times.
</Info>

## Row types and structure

### Personal row

A personal row grants access directly to one user without team context.

```json
{
  "userId": "user-123",
  "teamId": null,
  "accessLevel": "WRITE",
  "commissionPercentage": 25.0,
  "isPrimary": false
}
```

<Check>
  **Valid use case:** Independent agent or user outside team structure receiving direct commission allocation.
</Check>

### Team pool row

A team pool row represents the team's commission budget and team-level access.

```json
{
  "userId": null,
  "teamId": "team-456",
  "accessLevel": "WRITE",
  "commissionPercentage": 15.0,
  "originalTeamCommission": 40.0,
  "isPrimary": false
}
```

<Note>
  The `commissionPercentage` represents the **leftover** budget after team members are allocated. The `originalTeamCommission` is the initial team budget.
</Note>

### Team-member row

A team-member row assigns a specific user within a team context.

```json
{
  "userId": "user-789",
  "teamId": "team-456",
  "accessLevel": "WRITE",
  "commissionPercentage": 25.0,
  "isPrimary": true
}
```

<Warning>
  Every team-member row requires a matching team pool row with the same `teamId`.
</Warning>

## Access levels

Access levels determine what actions a stakeholder can perform on the entity.

| Level   | Permissions                                   | Can be primary? |
| ------- | --------------------------------------------- | --------------- |
| `READ`  | View entity and basic details                 | ❌ No           |
| `WRITE` | View and edit entity                          | ✅ Yes          |
| `FULL`  | View, edit, delete, and manage stakeholders   | ✅ Yes          |

<Tip>
  When a `READ` row is made primary, the frontend automatically promotes it to `WRITE` before submission.
</Tip>

### Team-member access capping

```typescript
// Team pool has WRITE access
const teamPoolAccess = "WRITE";

// Team member cannot have FULL if pool has WRITE
const memberAccess = "WRITE"; // ✅ Valid
const invalidAccess = "FULL"; // ❌ Invalid - exceeds team pool
```

## Primary assignee rules

<Steps>
  <Step title="Exactly one primary required">
    When stakeholders exist on a lead or deal, exactly one row must have `isPrimary: true`.
  </Step>
  
  <Step title="Primary cannot be READ">
    The primary row must have `WRITE` or `FULL` access level.
  </Step>
  
  <Step title="Frontend auto-promotion">
    If a READ row is selected as primary, the frontend promotes it to WRITE before sending to backend.
  </Step>
  
  <Step title="Contacts have no primary">
    Contact stakeholders are access-only and never use `isPrimary: true`.
  </Step>
</Steps>

## Commission allocation

### Agency cut (organization commission)

<Note>
  Only users with `finance.manage` permission can set the agency cut (`orgCommissionPercentage`).
</Note>

The agency cut reduces the allocatable commission pool:

```typescript
const totalDealValue = 100000;
const agencyCut = 10; // 10% to organization

const allocatablePercentage = 100 - agencyCut; // 90%
const allocatableDollars = totalDealValue * (allocatablePercentage / 100); // $90,000
```

### Commission calculation rules

<Tabs>
  <Tab title="Leads">
    ```json
    {
      "totalCommission": "0% or target%",
      "allowedValues": ["0", "target"],
      "example": {
        "target": 100,
        "allowed": [0, 100]
      }
    }
    ```
    
    Leads support `0%` total allocation for early assignment before deal creation.
  </Tab>
  
  <Tab title="Deals">
    ```json
    {
      "totalCommission": "100%",
      "required": true,
      "example": {
        "total": 100,
        "agencyCut": 10,
        "agentPool": 90,
        "allocated": 90
      }
    }
    ```
    
    Deals require exactly `100%` allocation at all times.
  </Tab>
</Tabs>

### Top-level allocation formula

```typescript
const topLevelTotal = 
  personalRows.reduce((sum, row) => sum + row.commissionPercentage, 0) +
  teamPoolRows.reduce((sum, row) => sum + row.originalTeamCommission, 0);

// Must equal target or 0 (leads only)
assert(topLevelTotal === target || (isLead && topLevelTotal === 0));
```

### Team budget allocation

```typescript
// Team pool defines the budget
const teamBudget = 40.0;

// Members allocated under the team
const memberAllocations = [
  { userId: "user-1", commission: 15.0 },
  { userId: "user-2", commission: 10.0 }
];

const memberTotal = 25.0;
const teamLeftover = teamBudget - memberTotal; // 15.0

// Team pool commission must equal leftover
assert(teamPoolRow.commissionPercentage === Math.max(0, teamLeftover));
```

<Warning>
  The sum of team-member commissions cannot exceed the team's `originalTeamCommission` budget.
</Warning>

## Management scopes

The backend computes stakeholder management scope based on user permissions and existing stakeholder relationships.

### Scope levels

<CardGroup cols={2}>
  <Card title="Organization scope" icon="building">
    User can manage all stakeholders on the entity.
    
    **Requirements:**
    - User has `crm.manage` permission, OR
    - User has `FULL` access to the entity
  </Card>
  
  <Card title="Team scope" icon="users">
    User can manage stakeholders from their teams.
    
    **Requirements:**
    - User has `crm.modify` permission
    - Can manage rows from teams they belong to
  </Card>
  
  <Card title="Self scope" icon="user">
    User can only manage their own stakeholder row.
    
    **Requirements:**
    - User has `FULL` access to the entity
    - Can only edit their personal row
  </Card>
  
  <Card title="None" icon="ban">
    User cannot manage stakeholders.
    
    **Conditions:**
    - User lacks necessary permissions
    - No stakeholder management access
  </Card>
</CardGroup>

### Scope authorization matrix

| User Permission | Stakeholder Access | Computed Scope | Can Manage                        |
| --------------- | ------------------ | -------------- | --------------------------------- |
| `crm.manage`    | Any                | `org`          | All rows                          |
| `crm.modify`    | `FULL`             | `org`          | All rows                          |
| `crm.modify`    | `WRITE`/`READ`     | `team`         | Rows from user's teams            |
| `crm.view`      | `FULL`             | `self`         | Only user's personal row          |
| `crm.view`      | `WRITE`/`READ`     | `none`         | No management access              |

## Create-time authorization

<Note>
  Source: `src/modules/crm/shared/stakeholder-create-authorization.util.ts`
</Note>

### Organization scope creation

Users with `crm.manage` or creator with `FULL` access can create any valid stakeholder structure.

```typescript
// ✅ Valid: Org-scoped creator
{
  requester: { userId: "creator-123", permissions: ["crm.manage"] },
  stakeholders: [
    { userId: "user-1", accessLevel: "WRITE", commissionPercentage: 50, isPrimary: true },
    { userId: "user-2", accessLevel: "READ", commissionPercentage: 50, isPrimary: false }
  ]
}
```

### Member self-assignment

Regular members (`crm.modify` or `crm.view`) can only create entities if they assign themselves with appropriate access.

```typescript
// ✅ Valid: Member self-assigns with FULL
{
  requester: { userId: "user-123", permissions: ["crm.modify"] },
  stakeholders: [
    { 
      userId: "user-123", 
      accessLevel: "FULL", 
      commissionPercentage: 100, 
      isPrimary: true 
    }
  ]
}

// ❌ Invalid: Member tries to assign someone else
{
  requester: { userId: "user-123", permissions: ["crm.modify"] },
  stakeholders: [
    { userId: "user-456", accessLevel: "WRITE", commissionPercentage: 100, isPrimary: true }
  ]
}
```

<Warning>
  Members must assign themselves with `FULL` access when creating leads or deals. `WRITE` or `READ` self-assignment is rejected.
</Warning>

### Team context rules

```typescript
// ✅ Valid: Member assigns self under their team
{
  requester: { 
    userId: "user-123", 
    teamIds: ["team-456"],
    permissions: ["crm.modify"] 
  },
  stakeholders: [
    { teamId: "team-456", accessLevel: "WRITE", originalTeamCommission: 100 },
    { 
      userId: "user-123", 
      teamId: "team-456", 
      accessLevel: "FULL", 
      commissionPercentage: 100,
      isPrimary: true 
    }
  ]
}

// ❌ Invalid: Member assigns under team they don't belong to
{
  requester: { 
    userId: "user-123", 
    teamIds: ["team-456"],
    permissions: ["crm.modify"] 
  },
  stakeholders: [
    { teamId: "team-789", accessLevel: "WRITE", originalTeamCommission: 100 },
    { 
      userId: "user-123", 
      teamId: "team-789", 
      accessLevel: "FULL", 
      commissionPercentage: 100,
      isPrimary: true 
    }
  ]
}
```

## Sync-time authorization

<Note>
  Source: `src/modules/crm/shared/stakeholder-sync-authorization.util.ts`
</Note>

### Organization scope sync

Users with org scope can make any valid structural changes.

```typescript
// ✅ Valid: Org-scoped user changes primary and reallocates
{
  scope: "org",
  changes: {
    added: [{ userId: "new-user", accessLevel: "WRITE", commissionPercentage: 30 }],
    updated: [{ id: "stake-1", isPrimary: true, commissionPercentage: 40 }],
    removed: ["stake-2"]
  }
}
```

### Team scope sync

Users with team scope can only modify rows from their teams.

```typescript
// ✅ Valid: Team-scoped user manages their team's rows
{
  scope: "team",
  requesterTeams: ["team-456"],
  changes: {
    updated: [
      { 
        id: "stake-1", 
        teamId: "team-456", 
        userId: "user-123",
        commissionPercentage: 50 
      }
    ]
  }
}

// ❌ Invalid: Team-scoped user tries to modify another team
{
  scope: "team",
  requesterTeams: ["team-456"],
  changes: {
    updated: [
      { 
        id: "stake-2", 
        teamId: "team-789", 
        commissionPercentage: 50 
      }
    ]
  }
}
```

<Warning>
  Team-scoped users cannot:
  - Add or remove team pool rows
  - Modify rows outside their teams
  - Change team budgets (`originalTeamCommission`)
</Warning>

### Self scope sync

Users with self scope can only modify their own personal row.

```typescript
// ✅ Valid: Self-scoped user updates their own row
{
  scope: "self",
  requesterId: "user-123",
  changes: {
    updated: [
      { 
        id: "stake-1", 
        userId: "user-123",
        teamId: null,
        isPrimary: true 
      }
    ]
  }
}

// ❌ Invalid: Self-scoped user tries to add another stakeholder
{
  scope: "self",
  requesterId: "user-123",
  changes: {
    added: [
      { userId: "user-456", accessLevel: "WRITE", commissionPercentage: 50 }
    ]
  }
}
```

<Check>
  Self-scoped users can:
  - Update their personal row details
  - Make themselves primary (if they have `WRITE` or `FULL`)
  - Adjust their commission within existing allocation
</Check>

const permissions = lead.permissions;

if (!permissions?.canManageStakeholders) {
  return <AccessDenied />;
}

const scope = permissions.stakeholderManagementScope; // "org" | "team" | "self" | "none"

### Permission checks

```typescript
// src/lib/permissions/lead-permissions.ts
export function canManageStakeholders(permissions?: LeadPermissions): boolean {
  if (!permissions) return false;
  
  return permissions.canManageStakeholders === true &&
         permissions.stakeholderManagementScope !== "none";
}

export function isOrgScopedManage(permissions?: LeadPermissions): boolean {
  return permissions?.stakeholderManagementScope === "org";
}

export function getManagementScope(permissions?: LeadPermissions): ManagementScope {
  return permissions?.stakeholderManagementScope || "none";
}
```

## Commission display and scaling

### Frontend visible percentage

The frontend displays commissions scaled to the agent pool (after agency cut).

```typescript
// src/lib/stakeholder-commission.ts
export function toVisiblePercentage(
  backendPercentage: number,
  agencyCut: number
): number {
  const agentPool = 100 - agencyCut;
  return (backendPercentage / agentPool) * 100;
}

export function toBackendPercentage(
  visiblePercentage: number,
  agencyCut: number
): number {
  const agentPool = 100 - agencyCut;
  return (visiblePercentage / 100) * agentPool;
}
```

### Example calculation

```typescript
// Deal value: $100,000
// Agency cut: 10%
// Agent pool: 90%

const stakeholder = {
  commissionPercentage: 45.0 // Backend value (45% of total deal)
};

const agencyCut = 10;
const visiblePercentage = toVisiblePercentage(45.0, 10); // 50%

// UI displays: "50% of $90,000 agent pool = $45,000"
// Backend stores: 45% of $100,000 = $45,000
```

<Info>
  The visible percentage represents the stakeholder's share of the **agent pool** after agency cut. The backend percentage represents the share of the **total deal value**.
</Info>

## Contact-specific rules

<Warning>
  Contacts use access-only stakeholder mode. Many lead/deal rules do not apply.
</Warning>

### Contact stakeholder differences

| Feature                      | Leads/Deals         | Contacts                |
| ---------------------------- | ------------------- | ----------------------- |
| Commission percentages       | ✅ Required         | ❌ Not used             |
| Primary assignee             | ✅ Exactly one      | ❌ Never used           |
| Self-assignment during create| ✅ Allowed          | ❌ Not allowed          |
| API structure                | Bulk sync           | Granular add/update/remove |
| `originalTeamCommission`     | ✅ Required for teams | ❌ Not used           |

### Contact validation rules

```typescript
// ✅ Valid contact stakeholder
{
  userId: "user-123",
  teamId: null,
  accessLevel: "WRITE",
  isPrimary: false // Always false for contacts
  // No commission fields
}

// ❌ Invalid: Contact with commission
{
  userId: "user-123",
  accessLevel: "WRITE",
  commissionPercentage: 50.0 // Not allowed on contacts
}

// ❌ Invalid: Contact with primary
{
  userId: "user-123",
  accessLevel: "WRITE",
  isPrimary: true // Not allowed on contacts
}
```

### Contact self-assignment restriction

<Steps>
  <Step title="Creation-time check">
    During contact creation, the backend validates that members cannot assign themselves.
  </Step>
  
  <Step title="Org scope required">
    Only users with `crm.manage` can assign stakeholders during contact creation.
  </Step>
  
  <Step title="Frontend enforcement">
    The contact creation modal hides the self-assignment option for non-admin users.
  </Step>
</Steps>

```typescript
// ❌ Invalid: Member tries to self-assign to new contact
{
  requester: { userId: "user-123", permissions: ["crm.modify"] },
  contactStakeholders: [
    { userId: "user-123", accessLevel: "FULL" }
  ]
}

// ✅ Valid: Admin assigns stakeholders during creation
{
  requester: { userId: "admin-456", permissions: ["crm.manage"] },
  contactStakeholders: [
    { userId: "user-123", accessLevel: "WRITE" },
    { userId: "user-789", accessLevel: "READ" }
  ]
}
```

### Contact management API

Unlike leads/deals, contacts use granular stakeholder operations:

<CodeGroup>

  userId: "user-123",
  teamId: null,

});
```

// PATCH /contacts/:id/stakeholders/:stakeholderId

// DELETE /contacts/:id/stakeholders/:stakeholderId

```

</CodeGroup>

## Common validation errors

<AccordionGroup>
  <Accordion title="No primary assignee">
    **Error:** "Exactly one stakeholder must be marked as primary"
    
    **Cause:** Payload has zero or multiple primary rows.
    
    **Fix:** Ensure exactly one row has `isPrimary: true`.
  </Accordion>
  
  <Accordion title="Primary has READ access">
    **Error:** "Primary stakeholder must have WRITE or FULL access"
    
    **Cause:** Primary row has `accessLevel: "READ"`.
    
    **Fix:** Set primary row to `"WRITE"` or `"FULL"`. Frontend auto-promotes READ to WRITE.
  </Accordion>
  
  <Accordion title="Commission total mismatch">
    **Error:** "Total commission must equal 100% (or 0% for leads)"
    
    **Cause:** Sum of all stakeholder commissions doesn't match target.
    
    **Fix:** Adjust commissions to sum to exactly 100% (deals) or 0%/target% (leads).
  </Accordion>
  
  <Accordion title="Team member exceeds budget">
    **Error:** "Team member commissions exceed team budget"
    
    **Cause:** Sum of team-member rows exceeds team's `originalTeamCommission`.
    
    **Fix:** Reduce member allocations or increase team budget.
  </Accordion>
  
  <Accordion title="Team leftover mismatch">
    **Error:** "Team pool commission must equal budget minus member total"
    
    **Cause:** Team pool row `commissionPercentage` doesn't equal `originalTeamCommission - sum(members)`.
    
    **Fix:** Calculate correct leftover: `Math.max(0, budget - memberTotal)`.
  </Accordion>
  
  <Accordion title="Member access exceeds team">
    **Error:** "Team member access level cannot exceed team pool access"
    
    **Cause:** Team-member row has higher access than parent team pool.
    
    **Fix:** Cap member access at team pool level or increase team pool access.
  </Accordion>
  
  <Accordion title="Duplicate stakeholder identity">
    **Error:** "Duplicate stakeholder: same (userId, teamId) appears twice"
    
    **Cause:** Two rows with identical `userId` and `teamId` combination.
    
    **Fix:** Remove or merge duplicate rows.
  </Accordion>
  
  <Accordion title="Team member without pool">
    **Error:** "Team member row requires matching team pool row"
    
    **Cause:** `userId + teamId` row exists without a `teamId`-only pool row.
    
    **Fix:** Add team pool row with matching `teamId` and budget.
  </Accordion>
  
  <Accordion title="Unauthorized scope violation">
    **Error:** "User cannot manage stakeholders outside their teams"
    
    **Cause:** Team-scoped user tried to modify another team's rows.
    
    **Fix:** Only modify rows from teams the user belongs to, or request org-scoped user assistance.
  </Accordion>
  
  <Accordion title="Self-scope excess modification">
    **Error:** "Self-scoped user can only modify their own row"
    
    **Cause:** User with self scope tried to add, remove, or modify other stakeholders.
    
    **Fix:** Limit changes to user's personal stakeholder row only.
  </Accordion>
</AccordionGroup>
