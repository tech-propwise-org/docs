This is the complete practical map of CRM stakeholder rules. In user-facing copy, "stakeholder" is rendered as **assignee** / **assignment**, but the internal API, entity, DTO, and service names still use "stakeholder".

<Warning>
The backend is the authority. Frontend checks are UX mirrors only. If frontend and backend disagree, the backend wins.
</Warning>

## Entity Applicability

| Entity  | Commission / primary | Mutation API                       | Sections in this doc                                     |
| ------- | -------------------- | ---------------------------------- | -------------------------------------------------------- |
| Lead    | Yes                  | `PUT /leads/:id/stakeholders` sync | Sections 3–20 (primary focus)                            |
| Deal    | Yes (100% floor)     | `PUT /deals/:id/stakeholders` sync | Same structural/scope rules as lead unless noted in §3   |
| Contact | **No** (access-only) | Granular add / update / remove     | **§21** and §3 contact exceptions — not lead self-assign |

Deal stakeholder rules mirror lead rules for commission, primary, and sync authorization unless `STAKEHOLDER_SYSTEM.md` states a deal-only invariant (for example deals cannot have zero stakeholders).

## Vocabulary

<AccordionGroup>
<Accordion title="Row Types">

| Term                      | Meaning                                                                                                                                           |
| ------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------- |
| Personal row              | `userId` set, `teamId` empty. Grants one user access directly.                                                                                    |
| Team pool / team-only row | `teamId` set, `userId` empty. Represents team-level access and, for leads/deals, the team commission budget / leftover row.                       |
| Team-member row           | Both `userId` and `teamId` set. Represents a user assigned under a specific team context.                                                         |

</Accordion>
<Accordion title="Access & Permissions">

| Term                      | Meaning                                                                                                                                           |
| ------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------- |
| Primary row               | Exactly one active lead/deal stakeholder row marked `isPrimary=true` whenever stakeholders exist. Contacts always use `isPrimary=false`.          |
| Access level              | `READ`, `WRITE`, or `FULL`. Determines what a stakeholder can do to the entity.                                                                   |
| Access-only (contact)     | Contact stakeholders grant permanent contact access only. No commission percentages and no primary assignee in UI or API.                         |
| Management scope          | Backend-computed stakeholder editing scope: `org`, `team`, `self`, or `none`.                                                                     |

</Accordion>
<Accordion title="Commission">

| Term                      | Meaning                                                                                                                                           |
| ------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------- |
| Agency cut                | `orgCommissionPercentage`. Only users with `finance.manage` can set it. Frontend displays the remaining agent pool as the allocatable percentage. |

</Accordion>

## Structural Validation Rules

These rules are mirrored in frontend `stakeholder-validation.ts` and backend `stakeholder-validation.utils.ts`.

<Warning>
All structural validation rules are enforced on both frontend and backend. The frontend provides immediate feedback, while the backend enforces the authoritative validation.
</Warning>

### Core Structural Rules

| Rule                                            | Details                                                                                                                |
| ----------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------- |
| At least one entry                              | A stakeholder payload must contain entries unless the lead is intentionally moved to pool by an allowed sync path.     |
| Every entry identifies someone                  | Each row must have `userId`, `teamId`, or both.                                                                        |
| Exactly one primary                             | Whenever stakeholders exist, exactly one row must be primary.                                                          |
| Primary cannot be READ                          | A primary row must be `WRITE` or `FULL`. The frontend auto-promotes READ to WRITE when a row becomes primary.          |
| No duplicate row identity                       | The same `(userId, teamId)` combination cannot appear twice.                                                           |

### Commission-Related Rules

| Rule                                            | Details                                                                                                                |
| ----------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------- |
| `originalTeamCommission` only on team pool rows | Only rows with `teamId` and no `userId` may carry `originalTeamCommission`.                                            |
| Team pool row requires budget                   | A team-only row must have `originalTeamCommission`.                                                                    |
| Team member requires pool                       | A `userId + teamId` row requires a matching `teamId`-only row.                                                         |
| Member access cannot exceed team access         | A team-member row cannot have higher access than its parent team pool row.                                             |
| No negative commissions                         | `commissionPercentage` must be `>= 0`. DTOs also cap it at `<= 100`.                                                   |
| Total commission must match target              | Leads allow `0%` or the configured target. Deals require `100%`.                                                       |
| Top-level allocation must match target          | Standalone users plus team budgets must equal the target, or `0%` when the lead-specific zero-total exception applies. |
| Team members cannot exceed budget               | Sum of member rows under a team must be less than or equal to the team budget.                                         |
| Team leftover must be exact                     | Team pool `commissionPercentage` must equal `originalTeamCommission - sum(member commissions)`, clamped at `0`.        |

### Entity-Specific Exceptions

<Tabs>
<Tab title="Leads">
**Lead-specific exceptions:**
- Leads allow a `0%` commission total. This supports early assignment before a deal exists.
</Tab>
<Tab title="Deals">
**Deal-specific rules:**
- Deals require exactly `100%` commission allocation
- Cannot have zero stakeholders
</Tab>
<Tab title="Contacts">
**Contact-specific rules:**
- No commission percentages
- No primary assignee concept
- Access-only stakeholder relationships
</Tab>
</Tabs>

## Frontend Validation Behavior

<Steps>
<Step title="Real-time validation">
The frontend performs validation as users interact with the stakeholder assignment section, providing immediate feedback for structural rule violations.
</Step>

<Step title="Primary row promotion">
When a user marks a READ-access row as primary, the frontend automatically promotes it to WRITE access level.
</Step>

<Step title="Commission calculation">
The frontend calculates team leftovers and validates commission totals in real-time as users modify percentages.
</Step>

<Step title="Pre-submission validation">
Before submitting to the backend, all structural rules are validated and any violations prevent submission.
</Step>
</Steps>

<Note>
Frontend validation is purely for user experience. The backend performs the authoritative validation and will reject any requests that violate the structural rules.
</Note>

## Backend Authorization Flow

The backend enforces stakeholder rules through multiple layers:

### Create-Time Authorization
- **File:** `stakeholder-create-authorization.util.ts`
- **Scope:** Validates initial stakeholder assignments during lead/deal creation
- **Rules:** Ensures user can assign themselves and others based on their permissions

### Sync-Time Authorization  
- **File:** `stakeholder-sync-authorization.util.ts`
- **Scope:** Validates stakeholder updates for existing entities
- **Rules:** Enforces team-scoped and self-scoped editing restrictions

### Entity-Specific Guards
- **Lead Service:** Handles lead-specific stakeholder logic
- **Contact Service:** Manages access-only contact assignee relationships
- **Access Service:** Provides resource-level permission checks

<Check>
All backend authorization is fail-closed - if permissions cannot be determined or are insufficient, the operation is denied.
</Check>

## Management Scope Computation

The backend computes a management scope that determines what stakeholder operations a user can perform:

| Scope | Can Modify |
|-------|------------|
| `org` | All stakeholders in the organization |
| `team` | Stakeholders from their teams |
| `self` | Only their own stakeholder entries |
| `none` | Cannot modify stakeholders |

<Tip>
The frontend receives this scope in the entity permissions and adjusts the UI accordingly - hiding add buttons, disabling edit controls, etc. when the user lacks sufficient scope.
</Tip>

## Contact Stakeholder Differences

Contact stakeholders operate differently from lead/deal stakeholders:

<Warning>
Contact stakeholders are **access-only** - they provide permanent access to contact records but have no commission or primary assignee concepts.
</Warning>

### Key Differences
- No commission percentages
- No primary stakeholder designation  
- Uses granular add/update/remove APIs instead of sync
- Self-assignment restrictions still apply
- Team-scoped management rules still apply

### API Endpoints
```typescript
// Contact-specific stakeholder endpoints
POST /contacts/:id/stakeholders     // Add stakeholder
PUT /contacts/:id/stakeholders/:stakeholderId  // Update stakeholder  
DELETE /contacts/:id/stakeholders/:stakeholderId  // Remove stakeholder
```

<Info>
Unlike leads and deals which use a sync endpoint that replaces all stakeholders, contacts use granular endpoints for individual stakeholder operations.
</Info>
