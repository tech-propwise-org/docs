This document provides the complete practical map of CRM stakeholder rules. In user-facing copy, "stakeholder" is rendered as **assignee** / **assignment**, but the internal API, entity, DTO, and service names still use "stakeholder".

<Warning>
The backend is the authority. Frontend checks are UX mirrors only. If frontend and backend disagree, the backend wins.
</Warning>

## Entity Applicability

| Entity  | Commission / primary | Mutation API                       | Sections in this doc                                     |
| ------- | -------------------- | ---------------------------------- | -------------------------------------------------------- |
| Lead    | Yes                  | `PUT /leads/:id/stakeholders` sync | Sections 3–20 (primary focus)                            |
| Deal    | Yes (100% floor)     | `PUT /deals/:id/stakeholders` sync | Same structural/scope rules as lead unless noted in §3   |
| Contact | **No** (access-only) | Granular add / update / remove     | **§21** and §3 contact exceptions — not lead self-assign |

<Note>
Deal stakeholder rules mirror lead rules for commission, primary, and sync authorization unless `STAKEHOLDER_SYSTEM.md` states a deal-only invariant (for example deals cannot have zero stakeholders).
</Note>

## Vocabulary

| Term                      | Meaning                                                                                                                                           |
| ------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------- |
| Personal row              | `userId` set, `teamId` empty. Grants one user access directly.                                                                                    |
| Team pool / team-only row | `teamId` set, `userId` empty. Represents team-level access and, for leads/deals, the team commission budget / leftover row.                       |
| Team-member row           | Both `userId` and `teamId` set. Represents a user assigned under a specific team context.                                                         |
| Primary row               | Exactly one active lead/deal stakeholder row marked `isPrimary=true` whenever stakeholders exist. Contacts always use `isPrimary=false`.          |
| Access level              | `READ`, `WRITE`, or `FULL`. Determines what a stakeholder can do to the entity.                                                                   |
| Access-only (contact)     | Contact stakeholders grant permanent contact access only. No commission percentages and no primary assignee in UI or API.                         |
| Agency cut                | `orgCommissionPercentage`. Only users with `finance.manage` can set it. Frontend displays the remaining agent pool as the allocatable percentage. |
| Management scope          | Backend-computed stakeholder editing scope: `org`, `team`, `self`, or `none`.                                                                     |

## Structural Validation Rules

<Note>
These rules are mirrored in frontend `stakeholder-validation.ts` and backend `stakeholder-validation.utils.ts`.
</Note>

<AccordionGroup>
<Accordion title="Basic Structure Rules">

| Rule                                            | Details                                                                                                                |
| ----------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------- |
| At least one entry                              | A stakeholder payload must contain entries unless the lead is intentionally moved to pool by an allowed sync path.     |
| Every entry identifies someone                  | Each row must have `userId`, `teamId`, or both.                                                                        |
| Exactly one primary                             | Whenever stakeholders exist, exactly one row must be primary.                                                          |
| Primary cannot be READ                          | A primary row must be `WRITE` or `FULL`. The frontend auto-promotes READ to WRITE when a row becomes primary.          |
| No duplicate row identity                       | The same `(userId, teamId)` combination cannot appear twice.                                                           |

</Accordion>

<Accordion title="Commission and Team Rules">

| Rule                                            | Details                                                                                                                |
| ----------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------- |
| `originalTeamCommission` only on team pool rows | Only rows with `teamId` and no `userId` may carry `originalTeamCommission`.                                            |
| Team pool row requires budget                   | A team-only row must have `originalTeamCommission`.                                                                    |
| Team member requires pool                       | A `userId + teamId` row requires a matching `teamId`-only row.                                                         |
| Member access cannot exceed team access         | A team-member row cannot have higher access than its parent team pool row.                                             |
| No negative commissions                         | `commissionPercentage` must be `>= 0`. DTOs also cap it at `<= 100`.                                                   |
| Total commission must match target              | Leads allow `0%` or the configured target. Deals require `100%`.                                                       |
| Top-level allocation must match target          | Standalone users plus team budgets must equal the target, or `0%` when the lead-specific zero-total exception applies. |
| Team members cannot exceed budget               | Sum of member rows under a team must be less than or equal to the team budget.                                         |
| Team leftover must be exact                     | Team pool `commissionPercentage` must equal `originalTeamCommission - sum(member commissions)`, clamped at `0`.        |

</Accordion>
</AccordionGroup>

### Lead-specific Exceptions

<Check>
Leads differ from deals in that leads allow a `0%` commission total. This supports early assignment before a deal exists.
</Check>

### Contact-specific Exceptions

<Warning>
Contacts are access-only entities:
- No commission percentages
- No primary assignee concept
- All rows use `isPrimary=false`
- Different API endpoints for stakeholder management
</Warning>

## Authorization Scopes

The backend computes management scope for stakeholder operations based on user permissions and current stakeholder assignments.

<CardGroup cols={2}>
<Card title="Org Scope" icon="building">
Full organization-wide stakeholder management. Can assign anyone to any team or personal capacity.
</Card>
<Card title="Team Scope" icon="users">
Can manage stakeholders within specific teams. Limited to team members and team-level assignments.
</Card>
<Card title="Self Scope" icon="user">
Can only manage their own stakeholder assignments. Most restrictive scope.
</Card>
<Card title="None Scope" icon="ban">
No stakeholder management permissions. Read-only access to stakeholder information.
</Card>
</CardGroup>

## Access Level Hierarchy

Access levels determine what actions a stakeholder can perform on an entity:

<Steps>
<Step title="READ">
View-only access to the entity. Cannot make modifications.
</Step>
<Step title="WRITE">
Can modify the entity but cannot delete or transfer ownership.
</Step>
<Step title="FULL">
Complete control including delete and transfer capabilities.
</Step>
</Steps>

<Note>
Team-member rows cannot have higher access than their parent team pool row. The frontend automatically caps member access appropriately.
</Note>

## Primary Assignee Rules

<Warning>
The primary assignee concept applies only to leads and deals, not contacts.
</Warning>

- Exactly one stakeholder must be marked as primary when stakeholders exist
- Primary stakeholders cannot have READ access level (auto-promoted to WRITE)
- Primary assignee represents the main point of responsibility for the entity
- Frontend automatically handles primary promotion when access levels change

## Commission Management

<Tip>
Commission percentages are only relevant for leads and deals. Contacts use access-only stakeholder assignments.
</Tip>

### Agency Cut
- Only users with `finance.manage` permission can set `orgCommissionPercentage`
- Frontend displays remaining percentage as allocatable to agents
- Agency cut is deducted before agent allocation

### Commission Validation
- Total allocations must equal target percentage (varies by entity type)
- Team member allocations cannot exceed team budget
- Negative commission percentages are not allowed
- Team leftover calculations must be exact

## Frontend Integration

The frontend stakeholder assignment section provides a unified interface for managing stakeholders across all entity types:

```typescript
<StakeholderAssignmentSection
  entityType="lead" // or "deal" or "contact"
  allowMemberSelfAssign={true}
  isTeamScopedManage={scope === 'team'}
  isSelfScopedManage={scope === 'self'}
  isAccessOnly={entityType === 'contact'}
/>
```

<Note>
The `isAccessOnly` prop gates contact-specific behavior and disables commission-related UI elements.
</Note>

## Error Handling

Common validation errors and their meanings:

<AccordionGroup>
<Accordion title="Structural Validation Errors">
- `STAKEHOLDERS_REQUIRED`: Empty stakeholder payload when entries are required
- `DUPLICATE_STAKEHOLDER_IDENTITY`: Same (userId, teamId) combination appears multiple times
- `PRIMARY_REQUIRED`: No primary stakeholder when stakeholders exist
- `PRIMARY_ACCESS_INVALID`: Primary stakeholder has READ access level
</Accordion>

<Accordion title="Commission Validation Errors">
- `COMMISSION_TOTAL_MISMATCH`: Total allocations don't match target percentage
- `TEAM_BUDGET_EXCEEDED`: Team member allocations exceed team budget
- `NEGATIVE_COMMISSION`: Commission percentage is below zero
- `TEAM_LEFTOVER_MISMATCH`: Team pool percentage doesn't match calculated leftover
</Accordion>

<Accordion title="Authorization Errors">
- `INSUFFICIENT_STAKEHOLDER_SCOPE`: User lacks permission for requested stakeholder operation
- `TEAM_MEMBER_ACCESS_EXCEEDS_TEAM`: Team member assigned higher access than team pool
- `SELF_ASSIGNMENT_NOT_ALLOWED`: User attempted self-assignment without permission
</Accordion>
</AccordionGroup>

This rulebook serves as the definitive reference for implementing and troubleshooting stakeholder functionality across the CRM system. Always defer to backend validation as the source of truth for enforcement.
