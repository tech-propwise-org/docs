## Vocabulary

<AccordionGroup>
  <Accordion title="Personal row">
    `userId` set, `teamId` empty. Grants one user access directly.
  </Accordion>
  
  <Accordion title="Team pool / team-only row">
    `teamId` set, `userId` empty. Represents team-level access and, for leads/deals, the team commission budget / leftover row.
  </Accordion>
  
  <Accordion title="Team-member row">
    Both `userId` and `teamId` set. Represents a user assigned under a specific team context.
  </Accordion>
  
  <Accordion title="Primary row">
    Exactly one active lead/deal stakeholder row marked `isPrimary=true` whenever stakeholders exist. Contacts always use `isPrimary=false`.
  </Accordion>
  
  <Accordion title="Access level">
    `READ`, `WRITE`, or `FULL`. Determines what a stakeholder can do to the entity.
  </Accordion>
  
  <Accordion title="Access-only (contact)">
    Contact stakeholders grant permanent contact access only. No commission percentages and no primary assignee in UI or API.
  </Accordion>
  
  <Accordion title="Agency cut">
    `orgCommissionPercentage`. Only users with `finance.manage` can set it. Frontend displays the remaining agent pool as the allocatable percentage.
  </Accordion>
  
  <Accordion title="Management scope">
    Backend-computed stakeholder editing scope: `org`, `team`, `self`, or `none`.
  </Accordion>
</AccordionGroup>

## Structural Validation Rules

<Note>
These rules are mirrored in frontend `stakeholder-validation.ts` and backend `stakeholder-validation.utils.ts`.
</Note>

### Core Rules

<Steps>
  <Step title="At least one entry">
    A stakeholder payload must contain entries unless the lead is intentionally moved to pool by an allowed sync path.
  </Step>
  
  <Step title="Every entry identifies someone">
    Each row must have `userId`, `teamId`, or both.
  </Step>
  
  <Step title="Exactly one primary">
    Whenever stakeholders exist, exactly one row must be primary.
  </Step>
  
  <Step title="Primary cannot be READ">
    A primary row must be `WRITE` or `FULL`. The frontend auto-promotes READ to WRITE when a row becomes primary.
  </Step>
</Steps>

### Identity and Duplication

| Rule | Details |
|------|---------|
| No duplicate row identity | The same `(userId, teamId)` combination cannot appear twice. |
| `originalTeamCommission` only on team pool rows | Only rows with `teamId` and no `userId` may carry `originalTeamCommission`. |
| Team pool row requires budget | A team-only row must have `originalTeamCommission`. |
| Team member requires pool | A `userId + teamId` row requires a matching `teamId`-only row. |

### Access and Commission Rules

<CardGroup cols={2}>
  <Card title="Member Access Capping" icon="shield-halved">
    A team-member row cannot have higher access than its parent team pool row.
  </Card>
  
  <Card title="No Negative Commissions" icon="percent">
    `commissionPercentage` must be `>= 0`. DTOs also cap it at `<= 100`.
  </Card>
  
  <Card title="Total Commission Match" icon="calculator">
    Leads allow `0%` or the configured target. Deals require `100%`.
  </Card>
  
  <Card title="Top-level Allocation Match" icon="chart-pie">
    Standalone users plus team budgets must equal the target, or `0%` when the lead-specific zero-total exception applies.
  </Card>
</CardGroup>

### Team Budget Rules

<Warning>
**Team members cannot exceed budget**

Sum of member rows under a team must be less than or equal to the team budget.
</Warning>

<Check>
**Team leftover must be exact**

Team pool `commissionPercentage` must equal `originalTeamCommission - sum(member commissions)`, clamped at `0`.
</Check>

### Lead-Specific Exceptions

<Tip>
Leads differ from deals:

- Leads allow a `0%` commission total. This supports early assignment before a deal exists.
- Deals always require `100%` commission allocation.
</Tip>

## Access Levels

### Level Hierarchy

```typescript
enum AccessLevel {
  READ = "READ",
  WRITE = "WRITE", 
  FULL = "FULL"
}
```

<Tabs>
  <Tab title="READ">
    - View entity details
    - Cannot modify entity
    - Cannot manage stakeholders
    - **Cannot be primary** (auto-promoted to WRITE if made primary)
  </Tab>
  
  <Tab title="WRITE">
    - All READ permissions
    - Modify entity fields
    - Cannot delete entity
    - Cannot manage stakeholders
    - **Can be primary**
  </Tab>
  
  <Tab title="FULL">
    - All WRITE permissions
    - Delete entity
    - Manage stakeholders (subject to management scope)
    - Transfer ownership
    - **Can be primary**
  </Tab>
</Tabs>

### Team-Member Access Capping

<CodeGroup>
```typescript Team Access Validation
// Team-member row access cannot exceed team pool access
function validateTeamMemberAccess(
  memberRow: StakeholderRow,
  teamPoolRow: StakeholderRow
): boolean {
  const accessRank = { READ: 0, WRITE: 1, FULL: 2 };
  
  return accessRank[memberRow.accessLevel] <= 
         accessRank[teamPoolRow.accessLevel];
}
```

```typescript Frontend Helper
// src/lib/stakeholder-access-level.ts
export function getMaxAllowedAccessForMember(
  teamAccess: AccessLevel
): AccessLevel {
  // Member cannot exceed team access
  return teamAccess;
}
```
</CodeGroup>

## Primary Row Rules

### Eligibility

<Steps>
  <Step title="Access level requirement">
    Primary row must be `WRITE` or `FULL`. `READ` rows are automatically promoted to `WRITE` when marked primary.
  </Step>
  
  <Step title="Uniqueness requirement">
    Exactly one row must be `isPrimary=true` when stakeholders exist.
  </Step>
  
  <Step title="Row type flexibility">
    Primary can be personal, team-member, or team-pool row (for leads/deals only).
  </Step>
</Steps>

<Warning>
**Contacts do not support primary rows.** All contact stakeholder rows must have `isPrimary=false`.
</Warning>

### Automatic Promotion

```typescript
// Frontend auto-promotion logic
function ensurePrimaryIsWritable(row: StakeholderRow): StakeholderRow {
  if (row.isPrimary && row.accessLevel === "READ") {
    return {
      ...row,
      accessLevel: "WRITE"
    };
  }
  return row;
}
```

## Commission Rules

### Agency Cut (Organization Commission)

<Info>
**Permission required:** `finance.manage`

Only users with finance management permissions can set `orgCommissionPercentage`.
</Info>

#### Frontend Display

```typescript
// Visible agent pool percentage
const agentPoolPercentage = 100 - orgCommissionPercentage;

// Example: 10% agency cut → 90% agent pool
```

<CodeGroup>
```typescript Validation
function validateAgencyCut(orgCommission: number): boolean {
  return orgCommission >= 0 && orgCommission <= 100;
}
```

```typescript Normalization
// src/lib/stakeholder-commission.ts
export function normalizeCommissionForBackend(
  visiblePercentage: number,
  agencyCut: number
): number {
  // Convert UI percentage to backend absolute
  return (visiblePercentage * (100 - agencyCut)) / 100;
}
```
</CodeGroup>

### Commission Allocation Rules

<Tabs>
  <Tab title="Leads">
    **Target:** Configurable percentage OR `0%`
    
    - Supports early assignment before deal conversion
    - Can have zero total commission
    - Top-level allocation must match target when non-zero
  </Tab>
  
  <Tab title="Deals">
    **Target:** Always `100%`
    
    - Cannot have zero stakeholders
    - Cannot have zero commission total
    - Top-level allocation must always equal 100%
  </Tab>
  
  <Tab title="Contacts">
    **Target:** N/A (access-only)
    
    - No commission fields
    - No `originalTeamCommission`
    - Access levels only
  </Tab>
</Tabs>

### Team Budget Mechanics

<Steps>
  <Step title="Team pool budget">
    Each team-only row has `originalTeamCommission` representing the team's total budget.
  </Step>
  
  <Step title="Member allocations">
    Team-member rows allocate portions of the team budget via `commissionPercentage`.
  </Step>
  
  <Step title="Leftover calculation">
    Team pool `commissionPercentage` = `originalTeamCommission - sum(member commissions)`, clamped at `0`.
  </Step>
  
  <Step title="Enforcement">
    Sum of member commissions cannot exceed `originalTeamCommission`.
  </Step>
</Steps>

#### Example

<CodeGroup>
```json Team Pool Row
{
  "teamId": "team-123",
  "userId": null,
  "accessLevel": "FULL",
  "isPrimary": false,
  "originalTeamCommission": 50.0,
  "commissionPercentage": 20.0  // 50 - 30 (members) = 20 leftover
}
```

```json Team Member Rows
[
  {
    "teamId": "team-123",
    "userId": "user-456",
    "accessLevel": "WRITE",
    "isPrimary": true,
    "commissionPercentage": 20.0
  },
  {
    "teamId": "team-123", 
    "userId": "user-789",
    "accessLevel": "READ",
    "isPrimary": false,
    "commissionPercentage": 10.0
  }
]
// Total member allocation: 30.0
// Remaining in pool: 20.0
```
</CodeGroup>

## Management Scope

<Note>
Backend computes stakeholder editing scope based on the actor's permissions and relationship to the entity.
</Note>

### Scope Levels

<CardGroup cols={2}>
  <Card title="org" icon="building">
    Full organization-wide stakeholder management
    
    - Can add/remove any user
    - Can modify all commission splits
    - Can change primary assignee
  </Card>
  
  <Card title="team" icon="users">
    Team-scoped stakeholder management
    
    - Can manage team pool and team members
    - Cannot modify personal rows of users outside managed teams
    - Can change primary if within team scope
  </Card>
  
  <Card title="self" icon="user">
    Self-only stakeholder management
    
    - Can only modify own personal row
    - Cannot add/remove other stakeholders
    - Cannot make self primary unless already primary
  </Card>
  
  <Card title="none" icon="ban">
    No stakeholder management permissions
    
    - Cannot modify any stakeholder rows
    - Read-only access to stakeholder list
  </Card>
</CardGroup>

### Backend Scope Resolution

```typescript
// src/modules/crm/lead/lead.service.ts
function resolveManagementScope(
  actor: User,
  lead: Lead
): ManagementScope {
  // Org-level permission check
  if (hasPermission(actor, "crm.leads.manage.org")) {
    return "org";
  }
  
  // Team-level permission check
  const managedTeams = getTeamsActorCanManage(actor);
  if (managedTeams.length > 0 && hasTeamStakeholder(lead, managedTeams)) {
    return { scope: "team", teamIds: managedTeams };
  }
  
  // Self-only check
  if (hasPersonalStakeholder(lead, actor.id)) {
    return "self";
  }
  
  return "none";
}
```

## Contact-Specific Rules

<Warning>
Contacts use **access-only** stakeholder mode. Many lead/deal rules do not apply.
</Warning>

### Key Differences

| Feature | Leads/Deals | Contacts |
|---------|-------------|----------|
| Primary assignee | Required (one) | **Not supported** |
| Commission fields | Yes | **No** |
| `originalTeamCommission` | Team pools only | **Never** |
| Self-assignment on create | Allowed with permission | **Rejected** |
| Mutation API | Sync (`PUT`) | Granular (add/update/remove) |

### Self-Row Rejection

```typescript
// Backend: src/modules/crm/contact/contact.service.ts
function validateContactStakeholders(
  stakeholders: StakeholderDto[],
  actorId: string
): void {
  stakeholders.forEach(row => {
    // Reject self-assignment
    if (row.userId === actorId && !row.teamId) {
      throw new ForbiddenException(
        "Cannot self-assign to contacts"
      );
    }
    
    // Reject primary flag
    if (row.isPrimary) {
      throw new BadRequestException(
        "Contacts do not support primary assignees"
      );
    }
  });
}
```

## Create-Time Authorization

### Lead Creation

<Steps>
  <Step title="Self-assignment permission check">
    If actor assigns themselves, verify `crm.leads.create.assign_self` permission.
  </Step>
  
  <Step title="Other user assignment check">
    If actor assigns other users, verify `crm.leads.create.assign_others` permission.
  </Step>
  
  <Step title="Team assignment check">
    If actor assigns teams, verify they can manage those teams or have org-level permission.
  </Step>
  
  <Step title="Structural validation">
    Run all structural validation rules from Section 3.
  </Step>
</Steps>

```typescript
// Backend: stakeholder-create-authorization.util.ts
export function authorizeLeadCreate(
  dto: CreateLeadDto,
  actor: User
): void {
  const { stakeholders } = dto;
  
  const hasSelfRow = stakeholders.some(
    s => s.userId === actor.id && !s.teamId
  );
  const hasOtherRows = stakeholders.some(
    s => s.userId !== actor.id || s.teamId
  );
  
  if (hasSelfRow && !hasPermission(actor, "crm.leads.create.assign_self")) {
    throw new ForbiddenException("Cannot self-assign on lead creation");
  }
  
  if (hasOtherRows && !hasPermission(actor, "crm.leads.create.assign_others")) {
    throw new ForbiddenException("Cannot assign others on lead creation");
  }
  
  validateStakeholders(stakeholders, "lead");
}
```

### Contact Creation

<Warning>
**Contacts reject self-assignment entirely.** Even if actor has `assign_self` permission, personal self-rows are forbidden.
</Warning>

```typescript
// Backend: contact.service.ts
export function authorizeContactCreate(
  dto: CreateContactDto,
  actor: User
): void {
  const { stakeholders = [] } = dto;
  
  // Reject any self-assignment
  const hasSelfRow = stakeholders.some(
    s => s.userId === actor.id && !s.teamId
  );
  
  if (hasSelfRow) {
    throw new ForbiddenException(
      "Self-assignment not allowed for contacts"
    );
  }
  
  // Validate access-only mode
  stakeholders.forEach(s => {
    if (s.isPrimary || s.commissionPercentage != null) {
      throw new BadRequestException(
        "Contacts are access-only entities"
      );
    }
  });
}
```

## Sync-Time Authorization

<Note>
Sync authorization applies to `PUT /leads/:id/stakeholders` and `PUT /deals/:id/stakeholders`. Contacts use granular endpoints instead.
</Note>

### Scope-Based Authorization

<Tabs>
  <Tab title="org scope">
    **Full access** to all stakeholder modifications
    
    - Add/remove any user or team
    - Modify all commission splits
    - Change primary assignee
    - Remove all stakeholders (leads only)
  </Tab>
  
  <Tab title="team scope">
    **Team-scoped access** to specific teams
    
    - Can modify team pool rows for managed teams
    - Can add/remove team members under managed teams
    - Cannot modify personal rows of non-team users
    - Can change primary if new primary is within team scope
  </Tab>
  
  <Tab title="self scope">
    **Self-only access**
    
    - Can only modify own personal row access level
    - Cannot modify commission percentage
    - Cannot add/remove self
    - Cannot change primary (unless already primary)
  </Tab>
</Tabs>

### Row-Level Authorization

```typescript
// Backend: stakeholder-sync-authorization.util.ts
export function authorizeRowModification(
  existingRow: Stakeholder | null,
  newRow: StakeholderDto | null,
  scope: ManagementScope,
  actorId: string
): void {
  // org scope: always allowed
  if (scope === "org") return;
  
  // team scope: check team membership
  if (scope.scope === "team") {
    const isTeamRow = 
      existingRow?.teamId && scope.teamIds.includes(existingRow.teamId) ||
      newRow?.teamId && scope.teamIds.includes(newRow.teamId);
    
    if (!isTeamRow && existingRow?.userId !== actorId) {
      throw new ForbiddenException("Cannot modify row outside team scope");
    }
  }
  
  // self scope: only own personal row
  if (scope === "self") {
    if (existingRow?.userId !== actorId || existingRow?.teamId) {
      throw new ForbiddenException("Can only modify own personal row");
    }
    
    // Cannot modify commission in self scope
    if (newRow && existingRow && 
        newRow.commissionPercentage !== existingRow.commissionPercentage) {
      throw new ForbiddenException("Cannot modify own commission");
    }
  }
}
```

<Steps>
  <Step title="Frontend structural validation">
    `stakeholder-validation.ts` runs all rules from Section 3 before submission.
  </Step>
  
  <Step title="Frontend authorization checks">
    Component verifies user has required permissions for the action (create self-assign, create assign-others, manage scope).

  </Step>

  </Step>
  
  <Step title="Database constraints">
    Entity stakeholder table enforces uniqueness and foreign key constraints.
  </Step>
</Steps>
