<Note>
**Status:** Reference document  
**Scope:** Lead, deal, and contact stakeholder/assignee validation, permissions, frontend behavior, and backend authorization
</Note>

This document is the complete practical map of CRM stakeholder rules. In user-facing copy, "stakeholder" is rendered as **assignee** / **assignment**, but the internal API, entity, DTO, and service names still use "stakeholder".

<Warning>
The backend is the authority. Frontend checks are UX mirrors only. If frontend and backend disagree, the backend wins.
</Warning>

## Entity applicability

| Entity  | Commission / primary | Mutation API                       | Coverage in this doc                                     |
| ------- | -------------------- | ---------------------------------- | -------------------------------------------------------- |
| Lead    | Yes                  | `PUT /leads/:id/stakeholders` sync | Sections 3–20 (primary focus)                            |
| Deal    | Yes (100% floor)     | `PUT /deals/:id/stakeholders` sync | Same structural/scope rules as lead unless noted in §3   |
| Contact | **No** (access-only) | Granular add / update / remove     | **§21** and §3 contact exceptions — not lead self-assign |

<Info>
Deal stakeholder rules mirror lead rules for commission, primary, and sync authorization unless the engineering spec states a deal-only invariant (for example, deals cannot have zero stakeholders).
</Info>

## Vocabulary

| Term                      | Meaning                                                                                                                                           |
| ------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------- |
| Personal row              | `userId` set, `teamId` empty. Grants one user access directly.                                                                                    |
| Team pool / team-only row | `teamId` set, `userId` empty. Represents team-level access and, for leads/deals, the team commission budget / leftover row.                       |
| Team-member row           | Both `userId` and `teamId` set. Represents a user assigned under a specific team context.                                                         |
| Primary row               | Exactly one active lead/deal stakeholder row marked `isPrimary=true` whenever stakeholders exist. Contacts always use `isPrimary=false`.          |
| Access level              | `READ`, `WRITE`, or `FULL`. Determines what a stakeholder can do to the entity.                                                                   |
| Access-only (contact)     | Contact stakeholders grant permanent contact access only. No commission percentages and no primary assignee in UI or API.                         |
| Agency cut                | `orgCommissionPercentage`. Only users with `finance.manage` can set it. Frontend displays the remaining agent pool as the allocatable percentage. |
| Management scope          | Backend-computed stakeholder editing scope: `org`, `team`, `self`, or `none`.                                                                     |

## Structural validation rules

These rules are mirrored in frontend `stakeholder-validation.ts` and backend `stakeholder-validation.utils.ts`.

<Note>
All structural validation rules apply to both frontend and backend validation layers.
</Note>

### Core validation rules

| Rule                                            | Details                                                                                                                |
| ----------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------- |
| At least one entry                              | A stakeholder payload must contain entries unless the lead is intentionally moved to pool by an allowed sync path.     |
| Every entry identifies someone                  | Each row must have `userId`, `teamId`, or both.                                                                        |
| Exactly one primary                             | Whenever stakeholders exist, exactly one row must be primary.                                                          |
| Primary cannot be READ                          | A primary row must be `WRITE` or `FULL`. The frontend auto-promotes READ to WRITE when a row becomes primary.          |
| No duplicate row identity                       | The same `(userId, teamId)` combination cannot appear twice.                                                           |
| `originalTeamCommission` only on team pool rows | Only rows with `teamId` and no `userId` may carry `originalTeamCommission`.                                            |
| Team pool row requires budget                   | A team-only row must have `originalTeamCommission`.                                                                    |
| Team member requires pool                       | A `userId + teamId` row requires a matching `teamId`-only row.                                                         |
| Member access cannot exceed team access         | A team-member row cannot have higher access than its parent team pool row.                                             |
| No negative commissions                         | `commissionPercentage` must be `>= 0`. DTOs also cap it at `<= 100`.                                                   |
| Total commission must match target              | Leads allow `0%` or the configured target. Deals require `100%`.                                                       |
| Top-level allocation must match target          | Standalone users plus team budgets must equal the target, or `0%` when the lead-specific zero-total exception applies. |
| Team members cannot exceed budget               | Sum of member rows under a team must be less than or equal to the team budget.                                         |
| Team leftover must be exact                     | Team pool `commissionPercentage` must equal `originalTeamCommission - sum(member commissions)`, clamped at `0`.        |

### Lead-specific exceptions

<Info>
Leads differ from deals in their commission requirements:
- Leads allow a `0%` commission total. This supports early assignment before a deal exists.
- Deals always require `100%` commission allocation.
</Info>

### Contact-specific exceptions

<Warning>
Contact stakeholders are access-only:
- No commission percentages
- No primary assignee concept
- `isPrimary` is always `false`
- Different validation rules apply (see §21)
</Warning>

## Access levels

Access levels determine what a stakeholder can do with an entity.

| Level   | Permissions                                          | Notes                                                   |
| ------- | ---------------------------------------------------- | ------------------------------------------------------- |
| `READ`  | View entity details                                  | Cannot be used for primary row                          |
| `WRITE` | View + edit entity fields                            | Minimum level for primary row                           |
| `FULL`  | View + edit + manage stakeholders + delete/transfer  | Full control over entity                                |

<Note>
When a row becomes primary, frontend automatically promotes `READ` to `WRITE` to meet the "primary cannot be READ" rule.
</Note>

### Access level capping for team members

Team-member rows cannot have higher access than their parent team pool row:

```typescript
// Example: Team pool has WRITE access
const teamPoolAccess = "WRITE";
const memberAccess = "FULL"; // ❌ Invalid - exceeds team access

// Valid member access levels when team pool is WRITE:
// - READ ✓
// - WRITE ✓
// - FULL ✗
```

## Commission calculation rules

### Agency cut (`orgCommissionPercentage`)

<Warning>
Only users with `finance.manage` permission can set or modify the agency cut.
</Warning>

The agency cut (organization commission percentage) is deducted from the total commission pool before agent allocation:

```typescript
// Example: 5% agency cut on 100% total
const totalCommission = 100;
const agencyCut = 5;
const agentPool = totalCommission - agencyCut; // 95%

// Frontend displays 95% as the allocatable percentage
```

### Team commission budgets

Team pool rows represent the commission budget allocated to a team:

<Steps>
  <Step title="Set team budget">
    Create a team-only row with `originalTeamCommission` (e.g., 50%)
  </Step>
  
  <Step title="Allocate to members">
    Add team-member rows under the team, each with a commission percentage
  </Step>
  
  <Step title="Calculate leftover">
    Team pool's `commissionPercentage` = `originalTeamCommission - sum(member commissions)`
  </Step>
</Steps>

```typescript
// Example: Team with 50% budget
const originalTeamCommission = 50;

// Member allocations
const member1Commission = 20;
const member2Commission = 15;
const memberTotal = member1Commission + member2Commission; // 35%

// Team pool leftover
const teamLeftover = originalTeamCommission - memberTotal; // 15%
```

<Check>
The team leftover must always equal `originalTeamCommission - sum(member commissions)`, clamped at `0`.
</Check>

### Commission validation

<Tabs>
  <Tab title="Leads">
    Leads allow two valid commission totals:
    - `0%` (early assignment before deal conversion)
    - Configured target percentage (typically 100%)
    
    ```typescript
    // Valid lead commission totals
    const validTotals = [0, targetPercentage];
    ```
  </Tab>
  
  <Tab title="Deals">
    Deals always require exactly `100%` commission allocation:
    
    ```typescript
    const totalCommission = stakeholders.reduce(
      (sum, s) => sum + s.commissionPercentage, 0
    );
    
    if (totalCommission !== 100) {
      throw new Error("Deal commission must total 100%");
    }
    ```
  </Tab>
  
  <Tab title="Contacts">
    Contacts are access-only and have no commission:
    
    ```typescript
    // Contact stakeholders
    const contactStakeholder = {
      userId: "user123",
      accessLevel: "READ",
      // No commissionPercentage field
      // No isPrimary field
    };
    ```
  </Tab>
</Tabs>

## Primary assignee rules

<Warning>
The primary assignee concept only applies to leads and deals. Contacts do not have a primary assignee.
</Warning>

### Primary assignee requirements

<Steps>
  <Step title="Exactly one primary">
    When stakeholders exist, exactly one row must have `isPrimary=true`
  </Step>
  
  <Step title="Access level minimum">
    Primary row must have `WRITE` or `FULL` access (not `READ`)
  </Step>
  
  <Step title="Cannot be team-only">
    Primary row must have a `userId` (personal or team-member row)
  </Step>
</Steps>

### Primary eligibility

A stakeholder row is eligible to be primary if:

1. Has a `userId` (personal or team-member)
2. Has `WRITE` or `FULL` access
3. Is not a team-only pool row

```typescript
function isPrimaryEligible(row: StakeholderRow): boolean {
  return (
    row.userId !== null &&
    row.userId !== undefined &&
    (row.accessLevel === "WRITE" || row.accessLevel === "FULL")
  );
}
```

### READ to WRITE promotion

When a READ-level row becomes primary, frontend automatically promotes it to WRITE:

```typescript
function makePrimary(row: StakeholderRow): StakeholderRow {
  if (row.accessLevel === "READ") {
    return { ...row, accessLevel: "WRITE", isPrimary: true };
  }
  return { ...row, isPrimary: true };
}
```

<Tip>
This automatic promotion ensures the structural validation rule "primary cannot be READ" is always satisfied.
</Tip>

## Row identity and uniqueness

### Row identity tuple

Each stakeholder row is uniquely identified by the `(userId, teamId)` tuple:

| Row type     | userId | teamId | Identity                    |
| ------------ | ------ | ------ | --------------------------- |
| Personal     | ✓      | null   | `(userId, null)`            |
| Team pool    | null   | ✓      | `(null, teamId)`            |
| Team-member  | ✓      | ✓      | `(userId, teamId)`          |

<Warning>
The same `(userId, teamId)` combination cannot appear twice in a stakeholder list.
</Warning>

### Duplicate detection

```typescript
function hasDuplicates(rows: StakeholderRow[]): boolean {
  const identities = new Set<string>();
  
  for (const row of rows) {
    const identity = `${row.userId || "null"}:${row.teamId || "null"}`;
    if (identities.has(identity)) {
      return true;
    }
    identities.add(identity);
  }
  
  return false;
}
```

## Team pool and member relationships

### Team pool requirements

A team-only row (pool) must have:

1. `teamId` set
2. `userId` empty/null
3. `originalTeamCommission` set
4. Commission budget > 0

```typescript
interface TeamPoolRow {
  teamId: string;
  userId: null;
  originalTeamCommission: number; // Must be > 0
  commissionPercentage: number;   // Calculated leftover
  accessLevel: "READ" | "WRITE" | "FULL";
  isPrimary: false;
}
```

### Team member requirements

A team-member row must have:

1. Both `userId` and `teamId` set
2. A matching team pool row with the same `teamId`
3. Access level ≤ team pool access level
4. Commission ≤ available team budget

```typescript
interface TeamMemberRow {
  userId: string;
  teamId: string;
  // No originalTeamCommission
  commissionPercentage: number;
  accessLevel: "READ" | "WRITE" | "FULL"; // Must be ≤ team pool access
  isPrimary: boolean; // Can be true if eligible
}
```

### Validation example

```typescript
// ✓ Valid team structure
const teamPool = {
  teamId: "team1",
  userId: null,
  originalTeamCommission: 50,
  commissionPercentage: 10, // 50 - 40 (allocated to members)
  accessLevel: "FULL"
};

const teamMember1 = {
  userId: "user1",
  teamId: "team1",
  commissionPercentage: 25,
  accessLevel: "WRITE", // ≤ FULL (team pool)
  isPrimary: true
};

const teamMember2 = {
  userId: "user2",
  teamId: "team1",
  commissionPercentage: 15,
  accessLevel: "READ", // ≤ FULL (team pool)
  isPrimary: false
};

// Member total: 25 + 15 = 40
// Team leftover: 50 - 40 = 10 ✓
// Member access levels ≤ team pool ✓
```

## Create-time authorization

### Lead and deal creation

<Info>
Stakeholder authorization during entity creation is handled by `stakeholder-create-authorization.util.ts`.
</Info>

#### Authorization scopes

| Scope | Can assign                                    | Restrictions                                              |
| ----- | --------------------------------------------- | --------------------------------------------------------- |
| Org   | Any user/team in organization                 | None                                                      |
| Team  | Users/teams in actor's teams                  | Cannot create personal rows for users outside actor teams |
| Self  | Only actor themselves                         | Single personal row for actor only                        |
| None  | Cannot create with stakeholders               | Must create unassigned (pool)                             |

#### Permission-based scope resolution

```typescript
// Org scope: User has leads.create.org
if (actor.hasPermission("leads.create.org")) {
  return "org";
}

// Team scope: User has leads.create.team
if (actor.hasPermission("leads.create.team")) {
  return "team";
}

// Self scope: User has leads.create.self
if (actor.hasPermission("leads.create.self")) {
  return "self";
}

// None: User has leads.create but no stakeholder permissions
return "none";
```

#### Member self-assignment

<Note>
Members can assign themselves when creating a lead if they have `leads.create.self` permission or higher.
</Note>

```typescript
const allowMemberSelfAssign = 
  scope === "self" || 
  scope === "team" || 
  scope === "org";
```

Frontend creates a pre-filled row for the actor when `allowMemberSelfAssign` is true.

### Contact creation

<Warning>
Contact creation has different authorization rules than leads/deals.
</Warning>

Contacts use access-only stakeholders and follow these rules:

1. No commission fields
2. No primary designation
3. Member **cannot** self-assign (unlike leads)
4. Only team-scoped or org-scoped creation allowed

```typescript
// Contact creation flow
const contactScope = resolveContactCreateScope(actor);

// Member self-assignment disabled for contacts
const allowMemberSelfAssign = false;

// Validate stakeholders are team or org users
validateContactCreateStakeholders(stakeholders, contactScope, actor);
```

## Sync authorization (edit existing)

### Management scope resolution

When editing stakeholders on an existing lead/deal, the backend computes a management scope based on the actor's current relationship to the entity:

| Scope | Description                                                            | Actor can modify                                    |
| ----- | ---------------------------------------------------------------------- | --------------------------------------------------- |
| `org` | Actor has org-wide manage permission                                   | All rows                                            |
| `team`| Actor has team manage permission and is stakeholder via managed team   | Team pool and members for actor's managed teams     |
| `self`| Actor is a personal stakeholder with FULL access                       | Only actor's personal row                           |
| `none`| Actor cannot modify stakeholders                                       | No modifications allowed                            |

<Info>
Management scope is computed by `computeStakeholderManagementScope()` in `lead.service.ts` or `deal.service.ts`.
</Info>

### Scope-specific rules

<Tabs>
  <Tab title="Org scope">
    **Full authority**: Actor can modify all stakeholder rows.
    
    ```typescript
    // Org-scoped actor can:
    // - Add/remove/edit any user or team
    // - Change primary assignee
    // - Modify commission splits
    // - Change access levels
    // - Set agency cut (if has finance.manage)
    ```
    
    No additional row-level authorization checks.
  </Tab>
  
  <Tab title="Team scope">
    **Team-limited authority**: Actor can only modify rows for teams they manage.
    
    ```typescript
    // Team-scoped actor can modify:
    const managedTeamIds = actor.getManagedTeamIds();
    
    // ✓ Team pool rows where teamId ∈ managedTeamIds
    // ✓ Team-member rows where teamId ∈ managedTeamIds
    // ✗ Personal rows for other users
    // ✗ Rows for teams not in managedTeamIds
    ```
    
    <Warning>
    Team-scoped actors cannot remove the last team they manage if it would leave them without management access.
    </Warning>
  </Tab>
  
  <Tab title="Self scope">
    **Personal row only**: Actor can only modify their own personal stakeholder row.
    
    ```typescript
    // Self-scoped actor can modify:
    // ✓ Their personal row (userId = actor.id, teamId = null)
    // ✗ Any other row
    
    // Typical use case: Adjusting own access level or removing self
    ```
    
    <Check>
    If actor removes their personal row and no other FULL access stakeholder exists, sync is rejected.
    </Check>
  </Tab>
  
  <Tab title="None scope">
    **No authority**: Actor cannot modify stakeholders.
    
    ```typescript
    // Sync request immediately rejected with 403 Forbidden
    throw new ForbiddenException(
      "You do not have permission to manage stakeholders"
    );
    ```
  </Tab>
</Tabs>

### Row-level authorization

After scope is determined, each row change is validated:

<Steps>
  <Step title="Determine operation">
    Compare new stakeholder list to existing to identify adds, updates, and deletes
  </Step>
  
  <Step title="Check scope coverage">
    For each modified row, verify it falls within actor's management scope
  </Step>
  
  <Step title="Validate structure">
    Run structural validation on the complete new stakeholder list
  </Step>
  
  <Step title="Check orphan protection">
    Ensure at least one FULL access stakeholder remains (unless moving to pool)
  </Step>
</Steps>

```typescript
// Row-level authorization example (team scope)
function authorizeRowChange(
  row: StakeholderRow,
  operation: "add" | "update" | "delete",
  scope: ManagementScope,
  actor: User
): void {
  if (scope === "org") {
    return; // Org scope can modify anything
  }
  
  if (scope === "team") {
    const managedTeamIds = actor.getManagedTeamIds();
    
    // Team pool or team-member row must be for managed team
    if (row.teamId && !managedTeamIds.includes(row.teamId)) {
      throw new ForbiddenException(
        `You cannot modify stakeholders for team ${row.teamId}`
      );
    }
    
    // Personal rows (no teamId) are forbidden for team scope
    if (!row.teamId && row.userId !== actor.id) {
      throw new ForbiddenException(
        "Team-scoped users cannot modify personal stakeholder rows"
      );
    }
    
    return;
  }
  
  if (scope === "self") {
    // Only actor's personal row allowed
    if (row.userId !== actor.id || row.teamId !== null) {
      throw new ForbiddenException(
        "You can only modify your own personal stakeholder row"
      );
    }
    return;
  }
  
  // scope === "none"
  throw new ForbiddenException(
    "You do not have permission to manage stakeholders"
  );
}
```

### Orphan protection

<Warning>
Stakeholder sync must not leave the entity orphaned (no one with FULL access).
</Warning>

Exception: Moving to pool is allowed if:

1. New stakeholder list is empty AND
2. Actor has org-wide move-to-pool permission

```typescript
function validateNotOrphaned(
  stakeholders: StakeholderRow[],
  allowMoveToPool: boolean
): void {
  if (stakeholders.length === 0) {
    if (!allowMoveToPool) {
      throw new BadRequestException(
        "Cannot remove all stakeholders without move-to-pool permission"
      );
    }
    return; // Intentional move to pool
  }
  
  const hasFullAccess = stakeholders.some(s => s.accessLevel === "FULL");
  
  if (!hasFullAccess) {
    throw new BadRequestException(
      "At least one stakeholder must have FULL access"
    );
  }
}
```

## Frontend permission flags

### Lead permissions object

The backend computes `lead.permissions` and sends it to the frontend. Frontend uses these flags to show/hide UI elements:

```typescript
interface LeadPermissions {
  canView: boolean;
  canModify: boolean;
  canDelete: boolean;
  canTransfer: boolean;
  canManageStakeholders: boolean;
  stakeholderManagementScope?: "org" | "team" | "self" | "none";
  managedTeamIds?: string[];
}
```

<Info>
Permissions are computed by `permission-compute.service.ts` and fail-closed (default `false`) when computation fails or user has no access.
</Info>

### Frontend permission checks

Frontend mirrors these flags in `lead-permissions.ts`:

```typescript
// Example: Show manage stakeholders button
const canShowManageButton = computed(() => {
  return (
    lead.value?.permissions?.canManageStakeholders === true &&
    lead.value?.permissions?.stakeholderManagementScope !== "none"
  );
});

// Example: Show delete lead button
const canShowDeleteButton = computed(() => {
  return lead.value?.permissions?.canDelete === true;
});
```

<Warning>
Frontend permission checks are UX-only. Backend always enforces actual authorization.
</Warning>

  </Tab>

## Agency cut (finance permission)

<Warning>
Only users with `finance.manage` permission can view or modify `orgCommissionPercentage` (agency cut).
</Warning>

### Agency cut field behavior

<Tabs>
  <Tab title="With finance.manage">
    **Full agency cut control**:
    
    ```typescript
    const canManageFinance = user.hasPermission("finance.manage");
    ```

### Frontend commission display

The visible agent pool percentage accounts for agency cut:

```typescript
function calculateVisiblePercentages(
  stakeholders: StakeholderRow[],
  orgCommissionPercentage: number
): { agentPool: number; allocatedToAgents: number; remaining: number } {
  const agentPool = 100 - orgCommissionPercentage;
  
  const allocatedToAgents = stakeholders.reduce(
    (sum, row) => sum + (row.commissionPercentage || 0),
    0
  );
  
  const remaining = agentPool - allocatedToAgents;
  
  return { agentPool, allocatedToAgents, remaining };
}
```

<Check>
Total commission must equal `100%` (agency cut + agent allocations) for deals, or `0%` or `100%` for leads.
</Check>

## Moving to pool

"Moving to pool" means removing all stakeholders, making the lead/deal available to the unassigned pool.

<Warning>
Only users with org-wide pool management permission can move leads/deals to pool.
</Warning>

### Pool move authorization

## Claiming from pool

Claiming allows a user to self-assign from the unassigned pool.

### Claim authorization

```typescript
// Backend authorization
function canClaimLead(actor: User, lead: Lead): boolean {
  // Lead must be in pool (no stakeholders)
  if (lead.stakeholders.length > 0) {
    return false;
  }
  
  // Actor must have claim permission

}
```

### Claim operation

<Steps>
  <Step title="Check lead is in pool">
    Lead must have zero stakeholders
  </Step>
  
  <Step title="Verify claim permission">
    Actor must have `leads.claim` permission
  </Step>
  
  <Step title="Create personal stakeholder">
    Add actor as personal stakeholder with FULL access and primary=true
  </Step>
  
  <Step title="Set commission">
    Set commission to 100% (or 0% for pre-deal leads)
  </Step>
</Steps>

```typescript

  const stakeholder = {
    userId: actor.id,
    teamId: null,
```

## Contact stakeholder rules (§21)

<Warning>
Contact stakeholders are fundamentally different from lead/deal stakeholders. They are access-only, with no commission or primary assignee concepts.
</Warning>

### Key differences

| Feature                  | Leads/Deals               | Contacts           |
| ------------------------ | ------------------------- | ------------------ |
| Commission               | Yes                       | No                 |
| Primary assignee         | Yes (exactly one)         | No (always false)  |
| Member self-assignment   | Yes (if allowed)          | **No**             |
| Mutation API             | `PUT .../stakeholders` (sync) | Granular add/update/remove |
| Agency cut               | Yes (`orgCommissionPercentage`) | No               |

### Contact structural validation

Contact stakeholders use a subset of lead/deal validation rules:

<Check>
**Rules that apply to contacts**:
- At least one entry
- Every entry identifies someone
- No duplicate row identity
- Team member requires pool
- Member access cannot exceed team access
</Check>

<Warning>
**Rules that DO NOT apply to contacts**:
- Exactly one primary (contacts have no primary)
- Primary cannot be READ (no primary concept)
- Commission-related rules (no commission)
- Agency cut rules (no agency cut)
</Warning>

### Contact create authorization

```typescript
// Contact creation authorization
function resolveContactCreateScope(actor: User): "org" | "team" | "none" {
  if (actor.hasPermission("contacts.create.org")) {
    return "org";
  }
  
  if (actor.hasPermission("contacts.create.team")) {
    return "team";
  }
  
  // No self-scope for contacts
  return "none";
}

// Member self-assignment DISABLED for contacts
const allowMemberSelfAssign = false;
```

<Info>
Unlike leads, contacts do not allow member self-assignment during creation. Only team-scoped or org-scoped stakeholder assignments are permitted.
</Info>

### Contact-specific validation

```typescript
// Contact service validation
function validateContactStakeholders(
  stakeholders: ContactStakeholderDto[]
): void {
  // Must have at least one stakeholder
  if (stakeholders.length === 0) {
    throw new BadRequestException(
      "Contact must have at least one assignee"
    );
  }
  
  // Each row must identify someone
  for (const stakeholder of stakeholders) {
    if (!stakeholder.userId && !stakeholder.teamId) {
      throw new BadRequestException(
        "Each assignee must have userId, teamId, or both"
      );
    }
  }
  
  // No duplicates
  if (hasDuplicateIdentities(stakeholders)) {
    throw new BadRequestException(
      "Cannot have duplicate assignee rows"
    );
  }
  
  // Team members require pool
  validateTeamMembersHavePool(stakeholders);
  
  // Member access cannot exceed team
  validateMemberAccessLevels(stakeholders);
  
  // No commission validation (access-only)
  // No primary validation (contacts have no primary)
  
  // At least one FULL access
  const hasFullAccess = stakeholders.some(s => s.accessLevel === "FULL");
  if (!hasFullAccess) {
    throw new BadRequestException(
      "At least one assignee must have FULL access"
    );
  }
}
```

### Contact self-row rejection

<Warning>
Backend explicitly rejects personal stakeholder rows during contact creation that match the actor's identity.
</Warning>

```typescript
// Contact service guard
function validateNotSelfAssignment(
  stakeholders: ContactStakeholderDto[],
  actorId: string
): void {
  const hasSelfRow = stakeholders.some(
    s => s.userId === actorId && !s.teamId
  );
  
  if (hasSelfRow) {
    throw new BadRequestException(
      "Cannot self-assign to contacts during creation. " +
      "Contact must be assigned to team or other users."
    );
  }
}
```

This prevents the lead/deal pattern where members can self-assign with `leads.create.self` permission. Contacts require team or org-level assignment.

## Common validation errors

<AccordionGroup>
  <Accordion title="Empty stakeholder list (without pool permission)">
    **Error**: `Cannot remove all stakeholders without move-to-pool permission`
    
    **Cause**: Trying to sync an empty stakeholder list without org-wide pool management permission.
    
    **Fix**: Either keep at least one stakeholder with FULL access, or obtain `leads.pool.manage` / `deals.pool.manage` permission.
  </Accordion>
  
  <Accordion title="No primary assignee">
    **Error**: `Exactly one stakeholder must be marked as primary`
    
    **Cause**: Stakeholder list has zero or multiple primary rows.
    
    **Fix**: Ensure exactly one row has `isPrimary=true`.
  </Accordion>
  
  <Accordion title="Primary with READ access">
    **Error**: `Primary stakeholder cannot have READ access level`
    
    **Cause**: Primary row has `accessLevel="READ"`.
    
    **Fix**: Set primary row to `WRITE` or `FULL`. Frontend auto-promotes READ to WRITE.
  </Accordion>
  
  <Accordion title="Duplicate row identity">
    **Error**: `Duplicate stakeholder identity: (userId, teamId)`
    
    **Cause**: Same `(userId, teamId)` combination appears twice.
    
    **Fix**: Remove duplicate rows. Each `(userId, teamId)` tuple must be unique.
  </Accordion>
  
  <Accordion title="Team member without pool">
    **Error**: `Team member row requires matching team pool row`
    
    **Cause**: Team-member row exists but no team-only row with same `teamId`.
    
    **Fix**: Add team pool row with matching `teamId` and `originalTeamCommission`.
  </Accordion>
  
  <Accordion title="Member access exceeds team">
    **Error**: `Team member access level cannot exceed team pool access level`
    
    **Cause**: Team-member row has higher access than parent team pool.
    
    **Fix**: Cap member access to team pool access level (e.g., if team has WRITE, member cannot have FULL).
  </Accordion>
  
  <Accordion title="Commission total mismatch">
    **Error**: `Total commission must equal 0% or 100% for leads` / `Total commission must equal 100% for deals`
    
    **Cause**: Sum of all stakeholder commissions doesn't match required total.
    
    **Fix**: Adjust commission percentages to match required total. Remember to account for agency cut.
  </Accordion>
  
  <Accordion title="Team members exceed budget">
    **Error**: `Team member commissions exceed team budget`
    
    **Cause**: Sum of team-member commissions > `originalTeamCommission`.
    
    **Fix**: Reduce member commissions or increase team budget (`originalTeamCommission`).
  </Accordion>
  
  <Accordion title="Team leftover mismatch">
    **Error**: `Team pool commission must equal originalTeamCommission - member allocations`
    
    **Cause**: Team pool `commissionPercentage` doesn't equal `originalTeamCommission - sum(member commissions)`.
    
    **Fix**: Recalculate team pool leftover. Frontend should auto-compute this value.
  </Accordion>
  
  <Accordion title="Unauthorized stakeholder modification">
    **Error**: `You cannot modify stakeholders for team X` / `You can only modify your own personal stakeholder row`
    
    **Cause**: Actor's management scope doesn't cover the modified rows.
    
    **Fix**: Only modify rows within your scope (org, team, or self). Check `lead.permissions.stakeholderManagementScope`.
  </Accordion>
  
  <Accordion title="No FULL access stakeholder">
    **Error**: `At least one stakeholder must have FULL access`
    
    **Cause**: Sync would leave entity orphaned with no FULL access stakeholder.
    
    **Fix**: Ensure at least one stakeholder has `accessLevel="FULL"` (unless moving to pool).
  </Accordion>
  
  <Accordion title="Contact member self-assignment">
    **Error**: `Cannot self-assign to contacts during creation`
    
    **Cause**: Contact stakeholder payload includes personal row for actor.
    
    **Fix**: Remove actor's personal row. Assign contact to team or other users instead.
  </Accordion>
  
  <Accordion title="Unauthorized agency cut modification">
    **Error**: `You do not have permission to modify organization commission`
    
    **Cause**: Actor lacks `finance.manage` permission but payload includes `orgCommissionPercentage`.
    
    **Fix**: Remove `orgCommissionPercentage` from payload if you lack finance permission.
  </Accordion>
</AccordionGroup>
