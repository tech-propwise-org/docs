## Structural Validation Rules

<Note>
These rules are mirrored in frontend `stakeholder-validation.ts` and backend `stakeholder-validation.utils.ts`.
</Note>

### Core Validation Rules

| Rule | Details |
| ---- | ------- |
| At least one entry | A stakeholder payload must contain entries unless the lead is intentionally moved to pool by an allowed sync path. |
| Every entry identifies someone | Each row must have `userId`, `teamId`, or both. |
| Exactly one primary | Whenever stakeholders exist, exactly one row must be primary. |
| Primary cannot be READ | A primary row must be `WRITE` or `FULL`. The frontend auto-promotes READ to WRITE when a row becomes primary. |
| No duplicate row identity | The same `(userId, teamId)` combination cannot appear twice. |

### Team and Commission Rules

| Rule | Details |
| ---- | ------- |
| `originalTeamCommission` only on team pool rows | Only rows with `teamId` and no `userId` may carry `originalTeamCommission`. |
| Team pool row requires budget | A team-only row must have `originalTeamCommission`. |
| Team member requires pool | A `userId + teamId` row requires a matching `teamId`-only row. |
| Member access cannot exceed team access | A team-member row cannot have higher access than its parent team pool row. |
| No negative commissions | `commissionPercentage` must be `>= 0`. DTOs also cap it at `<= 100`. |
| Total commission must match target | Leads allow `0%` or the configured target. Deals require `100%`. |
| Top-level allocation must match target | Standalone users plus team budgets must equal the target, or `0%` when the lead-specific zero-total exception applies. |
| Team members cannot exceed budget | Sum of member rows under a team must be less than or equal to the team budget. |
| Team leftover must be exact | Team pool `commissionPercentage` must equal `originalTeamCommission - sum(member commissions)`, clamped at `0`. |

### Lead-Specific Exceptions

<Info>
Leads differ from deals:

- Leads allow a `0%` commission total. This supports early assignment before a deal exists.
</Info>

### Contact-Specific Exceptions

<Warning>
Contacts differ from leads and deals:

- **No commission fields**: All commission-related fields (`commissionPercentage`, `originalTeamCommission`) are rejected at DTO level
- **No primary assignee**: `isPrimary` must always be `false`
- **Access-only purpose**: Stakeholders grant permanent contact access without financial implications
- **No self-row rejection on create**: Unlike leads, members can create contacts with their own personal row
- **Team-scoped management**: Non-admins can only manage assignees for contacts they have access to and teams they belong to
</Warning>

## Access Level Hierarchy

Access levels determine what actions a stakeholder can perform on an entity:

<Tabs>
  <Tab title="READ">
    - View the entity
    - View entity details
    - Cannot modify or delete
    - Cannot manage other stakeholders
  </Tab>
  
  <Tab title="WRITE">
    - All READ permissions
    - Edit entity fields
    - Cannot delete entity
    - Cannot manage stakeholders (unless additional permissions)
  </Tab>
  
  <Tab title="FULL">
    - All WRITE permissions
    - Delete the entity
    - Manage stakeholders (subject to scope restrictions)
    - Transfer ownership
  </Tab>
</Tabs>

<Tip>
When a team-member row is created, its access level is automatically capped to match or be lower than the parent team pool row's access level.
</Tip>

## Primary Assignee Rules

The primary assignee is the main point of contact and decision-maker for a lead or deal.

<Steps>
  <Step title="Exactly one primary">
    When stakeholders exist on a lead or deal, exactly one row must be marked `isPrimary=true`.
  </Step>
  
  <Step title="Minimum access level">
    The primary row must have `WRITE` or `FULL` access. `READ`-only rows cannot be primary.
  </Step>
  
  <Step title="Auto-promotion">
    The frontend automatically promotes a `READ` row to `WRITE` when a user attempts to mark it as primary.
  </Step>
  
  <Step title="Contact exception">
    Contacts do not use primary assignees. All contact stakeholder rows must have `isPrimary=false`.
  </Step>
</Steps>

## Commission Allocation

### Agency Cut

<Info>
The agency cut (`orgCommissionPercentage`) represents the organization's share of the commission. Only users with `finance.manage` permission can view or modify this field.
</Info>

<CodeGroup>
```typescript Frontend Display
// The frontend shows the "allocatable pool" to agents
const allocatablePercentage = 100 - orgCommissionPercentage;

// Example: If agency takes 20%, agents see 80% to allocate
```

```typescript Backend Validation
// Backend stores and validates the full 100% accounting
validateTotalCommission(stakeholders, targetPercentage);
```
</CodeGroup>

### Top-Level Allocation

The sum of all top-level commission allocations must equal the target:

- **Personal rows**: Each contributes its `commissionPercentage`
- **Team pool rows**: Each contributes its `originalTeamCommission`

<Check>
For leads, the target can be `0%` (unallocated) or the configured commission percentage.
</Check>

<Warning>
For deals, the total must always equal `100%`. Zero-commission deals are not allowed.
</Warning>

### Team Budget and Member Allocation

<Steps>
  <Step title="Set team budget">
    Create a team-only row with `originalTeamCommission` representing the team's total budget.
  </Step>
  
  <Step title="Allocate to members">
    Create team-member rows (with both `userId` and `teamId`) to allocate portions of the budget to specific team members.
  </Step>
  
  <Step title="Calculate leftover">
    The team pool row's `commissionPercentage` must equal:
    ```
    originalTeamCommission - sum(member commissions)
    ```
    This is clamped at `0` (cannot be negative).
  </Step>
  
  <Step title="Validate total">
    The sum of member allocations cannot exceed `originalTeamCommission`.
  </Step>
</Steps>

## Management Scopes

The backend computes a management scope for each stakeholder edit operation, determining what changes a user can make.

### Scope Levels

<Tabs>
  <Tab title="org">
    **Organization-wide management**
    
    - Granted to users with `crm.admin` permission
    - Can modify any stakeholder row
    - Can set agency cut
    - Can assign to any user or team
  </Tab>
  
  <Tab title="team">
    **Team-scoped management**
    
    - Granted to team managers or users with FULL access
    - Can modify stakeholder rows for their managed teams
    - Cannot change agency cut
    - Can assign team members from their teams
    - Can modify team pool rows for their teams
  </Tab>
  
  <Tab title="self">
    **Self-only management**
    
    - Limited to modifying the user's own stakeholder row
    - Cannot add new rows
    - Cannot remove their own row (would lose access)
    - Cannot change primary status
    - Can only adjust their own commission within constraints
  </Tab>
  
  <Tab title="none">
    **No management permission**
    
    - User cannot modify stakeholders
    - Read-only access to stakeholder list
    - All edit operations rejected
  </Tab>
</Tabs>

### Scope Resolution Logic

The backend determines management scope using this hierarchy:

<Steps>
  <Step title="Check org-level admin">
    If user has `crm.admin` â†’ scope is `org`
  </Step>
  
  <Step title="Check entity access level">
    If user has `FULL` access to the entity â†’ scope is `team` (limited to their teams)
  </Step>
  
  <Step title="Check team manager status">
    If user is a manager of any team with a stakeholder row â†’ scope is `team`
  </Step>
  
  <Step title="Check self stakeholder">
    If user has a personal stakeholder row â†’ scope is `self`
  </Step>
  
  <Step title="Default to none">
    Otherwise â†’ scope is `none`
  </Step>
</Steps>

## Create-Time Authorization

When creating a new lead, deal, or contact, the backend validates the initial stakeholder payload.

### Lead and Deal Creation

<CodeGroup>
```typescript Authorization Flow
// From stakeholder-create-authorization.util.ts
export function authorizeStakeholderCreate(
  user: User,
  stakeholders: StakeholderPayload[],
  entityType: 'lead' | 'deal'
): AuthorizationResult {
  // 1. Structural validation
  validateStakeholderStructure(stakeholders);
  
  // 2. Permission check
  if (user.hasPermission('crm.admin')) {
    return { allowed: true, scope: 'org' };
  }
  
  // 3. Member self-assignment rules
  if (isSelfAssignmentOnly(stakeholders, user.id)) {
    return { allowed: true, scope: 'self' };
  }
  
  // 4. Team-based assignment
  if (canAssignToManagedTeams(stakeholders, user.teams)) {
    return { allowed: true, scope: 'team' };
  }
  
  return { allowed: false, scope: 'none' };
}
```

```json Example Payload
{
  "name": "New Lead",
  "stakeholders": [
    {
      "userId": "current-user-id",
      "commissionPercentage": 100.00,
      "isPrimary": true,
      "accessLevel": "FULL"
    }
  ]
}
```
</CodeGroup>

<Info>
**Self-assignment for members:** Non-admin users can create leads/deals with themselves as the sole stakeholder. This is the primary self-serve creation path.
</Info>

<Warning>
**Agency cut restriction:** Only users with `finance.manage` can set `orgCommissionPercentage` during creation. Others must omit it or set it to `0`.
</Warning>

### Contact Creation

Contact creation differs from lead/deal creation:

<Tabs>
  <Tab title="No self-row rejection">
    Unlike leads, members **can** create contacts with their own personal stakeholder row without restriction.
  </Tab>
  
  <Tab title="Access-only validation">
    - No `commissionPercentage` allowed
    - No `originalTeamCommission` allowed
    - All rows must have `isPrimary: false`
  </Tab>
  
  <Tab title="Team assignment">
    Users can assign contacts to teams they belong to, subject to the same team-scoped rules as leads.
  </Tab>
</Tabs>

## Sync-Time Authorization

When updating stakeholders on an existing lead or deal via `PUT /:entity/:id/stakeholders`, the backend performs row-level authorization.

### Authorization Process

<Steps>
  <Step title="Compute management scope">
    Backend determines the user's scope: `org`, `team`, `self`, or `none`
  </Step>
  
  <Step title="Diff existing and new rows">
    Compare the current stakeholder set with the incoming payload to identify:
    - Added rows
    - Removed rows
    - Modified rows (commission, access level, primary status)
  </Step>
  
  <Step title="Authorize each change">
    For each row change, verify the user has permission:
    - `org` scope: can change anything
    - `team` scope: can only modify rows for managed teams
    - `self` scope: can only modify own row
  </Step>
  
  <Step title="Validate final state">
    After authorization passes, run structural validation on the complete new set
  </Step>
</Steps>

### Team-Scoped Management Rules

<AccordionGroup>
  <Accordion title="What team managers CAN do">
    - Add/remove team pool rows for their teams
    - Add/remove team-member rows for their team members
    - Modify commission allocations within their team budget
    - Change access levels for team rows (subject to hierarchy rules)
    - Modify their own personal row if they have one
  </Accordion>
  
  <Accordion title="What team managers CANNOT do">
    - Change the agency cut (`orgCommissionPercentage`)
    - Modify rows for other teams they don't manage
    - Remove other users' personal rows
    - Change primary status to a row outside their scope
    - Exceed team budget when allocating to members
  </Accordion>
</AccordionGroup>

### Self-Scoped Management Rules

<Warning>
Users with only `self` scope have very limited stakeholder management:

- Can only modify their own personal stakeholder row
- Cannot add new stakeholders
- Cannot remove themselves (would lose access)
- Can adjust their commission percentage within constraints
- Cannot change their access level (would require higher permission)
- Cannot change primary status
</Warning>

## Primary Assignee Promotion

When a row becomes primary, the frontend automatically ensures it meets access level requirements.

### Promotion Rules

<Steps>
  <Step title="Check current access level">
    Before marking a row as primary, check its `accessLevel`.
  </Step>
  
  <Step title="Auto-promote READ to WRITE">
    If the row has `READ` access, automatically upgrade it to `WRITE`.
    
    ```typescript
    if (row.accessLevel === 'READ' && row.isPrimary) {
      row.accessLevel = 'WRITE';
    }
    ```
  </Step>
  
  <Step title="Allow WRITE and FULL">
    Rows with `WRITE` or `FULL` access can become primary without changes.
  </Step>
  
  <Step title="Visual indicator">
    The UI shows a note: "Access level promoted to WRITE (required for primary assignee)"
  </Step>
</Steps>

<Info>
This promotion happens automatically in the frontend before submission. The backend enforces the same rule and will reject any primary row with `READ` access.
</Info>

### Team Member Primary Constraint

<Warning>
When a team-member row becomes primary and is promoted from READ to WRITE, the parent team pool row must also have at least WRITE access.

If the team pool row has only READ access, the operation is blocked with an error: "Cannot promote team member to primary: team access level is too low"
</Warning>

## Team Member Access Capping

Team-member rows inherit access level constraints from their parent team pool row.

### Capping Rules

When adding or editing a team-member row:

<Tabs>
  <Tab title="Team has FULL">
    Member can have `READ`, `WRITE`, or `FULL`
  </Tab>
  
  <Tab title="Team has WRITE">
    Member can have `READ` or `WRITE` (FULL is capped to WRITE)
  </Tab>
  
  <Tab title="Team has READ">
    Member can only have `READ` (WRITE and FULL are capped to READ)
  </Tab>
</Tabs>

### Frontend Implementation

```typescript
function capMemberAccessLevel(
  memberAccessLevel: AccessLevel,
  teamAccessLevel: AccessLevel
): AccessLevel {
  const levels: AccessLevel[] = ['READ', 'WRITE', 'FULL'];
  const memberRank = levels.indexOf(memberAccessLevel);
  const teamRank = levels.indexOf(teamAccessLevel);
  
  if (memberRank > teamRank) {
    return teamAccessLevel; // Cap to team level
  }
  
  return memberAccessLevel;
}
```

<Check>
The frontend applies capping automatically when users select access levels in dropdowns. Backend validation enforces the same constraint.
</Check>

## Zero-Commission Lead Exception

Leads have a special case that deals do not: they can have `0%` total commission.

### When Zero Commission Is Allowed

<Info>
Zero-commission leads support early-stage assignment before a deal is created:

- Lead is created with stakeholders for access control
- No commission percentages are allocated yet
- All stakeholder rows have `commissionPercentage: 0`
- Primary assignee is still required (but with 0% commission)
</Info>

### Validation Logic

```typescript
function validateLeadCommissionTotal(
  stakeholders: Stakeholder[],
  targetPercentage: number
): ValidationResult {
  const total = stakeholders.reduce(
    (sum, s) => sum + s.commissionPercentage,
    0
  );
  
  // Leads allow either 0% or the target percentage
  if (total === 0 || total === targetPercentage) {
    return { valid: true };
  }
  
  return {
    valid: false,
    error: `Total commission must be 0% or ${targetPercentage}%`
  };
}
```

### Deal Comparison

<Warning>
Deals do **not** support zero commission:

- Deals always require `100%` total commission allocation
- This enforces that commission is fully planned when a deal is signed
- Attempting to create or update a deal with `0%` total is rejected
</Warning>

## Pool Return Flow

Leads can be "returned to pool" by removing all stakeholders, but only under specific authorization conditions.

### Requirements for Pool Return

<Steps>
  <Step title="Management scope check">
    User must have `org` or `team` management scope. `self`-scoped users cannot return leads to pool.
  </Step>
  
  <Step title="Empty stakeholder payload">
    The sync payload must be an empty array: `stakeholders: []`
  </Step>
  
  <Step title="Authorization bypass">
    Normal "at least one entry" validation is bypassed for authorized pool returns
  </Step>
  
  <Step title="Entity update">
    Lead is marked as unassigned and appears in pool queries for eligible users
  </Step>
</Steps>

<Warning>
**Self-scoped restriction:** Users with only `self` management scope cannot return a lead to pool, even if they are the sole stakeholder. This prevents users from abandoning leads without oversight.
</Warning>

### Frontend Implementation

The manage stakeholders modal provides a "Return to Pool" button:

```typescript
function handleReturnToPool() {
  // Check scope
  if (managementScope === 'self') {
    showError('You cannot return this lead to pool. Contact your manager.');
    return;
  }
  
  // Confirm action
  if (!confirm('Remove all assignees and return lead to pool?')) {
    return;
  }
  
  // Submit empty array
  await leadApi.updateStakeholders(leadId, []);
}
```

## Contact Stakeholder Differences

Contacts use stakeholders for access control only, without commission or primary assignee concepts.

### Key Differences

<AccordionGroup>
  <Accordion title="No Commission Fields">
    Contact stakeholders **never** include:
    - `commissionPercentage`
    - `originalTeamCommission`
    
    Any attempt to include these fields is rejected with `400 Bad Request`.
  </Accordion>
  
  <Accordion title="No Primary Assignee">
    - `isPrimary` must always be `false`
    - All contact stakeholders have equal status
    - UI does not show primary selection controls
  </Accordion>
  
  <Accordion title="Granular Mutation API">
    Instead of sync, contacts use:
    - `POST /contacts/:id/stakeholders` - Add assignee
    - `PATCH /contacts/:id/stakeholders/:stakeholderId` - Update assignee
    - `DELETE /contacts/:id/stakeholders/:stakeholderId` - Remove assignee
  </Accordion>
  
  <Accordion title="Access-Only Purpose">
    Contact stakeholders grant:
    - Permanent visibility in contact lists
    - Ability to view contact details
    - Ability to modify contact (if WRITE or FULL)
    - No financial implications
  </Accordion>
</AccordionGroup>

### Contact-Specific Validation

```typescript
function validateContactStakeholder(row: StakeholderPayload): ValidationResult {
  // Must have identifiers
  if (!row.userId && !row.teamId) {
    return { valid: false, error: 'Must specify userId or teamId' };
  }
  
  // Cannot be primary
  if (row.isPrimary === true) {
    return { valid: false, error: 'Contact stakeholders cannot be primary' };
  }
  
  // Cannot have commission
  if (row.commissionPercentage || row.originalTeamCommission) {
    return { valid: false, error: 'Contact stakeholders do not have commission' };
  }
  
  // Must have valid access level
  if (!['READ', 'WRITE', 'FULL'].includes(row.accessLevel)) {
    return { valid: false, error: 'Invalid access level' };
  }
  
  return { valid: true };
}
```

### Contact Management Scope

<Warning>
Contact stakeholder management follows similar scope rules to leads, but with additional restrictions:

- **Org-scoped:** CRM admins can manage all contact assignees
- **Team-scoped:** Team managers can manage assignees for contacts they have access to and teams they manage
- **Self-scoped:** Users can modify their own contact stakeholder row (access level only, cannot remove self)
- **No self-row rejection on create:** Unlike leads, users can create contacts with themselves as a stakeholder without restriction
</Warning>

## Permissions Response Structure

The backend includes computed permissions in lead, deal, and contact API responses to inform frontend authorization.

### Lead Permissions

<CodeGroup>
```typescript TypeScript Interface
interface LeadPermissions {

  canModify: boolean;                         // Can edit lead fields
  canDelete: boolean;                         // Can delete the lead

  canManageStakeholders: boolean;             // Can edit stakeholder list
```

### Permission Computation

The backend computes permissions by:

<Steps>
  <Step title="Check system permissions">
    User's role-based permissions (`crm.admin`, `finance.manage`, etc.)
  </Step>
  
  <Step title="Check stakeholder access">
    User's access level if they are a stakeholder (READ, WRITE, FULL)
  </Step>
  
  <Step title="Check team relationships">
    Teams user belongs to and teams user manages
  </Step>
  
  <Step title="Combine and compute">
    Apply permission hierarchy to determine final capability set
  </Step>
</Steps>

<Info>
Permissions are computed per-entity and per-user, cached for the request lifecycle, and included in all entity detail responses.
</Info>

## Error Handling and Messages

The system provides specific error messages for common validation and authorization failures.

### Structural Validation Errors

<AccordionGroup>
  <Accordion title="Empty stakeholder list">
    **Error:** `Stakeholder list cannot be empty`  
    **When:** Attempting to create a lead/deal without stakeholders (except authorized pool return)  
    **Resolution:** Add at least one stakeholder or use the pool return flow
  </Accordion>
  
  <Accordion title="No primary assignee">
    **Error:** `Exactly one stakeholder must be marked as primary`  
    **When:** No row has `isPrimary: true`, or multiple rows are primary  
    **Resolution:** Mark exactly one row as primary
  </Accordion>
  
  <Accordion title="Primary with READ access">
    **Error:** `Primary stakeholder cannot have READ access level`  
    **When:** Attempting to make a READ row primary without promoting  
    **Resolution:** Frontend auto-promotes to WRITE; backend rejects if not promoted
  </Accordion>
  
  <Accordion title="Duplicate stakeholder">
    **Error:** `Duplicate stakeholder: (userId, teamId) combination already exists`  
    **When:** Same user-team combination appears twice  
    **Resolution:** Remove the duplicate row
  </Accordion>
  
  <Accordion title="Commission total mismatch">
    **Error:** `Total commission must equal ${targetPercentage}% (currently ${actualTotal}%)`  
    **When:** Sum of commissions doesn't match target (for deals, always 100%)  
    **Resolution:** Adjust commission percentages to match target
  </Accordion>
  
  <Accordion title="Team budget exceeded">
    **Error:** `Team ${teamName} member allocations (${memberTotal}%) exceed budget (${teamBudget}%)`  
    **When:** Sum of team-member commissions exceeds `originalTeamCommission`  
    **Resolution:** Reduce member allocations or increase team budget
  </Accordion>
</AccordionGroup>

### Authorization Errors

<AccordionGroup>
  <Accordion title="Insufficient permissions">
    **Error:** `You do not have permission to manage stakeholders for this ${entity}`  
    **When:** User lacks management scope to make the requested changes  
    **Resolution:** Contact an administrator or team manager
  </Accordion>
  
  <Accordion title="Cannot modify other teams">
    **Error:** `You can only manage stakeholders for teams you belong to or manage`  
    **When:** Team-scoped user attempts to modify stakeholder rows for unmanaged teams  
    **Resolution:** Remove changes to other teams' rows
  </Accordion>
  
  <Accordion title="Cannot set agency cut">
    **Error:** `Only users with finance.manage permission can set organization commission percentage`  
    **When:** User without `finance.manage` tries to set non-zero `orgCommissionPercentage`  
    **Resolution:** Remove agency cut from payload or contact a finance admin
  </Accordion>
  
  <Accordion title="Self-scope limitations">
    **Error:** `You can only modify your own stakeholder row`  
    **When:** Self-scoped user tries to add, remove, or modify other rows  
    **Resolution:** Contact a team manager to make broader changes
  </Accordion>
  
  <Accordion title="Cannot return to pool">
    **Error:** `You do not have permission to return this lead to the pool`  
    **When:** Self-scoped user attempts pool return  
    **Resolution:** Contact a team manager or admin
  </Accordion>
</AccordionGroup>

### Contact-Specific Errors

<AccordionGroup>
  <Accordion title="Commission on contact">
    **Error:** `Contact stakeholders cannot have commission percentages`  
    **When:** Attempting to set `commissionPercentage` or `originalTeamCommission` on contact stakeholder  
    **Resolution:** Remove commission fields from contact stakeholder payload
  </Accordion>
  
  <Accordion title="Primary on contact">
    **Error:** `Contact stakeholders cannot be marked as primary`  
    **When:** Setting `isPrimary: true` on contact stakeholder  
    **Resolution:** Set `isPrimary: false`
  </Accordion>
  
  <Accordion title="Contact team access">
    **Error:** `You can only manage contact assignees for contacts you have access to`  
    **When:** Attempting to modify contact stakeholders without contact access  
    **Resolution:** Request access to the contact first
  </Accordion>
</AccordionGroup>
