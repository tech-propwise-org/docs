<Note>
This is the complete practical guide to CRM stakeholder (assignee) rules. The backend is the authority - frontend checks are UX mirrors only. If frontend and backend disagree, the backend wins.
</Note>

## Overview

In user-facing copy, "stakeholder" is rendered as **assignee** / **assignment**, but the internal API, entity, DTO, and service names still use "stakeholder".

### Entity Applicability

| Entity  | Commission / Primary | Mutation API                       | Coverage                                     |
| ------- | -------------------- | ---------------------------------- | -------------------------------------------- |
| Lead    | Yes                  | `PUT /leads/:id/stakeholders` sync | Complete rules (primary focus)               |
| Deal    | Yes (100% floor)     | `PUT /deals/:id/stakeholders` sync | Same as lead unless noted                    |
| Contact | **No** (access-only) | Granular add / update / remove     | Access-only mode, no self-assign            |

<Warning>
Deal stakeholder rules mirror lead rules for commission, primary, and sync authorization unless the engineering spec states a deal-only invariant (e.g., deals cannot have zero stakeholders).
</Warning>

## Vocabulary

| Term | Definition |
|------|------------|
| **Personal row** | `userId` set, `teamId` empty. Grants one user direct access |
| **Team pool/team-only row** | `teamId` set, `userId` empty. Team-level access and commission budget |
| **Team-member row** | Both `userId` and `teamId` set. User assigned under specific team context |
| **Primary row** | Exactly one active lead/deal stakeholder marked `isPrimary=true` |
| **Access level** | `READ`, `WRITE`, or `FULL` - determines entity permissions |
| **Access-only (contact)** | Contact stakeholders grant access only, no commission/primary |
| **Agency cut** | `orgCommissionPercentage` - only `finance.manage` users can set |
| **Management scope** | Backend computed scope: `org`, `team`, `self`, or `none` |

## Structural Validation Rules

<Note>
These rules are mirrored in frontend `stakeholder-validation.ts` and backend `stakeholder-validation.utils.ts`.
</Note>

### Core Requirements

<Steps>
  <Step title="Minimum Entry Requirement">
    A stakeholder payload must contain entries unless the lead is intentionally moved to pool by an allowed sync path.
  </Step>
  
  <Step title="Identity Requirement">
    Each row must have `userId`, `teamId`, or both to identify someone.
  </Step>
  
  <Step title="Primary Assignment">
    Whenever stakeholders exist, exactly one row must be primary (`isPrimary=true`).
  </Step>
</Steps>

### Access Level Rules

<Warning>
Primary rows cannot have `READ` access level. The frontend auto-promotes READ to WRITE when a row becomes primary.
</Warning>

- **Primary cannot be READ**: Primary row must be `WRITE` or `FULL`
- **Team member access capping**: Team-member rows cannot exceed their parent team pool access level
- **No duplicate identities**: Same `(userId, teamId)` combination cannot appear twice

### Commission Rules

<Check>
Commission validation applies to leads and deals only. Contacts use access-only mode.
</Check>

#### Team Commission Requirements

- **Team pool budget**: Team-only rows must have `originalTeamCommission`
- **Member pool dependency**: `userId + teamId` rows require matching `teamId`-only row
- **Budget constraints**: Sum of member rows ≤ team budget
- **Exact leftover calculation**: Team pool `commissionPercentage = originalTeamCommission - sum(member commissions)`, clamped at `0`

#### Commission Totals

- **Non-negative values**: `commissionPercentage ≥ 0` (DTOs cap at `≤ 100`)
- **Total matching**: Must equal configured target or `0%`
- **Top-level allocation**: Standalone users + team budgets must equal target
- **Deal requirement**: Deals require `100%` total (no zero exception)

### Lead-Specific Exceptions

<Info>
Leads differ from deals in commission handling:
- Leads allow `0%` commission total (supports early assignment before deal exists)
- Deals require `100%` commission total
</Info>

## Authorization Scopes

The backend computes management scopes that determine what stakeholder operations a user can perform:

### Scope Types

| Scope | Description | Capabilities |
|-------|-------------|--------------|
| `org` | Organization-wide management | Full stakeholder CRUD across all entities |
| `team` | Team-scoped management | Manage stakeholders within user's teams |
| `self` | Self-only management | Add/remove only their own stakeholder row |
| `none` | No management access | Read-only stakeholder information |

### Frontend Permission Mapping

The frontend converts backend `permissions` objects into UI action flags:

```typescript
// Example permission check
const canManageStakeholders = lead.permissions?.stakeholders?.manage || false;
const canSelfAssign = lead.permissions?.stakeholders?.selfAssign || false;
```

<Warning>
Frontend permissions are fail-closed - when permissions are absent, access is denied.
</Warning>

## Contact-Specific Rules

<Tip>
Contacts use "access-only" mode with different behavior from leads/deals.
</Tip>

### Contact Differences

- **No commission percentages**: Contacts don't support commission allocation
- **No primary assignee**: All contact stakeholders use `isPrimary=false`
- **Granular API**: Uses separate add/update/remove endpoints instead of sync
- **Self-assignment restrictions**: Different self-assignment rules apply

### Contact Stakeholder Management

Contact stakeholder management uses row-diffing and individual API calls:

```typescript
// Contact API pattern
ContactApi.addStakeholder(contactId, stakeholderData)
ContactApi.updateStakeholder(contactId, stakeholderId, updates)
ContactApi.removeStakeholder(contactId, stakeholderId)
```

## Implementation Guidelines

### Frontend Development

<Steps>
  <Step title="Use Shared Components">
    Leverage `stakeholder-assignment-section.tsx` with appropriate props for entity type and permissions.
  </Step>
  
  <Step title="Mirror Backend Validation">
    Frontend validation should mirror backend rules but never replace them.
  </Step>
  
  <Step title="Handle Permissions Properly">
    Always check backend permissions and fail closed when permissions are missing.
  </Step>
  
  <Step title="Entity-Specific Behavior">
    Use `entityType` and `isAccessOnly` props to gate contact-specific behavior.
  </Step>
</Steps>

### Backend Development

<Steps>
  <Step title="Validate Structurally">
    Use `stakeholder-validation.utils.ts` for structural validation before business logic.
  </Step>
  
  <Step title="Check Authorization">
    Apply create/sync authorization utilities based on the operation type.
  </Step>
  
  <Step title="Compute Permissions">
    Ensure frontend receives proper permission objects for UI state management.
  </Step>
  
  <Step title="Entity-Specific Logic">
    Handle contact access-only mode and deal-specific requirements appropriately.
  </Step>
</Steps>
