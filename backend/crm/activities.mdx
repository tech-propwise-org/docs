The activity system uses a lightweight `activity` table as a timeline index. Full content is stored in separate channel tables. Notes are activities with `type='note'` and tasks are activities with `type='task'`.

## Architecture

```
ACTIVITY (lightweight timeline index)
├── target_entity_type → 'lead' | 'deal' | 'contact' | 'company'
├── target_entity_id → UUID of the target entity
├── contact_id (nullable) → for bubble-up aggregation
├── activity_type + reference_id → points to channel table
├── summary (for quick timeline display)
└── is_pinned, is_important
              │
              ▼ reference_id points to:
┌──────────┬──────────┬──────────┬──────────┬──────────┬──────────┬──────────┬──────────┐
│ call_log │ email_   │ sms_log  │ whatsapp │ meeting  │ site_    │ note     │ task     │
│          │ log      │          │          │          │ visit    │          │          │
│ Content  │ Content  │ Content  │ Content  │ Content  │ Content  │ Content  │ Content  │
│ ONLY     │ ONLY     │ ONLY     │ ONLY     │ ONLY     │ ONLY     │ ONLY     │ ONLY     │
│ (no FKs) │ (no FKs) │ (no FKs) │ (no FKs) │ (no FKs) │ (no FKs) │ (no FKs) │ (no FKs) │
└──────────┴──────────┴──────────┴──────────┴──────────┴──────────┴──────────┴──────────┘
```

<Info>
  Outcome, duration, and other detailed fields are stored on the **channel
  tables**, not on `activity` itself. Activity is purely a lightweight index for
  timeline display.
</Info>

## Activity table

```
activity
├── target_entity_type (lead | deal | contact | company)
├── target_entity_id (UUID)
├── contact_id (UUID, nullable) — for bubble-up visibility
├── activity_type (call, email, sms, whatsapp, meeting, site_visit, note, task, other)
├── reference_id → channel table record (nullable for note/task)
├── summary (text for quick display: "Called +1234567890 • 5 min • Connected")
├── is_pinned, is_important
├── performed_by_id, created_by_id, created_at
```

### Activity type classification

- **Communication types** (bubble up): call, email, sms, whatsapp, meeting, site_visit
- **Internal types** (stay on entity): note, task

## Visibility rules

| Target type          | contactId logic                                    | Visible on                                                |
| -------------------- | -------------------------------------------------- | --------------------------------------------------------- |
| Lead                 | Auto-filled from `lead.person.contact` (if exists) | Lead page + Contact page (if contact exists)              |
| Deal (communication) | Required from UI (selected contact)                | Deal page + Contact page                                  |
| Deal (internal note) | `null`                                             | Deal page only                                            |
| Contact              | `contact.id`                                       | Contact page + Company pages (via `contact_company_role`) |
| Company              | `null`                                             | Company page only                                         |

### Contact ID resolution logic

The system automatically populates `contact_id` based on the target entity for bubble-up visibility:

```typescript
// Company → null (no bubble up)
if (targetType === "company") return null;

// Contact → use contact itself
if (targetType === "contact") return targetId;

// Lead → get from lead.person.contact (if exists)
if (targetType === "lead") {
  const lead = await em.findOne(Lead, targetId, {
    populate: ["person.contact"],
  });
  return lead?.person?.contact?.id ?? null;
}

// Deal (communication) → require selectedContactId from UI
// Deal (internal note) → null
if (targetType === "deal") {
  if (isCommunicationActivity(activityType)) {
    if (!selectedContactId) throw new BadRequestException("Contact required");
    return selectedContactId;
  }
  return null;
}
```

### Visibility queries

```sql
-- Lead page timeline
WHERE target_entity_type = 'lead' AND target_entity_id = :leadId
ORDER BY is_pinned DESC, created_at DESC

-- Deal page timeline
WHERE target_entity_type = 'deal' AND target_entity_id = :dealId
ORDER BY is_pinned DESC, created_at DESC

-- Contact page timeline (aggregates from leads, deals, direct activities)
WHERE contact_id = :contactId
ORDER BY is_pinned DESC, created_at DESC

-- Company page timeline (direct + from all company contacts)
WHERE (target_entity_type = 'company' AND target_entity_id = :companyId)
   OR contact_id IN (
     SELECT contact_id FROM contact_company_role
     WHERE company_id = :companyId AND is_active = true
   )
ORDER BY is_pinned DESC, created_at DESC
```

### PersonChannel integration

Channel activities integrate with the person communication system through `person_channel`:

- **WhatsApp activities**: `whatsapp.from_number`/`to_number` maps to `person_channel.channel_identifier` where `channel_type='whatsapp'`
- **SMS activities**: `sms_log.from_number`/`to_number` maps to `person_channel.channel_identifier` where `channel_type='sms'`
- **Other channels**: Follow similar patterns for linking channel activities to person communication identities
- **Detailed identities**: `person_channel.channel_identity_id` provides detailed channel user information (WhatsAppUser, InstagramUser, etc.)

<Note>
Activities are created from different contexts:
- **Incoming messages**: System creates activities when receiving WhatsApp/SMS through person_channel mapping
- **Outgoing communications**: User-initiated calls, emails, messages create activities with proper channel linking
- **Meeting scheduling**: Calendar integrations create meeting activities with sync to external calendar systems
- **Site visits**: Real Estate module creates visit activities linked to property context
</Note>

## Entity relationships

The activity system connects with the broader CRM architecture through entity relationships:

### Person + Contact model

Activities link to the Person + Contact architecture for identity and business relationships:

- **Person**: Central identity layer storing communication preferences and channel identities via `person_channel`
- **Contact**: Business relationship layer where communication activities are aggregated via `contact_id` bubble-up
- **Lead**: Sales opportunity layer where activities auto-link to Person's Contact (if qualified)

### Entity stakeholder integration

Activities integrate with the unified stakeholder system for assignment and commission tracking:

```
entity_stakeholder
├── entity_type (lead | deal | contact | company)
├── entity_id → target entity
├── person_id → Person
├── role (primary_agent, co_agent, team_lead, closer, processor, admin)
├── commission_percentage, assignment_percentage
├── Communication preferences inherited from person.preferred_contact_method
```

### Real Estate module integration

Site visit activities connect with Real Estate module entities:

- **Property context**: Links to `ref_project_id` and property data
- **Lead property interest**: Site visits for `lead_property_interest` records
- **Deal property interest**: Site visits during transaction process
- **Unit ownership**: Historical site visits for property owners via Person linkage

## Activity creation workflow

<Steps>
<Step title="Create channel record">
Create the content record in the appropriate channel table (e.g., `call_log`). This stores the full data — phone number, direction, outcome, duration, notes.
</Step>
<Step title="Create activity record">
Create an `activity` record pointing to the channel record via `reference_id`. The system auto-fills `contact_id` for bubble-up visibility based on the target entity.

```
activity:
├── target_entity_type: 'lead'
├── target_entity_id: lead-123
├── contact_id: contact-456 (auto-filled from lead.person.contact)
├── activity_type: 'call'
├── reference_id: call-log-uuid
├── summary: "Called +1234567890 • 5 min • Connected"
```

</Step>
<Step title="Timeline display">
The timeline UI shows `activity.summary` (fast query, ordered by `created_at`). Clicking to expand fetches full channel details via `reference_id` on demand.
</Step>
</Steps>

### Activity creation patterns

<Tabs>
<Tab title="Communication activities">
Created when communication occurs through various channels:
- **Person lookup**: Find Person via person_channel for WhatsApp/SMS
- **Entity context**: Link to Lead/Deal/Contact based on current view or conversation context
- **Auto-assignment**: System determines contact_id for bubble-up visibility
</Tab>
<Tab title="Meeting scheduling">
Created from calendar integrations or manual scheduling:
- **Calendar sync**: External calendar events create meeting activities
- **Entity association**: Meetings link to current Lead/Deal context
- **Attendee mapping**: Person resolution for meeting participants
</Tab>
<Tab title="Site visits">
Created from Real Estate module interactions:
- **Property context**: Links to ref_project_id and property details
- **Lead association**: Site visits typically target Lead entities
- **Visit tracking**: Status updates from scheduled → completed with feedback
</Tab>
</Tabs>

## Notes system

Notes are activities with `type='note'`. They store internal information and follow the same polymorphic pattern as other activities through the Activity table.

### Note creation

Creating a note involves two records:

<Steps>
<Step title="Create note record">
Create a `note` record with content only: title, body, attachments, privacy settings. This table has no entity foreign keys.
</Step>
<Step title="Create activity record">
Create an `activity` record with:
- `target_entity_type` and `target_entity_id` for targeting
- `activity_type: 'note'`
- `reference_id: note-uuid`
- `contact_id` auto-resolved based on target entity
</Step>
</Steps>

### Note visibility

Notes are "internal" activities that do **not** bubble up from Lead/Deal to Contact.

| Target  | Note visibility                                    |
| ------- | -------------------------------------------------- |
| Lead    | Lead page only (`contactId = null`, internal type) |
| Deal    | Deal page only (`contactId = null`, internal type) |
| Contact | Contact page + Company pages (`contactId` set)     |
| Company | Company page only (`contactId = null`)             |

### Notes query pattern

Notes are retrieved via the activity table, not directly from the `note` table:

```sql
SELECT n.*, a.target_entity_type, a.target_entity_id
FROM activity a
JOIN note n ON a.reference_id = n.id
WHERE a.target_entity_type = :type AND a.target_entity_id = :id
  AND a.activity_type = 'note'
  AND a.is_deleted = false
ORDER BY n.is_pinned DESC, a.created_at DESC;
```

## Task system

Tasks are activities with `type='task'`. They represent action items and follow-ups that need to be completed.

### Task creation

Creating a task involves two records:

<Steps>
<Step title="Create task record">
Create a `task` record with task-specific fields: title, description, status, priority, due_date, assigned_to. This table has no entity foreign keys.
</Step>
<Step title="Create activity record">
Create an `activity` record with:
- `target_entity_type` and `target_entity_id` for targeting
- `activity_type: 'task'`
- `reference_id: task-uuid`
- `contact_id` auto-resolved based on target entity (null for internal tasks)
</Step>
</Steps>

### Task visibility

Tasks are "internal" activities that do **not** bubble up from Lead/Deal to Contact, similar to notes.

| Target  | Task visibility                                    |
| ------- | -------------------------------------------------- |
| Lead    | Lead page only (`contactId = null`, internal type) |
| Deal    | Deal page only (`contactId = null`, internal type) |
| Contact | Contact page + Company pages (`contactId` set)     |
| Company | Company page only (`contactId = null`)             |

### Task management

<AccordionGroup>
<Accordion title="Task status workflow">
- **pending**: Task created, not yet started
- **in_progress**: Task actively being worked on
- **completed**: Task finished successfully
- **cancelled**: Task no longer needed

Status transitions are tracked with timestamps and user IDs for audit purposes.
</Accordion>

<Accordion title="Task assignment and due dates">
- **assigned_to_id**: Links to User who should complete the task
- **due_date**: Optional deadline for task completion
- **priority**: Categorized as low, medium, high, or urgent
- **Notifications**: System can send reminders based on due dates and priority

Users can view tasks assigned to them across all entities in a unified task dashboard.
</Accordion>

<Accordion title="Task attachments and descriptions">
- **attachments[]**: JSONB array storing file references
- **description**: Rich text field for detailed task instructions
- **title**: Required brief description of the task

Attachments follow the same pattern as other entity attachments with URL, filename, size, and MIME type.
</Accordion>
</AccordionGroup>

### Tasks query pattern

Tasks are retrieved via the activity table:

```sql
SELECT t.*, a.target_entity_type, a.target_entity_id
FROM activity a
JOIN task t ON a.reference_id = t.id
WHERE a.target_entity_type = :type AND a.target_entity_id = :id
  AND a.activity_type = 'task'
  AND a.is_deleted = false
ORDER BY t.due_date ASC, a.created_at DESC;
```

## Query patterns

The activity system supports several key query patterns for different use cases:

### Timeline queries

<Tabs>
<Tab title="Entity timeline">
```sql
-- Get all activities for a specific entity
SELECT a.*, 
       CASE 
         WHEN a.activity_type = 'call' THEN cl.summary
         WHEN a.activity_type = 'email' THEN el.subject
         WHEN a.activity_type = 'note' THEN n.title
         ELSE a.summary
       END as display_summary
FROM activity a
LEFT JOIN call_log cl ON a.activity_type = 'call' AND a.reference_id = cl.id
LEFT JOIN email_log el ON a.activity_type = 'email' AND a.reference_id = el.id  
LEFT JOIN note n ON a.activity_type = 'note' AND a.reference_id = n.id
WHERE a.target_entity_type = :entityType 
  AND a.target_entity_id = :entityId
  AND a.is_deleted = false
ORDER BY a.is_pinned DESC, a.created_at DESC;
```
</Tab>
<Tab title="Contact bubble-up">
```sql
-- Get all activities that should appear on a contact timeline
SELECT a.*, 
       a.target_entity_type as origin_type,
       a.target_entity_id as origin_id
FROM activity a
WHERE a.contact_id = :contactId
  AND a.is_deleted = false
ORDER BY a.is_pinned DESC, a.created_at DESC;
```
</Tab>
<Tab title="Company aggregation">
```sql
-- Get all activities for a company (direct + from contacts)
SELECT a.*
FROM activity a
WHERE (
  -- Direct company activities
  (a.target_entity_type = 'company' AND a.target_entity_id = :companyId)
  OR
  -- Activities from company contacts
  a.contact_id IN (
    SELECT ccr.contact_id 
    FROM contact_company_role ccr
    WHERE ccr.company_id = :companyId AND ccr.is_active = true
  )
)
AND a.is_deleted = false
ORDER BY a.is_pinned DESC, a.created_at DESC;
```
</Tab>
</Tabs>

### Analytics and reporting

<Tabs>
<Tab title="Activity volume">
```sql
-- Activity volume by type and date
SELECT 
  DATE_TRUNC('day', a.created_at) as date,
  a.activity_type,
  COUNT(*) as activity_count
FROM activity a
WHERE a.created_at BETWEEN :startDate AND :endDate
  AND a.is_deleted = false
GROUP BY DATE_TRUNC('day', a.created_at), a.activity_type
ORDER BY date DESC, activity_type;
```
</Tab>
<Tab title="User performance">
```sql
-- User activity performance
SELECT 
  u.full_name,
  a.activity_type,
  COUNT(*) as total_activities,
  COUNT(CASE WHEN a.created_at >= CURRENT_DATE - INTERVAL '7 days' THEN 1 END) as this_week
FROM activity a
JOIN "user" u ON a.created_by_id = u.id
WHERE a.created_at >= CURRENT_DATE - INTERVAL '30 days'
  AND a.is_deleted = false
GROUP BY u.id, u.full_name, a.activity_type
ORDER BY total_activities DESC;
```
</Tab>
<Tab title="Entity engagement">
```sql
-- Most active entities by activity count
SELECT 
  a.target_entity_type,
  a.target_entity_id,
  COUNT(*) as activity_count,
  MAX(a.created_at) as last_activity
FROM activity a
WHERE a.created_at >= CURRENT_DATE - INTERVAL '30 days'
  AND a.is_deleted = false
GROUP BY a.target_entity_type, a.target_entity_id
HAVING COUNT(*) >= 5
ORDER BY activity_count DESC;
```

## Business rules

### Activity lifecycle rules

<AccordionGroup>
<Accordion title="Activity creation validation">
- **Target required**: `target_entity_type` + `target_entity_id` must be valid and exist
- **Archive protection**: Cannot create activities for archived entities
- **Channel validation**: Communication activities must have valid channel data
- **Contact resolution**: System auto-resolves contact_id based on entity relationships
- **Organization scope**: All activity components must belong to the same organization
- **User permissions**: Users must have write access to the target entity
</Accordion>

<Accordion title="Activity modification rules">
- **Immutable linking**: Cannot change target_entity after creation
- **Channel updates**: Updates go to channel table, not activity table
- **Summary sync**: Activity summary updates when channel content changes
- **Permission checks**: Only creator or assigned users can modify activities
- **Status transitions**: Task and meeting status changes must follow valid state transitions
- **Audit tracking**: All modifications are logged with user and timestamp
</Accordion>

<Accordion title="Activity deletion and archiving">
- **Soft deletion**: Activities marked as deleted, not physically removed
- **Cascade rules**: Deleting entity archives related activities
- **Channel cleanup**: Channel records can be cleaned up after activity deletion
- **Audit preservation**: Deleted activities preserved for audit trails
- **Referential integrity**: Cannot delete activities referenced by other system components
- **Permission requirements**: Delete permissions checked at both activity and entity levels
</Accordion>
</AccordionGroup>

## Data consistency guarantees

### Transaction boundaries

Activity creation follows ACID transaction principles:

- **Atomicity**: Channel record + Activity record created together or both fail
- **Consistency**: All foreign key relationships validated before commit
- **Isolation**: Concurrent activity creation properly serialized
- **Durability**: Committed activities persist through system failures

### Referential integrity

| Constraint | Description |
| ---------- | ----------- |
| Channel reference | Activity.reference_id must exist in matching channel table |
| Entity reference | target_entity_id must exist in the specified entity type table |
| Contact reference | contact_id must be valid Contact or null |
| User reference | All user_id fields must reference valid, active Users |
| Organization scope | All components must belong to same organization |

### Eventual consistency patterns

For external integrations (calendar, email providers), the system handles eventual consistency:

- **Calendar sync**: Local meeting records created immediately, external sync happens asynchronously
- **Email tracking**: Tracking events (opens, clicks) update activities eventually
- **Channel status**: Message delivery status updates propagate from external APIs

<Warning>
**Important**: Activities created from different sources (incoming messages, calendar sync, manual entry) follow different validation and creation patterns. Ensure proper context and entity linking for each source type.
</Warning>
