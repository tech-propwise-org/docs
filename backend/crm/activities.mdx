The activity system serves as the central communication and interaction hub for the CRM module. Built on a Person + Contact architecture, it uses a lightweight `activity` table as a timeline index with full content stored in specialized channel tables. The system integrates with the broader CRM ecosystem including leads, deals, contacts, companies, and Real Estate module components.

## Architecture overview

The activity system follows the CRM module's core design principles with Person as the identity layer and Contact as the business relationship layer:

```
PERSON (Identity Layer)
├── Central identity and preferences
├── person_channel (WhatsApp, Instagram, etc.)
├── person_email, person_phone, person_address
└── Links to: Lead, Contact, unit_ownership, site_visit
              │
              ▼
ACTIVITY (Timeline Index)
├── target_entity_type → 'lead' | 'deal' | 'contact' | 'company'
├── target_entity_id → UUID of target entity
├── contact_id (nullable) → for bubble-up aggregation
├── activity_type + reference_id → points to channel table
├── summary (for quick timeline display)
└── is_pinned, is_important
              │
              ▼ reference_id points to:
┌──────────┬──────────┬──────────┬──────────┬──────────┬──────────┬──────────┬──────────┐
│ call_log │ email_   │ sms_log  │ whatsapp │ meeting  │ site_    │ note     │ task     │
│          │ log      │          │          │          │ visit    │          │          │
│ Content  │ Content  │ Content  │ Content  │ Content  │ Content  │ Content  │ Content  │
│ ONLY     │ ONLY     │ ONLY     │ ONLY     │ ONLY     │ ONLY     │ ONLY     │ ONLY     │
│ (no FKs) │ (no FKs) │ (no FKs) │ (no FKs) │ (no FKs) │ (no FKs) │ (no FKs) │ (no FKs) │
└──────────┴──────────┴──────────┴──────────┴──────────┴──────────┴──────────┴──────────┘
```

<Info>
The activity system uses **separation of concerns**: the `activity` table handles entity relationships and timeline ordering, while channel tables store communication-specific data. This enables efficient querying and flexible content storage patterns.
</Info>

## Core activity table

```sql
activity
├── target_entity_type (lead | deal | contact | company)
├── target_entity_id (UUID)
├── contact_id (UUID, nullable) — for bubble-up visibility
├── activity_type (call, email, sms, whatsapp, meeting, site_visit, note, task, other)
├── reference_id → channel table record (nullable for note/task)
├── summary (text for quick display: "Called +1234567890 • 5 min • Connected")
├── is_pinned, is_important
├── performed_by_id, created_by_id, created_at
├── organization_id, is_deleted
```

### Activity type classification

<Tabs>
<Tab title="Communication types">
**Bubble up to Contact timeline**: call, email, sms, whatsapp, meeting, site_visit

These activities represent external communications and are visible on both the target entity and related Contact pages.
</Tab>
<Tab title="Internal types">
**Stay on target entity**: note, task

These activities represent internal team information and do not bubble up from Lead/Deal to Contact.
</Tab>
<Tab title="System types">
**Auto-generated**: other

System-generated activities like stage changes, assignments, or entity transfers.
</Tab>

## Person + Contact integration

The activity system is deeply integrated with the Person + Contact architecture for unified identity and business relationship management.

### Person-centric communication

All communication channels link to **Person** as the central identity:

```sql
person_channel
├── person_id → Person
├── channel_type (whatsapp, instagram, facebook, telegram, sms, webchat)
├── channel_identifier (phone number, username, PSID, etc.)
├── display_name, avatar_url
├── channel_identity_id → WhatsAppUser.id, InstagramUser.id, etc.
├── status (active, inactive, blocked, unsubscribed)
├── is_primary
├── Opt-in: marketing_opt_in, transactional_opt_in
├── Engagement: first_contact_at, last_message_at, message_count
└── Verification: is_verified, verified_at
```

<Note>
**Person preferences apply globally**: `person.preferred_contact_method`, `do_not_call`, and `do_not_email` settings affect all communication attempts across leads, contacts, and deals.
</Note>

### Contact bubble-up logic

The system automatically populates `contact_id` for communication bubble-up visibility:

```typescript
// Company → null (no bubble up)
if (targetType === "company") return null;

// Contact → use contact itself
if (targetType === "contact") return targetId;

// Lead → get from lead.person.contact (if qualified)
if (targetType === "lead") {
  const lead = await em.findOne(Lead, targetId, {
    populate: ["person.contact"],
  });
  return lead?.person?.contact?.id ?? null;
}

// Deal → require selectedContactId from UI for communication
// Deal → null for internal notes/tasks
if (targetType === "deal") {
  if (isCommunicationActivity(activityType)) {
    if (!selectedContactId) throw new BadRequestException("Contact required");
    return selectedContactId;
  }
  return null;
}
```

### Entity stakeholder integration

Activities integrate with the unified stakeholder system for proper assignment and commission tracking:

```sql
entity_stakeholder
├── entity_type (lead | deal | contact | company)
├── entity_id → target entity
├── person_id → Person
├── role (primary_agent, co_agent, team_lead, closer, processor, admin)
├── commission_percentage, assignment_percentage
├── assignment_type (manual, auto_assignment, transfer, lead_conversion)
├── assigned_at, assigned_by_id
├── Communication preferences inherited from person.preferred_contact_method
```

Activities show stakeholder context and can trigger assignment notifications based on stakeholder roles and permissions.

## Visibility and timeline rules

The activity system implements sophisticated visibility rules for proper timeline aggregation across the CRM entity hierarchy.

### Visibility matrix

| Target Entity Type | contact_id Logic | Visible On |
| ------------------ | ---------------- | ---------- |
| **Lead** | Auto-filled from `lead.person.contact` (if qualified) | Lead page + Contact page (if contact exists) |
| **Deal (communication)** | Required from UI (selected contact) | Deal page + Contact page |
| **Deal (internal note/task)** | `null` | Deal page only |
| **Contact** | `contact.id` | Contact page + Company pages (via `contact_company_role`) |
| **Company** | `null` | Company page only |

### Timeline queries

<Tabs>
<Tab title="Lead timeline">
Shows all activities for the lead, plus activities from any related Deal (if lead converted):

```sql
-- Direct lead activities
WHERE target_entity_type = 'lead' AND target_entity_id = :leadId
UNION
-- Related deal activities (if lead converted)
WHERE target_entity_type = 'deal' AND target_entity_id IN (
  SELECT id FROM deal WHERE originating_lead_id = :leadId
)
ORDER BY is_pinned DESC, created_at DESC
```
</Tab>
<Tab title="Deal timeline">
Shows all deal activities plus context from originating lead:

```sql
-- Direct deal activities
WHERE target_entity_type = 'deal' AND target_entity_id = :dealId
-- Optionally include lead context activities for reference
ORDER BY is_pinned DESC, created_at DESC
```
</Tab>
<Tab title="Contact timeline">
Aggregates activities from leads, deals, and direct contact interactions:

```sql
-- All activities that should bubble up to this contact
WHERE contact_id = :contactId
ORDER BY is_pinned DESC, created_at DESC
```
</Tab>
<Tab title="Company timeline">
Shows direct company activities plus aggregated contact activities:

```sql
WHERE (target_entity_type = 'company' AND target_entity_id = :companyId)
   OR contact_id IN (
     SELECT contact_id FROM contact_company_role
     WHERE company_id = :companyId AND is_active = true
   )
ORDER BY is_pinned DESC, created_at DESC
```
</Tab>
</Tabs>

### Real Estate integration visibility

Site visit activities integrate with the Real Estate module's property and unit system:

## Notes system

Notes are activities with `type='note'` that store internal team information within the polymorphic activity framework.

### Notes architecture

```sql
note
├── title (optional)
├── body (text content)
├── attachments[] (JSONB array: {url, filename, size, mimeType})
├── is_pinned
├── is_private
├── organization_id, created_by_id, created_at
```

<Steps>
<Step title="Create note content">
Create a `note` record with title, body, and attachments. This table stores content only with no entity foreign keys.
</Step>
<Step title="Create activity link">
Create an `activity` record linking the note to its target entity:
```sql
activity:
├── target_entity_type: 'lead' | 'deal' | 'contact' | 'company'
├── target_entity_id: target-uuid
├── contact_id: null (internal type - no bubble up from Lead/Deal)
├── activity_type: 'note'
├── reference_id: note-uuid
├── summary: note.title || truncate(note.body, 100)
```
</Step>
<Step title="Timeline display">
Notes appear in entity timelines with summary text. Full content loads on-demand via `reference_id`.
</Step>
</Steps>

```typescript
async function processOverdueTasks() {
```

## Activity creation workflows

The activity system supports multiple creation patterns based on the source and context of the interaction.

### Incoming communication workflow

<Steps>
<Step title="Channel message received">
External system (WhatsApp, SMS, Email) receives incoming message and creates initial channel record with communication details.
</Step>
<Step title="Person resolution">
System resolves Person via `person_channel` mapping:
```sql
-- Find person by WhatsApp number
SELECT p.* FROM person p
JOIN person_channel pc ON p.id = pc.person_id
WHERE pc.channel_type = 'whatsapp' 
  AND pc.channel_identifier = :from_number
  AND pc.organization_id = :org_id
```
If Person not found, create new Person + person_channel record.
</Step>
<Step title="Entity context determination">
Determine target entity for activity:
- **Active conversation**: Use existing conversation context (Lead/Deal)
- **No context**: Create activity on Person's Contact (if qualified) or create new Lead
- **Multiple contexts**: Use conversation disambiguation UI
</Step>
<Step title="Activity creation">
Create activity record linking channel content to target entity with proper bubble-up logic.
</Step>
</Steps>

### Outgoing communication workflow

<Steps>
<Step title="Entity context selection">
User initiates communication from Lead, Deal, Contact, or Company page, providing clear target entity context.
</Step>
<Step title="Contact selection for deals">
For Deal communications, user must select which Deal contact to communicate with (deals can have multiple contacts).
</Step>
<Step title="Channel record creation">
Create channel-specific record (call_log, email_log, etc.) with communication details and outcome.
</Step>
<Step title="Activity linking">
Create activity record linking to target entity with auto-resolved contact_id for bubble-up visibility.
</Step>
</Steps>

### Meeting and calendar integration

<Steps>
<Step title="Calendar event creation">
User creates meeting from CRM entity or external calendar system creates event with CRM context.
</Step>
<Step title="Attendee resolution">
Map meeting attendees to Person records where possible via email addresses in `person_email` table.
</Step>
<Step title="Entity association">
Link meeting to appropriate CRM entity (Lead for sales meetings, Deal for closing meetings, Contact for relationship meetings).
</Step>
<Step title="Sync maintenance">
Maintain two-way sync with external calendar for status updates, reschedules, and cancellations.
</Step>
</Steps>

## Query patterns and performance

The activity system implements optimized query patterns for efficient timeline display and reporting across the CRM module.

### Core timeline queries

<Tabs>
<Tab title="Entity activity timeline">
```sql
-- Optimized entity timeline with channel content
SELECT 
  a.id, a.activity_type, a.summary, a.created_at, a.is_pinned,
  CASE a.activity_type
    WHEN 'call' THEN json_build_object(
      'phone_number', cl.phone_number,
      'direction', cl.direction,
      'outcome', cl.outcome,
      'duration', cl.duration_seconds
    )
    WHEN 'email' THEN json_build_object(
      'subject', el.subject,
      'from_email', el.from_email,
      'to_emails', el.to_emails
    )
    WHEN 'note' THEN json_build_object(
      'title', n.title,
      'body', LEFT(n.body, 200)
    )
  END as channel_preview
FROM activity a
LEFT JOIN call_log cl ON a.activity_type = 'call' AND a.reference_id = cl.id
LEFT JOIN email_log el ON a.activity_type = 'email' AND a.reference_id = el.id  
LEFT JOIN note n ON a.activity_type = 'note' AND a.reference_id = n.id
WHERE a.target_entity_type = :entityType 
  AND a.target_entity_id = :entityId
  AND a.is_deleted = false
ORDER BY a.is_pinned DESC, a.created_at DESC
LIMIT 50;
```
</Tab>
<Tab title="Contact bubble-up timeline">
```sql
-- Contact timeline with source entity context
SELECT 
  a.id, a.activity_type, a.summary, a.created_at,
  a.target_entity_type as source_type,
  a.target_entity_id as source_id,

LEFT JOIN deal d ON a.target_entity_type = 'deal' AND a.target_entity_id = d.id
WHERE a.contact_id = :contactId
  AND a.is_deleted = false
ORDER BY a.is_pinned DESC, a.created_at DESC;
```

```sql
  MAX(a.created_at) as last_activity,
  COUNT(CASE WHEN a.activity_type IN ('call', 'email', 'whatsapp') THEN 1 END) as communications,
  COUNT(CASE WHEN a.activity_type = 'meeting' THEN 1 END) as meetings,
  -- Engagement score calculation
  (
    COUNT(*) * 1.0 +  -- Base activity volume
    COUNT(DISTINCT DATE_TRUNC('day', a.created_at)) * 2.0 +  -- Consistency bonus
    COUNT(CASE WHEN a.activity_type = 'meeting' THEN 1 END) * 5.0  -- Meeting weight
  ) as engagement_score

  AND a.is_deleted = false
  AND a.organization_id = :organizationId
GROUP BY a.target_entity_type, a.target_entity_id
HAVING COUNT(*) >= 3  -- Minimum activity threshold
ORDER BY engagement_score DESC;
```
