The activity system serves as the central communication and interaction hub for the CRM module. It uses a lightweight `activity` table as a timeline index with full content stored in specialized channel tables, enabling efficient querying and flexible content storage patterns.

## Architecture overview

### Design principles

The activity system is built on core architectural principles that separate concerns and optimize for performance:

1. **Timeline indexing**: The `activity` table indexes all interactions without storing full content
2. **Channel separation**: Specialized tables (call_log, email_log, whatsapp, etc.) store detailed communication data
3. **Polymorphic targeting**: Activities link to any CRM entity (lead, deal, contact, company) via entity_type/entity_id
4. **Contact bubble-up**: Communication activities automatically surface on contact timelines for unified visibility
5. **Efficient querying**: Timeline queries use the activity index; full content loads on-demand via reference_id

## Activity table structure

The core activity table serves as a lightweight timeline index linking to detailed channel content:

```sql
activity
├── target_entity_type (lead | deal | contact | company)
├── target_entity_id (UUID)
├── contact_id (UUID, nullable) — for bubble-up visibility
├── activity_type (call, email, sms, whatsapp, meeting, site_visit, note, task, other)
├── reference_id → channel table record (nullable for note/task)
├── summary (text for quick display: "Called +1234567890 • 5 min • Connected")
├── is_pinned, is_important
├── performed_by_id, created_by_id, created_at
├── organization_id, is_deleted
```

<Tabs>
<Tab title="Activity types">
- **call**: Phone call logged in call_log table
- **email**: Email communication in email_log table
- **sms**: SMS message in sms_log table
- **whatsapp**: WhatsApp message in whatsapp table
- **meeting**: In-person or virtual meeting in meeting table
- **site_visit**: Property site visit (Real Estate module)
- **note**: Internal note in note table
- **task**: Action item in task table
- **other**: Custom activity types
</Tab>
<Tab title="Summary formatting">
The `summary` field provides quick context for timeline display:
- **Call**: "Called +1234567890 • 5 min • Connected"
- **Email**: "Email: Re: Property inquiry"

- **Note**: Note title or truncated body text

## Contact bubble-up logic

The system automatically populates `contact_id` for communication bubble-up visibility, enabling contacts to show activities from related leads and deals:

```typescript
// Company → null (no bubble up)
if (targetType === "company") return null;

// Contact → use contact itself
if (targetType === "contact") return targetId;

// Lead → get from lead.person.contact (if qualified)
if (targetType === "lead") {
  const lead = await em.findOne(Lead, targetId, {
    populate: ["person.contact"],
  });
  return lead?.person?.contact?.id ?? null;
}

// Deal → require selectedContactId from UI for communication
// Deal → null for internal notes/tasks
if (targetType === "deal") {
  if (isCommunicationActivity(activityType)) {
    if (!selectedContactId) throw new BadRequestException("Contact required");
    return selectedContactId;
  }
  return null;
}
```

<Note>
**Deal communication context**: When creating communication activities on deals (calls, emails, WhatsApp), the UI must specify which deal participant (DealContact) the communication is with. Internal activities like notes and tasks don't require contact selection.
</Note>

## Activity service methods

The `ActivityService` provides core methods for activity creation, retrieval, and timeline management:

### Core service methods

<AccordionGroup>
<Accordion title="createActivityInTransaction">
Creates an activity record within a database transaction, ensuring atomic creation with channel content:

```typescript
async createActivityInTransaction(
  tx: EntityManager,
  input: CreateActivityInput
): Promise<Activity> {
  // 1. Validate target entity exists and is accessible
  await this.validateTargetEntity(tx, input.targetEntityType, input.targetEntityId);
  
  // 2. Resolve contact_id using bubble-up logic
  const contactId = await this.resolveContactIdForActivity(
    tx,
    input.targetEntityType,
    input.targetEntityId,
    input.activityType,
    input.selectedContactId
  );
  
  // 3. Create activity record
  const activity = tx.create(Activity, {
    targetEntityType: input.targetEntityType,
    targetEntityId: input.targetEntityId,
    contactId,
    activityType: input.activityType,
    referenceId: input.referenceId,
    summary: input.summary,
    isPinned: input.isPinned ?? false,
    isImportant: input.isImportant ?? false,
    performedById: input.performedById,
    createdById: input.createdById,
    organizationId: input.organizationId
  });
  
  await tx.persistAndFlush(activity);
  
  // 4. Invalidate timeline cache
  await this.invalidateTimelineCache(input.targetEntityType, input.targetEntityId);
  
  return activity;
}
```
</Accordion>

<Accordion title="getActivityTimeline">
Retrieves paginated activity timeline for an entity with optional type filtering:

```typescript
async getActivityTimeline(
  entityType: EntityType,
  entityId: string,
  options: {
    page?: number;
    limit?: number;
    activityTypes?: ActivityType[];
    includeChannelContent?: boolean;
  }
): Promise<PaginatedActivities> {
  const qb = this.activityRepository.createQueryBuilder('a');
  
  qb.where({
    targetEntityType: entityType,
    targetEntityId: entityId,
    isDeleted: false
  });
  
  if (options.activityTypes?.length) {
    qb.andWhere({ activityType: { $in: options.activityTypes } });
  }
  
  qb.orderBy({ isPinned: 'DESC', createdAt: 'DESC' });
  qb.limit(options.limit ?? 50);
  qb.offset(((options.page ?? 1) - 1) * (options.limit ?? 50));
  
  const [activities, total] = await qb.getResultAndCount();
  
  // Optionally load channel content
  if (options.includeChannelContent) {
    await this.enrichActivitiesWithChannelContent(activities);
  }
  
  return {
    items: activities,
    total,
    page: options.page ?? 1,
    limit: options.limit ?? 50
  };
}
```
</Accordion>

<Accordion title="getContactBubbleUpTimeline">
Retrieves all activities surfaced to a contact from related leads and deals:

```typescript
async getContactBubbleUpTimeline(
  contactId: string,
  options: TimelineOptions
): Promise<PaginatedActivities> {
  const qb = this.activityRepository.createQueryBuilder('a')
    .where({
      contactId,
      isDeleted: false
    })
    .leftJoinAndSelect('a.lead', 'l')
    .leftJoinAndSelect('a.deal', 'd')
    .orderBy({ isPinned: 'DESC', createdAt: 'DESC' });
  
  const [activities, total] = await qb.getResultAndCount();
  
  // Enrich with source entity context
  const enriched = activities.map(activity => ({
    ...activity,
    sourceType: activity.targetEntityType,
    sourceId: activity.targetEntityId,
    sourceTitle: activity.targetEntityType === 'lead' 
      ? activity.lead?.title 
      : activity.deal?.title
  }));
  
  return { items: enriched, total };
}
```
</Accordion>

<Accordion title="enrichActivitiesWithChannelContent">
Batch-loads channel content for activities to avoid N+1 queries:

```typescript
async enrichActivitiesWithChannelContent(
  activities: Activity[]
): Promise<void> {
  const byType = this.groupActivitiesByType(activities);
  
  // Batch load each channel type
  if (byType.call?.length) {
    const callIds = byType.call.map(a => a.referenceId);
    const calls = await this.callLogRepository.find({ id: { $in: callIds } });
    const callMap = new Map(calls.map(c => [c.id, c]));
    byType.call.forEach(a => a.channelContent = callMap.get(a.referenceId));
  }
  
  if (byType.email?.length) {
    const emailIds = byType.email.map(a => a.referenceId);
    const emails = await this.emailLogRepository.find({ id: { $in: emailIds } });
    const emailMap = new Map(emails.map(e => [e.id, e]));
    byType.email.forEach(a => a.channelContent = emailMap.get(a.referenceId));
  }
  
  // Similar batch loading for other channel types...
}
```
</Accordion>
</AccordionGroup>

### Query performance patterns

<Tip>
**Timeline query optimization**: Always query the `activity` table first for timeline ordering, then batch-load channel content only for visible items. This prevents expensive joins on large channel tables and enables efficient pagination.
</Tip>

<CodeGroup>
```typescript Efficient timeline loading
// ✅ GOOD: Query activity index, then load channel content on-demand
const activities = await activityRepo.find(
  { targetEntityType: 'lead', targetEntityId: leadId },
  { orderBy: { isPinned: 'DESC', createdAt: 'DESC' }, limit: 50 }
);

// Batch load channel content for visible activities only
await activityService.enrichActivitiesWithChannelContent(activities);
```

```typescript Inefficient join pattern
// ❌ BAD: Joining all channel tables upfront
const activities = await activityRepo.find(
  { targetEntityType: 'lead', targetEntityId: leadId },
  { 
    populate: ['callLog', 'emailLog', 'whatsapp', 'note', 'task'],
    orderBy: { createdAt: 'DESC' }
  }
);
// This creates expensive LEFT JOINs on all channel tables
```
</CodeGroup>

## Activity type registry

The `ActivityTypeRegistry` provides centralized configuration for activity types and their channel mappings:

```typescript
// Activity type definitions
export const ACTIVITY_TYPE_CONFIG = {
  call: {
    channelTable: 'call_log',
    requiresReference: true,
    isCommunication: true,
    summaryTemplate: (data: CallLog) => 
      `Called ${data.phoneNumber} • ${data.durationSeconds}s • ${data.outcome}`
  },
  email: {
    channelTable: 'email_log',
    requiresReference: true,
    isCommunication: true,
    summaryTemplate: (data: EmailLog) => 
      `Email: ${data.subject}`
  },
  whatsapp: {
    channelTable: 'whatsapp',
    requiresReference: true,
    isCommunication: true,
    summaryTemplate: (data: Whatsapp) => 
      `WhatsApp: ${data.messageType === 'text' ? data.textBody : data.messageType}`
  },
  note: {
    channelTable: 'note',
    requiresReference: true,
    isCommunication: false,
    summaryTemplate: (data: Note) => 
      data.title || data.body.substring(0, 100)
  },
  task: {
    channelTable: 'task',
    requiresReference: true,
    isCommunication: false,
    summaryTemplate: (data: Task) => 
      `${data.title}${data.dueDate ? ` • Due ${data.dueDate}` : ''}`
  }
};

// Helper functions
export function isCommunicationActivity(type: ActivityType): boolean {
  return ACTIVITY_TYPE_CONFIG[type]?.isCommunication ?? false;
}

export function getChannelTableForType(type: ActivityType): string {
  return ACTIVITY_TYPE_CONFIG[type]?.channelTable;
}

export function generateActivitySummary(
  type: ActivityType, 
  channelData: any
): string {
  const template = ACTIVITY_TYPE_CONFIG[type]?.summaryTemplate;
  return template ? template(channelData) : '';
}
```

### Creating notes

<Steps>
<Step title="Create note content">
Create a `note` record with title, body, and attachments. This table stores content only with no entity foreign keys.

Create an `activity` record linking the note to its target entity:
```sql
activity:
├── target_entity_type: 'lead' | 'deal' | 'contact' | 'company'
```

## Tasks system

```sql
task
├── title (required)
├── description (optional text)
├── status (pending, in_progress, completed, cancelled)
├── priority (low, medium, high, urgent)
├── due_date (nullable)
├── assigned_to_id → User (who should complete the task)
├── completed_at, completed_by_id
├── attachments[] (JSONB array: {url, filename, size, mimeType})
├── organization_id, created_by_id, created_at
```

<Tabs>
<Tab title="Task workflow">
<Steps>
<Step title="Task creation">
Create task record with title, description, priority, and optional due date.
</Step>

<Step title="Assignment">
Assign task to specific user via `assigned_to_id`. User receives notification.
</Step>

<Step title="Progress tracking">
Update task status as work progresses: pending → in_progress → completed.
</Step>

<Step title="Completion">
Mark completed with `completed_at` timestamp and `completed_by_id` user reference.
</Step>
</Steps>
</Tab>

```typescript
  meeting: 10,
  site_visit: 15,
  note: 1,
  task: 2
};

// Calculate entity engagement score
async function calculateEngagementScore(
  entityType: EntityType,
  entityId: string,
  timeWindow: number = 30 // days
): Promise<number> {
  const activities = await activityRepo.find({
    targetEntityType: entityType,
    targetEntityId: entityId,
    createdAt: { $gte: new Date(Date.now() - timeWindow * 86400000) }
  });
  
  return activities.reduce((score, activity) => {
    return score + (ENGAGEMENT_WEIGHTS[activity.activityType] || 1);
  }, 0);
}
```

### Transaction boundaries

<AccordionGroup>
<Accordion title="Activity creation transactions">
**Atomic activity + channel creation**:
```typescript
// All activity creation must be atomic across activity + channel tables
await database.transaction(async (tx) => {
  // 1. Create channel record (call_log, email_log, etc.)
  const channelRecord = await tx.insert(channelTable, channelData);
  
  // 2. Create activity linking record
  const activity = await tx.insert('activity', {
    target_entity_type: entityType,
    target_entity_id: entityId,
    activity_type: type,
    reference_id: channelRecord.id,
    contact_id: resolvedContactId, // computed via bubble-up logic

  // 3. Update entity engagement scores

});
```
