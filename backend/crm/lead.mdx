The Lead entity represents a sales opportunity or inquiry from a person. One person can have multiple leads — each inquiry is tracked separately.

## Lead entity structure

```
Lead
├── person_id (required) → Person
├── stage_id → lead_stage (pipeline position)
├── intent (buyer | seller | renter | landlord | investor) — single enum
├── reference_id (external reference)
├── source_channel_id → PersonChannel (channel attribution)
├── companyName (optional string for display — NO FK)
├── lead_source (enum, nullable), source_detail (text)
├── score (lead scoring), temperature (cold | warm | hot)
├── budget, budget_currency (default: AED)
├── industry, timeline
├── Pipeline dates: converted_at, disqualified_at
├── Disqualification: disqualification_reason (enum), disqualification_notes
└── Notes: use polymorphic 'note' table
```

<Warning>
`intent` is a **single enum value** (buyer, seller, renter, landlord, investor), not an array. `companyName` is display-only with no foreign key to Company — company associations happen at the Contact level.
</Warning>

## Architecture context

The Lead entity is part of the CRM module's multi-layered architecture:

### Person + Contact + Lead model

<AccordionGroup>
  <Accordion title="Identity separation">
    - **Person**: Hidden identity layer (single source of truth for personal details)
    - **Contact**: Business relationship layer (qualified customers)
    - **Lead**: Sales opportunity layer (unqualified inquiries)
    - **Deal**: Links to Contact, not Person directly
  </Accordion>
  <Accordion title="Module boundaries">
    CRM Core contains Person, Lead, Contact, Company, Deal entities with their supporting tables. Real Estate and Marketing modules integrate by linking to Person (for identity) rather than Contact (for business relationships).
  </Accordion>
  <Accordion title="Channel separation">
    Activity table indexes timeline; channel-specific tables (WhatsApp, Instagram) store full communication data. Leads link to channels via `source_channel_id` for attribution.
  </Accordion>
</AccordionGroup>

### Real Estate integration

Real Estate entities link to **Person** (not Contact) for identity tracking:

| Real Estate Entity       | Links To    | Rationale                                   |
| ------------------------ | ----------- | ------------------------------------------- |
| `unit_ownership`         | `person_id` | Ownership is about identity, not CRM status |
| `unit_transaction`       | `person_id` | Transaction party is an individual          |
| `site_visit`             | `person_id` | Who visited the property                    |
| `lead_property_interest` | `lead_id`   | Links to Lead for sales context             |
| `deal_property_interest` | `deal_id`   | Links to Deal for transaction context       |

## Lead lifecycle and stages

### Stage progression

Leads progress through pipeline stages that track their qualification status:

<Steps>
  <Step title="New">
    Initial stage for all new leads. Automatically assigned when a lead is created without a custom `stageId`.
  </Step>
  <Step title="Contacted">
    First contact has been made with the lead. Stage changes are tracked in `lead_stage_history` with timing data.
  </Step>
  <Step title="Qualified">
    Lead has been validated as a genuine opportunity with proper budget and timeline.
  </Step>
  <Step title="Converted">
    Lead has been converted to a Contact and potentially a Deal. `converted_at` timestamp is set.
  </Step>
  <Step title="Disqualified">
    Lead is no longer viable. `disqualified_at`, `disqualification_reason`, and `disqualification_notes` are set.
  </Step>
</Steps>

### Stage history tracking

Every stage change is recorded for analytics and audit purposes:

```
lead_stage_history
├── lead_id → Lead
├── from_stage_id → LeadStage, to_stage_id → LeadStage
├── changed_at, changed_by → User
├── change_reason, notes
├── time_in_previous_stage (minutes)
└── organization_id → Organization
```

**Analytics query example:**
```sql
-- Average time spent in each lead stage
SELECT 
  ls.name as stage_name,
  AVG(lsh.time_in_previous_stage) / 60.0 as avg_hours_in_stage
FROM lead_stage_history lsh
JOIN lead_stage ls ON ls.id = lsh.from_stage_id
WHERE lsh.organization_id = ?
GROUP BY ls.id, ls.name;
```

## Assignment & Commission System

### Assignment via EntityStakeholder

Leads use the unified `entity_stakeholder` table for assignment and commission tracking:

```
entity_stakeholder (for leads)
├── entity_type = 'lead'
├── entity_id = lead.id
├── user_id → User (nullable for team-only assignments)
├── team_id → Team (nullable)
├── role_type (primary, secondary, observer, manager)
├── commission_percentage (0-100)
├── assignment_type (manual, auto, imported, transferred)
└── Status: is_active, assigned_at, unassigned_at
```

### Pool leads

<Tabs>
  <Tab title="Unassigned pool">
    Leads with no active stakeholders go to the global pool:
    - No `entity_stakeholder` records with `is_active = true`
    - Available for manual assignment by managers
    - Can be auto-assigned based on team rotation rules
  </Tab>
  <Tab title="Team pool">
    Leads assigned to a team but not a specific user:
    - `entity_stakeholder` record with `team_id` but `user_id = NULL`
    - Team members can claim these leads
    - Requires `team_crm.manage` permission to create
  </Tab>
</Tabs>

### Assignment authorization

When creating leads via API, assignment follows authorization rules:

| Assignment scenario | Required permission | Notes |
|-------------------|-------------------|-------|
| Assign to self | None (always allowed) | Most common case for normal agents |
| Pool lead (global) | `crm.manage` | No `assignedUserId`, no `teamId` |
| Pool lead (team) | `team_crm.manage` in target team | Only `teamId` provided |
| Assign to another user | `crm.manage` | `assignedUserId ≠ userId` |

### Commission tracking

The system tracks commission allocations across multiple stakeholders:

```
commission_payment
├── entity_type = 'lead'
├── entity_id = lead.id  
├── stakeholder_id → EntityStakeholder
├── payment_amount, payment_currency
├── commission_percentage
├── payment_status (pending, paid, cancelled)
├── payment_date, payment_method
└── notes
```

<Note>
Commission percentages across all stakeholders for a lead should sum to 100%. The system validates this during stakeholder creation and updates.
</Note>

## Transfer System

### Transfer ownership

Leads can be transferred between users/teams using the `entity_transfer` system:

```
entity_transfer (for leads)
├── entity_type = 'lead'
├── entity_id = lead.id
├── From: from_user_id → User, from_team_id → Team
├── To: to_user_id → User, to_team_id → Team
├── transfer_reason, transfer_notes
├── Status: status (pending, completed, rejected)
└── Audit: requested_by → User, approved_by → User
```

### Transfer workflow

<Steps>
  <Step title="Request transfer">
    User requests transfer via API. Creates pending `EntityTransfer` record.
  </Step>
  <Step title="Authorization check">
    System validates user has permission to transfer lead (owns it, or has management rights).
  </Step>
  <Step title="Approval process">
    Depending on organization settings, transfer may require manager approval.
  </Step>
  <Step title="Execute transfer">
    Update `EntityStakeholder` records, mark transfer as completed, create activity log.
  </Step>
</Steps>

### Commission preservation

<Note>
When leads are transferred, commission structures can be preserved or modified. The system supports commission splits across multiple stakeholders and tracks commission payments through the `commission_payment` table.
</Note>

## Activity & Communication System

### Polymorphic activity tracking

Leads use the unified activity system for communication logging:

```
activity
├── entity_type = 'lead'
├── entity_id = lead.id
├── activity_type (call, email, meeting, note, task)
├── activity_subtype, direction (inbound/outbound)
├── occurred_at, duration, title, description
├── user_id → User (who performed the activity)
└── channel_metadata (links to channel-specific tables)
```

### Channel-specific data

While the activity table provides timeline indexing, full communication data is stored in channel-specific tables:

<Tabs>
  <Tab title="WhatsApp">
    `whatsapp_message` table stores full message content, media attachments, and conversation context.
  </Tab>
  <Tab title="Email">
    `email_message` table stores email headers, body content, and thread relationships.
  </Tab>
  <Tab title="Phone calls">
    `phone_call` table stores call records, recordings, and transcriptions.
  </Tab>
</Tabs>

### PersonChannel integration

The `source_channel_id` field links leads to specific communication channels through the `person_channel` table:

```
person_channel
├── person_id → Person
├── channel_type (whatsapp, instagram, facebook, telegram, sms, webchat)
├── channel_identifier (phone number, username, PSID, etc.)
├── display_name, avatar_url
├── channel_identity_id → WhatsAppUser.id, InstagramUser.id, etc.
├── status (active, inactive, blocked, unsubscribed)
├── is_primary
├── Opt-in: marketing_opt_in, transactional_opt_in
├── Engagement: first_contact_at, last_message_at, message_count
└── Verification: is_verified, verified_at
```

**Key rules for PersonChannel**:
- Channel belongs to Person, not Lead (Person-centric architecture)
- Lead can reference `source_channel_id` for attribution (which channel it came through)
- `channel_identity_id` links to detailed channel entities (WhatsAppUser, InstagramUser)
- One Person can have multiple channels of same type (e.g., multiple WhatsApp numbers)

## Stage History & Analytics

### Lead lifecycle analytics

The system provides comprehensive analytics through stage history tracking:

```sql
-- Lead conversion funnel analysis
SELECT 
  ls.name as stage_name,
  COUNT(DISTINCT lsh.lead_id) as leads_entered,
  COUNT(DISTINCT CASE WHEN next_lsh.to_stage_id IS NOT NULL THEN lsh.lead_id END) as leads_progressed,
  AVG(lsh.time_in_previous_stage) / 60.0 as avg_hours_in_stage
FROM lead_stage_history lsh
JOIN lead_stage ls ON ls.id = lsh.to_stage_id
LEFT JOIN lead_stage_history next_lsh ON next_lsh.lead_id = lsh.lead_id 
  AND next_lsh.changed_at > lsh.changed_at
WHERE lsh.organization_id = ? AND lsh.changed_at >= ?
GROUP BY ls.id, ls.name
ORDER BY ls.sort_order;
```

## Lead creation and person resolution

### Person resolution logic

<Steps>
  <Step title="Person ID provided">
    If `personId` is provided in the request, the system uses that existing person record.
  </Step>
  <Step title="Person lookup">
    If no `personId`, system searches by email/phone via `PersonService.findOrCreatePerson`.
  </Step>
  <Step title="Person creation">
    If no person found, creates a new person (requires at least `firstName`).
  </Step>
  <Step title="Duplicate prevention">
    If both `referenceId` and `leadSource` are provided, checks for existing lead with same combination to prevent duplicate imports.
  </Step>
</Steps>

### Person merge system

<Info>
The CRM includes a comprehensive person merge system for handling duplicates. When persons are merged, all lead assignments, activities, and property interests are automatically reassigned to the primary person record.
</Info>

**API Endpoint**: `POST /persons/:primaryPersonId/merge`

**Merge workflow includes**:
- Field selection from multiple sources
- Contact info consolidation (emails, phones, addresses)
- Lead reassignment to primary person
- Activity history preservation
- Commission payment reallocation

### Creation business rules

<AccordionGroup>
  <Accordion title="Required fields">
    - Must link to a Person (existing or created)
    - At least `firstName` required if creating new person
    - `organization_id` inherited from request context
  </Accordion>
  <Accordion title="Default assignments">
    - New leads assigned to "New" stage by default
    - Custom `stageId` can override default stage
    - If `assignedUserId` provided, creates primary stakeholder with 100% commission
    - No assignment = lead goes to pool
  </Accordion>
  <Accordion title="Property interests">
    - Optional property interests can be attached at creation
    - At least one must be marked `isPrimary` if any are provided
    - If none marked primary, first one is auto-set as primary
  </Accordion>
</AccordionGroup>

## Query Patterns

### Get lead with full context

```sql
-- Get lead with person, stakeholders, and property interests
SELECT 
  l.*,
  p.first_name, p.last_name, p.full_name,
  pe.email, pe.is_primary as is_primary_email,
  pp.phone, pp.is_primary as is_primary_phone,
  pc.channel_type, pc.channel_identifier,
  es.user_id, es.commission_percentage, es.role_type,
  lpi.property_id, lpi.interest_type, lpi.is_primary
FROM lead l
JOIN person p ON l.person_id = p.id
LEFT JOIN person_email pe ON pe.person_id = p.id AND pe.is_primary = true
LEFT JOIN person_phone pp ON pp.person_id = p.id AND pp.is_primary = true
LEFT JOIN person_channel pc ON pc.id = l.source_channel_id
LEFT JOIN entity_stakeholder es ON es.entity_type = 'lead' AND es.entity_id = l.id AND es.is_active = true
LEFT JOIN lead_property_interest lpi ON lpi.lead_id = l.id
WHERE l.organization_id = ? AND l.id = ?;
```

### User's assigned leads

```sql
-- Get all leads assigned to a specific user
SELECT 
  l.*,
  p.first_name, p.last_name, p.full_name,
  ls.name as stage_name,
  es.commission_percentage, es.role_type,
  COUNT(a.id) as activity_count,
  MAX(a.occurred_at) as last_activity
FROM lead l
JOIN entity_stakeholder es ON es.entity_type = 'lead' AND es.entity_id = l.id
JOIN person p ON l.person_id = p.id
JOIN lead_stage ls ON l.stage_id = ls.id
LEFT JOIN activity a ON a.entity_type = 'lead' AND a.entity_id = l.id
WHERE es.user_id = ? AND es.is_active = true
GROUP BY l.id
ORDER BY l.created_at DESC;
```

### Team performance analytics

```sql
-- Lead conversion rates by team
SELECT 
  t.name as team_name,
  COUNT(l.id) as total_leads,
  COUNT(l.converted_at) as converted_leads,
  AVG(l.score) as avg_lead_score,
  AVG(EXTRACT(EPOCH FROM (l.converted_at - l.created_at))/86400) as avg_days_to_convert
FROM team t
JOIN entity_stakeholder es ON es.team_id = t.id AND es.entity_type = 'lead'
JOIN lead l ON l.id = es.entity_id
WHERE l.organization_id = ? AND l.created_at >= ?
GROUP BY t.id, t.name;
```

## Business Rules

### Multiple leads per person

<Note>
One person can have multiple leads — each inquiry is tracked separately. This allows for:
- Different property interests over time
- Separate pipeline tracking per inquiry
- Independent assignment and commission structures
- Distinct source attribution per lead
</Note>

### Lead intent classification

The `intent` field categorizes the lead's primary interest:

| Intent | Description | Common sources |
|--------|-------------|----------------|
| `buyer` | Looking to purchase property | Website inquiries, broker referrals |
| `seller` | Looking to sell property | Listing requests, cold calls |
| `renter` | Looking to rent property | Rental platforms, social media |
| `landlord` | Looking to rent out property | Investment seminars, referrals |
| `investor` | Looking for investment opportunities | Networking events, direct marketing |

### Lead scoring and temperature

<Tabs>
  <Tab title="Lead scoring">
    Numeric score (0-100) based on qualification criteria:
    - Budget qualification (25 points)
    - Timeline urgency (25 points)
    - Property match (25 points)
    - Engagement level (25 points)
  </Tab>
  <Tab title="Temperature classification">
    Qualitative assessment of lead readiness:
    - **Cold**: Initial inquiry, minimal engagement
    - **Warm**: Some qualification, active communication
    - **Hot**: Highly qualified, ready to move forward
  </Tab>
</Tabs>

### Budget and financial qualification

Leads can specify budget constraints for property search:

- `budget`: Numeric value for maximum budget
- `budget_currency`: Currency code (defaults to AED)
- Used for property matching and qualification scoring
- Helps prioritize leads by financial capacity

## Lead conversion process

### Conversion to Contact and Deal

When a lead converts, it transitions to the Contact/Deal pipeline:

<Steps>
  <Step title="Contact creation">
    Creates a Contact record (or unarchives existing) for the same person. Contact inherits person relationship but starts fresh pipeline.
  </Step>
  <Step title="Deal creation (optional)">
    If converting to a deal, creates Deal linked to the Contact. Deal can copy primary `LeadPropertyInterest` to `DealPropertyInterest`.
  </Step>
  <Step title="Lead status update">
    Sets `converted_at` timestamp and moves lead to "Converted" stage. Original lead remains for historical tracking.
  </Step>
  <Step title="Stakeholder preservation">
    Lead stakeholder assignments are typically copied to the new Contact/Deal with same commission percentages.
  </Step>
</Steps>

### Property interest handling

<Tabs>
  <Tab title="Lead to Deal conversion">
    ```javascript
    // Deal created FROM Lead:
    // Copy primary LeadPropertyInterest → DealPropertyInterest (1:1)
    deal.propertyInterest.originatingInterest = leadPropertyInterest
    ```
  </Tab>
  <Tab title="Direct deal creation">
    ```javascript
    // Deal created directly (walk-in):
    // Create DealPropertyInterest with no originating interest
    deal.propertyInterest.originatingInterest = null
    ```
  </Tab>
</Tabs>

## Source tracking and attribution

### Lead sources

Leads track their origin through multiple attribution fields:

| Field | Purpose | Values |
|-------|---------|---------|
| `lead_source` | Primary source category | website, referral, advertisement, cold_call, social_media, etc. |
| `source_detail` | Specific source details | Campaign name, referrer, ad creative, etc. |
| `source_channel_id` | Communication channel attribution | Links to `PersonChannel` (WhatsApp, Instagram, etc.) |
| `original_source` | Person-level first touch | Set once when person first enters system |

### Source analytics

```sql
-- Lead to Contact conversion rates by source
SELECT 
  lead_source,
  COUNT(*) as total_leads,
  COUNT(converted_at) as converted_leads,
  (COUNT(converted_at) * 100.0 / COUNT(*)) as conversion_rate
FROM lead 
WHERE organization_id = ? AND created_at >= ?
GROUP BY lead_source;
```

## Data Consistency Guarantees

### Transaction boundaries

<Warning>
The CRM maintains several data consistency rules:
- Person merge operations are transactional
- Commission percentages across stakeholders should sum to 100%
- Lead stage changes must follow valid transitions
- Entity transfers require proper authorization
</Warning>

### Validation rules

<AccordionGroup>
  <Accordion title="Commission validation">
    - Total commission percentages must equal 100% across active stakeholders
    - Commission payments cannot exceed assigned percentages
    - Stakeholder roles must be valid for entity type
  </Accordion>
  <Accordion title="Stage transition validation">
    - Stage changes must follow configured pipeline flows
    - Converted leads cannot move to other stages
    - Disqualified leads require reason and notes
  </Accordion>
  <Accordion title="Assignment validation">
    - Users must have appropriate permissions for assignment
    - Team assignments require team membership
    - Primary stakeholder role is required for each entity
  </Accordion>
</AccordionGroup>

### Data integrity checks

The system performs regular integrity checks to ensure:
- All leads have valid person references
- Stakeholder commission allocations are balanced
- Activity records reference valid entities
- Stage history maintains proper chronological order
