The CRM module provides a comprehensive customer relationship management system with Person-centric identity management, lead/contact/deal pipelines, and unified stakeholder assignment across all entities.

## Architecture overview

### Design principles

<Tabs>
  <Tab title="Person + Contact model">
    - **Person**: Hidden identity layer (single source of truth for personal details)
    - **Contact**: Business relationship layer (qualified customers)  
    - **Lead**: Sales opportunity layer (unqualified inquiries)
    - **Deal**: Links to Contact, not Person directly
  </Tab>
  <Tab title="Unified patterns">
    - **Unified Stakeholder Model**: Single table for assignment and commission across leads/deals
    - **Polymorphic Patterns**: Notes, tags, and activities use entity_type/entity_id patterns
    - **Channel Separation**: Activity table indexes timeline; channel tables store full data
    - **Modular Design**: CRM core is independent; Real Estate, Marketing, Channels are optional modules
    - **Company via Contact**: Companies associate with Contact via ContactCompanyRole (not Person)
  </Tab>
</Tabs>

### Real Estate → CRM integration

Real Estate entities link to **Person** (not Contact) for identity:

| Real Estate Entity       | Links To    | Rationale                                   |
| ------------------------ | ----------- | ------------------------------------------- |
| `unit_ownership`         | `person_id` | Ownership is about identity, not CRM status |
| `unit_transaction`       | `person_id` | Transaction party is an individual          |
| `site_visit`             | `person_id` | Who visited the property                    |
| `lead_property_interest` | `lead_id`   | Links to Lead for sales context             |
| `deal_property_interest` | `deal_id`   | Links to Deal for transaction context       |

**Deal Property Interest Workflow**:

```javascript
// Deal created FROM Lead:
// Copy primary LeadPropertyInterest → DealPropertyInterest (1:1)
deal.propertyInterest.originatingInterest = leadPropertyInterest

// Deal created directly (walk-in):
// Create DealPropertyInterest with no originating interest
deal.propertyInterest.originatingInterest = null
```

## Core entities

### Person (central identity)

**Purpose**: Single source of truth for human identity and preferences.

```
Person
├── Identity: first_name, last_name, avatar_url, title
├── Demographics: date_of_birth
├── Social: website, linkedin_url, twitter_url
├── Preferences: preferred_contact_method, timezone
├── Languages: languages (unified array with code and proficiency per entry)
├── Communication Flags: do_not_call, do_not_email
├── Source Tracking: original_source
├── Merge Tracking: merged_into_id, merged_at, merged_by
├── Computed: full_name (getter: first_name + last_name)
└── Related Tables:
    ├── person_email (multiple emails, one primary)
    ├── person_phone (multiple phones, one primary)
    ├── person_address (multiple addresses, one primary)
    ├── person_channel (WhatsApp, Instagram, etc. identities)
    └── person_not_duplicate (deduplication override pairs)
```

**Key rules:**
- Every Lead, Contact must link to a Person
- Person preferences apply across all contexts (leads, deals, contacts)
- `original_source` is set once when person first enters system
- Languages array uses unified `UserLanguageEntry` format with code and proficiency per entry

#### PersonChannel (communication channels)

**Purpose**: Stores a person's communication channel identities (WhatsApp, Instagram, etc.).

```
person_channel
├── person_id → Person
├── channel_type (whatsapp, instagram, facebook, telegram, sms, webchat)
├── channel_identifier (phone number, username, PSID, etc.)
├── display_name, avatar_url
├── channel_identity_id → WhatsAppUser.id, InstagramUser.id, etc.
├── status (active, inactive, blocked, unsubscribed)
├── is_primary
├── Opt-in: marketing_opt_in, transactional_opt_in
├── Engagement: first_contact_at, last_message_at, message_count
└── Verification: is_verified, verified_at
```

**Key rules:**
- Similar pattern to `person_email`, `person_phone`, `person_address`
- Channel belongs to Person, not Lead (Person-centric architecture)
- Lead can reference `source_channel_id` for attribution (which channel it came through)
- `channel_identity_id` links to detailed channel entities (WhatsAppUser, InstagramUser)
- One Person can have multiple channels of same type (e.g., multiple WhatsApp numbers)

<Warning>
Channel belongs to Person, not Lead (Person-centric architecture). Lead can reference `source_channel_id` for attribution only.
</Warning>

#### PersonNotDuplicate (deduplication overrides)

**Purpose**: Records pairs of persons that have been manually confirmed as NOT duplicates. Prevents the deduplication system from repeatedly flagging the same pair.

```
person_not_duplicate
├── person1_id → Person
├── person2_id → Person
├── marked_by → User (who made the decision)
├── marked_at (when the decision was made)
├── organization_id → Organization
├── Unique constraint: (person1, person2, organization)
```

**Key rules:**
- Symmetric: if (A, B) is marked as not-duplicate, the system treats (B, A) equivalently
- Organization-scoped: each org maintains its own override decisions
- Used by `PersonNotDuplicateService` to exclude pairs from duplicate detection

#### Person merge system

**Purpose**: Consolidates duplicate persons into a single primary record, reassigning all related data.

**API Endpoint**: `POST /persons/:primaryPersonId/merge`

<Steps>
  <Step title="Validation">
    Verify primary person exists and is not deleted. Verify all secondary persons exist and are not deleted. All persons must be in the same organization.
  </Step>
  <Step title="Field selection">
    Accept fieldSelections: `Record<string, string>` (e.g., `{ "firstName": "primary", "lastName": "person-B-id" }`). For each field, pick the value from the specified source person. Fields not listed default to primary person's values.
  </Step>
  <Step title="Contact info merge">
    `mergeAllEmails`, `mergeAllPhones`, `mergeAllAddresses`, `mergeAllChannels` (boolean flags) — reassign all secondary contact info to primary (isPrimary reset to false).
  </Step>
  <Step title="Related data reassignment">
    Update all related entities to point to primary person:
    
    - **CRM entities**: Update person_id in leads, contacts  
    - **Real Estate**: Update person_id in unit_ownership, site_visit, unit_transaction
    - **Communication**: Update person_id in activities, notes, person_channels
    - **Related records**: Update person_id in person_email, person_phone, person_address
    
    Mark secondary persons as deleted with `merged_into_id` pointing to primary. Create audit trail via merge_history record.
  </Step>
</Steps>

### Lead entity

A lead represents a sales opportunity or inquiry from a person. One person can have multiple leads — each inquiry is tracked separately.

```
Lead
├── person_id (required) → Person
├── stage_id → lead_stage (pipeline position)
├── intent (buyer | seller | renter | landlord | investor) — single enum
├── reference_id (external reference)
├── source_channel_id → PersonChannel (channel attribution)
├── companyName (optional string for display — NO FK)
├── lead_source (enum, nullable), source_detail (text)
├── score (lead scoring), temperature (cold | warm | hot)
├── budget, budget_currency (default: AED)
├── industry, timeline
├── Pipeline dates: converted_at, disqualified_at
├── Disqualification: disqualification_reason (enum), disqualification_notes
└── Notes: use polymorphic 'note' table
```

<Warning>
`intent` is a **single enum value** (buyer, seller, renter, landlord, investor), not an array. `companyName` is display-only with no foreign key to Company — company associations happen at the Contact level.
</Warning>

### Contact entity

**Purpose**: Represents a qualified business relationship with a person.

```
Contact
├── person_id (required) → Person
├── stage_id → contact_stage
├── contact_source, source_detail
├── qualification_score, contact_temperature
├── Business context: industry, company_size, annual_revenue
├── Relationship: relationship_status, acquisition_channel
├── Pipeline: qualified_at, archived_at, reactivated_at
└── Archive reason: archive_reason, archive_notes
```

### Company entity

**Purpose**: Business entities that can be associated with contacts.

```
Company
├── name, description
├── industry, company_size, annual_revenue
├── website, linkedin_url
├── company_type (client, prospect, partner, vendor)
├── status (active, inactive, archived)
└── Related via ContactCompanyRole (not direct to Person)
```

### Deal entity

**Purpose**: Represents active sales transactions.

```
Deal
├── contact_id (required) → Contact (not Person directly)
├── stage_id → deal_stage
├── deal_type, deal_source
├── Financial: value, currency, commission_rate, expected_close_date
├── Pipeline: closed_won_at, closed_lost_at
├── Close details: close_reason, close_notes
└── Property context: deal_property_interests[]
```

## Assignment & commission system

### EntityStakeholder (unified assignment)

**Purpose**: Single table managing assignment and commission splits across all CRM entities (leads, contacts, deals).

```
entity_stakeholder
├── entity_type (lead, contact, deal, etc.)
├── entity_id (polymorphic foreign key)
├── user_id → User (nullable for team-only assignments)
├── team_id → Team (nullable)
├── role_type (primary, secondary, observer, manager)
├── commission_percentage (0-100)
├── assignment_type (manual, auto, imported, transferred)
├── Status: is_active, assigned_at, unassigned_at
├── assigned_by → User, unassigned_by → User
└── organization_id → Organization
```

**Key rules:**
- One entity can have multiple stakeholders (commission splits)
- Total commission_percentage should sum to 100% for active stakeholders
- `user_id = NULL` means team-only assignment (team pool)
- `team_id = NULL` means individual assignment without team context

### Commission payment tracking

```
commission_payment
├── entity_stakeholder_id → EntityStakeholder
├── deal_id → Deal (for context)
├── payment_amount, currency
├── payment_date, payment_status
├── payment_reference, payment_notes
├── created_by → User, approved_by → User
└── organization_id → Organization
```

## Transfer system

### EntityTransfer (ownership changes)

**Purpose**: Tracks ownership transfers between users/teams with full audit trail.

```
entity_transfer
├── entity_type, entity_id (polymorphic)
├── From: from_user_id → User, from_team_id → Team
├── To: to_user_id → User, to_team_id → Team
├── Transfer: transfer_type (manual, automatic, bulk), transfer_reason
├── Status: status (pending, completed, rejected), completed_at
├── Audit: requested_by → User, approved_by → User, rejected_by → User
├── Notes: transfer_notes, rejection_reason
└── organization_id → Organization
```

**Transfer workflow:**
<Steps>
  <Step title="Request transfer">
    User requests transfer via API. Creates pending EntityTransfer record.
  </Step>
  <Step title="Authorization check">
    System validates user has permission to transfer entity (owns it, or has management rights).
  </Step>
  <Step title="Approval process">
    Depending on organization settings, transfer may require manager approval.
  </Step>
  <Step title="Execute transfer">
    Update EntityStakeholder records, mark transfer as completed, create activity log.
  </Step>
</Steps>

## Activity & communication system

### Activity (timeline index)

**Purpose**: Unified activity timeline across all CRM entities. Stores lightweight index records pointing to detailed data in channel-specific tables.

```
activity
├── entity_type, entity_id (polymorphic)
├── activity_type (call, email, meeting, note, whatsapp_message, etc.)
├── activity_subtype (inbound, outbound, missed, etc.)
├── Summary: subject, summary (brief description)
├── Timing: activity_date, duration_minutes
├── Status: status (scheduled, completed, cancelled), completion_status
├── People: created_by → User, assigned_to → User
├── Channel data: channel_message_id (FK to WhatsAppMessage, Email, etc.)
├── Classification: is_billable, is_important
└── organization_id → Organization
```

**Key rules:**
- Activity stores timeline summary; channel tables store full data
- `channel_message_id` links to detailed records (WhatsAppMessage, EmailMessage, etc.)
- One activity can relate to multiple entities via polymorphic pattern
- Activity serves as the single timeline view across all communication channels

### Analytics patterns

**Conversion funnel analysis:**
```sql
-- Lead to Contact conversion rates by source
SELECT 
  lead_source,
  COUNT(*) as total_leads,
  COUNT(converted_at) as converted_leads,
  (COUNT(converted_at) * 100.0 / COUNT(*)) as conversion_rate
FROM lead 
WHERE organization_id = ? AND created_at >= ?
GROUP BY lead_source;
```

**Time in stage analysis:**
```sql
-- Average time spent in each lead stage
SELECT 
  ls.name as stage_name,
  AVG(lsh.time_in_previous_stage) / 60.0 as avg_hours_in_stage
FROM lead_stage_history lsh
JOIN lead_stage ls ON ls.id = lsh.from_stage_id
WHERE lsh.organization_id = ?
GROUP BY ls.id, ls.name;
```

## Query patterns

### Entity retrieval with relationships

<Tabs>
  <Tab title="Lead with full context">
    ```sql
    -- Get lead with person, stakeholders, and property interests
    SELECT 
      l.*,
      p.first_name, p.last_name, p.email,
      es.user_id, es.commission_percentage,
      lpi.property_id, lpi.interest_type
    FROM lead l
    JOIN person p ON l.person_id = p.id
    LEFT JOIN entity_stakeholder es ON es.entity_type = 'lead' AND es.entity_id = l.id
    LEFT JOIN lead_property_interest lpi ON lpi.lead_id = l.id
    WHERE l.organization_id = ? AND l.id = ?;
    ```
  </Tab>
  <Tab title="Contact with companies">
    ```sql
    -- Get contact with associated companies
    SELECT 
      c.*,
      p.first_name, p.last_name,
      comp.name as company_name,
      ccr.role as company_role
    FROM contact c
    JOIN person p ON c.person_id = p.id
    LEFT JOIN contact_company_role ccr ON ccr.contact_id = c.id
    LEFT JOIN company comp ON ccr.company_id = comp.id
    WHERE c.organization_id = ? AND c.id = ?;
    ```
  </Tab>
  <Tab title="Person with all contexts">
    ```sql
    -- Get person with all CRM contexts (leads, contacts, deals)
    SELECT 
      p.*,
      l.id as lead_id, l.stage_id as lead_stage,
      c.id as contact_id, c.stage_id as contact_stage,
      d.id as deal_id, d.value as deal_value
    FROM person p
    LEFT JOIN lead l ON l.person_id = p.id AND l.archived_at IS NULL
    LEFT JOIN contact c ON c.person_id = p.id AND c.archived_at IS NULL
    LEFT JOIN deal d ON d.contact_id = c.id AND d.closed_lost_at IS NULL
    WHERE p.organization_id = ? AND p.id = ?;
    ```
  </Tab>
</Tabs>

### Assignment and commission queries

```sql
-- Get user's assigned entities across all types
SELECT 
  es.entity_type,
  es.entity_id,
  es.commission_percentage,
  CASE 
    WHEN es.entity_type = 'lead' THEN l.created_at
    WHEN es.entity_type = 'contact' THEN c.created_at
    WHEN es.entity_type = 'deal' THEN d.created_at
  END as entity_created_at
FROM entity_stakeholder es
LEFT JOIN lead l ON es.entity_type = 'lead' AND es.entity_id = l.id
LEFT JOIN contact c ON es.entity_type = 'contact' AND es.entity_id = c.id
LEFT JOIN deal d ON es.entity_type = 'deal' AND es.entity_id = d.id
WHERE es.user_id = ? AND es.is_active = true;
```

### Activity and communication queries

```sql
-- Get activity timeline for an entity
SELECT 
  a.*,
  u.first_name, u.last_name,
  CASE 
    WHEN a.activity_type = 'whatsapp_message' THEN wm.message_content
    WHEN a.activity_type = 'email' THEN em.subject
    WHEN a.activity_type = 'call' THEN cl.call_notes
  END as detailed_content
FROM activity a
JOIN user u ON a.created_by = u.id
LEFT JOIN whatsapp_message wm ON a.channel_message_id = wm.id AND a.activity_type = 'whatsapp_message'
LEFT JOIN email_message em ON a.channel_message_id = em.id AND a.activity_type = 'email'
LEFT JOIN call_log cl ON a.channel_message_id = cl.id AND a.activity_type = 'call'
WHERE a.entity_type = ? AND a.entity_id = ?
ORDER BY a.activity_date DESC;
```

## Business rules

### Lead business rules

<AccordionGroup>
  <Accordion title="Creation and person resolution">
    - If `personId` is provided, the system uses that existing person
    - Otherwise it searches by email/phone via `PersonService.findOrCreatePerson`
    - If no person found, creates a new person (requires at least `firstName`)
    - If both `referenceId` and `leadSource` are provided, checks for existing lead with same combination to prevent duplicate imports
  </Accordion>
  <Accordion title="Stage and assignment">
    - New leads are assigned to the "New" stage by default
    - Custom `stageId` can override default stage assignment
    - If `assignedUserId` is provided, a primary stakeholder is created with 100% commission
    - If no `assignedUserId`, the lead goes to the Pool (no stakeholders = unassigned)
  </Accordion>
  <Accordion title="Property interests">
    - Optional property interests can be attached at creation
    - At least one must be marked `isPrimary`
    - If none is marked primary, the first one is auto-set as primary
  </Accordion>
</AccordionGroup>

### Assignment authorization rules

When creating entities via API endpoints, server-side validation enforces assignment rules based on permissions:

| Assignment scenario       | Required permission                 | Notes                                           |
| ------------------------- | ----------------------------------- | ----------------------------------------------- |
| Assign to self            | None (always allowed)               | Most common case for normal agents              |
| Pool entity (global)      | `crm.manage`                        | No `assignedUserId`, no `teamId`                |
| Pool entity (team-specific) | `team_crm.manage` in target team  | Only `teamId` provided — entity goes to team pool |
| Assign to another user    | `crm.manage`                        | `assignedUserId ≠ userId`                       |
| Team selection            | User must be member OR `crm.manage` | Validates `teamId` if specified                 |

### Contact and deal rules

<AccordionGroup>
  <Accordion title="Contact creation from lead">
    - When lead converts, a Contact is created (or unarchived) for the person
    - Contact inherits person relationship but starts fresh pipeline
    - Original lead remains for historical tracking
  </Accordion>
  <Accordion title="Deal creation">
    - Deals link to Contact, not Person directly
    - Deal can be created from Contact or Lead
    - If created from Lead, Deal copies primary LeadPropertyInterest to DealPropertyInterest
    - Deal created directly (walk-in) creates DealPropertyInterest with no originating interest
  </Accordion>
  <Accordion title="Company associations">
    - Companies associate with Contact via ContactCompanyRole (not Person)
    - One contact can have multiple company roles
    - Company information flows to deals through contact relationship
  </Accordion>
</AccordionGroup>

### Pipeline and stage rules

| Rule                       | Description                                                             |
| -------------------------- | ----------------------------------------------------------------------- |
| Multiple entities per person | One person can have multiple leads, one active contact, multiple deals |
| Stage changes tracked      | Every stage change creates a stage_history record with timing data     |
| Time tracking             | System tracks time spent in each stage for analytics                   |
| Conversion preserves data  | Lead conversion to contact preserves all historical data and activities |

## Data consistency guarantees

### Transaction boundaries

Critical operations use database transactions to ensure consistency:

<AccordionGroup>
  <Accordion title="Person merge operations">
    - All related entity updates within single transaction
    - Rollback on any failure to prevent partial merges
    - Concurrent merge prevention via row-level locking
  </Accordion>
  <Accordion title="Lead conversion">
    - Contact creation/reactivation + lead status update atomic
    - Stakeholder assignments preserved across conversion
    - Property interest transfer includes originating lead reference
  </Accordion>
  <Accordion title="Commission calculations">
    - Stakeholder percentage validation before commit
    - Commission payment records linked to immutable deal state
    - Audit trail for all commission adjustments
  </Accordion>
</AccordionGroup>

### Data validation rules

| Entity level          | Validation                                         | Impact                                    |
| --------------------- | -------------------------------------------------- | ----------------------------------------- |
| Organization boundary | All entities must belong to single organization    | Data isolation and access control         |
| Person uniqueness     | Email/phone combination prevents true duplicates   | Deduplication at creation time            |
| Commission integrity  | Active stakeholder percentages sum to 100%        | Prevents commission calculation errors    |
| Stage progression     | Invalid stage transitions blocked by business rules | Pipeline integrity and reporting accuracy |
