## Architecture Overview

The CRM module follows a multi-layered architecture with clear separation of concerns and modular design principles.

### Design Principles

<AccordionGroup>
  <Accordion title="Person + Contact Model">
    - **Person** is the hidden identity layer (single source of truth for personal details)
    - **Contact** is the business relationship layer (qualified customers)
    - **Lead** is the sales opportunity layer (unqualified inquiries)
    - **Deal** links to Contact, not Person directly
  </Accordion>
  <Accordion title="Unified Stakeholder Model">
    Single table for assignment and commission across leads/deals using polymorphic patterns.
  </Accordion>
  <Accordion title="Polymorphic Patterns">
    Notes, tags, and activities use entity_type/entity_id patterns for flexible data association.
  </Accordion>
  <Accordion title="Channel Separation">
    Activity table indexes timeline; channel tables store full data. PersonChannel provides attribution.
  </Accordion>
  <Accordion title="Modular Design">
    CRM core is independent; Real Estate, Lead Capture, Channels are optional modules.
  </Accordion>
  <Accordion title="Company via Contact">
    Companies associate with Contact via ContactCompanyRole (not Person).
  </Accordion>
  <Accordion title="Organization Membership Display">
    CRM DTOs batch-resolve organization membership status for user references, showing removed org members with `isActiveOrgMember: false` badges while preserving historical assignment data.
  </Accordion>
</AccordionGroup>

### AI Module Integration

The CRM module integrates with the fully implemented and operational AI module (`AiModule`) for comprehensive automated conversations, lead processing, and workflow-assisted follow-up. CRM owns the business entities and lifecycle rules; the AI module owns agent configuration, runtime execution, queueing, LLM integration, security controls, and activity logging.

The AI module provides:

- **AI Agent Templates**: 10 pre-configured agent types (Receptionist, Sales Qualification, Listing Inquiry, Off-Plan Inquiry, Appointment Booking, After-Hours, FAQ & Support, Campaign Lead Capture, Spam Handler, Human Handoff) seeded via AiAgentTemplateSeeder
- **Knowledge Base Integration**: FAQ, SNIPPET, DOCUMENT, PAGE types with chunking, embedding, and RAG capabilities through KnowledgeBaseService
- **Credit Management**: AI-credit affordability/gating via the subscription module's unified-wallet `CreditMeteringService` (shared org pool with per-user ceilings)
- **Queue-based Execution**: Reliable agent processing via AiAgentExecuteWorker with retry logic and error handling
- **Tool Registry**: Extensible system through AiAgentToolRegistryService and AiAgentActionService
- **OpenAI Integration**: Project provisioning and encrypted key management through OpenAiProjectProvisioner and OpenAiEncryptionService
- **Activity Logging**: Comprehensive audit trail via AiActivityLogService with filtering and analytics
- **Workflow Integration**: Bidirectional integration - AI agents can trigger workflows through the `trigger_workflow` action AND workflows can activate AI agents via AI_AGENT steps
- **Media Processing**: Audio/image processing via AiAgentMediaProcessorService
- **Optimization**: Instruction optimization via AiAgentOptimizeService with protected token preservation
- **Security**: SSRF protection via assertNotSsrf utility and AES-GCM encryption utilities
- **Conversation automation**: AI agents can respond to messaging conversations, use CRM context, and hand off to users or teams when configured actions require human follow-up
- **Lead source tracking**: Leads include `sourceConversation` so CRM workflows, AI agent context resolution, and downstream automation can trace the originating messaging conversation
- **CRM tool execution**: AI actions can search and update CRM-owned data through the AI module's fixed tool catalog, including contact-field updates and lead lifecycle stage changes through CRM services

### Real Estate Integration

Real Estate entities link to the CRM layer that owns their business context:

| Real Estate Entity       | Links To     | Rationale                                           |
| ------------------------ | ------------ | --------------------------------------------------- |
| `unit_ownership`         | `contact_id` | Ownership is represented by the org's owner Contact |
| `unit_transaction`       | `person_id`  | Transaction party is an individual                  |
| `site_visit`             | `person_id`  | Who visited the property                            |
| `lead_property_interest` | `lead_id`    | Links to Lead for sales context                     |
| `deal_property_interest` | `deal_id`    | Links to Deal for transaction context               |

Lead property interests remain readable when a lead is archived. Archived leads are read-only, but the CRM lead detail/property-interest endpoint must bypass the Lead `active` filter for the parent lead check so historical property interest context stays visible. Soft-deleted leads remain excluded and `GET /leads/:id` must return `404`, never a `200` response with a `null` body, when the service cannot resolve a non-deleted lead.

## Core Entities

### CRM Public IDs

All core CRM business records expose a stored `publicId` for user-facing references. The format is:

```
{ENTITY_CODE}-{ORG_PREFIX}-{SEQUENCE}
```

Examples: `LEAD-ADNS-001`, `PERS-ADNS-001`, `CONT-ADNS-001`, `COMP-ADNS-001`, `DEAL-ADNS-001`, `COMM-ADNS-001`, `USER-ADNS-001`, `TEAM-ADNS-001`, `LSTG-ADNS-001`, `DSTG-ADNS-001`.

<Note>
Entity codes make IDs globally unique within an organization across CRM record types. `ORG_PREFIX` is generated once from the organization's normalized alphanumeric name. Names of 4 characters or fewer use the whole normalized name (`WIK`); longer names use the first 2 and last 2 characters (`Adidas Operations` → `ADNS`, `Adidas` → `ADAS`).
</Note>

**Public ID Rules:**
- `SEQUENCE` is scoped by organization and entity type, padded to at least 3 digits, and not capped (`LEAD-ADNS-1000` is valid)
- IDs are allocated by `CrmPublicIdService` from `crm_public_id_counter` in the same transaction as entity creation
- Existing IDs do not change when the organization is renamed
- Entity codes: Lead=`LEAD`, Person=`PERS`, Contact=`CONT`, Company=`COMP`, Deal=`DEAL`, CommissionPayment=`COMM`, organization user membership=`USER`, Team=`TEAM`, LeadStage=`LSTG`, DealStage=`DSTG`
- Organization user memberships also receive stable fallback avatar colors from the user sequence (`organization_users.avatar_bg_color` + `avatar_text_color`). The hue uses a golden-angle spread across a finite HSL palette; colors are display metadata and may repeat in very large organizations, while `public_id` remains the unique user-facing membership identifier
- Global system stages have deterministic IDs based on `systemType` (e.g., `LSTG-NEW`, `LSTG-DISQUALIFIED`, `DSTG-CLOSED-WON`). Org-specific custom/override stages use the normal org-scoped sequence (`LSTG-ADNS-001`, `DSTG-ADNS-001`)

### Person (Central Identity)

**Purpose**: Single source of truth for human identity and preferences.

```
Person
├── public_id (coded org-scoped human ID, e.g. PERS-ADNS-001)
├── Identity: first_name, last_name, avatar_url, title
├── Demographics: date_of_birth, nationality
├── Links: links jsonb [{ type, url }] where type is website/facebook/instagram/linkedIn/twitter/tiktok
├── Preferences: timezone
├── Languages: languages (unified array with code and proficiency per entry)
├── Communication Flags: do_not_call, do_not_email
├── Source Tracking: original_source
├── Merge Tracking: merged_into_id, merged_at, merged_by
├── Computed: full_name (getter: first_name + last_name)
└── Related Tables:
    ├── person_email (multiple emails, one primary)
    ├── person_phone (multiple phones, one primary)
    ├── person_address (multiple addresses, one primary)
    ├── person_channel (WhatsApp, Instagram, etc. identities)
    └── person_not_duplicate (deduplication override pairs)
```

**Key Rules**:
- Every Lead, Contact must link to a Person
- `publicId` is generated once at creation using the shared CRM public ID allocator (`PERS-*`)
- Person preferences apply across all contexts (leads, deals, contacts)
- `nationality` is stored on Person so leads and contacts share the same identity profile value
- `original_source` is set once when person first enters system
- Languages array uses unified `UserLanguageEntry` format with code and proficiency per entry
- `links` stores web and social profile links as JSONB entries with `type` and `url`

### Lead Entity

**Purpose**: Sales opportunity layer for unqualified inquiries.

```
Lead
├── public_id (coded org-scoped lead ID, e.g. LEAD-ADNS-001)
├── person_id (required) → Person
├── stage_id → lead_stage (pipeline position)
├── intent (buyer | seller | renter | landlord | investor) — single enum
├── reference_id (external reference)
├── source_channel_id → PersonChannel (channel attribution)
├── companyName (optional string for display — NO FK)
├── lead_source (enum, nullable), source_detail (text)
├── score (lead scoring), temperature (cold | warm | hot)
├── budget, budget_currency (default: AED)
├── industry, timeline
├── Pipeline dates: converted_at, disqualified_at
├── Disqualification: disqualification_reason (enum), disqualification_notes
└── Notes: use polymorphic 'note' table
```

**Key Rules**:
- `intent` is a **single enum value** (buyer, seller, renter, landlord, investor), not an array
- `companyName` is display-only with no foreign key to Company — company associations happen at the Contact level
- One person can have multiple leads — each inquiry is tracked separately
- Lead list, kanban, and detail DTOs populate `person.channels` for messaging context

### Contact Entity

**Purpose**: Business relationship layer for qualified customers.

```
Contact
├── public_id (coded org-scoped contact ID, e.g. CONT-ADNS-001)
├── person_id → Person
├── contact_type (individual, business)
├── qualification_status, qualification_notes
├── customer_status (prospect, customer, former_customer)
├── relationship_manager_id → User
├── credit_limit, payment_terms
├── preferred_language
├── business_context, relationship_notes
├── lifecycle_stage_id → contact_stage
└── Status: is_active, archived_at, archived_by → User
```

### Company Entity

**Purpose**: Business entities that contacts can be associated with.

```
Company
├── public_id (coded org-scoped company ID, e.g. COMP-ADNS-001)
├── name, trade_name, logo_url
├── company_type, industry, company_size
├── registration_number, tax_id, license_number
├── website, description
├── financial_info: annual_revenue, credit_rating
├── addresses, phones, emails (via company_address, company_phone, company_email)
└── Status: is_active, archived_at
```

### Deal Entity

**Purpose**: Sales opportunities linked to contacts.

```
Deal
├── public_id (coded org-scoped deal ID, e.g. DEAL-ADNS-001)
├── contact_id → Contact (required)
├── deal_name, deal_value, deal_currency
├── expected_close_date, actual_close_date
├── probability (0-100)
├── stage_id → deal_stage
├── lost_reason, won_details
├── deal_source, source_detail
└── Status: status (open, won, lost)
```

### PersonChannel (Communication Channels)

**Purpose**: Stores a person's communication channel identities (WhatsApp, Instagram, etc.).

```
person_channel
├── person_id → Person
├── channel_type (whatsapp, instagram, facebook, telegram, sms, webchat)
├── channel_identifier (phone number, username, PSID, etc.)
├── display_name, avatar_url
├── channel_identity_id → WhatsAppUser.id, InstagramUser.id, etc.
├── status (active, inactive, blocked, unsubscribed)
├── is_primary
├── Opt-in: marketing_opt_in, transactional_opt_in
├── Engagement: first_contact_at, last_message_at, message_count
└── Verification: is_verified, verified_at
```

**Key Rules**:
- Similar pattern to `person_email`, `person_phone`, `person_address`
- Channel belongs to Person, not Lead (Person-centric architecture)
- Lead can reference `source_channel_id` for attribution (which channel it came through)
- `channel_identity_id` links to detailed channel entities (WhatsAppUser, InstagramUser)
- One Person can have multiple channels of same type (e.g., multiple WhatsApp numbers)
- Lead list, kanban, and detail DTOs populate `person.channels` so CRM surfaces can show or deep-link messaging actions without making a separate person-channel request

### PersonNotDuplicate (Deduplication Overrides)

**Purpose**: Records pairs of persons that have been manually confirmed as NOT duplicates. Prevents the deduplication system from repeatedly flagging the same pair.

```
person_not_duplicate
├── person1_id → Person
├── person2_id → Person
├── marked_by → User (who made the decision)
├── marked_at (when the decision was made)
├── organization_id → Organization
├── Unique constraint: (person1, person2, organization)
```

**Key Rules**:
- Symmetric: if (A, B) is marked as not-duplicate, the system treats (B, A) equivalently
- Organization-scoped: each org maintains its own override decisions
- Used by `PersonNotDuplicateService` to exclude pairs from duplicate detection

### Person Merge System

**Purpose**: Consolidates duplicate persons into a single primary record, reassigning all related data.

**API Endpoint**: `POST /persons/:primaryPersonId/merge`

**Merge Workflow**:

<Steps>
  <Step title="Validation">
    Verify primary person exists and is not deleted. Verify all secondary persons exist and are not deleted. All persons must be in the same organization.
  </Step>
  <Step title="Field selection">
    Accept fieldSelections: Record<string, string> (e.g., { "firstName": "primary", "lastName": "person-B-id" }). For each field, pick the value from the specified source person. Fields not listed default to primary person.
  </Step>
  <Step title="Data reassignment">
    Reassign all lead assignments, activities, and property interests to the primary person record.
  </Step>
  <Step title="Activity preservation">
    Maintain activity history and commission payment records through the merge process.
  </Step>
</Steps>

## Assignment & Commission System

### Assignment via EntityStakeholder

Leads use the unified `entity_stakeholder` table for assignment and commission tracking:

```
entity_stakeholder (for leads)
├── entity_type = 'lead'
├── entity_id = lead.id
├── user_id → User (nullable for team-only assignments)
├── team_id → Team (nullable)
├── role_type (primary, secondary, observer, manager)
├── commission_percentage (0-100)
├── assignment_type (manual, auto, imported, transferred)
└── Status: is_active, assigned_at, unassigned_at
```

### Pool leads

<Tabs>
  <Tab title="Unassigned pool">
    Leads with no active stakeholders go to the global pool:
    - No `entity_stakeholder` records with `is_active = true`
    - Available for manual assignment by managers
    - Can be auto-assigned based on team rotation rules
  </Tab>
  <Tab title="Team pool">
    Leads assigned to a team but not a specific user:
    - `entity_stakeholder` record with `team_id` but `user_id = NULL`
    - Team members can claim these leads
    - Requires `team_crm.manage` permission to create
  </Tab>
</Tabs>

### Assignment authorization

When creating leads via API, assignment follows authorization rules:

| Assignment scenario | Required permission | Notes |
|-------------------|-------------------|-------|
| Assign to self | None (always allowed) | Most common case for normal agents |
| Pool lead (global) | `crm.manage` | No `assignedUserId`, no `teamId` |
| Pool lead (team) | `team_crm.manage` in target team | Only `teamId` provided |
| Assign to another user | `crm.manage` | `assignedUserId ≠ userId` |

### Commission tracking

The system tracks commission allocations across multiple stakeholders:

```
commission_payment
├── public_id (coded org-scoped commission ID, e.g. COMM-ADNS-001)
├── entity_type = 'lead'
├── entity_id = lead.id  
├── stakeholder_id → EntityStakeholder
├── payment_amount, payment_currency
├── commission_percentage
├── payment_status (pending, paid, cancelled)
├── payment_date, payment_method
└── notes
```

<Note>
Commission percentages across all stakeholders for a lead should sum to 100%. The system validates this during stakeholder creation and updates.
</Note>

## Transfer System

### Transfer ownership

Leads can be transferred between users/teams using the `entity_transfer` system:

```
entity_transfer (for leads)
├── entity_type = 'lead'
├── entity_id = lead.id
├── From: from_user_id → User, from_team_id → Team
├── To: to_user_id → User, to_team_id → Team
├── transfer_reason, transfer_notes
├── Status: status (pending, completed, rejected)
└── Audit: requested_by → User, approved_by → User
```

### Transfer workflow

<Steps>
  <Step title="Request transfer">
    User requests transfer via API. Creates pending `EntityTransfer` record.
  </Step>
  <Step title="Authorization check">
    System validates user has permission to transfer lead (owns it, or has management rights).
  </Step>
  <Step title="Approval process">
    Depending on organization settings, transfer may require manager approval.
  </Step>
  <Step title="Execute transfer">
    Update `EntityStakeholder` records, mark transfer as completed, create activity log.
  </Step>
</Steps>

### Commission preservation

<Note>
When leads are transferred, commission structures can be preserved or modified. The system supports commission splits across multiple stakeholders and tracks commission payments through the `commission_payment` table.
</Note>

**Analytics query example:**
```sql
-- Average time spent in each lead stage
SELECT 
  ls.name as stage_name,
  AVG(lsh.time_in_previous_stage) / 60.0 as avg_hours_in_stage
FROM lead_stage_history lsh
JOIN lead_stage ls ON ls.id = lsh.from_stage_id
WHERE lsh.organization_id = ?
GROUP BY ls.id, ls.name;
```

## Query Patterns

### Get lead with full context

```sql
-- Get lead with person, stakeholders, and property interests
SELECT 
  l.*,
  p.first_name, p.last_name, p.full_name,
  pe.email, pe.is_primary as is_primary_email,
  pp.phone, pp.is_primary as is_primary_phone,
  pc.channel_type, pc.channel_identifier,
  es.user_id, es.commission_percentage, es.role_type,
  lpi.property_id, lpi.interest_type, lpi.is_primary
FROM lead l
JOIN person p ON l.person_id = p.id
LEFT JOIN person_email pe ON pe.person_id = p.id AND pe.is_primary = true
LEFT JOIN person_phone pp ON pp.person_id = p.id AND pp.is_primary = true
LEFT JOIN person_channel pc ON pc.id = l.source_channel_id
LEFT JOIN entity_stakeholder es ON es.entity_type = 'lead' AND es.entity_id = l.id AND es.is_active = true
LEFT JOIN lead_property_interest lpi ON lpi.lead_id = l.id
WHERE l.organization_id = ? AND l.id = ?;
```

### User's assigned leads

```sql
-- Get all leads assigned to a specific user
SELECT 
  l.*,
  p.first_name, p.last_name, p.full_name,
  ls.name as stage_name,
  es.commission_percentage, es.role_type,
  COUNT(a.id) as activity_count,
  MAX(a.occurred_at) as last_activity
FROM lead l
JOIN entity_stakeholder es ON es.entity_type = 'lead' AND es.entity_id = l.id
JOIN person p ON l.person_id = p.id
JOIN lead_stage ls ON l.stage_id = ls.id
LEFT JOIN activity a ON a.entity_type = 'lead' AND a.entity_id = l.id
WHERE es.user_id = ? AND es.is_active = true
GROUP BY l.id
ORDER BY l.created_at DESC;
```

### Team performance analytics

```sql
-- Lead conversion rates by team
SELECT 
  t.name as team_name,
  COUNT(l.id) as total_leads,
  COUNT(l.converted_at) as converted_leads,
  AVG(l.score) as avg_lead_score,
  AVG(EXTRACT(EPOCH FROM (l.converted_at - l.created_at))/86400) as avg_days_to_convert
FROM team t
JOIN entity_stakeholder es ON es.team_id = t.id AND es.entity_type = 'lead'
JOIN lead l ON l.id = es.entity_id
WHERE l.organization_id = ? AND l.created_at >= ?
GROUP BY t.id, t.name;
```

### Source attribution analysis

```sql
-- Lead quality by source channel
SELECT 
  pc.channel_type,
  COUNT(l.id) as total_leads,
  AVG(l.score) as avg_score,
  COUNT(l.converted_at) / COUNT(l.id)::float as conversion_rate
FROM lead l
LEFT JOIN person_channel pc ON l.source_channel_id = pc.id
WHERE l.organization_id = ? AND l.created_at >= ?
GROUP BY pc.channel_type
ORDER BY conversion_rate DESC;
```

### Lead property interest queries

```sql
-- Leads interested in specific property types
SELECT 
  l.id, l.public_id,
  p.first_name, p.last_name,
  lpi.interest_type, lpi.max_price, lpi.preferred_move_date
FROM lead l
JOIN person p ON l.person_id = p.id
JOIN lead_property_interest lpi ON lpi.lead_id = l.id
WHERE lpi.property_type = ? 
  AND lpi.max_price >= ?
  AND l.organization_id = ?
ORDER BY lpi.created_at DESC;
```

### Team workload distribution

```sql
-- Team workload by lead count and value
SELECT 
  u.first_name, u.last_name,
  COUNT(l.id) as lead_count,
  AVG(l.score) as avg_score,
  COUNT(CASE WHEN l.temperature = 'hot' THEN 1 END) as hot_leads
FROM organization_user u
LEFT JOIN entity_stakeholder es ON es.user_id = u.user_id AND es.entity_type = 'lead' AND es.is_active = true
LEFT JOIN lead l ON l.id = es.entity_id
WHERE u.organization_id = ?
GROUP BY u.id, u.first_name, u.last_name
ORDER BY lead_count DESC;
```

## Business Rules

### Multiple leads per person

<Note>
One person can have multiple leads — each inquiry is tracked separately. This allows for:
- Different property interests over time
- Separate pipeline tracking per inquiry
- Independent assignment and commission structures
- Distinct source attribution per lead
</Note>

### Lead intent classification

The `intent` field categorizes the lead's primary interest:

| Intent | Description | Common sources |
|--------|-------------|----------------|
| `buyer` | Looking to purchase property | Website inquiries, broker referrals |
| `seller` | Looking to sell property | Listing requests, cold calls |
| `renter` | Looking to rent property | Rental platforms, social media |
| `landlord` | Looking to rent out property | Investment seminars, referrals |
| `investor` | Looking for investment opportunities | Networking events, direct marketing |

### Lead scoring and temperature

<Tabs>
  <Tab title="Lead scoring">
    Numeric score (0-100) based on qualification criteria:
    - Budget qualification (25 points)
    - Timeline urgency (25 points)
    - Property match (25 points)
    - Engagement level (25 points)
  </Tab>
  <Tab title="Temperature classification">
    Qualitative assessment of lead readiness:
    - **Cold**: Initial inquiry, minimal engagement
    - **Warm**: Some qualification, active communication
    - **Hot**: Highly qualified, ready to move forward
  </Tab>
</Tabs>

### Commission validation rules

<AccordionGroup>
  <Accordion title="Commission allocation">
    - Total commission percentages must equal 100% across active stakeholders
    - Commission payments cannot exceed assigned percentages
    - Stakeholder roles must be valid for entity type
  </Accordion>
  <Accordion title="Stage transition validation">
    - Stage changes must follow configured pipeline flows
    - Converted leads cannot move to other stages
    - Disqualified leads require reason and notes
  </Accordion>
  <Accordion title="Assignment validation">
    - Users must have appropriate permissions for assignment
    - Team assignments require team membership
    - Primary stakeholder role is required for each entity
  </Accordion>
</AccordionGroup>

### Lead qualification criteria

<AccordionGroup>
  <Accordion title="Budget qualification">
    Leads must provide budget range or maximum price for property interests. Budget validation includes currency normalization and local market benchmarks.
  </Accordion>
  <Accordion title="Timeline validation">
    Timeline must be realistic for the property type and market conditions. System tracks timeline accuracy for scoring adjustments.
  </Accordion>
  <Accordion title="Contact information">
    Valid email or phone contact method required. Channel preferences tracked for communication optimization.
  </Accordion>
  <Accordion title="Property interest specificity">
    Generic interests receive lower scores. Specific location, property type, and feature requirements increase qualification scores.
  </Accordion>
</AccordionGroup>

### Data integrity rules

<Warning>
The system enforces several critical data integrity constraints:
- Every lead must have a valid person_id
- Commission percentages across stakeholders cannot exceed 100%
- Stage transitions must follow valid pipeline progressions
- Entity transfers require proper authorization
- Merged persons cannot be referenced by new records
- Lead property interests must have valid property references
</Warning>

### Lead lifecycle management

<AccordionGroup>
  <Accordion title="Lead creation rules">
    - Person must exist or be created atomically with lead
    - Source channel attribution is set once at creation
    - Initial stage assignment follows organization pipeline rules
    - Assignment respects user permissions and team membership
  </Accordion>
  <Accordion title="Lead conversion rules">
    - Converted leads become read-only (cannot change stages)
    - Property interests remain accessible after conversion
    - Assignment history is preserved through conversion
    - Commission structures transfer to resulting deal/contact
  </Accordion>
  <Accordion title="Lead archival rules">
    - Disqualified leads are archived, not deleted
    - Archived leads maintain all relationships and history
    - Property interests remain readable for historical context
    - Activities and notes are preserved indefinitely
  </Accordion>
</AccordionGroup>

## Data Consistency Guarantees

### Transaction boundaries

<Warning>
The CRM maintains several data consistency rules:
- Person merge operations are transactional
- Commission percentages across stakeholders should sum to 100%
- Lead stage changes must follow valid transitions
- Entity transfers require proper authorization
- Property interest creation validates property references
</Warning>

### Referential integrity

The system maintains referential integrity across related entities:

<AccordionGroup>
  <Accordion title="Person relationships">
    - Leads cannot exist without a valid person_id
    - Person deletion cascades to related leads (soft delete)
    - Person merges update all lead references atomically
    - Communication preferences flow from person to all leads
  </Accordion>
  <Accordion title="Stakeholder consistency">
    - Entity stakeholders must reference valid entities and users
    - Commission allocations are validated on creation/update
    - Inactive stakeholders don't receive new assignments
    - Team membership is validated for team assignments
  </Accordion>
  <Accordion title="Stage progression">
    - Stage changes follow configured pipeline rules
    - Historical stage data is preserved immutably
    - Analytics depend on accurate stage transition tracking
    - Converted leads cannot regress to earlier stages
  </Accordion>
  <Accordion title="Property interest integrity">
    - Property references are validated at creation
    - Interest types must align with lead intent
    - Price ranges are validated against market data
    - Timeline preferences are checked for reasonableness
  </Accordion>
</AccordionGroup>

### Audit trails

All significant lead operations maintain comprehensive audit trails:

```
audit_log
├── entity_type = 'lead'
├── entity_id = lead.id
├── operation_type (create, update, transfer, convert, assign)
├── field_changes (JSONB of before/after values)
├── user_id → User (who performed the operation)
├── timestamp, session_id, ip_address
├── business_context (reason, notes, approval_chain)
└── integration_source (api, webapp, mobile, import, automation)
```

**Audit coverage includes:**
- Assignment changes with full stakeholder history
- Stage progressions with reasons and timing
- Transfer requests and approval workflows
- Commission payment records and adjustments
- Activity and communication logs
- Property interest modifications
- Lead qualification scoring changes

### Data archival and retention

<AccordionGroup>
  <Accordion title="Lead archival policy">
    - Disqualified leads are archived after 90 days of inactivity
    - Converted leads maintain full history indefinitely
    - Archived leads preserve all relationships and audit trails
    - Property interests remain accessible for reporting
  </Accordion>
  <Accordion title="Personal data retention">
    - Person data follows regional privacy regulations (GDPR, CCPA)
    - Communication opt-out requests are honored immediately
    - Data anonymization for analytics while preserving business intelligence
    - Right to erasure balanced with business record requirements
  </Accordion>
  <Accordion title="Performance optimization">
    - Active leads indexed separately from archived leads
    - Historical analytics pre-computed for performance
    - Audit logs archived to separate storage after 2 years
    - Search indexes optimized for current lead operations
  </Accordion>
</AccordionGroup>
