Authoritative description of the CRM data-export endpoints (`GET /{entity}/export`) for **Lead**, **Deal**, **Contact**, and **Company**: what they return, who is allowed to call them, and exactly how the result set is scoped.

<Note>
**Scope of this documentation:** This covers user-facing CSV/XLSX exports of CRM entities. It does **not** cover import functionality, audit-log exports, or analytics report downloads.
</Note>

## Endpoints overview

| Entity  | Endpoint                   | Service method                   | Scoping model        |
| ------- | -------------------------- | -------------------------------- | -------------------- |
| Lead    | `GET /v1/leads/export`     | `LeadService.exportLeads`        | **Accessible union** |
| Deal    | `GET /v1/deals/export`     | `DealService.exportDeals`        | **Accessible union** |
| Contact | `GET /v1/contacts/export`  | `ContactService.exportContacts`  | **Accessible union** |
| Company | `GET /v1/companies/export` | `CompanyService.exportCompanies` | **Accessible union** |

<Info>
All four exports use the same **permission-free accessible-union model**: no `crm.view`, `sales.view`, or `team_crm.view` gate at the export endpoint, only the **data-portability policy** and the caller's own per-entity visibility.
</Info>

All four exports share:
- The same two hard gates
- The same RLS backstop
- The same filter/`selectedIds` narrowing semantics
- The same file-format rules

## Two-layer access model

Every export answer is the intersection of two independent questions:

<Steps>
<Step title="Data portability policy check">
**"Are you allowed to export at all?"** â†’ the **data-portability policy** gate (`assertDataPortabilityPolicy`). This is an org-level governance switch, not a CRM-visibility check.
</Step>

<Step title="Row-level scoping">
**"Which rows are you allowed to see?"** â†’ the **row-scoping** model, which differs per entity but follows the accessible union pattern.
</Step>
</Steps>

<Warning>
These layers are orthogonal. Passing the portability policy never widens row-level access, and vice versa. A user who clears the portability policy still only ever receives rows their CRM visibility allows.
</Warning>

## Data portability policy (Gate 1)

Enforced in the controller **before** the service runs:

```typescript
await this.accessService.assertDataPortabilityPolicy(user, 'export');
```

### Policy enforcement logic

<Steps>
<Step title="System bypass check">
System admin / system owner (`hasSystemBypass`) â†’ always allowed, short-circuit.
</Step>

<Step title="Policy resolution">
Resolve the org's `settings.dataExportPolicy`, defaulting to `DEFAULT_DATA_PORTABILITY_POLICY` when unset.
</Step>

<Step title="Permission validation">
- If policy is `'all'` â†’ **any authenticated member** of the org may export
- If policy is `'admins_only'` â†’ caller **must** hold `OrgPermissionKey.SYSTEM_ADMIN`, otherwise `403 ForbiddenException`
</Step>
</Steps>

### Policy values

| `dataExportPolicy`          | Who can export                                                 |
| --------------------------- | -------------------------------------------------------------- |
| `all`                       | Any authenticated org member (subject to row-scoping) |
| `admins_only` **(default)** | Only Admin / Owner (`system.admin`) + system bypass            |

<Warning>
**Important default:** `DEFAULT_DATA_PORTABILITY_POLICY = 'admins_only'`. Out of the box, **regular agents cannot export anything** until an Owner/Admin changes the org's export policy to `all`.
</Warning>

## Lead row scoping (Gate 2)

The lead export has **no `crm.view` gate** and **never calls `validateTypeAccess`**. After the portability gate, rows are scoped by the caller's permissions and memberships.

### Lead export scopes

| `type` value                                               | Rows returned                                                                  | Filename prefix |
| ---------------------------------------------------------- | ------------------------------------------------------------------------------ | --------------- |
| `my`                                                       | Only leads the user **created** or is a **direct stakeholder** on | `leads-my-`     |
| anything else â€” omitted, `all`, `pool`, `team`, `teamPool` | **Accessible union** of every tab the user can see                 | `leads-all-`    |

### Accessible union construction

```typescript
// type !== MY â†’ resolve the access predicate first (async), then build QB synchronously
const accessFragment =
  type !== LeadViewType.MY ? await this.resolveAccessibleLeadsFragment(em, user) : null;

if (type === LeadViewType.MY) {
  qb = this.buildLeadQueryForType(em, user, LeadViewType.MY); // created_by OR stakeholder
} else {
  qb = this.buildLeadQueryForType(em, user, LeadViewType.ALL); // base query, no access filter
  if (accessFragment) {
    qb.andWhere(accessFragment.sql, accessFragment.params); // OR-combined predicate
  }
}
```

### Fragment resolution logic

`resolveAccessibleLeadsFragment(em, user)`:

- **Returns `null`** when the user has org-wide CRM view (`crm.view` â†’ `canViewAll`). A `null` fragment means **no access predicate is added**, so the user sees every (non-deleted) lead in the org.
- **Otherwise** it ORs together fragments for each accessible tab:

<AccordionGroup>
<Accordion title="MY tab fragment">
Always included: `myLeadsFragment` - `created_by_id = user` OR a direct `entity_stakeholder` row for the user
</Accordion>

<Accordion title="POOL tab fragment">
Always included: `poolLeadsFragment` - `NOT EXISTS` any stakeholder row â†’ unassigned leads
</Accordion>

<Accordion title="TEAM tab fragment">
One `teamLeadsFragment` per team from `getAvailableTeams(em, user, orgId, false)` (teams where user holds `team_crm.view`), **only if** the `team` tab is available
</Accordion>

<Accordion title="TEAM_POOL tab fragment">
One `teamPoolLeadsFragment` per team from `getAllMemberTeams(em, user.id, false)` (teams the user is a member of), **only if** the `teamPool` tab is available
</Accordion>
</AccordionGroup>

### Tab availability rules

| Tab        | Available when                                |
| ---------- | --------------------------------------------- |
| `all`      | `crm.view` (`canViewAll`)                     |
| `my`       | always                                        |
| `pool`     | always â€” **anyone can see unassigned leads**  |
| `team`     | `crm.view` **or** `team_crm.view`             |
| `teamPool` | `crm.view` **or** the user belongs to â‰Ą1 team |

<Note>
**Product note on POOL:** Because `pool` is unconditionally available, a user with zero permissions and zero teams can still export **every unassigned lead in the org**. This is consistent with the list view behavior.
</Note>

### Implementation considerations

<Warning>
The `teamId` query parameter is accepted for backward compatibility but **ignored**. The union is always computed server-side from the caller's own memberships/permissions.
</Warning>

<Tip>
**For maintainers:** The union path builds the **unfiltered** `ALL` base query and then conditionally adds the predicate. Safety relies on the invariant that `resolveAccessibleLeadsFragment` returns `null` **if and only if** the user has `canViewAll`.
</Tip>

## Deal, Contact, and Company row scoping

Deal, Contact, and Company exports use the **same permission-free accessible-union model as leads**. `validateTypeAccess` is **never called** by these export endpoints.

### Export scopes

| Frontend scope | Sent `type`             | Rows returned                                                                        |
| -------------- | ----------------------- | ------------------------------------------------------------------------------------ |
| `my`           | `MY` (entity-specific)  | Only entities the user **created** or is a **direct stakeholder** on |
| `all`          | `ALL` (entity-specific) | **Accessible union** â€” every row the user can see across their tabs                  |
| `selected`     | `ALL` + `selectedIds`   | Same as `all` but AND-intersected with the supplied ID list                          |

### Per-entity accessible union

Each entity has a `resolveAccessible<Entity>Fragment(em, user)` method:

<Tabs>
<Tab title="Contacts & Companies">
- **Returns `null`** when user has `crm.view` (org-wide view permission)
- **Otherwise** returns OR-combined SQL predicate for:
  - **MY**: rows the user created or is a direct stakeholder on
  - **TEAM**: rows for each team where user holds `team_crm.view`
</Tab>

<Tab title="Deals">
- **Returns `null`** when user has `sales.view` (org-wide view permission)
- **Otherwise** returns OR-combined SQL predicate for:
  - **MY**: deals the user created or is a direct stakeholder on
  - **TEAM**: deals for each team where user holds `team_sales.view`
</Tab>

<Tab title="Contacts (special case)">
Additionally includes:
- **DEAL-LINKED contacts**: contacts linked to active deals, **if** the user holds `sales.view`
</Tab>
</Tabs>

<Note>
Non-privileged team members (members without `team_crm.view`) can still see rows where they are a personal stakeholder within that team's scope.
</Note>

<Warning>
The `teamId` query parameter is **ignored** for all three entities â€” the union is always computed server-side from the caller's own memberships/permissions.
</Warning>

## RLS backstop

Every export runs inside:

```typescript
return this.tenantContext.executeReadOnly(organizationId, async (em) => { 
  // ... export logic
});
```

This sets the tenant (organization) context, so **PostgreSQL row-level security physically scopes every query to the caller's organization**. Even if application-level logic fails, users cannot access data from other organizations.

## Filter and selectedIds behavior

All exports support optional query parameters for further narrowing results:

<CardGroup cols={2}>
<Card title="Filter parameters" icon="filter">
Standard entity filters (date ranges, status, etc.) are AND-combined with the access predicate
</Card>

<Card title="selectedIds parameter" icon="list-check">
When provided, results are AND-intersected with the supplied ID list for targeted exports
</Card>
</CardGroup>

## File format rules

All exports follow consistent formatting:

- **CSV**: Default format with proper escaping
- **XLSX**: Available as alternative format
- **Filename pattern**: `{entity}-{scope}-{timestamp}.{ext}`
- **Column selection**: Based on user's field visibility permissions

<Check>
Export files include only fields the user has permission to view in the standard CRM interface.
</Check>
