A contact represents the ongoing business relationship with a person. Contact is the **business layer** entity ‚Äî created when a lead converts.

## Architecture overview

The CRM follows a **Person + Contact Model** design:

- **Person** is the hidden identity layer (single source of truth for personal details)
- **Contact** is the business relationship layer (qualified customers)
- **Lead** is the sales opportunity layer (unqualified inquiries)  
- **Deal** links to Contact, not Person directly

This modular design keeps CRM core independent while allowing Real Estate, Marketing, and Channel modules to extend functionality.

### Real Estate integration

Real Estate entities link to the CRM layer that owns their business context:

| Real Estate Entity       | Links To     | Rationale                                           |
| ------------------------ | ------------ | --------------------------------------------------- |
| `unit_ownership`         | `contact_id` | Ownership is represented by the org's owner Contact |
| `unit_transaction`       | `person_id`  | Transaction party is an individual                  |
| `site_visit`             | `person_id`  | Who visited the property                            |
| `lead_property_interest` | `lead_id`    | Links to Lead for sales context                     |
| `deal_property_interest` | `deal_id`    | Links to Deal for transaction context               |

**Deal Property Interest Workflow**:

```
// Deal created FROM Lead:
// Copy the explicitly selected LeadPropertyInterest criteria ‚Üí DealPropertyInterest (1:1)
// If the lead interest has multiple InventoryUnit candidates, choose one unit for the deal.
deal.propertyInterest.originatingInterest = leadPropertyInterest

// Deal created directly (walk-in):
// Create DealPropertyInterest with no originating interest and at least one criterion
// (an inventory unit selection is sufficient on its own)
deal.propertyInterest.originatingInterest = null
```

**Access Pattern for Contact**:

```
// Contact page showing ownership
contact.unitOwnerships  // All ownerships
contact.unitOwnerships.filter(o => o.isActive)  // Current
contact.person  // Identity details for the owner Contact
```

## CRM public IDs

All core CRM business records expose a stored `publicId` for user-facing references. The format is:

```
{ENTITY_CODE}-{ORG_PREFIX}-{SEQUENCE}
```

Examples: `LEAD-ADNS-001`, `PERS-ADNS-001`, `CONT-ADNS-001`, `COMP-ADNS-001`, `DEAL-ADNS-001`, `COMM-ADNS-001`, `USER-ADNS-001`, `TEAM-ADNS-001`, `LSTG-ADNS-001`, `DSTG-ADNS-001`.

<Info>
Entity codes make IDs globally unique within an organization across CRM record types. Contact uses the `CONT` entity code for its public IDs.
</Info>

**Key rules:**

- `ORG_PREFIX` is generated once from the organization's normalized alphanumeric name
- Names of 4 characters or fewer use the whole normalized name (`WIK`)
- Longer names use the first 2 and last 2 characters (`Adidas Operations` ‚Üí `ADNS`)
- `SEQUENCE` is scoped by organization and entity type, padded to at least 3 digits
- IDs are allocated by `CrmPublicIdService` from `crm_public_id_counter` in the same transaction as entity creation
- Organization user memberships also receive stable fallback avatar colors from the user sequence
- Entity codes: Lead=`LEAD`, Person=`PERS`, Contact=`CONT`, Company=`COMP`, Deal=`DEAL`, CommissionPayment=`COMM`, User=`USER`, Team=`TEAM`, LeadStage=`LSTG`, DealStage=`DSTG`
- Global system stages use deterministic IDs based on `systemType` (e.g. `LSTG-NEW`, `LSTG-DISQUALIFIED`, `DSTG-CLOSED-WON`)

## Core entities

### Person (central identity)

**Purpose**: Single source of truth for human identity and preferences.

```
Person
‚îú‚îÄ‚îÄ public_id (coded org-scoped human ID, e.g. PERS-ADNS-001)
‚îú‚îÄ‚îÄ Identity: first_name, last_name, avatar_url, title
‚îú‚îÄ‚îÄ Demographics: date_of_birth, nationality
‚îú‚îÄ‚îÄ Links: links jsonb [{ type, url }] where type is website/facebook/instagram/linkedIn/twitter/tiktok
‚îú‚îÄ‚îÄ Preferences: timezone
‚îú‚îÄ‚îÄ Languages: languages (unified array with code and proficiency per entry)
‚îú‚îÄ‚îÄ Communication Flags: do_not_call, do_not_email
‚îú‚îÄ‚îÄ Source Tracking: original_source
‚îú‚îÄ‚îÄ Merge Tracking: merged_into_id, merged_at, merged_by
‚îú‚îÄ‚îÄ Computed: full_name (getter: first_name + last_name)
‚îî‚îÄ‚îÄ Related Tables:
    ‚îú‚îÄ‚îÄ person_email (multiple emails, one primary)
    ‚îú‚îÄ‚îÄ person_phone (multiple phones, one primary)
    ‚îú‚îÄ‚îÄ person_address (multiple addresses, one primary)
    ‚îú‚îÄ‚îÄ person_channel (WhatsApp, Instagram, etc. identities)
    ‚îî‚îÄ‚îÄ person_not_duplicate (deduplication override pairs)
```

**Key Rules**:

- Every Lead, Contact must link to a Person
- `publicId` is generated once at creation using the shared CRM public ID allocator (`PERS-*`)
- Person preferences apply across all contexts (leads, deals, contacts)
- `nationality` is stored on Person so leads and contacts share the same identity profile value
- `original_source` is set once when person first enters system
- Languages array uses unified `UserLanguageEntry` format with code and proficiency per entry
- `links` stores web and social profile links as JSONB entries with `type` and `url`

#### PersonNotDuplicate (deduplication overrides)

**Purpose**: Records pairs of persons that have been manually confirmed as NOT duplicates.

```

### Contact entity

**Purpose**: Business relationship layer for qualified customers.

```
Contact
‚îú‚îÄ‚îÄ public_id (coded org-scoped contact ID, e.g. CONT-ADNS-001)
‚îú‚îÄ‚îÄ person_id (required) ‚Üí Person (OneToOne)

‚îú‚îÄ‚îÄ Deals: dealContacts ‚Üí DealContact[]

**Key rules:**

- One Person has ONE Contact (unlike leads)
- Contact visibility is derived from linked lead assignments
- No direct assignment fields ‚Äî visibility is inherited
- Deals link to Contact, not Person
- Companies link via `ContactCompanyRole`
- `publicId` uses `CONT-*` format for user-facing references

<Warning>
Contacts can only be created when a lead converts. Direct contact creation without a lead is not allowed through standard workflows.
</Warning>

### Activity backfill on contact creation

When a Contact is created for a Person who has existing lead activities, the system backfills `contactId` to make historical lead activities visible on the new contact's timeline:

```typescript
// In ContactService.create(), after persisting contact:
await activityBackfillService.backfillContactActivities(
  contact.id,
  person.id,
  em,
);

// 1. Find all leads for this person
// 2. Update activities WHERE target_entity_type='lead' AND target_entity_id IN (leadIds)
// 3. SET contact_id = newContactId WHERE contact_id IS NULL
```

This ensures a newly created contact immediately has a complete activity history from all the person's leads.

### Visibility logic

Contact visibility is derived from the person's lead assignments:

```
Person: John Smith
‚îú‚îÄ‚îÄ Lead #1 ‚Üí assigned_to: Agent A
‚îú‚îÄ‚îÄ Lead #2 ‚Üí assigned_to: Agent B
‚îî‚îÄ‚îÄ Contact ‚Üí Visible to: Agent A, Agent B, Managers, Owners
             NOT visible to: Other agents without lead access
```

**Visibility query:**

```sql
SELECT DISTINCT c.* FROM contact c
JOIN person p ON c.person_id = p.id
WHERE EXISTS (
  SELECT 1 FROM lead l
  JOIN entity_stakeholder es ON es.entity_type = 'lead' AND es.entity_id = l.id
  WHERE l.person_id = p.id AND es.user_id = :user_id
    AND l.is_deleted = false AND es.is_deleted = false
) AND c.is_deleted = false AND c.is_archived = false;
```

## Business rules

### Contact creation rules

1. **One-to-one relationship**: Each person can have only one contact
2. **Lead conversion required**: Contacts are created through lead conversion process
3. **Visibility inheritance**: Contact visibility is derived from person's lead assignments
4. **Historical preservation**: Lead data and activities are preserved after conversion

### Assignment and transfer rules

1. **Stakeholder-based assignments**: Use entity_stakeholder for all contact assignments
2. **Commission splits**: Multiple agents can have commission splits on contact deals
3. **Transfer approvals**: Contact transfers require manager approval above certain value thresholds
4. **Visibility cascade**: Assignment changes affect visibility across related entities

### Data integrity rules

1. **Person relationship**: contact.person_id is required and must be unique
2. **Soft deletes**: Use is_deleted flag, never hard delete contacts with deal history
3. **Activity preservation**: Contact deletion must preserve related activity records
4. **Reference numbers**: When provided, reference_number must be unique within organization

## Data consistency guarantees

The CRM module ensures data consistency through several mechanisms:

### Transaction boundaries

<Info>
All contact operations are wrapped in database transactions to ensure atomicity. Complex operations like lead conversion and contact merging use distributed transactions across related entities.
</Info>

### Referential integrity

- **Person-Contact relationship**: Enforced at database level with foreign key constraints
- **Soft delete cascades**: Contact deletion triggers soft delete of related activities and notes
- **Assignment consistency**: EntityStakeholder records are validated against active contact status

### Eventual consistency

<Note>
Some derived fields (like lifetime value calculations) use eventual consistency patterns with background job processing to maintain system performance.
</Note>
