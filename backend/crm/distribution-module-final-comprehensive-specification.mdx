The Distribution Module automates lead assignment within organizations. When a new lead is created, the system evaluates org-defined rules to automatically assign the lead to the most appropriate agent — based on lead attributes, UserStatus online/away state, working-hours eligibility, language compatibility, and capacity.

<Info>
**Status:** Active — fully implemented  
**Module Path:** `src/modules/crm/distribution/`
</Info>

## Design principles

| Principle | Decision |
|-----------|----------|
| Async distribution | `createLead()` emits `LEAD_CREATED` after commit; a pg-boss worker handles distribution. Listener / emit failures are logged only — HTTP lead creation still returns success; manual assignment or backfill may be needed if enqueue never ran |
| Stakeholder system reuse | Distribution creates `EntityStakeholder` records via `EntityStakeholderService`, not a new paradigm |
| First-match-wins rules | Rules are evaluated top-to-bottom by priority; the first matching rule wins |
| Idempotency | Distribution engine checks for existing stakeholders or pending offers before running |
| No retroactive distribution | Existing leads are unaffected when rules are created; only new leads trigger distribution |
| pg-boss scheduling | Distribution queue uses pg-boss for reliability and retry guarantees |
| RLS compliance | All entities carry `organization_id` for row-level security |

### Distribution paths

The engine supports two execution paths:

**Path A — Org-level distribution** (`runDistribution`): triggered when a lead enters the org with no team context. Evaluates org-scoped rules, applies the org default method, and can bridge to Path B if a rule or default method routes to a team that has `distributionEnabled = true`.

**Path B — Team-level distribution** (`runTeamDistribution`): triggered directly when:
- A lead is created with a `teamId` in the event payload (team pool assignment)
- Path A determines the lead belongs to an auto-distributing team
- Idempotency check finds a single team-only stakeholder with auto-distribute enabled

<Note>
Path B evaluates team-scoped rules, uses team settings (with org fallback for capacity), and logs the team FK on the resulting `DistributionLog` record.
</Note>

### Component responsibilities

| Component | Responsibility |
|-----------|----------------|
| **DistributionEngine** | Orchestrator: receives a lead, evaluates rules, selects agent, creates assignment. Supports Path A (org) and Path B (team). |
| **RuleEvaluator** | Evaluates rule conditions against lead data; returns first matching rule |
| **LanguageMatcher** | Filters and ranks agents by language compatibility with the lead's person |
| **AgentSelector** | Applies the distribution method (round-robin, weighted, weighted-round-robin, direct) to the filtered agent pool |
| **DistributionCapacityService** | Two-phase capacity enforcement: Phase 1 `filterByCapacity()` (lead counts vs limits); Phase 2 `confirmCapacityAndAssign()` (advisory locks + atomic stakeholder creation). No entity of its own — queries `entity_stakeholder`. |
| **UserStatusService** | Pre-filters candidate agents to ONLINE status; filters by per-user working hours (`filterByWorkingHours`); provides `isWithinWorkingHours()` for org-level business hours check. |
| **DistributionListener** | Listens for `LEAD_CREATED` events and enqueues pg-boss jobs. The handler is fault-isolated (try/catch): settings lookup and enqueue errors are logged and do not fail `POST /v1/leads`. |
| **DistributionJobHandler** | pg-boss worker that processes distribution jobs |

<Warning>
**Master toggle behavior:**
- `distributionEnabled = false` (new-org default): Engine is off. `DistributionListener` and `LeadImportService` skip enqueue entirely — leads go to pool, no pg-boss jobs created.
- `distributionEnabled = true`: Engine is active. When toggled from `false` → `true` in `DistributionSettingsService.update()`, if `defaultMethod` is still `POOL` it is auto-upgraded to `ROUND_ROBIN` for a smooth first-run experience.
</Warning>

<Tip>
**Org backfill on toggle-on:** After the settings row is saved, `backfillUnassignedLeads()` runs asynchronously (fire-and-forget). It selects up to 2000 active leads in the org that have **no** `entity_stakeholder` rows for `entity_type = 'lead'` and batch-enqueues them via `DistributionJobHandler.enqueueBatch()` (priority 0).
</Tip>

## Distribution engine

### Core workflow

<Steps>
<Step title="Event processing">
Pg-boss worker receives `DistributionEvent` and validates the lead exists and is not already assigned
</Step>

<Step title="Path determination">
- If `teamId` present in event → Path B (team distribution)  
- If lead has single team stakeholder with auto-distribute enabled → Path B
- Otherwise → Path A (org distribution)
</Step>

<Step title="Rule evaluation">
Evaluate applicable rules by priority order (org-scoped for Path A, team-scoped for Path B)
</Step>

<Step title="Agent selection">
Apply distribution method (round-robin, weighted, etc.) to filtered candidate pool
</Step>

<Step title="Assignment creation">
Create `EntityStakeholder` record and log outcome in `DistributionLog`
</Step>
</Steps>

### Rule evaluation

The `RuleEvaluator` processes rules in ascending priority order and returns the first match:

```typescript
interface RuleCondition {
  field: string;
  operator: 'equals' | 'in' | 'contains' | 'range';
  value: any;
}

// Example conditions
{
  "leadSource": ["website", "referral"],
  "leadValue": { "min": 1000, "max": 10000 },
  "leadTags": ["vip", "urgent"],
  "personLanguages": ["en", "es"]
}
```

<Warning>
Rules are evaluated against lead and person data at distribution time. Changes to lead data after creation do not trigger re-evaluation.
</Warning>

### Language matching

The `LanguageMatcher` component filters agents based on language compatibility:

<Tabs>
<Tab title="STRICT mode">
Agent must have at least one language that exactly matches the lead's person languages
</Tab>
<Tab title="PREFERRED mode">  
Agents with matching languages are ranked higher, but non-matching agents are still included
</Tab>
</Tabs>

### Agent selection methods

<AccordionGroup>
<Accordion title="ROUND_ROBIN">
Selects the next agent in rotation using `last_assigned_index`. Index is atomically incremented after each assignment.
</Accordion>

<Accordion title="WEIGHTED">
Assigns leads based on agent weight ratios. Higher weights = more lead assignments.
</Accordion>

<Accordion title="WEIGHTED_ROUND_ROBIN">
Combines round-robin fairness with weighted distribution over time.
</Accordion>

<Accordion title="SPECIFIC_AGENT">
Directly assigns to a predetermined agent (if available and within capacity).
</Accordion>

<Accordion title="SPECIFIC_TEAM">
Routes lead to a specific team. If team has auto-distribution enabled, bridges to Path B.
</Accordion>
</AccordionGroup>

### Capacity enforcement

<Check>
Two-phase capacity checking ensures atomic assignment under concurrent load:
</Check>

**Phase 1 - Filtering:** `filterByCapacity()` removes agents who exceed limits based on current counts

**Phase 2 - Confirmation:** `confirmCapacityAndAssign()` uses advisory locks and rechecks capacity before creating the stakeholder record

```sql
-- Advisory lock pattern for capacity confirmation
SELECT pg_advisory_lock(hash_key);
-- Recheck current counts
-- Create stakeholder if still within limits  
SELECT pg_advisory_unlock(hash_key);
```

### Business hours gating

When enabled, the engine checks if the current time falls within the organization's business hours:

1. **Org-level check:** `Organization.settings.businessHours.enabled` AND `DistributionSettings.respectBusinessHours`
2. **Time validation:** Current time matches defined weekly schedule in org timezone
3. **Outside hours action:** Determined by `outsideHoursAction` setting

### Retry strategy

| Attempt | Delay | Action |
|---------|-------|--------|
| 1 | 0s | Initial attempt |
| 2 | 30s | First retry |
| 3 | 30s | Second retry |
| 4 | 30s | Final retry |
| Failed | - | Job marked as failed, alert sent |

## Edge case handling

### Concurrent assignments

Advisory locks prevent double-assignment under high concurrency:

```sql
-- Capacity confirmation with locking
SELECT pg_advisory_lock(user_capacity_lock_key);
-- Recheck counts and create stakeholder atomically
SELECT pg_advisory_unlock(user_capacity_lock_key);
```

### Agent status changes

Real-time agent status changes are handled gracefully:

- Agents going offline during distribution are skipped
- Capacity changes are reflected in next distribution cycle
- Working hours changes affect future assignments only

<Note>
The system does not reassign leads when agent status changes - manual reassignment may be needed.
</Note>

### Time zone handling

Business hours are evaluated in the organization's configured timezone:

```typescript
// Business hours evaluation
const orgTimezone = organization.settings.businessHours.timezone;
const currentTime = DateTime.now().setZone(orgTimezone);
const isWithinHours = businessHoursConfig.isWithinSchedule(currentTime);
```

### Large agent pools

Performance optimizations for organizations with many agents:

- Agent filtering is done in database queries where possible
- Capacity checks are batched for efficiency  
- Round-robin indices are updated atomically to prevent skew

CREATE POLICY team_distribution_settings_org_isolation ON team_distribution_settings
  FOR ALL TO authenticated
  USING (organization_id = current_setting('app.current_organization_id')::uuid);

-- Distribution rules isolation
CREATE POLICY distribution_rules_org_isolation ON distribution_rules
  FOR ALL TO authenticated
  USING (organization_id = current_setting('app.current_organization_id')::uuid);

-- Distribution logs isolation
CREATE POLICY distribution_log_org_isolation ON distribution_log
  FOR ALL TO authenticated
  USING (organization_id = current_setting('app.current_organization_id')::uuid);
