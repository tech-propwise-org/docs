The transfer system manages formal transfers of leads and deals between agents or teams, with an approval workflow and commission splitting that integrates with the broader stakeholder system and commission payments.

## entity_transfer

```
entity_transfer
├── entity_type (lead | deal)
├── entity_id
├── from_user_id (nullable), from_team_id (nullable)
├── to_user_id (nullable), to_team_id (nullable)
├── Commission handling:
│   ├── from_commission_total (snapshot of sender's commission at request time)
│   └── from_keeps_percentage (e.g., 30%)
├── Workflow:
│   ├── status (pending, approved, rejected, cancelled)
│   ├── reason, reject_reason
│   ├── requested_by_id, requested_at
│   ├── approved_by, approved_at, rejected_by, rejected_at
├── created_at, updated_at
├── organization_id
└── Database constraints:
    └── Unique partial index: (entity_type, entity_id) WHERE status = 'pending'
```

<Info>
The unique partial index ensures only one pending transfer can exist per entity at a time. This prevents conflicting transfer requests and maintains system integrity.
</Info>

## Transfer types

| From | To   | Description                      |
| ---- | ---- | -------------------------------- |
| User | User | Standard agent-to-agent transfer |
| User | Team | Agent transfers to a team pool   |
| Team | User | Team pool claim by a user        |
| Team | Team | Team-to-team reallocation        |

## Transfer request logic

The transfer system includes a comprehensive request workflow that tracks who initiated the transfer and when:

<AccordionGroup>
<Accordion title="Request initiation">
When a transfer is initiated, the system captures:
- `requested_by_id`: The user who initiated the transfer request
- `requested_at`: Timestamp of the request
- `reason`: Optional reason provided by the requestor
- `from_commission_total`: Snapshot of current commission percentage
</Accordion>

<Accordion title="Request validation">
The system validates:
- Requestor has permission to transfer the entity
- Target user/team exists and is valid
- No conflicting pending transfers exist (enforced by unique constraint)
- All parties are within the same organization
</Accordion>

<Accordion title="Commission snapshot">
The `from_commission_total` field captures the exact commission percentage at request time, ensuring calculations remain accurate even if stakeholder percentages change during the pending period.
</Accordion>
</AccordionGroup>

## Transfer workflow

<Steps>
<Step title="Transfer request">
Agent A requests transfer of Lead #1 to Agent B. An `entity_transfer` record is created with `from_commission_total` (snapshot) and `from_keeps_percentage`.

<Note>
The system validates that the transfer is valid (cannot transfer to self, must have valid from/to targets) and ensures only one pending transfer exists per entity via the unique partial index constraint.
</Note>
</Step>

<Step title="Pending review">
Agent B (or manager) reviews the request. They can approve with `approved_by`/`approved_at`, reject with `rejected_by`/`rejected_at`/`reject_reason`, or cancel.

<Warning>
**Auto-cancel:** If while a transfer is pending, any stakeholder on the entity is removed or has their commission reduced below the `from_commission_total`, the transfer is automatically cancelled because the commission split is now invalid.
</Warning>
</Step>

<Step title="Transfer approved">
On approval:
- Agent A's stakeholder is updated: `30%`, `isPrimary=false`
- Agent B's stakeholder is created: `isPrimary=true`, `70%`
- History records created in `entity_stakeholder_history` for both agents
- Transfer status set to `approved`
- Commission payment records updated to reflect new stakeholder arrangement
</Step>

<Step title="Downstream effects">
When a new deal is created from the lead later, all lead stakeholders are auto-copied to the deal with their commission splits preserved, maintaining the transfer results throughout the sales pipeline.
</Step>
</Steps>

## Commission calculation

The commission split is calculated using the formula:

```
to_commission = from_commission_total × (100% - from_keeps_percentage)
```

**Example:**
- Agent A has 100% commission on a lead
- Agent A transfers to Agent B, keeping 30%
- Result: Agent A keeps 30%, Agent B gets 70%

<Tip>
The `from_commission_total` field captures the transferor's commission percentage at the time of request, ensuring the calculation remains valid even if stakeholder percentages change later due to other system operations.
</Tip>

## Database constraints and indexes

The transfer system uses several database constraints and indexes to maintain data integrity:

<AccordionGroup>
<Accordion title="Unique partial index">
**Index:** `UNIQUE (entity_type, entity_id) WHERE status = 'pending'`

**Purpose:** Ensures only one pending transfer can exist per entity at any time. This prevents conflicting transfer requests and race conditions.

**Implementation:** PostgreSQL partial index that only applies to records with `status = 'pending'`, allowing multiple historical transfers (approved/rejected/cancelled) per entity.
</Accordion>

<Accordion title="Foreign key constraints">
- `entity_id` → `Lead.id` or `Deal.id` (polymorphic)
- `from_user_id` → `User.id`
- `from_team_id` → `Team.id`
- `to_user_id` → `User.id`
- `to_team_id` → `Team.id`
- `requested_by_id` → `User.id`
- `approved_by` → `User.id`
- `rejected_by` → `User.id`
- `organization_id` → `Organization.id`
</Accordion>

<Accordion title="Check constraints">
- Either `from_user_id` OR `from_team_id` must be set (not both)
- Either `to_user_id` OR `to_team_id` must be set (not both)
- `from_keeps_percentage` must be between 0 and 100
- `from_commission_total` must be >= 0
</Accordion>
</AccordionGroup>

## Auto-cancel triggers and implementation

Pending transfers are automatically cancelled through comprehensive system triggers when the underlying commission structure becomes invalid:

<AccordionGroup>
<Accordion title="Stakeholder removal trigger">
**When:** Any stakeholder on the entity (lead/deal) is completely removed from the stakeholder list.

**Implementation:** Database trigger on `entity_stakeholder` DELETE operations:
```sql
CREATE OR REPLACE FUNCTION cancel_transfers_on_stakeholder_removal()
RETURNS trigger AS $$
BEGIN
  UPDATE entity_transfer 
  SET status = 'cancelled', 
      updated_at = NOW()
  WHERE entity_type = OLD.entity_type 
    AND entity_id = OLD.entity_id 
    AND status = 'pending';
  RETURN OLD;
END;
$$ LANGUAGE plpgsql;
```

**Rationale:** The commission split calculation becomes invalid if the transferor is no longer a stakeholder.
</Accordion>

<Accordion title="Commission reduction trigger">
**When:** Any stakeholder has their commission percentage reduced below what was captured in `from_commission_total`.

**Implementation:** Database trigger on `entity_stakeholder` UPDATE operations:
```sql
CREATE OR REPLACE FUNCTION cancel_transfers_on_commission_reduction()
RETURNS trigger AS $$
BEGIN
  UPDATE entity_transfer 
  SET status = 'cancelled', 
      updated_at = NOW()
  WHERE entity_type = NEW.entity_type 
    AND entity_id = NEW.entity_id 
    AND status = 'pending'
    AND from_commission_total > NEW.commission_percentage;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;
```

**Rationale:** The promised commission split can no longer be fulfilled if the source commission is insufficient.
</Accordion>

<Accordion title="Entity lifecycle triggers">
**When:** The underlying entity (lead/deal) changes to a status that would invalidate the transfer.

**Scenarios:**
- Lead converted to deal (pending lead transfers cancelled)
- Deal closed/won/lost (pending deal transfers cancelled)
- Entity soft-deleted or archived

**Implementation:** Application-level logic in entity status change handlers:
```javascript
// In LeadService.convertToDeal()
await this.transferService.cancelPendingTransfers('lead', leadId, {
  reason: 'Lead converted to deal',
  cancelledBy: userId
});

// In DealService.markClosed()
await this.transferService.cancelPendingTransfers('deal', dealId, {
  reason: 'Deal closed',
  cancelledBy: userId
});
```

**Rationale:** Transfers become meaningless once the entity has moved beyond the transferable stage.
</Accordion>

<Accordion title="User/team deactivation triggers">
**When:** A user or team involved in a pending transfer is deactivated or deleted.

**Implementation:** Application-level cascade operations:
```javascript
// In UserService.deactivateUser()
await this.transferService.cancelTransfersForUser(userId, {
  reason: 'User deactivated',
  cancelledBy: adminUserId
});
```

**Rationale:** Prevents orphaned transfers and maintains system integrity.
</Accordion>
</AccordionGroup>

## Query patterns

The transfer system supports several key query patterns for efficient data access:

<AccordionGroup>
<Accordion title="Pending transfers for approval">
```sql
-- Get all pending transfers requiring user's approval
SELECT t.*, 
       e_from.full_name as from_name,
       e_to.full_name as to_name,
       CASE t.entity_type 
         WHEN 'lead' THEN l.title 
         WHEN 'deal' THEN d.title 
       END as entity_title
FROM entity_transfer t
LEFT JOIN users e_from ON t.from_user_id = e_from.id
LEFT JOIN users e_to ON t.to_user_id = e_to.id  
LEFT JOIN leads l ON t.entity_type = 'lead' AND t.entity_id = l.id
LEFT JOIN deals d ON t.entity_type = 'deal' AND t.entity_id = d.id
WHERE t.status = 'pending'
  AND (t.to_user_id = ? OR t.to_team_id IN (?))
  AND t.organization_id = ?
ORDER BY t.requested_at DESC;
```
</Accordion>

<Accordion title="Transfer history for entity">
```sql
-- Get complete transfer history for a lead/deal
SELECT t.*, 
       req.full_name as requested_by_name,
       app.full_name as approved_by_name,
       rej.full_name as rejected_by_name
FROM entity_transfer t
LEFT JOIN users req ON t.requested_by_id = req.id
LEFT JOIN users app ON t.approved_by = app.id
LEFT JOIN users rej ON t.rejected_by = rej.id
WHERE t.entity_type = ? 
  AND t.entity_id = ?
  AND t.organization_id = ?
ORDER BY t.created_at DESC;
```
</Accordion>

<Accordion title="Commission impact analysis">
```sql
-- Calculate commission impact of pending transfers
SELECT t.entity_type,
       t.entity_id,
       t.from_commission_total,
       t.from_keeps_percentage,
       (t.from_commission_total * (100 - t.from_keeps_percentage) / 100) as transfer_amount
FROM entity_transfer t
WHERE t.status = 'pending'
  AND t.organization_id = ?;
```
</Accordion>

<Accordion title="Auto-cancel trigger queries">
```sql
-- Find transfers that should be cancelled due to insufficient commission
SELECT t.id
FROM entity_transfer t
JOIN entity_stakeholder es ON t.entity_type = es.entity_type 
  AND t.entity_id = es.entity_id
WHERE t.status = 'pending'
  AND es.commission_percentage < t.from_commission_total;
```
</Accordion>
</AccordionGroup>

## Error handling and edge cases

<AccordionGroup>
<Accordion title="Validation errors">
**Invalid transfer targets:** Returns 400 error with specific validation message
- Cannot transfer to self
- Target user/team must exist and be active
- Must specify either user_id or team_id (not both) for from/to

**Permission errors:** Returns 403 error
- Requestor must have transfer permissions on the entity
- Organization boundaries must be respected
</Accordion>

<Accordion title="Concurrent modification">
**Race conditions:** The unique partial index prevents multiple pending transfers, but the system handles concurrent stakeholder modifications gracefully by auto-cancelling affected transfers.

**Stale data:** Commission snapshots protect against stale percentage calculations, but the system validates current state before applying approved transfers.
</Accordion>

<Accordion title="Orphaned transfers">
**Entity deletion:** If an entity is deleted while transfers are pending, cleanup jobs automatically cancel all related transfers.

**User deactivation:** If a user involved in a pending transfer is deactivated, the transfer is automatically cancelled.
</Accordion>

<Accordion title="Commission calculation edge cases">
**Zero commission transfers:** Allowed when `from_keeps_percentage = 0` and target is already a stakeholder. This transfers primary status without moving commission.

**Insufficient commission:** If stakeholder's commission drops below `from_commission_total` while transfer is pending, the transfer is auto-cancelled.

**Rounding precision:** Commission calculations use decimal precision to avoid floating-point errors in financial calculations.
</Accordion>
</AccordionGroup>

## Business rules implementation

The transfer system enforces comprehensive business rules through database constraints, application logic, and triggers:

<AccordionGroup>
<Accordion title="Transfer eligibility rules">
- **Same organization:** All parties must belong to the same organization
- **Active entities:** Can only transfer active leads/deals (not archived/deleted)
- **Stakeholder status:** Transferor must be a current stakeholder with sufficient commission
- **Target validity:** Target user/team must be active and have appropriate permissions
</Accordion>

<Accordion title="Commission rules">
- **Snapshot integrity:** `from_commission_total` preserves the commission amount at request time
- **Split validation:** `from_keeps_percentage` must be between 0-100%
- **Minimum thresholds:** Organizations can set minimum commission amounts for transfers
- **Team distributions:** When transferring to/from teams, commission is distributed according to team rules
</Accordion>

<Accordion title="Workflow rules">
- **Approval required:** All transfers require explicit approval from the recipient
- **Self-approval:** Users cannot approve their own outbound transfer requests
- **Manager override:** Managers can approve/reject transfers on behalf of their team members
- **Timeout handling:** Transfers can be configured to auto-expire after a specified period
</Accordion>

<Accordion title="History and audit rules">
- **Immutable history:** Transfer records are never deleted, only status-updated
- **Complete audit trail:** All status changes, approvals, and cancellations are logged
- **Commission tracking:** Changes to stakeholder records reference the originating transfer
- **Activity logging:** Transfer events appear in entity activity streams
</Accordion>
</AccordionGroup>
