The transfer system manages formal transfers of leads and deals between agents or teams, with an approval workflow and commission splitting that integrates with the broader stakeholder system and commission payments within the CRM architecture.

## Architecture integration

The transfer system is a core component of the CRM module that works within the Person + Contact model architecture:

<Tabs>
<Tab title="CRM core integration">
**Entity relationships:**
- Transfers operate on `Lead` and `Deal` entities (not `Person` directly)
- Integrates with unified stakeholder model via `entity_stakeholder`
- Uses polymorphic patterns for entity_type/entity_id references
- Maintains organization-scoped data boundaries

**Data flow:**
```
Person → Contact/Lead → Deal → Transfer → Stakeholder Updates → Commission Payments
```
</Tab>

<Tab title="Stakeholder system">
**Core integration points:**
- Modifies `entity_stakeholder` records on approval
- Creates `entity_stakeholder_history` for audit trail
- Preserves commission percentages and primary status
- Auto-copies stakeholder arrangements from leads to deals
</Tab>

<Tab title="Real estate module">
**Property interest handling:**
- Deal transfers maintain linkage to `deal_property_interest` records
- Preserves `site_visit` attribution through Person relationships
- Supports property-specific commission arrangements
- Maintains connection to originating `lead_property_interest`
</Tab>
</Tabs>

## Entity transfer schema

The `entity_transfer` table manages the complete transfer workflow with comprehensive tracking and validation:

```
entity_transfer
├── Entity reference:
│   ├── entity_type (lead | deal)
│   └── entity_id
├── Transfer parties:
│   ├── from_user_id (nullable), from_team_id (nullable)
│   └── to_user_id (nullable), to_team_id (nullable)
├── Commission handling:
│   ├── from_commission_total (snapshot of sender's commission at request time)
│   └── from_keeps_percentage (e.g., 30%)
├── Workflow tracking:
│   ├── status (pending, approved, rejected, cancelled)
│   ├── reason, reject_reason
│   ├── requested_by_id, requested_at
│   ├── approved_by, approved_at, rejected_by, rejected_at
├── Audit fields:
│   ├── created_at, updated_at
│   └── organization_id
└── Database constraints:
    ├── Unique partial index: (entity_type, entity_id) WHERE status = 'pending'
    ├── Check: Either from_user_id OR from_team_id (not both)
    ├── Check: Either to_user_id OR to_team_id (not both)
    ├── Check: from_keeps_percentage BETWEEN 0 AND 100
    └── Check: from_commission_total >= 0
```

<Info>
The unique partial index ensures only one pending transfer can exist per entity at a time. This prevents conflicting transfer requests and maintains system integrity while allowing multiple historical transfers per entity.
</Info>

## Transfer approval workflow

The transfer system implements a comprehensive approval workflow with validation and auto-cancel protection:

<Steps>
<Step title="Transfer request submission">
Agent A requests transfer of Lead #1 to Agent B. System creates `entity_transfer` record with:
- `from_commission_total`: snapshot of transferor's current commission percentage
- `from_keeps_percentage`: percentage the transferor will retain (0-100%)
- `status`: set to 'pending'

<Warning>
System validates transfer eligibility:
- Cannot transfer to self
- Must have valid from/to targets (user or team, not both)
- Transferor must be current stakeholder with sufficient commission
- Entity must be in transferable status (active lead/deal)
- All parties must be in same organization
</Warning>
</Step>

<Step title="Pending review period">
Transfer enters pending state where recipient can:
- Approve with `approved_by`/`approved_at`
- Reject with `rejected_by`/`rejected_at`/`reject_reason`
- Let it auto-expire (configurable timeout)

<Note>
**Auto-cancel triggers activate during pending period:**
- Stakeholder removal: If transferor is removed from entity
- Commission reduction: If transferor's commission drops below `from_commission_total`
- Entity lifecycle changes: Lead conversion, deal closure, entity deletion
- User/team deactivation: Any involved party becomes inactive
</Note>
</Step>

<Step title="Transfer approval execution">
On approval, system performs atomic operations:
- **Transferor stakeholder**: commission reduced to `from_keeps_percentage`, `is_primary` set to false
- **Recipient stakeholder**: created/updated with remaining commission, `is_primary` set to true
- **History tracking**: `entity_stakeholder_history` records created for both parties
- **Transfer status**: updated to 'approved'
- **Commission payments**: updated to reflect new ownership structure

<Info>
Commission split calculation:
```
to_commission = from_commission_total × (100% - from_keeps_percentage) / 100%
```
Example: 60% commission with 30% retention = 18% kept, 42% transferred
</Info>
</Step>

<Step title="Stakeholder record updates">
System updates stakeholder arrangements:
- Modifies existing `entity_stakeholder` records for transferor
- Creates new stakeholder record for recipient (if not already stakeholder)
- Transfers primary stakeholder flag appropriately
- Maintains commission percentage totals across all stakeholders
- Preserves other stakeholder relationships unchanged
</Step>

<Step title="Commission payment integration">
Financial system integration:
- Creates new `commission_payment` records for transferred amounts
- Updates future payment schedules to reflect new ownership
- Preserves historical payment records (no retroactive changes)
- Maintains commission liability accounting across stakeholder changes
- Applies commission caps and thresholds to new arrangements
</Step>

<Step title="Pipeline propagation">
When new deal is created from transferred lead:
- All lead stakeholders (including transfer results) auto-copied to deal
- Commission splits preserved throughout sales pipeline
- Ownership continuity maintained from lead through deal closure
- Property interests and attribution data follow stakeholder changes
</Step>
</Steps>

## Commission splitting mechanics

The commission split system uses precise calculations with commission snapshot protection:

<Tabs>
<Tab title="Calculation formula">
**Base calculation:**
```
to_commission = from_commission_total × (100% - from_keeps_percentage) / 100%
```

**Examples:**
- 100% commission, 30% retention: 30% kept, 70% transferred
- 60% commission, 20% retention: 48% kept, 12% transferred  
- 50% commission, 0% retention: 0% kept, 50% transferred (primary status only)
</Tab>

<Tab title="Commission snapshot protection">
**Snapshot integrity:** `from_commission_total` captures exact commission at request time, protecting against:
- Concurrent stakeholder modifications
- Other commission adjustments during pending period
- System changes that could invalidate transfer terms
- Race conditions between multiple operations

**Validation on approval:**
```javascript
// Verify transferor still has sufficient commission
const currentCommission = await getStakeholderCommission(entityType, entityId, fromUserId);
if (currentCommission < transfer.fromCommissionTotal) {
  await cancelTransfer(transferId, 'Insufficient commission remaining');
  throw new ValidationError('Commission no longer available for transfer');
}
```
</Tab>

<Tab title="Multi-stakeholder scenarios">
**Complex stakeholder arrangements:**
- Transfer preserves existing stakeholder relationships
- Only affects transferor and recipient commission percentages
- Other stakeholders remain unchanged
- Total commission across all stakeholders must not exceed 100%

**Team distribution:**
- Team commission distributed according to team member allocation rules
- Team member changes don't affect completed transfers
- Commission caps and minimums apply per team distribution policy
</Tab>
</Tabs>

<AccordionGroup>
<Accordion title="Zero-commission transfers">
**Primary status transfer:** When `from_keeps_percentage = 0`, transferor gives up all commission but transfer still creates proper stakeholder record for recipient.

**Use cases:**
- Pure primary status handoff without commission change
- Organizational restructuring where commission stays with original agent
- Manager delegation scenarios
- Temporary assignment for specific expertise

**Implementation:** System creates recipient stakeholder record with transferred commission even when amount is zero, maintaining proper primary stakeholder flag assignment.
</Accordion>

<Accordion title="Partial commission scenarios">
**Existing stakeholder transfers:** If recipient is already a stakeholder, their commission percentage increases by the transferred amount.

**Commission aggregation:**
```javascript
// Existing recipient with 20% commission receives 30% transfer
existingCommission = 20%;
transferredCommission = 30%;
newCommission = existingCommission + transferredCommission; // 50%
```

**Validation:** System ensures total commission across all stakeholders doesn't exceed 100% after transfer completion.
</Accordion>

<Accordion title="Team commission distribution">
**User-to-team transfers:** Commission transferred to team is distributed among team members according to:
- Team allocation percentages (if configured)
- Equal distribution among active members (default)
- Custom team commission rules (organization-specific)

**Team-to-user transfers:** Individual receives full commission amount from team allocation, reducing team's total commission accordingly.
</Accordion>
</AccordionGroup>

## Auto-cancel behavior

The system implements comprehensive auto-cancel triggers to maintain transfer integrity:

<AccordionGroup>
<Accordion title="Stakeholder modification triggers">
**Database trigger implementation:**
```sql
CREATE OR REPLACE FUNCTION cancel_transfers_on_stakeholder_removal()
RETURNS trigger AS $$
BEGIN
  UPDATE entity_transfer 
  SET status = 'cancelled', 
      updated_at = NOW(),
      reject_reason = 'Stakeholder removed from entity'
  WHERE entity_type = OLD.entity_type 
    AND entity_id = OLD.entity_id 
    AND status = 'pending'
    AND ((OLD.user_id = from_user_id) OR (OLD.team_id = from_team_id));
  RETURN OLD;
END;
$$ LANGUAGE plpgsql;
```

**Cancellation reasons:**
- Transferor completely removed from stakeholder list
- Commission reduced below `from_commission_total` threshold
- Stakeholder role or permissions changed to incompatible level
</Accordion>

<Accordion title="Commission reduction protection">
**Automatic validation trigger:**
```sql
CREATE OR REPLACE FUNCTION cancel_transfers_on_commission_reduction()
RETURNS trigger AS $$
BEGIN
  UPDATE entity_transfer 
  SET status = 'cancelled', 
      updated_at = NOW(),
      reject_reason = 'Insufficient commission remaining for transfer'
  WHERE entity_type = NEW.entity_type 
    AND entity_id = NEW.entity_id 
    AND status = 'pending'
    AND from_commission_total > NEW.commission_percentage
    AND ((NEW.user_id = from_user_id) OR (NEW.team_id = from_team_id));
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;
```

**Business rationale:** Prevents invalid transfers where promised commission split can no longer be fulfilled due to reduced stakeholder commission.
</Accordion>

## Business rules and validation

Comprehensive business rules ensure transfer integrity across organizational boundaries and system constraints:

<AccordionGroup>

- Cross-organization transfers blocked at API level with clear error messages

**Entity state requirements:**

```javascript
const validateTransferEligibility = async (entityType, entityId, fromUserId, toUserId) => {
  // Verify transferor is current stakeholder
  const transferor = await getStakeholder(entityType, entityId, fromUserId);
  if (!transferor) {
    throw new ValidationError('User is not a stakeholder on this entity');

  // Check minimum commission requirements
  const orgSettings = await getOrganizationSettings(transferor.organizationId);

    throw new ValidationError('Insufficient commission percentage for transfer');
  }
  
  // Verify target has appropriate permissions
  const canReceive = await checkEntityPermission(toUserId, entityType, entityId, 'receive_transfers');
  if (!canReceive) {
    throw new ValidationError('Target user cannot receive transfers for this entity type');
  }
  
  return true;
};
```

<Accordion title="Commission and financial validation">
**Commission integrity rules:**
- `from_commission_total` preserved exactly from request time to prevent calculation drift
- `from_keeps_percentage` must be 0-100% with decimal precision validation

**Financial boundary enforcement:**
```javascript
const validateCommissionSplit = async (entityType, entityId, transfer) => {
  const allStakeholders = await getEntityStakeholders(entityType, entityId);
  const otherStakeholdersTotal = allStakeholders
    .filter(s => s.userId !== transfer.fromUserId && s.teamId !== transfer.fromTeamId)
    .reduce((sum, s) => sum + s.commissionPercentage, 0);
  
  const retainedCommission = transfer.fromCommissionTotal * (transfer.fromKeepsPercentage / 100);
  const transferredCommission = transfer.fromCommissionTotal * ((100 - transfer.fromKeepsPercentage) / 100);
  
  const totalAfterTransfer = otherStakeholdersTotal + retainedCommission + transferredCommission;
  
  if (totalAfterTransfer > 100) {
    throw new ValidationError(`Transfer would result in ${totalAfterTransfer}% total commission (exceeds 100%)`);
  }
  
  return true;
};
```
</Accordion>

## Transfer completion workflow

After approval, the system executes a comprehensive completion workflow:

<Steps>
<Step title="Pre-completion validation">
System performs final validation before applying changes:
- Verify all parties still exist and are active
- Confirm entity hasn't changed state since transfer request
- Validate current commission levels match transfer assumptions
- Check for conflicting pending operations

```javascript
const validatePreCompletion = async (transfer) => {
  // Final entity state check
  const entity = await getEntity(transfer.entityType, transfer.entityId);
  if (entity.status !== transfer.originalEntityStatus) {
    throw new ConflictError('Entity status changed during transfer');
  }
  
  // Commission availability check  
  const currentCommission = await getStakeholderCommission(
    transfer.entityType, 
    transfer.entityId, 
    transfer.fromUserId
  );
  
  if (currentCommission < transfer.fromCommissionTotal) {
    throw new ConflictError('Insufficient commission for transfer completion');
  }
};
```
</Step>

<Step title="Atomic stakeholder updates">
Execute all stakeholder changes within single database transaction:

```sql
BEGIN TRANSACTION;

-- Update transferor's commission and primary status
UPDATE entity_stakeholder 
SET commission_percentage = :retainedCommission,
    is_primary = false,
    updated_at = NOW()
WHERE entity_type = :entityType 
  AND entity_id = :entityId 
  AND user_id = :fromUserId;

-- Create or update recipient stakeholder record
INSERT INTO entity_stakeholder 
  (entity_type, entity_id, user_id, commission_percentage, is_primary, role, created_at)
VALUES 
  (:entityType, :entityId, :toUserId, :transferredCommission, true, 'assigned', NOW())
ON CONFLICT (entity_type, entity_id, user_id) 
DO UPDATE SET 
  commission_percentage = entity_stakeholder.commission_percentage + :transferredCommission,
  is_primary = true,
  updated_at = NOW();

COMMIT;
```
</Step>

<Accordion title="Commission calculation edge cases">
**Floating point precision handling:**
```javascript
const calculateCommissionSplit = (fromCommissionTotal, fromKeepsPercentage) => {
  // Use decimal arithmetic to avoid floating point errors
  const Decimal = require('decimal.js');
  
  const total = new Decimal(fromCommissionTotal);
  const keepsPercent = new Decimal(fromKeepsPercentage);
  
  const retainedAmount = total.mul(keepsPercent.div(100));
  const transferredAmount = total.sub(retainedAmount);
  
  // Round to 2 decimal places for currency precision
  return {
    retained: retainedAmount.toDecimalPlaces(2).toNumber(),
    transferred: transferredAmount.toDecimalPlaces(2).toNumber()
  };
};
```

**Zero and boundary value handling:**
```javascript
// Handle edge cases in commission calculation
const validateCommissionBoundaries = (transfer) => {
  // Zero commission transfer (primary status only)
  if (transfer.fromCommissionTotal === 0) {
    return { type: 'primary_status_only', commission: 0 };
  }
  
  // 100% retention (no commission transferred)
  if (transfer.fromKeepsPercentage === 100) {
    return { type: 'status_change_only', commission: 0 };
  }
  
  // 0% retention (full commission transfer)
  if (transfer.fromKeepsPercentage === 0) {
    return { 
      type: 'full_transfer', 
      commission: transfer.fromCommissionTotal 
    };
  }
  
  // Standard split
  return { 
    type: 'split_transfer', 
    commission: calculateCommissionSplit(transfer.fromCommissionTotal, transfer.fromKeepsPercentage)
  };
};
```

## Business rules summary

| Rule Category | Rule | Enforcement Level |
|---------------|------|-------------------|
| **Approval** | Transfer requires explicit approval from recipient | API + Business Logic |
| **Commission** | `from_keeps_percentage` determines transferor retention (0-100%) | Database Check Constraint |
| **Snapshot** | `from_commission_total` preserves commission at request time | Application Logic |
| **Zero Transfer** | Allowed when `from_keeps_percentage = 0` (primary status only) | Application Logic |
| **Auto-Cancel** | Pending transfers cancelled when stakeholders removed/reduced | Database Triggers |
| **Transfer Types** | Supports user↔user, user↔team, team↔team scenarios | Application Logic |
| **Uniqueness** | Only one pending transfer per entity at any time | Database Unique Index |
| **Audit Trail** | All changes logged to `entity_stakeholder_history` | Application Logic |
| **Lifecycle** | Transfers cancelled on entity state changes (conversion, closure) | Application Logic |
| **Integration** | Property interests and related data preserved through transfers | Application Logic |
| **Organization** | All parties must belong to same organization | API Validation |
| **Permissions** | Requestor must have transfer permission on entity | API Authorization |
| **Immutability** | Final states (approved/rejected/cancelled) cannot be changed | Application Logic |
