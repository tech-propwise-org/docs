The transfer system manages formal transfers of leads and deals between agents or teams, with an approval workflow and commission splitting that integrates with the broader stakeholder system and commission payments within the CRM architecture.

## Architecture integration

The transfer system is a core component of the CRM module that works within the Person + Contact model architecture:

<Tabs>
<Tab title="CRM core integration">
**Entity relationships:**
- Transfers operate on `Lead` and `Deal` entities (not `Person` directly)
- Integrates with unified stakeholder model via `entity_stakeholder`
- Uses polymorphic patterns for entity_type/entity_id references
- Maintains organization-scoped data boundaries

**Data flow:**
```
Person → Contact/Lead → Deal → Transfer → Stakeholder Updates → Commission Payments
```
</Tab>

<Tab title="Stakeholder system">
**Core integration points:**
- Modifies `entity_stakeholder` records on approval
- Creates `entity_stakeholder_history` for audit trail
- Preserves commission percentages and primary status
- Auto-copies stakeholder arrangements from leads to deals
</Tab>

<Tab title="Real estate module">

- Preserves `site_visit` attribution through Person relationships

- Maintains connection to originating `lead_property_interest`

## entity_transfer

```
entity_transfer
├── Entity reference:
│   ├── entity_type (lead | deal)
│   └── entity_id
├── Transfer parties:
│   ├── from_user_id (nullable), from_team_id (nullable)
│   └── to_user_id (nullable), to_team_id (nullable)
├── Commission handling:
│   ├── from_commission_total (snapshot of sender's commission at request time)
│   └── from_keeps_percentage (e.g., 30%)
├── Workflow tracking:
│   ├── status (pending, approved, rejected, cancelled)
│   ├── reason, reject_reason
│   ├── requested_by_id, requested_at
│   ├── approved_by, approved_at, rejected_by, rejected_at
├── Audit fields:
│   ├── created_at, updated_at
│   └── organization_id
└── Database constraints:
    ├── Unique partial index: (entity_type, entity_id) WHERE status = 'pending'
    ├── Check: Either from_user_id OR from_team_id (not both)
    ├── Check: Either to_user_id OR to_team_id (not both)
    ├── Check: from_keeps_percentage BETWEEN 0 AND 100
    └── Check: from_commission_total >= 0
```

<Info>
The unique partial index ensures only one pending transfer can exist per entity at a time. This prevents conflicting transfer requests and maintains system integrity while allowing multiple historical transfers per entity.
</Info>

## Approval workflow system

The transfer system includes a comprehensive approval workflow that integrates with CRM validation and business rules:

<Steps>
<Step title="Transfer request">
Agent A requests transfer of Lead #1 to Agent B. An `entity_transfer` record is created with `from_commission_total` (snapshot) and `from_keeps_percentage`.

<Note>
The system validates that the transfer is valid (cannot transfer to self, must have valid from/to targets) and ensures only one pending transfer exists per entity via the unique partial index constraint. Commission snapshot protects against concurrent stakeholder modifications.
</Note>
</Step>

<Step title="Pending review">
Agent B (or manager) reviews the request. They can approve with `approved_by`/`approved_at`, reject with `rejected_by`/`rejected_at`/`reject_reason`, or cancel.

<Warning>
**Auto-cancel triggers:** If while a transfer is pending, any stakeholder on the entity is removed or has their commission reduced below the `from_commission_total`, the transfer is automatically cancelled because the commission split is now invalid. This includes entity lifecycle changes (lead conversion, deal closure) and user/team deactivation.
</Warning>
</Step>

<Step title="Transfer approval">
On approval, the system performs atomic updates:
- Agent A's stakeholder: commission reduced to `from_keeps_percentage`, `isPrimary=false` 
- Agent B's stakeholder: created with remaining commission, `isPrimary=true`
- History records created in `entity_stakeholder_history` for both agents
- Transfer status set to `approved`
- Commission payment records updated to reflect new arrangement
- Activity log entries created for audit trail
</Step>

<Step title="Commission splitting execution">
The commission split calculation uses:
```
to_commission = from_commission_total × (100% - from_keeps_percentage) / 100%
```

System performs atomic stakeholder updates:
- Transferor's commission adjusted to `from_keeps_percentage`
- Recipient receives calculated remaining commission
- Primary stakeholder flag transferred appropriately
- All changes wrapped in database transaction
</Step>

<Step title="Stakeholder history creation">
Complete audit trail is created:
- `entity_stakeholder_history` records for both parties
- Activity log entries for transfer completion
- Commission payment schedule updates
- Integration with broader CRM activity timeline
</Step>

<Step title="Downstream propagation">
When a new deal is created from the transferred lead, all lead stakeholders (including transfer results) are auto-copied to the deal with their commission splits preserved, maintaining ownership continuity throughout the sales pipeline.
</Step>
</Steps>

## Commission splitting calculations

The commission split is calculated using the formula:

```
to_commission = from_commission_total × (100% - from_keeps_percentage) / 100%
```

<Tabs>
<Tab title="Standard transfer">
- Agent A has 100% commission on a lead
- Agent A transfers to Agent B, keeping 30%
- Result: Agent A keeps 30%, Agent B gets 70%
- Formula: `70% = 100% × (100% - 30%) / 100%`
</Tab>

<Tab title="Partial commission transfer">
- Agent A has 60% commission on a deal (other stakeholders exist)
- Agent A transfers to Agent B, keeping 20%
- Result: Agent A keeps 20%, Agent B gets 40%
- Formula: `40% = 60% × (100% - 33.33%) / 100%` (keeping 20% of 60% = 33.33%)
</Tab>

<Tab title="Zero-commission transfer">
- Agent A has 50% commission, wants to transfer primary status only
- Agent A transfers to Agent B, keeping 0%
- Result: Agent A keeps 0%, Agent B gets 50% and becomes primary
- Special case: Used for primary status transfers without commission change
</Tab>
</Tabs>

<AccordionGroup>
<Accordion title="Commission snapshot integrity">
**Purpose:** The `from_commission_total` field captures the exact commission percentage at request time, ensuring calculations remain accurate even if stakeholder percentages change during the pending period.

**Business rationale:** Prevents situations where:
- Other stakeholder changes affect the promised commission split
- System modifications invalidate the transfer terms
- Concurrent operations create inconsistent commission states

**Implementation:** Database triggers and application logic protect this snapshot throughout the transfer lifecycle.
</Accordion>

<Accordion title="Stakeholder updates integration">
**Atomic operations:** Transfer approval creates or updates `entity_stakeholder` records atomically:
- Transferor's commission adjusted to `from_keeps_percentage`
- Recipient receives calculated remaining commission percentage
- Primary stakeholder flag transferred appropriately
- All changes wrapped in database transaction

**Commission payment integration:**
- Creates new `commission_payment` records for transferred amounts
- Updates future payment schedules and recipient allocation
- Preserves historical payment records (no retroactive changes)
- Maintains commission liability accounting across stakeholders
</Accordion>
</AccordionGroup>

<Tip>
The `from_commission_total` field captures the transferor's commission percentage at the time of request, ensuring the calculation remains valid even if stakeholder percentages change later due to other system operations or concurrent transfers.
</Tip>

## Auto-cancel behavior

The system implements comprehensive auto-cancel behavior through database triggers and application logic to maintain transfer integrity:

<AccordionGroup>
<Accordion title="Stakeholder removal triggers">
**Trigger condition:** Any stakeholder on the entity is completely removed from `entity_stakeholder`.

**Implementation:**
```sql
CREATE OR REPLACE FUNCTION cancel_transfers_on_stakeholder_removal()
RETURNS trigger AS $$
BEGIN
  UPDATE entity_transfer 
  SET status = 'cancelled', 
      updated_at = NOW(),
      reject_reason = 'Stakeholder removed from entity'
  WHERE entity_type = OLD.entity_type 
    AND entity_id = OLD.entity_id 
    AND status = 'pending';
  RETURN OLD;
END;
$$ LANGUAGE plpgsql;
```

**Business rationale:** The commission split calculation becomes invalid if the transferor is no longer a stakeholder, making the promised transfer impossible to fulfill.
</Accordion>

<Accordion title="Commission reduction triggers">
**Trigger condition:** Any stakeholder has their commission percentage reduced below what was captured in `from_commission_total`.

**Implementation:**
```sql
CREATE OR REPLACE FUNCTION cancel_transfers_on_commission_reduction()
RETURNS trigger AS $$
BEGIN
  UPDATE entity_transfer 
  SET status = 'cancelled', 
      updated_at = NOW(),
      reject_reason = 'Insufficient commission remaining'
  WHERE entity_type = NEW.entity_type 
    AND entity_id = NEW.entity_id 
    AND status = 'pending'
    AND from_commission_total > NEW.commission_percentage;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;
```

**Business rationale:** The promised commission split can no longer be fulfilled if the source commission is insufficient.
</Accordion>

<Accordion title="Entity lifecycle triggers">
**Application-level cancellation scenarios:**
- Lead converted to deal (pending lead transfers cancelled)
- Deal closed/won/lost (pending deal transfers cancelled)  
- Entity soft-deleted or archived
- Entity moved to non-transferable status

**Implementation example:**
```javascript
// In LeadService.convertToDeal()
await this.transferService.cancelPendingTransfers('lead', leadId, {
  reason: 'Lead converted to deal',
  cancelledBy: userId
});

// In DealService.markClosed()
await this.transferService.cancelPendingTransfers('deal', dealId, {
  reason: 'Deal closed',
  cancelledBy: userId
});
```

**Business rationale:** Transfers become meaningless once the entity has moved beyond the transferable stage in the CRM lifecycle.
</Accordion>

<Accordion title="User and team deactivation">
**Application-level cascade operations:**
```javascript
// In UserService.deactivateUser()
await this.transferService.cancelTransfersForUser(userId, {
  reason: 'User deactivated',
  cancelledBy: adminUserId
});

// In TeamService.deactivateTeam()
await this.transferService.cancelTransfersForTeam(teamId, {
  reason: 'Team deactivated', 
  cancelledBy: adminUserId
});
```

**Business rationale:** Prevents orphaned transfers and maintains system integrity when users/teams are no longer available to complete the transfer workflow.
</Accordion>

<Accordion title="Concurrent modification protection">
**Optimistic locking:** Transfer approval checks entity state immediately before applying changes:
```javascript
// Verify entity hasn't changed state since transfer request
const currentEntity = await this.getEntityWithLock(entityType, entityId);
if (currentEntity.lastModified > transfer.requestedAt) {
  await this.cancelTransfer(transferId, 'Entity modified during transfer');
  throw new ConflictError('Entity was modified during transfer');
}
```

**Commission validation:** Real-time validation of commission availability:
```javascript
// Verify transferor still has sufficient commission
const currentStakeholder = await this.getStakeholder(entityType, entityId, fromUserId);
if (currentStakeholder.commissionPercentage < transfer.fromCommissionTotal) {
  await this.cancelTransfer(transferId, 'Insufficient commission remaining');
  throw new ValidationError('Commission no longer available');
}
```
</Accordion>
</AccordionGroup>

## Business rules and validation

The transfer system enforces comprehensive business rules through multiple layers:

<AccordionGroup>
<Accordion title="Transfer eligibility rules">
**Organizational boundaries:**
- All parties must belong to the same organization
- Cross-organization transfers blocked at API level
- Organization-scoped data access enforced

**Entity state requirements:**
- Can only transfer active leads/deals (not archived/deleted)
- Entity must be in transferable status
- Lead conversion in progress blocks transfers
- Closed deals cannot be transferred

**Stakeholder prerequisites:**
- Transferor must be current stakeholder with sufficient commission
- Cannot transfer more commission than currently held
- Minimum commission thresholds (configurable per organization)
- Target must have appropriate permissions for entity type
</Accordion>

<Accordion title="Commission and financial rules">
**Commission integrity:**
- `from_commission_total` preserves exact commission at request time
- `from_keeps_percentage` must be between 0-100%
- Resulting commission split must sum correctly with other stakeholders
- No negative commission amounts allowed

**Team distribution rules:**
- Team commission distributed according to team member allocation rules
- Team member changes don't affect pending transfers
- Team-to-team transfers follow organizational hierarchy rules
- Commission caps and minimums apply to team distributions
</Accordion>

<Accordion title="Workflow and approval rules">
**Approval requirements:**
- All transfers require explicit approval from recipient
- Self-approval blocked (users cannot approve own outbound transfers)
- Manager override capabilities with proper authorization
- Timeout handling with configurable auto-expiration periods

**Status transition rules:**
- Pending → Approved: requires recipient approval
- Pending → Rejected: requires rejection reason
- Pending → Cancelled: can be system-triggered or manual
- Final states (approved/rejected/cancelled) are immutable
</Accordion>

<Accordion title="Data integrity and audit rules">
**Immutable audit trail:**
- Transfer records are never deleted, only status-updated
- Complete audit trail for all status changes and reasons
- All approvals, rejections, and cancellations logged with user attribution
- Historical data preserved for compliance and analytics

**Data consistency enforcement:**
- Atomic updates across stakeholder and transfer tables
- Commission totals validated before and after operations
- Referential integrity maintained across entity relationships
- Soft delete patterns preserve data relationships
</Accordion>

<Accordion title="Entity lifecycle validation">
**Lead-specific rules:**
- Cannot transfer leads that are currently being converted to deals
- Lead qualification status restrictions (configurable per organization)
- Source channel attribution preserved through transfer process
- Marketing campaign attribution maintained for commission tracking

**Deal-specific rules:**
- Cannot transfer deals in final stages (won, lost, cancelled)
- Deal value changes during transfer require re-approval
- Property interest associations maintained through transfer
- Contract and legal document ownership properly reassigned
</Accordion>
</AccordionGroup>

## Query patterns

The transfer system supports several key query patterns for efficient data access and reporting:

<AccordionGroup>
<Accordion title="Pending transfers for approval">
```sql
-- Get all pending transfers requiring user's approval
SELECT t.*, 
       e_from.full_name as from_name,
       e_to.full_name as to_name,
       CASE t.entity_type 
         WHEN 'lead' THEN l.title 
         WHEN 'deal' THEN d.title 
       END as entity_title,
       CASE t.entity_type
         WHEN 'lead' THEN l.status
         WHEN 'deal' THEN d.stage
       END as entity_status
FROM entity_transfer t
LEFT JOIN users e_from ON t.from_user_id = e_from.id
LEFT JOIN users e_to ON t.to_user_id = e_to.id  
LEFT JOIN teams tf ON t.from_team_id = tf.id
LEFT JOIN teams tt ON t.to_team_id = tt.id
LEFT JOIN leads l ON t.entity_type = 'lead' AND t.entity_id = l.id
LEFT JOIN deals d ON t.entity_type = 'deal' AND t.entity_id = d.id
WHERE t.status = 'pending'
  AND (t.to_user_id = ? OR t.to_team_id IN (?))
  AND t.organization_id = ?
ORDER BY t.requested_at DESC;
```
</Accordion>

<Accordion title="Transfer history for entity">
```sql
-- Get complete transfer history for a lead/deal with full context
SELECT t.*,
```

<Accordion title="Commission impact analysis">
```sql
-- Calculate commission impact of pending transfers by entity type
SELECT t.entity_type,
       COUNT(*) as pending_count,
       SUM(t.from_commission_total) as total_commission_at_risk,
       SUM(t.from_commission_total * t.from_keeps_percentage / 100) as total_retained,
       SUM(t.from_commission_total * (100 - t.from_keeps_percentage) / 100) as total_transferred,
       AVG(t.from_keeps_percentage) as avg_retention_rate

WHERE t.status = 'pending'
  AND t.organization_id = ?

```
</Accordion>

<Accordion title="Auto-cancel trigger queries">
```sql
-- Find transfers that should be cancelled due to insufficient commission
SELECT t.id, t.entity_type, t.entity_id, 
       t.from_commission_total, es.commission_percentage
FROM entity_transfer t
JOIN entity_stakeholder es ON t.entity_type = es.entity_type 
  AND t.entity_id = es.entity_id
  AND ((t.from_user_id IS NOT NULL AND es.user_id = t.from_user_id)
    OR (t.from_team_id IS NOT NULL AND es.team_id = t.from_team_id))

  AND es.commission_percentage < t.from_commission_total
  AND t.organization_id = ?;
```
</Accordion>

<Accordion title="Stakeholder query patterns">
```sql
-- Get current stakeholders for entity with transfer context
SELECT es.*, u.full_name as user_name, t.name as team_name,

       et.status as transfer_status
```

<Accordion title="Performance monitoring queries">
```sql
-- Transfer approval time analytics
SELECT entity_type,
       AVG(EXTRACT(EPOCH FROM (approved_at - requested_at))/3600) as avg_approval_hours,
       PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY EXTRACT(EPOCH FROM (approved_at - requested_at))/3600) as median_approval_hours,

  AND approved_at IS NOT NULL
  AND organization_id = ?

  AND organization_id = ?
  AND requested_at > NOW() - INTERVAL '30 days'
GROUP BY reject_reason
ORDER BY rejection_count DESC;
```
</Accordion>
</AccordionGroup>

**Race condition prevention:**
- Unique constraints prevent duplicate pending transfers
- Database-level commission percentage bounds checking
- Atomic compare-and-swap operations for critical updates
- Queue-based processing for high-volume scenarios
</Accordion>
</AccordionGroup>

## Database constraints and indexes

The transfer system uses comprehensive database constraints and indexes to maintain data integrity and performance:

<AccordionGroup>
<Accordion title="Unique partial index">
**Index definition:**
```sql
CREATE UNIQUE INDEX CONCURRENTLY idx_entity_transfer_unique_pending 
ON entity_transfer (entity_type, entity_id) 
WHERE status = 'pending';
```

**Purpose:** Ensures only one pending transfer can exist per entity at any time, preventing conflicting transfer requests and race conditions.

**Implementation details:**
- PostgreSQL partial index applies only to records with `status = 'pending'`
- Allows multiple historical transfers (approved/rejected/cancelled) per entity
- Supports high-concurrency scenarios with minimal lock contention
- Automatically maintained by PostgreSQL with MVCC consistency
</Accordion>

<Accordion title="Foreign key constraints">
**Entity relationships:**
- `entity_id` → polymorphic reference to `Lead.id` or `Deal.id`
- `from_user_id` → `User.id` (nullable with appropriate cascade rules)
- `from_team_id` → `Team.id` (nullable with appropriate cascade rules)
- `to_user_id` → `User.id` (nullable with appropriate cascade rules)
- `to_team_id` → `Team.id` (nullable with appropriate cascade rules)

**Audit relationships:**
- `requested_by_id` → `User.id` (required, no cascade to preserve history)
- `approved_by` → `User.id` (nullable, no cascade to preserve history)
- `rejected_by` → `User.id` (nullable, no cascade to preserve history)
- `organization_id` → `Organization.id` (required, restrict cascade)
</Accordion>

<Accordion title="Check constraints">
**Data validation constraints:**
```sql
-- Mutual exclusivity for from party
CHECK ((from_user_id IS NULL) != (from_team_id IS NULL))

-- Mutual exclusivity for to party  
CHECK ((to_user_id IS NULL) != (to_team_id IS NULL))

-- Commission percentage bounds
CHECK (from_keeps_percentage >= 0 AND from_keeps_percentage <= 100)

-- Commission total non-negative
CHECK (from_commission_total >= 0)

-- Entity type validation
CHECK (entity_type IN ('lead', 'deal'))

-- Status validation
CHECK (status IN ('pending', 'approved', 'rejected', 'cancelled'))
```
</Accordion>

<Accordion title="Performance indexes">
**Query optimization indexes:**
```sql
-- Pending transfers lookup
CREATE INDEX idx_entity_transfer_pending_to 
ON entity_transfer (to_user_id, to_team_id, organization_id) 
WHERE status = 'pending';

-- Entity history lookup
CREATE INDEX idx_entity_transfer_entity_history 
ON entity_transfer (entity_type, entity_id, created_at DESC);

-- User transfer history
CREATE INDEX idx_entity_transfer_user_activity 
ON entity_transfer (requested_by_id, created_at DESC);

-- Organization analytics
CREATE INDEX idx_entity_transfer_org_analytics 
ON entity_transfer (organization_id, status, entity_type, requested_at);

-- Commission impact queries
CREATE INDEX idx_entity_transfer_commission_analysis
ON entity_transfer (organization_id, status, from_commission_total, from_keeps_percentage)
WHERE status IN ('pending', 'approved');
```
</Accordion>
</AccordionGroup>

## Error handling and edge cases

<AccordionGroup>
<Accordion title="Validation errors">
**Invalid transfer targets (400 Bad Request):**
- Cannot transfer to self
- Target user/team must exist and be active
- Must specify either user_id or team_id (not both) for from/to
- Commission percentages must be valid (0-100%)

**Permission errors (403 Forbidden):**
- Requestor must have transfer permissions on the entity
- Organization boundaries must be respected
- Entity must be in transferable state

**Conflict errors (409 Conflict):**
- Only one pending transfer allowed per entity
- Concurrent modification detected during approval
</Accordion>

<Accordion title="Concurrent modification handling">
**Race conditions:** The unique partial index prevents multiple pending transfers, but the system handles concurrent stakeholder modifications gracefully by auto-cancelling affected transfers.

**Stale data protection:** Commission snapshots protect against stale percentage calculations, but the system validates current state before applying approved transfers with optimistic locking.

**Atomic operations:** Transfer approval uses database transactions to ensure stakeholder updates are applied atomically or rolled back completely.
</Accordion>

<Accordion title="Orphaned transfers">
**Entity deletion cleanup:** If an entity is soft-deleted while transfers are pending, cleanup jobs automatically cancel all related transfers with appropriate audit trails.

**User deactivation handling:** If a user involved in a pending transfer is deactivated, the transfer is automatically cancelled with proper notification to affected parties.

**Team restructuring:** When teams are dissolved or restructured, pending team transfers are cancelled or redirected according to business rules.
</Accordion>

<Accordion title="Commission calculation edge cases">
**Zero commission transfers:** Allowed when `from_keeps_percentage = 0` and target is already a stakeholder. This transfers primary status without moving commission percentage.

**Insufficient commission scenarios:** If stakeholder's commission drops below `from_commission_total` while transfer is pending, the transfer is auto-cancelled with detailed audit trail.

**Rounding precision:** Commission calculations use decimal precision with proper rounding rules to avoid floating-point errors in financial calculations.

**Team commission distribution:** When transferring to/from teams, commission is distributed according to team-specific rules and member allocation percentages.
</Accordion>

<Accordion title="Integration failure scenarios">
**Commission payment system failures:** If commission payment record creation fails during transfer approval, the entire transfer transaction is rolled back to maintain consistency.

**Activity logging failures:** Transfer operations continue even if activity logging fails, but failure alerts are generated for audit trail completeness.

**Real estate module integration:** Property interest associations are preserved even if related real estate operations fail, ensuring CRM data integrity.

**Notification system failures:** Transfer approvals proceed even if email/SMS notifications fail, with retry mechanisms for important status updates.
</Accordion>
</AccordionGroup>

## Validation rules

| Rule                          | Level      | Description                                                     |
| ----------------------------- | ---------- | --------------------------------------------------------------- |
| Cannot transfer to self       | API        | User cannot transfer entity to themselves                       |
| Valid from/to targets         | API        | Must specify valid user_id or team_id for both from and to     |
| One pending per entity        | Database   | Only one pending transfer allowed per entity (unique constraint) |
| Organization scope            | API        | All parties must be in same organization                        |
| Entity exists and accessible  | API        | Target entity (lead/deal) must exist and be accessible         |
| Stakeholder permissions       | API        | Requestor must have permission to transfer the entity          |
| Commission bounds             | Database   | from_keeps_percentage must be 0-100%, from_commission_total >= 0 |
| Status transitions            | API        | Valid status transition paths enforced                          |
| Mutual exclusivity            | Database   | Either user_id OR team_id for from/to (not both, not neither)  |

## Business rules summary

| Rule                 | Description                                                                                            |
| -------------------- | ------------------------------------------------------------------------------------------------------ |
| Approval required    | Transfer requires explicit approval from target user/team                                              |
| Commission preserved | `from_keeps_percentage` determines what transferor retains                                             |
| Commission snapshot  | `from_commission_total` snapshots sender's commission at request time                                  |
| Zero transfer        | Allowed when `from_keeps_percentage = 0` (transfers primary status without moving commission)          |
| Auto-cancel          | Pending transfers auto-cancelled when stakeholders removed or commission reduced below snapshot        |
| Team distribution    | Supports user→user, user→team, team→user, team→team with appropriate commission handling               |
| Single pending       | Only one pending transfer per entity (enforced by unique partial index)                               |
| Immutable history    | All changes logged to `entity_stakeholder_history` and activity streams                               |
| Entity lifecycle     | Transfers cancelled when entities change state (conversion, closure, deletion)                         |
| Cross-module impact  | Property interests, site visits, and other related data maintained through transfers                  |
