The transfer system manages formal transfers of leads and deals between agents or teams, with an approval workflow and commission splitting that integrates with the broader stakeholder system and commission payments.

## entity_transfer

```
entity_transfer
├── entity_type (lead | deal)
├── entity_id
├── from_user_id (nullable), from_team_id (nullable)
├── to_user_id (nullable), to_team_id (nullable)
├── Commission handling:
│   ├── from_commission_total (snapshot of sender's commission at request time)
│   └── from_keeps_percentage (e.g., 30%)
├── Workflow:
│   ├── status (pending, approved, rejected, cancelled)
│   ├── reason, reject_reason
│   ├── requested_by_id, requested_at
│   ├── approved_by, approved_at, rejected_by, rejected_at
├── created_at, updated_at
├── organization_id
└── Database constraints:
    └── Unique partial index: (entity_type, entity_id) WHERE status = 'pending'
```

<Info>
The unique partial index ensures only one pending transfer can exist per entity at a time. This prevents conflicting transfer requests and maintains system integrity.
</Info>

## Transfer types

| From | To   | Description                      |
| ---- | ---- | -------------------------------- |
| User | User | Standard agent-to-agent transfer |
| User | Team | Agent transfers to a team pool   |
| Team | User | Team pool claim by a user        |
| Team | Team | Team-to-team reallocation        |

## Transfer request logic

The transfer system includes a comprehensive request workflow that tracks who initiated the transfer and when:

<AccordionGroup>
<Accordion title="Request initiation">
When a transfer is initiated, the system captures:
- `requested_by_id`: The user who initiated the transfer request
- `requested_at`: Timestamp of the request
- `reason`: Optional reason provided by the requestor
- `from_commission_total`: Snapshot of current commission percentage
</Accordion>

<Accordion title="Request validation">
The system validates:
- Requestor has permission to transfer the entity
- Target user/team exists and is valid
- No conflicting pending transfers exist (enforced by unique constraint)
- All parties are within the same organization
</Accordion>

<Accordion title="Commission snapshot">
The `from_commission_total` field captures the exact commission percentage at request time, ensuring calculations remain accurate even if stakeholder percentages change during the pending period.
</Accordion>
</AccordionGroup>

## Transfer workflow

<Steps>
<Step title="Transfer request">
Agent A requests transfer of Lead #1 to Agent B. An `entity_transfer` record is created with `from_commission_total` (snapshot) and `from_keeps_percentage`.

<Note>
The system validates that the transfer is valid (cannot transfer to self, must have valid from/to targets) and ensures only one pending transfer exists per entity via the unique partial index constraint.
</Note>
</Step>

<Step title="Pending review">
Agent B (or manager) reviews the request. They can approve with `approved_by`/`approved_at`, reject with `rejected_by`/`rejected_at`/`reject_reason`, or cancel.

<Warning>
**Auto-cancel:** If while a transfer is pending, any stakeholder on the entity is removed or has their commission reduced below the `from_commission_total`, the transfer is automatically cancelled because the commission split is now invalid.
</Warning>
</Step>

<Step title="Transfer approved">
On approval:
- Agent A's stakeholder is updated: `30%`, `isPrimary=false`
- Agent B's stakeholder is created: `isPrimary=true`, `70%`
- History records created in `entity_stakeholder_history` for both agents
- Transfer status set to `approved`
- Commission payment records updated to reflect new stakeholder arrangement
</Step>

<Step title="Downstream effects">
When a new deal is created from the lead later, all lead stakeholders are auto-copied to the deal with their commission splits preserved, maintaining the transfer results throughout the sales pipeline.
</Step>
</Steps>

## Commission calculation

The commission split is calculated using the formula:

```
to_commission = from_commission_total × (100% - from_keeps_percentage)
```

**Example:**
- Agent A has 100% commission on a lead
- Agent A transfers to Agent B, keeping 30%
- Result: Agent A keeps 30%, Agent B gets 70%

<Tip>
The `from_commission_total` field captures the transferor's commission percentage at the time of request, ensuring the calculation remains valid even if stakeholder percentages change later due to other system operations.
</Tip>

## System integration

The transfer system integrates deeply with other CRM components:

<Tabs>
<Tab title="Stakeholder system">
Transfers modify the `entity_stakeholder` table, creating new stakeholder records and updating existing ones. All changes are tracked in `entity_stakeholder_history` for audit purposes.
</Tab>

<Tab title="Commission payments">
When transfers are approved, the `commission_payment` system is updated to reflect the new stakeholder arrangement. This ensures future commission calculations use the correct percentages.
</Tab>

<Tab title="Activity tracking">
Transfer events are logged in the activity system, providing a complete audit trail of ownership changes and the business context behind them.
</Tab>
</Tabs>

## Auto-cancel triggers and implementation

Pending transfers are automatically cancelled through system triggers when:

<AccordionGroup>
<Accordion title="Stakeholder removal trigger">
**When:** Any stakeholder on the entity (lead/deal) is completely removed from the stakeholder list.

**Implementation:** Database trigger on `entity_stakeholder` DELETE operations checks for pending transfers on the same entity and automatically updates their status to 'cancelled'.

**Rationale:** The commission split calculation becomes invalid if the transferor is no longer a stakeholder.
</Accordion>

<Accordion title="Commission reduction trigger">
**When:** Any stakeholder has their commission percentage reduced below what was captured in `from_commission_total`.

**Implementation:** Database trigger on `entity_stakeholder` UPDATE operations compares new commission percentage against pending transfer `from_commission_total` values.

**Rationale:** The promised commission split can no longer be fulfilled if the source commission is insufficient.
</Accordion>

<Accordion title="Entity lifecycle triggers">
**When:** The underlying entity (lead/deal) changes to a status that would invalidate the transfer (e.g., lead converted to deal, deal closed).

**Implementation:** Application-level logic in entity status change handlers checks for pending transfers and cancels them when appropriate.

**Rationale:** Transfers become meaningless once the entity has moved beyond the transferable stage.
</Accordion>
</AccordionGroup>

## Error handling and edge cases

<AccordionGroup>
<Accordion title="Validation errors">
**Invalid transfer targets:** Returns 400 error with specific validation message
- Cannot transfer to self
- Target user/team must exist and be active
- Must specify either user_id or team_id (not both) for from/to

**Permission errors:** Returns 403 error
- Requestor must have transfer permissions on the entity
- Organization boundaries must be respected
</Accordion>

<Accordion title="Concurrent modification">
**Race conditions:** The unique partial index prevents multiple pending transfers, but the system handles concurrent stakeholder modifications gracefully by auto-cancelling affected transfers.

**Stale data:** Commission snapshots protect against stale percentage calculations, but the system validates current state before applying approved transfers.
</Accordion>

<Accordion title="Orphaned transfers">
**Entity deletion:** If an entity is deleted while transfers are pending, cleanup jobs automatically cancel all related transfers.

**User deactivation:** If a user involved in a pending transfer is deactivated, the transfer is automatically cancelled.
</Accordion>
</AccordionGroup>

## Validation rules

| Rule                          | Description                                                     |
| ----------------------------- | --------------------------------------------------------------- |
| Cannot transfer to self       | User cannot transfer entity to themselves                       |
| Valid from/to targets         | Must specify valid user_id or team_id for both from and to     |
| One pending per entity        | Only one pending transfer allowed per entity (database constraint) |
| Organization scope            | All parties must be in same organization                        |
| Entity exists                 | Target entity (lead/deal) must exist and be accessible         |
| Stakeholder permissions       | Requestor must have permission to transfer the entity          |

## Business rules

| Rule                 | Description                                                                                            |
| -------------------- | ------------------------------------------------------------------------------------------------------ |
| Approval required    | Transfer requires target user/team approval                                                            |
| Commission preserved | `from_keeps_percentage` sets what transferor keeps                                                     |
| Commission snapshot  | `from_commission_total` snapshots sender's commission at request time                                  |
| 0% transfer          | Allowed only when target is already a stakeholder (transfers primary status without moving commission) |
| Auto-cancel          | Pending transfers auto-cancelled when a stakeholder is removed or has commission reduced               |
| Team transfers       | Supports user→user, user→team, team→user, team→team                                                    |
| One pending          | Only one pending transfer per entity (enforced by unique partial index)                                |
| History recorded     | All changes logged to `entity_stakeholder_history`                                                     |
