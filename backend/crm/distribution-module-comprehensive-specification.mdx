## Overview

The Distribution Module automates lead assignment within organizations. When a new lead is created, the system evaluates org-defined rules to automatically assign the lead to the most appropriate agent — based on lead attributes, agent availability, language compatibility, and capacity.

<Note>
**Module Status:** Active — fully implemented  
**Module Path:** `src/modules/crm/distribution/`
</Note>

### Design Principles

| Principle | Decision |
|-----------|----------|
| Async distribution | `createLead()` emits `LEAD_CREATED`; a pg-boss worker handles distribution — lead creation is never blocked |
| Stakeholder system reuse | Distribution creates `EntityStakeholder` records via `EntityStakeholderService`, not a new paradigm |
| First-match-wins rules | Rules are evaluated top-to-bottom by priority; the first matching rule wins |
| Idempotency | Distribution engine checks for existing stakeholders or pending offers before running |
| No retroactive distribution | Existing leads are unaffected when rules are created; only new leads trigger distribution |
| pg-boss scheduling | Distribution queue uses pg-boss for reliability and retry guarantees |
| RLS compliance | All entities carry `organization_id` for row-level security |

### Distribution Paths

The engine supports two execution paths:

<Tabs>
<Tab title="Path A - Org-level Distribution">
**Function:** `runDistribution`

**Triggered when:** A lead enters the org with no team context

**Process:**
- Evaluates org-scoped rules
- Applies the org default method
- Can bridge to Path B if a rule or default method routes to a team that has `distributionEnabled = true`
</Tab>

<Tab title="Path B - Team-level Distribution">
**Function:** `runTeamDistribution`

**Triggered when:**
- A lead is created with a `teamId` in the event payload (team pool assignment)
- Path A determines the lead belongs to an auto-distributing team
- Idempotency check finds a single team-only stakeholder with auto-distribute enabled

**Process:**
- Evaluates team-scoped rules
- Uses team settings (with org fallback for capacity)
- Logs the team FK on the resulting `DistributionLog` record
</Tab>
</Tabs>

### Component Responsibilities

| Component | Responsibility |
|-----------|----------------|
| **DistributionEngine** | Orchestrator: receives a lead, evaluates rules, selects agent, creates assignment. Supports Path A (org) and Path B (team). |
| **RuleEvaluator** | Evaluates rule conditions against lead data; returns first matching rule |
| **LanguageMatcher** | Filters and ranks agents by language compatibility with the lead's person |
| **AgentSelector** | Applies the distribution method (round-robin, weighted, weighted-round-robin, direct) to the filtered agent pool |
| **AgentAvailabilityService** | Checks agent capacity, business hours, leave status. Two-phase capacity enforcement with advisory locks. |
| **UserStatusService** | Pre-filters candidate agents to only those with ONLINE status |
| **DistributionListener** | Listens for `LEAD_CREATED` events and enqueues pg-boss jobs |
| **DistributionJobHandler** | pg-boss worker that processes distribution jobs |

<Warning>
**Master toggle behavior:**
- `distributionEnabled = false` (new-org default): Engine is off. `DistributionListener` and `LeadImportService` skip enqueue entirely — leads go to pool, no pg-boss jobs created.
- `distributionEnabled = true`: Engine is active. When toggled from `false` → `true` in `DistributionSettingsService.update()`, if `defaultMethod` is still `POOL` it is auto-upgraded to `ROUND_ROBIN` for a smooth first-run experience.
</Warning>

<Info>
**Business hours source:** Business hours schedule (timezone, weekly slots, enabled flag) is stored on `Organization.settings.businessHours` (`BusinessHoursConfig`), not on `DistributionSettings`. The `respectBusinessHours` field on this entity only controls whether the distribution engine gates against that org-level schedule.
</Info>

#### Rule Conditions — Supported Fields

| Field | Operator(s) | Example Value |
|-------|-------------|---------------|
| `leadSource` | `eq`, `in` | `'WEBSITE'`, `['WEBSITE', 'REFERRAL']` |
| `temperature` | `eq`, `in` | `'HOT'` |
| `language` | `eq` | `'ar'` (matched against `person.preferredLanguage`) |
| `budget` | `gte`, `lte`, `between` | `500000` |
| `tags` | `contains` | `['vip']` |
| `sourceChannel` | `eq`, `in` | `'WHATSAPP'` |
| `intent` | `eq` | `'BUY'` |
| `area` | `eq`, `in`, `contains` | `'Dubai Marina'`, `['JBR', 'Downtown Dubai']` |

<Note>
All string-based condition fields use **case-insensitive matching**. The `area` field requires data from `LeadPropertyInterest.preferredAreas[]` — the engine pre-loads the lead's property interests before calling the evaluator.
</Note>

**Scope & team rules:**
- **Org-level rules** (`scope = ORGANIZATION`): `team` is null. Evaluated during Path A.
- **Team-scoped rules** (`scope = TEAM`): `team` is set. Only evaluated during Path B for the matching team.

## Distribution Engine

### Core Engine Logic

The distribution engine operates through two main execution paths:

<Steps>
<Step title="Event Trigger">
Lead creation emits `LEAD_CREATED` event, captured by `DistributionListener`
</Step>

<Step title="Job Enqueuing">
If distribution is enabled, a pg-boss job is enqueued for asynchronous processing
</Step>

<Step title="Path Selection">
Engine determines execution path:
- **Path A (Org-level):** No team context or general org distribution
- **Path B (Team-level):** Team-specific distribution
</Step>

<Step title="Rule Evaluation">
Rules are evaluated in priority order (ascending), first match wins
</Step>

<Step title="Agent Selection">
Selected distribution method is applied to filtered agent pool
</Step>

<Step title="Assignment Creation">
`EntityStakeholder` record is created via `EntityStakeholderService`
</Step>
</Steps>

### Rule Evaluation Process

<CodeGroup>
```typescript Rule Evaluation
async evaluateRules(lead: Lead, scope: RuleScope, teamId?: string): Promise<DistributionRule | null> {
  const rules = await this.getRulesForEvaluation(lead.organizationId, scope, teamId);
  
  for (const rule of rules) {
    if (await this.evaluateRule(rule, lead)) {
      return rule;
    }
  }
  
  return null;
}

private async evaluateRule(rule: DistributionRule, lead: Lead): Promise<boolean> {
  // OR logic between condition groups
  for (const group of rule.conditionGroups) {
    if (await this.evaluateConditionGroup(group, lead)) {
      return true;
    }
  }
  return false;
}

private async evaluateConditionGroup(group: ConditionGroup, lead: Lead): Promise<boolean> {
  // AND logic within condition group
  for (const condition of group.conditions) {
    if (!await this.evaluateCondition(condition, lead)) {
      return false;
    }
  }
  return true;
}
```
</CodeGroup>

### Agent Selection Methods

<AccordionGroup>
<Accordion title="Round Robin">
Sequential assignment using an atomic cursor. The cursor is incremented after each assignment to ensure fair distribution.

```typescript
async selectAgentRoundRobin(agents: User[], cursor: number): Promise<User> {
  if (agents.length === 0) throw new Error('No agents available');
  
  const index = cursor % agents.length;
  return agents[index];
}
```
</Accordion>

<Accordion title="Weighted Selection">
Random selection based on agent weights. Higher weights increase selection probability.

```typescript
async selectAgentWeighted(agents: User[], weights: Record<string, number>): Promise<User> {
  const weightedAgents = agents.map(agent => ({
    agent,
    weight: weights[agent.id] || 1
  }));
  
  const totalWeight = weightedAgents.reduce((sum, item) => sum + item.weight, 0);
  const random = Math.random() * totalWeight;
  
  let currentWeight = 0;
  for (const item of weightedAgents) {
    currentWeight += item.weight;
    if (random <= currentWeight) {
      return item.agent;
    }
  }
  
  return weightedAgents[0].agent;
}
```
</Accordion>

<Accordion title="Weighted Round Robin">
Combines round-robin fairness with weighted distribution. Uses weight-adjusted cursor advancement.

```typescript
async selectAgentWeightedRoundRobin(
  agents: User[], 
  weights: Record<string, number>, 
  cursor: number
): Promise<User> {
  const weightedAgents = this.createWeightedSequence(agents, weights);
  const index = cursor % weightedAgents.length;
  return weightedAgents[index];
}
```
</Accordion>

<Accordion title="Direct Assignment">
Explicit assignment to a specific agent. Used when rules specify exact agent targeting.

```typescript
async selectAgentDirect(agentId: string): Promise<User> {
  const agent = await this.userService.findById(agentId);
  if (!agent) {
    throw new Error(`Direct assignment target not found: ${agentId}`);
  }
  return agent;
}
```
</Accordion>
</AccordionGroup>

## pg-boss Job Configuration

The distribution system uses pg-boss for reliable, asynchronous job processing.

### Job Configuration

<CodeGroup>
```typescript Job Setup
const JOB_NAME = 'lead-distribution';
const RETRY_LIMIT = 3;
const RETRY_DELAY = 5; // seconds
const EXPIRE_IN = '1 hour';

interface DistributionJobPayload {
  leadId: string;
  organizationId: string;
  teamId?: string;
  source: 'LEAD_CREATED' | 'TEAM_POOL' | 'MANUAL';
}
```

```typescript Job Handler
@OnWorkerEvent({ name: JOB_NAME })
async handleDistributionJob(job: Job<DistributionJobPayload>) {
  const { leadId, organizationId, teamId, source } = job.data;
  
  try {
    if (teamId) {
      await this.distributionEngine.runTeamDistribution(leadId, teamId);
    } else {
      await this.distributionEngine.runDistribution(leadId);
    }
    
    this.logger.log(`Distribution completed for lead ${leadId}`);
  } catch (error) {
    this.logger.error(`Distribution failed for lead ${leadId}`, error);
    throw error; // pg-boss will handle retry
  }
}
```
</CodeGroup>

### Retry Strategy

<Steps>
<Step title="Initial Attempt">
Job is processed immediately when enqueued
</Step>

<Step title="Failure Handling">
On failure, job is retried up to 3 times with exponential backoff
</Step>

<Step title="Dead Letter Queue">
After 3 failures, job moves to failed state for manual inspection
</Step>

<Step title="Monitoring">
Failed jobs trigger alerts and are logged for investigation
</Step>
</Steps>

## Edge Case Handling

### No Available Agents

<Steps>
<Step title="Check Capacity">
All agents at capacity or offline
</Step>

<Step title="Business Hours Check">
If outside hours and `respectBusinessHours = true`
</Step>

<Step title="Apply Outside Hours Action">
- `QUEUE`: Hold lead until business hours
- `POOL`: Send to unassigned pool
- `DUTY_AGENT`: Assign to designated duty agent
</Step>

<Step title="Fallback to Pool">
If duty agent unavailable, always fallback to pool
</Step>
</Steps>

### Rule Conflicts

<Warning>
Rules are evaluated in priority order (ascending). The first matching rule wins, preventing conflicts. If multiple rules have the same priority, database ordering determines precedence.
</Warning>

### Circular Team Routing

<Steps>
<Step title="Detection">
Track teams visited during distribution chain
</Step>

<Step title="Prevention">
Prevent routing to already-visited teams
</Step>

<Step title="Fallback">
Break cycle by routing to org pool
</Step>
</Steps>

### Capacity Race Conditions

<CodeGroup>
```typescript Advisory Locking
async checkAndReserveCapacity(agentId: string): Promise<boolean> {
  return this.em.getConnection().execute(`
    SELECT pg_advisory_lock(hashtext($1));
    
    WITH capacity_check AS (
      SELECT 
        COUNT(*) as current_active,
        (SELECT max_active_leads_per_agent FROM distribution_settings 
         WHERE organization_id = $2) as max_active
      FROM entity_stakeholders es
      WHERE es.user_id = $1 
        AND es.entity_type = 'LEAD'
        AND es.status = 'ACTIVE'
    )
    SELECT 
      CASE WHEN current_active < max_active THEN 1 ELSE 0 END as available
    FROM capacity_check;
    
    SELECT pg_advisory_unlock(hashtext($1));
  `, [agentId, organizationId]);
}
```
</CodeGroup>

### Database Optimizations

<CodeGroup>
```sql Key Indexes
-- Distribution rules evaluation
CREATE INDEX idx_distribution_rules_org_priority

CREATE INDEX idx_distribution_rules_team_scope 
ON distribution_rules(team_id, scope, priority) 
WHERE scope = 'TEAM' AND is_active = true AND is_deleted = false;

-- Agent capacity queries
CREATE INDEX idx_entity_stakeholders_agent_active 
ON entity_stakeholders(user_id, entity_type, status);

CREATE INDEX idx_distribution_log_org_created 
ON distribution_log(organization_id, created_at);
```

```sql Partial Indexes
-- Active rules only
CREATE INDEX idx_distribution_rules_active 
ON distribution_rules(organization_id, priority) 
WHERE is_active = true AND is_deleted = false;

-- Online agents only  
CREATE INDEX idx_users_online_agents 
ON users(organization_id, id) 
WHERE status = 'ONLINE' AND role = 'AGENT';
```
</CodeGroup>

### Caching Strategy

<Tabs>
<Tab title="Redis Caching">
- **Agent availability**: 30-second TTL
- **Business hours**: 1-hour TTL
- **Distribution settings**: 5-minute TTL
- **Rule evaluation results**: No caching (rules change frequently)
</Tab>

<Tab title="In-Memory Caching">
- **Round-robin cursors**: Cached for 1 minute to reduce DB writes
- **Team membership**: 5-minute TTL
- **User language preferences**: 10-minute TTL
</Tab>
</Tabs>

async handleLeadCreated(payload: LeadCreatedEvent) {
