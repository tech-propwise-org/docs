# Lead Capture Unified Specification v4

<Note>
**Purpose:** One universal, source-agnostic pipeline that turns any **captured** lead — Property Finder, Bayut/dubizzle, and later Meta Ads, website forms, etc. — into a CRM `Lead` with consistent **source fields**, **person reuse**, **deduplication**, and **assignment**.
</Note>

**Module:** `src/modules/crm/lead-capture/` (new).

**Core principle — reuse, don't rebuild.** This layer sits **above** the existing CRM primitives and orchestrates them. It does NOT reimplement lead creation, person matching, dedup, stakeholders, or distribution. Every source funnels through `LeadCaptureService.capture()`.

**Related docs:** `CRM_MODULE_SPECIFICATION.md`, `DISTRIBUTION_MODULE_SPECIFICATION.md`, `STAKEHOLDER_SYSTEM.md`, `PORTAL_SYNDICATION_SPECIFICATION.md`, `BAYUT_DUBIZZLE_PULL_API.md`, `PORTAL_FIELD_MAPPING_MATRIX.md`.

<Check>
**Implementation status: BUILT.** Module `src/modules/crm/lead-capture/` ships `LeadCaptureService` (`capture()` pipeline), `CapturedLeadInput`/`CaptureResult` contracts, `LeadCaptureSource` interface + `LeadCaptureSourceRegistry` (adapters self-register in `onModuleInit`), `LeadCaptureSettings` (org default) + `CapturedLead` ledger (idempotency, RLS), and the source-agnostic **`lead-ingestion`** pg-boss queue + `LeadIngestionWorker`.
</Check>

## Scope and non-goals

### In scope
- A canonical captured-lead contract
- A source-adapter pattern
- Orchestration of source-field mapping + identity resolution + dedup/attach + assignment
- Async transport (queue + worker) that feeds the pipeline

### Out of scope
<Warning>
Meta Ads and website-form ingestion are **not** designed here. The contract simply reserves the fields they will need (`externalLeadId`, `assignmentHint`) so they plug in later without a refactor. Outbound listing syndication lives in `PORTAL_SYNDICATION_SPECIFICATION.md`.
</Warning>

## Canonical contract

<CodeGroup>
```typescript CapturedLeadInput
interface CapturedLeadInput {
  orgId: string;
  source: LeadSource;                  // Bayut | PropertyFinder | Dubizzle | Website | Instagram | ...
  sourceDetail?: string;               // campaign / form / "WhatsApp message"
  inboundChannel?: InboundChannel;     // call | whatsapp | email | sms — the delivery channel of THIS inquiry
  referenceId?: string;                // listing / campaign / form id — attribution + dedup key
  externalLeadId?: string;             // the source's own lead id — hard idempotency key

  identity: {                          // for Person reuse, in match-priority order
    firstName?: string;
    lastName?: string;
    email?: string;
    phone?: string;
    channel?: { type: ChannelType; identifier: string };  // WhatsApp number / IG id / Messenger id
  };

  intent?: LeadIntent;
  propertyRef?: { listingId?: string; refUnitId?: number };  // resolves the attributed agent
  assignmentHint?: CaptureAssignmentConfig;                  // adapter's suggestion
  capturedAt?: Date;
  rawPayload: unknown;                 // stored verbatim for audit / replay
}
```

```typescript CaptureResult
interface CaptureResult {
  leadId: string;
  personId: string;
  outcome: 'created' | 'attached' | 'skipped';   // attached = re-inquiry on an open lead
  assignment: 'agent' | 'team_pool' | 'org_pool' | 'queued' | 'none';
}
```

```typescript LeadCaptureSource
interface LeadCaptureSource {
  readonly source: LeadSource;
  toCapturedInput(raw: unknown, ctx: { orgId: string }): CapturedLeadInput | CapturedLeadInput[];
}
```
</CodeGroup>

## Pipeline - LeadCaptureService.capture()

Runs entirely inside `tenantContext.executeInOrg(input.orgId)` with the **system actor** as `createdBy` (`resolveSystemActorId` → the per-org **System service user**).

<Tip>
**System actor availability:** `resolveSystemActorId` now returns the dedicated per-org System service user (`isSystemServiceUser = true`), which is always seeded at org provisioning and backfilled at startup for existing orgs. **Memberless orgs no longer skip capture**; their leads are captured and attributed to the System actor instead of being dropped.
</Tip>

<Steps>
<Step title="Idempotency guard">
Look up the `captured_lead` ledger by `(orgId, source, externalLeadId)`. If present → return the prior `CaptureResult` (no-op). This makes webhook retries and poll overlaps safe.
</Step>

<Step title="Source-field normalization">
Derive `leadSource`, `sourceDetail`, `referenceId`, and any `sourceChannel`/`sourceConversation`.
</Step>

<Step title="Identity resolution & person reuse">
Resolve/create the `Person` using existing person services.
</Step>

<Step title="Dedup / attach">
Decide create-new vs attach-to-open-lead using existing deduplication logic.
</Step>

<Step title="Assignment">
Resolve the directive and execute (stakeholder or distribution).
</Step>

<Step title="Create or attach + record ledger">
Create via `createLeadInTransaction` (or attach a re-inquiry activity), write the `captured_lead` ledger row (`status`, `leadId`, `personId`, `rawPayload`), and return `CaptureResult`.
</Step>
</Steps>

Failures inside a job bubble to the worker so pg-boss retries; the ledger guard makes retries idempotent.

### Read query - findCapturedLeadsForListing()

A read-only projection over the `captured_lead` ledger that powers the portal-syndication **listing detail "Leads" tab**. 

<Info>
Listing-targeted portal leads set `captured_lead.referenceId` to the local listing id. The method filters by `referenceId = listingId`, `source IN (PropertyFinder, Bayut, Dubizzle)`, and `leadId != null` (only `created`/`attached` rows), newest first.
</Info>

The resolved `Person` is loaded separately and the result is mapped to `ListingCapturedLead` with fields:
- `capturedLeadId`
- `leadId?`
- `personId?`  
- `source`
- `inboundChannel?`
- `fullName`
- `email?`
- `phone?`
- `outcome`
- `capturedAt?`

## Source-field policy

| Field | Rule |
|---|---|
| `Lead.leadSource` | `input.source` — the source of THIS inquiry (the portal/origin) |
| `Lead.sourceDetail` | `input.sourceDetail` (campaign name, "WhatsApp message", entity/agent/agency target, etc.). Human-readable; **not** the structured channel |
| `Lead.inboundChannel` | `input.inboundChannel` — the **structured delivery channel** of the inquiry (`InboundChannel`: `call \| whatsapp \| email \| sms`), distinct from `leadSource`. First-touch only; queryable/indexed |
| `Lead.referenceId` | `input.referenceId` (listing/campaign/form id). Attribution + part of the dedup key. Not unique |
| `Lead.sourceChannel` | the resolved `PersonChannel` when the lead arrived through a persistent messaging identity (WhatsApp/IG/Messenger) |
| `Lead.sourceConversation` | set when captured from a messaging conversation |
| `Person.originalSource` | set on FIRST creation only (first-touch attribution); never overwritten on later inquiries |

<Note>
**InboundChannel vocabulary** (`src/modules/shared/inbound-channel.enum.ts`): `email | whatsapp | call | sms`, stored as plain `text`. 

**Portal mapping:** 
- **Property Finder** `payload.channel` (`email | whatsapp | call`) → 1:1
- **Bayut/dubizzle** inferred from contact method in lead data
</Note>

## Identity resolution and person reuse

<Tabs>
<Tab title="Match priority order">
1. **Channel identity** (`PersonChannel`) — WhatsApp number, Instagram ID, Messenger ID
2. **Email** (exact match, case-insensitive) 
3. **Phone** (normalized international format)
4. **Name + fuzzy match** (fallback for poor data quality)
</Tab>

<Tab title="Creation logic">
If no existing person is found:
- Create new `Person` with `originalSource` = `input.source` (first-touch attribution)
- Set all available identity fields (name, email, phone, channels)
- Follow existing `PersonService.findOrCreatePersonFullInTransaction` logic
</Tab>
</Tabs>

## Deduplication and attachment

The system uses existing `LeadService.findDuplicateLeadMatchInTransaction` to determine if an incoming lead should:

<CardGroup cols={2}>
<Card title="Create new lead" icon="plus">
No existing open lead found for this person + reference combination
</Card>

<Card title="Attach to existing" icon="link">
Open lead exists - add as re-inquiry activity instead of creating duplicate
</Card>
</CardGroup>

<Warning>
**Dedup key components:**
- Person identity (resolved in previous step)
- Reference ID (listing/campaign/form)
- Lead status (only attach to open leads, not closed ones)
</Warning>

## Assignment strategies

<AccordionGroup>
<Accordion title="Agent assignment (listing-based)">
When `propertyRef.listingId` is provided:
1. Resolve listing agent via `LISTING_AGENT_RESOLVER` seam
2. Check agent is active in organization
3. Create direct stakeholder assignment
4. Bypass distribution engine
</Accordion>

<Accordion title="Team pool assignment">
When `assignmentHint` specifies team:
1. Validate team membership
2. Trigger distribution engine with team constraint
3. Follow existing distribution rules within team
</Accordion>

<Accordion title="Organization pool (default)">
When no specific assignment hint:
1. Use organization-wide distribution settings
2. Apply standard distribution rules (round-robin, capacity-based, etc.)
3. Fall back to organization pool if no rules match
</Accordion>

<Accordion title="Queued assignment">
When assignment fails or is deferred:
1. Mark lead as `queued` assignment status
2. Allow manual assignment later
3. Trigger alerts for unassigned leads
</Accordion>
</AccordionGroup>

## Transport and queue processing

The system uses pg-boss queue `lead-ingestion` for reliable, asynchronous processing:

```typescript
// Queue job payload
interface LeadIngestionJob {
  capturedLeadInput: CapturedLeadInput;
  metadata: {
    sourceAdapter: string;
    attemptNumber: number;
    enqueuedAt: Date;
  };
}
```

<Steps>
<Step title="Enqueue">
Source adapters enqueue `CapturedLeadInput` to `lead-ingestion` queue
</Step>

<Step title="Process">
`LeadIngestionWorker` dequeues and calls `LeadCaptureService.capture()`
</Step>

<Step title="Retry">
Failures trigger pg-boss retry with exponential backoff
</Step>

<Step title="Monitor">
Queue depth and failure rates are monitored for operational health
</Step>
</Steps>

## Source adapter pattern

Source modules implement `LeadCaptureSource` interface and self-register:

<CodeGroup>
```typescript Interface
interface LeadCaptureSource {
  readonly source: LeadSource;
  toCapturedInput(raw: unknown, ctx: { orgId: string }): CapturedLeadInput | CapturedLeadInput[];
}
```

```typescript Registration
// In source module's onModuleInit()
constructor(
  private leadCaptureRegistry: LeadCaptureSourceRegistry
) {}

onModuleInit() {
  this.leadCaptureRegistry.register(new BayutLeadCaptureAdapter());
  this.leadCaptureRegistry.register(new PfLeadCaptureAdapter());
}
```

```typescript Usage
// Portal webhook handler
@Post('/webhook/bayut')
async handleBayutWebhook(@Body() payload: unknown, @Query('orgId') orgId: string) {
  const adapter = this.registry.getSource('Bayut');
  const inputs = adapter.toCapturedInput(payload, { orgId });
  
  for (const input of Array.isArray(inputs) ? inputs : [inputs]) {
    await this.leadCaptureService.enqueue(input);
  }
  
  return { status: 'accepted' };
}
```
</CodeGroup>
