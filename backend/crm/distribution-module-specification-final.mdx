## Overview

The Distribution Module automates lead assignment within organizations. When a new lead is created, the system evaluates org-defined rules to automatically assign the lead to the most appropriate agent — based on lead attributes, UserStatus online/away state, working-hours eligibility, language compatibility, and capacity.

### Design Principles

| Principle | Decision |
|-----------|----------|
| **Async distribution** | `createLead()` emits `LEAD_CREATED` after commit; a pg-boss worker handles distribution. Listener / emit failures are logged only — HTTP lead creation still returns success; manual assignment or backfill may be needed if enqueue never ran. |
| **Stakeholder system reuse** | Distribution creates `EntityStakeholder` records via `EntityStakeholderService`, not a new paradigm |
| **First-match-wins rules** | Rules are evaluated top-to-bottom by priority; the first matching rule wins |
| **Idempotency** | Distribution engine checks for existing stakeholders or pending offers before running |
| **No retroactive distribution** | Existing leads are unaffected when rules are created; only new leads trigger distribution |
| **pg-boss scheduling** | Distribution queue uses pg-boss for reliability and retry guarantees |
| **RLS compliance** | All entities carry `organization_id` for row-level security |

### Distribution Paths

<Tabs>
<Tab title="Path A — Org-level">
**Triggered when:** A lead enters the org with no team context

- Evaluates org-scoped rules
- Applies the org default method
- Can bridge to Path B if a rule or default method routes to a team that has `distributionEnabled = true`
</Tab>

<Tab title="Path B — Team-level">
**Triggered when:**
- A lead is created with a `teamId` in the event payload (team pool assignment)
- A bulk-imported lead has a team-only assignment
- Path A determines the lead belongs to an auto-distributing team
- Idempotency check finds a single team-only stakeholder with auto-distribute enabled

Evaluates team-scoped rules, uses team settings (with org fallback for capacity), and logs the team FK on the resulting `DistributionLog` record.
</Tab>
</Tabs>

### Component Responsibilities

<AccordionGroup>
<Accordion title="Core Engine Components">
| Component | Responsibility |
|-----------|----------------|
| **DistributionEngine** | Orchestrator: receives a lead, evaluates rules, selects agent, creates assignment. Supports Path A (org) and Path B (team). |
| **RuleEvaluator** | Evaluates rule conditions against lead data; returns first matching rule |
| **LanguageMatcher** | Filters and ranks agents by language compatibility with the lead's person |
| **AgentSelector** | Applies the distribution method (round-robin, weighted, weighted-round-robin, direct) to the filtered agent pool |
</Accordion>

<Accordion title="Supporting Services">
| Component | Responsibility |
|-----------|----------------|
| **DistributionCapacityService** | Two-phase capacity enforcement: Phase 1 `filterByCapacity()` (lead counts vs limits); Phase 2 `confirmCapacityAndAssign()` (advisory locks + atomic stakeholder creation) |
| **UserStatusService** | Pre-filters candidate agents to ONLINE status; filters by per-user working hours; provides `isWithinWorkingHours()` for org-level business hours check |
| **DistributionListener** | Listens for `LEAD_CREATED` events and enqueues pg-boss jobs. Fault-isolated with try/catch for error logging |
| **DistributionJobHandler** | pg-boss worker that processes distribution jobs |
</Accordion>
</AccordionGroup>

## Distribution Engine

### Core Engine Flow

<Steps>
<Step title="Idempotency Check">
Verify lead doesn't already have assignments or pending distribution
</Step>
<Step title="Settings Validation">
Confirm distribution is enabled for org/team and within business hours
</Step>
<Step title="Rule Evaluation">
Apply rules in priority order, return first match
</Step>
<Step title="Agent Pool Assembly">
Gather candidate agents based on rule/default targeting
</Step>
<Step title="Agent Filtering">
Apply status, working hours, capacity, and language filters
</Step>
<Step title="Agent Selection">
Use distribution method to select final agent
</Step>
<Step title="Assignment Creation">
Create EntityStakeholder record and log distribution attempt
</Step>
</Steps>

### Distribution Methods

<AccordionGroup>
<Accordion title="ROUND_ROBIN">
**Logic:** Selects the agent with the oldest `last_assigned_at` timestamp

**Use case:** Equal distribution across all agents

```typescript
// Simplified logic
const sortedAgents = eligibleAgents.sort((a, b) => 
  (a.lastAssignedAt || new Date(0)).getTime() - 
  (b.lastAssignedAt || new Date(0)).getTime()
);
return sortedAgents[0];
```
</Accordion>

<Accordion title="WEIGHTED">
**Logic:** Random selection based on agent weights (higher weight = higher probability)

**Use case:** Senior agents get more leads than junior agents

```typescript
// Weighted random selection
const totalWeight = agents.reduce((sum, agent) => sum + agent.weight, 0);
const randomValue = Math.random() * totalWeight;
// Select agent based on cumulative weight ranges
```
</Accordion>

<Accordion title="WEIGHTED_ROUND_ROBIN">
**Logic:** Combines round-robin fairness with weighted preferences

**Use case:** Balanced distribution that still respects agent capabilities

```typescript
// Each agent gets opportunities proportional to weight
// but ensures fair rotation over time
```
</Accordion>

<Accordion title="DIRECT">
**Logic:** Assigns to specific agents listed in `target_agents`

**Use case:** Route high-value leads to specific senior agents

```typescript
const availableTargets = targetAgents.filter(agent => 
  agent.isEligible && hasCapacity(agent)
);
return availableTargets[0]; // First available
```
</Accordion>
</AccordionGroup>

### Language Matching

<Tip>
Language matching improves lead-agent compatibility by considering language preferences.
</Tip>

<Steps>
<Step title="Extract Lead Languages">
Get languages from `person.preferred_languages` or infer from location
</Step>
<Step title="Filter Compatible Agents">
Find agents with overlapping languages in their profiles
</Step>
<Step title="Apply Strict Mode">
If `language_matching_strict = true`, only return exact matches
If `false`, fall back to all agents when no matches found
</Step>
<Step title="Rank by Compatibility">
Prioritize agents with more language overlaps
</Step>
</Steps>

### Job Handler

<CodeGroup>

```typescript Job Processing
@Injectable()
export class DistributionJobHandler {
  async processDistribution(job: Job<DistributionJobData>) {
    const { leadId, organizationId, teamId } = job.data;
    
    try {
      if (teamId) {
        // Path B: Team-level distribution
        await this.engine.runTeamDistribution(leadId, teamId);
      } else {
        // Path A: Org-level distribution
        await this.engine.runDistribution(leadId, organizationId);
      }
      
      return { success: true };
    } catch (error) {
      this.logger.error('Distribution job failed', {
        leadId,
        organizationId,
        teamId,
        error: error.message,
        stack: error.stack
      });
      throw error; // Allows pg-boss retry
    }
  }
}
```

```typescript Job Enqueueing
// Single lead distribution
await this.pgBoss.send('lead-distribution', {
  leadId: lead.id,
  organizationId: lead.organizationId,
  teamId: lead.teamId // optional
});

// Bulk lead distribution
await this.pgBoss.sendBatch('lead-distribution', 
  leads.map(lead => ({
    data: {
      leadId: lead.id,
      organizationId: lead.organizationId,
      teamId: lead.teamId
    }
  }))
);
```

## Edge Case Handling

<AccordionGroup>
<Accordion title="No Eligible Agents">
**Scenario:** All agents are offline, at capacity, or outside working hours

**Handling:**
1. Log `NO_ELIGIBLE_AGENTS` status
2. Create distribution log entry with failure reason
3. Lead remains unassigned for manual intervention
4. Optional: Send notification to administrators
</Accordion>

<Accordion title="Concurrent Distribution">
**Scenario:** Multiple leads distributed simultaneously might exceed agent capacity

**Handling:**
1. Use advisory locks in `confirmCapacityAndAssign()`
2. Atomic capacity check + stakeholder creation
3. First transaction wins, others fail gracefully
4. Failed attempts logged and potentially retried
</Accordion>

<Accordion title="Agent Becomes Unavailable">
**Scenario:** Agent goes offline between selection and assignment

**Handling:**
1. Assignment still proceeds (agent may return online)
2. Real-time notifications may fail gracefully
3. Agent will see assignment when they return
4. No automatic reassignment to avoid conflicts
</Accordion>

<Accordion title="Rule Conflicts">
**Scenario:** Multiple rules could match the same lead

**Handling:**
1. Rules evaluated in strict priority order
2. First matching rule wins
3. Remaining rules ignored for that lead
4. Clear priority numbers prevent ambiguity
</Accordion>

<Accordion title="Business Hours Boundary">
**Scenario:** Lead created just before business hours end

**Handling:**
1. Distribution queued but delayed until next business day
2. Job retry logic respects business hours
3. Lead appears in "pending distribution" dashboards
4. Manual override available for urgent leads
</Accordion>
</AccordionGroup>

### Batch Processing

For high-volume scenarios:

```typescript
// Bulk distribution with batching
async enqueueBatch(leads: Lead[], batchSize = 100) {
  for (let i = 0; i < leads.length; i += batchSize) {
    const batch = leads.slice(i, i + batchSize);
    const jobs = batch.map(lead => ({
      data: {
        leadId: lead.id,
        organizationId: lead.organizationId,
        teamId: lead.teamId
      }
    }));
    
    await this.pgBoss.sendBatch('lead-distribution', jobs);
    
    // Rate limiting
    if (i + batchSize < leads.length) {
      await new Promise(resolve => setTimeout(resolve, 100));
    }
  }
}
```

## Integration Points

### Event System

<CodeGroup>

```typescript Events Emitted
// Lead creation triggers distribution
this.eventEmitter.emit('lead.created', {
  leadId: lead.id,
  organizationId: lead.organizationId,
  teamId: lead.teamId,
  skipEmitLeadCreated: false // Set to true for bulk imports
});

// Distribution completion
this.eventEmitter.emit('distribution.completed', {
  leadId,
  assignedUserId,
  distributionMethod,
  ruleId
});
```

```typescript Events Consumed
// Listen for lead creation
@OnEvent('lead.created')
async handleLeadCreated(payload: LeadCreatedEvent) {
  if (payload.skipEmitLeadCreated) return;
  
  try {
    await this.enqueueLead(payload);
  } catch (error) {
    this.logger.error('Failed to enqueue distribution', {
      leadId: payload.leadId,
      error: error.message
    });
  }
}
```

</CodeGroup>

### External Dependencies

| Service | Usage | Fallback Behavior |
|---------|--------|------------------|
| **UserStatusService** | Agent availability filtering | Skip status filter, log warning |
| **EntityStakeholderService** | Lead assignment creation | Fail distribution, retry job |
| **NotificationService** | Assignment notifications | Log failure, continue |
| **AuditService** | Activity logging | Best effort, don't block |
