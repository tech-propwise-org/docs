title: "Person"
description: "Central identity layer for all individuals — stores personal details, communication channels, deduplication overrides, and merge logic."
---

Person is the single source of truth for human identity and preferences. It is the hidden layer — not exposed directly in the frontend. Every lead and contact links to a Person.

## Person entity

```
Person
├── Identity: first_name, last_name, avatar_url, title
├── Demographics: date_of_birth
├── Social: website, linkedin_url, twitter_url
├── Preferences: preferred_contact_method, timezone, preferred_language
├── Communication flags: do_not_call, do_not_email
├── Source tracking: original_source, languages[]
├── Merge tracking: merged_into_id, merged_at, merged_by
├── Computed: full_name (getter: first_name + last_name)
└── Related tables:
    ├── person_email (multiple emails, one primary)
    ├── person_phone (multiple phones, one primary)
    ├── person_address (multiple addresses, one primary)
    ├── person_channel (WhatsApp, Instagram, etc.)
    └── person_not_duplicate (deduplication override pairs)
```

<Note>
  Person preferences apply across all contexts — leads, deals, and contacts. The
  `original_source` field is set once when the person first enters the system.
</Note>

## PersonChannel

Stores a person's communication channel identities (WhatsApp, Instagram, etc.).

```
person_channel
├── person_id → Person
├── channel_type (whatsapp, instagram, facebook, telegram, sms, webchat)
├── channel_identifier (phone number, username, PSID, etc.)
├── display_name, avatar_url
├── channel_identity_id → WhatsAppUser.id, InstagramUser.id, etc.
├── status (active, inactive, blocked, unsubscribed)
├── is_primary
├── Opt-in: marketing_opt_in, transactional_opt_in
├── Engagement: first_contact_at, last_message_at, message_count
└── Verification: is_verified, verified_at
```

<Info>
  Channels belong to Person, not Lead (person-centric architecture). A lead can
  reference `source_channel_id` for attribution. One person can have multiple
  channels of the same type.
</Info>

## PersonNotDuplicate

Records pairs of persons manually confirmed as NOT duplicates. Prevents repeated flagging by the deduplication system.

```
person_not_duplicate
├── person1_id → Person
├── person2_id → Person
├── marked_by → User
├── marked_at
├── organization_id → Organization
├── Unique constraint: (person1, person2, organization)
```

The relationship is symmetric — if (A, B) is marked as not-duplicate, (B, A) is treated equivalently. Each organization maintains its own override decisions.

## Person merge system

Consolidates duplicate persons into a single primary record, reassigning all related data.

**API endpoint:** `POST /persons/:primaryPersonId/merge`

<Steps>
<Step title="Validation">
Verify the primary person and all secondary persons exist, are not deleted, and belong to the same organization.
</Step>
<Step title="Field selection">
Accept `fieldSelections: Record<string, string>` (e.g., `{ "firstName": "primary", "lastName": "person-B-id" }`). For each field, pick the value from the specified source person. Fields not listed default to the primary person's values.
</Step>
<Step title="Contact info merge (opt-in per type)">
Three boolean flags control which contact info is reassigned from secondary persons to the primary:

- `mergeAllEmails` — reassign all secondary emails (`isPrimary` reset to `false`)
- `mergeAllPhones` — reassign all secondary phones (`isPrimary` reset to `false`)
- `mergeAllAddresses` — reassign all secondary addresses (`isPrimary` reset to `false`)

</Step>
<Step title="Channel merge (always)">
All channels from secondary persons are reassigned to the primary person (`isPrimary` reset to `false`). This is not optional since channels represent unique identities.
</Step>
<Step title="Lead and ownership reassignment">
All leads pointing to secondary persons are bulk-reassigned to the primary person via `nativeUpdate`. All `InventoryUnitOwnership` records are similarly reassigned.
</Step>
<Step title="Mark secondaries">
Each secondary person is marked with `mergedIntoId`, `mergedAt`, `mergedBy`, and soft-deleted (`isDeleted = true`).
</Step>
</Steps>

<Warning>
  Merge is **irreversible** — secondary persons are soft-deleted. Contact
  entities are NOT directly reassigned; since Contact has a `OneToOne` to
  Person, the primary person retains its contact if one exists.
</Warning>

## Person creation and deduplication

When a lead is created, the system resolves or creates a Person:

```typescript
async function findOrCreatePerson(data: CreateLeadDto) {
  // 1. Check by email
  const existingByEmail = await personEmailRepo.findOne({
    email: data.email,
    organization: data.organizationId,
  });
  if (existingByEmail) return existingByEmail.person;

  // 2. Check by phone
  const existingByPhone = await personPhoneRepo.findOne({
    phone_number: data.phone,
    organization: data.organizationId,
  });
  if (existingByPhone) return existingByPhone.person;

  // 3. Create new person + contact info
  const person = await personRepo.create({ ... });
  await personEmailRepo.create({ person, email: data.email, is_primary: true });
  await personPhoneRepo.create({ person, phone_number: data.phone, is_primary: true });

  return person;
}
```

If `personId` is provided directly in the lead creation request, the system uses that existing person instead of searching.

## Import services

- **`LeadImportService`** — Handles bulk lead import with person matching/creation, deduplication by `referenceId + leadSource`, and stakeholder assignment.
- **`ContactImportService`** — Handles bulk contact import with person matching, uniqueness validation (one contact per person per organization), and activity backfill.
