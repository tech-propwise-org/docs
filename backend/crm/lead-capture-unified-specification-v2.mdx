# Lead Capture Unified Specification v2

<Note>
**Purpose:** One universal, source-agnostic pipeline that turns any **captured** lead — Property Finder, Bayut/dubizzle, and later Meta Ads, website forms, etc. — into a CRM `Lead` with consistent **source fields**, **person reuse**, **deduplication**, and **assignment**.
</Note>

**Module:** `src/modules/crm/lead-capture/` (new).

**Core principle — reuse, don't rebuild.** This layer sits **above** the existing CRM primitives and orchestrates them. It does NOT reimplement lead creation, person matching, dedup, stakeholders, or distribution. Every source funnels through `LeadCaptureService.capture()`.

<Info>
**Implementation status: BUILT.** Module `src/modules/crm/lead-capture/` ships `LeadCaptureService` (`capture()` pipeline), `CapturedLeadInput`/`CaptureResult` contracts, `LeadCaptureSource` interface + `LeadCaptureSourceRegistry` (adapters self-register in `onModuleInit`), `LeadCaptureSettings` (org default) + `CapturedLead` ledger (idempotency, RLS), and the source-agnostic **`lead-ingestion`** pg-boss queue + `LeadIngestionWorker`.
</Info>

## Scope & Non-goals

### In Scope
- A canonical captured-lead contract
- A source-adapter pattern
- Orchestration of source-field mapping + identity resolution + dedup/attach + assignment
- Async transport (queue + worker) that feeds the pipeline

### Out of Scope
<Warning>
Meta Ads and website-form ingestion are **not** designed here. The contract simply reserves the fields they will need (`externalLeadId`, `assignmentHint`) so they plug in later without a refactor. Outbound listing syndication lives in `PORTAL_SYNDICATION_SPECIFICATION.md`.
</Warning>

## Reused Existing Components

<Note>
**Do NOT rebuild these components** - the lead capture service orchestrates existing CRM primitives.
</Note>

| Concern | Existing Component | Location |
|---|---|---|
| Lead creation (composable) | `LeadService.createLeadInTransaction(em, data, createdById, orgId, { stakeholders?, skipDistribution?, systemInitiated? })` | `crm/lead/lead.service.ts` |
| Lead dedup match | `LeadService.findDuplicateLeadMatchInTransaction` | `crm/lead/lead.service.ts` |
| Person reuse | `PersonService.findOrCreatePersonFullInTransaction`, `findByEmailInTransaction`, `findByPhoneInTransaction` | `crm/person/person.service.ts` |
| Channel identity | `PersonChannel` (channelType + channelIdentifier) + messaging matchers | `crm/person/channel/`, `messaging/services/contact-matcher.service.ts` |
| Assignment | `EntityStakeholderService` (direct) + `DistributionEngineService` (rules/pool) via `LeadCreatedEvent` → `DistributionListener` | `crm/entity-stakeholder/`, `crm/distribution/` |
| Source vocabulary | `LeadSource` enum | `shared/lead-source.enum.ts` |
| System actor / audit | `resolveSystemActorId` (returns the per-org **System service user**), `SYSTEM_AUDIT_CONTEXT` | `shared/org-member.utils.ts`, `common/tenant/` |

Lead already carries the source fields: `leadSource`, `sourceDetail`, `referenceId`, `sourceChannel` (PersonChannel FK), `sourceConversation` (Conversation FK). Person carries `originalSource` (first-touch).

## Canonical Contract

### CapturedLeadInput Interface

```typescript
interface CapturedLeadInput {
  orgId: string;
  source: LeadSource;                  // Bayut | PropertyFinder | Dubizzle | Website | Instagram | ...
  sourceDetail?: string;               // campaign / form / "WhatsApp message"
  inboundChannel?: InboundChannel;     // call | whatsapp | email | sms — the delivery channel of THIS inquiry
  referenceId?: string;                // listing / campaign / form id — attribution + dedup key
  externalLeadId?: string;             // the source's own lead id — hard idempotency key

  identity: {                          // for Person reuse, in match-priority order
    firstName?: string;
    lastName?: string;
    email?: string;
    phone?: string;
    channel?: { type: ChannelType; identifier: string };  // WhatsApp number / IG id / Messenger id
  };

  intent?: LeadIntent;
  propertyRef?: { listingId?: string; refUnitId?: number };  // resolves the attributed agent
  assignmentHint?: CaptureAssignmentConfig;                  // adapter's suggestion
  capturedAt?: Date;
  rawPayload: unknown;                 // stored verbatim for audit / replay
}
```

### CaptureResult Interface

```typescript
interface CaptureResult {
  leadId: string;
  personId: string;
  outcome: 'created' | 'attached' | 'skipped';   // attached = re-inquiry on an open lead
  assignment: 'agent' | 'team_pool' | 'org_pool' | 'queued' | 'none';
}
```

### LeadCaptureSource Interface

```typescript
interface LeadCaptureSource {
  readonly source: LeadSource;
  toCapturedInput(raw: unknown, ctx: { orgId: string }): CapturedLeadInput | CapturedLeadInput[];
}
```

## Pipeline — LeadCaptureService.capture()

Runs entirely inside `tenantContext.executeInOrg(input.orgId)` with the **system actor** as `createdBy` (`resolveSystemActorId` → the per-org **System service user**).

<Check>
**System actor availability:** `resolveSystemActorId` now returns the dedicated per-org System service user (`isSystemServiceUser = true`), which is always seeded at org provisioning and backfilled at startup for existing orgs. **Memberless orgs no longer skip capture** — their leads are captured and attributed to the System actor.
</Check>

<Steps>
<Step title="Idempotency guard">
Look up the `captured_lead` ledger by `(orgId, source, externalLeadId)`. If present → return the prior `CaptureResult` (no-op). This makes webhook retries and poll overlaps safe.
</Step>

<Step title="Source-field normalization">
Derive `leadSource`, `sourceDetail`, `referenceId`, and any `sourceChannel`/`sourceConversation`.
</Step>

<Step title="Identity resolution & person reuse">
Resolve/create the `Person` using existing PersonService methods.
</Step>

<Step title="Dedup / attach">
Decide create-new vs attach-to-open-lead using existing deduplication logic.
</Step>

<Step title="Assignment">
Resolve the directive and execute (stakeholder or distribution).
</Step>

<Step title="Create or attach + record ledger">
Create via `createLeadInTransaction` (or attach a re-inquiry activity), write the `captured_lead` ledger row (`status`, `leadId`, `personId`, `rawPayload`), and return `CaptureResult`.
</Step>
</Steps>

<Warning>
Failures inside a job bubble to the worker so pg-boss retries; the ledger guard makes retries idempotent.
</Warning>

### Read Query — findCapturedLeadsForListing()

A read-only projection over the `captured_lead` ledger that powers the portal-syndication **listing detail "Leads" tab**.

<Tip>
Listing-targeted portal leads set `captured_lead.referenceId` to the local listing id. The method filters `referenceId = listingId`, `source IN (PropertyFinder, Bayut, Dubizzle)`, and `leadId != null` (only `created`/`attached` rows), newest first.
</Tip>

The resolved `Person` is loaded separately and the result is mapped to `ListingCapturedLead` with:
- `capturedLeadId`, `leadId?`, `personId?`
- `source`, `inboundChannel?` 
- `fullName`, `email?`, `phone?`
- `outcome`, `capturedAt?`

## Source-field Policy

| Field | Rule |
|---|---|
| `Lead.leadSource` | `input.source` — the source of THIS inquiry (the portal/origin) |
| `Lead.sourceDetail` | `input.sourceDetail` (campaign name, "WhatsApp message", etc.). Human-readable; **not** the structured channel |
| `Lead.inboundChannel` | `input.inboundChannel` — the **structured delivery channel** (`InboundChannel`: `call | whatsapp | email | sms`), distinct from `leadSource`. First-touch only; indexed for analytics |
| `Lead.referenceId` | `input.referenceId` (listing/campaign/form id). Attribution + part of the dedup key. Not unique |
| `Lead.sourceChannel` | the resolved `PersonChannel` when the lead arrived through a persistent messaging identity |
| `Lead.sourceConversation` | set when captured from a messaging conversation |
| `Person.originalSource` | set on FIRST creation only (first-touch attribution); never overwritten |

<Note>
**`InboundChannel` vocabulary** (`src/modules/shared/inbound-channel.enum.ts`): `email | whatsapp | call | sms`, stored as plain `text` (no DB CHECK constraint).

**Portal mapping:** 
- **Property Finder** `payload.channel` (`email | whatsapp | call`) → 1:1
- **Bayut/dubizzle** inquiry context → inferred channel mapping
</Note>
