## Overview

The Lead Capture module provides a universal, source-agnostic pipeline that transforms any captured lead from Property Finder, Bayut/dubizzle, Meta Ads, website forms, and other sources into a CRM `Lead` with consistent source fields, person reuse, deduplication, and assignment.

<Note>
**Core principle:** Reuse, don't rebuild. This layer sits above existing CRM primitives and orchestrates them without reimplementing lead creation, person matching, dedup, stakeholders, or distribution.
</Note>

## Module Architecture

The module is located at `src/modules/crm/lead-capture/` and follows a dependency rule where it only depends on intra-CRM services. Source modules depend on lead-capture, never the reverse.

### File Structure

```
crm/lead-capture/
├── lead-capture.service.ts              # LeadCaptureService.capture()
├── lead-capture-source.interface.ts     # LeadCaptureSource adapter contract

```

## Canonical Contract

### CapturedLeadInput Interface

```typescript
interface CapturedLeadInput {
  orgId: string;
  source: LeadSource;                  // Bayut | PropertyFinder | Dubizzle | Website | Instagram
  sourceDetail?: string;               // Campaign / form / "WhatsApp message"
  inboundChannel?: InboundChannel;     // call | whatsapp | email | sms
  referenceId?: string;                // Listing / campaign / form id
  externalLeadId?: string;             // Source's own lead id for idempotency

  identity: {                          // For Person reuse, in match-priority order
    firstName?: string;
    lastName?: string;
    email?: string;
    phone?: string;
    channel?: { type: ChannelType; identifier: string };
  };

  intent?: LeadIntent;
  propertyRef?: { listingId?: string; refUnitId?: number };
  assignmentHint?: CaptureAssignmentConfig;
  capturedAt?: Date;
  rawPayload: unknown;                 // Stored verbatim for audit/replay
}
```

### CaptureResult Interface

```typescript
interface CaptureResult {
  leadId: string;
  personId: string;
  outcome: 'created' | 'attached' | 'skipped';
  assignment: 'agent' | 'team_pool' | 'org_pool' | 'queued' | 'none';
}
```

### LeadCaptureSource Interface

```typescript
interface LeadCaptureSource {
  readonly source: LeadSource;
  toCapturedInput(raw: unknown, ctx: { orgId: string }): CapturedLeadInput | CapturedLeadInput[];
}
```

## Processing Pipeline

The `LeadCaptureService.capture(input)` method processes leads through six stages:

<Steps>
<Step title="Idempotency Guard">
Checks the `captured_lead` ledger by `(orgId, source, externalLeadId)`. If present, returns the prior `CaptureResult` to handle webhook retries and poll overlaps safely.
</Step>

<Step title="Source Field Normalization">
Derives `leadSource`, `sourceDetail`, `referenceId`, and any `sourceChannel`/`sourceConversation` from the input.
</Step>

<Step title="Identity Resolution">
Resolves or creates the `Person` using existing person matching services with priority-based matching.
</Step>

<Step title="Deduplication">
Decides whether to create a new lead or attach to an existing open lead using dedup matching logic.
</Step>

<Step title="Assignment">
Resolves assignment directives and executes through stakeholder or distribution services.
</Step>

<Step title="Create or Attach">
Creates the lead via `createLeadInTransaction` or attaches a re-inquiry activity, writes the ledger row, and returns the result.
</Step>
</Steps>

<Warning>
All processing runs within `tenantContext.executeInOrg(input.orgId)` with the system actor as `createdBy`. Failures bubble to the worker for pg-boss retry handling.
</Warning>

## Source Field Mapping

The module maps input fields to lead properties with specific policies:

| Field | Mapping Rule |
|-------|--------------|
| `Lead.leadSource` | Maps to `input.source` - the portal/origin of the inquiry |
| `Lead.sourceDetail` | Maps to `input.sourceDetail` - human-readable campaign/context |
| `Lead.inboundChannel` | Maps to `input.inboundChannel` - structured delivery channel (first-touch only) |
| `Lead.referenceId` | Maps to `input.referenceId` - attribution and dedup key |
| `Lead.sourceChannel` | Resolved `PersonChannel` for messaging identities |
| `Lead.sourceConversation` | Set when captured from messaging conversation |
| `Person.originalSource` | Set on first creation only (first-touch attribution) |

### InboundChannel Vocabulary

<Info>
`InboundChannel` enum includes: `email`, `whatsapp`, `call`, `sms`. This is distinct from `leadSource` (the portal) and represents how the inquiry was delivered.
</Info>

Portal mapping examples:
- **Property Finder**: `payload.channel` maps 1:1 to InboundChannel
- **Bayut/dubizzle**: `sourceType` in raw payload maps to InboundChannel

## System Actor Handling

<Tip>
The system now uses dedicated per-org System service users (`isSystemServiceUser = true`) that are seeded at org provisioning and backfilled for existing orgs. This ensures memberless orgs can still capture leads.
</Tip>

The `resolveSystemActorId()` function:
- Returns the per-org System service user
- Provides proper audit attribution for system-initiated captures
- Handles edge cases with fallback to first human member

## Read Operations

### Listing Lead Queries

The module provides `findCapturedLeadsForListing(listingId, orgId)` for portal syndication:

```typescript
// Returns captured leads for a specific listing
interface ListingCapturedLead {
  capturedLeadId: string;
  leadId?: string;
  personId?: string;
  source: LeadSource;
  inboundChannel?: InboundChannel;
  fullName: string;
  email?: string;
  phone?: string;
  outcome: 'created' | 'attached' | 'skipped';
  capturedAt?: Date;
}
```

<Note>
This projection filters by `referenceId = listingId` for portal leads and includes only successful captures (`leadId != null`).
</Note>
