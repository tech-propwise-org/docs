<Note>
This module implements a universal, source-agnostic pipeline that transforms any captured lead from Property Finder, Bayut/dubizzle, and future sources like Meta Ads into CRM leads with consistent source fields, person reuse, deduplication, and assignment.
</Note>

## Overview

The Lead Capture module (`src/modules/crm/lead-capture/`) provides a unified ingestion pipeline that orchestrates existing CRM primitives rather than rebuilding them. Every lead source funnels through `LeadCaptureService.capture()` for consistent processing.

<Info>
**Implementation Status:** BUILT - The module ships with `LeadCaptureService`, source adapters, settings, and the `lead-ingestion` queue worker.
</Info>

### Core Principle

**Reuse, don't rebuild.** This layer sits above existing CRM primitives and orchestrates them without reimplementing lead creation, person matching, deduplication, stakeholders, or distribution logic.

## Capture Pipeline

The `LeadCaptureService.capture(input)` method processes leads through a standardized pipeline:

<Steps>
  <Step title="Initialize Context">
    Runs entirely inside `tenantContext.executeInOrg(input.orgId)` with the system actor as `createdBy` using `resolveSystemActorId`.
  </Step>

  <Step title="Idempotency Guard">
    Look up the `captured_lead` ledger by `(orgId, source, externalLeadId)`. If present, return the prior `CaptureResult` (no-op).
  </Step>

  <Step title="Source Field Normalization">
    Derive `leadSource`, `sourceDetail`, `referenceId`, and any `sourceChannel`/`sourceConversation` from the input.
  </Step>

  <Step title="Identity Resolution">
    Resolve or create the `Person` using existing person matching logic and channel identity systems.
  </Step>

  <Step title="Deduplication Check">
    Decide whether to create a new lead or attach to an existing open lead based on deduplication rules.
  </Step>

  <Step title="Assignment Resolution">
    Execute assignment directive through stakeholder assignment or distribution engine.
  </Step>

  <Step title="Create and Record">
    Create lead via `createLeadInTransaction` or attach re-inquiry activity, write `captured_lead` ledger row, and return `CaptureResult`.
  </Step>
</Steps>

<Note>
**System Actor Availability:** `resolveSystemActorId` now returns the dedicated per-org System service user, which is always seeded at org provisioning. Memberless orgs no longer skip capture.
</Note>

## Source Field Mapping Policy

<AccordionGroup>
  <Accordion title="Lead Source Fields">
    | Field | Rule |
    |-------|------|
    | `Lead.leadSource` | `input.source` - The portal/origin of THIS inquiry |
    | `Lead.sourceDetail` | `input.sourceDetail` - Human-readable campaign/context |
    | `Lead.inboundChannel` | `input.inboundChannel` - Structured delivery channel (first-touch only) |
    | `Lead.referenceId` | `input.referenceId` - Attribution + dedup key |
    | `Lead.sourceChannel` | Resolved `PersonChannel` for persistent messaging identities |
    | `Lead.sourceConversation` | Set when captured from messaging conversation |
    | `Person.originalSource` | Set on FIRST creation only (never overwritten) |
  </Accordion>
</AccordionGroup>

### InboundChannel Vocabulary

<Tabs>
  <Tab title="Enum Values">
    - `email` - Email inquiries
    - `whatsapp` - WhatsApp messages  
    - `call` - Phone calls
    - `sms` - SMS messages
  </Tab>
  
  <Tab title="Portal Mapping">
    **Property Finder:** `payload.channel` maps 1:1 to `InboundChannel`
    
    **Bayut/dubizzle:** Inferred from inquiry context and message content
  </Tab>
</Tabs>

<Warning>
`inboundChannel` (delivery method) is distinct from `sourceChannel` (persistent identity). The former describes HOW the inquiry arrived; the latter is the Person's channel identity.
</Warning>

## Error Handling and Reliability

<CardGroup cols={2}>
  <Card title="Idempotency" icon="shield-check">
    Webhook retries and poll overlaps are safe via the `captured_lead` ledger lookup by `(orgId, source, externalLeadId)`.
  </Card>
  
  <Card title="Failure Recovery" icon="rotate">
    Failures inside jobs bubble to the worker so pg-boss retries. The ledger guard makes retries idempotent.
  </Card>
</CardGroup>
