<Note>
**Purpose:** One universal, source-agnostic pipeline that turns any **captured** lead — Property Finder, Bayut/dubizzle, and later Meta Ads, website forms, etc. — into a CRM `Lead` with consistent **source fields**, **person reuse**, **deduplication**, and **assignment**.

**Module:** `src/modules/crm/lead-capture/` (new).

**Core principle — reuse, don't rebuild.** This layer sits **above** the existing CRM primitives and orchestrates them.
</Note>

<Info>
**Implementation status: BUILT.** Module `src/modules/crm/lead-capture/` ships `LeadCaptureService` with complete pipeline, contracts, adapters, settings, and async transport via pg-boss queue.
</Info>

## Scope & non-goals

**In scope:** a canonical captured-lead contract, a source-adapter pattern, and the orchestration of source-field mapping + identity resolution + dedup/attach + assignment, plus the async transport (queue + worker) that feeds it.

**Out of scope (by decision):** Meta Ads and website-form ingestion are **not** designed here. The contract simply reserves the fields they will need (`externalLeadId`, `assignmentHint`) so they plug in later without a refactor.

## Canonical contract

### CapturedLeadInput

```typescript
interface CapturedLeadInput {
  orgId: string;
  source: LeadSource;                  // Bayut | PropertyFinder | Dubizzle | Website | Instagram
  sourceDetail?: string;               // campaign / form / "WhatsApp message"
  inboundChannel?: InboundChannel;     // call | whatsapp | email | sms
  referenceId?: string;                // listing / campaign / form id
  externalLeadId?: string;             // source's own lead id - hard idempotency key

  identity: {                          // for Person reuse, in match-priority order
    firstName?: string;
    lastName?: string;
    email?: string;
    phone?: string;
    channel?: { type: ChannelType; identifier: string };
  };

  intent?: LeadIntent;
  propertyRef?: { listingId?: string; refUnitId?: number };
  assignmentHint?: CaptureAssignmentConfig;
  capturedAt?: Date;
  rawPayload: unknown;                 // stored verbatim for audit / replay
}
```

### CaptureResult

```typescript
interface CaptureResult {

}
```

## Pipeline - `LeadCaptureService.capture(input)`

Runs entirely inside `tenantContext.executeInOrg(input.orgId)` with the **system actor** as `createdBy`.

<Steps>
<Step title="Idempotency guard">
Look up the `captured_lead` ledger by `(orgId, source, externalLeadId)`. If present → return the prior `CaptureResult` (no-op).
</Step>

<Step title="Source-field normalization">
Derive `leadSource`, `sourceDetail`, `referenceId`, and any `sourceChannel`/`sourceConversation`.
</Step>

<Step title="Identity resolution & person reuse">
Resolve/create the `Person` using existing person matching logic.
</Step>

<Step title="Dedup / attach">
Decide create-new vs attach-to-open-lead using existing deduplication logic.
</Step>

<Step title="Assignment">
Resolve the directive and execute (stakeholder or distribution).
</Step>

<Step title="Create or attach + record ledger">
Create via `createLeadInTransaction` or attach a re-inquiry activity, write the `captured_lead` ledger row.
</Step>
</Steps>

<Info>
**System actor availability:** `resolveSystemActorId` now returns the dedicated per-org System service user, which is always seeded at org provisioning. Memberless orgs no longer skip capture.
</Info>

### Read query - `findCapturedLeadsForListing(listingId, orgId)`

A read-only projection over the `captured_lead` ledger that powers the portal-syndication **listing detail "Leads" tab**. 

The method:
- Filters `referenceId = listingId`, `source IN (PropertyFinder, Bayut, Dubizzle)`
- Only includes `leadId != null` (successful `created`/`attached` rows)
- Orders newest first
- Maps to `ListingCapturedLead` with person details and outcome

## Source-field policy

| Field | Rule |
|---|---|
| `Lead.leadSource` | `input.source` — the source of THIS inquiry (the portal/origin) |
| `Lead.sourceDetail` | `input.sourceDetail` (campaign name, "WhatsApp message", etc.) |
| `Lead.inboundChannel` | `input.inboundChannel` — structured delivery channel (`InboundChannel`) |
| `Lead.referenceId` | `input.referenceId` (listing/campaign/form id) - attribution + dedup key |
| `Lead.sourceChannel` | Resolved `PersonChannel` for persistent messaging identities |
| `Lead.sourceConversation` | Set when captured from a messaging conversation |
| `Person.originalSource` | Set on FIRST creation only (first-touch attribution) |

<Tip>
**`InboundChannel` vocabulary:** `email | whatsapp | call | sms`. Distinct from `leadSource` (the portal) - this is *how the inquiry arrived*.
</Tip>

## Identity resolution (Person reuse)

<Tabs>
<Tab title="Match Priority Order">
1. **Channel identity** (WhatsApp number, IG handle, etc.)
2. **Email** (exact match, case-insensitive)
3. **Phone** (E.164 normalized)
4. **Name + context** (fuzzy match within org)
</Tab>

<Tab title="Person Creation">
If no match found:
- Create new `Person` with `originalSource = input.source`
- Set primary contact method from available identity data
- Link to messaging channels if applicable
</Tab>
</Tabs>

## Deduplication logic

The system checks for duplicate leads using:

<AccordionGroup>
<Accordion title="Exact match criteria">
- Same `Person` + same `referenceId` (listing/property)
- Same `Person` + same `source` + recent timeframe
- Same external lead ID from source system
</Accordion>

<Accordion title="Attach vs Create decision">
**Attach to existing lead** if:
- Lead status is still "open" (not closed/converted)
- Same property interest or no property specified
- Within reasonable time window (configurable)

**Create new lead** if:
- No duplicate found
- Existing lead is closed/converted
- Different property interest
- Significant time gap suggests new genuine interest
</Accordion>
</AccordionGroup>

### Assignment strategies

```typescript
enum CaptureAssignmentStrategy {
  LISTING_AGENT = 'listing_agent',     // Agent who listed the property
  TEAM_POOL = 'team_pool',             // Round-robin within team
  ORG_POOL = 'org_pool',               // Round-robin organization-wide
  MANUAL = 'manual',                   // No auto-assignment
  CUSTOM_RULES = 'custom_rules'        // Use distribution engine rules
}
```

```typescript
// Queue job payload
interface LeadIngestionJob {
```

<Warning>
Failed captures are retried by pg-boss. The idempotency guard makes retries safe.
</Warning>

### Error handling

<Steps>
<Step title="Validation errors">
Invalid input data - logged and marked as permanently failed
</Step>

<Step title="Temporary failures">
Database connections, external API calls - retried with exponential backoff
</Step>

<Step title="Assignment failures">
When no agent can be assigned - lead created but queued for manual assignment
</Step>
</Steps>

## Configuration & settings

### Lead capture settings

```typescript
@Entity('lead_capture_settings')
export class LeadCaptureSettings {
  @PrimaryColumn('uuid')
  orgId: string;
  
  @Column({ type: 'enum', enum: CaptureAssignmentStrategy })
  defaultAssignmentStrategy: CaptureAssignmentStrategy;
  
  @Column({ type: 'boolean', default: true })
  enableDeduplication: boolean;
  
  @Column({ type: 'interval', nullable: true })
  deduplicationWindow: string; // PostgreSQL interval
  
  @Column({ type: 'boolean', default: true })
  autoAssignOnCapture: boolean;
}
```

### Org-level defaults

Each organization can configure:
- Default assignment strategy for captured leads
- Deduplication time window
- Whether to enable automatic assignment
- Fallback teams/pools for unassigned leads

<Check>
Settings are cached and refreshed periodically to avoid database hits on every capture.
</Check>
