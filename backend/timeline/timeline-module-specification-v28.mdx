## Overview

The Timeline module provides a unified view of chronological events across the PropWise CRM system. It aggregates data from multiple sources (audit logs, activities, and lead-level system events from distribution/escalation) to present a comprehensive timeline of events related to specific entities like leads, contacts, and companies.

The module serves as a central point for tracking historical changes and activities, enabling users to understand the complete lifecycle and interaction history of CRM entities.

## Architecture

The Timeline module follows a unified aggregation pattern, combining data from disparate sources into a single chronological view.

### Key architectural decisions

<CardGroup cols={2}>
  <Card title="Source abstraction" icon="filter">
    Uses a `TimelineSource` enum to specify which data sources to query (audit, activity, or all)
  </Card>
  <Card title="Unified response format" icon="layer-group">
    Normalizes different event types into a consistent timeline entry structure
  </Card>
  <Card title="Tenant-aware" icon="shield">
    All timeline queries are executed within the proper tenant context
  </Card>
  <Card title="Resolver pattern" icon="code-branch">
    Delegates timeline entry formatting to specialized resolver classes
  </Card>
</CardGroup>

<Info>
The module implements page-based pagination over merged timeline results and includes optional event details such as stage-change reasons and compact activity channel metadata.
</Info>

## Entity relationships

The Timeline module doesn't define its own entities but aggregates data from multiple sources:

<AccordionGroup>
  <Accordion title="Core data sources">
    - **AuditLog**: System audit entries tracking entity changes
    - **Activity**: User-created activities and interactions, including lightweight scheduled task/event index rows
    - **Activity Channel Tables**: `call_log`, `email_log`, `sms_log`, `meeting`, and `site_visit` rows are batch-hydrated for expandable activity details
  </Accordion>
  
  <Accordion title="Relationship tracking">
    - **ContactCompanyRole**: Relationship tracking between contacts and companies
    - **LeadStageHistory / DealStageHistory**: Stage transition history used to attach user-provided stage-change reasons
  </Accordion>
  
  <Accordion title="System events">
    - **DistributionLog / EscalationActionLog**: Lead-level system events for automatic assignment and escalation activity
    - **EntityTransfer**: Related lead/deal transfer request audit entries
    - **UnitAccessRequest**: Deal-linked unit access request audit entries
  </Accordion>
  
  <Accordion title="Property interests">
    - **LeadPropertyInterest / DealPropertyInterest**: Add/update/remove events for property interests
    - Timeline events: `LEAD_PROPERTY_INTEREST_ADDED`, `LEAD_PROPERTY_INTEREST_UPDATED`, `LEAD_PROPERTY_INTEREST_REMOVED`, `DEAL_PROPERTY_INTEREST_ADDED`, `DEAL_PROPERTY_INTEREST_UPDATED`, `DEAL_PROPERTY_INTEREST_REMOVED`
  </Accordion>
</AccordionGroup>

## API endpoints

### Get entity timeline

<CodeGroup>
```http Lead timeline
GET /v1/timeline/lead/:id
```

```http Deal timeline
GET /v1/timeline/deal/:id
```

```http Contact timeline
GET /v1/timeline/contact/:id
```

```http Company timeline
GET /v1/timeline/company/:id
```
</CodeGroup>

#### Parameters

| Parameter | Type | Required | Default | Description |
|-----------|------|----------|---------|-------------|
| `source` | `TimelineSource` | No | `all` | Data source filter (`audit`, `activity`, `system`, `all`) |
| `page` | `number` | No | 1 | Page number for merged timeline results |
| `limit` | `number` | No | 50 | Number of entries per page (max 200) |
| `from` | `ISO8601` | No | - | Filter events from this date |
| `to` | `ISO8601` | No | - | Filter events until this date |
| `activityType` | `string` | No | - | Single or comma-separated activity type filter |

#### Response

```typescript
{
  data: UnifiedTimelineItemDto[];
  page: number;
  limit: number;
  total: number;
  totalPages: number;
  hasMore: boolean;
}
```

#### Timeline entry structure

```typescript
{
  id: string;
  source: 'audit' | 'activity' | 'system';
  timestamp: string; // ISO8601
  message: string;
  accentColor?: string; // Optional display accent
  reason?: string; // Optional detail text

  // Audit-only fields
  eventType?: TimelineEventType;
  auditAction?: AuditAction;
  tableName?: string;
  recordId?: string;

  // Activity-only fields
  activityType?: ActivityType;
  referenceId?: string;
  targetEntityType?: ActivityTargetType;
  targetEntityId?: string;
  isPinned?: boolean;
  isImportant?: boolean;
  activityDetails?: {
    type: ActivityType;
    // Varies by activity type:
    // call: direction/status/outcome/phone/duration/recording/notes/purpose
    // email: direction/status/from/to/cc/bcc/subject/bodyPlain/provider/tracking/threading
    // sms: direction/status/from/to/body/provider/delivery/error
    // meeting: type/status/timing/location/link/outcome/notes/cancellation
    // site_visit: visit type/status/timing/property/feedback/notes/cancellation
  };

  // System-only fields (lead timelines)
  systemActionType?: 'distribution' | 'escalation';

  actor?: {
    id: string;
    name: string;
    email?: string;
    avatarUrl?: string;
  };
}
```

### Task-scoped feeds

<Warning>
Tasks expose their own feed endpoints from the CRM task module rather than the generic timeline controller.
</Warning>

<Tabs>
  <Tab title="Combined timeline">
    ```http
    GET /tasks/:id/timeline
    ```
    Returns combined task feed (`TaskComment` rows + task `AuditLog` events)
  </Tab>
  
  <Tab title="Activity only">
    ```http
    GET /tasks/:id/activity
    ```
    Returns only audit-backed task activity events
  </Tab>
  
  <Tab title="Comments only">
    ```http
    GET /tasks/:id/comments/feed
    ```
    Returns only comments in keyset-paginated feed shape
  </Tab>
</Tabs>

All three endpoints share the same `TaskTimelineFeedQueryDto` contract with `limit` (default 40, max 100) and optional `before` cursor.

### Commission payment timeline

```http
GET /commission-payments/:id/history
```

<Note>
Commission payments use a dedicated history endpoint that returns immutable `commission_payment_history` rows ordered oldest-first. The client sorts these by `createdAt` descending for display.
</Note>

### Org-wide activity feed (dashboard)

The `UnifiedTimelineService.getActivityFeedInTransaction()` method powers the dashboard Activity Feed widget with org-wide, actor-scoped event aggregation.

<Steps>
  <Step title="Source merging">
    Combines audit lifecycle events from curated tables with logged CRM activity rows
  </Step>
  <Step title="Actor filtering">
    Filters by `actorUserIds` based on caller's CRM/Sales scope (self/team/org)
  </Step>
  <Step title="Cursor pagination">
    Uses `(occurredAt DESC, id DESC)` ordering with in-memory merge and sort
  </Step>
</Steps>

## Business rules

<Check>
**Tenant isolation**: Timeline entries are strictly filtered by tenant context
</Check>

<Check>
**Permission-based access**: Users can only view timelines for entities they have access to
</Check>

### Core rules

1. **Chronological ordering**: Entries are always returned in reverse chronological order (newest first)
2. **Source filtering**: When `source=all`, entries from all available sources are merged and sorted by timestamp
3. **Pagination limits**: Maximum of 200 entries per request to prevent performance issues
4. **Date range filtering**: When date filters are applied, they are inclusive of boundary dates

### Display enhancements

<Tabs>
  <Tab title="Stage changes">
    - **Accent color**: Stage-change timeline items include `accentColor` set to the destination stage color
    - **Change reasons**: Include `reason` when matching stage history record has notes
    - **Matching logic**: Uses entity ID, from-stage ID, to-stage ID, and timestamp window
  </Tab>
  
  <Tab title="Lead disqualification">
    - **Panel display**: Shows `lead.disqualificationNotes` as visible reason detail
    - **Fallback**: Uses timeline item's `reason` when no written notes exist
    - **Message format**: Includes categorical disqualification reason (e.g., "Lead disqualified: Budget")
  </Tab>
  
  <Tab title="Contact merging">
    - **Activity repointing**: Both `Activity.contactId` and direct contact-target activities are re-pointed to surviving contact
    - **Timeline continuity**: Absorbed contact's timeline appears on surviving contact after merge
  </Tab>
</Tabs>

### Communication targeting

<Info>
Company-targeted communication activities can be logged directly against the company with `Activity.contactId = null`. When a linked company contact is selected, the system validates the `contact_company_role` relationship and stores the contact ID so the activity appears on both timelines.
</Info>
