## Overview

The Timeline module provides a unified view of chronological events across the PropWise CRM system. It aggregates data from multiple sources (audit logs, activities, and lead-level system events from distribution/escalation) to present a comprehensive timeline of events related to specific entities like leads, contacts, and companies.

<Info>
**Last Updated:** 2026-05-23  
**Status:** Active
</Info>

The module serves as a central point for tracking historical changes and activities, enabling users to understand the complete lifecycle and interaction history of CRM entities.

## Architecture

The Timeline module follows a unified aggregation pattern, combining data from disparate sources into a single chronological view.

### Key Architectural Decisions

<CardGroup cols={2}>
  <Card title="Source Abstraction" icon="filter">
    Uses a `TimelineSource` enum to specify which data sources to query (audit, activity, system, or all)
  </Card>
  <Card title="Unified Response Format" icon="table-list">
    Normalizes different event types into a consistent timeline entry structure with optional details
  </Card>
  <Card title="Tenant-Aware" icon="users">
    All timeline queries are executed within the proper tenant context
  </Card>
  <Card title="Resolver Pattern" icon="code-branch">
    Delegates timeline entry formatting to specialized resolver classes
  </Card>
</CardGroup>

### Pagination Support

The module implements page-based pagination over merged timeline results, supporting efficient browsing of large event histories.

## Entity Relationships

<Note>
The Timeline module doesn't define its own entities but aggregates data from multiple sources.
</Note>

### Data Sources

The Timeline module aggregates data from:

<AccordionGroup>
  <Accordion title="AuditLog">
    System audit entries tracking entity changes across all CRM entities
  </Accordion>
  
  <Accordion title="Activity">
    User-created activities and interactions, including lightweight scheduled task/event index rows for entity-linked task/event records that do not create a typed channel activity
  </Accordion>
  
  <Accordion title="Activity Channel Tables">
    Batch-hydrated records from:
    - `call_log`
    - `email_log`
    - `sms_log`
    - `meeting`
    - `site_visit`
    
    These are used for expandable activity details.
  </Accordion>
  
  <Accordion title="ContactCompanyRole">
    Relationship tracking between contacts and companies
  </Accordion>
  
  <Accordion title="Stage History Tables">
    - `LeadStageHistory`
    - `DealStageHistory`
    
    Stage transition history used to attach user-provided stage-change reasons to matching audit timeline entries.
  </Accordion>
  
  <Accordion title="Distribution & Escalation Logs">
    - `DistributionLog`
    - `EscalationActionLog`
    
    Lead-level system events for automatic assignment and escalation activity.
  </Accordion>
  
  <Accordion title="EntityTransfer">
    Related lead/deal transfer request audit entries are included on the owning entity timeline
  </Accordion>
  
  <Accordion title="UnitAccessRequest">
    Deal-linked unit access request audit entries are included on the related deal timeline
  </Accordion>
  
  <Accordion title="Property Interest Tables">
    - `LeadPropertyInterest`
    - `DealPropertyInterest`
    
    Add/update/remove events for property interests are included on the owning lead/deal timeline as curated audit entries.
    
    **Timeline Events:**
    - `LEAD_PROPERTY_INTEREST_ADDED`
    - `LEAD_PROPERTY_INTEREST_UPDATED`
    - `LEAD_PROPERTY_INTEREST_REMOVED`
    - `DEAL_PROPERTY_INTEREST_ADDED`
    - `DEAL_PROPERTY_INTEREST_UPDATED`
    - `DEAL_PROPERTY_INTEREST_REMOVED`
  </Accordion>
</AccordionGroup>

## API Endpoints

### Get Entity Timeline

Retrieves the timeline for a specific entity (lead, deal, contact, or company).

```http
GET /v1/timeline/{lead|deal|contact|company}/:id
```

#### Path Parameters

| Parameter | Type | Description |
|-----------|------|-------------|
| `entityType` | `string` | Entity type: `lead`, `deal`, `contact`, or `company` |
| `id` | `UUID` | UUID of the target entity |

#### Query Parameters

| Parameter | Type | Required | Default | Description |
|-----------|------|----------|---------|-------------|
| `source` | `TimelineSource` | No | `all` | Data source filter (`audit`, `activity`, `system`, `all`) |
| `page` | `number` | No | 1 | Page number for merged timeline results |
| `limit` | `number` | No | 50 | Number of entries per page (max 200) |
| `from` | `ISO8601` | No | - | Filter events from this date |
| `to` | `ISO8601` | No | - | Filter events until this date |
| `activityType` | `string` | No | - | Single or comma-separated activity type filter |

#### Response Structure

<CodeGroup>

```typescript Response
{
  data: UnifiedTimelineItemDto[];
  page: number;
  limit: number;
  total: number;
  totalPages: number;
  hasMore: boolean;
}
```

```typescript Timeline Entry
{
  id: string;
  source: 'audit' | 'activity' | 'system';
  timestamp: string; // ISO8601
  message: string;
  accentColor?: string; // Optional display accent; stage changes use destination stage color
  reason?: string; // Optional detail text; stage changes use matching stage history notes

  // Audit-only fields
  eventType?: TimelineEventType;
  auditAction?: AuditAction;
  tableName?: string;
  recordId?: string;

  // Activity-only fields
  activityType?: ActivityType;
  referenceId?: string;
  targetEntityType?: ActivityTargetType;
  targetEntityId?: string;
  isPinned?: boolean;
  isImportant?: boolean;
  activityDetails?: {
    type: ActivityType;
    // call: direction/status/outcome/phone/duration/recording/notes/purpose
    // email: direction/status/from/to/cc/bcc/subject/bodyPlain/provider/tracking/threading
    // sms: direction/status/from/to/body/provider/delivery/error
    // meeting: type/status/timing/location/link/outcome/notes/cancellation
    // site_visit: visit type/status/timing/property/feedback/notes/cancellation
  };

  // System-only fields (lead timelines)
  systemActionType?: 'distribution' | 'escalation';

  actor?: {
    id: string;
    name: string;
    email?: string;
    avatarUrl?: string;
  };
}
```

</CodeGroup>

### Task-Scoped Feeds

<Warning>
Tasks expose their own feed endpoints from the CRM task module rather than the generic `/timeline/:entityType/:entityId` controller.
</Warning>

<Tabs>
  <Tab title="Combined Feed">
    ```http
    GET /tasks/:id/timeline
    ```
    Returns the combined task feed (`TaskComment` rows + task `AuditLog` events).
  </Tab>
  
  <Tab title="Activity Only">
    ```http
    GET /tasks/:id/activity
    ```
    Returns only audit-backed task activity events.
  </Tab>
  
  <Tab title="Comments Feed">
    ```http
    GET /tasks/:id/comments/feed
    ```
    Returns only comments in the same keyset-paginated feed shape.
  </Tab>
</Tabs>

#### Shared Query Contract

All three endpoints share the same `TaskTimelineFeedQueryDto` contract:

| Parameter | Type | Required | Default | Description |
|-----------|------|----------|---------|-------------|
| `limit` | `number` | No | 40 | Max 100 |
| `before` | `string` | No | - | Optional cursor |

#### Response Format

```typescript
{
  items: TaskTimelineItem[]; // ascending order for fetched page
  hasMoreOlder: boolean;
  nextCursorBefore?: string;
}
```

<Info>
Use `GET /tasks/:id/comments` only when the caller needs the legacy full comment array.
</Info>

### Commission Payment Timeline

Commission payments do not use the generic unified timeline controller.

```http
GET /commission-payments/:id/history
```

<Note>
The commission panel Timeline tab reads this endpoint, which returns immutable `commission_payment_history` rows ordered oldest-first.
</Note>

**Display Behavior:**
- Side panel sorts rows by `createdAt` descending for display
- Newest event appears at the top, matching lead/deal panel timelines
- Notes entered during status changes are persisted on the history row `reason` field
- Displayed only in the Timeline tab

**Activity Notes:**
- Free-form commission panel notes use the Activity/Note system
- `targetEntityType='commission_payment'`
- Isolated to the panel Notes tab

```typescript
UnifiedTimelineService.getActivityFeedInTransaction(
  em, 
  orgId,
```

Powers the dashboard **Activity Feed** widget as an **org-wide, actor-scoped** "what happened" feed.

<Warning>
Distribution/escalation system rows are **not** included in the activity feed.
</Warning>

#### Actor Scope

| Value | Behavior |
|-------|----------|
| `actorUserIds` array | Filters audit rows by `userId` and activity rows by `performedBy` |
| `null` | No actor filter (org scope) |

The dashboard derives this from the caller's resolved CRM/Sales scope:
- **Self** → `[userId]`
- **Team** → self ∪ team members
- **Org** → `null`

#### Audit Resolution

Each audit row is resolved through `AuditTimelineResolver.resolveBatch` into:
- `kind` (`TimelineEventType`)
- Human-readable `message` (e.g., "Lead created", "Moved from New to Contacted", "Sarah assigned", "Deal won")
- Optional `accentColor` (destination stage color)

#### Cursor Pagination

<Steps>
  <Step title="Independent Ordering">
    Each source is independently ordered `(occurredAt DESC, id DESC)`
  </Step>
  
  <Step title="Over-Fetching">
    Method over-fetches `limit + 1` per source
  </Step>
  
  <Step title="Merging">
    Merges in memory and sorts globally
  </Step>
  
  <Step title="Slicing">
    Slices the top `limit` results
  </Step>
  
  <Step title="Response">
    Returns `{ items, hasMore }`
  </Step>
</Steps>

The caller (dashboard) encodes the last item's `{ occurredAt, id }` into its opaque cursor.

#### Return Shape

```typescript
UnifiedActivityFeedItem[] {
  id: string;
  source: 'audit' | 'activity';
  kind: TimelineEventType;
  message: string;
  occurredAt: string;
  accentColor?: string;
  targetEntityType?: string;
  targetEntityId?: string;
  actor?: {
    id: string;
    name: string;
    email?: string;
    avatarUrl?: string;
  };
}
```

<Info>
The dashboard module (`DashboardService.getActivityFeed`) is the only current caller. It maps `UnifiedActivityFeedItem` into `DashboardActivityItemDto`. See `DASHBOARD_MODULE_SPECIFICATION.md` for details.
</Info>

## Business Rules

<AccordionGroup>
  <Accordion title="1. Tenant Isolation" icon="shield">
    Timeline entries are strictly filtered by tenant context. All queries respect tenant boundaries.
  </Accordion>
  
  <Accordion title="2. Permission-Based Access" icon="lock">
    Users can only view timelines for entities they have access to. Access control is enforced at the entity level.
  </Accordion>
  
  <Accordion title="3. Chronological Ordering" icon="arrow-down-1-9">
    Entries are always returned in reverse chronological order (newest first) for consistent user experience.
  </Accordion>
  
  <Accordion title="4. Source Filtering" icon="filter">
    When `source=all`, entries from all available sources are merged and sorted by timestamp:
    
    - **Lead timelines** include: direct lead audit, related transfer audit, related lead-property-interest audit, activity, distribution, and escalation entries
    - **Deal timelines** include: direct deal audit, related transfer audit, related unit-access request audit, related deal-property-interest audit, and activity entries
    - **Contact/Company timelines** include: audit and activity entries
  </Accordion>
  
  <Accordion title="5. Pagination Limits" icon="list">
    Maximum of 200 entries per request to prevent performance issues. Default is 50 entries per page.
  </Accordion>
  
  <Accordion title="6. Date Range Filtering" icon="calendar-range">
    When date filters are applied, they are inclusive of the boundary dates.
  </Accordion>
  
  <Accordion title="7. Stage Accent Color" icon="palette">
    Lead and deal stage-change timeline items include `accentColor` set to the destination stage color so clients can render timeline indicators without parsing stage names or fetching stage metadata.
    
    **Override Architecture:**
    - System stages are GLOBAL rows whose ID is shared across orgs
    - Per-org color lives in `lead_stage_override` / `deal_stage_override`
    - `AuditTimelineResolver.batchResolveNames` loads the override map **per distinct org in the batch**
    - Keyed as `Map<orgId, Map<stageId, color>>`
    - Resolves each entry's accent against **its own** `entry.organizationId`
    - Falls back to the global stage color
    
    <Warning>
    This matters because system-admin audit queries run under RLS bypass and a single page can span multiple tenants. A single shared map would otherwise bleed one org's override color onto another org's events.
    </Warning>
  </Accordion>
  
  <Accordion title="8. Stage Change Reason" icon="comment">
    Lead and deal stage-change timeline items include `reason` when a matching stage history record has notes.
    
    **Matching Criteria:**
    - Entity ID
    - From-stage ID
    - To-stage ID
    - Short timestamp window (to prevent repeated transitions from receiving the wrong reason)
  </Accordion>
  
  <Accordion title="9. Lead Disqualification Panel Display" icon="ban">
    Lead disqualification audit messages include the categorical disqualification reason in `message` (for example, `Lead disqualified: Budget`).
    
    **Display Priority:**
    1. The lead panel Timeline tab displays `lead.disqualificationNotes` as the visible `Reason:` detail when notes exist
    2. Falls back to the timeline item's `reason` only when no written notes were saved
  </Accordion>
  
  <Accordion title="10. Contact Merge Repointing" icon="code-merge">
    Person duplicate merges can absorb duplicate Contact records.
    
    **During Merge:**
    - Both `Activity.contactId` and direct Contact-target activities are re-pointed to the surviving Contact
    - `targetEntityType='contact'`
    - `targetEntityId=<absorbedContactId>` is updated to the surviving contact ID
    
    This ensures timeline continuity after merge operations.
  </Accordion>
</AccordionGroup>
