## Overview

The Timeline module provides a unified view of chronological events across the PropWise CRM system. It aggregates data from multiple sources (audit logs and activities) to present a comprehensive timeline of events related to specific entities like leads, contacts, and companies. The module serves as a central point for tracking historical changes and activities, enabling users to understand the complete lifecycle and interaction history of CRM entities.

## Architecture

The Timeline module follows a unified aggregation pattern, combining data from disparate sources into a single chronological view.

### Key architectural decisions

<AccordionGroup>
<Accordion title="Source abstraction">
Uses a `TimelineSource` enum to specify which data sources to query (audit, activity, or all)
</Accordion>

<Accordion title="Unified response format">
Normalizes different event types into a consistent timeline entry structure
</Accordion>

<Accordion title="Tenant-aware">
All timeline queries are executed within the proper tenant context
</Accordion>

<Accordion title="Pagination support">
Implements cursor-based pagination for efficient timeline browsing
</Accordion>

<Accordion title="Resolver pattern">
Delegates timeline entry formatting to specialized resolver classes (e.g., `AuditTimelineResolver`)
</Accordion>
</AccordionGroup>

## Entity relationships

<Info>
The Timeline module doesn't define its own entities but aggregates data from existing system entities.
</Info>

The Timeline module aggregates data from:

- **AuditLog**: System audit entries tracking entity changes
- **Activity**: User-created activities and interactions
- **ContactCompanyRole**: Relationship tracking between contacts and companies

## API endpoints

### Get entity timeline

```http
GET /timeline/:entityType/:entityId
```

#### Parameters

- `entityType` (path): Type of entity (lead, contact, company, etc.)
- `entityId` (path): UUID of the target entity

#### Query parameters

| Parameter | Type | Required | Default | Description |
|-----------|------|----------|---------|-------------|
| `source` | `TimelineSource` | No | `all` | Data source filter (audit/activity/all) |
| `limit` | `number` | No | 50 | Number of entries per page (max 100) |
| `cursor` | `string` | No | - | Pagination cursor |
| `startDate` | `ISO8601` | No | - | Filter events from this date |
| `endDate` | `ISO8601` | No | - | Filter events until this date |

#### Response

<CodeGroup>
```typescript Response Structure
{
  items: UnifiedTimelineEntryDto[];
  hasNextPage: boolean;
  nextCursor?: string;
  totalCount?: number;
}
```

```typescript Timeline Entry Structure
{
  id: string;
  timestamp: string; // ISO8601
  type: TimelineEventType;
  source: TimelineSource;
  title: string;
  description?: string;
  user?: {
    id: string;
    name: string;
    email: string;
  };
  metadata?: Record<string, any>;
}
```
</CodeGroup>

## Business rules

<Steps>
<Step title="Tenant isolation">
Timeline entries are strictly filtered by tenant context
</Step>

<Step title="Permission-based access">
Users can only view timelines for entities they have access to
</Step>

<Step title="Chronological ordering">
Entries are always returned in reverse chronological order (newest first)
</Step>

<Step title="Source filtering">
When `source=all`, entries from all sources are merged and sorted by timestamp
</Step>

<Step title="Pagination limits">
Maximum of 100 entries per request to prevent performance issues
</Step>

<Step title="Date range filtering">
When date filters are applied, they are inclusive of the boundary dates
</Step>
</Steps>

## Service patterns

### UnifiedTimelineService

The core service implements several key patterns:

<Tabs>
<Tab title="Multi-source aggregation">
Queries multiple entity types and merges results into a unified timeline view
</Tab>

<Tab title="Cursor-based pagination">
Implements efficient pagination using timestamp cursors to avoid expensive OFFSET queries
</Tab>

<Tab title="Tenant context execution">
All database operations execute within tenant boundaries for proper data isolation
</Tab>

<Tab title="Error handling">
Graceful handling of missing entities and invalid queries with appropriate error responses
</Tab>
</Tabs>

### Resolver pattern

Timeline formatting is delegated to specialized resolvers:

- Each source type has its own resolver (e.g., `AuditTimelineResolver`)
- Resolvers handle source-specific formatting and metadata extraction
- Maintains consistency in timeline entry structure across different sources

## Performance considerations

<Tip>
The Timeline module is designed with performance in mind, using several optimization strategies.
</Tip>

### Optimization strategies

- **Database indexing**: Relies on proper indexing of timestamp fields for efficient querying
- **Pagination**: Uses cursor-based pagination to avoid expensive OFFSET queries
- **Query optimization**: Separate queries per source type, then merges in application layer
- **Caching**: No explicit caching implemented - relies on database query performance

<Check>
The cursor-based pagination approach ensures consistent performance even with large datasets.
</Check>
