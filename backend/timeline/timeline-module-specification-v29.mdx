## Overview

The Timeline module provides a unified view of chronological events across the PropWise CRM system. It aggregates data from multiple sources (audit logs, activities, and lead-level system events from distribution/escalation) to present a comprehensive timeline of events related to specific entities like leads, contacts, and companies.

<Note>
The module serves as a central point for tracking historical changes and activities, enabling users to understand the complete lifecycle and interaction history of CRM entities.
</Note>

## Architecture

The Timeline module follows a unified aggregation pattern, combining data from disparate sources into a single chronological view. Key architectural decisions:

### Core Patterns

<CardGroup cols={2}>
  <Card title="Source Abstraction" icon="filter">
    Uses a `TimelineSource` enum to specify which data sources to query (audit, activity, or all)
  </Card>
  <Card title="Unified Response Format" icon="layer-group">
    Normalizes different event types into a consistent timeline entry structure
  </Card>
  <Card title="Tenant-Aware" icon="shield">
    All timeline queries are executed within the proper tenant context
  </Card>
  <Card title="Resolver Pattern" icon="gear">
    Delegates timeline entry formatting to specialized resolver classes
  </Card>
</CardGroup>

### Key Features

- **Pagination Support**: Implements page-based pagination over the merged timeline results
- **Optional Event Details**: Stage-change reasons and compact activity channel metadata
- **Specialized Resolvers**: `AuditTimelineResolver`, `DistributionTimelineResolver`, `EscalationTimelineResolver`

## Entity Relationships

<Warning>
The Timeline module doesn't define its own entities but aggregates data from existing sources.
</Warning>

### Data Sources

<AccordionGroup>
  <Accordion title="Core Activity Sources">
    - **AuditLog**: System audit entries tracking entity changes
    - **Activity**: User-created activities and interactions
    - **Activity Channel Tables**: `call_log`, `email_log`, `sms_log`, `meeting`, and `site_visit` rows
  </Accordion>
  
  <Accordion title="Relationship Sources">
    - **ContactCompanyRole**: Relationship tracking between contacts and companies
    - **LeadStageHistory / DealStageHistory**: Stage transition history with user-provided reasons
  </Accordion>
  
  <Accordion title="System Event Sources">
    - **DistributionLog / EscalationActionLog**: Lead-level system events for automatic assignment
    - **EntityTransfer**: Related lead/deal transfer request audit entries
    - **UnitAccessRequest**: Deal-linked unit access request audit entries
  </Accordion>
  
  <Accordion title="Interest Tracking">
    - **LeadPropertyInterest / DealPropertyInterest**: Property interest changes
    - Timeline events: `LEAD_PROPERTY_INTEREST_ADDED`, `LEAD_PROPERTY_INTEREST_UPDATED`, `LEAD_PROPERTY_INTEREST_REMOVED`
    - Deal events: `DEAL_PROPERTY_INTEREST_ADDED`, `DEAL_PROPERTY_INTEREST_UPDATED`, `DEAL_PROPERTY_INTEREST_REMOVED`
  </Accordion>
</AccordionGroup>

## API Endpoints

### Get Entity Timeline

<CodeGroup>
```http Basic Request
GET /v1/timeline/{lead|deal|contact|company}/:id
```

```http With Parameters
GET /v1/timeline/lead/123e4567-e89b-12d3-a456-426614174000?source=all&page=1&limit=50
```
</CodeGroup>

#### Path Parameters

| Parameter | Type | Description |
|-----------|------|-------------|
| `leadId`, `dealId`, `contactId`, or `companyId` | UUID | Target entity identifier |

#### Query Parameters

| Parameter | Type | Required | Default | Description |
|-----------|------|----------|---------|-------------|
| `source` | `TimelineSource` | No | `all` | Data source filter (`audit`, `activity`, `system`, `all`) |
| `page` | `number` | No | 1 | Page number for merged timeline results |
| `limit` | `number` | No | 50 | Number of entries per page (max 200) |
| `from` | `ISO8601` | No | - | Filter events from this date |
| `to` | `ISO8601` | No | - | Filter events until this date |
| `activityType` | `string` | No | - | Single or comma-separated activity type filter |

#### Response Format

```typescript
{
  data: UnifiedTimelineItemDto[];
  page: number;
  limit: number;
  total: number;
  totalPages: number;
  hasMore: boolean;
}
```

### Task-Scoped Feeds

<Note>
Tasks expose their own feed endpoints from the CRM task module rather than the generic timeline controller.
</Note>

#### Available Endpoints

<Steps>
  <Step title="Combined Feed">
    ```http
    GET /tasks/:id/timeline
    ```
    Returns combined task feed (`TaskComment` rows + task `AuditLog` events)
  </Step>
  
  <Step title="Activity Only">
    ```http
    GET /tasks/:id/activity
    ```
    Returns only audit-backed task activity events
  </Step>
  
  <Step title="Comments Only">
    ```http
    GET /tasks/:id/comments/feed
    ```
    Returns only comments in keyset-paginated feed shape
  </Step>
</Steps>

#### Query Parameters

All three endpoints share the same `TaskTimelineFeedQueryDto` contract:

| Parameter | Type | Default | Max | Description |
|-----------|------|---------|-----|-------------|
| `limit` | `number` | 40 | 100 | Number of items per page |
| `before` | `string` | - | - | Optional cursor for pagination |

#### Response Format

```typescript
{
  items: TaskTimelineItem[]; // Ascending order for fetched page
  hasMoreOlder: boolean;
  nextCursorBefore?: string;
}
```

### Commission Payment Timeline

<Warning>
Commission payments do not use the generic unified timeline controller.
</Warning>

```http
GET /commission-payments/:id/history
```

- Returns immutable `commission_payment_history` rows ordered oldest-first
- Side panel sorts by `createdAt` descending for display
- Notes from status changes are persisted on the history row `reason` field
- Free-form notes use the Activity/Note system with `targetEntityType='commission_payment'`

## Business Rules

<AccordionGroup>
  <Accordion title="Access Control & Security">
    1. **Tenant Isolation**: Timeline entries are strictly filtered by tenant context
    2. **Permission-Based Access**: Users can only view timelines for entities they have access to
  </Accordion>
  
  <Accordion title="Data Organization & Display">
    3. **Chronological Ordering**: Entries are always returned in reverse chronological order (newest first)
    4. **Source Filtering**: When `source=all`, entries from all available sources are merged and sorted by timestamp
    5. **Pagination Limits**: Maximum of 200 entries per request to prevent performance issues
    6. **Date Range Filtering**: Date filters are inclusive of boundary dates
  </Accordion>
  
  <Accordion title="Visual Enhancement">
    7. **Stage Accent Color**: Lead and deal stage-change timeline items include `accentColor` set to destination stage color
    8. **Stage Change Reason**: Stage-change items include `reason` when matching stage history record has notes
  </Accordion>
  
  <Accordion title="Special Display Cases">
    9. **Lead Disqualification Panel**: Disqualification audit messages include categorical reason in `message`. Panel displays `lead.disqualificationNotes` as visible `Reason:` detail when notes exist
    10. **Contact Merge Repointing**: During person duplicate merges, both `Activity.contactId` and direct Contact-target activities are re-pointed to surviving Contact
    11. **Company Communication Targeting**: Company-targeted activities can be logged with `Activity.contactId = null`. If user selects linked contact, validates `contact_company_role` relationship
  </Accordion>
</AccordionGroup>
