The Timeline module provides a unified view of chronological events across the PropWise CRM system. It aggregates data from multiple sources (audit logs and activities) to present a comprehensive timeline of events related to specific entities like leads, contacts, and companies.

<Note>
The Timeline module serves as a central point for tracking historical changes and activities, enabling users to understand the complete lifecycle and interaction history of CRM entities.
</Note>

## Architecture

The Timeline module follows a unified aggregation pattern, combining data from disparate sources into a single chronological view.

### Key Architectural Decisions

<CardGroup cols={2}>
  <Card title="Source Abstraction" icon="filter">
    Uses a `TimelineSource` enum to specify which data sources to query (audit, activity, or all)
  </Card>
  <Card title="Unified Response Format" icon="table">
    Normalizes different event types into a consistent timeline entry structure
  </Card>
  <Card title="Tenant-Aware" icon="building">
    All timeline queries are executed within the proper tenant context
  </Card>
  <Card title="Pagination Support" icon="page">
    Implements cursor-based pagination for efficient timeline browsing
  </Card>
</CardGroup>

The module uses a **Resolver Pattern** that delegates timeline entry formatting to specialized resolver classes (e.g., `AuditTimelineResolver`).

## Entity Relationships

<Info>
The Timeline module doesn't define its own entities but aggregates data from existing sources:
</Info>

- **AuditLog**: System audit entries tracking entity changes
- **Activity**: User-created activities and interactions  
- **ContactCompanyRole**: Relationship tracking between contacts and companies

## API Reference

### Get Entity Timeline

Retrieve the chronological timeline for a specific entity.

```http
GET /timeline/:entityType/:entityId
```

#### Path Parameters

| Parameter | Type | Description |
|-----------|------|-------------|
| `entityType` | string | Type of entity (lead, contact, company, etc.) |
| `entityId` | string | UUID of the target entity |

#### Query Parameters

| Parameter | Type | Required | Default | Description |
|-----------|------|----------|---------|-------------|
| `source` | `TimelineSource` | No | `all` | Data source filter (audit/activity/all) |
| `limit` | number | No | 50 | Number of entries per page (max 100) |
| `cursor` | string | No | - | Pagination cursor |
| `startDate` | string | No | - | Filter events from this date (ISO8601) |
| `endDate` | string | No | - | Filter events until this date (ISO8601) |

#### Response Format

## Business Rules

<AccordionGroup>
  <Accordion title="Security & Access Control">
    - **Tenant Isolation**: Timeline entries are strictly filtered by tenant context
    - **Permission-Based Access**: Users can only view timelines for entities they have access to
  </Accordion>
  
  <Accordion title="Data Ordering & Pagination">
    - **Chronological Ordering**: Entries are always returned in reverse chronological order (newest first)
    - **Pagination Limits**: Maximum of 100 entries per request to prevent performance issues
  </Accordion>
  
  <Accordion title="Filtering & Source Management">
    - **Source Filtering**: When `source=all`, entries from all sources are merged and sorted by timestamp
    - **Date Range Filtering**: When date filters are applied, they are inclusive of the boundary dates
  </Accordion>
</AccordionGroup>

## Integration Points

### Audit Module Integration

<Steps>
  <Step title="Data Consumption">
    Consumes `AuditLog` entities to show system-generated change events
  </Step>
  <Step title="Timeline Formatting">
    Uses `AuditTimelineResolver` to format audit entries into timeline format
  </Step>
  <Step title="Noise Filtering">
    Filters out implementation details including stage tracking fields (`stage_entered_at`, `stageEnteredAt`) that are automatically updated alongside stage changes
  </Step>
</Steps>

- Integrates with `Activity` entities to show user-created activities
- Supports all activity target types (leads, contacts, companies, etc.)
- Leverages `ContactCompanyRole` for relationship context

## Service Architecture

### UnifiedTimelineService Patterns

The core service implements several key patterns:

<CardGroup cols={2}>
  <Card title="Multi-Source Aggregation" icon="merge">
    Queries multiple entity types and merges results
  </Card>
  <Card title="Cursor-Based Pagination" icon="forward">
    Implements efficient pagination using timestamp cursors
  </Card>
  <Card title="Tenant Context Execution" icon="shield">
    All database operations execute within tenant boundaries
  </Card>
  <Card title="Error Handling" icon="triangle-exclamation">
    Graceful handling of missing entities and invalid queries
  </Card>

### Resolver Pattern Implementation

<Note>
Timeline formatting is delegated to specialized resolvers to maintain consistency and separation of concerns.
</Note>

- Each source type has its own resolver (e.g., `AuditTimelineResolver`)
- Resolvers handle source-specific formatting and metadata extraction
- Maintains consistency in timeline entry structure across different sources

## Performance Considerations

<Warning>
The following performance considerations are critical for optimal timeline performance:
</Warning>

- **Database Indexing**: Relies on proper indexing of timestamp fields for efficient querying
- **Pagination**: Uses cursor-based pagination to avoid expensive OFFSET queries
- **Query Optimization**: Separate queries per source type, then merges in application layer
- **Caching**: No explicit caching implemented - relies on database query performance

<Tip>
For large datasets, consider implementing date range filtering to limit the scope of timeline queries and improve response times.
</Tip>
