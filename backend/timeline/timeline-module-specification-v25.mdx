The Timeline module provides a unified view of chronological events across the PropWise CRM system. It aggregates data from multiple sources (audit logs, activities, and lead-level system events from distribution/escalation) to present a comprehensive timeline of events related to specific entities like leads, contacts, and companies.

<Note>
The Timeline module serves as a central point for tracking historical changes and activities, enabling users to understand the complete lifecycle and interaction history of CRM entities.
</Note>

## Architecture

The Timeline module follows a unified aggregation pattern, combining data from disparate sources into a single chronological view.

### Key Architectural Decisions

- **Source Abstraction**: Uses a `TimelineSource` enum to specify which data sources to query (audit, activity, or all)
- **Unified Response Format**: Normalizes different event types into a consistent timeline entry structure, with optional event details such as stage-change reasons and compact activity channel metadata
- **Tenant-Aware**: All timeline queries are executed within the proper tenant context
- **Pagination Support**: Implements page-based pagination over the merged timeline results
- **Resolver Pattern**: Delegates timeline entry formatting to specialized resolver classes (e.g., `AuditTimelineResolver`, `DistributionTimelineResolver`, `EscalationTimelineResolver`)

## Entity Relationships

The Timeline module doesn't define its own entities but aggregates data from:

<AccordionGroup>
<Accordion title="Primary Data Sources">
- **AuditLog**: System audit entries tracking entity changes
- **Activity**: User-created activities and interactions, including lightweight scheduled task/event index rows for entity-linked task/event records that do not create a typed channel activity
- **Activity Channel Tables**: `call_log`, `email_log`, `sms_log`, `meeting`, and `site_visit` rows are batch-hydrated for expandable activity details
</Accordion>

<Accordion title="Supporting Data Sources">
- **ContactCompanyRole**: Relationship tracking between contacts and companies
- **LeadStageHistory / DealStageHistory**: Stage transition history used to attach user-provided stage-change reasons to matching audit timeline entries
- **DistributionLog / EscalationActionLog**: Lead-level system events for automatic assignment and escalation activity
- **EntityTransfer**: Related lead/deal transfer request audit entries are included on the owning entity timeline
- **UnitAccessRequest**: Deal-linked unit access request audit entries are included on the related deal timeline
</Accordion>
</AccordionGroup>

## API Endpoints

### Get Entity Timeline

<CodeGroup>
```http Get Lead Timeline
GET /v1/timeline/lead/:id
```

```http Get Deal Timeline
GET /v1/timeline/deal/:id
```

```http Get Contact Timeline
GET /v1/timeline/contact/:id
```

```http Get Company Timeline
GET /v1/timeline/company/:id
```
</CodeGroup>

**Path Parameters:**
- `leadId`, `dealId`, `contactId`, or `companyId`: UUID of the target entity

**Query Parameters:**

| Parameter | Type | Required | Default | Description |
|-----------|------|----------|---------|-------------|
| `source` | `TimelineSource` | No | `all` | Data source filter (`audit`, `activity`, `system`, `all`) |
| `page` | `number` | No | 1 | Page number for merged timeline results |
| `limit` | `number` | No | 50 | Number of entries per page (max 200) |
| `from` | `ISO8601` | No | - | Filter events from this date |
| `to` | `ISO8601` | No | - | Filter events until this date |
| `activityType` | `string` | No | - | Single or comma-separated activity type filter, applied to activity entries |

**Response:**

```typescript
{
  data: UnifiedTimelineItemDto[];
  page: number;
  limit: number;
  total: number;
  totalPages: number;
  hasMore: boolean;
}
```

### Timeline Entry Structure

```typescript
{
  id: string;
  source: 'audit' | 'activity' | 'system';
  timestamp: string; // ISO8601
  message: string;
  accentColor?: string; // Optional display accent; stage changes use destination stage color
  reason?: string; // Optional detail text; stage changes use the matching stage history notes

  // Audit-only
  eventType?: TimelineEventType;
  auditAction?: AuditAction;
  tableName?: string;
  recordId?: string;

  // Activity-only
  activityType?: ActivityType;
  referenceId?: string;
  targetEntityType?: ActivityTargetType;
  targetEntityId?: string;
  isPinned?: boolean;
  isImportant?: boolean;
  activityDetails?: {
    type: ActivityType;
    // call: direction/status/outcome/phone/duration/recording/notes/purpose
    // email: direction/status/from/to/cc/bcc/subject/bodyPlain/provider/tracking/threading
    // sms: direction/status/from/to/body/provider/delivery/error
    // meeting: type/status/timing/location/link/outcome/notes/cancellation
    // site_visit: visit type/status/timing/property/feedback/notes/cancellation
  };

  // System-only (lead timelines)
  systemActionType?: 'distribution' | 'escalation';

  actor?: {
    id: string;
    name: string;
    email?: string;
    avatarUrl?: string;
  };
}
```

### Task-Scoped Feeds

<Warning>
Tasks expose their own feed endpoints from the CRM task module rather than the generic `/timeline/:entityType/:entityId` controller.
</Warning>

<Tabs>
<Tab title="Combined Timeline">
```http
GET /tasks/:id/timeline
```
Returns the combined task feed (`TaskComment` rows + task `AuditLog` events).
</Tab>

<Tab title="Activity Only">
```http
GET /tasks/:id/activity
```
Returns only audit-backed task activity events.
</Tab>

<Tab title="Comments Only">
```http
GET /tasks/:id/comments/feed
```
Returns only comments in the same keyset-paginated feed shape.
</Tab>
</Tabs>

All three endpoints share the same `TaskTimelineFeedQueryDto` contract: `limit` (default 40, max 100) and optional `before` cursor. They return `TaskTimelineFeedResponse` with ascending items for the fetched page, `hasMoreOlder`, and `nextCursorBefore`.

<Note>
Use `GET /tasks/:id/comments` only when the caller needs the legacy full comment array.
</Note>

### Commission Payment Timeline

<Info>
Commission payments do not use the generic unified timeline controller. The commission panel Timeline tab reads `GET /commission-payments/:id/history`, which returns immutable `commission_payment_history` rows ordered oldest-first.
</Info>

The side panel sorts those rows by `createdAt` descending for display so the newest event appears at the top, matching lead/deal panel timelines. Notes entered during a status change are persisted on the history row `reason` field and displayed only in this Timeline tab.

## Business Rules

<Steps>
<Step title="Tenant Isolation">
Timeline entries are strictly filtered by tenant context
</Step>

<Step title="Permission-Based Access">
Users can only view timelines for entities they have access to
</Step>

<Step title="Chronological Ordering">
Entries are always returned in reverse chronological order (newest first)
</Step>

<Step title="Source Filtering">
When `source=all`, entries from all available sources are merged and sorted by timestamp. Lead timelines include direct lead audit, related transfer audit, activity, distribution, and escalation entries; deal timelines include direct deal audit, related transfer audit, related unit-access request audit, and activity entries; contact/company timelines include audit and activity entries.
</Step>

<Step title="Pagination Limits">
Maximum of 200 entries per request to prevent performance issues
</Step>

<Step title="Date Range Filtering">
When date filters are applied, they are inclusive of the boundary dates
</Step>
</Steps>

### Advanced Business Rules

<AccordionGroup>
<Accordion title="Stage Management">
- **Stage Accent Color**: Lead and deal stage-change timeline items include `accentColor` set to the destination stage color so clients can render timeline indicators without parsing stage names or fetching stage metadata.
- **Stage Change Reason**: Lead and deal stage-change timeline items include `reason` when a matching stage history record has notes. Matching uses entity ID, from-stage ID, to-stage ID, and a short timestamp window so repeated transitions do not receive the wrong reason.
</Accordion>

<Accordion title="Lead Disqualification">
Lead disqualification audit messages include the categorical disqualification reason in `message` (for example, `Lead disqualified: Budget`). The lead panel Timeline tab displays `lead.disqualificationNotes` as the visible `Reason:` detail when notes exist, falling back to the timeline item's `reason` only when no written notes were saved.
</Accordion>

<Accordion title="Contact and Company Management">
- **Contact Merge Repointing**: Person duplicate merges can absorb duplicate Contact records. During that merge, both `Activity.contactId` and direct Contact-target activities (`targetEntityType='contact'`, `targetEntityId=<absorbedContactId>`) are re-pointed to the surviving Contact, so the Contact timeline continues to show notes, calls, meetings, and message bridge activity after the absorbed Contact is soft-deleted.
- **Company Communication Targeting**: Company-targeted communication activities can be logged directly against the company with `Activity.contactId = null`. If the user selects a linked company contact, `ActivityService` validates the `contact_company_role` relationship and stores that contact ID so the activity appears on both the company timeline and the selected contact timeline.
</Accordion>

<Accordion title="Activity Details">
- **Scheduled Task/Event Visibility**: Creating a task or calendar event from a lead/deal/contact/company panel writes an `Activity` index row for that entity when the task/event does not already create a typed call/meeting/site-visit/email activity. Task rows use `activityType='task'` with a `Scheduled task · ...` summary; generic/custom event rows use `activityType='other'` with a `Scheduled event · ...` summary.
- **Expandable Activity Details**: Activity timeline rows keep `message` as the compact header but include `activityDetails` when `referenceId` points at a supported channel row. Email details include from/to/cc/bcc, subject, bodyPlain, provider, tracking, and threading metadata; SMS details include from/to/body/status/provider/delivery/error metadata; call/meeting/site-visit details expose their channel-specific fields.
</Accordion>
</AccordionGroup>

<Warning>
HTML email bodies are intentionally not included in the unified timeline payload.
</Warning>

## Integration Points

### AI Assistant Module

The Timeline module integrates with the AI Assistant through `TimelineAiToolsProvider`:

<CardGroup cols={2}>
<Card title="Read-Only Tool" icon="robot">
`getLeadTimeline` is exposed as a read-only AI assistant tool
</Card>
<Card title="Timeline Evidence" icon="clock">
Used for lead change-history prompts like "what changed on this lead yesterday?"
</Card>
</CardGroup>

The tool wraps `UnifiedTimelineService.getLeadTimeline()` and keeps the same `source`, `from`, `to`, `activityType`, and `limit` semantics as the HTTP timeline endpoint.

<Note>
Use this tool for lead change-history prompts so the assistant reads audit/activity/system timeline evidence instead of relying on recent activities alone.
</Note>

#### AI Tool Projection

The AI tool projection sanitizes body-bearing activity rows (`note`, `email`, `sms`, `whatsapp`, `instagram`, `messenger`):

- `summary.items[].message` is replaced with a safe label such as `Note activity logged`
- `hasBody` / `bodyLength` are exposed for shape
- The original `Activity.summary` text is routed through `untrustedContent[]` so the AI bridge wraps it as `<source author_controlled="true">`

<Check>
Audit and system messages remain unchanged because they are server-formatted.
</Check>

### Task / Event AI Adjacent Flows

Lead-linked task and event tools complement the timeline surface:

<Tabs>
<Tab title="Current Work">
Use `getLeadTasks` and `getLeadEvents` for "what open tasks are due?" or "what meetings are scheduled?"
</Tab>

<Tab title="Historical Activity">
Use `getLeadTimeline` for historical change/activity questions
</Tab>
</Tabs>

### Audit Module

The Timeline module consumes `AuditLog` entities to provide historical tracking of system changes and user actions across all CRM entities.

<Tip>
The Timeline module acts as a read-only consumer of audit data, providing a user-friendly chronological view of system events without modifying the underlying audit records.
</Tip>
