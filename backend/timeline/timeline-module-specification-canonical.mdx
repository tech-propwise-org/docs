## Overview

The Timeline module provides a unified view of chronological events across the PropWise CRM system. It aggregates data from multiple sources (audit logs, activities, and lead-level system events from distribution/escalation) to present a comprehensive timeline of events related to specific entities like leads, contacts, and companies.

<Note>
The module serves as a central point for tracking historical changes and activities, enabling users to understand the complete lifecycle and interaction history of CRM entities.
</Note>

## Architecture

The Timeline module follows a unified aggregation pattern, combining data from disparate sources into a single chronological view.

### Key Architectural Decisions

<AccordionGroup>
  <Accordion title="Source Abstraction">
    Uses a `TimelineSource` enum to specify which data sources to query (audit, activity, or all)
  </Accordion>
  
  <Accordion title="Unified Response Format">
    Normalizes different event types into a consistent timeline entry structure, with optional event details such as stage-change reasons and compact activity channel metadata
  </Accordion>
  
  <Accordion title="Tenant-Aware">
    All timeline queries are executed within the proper tenant context
  </Accordion>
  
  <Accordion title="Pagination Support">
    Implements page-based pagination over the merged timeline results
  </Accordion>
  
  <Accordion title="Resolver Pattern">
    Delegates timeline entry formatting to specialized resolver classes (e.g., `AuditTimelineResolver`, `DistributionTimelineResolver`, `EscalationTimelineResolver`)
  </Accordion>
</AccordionGroup>

## Entity Relationships

<Warning>
The Timeline module doesn't define its own entities but aggregates data from multiple sources.
</Warning>

### Data Sources

- **AuditLog**: System audit entries tracking entity changes
- **Activity**: User-created activities and interactions, including lightweight scheduled task/event index rows
- **Activity Channel Tables**: `call_log`, `email_log`, `sms_log`, `meeting`, and `site_visit` rows are batch-hydrated for expandable activity details
- **ContactCompanyRole**: Relationship tracking between contacts and companies
- **LeadStageHistory / DealStageHistory**: Stage transition history used to attach user-provided stage-change reasons
- **DistributionLog / EscalationActionLog**: Lead-level system events for automatic assignment and escalation
- **EntityTransfer**: Related lead/deal transfer request audit entries
- **UnitAccessRequest**: Deal-linked unit access request audit entries

## API Endpoints

### Get Entity Timeline

```http
GET /v1/timeline/{lead|deal|contact|company}/:id
```

#### Parameters

- `leadId`, `dealId`, `contactId`, or `companyId` (path): UUID of the target entity

#### Query Parameters

| Parameter | Type | Required | Default | Description |
|-----------|------|----------|---------|-------------|
| `source` | `TimelineSource` | No | `all` | Data source filter (`audit`, `activity`, `system`, `all`) |
| `page` | `number` | No | 1 | Page number for merged timeline results |
| `limit` | `number` | No | 50 | Number of entries per page (max 200) |
| `from` | `ISO8601` | No | - | Filter events from this date |
| `to` | `ISO8601` | No | - | Filter events until this date |
| `activityType` | `string` | No | - | Single or comma-separated activity type filter |

#### Response

<CodeGroup>

```typescript Response Format
{
  data: UnifiedTimelineItemDto[];
  page: number;
  limit: number;
  total: number;
  totalPages: number;
  hasMore: boolean;
}
```

```typescript Timeline Entry Structure
{
  id: string;
  source: 'audit' | 'activity' | 'system';
  timestamp: string; // ISO8601
  message: string;
  accentColor?: string; // Optional display accent
  reason?: string; // Optional detail text

  // Audit-only fields
  eventType?: TimelineEventType;
  auditAction?: AuditAction;
  tableName?: string;
  recordId?: string;

  // Activity-only fields
  activityType?: ActivityType;
  referenceId?: string;
  targetEntityType?: ActivityTargetType;
  targetEntityId?: string;
  isPinned?: boolean;
  isImportant?: boolean;
  activityDetails?: {
    type: ActivityType;
    // Dynamic based on activity type
  };

  // System-only fields (lead timelines)
  systemActionType?: 'distribution' | 'escalation';

  actor?: {
    id: string;
    name: string;
    email?: string;
    avatarUrl?: string;
  };
}
```

</CodeGroup>

### Task-Scoped Feeds

<Info>
Tasks expose their own feed endpoints from the CRM task module rather than the generic timeline controller.
</Info>

- `GET /tasks/:id/timeline` - Returns combined task feed (comments + audit events)
- `GET /tasks/:id/activity` - Returns only audit-backed task activity events
- `GET /tasks/:id/comments/feed` - Returns only comments in keyset-paginated format

All endpoints use `TaskTimelineFeedQueryDto` with:
- `limit` (default 40, max 100)
- Optional `before` cursor

### Commission Payment Timeline

<Note>
Commission payments use a separate endpoint: `GET /commission-payments/:id/history`
</Note>

Returns immutable `commission_payment_history` rows ordered oldest-first. The side panel sorts by `createdAt` descending for display.

## DTOs

### UnifiedTimelineQueryDto

```typescript
class UnifiedTimelineQueryDto {
  source?: TimelineSource = TimelineSource.ALL;
  from?: string; // ISO8601
  to?: string; // ISO8601
  page?: number = 1;
  limit?: number = 50; // min: 1, max: 200
  activityType?: string; // single value or comma-separated list
}
```

### TimelineSource Enum

```typescript
enum TimelineSource {
  AUDIT = 'audit',
  ACTIVITY = 'activity',
  SYSTEM = 'system',
  ALL = 'all',
}
```

## Business Rules

<Steps>
  <Step title="Tenant Isolation">
    Timeline entries are strictly filtered by tenant context
  </Step>
  
  <Step title="Permission-Based Access">
    Users can only view timelines for entities they have access to
  </Step>
  
  <Step title="Chronological Ordering">
    Entries are always returned in reverse chronological order (newest first)
  </Step>
  
  <Step title="Source Filtering">
    When `source=all`, entries from all available sources are merged and sorted by timestamp
  </Step>
  
  <Step title="Pagination Limits">
    Maximum of 200 entries per request to prevent performance issues
  </Step>
</Steps>

### Advanced Rules

<AccordionGroup>
  <Accordion title="Date Range Filtering">
    When date filters are applied, they are inclusive of the boundary dates
  </Accordion>
  
  <Accordion title="Stage Accent Color">
    Lead and deal stage-change timeline items include `accentColor` set to the destination stage color for client rendering
  </Accordion>
  
  <Accordion title="Stage Change Reason">
    Stage-change timeline items include `reason` when matching stage history records have notes. Matching uses entity ID, from-stage ID, to-stage ID, and timestamp window
  </Accordion>
  
  <Accordion title="Lead Disqualification Display">
    Disqualification audit messages include categorical reason in `message`. Panel displays `lead.disqualificationNotes` as visible detail when available
  </Accordion>
  
  <Accordion title="Contact Merge Repointing">
    During person duplicate merges, both `Activity.contactId` and direct Contact-target activities are re-pointed to the surviving Contact
  </Accordion>
  
  <Accordion title="Company Communication Targeting">
    Company-targeted communications can be logged with `Activity.contactId = null`. Selected contacts are validated through `contact_company_role` relationships
  </Accordion>
  
  <Accordion title="Scheduled Task/Event Visibility">
    Creating tasks/events from entity panels writes `Activity` index rows when they don't create typed channel activities
  </Accordion>
  
  <Accordion title="Expandable Activity Details">
    Activity rows include compact `message` headers with detailed `activityDetails` for supported channel types
  </Accordion>
</AccordionGroup>

## Integration Points

### Audit Module

<Tabs>
  <Tab title="Data Consumption">
    - Consumes `AuditLog` entities for system-generated change events
    - Uses `AuditTimelineResolver` for formatting
    - Filters out implementation noise (e.g., `stage_entered_at` fields)
  </Tab>
  
  <Tab title="Stage Processing">
    - Resolves stage names and colors in batch for stage-change entries
    - Enriches entries with `reason` from matching stage history
    - Exposes destination stage color as `accentColor`
  </Tab>
  
  <Tab title="Related Events">
    - Resolves transfer and unit-access request audit by matching JSONB snapshots
    - Handles entity relationships through audit service integration
  </Tab>
</Tabs>

### CRM Module

<CardGroup cols={2}>
  <Card title="Activity Integration" icon="calendar">
    Integrates with `Activity` entities for user-created activities across all target types
  </Card>
  
  <Card title="Channel Hydration" icon="database">
    Batch-hydrates channel rows for expandable activity details without additional requests
  </Card>
  
  <Card title="Relationship Context" icon="link">
    Leverages `ContactCompanyRole` for relationship validation and context
  </Card>
  
  <Card title="Communication Targeting" icon="message">
    Supports flexible company communication targeting with optional contact selection
  </Card>
</CardGroup>

<Warning>
HTML email bodies are intentionally not included in the unified timeline payload for performance reasons.
