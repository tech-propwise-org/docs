The Timeline module provides a unified view of chronological events across the PropWise CRM system. It aggregates data from multiple sources (audit logs, activities, and lead-level system events from distribution/escalation) to present a comprehensive timeline of events related to specific entities like leads, contacts, and companies.

## Overview

The module serves as a central point for tracking historical changes and activities, enabling users to understand the complete lifecycle and interaction history of CRM entities. It combines disparate data sources into a single chronological view with consistent formatting and pagination.

### Key capabilities

- **Multi-source aggregation** - Combines audit logs, activities, and system events
- **Unified formatting** - Normalizes different event types into consistent timeline entries
- **Flexible filtering** - Supports source type, date range, and activity type filters
- **Tenant isolation** - All queries execute within proper tenant context
- **Permission-aware** - Respects user access controls for entity viewing

The Timeline module follows a unified aggregation pattern with several key architectural decisions:

## Entity relationships

The Timeline module aggregates data from existing entities rather than defining its own:

- **AuditLog** - System audit entries tracking entity changes
- **Activity** - User-created activities and interactions  
- **ContactCompanyRole** - Relationship tracking between contacts and companies
- **LeadStageHistory / DealStageHistory** - Stage transition history with user-provided reasons
- **DistributionLog / EscalationActionLog** - Lead-level system events for automatic assignment and escalation

## API endpoints

### Get entity timeline

Retrieve chronological events for a specific CRM entity.

<CodeGroup>
```http Lead Timeline
GET /v1/timeline/lead/:id
```

```http Deal Timeline  
GET /v1/timeline/deal/:id
```

```http Contact Timeline
GET /v1/timeline/contact/:id
```

```http Company Timeline
GET /v1/timeline/company/:id
```
</CodeGroup>

**Path parameters:**
- `id` (UUID) - ID of the target entity

**Query parameters:**

| Parameter | Type | Required | Default | Description |
|-----------|------|----------|---------|-------------|
| `source` | `TimelineSource` | No | `all` | Data source filter (`audit`, `activity`, `system`, `all`) |
| `page` | `number` | No | 1 | Page number for merged timeline results |
| `limit` | `number` | No | 50 | Number of entries per page (max 200) |
| `from` | `ISO8601` | No | - | Filter events from this date |
| `to` | `ISO8601` | No | - | Filter events until this date |
| `activityType` | `string` | No | - | Single or comma-separated activity type filter |

**Response format:**

```typescript
{
  data: UnifiedTimelineItemDto[];
  page: number;
  limit: number;
  total: number;
  totalPages: number;
  hasMore: boolean;
}
```

### Timeline entry structure

Each timeline entry follows this unified structure:

```typescript
{
  id: string;
  source: 'audit' | 'activity' | 'system';
  timestamp: string; // ISO8601
  message: string;
  accentColor?: string; // Display accent; stage changes use destination stage color
  reason?: string; // Detail text; stage changes use matching stage history notes

  // Audit-specific fields
  eventType?: TimelineEventType;
  auditAction?: AuditAction;
  tableName?: string;
  recordId?: string;

  // Activity-specific fields
  activityType?: ActivityType;
  referenceId?: string;
  targetEntityType?: ActivityTargetType;
  targetEntityId?: string;
  isPinned?: boolean;
  isImportant?: boolean;

  // System-specific fields (lead timelines only)
  systemActionType?: 'distribution' | 'escalation';

  actor?: {
    id: string;
    name: string;
    email?: string;
    avatarUrl?: string;
  };
}
```

## Business rules

<Steps>
  <Step title="Tenant isolation">
    Timeline entries are strictly filtered by tenant context - users can only access timelines for entities within their tenant
  </Step>
  <Step title="Permission validation">
    Users can only view timelines for entities they have explicit access permissions for
  </Step>
  <Step title="Chronological ordering">
    Entries are always returned in reverse chronological order (newest first) for consistent user experience
  </Step>
  <Step title="Source aggregation">
    When `source=all`, entries from all available sources are merged and sorted by timestamp. Lead timelines include audit, activity, distribution, and escalation entries
  </Step>
  <Step title="Pagination limits">
    Maximum of 200 entries per request to prevent performance issues and ensure responsive API calls
  </Step>
</Steps>

### Special handling rules

<AccordionGroup>
  <Accordion title="Stage change enrichment">
    **Accent Color**: Lead and deal stage-change timeline items include `accentColor` set to the destination stage color for visual consistency.
    
    **Change Reason**: Stage-change items include `reason` when a matching stage history record has notes. Matching uses entity ID, from-stage ID, to-stage ID, and timestamp window.
  </Accordion>

  <Accordion title="Contact merge repointing">
    During person duplicate merges that absorb Contact records:
    - Both `Activity.contactId` and direct Contact-target activities are re-pointed to surviving Contact
    - Contact timeline continues showing notes, calls, meetings after absorbed Contact is soft-deleted
  </Accordion>

  <Accordion title="Date range filtering">
    When date filters (`from`/`to`) are applied:
    - Boundary dates are inclusive
    - Filters apply to the `timestamp` field across all source types
    - Combined with source filtering for precise control
  </Accordion>
</AccordionGroup>

## Integration points

The Timeline module integrates with multiple system components:

### Audit module integration

- Consumes `AuditLog` entities for system-generated change events
- Uses `AuditTimelineResolver` to format audit entries
- Filters implementation noise (e.g., `stage_entered_at` auto-updates)
- Enriches stage-change entries with destination stage colors and user-provided reasons

### CRM module integration

- Integrates with `Activity` entities for user-created activities
- Supports all activity target types (leads, contacts, companies)
- Leverages `ContactCompanyRole` for relationship context
- Handles activity filtering by type and importance flags

### Distribution module integration

- Lead timelines include `DistributionLog` rows as `source='system'`
- `DistributionTimelineResolver` maps assignment outcomes to timeline messages
- Covers pool assignments, fallback routing, and manual assignments

### Escalation module integration

- Lead timelines include `EscalationActionLog` rows with `systemActionType='escalation'`
- `EscalationTimelineResolver` formats notifications, redistributions, temperature changes
- Use `source=system` filter for distribution/escalation-only view

### User module integration

<Warning>
All timeline entries require proper user context for actor information and permission validation.
</Warning>

- Enriches timeline entries with user information (name, email, avatar)
- Ensures proper user references in timeline metadata
- Handles cases where original actors may no longer exist

## Service patterns

### UnifiedTimelineService

The core service implements several key patterns:

<Tabs>
  <Tab title="Multi-Source Aggregation">
    - Queries multiple entity types in parallel
    - Merges results by timestamp in application layer  
    - Maintains source type metadata for filtering
  </Tab>
  <Tab title="Page-Based Pagination">
    - Fetches bounded results from each source
    - Merges and sorts in memory
    - Applies page/limit slicing to unified result
  </Tab>
  <Tab title="Tenant Context">
    - All database operations execute within tenant boundaries
    - Automatic filtering by tenant ID across all sources
    - No cross-tenant data leakage possible
  </Tab>
</Tabs>

### Resolver pattern

Timeline formatting uses specialized resolvers for consistency:

```typescript
// Example resolver interface
interface TimelineResolver {
  resolve(entry: SourceEntry): UnifiedTimelineItemDto;
  canHandle(entry: SourceEntry): boolean;
}
```

- **AuditTimelineResolver** - Formats audit log entries with action descriptions
- **DistributionTimelineResolver** - Converts distribution events to user-friendly messages  
- **EscalationTimelineResolver** - Formats escalation actions with context
- **ActivityTimelineResolver** - Standardizes activity display format

## Performance considerations

<Tip>
The Timeline module is optimized for read-heavy workloads with proper indexing and query patterns.
</Tip>

### Database optimization

- **Indexing Strategy** - Relies on timestamp field indexes for efficient chronological queries
- **Query Optimization** - Separate queries per source type avoid complex joins
- **Bounded Fetching** - Limits source queries before merge to control memory usage

### Application layer

- **Memory Management** - Merges bounded result sets rather than full tables
- **Pagination Strategy** - Page/limit applied after merge reduces database load
- **No Caching** - Relies on database query performance and proper indexing

### Scaling considerations

- Timeline queries scale with entity activity volume
- Pagination prevents memory issues with high-activity entities  
- Source filtering allows targeted queries for specific use cases
- Date range filtering reduces query scope for historical analysis
