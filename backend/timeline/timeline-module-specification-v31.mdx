## Overview

The Timeline module provides a unified view of chronological events across the PropWise CRM system. It aggregates data from multiple sources (audit logs, activities, and lead-level system events from distribution/escalation) to present a comprehensive timeline of events related to specific entities like leads, contacts, and companies.

<Info>
The module serves as a central point for tracking historical changes and activities, enabling users to understand the complete lifecycle and interaction history of CRM entities.
</Info>

## Architecture

The Timeline module follows a unified aggregation pattern, combining data from disparate sources into a single chronological view.

### Key Architectural Decisions

<CardGroup cols={2}>
  <Card title="Source Abstraction" icon="filter">
    Uses a `TimelineSource` enum to specify which data sources to query (audit, activity, or all)
  </Card>
  <Card title="Unified Response Format" icon="arrows-merge">
    Normalizes different event types into a consistent timeline entry structure
  </Card>
  <Card title="Tenant-Aware" icon="shield">
    All timeline queries are executed within the proper tenant context
  </Card>
  <Card title="Resolver Pattern" icon="gear">
    Delegates timeline entry formatting to specialized resolver classes
  </Card>
</CardGroup>

<Note>
The module implements page-based pagination over the merged timeline results with optional event details such as stage-change reasons and compact activity channel metadata.
</Note>

## Entity Relationships

The Timeline module doesn't define its own entities but aggregates data from multiple sources:

<AccordionGroup>
  <Accordion title="Core Data Sources">
    - **AuditLog**: System audit entries tracking entity changes
    - **Activity**: User-created activities and interactions, including lightweight scheduled task/event index rows
    - **Activity Channel Tables**: `call_log`, `email_log`, `sms_log`, `meeting`, and `site_visit` rows are batch-hydrated for expandable activity details
  </Accordion>
  
  <Accordion title="Relationship Tracking">
    - **ContactCompanyRole**: Relationship tracking between contacts and companies
    - **LeadStageHistory / DealStageHistory**: Stage transition history used to attach user-provided stage-change reasons
  </Accordion>
  
  <Accordion title="System Events">
    - **DistributionLog / EscalationActionLog**: Lead-level system events for automatic assignment and escalation activity
    - **EntityTransfer**: Related lead/deal transfer request audit entries
    - **UnitAccessRequest**: Deal-linked unit access request audit entries
  </Accordion>
  
  <Accordion title="Property Interest Events">
    - **LeadPropertyInterest / DealPropertyInterest**: Add/update/remove events included as curated audit entries
    - Timeline events: `LEAD_PROPERTY_INTEREST_ADDED`, `LEAD_PROPERTY_INTEREST_UPDATED`, `LEAD_PROPERTY_INTEREST_REMOVED`, `DEAL_PROPERTY_INTEREST_ADDED`, `DEAL_PROPERTY_INTEREST_UPDATED`, `DEAL_PROPERTY_INTEREST_REMOVED`
  </Accordion>
</AccordionGroup>

## API Endpoints

### Get Entity Timeline

<CodeGroup>
```http GET /v1/timeline/{entity}/:id
GET /v1/timeline/lead/:id
GET /v1/timeline/deal/:id
GET /v1/timeline/contact/:id
GET /v1/timeline/company/:id
```
</CodeGroup>

**Path Parameters:**
- `leadId`, `dealId`, `contactId`, or `companyId` (UUID): Target entity identifier

**Query Parameters:**

| Parameter | Type | Required | Default | Description |
|-----------|------|----------|---------|-------------|
| `source` | `TimelineSource` | No | `all` | Data source filter (`audit`, `activity`, `system`, `all`) |
| `page` | `number` | No | 1 | Page number for merged timeline results |
| `limit` | `number` | No | 50 | Number of entries per page (max 200) |
| `from` | `ISO8601` | No | - | Filter events from this date |
| `to` | `ISO8601` | No | - | Filter events until this date |
| `activityType` | `string` | No | - | Single or comma-separated activity type filter |

**Response Structure:**

```typescript
{
  data: UnifiedTimelineItemDto[];
  page: number;
  limit: number;
  total: number;
  totalPages: number;
  hasMore: boolean;
}
```

### Timeline Entry Structure

```typescript
{
  id: string;
  source: 'audit' | 'activity' | 'system';
  timestamp: string; // ISO8601
  message: string;
  accentColor?: string; // Optional display accent
  reason?: string; // Optional detail text

  // Audit-specific fields
  eventType?: TimelineEventType;
  auditAction?: AuditAction;
  tableName?: string;
  recordId?: string;

  // Activity-specific fields
  activityType?: ActivityType;
  referenceId?: string;
  targetEntityType?: ActivityTargetType;
  targetEntityId?: string;
  isPinned?: boolean;
  isImportant?: boolean;
  activityDetails?: {
    type: ActivityType;
    // Dynamic details based on activity type
  };

  // System-specific fields (lead timelines only)
  systemActionType?: 'distribution' | 'escalation';

  actor?: {
    id: string;
    name: string;
    email?: string;
    avatarUrl?: string;
  };
}
```

### Task-Scoped Feeds

<Warning>
Tasks use dedicated endpoints from the CRM task module instead of the generic timeline controller.
</Warning>

<Tabs>
  <Tab title="Combined Feed">
    ```http
    GET /tasks/:id/timeline
    ```
    Returns combined task feed (`TaskComment` rows + task `AuditLog` events)
  </Tab>
  
  <Tab title="Activity Only">
    ```http
    GET /tasks/:id/activity
    ```
    Returns only audit-backed task activity events
  </Tab>
  
  <Tab title="Comments Feed">
    ```http
    GET /tasks/:id/comments/feed
    ```
    Returns only comments in keyset-paginated feed shape
  </Tab>
</Tabs>

All endpoints share the `TaskTimelineFeedQueryDto` contract with `limit` (default 40, max 100) and optional `before` cursor.

### Commission Payment Timeline

<Note>
Commission payments use a dedicated endpoint: `GET /commission-payments/:id/history`
</Note>

Returns immutable `commission_payment_history` rows ordered oldest-first. The client sorts by `createdAt` descending for display.

### Org-Wide Activity Feed

The `UnifiedTimelineService.getActivityFeedInTransaction()` method powers the dashboard Activity Feed widget with org-wide, actor-scoped aggregation.

**Key Features:**
- Merges audit lifecycle events and logged CRM activities
- Supports actor filtering by user IDs
- Uses cursor-based pagination
- Returns `UnifiedActivityFeedItem[]` format

## DTOs

### UnifiedTimelineQueryDto

```typescript
class UnifiedTimelineQueryDto {
  source?: TimelineSource = TimelineSource.ALL;
  from?: string; // ISO8601
  to?: string; // ISO8601
  page?: number = 1;
  limit?: number = 50; // min: 1, max: 200
  activityType?: string; // single value or comma-separated list
}
```

### TimelineSource Enum

```typescript
enum TimelineSource {
  AUDIT = 'audit',
  ACTIVITY = 'activity',
  SYSTEM = 'system',
  ALL = 'all',
}
```

## Business Rules

<Steps>
  <Step title="Tenant Isolation">
    Timeline entries are strictly filtered by tenant context
  </Step>
  
  <Step title="Permission-Based Access">
    Users can only view timelines for entities they have access to
  </Step>
  
  <Step title="Chronological Ordering">
    Entries are always returned in reverse chronological order (newest first)
  </Step>
  
  <Step title="Source Filtering">
    When `source=all`, entries from all available sources are merged and sorted by timestamp
  </Step>
</Steps>

### Advanced Rules

<AccordionGroup>
  <Accordion title="Pagination & Performance">
    - Maximum of 200 entries per request to prevent performance issues
    - Date range filtering is inclusive of boundary dates
  </Accordion>
  
  <Accordion title="Stage Change Handling">
    - Stage-change timeline items include `accentColor` set to the destination stage color
    - Include `reason` when matching stage history record has notes
    - Matching uses entity ID, stage IDs, and timestamp window
  </Accordion>
  
  <Accordion title="Lead Disqualification Display">
    - Disqualification audit messages include categorical reason in `message`
    - Panel displays `lead.disqualificationNotes` as visible detail
    - Falls back to timeline item's `reason` when no notes exist
  </Accordion>
  
  <Accordion title="Contact Merge Repointing">
    - Person duplicate merges can absorb duplicate Contact records
    - Both `Activity.contactId` and direct Contact-target activities are re-pointed to surviving Contact
    - Timeline continues to show activities after absorbed Contact is soft-deleted
  </Accordion>
  
  <Accordion title="Company Communication Targeting">
    - Company-targeted communications can be logged with `Activity.contactId = null`
    - If user selects linked company contact, validates `contact_company_role` relationship
    - Activity appears on both company and selected contact timelines
  </Accordion>
</AccordionGroup>
