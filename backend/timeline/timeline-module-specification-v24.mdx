## Overview

The Timeline module provides a unified view of chronological events across the PropWise CRM system. It aggregates data from multiple sources (audit logs, activities, and lead-level system events from distribution/escalation) to present a comprehensive timeline of events related to specific entities like leads, contacts, and companies.

<Note>
The Timeline module serves as a central point for tracking historical changes and activities, enabling users to understand the complete lifecycle and interaction history of CRM entities.
</Note>

**Last Updated:** 2026-05-23  
**Status:** Active

## Architecture

The Timeline module follows a unified aggregation pattern, combining data from disparate sources into a single chronological view.

### Key Architectural Decisions

<CardGroup cols={2}>
  <Card title="Source Abstraction" icon="filter">
    Uses a `TimelineSource` enum to specify which data sources to query (audit, activity, or all)
  </Card>
  <Card title="Unified Response Format" icon="table">
    Normalizes different event types into a consistent timeline entry structure
  </Card>
  <Card title="Tenant-Aware" icon="shield">
    All timeline queries are executed within the proper tenant context
  </Card>
  <Card title="Resolver Pattern" icon="code">
    Delegates timeline entry formatting to specialized resolver classes
  </Card>
</CardGroup>

<Info>
The module includes optional event details such as stage-change reasons and compact activity channel metadata for enhanced context.
</Info>

## Entity Relationships

The Timeline module doesn't define its own entities but aggregates data from multiple sources:

<AccordionGroup>
  <Accordion title="Core Data Sources">
    - **AuditLog**: System audit entries tracking entity changes
    - **Activity**: User-created activities and interactions, including lightweight scheduled task/event index rows
    - **Activity Channel Tables**: `call_log`, `email_log`, `sms_log`, `meeting`, and `site_visit` rows are batch-hydrated for expandable activity details
  </Accordion>
  
  <Accordion title="Relationship Tracking">
    - **ContactCompanyRole**: Relationship tracking between contacts and companies
    - **LeadStageHistory / DealStageHistory**: Stage transition history used to attach user-provided stage-change reasons
  </Accordion>
  
  <Accordion title="System Events">
    - **DistributionLog / EscalationActionLog**: Lead-level system events for automatic assignment and escalation activity
    - **EntityTransfer**: Related lead/deal transfer request audit entries
    - **UnitAccessRequest**: Deal-linked unit access request audit entries
  </Accordion>
</AccordionGroup>

## API Endpoints

### Get Entity Timeline

```http
GET /v1/timeline/{lead|deal|contact|company}/:id
```

**Path Parameters:**
- `leadId`, `dealId`, `contactId`, or `companyId` (UUID): Target entity identifier

**Query Parameters:**

| Parameter | Type | Required | Default | Description |
|-----------|------|----------|---------|-------------|
| `source` | `TimelineSource` | No | `all` | Data source filter (`audit`, `activity`, `system`, `all`) |
| `page` | `number` | No | 1 | Page number for merged timeline results |
| `limit` | `number` | No | 50 | Number of entries per page (max 200) |
| `from` | `ISO8601` | No | - | Filter events from this date |
| `to` | `ISO8601` | No | - | Filter events until this date |
| `activityType` | `string` | No | - | Single or comma-separated activity type filter |

**Response Structure:**

<CodeGroup>
```typescript Response Schema
{
  data: UnifiedTimelineItemDto[];
  page: number;
  limit: number;
  total: number;
  totalPages: number;
  hasMore: boolean;
}
```

```typescript Timeline Entry
{
  id: string;
  source: 'audit' | 'activity' | 'system';
  timestamp: string; // ISO8601
  message: string;
  accentColor?: string; // Optional display accent
  reason?: string; // Optional detail text

  // Audit-only fields
  eventType?: TimelineEventType;
  auditAction?: AuditAction;
  tableName?: string;
  recordId?: string;

  // Activity-only fields
  activityType?: ActivityType;
  referenceId?: string;
  targetEntityType?: ActivityTargetType;
  targetEntityId?: string;
  isPinned?: boolean;
  isImportant?: boolean;
  activityDetails?: {
    type: ActivityType;
    // Dynamic fields based on activity type
  };

  // System-only fields
  systemActionType?: 'distribution' | 'escalation';

  actor?: {
    id: string;
    name: string;
    email?: string;
    avatarUrl?: string;
  };
}
```
</CodeGroup>

### Task-Scoped Feeds

<Warning>
Tasks expose their own feed endpoints from the CRM task module rather than using the generic timeline controller.
</Warning>

<Steps>
  <Step title="Task Timeline">
    `GET /tasks/:id/timeline` - Returns combined task feed (`TaskComment` rows + task `AuditLog` events)
  </Step>
  <Step title="Task Activity">
    `GET /tasks/:id/activity` - Returns only audit-backed task activity events
  </Step>
  <Step title="Task Comments">
    `GET /tasks/:id/comments/feed` - Returns only comments in keyset-paginated feed shape
  </Step>
</Steps>

All endpoints share the `TaskTimelineFeedQueryDto` contract with `limit` (default 40, max 100) and optional `before` cursor.

### Commission Payment Timeline

<Note>
Commission payments use a dedicated endpoint: `GET /commission-payments/:id/history` instead of the unified timeline controller.
</Note>

Returns immutable `commission_payment_history` rows ordered oldest-first. The side panel sorts by `createdAt` descending for display.

## Business Rules

<Tabs>
  <Tab title="Access & Security">
    ### Access Control
    1. **Tenant Isolation**: Timeline entries are strictly filtered by tenant context
    2. **Permission-Based Access**: Users can only view timelines for entities they have access to

    ### Data Limits
    3. **Pagination Limits**: Maximum of 200 entries per request to prevent performance issues
    4. **Date Range Filtering**: Date filters are inclusive of boundary dates
  </Tab>

  <Tab title="Data Processing">
    ### Ordering & Filtering
    5. **Chronological Ordering**: Entries are always returned in reverse chronological order (newest first)
    6. **Source Filtering**: When `source=all`, entries from all available sources are merged and sorted by timestamp

    ### Enhanced Context
    7. **Stage Accent Color**: Stage-change timeline items include `accentColor` set to destination stage color
    8. **Stage Change Reason**: Stage-change items include `reason` when matching stage history record has notes
  </Tab>

  <Tab title="Special Cases">
    ### Display Rules
    9. **Lead Disqualification**: Disqualification messages include categorical reason in `message`
    10. **Contact Merge Repointing**: Person duplicate merges re-point activities to surviving Contact

    ### Communication Rules
    11. **Company Communication**: Company-targeted activities can be logged with or without linked contact
    12. **Scheduled Visibility**: Tasks/events create Activity index rows for timeline visibility
    13. **Expandable Details**: Activity rows include `activityDetails` when `referenceId` exists
  </Tab>
</Tabs>

## Integration Points

### AI Assistant Module

<CardGroup cols={1}>
  <Card title="Timeline AI Tools Provider" icon="robot">
    `TimelineAiToolsProvider` exposes `getLeadTimeline` as a read-only AI assistant tool for lead change-history prompts
  </Card>

<Steps>
  <Step title="Tool Registration">
    The provider wraps `UnifiedTimelineService.getLeadTimeline()` with the same semantics as the HTTP endpoint
  </Step>
  <Step title="Content Sanitization">
    Body-bearing activity rows are sanitized with safe labels and `untrustedContent[]` routing
  </Step>
  <Step title="Complementary Tools">
    Use `getLeadTasks` and `getLeadEvents` for current scheduled work, `getLeadTimeline` for historical questions
  </Step>
</Steps>

<Tip>
Use the timeline tool for questions like "what changed on this lead yesterday?" to provide audit/activity/system evidence to the AI assistant.
</Tip>

### Audit Module Integration

The Timeline module consumes `AuditLog` entries and transforms them into user-friendly timeline events, providing context and formatting for system-generated audit trails.

<Check>
All audit events maintain their original timestamp precision while being formatted for human readability in the unified timeline view.
</Check>
