## Overview

The Timeline module provides a unified view of chronological events across the PropWise CRM system. It aggregates data from multiple sources (audit logs, activities, and lead-level system events from distribution/escalation) to present a comprehensive timeline of events related to specific entities like leads, contacts, and companies.

<Info>
The module serves as a central point for tracking historical changes and activities, enabling users to understand the complete lifecycle and interaction history of CRM entities.
</Info>

**Last Updated:** 2026-05-23  
**Status:** Active

## Architecture

The Timeline module follows a unified aggregation pattern, combining data from disparate sources into a single chronological view.

### Key Architectural Decisions

<CardGroup cols={2}>
  <Card title="Source Abstraction" icon="layer-group">
    Uses a `TimelineSource` enum to specify which data sources to query (audit, activity, system, or all)
  </Card>
  <Card title="Unified Response Format" icon="align-justify">
    Normalizes different event types into a consistent timeline entry structure with optional event details
  </Card>
  <Card title="Tenant-Aware" icon="building">
    All timeline queries are executed within the proper tenant context
  </Card>
  <Card title="Pagination Support" icon="file-lines">
    Implements page-based pagination over merged timeline results
  </Card>
</CardGroup>

### Resolver Pattern

The module delegates timeline entry formatting to specialized resolver classes:

- `AuditTimelineResolver` - Handles audit log entries
- `DistributionTimelineResolver` - Processes distribution events
- `EscalationTimelineResolver` - Formats escalation actions

## API Endpoints

### Get Entity Timeline

Retrieves the chronological timeline for a specific entity.

```http
GET /v1/timeline/{lead|deal|contact|company}/:id
```

#### Path Parameters

- `leadId`, `dealId`, `contactId`, or `companyId` - UUID of the target entity

#### Query Parameters

| Parameter | Type | Required | Default | Description |
|-----------|------|----------|---------|-------------|
| `source` | `TimelineSource` | No | `all` | Data source filter (`audit`, `activity`, `system`, `all`) |
| `page` | `number` | No | 1 | Page number for merged timeline results |
| `limit` | `number` | No | 50 | Number of entries per page (max 200) |
| `from` | `ISO8601` | No | - | Filter events from this date |
| `to` | `ISO8601` | No | - | Filter events until this date |
| `activityType` | `string` | No | - | Single or comma-separated activity type filter |

#### Response

```typescript
{
  data: UnifiedTimelineItemDto[];
  page: number;
  limit: number;
  total: number;
  totalPages: number;
  hasMore: boolean;
}
```

#### Timeline Entry Structure

```typescript
{
  id: string;
  source: 'audit' | 'activity' | 'system';
  timestamp: string; // ISO8601
  message: string;
  accentColor?: string; // Optional display accent; stage changes use destination stage color
  reason?: string; // Optional detail text; stage changes use matching stage history notes

  // Audit-only fields
  eventType?: TimelineEventType;
  auditAction?: AuditAction;
  tableName?: string;
  recordId?: string;

  // Activity-only fields
  activityType?: ActivityType;
  referenceId?: string;
  targetEntityType?: ActivityTargetType;
  targetEntityId?: string;
  isPinned?: boolean;
  isImportant?: boolean;
  activityDetails?: {
    type: ActivityType;
    // call: direction/status/outcome/phone/duration/recording/notes/purpose
    // email: direction/status/from/to/cc/bcc/subject/bodyPlain/provider/tracking/threading
    // sms: direction/status/from/to/body/provider/delivery/error
    // meeting: type/status/timing/location/link/outcome/notes/cancellation
    // site_visit: visit type/status/timing/property/feedback/notes/cancellation
  };

  // System-only fields (lead timelines)
  systemActionType?: 'distribution' | 'escalation';

  // Actor information
  actor?: {
    id: string;
    name: string;
    email?: string;
    avatarUrl?: string;
  };
}
```

### Task-Scoped Feeds

<Warning>
Tasks expose their own feed endpoints from the CRM task module rather than the generic `/timeline/:entityType/:entityId` controller.
</Warning>

<Tabs>
  <Tab title="Combined Timeline">
    ```http
    GET /tasks/:id/timeline
    ```
    Returns combined task feed (`TaskComment` rows + task `AuditLog` events)
  </Tab>
  
  <Tab title="Activity Only">
    ```http
    GET /tasks/:id/activity
    ```
    Returns only audit-backed task activity events
  </Tab>
  
  <Tab title="Comments Feed">
    ```http
    GET /tasks/:id/comments/feed
    ```
    Returns only comments in keyset-paginated feed shape
  </Tab>
</Tabs>

#### Query Parameters

All three endpoints share the `TaskTimelineFeedQueryDto` contract:

- `limit` - Default 40, max 100
- `before` - Optional cursor for pagination

#### Response Format

Returns `TaskTimelineFeedResponse` with:
- Ascending items for the fetched page
- `hasMoreOlder` - Boolean indicating more historical items
- `nextCursorBefore` - Cursor for next page

<Note>
Use `GET /tasks/:id/comments` only when the caller needs the legacy full comment array.
</Note>

### Commission Payment Timeline

<Warning>
Commission payments do not use the generic unified timeline controller.
</Warning>

The commission panel Timeline tab reads immutable history records:

```http
GET /commission-payments/:id/history
```

#### Behavior

<Steps>
  <Step title="Data Source">
    Returns immutable `commission_payment_history` rows ordered oldest-first
  </Step>
  
  <Step title="Display Ordering">
    Side panel sorts rows by `createdAt` descending so newest event appears at the top
  </Step>
  
  <Step title="Status Change Notes">
    Notes entered during status change are persisted on the history row `reason` field and displayed only in Timeline tab
  </Step>
  
  <Step title="Free-Form Notes">
    Free-form commission panel notes use Activity/Note system (`targetEntityType='commission_payment'`) and are isolated to the Notes tab
  </Step>
</Steps>

</Info>

#### Sources Merged

<AccordionGroup>
  <Accordion title="Audit Lifecycle Events">
    Curated from `audit_log` for specific tables:
    - `lead`
    - `deal`
    - `contact`
    - `company`
    - `entity_stakeholder`
    - `entity_transfer`
    - `unit_access_request`
    
    <Note>Distribution/escalation system rows are not included</Note>
  </Accordion>
  
  <Accordion title="CRM Activity Rows">
    Logged activities including:
    - Notes
    - Calls
    - Emails
    - Meetings
    - Site visits
    - Other activity types
  </Accordion>
</AccordionGroup>

#### Actor Scope

| Scope | Filter Behavior |
|-------|-----------------|
| **Self** | `actorUserIds = [userId]` |
| **Team** | `actorUserIds = self ∪ team members` |
| **Org** | `actorUserIds = null` (no filter) |

The dashboard derives scope from the caller's resolved CRM/Sales permissions.

#### Resolution Process

<Steps>
  <Step title="Audit Resolution">
    Each audit row is resolved through `AuditTimelineResolver.resolveBatch` into:
    - `kind` (`TimelineEventType`)
    - Human-readable `message` (e.g., "Lead created", "Moved from New to Contacted")
    - Optional `accentColor` (destination stage color)
  </Step>
  
  <Step title="Cursor Pagination">
    - Each source independently ordered `(occurredAt DESC, id DESC)`
    - Method over-fetches `limit + 1` per source
    - Merges in memory and sorts globally
    - Slices top `limit` items
    - Returns `{ items, hasMore }`
  </Step>
  
  <Step title="Cursor Encoding">
    Caller (dashboard) encodes last item's `{ occurredAt, id }` into opaque cursor
  </Step>
</Steps>

#### Return Shape

```typescript
UnifiedActivityFeedItem[] {
  id: string;
  source: 'audit' | 'activity';
  kind: TimelineEventType;
  message: string;
  occurredAt: string;
  accentColor?: string;
  targetEntityType?: string;
  targetEntityId?: string;
  actor?: {
    id: string;
    name: string;
    email?: string;
    avatarUrl?: string;
  };
}
```

<Tip>
The dashboard module (`DashboardService.getActivityFeed`) maps `UnifiedActivityFeedItem` into `DashboardActivityItemDto`. See `DASHBOARD_MODULE_SPECIFICATION.md` for details.

## Business Rules

### Security & Access Control

<CardGroup cols={2}>
  <Card title="Tenant Isolation" icon="shield-halved">
    Timeline entries are strictly filtered by tenant context
  </Card>
  <Card title="Permission-Based Access" icon="lock">
    Users can only view timelines for entities they have access to
  </Card>
</CardGroup>

### Data Presentation

<Steps>
  <Step title="Chronological Ordering">
    Entries are always returned in reverse chronological order (newest first)
  </Step>
  
  <Step title="Source Filtering">
    When `source=all`, entries from all available sources are merged and sorted by timestamp:
    - **Lead timelines**: Direct lead audit + related transfer audit + related lead-property-interest audit + activity + distribution + escalation
    - **Deal timelines**: Direct deal audit + related transfer audit + related unit-access request audit + related deal-property-interest audit + activity
    - **Contact/Company timelines**: Audit + activity entries
  </Step>
  
  <Step title="Pagination Limits">
    Maximum of 200 entries per request to prevent performance issues
  </Step>
  
  <Step title="Date Range Filtering">
    When date filters are applied, they are inclusive of the boundary dates
  </Step>
</Steps>

### Stage Change Handling

<Accordion title="Stage Accent Color">
Lead and deal stage-change timeline items include `accentColor` set to the destination stage color so clients can render timeline indicators without parsing stage names or fetching stage metadata.

**Architecture Details:**

- System stages are GLOBAL rows with shared IDs across orgs
- Per-org colors live in `lead_stage_override` / `deal_stage_override` (override-table architecture, `CRM_MODULE_SPECIFICATION.md §2.6`)
- `AuditTimelineResolver.batchResolveNames` loads the override map **per distinct org in the batch**
- Map structure: `Map<orgId, Map<stageId, color>>`
- Each entry's accent is resolved against **its own** `entry.organizationId`
- Falls back to global stage color if no override exists

<Warning>
This multi-tenant resolution is critical because system-admin audit queries run under RLS bypass. A single page can span multiple tenants, and a shared map would bleed one org's override color onto another org's events.
</Warning>
</Accordion>

<Accordion title="Stage Change Reason">
Lead and deal stage-change timeline items include `reason` when a matching stage history record has notes.

**Matching Logic:**
- Entity ID
- From-stage ID
- To-stage ID
- Short timestamp window

This ensures repeated transitions do not receive the wrong reason.
</Accordion>

### Special Display Cases

<Accordion title="Lead Disqualification Panel Display">
Lead disqualification audit messages include the categorical disqualification reason in `message` (e.g., `Lead disqualified: Budget`).

**Timeline Tab Display Logic:**
1. Display `lead.disqualificationNotes` as visible `Reason:` detail when notes exist
2. Fall back to timeline item's `reason` only when no written notes were saved
</Accordion>

<Accordion title="Contact Merge Repointing">
Person duplicate merges can absorb duplicate Contact records.

**During Merge:**
- Both `Activity.contactId` and direct Contact-target activities are re-pointed to the surviving Contact
- Target activities: `targetEntityType='contact'`, `targetEntityId=<absorbedContactId>`
- Result: Surviving contact retains complete activity history from absorbed contacts
</Accordion>

## Implementation Notes

<Note>
The Timeline module is read-only. It aggregates and presents data but does not create or modify timeline entries directly. Entry creation happens at the source (audit logs, activities, distribution/escalation systems).
</Note>

<Tip>
For optimal performance, use specific source filters when you know which type of events you need, rather than always querying `source=all`.
</Tip>

<Check>
All timeline queries respect tenant boundaries and user permissions automatically through the resolver pattern and tenant-aware service methods.
</Check>
