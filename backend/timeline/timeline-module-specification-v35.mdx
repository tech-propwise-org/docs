## Overview

The Timeline module provides a unified view of chronological events across the PropWise CRM system. It aggregates data from multiple sources (audit logs, activities, and lead-level system events from distribution/escalation) to present a comprehensive timeline of events related to specific entities like leads, contacts, and companies.

The module serves as a central point for tracking historical changes and activities, enabling users to understand the complete lifecycle and interaction history of CRM entities.

## Architecture

The Timeline module follows a unified aggregation pattern, combining data from disparate sources into a single chronological view.

### Key architectural decisions

<AccordionGroup>
  <Accordion title="Source abstraction">
    Uses a `TimelineSource` enum to specify which data sources to query (audit, activity, or all)
  </Accordion>
  
  <Accordion title="Unified response format">
    Normalizes different event types into a consistent timeline entry structure, with optional event details such as stage-change reasons and compact activity channel metadata
  </Accordion>
  
  <Accordion title="Tenant-aware">
    All timeline queries are executed within the proper tenant context
  </Accordion>
  
  <Accordion title="Pagination support">
    Implements page-based pagination over the merged timeline results
  </Accordion>
  
  <Accordion title="Resolver pattern">
    Delegates timeline entry formatting to specialized resolver classes (e.g., `AuditTimelineResolver`, `DistributionTimelineResolver`, `EscalationTimelineResolver`)
  </Accordion>
</AccordionGroup>

## Entity relationships

The Timeline module doesn't define its own entities but aggregates data from:

- **AuditLog**: System audit entries tracking entity changes
- **Activity**: User-created activities and interactions, including lightweight scheduled task/event index rows for entity-linked task/event records that do not create a typed channel activity
- **Activity Channel Tables**: `call_log`, `email_log`, `sms_log`, `meeting`, and `site_visit` rows are batch-hydrated for expandable activity details
- **ContactCompanyRole**: Relationship tracking between contacts and companies
- **LeadStageHistory / DealStageHistory**: Stage transition history used to attach user-provided stage-change reasons to matching audit timeline entries
- **DistributionLog / EscalationActionLog**: Lead-level system events for automatic assignment and escalation activity
- **EntityTransfer**: Related lead/deal transfer request audit entries are included on the owning entity timeline
- **UnitAccessRequest**: Deal-linked unit access request audit entries are included on the related deal timeline
- **LeadPropertyInterest / DealPropertyInterest**: Add / update / remove of a lead's or deal's property interest is included on the owning lead/deal timeline as curated audit entries (matched via `lead_property_interest.lead_id` / `deal_property_interest.deal_id` JSONB snapshots)

### Timeline events for property interests

- `LEAD_PROPERTY_INTEREST_ADDED`
- `LEAD_PROPERTY_INTEREST_UPDATED`
- `LEAD_PROPERTY_INTEREST_REMOVED`
- `DEAL_PROPERTY_INTEREST_ADDED`
- `DEAL_PROPERTY_INTEREST_UPDATED`
- `DEAL_PROPERTY_INTEREST_REMOVED`

## API endpoints

### Get entity timeline

Retrieve the chronological timeline for a specific entity (lead, deal, contact, or company).

```http
GET /v1/timeline/{lead|deal|contact|company}/:id
```

#### Path parameters

| Parameter | Type | Description |
|-----------|------|-------------|
| `entityType` | `string` | Entity type: `lead`, `deal`, `contact`, or `company` |
| `id` | `UUID` | UUID of the target entity |

#### Query parameters

| Parameter | Type | Required | Default | Description |
|-----------|------|----------|---------|-------------|
| `source` | `TimelineSource` | No | `all` | Data source filter (`audit`, `activity`, `system`, `all`) |
| `page` | `number` | No | 1 | Page number for merged timeline results |
| `limit` | `number` | No | 50 | Number of entries per page (max 200) |
| `from` | `ISO8601` | No | - | Filter events from this date |
| `to` | `ISO8601` | No | - | Filter events until this date |
| `activityType` | `string` | No | - | Single or comma-separated activity type filter, applied to activity entries |

#### Response

```typescript
{
  data: UnifiedTimelineItemDto[];
  page: number;
  limit: number;
  total: number;
  totalPages: number;
  hasMore: boolean;
}
```

### Task-scoped feeds

<Warning>
Tasks expose their own feed endpoints from the CRM task module rather than the generic `/timeline/:entityType/:entityId` controller.
</Warning>

Tasks provide three specialized endpoints:

| Endpoint | Description | Content |
|----------|-------------|---------|
| `GET /tasks/:id/timeline` | Combined task feed | `TaskComment` rows + task `AuditLog` events |
| `GET /tasks/:id/activity` | Audit-only feed | Task audit events only |
| `GET /tasks/:id/comments/feed` | Comments-only feed | Comments in keyset-paginated feed shape |

<Info>
All three endpoints share the same `TaskTimelineFeedQueryDto` contract with `limit` (default 40, max 100) and optional `before` cursor.
</Info>

#### Response format

All task feed endpoints return `TaskTimelineFeedResponse`:

```typescript
{
  items: Array<TaskComment | AuditEvent>; // Ascending order for fetched page
  hasMoreOlder: boolean;
  nextCursorBefore?: string;
}
```

<Tip>
Use `GET /tasks/:id/comments` only when the caller needs the legacy full comment array.
</Tip>

### Commission payment timeline

<Warning>
Commission payments do not use the generic unified timeline controller.
</Warning>

The commission panel Timeline tab reads immutable history from a dedicated endpoint:

```http
GET /commission-payments/:id/history
```

#### Response characteristics

<Steps>
  <Step title="Data source">
    Returns immutable `commission_payment_history` rows ordered oldest-first
  </Step>
  
  <Step title="Display sorting">
    The side panel sorts rows by `createdAt` descending for display, so the newest event appears at the top
  </Step>
  
  <Step title="Status change notes">
    Notes entered during a status change are persisted on the history row `reason` field and displayed only in the Timeline tab
  </Step>
  
  <Step title="Free-form notes">
    Free-form commission panel notes use the Activity/Note system (`targetEntityType='commission_payment'`) and are isolated to the panel Notes tab
  </Step>
</Steps>

### Org-wide actor activity feed (dashboard)

The dashboard **Activity Feed** widget uses `UnifiedTimelineService.getActivityFeedInTransaction(em, orgId, { actorUserIds, cursor, limit })` to power an org-wide, actor-scoped "what happened" feed.

<Note>
Unlike entity-scoped timelines, this is an **org-wide, actor-scoped** feed for dashboard consumption.
</Note>

#### Data sources merged

<Tabs>
  <Tab title="Audit events">
    Lifecycle events from `audit_log` curated to `ACTIVITY_FEED_AUDIT_TABLES`:
    - `lead`
    - `deal`
    - `contact`
    - `company`
    - `entity_stakeholder`
    - `entity_transfer`
    - `unit_access_request`
  </Tab>
  
  <Tab title="Activity events">
    Logged CRM `activity` rows including:
    - Note
    - Call
    - Email
    - Meeting
    - SMS
    - Site Visit
  </Tab>
  
  <Tab title="Exclusions">
    Distribution/escalation system rows are **not** included in the activity feed.
  </Tab>
</Tabs>

#### Actor scope filtering

| Scope | Actor filter | Description |
|-------|-------------|-------------|
| Self | `[userId]` | Current user's activities only |
| Team | `[userId, ...teamMemberIds]` | User and their team members |
| Org | `null` | No actor filter (organization-wide) |

<Info>
The dashboard derives `actorUserIds` from the caller's resolved CRM/Sales scope.
</Info>

#### Audit resolution

Each audit row is resolved through `AuditTimelineResolver.resolveBatch` into:

- **kind**: `TimelineEventType` (e.g., `LEAD_CREATED`, `DEAL_STAGE_CHANGED`)
- **message**: Human-readable description (e.g., "Lead created", "Moved from New to Contacted", "Sarah assigned", "Deal won")
- **accentColor**: Optional color (destination stage color for stage changes)

#### Cursor pagination

<Steps>
  <Step title="Independent ordering">
    Each source is ordered `(occurredAt DESC, id DESC)`
  </Step>
  
  <Step title="Over-fetching">
    Method over-fetches `limit + 1` per source
  </Step>
  
  <Step title="In-memory merge">
    Results are merged in memory and sorted globally
  </Step>
  
  <Step title="Slicing">
    Top `limit` items are returned with `hasMore` indicator
  </Step>
  
  <Step title="Cursor encoding">
    Caller encodes last item's `{ occurredAt, id }` into opaque cursor
  </Step>
</Steps>

## DTOs

### UnifiedTimelineQueryDto

Query parameters for entity timeline requests.

```typescript
class UnifiedTimelineQueryDto {
  source?: TimelineSource = TimelineSource.ALL;
  from?: string; // ISO8601
  to?: string; // ISO8601
  page?: number = 1;
  limit?: number = 50; // min: 1, max: 200
  activityType?: string; // single value or comma-separated list
}
```

### TimelineSource enum

Source filter for timeline queries.

```typescript
enum TimelineSource {
  AUDIT = 'audit',      // Audit log entries only
  ACTIVITY = 'activity', // Activity entries only
  SYSTEM = 'system',    // System events (distribution/escalation)
  ALL = 'all',          // All sources merged
}
```

## Business rules

<AccordionGroup>
  <Accordion title="1. Tenant isolation">
    Timeline entries are strictly filtered by tenant context. Users can only see events within their organization.
  </Accordion>

  <Accordion title="2. Permission-based access">
    Users can only view timelines for entities they have access to based on their role and permissions.
  </Accordion>

  <Accordion title="3. Chronological ordering">
    Entries are always returned in reverse chronological order (newest first) for consistent user experience.
  </Accordion>

  <Accordion title="4. Source filtering">
    When `source=all`, entries from all available sources are merged and sorted by timestamp:
    
    - **Lead timelines**: Direct lead audit + related transfer audit + related lead-property-interest audit + activity + distribution + escalation entries
    - **Deal timelines**: Direct deal audit + related transfer audit + related unit-access request audit + related deal-property-interest audit + activity entries
    - **Contact/Company timelines**: Audit + activity entries
  </Accordion>

  <Accordion title="5. Pagination limits">
    Maximum of 200 entries per request to prevent performance issues.
  </Accordion>

  <Accordion title="6. Date range filtering">
    When date filters are applied, they are inclusive of the boundary dates.
  </Accordion>

  <Accordion title="7. Stage accent color">
    Lead and deal stage-change timeline items include `accentColor` set to the destination stage color so clients can render timeline indicators without parsing stage names or fetching stage metadata.
    
    **Override resolution:**
    - System stages are GLOBAL rows whose id is shared across orgs
    - Per-org color lives in `lead_stage_override` / `deal_stage_override` (override-table architecture)
    - `AuditTimelineResolver.batchResolveNames` loads the override map **per distinct org in the batch** (keyed `Map<orgId, Map<stageId, color>>`)
    - Each entry's accent is resolved against **its own** `entry.organizationId` (falling back to global stage color)
    
    <Warning>
    This matters because system-admin audit queries run under RLS bypass and a single page can span multiple tenants. A single shared map would otherwise bleed one org's override color onto another org's events.
    </Warning>
  </Accordion>

  <Accordion title="8. Stage change reason">
    Lead and deal stage-change timeline items include `reason` when a matching stage history record has notes.
    
    **Matching criteria:**
    - Entity ID
    - From-stage ID
    - To-stage ID
    - Short timestamp window (to prevent wrong reason on repeated transitions)
  </Accordion>

  <Accordion title="9. Lead disqualification panel display">
    Lead disqualification audit messages include the categorical disqualification reason in `message` (e.g., `Lead disqualified: Budget`).
    
    **Display logic:**
    - The lead panel Timeline tab displays `lead.disqualificationNotes` as the visible `Reason:` detail when notes exist
    - Falls back to the timeline item's `reason` only when no written notes were saved
  </Accordion>

  <Accordion title="10. Contact merge repointing">
    Person duplicate merges can absorb duplicate Contact records.
    
    **During merge:**
    - Both `Activity.contactId` and direct Contact-target activities (`targetEntityType='contact'`, `targetEntityId=<absorbedContactId>`) are re-pointed to the surviving Contact
    - This ensures the complete activity history remains accessible on the surviving contact's timeline
  </Accordion>
</AccordionGroup>

## Performance considerations

<Tip>
The Timeline module implements several optimizations for performance:
</Tip>

- **Batch resolution**: Entity names and metadata are resolved in batches to minimize database queries
- **Source-specific queries**: When a specific source is requested, only that source is queried
- **Index optimization**: Timeline queries rely on proper indexing of timestamp and entity ID columns
- **Pagination enforcement**: Strict limits prevent unbounded result sets
- **Cursor-based pagination**: Task and activity feeds use keyset pagination for efficient deep pagination
