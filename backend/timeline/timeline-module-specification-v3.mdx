## Overview

The Timeline module provides a unified view of chronological events across the PropWise CRM system. It aggregates data from multiple sources (audit logs and activities) to present a comprehensive timeline of events related to specific entities like leads, contacts, and companies.

<Info>
The module serves as a central point for tracking historical changes and activities, enabling users to understand the complete lifecycle and interaction history of CRM entities.
</Info>

**Last Updated:** 2024-12-19  
**Status:** Active

## Architecture

The Timeline module follows a unified aggregation pattern, combining data from disparate sources into a single chronological view.

### Key architectural decisions

<CardGroup cols={2}>
  <Card title="Source Abstraction" icon="database">
    Uses a `TimelineSource` enum to specify which data sources to query (audit, activity, or all)
  </Card>
  <Card title="Unified Response Format" icon="code">
    Normalizes different event types into a consistent timeline entry structure
  </Card>
  <Card title="Tenant-Aware" icon="shield">
    All timeline queries are executed within the proper tenant context
  </Card>
  <Card title="Pagination Support" icon="list">
    Implements cursor-based pagination for efficient timeline browsing
  </Card>
</CardGroup>

### Resolver pattern

<Note>
The module delegates timeline entry formatting to specialized resolver classes (e.g., `AuditTimelineResolver`) to maintain consistency across different data sources.
</Note>

## Entity Relationships

The Timeline module doesn't define its own entities but aggregates data from:

- **AuditLog**: System audit entries tracking entity changes
- **Activity**: User-created activities and interactions
- **ContactCompanyRole**: Relationship tracking between contacts and companies

## API Endpoints

### Get entity timeline

```http
GET /timeline/:entityType/:entityId
```

**Path parameters:**
- `entityType` (string): Type of entity (lead, contact, company, etc.)
- `entityId` (string): UUID of the target entity

**Query parameters:**

| Parameter | Type | Required | Default | Description |
|-----------|------|----------|---------|-------------|
| `source` | `TimelineSource` | No | `all` | Data source filter (audit/activity/all) |
| `limit` | `number` | No | 50 | Number of entries per page (max 100) |
| `cursor` | `string` | No | - | Pagination cursor |
| `startDate` | `ISO8601` | No | - | Filter events from this date |
| `endDate` | `ISO8601` | No | - | Filter events until this date |

**Response structure:**

<CodeGroup>
```typescript Response Format
{
  items: UnifiedTimelineEntryDto[];
  hasNextPage: boolean;
  nextCursor?: string;
  totalCount?: number;
}
```

  source: TimelineSource;
  title: string;
  description?: string;
  user?: {
    id: string;

</CodeGroup>

## Business Rules

<Steps>
  <Step title="Tenant isolation">
    Timeline entries are strictly filtered by tenant context to ensure data security and separation.
  </Step>
  
  <Step title="Permission-based access">
    Users can only view timelines for entities they have proper access permissions for.
  </Step>
  
  <Step title="Chronological ordering">
    Entries are always returned in reverse chronological order (newest first) for optimal user experience.
  </Step>
  
  <Step title="Source filtering">
    When `source=all`, entries from all sources are merged and sorted by timestamp for a unified view.
  </Step>
  
  <Step title="Pagination limits">
    Maximum of 100 entries per request to prevent performance issues and ensure responsive API behavior.
  </Step>
  
  <Step title="Date range filtering">
    When date filters are applied, they are inclusive of the boundary dates for precise querying.
  </Step>
</Steps>

## Integration Points

### Audit module

<Tabs>
  <Tab title="Data Consumption">
    - Consumes `AuditLog` entities to show system-generated change events
    - Uses `AuditTimelineResolver` to format audit entries into timeline format
  </Tab>
  
  <Tab title="Noise Filtering">
    Filters out noise from implementation details including stage tracking fields (`stage_entered_at`, `stageEnteredAt`) that are automatically updated alongside stage changes
  </Tab>
</Tabs>

### CRM module

- Integrates with `Activity` entities to show user-created activities
- Supports all activity target types (leads, contacts, companies, etc.)
- Leverages `ContactCompanyRole` for relationship context

### User module

- Enriches timeline entries with user information (name, email)
- Ensures proper user references in timeline metadata

### Auth module

- Enforces JWT authentication on all timeline endpoints
- Validates user permissions for accessed entities

## Service Patterns

### UnifiedTimelineService

The core service implements several key patterns:

<CardGroup cols={2}>
  <Card title="Multi-Source Aggregation" icon="merge">
    Queries multiple entity types and merges results into a unified timeline
  </Card>
  <Card title="Cursor-Based Pagination" icon="arrow-right">
    Implements efficient pagination using timestamp cursors
  </Card>
  <Card title="Tenant Context Execution" icon="building">
    All database operations execute within tenant boundaries
  </Card>
  <Card title="Error Handling" icon="exclamation-triangle">
    Graceful handling of missing entities and invalid queries
  </Card>
</CardGroup>

### Resolver pattern

Timeline formatting is delegated to specialized resolvers:

- Each source type has its own resolver (e.g., `AuditTimelineResolver`)
- Resolvers handle source-specific formatting and metadata extraction
- Maintains consistency in timeline entry structure across different sources

## Performance Considerations

<Warning>
The following performance aspects are critical for optimal timeline functionality:
</Warning>

- **Database Indexing**: Relies on proper indexing of timestamp fields for efficient querying
- **Pagination**: Uses cursor-based pagination to avoid expensive OFFSET queries
- **Query Optimization**: Separate queries per source type, then merges in application layer
- **Caching**: No explicit caching implemented - relies on database query performance

<Tip>
Consider implementing Redis caching for frequently accessed timelines to improve response times for high-traffic entities.
</Tip>
