## Overview

The Timeline module provides a unified view of chronological events across the PropWise CRM system. It aggregates data from multiple sources (audit logs, activities, and lead-level system events from distribution/escalation) to present a comprehensive timeline of events related to specific entities like leads, contacts, and companies.

<Info>
The module serves as a central point for tracking historical changes and activities, enabling users to understand the complete lifecycle and interaction history of CRM entities.
</Info>

## Architecture

The Timeline module follows a unified aggregation pattern, combining data from disparate sources into a single chronological view.

### Key Architectural Decisions

<CardGroup cols={2}>
  <Card title="Source Abstraction" icon="filter">
    Uses a `TimelineSource` enum to specify which data sources to query (audit, activity, or all)
  </Card>
  <Card title="Unified Response Format" icon="arrows-merge">
    Normalizes different event types into a consistent timeline entry structure
  </Card>
  <Card title="Tenant-Aware" icon="shield">
    All timeline queries are executed within the proper tenant context
  </Card>
  <Card title="Resolver Pattern" icon="code">
    Delegates timeline entry formatting to specialized resolver classes
  </Card>
</CardGroup>

<Note>
The module implements page-based pagination over merged timeline results and includes optional event details such as stage-change reasons and compact activity channel metadata.
</Note>

## Entity Relationships

The Timeline module doesn't define its own entities but aggregates data from multiple sources:

<AccordionGroup>
  <Accordion title="Core Data Sources">
    - **AuditLog**: System audit entries tracking entity changes
    - **Activity**: User-created activities and interactions
    - **Activity Channel Tables**: `call_log`, `email_log`, `sms_log`, `meeting`, and `site_visit` rows
    - **ContactCompanyRole**: Relationship tracking between contacts and companies
  </Accordion>
  
  <Accordion title="Stage History Sources">
    - **LeadStageHistory / DealStageHistory**: Stage transition history used to attach user-provided stage-change reasons
  </Accordion>
  
  <Accordion title="System Event Sources">
    - **DistributionLog / EscalationActionLog**: Lead-level system events for automatic assignment and escalation
    - **EntityTransfer**: Related lead/deal transfer request audit entries
    - **UnitAccessRequest**: Deal-linked unit access request audit entries
  </Accordion>
  
  <Accordion title="Property Interest Sources">
    - **LeadPropertyInterest / DealPropertyInterest**: Add/update/remove events for property interests
    
    Timeline events include:
    - `LEAD_PROPERTY_INTEREST_ADDED`
    - `LEAD_PROPERTY_INTEREST_UPDATED` 
    - `LEAD_PROPERTY_INTEREST_REMOVED`
    - `DEAL_PROPERTY_INTEREST_ADDED`
    - `DEAL_PROPERTY_INTEREST_UPDATED`
    - `DEAL_PROPERTY_INTEREST_REMOVED`
  </Accordion>
</AccordionGroup>

## API Endpoints

### Get Entity Timeline

Retrieve a paginated timeline for a specific entity.

```http
GET /v1/timeline/{lead|deal|contact|company}/:id
```

#### Parameters

- `leadId`, `dealId`, `contactId`, or `companyId` (path): UUID of the target entity

#### Query Parameters

| Parameter | Type | Required | Default | Description |
|-----------|------|----------|---------|-------------|
| `source` | `TimelineSource` | No | `all` | Data source filter (`audit`, `activity`, `system`, `all`) |
| `page` | `number` | No | 1 | Page number for merged timeline results |
| `limit` | `number` | No | 50 | Number of entries per page (max 200) |
| `from` | `ISO8601` | No | - | Filter events from this date |
| `to` | `ISO8601` | No | - | Filter events until this date |
| `activityType` | `string` | No | - | Single or comma-separated activity type filter |

#### Response Structure

<CodeGroup>
```typescript Response Format
{
  data: UnifiedTimelineItemDto[];
  page: number;
  limit: number;
  total: number;
  totalPages: number;
  hasMore: boolean;
}
```

```typescript Timeline Entry Structure
{
  id: string;
  source: 'audit' | 'activity' | 'system';
  timestamp: string; // ISO8601
  message: string;
  accentColor?: string; // Optional display accent
  reason?: string; // Optional detail text

  // Audit-only fields
  eventType?: TimelineEventType;
  auditAction?: AuditAction;
  tableName?: string;
  recordId?: string;

  // Activity-only fields
  activityType?: ActivityType;
  referenceId?: string;
  targetEntityType?: ActivityTargetType;
  targetEntityId?: string;
  isPinned?: boolean;
  isImportant?: boolean;
  activityDetails?: {
    type: ActivityType;
    // Expandable channel-specific details
  };

  // System-only fields (lead timelines)
  systemActionType?: 'distribution' | 'escalation';

  actor?: {
    id: string;
    name: string;
    email?: string;
    avatarUrl?: string;
  };
}
```
</CodeGroup>

### Task-Scoped Feeds

<Warning>
Tasks expose their own feed endpoints from the CRM task module rather than using the generic timeline controller.
</Warning>

Available task timeline endpoints:

- `GET /tasks/:id/timeline` - Combined task feed (comments + audit events)
- `GET /tasks/:id/activity` - Audit-backed task activity events only
- `GET /tasks/:id/comments/feed` - Comments only in keyset-paginated format

<Tabs>
  <Tab title="Query Parameters">
    All endpoints share the `TaskTimelineFeedQueryDto` contract:
    - `limit` (default 40, max 100)
    - `before` (optional cursor)
  </Tab>
  
  <Tab title="Response Format">
    Returns `TaskTimelineFeedResponse` with:
    - `items` (ascending order for the fetched page)
    - `hasMoreOlder` (boolean)
    - `nextCursorBefore` (cursor for pagination)
  </Tab>
</Tabs>

### Commission Payment Timeline

<Info>
Commission payments use a specialized endpoint: `GET /commission-payments/:id/history`
</Info>

This endpoint returns immutable `commission_payment_history` rows ordered oldest-first. The client sorts by `createdAt` descending for display, ensuring newest events appear at the top.

### Org-Wide Actor Activity Feed

The `UnifiedTimelineService.getActivityFeedInTransaction()` method powers the dashboard Activity Feed widget with org-wide, actor-scoped event aggregation.

<Steps>
  <Step title="Source Merging">
    Combines audit lifecycle events from curated tables (`lead`, `deal`, `contact`, `company`, `entity_stakeholder`, `entity_transfer`, `unit_access_request`) with logged CRM activity rows.
  </Step>
  
  <Step title="Actor Filtering">
    Filters by `actorUserIds` for both audit and activity sources. Null value means no actor filter (org scope).
  </Step>
  
  <Step title="Cursor Pagination">
    Each source ordered by `(occurredAt DESC, id DESC)` with independent over-fetching and in-memory merging.
  </Step>
</Steps>

## Business Rules

<Check>
**Security & Access Control**
</Check>

1. **Tenant Isolation**: Timeline entries are strictly filtered by tenant context
2. **Permission-Based Access**: Users can only view timelines for entities they have access to

<Check>
**Data Organization & Pagination**
</Check>

3. **Chronological Ordering**: Entries are always returned in reverse chronological order (newest first)
4. **Source Filtering**: When `source=all`, entries from all available sources are merged and sorted by timestamp
5. **Pagination Limits**: Maximum of 200 entries per request to prevent performance issues
6. **Date Range Filtering**: When date filters are applied, they are inclusive of the boundary dates

<Check>
**Visual Enhancement Features**
</Check>

7. **Stage Accent Color**: Lead and deal stage-change timeline items include `accentColor` set to the destination stage color. System stages use a per-org override architecture for color customization.

8. **Stage Change Reason**: Stage-change timeline items include `reason` when matching stage history records have notes. Matching uses entity ID, stage IDs, and timestamp windows.

<Check>
**Special Display Rules**
</Check>

9. **Lead Disqualification Panel Display**: Disqualification audit messages include categorical reasons in `message`. The timeline displays `lead.disqualificationNotes` as the visible reason detail when available.

10. **Contact Merge Repointing**: During person duplicate merges, both `Activity.contactId` and direct Contact-target activities are re-pointed to the surviving Contact to maintain timeline continuity.

<Tip>
The Timeline module's unified approach ensures consistent event presentation across all CRM entities while maintaining the flexibility to filter and customize views based on specific use cases.
</Tip>
