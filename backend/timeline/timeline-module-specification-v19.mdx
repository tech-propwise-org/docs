## Overview

The Timeline module provides a unified view of chronological events across the PropWise CRM system. It aggregates data from multiple sources (audit logs, activities, and lead-level system events from distribution/escalation) to present a comprehensive timeline of events related to specific entities like leads, contacts, and companies. 

The module serves as a central point for tracking historical changes and activities, enabling users to understand the complete lifecycle and interaction history of CRM entities.

## Architecture

The Timeline module follows a unified aggregation pattern, combining data from disparate sources into a single chronological view. Key architectural decisions include:

<CardGroup cols={2}>
  <Card title="Source abstraction" icon="database">
    Uses a `TimelineSource` enum to specify which data sources to query (audit, activity, or all)
  </Card>
  <Card title="Unified response format" icon="layer-group">
    Normalizes different event types into a consistent timeline entry structure
  </Card>
  <Card title="Tenant-aware" icon="shield">
    All timeline queries are executed within the proper tenant context
  </Card>
  <Card title="Pagination support" icon="list">
    Implements page-based pagination over the merged timeline results
  </Card>
</CardGroup>

<Note>
The module uses a resolver pattern that delegates timeline entry formatting to specialized resolver classes (e.g., `AuditTimelineResolver`, `DistributionTimelineResolver`, `EscalationTimelineResolver`).
</Note>

## Entity Relationships

The Timeline module doesn't define its own entities but aggregates data from:

<AccordionGroup>
  <Accordion title="AuditLog">
    System audit entries tracking entity changes
  </Accordion>
  <Accordion title="Activity">
    User-created activities and interactions
  </Accordion>
  <Accordion title="ContactCompanyRole">
    Relationship tracking between contacts and companies
  </Accordion>
  <Accordion title="LeadStageHistory / DealStageHistory">
    Stage transition history used to attach user-provided stage-change reasons to matching audit timeline entries
  </Accordion>
  <Accordion title="DistributionLog / EscalationActionLog">
    Lead-level system events for automatic assignment and escalation activity
  </Accordion>
</AccordionGroup>

## API Endpoints

### Get Entity Timeline

<CodeGroup>
```http GET /v1/timeline/{lead|deal|contact|company}/:id
GET /v1/timeline/lead/:id
GET /v1/timeline/deal/:id
GET /v1/timeline/contact/:id
GET /v1/timeline/company/:id
```

```javascript Example Request
const response = await fetch('/v1/timeline/lead/123e4567-e89b-12d3-a456-426614174000?source=all&page=1&limit=50', {
  headers: {
    'Authorization': 'Bearer <token>'
  }
});
```
</CodeGroup>

#### Parameters

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `leadId`, `dealId`, `contactId`, or `companyId` | UUID | Yes | UUID of the target entity (path parameter) |

#### Query Parameters

| Parameter | Type | Required | Default | Description |
|-----------|------|----------|---------|-------------|
| `source` | `TimelineSource` | No | `all` | Data source filter (`audit`, `activity`, `system`, `all`) |
| `page` | `number` | No | 1 | Page number for merged timeline results |
| `limit` | `number` | No | 50 | Number of entries per page (max 200) |
| `from` | `ISO8601` | No | - | Filter events from this date |
| `to` | `ISO8601` | No | - | Filter events until this date |
| `activityType` | `string` | No | - | Single or comma-separated activity type filter, applied to activity entries |

#### Response Format

```typescript
{
  data: UnifiedTimelineItemDto[];
  page: number;
  limit: number;
  total: number;
  totalPages: number;
  hasMore: boolean;
}
```

#### Timeline Entry Structure

<CodeGroup>
```typescript Base Structure
{
  id: string;
  source: 'audit' | 'activity' | 'system';
  timestamp: string; // ISO8601
  message: string;
  accentColor?: string; // Optional display accent; stage changes use destination stage color
  reason?: string; // Optional detail text; stage changes use the matching stage history notes

  actor?: {
    id: string;
    name: string;
    email?: string;
    avatarUrl?: string;
  };
}
```

```typescript Audit Fields
// Audit-only fields
{
  eventType?: TimelineEventType;
  auditAction?: AuditAction;
  tableName?: string;
  recordId?: string;
}
```

```typescript Activity Fields
// Activity-only fields
{
  activityType?: ActivityType;
  referenceId?: string;
  targetEntityType?: ActivityTargetType;
  targetEntityId?: string;
  isPinned?: boolean;
  isImportant?: boolean;
}
```

```typescript System Fields
// System-only fields (lead timelines)
{
  systemActionType?: 'distribution' | 'escalation';
}
```
</CodeGroup>

### Task-Scoped Feeds

<Warning>
Tasks expose their own feed endpoints from the CRM task module rather than the generic `/timeline/:entityType/:entityId` controller.
</Warning>

<Tabs>
  <Tab title="Combined Task Feed">
    ```http
    GET /tasks/:id/timeline
    ```
    Returns the combined task feed (`TaskComment` rows + task `AuditLog` events).
  </Tab>
  <Tab title="Task Activity Only">
    ```http
    GET /tasks/:id/activity
    ```
    Returns only audit-backed task activity events.
  </Tab>
  <Tab title="Comments Feed">
    ```http
    GET /tasks/:id/comments/feed
    ```
    Returns only comments in the same keyset-paginated feed shape.
  </Tab>
</Tabs>

All three endpoints share the same `TaskTimelineFeedQueryDto` contract:
- `limit` (default 40, max 100)
- Optional `before` cursor

They return `TaskTimelineFeedResponse` with ascending items for the fetched page, `hasMoreOlder`, and `nextCursorBefore`.

<Note>
Use `GET /tasks/:id/comments` only when the caller needs the legacy full comment array.
</Note>

## Business Rules

<Steps>
  <Step title="Tenant isolation">
    Timeline entries are strictly filtered by tenant context
  </Step>
  <Step title="Permission-based access">
    Users can only view timelines for entities they have access to
  </Step>
  <Step title="Chronological ordering">
    Entries are always returned in reverse chronological order (newest first)
  </Step>
  <Step title="Source filtering">
    When `source=all`, entries from all available sources are merged and sorted by timestamp. Lead timelines include audit, activity, distribution, and escalation entries; deal/contact/company timelines include audit and activity entries.
  </Step>
  <Step title="Pagination limits">
    Maximum of 200 entries per request to prevent performance issues
  </Step>
  <Step title="Date range filtering">
    When date filters are applied, they are inclusive of the boundary dates
  </Step>
</Steps>

### Enhanced Features

<AccordionGroup>
  <Accordion title="Stage Accent Color">
    Lead and deal stage-change timeline items include `accentColor` set to the destination stage color so clients can render timeline indicators without parsing stage names or fetching stage metadata.
  </Accordion>
  <Accordion title="Stage Change Reason">
    Lead and deal stage-change timeline items include `reason` when a matching stage history record has notes. Matching uses entity ID, from-stage ID, to-stage ID, and a short timestamp window so repeated transitions do not receive the wrong reason.
  </Accordion>
  <Accordion title="Contact Merge Repointing">
    Person duplicate merges can absorb duplicate Contact records. During that merge, both `Activity.contactId` and direct Contact-target activities (`targetEntityType='contact'`, `targetEntityId=<absorbedContactId>`) are re-pointed to the surviving Contact, so the Contact timeline continues to show notes, calls, meetings, and message bridge activity after the absorbed Contact is soft-deleted.
  </Accordion>
</AccordionGroup>

## Integration Points

### Audit Module

<CardGroup cols={1}>
  <Card title="Core Integration" icon="clock">
    - Consumes `AuditLog` entities to show system-generated change events
    - Uses `AuditTimelineResolver` to format audit entries into timeline format
    - Filters out noise from implementation details including stage tracking fields (`stage_entered_at`, `stageEnteredAt`)
    - Resolves stage names and destination stage colors in batch for lead/deal stage-change entries
    - Stage-change audit entries are enriched with `reason` from matching `LeadStageHistory` or `DealStageHistory` notes
  </Card>
</CardGroup>

### CRM Module

- Integrates with `Activity` entities to show user-created activities
- Supports all activity target types (leads, contacts, companies, etc.)
- Leverages `ContactCompanyRole` for relationship context

### Distribution Module

- Lead timelines include `DistributionLog` rows as `source='system'`
- `DistributionTimelineResolver` maps assignment, pool, and fallback outcomes to display-ready timeline messages

### Escalation Module

<Tip>
Use `source=system` to fetch only distribution/escalation entries for a lead timeline.
</Tip>

- Lead timelines include `EscalationActionLog` rows as `source='system'` with `systemActionType='escalation'`
- `EscalationTimelineResolver` formats notification, redistribution, temperature-change, and tag-add actions

### User Module

- Enriches timeline entries with user information (name, email)
- Ensures proper user references in timeline metadata

### Auth Module

- Enforces JWT authentication on all timeline endpoints
- Validates user permissions for accessed entities

## Service Patterns

### UnifiedTimelineService

The core service implements several key patterns:

<CardGroup cols={2}>
  <Card title="Multi-source aggregation" icon="merge">
    Queries multiple entity types and merges results
  </Card>
  <Card title="Page-based pagination" icon="page">
    Applies `page` and `limit` after source entries are merged and sorted
  </Card>
  <Card title="Tenant context execution" icon="building">
    All database operations execute within tenant boundaries
  </Card>
  <Card title="Error handling" icon="triangle-exclamation">
    Graceful handling of missing entities and invalid queries
  </Card>
</CardGroup>

### Resolver Pattern

Timeline formatting is delegated to specialized resolvers:

- Each source type has its own resolver (e.g., `AuditTimelineResolver`)
- Resolvers handle source-specific formatting and metadata extraction
- Maintains consistency in timeline entry structure across different sources

## Performance Considerations

<Warning>
The system relies on proper database indexing and bounded queries for optimal performance.
</Warning>

<AccordionGroup>
  <Accordion title="Database Indexing">
    Relies on proper indexing of timestamp fields for efficient querying
  </Accordion>
  <Accordion title="Pagination Strategy">
    Fetches bounded source lists, merges in memory, then applies page/limit slicing to the unified result
  </Accordion>
  <Accordion title="Query Optimization">
    Separate queries per source type, then merges in application layer
  </Accordion>
  <Accordion title="Caching">
    No explicit caching implemented - relies on database query performance
  </Accordion>
</AccordionGroup>
