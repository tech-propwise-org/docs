# Timeline Module

The Timeline module provides a unified view of chronological events across the PropWise CRM system. It aggregates data from multiple sources (audit logs and activities) to present a comprehensive timeline of events related to specific entities like leads, contacts, and companies.

<Info>
**Last Updated:** 2024-12-19  
**Status:** Active
</Info>

## Overview

The module serves as a central point for tracking historical changes and activities, enabling users to understand the complete lifecycle and interaction history of CRM entities. It combines disparate data sources into a single chronological view with consistent formatting and pagination.

## Architecture

The Timeline module follows a unified aggregation pattern with several key architectural decisions:

### Design Patterns

  </Card>

  </Card>
  <Card title="Tenant-Aware Operations" icon="shield">

### Key Features

- **Pagination Support**: Implements cursor-based pagination for efficient timeline browsing
- **Multi-Source Aggregation**: Combines audit logs and activities into unified chronological view
- **Permission Integration**: Respects user access controls for entity visibility

## Entity Relationships

### Data Sources

- **AuditLog**: System audit entries tracking entity changes
- **Activity**: User-created activities and interactions

## API Endpoints

### Get Entity Timeline

Retrieves a paginated timeline of events for a specific entity.

```http
GET /timeline/:entityType/:entityId
```

#### Path Parameters

| Parameter | Type | Description |
|-----------|------|-------------|
| `entityType` | `string` | Type of entity (lead, contact, company, etc.) |
| `entityId` | `string` | UUID of the target entity |

#### Query Parameters

| Parameter | Type | Required | Default | Description |
|-----------|------|----------|---------|-------------|
| `source` | `TimelineSource` | No | `all` | Data source filter (audit/activity/all) |
| `limit` | `number` | No | 50 | Number of entries per page (max 100) |
| `cursor` | `string` | No | - | Pagination cursor |
| `startDate` | `ISO8601` | No | - | Filter events from this date |
| `endDate` | `ISO8601` | No | - | Filter events until this date |

#### Response Format

<CodeGroup>

```typescript Response Structure
{
  items: UnifiedTimelineEntryDto[];
  hasNextPage: boolean;
  nextCursor?: string;
  totalCount?: number;
}
```

```typescript Timeline Entry
{
  id: string;
  timestamp: string; // ISO8601
  type: TimelineEventType;
  source: TimelineSource;
  title: string;
  description?: string;
  user?: {
    id: string;
    name: string;
    email: string;
  };
  metadata?: Record<string, any>;
}
```

</CodeGroup>

#### Example Request

```http
GET /timeline/lead/123e4567-e89b-12d3-a456-426614174000?source=all&limit=25&startDate=2024-01-01T00:00:00Z
```

## DTOs

### UnifiedTimelineQueryDto

<Accordion title="Query Parameters DTO">

```typescript
class UnifiedTimelineQueryDto {
  source?: TimelineSource = TimelineSource.ALL;
  limit?: number = 50; // min: 1, max: 100
  cursor?: string;
  startDate?: string; // ISO8601
  endDate?: string; // ISO8601
}
```

**Validation Rules:**
- `limit`: Must be between 1 and 100
- `startDate`/`endDate`: Must be valid ISO8601 formatted dates
- `cursor`: Opaque pagination token

</Accordion>

### TimelineSource Enum

```typescript
enum TimelineSource {
  AUDIT = 'audit',
  ACTIVITY = 'activity',
  ALL = 'all',
}
```

<Tabs>
  <Tab title="AUDIT">
    Returns only system-generated audit log entries
  </Tab>
  <Tab title="ACTIVITY">
    Returns only user-created activities
  </Tab>
  <Tab title="ALL">
    Returns merged entries from both sources, sorted chronologically
  </Tab>
</Tabs>

## Business Rules

<Steps>
  <Step title="Tenant Isolation">
    Timeline entries are strictly filtered by tenant context - users can only see data within their organization
  </Step>
  <Step title="Permission-Based Access">
    Users can only view timelines for entities they have proper access permissions for
  </Step>
  <Step title="Chronological Ordering">
    Entries are always returned in reverse chronological order (newest first) regardless of source
  </Step>
  <Step title="Source Filtering">
    When `source=all`, entries from all sources are merged and sorted by timestamp
  </Step>
</Steps>

### Additional Constraints

<Warning>
**Pagination Limits**: Maximum of 100 entries per request to prevent performance issues
</Warning>

<Note>
**Date Range Filtering**: When date filters are applied, they are inclusive of the boundary dates
</Note>

## Integration Points

  </Accordion>
</AccordionGroup>

- **Reference Integrity**: Ensures proper user references in timeline metadata

- **Permission Validation**: Validates user permissions for accessed entities

## Performance Considerations

<Tip>
The Timeline module is designed for optimal performance through several strategies:
</Tip>

### Optimization Strategies

- **Database Indexing**: Relies on proper indexing of timestamp fields for efficient querying
- **Cursor-Based Pagination**: Uses cursor-based pagination to avoid expensive OFFSET queries  
- **Query Optimization**: Separate queries per source type, then merges in application layer
- **No Explicit Caching**: Relies on database query performance rather than application-level caching

### Performance Monitoring

<Check>
Monitor query performance on timestamp-based indexes and consider query optimization if response times exceed acceptable thresholds.
