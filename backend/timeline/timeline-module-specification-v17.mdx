## Overview

The Timeline module provides a unified view of chronological events across the PropWise CRM system. It aggregates data from multiple sources (audit logs, activities, and lead-level system events from distribution/escalation) to present a comprehensive timeline of events related to specific entities like leads, contacts, and companies.

<Note>
The module serves as a central point for tracking historical changes and activities, enabling users to understand the complete lifecycle and interaction history of CRM entities.
</Note>

## Architecture

The Timeline module follows a unified aggregation pattern, combining data from disparate sources into a single chronological view.

### Key architectural decisions

- **Source Abstraction**: Uses a `TimelineSource` enum to specify which data sources to query (audit, activity, or all)
- **Unified Response Format**: Normalizes different event types into a consistent timeline entry structure, with optional event details such as stage-change reasons
- **Tenant-Aware**: All timeline queries are executed within the proper tenant context
- **Pagination Support**: Implements page-based pagination over the merged timeline results
- **Resolver Pattern**: Delegates timeline entry formatting to specialized resolver classes

<Tip>
The resolver pattern ensures consistent formatting while maintaining flexibility for different event types.
</Tip>

## API endpoints

### Get entity timeline

Retrieve the timeline for a specific entity with filtering and pagination options.

<CodeGroup>

```http GET /v1/timeline/{entityType}/:id
GET /v1/timeline/lead/:id
GET /v1/timeline/deal/:id
GET /v1/timeline/contact/:id
GET /v1/timeline/company/:id
```

</CodeGroup>

#### Path parameters

| Parameter | Type | Description |
|-----------|------|-------------|
| `leadId` | UUID | Lead identifier |
| `dealId` | UUID | Deal identifier |
| `contactId` | UUID | Contact identifier |
| `companyId` | UUID | Company identifier |

#### Query parameters

| Parameter | Type | Required | Default | Description |
|-----------|------|----------|---------|-------------|
| `source` | `TimelineSource` | No | `all` | Data source filter (`audit`, `activity`, `system`, `all`) |
| `page` | number | No | 1 | Page number for merged timeline results |
| `limit` | number | No | 50 | Number of entries per page (max 200) |
| `from` | ISO8601 | No | - | Filter events from this date |
| `to` | ISO8601 | No | - | Filter events until this date |
| `activityType` | string | No | - | Single or comma-separated activity type filter |

<Warning>
The maximum limit is 200 entries per request to prevent performance issues.
</Warning>

#### Response format

```typescript
{
  data: UnifiedTimelineItemDto[];
  page: number;
  limit: number;
  total: number;
  totalPages: number;
  hasMore: boolean;
}
```

#### Timeline entry structure

<AccordionGroup>
  <Accordion title="Complete timeline entry schema">

```typescript
{
  id: string;
  source: 'audit' | 'activity' | 'system';
  timestamp: string; // ISO8601
  message: string;
  accentColor?: string; // Optional display accent; stage changes use destination stage color
  reason?: string; // Optional detail text; stage changes use the matching stage history notes

  // Audit-only fields
  eventType?: TimelineEventType;
  auditAction?: AuditAction;
  tableName?: string;
  recordId?: string;

  // Activity-only fields
  activityType?: ActivityType;
  referenceId?: string;
  targetEntityType?: ActivityTargetType;
  targetEntityId?: string;
  isPinned?: boolean;
  isImportant?: boolean;

  // System-only fields (lead timelines)
  systemActionType?: 'distribution' | 'escalation';

  actor?: {
    id: string;
    name: string;
    email?: string;
    avatarUrl?: string;
  };
}
```

  </Accordion>
</AccordionGroup>

### Task-scoped feeds

<Info>
Tasks expose their own feed endpoints from the CRM task module rather than using the generic timeline controller.
</Info>

- `GET /tasks/:id/timeline` - Combined task feed (comments + audit events)
- `GET /tasks/:id/activity` - Only audit-backed task activity events
- `GET /tasks/:id/comments/feed` - Only comments in keyset-paginated feed format

All three endpoints use `TaskTimelineFeedQueryDto` with:
- `limit` (default 40, max 100)
- Optional `before` cursor

They return `TaskTimelineFeedResponse` with ascending items, `hasMoreOlder`, and `nextCursorBefore`.

class UnifiedTimelineQueryDto {

## Business rules

<Steps>
  <Step title="Security & Access">
    - **Tenant Isolation**: Timeline entries are strictly filtered by tenant context
    - **Permission-Based Access**: Users can only view timelines for entities they have access to
  </Step>
  
  <Step title="Data Ordering & Pagination">
    - **Chronological Ordering**: Entries are always returned in reverse chronological order (newest first)
    - **Pagination Limits**: Maximum of 200 entries per request to prevent performance issues
  </Step>
  
  <Step title="Filtering & Enhancement">
    - **Source Filtering**: When `source=all`, entries from all available sources are merged and sorted by timestamp
    - **Date Range Filtering**: When date filters are applied, they are inclusive of the boundary dates
    - **Stage Accent Color**: Stage-change timeline items include `accentColor` set to the destination stage color
    - **Stage Change Reason**: Stage-change timeline items include `reason` when a matching stage history record has notes
  </Step>
</Steps>

<Check>
Contact merge repointing: Person duplicate merges can absorb duplicate Contact records. During that merge, both `Activity.contactId` and direct Contact-target activities are re-pointed to the surviving Contact, ensuring timeline continuity.
</Check>

## Integration points

### Audit module

<Tabs>
  <Tab title="Data Sources">
    - Consumes `AuditLog` entities to show system-generated change events
    - Uses `AuditTimelineResolver` to format audit entries into timeline format
    - Filters out noise from implementation details including stage tracking fields
  </Tab>
  
  <Tab title="Stage Changes">
    - Resolves stage names and destination stage colors in batch for lead/deal stage-change entries
    - Stage-change audit entries are enriched with `reason` from matching stage history notes
    - The unified response exposes the color as `accentColor`
  </Tab>
</Tabs>

### CRM module

- Integrates with `Activity` entities to show user-created activities
- Supports all activity target types (leads, contacts, companies, etc.)
- Leverages `ContactCompanyRole` for relationship context

### Distribution module

- Lead timelines include `DistributionLog` rows as `source='system'`
- `DistributionTimelineResolver` maps assignment, pool, and fallback outcomes to display-ready timeline messages

### Escalation module

- Lead timelines include `EscalationActionLog` rows as `source='system'` with `systemActionType='escalation'`
- `EscalationTimelineResolver` formats notification, redistribution, temperature-change, and tag-add actions
- Use `source=system` to fetch only distribution/escalation entries for a lead timeline

### User & Auth modules

- **User Module**: Enriches timeline entries with user information (name, email)
- **Auth Module**: Enforces JWT authentication and validates user permissions for accessed entities

## Service patterns

### UnifiedTimelineService

The core service implements several key patterns:

<CardGroup cols={2}>
  <Card title="Data Aggregation" icon="layer-group">
    - **Multi-Source Aggregation**: Queries multiple entity types and merges results
    - **Page-Based Pagination**: Applies pagination after source entries are merged and sorted
  </Card>
  <Card title="Context & Error Handling" icon="shield-check">
    - **Tenant Context Execution**: All database operations execute within tenant boundaries
    - **Error Handling**: Graceful handling of missing entities and invalid queries
  </Card>
</CardGroup>

### Resolver pattern

Timeline formatting is delegated to specialized resolvers:

- Each source type has its own resolver (e.g., `AuditTimelineResolver`)
- Resolvers handle source-specific formatting and metadata extraction
- Maintains consistency in timeline entry structure across different sources

## Performance considerations

<Warning>
Performance optimization strategies are critical for timeline queries due to the volume of historical data.
</Warning>

- **Database Indexing**: Relies on proper indexing of timestamp fields for efficient querying
- **Pagination**: Fetches bounded source lists, merges in memory, then applies page/limit slicing to the unified result
- **Query Optimization**: Separate queries per source type, then merges in application layer
- **Caching**: No explicit caching implemented - relies on database query performance

<Tip>
Consider implementing caching strategies for frequently accessed timelines to improve response times.
</Tip>
