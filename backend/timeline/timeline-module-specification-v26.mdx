## Overview

The Timeline module provides a unified view of chronological events across the PropWise CRM system. It aggregates data from multiple sources (audit logs, activities, and lead-level system events from distribution/escalation) to present a comprehensive timeline of events related to specific entities like leads, contacts, and companies.

<Info>
The module serves as a central point for tracking historical changes and activities, enabling users to understand the complete lifecycle and interaction history of CRM entities.
</Info>

## Architecture

The Timeline module follows a unified aggregation pattern, combining data from disparate sources into a single chronological view.

### Key Architectural Decisions

<CardGroup cols={2}>
  <Card title="Source Abstraction" icon="filter">
    Uses a `TimelineSource` enum to specify which data sources to query (audit, activity, or all)
  </Card>
  <Card title="Unified Response Format" icon="layer-group">
    Normalizes different event types into a consistent timeline entry structure
  </Card>
  <Card title="Tenant-Aware" icon="shield">
    All timeline queries are executed within the proper tenant context
  </Card>
  <Card title="Pagination Support" icon="list">
    Implements page-based pagination over the merged timeline results
  </Card>
</CardGroup>

### Resolver Pattern

The module delegates timeline entry formatting to specialized resolver classes:

- `AuditTimelineResolver`
- `DistributionTimelineResolver` 
- `EscalationTimelineResolver`

## Entity Relationships

<Warning>
The Timeline module doesn't define its own entities but aggregates data from existing sources.
</Warning>

### Data Sources

<AccordionGroup>
  <Accordion title="Core Sources">
    - **AuditLog**: System audit entries tracking entity changes
    - **Activity**: User-created activities and interactions
    - **Activity Channel Tables**: `call_log`, `email_log`, `sms_log`, `meeting`, and `site_visit` rows
  </Accordion>
  
  <Accordion title="Relationship Sources">
    - **ContactCompanyRole**: Relationship tracking between contacts and companies
    - **LeadStageHistory / DealStageHistory**: Stage transition history
    - **DistributionLog / EscalationActionLog**: Lead-level system events
  </Accordion>
  
  <Accordion title="Additional Sources">
    - **EntityTransfer**: Related lead/deal transfer request audit entries
    - **UnitAccessRequest**: Deal-linked unit access request audit entries
    - **LeadPropertyInterest / DealPropertyInterest**: Property interest changes
  </Accordion>
</AccordionGroup>

## API Endpoints

### Get Entity Timeline

```http
GET /v1/timeline/{lead|deal|contact|company}/:id
```

**Path Parameters:**
- `leadId`, `dealId`, `contactId`, or `companyId`: UUID of the target entity

**Query Parameters:**

| Parameter | Type | Required | Default | Description |
|-----------|------|----------|---------|-------------|
| `source` | `TimelineSource` | No | `all` | Data source filter (`audit`, `activity`, `system`, `all`) |
| `page` | `number` | No | 1 | Page number for merged timeline results |
| `limit` | `number` | No | 50 | Number of entries per page (max 200) |
| `from` | `ISO8601` | No | - | Filter events from this date |
| `to` | `ISO8601` | No | - | Filter events until this date |
| `activityType` | `string` | No | - | Single or comma-separated activity type filter |

**Response Format:**

```json
{
  "data": "UnifiedTimelineItemDto[]",
  "page": "number",
  "limit": "number", 
  "total": "number",
  "totalPages": "number",
  "hasMore": "boolean"
}
```

### Timeline Entry Structure

<CodeGroup>
```typescript Core Fields
{
  id: string;
  source: 'audit' | 'activity' | 'system';
  timestamp: string; // ISO8601
  message: string;
  accentColor?: string; // Optional display accent
  reason?: string; // Optional detail text
}
```

```typescript Audit Fields
{
  // Audit-only fields
  eventType?: TimelineEventType;
  auditAction?: AuditAction;
  tableName?: string;

```

```typescript Activity Fields
{
  // Activity-only fields
  activityType?: ActivityType;
```


</CodeGroup>

### Task-Scoped Feeds

<Note>
Tasks expose their own feed endpoints from the CRM task module rather than the generic timeline controller.
</Note>

<Steps>
  <Step title="Task Timeline">
    `GET /tasks/:id/timeline` - Returns combined task feed (comments + audit events)
  </Step>
  <Step title="Task Activity">
    `GET /tasks/:id/activity` - Returns only audit-backed task activity events
  </Step>
  <Step title="Comments Feed">
    `GET /tasks/:id/comments/feed` - Returns only comments in keyset-paginated feed
  </Step>
</Steps>

All endpoints share the `TaskTimelineFeedQueryDto` contract with `limit` (default 40, max 100) and optional `before` cursor.

### Commission Payment Timeline

<Warning>
Commission payments do not use the generic unified timeline controller.
</Warning>

The commission panel Timeline tab reads `GET /commission-payments/:id/history`, which returns immutable `commission_payment_history` rows ordered oldest-first.

## DTOs

## Business Rules

### Core Rules

<Tabs>
  <Tab title="Security & Access">
    1. **Tenant Isolation**: Timeline entries are strictly filtered by tenant context
    2. **Permission-Based Access**: Users can only view timelines for entities they have access to
  </Tab>
  
  <Tab title="Data Ordering">
    3. **Chronological Ordering**: Entries are always returned in reverse chronological order (newest first)
    4. **Source Filtering**: When `source=all`, entries from all available sources are merged and sorted by timestamp
  </Tab>
  
  <Tab title="Performance">
    5. **Pagination Limits**: Maximum of 200 entries per request to prevent performance issues
    6. **Date Range Filtering**: When date filters are applied, they are inclusive of the boundary dates
  </Tab>
</Tabs>

### Advanced Rules

<AccordionGroup>
  <Accordion title="Stage Management">
    7. **Stage Accent Color**: Lead and deal stage-change timeline items include `accentColor` set to the destination stage color
    8. **Stage Change Reason**: Stage-change items include `reason` when a matching stage history record has notes
    9. **Lead Disqualification Panel Display**: Disqualification audit messages include the categorical reason in `message`
  </Accordion>
  
  <Accordion title="Contact & Company Management">
    10. **Contact Merge Repointing**: During person duplicate merges, activities are re-pointed to the surviving Contact
    11. **Company Communication Targeting**: Company-targeted communications can be logged directly against the company
  </Accordion>
  
  <Accordion title="Activity Management">
    12. **Scheduled Task/Event Visibility**: Creating tasks/events writes Activity index rows for entity visibility
    13. **Expandable Activity Details**: Activity rows include `activityDetails` for supported channel types
  </Accordion>
</AccordionGroup>

## Integration Points

### AI Assistant Module

<Check>
`TimelineAiToolsProvider` exposes `getLeadTimeline` as a read-only AI assistant tool.
</Check>

The tool wraps `UnifiedTimelineService.getLeadTimeline()` and maintains the same semantics as the HTTP timeline endpoint:

- **Parameters**: `source`, `from`, `to`, `activityType`, and `limit`
- **Use Case**: Lead change-history prompts such as "what changed on this lead yesterday?"
- **Security**: The AI tool projection sanitizes body-bearing activity rows for privacy

<Tip>
Use this tool for timeline evidence instead of relying on recent activities alone when the AI assistant needs historical context.
</Tip>
