## Overview

The Timeline module provides a unified view of chronological events across the PropWise CRM system. It aggregates data from multiple sources (audit logs, activities, and lead-level system events from distribution/escalation) to present a comprehensive timeline of events related to specific entities like leads, contacts, and companies.

<Info>
The module serves as a central point for tracking historical changes and activities, enabling users to understand the complete lifecycle and interaction history of CRM entities.
</Info>

## Architecture

The Timeline module follows a unified aggregation pattern, combining data from disparate sources into a single chronological view.

### Key Architectural Decisions

<CardGroup cols={2}>
  <Card title="Source Abstraction" icon="filter">
    Uses a `TimelineSource` enum to specify which data sources to query (audit, activity, or all)
  </Card>
  <Card title="Unified Response Format" icon="list">
    Normalizes different event types into a consistent timeline entry structure
  </Card>
  <Card title="Tenant-Aware" icon="shield">
    All timeline queries are executed within the proper tenant context
  </Card>
  <Card title="Resolver Pattern" icon="gear">
    Delegates timeline entry formatting to specialized resolver classes
  </Card>
</CardGroup>

## Entity Relationships

<Note>
The Timeline module doesn't define its own entities but aggregates data from multiple sources.
</Note>

### Data Sources

<AccordionGroup>
  <Accordion title="Audit Sources">
    - **AuditLog**: System audit entries tracking entity changes
    - **LeadStageHistory / DealStageHistory**: Stage transition history with user-provided reasons
    - **EntityTransfer**: Lead/deal transfer request audit entries
    - **UnitAccessRequest**: Deal-linked unit access request audit entries
    - **LeadPropertyInterest / DealPropertyInterest**: Property interest changes
  </Accordion>
  
  <Accordion title="Activity Sources">
    - **Activity**: User-created activities and interactions
    - **Activity Channel Tables**: `call_log`, `email_log`, `sms_log`, `meeting`, `site_visit`
    - **ContactCompanyRole**: Relationship tracking between contacts and companies
  </Accordion>
  
  <Accordion title="System Sources">
    - **DistributionLog**: Lead distribution events
    - **EscalationActionLog**: Lead escalation events
  </Accordion>
</AccordionGroup>

### Timeline Events

The following property interest events are tracked:

- `LEAD_PROPERTY_INTEREST_ADDED`
- `LEAD_PROPERTY_INTEREST_UPDATED`
- `LEAD_PROPERTY_INTEREST_REMOVED`
- `DEAL_PROPERTY_INTEREST_ADDED`
- `DEAL_PROPERTY_INTEREST_UPDATED`
- `DEAL_PROPERTY_INTEREST_REMOVED`

## API Endpoints

### Get Entity Timeline

```http
GET /v1/timeline/{lead|deal|contact|company}/:id
```

#### Parameters

| Parameter | Type | Description |
|-----------|------|-------------|
| `id` | UUID | Target entity identifier |

#### Query Parameters

| Parameter | Type | Required | Default | Description |
|-----------|------|----------|---------|-------------|
| `source` | `TimelineSource` | No | `all` | Data source filter |
| `page` | `number` | No | 1 | Page number for results |
| `limit` | `number` | No | 50 | Entries per page (max 200) |
| `from` | `ISO8601` | No | - | Filter events from date |
| `to` | `ISO8601` | No | - | Filter events until date |
| `activityType` | `string` | No | - | Activity type filter |

#### Response Structure

```typescript
{
  data: UnifiedTimelineItemDto[];
  page: number;
  limit: number;
  total: number;
  totalPages: number;
  hasMore: boolean;
}
```

### Timeline Entry Structure

<CodeGroup>
```typescript Base Structure
{
  id: string;
  source: 'audit' | 'activity' | 'system';
  timestamp: string; // ISO8601
  message: string;
  accentColor?: string;
  reason?: string;
  
  actor?: {
    id: string;
    name: string;
    email?: string;
    avatarUrl?: string;
  };
}
```

```typescript Audit Fields
{
  // Audit-specific fields
  eventType?: TimelineEventType;
  auditAction?: AuditAction;
  tableName?: string;
  recordId?: string;
}
```

```typescript Activity Fields
{
  // Activity-specific fields
  activityType?: ActivityType;
  referenceId?: string;
  targetEntityType?: ActivityTargetType;
  targetEntityId?: string;
  isPinned?: boolean;
  isImportant?: boolean;
  activityDetails?: {
    type: ActivityType;
    // Dynamic based on activity type
  };
}
```

```typescript System Fields
{
  // System-specific fields (lead timelines only)
  systemActionType?: 'distribution' | 'escalation';
}
```
</CodeGroup>

### Task-Scoped Feeds

<Warning>
Tasks use dedicated endpoints from the CRM task module, not the generic timeline controller.
</Warning>

Available task feed endpoints:

- `GET /tasks/:id/timeline` - Combined task feed (comments + audit events)
- `GET /tasks/:id/activity` - Audit-backed task activity events only
- `GET /tasks/:id/comments/feed` - Comments only in feed format

All endpoints share the `TaskTimelineFeedQueryDto` contract with keyset pagination.

### Commission Payment Timeline

<Note>
Commission payments use a dedicated history endpoint: `GET /commission-payments/:id/history`
</Note>

Returns immutable `commission_payment_history` rows with status change notes persisted in the `reason` field.

### Organization Activity Feed

The `UnifiedTimelineService.getActivityFeedInTransaction()` method powers the dashboard Activity Feed widget.

#### Features

<Tabs>
  <Tab title="Scope">
    - **Org-wide**: Aggregates events across the entire organization
    - **Actor-scoped**: Filters by specific user activities
    - **Mixed sources**: Combines audit events and CRM activities
  </Tab>
  
  <Tab title="Sources">
    - Audit events from curated tables (lead, deal, contact, company, etc.)
    - Logged CRM activity rows (notes, calls, emails, meetings)
    - Excludes distribution/escalation system events
  </Tab>
  
  <Tab title="Pagination">
    - Cursor-based pagination with `{ occurredAt, id }` encoding
    - Over-fetches from each source for optimal merging
    - Returns `{ items, hasMore }` structure
  </Tab>
</Tabs>

## DTOs

### UnifiedTimelineQueryDto

```typescript
class UnifiedTimelineQueryDto {
  source?: TimelineSource = TimelineSource.ALL;
  from?: string; // ISO8601
  to?: string; // ISO8601
  page?: number = 1;
  limit?: number = 50; // min: 1, max: 200
  activityType?: string; // single value or comma-separated list
}
```

### TimelineSource Enum

```typescript
enum TimelineSource {
  AUDIT = 'audit',
  ACTIVITY = 'activity',
  SYSTEM = 'system',
  ALL = 'all',
}
```

## Business Rules

<Steps>
  <Step title="Security & Access">
    - **Tenant Isolation**: Timeline entries are strictly filtered by tenant context
    - **Permission-Based Access**: Users can only view timelines for accessible entities
  </Step>
  
  <Step title="Data Ordering">
    - **Chronological Ordering**: Entries returned in reverse chronological order (newest first)
    - **Source Merging**: When `source=all`, entries from all sources are merged by timestamp
  </Step>
  
  <Step title="Performance Limits">
    - **Pagination Limits**: Maximum 200 entries per request
    - **Date Filtering**: Inclusive boundary dates when filters applied
  </Step>
  
  <Step title="Visual Enhancement">
    - **Stage Accent Colors**: Stage-change items include destination stage color
    - **Stage Change Reasons**: Include notes from matching stage history records
  </Step>
</Steps>

### Special Display Rules

<AccordionGroup>
  <Accordion title="Lead Disqualification">
    Lead disqualification audit messages include categorical reason in the message. The Timeline tab displays `lead.disqualificationNotes` as the visible reason detail when available.
  </Accordion>
  
  <Accordion title="Contact Merge Handling">
    During contact merges, both `Activity.contactId` and direct contact-target activities are re-pointed to the surviving contact, preserving timeline continuity.
  </Accordion>
  
  <Accordion title="Company Communication">
    Company-targeted communications can be logged directly against the company (`Activity.contactId = null`) or through linked contacts with validated relationships.
  </Accordion>
</AccordionGroup>
