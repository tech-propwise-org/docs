## Overview

The Timeline module provides a unified view of chronological events across the PropWise CRM system. It aggregates data from multiple sources (audit logs, activities, and lead-level system events from distribution/escalation) to present a comprehensive timeline of events related to specific entities like leads, contacts, and companies.

<Note>
The module serves as a central point for tracking historical changes and activities, enabling users to understand the complete lifecycle and interaction history of CRM entities.
</Note>

## Architecture

The Timeline module follows a unified aggregation pattern, combining data from disparate sources into a single chronological view.

### Key Architectural Decisions

<CardGroup cols={2}>
  <Card title="Source Abstraction" icon="filter">
    Uses a `TimelineSource` enum to specify which data sources to query (audit, activity, or all)
  </Card>
  <Card title="Unified Response Format" icon="arrows-merge">
    Normalizes different event types into a consistent timeline entry structure
  </Card>
  <Card title="Tenant-Aware" icon="shield">
    All timeline queries are executed within the proper tenant context
  </Card>
  <Card title="Resolver Pattern" icon="code">
    Delegates timeline entry formatting to specialized resolver classes
  </Card>
</CardGroup>

### Core Features

- **Pagination Support**: Implements page-based pagination over the merged timeline results
- **Optional Event Details**: Stage-change reasons and compact activity channel metadata
- **Specialized Resolvers**: `AuditTimelineResolver`, `DistributionTimelineResolver`, `EscalationTimelineResolver`

## Entity Relationships

<Warning>
The Timeline module doesn't define its own entities but aggregates data from existing sources.
</Warning>

### Data Sources

<AccordionGroup>
  <Accordion title="Primary Sources">
    - **AuditLog**: System audit entries tracking entity changes
    - **Activity**: User-created activities and interactions
    - **Activity Channel Tables**: `call_log`, `email_log`, `sms_log`, `meeting`, and `site_visit` rows
  </Accordion>
  
  <Accordion title="Relationship Sources">
    - **ContactCompanyRole**: Relationship tracking between contacts and companies
    - **LeadStageHistory / DealStageHistory**: Stage transition history for stage-change reasons
    - **DistributionLog / EscalationActionLog**: Lead-level system events for automatic assignment
  </Accordion>
  
  <Accordion title="Transfer & Access Sources">
    - **EntityTransfer**: Related lead/deal transfer request audit entries
    - **UnitAccessRequest**: Deal-linked unit access request audit entries
  </Accordion>
</AccordionGroup>

## API Endpoints

### Get Entity Timeline

<CodeGroup>
```http GET Request
GET /v1/timeline/{lead|deal|contact|company}/:id
```

```typescript Response Type
{
  data: UnifiedTimelineItemDto[];
  page: number;
  limit: number;
  total: number;
  totalPages: number;
  hasMore: boolean;
}
```
</CodeGroup>

#### Path Parameters

| Parameter | Type | Description |
|-----------|------|-------------|
| `leadId` | UUID | Lead entity identifier |
| `dealId` | UUID | Deal entity identifier |
| `contactId` | UUID | Contact entity identifier |
| `companyId` | UUID | Company entity identifier |

#### Query Parameters

| Parameter | Type | Required | Default | Description |
|-----------|------|----------|---------|-------------|
| `source` | `TimelineSource` | No | `all` | Data source filter |
| `page` | `number` | No | 1 | Page number for merged timeline results |
| `limit` | `number` | No | 50 | Number of entries per page (max 200) |
| `from` | `ISO8601` | No | - | Filter events from this date |
| `to` | `ISO8601` | No | - | Filter events until this date |
| `activityType` | `string` | No | - | Single or comma-separated activity type filter |

### Task-Scoped Feeds

<Info>
Tasks expose their own feed endpoints from the CRM task module rather than the generic timeline controller.
</Info>

<Steps>
  <Step title="Task Timeline">
    `GET /tasks/:id/timeline` - Combined task feed (comments + audit events)
  </Step>
  <Step title="Task Activity">
    `GET /tasks/:id/activity` - Only audit-backed task activity events
  </Step>
  <Step title="Task Comments">
    `GET /tasks/:id/comments/feed` - Only comments in keyset-paginated feed shape
  </Step>
</Steps>

**Shared Contract**: `TaskTimelineFeedQueryDto`
- `limit` (default 40, max 100)
- Optional `before` cursor
- Returns `TaskTimelineFeedResponse` with ascending items

### Commission Payment Timeline

<Warning>
Commission payments do not use the generic unified timeline controller.
</Warning>

The commission panel Timeline tab reads `GET /commission-payments/:id/history`, which returns immutable `commission_payment_history` rows:

- Ordered oldest-first from database
- Side panel sorts by `createdAt` descending for display
- Notes from status changes stored in `reason` field
- Free-form notes use Activity/Note system with `targetEntityType='commission_payment'`

## DTOs

### UnifiedTimelineQueryDto

```typescript
class UnifiedTimelineQueryDto {
  source?: TimelineSource = TimelineSource.ALL;
  from?: string; // ISO8601
  to?: string; // ISO8601
  page?: number = 1;
  limit?: number = 50; // min: 1, max: 200
  activityType?: string; // single value or comma-separated list
}
```

### TimelineSource Enum

```typescript
enum TimelineSource {
  AUDIT = 'audit',
  ACTIVITY = 'activity',
  SYSTEM = 'system',
  ALL = 'all',
}
```

## Business Rules

<AccordionGroup>
  <Accordion title="Security & Access">
    1. **Tenant Isolation**: Timeline entries are strictly filtered by tenant context
    2. **Permission-Based Access**: Users can only view timelines for entities they have access to
  </Accordion>
  
  <Accordion title="Data Handling">
    3. **Chronological Ordering**: Entries are always returned in reverse chronological order (newest first)
    4. **Source Filtering**: When `source=all`, entries from all available sources are merged and sorted by timestamp
    5. **Pagination Limits**: Maximum of 200 entries per request to prevent performance issues
    6. **Date Range Filtering**: Date filters are inclusive of boundary dates
  </Accordion>
  
  <Accordion title="Stage Management">
    7. **Stage Accent Color**: Stage-change timeline items include `accentColor` set to destination stage color
    8. **Stage Change Reason**: Stage-change items include `reason` when matching stage history has notes
    9. **Lead Disqualification Display**: Disqualification messages include categorical reason, with `lead.disqualificationNotes` as visible detail
  </Accordion>
  
  <Accordion title="Contact & Company Handling">
    10. **Contact Merge Repointing**: During person duplicate merges, activities are re-pointed to surviving Contact
    11. **Company Communication Targeting**: Company-targeted activities can be logged with or without linked contact
  </Accordion>
  
  <Accordion title="Task & Event Integration">
    12. **Scheduled Task/Event Visibility**: Tasks/events create `Activity` index rows when not already creating typed activities
    13. **Expandable Activity Details**: Activity rows include compact headers with expandable `activityDetails` for supported channels
  </Accordion>
</AccordionGroup>

<Tip>
HTML email bodies are intentionally not included in the unified timeline payload for performance reasons.
</Tip>

## Integration Points

### AI Assistant Module

<Steps>
  <Step title="Tool Provider">
    `TimelineAiToolsProvider` exposes `getLeadTimeline` as a read-only AI assistant tool
  </Step>
  <Step title="Service Integration">
    Tool wraps `UnifiedTimelineService.getLeadTimeline()` with same semantics as HTTP endpoint
  </Step>
  <Step title="Use Cases">
    Use for lead change-history prompts like "what changed on this lead yesterday?"
  </Step>
  <Step title="Content Sanitization">
    AI tool projection sanitizes body-bearing activities for security
  </Step>
</Steps>

**Content Sanitization Rules:**
- `summary.items[].message` replaced with safe labels (e.g., "Note activity logged")
- `hasBody` / `bodyLength` exposed for shape information
- Original `Activity.summary` routed through `untrustedContent[]` with author control flag
- Audit and system messages remain unchanged (server-formatted)
