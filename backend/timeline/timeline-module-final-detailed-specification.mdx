## Overview

The Timeline module provides a unified view of chronological events across the PropWise CRM system. It aggregates data from multiple sources (audit logs and activities) to present a comprehensive timeline of events related to specific entities like leads, contacts, and companies. The module serves as a central point for tracking historical changes and activities, enabling users to understand the complete lifecycle and interaction history of CRM entities.

## Architecture

The Timeline module follows a unified aggregation pattern, combining data from disparate sources into a single chronological view. Key architectural decisions:

<CardGroup cols={2}>
  <Card title="Source Abstraction" icon="filter">
    Uses a `TimelineSource` enum to specify which data sources to query (audit, activity, or all)
  </Card>
  <Card title="Unified Response Format" icon="list">
    Normalizes different event types into a consistent timeline entry structure
  </Card>
  <Card title="Tenant-Aware" icon="shield">
    All timeline queries are executed within the proper tenant context
  </Card>
  <Card title="Pagination Support" icon="arrows-split-up-and-left">
    Implements cursor-based pagination for efficient timeline browsing
  </Card>
</CardGroup>

<Note>
The module uses a **Resolver Pattern** that delegates timeline entry formatting to specialized resolver classes (e.g., `AuditTimelineResolver`) for maintainability and extensibility.
</Note>

## Entity Relationships

The Timeline module doesn't define its own entities but aggregates data from:

- **AuditLog**: System audit entries tracking entity changes
- **Activity**: User-created activities and interactions
- **ContactCompanyRole**: Relationship tracking between contacts and companies

## API Endpoints

### Get Entity Timeline

```http
GET /timeline/:entityType/:entityId
```

**Path Parameters:**

- `entityType`: Type of entity (lead, contact, company, etc.)
- `entityId`: UUID of the target entity

**Query Parameters:**

| Parameter | Type | Required | Default | Description |
|-----------|------|----------|---------|-------------|
| `source` | `TimelineSource` | No | `all` | Data source filter (audit/activity/all) |
| `limit` | `number` | No | 50 | Number of entries per page (max 100) |
| `cursor` | `string` | No | - | Pagination cursor |
| `startDate` | `ISO8601` | No | - | Filter events from this date |
| `endDate` | `ISO8601` | No | - | Filter events until this date |

**Response Structure:**

<CodeGroup>

```typescript Timeline Response
{
  items: UnifiedTimelineEntryDto[];
  hasNextPage: boolean;
  nextCursor?: string;
  totalCount?: number;
}
```

```typescript Timeline Entry

  id: string;
  timestamp: string; // ISO8601
  type: TimelineEventType;
  source: TimelineSource;
  title: string;
  description?: string;
  user?: {
    id: string;
    name: string;
    email: string;
  };
  metadata?: Record<string, any>;
}
```

</CodeGroup>

## Business Rules

<AccordionGroup>
  <Accordion title="Tenant Isolation">
    Timeline entries are strictly filtered by tenant context to ensure data security and isolation.
  </Accordion>
  
  <Accordion title="Permission-Based Access">
    Users can only view timelines for entities they have access to based on their role permissions.
  </Accordion>
  
  <Accordion title="Chronological Ordering">
    Entries are always returned in reverse chronological order (newest first) for intuitive browsing.
  </Accordion>
  
  <Accordion title="Source Filtering">
    When `source=all`, entries from all sources are merged and sorted by timestamp for unified view.
  </Accordion>
  
  <Accordion title="Pagination Limits">
    Maximum of 100 entries per request to prevent performance issues and ensure responsive API.
  </Accordion>
  
  <Accordion title="Date Range Filtering">
    When date filters are applied, they are inclusive of the boundary dates for accurate time-based filtering.
  </Accordion>
</AccordionGroup>

## Integration Points

### Audit Module

<Steps>
  <Step title="Audit Log Consumption">
    Consumes `AuditLog` entities to show system-generated change events
  </Step>
  
  <Step title="Timeline Formatting">
    Uses `AuditTimelineResolver` to format audit entries into timeline format
  </Step>
  
  <Step title="Noise Filtering">
    Filters out noise from implementation details including stage tracking fields (`stage_entered_at`, `stageEnteredAt`) that are automatically updated alongside stage changes
  </Step>
</Steps>

### CRM Module

- Integrates with `Activity` entities to show user-created activities
- Supports all activity target types (leads, contacts, companies, etc.)
- Leverages `ContactCompanyRole` for relationship context

### User Module

- Enriches timeline entries with user information (name, email)
- Ensures proper user references in timeline metadata

### Auth Module

- Enforces JWT authentication on all timeline endpoints
- Validates user permissions for accessed entities

## Service Patterns

### UnifiedTimelineService

The core service implements several key patterns:

<Tabs>
  <Tab title="Multi-Source Aggregation">
    Queries multiple entity types and merges results into a unified timeline view
  </Tab>
  
  <Tab title="Cursor-Based Pagination">
    Implements efficient pagination using timestamp cursors to avoid expensive OFFSET queries
  </Tab>
  
  <Tab title="Tenant Context">
    All database operations execute within tenant boundaries for proper isolation
  </Tab>
  
  <Tab title="Error Handling">
    Graceful handling of missing entities and invalid queries with meaningful error responses
  </Tab>
</Tabs>

### Resolver Pattern

Timeline formatting is delegated to specialized resolvers:

- Each source type has its own resolver (e.g., `AuditTimelineResolver`)
- Resolvers handle source-specific formatting and metadata extraction
- Maintains consistency in timeline entry structure across different sources

<Tip>
This pattern allows for easy extension when adding new timeline sources without modifying the core timeline logic.
</Tip>

## Performance Considerations

<Warning>
Proper performance optimization is critical for timeline functionality due to potentially large datasets.
</Warning>

### Database Optimization

- **Indexing**: Relies on proper indexing of timestamp fields for efficient querying
- **Query Strategy**: Separate queries per source type, then merges in application layer
- **Pagination**: Uses cursor-based pagination to avoid expensive OFFSET queries

### Caching Strategy

<Note>
No explicit caching is implemented currently - the system relies on database query performance and proper indexing.
</Note>

### Performance Limits

- Maximum 100 entries per request prevents excessive memory usage
- Cursor-based pagination ensures consistent performance regardless of result set size
- Date range filtering reduces query scope when specified
