## Overview

The Timeline module provides a unified view of chronological events across the PropWise CRM system. It aggregates data from multiple sources (audit logs, activities, and lead-level system events from distribution/escalation) to present a comprehensive timeline of events related to specific entities like leads, contacts, and companies.

<Note>
The module serves as a central point for tracking historical changes and activities, enabling users to understand the complete lifecycle and interaction history of CRM entities.
</Note>

## Architecture

The Timeline module follows a unified aggregation pattern, combining data from disparate sources into a single chronological view.

### Key Architectural Decisions

<AccordionGroup>
<Accordion title="Source Abstraction">
Uses a `TimelineSource` enum to specify which data sources to query (audit, activity, or all)
</Accordion>

<Accordion title="Unified Response Format">
Normalizes different event types into a consistent timeline entry structure, with optional event details such as stage-change reasons
</Accordion>

<Accordion title="Tenant-Aware">
All timeline queries are executed within the proper tenant context
</Accordion>

<Accordion title="Pagination Support">
Implements page-based pagination over the merged timeline results
</Accordion>

<Accordion title="Resolver Pattern">
Delegates timeline entry formatting to specialized resolver classes (e.g., `AuditTimelineResolver`, `DistributionTimelineResolver`, `EscalationTimelineResolver`)
</Accordion>
</AccordionGroup>

## Entity Relationships

<Info>
The Timeline module doesn't define its own entities but aggregates data from existing sources.
</Info>

**Data Sources:**
- **AuditLog**: System audit entries tracking entity changes
- **Activity**: User-created activities and interactions
- **ContactCompanyRole**: Relationship tracking between contacts and companies
- **LeadStageHistory / DealStageHistory**: Stage transition history used to attach user-provided stage-change reasons
- **DistributionLog / EscalationActionLog**: Lead-level system events for automatic assignment and escalation activity

## API Endpoints

### Get Entity Timeline

```http
GET /v1/timeline/{lead|deal|contact|company}/:id
```

#### Parameters

**Path Parameters:**
- `leadId`, `dealId`, `contactId`, or `companyId`: UUID of the target entity

#### Query Parameters

| Parameter | Type | Required | Default | Description |
|-----------|------|----------|---------|-------------|
| `source` | `TimelineSource` | No | `all` | Data source filter (`audit`, `activity`, `system`, `all`) |
| `page` | `number` | No | 1 | Page number for merged timeline results |
| `limit` | `number` | No | 50 | Number of entries per page (max 200) |
| `from` | `ISO8601` | No | - | Filter events from this date |
| `to` | `ISO8601` | No | - | Filter events until this date |
| `activityType` | `string` | No | - | Single or comma-separated activity type filter |

#### Response

<CodeGroup>
```typescript Response Format
{
  data: UnifiedTimelineItemDto[];
  page: number;
  limit: number;
  total: number;
  totalPages: number;
  hasMore: boolean;
}
```

```typescript Timeline Entry Structure
{
  id: string;
  source: 'audit' | 'activity' | 'system';
  timestamp: string; // ISO8601
  message: string;
  accentColor?: string; // Optional display accent
  reason?: string; // Optional detail text

  // Audit-only
  eventType?: TimelineEventType;
  auditAction?: AuditAction;
  tableName?: string;
  recordId?: string;

  // Activity-only
  activityType?: ActivityType;
  referenceId?: string;
  targetEntityType?: ActivityTargetType;
  targetEntityId?: string;
  isPinned?: boolean;
  isImportant?: boolean;

  // System-only (lead timelines)
  systemActionType?: 'distribution' | 'escalation';

  actor?: {
    id: string;
    name: string;
    email?: string;
    avatarUrl?: string;
  };
}
```
</CodeGroup>

### Task-Scoped Feeds

<Note>
Tasks expose their own feed endpoints from the CRM task module rather than the generic timeline controller.
</Note>

**Available Endpoints:**
- `GET /tasks/:id/timeline` - Combined task feed (comments + audit events)
- `GET /tasks/:id/activity` - Only audit-backed task activity events  
- `GET /tasks/:id/comments/feed` - Only comments in feed format

## DTOs

### UnifiedTimelineQueryDto

```typescript
class UnifiedTimelineQueryDto {
  source?: TimelineSource = TimelineSource.ALL;
  from?: string; // ISO8601
  to?: string; // ISO8601
  page?: number = 1;
  limit?: number = 50; // min: 1, max: 200
  activityType?: string; // single value or comma-separated list
}
```

### TimelineSource Enum

```typescript
enum TimelineSource {
  AUDIT = 'audit',
  ACTIVITY = 'activity',
  SYSTEM = 'system',
  ALL = 'all',
}
```

## Business Rules

<Steps>
<Step title="Tenant Isolation">
Timeline entries are strictly filtered by tenant context
</Step>

<Step title="Permission-Based Access">
Users can only view timelines for entities they have access to
</Step>

<Step title="Chronological Ordering">
Entries are always returned in reverse chronological order (newest first)
</Step>

<Step title="Source Filtering">
When `source=all`, entries from all available sources are merged and sorted by timestamp
</Step>

<Step title="Pagination Limits">
Maximum of 200 entries per request to prevent performance issues
</Step>

<Step title="Date Range Filtering">
When date filters are applied, they are inclusive of the boundary dates
</Step>
</Steps>

### Special Rules

<CardGroup cols={2}>
<Card title="Stage Accent Color" icon="palette">
Lead and deal stage-change timeline items include `accentColor` set to the destination stage color for visual indicators
</Card>

<Card title="Stage Change Reason" icon="comment">
Stage-change timeline items include `reason` when a matching stage history record has notes
</Card>

<Card title="Contact Merge Repointing" icon="merge">
During person duplicate merges, both `Activity.contactId` and direct Contact-target activities are re-pointed to the surviving Contact
</Card>
</CardGroup>

## Integration Points

### Audit Module

<Check>Consumes `AuditLog` entities to show system-generated change events</Check>

- Uses `AuditTimelineResolver` to format audit entries
- Filters out noise from implementation details including stage tracking fields
- Resolves stage names and destination stage colors in batch
- Enriches stage-change entries with `reason` from matching stage history

### CRM Module

<Check>Integrates with `Activity` entities to show user-created activities</Check>

- Supports all activity target types (leads, contacts, companies)
- Leverages `ContactCompanyRole` for relationship context

### Distribution Module

<Check>Lead timelines include `DistributionLog` rows as `source='system'`</Check>

- `DistributionTimelineResolver` maps assignment, pool, and fallback outcomes
- Provides display-ready timeline messages for distribution events

### Escalation Module

<Check>Lead timelines include `EscalationActionLog` rows as system events</Check>

- `EscalationTimelineResolver` formats various escalation actions
- Handles notification, redistribution, temperature-change, and tag-add actions

## Service Patterns

### UnifiedTimelineService

The core service implements several key patterns:

<AccordionGroup>
<Accordion title="Multi-Source Aggregation">
Queries multiple entity types and merges results into a unified timeline
</Accordion>

<Accordion title="Page-Based Pagination">
Applies `page` and `limit` after source entries are merged and sorted
</Accordion>

<Accordion title="Tenant Context Execution">
All database operations execute within tenant boundaries
</Accordion>

<Accordion title="Error Handling">
Graceful handling of missing entities and invalid queries
</Accordion>
</AccordionGroup>

### Resolver Pattern

Timeline formatting is delegated to specialized resolvers:

- Each source type has its own resolver (e.g., `AuditTimelineResolver`)
- Resolvers handle source-specific formatting and metadata extraction
- Maintains consistency in timeline entry structure across different sources

## Performance Considerations

<Warning>
The module fetches bounded source lists, merges in memory, then applies pagination slicing to the unified result.
</Warning>

**Key Performance Factors:**

- **Database Indexing**: Relies on proper indexing of timestamp fields for efficient querying
- **Pagination**: Memory-based merging with application-layer page/limit slicing
- **Query Optimization**: Separate queries per source type, then merges in application layer
- **Caching**: No explicit caching implemented - relies on database query performance

<Tip>
Consider implementing caching strategies for frequently accessed timelines to improve response times.
</Tip>
