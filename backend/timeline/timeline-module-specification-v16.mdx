## Overview

The Timeline module provides a unified view of chronological events across the PropWise CRM system. It aggregates data from multiple sources (audit logs, activities, and lead-level system events from distribution/escalation) to present a comprehensive timeline of events related to specific entities like leads, contacts, and companies.

<Info>
The module serves as a central point for tracking historical changes and activities, enabling users to understand the complete lifecycle and interaction history of CRM entities.
</Info>

## Architecture

The Timeline module follows a unified aggregation pattern, combining data from disparate sources into a single chronological view.

### Key architectural decisions

<CardGroup cols={2}>
<Card title="Source Abstraction" icon="filter">
Uses a `TimelineSource` enum to specify which data sources to query (audit, activity, or all)
</Card>
<Card title="Unified Response Format" icon="align-left">
Normalizes different event types into a consistent timeline entry structure, with optional event details
</Card>
<Card title="Tenant-Aware" icon="shield">
All timeline queries are executed within the proper tenant context
</Card>
<Card title="Pagination Support" icon="page">
Implements page-based pagination over the merged timeline results
</Card>
</CardGroup>

### Resolver pattern

<Note>
The module delegates timeline entry formatting to specialized resolver classes such as `AuditTimelineResolver`, `DistributionTimelineResolver`, and `EscalationTimelineResolver`.
</Note>

## Entity relationships

The Timeline module doesn't define its own entities but aggregates data from:

- **AuditLog**: System audit entries tracking entity changes
- **Activity**: User-created activities and interactions  
- **ContactCompanyRole**: Relationship tracking between contacts and companies
- **LeadStageHistory / DealStageHistory**: Stage transition history used to attach user-provided stage-change reasons
- **DistributionLog / EscalationActionLog**: Lead-level system events for automatic assignment and escalation activity

## API endpoints

### Get entity timeline

```http
GET /v1/timeline/{lead|deal|contact|company}/:id
```

#### Parameters

| Parameter | Location | Type | Required | Description |
|-----------|----------|------|----------|-------------|
| `leadId`, `dealId`, `contactId`, or `companyId` | path | UUID | Yes | Target entity identifier |

#### Query parameters

| Parameter | Type | Required | Default | Description |
|-----------|------|----------|---------|-------------|
| `source` | `TimelineSource` | No | `all` | Data source filter (`audit`, `activity`, `system`, `all`) |
| `page` | `number` | No | 1 | Page number for merged timeline results |
| `limit` | `number` | No | 50 | Number of entries per page (max 200) |
| `from` | `ISO8601` | No | - | Filter events from this date |
| `to` | `ISO8601` | No | - | Filter events until this date |
| `activityType` | `string` | No | - | Single or comma-separated activity type filter |

#### Response format

<CodeGroup>

```typescript Response Schema
{
  data: UnifiedTimelineItemDto[];
  page: number;
  limit: number;
  total: number;
  totalPages: number;
  hasMore: boolean;
}
```

```typescript Timeline Entry Structure
{
  id: string;
  source: 'audit' | 'activity' | 'system';
  timestamp: string; // ISO8601
  message: string;
  accentColor?: string; // Optional display accent
  reason?: string; // Optional detail text

  // Audit-only fields
  eventType?: TimelineEventType;
  auditAction?: AuditAction;
  tableName?: string;
  recordId?: string;

  // Activity-only fields
  activityType?: ActivityType;
  referenceId?: string;
  targetEntityType?: ActivityTargetType;
  targetEntityId?: string;
  isPinned?: boolean;
  isImportant?: boolean;

  // System-only fields (lead timelines)
  systemActionType?: 'distribution' | 'escalation';

  actor?: {
    id: string;
    name: string;
    email?: string;
    avatarUrl?: string;
  };
}
```

</CodeGroup>

### Task-scoped feeds

<Warning>
Tasks expose their own feed endpoints from the CRM task module rather than the generic timeline controller.
</Warning>

<AccordionGroup>
<Accordion title="Task timeline endpoints">

- `GET /tasks/:id/timeline` - Returns the combined task feed (`TaskComment` rows + task `AuditLog` events)
- `GET /tasks/:id/activity` - Returns only audit-backed task activity events  
- `GET /tasks/:id/comments/feed` - Returns only comments in the same keyset-paginated feed shape

All three endpoints share the same `TaskTimelineFeedQueryDto` contract with `limit` (default 40, max 100) and optional `before` cursor. They return `TaskTimelineFeedResponse` with ascending items, `hasMoreOlder`, and `nextCursorBefore`.

</Accordion>
</AccordionGroup>

## Data transfer objects

### UnifiedTimelineQueryDto

```typescript
class UnifiedTimelineQueryDto {
  source?: TimelineSource = TimelineSource.ALL;
  from?: string; // ISO8601
  to?: string; // ISO8601
  page?: number = 1;
  limit?: number = 50; // min: 1, max: 200
  activityType?: string; // single value or comma-separated list
}
```

### TimelineSource enum

```typescript
enum TimelineSource {
  AUDIT = 'audit',
  ACTIVITY = 'activity', 
  SYSTEM = 'system',
  ALL = 'all',
}
```

## Business rules

<Steps>
<Step title="Tenant isolation">
Timeline entries are strictly filtered by tenant context
</Step>
<Step title="Permission-based access">
Users can only view timelines for entities they have access to
</Step>
<Step title="Chronological ordering">
Entries are always returned in reverse chronological order (newest first)
</Step>
<Step title="Source filtering">
When `source=all`, entries from all available sources are merged and sorted by timestamp
</Step>
<Step title="Pagination limits">
Maximum of 200 entries per request to prevent performance issues
</Step>
</Steps>

### Advanced business rules

<Tabs>
<Tab title="Date Filtering">
When date filters are applied, they are inclusive of the boundary dates
</Tab>
<Tab title="Stage Accent Color">
Lead and deal stage-change timeline items include `accentColor` set to the destination stage color for client rendering
</Tab>
<Tab title="Stage Change Reason">
Stage-change timeline items include `reason` when a matching stage history record has notes, using entity ID, stage IDs, and timestamp matching
</Tab>
<Tab title="Contact Merge Repointing">
During person duplicate merges, both `Activity.contactId` and direct Contact-target activities are re-pointed to the surviving Contact
</Tab>
</Tabs>

## Integration points

### Audit module integration

<CardGroup cols={2}>
<Card title="Data Source" icon="database">
Consumes `AuditLog` entities to show system-generated change events
</Card>
<Card title="Formatting" icon="code">
Uses `AuditTimelineResolver` to format audit entries into timeline format
</Card>
<Card title="Noise Filtering" icon="filter">
Filters out implementation details including stage tracking fields
</Card>
<Card title="Stage Enhancement" icon="star">
Resolves stage names and colors, enriches with reason from stage history
</Card>
</CardGroup>

### CRM module integration

- Integrates with `Activity` entities to show user-created activities
- Supports all activity target types (leads, contacts, companies, etc.)
- Leverages `ContactCompanyRole` for relationship context

### Distribution module integration

<Info>
Lead timelines include `DistributionLog` rows as `source='system'` with `DistributionTimelineResolver` mapping assignment, pool, and fallback outcomes.
</Info>

### Escalation module integration

<Info>
Lead timelines include `EscalationActionLog` rows as `source='system'` with `systemActionType='escalation'`, formatting notification, redistribution, temperature-change, and tag-add actions.
</Info>

### User and auth modules

- **User Module**: Enriches timeline entries with user information (name, email)
- **Auth Module**: Enforces JWT authentication and validates user permissions

## Performance considerations

<Warning>
The following performance factors should be considered when implementing timeline functionality:
</Warning>

- **Database Indexing**: Relies on proper indexing of timestamp fields for efficient querying
- **Pagination**: Fetches bounded source lists, merges in memory, then applies page/limit slicing
- **Query Optimization**: Separate queries per source type, then merges in application layer
- **Caching**: No explicit caching implemented - relies on database query performance

<Tip>
For optimal performance, ensure proper database indexes are in place for timestamp-based queries across all source tables.
</Tip>
