## Overview

The Timeline module provides a unified view of chronological events across the PropWise CRM system. It aggregates data from multiple sources (audit logs and activities) to present a comprehensive timeline of events related to specific entities like leads, contacts, and companies. The module serves as a central point for tracking historical changes and activities, enabling users to understand the complete lifecycle and interaction history of CRM entities.

## Architecture

The Timeline module follows a unified aggregation pattern, combining data from disparate sources into a single chronological view. Key architectural decisions:

<CardGroup cols={2}>
  <Card title="Source Abstraction" icon="filter">
    Uses a `TimelineSource` enum to specify which data sources to query (audit, activity, or all)
  </Card>
  <Card title="Unified Response Format" icon="merge">
    Normalizes different event types into a consistent timeline entry structure, with optional event details such as stage-change reasons
  </Card>
  <Card title="Tenant-Aware" icon="shield">
    All timeline queries are executed within the proper tenant context
  </Card>
  <Card title="Pagination Support" icon="list">
    Implements page-based pagination over the merged timeline results
  </Card>
</CardGroup>

<Note>
The module uses a **Resolver Pattern** that delegates timeline entry formatting to specialized resolver classes (e.g., `AuditTimelineResolver`) for maintainable code organization.
</Note>

## Entity Relationships

The Timeline module doesn't define its own entities but aggregates data from:

- **AuditLog**: System audit entries tracking entity changes
- **Activity**: User-created activities and interactions  
- **ContactCompanyRole**: Relationship tracking between contacts and companies
- **LeadStageHistory / DealStageHistory**: Stage transition history used to attach user-provided stage-change reasons to matching audit timeline entries

## API Endpoints

### Get Entity Timeline

```http
GET /timeline/:entityType/:entityId
```

**Parameters:**

- `entityType` (path): Type of entity (lead, contact, company, etc.)
- `entityId` (path): UUID of the target entity

**Query Parameters:**

| Parameter | Type | Required | Default | Description |
|-----------|------|----------|---------|-------------|
| `source` | `TimelineSource` | No | `all` | Data source filter (`audit`, `activity`, `system`, `all`) |
| `page` | `number` | No | 1 | Page number for merged timeline results |
| `limit` | `number` | No | 50 | Number of entries per page (max 200) |
| `from` | `ISO8601` | No | - | Filter events from this date |
| `to` | `ISO8601` | No | - | Filter events until this date |
| `activityType` | `string` | No | - | Single or comma-separated activity type filter, applied to activity entries |

**Response:**

<CodeGroup>

```typescript Timeline Response
{
  data: UnifiedTimelineItemDto[];
  page: number;
  limit: number;
```

## Business Rules

<AccordionGroup>
  <Accordion title="Core Access & Security Rules">
    - **Tenant Isolation**: Timeline entries are strictly filtered by tenant context
    - **Permission-Based Access**: Users can only view timelines for entities they have access to
    - **Pagination Limits**: Maximum of 200 entries per request to prevent performance issues
  </Accordion>

  <Accordion title="Data Organization Rules">
    - **Chronological Ordering**: Entries are always returned in reverse chronological order (newest first)
    - **Source Filtering**: When `source=all`, entries from all sources are merged and sorted by timestamp
    - **Date Range Filtering**: When date filters are applied, they are inclusive of the boundary dates
  </Accordion>

  <Accordion title="Stage Change Enhancement">
    - **Stage Accent Color**: Lead and deal stage-change timeline items include `accentColor` set to the destination stage color so clients can render timeline indicators without parsing stage names or fetching stage metadata
    - **Stage Change Reason**: Lead and deal stage-change timeline items include `reason` when a matching stage history record has notes. Matching uses entity ID, from-stage ID, to-stage ID, and a short timestamp window so repeated transitions do not receive the wrong reason
  </Accordion>

  <Accordion title="Contact Merge Handling">
    **Contact Merge Repointing**: Person duplicate merges can absorb duplicate Contact records. During that merge, both `Activity.contactId` and direct Contact-target activities (`targetEntityType='contact'`, `targetEntityId=<absorbedContactId>`) are re-pointed to the surviving Contact, so the Contact timeline continues to show notes, calls, meetings, and message bridge activity after the absorbed Contact is soft-deleted.
  </Accordion>
</AccordionGroup>

## Integration Points

### Audit Module

<Steps>
  <Step title="Data Consumption">
    Consumes `AuditLog` entities to show system-generated change events
  </Step>
  <Step title="Timeline Formatting">
    Uses `AuditTimelineResolver` to format audit entries into timeline format
  </Step>
  <Step title="Noise Filtering">
    Filters out noise from implementation details including stage tracking fields (`stage_entered_at`, `stageEnteredAt`) that are automatically updated alongside stage changes
  </Step>
  <Step title="Stage Enhancement">
    - Resolves stage names and destination stage colors in batch for lead/deal stage-change entries
    - The unified response exposes the color as `accentColor`
    - Stage-change audit entries are enriched with `reason` from matching `LeadStageHistory` or `DealStageHistory` notes after audit message resolution
  </Step>
</Steps>

### CRM Module

- Integrates with `Activity` entities to show user-created activities
- Supports all activity target types (leads, contacts, companies, etc.)
- Leverages `ContactCompanyRole` for relationship context

### User Module

- Enriches timeline entries with user information (name, email)
- Ensures proper user references in timeline metadata

### Auth Module

- Enforces JWT authentication on all timeline endpoints
- Validates user permissions for accessed entities

## Service Patterns

### UnifiedTimelineService

The core service implements several key patterns:

<Tabs>
  <Tab title="Multi-Source Aggregation">
    Queries multiple entity types and merges results into a unified chronological view
  </Tab>
  <Tab title="Page-Based Pagination">
    Applies `page` and `limit` after source entries are merged and sorted
  </Tab>
  <Tab title="Tenant Context Execution">
    All database operations execute within tenant boundaries for data isolation
  </Tab>
  <Tab title="Error Handling">
    Graceful handling of missing entities and invalid queries
  </Tab>
</Tabs>

### Resolver Pattern

<Note>
Timeline formatting is delegated to specialized resolvers to maintain code organization and consistency.
</Note>

- Each source type has its own resolver (e.g., `AuditTimelineResolver`)
- Resolvers handle source-specific formatting and metadata extraction
- Maintains consistency in timeline entry structure across different sources

## Performance Considerations

<Warning>
The Timeline module handles large datasets and requires careful performance optimization.
</Warning>

<CardGroup cols={2}>
  <Card title="Database Indexing" icon="database">
    Relies on proper indexing of timestamp fields for efficient querying
  </Card>
  <Card title="Pagination Strategy" icon="page">
    Fetches bounded source lists, merges in memory, then applies page/limit slicing to the unified result
  </Card>
  <Card title="Query Optimization" icon="bolt">
    Separate queries per source type, then merges in application layer
  </Card>
  <Card title="Caching" icon="memory">
    No explicit caching implemented - relies on database query performance
  </Card>
</CardGroup>

<Tip>
Consider implementing caching strategies for frequently accessed timeline data if performance becomes an issue with high-volume usage.
</Tip>
