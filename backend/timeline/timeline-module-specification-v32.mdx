The Timeline module provides a unified view of chronological events across the PropWise CRM system. It aggregates data from multiple sources (audit logs, activities, and lead-level system events from distribution/escalation) to present a comprehensive timeline of events related to specific entities like leads, contacts, and companies.

## Overview

The Timeline module serves as a central point for tracking historical changes and activities, enabling users to understand the complete lifecycle and interaction history of CRM entities. It combines data from disparate sources into a single chronological view with consistent formatting and pagination.

<Info>
The module acts as an aggregation layer rather than storing its own data, pulling from existing audit logs, activities, and system events to create unified timelines.
</Info>

## Architecture

The Timeline module follows a unified aggregation pattern with these key architectural decisions:

### Core Design Patterns

- **Source Abstraction**: Uses a `TimelineSource` enum to specify which data sources to query (audit, activity, or all)
- **Unified Response Format**: Normalizes different event types into a consistent timeline entry structure
- **Tenant-Aware**: All timeline queries are executed within proper tenant context
- **Pagination Support**: Implements page-based pagination over merged timeline results
- **Resolver Pattern**: Delegates timeline entry formatting to specialized resolver classes

<Tabs>
  <Tab title="Timeline Sources">

    - **Distribution/Escalation**: Lead-level system events for automatic assignment

## API Endpoints

### Get Entity Timeline

<CodeGroup>
```http HTTP
GET /v1/timeline/{lead|deal|contact|company}/:id
```

```typescript TypeScript
// Example usage
const response = await fetch(`/v1/timeline/lead/${leadId}?source=all&page=1&limit=50`);
const timeline = await response.json();
```
</CodeGroup>

#### Parameters

**Path Parameters:**
- `leadId`, `dealId`, `contactId`, or `companyId`: UUID of the target entity

**Query Parameters:**

| Parameter | Type | Required | Default | Description |
|-----------|------|----------|---------|-------------|
| `source` | `TimelineSource` | No | `all` | Data source filter (`audit`, `activity`, `system`, `all`) |
| `page` | `number` | No | 1 | Page number for merged timeline results |
| `limit` | `number` | No | 50 | Number of entries per page (max 200) |
| `from` | `ISO8601` | No | - | Filter events from this date |
| `to` | `ISO8601` | No | - | Filter events until this date |
| `activityType` | `string` | No | - | Single or comma-separated activity type filter |

#### Response Structure

```typescript
{
  data: UnifiedTimelineItemDto[];
  page: number;
  limit: number;
  total: number;
  totalPages: number;
  hasMore: boolean;
}
```

### Timeline Entry Structure

Each timeline entry follows this unified structure:

<CodeGroup>
```typescript Base Structure
{
  id: string;
  source: 'audit' | 'activity' | 'system';
  timestamp: string; // ISO8601
  message: string;
  accentColor?: string; // Optional display accent
  reason?: string; // Optional detail text
  
  actor?: {
    id: string;
    name: string;
    email?: string;
    avatarUrl?: string;
  };
}
```

```typescript Audit Fields
{
  // Audit-only fields
  eventType?: TimelineEventType;
  auditAction?: AuditAction;
  tableName?: string;
  recordId?: string;
}
```

```typescript Activity Fields
{
  // Activity-only fields
  activityType?: ActivityType;
  referenceId?: string;
  targetEntityType?: ActivityTargetType;
  targetEntityId?: string;
  isPinned?: boolean;
  isImportant?: boolean;
  activityDetails?: {
    type: ActivityType;
    // Varies by activity type:
    // call: direction/status/outcome/phone/duration/recording/notes/purpose
    // email: direction/status/from/to/cc/bcc/subject/bodyPlain/provider/tracking
    // sms: direction/status/from/to/body/provider/delivery/error
    // meeting: type/status/timing/location/link/outcome/notes/cancellation
    // site_visit: visit type/status/timing/property/feedback/notes/cancellation
  };
}
```

```typescript System Fields
{
  // System-only fields (lead timelines)
  systemActionType?: 'distribution' | 'escalation';
}
```
</CodeGroup>

### Task-Scoped Feeds

<Warning>
Tasks use separate feed endpoints from the CRM task module, not the generic timeline controller.
</Warning>

<Steps>
  <Step title="Task Timeline Feed">
    `GET /tasks/:id/timeline` - Returns combined task feed (comments + audit events)
  </Step>
  <Step title="Task Activity Feed">
    `GET /tasks/:id/activity` - Returns only audit-backed task activity events  
  </Step>
  <Step title="Task Comments Feed">
    `GET /tasks/:id/comments/feed` - Returns only comments in paginated feed format
  </Step>
</Steps>

All task endpoints use `TaskTimelineFeedQueryDto` with:
- `limit` (default 40, max 100)
- Optional `before` cursor for pagination
- Returns ascending items with `hasMoreOlder` and `nextCursorBefore`

### Commission Payment Timeline

<Note>
Commission payments use a dedicated history endpoint: `GET /commission-payments/:id/history`
</Note>

This endpoint returns immutable `commission_payment_history` rows ordered oldest-first. The UI sorts by `createdAt` descending so newest events appear at the top. Status change notes are stored in the `reason` field.

### Org-Wide Activity Feed

The dashboard Activity Feed uses `UnifiedTimelineService.getActivityFeedInTransaction()` for org-wide, actor-scoped feeds:

<Tabs>
  <Tab title="Data Sources">
    - **Audit Events**: From curated tables (`lead`, `deal`, `contact`, `company`, `entity_stakeholder`, `entity_transfer`, `unit_access_request`)
    - **Activity Logs**: All CRM activity types (note/call/email/meeting/etc.)
    - **Exclusions**: Distribution/escalation system rows not included
  </Tab>
  
  <Tab title="Actor Filtering">
    - `actorUserIds` parameter filters by actor
    - `null` = org-wide scope (no actor filter)
    - Dashboard derives scope from user permissions:
      - Self → `[userId]`  
      - Team → self + team members
      - Org → `null`
  </Tab>
  
  <Tab title="Return Format">
    ```typescript
    UnifiedActivityFeedItem {
      id: string;
      source: 'audit' | 'activity';
      kind: TimelineEventType;
      message: string; // "Lead created", "Moved from New to Contacted"
      occurredAt: string;
      accentColor?: string;
      targetEntityType?: string;
      targetEntityId?: string;
      actor?: ActorInfo;
    }
    ```
  </Tab>
</Tabs>

## Business Rules

<CardGroup cols={2}>
  <Card title="Security & Access" icon="shield-check">
    - Tenant isolation enforced on all queries
    - Permission-based access to entity timelines
    - Users can only view timelines for accessible entities
  </Card>
  
  <Card title="Data Ordering" icon="clock">
    - Entries returned in reverse chronological order (newest first)
    - Chronological ordering maintained across all sources
    - Date range filters are inclusive of boundary dates
  </Card>
</CardGroup>

### Core Rules

1. **Source Filtering**: When `source=all`, entries from all available sources are merged and sorted by timestamp
2. **Pagination Limits**: Maximum of 200 entries per request to prevent performance issues  
3. **Stage Accent Color**: Lead and deal stage-change items include `accentColor` set to destination stage color
4. **Stage Change Reason**: Stage-change items include `reason` when matching stage history has notes
5. **Contact Merge Repointing**: During person duplicate merges, both `Activity.contactId` and Contact-target activities are re-pointed to the surviving Contact

### Special Display Rules

<Warning>
**Lead Disqualification Panel**: Lead disqualification audit messages include the categorical reason in `message`. The Timeline tab displays `lead.disqualificationNotes` as the visible reason detail when notes exist, falling back to the timeline item's `reason` only when no written notes were saved.
</Warning>

<Tip>
**Stage Color Resolution**: System stages are global with per-org color overrides. The resolver loads override maps per distinct org in the batch and resolves each entry's accent against its own `organizationId`, preventing color bleeding between tenants in system-admin queries.
</Tip>
