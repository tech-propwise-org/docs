## Overview

The Timeline module provides a unified view of chronological events across the PropWise CRM system. It aggregates data from multiple sources (audit logs, activities, and lead-level system events from distribution/escalation) to present a comprehensive timeline of events related to specific entities like leads, contacts, and companies. The module serves as a central point for tracking historical changes and activities, enabling users to understand the complete lifecycle and interaction history of CRM entities.

<Note>
**Last Updated:** 2024-12-19  
**Status:** Active
</Note>

## Architecture

The Timeline module follows a unified aggregation pattern, combining data from disparate sources into a single chronological view. Key architectural decisions:

<CardGroup cols={2}>
<Card title="Source abstraction" icon="filter">
Uses a `TimelineSource` enum to specify which data sources to query (audit, activity, or all)
</Card>

<Card title="Unified response format" icon="list">
Normalizes different event types into a consistent timeline entry structure, with optional event details such as stage-change reasons
</Card>

<Card title="Tenant-aware" icon="shield">
All timeline queries are executed within the proper tenant context
</Card>

<Card title="Pagination support" icon="page">
Implements page-based pagination over the merged timeline results
</Card>
</CardGroup>

### Resolver Pattern

The module delegates timeline entry formatting to specialized resolver classes:
- `AuditTimelineResolver`
- `DistributionTimelineResolver` 
- `EscalationTimelineResolver`

## Entity Relationships

<Info>
The Timeline module doesn't define its own entities but aggregates data from existing sources.
</Info>

The module integrates with the following entities:

- **AuditLog**: System audit entries tracking entity changes
- **Activity**: User-created activities and interactions
- **ContactCompanyRole**: Relationship tracking between contacts and companies
- **LeadStageHistory / DealStageHistory**: Stage transition history used to attach user-provided stage-change reasons to matching audit timeline entries
- **DistributionLog / EscalationActionLog**: Lead-level system events for automatic assignment and escalation activity

## API Endpoints

### Get Entity Timeline

```http
GET /v1/timeline/{lead|deal|contact|company}/:id
```

**Path Parameters:**
- `leadId`, `dealId`, `contactId`, or `companyId`: UUID of the target entity

**Query Parameters:**

| Parameter | Type | Required | Default | Description |
|-----------|------|----------|---------|-------------|
| `source` | `TimelineSource` | No | `all` | Data source filter (`audit`, `activity`, `system`, `all`) |
| `page` | `number` | No | 1 | Page number for merged timeline results |
| `limit` | `number` | No | 50 | Number of entries per page (max 200) |
| `from` | `ISO8601` | No | - | Filter events from this date |
| `to` | `ISO8601` | No | - | Filter events until this date |
| `activityType` | `string` | No | - | Single or comma-separated activity type filter, applied to activity entries |

**Response Structure:**

```typescript
{
  data: UnifiedTimelineItemDto[];
  page: number;
  limit: number;
  total: number;
  totalPages: number;
  hasMore: boolean;
}
```

### Timeline Entry Structure

<CodeGroup>

```typescript Timeline Entry
{
  id: string;
  source: 'audit' | 'activity' | 'system';
  timestamp: string; // ISO8601
  message: string;
  accentColor?: string; // Optional display accent; stage changes use destination stage color
  reason?: string; // Optional detail text; stage changes use the matching stage history notes

  // Audit-only
  eventType?: TimelineEventType;
  auditAction?: AuditAction;
  tableName?: string;
  recordId?: string;

  // Activity-only
  activityType?: ActivityType;
  referenceId?: string;
  targetEntityType?: ActivityTargetType;
  targetEntityId?: string;
  isPinned?: boolean;
  isImportant?: boolean;

  // System-only (lead timelines)
  systemActionType?: 'distribution' | 'escalation';

  actor?: {
    id: string;
    name: string;
    email?: string;
    avatarUrl?: string;
  };
}
```

```typescript Query DTO
class UnifiedTimelineQueryDto {
  source?: TimelineSource = TimelineSource.ALL;
  from?: string; // ISO8601
  to?: string; // ISO8601
  page?: number = 1;
  limit?: number = 50; // min: 1, max: 200
  activityType?: string; // single value or comma-separated list
}
```

```typescript Timeline Source
enum TimelineSource {
  AUDIT = 'audit',
  ACTIVITY = 'activity',
  SYSTEM = 'system',
  ALL = 'all',
}
```

</CodeGroup>

### Task-Scoped Feeds

<Warning>
Tasks expose their own feed endpoints from the CRM task module rather than the generic `/timeline/:entityType/:entityId` controller.
</Warning>

<Tabs>
<Tab title="Task Timeline">
```http
GET /tasks/:id/timeline
```
Returns the combined task feed (`TaskComment` rows + task `AuditLog` events).
</Tab>

<Tab title="Task Activity">
```http
GET /tasks/:id/activity
```
Returns only audit-backed task activity events.
</Tab>

<Tab title="Task Comments Feed">
```http
GET /tasks/:id/comments/feed
```
Returns only comments in the same keyset-paginated feed shape.
</Tab>
</Tabs>

All three endpoints share the same `TaskTimelineFeedQueryDto` contract:
- `limit` (default 40, max 100)
- Optional `before` cursor

They return `TaskTimelineFeedResponse` with:
- Ascending items for the fetched page
- `hasMoreOlder` flag
- `nextCursorBefore` cursor

<Note>
Use `GET /tasks/:id/comments` only when the caller needs the legacy full comment array.
</Note>

## Business Rules

<AccordionGroup>
<Accordion title="Security & Access Control">
1. **Tenant Isolation**: Timeline entries are strictly filtered by tenant context
2. **Permission-Based Access**: Users can only view timelines for entities they have access to
</Accordion>

<Accordion title="Data Ordering & Pagination">
3. **Chronological Ordering**: Entries are always returned in reverse chronological order (newest first)
4. **Pagination Limits**: Maximum of 200 entries per request to prevent performance issues
5. **Date Range Filtering**: When date filters are applied, they are inclusive of the boundary dates
</Accordion>

<Accordion title="Source Integration">
6. **Source Filtering**: When `source=all`, entries from all available sources are merged and sorted by timestamp. Lead timelines include audit, activity, distribution, and escalation entries; deal/contact/company timelines include audit and activity entries.
7. **Stage Accent Color**: Lead and deal stage-change timeline items include `accentColor` set to the destination stage color so clients can render timeline indicators without parsing stage names or fetching stage metadata.
8. **Stage Change Reason**: Lead and deal stage-change timeline items include `reason` when a matching stage history record has notes. Matching uses entity ID, from-stage ID, to-stage ID, and a short timestamp window so repeated transitions do not receive the wrong reason.
</Accordion>

<Accordion title="Data Consistency">
9. **Contact Merge Repointing**: Person duplicate merges can absorb duplicate Contact records. During that merge, both `Activity.contactId` and direct Contact-target activities (`targetEntityType='contact'`, `targetEntityId=<absorbedContactId>`) are re-pointed to the surviving Contact, so the Contact timeline continues to show notes, calls, meetings, and message bridge activity after the absorbed Contact is soft-deleted.
</Accordion>
</AccordionGroup>

## Integration Points

### Audit Module

<Steps>
<Step title="Consume audit logs">
Consumes `AuditLog` entities to show system-generated change events
</Step>

<Step title="Format entries">
Uses `AuditTimelineResolver` to format audit entries into timeline format
</Step>

<Step title="Filter noise">
Filters out implementation details including stage tracking fields (`stage_entered_at`, `stageEnteredAt`) that are automatically updated alongside stage changes
</Step>

<Step title="Resolve stage information">
Resolves stage names and destination stage colors in batch for lead/deal stage-change entries; the unified response exposes the color as `accentColor`
</Step>

<Step title="Enrich with reasons">
Stage-change audit entries are enriched with `reason` from matching `LeadStageHistory` or `DealStageHistory` notes after audit message resolution
</Step>
</Steps>

### CRM Module

- Integrates with `Activity` entities to show user-created activities
- Supports all activity target types (leads, contacts, companies, etc.)
- Leverages `ContactCompanyRole` for relationship context

### Distribution Module

- Lead timelines include `DistributionLog` rows as `source='system'`
- `DistributionTimelineResolver` maps assignment, pool, and fallback outcomes to display-ready timeline messages

### Escalation Module

- Lead timelines include `EscalationActionLog` rows as `source='system'` with `systemActionType='escalation'`
- `EscalationTimelineResolver` formats notification, redistribution, temperature-change, and tag-add actions

<Tip>
Use `source=system` to fetch only distribution/escalation entries for a lead timeline.
</Tip>

### User Module

- Enriches timeline entries with user information (name, email)
- Ensures proper user references in timeline metadata

### Auth Module

- Enforces JWT authentication on all timeline endpoints
- Validates user permissions for accessed entities

## Service Patterns

### UnifiedTimelineService

The core service implements several key patterns:

<CardGroup cols={2}>
<Card title="Multi-source aggregation" icon="merge">
Queries multiple entity types and merges results
</Card>

<Card title="Page-based pagination" icon="list-ol">
Applies `page` and `limit` after source entries are merged and sorted
</Card>

<Card title="Tenant context execution" icon="building">
All database operations execute within tenant boundaries
</Card>

<Card title="Error handling" icon="exclamation-triangle">
Graceful handling of missing entities and invalid queries
</Card>
</CardGroup>

### Resolver Pattern

Timeline formatting is delegated to specialized resolvers:

- Each source type has its own resolver (e.g., `AuditTimelineResolver`)
- Resolvers handle source-specific formatting and metadata extraction
- Maintains consistency in timeline entry structure across different sources

## Performance Considerations

<Check>
**Database Indexing**: Relies on proper indexing of timestamp fields for efficient querying
</Check>

<Check>
**Pagination**: Fetches bounded source lists, merges in memory, then applies page/limit slicing to the unified result
</Check>

<Check>
**Query Optimization**: Separate queries per source type, then merges in application layer
</Check>

<Warning>
**Caching**: No explicit caching implemented - relies on database query performance
