## Overview

The Timeline module provides a unified view of chronological events across the PropWise CRM system. It aggregates data from multiple sources (audit logs and activities) to present a comprehensive timeline of events related to specific entities like leads, contacts, and companies. The module serves as a central point for tracking historical changes and activities, enabling users to understand the complete lifecycle and interaction history of CRM entities.

<Note>
The Timeline module was last updated on **2024-12-19** and is currently in **Active** status.
</Note>

## Architecture

The Timeline module follows a unified aggregation pattern, combining data from disparate sources into a single chronological view.

### Key architectural decisions

<CardGroup cols={2}>
<Card title="Source abstraction" icon="filter">
Uses a `TimelineSource` enum to specify which data sources to query (audit, activity, or all)
</Card>
<Card title="Unified response format" icon="align-left">
Normalizes different event types into a consistent timeline entry structure, with optional event details such as stage-change reasons
</Card>
<Card title="Tenant-aware" icon="shield">
All timeline queries are executed within the proper tenant context
</Card>
<Card title="Resolver pattern" icon="code">
Delegates timeline entry formatting to specialized resolver classes (e.g., `AuditTimelineResolver`)
</Card>
</CardGroup>

The module also implements **pagination support** with page-based pagination over the merged timeline results.

## Entity relationships

<Warning>
The Timeline module doesn't define its own entities but aggregates data from existing sources.
</Warning>

The Timeline module aggregates data from:

- **AuditLog**: System audit entries tracking entity changes
- **Activity**: User-created activities and interactions  
- **ContactCompanyRole**: Relationship tracking between contacts and companies
- **LeadStageHistory / DealStageHistory**: Stage transition history used to attach user-provided stage-change reasons to matching audit timeline entries

## API endpoints

### Get entity timeline

Retrieves a paginated timeline of events for a specific entity.

```http
GET /timeline/:entityType/:entityId
```

**Path parameters:**

| Parameter | Type | Description |
|-----------|------|-------------|
| `entityType` | string | Type of entity (lead, contact, company, etc.) |
| `entityId` | string | UUID of the target entity |

**Query parameters:**

| Parameter | Type | Required | Default | Description |
|-----------|------|----------|---------|-------------|
| `source` | `TimelineSource` | No | `all` | Data source filter (`audit`, `activity`, `system`, `all`) |
| `page` | number | No | 1 | Page number for merged timeline results |
| `limit` | number | No | 50 | Number of entries per page (max 200) |
| `from` | ISO8601 | No | - | Filter events from this date |
| `to` | ISO8601 | No | - | Filter events until this date |
| `activityType` | string | No | - | Single or comma-separated activity type filter, applied to activity entries |

<Tip>
The `limit` parameter has a maximum value of 200 entries per request to prevent performance issues.
</Tip>

**Response structure:**

<CodeGroup>
```typescript Response
{
  data: UnifiedTimelineItemDto[];
  page: number;
  limit: number;
  total: number;
  totalPages: number;
  hasMore: boolean;
}
```

```typescript Timeline entry structure
{
  id: string;
  source: 'audit' | 'activity' | 'system';
  timestamp: string; // ISO8601
  message: string;
  accentColor?: string; // Optional display accent; stage changes use destination stage color
  reason?: string; // Optional detail text; stage changes use the matching stage history notes

  // Audit-only
  eventType?: TimelineEventType;
  auditAction?: AuditAction;
  tableName?: string;
  recordId?: string;

  // Activity-only
  activityType?: ActivityType;
  referenceId?: string;
  targetEntityType?: ActivityTargetType;
  targetEntityId?: string;
  isPinned?: boolean;
  isImportant?: boolean;

  // System-only
  systemActionType?: 'distribution' | 'escalation';

  actor?: {
    id: string;
    name: string;
    email?: string;
    avatarUrl?: string;
  };
}
```
</CodeGroup>

## Business rules

<Steps>
<Step title="Tenant isolation">
Timeline entries are strictly filtered by tenant context
</Step>
<Step title="Permission-based access">
Users can only view timelines for entities they have access to
</Step>
<Step title="Chronological ordering">
Entries are always returned in reverse chronological order (newest first)
</Step>
<Step title="Source filtering">
When `source=all`, entries from all sources are merged and sorted by timestamp
</Step>
<Step title="Pagination limits">
Maximum of 200 entries per request to prevent performance issues
</Step>
<Step title="Date range filtering">
When date filters are applied, they are inclusive of the boundary dates
</Step>
</Steps>

### Stage change enhancements

<Info>
Lead and deal stage-change timeline items include special enhancements for better user experience.
</Info>

- **Stage accent color**: Timeline items include `accentColor` set to the destination stage color so clients can render timeline indicators without parsing stage names or fetching stage metadata
- **Stage change reason**: Timeline items include `reason` when a matching stage history record has notes. Matching uses entity ID, from-stage ID, to-stage ID, and a short timestamp window so repeated transitions do not receive the wrong reason

### Contact merge repointing

<Note>
During person duplicate merges, both `Activity.contactId` and direct Contact-target activities are re-pointed to the surviving Contact, ensuring timeline continuity after absorbed contacts are soft-deleted.
</Note>

## Integration points

<Tabs>
<Tab title="Audit Module">

- Consumes `AuditLog` entities to show system-generated change events
- Uses `AuditTimelineResolver` to format audit entries into timeline format
- Filters out noise from implementation details including stage tracking fields (`stage_entered_at`, `stageEnteredAt`) that are automatically updated alongside stage changes
- Resolves stage names and destination stage colors in batch for lead/deal stage-change entries; the unified response exposes the color as `accentColor`
- Stage-change audit entries are enriched with `reason` from matching `LeadStageHistory` or `DealStageHistory` notes after audit message resolution

</Tab>
<Tab title="CRM Module">

- Integrates with `Activity` entities to show user-created activities
- Supports all activity target types (leads, contacts, companies, etc.)
- Leverages `ContactCompanyRole` for relationship context

</Tab>
<Tab title="User Module">

- Enriches timeline entries with user information (name, email)
- Ensures proper user references in timeline metadata

</Tab>
<Tab title="Auth Module">

- Enforces JWT authentication on all timeline endpoints
- Validates user permissions for accessed entities

</Tab>
</Tabs>

## Service patterns

### UnifiedTimelineService

The core service implements several key patterns:

<AccordionGroup>
<Accordion title="Multi-source aggregation">
Queries multiple entity types and merges results into a unified timeline view.
</Accordion>

<Accordion title="Page-based pagination">
Applies `page` and `limit` parameters after source entries are merged and sorted chronologically.
</Accordion>

<Accordion title="Tenant context execution">
All database operations execute within tenant boundaries to ensure data isolation.
</Accordion>

<Accordion title="Error handling">
Graceful handling of missing entities and invalid queries with appropriate error responses.
</Accordion>
</AccordionGroup>

### Resolver pattern

Timeline formatting is delegated to specialized resolvers:

- Each source type has its own resolver (e.g., `AuditTimelineResolver`)
- Resolvers handle source-specific formatting and metadata extraction
- Maintains consistency in timeline entry structure across different sources

## Performance considerations

<Warning>
The Timeline module relies on proper database optimization and indexing for efficient performance.
</Warning>

Key performance considerations include:

- **Database indexing**: Relies on proper indexing of timestamp fields for efficient querying
- **Pagination**: Fetches bounded source lists, merges in memory, then applies page/limit slicing to the unified result
- **Query optimization**: Separate queries per source type, then merges in application layer
- **Caching**: No explicit caching implemented - relies on database query performance

<Check>
The module is designed to handle high-volume timeline queries efficiently through proper pagination and query optimization strategies.
</Check>
