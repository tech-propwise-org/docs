## Overview

The Timeline module provides a unified view of chronological events across the PropWise CRM system. It aggregates data from multiple sources (audit logs and activities) to present a comprehensive timeline of events related to specific entities like leads, contacts, and companies. The module serves as a central point for tracking historical changes and activities, enabling users to understand the complete lifecycle and interaction history of CRM entities.

## Architecture

The Timeline module follows a unified aggregation pattern, combining data from disparate sources into a single chronological view. Key architectural decisions:

- **Source Abstraction**: Uses a `TimelineSource` enum to specify which data sources to query (audit, activity, or all)
- **Unified Response Format**: Normalizes different event types into a consistent timeline entry structure
- **Tenant-Aware**: All timeline queries are executed within the proper tenant context
- **Pagination Support**: Implements cursor-based pagination for efficient timeline browsing
- **Resolver Pattern**: Delegates timeline entry formatting to specialized resolver classes (e.g., `AuditTimelineResolver`)

<Info>
The module doesn't define its own entities but aggregates data from existing sources, making it a pure aggregation service.
</Info>

## Entity relationships

The Timeline module doesn't define its own entities but aggregates data from:

- **AuditLog**: System audit entries tracking entity changes
- **Activity**: User-created activities and interactions
- **ContactCompanyRole**: Relationship tracking between contacts and companies

## API endpoints

### Get entity timeline

Retrieves the timeline for a specific entity with optional filtering and pagination.

```http
GET /timeline/:entityType/:entityId
```

**Path parameters:**
- `entityType` (string): Type of entity (lead, contact, company, etc.)
- `entityId` (string): UUID of the target entity

**Query parameters:**

| Parameter | Type | Required | Default | Description |
|-----------|------|----------|---------|-------------|
| `source` | `TimelineSource` | No | `all` | Data source filter (audit/activity/all) |
| `limit` | `number` | No | 50 | Number of entries per page (max 100) |
| `cursor` | `string` | No | - | Pagination cursor |
| `startDate` | `ISO8601` | No | - | Filter events from this date |
| `endDate` | `ISO8601` | No | - | Filter events until this date |

**Response format:**

<CodeGroup>
```typescript Response Structure
{
  items: UnifiedTimelineEntryDto[];
  hasNextPage: boolean;
  nextCursor?: string;
  totalCount?: number;
}
```

```typescript Timeline Entry
{
  id: string;
  timestamp: string; // ISO8601
  type: TimelineEventType;
  source: TimelineSource;
  title: string;
  description?: string;
  user?: {
    id: string;
    name: string;
    email: string;
  };
  metadata?: Record<string, any>;
}
```
</CodeGroup>

<Tip>
Use cursor-based pagination for optimal performance when browsing large timelines. The `nextCursor` value should be passed to subsequent requests.
</Tip>

## DTOs

### UnifiedTimelineQueryDto

```typescript
class UnifiedTimelineQueryDto {
  source?: TimelineSource = TimelineSource.ALL;
  limit?: number = 50; // min: 1, max: 100
  cursor?: string;
  startDate?: string; // ISO8601
  endDate?: string; // ISO8601
}
```

### TimelineSource enum

```typescript
enum TimelineSource {
  AUDIT = 'audit',
  ACTIVITY = 'activity',
  ALL = 'all',
}
```

<Warning>
The maximum limit per request is 100 entries to prevent performance issues. Larger datasets should be accessed through pagination.
</Warning>

## Business rules

<AccordionGroup>
  <Accordion title="Core access and security rules">
    1. **Tenant Isolation**: Timeline entries are strictly filtered by tenant context
    2. **Permission-Based Access**: Users can only view timelines for entities they have access to
  </Accordion>
  
  <Accordion title="Data ordering and filtering rules">
    3. **Chronological Ordering**: Entries are always returned in reverse chronological order (newest first)
    4. **Source Filtering**: When `source=all`, entries from all sources are merged and sorted by timestamp
    5. **Date Range Filtering**: When date filters are applied, they are inclusive of the boundary dates
  </Accordion>
  
  <Accordion title="Performance and pagination rules">
    6. **Pagination Limits**: Maximum of 100 entries per request to prevent performance issues
  </Accordion>
</AccordionGroup>

## Service patterns

### UnifiedTimelineService

The core service implements several key patterns:

<Steps>
  <Step title="Multi-source aggregation">
    Queries multiple entity types and merges results into a unified timeline view
  </Step>
  
  <Step title="Cursor-based pagination">
    Implements efficient pagination using timestamp cursors to avoid expensive OFFSET queries
  </Step>
  
  <Step title="Tenant context execution">
    All database operations execute within tenant boundaries to ensure data isolation
  </Step>
  
  <Step title="Error handling">
    Graceful handling of missing entities and invalid queries with appropriate error responses
  </Step>
</Steps>

### Resolver pattern

Timeline formatting is delegated to specialized resolvers:

- Each source type has its own resolver (e.g., `AuditTimelineResolver`)
- Resolvers handle source-specific formatting and metadata extraction
- Maintains consistency in timeline entry structure across different sources

<Info>
The resolver pattern allows for easy extension when adding new timeline sources without modifying the core service logic.
</Info>
