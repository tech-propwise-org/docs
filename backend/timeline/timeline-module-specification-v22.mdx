## Overview

The Timeline module provides a unified view of chronological events across the PropWise CRM system. It aggregates data from multiple sources (audit logs, activities, and lead-level system events from distribution/escalation) to present a comprehensive timeline of events related to specific entities like leads, contacts, and companies.

The module serves as a central point for tracking historical changes and activities, enabling users to understand the complete lifecycle and interaction history of CRM entities.

## Architecture

The Timeline module follows a unified aggregation pattern, combining data from disparate sources into a single chronological view.

### Key architectural decisions

- **Source Abstraction**: Uses a `TimelineSource` enum to specify which data sources to query (audit, activity, or all)
- **Unified Response Format**: Normalizes different event types into a consistent timeline entry structure, with optional event details such as stage-change reasons
- **Tenant-Aware**: All timeline queries are executed within the proper tenant context
- **Pagination Support**: Implements page-based pagination over the merged timeline results
- **Resolver Pattern**: Delegates timeline entry formatting to specialized resolver classes (e.g., `AuditTimelineResolver`, `DistributionTimelineResolver`, `EscalationTimelineResolver`)

## Entity Relationships

The Timeline module doesn't define its own entities but aggregates data from:

<AccordionGroup>
<Accordion title="Core entities">
- **AuditLog**: System audit entries tracking entity changes
- **Activity**: User-created activities and interactions, including lightweight scheduled task/event index rows for entity-linked task/event records that do not create a typed channel activity
- **ContactCompanyRole**: Relationship tracking between contacts and companies
</Accordion>

<Accordion title="Stage history">
- **LeadStageHistory / DealStageHistory**: Stage transition history used to attach user-provided stage-change reasons to matching audit timeline entries
</Accordion>

<Accordion title="System events">
- **DistributionLog / EscalationActionLog**: Lead-level system events for automatic assignment and escalation activity
- **EntityTransfer**: Related lead/deal transfer request audit entries are included on the owning entity timeline
- **UnitAccessRequest**: Deal-linked unit access request audit entries are included on the related deal timeline
</Accordion>
</AccordionGroup>

## API Endpoints

### Get Entity Timeline

```http
GET /v1/timeline/{lead|deal|contact|company}/:id
```

**Parameters:**

- `leadId`, `dealId`, `contactId`, or `companyId` (path): UUID of the target entity

**Query Parameters:**

| Parameter | Type | Required | Default | Description |
|-----------|------|----------|---------|-------------|
| `source` | `TimelineSource` | No | `all` | Data source filter (`audit`, `activity`, `system`, `all`) |
| `page` | `number` | No | 1 | Page number for merged timeline results |
| `limit` | `number` | No | 50 | Number of entries per page (max 200) |
| `from` | `ISO8601` | No | - | Filter events from this date |
| `to` | `ISO8601` | No | - | Filter events until this date |
| `activityType` | `string` | No | - | Single or comma-separated activity type filter, applied to activity entries |

**Response:**

<CodeGroup>
```typescript Response format
{
  data: UnifiedTimelineItemDto[];
  page: number;
  limit: number;
  total: number;
  totalPages: number;
  hasMore: boolean;
}
```

```typescript Timeline entry structure
{
  id: string;
  source: 'audit' | 'activity' | 'system';
  timestamp: string; // ISO8601
  message: string;
  accentColor?: string; // Optional display accent; stage changes use destination stage color
  reason?: string; // Optional detail text; stage changes use the matching stage history notes

  // Audit-only
  eventType?: TimelineEventType;
  auditAction?: AuditAction;
  tableName?: string;
  recordId?: string;

  // Activity-only
  activityType?: ActivityType;
  referenceId?: string;
  targetEntityType?: ActivityTargetType;
  targetEntityId?: string;
  isPinned?: boolean;
  isImportant?: boolean;

  // System-only (lead timelines)
  systemActionType?: 'distribution' | 'escalation';

  actor?: {
    id: string;
    name: string;
    email?: string;
    avatarUrl?: string;
  };
}
```
</CodeGroup>

### Task-Scoped Feeds

<Info>
Tasks expose their own feed endpoints from the CRM task module rather than the generic `/timeline/:entityType/:entityId` controller.
</Info>

- `GET /tasks/:id/timeline` returns the combined task feed (`TaskComment` rows + task `AuditLog` events)
- `GET /tasks/:id/activity` returns only audit-backed task activity events
- `GET /tasks/:id/comments/feed` returns only comments in the same keyset-paginated feed shape

All three endpoints share the same `TaskTimelineFeedQueryDto` contract: `limit` (default 40, max 100) and optional `before` cursor. They return `TaskTimelineFeedResponse` with ascending items for the fetched page, `hasMoreOlder`, and `nextCursorBefore`.

<Tip>
Use `GET /tasks/:id/comments` only when the caller needs the legacy full comment array.
</Tip>

### Commission Payment Timeline

<Warning>
Commission payments do not use the generic unified timeline controller.
</Warning>

The commission panel Timeline tab reads `GET /commission-payments/:id/history`, which returns immutable `commission_payment_history` rows ordered oldest-first. The side panel sorts those rows by `createdAt` descending for display so the newest event appears at the top, matching lead/deal panel timelines.

Notes entered during a status change are persisted on the history row `reason` field and displayed only in this Timeline tab. Free-form commission panel notes use the Activity/Note system (`targetEntityType='commission_payment'`) and are isolated to the panel Notes tab.

## Business Rules

<Steps>
<Step title="Security and access">
**Tenant Isolation**: Timeline entries are strictly filtered by tenant context

**Permission-Based Access**: Users can only view timelines for entities they have access to
</Step>

<Step title="Data ordering and filtering">
**Chronological Ordering**: Entries are always returned in reverse chronological order (newest first)

**Source Filtering**: When `source=all`, entries from all available sources are merged and sorted by timestamp. Lead timelines include direct lead audit, related transfer audit, activity, distribution, and escalation entries; deal timelines include direct deal audit, related transfer audit, related unit-access request audit, and activity entries; contact/company timelines include audit and activity entries.

**Date Range Filtering**: When date filters are applied, they are inclusive of the boundary dates
</Step>

<Step title="Performance limits">
**Pagination Limits**: Maximum of 200 entries per request to prevent performance issues
</Step>

<Step title="Stage changes and visual indicators">
**Stage Accent Color**: Lead and deal stage-change timeline items include `accentColor` set to the destination stage color so clients can render timeline indicators without parsing stage names or fetching stage metadata.

**Stage Change Reason**: Lead and deal stage-change timeline items include `reason` when a matching stage history record has notes. Matching uses entity ID, from-stage ID, to-stage ID, and a short timestamp window so repeated transitions do not receive the wrong reason.
</Step>

<Step title="Special display rules">
**Lead Disqualification Panel Display**: Lead disqualification audit messages include the categorical disqualification reason in `message` (for example, `Lead disqualified: Budget`). The lead panel Timeline tab displays `lead.disqualificationNotes` as the visible `Reason:` detail when notes exist, falling back to the timeline item's `reason` only when no written notes were saved.

**Contact Merge Repointing**: Person duplicate merges can absorb duplicate Contact records. During that merge, both `Activity.contactId` and direct Contact-target activities (`targetEntityType='contact'`, `targetEntityId=<absorbedContactId>`) are re-pointed to the surviving Contact, so the Contact timeline continues to show notes, calls, meetings, and message bridge activity after the absorbed Contact is soft-deleted.

**Scheduled Task/Event Visibility**: Creating a task or calendar event from a lead/deal/contact/company panel writes an `Activity` index row for that entity when the task/event does not already create a typed call/meeting/site-visit/email activity. Task rows use `activityType='task'` with a `Scheduled task Â· ...` summary; generic/custom event rows use `activityType='other'` with a `Scheduled event Â· ...` summary. Typed task/event channel flows keep the single typed activity row to avoid duplicates.
</Step>
</Steps>

## Service Patterns

### UnifiedTimelineService

The core service implements several key patterns:

<Tabs>
<Tab title="Multi-source aggregation">
Queries multiple entity types and merges results into a unified timeline view.
</Tab>

<Tab title="Page-based pagination">
Applies `page` and `limit` parameters after source entries are merged and sorted chronologically.
</Tab>

<Tab title="Tenant context execution">
All database operations execute within tenant boundaries to ensure data isolation.
</Tab>

<Tab title="Error handling">
Graceful handling of missing entities and invalid queries with appropriate error responses.
</Tab>
</Tabs>

### Resolver Pattern

Timeline formatting is delegated to specialized resolvers:

- Each source type has its own resolver (e.g., `AuditTimelineResolver`)
- Resolvers handle source-specific formatting and metadata extraction
- Maintains consistency in timeline entry structure across different sources
