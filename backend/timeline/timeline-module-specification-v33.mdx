## Overview

The Timeline module provides a unified view of chronological events across the PropWise CRM system. It aggregates data from multiple sources (audit logs, activities, and lead-level system events from distribution/escalation) to present a comprehensive timeline of events related to specific entities like leads, contacts, and companies.

<Info>
The module serves as a central point for tracking historical changes and activities, enabling users to understand the complete lifecycle and interaction history of CRM entities.
</Info>

## Architecture

The Timeline module follows a unified aggregation pattern, combining data from disparate sources into a single chronological view.

### Key Architectural Decisions

<CardGroup cols={2}>
  <Card title="Source Abstraction" icon="layer-group">
    Uses a `TimelineSource` enum to specify which data sources to query (audit, activity, or all)
  </Card>
  <Card title="Unified Response Format" icon="timeline">
    Normalizes different event types into a consistent timeline entry structure
  </Card>
  <Card title="Tenant-Aware" icon="shield-check">
    All timeline queries are executed within the proper tenant context
  </Card>
  <Card title="Resolver Pattern" icon="code">
    Delegates timeline entry formatting to specialized resolver classes
  </Card>
</CardGroup>

### Additional Features

- **Pagination Support**: Implements page-based pagination over the merged timeline results
- **Optional Event Details**: Includes stage-change reasons and compact activity channel metadata

## Entity Relationships

<Note>
The Timeline module doesn't define its own entities but aggregates data from multiple sources.
</Note>

<AccordionGroup>
  <Accordion title="Core Data Sources">
    - **AuditLog**: System audit entries tracking entity changes
    - **Activity**: User-created activities and interactions
    - **Activity Channel Tables**: `call_log`, `email_log`, `sms_log`, `meeting`, and `site_visit` rows
    - **ContactCompanyRole**: Relationship tracking between contacts and companies
  </Accordion>
  
  <Accordion title="Stage History Sources">
    - **LeadStageHistory / DealStageHistory**: Stage transition history for stage-change reasons
    - **DistributionLog / EscalationActionLog**: Lead-level system events
    - **EntityTransfer**: Related lead/deal transfer request audit entries
    - **UnitAccessRequest**: Deal-linked unit access request audit entries
  </Accordion>
  
  <Accordion title="Property Interest Events">
    - **LeadPropertyInterest / DealPropertyInterest**: Property interest changes
    - Timeline events: `LEAD_PROPERTY_INTEREST_ADDED`, `LEAD_PROPERTY_INTEREST_UPDATED`, `LEAD_PROPERTY_INTEREST_REMOVED`
    - Deal events: `DEAL_PROPERTY_INTEREST_ADDED`, `DEAL_PROPERTY_INTEREST_UPDATED`, `DEAL_PROPERTY_INTEREST_REMOVED`
  </Accordion>
</AccordionGroup>

## API Endpoints

### Get Entity Timeline

<CodeGroup>
```http GET
GET /v1/timeline/{lead|deal|contact|company}/:id
```
</CodeGroup>

**Path Parameters:**
- `leadId`, `dealId`, `contactId`, or `companyId`: UUID of the target entity

**Query Parameters:**

| Parameter | Type | Required | Default | Description |
|-----------|------|----------|---------|-------------|
| `source` | `TimelineSource` | No | `all` | Data source filter (`audit`, `activity`, `system`, `all`) |
| `page` | `number` | No | 1 | Page number for merged timeline results |
| `limit` | `number` | No | 50 | Number of entries per page (max 200) |
| `from` | `ISO8601` | No | - | Filter events from this date |
| `to` | `ISO8601` | No | - | Filter events until this date |
| `activityType` | `string` | No | - | Single or comma-separated activity type filter |

**Response Structure:**

```typescript
{
  data: UnifiedTimelineItemDto[];
  page: number;
  limit: number;
  total: number;
  totalPages: number;
  hasMore: boolean;
}
```

### Task-Scoped Feeds

<Warning>
Tasks use dedicated endpoints rather than the generic timeline controller.
</Warning>

- `GET /tasks/:id/timeline` - Combined task feed (comments + audit events)
- `GET /tasks/:id/activity` - Audit-backed task activity events only
- `GET /tasks/:id/comments/feed` - Comments only in keyset-paginated format

**Query Parameters:**
- `limit`: Default 40, max 100
- `before`: Optional cursor for pagination

### Commission Payment Timeline

<Note>
Commission payments have their own specialized timeline endpoint.
</Note>

- `GET /commission-payments/:id/history` - Returns immutable commission payment history
- Notes are persisted on the `reason` field of history rows
- Activity/Note system is isolated to the Notes tab

### Org-Wide Actor Activity Feed

The dashboard Activity Feed uses `UnifiedTimelineService.getActivityFeedInTransaction()`:

<Steps>
  <Step title="Source Aggregation">
    Merges audit lifecycle events from curated tables with logged CRM activity rows
  </Step>
  <Step title="Actor Filtering">
    Filters by `actorUserIds` based on caller's CRM/Sales scope (self, team, or org-wide)
  </Step>
  <Step title="Cursor Pagination">
    Uses `(occurredAt DESC, id DESC)` ordering with cursor-based pagination
  </Step>
  <Step title="Response Formatting">
    Returns `UnifiedActivityFeedItem[]` with resolved messages and accent colors
  </Step>
</Steps>

## DTOs

### UnifiedTimelineQueryDto

```typescript
class UnifiedTimelineQueryDto {
  source?: TimelineSource = TimelineSource.ALL;
  from?: string; // ISO8601
  to?: string; // ISO8601

  limit?: number = 50; // min: 1, max: 200

}
```

### TimelineSource Enum

```typescript
enum TimelineSource {
  AUDIT = 'audit',
  ACTIVITY = 'activity',
  SYSTEM = 'system',
  ALL = 'all',
}
```

## Business Rules

<AccordionGroup>
  <Accordion title="Security and Access Control">
    1. **Tenant Isolation**: Timeline entries are strictly filtered by tenant context
    2. **Permission-Based Access**: Users can only view timelines for entities they have access to
  </Accordion>
  
  <Accordion title="Data Organization and Filtering">
    3. **Chronological Ordering**: Entries are always returned in reverse chronological order (newest first)
    4. **Source Filtering**: When `source=all`, entries from all available sources are merged and sorted by timestamp
    5. **Pagination Limits**: Maximum of 200 entries per request to prevent performance issues
    6. **Date Range Filtering**: When date filters are applied, they are inclusive of the boundary dates
  </Accordion>
  
  <Accordion title="Visual Enhancements">
    7. **Stage Accent Color**: Lead and deal stage-change timeline items include `accentColor` set to the destination stage color
    8. **Stage Change Reason**: Timeline items include `reason` when matching stage history records have notes
  </Accordion>
  
  <Accordion title="Special Display Rules">
    9. **Lead Disqualification Panel Display**: Shows `lead.disqualificationNotes` as the visible reason, falling back to timeline item's `reason`
    10. **Contact Merge Repointing**: During person duplicate merges, both `Activity.contactId` and direct Contact-target activities are re-pointed to the surviving Contact
  </Accordion>
</AccordionGroup>

<Tip>
The Timeline module's unified approach ensures consistent user experience across all CRM entities while maintaining the flexibility to display source-specific information when needed.
</Tip>
