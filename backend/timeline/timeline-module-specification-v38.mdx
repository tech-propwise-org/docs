## Overview

The Timeline module provides a unified view of chronological events across the PropWise CRM system. It aggregates data from multiple sources (audit logs, activities, and lead-level system events from distribution/escalation) to present a comprehensive timeline of events related to specific entities like leads, contacts, and companies.

<Info>
The module serves as a central point for tracking historical changes and activities, enabling users to understand the complete lifecycle and interaction history of CRM entities.
</Info>

**Status:** Active  
**Last Updated:** 2026-05-23

## Architecture

The Timeline module follows a unified aggregation pattern, combining data from disparate sources into a single chronological view.

### Key Architectural Decisions

<CardGroup cols={2}>
  <Card title="Source Abstraction" icon="filter">
    Uses a `TimelineSource` enum to specify which data sources to query (audit, activity, system, or all)
  </Card>
  <Card title="Unified Response Format" icon="table">
    Normalizes different event types into a consistent timeline entry structure
  </Card>
  <Card title="Tenant-Aware" icon="building">
    All timeline queries are executed within the proper tenant context
  </Card>
  <Card title="Pagination Support" icon="page">
    Implements page-based pagination over the merged timeline results
  </Card>
</CardGroup>

<Note>
The module uses a **Resolver Pattern** that delegates timeline entry formatting to specialized resolver classes like `AuditTimelineResolver`, `DistributionTimelineResolver`, and `EscalationTimelineResolver`.
</Note>

## Entity Relationships

The Timeline module doesn't define its own entities but aggregates data from multiple sources:

<AccordionGroup>
  <Accordion title="Core Data Sources">
    - **AuditLog**: System audit entries tracking entity changes
    - **Activity**: User-created activities and interactions, including lightweight scheduled task/event index rows
    - **Activity Channel Tables**: `call_log`, `email_log`, `sms_log`, `meeting`, and `site_visit` rows are batch-hydrated for expandable activity details
  </Accordion>

  <Accordion title="Relationship & History Tables">
    - **ContactCompanyRole**: Relationship tracking between contacts and companies
    - **LeadStageHistory / DealStageHistory**: Stage transition history used to attach user-provided stage-change reasons
    - **DistributionLog / EscalationActionLog**: Lead-level system events for automatic assignment and escalation
  </Accordion>

  <Accordion title="Related Entity Events">
    - **EntityTransfer**: Related lead/deal transfer request audit entries are included on the owning entity timeline
    - **UnitAccessRequest**: Deal-linked unit access request audit entries are included on the related deal timeline
    - **LeadPropertyInterest / DealPropertyInterest**: Add/update/remove operations are included as curated audit entries
  </Accordion>
</AccordionGroup>

## API Endpoints

### Get Entity Timeline

Retrieves the timeline for a specific entity (lead, deal, contact, or company).

```http
GET /v1/timeline/{lead|deal|contact|company}/:id
```

#### Path Parameters

| Parameter | Type | Description |
|-----------|------|-------------|
| `entityType` | string | One of: `lead`, `deal`, `contact`, `company` |
| `id` | UUID | ID of the target entity |

#### Query Parameters

| Parameter | Type | Required | Default | Description |
|-----------|------|----------|---------|-------------|
| `source` | `TimelineSource` | No | `all` | Data source filter (`audit`, `activity`, `system`, `all`) |
| `page` | number | No | 1 | Page number for merged timeline results |
| `limit` | number | No | 50 | Number of entries per page (max 200) |
| `from` | ISO8601 | No | - | Filter events from this date |
| `to` | ISO8601 | No | - | Filter events until this date |
| `activityType` | string | No | - | Single or comma-separated activity type filter |

#### Response

```typescript
{
  data: UnifiedTimelineItemDto[];
  page: number;
  limit: number;
  total: number;
  totalPages: number;
  hasMore: boolean;
}
```

#### Timeline Entry Structure

```typescript
interface UnifiedTimelineItemDto {
  id: string;
  source: 'audit' | 'activity' | 'system';
  timestamp: string; // ISO8601
  message: string;
  accentColor?: string; // Optional display accent; stage changes use destination stage color
  reason?: string; // Optional detail text; stage changes use matching stage history notes

  // Audit-only fields
  eventType?: TimelineEventType;
  auditAction?: AuditAction;
  tableName?: string;
  recordId?: string;

  // Activity-only fields
  activityType?: ActivityType;
  referenceId?: string;
  targetEntityType?: ActivityTargetType;
  targetEntityId?: string;
  isPinned?: boolean;
  isImportant?: boolean;
  activityDetails?: {
    type: ActivityType;
    // call: direction/status/outcome/phone/duration/recording/notes/purpose
    // email: direction/status/from/to/cc/bcc/subject/bodyPlain/provider/tracking/threading
    // sms: direction/status/from/to/body/provider/delivery/error
    // meeting: type/status/timing/location/link/outcome/notes/cancellation
    // site_visit: visit type/status/timing/property/feedback/notes/cancellation
  };

  // System-only fields (lead timelines)
  systemActionType?: 'distribution' | 'escalation';

  actor?: {
    id: string;
    name: string;
    email?: string;
    avatarUrl?: string;
  };
}
```

<Tip>
The `activityDetails` field provides expandable channel-specific metadata for activities. Each activity type includes relevant fields like call duration, email subject, SMS body, meeting location, or site visit feedback.
</Tip>

### Task-Scoped Feeds

Tasks expose their own feed endpoints from the CRM task module rather than the generic timeline controller.

<Tabs>
  <Tab title="Combined Timeline">
    ```http
    GET /tasks/:id/timeline
    ```
    Returns the combined task feed (`TaskComment` rows + task `AuditLog` events).
  </Tab>
  
  <Tab title="Activity Only">
    ```http
    GET /tasks/:id/activity
    ```
    Returns only audit-backed task activity events.
  </Tab>
  
  <Tab title="Comments Feed">
    ```http
    GET /tasks/:id/comments/feed
    ```
    Returns only comments in the same keyset-paginated feed shape.
  </Tab>
</Tabs>

<Info>
All three endpoints share the same `TaskTimelineFeedQueryDto` contract with `limit` (default 40, max 100) and optional `before` cursor. They return `TaskTimelineFeedResponse` with ascending items for the fetched page, `hasMoreOlder`, and `nextCursorBefore`.
</Info>

<Warning>
Use `GET /tasks/:id/comments` only when the caller needs the legacy full comment array.
</Warning>

### Commission Payment Timeline

Commission payments use a dedicated history endpoint instead of the generic unified timeline controller.

```http
GET /commission-payments/:id/history
```

<Steps>
  <Step title="Fetch History">
    The commission panel Timeline tab calls this endpoint to retrieve immutable `commission_payment_history` rows ordered oldest-first.
  </Step>
  
  <Step title="Client-Side Sorting">
    The side panel sorts those rows by `createdAt` descending for display so the newest event appears at the top, matching lead/deal panel timelines.
  </Step>
  
  <Step title="Display Notes">
    Notes entered during a status change are persisted on the history row `reason` field and displayed only in this Timeline tab.
  </Step>
</Steps>

<Note>
Free-form commission panel notes use the Activity/Note system (`targetEntityType='commission_payment'`) and are isolated to the panel Notes tab.
</Note>

### Org-Wide Actor Activity Feed

The dashboard **Activity Feed** widget is powered by an org-wide, actor-scoped feed.

```typescript
UnifiedTimelineService.getActivityFeedInTransaction(
  em, 
  orgId, 
  { actorUserIds, cursor, limit }
)
```

#### Feed Characteristics

<CardGroup cols={2}>
  <Card title="Sources Merged" icon="merge">
    Audit lifecycle events from curated tables + logged CRM activity rows (note/call/email/meeting). Distribution/escalation system rows are **not** included.
  </Card>
  
  <Card title="Actor Scope" icon="user">
    `actorUserIds` filters audit rows by `userId` and activity rows by `performedBy`. `null` = no actor filter (org scope).
  </Card>
  
  <Card title="Audit Resolution" icon="magnifying-glass">
    Each audit row is resolved through `AuditTimelineResolver.resolveBatch` into a `kind`, human `message`, and optional `accentColor`.
  </Card>
  
  <Card title="Cursor Pagination" icon="arrows-left-right">
    Each source is independently ordered `(occurredAt DESC, id DESC)` with over-fetching, merging, and global sorting.
  </Card>
</CardGroup>

### TimelineSource Enum

Specifies which data sources to include in timeline queries.

```typescript
enum TimelineSource {
  AUDIT = 'audit',      // Audit log entries only
  ACTIVITY = 'activity', // Activity entries only
  SYSTEM = 'system',     // System events (distribution/escalation)
  ALL = 'all',          // All sources merged
}
```

## Business Rules

<AccordionGroup>
  <Accordion title="Tenant Isolation">
    Timeline entries are strictly filtered by tenant context. Users can only access timelines for entities within their organization.
  </Accordion>

  <Accordion title="Permission-Based Access">
    Users can only view timelines for entities they have access to based on their role and permissions.
  </Accordion>

  <Accordion title="Chronological Ordering">
    Entries are always returned in reverse chronological order (newest first) to show the most recent activity at the top.
  </Accordion>

  <Accordion title="Source Filtering">
    When `source=all`, entries from all available sources are merged and sorted by timestamp:
    
    - **Lead timelines**: Direct lead audit + related transfer audit + related lead-property-interest audit + activity + distribution + escalation entries
    - **Deal timelines**: Direct deal audit + related transfer audit + related unit-access request audit + related deal-property-interest audit + activity entries
    - **Contact/Company timelines**: Audit + activity entries
  </Accordion>

  <Accordion title="Pagination Limits">
    Maximum of 200 entries per request to prevent performance issues. The default limit is 50 entries per page.
  </Accordion>

  <Accordion title="Date Range Filtering">
    When date filters are applied via `from` and `to` parameters, they are inclusive of the boundary dates.
  </Accordion>

  <Accordion title="Stage Accent Color">
    Lead and deal stage-change timeline items include `accentColor` set to the destination stage color. This allows clients to render timeline indicators without parsing stage names or fetching stage metadata.
    
    **Multi-Tenant Resolution**: `AuditTimelineResolver.batchResolveNames` loads the override map per distinct org in the batch (keyed `Map<orgId, Map<stageId, color>>`) and resolves each entry's accent against its own `entry.organizationId`, falling back to the global stage color. This is critical because system-admin audit queries run under RLS bypass and a single page can span multiple tenants.
  </Accordion>

  <Accordion title="Stage Change Reason">
    Lead and deal stage-change timeline items include `reason` when a matching stage history record has notes. Matching uses:
    
    - Entity ID
    - From-stage ID
    - To-stage ID
    - Short timestamp window
    
    This prevents repeated transitions from receiving the wrong reason.
  </Accordion>

  <Accordion title="Lead Disqualification Panel Display">
    Lead disqualification audit messages include the categorical disqualification reason in `message` (e.g., "Lead disqualified: Budget").
    
    The lead panel Timeline tab displays:
    1. `lead.disqualificationNotes` as the visible "Reason:" detail when notes exist
    2. Falls back to the timeline item's `reason` only when no written notes were saved
  </Accordion>

  <Accordion title="Contact Merge Repointing">
    Person duplicate merges can absorb duplicate Contact records. During merge:
    
    - Both `Activity.contactId` and direct Contact-target activities (`targetEntityType='contact'`, `targetEntityId=<absorbedContactId>`) are re-pointed to the surviving Contact
    - This ensures timeline continuity after merge operations
  </Accordion>
</AccordionGroup>

## Implementation Examples

### Fetching a Lead Timeline

```typescript
// Fetch all timeline sources for a lead
const response = await fetch(
  `/v1/timeline/lead/${leadId}?source=all&page=1&limit=50`
);
const timeline = await response.json();

console.log(`Total entries: ${timeline.total}`);
timeline.data.forEach(entry => {
  console.log(`[${entry.timestamp}] ${entry.message}`);
  if (entry.accentColor) {
    console.log(`  Stage color: ${entry.accentColor}`);
  }
  if (entry.reason) {
    console.log(`  Reason: ${entry.reason}`);
  }
});
```

### Filtering by Activity Type

```typescript
// Fetch only call and email activities
const response = await fetch(
  `/v1/timeline/contact/${contactId}?source=activity&activityType=call,email`
);
const activities = await response.json();
```

### Date Range Filtering

```typescript
// Fetch timeline entries from the last 30 days
const thirtyDaysAgo = new Date();
thirtyDaysAgo.setDate(thirtyDaysAgo.getDate() - 30);

const response = await fetch(
  `/v1/timeline/deal/${dealId}?from=${thirtyDaysAgo.toISOString()}`
);
const recentTimeline = await response.json();
```

### Task Timeline with Pagination

```typescript
// Fetch task timeline with cursor pagination
const response = await fetch(
  `/tasks/${taskId}/timeline?limit=40&before=${cursor}`
);
const taskFeed = await response.json();

if (taskFeed.hasMoreOlder) {
  console.log(`Next cursor: ${taskFeed.nextCursorBefore}`);
}
```

## Performance Considerations

<Warning>
**Large Result Sets**: When querying timelines for entities with extensive history, always use pagination. The 200-entry limit per request prevents memory and performance issues.
</Warning>

<Tip>
**Source Filtering**: Use specific source filters (`audit`, `activity`, `system`) instead of `all` when you only need a subset of timeline data. This reduces database load and improves response times.
</Tip>

<Check>
**Cursor Pagination**: For task timelines and activity feeds, cursor-based pagination is more efficient than offset pagination for large datasets as it maintains consistent performance regardless of page depth.
</Check>
