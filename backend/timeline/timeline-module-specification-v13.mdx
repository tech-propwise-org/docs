## Overview

The Timeline module provides a unified view of chronological events across the PropWise CRM system. It aggregates data from multiple sources (audit logs, activities, and lead-level system events from distribution/escalation) to present a comprehensive timeline of events related to specific entities like leads, contacts, and companies.

<Note>
The module serves as a central point for tracking historical changes and activities, enabling users to understand the complete lifecycle and interaction history of CRM entities.
</Note>

## Architecture

The Timeline module follows a unified aggregation pattern, combining data from disparate sources into a single chronological view.

### Key Architectural Decisions

<CardGroup cols={2}>
  <Card title="Source Abstraction" icon="filter">
    Uses a `TimelineSource` enum to specify which data sources to query (audit, activity, or all)
  </Card>
  <Card title="Unified Response Format" icon="merge">
    Normalizes different event types into a consistent timeline entry structure
  </Card>
  <Card title="Tenant-Aware" icon="shield">
    All timeline queries are executed within the proper tenant context
  </Card>
  <Card title="Resolver Pattern" icon="code">
    Delegates timeline entry formatting to specialized resolver classes
  </Card>
</CardGroup>

## Entity Relationships

The Timeline module doesn't define its own entities but aggregates data from:

<AccordionGroup>
  <Accordion title="Core Data Sources">
    - **AuditLog**: System audit entries tracking entity changes
    - **Activity**: User-created activities and interactions
    - **ContactCompanyRole**: Relationship tracking between contacts and companies
  </Accordion>
  
  <Accordion title="Stage History Sources">
    - **LeadStageHistory / DealStageHistory**: Stage transition history used to attach user-provided stage-change reasons to matching audit timeline entries
  </Accordion>
  
  <Accordion title="System Event Sources">
    - **DistributionLog / EscalationActionLog**: Lead-level system events for automatic assignment and escalation activity
  </Accordion>
</AccordionGroup>

## API Endpoints

### Get Entity Timeline

Retrieve the chronological timeline for a specific entity.

```http
GET /v1/timeline/{lead|deal|contact|company}/:id
```

#### Path Parameters

| Parameter | Type | Description |
|-----------|------|-------------|
| `leadId`, `dealId`, `contactId`, or `companyId` | UUID | Identifier of the target entity |

#### Query Parameters

| Parameter | Type | Required | Default | Description |
|-----------|------|----------|---------|-------------|
| `source` | `TimelineSource` | No | `all` | Data source filter (`audit`, `activity`, `system`, `all`) |
| `page` | `number` | No | 1 | Page number for merged timeline results |
| `limit` | `number` | No | 50 | Number of entries per page (max 200) |
| `from` | `ISO8601` | No | - | Filter events from this date |
| `to` | `ISO8601` | No | - | Filter events until this date |
| `activityType` | `string` | No | - | Single or comma-separated activity type filter |

<Warning>
The `limit` parameter has a maximum value of 200 entries per request to prevent performance issues.
</Warning>

## Business Rules

<Steps>
  <Step title="Tenant Isolation">
    Timeline entries are strictly filtered by tenant context to ensure data security
  </Step>
  
  <Step title="Permission-Based Access">
    Users can only view timelines for entities they have access to based on their permissions
  </Step>
  
  <Step title="Chronological Ordering">
    Entries are always returned in reverse chronological order (newest first)
  </Step>
  
  <Step title="Source Filtering">
    When `source=all`, entries from all available sources are merged and sorted by timestamp
  </Step>
</Steps>

### Additional Rules

<Info>
**Stage Accent Color**: Lead and deal stage-change timeline items include `accentColor` set to the destination stage color for enhanced UI rendering.
</Info>

<Info>
**Stage Change Reason**: Timeline items include `reason` when matching stage history records have notes, using entity ID, stage IDs, and timestamp matching.
</Info>

<Tip>
**Contact Merge Repointing**: During person duplicate merges, both `Activity.contactId` and direct Contact-target activities are re-pointed to the surviving Contact to maintain timeline continuity.
</Tip>

## Integration Points

### Audit Module

- Consumes `AuditLog` entities to show system-generated change events
- Uses `AuditTimelineResolver` to format audit entries into timeline format
- Filters out implementation noise including stage tracking fields
- Enriches stage-change entries with destination stage colors and reasons

### CRM Module

- Integrates with `Activity` entities to show user-created activities
- Supports all activity target types (leads, contacts, companies, etc.)
- Leverages `ContactCompanyRole` for relationship context

### Distribution Module

- Lead timelines include `DistributionLog` rows as `source='system'`
- `DistributionTimelineResolver` maps assignment, pool, and fallback outcomes

### Escalation Module

- Lead timelines include `EscalationActionLog` rows with `systemActionType='escalation'`
- `EscalationTimelineResolver` formats notification, redistribution, and other actions
- Use `source=system` to fetch only distribution/escalation entries

### User Module

- Enriches timeline entries with user information (name, email, avatar)
- Ensures proper user references in timeline metadata

### Auth Module

- Enforces JWT authentication on all timeline endpoints
- Validates user permissions for accessed entities

## Service Patterns

### UnifiedTimelineService

<Tabs>
  <Tab title="Core Patterns">
    - **Multi-Source Aggregation**: Queries multiple entity types and merges results
    - **Page-Based Pagination**: Applies pagination after source entries are merged and sorted
    - **Tenant Context Execution**: All database operations execute within tenant boundaries
    - **Error Handling**: Graceful handling of missing entities and invalid queries
  </Tab>
  
  <Tab title="Resolver Pattern">
    - Each source type has its own resolver (e.g., `AuditTimelineResolver`)
    - Resolvers handle source-specific formatting and metadata extraction
    - Maintains consistency in timeline entry structure across different sources
  </Tab>
</Tabs>
