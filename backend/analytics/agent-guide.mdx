description: "Step-by-step playbook for instrumenting analytics in features, covering what to track, why, and how to implement it correctly"
---

A playbook for an AI agent (or engineer) asked to "add analytics" to a feature. It answers three questions in order: **what** to track, **why**, and **how** to wire it. Pair with the [Analytics System](/backend/analytics/system) (the primitives) and [Analytics Event Catalog](/backend/analytics/event-catalog) (what already exists — check it first to avoid duplicates and to match naming).

---

## Mindset: Events serve questions, not the other way around

<Warning>
Do **not** start by listing every button. Start from a business question and work backwards. Every event you add must earn its place in a named funnel or metric. If you can't say which question an event answers, don't add it.
</Warning>

The questions worth instrumenting almost always fall into:

<CardGroup cols={2}>
  <Card title="Activation" icon="rocket">
    Did a new org reach value? (`org_created` → `integration_connected` → first `lead_created`)
  </Card>
  <Card title="Retention / Engagement" icon="repeat">
    Do they come back / use the feature? (`*_viewed`, weekly actives)
  </Card>
  <Card title="Conversion Funnel" icon="filter">
    Where do they drop off? (`opened` → `submitted` → server `completed`)
  </Card>
  <Card title="Revenue / Supply" icon="dollar-sign">
    What drives money? (`unit_added` → `unit_transaction_recorded`)
  </Card>
  <Card title="Friction / Quality" icon="triangle-exclamation">
    What fails or frustrates? (`*_failed`, paywall shown, abandonment)
  </Card>
</CardGroup>

### Anti-patterns

<Warning>
Do **not** track:
- Per-keystroke noise
- Success-only events (capture failure too or the funnel lies)
- Anything already answerable from the DB/git
- Raw values that belong in a bucket
</Warning>

## Decide what to track

Map the flow to events using this process:

<Steps>
  <Step title="Sketch the flow">
    Draw the sequence of states a user/org passes through from start to completion.
  </Step>
  
  <Step title="Identify meaningful transitions">
    For each state transition, ask: **"Would a drop here change a decision?"**
    
    If yes → create an event. If no → skip it.
  </Step>
  
  <Step title="Collapse noise">
    Keep one event per real stage, not per re-render.
    
    - Multi-step wizard → one event per step advance + one `*_abandoned` for drop-off
    - Bulk operation → one summary event with counts, never one event per row
  </Step>
  
  <Step title="Split client vs server truth">
    <Tabs>
      <Tab title="Server (Backend)">
        **Server-authoritative outcomes**
        
        Created, completed, won, counts, status changes
        
        ```typescript
        // Backend owns the truth
        this.dealAnalytics.dealCreated(ctx, deal);
        ```
      </Tab>
      <Tab title="Client (Frontend)">
        **Intent / engagement / drop-off**
        
        Opened, switched, viewed, abandoned (the server never sees an abandon)
        
        ```typescript
        // Frontend captures intent
        captureEvent(FEATURE.DEALS, EVENTS.DEAL_FORM_OPENED, { source });
        ```
      </Tab>
    </Tabs>
  </Step>
  
  <Step title="Validate with funnels">
    Write the funnel(s) the events feed **before** coding. This validates the set is sufficient and minimal.
  </Step>
</Steps>

## Decide dimensions

For each event, list properties that let you **slice** the funnel.

### Allowed dimension types

<AccordionGroup>
  <Accordion title="Enums / Categoricals">
    Status, type, source, role, provider, side (the system-type, never an org-customizable display name).
    
    ```typescript
    { 
      status: 'active',
      provider: 'salesforce',
      integration_type: 'crm'
    }
    ```
  </Accordion>
  
  <Accordion title="Buckets">
    Use `bucketValue` (money), `bucketDays` (duration/tenure), `bucketCount` (counts). **Never the raw figure.**
    
    ```typescript
    {
      deal_value_bucket: '10k-50k',
      days_to_close_bucket: '30-60',
      contacts_count_bucket: '10-25'
    }
    ```
  </Accordion>
  
  <Accordion title="Booleans">
    Simple flags for binary conditions.
    
    ```typescript
    {
      has_company_link: true,
      had_errors: false,
      is_primary: true
    }
    ```
  </Accordion>
  
  <Accordion title="Opaque IDs">
    `deal_id`, `entity_id` as funnel join keys (not content).
    
    ```typescript
    {
      deal_id: 'deal_abc123',
      entity_id: 'entity_xyz789'
    }
    ```
  </Accordion>
  
  <Accordion title="Changed-field tracking">
    Send field **names** (schema), never their values.
    
    ```typescript
    {
      fields_changed: ['status', 'owner', 'amount'],
      fields_changed_count: 3
    }
    ```
  </Accordion>
</AccordionGroup>

### PII hard-exclusions

<Warning>
**Never track:** names, emails, phones, addresses, notes, titles, message text, record contents, raw money, credentials, raw errors, IPs.

When in doubt, bucket it or drop it.
</Warning>

## Decide attribution

Choose the right tracking method based on context:

| Context | Method | distinctId |
|---------|--------|------------|
| User action in a request | `track(...)` | `userId` from CLS |
| System / webhook / cron / **queue worker** | `trackForOrg(..., orgId)` | `org-{orgId}` |
| Frontend (browser) | `captureEvent(feature, event, props)` | ambient posthog-js identity |

<Info>
All methods attach the `organization` group → funnels aggregate by org. If a funnel's conversion unit is an entity (e.g. a deal), aggregate by a shared id instead (HogQL `coalesce(...)`).
</Info>

## Backend implementation

<Steps>
  <Step title="Add feature tag">
    Add to `POSTHOG_FEATURE` in `posthog.constants.ts` (and `FEATURE` in the FE `features.ts` if there's a client side).
    
    ```typescript
    export const POSTHOG_FEATURE = {
      // ... existing features
      CONTACT_IMPORT: 'contact_import',
    } as const;
    ```
  </Step>
  
  <Step title="Define event names">
    Add a `<DOMAIN>_EVENTS` const block in `posthog-events.constants.ts`.
    
    ```typescript
    export const CONTACT_IMPORT_EVENTS = {
      SUBMITTED: 'contact_import_submitted',
      COMPLETED: 'contact_import_completed',
      FAILED: 'contact_import_failed',
    } as const;
    ```
  </Step>
  
  <Step title="Create per-domain service">
    Create `modules/<domain>/<domain>-analytics.service.ts`. Mirror `deal-analytics.service.ts`:
    
    - Constructor injects `ProductAnalyticsService`
    - Private `emit()` wraps `track` in `try/catch`
    - Private `emitForOrg()` wrapping `trackForOrg` if a worker path exists
    - One method per event doing enum mapping + bucketing
    
    ```typescript
    @Injectable()
    export class ContactImportAnalyticsService {
      constructor(
        @Optional() private readonly analytics?: ProductAnalyticsService,
      ) {}
    
      private emit(
        event: string,
        userId: string,
        orgId: string,
        props: Record<string, any>,
      ): void {
        try {
          this.analytics?.track(event, userId, orgId, props);
        } catch (err) {
          // Silent fail - analytics should never break features
        }
      }
    
      contactImportCompleted(
        ctx: RequestContext,
        result: { contactsCreated: number; contactsUpdated: number; errors: number },
      ): void {
        this.emit(
          CONTACT_IMPORT_EVENTS.COMPLETED,
          ctx.userId,
          ctx.orgId,
          {
            contacts_created_bucket: bucketCount(result.contactsCreated),
            contacts_updated_bucket: bucketCount(result.contactsUpdated),
            had_errors: result.errors > 0,
          },
        );
      }
    }
    ```
  </Step>
  
  <Step title="Add test coverage">
    Add a `*.service.spec.ts`:
    
    - Assert bucketed/enum dimensions are present
    - Assert raw values are NOT present
    - Assert it never throws
    
    ```typescript
    describe('ContactImportAnalyticsService', () => {
      it('should bucket contact counts', () => {
        // ... test implementation
      });
      
      it('should never include raw contact counts', () => {
        // ... test implementation
      });
      
      it('should not throw on analytics failure', () => {
        // ... test implementation
      });
    });
    ```
  </Step>
  
  <Step title="Register the service">
    Add the service to its module's `providers` (and `exports` if another module's worker needs it).
    
    ```typescript
    @Module({
      providers: [
        ContactImportService,
        ContactImportAnalyticsService, // Add here
      ],
      exports: [ContactImportAnalyticsService], // If needed by workers
    })
    export class ContactImportModule {}
    ```
  </Step>
  
  <Step title="Wire call sites">
    Capture data inside the txn into an outer `let`, fire **after** commit. Keep it a one-liner.
    
    ```typescript
    async importContacts(ctx: RequestContext, file: File): Promise<void> {
      let result: ImportResult;
      
      await this.db.transaction(async (txn) => {
        result = await this.processImport(txn, file);
      });
      
      // Fire after commit
      this.contactImportAnalytics?.contactImportCompleted(ctx, result);
    }
    ```
  </Step>
  
  <Step title="Worker path (if applicable)">
    Fire `trackForOrg`-backed methods from the queue handler. Read final counts via a service snapshot getter (raw counts never leave the worker — bucket them in the analytics service).
    
    ```typescript
    private emitForOrg(
      event: string,
      orgId: string,
      props: Record<string, any>,
    ): void {
      try {
        this.analytics?.trackForOrg(event, orgId, props);
      } catch (err) {
        // Silent fail
      }
    }
    ```
  </Step>
</Steps>

### Constructor injection gotcha

<Warning>
Adding a constructor param breaks specs differently by construction style. **Check the spec first** (`grep "new <Service>(" ...spec.ts` vs `Test.createTestingModule`).
</Warning>

<Tip>
**Make the new param trailing + optional** (`?`) so positional specs (`new Service(a,b,c)`) keep compiling without the extra arg, and guard every call with `?.`.

**Test-module specs** also need **`@Optional()`** on the param, or DI fails to resolve a non-global provider and the whole suite errors.
</Tip>

Either way, analytics is best-effort, so `?.`-guarded optional is correct semantically too.

## Frontend implementation

<Steps>
  <Step title="Define events">
    Add the events to the right `*_EVENTS` const in `src/lib/analytics/events.ts`, add the `*EventName` type, and union it into `AnalyticsEventName`.
    
    ```typescript
    export const CONTACT_IMPORT_EVENTS = {
      FORM_OPENED: 'contact_import_form_opened',
      FILE_SELECTED: 'contact_import_file_selected',
      SUBMITTED: 'contact_import_submitted',
      ABANDONED: 'contact_import_abandoned',
    } as const;
    
    export type ContactImportEventName = 
      typeof CONTACT_IMPORT_EVENTS[keyof typeof CONTACT_IMPORT_EVENTS];
    
    export type AnalyticsEventName = 
      | DealEventName
      | ContactImportEventName
      | ...;
    ```
  </Step>
  
  <Step title="Fire events at handlers">
    Use `captureEvent(FEATURE.<x>, <EVENTS>.<NAME>, { ...dims })` at the handler/callback.
    
    ```typescript
    const handleSubmit = async (file: File) => {
      captureEvent(
        FEATURE.CONTACT_IMPORT,
        CONTACT_IMPORT_EVENTS.SUBMITTED,
        {
          file_type: file.type,
          file_size_bucket: bucketFileSize(file.size),
        }
      );
      
      await importContacts(file);
    };
    ```
  </Step>
  
  <Step title="Mount-once view events">
    Create an engagement island with a `useRef` guard (`src/components/analytics/*`).
    
    ```typescript
    export const ContactImportViewTracker: React.FC = () => {
      const tracked = useRef(false);
      
      useEffect(() => {
        if (!tracked.current) {
          captureEvent(
            FEATURE.CONTACT_IMPORT,
            CONTACT_IMPORT_EVENTS.FORM_OPENED,
            {}
          );
          tracked.current = true;
        }
      }, []);
      
      return null;
    };
    ```
  </Step>
  
  <Step title="Generic components (if applicable)">
    **Generic/shared components** (used by multiple features) → don't hardcode a feature's events inside them.
    
    Add **optional callback props** (`onFileSelected`, `onAbandoned`, …) and let the feature-specific wrapper supply handlers that call `captureEvent`. Keeps siblings uninstrumented.
    
    ```typescript
    // Generic component
    interface FileUploaderProps {
      onFileSelected?: (file: File) => void;
      onAbandoned?: () => void;
    }
    
    // Feature-specific wrapper
    <FileUploader
      onFileSelected={(file) => {
        captureEvent(FEATURE.CONTACT_IMPORT, EVENTS.FILE_SELECTED, {
          file_type: file.type
        });
      }}
    />
    ```
  </Step>
</Steps>

## Verification checklist

Run these checks for every change:

<Steps>
  <Step title="Backend typecheck">
    ```bash
    npx tsc --noEmit -p tsconfig.json
    ```
    
    Then grep your files. Known pre-existing errors (ignore): `event.service.spec`, `task.service.spec`, `contact.controller.spec`, two `calendar/*e2e`, ai-agent `InboundPersistedEvent`.
  </Step>
  
  <Step title="Run specs">
    ```bash
    npx jest <name>.spec
    ```
    
    Run the touched services' specs — confirm no constructor-arity / DI regressions.
  </Step>
  
  <Step title="Lint">
    Backend:
    ```bash
    npx eslint --fix <files>
    ```
    
    Frontend:
    ```bash
    npx tsc --noEmit
    npx eslint <files>
    ```
  </Step>
  
  <Step title="Update the catalog">
    Add the new events to the [Analytics Event Catalog](/backend/analytics/event-catalog) under their feature with:
    - Fired from (backend/frontend)
    - Attribution method
    - Dimensions
    - Funnel membership
    - Status
  </Step>
</Steps>

## Naming conventions

Follow these rules for consistent event names:

<Check>
**Format:** `snake_case`, present/past as fits: `lead_created`, `contact_import_submitted`
</Check>

<Check>
**Shape:** `<object>_<verb>` or `<feature>_<object>_<verb>` when collision-prone (`contact_import_completed`)
</Check>

<Check>
**FE ↔ BE:** A server-truth outcome and its client-intent counterpart are distinct events (`contact_import_submitted` FE vs `contact_import_completed` BE), joined by the org group
</Check>

<Check>
**Registry alignment:** Keep the FE registry and BE constants names identical where the same logical event exists
</Check>

## Decision flow

Quick reference for the instrumentation process:

```
New flow to instrument
   │
   ├─ What question? (activation / retention / funnel / revenue / friction)
   │     └─ none? → don't track
   │
   ├─ Which state transitions change a decision? 
   │     └─ those are your events (one per stage)
   │
   ├─ Server truth or browser intent?
   │     ├─ server  → ProductAnalyticsService.track / trackForOrg (worker)
   │     └─ browser → captureEvent
   │
   ├─ Dimensions: enums + buckets + ids only
   │     └─ PII? → bucket or drop
   │
   ├─ Wire: feature tag → event const → per-domain service (+spec) 
   │        → register → post-commit call
   │        (trailing-optional + @Optional; worker → trackForOrg + counts snapshot)
   │
   └─ Verify (tsc + specs + lint) → update the catalog → build/extend the PostHog funnel
```

<Tip>
Always start with the business question, not the implementation. If you can't articulate what decision the event informs, don't add it.
</Tip>
