This is the **single source of truth** for every product-analytics event fired at PropWise — backend (`posthog-node`) and frontend (`posthog-js`) — grouped by feature. This defines the contract: event name, where it fires, how it's attributed, its non-PII dimensions, the funnel(s) it feeds, and its status.
</Info>

## Event Implementation

Event names live in:
- **Backend**: `posthog-events.constants.ts`
- **Frontend**: `src/lib/analytics/events.ts`
- **Feature tags**: `posthog.constants.ts` (BE) / `features.ts` (FE)

<Warning>
Keep this catalog in sync when adding or wiring an event.
</Warning>

## Conventions

<Tabs>
  <Tab title="Backend">
    - **track** = `ProductAnalyticsService.track` (user action, `distinctId = userId` from CLS)
    - **trackForOrg** = org-attributed (`org-{id}`, no request: webhook/cron/worker)
  </Tab>
  <Tab title="Frontend">
    - **captureEvent (FE)** = `posthog-js` client event
  </Tab>
</Tabs>

<Note>
All events attach the `organization` group, so funnels aggregate by org unless noted. All magnitudes are bucketed (`bucketValue`/`bucketDays`/`bucketCount`); ids are opaque join keys, never content.
</Note>

## Onboarding (`onboarding`)

Activation — org creation and the setup wizard. The funnel ROOT that every downstream activation funnel (`integration_connected`, first `lead_created`, etc.) hangs off.

<Info>
Backend-only events emitted from `OrganizationController` around org create + the setup-progress endpoints.
</Info>

### Events

| Event | Fired from | Attribution | Dimensions (non-PII) | Status |
|-------|------------|-------------|---------------------|--------|
| `org_created` | `OrganizationController.create` (after the org persists) | BE explicit ids (the new org isn't in the request CLS yet — `distinctId=userId`, `org=newOrgId`) | `plan_tier_at_creation` | ✅ DONE |
| `onboarding_started` | `OrganizationController.create` (same call) | BE explicit ids | — | ✅ DONE |
| `onboarding_step_completed` | `POST setup-progress/advance` when a non-terminal step moves forward | track (user) | `step`, `step_index` | ✅ DONE |
| `onboarding_completed` | `setup-progress/advance` **or** `/complete` on the transition into a terminal state (fire-once) | track (user) | `completion_status`, `days_to_complete_bucket` | ✅ DONE |
| `onboarding_wizard_abandoned` | `create-organization-page-content.tsx` "explore on my own" skip (`handleExploreOwn`) | captureEvent (FE) | `last_step` (the `view.kind` screen enum). Complements `onboarding_completed`(SKIPPED) with WHERE they bailed | ✅ DONE |

### Funnels (org group)

- **Activation (the root):** `org_created` → `onboarding_started` → `onboarding_step_completed`(×N) → `onboarding_completed` → first `integration_connected` / `lead_created`
- **Setup drop-off:** `onboarding_started` with no `onboarding_completed`; where they stall via the `step` distribution

### Implementation Notes

<Warning>
**Unique Attribution:** `org_created` + `onboarding_started` pass EXPLICIT ids (the new org is not yet in the request CLS). `onboarding_step_completed` / `_completed` fire only on a real forward transition (guarded against re-fires).
</Warning>

<Note>
- Setup endpoints require `system.admin`
- NEVER track org name / contact details
- True tab-close abandon isn't reliably catchable
</Note>

- **Adoption:** `ai_assistant_opened` → `ai_assistant_conversation_started` → `ai_assistant_message_sent` → `ai_assistant_response_received`

- **Mode adoption:** `ai_mode` distribution; `ai_assistant_mode_changed` frequency

### Implementation Notes

<Check>
**Server-authoritative metrics:** Duration, tokens, tool calls, TTFT come from backend `ai_assistant_turn_completed`. Frontend events capture interaction timing and UX signals.
</Check>

<Tip>
**Intent classification:** The `intent_group` dimension groups tool usage into high-level categories (research, data entry, analysis, etc.) for product insights without exposing exact prompts.
</Tip>

### Funnels (org)

- **Integration activation:** `integration_connected` → first `integration_sync_completed` → entity created from sync
- **Sync health:** `integration_sync_started` → `integration_sync_completed` (success rate, duration, volume)
- **Error recovery:** `integration_sync_failed` / `integration_disconnected`(auth_failed) → `integration_reconnected`
- **Data flow adoption:** `integration_connected` → `integration_field_mapped` → entity creation rate

### Implementation Notes

<Warning>
**No user context:** All sync/webhook events use `trackForOrg` (background processes). Only connection/config events have user attribution.
</Warning>

<Note>
- Sync events aggregate at the **job level** (not per-record) to avoid volume explosion
- `integration_sync_error` fires for **individual** record failures (sampled if volume is extreme)
- Webhook events track receipt **and** processing separately (receipt = HTTP 200; processing = entity mutation)
- Never log API keys, tokens, or provider-specific record IDs
</Note>

<Tip>
**Error categorization:** `failure_reason` and `error_type` use a fixed taxonomy (auth, rate_limit, network, validation, unknown) for aggregation across providers.
</Tip>

### Implementation Notes

<Warning>
**Explicit attribution:** `user_signed_up` and `org_created` require explicit `distinctId` (the user/org doesn't exist in CLS yet).
</Warning>

<Note>
- Performance events (`page_loaded`) are sampled (not every page load) to avoid volume explosion
- Error events sanitize stack traces and user input (never log request bodies, query params)
- Feature flag events fire on **evaluation** (not every read from cache)
- Import/export events track **initiation** (FE) and **completion** (BE) separately
</Note>

<Tip>
**Route sanitization:** All route/endpoint dimensions use **patterns** (e.g., `/properties/:id` not `/properties/12345`) to enable aggregation while preserving meaningful structure.
</Tip>

## Maintenance & Governance

<Steps>
  <Step title="Keep constants in sync">
    When adding/modifying events, update both the code constants (`posthog-events.constants.ts`, `events.ts`) AND this catalog.
  </Step>
  
  <Step title="Review dimensions quarterly">
    Audit dimension cardinality and usefulness. Deprecate unused dimensions; add new ones via PR.
  </Step>
  
  <Step title="Monitor event volume">
    High-cardinality events (especially FE interaction events) should be sampled or aggregated to avoid cost explosion.
  </Step>
  
  <Step title="Validate in staging">
    Every new event MUST fire correctly in staging before production deployment. Use PostHog's event debugger.
  </Step>
</Steps>

<Warning>
**PII Guardrails:** Never emit user email, phone, address, property address, prompt text, or any other PII. Use opaque IDs (`userId`, `propertyId`) as join keys only.
</Warning>
