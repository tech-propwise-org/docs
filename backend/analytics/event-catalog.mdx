<Note>
This is the **single source of truth** for every product-analytics event fired at PropWise — backend (`posthog-node`) and frontend (`posthog-js`) — grouped by feature.
</Note>

## Overview

This catalog documents the **contract** for each event: event name, where it fires, how it's attributed, its non-PII dimensions, the funnel(s) it feeds, and its implementation status.

### Code Locations

- **Backend events**: `posthog-events.constants.ts`
- **Frontend events**: `src/lib/analytics/events.ts`
- **Feature tags**: `posthog.constants.ts` (BE) / `features.ts` (FE)

<Warning>
Keep this catalog in sync when adding or wiring an event.
</Warning>

### Conventions

<AccordionGroup>
  <Accordion title="Attribution Methods">
    - **track** = `ProductAnalyticsService.track` (user action, `distinctId = userId` from CLS)
    - **trackForOrg** = org-attributed (`org-{id}`, no request: webhook/cron/worker)
    - **captureEvent (FE)** = `posthog-js` client event
  </Accordion>
  
  <Accordion title="Data Standards">
    - All events attach the `organization` group, so funnels aggregate by org unless noted
    - All magnitudes are bucketed (`bucketValue`/`bucketDays`/`bucketCount`)
    - IDs are opaque join keys, never content
    - Non-PII dimensions only
  </Accordion>
</AccordionGroup>

## Onboarding (`onboarding`)

Activation — org creation and the setup wizard. The **funnel ROOT** that every downstream activation funnel (`integration_connected`, first `lead_created`, etc.) hangs off.

<Info>
Backend-only events, emitted from `OrganizationController` around org create and setup-progress endpoints.
</Info>

### Events

| Event | Fired From | Attribution | Dimensions (non-PII) | Status |
|-------|-----------|-------------|---------------------|--------|
| `org_created` | `OrganizationController.create` (after the org persists) | BE explicit ids (the new org isn't in the request CLS yet — `distinctId=userId`, `org=newOrgId`) | `plan_tier_at_creation` | ✅ DONE |
| `onboarding_started` | `OrganizationController.create` (same call) | BE explicit ids | — | ✅ DONE |
| `onboarding_step_completed` | `POST setup-progress/advance` when a non-terminal step moves forward | track (user) | `step`, `step_index` | ✅ DONE |
| `onboarding_completed` | `setup-progress/advance` **or** `/complete` on the transition into a terminal state (fire-once) | track (user) | `completion_status`, `days_to_complete_bucket` | ✅ DONE |
| `onboarding_wizard_abandoned` | `create-organization-page-content.tsx` "explore on my own" skip (`handleExploreOwn`) | captureEvent (FE) | `last_step` (the `view.kind` screen enum) | ✅ DONE |

### Funnels (org group)

<Steps>
  <Step title="Organization Creation">
    `org_created` → `onboarding_started`
  </Step>
  
  <Step title="Setup Progress">
    `onboarding_step_completed` (×N steps)
  </Step>
  
  <Step title="Completion">
    `onboarding_completed` → first `integration_connected` / `lead_created`
  </Step>
</Steps>

**Setup drop-off:** `onboarding_started` with no `onboarding_completed`; where they stall via the `step` distribution

### Implementation Notes

<Note>
**Unique attribution:** `org_created` + `onboarding_started` pass EXPLICIT ids because the new org is not yet in the request CLS.
</Note>

<Check>
- `onboarding_step_completed` / `_completed` fire only on a real forward transition (guarded against re-fires)
- Setup endpoints require `system.admin` role
- NEVER include org name or contact details
</Check>

<Tip>
`onboarding_wizard_abandoned` complements `onboarding_completed(SKIPPED)` with WHERE they bailed. True tab-close abandon isn't reliably catchable.
</Tip>

## Propilot (`propilot`)

Behavioral instrumentation for the Propilot AI assistant (the `/ai` workspace) — adoption, retention, intent, effort mix, drop-off, and quality.

<Info>
This tracks user behavior, NOT AI cost/credit tracking (that stays in `ai_usage_event` / `ai_credit_ledger`).
</Info>

### Events

#### `ai_assistant_turn_completed`

**Fired from:** `AiAssistantOrchestratorService.finalizeOnce()` (1 per turn, every terminal state)

**Attribution:** BE `capture` (`distinctId = userId`, `groups = { organization }`)

**Dimensions:**

<AccordionGroup>
  <Accordion title="Configuration Dimensions">
    - `ai_tier` — lite | standard | max
    - `ai_mode` — propwise | market_regulations
    - `ai_deep_search` — boolean
    - `research_depth` — fast | deep
    - `model` — model identifier
    - `surface` — sheet | workspace
    - `scope_type` — global | entity
  </Accordion>
  
  <Accordion title="Execution Dimensions">
    - `intent_group` — dominant-tool proxy / no_tool / market_no_tool
    - `tool_names` — list of tools used
    - `tool_call_count` — number of tool invocations
    - `tool_error_count` — sanitized bridge tool failures this turn
    - `web_search_call_count` — number of web searches
    - `step_count` — execution steps
  </Accordion>
  
  <Accordion title="Performance Dimensions">
    - `input_tokens` — tokens consumed
    - `output_tokens` — tokens generated
    - `cached_input_tokens` — cached tokens used
    - `duration_ms` — total turn duration
    - `time_to_first_token_ms` — server-authoritative; null when no text streamed
    - `finish_reason` — completion reason
    - `status` — complete | failed | aborted
  </Accordion>
  
  <Accordion title="Context Dimensions">
    - `feature` — feature tag
    - `conversation_id` — conversation identifier
    - `message_id` — message identifier
    - `conversation_turn_index` — turn number in conversation
  </Accordion>
</AccordionGroup>

**Status:** ✅ DONE

## Best Practices

<Steps>
  <Step title="Review the Event Contract">
    Before adding a new event, review the existing catalog to ensure consistency with naming conventions and dimensions.
  </Step>
  
  <Step title="Update Constants">
    Add the event name to the appropriate constants file (`posthog-events.constants.ts` or `events.ts`).
  </Step>
  
  <Step title="Document the Event">
    Update this catalog with the event contract: name, attribution, dimensions, funnel relationships, and status.
  </Step>
  
  <Step title="Test Attribution">
    Verify that the event is properly attributed and includes all required dimensions before deploying.
  </Step>
</Steps>

<Warning>
**Privacy First:** Never include PII (personally identifiable information) in event dimensions. Use opaque IDs as join keys, never content or names.
</Warning>
