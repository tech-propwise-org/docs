The single source of truth for **every** product-analytics event fired at Propwise — backend (`posthog-node`) and frontend (`posthog-js`) — grouped by feature. This is the **contract**: event name, where it fires, how it's attributed, its non-PII dimensions, the funnel(s) it feeds, and its status.

<Info>
  **Related documentation:**
  - [Analytics System Architecture](./analytics-system) — primitives, attribution, buckets, guardrails
  - [Analytics Agent Guide](./analytics-agent-guide) — how to add tracking to a new flow
</Info>

<Note>
  Event names live in `posthog-events.constants.ts` (BE) and `src/lib/analytics/events.ts` (FE); feature tags in `posthog.constants.ts` (BE) / `features.ts` (FE). Keep this catalog in sync when adding or wiring an event.
</Note>

## Conventions

**Tracking methods:**
- **track** = `ProductAnalyticsService.track` (user action, `distinctId = userId` from CLS)
- **trackForOrg** = org-attributed (`org-{id}`, no request: webhook/cron/worker)
- **captureEvent (FE)** = `posthog-js` client event

**Data principles:**
- All events attach the `organization` group, so funnels aggregate by org unless noted
- All magnitudes are bucketed (`bucketValue`/`bucketDays`/`bucketCount`)
- IDs are opaque join keys, never content
- No PII in dimensions

## Onboarding (`onboarding`)

Activation — org creation and the setup wizard. The funnel ROOT that every downstream activation funnel (`integration_connected`, first `lead_created`, etc.) hangs off. Backend-only; emitted from `OrganizationController` around org create + the setup-progress endpoints.

### Events

<AccordionGroup>
  <Accordion title="org_created">
    **Fired from:** `OrganizationController.create` (after the org persists)
    
    **Attribution:** BE explicit ids (the new org isn't in the request CLS yet — `distinctId=userId`, `org=newOrgId`)
    
    **Dimensions:**
    - `plan_tier_at_creation`
    
    **Status:** ✅ DONE
  </Accordion>

  <Accordion title="onboarding_started">
    **Fired from:** `OrganizationController.create` (same call)
    
    **Attribution:** BE explicit ids
    
    **Dimensions:** —
    
    **Status:** ✅ DONE
  </Accordion>

  <Accordion title="onboarding_step_completed">
    **Fired from:** `POST setup-progress/advance` when a non-terminal step moves forward
    
    **Attribution:** track (user)
    
    **Dimensions:**
    - `step`
    - `step_index`
    
    **Status:** ✅ DONE
  </Accordion>

  <Accordion title="onboarding_completed">
    **Fired from:** `setup-progress/advance` **or** `/complete` on the transition into a terminal state (fire-once)
    
    **Attribution:** track (user)
    
    **Dimensions:**
    - `completion_status`
    - `days_to_complete_bucket`
    
    **Status:** ✅ DONE
  </Accordion>

  <Accordion title="onboarding_wizard_abandoned">
    **Fired from:** `create-organization-page-content.tsx` "explore on my own" skip (`handleExploreOwn`)
    
    **Attribution:** captureEvent (FE)
    
    **Dimensions:**
    - `last_step` (the `view.kind` screen enum)
    
    <Note>
      Complements `onboarding_completed`(SKIPPED) with WHERE they bailed; true tab-close abandon isn't reliably catchable
    </Note>
    
    **Status:** ✅ DONE
  </Accordion>
</AccordionGroup>

### Funnels (org group)

<Steps>
  <Step title="Activation (the root)">
    `org_created` → `onboarding_started` → `onboarding_step_completed`(×N) → `onboarding_completed` → first `integration_connected` / `lead_created`
  </Step>
  
  <Step title="Setup drop-off">
    `onboarding_started` with no `onboarding_completed`; where they stall via the `step` distribution
  </Step>
</Steps>

### Implementation notes

<Warning>
  **Unique attribution:** `org_created` + `onboarding_started` pass EXPLICIT ids (the new org is not yet in the request CLS). `onboarding_step_completed` / `_completed` fire only on a real forward transition (guarded against re-fires).
</Warning>

<Info>
  Setup endpoints require `system.admin`. NEVER log org name / contact details.
</Info>

## Propilot (`propilot`)

Behavioral instrumentation for the Propilot AI assistant (the `/ai` workspace) — adoption, retention, intent, effort mix, drop-off, and quality. Not AI cost/credit tracking (that stays in `ai_usage_event` / `ai_credit_ledger`).

### Events

<AccordionGroup>
  <Accordion title="ai_assistant_turn_completed">
    **Fired from:** `AiAssistantOrchestratorService.finalizeOnce()` (1 per turn, every terminal state)
    
    **Attribution:** BE `capture` (`distinctId = userId`, `groups = { organization }`)
    
    **Dimensions:**
    - `ai_tier` (lite|standard|max)
    - `ai_mode` (propwise|market_regulations)
    - `ai_deep_search`
    - `research_depth` (fast|deep)
    - `model`
    - `surface` (sheet|workspace)
    - `scope_type` (global|entity)
    - `intent_group` (dominant-tool proxy / `no_tool` / `market_no_tool`)
    - `tool_names`
    - `tool_call_count`
    - `tool_error_count` (sanitized bridge tool failures this turn)
    - `web_search_call_count`
    - `input_tokens`
    - `output_tokens`
    - `cached_input_tokens`
    - `step_count`
    - `duration_ms`
    - `time_to_first_token_ms` (server-authoritative; null when no text streamed)
    - `finish_reason`
    - `status` (complete|failed|aborted)
    - `feature`
    - `conversation_id`
    - `message_id`
    - `conversation_turn_index`
    
    **Status:** ✅ DONE
  </Accordion>

  <Accordion title="ai_assistant_opened">
    **Fired from:** `src/components/ai-assistant/workspace/*`, `src/lib/propilot-analytics.ts`
    
    **Attribution:** captureEvent (FE)
    
    **Dimensions:** adoption — one per `/ai` mount
    
    **Status:** ✅ DONE
  </Accordion>

  <Accordion title="ai_assistant_conversation_started">
    **Fired from:** FE propilot-analytics
    
    **Attribution:** captureEvent (FE)
    
    **Dimensions:** new thread
    
    **Status:** ✅ DONE
  </Accordion>

  <Accordion title="ai_assistant_message_sent">
    **Fired from:** FE propilot-analytics
    
    **Attribution:** captureEvent (FE)
    
    **Dimensions:**
    - tier/mode/deep_search
    - `is_new_chat`
    - `is_starter_prompt`
    - `mention_count`
    - `message_length`
    
    **Status:** ✅ DONE
  </Accordion>

  <Accordion title="ai_assistant_response_received">
    **Fired from:** FE propilot-analytics
    
    **Attribution:** captureEvent (FE)
    
    **Dimensions:**
    - completion + client-measured `duration_ms` (carries TTFT client-side)
    
    **Status:** ✅ DONE
  </Accordion>
</AccordionGroup>

### Propilot funnels

<Tip>
  **Key metrics:** Adoption (opened), engagement (messages sent/received), intent mix (tool usage patterns), quality (completion rates, errors), and performance (TTFT, duration).
</Tip>

**Analysis patterns:**
- **Adoption:** `ai_assistant_opened` → `ai_assistant_conversation_started` → `ai_assistant_message_sent`
- **Intent distribution:** Aggregate `intent_group` from `ai_assistant_turn_completed`
- **Quality:** `status` = complete vs. failed/aborted, `tool_error_count` distribution
- **Performance:** P50/P90/P99 of `duration_ms` and `time_to_first_token_ms` by `ai_tier` and `research_depth`

## Event maintenance

<Steps>
  <Step title="Add new event">
    1. Define constant in `posthog-events.constants.ts` (BE) or `src/lib/analytics/events.ts` (FE)
    2. Update feature tags in `posthog.constants.ts` (BE) or `features.ts` (FE)
    3. Add entry to this catalog with status
  </Step>
  
  <Step title="Update this catalog">
    Keep this document in sync when adding, modifying, or deprecating events. Include the status (DONE / deferred / Phase N).
  </Step>
  
  <Step title="Review dimensions">
    Ensure all dimensions are non-PII and bucketed appropriately for magnitude values.
  </Step>
</Steps>

<Check>
  All events in this catalog follow the analytics system primitives and attribution rules defined in the [Analytics System Architecture](./analytics-system).
</Check>
