The Analytics module provided comprehensive reporting and analytics capabilities for the PropWise CRM system. It generated insights and metrics for leads and deals, supporting both individual and team-level performance analysis with robust role-based access controls.

## Architecture

The Analytics module followed the standard PropWise module architecture with clear separation of concerns:

<CardGroup cols={2}>
  <Card title="Controller Layer" icon="server">
    `AnalyticsController` handles HTTP requests and applies authentication/authorization guards
  </Card>
  <Card title="Service Layer" icon="gear">
    `AnalyticsService` contains business logic for generating analytics data
  </Card>
  <Card title="DTO Layer" icon="file-code">
    DTOs for request validation and response formatting
  </Card>
  <Card title="Integration Layer" icon="link">
    Connects with CRM entities and RBAC system
  </Card>
</CardGroup>

<Note>
The module leveraged MikroORM for database queries and integrated with the Permission service for access control.
</Note>

### GET /analytics

Retrieved analytics data for leads or deals.

**Authentication:** JWT Bearer token required

<CodeGroup>
  ```bash cURL
  curl -X GET "https://api.propwise.com/analytics?type=leads&teamId=123e4567-e89b-12d3-a456-426614174000" \
  ```


## Business rules

### Access control

<Steps>
  <Step title="Team-based access">
    Users can only view analytics for teams they belong to
  </Step>
  <Step title="Performance filtering">
    The `myPerformance` parameter restricts data to the current user's records
  </Step>
  <Step title="Permission integration">
    Analytics access is governed by the RBAC permission system
  </Step>
</Steps>

### Data filtering

<Warning>
Date ranges must be provided as valid ISO date strings. Invalid date formats will result in a 400 Bad Request error.
</Warning>

- **Date range**: `startDate` and `endDate` filter records within the specified period
- **Team scope**: `teamId` parameter limits results to a specific team's data  
- **User scope**: `myPerformance` parameter restricts to current user's activities

## Distribution & Escalation Analytics

**Last Updated:** 2026-03-11  
**Status:** Active âś…  
**Source modules:** `src/modules/distribution/analytics/` and `src/modules/escalation/analytics/`

The Distribution and Escalation Analytics endpoints expose aggregated metrics and a paginated raw event log from the Lead Auto-Distribution & Escalation System. They are distinct from the general Analytics module (which has been removed) and live in their respective feature modules.

<Warning>
Both analytics services use **raw SQL via `em.execute()`** for performance reasons. This approach is optimized for complex aggregations and append-only analytics reads.
</Warning>

### Key characteristics

Both analytics services use **raw SQL via `em.execute()`** instead of MikroORM QueryBuilder for the following reasons:

- `AVG()` and `COUNT(*) FILTER (WHERE â€¦)` aggregations are awkward to express via the ORM
- Analytics reads are append-only and never touch the IdentityMap, so entity loading overhead is unnecessary
- Consistent style with `EscalationSchedulerService` which also uses `em.execute()` for cross-org scans

All three endpoints share the same 90-day maximum date-range constraint. The constraint is enforced at two layers:

<Steps>
  <Step title="DTO layer validation">
    A custom `@ValidateDateRange()` class-validator decorator rejects requests before they reach the service
  </Step>
  <Step title="Service layer validation">
    `validateDateRange()` throws `BadRequestException` as a defense-in-depth guard for callers that bypass the validation pipe (e.g. internal service-layer calls, tests with manually constructed DTOs)
  </Step>
</Steps>

### Permission requirements

All Distribution & Escalation Analytics endpoints require the **`VIEW_DISTRIBUTION_ANALYTICS`** permission:

<AccordionGroup>
  <Accordion title="Permission details">
    | Permission | Description |
    |---|---|
    | `VIEW_DISTRIBUTION_ANALYTICS` | Read-only access to distribution logs, distribution analytics, and escalation analytics |

    **Default role assignments:**
    - **System Admin**: granted
    - **Owner**: granted  
    - **Manager**: granted
  </Accordion>

  <Accordion title="Implementation">
    Permission checks are applied via:
    ```typescript
    @CheckAccess({ 
      permissions: [OrgPermissionKey.VIEW_DISTRIBUTION_ANALYTICS] 
    })
    ```
    
    Controllers use `AuthGuard` + `AccessGuard` at the class level.
  </Accordion>
</AccordionGroup>

### Date range constraints

<Note>
All three endpoints **require `dateFrom` and `dateTo`** as mandatory ISO 8601 query parameters.
</Note>

**Validation rules:**
- `dateTo >= dateFrom` 
- Range must not exceed **90 days**
- Enforced at both DTO and service layers for defense-in-depth

The 90-day constraint prevents unbounded table scans on `distribution_log`, `escalation_tracker`, and `escalation_action_log` which can hold months of data. Violations return HTTP 400 with a descriptive message.

### Technical implementation

The Distribution & Escalation Analytics use raw SQL via `em.execute()` instead of the MikroORM QueryBuilder for optimal performance:

<AccordionGroup>
  <Accordion title="Raw SQL advantages">
    - **Complex aggregations**: `AVG()` and `COUNT(*) FILTER (WHERE â€¦)` are easier to express in raw SQL
    - **Performance optimization**: No entity loading overhead for append-only analytics reads
    - **Consistency**: Matches the approach used in `EscalationSchedulerService` for cross-org scans
  </Accordion>

  <Accordion title="Aggregation examples">
    The analytics leverage advanced SQL aggregations that are difficult to express via ORM:
    
    ```sql
    SELECT 
      AVG(distribution_time) as avg_distribution_time,
      COUNT(*) FILTER (WHERE status = 'SUCCESS') as success_count,
      COUNT(*) FILTER (WHERE status = 'FAILED') as failure_count
    FROM distribution_log
    WHERE created_at BETWEEN ? AND ?
      AND organization_id = ?
    ```
  </Accordion>

  <Accordion title="Data sources">
    The analytics queries operate on three main tables:
    - **`distribution_log`**: Records all distribution attempts and outcomes
    - **`escalation_tracker`**: Tracks escalation events and timelines  
    - **`escalation_action_log`**: Detailed log of escalation actions taken
  </Accordion>
</AccordionGroup>

<Info>
All three endpoints follow the same permission model (`VIEW_DISTRIBUTION_ANALYTICS`) and 90-day date range constraints. The SQL implementation approach provides optimal performance for analytics workloads while maintaining consistency with the broader system architecture.
</Info>
