<Note>
The Analytics module is **ACTIVE ✅** and powers the `/analytics` page via `/analytics/dashboard` and `/analytics/permissions` endpoints (both gated by `@CheckAccess(ANALYTICS_VIEW)`). It also provides an additive, de-gated, `ResolvedScope`-driven path consumed by the role-aware Dashboard module.
</Note>

The Analytics module provides comprehensive reporting and analytics capabilities for the PropWise CRM system. It generates insights and metrics for leads and deals, supporting both individual and team-level performance analysis with robust role-based access controls. The module includes dashboard reuse functionality and a new scoping system for cross-module integration.

## Dashboard reuse architecture

The Analytics module exposes two composable methods for dashboard integration:

<AccordionGroup>
  <Accordion title="getAnalyticsForScopeInTransaction()">
    **Signature:** `getAnalyticsForScopeInTransaction(em, type: AnalyticsType, scope: ResolvedScope, range?)`
    
    Takes the caller's transactional `EntityManager` (no `TenantContext` wrapping) and a resolved per-domain scope. Returns the same `AnalyticsResponse` shape. The actor filter is derived from `scope.level`:
    
    - **`self`** → `created_by = scope.userId OR EXISTS entity_stakeholder(user_id = scope.userId)`
    - **`team`** → `EXISTS entity_stakeholder es WHERE es.team_id IN (scope.teamIds)` (pool-inclusive)
    - **`org`** → no actor filter (RLS still isolates the tenant)
  </Accordion>

  <Accordion title="applyScopeFilter()">
    **Signature:** `applyScopeFilter(qb, alias, 'lead' | 'deal', scope)`
    
    The reusable scope-filter helper that applies the appropriate filtering logic based on the resolved scope level.
  </Accordion>
</AccordionGroup>

<Info>
The dashboard module invokes `getAnalyticsForScopeInTransaction()` once per `(type, resolvedScope)` pair so each sub-metric can carry a different clamped level.
</Info>

The Analytics module implements a `ResolvedScope` system with three access levels:

## Back-compatibility guarantees

<Warning>
The legacy `getAnalytics(userId, orgId, query)` signature, its `myPerformance` / `teamId` / "must select a team" behavior, and the controller guards are unchanged. The dashboard path is a SEPARATE method - the legacy method is NOT rewired through it.
</Warning>

Both legacy and dashboard paths now filter team scope via `entity_stakeholder.team_id`:
- **Dashboard path**: Uses `es.team_id IN (?)` (Knex expands the id array)
- **Legacy path**: Uses `?teamId=` query with `es.team_id = ?`

## API Endpoints

### GET /analytics

Retrieve analytics data for leads or deals.

**Authentication:** Required (JWT Bearer token)

**Query Parameters:**

- `type` (required): "leads" or "deals"
- `teamId` (optional): UUID of team to filter by
- `myPerformance` (optional): boolean to filter for user's own performance
- `startDate` (optional): ISO date string for period start
- `endDate` (optional): ISO date string for period end

**Response:** `AnalyticsResponse` object containing requested analytics data

**Status Codes:**

- 200: Success
- 401: Unauthorized
- 403: Forbidden (insufficient permissions)
- 400: Bad Request (invalid parameters)

### GET /analytics/dashboard

Retrieve analytics data for the analytics dashboard page.

**Authentication:** Required (JWT Bearer token)
**Authorization:** `@CheckAccess(ANALYTICS_VIEW)`

**Query Parameters:**

- `type` (required): "leads" or "deals"
- `teamId` (optional): UUID of team to filter by
- `myPerformance` (optional): boolean to filter for user's own performance
- `startDate` (optional): ISO date string for period start
- `endDate` (optional): ISO date string for period end

**Response:** `AnalyticsResponse` object containing requested analytics data

**Status Codes:**

- 200: Success
- 401: Unauthorized
- 403: Forbidden (insufficient permissions)
- 400: Bad Request (invalid parameters)

Retrieve user's analytics permissions and available scopes.

**Authentication:** Required (JWT Bearer token)
**Authorization:** `@CheckAccess(ANALYTICS_VIEW)`

**Response:** Permission metadata including accessible teams and scope levels

**Status Codes:**

- 200: Success

## Business Rules

### Access Control

<Steps>
  <Step title="Permission-based access">
    Analytics dashboard endpoints require `ANALYTICS_VIEW` permission
  </Step>
  <Step title="Scope-based filtering">
    Data access is restricted based on the user's resolved scope level (`self`, `team`, `org`)
  </Step>
  <Step title="Team membership">
    Team-scoped access requires membership in the target team(s)
  </Step>
</Steps>

### Data Filtering

1. **Date Range**: When provided, `startDate` and `endDate` filter records within the specified period
2. **Team Scope**: `teamId` parameter limits results to a specific team's data via `entity_stakeholder.team_id`
3. **User Scope**: `myPerformance` parameter restricts to current user's activities
4. **Resolved Scope**: Dashboard integration uses `ResolvedScope` for fine-grained access control

## Distribution & Escalation Analytics

**Last Updated:** 2026-03-11  
**Status:** Active ✅  
**Source modules:** `src/modules/distribution/analytics/` and `src/modules/escalation/analytics/`

<Note>
The Distribution and Escalation Analytics endpoints are distinct from the general Analytics module and live in their respective feature modules. They expose aggregated metrics and a paginated raw event log from the Lead Auto-Distribution & Escalation System.
</Note>

### Architecture characteristics

Both analytics services use **raw SQL via `em.execute()`** instead of MikroORM QueryBuilder for the following reasons:

- `AVG()` and `COUNT(*) FILTER (WHERE …)` aggregations are awkward to express via the ORM
- Analytics reads are append-only and never touch the IdentityMap, so entity loading overhead is unnecessary
- Consistent style with `EscalationSchedulerService` which also uses `em.execute()` for cross-org scans

### 90-Day date range constraints

<Warning>
All endpoints **require `dateFrom` and `dateTo`** as mandatory ISO 8601 query parameters with a maximum 90-day range.
</Warning>

**Validation rules:**
- `dateTo >= dateFrom` 
- Range must not exceed **90 days** (`(dateTo - dateFrom) ≤ 90 days`)
- Enforced at both DTO and service layers for defense-in-depth

The 90-day constraint prevents unbounded table scans on `distribution_log`, `escalation_tracker`, and `escalation_action_log` which can hold months of data.

<Steps>
  <Step title="DTO layer validation">
    A custom `@ValidateDateRange()` class-validator decorator rejects requests before they reach the service
  </Step>
  <Step title="Service layer validation">
    `validateDateRange()` throws `BadRequestException` as a defense-in-depth guard for internal calls
  </Step>
</Steps>

### SQL query methodology

<AccordionGroup>
  <Accordion title="Raw SQL advantages">
    - **Complex aggregations**: `AVG()` and `COUNT(*) FILTER (WHERE …)` are easier to express in raw SQL
    - **Performance optimization**: No entity loading overhead for append-only analytics reads
    - **Consistency**: Matches the approach used in `EscalationSchedulerService` for cross-org scans
  </Accordion>

  <Accordion title="Aggregation examples">
    The analytics leverage advanced SQL aggregations that are difficult to express via ORM:
    
    ```sql
    SELECT 
      AVG(distribution_time) as avg_distribution_time,
      COUNT(*) FILTER (WHERE status = 'SUCCESS') as success_count,
      COUNT(*) FILTER (WHERE status = 'FAILED') as failure_count
    FROM distribution_log
    WHERE created_at BETWEEN ? AND ?
      AND organization_id = ?
    ```
  </Accordion>

  <Accordion title="Data sources">
    The analytics queries operate on three main tables:
    - **`distribution_log`**: Records all distribution attempts and outcomes
    - **`escalation_tracker`**: Tracks escalation events and timelines  
    - **`escalation_action_log`**: Detailed log of escalation actions taken
  </Accordion>
</AccordionGroup>

<Info>
All Distribution & Escalation Analytics endpoints follow the same 90-day date range constraints and use raw SQL for optimal performance. The permission model allows both analytics users and CRM managers to access relevant audit trails.
</Info>
