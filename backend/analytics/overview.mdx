The Analytics module provides comprehensive reporting and analytics capabilities for the PropWise CRM system. It generates insights and metrics for leads and deals, supporting both individual and team-level performance analysis with robust role-based access controls.

## Architecture

The Analytics module follows the standard PropWise module architecture with clear separation of concerns:

<CardGroup cols={2}>
  <Card title="Controller Layer" icon="server">
    `AnalyticsController` handles HTTP requests and applies authentication/authorization guards
  </Card>
  <Card title="Service Layer" icon="gear">
    `AnalyticsService` contains business logic for generating analytics data
  </Card>
  <Card title="DTO Layer" icon="file-code">
    DTOs for request validation and response formatting
  </Card>
  <Card title="Integration Layer" icon="link">
    Connects with CRM entities and RBAC system
  </Card>
</CardGroup>

<Note>
The module leverages MikroORM for database queries and integrates with the Permission service for access control.
</Note>

## Data structures

### Request parameters

<AccordionGroup>
  <Accordion title="GetAnalyticsQueryDto">
    Request DTO for analytics queries:

    | Field | Type | Required | Description |
    |-------|------|----------|-------------|
    | type | AnalyticsType | Yes | Type of analytics (leads or deals) |
    | teamId | string (UUID) | No | Filter by specific team |
    | myPerformance | boolean | No | Filter for current user's performance only |
    | startDate | string (ISO date) | No | Start date for analytics period |
    | endDate | string (ISO date) | No | End date for analytics period |
  </Accordion>

  <Accordion title="AnalyticsType enum">
    ```typescript
    enum AnalyticsType {
      LEADS = 'leads',
      DEALS = 'deals'
    }
    ```
  </Accordion>
</AccordionGroup>

### Response structure

The `AnalyticsResponse` object contains:

| Field | Type | Description |
|-------|------|-------------|
| sourceDistribution | SourceDistribution[] | Breakdown by source |
| stageDistribution | StageDistribution[] | Breakdown by stage |
| teamAnalyticsPermission | TeamAnalyticsPermission | User's permission level |

## API endpoint

### GET /analytics

Retrieve analytics data for leads or deals.

**Authentication:** JWT Bearer token required

<CodeGroup>
  ```bash cURL
  curl -X GET "https://api.propwise.com/analytics?type=leads&teamId=123e4567-e89b-12d3-a456-426614174000" \
    -H "Authorization: Bearer your_jwt_token"
  ```

  ```javascript JavaScript
  const response = await fetch('/analytics?type=deals&myPerformance=true', {
    headers: {
      'Authorization': 'Bearer your_jwt_token'
    }
  });
  const analytics = await response.json();
  ```
</CodeGroup>

#### Query parameters

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `type` | string | Yes | "leads" or "deals" |
| `teamId` | string | No | UUID of team to filter by |
| `myPerformance` | boolean | No | Filter for user's own performance |
| `startDate` | string | No | ISO date string for period start |
| `endDate` | string | No | ISO date string for period end |

#### Response codes

<Tabs>
  <Tab title="Success">
    **200 OK** - Returns `AnalyticsResponse` object with requested analytics data
  </Tab>
  <Tab title="Client Errors">
    - **400 Bad Request** - Invalid parameters
    - **401 Unauthorized** - Missing or invalid authentication
    - **403 Forbidden** - Insufficient permissions
  </Tab>
</Tabs>

## Business rules

### Access control

<Steps>
  <Step title="Team-based access">
    Users can only view analytics for teams they belong to
  </Step>
  <Step title="Performance filtering">
    The `myPerformance` parameter restricts data to the current user's records
  </Step>
  <Step title="Permission integration">
    Analytics access is governed by the RBAC permission system
  </Step>
</Steps>

### Data filtering

<Warning>
Date ranges must be provided as valid ISO date strings. Invalid date formats will result in a 400 Bad Request error.
</Warning>

- **Date range**: `startDate` and `endDate` filter records within the specified period
- **Team scope**: `teamId` parameter limits results to a specific team's data  
- **User scope**: `myPerformance` parameter restricts to current user's activities

## Integration points

### CRM module dependencies

The Analytics module integrates with several CRM entities:

- **Deal Entity**: Source of deal-related analytics data
- **Lead Entity**: Source of lead-related analytics data
- **Activity Entity**: Provides activity-based metrics
- **LeadStage/DealStage Entities**: Used for stage distribution analytics

### RBAC module integration

<Info>
All analytics data is filtered through the RBAC system to ensure users only see authorized information.
</Info>

- **User Entity**: User context for performance filtering
- **Team Entity**: Team-based data filtering and permissions
- **PermissionService**: Authorization for analytics access

## Implementation patterns

### Security patterns

- All endpoints protected by `AuthGuard`
- User context retrieved via `@GetCurrentUser()` decorator
- Permission-based data filtering through service layer

### Query patterns

- Uses MikroORM EntityManager for database operations
- Implements date range filtering for time-based analytics
- Supports both team-wide and individual performance metrics

### Response patterns

<Tip>
Permission metadata is included in all analytics responses to help clients understand the user's access level.
</Tip>

- Structured response objects with typed distributions
- Permission metadata included in responses
- Consistent error handling through NestJS exception filters

### Validation patterns

- Class-validator decorators for input validation
- Custom transform decorators for boolean parameter handling
- Enum validation for analytics type selection
