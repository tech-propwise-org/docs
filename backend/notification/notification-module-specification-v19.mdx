## Overview

The notification module provides a comprehensive system for managing and delivering notifications across multiple channels within the PropWise CRM. It supports in-app and email delivery channels, user preferences management, action-based notifications with handlers, and event-driven notification creation from domain events.

<Note>
The module follows an event-driven architecture where domain modules emit events that are captured by notification listeners, which then create appropriate notifications based on registered notification types and user preferences.
</Note>

### Design Decisions

<AccordionGroup>
  <Accordion title="Multi-Channel Architecture">
    Notifications can be delivered via multiple channels simultaneously based on user preferences, providing flexibility in how users receive information.
  </Accordion>
  
  <Accordion title="Recipient-Centric Design">
    Each user gets their own recipient record with independent status tracking, allowing for personalized notification states.
  </Accordion>
  
  <Accordion title="Action-Driven Workflow">
    Notifications can include actions that trigger business logic when clicked, enabling interactive notifications.
  </Accordion>
  
  <Accordion title="Event-Driven Creation">
    Domain events automatically trigger notification creation through listeners, ensuring loose coupling between modules.
  </Accordion>
  
  <Accordion title="Retry Mechanism">
    Failed deliveries are retried with exponential backoff to handle temporary delivery issues gracefully.
  </Accordion>
  
  <Accordion title="Organization Scoping">
    All notifications are scoped to organizations for multi-tenancy support.
  </Accordion>
</AccordionGroup>

## Entities

### NotificationEvent

Represents the core notification record that can target multiple recipients.

```typescript
{
  id: string;                           // UUID primary key
  type: NotificationType;               // LEAD_ASSIGNED, TEAM_MEMBER_ADDED, etc.
  category: NotificationCategory;       // CRM, REAL_ESTATE, RBAC, MESSAGING
  priority: NotificationPriority;       // LOW, NORMAL, HIGH, URGENT
  title: string;                        // Display title
  message: string;                      // Notification content
  data: Record<string, any>;            // Additional contextual data
  organization: Organization;           // Organization scope
  triggeredBy: User;                    // User who caused the notification
  expiresAt?: Date;                     // Optional expiry date
  createdAt: Date;
  updatedAt: Date;
  recipients: Collection<NotificationRecipient>;
}
```

### NotificationRecipient

Per-user instance of a notification with status and action tracking.

```typescript
{
  id: string;                           // UUID primary key
  notificationEvent: NotificationEvent; // Parent notification
  user: User;                          // Target user
  status: NotificationStatus;          // UNREAD, READ, DISMISSED
  readAt?: Date;                       // When marked as read
  dismissedAt?: Date;                  // When dismissed
  actionsTaken: ActionResult[];        // History of actions performed
  createdAt: Date;
  updatedAt: Date;
  deliveries: Collection<NotificationDelivery>;
}
```

### NotificationDelivery

Tracks delivery attempts per channel with retry logic.

```typescript
{
  id: string;                           // UUID primary key
  notificationRecipient: NotificationRecipient;
  channel: Channel;                     // EMAIL, IN_APP
  status: DeliveryStatus;              // PENDING, SENT, FAILED, RETRYING
  sentAt?: Date;                       // Successful delivery timestamp
  failedAt?: Date;                     // Last failure timestamp
  error?: string;                      // Last error message
  retryCount: number;                  // Number of retry attempts
  nextRetryAt?: Date;                  // Next retry schedule
  createdAt: Date;
  updatedAt: Date;
}
```

### NotificationPreference

User preferences for notification types and channels per organization.

```typescript
{
  id: string;                           // UUID primary key
  user: User;                          // Target user
  organization: Organization;           // Organization scope
  muteAll: boolean;                    // Global mute flag
  channelPreferences: Record<Channel, boolean>; // Per-channel preferences
  typePreferences: Record<NotificationType, {
    enabled: boolean;
    channels: Channel[];
  }>;                                  // Per-type channel preferences
  createdAt: Date;
  updatedAt: Date;
}
```

## Business Rules

### Notification Creation

<Steps>
  <Step title="Access Validation">
    Notifications are created only for users with appropriate access to the related entity
  </Step>
  
  <Step title="Preference Check">
    User preferences are checked before creating notifications
  </Step>
  
  <Step title="Mute All Check">
    If `muteAll` is enabled, no notifications are created
  </Step>
  
  <Step title="Channel Selection">
    Only enabled channels from preferences are used for delivery
  </Step>
</Steps>

### Delivery Logic

<Warning>
Failed deliveries are retried with exponential backoff: 1min, 5min, 30min, 2hr, 12hr. After 5 failed attempts, delivery is marked as permanently failed.
</Warning>

1. **In-app notifications** are always created regardless of delivery success
2. **Email deliveries** are attempted based on user preferences
3. **Retry mechanism** handles temporary failures gracefully
4. **Permanent failures** are recorded after retry limit is reached

### Action Handling

<Info>
Actions can only be taken on notifications in UNREAD or READ status. Taking an action may change notification status to read.
</Info>

- Action handlers can return redirect URLs for frontend navigation
- Action results are stored in the recipient's `actionsTaken` array
- Multiple actions can be taken on the same notification

### Expiration

<Check>
Expired notifications cannot have actions taken on them. A cleanup job removes expired notifications and their deliveries automatically.
</Check>

- Action handlers receive `onExpire` callback for cleanup logic
- Expiration prevents stale actions from being executed

## Patterns and Conventions

### Notification Types

Notification types are centrally managed and configured:

```typescript
// Enumerated in NotificationType enum
export enum NotificationType {
  LEAD_ASSIGNED = 'LEAD_ASSIGNED',
  TEAM_MEMBER_ADDED = 'TEAM_MEMBER_ADDED',
  // ... other types
}

// Registered in NotificationTypeRegistry with configuration
@Injectable()
export class NotificationTypeRegistry {
  // Type configuration with channels, actions, and handlers
}
```

### Channel Implementation

<Tip>
All channels implement the `NotificationChannel` interface, making it easy to add new delivery mechanisms like SMS or push notifications.
</Tip>

- Channels are auto-discovered and registered at startup
- Support for easy addition of new channels (SMS, Push, etc.)
- Consistent interface across all delivery mechanisms

### Action Handlers

Action handlers follow a consistent pattern for processing notification actions:

```typescript
@NotificationActionHandler({ name: 'approve-request' })
export class ApprovalHandler extends BaseNotificationHandler {
  async onCreate(notification: NotificationEvent): Promise<void> {
    // Setup logic when notification is created
  }

  async onAction(
    recipient: NotificationRecipient,
    actionKey: string,
    data: any
  ): Promise<ActionResult> {
    // Process the action
  }

  async onExpire(notification: NotificationEvent): Promise<void> {
    // Cleanup logic when notification expires
  }
}
```

### Event Naming Convention

Events follow a consistent naming pattern for easy identification:

- **Domain-prefixed**: `crm.lead-assigned`, `rbac.team-member-added`
- **Kebab-case** for consistency
- **Centralized** in domain-specific event files

### Error Handling

<Warning>
The notification system is designed to be resilient. Delivery failures are logged but don't break the notification flow, and handler errors are caught and logged with context.
</Warning>

- Delivery failures are logged but don't break the notification flow
- Invalid action attempts return appropriate error responses
- Handler errors are caught and logged with context
- Graceful degradation ensures system stability
