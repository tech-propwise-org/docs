The notification module provides a comprehensive system for managing and delivering notifications across multiple channels within the PropWise CRM. It supports in-app and email delivery channels, user preferences management, action-based notifications with handlers, and event-driven notification creation from domain events.

<Note>
Last Updated: 2024-12-19  
Status: Active
</Note>

## Architecture Overview

The module follows an event-driven architecture where domain modules emit events that are captured by notification listeners, which then create appropriate notifications based on registered notification types and user preferences.

## Entity Specifications

### NotificationEvent

Represents the core notification record that can target multiple recipients.

```typescript
{
  id: string;                           // UUID primary key
  type: NotificationType;               // LEAD_ASSIGNED, TEAM_MEMBER_ADDED, etc.
  category: NotificationCategory;       // CRM, REAL_ESTATE, RBAC, MESSAGING
  priority: NotificationPriority;       // LOW, NORMAL, HIGH, URGENT
  title: string;                        // Display title
  message: string;                      // Notification content
  data: Record<string, any>;            // Additional contextual data
  organization: Organization;           // Organization scope
  triggeredBy: User;                    // User who caused the notification
  expiresAt?: Date;                     // Optional expiry date
  createdAt: Date;
  updatedAt: Date;
  recipients: Collection<NotificationRecipient>;
}
```

### NotificationRecipient

Per-user instance of a notification with status and action tracking.

```typescript
{
  id: string;                           // UUID primary key
  notificationEvent: NotificationEvent; // Parent notification
  user: User;                          // Target user
  status: NotificationStatus;          // UNREAD, READ, DISMISSED
  readAt?: Date;                       // When marked as read
  dismissedAt?: Date;                  // When dismissed
  actionsTaken: ActionResult[];        // History of actions performed
  createdAt: Date;
  updatedAt: Date;
  deliveries: Collection<NotificationDelivery>;
}
```

### NotificationDelivery

Tracks delivery attempts per channel with retry logic.

```typescript
{
  id: string;                           // UUID primary key
  notificationRecipient: NotificationRecipient;
  channel: Channel;                     // EMAIL, IN_APP
  status: DeliveryStatus;              // PENDING, SENT, FAILED, RETRYING
  sentAt?: Date;                       // Successful delivery timestamp
  failedAt?: Date;                     // Last failure timestamp
  error?: string;                      // Last error message
  retryCount: number;                  // Number of retry attempts
  nextRetryAt?: Date;                  // Next retry schedule
  createdAt: Date;
  updatedAt: Date;
}
```

### NotificationPreference

User preferences for notification types and channels per organization.

```typescript
{
  id: string;                           // UUID primary key
  user: User;                          // Target user
  organization: Organization;           // Organization scope
  muteAll: boolean;                    // Global mute flag
  channelPreferences: Record<Channel, boolean>; // Per-channel preferences
  typePreferences: Record<NotificationType, {
    enabled: boolean;
    channels: Channel[];
  }>;                                  // Per-type channel preferences
  createdAt: Date;
  updatedAt: Date;
}
```

## API Reference

### Get Notifications

Retrieve paginated notifications for the authenticated user.

```http
GET /notifications?page=1&limit=20&status=UNREAD
```

**Response:**
```json
{
  "items": [...],
  "total": 45,
  "page": 1,
  "limit": 20
}
```

### Mark as Read

Mark a specific notification as read.

```http
PATCH /notifications/:id/read
```

**Response:**
```json
{
  "success": true,
  "redirectUrl": "/crm/leads/123"
}
```

### Dismiss Notification

Remove a notification from the user's feed.

```http
DELETE /notifications/:id
```

**Response:**
```json
{
  "success": true
}
```

### Take Action

Execute an action associated with a notification.

```http
POST /notifications/:id/actions/:actionKey
```

**Request Body:**
```json
{
  "data": {
    "customField": "value"
  }
}
```

**Response:**
```json
{
  "success": true,
  "message": "Action completed successfully",
  "redirectUrl": "/crm/leads/123",
  "result": {...}
}
```

## Business Rules

### Notification Creation

<Steps>
  <Step title="Access Control Check">
    Notifications are created only for users with appropriate access to the related entity
  </Step>
  <Step title="User Preferences Validation">
    User preferences are checked before creating notifications
  </Step>
  <Step title="Mute All Check">
    If `muteAll` is enabled, no notifications are created
  </Step>
  <Step title="Channel Selection">
    Only enabled channels from preferences are used for delivery
  </Step>
</Steps>

### Delivery Logic

<Warning>
In-app notifications are always created regardless of delivery success, but email deliveries depend on user preferences.
</Warning>

1. **In-app notifications**: Always created for enabled notification types
2. **Email deliveries**: Attempted based on user channel preferences
3. **Retry mechanism**: Failed deliveries use exponential backoff (1min, 5min, 30min, 2hr, 12hr)
4. **Permanent failure**: After 5 failed attempts, delivery is marked as permanently failed

### Action Handling

<Info>
Actions provide interactive functionality within notifications, allowing users to take immediate action without navigating away.
</Info>

- Actions can only be taken on notifications in `UNREAD` or `READ` status
- Taking an action may automatically change notification status to read
- Action handlers can return redirect URLs for frontend navigation
- Action results are stored in the recipient's `actionsTaken` array

### Expiration Management

<Check>
Expired notifications are automatically cleaned up through scheduled jobs to maintain system performance.
</Check>

- Expired notifications cannot have actions taken
- Cleanup job removes expired notifications and their deliveries
- Action handlers receive `onExpire` callback for cleanup logic

## Patterns and Conventions

### Notification Types

All notification types are enumerated and registered with specific configurations:

```typescript
// Notification type registration
@NotificationType({
  name: NotificationType.LEAD_ASSIGNED,
  category: NotificationCategory.CRM,
  defaultChannels: [Channel.EMAIL, Channel.IN_APP],
  actions: ['view_lead', 'contact_lead']
})
```

### Channel Implementation

<Tip>
New channels can be easily added by implementing the `NotificationChannel` interface and following the auto-discovery pattern.
</Tip>

All channels implement the `NotificationChannel` interface:
- Auto-discovered and registered at startup
- Support for easy addition of new channels (SMS, Push, etc.)
- Consistent error handling and retry logic

### Action Handlers

Action handlers provide business logic for notification interactions:

```typescript
@NotificationActionHandler({ name: 'view_lead' })
export class ViewLeadHandler extends BaseNotificationHandler {
  async onAction(notification: NotificationRecipient, data?: any) {
    // Business logic here
    return {
      success: true,
      redirectUrl: `/crm/leads/${notification.data.leadId}`
    };
  }
}
```

### Event Naming Convention

Events follow a consistent naming pattern:
- **Domain-prefixed**: `crm.lead-assigned`, `rbac.team-member-added`
- **Kebab-case**: For consistency across the system
- **Centralized**: Defined in domain-specific event files

### Error Handling Strategy

<Warning>
Robust error handling ensures that notification failures don't impact the core business processes.
</Warning>

- **Delivery failures**: Logged but don't break the notification flow
- **Invalid actions**: Return appropriate error responses
- **Handler errors**: Caught and logged with full context for debugging

The notification module provides a flexible, scalable foundation for all notification needs within the PropWise CRM system, supporting current requirements while enabling future enhancements through its modular architecture.
