## Overview

The notification module provides a comprehensive system for managing and delivering notifications across multiple channels within the PropWise CRM. It supports in-app and email delivery channels, user preferences management, action-based notifications with handlers, and event-driven notification creation from domain events.

The module follows an event-driven architecture where domain modules emit events that are captured by notification listeners, which then create appropriate notifications based on registered notification types and user preferences.

## Architecture

### Core components

The notification system is built around several interconnected components:

- **Notification Events**: Central notification records that can have multiple recipients
- **Notification Recipients**: Per-user instances of notifications with status tracking
- **Notification Deliveries**: Per-channel delivery attempts with retry logic
- **Notification Channels**: Pluggable delivery mechanisms (email, in-app)
- **Action Handlers**: Business logic processors for notification actions
- **Preference System**: User-configurable notification settings per organization

### Design decisions

<AccordionGroup>
  <Accordion title="Multi-Channel Architecture">
    Notifications can be delivered via multiple channels simultaneously based on user preferences. This allows users to receive the same notification through both in-app and email channels if configured.
  </Accordion>

  <Accordion title="Recipient-Centric Design">
    Each user gets their own recipient record with independent status tracking. This enables per-user read/dismissed states and action tracking without affecting other recipients.
  </Accordion>

  <Accordion title="Action-Driven Workflow">
    Notifications can include actions that trigger business logic when clicked. Actions are handled by registered action handlers that execute specific workflows.
  </Accordion>

  <Accordion title="Event-Driven Creation">
    Domain events automatically trigger notification creation through listeners. This decouples notification logic from core business logic.
  </Accordion>

  <Accordion title="Retry Mechanism">
    Failed deliveries are retried with exponential backoff. This ensures transient failures don't result in lost notifications.
  </Accordion>

  <Accordion title="Organization Scoping">
    All notifications are scoped to organizations for multi-tenancy. This ensures proper data isolation and access control.
  </Accordion>

  <Accordion title="Generic Grouping/Deduplication">
    Any notification type can opt into windowed aggregation (one digest notification + one digest email per window) or collapse/throttle, configured declaratively per type. See [Grouping & Deduplication](#grouping--deduplication) for details.
  </Accordion>
</AccordionGroup>

## Entities

### NotificationEvent

Represents the core notification record that can target multiple recipients.

```typescript
{
  id: string;                           // UUID primary key
  type: NotificationType;               // LEAD_ASSIGNED, TEAM_MEMBER_ADDED, etc.
  category: NotificationCategory;       // crm, properties, access, document, system, messaging
  priority: NotificationPriority;       // LOW, NORMAL, HIGH, URGENT
  title: string;                        // Display title
  message: string;                      // Notification content
  data: Record<string, any>;            // Additional contextual data
  organization: Organization;           // Organization scope
  triggeredBy: User;                    // User who caused the notification
  expiresAt?: Date;                     // Optional expiry date
  createdAt: Date;
  updatedAt: Date;
  recipients: Collection<NotificationRecipient>;
}
```

<Info>
The `type` field determines which notification type configuration is used, including delivery channels, action handlers, and grouping behavior.
</Info>

### NotificationRecipient

Per-user instance of a notification with status and action tracking.

```typescript
{
  id: string;                           // UUID primary key
  notificationEvent: NotificationEvent; // Parent notification
  user: User;                          // Target user
  status: NotificationStatus;          // UNREAD, READ, DISMISSED
  readAt?: Date;                       // When marked as read
  dismissedAt?: Date;                  // When dismissed
  actionsTaken: ActionResult[];        // History of actions performed
  createdAt: Date;
  updatedAt: Date;
  deliveries: Collection<NotificationDelivery>;
}
```

<Tip>
Each recipient maintains independent status tracking, allowing different users to mark the same notification as read or dismissed independently.
</Tip>

### NotificationDelivery

Tracks delivery attempts per channel with retry logic.

```typescript
{
  id: string;                           // UUID primary key
  notificationRecipient: NotificationRecipient;
  channel: Channel;                     // EMAIL, IN_APP
  status: DeliveryStatus;              // PENDING, SENT, FAILED, RETRYING
  sentAt?: Date;                       // Successful delivery timestamp
  failedAt?: Date;                     // Last failure timestamp
  error?: string;                      // Last error message
  retryCount: number;                  // Number of retry attempts
  nextRetryAt?: Date;                  // Next retry schedule
  createdAt: Date;
  updatedAt: Date;
}
```

<Warning>
Failed deliveries are automatically retried with exponential backoff. Monitor the `retryCount` and `error` fields to identify persistent delivery issues.
</Warning>

### NotificationPreference

User preferences for notification types and channels per organization.

```typescript
{
  id: string;                          // UUID primary key
  user: User;                          // Target user
  organization: Organization;          // Organization scope
  channelPreferences: Record<Channel, boolean>; // Per-channel preferences
  typePreferences: Record<
    NotificationType,
    {
      enabled: boolean;
      channels: Channel[];
    }
  >;                                   // Per-type channel preferences
  createdAt: Date;
  updatedAt: Date;
}
```

## AI-related notification types

The notification system includes specialized types for AI agent operations and messaging events:

| Type | Category | Description |
|------|----------|-------------|
| `AI_CREDIT_LOW` | AI | Organization AI credits are running low |
| `AI_CREDIT_EXHAUSTED` | AI | Organization has exhausted AI credits |
| `CONVERSATION_TRANSFERRED` | MESSAGING | Conversation assigned to different user/team |
| `AI_AGENT_FAILED` | AI | AI agent execution failed with error |
| `AI_AGENT_BUDGET_EXCEEDED` | AI | AI agent exceeded token budget limits |
| `AI_AGENT_ACTIVATED` | AI | AI agent has been activated |
| `AI_AGENT_DEACTIVATED` | AI | AI agent has been deactivated |

<Note>
These notifications are triggered by AI agent actions and messaging events, following the same event-driven pattern as CRM notifications. The AI module integrates with the notification system for credit alerts, agent status changes, and execution failures.
</Note>

## Grouping & deduplication

A generic engine lets **any** notification type collapse a burst of related notifications into a single aggregated in-app notification + a single digest email, instead of N separate pings. It is configured declaratively per type via a `grouping` block on the type's registry config — no bespoke per-type code is required.

<Info>
**Engine status:** Built. The legacy ad-hoc dedup implementations listed in `NOTIFICATION_CATALOG.md` are scheduled to migrate onto it.
</Info>

### How it works

<Steps>
  <Step title="Detect">
    When a grouped type is created, a deterministic **group key** is computed from `type + org-scope + recipientId + configured payload fields`. The field segment is **JSON-encoded** (a JSON array of the stringified field values), not a raw `|` join, so a payload value containing a `:` / `|` delimiter cannot collide across different field splits. Notifications sharing a key for a recipient land in the same window.
  </Step>

  <Step title="Collect">
    The first notification opens a `NotificationGroup` window (and inserts a `NotificationGroupItem`); subsequent matching notifications append items. The user sees nothing yet (`hold` mode) — or the first one is delivered immediately and the rest fold in (`emit-first-immediately` mode).
  </Step>

  <Step title="Flush">
    A pg-boss job fires when the window closes. The worker aggregates the collected items, renders one aggregated notification, and creates it through the **normal** create path (so preferences/mute, channels, and delivery are handled identically to any other notification).
  </Step>
</Steps>

### Window semantics

The grouping engine supports multiple window strategies:

<Tabs>
  <Tab title="Fixed (Tumbling)">
    Opens at the first item, flushes a fixed duration later. Provides predictable latency — used by the `activity_*` pilot (15-minute digest).
    
    ```typescript
    {
      windowKind: 'FIXED',
      windowMs: 900000  // 15 minutes
    }
    ```
  </Tab>

  <Tab title="Sliding (Debounce)">
    Each new item pushes the flush time forward, hard-capped by a max duration. Best for burst suppression.
    
    ```typescript
    {
      windowKind: 'SLIDING',
      windowMs: 300000,      // 5 minutes
      maxWindowMs: 3600000   // 1 hour hard cap
    }
    ```
  </Tab>
</Tabs>

#### Additional window features

<CardGroup cols={2}>
  <Card title="Early Flush" icon="forward-fast">
    An optional `maxGroupSize` flushes immediately once reached, so a flood never produces an unbounded digest. On early flush the window's `flushAt` is pulled back to **now** (matching the immediate job that is enqueued).
  </Card>

  <Card title="Window Clamp" icon="clamp">
    A resolved window duration is clamped to `[MIN_GROUP_WINDOW_MS, MAX_GROUP_WINDOW_MS]` (1 ms … 7 days) before it is snapshotted, so an oversized value can never overflow the `int4` `window_ms` column.
  </Card>
</CardGroup>

### Modes

The grouping engine supports two aggregation modes:

<Tabs>
  <Tab title="Digest Mode">
    Render an itemized list ("7 activity updates"). The `payload.items` field carries each item for the in-app list + digest email, each with its own deep-link.
    
    ```typescript
    {
      mode: 'digest',
      collectItems: true,
      renderTemplate: 'digest-list'
    }
    ```
  </Tab>

  <Tab title="Collapse Mode">
    Render a single representative + count ("Seat limit reached (+3 more)"). No item list needed.
    
    ```typescript
    {
      mode: 'collapse',
      collectItems: false,
      renderTemplate: 'single-with-count'
    }
    ```
  </Tab>
</Tabs>

### Entities (grouping)

#### NotificationGroup

The open aggregation-window state — one row per `(recipient, type, groupKey, org-scope)` while live.

```typescript
{
  id: string;
  organization?: Organization;       // NULL reserved for tenantless grouping
  recipient: User;                   // windows are always per-recipient
  type: NotificationType;
  groupKey: string;                  // detection discriminator (JSON-encoded field segment)
  status: NotificationGroupStatus;   // OPEN | FLUSHED | CANCELLED (FLUSHING reserved)
  windowKind: GroupWindowKind;       // FIXED | SLIDING (snapshotted at open)
  openedAt: Date;
  flushAt: Date;                     // next scheduled flush (advances for SLIDING)
  maxAt: Date;                       // hard ceiling for SLIDING
  itemCount: number;                 // running count (drives maxGroupSize + collapse)
  scheduledJobId?: string;           // pg-boss flush job id
  firstEventId?: string;             // emit-first-immediately: event updated in place at flush
  flushedEventId?: string;           // set on flush — double-flush idempotency guard
}
```

#### NotificationGroupItem

Insert-only collected item inside a window (skipped for count-only collapse).

```typescript
{
  id: string;
  group: NotificationGroup;          // FK, cascade delete
  title?: string;
  message?: string;
  payload: Record<string, any>;      // per-item snapshot for rendering + per-item deep-links
  senderId?: string;
  priority?: NotificationPriority;   // flush picks the highest as the grouped priority
}
```

### Concurrency & correctness

<Warning>
The grouping engine implements multiple safeguards to ensure correctness under high concurrency:
</Warning>

<AccordionGroup>
  <Accordion title="Window Uniqueness">
    A **partial unique index** over `(recipient, type, groupKey, COALESCE(org, sentinel)) WHERE status = 'open'` enforces "one live window per scope". Concurrent openers race-recover via the same optimistic-insert + unique-conflict pattern as `idempotencyKey`.
  </Accordion>

  <Accordion title="Insert-Only Collection">
    Item collection is an **insert-only child table** (not a JSONB array) so high-fan-in windows don't serialize on a single row lock.
  </Accordion>

  <Accordion title="Double-Flush Prevention">
    **Double-flush is impossible** through multiple mechanisms:
    - Per-window pg-boss `singletonKey` (policy `stately`)
    - `PESSIMISTIC_WRITE` lock at flush
    - Terminal-status guard
    - Persisted `flushedEventId`
  </Accordion>

  <Accordion title="Recovery Mechanism">
    A **recovery cron** (`notification-group-recovery`, every 10 min) re-enqueues windows whose flush job was lost or is overdue. The orphan scan re-picks an OPEN window only when:
    - `scheduled_job_id IS NULL`, **or**
    - `flush_at < NOW() - INTERVAL '2 minutes'`
    
    Every flush path must keep `flush_at` aligned with the actually-enqueued job time — including **early flush**, which sets `flush_at = now`. This prevents a lost early-flush job from hiding behind a still-future `flush_at` and stalling the digest.
    
    The orphan scan filters `organization_id IS NOT NULL` because the engine is **org-scoped** — `append()` always opens windows with a real org and recovery re-enqueues via `executeInOrg(orgId)`. Tenantless grouping is unsupported, so no tenantless window is ever stranded.
  </Accordion>

  <Accordion title="Abandoned Window Guard">
    If the org (or recipient) is deleted between a window opening and its flush, `createNotificationInTransaction` throws `NotFoundException` at flush. The flush worker catches it and marks the window as `CANCELLED`.
  </Accordion>
</AccordionGroup>
