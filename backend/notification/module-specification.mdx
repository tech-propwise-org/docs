## Overview

The notification module provides a comprehensive system for managing and delivering notifications across multiple channels within the PropWise CRM. It supports in-app and email delivery channels, user preferences management, action-based notifications with handlers, and event-driven notification creation from domain events.

<Info>
The module follows an event-driven architecture where domain modules emit events that are captured by notification listeners, which then create appropriate notifications based on registered notification types and user preferences.
</Info>

**Last Updated:** 2024-12-19  
**Status:** Active

### Design Decisions

<AccordionGroup>
  <Accordion title="Multi-Channel Architecture">
    Notifications can be delivered via multiple channels simultaneously based on user preferences, allowing for flexible delivery strategies.
  </Accordion>
  
  <Accordion title="Recipient-Centric Design">
    Each user gets their own recipient record with independent status tracking, enabling personalized notification experiences.
  </Accordion>
  
  <Accordion title="Action-Driven Workflow">
    Notifications can include actions that trigger business logic when clicked, creating interactive notification experiences.
  </Accordion>
  
  <Accordion title="Event-Driven Creation">
    Domain events automatically trigger notification creation through listeners, ensuring loose coupling between modules.
  </Accordion>
  
  <Accordion title="Retry Mechanism">
    Failed deliveries are retried with exponential backoff to handle temporary delivery issues gracefully.
  </Accordion>
  
  <Accordion title="Organization Scoping">
    All notifications are scoped to organizations for multi-tenancy isolation and security.
  </Accordion>
</AccordionGroup>

## Entities

### NotificationEvent

Represents the core notification record that can target multiple recipients.

```typescript
{
  id: string;                           // UUID primary key
  type: NotificationType;               // LEAD_ASSIGNED, TEAM_MEMBER_ADDED, etc.
  category: NotificationCategory;       // CRM, REAL_ESTATE, RBAC, MESSAGING
  priority: NotificationPriority;       // LOW, NORMAL, HIGH, URGENT
  title: string;                        // Display title
  message: string;                      // Notification content
  data: Record<string, any>;            // Additional contextual data
  organization: Organization;           // Organization scope
  triggeredBy: User;                    // User who caused the notification
  expiresAt?: Date;                     // Optional expiry date
  createdAt: Date;
  updatedAt: Date;
  recipients: Collection<NotificationRecipient>;
}
```

### NotificationRecipient

Per-user instance of a notification with status and action tracking.

```typescript
{
  id: string;                           // UUID primary key
  notificationEvent: NotificationEvent; // Parent notification
  user: User;                          // Target user
  status: NotificationStatus;          // UNREAD, READ, DISMISSED
  readAt?: Date;                       // When marked as read
  dismissedAt?: Date;                  // When dismissed
  actionsTaken: ActionResult[];        // History of actions performed
  createdAt: Date;
  updatedAt: Date;
  deliveries: Collection<NotificationDelivery>;
}
```

### NotificationDelivery

Tracks delivery attempts per channel with retry logic.

```typescript
{
  id: string;                           // UUID primary key
  notificationRecipient: NotificationRecipient;
  channel: Channel;                     // EMAIL, IN_APP
  status: DeliveryStatus;              // PENDING, SENT, FAILED, RETRYING
  sentAt?: Date;                       // Successful delivery timestamp
  failedAt?: Date;                     // Last failure timestamp
  error?: string;                      // Last error message
  retryCount: number;                  // Number of retry attempts
  nextRetryAt?: Date;                  // Next retry schedule
  createdAt: Date;
  updatedAt: Date;
}
```

### NotificationPreference

User preferences for notification types and channels per organization.

```typescript
{
  id: string;                           // UUID primary key
  user: User;                          // Target user
  organization: Organization;           // Organization scope
  muteAll: boolean;                    // Global mute flag
  channelPreferences: Record<Channel, boolean>; // Per-channel preferences
  typePreferences: Record<NotificationType, {
    enabled: boolean;
    channels: Channel[];
  }>;                                  // Per-type channel preferences
  createdAt: Date;
  updatedAt: Date;
}
```

## API Endpoints

### Get Notifications

Retrieve paginated notifications for the current user.

```http

```

## Business Rules

### Notification Creation

<Steps>
  <Step title="Access Validation">
    Notifications are created only for users with appropriate access to the related entity
  </Step>
  
  <Step title="Preference Check">
    User preferences are checked before creating notifications
  </Step>
  
  <Step title="Mute Handling">
    If `muteAll` is enabled, no notifications are created
  </Step>
  
  <Step title="Channel Selection">
    Only enabled channels from preferences are used for delivery
  </Step>
</Steps>

### Delivery Logic

<Warning>
Failed deliveries are retried with exponential backoff timing: 1min, 5min, 30min, 2hr, 12hr. After 5 failed attempts, delivery is marked as permanently failed.
</Warning>

1. **In-app notifications** are always created regardless of delivery success
2. **Email deliveries** are attempted based on user preferences
3. **Retry mechanism** handles temporary delivery failures gracefully
4. **Permanent failure** after maximum retry attempts

### Action Handling

<Note>
Actions can only be taken on notifications in UNREAD or READ status. Expired notifications cannot have actions executed.
</Note>

- Taking an action may automatically change notification status to read
- Action handlers can return redirect URLs for frontend navigation
- Action results are stored in the recipient's `actionsTaken` array for audit purposes

### Expiration

<Tip>
The cleanup job automatically removes expired notifications and their associated deliveries to maintain system performance.
</Tip>

- Expired notifications block action execution
- Action handlers receive `onExpire` callback for cleanup logic
- Automated cleanup maintains database efficiency

## Patterns and Conventions

### Notification Types

<Check>
All notification types are enumerated in `NotificationType` enum and registered in `NotificationTypeRegistry` with complete configuration.
</Check>

Configuration includes:
- Channel preferences and defaults
- Action definitions and handlers
- Template mappings for different channels

### Channel Implementation

All notification channels follow a consistent pattern:

```typescript
interface NotificationChannel {
  name: string;
  deliver(notification: NotificationRecipient): Promise<DeliveryResult>;
  supports(type: NotificationType): boolean;
}
```

- Channels implement the `NotificationChannel` interface
- Auto-discovery and registration at application startup
- Easy addition of new channels (SMS, Push notifications, etc.)

### Action Handlers

<CodeGroup>
  ```typescript Handler Structure
  @NotificationActionHandler({ name: 'approve-request' })
  class ApprovalHandler extends BaseNotificationHandler {
    async onCreate(notification: NotificationRecipient): Promise<void> {
      // Setup logic when notification is created
    }
    
    async onAction(notification: NotificationRecipient, data?: any): Promise<ActionResult> {
      // Handle action execution
    }
    
    async onExpire(notification: NotificationRecipient): Promise<void> {
      // Cleanup when notification expires
    }
  }
  ```

  ```typescript Auto-Discovery
  // Handlers are automatically discovered via decorator scanning
  // and registered in the NotificationHandlerExplorer
  ```
</CodeGroup>

### Event Naming Convention

Events follow a consistent naming pattern for easy identification and routing:

- **Domain-prefixed**: `crm.lead-assigned`, `rbac.team-member-added`
- **Kebab-case**: Consistent formatting across all event names
- **Centralized**: Defined in domain-specific event constant files

### Error Handling

<Warning>
All error scenarios are handled gracefully to prevent notification failures from breaking business workflows.
</Warning>

- **Delivery failures** are logged but don't interrupt the notification creation flow
- **Invalid action attempts** return appropriate error responses with context
- **Handler errors** are caught and logged with full context for debugging
- **Retry logic** handles transient failures automatically
