## Overview

The notification module provides a comprehensive system for managing and delivering notifications across multiple channels within the PropWise CRM. It supports in-app and email delivery channels, user preferences management, action-based notifications with handlers, and event-driven notification creation from domain events.

<Info>
The module follows an event-driven architecture where domain modules emit events that are captured by notification listeners, which then create appropriate notifications based on registered notification types and user preferences.
</Info>

**Last Updated:** 2024-12-19  
**Status:** Active

## Architecture

### Core Components

The notification system is built around several key components that work together to provide a flexible and scalable solution:

- **Notification Events**: Central notification records that can have multiple recipients
- **Notification Recipients**: Per-user instances of notifications with status tracking
- **Notification Deliveries**: Per-channel delivery attempts with retry logic
- **Notification Channels**: Pluggable delivery mechanisms (email, in-app)
- **Action Handlers**: Business logic processors for notification actions
- **Preference System**: User-configurable notification settings per organization

### Design Decisions

<AccordionGroup>
<Accordion title="Multi-Channel Architecture">
Notifications can be delivered via multiple channels simultaneously based on user preferences. This allows users to receive important notifications through their preferred medium while maintaining flexibility.
</Accordion>

<Accordion title="Recipient-Centric Design">
Each user gets their own recipient record with independent status tracking. This ensures that notification states (read, dismissed) are maintained per user without affecting other recipients.
</Accordion>

<Accordion title="Action-Driven Workflow">
Notifications can include actions that trigger business logic when clicked. This enables interactive notifications that can perform operations like approving requests or navigating to specific pages.
</Accordion>

<Accordion title="Event-Driven Creation">
Domain events automatically trigger notification creation through listeners. This decouples notification logic from business logic and ensures consistent notification delivery.
</Accordion>

<Accordion title="Organization Scoping">
All notifications are scoped to organizations for multi-tenancy. This ensures data isolation and proper access control across different organizations.
</Accordion>
</AccordionGroup>

## Entities

### NotificationEvent

Represents the core notification record that can target multiple recipients.

```typescript
{
  id: string;                           // UUID primary key
  type: NotificationType;               // LEAD_ASSIGNED, TEAM_MEMBER_ADDED, etc.
  category: NotificationCategory;       // CRM, REAL_ESTATE, RBAC, MESSAGING
  priority: NotificationPriority;       // LOW, NORMAL, HIGH, URGENT
  title: string;                        // Display title
  message: string;                      // Notification content
  data: Record<string, any>;            // Additional contextual data
  organization: Organization;           // Organization scope
  triggeredBy: User;                    // User who caused the notification
  expiresAt?: Date;                     // Optional expiry date
  createdAt: Date;
  updatedAt: Date;
  recipients: Collection<NotificationRecipient>;
}
```

### NotificationRecipient

Per-user instance of a notification with status and action tracking.

```typescript
{
  id: string;                           // UUID primary key
  notificationEvent: NotificationEvent; // Parent notification
  user: User;                          // Target user
  status: NotificationStatus;          // UNREAD, READ, DISMISSED
  readAt?: Date;                       // When marked as read
  dismissedAt?: Date;                  // When dismissed
  actionsTaken: ActionResult[];        // History of actions performed
  createdAt: Date;
  updatedAt: Date;
  deliveries: Collection<NotificationDelivery>;
}
```

### NotificationDelivery

Tracks delivery attempts per channel with retry logic.

```typescript
{
  id: string;                           // UUID primary key
  notificationRecipient: NotificationRecipient;
  channel: Channel;                     // EMAIL, IN_APP
  status: DeliveryStatus;              // PENDING, SENT, FAILED, RETRYING
  sentAt?: Date;                       // Successful delivery timestamp
  failedAt?: Date;                     // Last failure timestamp
  error?: string;                      // Last error message
  retryCount: number;                  // Number of retry attempts
  nextRetryAt?: Date;                  // Next retry schedule
  createdAt: Date;
  updatedAt: Date;
}
```

### NotificationPreference

User preferences for notification types and channels per organization.

```typescript
{
  id: string;                           // UUID primary key
  user: User;                          // Target user
  organization: Organization;           // Organization scope
  muteAll: boolean;                    // Global mute flag
  channelPreferences: Record<Channel, boolean>; // Per-channel preferences
  typePreferences: Record<NotificationType, {
    enabled: boolean;
    channels: Channel[];
  }>;                                  // Per-type channel preferences
  createdAt: Date;
  updatedAt: Date;
}
```

## API Endpoints

### Get Notifications

Retrieve notifications for the authenticated user with pagination and filtering options.

```http
GET /notifications?page=1&limit=20&status=UNREAD
```

**Query Parameters:**
- `page`: Page number (default: 1)
- `limit`: Items per page (default: 20, max: 100)
- `status`: Filter by status (UNREAD, READ, DISMISSED)

**Response:**
```json
{
  "items": [...],
  "total": 45,
  "page": 1,
  "limit": 20
}
```

### Mark as Read

Mark a specific notification as read for the authenticated user.

```http
PATCH /notifications/:id/read
```

**Response:**
```json
{
  "success": true,
  "redirectUrl": "/dashboard/leads/123"
}
```

<Note>
The `redirectUrl` is optional and provided when the notification has a default navigation target.
</Note>

### Dismiss Notification

Remove a notification from the user's notification list.

```http
DELETE /notifications/:id
```

**Response:**
```json
{
  "success": true
}
```

### Take Action

Execute a specific action associated with a notification.

```http
POST /notifications/:id/actions/:actionKey
```

**Request Body:**
```json
{
  "data": {
    "additionalParam": "value"
  }
}
```

**Response:**
```json
{
  "success": true,
  "message": "Lead approved successfully",
  "redirectUrl": "/dashboard/leads/123",
  "result": {...}
}
```

## Business Rules

### Notification Creation

<Steps>
<Step title="Access Validation">
Notifications are created only for users with appropriate access to the related entity. The system verifies permissions before adding users as recipients.
</Step>

<Step title="Preference Check">
User preferences are evaluated to determine if notifications should be created. If `muteAll` is enabled, no notifications are generated.
</Step>

<Step title="Channel Selection">
Only enabled channels from user preferences are used for delivery. This respects user communication preferences.
</Step>
</Steps>

### Delivery Logic

<Warning>
Failed deliveries are retried with exponential backoff intervals: 1 minute, 5 minutes, 30 minutes, 2 hours, and 12 hours. After 5 failed attempts, delivery is marked as permanently failed.
</Warning>

1. **In-app notifications** are always created regardless of delivery success
2. **Email deliveries** are attempted based on user preferences
3. **Retry mechanism** handles temporary failures with progressive delays
4. **Permanent failure** classification after exhausting retry attempts

### Action Handling

<Check>
Actions can only be taken on notifications in UNREAD or READ status. Dismissed notifications cannot trigger actions.
</Check>

- Taking an action may automatically change notification status to read
- Action handlers can return redirect URLs for frontend navigation
- Action results are stored in the recipient's `actionsTaken` array for audit purposes
- Multiple actions can be defined per notification type

### Expiration

Notifications can include an optional expiration date for time-sensitive content:

- Expired notifications cannot have actions taken on them
- Cleanup jobs remove expired notifications and their deliveries
- Action handlers receive `onExpire` callback for cleanup logic
- Expiration is checked before allowing any notification interactions

## Patterns and Conventions

### Notification Types

Notification types are centrally managed through a registry system:

```typescript
// Types are enumerated and registered with configuration
NotificationType {
  LEAD_ASSIGNED = 'LEAD_ASSIGNED',
  TEAM_MEMBER_ADDED = 'TEAM_MEMBER_ADDED',
  // ... other types
}
```

Each type includes:
- Default channel preferences
- Available action definitions  
- Handler mappings for business logic

### Channel Implementation

<Tip>
All channels implement the `NotificationChannel` interface, making it easy to add new delivery methods like SMS or push notifications.
</Tip>

Channels are:
- Auto-discovered and registered at startup
- Pluggable and extensible
- Support custom configuration per organization
- Include built-in retry and error handling

### Action Handlers

Action handlers follow a consistent pattern:

```typescript
@NotificationActionHandler({ name: 'approve-lead' })
export class ApproveLeadHandler extends BaseNotificationHandler {
  async onCreate(notification: NotificationEvent) {
    // Setup logic when notification is created
  }

  async onAction(action: string, data: any, recipient: NotificationRecipient) {
    // Handle the action execution
    return { success: true, redirectUrl: '/dashboard/leads' };
  }

  async onExpire(notification: NotificationEvent) {
    // Cleanup when notification expires
  }
}
```

Handlers are auto-discovered and provide lifecycle hooks for complete notification management.

### Event Naming Convention

Events follow a consistent naming pattern:

- **Domain-prefixed**: `crm.lead-assigned`, `rbac.team-member-added`
- **Kebab-case**: For consistency across the application
- **Centralized**: Defined in domain-specific event files
- **Descriptive**: Clear indication of the triggering action

### Error Handling

<Warning>
All error scenarios are handled gracefully to ensure the notification system remains resilient and doesn't impact core business operations.
</Warning>

- **Delivery failures** are logged but don't break the notification flow
- **Invalid action attempts** return appropriate error responses  
- **Handler errors** are caught and logged with full context
- **Retry logic** handles transient failures automatically
- **Circuit breakers** prevent cascade failures in external integrations
