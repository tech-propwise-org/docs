The notification module provides a comprehensive system for managing and delivering notifications across multiple channels within the PropWise CRM. It supports in-app and email delivery channels, user preferences management, action-based notifications with handlers, and event-driven notification creation from domain events.

<Note>
The module follows an event-driven architecture where domain modules emit events that are captured by notification listeners, which then create appropriate notifications based on registered notification types and user preferences.
</Note>

### Design Decisions

<AccordionGroup>
  <Accordion title="Multi-Channel Architecture">
    Notifications can be delivered via multiple channels simultaneously based on user preferences. This allows users to receive notifications through their preferred communication channels.
  </Accordion>
  
  <Accordion title="Recipient-Centric Design">
    Each user gets their own recipient record with independent status tracking, enabling personalized notification management and status control.
  </Accordion>
  
  <Accordion title="Action-Driven Workflow">
    Notifications can include actions that trigger business logic when clicked, providing interactive notification experiences.
  </Accordion>
  
  <Accordion title="Event-Driven Creation">
    Domain events automatically trigger notification creation through listeners, ensuring loose coupling between modules.
  </Accordion>
  
  <Accordion title="Retry Mechanism">
    Failed deliveries are retried with exponential backoff to handle temporary delivery failures gracefully.
  </Accordion>
  
  <Accordion title="Organization Scoping">
    All notifications are scoped to organizations for multi-tenancy isolation and security.
  </Accordion>
</AccordionGroup>

## Entities

### NotificationEvent

Represents the core notification record that can target multiple recipients.

```typescript
{
  id: string;                           // UUID primary key
  type: NotificationType;               // LEAD_ASSIGNED, TEAM_MEMBER_ADDED, etc.
  category: NotificationCategory;       // crm, properties, access, document, system, messaging
  priority: NotificationPriority;       // LOW, NORMAL, HIGH, URGENT
  title: string;                        // Display title
  message: string;                      // Notification content
  data: Record<string, any>;            // Additional contextual data
  organization: Organization;           // Organization scope
  triggeredBy: User;                    // User who caused the notification
  expiresAt?: Date;                     // Optional expiry date
  createdAt: Date;
  updatedAt: Date;
  recipients: Collection<NotificationRecipient>;
}
```

### NotificationRecipient

Per-user instance of a notification with status and action tracking.

```typescript
{
  id: string;                           // UUID primary key
  notificationEvent: NotificationEvent; // Parent notification
  user: User;                          // Target user
  status: NotificationStatus;          // UNREAD, READ, DISMISSED
  readAt?: Date;                       // When marked as read
  dismissedAt?: Date;                  // When dismissed
  actionsTaken: ActionResult[];        // History of actions performed
  createdAt: Date;
  updatedAt: Date;
  deliveries: Collection<NotificationDelivery>;
}
```

### NotificationDelivery

Tracks delivery attempts per channel with retry logic.

```typescript
{
  id: string;                           // UUID primary key
  notificationRecipient: NotificationRecipient;
  channel: Channel;                     // EMAIL, IN_APP
  status: DeliveryStatus;              // PENDING, SENT, FAILED, RETRYING
  sentAt?: Date;                       // Successful delivery timestamp
  failedAt?: Date;                     // Last failure timestamp
  error?: string;                      // Last error message
  retryCount: number;                  // Number of retry attempts
  nextRetryAt?: Date;                  // Next retry schedule
  createdAt: Date;
  updatedAt: Date;
}
```

### NotificationPreference

User preferences for notification types and channels per organization.

```typescript
{
  id: string; // UUID primary key
  user: User; // Target user
  organization: Organization; // Organization scope
  channelPreferences: Record<Channel, boolean>; // Per-channel preferences
  typePreferences: Record<
    NotificationType,
    {
      enabled: boolean;
      channels: Channel[];
    }
  >; // Per-type channel preferences
  createdAt: Date;
  updatedAt: Date;
}
```

## Notification Types

### AI-Related Notifications

<Warning>
AI-related notifications are critical for system monitoring and should be configured with appropriate urgency levels.
</Warning>

| Type                       | Category  | Description                                  |
| -------------------------- | --------- | -------------------------------------------- |
| `AI_CREDIT_LOW`            | AI        | Organization AI credits are running low      |
| `AI_CREDIT_EXHAUSTED`      | AI        | Organization has exhausted AI credits        |
| `CONVERSATION_TRANSFERRED` | MESSAGING | Conversation assigned to different user/team |
| `AI_AGENT_FAILED`          | AI        | AI agent execution failed with error         |
| `AI_AGENT_BUDGET_EXCEEDED` | AI        | AI agent exceeded token budget limits        |
| `AI_AGENT_ACTIVATED`       | AI        | AI agent has been activated                  |
| `AI_AGENT_DEACTIVATED`     | AI        | AI agent has been deactivated                |

<Note>
These notifications are triggered by AI agent actions and messaging events, following the same event-driven pattern as CRM notifications. The AI module integrates with the notification system for credit alerts, agent status changes, and execution failures.
</Note>

## Business Rules

### Notification Creation

<Steps>
  <Step title="Access Validation">
    Notifications are created only for users with appropriate access to the related entity.
  </Step>
  
  <Step title="Preference Check">
    User preferences are checked before creating notifications to respect user settings.
  </Step>
  
  <Step title="Mutable Override">
    Non-mutable types always deliver regardless of parent mute overrides.
  </Step>
  
  <Step title="Channel Selection">
    Only enabled channels from preferences are used for delivery.
  </Step>
</Steps>

### Preference Taxonomy

<Info>
Top-level preference categories are `crm`, `properties`, `access`, `document`, `system`, and `messaging`. Commission payments, tasks, calendar events, and reminders are grouped under **CRM** via subcategories (`commission`, `tasks`, `events`, `reminders`) — there are no separate `commission`, `task`, or `event` category keys in `categorySettings`.
</Info>

### Delivery Logic

<AccordionGroup>
  <Accordion title="In-App Notifications">
    In-app notifications are always created regardless of delivery success to ensure users have access to notification history.
  </Accordion>
  
  <Accordion title="Email Deliveries">
    Email deliveries are attempted based on user preferences and channel settings.
  </Accordion>
  
  <Accordion title="Retry Strategy">
    Failed deliveries are retried with exponential backoff: 1min, 5min, 30min, 2hr, 12hr.
  </Accordion>
  
  <Accordion title="Permanent Failure">
    After 5 failed attempts, delivery is marked as permanently failed.
  </Accordion>
</AccordionGroup>

### Action Handling

<Warning>
Actions can only be taken on notifications in UNREAD or READ status. Expired or dismissed notifications cannot execute actions.
</Warning>

- Taking an action may change notification status to read automatically
- Action handlers can return redirect URLs for frontend navigation
- Action results are stored in the recipient's `actionsTaken` array for audit purposes

### Expiration

<Check>
The system automatically handles notification expiration through cleanup jobs.
</Check>

- Expired notifications cannot have actions taken on them
- Cleanup job removes expired notifications and their deliveries
- Action handlers receive `onExpire` callback for cleanup logic

## Patterns and Conventions

### Notification Types

<Steps>
  <Step title="Enumeration">
    All notification types are enumerated in the `NotificationType` enum for type safety.
  </Step>
  
  <Step title="Registry">
    Types are registered in `NotificationTypeRegistry` with configuration including channel preferences, action definitions, and handler mappings.
  </Step>
  
  <Step title="Configuration">
    Each type includes default channel preferences and available actions.
  </Step>
</Steps>

### Channel Implementation

<Tip>
The channel system is designed for extensibility. Adding new channels like SMS or Push notifications requires implementing the `NotificationChannel` interface.
</Tip>

- All channels implement `NotificationChannel` interface
- Channels are auto-discovered and registered at startup
- Support for easy addition of new channels (SMS, Push, etc.)

### Action Handlers

Action handlers provide the business logic for interactive notifications:

```typescript
@NotificationActionHandler({ name: 'approve-lead' })
export class ApproveLeadHandler extends BaseNotificationHandler {
  async onAction(recipient: NotificationRecipient, actionData?: any) {
    // Implementation logic
    return { success: true, redirectUrl: '/leads' };
  }
}
```

- Extend `BaseNotificationHandler` abstract class
- Decorated with `@NotificationActionHandler({ name })`
- Auto-discovered via `NotificationHandlerExplorer`
- Provide lifecycle hooks: `onCreate`, `onAction`, `onExpire`

### Event Naming

<Info>
Consistent event naming helps maintain clarity across the system and makes debugging easier.
</Info>

- Domain-prefixed: `crm.lead-assigned`, `rbac.team-member-added`
- Kebab-case for consistency
- Centralized in domain-specific event files

### Error Handling

<Warning>
The notification system is designed to be resilient. Delivery failures should not impact the core business processes.
</Warning>

- Delivery failures are logged but don't break the notification flow
- Invalid action attempts return appropriate error responses
- Handler errors are caught and logged with context for debugging
