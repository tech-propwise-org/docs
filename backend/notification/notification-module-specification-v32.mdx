## Overview

The notification module provides a comprehensive system for managing and delivering notifications across multiple channels within the PropWise CRM. It supports in-app and email delivery channels, user preferences management, action-based notifications with handlers, and event-driven notification creation from domain events.

The module follows an event-driven architecture where domain modules emit events that are captured by notification listeners, which then create appropriate notifications based on registered notification types and user preferences.

## Architecture

### Core Components

The notification system is built around several key components that work together to provide a robust notification infrastructure:

- **Notification Events**: Central notification records that can have multiple recipients
- **Notification Recipients**: Per-user instances of notifications with status tracking
- **Notification Deliveries**: Per-channel delivery attempts with retry logic
- **Notification Channels**: Pluggable delivery mechanisms (email, in-app)
- **Action Handlers**: Business logic processors for notification actions
- **Preference System**: User-configurable notification settings per organization

### Design Decisions

<AccordionGroup>
  <Accordion title="Multi-Channel Architecture">
    Notifications can be delivered via multiple channels simultaneously based on user preferences, allowing users to receive notifications through their preferred communication methods.
  </Accordion>
  <Accordion title="Recipient-Centric Design">
    Each user gets their own recipient record with independent status tracking, enabling personalized notification experiences and individual action handling.
  </Accordion>
  <Accordion title="Action-Driven Workflow">
    Notifications can include actions that trigger business logic when clicked, providing interactive notification experiences that drive user engagement.
  </Accordion>
  <Accordion title="Event-Driven Creation">
    Domain events automatically trigger notification creation through listeners, ensuring notifications are created in response to real business events.
  </Accordion>
  <Accordion title="Retry Mechanism">
    Failed deliveries are retried with exponential backoff to handle temporary delivery issues gracefully.
  </Accordion>
  <Accordion title="Organization Scoping">
    All notifications are scoped to organizations for multi-tenancy, ensuring data isolation between different organizations.
  </Accordion>
</AccordionGroup>

## Entities

### NotificationEvent

Represents the core notification record that can target multiple recipients.

```typescript
{
  id: string;                           // UUID primary key
  type: NotificationType;               // LEAD_ASSIGNED, TEAM_MEMBER_ADDED, etc.
  category: NotificationCategory;       // crm, properties, access, document, system, messaging
  priority: NotificationPriority;       // LOW, NORMAL, HIGH, URGENT
  title: string;                        // Display title
  message: string;                      // Notification content
  data: Record<string, any>;            // Additional contextual data
  organization: Organization;           // Organization scope
  triggeredBy: User;                    // User who caused the notification
  expiresAt?: Date;                     // Optional expiry date
  createdAt: Date;
  updatedAt: Date;
  recipients: Collection<NotificationRecipient>;
}
```

### NotificationRecipient

Per-user instance of a notification with status and action tracking.

```typescript
{
  id: string;                           // UUID primary key
  notificationEvent: NotificationEvent; // Parent notification
  user: User;                          // Target user
  status: NotificationStatus;          // UNREAD, READ, DISMISSED
  readAt?: Date;                       // When marked as read
  dismissedAt?: Date;                  // When dismissed
  actionsTaken: ActionResult[];        // History of actions performed
  createdAt: Date;
  updatedAt: Date;
  deliveries: Collection<NotificationDelivery>;
}
```

### NotificationDelivery

Tracks delivery attempts per channel with retry logic.

```typescript
{
  id: string;                           // UUID primary key
  notificationRecipient: NotificationRecipient;
  channel: Channel;                     // EMAIL, IN_APP
  status: DeliveryStatus;              // PENDING, SENT, FAILED, RETRYING
  sentAt?: Date;                       // Successful delivery timestamp
  failedAt?: Date;                     // Last failure timestamp
  error?: string;                      // Last error message
  retryCount: number;                  // Number of retry attempts
  nextRetryAt?: Date;                  // Next retry schedule
  createdAt: Date;
  updatedAt: Date;
}
```

### NotificationPreference

User preferences for notification types and channels per organization.

```typescript
{
  id: string; // UUID primary key
  user: User; // Target user
  organization: Organization; // Organization scope
  channelPreferences: Record<Channel, boolean>; // Per-channel preferences
  typePreferences: Record<
    NotificationType,
    {
      enabled: boolean;
      channels: Channel[];
    }
  >; // Per-type channel preferences
  createdAt: Date;
  updatedAt: Date;
}
```

## API Endpoints

### Get Notifications

Retrieve notifications for the current user with pagination and filtering options.

```http
GET /notifications?page=1&limit=20&status=UNREAD
```

<Tabs>
  <Tab title="Response">
    ```typescript
    {
      items: NotificationDto[];
      total: number;
      page: number;
      limit: number;
    }
    ```
  </Tab>
  <Tab title="Query Parameters">
    - `page`: Page number (default: 1)
    - `limit`: Items per page (default: 20, max: 100)
    - `status`: Filter by status (UNREAD, READ, DISMISSED)
    - `category`: Filter by notification category
    - `priority`: Filter by priority level
  </Tab>
</Tabs>

### Mark as Read

Mark a specific notification as read.

```http
PATCH /notifications/:id/read
```

**Response:**
```typescript
{
  success: boolean;
  redirectUrl?: string;
}
```

### Dismiss Notification

Remove a notification from the user's notification list.

```http
DELETE /notifications/:id
```

**Response:**
```typescript
{
  success: boolean;
}
```

### Take Action

Execute a specific action associated with a notification.

```http
POST /notifications/:id/actions/:actionKey
```

<CodeGroup>
```typescript Request Body
{
  data?: Record<string, any>;
}
```

```typescript Response
{
  success: boolean;
  message?: string;
  redirectUrl?: string;
  result?: any;
}
```
</CodeGroup>

### Get Preferences

Retrieve the current user's notification preferences.

```http
GET /notifications/preferences
```

### Update Preferences

Update notification preferences for channels and notification types.

```http
PUT /notifications/preferences
```

**Request Body:**
```typescript
{
  channelPreferences?: Record<Channel, boolean>;
  typePreferences?: Record<NotificationType, {
    enabled: boolean;
    channels: Channel[];
  }>;
}
```

## Business Rules

### Notification Creation

<Steps>
  <Step title="Access Validation">
    Notifications are created only for users with appropriate access to the related entity, ensuring security and relevance.
  </Step>
  <Step title="Preference Checking">
    User preferences are checked before creating notifications to respect user communication preferences.
  </Step>
  <Step title="Mutable Type Handling">
    Non-mutable types always deliver regardless of parent mute overrides, ensuring critical notifications are not missed.
  </Step>
  <Step title="Channel Selection">
    Only enabled channels from preferences are used for delivery, respecting user communication channel choices.
  </Step>
</Steps>

### Preference Taxonomy

<Important>
Top-level preference categories are `crm`, `properties`, `access`, `document`, `system`, and `messaging`. Commission payments, tasks, calendar events, and reminders are grouped under **CRM** via subcategories (`commission`, `tasks`, `events`, `reminders`) — there are no separate `commission`, `task`, or `event` category keys in `categorySettings`.
</Important>

### Delivery Logic

<CardGroup cols={2}>
  <Card title="In-App Notifications" icon="bell">
    Always created regardless of delivery success to ensure users can access notifications within the application.
  </Card>
  <Card title="Email Deliveries" icon="envelope">
    Attempted based on user preferences with retry logic for failed attempts.
  </Card>
  <Card title="Retry Strategy" icon="arrows-rotate">
    Failed deliveries are retried with exponential backoff: 1min, 5min, 30min, 2hr, 12hr.
  </Card>
  <Card title="Permanent Failure" icon="circle-exclamation">
    After 5 failed attempts, delivery is marked as permanently failed.
  </Card>
</CardGroup>

### Action Handling

<Warning>
Actions can only be taken on notifications in UNREAD or READ status. Expired or dismissed notifications cannot execute actions.
</Warning>

- Taking an action may change notification status to read
- Action handlers can return redirect URLs for frontend navigation
- Action results are stored in the recipient's `actionsTaken` array for audit purposes

### Expiration

- Expired notifications cannot have actions taken
- Cleanup job removes expired notifications and their deliveries
- Action handlers receive `onExpire` callback for cleanup logic

## Patterns and Conventions

### Notification Types

<Tip>
All notification types follow a consistent pattern for easy maintenance and extension.
</Tip>

- Enumerated in `NotificationType` enum
- Registered in `NotificationTypeRegistry` with configuration
- Include channel preferences, action definitions, and handler mappings

### Channel Implementation

All notification channels follow a consistent interface pattern:

- All channels implement `NotificationChannel` interface
- Channels are auto-discovered and registered at startup
- Support for easy addition of new channels (SMS, Push, etc.)

### Action Handlers

Action handlers provide a standardized way to process notification actions:

- Extend `BaseNotificationHandler` abstract class
- Decorated with `@NotificationActionHandler({ name })`
- Auto-discovered via `NotificationHandlerExplorer`
- Provide lifecycle hooks: `onCreate`, `onAction`, `onExpire`

### Event Naming

<Check>
Events follow a consistent naming convention for easy identification and routing.
</Check>

- Domain-prefixed: `crm.lead-assigned`, `rbac.team-member-added`
- Kebab-case for consistency
- Centralized in domain-specific event files

### Error Handling

The notification system implements comprehensive error handling:

- Delivery failures are logged but don't break the notification flow
- Invalid action attempts return appropriate error responses
- Handler errors are caught and logged with context for debugging
