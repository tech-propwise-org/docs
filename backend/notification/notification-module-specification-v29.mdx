## Overview

The notification module provides a comprehensive system for managing and delivering notifications across multiple channels within the PropWise CRM. It supports in-app and email delivery channels, user preferences management, action-based notifications with handlers, and event-driven notification creation from domain events.

<Info>
The module follows an event-driven architecture where domain modules emit events that are captured by notification listeners, which then create appropriate notifications based on registered notification types and user preferences.
</Info>

### Design decisions

<AccordionGroup>
  <Accordion title="Multi-channel architecture">
    Notifications can be delivered via multiple channels simultaneously based on user preferences, ensuring users receive notifications through their preferred communication methods.
  </Accordion>
  
  <Accordion title="Recipient-centric design">
    Each user gets their own recipient record with independent status tracking, allowing for granular control over notification states per user.
  </Accordion>
  
  <Accordion title="Action-driven workflow">
    Notifications can include actions that trigger business logic when clicked, enabling interactive notifications that drive user engagement.
  </Accordion>
  
  <Accordion title="Event-driven creation">
    Domain events automatically trigger notification creation through listeners, ensuring consistent notification delivery across the system.
  </Accordion>
  
  <Accordion title="Retry mechanism">
    Failed deliveries are retried with exponential backoff to handle temporary delivery failures gracefully.
  </Accordion>
  
  <Accordion title="Organization scoping">
    All notifications are scoped to organizations for multi-tenancy, ensuring proper data isolation.
  </Accordion>
</AccordionGroup>

## Entities

### NotificationEvent

Represents the core notification record that can target multiple recipients.

```typescript
{
  id: string;                           // UUID primary key
  type: NotificationType;               // LEAD_ASSIGNED, TEAM_MEMBER_ADDED, etc.
  category: NotificationCategory;       // crm, properties, access, document, system, messaging
  priority: NotificationPriority;       // LOW, NORMAL, HIGH, URGENT
  title: string;                        // Display title
  message: string;                      // Notification content
  data: Record<string, any>;            // Additional contextual data
  organization: Organization;           // Organization scope
  triggeredBy: User;                    // User who caused the notification
  expiresAt?: Date;                     // Optional expiry date
  createdAt: Date;
  updatedAt: Date;
  recipients: Collection<NotificationRecipient>;
}
```

### NotificationRecipient

Per-user instance of a notification with status and action tracking.

```typescript
{
  id: string;                           // UUID primary key
  notificationEvent: NotificationEvent; // Parent notification
  user: User;                          // Target user
  status: NotificationStatus;          // UNREAD, READ, DISMISSED
  readAt?: Date;                       // When marked as read
  dismissedAt?: Date;                  // When dismissed
  actionsTaken: ActionResult[];        // History of actions performed
  createdAt: Date;
  updatedAt: Date;
  deliveries: Collection<NotificationDelivery>;
}
```

### NotificationDelivery

Tracks delivery attempts per channel with retry logic.

```typescript
{
  id: string;                           // UUID primary key
  notificationRecipient: NotificationRecipient;
  channel: Channel;                     // EMAIL, IN_APP
  status: DeliveryStatus;              // PENDING, SENT, FAILED, RETRYING
  sentAt?: Date;                       // Successful delivery timestamp
  failedAt?: Date;                     // Last failure timestamp
  error?: string;                      // Last error message
  retryCount: number;                  // Number of retry attempts
  nextRetryAt?: Date;                  // Next retry schedule
  createdAt: Date;
  updatedAt: Date;
}
```

### NotificationPreference

User preferences for notification types and channels per organization.

```typescript
{
  id: string; // UUID primary key
  user: User; // Target user
  organization: Organization; // Organization scope
  channelPreferences: Record<Channel, boolean>; // Per-channel preferences
  typePreferences: Record<
    NotificationType,
    {
      enabled: boolean;
      channels: Channel[];
    }
  >; // Per-type channel preferences
  createdAt: Date;
  updatedAt: Date;
}
```

## API endpoints

### Get notifications

Retrieve paginated list of notifications for the current user.

```http
GET /notifications?page=1&limit=20&status=UNREAD
```

**Response:**

```typescript
{
  items: NotificationDto[];
  total: number;
  page: number;
  limit: number;
}
```

### Mark as read

Mark a notification as read and optionally get redirect URL.

```http
PATCH /notifications/:id/read
```

**Response:**

```typescript
{
  success: boolean;
  redirectUrl?: string;
}
```

### Dismiss notification

Dismiss a notification (soft delete).

```http
DELETE /notifications/:id
```

**Response:**

```typescript
{
  success: boolean;
}
```

### Take action

Execute a notification action handler.

```http
POST /notifications/:id/actions/:actionKey
```

**Request body:**

```typescript
{
  data?: Record<string, any>;
}
```

**Response:**

```typescript
{
  success: boolean;
  message?: string;
  redirectUrl?: string;
  result?: any;
}
```

### Get preferences

Retrieve current user's notification preferences.

```http
GET /notifications/preferences
```

### Update preferences

Update user's notification preferences.

```http
PUT /notifications/preferences
```

**Request body:**

```typescript
{
  channelPreferences?: Record<Channel, boolean>;
  typePreferences?: Record<NotificationType, {
    enabled: boolean;
    channels: Channel[];
  }>;
}
```

## Business rules

### Notification creation

<Steps>
  <Step title="Access validation">
    Notifications are created only for users with appropriate access to the related entity.
  </Step>
  
  <Step title="Preference check">
    User preferences are checked before creating notifications to ensure they want to receive them.
  </Step>
  
  <Step title="Non-mutable types">
    Non-mutable types always deliver regardless of parent mute overrides.
  </Step>
  
  <Step title="Channel filtering">
    Only enabled channels from preferences are used for delivery.
  </Step>
</Steps>

### Preference taxonomy

<Warning>
Top-level preference categories are `crm`, `properties`, `access`, `document`, `system`, and `messaging`. Commission payments, tasks, calendar events, and reminders are grouped under **CRM** via subcategories (`commission`, `tasks`, `events`, `reminders`) — there are no separate `commission`, `task`, or `event` category keys in `categorySettings`.
</Warning>

### Delivery logic

<Tabs>
  <Tab title="In-app">
    In-app notifications are always created regardless of delivery success to ensure users can see them in the UI.
  </Tab>
  
  <Tab title="Email">
    Email deliveries are attempted based on user preferences and can be disabled per notification type.
  </Tab>
  
  <Tab title="Retry mechanism">
    Failed deliveries are retried with exponential backoff: 1min, 5min, 30min, 2hr, 12hr. After 5 failed attempts, delivery is marked as permanently failed.
  </Tab>
</Tabs>

### Action handling

<Check>Actions can only be taken on notifications in UNREAD or READ status</Check>
<Check>Taking an action may change notification status to read</Check>
<Check>Action handlers can return redirect URLs for frontend navigation</Check>
<Check>Action results are stored in the recipient's `actionsTaken` array</Check>

### Expiration

<Warning>
- Expired notifications cannot have actions taken
- Cleanup job removes expired notifications and their deliveries  
- Action handlers receive `onExpire` callback for cleanup logic
</Warning>

## Integration points

### Event system

The notification module integrates with the event system to receive domain events and convert them into notifications:

- **Domain events**: CRM, Real Estate, RBAC, and Messaging modules emit events
- **Event listeners**: Convert domain events to notifications based on business rules
- **Event emitter**: NestJS EventEmitter for async event processing

### User module

- User entity relationships for recipients and preferences
- Access control integration for notification visibility

### Organization module

- Organization scoping for all notifications
- Multi-tenant isolation

### Email service

- External email delivery via configured email provider
- Template-based email generation

### WebSocket gateway

- Real-time delivery of in-app notifications
- Live notification count updates

## Patterns and conventions

### Notification types

<CodeGroup>

```typescript Notification type enum
enum NotificationType {
  LEAD_ASSIGNED = 'LEAD_ASSIGNED',
  TEAM_MEMBER_ADDED = 'TEAM_MEMBER_ADDED',
  AI_CREDIT_LOW = 'AI_CREDIT_LOW',
  // ... other types
}
```

```typescript Type registration
NotificationTypeRegistry.register({
  type: NotificationType.LEAD_ASSIGNED,
  channels: [Channel.EMAIL, Channel.IN_APP],
  actions: ['accept', 'decline'],
  handlerClass: LeadAssignmentHandler
});
```

</CodeGroup>

### Channel implementation

All channels must implement the `NotificationChannel` interface:

```typescript
interface NotificationChannel {
  readonly name: Channel;
  
  deliver(
    recipient: NotificationRecipient,
    context: DeliveryContext
  ): Promise<DeliveryResult>;
  
  isEnabled(user: User, organization: Organization): Promise<boolean>;
}
```

### Action handlers

<Steps>
  <Step title="Extend base class">
    All action handlers must extend `BaseNotificationHandler` abstract class.
  </Step>
  
  <Step title="Add decorator">
    Use `@NotificationActionHandler({ name })` decorator for auto-discovery.
  </Step>
  
  <Step title="Implement lifecycle hooks">
    Provide `onCreate`, `onAction`, and `onExpire` lifecycle hooks as needed.
  </Step>
</Steps>

```typescript
@NotificationActionHandler({ name: 'lead-assignment' })
export class LeadAssignmentHandler extends BaseNotificationHandler {
  async onCreate(event: NotificationEvent): Promise<void> {
    // Setup logic when notification is created
  }
  
  async onAction(
    recipient: NotificationRecipient, 
    action: string, 
    data?: any
  ): Promise<ActionResult> {
    // Handle action execution
    if (action === 'accept') {
      // Accept logic
      return { success: true, redirectUrl: '/leads/123' };
    }
    
    throw new Error(`Unknown action: ${action}`);
  }
  
  async onExpire(event: NotificationEvent): Promise<void> {
    // Cleanup logic when notification expires
  }
}
```

### Event naming

<Tip>
Events follow a consistent naming pattern for easy identification and routing:
- Domain-prefixed: `crm.lead-assigned`, `rbac.team-member-added`
- Kebab-case for consistency
- Centralized in domain-specific event files
</Tip>

### Error handling

The notification system implements comprehensive error handling:

- **Delivery failures**: Logged but don't break the notification flow
- **Invalid actions**: Return appropriate error responses
- **Handler errors**: Caught and logged with context for debugging

<Note>
All errors are logged with sufficient context to enable effective troubleshooting while maintaining system stability.
</Note>
