## Overview

The notification module provides a comprehensive system for managing and delivering notifications across multiple channels within the PropWise CRM. It supports in-app and email delivery channels, user preferences management, action-based notifications with handlers, and event-driven notification creation from domain events.

<Info>
The module follows an event-driven architecture where domain modules emit events that are captured by notification listeners, which then create appropriate notifications based on registered notification types and user preferences.
</Info>

### Design Decisions

<AccordionGroup>
  <Accordion title="Multi-Channel Architecture">
    Notifications can be delivered via multiple channels simultaneously based on user preferences, allowing flexible delivery options.
  </Accordion>
  
  <Accordion title="Recipient-Centric Design">
    Each user gets their own recipient record with independent status tracking, enabling per-user notification management.
  </Accordion>
  
  <Accordion title="Action-Driven Workflow">
    Notifications can include actions that trigger business logic when clicked, creating interactive notification experiences.
  </Accordion>
  
  <Accordion title="Event-Driven Creation">
    Domain events automatically trigger notification creation through listeners, ensuring real-time notification delivery.
  </Accordion>
  
  <Accordion title="Retry Mechanism">
    Failed deliveries are retried with exponential backoff to handle temporary delivery issues gracefully.
  </Accordion>
  
  <Accordion title="Organization Scoping">
    All notifications are scoped to organizations for multi-tenancy support and data isolation.
  </Accordion>
</AccordionGroup>

## Entities

### NotificationEvent

Represents the core notification record that can target multiple recipients.

<CodeGroup>

```typescript TypeScript Interface
interface NotificationEvent {
  id: string;                           // UUID primary key
  type: NotificationType;               // LEAD_ASSIGNED, TEAM_MEMBER_ADDED, etc.
  category: NotificationCategory;       // crm, properties, access, document, system, messaging
  priority: NotificationPriority;       // LOW, NORMAL, HIGH, URGENT
  title: string;                        // Display title
  message: string;                      // Notification content
  data: Record<string, any>;            // Additional contextual data
  organization: Organization;           // Organization scope
  triggeredBy: User;                    // User who caused the notification
  expiresAt?: Date;                     // Optional expiry date
  createdAt: Date;
  updatedAt: Date;
  recipients: Collection<NotificationRecipient>;
}
```

</CodeGroup>

### NotificationRecipient

Per-user instance of a notification with status and action tracking.

<CodeGroup>

```typescript TypeScript Interface
interface NotificationRecipient {
  id: string;                           // UUID primary key
  notificationEvent: NotificationEvent; // Parent notification
  user: User;                          // Target user
  status: NotificationStatus;          // UNREAD, READ, DISMISSED
  readAt?: Date;                       // When marked as read
  dismissedAt?: Date;                  // When dismissed
  actionsTaken: ActionResult[];        // History of actions performed
  createdAt: Date;
  updatedAt: Date;
  deliveries: Collection<NotificationDelivery>;
}
```

</CodeGroup>

### NotificationDelivery

Tracks delivery attempts per channel with retry logic.

<CodeGroup>

```typescript TypeScript Interface
interface NotificationDelivery {
  id: string;                           // UUID primary key
  notificationRecipient: NotificationRecipient;
  channel: Channel;                     // EMAIL, IN_APP
  status: DeliveryStatus;              // PENDING, SENT, FAILED, RETRYING
  sentAt?: Date;                       // Successful delivery timestamp
  failedAt?: Date;                     // Last failure timestamp
  error?: string;                      // Last error message
  retryCount: number;                  // Number of retry attempts
  nextRetryAt?: Date;                  // Next retry schedule
  createdAt: Date;
  updatedAt: Date;
}
```

</CodeGroup>

### NotificationPreference

User preferences for notification types and channels per organization.

<CodeGroup>

```typescript TypeScript Interface
interface NotificationPreference {
  id: string;                          // UUID primary key
  user: User;                         // Target user
  organization: Organization;          // Organization scope
  channelPreferences: Record<Channel, boolean>; // Per-channel preferences
  typePreferences: Record<
    NotificationType,
    {
      enabled: boolean;
      channels: Channel[];
    }
  >;                                  // Per-type channel preferences
  createdAt: Date;
  updatedAt: Date;
}
```

</CodeGroup>

## Business Rules

### Notification Creation

<Steps>
  <Step title="Access Control Check">
    Notifications are created only for users with appropriate access to the related entity
  </Step>
  
  <Step title="Preference Validation">
    User preferences are checked before creating notifications to respect user settings
  </Step>
  
  <Step title="Non-Mutable Override">
    Non-mutable types always deliver regardless of parent mute overrides for critical notifications
  </Step>
  
  <Step title="Channel Selection">
    Only enabled channels from preferences are used for delivery
  </Step>
</Steps>

### Preference Taxonomy

<Warning>
Top-level preference categories are `crm`, `properties`, `access`, `document`, `system`, and `messaging`. Commission payments, tasks, calendar events, and reminders are grouped under **CRM** via subcategories (`commission`, `tasks`, `events`, `reminders`) — there are no separate `commission`, `task`, or `event` category keys in `categorySettings`.
</Warning>

### Delivery Logic

<Tabs>
  <Tab title="In-App Notifications">
    - Always created regardless of delivery success
    - Delivered via WebSocket for real-time updates
    - Persist until user dismisses or they expire
  </Tab>
  
  <Tab title="Email Deliveries">
    - Attempted based on user preferences
    - Failed deliveries are retried with exponential backoff
    - Retry schedule: 1min, 5min, 30min, 2hr, 12hr
    - After 5 failed attempts, delivery is marked as permanently failed
  </Tab>
</Tabs>

### Action Handling

<Check>
Actions can only be taken on notifications in UNREAD or READ status
</Check>

<Check>
Taking an action may change notification status to read automatically
</Check>

<Check>
Action handlers can return redirect URLs for frontend navigation
</Check>

<Check>
Action results are stored in the recipient's `actionsTaken` array for audit trails
</Check>

### Expiration

<Warning>
Expired notifications cannot have actions taken on them. A cleanup job removes expired notifications and their deliveries. Action handlers receive `onExpire` callback for cleanup logic.
</Warning>

## Patterns and Conventions

### Notification Types

<Tip>
Notification types are enumerated in `NotificationType` enum, registered in `NotificationTypeRegistry` with configuration, and include channel preferences, action definitions, and handler mappings.
</Tip>

### Channel Implementation

All channels follow a consistent pattern:

<Steps>
  <Step title="Interface Implementation">
    All channels implement the `NotificationChannel` interface
  </Step>
  
  <Step title="Auto-Discovery">
    Channels are auto-discovered and registered at application startup
  </Step>
  
  <Step title="Extensibility">
    Easy addition of new channels (SMS, Push notifications, etc.)
  </Step>
</Steps>

### Action Handlers

<CodeGroup>

```typescript Handler Pattern
@NotificationActionHandler({ name: 'approve-request' })
export class ApprovalHandler extends BaseNotificationHandler {
  async onCreate(notification: NotificationEvent): Promise<void> {
    // Setup logic when notification is created
  }

  async onAction(
    recipient: NotificationRecipient,
    actionKey: string,
    data?: any
  ): Promise<ActionResult> {
    // Handle the action logic
    return {
      success: true,
      message: 'Request approved',
      redirectUrl: '/dashboard'
    };
  }

  async onExpire(notification: NotificationEvent): Promise<void> {
    // Cleanup logic when notification expires
  }
}
```

</CodeGroup>

Action handlers are:
- Extended from `BaseNotificationHandler` abstract class
- Decorated with `@NotificationActionHandler({ name })`
- Auto-discovered via `NotificationHandlerExplorer`
- Provide lifecycle hooks: `onCreate`, `onAction`, `onExpire`

### Event Naming Convention

<Info>
Events follow a consistent naming pattern for easy identification and routing.
</Info>

- **Domain-prefixed**: `crm.lead-assigned`, `rbac.team-member-added`
- **Kebab-case**: For consistency across all event names
- **Centralized**: Defined in domain-specific event files

### Error Handling

<Warning>
The system handles errors gracefully to prevent notification failures from breaking core functionality.
</Warning>

<Tabs>
  <Tab title="Delivery Failures">
    Delivery failures are logged but don't break the notification flow. The system continues processing other notifications.
  </Tab>
  
  <Tab title="Invalid Actions">
    Invalid action attempts return appropriate error responses with clear messaging for debugging.
  </Tab>
  
  <Tab title="Handler Errors">
    Handler errors are caught and logged with full context for troubleshooting and monitoring.
  </Tab>
</Tabs>
