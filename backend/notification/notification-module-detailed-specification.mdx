## Overview

The notification module provides a comprehensive system for managing and delivering notifications across multiple channels within the PropWise CRM. It supports in-app and email delivery channels, user preferences management, action-based notifications with handlers, and event-driven notification creation from domain events.

<Info>
The module follows an event-driven architecture where domain modules emit events that are captured by notification listeners, which then create appropriate notifications based on registered notification types and user preferences.
</Info>

## Architecture

### Core Components

The notification system is built around several key architectural components:

- **Notification Events**: Central notification records that can have multiple recipients
- **Notification Recipients**: Per-user instances of notifications with status tracking
- **Notification Deliveries**: Per-channel delivery attempts with retry logic
- **Notification Channels**: Pluggable delivery mechanisms (email, in-app)
- **Action Handlers**: Business logic processors for notification actions
- **Preference System**: User-configurable notification settings per organization

### Design Decisions

<AccordionGroup>
<Accordion title="Multi-Channel Architecture">
Notifications can be delivered via multiple channels simultaneously based on user preferences. This allows users to receive the same notification through email and in-app channels if desired.
</Accordion>

<Accordion title="Recipient-Centric Design">
Each user gets their own recipient record with independent status tracking. This enables granular control over notification states per user while maintaining a shared notification event.
</Accordion>

<Accordion title="Action-Driven Workflow">
Notifications can include actions that trigger business logic when clicked. This enables interactive notifications that can perform operations like lead assignment or task completion.
</Accordion>

<Accordion title="Event-Driven Creation">
Domain events automatically trigger notification creation through listeners. This decouples notification logic from business domains while ensuring relevant events generate appropriate notifications.
</Accordion>

<Accordion title="Organization Scoping">
All notifications are scoped to organizations for multi-tenancy. This ensures complete data isolation between different client organizations.
</Accordion>
</AccordionGroup>

## Entities

### NotificationEvent

Represents the core notification record that can target multiple recipients.

```typescript
{
  id: string;                           // UUID primary key
  type: NotificationType;               // LEAD_ASSIGNED, TEAM_MEMBER_ADDED, etc.
  category: NotificationCategory;       // CRM, REAL_ESTATE, RBAC, MESSAGING
  priority: NotificationPriority;       // LOW, NORMAL, HIGH, URGENT
  title: string;                        // Display title
  message: string;                      // Notification content
  data: Record<string, any>;            // Additional contextual data
  organization: Organization;           // Organization scope
  triggeredBy: User;                    // User who caused the notification
  expiresAt?: Date;                     // Optional expiry date
  createdAt: Date;
  updatedAt: Date;
  recipients: Collection<NotificationRecipient>;
}
```

### NotificationRecipient

Per-user instance of a notification with status and action tracking.

```typescript
{
  id: string;                           // UUID primary key
  notificationEvent: NotificationEvent; // Parent notification
  user: User;                          // Target user
  status: NotificationStatus;          // UNREAD, READ, DISMISSED
  readAt?: Date;                       // When marked as read
  dismissedAt?: Date;                  // When dismissed
  actionsTaken: ActionResult[];        // History of actions performed
  createdAt: Date;
  updatedAt: Date;
  deliveries: Collection<NotificationDelivery>;
}
```

### NotificationDelivery

Tracks delivery attempts per channel with retry logic.

```typescript
{
  id: string;                           // UUID primary key
  notificationRecipient: NotificationRecipient;
  channel: Channel;                     // EMAIL, IN_APP
  status: DeliveryStatus;              // PENDING, SENT, FAILED, RETRYING
  sentAt?: Date;                       // Successful delivery timestamp
  failedAt?: Date;                     // Last failure timestamp
  error?: string;                      // Last error message
  retryCount: number;                  // Number of retry attempts
  nextRetryAt?: Date;                  // Next retry schedule
  createdAt: Date;
  updatedAt: Date;
}
```

### NotificationPreference

User preferences for notification types and channels per organization.

```typescript
{
  id: string;                           // UUID primary key
  user: User;                          // Target user
  organization: Organization;           // Organization scope
  muteAll: boolean;                    // Global mute flag
  channelPreferences: Record<Channel, boolean>; // Per-channel preferences
  typePreferences: Record<NotificationType, {
    enabled: boolean;
    channels: Channel[];
  }>;                                  // Per-type channel preferences
  createdAt: Date;
  updatedAt: Date;
}
```

**Response:**
```typescript
{
  success: boolean;
  redirectUrl?: string;
}
```

```http

```

**Response:**
```typescript
{
  success: boolean;
}
```
</Tab>
</Tabs>

## Business Rules

### Notification Creation Rules

<Steps>
<Step title="Access Validation">
Notifications are created only for users with appropriate access to the related entity. The system verifies user permissions before generating any notification records.
</Step>

<Step title="Preference Checking">
User preferences are evaluated to determine if notifications should be created. If `muteAll` is enabled, no notifications are generated for that user.
</Step>

<Step title="Channel Selection">
Only enabled channels from user preferences are used for delivery attempts. Disabled channels are skipped entirely.
</Step>
</Steps>

### Delivery Logic

<Warning>
Failed deliveries are retried with exponential backoff: 1min, 5min, 30min, 2hr, 12hr. After 5 failed attempts, delivery is marked as permanently failed.
</Warning>

- **In-app notifications**: Always created regardless of delivery success
- **Email deliveries**: Attempted based on user preferences and email availability
- **Retry mechanism**: Automatic retry with increasing delays for failed deliveries
- **Permanent failure**: After maximum retry attempts, deliveries are marked as failed

### Action Handling

<Check>
Actions can only be taken on notifications in UNREAD or READ status. Dismissed notifications cannot have actions executed.
</Check>

<Note>
Taking an action may automatically change notification status to read, and action handlers can return redirect URLs for frontend navigation.
</Note>

- Action results are stored in the recipient's `actionsTaken` array
- Handlers receive context about the notification and user
- Multiple actions can be taken on the same notification

### Expiration Management

- Expired notifications cannot have actions taken
- Cleanup jobs automatically remove expired notifications and their deliveries
- Action handlers receive `onExpire` callback for cleanup logic
- Expiration is checked before any action execution

## Patterns and Conventions

### Notification Types

<Tip>
All notification types are enumerated in `NotificationType` enum and registered in `NotificationTypeRegistry` with full configuration including channel preferences and action definitions.
</Tip>

### Channel Implementation

All notification channels follow a consistent pattern:

```typescript
interface NotificationChannel {
  getName(): string;
  canHandle(channel: Channel): boolean;
  send(notification: NotificationRecipient, content: NotificationContent): Promise<void>;
}
```

Channels are:
- Auto-discovered and registered at startup
- Easily extensible for new delivery methods (SMS, Push, etc.)
- Independently configurable per user

### Action Handlers

Action handlers provide the business logic for notification interactions:

```typescript
@NotificationActionHandler({ name: 'approve-lead' })
class LeadApprovalHandler extends BaseNotificationHandler {
  async onCreate(notification: NotificationEvent): Promise<void> {
    // Setup logic when notification is created
  }
  
  async onAction(recipient: NotificationRecipient, actionKey: string, data?: any): Promise<ActionResult> {
    // Handle action execution
  }
  
  async onExpire(notification: NotificationEvent): Promise<void> {
    // Cleanup when notification expires
  }
}
```

<Steps>
<Step title="Discovery">
Handlers are auto-discovered via `NotificationHandlerExplorer` using decorators
</Step>
<Step title="Lifecycle">
Handlers provide lifecycle hooks for creation, action, and expiration events
</Step>
<Step title="Registration">
Handlers are registered with notification types for automatic invocation
</Step>
</Steps>

### Event Naming Conventions

- **Domain-prefixed**: `crm.lead-assigned`, `rbac.team-member-added`
- **Kebab-case**: Consistent lowercase with hyphens for readability
- **Centralized**: Defined in domain-specific event files for organization
- **Descriptive**: Clear indication of the triggering action or state change

### Error Handling Strategy

<Warning>
Delivery failures are logged but don't break the notification flow. Invalid action attempts return appropriate error responses with context.
</Warning>

- Handler errors are caught and logged with full context
- Delivery retries continue even after individual channel failures
- User experience remains uninterrupted by backend notification issues
- Comprehensive error tracking enables debugging and monitoring
