The notification module provides a comprehensive system for managing and delivering notifications across multiple channels within the PropWise CRM. It supports in-app and email delivery channels, user preferences management, action-based notifications with handlers, and event-driven notification creation from domain events.

<Info>
Last Updated: 2024-12-19 | Status: Active
</Info>

The module follows an event-driven architecture where domain modules emit events that are captured by notification listeners, which then create appropriate notifications based on registered notification types and user preferences.

### Design Decisions

<AccordionGroup>
  <Accordion title="Multi-Channel Architecture">
    Notifications can be delivered via multiple channels simultaneously based on user preferences. This allows users to receive notifications through their preferred communication method while maintaining consistency across channels.
  </Accordion>
  
  <Accordion title="Recipient-Centric Design">
    Each user gets their own recipient record with independent status tracking. This enables per-user notification states without affecting other recipients of the same notification event.
  </Accordion>
  
  <Accordion title="Action-Driven Workflow">
    Notifications can include actions that trigger business logic when clicked. This transforms notifications from passive information displays into interactive workflow triggers.
  </Accordion>
  
  <Accordion title="Event-Driven Creation">
    Domain events automatically trigger notification creation through listeners. This decouples notification logic from business logic while ensuring consistent notification delivery.
  </Accordion>
  
  <Accordion title="Retry Mechanism">
    Failed deliveries are retried with exponential backoff to handle temporary delivery issues without overwhelming external services.
  </Accordion>
  
  <Accordion title="Organization Scoping">
    All notifications are scoped to organizations for multi-tenancy, ensuring proper data isolation and security.
  </Accordion>
</AccordionGroup>

## Entities

### NotificationEvent

Represents the core notification record that can target multiple recipients.

```typescript
{
  id: string;                           // UUID primary key
  type: NotificationType;               // LEAD_ASSIGNED, TEAM_MEMBER_ADDED, etc.
  category: NotificationCategory;       // CRM, REAL_ESTATE, RBAC, MESSAGING
  priority: NotificationPriority;       // LOW, NORMAL, HIGH, URGENT
  title: string;                        // Display title
  message: string;                      // Notification content
  data: Record<string, any>;            // Additional contextual data
  organization: Organization;           // Organization scope
  triggeredBy: User;                    // User who caused the notification
  expiresAt?: Date;                     // Optional expiry date
  createdAt: Date;
  updatedAt: Date;
  recipients: Collection<NotificationRecipient>;
}
```

<Note>
The `data` field contains contextual information specific to the notification type, such as lead IDs, property details, or team information.
</Note>

### NotificationRecipient

Per-user instance of a notification with status and action tracking.

```typescript
{
  id: string;                           // UUID primary key
  notificationEvent: NotificationEvent; // Parent notification
  user: User;                          // Target user
  status: NotificationStatus;          // UNREAD, READ, DISMISSED
  readAt?: Date;                       // When marked as read
  dismissedAt?: Date;                  // When dismissed
  actionsTaken: ActionResult[];        // History of actions performed
  createdAt: Date;
  updatedAt: Date;
  deliveries: Collection<NotificationDelivery>;
}
```

### NotificationDelivery

Tracks delivery attempts per channel with retry logic.

```typescript
{
  id: string;                           // UUID primary key
  notificationRecipient: NotificationRecipient;
  channel: Channel;                     // EMAIL, IN_APP
  status: DeliveryStatus;              // PENDING, SENT, FAILED, RETRYING
  sentAt?: Date;                       // Successful delivery timestamp
  failedAt?: Date;                     // Last failure timestamp
  error?: string;                      // Last error message
  retryCount: number;                  // Number of retry attempts
  nextRetryAt?: Date;                  // Next retry schedule
  createdAt: Date;
  updatedAt: Date;
}
```

<Warning>
After 5 failed delivery attempts, the delivery is marked as permanently failed and no further retries are attempted.
</Warning>

### NotificationPreference

User preferences for notification types and channels per organization.

```typescript
{
  id: string;                           // UUID primary key
  user: User;                          // Target user
  organization: Organization;           // Organization scope
  muteAll: boolean;                    // Global mute flag
  channelPreferences: Record<Channel, boolean>; // Per-channel preferences
  typePreferences: Record<NotificationType, {
    enabled: boolean;
    channels: Channel[];
  }>;                                  // Per-type channel preferences
  createdAt: Date;
  updatedAt: Date;
}
```

## API Endpoints

### Get Notifications

Retrieve notifications for the current user with optional filtering.

```http
GET /notifications?page=1&limit=20&status=UNREAD
```

**Query Parameters:**
- `page` (optional): Page number for pagination
- `limit` (optional): Number of items per page
- `status` (optional): Filter by notification status

**Response:**
```typescript
{
  items: NotificationDto[];
  total: number;
  page: number;
  limit: number;
}
```

### Mark as Read

Mark a specific notification as read for the current user.

```http
PATCH /notifications/:id/read
```

**Response:**
```typescript
{
  success: boolean;
  redirectUrl?: string;
}
```

<Tip>
The `redirectUrl` field may contain a URL to redirect the user to relevant content related to the notification.
</Tip>

### Dismiss Notification

Remove a notification from the user's notification list.

```http
DELETE /notifications/:id
```

**Response:**
```typescript
{
  success: boolean;
}
```

### Take Action

Execute a specific action associated with a notification.

```http
POST /notifications/:id/actions/:actionKey
```

**Request Body:**
```typescript
{
  data?: Record<string, any>;
}
```

**Response:**
```typescript
{
  success: boolean;
  message?: string;
  redirectUrl?: string;
  result?: any;
}
```

### Get Preferences

Retrieve notification preferences for the current user in their organization.

```http
GET /notifications/preferences
```

### Update Preferences

Update notification preferences for the current user.

```http
PUT /notifications/preferences
```

**Request Body:**
```typescript
{
  muteAll?: boolean;
  channelPreferences?: Record<Channel, boolean>;
  typePreferences?: Record<NotificationType, {
    enabled: boolean;
    channels: Channel[];
  }>;
}
```

## Business Rules

### Notification Creation

<Steps>
  <Step title="Access Verification">
    Notifications are created only for users with appropriate access to the related entity. This ensures users don't receive notifications for resources they cannot view.
  </Step>
  
  <Step title="Preference Check">
    User preferences are checked before creating notifications. If a user has disabled a notification type or channel, those notifications will not be created.
  </Step>
  
  <Step title="Mute All Check">
    If `muteAll` is enabled in user preferences, no notifications are created for that user regardless of other settings.
  </Step>
  
  <Step title="Channel Selection">
    Only enabled channels from user preferences are used for delivery attempts.
  </Step>
</Steps>

### Delivery Logic

<Note>
The delivery system operates independently for each channel, allowing for different success/failure states per delivery method.
</Note>

1. **In-app notifications** are always created regardless of delivery success
2. **Email deliveries** are attempted based on user preferences
3. **Failed deliveries** are retried with exponential backoff: 1min → 5min → 30min → 2hr → 12hr
4. **Permanent failure** occurs after 5 failed attempts

### Action Handling

<Warning>
Actions can only be taken on notifications in UNREAD or READ status. Dismissed notifications cannot have actions performed.
</Warning>

- Taking an action may automatically change notification status to read
- Action handlers can return redirect URLs for frontend navigation
- Action results are stored in the recipient's `actionsTaken` array for audit purposes

### Expiration

<Check>
Expired notifications are automatically handled by the cleanup system to prevent resource accumulation.
</Check>

- Expired notifications cannot have actions taken
- Cleanup job removes expired notifications and their deliveries
- Action handlers receive `onExpire` callback for cleanup logic

## Patterns and Conventions

### Notification Types

<CodeGroup>
```typescript Notification Type Definition
enum NotificationType {
  LEAD_ASSIGNED = 'LEAD_ASSIGNED',
  TEAM_MEMBER_ADDED = 'TEAM_MEMBER_ADDED',
  PROPERTY_VIEWED = 'PROPERTY_VIEWED',
  // ... additional types
}
```

```typescript Type Registry
const NotificationTypeRegistry = {
  [NotificationType.LEAD_ASSIGNED]: {
    channels: [Channel.EMAIL, Channel.IN_APP],
    actions: ['view_lead', 'assign_agent'],
    handler: 'LeadNotificationHandler'
  }
  // ... additional configurations
}
```
</CodeGroup>

- Enumerated in `NotificationType` enum
- Registered in `NotificationTypeRegistry` with configuration
- Include channel preferences, action definitions, and handler mappings

### Channel Implementation

All notification channels follow a consistent interface pattern:

```typescript
interface NotificationChannel {
  send(notification: NotificationRecipient): Promise<DeliveryResult>;
  supports(type: NotificationType): boolean;
  getName(): string;
}
```

<Tip>
Channels are auto-discovered and registered at startup, making it easy to add new delivery methods like SMS or push notifications.
</Tip>

### Action Handlers

Action handlers provide the business logic for notification interactions:

```typescript
@NotificationActionHandler({ name: 'LeadNotificationHandler' })
class LeadNotificationHandler extends BaseNotificationHandler {
  async onCreate(notification: NotificationRecipient): Promise<void> {
    // Logic when notification is created
  }
  
  async onAction(action: string, data: any): Promise<ActionResult> {
    // Logic when action is taken
  }
  
  async onExpire(notification: NotificationRecipient): Promise<void> {
    // Cleanup logic when notification expires
  }
}
```

- Extend `BaseNotificationHandler` abstract class
- Decorated with `@NotificationActionHandler({ name })`
- Auto-discovered via `NotificationHandlerExplorer`
- Provide lifecycle hooks: `onCreate`, `onAction`, `onExpire`

### Event Naming Convention

<Info>
Consistent event naming ensures clear understanding and easy debugging of the notification flow.
</Info>

- **Domain-prefixed**: `crm.lead-assigned`, `rbac.team-member-added`
- **Kebab-case** for consistency across the application
- **Centralized** in domain-specific event files

### Error Handling

The notification system implements robust error handling:

- **Delivery failures** are logged but don't break the notification flow
- **Invalid action attempts** return appropriate error responses
- **Handler errors** are caught and logged with full context
- **Retry logic** handles transient failures automatically

<Warning>
While the system is designed to be fault-tolerant, monitoring delivery success rates is recommended to identify systemic issues.
</Warning>
