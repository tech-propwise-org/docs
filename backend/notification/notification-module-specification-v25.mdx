## Overview

The notification module provides a comprehensive system for managing and delivering notifications across multiple channels within the PropWise CRM. It supports in-app and email delivery channels, user preferences management, action-based notifications with handlers, and event-driven notification creation from domain events.

<Info>
The module follows an event-driven architecture where domain modules emit events that are captured by notification listeners, which then create appropriate notifications based on registered notification types and user preferences.
</Info>

### Design Decisions

<AccordionGroup>
  <Accordion title="Multi-Channel Architecture">
    Notifications can be delivered via multiple channels simultaneously based on user preferences. This allows users to receive notifications through their preferred communication methods.
  </Accordion>
  
  <Accordion title="Recipient-Centric Design">
    Each user gets their own recipient record with independent status tracking. This enables personalized notification states and actions per user.
  </Accordion>
  
  <Accordion title="Action-Driven Workflow">
    Notifications can include actions that trigger business logic when clicked. This creates interactive notifications that can perform operations directly from the notification interface.
  </Accordion>
  
  <Accordion title="Event-Driven Creation">
    Domain events automatically trigger notification creation through listeners. This decouples notification creation from business logic and ensures consistent notification behavior.
  </Accordion>
  
  <Accordion title="Organization Scoping">
    All notifications are scoped to organizations for multi-tenancy. This ensures proper data isolation in multi-tenant environments.
  </Accordion>
</AccordionGroup>

## Entities

### NotificationEvent

Represents the core notification record that can target multiple recipients.

```typescript
{
  id: string;                           // UUID primary key
  type: NotificationType;               // LEAD_ASSIGNED, TEAM_MEMBER_ADDED, etc.
  category: NotificationCategory;       // CRM, REAL_ESTATE, RBAC, MESSAGING
  priority: NotificationPriority;       // LOW, NORMAL, HIGH, URGENT
  title: string;                        // Display title
  message: string;                      // Notification content
  data: Record<string, any>;            // Additional contextual data
  organization: Organization;           // Organization scope
  triggeredBy: User;                    // User who caused the notification
  expiresAt?: Date;                     // Optional expiry date
  createdAt: Date;
  updatedAt: Date;
  recipients: Collection<NotificationRecipient>;
}
```

### NotificationRecipient

Per-user instance of a notification with status and action tracking.

```typescript
{
  id: string;                           // UUID primary key
  notificationEvent: NotificationEvent; // Parent notification
  user: User;                          // Target user
  status: NotificationStatus;          // UNREAD, READ, DISMISSED
  readAt?: Date;                       // When marked as read
  dismissedAt?: Date;                  // When dismissed
  actionsTaken: ActionResult[];        // History of actions performed
  createdAt: Date;
  updatedAt: Date;
  deliveries: Collection<NotificationDelivery>;
}
```

### NotificationDelivery

Tracks delivery attempts per channel with retry logic.

```typescript
{
  id: string;                           // UUID primary key
  notificationRecipient: NotificationRecipient;
  channel: Channel;                     // EMAIL, IN_APP
  status: DeliveryStatus;              // PENDING, SENT, FAILED, RETRYING
  sentAt?: Date;                       // Successful delivery timestamp
  failedAt?: Date;                     // Last failure timestamp
  error?: string;                      // Last error message
  retryCount: number;                  // Number of retry attempts
  nextRetryAt?: Date;                  // Next retry schedule
  createdAt: Date;
  updatedAt: Date;
}
```

### NotificationPreference

User preferences for notification types and channels per organization.

```typescript
{
  id: string;                           // UUID primary key
  user: User;                          // Target user
  organization: Organization;           // Organization scope
  muteAll: boolean;                    // Global mute flag
  channelPreferences: Record<Channel, boolean>; // Per-channel preferences
  typePreferences: Record<NotificationType, {
    enabled: boolean;
    channels: Channel[];
  }>;                                  // Per-type channel preferences
  createdAt: Date;
  updatedAt: Date;
}
```

## API Endpoints

### Get Notifications

Retrieve notifications for the current user with pagination and filtering options.

```http
GET /notifications?page=1&limit=20&status=UNREAD
```

**Response:**
```typescript
{
  items: NotificationDto[];
  total: number;
  page: number;
  limit: number;
}
```

### Mark as Read

Mark a notification as read and potentially receive a redirect URL.

```http
PATCH /notifications/:id/read
```

**Response:**
```typescript
{
  success: boolean;
  redirectUrl?: string;
}
```

### Dismiss Notification

Remove a notification from the user's notification list.

```http
DELETE /notifications/:id
```

**Response:**
```typescript
{
  success: boolean;
}
```

### Take Action

Execute a specific action associated with a notification.

```http
POST /notifications/:id/actions/:actionKey
```

**Request Body:**
```typescript
{
  data?: Record<string, any>;
}
```

**Response:**
```typescript
{
  success: boolean;
  message?: string;
  redirectUrl?: string;
  result?: any;
}
```

## Business Rules

### Notification Creation

<Steps>
  <Step title="Access Validation">
    Notifications are created only for users with appropriate access to the related entity
  </Step>
  <Step title="Preference Check">
    User preferences are checked before creating notifications
  </Step>
  <Step title="Mute Validation">
    If `muteAll` is enabled, no notifications are created
  </Step>
  <Step title="Channel Selection">
    Only enabled channels from preferences are used for delivery
  </Step>
</Steps>

### Delivery Logic

<Warning>
Failed deliveries are retried with exponential backoff intervals: 1 minute, 5 minutes, 30 minutes, 2 hours, and 12 hours. After 5 failed attempts, delivery is marked as permanently failed.
</Warning>

1. **In-app notifications** are always created regardless of delivery success
2. **Email deliveries** are attempted based on user preferences
3. **Retry mechanism** uses exponential backoff for failed deliveries
4. **Permanent failure** occurs after 5 consecutive failed attempts

### Action Handling

<Check>
Actions can only be taken on notifications in UNREAD or READ status
</Check>

- Taking an action may automatically change notification status to read
- Action handlers can return redirect URLs for frontend navigation
- Action results are stored in the recipient's `actionsTaken` array for audit purposes

### Expiration

<Info>
Expired notifications cannot have actions taken on them, and a cleanup job periodically removes expired notifications and their deliveries.
</Info>

- Action handlers receive `onExpire` callback for cleanup logic
- Expired notifications are automatically cleaned up by background jobs

## Patterns and Conventions

### Notification Types

All notification types follow a consistent registration pattern:

```typescript
// Enumerated in NotificationType enum
enum NotificationType {
  LEAD_ASSIGNED = 'LEAD_ASSIGNED',
  TEAM_MEMBER_ADDED = 'TEAM_MEMBER_ADDED',
  // ... other types
}

// Registered in NotificationTypeRegistry with configuration
NotificationTypeRegistry.register({
  type: NotificationType.LEAD_ASSIGNED,
  channels: [Channel.EMAIL, Channel.IN_APP],
  actions: ['view-lead', 'assign-to-self'],
  handler: 'LeadNotificationHandler'
});
```

### Channel Implementation

<Tip>
All channels implement the `NotificationChannel` interface, making it easy to add new channels like SMS or push notifications.
</Tip>

```typescript
interface NotificationChannel {
  deliver(recipient: NotificationRecipient): Promise<void>;
  supports(type: NotificationType): boolean;
}
```

### Action Handlers

Action handlers follow a decorator-based pattern for auto-discovery:

```typescript
@NotificationActionHandler({ name: 'LeadNotificationHandler' })
export class LeadNotificationHandler extends BaseNotificationHandler {
  async onCreate(recipient: NotificationRecipient): Promise<void> {
    // Called when notification is created
  }
  
  async onAction(recipient: NotificationRecipient, action: string, data?: any): Promise<ActionResult> {
    // Called when user takes an action
  }
  
  async onExpire(recipient: NotificationRecipient): Promise<void> {
    // Called when notification expires
  }
}
```

### Event Naming Conventions

Events follow a consistent naming pattern across all domains:

- **Domain-prefixed**: `crm.lead-assigned`, `rbac.team-member-added`
- **Kebab-case**: All lowercase with hyphens for word separation
- **Centralized**: Defined in domain-specific event files for maintainability

### Error Handling

<Warning>
The system is designed to be fault-tolerant. Delivery failures are logged but don't break the notification flow, and handler errors are caught and logged with context.
</Warning>

- Invalid action attempts return appropriate error responses
- Failed deliveries are retried automatically
- System continues operating even if individual notifications fail
