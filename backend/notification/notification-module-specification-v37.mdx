## Overview

The notification module provides a comprehensive system for managing and delivering notifications across multiple channels within the PropWise CRM. It supports in-app and email delivery channels, user preferences management, action-based notifications with handlers, and event-driven notification creation from domain events.

The module follows an event-driven architecture where domain modules emit events that are captured by notification listeners, which then create appropriate notifications based on registered notification types and user preferences.

### Design decisions

<AccordionGroup>
  <Accordion title="Multi-Channel Architecture">
    Notifications can be delivered via multiple channels simultaneously based on user preferences
  </Accordion>
  
  <Accordion title="Recipient-Centric Design">
    Each user gets their own recipient record with independent status tracking
  </Accordion>
  
  <Accordion title="Action-Driven Workflow">
    Notifications can include actions that trigger business logic when clicked
  </Accordion>
  
  <Accordion title="Event-Driven Creation">
    Domain events automatically trigger notification creation through listeners
  </Accordion>
  
  <Accordion title="Retry Mechanism">
    Failed deliveries are retried with exponential backoff
  </Accordion>
  
  <Accordion title="Organization Scoping">
    All notifications are scoped to organizations for multi-tenancy
  </Accordion>
  
  <Accordion title="Generic Grouping/Deduplication">
    Any notification type can opt into windowed aggregation (one digest notification + one digest email per window) or collapse/throttle, configured declaratively per type
  </Accordion>

## Entities

### NotificationEvent

Represents the core notification record that can target multiple recipients.

```typescript
{
  id: string;                           // UUID primary key
  type: NotificationType;               // LEAD_ASSIGNED, TEAM_MEMBER_ADDED, etc.
  category: NotificationCategory;       // crm, properties, access, document, system, messaging
  priority: NotificationPriority;       // LOW, NORMAL, HIGH, URGENT
  title: string;                        // Display title
  message: string;                      // Notification content
  data: Record<string, any>;            // Additional contextual data
  organization: Organization;           // Organization scope
  triggeredBy: User;                    // User who caused the notification
  expiresAt?: Date;                     // Optional expiry date
  createdAt: Date;
  updatedAt: Date;
  recipients: Collection<NotificationRecipient>;
}
```

<Info>
The `NotificationEvent` entity serves as the parent record for all notification recipients, enabling efficient multi-recipient notifications while maintaining per-user state tracking.
</Info>

### NotificationRecipient

Per-user instance of a notification with status and action tracking.

```typescript
{
  id: string;                           // UUID primary key
  notificationEvent: NotificationEvent; // Parent notification
  user: User;                          // Target user
  status: NotificationStatus;          // UNREAD, READ, DISMISSED
  readAt?: Date;                       // When marked as read
  dismissedAt?: Date;                  // When dismissed
  actionsTaken: ActionResult[];        // History of actions performed
  createdAt: Date;
  updatedAt: Date;
  deliveries: Collection<NotificationDelivery>;
}
```

### NotificationDelivery

Tracks delivery attempts per channel with retry logic.

```typescript
{
  id: string;                           // UUID primary key
  notificationRecipient: NotificationRecipient;
  channel: Channel;                     // EMAIL, IN_APP
  status: DeliveryStatus;              // PENDING, SENT, FAILED, RETRYING
  sentAt?: Date;                       // Successful delivery timestamp
  failedAt?: Date;                     // Last failure timestamp
  error?: string;                      // Last error message
  retryCount: number;                  // Number of retry attempts
  nextRetryAt?: Date;                  // Next retry schedule
  createdAt: Date;
  updatedAt: Date;
}
```

<Warning>
Failed deliveries are automatically retried with exponential backoff. Ensure proper error handling and monitoring to prevent notification delivery issues.
</Warning>

### NotificationPreference

User preferences for notification types and channels per organization.

```typescript
{
  id: string; // UUID primary key
  user: User; // Target user
  organization: Organization; // Organization scope
  channelPreferences: Record<Channel, boolean>; // Per-channel preferences
  typePreferences: Record<
    NotificationType,
    {
      enabled: boolean;
      channels: Channel[];
    }
  >; // Per-type channel preferences
  createdAt: Date;
  updatedAt: Date;
}
```

## Grouping & deduplication

A generic engine lets **any** notification type collapse a burst of related notifications into a single aggregated in-app notification + a single digest email, instead of N separate pings. It is configured declaratively per type via a `grouping` block on the type's registry config — no bespoke per-type code is required.

<Tip>
**Engine status:** Built and active. Legacy ad-hoc deduplication implementations listed in `NOTIFICATION_CATALOG.md` are scheduled to migrate onto this unified engine.
</Tip>

### How it works

<Steps>
  <Step title="Detect">
    When a grouped type is created, a deterministic **group key** is computed from `type + org-scope + recipientId + configured payload fields`. The field segment is **JSON-encoded** (a JSON array of the stringified field values), not a raw `|` join, so a payload value containing a `:` / `|` delimiter cannot collide across different field splits. Notifications sharing a key for a recipient land in the same window.
  </Step>
  
  <Step title="Collect">
    The first notification opens a `NotificationGroup` window (and inserts a `NotificationGroupItem`); subsequent matching notifications append items. The user sees nothing yet (`hold` mode) — or the first one is delivered immediately and the rest fold in (`emit-first-immediately` mode).
  </Step>
  
  <Step title="Flush">
    A pg-boss job fires when the window closes. The worker aggregates the collected items, renders one aggregated notification, and creates it through the **normal** create path (so preferences/mute, channels, and delivery are handled identically to any other notification).
  </Step>
</Steps>

### Window semantics

<Tabs>
  <Tab title="Fixed (Tumbling)">
    Opens at the first item, flushes a fixed duration later. Predictable latency — used by the `activity_*` pilot (15-minute digest).
  </Tab>
  
  <Tab title="Sliding (Debounce)">
    Each new item pushes the flush time forward, hard-capped by a max duration. Best for burst suppression.
  </Tab>
  
  <Tab title="Early Flush">
    An optional `maxGroupSize` flushes immediately once reached, so a flood never produces an unbounded digest. On early flush the window's `flushAt` is pulled back to **now** (matching the immediate job that is enqueued), so if that job is lost the recovery cron re-enqueues it ~2 min later instead of waiting out the original (future) window end.
  </Tab>
  
  <Tab title="Window Clamp">
    A resolved window duration (`resolveWindowMs` or the static config) is clamped to `[MIN_GROUP_WINDOW_MS, MAX_GROUP_WINDOW_MS]` (1 ms … 7 days, in `notification-group.constants.ts`) before it is snapshotted, so an oversized value can never overflow the `int4` `window_ms` column on insert.
  </Tab>
</Tabs>

### Modes

<CardGroup cols={2}>
  <Card title="Digest" icon="list">
    Render an itemized list ("7 activity updates"); `payload.items` carries each item for the in-app list + digest email, each with its own deep-link.
  </Card>
  
  <Card title="Collapse" icon="compress">
    Render a single representative + count ("Seat limit reached (+3 more)"); no item list needed (`collectItems: false`).
  </Card>

### Entities (grouping)

#### NotificationGroup

The open aggregation-window state — one row per `(recipient, type, groupKey, org-scope)` while live.

```typescript
{
  id: string;
  organization?: Organization;       // NULL reserved for tenantless grouping
  recipient: User;                   // windows are always per-recipient
  type: NotificationType;
  groupKey: string;                  // detection discriminator (JSON-encoded field segment)
  status: NotificationGroupStatus;   // OPEN | FLUSHED | CANCELLED (FLUSHING reserved)
  windowKind: GroupWindowKind;       // FIXED | SLIDING (snapshotted at open)
  openedAt: Date;
  flushAt: Date;                     // next scheduled flush (advances for SLIDING)
  maxAt: Date;                       // hard ceiling for SLIDING
  itemCount: number;                 // running count (drives maxGroupSize + collapse)
  scheduledJobId?: string;           // pg-boss flush job id
  firstEventId?: string;             // emit-first-immediately: event updated in place at flush
  flushedEventId?: string;           // set on flush — double-flush idempotency guard
}
```

#### NotificationGroupItem

Insert-only collected item inside a window (skipped for count-only collapse).

```typescript
{
  id: string;
  group: NotificationGroup;          // FK, cascade delete
  title?: string;
  message?: string;
  payload: Record<string, any>;      // per-item snapshot for rendering + per-item deep-links
  senderId?: string;
  priority?: NotificationPriority;   // flush picks the highest as the grouped priority
}
```

### Concurrency & correctness

<AccordionGroup>
  <Accordion title="Single Live Window Per Scope">
    A **partial unique index** over `(recipient, type, groupKey, COALESCE(org, sentinel)) WHERE status = 'open'` enforces "one live window per scope". Concurrent openers race-recover via the same optimistic-insert + unique-conflict pattern as `idempotencyKey`.
  </Accordion>
  
  <Accordion title="Insert-Only Collection">
    Item collection is an **insert-only child table** (not a JSONB array) so high-fan-in windows don't serialize on a single row lock.
  </Accordion>
  
  <Accordion title="Double-Flush Prevention">
    **Double-flush is impossible**: per-window pg-boss `singletonKey` (policy `stately`) + a `PESSIMISTIC_WRITE` lock at flush + a terminal-status guard + the persisted `flushedEventId`.
  </Accordion>
  
  <Accordion title="Recovery Mechanism">
    A **recovery cron** (`notification-group-recovery`, every 10 min) re-enqueues windows whose flush job was lost or is overdue (mirrors `reminder-recovery`). The orphan scan re-picks an OPEN window only when `scheduled_job_id IS NULL` **or** `flush_at < NOW() - INTERVAL '2 minutes'`, so every flush path must keep `flush_at` aligned with the actually-enqueued job time — including **early flush**, which sets `flush_at = now` (otherwise a lost early-flush job would hide behind a still-future `flush_at` and stall the digest by up to the full window length).
    
    The orphan scan also filters `organization_id IS NOT NULL` because the engine is **org-scoped** — `append()` always opens windows with a real org and recovery re-enqueues via `executeInOrg(orgId)`; tenantless grouping is unsupported, so no tenantless window is ever stranded.
  </Accordion>
  
  <Accordion title="Abandoned Window Guard">
    **CANCELLED status**: If the org (or recipient) is deleted between a window opening and its flush, `createNotificationInTransaction` throws `NotFoundException` at flush. The flush worker catches it and marks the window as CANCELLED, preventing zombie windows from accumulating.
  </Accordion>
</AccordionGroup>

<Check>
The grouping engine provides strong consistency guarantees through database-level constraints, optimistic locking patterns, and comprehensive recovery mechanisms. All edge cases around concurrent operations, failed jobs, and deleted entities are handled gracefully.
</Check>
