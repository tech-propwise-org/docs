The notification module provides a comprehensive system for creating, dispatching, and managing notifications across the PropWise CRM platform. It supports multi-channel delivery (in-app, email), user preferences, action handling, and automated retry mechanisms.

<Note>
The module follows an event-driven architecture where domain events from other modules (CRM, Real Estate, RBAC) trigger notification creation through event listeners.
</Note>

## Architecture

### Core Components

The notification system is built around six key components that work together to provide reliable, multi-channel notification delivery:

### Design Decisions

<AccordionGroup>
  <Accordion title="Multi-tenant Isolation">
    All notifications are scoped to organizations, ensuring complete data isolation between tenants.
  </Accordion>
  
  <Accordion title="Channel Abstraction">
    Pluggable channel system supporting multiple delivery methods with consistent interfaces.
  </Accordion>
  
  <Accordion title="Action-based Workflow">
    Notifications can include interactive actions (approve/reject/etc.) for streamlined workflows.
  </Accordion>
  
  <Accordion title="Retry Mechanism">
    Failed deliveries are automatically retried with exponential backoff for improved reliability.
  </Accordion>
  
  <Accordion title="User Preferences">
    Fine-grained control over notification types and channels per user and organization.
  </Accordion>
  
  <Accordion title="Event-driven">
    Decoupled from domain logic through event emitter pattern for better maintainability.
  </Accordion>
</AccordionGroup>

## Core Entities

### NotificationEvent

Core notification entity representing a notification template/type.

```typescript
interface NotificationEvent {
  id: string;
  type: NotificationType;
  category: NotificationCategory;
  priority: NotificationPriority;
  title: string;
  message: string;
  data?: Record<string, any>;
  actionData?: Record<string, any>;
  expiresAt?: Date;
  createdAt: Date;
  updatedAt: Date;
  triggeredBy: User;
  organization: Organization;
  recipients: Collection<NotificationRecipient>;
}
```

### NotificationRecipient

Links users to notification events with individual status tracking.

```typescript
interface NotificationRecipient {
  id: string;
  user: User;
  event: NotificationEvent;
  status: NotificationStatus; // PENDING, READ, DISMISSED, EXPIRED
  readAt?: Date;
  dismissedAt?: Date;
  actionResults: ActionResult[];
  createdAt: Date;
  updatedAt: Date;
  deliveries: Collection<NotificationDelivery>;
}
```

### NotificationDelivery

Tracks delivery attempts for each channel.

```typescript
interface NotificationDelivery {
  id: string;
  notificationRecipient: NotificationRecipient;
  channel: Channel; // IN_APP, EMAIL
  status: DeliveryStatus; // PENDING, SENT, DELIVERED, FAILED, RETRYING
  attempts: number;
  lastAttemptAt?: Date;
  nextRetryAt?: Date;
  deliveredAt?: Date;
  failureReason?: string;
  createdAt: Date;
  updatedAt: Date;
}
```

### NotificationPreference

Per-user, per-organization notification preferences.

```typescript
interface NotificationPreference {
  id: string;
  user: User;
  organization: Organization;
  muteAll: boolean;
  channelPreferences: Record<Channel, boolean>;
  typePreferences: Record<NotificationType, boolean>;
  createdAt: Date;
  updatedAt: Date;
}
```

## Business Rules and Invariants

### Notification Creation Rules

<Check>All notifications must be associated with an organization</Check>
<Check>Notifications can have multiple recipients within the same organization</Check>
<Check>Action data is preserved separately from display data for security</Check>
<Check>Expiration dates are optional but enforced if set</Check>

### Delivery Rules

<Steps>
  <Step title="Priority Delivery">
    In-app notifications are always attempted first for immediate visibility
  </Step>
  
  <Step title="Email Delivery">
    Email delivery respects user preferences and organization settings
  </Step>
  
  <Step title="Retry Logic">
    Failed deliveries are retried up to 5 times with exponential backoff
  </Step>
  
  <Step title="Timeout Handling">
    Deliveries stop retrying after 24 hours to prevent infinite loops
  </Step>
</Steps>

### User Preference Rules

<Warning>
`muteAll = true` blocks all notifications except critical system alerts that cannot be disabled for security reasons.
</Warning>

- **Channel Override**: Channel preferences override type preferences (email disabled = no email regardless of type)
- **Default Settings**: New users get default preferences based on organization settings  
- **Organization Scope**: Preferences are per-organization (users can have different settings per org)

### Action Handling Rules

<Info>
Actions provide interactive capabilities within notifications, enabling users to respond directly without navigating to separate pages.
</Info>

- Actions can only be performed **once** per notification recipient
- Action results are **immutable** once recorded
- **Expired** notifications cannot have actions performed
- Action handlers can return **redirect URLs** for frontend navigation

## Implementation Patterns

### Event Listeners

Use the `@OnEvent()` decorator with specific event names and implement proper error handling:

```typescript
@OnEvent('crm.lead-assigned')
async handleLeadAssigned(event: LeadAssignedEvent) {
  try {
    await this.notificationService.create({
      type: NotificationType.LEAD_ASSIGNED,
      organizationId: event.organizationId,
      recipientIds: [event.assignedUserId],
      data: { leadId: event.leadId }
    });
  } catch (error) {
    // Log error but don't block event processing
    this.logger.error('Failed to create notification', error);
  }
}
```

### Channel Implementation

<Tip>
Channels should implement graceful degradation and comprehensive error handling for external service failures.
</Tip>

Implement the `NotificationChannel` interface with proper dependency injection:

```typescript
@Injectable()
export class EmailChannel implements NotificationChannel {
  async deliver(delivery: NotificationDelivery): Promise<DeliveryResult> {
    try {
      await this.emailService.send({
        to: delivery.notificationRecipient.user.email,
        template: this.getTemplate(delivery),
        data: delivery.notificationRecipient.event.data
      });
      
      return { success: true };
    } catch (error) {
      return { 
        success: false, 
        error: error.message,
        retryable: this.isRetryableError(error)
      };
    }
  }
}
```

### Action Handlers

Extend `BaseNotificationHandler` and use the `@NotificationActionHandler()` decorator:

```typescript
@NotificationActionHandler(NotificationType.LEAD_APPROVAL)
export class LeadApprovalHandler extends BaseNotificationHandler {
  // Validation during notification creation
  onCreate(event: NotificationEvent): boolean {
    return event.data?.leadId && event.data?.requesterId;
  }

  // Handle user actions
  async onAction(
    action: string, 
    data: any, 
    recipient: NotificationRecipient
  ): Promise<ActionResult> {
    if (action === 'approve') {
      await this.leadService.approve(
        recipient.event.data.leadId, 
        data.comment
      );
      
      return {
        success: true,
        message: 'Lead approved successfully',
        redirectUrl: `/leads/${recipient.event.data.leadId}`
      };
    }
    
    throw new BadRequestException('Invalid action');
  }

  // Cleanup for expired notifications
  async onExpire(recipient: NotificationRecipient): Promise<void> {
    // Handle time-sensitive cleanup
    await this.leadService.handleExpiredApproval(
      recipient.event.data.leadId
    );
  }
}
```

### Type Registry Configuration

The `NotificationTypeRegistry` provides centralized configuration:

```typescript
// Maps notification types to templates, channels, and handlers
registry.register(NotificationType.LEAD_ASSIGNED, {
  channels: [Channel.IN_APP, Channel.EMAIL],
  handler: LeadAssignedHandler,
  actions: ['view', 'accept'],
  defaultExpiration: '7d',
  priority: NotificationPriority.MEDIUM
});
```
