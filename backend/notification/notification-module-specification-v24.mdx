## Overview

The notification module provides a comprehensive system for managing and delivering notifications across multiple channels within the PropWise CRM. It supports in-app and email delivery channels, user preferences management, action-based notifications with handlers, and event-driven notification creation from domain events.

<Info>
The module follows an event-driven architecture where domain modules emit events that are captured by notification listeners, which then create appropriate notifications based on registered notification types and user preferences.
</Info>

**Last Updated:** 2024-12-19  
**Status:** Active

## Architecture

### Core Components

The notification system consists of several key components working together:

- **Notification Events**: Central notification records that can have multiple recipients
- **Notification Recipients**: Per-user instances of notifications with status tracking
- **Notification Deliveries**: Per-channel delivery attempts with retry logic
- **Notification Channels**: Pluggable delivery mechanisms (email, in-app)
- **Action Handlers**: Business logic processors for notification actions
- **Preference System**: User-configurable notification settings per organization

## Entities

### NotificationEvent

Represents the core notification record that can target multiple recipients.

```typescript
{
  id: string;                           // UUID primary key
  type: NotificationType;               // LEAD_ASSIGNED, TEAM_MEMBER_ADDED, etc.
  category: NotificationCategory;       // CRM, REAL_ESTATE, RBAC, MESSAGING
  priority: NotificationPriority;       // LOW, NORMAL, HIGH, URGENT
  title: string;                        // Display title
  message: string;                      // Notification content
  data: Record<string, any>;            // Additional contextual data
  organization: Organization;           // Organization scope
  triggeredBy: User;                    // User who caused the notification
  expiresAt?: Date;                     // Optional expiry date
  createdAt: Date;
  updatedAt: Date;
  recipients: Collection<NotificationRecipient>;
}
```

### NotificationRecipient

Per-user instance of a notification with status and action tracking.

```typescript
{
  id: string;                           // UUID primary key
  notificationEvent: NotificationEvent; // Parent notification
  user: User;                          // Target user
  status: NotificationStatus;          // UNREAD, READ, DISMISSED
  readAt?: Date;                       // When marked as read
  dismissedAt?: Date;                  // When dismissed
  actionsTaken: ActionResult[];        // History of actions performed
  createdAt: Date;
  updatedAt: Date;
  deliveries: Collection<NotificationDelivery>;
}
```

### NotificationDelivery

Tracks delivery attempts per channel with retry logic.

```typescript
{
  id: string;                           // UUID primary key
  notificationRecipient: NotificationRecipient;
  channel: Channel;                     // EMAIL, IN_APP
  status: DeliveryStatus;              // PENDING, SENT, FAILED, RETRYING
  sentAt?: Date;                       // Successful delivery timestamp
  failedAt?: Date;                     // Last failure timestamp
  error?: string;                      // Last error message
  retryCount: number;                  // Number of retry attempts
  nextRetryAt?: Date;                  // Next retry schedule
  createdAt: Date;
  updatedAt: Date;
}
```

### NotificationPreference

User preferences for notification types and channels per organization.

```typescript
{
  id: string;                           // UUID primary key
  user: User;                          // Target user
  organization: Organization;           // Organization scope
  muteAll: boolean;                    // Global mute flag
  channelPreferences: Record<Channel, boolean>; // Per-channel preferences
  typePreferences: Record<NotificationType, {
    enabled: boolean;
    channels: Channel[];
  }>;                                  // Per-type channel preferences
  createdAt: Date;
  updatedAt: Date;
}
```

## API Endpoints

### Get Notifications

Retrieve paginated list of notifications for the current user.

```http
GET /notifications?page=1&limit=20&status=UNREAD
```

**Query Parameters:**
- `page` (optional): Page number (default: 1)
- `limit` (optional): Items per page (default: 20, max: 100)
- `status` (optional): Filter by status (UNREAD, READ, DISMISSED)

**Response:**
```json
{
  "items": [
    {
      "id": "123e4567-e89b-12d3-a456-426614174000",
      "type": "LEAD_ASSIGNED",
      "category": "CRM",
      "priority": "NORMAL",
      "title": "New Lead Assigned",
      "message": "You have been assigned a new lead: John Doe",
      "status": "UNREAD",
      "createdAt": "2024-12-19T10:00:00Z",
      "actions": [
        {
          "key": "view_lead",
          "label": "View Lead",
          "type": "PRIMARY"
        }
      ]
    }
  ],
  "total": 15,
  "page": 1,
  "limit": 20
}
```

### Mark as Read

Mark a notification as read and optionally get redirect URL.

```http
PATCH /notifications/:id/read
```

**Response:**
```json
{
  "success": true,
  "redirectUrl": "/leads/123"
}
```

### Dismiss Notification

Remove a notification from the user's list.

```http
DELETE /notifications/:id
```

**Response:**
```json
{
  "success": true
}
```

### Take Action

Execute a specific action associated with a notification.

```http
POST /notifications/:id/actions/:actionKey
```

**Request Body:**
```json
{
  "data": {
    "customParam": "value"
  }
}
```

**Response:**
```json
{
  "success": true,
  "message": "Action completed successfully",
  "redirectUrl": "/leads/123",
  "result": {
    "updatedField": "newValue"
  }
}
```

### Get Preferences

Retrieve user's notification preferences for the current organization.

```http
GET /notifications/preferences
```

**Response:**
```json
{
  "muteAll": false,
  "channelPreferences": {
    "EMAIL": true,
    "IN_APP": true
  },
  "typePreferences": {
    "LEAD_ASSIGNED": {
      "enabled": true,
      "channels": ["EMAIL", "IN_APP"]
    },
    "TASK_DUE": {
      "enabled": false,
      "channels": []
    }
  }
}
```

### Update Preferences

Update user's notification preferences.

```http
PUT /notifications/preferences
```

**Request Body:**
```json

    "EMAIL": true,

  },

## Business Rules

### Notification Creation

<Steps>
  <Step title="Access Validation">
    Notifications are created only for users with appropriate access to the related entity
  </Step>
  <Step title="Preference Check">
    User preferences are checked before creating notifications
  </Step>
  <Step title="Mute Validation">
    If `muteAll` is enabled, no notifications are created
  </Step>
  <Step title="Channel Selection">
    Only enabled channels from preferences are used for delivery
  </Step>
</Steps>

### Delivery Logic

<Warning>
Failed deliveries are retried with exponential backoff: 1min, 5min, 30min, 2hr, 12hr. After 5 failed attempts, delivery is marked as permanently failed.
</Warning>

1. **In-app notifications** are always created regardless of delivery success
2. **Email deliveries** are attempted based on user preferences
3. **Retry mechanism** uses exponential backoff for failed deliveries
4. **Permanent failure** occurs after 5 failed attempts

### Action Handling

<AccordionGroup>
  <Accordion title="Action Requirements">
    - Actions can only be taken on notifications in UNREAD or READ status
    - Taking an action may change notification status to read
    - Action handlers can return redirect URLs for frontend navigation
    - Action results are stored in the recipient's `actionsTaken` array
  </Accordion>
  
  <Accordion title="Expiration Rules">
    - Expired notifications cannot have actions taken
    - Cleanup job removes expired notifications and their deliveries
    - Action handlers receive `onExpire` callback for cleanup logic
  </Accordion>
</AccordionGroup>

@Injectable()
export class NotificationTypeRegistry {
  private types = new Map<NotificationType, NotificationTypeConfig>();

    this.types.set(type, config);

interface NotificationChannel {
  readonly name: string;

  supports(type: NotificationType): boolean;
}
```

### Action Handlers

<Tip>
Action handlers are auto-discovered via the `NotificationHandlerExplorer` and provide lifecycle hooks for complete notification management.
</Tip>

```typescript
@NotificationActionHandler({ name: 'lead-assignment' })
export class LeadAssignmentHandler extends BaseNotificationHandler {
  async onCreate(notification: NotificationRecipient): Promise<void> {
    // Initialize notification data
  }
  
  async onAction(
    notification: NotificationRecipient,
    action: string,
    data?: any
  ): Promise<ActionResult> {
    switch (action) {
      case 'view_lead':
        return {
          success: true,
          redirectUrl: `/leads/${notification.notificationEvent.data.leadId}`
        };
      default:
        throw new Error(`Unknown action: ${action}`);
    }
  }
  
  async onExpire(notification: NotificationRecipient): Promise<void> {
    // Cleanup expired notification
  }
}
```

### Event Naming Conventions

<Check>
Events follow a consistent naming pattern: domain-prefixed with kebab-case for consistency.
</Check>

- **CRM Events**: `crm.lead-assigned`, `crm.deal-status-changed`
- **RBAC Events**: `rbac.team-member-added`, `rbac.role-assigned`
- **AI Events**: `ai.credit-low`, `ai.agent-failed`
- **Messaging Events**: `messaging.conversation-transferred`

### Error Handling

The system implements comprehensive error handling:

1. **Delivery failures** are logged but don't break the notification flow
2. **Invalid action attempts** return appropriate error responses
3. **Handler errors** are caught and logged with context
4. **Retry logic** ensures reliable delivery across all channels

<Note>
The notification module is designed to be fault-tolerant, ensuring that system failures don't prevent critical notifications from being delivered to users.
</Note>
