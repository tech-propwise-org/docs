<Note>
**Source of truth:** `notification-type.enum.ts`, `notification-type.registry.ts`, and the domain listeners under `src/modules/notification/listeners/` plus external emitters (AI agent, escalation, distribution cron, workflow engine, reminder scheduler).

**Convention:** "Sender" = the acting user; in most bulk paths the sender is filtered OUT of the recipient list unless the caller explicitly opts into sender delivery. Channels are registry candidates; user preferences and per-reminder overrides decide the final delivery channels.
</Note>

## Understanding Notification Components

<CardGroup cols={2}>
  <Card title="Categories" icon="folder">
    - **properties** (formerly `real_estate`)
    - **crm**
    - **access** (formerly `team`)
    - **system**
    - **messaging**
  </Card>
  <Card title="Actions" icon="hand-pointer">
    **In-app actions:**
    - `navigate` = opens the linked entity
    - `approve`/`reject`/`rsvp_*` = change notification status
    
    **Email behavior:** Only sends entity deep-link buttons for `navigate` actions. Interactive actions require opening the app.
  </Card>
</CardGroup>

<Info>
**Subcategory** is derived from type ‚Äî never stored in DB. See `NotificationSubcategory` enum.

**Setting visibility:** Some notification types carry a `visibilityPermissions` list in the registry. When present, `NotificationPreferenceService.getPreferences()` filters those types out of the preference catalog **before** sending it to the frontend ‚Äî users who do not hold any of the listed permissions simply do not see the toggle row in the UI. This is **cosmetic only**: delivery logic is unchanged (stakeholders still receive the notification). See ¬ß25 for the complete gate table.
</Info>

<Info>
System-initiated stakeholder creations (distribution/escalation engines) emit `lead_assigned` instead of `lead_stakeholder_added` to avoid duplicate/confusing pings.
</Info>

<Tip>
**Grouped:** `lead_stakeholder_commission_changed` is grouped ‚Äî all lead-commission changes to a user within a **1-minute window** collapse into one in-app notification + one digest email (`groupKeyFields: []`, `mode: digest`). The digest header counts **distinct leads** (by `entityId`), not rows. 

When a single change touches a recipient twice on the **same** lead (e.g. a team leader whose **personal** cut drops while their **team-pool** cut rises) the digest reconstructs **one concise sentence** from the item payloads: `{actor} updated commissions on {lead} ‚Äî your share: A% ‚Üí B%, {Team}: C% ‚Üí D%.` (never "2 leads"; falls back to the raw item messages if payload fields are missing). Per-item messages use the arrow form `{actor} updated your commission on {lead}: A% ‚Üí B%.` See ¬ß27.
</Tip>

## 5. CRM ‚Äî Activity Stakeholder Inbox (granular)

**Category:** `crm` ¬∑ **Subcategory:** `activities`

<Info>
Triggered when an activity is logged/scheduled on a lead or contact. Sender = the performer; recipients = lead+linked-contact stakeholders with `team.crm.view`/`team.crm.manage` (excl. performer). All MEDIUM priority, IN_APP + EMAIL candidates.

**All types have the same action:** View record (navigates using the payload `entityType` / `entityId`).
</Info>

<Warning>
Create/schedule only ‚Äî editing an activity does NOT ping.
</Warning>

### Activity Notification Types

- `activity_call_logged`
- `activity_email_logged`
- `activity_sms_logged`
- `activity_meeting_scheduled`
- `activity_site_visit_scheduled`
- `activity_note_added`
- `activity_marked_important`

<Tip>
**Grouped (digest).** These types are the **pilot** for the generic grouping engine (see ¬ß27). A burst of activity logged on the same lead/contact for a stakeholder collapses into **one** in-app digest + **one** digest email per **15-minute fixed window** (`groupKeyFields: ['entityId']`, `mode: 'digest'`) instead of N separate pings. Grouping is keyed per notification type, so e.g. several `activity_call_logged` on one lead group together.
</Tip>

## 27. Notification Grouping (Digest Mode)

<Info>
The notification system supports **digest mode** to prevent notification fatigue. When enabled for a notification type, multiple notifications of the same type are collapsed within a time window.
</Info>

### Grouping Configuration

Grouping is configured per notification type in the registry:

```typescript
{
  groupKeyFields: ['entityId'], // Fields to group by
  mode: 'digest',               // Grouping mode
  windowMinutes: 15             // Time window for grouping
}
```

### Grouped Notification Types

<Tabs>
  <Tab title="Commission Changes">
    **Type:** `lead_stakeholder_commission_changed`
    
    **Window:** 1 minute
    
    **Grouping:** All lead-commission changes to a user within the window collapse into one in-app notification + one digest email.
    
    **Behavior:**
    - Counts distinct leads by `entityId`
    - Single change on same lead reconstructs one sentence
    - Example: `{actor} updated commissions on {lead} ‚Äî your share: A% ‚Üí B%, {Team}: C% ‚Üí D%.`
  </Tab>
  
  <Tab title="Activity Logs">
    **Types:** All `activity_*` types
    
    **Window:** 15 minutes (fixed)
    
    **Grouping:** Activity logged on the same lead/contact collapses per notification type.
    
    **Behavior:**
    - Groups by `entityId` per type
    - Example: Multiple `activity_call_logged` on one lead ‚Üí one digest
    - Different activity types remain separate
  </Tab>
</Tabs>

<Warning>
Grouping is **per notification type**. For example, `activity_call_logged` and `activity_email_logged` on the same entity will create **two separate** digests, even within the same time window.
</Warning>

## Best Practices

<Steps>
  <Step title="Sender filtering">
    In most bulk notification paths, the acting user (sender) is automatically filtered out of the recipient list unless explicitly opted in.
  </Step>
  
  <Step title="Stakeholder resolution">
    Recipients are resolved from entity stakeholders using appropriate permissions (`team.crm.view`, `team.crm.manage`, etc.).
  </Step>
  
  <Step title="Channel candidates">
    Channels listed in the registry are candidates; final delivery channels are determined by user preferences and per-reminder overrides.
  </Step>
  
  <Step title="Interactive actions">
    Email notifications only send deep-link buttons for `navigate` actions. Interactive actions (approve/reject/RSVP) require the recipient to open the app.
  </Step>
</Steps>

<Check>
All notification types in this catalog are registered in the system and actively monitored. Changes to notification behavior require updates to both the registry and relevant domain listeners.
</Check>
