description: "Complete catalog of all notification types, their triggers, recipients, and delivery channels across the PropWise CRM system"

<Note>
This catalog serves as the source of truth for all notification types, derived from `notification-type.enum.ts`, `notification-type.registry.ts`, and domain listeners under `src/modules/notification/listeners/`.
</Note>

## Overview

### Key Concepts

<AccordionGroup>
  <Accordion title="Sender & Recipients">
    **Sender** = the acting user who triggers the notification. In most bulk paths, the sender is filtered OUT of the recipient list unless the caller explicitly opts into sender delivery.
  </Accordion>

  <Accordion title="Channels & Preferences">
    Channels are registry candidates; user preferences and per-reminder overrides decide the final delivery channels.
  </Accordion>

  <Accordion title="Actions">
    Lists the in-app notification actions available to recipients:
    - `navigate` = opens the linked entity
    - `approve`/`reject`/`rsvp_*` = in-app-only actions that change notification status
    
    Email sends entity deep-link buttons for `navigate` actions only; interactive actions (approve/reject/RSVP) require the recipient to open the app.
  </Accordion>

  <Accordion title="Categories">
    - `properties` (formerly `real_estate`)
    - `crm`
    - `access` (formerly `team`)
    - `system`
    - `messaging`
    
    **Subcategory** is derived from type — never stored in DB. See `NotificationSubcategory` enum.
  </Accordion>

  <Accordion title="Setting Visibility">
    Some notification types carry a `visibilityPermissions` list in the registry. When present, `NotificationPreferenceService.getPreferences()` filters those types out of the preference catalog **before** sending it to the frontend — users who do not hold any of the listed permissions simply do not see the toggle row in the UI.
    
    <Warning>
    This is **cosmetic only**: delivery logic is unchanged (stakeholders still receive the notification).
    </Warning>
  </Accordion>
</AccordionGroup>

<Tip>
**Grouped notification:** `lead_stakeholder_commission_changed` is grouped — all lead-commission changes to a user within a **1-minute window** collapse into one in-app notification + one digest email (`groupKeyFields: []`, `mode: digest`).

The digest header counts **distinct leads** (by `entityId`), not rows. When a single change touches a recipient twice on the **same** lead (e.g. a team leader whose **personal** cut drops while their **team-pool** cut rises) the digest reconstructs **one concise sentence** from the item payloads:

`{actor} updated commissions on {lead} — your share: A% → B%, {Team}: C% → D%.`

Per-item messages use the arrow form `{actor} updated your commission on {lead}: A% → B%.`
</Tip>

## 5. CRM — Activity Stakeholder Inbox (granular)

**Category:** `crm` · **Subcategory:** `activities`

<Info>
Triggered when an activity is logged/scheduled on a lead or contact. Sender = the performer; recipients = lead+linked-contact stakeholders with `team.crm.view`/`team.crm.manage` (excl. performer). All MEDIUM priority, IN_APP + EMAIL candidates.
</Info>

<Warning>
Create/schedule only — editing an activity does NOT ping.
</Warning>

<Tip>
**Grouped (digest).** These types are the **pilot** for the generic grouping engine. A burst of activity logged on the same lead/contact for a stakeholder collapses into **one** in-app digest + **one** digest email per **15-minute fixed window** (`groupKeyFields: ['entityId']`, `mode: 'digest'`) instead of N separate pings. Grouping is keyed per notification type, so e.g. several `activity_call_logged` on one lead group together.
</Tip>

### Activity notification types

All activity types share common properties:

| Property | Value |
|----------|-------|
| **Sender** | Activity performer |
| **Recipients** | Lead + linked-contact stakeholders with `team.crm.view` or `team.crm.manage` (excl. performer) |
| **Priority** | MEDIUM |
| **Actions** | View record (navigates using the payload `entityType` / `entityId`) |
| **Channels** | IN_APP + EMAIL candidates |

**Available types:**

- `activity_call_logged`
- `activity_email_logged`
- `activity_sms_logged`
- `activity_meeting_scheduled`
- `activity_site_visit_scheduled`
- `activity_note_added`
- `activity_marked_important`

## 25. Permission Gates for Setting Visibility

<Note>
This table shows which notification types require specific permissions to appear in a user's preference settings UI. Types not listed here are visible to all users.
</Note>

| Notification Type(s) | Required Permission(s) | Notes |
|---------------------|----------------------|-------|
| `unit_access_*` | Any real-estate or sales permission | Property access workflow |
| `listing_*` | `real_estate.view` OR `real_estate.manage` OR `real_estate.listing.publish` | Listing management |
| `distribution_failed` | `team.manage` | Admin-only distribution alerts |
| `escalation_failed` | `team.manage` | Admin-only escalation alerts |
| `ai_*` | Visible to all | AI features are org-wide |
| `workflow_*` | Visible to all | Workflows can involve any user |
| `import_*` | Visible to all | Any user can import data |

<Warning>
**Setting visibility is cosmetic only.** Even if a notification type is hidden from a user's preferences, they will still receive notifications of that type if they are stakeholders in the relevant entities. The visibility gates only control whether the toggle appears in the settings UI.
</Warning>

## 27. Notification Grouping

<Info>
Notification grouping is a feature that collapses multiple related notifications into a single digest to prevent notification fatigue.
</Info>

### Grouping Strategies

<Tabs>
  <Tab title="Digest Mode">
    Multiple notifications of the same type for the same entity are collapsed into a single notification with a summary.
    
    **Example:** Multiple activity logs on the same lead within 15 minutes collapse into:
    ```
    "5 activities logged on Lead: John Smith"
    ```
    
    - **Window:** Fixed time window (e.g., 15 minutes, 1 minute)
    - **Key fields:** `entityId` (groups by same entity)
    - **Delivery:** One in-app notification + one digest email
  </Tab>
  
  <Tab title="Count Mode">
    Shows a count of related notifications without individual details.
    
    **Example:**
    ```
    "3 commission changes across 2 leads"
    ```
    
    - Used for bulk updates
    - Provides aggregate statistics
    - Links to full details view
  </Tab>
  
  <Tab title="Smart Condensing">
    When multiple changes affect the same entity/recipient combination, the system reconstructs a concise summary.
    
    **Example (commission change):**
    ```
    "Jane updated commissions on Lead: ABC Corp 
    — your share: 2% → 3%, Sales Team: 5% → 4%"
    ```
    
    Instead of two separate notifications.
  </Tab>
</Tabs>

### Currently Grouped Types

<AccordionGroup>
  <Accordion title="lead_stakeholder_commission_changed">
    - **Window:** 1 minute
    - **Mode:** Digest with smart condensing
    - **Key:** All changes to a user (empty `groupKeyFields`)
    - **Summary:** Counts distinct leads, reconstructs per-lead sentences
  </Accordion>
  
  <Accordion title="Activity notifications (pilot)">
    All `activity_*` types:
    - **Window:** 15 minutes (fixed)
    - **Mode:** Digest
    - **Key:** `entityId` (same lead/contact)
    - **Summary:** Lists activity types and count
  </Accordion>
  
  <Accordion title="deal_stakeholder_commission_changed">
    - **Window:** 1 minute
    - **Mode:** Digest with smart condensing
    - **Key:** All changes to a user
    - **Summary:** Same pattern as lead commissions
  </Accordion>
</AccordionGroup>

<Tip>
The grouping engine is designed to be extensible. New notification types can opt into grouping by configuring `groupKeyFields` and `groupingMode` in the notification registry.
</Tip>
