## Key Concepts

<Info>
**Sender Convention**: The "sender" is the acting user. In bulk notification paths, the sender is typically filtered out of the recipient list unless explicitly opted in for delivery.
</Info>

<Tip>
**Delivery Channels**: Channels are registry candidates, but final delivery is determined by user preferences and per-reminder overrides.
</Tip>

### Notification Actions

The **Actions** column describes available in-app notification actions:

- `navigate` - Opens the linked entity
- `approve`/`reject` - In-app only actions that change notification status  
- `rsvp_*` - RSVP actions for events
- Email notifications only include deep-link buttons for `navigate` actions

<Warning>
Interactive actions (approve/reject/RSVP) require recipients to open the app - they cannot be performed via email.
</Warning>

### Setting Visibility

Some notification types include `visibilityPermissions` in the registry. When present, `NotificationPreferenceService.getPreferences()` filters these types from users who don't hold the required permissions. This is cosmetic only - delivery logic remains unchanged.

## Properties Notifications

### Unit Access

<Info>
**Category**: `properties` · **Subcategory**: `unit_access`  
**Visibility**: Requires any real-estate or sales permission
</Info>

| Type | Trigger | Sender | Recipients | Purpose | Priority | Actions |
|------|---------|--------|------------|---------|----------|---------|
| `unit_access_request` | Agent requests unit details access | Requesting agent | Unit manager | Manager approval required | HIGH | View entity · Approve · Reject · View Unit · View Deal |
| `unit_access_approved` | Manager approves access request | Approver | Original requester | Access granted notification | HIGH | View Unit |
| `unit_access_rejected` | Manager rejects access request | Rejecter | Original requester | Denial notification with reason | MEDIUM | — |

### Listing Approval Workflow

<Info>
**Category**: `properties` · **Subcategory**: `listing`  
**Visibility**: Requires `real_estate.view`, `real_estate.manage`, or `real_estate.listing.publish`
</Info>

| Type | Trigger | Sender | Recipients | Purpose | Priority | Actions |
|------|---------|--------|------------|---------|----------|---------|
| `listing_approval_requested` | Non-publisher submits listing | Submitter | Org approvers with `real_estate.manage` | Approval workflow initiation | HIGH | Review listing |
| `listing_approved` | Manager approves listing | Approver | Listing publisher (`createdBy`) | Approval confirmation | HIGH | View listing |
| `listing_rejected` | Manager rejects listing | Rejecter | Listing publisher (`createdBy`) | Rejection notice with reason | HIGH | View listing |
| `listing_deleted` | Listing deleted by non-publisher | Deleter | Listing publisher (`createdBy`) | Deletion notification | MEDIUM | — |

### Activity Stakeholder Inbox

<Info>
**Category**: `crm` · **Subcategory**: `activities`  
**Recipients**: Lead + linked-contact stakeholders with `team.crm.view` or `team.crm.manage` (excluding performer)  
**Priority**: MEDIUM for all activity types  
**Channels**: IN_APP + EMAIL candidates
</Info>

<Tip>
Activity notifications are only sent when activities are created or scheduled - editing existing activities does not trigger notifications.
</Tip>

**Activity Types**:
- `activity_call_logged`
- `activity_email_logged` 
- `activity_sms_logged`
- `activity_meeting_scheduled`
- `activity_site_visit_scheduled`
- `activity_note_added`
- `activity_marked_important`

**Actions**: View record (navigates using payload `entityType`/`entityId`)

## System Architecture

<Steps>
  <Step title="Notification Generation">
    Notifications are generated by domain listeners under `src/modules/notification/listeners/` and external emitters (AI agent, escalation engine, distribution cron, workflow engine, reminder scheduler).
  </Step>

  <Step title="Registry Processing">
    The notification registry determines available channels and default preferences based on type configuration.
  </Step>

  <Step title="Preference Filtering">
    User preferences and per-reminder overrides determine final delivery channels.
  </Step>

  <Step title="Delivery">
    Notifications are delivered through configured channels (in-app, email, etc.) based on final preferences.
  </Step>
</Steps>

<Warning>
The notification catalog is continuously evolving. Always refer to the source files (`notification-type.enum.ts`, `notification-type.registry.ts`) for the most current implementation details.
</Warning>
