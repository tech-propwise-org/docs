<Info>
This catalog serves as the source of truth for all notification types in the PropWise CRM system. It's based on `notification-type.enum.ts`, `notification-type.registry.ts`, and domain listeners in the notification module.
</Info>

## Overview

The PropWise notification system delivers real-time updates to users across multiple channels (in-app, email, SMS) based on business events and user actions. This catalog documents every notification type, its triggers, recipients, and available actions.

### Key Concepts

<Tabs>
  <Tab title="Sender & Recipients">
    - **Sender**: The user performing the action that triggers the notification
    - **Recipients**: Users who receive the notification (sender is typically excluded unless explicitly opted in)
    - **Stakeholders**: Users with permissions or assignments related to the entity
  </Tab>
  <Tab title="Categories">
    - **Properties**: Real estate listings, units, and property management
    - **CRM**: Lead and contact management workflows
    - **Access**: Team permissions and access requests
    - **System**: Platform-wide notifications and alerts
    - **Messaging**: Communication and collaboration features
  </Tab>
  <Tab title="Actions">
    - **Navigate**: Opens the linked entity in the app
    - **Interactive**: In-app actions like approve/reject/RSVP (email shows deep-link buttons only)
    - **View**: Standard entity viewing action
  </Tab>
</Tabs>

<Warning>
Some notification types have visibility permissions that filter them from user preference settings. However, delivery logic remains unchanged - stakeholders still receive these notifications regardless of UI visibility.
</Warning>

### Activity Notifications

<AccordionGroup>
  <Accordion title="Activity Stakeholder Inbox">
    
**Category:** `crm` · **Subcategory:** `activities`

**Recipients:** Lead + linked-contact stakeholders with `team.crm.view` or `team.crm.manage` permissions (excluding performer)  
**Priority:** MEDIUM  
**Channels:** IN_APP + EMAIL candidates  
**Actions:** View record (navigates using payload `entityType` / `entityId`)

**Activity Types:**
- `activity_call_logged`
- `activity_email_logged`
- `activity_sms_logged`
- `activity_meeting_scheduled`
- `activity_site_visit_scheduled`
- `activity_note_added`
- `activity_marked_important`

<Note>
Activity notifications are only triggered when activities are created or scheduled. Editing existing activities does not generate notifications.
</Note>

## Notification Behavior

### Delivery Rules

<Steps>
  <Step title="Trigger Detection">
    System events and user actions trigger notifications based on domain listeners and external emitters (AI agent, escalation, distribution cron, workflow engine, reminder scheduler).
  </Step>
  <Step title="Recipient Calculation">
    Recipients are determined based on stakeholder relationships, permissions, and team assignments. The sender is typically excluded unless explicitly opted in.
  </Step>
  <Step title="Channel Selection">
    Delivery channels are determined by user preferences and per-reminder overrides from the notification registry.
  </Step>
  <Step title="Action Processing">
    In-app notifications support interactive actions, while email notifications provide deep-link buttons for navigation actions only.
  </Step>
</Steps>

### Priority Levels

<CardGroup cols={2}>
  <Card title="HIGH Priority" icon="exclamation-triangle">
    Critical business events requiring immediate attention (approvals, assignments)
  </Card>
  <Card title="MEDIUM Priority" icon="circle-info">
    Important updates that don't require immediate action (status changes, assignments)
  </Card>
  <Card title="LOW Priority" icon="circle-dot">
    Informational updates (archive actions, access changes)
  </Card>
</CardGroup>

<Tip>
For complete permission gates and visibility rules, refer to the notification preferences documentation in section §25 of the backend specification.
</Tip>
