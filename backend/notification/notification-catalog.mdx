## Conventions & Delivery Rules

<Info>
**Sender filtering:** "Sender" refers to the acting user. In most bulk notification paths, the sender is automatically filtered OUT of the recipient list unless the caller explicitly opts into sender delivery.
</Info>

### Channel Selection

Channels are registry candidates — final delivery channels are determined by:
- User notification preferences
- Per-reminder channel overrides (when applicable)

### Actions Column

Lists the in-app notification actions available to recipients:

- **`navigate`** — Opens the linked entity
- **`approve`/`reject`/`rsvp_*`** — In-app-only actions that change notification status

<Warning>
Email sends entity deep-link buttons for `navigate` actions only. Interactive actions (approve/reject/RSVP) require the recipient to open the app.
</Warning>

### Categories & Subcategories

**Categories:**
- `properties` (formerly `real_estate`)
- `crm`
- `access` (formerly `team`)
- `system`
- `messaging`

**Subcategory** is derived from notification type — never stored in database. See `NotificationSubcategory` enum.

### Setting Visibility

<Tip>
Some notification types carry a `visibilityPermissions` list in the registry. When present, `NotificationPreferenceService.getPreferences()` filters those types out of the preference catalog **before** sending it to the frontend.
</Tip>

Users who do not hold any of the listed permissions simply do not see the toggle row in the UI. This is **cosmetic only** — delivery logic is unchanged (stakeholders still receive the notification). See [Permission Gates](#permission-gates) for the complete gate table.

### lead_stakeholder_commission_changed

| Property | Value |
|----------|-------|
| **When sent** | A lead stakeholder's commission % changes |
| **Sender** | Changer |
| **Recipients** | Affected user |
| **Why** | Commission updated |
| **Priority** | LOW |
| **Actions** | View Lead |

<Info>
**Grouped notification:** `lead_stakeholder_commission_changed` is grouped — all lead-commission changes to a user within a **1-minute window** collapse into one in-app notification + one digest email (`groupKeyFields: []`, `mode: digest`).
</Info>

The digest header counts **distinct leads** (by `entityId`), not rows. When a single change touches a recipient twice on the **same** lead (e.g. a team leader whose **personal** cut drops while their **team-pool** cut rises) the digest reconstructs **one concise sentence** from the item payloads:

```
{actor} updated commissions on {lead} — your share: A% → B%, {Team}: C% → D%.
```

Never shows "2 leads"; falls back to raw item messages if payload fields are missing. Per-item messages use the arrow form:

```
{actor} updated your commission on {lead}: A% → B%.
```

## CRM — Activity Stakeholder Inbox (Granular)

<CardGroup cols={2}>
  <Card title="Category" icon="users">
    `crm`
  </Card>
  <Card title="Subcategory" icon="calendar">
    `activities`
  </Card>
</CardGroup>

Triggered when an activity is logged/scheduled on a lead or contact.

- **Sender:** The performer
- **Recipients:** Lead + linked-contact stakeholders with `team.crm.view`/`team.crm.manage` (excl. performer)
- **Priority:** MEDIUM (all types)
- **Channels:** IN_APP + EMAIL candidates

<Note>
Create/schedule only — editing an activity does NOT ping.
</Note>

<Info>
**Grouped (digest):** These types are the **pilot** for the generic grouping engine. A burst of activity logged on the same lead/contact for a stakeholder collapses into **one** in-app digest + **one** digest email per **15-minute fixed window** (`groupKeyFields: ['entityId']`, `mode: 'digest'`) instead of N separate pings. Grouping is keyed per notification type, so e.g. several `activity_call_logged` on one lead group together.
</Info>

## Grouping & Digest Rules

<Info>
Propwise implements notification grouping to prevent notification fatigue when multiple related events occur in quick succession.
</Info>

### Grouping Configuration

Grouping behavior is configured per notification type in the registry:

- **`groupKeyFields`** — Array of payload fields used to group notifications (e.g., `['entityId']`)
- **`mode`** — Either `'digest'` or `'collapse'`
- **Time window** — Fixed duration for grouping (varies by notification type)

### Current Grouped Types

<Tabs>
  <Tab title="Commission Changes">
    **Type:** `lead_stakeholder_commission_changed`
    
    - **Window:** 1 minute
    - **Key fields:** `[]` (groups all commission changes)
    - **Mode:** `digest`
    - **Counts:** Distinct leads affected
  </Tab>
  <Tab title="Activities">
    **Types:** All `activity_*` notifications
    
    - **Window:** 15 minutes (fixed)
    - **Key fields:** `['entityId']`
    - **Mode:** `digest`
    - **Groups:** Per notification type + entity
  </Tab>
</Tabs>

### Digest Message Construction

<Steps>
  <Step title="Collect items">
    Group notification items by time window and key fields
  </Step>
  <Step title="Count entities">
    Count distinct entities (by `entityId`) affected
  </Step>
  <Step title="Build message">
    Construct concise summary from collected payloads
  </Step>
  <Step title="Fallback">
    If payload fields missing, use raw item messages
  </Step>
</Steps>

## Best Practices

<Tip>
**For developers extending the notification system:**
</Tip>

1. **Always exclude sender** from recipient lists unless explicitly required
2. **Use appropriate priority levels** — HIGH for actionable items, LOW for informational
3. **Provide clear action paths** — ensure recipients can navigate to relevant entities
4. **Consider grouping** for high-frequency notification types
5. **Test permission gates** to ensure proper setting visibility
6. **Document new types** in this catalog with complete metadata

<Warning>
Interactive actions (approve/reject/RSVP) are **in-app only**. Email notifications for these types should only include navigation deep-links, not action buttons.
</Warning>
