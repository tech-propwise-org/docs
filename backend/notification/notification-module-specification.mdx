## Overview

The notification module provides a comprehensive system for managing and delivering notifications across multiple channels within the PropWise CRM. It supports in-app and email delivery channels, user preferences management, action-based notifications with handlers, and event-driven notification creation from domain events.

<Info>
The module follows an event-driven architecture where domain modules emit events that are captured by notification listeners, which then create appropriate notifications based on registered notification types and user preferences.
</Info>

**Last Updated:** 2024-12-19  
**Status:** Active

## Architecture

### Core Components

The notification system is built around several key components that work together to provide a flexible and scalable notification infrastructure:

- **Notification Events**: Central notification records that can have multiple recipients
- **Notification Recipients**: Per-user instances of notifications with status tracking
- **Notification Deliveries**: Per-channel delivery attempts with retry logic
- **Notification Channels**: Pluggable delivery mechanisms (email, in-app)
- **Action Handlers**: Business logic processors for notification actions
- **Preference System**: User-configurable notification settings per organization

### Design Decisions

<AccordionGroup>
  <Accordion title="Multi-Channel Architecture">
    Notifications can be delivered via multiple channels simultaneously based on user preferences. This allows users to receive notifications through their preferred communication methods.
  </Accordion>

  <Accordion title="Recipient-Centric Design">
    Each user gets their own recipient record with independent status tracking. This enables personalized notification management and prevents cross-user interference.
  </Accordion>

  <Accordion title="Action-Driven Workflow">
    Notifications can include actions that trigger business logic when clicked. This transforms notifications from passive information into interactive workflow triggers.
  </Accordion>

  <Accordion title="Event-Driven Creation">
    Domain events automatically trigger notification creation through listeners. This decouples notification logic from business logic while maintaining consistency.
  </Accordion>

  <Accordion title="Retry Mechanism">
    Failed deliveries are retried with exponential backoff to handle temporary service issues without overwhelming external services.
  </Accordion>

  <Accordion title="Organization Scoping">
    All notifications are scoped to organizations for multi-tenancy, ensuring complete data isolation between different clients.
  </Accordion>
</AccordionGroup>

## Entities

### NotificationEvent

Represents the core notification record that can target multiple recipients.

```typescript
{
  id: string;                           // UUID primary key
  type: NotificationType;               // LEAD_ASSIGNED, TEAM_MEMBER_ADDED, etc.
  category: NotificationCategory;       // CRM, REAL_ESTATE, RBAC, MESSAGING
  priority: NotificationPriority;       // LOW, NORMAL, HIGH, URGENT
  title: string;                        // Display title
  message: string;                      // Notification content
  data: Record<string, any>;            // Additional contextual data
  organization: Organization;           // Organization scope
  triggeredBy: User;                    // User who caused the notification
  expiresAt?: Date;                     // Optional expiry date
  createdAt: Date;
  updatedAt: Date;
  recipients: Collection<NotificationRecipient>;
}
```

### NotificationRecipient

Per-user instance of a notification with status and action tracking.

```typescript
{
  id: string;                           // UUID primary key
  notificationEvent: NotificationEvent; // Parent notification
  user: User;                          // Target user
  status: NotificationStatus;          // UNREAD, READ, DISMISSED
  readAt?: Date;                       // When marked as read
  dismissedAt?: Date;                  // When dismissed
  actionsTaken: ActionResult[];        // History of actions performed
  createdAt: Date;
  updatedAt: Date;
  deliveries: Collection<NotificationDelivery>;
}
```

### NotificationDelivery

Tracks delivery attempts per channel with retry logic.

```typescript
{
  id: string;                           // UUID primary key
  notificationRecipient: NotificationRecipient;
  channel: Channel;                     // EMAIL, IN_APP
  status: DeliveryStatus;              // PENDING, SENT, FAILED, RETRYING
  sentAt?: Date;                       // Successful delivery timestamp
  failedAt?: Date;                     // Last failure timestamp
  error?: string;                      // Last error message
  retryCount: number;                  // Number of retry attempts
  nextRetryAt?: Date;                  // Next retry schedule
  createdAt: Date;
  updatedAt: Date;
}
```

### NotificationPreference

User preferences for notification types and channels per organization.

```typescript
{
  id: string;                           // UUID primary key
  user: User;                          // Target user
  organization: Organization;           // Organization scope
  muteAll: boolean;                    // Global mute flag
  channelPreferences: Record<Channel, boolean>; // Per-channel preferences
  typePreferences: Record<NotificationType, {
    enabled: boolean;
    channels: Channel[];
  }>;                                  // Per-type channel preferences
  createdAt: Date;
  updatedAt: Date;
}
```

## API Endpoints

### Get Notifications

Retrieve paginated notifications for the authenticated user.

```http
GET /notifications?page=1&limit=20&status=UNREAD
```

<Tabs>
  <Tab title="Response">
    ```typescript
    {
      items: NotificationDto[];
      total: number;
      page: number;
      limit: number;
    }
    ```
  </Tab>

  <Tab title="Query Parameters">
    - `page` (optional): Page number for pagination
    - `limit` (optional): Number of items per page
    - `status` (optional): Filter by notification status
  </Tab>
</Tabs>

### Mark as Read

Mark a specific notification as read.

```http
PATCH /notifications/:id/read
```

**Response:**
```typescript
{
  success: boolean;
  redirectUrl?: string;
}
```

### Dismiss Notification

Remove a notification from the user's list.

```http
DELETE /notifications/:id
```

**Response:**
```typescript
{
  success: boolean;
}
```

Execute a specific action associated with a notification.

<CodeGroup>

  }

## Business Rules

### Notification Creation

<Check>
Notifications are created only for users with appropriate access to the related entity
</Check>

<Steps>
  <Step title="Access Validation">
    Verify user permissions for the entity referenced in the notification
  </Step>
  
  <Step title="Preference Check">
    Check user preferences before creating notifications
  </Step>
  
  <Step title="Mute Validation">
    If `muteAll` is enabled, skip notification creation entirely
  </Step>
  
  <Step title="Channel Selection">
    Only use enabled channels from user preferences for delivery
  </Step>
</Steps>

### Delivery Logic

<Warning>
Failed deliveries are retried with exponential backoff: 1min, 5min, 30min, 2hr, 12hr. After 5 failed attempts, delivery is marked as permanently failed.
</Warning>

- **In-app notifications** are always created regardless of delivery success
- **Email deliveries** are attempted based on user preferences
- **Retry mechanism** handles temporary failures gracefully

### Action Handling

<Note>
Actions can only be taken on notifications in UNREAD or READ status. Expired notifications cannot have actions taken.
</Note>

Key behaviors:
- Taking an action may change notification status to read
- Action handlers can return redirect URLs for frontend navigation
- Action results are stored in the recipient's `actionsTaken` array

### Expiration

- Expired notifications cannot have actions taken
- Cleanup job removes expired notifications and their deliveries
- Action handlers receive `onExpire` callback for cleanup logic

## Patterns and Conventions

### Notification Types

<Tip>
Notification types are enumerated and registered with configuration including channel preferences, action definitions, and handler mappings.
</Tip>

### Channel Implementation

All channels implement the `NotificationChannel` interface:

```typescript
interface NotificationChannel {
  name: string;
  deliver(notification: NotificationRecipient): Promise<DeliveryResult>;
  isAvailable(): boolean;
}
```

Channels are auto-discovered and registered at startup, supporting easy addition of new channels (SMS, Push, etc.).

### Action Handlers

Action handlers follow a consistent pattern:

```typescript
@NotificationActionHandler({ name: 'approve-lead' })
export class ApproveLeadHandler extends BaseNotificationHandler {
  async onCreate(notification: NotificationRecipient): Promise<void> {
    // Initialization logic
  }
  
  async onAction(notification: NotificationRecipient, data?: any): Promise<ActionResult> {
    // Action execution logic
  }
  
  async onExpire(notification: NotificationRecipient): Promise<void> {
    // Cleanup logic
  }
}
```

### Event Naming Convention

<AccordionGroup>
  <Accordion title="Domain Prefix">
    Events are prefixed with their domain: `crm.lead-assigned`, `rbac.team-member-added`
  </Accordion>
  
  <Accordion title="Kebab Case">
    All event names use kebab-case for consistency
  </Accordion>
  
  <Accordion title="Centralized Definition">
    Events are centralized in domain-specific event files
  </Accordion>
</AccordionGroup>

### Error Handling

- **Delivery failures** are logged but don't break the notification flow
- **Invalid action attempts** return appropriate error responses  
- **Handler errors** are caught and logged with full context
- **Graceful degradation** ensures system stability under failure conditions
