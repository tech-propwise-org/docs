## Executive Summary

The calendar system operates as a distributed sync system with three distinct layers, managing complex interactions between CRM events and external calendar providers.

<Warning>
The calendar integration is not a simple "push event to Google" feature. It's a distributed sync system with two hostile external providers, eventually consistent mailboxes, webhook retries, token expiry, RLS, notifications, and user actions happening from both CRM and external calendars.
</Warning>

### System Architecture

The system has three calendar layers:

1. **CRM calendar feed**: Primary routes for personal workload, organization stacked calendar, and team calendars
2. **CRM events**: Internal event source of truth with invitees, RSVP status, and notifications
3. **Calendar integrations**: Google and Outlook drivers for bidirectional sync

## CRM Calendar Feed Endpoints

### Personal Calendar (`/me`)

<CodeGroup>
```http GET /v1/calendar/me
GET /v1/calendar/me
```
</CodeGroup>

Returns personal workload including:
- CRM tasks/subtasks
- CRM events 
- External items (when viewing yourself)

### Organization Calendar (`/org`)

<CodeGroup>
```http GET /v1/calendar/org
GET /v1/calendar/org?userIds=1,2,3
```
</CodeGroup>

Returns stacked calendar for 1-10 other org members:
- Response format: `{ rosterUserIds, itemsByUserId }`
- Personal-scope tasks/subtasks + CRM events per subject
- No external items included
- Viewer ID in query is ignored

### Team Calendar (`/teams/:teamId`)

<CodeGroup>
```http GET /v1/calendar/teams/:teamId
GET /v1/calendar/teams/123
```
</CodeGroup>

Access requires:
- `team.admin`, `team_crm.*`, or `team_sales.*` permissions on that team
- OR org admin/owner role

Returns CRM-only items per roster member (authenticated user excluded from roster).

<Note>
Use `/me` endpoint for self-calendar access.
</Note>

### Agenda Endpoints

#### Personal Agenda (`/me/agenda`)

<CodeGroup>
```http GET /v1/calendar/me/agenda
GET /v1/calendar/me/agenda?from=2024-01-01T00:00:00Z
```
</CodeGroup>

Serves personal merged agenda list:
- Optional inclusive `from` parameter
- Forward scan up to server lookahead cap
- **No** `limit`/`cursor` pagination
- Response includes `truncated: boolean` flag

<Info>
`truncated: true` indicates the merge loop exited before reaching lookahead horizon. `truncated: false` means full lookahead window was scanned.
</Info>

#### Timeline Agenda (`/me/agenda/timeline`)

<CodeGroup>
```http GET /v1/calendar/me/agenda/timeline
GET /v1/calendar/me/agenda/timeline?cursor=eyJ2IjoxLCJuZXh0RGF5IjoiMjAyNC0wMS0wMiJ9
```
</CodeGroup>

Serves CRM web agenda list with **day-window pagination**:

<Steps>
<Step title="Merged Probe">
Fetch merged window from page anchor day + 60 days lookahead
</Step>
<Step title="Local Window">
Convert to viewer `Time-Zone` header for `fromDay`/`toDay`
</Step>
<Step title="Authoritative Refetch">
Full refetch for that day window
</Step>
<Step title="Next Cursor">
Generate `nextCursor` for following page
</Step>
</Steps>

Response format:
```json
{
  "window": {
    "fromDay": "2024-01-01",
    "toDay": "2024-01-31", 
    "timeZone": "America/New_York"
  },
  "items": [...],
  "nextCursor": "eyJ2IjoxLCJuZXh0RGF5IjoiMjAyNC0wMi0wMSJ9",
  "hasMore": true
}
```

### Calendar Event Mapping

The `calendar_event_mapping` table links CRM events to external calendar events:

- `eventId`: CRM event reference
- `calendarIntegration`: Provider connection
- `externalEventId`: Provider-specific event ID
- `iCalUid`: Universal event identifier
- `lastOutboundFingerprint`: Change detection for sync loops

## Webhook Security

<Warning>
Webhook authenticity verification is critical for security. Each provider has different validation mechanisms.
</Warning>

### Google Calendar Webhooks

Google uses separate verification tokens:

<CodeGroup>
```typescript Google Webhook Validation
// watchChannelVerifyToken is sent in events.watch
// and echoed as X-Goog-Channel-Token
const incomingToken = req.headers['x-goog-channel-token'];
const expectedToken = integration.watchChannelVerifyToken;

if (incomingToken !== expectedToken) {
  throw new UnauthorizedException('Invalid webhook token');
}

// Also verify resource ID when present
const resourceId = req.headers['x-goog-resource-id'];
if (integration.watchResourceId && resourceId !== integration.watchResourceId) {
  throw new UnauthorizedException('Invalid resource ID');
}
```
</CodeGroup>

<Note>
`watchChannelVerifyToken` is **separate** from channel `id`. Legacy rows may incorrectly equate the two.
</Note>

### Outlook Calendar Webhooks

Outlook requires `clientState` validation and JWT verification:

<CodeGroup>
```typescript Outlook Webhook Validation
// clientState must equal integration UUID
const clientState = notification.clientState;
if (clientState !== integration.id) {
  throw new UnauthorizedException('Invalid client state');
}

// Verify JWTs when validationTokens present
if (body.validationTokens?.length) {
  await verifyMicrosoftGraphWebhookTokens(
    body.validationTokens,
    audience // defaults to calendar app client ID
  );
}
```
</CodeGroup>

<Warning>
In **production** environments, `GOOGLE_CALENDAR_WEBHOOK_URL` and `OUTLOOK_CALENDAR_WEBHOOK_URL` must be **https** and cannot target localhost.
</Warning>

## Sync Mechanisms

### Event Sync Model

The intended sync model follows these principles:

<Check>
**CRM events are the canonical business record**
</Check>

<Check>
**Creator calendar copies can update CRM event details when changed externally**
</Check>

<Check>
**Invitee calendar copies can update RSVP only**
</Check>

<Check>
**External-only items are displayed read-only and not imported into CRM**
</Check>

<Check>
**Provider echoes are suppressed with change IDs, fingerprints, ETags, and time windows**
</Check>

### Outbound Sync

When CRM events change, the system pushes updates to connected calendars:

<CodeGroup>
```typescript Outbound Sync Flow
// 1. Event change detected
const changeFingerprint = generateEventFingerprint(crmEvent);

// 2. Check for mapping
const mapping = await findEventMapping(crmEvent.id);

// 3. Push to provider
if (mapping) {
  await calendarDriver.updateEvent(mapping.externalEventId, crmEvent);
} else {
  const externalId = await calendarDriver.createEvent(crmEvent);
  await createEventMapping(crmEvent.id, externalId);
}

// 4. Update fingerprint
mapping.lastOutboundFingerprint = changeFingerprint;
```
</CodeGroup>

### Inbound Sync

Provider webhooks trigger processing of external changes:

<CodeGroup>
```typescript Inbound Webhook Processing
// 1. Authenticate webhook
await validateWebhookSignature(req);

// 2. Process change notifications
for (const notification of notifications) {
  const mapping = await findMappingByExternalId(notification.eventId);
  
  if (mapping) {
    // Update existing CRM event
    await syncExternalChangesToCrm(mapping, notification);
  }
  // External-only events are not imported
}

// 3. Update sync tokens
await updateProviderSyncToken(integration, newSyncToken);
```
</CodeGroup>

## Operational Considerations

### Sync Health Monitoring

The system tracks sync health through several metrics:

<Tabs>
<Tab title="Timestamps">
- `lastInboundWebhookAt`: Last successful webhook processing
- `lastOutboundSyncAt`: Last successful CRM→provider push
- `lastSyncErrorAt`: Most recent error occurrence
</Tab>

<Tab title="Error Tracking">
- `lastSyncErrorMessage`: Bounded error description
- Webhook retry mechanisms with exponential backoff
- Token refresh handling for expired credentials
</Tab>

<Tab title="Watch Renewal">
- Google: Automatic channel renewal before expiration
- Outlook: Subscription lifecycle management
- Proactive monitoring of watch health
</Tab>
</Tabs>

### Known Risk Areas

<Warning>
The biggest risk is that the system is trying to make two different calendar providers behave like one coherent event bus. This is inherently fragile.
</Warning>

Critical risk areas include:

<AccordionGroup>
<Accordion title="In-Memory Cache">
Cache invalidation issues can cause stale data to be served or synced
</Accordion>

<Accordion title="Outbound Sync Without Durable Queue">
Listener path lacks durability guarantees for CRM-originated changes
</Accordion>

<Accordion title="Provider-Specific Edge Cases">
- Recurring event handling differences
- All-day event timezone interpretation
- Deleted invitee management
- Outlook mailbox-local event IDs
</Accordion>

<Accordion title="Race Conditions">
Simultaneous updates from CRM and external calendars can create conflicts
</Accordion>
</AccordionGroup>

### Durable Operations

<Info>
**Outlook shadow mapping retries are durable:** Uses `calendar_sync_outbox` with operation `outlook-shadow-reconcile` and pg-boss worker with backoff, not in-process timers.
</Info>

The `calendar_sync_outbox` table provides durable queuing for certain operations:
- RLS-protected like other org tables
- Handles retry logic with exponential backoff
- Ensures operations survive service restarts
