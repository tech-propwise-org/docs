<Warning>
The calendar integration is not a simple "push event to Google" feature. It is a distributed sync system with two hostile external providers, eventually consistent mailboxes, webhook retries, token expiry, RLS, notifications, and user actions happening from both CRM and external calendars.
</Warning>

## Executive Summary

The system has three calendar layers:

### 1. CRM Calendar Feed

Primary routes include:
- **`GET /v1/calendar/me`** - Personal workload: CRM tasks/subtasks + CRM events + external items when viewing yourself
- **`GET /v1/calendar/org`** - Organization stacked calendar: `userIds` = 1–10 other org members, response `{ rosterUserIds, itemsByUserId }`
- **`GET /v1/calendar/teams/:teamId`** - Team calendar with appropriate permissions

### 2. CRM Events

`EventService` owns the internal event source of truth, invitees, RSVP status, soft deletion, and notification emission.

### 3. Calendar Integrations

Google and Outlook drivers push CRM events outward and process provider webhooks back into CRM.

<Note>
The intended model is that CRM events are the canonical business record, with creator calendar copies able to update CRM event details when changed externally, while invitee calendar copies can only update RSVP status.
</Note>

## Personal Agenda Timeline

The **`GET /v1/calendar/me/agenda/timeline`** endpoint serves the CRM web agenda list with day-window pagination:

<Steps>
<Step title="Viewer Zone Detection">
Requires the standard `Time-Zone` request header (IANA). Invalid/missing values default to `UTC`.
</Step>

<Step title="Probe Window">
From the page anchor day, fetch a merged window of `CALENDAR_TIMELINE_PROBE_LOOKAHEAD_DAYS` (default 60) days.
</Step>

<Step title="Sort and Filter">
- Tasks by **due** (`startDate` on the DTO mirrors due)
- CRM events and externals by **start**
- Stable tie-break by source + id
</Step>

<Step title="Day Span Calculation">
Union of local calendar days each row touches, with `fromDay` floored at the page anchor.
</Step>

<Step title="Authoritative Fetch">
`collectMergedCalendarItems` for the calculated day range in the viewer's timezone.
</Step>
</Steps>

### Response Format

```json
{
  "window": {
    "fromDay": "2024-01-15",
    "toDay": "2024-01-20",
    "timeZone": "America/New_York"
  },
  "items": [...],
  "nextCursor": "base64-encoded-json",
  "hasMore": true
}
```

<Info>
The legacy `GET /v1/calendar/me/agenda` endpoint remains for callers that want a single forward scan without pagination.
</Info>

## OAuth Security

### Signed State Tokens

Calendar OAuth flows use `SignedPayloadService` with HMAC-SHA256 signed state tokens:

```typescript
// State payload structure
{
  userId: string,
  organizationId: string,
  purpose: 'calendar.oauth.state',
  iat: number,
  exp: number
}
```

<Check>
This prevents cross-site request forgery and replay attacks by including user/org context and expiration.
</Check>

### Provider-Specific Security

<Tabs>
<Tab title="Google Calendar">
- Separate `watchChannelVerifyToken` sent as `token` in `events.watch`
- Echoed as `X-Goog-Channel-Token` in webhook notifications
- `watchResourceId` must match `X-Goog-Resource-Id` when present
</Tab>
<Tab title="Outlook Calendar">
- Requires `clientState` to equal the integration UUID
- JWT validation for `validationTokens` in change notification batches
- JWKS verification with configured audience
</Tab>
</Tabs>

## Webhook Handling

### Production Requirements

<Warning>
In production (`buildAppConfig().isProduction`), `GOOGLE_CALENDAR_WEBHOOK_URL` and `OUTLOOK_CALENDAR_WEBHOOK_URL` must be **https** and must not target localhost.
</Warning>

### Authenticity Verification

The system implements comprehensive webhook authenticity checks:

<Steps>
<Step title="Google Verification">
- Verify `X-Goog-Channel-Token` matches stored `watchChannelVerifyToken`
- Check `X-Goog-Resource-Id` matches `watchResourceId` when present
- Validate channel ID and resource ID consistency
</Step>

<Step title="Outlook Verification">
- Verify `clientState` equals integration UUID
- For batches with `validationTokens`, verify JWTs using JWKS
- Check audience matches `app.outlookCalendar.webhookValidationAudience`
</Step>
</Steps>

## Sync Mechanisms

### Echo Suppression

The system uses multiple techniques to prevent sync loops:

- **Change IDs**: Track outbound change identifiers
- **Fingerprints**: Hash event content to detect modifications
- **ETags**: Use provider ETags when available
- **Time Windows**: Short suppression windows for recent changes

### Durable Operations

<Note>
Outlook shadow mapping retries are durable via `calendar_sync_outbox` operation `outlook-shadow-reconcile` + pg-boss execute worker with backoff, not in-process timers.
</Note>

### Row Level Security

```sql

```

## Event Access Control

Events use a creator + invitee + org-calendar policy model:

<AccordionGroup>
<Accordion title="ACCESS Permissions">
- Creator OR invitee OR org-calendar policy
- Same entity-type filters as `GET /v1/calendar/org`
- Team-calendar visibility rules apply
</Accordion>

<Accordion title="MODIFY/DELETE Permissions">
- Creator has full modification rights
- Invitees can only update RSVP status
- Org admins have elevated permissions
</Accordion>
</AccordionGroup>

## Production Considerations

### Known Risk Areas

<Warning>
The biggest risk is that the system is trying to make two different calendar providers behave like one coherent event bus. That is inherently fragile.
</Warning>

Key areas of concern:

1. **In-memory cache** - Potential for inconsistency
2. **CRM-originated outbound sync** - No durable queue for listener path
3. **Provider-specific edge cases**:
   - Recurring events
   - All-day events
   - Deleted invitees
   - Outlook mailbox-local event IDs

## Development Guidelines

### Constants Management

<Check>
All magic numbers are centralized in `src/common/constants/event-calendar-sync.constants.ts` including timeouts, backoff intervals, HTTP status aliases, and all-day UTC components.
</Check>

### Error Handling

The system implements comprehensive error handling:

- **Retry logic** with exponential backoff
- **Circuit breaker** patterns for provider failures
- **Graceful degradation** when external services are unavailable
- **Detailed logging** for debugging sync issues

### Testing Considerations

When testing calendar integrations:

1. Mock external provider responses
2. Test webhook authenticity verification
3. Verify echo suppression mechanisms
4. Test OAuth state token validation
5. Validate timezone handling edge cases
