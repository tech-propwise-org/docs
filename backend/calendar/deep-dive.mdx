<Warning>
The calendar integration is not a simple "push event to Google" feature. It is a distributed sync system with two hostile external providers, eventually consistent mailboxes, webhook retries, token expiry, RLS, notifications, and user actions happening from both CRM and external calendars.
</Warning>

This document explains how the current calendar system actually works. The calendar integration consists of three main layers that work together to provide seamless calendar synchronization across platforms.

### Intended Model

<Steps>
  <Step title="CRM Events as Source of Truth">
    CRM events serve as the canonical business record for all calendar activities
  </Step>
  <Step title="Creator Update Permissions">
    Creator calendar copies can update CRM event details when changed externally
  </Step>
  <Step title="Invitee RSVP Only">
    Invitee calendar copies can only update RSVP status
  </Step>
  <Step title="External Read-Only Display">
    External-only items are displayed read-only and not imported into CRM
  </Step>
  <Step title="Echo Suppression">
    Provider echoes are suppressed with change IDs, fingerprints, ETags, and time windows
  </Step>
</Steps>

## API Endpoints

### Personal Calendar Feed

<Tabs>
  <Tab title="Personal Workload">
    **`GET /v1/calendar/me`** - Personal workload including:
    - CRM tasks and subtasks
    - CRM events
    - External items (when viewing yourself)
  </Tab>
  <Tab title="Legacy Support">
    **`GET /v1/calendar`** and **`GET /v1/calendar/agenda`** delegate to the same handlers as `/me`
  </Tab>
</Tabs>

### Organization Calendar Views

<AccordionGroup>
  <Accordion title="Organization Stacked Calendar">
    **`GET /v1/calendar/org`** - Organization stacked calendar view:
    - `userIds` parameter: 1-10 other org members
    - Response: `{ rosterUserIds, itemsByUserId }`
    - Includes personal-scope tasks/subtasks + CRM events per subject
    - No external items included
    - Viewer ID in query is ignored
  </Accordion>
  
  <Accordion title="Team Calendar">
    **`GET /v1/calendar/teams/:teamId`** - Team-specific calendar view:
    - Required permissions: `team.admin`, `team_crm.*`, `team_sales.*`, or org admin/owner
    - CRM-only items per other roster members
    - Authenticated user excluded from roster
    - Use `/me` for self-calendar
  </Accordion>
</AccordionGroup>

### Agenda Endpoints

<Tip>
The agenda endpoints provide specialized views for timeline and list-based calendar interfaces.
</Tip>

#### Personal Agenda List
**`GET /v1/calendar/me/agenda`** - Personal merged agenda list:
- Optional inclusive `from` parameter
- Forward scan up to server lookahead cap
- **No** `limit`/`cursor` pagination
- Response includes `truncated` boolean:
  - `true`: Merge loop exited before reaching lookahead horizon
  - `false`: Scan completed full lookahead window

#### Timeline View
**`GET /v1/calendar/me/agenda/timeline`** - CRM web agenda with day-window pagination:
- Merged probe → local `fromDay`/`toDay` in viewer `Time-Zone` header
- Full refetch for that window
- `nextCursor` for following page

## Personal Agenda Timeline Deep Dive

<Info>
Used by the CRM web agenda view for infinite scroll. Returns the same merged shapes as `/calendar/me` using `CalendarItemDto`.
</Info>

### Configuration Requirements

<Steps>
  <Step title="Viewer Zone Setup">
    Requires standard `Time-Zone` request header (IANA format)
    - Invalid/missing timezone defaults to `UTC`
    - Handled by `TimezoneMiddleware` / `TimezoneParam`
  </Step>
  
  <Step title="Sorting Logic">
    - **Tasks**: Sorted by due date (`startDate` on DTO mirrors due)
    - **CRM Events & Externals**: Sorted by start time
    - **Tie-breaker**: Stable sort by source + id
  </Step>
</Steps>

### Processing Flow

<Tabs>
  <Tab title="Probe Phase">
    From the page anchor day:
    1. Fetch merged window of `CALENDAR_TIMELINE_PROBE_LOOKAHEAD_DAYS` (default 60 days)
    2. Merge-sort results
    3. Apply first-page visibility rule:
       - Tasks: `max(from, now)`
       - Events/externals: Keep when they overlap viewer's local "today"
    4. Take first `CALENDAR_TIMELINE_PAGE_TARGET` (40) rows
  </Tab>
  
  <Tab title="Day Span Calculation">
    1. Calculate union of local calendar days each row touches
    2. Date-only tasks use `dueDate` YYYY-MM-DD prefix
    3. Timed items use Luxon in viewer zone
    4. All-day events walk date labels
    5. `fromDay` is floored at page anchor
    6. Clamp to `CALENDAR_TIMELINE_MAX_DAY_SPAN` days
  </Tab>
  
  <Tab title="Response Generation">
    ```json
    {
      "window": {
        "fromDay": "2024-01-01",
        "toDay": "2024-01-31",
        "timeZone": "America/New_York"
      },
      "items": [...],
      "nextCursor": "base64EncodedJson",
      "hasMore": true
    }
    ```
  </Tab>
</Tabs>

## Data Model

### Calendar Integration Entity

The `calendar_integration` table stores one active external calendar account per user, organization, and provider.

<CodeGroup>
```typescript Key Fields
interface CalendarIntegration {
  user: User;                    // CRM user who connected
  organization: Organization;    // Tenant boundary
  provider: 'google' | 'outlook'; // External provider
  calendarEmail: string;         // External account email
  
  // Encrypted tokens
  encryptedAccessToken?: string; // AES-GCM encrypted (nullable after disconnect)
  encryptedRefreshToken?: string; // AES-GCM encrypted when issued
  tokenExpiresAt?: Date;         // Refresh threshold
  
  // Status tracking
  isActive: boolean;
  isDeleted: boolean;
}
```

```typescript Google-Specific Fields
interface GoogleCalendarFields {
  watchChannelId?: string;        // Google push watch channel ID
  watchChannelVerifyToken?: string; // Opaque secret for X-Goog-Channel-Token
  watchResourceId?: string;       // Resource identifier
  watchExpiration?: Date;         // Watch expiration time
  googleSyncToken?: string;       // Incremental sync token
}
```

```typescript Outlook-Specific Fields
interface OutlookCalendarFields {
  outlookSubscriptionId?: string;     // Microsoft Graph subscription ID
  outlookSubscriptionExpiration?: Date; // Subscription expiration
}
```

```typescript Observability Fields
interface SyncObservability {
  lastInboundWebhookAt?: Date;    // Last successful webhook processing
  lastOutboundSyncAt?: Date;      // Last successful CRM→provider push
  lastSyncErrorAt?: Date;         // Last provider/webhook failure
  lastSyncErrorMessage?: string;  // Bounded error message
}
```
</CodeGroup>

<Note>
The unique active index ensures one row per `(user, organization, provider)`. Reconnecting the same provider creates a new row and soft-deletes the old one.
</Note>

## Security & Authentication

### Webhook Authenticity

<Warning>
Provider webhook authentication is critical for preventing malicious calendar updates.
</Warning>

<Tabs>
  <Tab title="Google Calendar">
    Google stores a **separate** `watchChannelVerifyToken`:
    - Sent as `token` in `events.watch` request
    - Echoed as `X-Goog-Channel-Token` header
    - **Not** the same as channel `id` (legacy rows may incorrectly equate them)
    - When `watchResourceId` is present, must match `X-Goog-Resource-Id`
  </Tab>
  
  <Tab title="Microsoft Outlook">
    Outlook requires `clientState` validation:
    - Must equal the integration UUID
    - When Microsoft Graph includes `validationTokens` on change-notification batch:
      - JWTs are verified using JWKS
      - Handler returns `202` only after verification
      - Audience: `app.outlookCalendar.webhookValidationAudience` (defaults to calendar app client ID)
  </Tab>
</Tabs>

### Production Security Requirements

<Check>
In production (`buildAppConfig().isProduction`):
- `GOOGLE_CALENDAR_WEBHOOK_URL` must be HTTPS and not localhost
- `OUTLOOK_CALENDAR_WEBHOOK_URL` must be HTTPS and not localhost
</Check>

## Sync Architecture & Risk Points

### Durability Guarantees

<Info>
Outlook shadow mapping retries are durable using `calendar_sync_outbox` with pg-boss worker and backoff, not in-process timers.
</Info>

**Row-Level Security**: `calendar_sync_outbox` is RLS-protected like other org tables (`Migration20260428180000_EnableCalendarSyncOutboxRls`).

### Production Risk Areas

<Warning>
The system attempts to make two different calendar providers behave like one coherent event bus. This is inherently fragile.
</Warning>

**High-Risk Components**:
- In-memory cache management
- CRM-originated outbound sync without durable queue (listener path)
- Provider-specific edge cases:
  - Recurring events
  - All-day events
  - Deleted invitees
  - Outlook mailbox-local event IDs
