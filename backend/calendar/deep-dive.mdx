## Executive Summary

The calendar system operates as a distributed sync system with three distinct layers that manage the complexity of integrating with external calendar providers while maintaining data consistency and user experience.

<Warning>
The calendar integration is not a simple "push event to Google" feature. It is a distributed sync system with two hostile external providers, eventually consistent mailboxes, webhook retries, token expiry, RLS, notifications, and user actions happening from both CRM and external calendars.
</Warning>

### Three Calendar Layers

1. **CRM calendar feed**: Primary routes for personal workload, organization stacked calendar, and team calendars
2. **CRM events**: `EventService` owns the internal event source of truth, invitees, RSVP status, soft deletion, and notification emission
3. **Calendar integrations**: Google and Outlook drivers push CRM events outward and process provider webhooks back into CRM

## Architecture Overview

### Intended Model

<Check>
- CRM events are the canonical business record
- Creator calendar copies can update CRM event details when changed externally
- Invitee calendar copies can update RSVP only
- External-only items are displayed read-only and are not imported into CRM
- Provider echoes are suppressed with change IDs, fingerprints, ETags, and short time windows
</Check>

## API Routes

### Personal Calendar Feed

The primary route **`GET /v1/calendar/me`** serves personal workload including:
- CRM tasks/subtasks
- CRM events
- External items when viewing yourself

### Organization Calendar

**`GET /v1/calendar/org`** provides organization stacked calendar with:
- `userIds` parameter for 1-10 other org members
- Response format: `{ rosterUserIds, itemsByUserId }`
- Personal-scope tasks/subtasks + CRM events per subject
- No external items included
- Viewer ID in query is ignored

### Team Calendar

**`GET /v1/calendar/teams/:teamId`** requires:
- `team.admin`, `team_crm.*`, or `team_sales.*` permissions on that team
- Org admin/owner access
- CRM-only data per roster member
- Authenticated user excluded from roster (use `/me` for self)

### Legacy Routes

<Info>
Legacy **`GET /v1/calendar`** and **`GET /v1/calendar/agenda`** delegate to the same handlers as **`/me`** and **`/me/agenda`**.
</Info>

## Agenda Endpoints

### Personal Agenda

**`GET /v1/calendar/me/agenda`** serves the personal merged agenda list:
- Optional inclusive `from` parameter
- Forward scan up to server lookahead cap
- No `limit`/`cursor` pagination
- Response includes `truncated: true/false` indicator

<Note>
When `truncated: true`, the merge loop exited before reaching the lookahead horizon. When `truncated: false`, the scan ran through the full lookahead window.
</Note>

### Timeline Agenda

**`GET /v1/calendar/me/agenda/timeline`** serves the CRM web agenda list with day-window pagination:
- Merged probe → local `fromDay`/`toDay` in viewer `Time-Zone` header
- Full refetch for that window
- `nextCursor` for following page

## Personal Agenda Timeline Implementation

### Timezone Handling

<Steps>
<Step title="Timezone Header">
Requires the standard **`Time-Zone`** request header (IANA format). Invalid or missing values default to `UTC`.
</Step>

<Step title="Sort Logic">
- Tasks sorted by **due** date (`startDate` on DTO mirrors due)
- CRM events and externals sorted by **start** time
- Stable tie-break by source + id
</Step>

<Step title="Probe Window">
From the page anchor day, fetch a merged window of `CALENDAR_TIMELINE_PROBE_LOOKAHEAD_DAYS` (default 60) days, merge-sort, and apply first-page visibility rules.
</Step>
</Steps>

### Day Span Calculation

The system calculates the union of local calendar days each row touches:
- Date-only tasks use `dueDate` `YYYY-MM-DD` prefix
- Timed items use Luxon in the viewer zone
- All-day events walk date labels

<Warning>
`fromDay` is floored at the page anchor to prevent cursor repetition. Multi-day CRM events/externals can contribute keys before the anchor while still overlapping the probe window.
</Warning>

### Response Format

```json
{
  "window": {
    "fromDay": "2024-01-15",
    "toDay": "2024-01-20",
    "timeZone": "America/New_York"
  },
  "items": [...],
  "nextCursor": "base64-encoded-cursor",
  "hasMore": true
}
```

## Security & Authentication

### OAuth State Security

<Tabs>
<Tab title="Implementation">
Calendar OAuth flows use **`SignedPayloadService`** with HMAC-SHA256 signed state tokens. The state payload includes `{ userId, organizationId }` with purpose `calendar.oauth.state` and configurable TTL.
</Tab>

<Tab title="Benefits">
- Prevents cross-site request forgery
- Prevents replay attacks
- Secure token validation
</Tab>
</Tabs>

### Webhook Authenticity

#### Google Calendar

<CodeGroup>
```typescript Google Verification
// Google stores separate watchChannelVerifyToken
// Sent as 'token' in events.watch
// Echoed as X-Goog-Channel-Token
// NOT the same as channel id
```
</CodeGroup>

- Uses separate `watchChannelVerifyToken`
- Must match `X-Goog-Resource-Id` when `watchResourceId` is present
- Legacy rows may incorrectly equate token with channel ID

#### Outlook Calendar

<CodeGroup>
```typescript Outlook Verification
// Requires clientState to equal integration UUID
// Microsoft Graph validationTokens are JWT verified
// Uses JWKS before returning 202
```
</CodeGroup>

- Requires `clientState` to equal integration UUID
- JWT verification for `validationTokens` in change-notification batches
- Audience validation using `app.outlookCalendar.webhookValidationAudience`

<Warning>
In production (`buildAppConfig().isProduction`), both `GOOGLE_CALENDAR_WEBHOOK_URL` and `OUTLOOK_CALENDAR_WEBHOOK_URL` must be HTTPS and must not target localhost.
</Warning>

## Event Access Control

### Permission Model

Events use a creator + invitee + org-calendar policy model:

<AccordionGroup>
<Accordion title="ACCESS Permissions">
- Creator of the event
- Invitee to the event
- Org-calendar policy (same entity-type filters as `GET /v1/calendar/org`)
- Team-calendar visibility
</Accordion>

<Accordion title="MODIFY/DELETE Permissions">
- Creator only
- System admins bypass via OR-mode permissions
</Accordion>

<Accordion title="RSVP Permissions">
`PATCH /events/:id/rsvp` uses the same OR logic as ACCESS permissions
</Accordion>
</AccordionGroup>

## Data Synchronization

### Sync Outbox

<Info>
**`calendar_sync_outbox`** operations use durable pg-boss workers with backoff, not in-process timers. The table is RLS-protected like other org tables.
</Info>

Outlook shadow mapping retries are durable:
- `calendar_sync_outbox` operation `outlook-shadow-reconcile`
- pg-boss execute worker with backoff
- RLS protection enabled

### Echo Suppression

Provider echoes are suppressed using multiple mechanisms:
- Change IDs
- Fingerprints
- ETags
- Short time windows

## Risk Assessment

<Warning>
The biggest risk is that the system is trying to make two different calendar providers behave like one coherent event bus. This is inherently fragile.
</Warning>

### High-Risk Areas

1. **In-memory cache** - Potential data loss during restarts
2. **CRM-originated outbound sync** - No durable queue for listener path
3. **Provider-specific edge cases**:
   - Recurring events
   - All-day events
   - Deleted invitees
   - Outlook mailbox-local event IDs

### Mitigations

The code includes several guardrails:
- Webhook authenticity verification
- Durable retry mechanisms for critical operations
- RLS protection on sync tables
- Comprehensive error handling
