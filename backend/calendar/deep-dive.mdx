<Warning>
The calendar integration is not a simple "push event to Google" feature. It is a distributed sync system with two hostile external providers, eventually consistent mailboxes, webhook retries, token expiry, RLS, notifications, and user actions happening from both CRM and external calendars.
</Warning>

## Architecture Overview

The system has three calendar layers that work together to provide a unified calendar experience:

  <Card title="CRM Calendar Feed" icon="calendar">
    Primary routes for personal, organization, and team calendar views
  </Card>
  <Card title="CRM Events" icon="calendar-days">
    Internal event source of truth with invitees, RSVP status, and notifications
  </Card>
  <Card title="Calendar Integrations" icon="arrows-rotate">
    Google and Outlook drivers for bidirectional sync
  </Card>
</CardGroup>

### Intended Data Model

<Steps>
  <Step title="CRM Events as Canonical Source">
    CRM events are the canonical business record for all calendar data
  </Step>
  <Step title="Creator Calendar Updates">
    Creator calendar copies can update CRM event details when changed externally
  </Step>
  <Step title="Invitee RSVP Only">
    Invitee calendar copies can only update RSVP status
  </Step>
  <Step title="Read-Only External Items">
    External-only items are displayed read-only and are not imported into CRM
  </Step>
  <Step title="Echo Suppression">
    Provider echoes are suppressed with change IDs, fingerprints, ETags, and short time windows
  </Step>
</Steps>

## Calendar Feed Endpoints

### Personal Calendar (`GET /v1/calendar/me`)

Returns personal workload including:
- CRM tasks and subtasks
- CRM events
- External calendar items (when viewing yourself)

### Organization Calendar (`GET /v1/calendar/org`)

<Note>
**Stacked calendar view**: Query parameter `userIds` accepts 1–10 other org members. Response format: `{ rosterUserIds, itemsByUserId }`
</Note>

Returns per-user data including:
- Personal-scope tasks/subtasks
- CRM events per subject
- **No external items included**
- Viewer ID in query is ignored

### Team Calendar (`GET /v1/calendar/teams/:teamId`)

<Info>
Requires `team.admin`, `team_crm.*`, or `team_sales.*` permissions on the team, or org admin/owner role
</Info>

- Shows CRM-only data per roster member
- **Authenticated user is excluded from roster**
- Use `/me` endpoint for self-calendar view

### Legacy Endpoints

- `GET /v1/calendar` → delegates to `/me` handler
- `GET /v1/calendar/agenda` → delegates to `/me/agenda` handler

## Agenda Endpoints

### Personal Agenda (`GET /v1/calendar/me/agenda`)

<Tabs>
  <Tab title="Features">
    - Optional inclusive `from` parameter
    - Forward scan up to server lookahead cap
    - **No** `limit`/`cursor` pagination
    - Returns `truncated` boolean indicator
  </Tab>
  <Tab title="Response Format">
    ```json
    {
      "items": [...],
      "truncated": true  // or false
    }
    ```
    - `truncated: true` → merge loop exited before reaching lookahead horizon
    - `truncated: false` → scan completed full lookahead window
  </Tab>
</Tabs>

### Timeline Agenda (`GET /v1/calendar/me/agenda/timeline`)

<Warning>
Used by CRM web agenda list with **day-window pagination**. This is the primary endpoint for infinite scroll functionality.
</Warning>

<Steps>
  <Step title="Viewer Zone Setup">
    Requires `Time-Zone` header (IANA format). Invalid/missing defaults to UTC
  </Step>
  <Step title="Probe Window">
    Fetches merged window of `CALENDAR_TIMELINE_PROBE_LOOKAHEAD_DAYS` (default 60 days)
  </Step>
  <Step title="Day Span Calculation">
    Union of local calendar days each item touches, with `fromDay` floored at page anchor
  </Step>
  <Step title="Authoritative Fetch">
    Full refetch for calculated window with interval overlap logic
  </Step>
  <Step title="Response Generation">
    Returns `nextCursor` for following page with day-based pagination
  </Step>
</Steps>

#### Sort Order
- **Tasks**: By due date (`startDate` on DTO mirrors due)
- **CRM Events & Externals**: By start time
- **Tie-breaker**: Stable sort by source + ID

### Webhook Security

<Tabs>
  <Tab title="Google Calendar">
    - Stores separate `watchChannelVerifyToken`
    - Sent as `token` in `events.watch`
    - Echoed as `X-Goog-Channel-Token`
    - **Not the same as channel ID**
    - Requires `X-Goog-Resource-Id` match when `watchResourceId` present
  </Tab>
  
  <Tab title="Outlook Calendar">
    - Requires `clientState` to equal integration UUID
    - Microsoft Graph `validationTokens` verified as JWTs (JWKS)
    - Audience: `app.outlookCalendar.webhookValidationAudience`
    - Returns `202` after JWT verification
  </Tab>
</Tabs>

<Warning>
In **production** (`buildAppConfig().isProduction`):
- `GOOGLE_CALENDAR_WEBHOOK_URL` must be **https** and not localhost
- `OUTLOOK_CALENDAR_WEBHOOK_URL` must be **https** and not localhost
</Warning>

## OAuth State Security

<Steps>
  <Step title="Intent Creation">
    SPA calls authenticated `POST /auth/google-calendar/intent` or `POST /auth/outlook-calendar/intent`
  </Step>
  <Step title="Intent Storage">
    Backend stores short-lived `calendar_oauth_intent` row and returns opaque `intentId`
  </Step>
  <Step title="Authorization">
    Popup form POSTs `intentId` to `/authorize` endpoint
  </Step>
  <Step title="Atomic Consumption">
    `/authorize` atomically consumes row with `UPDATE ... RETURNING` before redirecting
  </Step>
</Steps>

## Sync Mechanisms

### Outbound Sync

<Note>
**Risk**: CRM-originated outbound sync without durable queue on listener path
</Note>

- Events pushed from CRM to external calendars
- Change detection using fingerprints and ETags
- Echo suppression with change IDs and time windows

### Inbound Sync

- Webhook-driven updates from external providers
- Provider-specific edge cases around:
  - Recurring events
  - All-day events  
  - Deleted invitees
  - Outlook mailbox-local event IDs

### Durable Operations

<Check>
**Outlook shadow mapping retries are durable**: Uses `calendar_sync_outbox` with pg-boss execute worker (with backoff), not in-process timers
</Check>

## Production Risks

<Warning>
**Critical production risks to monitor:**

1. **Long-lived tenant transactions** that hold DB connections while polling Google/Graph
2. **CRM-originated outbound sync** without durable queue (listener path)  
3. **Provider-specific edge cases** around recurring events, all-day events, deleted invitees
4. **Outlook mailbox-local event IDs** causing sync conflicts
5. **Webhook authenticity** validation failures
6. **Token expiry** during critical sync operations
</Warning>

### Operational Considerations

<Tabs>
  <Tab title="Monitoring">
    - Watch for webhook delivery failures
    - Monitor OAuth token refresh rates
    - Track sync health columns
    - Alert on long-running transactions
  </Tab>
  
  <Tab title="Throttling">
    - Public calendar webhooks are throttled
    - Outlook batches are capped before async sync fan-out
    - Provider rate limits must be respected
  </Tab>
  
  <Tab title="Error Handling">
    - Durable retry mechanisms for critical operations
    - Graceful degradation when providers are unavailable
    - Clear error reporting for OAuth failures
  </Tab>
</Tabs>

## Timeline Middleware

<Info>
All calendar endpoints support the `Time-Zone` request header (IANA timezone). The `TimezoneMiddleware` validates the header and falls back to UTC for invalid values. This affects task list buckets and calendar timeline day calculations.
</Info>

The personal merge helper `CalendarService.collectMergedCalendarItems` is **personal-merge only** and includes:
- Tasks + CRM events for the subject
- External items when subject is the viewer

Organization stacked feeds use `getOrgCalendarItems`, not the personal merge helper.
