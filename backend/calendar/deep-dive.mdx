<Warning>
The calendar integration is not a simple "push event to Google" feature. It is a distributed sync system with two hostile external providers, eventually consistent mailboxes, webhook retries, token expiry, RLS, notifications, and user actions happening from both CRM and external calendars.
</Warning>

## Executive summary

The system has three calendar layers:

1. **CRM calendar feed**: Primary routes for personal workload, organization stacked calendar, and team calendars
2. **CRM events**: `EventService` owns the internal event source of truth, invitees, RSVP status, soft deletion, and notification emission
3. **Calendar integrations**: Google and Outlook drivers push CRM events outward and process provider webhooks back into CRM

<Info>
The intended model is that CRM events are the canonical business record, with calendar integrations providing bidirectional sync capabilities while maintaining data integrity.
</Info>

## Calendar API endpoints

### Personal calendar endpoints

<CodeGroup>

```http GET /v1/calendar/me
Personal workload: CRM tasks/subtasks + CRM events + external items when viewing yourself
```

```http GET /v1/calendar/me/agenda
Personal merged agenda list with forward scan up to server lookahead cap
- Optional inclusive `from` parameter
- No `limit`/`cursor` pagination
- Returns `truncated: true/false` to indicate if more items exist
```

```http GET /v1/calendar/me/agenda/timeline
CRM web agenda list with day-window pagination
- Uses viewer `Time-Zone` header
- Day-window pagination with `nextCursor`
- Infinite scroll support
```

</CodeGroup>

### Organization and team endpoints

<CodeGroup>

```http GET /v1/calendar/org
Organization stacked calendar
- `userIds` parameter for 1-10 other org members
- Response: `{ rosterUserIds, itemsByUserId }`
- Personal-scope tasks/subtasks + CRM events per subject
- No externals included
```

```http GET /v1/calendar/teams/:teamId
Team calendar access
- Requires `team.admin`, `team_crm.*`, `team_sales.*`, or org admin/owner
- CRM-only per roster member
- Authenticated user excluded from roster (use `/me` for self)
```

</CodeGroup>

<Note>
Legacy endpoints `GET /v1/calendar` and `GET /v1/calendar/agenda` delegate to the same handlers as `/me` and `/me/agenda` respectively.
</Note>

## Calendar integration architecture

### Core services

The calendar integration is built around several key services:

<AccordionGroup>
<Accordion title="CalendarIntegrationService">
Main orchestration service that coordinates between CRM events and external calendar providers. Handles the high-level sync logic and provider abstraction.
</Accordion>

<Accordion title="EventCalendarSyncService">
Manages the bidirectional sync between CRM events and external calendars. Handles change detection, conflict resolution, and sync state management.
</Accordion>

<Accordion title="CalendarWebhookService">
Processes incoming webhooks from Google Calendar and Outlook. Implements echo suppression and handles provider-specific webhook formats.
</Accordion>

<Accordion title="CalendarWatchRenewalHandler">
Background service that maintains webhook subscriptions with external providers before they expire.
</Accordion>
</AccordionGroup>

### Provider drivers

<Tabs>
<Tab title="Google Calendar">
- **File**: `src/modules/integrations/calendar-integration/drivers/google-calendar.driver.ts`
- **Authentication**: OAuth 2.0 with refresh tokens
- **Webhook verification**: Separate `watchChannelVerifyToken` (sent as `token` in `events.watch`, echoed as `X-Goog-Channel-Token`)
- **Resource matching**: `watchResourceId` must match `X-Goog-Resource-Id` header
</Tab>

<Tab title="Outlook Calendar">
- **File**: `src/modules/integrations/calendar-integration/drivers/outlook-calendar.driver.ts`
- **Authentication**: Microsoft Graph OAuth 2.0
- **Webhook verification**: `clientState` must equal integration UUID
- **JWT validation**: Microsoft Graph `validationTokens` verified via JWKS in production
- **Shadow mapping**: Uses durable retry via `calendar_sync_outbox` and pg-boss worker
</Tab>
</Tabs>

## OAuth flow security

Calendar OAuth flows use `SignedPayloadService` with HMAC-SHA256 signed state tokens to prevent CSRF and replay attacks.

<Steps>
<Step title="State token generation">
The service creates a signed token with:
- Purpose: `calendar.oauth.state`
- Payload: `{ userId, organizationId }`
- TTL: Configurable via `CALENDAR_OAUTH_STATE_TTL_MS`
</Step>

<Step title="OAuth redirect">
User is redirected to provider with the signed state token
</Step>

<Step title="Callback verification">
On callback, the state token is verified for authenticity and expiration before processing the authorization code
</Step>
</Tabs>

<Warning>
In production (`buildAppConfig().isProduction`), `GOOGLE_CALENDAR_WEBHOOK_URL` and `OUTLOOK_CALENDAR_WEBHOOK_URL` must be HTTPS and must not target localhost.
</Warning>

## Personal agenda timeline

The timeline endpoint (`GET /v1/calendar/me/agenda/timeline`) powers the CRM web agenda view with infinite scroll capabilities.

### Request parameters

<AccordionGroup>
<Accordion title="Time-Zone header">
**Required**: IANA timezone identifier (e.g., `America/New_York`)
- Invalid/missing values fall back to `UTC`
- Processed by `TimezoneMiddleware`
</Accordion>

<Accordion title="Cursor pagination">
- `cursor`: Base64 JSON with `{ v, nextDay }`
- `nextDay`: Local calendar day after the current page's `toDay`
- No `limit` parameter - uses server-configured page size
</Accordion>
</AccordionGroup>

### Timeline algorithm

<Steps>
<Step title="Probe window">
From the page anchor day, fetch a merged window of `CALENDAR_TIMELINE_PROBE_LOOKAHEAD_DAYS` (default 60) days
</Step>

<Step title="Sort and filter">
- Tasks by due date (`startDate` on DTO mirrors due)
- CRM events and externals by start time
- Apply visibility rules: `max(from, now)` for tasks, overlap with viewer's local "today" for events
</Step>

<Step title="Day span calculation">
Union of local calendar days each row touches, floored at page anchor to prevent cursor repetition
</Step>

<Step title="Authoritative fetch">
`collectMergedCalendarItems` for the calculated day span in viewer timezone
</Step>

<Step title="Response format">
```json
{
  "window": { "fromDay": "2024-01-15", "toDay": "2024-01-20", "timeZone": "America/New_York" },
  "items": [...],
  "nextCursor": "eyJ2IjoxLCJuZXh0RGF5IjoiMjAyNC0wMS0yMSJ9",
  "hasMore": true
}
```
</Step>
</Steps>

## Sync mechanisms

### Echo suppression

The system implements multiple strategies to prevent sync loops between CRM and external calendars:

<Tabs>
<Tab title="Change IDs">
Track provider-specific change identifiers to detect when an update originated from the same system
</Tab>

<Tab title="Fingerprints">
Hash event content to identify unchanged events and avoid unnecessary sync operations
</Tab>

<Tab title="ETags">
Use provider ETags for optimistic concurrency control and change detection
</Tab>

<Tab title="Time windows">
Short time windows around sync operations to suppress immediate echoes
</Tab>
</Tabs>

### Outbound sync

<Warning>
CRM-originated outbound sync uses in-process listeners without durable queuing on the primary path. Outlook shadow mapping retries are durable via `calendar_sync_outbox` and pg-boss workers.
</Warning>

### Webhook processing

<Steps>
<Step title="Webhook authentication">
- Google: Verify `X-Goog-Channel-Token` matches stored `watchChannelVerifyToken`
- Outlook: Verify `clientState` equals integration UUID
- Microsoft Graph: Validate JWT tokens in production environments
</Step>

<Step title="Echo suppression">
Check change IDs, fingerprints, and timing to avoid processing self-originated changes
</Step>

<Step title="Sync processing">
Apply changes to CRM events based on user permissions:
- Creator: Can update all event details
- Invitee: Can only update RSVP status
- Read-only: External-only items displayed but not imported
</Step>
</Steps>

## Production considerations

### Known risks

<Warning>
The system tries to make two different calendar providers behave like one coherent event bus, which is inherently fragile.
</Warning>

<AccordionGroup>
<Accordion title="In-memory cache">
Calendar cache is stored in memory and not persisted across restarts, potentially causing temporary inconsistencies
</Accordion>

<Accordion title="Sync queue durability">
Primary outbound sync path lacks durable queueing, though Outlook shadow reconciliation uses pg-boss for reliability
</Accordion>

<Accordion title="Provider edge cases">
- Recurring events have provider-specific handling
- All-day events require careful timezone handling
- Deleted invitees may not sync properly across all providers
- Outlook uses mailbox-local event IDs that can cause mapping issues
</Accordion>
</AccordionGroup>

### Security measures

<Check>
**RLS Protection**: `calendar_sync_outbox` is protected by row-level security like other org tables
</Check>

<Check>
**HTTPS Requirements**: Production webhook URLs must use HTTPS and cannot target localhost
</Check>

<Check>
**Token Encryption**: OAuth tokens are encrypted at rest in the database
</Check>

### Monitoring and health

The system tracks sync health metrics:

- `syncHealthLastSuccess`: Timestamp of last successful sync
- `syncHealthConsecutiveFailures`: Counter for consecutive sync failures
- Webhook subscription renewal before expiration
- Provider-specific error handling and retry logic
