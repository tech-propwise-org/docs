This document explains how the current calendar system actually works. It is intentionally blunt: the calendar integration is not a simple "push event to Google" feature. It is a distributed sync system with two hostile external providers, eventually consistent mailboxes, webhook retries, token expiry, RLS, notifications, and user actions happening from both CRM and external calendars.

## Executive Summary

The system has three calendar layers:

  <Card title="CRM Calendar Feed" icon="calendar">
    Primary routes for personal workload, organization stacked calendar, and team calendars
  </Card>
  <Card title="CRM Events" icon="calendar-days">

  </Card>
  <Card title="Calendar Integrations" icon="link">

### Calendar Feed Routes

The CRM calendar system provides several API endpoints:

- **`GET /v1/calendar/me`** - Personal workload: CRM tasks/subtasks + CRM events + external items when viewing yourself
- **`GET /v1/calendar/org`** - Organization stacked calendar: 1â€“10 other org members, returns `{ rosterUserIds, itemsByUserId }`
- **`GET /v1/calendar/teams/:teamId`** - Team calendar for members with appropriate permissions
- **`GET /v1/calendar/me/agenda`** - Personal merged agenda list with forward scan
- **`GET /v1/calendar/me/agenda/timeline`** - CRM web agenda list with day-window pagination

<Note>
Legacy `GET /v1/calendar` and `GET /v1/calendar/agenda` delegate to the same handlers as `/me` endpoints.
</Note>

### Intended Model

The calendar system follows these principles:

- CRM events are the canonical business record
- Creator calendar copies can update CRM event details when changed externally
- Invitee calendar copies can update RSVP only
- External-only items are displayed read-only and are not imported into CRM
- Provider echoes are suppressed with change IDs, fingerprints, ETags, and short time windows

<Warning>
The biggest risk is that the system is trying to make two different calendar providers behave like one coherent event bus. That is inherently fragile.
</Warning>

## Security & Authentication

### Webhook Authenticity

<Tabs>
  <Tab title="Google Calendar">
    Google stores a **separate** `watchChannelVerifyToken` (sent as the `token` in `events.watch` and echoed as `X-Goog-Channel-Token`). It is not the same as the channel `id`â€”legacy rows may still equate the two.

    When `watchResourceId` is present, inbound notifications must also match `X-Goog-Resource-Id`.
  </Tab>
  <Tab title="Outlook">
    Outlook requires `clientState` to equal the integration UUID.

    When Microsoft Graph includes `validationTokens` on a change-notification batch, JWTs are verified (JWKS) before the handler returns `202` (audience: `app.outlookCalendar.webhookValidationAudience`, defaulting to the calendar app client id).
  </Tab>
</Tabs>

<Check>
In **production** (`buildAppConfig().isProduction`), `GOOGLE_CALENDAR_WEBHOOK_URL` and `OUTLOOK_CALENDAR_WEBHOOK_URL` must be **https** and must not target localhost.
</Check>

## Personal Agenda Timeline

The `GET /v1/calendar/me/agenda/timeline` endpoint is used by the CRM web **agenda** view for infinite scroll.

### Request Requirements

<Steps>
  <Step title="Timezone Header">
    Requires the standard **`Time-Zone`** request header (IANA). Invalid/missing values default to `UTC`.
  </Step>
  <Step title="Sorting Logic">
    - Tasks by **due** (`startDate` on the DTO mirrors due)
    - CRM events and externals by **start**
    - Stable tie-break by source + id
  </Step>
</Steps>

### Processing Flow

<AccordionGroup>
  <Accordion title="Probe Phase">
    From the page anchor day, fetch a merged window of `CALENDAR_TIMELINE_PROBE_LOOKAHEAD_DAYS` (default 60) days, merge-sort, apply the first-page visibility rule (`max(from, now)` for tasks; events/externals also kept when they overlap the viewer's local "today").
  </Accordion>
  <Accordion title="Day Span Calculation">
    Union of local calendar days each row touches. `fromDay` is floored at the page anchor to prevent cursor repetition. `D_min` / `D_max` are clamped to `CALENDAR_TIMELINE_MAX_DAY_SPAN` days.
  </Accordion>
  <Accordion title="Authoritative Fetch">
    `collectMergedCalendarItems` for `[startOf(D_min), endOf(D_max)]` in the viewer zone. CRM events use interval overlap in `EventService.getCalendarEvents`.
  </Accordion>
</AccordionGroup>

### Response Format

```json
{
  "window": {
    "fromDay": "2024-01-15",
    "toDay": "2024-02-14",
    "timeZone": "America/New_York"
  },
  "items": [...],
  "nextCursor": "base64-encoded-cursor",
  "hasMore": true
}
```

<Note>
The legacy `GET /v1/calendar/me/agenda` endpoint remains for callers that want a single forward scan without pagination.
</Note>

## Data Model

### Calendar Integration Entity

The `calendar_integration` table stores one active external calendar account per user, organization, and provider.

<Tabs>
  <Tab title="Core Fields">
    - `user`: CRM user who connected the calendar
    - `organization`: tenant boundary
    - `provider`: `google` or `outlook`
    - `calendarEmail`: external account email for display and attendee matching
  </Tab>
  <Tab title="Security Fields">
    - `encryptedAccessToken`: AES-GCM encrypted access token (nullable after disconnect)
    - `encryptedRefreshToken`: AES-GCM encrypted refresh token when issued
    - `tokenExpiresAt`: refresh threshold source
  </Tab>
  <Tab title="Google Fields">
    - `watchChannelId`: Google push watch channel ID
    - `watchChannelVerifyToken`: Opaque secret echoed as `X-Goog-Channel-Token`
    - `watchResourceId`: Google resource identifier
    - `watchExpiration`: Watch expiration timestamp
    - `googleSyncToken`: Google incremental sync token
  </Tab>
  <Tab title="Outlook Fields">
    - `outlookSubscriptionId`: Microsoft Graph subscription ID
    - `outlookSubscriptionExpiration`: Subscription expiration timestamp
  </Tab>
  <Tab title="Health Fields">
    - `lastInboundWebhookAt`: Last successful webhook processing
    - `lastOutboundSyncAt`: Last successful CRMâ†’provider push
    - `lastSyncErrorAt`: Last provider/webhook/processing failure
    - `lastSyncErrorMessage`: Bounded error message
    - `isActive`, `isDeleted`: Connection status and soft delete
  </Tab>
</Tabs>

## Known Risks & Considerations

<Tip>
The code has several good guardrails, but some areas are still production-dangerous.
</Tip>

### High-Risk Areas

- **In-memory cache** - Can lead to inconsistencies
- **CRM-originated outbound sync without durable queue** - Listener path vulnerability
- **Provider-specific edge cases** around:
  - Recurring events
  - All-day events
  - Deleted invitees
  - Outlook mailbox-local event IDs

### Outlook Shadow Mapping

<Check>
Outlook shadow mapping retries are durable: `calendar_sync_outbox` operation `outlook-shadow-reconcile` + pg-boss execute worker (with backoff), not in-process timers.
</Check>

- `src/modules/crm/calendar/calendar-timeline.constants.ts` - Timeline pagination settings

<Note>
