**Architecture Complexity Notice**: The calendar integration is not a simple "push event to Google" feature. It is a distributed sync system with two hostile external providers, eventually consistent mailboxes, webhook retries, token expiry, RLS, notifications, and user actions happening from both CRM and external calendars.

### Calendar feed routes

The CRM calendar feed provides three main access patterns:

- **`GET /v1/calendar/me`** — Personal workload: CRM tasks/subtasks + CRM events + external items when viewing yourself
- **`GET /v1/calendar/org`** — Organization stacked calendar: `userIds` = 1–10 **other** org members, response `{ rosterUserIds, itemsByUserId }` — personal-scope tasks/subtasks + CRM events per subject, no externals; viewer id in the query is ignored
- **`GET /v1/calendar/teams/:teamId`** — `team.admin`, `team_crm.*`, or `team_sales.*` on that team, or org admin/owner: CRM-only per **other** roster member — the authenticated user is excluded from the roster; use **`/me`** for self

<Note>
Legacy **`GET /v1/calendar`** and **`GET /v1/calendar/agenda`** delegate to the same handlers as **`/me`** and **`/me/agenda`**.
</Note>

### Agenda endpoints

**`GET /v1/calendar/me/agenda`** (and legacy `/calendar/agenda`) serve the personal merged agenda list:
- Optional inclusive `from`
- Forward scan up to a server lookahead cap
- **No** `limit`/`cursor` pagination (see `calendar-agenda.constants.ts`)
- Response includes **`truncated: true`** when the merge loop exits before reaching the lookahead horizon (iteration cap)
- **`truncated: false`** when the scan ran through the full lookahead window

<Info>
Clients can use the `truncated` flag to show that more items may exist beyond what was merged.
</Info>

**`GET /v1/calendar/me/agenda/timeline`** serves the **CRM web agenda list** with **day-window pagination**:
- Merged probe → local `fromDay`/`toDay` in the viewer `Time-Zone` header
- Full refetch for that window → `nextCursor` for the following page
- See `calendar-timeline.*` for implementation details

### Intended model

The calendar integration follows these principles:

<Steps>
  <Step title="CRM events as canonical source">
    CRM events are the canonical business record
  </Step>
  <Step title="Creator calendar updates">
    Creator calendar copies can update CRM event details when changed externally
  </Step>
  <Step title="Invitee RSVP updates">
    Invitee calendar copies can update RSVP only
  </Step>
  <Step title="External items read-only">
    External-only items are displayed read-only and are not imported into CRM
  </Step>
  <Step title="Echo suppression">
    Provider echoes are suppressed with change IDs, fingerprints, ETags, and short time windows
  </Step>
</Steps>

### System risks

<Warning>
**Primary Risk**: The system is trying to make two different calendar providers behave like one coherent event bus. That is inherently fragile.
</Warning>

The code has several good guardrails, but some areas are still production-dangerous:

- CRM-originated outbound sync without durable queue (listener path)
- Remaining provider HTTP inside tenant transactions (now **without** multi-second Outlook shadow polling)
- Provider-specific edge cases around recurring events, all-day events, deleted invitees, and Outlook mailbox-local event IDs

### Webhook authenticity

<Tabs>
  <Tab title="Google Calendar">
    Google stores a **separate** `watchChannelVerifyToken`:
    - Sent as the `token` in `events.watch`
    - Echoed as `X-Goog-Channel-Token`
    - **Not** the same as the channel `id` (legacy rows may still equate the two)
    
    When `watchResourceId` is present, inbound notifications must also match `X-Goog-Resource-Id`.
  </Tab>
  <Tab title="Outlook Calendar">
    Outlook requires `clientState` to equal the integration UUID.
    
    When Microsoft Graph includes **`validationTokens`** on a change-notification batch:
    - JWTs are verified (JWKS) before the handler returns `202`
    - Audience: `app.outlookCalendar.webhookValidationAudience`
    - Defaults to the calendar app client id
  </Tab>
</Tabs>

<Check>
In **production** (`buildAppConfig().isProduction`), `GOOGLE_CALENDAR_WEBHOOK_URL` and `OUTLOOK_CALENDAR_WEBHOOK_URL` must be **https** and must not target localhost.
</Check>

### Additional safeguards

- Public calendar webhooks are throttled
- Outlook batches are capped before async sync fan-out
- **Outlook shadow mapping retries are durable**: `calendar_sync_outbox` operation `outlook-shadow-reconcile` + pg-boss execute worker (with backoff), not in-process timers
- **`calendar_sync_outbox`, `calendar_external_cache`, and `calendar_oauth_intent` are RLS-protected** like other org tables

## Personal agenda timeline

**Endpoint**: `GET /v1/calendar/me/agenda/timeline`

Used by the CRM web **agenda** view for infinite scroll. Returns same merged shapes as **`/calendar/me`** (`CalendarItemDto`).

### Timeline behavior

<Tabs>
  <Tab title="Viewer zone">
    Requires the standard **`Time-Zone`** request header (IANA format).
    
    - Invalid/missing timezone → defaults to `UTC`
    - See `TimezoneMiddleware` / `TimezoneParam`
  </Tab>
  <Tab title="Sort order">
    **Sort rules** (`calendar-timeline.util.ts`):
    - **Tasks**: By **due** (`startDate` on the DTO mirrors due)
    - **CRM events and externals**: By **start**
    - **Stable tie-break**: By source + id
  </Tab>
  <Tab title="Probe logic">
    **Probe algorithm**:
    
    1. From the page anchor day, fetch a merged window of **`CALENDAR_TIMELINE_PROBE_LOOKAHEAD_DAYS`** (default 60) days
    2. Merge-sort all items
    3. Apply first-page visibility rule:
       - `max(from, now)` for tasks
       - Events/externals kept when they **overlap the viewer's local "today"** even if `start` is before the floor
    4. Take the first **`CALENDAR_TIMELINE_PAGE_TARGET`** (40) rows
  </Tab>
</Tabs>

### Day span calculation

**Union of local calendar days** each row touches:
- **Date-only tasks**: Use `dueDate` `YYYY-MM-DD` prefix
- **Timed items**: Use Luxon in the viewer zone
- **All-day events**: Walk date labels

<Info>
**`fromDay` is floored at the page anchor** (`anchorBoundedDaySpan`): Multi-day CRM events/externals can contribute keys **before** the anchor while still overlapping the probe window.

Without the floor, clamp + `nextCursor` could repeat and clients would refetch the same cursor.
</Info>

`D_min` / `D_max` are then clamped to **`CALENDAR_TIMELINE_MAX_DAY_SPAN`** days.

### Authoritative fetch

`collectMergedCalendarItems` for `[startOf(D_min), endOf(D_max)]` in the viewer zone.

<Note>
CRM events already use **interval overlap** in `EventService.getCalendarEvents`.
</Note>

### Response format

```typescript
{
  window: {
    fromDay: string,      // YYYY-MM-DD
    toDay: string,        // YYYY-MM-DD
    timeZone: string      // IANA timezone
  },
  items: CalendarItemDto[],
  nextCursor: string | null,
  hasMore: boolean
}
```

#### Timeline items structure

Timeline items include optional **`agendaDayKeys: string[]`** for this endpoint only:

- Server-authoritative viewer-local day buckets the row should render in
- Clipped to the response window (`fromDay`–`toDay`)
- Later pages cannot inject content into already-rendered earlier days

<Warning>
**Items whose clipped `agendaDayKeys` is empty are dropped server-side** before the response is built.

They do not belong to this page window and will appear correctly on the next page.
</Warning>

#### Client handling rules

<Steps>
  <Step title="Group by agendaDayKeys">
    Clients must group agenda rows by `agendaDayKeys` when present
  </Step>
  <Step title="Skip empty arrays">
    If `agendaDayKeys` is a defined but empty array, the row must be skipped entirely
    
    Do **not** fall back to browser-local date parsing, which would misplace the row
  </Step>
  <Step title="Fallback for missing keys">
    Browser-local parsing is only a fallback for rows where `agendaDayKeys` is entirely absent (non-timeline callers, legacy paths)
  </Step>
  <Step title="Use response window">
    Use the response `window.toDay` as the loaded-through boundary
  </Step>
</Steps>

#### Next cursor format

`nextCursor` is base64 JSON with:
```json
{
  "v": 1,
  "nextDay": "YYYY-MM-DD"  // Local calendar day after toDay
}
```

### West-of-UTC all-day leak (fixed)

<Warning>
**Historical issue**: All-day CRM events are stored as UTC midnight (`T00:00:00.000Z`).

The authoritative UTC refetch range for a page ending at local `toDay` extends to `endOf(toDay)` in the viewer zone, which for zones west of UTC (e.g., America/Los_Angeles) is already **into the next UTC day**.
</Warning>

**Example scenario**:
- All-day event on `2026-05-15` stored as `2026-05-15T00:00:00.000Z`
- Page window `toDay = 2026-05-14` in `America/Los_Angeles`
- `endOf(2026-05-14)` in LA = `2026-05-15T06:59:59.999Z`
- Event at `2026-05-15T00:00:00.000Z` falls within the fetch range
- Without proper clipping, it would render in the current page instead of waiting for `nextCursor`

**Solution**: The `agendaDayKeys` clipping ensures:
- Event gets `agendaDayKeys = ["2026-05-15"]`
- Page window is `["2026-05-14"]`
- Clipped intersection is empty
- Event is dropped from the current response
- Event appears correctly on the next page when `fromDay = 2026-05-15`

<Check>
This fix prevents items from leaking backward into already-loaded pages.
</Check>

## CRM events deep dive

### Event lifecycle

<Steps>
  <Step title="Creation">
    **EventService** creates the internal event source of truth
    - Assigns unique event ID
    - Stores event details (title, description, times, location)
    - Records creator and invitees
    - Sets initial RSVP status for all invitees
  </Step>
  <Step title="Invitations">
    System sends notifications to invitees
    - Uses `event-notification-recipients.util.ts` for RSVP-aware recipient sets
    - Distinguishes between CRM notifications and time-change notifications
  </Step>
  <Step title="Calendar sync">
    **EventCalendarSyncService** pushes to external calendars
    - Creator gets full event control in their calendar
    - Invitees get read-only copies in their calendars
    - Sync operations are tracked in `calendar_event_mapping`
  </Step>
  <Step title="Updates">
    Changes can originate from CRM or external calendars
    - Creator changes in external calendar → updates CRM event
    - Invitee changes in external calendar → updates RSVP only
    - CRM changes → pushed to all external calendars
    - Echo suppression prevents infinite loops
  </Step>
  <Step title="Deletion">
    Events support soft deletion
    - Marked as deleted in CRM
    - Removed from external calendars
    - Historical data preserved for audit
  </Step>
</Steps>

### RSVP status tracking

Each invitee maintains an RSVP status stored in the `event_invitee` table:

| Status | Meaning | Can update via external calendar? |
|--------|---------|-----------------------------------|
| `pending` | Not yet responded | ✅ Yes |
| `accepted` | Confirmed attendance | ✅ Yes |
| `tentative` | Maybe attending | ✅ Yes |
| `declined` | Not attending | ✅ Yes |

<Note>
When an invitee updates their RSVP in Google Calendar or Outlook, the webhook handler:
1. Identifies the user via calendar integration
2. Locates the corresponding CRM event via `calendar_event_mapping`
3. Updates only the RSVP status (not event details)
4. Emits notification if configured
</Note>

### Driver registry

**Location**: `src/modules/integrations/calendar-integration/drivers/calendar-driver.registry.ts`

The registry provides a factory pattern for driver instantiation:

```typescript
class CalendarDriverRegistry {
  private drivers: Map<CalendarProvider, CalendarDriver>;
  
  getDriver(provider: CalendarProvider): CalendarDriver {
    const driver = this.drivers.get(provider);
    if (!driver) {
      throw new Error(`No driver registered for provider: ${provider}`);
    }
    return driver;
  }
  
  registerDriver(provider: CalendarProvider, driver: CalendarDriver): void {
    this.drivers.set(provider, driver);
  }
}
```

<Check>
Drivers are registered at module initialization time, ensuring they're available before any calendar operations occur.
</Check>

### OAuth flow

<Steps>
  <Step title="Authorization URL generation">
    ```typescript
    getAuthorizationUrl(state: string, redirectUri: string): string {
      return `https://accounts.google.com/o/oauth2/v2/auth?` +
        `client_id=${config.google.clientId}&` +
        `redirect_uri=${encodeURIComponent(redirectUri)}&` +
        `response_type=code&` +
        `scope=${encodeURIComponent(GOOGLE_CALENDAR_SCOPES)}&` +
        `access_type=offline&` +
        `prompt=consent&` +
        `state=${state}`;
    }
    ```
    
    Scopes requested:
    - `https://www.googleapis.com/auth/calendar` — Full calendar access
    - `https://www.googleapis.com/auth/calendar.events` — Event management
  </Step>
  <Step title="Code exchange">
    ```typescript
    async exchangeCodeForTokens(code: string, redirectUri: string): Promise<TokenSet> {
      const response = await axios.post('https://oauth2.googleapis.com/token', {
        code,
        client_id: config.google.clientId,
        client_secret: config.google.clientSecret,
        redirect_uri: redirectUri,
        grant_type: 'authorization_code',
      });
      
      return {
        accessToken: response.data.access_token,
        refreshToken: response.data.refresh_token,
        expiresAt: new Date(Date.now() + response.data.expires_in * 1000),
      };
    }
    ```
  </Step>
  <Step title="Token refresh">
    ```typescript
    async refreshAccessToken(refreshToken: string): Promise<TokenSet> {
      const response = await axios.post('https://oauth2.googleapis.com/token', {
        refresh_token: refreshToken,
        client_id: config.google.clientId,
        client_secret: config.google.clientSecret,
        grant_type: 'refresh_token',
      });
      
      // Google may or may not return a new refresh token
      return {
        accessToken: response.data.access_token,
        refreshToken: response.data.refresh_token || refreshToken,
        expiresAt: new Date(Date.now() + response.data.expires_in * 1000),
      };
    }
    ```
  </Step>
</Steps>

<Warning>
Google refresh tokens can be invalidated if:
- User revokes access in Google account settings
- 6 months pass without the refresh token being used
- Too many tokens are issued for the same client/user pair
</Warning>

## Echo suppression

Echo suppression prevents infinite sync loops when the same change bounces between CRM and external calendars.

### Change ID generation

Each outbound sync generates a unique change ID:

```typescript
function generateChangeId(): string {
  return `${Date.now()}.${randomUUID().slice(0, 8)}`;
}
```

This change ID is stored in:
- **Google**: `extendedProperties.private.crmChangeId`
- **Outlook**: Custom extension `crmChangeId`
- **CRM**: `calendar_event_mapping.lastOutboundChangeId`

### Inbound change detection

When processing a webhook notification:

<Steps>
  <Step title="Extract change ID">
    ```typescript
    const externalChangeId = event.extendedProperties?.private?.crmChangeId || 
                            event.extensions?.find(e => e.extensionName === 'com.yourcrm.event')?.crmChangeId;
    ```
  </Step>
  <Step title="Compare with stored value">
    ```typescript
    const mapping = await this.findMapping(event);
    
    if (mapping && mapping.lastInboundChangeId === externalChangeId) {
      // This is our own echo, skip processing
      this.logger.debug('Suppressing echo', {
        eventId: mapping.crmEventId,
        changeId: externalChangeId,
      });
      return;
    }
    ```
  </Step>
  <Step title="Update stored change ID">
    ```typescript
    // Process the change
    await this.processExternalChange(event, mapping);
    
    // Update last inbound change ID
    await this.updateMapping(mapping, {
      lastInboundChangeId: externalChangeId,
    });
    ```
  </Step>
</Steps>

### Fingerprint comparison

For additional safety, the system compares event fingerprints:

```typescript
function computeEventFingerprint(event: Event): string {
  const data = {
    title: event.title,
    startTime: event.startTime.getTime(),
    endTime: event.endTime.getTime(),
    location: event.location,
    inviteeIds: event.invitees.map(i => i.id).sort(),
  };
  
  return createHash('sha256')
    .update(JSON.stringify(data))
    .digest('hex')
    .slice(0, 16);
}
```

<Tip>
Fingerprints catch cases where:
- Change IDs are lost or corrupted
- Multiple changes happen in rapid succession
- Provider APIs don't preserve custom properties reliably
</Tip>

### Time window suppression

Changes that occur within a short time window of an outbound sync are also suppressed:

```typescript
const ECHO_SUPPRESSION_WINDOW_MS = 30_000; // 30 seconds

async function shouldSuppressEcho(mapping: CalendarEventMapping, 
                                 externalEvent: ExternalEvent): Promise<boolean> {
  // Check change ID
  if (mapping.lastInboundChangeId === externalEvent.changeId) {
    return true;
  }
  
  // Check fingerprint
  if (mapping.lastOutboundFingerprint === externalEvent.fingerprint) {
    return true;
  }
  
  // Check time window
  if (mapping.lastOutboundSyncAt) {
    const elapsed = Date.now() - mapping.lastOutboundSyncAt.getTime();
    if (elapsed < ECHO_SUPPRESSION_WINDOW_MS) {
      this.logger.debug('Suppressing change within echo window', {
        elapsed,
        threshold: ECHO_SUPPRESSION_WINDOW_MS,
      });
      return true;
    }
  }
  
  return false;
}
```

<Warning>
Echo suppression is best-effort. In rare cases (rapid manual changes, provider bugs, race conditions), echoes may still occur. The system relies on users noticing and reporting these issues.
</Warning>

### Processing flow

<Steps>
  <Step title="Enqueue operation">
    ```typescript
    async enqueueOperation(operation: OutboxOperation): Promise<void> {
      await this.outboxRepository.insert({
        organizationId: operation.organizationId,
        operation: operation.type,
        payload: operation.payload,
        idempotencyKey: operation.idempotencyKey,
        nextAttemptAt: new Date(),
      });
      
      // Trigger pg-boss job
      await this.jobQueue.send('calendar-sync-outbox-execute', {
        organizationId: operation.organizationId,
      });
    }
    ```
  </Step>
  <Step title="Execute worker picks up job">
    ```typescript
    @Process('calendar-sync-outbox-execute')
    async executeOutboxJob(job: Job<{ organizationId: string }>): Promise<void> {
      // Set RLS context
      await this.setOrganizationContext(job.data.organizationId);
      
      // Find pending operations
      const operations = await this.outboxRepository.find({
        organizationId: job.data.organizationId,
        status: 'pending',
        nextAttemptAt: { $lte: new Date() },
      }, {
        orderBy: { nextAttemptAt: 'ASC' },
        limit: 10,
      });
      
      // Process each operation
      for (const op of operations) {
        await this.processOperation(op);
      }
    }
    ```
  </Step>
  <Step title="Process individual operation">
    ```typescript
    async processOperation(outboxItem: CalendarSyncOutbox): Promise<void> {
      // Mark as processing
      await this.outboxRepository.update(outboxItem.id, {
        status: 'processing',
        lastAttemptAt: new Date(),
        attempts: outboxItem.attempts + 1,
      });
      
      try {
        // Execute operation
        await this.executeOperation(outboxItem);
        
        // Mark as completed
        await this.outboxRepository.update(outboxItem.id, {
          status: 'completed',
          completedAt: new Date(),
        });
      } catch (error) {
        await this.handleOperationError(outboxItem, error);
      }
    }
    ```
  </Step>
  <Step title="Handle errors with backoff">
    ```typescript
    async handleOperationError(outboxItem: CalendarSyncOutbox, error: Error): Promise<void> {
      const nextAttempt = outboxItem.attempts >= outboxItem.maxAttempts
        ? null
        : this.calculateNextAttempt(outboxItem.attempts);
      
      await this.outboxRepository.update(outboxItem.id, {
        status: nextAttempt ? 'pending' : 'failed',
        lastError: error.message,
        nextAttemptAt: nextAttempt,
      });
      
      if (!nextAttempt) {
        this.logger.error('Outbox operation exhausted retries', {
          id: outboxItem.id,
          operation: outboxItem.operation,
          attempts: outboxItem.attempts,
        });
      }
    }
    
    calculateNextAttempt(attempts: number): Date {
      // Exponential backoff: 1min, 2min, 4min, 8min, 16min
      const delayMs = Math.min(
        Math.pow(2, attempts) * 60 * 1000,
        16 * 60 * 1000
      );
      return new Date(Date.now() + delayMs);
    }
    ```
  </Step>
</Steps>

<Check>
The outbox pattern ensures:
- Operations survive process restarts
- Failed operations are retried with backoff
- Duplicate operations are prevented via idempotency keys
- Organization RLS context is preserved
</Check>

    ```typescript
      const expiresIn = integration.tokenExpiresAt.getTime() - Date.now();
      const threshold = 5 * 60 * 1000; // 5 minutes
      
      if (expiresIn < threshold) {
        await this.refreshToken(integration);
      }
    }
    ```
  </Step>
  <Step title="Reactive refresh on 401">
    If an API call fails with 401, refresh and retry:
    
    ```typescript
    async callProviderApi<T>(integration: CalendarIntegration,
                            apiCall: () => Promise<T>): Promise<T> {
      try {
        return await apiCall();
      } catch (error) {
        if (error.statusCode === 401) {
          await this.refreshToken(integration);
          return await apiCall(); // Retry once
        }
        throw error;
      }
    }
    ```
  </Step>
  <Step title="Handle refresh failures">
    If refresh fails, mark integration as broken:
    
    ```typescript
    async refreshToken(integration: CalendarIntegration): Promise<void> {
      try {
        const driver = this.registry.getDriver(integration.provider);
        const tokens = await driver.refreshAccessToken(integration.refreshToken);
        
        await this.integrationRepository.update(integration.id, {
          accessToken: await this.encrypt(tokens.accessToken),
          refreshToken: await this.encrypt(tokens.refreshToken),
          tokenExpiresAt: tokens.expiresAt,
          syncHealth: 'healthy',
          lastSyncAt: new Date(),
        });
      } catch (error) {
        await this.integrationRepository.update(integration.id, {
          syncHealth: 'token_expired',
          lastSyncError: error.message,
        });
        
        // Notify user to reconnect
        await this.notifyTokenExpired(integration.userId);
      }
    }
    ```
  </Step>
</Steps>

<Tip>
The `syncHealth` column on `calendar_integration` tracks integration status:
- `healthy` — Working normally
- `token_expired` — User needs to reconnect
- `webhook_failed` — Webhook setup failed
- `sync_error` — Recent sync errors occurred
</Tip>

### Webhook renewal monitoring

Both Google and Outlook webhook subscriptions expire and must be renewed proactively.

**Renewal job** (`calendar-watch-renewal.handler.ts`):

```typescript
@Cron('0 */6 * * *') // Every 6 hours
async renewExpiringWatches(): Promise<void> {
  const threshold = new Date(Date.now() + 24 * 60 * 60 * 1000); // 24 hours
  
  const expiringIntegrations = await this.integrationRepository.find({
    watchExpiration: { $lte: threshold },
    syncHealth: 'healthy',
  });
  
  for (const integration of expiringIntegrations) {
    try {
      const driver = this.registry.getDriver(integration.provider);
      const subscription = await driver.renewWebhook(integration);
      
      await this.integrationRepository.update(integration.id, {
        watchChannelId: subscription.channelId || subscription.subscriptionId,
        watchResourceId: subscription.resourceId,
        watchExpiration: subscription.expiration,
      });
      
      this.logger.info('Renewed webhook watch', {
        integrationId: integration.id,
        provider: integration.provider,
        newExpiration: subscription.expiration,
      });
    } catch (error) {
      this.logger.error('Failed to renew webhook watch', {
        integrationId: integration.id,
        error: error.message,
      });
      
      await this.integrationRepository.update(integration.id, {
        syncHealth: 'webhook_failed',
        lastSyncError: `Webhook renewal failed: ${error.message}`,
      });
    }
  }
}
```

<Warning>
If webhook renewal fails, the integration will miss change notifications until the user reconnects or the renewal job succeeds.
</Warning>
