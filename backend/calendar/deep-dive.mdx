<Warning>
This document is intentionally blunt: the calendar integration is not a simple "push event to Google" feature. It is a distributed sync system with two hostile external providers, eventually consistent mailboxes, webhook retries, token expiry, RLS, notifications, and user actions happening from both CRM and external calendars.
</Warning>

## Executive summary

The system has three calendar layers:

<Steps>
  <Step title="CRM calendar feed">
    Primary routes serve personal workload, organization stacked calendars, and team calendars with CRM tasks/subtasks, events, and external items
  </Step>
  
  <Step title="CRM events">
    `EventService` owns the internal event source of truth, invitees, RSVP status, soft deletion, and notification emission
  </Step>
  
  <Step title="Calendar integrations">
    Google and Outlook drivers push CRM events outward and process provider webhooks back into CRM
  </Step>
</Steps>

### Intended model

- CRM events are the canonical business record
- Creator calendar copies can update CRM event details when changed externally
- Invitee calendar copies can update RSVP only
- External-only items are displayed read-only and are not imported into CRM
- Provider echoes are suppressed with change IDs, fingerprints, ETags, and short time windows

<Warning>
The biggest risk is that the system is trying to make two different calendar providers behave like one coherent event bus. That is inherently fragile.
</Warning>

## Calendar feed endpoints

### Personal calendar (`GET /v1/calendar/me`)

Personal workload view including:
- CRM tasks and subtasks
- CRM events
- External calendar items (when viewing yourself)

<Note>
Legacy `GET /v1/calendar` delegates to the same handler as `/me`.
</Note>

### Organization calendar (`GET /v1/calendar/org`)

Organization **stacked** calendar with `userIds` parameter for 1–10 **other** org members.

**Response structure:**
```json
{
  "rosterUserIds": [...],
  "itemsByUserId": {
    "userId1": [...],
    "userId2": [...]
  }
}
```

**Content per user:**
- Personal-scope tasks/subtasks
- CRM events per subject
- No external calendar items
- Viewer ID in query is ignored

### Team calendar (`GET /v1/calendar/teams/:teamId`)

**Required permissions:**
- `team.admin` on that team, OR
- `team_crm.*` on that team, OR
- `team_sales.*` on that team, OR
- Organization admin/owner role

**Behavior:**
- CRM-only items per **other** roster member
- The authenticated user is excluded from the roster
- Use `/me` endpoint for self view

### Personal agenda (`GET /v1/calendar/me/agenda`)

Serves the personal merged agenda list with:
- Optional inclusive `from` parameter
- Forward scan up to server lookahead cap
- **No** `limit`/`cursor` pagination (see `calendar-agenda.constants.ts`)

**Response includes truncation indicator:**

```typescript
{
  items: [...],
  truncated: boolean  // true = exited before lookahead horizon
                      // false = scanned full lookahead window
}
```

<Info>
Clients can use the `truncated` flag to show that more items may exist beyond what was merged.
</Info>

<Note>
Legacy `GET /v1/calendar/agenda` delegates to `/me/agenda`.
</Note>

### Personal agenda timeline (`GET /v1/calendar/me/agenda/timeline`)

Serves the **CRM web agenda list** with **day-window pagination** for infinite scroll.

<Tabs>
  <Tab title="Request">
    **Required header:**
    - `Time-Zone`: IANA timezone identifier (invalid/missing defaults to `UTC`)
    
    **Query parameters:**
    - `cursor`: Optional base64 encoded pagination token
  </Tab>
  
  <Tab title="Response">
    ```json
    {
      "window": {
        "fromDay": "2024-01-15",
        "toDay": "2024-03-15",
        "timeZone": "America/New_York"
      },
      "items": [...],
      "nextCursor": "eyJ2IjoxLCJuZXh0RGF5IjoiMjAyNC0wMy0xNiJ9",
      "hasMore": true
    }
    ```
  </Tab>
</Tabs>

#### Timeline algorithm

<Steps>
  <Step title="Viewer zone">
    Extract timezone from `Time-Zone` header via `TimezoneMiddleware` / `TimezoneParam`
  </Step>
  
  <Step title="Sort order">
    - Tasks by **due** (`startDate` on DTO mirrors due)
    - CRM events and externals by **start**
    - Stable tie-break by source + id (`calendar-timeline.util.ts`)
  </Step>
  
  <Step title="Probe window">
    From page anchor day, fetch merged window of `CALENDAR_TIMELINE_PROBE_LOOKAHEAD_DAYS` (default 60 days)
  </Step>
  
  <Step title="Merge and filter">
    - Merge-sort items
    - Apply first-page visibility rule: `max(from, now)` for tasks
    - Keep events/externals that overlap viewer's local "today" even if start is before floor
    - Take first `CALENDAR_TIMELINE_PAGE_TARGET` (40) rows
  </Step>
  
  <Step title="Day span calculation">
    Union of local calendar days each row touches:
    - Date-only tasks use `dueDate` YYYY-MM-DD prefix
    - Timed items use Luxon in viewer zone
    - All-day events walk date labels
    - `fromDay` is floored at page anchor (`anchorBoundedDaySpan`)
    - Clamp to `CALENDAR_TIMELINE_MAX_DAY_SPAN` days
  </Step>
  
  <Step title="Authoritative fetch">
    `collectMergedCalendarItems` for `[startOf(D_min), endOf(D_max)]` in viewer zone
  </Step>
</Steps>

#### Timeline item day keys

Timeline items include optional `agendaDayKeys: string[]` for this endpoint only.

<Info>
`agendaDayKeys` are server-authoritative viewer-local day buckets the row should render in, clipped to the response window (`fromDay`–`toDay`).
</Info>

**Client rendering rules:**

<CodeGroup>
```typescript Client Logic
// Items with clipped agendaDayKeys as empty array are dropped server-side
// Clients must group by agendaDayKeys when present

if (item.agendaDayKeys !== undefined) {
  if (item.agendaDayKeys.length === 0) {
    // Skip entirely - doesn't belong to this page window
    continue;
  }
  // Group by server-provided day keys
  item.agendaDayKeys.forEach(day => {
    agendaBuckets[day].push(item);
  });
} else {
  // Fallback to browser-local parsing only if agendaDayKeys absent
  // (non-timeline callers, legacy paths)
  const localDay = parseLocalDay(item);
  agendaBuckets[localDay].push(item);
}
```
</CodeGroup>

<Warning>
Do **not** fall back to browser-local date parsing when `agendaDayKeys` is an empty array. This would misplace the row.
</Warning>

**Cursor structure:**
```json
{
  "v": 1,
  "nextDay": "2024-03-16"
}
```

Use `window.toDay` as the loaded-through boundary. `nextDay` is the local calendar day **after** `toDay`.

#### West-of-UTC all-day leak (fixed)

All-day CRM events are stored as UTC midnight (`T00:00:00.000Z`). The authoritative UTC refetch range for a page ending at local `toDay` extends to `endOf(toDay)` in the viewer zone, which may be the next UTC day for western timezones. This prevented multi-day events from leaking into future pages where they would be dropped by the window clipping logic.

## OAuth flow

### SignedPayloadService

HMAC-SHA256 tokens with format: `v1.<base64url(envelope)>.<base64url(sig)>`

**Envelope structure:**
```json
{
  "purpose": "calendar.oauth.state",
  "iat": 1234567890,
  "exp": 1234571490,
  "payload": {
    "userId": "...",
    "organizationId": "..."
  }
}
```

**Configuration:**
- Purpose: `calendar.oauth.state`
- Secret: `app.signedPayload.stateSecret`
- TTL: `CALENDAR_OAUTH_STATE_TTL_MS`

### OAuth flow architecture

<Steps>
  <Step title="Initiate flow">
    User clicks "Connect Google Calendar" or "Connect Outlook"
    - Generate signed state token via `SignedPayloadService`
    - Redirect to provider authorization URL
  </Step>
  
  <Step title="Provider authorization">
    User grants permissions on provider's OAuth consent screen
  </Step>
  
  <Step title="Callback handling">
    Provider redirects to callback endpoint with authorization code and state
    - Verify state token via `SignedPayloadService`
    - Exchange authorization code for access token and refresh token
    - Store integration record
  </Step>
  
  <Step title="Initial sync">
    Set up webhook subscription for future changes
  </Step>
</Steps>

**Controllers:**
- `google-calendar-oauth.controller.ts`: Handles Google OAuth flow
- `outlook-calendar-oauth.controller.ts`: Handles Microsoft OAuth flow

**Shared service:**
- `calendar-oauth-flow.service.ts`: Common state signing/verification and callback helpers

## Webhook security

### Google Calendar webhooks

<Info>
Google stores a **separate** `watchChannelVerifyToken` that is sent as the `token` in `events.watch` and echoed as `X-Goog-Channel-Token`. It is not the same as the channel `id`.
</Info>

**Security checks:**
1. Verify `X-Goog-Channel-Token` matches stored `watchChannelVerifyToken`
2. When `watchResourceId` is present, verify `X-Goog-Resource-Id` matches
3. Legacy rows may still equate token and channel ID

<Warning>
Legacy rows may still equate the two (token and channel ID). Handle gracefully for backward compatibility.
</Warning>

### Outlook webhooks

**Security requirements:**

<Steps>
  <Step title="Client state validation">
    Verify `clientState` in notification equals the integration UUID
  </Step>
  
  <Step title="JWT validation">
    When Microsoft Graph includes `validationTokens` on a change-notification batch:
    - Verify JWTs using JWKS
    - Check audience: `app.outlookCalendar.webhookValidationAudience` (defaults to calendar app client ID)
    - Return `202` only after successful verification
  </Step>
</Steps>

**Implementation:**
- `microsoft-graph-webhook-validation.util.ts`: JWT verification via `jose` library

### Production requirements

<Warning>
In **production** (`buildAppConfig().isProduction`):
- `GOOGLE_CALENDAR_WEBHOOK_URL` must be **https** and must not target localhost
- `OUTLOOK_CALENDAR_WEBHOOK_URL` must be **https** and must not target localhost
- Public calendar webhooks are throttled
- Outlook batches are capped before async sync fan-out
</Warning>

**Echo suppression mechanism:**

The system uses multiple fingerprinting strategies to prevent infinite sync loops:

1. **Change ID:** Unique identifier generated on each CRM change
2. **ETag:** Provider-supplied version identifier
3. **Fingerprint:** Hash of event content sent to provider
4. **Time windows:** Short grace periods for webhook processing

## Sync architecture

### Outbound sync (CRM → Provider)

When a CRM event is created or updated:

<Steps>
  <Step title="Detect changes">
    `event-change-detection.util.ts` performs deep equality checks:
    - Custom fields comparison
    - Sorted ID arrays for invitees
    - Time and location changes
  </Step>
  
  <Step title="Generate fingerprint">
    Hash the outbound event data to create `lastOutboundFingerprint`
  </Step>
  
  <Step title="Queue sync operation">
    Insert into `calendar_sync_outbox` with idempotency key
  </Step>
  
  <Step title="Process via pg-boss">
    Worker picks up queued operation with backoff retry
  </Step>
  
  <Step title="Call provider API">
    Use appropriate driver (Google or Outlook) to update calendar
  </Step>
  
  <Step title="Update mapping">
    Store provider event ID, ETag, and fingerprint in `calendar_event_mapping`
  </Step>
</Steps>

<Warning>
Some areas are still production-dangerous: CRM-originated outbound sync without durable queue (listener path), remaining provider HTTP inside tenant transactions, and provider-specific edge cases.
</Warning>

### Inbound sync (Provider → CRM)

When a webhook notification arrives:

<Steps>
  <Step title="Authenticate webhook">
    Verify tokens/signatures per provider requirements
  </Step>
  
  <Step title="Fetch event details">
    Call provider API to get full event data
  </Step>
  
  <Step title="Check echo suppression">
    Compare against stored fingerprint, ETag, change ID, and time windows
    
    If this is an echo of our own outbound sync, skip processing
  </Step>
  
  <Step title="Determine update type">
    - **Creator integration:** Can update full event details in CRM
    - **Invitee integration:** Can only update RSVP status
  </Step>
  
  <Step title="Update CRM event">
    Apply changes to CRM event via `EventService`
  </Step>
  
  <Step title="Emit notifications">
    Use `event-notification-recipients.util.ts` to determine recipients:
    - RSVP-aware recipient sets
    - CRM vs time-change notification logic
  </Step>
</Steps>

<Tip>
Provider echoes are suppressed with change IDs, fingerprints, ETags, and short time windows to prevent infinite loops.
</Tip>

### Watch renewal

Calendar webhook subscriptions expire and must be renewed:

**Google Calendar:**
- Max lifetime: 7 days (configurable)
- Renewal handler runs daily via scheduled job
- Calls `events.watch` API to extend subscription

**Outlook Calendar:**
- Max lifetime: 3 days (Graph API limit)
- Renewal handler runs more frequently
- Calls Graph subscription PATCH API

**Implementation:**
- `calendar-watch-renewal.handler.ts`: Scheduled job handler
- Queries integrations with expiring subscriptions
- Renews in batches with error handling

## Provider-specific edge cases

### Google Calendar

<AccordionGroup>
  <Accordion title="Recurring events">
    - Single instance modifications create separate event IDs
    - Must track both master event and instance overrides
    - iCalendar UID links instances to master
  </Accordion>
  
  <Accordion title="All-day events">
    - Stored as date-only (no time component)
    - Timezone handling differs from timed events
    - Use `CALENDAR_ALL_DAY_UTC_HOUR` constant (0) for UTC midnight conversion
  </Accordion>
  
  <Accordion title="Deleted invitees">
    - Removal triggers webhook notification
    - Must sync deletions back to CRM
    - Check `status: 'cancelled'` in attendee array
  </Accordion>
  
  <Accordion title="Channel expiration">
    - Max 7 days subscription lifetime
    - Must renew before expiration
    - Handle expired channel gracefully
  </Accordion>
</AccordionGroup>

### Outlook Calendar

<AccordionGroup>
  <Accordion title="Mailbox-local event IDs">
    - Event IDs are specific to the mailbox
    - Cannot directly share event IDs between users
    - Must use iCalendar UID for cross-mailbox matching
  </Accordion>
  
  <Accordion title="Shadow mapping">
    When CRM event has multiple Outlook invitees:
    - Creator's event ID is the canonical mapping
    - Invitees' mailboxes receive copies with different IDs
    - Shadow reconciliation matches via iCalUid
    - Reconciliation retries are durable via `calendar_sync_outbox`
  </Accordion>
  
  <Accordion title="Recurring events">
    - Master event has series master ID
    - Exceptions have occurrence IDs
    - Graph API requires special handling for instances
  </Accordion>
  
  <Accordion title="Subscription expiration">
    - Max 3 days (4230 minutes) lifetime
    - More aggressive renewal schedule needed
    - Handle 404 on expired subscriptions
  </Accordion>
  
  <Accordion title="Validation tokens">
    - Graph may send validation JWT in notification batch
    - Must verify signature before returning 202
    - Uses `jose` library for JWT verification
    - Audience claim must match configured webhook audience
  </Accordion>
</AccordionGroup>

<Warning>
Outlook mailbox-local event IDs are a major source of complexity. Always use iCalUid for cross-mailbox event correlation.
</Warning>

## Known issues and weak points

<Warning>
The calendar integration has several known fragile areas that require careful attention:
</Warning>

### Critical weak points

<CardGroup cols={2}>
  <Card title="Listener path sync" icon="warning">
    CRM-originated outbound sync without durable queue on listener path
  </Card>
  
  <Card title="Provider HTTP in transactions" icon="database">
    Remaining provider HTTP calls inside tenant transactions (Outlook shadow polling removed)
  </Card>
  
  <Card title="Recurring event edge cases" icon="repeat">
    Provider-specific handling of recurring events, instances, and exceptions
  </Card>
  
  <Card title="All-day timezone handling" icon="clock">
    All-day events stored as UTC midnight can leak across day boundaries for western timezones
  </Card>

### Race conditions

<Steps>
  <Step title="Concurrent modifications">
    User edits CRM event while webhook is processing provider change
    - Mitigation: Change ID and fingerprint comparison
    - Not fully transactional across systems
  </Step>
  
  <Step title="Token expiry during sync">
    Access token expires between operation queuing and execution
    - Mitigation: Refresh token flow in driver
    - May still fail if refresh token invalid
  </Step>
  
  <Step title="Webhook delivery delays">
    Provider webhooks may arrive out of order or delayed
    - Mitigation: Use ETag and sequence numbers where available
    - Cannot guarantee perfect ordering
  </Step>
</Steps>

### Provider-specific risks

<Tabs>
  <Tab title="Google">
    - Channel expiration if renewal job fails
    - Quota limits on API calls
    - Rate limiting during bulk operations
    - Sync token invalidation requiring full resync
  </Tab>
  
  <Tab title="Outlook">
    - Short subscription lifetime (3 days)
    - Mailbox-local event IDs requiring shadow mapping
    - Graph API throttling and complex retry headers
    - Validation token verification on critical path
    - Cross-mailbox event correlation complexity
  </Tab>
</Tabs>

## Recommendations

### Architectural improvements

<Steps>
  <Step title="Full outbound queue">
    Move **all** outbound sync operations to `calendar_sync_outbox`, including listener paths
    - Currently only partial coverage
    - Would make retry logic consistent
  </Step>
  
  <Step title="Remove provider HTTP from transactions">
    Extract remaining provider API calls from tenant transaction boundaries
    - Queue operations instead of inline HTTP

<Warning>
Disabling an integration should be a graceful operation that cleans up provider state.
</Warning>

<Steps>
  <Step title="Mark inactive">
    Set `calendar_integration.isActive = false`
  </Step>
  
  <Step title="Cancel webhook subscription">

  </Step>

### Emergency rollback

If calendar sync causes production issues:

<CodeGroup>
```sql Stop All Sync
-- Disable all calendar integrations
UPDATE calendar_integration 
SET is_active = false 
WHERE is_active = true;

-- Clear pending outbox operations
UPDATE calendar_sync_outbox 
SET status = 'cancelled' 
WHERE status IN ('pending', 'processing');
```

```sql Stop Specific Provider
-- Disable only Google integrations
UPDATE calendar_integration 
SET is_active = false 
WHERE provider = 'google' AND is_active = true;

-- Or only Outlook
UPDATE calendar_integration 
SET is_active = false 
WHERE provider = 'outlook' AND is_active = true;
```
</CodeGroup>

<Info>
After rollback, webhook subscriptions will eventually expire. The system will not attempt renewal for inactive integrations.
</Info>

## Support and troubleshooting

### Common issues

<AccordionGroup>
  <Accordion title="Events not syncing to provider">
    **Symptoms:** CRM events created but not appearing in Google/Outlook
    
    **Debug steps:**
    1. Check `calendar_sync_outbox` for pending/failed operations
    2. Verify integration `isActive = true`
    3. Check access token not expired
    4. Review `lastSyncError` on integration
    5. Look for provider API rate limit errors
  </Accordion>
  
  <Accordion title="Duplicate events appearing">
    **Symptoms:** Same event showing multiple times in calendar
    
    **Possible causes:**
    - Echo suppression failure
    - Multiple integrations for same user
    - Webhook replay after timeout
    
    **Remediation:**
    - Check `calendar_event_mapping` for duplicate provider event IDs
    - Verify fingerprint and ETag tracking
    - Review webhook processing logs for retries
  </Accordion>
  
  <Accordion title="RSVP changes not syncing back">
    **Symptoms:** User responds in Outlook but CRM not updated
    
    **Debug steps:**
    1. Verify webhook subscription active
    2. Check webhook delivery logs
    3. Confirm integration has `isActive = true`
    4. Look for echo suppression false positive
    5. Check if user is marked as invitee vs creator
  </Accordion>
  
  <Accordion title="Subscription expired errors">
    **Symptoms:** 404 or expired subscription errors in logs
    
    **Resolution:**
    - Renewal job may have failed
    - Check `watchExpiration` or `subscriptionExpiration`
    - Manually trigger renewal for affected integrations
    - Subscribe again if renewal fails
  </Accordion>
</AccordionGroup>

### Diagnostic queries

<CodeGroup>
```sql Integration Health
-- Check integration health
SELECT 
  ci.id,
  ci.provider,
  ci.provider_user_id,
  ci.last_sync_at,
  ci.last_sync_error,
  ci.watch_expiration,
  ci.subscription_expiration,
  CASE 
    WHEN ci.provider = 'google' THEN ci.watch_expiration < NOW() + INTERVAL '1 day'
    WHEN ci.provider = 'outlook' THEN ci.subscription_expiration < NOW() + INTERVAL '12 hours'
  END as needs_renewal_soon
FROM calendar_integration ci
WHERE ci.is_active = true
ORDER BY ci.last_sync_at NULLS FIRST;
```

```sql Pending Operations
-- Check pending sync operations
SELECT 
  cso.id,
  cso.operation,
  cso.status,
  cso.attempts,
  cso.scheduled_for,
  cso.last_attempt_at,
  cso.last_error
FROM calendar_sync_outbox cso
WHERE cso.status IN ('pending', 'processing')
ORDER BY cso.scheduled_for;
```

```sql Recent Errors
-- Find recent sync errors
SELECT 
  ci.provider,
  ci.provider_user_id,
  ci.last_sync_error,
  ci.last_sync_at,
  COUNT(*) as error_count
FROM calendar_integration ci
WHERE ci.last_sync_error IS NOT NULL
  AND ci.last_sync_at > NOW() - INTERVAL '1 day'
GROUP BY 1, 2, 3, 4
ORDER BY error_count DESC;
```

```sql Event Mapping Stats
-- Event mapping statistics
SELECT 
  cem.provider,
  cem.is_creator,
  COUNT(*) as mapping_count,
  COUNT(DISTINCT cem.crm_event_id) as unique_crm_events,
  AVG(EXTRACT(EPOCH FROM (NOW() - cem.last_outbound_at))) as avg_seconds_since_outbound
FROM calendar_event_mapping cem
GROUP BY 1, 2;
```
</CodeGroup>
