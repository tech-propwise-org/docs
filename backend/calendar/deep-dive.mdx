## Executive Summary

The calendar system operates across three interconnected layers that handle complex synchronization between CRM data and external calendar providers:

<CardGroup cols={3}>
  <Card title="CRM Calendar Feed" icon="calendar">
    Primary routes for personal, organizational, and team calendar views
  </Card>
  <Card title="CRM Events" icon="event">
    Internal event source of truth with invitees and RSVP management
  </Card>
  <Card title="Calendar Integrations" icon="sync">
    Google and Outlook drivers for bidirectional synchronization
  </Card>
</CardGroup>

### Calendar Feed Routes

The system provides multiple calendar view endpoints:

- **`GET /v1/calendar/me`**: Personal workload (CRM tasks/subtasks + CRM events + external items)
- **`GET /v1/calendar/org`**: Organization stacked calendar (1-10 other org members)
- **`GET /v1/calendar/teams/:teamId`**: Team calendar views with proper permissions
- **`GET /v1/calendar/me/agenda`**: Personal merged agenda list
- **`GET /v1/calendar/me/agenda/timeline`**: CRM web agenda with day-window pagination

<Warning>
The system is trying to make two different calendar providers behave like one coherent event bus. This is inherently fragile and requires careful operational monitoring.
</Warning>

## Architecture Overview

### Intended Model

<Steps>
  <Step title="CRM Events as Source of Truth">
    CRM events serve as the canonical business record for all event data
  </Step>
  <Step title="Creator Calendar Synchronization">
    Creator calendar copies can update CRM event details when changed externally
  </Step>
  <Step title="Invitee RSVP Updates">
    Invitee calendar copies can only update RSVP status, not event details
  </Step>
  <Step title="External Item Display">
    External-only items are displayed read-only and are not imported into CRM
  </Step>
  <Step title="Echo Suppression">
    Provider echoes are suppressed using change IDs, fingerprints, ETags, and time windows
  </Step>
</Steps>

### Security Considerations

<Info>
**Webhook Authenticity:** The system implements different security models for each provider:
- **Google**: Separate `watchChannelVerifyToken` (distinct from channel `id`)
- **Outlook**: `clientState` must equal the integration UUID
- **Microsoft Graph**: JWT verification for `validationTokens` in production
</Info>

## Personal Agenda Timeline

The **`GET /v1/calendar/me/agenda/timeline`** endpoint powers the CRM web agenda view with infinite scroll capability.

### Implementation Details

<Tabs>
  <Tab title="Viewer Zone">
    Requires the standard **`Time-Zone`** request header (IANA format):
    - Invalid/missing timezone defaults to `UTC`
    - Handled by `TimezoneMiddleware` / `TimezoneParam`
  </Tab>
  
  <Tab title="Sort Logic">
    - **Tasks**: Sorted by due date (`startDate` on DTO mirrors due)
    - **CRM Events & Externals**: Sorted by start time
    - **Tie-breaker**: Stable sorting by source + id
  </Tab>
  
  <Tab title="Probe Strategy">
    - Fetch merged window of `CALENDAR_TIMELINE_PROBE_LOOKAHEAD_DAYS` (default: 60 days)
    - Apply visibility rules: `max(from, now)` for tasks
    - Events/externals kept when overlapping viewer's local "today"
    - Take first `CALENDAR_TIMELINE_PAGE_TARGET` (40) rows
  </Tab>
</Tabs>

### Response Structure

```json
{
  "window": {
    "fromDay": "2024-01-15",
    "toDay": "2024-01-20",
    "timeZone": "America/New_York"
  },
  "items": [...],
  "nextCursor": "base64EncodedCursor",
  "hasMore": true
}
```

<Check>
The `nextCursor` contains base64 JSON with `{ v, nextDay }` where `nextDay` is the local calendar day after `toDay`.
</Check>

## Data Model

### Calendar Integration Entity

The `CalendarIntegration` entity stores one active external calendar account per user, organization, and provider.

<CodeGroup>
```typescript Key Fields
interface CalendarIntegration {
  user: User;                    // CRM user who connected calendar
  organization: Organization;    // Tenant boundary
  provider: 'google' | 'outlook'; // Calendar provider
  calendarEmail: string;         // External account email
  encryptedAccessToken?: string; // AES-GCM encrypted (nullable after disconnect)
  encryptedRefreshToken?: string; // AES-GCM encrypted when issued
  tokenExpiresAt?: Date;         // Refresh threshold
}
```

```typescript Google Watch State
interface GoogleWatchFields {
  watchChannelId?: string;
  watchChannelVerifyToken?: string; // Distinct from channel ID
  watchResourceId?: string;
  watchExpiration?: Date;
  googleSyncToken?: string;       // Incremental sync token
}
```

```typescript Outlook Subscription
interface OutlookFields {
  outlookSubscriptionId?: string;
  outlookSubscriptionExpiration?: Date;
}
```

```typescript Health Monitoring
interface HealthFields {
  lastInboundWebhookAt?: Date;    // Last successful webhook processing
  lastOutboundSyncAt?: Date;      // Last CRM→provider push
  lastSyncErrorAt?: Date;         // Last failure timestamp
  lastSyncErrorMessage?: string;  // Bounded error message
  isActive: boolean;              // Connection status
  isDeleted: boolean;             // Soft delete flag
}
```
</CodeGroup>

### Calendar Event Mapping

The `CalendarEventMapping` entity tracks the relationship between CRM events and external calendar events.

<Warning>
The unique active index allows one row per `(user, organization, provider)`. Reconnecting the same provider will update the existing row rather than creating duplicates.
</Warning>

## Risk Assessment

### High-Risk Areas

<CardGroup cols={2}>
  <Card title="In-Memory Cache" icon="memory">
    Potential data inconsistency during high concurrency scenarios
  </Card>
  <Card title="Sync Without Durable Queue" icon="queue">
    CRM-originated outbound sync lacks durable queue on listener path
  </Card>
  <Card title="Recurring Events" icon="repeat">
    Provider-specific edge cases around recurring event handling
  </Card>
  <Card title="All-Day Events" icon="clock">
    Timezone and boundary handling complexities
  </Card>
</CardGroup>

### Mitigation Strategies

<Steps>
  <Step title="Implement Monitoring">
    Set up comprehensive monitoring for sync failures and webhook processing delays
  </Step>
  <Step title="Add Durable Queues">
    Implement durable queuing for critical sync operations
  </Step>
  <Step title="Test Edge Cases">
    Thoroughly test recurring events, all-day events, and deleted invitees scenarios
  </Step>
  <Step title="Provider-Specific Handling">
    Implement provider-specific logic for Outlook mailbox-local event IDs
  </Step>
</Steps>

## Operational Considerations

### Production Requirements

<Info>
In production environments (`buildAppConfig().isProduction`):
- `GOOGLE_CALENDAR_WEBHOOK_URL` must be HTTPS and not target localhost
- `OUTLOOK_CALENDAR_WEBHOOK_URL` must be HTTPS and not target localhost
- Outlook shadow mapping retries use durable `calendar_sync_outbox` operations
</Info>

### Database Hygiene

<Note>
**MikroORM Snapshot Management:** The `migrations/.snapshot-neondb.json` serves as the baseline for migration checks. If `migration:check` reports large unrelated diffs, refresh the snapshot using the standard process before creating new migrations.
</Note>

### Row-Level Security

The `calendar_sync_outbox` table is RLS-protected like other organization tables:
- Implemented in `Migration20260428180000_EnableCalendarSyncOutboxRls`
- Ensures proper tenant isolation for sync operations

<Tip>
Outlook shadow mapping retries are durable and use pg-boss execute workers with backoff, not in-process timers, providing better reliability for sync operations.
</Tip>
