<Warning>
The calendar integration is not a simple "push event to Google" feature. It is a distributed sync system with two hostile external providers, eventually consistent mailboxes, webhook retries, token expiry, RLS, notifications, and user actions happening from both CRM and external calendars.
</Warning>

## Executive Summary

The system has three calendar layers:

  <Card title="CRM Events" icon="calendar-days">

  </Card>
  <Card title="Calendar Integrations" icon="arrows-rotate">

### Core API Endpoints

**Personal Calendar:**
- `GET /v1/calendar/me` - Personal workload (CRM tasks/subtasks + CRM events + external items)
- `GET /v1/calendar/me/agenda` - Personal merged agenda list with forward scan
- `GET /v1/calendar/me/agenda/timeline` - CRM web agenda with day-window pagination

**Organization Calendar:**
- `GET /v1/calendar/org` - Organization stacked calendar for 1-10 other org members
- `GET /v1/calendar/teams/:teamId` - Team calendar for authorized members

**Legacy Endpoints:**
- `GET /v1/calendar` - Delegates to `/me` handler
- `GET /v1/calendar/agenda` - Delegates to `/me/agenda` handler

## Architecture Overview

### Intended Model

<Steps>
  <Step title="CRM Events as Source of Truth">
    CRM events are the canonical business record with full lifecycle management
  </Step>
  <Step title="Creator Calendar Sync">
    Creator calendar copies can update CRM event details when changed externally
  </Step>
  <Step title="Invitee RSVP Only">
    Invitee calendar copies can update RSVP status only
  </Step>
  <Step title="External Items Read-Only">
    External-only items are displayed read-only and are not imported into CRM
  </Step>
  <Step title="Echo Suppression">
    Provider echoes are suppressed with change IDs, fingerprints, ETags, and short time windows
  </Step>
</Steps>

### Risk Assessment

<Warning>
The biggest risk is trying to make two different calendar providers behave like one coherent event bus. This is inherently fragile.
</Warning>

**Production-Dangerous Areas:**
- In-memory cache
- CRM-originated outbound sync without durable queue (listener path)
- Provider-specific edge cases around recurring events, all-day events, deleted invitees
- Outlook mailbox-local event IDs

## Webhook Security

### Google Calendar

<Note>
Google stores a **separate** `watchChannelVerifyToken` sent as `token` in `events.watch` and echoed as `X-Goog-Channel-Token`. This is not the same as the channel `id`.
</Note>

**Verification Requirements:**
- `X-Goog-Channel-Token` must match stored `watchChannelVerifyToken`
- When `watchResourceId` is present, `X-Goog-Resource-Id` must match
- Legacy rows may incorrectly equate token and channel ID

### Outlook Calendar

**Client State Verification:**
- `clientState` must equal the integration UUID
- Microsoft Graph `validationTokens` are verified as JWTs using JWKS
- Audience: `app.outlookCalendar.webhookValidationAudience` (defaults to calendar app client ID)

<Warning>
In production (`buildAppConfig().isProduction`), both `GOOGLE_CALENDAR_WEBHOOK_URL` and `OUTLOOK_CALENDAR_WEBHOOK_URL` must be HTTPS and must not target localhost.
</Warning>

## Personal Agenda Timeline

The `/v1/calendar/me/agenda/timeline` endpoint powers the CRM web agenda view with infinite scroll capability.

### Request Parameters

<Info>
Requires the standard `Time-Zone` request header (IANA timezone). Invalid or missing values default to UTC.
</Info>

### Timeline Logic

<Steps>
  <Step title="Viewer Zone Detection">
    Parse `Time-Zone` header, fallback to UTC for invalid values
  </Step>
  <Step title="Probe Window">
    From anchor day, fetch merged window of `CALENDAR_TIMELINE_PROBE_LOOKAHEAD_DAYS` (default: 60 days)
  </Step>
  <Step title="Merge and Sort">
    - Tasks by due date (`startDate` mirrors due)
    - CRM events and externals by start time
    - Stable tie-break by source + ID
  </Step>
  <Step title="Apply Visibility Rules">
    - Tasks: `max(from, now)` floor
    - Events/externals: kept when overlapping viewer's local "today"
  </Step>
  <Step title="Page Formation">
    Take first `CALENDAR_TIMELINE_PAGE_TARGET` (40) rows
  </Step>
</Steps>

### Response Format

```typescript
{
  window: {
    fromDay: string,    // YYYY-MM-DD
    toDay: string,      // YYYY-MM-DD  
    timeZone: string    // IANA timezone
  },
  items: CalendarItemDto[],
  nextCursor: string,   // Base64 JSON: { v, nextDay }
  hasMore: boolean
}
```

## Sync Mechanisms

### Echo Suppression

<Tip>
The system uses multiple techniques to prevent infinite sync loops when calendar events are modified.
</Tip>

**Suppression Techniques:**
- Change IDs for operation tracking
- Content fingerprints for duplicate detection  
- ETags for version control
- Short time windows for echo detection

### Durable Operations

**Outlook Shadow Mapping:**
- Uses `calendar_sync_outbox` for retry durability
- Operation type: `outlook-shadow-reconcile` 
- Executed by pg-boss worker with backoff
- **Not** in-process timers

### Row Level Security

<Warning>
`calendar_sync_outbox` is RLS-protected like other org tables via `Migration20260428180000_EnableCalendarSyncOutboxRls`.
</Warning>

## Event Access Control

Events use a creator + invitee + org-calendar policy model:

<Steps>
  <Step title="ACCESS Policy">
    Creator OR invitee OR org-calendar policy OR team-calendar visibility
  </Step>
  <Step title="MODIFY Policy">
    Creator OR (invitee with RSVP-only changes)
  </Step>
  <Step title="DELETE Policy">  
    Creator OR org admin/owner
  </Step>
</Steps>

## Timezone Middleware

All calendar endpoints support the `Time-Zone` request header for IANA timezone specification.

**Behavior:**
- Validates IANA timezone format
- Falls back to UTC for invalid values
- Affects task list buckets and timeline day calculations
- Implemented via `TimezoneMiddleware` and `TimezoneParam`
