<Note>
This document explains how the current calendar system actually works. The calendar integration is not a simple "push event to Google" feature—it's a distributed sync system with two external providers, eventually consistent mailboxes, webhook retries, token expiry, and user actions happening from both CRM and external calendars.
</Note>

## Executive Summary

The system has three calendar layers:

1. **CRM calendar feed**: Primary routes for viewing calendar data across different scopes
2. **CRM events**: Internal event source of truth with invitees and RSVP status
3. **Calendar integrations**: Google and Outlook drivers for bidirectional sync

### Integration Model

<Info>
The intended model is:
- CRM events are the canonical business record
- Creator calendar copies can update CRM event details when changed externally
- Invitee calendar copies can update RSVP only
- External-only items are displayed read-only and are not imported into CRM
- Provider echoes are suppressed with change IDs, fingerprints, ETags, and short time windows
</Info>

<Warning>
The biggest risk is that the system is trying to make two different calendar providers behave like one coherent event bus. That is inherently fragile.
</Warning>

## CRM Calendar Feed

### Primary Routes

<Tabs>
  <Tab title="Personal Calendar">
    **`GET /v1/calendar/me`**
    
    Personal workload including:
    - CRM tasks/subtasks
    - CRM events
    - External items (when viewing yourself)
    
    Legacy **`GET /v1/calendar`** delegates to the same handler.
  </Tab>
  
  <Tab title="Organization Calendar">
    **`GET /v1/calendar/org`**
    
    Organization stacked calendar:
    - `userIds` = 1–10 other org members
    - Response: `{ rosterUserIds, itemsByUserId }`
    - Personal-scope tasks/subtasks + CRM events per subject
    - No externals included
    - Viewer ID in query is ignored
  </Tab>
  
  <Tab title="Team Calendar">
    **`GET /v1/calendar/teams/:teamId`**
    
    Requires: `team.admin`, `team_crm.*`, `team_sales.*`, or org admin/owner
    - CRM-only per other roster member
    - Authenticated user excluded from roster
    - Use **`/me`** for self-view
  </Tab>
</Tabs>

### Agenda Endpoints

#### Personal Agenda List

**`GET /v1/calendar/me/agenda`** (legacy: `/calendar/agenda`)

<Steps>
  <Step title="Query Parameters">
    - Optional inclusive `from` parameter
    - Forward scan up to server lookahead cap
    - **No** `limit`/`cursor` pagination
  </Step>
  
  <Step title="Response Format">
    ```json
    {
      "items": [...],
      "truncated": boolean
    }
    ```
    
    - `truncated: true` when merge loop exits before reaching lookahead horizon
    - `truncated: false` when scan ran through full lookahead window
  </Step>
</Tabs>

#### Timeline Agenda (Web View)

**`GET /v1/calendar/me/agenda/timeline`**

<Note>
Used by the CRM web agenda view for infinite scroll. Uses day-window pagination with merged probe → local `fromDay`/`toDay` in viewer `Time-Zone` header.
</Note>

<Steps>
  <Step title="Timezone Requirements">
    Requires **`Time-Zone`** request header (IANA format)
    - Invalid/missing defaults to `UTC`
    - Handled by `TimezoneMiddleware`
  </Step>
  
  <Step title="Sorting Logic">
    - **Tasks**: by due date (`startDate` mirrors due)
    - **CRM events and externals**: by start time
    - **Tie-break**: stable by source + id
  </Step>
  
  <Step title="Probe Window">
    From page anchor day, fetch merged window of `CALENDAR_TIMELINE_PROBE_LOOKAHEAD_DAYS` (default: 60 days)
  </Step>
  
  <Step title="Response Structure">
    ```json
    {
      "window": {
        "fromDay": "YYYY-MM-DD",
        "toDay": "YYYY-MM-DD", 
        "timeZone": "America/New_York"
      },
      "items": [...],
      "nextCursor": "base64-encoded-cursor",
      "hasMore": boolean
    }
    ```
  </Step>
</Steps>

## Integration Architecture

### OAuth State Security

<Check>
Calendar OAuth flows use `SignedPayloadService` with HMAC-SHA256 signed state tokens for CSRF protection.
</Check>

```typescript
// OAuth state payload structure
{
  userId: string,
  organizationId: string,
  purpose: 'calendar.oauth.state',
  iat: number,
  exp: number
}
```

The signed payload format is: `v1.<base64url(envelope)>.<base64url(signature)>`

### Webhook Security

<Tabs>
  <Tab title="Google Calendar">
    **Channel Verification:**
    - Stores separate `watchChannelVerifyToken`
    - Sent as `token` in `events.watch` 
    - Echoed as `X-Goog-Channel-Token`
    - When `watchResourceId` present, must match `X-Goog-Resource-Id`
    
    <Warning>
    Legacy rows may equate the channel `id` with the verify token—these are different values.
    </Warning>
  </Tab>
  
  <Tab title="Outlook Calendar">
    **Client State Validation:**
    - Requires `clientState` to equal integration UUID
    - Microsoft Graph `validationTokens` verified as JWTs (JWKS)
    - Audience: `app.outlookCalendar.webhookValidationAudience`
    - Returns `202` after JWT verification
  </Tab>
</Tabs>

<Info>
In **production** (`buildAppConfig().isProduction`), webhook URLs must be HTTPS and cannot target localhost.
</Info>

### Sync Mechanisms

#### Outbound Sync

CRM events are pushed to external calendars when:
- Event is created/updated/deleted
- Invitees are added/removed
- RSVP status changes

<Note>
Outlook shadow mapping retries are durable via `calendar_sync_outbox` + pg-boss worker (with backoff), not in-process timers.
</Note>

#### Inbound Sync

External calendar changes are processed through webhooks:

<Steps>
  <Step title="Webhook Validation">
    Verify webhook authenticity using provider-specific tokens
  </Step>
  
  <Step title="Change Detection">
    Use ETags, fingerprints, and change IDs to suppress provider echoes
  </Step>
  
  <Step title="Permission Check">
    - **Creator**: Can update all event details
    - **Invitee**: Can only update RSVP status
  </Step>
  
  <Step title="CRM Update">
    Apply changes to canonical CRM event record
  </Step>
</Steps>

## Data Model

### Calendar Integration Entity

```sql
CREATE TABLE calendar_integrations (
  id UUID PRIMARY KEY,
  user_id UUID NOT NULL,
  organization_id UUID NOT NULL,
  provider VARCHAR(50) NOT NULL,
  provider_calendar_id VARCHAR(255),
  access_token TEXT,
  refresh_token TEXT,
  token_expires_at TIMESTAMPTZ,
  watch_channel_id VARCHAR(255),
  watch_channel_verify_token VARCHAR(255),
  watch_resource_id VARCHAR(255),
  watch_expires_at TIMESTAMPTZ,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW()
);
```

### Event Mapping Entity

```sql
CREATE TABLE calendar_event_mappings (
  id UUID PRIMARY KEY,
  event_id UUID NOT NULL,
  calendar_integration_id UUID NOT NULL,
  provider_event_id VARCHAR(255) NOT NULL,
  ical_uid VARCHAR(255),
  last_outbound_fingerprint TEXT,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW()
);
```

### Sync Outbox

<Check>
`calendar_sync_outbox` is RLS-protected like other org tables.
</Check>

```sql
CREATE TABLE calendar_sync_outbox (
  id UUID PRIMARY KEY,
  organization_id UUID NOT NULL,
  operation VARCHAR(100) NOT NULL,
  payload JSONB NOT NULL,
  idempotency_key VARCHAR(255),
  status VARCHAR(50) DEFAULT 'pending',
  retry_count INTEGER DEFAULT 0,
  next_retry_at TIMESTAMPTZ,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW()
);
```

## Event Access Control

Events use a creator + invitee + org-calendar policy model:

<AccordionGroup>
  <Accordion title="ACCESS Permission">
    - Creator OR
    - Invitee OR  
    - Org-calendar policy (same filters as `GET /v1/calendar/org`) OR
    - Team-calendar visibility
  </Accordion>
  
  <Accordion title="MODIFY Permission">
    - Creator OR
    - Org admin/owner OR
    - Team admin (for team-scoped events)
  </Accordion>
  
  <Accordion title="DELETE Permission">
    - Creator OR
    - Org admin/owner
  </Accordion>
</AccordionGroup>
