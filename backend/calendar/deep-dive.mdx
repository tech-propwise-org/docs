# Calendar System Deep Dive

<Warning>
The calendar integration is not a simple "push event to Google" feature. It is a distributed sync system with two hostile external providers, eventually consistent mailboxes, webhook retries, token expiry, RLS, notifications, and user actions happening from both CRM and external calendars.
</Warning>

## Executive Summary

The system has three calendar layers:

<CardGroup cols={3}>
  <Card title="CRM Calendar Feed" icon="calendar">
    Primary routes for personal workload, organization stacked calendar, and team calendars
  </Card>
  <Card title="CRM Events" icon="calendar-days">
    EventService owns internal event source of truth, invitees, RSVP status, and notifications
  </Card>
  <Card title="Calendar Integrations" icon="sync">
    Google and Outlook drivers for bidirectional sync with external providers
  </Card>
</CardGroup>

### API Routes

<Tabs>
  <Tab title="Personal Calendar">
    - **`GET /v1/calendar/me`** - Personal workload (CRM tasks/subtasks + CRM events + external items when viewing yourself)
    - **`GET /v1/calendar/me/agenda`** - Personal merged agenda list with optional `from` parameter
    - **`GET /v1/calendar/me/agenda/timeline`** - CRM web agenda list with day-window pagination
  </Tab>
  <Tab title="Organization Calendar">
    - **`GET /v1/calendar/org`** - Organization stacked calendar for 1-10 other org members
    - Returns `{ rosterUserIds, itemsByUserId }` with personal-scope tasks/subtasks + CRM events per subject
  </Tab>
  <Tab title="Team Calendar">
    - **`GET /v1/calendar/teams/:teamId`** - Team calendar for authorized members
    - Requires `team.admin`, `team_crm.*`, or `team_sales.*` permissions
    - Excludes authenticated user from roster (use `/me` for self)
  </Tab>
</Tabs>

<Info>
Legacy routes `GET /v1/calendar` and `GET /v1/calendar/agenda` delegate to the same handlers as `/me` routes.
</Info>

## Architecture Model

The intended model follows these principles:

<Steps>
  <Step title="CRM Events as Source of Truth">
    CRM events are the canonical business record for all event data and metadata.
  </Step>
  <Step title="Creator Calendar Updates">
    Creator calendar copies can update CRM event details when changed externally.
  </Step>
  <Step title="Invitee RSVP Updates">
    Invitee calendar copies can update RSVP status only, not event details.
  </Step>
  <Step title="External Items Display">
    External-only items are displayed read-only and are not imported into CRM.
  </Step>
  <Step title="Echo Suppression">
    Provider echoes are suppressed with change IDs, fingerprints, ETags, and short time windows.
  </Step>
</Steps>

## Personal Agenda Timeline

The `/v1/calendar/me/agenda/timeline` endpoint serves the CRM web agenda view with infinite scroll support.

### Request Parameters

<CodeGroup>
```typescript API Request
GET /v1/calendar/me/agenda/timeline
Headers:
  Time-Zone: America/New_York
Query:
  cursor?: string (base64 JSON with nextDay)
```

```json Response Format
{
  "window": {
    "fromDay": "2024-01-15",
    "toDay": "2024-03-15", 
    "timeZone": "America/New_York"
  },
  "items": [...], // CalendarItemDto[]
  "nextCursor": "eyJ2IjoxLCJuZXh0RGF5IjoiMjAyNC0wMy0xNiJ9",
  "hasMore": true
}
```
</CodeGroup>

### Timeline Processing

<Steps>
  <Step title="Viewer Zone Detection">
    Requires `Time-Zone` request header (IANA). Invalid/missing defaults to `UTC`.
  </Step>
  <Step title="Probe Window">
    From anchor day, fetch `CALENDAR_TIMELINE_PROBE_LOOKAHEAD_DAYS` (default 60) days of merged data.
  </Step>
  <Step title="Sort and Filter">
    - Tasks by **due date** (`startDate` mirrors due)
    - CRM events and externals by **start time**
    - Apply visibility rule: `max(from, now)` for tasks; events kept when overlapping viewer's local "today"
  </Step>
  <Step title="Day Span Calculation">
    Calculate union of local calendar days, floored at page anchor and clamped to `CALENDAR_TIMELINE_MAX_DAY_SPAN`.
  </Step>
  <Step title="Authoritative Fetch">
    Re-fetch `collectMergedCalendarItems` for exact day span to ensure consistency.
  </Step>
</Steps>

<Note>
The legacy `GET /v1/calendar/me/agenda` endpoint remains for callers that want a single forward scan without pagination.
</Note>

## Security Implementation

### OAuth State Security

<CodeGroup>
```typescript OAuth Flow
// SignedPayloadService with HMAC-SHA256
const stateToken = signedPayloadService.signState({
  purpose: 'calendar.oauth.state',
  payload: { userId, organizationId },
  ttl: CALENDAR_OAUTH_STATE_TTL_MS
});

// Token format: v1.<base64url(envelope)>.<base64url(signature)>
```

```typescript State Verification
const { userId, organizationId } = signedPayloadService.verifyState(
  stateToken,
  'calendar.oauth.state'
);
```
</CodeGroup>

### Webhook Authentication

<Tabs>
  <Tab title="Google Calendar">
    - Separate `watchChannelVerifyToken` sent as `token` in `events.watch`
    - Echoed as `X-Goog-Channel-Token` header
    - Must match `X-Goog-Resource-Id` when `watchResourceId` is present
  </Tab>
  <Tab title="Outlook Calendar">
    - Requires `clientState` to equal integration UUID
    - JWT verification for `validationTokens` in change-notification batches
    - JWKS verification with audience validation
  </Tab>
</Tabs>

<Warning>
In production, `GOOGLE_CALENDAR_WEBHOOK_URL` and `OUTLOOK_CALENDAR_WEBHOOK_URL` must be HTTPS and cannot target localhost.
</Warning>

## Data Durability

### Sync Outbox

The `calendar_sync_outbox` table provides durable queuing for sync operations:

<CodeGroup>
```sql Schema
CREATE TABLE calendar_sync_outbox (
  id UUID PRIMARY KEY,
  organization_id UUID NOT NULL,
  operation TEXT NOT NULL,
  payload JSONB NOT NULL,
  status TEXT NOT NULL,
  created_at TIMESTAMP WITH TIME ZONE NOT NULL,
  updated_at TIMESTAMP WITH TIME ZONE NOT NULL
);
```

```typescript Usage
// Outlook shadow mapping retries are durable
await syncOutboxService.enqueue({
  operation: 'outlook-shadow-reconcile',
  payload: { eventId, mappingId },
  organizationId
});
```
</CodeGroup>

<Check>
The `calendar_sync_outbox` table is RLS-protected like other organization tables via `Migration20260428180000_EnableCalendarSyncOutboxRls`.
</Check>

## Risk Factors

<Warning>
The system is trying to make two different calendar providers behave like one coherent event bus. This is inherently fragile.
</Warning>

Key risk areas include:

- **In-memory cache** - Potential data loss during restarts
- **CRM-originated outbound sync** - No durable queue for listener path
- **Provider-specific edge cases** - Recurring events, all-day events, deleted invitees
- **Outlook mailbox-local event IDs** - Provider-specific identifier complications

<Tip>
The code has several good guardrails, but these areas require careful monitoring in production environments.
</Tip>
