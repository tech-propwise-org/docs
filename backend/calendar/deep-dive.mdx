This document explains how the current calendar system actually works. It is intentionally blunt: the calendar integration is not a simple "push event to Google" feature. It is a distributed sync system with two hostile external providers, eventually consistent mailboxes, webhook retries, token expiry, RLS, notifications, and user actions happening from both CRM and external calendars.

## Executive summary

The system has three calendar layers:

### 1. CRM calendar feed

Primary routes include:

- **`GET /v1/calendar/me`** - Personal workload: CRM tasks/subtasks + CRM events + external items when viewing yourself
- **`GET /v1/calendar/org`** - Organization stacked calendar: `userIds` = 1–10 other org members, response `{ rosterUserIds, itemsByUserId }` — personal-scope tasks/subtasks + CRM events per subject, no externals; viewer id in the query is ignored
- **`GET /v1/calendar/teams/:teamId`** - `team.admin`, `team_crm.*`, or `team_sales.*` on that team, or org admin/owner: CRM-only per other roster member — the authenticated user is excluded from the roster; use `/me` for self

Legacy **`GET /v1/calendar`** and **`GET /v1/calendar/agenda`** delegate to the same handlers as **`/me`** and **`/me/agenda`**.

### 2. CRM events

`EventService` owns the internal event source of truth, invitees, RSVP status, soft deletion, and notification emission.

### 3. Calendar integrations

Google and Outlook drivers push CRM events outward and process provider webhooks back into CRM.

## Production considerations

The code has several good guardrails, but some areas are still production-dangerous:

- In-memory cache
- CRM-originated outbound sync without durable queue (listener path)
- Provider-specific edge cases around recurring events, all-day events, deleted invitees
- Outlook mailbox-local event IDs

### Webhook authenticity

<Tabs>
  <Tab title="Google">
    Google stores a **separate** `watchChannelVerifyToken` (sent as the `token` in `events.watch` and echoed as `X-Goog-Channel-Token`); it is not the same as the channel `id`—legacy rows may still equate the two. When `watchResourceId` is present, inbound notifications must also match `X-Goog-Resource-Id`.
  </Tab>
  <Tab title="Outlook">
    Outlook requires `clientState` to equal the integration UUID. When Microsoft Graph includes `validationTokens` on a change-notification batch, JWTs are verified (JWKS) before the handler returns `202` (audience: `app.outlookCalendar.webhookValidationAudience`, defaulting to the calendar app client id).
  </Tab>
</Tabs>

<Info>
In **production** (`buildAppConfig().isProduction`), `GOOGLE_CALENDAR_WEBHOOK_URL` and `OUTLOOK_CALENDAR_WEBHOOK_URL` must be **https** and must not target localhost.
</Info>

### Outlook shadow mapping

<Check>
**Outlook shadow mapping retries are durable:** `calendar_sync_outbox` operation `outlook-shadow-reconcile` + pg-boss execute worker (with backoff), not in-process timers.
</Check>

<Note>
**`calendar_sync_outbox` is RLS-protected** like other org tables (`Migration20260428180000_EnableCalendarSyncOutboxRls`).
</Note>

## Personal agenda timeline

**`GET /v1/calendar/me/agenda/timeline`** is used by the CRM web **agenda** view for infinite scroll. Same merged shapes as **`/calendar/me`** (`CalendarItemDto`).

<Steps>
  <Step title="Viewer zone">
    Requires the standard **`Time-Zone`** request header (IANA); invalid/missing → `UTC` (see `TimezoneMiddleware` / `TimezoneParam`).
  </Step>
  
  <Step title="Sort">
    Tasks by **due** (`startDate` on the DTO mirrors due); CRM events and externals by **start**; stable tie-break by source + id (`calendar-timeline.util.ts`).
  </Step>
  
  <Step title="Probe">
    From the page anchor day, fetch a merged window of **`CALENDAR_TIMELINE_PROBE_LOOKAHEAD_DAYS`** (default 60) days, merge-sort, apply the first-page visibility rule (`max(from, now)` for tasks; events/externals also kept when they **overlap the viewer's local "today"** even if `start` is before the floor), take the first **`CALENDAR_TIMELINE_PAGE_TARGET`** (40) rows.
  </Step>
  
  <Step title="Day span">
    Union of local calendar days each of those rows touches (date-only tasks use `dueDate` `YYYY-MM-DD` prefix; timed items use Luxon in the viewer zone; all-day events walk date labels). **`fromDay` is floored at the page anchor** (`anchorBoundedDaySpan`): multi-day CRM events / externals can contribute keys **before** the anchor while still overlapping the probe window; without the floor, clamp + `nextCursor` could repeat and clients would refetch the same cursor.
  </Step>
  
  <Step title="Authoritative fetch">
    `collectMergedCalendarItems` for `[startOf(D_min), endOf(D_max)]` in the viewer zone (CRM events already use **interval overlap** in `EventService.getCalendarEvents`).
  </Step>
  
  <Step title="Response">
    `{ window: { fromDay, toDay, timeZone }, items, nextCursor, hasMore }`. `nextCursor` is base64 JSON with `{ v, nextDay }` where `nextDay` is the local calendar day **after** `toDay`.
  </Step>
</Steps>

<Tip>
The legacy **`GET /v1/calendar/me/agenda`** endpoint remains for callers that want a single forward scan without pagination. **`CalendarService.collectMergedCalendarItems`** is **personal-merge only** (tasks + CRM events for the subject + externals when subject is the viewer); organization stacked feeds use **`getOrgCalendarItems`**, not this helper.
</Tip>

## Data model

### `calendar_integration`

`CalendarIntegration` stores one active external calendar account per user, organization, and provider.

**Important fields:**

| Field | Description |
|-------|-------------|
| `user` | CRM user who connected the calendar |
| `organization` | Tenant boundary |
| `provider` | `google` or `outlook` |
| `calendarEmail` | External account email used for display and attendee matching |
| `encryptedAccessToken` | AES-GCM encrypted access token — **nullable** after disconnect |
| `encryptedRefreshToken` | AES-GCM encrypted refresh token when issued |
| `tokenExpiresAt` | Refresh threshold source |
| `watchChannelId`, `watchChannelVerifyToken`, `watchResourceId`, `watchExpiration` | Google push watch state |
| `googleSyncToken` | Google incremental sync token |
| `outlookSubscriptionId`, `outlookSubscriptionExpiration` | Microsoft Graph subscription state |
| `lastInboundWebhookAt`, `lastOutboundSyncAt` | Observability timestamps |
| `lastSyncErrorAt`, `lastSyncErrorMessage` | Last provider/webhook/processing failure |
| `isActive`, `isDeleted` | Connection status and soft delete |

<Note>
**`watchChannelVerifyToken`** is the opaque secret Google echoes as `X-Goog-Channel-Token` (distinct from channel `id`).

The unique active index is now effectively one row per `(user, organization, provider)`. Reconnecting the same provider creates a new row and soft-deletes the previous one.
</Note>
