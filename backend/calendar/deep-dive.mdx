The calendar system operates as a distributed sync system with three distinct layers that must work together coherently:

### Core API Routes

<AccordionGroup>

    - **`GET /v1/calendar/me/agenda/timeline`** - CRM web agenda with day-window pagination
    - **`GET /v1/calendar`** (legacy) - Delegates to `/me` handler
    - **`GET /v1/calendar/agenda`** (legacy) - Delegates to `/me/agenda` handler

## Data Architecture

### Calendar Integration Entity

The `calendar_integration` table stores one active external calendar account per user, organization, and provider:

```typescript
interface CalendarIntegration {
  user: User;                           // CRM user who connected
  organization: Organization;           // Tenant boundary
  provider: 'google' | 'outlook';      // External provider
  calendarEmail: string;               // External account email
  encryptedAccessToken: string | null; // AES-GCM encrypted (null after disconnect)
  encryptedRefreshToken: string;       // AES-GCM encrypted refresh token
  tokenExpiresAt: Date;                // Refresh threshold
  
  // Google-specific fields
  watchChannelId: string;
  watchChannelVerifyToken: string;     // Distinct from channel ID
  watchResourceId: string;
  watchExpiration: Date;
  googleSyncToken: string;
  
  // Outlook-specific fields
  outlookSubscriptionId: string;
  outlookSubscriptionExpiration: Date;
  
  // Health monitoring
  lastInboundWebhookAt: Date;
  lastOutboundSyncAt: Date;
  lastSyncErrorAt: Date;
  lastSyncErrorMessage: string;
  
  // Status
  isActive: boolean;
  isDeleted: boolean;
}
```

<Note>
The unique active index ensures one row per `(user, organization, provider)`. Reconnecting the same provider creates a new row and soft-deletes the previous one.
</Note>

### Calendar Event Mapping

The `calendar_event_mapping` table tracks synchronization between CRM events and external calendar items:

```typescript
interface CalendarEventMapping {
```

## Personal Agenda Timeline

The `/v1/calendar/me/agenda/timeline` endpoint powers the CRM web agenda view with infinite scroll pagination:

<Steps>
  <Step title="Timezone Detection">
    Requires the `Time-Zone` header (IANA format). Invalid/missing defaults to UTC via `TimezoneMiddleware`.
  </Step>

  <Step title="Probe Window">
    From the anchor day, fetch a merged window of `CALENDAR_TIMELINE_PROBE_LOOKAHEAD_DAYS` (default 60 days).
  </Step>

  <Step title="Merge and Sort">
    - **Tasks**: Sorted by due date (`startDate` mirrors due)
    - **CRM Events & Externals**: Sorted by start time
    - **Tie-breaker**: Stable sort by source + ID
  </Step>

  <Step title="Visibility Rules">
    - Tasks: `max(from, now)` floor
    - Events/Externals: Kept when overlapping viewer's local "today"
    - Take first `CALENDAR_TIMELINE_PAGE_TARGET` (40) rows
  </Step>

  <Step title="Day Span Calculation">
    Union of local calendar days each row touches, floored at page anchor to prevent cursor repeats.
  </Step>
</Steps>

### Response Format

```typescript
interface TimelineResponse {
  window: {
    fromDay: string;    // YYYY-MM-DD

    timeZone: string;   // Viewer's timezone
```

## Security & Authentication

### Webhook Security

<Tabs>
  <Tab title="Google Calendar">
    - Stores separate `watchChannelVerifyToken` (not the same as channel ID)
    - Echoed as `X-Goog-Channel-Token` header
    - Must match `X-Goog-Resource-Id` when `watchResourceId` present
    - Legacy rows may incorrectly equate token and channel ID
  </Tab>

  <Tab title="Outlook Calendar">
    - Requires `clientState` to equal integration UUID
    - Microsoft Graph `validationTokens` verified as JWTs (JWKS)
    - Audience: `app.outlookCalendar.webhookValidationAudience` 
    - Returns `202` only after JWT verification
  </Tab>
</Tabs>

<Warning>
In production (`buildAppConfig().isProduction`), webhook URLs must be HTTPS and cannot target localhost:
- `GOOGLE_CALENDAR_WEBHOOK_URL`
- `OUTLOOK_CALENDAR_WEBHOOK_URL`
</Warning>

### Row Level Security

The `calendar_sync_outbox` table is RLS-protected like other organizational tables via `Migration20260428180000_EnableCalendarSyncOutboxRls`.

## Sync Architecture

### Intended Model

<Check>**CRM events are the canonical business record**</Check>

- Creator calendar copies can update CRM event details when changed externally
- Invitee calendar copies can update RSVP status only  
- External-only items are displayed read-only (not imported into CRM)
- Provider echoes are suppressed with change IDs, fingerprints, ETags, and time windows

### Echo Suppression

The system uses multiple mechanisms to prevent sync loops:

1. **Change IDs**: Track outbound change fingerprints
2. **ETags**: Detect actual provider-side changes
3. **Time Windows**: Short-circuit recent outbound pushes
4. **Fingerprints**: Content hashing for deep change detection

### Durable Operations

<Info>
Outlook shadow mapping retries are durable via `calendar_sync_outbox` with operation type `outlook-shadow-reconcile` + pg-boss execution worker (with backoff), not in-process timers.
</Info>

## Production Risks

<Warning>
The system attempts to make two different calendar providers behave like one coherent event bus. This is inherently fragile.
</Warning>

### Known Weak Points

1. **In-memory cache**: Potential data loss on server restart
2. **CRM-originated sync**: No durable queue for listener path  
3. **Provider edge cases**: 
   - Recurring events handling
   - All-day event timezone issues
   - Deleted invitee reconciliation
   - Outlook mailbox-local event IDs

### Monitoring

Track these metrics for system health:

- `lastInboundWebhookAt`: Recent webhook processing
- `lastOutboundSyncAt`: Successful CRM→provider pushes  
- `lastSyncErrorAt`/`lastSyncErrorMessage`: Failure tracking
- Webhook retry counts and backoff behavior
- Token refresh success rates
