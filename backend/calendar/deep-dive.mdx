The calendar system is a distributed sync system that integrates CRM events with Google Calendar and Outlook through OAuth, webhooks, and bidirectional synchronization. This guide provides a complete technical overview of the architecture, integration flows, and implementation details.

<Warning>
The calendar integration is not a simple "push event to Google" feature. It's a distributed sync system with two external providers, eventually consistent mailboxes, webhook retries, token expiry, RLS, notifications, and user actions happening from both CRM and external calendars.
</Warning>

## System Architecture

The system operates on three calendar layers:

### 1. CRM Calendar Feed

Primary routes for accessing calendar data:

<Tabs>
<Tab title="Personal Calendar">
**`GET /v1/calendar/me`** - Personal workload including CRM tasks/subtasks, CRM events, and external items when viewing yourself.
</Tab>
<Tab title="Organization Calendar">  
**`GET /v1/calendar/org`** - Organization stacked calendar for 1-10 other org members, returning `{ rosterUserIds, itemsByUserId }` with personal-scope tasks/subtasks and CRM events per subject (no externals).
</Tab>
<Tab title="Team Calendar">
**`GET /v1/calendar/teams/:teamId`** - Requires `team.admin`, `team_crm.*`, or `team_sales.*` permissions on the team, or org admin/owner access. Shows CRM-only data for other roster members (authenticated user excluded).
</Tab>
</Tabs>

<Note>
Legacy `GET /v1/calendar` and `GET /v1/calendar/agenda` delegate to the same handlers as `/me` endpoints.
</Note>

### 2. CRM Events

`EventService` owns the internal event source of truth, managing:
- Event invitees and RSVP status
- Soft deletion
- Notification emission
- Access control (creator + invitee + org-calendar policy)

### 3. Calendar Integrations

Google and Outlook drivers handle:
- Pushing CRM events outward to provider calendars
- Processing provider webhooks back into CRM
- OAuth authentication and token management

## Personal Agenda Timeline

The `/v1/calendar/me/agenda/timeline` endpoint serves the CRM web agenda list with day-window pagination.

<Steps>
<Step title="Timezone Processing">
Requires the `Time-Zone` request header (IANA timezone). Invalid or missing values default to `UTC` via `TimezoneMiddleware`.
</Step>

<Step title="Sorting Logic">
- Tasks sorted by **due date** (`startDate` on DTO mirrors due)
- CRM events and externals sorted by **start time**
- Stable tie-break by source + id
</Step>

<Step title="Probe Window">
From the anchor day, fetches a merged window of `CALENDAR_TIMELINE_PROBE_LOOKAHEAD_DAYS` (default 60) days, merge-sorts results, and applies visibility rules.
</Step>

<Step title="Day Span Calculation">
Calculates the union of local calendar days each row touches, with `fromDay` floored at the page anchor to prevent cursor repetition.
</Step>

<Step title="Authoritative Fetch">
Performs `collectMergedCalendarItems` for the calculated date range in the viewer's timezone.
</Step>
</Steps>

## OAuth State Security

<Info>
Calendar OAuth flows use `SignedPayloadService` with HMAC-SHA256 signed state tokens to prevent CSRF and replay attacks.
</Info>

The OAuth state includes:
- Purpose: `calendar.oauth.state`
- Payload: `{ userId, organizationId }`
- Configurable TTL via `CALENDAR_OAUTH_STATE_TTL_MS`
- Secret: `app.signedPayload.stateSecret`

Token format: `v1.<base64url(envelope)>.<base64url(sig)>`

## Sync Architecture

### Outbound Sync (CRM → Provider)

<Steps>
<Step title="Event Creation/Update">
When CRM events are created or updated, `EventCalendarSyncService` handles outbound synchronization.
</Step>

<Step title="Calendar Mapping">
`CalendarEventMapping` entities track the relationship between CRM events and provider calendar entries.
</Step>

<Step title="Echo Suppression">
Change IDs, fingerprints, ETags, and short time windows prevent processing of echoed changes.
</Step>

<Step title="Error Handling">
Failed syncs are queued in `calendar_sync_outbox` with durable retry logic via pg-boss.
</Step>
</Steps>

### Inbound Sync (Provider → CRM)

<Steps>
<Step title="Webhook Receipt">
Provider webhooks are received at `/webhooks/calendar/google` and `/webhooks/calendar/outlook`.
</Step>

<Step title="Authentication">
Webhook authenticity is verified using provider-specific tokens and signatures.
</Step>

<Step title="Change Processing">
`CalendarWebhookService` processes changes and updates CRM events appropriately.
</Step>

<Step title="Permission Checks">
Only creators can update event details; invitees can only update RSVP status.
</Step>
</Steps>

## Risk Areas

<Warning>
The system attempts to make two different calendar providers behave like one coherent event bus, which is inherently fragile.
</Warning>

Key risk areas include:

1. **In-memory caching** without persistence across restarts
2. **CRM-originated outbound sync** without durable queuing on the listener path  
3. **Provider-specific edge cases** around:
   - Recurring events
   - All-day events
   - Deleted invitees
   - Outlook mailbox-local event IDs

<Tip>
The code includes good guardrails, but production deployments should monitor sync health metrics and implement alerting for stuck queues or repeated failures.
</Tip>
