# Calendar System Deep Dive

This document explains how the current calendar system actually works. The calendar integration is not a simple "push event to Google" feature—it's a distributed sync system with two hostile external providers, eventually consistent mailboxes, webhook retries, token expiry, RLS, notifications, and user actions happening from both CRM and external calendars.

<Warning>
The system is trying to make two different calendar providers behave like one coherent event bus. This is inherently fragile and requires careful attention to production risks.
</Warning>

## Executive Summary

The system has three calendar layers:

### 1. CRM Calendar Feed

Primary routes include:
- **`GET /v1/calendar/me`** - Personal workload (CRM tasks/subtasks + CRM events + external items when viewing yourself)
- **`GET /v1/calendar/org`** - Organization stacked calendar (1-10 other org members)
- **`GET /v1/calendar/teams/:teamId`** - Team calendar for authorized members
- **`GET /v1/calendar/me/agenda`** - Personal merged agenda list
- **`GET /v1/calendar/me/agenda/timeline`** - CRM web agenda with day-window pagination

### 2. CRM Events

`EventService` owns the internal event source of truth, including:
- Event details and metadata
- Invitee management
- RSVP status tracking
- Soft deletion
- Notification emission

### 3. Calendar Integrations

Google and Outlook drivers that:
- Push CRM events outward to external calendars
- Process provider webhooks back into CRM
- Handle OAuth flows and token management

## Security Features

### Webhook Authenticity

<Tabs>
<Tab title="Google Calendar">
Google stores a separate `watchChannelVerifyToken` that is:
- Sent as the `token` in `events.watch`
- Echoed as `X-Goog-Channel-Token`
- **Not** the same as the channel `id`

<Note>Legacy rows may still equate the token and ID. When `watchResourceId` is present, inbound notifications must also match `X-Goog-Resource-Id`.</Note>
</Tab>

<Tab title="Outlook Calendar">
Outlook requires `clientState` to equal the integration UUID.

When Microsoft Graph includes `validationTokens` on a change-notification batch, JWTs are verified (JWKS) before the handler returns `202`.

In production, webhook URLs must be HTTPS and not target localhost.
</Tab>
</Tabs>

### OAuth State Security

Calendar OAuth flows use `SignedPayloadService` with HMAC-SHA256 signed state tokens:

```typescript
// State token format: v1.<base64url(envelope)>.<base64url(sig)>
// Envelope contains:
{
  purpose: 'calendar.oauth.state',
  iat: timestamp,
  exp: timestamp,
  payload: { userId, organizationId }
}
```

<Info>
The service is provided globally via `CryptoModule` and supports multiple purposes to prevent cross-feature token replay attacks.
</Info>

## Personal Agenda Timeline

The `GET /v1/calendar/me/agenda/timeline` endpoint is used by the CRM web agenda view for infinite scroll.

<Steps>
<Step title="Viewer Zone">
Requires the standard `Time-Zone` request header (IANA). Invalid/missing values default to `UTC`.
</Step>

<Step title="Sort Logic">
- Tasks by **due date** (`startDate` on DTO mirrors due)
- CRM events and externals by **start time**
- Stable tie-break by source + id
</Step>

<Step title="Probe Window">
From the page anchor day, fetch a merged window of `CALENDAR_TIMELINE_PROBE_LOOKAHEAD_DAYS` (default 60) days, merge-sort, and apply visibility rules.
</Step>

<Step title="Response Format">
Returns `{ window: { fromDay, toDay, timeZone }, items, nextCursor, hasMore }`
</Step>
</Steps>

### Timezone Middleware

All calendar endpoints support the `Time-Zone` request header (IANA timezone). The `TimezoneMiddleware` validates the header and falls back to `UTC` for invalid values.

## Sync Mechanisms

### Outbound Sync

<Warning>
CRM-originated outbound sync uses listener path without durable queue, which is production-dangerous.
</Warning>

The system uses `calendar_sync_outbox` for durable operations:
- Outlook shadow mapping retries are durable
- Uses pg-boss execute worker with backoff
- RLS-protected like other org tables

### Echo Suppression

Provider echoes are suppressed using:
- Change IDs
- Fingerprints
- ETags
- Short time windows

### Recurring Events Handling

<Tip>
Provider-specific edge cases exist around recurring events, all-day events, deleted invitees, and Outlook mailbox-local event IDs.
</Tip>

## Production Considerations

### High-Risk Areas

<Warning>
The following areas require careful monitoring in production:
- In-memory cache management
- CRM-originated outbound sync without durable queue
- Provider-specific edge cases
- Webhook authenticity validation
- Token expiry handling
</Warning>

### Migration Snapshot Hygiene

<Check>
`migrations/.snapshot-neondb.json` is the baseline for `npx mikro-orm migration:check`. If migration check reports large diffs unrelated to your change, refresh the snapshot using the standard process.
</Check>

### Database Tables

Key tables in the calendar system:
- `calendar_integration` - Integration configurations
- `calendar_event_mapping` - Event mappings between CRM and providers
- `calendar_sync_outbox` - Durable sync operations queue

## Monitoring and Health

The system includes sync health columns and monitoring capabilities:
- Integration status tracking
- Sync operation success/failure rates
- Webhook delivery monitoring
- OAuth token validity tracking

Monitor the calendar sync outbox for failed operations and implement alerting for webhook delivery failures.
