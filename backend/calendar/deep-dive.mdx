## Executive Summary

The calendar system operates as a distributed sync engine across three primary layers, managing complex interactions between CRM data and external calendar providers.

<Warning>
This is not a simple "push event to Google" feature. The calendar integration is a distributed sync system with two hostile external providers, eventually consistent mailboxes, webhook retries, token expiry, RLS, notifications, and user actions happening from both CRM and external calendars.
</Warning>

### System Architecture

The system has three calendar layers:

1. **CRM calendar feed**: Primary routes for personal workload, organization stacked calendar, and team calendars
2. **CRM events**: Internal event source of truth with invitees, RSVP status, and notifications
3. **Calendar integrations**: Google and Outlook drivers for bidirectional sync

## API Endpoints Overview

### Personal Calendar Routes

<Tabs>
<Tab title="Personal Workload">
**`GET /v1/calendar/me`**

Personal workload including:
- CRM tasks/subtasks
- CRM events 
- External items (when viewing yourself)
</Tab>

<Tab title="Organization Calendar">
**`GET /v1/calendar/org`**

Organization stacked calendar with `userIds` parameter (1-10 other org members):
- Response: `{ rosterUserIds, itemsByUserId }`
- Personal-scope tasks/subtasks + CRM events per subject
- No externals included
- Viewer ID in query is ignored
</Tab>

<Tab title="Team Calendar">
**`GET /v1/calendar/teams/:teamId`**

Requires `team.admin`, `team_crm.*`, or `team_sales.*` permissions on that team, or org admin/owner:
- CRM-only per other roster member
- Authenticated user excluded from roster
- Use `/me` for self
</Tab>
</Tabs>

## Personal Agenda Timeline

The timeline endpoint (`GET /v1/calendar/me/agenda/timeline`) powers the CRM web agenda view with infinite scroll capabilities.

<Steps>
<Step title="Timezone Validation">
Requires the standard `Time-Zone` request header (IANA format). Invalid or missing values default to `UTC`.
</Step>

<Step title="Probe Window">
From the page anchor day, fetches a merged window of `CALENDAR_TIMELINE_PROBE_LOOKAHEAD_DAYS` (default 60) days.
</Step>

<Step title="Sorting and Filtering">
- **Tasks**: Sorted by due date (`startDate` mirrors due)
- **CRM events and externals**: Sorted by start time
- **Stable tie-break**: By source + id
- **Visibility rule**: `max(from, now)` for tasks; events/externals kept when overlapping viewer's local "today"
</Step>

<Step title="Day Span Calculation">
Calculates the union of local calendar days each row touches, with `fromDay` floored at the page anchor to prevent cursor repetition.
</Step>

<Step title="Response Generation">
Returns `{ window: { fromDay, toDay, timeZone }, items, nextCursor, hasMore }` with base64 JSON cursor containing `{ v, nextDay }`.
</Step>
</Steps>

## Calendar Integration Model

### Intended Sync Behavior

<AccordionGroup>
<Accordion title="CRM Events as Source of Truth">
- CRM events are the canonical business record
- Creator calendar copies can update CRM event details when changed externally
- Invitee calendar copies can update RSVP only
- External-only items are displayed read-only and not imported into CRM
</Accordion>

<Accordion title="Echo Suppression">
Provider echoes are suppressed using:
- Change IDs
- Fingerprints
- ETags
- Short time windows
</Accordion>
</AccordionGroup>

### Risk Assessment

<Warning>
The biggest risk is that the system attempts to make two different calendar providers behave like one coherent event bus. This is inherently fragile.
</Warning>

Production-dangerous areas include:

- In-memory cache
- CRM-originated outbound sync without durable queue (listener path)
- Provider-specific edge cases around recurring events, all-day events, deleted invitees
- Outlook mailbox-local event IDs

## Security and Authentication

### Webhook Authenticity

<Tabs>
<Tab title="Google Calendar">
- Stores separate `watchChannelVerifyToken` (sent as `token` in `events.watch`)
- Echoed as `X-Goog-Channel-Token` (not the same as channel `id`)
- When `watchResourceId` present, inbound notifications must match `X-Goog-Resource-Id`
- Legacy rows may equate token and ID incorrectly
</Tab>

<Tab title="Outlook Calendar">
- Requires `clientState` to equal the integration UUID
- When Microsoft Graph includes `validationTokens` on change-notification batch:
  - JWTs verified (JWKS) before handler returns `202`
  - Audience: `app.outlookCalendar.webhookValidationAudience`
  - Defaults to calendar app client ID
</Tab>
</Tabs>

### Production Requirements

<Check>
In production (`buildAppConfig().isProduction`):
- `GOOGLE_CALENDAR_WEBHOOK_URL` must be HTTPS and not target localhost
- `OUTLOOK_CALENDAR_WEBHOOK_URL` must be HTTPS and not target localhost
</Check>

### OAuth State Security

Calendar OAuth flows use `SignedPayloadService` with HMAC-SHA256 signed state tokens:

```typescript
// State payload structure
{
  userId: string,
  organizationId: string
}
```

- **Purpose**: `calendar.oauth.state`
- **TTL**: `CALENDAR_OAUTH_STATE_TTL_MS`
- **Secret**: `app.signedPayload.stateSecret`
- **Format**: `v1.<base64url(envelope)>.<base64url(sig)>`

## Data Persistence

### Durable Operations

<Info>
Outlook shadow mapping retries are durable using `calendar_sync_outbox` operation `outlook-shadow-reconcile` + pg-boss execute worker (with backoff), not in-process timers.

## Event Access Control

Events use a creator + invitee + org-calendar policy model:

- **ACCESS**: Creator OR invitee OR org-calendar policy OR team-calendar visibility
- **MODIFY/DELETE**: Creator-only with additional business rules
- **RSVP**: Invitees can update their own status

<Note>
The access control model ensures proper data isolation while allowing necessary collaboration features.
</Note>
