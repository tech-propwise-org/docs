This document explains how the calendar system actually works in production. The calendar integration is a distributed sync system with external providers, webhook retries, token expiry, RLS, and user actions happening from both CRM and external calendars.

<Warning>
The system is trying to make two different calendar providers behave like one coherent event bus. This is inherently fragile and requires careful operational monitoring.
</Warning>

## System Architecture

The system has three calendar layers:

### 1. CRM Calendar Feed

Primary routes for calendar data access:

<Tabs>
  <Tab title="Personal Calendar">
    **`GET /v1/calendar/me`** - Personal workload including:
    - CRM tasks/subtasks 
    - CRM events
    - External items (when viewing yourself)
  </Tab>
  <Tab title="Organization Calendar">
    **`GET /v1/calendar/org`** - Organization stacked calendar:
    - Query: `userIds` = 1–10 other org members
    - Response: `{ rosterUserIds, itemsByUserId }`
    - Includes: Personal-scope tasks/subtasks + CRM events per subject
    - Excludes: External items, viewer ID ignored
  </Tab>
  <Tab title="Team Calendar">
    **`GET /v1/calendar/teams/:teamId`** - Team-specific calendar:
    - Access: `team.admin`, `team_crm.*`, `team_sales.*`, or org admin/owner
    - Scope: CRM-only per roster member
    - Note: Authenticated user excluded from roster (use `/me` for self)
  </Tab>
</Tabs>

<Info>
Legacy endpoints `GET /v1/calendar` and `GET /v1/calendar/agenda` delegate to the same handlers as `/me` and `/me/agenda`.
</Info>

### 2. CRM Events

`EventService` owns the internal event source of truth, including:
- Event details and metadata
- Invitee management
- RSVP status tracking
- Soft deletion
- Notification emission

### 3. Calendar Integrations

Google and Outlook drivers that:
- Push CRM events outward to external calendars
- Process provider webhooks back into CRM
- Handle OAuth flows and token management

## Calendar Timeline API

The timeline endpoint serves the CRM web agenda view with infinite scroll support.

### Endpoint Details

**`GET /v1/calendar/me/agenda/timeline`**

<Steps>
  <Step title="Timezone handling">
    Requires `Time-Zone` request header (IANA format). Invalid/missing values default to `UTC`.
  </Step>
  <Step title="Sorting logic">
    - Tasks: By due date (`startDate` on DTO mirrors due)
    - CRM events and externals: By start time
    - Tie-breaker: Source + ID for stability
  </Step>
  <Step title="Probe window">
    From anchor day, fetch merged window of `CALENDAR_TIMELINE_PROBE_LOOKAHEAD_DAYS` (default 60 days).
  </Step>
  <Step title="Page processing">
    Take first `CALENDAR_TIMELINE_PAGE_TARGET` (40) rows after applying visibility rules.
  </Step>
</Steps>

### Response Format

```json
{
  "window": {
    "fromDay": "2024-01-15",
    "toDay": "2024-02-14", 
    "timeZone": "America/New_York"
  },
  "items": [...],
  "nextCursor": "eyJ2IjoxLCJuZXh0RGF5IjoiMjAyNC0wMi0xNSJ9",
  "hasMore": true
}
```

<Note>
`nextCursor` contains base64 JSON with `{ v, nextDay }` where `nextDay` is the local calendar day after `toDay`.
</Note>

```typescript
// Token format: v1.<base64url(envelope)>.<base64url(sig)>
// Envelope contains: { purpose, iat, exp, payload }

```

### Webhook Authentication

<Tabs>
  <Tab title="Google Calendar">
    - Stores separate `watchChannelVerifyToken`
    - Sent as `token` in `events.watch`
    - Echoed as `X-Goog-Channel-Token` header
    - Must match `X-Goog-Resource-Id` when `watchResourceId` present
  </Tab>
  <Tab title="Outlook/Microsoft Graph">
    - Requires `clientState` to equal integration UUID
    - Validates JWT `validationTokens` via JWKS
    - Audience: `app.outlookCalendar.webhookValidationAudience`
    - Returns `202` only after successful verification
  </Tab>
</Tabs>

<Warning>
In production (`buildAppConfig().isProduction`), webhook URLs must be HTTPS and cannot target localhost.
</Warning>

## Sync Mechanism

### Intended Model

<Check>CRM events are the canonical business record</Check>
<Check>Creator calendar copies can update CRM event details when changed externally</Check>
<Check>Invitee calendar copies can update RSVP only</Check>
<Check>External-only items are displayed read-only and not imported into CRM</Check>
<Check>Provider echoes are suppressed with change IDs, fingerprints, ETags, and time windows</Check>

### Echo Suppression Strategy

The system prevents sync loops using multiple mechanisms:

1. **Change IDs**: Track outbound operations with unique identifiers
2. **Fingerprints**: Hash event content to detect actual changes
3. **ETags**: Use provider-specific versioning when available  
4. **Time Windows**: Short-term suppression of expected echoes

### Outlook Shadow Mapping

<Tip>
Outlook shadow mapping retries are durable using `calendar_sync_outbox` operations with pg-boss workers and backoff, not in-process timers.
</Tip>

## Operational Considerations

### Production Risks

<Warning>
The following areas require careful monitoring in production:
</Warning>

1. **In-memory cache**: Can become inconsistent during high load
2. **CRM-originated outbound sync**: Uses listeners without durable queue
3. **Provider edge cases**: Recurring events, all-day events, deleted invitees
4. **Outlook mailbox-local IDs**: Provider-specific identifier challenges

### MikroORM Snapshot Hygiene

<Steps>
  <Step title="Check migration status">
    Run `npx mikro-orm migration:check` to verify snapshot alignment
  </Step>
  <Step title="Handle drift">
    If large unrelated diffs appear, refresh snapshot using standard MikroORM process
  </Step>
  <Step title="Avoid broken migrations">
    Do not merge `migration:create` dumps that recreate unrelated tables
  </Step>
</Steps>

<Note>
`migrations/.snapshot-neondb.json` is the baseline for migration checks. Keep it synchronized with team entity metadata expectations.
</Note>

### RLS Protection

The `calendar_sync_outbox` table is RLS-protected like other organization tables via `Migration20260428180000_EnableCalendarSyncOutboxRls`.

## Event Access Control

Events use a creator + invitee + org-calendar policy model:

- **ACCESS**: Creator OR invitee OR org-calendar policy OR team-calendar visibility
- **MODIFY/DELETE**: Restricted based on role and relationship to event

This ensures proper data isolation while maintaining necessary collaboration features.
