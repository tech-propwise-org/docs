The calendar system is a distributed sync architecture managing CRM events, Google Calendar integration, Outlook/Microsoft Graph integration, OAuth flows, webhooks, and complex sync mechanisms. This guide provides a complete technical overview of the implementation.

<Warning>
The calendar integration is not a simple "push event to Google" feature. It is a distributed sync system with two hostile external providers, eventually consistent mailboxes, webhook retries, token expiry, RLS, notifications, and user actions happening from both CRM and external calendars.
</Warning>

## System Architecture

The system operates with three distinct calendar layers:

### 1. CRM Calendar Feed

The primary calendar feed routes include:

- **`GET /v1/calendar/me`** - Personal workload (CRM tasks/subtasks + CRM events + external items when viewing yourself)
- **`GET /v1/calendar/org`** - Organization stacked calendar (1-10 other org members)
- **`GET /v1/calendar/teams/:teamId`** - Team calendar with appropriate permissions

<Info>
Legacy endpoints `GET /v1/calendar` and `GET /v1/calendar/agenda` delegate to the same handlers as `/me` and `/me/agenda`.
</Info>

### 2. CRM Events

`EventService` owns the internal event source of truth, including:
- Event invitees and RSVP status
- Soft deletion handling
- Notification emission

### 3. Calendar Integrations

Google and Outlook drivers handle:
- Pushing CRM events outward to external calendars
- Processing provider webhooks back into CRM
- OAuth authentication flows

## Personal Agenda Timeline

The `/v1/calendar/me/agenda/timeline` endpoint provides infinite scroll functionality for the CRM web agenda view.

<Steps>
  <Step title="Viewer Zone Processing">
    Requires the standard `Time-Zone` request header (IANA format). Invalid or missing values default to UTC.
  </Step>
  
  <Step title="Sorting Strategy">
    - Tasks sorted by due date (`startDate` on DTO mirrors due)
    - CRM events and externals sorted by start time
    - Stable tie-break by source + id
  </Step>
  
  <Step title="Probe Window">
    From the page anchor day, fetch a merged window of `CALENDAR_TIMELINE_PROBE_LOOKAHEAD_DAYS` (default 60) days.
  </Step>
  
  <Step title="Day Span Calculation">
    Union of local calendar days each row touches, with `fromDay` floored at the page anchor to prevent cursor repetition.
  </Step>
  
  <Step title="Authoritative Fetch">
    `collectMergedCalendarItems` for the calculated date range in viewer timezone.
  </Step>
</Steps>

### Response Format

```json
{
  "window": {
    "fromDay": "2024-01-15",
    "toDay": "2024-03-15", 
    "timeZone": "America/New_York"
  },
  "items": [...],
  "nextCursor": "base64EncodedCursor",
  "hasMore": true
}
```

## Webhook Security

<AccordionGroup>
  <Accordion title="Google Calendar Webhook Authentication">
    Google stores a separate `watchChannelVerifyToken` sent as the `token` in `events.watch` and echoed as `X-Goog-Channel-Token`. When `watchResourceId` is present, inbound notifications must match `X-Goog-Resource-Id`.

    <Warning>
    The verify token is not the same as the channel ID. Legacy rows may incorrectly equate the two.
    </Warning>
  </Accordion>

  <Accordion title="Outlook Webhook Validation">
    Outlook requires `clientState` to equal the integration UUID. When Microsoft Graph includes `validationTokens` on a change-notification batch, JWTs are verified using JWKS before the handler returns `202`.

    The audience for validation is `app.outlookCalendar.webhookValidationAudience`, defaulting to the calendar app client ID.
  </Accordion>

  <Accordion title="Production Requirements">
    In production (`buildAppConfig().isProduction`), both `GOOGLE_CALENDAR_WEBHOOK_URL` and `OUTLOOK_CALENDAR_WEBHOOK_URL` must be:
    - HTTPS endpoints
    - Not targeting localhost
  </Accordion>
</AccordionGroup>

## Sync Mechanisms

### Echo Suppression

The system prevents infinite sync loops using multiple techniques:

- **Change IDs** - Track modifications to prevent re-processing
- **Fingerprints** - Content hashing to detect actual changes  
- **ETags** - Provider-specific versioning
- **Time Windows** - Short windows to suppress rapid echoes

### Durable Operations

<Check>
Outlook shadow mapping retries use durable `calendar_sync_outbox` operations with pg-boss execution workers and backoff, not in-process timers.
</Check>

The `calendar_sync_outbox` table is RLS-protected like other org tables via `Migration20260428180000_EnableCalendarSyncOutboxRls`.

## Event Access Control

Events use a multi-layered access control model:

- **Creator access** - Event creator has full permissions
- **Invitee access** - Event invitees can view and update RSVP status  
- **Organization calendar policy** - Org-wide visibility rules
- **Team calendar visibility** - Team-specific access controls

<Warning>
The intended model assumes CRM events are the canonical business record. External changes should only update CRM event details when made by the creator, or RSVP status when made by invitees.
</Warning>

## Known Risks

<AccordionGroup>
  <Accordion title="Provider Coherence">
    The system attempts to make two different calendar providers behave like one coherent event bus, which is inherently fragile. While good guardrails exist, this remains the primary architectural risk.
  </Accordion>

  <Accordion title="In-Memory Cache">
    Some calendar operations rely on in-memory caching which can lead to inconsistencies in multi-instance deployments or during restarts.
  </Accordion>

  <Accordion title="Sync Without Durable Queue">  
    CRM-originated outbound sync lacks a durable queue for the listener path, potentially losing events during failures.
  </Accordion>

  <Accordion title="Provider Edge Cases">
    Provider-specific edge cases around recurring events, all-day events, deleted invitees, and Outlook mailbox-local event IDs can cause sync inconsistencies.
  </Accordion>
</AccordionGroup>

## Best Practices

<Tabs>
  <Tab title="Development">
    - Always use constants from `event-calendar-sync.constants.ts`
    - Test webhook authentication thoroughly
    - Verify timezone handling in all calendar operations
    - Use the timeline probe for pagination instead of cursor-based approaches
  </Tab>
  
  <Tab title="Operations">
    - Monitor webhook delivery failures and token refresh issues
    - Watch for echo suppression effectiveness in logs
    - Track sync outbox operation success rates
    - Verify OAuth state token validation is working
  </Tab>
  
  <Tab title="Security">
    - Ensure production webhook URLs are HTTPS and non-localhost
    - Validate JWT tokens for Microsoft Graph webhooks
    - Use signed payload service for all OAuth state management
    - Implement proper RLS on all calendar-related tables
  </Tab>
</Tabs>
