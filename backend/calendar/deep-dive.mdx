<Warning>
The calendar integration is not a simple "push event to Google" feature. It is a distributed sync system with two hostile external providers, eventually consistent mailboxes, webhook retries, token expiry, RLS, notifications, and user actions happening from both CRM and external calendars.
</Warning>

## Executive summary

The system has three calendar layers:

1. **CRM calendar feed**: Primary routes for personal workload, organization stacked calendar, and team calendars
2. **CRM events**: Internal event source of truth with invitees, RSVP status, and notifications
3. **Calendar integrations**: Google and Outlook drivers for bidirectional sync

### Intended model

- CRM events are the canonical business record
- Creator calendar copies can update CRM event details when changed externally
- Invitee calendar copies can update RSVP only
- External-only items are displayed read-only and are not imported into CRM
- Provider echoes are suppressed with change IDs, fingerprints, ETags, and short time windows

<Note>
The biggest risk is that the system is trying to make two different calendar providers behave like one coherent event bus. That is inherently fragile.
</Note>

## Calendar feed routes

### Personal calendar feed

**`GET /v1/calendar/me`** - Personal workload including:
- CRM tasks/subtasks 
- CRM events
- External items when viewing yourself

**`GET /v1/calendar/me/agenda`** - Personal merged agenda list with:
- Optional inclusive `from` parameter
- Forward scan up to server lookahead cap
- No `limit`/`cursor` pagination
- Response includes `truncated: true/false` to indicate if more items exist

**`GET /v1/calendar/me/agenda/timeline`** - CRM web agenda list with day-window pagination:
- Merged probe → local `fromDay`/`toDay` in viewer `Time-Zone` header
- Full refetch for that window
- `nextCursor` for following page

### Organization calendar feed

**`GET /v1/calendar/org`** - Organization stacked calendar:
- Query parameter: `userIds` (1-10 other org members)
- Response: `{ rosterUserIds, itemsByUserId }`
- Personal-scope tasks/subtasks + CRM events per subject
- No externals included
- Viewer ID in query is ignored

### Team calendar feed

**`GET /v1/calendar/teams/:teamId`** - Team calendar requiring:
- `team.admin`, `team_crm.*`, or `team_sales.*` permissions on that team
- OR org admin/owner
- CRM-only per other roster member
- Authenticated user excluded from roster (use `/me` for self)

<Info>
Legacy routes `GET /v1/calendar` and `GET /v1/calendar/agenda` delegate to the same handlers as `/me` and `/me/agenda`.
</Info>

## Personal agenda timeline

The `/v1/calendar/me/agenda/timeline` endpoint serves the CRM web agenda view with infinite scroll support.

<Steps>
<Step title="Viewer zone setup">
Requires the standard `Time-Zone` request header (IANA timezone). Invalid/missing values default to `UTC`.
</Step>

<Step title="Sorting strategy">
- Tasks by **due** (`startDate` on DTO mirrors due)
- CRM events and externals by **start** 
- Stable tie-break by source + id
</Step>

<Step title="Probe window">
From the page anchor day, fetch a merged window of `CALENDAR_TIMELINE_PROBE_LOOKAHEAD_DAYS` (default 60) days, merge-sort, and apply visibility rules.
</Step>

<Step title="Day span calculation">
Union of local calendar days each row touches, with `fromDay` floored at the page anchor to prevent cursor repetition.
</Step>

<Step title="Authoritative fetch">
`collectMergedCalendarItems` for `[startOf(D_min), endOf(D_max)]` in the viewer zone.
</Step>
</Steps>

**Response format:**
```json
{
  "window": {
    "fromDay": "2024-01-15",
    "toDay": "2024-01-20", 
    "timeZone": "America/New_York"
  },
  "items": [...],
  "nextCursor": "eyJ2IjoxLCJuZXh0RGF5IjoiMjAyNC0wMS0yMSJ9",
  "hasMore": true
}
```

## OAuth state security

Calendar OAuth flows use `SignedPayloadService` with HMAC-SHA256 signed state tokens.

### State token format
```
v1.<base64url(envelope)>.<base64url(signature)>
```

**Envelope contents:**
- `purpose`: `calendar.oauth.state`
- `iat`: issued at timestamp
- `exp`: expiration timestamp  
- `payload`: `{ userId, organizationId }`

<Tip>
This prevents cross-site request forgery and replay attacks. The service supports multiple purposes to prevent cross-feature token replay attacks.
</Tip>

## Webhook security

### Google Calendar webhooks

- Stores separate `watchChannelVerifyToken` (sent as `token` in `events.watch`)
- Echoed as `X-Goog-Channel-Token` header
- When `watchResourceId` is present, notifications must match `X-Goog-Resource-Id`

### Outlook Calendar webhooks  

- Requires `clientState` to equal the integration UUID
- Microsoft Graph `validationTokens` are JWT-verified (JWKS)
- Audience: `app.outlookCalendar.webhookValidationAudience`

<Warning>
In production (`buildAppConfig().isProduction`), `GOOGLE_CALENDAR_WEBHOOK_URL` and `OUTLOOK_CALENDAR_WEBHOOK_URL` must be HTTPS and must not target localhost.
</Warning>

## Sync mechanisms

### Outbound sync

CRM events are pushed to provider calendars when:
- Event is created/updated/deleted
- Invitee list changes
- RSVP status updates

### Echo suppression

The system prevents infinite sync loops using:
- Change IDs and fingerprints
- ETags for version tracking  
- Short time windows for change detection
- Provider-specific echo patterns

### Outlook shadow mapping

Outlook shadow mapping retries are durable:
- `calendar_sync_outbox` operation `outlook-shadow-reconcile`
- pg-boss execute worker with backoff
- Not in-process timers

<Check>
`calendar_sync_outbox` is RLS-protected like other org tables via `Migration20260428180000_EnableCalendarSyncOutboxRls`.
</Check>

## MikroORM snapshot hygiene

<Warning>
`migrations/.snapshot-neondb.json` is the baseline for `npx mikro-orm migration:check` / `migration:create`. If `migration:check` reports a large diff unrelated to your change, the snapshot has drifted from the expected entity metadata.
</Warning>

**To refresh the snapshot:**
1. Ensure dev database has all migrations applied
2. Regenerate snapshot per MikroORM guidance  
3. Do **not** merge a `migration:create` dump that tries to recreate unrelated tables

## Event access control

Events use a creator + invitee + org-calendar policy model:

### Access permissions
- **VIEW**: Creator OR invitee OR org-calendar policy OR team-calendar visibility
- **MODIFY**: Creator OR invitee (RSVP only) OR org admin
- **DELETE**: Creator OR org admin

### RLS implementation
Calendar tables inherit the organization-scoped RLS pattern used throughout the system.

## Weak points and recommendations

1. **In-memory cache**: Consider persistent caching for calendar data
2. **Sync queue durability**: Implement durable queues for all outbound sync operations  
3. **Provider edge cases**: Handle recurring events, all-day events, and deleted invitees consistently
4. **Outlook mailbox-local IDs**: Account for provider-specific event ID semantics
5. **Webhook authenticity**: Ensure all webhook validations are production-hardened

<Tip>
The calendar system works well for standard use cases but requires careful monitoring and error handling for edge cases involving external provider inconsistencies.
</Tip>
