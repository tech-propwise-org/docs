<Warning>
This document is intentionally blunt: the calendar integration is not a simple "push event to Google" feature. It is a distributed sync system with two hostile external providers, eventually consistent mailboxes, webhook retries, token expiry, RLS, notifications, and user actions happening from both CRM and external calendars.
</Warning>

### CRM Calendar Feed Routes

<AccordionGroup>
  <Accordion title="GET /v1/calendar/me - Personal Workload">
    Personal workload calendar showing:
    - CRM tasks and subtasks
    - CRM events
    - External items when viewing yourself
  </Accordion>

  <Accordion title="GET /v1/calendar/org - Organization Stacked Calendar">
    Organization-wide stacked calendar (`userIds` = 1–10 other org members):
    - Response format: `{ rosterUserIds, itemsByUserId }`
    - Personal-scope tasks/subtasks + CRM events per subject
    - No externals included
    - Viewer ID in query is ignored
  </Accordion>

  <Accordion title="GET /v1/calendar/teams/:teamId - Team Calendar">
    Team calendar requiring `team.admin`, `team_crm.*`, or `team_sales.*` permissions:
    - Shows CRM-only items per other roster member
    - Authenticated user excluded from roster
    - Use `/me` endpoint for self-view
  </Accordion>

  <Accordion title="GET /v1/calendar/me/agenda - Personal Agenda List">
    Personal merged agenda list with:
    - Optional inclusive `from` parameter
    - Forward scan up to server lookahead cap
    - **No** `limit`/`cursor` pagination (see `calendar-agenda.constants.ts`)
    - Response includes `truncated: true` when merge loop exits before lookahead horizon
    - `truncated: false` when scan completes full lookahead window
  </Accordion>

  <Accordion title="GET /v1/calendar/me/agenda/timeline - Web Agenda Timeline">
    CRM web agenda list with day-window pagination:
    - Merged probe → local `fromDay`/`toDay` in viewer `Time-Zone` header
    - Full refetch for that window
    - `nextCursor` for following page
    - See `calendar-timeline.*` files
  </Accordion>
</AccordionGroup>

<Note>
Legacy routes `GET /v1/calendar` and `GET /v1/calendar/agenda` delegate to the same handlers as `/me` and `/me/agenda`.
</Note>

## Architecture Risks

<Warning>
The biggest risk is that the system is trying to make two different calendar providers behave like one coherent event bus. That is inherently fragile.
</Warning>

### Known Production Dangers

- CRM-originated outbound sync without durable queue (listener path)
- Remaining provider HTTP inside tenant transactions (now **without** multi-second Outlook shadow polling)
- Provider-specific edge cases around:
  - Recurring events
  - All-day events
  - Deleted invitees
  - Outlook mailbox-local event IDs

### Security: Webhook Authenticity

<Tabs>
  <Tab title="Google Calendar">
    Google stores a **separate** `watchChannelVerifyToken`:
    - Sent as `token` in `events.watch` request
    - Echoed as `X-Goog-Channel-Token` header
    - **Not** the same as channel `id` (legacy rows may incorrectly equate them)
    - When `watchResourceId` is present, inbound notifications must match `X-Goog-Resource-Id`
  </Tab>
  
  <Tab title="Outlook Calendar">
    Outlook verification requirements:
    - `clientState` must equal the integration UUID
    - When Microsoft Graph includes `validationTokens` on change-notification batch:
      - JWTs verified via JWKS before returning `202`
      - Audience: `app.outlookCalendar.webhookValidationAudience`
      - Defaults to calendar app client ID
  </Tab>
</Tabs>

<Check>
In **production** (`buildAppConfig().isProduction`), `GOOGLE_CALENDAR_WEBHOOK_URL` and `OUTLOOK_CALENDAR_WEBHOOK_URL` must be **https** and must not target localhost.
</Check>

### Durability Guarantees

<Info>
- Public calendar webhooks are throttled
- Outlook batches are capped before async sync fan-out
- **Outlook shadow mapping retries are durable:** `calendar_sync_outbox` operation `outlook-shadow-reconcile` + pg-boss execute worker (with backoff), not in-process timers
- **RLS Protection:** `calendar_sync_outbox`, `calendar_external_cache`, and `calendar_oauth_intent` are RLS-protected like other org tables
</Info>

- Processed by `TimezoneMiddleware` / `TimezoneParam`

### Sort Order

<Tabs>
  <Tab title="Tasks">
    Sorted by **due date** (`startDate` on DTO mirrors due)
  </Tab>
  
  <Tab title="CRM Events & Externals">
    Sorted by **start time**
  </Tab>
  
  <Tab title="Tie-Breaking">
    Stable tie-break by source + id (see `calendar-timeline.util.ts`)
  </Tab>
</Tabs>

### Timeline Algorithm

<Steps>
  <Step title="Probe Window">
    From page anchor day, fetch merged window of `CALENDAR_TIMELINE_PROBE_LOOKAHEAD_DAYS` (default: 60 days)
  </Step>
  
  <Step title="Merge and Sort">
    Merge-sort items and apply first-page visibility rule:
    - Tasks: `max(from, now)`
    - Events/externals: kept when they overlap viewer's local "today" even if start is before the floor
  </Step>
  
  <Step title="Page Target">
    Take first `CALENDAR_TIMELINE_PAGE_TARGET` (40) rows
  </Step>
  
  <Step title="Calculate Day Span">
    Union of local calendar days each row touches:
    - Date-only tasks use `dueDate` `YYYY-MM-DD` prefix
    - Timed items use Luxon in viewer zone
    - All-day events walk date labels
    
    **Important:** `fromDay` is floored at page anchor (`anchorBoundedDaySpan`)
  </Step>
  
  <Step title="Clamp to Max Span">
    `D_min` / `D_max` clamped to `CALENDAR_TIMELINE_MAX_DAY_SPAN` days
  </Step>
  
  <Step title="Authoritative Fetch">
    `collectMergedCalendarItems` for `[startOf(D_min), endOf(D_max)]` in viewer zone
    
    CRM events use **interval overlap** in `EventService.getCalendarEvents`
  </Step>
</Steps>

### Day Keys and Client Rendering

<Warning>
Timeline items include optional `agendaDayKeys: string[]` for this endpoint only.
</Warning>

**Server-Side Processing:**
- Day keys are viewer-local day buckets the row should render in
- Clipped to response window (`fromDay`–`toDay`)
- Later pages cannot inject content into already-rendered earlier days
- **Items with empty clipped `agendaDayKeys` are dropped before response**

**Client-Side Rules:**

<Tabs>
  <Tab title="With agendaDayKeys">
    - Group agenda rows by `agendaDayKeys` when present
    - If `agendaDayKeys` is defined but empty array, skip the row entirely
    - Do **not** fall back to browser-local date parsing
  </Tab>
  
  <Tab title="Without agendaDayKeys">
    Browser-local parsing is only a fallback for:
    - Rows where `agendaDayKeys` is entirely absent
    - Non-timeline callers
    - Legacy paths
  </Tab>
  
  <Tab title="Pagination">
    - Use response `window.toDay` as loaded-through boundary
    - `nextCursor` is base64 JSON: `{ v, nextDay }`
    - `nextDay` is local calendar day **after** `toDay`
  </Tab>
</Tabs>

### All-Day Event Handling

<Check>
**West-of-UTC all-day leak (fixed)**
</Check>

All-day CRM events are stored as UTC midnight (`T00:00:00.000Z`). The authoritative UTC refetch range for a page ending at local `toDay` extends to `endOf(toDay)` in the viewer zone, which may reach into the following UTC day for western timezones.

**Example:**
- Viewer in `America/Los_Angeles` (UTC-8)
- Page window ends at local `2024-01-15`
- `endOf("2024-01-15")` = `2024-01-16T07:59:59.999Z`
- All-day event on `2024-01-16T00:00:00.000Z` falls within this UTC range
- Without the day-key clip, would incorrectly appear on the current page

**Solution:**
The server generates `agendaDayKeys` using the viewer's timezone, then clips to `[fromDay, toDay]` before sending. Any all-day event that belongs to a later local day will have an empty clipped array and be dropped from the response.

## Event Sync Lifecycle

### Outbound: CRM → Provider

<Steps>
  <Step title="CRM Event Creation">
    User creates event in CRM via `EventService.createEvent()`
  </Step>
  
  <Step title="Sync Trigger">
    `EventCalendarSyncService` detects event needs sync:
    
    - Creator has active calendar integration
    - Event not marked as external-only
    - No existing mapping or mapping needs update
  </Step>
  
  <Step title="Driver Push">
    Appropriate driver pushes to provider:
    
    ```typescript
    const providerEvent = await driver.pushEvent({
      summary: crmEvent.title,
      start: crmEvent.startTime,
      end: crmEvent.endTime,
      attendees: crmEvent.invitees,
      // ... other fields
    });
    ```
  </Step>
  
  <Step title="Mapping Creation">
    Create `CalendarEventMapping` record:
    
    ```typescript
    {
      crmEventId: crmEvent.id,
      integrationId: integration.id,
      providerEventId: providerEvent.id,
      iCalUid: providerEvent.iCalUid,
      lastOutboundFingerprint: computeFingerprint(crmEvent),
      lastOutboundChangeId: uuid()
    }
    ```
  </Step>
  
  <Step title="Echo Suppression">
    Store change ID and fingerprint to suppress webhook echo when provider notifies us of the change we just made
  </Step>
</Steps>

### Inbound: Provider → CRM

<Steps>
  <Step title="Webhook Reception">
    Provider sends notification to webhook endpoint:
    
    - Google: `/webhooks/google-calendar`
    - Outlook: `/webhooks/outlook-calendar`
  </Step>
  
  <Step title="Webhook Validation">
    Verify webhook authenticity (see Security section above)
  </Step>
  
  <Step title="Change Detection">
    Fetch updated event from provider and compare with CRM state:
    
    ```typescript
    const providerEvent = await driver.fetchEvent(eventId);
    const mapping = await findMapping(eventId);
    const crmEvent = await EventService.findById(mapping.crmEventId);
    ```
  </Step>
  
  <Step title="Echo Suppression Check">
    ```typescript
    if (isRecentEcho(providerEvent, mapping)) {
      // Skip - this is our own change echoing back
      return;
    }
    ```
    
    Checks:
    - Change ID match (if provider supports)
    - Fingerprint match
    - Recent timestamp window
  </Step>
  
  <Step title="Apply Changes">
    Update CRM event based on user role:
    
    <Tabs>
      <Tab title="Creator Calendar">
        Can update all event details:
        - Title, description
        - Start/end times
        - Location
        - Custom fields
        - Attendee list
      </Tab>
      
      <Tab title="Invitee Calendar">
        Can only update RSVP status:
        - Accept
        - Decline
        - Tentative
        - No response
      </Tab>
    </Tabs>
  </Step>
  
  <Step title="Notification Dispatch">
    `EventService` emits notifications based on change type:
    
    - Time/location changes → all attendees
    - RSVP updates → creator only
    - See `event-notification-recipients.util.ts`
  </Step>
</Steps>

## Watch Channel Renewal

Both Google and Outlook require periodic webhook subscription renewal.

### Renewal Handler

<CodeGroup>

```typescript calendar-watch-renewal.handler.ts
@Injectable()
export class CalendarWatchRenewalHandler {
  @Cron('0 * * * *')  // Hourly
  async renewExpiringSoon() {
    const expiringIntegrations = await this.findExpiringSoon(
      Duration.fromHours(24)
    );
    
    for (const integration of expiringIntegrations) {
      await this.renewWebhook(integration);
    }
  }
}
```

```typescript Renewal Logic
async renewWebhook(integration: CalendarIntegration) {
  const driver = this.registry.get(integration.provider);
  
  try {
    const newConfig = await driver.setupWebhook(integration);
    
    await this.integrationRepo.update(integration.id, {
      watchChannelId: newConfig.channelId,
      watchResourceId: newConfig.resourceId,
      watchExpiresAt: newConfig.expiresAt
    });
    
    this.logger.info('Webhook renewed', { integrationId: integration.id });
  } catch (error) {
    await this.markSyncUnhealthy(integration, error);
  }
}
```

</CodeGroup>

<Warning>
If webhook renewal fails multiple times, the integration is marked unhealthy and stops syncing. User must reconnect their calendar.
</Warning>

## External Cache

The `calendar_external_cache` table stores read-only external calendar items for display in CRM calendars.

### Purpose

<Info>
External items are events that exist in the user's Google/Outlook calendar but were not created by CRM. They are displayed in merged calendar views but cannot be edited in CRM.
</Info>

### Schema

```sql
CREATE TABLE calendar_external_cache (
  id UUID PRIMARY KEY,
  organization_id UUID NOT NULL,
  integration_id UUID NOT NULL,
  provider_event_id VARCHAR(255) NOT NULL,
  ical_uid VARCHAR(500),
  
  summary TEXT,
  description TEXT,
  location TEXT,
  start_time TIMESTAMPTZ NOT NULL,
  end_time TIMESTAMPTZ NOT NULL,
  is_all_day BOOLEAN NOT NULL DEFAULT false,
  
  provider_etag VARCHAR(255),
  provider_updated_at TIMESTAMPTZ,
  
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
```

## Change Detection and Echo Suppression

### Why Echo Suppression is Critical

When CRM pushes an event update to Google/Outlook, the provider immediately notifies us back via webhook. Without suppression, this creates an infinite loop:

1. CRM updates event title
2. Push to Google Calendar
3. Google webhook fires: "Event updated"
4. CRM processes webhook, sees title change
5. Push back to Google (no actual change)
6. Google webhook fires again...

### Suppression Strategies

<Tabs>
  <Tab title="Change ID">
    Generate UUID for each outbound change:
    
    ```typescript
    const changeId = uuid();
    
    await driver.pushEvent(event, {
      extendedProperties: {
        private: { crmChangeId: changeId }
      }
    });
    
    await mapping.update({ lastOutboundChangeId: changeId });
    ```
    
    When webhook arrives:
    ```typescript
    if (providerEvent.extendedProperties?.private?.crmChangeId === 
        mapping.lastOutboundChangeId) {
      return; // Our echo, suppress
    }
    ```
  </Tab>
  
  <Tab title="Content Fingerprint">
    Hash relevant event fields:
    
    ```typescript
    function computeFingerprint(event: CrmEvent): string {
      return createHash('sha256')
        .update(JSON.stringify({
          title: event.title,
          description: event.description,
          start: event.startTime.toISOString(),
          end: event.endTime.toISOString(),
          location: event.location,
          attendeeIds: event.inviteeIds.sort()
        }))
        .digest('hex');
    }
    ```
    
    When webhook arrives:
    ```typescript
    const inboundFingerprint = computeFingerprint(providerEvent);
    if (inboundFingerprint === mapping.lastOutboundFingerprint) {
      return; // Exact match, suppress
    }
    ```
  </Tab>
  
  <Tab title="Time Window">
    Suppress echoes within short time window:
    
    ```typescript
    const ECHO_WINDOW_MS = 5000; // 5 seconds
    
    if (providerEvent.updated < mapping.lastOutboundAt + ECHO_WINDOW_MS) {
      // Likely our echo, be conservative
      return;
    }
    ```
  </Tab>
  
  <Tab title="Provider ETag">
    Use provider-specific version markers:
    
    ```typescript
    // Google: etag header
    // Outlook: @odata.etag property
    
    if (providerEvent.etag === mapping.lastKnownEtag) {
      return; // No actual change
    }
    ```
  </Tab>
</Tabs>

<Warning>
The system uses **all four strategies** in combination. A webhook is only processed if it passes all suppression checks. This multi-layer approach prevents false positives while catching genuine external changes.
</Warning>

### Deep Change Detection

For non-echo webhooks, determine what actually changed:

<CodeGroup>

```typescript event-change-detection.util.ts
export interface EventChanges {
  hasTimeChange: boolean;
  hasLocationChange: boolean;
  hasAttendeeChange: boolean;
  hasCustomFieldChange: boolean;
  rsvpChanges: Map<string, RsvpStatus>;
}

export function detectChanges(
  before: CrmEvent,
  after: ProviderEvent
): EventChanges {
  return {
    hasTimeChange: 
      !before.startTime.equals(after.start) ||
      !before.endTime.equals(after.end),
    
    hasLocationChange:
      before.location !== after.location,
    
    hasAttendeeChange:
      !arraysEqual(
        sortBy(before.inviteeIds),
        sortBy(after.attendees.map(a => a.crmUserId))
      ),
    
    hasCustomFieldChange:
      !deepEqual(before.customFields, after.customFields),
    
    rsvpChanges: detectRsvpChanges(before, after)
  };
}
```

```typescript Notification Dispatch
const changes = detectChanges(crmEvent, providerEvent);

if (changes.hasTimeChange || changes.hasLocationChange) {
  // Notify all attendees
  await notificationService.sendEventUpdate(
    crmEvent,
    await getEventAttendeeRecipients(crmEvent)
  );
}

if (changes.rsvpChanges.size > 0) {
  // Notify creator only
  await notificationService.sendRsvpUpdates(
    crmEvent,
    changes.rsvpChanges,
    [crmEvent.createdBy]
  );
}
```

## Recurring Events

<Warning>
Recurring events are one of the most complex areas. Google and Outlook handle recurrence very differently.
</Warning>

### Current Limitations

<Info>
**CRM does not currently support creating recurring events.** All recurring event handling is read-only for display purposes.
</Info>

When a user creates a recurring event in Google/Outlook:

1. The series appears in external cache as individual occurrences
2. Each occurrence is a separate cache entry
3. Occurrences are read-only in CRM calendar views
4. No sync back to CRM events table

### Provider Differences

<Tabs>
  <Tab title="Google Calendar">
    Google uses RFC 5545 (iCalendar) recurrence rules:
    
    ```json
    {
      "summary": "Weekly Team Sync",
      "start": { "dateTime": "2024-01-15T10:00:00Z" },
      "end": { "dateTime": "2024-01-15T11:00:00Z" },
      "recurrence": [
        "RRULE:FREQ=WEEKLY;BYDAY=MO;COUNT=10"
      ]
    }
    ```
    
    - Recurring event has a master event ID
    - Individual occurrences have `recurringEventId` pointing to master
    - Can query specific occurrence: `events.get(eventId + '_20240115T100000Z')`
  </Tab>
  
  <Tab title="Outlook Calendar">
    Outlook uses Microsoft's pattern/range structure:
    
    ```json
    {
      "subject": "Weekly Team Sync",
      "start": { "dateTime": "2024-01-15T10:00:00" },
      "end": { "dateTime": "2024-01-15T11:00:00" },
      "recurrence": {
        "pattern": {
          "type": "weekly",
          "daysOfWeek": ["monday"],
          "interval": 1
        },
        "range": {
          "type": "numbered",
          "numberOfOccurrences": 10
        }
      }
    }
    ```
    
    - Series master has `type: "seriesMaster"`
    - Occurrences have `type: "occurrence"`
    - Exception instances have `type: "exception"`
  </Tab>
</Tabs>

If CRM adds recurring event support:

  </Step>
  
  <Step title="Series Expansion">
    Server expands series for calendar queries:
    ```typescript

    ```
  </Step>
  
  <Step title="Exception Handling">
    Track modified/deleted occurrences separately
  </Step>
  
  <Step title="Sync Logic">

    - Handle single-occurrence edits

  </Step>
</Steps>

## Token Management

### Access Token Lifecycle

<Steps>
  <Step title="Initial Grant">
    OAuth flow provides:
    - Access token (short-lived: 1 hour)
    - Refresh token (long-lived: until revoked)
  </Step>
  
  <Step title="Token Storage">
    Encrypted in `calendar_integration` table:
    
    ```typescript
    {
      accessToken: await encryption.encrypt(tokens.access_token),
      refreshToken: await encryption.encrypt(tokens.refresh_token),
      expiresAt: new Date(Date.now() + tokens.expires_in * 1000)
    }
    ```
  </Step>
  
  <Step title="Automatic Refresh">
    Before making API calls:
    
    ```typescript
    async getValidAccessToken(integration: CalendarIntegration) {
      if (integration.expiresAt > addMinutes(now(), 5)) {
        return integration.accessToken;
      }
      
      const newTokens = await this.refreshToken(integration);
      await this.updateIntegration(integration.id, newTokens);
      return newTokens.accessToken;
    }
    ```
  </Step>
  
  <Step title="Refresh Failure">
    If refresh fails (revoked, expired):
    
    - Mark integration as `status: 'disconnected'`
    - Stop sync operations
    - Notify user to reconnect
  </Step>
</Steps>

### Security Best Practices

<Check>
**Current Implementation:**
- Tokens encrypted at rest using `EncryptionService`
- Refresh tokens never logged
- Token refresh happens automatically before expiry
- Failed refreshes trigger graceful degradation
</Check>

<Warning>
**Avoid:**
- Logging decrypted tokens
- Sending tokens to frontend
- Storing tokens in external cache or session storage
- Using expired tokens (always check `expiresAt`)

### Symptom: Infinite Sync Loops

<Warning>
This is the most critical failure mode. Signs include:
- High API rate limit errors
- Duplicate event notifications
- Database transaction conflicts on mappings
</Warning>

<Steps>
  <Step title="Identify Looping Event">
    Query recent mapping updates:
    
    ```sql
    SELECT 
      cem.crm_event_id,
      e.title,
      cem.updated_at,
      COUNT(*) OVER (
        PARTITION BY cem.crm_event_id 
        ORDER BY cem.updated_at
        RANGE BETWEEN INTERVAL '5 minutes' PRECEDING AND CURRENT ROW
      ) as recent_updates
    FROM calendar_event_mapping cem
    JOIN events e ON e.id = cem.crm_event_id
    WHERE cem.updated_at > NOW() - INTERVAL '1 hour'
    ORDER BY recent_updates DESC;
    ```
    
    Events with `recent_updates > 10` are likely looping
  </Step>
  
  <Step title="Check Echo Suppression">
    Review mapping suppression fields:
    
    ```sql
    SELECT 
      last_outbound_change_id,
      last_outbound_fingerprint,
      last_outbound_at,
      last_inbound_change_id
    FROM calendar_event_mapping
    WHERE crm_event_id = '<looping-event-id>';
    ```
    
    If these aren't updating, echo suppression is failing
  </Step>
  
  <Step title="Temporary Loop Break">
    Emergency stop for specific event:
    
    ```sql
    -- Soft delete mapping to stop sync
    UPDATE calendar_event_mapping
    SET deleted_at = NOW()
    WHERE crm_event_id = '<looping-event-id>';
    ```
    
    This preserves data while stopping the loop
  </Step>
  
  <Step title="Root Cause Analysis">
    Common causes:
    - Provider returning stale ETag
    - Fingerprint not including all changed fields
    - Webhook arriving before outbound transaction commits
    - Provider modifying fields automatically (e.g., location normalization)
  </Step>
</Steps>

### Symptom: All-Day Events Wrong Day

<Info>
All-day events are particularly tricky with timezone conversions.
</Info>

<Steps>
  <Step title="Verify Storage">
    All-day events should store as UTC midnight:
    
    ```sql
    SELECT 
      title,
      start_time,
      end_time,
      is_all_day
    FROM events
    WHERE is_all_day = true
      AND start_time::time != '00:00:00'::time;
    ```
    
    Non-midnight times indicate incorrect storage
  </Step>
  
  <Step title="Check Timeline Response">
    Review `agendaDayKeys` in API response:
    
    ```json
    {
      "title": "All-Day Event",
      "startDate": "2024-01-15",
      "isAllDay": true,
      "agendaDayKeys": ["2024-01-15"]  // Should match startDate
    }
    ```
  </Step>
  
  <Step title="Verify Client Timezone">
    Ensure `Time-Zone` header is set correctly:
    
    ```
    GET /v1/calendar/me/agenda/timeline
    Time-Zone: America/New_York
    ```
    
    Missing header defaults to UTC
  </Step>
  
  <Step title="Test Edge Case">
    Create test all-day event in western timezone (e.g., `America/Los_Angeles`):
    
    - Storage: `2024-01-15T00:00:00.000Z`
    - Display: Should appear on `2024-01-15` in all timezones
    - Bug would show `2024-01-14` for UTC-8 viewers
  </Step>
</Steps>

```sql Query Recent Webhook Activity
SELECT 
  ci.provider,
  ci.user_id,
  cem.crm_event_id,
  e.title as event_title,
  cem.updated_at as last_mapping_update,
  cem.last_inbound_change_id,
  cem.last_outbound_change_id
FROM calendar_integration ci
JOIN calendar_event_mapping cem ON cem.integration_id = ci.id
JOIN events e ON e.id = cem.crm_event_id
WHERE cem.updated_at > NOW() - INTERVAL '1 hour'
ORDER BY cem.updated_at DESC;
```
