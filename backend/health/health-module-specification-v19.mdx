## Overview

The health module provides comprehensive application health monitoring through standardized health check endpoints. Built on NestJS Terminus, it implements a two-endpoint strategy distinguishing between liveness and readiness probes for container orchestration platforms like Kubernetes.

<Info>
The module monitors critical application dependencies including database connectivity, memory usage, and background job queue status through custom health indicators.
</Info>

The module is registered in `app.module.ts` and provides dedicated health check endpoints separate from the main application controller.

## API Endpoints

### GET /health/live

<Tip>
**Purpose:** Liveness probe - indicates if the process can handle requests
</Tip>

```typescript
@Get('live')
@PublicEndpoint()
@HealthCheck()
liveness(): Promise<HealthCheckResult>
```

**Response:**
```json
{
  "status": "ok",
  "info": {},
  "error": {},
  "details": {}
}
```

**Use Cases:**
- Container orchestration restart decisions
- Load balancer routing decisions
- Basic process health verification

### GET /health/ready

<Tip>
**Purpose:** Readiness probe - indicates if the application can serve traffic
</Tip>

```typescript
@Get('ready')
@PublicEndpoint()
@HealthCheck()
readiness(): Promise<HealthCheckResult>
```

**Health Checks Performed:**
- Database connectivity (MikroORM)
- PgBoss job queue status

**Response Format:**
```json
{
  "status": "ok|error",
  "info": {
    "database": { "status": "up" },
    "pgboss": { "status": "up" }
  },
  "error": {},
  "details": {
    "database": { "status": "up" },
    "pgboss": { "status": "up" }
  }
}
```

### Scheduled Data Integrity Audit

<Warning>
This audit is **not wired into `/health/ready`** and has no public `/health/data-integrity` endpoint.
</Warning>

**Purpose:** Detect orphan rows resulting from soft-delete cascade gaps without exposing an unauthenticated, high-cost HTTP route.

The daily queued job runs at `30 3 * * *` by default (override via `DATA_INTEGRITY_AUDIT_CRON`) through `DataIntegrityAuditHandler`. It logs every run, emits a warn-level application log when unhealthy orphans are found, and sends a grouped Sentry error event when `totalOrphans > 0`.

<Steps>
  <Step title="Audit Execution">
    Runs ~74 count queries grouped into module buckets using parallel `Promise.all` queries
  </Step>
  
  <Step title="Coverage Analysis">
    Audits 12 different module buckets covering core entities, sub-entities, activities, and junctions
  </Step>
  
  <Step title="Error Reporting">
    Emits Sentry error events with detailed context when orphan rows are detected
  </Step>
</Steps>

<AccordionGroup>
  <Accordion title="Audit Coverage Details">
    
**Core (12)** — Lead/Deal/Contact/Company/Listing with deleted persons/orgs/stages/creators

**Person & Company sub-entities (10)** — Person.organization, Person.createdBy, Person{Email,Phone,Address,Channel}.person, PersonNotDuplicate (either side), Company{Email,Phone,Address}.company

**Activity stack + Reminder (10)** — Activity, Note, CallLog, EmailLog, SmsLog, Meeting, SiteVisit organization refs; Activity.performedBy and Note.createdBy creator refs; Reminder.user

**Tasks + Events + Tag (10)** — Tag.organization; Task.organization, Task.createdBy; TaskComment.task; TaskAssignee.task; TaskTag.tag; Event.organization, Event.createdBy; EventInvitee.event; EventTag.event

**Other modules** — Stakeholder, Transfer, Stage histories, Junctions, Commission, DealDocument, Auth sessions, RBAC, Distribution, Escalation, Notifications

  </Accordion>
</AccordionGroup>

<Note>
Soft-deleted creator refs are informational only and do **not** contribute to `totalOrphans`. User churn is normal, and historical records should still display the deleted creator when needed.
</Note>

**Why separate from `/health/ready`:** Orphan rows do not affect whether the instance can serve traffic — they are a data-integrity concern, not a service-availability concern. Wiring this into the readiness probe would block rollouts unrelated to the symptom.

## Health Indicators

### PgBossHealthIndicator

Custom health indicator for the PgBoss background job service. Uses the `HealthIndicatorService` API (NestJS Terminus v11+) instead of the deprecated `HealthIndicator` base class.

```typescript
@Injectable()
export class PgBossHealthIndicator {
  constructor(
    @Inject(QUEUE_SERVICE) private readonly queueService: IQueueService,
    private readonly healthIndicatorService: HealthIndicatorService,
  ) {}

  async isHealthy(key: string) {
    const indicator = this.healthIndicatorService.check(key);
    if (!this.queueService.isStarted) {
      return indicator.down({ reason: 'boss.start() has not completed' });
    }
    return indicator.up();
  }
}
```

<Check>
Uses `HealthIndicatorService.check(key)` with `.up()` / `.down()` instead of the legacy `extends HealthIndicator` + `HealthCheckError` pattern
</Check>

**Implementation Notes:**
- Checks `IQueueService.isStarted` flag rather than database queries
- Lightweight check avoiding additional database load
- Returns healthy when PgBoss is properly started and connected

## Business Rules

### Health Check Behavior

<Tabs>
  <Tab title="Liveness Checks">
    Always return 200 if the process can handle HTTP requests
  </Tab>
  
  <Tab title="Readiness Checks">
    Return 503 if any critical dependency is unavailable
  </Tab>
  
  <Tab title="Public Access">
    All health endpoints are publicly accessible (no authentication required)
  </Tab>
</Tabs>

- Health endpoints are covered by the global `AppThrottlerGuard`; clients that exceed the configured limit receive 429
- Health checks use version-neutral routing

### Failure Conditions

<Warning>
The following conditions will cause health check failures:
</Warning>

- Database connection lost or unresponsive
- Memory usage exceeds configured thresholds
- PgBoss service not started or connection failed
- Any critical service dependency unavailable

### Response Standards

<CodeGroup>
```json HTTP 200 - All Checks Passing

  "status": "ok",
  "info": {
    "database": { "status": "up" },
    "pgboss": { "status": "up" }
  },
  "error": {},
  "details": {}
}
```

```json HTTP 503 - Check Failures
{
  "status": "error",
  "info": {},
  "error": {
    "database": { "status": "down", "message": "Connection timeout" }
  },
  "details": {}
}
```

```json HTTP 429 - Rate Limited
{
  "statusCode": 429,
  "message": "ThrottlerException: Too Many Requests"
}
```
</CodeGroup>

### Error Handling

```typescript
// Health check failures use indicator.down() (NestJS Terminus v11+)
const indicator = this.healthIndicatorService.check(key);
return indicator.down({ reason: 'Service not started' });

// The legacy HealthCheckError + extends HealthIndicator pattern is deprecated
// and will be removed in the next major Terminus release.
```

## Observability Stack

Beyond health check endpoints, the application uses a layered observability architecture:

### Sentry (Error Tracking & Performance)

<Tabs>
  <Tab title="Backend">
    **Configuration** (`@sentry/nestjs`):
    - SDK initialized in `src/instrument.ts` (loaded before NestFactory)
    - `SentryModule.forRoot()` + `SentryGlobalFilter` as `APP_FILTER` captures all unhandled exceptions
    - `LogContextInterceptor` enriches Sentry events with user and organization context
    - `DataIntegrityAuditHandler` explicitly emits grouped error-level events for orphan rows
    - `tracesSampleRate: 0.2` in production (all error events captured at 100%)
    - Health transactions (`GET /health*`) excluded via `ignoreTransactions`
  </Tab>
  
  <Tab title="Frontend">
    **Configuration** (`@sentry/nextjs`):
    - Client config with Session Replay and Breadcrumbs
    - Server and Edge configs for SSR/middleware error capture
    - `next.config.ts` wrapped with `withSentryConfig` for source map uploads
    - `global-error.tsx` root error boundary reports to Sentry
    - `AuthProvider` sets user context and clears on logout
    - `tunnelRoute: '/monitoring'` — proxies Sentry events to avoid ad-blockers
  </Tab>
</Tabs>

**Environment Variables:**
- `SENTRY_DSN` — backend DSN
- `NEXT_PUBLIC_SENTRY_DSN` — frontend DSN
- `SENTRY_AUTH_TOKEN` — optional, for source map uploads during build

### Better Stack (Uptime + Error Logs)

**Uptime Monitoring:**
- External HTTP monitoring of health endpoints
- Multi-region availability checks
- Alert integration for downtime notifications
- Historical uptime metrics and SLA tracking
