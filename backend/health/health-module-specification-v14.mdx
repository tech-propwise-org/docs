## Overview

The health module provides comprehensive application health monitoring through standardized health check endpoints. Built on NestJS Terminus, it implements a two-endpoint strategy distinguishing between liveness and readiness probes for container orchestration platforms like Kubernetes.

<Note>
The module monitors critical application dependencies including database connectivity, memory usage, and background job queue status through custom health indicators.
</Note>

The module is registered in `app.module.ts` and provides dedicated health check endpoints separate from the main application controller.

## API endpoints

### Liveness probe

<Info>
The liveness probe indicates if the process can handle requests and is used by container orchestration for restart decisions.
</Info>

```http
GET /health/live
```

```typescript
@Get('live')
@PublicEndpoint()
@HealthCheck()
liveness(): Promise<HealthCheckResult>
```

**Response:**
```json
{
  "status": "ok",
  "info": {},
  "error": {},
  "details": {}
}
```

**Use cases:**
- Container orchestration restart decisions
- Load balancer routing decisions
- Basic process health verification

### Readiness probe

<Warning>
The readiness probe indicates if the application can serve traffic. Returns 503 if any critical dependency is unavailable.
</Warning>

```http
GET /health/ready
```

```typescript
@Get('ready')
@PublicEndpoint()
@HealthCheck()
readiness(): Promise<HealthCheckResult>
```

**Health checks performed:**
- Database connectivity (MikroORM)
- Memory usage monitoring
- PgBoss job queue status

**Response format:**
```json
{
  "status": "ok|error",
  "info": {
    "database": { "status": "up" },
    "memory_heap": { "status": "up" },
    "pgboss": { "status": "up" }
  },
  "error": {},
  "details": {
    "database": { "status": "up" },
    "memory_heap": { "status": "up" },
    "pgboss": { "status": "up" }
  }
}
```

## Health indicators

### PgBoss health indicator

Custom health indicator for the PgBoss background job service. Uses the `HealthIndicatorService` API (NestJS Terminus v11+) instead of the deprecated `HealthIndicator` base class.

```typescript
@Injectable()
export class PgBossHealthIndicator {
  constructor(
    @Inject(QUEUE_SERVICE) private readonly queueService: IQueueService,
    private readonly healthIndicatorService: HealthIndicatorService,
  ) {}

  async isHealthy(key: string) {
    const indicator = this.healthIndicatorService.check(key);
    if (!this.queueService.isStarted) {
      return indicator.down({ reason: 'boss.start() has not completed' });
    }
    return indicator.up();
  }
}
```

<Tip>
Uses `HealthIndicatorService.check(key)` with `.up()` / `.down()` instead of the legacy `extends HealthIndicator` + `HealthCheckError` pattern.
</Tip>

**Implementation notes:**
- Checks `IQueueService.isStarted` flag rather than database queries
- Lightweight check avoiding additional database load
- Returns healthy when PgBoss is properly started and connected

## Business rules

### Health check behavior

<Steps>
  <Step title="Liveness checks">
    Always return 200 if the process can handle HTTP requests
  </Step>
  <Step title="Readiness checks">
    Return 503 if any critical dependency is unavailable
  </Step>
  <Step title="Public access">
    All health endpoints are publicly accessible (no authentication required)
  </Step>
  <Step title="Version neutral">
    Health checks use version-neutral routing
  </Step>
</Steps>

### Failure conditions

<AccordionGroup>
  <Accordion title="Database issues">
    Database connection lost or unresponsive
  </Accordion>
  <Accordion title="Memory issues">
    Memory usage exceeds configured thresholds
  </Accordion>
  <Accordion title="Queue issues">
    PgBoss service not started or connection failed
  </Accordion>
  <Accordion title="Service dependencies">
    Any critical service dependency unavailable
  </Accordion>
</AccordionGroup>

## Patterns and conventions

### Endpoint naming

```
/health/live   - Liveness probe
/health/ready  - Readiness probe
```

Both use version neutral routing (`VERSION_NEUTRAL`)

### Response standards

<Tabs>
  <Tab title="Success">
    **HTTP 200**: All checks passing
    ```json
    {
      "status": "ok",
      "info": { /* check details */ },
      "error": {},
      "details": { /* all check results */ }
    }
    ```
  </Tab>
  <Tab title="Failure">
    **HTTP 503**: One or more checks failing
    ```json
    {
      "status": "error",
      "info": {},
      "error": { /* failed checks */ },
      "details": { /* all check results */ }
    }
    ```
  </Tab>
</Tabs>

### Error handling

<CodeGroup>
```typescript Modern (v11+)
// Health check failures use indicator.down()
const indicator = this.healthIndicatorService.check(key);
return indicator.down({ reason: 'Service not started' });
```

```typescript Legacy (deprecated)
// The HealthCheckError + extends HealthIndicator pattern
// is deprecated and will be removed
throw new HealthCheckError('Service not started', {
  [key]: { status: 'down', reason: 'Service not started' }
});
```
</CodeGroup>

## Observability stack

Beyond health check endpoints, the application uses a layered observability architecture:

### Sentry (Error tracking & Performance)

<Tabs>
  <Tab title="Backend">
    **Configuration** (`@sentry/nestjs`):
    - SDK initialized in `src/instrument.ts` (loaded before NestFactory)
    - `SentryModule.forRoot()` + `SentryGlobalFilter` as `APP_FILTER`
    - `LogContextInterceptor` enriches events with user context
    - `tracesSampleRate: 0.2` in production
    - `nodeProfilingIntegration()` for performance profiling

    **Context enrichment:**
    ```typescript
    Sentry.setUser({ id, email });
    Sentry.setTag('organization_id', ...);
    ```

    <Note>
    Health transactions (`GET /health*`) are excluded via `ignoreTransactions`
    </Note>
  </Tab>
  <Tab title="Frontend">
    **Configuration** (`@sentry/nextjs`):
    - Client config with Session Replay and Breadcrumbs
    - Server and Edge configs for SSR/middleware error capture
    - `next.config.ts` wrapped with `withSentryConfig`
    - `global-error.tsx` root error boundary

    **Replay settings:**
    - `replaysOnErrorSampleRate: 1.0` — always captures DOM replay on errors
    - `replaysSessionSampleRate: 0` — no recording of healthy sessions
    - `tunnelRoute: '/monitoring'` — proxies events to avoid ad-blockers
  </Tab>
</Tabs>

**Environment variables:**
- `SENTRY_DSN` — backend DSN
- `NEXT_PUBLIC_SENTRY_DSN` — frontend DSN
- `SENTRY_AUTH_TOKEN` — optional, for source map uploads

### Better Stack (Uptime + Error logs)

<CardGroup cols={2}>
  <Card title="Uptime monitoring" icon="heartbeat">
    Configured in Better Stack dashboard (no code) — pings `GET /health/ready` every 30–60 seconds
  </Card>
  <Card title="Logtail log shipping" icon="file-lines">
    Multi-transport: stdout at `info` level + Logtail at `warn` level (production only)
  </Card>
</CardGroup>

**Log shipping** (`@logtail/pino`):
- Stdout for Render log viewer (all info+ logs)
- Logtail for Better Stack (warn+ only to control volume/cost)
- Requires `BETTERSTACK_SOURCE_TOKEN` for production shipping

### Pino structured logging

Configured in `app.module.ts` via `nestjs-pino` (`LoggerModule`):

| Field | Source | When |
|-------|---------|------|
| `requestId` | CLS (`ClsService.getId()`) | Every request |
| `userId` | `LogContextInterceptor` | After auth guard |
| `organizationId` | `LogContextInterceptor` | After auth guard |

**Log level classification:**

<Steps>
  <Step title="Error level">
    `statusCode >= 500` or `err` present
  </Step>
  <Step title="Warn level">
    `statusCode >= 400`
  </Step>
  <Step title="Info level">
    All other requests
  </Step>
</Steps>

**Serializers and redaction:**
- `req` — `id`, `method`, `url` (sensitive headers redacted)
- `res` — `statusCode`
- `err` — `type`, `message`, `stack`

<Warning>
Authorization headers, cookies, password fields, and tokens are replaced with `[REDACTED]`
</Warning>

**Auto-logging exclusions:** `/health`, `/health/live`, `/health/ready`

### Observability role summary

| Tool | Role | What it captures |
|------|------|------------------|
| **Sentry** | Error tracking + performance + session replay | Stack traces, user context, DOM replay, correlated traces |
| **Better Stack Uptime** | "Is the app alive?" | Pings `/health/ready`, alerts on downtime |
| **Better Stack Logtail** | Error log archive | 4xx/5xx request logs with user context |
| **Render stdout** | Ad-hoc debugging | All pino JSON logs (info+) in Render log viewer |

<Check>
This layered approach provides comprehensive observability while keeping costs manageable through selective log shipping and sampling rates.
