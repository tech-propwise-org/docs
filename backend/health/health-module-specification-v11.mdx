The health module provides comprehensive application health monitoring through standardized health check endpoints. Built on NestJS Terminus, it implements a two-endpoint strategy distinguishing between liveness and readiness probes for container orchestration platforms like Kubernetes.

<Note>
The module monitors critical application dependencies including database connectivity, memory usage, and background job queue status through custom health indicators.
</Note>

## Architecture

### Design Decisions

The health module follows cloud-native best practices with several key architectural decisions:

- **Two-endpoint strategy**: Separate liveness (`/health/live`) and readiness (`/health/ready`) endpoints
- **Custom health indicators**: Extended Terminus framework with PgBoss job queue monitoring
- **Public endpoints**: Health checks bypass authentication for load balancer and orchestration access
- **Lightweight checks**: Focused on essential service availability without heavy database operations

## API Endpoints

### Liveness Probe

**Endpoint:** `GET /health/live`

<Info>
Liveness probe indicates if the process can handle requests. Used by container orchestration for restart decisions.
</Info>

```typescript
@Get('live')
@PublicEndpoint()
@HealthCheck()
liveness(): Promise<HealthCheckResult>
```

**Response:**
```json
{
  "status": "ok",
  "info": {},
  "error": {},
  "details": {}
}
```

**Use Cases:**
- Container orchestration restart decisions
- Load balancer routing decisions
- Basic process health verification

### Readiness Probe

**Endpoint:** `GET /health/ready`

<Warning>
Readiness probe indicates if the application can serve traffic. Returns 503 if any critical dependency is unavailable.
</Warning>

```typescript
@Get('ready')
@PublicEndpoint()
@HealthCheck()
readiness(): Promise<HealthCheckResult>
```

**Health Checks Performed:**
- Database connectivity (MikroORM)
- Memory usage monitoring
- PgBoss job queue status

**Response Format:**
```json
{
  "status": "ok|error",
  "info": {
    "database": { "status": "up" },
    "memory_heap": { "status": "up" },
    "pgboss": { "status": "up" }
  },
  "error": {},
  "details": {
    "database": { "status": "up" },
    "memory_heap": { "status": "up" },
    "pgboss": { "status": "up" }
  }
}
```

## Health Indicators

### PgBossHealthIndicator

Custom health indicator for the PgBoss background job service using the modern `HealthIndicatorService` API:

<CodeGroup>
```typescript PgBossHealthIndicator
@Injectable()
export class PgBossHealthIndicator {
  constructor(
    @Inject(QUEUE_SERVICE) private readonly queueService: IQueueService,
    private readonly healthIndicatorService: HealthIndicatorService,
  ) {}

  async isHealthy(key: string) {
    const indicator = this.healthIndicatorService.check(key);
    if (!this.queueService.isStarted) {
      return indicator.down({ reason: 'boss.start() has not completed' });
    }
    return indicator.up();
  }
}
```
</CodeGroup>

<Tip>
This implementation uses the NestJS Terminus v11+ `HealthIndicatorService.check(key)` pattern instead of the deprecated `extends HealthIndicator` approach.
</Tip>

**Implementation Notes:**
- Checks `IQueueService.isStarted` flag rather than database queries
- Lightweight check avoiding additional database load
- Returns healthy when PgBoss is properly started and connected

## Business Rules

### Health Check Behavior

<Steps>
  <Step title="Liveness checks">
    Always return 200 if the process can handle HTTP requests
  </Step>
  <Step title="Readiness checks">
    Return 503 if any critical dependency is unavailable
  </Step>
  <Step title="Public access">
    All health endpoints are publicly accessible (no authentication required)
  </Step>
  <Step title="Version neutral">
    Health checks use version-neutral routing
  </Step>
</Steps>

### Failure Conditions

<Warning>
The following conditions will cause readiness checks to fail:
</Warning>

- Database connection lost or unresponsive
- Memory usage exceeds configured thresholds
- PgBoss service not started or connection failed
- Any critical service dependency unavailable

### Endpoint Naming

| Endpoint | Purpose | Routing |
|----------|---------|---------|
| `/health/live` | Liveness probe | Version neutral |
| `/health/ready` | Readiness probe | Version neutral |

### Response Standards

<Check>
HTTP 200: All checks passing
</Check>

<Warning>
HTTP 503: One or more checks failing
</Warning>

### Error Handling

<CodeGroup>
```typescript Modern Approach (v11+)
// Use HealthIndicatorService.check() with .up()/.down()
const indicator = this.healthIndicatorService.check(key);
return indicator.down({ reason: 'Service not started' });
```

```typescript Deprecated Approach
// The legacy HealthCheckError + extends HealthIndicator pattern
// is deprecated and will be removed in the next major release
throw new HealthCheckError('Service failed', result);
```
</CodeGroup>

## Observability Stack

Beyond health check endpoints, the application uses a comprehensive layered observability architecture:

### Sentry (Error Tracking & Performance)

<Tabs>
  <Tab title="Backend">
    **Integration**: `@sentry/nestjs`
    
    - SDK initialized in `src/instrument.ts` (loaded before NestFactory)
    - `SentryModule.forRoot()` + `SentryGlobalFilter` captures all unhandled exceptions
    - `LogContextInterceptor` enriches events with user context
    - `tracesSampleRate: 0.2` in production
    - `nodeProfilingIntegration()` for performance profiling
    - Health transactions excluded via `ignoreTransactions`

    **Environment Variables:**
    - `SENTRY_DSN` — backend DSN
    - `SENTRY_AUTH_TOKEN` — optional, for source map uploads
  </Tab>
  
  <Tab title="Frontend">
    **Integration**: `@sentry/nextjs`
    
    - Client config with Session Replay and Breadcrumbs
    - Server and Edge configs for SSR/middleware error capture
    - `global-error.tsx` root error boundary
    - `AuthProvider` manages user context
    - `replaysOnErrorSampleRate: 1.0` — always captures DOM replay on errors
    - `tunnelRoute: '/monitoring'` — proxies events to avoid ad-blockers

    **Environment Variables:**
    - `NEXT_PUBLIC_SENTRY_DSN` — frontend DSN
  </Tab>
</Tabs>

### Pino Structured Logging

Configured in `app.module.ts` via `nestjs-pino` with contextual field enrichment:

| Field | Source | When |
|-------|--------|------|
| `requestId` | CLS (`ClsService.getId()`) | Every request |
| `userId` | `LogContextInterceptor` | After auth guard resolves |
| `organizationId` | `LogContextInterceptor` | After auth guard resolves |

<Note>
Log level classification: `error` (≥500 or exception), `warn` (≥400), `info` (other requests)
</Note>

**Security Features:**
- Authorization headers, cookies, passwords, and tokens are `[REDACTED]`
- Health endpoints (`/health*`) excluded from automatic request logging
