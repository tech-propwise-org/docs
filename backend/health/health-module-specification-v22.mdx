## Overview

The health module provides comprehensive application health monitoring through standardized health check endpoints. Built on NestJS Terminus, it implements a two-endpoint strategy distinguishing between liveness and readiness probes for container orchestration platforms like Kubernetes.

<Info>
The module monitors critical application dependencies including database connectivity, memory usage, and background job queue status through custom health indicators.
</Info>

The module is registered in `app.module.ts` and provides dedicated health check endpoints separate from the main application controller.

## Architecture

### Design Decisions

<Tabs>
  <Tab title="Two-Endpoint Strategy">
    Separate liveness (`/health/live`) and readiness (`/health/ready`) endpoints following cloud-native best practices

  <Tab title="Public Access">
    Health checks bypass authentication for load balancer and orchestration access
  </Tab>
  <Tab title="Rate Limiting">

## API Endpoints

### GET /health/live

**Purpose:** Liveness probe - indicates if the process can handle requests

```typescript
@Get('live')
@PublicEndpoint()
@HealthCheck()
liveness(): Promise<HealthCheckResult>
```

<CodeGroup>
```json Response
{
  "status": "ok",
  "info": {},
  "error": {},
  "details": {}
}
```
</CodeGroup>

<Note>
Use cases include container orchestration restart decisions, load balancer routing decisions, and basic process health verification.
</Note>

### GET /health/ready

**Purpose:** Readiness probe - indicates if the application can serve traffic

```typescript
@Get('ready')
@PublicEndpoint()
@HealthCheck()
readiness(): Promise<HealthCheckResult>
```

**Health Checks Performed:**
- Database connectivity (MikroORM)
- PgBoss job queue status

<CodeGroup>
```json Success Response
{
  "status": "ok",
  "info": {
    "database": { "status": "up" },
    "pgboss": { "status": "up" }
  },
  "error": {},
  "details": {
    "database": { "status": "up" },
    "pgboss": { "status": "up" }
  }
}
```

```json Error Response
{
  "status": "error",
  "info": {},

    "database": { "status": "down", "message": "Connection timeout" }

  "details": {
```

## Data Integrity Audit

<Warning>
The scheduled data-integrity audit is **not wired into `/health/ready`** and has no public HTTP endpoint for security reasons.
</Warning>

**Purpose:** Detect orphan rows resulting from soft-delete cascade gaps without exposing an unauthenticated, high-cost HTTP route.

The daily queued job runs at `30 3 * * *` by default (override via `DATA_INTEGRITY_AUDIT_CRON`) through `DataIntegrityAuditHandler`.

## Health Indicators

### PgBossHealthIndicator

Custom health indicator using the modern `HealthIndicatorService` API (NestJS Terminus v11+):

```typescript
@Injectable()
export class PgBossHealthIndicator {
  constructor(
    @Inject(QUEUE_SERVICE) private readonly queueService: IQueueService,
    private readonly healthIndicatorService: HealthIndicatorService,
  ) {}

  async isHealthy(key: string) {
    const indicator = this.healthIndicatorService.check(key);
    if (!this.queueService.isStarted) {
      return indicator.down({ reason: 'boss.start() has not completed' });
    }
    return indicator.up();
  }
}
```

<Check>
Uses the modern `.up()` / `.down()` pattern instead of the deprecated `extends HealthIndicator` base class.
</Check>

## Business Rules

### Health Check Behavior

1. **Liveness checks** always return 200 if the process can handle HTTP requests
2. **Readiness checks** return 503 if any critical dependency is unavailable
3. All health endpoints are publicly accessible (no authentication required)
4. Health endpoints are covered by the global `AppThrottlerGuard`
5. Health checks use version-neutral routing

### HTTP Status Codes

- **200**: All checks passing
- **503**: One or more checks failing
- **429**: Request exceeds the global throttling policy

### Failure Conditions

<Warning>
The following conditions will cause readiness checks to fail:
</Warning>

- Database connection lost or unresponsive
- Memory usage exceeds configured thresholds
- PgBoss service not started or connection failed
- Any critical service dependency unavailable

### Sentry Integration

<Tabs>
  <Tab title="Backend">
