## Overview

The health module provides comprehensive application health monitoring through standardized health check endpoints. Built on NestJS Terminus, it implements a two-endpoint strategy distinguishing between liveness and readiness probes for container orchestration platforms like Kubernetes.

<Info>
The module monitors critical application dependencies including database connectivity, memory usage, and background job queue status through custom health indicators.
</Info>

The module is registered in `app.module.ts` and provides dedicated health check endpoints separate from the main application controller.

## Health Indicators

### PgBossHealthIndicator

<Note>
This custom health indicator uses the `HealthIndicatorService` API (NestJS Terminus v11+) instead of the deprecated `HealthIndicator` base class.
</Note>

```typescript
@Injectable()
export class PgBossHealthIndicator {
  constructor(
    @Inject(QUEUE_SERVICE) private readonly queueService: IQueueService,
    private readonly healthIndicatorService: HealthIndicatorService,
  ) {}

  async isHealthy(key: string) {
    const indicator = this.healthIndicatorService.check(key);
    if (!this.queueService.isStarted) {
      return indicator.down({ reason: 'boss.start() has not completed' });
    }
    return indicator.up();
  }
}
```

<AccordionGroup>
  <Accordion title="Implementation Notes">
    - Uses `HealthIndicatorService.check(key)` with `.up()` / `.down()` instead of the legacy `extends HealthIndicator` + `HealthCheckError` pattern
    - Checks `IQueueService.isStarted` flag rather than database queries
    - Lightweight check avoiding additional database load
    - Returns healthy when PgBoss is properly started and connected
  </Accordion>
</AccordionGroup>

## Business Rules

### Health Check Behavior

<Check>Liveness checks always return 200 if the process can handle HTTP requests</Check>
<Warning>Readiness checks return 503 if any critical dependency is unavailable</Warning>
<Info>All health endpoints are publicly accessible (no authentication required)</Info>
<Tip>Health checks use version-neutral routing</Tip>

### Failure Conditions

- Database connection lost or unresponsive
- Memory usage exceeds configured thresholds
- PgBoss service not started or connection failed
- Any critical service dependency unavailable

## Patterns and Conventions

### Endpoint Naming

- `/health/live` - Liveness probe
- `/health/ready` - Readiness probe
- Version neutral routing (`VERSION_NEUTRAL`)

### Response Standards

- **HTTP 200**: All checks passing
- **HTTP 503**: One or more checks failing
- Consistent JSON response format following Terminus conventions

### Error Handling

<Warning>
The legacy `HealthCheckError` + `extends HealthIndicator` pattern is deprecated and will be removed in the next major Terminus release.
</Warning>

```typescript
// Modern approach (NestJS Terminus v11+)
const indicator = this.healthIndicatorService.check(key);
return indicator.down({ reason: 'Service not started' });
```

## Observability Stack

Beyond health check endpoints, the application uses a layered observability architecture:

### Sentry (Error Tracking & Performance)

<Tabs>
  <Tab title="Backend">
    **Setup** (`@sentry/nestjs`):
    - SDK initialized in `src/instrument.ts` (loaded before NestFactory)
    - `SentryModule.forRoot()` + `SentryGlobalFilter` as `APP_FILTER` captures all unhandled exceptions
    - `LogContextInterceptor` enriches Sentry events with user context
    - Health transactions excluded via `ignoreTransactions`

    **Configuration:**
    ```typescript
    {
      tracesSampleRate: 0.2, // 20% of transactions in production
      nodeProfilingIntegration(), // Performance profiling
      // Error events captured at 100%
    }
    ```
  </Tab>
  <Tab title="Frontend">
    **Setup** (`@sentry/nextjs`):
    - Client config with Session Replay and Breadcrumbs
    - Server and Edge configs for SSR/middleware error capture
    - `global-error.tsx` root error boundary
    - `tunnelRoute: '/monitoring'` to avoid ad-blockers

    **Session Replay:**
    ```typescript
    {
      replaysOnErrorSampleRate: 1.0, // Always capture replay on error
      replaysSessionSampleRate: 0     // No recording of healthy sessions
    }
    ```
  </Tab>
</Tabs>

**Environment Variables:**
- `SENTRY_DSN` — Backend DSN
- `NEXT_PUBLIC_SENTRY_DSN` — Frontend DSN
- `SENTRY_AUTH_TOKEN` — Optional, for source map uploads

### Better Stack (Uptime + Error Logs)

<CardGroup cols={2}>
  <Card title="Uptime Monitoring" icon="clock">
    Configured in Better Stack dashboard - pings `GET /health/ready` every 30-60 seconds
  </Card>
  <Card title="Logtail Log Shipping" icon="file-text">
    Multi-transport setup with `@logtail/pino` - stdout at `info` + Logtail at `warn` level
  </Card>

**Log Transport Strategy:**
- **Production with token**: stdout (`info` level) + Logtail (`warn` level)
- **Without token**: stdout only (JSON format)
- **Benefit**: Low cost, error searchability preserved

**Environment Variables:**
- `BETTERSTACK_SOURCE_TOKEN` — Logtail source token (production only)

### Pino Structured Logging

<Steps>
  <Step title="Request Context">
    `requestId` from CLS (`ClsService.getId()`) on every request
  </Step>
  <Step title="User Context">
    `userId` and `organizationId` added after auth guard via `LogContextInterceptor`
  </Step>
  <Step title="Log Level Classification">
    - `error`: statusCode >= 500 or exception present
    - `warn`: statusCode >= 400
    - `info`: all other requests
  </Step>
</Steps>

<AccordionGroup>
  <Accordion title="Log Field Details">
    | Field | Source | When |
    |---|---|---|
    | `requestId` | CLS (`ClsService.getId()`) | Every request (middleware layer) |
    | `userId` | `LogContextInterceptor` (`logger.assign()`) | After auth guard resolves `request.user` |
    | `organizationId` | `LogContextInterceptor` | After auth guard resolves `request.user` |
  </Accordion>
  <Accordion title="Serializers & Redaction">
    **Serializers:**
    - `req` — `id`, `method`, `url` (minimal, sensitive headers redacted)
    - `res` — `statusCode`
    - `err` — `type`, `message`, `stack`

    **Redaction:** Authorization header, cookies, password fields, and tokens are replaced with `[REDACTED]`.

    **Exclusions:** `/health`, `/health/live`, `/health/ready` excluded from automatic request logging.
  </Accordion>
</AccordionGroup>
