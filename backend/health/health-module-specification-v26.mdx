## Overview

The health module provides comprehensive application health monitoring through standardized health check endpoints. Built on NestJS Terminus, it implements a two-endpoint strategy distinguishing between liveness and readiness probes for container orchestration platforms like Kubernetes.

<Note>
The module monitors critical application dependencies including database connectivity, memory usage, and background job queue status through custom health indicators.
</Note>

The module is registered in `app.module.ts` and provides dedicated health check endpoints separate from the main application controller.

## API Endpoints

### GET /health/live

**Purpose:** Liveness probe - indicates if the process can handle requests

```typescript
@Get('live')
@PublicEndpoint()
@HealthCheck()
liveness(): Promise<HealthCheckResult>
```

**Response:**

```json
{
  "status": "ok",
  "info": {},
  "error": {},
  "details": {}
}
```

**Use Cases:**
- Container orchestration restart decisions
- Load balancer routing decisions
- Basic process health verification

### GET /health/ready

**Purpose:** Readiness probe - indicates if the application can serve traffic

```typescript
@Get('ready')
@PublicEndpoint()
@HealthCheck()
readiness(): Promise<HealthCheckResult>
```

**Health Checks Performed:**
- Database connectivity (MikroORM)
- PgBoss job queue status
- Search provider (`SearchProviderHealthIndicator` — Phase 1A)

**Response Format:**

```json
{
  "status": "ok|error",
  "info": {
    "database": { "status": "up" },
    "pgboss": { "status": "up" },
    "search": { "status": "up" }
  },
  "error": {},
  "details": {
    "database": { "status": "up" },
    "pgboss": { "status": "up" },
    "search": { "status": "up" }
  }
}
```

## Scheduled Data-Integrity Audit

<Warning>
The audit is **not wired into `/health/ready`** and has no public `/health/data-integrity` endpoint.
</Warning>

**Purpose:** Detect orphan rows resulting from soft-delete cascade gaps without exposing an unauthenticated, high-cost HTTP route.

The daily queued job runs at `30 3 * * *` by default (override via `DATA_INTEGRITY_AUDIT_CRON`) through `DataIntegrityAuditHandler`. It logs every run, emits a warn-level application log when unhealthy orphans are found, and sends a grouped Sentry error event when `totalOrphans > 0`.

### Why Separate from Readiness Probe

Orphan rows do not affect whether the instance can serve traffic — they are a data-integrity concern, not a service-availability concern. Wiring this into the readiness probe would block rollouts unrelated to the symptom.

## Health Indicators

### PgBossHealthIndicator

<Tip>
Uses `HealthIndicatorService.check(key)` with `.up()` / `.down()` instead of the legacy `extends HealthIndicator` pattern.
</Tip>

Custom health indicator for the PgBoss background job service:

```typescript
@Injectable()
export class PgBossHealthIndicator {
  constructor(
    @Inject(QUEUE_SERVICE) private readonly queueService: IQueueService,
    private readonly healthIndicatorService: HealthIndicatorService,
  ) {}

  async isHealthy(key: string) {
    const indicator = this.healthIndicatorService.check(key);
    const result = await this.queueService.checkHealth();
    if (!result.healthy) {
      return indicator.down({ reason: result.reason ?? 'queue backend unhealthy' });
    }
    return indicator.up({ latencyMs: result.latencyMs });
  }
}
```

**Why not just check `isStarted`?**

The previous implementation reported healthy whenever `isStarted === true`. That flag is set once after `boss.start()` resolves and only flips back on `stop()` — so if pg-boss loses its connection or its supervisor loop stalls after startup, the flag stays `true` and a **dead queue keeps reporting healthy**.

`checkHealth()` closes this gap by round-tripping a trivial `SELECT 1` through pg-boss's own connection pool (bounded by `QUEUE_HEALTH_PROBE_TIMEOUT_MS`, default 2000ms).

**Down states cover:**

| Condition | Reason returned |
|-----------|----------------|
| `boss.start()` not yet complete | `boss.start() has not completed` |
| Probe query rejects (dropped connection) | the underlying driver error message |
| Probe exceeds the timeout (stalled/hung queue) | `queue health probe timed out after <n>ms` |

<Check>
**Pool coverage:** the probe uses pg-boss's connection (the superuser `DATABASE_URL` pool), which is distinct from the ORM connection (`RUNTIME_DATABASE_URL`) that the readiness `database` check pings.
</Check>

### SearchProviderHealthIndicator

Custom readiness indicator for the active `SearchProvider` implementation (`PostgresSearchProvider` in Phase 1). Mirrors the `PgBossHealthIndicator` pattern: a thin Terminus wrapper in `src/modules/health/` that delegates to the provider interface.

**Contract (`SearchProvider.healthCheck()`):**

<Tabs>
<Tab title="Phase 1B Steps 1–2 (current)">
Probes `information_schema` for `search_document` table existence via `executeReadOnlyWithBypass`. Returns `up` when the table is reachable; read path (`search()`) and write paths (`upsert` / `delete` + ACL diff) are live.

<Note>
`reindexOrg` still throws `SearchProviderNotImplementedError` in this phase.
</Note>
</Tab>

<Tab title="Phase 1B+ (target)">
Optional row-count sanity, pg-boss `search.reindex` queue depth threshold, and report `down` when indexer backlog exceeds SLO.
</Tab>
</Tabs>
