## Overview

The health module provides comprehensive application health monitoring through standardized health check endpoints. Built on NestJS Terminus, it implements a two-endpoint strategy distinguishing between liveness and readiness probes for container orchestration platforms like Kubernetes.

<Note>
The module monitors critical application dependencies including database connectivity, memory usage, and background job queue status through custom health indicators. The module is registered in `app.module.ts` and provides dedicated health check endpoints separate from the main application controller.
</Note>

## Architecture

### Design Decisions

The health module follows cloud-native best practices with several key architectural decisions:

- **Two-endpoint strategy**: Separate liveness (`/health/live`) and readiness (`/health/ready`) endpoints following cloud-native best practices
- **Custom health indicators**: Extended Terminus framework with PgBoss job queue monitoring  
- **Public endpoints**: Health checks bypass authentication for load balancer and orchestration access
- **Rate limited public surface**: Health checks use the app-wide throttler to reduce abuse risk while remaining reachable by probes
- **Lightweight checks**: Focused on essential service availability without heavy database operations
- **Data integrity monitoring**: Tracks informational counts for user org membership drift after removal

### Data Integrity Audit Enhancements

The `DataIntegrityAuditService` includes comprehensive monitoring capabilities:

<Tip>
**Informational count tracking**: The service now includes `INFORMATIONAL_COUNT_FIELDS` alongside orphan and creator reference counts for comprehensive data integrity monitoring.
</Tip>

Key tracking metrics include:

- `stakeholdersWithoutActiveUserOrgRoleCount` - tracks stakeholder rows where the user no longer has an active UserOrgRole (expected after org removal, see ORG_USER_REMOVAL_SPECIFICATION.md Phase 4.3)
- `totalInformationalCounts` - sum of membership drift metrics (not unhealthy orphans)
- Classification requirement: new audit counts must be explicitly categorized as orphan, creator reference, or informational
- The service includes these counts in the daily audit alongside existing orphan and creator reference tracking

### Scheduled Data Integrity Audit

<Warning>
The data integrity audit is **not wired into `/health/ready`** and has no public `/health/data-integrity` endpoint. This is intentional to prevent blocking service readiness due to data integrity issues.
</Warning>

**Purpose**: Detect orphan rows resulting from soft-delete cascade gaps without exposing an unauthenticated, high-cost HTTP route.

The daily queued job runs at `30 3 * * *` by default (override via `DATA_INTEGRITY_AUDIT_CRON`) through `DataIntegrityAuditHandler`. It logs every run, emits a warn-level application log when unhealthy orphans are found, and sends a grouped Sentry error event when `totalOrphans > 0`.

<Steps>
  <Step title="Core Entity Audits (12 checks)">
    Lead/Deal/Contact/Company/Listing with deleted persons/orgs/stages/creators (the original 12 fixed relations from `Docs/SOFT_DELETE_FILTER_STANDARD.md` §6)
  </Step>
  
  <Step title="Person & Company Sub-entities (10 checks)">
    Person.organization, Person.createdBy, Person{Email,Phone,Address,Channel}.person, PersonNotDuplicate (either side), Company{Email,Phone,Address}.company
  </Step>
  
  <Step title="Activity Stack + Reminder (10 checks)">
    Activity, Note, CallLog, EmailLog, SmsLog, Meeting, SiteVisit organization refs; Activity.performedBy and Note.createdBy creator refs; Reminder.user
  </Step>
  
  <Step title="Tasks + Events + Tag (10 checks)">
    Tag.organization; Task.organization, Task.createdBy; TaskComment.task; TaskAssignee.task; TaskTag.tag; Event.organization, Event.createdBy; EventInvitee.event; EventTag.event
  </Step>
  
  <Step title="Additional Entity Audits">
    Stakeholder + Transfer + Stage histories, Junctions + Commission + DealDocument, Auth sessions, RBAC junctions, Distribution, Escalation, and Notifications
  </Step>
</Steps>

<Info>
**Why separate from `/health/ready`**: Orphan rows do not affect whether the instance can serve traffic — they are a data-integrity concern, not a service-availability concern. Wiring this into the readiness probe would block rollouts unrelated to the symptom.
</Info>

**Companion Infrastructure**:

- **Sentry alert event**: Emitted at `error` level with tags `subsystem=health`, `check=data-integrity`, `queue=data-integrity-audit`
- **One-off repair script**: `pnpm repair:orphaned-lead-persons --dry-run` then without the flag
- See `Docs/QUERY_OPTIMIZATION_PATTERNS.md` §5.5 for root-cause mechanics and `Docs/CRM_MODULE_SPECIFICATION.md` §2.1.3 for mergePersons contract

## Health Indicators

### PgBossHealthIndicator

Custom health indicator for the PgBoss background job service using the modern `HealthIndicatorService` API.

<CodeGroup>

```typescript Implementation
@Injectable()
export class PgBossHealthIndicator {
  constructor(
    @Inject(QUEUE_SERVICE) private readonly queueService: IQueueService,
    private readonly healthIndicatorService: HealthIndicatorService,
  ) {}

  async isHealthy(key: string) {
    const indicator = this.healthIndicatorService.check(key);
    const result = await this.queueService.checkHealth();
    if (!result.healthy) {
      return indicator.down({ reason: result.reason ?? 'queue backend unhealthy' });
    }
    return indicator.up({ latencyMs: result.latencyMs });
  }
}
```

</CodeGroup>

<Note>
Uses `HealthIndicatorService.check(key)` with `.up()` / `.down()` instead of the legacy `extends HealthIndicator` + `HealthCheckError` pattern.
</Note>

**Active Health Probing**: Rather than checking the `isStarted` boolean, this implementation uses `IQueueService.checkHealth()` for active liveness probes. This prevents scenarios where a dead queue continues reporting healthy after losing connection.

<AccordionGroup>
  <Accordion title="Health Check Conditions">
    
| Condition | Reason Returned |
|-----------|----------------|
| `boss.start()` not yet complete | `boss.start() has not completed` |
| Probe query rejects (dropped connection) | Underlying driver error message |
| Probe exceeds timeout (stalled/hung queue) | `queue health probe timed out after <n>ms` |

  </Accordion>
</AccordionGroup>

**Pool Coverage**: The probe uses pg-boss's connection (superuser `DATABASE_URL` pool), distinct from the ORM connection (`RUNTIME_DATABASE_URL`) that the readiness `database` check pings.

### SearchProviderHealthIndicator

Custom readiness indicator for the active `SearchProvider` implementation (`PostgresSearchProvider` in Phase 1).

<Tabs>
  <Tab title="Phase 1B Steps 1-2 (Current)">
    Probes `information_schema` for `search_document` table existence via `executeReadOnlyWithBypass`. Returns `up` when the table is reachable; read path (`search()`) and write paths (`upsert` / `delete` + ACL diff) are live. 
    
    <Warning>
    `reindexOrg` still throws `SearchProviderNotImplementedError` in this phase.
    </Warning>
  </Tab>
  
  <Tab title="Phase 1B+ (Target)">
    Optional row-count sanity, pg-boss `search.reindex` queue depth threshold, and report `down` when indexer backlog exceeds SLO.
  </Tab>
</Tabs>

**Contract**: Follows the `SearchProvider.healthCheck()` interface as defined in `Docs/SEARCH_MODULE_SPECIFICATION.md` §11.1, providing a thin Terminus wrapper that delegates to the provider interface.

<Check>
This design ensures `HealthModule` does not import `SearchModule` internals beyond the `SEARCH_PROVIDER` token, maintaining proper module boundaries.
</Check>
