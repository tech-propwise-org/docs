## Overview

The health module provides comprehensive application health monitoring through standardized health check endpoints. Built on NestJS Terminus, it implements a two-endpoint strategy distinguishing between liveness and readiness probes for container orchestration platforms like Kubernetes.

The module monitors critical application dependencies including database connectivity, memory usage, and background job queue status through custom health indicators. The module is registered in `app.module.ts` and provides dedicated health check endpoints separate from the main application controller.

## Architecture

### Design decisions

The health module follows cloud-native best practices with several key architectural decisions:

<CardGroup cols={2}>
  <Card title="Two-endpoint strategy" icon="route">
    Separate liveness (`/health/live`) and readiness (`/health/ready`) endpoints following cloud-native best practices
  </Card>
  <Card title="Custom health indicators" icon="gauge">
    Extended Terminus framework with PgBoss job queue monitoring
  </Card>
  <Card title="Public endpoints" icon="unlock">
    Health checks bypass authentication for load balancer and orchestration access
  </Card>
  <Card title="Lightweight checks" icon="feather">

  </Card>

<Note>
Health checks use the app-wide throttler to reduce abuse risk while remaining reachable by probes.
</Note>

## API Endpoints

### GET /health/live

**Purpose:** Liveness probe - indicates if the process can handle requests

```typescript
@Get('live')
@PublicEndpoint()
@HealthCheck()
liveness(): Promise<HealthCheckResult>
```

**Response:**
```json
{
  "status": "ok",
  "info": {},
  "error": {},
  "details": {}
}
```

**Use cases:**
- Container orchestration restart decisions
- Load balancer routing decisions
- Basic process health verification

### GET /health/ready

**Purpose:** Readiness probe - indicates if the application can serve traffic

```typescript
@Get('ready')
@PublicEndpoint()
@HealthCheck()
readiness(): Promise<HealthCheckResult>
```

**Health checks performed:**
- Database connectivity (MikroORM)
- PgBoss job queue status

**Response format:**
```json
{
  "status": "ok|error",
  "info": {
    "database": { "status": "up" },
    "pgboss": { "status": "up" }
  },
  "error": {},
  "details": {
    "database": { "status": "up" },
    "pgboss": { "status": "up" }
  }
}
```

## Data Integrity Audit

<Warning>
The data integrity audit is **not wired into `/health/ready`** and has no public HTTP endpoint. It runs as a scheduled background job.
</Warning>

### Purpose

Detect orphan rows resulting from soft-delete cascade gaps without exposing an unauthenticated, high-cost HTTP route. The audit runs daily at `30 3 * * *` by default (override via `DATA_INTEGRITY_AUDIT_CRON`) through `DataIntegrityAuditHandler`.

### Audit behavior

<Steps>
  <Step title="Schedule execution">
    The daily queued job runs at the configured cron schedule
  </Step>
  <Step title="Run audit checks">
    Executes ~74 count queries grouped into module buckets
  </Step>
  <Step title="Log results">
    Logs every run and emits warn-level application logs for unhealthy orphans
  </Step>
  <Step title="Alert on issues">
    Sends grouped Sentry error events when `totalOrphans > 0`
  </Step>
</Steps>

### Audit coverage

<AccordionGroup>
  <Accordion title="Core entities (12 checks)">
    Lead/Deal/Contact/Company/Listing with deleted persons/organizations/stages/creators from the original 12 fixed relations in `Docs/SOFT_DELETE_FILTER_STANDARD.md` §6
  </Accordion>

  <Accordion title="Person & Company sub-entities (10 checks)">
    - Person.organization, Person.createdBy
    - Person{Email,Phone,Address,Channel}.person
    - PersonNotDuplicate (either side)
    - Company{Email,Phone,Address}.company
  </Accordion>

  <Accordion title="Activity stack + Reminder (10 checks)">
    - Activity, Note, CallLog, EmailLog, SmsLog, Meeting, SiteVisit organization refs
    - Activity.performedBy and Note.createdBy creator refs
    - Reminder.user
  </Accordion>

  <Accordion title="Tasks + Events + Tag (10 checks)">
    - Tag.organization
    - Task.organization, Task.createdBy
    - TaskComment.task, TaskAssignee.task, TaskTag.tag
    - Event.organization, Event.createdBy
    - EventInvitee.event, EventTag.event
  </Accordion>

  <Accordion title="Stakeholder + Transfer + Stage histories (7 checks)">
    - EntityStakeholder.organization, EntityStakeholderHistory.organization
    - EntityTransfer.organization
    - LeadStageHistory.{lead,stage}, DealStageHistory.{deal,stage}
  </Accordion>

  <Accordion title="Additional modules">
    - **Junctions + Commission + DealDocument**: DealContact, ContactCompanyRole, CommissionPayment entities
    - **Auth sessions (3)**: AuthSession.user, OrgSession.organization, UserDevice.user
    - **RBAC junctions (6)**: Team, TeamMembership, UserOrgRole, Invitation entities
    - **Distribution (3)**: DistributionRule, DistributionLog, TeamDistributionSettings
    - **Escalation (3)**: EscalationRule, EscalationTracker entities

  </Accordion>
</AccordionGroup>

<Info>
Soft-deleted creator refs are informational only and do **not** contribute to `totalOrphans`. User churn is normal, and historical records should still display the deleted creator when needed.
</Info>

### Why separate from readiness probe

Orphan rows do not affect whether the instance can serve traffic — they are a data-integrity concern, not a service-availability concern. Wiring this into the readiness probe would block rollouts unrelated to the symptom.

### Companion infrastructure

- **Sentry alert event**: Emitted at `error` level with tags `subsystem=health`, `check=data-integrity`, `queue=data-integrity-audit`
- **One-off repair script**: `pnpm repair:orphaned-lead-persons --dry-run` then without the flag
- See `Docs/QUERY_OPTIMIZATION_PATTERNS.md` §5.5 for root-cause analysis

## Health Indicators

### PgBossHealthIndicator

Custom health indicator for the PgBoss background job service. Uses the `HealthIndicatorService` API (NestJS Terminus v11+) instead of the deprecated `HealthIndicator` base class.

```typescript
@Injectable()
export class PgBossHealthIndicator {
  constructor(
    @Inject(QUEUE_SERVICE) private readonly queueService: IQueueService,
    private readonly healthIndicatorService: HealthIndicatorService,
  ) {}

  async isHealthy(key: string) {
    const indicator = this.healthIndicatorService.check(key);
    if (!this.queueService.isStarted) {
      return indicator.down({ reason: 'boss.start() has not completed' });
    }
    return indicator.up();
  }
}
```

**Implementation notes:**
- Uses `HealthIndicatorService.check(key)` with `.up()` / `.down()` instead of the legacy pattern
- Checks `IQueueService.isStarted` flag rather than database queries
- Lightweight check avoiding additional database load
- Returns healthy when PgBoss is properly started and connected

## Business Rules

### Health check behavior

<Check>
**Liveness checks** always return 200 if the process can handle HTTP requests
</Check>

<Check>
**Readiness checks** return 503 if any critical dependency is unavailable
</Check>

<Check>
All health endpoints are publicly accessible (no authentication required)
</Check>

<Check>
Health endpoints are covered by the global `AppThrottlerGuard`
</Check>

### Failure conditions

- Database connection lost or unresponsive
- Memory usage exceeds configured thresholds
- PgBoss service not started or connection failed
- Any critical service dependency unavailable

### Endpoint naming

### Response standards

<Tabs>
  <Tab title="Success">
    **HTTP 200**: All checks passing
  </Tab>
  <Tab title="Service Error">
    **HTTP 503**: One or more checks failing
  </Tab>
  <Tab title="Rate Limited">
    **HTTP 429**: Request exceeds the global throttling policy
  </Tab>
</Tabs>

Consistent JSON response format following Terminus conventions.

### Error handling

```typescript
// Health check failures use indicator.down() (NestJS Terminus v11+)
const indicator = this.healthIndicatorService.check(key);
return indicator.down({ reason: 'Service not started' });

// The legacy HealthCheckError + extends HealthIndicator pattern is deprecated
// and will be removed in the next major Terminus release.
```

## Observability Stack

Beyond health check endpoints, the application uses a layered observability architecture:

### Sentry (Error tracking & Performance)

<Tabs>
  <Tab title="Backend">
    **Backend** (`@sentry/nestjs`):
    - SDK initialized in `src/instrument.ts` (loaded before NestFactory)
    - `SentryModule.forRoot()` + `SentryGlobalFilter` as `APP_FILTER` captures all unhandled exceptions
    - `LogContextInterceptor` enriches Sentry events with user and organization context
    - `DataIntegrityAuditHandler` explicitly emits grouped error-level events for orphan rows
    - `tracesSampleRate: 0.2` in production (all error events captured at 100%)
    - `nodeProfilingIntegration()` for performance profiling
    - Health transactions (`GET /health*`) are excluded via `ignoreTransactions`
  </Tab>
  <Tab title="Frontend">
    **Frontend** (`@sentry/nextjs`):
    - Client config with Session Replay and Breadcrumbs
    - Server and Edge configs for SSR/middleware error capture
    - `next.config.ts` wrapped with `withSentryConfig` for source map uploads
    - `global-error.tsx` root error boundary reports to Sentry
    - `AuthProvider` sets user context and organization tags
    - `replaysOnErrorSampleRate: 1.0` — always captures DOM replay on errors
    - `tunnelRoute: '/monitoring'` — proxies events to avoid ad-blockers
  </Tab>
</Tabs>

**Environment variables:**
- `SENTRY_DSN` — backend DSN
- `NEXT_PUBLIC_SENTRY_DSN` — frontend DSN
- `SENTRY_AUTH_TOKEN` — optional, for source map uploads during build

<Tip>
Frontend and backend can use the same Sentry project for correlated traces across the full stack.
</Tip>
