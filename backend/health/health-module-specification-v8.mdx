## Overview

The health module provides comprehensive application health monitoring through standardized health check endpoints. Built on NestJS Terminus, it implements a two-endpoint strategy distinguishing between liveness and readiness probes for container orchestration platforms like Kubernetes.

<Info>
The module monitors critical application dependencies including database connectivity, memory usage, and background job queue status through custom health indicators.
</Info>

The module is registered in `app.module.ts` and provides dedicated health check endpoints separate from the main application controller.

## Architecture

## Health Indicators

### PgBossHealthIndicator

Custom health indicator for the PgBoss background job service. Uses the modern `HealthIndicatorService` API (NestJS Terminus v11+) instead of the deprecated `HealthIndicator` base class.

```typescript
@Injectable()
export class PgBossHealthIndicator {
  constructor(
    @Inject(QUEUE_SERVICE) private readonly queueService: IQueueService,
    private readonly healthIndicatorService: HealthIndicatorService,
  ) {}

  async isHealthy(key: string) {
    const indicator = this.healthIndicatorService.check(key);
    if (!this.queueService.isStarted) {
      return indicator.down({ reason: 'boss.start() has not completed' });
    }
    return indicator.up();
  }
}
```

<Note>
The implementation uses `HealthIndicatorService.check(key)` with `.up()` / `.down()` instead of the legacy `extends HealthIndicator` + `HealthCheckError` pattern, which is deprecated and will be removed in the next major Terminus release.
</Note>

**Implementation Details:**
- Checks `IQueueService.isStarted` flag rather than database queries
- Lightweight check avoiding additional database load
- Returns healthy when PgBoss is properly started and connected

## Business Rules

### Health Check Behavior

<Steps>
  <Step title="Liveness Check Logic">
    Always returns 200 if the process can handle HTTP requests - this is a basic process health verification
  </Step>
  <Step title="Readiness Check Logic">
    Returns 503 if any critical dependency is unavailable, preventing traffic routing to unhealthy instances
  </Step>
  <Step title="Authentication Bypass">
    All health endpoints are publicly accessible with no authentication required for load balancer access
  </Step>
  <Step title="Version Neutral Routing">
    Health checks use version-neutral routing to remain consistent across API versions
  </Step>
</Steps>

### Failure Conditions

<Warning>
The following conditions will cause readiness checks to fail:
</Warning>

- Database connection lost or unresponsive
- Memory usage exceeds configured thresholds
- PgBoss service not started or connection failed
- Any critical service dependency unavailable

## Patterns and Conventions

### Endpoint Naming

- `/health/live` - Liveness probe
- `/health/ready` - Readiness probe  
- Version neutral routing (`VERSION_NEUTRAL`)

### Response Standards

- **HTTP 200**: All checks passing
- **HTTP 503**: One or more checks failing
- Consistent JSON response format following Terminus conventions

### Error Handling

```typescript
// Modern approach (NestJS Terminus v11+)
const indicator = this.healthIndicatorService.check(key);
return indicator.down({ reason: 'Service not started' });

// Legacy pattern (deprecated)
// throw new HealthCheckError('Service failed', result);
```

<Tip>
Always use the modern `HealthIndicatorService.check()` pattern for new health indicators to ensure compatibility with future Terminus versions.
</Tip>

## Observability Stack

Beyond health check endpoints, the application uses a layered observability architecture for comprehensive monitoring and debugging.

### Sentry (Error Tracking & Performance)

<Tabs>
  <Tab title="Backend Configuration">
    **Setup** (`@sentry/nestjs`):
    - SDK initialized in `src/instrument.ts` (loaded before NestFactory)
    - `SentryModule.forRoot()` + `SentryGlobalFilter` as `APP_FILTER`
    - `LogContextInterceptor` enriches events with user context
    
    **Configuration**:
    - `tracesSampleRate: 0.2` in production
    - `nodeProfilingIntegration()` for performance profiling
    - Health transactions excluded via `ignoreTransactions`
  </Tab>
  <Tab title="Frontend Configuration">
    **Setup** (`@sentry/nextjs`):
    - Client config with Session Replay and Breadcrumbs
    - Server and Edge configs for SSR/middleware error capture
    - `next.config.ts` wrapped with `withSentryConfig`
    
    **Configuration**:
    - `replaysOnErrorSampleRate: 1.0` — always capture DOM replay on errors
    - `replaysSessionSampleRate: 0` — no healthy session recording
    - `tunnelRoute: '/monitoring'` — proxy to avoid ad-blockers
  </Tab>
</Tabs>

**Environment Variables:**
- `SENTRY_DSN` — Backend DSN
- `NEXT_PUBLIC_SENTRY_DSN` — Frontend DSN
- `SENTRY_AUTH_TOKEN` — Optional, for source map uploads

### Better Stack (Uptime + Error Logs)

<CardGroup cols={2}>
  <Card title="Uptime Monitoring" icon="clock">
    Dashboard-configured monitoring pings `/health/ready` every 30-60 seconds
  </Card>
  <Card title="Logtail Log Shipping" icon="file-lines">
    Multi-transport logging with selective error log shipping to reduce costs
  </Card>
</CardGroup>

**Logtail Configuration** (`@logtail/pino`):
- **Production**: stdout (info level) + Logtail (warn level)
- **Development**: stdout only (JSON format)
- Only 4xx/5xx errors reach Better Stack for cost optimization

**Environment Variables:**
- `BETTERSTACK_SOURCE_TOKEN` — Logtail source token (production only)

### Pino Structured Logging

Configured in `app.module.ts` via `nestjs-pino` with contextual enrichment:

| Field | Source | When Applied |
|-------|--------|--------------|
| `requestId` | CLS (`ClsService.getId()`) | Every request (middleware layer) |
| `userId` | `LogContextInterceptor` | After auth guard resolves |
| `organizationId` | `LogContextInterceptor` | After auth guard resolves |

**Log Level Classification**:
- `error` — `statusCode >= 500` or exception present
- `warn` — `statusCode >= 400`  
- `info` — all other requests

<Note>
Health endpoints (`/health`, `/health/live`, `/health/ready`) are excluded from automatic request logging to reduce noise.
</Note>

### Observability Summary

| Tool | Primary Role | Captures |
|------|--------------|----------|
| **Sentry** | Error tracking + performance | Stack traces, user context, DOM replay, distributed traces |
| **Better Stack Uptime** | Service availability | `/health/ready` endpoint monitoring with alerting |
| **Better Stack Logtail** | Error log archive | 4xx/5xx logs with user/org context and request correlation |
| **Render stdout** | Development debugging | All pino JSON logs visible in Render log viewer |

This multi-layered approach ensures comprehensive observability while maintaining cost efficiency through selective data collection and retention policies.
