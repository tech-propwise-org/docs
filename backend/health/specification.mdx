## Overview

The health module provides comprehensive application health monitoring through standardized health check endpoints. Built on NestJS Terminus, it implements a two-endpoint strategy distinguishing between liveness and readiness probes for container orchestration platforms like Kubernetes.

<Note>
The module monitors critical application dependencies including database connectivity, memory usage, and background job queue status through custom health indicators. The module is registered in `app.module.ts` and provides dedicated health check endpoints separate from the main application controller.
</Note>

**Last Updated:** 2024-12-19  
**Status:** Active

## Architecture

### Design Decisions

The health module follows cloud-native best practices with these key architectural decisions:

- **Two-endpoint strategy**: Separate liveness (`/health/live`) and readiness (`/health/ready`) endpoints following cloud-native best practices
- **Custom health indicators**: Extended Terminus framework with PgBoss job queue monitoring
- **Public endpoints**: Health checks bypass authentication for load balancer and orchestration access
- **Lightweight checks**: Focused on essential service availability without heavy database operations

## API Endpoints

### Liveness Probe

<CodeGroup>
```typescript Endpoint Definition
@Get('live')
@PublicEndpoint()
@HealthCheck()
liveness(): Promise<HealthCheckResult>
```

```http HTTP Request
GET /health/live
```

```json Response
{
  "status": "ok",
  "info": {},
  "error": {},
  "details": {}
}
```
</CodeGroup>

<Info>
**Purpose:** Liveness probe indicates if the process can handle requests and is used for container orchestration restart decisions and load balancer routing.
</Info>

**Use Cases:**
- Container orchestration restart decisions
- Load balancer routing decisions
- Basic process health verification

### Readiness Probe

<CodeGroup>
```typescript Endpoint Definition
@Get('ready')
@PublicEndpoint()
@HealthCheck()
readiness(): Promise<HealthCheckResult>
```

```http HTTP Request
GET /health/ready
```

```json Success Response
{
  "status": "ok",
  "info": {
    "database": { "status": "up" },
    "memory_heap": { "status": "up" },
    "pgboss": { "status": "up" }
  },
  "error": {},
  "details": {
    "database": { "status": "up" },
    "memory_heap": { "status": "up" },
    "pgboss": { "status": "up" }
  }
}
```

```json Error Response
{
  "status": "error",
  "info": {},
  "error": {
    "database": { "status": "down", "message": "Connection failed" }
  },
  "details": {
    "database": { "status": "down", "message": "Connection failed" },
    "memory_heap": { "status": "up" },
    "pgboss": { "status": "up" }
  }
}
```
</CodeGroup>

<Warning>
The readiness probe performs comprehensive health checks and may return HTTP 503 if any critical dependency is unavailable.
</Warning>

**Health Checks Performed:**
- Database connectivity (MikroORM)
- Memory usage monitoring
- PgBoss job queue status

## Health Indicators

### PgBossHealthIndicator

Custom health indicator for the PgBoss background job service.

<CodeGroup>
```typescript Implementation
@Injectable()
export class PgBossHealthIndicator extends HealthIndicator {
  isHealthy(key: string): Promise<HealthIndicatorResult>
}
```

```typescript Usage Example
// In readiness check
const pgBossCheck = () => this.pgBossHealth.isHealthy('pgboss');
```
</CodeGroup>

<Tip>
The PgBoss health indicator checks the `PgBossService.isStarted` flag rather than performing database queries, making it a lightweight check that avoids additional database load.
</Tip>

**Implementation Notes:**
- Checks `PgBossService.isStarted` flag rather than database queries
- Lightweight check avoiding additional database load
- Returns healthy when PgBoss is properly started and connected

## Business Rules

### Health Check Behavior

<Steps>
  <Step title="Liveness Check Logic">
    Liveness checks always return 200 if the process can handle HTTP requests - this indicates the container is alive and doesn't need restarting.
  </Step>
  
  <Step title="Readiness Check Logic">
    Readiness checks return 503 if any critical dependency is unavailable - this prevents traffic routing to unhealthy instances.
  </Step>
  
  <Step title="Authentication Bypass">
    All health endpoints are publicly accessible with no authentication required for load balancer and orchestration access.
  </Step>
  
  <Step title="Version Neutral Routing">
    Health checks use version-neutral routing to ensure consistent endpoint availability across API versions.
  </Step>
</Steps>

### Failure Conditions

<Warning>
The following conditions will cause readiness checks to fail:

- Database connection lost or unresponsive
- Memory usage exceeds configured thresholds  
- PgBoss service not started or connection failed
- Any critical service dependency unavailable
</Warning>

## Integration Points

### External Integrations

<Tabs>
  <Tab title="Load Balancers">
    Use liveness endpoint for routing decisions and traffic distribution to healthy instances.
    
    ```yaml Example Configuration
    health_check:
      path: /health/live
      interval: 30s
      timeout: 5s
    ```
  </Tab>
  
  <Tab title="Kubernetes">
    Configure as liveness and readiness probes in pod specifications.
    
    ```yaml Pod Configuration
    livenessProbe:
      httpGet:
        path: /health/live
        port: 3000
      initialDelaySeconds: 30
      periodSeconds: 10
    
    readinessProbe:
      httpGet:
        path: /health/ready
        port: 3000
      initialDelaySeconds: 5
      periodSeconds: 5
    ```
  </Tab>
  
  <Tab title="Monitoring Systems">
    Scrape health endpoints for alerting and observability.
    
    ```yaml Prometheus Config
    - job_name: 'health-checks'
      static_configs:
        - targets: ['app:3000']
      metrics_path: '/health/ready'
    ```
  </Tab>
</Tabs>

## Patterns and Conventions

### Endpoint Naming

<Check>
**Consistent Naming Convention:**
- `/health/live` - Liveness probe
- `/health/ready` - Readiness probe  
- Version neutral routing (`VERSION_NEUTRAL`)
</Check>

### Response Standards

<CodeGroup>
```typescript Success Response
// HTTP 200: All checks passing
{
  "status": "ok",
  "info": { /* check details */ },
  "error": {},
  "details": { /* all check results */ }
}
```

```typescript Failure Response
// HTTP 503: One or more checks failing
{
  "status": "error", 
  "info": { /* passing checks */ },
  "error": { /* failing checks */ },
  "details": { /* all check results */ }
}
```
</CodeGroup>

### Error Handling

Health check failures are handled through the Terminus framework's error system:

```typescript
// Health check failures throw HealthCheckError
throw new HealthCheckError('PgBoss check failed', {
  pgboss: { status: 'down', message: 'Service not started' }
});
```
