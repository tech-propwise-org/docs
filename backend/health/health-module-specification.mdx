## Overview

The health module provides comprehensive application health monitoring through standardized health check endpoints. Built on NestJS Terminus, it implements a two-endpoint strategy distinguishing between liveness and readiness probes for container orchestration platforms like Kubernetes.

<Info>
The module monitors critical application dependencies including database connectivity, memory usage, and background job queue status through custom health indicators.
</Info>

The module is registered in `app.module.ts` and provides dedicated health check endpoints separate from the main application controller.

**Last Updated:** 2026-06-03  
**Status:** Active

## Architecture

### Design Decisions

<CardGroup cols={2}>
  <Card title="Two-Endpoint Strategy" icon="arrows-split-up-and-left">
    Separate liveness (`/health/live`) and readiness (`/health/ready`) endpoints following cloud-native best practices
  </Card>
  <Card title="Custom Health Indicators" icon="stethoscope">
    Extended Terminus framework with PgBoss job queue and search provider monitoring
  </Card>
  <Card title="Public Endpoints" icon="globe">
    Health checks bypass authentication for load balancer and orchestration access
  </Card>
  <Card title="Rate Limited" icon="gauge-high">
    Health checks use the app-wide throttler to reduce abuse risk while remaining reachable by probes
  </Card>
</CardGroup>

<Note>
**Lightweight checks**: Focused on essential service availability without heavy database operations to ensure fast probe responses.
</Note>

### Dependencies

- **@nestjs/terminus** - Core health check framework
- **PgBossService** - Background job queue monitoring
- **MikroORM** - Database connectivity checks
- **SearchProvider** - Search service availability (Phase 1A+)
- Memory monitoring for resource usage

### Data Integrity Audit Enhancements

The `DataIntegrityAuditService` tracks three categories of metrics:

<AccordionGroup>
  <Accordion title="Informational Count Fields">
    Tracks membership drift metrics that are **expected** after certain operations:
    
    - `stakeholdersWithoutActiveUserOrgRoleCount` - Stakeholder rows where the user no longer has an active UserOrgRole (expected after org removal, see ORG_USER_REMOVAL_SPECIFICATION.md Phase 4.3)
    - `totalInformationalCounts` - Sum of membership drift metrics (not unhealthy orphans)
  </Accordion>
  
  <Accordion title="Orphan Counts">
    Tracks entity rows with deleted foreign key references that indicate data integrity issues requiring remediation.
  </Accordion>
  
  <Accordion title="Creator Reference Counts">
    Tracks soft-deleted creator references which are informational only and do **not** contribute to `totalOrphans`.
  </Accordion>
</AccordionGroup>

<Warning>
**Classification requirement**: New audit counts must be explicitly categorized as orphan, creator reference, or informational.
</Warning>

## Health Indicators

### PgBossHealthIndicator

Custom health indicator for the PgBoss background job service using the modern `HealthIndicatorService` API (NestJS Terminus v11+).

```typescript
@Injectable()
export class PgBossHealthIndicator {
  constructor(
    @Inject(QUEUE_SERVICE) private readonly queueService: IQueueService,
    private readonly healthIndicatorService: HealthIndicatorService,
  ) {}

  async isHealthy(key: string) {
    const indicator = this.healthIndicatorService.check(key);
    const result = await this.queueService.checkHealth();
    if (!result.healthy) {
      return indicator.down({ reason: result.reason ?? 'queue backend unhealthy' });
    }
    return indicator.up({ latencyMs: result.latencyMs });
  }
}
```

<Tip>
Uses `HealthIndicatorService.check(key)` with `.up()` / `.down()` instead of the legacy `extends HealthIndicator` + `HealthCheckError` pattern.
</Tip>

#### Active Liveness Probe

<Warning>
**Why not just check `isStarted`?** The previous implementation reported healthy whenever `isStarted === true`. That flag is set once after `boss.start()` resolves and only flips back on `stop()` — if pg-boss loses its connection or its supervisor loop stalls after startup, the flag stays `true` and a **dead queue keeps reporting healthy**.
</Warning>

`checkHealth()` closes this gap by round-tripping a trivial `SELECT 1` through pg-boss's own connection pool (bounded by `QUEUE_HEALTH_PROBE_TIMEOUT_MS`, default 2000ms).

**Down State Reasons:**

| Condition | Reason Returned |
|-----------|----------------|
| `boss.start()` not yet complete | `boss.start() has not completed` (no query issued) |
| Probe query rejects (dropped connection) | The underlying driver error message |
| Probe exceeds timeout (stalled/hung queue) | `queue health probe timed out after <n>ms` |

<Info>
**Pool coverage**: The probe uses pg-boss's connection (the superuser `DATABASE_URL` pool), which is distinct from the ORM connection (`RUNTIME_DATABASE_URL`) that the readiness `database` check pings. The two checks therefore exercise different pools — a stalled pg-boss connection is caught here even when the ORM ping still succeeds.
</Info>

### SearchProviderHealthIndicator

Custom readiness indicator for the active `SearchProvider` implementation (`PostgresSearchProvider` in Phase 1). Mirrors the `PgBossHealthIndicator` pattern: a thin Terminus wrapper that delegates to the provider interface.

<Note>
`HealthModule` does not import `SearchModule` internals beyond the `SEARCH_PROVIDER` token.
</Note>

#### SearchProvider Contract

The `SearchProvider.healthCheck()` method behavior varies by implementation phase:

<Tabs>
  <Tab title="Phase 1B Steps 1–2 (Current)">
    Probes `information_schema` for `search_document` table existence via `executeReadOnlyWithBypass`.
    
    - Returns `up` when the table is reachable
    - Read path (`search()`) and write paths (`upsert` / `delete` + ACL diff) are live
    - `reindexOrg` still throws `SearchProviderNotImplementedError`
  </Tab>
  
  <Tab title="Phase 1B+ (Target)">
    Enhanced health checks including:
    
    - Optional row-count sanity checks
    - pg-boss `search.reindex` queue depth threshold monitoring
    - Reports `down` when indexer backlog exceeds SLO
  </Tab>
</Tabs>

See `Docs/SEARCH_MODULE_SPECIFICATION.md` §11.1 for complete contract details.

## Data Integrity Audit

### Overview

<Warning>
The scheduled data-integrity audit is **not wired into `/health/ready`** and has no public `/health/data-integrity` endpoint.
</Warning>

**Purpose:** Detect orphan rows resulting from soft-delete cascade gaps without exposing an unauthenticated, high-cost HTTP route.

The daily queued job runs at `30 3 * * *` by default (override via `DATA_INTEGRITY_AUDIT_CRON`) through `DataIntegrityAuditHandler`. It logs every run, emits a warn-level application log when unhealthy orphans are found, and sends a grouped Sentry error event when `totalOrphans > 0`.

<Info>
**Why separate from `/health/ready`?** Orphan rows do not affect whether the instance can serve traffic — they are a data-integrity concern, not a service-availability concern. Wiring this into the readiness probe would block rollouts unrelated to the symptom.
</Info>

### Creator Reference Handling

<Check>
Soft-deleted creator refs are **informational only** and do **not** contribute to `totalOrphans`.
</Check>

Creator references tracked:
- `dealsWithDeletedCreators`
- `contactsWithDeletedCreators`
- `companiesWithDeletedCreators`
- `listingsWithDeletedCreators`
- `personsWithDeletedCreators`
- `activitiesWithDeletedPerformers`
- `notesWithDeletedCreators`
- `tasksWithDeletedCreators`
- `eventsWithDeletedCreators`
- `invitationsWithDeletedCreators`

<Info>
User churn is normal, and historical records should still display the deleted creator when needed.
</Info>
