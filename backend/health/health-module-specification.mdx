## Overview

The health module provides comprehensive application health monitoring through standardized health check endpoints. Built on NestJS Terminus, it implements a two-endpoint strategy distinguishing between liveness and readiness probes for container orchestration platforms like Kubernetes.

The module monitors critical application dependencies including database connectivity, memory usage, and background job queue status through custom health indicators. The module is registered in `app.module.ts` and provides dedicated health check endpoints separate from the main application controller.

### Data Integrity Audit Enhancements

The `DataIntegrityAuditService` includes comprehensive tracking of data integrity metrics:

<Tabs>
  <Tab title="Orphan Counts">
    Tracks rows that reference soft-deleted parent entities (unhealthy state requiring remediation)
  </Tab>
  <Tab title="Creator Reference Counts">
    Monitors entities with soft-deleted creator users (informational only - user churn is expected)
  </Tab>
  <Tab title="Informational Counts">
    Tracks membership drift metrics like `stakeholdersWithoutActiveUserOrgRoleCount` (expected after org removal)
  </Tab>
</Tabs>

**Classification Requirement:**

All new audit counts must be explicitly categorized as:
- **Orphan counts** - contribute to `totalOrphans` (unhealthy)
- **Creator reference counts** - informational only
- **Informational counts** - includes `totalInformationalCounts` sum

See `ORG_USER_REMOVAL_SPECIFICATION.md` Phase 4.3 for stakeholder membership drift behavior.

## API Endpoints

### GET /health/live

Liveness probe - indicates if the process can handle requests.

```typescript
@Get('live')
@PublicEndpoint()
@HealthCheck()
liveness(): Promise<HealthCheckResult>
```

**Response:**

```json
{
  "status": "ok",
  "info": {},
  "error": {},
  "details": {}
}
```

**Use Cases:**

<Steps>
  <Step title="Container Orchestration">
    Kubernetes uses this endpoint to determine if the pod needs to be restarted
  </Step>
  <Step title="Load Balancer Routing">
    Load balancers check this endpoint to route traffic to healthy instances
  </Step>
  <Step title="Basic Health Verification">
    Simple check that the process is running and can respond to requests
  </Step>
</Steps>

### GET /health/ready

Readiness probe - indicates if the application can serve traffic.

```typescript
@Get('ready')
@PublicEndpoint()
@HealthCheck()
readiness(): Promise<HealthCheckResult>
```

**Health Checks Performed:**

<AccordionGroup>
  <Accordion title="Database Connectivity">
    Verifies MikroORM connection to the runtime database pool (`RUNTIME_DATABASE_URL`)
  </Accordion>
  <Accordion title="PgBoss Job Queue">
    Active liveness probe through `IQueueService.checkHealth()` - executes a test query through pg-boss's connection pool
  </Accordion>
  <Accordion title="Search Provider">
    Phase 1A check via `SearchProviderHealthIndicator` - validates search infrastructure readiness
  </Accordion>
</AccordionGroup>

**Response Format:**

```json
{
  "status": "ok|error",
  "info": {
    "database": { "status": "up" },
    "pgboss": { "status": "up" },
    "search": { "status": "up" }
  },
  "error": {},
  "details": {
    "database": { "status": "up" },
    "pgboss": { "status": "up" },
    "search": { "status": "up" }
  }
}
```

### Scheduled Data-Integrity Audit

<Warning>
The data integrity audit is **not wired into `/health/ready`** and has no public `/health/data-integrity` endpoint. This is a scheduled background job for monitoring, not a service availability check.
</Warning>

**Purpose:** Detect orphan rows resulting from soft-delete cascade gaps without exposing an unauthenticated, high-cost HTTP route.

The daily queued job runs at `30 3 * * *` by default (override via `DATA_INTEGRITY_AUDIT_CRON`) through `DataIntegrityAuditHandler`. It logs every run, emits a warn-level application log when unhealthy orphans are found, and sends a grouped Sentry error event when `totalOrphans > 0`.

#### Why Separate from Readiness?

Orphan rows do not affect whether the instance can serve traffic — they are a data-integrity concern, not a service-availability concern. Wiring this into the readiness probe would block rollouts unrelated to the symptom.

## Health Indicators

### PgBossHealthIndicator

Custom health indicator for the PgBoss background job service. Uses the `HealthIndicatorService` API (NestJS Terminus v11+) instead of the deprecated `HealthIndicator` base class.

```typescript
@Injectable()
export class PgBossHealthIndicator {
  constructor(
    @Inject(QUEUE_SERVICE) private readonly queueService: IQueueService,
    private readonly healthIndicatorService: HealthIndicatorService,
  ) {}

  async isHealthy(key: string) {
    const indicator = this.healthIndicatorService.check(key);
    const result = await this.queueService.checkHealth();
    if (!result.healthy) {
      return indicator.down({ reason: result.reason ?? 'queue backend unhealthy' });
    }
    return indicator.up({ latencyMs: result.latencyMs });
  }
}
```

<Warning>
**Why not just check `isStarted`?**

The previous implementation reported healthy whenever `isStarted === true`. That flag is set once after `boss.start()` resolves and only flips back on `stop()` — so if pg-boss loses its connection or its supervisor loop stalls after startup, the flag stays `true` and a **dead queue keeps reporting healthy**.
</Warning>

`checkHealth()` closes this gap by round-tripping a trivial `SELECT 1` through pg-boss's own connection pool (bounded by `QUEUE_HEALTH_PROBE_TIMEOUT_MS`, default 2000ms).

#### Down States

| Condition | Reason Returned |
|-----------|----------------|
| `boss.start()` not yet complete | `boss.start() has not completed` (no query issued) |
| Probe query rejects (dropped connection) | The underlying driver error message |
| Probe exceeds timeout (stalled/hung queue) | `queue health probe timed out after <n>ms` |

<Note>
**Pool Coverage:** The probe uses pg-boss's connection (the superuser `DATABASE_URL` pool), which is distinct from the ORM connection (`RUNTIME_DATABASE_URL`) that the readiness `database` check pings. The two checks therefore exercise different pools — a stalled pg-boss connection is caught here even when the ORM ping still succeeds.
</Note>

#### Implementation Notes

<Steps>
  <Step title="Modern Terminus API">
    Uses `HealthIndicatorService.check(key)` with `.up()` / `.down()` instead of the legacy `extends HealthIndicator` + `HealthCheckError` pattern
  </Step>
  <Step title="Active Liveness Probe">
    Delegates to `IQueueService.checkHealth()` for an **active** liveness probe rather than reading the `isStarted` boolean
  </Step>
  <Step title="Timeout Protection">
    Bounded by `QUEUE_HEALTH_PROBE_TIMEOUT_MS` (default 2000ms) to prevent hanging health checks
  </Step>
</Steps>

### SearchProviderHealthIndicator

Custom readiness indicator for the active `SearchProvider` implementation (`PostgresSearchProvider` in Phase 1). Mirrors the `PgBossHealthIndicator` pattern: a thin Terminus wrapper in `src/modules/health/` that delegates to the provider interface so `HealthModule` does not import `SearchModule` internals beyond the `SEARCH_PROVIDER` token.

#### Contract (`SearchProvider.healthCheck()`)

<Tabs>
  <Tab title="Phase 1B Steps 1–2 (Current)">
    Probes `information_schema` for `search_document` table existence via `executeReadOnlyWithBypass`.
    
    **Returns:**
    - `up` when the table is reachable
    - Read path (`search()`) and write paths (`upsert` / `delete` + ACL diff) are live
    - `reindexOrg` still throws `SearchProviderNotImplementedError`
  </Tab>
  
  <Tab title="Phase 1B+ (Target)">
    Enhanced health checks including:
    - Optional row-count sanity checks
    - pg-boss `search.reindex` queue depth threshold monitoring
    - Reports `down` when indexer backlog exceeds SLO
  </Tab>
</Tabs>
