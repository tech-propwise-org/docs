## Overview

The health module provides comprehensive application health monitoring through standardized health check endpoints. Built on NestJS Terminus, it implements a two-endpoint strategy distinguishing between liveness and readiness probes for container orchestration platforms like Kubernetes.

<Info>
The module monitors critical application dependencies including database connectivity, memory usage, and background job queue status through custom health indicators.
</Info>

The module is registered in `app.module.ts` and provides dedicated health check endpoints separate from the main application controller.

**Last Updated:** 2024-12-19  
**Status:** Active

## Architecture

### Design decisions

The health module follows cloud-native best practices with several key architectural decisions:

<CardGroup cols={2}>
  <Card title="Two-endpoint strategy" icon="route">
    Separate liveness (`/health/live`) and readiness (`/health/ready`) endpoints following cloud-native best practices
  </Card>
  <Card title="Custom health indicators" icon="gauge">
    Extended Terminus framework with PgBoss job queue monitoring
  </Card>
  <Card title="Public endpoints" icon="unlock">
    Health checks bypass authentication for load balancer and orchestration access
  </Card>
  <Card title="Lightweight checks" icon="feather">
    Focused on essential service availability without heavy database operations
  </Card>
</CardGroup>

### Dependencies

The health module relies on the following key dependencies:

- **@nestjs/terminus** - Core health check framework
- **PgBossService** - Background job queue monitoring
- **MikroORM** - Database connectivity checks
- **Memory monitoring** - Resource usage tracking

## API Endpoints

### GET /health/live

**Purpose:** Liveness probe - indicates if the process can handle requests

<CodeGroup>
```typescript TypeScript Interface
@Get('live')
@PublicEndpoint()
@HealthCheck()
liveness(): Promise<HealthCheckResult>
```

```json Response
{
  "status": "ok",
  "info": {},
  "error": {},
  "details": {}
}
```
</CodeGroup>

<Note>
The liveness endpoint is used by container orchestration platforms to determine if a process should be restarted.
</Note>

**Use cases:**
- Container orchestration restart decisions
- Load balancer routing decisions
- Basic process health verification

### GET /health/ready

**Purpose:** Readiness probe - indicates if the application can serve traffic

<CodeGroup>
```typescript TypeScript Interface
@Get('ready')
@PublicEndpoint()
@HealthCheck()
readiness(): Promise<HealthCheckResult>
```

```json Successful Response
{
  "status": "ok",
  "info": {
    "database": { "status": "up" },
    "memory_heap": { "status": "up" },
    "pgboss": { "status": "up" }
  },
  "error": {},
  "details": {
    "database": { "status": "up" },
    "memory_heap": { "status": "up" },
    "pgboss": { "status": "up" }
  }
}
```

```json Error Response
{
  "status": "error",
  "info": {
    "database": { "status": "up" },
    "memory_heap": { "status": "up" }
  },
  "error": {
    "pgboss": { "status": "down", "message": "Service not started" }
  },
  "details": {
    "database": { "status": "up" },
    "memory_heap": { "status": "up" },
    "pgboss": { "status": "down", "message": "Service not started" }
  }
}
```
</CodeGroup>

**Health checks performed:**
- Database connectivity (MikroORM)
- Memory usage monitoring
- PgBoss job queue status

<Warning>
The readiness endpoint returns HTTP 503 if any critical dependency is unavailable, preventing traffic routing to unhealthy instances.
</Warning>

## Health Indicators

### PgBossHealthIndicator

Custom health indicator for the PgBoss background job service.

```typescript
@Injectable()
export class PgBossHealthIndicator extends HealthIndicator {
  isHealthy(key: string): Promise<HealthIndicatorResult>
}
```

<Tip>
The PgBoss health indicator checks the `PgBossService.isStarted` flag rather than performing database queries, keeping the check lightweight and avoiding additional database load.
</Tip>

**Implementation notes:**
- Checks `PgBossService.isStarted` flag rather than database queries
- Lightweight check avoiding additional database load
- Returns healthy when PgBoss is properly started and connected

## Business Rules

### Health check behavior

<Steps>
  <Step title="Liveness checks">
    Always return 200 if the process can handle HTTP requests. This is a basic process health verification.
  </Step>
  
  <Step title="Readiness checks">
    Return 503 if any critical dependency is unavailable. This prevents traffic routing to unhealthy instances.
  </Step>
  
  <Step title="Public access">
    All health endpoints are publicly accessible with no authentication required for load balancer and orchestration access.
  </Step>
  
  <Step title="Version neutral">
    Health checks use version-neutral routing to ensure consistency across API versions.
  </Step>
</Steps>

### Failure conditions

The following conditions will cause health checks to fail:

- Database connection lost or unresponsive
- Memory usage exceeds configured thresholds
- PgBoss service not started or connection failed
- Any critical service dependency unavailable

## Patterns and Conventions

### Endpoint naming

The health module follows consistent naming conventions:

- `/health/live` - Liveness probe
- `/health/ready` - Readiness probe
- Version neutral routing (`VERSION_NEUTRAL`)

### Response standards

<Check>
**HTTP 200:** All checks passing - service is healthy and ready to serve traffic
</Check>

<Warning>
**HTTP 503:** One or more checks failing - service should not receive traffic
</Warning>

All responses follow consistent JSON format according to Terminus conventions.

### Error handling

Health check failures are handled through the standard Terminus error mechanism:

```typescript
// Health check failures throw HealthCheckError
throw new HealthCheckError('PgBoss check failed', {
  pgboss: { status: 'down', message: 'Service not started' }
});
```
