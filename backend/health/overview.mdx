## Overview

The health module provides comprehensive application health monitoring through standardized health check endpoints. Built on NestJS Terminus, it implements a two-endpoint strategy distinguishing between liveness and readiness probes for container orchestration platforms like Kubernetes.

<Info>
The module monitors critical application dependencies including database connectivity, memory usage, and background job queue status through custom health indicators.
</Info>

## Architecture

### Design Decisions

The health module follows cloud-native best practices with several key architectural decisions:

- **Two-endpoint strategy**: Separate liveness (`/health/live`) and readiness (`/health/ready`) endpoints
- **Custom health indicators**: Extended Terminus framework with PgBoss job queue monitoring  
- **Public endpoints**: Health checks bypass authentication for load balancer and orchestration access
- **Lightweight checks**: Focused on essential service availability without heavy database operations

## Health Indicators

### Custom PgBoss Indicator

```typescript
@Injectable()
export class PgBossHealthIndicator extends HealthIndicator {
  isHealthy(key: string): Promise<HealthIndicatorResult>
}
```

<Tip>
The PgBoss health indicator checks the `PgBossService.isStarted` flag rather than performing database queries, keeping the check lightweight and avoiding additional database load.
</Tip>

**Implementation characteristics:**
- Checks service startup status flag
- Avoids database round-trips
- Returns healthy when PgBoss is properly started and connected

### Built-in Indicators

## Business Rules

### Health Check Behavior

<Warning>
Liveness and readiness probes serve different purposes and should not be confused:
</Warning>

1. **Liveness checks** always return 200 if the process can handle HTTP requests
2. **Readiness checks** return 503 if any critical dependency is unavailable
3. All health endpoints are publicly accessible (no authentication required)
4. Health checks use version-neutral routing

### Failure Conditions

The following conditions will cause readiness checks to fail:

- Database connection lost or unresponsive
- Memory usage exceeds configured thresholds
- PgBoss service not started or connection failed
- Any critical service dependency unavailable

## Implementation Patterns

### Endpoint Configuration

```typescript
@Get('live')
@PublicEndpoint()
@HealthCheck()
liveness(): Promise<HealthCheckResult>

@Get('ready')  
@PublicEndpoint()
@HealthCheck()
readiness(): Promise<HealthCheckResult>
```

<Check>
Both endpoints use `@PublicEndpoint()` to bypass authentication and `VERSION_NEUTRAL` routing for consistent access across API versions.
</Check>

### Response Standards

- **HTTP 200**: All checks passing
- **HTTP 503**: One or more checks failing
- Consistent JSON response format following Terminus conventions

### Error Handling

```typescript
// Health check failures throw HealthCheckError
throw new HealthCheckError('PgBoss check failed', {
  pgboss: { 
    status: 'down', 
    message: 'Service not started' 
  }
});
```
