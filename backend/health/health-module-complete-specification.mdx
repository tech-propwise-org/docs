## Overview

The health module provides comprehensive application health monitoring through standardized health check endpoints. Built on NestJS Terminus, it implements a two-endpoint strategy distinguishing between liveness and readiness probes for container orchestration platforms like Kubernetes.

<Info>
The module monitors critical application dependencies including database connectivity, memory usage, and background job queue status through custom health indicators.
</Info>

The module is registered in `app.module.ts` and provides dedicated health check endpoints separate from the main application controller.

**Last Updated:** 2025-04-17  
**Status:** Active

<CodeGroup>

## Health Indicators

### PgBossHealthIndicator

Custom health indicator for the PgBoss background job service using the modern NestJS Terminus v11+ API.

<Warning>
This implementation uses `HealthIndicatorService` instead of the deprecated `HealthIndicator` base class pattern.
</Warning>

```typescript
@Injectable()
export class PgBossHealthIndicator {
  constructor(
    @Inject(QUEUE_SERVICE) private readonly queueService: IQueueService,
    private readonly healthIndicatorService: HealthIndicatorService,
  ) {}

  async isHealthy(key: string) {
    const indicator = this.healthIndicatorService.check(key);
    if (!this.queueService.isStarted) {
      return indicator.down({ reason: 'boss.start() has not completed' });
    }
    return indicator.up();
  }
}
```

<Steps>
  <Step title="Dependency Injection">
    Injects the queue service and health indicator service
  </Step>
  <Step title="Health Check Logic">
    Verifies that PgBoss service is properly started
  </Step>
  <Step title="Response Generation">
    Returns appropriate health status using the modern API
  </Step>
</Steps>

**Implementation Notes:**
- Uses `HealthIndicatorService.check(key)` with `.up()` / `.down()` methods
- Checks `IQueueService.isStarted` flag rather than database queries
- Lightweight check avoiding additional database load
- Returns healthy when PgBoss is properly started and connected

## Business Rules

### Health Check Behavior

<Check>
**Liveness checks** always return 200 if the process can handle HTTP requests
</Check>

<Check>
**Readiness checks** return 503 if any critical dependency is unavailable
</Check>

<Check>
All health endpoints are publicly accessible (no authentication required)
</Check>

<Check>
Health checks use version-neutral routing
</Check>

### Failure Conditions

The following conditions will cause health checks to fail:

- Database connection lost or unresponsive
- Memory usage exceeds configured thresholds
- PgBoss service not started or connection failed
- Any critical service dependency unavailable

## Patterns and Conventions

### Endpoint Naming

<CodeGroup>
```http Liveness Endpoint
GET /health/live
```

```http Readiness Endpoint  
GET /health/ready
```
</CodeGroup>

Both endpoints use version neutral routing (`VERSION_NEUTRAL`).

### Response Standards

<Tabs>
  <Tab title="Success Response">
    **HTTP 200:** All checks passing
    
    ```json
    {
      "status": "ok",
      "info": { /* check details */ },
      "error": {},
      "details": { /* all check results */ }
    }
    ```
  </Tab>
  <Tab title="Failure Response">
    **HTTP 503:** One or more checks failing
    
    ```json
    {
      "status": "error",
      "info": { /* passing checks */ },
      "error": { /* failing checks */ },
      "details": { /* all check results */ }
    }
    ```
  </Tab>
</Tabs>

### Error Handling

<Warning>
The legacy `HealthCheckError` + `extends HealthIndicator` pattern is deprecated and will be removed in the next major Terminus release.
</Warning>

**Modern Approach (Recommended):**
```typescript
// Health check failures use indicator.down() (NestJS Terminus v11+)
const indicator = this.healthIndicatorService.check(key);
return indicator.down({ reason: 'Service not started' });
```

**Legacy Approach (Deprecated):**
```typescript
// The legacy HealthCheckError + extends HealthIndicator pattern is deprecated
// and will be removed in the next major Terminus release.
```
