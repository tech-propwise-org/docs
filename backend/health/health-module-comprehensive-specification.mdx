## Overview

The health module provides comprehensive application health monitoring through standardized health check endpoints. Built on NestJS Terminus, it implements a two-endpoint strategy distinguishing between liveness and readiness probes for container orchestration platforms like Kubernetes.

<Info>
The module monitors critical application dependencies including database connectivity, memory usage, and background job queue status through custom health indicators.
</Info>

The module is registered in `app.module.ts` and provides dedicated health check endpoints separate from the main application controller.

## Architecture

### Design Decisions

The health module follows cloud-native best practices with these key architectural decisions:

- **Two-endpoint strategy**: Separate liveness (`/health/live`) and readiness (`/health/ready`) endpoints following cloud-native best practices
- **Custom health indicators**: Extended Terminus framework with PgBoss job queue monitoring
- **Public endpoints**: Health checks bypass authentication for load balancer and orchestration access
- **Lightweight checks**: Focused on essential service availability without heavy database operations

## Health Indicators

### PgBoss Health Indicator

<Note>
The PgBoss health indicator uses the new `HealthIndicatorService` API from NestJS Terminus v11+ instead of the deprecated `HealthIndicator` base class pattern.
</Note>

<CodeGroup>

```typescript PgBoss Health Indicator
@Injectable()
export class PgBossHealthIndicator {
  constructor(
    @Inject(QUEUE_SERVICE) private readonly queueService: IQueueService,
    private readonly healthIndicatorService: HealthIndicatorService,
  ) {}

  async isHealthy(key: string) {
    const indicator = this.healthIndicatorService.check(key);
    if (!this.queueService.isStarted) {
      return indicator.down({ reason: 'boss.start() has not completed' });
    }
    return indicator.up();
  }
}
```

```typescript Legacy Pattern (Deprecated)
// The following pattern is deprecated and will be removed
// in the next major Terminus release

export class PgBossHealthIndicator extends HealthIndicator {
  async isHealthy(key: string) {
    if (!this.queueService.isStarted) {
      throw new HealthCheckError('PgBoss not started', {
        [key]: { status: 'down', reason: 'boss.start() has not completed' }
      });
    }
    return this.getStatus(key, true);
  }
}
```

</CodeGroup>

**Implementation Notes:**

<Steps>
<Step title="Use Modern API">
Uses `HealthIndicatorService.check(key)` with `.up()` / `.down()` instead of the legacy `extends HealthIndicator` + `HealthCheckError` pattern
</Step>
<Step title="Lightweight Checking">
Checks `IQueueService.isStarted` flag rather than database queries
</Step>
<Step title="Avoid Database Load">
Lightweight check avoiding additional database load during health verification
</Step>
<Step title="Clear Status Indication">
Returns healthy when PgBoss is properly started and connected
</Step>
</Steps>

## Business Rules

### Health Check Behavior

<AccordionGroup>
<Accordion title="Liveness Check Rules">
1. **Always returns 200** if the process can handle HTTP requests
2. **No dependency validation** - only process-level health
3. **Fast response time** - minimal processing overhead
4. **Container restart trigger** - failure indicates need for process restart
</Accordion>

<Accordion title="Readiness Check Rules">
1. **Returns 503** if any critical dependency is unavailable
2. **Validates all dependencies** - database, memory, queue service
3. **Traffic routing decision** - determines if instance can serve requests
4. **Graceful degradation** - temporary unavailability without restart
</Accordion>

<Accordion title="Security Rules">
- All health endpoints are **publicly accessible** (no authentication required)
- Health checks use **version-neutral routing**
- **No sensitive data** exposed in health check responses
</Accordion>
</AccordionGroup>

### Failure Conditions

<Warning>
Readiness checks will fail under these conditions:
</Warning>

- Database connection lost or unresponsive
- Memory usage exceeds configured thresholds  
- PgBoss service not started or connection failed
- Any critical service dependency unavailable

### Error Handling

<CodeGroup>

```typescript Modern Approach (Recommended)
// Health check failures use indicator.down() (NestJS Terminus v11+)
const indicator = this.healthIndicatorService.check(key);
return indicator.down({ reason: 'Service not started' });
```

```typescript Legacy Approach (Deprecated)
// The legacy HealthCheckError + extends HealthIndicator pattern is deprecated
// and will be removed in the next major Terminus release.
throw new HealthCheckError('Service failed', {
  [key]: { status: 'down', reason: 'Service not started' }
});
```

</CodeGroup>

## Observability Stack

<Info>
Beyond health check endpoints, the application uses a layered observability architecture combining error tracking, uptime monitoring, and structured logging.
</Info>

### Sentry (Error Tracking & Performance)

<Tabs>
<Tab title="Backend Configuration">
**Setup** (`@sentry/nestjs`):
- SDK initialized in `src/instrument.ts` (loaded before NestFactory)
- `SentryModule.forRoot()` + `SentryGlobalFilter` as `APP_FILTER` captures all unhandled exceptions
- `LogContextInterceptor` enriches Sentry events with user context

**Configuration:**
- `tracesSampleRate: 0.2` in production (all error events captured at 100%)
- `nodeProfilingIntegration()` for performance profiling
- Health transactions (`GET /health*`) excluded via `ignoreTransactions`

**Context Enrichment:**
```typescript
// User context
Sentry.setUser({ id, email })

// Organization tagging  
Sentry.setTag('organization_id', organizationId)
```
</Tab>

<Tab title="Frontend Configuration">
**Setup** (`@sentry/nextjs`):
- Client config (`sentry.client.config.ts`) with Session Replay and Breadcrumbs
- Server and Edge configs for SSR/middleware error capture
- `next.config.ts` wrapped with `withSentryConfig` for source map uploads

**Error Boundaries:**
- `global-error.tsx` root error boundary reports to Sentry
- `AuthProvider` manages user context on login/logout

**Session Replay:**
- `replaysOnErrorSampleRate: 1.0` — always captures DOM replay when error occurs
- `replaysSessionSampleRate: 0` — no recording of healthy sessions
- `tunnelRoute: '/monitoring'` — proxies events to avoid ad-blockers
</Tab>
</Tabs>

**Environment Variables:**
- `SENTRY_DSN` — backend DSN
- `NEXT_PUBLIC_SENTRY_DSN` — frontend DSN (can be same project for correlated traces)
- `SENTRY_AUTH_TOKEN` — optional, for source map uploads during build
