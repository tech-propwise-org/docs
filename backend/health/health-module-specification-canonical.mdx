## Overview

The health module provides comprehensive application health monitoring through standardized health check endpoints. Built on NestJS Terminus, it implements a two-endpoint strategy distinguishing between liveness and readiness probes for container orchestration platforms like Kubernetes.

<Note>
The module monitors critical application dependencies including database connectivity, memory usage, and background job queue status through custom health indicators.
</Note>

The module is registered in `app.module.ts` and provides dedicated health check endpoints separate from the main application controller.

## Architecture

### Design Decisions

<AccordionGroup>
<Accordion title="Two-endpoint Strategy">
Separate liveness (`/health/live`) and readiness (`/health/ready`) endpoints following cloud-native best practices for container orchestration.
</Accordion>

<Accordion title="Custom Health Indicators">
Extended Terminus framework with PgBoss job queue monitoring and database connectivity checks.
</Accordion>

<Accordion title="Public Endpoints">
Health checks bypass authentication for load balancer and orchestration access while using app-wide throttling.
</Accordion>

<Accordion title="Lightweight Checks">
Focused on essential service availability without heavy database operations.
</Accordion>
</AccordionGroup>

### Dependencies

- `@nestjs/terminus` - Core health check framework
- `PgBossService` - Background job queue monitoring  
- `MikroORM` - Database connectivity checks
- Memory monitoring for resource usage

## API Endpoints

### GET /health/live

**Purpose:** Liveness probe - indicates if the process can handle requests

### GET /health/ready

**Purpose:** Readiness probe - indicates if the application can serve traffic

<CodeGroup>
```typescript Controller

@PublicEndpoint()
@HealthCheck()
readiness(): Promise<HealthCheckResult>
```

```json Response

  "info": {

    "pgboss": { "status": "up" }
```

</CodeGroup>

**Health Checks Performed:**
- Database connectivity (MikroORM)
- PgBoss job queue status

### Scheduled Data Integrity Audit

<Warning>
The audit is **not wired into `/health/ready`** and has no public `/health/data-integrity` endpoint. This is intentional - orphan rows do not affect service availability.
</Warning>

**Purpose:** Detect orphan rows resulting from soft-delete cascade gaps without exposing an unauthenticated, high-cost HTTP route.

The daily queued job runs at `30 3 * * *` by default (override via `DATA_INTEGRITY_AUDIT_CRON`) through `DataIntegrityAuditHandler`. It logs every run, emits a warn-level application log when unhealthy orphans are found, and sends a grouped Sentry error event when `totalOrphans > 0`.

<Steps>
<Step title="Audit Execution">
The audit runs ~74 count queries grouped into module buckets with parallel `Promise.all` execution.
</Step>

<Step title="Coverage Areas">
Checks 11 module buckets including Core entities, Person/Company sub-entities, Activity stack, Tasks/Events, and RBAC junctions.
</Step>

<Step title="Reporting">
Emits Sentry error events with tags `subsystem=health`, `check=data-integrity`, and remediation commands.
</Step>
</Steps>

**Audit Coverage:**

<Tabs>
<Tab title="Core Entities">
- Lead/Deal/Contact/Company/Listing with deleted persons/orgs/stages/creators
- 12 fixed relations from soft-delete standards
</Tab>

<Tab title="Sub-Entities">
- Person.organization, Person.createdBy
- Person/Company Email/Phone/Address references
- PersonNotDuplicate relationships
</Tab>

<Tab title="Activity Stack">
- Activity, Note, CallLog, EmailLog, SmsLog, Meeting, SiteVisit
- Organization and creator references
- Reminder.user relationships
</Tab>

<Tab title="Tasks & Events">
- Tag.organization references
- Task and Event with organization/creator links
- Comment, Assignee, and Tag junction tables
</Tab>
</Tabs>

<Info>
Soft-deleted creator refs are informational only and do **not** contribute to `totalOrphans`. User churn is normal, and historical records should retain deleted creator references.
</Info>

**Companion Infrastructure:**
- **Sentry alert event:** Error-level with fingerprint `data-integrity-orphans`
- **Repair script:** `pnpm repair:orphaned-lead-persons --dry-run`
- **Root cause:** MikroORM v6 `autoJoinRefsForFilters` behavior

## Health Indicators

### PgBossHealthIndicator

Custom health indicator using the modern `HealthIndicatorService` API (NestJS Terminus v11+):

```typescript
@Injectable()
export class PgBossHealthIndicator {
  constructor(
    @Inject(QUEUE_SERVICE) private readonly queueService: IQueueService,
    private readonly healthIndicatorService: HealthIndicatorService,
  ) {}

  async isHealthy(key: string) {
    const indicator = this.healthIndicatorService.check(key);
    if (!this.queueService.isStarted) {
      return indicator.down({ reason: 'boss.start() has not completed' });
    }
    return indicator.up();
  }
}
```

<Note>
Uses `HealthIndicatorService.check(key)` with `.up()` / `.down()` instead of the legacy `extends HealthIndicator` + `HealthCheckError` pattern.
</Note>

**Implementation Notes:**
- Checks `IQueueService.isStarted` flag rather than database queries
- Lightweight check avoiding additional database load
- Returns healthy when PgBoss is properly started and connected

## Business Rules

### Health Check Behavior

<CardGroup cols={2}>
<Card title="Liveness Checks" icon="heartbeat">
Always return 200 if the process can handle HTTP requests
</Card>

<Card title="Readiness Checks" icon="shield-check">
Return 503 if any critical dependency is unavailable
</Card>

<Card title="Public Access" icon="globe">
All health endpoints are publicly accessible (no authentication required)
</Card>

<Card title="Rate Limiting" icon="gauge">
Health endpoints are covered by the global `AppThrottlerGuard`
</Card>
</CardGroup>

### Failure Conditions

<Warning>
The following conditions will cause readiness check failures:
</Warning>

- Database connection lost or unresponsive
- Memory usage exceeds configured thresholds
- PgBoss service not started or connection failed
- Any critical service dependency unavailable


## Observability Stack

Beyond health check endpoints, the application uses a layered observability architecture:

### Sentry (Error Tracking & Performance)

<Tabs>
<Tab title="Backend">
**Configuration** (`@sentry/nestjs`):
- SDK initialized in `src/instrument.ts` (loaded before NestFactory)
- `SentryModule.forRoot()` + `SentryGlobalFilter` as `APP_FILTER`
- `LogContextInterceptor` enriches events with user/org context
- `tracesSampleRate: 0.2` in production
- Health transactions excluded via `ignoreTransactions`

**Data Integrity Integration:**
- `DataIntegrityAuditHandler` emits grouped error events
- Tags: `check=data-integrity`, fingerprint `data-integrity-orphans`
</Tab>

<Tab title="Frontend">
**Configuration** (`@sentry/nextjs`):
- Client config with Session Replay and Breadcrumbs
- Server and Edge configs for SSR/middleware capture
- `global-error.tsx` root error boundary
- `replaysOnErrorSampleRate: 1.0` — always captures DOM replay on error
- `tunnelRoute: '/monitoring'` — proxies events to avoid ad-blockers

**User Context:**
- `AuthProvider` sets user/org context on login
- Clears context on logout
</Tab>
</Tabs>

**Environment Variables:**
- `SENTRY_DSN` — backend DSN
- `NEXT_PUBLIC_SENTRY_DSN` — frontend DSN
- `SENTRY_AUTH_TOKEN` — for source map uploads
