## Overview

The health module provides comprehensive application health monitoring through standardized health check endpoints. Built on NestJS Terminus, it implements a two-endpoint strategy distinguishing between liveness and readiness probes for container orchestration platforms like Kubernetes.

<Info>
The module monitors critical application dependencies including database connectivity, memory usage, and background job queue status through custom health indicators.
</Info>

The module is registered in `app.module.ts` and provides dedicated health check endpoints separate from the main application controller.

## API Endpoints

### Liveness Probe

<CodeGroup>
```typescript TypeScript
@Get('live')
@PublicEndpoint()
@HealthCheck()
liveness(): Promise<HealthCheckResult>
```

```json Response
{
  "status": "ok",
  "info": {},
  "error": {},
  "details": {}
}
```
</CodeGroup>

<Note>
**Purpose:** Indicates if the process can handle requests - used for container orchestration restart decisions and load balancer routing.
</Note>

**Use Cases:**
- Container orchestration restart decisions
- Load balancer routing decisions
- Basic process health verification

## Health Indicators

### Custom PgBoss Health Indicator

<Warning>
This implementation uses the modern `HealthIndicatorService` API (NestJS Terminus v11+) instead of the deprecated `HealthIndicator` base class.
</Warning>

<CodeGroup>
```typescript Implementation
@Injectable()
export class PgBossHealthIndicator {
  constructor(
    @Inject(QUEUE_SERVICE) private readonly queueService: IQueueService,
    private readonly healthIndicatorService: HealthIndicatorService,
  ) {}

  async isHealthy(key: string) {
    const indicator = this.healthIndicatorService.check(key);
    if (!this.queueService.isStarted) {
      return indicator.down({ reason: 'boss.start() has not completed' });
    }
    return indicator.up();
  }
}
```

```typescript Legacy (Deprecated)
// DEPRECATED: This pattern will be removed in future Terminus releases
export class PgBossHealthIndicator extends HealthIndicator {
  // Don't use this approach
}
```
</CodeGroup>

**Implementation Notes:**
- Uses `HealthIndicatorService.check(key)` with `.up()` / `.down()` methods
- Checks `IQueueService.isStarted` flag rather than database queries
- Lightweight check avoiding additional database load
- Returns healthy when PgBoss is properly started and connected

## Business Rules

### Health Check Behavior

<Steps>
<Step title="Liveness Check Logic">
Always returns 200 if the process can handle HTTP requests - this is a basic process health verification
</Step>
<Step title="Readiness Check Logic">
Returns 503 if any critical dependency is unavailable, ensuring traffic is only routed to fully functional instances
</Step>
<Step title="Public Access">
All health endpoints are publicly accessible with no authentication required for infrastructure monitoring
</Step>
<Step title="Version Neutral">
Health checks use version-neutral routing for consistent access across API versions
</Step>
</Steps>

### Failure Conditions

<AccordionGroup>
<Accordion title="Database Failures">
- Database connection lost or unresponsive
- Connection pool exhausted
- Database server unavailable
</Accordion>
<Accordion title="Memory Issues">
- Memory usage exceeds configured thresholds
- Heap memory pressure indicators
- Out of memory conditions approaching
</Accordion>
<Accordion title="Queue Service Problems">
- PgBoss service not started or connection failed
- Job queue processing halted
- Background service dependencies unavailable
</Accordion>
</AccordionGroup>

## Patterns and Conventions

### Endpoint Naming Convention

<CodeGroup>
```bash Liveness
GET /health/live
```

```bash Readiness  
GET /health/ready
```
</CodeGroup>

<Tip>
Both endpoints use version-neutral routing (`VERSION_NEUTRAL`) for consistent access across API versions.
</Tip>

### Response Standards

<Tabs>
<Tab title="Success Response">
- **HTTP 200**: All checks passing
- Consistent JSON format following Terminus conventions
- Detailed status information in `info` and `details` sections
</Tab>
<Tab title="Failure Response">
- **HTTP 503**: One or more checks failing
- Error details in `error` section with specific failure reasons
- Partial success information still included
</Tab>
</Tabs>

### Modern Error Handling

<CodeGroup>
```typescript Recommended (v11+)
// Modern approach using HealthIndicatorService
const indicator = this.healthIndicatorService.check(key);
return indicator.down({ reason: 'Service not started' });
```

```typescript Deprecated
// DEPRECATED: Legacy pattern being removed
throw new HealthCheckError(
  'PgBoss check failed',
  this.getStatus('pgboss', false, { reason: 'Service not started' })
);
```
</CodeGroup>

<Warning>
The legacy `HealthCheckError` + `extends HealthIndicator` pattern is deprecated and will be removed in the next major Terminus release.
</Warning>
