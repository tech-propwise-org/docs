## Overview

The health module provides comprehensive application health monitoring through standardized health check endpoints. Built on NestJS Terminus, it implements a two-endpoint strategy distinguishing between liveness and readiness probes for container orchestration platforms like Kubernetes.

<Note>
The module monitors critical application dependencies including database connectivity, memory usage, and background job queue status through custom health indicators. The module is registered in `app.module.ts` and provides dedicated health check endpoints separate from the main application controller.
</Note>

  <Card title="Lightweight Checks" icon="feather">

  </Card>

### Dependencies

- `@nestjs/terminus` - Core health check framework
- `PgBossService` - Background job queue monitoring
- `MikroORM` - Database connectivity checks

<Warning>
Health checks use the app-wide throttler to reduce abuse risk while remaining reachable by probes. Clients that exceed the configured limit receive HTTP 429.
</Warning>

## Data Integrity Audit

<Info>
The scheduled data-integrity audit detects orphan rows resulting from soft-delete cascade gaps without exposing an unauthenticated, high-cost HTTP route. The audit is **not wired into `/health/ready`** and has no public endpoint.
</Info>

### Audit Schedule

The daily queued job runs at `30 3 * * *` by default through `DataIntegrityAuditHandler`:

- Override schedule via `DATA_INTEGRITY_AUDIT_CRON` environment variable
- Logs every run execution
- Emits warn-level application log when unhealthy orphans are found
- Sends grouped Sentry error event when `totalOrphans > 0`

### Audit Coverage

<Steps>
  <Step title="Core Entities (12 checks)">
    Lead/Deal/Contact/Company/Listing with deleted persons/organizations/stages/creators
  </Step>
  
  <Step title="Person & Company Sub-entities (10 checks)">
    Person.organization, Person.createdBy, Person{Email,Phone,Address,Channel}.person, PersonNotDuplicate, Company{Email,Phone,Address}.company
  </Step>
  
  <Step title="Activity Stack + Reminder (10 checks)">
    Activity, Note, CallLog, EmailLog, SmsLog, Meeting, SiteVisit organization refs; Activity.performedBy and Note.createdBy creator refs; Reminder.user
  </Step>
  
  <Step title="Tasks + Events + Tag (10 checks)">
    Tag.organization; Task.organization, Task.createdBy; TaskComment.task; TaskAssignee.task; TaskTag.tag; Event.organization, Event.createdBy; EventInvitee.event; EventTag.event
  </Step>
  
  <Step title="Additional Coverage (31 checks)">
    Stakeholder + Transfer + Stage histories, Junctions + Commission + DealDocument, Auth sessions, RBAC junctions, Distribution, Escalation, Notifications
  </Step>
</Steps>

<Tip>
Soft-deleted creator references are informational only and do **not** contribute to `totalOrphans`. User churn is normal, and historical records should still display the deleted creator when needed.
</Tip>

### Why Separate from Readiness?

Orphan rows do not affect whether the instance can serve traffic — they are a data-integrity concern, not a service-availability concern. Wiring this into the readiness probe would block rollouts unrelated to the symptom.

## Health Indicators

### PgBoss Health Indicator

Custom health indicator for the PgBoss background job service using the modern `HealthIndicatorService` API:

```typescript
@Injectable()
export class PgBossHealthIndicator {
  constructor(
    @Inject(QUEUE_SERVICE) private readonly queueService: IQueueService,
    private readonly healthIndicatorService: HealthIndicatorService,
  ) {}

  async isHealthy(key: string) {
    const indicator = this.healthIndicatorService.check(key);
    if (!this.queueService.isStarted) {
      return indicator.down({ reason: 'boss.start() has not completed' });
    }
    return indicator.up();
  }
}
```

<Note>
Uses `HealthIndicatorService.check(key)` with `.up()` / `.down()` instead of the legacy `extends HealthIndicator` + `HealthCheckError` pattern that will be removed in the next major Terminus release.
</Note>

### Implementation Details

- Checks `IQueueService.isStarted` flag rather than database queries
- Lightweight check avoiding additional database load
- Returns healthy when PgBoss is properly started and connected

- **MikroOrmHealthIndicator**: Database connectivity verification
- **MemoryHealthIndicator**: Heap memory usage monitoring

## Business Rules

### Health Check Behavior

<Check>
**Liveness checks** always return 200 if the process can handle HTTP requests
</Check>

<Check>
**Readiness checks** return 503 if any critical dependency is unavailable
</Check>

<Check>
All health endpoints are publicly accessible (no authentication required)
</Check>

<Check>
Health endpoints are covered by the global `AppThrottlerGuard`
</Check>

<Check>
Health checks use version-neutral routing
</Check>

### Failure Conditions

<Warning>
Health checks will fail under these conditions:
- Database connection lost or unresponsive
- Memory usage exceeds configured thresholds  
- PgBoss service not started or connection failed
- Any critical service dependency unavailable
</Warning>

## Observability Stack

### Sentry Integration

<Tabs>
  <Tab title="Backend">
    **Setup:**
    - SDK initialized in `src/instrument.ts` (loaded before NestFactory)
    - `SentryModule.forRoot()` + `SentryGlobalFilter` captures unhandled exceptions
    - `LogContextInterceptor` enriches events with user context
    
    **Configuration:**
    - `tracesSampleRate: 0.2` in production
    - Health transactions excluded via `ignoreTransactions`
    - `nodeProfilingIntegration()` for performance profiling
  </Tab>
  
  <Tab title="Frontend">
    **Setup:**
    - Client/server/edge configs for comprehensive coverage
    - `withSentryConfig` wrapper for source map uploads
    - `global-error.tsx` root error boundary
    
    **Configuration:**
    - `replaysOnErrorSampleRate: 1.0` - always captures DOM replay on errors
    - `replaysSessionSampleRate: 0` - no recording of healthy sessions
    - `tunnelRoute: '/monitoring'` - proxies events to avoid ad-blockers
  </Tab>
</Tabs>
