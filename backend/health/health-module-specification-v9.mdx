## Overview

The health module provides comprehensive application health monitoring through standardized health check endpoints. Built on NestJS Terminus, it implements a two-endpoint strategy distinguishing between liveness and readiness probes for container orchestration platforms like Kubernetes.

<Info>
The module monitors critical application dependencies including database connectivity, memory usage, and background job queue status through custom health indicators.
</Info>

The module is registered in `app.module.ts` and provides dedicated health check endpoints separate from the main application controller.

## Architecture

### Design Decisions

- **Two-endpoint strategy**: Separate liveness (`/health/live`) and readiness (`/health/ready`) endpoints following cloud-native best practices
- **Custom health indicators**: Extended Terminus framework with PgBoss job queue monitoring
- **Public endpoints**: Health checks bypass authentication for load balancer and orchestration access
- **Lightweight checks**: Focused on essential service availability without heavy database operations

## API Endpoints

### GET /health/live

**Purpose:** Liveness probe - indicates if the process can handle requests

```typescript
@Get('live')
@PublicEndpoint()
@HealthCheck()
liveness(): Promise<HealthCheckResult>
```

**Response:**
```json
{
  "status": "ok",
  "info": {},
  "error": {},
  "details": {}
}
```

<Tip>
Use for container orchestration restart decisions, load balancer routing, and basic process health verification.
</Tip>

### GET /health/ready

**Purpose:** Readiness probe - indicates if the application can serve traffic

```typescript
@Get('ready')
@PublicEndpoint()
@HealthCheck()
readiness(): Promise<HealthCheckResult>
```

**Health Checks Performed:**
- Database connectivity (MikroORM)
- Memory usage monitoring
- PgBoss job queue status

**Response Format:**
```json
{
  "status": "ok|error",
  "info": {
    "database": { "status": "up" },
    "memory_heap": { "status": "up" },
    "pgboss": { "status": "up" }
  },
  "error": {},
  "details": {
    "database": { "status": "up" },
    "memory_heap": { "status": "up" },
    "pgboss": { "status": "up" }
  }
}
```

## Health Indicators

### PgBossHealthIndicator

Custom health indicator for the PgBoss background job service. Uses the `HealthIndicatorService` API (NestJS Terminus v11+) instead of the deprecated `HealthIndicator` base class.

```typescript
@Injectable()
export class PgBossHealthIndicator {
  constructor(
    @Inject(QUEUE_SERVICE) private readonly queueService: IQueueService,
    private readonly healthIndicatorService: HealthIndicatorService,
  ) {}

  async isHealthy(key: string) {
    const indicator = this.healthIndicatorService.check(key);
    if (!this.queueService.isStarted) {
      return indicator.down({ reason: 'boss.start() has not completed' });
    }
    return indicator.up();
  }
}
```

<Note>
Uses `HealthIndicatorService.check(key)` with `.up()` / `.down()` instead of the legacy `extends HealthIndicator` + `HealthCheckError` pattern.
</Note>

**Implementation Notes:**
- Checks `IQueueService.isStarted` flag rather than database queries
- Lightweight check avoiding additional database load
- Returns healthy when PgBoss is properly started and connected

## Business Rules

### Health Check Behavior

<Steps>
  <Step title="Liveness checks">
    Always return 200 if the process can handle HTTP requests
  </Step>
  <Step title="Readiness checks">
    Return 503 if any critical dependency is unavailable
  </Step>
  <Step title="Public access">
    All health endpoints are publicly accessible (no authentication required)
  </Step>
  <Step title="Version neutral routing">
    Health checks use version-neutral routing
  </Step>
</Steps>

### Failure Conditions

<Warning>
The following conditions will cause health checks to fail:
- Database connection lost or unresponsive
- Memory usage exceeds configured thresholds
- PgBoss service not started or connection failed
- Any critical service dependency unavailable
</Warning>

```typescript

```

## Observability Stack

Beyond health check endpoints, the application uses a layered observability architecture:

### Sentry (Error Tracking & Performance)

<Tabs>
  <Tab title="Backend">
    **SDK Configuration** (`@sentry/nestjs`):
    - SDK initialized in `src/instrument.ts` (loaded before NestFactory)
    - `SentryModule.forRoot()` + `SentryGlobalFilter` as `APP_FILTER` captures all unhandled exceptions
    - `LogContextInterceptor` enriches Sentry events with `Sentry.setUser({ id, email })` and `Sentry.setTag('organization_id', ...)`
    - `tracesSampleRate: 0.2` in production (all error events are captured at 100%)
    - `nodeProfilingIntegration()` for performance profiling
    - Health transactions (`GET /health*`) are excluded via `ignoreTransactions`
  </Tab>
  <Tab title="Frontend">
    **Configuration** (`@sentry/nextjs`):
    - Client config (`sentry.client.config.ts`) with Session Replay and Breadcrumbs
    - Server and Edge configs for SSR/middleware error capture
    - `next.config.ts` wrapped with `withSentryConfig` for source map uploads
    - `global-error.tsx` root error boundary reports to Sentry
    - `AuthProvider` sets `Sentry.setUser()` on login/refresh, `Sentry.setTag('organization_id', ...)` on org selection, and clears user on logout
    - `replaysOnErrorSampleRate: 1.0` — always captures a DOM replay when an error occurs
    - `replaysSessionSampleRate: 0` — no recording of healthy sessions
    - `tunnelRoute: '/monitoring'` — proxies Sentry events to avoid ad-blockers
  </Tab>
</Tabs>

**Environment variables:**
- `SENTRY_DSN` — backend DSN
- `NEXT_PUBLIC_SENTRY_DSN` — frontend DSN (can be same project for correlated traces)
- `SENTRY_AUTH_TOKEN` — optional, for source map uploads during build

### Better Stack (Uptime + Error Logs)

**Uptime monitoring** is configured in the Better Stack dashboard (no code) — pings `GET /health/ready` every 30–60 seconds.

**Logtail log shipping** (`@logtail/pino`) — in production with `BETTERSTACK_SOURCE_TOKEN` set:

<Check>
Multi-transport: stdout at `info` level (for Render log viewer) + Logtail at `warn` level (only 4xx/5xx reach Better Stack)
</Check>

- Keeps log volume and cost low while preserving error searchability
- Without the token, only stdout (JSON) is used

**Environment variables:**
- `BETTERSTACK_SOURCE_TOKEN` — Logtail source token (production only)

### Pino Structured Logging

Configured in `app.module.ts` via `nestjs-pino` (`LoggerModule`):

| Field | Source | When |
|---|---|---|
| `requestId` | CLS (`ClsService.getId()`) | Every request (middleware layer) |
| `userId` | `LogContextInterceptor` (`logger.assign()`) | After auth guard resolves `request.user` |
| `organizationId` | `LogContextInterceptor` | After auth guard resolves `request.user` |

**Log level classification** (`customLogLevel`):
- `error` — `statusCode >= 500` or `err` present
- `warn` — `statusCode >= 400`
- `info` — all other requests

**Serializers:**
- `req` — `id`, `method`, `url` (minimal, sensitive headers redacted)
- `res` — `statusCode`
- `err` — `type`, `message`, `stack` (captures exception details for log-level diagnosis)

<Warning>
**Redaction:** Authorization header, cookies, password fields, and tokens are replaced with `[REDACTED]`.
</Warning>

**Auto-logging exclusions:** `/health`, `/health/live`, `/health/ready` are excluded from automatic request logging.

### Observability Role Summary

| Tool | Role | What it captures |
|---|---|---|
| **Sentry** | Error tracking + performance + session replay | Stack traces, user context, DOM replay, correlated frontend↔backend traces |
| **Better Stack Uptime** | "Is the app alive?" | Pings `/health/ready`, alerts on downtime |
| **Better Stack Logtail** | Error log archive | 4xx/5xx request logs with `userId`, `organizationId`, `requestId`, error details |
| **Render stdout** | Ad-hoc debugging | All pino JSON logs (info+) visible in Render log viewer |
