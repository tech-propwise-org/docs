The health module provides comprehensive application health monitoring through standardized health check endpoints. Built on NestJS Terminus, it implements a two-endpoint strategy distinguishing between liveness and readiness probes for container orchestration platforms like Kubernetes.

<Note>
The module monitors critical application dependencies including database connectivity, memory usage, and background job queue status through custom health indicators.
</Note>

## Architecture

### Design Decisions

The health module follows cloud-native best practices with several key architectural decisions:

- **Two-endpoint strategy**: Separate liveness (`/health/live`) and readiness (`/health/ready`) endpoints
- **Custom health indicators**: Extended Terminus framework with PgBoss job queue monitoring
- **Public endpoints**: Health checks bypass authentication for load balancer and orchestration access
- **Rate limited public surface**: Health checks use the app-wide throttler to reduce abuse risk
- **Lightweight checks**: Focused on essential service availability without heavy database operations

## Data Integrity Audit

<Warning>
The data integrity audit is **not wired into `/health/ready`** and has no public HTTP endpoint. It runs as a scheduled background job to avoid exposing an unauthenticated, high-cost route.
</Warning>

### Overview

The daily queued job runs at `30 3 * * *` by default (configurable via `DATA_INTEGRITY_AUDIT_CRON`) through `DataIntegrityAuditHandler`. It detects orphan rows resulting from soft-delete cascade gaps.

<Steps>
  <Step title="Scheduled Execution">
    Job runs daily at 3:30 AM (configurable via environment variable)
  </Step>
  <Step title="Audit Categories">
    Executes ~74 count queries grouped into module buckets
  </Step>
  <Step title="Logging & Alerting">
    Logs every run, emits warn-level logs for unhealthy orphans, sends Sentry error events when `totalOrphans > 0`
  </Step>
</Steps>

  <Accordion title="Core Entities (12 checks)">

## Health Indicators

### PgBossHealthIndicator

Custom health indicator for the PgBoss background job service using the modern NestJS Terminus v11+ API:

```typescript
@Injectable()
export class PgBossHealthIndicator {
  constructor(
    @Inject(QUEUE_SERVICE) private readonly queueService: IQueueService,
    private readonly healthIndicatorService: HealthIndicatorService,
  ) {}

  async isHealthy(key: string) {
    const indicator = this.healthIndicatorService.check(key);
    if (!this.queueService.isStarted) {
      return indicator.down({ reason: 'boss.start() has not completed' });
    }
    return indicator.up();
  }
}
```

<Tip>
Uses `HealthIndicatorService.check(key)` with `.up()` / `.down()` instead of the legacy `extends HealthIndicator` + `HealthCheckError` pattern, which is deprecated in Terminus v11+.
</Tip>

**Implementation Notes:**
- Checks `IQueueService.isStarted` flag rather than database queries
- Lightweight check avoiding additional database load
- Returns healthy when PgBoss is properly started and connected

## Business Rules

### Health Check Behavior

1. **Liveness checks** always return 200 if the process can handle HTTP requests
2. **Readiness checks** return 503 if any critical dependency is unavailable
3. All health endpoints are publicly accessible (no authentication required)
4. Health endpoints are covered by the global `AppThrottlerGuard`
5. Health checks use version-neutral routing

### Response Codes

<Tabs>
  <Tab title="Success">
    **HTTP 200**: All checks passing
  </Tab>
  <Tab title="Service Unavailable">
    **HTTP 503**: One or more checks failing
  </Tab>
  <Tab title="Rate Limited">
    **HTTP 429**: Request exceeds the global throttling policy
  </Tab>
</Tabs>

### Failure Conditions

- Database connection lost or unresponsive
- Memory usage exceeds configured thresholds
- PgBoss service not started or connection failed
- Any critical service dependency unavailable

## Observability Stack

Beyond health check endpoints, the application uses a layered observability architecture:

### Sentry (Error Tracking & Performance)

<Tabs>
  <Tab title="Backend">
    **@sentry/nestjs Integration:**
    - SDK initialized in `src/instrument.ts` (loaded before NestFactory)
    - `SentryModule.forRoot()` + `SentryGlobalFilter` captures all unhandled exceptions
    - `LogContextInterceptor` enriches events with user context
    - `DataIntegrityAuditHandler` emits grouped error events for unhealthy orphan rows
    - `tracesSampleRate: 0.2` in production
    - Health transactions excluded via `ignoreTransactions`
  </Tab>
  
  <Tab title="Frontend">
    **@sentry/nextjs Integration:**
    - Client config with Session Replay and Breadcrumbs
    - Server and Edge configs for SSR/middleware error capture
    - `next.config.ts` wrapped with `withSentryConfig`
    - `global-error.tsx` root error boundary
    - `AuthProvider` sets user context
    - `tunnelRoute: '/monitoring'` to avoid ad-blockers
  </Tab>
</Tabs>

**Environment Variables:**
- `SENTRY_DSN` - backend DSN
- `NEXT_PUBLIC_SENTRY_DSN` - frontend DSN
- `SENTRY_AUTH_TOKEN` - optional, for source map uploads

## Patterns and Conventions

### Endpoint Naming

- `/health/live` - Liveness probe
- `/health/ready` - Readiness probe
- Version neutral routing (`VERSION_NEUTRAL`)

### Error Handling

<CodeGroup>
```typescript Modern Pattern (v11+)
// Health check failures use indicator.down()
const indicator = this.healthIndicatorService.check(key);
return indicator.down({ reason: 'Service not started' });
```

```typescript Legacy Pattern (Deprecated)
// The legacy HealthCheckError + extends HealthIndicator pattern
// is deprecated and will be removed in the next major Terminus release
throw new HealthCheckError('Check failed', {});
```
</CodeGroup>
