## Overview

The health module provides comprehensive application health monitoring through standardized health check endpoints. Built on NestJS Terminus, it implements a two-endpoint strategy distinguishing between liveness and readiness probes for container orchestration platforms like Kubernetes.

<Info>
The module monitors critical application dependencies including database connectivity, memory usage, and background job queue status through custom health indicators.
</Info>

The module is registered in `app.module.ts` and provides dedicated health check endpoints separate from the main application controller.

## Architecture

### Design Decisions

The health module follows several key architectural principles:

<CardGroup cols={2}>
  <Card title="Two-endpoint Strategy" icon="heart-pulse">

  </Card>
  <Card title="Custom Health Indicators" icon="stethoscope">

  </Card>
  <Card title="Public Endpoints" icon="unlock">
    Health checks bypass authentication for load balancer and orchestration access

  <Card title="Lightweight Checks" icon="feather">

### Dependencies

The health module relies on the following key dependencies:

- **`@nestjs/terminus`** - Core health check framework
- **`PgBossService`** - Background job queue monitoring
- **`MikroORM`** - Database connectivity checks
- **Memory monitoring** - Resource usage tracking

## API Endpoints

### GET /health/live

**Purpose:** Liveness probe - indicates if the process can handle requests

```typescript
@Get('live')
@PublicEndpoint()
@HealthCheck()
liveness(): Promise<HealthCheckResult>
```

<Tabs>
  <Tab title="Response">
    ```json
    {
      "status": "ok",
      "info": {},
      "error": {},
      "details": {}
    }
    ```
  </Tab>
  <Tab title="Use Cases">
    - Container orchestration restart decisions
    - Load balancer routing decisions
    - Basic process health verification
  </Tab>
</Tabs>

### GET /health/ready

**Purpose:** Readiness probe - indicates if the application can serve traffic

```typescript
@Get('ready')
@PublicEndpoint()
@HealthCheck()
readiness(): Promise<HealthCheckResult>
```

<Note>
The readiness endpoint performs comprehensive health checks including database connectivity, memory usage monitoring, and PgBoss job queue status.
</Note>

#### Health Checks Performed

<Steps>
  <Step title="Database Connectivity">
    Verifies MikroORM database connection is active and responsive
  </Step>
  <Step title="Memory Usage Monitoring">
    Checks heap memory usage against configured thresholds
  </Step>
  <Step title="PgBoss Job Queue Status">
    Validates background job service is started and connected
  </Step>
</Steps>

#### Response Format

```json
{
  "status": "ok|error",
  "info": {
    "database": { "status": "up" },
    "memory_heap": { "status": "up" },
    "pgboss": { "status": "up" }
  },
  "error": {},
  "details": {
    "database": { "status": "up" },
    "memory_heap": { "status": "up" },
    "pgboss": { "status": "up" }
  }
}
```

## Health Indicators

### PgBossHealthIndicator

Custom health indicator for the PgBoss background job service.

```typescript
@Injectable()
export class PgBossHealthIndicator extends HealthIndicator {
  isHealthy(key: string): Promise<HealthIndicatorResult>
}
```

<Tip>
The PgBoss health indicator checks the `PgBossService.isStarted` flag rather than performing database queries, keeping the check lightweight and avoiding additional database load.
</Tip>

#### Implementation Details

- Returns healthy when PgBoss is properly started and connected
- Avoids heavy database operations for performance
- Provides quick service availability verification

## Business Rules

### Health Check Behavior

<Check>
**Liveness checks** always return 200 if the process can handle HTTP requests
</Check>

<Warning>
**Readiness checks** return 503 if any critical dependency is unavailable
</Warning>

Additional behavioral rules:

- All health endpoints are publicly accessible (no authentication required)
- Health checks use version-neutral routing
- Consistent response format following Terminus conventions

### Failure Conditions

The health module will report failures under the following conditions:

- Database connection lost or unresponsive
- Memory usage exceeds configured thresholds
- PgBoss service not started or connection failed
- Any critical service dependency unavailable

## Patterns and Conventions

### Endpoint Naming

<CodeGroup>
```typescript Liveness
// GET /health/live
@Get('live')
@PublicEndpoint()
@HealthCheck()
```

```typescript Readiness
// GET /health/ready  
@Get('ready')
@PublicEndpoint()
@HealthCheck()
```
</CodeGroup>

All endpoints use version neutral routing (`VERSION_NEUTRAL`) to ensure consistent access regardless of API versioning.

### Response Standards

- **HTTP 200**: All checks passing
- **HTTP 503**: One or more checks failing
- Consistent JSON response format following Terminus conventions

### Error Handling

Health check failures are handled through structured error responses:

```typescript
// Health check failures throw HealthCheckError
throw new HealthCheckError('PgBoss check failed', {
  pgboss: { status: 'down', message: 'Service not started' }
});
```
