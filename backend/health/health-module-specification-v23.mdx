## Overview

The health module provides comprehensive application health monitoring through standardized health check endpoints. Built on NestJS Terminus, it implements a two-endpoint strategy distinguishing between liveness and readiness probes for container orchestration platforms like Kubernetes.

<Info>
The module monitors critical application dependencies including database connectivity, memory usage, and background job queue status through custom health indicators.
</Info>

The module is registered in `app.module.ts` and provides dedicated health check endpoints separate from the main application controller.

**Last Updated:** 2026-05-17  
**Status:** Active

### Scheduled Data-Integrity Audit

<Warning>
The data integrity audit is **not wired into `/health/ready`** and has no public `/health/data-integrity` endpoint. This is intentional to separate data integrity concerns from service availability.
</Warning>

**Purpose:** Detect orphan rows resulting from soft-delete cascade gaps without exposing an unauthenticated, high-cost HTTP route.

The daily queued job runs at `30 3 * * *` by default (override via `DATA_INTEGRITY_AUDIT_CRON`) through `DataIntegrityAuditHandler`. It logs every run, emits a warn-level application log when unhealthy orphans are found, and sends a grouped Sentry error event when `totalOrphans > 0`.

## Health Indicators

### PgBossHealthIndicator

Custom health indicator for the PgBoss background job service using the `HealthIndicatorService` API (NestJS Terminus v11+).

```typescript
@Injectable()
export class PgBossHealthIndicator {
  constructor(
    @Inject(QUEUE_SERVICE) private readonly queueService: IQueueService,
    private readonly healthIndicatorService: HealthIndicatorService,
  ) {}

  async isHealthy(key: string) {
    const indicator = this.healthIndicatorService.check(key);
    if (!this.queueService.isStarted) {
      return indicator.down({ reason: 'boss.start() has not completed' });
    }
    return indicator.up();
  }
}
```

<Check>
Uses `HealthIndicatorService.check(key)` with `.up()` / `.down()` instead of the legacy `extends HealthIndicator` + `HealthCheckError` pattern
</Check>

**Implementation Notes:**
- Checks `IQueueService.isStarted` flag rather than database queries
- Lightweight check avoiding additional database load
- Returns healthy when PgBoss is properly started and connected

## Business Rules

### Health Check Behavior

<Steps>
  <Step title="Liveness Check Processing">
    Liveness checks always return 200 if the process can handle HTTP requests
  </Step>
  <Step title="Readiness Check Processing">
    Readiness checks return 503 if any critical dependency is unavailable
  </Step>
  <Step title="Access Control">
    All health endpoints are publicly accessible (no authentication required) but covered by global throttling
  </Step>
  <Step title="Routing">
    Health checks use version-neutral routing
  </Step>
</Steps>

### Failure Conditions

- Database connection lost or unresponsive
- Memory usage exceeds configured thresholds
- PgBoss service not started or connection failed
- Any critical service dependency unavailable

```typescript

const indicator = this.healthIndicatorService.check(key);
return indicator.down({ reason: 'Service not started' });

// and will be removed in the next major Terminus release.
```

### Sentry (Error Tracking & Performance)

<Tabs>
  <Tab title="Backend Configuration">
    **Setup**: `@sentry/nestjs` SDK initialized in `src/instrument.ts`
    
    **Integration**: `SentryModule.forRoot()` + `SentryGlobalFilter` as `APP_FILTER`
    
    **Context Enrichment**: `LogContextInterceptor` enriches events with user/org data
    
    **Sampling**: `tracesSampleRate: 0.2` in production, health transactions excluded
    
    **Profiling**: `nodeProfilingIntegration()` for performance analysis
  </Tab>
  <Tab title="Frontend Configuration">
    **Setup**: `@sentry/nextjs` with Session Replay and Breadcrumbs
    
    **Coverage**: Client, server, and edge configs for complete error capture
    
    **Error Boundaries**: `global-error.tsx` root boundary reports to Sentry
    
    **Session Replay**: 100% on errors, 0% on healthy sessions
    
    **Tunneling**: `/monitoring` route to avoid ad-blockers
  </Tab>
  <Tab title="Environment Variables">
    ```bash
    SENTRY_DSN=                    # Backend DSN
    NEXT_PUBLIC_SENTRY_DSN=        # Frontend DSN
    SENTRY_AUTH_TOKEN=             # Optional, for source map uploads
    ```
  </Tab>
</Tabs>
