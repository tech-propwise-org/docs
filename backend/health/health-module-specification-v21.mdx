## Overview

The health module provides comprehensive application health monitoring through standardized health check endpoints. Built on NestJS Terminus, it implements a two-endpoint strategy distinguishing between liveness and readiness probes for container orchestration platforms like Kubernetes.

<Info>
The module monitors critical application dependencies including database connectivity, memory usage, and background job queue status through custom health indicators.
</Info>

The module is registered in `app.module.ts` and provides dedicated health check endpoints separate from the main application controller.

## Architecture

### Design Decisions

The health module follows cloud-native best practices with these key architectural decisions:

<CardGroup cols={2}>
  <Card title="Two-Endpoint Strategy" icon="heart-pulse">
    Separate liveness (`/health/live`) and readiness (`/health/ready`) endpoints for optimal orchestration
  </Card>
  <Card title="Custom Health Indicators" icon="stethoscope">
    Extended Terminus framework with PgBoss job queue monitoring capabilities
  </Card>
  <Card title="Public Endpoints" icon="globe">
    Health checks bypass authentication for load balancer and orchestration access
  </Card>
  <Card title="Rate Limited" icon="shield">
    Health checks use app-wide throttler to reduce abuse risk while remaining reachable
  </Card>
</CardGroup>

<Note>
Lightweight checks are focused on essential service availability without heavy database operations.
</Note>

## Data Integrity Auditing

### Scheduled Data-Integrity Audit

<Warning>
This audit is **not wired into `/health/ready`** and has no public `/health/data-integrity` endpoint. It's a separate concern from service availability.
</Warning>

The daily queued job runs at `30 3 * * *` by default (override via `DATA_INTEGRITY_AUDIT_CRON`) through `DataIntegrityAuditHandler`. It detects orphan rows resulting from soft-delete cascade gaps without exposing an unauthenticated, high-cost HTTP route.

<Steps>
  <Step title="Audit Execution">
    Runs ~74 count queries grouped into module buckets with parallel `Promise.all` execution
  </Step>
  <Step title="Logging">
    Logs every run and emits warn-level application log when unhealthy orphans are found
  </Step>
  <Step title="Alerting">
    Sends grouped Sentry error event when `totalOrphans > 0`
  </Step>
</Steps>

## Health Indicators

### PgBossHealthIndicator

Custom health indicator for the PgBoss background job service using the modern `HealthIndicatorService` API.

```typescript
@Injectable()
export class PgBossHealthIndicator {
  constructor(
    @Inject(QUEUE_SERVICE) private readonly queueService: IQueueService,
    private readonly healthIndicatorService: HealthIndicatorService,
  ) {}

  async isHealthy(key: string) {
    const indicator = this.healthIndicatorService.check(key);
    if (!this.queueService.isStarted) {
      return indicator.down({ reason: 'boss.start() has not completed' });
    }
    return indicator.up();
  }
}
```

<Tip>
Uses `HealthIndicatorService.check(key)` with `.up()` / `.down()` instead of the legacy `extends HealthIndicator` pattern which is deprecated in NestJS Terminus v11+.
</Tip>

**Implementation Notes:**
- Checks `IQueueService.isStarted` flag rather than database queries
- Lightweight check avoiding additional database load
- Returns healthy when PgBoss is properly started and connected

## Business Rules

### Health Check Behavior

<Check>
**Liveness checks** always return 200 if the process can handle HTTP requests
</Check>

<Check>
**Readiness checks** return 503 if any critical dependency is unavailable
</Check>

<Check>
All health endpoints are publicly accessible (no authentication required)
</Check>

<Check>
Health endpoints are covered by the global `AppThrottlerGuard`
</Check>

<Check>
Health checks use version-neutral routing
</Check>

### Failure Conditions

- Database connection lost or unresponsive
- Memory usage exceeds configured thresholds
- PgBoss service not started or connection failed
- Any critical service dependency unavailable

## Patterns and Conventions

### Endpoint Naming

- `/health/live` - Liveness probe
- `/health/ready` - Readiness probe
- Version neutral routing (`VERSION_NEUTRAL`)

### Response Standards

| Status Code | Condition | Description |
|-------------|-----------|-------------|
| HTTP 200 | All checks passing | Healthy state |
| HTTP 503 | One or more checks failing | Unhealthy state |
| HTTP 429 | Rate limit exceeded | Throttled request |

All responses follow consistent JSON format using Terminus conventions.

### Error Handling

```typescript
// Modern health check pattern (NestJS Terminus v11+)
const indicator = this.healthIndicatorService.check(key);
return indicator.down({ reason: 'Service not started' });

// Legacy pattern (deprecated)
// throw new HealthCheckError('Service not started', result);
```

## Observability Stack

Beyond health check endpoints, the application uses a layered observability architecture:

### Sentry (Error Tracking & Performance)

<Tabs>
  <Tab title="Backend">
    **Implementation** (`@sentry/nestjs`):
    - SDK initialized in `src/instrument.ts` (loaded before NestFactory)
    - `SentryModule.forRoot()` + `SentryGlobalFilter` captures unhandled exceptions
    - `LogContextInterceptor` enriches events with user context
    - `DataIntegrityAuditHandler` emits grouped error events for orphan detection
    
    **Configuration:**
    - `tracesSampleRate: 0.2` in production
    - `nodeProfilingIntegration()` for performance profiling
    - Health transactions excluded via `ignoreTransactions`
  </Tab>

  <Tab title="Frontend">
    **Implementation** (`@sentry/nextjs`):
    - Client config with Session Replay and Breadcrumbs
    - Server and Edge configs for SSR/middleware error capture
    - `withSentryConfig` wrapper for source map uploads
    - `global-error.tsx` root error boundary
    
    **Configuration:**
    - `replaysOnErrorSampleRate: 1.0` - always captures DOM replay on errors
    - `replaysSessionSampleRate: 0` - no recording of healthy sessions
    - `tunnelRoute: '/monitoring'` - proxies events to avoid ad-blockers
  </Tab>
</Tabs>

**Environment Variables:**
- `SENTRY_DSN` - Backend DSN
- `NEXT_PUBLIC_SENTRY_DSN` - Frontend DSN
- `SENTRY_AUTH_TOKEN` - Optional, for source map uploads
