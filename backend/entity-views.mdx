## Overview

The CRM module exposes Kanban and List view APIs for **leads**, **deals**, **contacts**, **companies**, and **commission payments**. Every entity follows the same structural patterns, making it straightforward to build generic frontend components.

<Info>
**Source of truth:** `Docs/CRM_ENTITY_VIEWS.md` in the backend repository.
</Info>

### Design principles

1. **Flexible `type` parameter** — The `type` query parameter is optional; omitting it returns all available tabs.
2. **Unified data structure** — All APIs use the same `TabData` interface.
3. **Consistent patterns** — All entity APIs follow identical structural patterns (flat response, `availableTeams`, parallel fetching).
4. **Performance optimized** — Parallel fetching, SQL `EXISTS` subqueries, and DB-level pagination.
5. **`availableTeams` metadata** — When no `type` is specified, the response includes `availableTeams` so the frontend knows which teams are accessible.

## Response structure

### TabData interface

All view APIs use this common data structure:

```typescript TabData
interface TabData {
  data: EntityDto[];
  total: number;
  page: number;
  limit: number;
  totalPages: number;
  hasMore?: boolean;
}
```

### Kanban responses

Kanban responses nest data by stage. Staged views return `Record<stageId, TabData>`, while pool views return `TabData` directly.

<Tabs>
<Tab title="Lead Kanban">
```typescript
{
  all?: Record<stageId, ListTabData>,
  my?: Record<stageId, ListTabData>,
  team?: Record<teamId, Record<stageId, ListTabData>>,
  pool?: ListTabData,
  teamPool?: Record<teamId, ListTabData>,
  availableTeams?: Array<{ id: string; name: string }>
}
```
</Tab>

<Tab title="Deal Kanban">
```typescript
{
  all?: Record<stageId, DealTabData>,
  my?: Record<stageId, DealTabData>,
  team?: Record<teamId, Record<stageId, DealTabData>>,
  availableTeams?: Array<{ id: string; name: string }>
}
```

<Warning>
Deals do **not** have `pool` or `teamPool` views. Only leads have pool views.
</Warning>
</Tab>
</Tabs>

When a `type` parameter is provided, the response contains only that specific tab (no `availableTeams`).

### Kanban vs list differences

| Feature | Kanban | List |
| --- | --- | --- |
| Stage nesting | `Record<stageId, TabData>` for staged tabs | `TabData` directly |
| Team nesting | `Record<teamId, Record<stageId, TabData>>` | `Record<teamId, TabData>` |
| Pool nesting | `TabData` directly | `TabData` directly |

## View types

### Lead view types

| Type | Description | Has stages | Team nested |
| --- | --- | --- | --- |
| `all` | All organization leads | Yes | No |
| `my` | Leads assigned to current user | Yes | No |
| `team` | Team leads | Yes | Yes |
| `pool` | Unassigned organization leads | No | No |
| `teamPool` | Unassigned team leads | No | Yes |

### Deal view types

| Type | Description | Has stages | Team nested |
| --- | --- | --- | --- |
| `all` | All organization deals | Yes | No |
| `my` | Deals assigned to current user | Yes | No |
| `team` | Team deals | Yes | Yes |

### Contact, company, and commission payment view types

These entities share the same view type structure — `all`, `my`, `team` — with no stages and no pool views.

| Type | Description | Team nested |
| --- | --- | --- |
| `all` | All organization records | No |
| `my` | Records linked to my leads/deals | No |
| `team` | Records linked to team leads/deals | Yes |

### Archived items behavior

Both lead and deal list APIs support a `showArchived` query parameter:

- **`showArchived` is `false` or omitted (default)** — returns only active (non-archived) items. Default sort is `createdAt DESC`.
- **`showArchived` is `true`** — returns only archived items. Default sort switches to `archivedAt DESC`.

<Note>
This is a mutually exclusive filter — you see either active or archived items, never both in the same response.
</Note>

## Pool and team pool behavior

Pool views apply to **leads only**. Pool and team pool leads do not have stages — they exist in a "New" stage by default and cannot be viewed in staged views until they are claimed by a user or assigned to a team.

The response structure reflects this: `pool` and `teamPool` return `TabData` directly, not `Record<stageId, TabData>`.

</Accordion>

Returns a single stage for "load more" pagination.
</Accordion>

<Accordion title="type = team (no teamId) — all accessible teams">
```
GET /leads?type=team&limit=10
GET /deals?type=team&limit=10
```

Returns all teams the user has access to, with stages nested under each team.
</Accordion>

<Accordion title="type = team + teamId — specific team">
```
GET /leads?type=team&teamId=uuid&limit=10
```

Returns a specific team with all its stages.
</Accordion>

<Accordion title="type = team + teamId + stageId — team stage pagination">

## Permissions and access control

### Lead view permissions

| View type | Required permission | Notes |
| --- | --- | --- |
| `all` | `system.admin`, `crm.view`, or `crm.manage` | Organization-wide view |
| `my` | Any authenticated user | Leads the user created or is a stakeholder on |
| `team` | `system.admin`, `crm.view`, `crm.manage`, `team_crm.view`, or `team_crm.manage` | Org-level permissions see all teams; team-level see only permitted teams |
| `pool` | Any authenticated user | Unassigned leads at organization level |
| `teamPool` | Team membership (no specific permission) | Any team member can see their team's unassigned leads |

### Deal view permissions

| View type | Required permission | Notes |
| --- | --- | --- |
| `all` | `system.admin`, `sales.view`, or `sales.manage` | Organization-wide view |
| `my` | Any authenticated user | Deals the user is a stakeholder on |
| `team` | `system.admin`, `sales.view`, `sales.manage`, `team_sales.view`, or `team_sales.manage` | Org-level permissions see all teams; team-level see only permitted teams |

### Contact and company view permissions

| View type | Required permission | Notes |
| --- | --- | --- |
| `all` | `system.admin`, `crm.view`, or `crm.manage` | Organization-wide view |
| `my` | Any authenticated user | Records linked to the user's leads/deals |
| `team` | `system.admin`, `crm.view`, `crm.manage`, `team_crm.view`, or `team_crm.manage` | Org-level permissions see all teams; team-level see only permitted teams |

| View type | Required permission | Notes |
| --- | --- | --- |
| `all` | `system.admin`, `sales.view`, or `sales.manage` | Organization-wide view |
| `my` | Any authenticated user | Payments where the user is the recipient |
| `team` | `system.admin`, `sales.view`, `sales.manage`, `team_sales.view`, or `team_sales.manage` | Org-level permissions see all teams; team-level see only permitted teams |

### Key access control rules

<Warning>
**Team vs team pool access** — The `team` tab requires explicit `team_crm.view` or `team_crm.manage` permissions, but the `teamPool` tab is available to **any team member** regardless of permissions. This allows agents to see and claim leads from their team's pool without needing CRM management permissions.
</Warning>

**Team pool visibility rules:**

- Uses `getAllMemberTeams()` to show all teams the user belongs to.
- A lead appears in **Team X's pool** when Team X has a team-only stakeholder row (`team_id=X`, `user_id IS NULL`) **and** no stakeholder row exists with `team_id=X` and `user_id IS NOT NULL`.
- A user holding the lead with `team_id=NULL` (global pool claim) or `team_id=Y` (different team) does **not** remove the lead from Team X's pool.
- Only a user row with `team_id=X` removes the lead from Team X's pool.

### SQL access control patterns

| View type | SQL filter |
| --- | --- |
| `all` | No filter (org-wide access validated by permission check) |
| `my` | `lead.created_by_id = user_id` OR `EXISTS (stakeholder with user_id)` |
| `pool` | `NOT EXISTS (any stakeholder)` |
| `team` | `EXISTS (stakeholder with team_id = ?)` |
| `teamPool` | `EXISTS (stakeholder with team_id=? AND user_id IS NULL) AND NOT EXISTS (stakeholder with team_id=? AND user_id IS NOT NULL)` |

### Contact access control

- **ALL** — No filter (user has org-wide access).
- **MY** — Contact's person has a lead where the user is a stakeholder, or the contact is linked to a deal where the user is a stakeholder.
- **TEAM** — Contact's person has a lead where the team is a stakeholder, or the contact is linked to a deal where the team is a stakeholder.

### Company access control

- **ALL** — No filter (user has org-wide access).
- **MY** — Company has a `ContactCompanyRole` linking to a contact whose person has a lead where the user is a stakeholder, or to a contact linked to a deal where the user is a stakeholder.
- **TEAM** — Same as MY but with `team_id` instead of `user_id` in stakeholder checks.

### Commission payment access control

- **ALL** — No filter (user has org-wide access).
- **MY** — `user_id = current_user_id`.
- **TEAM** — `EXISTS (SELECT 1 FROM entity_stakeholder WHERE entity_type='deal' AND team_id=?)`.

## Performance optimizations

<Tip>
Use the `type` parameter to fetch only the tab the user is viewing. This avoids fetching all tabs in parallel and reduces load.
</Tip>

1. **Parallel tab fetching** — All tabs are fetched in parallel on initial load when no `type` is specified.
2. **SQL EXISTS subqueries** — Prevent N+1 queries for access control checks.
3. **DB-level pagination** — Uses `LIMIT`/`OFFSET` for efficient pagination.
4. **Selective fetching** — When `type` is provided, only that tab is fetched.
