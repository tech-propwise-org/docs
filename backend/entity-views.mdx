## Overview

This document describes the architecture and implementation of the Kanban and List view APIs for **Leads**, **Deals**, **Contacts**, **Companies**, and **Commission Payments** in the CRM module. All entity types follow the same structural patterns for consistency, making it easier to build generic frontend components.

<Info>
  **Source of truth:** `Docs/CRM_ENTITY_VIEWS.md` in the backend repository.
</Info>

### Key design principles

1. **Flexible `type` parameter** â€” The `type` query parameter is optional; when omitted, all available tabs are returned
2. **Unified data structure** â€” All APIs use the same `TabData` interface
3. **Consistent patterns** â€” All entity APIs follow identical structural patterns (flat response, `availableTeams`, parallel fetching)
4. **Performance optimized** â€” Parallel fetching, SQL EXISTS subqueries, and DB-level pagination
5. **`availableTeams` metadata** â€” When no `type` is specified, the response includes `availableTeams` so the frontend knows which teams are accessible

## Response structure

### TabData interface

Both Kanban and List APIs use the same data structure for tab data:

```typescript TabData
interface TabData {
  data: EntityDto[]; // Array of leads or deals
  total: number; // Total count
  page: number; // Current page
  limit: number; // Items per page
  totalPages: number; // Total pages
  hasMore?: boolean; // Whether more pages exist
}
```

### Lead Kanban response

**Without `type` parameter** â€” Returns all available tabs:

```typescript
{
  all?: Record<stageId, ListTabData>,           // If user has CRM permissions
  my?: Record<stageId, ListTabData>,            // Always present
  team?: Record<teamId, Record<stageId, ListTabData>>, // If has team access
  pool?: ListTabData,                            // If has pool access (no stages)
  teamPool?: Record<teamId, ListTabData>,        // If belongs to teams (no stages)
  availableTeams?: Array<{ id: string; name: string }> // Teams the user can access
}
```

**With `type` parameter** â€” Returns only that specific tab with the same nested structure (no `availableTeams`).

### Deal Kanban response

**Without `type` parameter** â€” Returns all available tabs:

```typescript
{
  all?: Record<stageId, DealTabData>,           // If user has sales permissions
  my?: Record<stageId, DealTabData>,            // Always present
  team?: Record<teamId, Record<stageId, DealTabData>>, // If has team access
  availableTeams?: Array<{ id: string; name: string }> // Teams the user can access
}
```

**With `type` parameter** â€” Returns only that specific tab with the same nested structure (no `availableTeams`).

<Warning>
Deals do **NOT** have `pool` or `teamPool` views. Only leads have these views.
</Warning>

### Archived items behavior

Both lead and deal list APIs support a `showArchived` query parameter:

- **`showArchived` is `false` or omitted (default)** â€” returns only active (non-archived) items. Default sort is `createdAt DESC`.
- **`showArchived` is `true`** â€” returns only archived items. Default sort switches to `archivedAt DESC`.

<Note>
  This is a mutually exclusive filter â€” you see either active or archived items,
  never both in the same response.
</Note>

## Pool and team pool behavior

Pool views apply to **leads only**. Pool and team pool leads do not have stages â€” they exist in a "New" stage by default and cannot be viewed in staged views until they are claimed by a user or assigned to a team.

The response structure reflects this: `pool` and `teamPool` return `TabData` directly, not `Record<stageId, TabData>`.

## Permissions and access control

### Lead view permissions

| View type  | Required permission                                                             | Notes                                                                    |
| ---------- | ------------------------------------------------------------------------------- | ------------------------------------------------------------------------ |
| `all`      | `system.admin`, `crm.view`, or `crm.manage`                                     | Organization-wide view                                                   |
| `my`       | Any authenticated user                                                          | Leads the user created or is a stakeholder on                            |
| `team`     | `system.admin`, `crm.view`, `crm.manage`, `team_crm.view`, or `team_crm.manage` | Org-level permissions see all teams; team-level see only permitted teams |
| `pool`     | Any authenticated user                                                          | Unassigned leads at organization level                                   |
| `teamPool` | Team membership (no specific permission)                                        | Any team member can see their team's unassigned leads                    |

### Key access control rules

<Warning>
  **Team vs team pool access** â€” The `team` tab requires explicit
  `team_crm.view` or `team_crm.manage` permissions, but the `teamPool` tab is
  available to **any team member** regardless of permissions. This allows agents
  to see and claim leads from their team's pool without needing CRM management
  permissions.
</Warning>

**Team pool visibility rules:**

- Uses `getAllMemberTeams()` to show all teams the user belongs to.
- A lead appears in **Team X's pool** when Team X has a team-only stakeholder row (`team_id=X`, `user_id IS NULL`) **and** no stakeholder row exists with `team_id=X` and `user_id IS NOT NULL`.
- A user holding the lead with `team_id=NULL` (global pool claim) or `team_id=Y` (different team) does **not** remove the lead from Team X's pool.
- Only a user row with `team_id=X` removes the lead from Team X's pool.

### SQL access control patterns

| View type  | SQL filter                                                                                                                    |
| ---------- | ----------------------------------------------------------------------------------------------------------------------------- |
| `all`      | No filter (org-wide access validated by permission check)                                                                     |
| `my`       | `lead.created_by_id = user_id` OR `EXISTS (stakeholder with user_id)`                                                         |
| `pool`     | `NOT EXISTS (any stakeholder)`                                                                                                |
| `team`     | `EXISTS (stakeholder with team_id = ?)`                                                                                       |
| `teamPool` | `EXISTS (stakeholder with team_id=? AND user_id IS NULL) AND NOT EXISTS (stakeholder with team_id=? AND user_id IS NOT NULL)` |

### Contact access control

- **ALL** â€” No filter (user has org-wide access).
- **MY** â€” Contact's person has a lead where the user is a stakeholder, or the contact is linked to a deal where the user is a stakeholder.
- **TEAM** â€” Contact's person has a lead where the team is a stakeholder, or the contact is linked to a deal where the team is a stakeholder.

### Company access control

- **ALL** â€” No filter (user has org-wide access).
- **MY** â€” Company has a `ContactCompanyRole` linking to a contact whose person has a lead where the user is a stakeholder, or to a contact linked to a deal where the user is a stakeholder.
- **TEAM** â€” Same as MY but with `team_id` instead of `user_id` in stakeholder checks.

### Commission payment access control

- **ALL** â€” No filter (user has org-wide access).
- **MY** â€” `user_id = current_user_id`.
- **TEAM** â€” `EXISTS (SELECT 1 FROM entity_stakeholder WHERE entity_type='deal' AND team_id=?)`.

## Performance optimizations

<Tip>
  Use the `type` parameter to fetch only the tab the user is viewing. This
  avoids fetching all tabs in parallel and reduces load.
</Tip>

1. **Parallel tab fetching** â€” All tabs are fetched in parallel on initial load when no `type` is specified.
2. **SQL EXISTS subqueries** â€” Prevent N+1 queries for access control checks.
3. **DB-level pagination** â€” Uses `LIMIT`/`OFFSET` for efficient pagination.
4. **Selective fetching** â€” When `type` is provided, only that tab is fetched.
