## Overview

The Escalation Module automates responses when assigned leads go stale. A scheduled engine detects trigger conditions (no first contact, went cold) and executes tiered escalation actions — notifications, temperature changes, tag additions, and redistribution to new agents.

<Info>
**Status:** Active — fully implemented  
**Module Path:** `src/modules/crm/escalation/`
</Info>

### Design Principles

| Principle               | Decision                                                                                   |
| ----------------------- | ------------------------------------------------------------------------------------------ |
| pg-boss scheduling      | Escalation scheduler uses pg-boss recurring job for reliability                            |
| Tiered actions          | Rules have ordered tiers with configurable delays; actions execute in sequence             |
| Auto-resolution         | Events (activity, stage change, reassignment) automatically resolve active trackers        |
| Idempotency             | Partial unique index + `ON CONFLICT DO NOTHING` prevents duplicate trackers                |
| Distribution delegation | Reassignment uses the distribution engine (`REDISTRIBUTE` action), not a separate paradigm |
| RLS compliance          | All entities carry `organization_id` for row-level security                                |

### Component Responsibilities

| Component                  | Responsibility                                                                                      |
| -------------------------- | --------------------------------------------------------------------------------------------------- |
| **EscalationScheduler**    | pg-boss recurring job that runs every 60 seconds to detect new triggers and process due escalations |
| **TriggerDetector**        | Scans leads for unmet conditions (no first contact, went cold); creates tracker records             |
| **TierExecutor**           | Executes escalation tier actions (notify, redistribute, change temp, add tag)                       |
| **EscalationAutoResolver** | Listens to domain events and resolves active trackers when conditions change                        |
| **EscalationRuleService**  | CRUD for escalation rules; handles tracker cancellation on deactivation/deletion                    |

## Entity Specifications

### EscalationRule

Defines when and how a lead should be escalated. Evaluated by `TriggerDetector`.

<Note>
Rules are evaluated in ascending `priority` order (lower number = higher priority). Active rules must use unique priorities within the organization.
</Note>

| Column                 | Type      | Notes                                                                        |
| ---------------------- | --------- | ---------------------------------------------------------------------------- |
| id                     | uuid PK   |                                                                              |
| organization_id        | uuid FK   | RLS                                                                          |
| name                   | varchar   | Human-readable rule name                                                     |
| is_active              | bool      | default true                                                                 |
| priority               | int       | Evaluation order                                                             |
| trigger_type           | enum      | `NO_FIRST_CONTACT`, `WENT_COLD`                                              |
| trigger_config         | jsonb     | `{thresholdMinutes?, thresholdValue?, thresholdUnit?}`                       |
| conditions             | jsonb     | `EscalationCondition[]` — AND-joined applicability filters; `[]` = all leads |
| respect_business_hours | bool      | default true. References org business hours schedule.                        |
| created_by             | uuid FK   |                                                                              |
| created_at, updated_at | timestamp |                                                                              |
| is_deleted             | bool      | soft delete                                                                  |

#### EscalationCondition Shape

```typescript
interface EscalationCondition {
  field: 'temperature' | 'leadSource' | 'language' | 'sourceChannel';
  operator: 'eq' | 'in';
  value: string | string[];
}
```

#### SQL Field Mapping

Used by `TriggerDetector.buildApplicabilityExtraWhere`:

| Field           | SQL Column         | Table  | Notes                                                                                        |
| --------------- | ------------------ | ------ | -------------------------------------------------------------------------------------------- |
| `temperature`   | `l.temperature`    | lead   |                                                                                              |
| `leadSource`    | `l.lead_source`    | lead   |                                                                                              |
| `sourceChannel` | `l.source_channel` | lead   |                                                                                              |
| `language`      | `p.languages`      | person | Adds `LEFT JOIN person p ON p.id = l.person_id`; matches JSONB entries by `languages[].code` |

### EscalationTier

Each tier in an escalation rule represents a delayed action set. Tiers execute in `tier_order` sequence.

| Column             | Type    | Notes                                                                                                                                     |
| ------------------ | ------- | ----------------------------------------------------------------------------------------------------------------------------------------- |
| id                 | uuid PK |                                                                                                                                           |
| escalation_rule_id | uuid FK |                                                                                                                                           |
| organization_id    | uuid FK | RLS                                                                                                                                       |
| tier_order         | int     | 1, 2, 3... (max 10)                                                                                                                       |
| delay_minutes      | int     | Tier 1 (lowest tier_order): always 0 — threshold is the sole timing control. Subsequent tiers: minutes after the previous tier completed. |
| actions            | jsonb   | `TierAction[]` — see Tier Actions below                                                                                                   |

#### Tier Action Types

| Action Type          | Parameters                               | Resolution                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| -------------------- | ---------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `NOTIFY_AGENT`       | `message?: string`                       | Resolved from lead's current stakeholder (assigned agent)                                                                                                                                                                                                                                                                                                                                                                                                                              |
| `NOTIFY_ADMIN`       | `message?: string`                       | **Self-resolving** — queries all org users with the `system.admin` permission key via `UserOrgRole → RolePermission → Permission`. Skipped if no admin users found.                                                                                                                                          |
| `REDISTRIBUTE`       | `{distributionSettings?: {mode?, roleId?}}` | Uses the distribution engine to reassign the lead. Default: org distribution settings.                                                                                                                                                                                                                                                                                                                                                                                                |
| `CHANGE_TEMPERATURE` | `{temperature: string}`                  | Updates lead temperature directly                                                                                                                                                                                                                                                                                                                                                                                                                                                       |
| `ADD_TAG`            | `{tagName: string}`                      | Adds tag to lead; creates tag if it doesn't exist                                                                                                                                                                                                                                                                                                                                                                                                                                      |

### EscalationTracker

Tracks the escalation progress for a specific lead. Created when trigger conditions are met, resolved when conditions change.

| Column                | Type      | Notes                                                                        |
| --------------------- | --------- | ---------------------------------------------------------------------------- |
| id                    | uuid PK   |                                                                              |
| organization_id       | uuid FK   | RLS                                                                          |
| escalation_rule_id    | uuid FK   |                                                                              |
| lead_id               | uuid FK   |                                                                              |
| triggered_at          | timestamp | When trigger condition was first detected                                    |
| trigger_reason        | varchar   | Human-readable trigger description                                           |
| current_tier          | int       | Current tier being processed (1-based)                                      |
| next_action_due_at    | timestamp | When next tier should execute                                                |
| resolved_at           | timestamp | null if active                                                               |
| resolution_reason     | varchar   | Why tracker was resolved                                                     |
| resolution_event_type | varchar   | Event type that caused resolution                                            |

<Warning>
Partial unique index prevents duplicate active trackers: `(escalation_rule_id, lead_id) WHERE resolved_at IS NULL`
</Warning>

## Escalation Engine

### EscalationScheduler

<Info>
Runs as a pg-boss recurring job every 60 seconds with the key `escalation-scheduler`.
</Info>

<Steps>
<Step title="Detect new triggers">
Scans all active escalation rules and identifies leads that meet trigger conditions but don't have active trackers
</Step>

<Step title="Process due escalations">
Finds trackers with `next_action_due_at <= NOW()` and executes the appropriate tier actions
</Step>

<Step title="Handle business hours">
Skips processing if `respect_business_hours` is true and current time is outside business hours
</Step>
</Steps>

### TriggerDetector

Identifies leads that meet escalation trigger conditions:

#### NO_FIRST_CONTACT
- Lead has stakeholder assigned
- No activity records with direction `OUTBOUND` 
- Time since assignment > threshold

#### WENT_COLD
- Lead was previously `HOT` or `WARM`
- Currently `COLD` 
- Time since temperature change > threshold

<Tip>
The detector uses optimized SQL queries with proper indexing to scan large lead volumes efficiently.
</Tip>

### TierExecutor

Executes actions for a specific escalation tier:

<Tabs>
<Tab title="Notification Actions">
- **NOTIFY_AGENT**: Sends notification to lead's assigned stakeholder
- **NOTIFY_ADMIN**: Queries users with admin permissions and sends notifications
</Tab>

<Tab title="Lead Actions">
- **REDISTRIBUTE**: Uses distribution engine to reassign lead
- **CHANGE_TEMPERATURE**: Updates lead temperature
- **ADD_TAG**: Adds specified tag to lead
</Tab>
</Tabs>

### EscalationAutoResolver

Event-driven component that automatically resolves escalation trackers when conditions change:

| Event Type | Resolution Trigger |
|------------|-------------------|
| `lead.activity.created` | New outbound activity resolves `NO_FIRST_CONTACT` trackers |
| `lead.temperature.changed` | Temperature change to non-COLD resolves `WENT_COLD` trackers |
| `lead.stakeholder.changed` | Reassignment resolves all active trackers for the lead |
| `lead.stage.changed` | Stage change resolves all active trackers |

## Edge Case Handling

### Business Hours Respect

<Steps>
<Step title="Schedule Check">
When `respectBusinessHours` is true, the scheduler checks current time against organization business hours
</Step>

<Step title="Delayed Execution">
If outside business hours, escalation processing is deferred until the next business hour period
</Step>

<Step title="Catch-up Processing">
When business hours resume, all pending escalations are processed immediately
</Step>
</Steps>

### Concurrent Modifications

- **Rule Updates**: Active tracker cancellation prevents orphaned escalations
- **Lead Reassignment**: Auto-resolver immediately cancels trackers to prevent duplicate notifications
- **Duplicate Prevention**: Unique constraints prevent multiple active trackers per rule-lead pair

### Error Recovery

<Tip>
Failed escalation actions are logged but don't block subsequent tier execution. The system continues processing other actions and advances to the next tier on schedule.
</Tip>

## Performance & Scaling

### Database Optimization

- **Indexes**: Optimized for trigger detection queries and tracker lookups
- **Partitioning**: Action logs can be partitioned by date for long-term performance
- **Query Batching**: Bulk operations reduce database round trips

### Memory Management

- **Lazy Loading**: Tier actions and conditions loaded on demand
- **Connection Pooling**: Efficient database connection reuse
- **Event Buffering**: Auto-resolver batches rapid successive events

## Integration Points

### Distribution Engine
- Escalation `REDISTRIBUTE` actions delegate to the distribution service
- Respects organization distribution settings and agent availability

### Notification System
- All notification actions use the centralized notification service
- Supports multiple delivery channels (email, in-app, webhook)

### Activity Tracking
- Monitors lead activities to detect first contact events
- Integrates with activity creation pipeline for real-time updates

### Event System
- Uses EventEmitter2 for decoupled communication
- Enables real-time escalation resolution based on lead changes
