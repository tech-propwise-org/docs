## Overview

The Escalation Module automates responses when assigned leads go stale. A scheduled engine detects trigger conditions (no first contact, went cold) and executes tiered escalation actions — notifications, temperature changes, tag additions, and redistribution to new agents.

<Info>
**Status:** Active — fully implemented  
**Module Path:** `src/modules/crm/escalation/`
</Info>

### Design Principles

<CardGroup cols={2}>
  <Card title="pg-boss Scheduling" icon="clock">

  </Card>
  <Card title="Tiered Actions" icon="layers">
    Rules have ordered tiers with configurable delays; actions execute in sequence
  </Card>
  <Card title="Auto-resolution" icon="check-circle">
    Events (activity, stage change, reassignment) automatically resolve active trackers
  </Card>

  </Card>

### Component Responsibilities

| Component | Responsibility |
|-----------|----------------|
| **EscalationScheduler** | pg-boss recurring job that runs every 60 seconds to detect new triggers and process due escalations |
| **TriggerDetector** | Scans leads for unmet conditions (no first contact, went cold); creates tracker records |
| **TierExecutor** | Executes escalation tier actions (notify, redistribute, change temp, add tag) |
| **EscalationAutoResolver** | Listens to domain events and resolves active trackers when conditions change |
| **EscalationRuleService** | CRUD for escalation rules; handles tracker cancellation on deactivation/deletion |

## Entity Specifications

### EscalationRule

<Note>
Defines when and how a lead should be escalated. Evaluated by `TriggerDetector`.
</Note>

| Column | Type | Notes |
|--------|------|-------|
| id | uuid PK | |
| organization_id | uuid FK | RLS |
| name | varchar | Human-readable rule name |
| is_active | bool | default true |
| priority | int | Evaluation order |
| trigger_type | enum | `NO_FIRST_CONTACT`, `WENT_COLD` |
| trigger_config | jsonb | `{thresholdMinutes?, thresholdValue?, thresholdUnit?}` |
| conditions | jsonb | `EscalationCondition[]` — AND-joined applicability filters; `[]` = all leads |
| respect_business_hours | bool | default true. References org business hours schedule. |
| created_by | uuid FK | |
| created_at, updated_at | timestamp | |
| is_deleted | bool | soft delete |

<Warning>
Rules are evaluated in ascending `priority` order (lower number = higher priority). Active rules must use unique priorities within the organization.
</Warning>

**EscalationCondition Shape:**

```typescript
interface EscalationCondition {
  field: 'temperature' | 'leadSource' | 'language' | 'sourceChannel';
  operator: 'eq' | 'in';
  value: string | string[];
}
```

**SQL Field Mapping:**

| Field | SQL Column | Table | Notes |
|-------|------------|-------|-------|
| `temperature` | `l.temperature` | lead | |
| `leadSource` | `l.lead_source` | lead | |
| `sourceChannel` | `l.source_channel` | lead | |
| `language` | `p.languages` | person | Adds `LEFT JOIN person p ON p.id = l.person_id`; matches JSONB entries by `languages[].code` |

### EscalationTier

Each tier in an escalation rule represents a delayed action set. Tiers execute in `tier_order` sequence.

| Column | Type | Notes |
|--------|------|-------|
| id | uuid PK | |
| escalation_rule_id | uuid FK | |
| organization_id | uuid FK | RLS |
| tier_order | int | 1, 2, 3... (max 10) |
| delay_minutes | int | Tier 1: always 0 — threshold is the sole timing control. Subsequent tiers: minutes after the previous tier completed. |
| actions | jsonb | `TierAction[]` — see Tier Actions below |

**Tier Action Types:**

| Action Type | Parameters | Resolution |
|-------------|------------|------------|
| `NOTIFY_AGENT` | `message?: string` | Resolved from lead's current stakeholder (assigned agent) |
| `NOTIFY_ADMIN` | `message?: string` | **Self-resolving** — queries all org users with the `system.admin` permission key |
| `NOTIFY_CUSTOM` | `userIds: string[], message?: string` | Uses provided user IDs |
| `CHANGE_TEMPERATURE` | `temperature: LeadTemperature` | Updates lead temperature |
| `ADD_TAG` | `tagNames: string[]` | Creates tags if they don't exist |
| `REDISTRIBUTE` | `distributionRuleIds?: string[]` | Uses distribution engine for reassignment |

## Escalation Engine

### Trigger Detection

<Steps>
<Step title="Query Active Rules">
The TriggerDetector queries all active escalation rules for the organization, ordered by priority
</Step>
<Step title="Build Lead Query">
For each trigger type, builds a SQL query to find leads that meet the trigger conditions
</Step>
<Step title="Apply Conditions">
Filters leads based on the rule's conditions (temperature, source, etc.)
</Step>
<Step title="Create Trackers">
Creates EscalationTracker records for matching leads (with conflict handling for idempotency)
</Step>
</Steps>

**No First Contact Trigger:**
```sql
SELECT l.id FROM lead l
LEFT JOIN activity a ON a.lead_id = l.id AND a.activity_type = 'FIRST_CONTACT'
WHERE l.organization_id = ? 
  AND l.stakeholder_id IS NOT NULL 
  AND a.id IS NULL
  AND l.created_at <= (NOW() - INTERVAL '? minutes')
```

**Went Cold Trigger:**
```sql
SELECT l.id FROM lead l
WHERE l.organization_id = ? 
  AND l.stakeholder_id IS NOT NULL
  AND l.temperature IN ('HOT', 'WARM')
  AND l.last_activity_at <= (NOW() - INTERVAL '? minutes')
```

### Auto-Resolution

The EscalationAutoResolver listens for domain events and automatically resolves active trackers:

<AccordionGroup>
<Accordion title="Lead Activity Events">
- `lead.activity.created` - Resolves trackers with reason 'LEAD_ACTIVITY'
- Applies to all tracker types
</Accordion>
<Accordion title="Lead Stage Changes">
- `lead.stage.changed` - Resolves trackers with reason 'STAGE_CHANGED'  
- Only if new stage indicates progress
</Accordion>
<Accordion title="Lead Reassignment">
- `lead.stakeholder.changed` - Resolves trackers with reason 'REASSIGNMENT'
- Applies when stakeholder changes to a different user
</Accordion>
<Accordion title="Lead Temperature Changes">
- `lead.temperature.changed` - Resolves WENT_COLD trackers with reason 'TEMPERATURE_IMPROVED'
- Only when temperature changes to HOT or WARM
</Accordion>
</AccordionGroup>

## Edge Case Handling

### Business Hours Considerations

<Note>
When `respectBusinessHours` is enabled, escalation timing calculations exclude non-business hours.
</Note>

- **Trigger Detection**: Threshold calculations use business hours only
- **Tier Delays**: Timer pauses during non-business hours
- **Action Execution**: Notifications may be delayed until business hours resume

### Concurrent Modifications

<AccordionGroup>
<Accordion title="Lead Updates During Escalation">
Auto-resolver handles concurrent lead updates that should resolve trackers. Uses event-driven resolution to avoid race conditions.
</Accordion>
<Accordion title="Rule Priority Conflicts">
Frontend prevents priority conflicts during editing. Backend validates and rejects conflicting priorities with clear error messages.
</Accordion>
<Accordion title="Multiple Rule Matches">
Higher priority rules (lower priority number) take precedence. Only one tracker created per lead per rule.
</Accordion>
</AccordionGroup>

### Failure Scenarios

| Scenario | Handling |
|----------|----------|
| **Notification Failure** | Logged as FAILED status, escalation continues to next tier |
| **Redistribution Failure** | Logged with error details, may retry based on distribution engine settings |
| **Database Connection Loss** | pg-boss handles job retry with exponential backoff |
| **Invalid Action Config** | Validation prevents invalid configs; runtime errors skip action and continue |

## Performance & Scaling

### Optimization Strategies

<Tip>
The escalation engine is designed for high-volume environments with thousands of leads.
</Tip>

<Steps>
<Step title="Batch Processing">
TriggerDetector processes leads in batches of 1000 to manage memory usage
</Step>
<Step title="Indexed Queries">
Critical database indexes on lead.created_at, lead.last_activity_at, and escalation_tracker fields
</Step>
<Step title="Async Actions">
All escalation actions execute asynchronously to avoid blocking the main escalation loop
</Step>
<Step title="Smart Scheduling">
pg-boss recurring job frequency can be adjusted based on escalation volume
</Step>
</Steps>

### Database Indexes

```sql
-- Critical performance indexes
CREATE INDEX idx_lead_escalation_nocont ON lead(organization_id, created_at) 
  WHERE stakeholder_id IS NOT NULL;

CREATE INDEX idx_lead_escalation_cold ON lead(organization_id, last_activity_at, temperature) 
  WHERE stakeholder_id IS NOT NULL;

CREATE INDEX idx_escalation_tracker_execution ON escalation_tracker(next_execution_at) 
  WHERE is_resolved = false;

-- RLS performance
CREATE INDEX idx_escalation_rule_org ON escalation_rule(organization_id) 
  WHERE is_deleted = false;
```
