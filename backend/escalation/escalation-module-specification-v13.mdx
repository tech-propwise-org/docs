# Escalation Module Specification

**Status:** Active — fully implemented  
**Module Path:** `src/modules/crm/escalation/`

<Note>
This module automates responses when assigned leads go stale, providing tiered escalation actions including notifications, temperature changes, tag additions, and redistribution to new agents.
</Note>

## Overview

The Escalation Module automates responses when assigned leads go stale. A scheduled engine detects trigger conditions (no first contact, went cold) and executes tiered escalation actions — notifications, temperature changes, tag additions, and redistribution to new agents.

### Design Principles

| Principle | Decision |
|-----------|----------|
| pg-boss scheduling | Escalation scheduler uses pg-boss recurring job for reliability |
| Tiered actions | Rules have ordered tiers with configurable delays; actions execute in sequence |
| Auto-resolution | Events (activity, stage change, reassignment) automatically resolve active trackers |
| Idempotency | Partial unique index + `ON CONFLICT DO NOTHING` prevents duplicate trackers |
| Distribution delegation | Reassignment uses the distribution engine (`REDISTRIBUTE` action), not a separate paradigm |
| RLS compliance | All entities carry `organization_id` for row-level security |

### Component responsibilities

| Component | Responsibility |
|-----------|----------------|
| **EscalationScheduler** | pg-boss recurring job that runs every 60 seconds to detect new triggers and process due escalations |
| **TriggerDetector** | Scans leads for unmet conditions (no first contact, went cold); creates tracker records |
| **TierExecutor** | Executes escalation tier actions (notify, redistribute, change temp, add tag) |
| **EscalationAutoResolver** | Listens to domain events and resolves active trackers when conditions change |
| **EscalationRuleService** | CRUD for escalation rules; handles tracker cancellation on deactivation/deletion |

## Entity Specifications

### EscalationRule

Defines when and how a lead should be escalated. Evaluated by `TriggerDetector`.

| Column | Type | Notes |
|--------|------|-------|
| id | uuid PK | |
| organization_id | uuid FK | RLS |
| name | varchar | Human-readable rule name |
| is_active | bool | default true |
| priority | int | Evaluation order |
| trigger_type | enum | `NO_FIRST_CONTACT`, `WENT_COLD` |
| trigger_config | jsonb | `{thresholdMinutes?, thresholdValue?, thresholdUnit?}` |
| conditions | jsonb | `EscalationCondition[]` — AND-joined applicability filters; `[]` = all leads |
| respect_business_hours | bool | default true. References org business hours schedule. |
| created_by | uuid FK | |
| created_at, updated_at | timestamp | |
| is_deleted | bool | soft delete |

<Warning>
Rules are evaluated in ascending `priority` order (lower number = higher priority). Active rules must use unique priorities within the organization.
</Warning>

### EscalationTier

Each tier in an escalation rule represents a delayed action set. Tiers execute in `tier_order` sequence.

| Column | Type | Notes |
|--------|------|-------|
| id | uuid PK | |
| escalation_rule_id | uuid FK | |
| organization_id | uuid FK | RLS |
| tier_order | int | 1, 2, 3... (max 10) |
| delay_minutes | int | Tier 1 (lowest tier_order): always 0 — threshold is the sole timing control. Subsequent tiers: minutes after the previous tier completed. |
| actions | jsonb | `TierAction[]` — see Tier Actions below |

### Tier Action Types

<Tabs>
  <Tab title="Notification Actions">
    | Action Type | Parameters | Resolution |
    |-------------|------------|------------|
    | `NOTIFY_AGENT` | `message?: string` | Resolved from lead's current stakeholder (assigned agent) |
    | `NOTIFY_ADMIN` | `message?: string` | **Self-resolving** — queries all org users with the `system.admin` permission key via `UserOrgRole → RolePermission → Permission`. Skipped if no admin users found. |
    | `NOTIFY_TEAM_LEAD` | `message?: string` | Resolved from assigned agent's direct manager |
    | `NOTIFY_CUSTOM_USERS` | `userIds: string[], message?: string` | Static user list |
  </Tab>
  
  <Tab title="State Actions">
    | Action Type | Parameters | Resolution |
    |-------------|------------|------------|
    | `CHANGE_TEMPERATURE` | `temperature: 'hot' \| 'warm' \| 'cold'` | Updates lead temperature |
    | `ADD_TAG` | `tagId: string` | Adds tag to lead |
    | `REDISTRIBUTE` | `config?: DistributionConfig` | Delegates to distribution engine |
  </Tab>
</Tabs>

## Escalation Engine

### EscalationScheduler

<Steps>
  <Step title="Job Configuration">
    - **Job name:** `'escalation-scheduler'`
    - **Schedule:** Every 60 seconds (`'*/60 * * * * *'`)
    - **Concurrency:** 1 (singleton pattern)
    - **Retry:** 3 attempts with exponential backoff
  </Step>
  
  <Step title="Trigger Detection">
    Calls `TriggerDetector.detectAndCreateTrackers()` to scan for new escalation triggers
  </Step>
  
  <Step title="Tier Execution">
    Calls `TierExecutor.processDueEscalations()` to execute pending tier actions
  </Step>
</Steps>

### TriggerDetector

Scans leads for escalation trigger conditions and creates tracker records.

<Tabs>
  <Tab title="NO_FIRST_CONTACT">
    **Logic:** Lead has been assigned for X minutes without any activity where `direction = 'OUTBOUND'`
    
    **SQL Query Pattern:**
    ```sql
    SELECT l.* FROM lead l
    WHERE l.organization_id = ?
      AND l.assigned_to IS NOT NULL
      AND l.stage != 'CLOSED'
      AND l.assigned_at <= NOW() - INTERVAL '? minutes'
      AND NOT EXISTS (
        SELECT 1 FROM activity a 
        WHERE a.lead_id = l.id 
          AND a.direction = 'OUTBOUND'
      )
      AND NOT EXISTS (
        SELECT 1 FROM escalation_tracker et
        WHERE et.lead_id = l.id 
          AND et.status IN ('PENDING', 'IN_PROGRESS')
      )
    ```
  </Tab>
  
  <Tab title="WENT_COLD">
    **Logic:** Lead's last activity was X hours/days ago, indicating it went cold
    
    **SQL Query Pattern:**
    ```sql
    SELECT l.* FROM lead l
    WHERE l.organization_id = ?
      AND l.assigned_to IS NOT NULL  
      AND l.stage != 'CLOSED'
      AND l.last_activity_at <= NOW() - INTERVAL '? hours/days'
      AND NOT EXISTS (
        SELECT 1 FROM escalation_tracker et
        WHERE et.lead_id = l.id 
          AND et.status IN ('PENDING', 'IN_PROGRESS')
      )
    ```
  </Tab>
</Tabs>

### TierExecutor

Processes escalation trackers that are due for execution.

<Steps>
  <Step title="Query Due Escalations">
    ```sql
    SELECT * FROM escalation_tracker
    WHERE status = 'IN_PROGRESS'
      AND next_execution_at <= NOW()
      AND organization_id = ?
    ```
  </Step>
  
  <Step title="Execute Tier Actions">
    For each due tracker, execute all actions in the current tier sequentially
  </Step>
  
  <Step title="Update Tracker State">
    - If more tiers exist: advance to next tier and set `next_execution_at`
    - If final tier: mark as `RESOLVED` with reason `COMPLETED`
  </Step>
</Steps>

### EscalationAutoResolver

Listens to domain events and automatically resolves active trackers when conditions change.

<AccordionGroup>
  <Accordion title="Activity Events">
    **Event:** `activity.created`  
    **Condition:** Activity direction is `'OUTBOUND'`  
    **Action:** Resolve with reason `ACTIVITY_OCCURRED`
  </Accordion>
  
  <Accordion title="Lead Stage Changes">
    **Event:** `lead.stage_changed`  
    **Condition:** New stage is `'CLOSED'`  
    **Action:** Resolve with reason `STAGE_CHANGED`
  </Accordion>
  
  <Accordion title="Lead Reassignment">
    **Event:** `lead.assigned`  
    **Condition:** `assigned_to` field changed  
    **Action:** Resolve with reason `REASSIGNED`
  </Accordion>
</AccordionGroup>

## Edge Case Handling

<Tabs>
  <Tab title="Business Hours">
    **Scenario:** Escalation scheduled outside business hours  
    **Handling:** 
    - If `respect_business_hours = true`, delay execution to next business hour
    - Use organization's business hours configuration
    - Recalculate `next_execution_at` accordingly
  </Tab>
  
  <Tab title="Agent Unavailable">
    **Scenario:** Assigned agent is inactive/deleted  
    **Handling:**
    - `NOTIFY_AGENT` actions are skipped with status `SKIPPED`
    - Log reason in `execution_details`
    - Continue with other actions in the tier
  </Tab>
  
  <Tab title="Duplicate Trackers">
    **Scenario:** Concurrent trigger detection creates duplicate trackers  
    **Handling:**
    - Partial unique index prevents duplicates
    - `ON CONFLICT DO NOTHING` in insert statement
    - Latest trigger attempt is silently ignored
  </Tab>
  
  <Tab title="Rule Deletion During Execution">
    **Scenario:** Rule is deleted while tracker is active  
    **Handling:**
    - Active trackers are resolved with reason `RULE_DELETED`
    - No new triggers are created for deleted rules
    - Audit trail is preserved
  </Tab>
</Tabs>

## Performance & Scaling

### Optimization Strategies

<Steps>
  <Step title="Database Indexing">
    - Index on `(organization_id, status, next_execution_at)` for due escalations
    - Index on `(organization_id, lead_id, status)` for resolution queries
    - Composite index on `(assigned_at, assigned_to)` for trigger detection
  </Step>
  
  <Step title="Query Optimization">
    - Batch processing of due escalations (limit 100 per cycle)
    - Efficient EXISTS queries for trigger detection
    - Eager loading of related entities
  </Step>
  
  <Step title="Job Scheduling">
    - pg-boss singleton pattern prevents concurrent execution
    - Configurable batch sizes for large organizations
    - Exponential backoff for failed executions
  </Step>
</Steps>
