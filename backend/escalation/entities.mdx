The Escalation Module manages automated responses when leads go stale through a structured entity system that includes rules, tiers, trackers, and action logs.

## Entity Overview

The escalation system consists of four main entities that work together to automate lead management:

<CardGroup cols={2}>
  <Card title="EscalationRule" icon="gear">
    Defines trigger conditions and when escalations should occur
  </Card>
  <Card title="EscalationTier" icon="layer-group">
    Configures sequential actions with delays for each rule
  </Card>
  <Card title="EscalationTracker" icon="location-crosshairs">
    Tracks escalation state for specific leads
  </Card>
  <Card title="EscalationActionLog" icon="list">
    Records all executed actions for analytics
  </Card>
</CardGroup>

## EscalationRule Entity

Defines when and how a lead should be escalated. Evaluated by the `TriggerDetector` component.

### Schema

| Column | Type | Notes |
|--------|------|-------|
| id | uuid PK | |
| organization_id | uuid FK | RLS compliance |
| name | varchar | Human-readable rule name |
| is_active | bool | default true |
| priority | int | Evaluation order |
| trigger_type | enum | `NO_FIRST_CONTACT`, `WENT_COLD` |
| trigger_config | jsonb | Threshold configuration |
| conditions | jsonb | Applicability filters |
| respect_business_hours | bool | default true |
| created_by | uuid FK | |
| created_at, updated_at | timestamp | |
| is_deleted | bool | Soft delete flag |

### Trigger Configuration

The `trigger_config` jsonb field contains threshold settings:

```json
{
  "thresholdMinutes": 1440,
  "thresholdValue": 24,
  "thresholdUnit": "hours"
}
```

### Conditions System

The `conditions` field contains an array of `EscalationCondition` objects that are AND-joined:

```typescript
interface EscalationCondition {
  field: 'temperature' | 'leadSource' | 'language' | 'sourceChannel';
  operator: 'eq' | 'in';
  value: string | string[];
}
```

<Note>
Empty conditions array (`[]`) means the rule applies to all leads.
</Note>

### Field Mapping

When evaluating conditions, fields map to SQL columns as follows:

| Field | SQL Column | Table | Notes |
|-------|-----------|-------|-------|
| `temperature` | `l.temperature` | lead | |
| `leadSource` | `l.lead_source` | lead | |
| `sourceChannel` | `l.source_channel` | lead | |
| `language` | `p.language` | person | Requires JOIN |

## EscalationTier Entity

Each tier represents a delayed action set that executes in sequence based on `tier_order`.

### Schema

| Column | Type | Notes |
|--------|------|-------|
| id | uuid PK | |
| escalation_rule_id | uuid FK | |
| organization_id | uuid FK | RLS compliance |
| tier_order | int | 1, 2, 3... (max 10) |
| delay_minutes | int | Delay after previous tier |
| actions | jsonb | Array of `TierAction` objects |

<Warning>
Tier 1 must have `delay_minutes = 0`. The rule's threshold controls initial timing.
</Warning>

### Tier Actions

The `actions` jsonb field contains an array of action configurations:

<Tabs>
  <Tab title="Notification Actions">
    ```json
    { "type": "NOTIFY_AGENT", "message": "Lead needs attention" }
    { "type": "NOTIFY_ADMIN", "message": "Escalation alert" }
    { "type": "NOTIFY_TEAM_LEAD" }
    ```
  </Tab>
  <Tab title="Lead Modification">
    ```json
    { "type": "CHANGE_TEMPERATURE", "temperature": "hot" }
    { "type": "ADD_TAG", "tagIds": ["tag-uuid-1", "tag-uuid-2"] }
    ```
  </Tab>
  <Tab title="Redistribution">
    ```json
    { "type": "REDISTRIBUTE" }
    ```
  </Tab>
</Tabs>

### Action Types

<AccordionGroup>
  <Accordion title="NOTIFY_AGENT">
    Notifies the lead's currently assigned agent.
    
    **Parameters:** `message?: string`
    
    **Resolution:** Uses lead's current stakeholder
  </Accordion>

  <Accordion title="NOTIFY_ADMIN">
    Notifies all organization administrators.
    
    **Parameters:** `message?: string`
    
    **Resolution:** Self-resolving - queries users with `system.admin` permission. Skipped if no admins found.
  </Accordion>

  <Accordion title="NOTIFY_TEAM_LEAD">
    Notifies all team leaders in the lead's assigned team.
    
    **Parameters:** `message?: string`
    
    **Resolution:** Self-resolving - queries team members with `team.admin` permission. Skipped if no team or leaders.
  </Accordion>

  <Accordion title="REDISTRIBUTE">
    Reassigns the lead to a new agent using the distribution engine.
    
    **Parameters:** None
    
    **Constraints:** Must be in the last tier only
    
    **Resolution:** Calls `DistributionEngineService.redistribute()`, creates distribution log entry
  </Accordion>

  <Accordion title="CHANGE_TEMPERATURE">
    Updates the lead's temperature directly.
    
    **Parameters:** `temperature: 'hot' | 'warm' | 'cold'`
    
    **Implementation:** Direct entity update bypassing validation guards
  </Accordion>

  <Accordion title="ADD_TAG">
    Appends tags to the lead's tag array.
    
    **Parameters:** `tagIds: string[]`
    
    **Implementation:** Deduplicates with existing tags
  </Accordion>
</AccordionGroup>

## EscalationTracker Entity

Tracks the escalation state of a specific lead against a specific rule.

### Schema

| Column | Type | Notes |
|--------|------|-------|
| id | uuid PK | |
| lead_id | uuid FK | |
| escalation_rule_id | uuid FK | |
| organization_id | uuid FK | RLS compliance |
| current_tier | int | 0 = triggered, not escalated |
| trigger_fired_at | timestamp | Initial detection time |
| next_escalation_at | timestamp | Scheduler query target |
| status | enum | `ACTIVE`, `RESOLVED`, `CANCELLED` |
| resolved_at | timestamp nullable | |
| resolved_by | enum nullable | Resolution reason |
| history | jsonb | Tracker history entries |
| created_at | timestamp | |

### Status Values

| Status | Description |
|--------|-------------|
| `ACTIVE` | Currently tracking escalation |
| `RESOLVED` | Automatically resolved by system event |
| `CANCELLED` | Manually cancelled or rule deactivated |

### Resolution Reasons

<Tabs>
  <Tab title="Manual">
    - `MANUAL` - User explicitly resolved via UI/API
  </Tab>
  <Tab title="Automatic">
    - `AUTO_ACTIVITY` - New activity added to lead
    - `AUTO_STAGE_CHANGE` - Lead moved to different stage
    - `AUTO_REASSIGNMENT` - Lead reassigned to different agent
    - `AUTO_ARCHIVED` - Lead archived
    - `AUTO_DELETED` - Lead deleted
    - `AUTO_ORPHANED` - Lead stakeholder removed
    - `REDISTRIBUTED` - Successfully redistributed to new agent
  </Tab>
</Tabs>

### Key Indexes

| Index | Columns | Type | Purpose |
|-------|---------|------|---------|
| `uq_escalation_tracker_lead_rule` | `(lead_id, escalation_rule_id) WHERE status = 'ACTIVE'` | Partial unique | Prevents duplicate active trackers |
| `idx_escalation_tracker_next_at` | `(next_escalation_at, status)` | Composite | Scheduler queries |
| `idx_escalation_tracker_lead` | `(lead_id, status)` | Composite | Auto-resolver lookups |
| `idx_escalation_tracker_org_status` | `(organization_id, status)` | Composite | Per-org statistics |

### Idempotency

The system prevents duplicate trackers using a partial unique index combined with `ON CONFLICT DO NOTHING`:

```sql
INSERT INTO escalation_tracker
  (id, lead_id, escalation_rule_id, organization_id, trigger_fired_at,
   next_escalation_at, status, history, current_tier, created_at)
VALUES (gen_random_uuid(), $1, $2, $3, $4, $5, 'ACTIVE', '[]', 0, NOW())
ON CONFLICT (lead_id, escalation_rule_id) WHERE status = 'ACTIVE' DO NOTHING;
```

<Warning>
Always use raw SQL with `ON CONFLICT DO NOTHING` for tracker creation. Never use `em.persistAndFlush()` as it will throw `UniqueViolationError` and abort the scheduler.
</Warning>

## EscalationActionLog Entity

Normalized table recording every escalation tier action execution for analytics.

### Schema

| Column | Type | Notes |
|--------|------|-------|
| id | uuid PK | |
| tracker_id | uuid FK | References escalation_tracker |
| organization_id | uuid FK | RLS compliance |
| tier_order | int | Which tier triggered this action |
| action_type | varchar | Action type identifier |
| action_params | jsonb nullable | Serialized parameters |
| result | enum | `SUCCESS`, `FAILED`, `SKIPPED` |
| executed_at | timestamp | Execution timestamp |

### Result Types

| Result | Description |
|--------|-------------|
| `SUCCESS` | Action completed successfully |
| `FAILED` | Action failed due to error |
| `SKIPPED` | Action skipped (e.g., no recipients found) |

### Analytics Indexes

| Index | Columns | Purpose |
|-------|---------|---------|
| `(organization_id, action_type)` | Action type analytics |
| `(organization_id, executed_at)` | Time-range queries |
| `(tracker_id)` | Lookup all actions for a tracker |

</Tip>
