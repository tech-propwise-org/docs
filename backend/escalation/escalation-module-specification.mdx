The Escalation Module automates responses when assigned leads go stale. A scheduled engine detects trigger conditions (no first contact, went cold) and executes tiered escalation actions — notifications, temperature changes, tag additions, and redistribution to new agents.

<Note>
**Status:** Active — fully implemented  
**Module Path:** `src/modules/crm/escalation/`
</Note>

## Design Principles

| Principle | Decision |
|-----------|----------|
| pg-boss scheduling | Escalation scheduler uses pg-boss recurring job for reliability |
| Tiered actions | Rules have ordered tiers with configurable delays; actions execute in sequence |
| Auto-resolution | Events (activity, stage change, reassignment) automatically resolve active trackers |
| Idempotency | Partial unique index + `ON CONFLICT DO NOTHING` prevents duplicate trackers |
| Distribution delegation | Reassignment uses the distribution engine (`REDISTRIBUTE` action), not a separate paradigm |
| RLS compliance | All entities carry `organization_id` for row-level security |

### Component Responsibilities

| Component | Responsibility |
|-----------|----------------|
| **EscalationScheduler** | pg-boss recurring job that runs every 60 seconds to detect new triggers and process due escalations |
| **TriggerDetector** | Scans leads for unmet conditions (no first contact, went cold); creates tracker records |
| **TierExecutor** | Executes escalation tier actions (notify, redistribute, change temp, add tag) |
| **EscalationAutoResolver** | Listens to domain events and resolves active trackers when conditions change |
| **EscalationRuleService** | CRUD for escalation rules; handles tracker cancellation on deactivation/deletion |

### EscalationTier

Each tier in an escalation rule represents a delayed action set. Tiers execute in `tier_order` sequence.

| Column | Type | Notes |
|--------|------|-------|
| id | uuid PK | |
| escalation_rule_id | uuid FK | |
| organization_id | uuid FK | RLS |
| tier_order | int | 1, 2, 3... (max 10) |
| delay_minutes | int | Tier 1 (lowest tier_order): always 0. Subsequent tiers: minutes after previous tier |
| actions | jsonb | `TierAction[]` |

<Warning>
Tier 1 delay is always 0 — the threshold is the sole timing control. Subsequent tiers delay after the previous tier completed.
</Warning>

#### Tier Action Types

<AccordionGroup>
<Accordion title="NOTIFY Actions">
| Action Type | Parameters | Resolution |
|-------------|------------|------------|
| `NOTIFY_AGENT` | `message?: string` | Resolved from lead's current stakeholder (assigned agent) |
| `NOTIFY_ADMIN` | `message?: string` | **Self-resolving** — queries all org users with `system.admin` permission |
| `NOTIFY_TEAM_LEAD` | `message?: string` | **Self-resolving** — queries team members with `team.admin` permission |

```typescript
{ "type": "NOTIFY_AGENT", "message": "Lead needs attention" }
{ "type": "NOTIFY_ADMIN", "message": "Escalation alert" }
{ "type": "NOTIFY_TEAM_LEAD" }
```
</Accordion>

<Accordion title="REDISTRIBUTE Action">
| Action Type | Parameters | Resolution |
|-------------|------------|------------|
| `REDISTRIBUTE` | _(no params)_ | **Distribution engine delegation** — removes current stakeholders, calls `DistributionEngineService.redistribute()` |

<Warning>
Must be in the **last tier** only. The API rejects rules where REDISTRIBUTE appears in an intermediate tier.
</Warning>

```typescript
{ "type": "REDISTRIBUTE" }
```
</Accordion>

<Accordion title="UPDATE Actions">
| Action Type | Parameters | Resolution |
|-------------|------------|------------|
| `CHANGE_TEMPERATURE` | `temperature: 'hot' \| 'warm' \| 'cold'` | **Direct entity update** — bypasses guards for archived leads |
| `ADD_TAG` | `tagIds: string[]` | **Direct entity update** — appends to `lead.tagIds` with deduplication |

```typescript
{ "type": "CHANGE_TEMPERATURE", "temperature": "hot" }
{ "type": "ADD_TAG", "tagIds": ["tag-uuid-1", "tag-uuid-2"] }
```
</Accordion>
</AccordionGroup>

### EscalationTracker

Tracks the escalation state of a specific lead against a specific rule.

| Column | Type | Notes |
|--------|------|-------|
| id | uuid PK | |
| lead_id | uuid FK | |
| escalation_rule_id | uuid FK | |
| organization_id | uuid FK | RLS |
| current_tier | int | 0 = triggered but not yet escalated |
| trigger_fired_at | timestamp | when condition was first detected |
| next_escalation_at | timestamp | indexed for scheduler query |
| status | enum | `ACTIVE`, `RESOLVED`, `CANCELLED` |
| resolved_at | timestamp nullable | |
| resolved_by | enum nullable | See ResolvedBy enum below |
| history | jsonb | `TrackerHistoryEntry[]` — append-only |
| created_at | timestamp | |

#### Key Indexes

<Info>
Critical indexes for performance and data integrity:
</Info>

| Index | Columns | Type | Purpose |
|-------|---------|------|---------|
| `uq_escalation_tracker_lead_rule` | `(lead_id, escalation_rule_id) WHERE status = 'ACTIVE'` | Partial unique | Prevents duplicate ACTIVE trackers |
| `idx_escalation_tracker_next_at` | `(next_escalation_at, status)` | Composite | Primary scheduler query |
| `idx_escalation_tracker_lead` | `(lead_id, status)` | Composite | Auto-resolver lookups |
| `idx_escalation_tracker_org_status` | `(organization_id, status)` | Composite | Per-org active counts |

#### Idempotency Guarantee

<CodeGroup>
```sql Idempotent Insert
INSERT INTO escalation_tracker
  (id, lead_id, escalation_rule_id, organization_id, trigger_fired_at,
   next_escalation_at, status, history, current_tier, created_at)
VALUES (gen_random_uuid(), $1, $2, $3, $4, $5, 'ACTIVE', '[]', 0, NOW())
ON CONFLICT (lead_id, escalation_rule_id) WHERE status = 'ACTIVE' DO NOTHING;
```
</CodeGroup>

<Warning>
`TriggerDetector` must **never** use `em.persistAndFlush()` for tracker creation — always use raw `execute()` with `ON CONFLICT DO NOTHING`.
</Warning>

## Escalation Engine

### Scheduler Flow

<Steps>
<Step title="Job Registration">
pg-boss recurring job runs every 60 seconds
</Step>

<Step title="Trigger Detection">
`TriggerDetector` scans leads for unmet conditions based on active rules
</Step>

<Step title="Tracker Creation">
Creates new trackers with idempotent `ON CONFLICT DO NOTHING` insert
</Step>

<Step title="Tier Execution">
`TierExecutor` processes due escalations from `next_escalation_at` index
</Step>

<Step title="Auto-Resolution">
Event listeners automatically resolve trackers when conditions change
</Step>
</Steps>

### Trigger Types

<Tabs>
<Tab title="NO_FIRST_CONTACT">
**Condition:** Lead has no activities and was created beyond the threshold time.

**Query Logic:**
```sql
SELECT l.* FROM lead l
WHERE l.organization_id = $1
  AND l.created_at <= $2  -- threshold time ago
  AND NOT EXISTS (
    SELECT 1 FROM activity a 
    WHERE a.lead_id = l.id
  )
```

**Configuration:**
- `thresholdMinutes`: Minutes since lead creation
</Tab>

<Tab title="WENT_COLD">
**Condition:** Lead's last activity is older than the threshold AND current temperature is not 'cold'.

**Query Logic:**
```sql
SELECT l.* FROM lead l
WHERE l.organization_id = $1
  AND l.temperature != 'cold'
  AND l.last_activity_at <= $2  -- threshold time ago
```

**Configuration:**
- `thresholdMinutes`: Minutes since last activity
</Tab>
</Tabs>

### Analytics Queries

<Accordion title="Sample Analytics Queries">
```sql
-- Escalation volume by day
SELECT DATE(executed_at) as date, COUNT(*) as escalations
FROM escalation_action_log 
WHERE organization_id = $1 
  AND executed_at >= $2
GROUP BY DATE(executed_at)
ORDER BY date;

-- Rule performance summary
SELECT 
  er.name as rule_name,
  COUNT(et.id) as total_triggered,
  AVG(EXTRACT(EPOCH FROM (et.resolved_at - et.trigger_fired_at))/60) as avg_resolution_minutes,
  COUNT(CASE WHEN et.resolved_by = 'REDISTRIBUTED' THEN 1 END) as redistributed_count
FROM escalation_rule er
LEFT JOIN escalation_tracker et ON et.escalation_rule_id = er.id
WHERE er.organization_id = $1
  AND et.created_at >= $2
GROUP BY er.id, er.name
ORDER BY total_triggered DESC;
```
</Accordion>

## Edge Case Handling

### Business Hours Respect

<Info>
When `respect_business_hours = true`, the system adjusts scheduling to align with organization business hours.
</Info>

### Failed Action Handling

<Warning>
Failed actions are logged but don't prevent tier progression. The system continues with remaining actions in the tier.
</Warning>

### Orphaned Lead Scenarios

When leads lose their assigned stakeholders, active trackers are automatically resolved with `resolved_by = AUTO_ORPHANED`.

### Rule Deactivation Impact

<Steps>
<Step title="Rule Deactivated">
User sets `is_active = false` on escalation rule
</Step>

<Step title="Tracker Cancellation">
System automatically cancels all `ACTIVE` trackers for that rule
</Step>

<Step title="Status Update">
Trackers marked as `CANCELLED` with `resolved_by = MANUAL`
</Step>
</Steps>

## Performance & Scaling

### Query Optimization

Critical indexes ensure sub-second query performance:

- Scheduler query: `(next_escalation_at, status)` composite index
- Lead lookup: `(lead_id, status)` for auto-resolution
- Analytics: `(organization_id, executed_at)` for time-range queries

### Batch Processing

The scheduler processes escalations in batches to prevent long-running transactions:

```typescript
const BATCH_SIZE = 100;
const dueTrackers = await this.findDueEscalations(batchSize: BATCH_SIZE);
```

### Memory Usage

Tracker history is stored as JSONB for quick reads, with authoritative data in the normalized `escalation_action_log` table for complex analytics.

## Integration Points

### Distribution Engine

<Note>
The `REDISTRIBUTE` action delegates to `DistributionEngineService.redistribute()`, maintaining consistency with the core distribution logic.
</Note>

### Notification System

All notification actions use the centralized `NotificationService`, respecting user preferences and delivery channels.

### Event System

The auto-resolver subscribes to domain events:

- `lead.activity.created`
- `lead.stage.changed` 
- `lead.stakeholder.assigned`
- `lead.archived`
- `lead.deleted`

### User Management

Notification resolution queries the permission system via `UserOrgRole → RolePermission → Permission` relationships for dynamic recipient lists.
