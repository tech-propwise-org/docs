The Escalation Engine is the core component that automates responses when assigned leads go stale. It detects trigger conditions and executes tiered escalation actions through a scheduled process.

## Overview

The escalation engine operates on three main principles:

<CardGroup cols={2}>
  <Card title="Automated Detection" icon="radar">
    Continuously scans for leads that meet escalation trigger conditions
  </Card>
  <Card title="Tiered Execution" icon="layer-group">
    Executes escalation actions in configurable tiers with delays
  </Card>
  <Card title="Smart Resolution" icon="check-circle">
    Automatically resolves trackers when conditions change
  </Card>
  <Card title="Reliable Scheduling" icon="clock">
    Uses pg-boss for robust recurring job execution
  </Card>
</CardGroup>

## Architecture Components

### Core Engine Components

The escalation engine consists of four primary components:

<AccordionGroup>
  <Accordion title="EscalationScheduler">
    **Purpose:** pg-boss recurring job that runs every 60 seconds
    
    **Responsibilities:**
    - Detect new trigger conditions
    - Process due escalations
    - Coordinate trigger detection and tier execution
  </Accordion>

  <Accordion title="TriggerDetector">
    **Purpose:** Scan leads for escalation conditions
    
    **Responsibilities:**
    - Identify leads meeting trigger criteria
    - Create tracker records with idempotency
    - Apply rule conditions and filters
  </Accordion>

  <Accordion title="TierExecutor">
    **Purpose:** Execute escalation tier actions
    
    **Responsibilities:**
    - Process notification actions
    - Handle lead redistribution
    - Update lead properties (temperature, tags)
    - Log all action results
  </Accordion>

  <Accordion title="EscalationAutoResolver">
    **Purpose:** Listen for domain events and resolve trackers
    
    **Responsibilities:**
    - Resolve trackers on lead activity
    - Handle stage changes and reassignments
    - Process archived/deleted leads
  </Accordion>
</AccordionGroup>

## Trigger Types

The engine supports two main trigger types for detecting escalation conditions:

<Tabs>
  <Tab title="NO_FIRST_CONTACT">
    Triggers when a lead has been assigned but no initial contact activity has been recorded within the threshold period.

    **Configuration:**
    ```json
    {
      "thresholdMinutes": 1440,  // 24 hours
      "thresholdValue": 1,
      "thresholdUnit": "days"
    }
    ```
  </Tab>
  <Tab title="WENT_COLD">
    Triggers when a lead's temperature has been set to "cold" and remains unchanged for the threshold period.

    **Configuration:**
    ```json
    {
      "thresholdMinutes": 2880,  // 48 hours
      "thresholdValue": 2,
      "thresholdUnit": "days"
    }
    ```
  </Tab>
</Tabs>

## Escalation Tiers

Each escalation rule can have multiple tiers that execute in sequence:

### Tier Configuration

<Steps>
  <Step title="Tier 1 (Initial)">
    - `delay_minutes`: Always 0 (threshold controls timing)
    - Actions execute immediately when trigger fires
  </Step>
  
  <Step title="Subsequent Tiers">
    - `delay_minutes`: Minutes after previous tier completed
    - Actions execute after delay period
  </Step>
  
  <Step title="Final Tier">
    - Can include `REDISTRIBUTE` action
    - Automatically resolves tracker if redistribution succeeds
  </Step>
</Steps>

### Available Actions

<CodeGroup>
```json Notification Actions
{
  "type": "NOTIFY_AGENT",
  "message": "Lead needs attention"
}

{
  "type": "NOTIFY_ADMIN", 
  "message": "Escalation alert"
}

{
  "type": "NOTIFY_TEAM_LEAD"
}
```

```json Lead Modifications
{
  "type": "CHANGE_TEMPERATURE",
  "temperature": "hot"
}

{
  "type": "ADD_TAG",
  "tagIds": ["tag-uuid-1", "tag-uuid-2"]
}
```

```json Redistribution
{
  "type": "REDISTRIBUTE"
}
```
</CodeGroup>

<Note>
The `REDISTRIBUTE` action must be placed in the final tier only. The API will reject rules where redistribution appears in intermediate tiers.
</Note>

## Tracker Lifecycle

### Tracker States

<Tabs>
  <Tab title="ACTIVE">
    Tracker is actively monitoring the lead and can execute escalation tiers.
    
    **Transitions to:**
    - `RESOLVED` - when auto-resolution conditions are met
    - `CANCELLED` - when rule is deactivated/deleted
  </Tab>
  
  <Tab title="RESOLVED">
    Tracker has been resolved due to lead activity or successful redistribution.
    
    **Resolution triggers:**
    - New lead activity
    - Stage change
    - Manual reassignment
    - Lead archived/deleted
    - Successful redistribution
  </Tab>
  
  <Tab title="CANCELLED">
    Tracker was cancelled due to rule changes.
    
    **Cancellation triggers:**
    - Rule deactivated
    - Rule deleted
    - Manual intervention
  </Tab>
</Tabs>

### Idempotency Protection

The engine uses a partial unique index to prevent duplicate trackers:

```sql
CREATE UNIQUE INDEX uq_escalation_tracker_lead_rule 
ON escalation_tracker (lead_id, escalation_rule_id) 
WHERE status = 'ACTIVE';
```

<Warning>
Tracker creation must always use `INSERT ... ON CONFLICT DO NOTHING` to prevent duplicate creation errors during concurrent execution.
</Warning>

## Business Hours Handling

When `respect_business_hours` is enabled on a rule:

<Steps>
  <Step title="Threshold Calculation">
    Only business hours count toward the trigger threshold
  </Step>
  
  <Step title="Execution Timing">
    Escalation actions only execute during business hours
  </Step>
  
  <Step title="Delay Calculation">
    Tier delays respect business hour boundaries
  </Step>
</Steps>

<Tip>
Business hours are defined per organization and reference the organization's configured schedule.
</Tip>

## Performance Considerations

### Indexing Strategy

Key indexes for optimal performance:

| Index | Purpose | Columns |
|-------|---------|---------|
| Scheduler Query | Next escalation lookup | `(next_escalation_at, status)` |
| Auto-Resolution | Lead-based lookups | `(lead_id, status)` |
| Analytics | Per-org tracking | `(organization_id, status)` |

### Batch Processing

<Steps>
  <Step title="Trigger Detection">
    Processes leads in configurable batch sizes to prevent memory issues
  </Step>
  
  <Step title="Tier Execution">
    Executes actions in parallel where possible
  </Step>
  
  <Step title="Action Logging">
    Bulk inserts action log entries for efficiency
  </Step>
</Steps>

## Error Handling

### Action Failures

<AccordionGroup>
  <Accordion title="Notification Failures">
    - Logged as `FAILED` in action log
    - Does not prevent other actions in the tier
    - Tier progression continues normally
  </Accordion>
  
  <Accordion title="Redistribution Failures">
    - Logged as `FAILED` in action log
    - Tracker remains `ACTIVE`
    - Next tier can still execute if configured
  </Accordion>
  
  <Accordion title="Lead Update Failures">
    - Logged as `FAILED` in action log
    - Other actions in tier continue
    - Validation errors are captured
  </Accordion>
</AccordionGroup>

### System Failures

<Warning>
If the scheduler job fails mid-execution, pg-boss will retry based on the configured retry policy. All escalation operations are designed to be idempotent to handle retries safely.
</Warning>

## Monitoring and Analytics

### Action Logging

All escalation actions are logged in the `escalation_action_log` table:

```typescript
interface ActionLogEntry {
  tracker_id: string;
  tier_order: number;
  action_type: EscalationActionType;
  action_params: object;
  result: 'SUCCESS' | 'FAILED' | 'SKIPPED';
  executed_at: Date;
}
```
