## Overview

The Escalation Module implements comprehensive security through role-based permissions and row-level security (RLS) policies. All entities carry organization-scoped access controls, while analytics provide insights into escalation patterns and performance metrics.

<Note>
All escalation entities enforce organization-level isolation through RLS policies to ensure tenant data separation.
</Note>

## Row-Level Security (RLS) Policies

### Entity RLS Implementation

All escalation entities include `organization_id` for tenant isolation:

```sql
-- Escalation Rule RLS Policy
CREATE POLICY escalation_rule_org_policy ON escalation_rule
  USING (organization_id = current_setting('app.current_organization_id')::uuid);

-- Escalation Tracker RLS Policy  
CREATE POLICY escalation_tracker_org_policy ON escalation_tracker
  USING (organization_id = current_setting('app.current_organization_id')::uuid);

-- Escalation Tier RLS Policy
CREATE POLICY escalation_tier_org_policy ON escalation_tier
  USING (organization_id = current_setting('app.current_organization_id')::uuid);

-- Escalation Action Log RLS Policy
CREATE POLICY escalation_action_log_org_policy ON escalation_action_log
  USING (organization_id = current_setting('app.current_organization_id')::uuid);
```

### Security Context

<Steps>
  <Step title="Request Authentication">
    User authentication establishes organization context through JWT token validation.
  </Step>
  
  <Step title="Organization Context Setting">
    `TenantContext` service sets `app.current_organization_id` for the database session.
  </Step>
  
  <Step title="RLS Policy Enforcement">
    PostgreSQL RLS policies automatically filter queries to organization-scoped data.
  </Step>
  
  <Step title="Permission Validation">
    API endpoints validate user permissions before executing operations.
  </Step>
</Steps>

<Warning>
The escalation scheduler runs with elevated privileges to process all organizations. It bypasses RLS by using a system-level database connection.
</Warning>

## Notification Security

### Notification Target Resolution

Escalation actions resolve notification recipients through secure queries:

<AccordionGroup>
  <Accordion title="Admin Notifications">
    ```sql
    SELECT DISTINCT u.id, u.email, u.name
    FROM "user" u
    JOIN user_org_role uor ON u.id = uor.user_id
    JOIN role_permission rp ON uor.role_id = rp.role_id
    JOIN permission p ON rp.permission_id = p.id
    WHERE uor.organization_id = $1
      AND p.key = 'system.admin'
      AND u.is_active = true
    ```
  </Accordion>
  
  <Accordion title="Team Lead Notifications">
    ```sql
    SELECT DISTINCT u.id, u.email, u.name
    FROM "user" u
    JOIN user_org_role uor ON u.id = uor.user_id
    JOIN role_permission rp ON uor.role_id = rp.role_id
    JOIN permission p ON rp.permission_id = p.id
    JOIN user_team_assignment uta ON u.id = uta.user_id
    JOIN lead_stakeholder ls ON uta.team_id = ls.stakeholder_id
    WHERE ls.lead_id = $1
      AND ls.stakeholder_type = 'TEAM'
      AND p.key = 'team.admin'
      AND u.is_active = true
    ```
  </Accordion>
  
  <Accordion title="Agent Notifications">
    Direct resolution from lead's assigned agent stakeholder with active user validation.
  </Accordion>
</AccordionGroup>

<Info>
Notification actions are automatically skipped if no valid recipients are found, preventing escalation failures.
</Info>

### Analytics Queries

The `escalation_action_log` provides rich data for analytics:

<Tabs>
  <Tab title="Escalation Volume">
    ```sql
    SELECT 
      DATE_TRUNC('day', executed_at) as day,
      action_type,
      COUNT(*) as action_count
    FROM escalation_action_log
    WHERE organization_id = $1
      AND executed_at >= $2
    GROUP BY day, action_type
    ORDER BY day DESC;
    ```
  </Tab>
  
  <Tab title="Success Rates">
    ```sql
    SELECT 
      action_type,
      result,
      COUNT(*) as count,
      ROUND(COUNT(*) * 100.0 / SUM(COUNT(*)) OVER (PARTITION BY action_type), 2) as percentage
    FROM escalation_action_log
    WHERE organization_id = $1
    GROUP BY action_type, result;
    ```
  </Tab>
  
  <Tab title="Trigger Distribution">
    ```sql
    SELECT 
      er.trigger_type,
      COUNT(et.id) as tracker_count,
      AVG(EXTRACT(EPOCH FROM (et.resolved_at - et.trigger_fired_at))/60) as avg_resolution_minutes
    FROM escalation_tracker et
    JOIN escalation_rule er ON et.escalation_rule_id = er.id
    WHERE et.organization_id = $1
      AND et.status = 'RESOLVED'
    GROUP BY er.trigger_type;
    ```
  </Tab>
</Tabs>

## Audit Trail

### Comprehensive Logging

Every escalation action creates an audit trail:

- **Escalation Tracker History**: JSON append-only log of state changes
- **Action Log Entries**: Normalized records of all tier executions  
- **Resolution Tracking**: Detailed resolution reasons and timestamps

### Data Retention

<Warning>
Escalation logs are retained indefinitely for compliance. Consider implementing data archiving policies for long-term storage management.
</Warning>
