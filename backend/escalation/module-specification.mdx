## Overview

The Escalation Module automates responses when assigned leads go stale. A scheduled engine detects trigger conditions (no first contact, went cold) and executes tiered escalation actions — notifications, temperature changes, tag additions, and redistribution to new agents.

<Info>
**Module Status:** Active — fully implemented  
**Module Path:** `src/modules/crm/escalation/`
</Info>

### Design Principles

| Principle | Decision |
|-----------|----------|
| pg-boss scheduling | Escalation scheduler uses pg-boss recurring job for reliability |
| Tiered actions | Rules have ordered tiers with configurable delays; actions execute in sequence |
| Auto-resolution | Events (activity, stage change, reassignment) automatically resolve active trackers |
| Idempotency | Partial unique index + `ON CONFLICT DO NOTHING` prevents duplicate trackers |
| Distribution delegation | Reassignment uses the distribution engine (`REDISTRIBUTE` action), not a separate paradigm |
| RLS compliance | All entities carry `organization_id` for row-level security |

### Component Responsibilities

| Component | Responsibility |
|-----------|----------------|
| **EscalationScheduler** | pg-boss recurring job that runs every 60 seconds to detect new triggers and process due escalations |
| **TriggerDetector** | Scans leads for unmet conditions (no first contact, went cold); creates tracker records |
| **TierExecutor** | Executes escalation tier actions (notify, redistribute, change temp, add tag) |
| **EscalationAutoResolver** | Listens to domain events and resolves active trackers when conditions change |
| **EscalationRuleService** | CRUD for escalation rules; handles tracker cancellation on deactivation/deletion |

### EscalationTier

Each tier in an escalation rule represents a delayed action set. Tiers execute in `tier_order` sequence.

| Column | Type | Notes |
|--------|------|-------|
| id | uuid PK | |
| escalation_rule_id | uuid FK | |
| organization_id | uuid FK | RLS |
| tier_order | int | 1, 2, 3... (max 10) |
| delay_minutes | int | Tier 1 (lowest tier_order): always 0 — threshold is the sole timing control. Subsequent tiers: minutes after the previous tier completed. |
| actions | jsonb | `TierAction[]` — see Tier Actions below |

<AccordionGroup>
<Accordion title="Tier Action Types">

| Action Type | Parameters | Resolution |
|-------------|------------|------------|
| `NOTIFY_AGENT` | `message?: string` | Resolved from lead's current stakeholder (assigned agent) |
| `NOTIFY_ADMIN` | `message?: string` | **Self-resolving** — queries all org users with the `system.admin` permission key via `UserOrgRole → RolePermission → Permission`. Skipped if no admin users found. |
| `NOTIFY_TEAM_LEAD` | `message?: string` | **Self-resolving** — queries all team members with the `team.admin` permission key in the lead's assigned team. Skipped if the lead has no team stakeholder or no team leaders exist. Notifies ALL team leaders. |
| `REDISTRIBUTE` | _(no params)_ | **Distribution engine delegation** — removes current stakeholders, calls `DistributionEngineService.redistribute()` which re-runs the full pipeline excluding the current assignee. |
| `CHANGE_TEMPERATURE` | `temperature: 'hot' \| 'warm' \| 'cold'` | **Direct entity update** — sets `lead.temperature` directly without going through `LeadService.update()` |
| `ADD_TAG` | `tagIds: string[]` | **Direct entity update** — appends `tagIds` to `lead.tagIds` (jsonb string array), deduplicating existing entries |

</Accordion>

<Accordion title="Action Configuration Examples">

```typescript
{ "type": "NOTIFY_AGENT", "message": "Lead needs attention" }
{ "type": "NOTIFY_ADMIN", "message": "Escalation alert" }
{ "type": "NOTIFY_TEAM_LEAD" }
{ "type": "REDISTRIBUTE" }
{ "type": "CHANGE_TEMPERATURE", "temperature": "hot" }
{ "type": "ADD_TAG", "tagIds": ["tag-uuid-1", "tag-uuid-2"] }
```

</Accordion>
</AccordionGroup>

<Warning>
`REDISTRIBUTE` actions must be in the **last tier** only. The API rejects rules where REDISTRIBUTE appears in an intermediate tier.
</Warning>

### EscalationTracker

Tracks the escalation state of a specific lead against a specific rule.

| Column | Type | Notes |
|--------|------|-------|
| id | uuid PK | |
| lead_id | uuid FK | |
| escalation_rule_id | uuid FK | |
| organization_id | uuid FK | RLS |
| current_tier | int | 0 = triggered but not yet escalated; increments with each tier fired |
| trigger_fired_at | timestamp | when the trigger condition was first detected |
| next_escalation_at | timestamp | indexed for scheduler query; null after all tiers complete |
| status | enum | `ACTIVE`, `RESOLVED`, `CANCELLED` |
| resolved_at | timestamp nullable | |
| resolved_by | enum nullable | See ResolvedBy values below |
| history | jsonb | `TrackerHistoryEntry[]` — append-only summary |
| created_at | timestamp | |

#### Key Indexes

| Index | Columns | Type | Notes |
|-------|---------|------|-------|
| `uq_escalation_tracker_lead_rule` | `(lead_id, escalation_rule_id) WHERE status = 'ACTIVE'` | Partial unique | Prevents duplicate ACTIVE trackers for the same lead+rule |
| `idx_escalation_tracker_next_at` | `(next_escalation_at, status)` | Composite | Primary scheduler query |
| `idx_escalation_tracker_lead` | `(lead_id, status)` | Composite | Auto-resolver lookups by lead |
| `idx_escalation_tracker_org_status` | `(organization_id, status)` | Composite | Per-org active tracker counts |

<Note>
**Idempotency guarantee:** The partial unique index prevents duplicate trackers. `TriggerDetector` uses `INSERT ... ON CONFLICT ... DO NOTHING` to prevent `UniqueViolationError`.
</Note>

## Escalation Engine

### Scheduler Architecture

The escalation engine operates on a 60-second recurring pg-boss job that performs two main functions:

<Steps>
<Step title="Trigger Detection">
Scans for leads that meet escalation trigger conditions and creates new tracker records
</Step>
<Step title="Tier Processing">
Processes existing trackers that are due for their next escalation tier
</Step>
</Steps>

### TriggerDetector Logic

<CodeGroup>

```sql Trigger Detection Query
SELECT DISTINCT l.id as lead_id, er.id as rule_id
FROM lead l
LEFT JOIN escalation_tracker et ON et.lead_id = l.id 
  AND et.escalation_rule_id = er.id 
  AND et.status = 'ACTIVE'
JOIN escalation_rule er ON er.organization_id = l.organization_id
WHERE l.organization_id = $1
  AND l.is_archived = false
  AND l.is_deleted = false
  AND er.is_active = true
  AND er.is_deleted = false
  AND et.id IS NULL  -- No active tracker exists
  AND [TRIGGER_CONDITION]
  AND [APPLICABILITY_CONDITIONS]
ORDER BY er.priority ASC
```

```typescript Trigger Conditions
// NO_FIRST_CONTACT
const noFirstContactCondition = `
  NOT EXISTS (
    SELECT 1 FROM activity a 
    WHERE a.lead_id = l.id 
    AND a.is_first_contact = true
  )
  AND l.created_at <= NOW() - INTERVAL '${thresholdMinutes} minutes'
`;

// WENT_COLD
const wentColdCondition = `
  l.temperature = 'cold'
  AND l.temperature_changed_at <= NOW() - INTERVAL '${thresholdMinutes} minutes'
`;
```

</CodeGroup>

### Auto-Resolution

The `EscalationAutoResolver` listens to domain events and automatically resolves active trackers:

| Event | Resolution Trigger | ResolvedBy |
|-------|-------------------|------------|
| `lead.activity.created` | New activity added | `AUTO_ACTIVITY` |
| `lead.stage.changed` | Stage transition | `AUTO_STAGE_CHANGE` |
| `lead.stakeholder.assigned` | New assignment | `AUTO_REASSIGNMENT` |
| `lead.archived` | Lead archived | `AUTO_ARCHIVED` |
| `lead.deleted` | Lead deleted | `AUTO_DELETED` |
| `lead.stakeholder.removed` | All stakeholders removed | `AUTO_ORPHANED` |

<Warning>
Auto-resolution occurs **immediately** when events are emitted, not during the scheduled run. This prevents unnecessary escalation actions on leads that no longer meet trigger conditions.
</Warning>

## Performance & Scaling

### Database Optimization

<Steps>
<Step title="Index Strategy">
Critical indexes for scheduler performance:
- `(next_escalation_at, status)` for due tracker queries
- `(organization_id, status)` for per-org active counts
- Partial unique index on `(lead_id, escalation_rule_id)` for idempotency
</Step>

<Step title="Query Optimization">
- Use `ON CONFLICT DO NOTHING` for tracker creation idempotency
- Batch process trackers in chunks of 100
- Direct entity updates for temperature/tag changes bypass service layer validation
</Step>

<Step title="pg-boss Configuration">
```javascript
{
  newJobCheckInterval: 2000,
  archiveCompletedAfterSeconds: 86400, // 24 hours
  deleteAfterDays: 7,
  retentionDays: 7
}
```
</Step>
</Runs>

### Scaling Considerations

| Component | Scaling Strategy |
|-----------|-----------------|
| **Scheduler** | Single instance with pg-boss coordination; horizontal scaling via job distribution |
| **Trigger Detection** | Organization-based partitioning; parallel processing per tenant |
| **Action Execution** | Async notification dispatch; batch redistribution calls |
| **Analytics** | Read replicas for historical queries; materialized views for aggregations |

<Tip>
For large organizations (>10k leads), consider implementing trigger condition pre-filtering at the application level before running full SQL queries.
</Tip>

## Integration Points

### Distribution Engine

The escalation module delegates lead redistribution to the existing distribution engine:

```typescript
// REDISTRIBUTE action execution
await this.distributionEngineService.redistribute(leadId, {
  excludeUserIds: [currentAssigneeId],
  distributionMethod: 'REDISTRIBUTION',
  reason: `Escalation tier ${tierOrder} redistribution`
});
```

### Notification System

All notification actions use the centralized notification service:

```typescript
await this.notificationService.send({
  type: 'ESCALATION_ALERT',
  recipientIds: [userId],
  templateData: {
    leadName: lead.person.name,
    ruleName: rule.name,
    message: action.message
  }
});
```

### Event System

Auto-resolution relies on domain events emitted by other modules:

```typescript
@OnEvent('lead.activity.created')
async handleActivityCreated(payload: ActivityCreatedEvent) {
  await this.escalationAutoResolver.resolveByActivity(payload.leadId);
}

@OnEvent('lead.stakeholder.assigned')
async handleStakeholderAssigned(payload: StakeholderAssignedEvent) {
  await this.escalationAutoResolver.resolveByReassignment(payload.leadId);
}
```

<Check>
The escalation module is fully integrated with the CRM's event-driven architecture, ensuring automatic resolution when lead conditions change.
</Check>
