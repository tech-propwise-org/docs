## Overview

The Escalation Module automates responses when assigned leads go stale. A scheduled engine detects trigger conditions (no first contact, went cold) and executes tiered escalation actions — notifications, temperature changes, tag additions, and redistribution to new agents.

<Info>
**Status:** Active — fully implemented  
**Module Path:** `src/modules/crm/escalation/`
</Info>

### Design principles

| Principle | Decision |
|-----------|----------|
| pg-boss scheduling | Escalation scheduler uses pg-boss recurring job for reliability |
| Tiered actions | Rules have ordered tiers with configurable delays; actions execute in sequence |
| Auto-resolution | Events (activity, stage change, reassignment) automatically resolve active trackers |
| Idempotency | Partial unique index + `ON CONFLICT DO NOTHING` prevents duplicate trackers |
| Distribution delegation | Reassignment uses the distribution engine (`REDISTRIBUTE` action), not a separate paradigm |
| RLS compliance | All entities carry `organization_id` for row-level security |

## Entity specifications

### EscalationRule

Defines when and how a lead should be escalated. Evaluated by `TriggerDetector`.

| Column | Type | Notes |
|--------|------|-------|
| id | uuid PK | |
| organization_id | uuid FK | RLS |
| name | varchar | Human-readable rule name |
| is_active | bool | default true |
| priority | int | Evaluation order |
| trigger_type | enum | `NO_FIRST_CONTACT`, `WENT_COLD` |
| trigger_config | jsonb | `{thresholdMinutes?, thresholdValue?, thresholdUnit?}` |
| condition_groups | jsonb | `[{conditions:[{field,operator,value}]}]` — AND-within-OR groups; `[]` = all leads |
| respect_business_hours | bool | default true. References org business hours schedule. |
| created_by | uuid FK | |
| created_at, updated_at | timestamp | |
| is_deleted | bool | soft delete |

<Warning>
Rules are evaluated in ascending `priority` order (lower number = higher priority). Active rules must use unique priorities within the organization.
</Warning>

#### Priority management

When creating a rule, the frontend defaults `priority` to one greater than the highest active escalation rule priority from the loaded rule list; edit mode preserves the existing rule priority. The frontend create/edit sheet disables submission when an active rule would reuse another active rule's priority, and rule cards block reactivation of paused rules with a conflicting priority.

The backend remains the source of truth for concurrent edits and enforces the invariant on create, priority update, and reactivation: if another active, non-deleted rule in the same organization already uses the requested priority, the write is rejected with `400 Bad Request`. Inactive rules may keep duplicate priorities until activation.

#### Duplicate rule prevention

Rule `name` is a display label only — duplicate names are allowed. The backend rejects create/update when another **non-deleted** rule in the same organization has an identical **behavior fingerprint**: `triggerType`, normalized `triggerConfig`, canonical `conditionGroups`, and canonical tiers/actions (`tierOrder`, `delayMinutes`, action type + params).

<Note>
Comparison ignores group/condition ordering within JSONB and treats legacy `thresholdMinutes` as equivalent to `thresholdValue` + `thresholdUnit: MINUTES`. On conflict the API returns `400 Bad Request` with message `An escalation rule with identical trigger, conditions, and actions already exists.`
</Note>

#### Applicability conditions

Escalation reuses the shared rule-condition module (`src/modules/crm/shared/rule-conditions/`). Stored shape matches distribution rules: `ConditionGroup[]` where conditions within a group are AND-joined and multiple groups are OR-joined.

```typescript
interface ConditionGroup {
  conditions: RuleCondition[]; // AND within group
}
// A lead matches when ANY group fully passes. Empty conditionGroups[] = all leads.
```

Used by `LeadScanService.buildApplicabilityExtraWhere`:

| `leadSource` | `l.lead_source` | lead | eq, in | Case-insensitive |
| `intent` | `l.intent` | lead | eq | Case-insensitive |

| `language` | `p.languages` | person | eq | `LEFT JOIN person p ON p.id = l.person_id`; matches JSONB `languages[].code` |
| `area` | wished snapshot names | lead_property_interest | eq, in, contains | `EXISTS` subquery flattening area snapshots |

## Escalation engine

### EscalationScheduler

<Steps>
<Step title="Job Registration">
Register pg-boss recurring job on module initialization:
```typescript
await this.pgBoss.schedule(
  'escalation-scheduler',
  '*/1 * * * *', // Every minute
  null,
  { singletonKey: 'escalation-scheduler' }
);
```
</Step>

<Step title="Trigger Detection">
Scan for new escalation triggers using `TriggerDetector`
</Step>

<Step title="Due Processing">
Process trackers with `next_tier_due_at <= NOW()` using `TierExecutor`
</Step>

<Step title="Cleanup">
Update tracker status and schedule next tier execution
</Step>
</Steps>

### TriggerDetector

Scans leads to identify new escalation conditions:

<Tabs>
<Tab title="NO_FIRST_CONTACT">
```sql
SELECT l.* FROM lead l
WHERE l.organization_id = ?
  AND l.assigned_agent_id IS NOT NULL
  AND l.stage != 'COMPLETED'
  AND l.created_at <= (NOW() - INTERVAL '? minutes')
  AND NOT EXISTS (
    SELECT 1 FROM activity a 
    WHERE a.lead_id = l.id 
    AND a.is_first_contact = true
  )
```
</Tab>

<Tab title="WENT_COLD">
```sql
SELECT l.* FROM lead l
WHERE l.organization_id = ?
  AND l.assigned_agent_id IS NOT NULL  
  AND l.stage != 'COMPLETED'
  AND l.temperature = 'COLD'
  AND l.updated_at <= (NOW() - INTERVAL '? minutes')
```
</Tab>
</Tabs>

### TierExecutor

Executes actions for due escalation tiers:

<CodeGroup>
```typescript Notification Actions
async executeNotifyAssignee(tracker: EscalationTracker, action: EscalationAction) {
  const lead = await this.leadService.findById(tracker.leadId);
  const config = action.actionConfig as NotificationActionConfig;
  
  await this.notificationService.send({
    userId: lead.assignedAgentId,
    type: 'ESCALATION_ALERT',
    title: 'Lead Escalation',
    message: config.message,
    leadId: lead.id
  });
}
```

```typescript Redistribution Action  
async executeRedistribute(tracker: EscalationTracker, action: EscalationAction) {
  const config = action.actionConfig as RedistributeActionConfig;
  
  await this.distributionEngineService.redistributeLead(
    tracker.leadId,
    config.reason
  );
}
```

```typescript Temperature Change
async executeChangeTemperature(tracker: EscalationTracker, action: EscalationAction) {
  const config = action.actionConfig as ChangeTemperatureActionConfig;
  
  await this.leadService.updateTemperature(
    tracker.leadId,
    config.newTemperature
  );
}
```
</CodeGroup>

### EscalationAutoResolver

Listens for domain events that should resolve active escalation trackers:

```typescript
@OnEvent('lead.activity.created')
async onActivityCreated(payload: { leadId: string; isFirstContact: boolean }) {
  if (payload.isFirstContact) {
    await this.resolveTrackersForLead(
      payload.leadId,
      ['NO_FIRST_CONTACT'],
      'FIRST_CONTACT_MADE'
    );
  }
}

@OnEvent('lead.temperature.changed')
async onTemperatureChanged(payload: { leadId: string; newTemperature: string }) {
  if (payload.newTemperature !== 'COLD') {
    await this.resolveTrackersForLead(
      payload.leadId,
      ['WENT_COLD'],
      'TEMPERATURE_IMPROVED'
    );
  }
}
```

```sql
  er.trigger_type,
  AVG(EXTRACT(EPOCH FROM (et.updated_at - et.triggered_at))/3600) as avg_hours
```

## Edge case handling

### Business hours respect

<Steps>
<Step title="Trigger Detection">
When `respectBusinessHours` is true, threshold calculations exclude non-business hours from elapsed time
</Step>

<Step title="Action Scheduling">  
Tier execution is delayed until next business hours if rule requires it
</Step>

<Step title="Timezone Handling">
Business hours are evaluated in the organization's configured timezone
</Step>
</Steps>

### Concurrent modifications

<Warning>
**Lead Reassignment:** If a lead is reassigned while escalation is active, existing tracker is resolved with reason `LEAD_REASSIGNED`
</Warning>

<Note>
**Rule Deactivation:** When a rule is deactivated, all active trackers using that rule are cancelled with reason `RULE_DEACTIVATED`
</Note>

### Data consistency

- **Idempotent Triggers:** Unique constraint prevents duplicate active trackers
- **Atomic Tier Execution:** All actions in a tier execute in a database transaction
- **Graceful Failures:** Failed individual actions don't prevent tier completion

