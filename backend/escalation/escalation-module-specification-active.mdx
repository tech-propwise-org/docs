## Overview

The Escalation Module automates responses when assigned leads go stale. A scheduled engine detects trigger conditions (no first contact, went cold) and executes tiered escalation actions — notifications, temperature changes, tag additions, and redistribution to new agents.

<Info>
**Module Status:** Active — fully implemented  
**Module Path:** `src/modules/crm/escalation/`
</Info>

### Design Principles

<CardGroup cols={2}>
  <Card title="pg-boss Scheduling" icon="clock">
    Escalation scheduler uses pg-boss recurring job for reliability
  </Card>
  <Card title="Tiered Actions" icon="layer-group">
    Rules have ordered tiers with configurable delays; actions execute in sequence
  </Card>
  <Card title="Auto-Resolution" icon="check-circle">
    Events (activity, stage change, reassignment) automatically resolve active trackers
  </Card>
  <Card title="Idempotency" icon="shield-check">
    Partial unique index + `ON CONFLICT DO NOTHING` prevents duplicate trackers
  </Card>
  <Card title="Distribution Delegation" icon="arrows-split-up-and-left">
    Reassignment uses the distribution engine (`REDISTRIBUTE` action), not a separate paradigm
  </Card>
  <Card title="RLS Compliance" icon="lock">
    All entities carry `organization_id` for row-level security
  </Card>
</CardGroup>

### Component Responsibilities

| Component | Responsibility |
|-----------|---------------|
| **EscalationScheduler** | pg-boss recurring job that runs every 60 seconds to detect new triggers and process due escalations |
| **TriggerDetector** | Scans leads for unmet conditions (no first contact, went cold); creates tracker records |
| **TierExecutor** | Executes escalation tier actions (notify, redistribute, change temp, add tag) |
| **EscalationAutoResolver** | Listens to domain events and resolves active trackers when conditions change |
| **EscalationRuleService** | CRUD for escalation rules; handles tracker cancellation on deactivation/deletion |

## Entity Specifications

### EscalationRule

Defines when and how a lead should be escalated. Evaluated by `TriggerDetector`.

<AccordionGroup>
  <Accordion title="Core Schema Fields">
    | Column | Type | Notes |
    |--------|------|-------|
    | `id` | uuid PK | Primary key |
    | `organization_id` | uuid FK | Row-level security |
    | `name` | varchar | Human-readable rule name |
    | `is_active` | bool | default true |
    | `priority` | int | Evaluation order |
    | `trigger_type` | enum | `NO_FIRST_CONTACT`, `WENT_COLD` |
    | `trigger_config` | jsonb | `{thresholdMinutes?, thresholdValue?, thresholdUnit?}` |
    | `condition_groups` | jsonb | `[{conditions:[{field,operator,value}]}]` — AND-within-OR groups; `[]` = all leads |
    | `respect_business_hours` | bool | default true. References org business hours schedule |
    | `created_by` | uuid FK | User who created the rule |
    | `created_at`, `updated_at` | timestamp | Audit timestamps |
    | `is_deleted` | bool | Soft delete flag |
  </Accordion>

  <Accordion title="Priority Management">
    Rules are evaluated in ascending `priority` order (lower number = higher priority). Active rules must use unique priorities within the organization.

    <Steps>
      <Step title="Priority Assignment">
        When creating a rule, the frontend defaults `priority` to one greater than the highest active escalation rule priority from the loaded rule list
      </Step>
      <Step title="Edit Mode">
        Edit mode preserves the existing rule priority
      </Step>
      <Step title="Validation">
        The frontend disables submission when an active rule would reuse another active rule's priority, and rule cards block reactivation of paused rules with a conflicting priority
      </Step>
      <Step title="Backend Enforcement">
        Backend enforces the invariant on create, priority update, and reactivation. Returns `400 Bad Request` if another active, non-deleted rule in the same organization already uses the requested priority
      </Step>
    </Steps>

    <Note>
    Inactive rules may keep duplicate priorities until activation
    </Note>
  </Accordion>

  <Accordion title="Duplicate Rule Prevention">
    Rule `name` is a display label only — duplicate names are allowed. 

    The backend rejects create/update when another **non-deleted** rule in the same organization has an identical **behavior fingerprint**:
    - `triggerType`
    - Normalized `triggerConfig`
    - Canonical `conditionGroups`
    - Canonical tiers/actions (`tierOrder`, `delayMinutes`, action type + params)

    <Warning>
    Comparison ignores group/condition ordering within JSONB and treats legacy `thresholdMinutes` as equivalent to `thresholdValue` + `thresholdUnit: MINUTES`
    </Warning>

    On conflict the API returns `400 Bad Request` with message:
    ```
    An escalation rule with identical trigger, conditions, and actions already exists.
    ```

    **Fingerprint helpers location:**
    - `src/modules/crm/escalation/escalation-rule-fingerprint.util.ts`
    - Shared condition canonicalization: `src/modules/crm/shared/rule-conditions/rule-fingerprint.util.ts`
  </Accordion>

  <Accordion title="Applicability Conditions">
    Escalation reuses the shared rule-condition module (`src/modules/crm/shared/rule-conditions/`). Stored shape matches distribution rules: `ConditionGroup[]` where conditions within a group are AND-joined and multiple groups are OR-joined.

    ```typescript
    interface ConditionGroup {
      conditions: RuleCondition[]; // AND within group
    }
    // A lead matches when ANY group fully passes. 
    // Empty conditionGroups[] = all leads.
    ```

    **API Shape:**
    - `CreateEscalationRuleDto` / `UpdateEscalationRuleDto` accept optional `conditionGroups?: ConditionGroupDto[]`
    - `EscalationRuleDto` returns `conditionGroups: ConditionGroup[]`
    - Omitted or empty `conditionGroups` means the rule applies to all leads
    - Validation uses shared `RuleConditionDto` / `ConditionGroupDto` classes from `src/modules/crm/shared/rule-conditions/`

    <Info>
    Same 8 fields, 6 operators, and value-shape validation as distribution rules (see Distribution Module Specification §3.3)
    </Info>
  </Accordion>

  <Accordion title="Migration History">
    **Migration:** `Migration20260607120000_EscalationConditionsToGroups`

    - Replaced flat `conditions` JSONB column with `condition_groups`
    - **up():** Wraps each legacy flat array in a single `{ conditions: [...] }` group
    - **down():** Flattens all group conditions back into a flat array (best-effort)

    <Note>
    Distribution rules already used `condition_groups` — escalation now shares the same shape and validators
    </Note>
  </Accordion>

  <Accordion title="SQL Field Mapping">
    Used by `LeadScanService.buildApplicabilityExtraWhere`:

    | Field | SQL Column / Expression | Table / Join | Operators | Notes |
    |-------|------------------------|--------------|-----------|-------|
    | `temperature` | `l.temperature` | lead | eq, in | Case-insensitive |
    | `leadSource` | `l.lead_source` | lead | eq, in | Case-insensitive |
    | `intent` | `l.intent` | lead | eq | Case-insensitive |
    | `budget` | `l.budget` | lead | eq, gte, lte, between | Numeric; `between` accepts `{ min, max }` or `[min, max]` |
    | `tags` | `l.tag_ids` | lead | contains | `EXISTS` + `jsonb_array_elements_text` + `IN (?)` per label — lead has any of the tag labels |
    | `sourceChannel` | `pc.channel_type` | person_channel | eq, in | `LEFT JOIN person_channel pc ON pc.id = l.source_channel_id AND pc.is_deleted = false` |
    | `language` | `p.languages` | person | eq | `LEFT JOIN person p ON p.id = l.person_id`; matches JSONB `languages[].code` |
    | `area` | wished snapshot names | lead_property_interest | eq, in, contains | `EXISTS` subquery flattening `wished_area_snapshots`, `wished_community_snapshots`, etc.; case-insensitive name match |

    <Warning>
    Unknown `field` keys (corrupt/legacy JSONB only — HTTP DTO validation rejects them on write) are skipped with a warning. Groups whose conditions are all empty or skipped produce `AND FALSE` (no leads match).
    </Warning>
  </Accordion>
</AccordionGroup>

### EscalationTier

Each tier in an escalation rule represents a delayed action set. Tiers execute in `tier_order` sequence.

| Column | Type | Notes |
|--------|------|-------|
| `id` | uuid PK | Primary key |
| `escalation_rule_id` | uuid FK | Parent rule |
| `tier_order` | int | Execution order (1, 2, 3...) |
| `delay_minutes` | int | Cumulative minutes from trigger detection |
| `actions` | jsonb | Array of action objects |
| `created_at`, `updated_at` | timestamp | Audit timestamps |

<Tabs>

### EscalationTracker

Records the escalation lifecycle for a specific lead under a specific rule.

<AccordionGroup>
  <Accordion title="Schema Definition">
    | Column | Type | Notes |
    |--------|------|-------|
    | `id` | uuid PK | Primary key |
    | `organization_id` | uuid FK | RLS |
    | `lead_id` | uuid FK | Target lead |
    | `escalation_rule_id` | uuid FK | Applied rule |
    | `triggered_at` | timestamp | When trigger condition was met |
    | `trigger_type` | enum | `NO_FIRST_CONTACT`, `WENT_COLD` |
    | `status` | enum | `ACTIVE`, `RESOLVED`, `CANCELLED` |
    | `current_tier` | int | Last executed tier_order (0 = none yet) |
    | `next_tier_due_at` | timestamp | When next tier should execute (null if all done) |
    | `resolved_at` | timestamp | When status became RESOLVED/CANCELLED |
    | `resolution_reason` | varchar | e.g., "ACTIVITY_DETECTED", "STAGE_CHANGED", "RULE_DEACTIVATED" |
    | `metadata` | jsonb | Additional context |
    | `created_at`, `updated_at` | timestamp | Audit timestamps |
  </Accordion>

  <Accordion title="Unique Constraint">
    ```sql
    UNIQUE (lead_id, escalation_rule_id) 
    WHERE status = 'ACTIVE' AND is_deleted = false
    ```

    <Info>
    This partial unique index ensures only one active escalation tracker exists per lead+rule combination, enabling idempotent tracker creation via `ON CONFLICT DO NOTHING`
    </Info>
  </Accordion>

  <Accordion title="Status Transitions">
    <Steps>
      <Step title="ACTIVE">
        Tracker is created when trigger condition is detected
      </Step>
      <Step title="RESOLVED">
        Auto-resolved when lead activity occurs, stage changes, or lead is reassigned
      </Step>
      <Step title="CANCELLED">
        Manually cancelled or rule is deactivated/deleted
      </Step>
    </Steps>
  </Accordion>
</AccordionGroup>

## Escalation Engine

### Scheduler Architecture

The escalation engine runs as a **pg-boss recurring job** named `escalation:check` with a 60-second interval.

<Steps>
  <Step title="Job Registration">
    On module initialization, `EscalationScheduler.onModuleInit()` registers the recurring job:
    ```typescript
    await this.pgBoss.schedule(
      'escalation:check',
      '*/1 * * * *', // every minute
      {},
      { tz: 'UTC' }
    );
    ```
  </Step>
  <Step title="Job Processing">
    Each execution calls `EscalationScheduler.processEscalations()`:
    - Fetch all active rules (ordered by priority)
    - For each rule, run trigger detection
    - Process due escalation tiers
  </Step>
  <Step title="Error Handling">
    Failures are logged but don't prevent subsequent rule processing. pg-boss handles retries and dead-letter queueing.
  </Step>
</Steps>

<Warning>
**Concurrent Execution:** pg-boss ensures only one worker processes `escalation:check` at a time per queue, preventing duplicate tracker creation
</Warning>

### Trigger Detection

`TriggerDetector` scans leads for escalation conditions.

<Tabs>
  <Tab title="NO_FIRST_CONTACT">
    ```typescript
    // Scan criteria:
    // 1. Lead is assigned to a user
    // 2. Lead stage is not WON/LOST/ARCHIVED
    // 3. No activity records exist for this lead
    // 4. Assignment timestamp + threshold has passed
    // 5. Lead matches rule conditionGroups (if any)
    // 6. No active tracker exists for this lead+rule

    const leads = await leadScanService.scanForNoFirstContact({
      ruleId: rule.id,
      thresholdMinutes: convertToMinutes(rule.triggerConfig),
      conditionGroups: rule.conditionGroups ?? [],
      respectBusinessHours: rule.respectBusinessHours,
    });
    ```
  </Tab>
  <Tab title="WENT_COLD">
    ```typescript
    // Scan criteria:
    // 1. Lead temperature is COLD
    // 2. Lead is assigned to a user
    // 3. Lead stage is not WON/LOST/ARCHIVED
    // 4. Last activity timestamp + threshold has passed
    // 5. Lead matches rule conditionGroups (if any)
    // 6. No active tracker exists for this lead+rule

    const leads = await leadScanService.scanForWentCold({
      ruleId: rule.id,
      thresholdMinutes: convertToMinutes(rule.triggerConfig),
      conditionGroups: rule.conditionGroups ?? [],
      respectBusinessHours: rule.respectBusinessHours,
    });
    ```
  </Tab>
</Tabs>

<Info>
**Business Hours Respect:** When `respectBusinessHours` is true, threshold calculations exclude non-business hours based on the organization's business hours schedule
</Info>

### Tracker Creation

<Steps>
  <Step title="Idempotent Insert">
    For each matching lead, create a tracker record:
    ```typescript
    const tracker = escalationTrackerRepo.create({
      organizationId,
      leadId,
      escalationRuleId: rule.id,
      triggeredAt: new Date(),
      triggerType: rule.triggerType,
      status: EscalationTrackerStatus.ACTIVE,
      currentTier: 0,
      nextTierDueAt: calculateNextTierDueAt(rule.tiers[0], new Date()),
    });

    await escalationTrackerRepo.persistAndFlush(tracker); // ON CONFLICT DO NOTHING
    ```
  </Step>
  <Step title="Conflict Handling">
    The unique constraint ensures duplicate trackers are silently ignored if they already exist
  </Step>
  <Step title="Event Emission">
    On successful creation, emit `escalation.tracker.created` event for observability
  </Step>
</Steps>

### Tier Execution

`TierExecutor` processes due escalation tiers.

<AccordionGroup>
  <Accordion title="Execution Flow">
    <Steps>
      <Step title="Fetch Due Trackers">
        ```sql
        SELECT * FROM escalation_tracker
        WHERE status = 'ACTIVE'
          AND next_tier_due_at <= NOW()
          AND is_deleted = false
        ORDER BY next_tier_due_at ASC
        ```
      </Step>
      <Step title="Load Rule & Tier">
        For each tracker, load the associated rule and the next tier to execute
      </Step>
      <Step title="Execute Actions">
        Sequentially execute all actions in the tier's `actions` array
      </Step>
      <Step title="Update Tracker">
        Increment `currentTier`, calculate `nextTierDueAt` for the next tier (or set to null if complete)
      </Step>
      <Step title="Log Results">
        Create `EscalationActionLog` entries for each executed action
      </Step>
    </Steps>
  </Accordion>

  <Accordion title="Action Execution Details">
    <Tabs>
      <Tab title="NOTIFY">
        ```typescript
        await notificationService.createEscalationNotification({
          organizationId,
          leadId,
          recipientRoles: params.recipientRoles,
          message: params.message,
          escalationRuleId: rule.id,
          tierOrder: tier.tierOrder,
        });
        // Creates in-app notifications for all online users with specified roles
        ```
      </Tab>
      <Tab title="CHANGE_TEMPERATURE">
        ```typescript
        await leadService.updateLeadTemperature({
          leadId,
          temperature: params.newTemperature,
          reason: 'ESCALATION',
          escalationRuleId: rule.id,
        });
        // Updates lead.temperature and logs the change
        ```
      </Tab>
      <Tab title="ADD_TAG">
        ```typescript
        await leadService.addTagToLead({
          leadId,
          tagId: params.tagId,
          source: 'ESCALATION',
          escalationRuleId: rule.id,
        });
        // Adds tag to lead.tag_ids array if not already present
        ```
      </Tab>
      <Tab title="REDISTRIBUTE">
        ```typescript
        await distributionEngineService.redistributeLead({
          leadId,
          reason: 'ESCALATION',
          escalationRuleId: rule.id,
          organizationId,
        });
        // Delegates to distribution engine for reassignment
        ```
      </Tab>
    </Tabs>
  </Accordion>

  <Accordion title="Error Handling">
    - Individual action failures are logged but don't halt tier execution
    - Failed actions create log entries with `status: FAILED` and `error_message`
    - Tracker progression continues to next tier even if some actions fail
    - Critical errors (e.g., DB connection loss) bubble up and fail the entire job

    <Warning>
    **Partial Tier Execution:** If 3 actions exist and action 2 fails, actions 1 and 3 still execute. The tracker moves to the next tier regardless.
    </Warning>
  </Accordion>
</AccordionGroup>

### Auto-Resolution

`EscalationAutoResolver` listens for domain events and automatically resolves active trackers.

<Tabs>
  <Tab title="Resolved Triggers">
    | Event | Reason | SQL Update |
    |-------|--------|------------|
    | `lead.activity.created` | `ACTIVITY_DETECTED` | Resolve all `ACTIVE` trackers for the lead |
    | `lead.stage.changed` | `STAGE_CHANGED` | Resolve if new stage is WON/LOST/ARCHIVED |
    | `lead.assigned` | `REASSIGNED` | Resolve all `ACTIVE` trackers for the lead |
    | `escalation.rule.deactivated` | `RULE_DEACTIVATED` | Cancel all `ACTIVE` trackers for the rule |
    | `escalation.rule.deleted` | `RULE_DELETED` | Cancel all `ACTIVE` trackers for the rule |
  </Tab>
  <Tab title="Implementation">
    ```typescript
    @OnEvent('lead.activity.created')
    async handleActivityCreated(payload: { leadId: string; organizationId: string }) {
      await this.escalationTrackerRepo.nativeUpdate(
        {
          leadId: payload.leadId,
          status: EscalationTrackerStatus.ACTIVE,
          isDeleted: false,
        },
        {
          status: EscalationTrackerStatus.RESOLVED,
          resolvedAt: new Date(),
          resolutionReason: 'ACTIVITY_DETECTED',
        }
      );
    }
    ```
  </Tab>
</Tabs>

<Info>
**Event-Driven Design:** Auto-resolution is implemented via event listeners, ensuring loose coupling between modules
</Info>


- At least one tier with one action required
</Note>

All escalation entities filter by `organization_id` automatically via MikroORM global filter:

<Warning>

## Edge Case Handling

<AccordionGroup>
  <Accordion title="Lead Reassignment During Escalation">
    **Scenario:** Lead is reassigned while an active escalation tracker exists.

    **Behavior:**
    - `EscalationAutoResolver` listens for `lead.assigned` event
    - All active trackers for that lead are marked `RESOLVED` with reason `REASSIGNED`
    - No further tiers execute
    - If the new agent also doesn't contact the lead, trigger detection will create a new tracker on the next scheduler run

    <Info>
    This prevents "ghost escalations" where actions target the old agent after reassignment
    </Info>
  </Accordion>

  <Accordion title="Rule Deactivation with Active Trackers">
    **Scenario:** Admin deactivates a rule that has 20 active trackers.

    **Behavior:**
    - `EscalationRuleService.update()` detects `isActive: false` transition
    - All active trackers for that rule are marked `CANCELLED` with reason `RULE_DEACTIVATED`
    - No further tiers execute
    - Tracker history remains in the database for analytics

    <Warning>
    Deactivated rules can be reactivated. This does NOT resurrect cancelled trackers — new triggers must be detected.
    </Warning>
  </Accordion>

  <Accordion title="Rule Deletion with Active Trackers">
    **Scenario:** Admin soft-deletes a rule with active trackers.

    **Behavior:**
    - `EscalationRuleService.remove()` soft-deletes the rule (`is_deleted = true`)
    - All active trackers are marked `CANCELLED` with reason `RULE_DELETED`
    - Deleted rules are excluded from scheduler runs
    - Tracker and action log history preserved for compliance

    <Note>
    Soft-deleted rules can be restored via database admin tools if needed for audit purposes
    </Note>
  </Accordion>

  <Accordion title="Tier Action Partial Failure">
    **Scenario:** Tier has 3 actions: NOTIFY (success), ADD_TAG (failed), REDISTRIBUTE (success).

    **Behavior:**
    - All three actions execute sequentially
    - Each action creates a separate `EscalationActionLog` entry
    - Failed action logs include `status: FAILED` and `error_message`
    - Tracker still advances to `currentTier++` and schedules the next tier
    - Metrics increment `escalation_actions_executed_total{status="failed"}`

    <Tip>
    Monitor failed action metrics to detect systemic issues (e.g., notification service down)
    </Tip>
  </Accordion>

  <Accordion title="Duplicate Tracker Creation Race">
    **Scenario:** Two scheduler workers detect the same trigger simultaneously.

    **Behavior:**
    - Both workers attempt to insert a tracker with identical `(lead_id, escalation_rule_id)`
    - Partial unique index causes one insert to succeed, the other fails with constraint violation
    - MikroORM's `persistAndFlush` with `ON CONFLICT DO NOTHING` silently ignores the duplicate
    - Only one tracker exists; no double-escalation occurs

    <Check>
    This idempotency guarantee is critical for distributed scheduler deployments
    </Check>
  </Accordion>

  <Accordion title="Business Hours Boundary">
    **Scenario:** Rule has `respectBusinessHours: true`. Trigger threshold elapses during weekend.

    **Behavior:**
    - `LeadScanService` uses `BusinessHoursService` to calculate elapsed business minutes
    - Weekend/holiday hours are excluded from the threshold calculation
    - Trigger detection only fires after threshold business minutes have passed
    - Example: 24-hour threshold with 9am-5pm weekday hours = 3 business days

    <Info>
    Business hours configuration is per-organization in the `organization_settings` table
    </Info>
  </Accordion>

  <Accordion title="Missing Tier Configuration">
    **Scenario:** Tracker has `currentTier: 2`, but rule only has tiers 1 and 3 (tier 2 was deleted).

    **Behavior:**
    - `TierExecutor.getNextTier()` searches for `tier_order > currentTier`
    - Finds tier 3 and executes it normally
    - Gap in tier sequence is benign — execution continues with next available tier

    <Warning>
    Frontend should prevent tier deletion if active trackers exist, but backend handles it gracefully
    </Warning>
  </Accordion>
</AccordionGroup>

## Performance & Scaling

### Scheduler Optimization

<Tabs>
  <Tab title="Query Batching">
    ```typescript
    // Bad: N+1 queries for each rule
    for (const rule of rules) {
      const leads = await findLeadsForRule(rule);
      for (const lead of leads) {
        await createTracker(lead, rule);
      }
    }

    // Good: Batch inserts per rule
    for (const rule of rules) {
      const leads = await findLeadsForRule(rule); // Single query
      const trackers = leads.map(lead => createTrackerEntity(lead, rule));
      await trackerRepo.persistAndFlush(trackers); // Bulk insert with ON CONFLICT
    }
    ```
  </Tab>
  <Tab title="Index Strategy">
    Critical indexes for scheduler performance:

    ```sql
    -- Trigger detection
    CREATE INDEX idx_lead_assigned_at 
      ON lead(assigned_to_id, assigned_at) 
      WHERE stage NOT IN ('WON', 'LOST', 'ARCHIVED');

    CREATE INDEX idx_lead_last_activity 
      ON lead(last_activity_at, temperature) 
      WHERE stage NOT IN ('WON', 'LOST', 'ARCHIVED');

    -- Tracker lookup (idempotency)
    CREATE UNIQUE INDEX idx_tracker_active_unique 
      ON escalation_tracker(lead_id, escalation_rule_id) 
      WHERE status = 'ACTIVE' AND is_deleted = false;

    -- Tier due processing
    CREATE INDEX idx_tracker_next_due 
      ON escalation_tracker(next_tier_due_at) 
      WHERE status = 'ACTIVE';
    ```
  </Tab>
  <Tab title="Pagination">
    ```typescript
    // Process large result sets in chunks
    const BATCH_SIZE = 500;
    let offset = 0;
    let hasMore = true;

    while (hasMore) {
      const leads = await leadScanService.scanForTrigger({
        rule,
        limit: BATCH_SIZE,
        offset,
      });

      await processLeadBatch(leads);
      
      hasMore = leads.length === BATCH_SIZE;
      offset += BATCH_SIZE;
    }
    ```
  </Tab>
</Tabs>

### Horizontal Scaling

<Steps>
  <Step title="pg-boss Queue Partitioning">
    pg-boss supports multiple workers processing the same queue. Set `teamSize` in job options:
    ```typescript
    await pgBoss.work('escalation:check', { teamSize: 4 }, handler);
    ```
  </Step>
  <Step title="Rule Sharding">
    For organizations with hundreds of rules, partition processing by rule priority ranges:
    ```typescript
    await pgBoss.schedule('escalation:check:p1-10', '*/1 * * * *', { priorityMin: 1, priorityMax: 10 });
    await pgBoss.schedule('escalation:check:p11-20', '*/1 * * * *', { priorityMin: 11, priorityMax: 20 });
    ```
  </Step>
  <Step title="Read Replicas">
    Route heavy lead scans to read replicas:
    ```typescript
    const leads = await em.fork({ useReplica: true }).find(Lead, { ... });
    ```
  </Step>
</Steps>

- **100 rules, 100K leads:** ~90s scan time, ~30s tracker creation

## RLS Policies

All escalation tables enforce row-level security via PostgreSQL RLS policies.

<CodeGroup>

```sql escalation_rule
CREATE POLICY escalation_rule_org_isolation ON escalation_rule
  USING (organization_id = current_setting('app.current_organization_id')::uuid);

-- Admin users can manage all rules in their org
-- Agents can view rules but not modify
CREATE POLICY escalation_rule_admin_full ON escalation_rule
  FOR ALL
  USING (
    organization_id = current_setting('app.current_organization_id')::uuid
    AND current_setting('app.current_user_role') IN ('ADMIN', 'MANAGER')
  );

CREATE POLICY escalation_rule_agent_read ON escalation_rule
  FOR SELECT
  USING (
    organization_id = current_setting('app.current_organization_id')::uuid
  );
```

```sql escalation_tracker
CREATE POLICY escalation_tracker_org_isolation ON escalation_tracker
  USING (organization_id = current_setting('app.current_organization_id')::uuid);

-- Agents can only view trackers for leads assigned to them
CREATE POLICY escalation_tracker_agent_own ON escalation_tracker
  FOR SELECT
  USING (
    organization_id = current_setting('app.current_organization_id')::uuid
    AND (
      current_setting('app.current_user_role') IN ('ADMIN', 'MANAGER', 'TEAM_LEAD')
      OR lead_id IN (
        SELECT id FROM lead 
        WHERE assigned_to_id = current_setting('app.current_user_id')::uuid
      )
    )
  );
```

```sql escalation_action_log
CREATE POLICY escalation_action_log_org_isolation ON escalation_action_log
  USING (organization_id = current_setting('app.current_organization_id')::uuid);

-- Only admins/managers can view action logs
CREATE POLICY escalation_action_log_admin ON escalation_action_log
  FOR SELECT
  USING (
    organization_id = current_setting('app.current_organization_id')::uuid
    AND current_setting('app.current_user_role') IN ('ADMIN', 'MANAGER')
  );
```

</CodeGroup>

<Warning>
**Session Variable Setup:** Every request must call `SET app.current_organization_id = ?` and `SET app.current_user_role = ?` before querying. This is handled automatically by the `TenantContextInterceptor`.
</Warning>

    - Backend does NOT prevent this — it's a configuration responsibility
