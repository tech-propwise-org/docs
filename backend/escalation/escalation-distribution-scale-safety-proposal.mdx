## Executive summary

The escalation and distribution engines are **architecturally sound for production** at ~100 orgs: two-stage escalation dispatch, `singletonKey` dedup, bounded trigger scans, and distribution batch caps prevent runaway crashes. The real constraints are **connection-pool budget**, **unbounded due-tracker execution per org job**, **scan query cost at large lead volume**, and **two correctness gaps** (silent dropped distribution enqueues and destructive `REDISTRIBUTE` when distribution is off).

This document proposes a prioritized fix plan: pool-safe throughput bounds, self-healing reconciliation, observability, and redistribution safety â€” without increasing concurrent transactions beyond what `DB_POOL_MAX` can sustain.

## Current architecture (reference)

### Escalation (two-stage pg-boss model)

| Stage | Queue / cron | Behavior |
| ----- | ------------ | -------- |
| **Stage 1 â€” Dispatcher** | `escalation-scheduler`, cron `* * * * *` (~60s) | Cluster-wide tick: `UNION` of orgs with active rules or active trackers â†’ one `escalation-org` job per org (`singletonKey: esc-org:<orgId>`) |
| **Stage 2 â€” Per-org worker** | `escalation-org`, `batchSize: 5`, `stately` policy | Up to 5 orgs in parallel; per org: `TriggerDetector.scan()` (LIMIT 500/rule) â†’ due trackers â†’ sequential `executeTrackerTier()` |

<Check>
**Safeguards already in place:** partial unique index + `ON CONFLICT DO NOTHING` on tracker insert; `singletonKey` drops duplicate org jobs while one is in-flight; per-tracker try/catch; `expireInSeconds: 120` on org jobs.
</Check>

### Distribution (event-driven + pg-boss)

| Path | Trigger | Worker |
| ---- | ------- | ------ |
| **Single lead** | `LEAD_CREATED` â†’ `DistributionListener` â†’ `enqueueSingle` (priority 10) | `distribution` queue, `batchSize: 10` |
| **Bulk / backfill** | Import or toggle-on backfill â†’ `enqueueBatch` (priority 0) | Same worker |
| **Escalation REDISTRIBUTE** | `TierExecutor` â†’ `DistributionEngineService.redistribute()` | Direct call (no toggle check) |

<Check>
**Safeguards already in place:** `singletonKey: lead:<leadId>`; idempotency in engine; two-phase capacity (advisory lock + re-count); post-commit `LEAD_ASSIGNED` notifications; `FAILED` `distribution_log` on exhausted worker retries.
</Check>

<Info>
**What "noisy" local dev usually is**
- Cron dispatcher every ~60s + per-org `.log()` even when `+0 trackers, 0 tiers`.
- `EscalationAutoResolver` logging on every activity/stage/stakeholder event.
- Not extra load â€” **log volume**, not a runaway loop.
</Info>

## Governing constraint: connection pool budget

Production default: **`DB_POOL_MAX = 20` per app instance** (`app.config.ts` â†’ `mikro-orm.config.ts`). Every `TenantContext.executeInOrg()` holds a connection for the transaction duration.

### Concurrent consumers (worst-case ceiling)

| Consumer | Concurrent transactions | Notes |
| -------- | ------------------------ | ----- |
| Distribution worker | **Up to 10** | `processJobs` uses `Promise.allSettled` over full batch (`distribution-job.handler.ts`) |
| Escalation org worker | **Up to 5** | One connection per org in batch; tracker tiers run **sequentially** per org |
| HTTP / other crons | Remainder | Auth, messaging, search maintenance, user-status, etc. |

<Warning>
**Worst case on one instance:** ~**15 of 20** slots during overlap (bulk import + escalation cycle). This is why **parallelizing the inner escalation tracker loop is the wrong lever** â€” it would multiply against the 5-org batch and blow the pool.
</Warning>

### Multi-instance note

pg-boss **cron dispatchers** are cluster-wide (one tick), but **workers** (`escalation-org`, `distribution`) run on **every** app instance. Total pressure on Neon â‰ˆ **`N_instances Ã— DB_POOL_MAX`**, plus pg-boss's own pool (`DATABASE_URL`). Size pools before scaling horizontally.

<Tip>
Treat "total concurrent transactions" as a **fixed budget**. Prefer bounds, sequential inner loops, and horizontal scale with **lower** per-instance `DB_POOL_MAX` â€” not larger batches.
</Tip>

## Scaling risks and mitigations

### Risk 1 â€” Scan query cost (steady-state, large lead tables)

`TriggerDetector.scan()` runs **one multi-table join per active rule per org every ~60s** (`lead â‹ˆ entity_stakeholder â‹ˆ activity`, `LIMIT 500` on results only).

At 100 orgs Ã— 3 rules â‰ˆ **300 scan queries/minute**. Unindexed paths degrade before the pool saturates.

| Mitigation | Priority |
| ---------- | -------- |
| `EXPLAIN (ANALYZE, BUFFERS)` on NFC + WENT_COLD scans against production-sized data | **Gate (P0)** |
| Confirm indexes on `entity_stakeholder`, `activity`, `lead`, `escalation_tracker.next_escalation_at` | **Gate (P0)** |

### Risk 2 â€” Unbounded due-tracker loop (concrete bug)

Per org, **all** due trackers are selected (no `LIMIT`) and processed sequentially. A heavy org (e.g. business-hours backlog) can exceed **`expireInSeconds: 120`** â†’ pg-boss expiry â†’ retries â†’ wasted work.

Heavy tiers (`REDISTRIBUTE`, `NOTIFY_ADMIN`) can exceed naive ~100ms/tracker estimates.

| Mitigation | Priority |
| ---------- | -------- |
| **Phase 1a:** `ORDER BY next_escalation_at ASC LIMIT $DUE_TRACKER_BATCH_LIMIT` (~50â€“100, or wall-clock budget ~60s) on due-tracker query | **P1 â€” do first** |
| Keep inner loop **sequential** (pool-safe) | Required |
| Drain remainder on next cron tick (~5 min for 1000 trackers @ 200/batch) | Default |
| Optional **Phase 1b:** continuation job with distinct `singletonKey` (`esc-org-cont:<orgId>`) only if drain latency is unacceptable | **P5 â€” defer** |

<Note>
Mirror precedent: `SCAN_BATCH_LIMIT = 500` in `trigger-detector.ts`.
</Note>

### Risk 3 â€” Silent dropped distribution enqueue

`DistributionListener.enqueueWithRetry` retries 3Ã—, then **only logs** â€” lead is created (HTTP 201) but **no job** and **no `distribution_log`**.

| Mitigation | Priority |
| ---------- | -------- |
| **Phase 2:** Stuck-lead reconciliation cron (~10â€“15 min), re-enqueue via `enqueueBatch` | **P2 â€” highest silent-failure value** |
| **Phase 3a:** On enqueue exhaustion, write `distribution_log` with `outcome: FAILED`, `failureReason: enqueue_failed` | **P3** |
| **Phase 3b:** Metrics: failed logs window, `escalation-org` retry exhaustion, pg-boss failed-job depth | **P3** |

**Stuck-lead definition (all must hold):**
- Distribution enabled for org/team
- Created > grace period (e.g. 10 min)
- Not archived / not deleted
- **No** user or team stakeholder
- **No** terminal `distribution_log` outcome: `ASSIGNED | FALLBACK_POOL | FALLBACK_TEAM | QUEUED | ALREADY_ASSIGNED` (zero logs, or only `FAILED` logs)

<Warning>
**Critical filter:** `FALLBACK_POOL` is terminal â€” leads intentionally in the pool must not be re-enqueued.

**False-positive guard:** Manual unassignment leaves no terminal log. Sweeper should target **never-attempted** leads (e.g. no `distribution_log` at all, or only `enqueue_failed` / worker `FAILED`) â€” not leads an admin deliberately left unassigned.
</Warning>

<Info>
**Module boundary:** No cross-module raw SQL. Expose e.g. `LeadService.findUndistributedLeadIdsInTransaction(...)` + compose with `EntityStakeholderService`; join `distribution_log` in distribution module only.
</Info>

### Risk 4 â€” Pool saturation under burst

Bulk import (`priority: 0`, batch 10) overlapping escalation cycle.

| Mitigation | Priority |
| ---------- | -------- |
| Size `DB_POOL_MAX` so `N Ã— max` < Neon ceiling | **Gate** |
| Optional: env-configurable `distribution` `batchSize` | **P5** |
| Horizontal scale: pg-boss `SKIP LOCKED` â€” safe multi-instance; reduce per-instance pool | When volume demands |

### Risk 5 â€” Observability gap

Operators cannot see queue depth, per-org cycle duration, or enqueue failures without log diving.

| Mitigation | Priority |
| ---------- | -------- |
| Wire `PgBossHealthIndicator` into readiness | **P3** |
| Reduce per-cycle `.log()` to `.debug()` when `+0 trackers, 0 tiers` | Low effort |
| Optional prod kill-switch env for escalation cron (test has `ESCALATION_SKIP_PGBOSS_WORKERS`) | Nice-to-have |

## Correctness gap: REDISTRIBUTE vs distribution toggle

### Investigation summary

We analyzed escalation `REDISTRIBUTE` when org-level distribution is **off**.

### What we found (verified in code)

<AccordionGroup>
<Accordion title="1. distributionEnabled is a trigger gate, not an engine kill-switch">
Checked in `DistributionListener.handleLeadCreated` and team Path B routing only. Org pipeline (`runDistribution`) still runs when off â€” **"off" = manual assignment mode**, not "engine disabled".
</Accordion>

<Accordion title="2. Escalation REDISTRIBUTE bypasses the toggle">
`TierExecutor.handleRedistribute` â†’ `DistributionEngineService.redistribute()` with **no** `distributionEnabled` check (`tier-executor.ts`).
</Accordion>

<Accordion title="3. redistribute() removes the current agent up-front">
`removeAllForEntityInTransaction` runs **before** the pipeline (`distribution-engine.service.ts`), with `excludeAgentIds: [previousAgent]`.
</Accordion>

<Accordion title="4. Default off state â†’ POOL fallback">
No matching rule + `defaultMethod = POOL` â†’ `FALLBACK_POOL` log, **no** new stakeholder, agent availability ignored.
</Accordion>

<Accordion title="5. Auto-create side effect">
Missing `DistributionSettings` row is created with `distributionEnabled = false`, `defaultMethod = POOL` â€” then pipeline proceeds anyway.
</Accordion>
</AccordionGroup>

### The gap

`REDISTRIBUTE` is **destructive** when distribution is off: removes Agent X, fails to reassign, leaves lead **orphaned in pool**. Escalation logs tier action `FAILED`; un-assignment is already committed.

| Problem | Description |
| ------- | ----------- |
| **A â€” Intent mismatch** | Escalation overrides operator's "distribution off / manual mode". |
| **B â€” Decision ordering** | Agent removed **before** placement confirmed. Removal + redistribution share one transaction (atomic by design); non-`ASSIGNED` outcomes (`FALLBACK_POOL`, `NO_ELIGIBLE_AGENTS`, `FAILED`) **commit** with agent gone. Also occurs when distribution is **on** but no one is eligible. |

### Proposed fix (redistribution safety)

| Part | Change | Fixes |
| ---- | ------ | ----- |
| **Part 1 â€” Toggle guard** | In `handleRedistribute`, before any removal: `getDistributionFlags(orgId, teamId)`. If both org and team off â†’ `ActionResult.SKIPPED`, `reason: distribution_disabled`. Other tier actions still run. | Intent (A) |
| **Part 2 â€” Confirm before remove** | In `redistribute()`, only remove stakeholders **after** confirmed replacement (`ASSIGNED`). On pool / no eligible / failed â†’ retain assignment, return non-assigned outcome. | Safety (B), all failure modes |
| **Part 3 â€” UI warning (optional)** | Warn when saving escalation rule with `REDISTRIBUTE` tier while distribution disabled. | UX only |
