## Executive Summary

The escalation and distribution engines are **architecturally sound for production** at ~100 orgs: two-stage escalation dispatch, `singletonKey` dedup, bounded trigger scans, and distribution batch caps prevent runaway crashes. The real constraints are **connection-pool budget**, **unbounded due-tracker execution per org job**, **scan query cost at large lead volume**, and **two correctness gaps** (silent dropped distribution enqueues and destructive `REDISTRIBUTE` when distribution is off).

This document proposes a prioritized fix plan: pool-safe throughput bounds, self-healing reconciliation, observability, and redistribution safety — without increasing concurrent transactions beyond what `DB_POOL_MAX` can sustain.

## Current Architecture (Reference)

### Escalation (Two-Stage pg-boss Model)

The escalation system uses a two-stage architecture to safely distribute work across multiple organizations:

| Stage | Queue / Cron | Behavior |
|-------|--------------|----------|
| **Stage 1 — Dispatcher** | `escalation-scheduler`, cron `* * * * *` (~60s) | Cluster-wide tick: `UNION` of orgs with active rules or active trackers → one `escalation-org` job per org (`singletonKey: esc-org:<orgId>`) |
| **Stage 2 — Per-org worker** | `escalation-org`, `batchSize: 5`, `stately` policy | Up to 5 orgs in parallel; per org: `TriggerDetector.scan()` (LIMIT 500/rule) → due trackers → sequential `executeTrackerTier()` |

<Check>
**Safeguards already in place:**
- Partial unique index + `ON CONFLICT DO NOTHING` on tracker insert
- `singletonKey` drops duplicate org jobs while one is in-flight
- Per-tracker try/catch
- `expireInSeconds: 120` on org jobs
</Check>

### Distribution (Event-Driven + pg-boss)

Distribution handles lead assignment through multiple pathways:

| Path | Trigger | Worker |
|------|---------|--------|
| **Single lead** | `LEAD_CREATED` → `DistributionListener` → `enqueueSingle` (priority 10) | `distribution` queue, `batchSize: 10` |
| **Bulk / backfill** | Import or toggle-on backfill → `enqueueBatch` (priority 0) | Same worker |
| **Escalation REDISTRIBUTE** | `TierExecutor` → `DistributionEngineService.redistribute()` | Direct call (no toggle check) |

<Check>
**Safeguards already in place:**
- `singletonKey: lead:<leadId>`
- Idempotency in engine
- Two-phase capacity (advisory lock + re-count)
- Post-commit `LEAD_ASSIGNED` notifications
- `FAILED` `distribution_log` on exhausted worker retries
</Check>

### What "Noisy" Local Dev Usually Is

<Info>
The following behaviors are **log volume**, not runaway loops:
- Cron dispatcher every ~60s + per-org `.log()` even when `+0 trackers, 0 tiers`
- `EscalationAutoResolver` logging on every activity/stage/stakeholder event
</Info>

## Governing Constraint: Connection Pool Budget

Production default: **`DB_POOL_MAX = 20` per app instance** (`app.config.ts` → `mikro-orm.config.ts`). Every `TenantContext.executeInOrg()` holds a connection for the transaction duration.

### Concurrent Consumers (Worst-Case Ceiling)

| Consumer | Concurrent Transactions | Notes |
|----------|-------------------------|-------|
| Distribution worker | **Up to 10** | `processJobs` uses `Promise.allSettled` over full batch (`distribution-job.handler.ts`) |
| Escalation org worker | **Up to 5** | One connection per org in batch; tracker tiers run **sequentially** per org |
| HTTP / other crons | Remainder | Auth, messaging, search maintenance, user-status, etc. |

<Warning>
**Worst case on one instance:** ~**15 of 20** slots during overlap (bulk import + escalation cycle). This is why **parallelizing the inner escalation tracker loop is the wrong lever** — it would multiply against the 5-org batch and blow the pool.
</Warning>

### Multi-Instance Note

pg-boss **cron dispatchers** are cluster-wide (one tick), but **workers** (`escalation-org`, `distribution`) run on **every** app instance. Total pressure on Neon ≈ **`N_instances × DB_POOL_MAX`**, plus pg-boss's own pool (`DATABASE_URL`).

<Tip>
Size pools before scaling horizontally. Treat "total concurrent transactions" as a **fixed budget**. Prefer bounds, sequential inner loops, and horizontal scale with **lower** per-instance `DB_POOL_MAX` — not larger batches.
</Tip>

## Scaling Risks and Mitigations

### Risk 1 — Scan Query Cost (Steady-State, Large Lead Tables)

`TriggerDetector.scan()` runs **one multi-table join per active rule per org every ~60s** (`lead ⋈ entity_stakeholder ⋈ activity`, `LIMIT 500` on results only).

At 100 orgs × 3 rules ≈ **300 scan queries/minute**. Unindexed paths degrade before the pool saturates.

<AccordionGroup>
  <Accordion title="Mitigation Steps">
    | Mitigation | Priority |
    |------------|----------|
    | `EXPLAIN (ANALYZE, BUFFERS)` on NFC + WENT_COLD scans against production-sized data | **Gate (P0)** |
    | Confirm indexes on `entity_stakeholder`, `activity`, `lead`, `escalation_tracker.next_escalation_at` | **Gate (P0)** |
  </Accordion>
</AccordionGroup>

### Risk 2 — Unbounded Due-Tracker Loop (Concrete Bug)

Per org, **all** due trackers are selected (no `LIMIT`) and processed sequentially. A heavy org (e.g. business-hours backlog) can exceed **`expireInSeconds: 120`** → pg-boss expiry → retries → wasted work.

Heavy tiers (`REDISTRIBUTE`, `NOTIFY_ADMIN`) can exceed naive ~100ms/tracker estimates.

<Warning>
This is a concrete bug that must be addressed before production deployment at scale.
</Warning>

<Steps>
  <Step title="Phase 1a: Add batch limit (P1 — do first)">
    `ORDER BY next_escalation_at ASC LIMIT $DUE_TRACKER_BATCH_LIMIT` (~50–100, or wall-clock budget ~60s) on due-tracker query
  </Step>
  
  <Step title="Keep inner loop sequential (Required)">
    Pool-safe processing — do not parallelize tracker execution
  </Step>
  
  <Step title="Drain remainder on next tick (Default)">
    ~5 min for 1000 trackers @ 200/batch
  </Step>
  
  <Step title="Optional Phase 1b: Continuation job (P5 — defer)">
    Distinct `singletonKey` (`esc-org-cont:<orgId>`) only if drain latency is unacceptable
  </Step>
</Steps>

<Info>
Mirror precedent: `SCAN_BATCH_LIMIT = 500` in `trigger-detector.ts`.
</Info>

### Risk 3 — Silent Dropped Distribution Enqueue

`DistributionListener.enqueueWithRetry` retries 3×, then **only logs** — lead is created (HTTP 201) but **no job** and **no `distribution_log`**.

<AccordionGroup>
  <Accordion title="Mitigation Plan">
    | Mitigation | Priority |
    |------------|----------|
    | **Phase 2:** Stuck-lead reconciliation cron (~10–15 min), re-enqueue via `enqueueBatch` | **P2 — highest silent-failure value** |
    | **Phase 3a:** On enqueue exhaustion, write `distribution_log` with `outcome: FAILED`, `failureReason: enqueue_failed` | **P3** |
    | **Phase 3b:** Metrics: failed logs window, `escalation-org` retry exhaustion, pg-boss failed-job depth | **P3** |
  </Accordion>
  
  <Accordion title="Stuck-Lead Definition">
    **All must hold:**
    - Distribution enabled for org/team
    - Created > grace period (e.g. 10 min)
    - Not archived / not deleted
    - **No** user or team stakeholder
    - **No** terminal `distribution_log` outcome: `ASSIGNED | FALLBACK_POOL | FALLBACK_TEAM | QUEUED | ALREADY_ASSIGNED` (zero logs, or only `FAILED` logs)
  </Accordion>
  
  <Accordion title="Critical Filters">
    **`FALLBACK_POOL` is terminal** — leads intentionally in the pool must not be re-enqueued.
    
    **False-positive guard:** Manual unassignment leaves no terminal log. Sweeper should target **never-attempted** leads (e.g. no `distribution_log` at all, or only `enqueue_failed` / worker `FAILED`) — not leads an admin deliberately left unassigned.
    
    **Module boundary:** No cross-module raw SQL. Expose e.g. `LeadService.findUndistributedLeadIdsInTransaction(...)` + compose with `EntityStakeholderService`; join `distribution_log` in distribution module only.
  </Accordion>
</AccordionGroup>

### Risk 4 — Pool Saturation Under Burst

Bulk import (`priority: 0`, batch 10) overlapping escalation cycle.

| Mitigation | Priority |
|------------|----------|
| Size `DB_POOL_MAX` so `N × max` < Neon ceiling | **Gate** |
| Optional: env-configurable `distribution` `batchSize` | **P5** |
| Horizontal scale: pg-boss `SKIP LOCKED` — safe multi-instance; reduce per-instance pool | When volume demands |

### Risk 5 — Observability Gap

Operators cannot see queue depth, per-org cycle duration, or enqueue failures without log diving.

| Mitigation | Priority |
|------------|----------|
| Wire `PgBossHealthIndicator` into readiness | **P3** |
| Reduce per-cycle `.log()` to `.debug()` when `+0 trackers, 0 tiers` | Low effort |
| Optional prod kill-switch env for escalation cron (test has `ESCALATION_SKIP_PGBOSS_WORKERS`) | Nice-to-have |

## Correctness Gap: REDISTRIBUTE vs Distribution Toggle

### Investigation Summary

We analyzed escalation `REDISTRIBUTE` when org-level distribution is **off**.

### What We Found (Verified in Code)

<Steps>
  <Step title="distributionEnabled is a trigger gate, not an engine kill-switch">
    Checked in `DistributionListener.handleLeadCreated` and team Path B routing only. Org pipeline (`runDistribution`) still runs when off — **"off" = manual assignment mode**, not "engine disabled".
  </Step>
  
  <Step title="Escalation REDISTRIBUTE bypasses the toggle">
    `TierExecutor.handleRedistribute` → `DistributionEngineService.redistribute()` with **no** `distributionEnabled` check (`tier-executor.ts`).
  </Step>
  
  <Step title="redistribute() removes the current agent up-front">
    `removeAllForEntityInTransaction` runs **before** the pipeline (`distribution-engine.service.ts`), with `excludeAgentIds: [previousAgent]`.
  </Step>
  
  <Step title="Default off state → POOL fallback">
    No matching rule + `defaultMethod = POOL` → `FALLBACK_POOL` log, **no** new stakeholder, agent availability ignored.
  </Step>
  
  <Step title="Auto-create side effect">
    Missing `DistributionSettings` row is created with `distributionEnabled = false`, `defaultMethod = POOL` — then pipeline proceeds anyway.
  </Step>
</Steps>

### The Gap

<Warning>
`REDISTRIBUTE` is **destructive** when distribution is off: removes Agent X, fails to reassign, leaves lead **orphaned in pool**. Escalation logs tier action `FAILED`; un-assignment is already committed.
</Warning>

<Tabs>
  <Tab title="Problem A — Intent Mismatch">
    Escalation overrides operator's "distribution off / manual mode".
  </Tab>
  
  <Tab title="Problem B — Decision Ordering">
    Agent removed **before** placement confirmed. Removal + redistribution share one transaction (atomic by design); non-`ASSIGNED` outcomes (`FALLBACK_POOL`, `NO_ELIGIBLE_AGENTS`, `FAILED`) **commit** with agent gone. Also occurs when distribution is **on** but no one is eligible.
  </Tab>
</Tabs>

### Proposed Fix (Redistribution Safety)

<CodeGroup>

```typescript Part 1 — Toggle Guard
// In handleRedistribute, before any removal:
const flags = await getDistributionFlags(orgId, teamId);
if (!flags.orgEnabled && !flags.teamEnabled) {
  return ActionResult.SKIPPED('distribution_disabled');
}
// Other tier actions still run
```

```typescript Part 2 — Confirm Before Remove
// In redistribute(), only remove stakeholders AFTER confirmed replacement:
const outcome = await runDistributionPipeline(leadId, orgId);
if (outcome === 'ASSIGNED') {
  await removeAllForEntityInTransaction(leadId, excludeAgentIds);
  return outcome;
}
// On pool / no eligible / failed → retain assignment
return outcome;
```

```typescript Part 3 — UI Warning (Optional)
// Warn when saving escalation rule with REDISTRIBUTE tier
// while distribution disabled
if (hasTier('REDISTRIBUTE') && !distributionEnabled) {
  showWarning('REDISTRIBUTE tier will not reassign leads while distribution is disabled');
}
```

</CodeGroup>

| Part | Change | Fixes |
|------|--------|-------|
| **Part 1 — Toggle guard** | In `handleRedistribute`, before any removal: `getDistributionFlags(orgId, teamId)`. If both org and team off → `ActionResult.SKIPPED`, `reason: distribution_disabled`. Other tier actions still run. | Intent (A) |
| **Part 2 — Confirm before remove** | In `redistribute()`, only remove stakeholders **after** confirmed replacement (`ASSIGNED`). On pool / no eligible / failed → retain assignment, return non-assigned outcome. | Safety (B), all failure modes |
| **Part 3 — UI warning (optional)** | Warn when saving escalation rule with `REDISTRIBUTE` tier while distribution disabled. | UX only |
