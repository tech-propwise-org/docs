<Note>
The Escalation Engine focuses on **reassignment workflows** for leads requiring ownership changes. For lead activity monitoring with **agent notifications** (without forced reassignment), see the separate **Activity Monitor module**.

Both systems complement each other:
- **Escalation Engine**: Handles reassignment when agents fail to make first contact
- **Activity Monitor**: Handles notification alerts for ongoing lead activity issues
</Note>

The escalation module provides automated **reassignment workflows** for leads that were never contacted after assignment (`NO_FIRST_CONTACT` trigger only). It enables organizations to define tiered escalation rules that progressively take actions (notifications, reassignments, etc.) when an agent fails to make first contact with a newly assigned lead.

The module consists of three main components:

<CardGroup cols={3}>
  <Card title="Escalation Rules" icon="gear">
    Define when and how escalations should trigger
  </Card>
  <Card title="Escalation Trackers" icon="clock">
    Track active escalations and their execution history
  </Card>
  <Card title="Escalation Tiers" icon="layer-group">
    Multi-level actions that execute in sequence with configurable delays
  </Card>
</CardGroup>

## Architecture

The escalation system follows an event-driven architecture with scheduled evaluation of trigger conditions:

<Steps>
  <Step title="Rule Definition">
    Organizations create escalation rules with specific triggers and multi-tier actions
  </Step>
  <Step title="Condition Evaluation">
    Background jobs periodically evaluate leads against active escalation rules
  </Step>
  <Step title="Tracker Creation">
    When conditions are met, an escalation tracker is created to manage execution
  </Step>
  <Step title="Tier Execution">
    Actions are executed in sequence based on tier delays and configurations
  </Step>
  <Step title="Resolution">
    Escalations resolve automatically when conditions change or manually by users
  </Step>
</Steps>

## Core Entities

### EscalationRule

Defines the conditions under which escalations should trigger and the actions to take.

| Field | Type | Description |
|-------|------|-------------|
| `id` | UUID | Primary key |
| `organization` | Organization | RLS boundary |
| `createdBy` | User | Rule creator |
| `name` | string | Human-readable rule name |
| `description` | string | Optional rule description |
| `isActive` | boolean | Whether rule is currently enabled |
| `triggerType` | TriggerType | Type of condition that triggers escalation |
| `triggerConfig` | JSON | Threshold configuration |
| `applyTo` | EscalationApplyTo | Scope of leads this rule applies to |
| `applyToConfig` | JSON | Additional scope configuration |
| `tiers` | EscalationTier[] | Ordered list of escalation actions |

#### Supported Trigger Types

<Tabs>
  <Tab title="Active">
    | Type | Description |
    |------|-------------|
    | `NO_FIRST_CONTACT` | Fires when an agent never makes first contact after assignment |
  </Tab>
  
  <Tab title="Deprecated">
    <Warning>
    The following trigger types have been moved to the Activity Monitor module:
    - `NO_ACTIVITY`
    - `STALE_LEAD` 
    - `STAGE_STUCK`
    - `CUSTOM`
    </Warning>
  </Tab>
</Tabs>

#### Trigger Configuration

For `NO_FIRST_CONTACT` triggers, configure the threshold in minutes:

```json
{
  "thresholdMinutes": 60
}
```

<Info>
The `thresholdHours`, `thresholdDays`, and `stageIds` fields are kept for historical data readability only and are not used for new rule evaluation.
</Info>

### EscalationTier

Individual action levels within an escalation rule, executed in sequence.

| Field | Type | Description |
|-------|------|-------------|
| `id` | UUID | Primary key |
| `organization` | Organization | RLS boundary |
| `escalationRule` | EscalationRule | Parent rule |
| `tierOrder` | number | Execution order (1, 2, 3, etc.) |
| `delayHours` | number | Hours to wait before executing this tier |
| `actions` | JSON[] | Array of actions to execute |

#### Supported Action Types

<AccordionGroup>
  <Accordion title="Notification Actions">
    | Type | Parameters | Description |
    |------|-----------|-------------|
    | `NOTIFY_AGENT` | `message?` | Notify the lead's currently assigned agent |
    | `NOTIFY_ADMIN` | `message?` | Self-resolving — notifies all org users with `system.admin` permission |
    | `NOTIFY_TEAM_LEAD` | `message?` | Self-resolving — notifies all team members with `team.admin` permission in the lead's team |
  </Accordion>
  
  <Accordion title="Lead Management Actions">
    | Type | Parameters | Description |
    |------|-----------|-------------|
    | `REDISTRIBUTE` | _(no params)_ | Delegates to DistributionEngine; removes current stakeholders and re-runs full pipeline |
    | `CHANGE_TEMPERATURE` | `temperature: 'hot' \| 'warm' \| 'cold'` | Change lead temperature |
    | `ADD_TAG` | `tagIds: string[]` | Add tags to the lead |
  </Accordion>
</AccordionGroup>

<Warning>
`REDISTRIBUTE` actions must be placed in the last tier. On successful assignment, the tracker is immediately resolved and no further tiers will execute.
</Warning>

#### Action Configuration Examples

<CodeGroup>
```json Notify Agent
{
  "type": "NOTIFY_AGENT",
  "message": "Lead needs attention"
}
```

```json Notify Admins
{
  "type": "NOTIFY_ADMIN", 
  "message": "Escalation alert"
}
```

```json Redistribute Lead
{
  "type": "REDISTRIBUTE"
}
```

```json Change Temperature
{
  "type": "CHANGE_TEMPERATURE",
  "temperature": "hot"
}
```

```json Add Tags
{
  "type": "ADD_TAG",
  "tagIds": ["VIP", "Hot Buyer"]
}
```
</CodeGroup>

### EscalationTracker

Tracks active escalations and their execution history for specific leads.

| Field | Type | Description |
|-------|------|-------------|
| `id` | UUID | Primary key |
| `organization` | Organization | RLS boundary |
| `lead` | Lead | Lead being escalated |
| `escalationRule` | EscalationRule | Rule that triggered escalation |
| `status` | EscalationStatus | Current escalation state |
| `triggeredAt` | Date | When escalation was first triggered |
| `resolvedAt` | Date | When escalation was resolved |
| `resolvedBy` | ResolvedBy | How escalation was resolved |
| `currentTier` | number | Currently executing tier (1-based) |
| `nextTierAt` | Date | When next tier should execute |
| `executionHistory` | JSON[] | Log of all tier executions |

#### Execution History Format

```json
{
  "tier": 1,
  "executedAt": "2024-01-15T10:30:00Z",
  "actions": [
    {
      "type": "NOTIFY_AGENT",
      "status": "SUCCESS", 
      "details": { "notificationId": "uuid" }
    }
  ]
}
```

## Business Rules

### Trigger Evaluation

<Check>
- Rules are evaluated periodically by background jobs (typically every 15-30 minutes)
- Only active rules (`isActive = true`) are considered
- Leads with existing active escalations are skipped to prevent duplicates
- Trigger conditions account for business hours and timezone settings
</Check>

### Tier Execution

- Tiers execute in order based on `tierOrder` field
- `delayHours` is calculated from the escalation's `triggeredAt` timestamp
- If a tier fails partially, successful actions are not retried
- Execution continues to next tier regardless of individual action failures

### Resolution Conditions

<Tabs>
  <Tab title="Manual">
    **MANUAL** - User explicitly resolves via UI/API
  </Tab>
  
  <Tab title="Automatic">
    - **AUTO_ACTIVITY** - New activity added to lead
    - **AUTO_STAGE_CHANGE** - Lead moves to different stage
    - **AUTO_REASSIGNMENT** - Lead reassigned to different agent
    - **REDISTRIBUTED** - Resolved immediately after successful redistribution
  </Tab>
</Tabs>

### Scope Application

| Scope | Description |
|-------|-------------|
| **ALL** | Applies to all leads in organization |
| **SPECIFIC_STAGES** | Only leads in configured stages |
| **SPECIFIC_TEMPERATURES** | Only leads with configured temperatures |

API endpoints are currently being developed. This section will be updated when controllers are implemented.

### Escalation Rules Management

#### List escalation rules
```http
GET /escalation-rules
```
**Authorization:** `MANAGE_ESCALATION` permission

#### Create escalation rule
```http
POST /escalation-rules
```
**Authorization:** `MANAGE_ESCALATION` permission

#### Update escalation rule
```http
PUT /escalation-rules/:id
```
**Authorization:** `MANAGE_ESCALATION` permission

#### Delete escalation rule
```http
DELETE /escalation-rules/:id
```
**Authorization:** `MANAGE_ESCALATION` permission

<Warning>
Deleting a rule will immediately cancel all active trackers for that rule.
</Warning>

### Escalation Analytics

#### Get escalation metrics
```http
GET /escalation-analytics?dateFrom={ISO8601}&dateTo={ISO8601}&ruleId={optional}
```
**Authorization:** `VIEW_DISTRIBUTION_ANALYTICS` permission

**Query Parameters:**
- `dateFrom` (required): Start date (ISO 8601)
- `dateTo` (required): End date (ISO 8601, max 90 days from dateFrom)
- `ruleId` (optional): Filter by specific rule

## Implementation Patterns

### Naming Conventions

- Entity names use singular form: `EscalationRule`, `EscalationTier`
- Enum values use UPPER_SNAKE_CASE: `NO_FIRST_CONTACT`, `NOTIFY_AGENT`
- Database indexes follow pattern: `idx_{table}_{columns}`

### JSON Configuration

- All configuration objects use camelCase field names
- Optional fields are omitted rather than set to null
- Action configurations include `type` field for polymorphic handling

### Error Handling

<Tip>
- Escalation execution failures are logged but do not prevent progression
- Invalid configurations are validated at rule creation time
- Orphaned escalations are cleaned up by maintenance jobs
</Tip>

### Performance Considerations

- Escalation evaluation is optimized with database indexes on trigger-relevant fields
- Large organizations may require partitioning of tracker history
- Tier delays are calculated using database timestamps to handle server restarts
