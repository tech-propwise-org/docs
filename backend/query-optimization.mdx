## Query optimization patterns

### EXISTS subqueries for access control

Checking if a user has access to entities often requires loading IDs first, then filtering. This creates two database round trips and memory overhead for large ID arrays.

<Tabs>
<Tab title="Bad pattern">
```typescript
// Step 1: Get all entity IDs where user is stakeholder
const stakeholderLeadIds = await this.em
  .find(EntityStakeholder, {
    entityType: 'lead',
    user: userId,
    isDeleted: false,
  })
  .then((stakes) => stakes.map((s) => s.entityId));

// Step 2: Use IDs in main query
queryBuilder.andWhere({ id: { $in: stakeholderLeadIds } });
```

<Warning>
Two database round trips, memory overhead for large ID arrays, and PostgreSQL `IN` clause parameter limits (~32K parameters) make this pattern unsafe at scale.
</Warning>
</Tab>

<Tab title="Good pattern">
```typescript
queryBuilder.andWhere(
  `(
    lead.created_by_id = ?
    OR EXISTS (
      SELECT 1 FROM entity_stakeholder es
      WHERE es.entity_type = 'lead'
      AND es.entity_id = lead.id
      AND es.user_id = ?
      AND es.is_deleted = false
    )
  )`,
  [userId, userId],
);
```

<Tip>
Single database round trip. The database optimizer uses indexes efficiently, there is no memory overhead for ID arrays, and it works with any number of records.
</Tip>
</Tab>
</Tabs>

### NOT EXISTS for pool/unassigned queries

Finding entities without a primary stakeholder (lead pool) should use `NOT EXISTS` instead of loading all entities and filtering in application code.

<Tabs>
<Tab title="Bad pattern">
```typescript
// Get all leads
const allLeads = await this.em.find(Lead, { isDeleted: false });

// Get leads with primary stakeholder
const assignedIds = await this.em
  .find(EntityStakeholder, {
    entityType: 'lead',
    isPrimary: true,
    isDeleted: false,
  })
  .then((s) => s.map((x) => x.entityId));

// Filter in application
const unassignedLeads = allLeads.filter((l) => !assignedIds.includes(l.id));
```
</Tab>

<Tab title="Good pattern">
```typescript
queryBuilder.andWhere(
  `NOT EXISTS (
    SELECT 1 FROM entity_stakeholder es
    WHERE es.entity_type = 'lead'
    AND es.entity_id = lead.id
    AND es.is_primary = true
    AND es.is_deleted = false
  )`,
);
```
</Tab>
</Tabs>

Use `NOT EXISTS` for lead pool (unassigned leads), orphaned entities, and records missing required relationships.

### IN with subquery for team access

Fetching entities assigned to teams where the user has permissions uses a small `IN` clause, which is acceptable because team count per user is typically under 10.

```typescript
// First, get team IDs (this is usually a small set)
// Use AccessService for centralized permission checking
const teamIds = await this.accessService.getUserTeamsWithPermissions(userId, [
  'team_crm.view',
  'team_crm.manage',
]);

if (teamIds.length === 0) {
  return { items: [], total: 0, page: 1, limit: 10, hasMore: false };
}

// Use IN clause with team IDs (small set is OK)
const teamIdsParam = teamIds.map(() => '?').join(', ');
queryBuilder.andWhere(
  `EXISTS (
    SELECT 1 FROM entity_stakeholder es
    WHERE es.entity_type = 'lead'
    AND es.entity_id = lead.id
    AND es.team_id IN (${teamIdsParam})
    AND es.is_deleted = false
  )`,
  teamIds,
);
```

<Tip>
Use `AccessService.getUserTeamsWithPermissions()` for centralized permission checking. This is acceptable because team count per user is small and avoids complex subquery for team membership.
</Tip>

### Proper JOIN loading

Prevent N+1 queries by eagerly loading related entities with `leftJoinAndSelect`.

<Tabs>
<Tab title="Bad pattern">
```typescript
const leads = await this.em.find(Lead, { isDeleted: false });
// Each access to lead.person triggers a query
for (const lead of leads) {
  console.log(lead.person.firstName); // Query per lead!
}
```
</Tab>

<Tab title="Good pattern">
```typescript
const queryBuilder = this.em.createQueryBuilder(Lead, 'lead');

// Load all needed relations upfront
queryBuilder.leftJoinAndSelect('lead.person', 'person');
queryBuilder.leftJoinAndSelect('person.emails', 'emails');
queryBuilder.leftJoinAndSelect('person.phones', 'phones');
queryBuilder.leftJoinAndSelect('lead.stage', 'stage');
queryBuilder.leftJoinAndSelect('lead.company', 'company');
queryBuilder.leftJoinAndSelect('lead.createdBy', 'createdBy');
```
</Tab>
</Tabs>

**Join decision matrix:**

| Relation        | Always join? | Notes                           |
| --------------- | ------------ | ------------------------------- |
| `person`        | ✅ Yes       | Core identity, always needed    |
| `person.emails` | ✅ Yes       | Primary email usually displayed |
| `person.phones` | ✅ Yes       | Primary phone usually displayed |
| `stage`         | ✅ Yes       | Required for pipeline views     |
| `company`       | ⚡ Optional  | Join if showing company info    |
| `stakeholders`  | ❌ No        | Separate query or lazy load     |
| `activities`    | ❌ No        | Paginated separately            |
| `notes`         | ❌ No        | Paginated separately            |

### When to use raw SQL

Use **MikroORM QueryBuilder** for standard CRUD operations, simple filters and joins, `ORDER BY` / `LIMIT` / `OFFSET`, and basic aggregations.

Use **raw SQL** for complex subqueries (`EXISTS`, `NOT EXISTS`, correlated), window functions, CTEs (Common Table Expressions), database-specific optimizations, and complex `GROUP BY` with `HAVING`.

**Example - Raw SQL for Complex Access Control:**

```typescript
// Complex access pattern requiring raw SQL
queryBuilder.andWhere(
  `(
    lead.created_by_id = ?
    OR EXISTS (
      SELECT 1 FROM entity_stakeholder es
      WHERE es.entity_type = 'lead'
      AND es.entity_id = lead.id
      AND (
        es.user_id = ?
        OR es.team_id IN (
          SELECT team_id FROM team_member
          WHERE user_id = ? AND role = 'leader'
        )
      )
      AND es.is_deleted = false
    )
  )`,
  [userId, userId, userId],
);
```

### View type architecture

Multiple tabs/views require different access control logic (My Leads, All Leads, Pool, Team Leads, etc.). Use a single endpoint with an optional `type` parameter instead of separate endpoints.

<Tabs>
<Tab title="Bad pattern">
```typescript
// Separate endpoints lead to code duplication
@Get('/my')
async getMyLeads() { ... }

@Get('/all')
async getAllLeads() { ... }

@Get('/pool')
async getPoolLeads() { ... }

@Get('/team')
async getTeamLeads() { ... }
```
</Tab>

<Tab title="Good pattern">
```typescript
// Single endpoint with optional type discrimination
export enum LeadViewType {
  ALL = 'all',
  MY = 'my',
  POOL = 'pool',
  TEAM = 'team',
  TEAM_POOL = 'teamPool',
}

@Get()
async getLeads(
  @Query('type') type?: LeadViewType,  // Optional: omit to fetch all tabs
  @Query('teamId') teamId?: string,
) {
  return this.service.getLeads(user, { type, teamId, ...params });
}
```
</Tab>
</Tabs>

**Behavior:**

- **No `type` parameter**: Returns all available tabs concurrently (initial page load)
- **With `type` parameter**: Returns only that specific tab (for "load more" pagination)

This gives you a single source of truth for query building, consistent filter/pagination logic, and easy extensibility for new view types.

### Permission validation before query building

Building complex queries for users who lack access wastes resources. Validate permissions first, then build the query.

```typescript
private validateTypeAccess(user: any, type: LeadViewType, teamId?: string): void {
  const perms = this.getLeadPermissions(user);

  switch (type) {
    case LeadViewType.ALL:
      if (!perms.canViewAll) {
        throw new ForbiddenException(
          'Access denied: requires system.admin, crm.view, or crm.manage permission',
        );
      }
      break;

    case LeadViewType.TEAM:
      if (!perms.canViewTeam) {
        throw new ForbiddenException(
          'Access denied: requires team_crm.view or team_crm.manage permission',
        );
      }
      if (!teamId) {
        throw new ForbiddenException('teamId is required for team view type');
      }
      break;

    case LeadViewType.MY:
    case LeadViewType.POOL:
      break;
  }
}

async getLeads(user: any, query: LeadsQuery): Promise<LeadsResponse> {
  this.validateTypeAccess(user, query.type, query.teamId);

  const qb = this.buildQueryForType(user, query.type, query.teamId);
  // ...
}
```

<Tip>
Fail-fast principle prevents wasted database queries. Centralized permission logic is easier to audit and produces clear error messages for the frontend.
</Tip>

### Reusable access control filter methods

Extract access control logic into private filter methods per view type so they are testable in isolation and reusable across Kanban and List views.

```typescript
private applyMyLeadsFilter(qb: any, userId: string): void {
  qb.andWhere(
    `(
      lead.created_by_id = ?
      OR EXISTS (
        SELECT 1 FROM entity_stakeholder es
        WHERE es.entity_type = 'lead'
        AND es.entity_id = lead.id
        AND es.user_id = ?
        AND es.is_deleted = false
      )
    )`,
    [userId, userId],
  );
}

private applyPoolLeadsFilter(qb: any): void {
  qb.andWhere(
    `NOT EXISTS (
      SELECT 1 FROM entity_stakeholder es
      WHERE es.entity_type = 'lead'
      AND es.entity_id = lead.id
      AND es.is_deleted = false
    )`,
  );
}

private applyTeamLeadsFilter(qb: any, teamId: string): void {
  qb.andWhere(
    `EXISTS (
      SELECT 1 FROM entity_stakeholder es
      WHERE es.entity_type = 'lead'
      AND es.entity_id = lead.id
      AND es.team_id = ?
      AND es.is_deleted = false
    )`,
    [teamId],
  );
}

private applyTeamPoolLeadsFilter(qb: any, teamId: string): void {
  qb.andWhere(
    `EXISTS (
      SELECT 1 FROM entity_stakeholder es
      WHERE es.entity_type = 'lead'
      AND es.entity_id = lead.id
      AND es.team_id = ?
      AND es.user_id IS NULL
      AND es.is_deleted = false
    )`,
    [teamId],
  );
}
```

Wire them together in a switch:

```typescript
private buildLeadQueryForType(user: any, type: LeadViewType, teamId?: string) {
  const qb = this.em.createQueryBuilder(Lead, 'lead');

  switch (type) {
    case LeadViewType.ALL:
      break;
    case LeadViewType.MY:
      this.applyMyLeadsFilter(qb, user.id);
      break;
    case LeadViewType.POOL:
      this.applyPoolLeadsFilter(qb);
      break;
    case LeadViewType.TEAM:
      this.applyTeamLeadsFilter(qb, teamId!);
      break;
    case LeadViewType.TEAM_POOL:
      this.applyTeamPoolLeadsFilter(qb, teamId!);
      break;
  }

  return qb;
}
```

### Centralized team permission lookup

Use `AccessService` for permission-based team lookups instead of hardcoding role names in each service.

<Tabs>
<Tab title="Bad pattern">
```typescript
private async getUserLeadershipTeams(userId: string): Promise<string[]> {
  const memberships = await this.em.find(TeamMembership, { user: userId });
  return memberships
    .filter((tm) => tm.teamRoles.some((tmr) => tmr.role.name === 'Team Leader'))
    .map((tm) => tm.team.id);
}
```

<Warning>
Hardcoded role names are fragile and duplicated across every service that needs team checks.
</Warning>
</Tab>

<Tab title="Good pattern">
```typescript
async getUserTeamsWithPermissions(
  userId: string,
  requiredPermissions: string[],
): Promise<string[]> {
  const memberships = await this.em.find(TeamMembership, {
    user: userId,
    isDeleted: false,
  }, {
    populate: ['team', 'teamRoles', 'teamRoles.role', 'teamRoles.role.permissions'],
  });

  return memberships
    .filter((m) => m.teamRoles.some((tr) =>
      tr.role.permissions.some((p) => requiredPermissions.includes(p.key))
    ))
    .map((m) => m.team.id);
}

const teamIds = await this.accessService.getUserTeamsWithPermissions(userId, [
  'team_crm.view',
  'team_crm.manage',
]);
```

</Tab>

## Access control patterns

### User access levels

```typescript
async getLeadsWithAccessControl(
  userId: string,
  userRole: string,
  orgId: string,
): Promise<Lead[]> {
  const queryBuilder = this.em.createQueryBuilder(Lead, 'lead');

  queryBuilder.andWhere({ isDeleted: false, organization: orgId });

  if (userRole === 'admin' || userRole === 'owner') {
    // Full org access
  } else if (userRole === 'manager') {
    const teamIds = await this.getUserManagedTeams(userId);
    this.applyTeamAccessFilter(queryBuilder, teamIds);
  } else {
    this.applyUserAccessFilter(queryBuilder, userId);
  }

  return queryBuilder.getResultList();
}
```

### Contact visibility

Contacts are visible via both Lead and Deal stakeholder chains. A contact is accessible if the user is a stakeholder on any lead linked through the contact's person, or on any deal the contact is linked to.

```typescript
async getMyContacts(userId: string, orgId: string): Promise<Contact[]> {
  const queryBuilder = this.em.createQueryBuilder(Contact, 'contact');

  queryBuilder.leftJoinAndSelect('contact.person', 'person');
  queryBuilder.andWhere({ isDeleted: false, organization: orgId });

  queryBuilder.andWhere(
    `(
      EXISTS (
        SELECT 1 FROM lead l
        JOIN entity_stakeholder es ON es.entity_type = 'lead'
          AND es.entity_id = l.id
        WHERE l.person_id = contact.person_id
        AND es.user_id = ?
        AND es.is_deleted = false
        AND l.is_deleted = false
      )
      OR
      EXISTS (
        SELECT 1 FROM deal_contact dc
        JOIN entity_stakeholder es ON es.entity_type = 'deal'
          AND es.entity_id = dc.deal_id
        WHERE dc.contact_id = contact.id
        AND es.user_id = ?
        AND es.is_deleted = false
        AND dc.is_deleted = false
      )
    )`,
    [userId, userId],
  );

  return queryBuilder.getResultList();
}
```

### Company visibility

Companies are visible via the `ContactCompanyRole` -> Contact -> Lead/Deal stakeholder chain.

```typescript
async getMyCompanies(userId: string, orgId: string): Promise<Company[]> {
  const queryBuilder = this.em.createQueryBuilder(Company, 'company');

  queryBuilder.andWhere({ isDeleted: false });

  queryBuilder.andWhere(
    `(
      EXISTS (
        SELECT 1 FROM contact_company_role ccr
        JOIN contact c ON c.id = ccr.contact_id
        JOIN lead l ON l.person_id = c.person_id
        JOIN entity_stakeholder es ON es.entity_type = 'lead'
          AND es.entity_id = l.id
        WHERE ccr.company_id = company.id
        AND es.user_id = ?
        AND es.is_deleted = false
        AND l.is_deleted = false
        AND ccr.is_deleted = false
      )
      OR
      EXISTS (
        SELECT 1 FROM contact_company_role ccr
        JOIN deal_contact dc ON dc.contact_id = ccr.contact_id
        JOIN entity_stakeholder es ON es.entity_type = 'deal'
          AND es.entity_id = dc.deal_id
        WHERE ccr.company_id = company.id
        AND es.user_id = ?
        AND es.is_deleted = false
        AND dc.is_deleted = false
        AND ccr.is_deleted = false
      )
    )`,
    [userId, userId],
  );

  return queryBuilder.getResultList();
}
```

## Pagination patterns

### Always get total count

Clone the query builder before applying `LIMIT`/`OFFSET` so the count reflects the full filtered result set, not just the current page.

<Tabs>
<Tab title="Bad pattern">
```typescript
let total = 0;
if (page === 1) {
  total = await queryBuilder.clone().getCount();
}
```
</Tab>

<Tab title="Good pattern">
```typescript
const countQuery = queryBuilder.clone();
const total = await countQuery.getCount();

queryBuilder.limit(limit).offset((page - 1) * limit);
const items = await queryBuilder.getResultList();

return {
  items,
  total,
  page,
  limit,
  hasMore: page * limit < total,
};
```
</Tab>
</Tabs>

### Kanban/board pagination

For board views, each column needs limited items plus a `hasMore` indicator. Use `Promise.all` to query all stages concurrently.

```typescript
async getLeadsByStages(
  userId: string,
  stageIds: string[],
  perStage: number = 20,
): Promise<Record<string, { leads: Lead[]; hasMore: boolean; count: number }>> {
  const result: Record<string, { leads: Lead[]; hasMore: boolean; count: number }> = {};

  await Promise.all(stageIds.map(async (stageId) => {
    const queryBuilder = this.em.createQueryBuilder(Lead, 'lead');

    this.applyAccessControl(queryBuilder, userId);
    queryBuilder.andWhere({ stage: stageId, isDeleted: false });

    const count = await queryBuilder.clone().getCount();

    queryBuilder.limit(perStage + 1);
    const leads = await queryBuilder.getResultList();

    const hasMore = leads.length > perStage;
    if (hasMore) leads.pop();

    result[stageId] = { leads, hasMore, count };
  }));

  return result;
}
```

### Cursor-based pagination

When offset pagination becomes slow on large datasets, use cursor-based pagination with a compound sort key.

```typescript
async getLeadsWithCursor(
  cursor?: string,
  limit: number = 20,
): Promise<{ items: Lead[]; nextCursor?: string }> {
  const queryBuilder = this.em.createQueryBuilder(Lead, 'lead');

  if (cursor) {
    const { id, createdAt } = JSON.parse(Buffer.from(cursor, 'base64').toString());
    queryBuilder.andWhere(
      `(lead.created_at < ? OR (lead.created_at = ? AND lead.id < ?))`,
      [createdAt, createdAt, id],
    );
  }

  queryBuilder.orderBy({ createdAt: 'DESC', id: 'DESC' });
  queryBuilder.limit(limit + 1);

  const items = await queryBuilder.getResultList();
  const hasMore = items.length > limit;

  if (hasMore) items.pop();

  const nextCursor = hasMore && items.length > 0
    ? Buffer.from(JSON.stringify({
        id: items[items.length - 1].id,
        createdAt: items[items.length - 1].createdAt,
      })).toString('base64')
    : undefined;

  return { items, nextCursor };
}
```

## Business logic verification

This section covers generic patterns for implementing business logic. For module-specific rules, see the [module-specific business rules](#module-specific-business-rules) section.

### Business logic checklist template

When implementing any entity, verify these common patterns:

| Pattern                | Implementation                                      | Verification                               |
| ---------------------- | --------------------------------------------------- | ------------------------------------------ |
| Required relationships | FK constraints with `NOT NULL`                      | Schema validation                          |
| Deduplication          | Check unique fields before create                   | Unit test with duplicate data              |
| History/audit tracking | Create history record on state changes              | Verify history records exist after update  |
| Cascading creates      | Auto-create related entities when parent is created | Check related tables after creation        |
| State transitions      | Validate allowed transitions, record changes        | Test invalid transitions throw errors      |
| Soft delete handling   | Always filter `is_deleted = false` in queries       | Verify deleted items don't appear in lists |
| Organization isolation | Always include `organization_id` in queries (RLS)   | Multi-tenant tests                         |

### Deduplication pattern

Check for duplicates before creating any entity that has uniqueness constraints.

```typescript
async createEntity(dto: CreateEntityDto): Promise<Entity> {
  if (dto.uniqueField1 && dto.uniqueField2) {
    const existing = await this.em.findOne(Entity, {
      organization: dto.organizationId,
      uniqueField1: dto.uniqueField1,
      uniqueField2: dto.uniqueField2,
      isDeleted: false,
    });

    if (existing) {
      throw new ConflictException(
        `Entity with this combination already exists`,
      );
    }
  }

  // Proceed with creation...
}
```

| Module      | Entity        | Unique fields                   |
| ----------- | ------------- | ------------------------------- |
| CRM         | Lead          | `referenceId + leadSource`      |
| CRM         | Person        | `email` or `phone` (soft match) |
| Real Estate | Unit          | `development_id + unit_number`  |
| Real Estate | Development   | `name` (within org)             |
| Marketing   | Campaign      | `name` (within org)             |
| Channels    | WhatsApp User | `phone_number`                  |

### History/audit recording pattern

Record state changes in a history table for audit trails and analytics.

```typescript
async updateEntityState(
  entityId: string,
  newStateId: string,
  userId: string,
): Promise<Entity> {
  const entity = await this.em.findOneOrFail(Entity, entityId);
  const previousStateId = entity.state?.id;

  if (previousStateId === newStateId) return entity;

  const lastHistory = await this.em.findOne(EntityStateHistory, {
    entity: entityId,
  }, { orderBy: { changedAt: 'DESC' } });

  const durationStart = lastHistory?.changedAt || entity.createdAt;
  const durationSeconds = Math.floor((Date.now() - durationStart.getTime()) / 1000);

  const history = this.em.create(EntityStateHistory, {
    entity: entityId,
    previousState: previousStateId,
    state: newStateId,
    durationInPreviousState: durationSeconds,
    changedBy: userId,
    changedAt: new Date(),
  });

  entity.state = this.em.getReference(State, newStateId);

  await this.em.persistAndFlush([history, entity]);

  return entity;
}
```

**Use cases by module:**

| Module      | Entity       | State field | History table          |
| ----------- | ------------ | ----------- | ---------------------- |
| CRM         | Lead         | `stage_id`  | `lead_stage_history`   |
| CRM         | Deal         | `stage_id`  | `deal_stage_history`   |
| Real Estate | Unit         | `status`    | `unit_status_history`  |
| Marketing   | Campaign     | `status`    | `campaign_history`     |
| Channels    | Conversation | `status`    | `conversation_history` |

### Aggregate validation pattern

Validate that related records do not exceed business constraints.

```typescript
async validateAggregate(
  parentId: string,
  newValue: number,
  excludeChildId?: string,
): Promise<void> {
  const children = await this.em.find(ChildEntity, {
    parentId,
    isDeleted: false,
    ...(excludeChildId ? { id: { $ne: excludeChildId } } : {}),
  });

  const lockedTotal = children
    .filter(c => c.isLocked)
    .reduce((sum, c) => sum + (c.value || 0), 0);

  const projectedTotal = lockedTotal + newValue;

  if (projectedTotal > MAX_ALLOWED) {
    throw new BadRequestException(
      `Total would exceed ${MAX_ALLOWED}. Current: ${lockedTotal}, ` +
      `New: ${newValue}, Projected: ${projectedTotal}`,
    );
  }
}
```

| Module      | Constraint                        | Max value    |
| ----------- | --------------------------------- | ------------ |
| CRM         | Stakeholder commission total      | 100%         |
| Real Estate | Unit floor allocation percentage  | 100%         |
| Marketing   | Campaign budget allocation        | Total budget |
| Inventory   | Stock allocation across locations | Total stock  |

### Cascading entity creation pattern

When creating an entity from a parent, copy relevant related records.

```typescript
async createChildFromParent(
  parentId: string,
  childData: CreateChildDto,
): Promise<ChildEntity> {
  const parent = await this.em.findOneOrFail(ParentEntity, parentId, {
    populate: ['relatedRecords'],
  });

  const child = this.em.create(ChildEntity, {
    ...childData,
    parent: parent,
    person: parent.person,
    organization: parent.organization,
  });

  const recordsToCopy = await this.em.find(RelatedRecord, {
    parentId,
    shouldCascade: true,
    isDeleted: false,
  });

  for (const record of recordsToCopy) {
    const copiedRecord = this.em.create(RelatedRecord, {
      ...record,
      id: undefined,
      parentId: child.id,
      parentType: 'child',
    });
    this.em.persist(copiedRecord);
  }

  await this.em.persistAndFlush(child);

  return child;
}
```

| Module      | Parent to child          | Cascaded records                |
| ----------- | ------------------------ | ------------------------------- |
| CRM         | Lead to Deal             | Stakeholders (with future flag) |
| CRM         | Lead to Contact          | Person link, tags               |
| Real Estate | Development to Unit      | Amenities, default pricing      |
| Marketing   | Campaign to Campaign Run | Audience segments, content      |

## Anti-patterns to avoid

### Loading all then filtering

<Warning>
Never load all records and filter in application memory. This wastes bandwidth, memory, and CPU.
</Warning>

```typescript
// Never do this
const allLeads = await this.em.find(Lead, {});
const myLeads = allLeads.filter((l) => l.createdBy.id === userId);
```

### Memory-based pagination

<Tabs>
<Tab title="Bad pattern">
```typescript
async getLeads(page: number, limit: number): Promise<Lead[]> {
  const allLeads = await this.em.find(Lead, { isDeleted: false });
  const start = (page - 1) * limit;
  return allLeads.slice(start, start + limit);
}
```

With 10,000 leads this loads ALL records into memory, then throws away 9,950. Memory usage is O(N) instead of O(limit).
</Tab>

<Tab title="Good pattern">
```typescript
async getLeads(page: number, limit: number): Promise<Lead[]> {
  const qb = this.em.createQueryBuilder(Lead, 'lead');
  qb.andWhere({ isDeleted: false });
  qb.limit(limit).offset((page - 1) * limit);
  return qb.getResultList();
}
```
</Tab>
</Tabs>

### N+1 access control checks

<Tabs>
<Tab title="Bad pattern">
```typescript
async getAccessibleLeads(userId: string, leads: Lead[]): Promise<Lead[]> {
  const accessibleLeads: Lead[] = [];

  for (const lead of leads) {
    const canAccess = await this.checkLeadAccess(userId, lead.id);
    if (canAccess) {
      accessibleLeads.push(lead);
    }
  }

  return accessibleLeads;
}
```

100 leads x 1 access check query = 101 queries.
</Tab>

<Tab title="Good pattern">
```typescript
async getAccessibleLeads(userId: string): Promise<Lead[]> {
  const qb = this.em.createQueryBuilder(Lead, 'lead');
  qb.andWhere({ isDeleted: false });

  qb.andWhere(
    `(
      lead.created_by_id = ?
      OR EXISTS (
        SELECT 1 FROM entity_stakeholder es
        WHERE es.entity_type = 'lead'
        AND es.entity_id = lead.id
        AND es.user_id = ?
        AND es.is_deleted = false
      )
    )`,
    [userId, userId],
  );

  return qb.getResultList();
}
```

With the `EXISTS` pattern, a single query replaces 101 — roughly a 100x performance improvement.
</Tab>
</Tabs>

### Multiple sequential queries

<Tabs>
<Tab title="Bad pattern">
```typescript
const lead = await this.em.findOne(Lead, id);
const person = await this.em.findOne(Person, lead.personId);
const emails = await this.em.find(PersonEmail, { person: person.id });
```
</Tab>

<Tab title="Good pattern">
```typescript
const lead = await this.em.findOne(Lead, id, {
  populate: ['person', 'person.emails'],
});
```
</Tab>
</Tabs>

### N+1 in loops

<Tabs>
<Tab title="Bad pattern">
```typescript
const leads = await this.em.find(Lead, {});
for (const lead of leads) {
  const stakeholders = await this.em.find(EntityStakeholder, { entityId: lead.id });
}
```
</Tab>

<Tab title="Good pattern">
```typescript
const leads = await this.em.find(Lead, {});
const leadIds = leads.map((l) => l.id);
const allStakeholders = await this.em.find(EntityStakeholder, {
  entityType: 'lead',
  entityId: { $in: leadIds },
});
const stakeholdersByLead = groupBy(allStakeholders, 'entityId');
```
</Tab>
</Tabs>

### Unbounded queries

<Tabs>
<Tab title="Bad pattern">
```typescript
const allActivities = await this.em.find(Activity, { personId });
```
</Tab>

<Tab title="Good pattern">
```typescript
const activities = await this.em.find(
  Activity,
  { personId },
  {
    limit: 50,
    orderBy: { createdAt: 'DESC' },
  },
);
```
</Tab>
</Tabs>

### String concatenation for SQL

<Warning>
Never interpolate user input directly into SQL strings. Always use parameterized queries.
</Warning>

<Tabs>
<Tab title="Bad pattern">
```typescript
const query = `SELECT * FROM lead WHERE name = '${userInput}'`;
```
</Tab>

<Tab title="Good pattern">
```typescript
queryBuilder.andWhere('lead.name = ?', [userInput]);
```
</Tab>
</Tabs>

### Separate endpoints per view type

Use a single endpoint with an optional `type` parameter instead of creating separate endpoints for each view type. The service fetches all tabs the user has permission to see in parallel using `Promise.all` on initial load, and targets a single tab when the frontend passes a `type` for pagination.

```typescript
// Single endpoint, optional type parameter
@Get()
async getLeads(@Query('type') type?: LeadViewType) {
  return this.service.getLeads(user, { type });
}
```

## Module review checklist

Use these checklists when reviewing or implementing any module.

<AccordionGroup>
<Accordion title="Query optimization">
- All list endpoints use proper pagination with total count
- Access control uses `EXISTS` subqueries (not load-then-filter)
- Related entities loaded with `leftJoinAndSelect` (not lazy)
- No N+1 query patterns
- No memory-based pagination (slice/filter after fetch)
- No N+1 access control checks in loops
- Raw SQL used only when necessary (complex subqueries)
- Indexes exist for common query patterns
- View types use single endpoint with optional type parameter (not separate endpoints)
- No `type` fetches all available tabs; with `type` fetches single tab (load more)
- Permission validation happens before query building
- Response shape determines tab visibility (no `availableTabs`/`availableTeams` metadata)
- Team permission checks use `AccessService.getUserTeamsWithPermissions`
</Accordion>

<Accordion title="Business logic (generic patterns)">
**For any entity with state/stage:**

- State transitions recorded in history table
- Duration in previous state calculated
- Invalid transitions throw appropriate errors
- Changed by user tracked

**For any entity with relationships:**

- Required relationships have `NOT NULL` FK constraints
- Deduplication checked on create (if applicable)
- Cascading creates handled (if applicable)
- Soft delete respects relationship integrity

**For any entity with access control:**

- `EXISTS` subquery for stakeholder/assignment check
- `NOT EXISTS` for unassigned/pool queries
- Team-based access handled
- Creator always has access

**For any entity with aggregates:**

- Sum validations enforced (e.g., percentages do not exceed 100%)
- Locked values protected from changes
- Cascade updates to child aggregates
</Accordion>

<Accordion title="DTO and serialization">
- Uses `class-transformer` with `@Expose` decorators
- Groups defined for LIST, DETAIL, FULL views
- Service returns DTO instances (`plainToInstance`)
- Nested DTOs properly typed
- No sensitive data leakage
</Accordion>

<Accordion title="Error handling">
- Proper HTTP status codes
- Descriptive error messages
- Validation errors from `class-validator`
- Not Found handled gracefully
- Conflict errors for duplicates
</Accordion>

<Accordion title="Code quality">
- Service methods focused (single responsibility)
- Controller is thin (validation + routing only)
- Reusable helper methods for common patterns
- TypeScript strict mode passes
- No unused imports or dead code
</Accordion>
</AccordionGroup>

<AccordionGroup>
<Accordion title="CRM module — Lead">
**Reference:** CRM module specification

- Person-centric: Lead always links to Person
- Deduplication: Check `referenceId + leadSource` on create
- Stakeholder: Auto-create owner on lead creation
- Stage history: Record on every stage change
- Pool: `NOT EXISTS` query for unassigned leads
- Conversion: Creates Contact + Deal, copies stakeholders

- `applyTeamLeadsFilter`: `EXISTS` stakeholder for `team_id`
- `applyTeamPoolLeadsFilter`: `EXISTS` team stakeholder AND `user_id IS NULL`

- Kanban view uses `Promise.all` for parallel stage queries

- Response shape determines tab visibility (no `availableTabs` metadata)
