## Core Principle

The timezone handling system in PropWise CRM follows a clear separation of concerns:

<CardGroup cols={2}>
  <Card title="Timed Events" icon="clock">
    Backend stores UTC timestamps, frontend converts to local timezone for display
  </Card>
  <Card title="All-Day Events" icon="calendar-day">
    Backend sends date labels, frontend expands to local midnight â†’ 23:59
  </Card>
</CardGroup>

| Event type                             | Backend contract                                     | Frontend display                                        |
| -------------------------------------- | ---------------------------------------------------- | ------------------------------------------------------- |
| **Timed event** (`hasDueTime=true`)    | UTC ISO string (`2026-04-22T10:00:00.000Z`)          | Convert to local timezone using `Intl` / `date-fns-tz`  |
| **All-day event** (`hasDueTime=false`) | UTC midnight date-label (`2026-04-22T00:00:00.000Z`) | Extract date portion â†’ expand to local midnight â†’ 23:59 |

<Note>
The key insight: an all-day event is a **calendar date**, not a UTC instant.
`April 22` means April 22 regardless of which timezone the viewer is in.
</Note>

## HTTP Time-Zone Header

### Task List & Kanban Stats

The CRM web app attaches **`Time-Zone`** header on authenticated requests using `Intl.DateTimeFormat().resolvedOptions().timeZone`. Nest applies **`TimezoneMiddleware`** globally with these behaviors:

<Steps>
  <Step title="Validation">
    Header is validated against known IANA zones using `Intl.supportedValuesOf('timeZone')`
  </Step>
  <Step title="Fallback">
    Missing or invalid values fall back to `UTC` (no HTTP 400 error)
  </Step>
  <Step title="Request Context">
    Middleware sets `req.timezone` for use by `@TimezoneParam()` decorator
  </Step>
</Steps>

**Affected Endpoints:** `GET /tasks/list`, `GET /tasks/kanban`

<Tabs>
  <Tab title="Timed Tasks (hasDueTime: true)">
    `TaskService` uses Luxon to compute **[start, next midnight)** of "today" in the viewer's zone as UTC instants and compares `dueDate` (`timestamptz`) to that window.
  </Tab>
  <Tab title="Date-Only Tasks (hasDueTime: false)">
    Buckets compare **`(due_date AT TIME ZONE 'UTC')::date`** (semantic calendar day, aligned with noon-UTC storage) to **`(timezone(<viewer_tz>, now()))::date`** â€” not raw instant bounds.
  </Tab>
</Tabs>

<Warning>
JSON responses are unchanged â€” task `dueDate` fields remain UTC ISO strings; the header affects **grouping and counts**, not field serialization.
</Warning>

### Personal Agenda Timeline

**Endpoint:** `GET /v1/calendar/me/agenda/timeline`

The server interprets **probe anchor**, **first-page visibility**, **local calendar-day span** (`fromDay` / `toDay` in the response), and **`nextCursor`** (`nextDay`) in the viewer's IANA zone using the same middleware fallback as tasks.

<Note>
Multi-day CRM events and externals that **overlap** the viewer's local "today" can appear on the first page even when `start` is before `max(from, now)`.
</Note>

## Timed Events â€” UTC Round-Trip

Timed events follow a straightforward UTC contract with no special handling needed:

```javascript
// Task created
dueDate = "2026-04-22T10:00:00.000Z"   // stored as timestamptz in PG

// Backend response
{
  startDate: "2026-04-22T10:00:00.000Z",
  endDate: "2026-04-22T11:00:00.000Z"
}

// Frontend display
// JavaScript new Date(isoString) automatically converts to local time
// e.g., 13:00 in UTC+3
```

<Tip>
JavaScript `new Date(isoString)` already produces a `Date` in local time. Any date library that accepts ISO 8601 strings will handle the offset automatically.
</Tip>

## All-Day Events â€” Floating Date Model

### CRM Tasks (`source: 'task'`, `hasDueTime: false`)

Task **`dueDate`** for date-only dues is stored as **noon UTC** on the calendar day (`YYYY-MM-DDT12:00:00.000Z`) so the date prefix stays stable.

<CodeGroup>
```typescript Backend (calendar.service.ts â†’ toCalendarTask)
// Extract the YYYY-MM-DD portion â€” no timezone shift
const iso = dueDate.toISOString().substring(0, 10); // e.g. "2026-04-22"
startDate = new Date(`${iso}T00:00:00.000Z`); // UTC midnight â€” a date label
endDate = new Date(`${iso}T23:59:59.999Z`); // UTC end â€” also a date label
```

```typescript Frontend (use-calendar-events.ts â†’ calendarTaskToEvent)
if (!item.hasDueTime && item.dueDate) {
  const [y, m, d] = item.dueDate.substring(0, 10).split('-').map(Number);
  startDate = new Date(y, m - 1, d, 0, 0, 0, 0).toISOString(); // local midnight
  endDate = new Date(y, m - 1, d, 23, 59, 59, 999).toISOString(); // local 23:59
}
```
</CodeGroup>

<Check>
`new Date(y, m-1, d, â€¦)` (multi-argument constructor) **always uses the browser's local timezone**, so the event appears on April 22 no matter what offset the user is in.
</Check>

### External Calendar Events (Google / Outlook)

External providers follow the Google Calendar convention: all-day events arrive as date-only strings with **exclusive end dates**.

```typescript
// Frontend (use-calendar-events.ts â†’ calendarExternalItemToEvent)
if (item.isAllDay && item.startDate && item.endDate) {
  const [sy, sm, sd] = item.startDate.substring(0, 10).split('-').map(Number);
  startDate = new Date(sy, sm - 1, sd, 0, 0, 0, 0).toISOString();

  // Subtract one from end day (exclusive â†’ inclusive)
  const [ey, em, ed] = item.endDate.substring(0, 10).split('-').map(Number);
  endDate = new Date(ey, em - 1, ed - 1, 23, 59, 59, 999).toISOString();
}
```

<Note>
The subtraction `ed - 1` converts the exclusive end to an inclusive one.
</Note>

## Date Constructor Best Practices

<Warning>
**Avoid `new Date("YYYY-MM-DD")`** â€” it's parsed as UTC midnight per ECMAScript spec, causing day shifts in non-UTC timezones.
</Warning>

```typescript
// âťŚ Wrong â€” parsed as UTC midnight, shifts day in non-UTC timezones
new Date('2026-04-22');

// âś… Correct â€” uses local timezone
new Date(2026, 3, 22); // month is 0-indexed
new Date(y, m - 1, d, 0, 0, 0, 0);
```

In a UTC+3 timezone, `new Date("2026-04-22")` becomes `2026-04-21T21:00:00` local time â€” the event shifts to the previous day.

## API Contract

### Query Parameters

| Param            | Type         | Notes                                                           |
| ---------------- | ------------ | --------------------------------------------------------------- |
| `from`           | ISO 8601 UTC | Window start. Frontend sends `subMonths(now, 3).toISOString()`. |
| `to`             | ISO 8601 UTC | Window end. Frontend sends `addMonths(now, 6).toISOString()`.   |
| `userId`         | UUID         | Optional filter.                                                |
| `integrationIds` | string       | Comma-separated CalendarIntegration IDs.                        |

### Agenda Endpoint

**`GET /v1/calendar/agenda`** accepts optional inclusive `from` as UTC ISO: lower bound for merged rows. The web app passes **`startOfDay(selectedDate).toISOString()`** where `selectedDate` uses the same local-date rule.

<Info>
**No `timeZone` parameter.** The backend is timezone-agnostic for all-day events. All date logic lives in the frontend.
</Info>

### Response Fields for `CalendarTaskItemDto`

| Field        | Type         | Notes                                                                                                                              |
| ------------ | ------------ | ---------------------------------------------------------------------------------------------------------------------------------- |
| `startDate`  | ISO 8601 UTC | For `hasDueTime=true`: exact start. For `hasDueTime=false`: floating date label â€” **frontend must override with local expansion**. |
| `endDate`    | ISO 8601 UTC | Same rules as `startDate`.                                                                                                         |
| `dueDate`    | ISO 8601 UTC | Source of truth for the date portion on all-day tasks. Frontend reads `substring(0, 10)` to extract `YYYY-MM-DD`.                  |
| `hasDueTime` | boolean      | `false` â†’ all-day event, frontend applies floating-date expansion.                                                                 |

## Libraries and Tools

<AccordionGroup>
  <Accordion title="Frontend Libraries">
    - **Native `Date` constructor**: All-day local expansion (`new Date(y, m-1, d, â€¦)`)
    - **`date-fns`** (`parseISO`, `isSameDay`, etc.): General formatting and arithmetic
    - **`Intl.DateTimeFormat().resolvedOptions().timeZone`**: Get user's IANA tz for display of timed events
  </Accordion>
  
  <Accordion title="Backend Libraries">
    - **`luxon`**: Used in `viewer-calendar-day.util` for task list/stats day bounds (timed dues only)
    - **PostgreSQL `timezone()` function**: For date-only task buckets with UTC calendar dates
  </Accordion>
</AccordionGroup>

<Note>
`luxon` is **not used** in the calendar (`GET /v1/calendar`) pipeline. Task list/kanban stats use it only to derive **[todayStart, todayEnd)** instants for **timed** tasks.
</Note>

## Common Anti-Patterns

<Warning>
Avoid these common timezone handling mistakes:
</Warning>

| Anti-pattern                                           | Why it breaks                                                    | Fix                                                    |
| ------------------------------------------------------ | ---------------------------------------------------------------- | ------------------------------------------------------ |
| `new Date("2026-04-22")` for all-day display           | UTC parse â†’ wrong day in UTC+/-                                  | Use `new Date(y, m-1, d, â€¦)`                           |
| `date.setHours(0, 0, 0)` on the server                 | Applies server timezone (almost never UTC)                       | Backend emits UTC date-label; frontend expands locally |
| Sending `timeZone` in API query                        | Couples backend to client clock; server has no reliable fallback | All-day logic lives in the frontend only               |
| `toLocaleDateString()` for ISO output                  | Locale-dependent, non-deterministic                              | Use `.toISOString()` after local construction          |
| Using `startDate`/`endDate` directly for all-day tasks | Backend values are placeholders only                             | Extract date from `dueDate` and expand locally         |

<Tip>
Remember: the backend is timezone-agnostic for all-day events. All date expansion logic should live in the frontend to ensure consistent behavior across different user timezones.
</Tip>
