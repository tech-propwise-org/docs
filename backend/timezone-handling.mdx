<Note>
**TL;DR** â€” The API is pure UTC. The frontend converts to local time for display.
All-day / date-only events use the **floating-date model** (same as Google Calendar):
the backend sends a neutral date label; the frontend expands it to local
midnight â†’ 23:59 using `new Date(y, m-1, d, â€¦)`.
</Note>

## Core Principle

The PropWise calendar system handles timezone conversions differently based on event type:

| Event type                             | Backend contract                                     | Frontend display                                        |
| -------------------------------------- | ---------------------------------------------------- | ------------------------------------------------------- |
| **Timed event** (`hasDueTime=true`)    | UTC ISO string (`2026-04-22T10:00:00.000Z`)          | Convert to local timezone using `Intl` / `date-fns-tz`  |
| **All-day event** (`hasDueTime=false`) | UTC midnight date-label (`2026-04-22T00:00:00.000Z`) | Extract date portion â†’ expand to local midnight â†’ 23:59 |

<Info>
The key insight: an all-day event is a **calendar date**, not a UTC instant.
`April 22` means April 22 regardless of which timezone the viewer is in.
Storing `2026-04-22T00:00:00.000Z` and letting the frontend expand it locally is
the same strategy Google Calendar uses (`start.date` / `end.date` fields).
</Info>

## HTTP Time-Zone Header

### Task List & Kanban Stats

The CRM web app attaches **`Time-Zone`** on authenticated requests (`Intl.DateTimeFormat().resolvedOptions().timeZone`). Nest applies **`TimezoneMiddleware`** globally: the header is validated against known IANA zones (`Intl.supportedValuesOf('timeZone')`).

<Warning>
Missing or invalid values fall back to `UTC` (no HTTP 400 error).
</Warning>

The middleware sets `req.timezone` for use by `@TimezoneParam()` decorator.

**Endpoints:** `GET /tasks/list`, `GET /tasks/kanban`

#### Behavior

For **`hasDueTime: true`**, `TaskService` uses Luxon to compute **[start, next midnight)** of "today" in the viewer's zone as UTC instants and compares `dueDate` (`timestamptz`) to that window. 

For **`hasDueTime: false`** (date-only dues), buckets compare **`(due_date AT TIME ZONE 'UTC')::date`** (semantic calendar day, aligned with noon-UTC storage) to **`(timezone(<viewer_tz>, now()))::date`** â€” not raw instant bounds.

<Note>
JSON responses are unchanged â€” task `dueDate` fields remain UTC ISO strings; the header affects **grouping and counts**, not field serialization.
</Note>

#### List Sort

**List sort (`sort=dueDate` / default):** Within each overdue / today / upcoming / completed bucket, rows are ordered by:

1. **Calendar due day** first (same rules as buckets)
2. **`due_date` instant** for timed tasks
3. **Date-only** rows (`hasDueTime: false`) **after** all timed tasks on that calendar day
4. `title`

<Warning>
Requires **`Time-Zone`** header so timed dues sort by the viewer's local calendar date.
</Warning>

### Personal Agenda Timeline

**Endpoint:** `GET /v1/calendar/me/agenda/timeline`

The server interprets **probe anchor**, **first-page visibility**, **local calendar-day span** (`fromDay` / `toDay` in the response), and **`nextCursor`** (`nextDay`) in the viewer's IANA zone.

- Tasks use **due** for ordering (`CalendarItemDto.startDate` mirrors due)
- CRM events and externals use **start**
- Multi-day CRM events and externals that **overlap** the viewer's local "today" can appear on the first page even when `start` is before `max(from, now)`

## Timed Events â€” UTC Round-Trip

```
Task created: dueDate = "2026-04-22T10:00:00.000Z"   (stored as timestamptz in PG)
Backend:      startDate = dueDate,  endDate = startDate + 1h
API response: { startDate: "2026-04-22T10:00:00.000Z",
                endDate:   "2026-04-22T11:00:00.000Z" }
Frontend:     calendar grid converts to local time (e.g. 13:00 in UTC+3)
```

<Tip>
**No special handling needed.** JavaScript `new Date(isoString)` already produces a
`Date` in local time. Any date library that accepts ISO 8601 strings will handle the
offset automatically.
</Tip>

## All-Day Events â€” Floating Date Model

### CRM Tasks

Task **`dueDate`** for date-only dues is stored as **noon UTC** on the calendar day (`YYYY-MM-DDT12:00:00.000Z`) so the date prefix stays stable.

<Note>
UI must treat the due **day** as the **`YYYY-MM-DD` prefix**, not the local calendar day of the instant alone.
</Note>

#### Backend Implementation

```typescript
// Extract the YYYY-MM-DD portion â€” no timezone shift
const iso = dueDate.toISOString().substring(0, 10); // e.g. "2026-04-22"
startDate = new Date(`${iso}T00:00:00.000Z`); // UTC midnight â€” a date label
endDate = new Date(`${iso}T23:59:59.999Z`); // UTC end â€” also a date label
```

The backend also returns `dueDate` as a raw ISO string and `hasDueTime: false`.

#### Frontend Implementation

```typescript
if (!item.hasDueTime && item.dueDate) {
  const [y, m, d] = item.dueDate.substring(0, 10).split('-').map(Number);
  startDate = new Date(y, m - 1, d, 0, 0, 0, 0).toISOString(); // local midnight
  endDate = new Date(y, m - 1, d, 23, 59, 59, 999).toISOString(); // local 23:59
}
```

<Check>
`new Date(y, m-1, d, â€¦)` (multi-argument constructor) **always uses the browser's
local timezone**, so the event appears on April 22 no matter what offset the user is in.
</Check>

### External Calendar Events

External providers already follow the Google Calendar convention: all-day events
arrive as date-only strings (`"2026-04-22"`, not ISO timestamps) and use an
**exclusive end date** (April 22 all-day â†’ `endDate = "2026-04-23"`).

```typescript
if (item.isAllDay && item.startDate && item.endDate) {
  const [sy, sm, sd] = item.startDate.substring(0, 10).split('-').map(Number);
  startDate = new Date(sy, sm - 1, sd, 0, 0, 0, 0).toISOString();

  // Subtract one from end day (exclusive â†’ inclusive)
  const [ey, em, ed] = item.endDate.substring(0, 10).split('-').map(Number);
  endDate = new Date(ey, em - 1, ed - 1, 23, 59, 59, 999).toISOString();
}
```

The subtraction `ed - 1` converts the exclusive end to an inclusive one.

## Date Constructor Best Practices

<Warning>
`new Date("YYYY-MM-DD")` is parsed as **UTC midnight** per the ECMAScript spec.
In a UTC+3 timezone this becomes `2026-04-21T21:00:00` local time â€” the event
shifts to the previous day.
</Warning>

Always use the multi-argument constructor:

<CodeGroup>
```typescript Wrong
// âťŚ Wrong â€” parsed as UTC midnight, shifts day in non-UTC timezones
new Date('2026-04-22');
```

```typescript Correct
// âś… Correct â€” uses local timezone
new Date(2026, 3, 22); // month is 0-indexed
new Date(y, m - 1, d, 0, 0, 0, 0);
```
</CodeGroup>

## API Contract

### Query Parameters

<AccordionGroup>
<Accordion title="GET /v1/calendar Parameters">
| Param            | Type         | Notes                                                           |
| ---------------- | ------------ | --------------------------------------------------------------- |
| `from`           | ISO 8601 UTC | Window start. Frontend sends `subMonths(now, 3).toISOString()`. |
| `to`             | ISO 8601 UTC | Window end. Frontend sends `addMonths(now, 6).toISOString()`.   |
| `userId`         | UUID         | Optional filter.                                                |
| `integrationIds` | string       | Comma-separated CalendarIntegration IDs.                        |
</Accordion>

<Accordion title="GET /v1/calendar/agenda Parameters">
**`GET /v1/calendar/agenda`** â€” optional inclusive `from` as UTC ISO: lower bound for which merged rows are returned. The web app passes **`startOfDay(selectedDate).toISOString()`** where `selectedDate` is the viewer's calendar day from `?date=yyyy-MM-dd` (parsed with `new Date(y, m-1, d)`, same local-date rule).

The response is **`{ items }`** only (same shapes as `GET /v1/calendar`); there is **no cursor pagination** â€” the backend scans forward up to an internal cap and returns every item in range.
</Accordion>
</AccordionGroup>

<Info>
**No `timeZone` parameter.** The backend is timezone-agnostic for all-day events.
All date logic lives in the frontend.
</Info>

### Response Fields

#### CalendarTaskItemDto

| Field        | Type         | Notes                                                                                                                              |
| ------------ | ------------ | ---------------------------------------------------------------------------------------------------------------------------------- |
| `startDate`  | ISO 8601 UTC | For `hasDueTime=true`: exact start. For `hasDueTime=false`: floating date label â€” **frontend must override with local expansion**. |
| `endDate`    | ISO 8601 UTC | Same rules as `startDate`.                                                                                                         |
| `dueDate`    | ISO 8601 UTC | Source of truth for the date portion on all-day tasks. Frontend reads `substring(0, 10)` to extract `YYYY-MM-DD`.                  |
| `hasDueTime` | boolean      | `false` â†’ all-day event, frontend applies floating-date expansion.                                                                 |

</Warning>

## Anti-Patterns

<CardGroup cols={1}>
<Card title="Common Mistakes to Avoid">
| Anti-pattern                                           | Why it breaks                                                    | Fix                                                    |
| ------------------------------------------------------ | ---------------------------------------------------------------- | ------------------------------------------------------ |
| `new Date("2026-04-22")` for all-day display           | UTC parse â†’ wrong day in UTC+/-                                  | Use `new Date(y, m-1, d, â€¦)`                           |
| `date.setHours(0, 0, 0)` on the server                 | Applies server timezone (almost never UTC)                       | Backend emits UTC date-label; frontend expands locally |
| Sending `timeZone` in API query                        | Couples backend to client clock; server has no reliable fallback | All-day logic lives in the frontend only               |
| `toLocaleDateString()` for ISO output                  | Locale-dependent, non-deterministic                              | Use `.toISOString()` after local construction          |
| Using `startDate`/`endDate` directly for all-day tasks | Backend values are placeholders                                   | Extract date from `dueDate` field instead              |
</Card>
</CardGroup>
