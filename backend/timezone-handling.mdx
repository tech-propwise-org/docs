<Note>
**TL;DR** тАФ The API is pure UTC. The frontend converts to local time for display.
All-day / date-only events use the **floating-date model** (same as Google Calendar):
the backend sends a neutral date label; the frontend expands it to local
midnight тЖТ 23:59 using `new Date(y, m-1, d, тАж)`.
</Note>

## Core Principle

The system handles two fundamentally different event types:

| Event type                             | Backend contract                                     | Frontend display                                        |
| -------------------------------------- | ---------------------------------------------------- | ------------------------------------------------------- |
| **Timed event** (`hasDueTime=true`)    | UTC ISO string (`2026-04-22T10:00:00.000Z`)          | Convert to local timezone using `Intl` / `date-fns-tz`  |
| **All-day event** (`hasDueTime=false`) | UTC midnight date-label (`2026-04-22T00:00:00.000Z`) | Extract date portion тЖТ expand to local midnight тЖТ 23:59 |

<Info>
The key insight: an all-day event is a **calendar date**, not a UTC instant.
`April 22` means April 22 regardless of which timezone the viewer is in.
Storing `2026-04-22T00:00:00.000Z` and letting the frontend expand it locally is
the same strategy Google Calendar uses (`start.date` / `end.date` fields).
</Info>

## HTTP `Time-Zone` Header

### Task List & Kanban Stats

The CRM web app attaches **`Time-Zone`** on authenticated requests (`Intl.DateTimeFormat().resolvedOptions().timeZone`). Nest applies **`TimezoneMiddleware`** globally: the header is validated against known IANA zones (`Intl.supportedValuesOf('timeZone')`).

<Warning>
**Missing or invalid values fall back to `UTC`** (no HTTP 400 error). The middleware sets `req.timezone` for use by `@TimezoneParam()` decorator.
</Warning>

**Endpoints:** `GET /tasks/list`, `GET /tasks/kanban`

#### Behavior

<Steps>
<Step title="Timed tasks (hasDueTime: true)">
`TaskService` uses Luxon to compute **[start, next midnight)** of "today" in the viewer's zone as UTC instants and compares `dueDate` (`timestamptz`) to that window.
</Step>

<Step title="Date-only tasks (hasDueTime: false)">
Buckets compare **`(due_date AT TIME ZONE 'UTC')::date`** (semantic calendar day, aligned with noon-UTC storage) to **`(timezone(<viewer_tz>, now()))::date`** тАФ not raw instant bounds.

This ensures a task shown as "30 Apr" cannot land in "today" when the viewer's local day is already "1 May".
</Step>

<Step title="Stats computation">
List **stats** (`overdue`, `dueToday`) use the same rules for consistent counting.
</Step>
</Steps>

<Check>
**JSON responses are unchanged** тАФ task `dueDate` fields remain UTC ISO strings; the header affects **grouping and counts**, not field serialization.
</Check>

#### List Sort (`sort=dueDate` / default)

Within each overdue / today / upcoming / completed bucket, rows are ordered by:

1. **Calendar due day** first (same rules as buckets: local calendar date from `due_date` when `hasDueTime`, else `(due_date AT TIME ZONE 'UTC')::date`)
2. **`due_date` instant** for timed tasks
3. **Date-only** rows (`hasDueTime: false`) **after** all timed tasks on that calendar day
4. `title`

<Note>
Requires **`Time-Zone`** header so timed dues sort by the viewer's local calendar date.
</Note>

### Personal Agenda Timeline

**Endpoint:** `GET /v1/calendar/me/agenda/timeline`

The server interprets **probe anchor**, **first-page visibility**, **local calendar-day span** (`fromDay` / `toDay` in the response), and **`nextCursor`** (`nextDay`) in the viewer's IANA zone (same middleware fallback as tasks).

#### Ordering Rules

- **Tasks** use **due** for ordering (`CalendarItemDto.startDate` mirrors due)
- **CRM events and externals** use **start**

<Info>
Multi-day CRM events and externals that **overlap** the viewer's local "today" can appear on the first page even when `start` is before `max(from, now)`.
</Info>

Timeline rows include optional **`agendaDayKeys`** clipped to the response window. The frontend agenda list groups by those keys and clamps to `window.toDay` instead of re-grouping from browser-parsed `startDate` / `endDate`.

<Tip>
See `Docs/CALENDAR_SYSTEM_DEEP_DIVE.md` тАФ **Personal agenda timeline** for detailed implementation.
</Tip>

## Timed Events тАФ UTC Round-Trip

Timed events follow a straightforward UTC round-trip pattern:

```typescript
// Task created: dueDate = "2026-04-22T10:00:00.000Z" (stored as timestamptz in PG)
// Backend: startDate = dueDate, endDate = startDate + 1h

// API response:
{
  startDate: "2026-04-22T10:00:00.000Z",
  endDate: "2026-04-22T11:00:00.000Z"
}

// Frontend: calendar grid converts to local time (e.g. 13:00 in UTC+3)
```

<Check>
**No special handling needed.** JavaScript `new Date(isoString)` already produces a `Date` in local time. Any date library that accepts ISO 8601 strings will handle the offset automatically.
</Check>

## All-Day Events тАФ Floating Date Model

### CRM Tasks (`source: 'task'`, `hasDueTime: false`)

Task **`dueDate`** for date-only dues is stored as **noon UTC** on the calendar day (`YYYY-MM-DDT12:00:00.000Z`) so the date prefix stays stable.

<Warning>
UI must treat the due **day** as the **`YYYY-MM-DD` prefix** (see task detail panel / `parseTaskDueAsLocalDate`), not the local calendar day of the instant alone.
</Warning>

#### Backend Implementation

**`calendar.service.ts тЖТ toCalendarTask`**

```typescript
// Extract the YYYY-MM-DD portion тАФ no timezone shift
const iso = dueDate.toISOString().substring(0, 10); // e.g. "2026-04-22"
startDate = new Date(`${iso}T00:00:00.000Z`); // UTC midnight тАФ a date label
endDate = new Date(`${iso}T23:59:59.999Z`); // UTC end тАФ also a date label
```

The backend also returns `dueDate` as a raw ISO string and `hasDueTime: false`.

#### Frontend Implementation

**`use-calendar-events.ts тЖТ calendarTaskToEvent`**

```typescript
if (!item.hasDueTime && item.dueDate) {
  const [y, m, d] = item.dueDate.substring(0, 10).split('-').map(Number);
  startDate = new Date(y, m - 1, d, 0, 0, 0, 0).toISOString(); // local midnight
  endDate = new Date(y, m - 1, d, 23, 59, 59, 999).toISOString(); // local 23:59
}
```

<Info>
`new Date(y, m-1, d, тАж)` (multi-argument constructor) **always uses the browser's local timezone**, so the event appears on April 22 no matter what offset the user is in.
</Info>

`isAllDay: !item.hasDueTime` is also set on the returned `ICalendarEvent` so the calendar UI can render the event in the all-day row instead of the time grid.

### External Calendar Events (Google / Outlook)

External providers already follow the Google Calendar convention: all-day events arrive as date-only strings (`"2026-04-22"`, not ISO timestamps) and use an **exclusive end date** (April 22 all-day тЖТ `endDate = "2026-04-23"`).

**`use-calendar-events.ts тЖТ calendarExternalItemToEvent`**

```typescript
if (item.isAllDay && item.startDate && item.endDate) {
  const [sy, sm, sd] = item.startDate.substring(0, 10).split('-').map(Number);
  startDate = new Date(sy, sm - 1, sd, 0, 0, 0, 0).toISOString();

  // Subtract one from end day (exclusive тЖТ inclusive)
  const [ey, em, ed] = item.endDate.substring(0, 10).split('-').map(Number);
  endDate = new Date(ey, em - 1, ed - 1, 23, 59, 59, 999).toISOString();
}
```

<Note>
The subtraction `ed - 1` converts the exclusive end to an inclusive one.
</Note>

## Date Constructor Pitfall

<Warning>
`new Date("YYYY-MM-DD")` is parsed as **UTC midnight** per the ECMAScript spec. In a UTC+3 timezone this becomes `2026-04-21T21:00:00` local time тАФ the event shifts to the previous day.
</Warning>

Always use the multi-argument constructor:

<CodeGroup>

```typescript тЭМ Wrong
// Parsed as UTC midnight, shifts day in non-UTC timezones
new Date('2026-04-22');
```

```typescript тЬЕ Correct
// Uses local timezone
new Date(2026, 3, 22); // month is 0-indexed
new Date(y, m - 1, d, 0, 0, 0, 0);
```

</CodeGroup>

## API Contract

### Query Parameters

<AccordionGroup>
<Accordion title="from (ISO 8601 UTC)">
Window start. Frontend sends `subMonths(now, 3).toISOString()`.
</Accordion>

<Accordion title="to (ISO 8601 UTC)">
Window end. Frontend sends `addMonths(now, 6).toISOString()`.
</Accordion>

<Accordion title="userId (UUID)">
Optional filter for specific user.
</Accordion>

<Accordion title="integrationIds (string)">
Comma-separated CalendarIntegration IDs.
</Accordion>
</AccordionGroup>

### Agenda Endpoint

**`GET /v1/calendar/agenda`** тАФ optional inclusive `from` as UTC ISO: lower bound for which merged rows are returned.

The web app passes **`startOfDay(selectedDate).toISOString()`** where `selectedDate` is the viewer's calendar day from `?date=yyyy-MM-dd` (parsed with `new Date(y, m-1, d)`, same local-date rule as above).

<Info>
The response is **`{ items }`** only (same shapes as `GET /v1/calendar`); there is **no cursor pagination** тАФ the backend scans forward up to an internal cap and returns every item in range.
</Info>

<Note>
**No `timeZone` parameter.** The backend is timezone-agnostic for all-day events. All date logic lives in the frontend.
</Note>

### Response Fields for `CalendarTaskItemDto`

| Field        | Type         | Description                                                                                                                        |
| ------------ | ------------ | ---------------------------------------------------------------------------------------------------------------------------------- |
| `startDate`  | ISO 8601 UTC | For `hasDueTime=true`: exact start. For `hasDueTime=false`: floating date label тАФ **frontend must override with local expansion**. |
| `endDate`    | ISO 8601 UTC | Same rules as `startDate`.                                                                                                         |
| `dueDate`    | ISO 8601 UTC | Source of truth for the date portion on all-day tasks. Frontend reads `substring(0, 10)` to extract `YYYY-MM-DD`.                  |
| `hasDueTime` | boolean      | `false` тЖТ all-day event, frontend applies floating-date expansion.                                                                 |

## Libraries

<Tabs>
  <Tab title="Frontend">

    | -------------------------------------------------- | ----------------------------------------------- |

    | `date-fns` (`parseISO`, `isSameDay`, etc.)         | General formatting and arithmetic               |
    | `Intl.DateTimeFormat().resolvedOptions().timeZone` | Get user's IANA tz for display of timed events  |
  </Tab>
  <Tab title="Backend">
    | Library                                                | Usage                                                                                                      |
    | ------------------------------------------------------ | ---------------------------------------------------------------------------------------------------------- |
    | `luxon` (task list / stats day bounds for timed dues) | Used in `viewer-calendar-day.util` тАФ **not** in the calendar merge pipeline |
  </Tab>
</Tabs>

<Warning>
`luxon` is **not used** in the calendar (`GET /v1/calendar`) pipeline. Task list/kanban stats use it only to derive **[todayStart, todayEnd)** instants for **timed** tasks; date-only task buckets use PostgreSQL `timezone()` + UTC calendar dates instead.
</Warning>

## Anti-Patterns

<CardGroup cols={2}>
<Card title="Wrong Date Constructor" icon="xmark">
`new Date("2026-04-22")` for all-day display

**Why it breaks:** UTC parse тЖТ wrong day in UTC+/-

**Fix:** Use `new Date(y, m-1, d, тАж)`
</Card>

<Card title="Server-Side setHours" icon="xmark">
`date.setHours(0, 0, 0)` on the server

**Why it breaks:** Applies server timezone (almost never UTC)

**Fix:** Backend emits UTC date-label; frontend expands locally
</Card>

<Card title="TimeZone Query Param" icon="xmark">
Sending `timeZone` in API query

**Why it breaks:** Couples backend to client clock; server has no reliable fallback

**Fix:** All-day logic lives in the frontend only
</Card>

<Card title="Locale Date Strings" icon="xmark">
`toLocaleDateString()` for ISO output

**Why it breaks:** Produces locale-specific formats, not ISO 8601

**Fix:** Use `toISOString()` or date library formatting
</Card>
</CardGroup>

<Tip>
When in doubt, remember: **UTC for timed events, local date expansion for all-day events**. The backend is timezone-agnostic; the frontend handles all display logic.
</Tip>
