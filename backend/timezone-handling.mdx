<Note>
**TL;DR** — The API is pure UTC. The frontend converts to local time for display.
All-day / date-only events use the **floating-date model** (same as Google Calendar):
the backend sends a neutral date label; the frontend expands it to local
midnight → 23:59 using `new Date(y, m-1, d, …)`.
</Note>

## Core Principle

PropWise CRM follows a clean separation between backend UTC storage and frontend timezone conversion:

| Event type                             | Backend contract                                     | Frontend display                                        |
| -------------------------------------- | ---------------------------------------------------- | ------------------------------------------------------- |
| **Timed event** (`hasDueTime=true`)    | UTC ISO string (`2026-04-22T10:00:00.000Z`)          | Convert to local timezone using `Intl` / `date-fns-tz`  |
| **All-day event** (`hasDueTime=false`) | UTC midnight date-label (`2026-04-22T00:00:00.000Z`) | Extract date portion → expand to local midnight → 23:59 |

<Tip>
The key insight: an all-day event is a **calendar date**, not a UTC instant.
`April 22` means April 22 regardless of which timezone the viewer is in.
</Tip>

## HTTP Time-Zone Header

### Task List & Kanban Stats

The CRM web app attaches **`Time-Zone`** header on authenticated requests using `Intl.DateTimeFormat().resolvedOptions().timeZone`. Nest applies **`TimezoneMiddleware`** globally with the following behavior:

- Header is validated against known IANA zones (`Intl.supportedValuesOf('timeZone')`)
- **Missing or invalid values fall back to `UTC`** (no HTTP 400 error)
- Middleware sets `req.timezone` for use by `@TimezoneParam()` decorator

**Affected endpoints:** 
- `GET /tasks/list`
- `GET /tasks/kanban`

<Tabs>
<Tab title="Timed Tasks (hasDueTime: true)">
`TaskService` uses Luxon to compute **[start, next midnight)** of "today" in the viewer's zone as UTC instants and compares `dueDate` (`timestamptz`) to that window.
</Tab>
<Tab title="Date-only Tasks (hasDueTime: false)">
Buckets compare **`(due_date AT TIME ZONE 'UTC')::date`** (semantic calendar day, aligned with noon-UTC storage) to **`(timezone(<viewer_tz>, now()))::date`** — not raw instant bounds.
</Tab>
</Tabs>

<Warning>
JSON responses are unchanged — task `dueDate` fields remain UTC ISO strings; the header affects **grouping and counts**, not field serialization.
</Warning>

### List Sort Behavior

Within each bucket (overdue / today / upcoming / completed), rows are ordered by:

<Steps>
<Step title="Calendar due day">
Local calendar date from `due_date` when `hasDueTime`, else `(due_date AT TIME ZONE 'UTC')::date`
</Step>
<Step title="Due date instant">
For timed tasks only
</Step>
<Step title="Date-only tasks">
`hasDueTime: false` rows **after** all timed tasks on that calendar day
</Step>
<Step title="Task title">
Final sort criteria
</Step>
</Steps>

### Personal Agenda Timeline

**Endpoint:** `GET /v1/calendar/me/agenda/timeline`

The server interprets **probe anchor**, **first-page visibility**, **local calendar-day span** (`fromDay` / `toDay` in response), and **`nextCursor`** (`nextDay`) in the viewer's IANA zone.

- Tasks use **due** for ordering (`CalendarItemDto.startDate` mirrors due)
- CRM events and externals use **start**
- Multi-day events that **overlap** the viewer's local "today" can appear on the first page even when `start` is before `max(from, now)`

## Timed Events — UTC Round-Trip

For events with specific times, the flow is straightforward:

```typescript
// Task created
dueDate = "2026-04-22T10:00:00.000Z"   // stored as timestamptz in PostgreSQL

// Backend response
{
  startDate: "2026-04-22T10:00:00.000Z",
  endDate: "2026-04-22T11:00:00.000Z"
}

// Frontend automatically converts to local time (e.g. 13:00 in UTC+3)
```

<Info>
No special handling needed. JavaScript `new Date(isoString)` already produces a `Date` in local time. Any date library that accepts ISO 8601 strings will handle the offset automatically.
</Info>

## All-Day Events — Floating Date Model

### CRM Tasks (source: 'task', hasDueTime: false)

Task **`dueDate`** for date-only dues is stored as **noon UTC** on the calendar day (`YYYY-MM-DDT12:00:00.000Z`) so the date prefix stays stable.

<CodeGroup>
```typescript Backend (calendar.service.ts → toCalendarTask)
// Extract the YYYY-MM-DD portion — no timezone shift
const iso = dueDate.toISOString().substring(0, 10); // e.g. "2026-04-22"
startDate = new Date(`${iso}T00:00:00.000Z`); // UTC midnight — a date label
endDate = new Date(`${iso}T23:59:59.999Z`); // UTC end — also a date label
```

```typescript Frontend (use-calendar-events.ts → calendarTaskToEvent)
if (!item.hasDueTime && item.dueDate) {
  const [y, m, d] = item.dueDate.substring(0, 10).split('-').map(Number);
  startDate = new Date(y, m - 1, d, 0, 0, 0, 0).toISOString(); // local midnight
  endDate = new Date(y, m - 1, d, 23, 59, 59, 999).toISOString(); // local 23:59
}
```
</CodeGroup>

<Check>
`new Date(y, m-1, d, …)` (multi-argument constructor) **always uses the browser's local timezone**, so the event appears on April 22 no matter what offset the user is in.
</Check>

### External Calendar Events (Google / Outlook)

External providers follow the Google Calendar convention with date-only strings and exclusive end dates:

```typescript
// Frontend (use-calendar-events.ts → calendarExternalItemToEvent)
if (item.isAllDay && item.startDate && item.endDate) {
  const [sy, sm, sd] = item.startDate.substring(0, 10).split('-').map(Number);
  startDate = new Date(sy, sm - 1, sd, 0, 0, 0, 0).toISOString();

  // Subtract one from end day (exclusive → inclusive)
  const [ey, em, ed] = item.endDate.substring(0, 10).split('-').map(Number);
  endDate = new Date(ey, em - 1, ed - 1, 23, 59, 59, 999).toISOString();
}
```

## Date Constructor Best Practices

<Warning>
**Avoid `new Date("YYYY-MM-DD")`**

`new Date("2026-04-22")` is parsed as **UTC midnight** per the ECMAScript spec. In a UTC+3 timezone this becomes `2026-04-21T21:00:00` local time — the event shifts to the previous day.
</Warning>

<CodeGroup>
```typescript ❌ Wrong
// Parsed as UTC midnight, shifts day in non-UTC timezones
new Date('2026-04-22');
```

```typescript ✅ Correct
// Uses local timezone
new Date(2026, 3, 22); // month is 0-indexed
new Date(y, m - 1, d, 0, 0, 0, 0);
```
</CodeGroup>

## API Contract

### Calendar Endpoints

<AccordionGroup>
<Accordion title="GET /v1/calendar - Query Parameters">
| Param            | Type         | Notes                                                           |
| ---------------- | ------------ | --------------------------------------------------------------- |
| `from`           | ISO 8601 UTC | Window start. Frontend sends `subMonths(now, 3).toISOString()`. |
| `to`             | ISO 8601 UTC | Window end. Frontend sends `addMonths(now, 6).toISOString()`.   |
| `userId`         | UUID         | Optional filter.                                                |
| `integrationIds` | string       | Comma-separated CalendarIntegration IDs.                        |
</Accordion>

<Accordion title="GET /v1/calendar/agenda - Specific Behavior">
Optional inclusive `from` as UTC ISO: lower bound for merged rows. The web app passes **`startOfDay(selectedDate).toISOString()`** where `selectedDate` is parsed with `new Date(y, m-1, d)`.

Response format: **`{ items }`** only (same shapes as `GET /v1/calendar`). No cursor pagination — backend scans forward up to internal cap.
</Accordion>
</AccordionGroup>

<Info>
**No `timeZone` parameter.** The backend is timezone-agnostic for all-day events. All date logic lives in the frontend.
</Info>

### Response Fields for CalendarTaskItemDto

| Field        | Type         | Notes                                                                                                                              |
| ------------ | ------------ | ---------------------------------------------------------------------------------------------------------------------------------- |
| `startDate`  | ISO 8601 UTC | For `hasDueTime=true`: exact start. For `hasDueTime=false`: floating date label — **frontend must override with local expansion**. |
| `endDate`    | ISO 8601 UTC | Same rules as `startDate`.                                                                                                         |
| `dueDate`    | ISO 8601 UTC | Source of truth for the date portion on all-day tasks. Frontend reads `substring(0, 10)` to extract `YYYY-MM-DD`.                  |
| `hasDueTime` | boolean      | `false` → all-day event, frontend applies floating-date expansion.                                                                 |

## Libraries & Tools

<CardGroup cols={2}>
<Card title="Frontend Libraries" icon="browser">
- Native `Date` constructor for all-day local expansion
- `date-fns` for formatting and arithmetic
- `Intl.DateTimeFormat().resolvedOptions().timeZone` for user's IANA tz
</Card>

<Card title="Backend Libraries" icon="server">
- `luxon` for task list/stats day bounds (timed dues only)
- PostgreSQL `timezone()` + UTC calendar dates for date-only tasks
</Card>
</CardGroup>

<Note>
`luxon` is **not used** in the calendar (`GET /v1/calendar`) pipeline. Task list/kanban stats use it only to derive **[todayStart, todayEnd)** instants for **timed** tasks.
</Note>

## Common Anti-Patterns

<AccordionGroup>
<Accordion title="Date Constructor Issues">
| Anti-pattern | Why it breaks | Fix |
| --- | --- | --- |
| `new Date("2026-04-22")` for all-day display | UTC parse → wrong day in UTC+/- | Use `new Date(y, m-1, d, …)` |
| `toLocaleDateString()` for ISO output | Locale-dependent, non-deterministic | Use `.toISOString()` after local construction |
</Accordion>

<Accordion title="Backend Timezone Coupling">
| Anti-pattern | Why it breaks | Fix |
| --- | --- | --- |
| `date.setHours(0, 0, 0)` on the server | Applies server timezone (almost never UTC) | Backend emits UTC date-label; frontend expands locally |
| Sending `timeZone` in API query | Couples backend to client clock; no reliable fallback | All-day logic lives in the frontend only |
</Accordion>

<Accordion title="Data Handling">
| Anti-pattern | Why it breaks | Fix |
| --- | --- | --- |
| Using `startDate`/`endDate` directly for all-day tasks | Backend values are placeholders for floating dates | Extract date from `dueDate` and expand locally |
</Accordion>
</AccordionGroup>
