<Note>
**TL;DR** — The API is pure UTC. The frontend converts to local time for display.
All-day / date-only events use the **floating-date model** (same as Google Calendar):
the backend sends a neutral date label; the frontend expands it to local
midnight → 23:59 using `new Date(y, m-1, d, …)`.
</Note>

## Core Principle

The PropWise CRM handles timezone conversion using a clear separation of concerns between backend and frontend:

| Event type                             | Backend contract                                     | Frontend display                                        |
| -------------------------------------- | ---------------------------------------------------- | ------------------------------------------------------- |
| **Timed event** (`hasDueTime=true`)    | UTC ISO string (`2026-04-22T10:00:00.000Z`)          | Convert to local timezone using `Intl` / `date-fns-tz`  |
| **All-day event** (`hasDueTime=false`) | UTC midnight date-label (`2026-04-22T00:00:00.000Z`) | Extract date portion → expand to local midnight → 23:59 |

<Info>
The key insight: an all-day event is a **calendar date**, not a UTC instant.
`April 22` means April 22 regardless of which timezone the viewer is in.
Storing `2026-04-22T00:00:00.000Z` and letting the frontend expand it locally is
the same strategy Google Calendar uses (`start.date` / `end.date` fields).
</Info>

## HTTP Time-Zone Header

### Task List & Kanban Stats

The CRM web app attaches **`Time-Zone`** on authenticated requests using `Intl.DateTimeFormat().resolvedOptions().timeZone`. 

<Steps>
<Step title="Middleware validation">
Nest applies **`TimezoneMiddleware`** globally to validate the header against known IANA zones using `Intl.supportedValuesOf('timeZone')`
</Step>
<Step title="Fallback handling">
**Missing or invalid values fall back to `UTC`** (no HTTP 400 error)
</Step>
<Step title="Request decoration">
The middleware sets `req.timezone` for use by `@TimezoneParam()` decorator
</Step>
</Steps>

**Affected endpoints:**
- `GET /tasks/list`
- `GET /tasks/kanban`

#### Behavior for Different Task Types

<Tabs>
<Tab title="Timed Tasks (hasDueTime: true)">
`TaskService` uses Luxon to compute **[start, next midnight)** of "today" in the viewer's zone as UTC instants and compares `dueDate` (`timestamptz`) to that window.
</Tab>
<Tab title="Date-only Tasks (hasDueTime: false)">
Buckets compare **`(due_date AT TIME ZONE 'UTC')::date`** (semantic calendar day, aligned with noon-UTC storage) to **`(timezone(<viewer_tz>, now()))::date`** — not raw instant bounds.

This ensures a task shown as "30 Apr" cannot land in "today" when the viewer's local day is already "1 May".
</Tab>
</Tabs>

<Warning>
**JSON responses are unchanged** — task `dueDate` fields remain UTC ISO strings; the header affects **grouping and counts**, not field serialization.
</Warning>

#### List Sorting Rules

For `sort=dueDate` (default), within each bucket (overdue / today / upcoming / completed):

1. **Calendar due day** first (local calendar date from `due_date` when `hasDueTime`, else `(due_date AT TIME ZONE 'UTC')::date`)
2. **`due_date` instant** for timed tasks
3. **Date-only** rows (`hasDueTime: false`) **after** all timed tasks on that calendar day
4. `title`

<Note>
Requires **`Time-Zone`** header so timed dues sort by the viewer's local calendar date.
</Note>

### Personal Agenda Timeline

**Endpoint:** `GET /v1/calendar/me/agenda/timeline`

The server interprets **probe anchor**, **first-page visibility**, **local calendar-day span** (`fromDay` / `toDay` in the response), and **`nextCursor`** (`nextDay`) in the viewer's IANA zone.

<Info>
Tasks use **due** for ordering (`CalendarItemDto.startDate` mirrors due); CRM events and externals use **start**. Multi-day CRM events and externals that **overlap** the viewer's local "today" can appear on the first page even when `start` is before `max(from, now)`.
</Info>

## Timed Events — UTC Round-Trip

For events with specific times, the system maintains a clean UTC round-trip:

```typescript
// Task created
dueDate = "2026-04-22T10:00:00.000Z"   // stored as timestamptz in PG

// Backend processing
startDate = dueDate
endDate = startDate + 1h

// API response
{
  startDate: "2026-04-22T10:00:00.000Z",
  endDate: "2026-04-22T11:00:00.000Z"
}

// Frontend display
// Calendar grid converts to local time (e.g. 13:00 in UTC+3)
```

<Tip>
**No special handling needed.** JavaScript `new Date(isoString)` already produces a `Date` in local time. Any date library that accepts ISO 8601 strings will handle the offset automatically.
</Tip>

## All-Day Events — Floating Date Model

### CRM Tasks

For tasks with `source: 'task'` and `hasDueTime: false`:

<CodeGroup>
```typescript Backend (calendar.service.ts → toCalendarTask)
// Extract the YYYY-MM-DD portion — no timezone shift
const iso = dueDate.toISOString().substring(0, 10); // e.g. "2026-04-22"
startDate = new Date(`${iso}T00:00:00.000Z`); // UTC midnight — a date label
endDate = new Date(`${iso}T23:59:59.999Z`); // UTC end — also a date label
```

```typescript Frontend (use-calendar-events.ts → calendarTaskToEvent)
if (!item.hasDueTime && item.dueDate) {
  const [y, m, d] = item.dueDate.substring(0, 10).split('-').map(Number);
  startDate = new Date(y, m - 1, d, 0, 0, 0, 0).toISOString(); // local midnight
  endDate = new Date(y, m - 1, d, 23, 59, 59, 999).toISOString(); // local 23:59
}
```
</CodeGroup>

<Note>
Task **`dueDate`** for date-only dues is stored as **noon UTC** on the calendar day (`YYYY-MM-DDT12:00:00.000Z`) so the date prefix stays stable. The UI must treat the due **day** as the **`YYYY-MM-DD` prefix**, not the local calendar day of the instant alone.
</Note>

The `new Date(y, m-1, d, …)` (multi-argument constructor) **always uses the browser's local timezone**, so the event appears on April 22 no matter what offset the user is in.

### External Calendar Events

External providers (Google/Outlook) follow the Google Calendar convention:

<CodeGroup>
```typescript Frontend (use-calendar-events.ts → calendarExternalItemToEvent)
if (item.isAllDay && item.startDate && item.endDate) {
  const [sy, sm, sd] = item.startDate.substring(0, 10).split('-').map(Number);
  startDate = new Date(sy, sm - 1, sd, 0, 0, 0, 0).toISOString();

  // Subtract one from end day (exclusive → inclusive)
  const [ey, em, ed] = item.endDate.substring(0, 10).split('-').map(Number);
  endDate = new Date(ey, em - 1, ed - 1, 23, 59, 59, 999).toISOString();
}
```
</CodeGroup>

<Info>
External all-day events arrive as date-only strings (`"2026-04-22"`, not ISO timestamps) and use an **exclusive end date** (April 22 all-day → `endDate = "2026-04-23"`). The subtraction `ed - 1` converts the exclusive end to an inclusive one.
</Info>

## Date Constructor Best Practices

<Warning>
**Avoid `new Date("2026-04-22")`** — This is parsed as **UTC midnight** per the ECMAScript spec. In a UTC+3 timezone this becomes `2026-04-21T21:00:00` local time, shifting the event to the previous day.
</Warning>

<CodeGroup>
```typescript ❌ Wrong
// Parsed as UTC midnight, shifts day in non-UTC timezones
new Date('2026-04-22');
```

```typescript ✅ Correct
// Uses local timezone
new Date(2026, 3, 22); // month is 0-indexed
new Date(y, m - 1, d, 0, 0, 0, 0);
```
</CodeGroup>

## API Reference

### Query Parameters

| Parameter        | Type         | Description                                                     |
| ---------------- | ------------ | --------------------------------------------------------------- |
| `from`           | ISO 8601 UTC | Window start. Frontend sends `subMonths(now, 3).toISOString()` |
| `to`             | ISO 8601 UTC | Window end. Frontend sends `addMonths(now, 6).toISOString()`   |
| `userId`         | UUID         | Optional filter                                                 |
| `integrationIds` | string       | Comma-separated CalendarIntegration IDs                         |

### Agenda Endpoint

**`GET /v1/calendar/agenda`** accepts optional inclusive `from` as UTC ISO for lower bound filtering. The web app passes **`startOfDay(selectedDate).toISOString()`** where `selectedDate` is parsed from `?date=yyyy-MM-dd` using local-date rules.

<Note>
**No `timeZone` parameter.** The backend is timezone-agnostic for all-day events. All date logic lives in the frontend.
</Note>

### Response Fields for CalendarTaskItemDto

| Field        | Type         | Description                                                                                                                        |
| ------------ | ------------ | ---------------------------------------------------------------------------------------------------------------------------------- |
| `startDate`  | ISO 8601 UTC | For `hasDueTime=true`: exact start. For `hasDueTime=false`: floating date label — **frontend must override with local expansion** |
| `endDate`    | ISO 8601 UTC | Same rules as `startDate`                                                                                                         |
| `dueDate`    | ISO 8601 UTC | Source of truth for the date portion on all-day tasks. Frontend reads `substring(0, 10)` to extract `YYYY-MM-DD`                 |
| `hasDueTime` | boolean      | `false` → all-day event, frontend applies floating-date expansion                                                                 |

## Library Usage

<CardGroup cols={2}>
<Card title="Frontend Libraries" icon="code">
- **Native Date constructor** - All-day local expansion
- **date-fns** - General formatting and arithmetic  
- **Intl.DateTimeFormat** - Get user's IANA timezone
</Card>
<Card title="Backend Libraries" icon="server">
- **luxon** - Task list/stats day bounds for timed dues only
- **PostgreSQL timezone()** - Date-only task buckets
</Card>
</CardGroup>

<Warning>
`luxon` is **not used** in the calendar (`GET /v1/calendar`) pipeline. Task list/kanban stats use it only to derive **[todayStart, todayEnd)** instants for **timed** tasks; date-only task buckets use PostgreSQL `timezone()` + UTC calendar dates instead.
</Warning>

## Common Anti-Patterns

<AccordionGroup>
<Accordion title="Wrong date parsing for all-day display">
**Problem:** `new Date("2026-04-22")` for all-day display

**Why it breaks:** UTC parse → wrong day in UTC+/-

**Fix:** Use `new Date(y, m-1, d, …)`
</Accordion>

<Accordion title="Server-side timezone manipulation">
**Problem:** `date.setHours(0, 0, 0)` on the server

**Why it breaks:** Applies server timezone (almost never UTC)

**Fix:** Backend emits UTC date-label; frontend expands locally
</Accordion>

<Accordion title="API timezone coupling">
**Problem:** Sending `timeZone` in API query

**Why it breaks:** Couples backend to client clock; server has no reliable fallback

**Fix:** All-day logic lives in the frontend only
</Accordion>

<Accordion title="Locale-dependent output">
**Problem:** `toLocaleDateString()` for ISO output

**Why it breaks:** Locale-dependent, non-deterministic

**Fix:** Use `.toISOString()` after local construction
</Accordion>

<Accordion title="Direct field usage for all-day tasks">
**Problem:** Using `startDate`/`endDate` directly for all-day tasks

**Why it breaks:** Backend values are placeholders for floating dates

**Fix:** Extract date from `dueDate` and expand locally when `hasDueTime: false`
</Accordion>
</AccordionGroup>
