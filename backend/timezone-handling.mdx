<Note>
**TL;DR** — The API is pure UTC. The frontend converts to local time for display.
All-day / date-only events use the **floating-date model** (same as Google Calendar):
the backend sends a neutral date label; the frontend expands it to local
midnight → 23:59 using `new Date(y, m-1, d, …)`.
</Note>

## HTTP `Time-Zone` Header

### Task List & Kanban Stats

The CRM web app attaches **`Time-Zone`** on authenticated requests using `Intl.DateTimeFormat().resolvedOptions().timeZone`. Nest applies **`TimezoneMiddleware`** globally with the following behavior:

- Header is validated against known IANA zones (`Intl.supportedValuesOf('timeZone')`)
- **Missing or invalid values fall back to `UTC`** (no HTTP 400)
- Middleware sets `req.timezone` for use by `@TimezoneParam()` decorator

**Affected Endpoints:**
- `GET /tasks/list`
- `GET /tasks/kanban`

<Info>
For **`hasDueTime: true`**, `TaskService` uses Luxon to compute **[start, next midnight)** of "today" in the viewer's zone as UTC instants and compares `dueDate` (`timestamptz`) to that window.

For **`hasDueTime: false`** (date-only dues), buckets compare **`(due_date AT TIME ZONE 'UTC')::date`** to **`(timezone(<viewer_tz>, now()))::date`** — not raw instant bounds.
</Info>

<Warning>
JSON responses are unchanged — task `dueDate` fields remain UTC ISO strings; the header affects **grouping and counts**, not field serialization.
</Warning>

#### List Sort Behavior

When using `sort=dueDate` (or default), rows are ordered within each bucket by:

1. **Calendar due day** first (using timezone-aware calendar date rules)
2. **`due_date` instant** for timed tasks
3. **Date-only** rows (`hasDueTime: false`) **after** all timed tasks on that calendar day
4. `title` as final tiebreaker

### Personal Agenda Timeline

**Endpoint:** `GET /v1/calendar/me/agenda/timeline`

The server interprets the following in the viewer's IANA zone:
- **Probe anchor**
- **First-page visibility**
- **Local calendar-day span** (`fromDay` / `toDay` in response)
- **`nextCursor`** (`nextDay`)

<Tip>
Multi-day CRM events and externals that **overlap** the viewer's local "today" can appear on the first page even when `start` is before `max(from, now)`.
</Tip>

<Check>
The key insight: an all-day event is a **calendar date**, not a UTC instant. `April 22` means April 22 regardless of which timezone the viewer is in.
</Check>

## Timed Events - UTC Round-Trip

For events with specific times, the flow is straightforward:

```typescript
// Task created
dueDate = "2026-04-22T10:00:00.000Z"   // stored as timestamptz in PG

// Backend response
{
  startDate: "2026-04-22T10:00:00.000Z",
  endDate:   "2026-04-22T11:00:00.000Z"
}

// Frontend automatically converts to local time (e.g. 13:00 in UTC+3)
```

<Note>
No special handling needed. JavaScript `new Date(isoString)` already produces a `Date` in local time. Any date library that accepts ISO 8601 strings will handle the offset automatically.
</Note>

## All-Day Events - Floating Date Model

### CRM Tasks

For tasks with `source: 'task'` and `hasDueTime: false`:

**Backend Implementation** (`calendar.service.ts → toCalendarTask`):

```typescript
// Extract the YYYY-MM-DD portion — no timezone shift
const iso = dueDate.toISOString().substring(0, 10); // e.g. "2026-04-22"
startDate = new Date(`${iso}T00:00:00.000Z`); // UTC midnight — a date label
endDate = new Date(`${iso}T23:59:59.999Z`); // UTC end — also a date label
```

**Frontend Implementation** (`use-calendar-events.ts → calendarTaskToEvent`):

```typescript
if (!item.hasDueTime && item.dueDate) {
  const [y, m, d] = item.dueDate.substring(0, 10).split('-').map(Number);
  startDate = new Date(y, m - 1, d, 0, 0, 0, 0).toISOString(); // local midnight
  endDate = new Date(y, m - 1, d, 23, 59, 59, 999).toISOString(); // local 23:59
}
```

<Warning>
Task `dueDate` for date-only dues is stored as **noon UTC** on the calendar day (`YYYY-MM-DDT12:00:00.000Z`) so the date prefix stays stable. UI must treat the due **day** as the **`YYYY-MM-DD` prefix**.
</Warning>

### External Calendar Events

External providers (Google/Outlook) follow the Google Calendar convention:

```typescript
if (item.isAllDay && item.startDate && item.endDate) {
  const [sy, sm, sd] = item.startDate.substring(0, 10).split('-').map(Number);
  startDate = new Date(sy, sm - 1, sd, 0, 0, 0, 0).toISOString();

  // Subtract one from end day (exclusive → inclusive)
  const [ey, em, ed] = item.endDate.substring(0, 10).split('-').map(Number);
  endDate = new Date(ey, em - 1, ed - 1, 23, 59, 59, 999).toISOString();
}
```

<Info>
All-day events from external providers use an **exclusive end date** (April 22 all-day → `endDate = "2026-04-23"`). The subtraction `ed - 1` converts this to an inclusive end.
</Info>

## Date Constructor Best Practices

<Tabs>
<Tab title="❌ Wrong">
```typescript
// Parsed as UTC midnight, shifts day in non-UTC timezones
new Date('2026-04-22');
```
</Tab>
<Tab title="✅ Correct">
```typescript
// Uses local timezone
new Date(2026, 3, 22); // month is 0-indexed
new Date(y, m - 1, d, 0, 0, 0, 0);
```
</Tab>
</Tabs>

<Warning>
`new Date("YYYY-MM-DD")` is parsed as **UTC midnight** per the ECMAScript spec. In a UTC+3 timezone this becomes `2026-04-21T21:00:00` local time — the event shifts to the previous day.
</Warning>

## API Contracts

### Query Parameters

| Parameter | Type | Description |
|-----------|------|-------------|
| `from` | ISO 8601 UTC | Window start. Frontend sends `subMonths(now, 3).toISOString()` |
| `to` | ISO 8601 UTC | Window end. Frontend sends `addMonths(now, 6).toISOString()` |
| `userId` | UUID | Optional filter |
| `integrationIds` | string | Comma-separated CalendarIntegration IDs |

### Calendar Agenda Endpoint

**`GET /v1/calendar/agenda`** accepts an optional inclusive `from` parameter as UTC ISO string. The web app passes **`startOfDay(selectedDate).toISOString()`** where `selectedDate` uses the same local-date rule.

<Note>
**No `timeZone` parameter.** The backend is timezone-agnostic for all-day events. All date logic lives in the frontend.
</Note>

### Response Fields for CalendarTaskItemDto

<AccordionGroup>
<Accordion title="startDate">
**Type:** ISO 8601 UTC
- For `hasDueTime=true`: exact start time
- For `hasDueTime=false`: floating date label — **frontend must override with local expansion**
</Accordion>
<Accordion title="endDate">
**Type:** ISO 8601 UTC  
Same rules as `startDate`
</Accordion>
<Accordion title="dueDate">
**Type:** ISO 8601 UTC  
Source of truth for the date portion on all-day tasks. Frontend reads `substring(0, 10)` to extract `YYYY-MM-DD`
</Accordion>
<Accordion title="hasDueTime">
**Type:** boolean  
`false` → all-day event, frontend applies floating-date expansion
</Accordion>
</AccordionGroup>

## Libraries Used

| Layer | Library | Usage |
|-------|---------|-------|
| Frontend | Native `Date` constructor | All-day local expansion (`new Date(y, m-1, d, …)`) |
| Frontend | `date-fns` | General formatting and arithmetic |
| Frontend | `Intl.DateTimeFormat()` | Get user's IANA timezone for display of timed events |
| Backend | `luxon` | Task list/stats day bounds for timed dues only |

<Info>
`luxon` is **not used** in the calendar (`GET /v1/calendar`) pipeline. Task list/kanban stats use it only to derive **[todayStart, todayEnd)** instants for **timed** tasks; date-only task buckets use PostgreSQL `timezone()` + UTC calendar dates instead.
</Info>

## Anti-Patterns to Avoid

<Steps>
<Step title="Don't use new Date() with date strings">
❌ `new Date("2026-04-22")` for all-day display  
**Problem:** UTC parse → wrong day in UTC+/-  
**Fix:** Use `new Date(y, m-1, d, …)`
</Step>

<Step title="Don't apply timezone logic on server">
❌ `date.setHours(0, 0, 0)` on the server  
**Problem:** Applies server timezone (almost never UTC)  
**Fix:** Backend emits UTC date-label; frontend expands locally
</Step>

<Step title="Don't send timezone in API queries">
❌ Sending `timeZone` in API query  
**Problem:** Couples backend to client clock; server has no reliable fallback  
**Fix:** All-day logic lives in the frontend only
</Step>

<Step title="Don't use locale-dependent formatting for ISO output">
❌ `toLocaleDateString()` for ISO output  
**Problem:** Locale-dependent, non-deterministic  
**Fix:** Use `.toISOString()` after local construction
</Step>

<Step title="Don't use backend date fields directly for all-day tasks">
❌ Using `startDate`/`endDate` directly for all-day tasks  
**Problem:** Backend values are placeholders  
**Fix:** Frontend must override with local date expansion
</Step>
</Steps>
