## Core Principle

The PropWise CRM calendar system uses a strict separation between backend UTC storage and frontend local timezone conversion. This ensures consistent behavior across different timezones while maintaining the user experience of local time display.

| Event type | Backend contract | Frontend display |
|------------|------------------|------------------|
| **Timed event** (`hasDueTime=true`) | UTC ISO string (`2026-04-22T10:00:00.000Z`) | Convert to local timezone using `Intl` / `date-fns-tz` |
| **All-day event** (`hasDueTime=false`) | UTC midnight date-label (`2026-04-22T00:00:00.000Z`) | Extract date portion â†’ expand to local midnight â†’ 23:59 |

<Info>
The key insight: an all-day event is a **calendar date**, not a UTC instant.
`April 22` means April 22 regardless of which timezone the viewer is in.
</Info>

## HTTP Time-Zone Header

### Task List & Kanban Stats

The CRM web app attaches **`Time-Zone`** header on authenticated requests using `Intl.DateTimeFormat().resolvedOptions().timeZone`. The Nest application applies **`TimezoneMiddleware`** with these behaviors:

- Header is validated against known IANA zones (`Intl.supportedValuesOf('timeZone')`)
- **Missing or invalid values fall back to `UTC`** (no HTTP 400 error)

**Affected Endpoints:**
- `GET /tasks/list`
- `GET /tasks/kanban`

<Steps>
<Step title="Timed Tasks (hasDueTime: true)">
`TaskService` uses Luxon to compute **[start, next midnight)** of "today" in the viewer's zone as UTC instants and compares `dueDate` (`timestamptz`) to that window.
</Step>

<Step title="Date-only Tasks (hasDueTime: false)">
Buckets compare **`(due_date AT TIME ZONE 'UTC')::date`** (semantic calendar day, aligned with noon-UTC storage) to **`(timezone(<viewer_tz>, now()))::date`** â€” not raw instant bounds.
</Step>
</Steps>

<Warning>
JSON responses are unchanged â€” task `dueDate` fields remain UTC ISO strings. The header affects **grouping and counts**, not field serialization.
</Warning>

### Personal Agenda Timeline

**Endpoint:** `GET /v1/calendar/me/agenda/timeline`

The server interprets **probe anchor**, **first-page visibility**, **local calendar-day span** (`fromDay` / `toDay` in response), and **`nextCursor`** (`nextDay`) in the viewer's IANA zone using the same middleware fallback as tasks.

## Timed Events â€” UTC Round-Trip

Timed events follow a straightforward UTC contract with automatic frontend conversion:

```typescript
// Task created
dueDate = "2026-04-22T10:00:00.000Z"   // stored as timestamptz in PG

// Backend response
{
  startDate: "2026-04-22T10:00:00.000Z",
  endDate:   "2026-04-22T11:00:00.000Z"
}

// Frontend automatically converts to local time (e.g. 13:00 in UTC+3)
```

<Tip>
No special handling needed. JavaScript `new Date(isoString)` already produces a `Date` in local time. Any date library that accepts ISO 8601 strings will handle the offset automatically.
</Tip>

## All-Day Events â€” Floating Date Model

### CRM Tasks (`source: 'task'`, `hasDueTime: false`)

Task **`dueDate`** for date-only dues is stored as **noon UTC** on the calendar day (`YYYY-MM-DDT12:00:00.000Z`) so the date prefix stays stable.

<CodeGroup>
```typescript Backend (calendar.service.ts â†’ toCalendarTask)
// Extract the YYYY-MM-DD portion â€” no timezone shift
const iso = dueDate.toISOString().substring(0, 10); // e.g. "2026-04-22"
startDate = new Date(`${iso}T00:00:00.000Z`); // UTC midnight â€” a date label
endDate = new Date(`${iso}T23:59:59.999Z`); // UTC end â€” also a date label
```

```typescript Frontend (use-calendar-events.ts â†’ calendarTaskToEvent)
if (!item.hasDueTime && item.dueDate) {
  const [y, m, d] = item.dueDate.substring(0, 10).split('-').map(Number);
  startDate = new Date(y, m - 1, d, 0, 0, 0, 0).toISOString(); // local midnight
  endDate = new Date(y, m - 1, d, 23, 59, 59, 999).toISOString(); // local 23:59
}
```
</CodeGroup>

<Check>
`new Date(y, m-1, d, â€¦)` (multi-argument constructor) **always uses the browser's local timezone**, so the event appears on April 22 no matter what offset the user is in.
</Check>

### External Calendar Events (Google / Outlook)

External providers follow the Google Calendar convention with date-only strings and exclusive end dates:

```typescript Frontend (use-calendar-events.ts â†’ calendarExternalItemToEvent)
if (item.isAllDay && item.startDate && item.endDate) {
  const [sy, sm, sd] = item.startDate.substring(0, 10).split('-').map(Number);
  startDate = new Date(sy, sm - 1, sd, 0, 0, 0, 0).toISOString();

  // Subtract one from end day (exclusive â†’ inclusive)
  const [ey, em, ed] = item.endDate.substring(0, 10).split('-').map(Number);
  endDate = new Date(ey, em - 1, ed - 1, 23, 59, 59, 999).toISOString();
}
```

## Date Constructor Best Practices

<Warning>
`new Date("YYYY-MM-DD")` is parsed as **UTC midnight** per the ECMAScript spec.
In a UTC+3 timezone this becomes `2026-04-21T21:00:00` local time â€” the event
shifts to the previous day.
</Warning>

<Tabs>
<Tab title="âťŚ Wrong">
```typescript
// Parsed as UTC midnight, shifts day in non-UTC timezones
new Date('2026-04-22');
```
</Tab>
<Tab title="âś… Correct">
```typescript
// Uses local timezone
new Date(2026, 3, 22); // month is 0-indexed
new Date(y, m - 1, d, 0, 0, 0, 0);
```
</Tab>
</Tabs>

## API Contract

### Query Parameters

| Parameter | Type | Description |
|-----------|------|-------------|
| `from` | ISO 8601 UTC | Window start. Frontend sends `subMonths(now, 3).toISOString()` |
| `to` | ISO 8601 UTC | Window end. Frontend sends `addMonths(now, 6).toISOString()` |
| `userId` | UUID | Optional filter |
| `integrationIds` | string | Comma-separated CalendarIntegration IDs |

<Info>
**No `timeZone` parameter.** The backend is timezone-agnostic for all-day events.
All date logic lives in the frontend.
</Info>

### Response Fields for CalendarTaskItemDto

| Field | Type | Description |
|-------|------|-------------|
| `startDate` | ISO 8601 UTC | For `hasDueTime=true`: exact start. For `hasDueTime=false`: floating date label |
| `endDate` | ISO 8601 UTC | Same rules as `startDate` |
| `dueDate` | ISO 8601 UTC | Source of truth for the date portion on all-day tasks |
| `hasDueTime` | boolean | `false` â†’ all-day event, frontend applies floating-date expansion |

## Libraries Used

<Card title="Frontend Libraries" icon="browser">

<Card title="Backend Libraries" icon="server">
- `luxon` for task list/stats day bounds (timed dues only)
- PostgreSQL `timezone()` for date-only task buckets

<Note>
`luxon` is **not used** in the calendar (`GET /v1/calendar`) pipeline. Task list/kanban stats use it only to derive **[todayStart, todayEnd)** instants for **timed** tasks.
</Note>

## Common Anti-Patterns

<AccordionGroup>
<Accordion title="Wrong Date Constructor Usage">
**Problem:** `new Date("2026-04-22")` for all-day display

**Why it breaks:** UTC parse â†’ wrong day in UTC+/-

**Fix:** Use `new Date(y, m-1, d, â€¦)`
</Accordion>

<Accordion title="Server-side Timezone Manipulation">
**Problem:** `date.setHours(0, 0, 0)` on the server

**Why it breaks:** Applies server timezone (almost never UTC)

**Fix:** Backend emits UTC date-label; frontend expands locally
</Accordion>

<Accordion title="API Timezone Parameters">
**Problem:** Sending `timeZone` in API query

**Why it breaks:** Couples backend to client clock; server has no reliable fallback

**Fix:** All-day logic lives in the frontend only
</Accordion>

<Accordion title="Locale-dependent Output">
**Problem:** `toLocaleDateString()` for ISO output

**Why it breaks:** Locale-dependent, non-deterministic

**Fix:** Use `.toISOString()` after local construction
</Accordion>

<Accordion title="Direct Field Usage for All-day Tasks">
**Problem:** Using `startDate`/`endDate` directly for all-day tasks

**Why it breaks:** Backend values are placeholders

**Fix:** Always check `hasDueTime` and re-derive from `dueDate`
</Accordion>
</AccordionGroup>
