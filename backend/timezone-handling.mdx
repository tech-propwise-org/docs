<Note>
**TL;DR** — The API is pure UTC. The frontend converts to local time for display. All-day / date-only events use the **floating-date model** (same as Google Calendar): the backend sends a neutral date label; the frontend expands it to local midnight → 23:59 using `new Date(y, m-1, d, …)`.
</Note>

## Core Principle

The PropWise backend follows a strict UTC-based architecture for timezone handling, with smart frontend expansion for all-day events:

| Event type | Backend contract | Frontend display |
|------------|------------------|------------------|
| **Timed event** (`hasDueTime=true`) | UTC ISO string (`2026-04-22T10:00:00.000Z`) | Convert to local timezone using `Intl` / `date-fns-tz` |
| **All-day event** (`hasDueTime=false`) | UTC midnight date-label (`2026-04-22T00:00:00.000Z`) | Extract date portion → expand to local midnight → 23:59 |

<Info>
An all-day event is a **calendar date**, not a UTC instant. `April 22` means April 22 regardless of which timezone the viewer is in. This is the same strategy Google Calendar uses (`start.date` / `end.date` fields).
</Info>

## HTTP `Time-Zone` Header

The CRM web app attaches a **`Time-Zone`** header on authenticated requests using `Intl.DateTimeFormat().resolvedOptions().timeZone`. Nest applies **`TimezoneMiddleware`** globally to validate and process this header.

### Middleware Behavior

<Steps>
<Step title="Validation">
The header is validated against known IANA zones using `Intl.supportedValuesOf('timeZone')`
</Step>

<Step title="Fallback">
Missing or invalid values fall back to `UTC` (no HTTP 400 error)
</Step>

<Step title="Request Context">
The middleware sets `req.timezone` for use by the `@TimezoneParam()` decorator
</Step>
</Steps>

### Task List & Kanban Stats

**Endpoints:** `GET /tasks/list`, `GET /tasks/kanban`

<Tabs>
<Tab title="Timed Tasks (hasDueTime: true)">
`TaskService` uses Luxon to compute **[start, next midnight)** of "today" in the viewer's zone as UTC instants and compares `dueDate` (`timestamptz`) to that window.

```typescript
// Example: Computing today's bounds
const todayStart = DateTime.now().setZone(viewerTz).startOf('day').toUTC();
const todayEnd = todayStart.plus({ days: 1 });
```
</Tab>

<Tab title="Date-Only Tasks (hasDueTime: false)">
Buckets compare **`(due_date AT TIME ZONE 'UTC')::date`** (semantic calendar day, aligned with noon-UTC storage) to **`(timezone(<viewer_tz>, now()))::date`**.

```sql
-- Example: Calendar day comparison
WHERE (due_date AT TIME ZONE 'UTC')::date = (timezone($1, now()))::date
```

This ensures a task shown as "30 Apr" cannot land in "today" when the viewer's local day is already "1 May".
</Tab>
</Tabs>

<Warning>
JSON responses are **unchanged** — task `dueDate` fields remain UTC ISO strings. The `Time-Zone` header affects **grouping and counts**, not field serialization.
</Warning>

### List Sort Behavior

When `sort=dueDate` (or default), tasks are ordered within each bucket:

1. **Calendar due day** first (using same rules as buckets)
2. **`due_date` instant** for timed tasks
3. **Date-only rows** (`hasDueTime: false`) **after** all timed tasks on that calendar day
4. **`title`** as final tiebreaker

<Note>
Sort requires the **`Time-Zone`** header so timed dues can be sorted by the viewer's local calendar date.
</Note>

### Personal Agenda Timeline

**Endpoint:** `GET /v1/calendar/me/agenda/timeline`

The server interprets probe anchor, first-page visibility, and local calendar-day span (`fromDay` / `toDay`) in the viewer's IANA zone.

<Accordion title="Timeline Ordering">
- **Tasks:** Use `due` for ordering (`CalendarItemDto.startDate` mirrors due)
- **CRM events and externals:** Use `start`
- **Multi-day events:** Can appear on the first page when they overlap the viewer's local "today", even if `start` is before `max(from, now)`
</Accordion>

Timeline rows include optional **`agendaDayKeys`** clipped to the response window. The frontend agenda list groups by those keys and clamps to `window.toDay` instead of re-grouping from browser-parsed `startDate` / `endDate`.

<Tip>
See `Docs/CALENDAR_SYSTEM_DEEP_DIVE.md` — **Personal agenda timeline** for complete implementation details.
</Tip>

## Timed Events — UTC Round-Trip

Timed events use straightforward UTC instant storage with no special handling required.

```typescript
// Backend storage
Task created: dueDate = "2026-04-22T10:00:00.000Z"   // stored as timestamptz in PG
Backend:      startDate = dueDate,  endDate = startDate + 1h

// API response
{
  startDate: "2026-04-22T10:00:00.000Z",
  endDate:   "2026-04-22T11:00:00.000Z"
}

// Frontend handling
// JavaScript new Date(isoString) automatically converts to local time (e.g. 13:00 in UTC+3)
```

<Check>
JavaScript `new Date(isoString)` already produces a `Date` in local time. Any date library that accepts ISO 8601 strings will handle the offset automatically.
</Check>

## All-Day Events — Floating Date Model

### CRM Tasks (`source: 'task'`, `hasDueTime: false`)

Task **`dueDate`** for date-only dues is stored as **noon UTC** on the calendar day (`YYYY-MM-DDT12:00:00.000Z`) so the date prefix stays stable.

<Info>
UI must treat the due **day** as the **`YYYY-MM-DD` prefix**, not the local calendar day of the instant alone.
</Info>

#### Backend Implementation

```typescript
// calendar.service.ts → toCalendarTask
// Extract the YYYY-MM-DD portion — no timezone shift
const iso = dueDate.toISOString().substring(0, 10); // e.g. "2026-04-22"
startDate = new Date(`${iso}T00:00:00.000Z`); // UTC midnight — a date label
endDate = new Date(`${iso}T23:59:59.999Z`); // UTC end — also a date label
```

The backend also returns `dueDate` as a raw ISO string and `hasDueTime: false`.

#### Frontend Implementation

```typescript
// use-calendar-events.ts → calendarTaskToEvent
if (!item.hasDueTime && item.dueDate) {
  const [y, m, d] = item.dueDate.substring(0, 10).split('-').map(Number);
  startDate = new Date(y, m - 1, d, 0, 0, 0, 0).toISOString(); // local midnight
  endDate = new Date(y, m - 1, d, 23, 59, 59, 999).toISOString(); // local 23:59
}
```

<Note>
`new Date(y, m-1, d, …)` (multi-argument constructor) **always uses the browser's local timezone**, so the event appears on April 22 no matter what offset the user is in.
</Note>

`isAllDay: !item.hasDueTime` is also set on the returned `ICalendarEvent` so the calendar UI can render the event in the all-day row instead of the time grid.

### External Calendar Events (Google / Outlook)

External providers already follow the Google Calendar convention: all-day events arrive as date-only strings (`"2026-04-22"`, not ISO timestamps) and use an **exclusive end date**.

<Info>
April 22 all-day → `endDate = "2026-04-23"` (exclusive)
</Info>

```typescript
// use-calendar-events.ts → calendarExternalItemToEvent
if (item.isAllDay && item.startDate && item.endDate) {
  const [sy, sm, sd] = item.startDate.substring(0, 10).split('-').map(Number);
  startDate = new Date(sy, sm - 1, sd, 0, 0, 0, 0).toISOString();

  // Subtract one from end day (exclusive → inclusive)
  const [ey, em, ed] = item.endDate.substring(0, 10).split('-').map(Number);
  endDate = new Date(ey, em - 1, ed - 1, 23, 59, 59, 999).toISOString();
}
```

The subtraction `ed - 1` converts the exclusive end to an inclusive one.

## Date Constructor Best Practices

<Warning>
`new Date("YYYY-MM-DD")` is parsed as **UTC midnight** per the ECMAScript spec. In a UTC+3 timezone this becomes `2026-04-21T21:00:00` local time — the event shifts to the previous day.
</Warning>

### Correct Usage

Always use the multi-argument constructor for all-day dates:

<CodeGroup>
```typescript Wrong
// ❌ Parsed as UTC midnight, shifts day in non-UTC timezones
new Date('2026-04-22');
```

```typescript Correct
// ✅ Uses local timezone
new Date(2026, 3, 22); // month is 0-indexed
new Date(y, m - 1, d, 0, 0, 0, 0);
```
</CodeGroup>

## API Contract

### Calendar Endpoints

<AccordionGroup>
<Accordion title="Query Parameters">
| Param | Type | Notes |
|-------|------|-------|
| `from` | ISO 8601 UTC | Window start. Frontend sends `subMonths(now, 3).toISOString()`. |
| `to` | ISO 8601 UTC | Window end. Frontend sends `addMonths(now, 6).toISOString()`. |
| `userId` | UUID | Optional filter. |
| `integrationIds` | string | Comma-separated CalendarIntegration IDs. |

<Note>
**No `timeZone` parameter.** The backend is timezone-agnostic for all-day events. All date logic lives in the frontend.
</Note>
</Accordion>

<Accordion title="GET /v1/calendar/agenda">
Optional inclusive `from` as UTC ISO: lower bound for which merged rows are returned.

The web app passes **`startOfDay(selectedDate).toISOString()`** where `selectedDate` is the viewer's calendar day from `?date=yyyy-MM-dd` (parsed with `new Date(y, m-1, d)`, same local-date rule).

**Response:** `{ items }` only (same shapes as `GET /v1/calendar`)

<Info>
No cursor pagination — the backend scans forward up to an internal cap and returns every item in range.
</Info>
</Accordion>
</AccordionGroup>

### CalendarTaskItemDto Response Fields

| Field | Type | Notes |
|-------|------|-------|
| `startDate` | ISO 8601 UTC | For `hasDueTime=true`: exact start. For `hasDueTime=false`: floating date label — **frontend must override with local expansion**. |
| `endDate` | ISO 8601 UTC | Same rules as `startDate`. |
| `dueDate` | ISO 8601 UTC | Source of truth for the date portion on all-day tasks. Frontend reads `substring(0, 10)` to extract `YYYY-MM-DD`. |
| `hasDueTime` | boolean | `false` → all-day event, frontend applies floating-date expansion. |

## Libraries & Tools

<CardGroup cols={2}>
<Card title="Frontend" icon="browser">
- Native `Date` constructor for all-day local expansion
- `date-fns` for formatting and arithmetic
- `Intl.DateTimeFormat()` for user's IANA timezone
</Card>

<Card title="Backend" icon="server">
- `luxon` for task list/stats day bounds (timed dues only)
- **Not used** in calendar merge pipeline
- PostgreSQL `timezone()` for date-only task buckets
</Card>

### Library Usage Details

| Layer | Library | Usage |
|-------|---------|-------|
| Frontend | Native `Date` constructor | All-day local expansion (`new Date(y, m-1, d, …)`) |
| Frontend | `date-fns` (`parseISO`, `isSameDay`, etc.) | General formatting and arithmetic |
| Frontend | `Intl.DateTimeFormat().resolvedOptions().timeZone` | Get user's IANA tz for **display** of timed events |
| Backend | `luxon` (task list / stats only) | Used in `viewer-calendar-day.util` — **not** in the calendar merge pipeline |

<Warning>
`luxon` is **not used** in the calendar (`GET /v1/calendar`) pipeline. Task list/kanban stats use it only to derive **[todayStart, todayEnd)** instants for **timed** tasks; date-only task buckets use PostgreSQL `timezone()` + UTC calendar dates instead.
</Warning>

## Anti-Patterns to Avoid

<AccordionGroup>
<Accordion title="new Date('2026-04-22') for all-day display">
**Problem:** UTC parse → wrong day in UTC+/-

**Fix:** Use `new Date(y, m-1, d, …)`

```typescript
// ❌ Wrong
const date = new Date('2026-04-22'); // UTC parse

// ✅ Correct
const [y, m, d] = '2026-04-22'.split('-').map(Number);
const date = new Date(y, m - 1, d, 0, 0, 0, 0);
```
</Accordion>

<Accordion title="date.setHours(0, 0, 0) on the server">
**Problem:** Applies server timezone (almost never UTC)

**Fix:** Backend emits UTC date-label; frontend expands locally

```typescript
// ❌ Wrong (server-side)
const date = new Date();
date.setHours(0, 0, 0, 0); // Uses server timezone!

// ✅ Correct (server-side)
const iso = dueDate.toISOString().substring(0, 10);
const startDate = new Date(`${iso}T00:00:00.000Z`);
```
</Accordion>

<Accordion title="Sending timeZone in API query">
**Problem:** Couples backend to client clock; server has no reliable fallback

**Fix:** All-day logic lives in the frontend only

<Info>
The backend should remain timezone-agnostic for all-day events. Use the `Time-Zone` header for server-side operations that require timezone awareness (like task list bucketing).
</Info>
</Accordion>

<Accordion title="toLocaleDateString() for ISO output">
**Problem:** Locale-dependent formatting, not ISO 8601

**Fix:** Use `toISOString()` or format with `date-fns`

```typescript
// ❌ Wrong
const dateStr = date.toLocaleDateString(); // "4/22/2026" or "22/4/2026"

// ✅ Correct
const dateStr = date.toISOString().substring(0, 10); // "2026-04-22"
// or
import { format } from 'date-fns';
const dateStr = format(date, 'yyyy-MM-dd');
```
</Accordion>
</AccordionGroup>

## Summary

<Steps>
<Step title="Timed Events">
Store and transmit as UTC ISO strings. Frontend automatically converts to local time for display.
</Step>

<Step title="All-Day Events">
Backend sends UTC date-labels (midnight markers). Frontend expands to local midnight → 23:59 using multi-argument `Date` constructor.
</Step>

<Step title="Time-Zone Header">
Use for server-side operations requiring timezone awareness (task bucketing, agenda timeline). Falls back to UTC if missing or invalid.
</Step>

<Step title="Libraries">
- Frontend: Native `Date`, `date-fns`, `Intl`
- Backend: `luxon` for task stats only, PostgreSQL for date-only comparisons
</Step>
</Steps>

<Tip>
This architecture ensures consistent date handling across timezones while maintaining a clean separation between backend (UTC-centric) and frontend (local-time aware) concerns.
</Tip>
