<Note>
**TL;DR** — The API is pure UTC. The frontend converts to local time for display.
All-day / date-only events use the **floating-date model** (same as Google Calendar):
the backend sends a neutral date label; the frontend expands it to local
midnight → 23:59 using `new Date(y, m-1, d, …)`.
</Note>

## Core Principle

The PropWise calendar system maintains a clear separation of concerns between backend and frontend timezone handling:

| Event type                             | Backend contract                                     | Frontend display                                        |
| -------------------------------------- | ---------------------------------------------------- | ------------------------------------------------------- |
| **Timed event** (`hasDueTime=true`)    | UTC ISO string (`2026-04-22T10:00:00.000Z`)          | Convert to local timezone using `Intl` / `date-fns-tz`  |
| **All-day event** (`hasDueTime=false`) | UTC midnight date-label (`2026-04-22T00:00:00.000Z`) | Extract date portion → expand to local midnight → 23:59 |

<Info>
The key insight: an all-day event is a **calendar date**, not a UTC instant.
`April 22` means April 22 regardless of which timezone the viewer is in.
Storing `2026-04-22T00:00:00.000Z` and letting the frontend expand it locally is
the same strategy Google Calendar uses (`start.date` / `end.date` fields).
</Info>

## HTTP `Time-Zone` Header

### Task List & Kanban Stats

The CRM web app attaches **`Time-Zone`** on authenticated requests using `Intl.DateTimeFormat().resolvedOptions().timeZone`. Nest applies **`TimezoneMiddleware`** globally:

- The header is validated against known IANA zones (`Intl.supportedValuesOf('timeZone')`)
- **Missing or invalid values fall back to `UTC`** (no HTTP 400)
- The middleware sets `req.timezone` for use by `@TimezoneParam()` decorator

**Affected endpoints:**
- `GET /tasks/list`
- `GET /tasks/kanban`

#### Behavior for Task Bucketing

<Tabs>
  <Tab title="Timed Tasks (hasDueTime: true)">
    `TaskService` uses Luxon to compute **[start, next midnight)** of "today" in the viewer's zone as UTC instants and compares `dueDate` (`timestamptz`) to that window.
  </Tab>
  <Tab title="Date-Only Tasks (hasDueTime: false)">
    Buckets compare **`(due_date AT TIME ZONE 'UTC')::date`** (semantic calendar day, aligned with noon-UTC storage) to **`(timezone(<viewer_tz>, now()))::date`** — not raw instant bounds.
    
    This ensures a task shown as "30 Apr" cannot land in "today" when the viewer's local day is already "1 May".
  </Tab>
</Tabs>

<Warning>
**JSON responses are unchanged** — task `dueDate` fields remain UTC ISO strings. The header affects **grouping and counts**, not field serialization.
</Warning>

#### List Sort Order

When using `sort=dueDate` (default), within each bucket (overdue / today / upcoming / completed), rows are ordered by:

<Steps>
  <Step title="Calendar Due Day">
    Local calendar date from `due_date` when `hasDueTime`, else `(due_date AT TIME ZONE 'UTC')::date`
  </Step>
  <Step title="Due Date Instant">
    For timed tasks, order by `due_date` instant
  </Step>
  <Step title="Date-Only Tasks">
    All date-only rows (`hasDueTime: false`) appear **after** timed tasks on the same calendar day
  </Step>
  <Step title="Title">
    Finally sort by `title`
  </Step>
</Steps>

<Note>
Requires **`Time-Zone`** header so timed dues sort by the viewer's local calendar date.
</Note>

### Personal Agenda Timeline

**Endpoint:** `GET /v1/calendar/me/agenda/timeline`

The server interprets **probe anchor**, **first-page visibility**, **local calendar-day span** (`fromDay` / `toDay` in the response), and **`nextCursor`** (`nextDay`) in the viewer's IANA zone (same middleware fallback as tasks).

- **Tasks:** Use **due** for ordering (`CalendarItemDto.startDate` mirrors due)
- **CRM events and externals:** Use **start**
- **Multi-day events:** Events that **overlap** the viewer's local "today" can appear on the first page even when `start` is before `max(from, now)`

Timeline rows include optional **`agendaDayKeys`** clipped to the response window. The frontend agenda list groups by those keys and clamps to `window.toDay` instead of re-grouping from browser-parsed `startDate` / `endDate`.

<Tip>
See `Docs/CALENDAR_SYSTEM_DEEP_DIVE.md` — **Personal agenda timeline** for detailed implementation.
</Tip>

## Timed Events — UTC Round-Trip

Timed events follow a straightforward UTC round-trip pattern with no special handling required:

```typescript
// Task created
dueDate = "2026-04-22T10:00:00.000Z"   // stored as timestamptz in PostgreSQL

// Backend processing
startDate = dueDate
endDate = startDate + 1h

// API response
{
  startDate: "2026-04-22T10:00:00.000Z",
  endDate:   "2026-04-22T11:00:00.000Z"
}

// Frontend display
// Calendar grid converts to local time (e.g. 13:00 in UTC+3)
```

<Info>
JavaScript `new Date(isoString)` already produces a `Date` in local time. Any date library that accepts ISO 8601 strings will handle the offset automatically.
</Info>

## All-Day Events — Floating Date Model

### CRM Tasks

Task **`dueDate`** for date-only dues is stored as **noon UTC** on the calendar day (`YYYY-MM-DDT12:00:00.000Z`) so the date prefix stays stable.

<Warning>
The UI must treat the due **day** as the **`YYYY-MM-DD` prefix**, not the local calendar day of the instant alone. See task detail panel / `parseTaskDueAsLocalDate`.
</Warning>

#### Backend Implementation

```typescript calendar.service.ts
// Extract the YYYY-MM-DD portion — no timezone shift
const iso = dueDate.toISOString().substring(0, 10); // e.g. "2026-04-22"
startDate = new Date(`${iso}T00:00:00.000Z`); // UTC midnight — a date label
endDate = new Date(`${iso}T23:59:59.999Z`); // UTC end — also a date label
```

The backend also returns `dueDate` as a raw ISO string and `hasDueTime: false`.

#### Frontend Implementation

```typescript use-calendar-events.ts
if (!item.hasDueTime && item.dueDate) {
  const [y, m, d] = item.dueDate.substring(0, 10).split('-').map(Number);
  startDate = new Date(y, m - 1, d, 0, 0, 0, 0).toISOString(); // local midnight
  endDate = new Date(y, m - 1, d, 23, 59, 59, 999).toISOString(); // local 23:59
}
```

<Check>
`new Date(y, m-1, d, …)` (multi-argument constructor) **always uses the browser's local timezone**, so the event appears on April 22 no matter what offset the user is in.
</Check>

`isAllDay: !item.hasDueTime` is also set on the returned `ICalendarEvent` so the calendar UI can render the event in the all-day row instead of the time grid.

### External Calendar Events

External providers (Google / Outlook) already follow the Google Calendar convention:

- All-day events arrive as date-only strings (`"2026-04-22"`, not ISO timestamps)
- Use an **exclusive end date** (April 22 all-day → `endDate = "2026-04-23"`)

```typescript use-calendar-events.ts
if (item.isAllDay && item.startDate && item.endDate) {
  const [sy, sm, sd] = item.startDate.substring(0, 10).split('-').map(Number);
  startDate = new Date(sy, sm - 1, sd, 0, 0, 0, 0).toISOString();

  // Subtract one from end day (exclusive → inclusive)
  const [ey, em, ed] = item.endDate.substring(0, 10).split('-').map(Number);
  endDate = new Date(ey, em - 1, ed - 1, 23, 59, 59, 999).toISOString();
}
```

The subtraction `ed - 1` converts the exclusive end to an inclusive one.

## Date Parsing Best Practices

<Warning>
**Why NOT `new Date("2026-04-22")`**

`new Date("YYYY-MM-DD")` is parsed as **UTC midnight** per the ECMAScript spec.
In a UTC+3 timezone this becomes `2026-04-21T21:00:00` local time — the event shifts to the previous day.
</Warning>

Always use the multi-argument constructor:

<CodeGroup>

```typescript ❌ Wrong
// Parsed as UTC midnight, shifts day in non-UTC timezones
new Date('2026-04-22');
```

```typescript ✅ Correct
// Uses local timezone
new Date(2026, 3, 22); // month is 0-indexed
new Date(y, m - 1, d, 0, 0, 0, 0);
```

</CodeGroup>

## API Contract

### Query Parameters

| Param            | Type         | Notes                                                           |
| ---------------- | ------------ | --------------------------------------------------------------- |
| `from`           | ISO 8601 UTC | Window start. Frontend sends `subMonths(now, 3).toISOString()`. |
| `to`             | ISO 8601 UTC | Window end. Frontend sends `addMonths(now, 6).toISOString()`.   |
| `userId`         | UUID         | Optional filter.                                                |
| `integrationIds` | string       | Comma-separated CalendarIntegration IDs.                        |

#### Agenda Endpoint

**`GET /v1/calendar/agenda`** accepts optional inclusive `from` as UTC ISO: lower bound for which merged rows are returned.

The web app passes **`startOfDay(selectedDate).toISOString()`** where `selectedDate` is the viewer's calendar day from `?date=yyyy-MM-dd` (parsed with `new Date(y, m-1, d)`, same local-date rule).

**Response:** `{ items }` only (same shapes as `GET /v1/calendar`)

<Info>
There is **no cursor pagination** — the backend scans forward up to an internal cap and returns every item in range.
</Info>

<Note>
**No `timeZone` parameter.** The backend is timezone-agnostic for all-day events. All date logic lives in the frontend.
</Note>

### Response Fields for CalendarTaskItemDto

| Field        | Type         | Notes                                                                                                                              |
| ------------ | ------------ | ---------------------------------------------------------------------------------------------------------------------------------- |
| `startDate`  | ISO 8601 UTC | For `hasDueTime=true`: exact start. For `hasDueTime=false`: floating date label — **frontend must override with local expansion**. |
| `endDate`    | ISO 8601 UTC | Same rules as `startDate`.                                                                                                         |
| `dueDate`    | ISO 8601 UTC | Source of truth for the date portion on all-day tasks. Frontend reads `substring(0, 10)` to extract `YYYY-MM-DD`.                  |
| `hasDueTime` | boolean      | `false` → all-day event, frontend applies floating-date expansion.                                                                 |

## Common Anti-Patterns

Avoid these common mistakes when handling timezones:

<AccordionGroup>
  <Accordion title="❌ Using new Date('2026-04-22') for all-day display">
    **Problem:** UTC parse → wrong day in UTC+/- timezones
    
    **Fix:** Use `new Date(y, m-1, d, …)` multi-argument constructor
  </Accordion>

  <Accordion title="❌ Calling date.setHours(0, 0, 0) on the server">
    **Problem:** Applies server timezone (almost never UTC)
    
    **Fix:** Backend emits UTC date-label; frontend expands locally
  </Accordion>

  <Accordion title="❌ Sending timeZone in API query">
    **Problem:** Couples backend to client clock; server has no reliable fallback
    
    **Fix:** All-day logic lives in the frontend only
  </Accordion>

  <Accordion title="❌ Using toLocaleDateString() for ISO output">
    **Problem:** Produces locale-formatted strings, not ISO 8601
    
    **Fix:** Use `toISOString()` or `date-fns` formatting functions
  </Accordion>
</AccordionGroup>

## Summary

<Steps>
  <Step title="Backend Sends UTC">
    All timestamps are UTC ISO 8601 strings. All-day events use UTC midnight as date labels.
  </Step>
  <Step title="Frontend Expands Locally">
    Timed events convert to local timezone automatically. All-day events use `new Date(y, m-1, d, …)` for local expansion.
  </Step>
  <Step title="Time-Zone Header for Bucketing">
    Task list and agenda endpoints use the `Time-Zone` header to determine local calendar day boundaries for grouping.
  </Step>
  <Step title="Floating Date Model">
    All-day events follow the Google Calendar convention: calendar dates, not UTC instants.
  </Step>
</Steps>
