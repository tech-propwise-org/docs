<Note>
**TL;DR** — The API is pure UTC. The frontend converts to local time for display.
All-day / date-only events use the **floating-date model** (same as Google Calendar):
the backend sends a neutral date label; the frontend expands it to local
midnight → 23:59 using `new Date(y, m-1, d, …)`.
</Note>

## Core Principle

PropWise CRM follows a strict timezone separation model where the backend maintains timezone-agnostic contracts and the frontend handles all timezone conversion for display purposes.

| Event type                             | Backend contract                                     | Frontend display                                        |
| -------------------------------------- | ---------------------------------------------------- | ------------------------------------------------------- |
| **Timed event** (`hasDueTime=true`)    | UTC ISO string (`2026-04-22T10:00:00.000Z`)          | Convert to local timezone using `Intl` / `date-fns-tz`  |
| **All-day event** (`hasDueTime=false`) | UTC midnight date-label (`2026-04-22T00:00:00.000Z`) | Extract date portion → expand to local midnight → 23:59 |

<Info>
The key insight: an all-day event is a **calendar date**, not a UTC instant.
`April 22` means April 22 regardless of which timezone the viewer is in.
This is the same strategy Google Calendar uses (`start.date` / `end.date` fields).
</Info>

## HTTP `Time-Zone` Header

### Task List & Kanban Stats

The CRM web app attaches **`Time-Zone`** header on authenticated requests using `Intl.DateTimeFormat().resolvedOptions().timeZone`. Nest applies **`TimezoneMiddleware`** globally with the following behavior:

- Header is validated against known IANA zones (`Intl.supportedValuesOf('timeZone')`)
- **Missing or invalid values fall back to `UTC`** (no HTTP 400 error)
- Middleware sets `req.timezone` for use by `@TimezoneParam()` decorator

**Affected Endpoints:**
- `GET /tasks/list`
- `GET /tasks/kanban`

<Steps>
<Step title="Timed Tasks (hasDueTime: true)">
`TaskService` uses Luxon to compute **[start, next midnight)** of "today" in the viewer's zone as UTC instants and compares `dueDate` (`timestamptz`) to that window.
</Step>

<Step title="Date-only Tasks (hasDueTime: false)">
Buckets compare **`(due_date AT TIME ZONE 'UTC')::date`** (semantic calendar day, aligned with noon-UTC storage) to **`(timezone(<viewer_tz>, now()))::date`** — not raw instant bounds.
</Step>

<Step title="Response Handling">
JSON responses are unchanged — task `dueDate` fields remain UTC ISO strings; the header affects **grouping and counts**, not field serialization.
</Step>
</Steps>

<Warning>
A task shown as "30 Apr" cannot land in "today" when the viewer's local day is already "1 May" due to the semantic calendar day comparison.
</Warning>

### Personal Agenda Timeline

**Endpoint:** `GET /v1/calendar/me/agenda/timeline`

The server interprets **probe anchor**, **first-page visibility**, **local calendar-day span** (`fromDay` / `toDay` in the response), and **`nextCursor`** (`nextDay`) in the viewer's IANA zone using the same middleware fallback as tasks.

- Tasks use **due date** for ordering (`CalendarItemDto.startDate` mirrors due)
- CRM events and externals use **start date**
- Multi-day events that **overlap** the viewer's local "today" can appear on the first page

## Timed Events — UTC Round-Trip

Timed events follow a straightforward UTC contract with no special handling needed:

```typescript
// Task creation
Task created: dueDate = "2026-04-22T10:00:00.000Z"   // stored as timestamptz in PG

// Backend processing
Backend: startDate = dueDate, endDate = startDate + 1h

// API response
{
  startDate: "2026-04-22T10:00:00.000Z",
  endDate: "2026-04-22T11:00:00.000Z"
}

// Frontend display
// Calendar grid converts to local time (e.g. 13:00 in UTC+3)
```

<Tip>
JavaScript `new Date(isoString)` already produces a `Date` in local time. Any date library that accepts ISO 8601 strings will handle the offset automatically.
</Tip>

## All-Day Events — Floating Date Model

### CRM Tasks (source: 'task', hasDueTime: false)

Task **`dueDate`** for date-only dues is stored as **noon UTC** on the calendar day (`YYYY-MM-DDT12:00:00.000Z`) so the date prefix stays stable.

<CodeGroup>

```typescript Backend (calendar.service.ts → toCalendarTask)
// Extract the YYYY-MM-DD portion — no timezone shift
const iso = dueDate.toISOString().substring(0, 10); // e.g. "2026-04-22"
startDate = new Date(`${iso}T00:00:00.000Z`); // UTC midnight — a date label
endDate = new Date(`${iso}T23:59:59.999Z`); // UTC end — also a date label
```

```typescript Frontend (use-calendar-events.ts → calendarTaskToEvent)
if (!item.hasDueTime && item.dueDate) {
  const [y, m, d] = item.dueDate.substring(0, 10).split('-').map(Number);
  startDate = new Date(y, m - 1, d, 0, 0, 0, 0).toISOString(); // local midnight
  endDate = new Date(y, m - 1, d, 23, 59, 59, 999).toISOString(); // local 23:59
}
```

</CodeGroup>

<Check>
`new Date(y, m-1, d, …)` (multi-argument constructor) **always uses the browser's local timezone**, so the event appears on April 22 no matter what offset the user is in.
</Check>

### External Calendar Events (Google / Outlook)

External providers follow the Google Calendar convention with date-only strings and exclusive end dates:

```typescript Frontend (use-calendar-events.ts → calendarExternalItemToEvent)
if (item.isAllDay && item.startDate && item.endDate) {
  const [sy, sm, sd] = item.startDate.substring(0, 10).split('-').map(Number);
  startDate = new Date(sy, sm - 1, sd, 0, 0, 0, 0).toISOString();

  // Subtract one from end day (exclusive → inclusive)
  const [ey, em, ed] = item.endDate.substring(0, 10).split('-').map(Number);
  endDate = new Date(ey, em - 1, ed - 1, 23, 59, 59, 999).toISOString();
}
```

The subtraction `ed - 1` converts the exclusive end to an inclusive one.

## Date Constructor Best Practices

<Warning>
`new Date("YYYY-MM-DD")` is parsed as **UTC midnight** per the ECMAScript spec.
In a UTC+3 timezone this becomes `2026-04-21T21:00:00` local time — the event shifts to the previous day.
</Warning>

<Tabs>
<Tab title="❌ Wrong">
```typescript
// Parsed as UTC midnight, shifts day in non-UTC timezones
new Date('2026-04-22');
```
</Tab>

<Tab title="✅ Correct">
```typescript
// Uses local timezone
new Date(2026, 3, 22); // month is 0-indexed
new Date(y, m - 1, d, 0, 0, 0, 0);
```
</Tab>
</Tabs>

## API Contract

### Query Parameters

| Parameter        | Type         | Description                                                     |
| ---------------- | ------------ | --------------------------------------------------------------- |
| `from`           | ISO 8601 UTC | Window start. Frontend sends `subMonths(now, 3).toISOString()` |
| `to`             | ISO 8601 UTC | Window end. Frontend sends `addMonths(now, 6).toISOString()`   |
| `userId`         | UUID         | Optional filter for specific user                               |
| `integrationIds` | string       | Comma-separated CalendarIntegration IDs                         |

<Info>
**`GET /v1/calendar/agenda`** accepts optional inclusive `from` as UTC ISO: lower bound for merged rows. The web app passes **`startOfDay(selectedDate).toISOString()`** where `selectedDate` uses `new Date(y, m-1, d)` for local-date parsing.
</Info>

### Response Fields for CalendarTaskItemDto

| Field        | Type         | Description                                                                                                |
| ------------ | ------------ | ---------------------------------------------------------------------------------------------------------- |
| `startDate`  | ISO 8601 UTC | For `hasDueTime=true`: exact start. For `hasDueTime=false`: floating date label requiring frontend expansion |
| `endDate`    | ISO 8601 UTC | Same rules as `startDate`                                                                                  |
| `dueDate`    | ISO 8601 UTC | Source of truth for date portion. Frontend reads `substring(0, 10)` to extract `YYYY-MM-DD`              |
| `hasDueTime` | boolean      | `false` → all-day event, frontend applies floating-date expansion                                          |

<Note>
**No `timeZone` parameter needed.** The backend is timezone-agnostic for all-day events. All date logic lives in the frontend.
</Note>

## Libraries and Tools

| Layer    | Library                                     | Usage                                                    |
| -------- | ------------------------------------------- | -------------------------------------------------------- |
| Frontend | Native `Date` constructor                   | All-day local expansion (`new Date(y, m-1, d, …)`)      |
| Frontend | `date-fns` (`parseISO`, `isSameDay`, etc.)  | General formatting and arithmetic                        |
| Frontend | `Intl.DateTimeFormat().resolvedOptions()`   | Get user's IANA timezone for display of timed events    |
| Backend  | `luxon` (task list/stats only)             | Day bounds for timed dues in `viewer-calendar-day.util` |

<Warning>
`luxon` is **not used** in the calendar (`GET /v1/calendar`) pipeline. Task list/kanban stats use it only for **timed** tasks; date-only task buckets use PostgreSQL `timezone()` + UTC calendar dates.
</Warning>

## Common Anti-Patterns

<AccordionGroup>
<Accordion title="Wrong Date Constructor Usage">

**Problem:** `new Date("2026-04-22")` for all-day display  
**Why it breaks:** UTC parse → wrong day in UTC+/-  
**Fix:** Use `new Date(y, m-1, d, …)`

</Accordion>

<Accordion title="Server-Side Timezone Manipulation">

**Problem:** `date.setHours(0, 0, 0)` on the server  
**Why it breaks:** Applies server timezone (almost never UTC)  
**Fix:** Backend emits UTC date-label; frontend expands locally

</Accordion>

<Accordion title="API Timezone Parameters">

**Problem:** Sending `timeZone` in API query  
**Why it breaks:** Couples backend to client clock; server has no reliable fallback  
**Fix:** All-day logic lives in the frontend only

</Accordion>

<Accordion title="Locale-Dependent Output">

**Problem:** `toLocaleDateString()` for ISO output  
**Why it breaks:** Locale-dependent, non-deterministic  
**Fix:** Use `.toISOString()` after local construction

</Accordion>

<Accordion title="Direct Backend Values for All-Day">

**Problem:** Using `startDate`/`endDate` directly for all-day tasks  
**Why it breaks:** Backend values are placeholders for floating dates  
**Fix:** Extract date portion from `dueDate` and expand locally

</Accordion>
</AccordionGroup>
