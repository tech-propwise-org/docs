<Note>
**TL;DR** — The API is pure UTC. The frontend converts to local time for display.
All-day / date-only events use the **floating-date model** (same as Google Calendar):
the backend sends a neutral date label; the frontend expands it to local
midnight → 23:59 using `new Date(y, m-1, d, …)`.
</Note>

## HTTP Time-Zone header (task list & kanban stats)

The CRM web app attaches **`Time-Zone`** on authenticated requests (`Intl.DateTimeFormat().resolvedOptions().timeZone`). Nest applies **`TimezoneMiddleware`** globally: the header is validated against known IANA zones (`Intl.supportedValuesOf('timeZone')`); **missing or invalid values fall back to `UTC`** (no HTTP 400). The middleware sets `req.timezone` for use by `@TimezoneParam()` decorator.

**Endpoints:** `GET /tasks/list`, `GET /tasks/kanban`.

**Behavior:** For **`hasDueTime: true`**, `TaskService` uses Luxon to compute **[start, next midnight)** of "today" in the viewer's zone as UTC instants and compares `dueDate` (`timestamptz`) to that window. For **`hasDueTime: false`** (date-only dues), buckets compare **`(due_date AT TIME ZONE 'UTC')::date`** (semantic calendar day, aligned with noon-UTC storage) to **`(timezone(<viewer_tz>, now()))::date`** — not raw instant bounds — so a task shown as "30 Apr" cannot land in "today" when the viewer's local day is already "1 May". List **stats** (`overdue`, `dueToday`) use the same rules.

<Warning>
**JSON responses are unchanged** — task `dueDate` fields remain UTC ISO strings; the header affects **grouping and counts**, not field serialization.
</Warning>

**List sort (`sort=dueDate` / default):** Within each overdue / today / upcoming / completed bucket, rows are ordered by **calendar due day** first (same rules as buckets: local calendar date from `due_date` when `hasDueTime`, else `(due_date AT TIME ZONE 'UTC')::date`), then by **`due_date` instant** for timed tasks, then **date-only** rows (`hasDueTime: false`) **after** all timed tasks on that calendar day, then `title`. Requires **`Time-Zone`** so timed dues sort by the viewer's local calendar date.

## HTTP Time-Zone header — personal agenda timeline

**Endpoint:** `GET /v1/calendar/me/agenda/timeline`.

**Behavior:** The server interprets **probe anchor**, **first-page visibility**, **local calendar-day span** (`fromDay` / `toDay` in the response), and **`nextCursor`** (`nextDay`) in the viewer's IANA zone (same middleware fallback as tasks). Tasks use **due** for ordering (`CalendarItemDto.startDate` mirrors due); CRM events and externals use **start**. Multi-day CRM events and externals that **overlap** the viewer's local "today" can appear on the first page even when `start` is before `max(from, now)`.

## Core principle

<Tabs>
  <Tab title="Timed events">
    **Backend contract:** UTC ISO string (`2026-04-22T10:00:00.000Z`)
    
    **Frontend display:** Convert to local timezone using `Intl` / `date-fns-tz`
  </Tab>
  <Tab title="All-day events">
    **Backend contract:** UTC midnight date-label (`2026-04-22T00:00:00.000Z`)
    
    **Frontend display:** Extract date portion → expand to local midnight → 23:59
  </Tab>
</Tabs>

<Info>
The key insight: an all-day event is a **calendar date**, not a UTC instant.
`April 22` means April 22 regardless of which timezone the viewer is in.
Storing `2026-04-22T00:00:00.000Z` and letting the frontend expand it locally is
the same strategy Google Calendar uses (`start.date` / `end.date` fields).
</Info>

## Timed events — UTC round-trip

**No special handling needed.** JavaScript `new Date(isoString)` already produces a `Date` in local time. Any date library that accepts ISO 8601 strings will handle the offset automatically.

```typescript
// Task created: dueDate = "2026-04-22T10:00:00.000Z" (stored as timestamptz in PG)
// Backend: startDate = dueDate, endDate = startDate + 1h
// API response:
{
  "startDate": "2026-04-22T10:00:00.000Z",
  "endDate": "2026-04-22T11:00:00.000Z"
}
// Frontend: calendar grid converts to local time (e.g. 13:00 in UTC+3)
```

## All-day events — floating date model

### CRM tasks (`source: 'task'`, `hasDueTime: false`)

Task **`dueDate`** for date-only dues is stored as **noon UTC** on the calendar day (`YYYY-MM-DDT12:00:00.000Z`) so the date prefix stays stable; UI must treat the due **day** as the **`YYYY-MM-DD` prefix** (see task detail panel / `parseTaskDueAsLocalDate`), not the local calendar day of the instant alone.

<Steps>
<Step title="Backend processing">
```typescript
// Extract the YYYY-MM-DD portion — no timezone shift
const iso = dueDate.toISOString().substring(0, 10); // e.g. "2026-04-22"
startDate = new Date(`${iso}T00:00:00.000Z`); // UTC midnight — a date label
endDate = new Date(`${iso}T23:59:59.999Z`); // UTC end — also a date label
```

The backend also returns `dueDate` as a raw ISO string and `hasDueTime: false`.
</Step>

<Step title="Frontend processing">
```typescript
if (!item.hasDueTime && item.dueDate) {
  const [y, m, d] = item.dueDate.substring(0, 10).split('-').map(Number);
  startDate = new Date(y, m - 1, d, 0, 0, 0, 0).toISOString(); // local midnight
  endDate = new Date(y, m - 1, d, 23, 59, 59, 999).toISOString(); // local 23:59
}
```

`new Date(y, m-1, d, …)` (multi-argument constructor) **always uses the browser's local timezone**, so the event appears on April 22 no matter what offset the user is in.

`isAllDay: !item.hasDueTime` is also set on the returned `ICalendarEvent` so the calendar UI can render the event in the all-day row instead of the time grid.
</Step>
</Steps>

### External calendar events (Google / Outlook)

External providers already follow the Google Calendar convention: all-day events arrive as date-only strings (`"2026-04-22"`, not ISO timestamps) and use an **exclusive end date** (April 22 all-day → `endDate = "2026-04-23"`).

```typescript
if (item.isAllDay && item.startDate && item.endDate) {
  const [sy, sm, sd] = item.startDate.substring(0, 10).split('-').map(Number);
  startDate = new Date(sy, sm - 1, sd, 0, 0, 0, 0).toISOString();

  // Subtract one from end day (exclusive → inclusive)
  const [ey, em, ed] = item.endDate.substring(0, 10).split('-').map(Number);
  endDate = new Date(ey, em - 1, ed - 1, 23, 59, 59, 999).toISOString();
}
```

The subtraction `ed - 1` converts the exclusive end to an inclusive one.

## Why NOT `new Date("2026-04-22")`

<Warning>
`new Date("YYYY-MM-DD")` is parsed as **UTC midnight** per the ECMAScript spec.
In a UTC+3 timezone this becomes `2026-04-21T21:00:00` local time — the event
shifts to the previous day.
</Warning>

Always use the multi-argument constructor:

<CodeGroup>
```typescript Wrong
// ❌ Wrong — parsed as UTC midnight, shifts day in non-UTC timezones
new Date('2026-04-22');
```

```typescript Correct
// ✅ Correct — uses local timezone
new Date(2026, 3, 22); // month is 0-indexed
new Date(y, m - 1, d, 0, 0, 0, 0);
```
</CodeGroup>

## API contract

### Query parameters

| Param            | Type         | Notes                                                           |
| ---------------- | ------------ | --------------------------------------------------------------- |
| `from`           | ISO 8601 UTC | Window start. Frontend sends `subMonths(now, 3).toISOString()`. |
| `to`             | ISO 8601 UTC | Window end. Frontend sends `addMonths(now, 6).toISOString()`.   |
| `userId`         | UUID         | Optional filter.                                                |
| `integrationIds` | string       | Comma-separated CalendarIntegration IDs.                        |

**`GET /v1/calendar/agenda`** — optional inclusive `from` as UTC ISO: lower bound for which merged rows are returned. The web app passes **`startOfDay(selectedDate).toISOString()`** where `selectedDate` is the viewer's calendar day from `?date=yyyy-MM-dd` (parsed with `new Date(y, m-1, d)`, same local-date rule as above). The response is **`{ items }`** only (same shapes as `GET /v1/calendar`); there is **no cursor pagination** — the backend scans forward up to an internal cap and returns every item in range.

<Note>
**No `timeZone` parameter.** The backend is timezone-agnostic for all-day events.
All date logic lives in the frontend.
</Note>

### Response fields for `CalendarTaskItemDto`

| Field        | Type         | Notes                                                                                                                              |
| ------------ | ------------ | ---------------------------------------------------------------------------------------------------------------------------------- |
| `startDate`  | ISO 8601 UTC | For `hasDueTime=true`: exact start. For `hasDueTime=false`: floating date label — **frontend must override with local expansion**. |
| `endDate`    | ISO 8601 UTC | Same rules as `startDate`.                                                                                                         |
| `dueDate`    | ISO 8601 UTC | Source of truth for the date portion on all-day tasks. Frontend reads `substring(0, 10)` to extract `YYYY-MM-DD`.                  |
| `hasDueTime` | boolean      | `false` → all-day event, frontend applies floating-date expansion.                                                                 |

## Libraries

| Layer    | Library                                                    | Usage                                                                       |
| -------- | ---------------------------------------------------------- | --------------------------------------------------------------------------- |
| Frontend | Native `Date` constructor                                  | All-day local expansion (`new Date(y, m-1, d, …)`)                          |
| Frontend | `date-fns` (`parseISO`, `isSameDay`, etc.)                 | General formatting and arithmetic                                           |
| Frontend | `Intl.DateTimeFormat().resolvedOptions().timeZone`         | Get user's IANA tz for **display** of timed events                          |
| Backend  | `luxon` (task list / stats day bounds for timed dues only) | Used in `viewer-calendar-day.util` — **not** in the calendar merge pipeline |

<Tip>
`luxon` is **not used** in the calendar (`GET /v1/calendar`) pipeline. Task list/kanban stats use it only to derive **[todayStart, todayEnd)** instants for **timed** tasks; date-only task buckets use PostgreSQL `timezone()` + UTC calendar dates instead.
</Tip>

## Anti-patterns

<AccordionGroup>
<Accordion title="new Date('2026-04-22') for all-day display">
**Why it breaks:** UTC parse → wrong day in UTC+/-

**Fix:** Use `new Date(y, m-1, d, …)`
</Accordion>

<Accordion title="date.setHours(0, 0, 0) on the server">
**Why it breaks:** Applies server timezone (almost never UTC)

**Fix:** Backend emits UTC date-label; frontend expands locally
</Accordion>

<Accordion title="Sending timeZone in API query">
**Why it breaks:** Couples backend to client clock; server has no reliable fallback

**Fix:** All-day logic lives in the frontend only
</Accordion>

<Accordion title="toLocaleDateString() for ISO output">
**Why it breaks:** Locale-dependent, non-deterministic

**Fix:** Use `.toISOString()` after local construction
</Accordion>

<Accordion title="Using startDate/endDate directly for all-day tasks">
**Why it breaks:** Backend values are placeholders

**Fix:** Extract date from `dueDate` field and expand with local constructor
</Accordion>
</AccordionGroup>
