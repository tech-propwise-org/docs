## Core Principle

PropWise CRM uses a clear separation of concerns for timezone handling:

| Event type | Backend contract | Frontend display |
| --- | --- | --- |
| **Timed event** (`hasDueTime=true`) | UTC ISO string (`2026-04-22T10:00:00.000Z`) | Convert to local timezone using `Intl` / `date-fns-tz` |
| **All-day event** (`hasDueTime=false`) | UTC midnight date-label (`2026-04-22T00:00:00.000Z`) | Extract date portion â†’ expand to local midnight â†’ 23:59 |

<Info>
The key insight: an all-day event is a **calendar date**, not a UTC instant.
`April 22` means April 22 regardless of which timezone the viewer is in.
This follows the same strategy Google Calendar uses with `start.date` / `end.date` fields.
</Info>

## HTTP Time-Zone Header

### Task List & Kanban Stats

The CRM web app attaches **`Time-Zone`** header on authenticated requests using `Intl.DateTimeFormat().resolvedOptions().timeZone`. The Nest.js backend applies **`TimezoneMiddleware`** globally with these behaviors:

- Header is validated against known IANA zones (`Intl.supportedValuesOf('timeZone')`)
- **Missing or invalid values fall back to `UTC`** (no HTTP 400 error)
- Middleware sets `req.timezone` for use by `@TimezoneParam()` decorator

**Affected Endpoints:**
- `GET /tasks/list`
- `GET /tasks/kanban`

<Steps>
<Step title="Timed Tasks (hasDueTime: true)">
`TaskService` uses Luxon to compute **[start, next midnight)** of "today" in the viewer's zone as UTC instants and compares `dueDate` (`timestamptz`) to that window.
</Step>

<Step title="Date-only Tasks (hasDueTime: false)">
Buckets compare **`(due_date AT TIME ZONE 'UTC')::date`** (semantic calendar day, aligned with noon-UTC storage) to **`(timezone(<viewer_tz>, now()))::date`** using calendar day comparison, not raw instant bounds.
</Step>
</Steps>

<Warning>
JSON responses are unchanged â€” task `dueDate` fields remain UTC ISO strings. The header affects **grouping and counts**, not field serialization.
</Warning>

### Personal Agenda Timeline

**Endpoint:** `GET /v1/calendar/me/agenda/timeline`

The server interprets **probe anchor**, **first-page visibility**, **local calendar-day span** (`fromDay` / `toDay` in the response), and **`nextCursor`** (`nextDay`) in the viewer's IANA zone using the same middleware fallback as tasks.

- Tasks use **due** for ordering (`CalendarItemDto.startDate` mirrors due)
- CRM events and externals use **start**
- Multi-day events that **overlap** the viewer's local "today" can appear on the first page even when `start` is before `max(from, now)`

## Timed Events â€” UTC Round-Trip

Timed events follow a straightforward UTC contract:

```typescript
// Task created
dueDate = "2026-04-22T10:00:00.000Z"   // stored as timestamptz in PostgreSQL

// Backend processing
startDate = dueDate
endDate = startDate + 1h

// API response
{
  startDate: "2026-04-22T10:00:00.000Z",
  endDate: "2026-04-22T11:00:00.000Z"
}

// Frontend display
// calendar grid converts to local time (e.g. 13:00 in UTC+3)
```

<Tip>
No special handling needed. JavaScript `new Date(isoString)` already produces a `Date` in local time. Any date library that accepts ISO 8601 strings will handle the offset automatically.
</Tip>

## All-Day Events â€” Floating Date Model

### CRM Tasks (source: 'task', hasDueTime: false)

Task **`dueDate`** for date-only dues is stored as **noon UTC** on the calendar day (`YYYY-MM-DDT12:00:00.000Z`) so the date prefix stays stable. The UI must treat the due **day** as the **`YYYY-MM-DD` prefix**, not the local calendar day of the instant alone.

<CodeGroup>
```typescript Backend (calendar.service.ts â†’ toCalendarTask)
// Extract the YYYY-MM-DD portion â€” no timezone shift
const iso = dueDate.toISOString().substring(0, 10); // e.g. "2026-04-22"
startDate = new Date(`${iso}T00:00:00.000Z`); // UTC midnight â€” a date label
endDate = new Date(`${iso}T23:59:59.999Z`); // UTC end â€” also a date label
```

```typescript Frontend (use-calendar-events.ts â†’ calendarTaskToEvent)
if (!item.hasDueTime && item.dueDate) {
  const [y, m, d] = item.dueDate.substring(0, 10).split('-').map(Number);
  startDate = new Date(y, m - 1, d, 0, 0, 0, 0).toISOString(); // local midnight
  endDate = new Date(y, m - 1, d, 23, 59, 59, 999).toISOString(); // local 23:59
}
```
</CodeGroup>

<Check>
`new Date(y, m-1, d, â€¦)` (multi-argument constructor) **always uses the browser's local timezone**, so the event appears on April 22 no matter what offset the user is in.
</Check>

### External Calendar Events (Google / Outlook)

External providers already follow the Google Calendar convention: all-day events arrive as date-only strings (`"2026-04-22"`, not ISO timestamps) and use an **exclusive end date** (April 22 all-day â†’ `endDate = "2026-04-23"`).

```typescript Frontend (use-calendar-events.ts â†’ calendarExternalItemToEvent)
if (item.isAllDay && item.startDate && item.endDate) {
  const [sy, sm, sd] = item.startDate.substring(0, 10).split('-').map(Number);
  startDate = new Date(sy, sm - 1, sd, 0, 0, 0, 0).toISOString();

  // Subtract one from end day (exclusive â†’ inclusive)
  const [ey, em, ed] = item.endDate.substring(0, 10).split('-').map(Number);
  endDate = new Date(ey, em - 1, ed - 1, 23, 59, 59, 999).toISOString();
}
```

The subtraction `ed - 1` converts the exclusive end to an inclusive one.

## Date Constructor Best Practices

<Warning>
**Never use `new Date("YYYY-MM-DD")` for all-day events**

`new Date("YYYY-MM-DD")` is parsed as **UTC midnight** per the ECMAScript spec.
In a UTC+3 timezone this becomes `2026-04-21T21:00:00` local time â€” the event shifts to the previous day.
</Warning>

<Tabs>
<Tab title="âťŚ Wrong">
```typescript
// Parsed as UTC midnight, shifts day in non-UTC timezones
new Date('2026-04-22');
```
</Tab>

<Tab title="âś… Correct">
```typescript
// Uses local timezone
new Date(2026, 3, 22); // month is 0-indexed
new Date(y, m - 1, d, 0, 0, 0, 0);
```
</Tab>
</Tabs>

## API Contract

### Query Parameters

| Parameter | Type | Description |
| --- | --- | --- |
| `from` | ISO 8601 UTC | Window start. Frontend sends `subMonths(now, 3).toISOString()` |
| `to` | ISO 8601 UTC | Window end. Frontend sends `addMonths(now, 6).toISOString()` |
| `userId` | UUID | Optional filter |
| `integrationIds` | string | Comma-separated CalendarIntegration IDs |

<Info>
**No `timeZone` parameter needed.** The backend is timezone-agnostic for all-day events. All date logic lives in the frontend.
</Info>

### Calendar Agenda Endpoint

**`GET /v1/calendar/agenda`** accepts an optional inclusive `from` parameter as UTC ISO, serving as a lower bound for returned merged rows. The web app passes **`startOfDay(selectedDate).toISOString()`** where `selectedDate` is parsed from `?date=yyyy-MM-dd` using `new Date(y, m-1, d)`.

Response format: **`{ items }`** only (same shapes as `GET /v1/calendar`) with **no cursor pagination**.

### Response Fields for CalendarTaskItemDto

| Field | Type | Notes |
| --- | --- | --- |
| `startDate` | ISO 8601 UTC | For `hasDueTime=true`: exact start. For `hasDueTime=false`: floating date label |
| `endDate` | ISO 8601 UTC | Same rules as `startDate` |
| `dueDate` | ISO 8601 UTC | Source of truth for date portion on all-day tasks |
| `hasDueTime` | boolean | `false` â†’ all-day event, frontend applies floating-date expansion |

<Warning>
For all-day tasks, **frontend must override** `startDate`/`endDate` with local expansion. Backend values are placeholders.
</Warning>

## Libraries & Tools

<Note>
`luxon` is **not used** in the calendar (`GET /v1/calendar`) pipeline. Task list/kanban stats use it only to derive **[todayStart, todayEnd)** instants for **timed** tasks.
</Note>

## Common Anti-Patterns

<AccordionGroup>
<Accordion title="âťŚ new Date('2026-04-22') for all-day display">
**Problem:** UTC parse causes wrong day in UTC+/- timezones

**Fix:** Use `new Date(y, m-1, d, â€¦)` multi-argument constructor
</Accordion>

<Accordion title="âťŚ date.setHours(0, 0, 0) on the server">
**Problem:** Applies server timezone (almost never UTC)

**Fix:** Backend emits UTC date-label; frontend expands locally
</Accordion>

<Accordion title="âťŚ Sending timeZone in API query">
**Problem:** Couples backend to client clock; server has no reliable fallback

**Fix:** All-day logic lives in the frontend only
</Accordion>

<Accordion title="âťŚ toLocaleDateString() for ISO output">
**Problem:** Locale-dependent, non-deterministic formatting

**Fix:** Use `.toISOString()` after local construction
</Accordion>

<Accordion title="âťŚ Using startDate/endDate directly for all-day tasks">
**Problem:** Backend values are UTC placeholders, not local expansions

**Fix:** Extract date from `dueDate` and rebuild with local constructor
</Accordion>
</AccordionGroup>
