<Note>
**TL;DR** â€” The API is pure UTC. The frontend converts to local time for display. All-day / date-only events use the **floating-date model** (same as Google Calendar): the backend sends a neutral date label; the frontend expands it to local midnight â†’ 23:59 using `new Date(y, m-1, d, â€¦)`.
</Note>

## Core Principle

PropWise handles two fundamentally different types of temporal data:

| Event type                             | Backend contract                                     | Frontend display                                        |
| -------------------------------------- | ---------------------------------------------------- | ------------------------------------------------------- |
| **Timed event** (`hasDueTime=true`)    | UTC ISO string (`2026-04-22T10:00:00.000Z`)          | Convert to local timezone using `Intl` / `date-fns-tz`  |
| **All-day event** (`hasDueTime=false`) | UTC midnight date-label (`2026-04-22T00:00:00.000Z`) | Extract date portion â†’ expand to local midnight â†’ 23:59 |

<Info>
An all-day event is a **calendar date**, not a UTC instant. "April 22" means April 22 regardless of which timezone the viewer is in. Storing `2026-04-22T00:00:00.000Z` and letting the frontend expand it locally is the same strategy Google Calendar uses (`start.date` / `end.date` fields).
</Info>

## HTTP Time-Zone Header

### Task List & Kanban Stats

The CRM web app attaches `Time-Zone` header on authenticated requests using `Intl.DateTimeFormat().resolvedOptions().timeZone`. Nest applies `TimezoneMiddleware` globally:

<Steps>

<Step title="Validate timezone">
The header is validated against known IANA zones using `Intl.supportedValuesOf('timeZone')`.
</Step>

<Step title="Apply fallback">
Missing or invalid values fall back to `UTC` (no HTTP 400 error).
</Step>

<Step title="Set request context">
The middleware sets `req.timezone` for use by `@TimezoneParam()` decorator.
</Step>

</Steps>

**Affected endpoints:**
- `GET /tasks/list`
- `GET /tasks/kanban`

#### Behavior for Timed Tasks

For **`hasDueTime: true`**, `TaskService` uses Luxon to compute **[start, next midnight)** of "today" in the viewer's zone as UTC instants and compares `dueDate` (`timestamptz`) to that window.

#### Behavior for Date-Only Tasks

For **`hasDueTime: false`** (date-only dues), buckets compare:
- **`(due_date AT TIME ZONE 'UTC')::date`** (semantic calendar day, aligned with noon-UTC storage)
- **`(timezone(<viewer_tz>, now()))::date`**

This ensures a task shown as "30 Apr" cannot land in "today" when the viewer's local day is already "1 May".

<Warning>
JSON responses are unchanged â€” task `dueDate` fields remain UTC ISO strings. The header affects **grouping and counts**, not field serialization.
</Warning>

#### List Sort Behavior

When using `sort=dueDate` (or default), within each bucket (overdue / today / upcoming / completed):

1. **Primary sort**: Calendar due day (local calendar date from `due_date` when `hasDueTime`, else `(due_date AT TIME ZONE 'UTC')::date`)
2. **Secondary sort**: `due_date` instant for timed tasks
3. **Tertiary sort**: Date-only rows (`hasDueTime: false`) appear **after** all timed tasks on that calendar day
4. **Final sort**: `title`

<Note>
Requires `Time-Zone` header so timed dues sort by the viewer's local calendar date.
</Note>

### Personal Agenda Timeline

**Endpoint:** `GET /v1/calendar/me/agenda/timeline`

The server interprets:
- Probe anchor
- First-page visibility
- Local calendar-day span (`fromDay` / `toDay` in the response)
- `nextCursor` (`nextDay`)

All interpretations use the viewer's IANA zone (same middleware fallback as tasks).

<AccordionGroup>

<Accordion title="Ordering rules">
- **Tasks**: Use `due` for ordering (`CalendarItemDto.startDate` mirrors due)
- **CRM events and externals**: Use `start`
</Accordion>

<Accordion title="Multi-day event handling">
Multi-day CRM events and externals that **overlap** the viewer's local "today" can appear on the first page even when `start` is before `max(from, now)`.
</Accordion>

<Accordion title="Response structure">
Timeline rows include optional **`agendaDayKeys`** clipped to the response window. The frontend agenda list groups by those keys and clamps to `window.toDay` instead of re-grouping from browser-parsed `startDate` / `endDate`.
</Accordion>

</AccordionGroup>

## Timed Events â€” UTC Round-Trip

Timed events use straightforward UTC storage and conversion:

```typescript
// Task created
dueDate = "2026-04-22T10:00:00.000Z"   // stored as timestamptz in PG

// Backend
startDate = dueDate
endDate = startDate + 1h

// API response
{
  startDate: "2026-04-22T10:00:00.000Z",
  endDate: "2026-04-22T11:00:00.000Z"
}

// Frontend
// Calendar grid converts to local time (e.g. 13:00 in UTC+3)
```

<Check>
**No special handling needed.** JavaScript `new Date(isoString)` already produces a `Date` in local time. Any date library that accepts ISO 8601 strings will handle the offset automatically.
</Check>

## All-Day Events â€” Floating Date Model

### CRM Tasks

Task `dueDate` for date-only dues is stored as **noon UTC** on the calendar day (`YYYY-MM-DDT12:00:00.000Z`) so the date prefix stays stable.

<Warning>
The UI must treat the due **day** as the **`YYYY-MM-DD` prefix** (see task detail panel / `parseTaskDueAsLocalDate`), not the local calendar day of the instant alone.
</Warning>

<CodeGroup>

```typescript Backend (calendar.service.ts â†’ toCalendarTask)
// Extract the YYYY-MM-DD portion â€” no timezone shift
const iso = dueDate.toISOString().substring(0, 10); // e.g. "2026-04-22"
startDate = new Date(`${iso}T00:00:00.000Z`); // UTC midnight â€” a date label
endDate = new Date(`${iso}T23:59:59.999Z`); // UTC end â€” also a date label
```

```typescript Frontend (use-calendar-events.ts â†’ calendarTaskToEvent)
if (!item.hasDueTime && item.dueDate) {
  const [y, m, d] = item.dueDate.substring(0, 10).split('-').map(Number);
  startDate = new Date(y, m - 1, d, 0, 0, 0, 0).toISOString(); // local midnight
  endDate = new Date(y, m - 1, d, 23, 59, 59, 999).toISOString(); // local 23:59
}
```

</CodeGroup>

<Info>
`new Date(y, m-1, d, â€¦)` (multi-argument constructor) **always uses the browser's local timezone**, so the event appears on April 22 no matter what offset the user is in.
</Info>

The backend also returns:
- `dueDate` as a raw ISO string
- `hasDueTime: false`
- `isAllDay: !item.hasDueTime` is set on the returned `ICalendarEvent` so the calendar UI can render the event in the all-day row instead of the time grid

### External Calendar Events (Google / Outlook)

External providers already follow the Google Calendar convention:
- All-day events arrive as date-only strings (`"2026-04-22"`, not ISO timestamps)
- Use an **exclusive end date** (April 22 all-day â†’ `endDate = "2026-04-23"`)

```typescript Frontend (use-calendar-events.ts â†’ calendarExternalItemToEvent)
if (item.isAllDay && item.startDate && item.endDate) {
  const [sy, sm, sd] = item.startDate.substring(0, 10).split('-').map(Number);
  startDate = new Date(sy, sm - 1, sd, 0, 0, 0, 0).toISOString();

  // Subtract one from end day (exclusive â†’ inclusive)
  const [ey, em, ed] = item.endDate.substring(0, 10).split('-').map(Number);
  endDate = new Date(ey, em - 1, ed - 1, 23, 59, 59, 999).toISOString();
}
```

<Note>
The subtraction `ed - 1` converts the exclusive end to an inclusive one.
</Note>

## Date Constructor Pitfalls

<Warning>
**Never use `new Date("YYYY-MM-DD")` for all-day events**

`new Date("YYYY-MM-DD")` is parsed as **UTC midnight** per the ECMAScript spec. In a UTC+3 timezone this becomes `2026-04-21T21:00:00` local time â€” the event shifts to the previous day.
</Warning>

<CodeGroup>

```typescript âťŚ Wrong
// Parsed as UTC midnight, shifts day in non-UTC timezones
new Date('2026-04-22');
```

```typescript âś… Correct
// Uses local timezone
new Date(2026, 3, 22); // month is 0-indexed
new Date(y, m - 1, d, 0, 0, 0, 0);
```

</CodeGroup>

## API Contract

### Query Parameters

| Param            | Type         | Notes                                                           |
| ---------------- | ------------ | --------------------------------------------------------------- |
| `from`           | ISO 8601 UTC | Window start. Frontend sends `subMonths(now, 3).toISOString()`. |
| `to`             | ISO 8601 UTC | Window end. Frontend sends `addMonths(now, 6).toISOString()`.   |
| `userId`         | UUID         | Optional filter.                                                |
| `integrationIds` | string       | Comma-separated CalendarIntegration IDs.                        |

### Agenda Endpoint

`GET /v1/calendar/agenda` accepts optional inclusive `from` as UTC ISO: lower bound for which merged rows are returned.

The web app passes **`startOfDay(selectedDate).toISOString()`** where:
- `selectedDate` is the viewer's calendar day from `?date=yyyy-MM-dd`
- Parsed with `new Date(y, m-1, d)`, same local-date rule as above

**Response format:**
- Returns **`{ items }`** only (same shapes as `GET /v1/calendar`)
- **No cursor pagination** â€” the backend scans forward up to an internal cap and returns every item in range

<Info>
**No `timeZone` parameter.** The backend is timezone-agnostic for all-day events. All date logic lives in the frontend.
</Info>

### Response Fields for CalendarTaskItemDto

| Field        | Type         | Notes                                                                                                                              |
| ------------ | ------------ | ---------------------------------------------------------------------------------------------------------------------------------- |
| `startDate`  | ISO 8601 UTC | For `hasDueTime=true`: exact start. For `hasDueTime=false`: floating date label â€” **frontend must override with local expansion**. |
| `endDate`    | ISO 8601 UTC | Same rules as `startDate`.                                                                                                         |
| `dueDate`    | ISO 8601 UTC | Source of truth for the date portion on all-day tasks. Frontend reads `substring(0, 10)` to extract `YYYY-MM-DD`.                  |
| `hasDueTime` | boolean      | `false` â†’ all-day event, frontend applies floating-date expansion.                                                                 |

## Anti-Patterns

<AccordionGroup>

<Accordion title="âťŚ Using new Date('2026-04-22') for all-day display">
**Why it breaks:** UTC parse â†’ wrong day in UTC+/-

**Fix:** Use `new Date(y, m-1, d, â€¦)`
</Accordion>

<Accordion title="âťŚ Using date.setHours(0, 0, 0) on the server">
**Why it breaks:** Applies server timezone (almost never UTC)

**Fix:** Backend emits UTC date-label; frontend expands locally
</Accordion>

<Accordion title="âťŚ Sending timeZone in API query">
**Why it breaks:** Couples backend to client clock; server has no reliable fallback

**Fix:** All-day logic lives in the frontend only
</Accordion>

<Accordion title="âťŚ Using toLocaleDateString() for ISO output">
**Why it breaks:** Locale-dependent format; not machine-readable

**Fix:** Use `toISOString()` or `date-fns` `format(date, 'yyyy-MM-dd')`
</Accordion>

</AccordionGroup>
