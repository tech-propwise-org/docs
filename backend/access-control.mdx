## Overview

The access control and permissions system has **two layers** that work together:

| Layer | Purpose | When it runs | Result |
|-------|---------|-------------|--------|
| **Endpoint Protection** | Decides if a request is allowed | Before controller handler (NestJS guard) | 403 Forbidden or allow through |
| **Response Permissions** | Tells the frontend what the user can do | Inside the service, after data is fetched | `permissions` object on each resource |

```
Request → AuthGuard → AccessGuard (Layer 1: allow or 403)
  → Controller → Service → PermissionComputeService (Layer 2: attach permissions)
    → Response with { data, permissions }
```

---

# Layer 1: Endpoint Protection

This layer decides **whether a request is allowed to proceed**. It runs as a NestJS guard before the controller handler. If the user lacks access, the request is rejected with a 403.

## Core concepts

### Access check types

| Type                    | Description                                 | Use case                          |
| ----------------------- | ------------------------------------------- | --------------------------------- |
| **Permission Check**    | RBAC permission (e.g., `crm.manage`)        | List endpoints, create operations |
| **Resource Check**      | Stakeholder-based access to specific entity | Single-entity operations          |
| **Bulk Resource Check** | Batch access check for multiple entities    | Bulk operations                   |

### Access modes

| Mode    | Behavior                                               | Default |
| ------- | ------------------------------------------------------ | ------- |
| **OR**  | User passes if they have permission OR resource access | Yes     |
| **AND** | User must have BOTH permission AND resource access     | No      |

### Resource types

```typescript
enum ResourceType {
  LEAD = 'LEAD',
  DEAL = 'DEAL',
  CONTACT = 'CONTACT',
  COMPANY = 'COMPANY',
  PERSON = 'PERSON',
  UNIT = 'UNIT',
  TEAM = 'TEAM',
}
```

### Resource actions

```typescript
enum ResourceAction {
  ACCESS = 'ACCESS', // Can view
  MODIFY = 'MODIFY', // Can update
  DELETE = 'DELETE', // Can delete
}
```

## Single-resource endpoints

### @CheckAccess decorator

The `@CheckAccess` decorator combines permission and resource checks:

```typescript
@CheckAccess({
  permissions: string[],           // RBAC permissions (any match = pass)
  resource?: {
    type: ResourceType,            // Entity type
    action: ResourceAction,        // ACCESS, MODIFY, DELETE
    paramName?: string,            // Route param name (default: 'id')
  },
  mode?: AccessMode,               // OR (default) or AND
})
```

### Usage patterns

<Tabs>
<Tab title="Permission only">
Used for list and create endpoints where no specific resource ID is involved.

```typescript
@Get()
@CheckAccess({ permissions: ['crm.view', 'crm.manage'] })
async findAll() {}

@Post()
@CheckAccess({ permissions: ['crm.create', 'crm.manage'] })
async create(@Body() dto: CreateLeadDto) {}
```
</Tab>

<Tab title="Permission OR resource">
Used for view, update, and delete endpoints. The user passes if they have the permission **or** stakeholder access to the resource.

```typescript
@Get(':id')
@CheckAccess({
  permissions: ['crm.view', 'crm.manage'],
  resource: { type: ResourceType.LEAD, action: ResourceAction.ACCESS },
})
async findOne(@Param('id') id: string) {}

@Put(':id')
@CheckAccess({
  permissions: ['crm.manage'],
  resource: { type: ResourceType.LEAD, action: ResourceAction.MODIFY },
})
async update(@Param('id') id: string, @Body() dto: UpdateLeadDto) {}

@Delete(':id')
@CheckAccess({
  permissions: ['crm.manage'],
  resource: { type: ResourceType.LEAD, action: ResourceAction.DELETE },
})
async remove(@Param('id') id: string) {}
```
</Tab>

<Tab title="Permission AND resource">
Used for sensitive operations where the user must have **both** the permission and stakeholder access.

```typescript
@Put(':id/transfer')
@CheckAccess({
  permissions: ['crm.transfer'],
  resource: { type: ResourceType.LEAD, action: ResourceAction.MODIFY },
  mode: AccessMode.AND,
})
async transfer(@Param('id') id: string, @Body() dto: TransferDto) {}
```
</Tab>

<Tab title="Resource only">
Used when only stakeholder access matters, with no RBAC permission required.

```typescript
@Get(':id')
@CheckAccess({
  resource: { type: ResourceType.LEAD, action: ResourceAction.ACCESS },

```
</Tab>

<Tab title="Custom param">
When the route parameter is not `:id`, specify `paramName`.

### Permission-only with team permissions (stakeholder management)

For endpoints where resource-level access should NOT grant permission (e.g., managing stakeholders on an entity), use permission-only checks that include both org-level and team-level management permissions:

```typescript
// Stakeholder management on leads - permission-only, NO resource check
@Post(':id/stakeholders')
@CheckAccess({
  permissions: [OrgPermissionKey.CRM_MANAGE, TeamPermissionKey.TEAM_CRM_MANAGE],
})
async createStakeholder() {}

// Stakeholder management on deals - permission-only, NO resource check
@Post(':id/stakeholders')
@CheckAccess({
  permissions: [OrgPermissionKey.SALES_MANAGE, TeamPermissionKey.TEAM_SALES_MANAGE],
})
async createStakeholder() {}
```

<Note>
Having `accessLevel: MODIFY` on an entity (via being a stakeholder) should not grant permission to add, remove, or modify other stakeholders. Stakeholder management is an administrative function that requires explicit RBAC permissions.
</Note>

**Two-tier access for team managers:**

1. **Guard level:** `TEAM_CRM_MANAGE` / `TEAM_SALES_MANAGE` passes the guard (user has this permission on at least one team).
2. **Controller level:** Business logic further validates:
   - Team has a stakeholder on the specific entity (enforced on `createStakeholder`, `batchUpdate`, `removeStakeholder`)
   - Updates/adjustments only target the team's own stakeholders or team member stakeholders (enforced on `createStakeholder`, `batchUpdate`)
   - Removals only target the team's own stakeholders (enforced on `removeStakeholder`, `batchUpdate` via `removeStakeholderId`)
   - Budget constraints for commission allocation (`createStakeholder`)
   - Primary restriction: cannot set primary unless a team stakeholder already holds primary (`createStakeholder`, `batchUpdate`)

See [Stakeholder API](/backend/stakeholder-api) for full details on team manager constraints.

### Team-scoped permission check (`teamCheck`)

For endpoints that should be accessible to team admins/members within a specific team context, use `teamCheck`. The `permissions` field acts as an org-level bypass:

```typescript
@Get()
@CheckAccess({
  permissions: [OrgPermissionKey.SYSTEM_ADMIN, OrgPermissionKey.CRM_VIEW, OrgPermissionKey.CRM_MANAGE],
  teamCheck: {
    teamPermissions: [TeamPermissionKey.TEAM_ADMIN, TeamPermissionKey.TEAM_CRM_MANAGE],
    teamIdParam: 'teamId',
  },
})
async findAll(@Query('teamId') teamId?: string) {}
```

**Flow:**

1. Check org-level bypass permissions — if user has any bypass permission (e.g. `system.admin`, `crm.view`, `crm.manage`), access is granted immediately
2. Extract `teamId` from the request using `teamIdParam`:
   - **If `teamId` present** — check if user has any of `teamPermissions` in that specific team
   - **If `teamId` absent** — check if user has any of `teamPermissions` in ANY of their teams (any-team fallback). This allows team leaders to access endpoints like user selection dropdowns without specifying a team.
3. If neither bypass nor team check passes — throw 403

### Open directory endpoint (`GET /users/directory`)

For UI surfaces where any authenticated org member needs to pick a user (e.g. transfer modal, mentions), use the dedicated directory endpoint instead of `GET /users`. It returns only `id`, `firstName`, `lastName`, and `avatarUrl` — no sensitive data — and requires no additional permission beyond being logged in.

```typescript
@Get('directory')
async findDirectory(@GetCurrentUser() user: RequestUser): Promise<PaginatedResponse<UserDirectoryDto>> {
  return this.userService.findDirectory(user.organizationId, ...);
}
```

<Tip>
Use `UserApi.getDirectory()` on the frontend whenever you need a permission-free user picker.
</Tip>

<Warning>
When `teamCheck` grants access via team permissions (not org bypass), the **controller must scope the returned data to the team**. The guard only authorizes the request — it does NOT filter results.
</Warning>

```typescript
const effectiveTeamIds = teamIds
  ? teamIds.split(',').filter(Boolean)
  : teamId
    ? [teamId]
    : undefined;

return this.userService.findAllPaginated({ teamIds: effectiveTeamIds, ... }, orgId);
```

## Bulk-resource endpoints

Bulk endpoints check per-item access in a batch-optimized manner with partial success support. Users with org-level bypass permissions (e.g., `CRM_MANAGE`) can operate on all resources; others are checked per-item via stakeholder access.

### Bulk resource configuration

```typescript
@CheckAccess({
  permissions: [OrgPermissionKey.CRM_MANAGE],
  bulkResource: {
    type: ResourceType.LEAD,
    action: ResourceAction.MODIFY,
    paramName: 'leadIds',
    strategy: 'partial',
  },
})
```

### Strategy options

| Strategy  | Behavior                                                       | Use case                         |
| --------- | -------------------------------------------------------------- | -------------------------------- |
| `partial` | Allow operation, filter inaccessible items                     | Archive, unarchive (recommended) |
| `all`     | Require access to ALL items, reject entire request if any fail | Critical operations              |

### Bulk endpoint flow

```
User Request → AccessGuard
                    ↓
        Check bypass permissions?
                    ↓
         ┌──────────┴──────────┐
         │                     │
      YES: Allow all        NO: Batch check
         │                     │
         └─────────┬───────────┘
                   ↓
        Set bulkAccessContext on request
                   ↓
           Controller extracts context
                   ↓
           Service filters items
                   ↓
        Return partial results
```

### Controller implementation

```typescript
@Post('bulk-archive')
@CheckAccess({
  permissions: [OrgPermissionKey.CRM_MANAGE],
  bulkResource: {
    type: ResourceType.LEAD,
    action: ResourceAction.MODIFY,
    paramName: 'leadIds',
    strategy: 'partial',
  },
})
async bulkArchive(
  @Req() request: any,
  @Body() body: { leadIds: string[] },
  @GetCurrentUser() user: RequestUser,
) {
  const bulkAccessContext = request.bulkAccessContext;
  return this.service.bulkArchiveLeads(
    body.leadIds,
    user.id,
    user.organizationId!,
    bulkAccessContext,
  );
}
```

### Service implementation

```typescript
async bulkArchiveLeads(
  leadIds: string[],
  userId: string,
  organizationId: string,
  bulkAccessContext?: BulkAccessContext,
): Promise<{
  success: boolean;
  archivedCount: number;
  failedCount: number;
  failures: Array<{ leadId: string; reason: string }>;
}> {
  const results = {
    success: true,
    archivedCount: 0,
    failedCount: 0,
    failures: [] as Array<{ leadId: string; reason: string }>,
  };

  // Pre-filter inaccessible items
  if (bulkAccessContext?.accessMap && !bulkAccessContext.hasBypassPermission) {
    for (const leadId of leadIds) {
      if (bulkAccessContext.accessMap.get(leadId) === false) {
        results.failures.push({
          leadId,
          reason: 'Access denied - insufficient permissions',
        });
        results.failedCount++;
      }
    }
  }

  // Process only accessible items
  for (const leadId of leadIds) {
    if (results.failures.some(f => f.leadId === leadId)) continue;

    try {
      await this.archiveLead(leadId, userId, organizationId);
      results.archivedCount++;
    } catch (error) {
      results.failures.push({ leadId, reason: error.message });
      results.failedCount++;
    }
  }

  return results;
}
```

### BulkAccessContext type

```typescript
interface BulkAccessContext {
  hasBypassPermission: boolean;         // User has org-level permission
  resourceIds: string[];                // Original array of resource IDs from request
  accessMap: Map<string, boolean> | null; // resourceId → hasAccess (null if bypass)
}
```

## Access rules by resource type

<Note>
**Dual-record note:** A user may have multiple stakeholder rows on the same entity (personal + team-scoped). The access system resolves the **highest** `accessLevel` across all rows. For example, a personal stake with READ and a team-scoped stake with FULL resolves to FULL. Single-resource checks use `em.find()` + highest-level resolution; bulk checks already iterate all rows.
</Note>

### Leads

| Action | Access granted if                                                                |
| ------ | -------------------------------------------------------------------------------- |
| ACCESS | User is stakeholder, creator, lead in pool, or team member                       |
| MODIFY | User is stakeholder (WRITE/FULL), creator, team leader, or has `team_crm.manage` |
| DELETE | User is creator, stakeholder (FULL), team leader, or has `team_crm.manage`       |

### Deals

| Action | Access granted if                                                                                                              |
| ------ | ------------------------------------------------------------------------------------------------------------------------------ |
| ACCESS | User is stakeholder, creator, stakeholder of originating lead, or team member                                                  |
| MODIFY | User is stakeholder (WRITE/FULL), creator, team leader, has `team_sales.manage`, or originating lead stakeholder (WRITE/FULL)  |
| DELETE | User is creator, stakeholder (FULL), team leader, or has `team_sales.manage`                                                   |

### Contacts (derived access)

| Action | Access granted if                                                                |
| ------ | -------------------------------------------------------------------------------- |
| ACCESS | User has READ+ access to ANY lead/deal for this contact's person                 |
| MODIFY | User has WRITE+ access to ANY lead/deal for this contact's person, or is creator |
| DELETE | User has FULL access to ANY lead/deal for this contact's person, or is creator   |

### Companies (derived access)

| Action | Access granted if                                                    |
| ------ | -------------------------------------------------------------------- |
| ACCESS | User has READ+ access to ANY contact at this company                 |
| MODIFY | User has WRITE+ access to ANY contact at this company, or is creator |
| DELETE | User has FULL access to ANY contact at this company, or is creator   |

### Persons (derived access)

| Action | Access granted if                                                      |
| ------ | ---------------------------------------------------------------------- |
| ACCESS | User has READ+ access to ANY lead/deal for this person                 |
| MODIFY | User has WRITE+ access to ANY lead/deal for this person, or is creator |
| DELETE | User has FULL access to ANY lead/deal for this person, or is creator   |

### Units

| Action | Access granted if    |
| ------ | -------------------- |
| ACCESS | User is unit manager |
| MODIFY | User is unit manager |
| DELETE | User is unit manager |

### Teams

| Action | Access granted if                                              |
| ------ | -------------------------------------------------------------- |
| ACCESS | User is an active member of the team (any role)                |
| MODIFY | Not supported via resource check — use `teamCheck` instead     |
| DELETE | Not supported via resource check — use `teamCheck` instead     |

## Lead creation assignment authorization

The `POST /leads` endpoint has **no `@CheckAccess` decorator** — any authenticated user can create leads. However, **server-side validation** in `LeadService.validateLeadAssignmentAuthorization()` enforces assignment rules based on who the lead is assigned to and which team it's assigned to.

### Authorization matrix

| Scenario                                          | Required permission                         | Notes                                                 |
| ------------------------------------------------- | ------------------------------------------- | ----------------------------------------------------- |
| **Self-assignment** (`assignedUserId === userId`) | None                                        | Always allowed for any user                           |
| **Pool lead** (no `assignedUserId`) — Global Pool | `crm.manage`                                | Only org admins can create global pool leads          |
| **Pool lead** (no `assignedUserId`) — Team Pool   | `team_crm.manage` in target team            | Team managers can create pool leads for their team    |
| **Assign to another user**                        | `crm.manage`                                | Only org admins can assign to others                  |
| **Team validation** (if `teamId` provided)        | User must be member of team OR `crm.manage` | Users can only assign to their own teams unless admin |

### Rules in detail

<AccordionGroup>
<Accordion title="Rule 1 — Self-assignment (always allowed)">
```typescript
if (assignedUserId === user.id) {
  // If team specified, validate user is member of that team (unless crm.manage)
  if (teamId && !hasCrmManage) {
    checkTeamMembership(user.id, teamId); // Must be member
  }
  return; // Authorized
}
```
</Accordion>

<Accordion title="Rule 2 — Pool lead (no assignedUserId)">
```typescript
if (!assignedUserId) {
  if (hasCrmManage) return; // crm.manage → global or team pool allowed

  if (teamId) {
    // Check team_crm.manage in specific team
    if (hasTeamPermission(user.id, teamId, 'team_crm.manage')) {
      return; // Authorized for team pool
    }
    throw ForbiddenException('Need team_crm.manage in this team');
  }

  // No team → trying to create global pool lead without crm.manage
  throw ForbiddenException('Need crm.manage for global pool leads');
}
```
</Accordion>

<Accordion title="Rule 3 — Assign to another user">
```typescript
if (assignedUserId !== user.id) {
  if (!hasCrmManage) {
    throw ForbiddenException('Only crm.manage users can assign to others');
  }
  // crm.manage can assign to anyone, validate team exists if specified
  return; // Authorized
}
```
</Accordion>
</AccordionGroup>

### Frontend assignment behavior

The frontend enforces these rules with dynamic UI behavior:

| User type                                 | Assigned To field                             | Team selection           | Pool lead?             |
| ----------------------------------------- | --------------------------------------------- | ------------------------ | ---------------------- |
| **Normal agent** (no permissions)         | Self (locked)                                 | Optional (my teams)      | NO                     |
| **User with `team_crm.manage` in Team X** | Self (default, unlocked when Team X selected) | Optional (my teams)      | YES (only Team X pool) |
| **User with `crm.manage`**                | Anyone (open)                                 | Any team (all org teams) | YES (any pool)         |

**Dynamic enable/disable logic:**

- Normal agent: "Assigned To" is disabled, pre-filled with self
- User selects a team where they have `team_crm.manage` → "Assigned To" becomes enabled (can clear to send to pool)
- User selects a team where they DON'T have `team_crm.manage` → "Assigned To" stays disabled, auto-set to self

## Claiming leads from team pool

**Endpoint:** `POST /leads/:id/stakeholders`

### Authorization matrix for claiming

| User type                                              | Can claim? | Restrictions                                                         |

# Layer 2: Response Permissions

This layer computes **what actions the current user can perform** on each resource and attaches that information to the API response. The frontend reads these permissions directly for UI decisions (show/hide buttons, enable/disable actions) without duplicating access logic.

---

## Which entities have response permissions

### Entities WITH response permissions

These are **CRM business entities** with complex access rules involving stakeholders, teams, and creators:

| Entity | Has `permissions`? | Has `assignedTeam`? | Reason |
|--------|-------------------|---------------------|--------|
| **Lead** | Yes | Yes | Has EntityStakeholder, team assignment, creator access |
| **Deal** | Yes | Yes | Has EntityStakeholder, team assignment, creator access |
| **Contact** | Yes | No | Derived access from related Leads/Deals |
| **Company** | Yes | No | Derived access from related Contacts/Deals |

### Entities WITHOUT response permissions

These are **configuration/admin entities** with simple permission-based access:

| Entity | Why no response permissions |
|--------|----------------------------|
| User | Admin-only access (`system.admin`) |
| Team | Admin or team membership access |
| Role | Admin-only access (`system.admin`, `system.owner`) |
| Commission Payment | Recipient-based or `sales.manage` permission |
| Entity Transfer | Sender/recipient-based access |
| Invitation | Admin-only access |
| Activity | Read-only timeline data |
| Stage History | Read-only audit data |

## PermissionComputeService

**Location:** `src/modules/auth/services/permission-compute.service.ts`

This service computes permissions using the existing `AccessService` methods:

```typescript
@Injectable()
export class PermissionComputeService {
  constructor(private readonly accessService: AccessService) {}

  // Single resource methods
  async computeLeadPermissionsInTransaction(em, user, leadId): Promise<ResourcePermissionsDto>;
  async computeDealPermissionsInTransaction(em, user, dealId): Promise<ResourcePermissionsDto>;
  async computeContactPermissionsInTransaction(em, user, contactId): Promise<ResourcePermissionsDto>;
  async computeCompanyPermissionsInTransaction(em, user, companyId): Promise<ResourcePermissionsDto>;

  // Optimized bulk methods (used by services for list views)
  async computeAllBulkLeadPermissionsWithTeamsInTransaction(
    em, user, leadIds, preloadedLeads?
  ): Promise<{ permissions: Map<string, ResourcePermissionsDto> }>;

  async computeAllBulkDealPermissionsWithTeamsInTransaction(
    em, user, dealIds, preloadedDeals?
  ): Promise<{ permissions: Map<string, ResourcePermissionsDto> }>;

  // Standard bulk methods (3 queries per action type)
  async computeBulkDealPermissionsInTransaction(em, user, dealIds): Promise<Map<string, ResourcePermissionsDto>>;
  async computeBulkContactPermissionsInTransaction(em, user, contactIds): Promise<Map<string, ResourcePermissionsDto>>;
  async computeBulkCompanyPermissionsInTransaction(em, user, companyIds): Promise<Map<string, ResourcePermissionsDto>>;

  // Team assignment helpers
  async getLeadAssignedTeamInTransaction(em, leadId): Promise<TeamBasicDto | null>;
  async getBulkLeadAssignedTeamsInTransaction(em, leadIds): Promise<Map<string, TeamBasicDto>>;
  async getDealAssignedTeamInTransaction(em, dealId): Promise<TeamBasicDto | null>;
  async getBulkDealAssignedTeamsInTransaction(em, dealIds): Promise<Map<string, TeamBasicDto>>;
}
```

**Permission derivation rules:**

- `canAssign` = `canView && canManageStakeholders(user)` where `canManageStakeholders` checks for `crm.manage`, `system.admin`, `system.owner`, or `team_crm.manage`
- `canTransfer` = `canEdit`
- `canArchive` = `canDelete`
- `canUnarchive` = `canDelete`
- For Contacts/Companies: no `canAssign` or `canTransfer` (these are derived-access entities)

```typescript
private async enrichLeadsWithPermissions(
  em: EntityManager,

  leads: Lead[],

  if (!leads || leads.length === 0) return [];

  const leadIds = leads.map((lead) => lead.id);

  const { permissions: permissionsMap } =
```

## Complete controller example

```typescript
import { Controller, Get, Post, Put, Delete, Param, Body, Req, UseGuards } from '@nestjs/common';
import { AuthGuard } from '@/modules/auth/auth.guard';
import { AccessGuard } from '@/modules/auth/guards/access.guard';
import { CheckAccess, AccessMode, ResourceType, ResourceAction } from '@/modules/auth/decorators';
import { GetCurrentUser } from '@/modules/auth/decorators/get-current-user.decorator';
import { OrgPermissionKey } from '@/modules/rbac/permission/permission.constants';

@Controller('leads')
@UseGuards(AuthGuard, AccessGuard)
export class LeadController {
  // === List & Create (Permission Only) ===

  @Get()
  @CheckAccess({ permissions: ['crm.view', 'crm.manage'] })
  async findAll() {}

  @Post()
  @CheckAccess({ permissions: ['crm.create', 'crm.manage'] })
  async create(@Body() dto: CreateLeadDto) {}

  // === Single Resource (Permission OR Resource) ===

  @Get(':id')
  @CheckAccess({
    permissions: ['crm.view', 'crm.manage'],
    resource: { type: ResourceType.LEAD, action: ResourceAction.ACCESS },
  })
  async findOne(@Param('id') id: string) {}

  @Put(':id')
  @CheckAccess({
    permissions: ['crm.manage'],
    resource: { type: ResourceType.LEAD, action: ResourceAction.MODIFY },
  })
  async update(@Param('id') id: string, @Body() dto: UpdateLeadDto) {}

  @Delete(':id')
  @CheckAccess({
    permissions: ['crm.manage'],
    resource: { type: ResourceType.LEAD, action: ResourceAction.DELETE },
  })
  async remove(@Param('id') id: string) {}

  // === Sensitive Operation (Permission AND Resource) ===

  @Put(':id/transfer')
  @CheckAccess({
    permissions: ['crm.transfer'],
    resource: { type: ResourceType.LEAD, action: ResourceAction.MODIFY },
    mode: AccessMode.AND,
  })
  async transfer(@Param('id') id: string, @Body() dto: TransferDto) {}

  // === Bulk Operations ===

  @Post('bulk-archive')
  @CheckAccess({
    permissions: [OrgPermissionKey.CRM_MANAGE],
    bulkResource: {
      type: ResourceType.LEAD,
      action: ResourceAction.MODIFY,
      paramName: 'leadIds',
      strategy: 'partial',
    },
  })
  async bulkArchive(
    @Req() request: any,
    @Body() body: { leadIds: string[] },
    @GetCurrentUser() user: RequestUser,
  ) {
    const bulkAccessContext = request.bulkAccessContext;
    return this.leadService.bulkArchiveLeads(
      body.leadIds,
      user.id,
      user.organizationId!,
      bulkAccessContext,
    );
  }

  @Post('bulk-delete')
  @CheckAccess({
    permissions: [OrgPermissionKey.CRM_MANAGE],
    bulkResource: {
      type: ResourceType.LEAD,
      action: ResourceAction.DELETE,
      paramName: 'leadIds',
      strategy: 'partial',
    },
  })
  async bulkDelete(
    @Req() request: any,
    @Body() body: { leadIds: string[] },
    @GetCurrentUser() user: RequestUser,
  ) {
    const bulkAccessContext = request.bulkAccessContext;
    return this.leadService.bulkDeleteLeads(
      body.leadIds,
      user.id,
      user.organizationId!,
      bulkAccessContext,
    );
  }
}
```

## Performance considerations

### Layer 1: Endpoint protection

**Single-resource checks:**
- Uses indexed lookups on `entity_id`, `user_id`, `team_id`
- Single query per resource check

**Bulk resource checks:**
- **Batch queries**: All access checks use `$in` clauses
- **Single transaction**: Batch access check runs in one `executeReadOnly` transaction
- **In-memory processing**: After batch fetch, access logic runs in-memory
- **Typical query count**: ~4 queries regardless of N resources

| Strategy  | Max items per request |
| --------- | --------------------- |
| `partial` | 100                   |
| `all`     | 50                    |

### Layer 2: Response permissions

<Tip>
Always use the optimized bulk methods for list views. They compute all permission types in a single pass with 2-4 queries total.
</Tip>

```typescript
// 2-4 queries total (optimized single-pass)
const { permissions } = await this.permissionComputeService
  .computeAllBulkLeadPermissionsWithTeamsInTransaction(em, user, leadIds, leads);

const { permissions } = await this.permissionComputeService
  .computeAllBulkDealPermissionsWithTeamsInTransaction(em, user, dealIds, deals);
```

<Warning>
Never loop over individual permission computations. This causes N+1 queries.

```typescript
// 300-500 queries for 100 items
for (const lead of leads) {
  const permissions = await this.permissionComputeService
    .computeLeadPermissionsInTransaction(em, user, lead.id);
}
```
</Warning>

**Query count comparison:**

| Operation | Method | Database queries |
|-----------|--------|-----------------|
| 1 lead/deal | Single method | 3-5 queries |
| 100 leads/deals | Loop (N+1) | 300-500 queries |
| 100 leads/deals | Optimized bulk (`computeAll*`) | 2-4 queries |

All permission checks happen within the same EntityManager transaction, maintaining RLS context.
