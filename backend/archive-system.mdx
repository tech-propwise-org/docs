## Overview

The archive system provides a **two-tier removal model** for CRM business entities. Instead of immediately deleting records, entities go through an archive phase first, giving users a reversible option before permanent removal.

<Tabs>
<Tab title="Archive (tier 1)">
- **Purpose:** Make an entity inactive while preserving data for reporting, auditing, and relationship integrity
- **Visibility:** Hidden from active views, accessible via filters/toggles
- **Data:** Preserved completely — all relationships and history remain intact
- **Reversible:** Yes — unarchive restores the entity to active state
- **Operations:** Most operations are blocked on archived entities
- **Permission:** `crm.manage`
</Tab>

<Tab title="Soft delete (tier 2)">
- **Purpose:** Remove archived business entities that are no longer needed
- **Requirement:** Entity must be archived for at least **30 days**
- **Confirmation:** User must type the full entity name to confirm
- **Cascade:** Soft-deletes all related records (stakeholders, history, activities, notes)
- **Visibility:** Completely hidden from all views
- **Data:** Soft-deleted (`isDeleted = true`) — not physically removed
- **Reversible:** No standard undelete flow (requires manual database intervention)
- **Permission:** `system.admin`
</Tab>

<Tab title="Delete (config entities)">
- **Purpose:** Remove configuration items that are no longer needed
- **Visibility:** Completely hidden from all views
- **Data:** Soft-deleted (`isDeleted = true`) — not physically removed
- **Reversible:** No standard undelete flow (requires manual intervention)
</Tab>
</Tabs>

<Warning>
  The two-tier system exists for safety. Tier 1 (archive) is quick and
  reversible. Tier 2 (soft delete) requires a 30-day cooling-off period,
  `system.admin` permission, and explicit name confirmation. You cannot skip
  tier 1.
</Warning>

## Archive implementation pattern

### Entity fields

Add these three fields to any entity that supports archiving:

```typescript
@Property({ type: 'boolean', default: false })
@Index()
isArchived: boolean = false;

@Property({ nullable: true })
archivedAt?: Date;

@ManyToOne(() => User, { nullable: true })
archivedBy?: User;
```

### Entity filter update

Update the default `active` filter to exclude both deleted and archived records:

```typescript
@Filter({
  name: 'active',
  default: true,
  cond: { isDeleted: false, isArchived: false }
})
```

### Index updates

Update existing indexes and add a dedicated archive index for query performance:

```typescript
@Index({
  name: 'idx_lead_pipeline',
  properties: ['organization', 'stage', 'isDeleted', 'isArchived']
})

@Index({
  name: 'idx_lead_archived',
  properties: ['organization', 'isArchived', 'archivedAt']
})
```

## Pre-archive validations

Before archiving an entity, validate these business rules:

1. **Blocking dependencies** — What related entities would make archiving invalid?
2. **Cascading updates** — What related records need status updates?
3. **Status restrictions** — Can entities in any status be archived?

## Actions on archive

When an entity is archived, related records are preserved — not deleted or cascaded. Define what happens to related entities when the main entity is archived:

### Lead implementation decisions

| Related entity        | Action             | Reason                           |
| --------------------- | ------------------ | -------------------------------- |
| EntityStakeholder     | Keep as-is         | Historical assignment records    |
| LeadStageHistory      | Keep as-is         | Timeline/audit purposes          |
| Activities            | Keep as-is         | Historical communication records |
| LeadPropertyInterest  | Keep visible       | Property interest data valuable  |
| Deals.originatingLead | Maintain reference | Deal lineage for reporting       |

<Note>
The archive operation only changes the entity's own `isArchived`, `archivedAt`, and `archivedBy` fields. All related records remain untouched and visible in the archived entity's detail view.
</Note>

## Edge cases

### Create from archived entity

You cannot create child entities from an archived parent. For example, creating a deal from an archived lead is blocked:

```typescript
if (originatingLead && originatingLead.isArchived) {
  throw new BadRequestException(
    'Cannot create deal from an archived lead. Please unarchive the lead first.',
  );
}
```

### Bulk archive

Bulk operations use a **partial success** strategy — valid items are archived and failures are reported individually:

```typescript
async bulkArchiveLeads(
  leadIds: string[],
  userId: string,
  organizationId: string,
): Promise<{
  success: string[];
  failed: { id: string; reason: string }[];
}> {
  const results = { success: [], failed: [] };

  for (const leadId of leadIds) {
    try {
      await this.archiveLead(leadId, userId, organizationId);
      results.success.push(leadId);
    } catch (error) {
      results.failed.push({ id: leadId, reason: error.message });
    }
  }

  return results;
}
```

### Unarchive with deleted dependencies

If a required dependency was deleted while the entity was archived (e.g., the assigned stage), the system recovers by assigning to the "New" system stage:

```typescript
if (!entity.stage || entity.stage.isDeleted) {
  const newStage = await this.stageService.getNewStageInTransaction(em);
  entity.stage = newStage;
  entity.stageEnteredAt = new Date();
}
```

### Operations blocked on archived entities

All of the following operations throw `BadRequestException` with a message guiding the user to unarchive first:

| Operation                 | Reason                                                         |
| ------------------------- | -------------------------------------------------------------- |
| Update entity             | Prevents accidental modifications to historical records        |
| Create activities         | Activities should only be added to active entities             |
| Create notes              | Notes should only be added to active entities                  |
| Create/update calls       | Communication logs should only be added to active entities     |
| Create/update meetings    | Meeting records should only be added to active entities        |
| Create/update emails      | Email logs should only be added to active entities             |
| Create/update SMS         | SMS logs should only be added to active entities               |
| Create/update site visits | Site visit records should only be added to active entities     |
| Request transfers         | Transfers should only apply to active entities in the workflow |
| Create child entities     | New child entities should only originate from active parents   |

<Tip>
  Users must unarchive the entity first to perform any blocked operation, then
  can re-archive if needed.
</Tip>

### Person entity exception

Person does **not** have archive functionality because it is shared between Lead and Contact. Archiving a Person would affect multiple business entities unexpectedly. Instead, archive the Lead or Contact that references the Person.

## Database schema changes

### Migration template

Use this template when adding archive support to a new entity:

```typescript
import { Migration } from '@mikro-orm/migrations';

export class Migration_add_[entity]_archive extends Migration {
  async up(): Promise<void> {
    this.addSql(`
      ALTER TABLE "[entity]"
      ADD COLUMN IF NOT EXISTS "is_archived" boolean NOT NULL DEFAULT false,
      ADD COLUMN IF NOT EXISTS "archived_at" timestamptz,
      ADD COLUMN IF NOT EXISTS "archived_by_id" uuid REFERENCES "user"(id);
    `);

    this.addSql(`
      CREATE INDEX IF NOT EXISTS "idx_[entity]_archived"
      ON "[entity]" ("organization_id", "is_archived", "archived_at");
    `);

    this.addSql(`DROP INDEX IF EXISTS "idx_[entity]_pipeline";`);
    this.addSql(`
      CREATE INDEX "idx_[entity]_pipeline"
      ON "[entity]" ("organization_id", "stage_id", "is_deleted", "is_archived");
    `);
  }

  async down(): Promise<void> {
    this.addSql(`ALTER TABLE "[entity]" DROP COLUMN IF EXISTS "is_archived";`);
    this.addSql(`ALTER TABLE "[entity]" DROP COLUMN IF EXISTS "archived_at";`);
    this.addSql(`ALTER TABLE "[entity]" DROP COLUMN IF EXISTS "archived_by_id";`);
    this.addSql(`DROP INDEX IF EXISTS "idx_[entity]_archived";`);
  }
}
```

## API endpoints

### Archive endpoints

```typescript
// Archive single entity
@Post(':id/archive')
async archive(@Param('id') id: string): Promise<{ success: boolean; archivedAt: Date }>

// Unarchive single entity
@Post(':id/unarchive')
async unarchive(@Param('id') id: string): Promise<{ success: boolean }>

// Bulk archive
@Post('bulk-archive')
async bulkArchive(@Body() dto: { ids: string[] }): Promise<{
  success: boolean;
  archivedCount: number;
  failedCount: number;
  failures: Array<{ id: string; reason: string }>;
}>

// Bulk unarchive
@Post('bulk-unarchive')
async bulkUnarchive(@Body() dto: { ids: string[] }): Promise<{
  success: boolean;
  unarchivedCount: number;
  failedCount: number;
  failures: Array<{ id: string; reason: string }>;
}>
```

### Delete endpoints

```typescript
// Soft delete with cascade (requires 30-day archive + name confirmation)
@Delete(':id')
async delete(
  @Param('id') id: string,
  @Body() body: { confirmName: string }
): Promise<{
  success: boolean;
  deletedAt: Date;
  relatedRecordsDeleted: { /* entity-specific counts */ };
}>

// Bulk soft delete (requires 30-day archive for all)
@Post('bulk-delete')
async bulkDelete(@Body() dto: { ids: string[] }): Promise<{
  success: boolean;
  deletedCount: number;
  failedCount: number;
  failures: Array<{ id: string; reason: string }>;
}>
```

<Warning>
  Delete requirements: entity must be archived (`isArchived = true`) for at
  least 30 days, user must have `system.admin` permission, and explicit name
  confirmation is required. Cascade soft-deletes all related records and
  nullifies child entity references. The Person entity is never deleted.
</Warning>

### List view with archive filter

Add `showArchived` query parameter to list endpoints:

```typescript
@ApiQuery({
  name: 'showArchived',
  required: false,
  type: Boolean,
  description: 'If true, show only archived; if false/undefined (default), show only active',
})
async getList(
  @Query('showArchived') showArchived?: boolean,
): Promise<ListResponse> {
  // In service: filter by isArchived = showArchived === true
}
```

### Kanban view with archive column

Support `stageId='archived'` for infinite scroll on the archived column:

```typescript
@ApiQuery({
  name: 'stageId',
  description: 'Stage UUID for regular stages, or "archived" for archived column',
})
```

## Query patterns

### List view query

```typescript
qb.andWhere({ isArchived: query.showArchived === true });

const defaultSort = query.showArchived === true ? "archivedAt" : "createdAt";
```

### Kanban view query

```typescript
const targetStages = query.stageId
  ? query.stageId === "archived"
    ? []
    : allStages.filter((s) => s.id === query.stageId)
  : allStages;

if (!query.stageId || query.stageId === "archived") {
  const archivedQb = this.buildQueryForType(em, user, query.type, query.teamId);
  archivedQb.andWhere({ isArchived: true });
  // pagination, sorting by archivedAt DESC
}

// Regular stages exclude archived
qb.andWhere({ stage: stage.id, isArchived: false });
```

## Implementation checklist

Use this checklist when adding archive support to a new entity.

<AccordionGroup>
<Accordion title="Planning phase">
- Identify blocking dependencies (what prevents archive?)
- Identify cascading updates (what records need updating?)
- Define status restrictions (which statuses can be archived?)
- Define related entity behavior (keep/archive/update?)
- Define edge cases (create from archived, bulk, unarchive scenarios)
</Accordion>

<Accordion title="Implementation phase">
- **Entity:** Add `isArchived`, `archivedAt`, `archivedBy` fields
- **Entity:** Update `@Filter` to exclude archived from active
- **Entity:** Add/update indexes for `isArchived`
- **Migration:** Create migration for schema changes
- **Service:** Implement `archive[Entity]()` with validations
- **Service:** Implement `unarchive[Entity]()` with recovery logic
- **Service:** Implement `bulkArchive[Entity]()` with partial success
- **Service:** Implement `bulkUnarchive[Entity]()` with partial success
- **Service:** Update `update[Entity]()` to block archived entities
- **Service:** Update `delete[Entity]()` to require 30-day archive
- **Service:** Update `bulkDelete[Entity]()` to require 30-day archive with partial success
- **Service:** Update list queries to support `showArchived` filter
- **Service:** Update kanban queries to support `stageId='archived'`
- **Service:** Add guard in transfer request to block archived entities
- **Controller:** Add archive/unarchive/bulk-archive/bulk-unarchive endpoints
- **Controller:** Add `showArchived` query param to list endpoint
- **Controller:** Update delete endpoint docs with 30-day requirement
- **Controller:** Update Swagger documentation
- **DTO:** Add `isArchived`, `archivedAt`, `archivedBy` to response DTO
- **Activity Service:** Add validation to block activity creation on archived entities
- **Related Services:** Add guards to prevent invalid operations on archived entities
</Accordion>

<Accordion title="Validation phase">
- Run migration successfully
- TypeScript compiles without errors
- Linter passes
- Test archive flow
- Test unarchive flow
- Test bulk archive
- Test list view with `showArchived` filter
- Test kanban view with archived column pagination
- Test edge cases (create from archived, deleted dependencies)
</Accordion>
</AccordionGroup>
