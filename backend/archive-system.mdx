## Overview

The archive system provides a **two-tier removal model** for CRM business entities. Instead of immediately deleting records, entities go through an archive phase first, giving users a reversible option before permanent removal.

<Tabs>
<Tab title="Archive (tier 1)">
- **Purpose:** Make an entity inactive while preserving data for reporting, auditing, and relationship integrity
- **Visibility:** Hidden from active views, accessible via filters/toggles
- **Data:** Preserved completely — all relationships and history remain intact
- **Reversible:** Yes — unarchive restores the entity to active state
- **Operations:** Most operations are blocked on archived entities
- **Permission:** Creator ownership, `FULL` stakeholder access, or org manage permission (`crm.manage`; `sales.manage` for deals)
</Tab>

<Tab title="Soft delete (tier 2)">
- **Purpose:** Remove archived business entities that are no longer needed
- **Requirement:** Entity must be archived for at least **30 days**
- **Confirmation:** User must type the full entity name to confirm
- **Cascade:** Soft-deletes related records that support soft delete (stakeholders, history, activities, notes, etc.) and status-cancels workflow records such as pending transfers
- **Visibility:** Completely hidden from all views
- **Data:** Soft-deleted (`isDeleted = true`) — not physically removed
- **Reversible:** No standard undelete flow (requires manual database intervention)
- **Permission:** Creator ownership, `FULL` stakeholder access, or org manage permission (`crm.manage`; `sales.manage` for deals)
</Tab>

<Tab title="Delete (config entities)">
- **Purpose:** Remove configuration items that are no longer needed
- **Visibility:** Completely hidden from all views
- **Data:** Soft-deleted (`isDeleted = true`) — not physically removed
- **Reversible:** No standard undelete flow (requires manual intervention)
</Tab>
</Tabs>

<Warning>
  The two-tier system exists for safety. Tier 1 (archive) is quick and
  reversible. Tier 2 (soft delete) requires a 30-day cooling-off period and
  explicit name confirmation. You cannot skip tier 1.
</Warning>

## Actor permission vs lifecycle state

The backend separates **who has archive/delete authority** from **what lifecycle action is currently valid**:

- **Actor authority** comes from org manage permission, creator ownership, direct `FULL` stakeholder access, or team-scoped `FULL` stakeholder access plus the matching team manage permission (`team_crm.manage` for leads/contacts/companies, `team_sales.manage` for deals)
- **Creator ownership** intentionally remains delete/archive actor authority even if the creator is no longer a current stakeholder
- **`canArchive`** is true only when actor authority passes and the record is active (`isArchived=false`, `isDeleted=false`)
- **`canUnarchive`** is true only when actor authority passes and the record is archived but not deleted
- **`canDelete`** is true only when actor authority passes, the record is archived, not deleted, and `archivedAt` is at least 30 days old

<Note>
`ResourcePermissionsDto.fullAccess()` must not be used for archivable CRM business entities because it cannot represent this lifecycle state machine. Lead, deal, contact, and company permissions are computed explicitly by `PermissionComputeService`.
</Note>

## Archive implementation pattern

### Entity fields

Add these three fields to any entity that supports archiving:

```typescript
@Property({ type: 'boolean', default: false })
@Index()
isArchived: boolean = false;

@Property({ nullable: true })
archivedAt?: Date;

@ManyToOne(() => User, { nullable: true })
archivedBy?: User;
```

### Entity filter update

Declare TWO orthogonal default filters — one for soft-delete, one for archive — and intentionally do NOT overlap their conditions:

```typescript
// Both filters default to true, so standard queries apply isDeleted=false AND isArchived=false.
// The `active` filter intentionally omits isDeleted to avoid a redundant WHERE clause — the
// separate `isDeleted` filter (also default: true) already handles soft-delete exclusion.
// INVARIANT: Never disable `isDeleted` while keeping `active` — that would return soft-deleted records.
// To include archived rows, use `{ filters: { active: false } }`.
@Filter({ name: 'isDeleted', default: true, cond: { isDeleted: false } })
@Filter({ name: 'active',    default: true, cond: { isArchived: false } })
```

**Why orthogonal:** A single combined filter (e.g. `cond: { isDeleted: false, isArchived: false }`) is redundant with the separate `isDeleted` filter and creates a misleading mental model. Worse, it amplifies the impact of MikroORM v6's `autoJoinRefsForFilters` (see `Docs/QUERY_OPTIMIZATION_PATTERNS.md`). Keep the two concerns separate.

**Canonical include-archived read:**

```typescript
em.findOne(Lead, { id, isDeleted: false }, { filters: { active: false } })
```

## Pre-archive validations

Before archiving an entity, validate blocking dependencies and plan cascade updates. Ask these questions:

<Steps>
<Step title="Check blocking dependencies">
What related entities would make archiving invalid? (Example: active deals prevent lead archive)
</Step>

<Step title="Validate cascading updates">
What related records need status updates? (Example: pending transfers must be cancelled)
</Step>

<Step title="Check status restrictions">
Can entities in any status be archived? (Example: deals can be archived in open/won/lost status)
</Step>
</Steps>

<Tabs>
<Tab title="Lead">
**Blocking validation:**

```typescript
// VALIDATION: Check for active deals (BLOCK)
const activeDealCount = await em
  .createQueryBuilder(Deal, 'd')
  .select('COUNT(*)')
  .where({
    originatingLead: leadId,
    isClosed: false,
    isArchived: false,
    isDeleted: false,
  })
  .execute('get', false);

if (parseInt(activeDealCount[0]?.count || '0') > 0) {
  throw new BadRequestException(
    `Cannot archive lead: ${count} active deal(s) originated from this lead. ` +
      'Close or archive the deals first.',
  );
}
```

**Cascade action:**

```typescript
// ACTION: Cancel pending transfers
await em.nativeUpdate(
  'entity_transfer',
  {
    entity_type: StakeholderEntityType.LEAD,
    entity_id: leadId,
    status: 'pending',
  },
  {
    status: 'cancelled',
    reject_reason: 'Lead archived',
    rejected_at: new Date(),
    rejected_by_id: userId,
  },
);
```
</Tab>

<Tab title="Contact">
**Blocking validation:**

```typescript
// VALIDATION: Check for active deals via DealContact
const activeDealCount = await em
  .createQueryBuilder(Deal, 'd')
  .leftJoin('d.contacts', 'dc')
  .select('COUNT(*)')
  .where({
    'dc.contact': contactId,
    'd.isClosed': false,
    'd.isArchived': false,
    'd.isDeleted': false,
  })
  .execute('get', false);

if (parseInt(activeDealCount[0]?.count || '0') > 0) {
  throw new BadRequestException(
    `Cannot archive contact: ${count} active deal(s) associated with this contact. ` +
      'Close or archive the deals first.',
  );
}
```

**No cascade updates needed** — contacts don't have transfers.
</Tab>

<Tab title="Deal">
**Blocking validation:**

```typescript
// VALIDATION: Check for unpaid commissions
const unpaidCommissionCount = await em
  .createQueryBuilder(CommissionPayment, 'cp')
  .select('COUNT(*)')
  .where({
    deal: dealId,
    status: { $ne: 'paid' },
    isDeleted: false,
  })
  .execute('get', false);

if (parseInt(unpaidCommissionCount[0]?.count || '0') > 0) {
  throw new BadRequestException(
    `Cannot archive deal: ${count} unpaid commission(s) exist. ` +
      'Pay or cancel commissions first.',
  );
}
```

**Cascade action:**

```typescript
// ACTION: Cancel pending transfers
await em.nativeUpdate(
  'entity_transfer',
  {
    entity_type: StakeholderEntityType.DEAL,
    entity_id: dealId,
    status: 'pending',
  },
  {
    status: 'cancelled',
    reject_reason: 'Deal archived',
    rejected_at: new Date(),
    rejected_by_id: userId,
  },
);
```
</Tab>

<Tab title="Company">
**No blocking validations** — companies can be archived freely regardless of active contacts or deals.

**No cascade updates needed** — companies don't have transfers.
</Tab>

<Tab title="Conversation">
**No blocking validations** — conversations can be archived freely.

**No cascade updates needed** — conversations don't have complex dependencies.
</Tab>
</Tabs>

## Actions on archive

When an entity is archived, related records are **preserved** — not deleted or cascaded. The archive operation only changes the entity's own `isArchived`, `archivedAt`, and `archivedBy` fields. All related records remain untouched and visible in the archived entity's detail view.

<Note>
Define what happens to related entities when the main entity is archived. This is about visibility and reference integrity, not deletion.
</Note>

### Lead archive behavior

| Related entity        | Action             | Reason                           |
| --------------------- | ------------------ | -------------------------------- |
| EntityStakeholder     | Keep as-is         | Historical assignment records    |
| LeadStageHistory      | Keep as-is         | Timeline/audit purposes          |
| Activities            | Keep as-is         | Historical communication records |
| LeadPropertyInterest  | Keep visible       | Property interest data valuable  |
| Deals.originatingLead | Maintain reference | Deal lineage for reporting       |

## Edge cases

### Operations blocked on archived entities

All of the following operations throw `BadRequestException` with a message guiding the user to unarchive first:

<Tabs>
<Tab title="Update operations">
| Operation                 | Reason                                                         |
| ------------------------- | -------------------------------------------------------------- |
| Update entity             | Prevents accidental modifications to historical records        |
| Stage changes             | Stage progression should only apply to active entities         |
</Tab>

<Tab title="Activity operations">
| Operation                 | Reason                                                         |
| ------------------------- | -------------------------------------------------------------- |
| Create activities         | Activities should only be added to active entities             |
| Create notes              | Notes should only be added to active entities                  |
| Create/update calls       | Communication logs should only be added to active entities     |
| Create/update meetings    | Meeting records should only be added to active entities        |
| Create/update emails      | Email logs should only be added to active entities             |
| Create/update SMS         | SMS logs should only be added to active entities               |
| Create/update site visits | Site visit records should only be added to active entities     |
</Tab>

<Tab title="Workflow operations">
| Operation                 | Reason                                                         |
| ------------------------- | -------------------------------------------------------------- |
| Request transfers         | Transfers should only apply to active entities in the workflow |
| Create child entities     | New child entities should only originate from active parents   |
| Process commissions       | Commission calculations should only apply to active deals      |
</Tab>
</Tabs>

<Tip>
  Users must unarchive the entity first to perform any blocked operation, then
  can re-archive if needed.
</Tip>

### Create from archived entity

You cannot create child entities from an archived parent. For example, creating a deal from an archived lead is blocked:

```typescript
if (originatingLead && originatingLead.isArchived) {
  throw new BadRequestException(
    'Cannot create deal from an archived lead. Please unarchive the lead first.',
  );
}
```

<Tip>
**Question to ask**: What child entities need this validation? For each entity that references an archived parent, prevent creation by checking `parent.isArchived`.
</Tip>

### Unarchive with deleted dependencies

If a required dependency was deleted while the entity was archived (e.g., the assigned stage), the system recovers by assigning to the "New" system stage:

```typescript
if (!entity.stage || entity.stage.isDeleted) {
  const newStage = await this.stageService.getNewStageInTransaction(em);
  entity.stage = newStage;
  entity.stageEnteredAt = new Date();
}
```

### Bulk operations

Bulk operations use a **partial success** strategy — valid items are archived and failures are reported individually:

```typescript
async bulkArchiveLeads(
  leadIds: string[],
  userId: string,
  organizationId: string,
): Promise<{
  success: string[];
  failed: { id: string; reason: string }[];
}> {
  const results = { success: [], failed: [] };

  for (const leadId of leadIds) {
    try {
      await this.archiveLead(leadId, userId, organizationId);
      results.success.push(leadId);
    } catch (error) {
      results.failed.push({ id: leadId, reason: error.message });
    }
  }

  return results;
}
```

### Person entity exception

Person does **not** have archive functionality because it is shared between Lead and Contact. Archiving a Person would affect multiple business entities unexpectedly. Instead, archive the Lead or Contact that references the Person.

## Database schema changes

### Migration template

Use this template when adding archive support to a new entity:

```sql
ALTER TABLE "[entity]"
ADD COLUMN IF NOT EXISTS "is_archived" boolean NOT NULL DEFAULT false,
ADD COLUMN IF NOT EXISTS "archived_at" timestamptz,
ADD COLUMN IF NOT EXISTS "archived_by_id" uuid REFERENCES "user"(id);

CREATE INDEX IF NOT EXISTS "idx_[entity]_archived"
ON "[entity]" ("organization_id", "is_archived", "archived_at");

DROP INDEX IF EXISTS "idx_[entity]_pipeline";
CREATE INDEX "idx_[entity]_pipeline"
ON "[entity]" ("organization_id", "stage_id", "is_deleted", "is_archived");
```

<Note>
Replace `[entity]` with the actual table name. Adjust the pipeline index to match your entity's specific columns (e.g., some entities may not have `stage_id`).
</Note>

### Indexing strategy

Archive-aware queries need composite indexes that include `isArchived`:

- **List view index:** `(organization_id, is_archived, archived_at)` for filtering and sorting
- **Pipeline index:** `(organization_id, stage_id, is_deleted, is_archived)` for kanban views
- **Filtering index:** `(organization_id, is_archived, is_deleted)` for active/archived toggles

## API endpoints

### Archive endpoints

<CodeGroup>

```typescript Archive single entity
@Post(':id/archive')
async archive(
  @Param('id') id: string,
  @CurrentUser() user: User,
): Promise<{ success: boolean; archivedAt: Date }> {
  const entity = await this.service.archiveEntity(id, user.id, user.organizationId);
  return { success: true, archivedAt: entity.archivedAt };
}
```

```typescript Unarchive single entity
@Post(':id/unarchive')
async unarchive(
  @Param('id') id: string,
  @CurrentUser() user: User,
): Promise<{ success: boolean }> {
  await this.service.unarchiveEntity(id, user.id, user.organizationId);
  return { success: true };
}
```

```typescript Bulk archive
@Post('bulk-archive')
async bulkArchive(
  @Body() dto: { ids: string[] },
  @CurrentUser() user: User,
): Promise<{
  success: boolean;
  archivedCount: number;
  failedCount: number;
  failures: Array<{ id: string; reason: string }>;
}> {
  const result = await this.service.bulkArchiveEntity(
    dto.ids,
    user.id,
    user.organizationId,
  );
  return {
    success: result.failed.length === 0,
    archivedCount: result.success.length,
    failedCount: result.failed.length,
    failures: result.failed,
  };
}
```

```typescript Bulk unarchive

    success: result.failed.length === 0,
```

<Warning>
  Delete requirements: entity must be archived (`isArchived = true`) for at
  least 30 days, user must have proper permission (creator, `FULL` stakeholder access, or org manage), and explicit name
  confirmation is required for single deletes. Cascade soft-deletes all related records and
  nullifies child entity references. The Person entity is never deleted.
</Warning>

## Query patterns

### List view query

```typescript
// Filter by archive status
qb.andWhere({ isArchived: query.showArchived === true });

// Sort by appropriate date field
const defaultSort = query.showArchived === true ? "archivedAt" : "createdAt";
qb.orderBy({ [defaultSort]: 'DESC' });
```

### Kanban view query

```typescript
// Determine target stages
const targetStages = query.stageId
  ? query.stageId === "archived"
    ? []
    : allStages.filter((s) => s.id === query.stageId)
  : allStages;

// Build archived column query
if (!query.stageId || query.stageId === "archived") {
  const archivedQb = this.buildQueryForType(em, user, query.type, query.teamId);
  archivedQb.andWhere({ isArchived: true });
  archivedQb.orderBy({ archivedAt: 'DESC' });
  // Apply pagination for infinite scroll
}

// Regular stages exclude archived
for (const stage of targetStages) {
  qb.andWhere({ stage: stage.id, isArchived: false });
}
```

### Joins for archive metadata

```typescript
// In query builder
queryBuilder.leftJoinAndSelect("entity.archivedBy", "archivedBy");

// In findOne/findAndCount options
const options = {
  populate: ["archivedBy", /* other fields */],
};
```

### Loading strategies

Use `filters: { active: false }` to include archived entities in reads:

```typescript
// Standard read (active only)
const entity = await em.findOne(Lead, { id });

// Include archived
const entity = await em.findOne(
  Lead,
  { id, isDeleted: false },
  { filters: { active: false } }
);
```

<Tabs>

### Archive decisions

| Aspect                       | Decision                                           |
| ---------------------------- | -------------------------------------------------- |
| Blocking dependencies        | Active deals — block                               |
| Cascading updates            | Pending transfers — cancel                         |
| Status restrictions          | Any status can be archived                         |
| Stakeholders                 | Keep as-is                                         |

| Update archived lead         | Block                                              |
| Create activities/notes      | Block                                              |
| Create transfers             | Block                                              |

### Soft delete decisions
