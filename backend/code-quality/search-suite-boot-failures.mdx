# Search Suite Boot Failures

<Note>
**Status:** FIXED & verified. Two independent boot blockers were found and resolved.
**Discovered:** 2026-05-29 while running `pnpm test:e2e:queue:search`.
</Note>

## Overview

The search suite boot failures were caused by two distinct issues that prevented the PropWise CRM backend from starting during e2e test execution. This document provides a comprehensive analysis of both root causes and their verified solutions.

| --- | --- | --- | --- |
| 1 | `EntityStakeholderModule.providers[0]` is `undefined` → `CircularDependencyException` | Runtime ES-module (CommonJS) load-order cycle | `unit-access.service.ts` imports `User` from `@/modules/user/user.entity` instead of the `@/modules/user` **barrel** |
| 2 | `Nest can't resolve … LeadAiToolsProvider … LeadAiService at index [0]` | **Missing provider** (not a cycle) | Register `LeadAiService` in `LeadModule` `providers` + `exports` |

## Root Cause Analysis

### Blocker #1: Runtime ES-Module Load-Order Cycle

<Warning>
This is a **CommonJS load-order cycle**, not a NestJS DI cycle. `forwardRef()` cannot fix this type of issue.
</Warning>

#### The Symptom

`pnpm test:e2e:queue:search` failed during Nest test-app boot with:

```bash
A circular dependency has been detected inside EntityStakeholderModule.
Please, make sure that each side of a bidirectional relationships are decorated with "forwardRef()".
```

Investigation revealed that `EntityStakeholderModule`'s provider list at runtime was:

```ts
Providers: [
  undefined,                       // <-- should be EntityStakeholderService
  EntityStakeholderHistoryService,
  StakeholderSyncService,
]
```

#### The Cycle Path

The circular dependency flows through the following path:

```
entity-stakeholder.service.ts        (line 63) imports
  → unit-access.service.ts           (line 26) imports the `@/modules/user` BARREL
    → user/index.ts                  `export * from './user.module' | './user.controller'`
      → user.controller.ts           imports 5 CRM services + the `../auth` barrel
        → commission-payment.service.ts / lead.service.ts / deal.service.ts /
          contact.service.ts / entity-transfer.service.ts
            → back into entity-stakeholder.service.ts / entity-stakeholder.module.ts
```

<Info>
The amplifier is the **barrel import**: `unit-access.service.ts` only needs the `User` entity, but importing `@/modules/user` pulls in `user.controller.ts` which is a heavy DI hub.
</Info>

#### Why It's Load-Order Sensitive

The issue is latent and only manifests when certain entry points are loaded first:

| Entry point | `providers[0]` result |
| --- | --- |
| `entity-stakeholder.module.ts` loaded first (≈ `AppModule` / `pnpm start`) | `EntityStakeholderService` ✅ |
| `search-backfill.service.ts` (what `test/helpers/search-e2e.ts` imports) | `UNDEFINED` ❌ |
| `commission-payment.service.ts` loaded first | `UNDEFINED` ❌ |

### Blocker #2: Missing Provider Registration

Once blocker #1 was resolved, the boot process advanced but hit a second issue:

```bash
Nest can't resolve dependencies of the LeadAiToolsProvider (?).
Please make sure that the argument LeadAiService at index [0] is available in the LeadModule module.
```

<Note>
This is **not** a circular dependency - it's a straightforward missing provider registration issue.
</Note>

`LeadAiService` was being injected by three consumers but wasn't registered as a provider in any module:
- `LeadAiToolsProvider` (registered in both `LeadModule` and `AiAssistantModule`)
- `AiContextBuilderService`
- `AiAssistantOrchestratorService`

## Solutions

### Fix for Blocker #1: Direct Entity Import

<Steps>
  <Step title="Identify the problematic barrel import">
    The issue was in `src/modules/real-estate/inventory/unit-access/unit-access.service.ts` importing the entire user module barrel.
  </Step>
  
  <Step title="Replace barrel import with direct entity import">
    Change the import to target the specific entity file:
    
    ```ts
    // src/modules/real-estate/inventory/unit-access/unit-access.service.ts
    - import { User } from '@/modules/user';
    + import { User } from '@/modules/user/user.entity';
    ```
  </Step>
  
  <Step title="Verify the fix">
    This severs `user.controller.ts` and the entire downstream CRM-service graph from `unit-access.service.ts`'s import tree.
  </Step>
</Steps>

#### Additional Hardening

To prevent similar issues, redirect remaining barrel imports to specific DTO modules:

```ts
// event.service.ts, notification.dto.ts, notification-preference.dto.ts
- import { UserDto } from '@/modules/user';            // (or UserDirectoryDto)
+ import { UserDto } from '@/modules/user/user.dto';
```

<Check>
After this change, **no file under `src/` imports the `@/modules/user` barrel**, eliminating this entire class of circular dependencies.
</Check>

### Fix for Blocker #2: Register Missing Provider

<Steps>
  <Step title="Add LeadAiService to LeadModule providers">
    ```ts
    // src/modules/crm/lead/lead.module.ts
    import { LeadAiService } from './lead-ai.service';
    
    @Module({
      providers: [LeadService, LeadAiService, ...],
      exports: [MikroOrmModule, LeadService, LeadAiService, ...],
    })
    ```
  </Step>
  
  <Step title="Export the service for external modules">
    The service needs to be exported so `AiAssistantModule` can resolve it for its own consumers.
  </Step>
</Steps>

### Regression Prevention

A `madge`-based baseline-delta CI guard was implemented:

<Tabs>
  <Tab title="Check for new cycles">
    ```bash
    pnpm lint:circular
    ```
    Fails if a cycle appears that is not in the baseline.
  </Tab>
  
  <Tab title="Update baseline">
    ```bash
    pnpm lint:circular:update
    ```
    Regenerate the baseline (use only after review).
  </Tab>
</Tabs>

<Warning>
A hard "zero cycles" gate is not viable since the codebase has ~156 intentional NestJS module cycles resolved by `forwardRef()`. The baseline-delta approach only catches **new** runtime cycles.
</Warning>

## Important Corrections

<Warning>
The original draft proposed using DI tokens with `import type` to break the circular dependency. **This approach was tested and does NOT work.** The verified fix is the direct entity import approach documented above.
</Warning>

The token-based approach fails because:
- `commission-payment.module.ts` imports `EntityStakeholderModule` directly
- `user.controller.ts` imports five services that each import `entity-stakeholder.service.ts`
- These paths re-enter the module while it's still loading, regardless of type-only service imports
