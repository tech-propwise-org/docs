## Overview

The search suite boot failures were caused by two distinct issues that prevented the PropWise CRM backend from starting properly during end-to-end testing.

| # | Blocker | Class | Fix that actually works |
| --- | --- | --- | --- |
| 1 | `EntityStakeholderModule.providers[0]` is `undefined` → `CircularDependencyException` | Runtime ES-module (CommonJS) load-order cycle | `unit-access.service.ts` imports `User` from `@/modules/user/user.entity` instead of the `@/modules/user` **barrel** (1 line) |
| 2 | `Nest can't resolve … LeadAiToolsProvider … LeadAiService at index [0]` | **Missing provider** (not a cycle) | Register `LeadAiService` in `LeadModule` `providers` + `exports` |

<Warning>
**Important correction:** the first draft recommended breaking `commission-payment.service.ts → entity-stakeholder.service.ts` with a DI token + `import type` ("Option A"). **That fix does not work** — see the "Why Option A Failed" section below.
</Warning>

## Symptom Analysis

### Initial Error
`pnpm test:e2e:queue:search` failed during Nest test-app boot with:

```bash
A circular dependency has been detected inside EntityStakeholderModule.
Please, make sure that each side of a bidirectional relationships are decorated with "forwardRef()".
```

### Runtime Investigation
`NestContainer.addProvider()` throws this when it receives `undefined` as a provider. `EntityStakeholderModule`'s provider list at runtime confirmed the issue:

```typescript
Providers: [
  undefined,                       // <-- should be EntityStakeholderService
  EntityStakeholderHistoryService,
  StakeholderSyncService,
]
```

## Root Cause Analysis

### Blocker #1: Runtime ES-Module Load-Order Cycle

This is a **CommonJS load-order cycle**, not a NestJS DI cycle. When file A imports file B, Node must finish executing B before A continues. If B (directly or transitively) imports A back **while A is still executing**, Node returns A's **partial** `module.exports` — so a value imported from A at that instant is `undefined`.

#### The Cycle Path
```
entity-stakeholder.service.ts        (line 63) imports
  → unit-access.service.ts           (line 26) imports the `@/modules/user` BARREL
    → user/index.ts                  `export * from './user.module' | './user.controller'`
      → user.controller.ts           imports 5 CRM services + the `../auth` barrel
        → commission-payment.service.ts / lead.service.ts / deal.service.ts /
          contact.service.ts / entity-transfer.service.ts
            → back into entity-stakeholder.service.ts / entity-stakeholder.module.ts
```

<Info>
The amplifier is the **barrel**: `unit-access.service.ts` only needs the `User` *entity*, but importing `@/modules/user` pulls in `user.controller.ts` (a heavy DI hub that imports multiple CRM services + the `auth` barrel).
</Info>

#### Load-Order Sensitivity
`EntityStakeholderModule.providers[0]` is only `undefined` when a file that **starts** the cycle is loaded before `entity-stakeholder.module.ts`:

| Entry point | `providers[0]` |
| --- | --- |
| `entity-stakeholder.module.ts` loaded first (≈ `AppModule` / `pnpm start`) | `EntityStakeholderService` ✅ |
| `search-backfill.service.ts` (what `test/helpers/search-e2e.ts` top-imports) | `UNDEFINED` ❌ |
| `commission-payment.service.ts` loaded first | `UNDEFINED` ❌ |

### Why `forwardRef()` Doesn't Help

`forwardRef()` defers a **token lookup inside Nest's DI container**; it assumes the class reference is *defined* at decorator-evaluation time. Here the class reference is `undefined` because the ES-module cycle returned partial exports. There is nothing for the `forwardRef` thunk to resolve.

<Tip>
The cycle has to be broken at the **file-import** layer, not the DI layer.
</Tip>

## Why Option A Failed

The first draft proposed making `commission-payment.service.ts` import `EntityStakeholderService` as a **type-only** import and inject it via an `ENTITY_STAKEHOLDER_SERVICE` symbol token.

**This was tested and does NOT fix the bug.** With `commission-payment.service.ts` using `import type`, `EntityStakeholderModule.providers[0]` is **still `undefined`** under the search entry point.

**Reason:** The slot is poisoned by the **module/controller graph**, not by the `commission-payment.service → entity-stakeholder.service` edge:

- `commission-payment.module.ts` (line 23) imports `EntityStakeholderModule` **directly**
- `user.controller.ts` imports **five** services that each import `entity-stakeholder.service.ts`

## The Verified Fix

<Steps>
<Step title="Fix Blocker #1: Direct Entity Import">
Import the `User` **entity** directly instead of the `@/modules/user` **barrel**:

```typescript
// src/modules/real-estate/inventory/unit-access/unit-access.service.ts
- import { User } from '@/modules/user';
+ import { User } from '@/modules/user/user.entity';
```

This severs `user.controller.ts` (and the entire downstream CRM-service / `auth` graph) from `unit-access.service.ts`'s import tree.
</Step>

<Step title="Harden Against Similar Issues">
Remove barrel imports from other services and DTOs:

```typescript
// event.service.ts, notification.dto.ts, notification-preference.dto.ts
- import { UserDto } from '@/modules/user';            // (or UserDirectoryDto)
+ import { UserDto } from '@/modules/user/user.dto';
```

After this, **no file under `src/` imports the `@/modules/user` barrel**.
</Step>

<Step title="Fix Blocker #2: Register Missing Provider">
`LeadAiService` was being injected but never provided. Add it to `LeadModule`:

```typescript
// src/modules/crm/lead/lead.module.ts
import { LeadAiService } from './lead-ai.service';

@Module({
  providers: [LeadService, LeadAiService, ...],
  exports: [MikroOrmModule, LeadService, LeadAiService, ...],
})
```
</Step>
</Steps>

## Blocker #2 Details

Once blocker #1 was cleared, boot advanced and hit:

```bash
Nest can't resolve dependencies of the LeadAiToolsProvider (?).
Please make sure that the argument LeadAiService at index [0] is available in the LeadModule module.
```

<Note>
This is **not** a cycle and **not** load-order-dependent. `LeadAiService` was **injected** by three consumers but registered as a provider in **no module**.
</Note>

### The Missing Provider Issue
- `LeadAiService` is injected by:
  - `LeadAiToolsProvider` (registered in both `LeadModule` and `AiAssistantModule`)
  - `AiContextBuilderService`
  - `AiAssistantOrchestratorService`
- But it was never registered in any module's `providers` array

## Regression Guard: Madge Baseline-Delta

A baseline-delta guard prevents new runtime cycles without failing on existing intentional NestJS module relationships:

<Tabs>
<Tab title="Implementation">
- `scripts/check-circular-deps.ts` runs `madge` with `skipTypeImports`
- Canonicalizes each cycle and diffs against `scripts/circular-deps.baseline.json`
- Only fails on **new** cycles not in the baseline
</Tab>

<Tab title="Commands">
```bash
# Check for new cycles
pnpm lint:circular

# Update baseline (use only after review)
pnpm lint:circular:update
```
</Tab>
</Tabs>

<Warning>
A hard "zero cycles" gate is **not viable**: the codebase already has ~156 file-level cycles, the overwhelming majority of which are intentional NestJS `*.module.ts` ↔ `*.module.ts` relationships resolved by `forwardRef()`.
</Warning>
