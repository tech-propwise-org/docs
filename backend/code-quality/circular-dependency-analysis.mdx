## TL;DR

| # | Blocker | Class | Fix that actually works |
|---|---------|-------|-------------------------|
| 1 | `EntityStakeholderModule.providers[0]` is `undefined` ŌåÆ `CircularDependencyException` | Runtime ES-module (CommonJS) load-order cycle | `unit-access.service.ts` imports `User` from `@/modules/user/user.entity` instead of the `@/modules/user` **barrel** (1 line) |
| 2 | `Nest can't resolve ŌĆ” LeadAiToolsProvider ŌĆ” LeadAiService at index [0]` | **Missing provider** (not a cycle) | Register `LeadAiService` in `LeadModule` `providers` + `exports` |

Both were proven by reproducing the failure, applying the fix, and re-checking. A `madge`-based **baseline-delta** CI guard (`pnpm lint:circular`) was added to catch new runtime cycles.

<Warning>
**Important correction to the original draft of this document:** the first draft recommended breaking `commission-payment.service.ts ŌåÆ entity-stakeholder.service.ts` with a DI token + `import type` ("Option A"). **That fix does not work** ŌĆö see the "Why Option A Failed" section below. The verified fix is the `unit-access` barrel import.
</Warning>

## Symptom

`pnpm test:e2e:queue:search` failed during Nest test-app boot with:

```bash
A circular dependency has been detected inside EntityStakeholderModule.
Please, make sure that each side of a bidirectional relationships are decorated with "forwardRef()".
```

`NestContainer.addProvider()` throws this when it receives `undefined` as a provider. `EntityStakeholderModule`'s provider list at runtime confirmed it:

```typescript
Providers: [
  undefined,                       // <-- should be EntityStakeholderService
  EntityStakeholderHistoryService,
  StakeholderSyncService,
]
```

## Root Cause of Blocker #1: Runtime ES-Module Load-Order Cycle

This is a **CommonJS load-order cycle**, not a NestJS DI cycle. When file A imports file B, Node must finish executing B before A continues. If B (directly or transitively) imports A back **while A is still executing**, Node returns A's **partial** `module.exports` ŌĆö so a value imported from A at that instant is `undefined`. `forwardRef()` cannot fix this: by the time Nest reads the `@Module({ providers: [...] })` metadata, the slot is already frozen as `undefined`.

### The Cycle

```
entity-stakeholder.service.ts        (line 63) imports
  ŌåÆ unit-access.service.ts           (line 26) imports the `@/modules/user` BARREL
    ŌåÆ user/index.ts                  `export * from './user.module' | './user.controller'`
      ŌåÆ user.controller.ts           imports 5 CRM services + the `../auth` barrel
        ŌåÆ commission-payment.service.ts / lead.service.ts / deal.service.ts /
          contact.service.ts / entity-transfer.service.ts  (and, via `../auth` ŌåÆ
          auth.module ŌåÆ rbac.module ŌåÆ ŌĆ” ŌåÆ commission-payment.module)
            ŌåÆ back into entity-stakeholder.service.ts / entity-stakeholder.module.ts
```

<Info>
The amplifier is the **barrel**: `unit-access.service.ts` only needs the `User` *entity*, but importing `@/modules/user` pulls in `user.controller.ts` (a heavy DI hub that imports `commission-payment`, `lead`, `deal`, `contact`, `entity-transfer` services + the `auth` barrel). Because `unit-access.service.ts` is itself loaded by `entity-stakeholder.service.ts`, that whole graph becomes part of `EntityStakeholderService`'s transitive load and loops back.
</Info>

### Why It Is Load-Order-Sensitive (Latent)

`EntityStakeholderModule.providers[0]` is only `undefined` when a file that **starts** the cycle is loaded before `entity-stakeholder.module.ts`:

| Entry point | `providers[0]` |
|-------------|----------------|
| `entity-stakeholder.module.ts` loaded first (Ōēł `AppModule` / `pnpm start`) | `EntityStakeholderService` Ō£ģ |
| `search-backfill.service.ts` (what `test/helpers/search-e2e.ts` top-imports) | `UNDEFINED` ŌØī |
| `commission-payment.service.ts` loaded first | `UNDEFINED` ŌØī |

The search suite is the first place in the repo to top-import `SearchBackfillService` (which top-imports `ContactService`/`CompanyService`/`DealService`/`LeadService`), so it is the first caller to pre-walk CRM services in the order that exposes the latent cycle.

## Why `forwardRef()` Doesn't Help

`forwardRef()` defers a **token lookup inside Nest's DI container**; it assumes the class reference is *defined* at decorator-evaluation time. Here the class reference is `undefined` because the ES-module cycle returned partial exports. There is nothing for the `forwardRef` thunk to resolve. The cycle has to be broken at the **file-import** layer.

## Why "Option A" Failed (Token + `import type`)

The first draft proposed: make `commission-payment.service.ts` import `EntityStakeholderService` as a **type-only** import and inject it via an `ENTITY_STAKEHOLDER_SERVICE` symbol token.

<Warning>
**This was tested and it does NOT fix the bug.** With `commission-payment.service.ts` using `import type`, `EntityStakeholderModule.providers[0]` is **still `undefined`** under the search entry point.
</Warning>

**Why:** the slot is poisoned by the **module/controller graph**, not by the `commission-payment.service ŌåÆ entity-stakeholder.service` edge:

- `commission-payment.module.ts` (line 23) imports `EntityStakeholderModule` **directly**, and
- `user.controller.ts` imports **five** services that each import `entity-stakeholder.service.ts`.

All of those re-enter `entity-stakeholder.module.ts` / `EntityStakeholderService` while `entity-stakeholder.service.ts` is still mid-load. Option A removes one *consumer* edge but leaves the actual poisoning path intact.

## The Verified Fix for Blocker #1

<Steps>
  <Step title="Import the User entity directly">
    Import the `User` **entity** directly instead of the `@/modules/user` **barrel**:

    ```typescript src/modules/real-estate/inventory/unit-access/unit-access.service.ts
    - import { User } from '@/modules/user';
    + import { User } from '@/modules/user/user.entity';
    ```

    This severs `user.controller.ts` (and the entire downstream CRM-service / `auth` graph) from `unit-access.service.ts`'s import tree, so `EntityStakeholderService` is no longer re-entered mid-load.
  </Step>

  <Step title="Verify the fix">
    Verified: `providers[0]` becomes `EntityStakeholderService` under **both** the search entry point and the `commission-payment`-first order, and loading `unit-access.service.ts` no longer pulls in `user.controller.ts`.
  </Step>
</Steps>

<Tip>
This also matches the existing convention ŌĆö `entity-stakeholder.service.ts` itself imports `User` from `../../user/user.entity` directly. The barrel import was the inconsistency.
</Tip>

### Hardening (Same Class of Bug)

Removing the controller re-export from `user/index.ts` would be a **no-op** (the barrel also `export *`s `user.module`, which imports the controller anyway). The effective hardening is to stop **services/DTOs** from importing the heavy barrel. The remaining barrel importers were redirected to the leaf DTO module:

```typescript
// event.service.ts, notification.dto.ts, notification-preference.dto.ts
- import { UserDto } from '@/modules/user';            // (or UserDirectoryDto)
+ import { UserDto } from '@/modules/user/user.dto';
```

<Check>
After this, **no file under `src/` imports the `@/modules/user` barrel**, closing the whole class of "service/DTO ŌåÆ user barrel ŌåÆ controller hub" cycles.
</Check>

## Blocker #2: `LeadAiService` Missing Provider

Once blocker #1 was cleared, boot advanced and hit:

```bash
Nest can't resolve dependencies of the LeadAiToolsProvider (?).
Please make sure that the argument LeadAiService at index [0] is available in the LeadModule module.
```

<Info>
This is **not** a cycle and **not** load-order-dependent. `LeadAiService` (a normal `@Injectable()` in `src/modules/crm/lead/lead-ai.service.ts`) is **injected** by three consumers ŌĆö `LeadAiToolsProvider` (registered in both `LeadModule` and `AiAssistantModule`), `AiContextBuilderService`, and `AiAssistantOrchestratorService` ŌĆö but it was registered as a provider in **no module**.
</Info>

A normal-order `AppModule` compile (no search pre-walk) fails with the identical error, so the backend could not boot at all on this branch.

### Fix for Blocker #2

```typescript src/modules/crm/lead/lead.module.ts
import { LeadAiService } from './lead-ai.service';

@Module({
  providers: [ LeadService, LeadAiService, ... ],
  exports:   [ MikroOrmModule, LeadService, LeadAiService, ... ]
})
```

`LeadModule` provides it (for its own `LeadAiToolsProvider`) and **exports** it so `AiAssistantModule` (which `forwardRef`-imports `LeadModule`) can resolve it for its own `LeadAiToolsProvider` / `AiContextBuilderService` / `AiAssistantOrchestratorService`. No new import cycle is introduced (the module already transitively imported `lead-ai.service.ts` via `lead-ai-tools.provider.ts`; the `madge` guard confirms 0 new cycles).

## Regression Guard: Madge Baseline-Delta

A hard "zero cycles" gate is **not viable**: the codebase already has ~156 file-level cycles, the overwhelming majority of which are intentional NestJS `*.module.ts` Ōåö `*.module.ts` relationships resolved by `forwardRef()`. So instead of failing on all of them, a baseline-delta guard fails only on **new** runtime cycles:

<Steps>
  <Step title="Run madge with skipTypeImports">
    `scripts/check-circular-deps.ts` runs `madge` with `skipTypeImports` (type-only imports are erased at runtime and cannot cause a load-order cycle), canonicalizes each cycle, and diffs against `scripts/circular-deps.baseline.json`.
  </Step>

  <Step title="Check for new cycles">
    Run the check:
    ```bash
    pnpm lint:circular
    ```
    Fails if a cycle appears that is not in the baseline.
  </Step>

  <Step title="Update baseline (after review)">
    ```bash
    pnpm lint:circular:update
    ```
    <Warning>Use only after review.</Warning>
  </Step>
</Steps>

<Check>
Validated: temporarily restoring the `unit-access` barrel import makes the guard fail and prints the exact new cycle (`ŌĆ” entity-stakeholder.service.ts > unit-access.service.ts > user/index.ts > user.controller.ts ŌĆ”`).
</Check>
