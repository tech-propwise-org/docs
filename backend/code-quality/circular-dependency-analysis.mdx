## Overview

This document details the investigation and resolution of two critical boot-blocking issues discovered during E2E testing of the search queue functionality.

| # | Blocker | Class | Fix that actually works |
|---|---------|-------|------------------------|
| 1 | `EntityStakeholderModule.providers[0]` is `undefined` → `CircularDependencyException` | Runtime ES-module (CommonJS) load-order cycle | `unit-access.service.ts` imports `User` from `@/modules/user/user.entity` instead of the `@/modules/user` **barrel** (1 line) |
| 2 | `Nest can't resolve … LeadAiToolsProvider … LeadAiService at index [0]` | **Missing provider** (not a cycle) | Register `LeadAiService` in `LeadModule` `providers` + `exports` |

<Warning>
**Important correction:** The first draft recommended breaking `commission-payment.service.ts → entity-stakeholder.service.ts` with a DI token + `import type` ("Option A"). **That fix does not work** — see the "Why Option A Failed" section below.
</Warning>

## Symptom

The `pnpm test:e2e:queue:search` command failed during Nest test-app boot with:

```bash
A circular dependency has been detected inside EntityStakeholderModule.
Please, make sure that each side of a bidirectional relationships are decorated with "forwardRef()".
```

`NestContainer.addProvider()` throws this when it receives `undefined` as a provider. Runtime inspection of `EntityStakeholderModule`'s provider list confirmed the issue:

```typescript
Providers: [
  undefined,                       // <-- should be EntityStakeholderService
  EntityStakeholderHistoryService,
  StakeholderSyncService,
]
```

## Root Cause: Blocker #1 (Runtime ES-Module Cycle)

### Understanding the Issue

This is a **CommonJS load-order cycle**, not a NestJS DI cycle. When file A imports file B, Node must finish executing B before A continues. If B (directly or transitively) imports A back **while A is still executing**, Node returns A's **partial** `module.exports` — so a value imported from A at that instant is `undefined`.

<Info>
`forwardRef()` cannot fix this: by the time Nest reads the `@Module({ providers: [...] })` metadata, the slot is already frozen as `undefined`.
</Info>

### The Cycle Path

```
entity-stakeholder.service.ts        (line 63) imports
  → unit-access.service.ts           (line 26) imports the `@/modules/user` BARREL
    → user/index.ts                  `export * from './user.module' | './user.controller'`
      → user.controller.ts           imports 5 CRM services + the `../auth` barrel
        → commission-payment.service.ts / lead.service.ts / deal.service.ts /
          contact.service.ts / entity-transfer.service.ts
            → back into entity-stakeholder.service.ts / entity-stakeholder.module.ts
```

### The Amplifier: Barrel Exports

The amplifier is the **barrel**: `unit-access.service.ts` only needs the `User` *entity*, but importing `@/modules/user` pulls in `user.controller.ts` (a heavy DI hub that imports `commission-payment`, `lead`, `deal`, `contact`, `entity-transfer` services + the `auth` barrel).

Because `unit-access.service.ts` is itself loaded by `entity-stakeholder.service.ts`, that whole graph becomes part of `EntityStakeholderService`'s transitive load and loops back.

### Load-Order Sensitivity

`EntityStakeholderModule.providers[0]` is only `undefined` when a file that **starts** the cycle is loaded before `entity-stakeholder.module.ts`:

| Entry point | `providers[0]` |
|-------------|----------------|
| `entity-stakeholder.module.ts` loaded first (≈ `AppModule` / `pnpm start`) | `EntityStakeholderService` ✅ |
| `search-backfill.service.ts` (what `test/helpers/search-e2e.ts` top-imports) | `UNDEFINED` ❌ |
| `commission-payment.service.ts` loaded first | `UNDEFINED` ❌ |

<Tip>
The search suite is the first place in the repo to top-import `SearchBackfillService` (which top-imports `ContactService`/`CompanyService`/`DealService`/`LeadService`), so it is the first caller to pre-walk CRM services in the order that exposes the latent cycle.
</Tip>

## Why `forwardRef()` Doesn't Help

`forwardRef()` defers a **token lookup inside Nest's DI container**; it assumes the class reference is *defined* at decorator-evaluation time. Here the class reference is `undefined` because the ES-module cycle returned partial exports. There is nothing for the `forwardRef` thunk to resolve.

<Warning>
The cycle must be broken at the **file-import** layer, not the DI layer.
</Warning>

## Why Option A Failed (Token + `import type`)

The first draft proposed: make `commission-payment.service.ts` import `EntityStakeholderService` as a **type-only** import and inject it via an `ENTITY_STAKEHOLDER_SERVICE` symbol token.

### Evidence of Failure

**This was tested and it does NOT fix the bug.** With `commission-payment.service.ts` using `import type`, `EntityStakeholderModule.providers[0]` is **still `undefined`** under the search entry point.

### Why It Failed

The slot is poisoned by the **module/controller graph**, not by the `commission-payment.service → entity-stakeholder.service` edge:

- `commission-payment.module.ts` (line 23) imports `EntityStakeholderModule` **directly**
- `user.controller.ts` imports **five** services that each import `entity-stakeholder.service.ts`

All of those re-enter `entity-stakeholder.module.ts` / `EntityStakeholderService` while `entity-stakeholder.service.ts` is still mid-load. Option A removes one *consumer* edge but leaves the actual poisoning path intact.

## The Verified Fix for Blocker #1

### Primary Fix (1 Line)

Import the `User` **entity** directly instead of the `@/modules/user` **barrel**:

```typescript src/modules/real-estate/inventory/unit-access/unit-access.service.ts
- import { User } from '@/modules/user';
+ import { User } from '@/modules/user/user.entity';
```

<Check>
This severs `user.controller.ts` (and the entire downstream CRM-service / `auth` graph) from `unit-access.service.ts`'s import tree, so `EntityStakeholderService` is no longer re-entered mid-load.
</Check>

**Verification:** `providers[0]` becomes `EntityStakeholderService` under **both** the search entry point and the `commission-payment`-first order, and loading `unit-access.service.ts` no longer pulls in `user.controller.ts`.

<Info>
This also matches the existing convention — `entity-stakeholder.service.ts` itself imports `User` from `../../user/user.entity` directly. The barrel import was the inconsistency.
</Info>

### Hardening (Same Class of Bug)

Removing the controller re-export from `user/index.ts` would be a **no-op** (the barrel also `export *`s `user.module`, which imports the controller anyway). The effective hardening is to stop **services/DTOs** from importing the heavy barrel.

The remaining barrel importers were redirected to the leaf DTO module:

```typescript Multiple Files
// event.service.ts, notification.dto.ts, notification-preference.dto.ts
- import { UserDto } from '@/modules/user';            // (or UserDirectoryDto)
+ import { UserDto } from '@/modules/user/user.dto';
```

<Check>
After this, **no file under `src/` imports the `@/modules/user` barrel**, closing the whole class of "service/DTO → user barrel → controller hub" cycles.
</Check>

## Root Cause: Blocker #2 (Missing Provider)

Once blocker #1 was cleared, boot advanced and hit:

```bash
Nest can't resolve dependencies of the LeadAiToolsProvider (?).
Please make sure that the argument LeadAiService at index [0] is available in the LeadModule module.
```

<Warning>
This is **not** a cycle and **not** load-order-dependent. `LeadAiService` is **injected** by three consumers but was registered as a provider in **no module**.
</Warning>

### The Issue

`LeadAiService` (a normal `@Injectable()` in `src/modules/crm/lead/lead-ai.service.ts`) is injected by:

- `LeadAiToolsProvider` (registered in both `LeadModule` and `AiAssistantModule`)
- `AiContextBuilderService`
- `AiAssistantOrchestratorService`

A normal-order `AppModule` compile (no search pre-walk) fails with the identical error, so the backend could not boot at all on this branch.

### The Fix for Blocker #2

```typescript src/modules/crm/lead/lead.module.ts
import { LeadAiService } from './lead-ai.service';

@Module({
  // ...
  providers: [ LeadService, LeadAiService, /* ... */ ],
  exports:   [ MikroOrmModule, LeadService, LeadAiService, /* ... */ ]
})
```

<Info>
`LeadModule` provides it (for its own `LeadAiToolsProvider`) and **exports** it so `AiAssistantModule` (which `forwardRef`-imports `LeadModule`) can resolve it for its own consumers.
</Info>

<Check>
No new import cycle is introduced — the module already transitively imported `lead-ai.service.ts` via `lead-ai-tools.provider.ts`. The `madge` guard confirms 0 new cycles.
</Check>

## Regression Guard: Madge Baseline-Delta

A hard "zero cycles" gate is **not viable**: the codebase already has ~156 file-level cycles, the overwhelming majority of which are intentional NestJS `*.module.ts` ↔ `*.module.ts` relationships resolved by `forwardRef()`.

### Implementation

Instead of failing on all cycles, a baseline-delta guard fails only on **new** runtime cycles:

- `scripts/check-circular-deps.ts` runs `madge` with `skipTypeImports` (type-only imports are erased at runtime and cannot cause a load-order cycle)
- Canonicalizes each cycle and diffs against `scripts/circular-deps.baseline.json`

### Commands

<CodeGroup>

```bash Check for New Cycles
pnpm lint:circular
```

```bash Update Baseline (After Review)
pnpm lint:circular:update
```

</CodeGroup>

<Tip>
**Validation:** Temporarily restoring the `unit-access` barrel import makes the guard fail and prints the exact new cycle:

```
… entity-stakeholder.service.ts > unit-access.service.ts > 
user/index.ts > user.controller.ts …
```
</Tip>

## Key Takeaways

<CardGroup cols={2}>

<Card title="Barrel Exports Are Dangerous" icon="warning">
Importing from barrel files (`index.ts`) in services can create massive transitive dependency graphs that introduce latent circular dependencies.
</Card>

<Card title="Type-Only Imports Aren't Magic" icon="circle-info">
`import type` only helps if the problematic import is itself the cycle entry point. If the cycle exists in the module/controller graph, type-only imports won't fix it.
</Card>

<Card title="Load Order Matters" icon="arrow-right-arrow-left">
ES module cycles are load-order-dependent. A cycle may only manifest under specific entry points (like E2E test suites with different import patterns).
</Card>

<Card title="Baseline-Delta Guards" icon="shield-check">
In large codebases with intentional cycles (NestJS modules), prevent regressions with baseline-delta guards instead of "zero tolerance" rules.
</Card>

</CardGroup>

## Best Practices

<Steps>

<Step title="Import Leaf Entities Directly">
Services should import entities directly from their `.entity.ts` files rather than from barrel exports that may pull in controllers and other heavy dependencies.

```typescript
// Good
import { User } from '@/modules/user/user.entity';

// Bad
import { User } from '@/modules/user';  // Pulls in controller, module, etc.
```
</Step>

<Step title="Verify Provider Registration">
If you create a new injectable service, ensure it's registered in the appropriate module's `providers` array. If other modules need it, add it to `exports`.
</Step>

<Step title="Run Circular Dependency Checks">
Before committing, run `pnpm lint:circular` to catch new runtime cycles. Only update the baseline after careful review.
</Step>

<Step title="Test Multiple Entry Points">
Circular dependencies can be latent and only manifest under specific load orders. Test your changes with different entry points (app start, different test suites, etc.).
</Step>

</Steps>
