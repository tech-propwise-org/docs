The search-suite experienced boot failures due to ES-module load-order cycles. This analysis documents the root causes and verified fixes for both blocking issues.

## Symptoms

`pnpm test:e2e:queue:search` failed during Nest test-app boot with:

```bash
A circular dependency has been detected inside EntityStakeholderModule.
Please, make sure that each side of a bidirectional relationships are decorated with "forwardRef()".
```

Investigation revealed that `NestContainer.addProvider()` throws this when it receives `undefined` as a provider. The `EntityStakeholderModule`'s provider list at runtime confirmed the issue:

```typescript
Providers: [
  undefined,                       // <-- should be EntityStakeholderService
  EntityStakeholderHistoryService,
  StakeholderSyncService,
]
```

## Root cause analysis

### Blocker #1: Runtime ES-module load-order cycle

This is a **CommonJS load-order cycle**, not a NestJS DI cycle. When file A imports file B, Node must finish executing B before A continues. If B (directly or transitively) imports A back **while A is still executing**, Node returns A's **partial** `module.exports` — so a value imported from A at that instant is `undefined`.

<Info>
`forwardRef()` cannot fix this: by the time Nest reads the `@Module({ providers: [...] })` metadata, the slot is already frozen as `undefined`.
</Info>

#### The cycle path

```
entity-stakeholder.service.ts        (line 63) imports
  → unit-access.service.ts           (line 26) imports the `@/modules/user` BARREL
    → user/index.ts                  `export * from './user.module' | './user.controller'`
      → user.controller.ts           imports 5 CRM services + the `../auth` barrel
        → commission-payment.service.ts / lead.service.ts / deal.service.ts /
          contact.service.ts / entity-transfer.service.ts
            → back into entity-stakeholder.service.ts / entity-stakeholder.module.ts
```

The amplifier is the **barrel**: `unit-access.service.ts` only needs the `User` *entity*, but importing `@/modules/user` pulls in `user.controller.ts` (a heavy DI hub that imports multiple services).

#### Load-order sensitivity

`EntityStakeholderModule.providers[0]` is only `undefined` when a file that **starts** the cycle is loaded before `entity-stakeholder.module.ts`:

| Entry point | `providers[0]` |
| --- | --- |
| `entity-stakeholder.module.ts` loaded first (≈ `AppModule` / `pnpm start`) | `EntityStakeholderService` ✅ |
| `search-backfill.service.ts` (what `test/helpers/search-e2e.ts` top-imports) | `UNDEFINED` ❌ |
| `commission-payment.service.ts` loaded first | `UNDEFINED` ❌ |

### Why existing solutions don't work

#### ForwardRef limitation

`forwardRef()` defers a **token lookup inside Nest's DI container**; it assumes the class reference is *defined* at decorator-evaluation time. Here the class reference is `undefined` because the ES-module cycle returned partial exports.

#### Why "Option A" was rejected

<Warning>
The token + `import type` approach was tested and **does NOT fix the bug**. With `commission-payment.service.ts` using `import type`, `EntityStakeholderModule.providers[0]` is **still `undefined`** under the search entry point.
</Warning>

**Why it fails:** the slot is poisoned by the **module/controller graph**, not by the `commission-payment.service → entity-stakeholder.service` edge:

- `commission-payment.module.ts` imports `EntityStakeholderModule` **directly**
- `user.controller.ts` imports **five** services that each import `entity-stakeholder.service.ts`

## Verified fixes

### Fix for blocker #1

<Check>
Import the `User` **entity** directly instead of the `@/modules/user` **barrel**:
</Check>

```typescript
// src/modules/real-estate/inventory/unit-access/unit-access.service.ts
- import { User } from '@/modules/user';
+ import { User } from '@/modules/user/user.entity';
```

This severs `user.controller.ts` from `unit-access.service.ts`'s import tree, preventing `EntityStakeholderService` from being re-entered mid-load.

#### Additional hardening

Remove barrel imports from other service files:

```typescript
// event.service.ts, notification.dto.ts, notification-preference.dto.ts
- import { UserDto } from '@/modules/user';            // (or UserDirectoryDto)
+ import { UserDto } from '@/modules/user/user.dto';
```

After this change, **no file under `src/` imports the `@/modules/user` barrel**, closing the whole class of "service/DTO → user barrel → controller hub" cycles.

### Fix for blocker #2

<Info>
Once blocker #1 was cleared, boot advanced and hit a missing provider error for `LeadAiService`.
</Info>

This is **not** a cycle issue. `LeadAiService` was injected by three consumers but registered as a provider in **no module**.

#### The fix

```typescript
// src/modules/crm/lead/lead.module.ts
import { LeadAiService } from './lead-ai.service';

@Module({
  providers: [ LeadService, LeadAiService, ... ],
  exports:   [ MikroOrmModule, LeadService, LeadAiService, ... ]
})
```

`LeadModule` provides it and **exports** it so `AiAssistantModule` can resolve it for its consumers.

## Regression guard

A baseline-delta CI guard prevents new runtime cycles:

<Steps>
  <Step title="Baseline approach">
    Hard "zero cycles" gate is not viable due to ~156 existing file-level cycles (mostly intentional NestJS module relationships resolved by `forwardRef()`).
  </Step>
  
  <Step title="Implementation">
    `scripts/check-circular-deps.ts` runs `madge` with `skipTypeImports`, canonicalizes cycles, and diffs against `scripts/circular-deps.baseline.json`.
  </Step>
  
  <Step title="Commands">
    - `pnpm lint:circular` — fails if new cycles appear
    - `pnpm lint:circular:update` — regenerate baseline after review
  </Step>
</Steps>

<Tip>
Type-only imports are erased at runtime and cannot cause load-order cycles, so they're safely excluded from the analysis.
</Tip>
