This document details the root cause analysis and verified fixes for two critical boot blockers discovered while running the search suite end-to-end tests.

<Warning>
The originally proposed fix using DI tokens and `import type` was tested and **does not work**. This document reflects the actual verified solution.
</Warning>

## Executive Summary

Two independent boot blockers were discovered and resolved on 2026-05-29:

<CardGroup cols={2}>
  <Card title="Blocker #1" icon="circular-arrow">
    **EntityStakeholderModule** circular dependency causing `undefined` provider
  </Card>
  <Card title="Blocker #2" icon="exclamation-triangle">
    **LeadAiService** missing provider registration in LeadModule
  </Card>

| Issue | Root Cause | Solution |
| --- | --- | --- |
| EntityStakeholderModule.providers[0] is `undefined` | Runtime ES-module load-order cycle | Import `User` entity directly instead of barrel export |
| LeadAiService dependency resolution fails | Missing provider registration | Register LeadAiService in LeadModule providers and exports |

## Root Cause: ES-Module Load-Order Cycle

<Note>
This is a **CommonJS load-order cycle**, not a NestJS DI cycle. `forwardRef()` cannot fix this type of issue.
</Note>

### The Dependency Chain

The circular dependency follows this path:

```
entity-stakeholder.service.ts
  → unit-access.service.ts
    → @/modules/user barrel (index.ts)
      → user.controller.ts
        → commission-payment.service.ts + other CRM services
          → back to entity-stakeholder.service.ts
```

### Why Load Order Matters

The issue is latent and depends on which file initiates the loading:

| Entry Point | Result |
| --- | --- |
| `entity-stakeholder.module.ts` loaded first | ✅ `EntityStakeholderService` |
| `search-backfill.service.ts` loaded first | ❌ `UNDEFINED` |
| `commission-payment.service.ts` loaded first | ❌ `UNDEFINED` |

<Info>
The search suite is the first code path to import `SearchBackfillService`, which pre-walks CRM services in an order that exposes this latent cycle.
</Info>

## Why Common Solutions Don't Work

<AccordionGroup>
  <Accordion title="forwardRef() Limitation">
    `forwardRef()` defers token lookup inside Nest's DI container, but requires the class reference to be defined. Here, the class reference is `undefined` due to the ES-module cycle returning partial exports.
  </Accordion>

  <Accordion title="Token + import type Approach">
    **Tested and verified as ineffective.** Using `import type` for `EntityStakeholderService` in `commission-payment.service.ts` and injecting via token still results in `undefined` providers because:
    
    - `commission-payment.module.ts` directly imports `EntityStakeholderModule`
    - `user.controller.ts` imports five services that each import `entity-stakeholder.service.ts`
    
    The module/controller graph still re-enters `entity-stakeholder.module.ts` during load.
  </Accordion>
</AccordionGroup>

## Verified Solutions

### Fix #1: Direct Entity Import

<CodeGroup>
```ts src/modules/real-estate/inventory/unit-access/unit-access.service.ts
// Before
import { User } from '@/modules/user';

// After
import { User } from '@/modules/user/user.entity';
```
</CodeGroup>

<Check>
This change severs the connection to `user.controller.ts` and the downstream CRM service graph, preventing `EntityStakeholderService` from being re-entered during load.
</Check>

### Fix #2: LeadAiService Registration

<CodeGroup>
```ts src/modules/crm/lead/lead.module.ts
import { LeadAiService } from './lead-ai.service';

@Module({
  // ... existing imports
  providers: [
    LeadService, 
    LeadAiService,  // Add this
    // ... other providers
  ],
  exports: [
    MikroOrmModule, 
    LeadService, 
    LeadAiService,  // Add this
    // ... other exports
  ],
})
export class LeadModule {}
```
</CodeGroup>

### Additional Hardening

To prevent similar issues, redirect remaining barrel imports to specific modules:

<CodeGroup>
```ts event.service.ts, notification.dto.ts, notification-preference.dto.ts
// Before
import { UserDto } from '@/modules/user';

// After  
import { UserDto } from '@/modules/user/user.dto';
```
</CodeGroup>

<Tip>
After these changes, no file under `src/` imports the `@/modules/user` barrel, closing the entire class of "service/DTO → user barrel → controller hub" cycles.
</Tip>

## Regression Prevention

A `madge`-based baseline-delta CI guard prevents new runtime cycles:

<Steps>
  <Step title="Cycle Detection">
    `scripts/check-circular-deps.ts` runs `madge` with `skipTypeImports` to identify runtime cycles only
  </Step>

  <Step title="Baseline Comparison">
    Compare against `scripts/circular-deps.baseline.json` to identify new cycles
  </Step>

  <Step title="CI Integration">
    - `pnpm lint:circular` - fails on new cycles
    - `pnpm lint:circular:update` - regenerate baseline after review
  </Step>
</Steps>

<Warning>
A "zero cycles" gate isn't viable due to ~156 existing intentional NestJS module relationships resolved by `forwardRef()`. The baseline approach catches only new problematic cycles.
</Warning>
