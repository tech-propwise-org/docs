<Warning>
**Status:** Project-wide standard — MANDATORY for all entities, services, and DTOs. This document is the authoritative reference for the soft-delete / archive filter pattern.
</Warning>

This project uses **MikroORM v6** with `loadStrategy: JOINED`, `autoJoinRefsForFilters: true`, and `filtersOnRelations: true`. Combined with our `isDeleted` and `active` entity filters, this produces a **silent data-loss bug** if entities are not declared correctly.

<Note>
**Critical Bug Pattern:** A row whose own `is_deleted = false` will silently disappear from every query if any of its non-nullable owner relations points at a soft-deleted target.
</Note>

## Quick Reference Rules

| What | Rule |
|------|------|
| Non-nullable `@ManyToOne` / `@OneToOne(owner)` to an entity that declares `@Filter({ name: 'isDeleted', ... })` | Add `{ filters: { isDeleted: false } }` to the relation options |
| `@Filter({ name: 'active', ... })` on a soft-deletable + archivable entity | `cond` MUST be `{ isArchived: false }` only — NEVER `{ isDeleted: false, isArchived: false }` |
| Bulk re-point inside a merge service (`em.nativeUpdate(...)` that re-points children of an entity that is about to be soft-deleted) | Pass `{ filters: false }` so the re-point hits archived AND soft-deleted children |
| `{ filters: false }` anywhere else | Allowed only in explicit-business-intent categories (merge / restore / conflict-detection / audit-timeline / notification fan-out / admin / seed). Every call site MUST have a code comment naming the category. |
| User reference on a new entity | Classify per user reference tiers — `createdBy`/`updatedBy`/`archivedBy`/etc. are Tier 1 (audit ref); `assignedTo`/`owner`/`recipient`/`user` are Tier 2 or 3 |
| DTO with a Tier-3 owner relation (`person`, `organization`) | If `populated.isDeleted === true` → DTO MUST use placeholder / omit / 4xx. NEVER serialize the full Tier-3 payload when soft-deleted |
| New entity with non-nullable owners | Update the audit table + extend `DataIntegrityAuditService` for Tier 2/3 + add the two mandatory tests |

<Warning>
If you skip any of these rules, the regression will not show up in unit tests — it surfaces as "this row exists in the DB but the API returns nothing" weeks later.
</Warning>

## Root Cause Analysis

### Why Rows Silently Disappear

MikroORM v6 has three defaults that interact badly with soft-deletable graphs:

```typescript
// mikro-orm.config.ts (defaults — NOT to be disabled globally)
loadStrategy: LoadStrategy.JOINED; // populated relations are joined into the main SELECT
autoJoinRefsForFilters: true; // auto-add :ref JOIN for any relation whose target has filters
filtersOnRelations: true; // apply target-entity filters via JOIN ON
```

For every M:1 / 1:1-owner relation on the queried entity, MikroORM checks whether the target entity declares any default-enabled filters. If so, it adds an auto `:ref` JOIN to that relation **even if you did not populate it**, so the filter can be applied as a JOIN `ON` condition.

<Info>
For a **non-nullable** M:1 the resulting JOIN type is `INNER JOIN`. This means if the target is soft-deleted, the entire row disappears from results.
</Info>

### The Failure Mode

`em.find(Lead, { id: leadId })` becomes:

```sql
SELECT lead.*
  FROM lead
  INNER JOIN person p ON p.id = lead.person_id AND p.is_deleted = false
  INNER JOIN organization o ON o.id = lead.organization_id AND o.is_deleted = false
 WHERE lead.id = $1
   AND lead.is_deleted = false;
```

If **any** of those non-nullable joined targets fails the filter (e.g. `Person.isDeleted = true`), the **lead row is silently dropped** — even though `lead.is_deleted = false`.

### Real Production Examples

<AccordionGroup>
<Accordion title="Person Merge Scenario">
Two persons merged → secondary `Person.isDeleted = true` → an archived Lead still pointing at the secondary Person. Result: The archived Lead was invisible everywhere even though it existed in the DB.
</Accordion>

<Accordion title="DealStage Cleanup">
`DealStage` cleanup → custom stage soft-deleted while a Deal still pointed at it. Result: The Deal vanished from list endpoints.
</Accordion>

<Accordion title="Agent Departure">
Agent leaves the org → `User.isDeleted = true` → every Deal / Contact / Company / Listing they `createdBy` would have vanished if we shipped the relation override blindly without the Tier-1 carve-out.
</Accordion>
</AccordionGroup>

### Special Case: excludeAiServiceUser Filter

The `excludeAiServiceUser` filter is **OPT-IN** (`default: false`) to prevent the same issue with service accounts:

```typescript
@Filter({
  name: 'excludeAiServiceUser',
  default: false,
  cond: { isAiServiceUser: false, isSystemServiceUser: false },
})
```

<Note>
This filter was originally `default: true` but caused service-stamped rows to disappear. The solution was to invert the default and explicitly enable it where needed.
</Note>

## The Three-Part Fix

The fix has **three coordinated parts**. Skipping any one re-introduces the bug.

### Part 1: Per-Relation Filter Override (MANDATORY)

<Steps>
<Step title="Declare Filter Override">
Add `{ filters: { isDeleted: false } }` on **every non-nullable** `@ManyToOne` / `@OneToOne(owner)` relation to a soft-deletable entity:

```typescript
@Entity()
export class Lead extends BaseEntity {
  @ManyToOne(() => Person, { 
    nullable: false,
    filters: { isDeleted: false } // MANDATORY
  })
  person!: Ref<Person>;

  @ManyToOne(() => Organization, { 
    nullable: false,
    filters: { isDeleted: false } // MANDATORY
  })
  organization!: Ref<Organization>;
}
```
</Step>

<Step title="Apply to All Entity Types">
This applies to:
- Domain entities (`Lead`, `Deal`, `Contact`, etc.)
- Junction/bridge entities (`PersonOrganization`, `UserOrgRole`)  
- Event/audit entities (`ContactChangeEvent`, `DealProgressEvent`)
- Configuration entities (`CustomField`, `Webhook`, `EmailTemplate`)
</Step>
</Steps>

### Part 2: User Reference Classification

User references are classified into three tiers based on business intent:

<Tabs>
<Tab title="Tier 1: Audit References">
**No filter override needed** - These are audit trails that must survive user soft-deletion:

```typescript
@ManyToOne(() => User, { nullable: false })
createdBy!: Ref<User>; // NO filters override

@ManyToOne(() => User, { nullable: true })
updatedBy?: Ref<User>; // NO filters override
```

Examples: `createdBy`, `updatedBy`, `archivedBy`, `deletedBy`, `changedBy`, `performedBy`, `triggeredBy`
</Tab>

<Tab title="Tier 2: Functional References">
**Requires filter override** - Business-critical assignments:

```typescript
@ManyToOne(() => User, { 
  nullable: false,
  filters: { isDeleted: false } // REQUIRED
})
assignedTo!: Ref<User>;
```

Examples: `assignedTo`, `handledBy`, `responsibleAgent`, `primaryContact`
</Tab>

<Tab title="Tier 3: Owner References">
**Requires filter override AND DTO handling** - Core business entities:

```typescript
@ManyToOne(() => Person, { 
  nullable: false,
  filters: { isDeleted: false } // REQUIRED
})
person!: Ref<Person>;
```

Examples: `person`, `organization`, `contact`, `company`, `property`
</Tab>
</Tabs>

### Part 3: DTO Soft-Delete Handling

For Tier-3 owner relations in DTOs:

<CodeGroup>
```typescript Strategy A: Placeholder
// If populated.isDeleted === true, use placeholder
if (lead.person.isDeleted) {
  dto.person = {
    id: lead.person.id,
    name: "[Deleted Person]",
    // Minimal safe fields only
  };
}
```

```typescript Strategy B: Omit
// If populated.isDeleted === true, omit from DTO
if (!lead.person.isDeleted) {
  dto.person = this.personToDto(lead.person);
}
// DTO field becomes undefined/null
```

```typescript Strategy C: HTTP 4xx
// If populated.isDeleted === true, throw error
if (lead.person.isDeleted) {
  throw new BadRequestException(
    `Cannot display lead ${leadId} - associated person was deleted`
  );
}
```
</CodeGroup>

<Warning>
**NEVER serialize the full Tier-3 payload when soft-deleted.** This prevents data leakage of sensitive information from deleted entities.
</Warning>

## Implementation Requirements

### For New Entities

<Steps>
<Step title="Add Filter Overrides">
Follow the relation override rules above for all non-nullable references to soft-deletable entities.
</Step>

<Step title="Update Audit Systems">
- Update the audit table in query optimization patterns documentation
- Extend `DataIntegrityAuditService` for Tier 2/3 relations
- The daily cron will pick up the new entity automatically
</Step>

<Step title="Add Mandatory Tests">
Add these two tests to prevent regressions:

```typescript
describe('Soft-delete filter compliance', () => {
  it('should not drop rows when non-nullable owner is soft-deleted', async () => {
    // Create entity with owner
    // Soft-delete the owner
    // Verify entity is still findable
  });

  it('should handle archived state independently of soft-delete', async () => {
    // Test that isArchived filter works correctly
    // When owner is soft-deleted but entity is not archived
  });
});
```
</Step>
</Steps>

### Active Filter Pattern

For entities that support both soft-delete AND archiving:

```typescript
@Filter({
  name: 'active',
  default: true,
  cond: { isArchived: false } // ONLY isArchived, NEVER include isDeleted
})
```

<Warning>
The `active` filter condition MUST be `{ isArchived: false }` only — NEVER `{ isDeleted: false, isArchived: false }`. The `isDeleted` filtering is handled by the separate `isDeleted` filter.
</Warning>

### Bulk Operations

For bulk re-pointing operations in merge services:

```typescript
// When moving children before soft-deleting parent
await em.nativeUpdate(
  ChildEntity,
  { parent: oldParent },
  { parent: newParent },
  { filters: false } // REQUIRED: hits archived AND soft-deleted children
);
```

<Tip>
Always add a code comment identifying the business intent category when using `{ filters: false }`.
</Tip>

## Testing Strategy

### Unit Test Limitations

<Warning>
The regression will **not** show up in unit tests that only create fresh test data. The bug only manifests when querying entities that reference previously soft-deleted targets.
</Warning>

### Required Integration Tests

For every new entity with non-nullable owner relations:

1. **Soft-delete survival test** - Verify entity remains queryable when its owner is soft-deleted
2. **Archive independence test** - Verify `isArchived` filtering works independently of soft-delete state

### Monitoring

The `DataIntegrityAuditService` runs daily to detect entities that have become unreachable due to soft-deleted references. Extend this service when adding new Tier 2/3 relations.

<Info>
This is a background audit process - do NOT add HTTP endpoints for integrity checks.
</Info>
