This document is the authoritative reference for the soft-delete/archive filter pattern. See `QUERY_OPTIMIZATION_PATTERNS.md` §5.5 for MikroORM internals deep-dive.

## Overview

This project uses **MikroORM v6** with `loadStrategy: JOINED`, `autoJoinRefsForFilters: true`, and `filtersOnRelations: true`. Combined with our `isDeleted` and `active` entity filters, this produces a **silent data-loss bug** if entities are not declared correctly.

<Note>
**Critical Issue:** A row whose own `is_deleted = false` will silently disappear from every query if any of its non-nullable owner relations points at a soft-deleted target.
</Note>

The fix is **structural** and lives on the entity declaration. Every new entity MUST follow it.

### Essential Rules

| What | Rule |
|------|------|
| Non-nullable `@ManyToOne` / `@OneToOne(owner)` to soft-deletable entity | Add `{ filters: { isDeleted: false } }` to relation options |
| `@Filter({ name: 'active' })` on soft-deletable + archivable entity | `cond` MUST be `{ isArchived: false }` only — NEVER `{ isDeleted: false, isArchived: false }` |
| Non-nullable relation to ARCHIVABLE target | Decide per §2.2.1: record/history/financial child → override BOTH `{ isDeleted: false, active: false }`; active-view junction → `{ isDeleted: false }` only |
| Bulk re-point in merge service | Pass `{ filters: false }` so update hits archived AND soft-deleted children |
| `{ filters: false }` elsewhere | Allowed only in explicit categories (§2.3). MUST have code comment naming the category |
| User reference on new entity | Classify per §3.1.1 — audit refs are Tier 1; business refs are Tier 2/3 |
| DTO with Tier-3 owner relation | If `populated.isDeleted === true` → use placeholder/omit/4xx. NEVER serialize full soft-deleted payload |
| New entity with non-nullable owners | Update audit table in `QUERY_OPTIMIZATION_PATTERNS.md` + extend `DataIntegrityAuditService` + add mandatory tests |

<Warning>
If you skip any of these rules, the regression will not show up in unit tests — it surfaces as "this row exists in the DB but the API returns nothing" weeks later.
</Warning>

## Understanding the Problem

### Why Rows Silently Disappear

MikroORM v6 (`@mikro-orm/core@6.6.7`) has three defaults that interact badly with soft-deletable graphs:

```typescript mikro-orm.config.ts
// These are defaults — NOT to be disabled globally
loadStrategy: LoadStrategy.JOINED; // populated relations are joined into main SELECT
autoJoinRefsForFilters: true; // auto-add :ref JOIN for any relation whose target has filters
filtersOnRelations: true; // apply target-entity filters via JOIN ON
```

<Steps>
  <Step title="Filter Detection">
    For every M:1 / 1:1-owner relation on the queried entity, MikroORM checks whether the target entity declares any default-enabled filters (e.g. `@Filter({ name: 'isDeleted', default: true, cond: { isDeleted: false } })`)
  </Step>
  
  <Step title="Auto-Join Creation">
    If a filter exists, MikroORM adds an auto `:ref` JOIN to that relation **even if you did not populate it**, so the filter can be applied as a JOIN `ON` condition
  </Step>
  
  <Step title="Inner Join for Non-Nullable">
    For **non-nullable** M:1, the resulting JOIN type is `INNER JOIN`:
    
    ```typescript
    const mandatoryToOneProperty =
      [ReferenceKind.MANY_TO_ONE, ReferenceKind.ONE_TO_ONE].includes(prop.kind) && !prop.nullable;
    const joinType = (hint.filter && !prop.nullable) || mandatoryToOneProperty ? INNER_JOIN : LEFT_JOIN;
    ```
  </Step>
</Steps>

#### The Failure Mode

A simple query like `em.find(Lead, { id: leadId })` becomes:

```sql
SELECT lead.*
  FROM lead
  INNER JOIN person p ON p.id = lead.person_id AND p.is_deleted = false
  INNER JOIN organization o ON o.id = lead.organization_id AND o.is_deleted = false
 WHERE lead.id = $1
   AND lead.is_deleted = false;
```

<Warning>
If **any** of those non-nullable joined targets fails the filter (e.g. `Person.isDeleted = true` because the lead's person was merged), the **lead row is silently dropped** — even though `lead.is_deleted = false`.
</Warning>

<Info>
For nullable relations the JOIN is `LEFT JOIN` and the relation hydrates to `null` instead of dropping the row (MikroORM 6.6 reverted the v6.5 "Strict relation filters" regression).
</Info>

### Special Case: excludeAiServiceUser Filter

The `User` entity declares a second filter that hides **all non-human service accounts**:

```typescript
@Filter({
  name: 'excludeAiServiceUser',
  default: false,
  cond: { isAiServiceUser: false, isSystemServiceUser: false },
})
```

<Info>
This filter was originally `default: true` but caused service-stamped rows to disappear. It's now **opt-in** (`default: false`).
</Info>

#### Service Accounts

1. **AI agent service account** (`isAiServiceUser = true`) — organization-scope AI agents act as this row
2. **System automation actor** (`isSystemServiceUser = true`) — distribution/escalation/lead-capture stamp as `createdBy`/`assignedBy`/`performedBy`

#### Resolution Strategy

<Tabs>
  <Tab title="User-Facing Enumerations">
    Must re-exclude service accounts explicitly:
    
    | Query Shape | How to Exclude |
    |------------|----------------|
    | Direct `User` root query (listings, pickers) | `{ filters: { excludeAiServiceUser: true } }` |
    | `UserOrgRole` enumeration | `user: { isAiServiceUser: false }` in `where` |
    | Raw SQL over `user_org_roles` | `JOIN "user" u ON u.id = uor.user_id AND u.is_ai_service_user = false` |
    | `QueryBuilder` user lists | `qb.andWhere({ isAiServiceUser: false, isSystemServiceUser: false })` |
    | Auth lookups | `{ email, isAiServiceUser: false, isSystemServiceUser: false }` |
  </Tab>
  
  <Tab title="Inherently Service-Free">
    No changes needed for:
    
    - Queries scoped to specific `user: <id>`
    - Queries filtered by `User.organizations` (neither account is in this M2M)
    - `TeamMembership`-based queries (neither has team membership)
    - `UserOrgRole`-based queries (System actor has no `UserOrgRole`)
  </Tab>
  
  <Tab title="DTO Exposure">
    Service accounts can surface as actors (`createdBy`/`changedBy`/`assignedBy`/`performedBy`).
    
    <Check>
    Their synthetic emails (`ai-user+<orgId>@propwise.com` / `system-user+<orgId>@propwise.com`) are **not** masked — they're non-routable internal addresses and display names ("AI USER"/"System Automation") identify them clearly.
    </Check>
  </Tab>
</Tabs>

## Implementation Standards

### Entity-Level Filter Override

<Note>
The fix is **structural**. It lives on the entity declaration, not in query code.
</Note>

#### Basic Pattern

```typescript
@Entity()
@Filter({ name: 'isDeleted', default: true, cond: { isDeleted: false } })
export class Lead extends SoftDeletableEntity {
  @ManyToOne(() => Person, { nullable: false, filters: { isDeleted: false } })
  person!: Ref<Person>;

  @ManyToOne(() => Organization, { nullable: false, filters: { isDeleted: false } })
  organization!: Ref<Organization>;

  // ... other relations
}
```

<Warning>
**NEVER** add `{ filters: false }` to a non-nullable owner relation unless you're in one of the explicit categories (§2.3). Use `{ filters: { isDeleted: false } }` instead.
</Warning>

### Archive Filter Guidelines

#### The Active Filter Definition

For entities that are both soft-deletable AND archivable:

```typescript
@Filter({
  name: 'active',
  default: true,
  cond: { isArchived: false } // ONLY isArchived, NOT isDeleted
})
```

<Tip>
The `active` filter MUST check `isArchived` only. Never combine `{ isDeleted: false, isArchived: false }` — this creates the same silent data loss when the `isDeleted` global filter is disabled.
</Tip>

#### Child Relations to Archivable Targets

When adding a non-nullable relation to an **archivable** target (one that declares `@Filter({ name: 'active' })`):

<Steps>
  <Step title="Classify the Child Entity">
    Determine whether the child should persist across the owner's archive or drop with it
  </Step>
  
  <Step title="Apply Appropriate Override">
    <Tabs>
      <Tab title="Persist Across Archive">
        **For:** Record/history/financial/document/tracker children
        
        ```typescript
        @ManyToOne(() => Deal, {
          nullable: false,
          filters: { isDeleted: false, active: false }
        })
        deal!: Ref<Deal>;
        ```
        
        <Info>
        `active: false` disables the `active` filter for this relation, so children remain visible when the owner is archived.
        </Info>
      </Tab>
      
      <Tab title="Drop on Archive">
        **For:** Active-view junction entities
        
        ```typescript
        @ManyToOne(() => Deal, {
          nullable: false,
          filters: { isDeleted: false }
        })
        deal!: Ref<Deal>;
        ```
        
        <Info>
        No `active: false` override — the child drops when owner is archived. Callers must opt into archived rows explicitly.
        </Info>
      </Tab>
    </Tabs>
  </Step>
</Steps>

<CardGroup cols={2}>

    - `DealStakeholder` (record of involvement)
    - `ActivityLog` (audit trail)

### When filters: false Is Allowed

<Warning>
`{ filters: false }` disables **all** filters globally. Use with extreme caution.
</Warning>

#### Permitted Categories

<AccordionGroup>
  <Accordion title="1. Merge/Re-point Operations">
    **Use case:** Bulk re-parenting of children when merging entities
    
    ```typescript
    // Merge service re-pointing children from source to target
    await em.nativeUpdate(
      Lead,
      { person: sourcePersonId },
      { person: targetPersonId },
      { filters: false } // MUST hit archived AND soft-deleted children
    );
    ```
    
    <Check>
    **Required comment:** `// filters: false — merge re-point must hit all children`
    </Check>
  </Accordion>
  
  <Accordion title="2. Restore Operations">
    **Use case:** Un-deleting soft-deleted entities
    
    ```typescript
    // Restore service
    const deleted = await em.findOne(
      Person,
      { id },
      { filters: false } // Must find soft-deleted row to restore
    );
    ```
    
    <Check>
    **Required comment:** `// filters: false — restore operation`
    </Check>
  </Accordion>
  
  <Accordion title="3. Conflict Detection">
    **Use case:** Checking for duplicates including soft-deleted rows
    
    ```typescript
    const existing = await em.findOne(
      Organization,
      { email: normalizedEmail },
      { filters: false } // Detect conflicts with soft-deleted orgs
    );
    ```
    
    <Check>
    **Required comment:** `// filters: false — conflict detection across all rows`
    </Check>
  </Accordion>
  
  <Accordion title="4. Audit Timeline">
    **Use case:** Historical views showing all state transitions
    
    ```typescript
    const timeline = await em.find(
      ActivityLog,
      { entityId, entityType },
      { filters: false, orderBy: { createdAt: 'ASC' } }
    );
    ```
    
    <Check>
    **Required comment:** `// filters: false — audit timeline includes deleted`
    </Check>
  </Accordion>
  
  <Accordion title="5. Notification Fan-out">
    **Use case:** Ensuring notifications reach all stakeholders
    
    ```typescript
    const stakeholders = await em.find(
      DealStakeholder,
      { deal: dealId },
      { filters: false } // Notify even if deal is archived
    );
    ```
    
    <Check>
    **Required comment:** `// filters: false — notification fan-out to all stakeholders`
    </Check>
  </Accordion>
  
  <Accordion title="6. Admin/Support Tools">
    **Use case:** Admin interfaces showing complete data
    
    ```typescript
    // Admin controller (non-tenant route)
    const allOrgs = await em.find(
      Organization,
      {},
      { filters: false } // Admin view sees all
    );
    ```
    
    <Check>
    **Required comment:** `// filters: false — admin tool`
    </Check>
  </Accordion>
  
  <Accordion title="7. Seed/Migration Scripts">
    **Use case:** Data population and schema updates
    
    ```typescript
    // Seed script
    const user = await em.findOne(
      User,
      { email: seedEmail },
      { filters: false } // Idempotent seed finds existing
    );
    ```
    
    <Check>
    **Required comment:** `// filters: false — seed/migration idempotence`
    </Check>
  </Accordion>
</AccordionGroup>

<Warning>
Every `{ filters: false }` call site MUST have a code comment naming one of these seven categories. Code review will reject uncommented usage.
</Warning>

## User Reference Patterns

### Three-Tier Classification

<Tabs>
  <Tab title="Tier 1: Audit References">
    **Never override filters.** Must work with soft-deleted users.
    
    ```typescript
    @ManyToOne(() => User, { nullable: false })
    createdBy!: Ref<User>;
    
    @ManyToOne(() => User, { nullable: true })
    updatedBy?: Ref<User>;
    
    @ManyToOne(() => User, { nullable: true })
    archivedBy?: Ref<User>;
    ```
    
    <Info>
    **Rationale:** Audit trail must remain intact. A soft-deleted user doesn't invalidate who created/modified a record.
    </Info>
    
    **Examples:**
    - `createdBy`, `updatedBy`, `deletedBy`, `archivedBy`
    - `performedBy` (activity log)
    - `assignedBy`, `completedBy` (task history)
  </Tab>
  
  <Tab title="Tier 2: Business References">
    **Add `{ filters: { isDeleted: false } }` override.**
    
    ```typescript
    @ManyToOne(() => User, {
      nullable: false,
      filters: { isDeleted: false }
    })
    assignedTo!: Ref<User>;
    ```
    
    <Info>
    **Rationale:** Business-meaningful roles must point at active users. If the user is soft-deleted, the parent entity should drop from active queries.
    </Info>
    
    **Examples:**
    - `assignedTo` (task/deal assignment)
    - `owner` (record ownership)
    - `reviewer`, `approver` (workflow roles)
  </Tab>
  
  <Tab title="Tier 3: Owner References">
    **Add `{ filters: { isDeleted: false } }` + special DTO handling.**
    
    ```typescript
    @ManyToOne(() => User, {
      nullable: false,
      filters: { isDeleted: false }
    })
    user!: Ref<User>;
    ```
    
    <Warning>
    **Critical:** DTOs must handle soft-deleted Tier-3 refs with placeholder/omit/4xx strategies (§3.3.1). Never serialize full soft-deleted payload.
    </Warning>
    
    **Examples:**
    - `user` (chat participant, note author)
    - `recipient` (email, notification)
    - Primary person/organization on transactional entities
  </Tab>
</Tabs>

### DTO Handling for Soft-Deleted References

<Steps>
  <Step title="Detection">
    Check if the populated reference is soft-deleted:
    
    ```typescript
    if (entity.user?.isDeleted) {
      // Handle soft-deleted reference
    }
    ```
  </Step>
  
  <Step title="Apply Strategy">
    Choose one of three strategies:
    
    <Tabs>
      <Tab title="Strategy A: Placeholder">
        Replace with minimal placeholder data
        
        ```typescript
        return {
          ...baseDto,
          user: entity.user?.isDeleted
            ? { id: entity.user.id, name: '[Deleted User]', isDeleted: true }
            : this.userMapper.toDto(entity.user)
        };
        ```
        
        <Tip>Best for: UI contexts where showing "something" is better than null</Tip>
      </Tab>
      
      <Tab title="Strategy B: Omit">
        Omit the reference entirely
        
        ```typescript
        const dto: ChatMessageDto = {
          id: entity.id,
          content: entity.content,
          // user field omitted if deleted
        };
        
        if (entity.user && !entity.user.isDeleted) {
          dto.user = this.userMapper.toDto(entity.user);
        }
        
        return dto;
        ```
        
        <Tip>Best for: Optional references where absence is acceptable</Tip>
      </Tab>
      
      <Tab title="Strategy C: Error">
        Return 404 or other error
        
        ```typescript
        if (entity.user?.isDeleted) {
          throw new NotFoundException(
            `User ${entity.user.id} has been deleted`
          );
        }
        ```
        
        <Tip>Best for: Critical references where deletion should block access</Tip>
      </Tab>
    </Tabs>
  </Step>
</Tabs>

<Warning>
**NEVER** serialize the full Tier-3 payload (email, phone, address, etc.) when `isDeleted === true`. This prevents data leakage after soft-deletion.
</Warning>

## Testing Requirements

### Mandatory Test Coverage

Every entity with non-nullable owner relations MUST have these two tests:

<CodeGroup>
```typescript test-1-own-row-survives.spec.ts
describe('Lead soft-delete resilience', () => {
  it('should return lead even when person is soft-deleted', async () => {
    // Arrange
    const person = await personFactory.create();
    const org = await orgFactory.create();
    const lead = await leadFactory.create({ person, organization: org });
    
    // Act: soft-delete the person
    person.isDeleted = true;
    await em.flush();
    em.clear();
    
    // Assert: lead still queryable
    const found = await em.findOne(Lead, { id: lead.id });
    expect(found).toBeDefined();
    expect(found!.id).toBe(lead.id);
  });
});
```

```typescript test-2-populated-relation-null.spec.ts
describe('Lead soft-delete resilience', () => {
  it('should hydrate soft-deleted person when explicitly populated', async () => {
    // Arrange
    const person = await personFactory.create();
    const org = await orgFactory.create();
    const lead = await leadFactory.create({ person, organization: org });
    
    person.isDeleted = true;
    await em.flush();
    em.clear();
    
    // Act: populate the soft-deleted person
    const found = await em.findOne(
      Lead,
      { id: lead.id },
      { populate: ['person'] }
    );
    
    // Assert: person is loaded (because of filter override)
    expect(found).toBeDefined();
    expect(found!.person.id).toBe(person.id);
    expect(found!.person.isDeleted).toBe(true);
  });
});
```
</CodeGroup>

<Check>
**Why these tests matter:** Without them, you won't catch the silent data-loss bug until production. The entity passes unit tests because you're testing the "happy path" where nothing is soft-deleted.
</Check>

## Common Pitfalls

<Warning>
**These mistakes WILL cause silent data loss in production:**
</Warning>

<AccordionGroup>
  <Accordion title="Forgetting Filter Override on Non-Nullable M:1">
    ```typescript
    // ❌ WRONG — will drop when person is soft-deleted
    @ManyToOne(() => Person, { nullable: false })
    person!: Ref<Person>;
    
    // ✅ CORRECT
    @ManyToOne(() => Person, {
      nullable: false,
      filters: { isDeleted: false }
    })
    person!: Ref<Person>;
    ```
  </Accordion>
  
  <Accordion title="Wrong Active Filter Definition">
    ```typescript
    // ❌ WRONG — causes same issue when isDeleted filter disabled
    @Filter({
      name: 'active',
      default: true,
      cond: { isDeleted: false, isArchived: false }
    })
    
    // ✅ CORRECT
    @Filter({
      name: 'active',
      default: true,
      cond: { isArchived: false }
    })
    ```
  </Accordion>
  
  <Accordion title="Using { filters: false } Without Comment">
    ```typescript
    // ❌ WRONG — no category comment
    const lead = await em.findOne(Lead, { id }, { filters: false });
    
    // ✅ CORRECT
    // filters: false — restore operation
    const lead = await em.findOne(Lead, { id }, { filters: false });
    ```
  </Accordion>
  
  <Accordion title="Serializing Full Soft-Deleted Tier-3 Payload">
    ```typescript
    // ❌ WRONG — leaks deleted user data
    return {
      user: this.userMapper.toDto(entity.user) // includes email, phone if deleted
    };
    
    // ✅ CORRECT
    return {
      user: entity.user?.isDeleted
        ? { id: entity.user.id, name: '[Deleted User]', isDeleted: true }
        : this.userMapper.toDto(entity.user)
    };
    ```
  </Accordion>
  
  <Accordion title="Missing Mandatory Tests">
    ```typescript
    // ❌ WRONG — only testing happy path
    it('should find lead', async () => {
      const lead = await leadFactory.create();
      const found = await em.findOne(Lead, { id: lead.id });
      expect(found).toBeDefined();
    });
    
    // ✅ CORRECT — test with soft-deleted owner
    it('should find lead even when person soft-deleted', async () => {
      const lead = await leadFactory.create();
      lead.person.isDeleted = true;
      await em.flush();
      em.clear();
      
      const found = await em.findOne(Lead, { id: lead.id });
      expect(found).toBeDefined();
    });
    ```
