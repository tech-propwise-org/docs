## TL;DR ‚Äî What every backend dev needs to know

This project uses **MikroORM v6** with `loadStrategy: JOINED`, `autoJoinRefsForFilters: true`, and `filtersOnRelations: true`. Combined with our `isDeleted` and `active` entity filters, this produces a **silent data-loss bug** if entities are not declared correctly:

<Note>
**A row whose own `is_deleted = false` will silently disappear from every query if any of its non-nullable owner relations points at a soft-deleted target.**
</Note>

The fix is **structural** and lives on the entity declaration. Every new entity MUST follow it.

| What | Rule |
| --- | --- |
| Non-nullable `@ManyToOne` / `@OneToOne(owner)` to an entity that declares `@Filter({ name: 'isDeleted', ... })` | Add `{ filters: { isDeleted: false } }` to the relation options |
| `@Filter({ name: 'active', ... })` on a soft-deletable + archivable entity | `cond` MUST be `{ isArchived: false }` only ‚Äî NEVER `{ isDeleted: false, isArchived: false }` |
| Bulk re-point inside a merge service (`em.nativeUpdate(...)` that re-points children of an entity that is about to be soft-deleted) | Pass `{ filters: false }` so the re-point hits archived AND soft-deleted children |
| `{ filters: false }` anywhere else | Allowed only in the explicit-business-intent categories in ¬ß2.3 (merge / restore / conflict-detection / audit-timeline / notification fan-out / admin / seed). Every call site MUST have a code comment naming the category. |
| User reference on a new entity | Classify per ¬ß3.1.1 ‚Äî `createdBy`/`updatedBy`/`archivedBy`/etc. are Tier 1 (audit ref); `assignedTo`/`owner`/`recipient`/`user` are Tier 2 or 3 |
| DTO with a Tier-3 owner relation (`person`, `organization`) | If `populated.isDeleted === true` ‚Üí DTO MUST use placeholder / omit / 4xx (Strategies A / B / C in ¬ß3.3.1). NEVER serialize the full Tier-3 payload when soft-deleted |
| New entity with non-nullable owners | Update the audit table in `QUERY_OPTIMIZATION_PATTERNS.md` ¬ß5.5 + extend `DataIntegrityAuditService` for Tier 2/3 (the daily cron picks it up ‚Äî do NOT add an HTTP endpoint) + add the two mandatory tests from ¬ß4.1 |

<Warning>
If you skip any of these, the regression will not show up in unit tests ‚Äî it surfaces as "this row exists in the DB but the API returns nothing" weeks later.
</Warning>

## Cause ‚Äî Why rows silently disappear

MikroORM v6 (`@mikro-orm/core@6.6.7`) has three defaults that interact badly with soft-deletable graphs:

```typescript
// mikro-orm.config.ts (defaults ‚Äî NOT to be disabled globally)
loadStrategy: LoadStrategy.JOINED;       // populated relations are joined into the main SELECT
autoJoinRefsForFilters: true;            // auto-add :ref JOIN for any relation whose target has filters
filtersOnRelations: true;                // apply target-entity filters via JOIN ON
```

For every M:1 / 1:1-owner relation on the queried entity, MikroORM checks whether the target entity declares any default-enabled filters (e.g. `@Filter({ name: 'isDeleted', default: true, cond: { isDeleted: false } })`). If so, it adds an auto `:ref` JOIN to that relation **even if you did not populate it**, so the filter can be applied as a JOIN `ON` condition.

For a **non-nullable** M:1 the resulting JOIN type is `INNER JOIN` (see `AbstractSqlDriver.getFieldsForJoinedLoad`):

```typescript
const mandatoryToOneProperty =
  [ReferenceKind.MANY_TO_ONE, ReferenceKind.ONE_TO_ONE].includes(prop.kind) && !prop.nullable;
const joinType =
  (hint.filter && !prop.nullable) || mandatoryToOneProperty ? INNER_JOIN : LEFT_JOIN;
```

**The failure mode**: `em.find(Lead, { id: leadId })` becomes (roughly):

```sql
SELECT lead.*
  FROM lead
  INNER JOIN person p ON p.id = lead.person_id AND p.is_deleted = false
  INNER JOIN organization o ON o.id = lead.organization_id AND o.is_deleted = false
 WHERE lead.id = $1
   AND lead.is_deleted = false;
```

<Warning>
If **any** of those non-nullable joined targets fails the filter (e.g. `Person.isDeleted = true` because the lead's person was merged into another person), the **lead row is silently dropped** ‚Äî even though `lead.is_deleted = false`.
</Warning>

For nullable relations the JOIN is `LEFT JOIN` and the relation hydrates to `null` instead of dropping the row (see MikroORM 6.6 release notes ‚Äî the v6.5 "Strict relation filters" regression was reverted in 6.6).

### Real production examples this caused

| Scenario | Symptom |
| --- | --- |
| Two persons merged ‚Üí secondary `Person.isDeleted = true` ‚Üí an archived Lead still pointing at the secondary Person | The archived Lead was invisible everywhere even though it existed in the DB |
| `DealStage` cleanup ‚Üí custom stage soft-deleted while a Deal still pointed at it | The Deal vanished from list endpoints |
| Agent leaves the org ‚Üí `User.isDeleted = true` ‚Üí every Deal / Contact / Company / Listing they `createdBy` would have vanished if we shipped the relation override blindly without the Tier-1 carve-out |

## The fix ‚Äî Three coordinated parts

The fix has **three** parts. Skipping any one of them re-introduces the bug.

### Per-relation `{ filters: { isDeleted: false } }` override (MANDATORY structural)

Declare `{ filters: { isDeleted: false } }` on **every non-nullable** `@ManyToOne` / `@OneToOne(owner)` whose target entity has an `isDeleted` filter:

```typescript
// ‚úÖ CORRECT ‚Äî surgical, structural, zero side effects on standalone queries
@ManyToOne(() => Person, { filters: { isDeleted: false } })
person!: Person;

@ManyToOne(() => Organization, { filters: { isDeleted: false } })
organization!: Organization;

@ManyToOne(() => User, { filters: { isDeleted: false } })
createdBy!: User;
```

This disables the target's `isDeleted` filter **only** for the auto-`:ref` JOIN MikroORM adds for this relation. Standalone `em.find(Person, ...)` / `em.findOne(Organization, ...)` queries still apply the target's `isDeleted` filter ‚Äî the override is local to this relation site.

<Info>
**Side effect to be aware of:** when you explicitly populate this relation (`populate: ['person']`), the populate JOIN ALSO loses the target's `isDeleted` filter. A soft-deleted populated row will hydrate fully instead of being filtered out. This is acceptable but each Tier-2/Tier-3 relation MUST add a DTO warn-log (see ¬ß3 below).
</Info>

### `active` filter cleanup (MANDATORY for archivable entities)

Every archivable entity should have **two** entity-level filters:

```typescript
@Entity()
@Filter({ name: 'isDeleted', default: true, cond: { isDeleted: false } })
@Filter({ name: 'active', default: true, cond: { isArchived: false } })  // ‚Üê NOT { isDeleted: false, isArchived: false }
export class Lead { ... }
```

Both filters default to `true`, so all standard queries apply `isDeleted = false AND isArchived = false`. The `active` filter MUST omit `isDeleted` because the standalone `isDeleted` filter already handles soft-delete exclusion. The redundant clause is dangerous because it creates the appearance that disabling `active` also restores soft-deleted rows ‚Äî it doesn't, and reasoning about it correctly requires understanding both filters at once.

<Note>
**INVARIANTS** (already documented inline on every fixed entity):

- Never disable `isDeleted` while keeping `active` ‚Äî that would return soft-deleted records.
- To include archived rows, use `{ filters: { active: false } }`.
- To include soft-deleted rows, use `{ filters: { isDeleted: false } }` (rare ‚Äî typically only in repair scripts and merge code).
- To include both, use `{ filters: false }` (only valid in merge / cascade / RLS-bypass paths ‚Äî see ¬ß2.3).
</Note>

### `filters: false` is reserved for explicit business intent (MANDATORY framing)

`{ filters: false }` disables BOTH `isDeleted` and `active` filters at once. It is **never** a workaround for the `autoJoinRefsForFilters` symptom from ¬ß1 ‚Äî the per-relation override in ¬ß2.1 is the structural fix for that. Use `{ filters: false }` only when the code path **explicitly intends** to read soft-deleted or archived rows.

#### When `{ filters: false }` IS legitimate

A request handler / service / job may use `{ filters: false }` if and only if it falls into one of these categories. **Every call site MUST have a code comment naming the category.**

| Category | When to use | Canonical example |
| --- | --- | --- |
| **Merge / repair / cascade** | Re-pointing or cascading children of a row about to be soft-deleted, so archived/soft-deleted children get re-pointed too | `PersonService.mergePersons`, `ContactService.mergeContacts` |
| **Restore / unarchive / revive** | Resolver / endpoint that must find a soft-archived row to bring it back; the resolver is shared between archive and unarchive | `ContactService` archive resolver, `unarchive*` endpoints |
| **Conflict detection on re-create** | Looking up a soft-deleted row by unique key to prevent constraint violation, or to revive in place of `INSERT` | `TeamMembershipService.create*` (find soft-deleted membership before re-add) |
| **Audit / timeline / history rendering** | Fetching deleted stages / users / rules so historical entries render correctly | `audit-timeline.resolver.ts`, `escalation-timeline.resolver.ts` |
| **Notification fan-out across deleted/archived parents** | Loading archived deals/leads so their creators are still notified when a stakeholder changes | `ContactService.removeStakeholder*`, `CompanyService` notification fan-out |
| **Admin / system-admin maintenance** | System-admin tools that intentionally surface soft-deleted records | `system-admin.service.ts` |
| **Seed / migration / fixture / repair scripts** | Seeders, migrations, repair CLI scripts | `CrmFakerSeeder`, `UserSeeder`, `repair-orphaned-lead-persons.ts` |

#### Canonical merge example (still the most important use case)

```typescript
// ‚úÖ CORRECT ‚Äî merge: re-point archived AND soft-deleted children before soft-deleting secondary.
//    `filters: false` is required so the re-point hits archived/soft-deleted leads too;
//    without it those rows would become orphans pointing at a soft-deleted Person and would
//    silently disappear the next time something queries them (see ¬ß1).
await em.nativeUpdate(
  'Lead',
  { person: { $in: uniqueSecondaryIds } },
  { person: primaryPersonId },
  { filters: false },
);

secondary.isDeleted = true;
```

#### When `{ filters: false }` IS NOT legitimate

```typescript
// ‚ùå NEVER ‚Äî using `filters: false` to "fix" a list endpoint that is silently dropping rows.
//    The right fix is the per-relation override in ¬ß2.1 on the entity declaration, not at the
//    query site. `filters: false` here would also leak soft-deleted populated rows to the response.
async getLeads(...) {
  return em.find(Lead, { ... }, { filters: false });
}

// ‚ùå NEVER ‚Äî using `filters: false` on an unauthenticated public endpoint. It bypasses BOTH
//    the soft-delete guard AND the archive guard, and combined with `executeReadOnlyWithBypass`
//    it bypasses RLS too. That is a data-leak vector across tenants.
@Get('search')
@PublicEndpoint()
async publicSearch(...) {
  return em.find(Entity, { ... }, { filters: false });
}
```

#### What to use instead, by intent

| You want to read‚Ä¶ | Use this |
| --- | --- |
| Active rows only (default behavior) | No options ‚Äî both filters are on by default |
| Active + archived rows (e.g. detail page that shows archived items too) | `{ filters: { active: false } }` ‚Äî keeps `isDeleted = false` enforced |
| Soft-deleted rows on a specific lookup (e.g. revive flow) | `{ filters: { isDeleted: false } }` on the standalone query ‚Äî be explicit about why |
| Both archived AND soft-deleted (merge / repair / audit / notification fan-out ‚Äî see table above) | `{ filters: false }` with mandatory comment naming the category |
