title: "Soft Delete Filter Standard"
description: "Project-wide mandatory standard for handling soft-delete filters in MikroORM to prevent silent data loss"
---

<Warning>
**Status:** Project-wide standard — MANDATORY for all entities, services, and DTOs
</Warning>

This project uses **MikroORM v6** with specific configurations that can cause silent data loss if entities are not declared correctly. This standard prevents rows from silently disappearing when related entities are soft-deleted.

## Quick Reference

<Note>
A row whose own `is_deleted = false` will silently disappear from every query if any of its non-nullable owner relations points at a soft-deleted target.
</Note>

| Scenario | Required Action |
|----------|----------------|
| Non-nullable `@ManyToOne` / `@OneToOne(owner)` to soft-deletable entity | Add `{ filters: { isDeleted: false } }` to relation options |
| `@Filter({ name: 'active' })` on soft-deletable + archivable entity | Use `{ isArchived: false }` only — NEVER `{ isDeleted: false, isArchived: false }` |
| Bulk re-point in merge service | Pass `{ filters: false }` to hit archived AND soft-deleted children |
| User reference on new entity | Classify per audit tier (see User Reference Classification) |
| DTO with Tier-3 owner relation | Use placeholder/omit/4xx when `populated.isDeleted === true` |

## Root Cause

<Info>
MikroORM v6 has three defaults that interact badly with soft-deletable entity graphs:
- `loadStrategy: JOINED`
- `autoJoinRefsForFilters: true` 
- `filtersOnRelations: true`
</Info>

For every non-nullable `@ManyToOne` / `@OneToOne(owner)` relation, MikroORM automatically adds an `INNER JOIN` to apply the target entity's default filters, even when the relation isn't populated.

### The Failure Mode

A query like `em.find(Lead, { id: leadId })` becomes:

```sql
SELECT lead.*
  FROM lead
  INNER JOIN person p ON p.id = lead.person_id AND p.is_deleted = false
  INNER JOIN organization o ON o.id = lead.organization_id AND o.is_deleted = false
 WHERE lead.id = $1
   AND lead.is_deleted = false;
```

If **any** joined target fails the filter (e.g., `Person.isDeleted = true`), the lead row is silently dropped.

### Production Examples

<AccordionGroup>
<Accordion title="Person Merge Scenario">
Two persons merged → secondary `Person.isDeleted = true` → archived Lead still pointing at secondary Person → Lead became invisible everywhere despite existing in DB.
</Accordion>

<Accordion title="Deal Stage Cleanup">
Custom stage soft-deleted while Deal still pointed at it → Deal vanished from list endpoints.
</Accordion>

<Accordion title="Agent Departure">
Agent leaves org → `User.isDeleted = true` → every Deal/Contact/Company/Listing they created would vanish without proper tier classification.
</Accordion>
</AccordionGroup>

## The Fix: Three Coordinated Parts

### 1. Per-Relation Filter Override (MANDATORY)

<Steps>
<Step title="Identify Non-Nullable Relations">
Find all non-nullable `@ManyToOne` / `@OneToOne(owner)` relations pointing to entities with `@Filter({ name: 'isDeleted' })`.
</Step>

<Step title="Add Filter Override">
Add `{ filters: { isDeleted: false } }` to the relation options:

```typescript
@Entity()
export class Lead {
  @ManyToOne(() => Person, { 
    nullable: false,
    filters: { isDeleted: false } // MANDATORY override
  })
  person!: Person;
  
  @ManyToOne(() => Organization, { 
    nullable: false,
    filters: { isDeleted: false } // MANDATORY override
  })
  organization!: Organization;
}
```
</Step>

<Step title="Test the Override">
Verify that querying the entity still works when related entities are soft-deleted.
</Step>
</Steps>

### 2. Correct `active` Filter Pattern

For entities that are both soft-deletable AND archivable:

<CodeGroup>
```typescript Correct Pattern
@Filter({
  name: 'active',
  default: true,
  cond: { isArchived: false } // Only archived check
})
export class Deal {
  @Property()
  isDeleted = false;
  
  @Property() 
  isArchived = false;
}
```

```typescript Wrong Pattern
@Filter({
  name: 'active', 
  default: true,
  cond: { isDeleted: false, isArchived: false } // NEVER do this
})
```
</CodeGroup>

<Warning>
Including `isDeleted: false` in the `active` filter creates a double-filter condition that breaks the override mechanism.
</Warning>

### 3. Strategic Use of `{ filters: false }`

<Tip>
`{ filters: false }` is allowed only in specific business-intent categories. Every usage MUST have a code comment naming the category.
</Tip>

**Allowed Categories:**
- Merge operations
- Restore operations  
- Conflict detection
- Audit timeline
- Notification fan-out
- Admin operations
- Data seeding

```typescript
// Bulk re-point during merge - Category: merge
await em.nativeUpdate(Lead, 
  { person: oldPersonId }, 
  { person: newPersonId },
  { filters: false } // Hits archived AND soft-deleted children
);
```

## User Reference Classification

### Tier 1: Audit References (Always Override)
- `createdBy`, `updatedBy`, `deletedBy`, `archivedBy`
- `assignedBy`, `performedBy`, `approvedBy`

```typescript
@ManyToOne(() => User, { 
  nullable: false,
  filters: { isDeleted: false } // Always override for audit refs
})
createdBy!: User;
```

### Tier 2: Functional References (Override)
- `assignedTo`, `owner`, `agent`, `manager`

### Tier 3: Business Entity References (Override + DTO Strategy)
- `person`, `organization`, `contact`, `company`

<Note>
For Tier 3 references, when `populated.isDeleted === true`, the DTO must use one of:
- **Strategy A:** Placeholder (`"[Deleted Person]"`)
- **Strategy B:** Omit the field
- **Strategy C:** Return 4xx error
</Note>

## Special Case: excludeAiServiceUser Filter

The `excludeAiServiceUser` filter uses an **inverted default** (`default: false`):

```typescript
@Filter({
  name: 'excludeAiServiceUser',
  default: false, // Inverted default
  cond: { isAiServiceUser: false, isSystemServiceUser: false }
})
```

### When to Enable Explicitly

<Tabs>
<Tab title="User Listings">
```typescript
// Direct User queries need explicit exclusion
const users = await em.find(User, {}, { 
  filters: { excludeAiServiceUser: true } 
});
```
</Tab>

<Tab title="UserOrgRole Queries">
```typescript
// System actor has no UserOrgRole, AI agent excluded by where clause
const roles = await em.find(UserOrgRole, {
  user: { isAiServiceUser: false }
});
```
</Tab>

<Tab title="Auth Lookups">
```typescript
// Explicit exclusion in auth flows
const user = await em.findOne(User, {
  email,
  isAiServiceUser: false,
  isSystemServiceUser: false
});
```
</Tab>
</Tabs>

## Common Pitfalls

<AccordionGroup>
<Accordion title="Forgetting Filter Override">
**Symptom:** Rows exist in DB but API returns empty results
**Fix:** Add `{ filters: { isDeleted: false } }` to all non-nullable relations
</Accordion>

<Accordion title="Wrong active Filter Pattern">
**Symptom:** Override doesn't work, still getting silent drops
**Fix:** Remove `isDeleted: false` from `active` filter condition
</Accordion>

<Accordion title="Inappropriate filters: false Usage">
**Symptom:** Soft-deleted data leaking to end users
**Fix:** Only use `{ filters: false }` in approved categories with code comments
</Accordion>
</AccordionGroup>

## Verification Commands

```bash
# Check for entities missing filter overrides
grep -r "@ManyToOne\|@OneToOne" src/entities/ | grep "nullable: false" | grep -v "filters:"

# Find inappropriate active filter patterns  
grep -r "cond:.*isDeleted.*isArchived" src/entities/

# Audit filters: false usage
grep -r "filters: false" src/ | grep -v "// Category:"
```
