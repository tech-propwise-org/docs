<Warning>
**Project-wide standard — MANDATORY** for all entities, services, and DTOs. This prevents silent data loss in production where rows disappear from queries due to MikroORM's auto-join behavior.
</Warning>

## TL;DR — What every backend dev needs to know

This project uses **MikroORM v6** with `loadStrategy: JOINED`, `autoJoinRefsForFilters: true`, and `filtersOnRelations: true`. Combined with our `isDeleted` and `active` entity filters, this produces a **silent data-loss bug** if entities are not declared correctly:

<Note>
**A row whose own `is_deleted = false` will silently disappear from every query if any of its non-nullable owner relations points at a soft-deleted target.**
</Note>

The fix is **structural** and lives on the entity declaration. Every new entity MUST follow it.

### Quick reference table

| What | Rule |
|------|------|
| Non-nullable `@ManyToOne` / `@OneToOne(owner)` to an entity that declares `@Filter({ name: 'isDeleted', ... })` | Add `{ filters: { isDeleted: false } }` to the relation options |
| `@Filter({ name: 'active', ... })` on a soft-deletable + archivable entity | `cond` MUST be `{ isArchived: false }` only — NEVER `{ isDeleted: false, isArchived: false }` |
| Bulk re-point inside a merge service (`em.nativeUpdate(...)` that re-points children of an entity that is about to be soft-deleted) | Pass `{ filters: false }` so the re-point hits archived AND soft-deleted children |
| `{ filters: false }` anywhere else | Allowed only in the explicit-business-intent categories in §2.3. Every call site MUST have a code comment naming the category. |
| User reference on a new entity | Classify per §3.1.1 — `createdBy`/`updatedBy`/`archivedBy`/etc. are Tier 1 (audit ref); `assignedTo`/`owner`/`recipient`/`user` are Tier 2 or 3 |
| DTO with a Tier-3 owner relation (`person`, `organization`) | If `populated.isDeleted === true` → DTO MUST use placeholder / omit / 4xx (Strategies A / B / C in §3.3.1). NEVER serialize the full Tier-3 payload when soft-deleted |
| New entity with non-nullable owners | Update the audit table in `QUERY_OPTIMIZATION_PATTERNS.md` §5.5 + extend `DataIntegrityAuditService` for Tier 2/3 + add the two mandatory tests from §4.1 |

<Warning>
If you skip any of these, the regression will not show up in unit tests — it surfaces as "this row exists in the DB but the API returns nothing" weeks later.
</Warning>

## Why rows silently disappear

MikroORM v6 (`@mikro-orm/core@6.6.7`) has three defaults that interact badly with soft-deletable graphs:

```typescript
// mikro-orm.config.ts (defaults — NOT to be disabled globally)
loadStrategy: LoadStrategy.JOINED; // populated relations are joined into the main SELECT
autoJoinRefsForFilters: true; // auto-add :ref JOIN for any relation whose target has filters
filtersOnRelations: true; // apply target-entity filters via JOIN ON
```

For every M:1 / 1:1-owner relation on the queried entity, MikroORM checks whether the target entity declares any default-enabled filters (e.g. `@Filter({ name: 'isDeleted', default: true, cond: { isDeleted: false } })`). If so, it adds an auto `:ref` JOIN to that relation **even if you did not populate it**, so the filter can be applied as a JOIN `ON` condition.

For a **non-nullable** M:1 the resulting JOIN type is `INNER JOIN`:

```typescript
const mandatoryToOneProperty =
  [ReferenceKind.MANY_TO_ONE, ReferenceKind.ONE_TO_ONE].includes(prop.kind) && !prop.nullable;
const joinType = (hint.filter && !prop.nullable) || mandatoryToOneProperty ? INNER_JOIN : LEFT_JOIN;
```

### The failure mode

`em.find(Lead, { id: leadId })` becomes (roughly):

```sql
SELECT lead.*
  FROM lead
  INNER JOIN person p ON p.id = lead.person_id AND p.is_deleted = false
  INNER JOIN organization o ON o.id = lead.organization_id AND o.is_deleted = false
 WHERE lead.id = $1
   AND lead.is_deleted = false;
```

If **any** of those non-nullable joined targets fails the filter (e.g. `Person.isDeleted = true` because the lead's person was merged into another person), the **lead row is silently dropped** — even though `lead.is_deleted = false`.

<Info>
For nullable relations the JOIN is `LEFT JOIN` and the relation hydrates to `null` instead of dropping the row (see MikroORM 6.6 release notes — the v6.5 "Strict relation filters" regression was reverted in 6.6).
</Info>

### Real production examples

| Scenario | Symptom |
|----------|---------|
| Two persons merged → secondary `Person.isDeleted = true` → an archived Lead still pointing at the secondary Person | The archived Lead was invisible everywhere even though it existed in the DB |
| `DealStage` cleanup → custom stage soft-deleted while a Deal still pointed at it | The Deal vanished from list endpoints |
| Agent leaves the org → `User.isDeleted = true` → every Deal / Contact / Company / Listing they `createdBy` would have vanished if we shipped the relation override blindly without the Tier-1 carve-out |

## The fix — Three coordinated parts

The fix has **three** parts. Skipping any one of them re-introduces the bug.

### Per-relation filter override (MANDATORY structural)

<Check>
**STRUCTURAL FIX:** Declare `{ filters: { isDeleted: false } }` on **every non-nullable** `@ManyToOne` / `@OneToOne(owner)` whose target entity has an `isDeleted` filter.
</Check>

```typescript
// ✅ CORRECT — surgical, structural, zero side effects on standalone queries
@ManyToOne(() => Person, { filters: { isDeleted: false } })
person!: Person;

@ManyToOne(() => Organization, { filters: { isDeleted: false } })
organization!: Organization;

@ManyToOne(() => User, { filters: { isDeleted: false } })
createdBy!: User;
```

This disables the target's `isDeleted` filter **only** for the auto-`:ref` JOIN MikroORM adds for this relation. Standalone `em.find(Person, ...)` / `em.findOne(Organization, ...)` queries still apply the target's `isDeleted` filter — the override is local to this relation site.

<Note>
**Side effect to be aware of:** when you explicitly populate this relation (`populate: ['person']`), the populate JOIN ALSO loses the target's `isDeleted` filter. A soft-deleted populated row will hydrate fully instead of being filtered out. This is acceptable but each Tier-2/Tier-3 relation MUST add a DTO warn-log (see §3 below).
</Note>

Nullable attribution relations may also need the override when product behavior keeps historical references after the target is soft-deleted and callers populate the relation.

### Active filter cleanup (MANDATORY for archivable entities)

Every archivable entity should have **two** entity-level filters:

```typescript
@Entity()
@Filter({ name: 'isDeleted', default: true, cond: { isDeleted: false } })
@Filter({ name: 'active', default: true, cond: { isArchived: false } })  // ← NOT { isDeleted: false, isArchived: false }
export class Lead { ... }
```

Both filters default to `true`, so all standard queries apply `isDeleted = false AND isArchived = false`. The `active` filter MUST omit `isDeleted` because the standalone `isDeleted` filter already handles soft-delete exclusion.

<Warning>
**INVARIANTS:**
- Never disable `isDeleted` while keeping `active` — that would return soft-deleted records
- To include archived rows, use `{ filters: { active: false } }`
- To include soft-deleted rows, use `{ filters: { isDeleted: false } }` (rare — typically only in repair scripts and merge code)
- To include both, use `{ filters: false }` (only valid in merge / cascade / RLS-bypass paths — see below)
</Warning>

### `filters: false` is reserved for explicit business intent

`{ filters: false }` disables BOTH `isDeleted` and `active` filters at once. It is **never** a workaround for the `autoJoinRefsForFilters` symptom — the per-relation override above is the structural fix for that.

<Steps>
<Step title="Check if your use case is legitimate">
Use `{ filters: false }` only when the code path **explicitly intends** to read soft-deleted or archived rows.
</Step>

<Step title="Add mandatory comment">
**Every call site MUST have a code comment naming the category.**
</Step>

<Step title="Verify category matches approved list">
Ensure your use case falls into one of the approved categories below.
</Step>
</Steps>

#### When `{ filters: false }` IS legitimate

<AccordionGroup>
<Accordion title="Merge / cascade operations">
When merging entities or cascading soft-deletes, you need to process ALL children regardless of their soft-delete status.

```typescript
// Category: merge — re-pointing children before soft-delete
await em.nativeUpdate(Lead, 
  { person: sourcePersonId }, 
  { person: targetPersonId }, 
  { filters: false }
);
```
</Accordion>

<Accordion title="Restore operations">
Restoring a soft-deleted entity requires reading it first.

```typescript
// Category: restore — need to read soft-deleted entity to restore it
const deletedPerson = await em.findOne(Person, 
  { id: personId }, 
  { filters: false }
);
```
</Accordion>

<Accordion title="Conflict detection">
Checking for naming conflicts must include soft-deleted entities to prevent reuse of identifiers.

```typescript
// Category: conflict-detection — prevent duplicate slugs
const existingOrg = await em.findOne(Organization, 
  { slug: newSlug }, 
  { filters: false }
);
```
</Accordion>

<Accordion title="Audit timeline">
Audit logs and activity feeds need complete historical context.

```typescript
// Category: audit-timeline — complete historical view
const allActivities = await em.find(Activity, 
  { entityId: targetId }, 
  { filters: false, orderBy: { createdAt: 'ASC' } }
);
```
</Accordion>
</AccordionGroup>

## User reference classification

### Tier classification system

<Tabs>
<Tab title="Tier 1: Audit references">
**NEVER** gets the `{ filters: { isDeleted: false } }` override.

These are pure audit trail — who performed the action, not who the action affects.

```typescript
// ✅ CORRECT — no filter override for audit refs
@ManyToOne(() => User)
createdBy!: User;

@ManyToOne(() => User)
updatedBy!: User;

@ManyToOne(() => User)
archivedBy!: User;
```

Examples: `createdBy`, `updatedBy`, `archivedBy`, `mergedBy`, `approvedBy`
</Tab>

<Tab title="Tier 2: Operational assignment">
**DOES** get the `{ filters: { isDeleted: false } }` override + DTO handling.

These affect current business operations.

```typescript
// ✅ CORRECT — gets filter override
@ManyToOne(() => User, { filters: { isDeleted: false } })
assignedTo!: User;
```

Examples: `assignedTo`, `owner` (when operational), `reviewer`, `approver`
</Tab>

<Tab title="Tier 3: Core business entity">
**DOES** get the `{ filters: { isDeleted: false } }` override + mandatory DTO handling.

These are fundamental to the entity's purpose.

```typescript
// ✅ CORRECT — gets filter override  
@ManyToOne(() => Person, { filters: { isDeleted: false } })
person!: Person;

@ManyToOne(() => Organization, { filters: { isDeleted: false } })
organization!: Organization;
```

Examples: `person`, `organization`, `company`, `contact`
</Tab>
</Tabs>

### DTO handling for Tier 2/3

When a Tier 2/3 relation is populated and `populated.isDeleted === true`, the DTO MUST handle it explicitly:

<CodeGroup>
```typescript Strategy A: Placeholder
// In PersonDto.fromEntity()
if (person.isDeleted) {
  logger.warn(`DTO serializing soft-deleted Person ${person.id} for Lead ${leadId}`);
  return {
    id: person.id,
    name: '[Deleted Contact]',
    // ... other placeholder fields
  };
}
```

```typescript Strategy B: Omit
// In LeadDto.fromEntity() 
if (lead.person?.isDeleted) {
  logger.warn(`Omitting soft-deleted Person ${lead.person.id} from Lead ${lead.id} DTO`);
  return {
    // ... lead fields without person
    personId: lead.person.id, // Keep the FK for reference
  };
}
```

```typescript Strategy C: 4xx error
// In request handler
if (lead.person?.isDeleted) {
  logger.warn(`Request for Lead ${leadId} with soft-deleted Person ${lead.person.id}`);
  throw new BadRequestException('Referenced contact is no longer available');
}
```
</CodeGroup>

<Warning>
**NEVER serialize the full Tier-3 payload when soft-deleted.** This prevents data leaks and maintains referential integrity in the API layer.
</Warning>

## Testing requirements

### Mandatory tests for new entities

Every new entity with non-nullable owners MUST include these two tests:

<CodeGroup>
```typescript Test 1: Relation override prevents silent disappearance
describe('Entity visibility with soft-deleted relations', () => {
  it('should find entity even when non-nullable relation target is soft-deleted', async () => {
    // Arrange: create entity with valid relation
    const person = await createPerson({ name: 'John Doe' });
    const lead = await createLead({ person });
    
    // Act: soft-delete the relation target
    await softDeletePerson(person.id);
    
    // Assert: entity is still findable
    const foundLead = await em.findOne(Lead, { id: lead.id });
    expect(foundLead).toBeTruthy();
    expect(foundLead!.id).toBe(lead.id);
  });
});
```

```typescript Test 2: DTO handles soft-deleted populated relations  
describe('DTO serialization with soft-deleted relations', () => {
  it('should handle soft-deleted populated relation appropriately', async () => {
    // Arrange: create entity with valid relation
    const person = await createPerson({ name: 'John Doe' });
    const lead = await createLead({ person });
    
    // Act: soft-delete relation target and populate
    await softDeletePerson(person.id);
    const leadWithPerson = await em.findOne(Lead, 
      { id: lead.id }, 
      { populate: ['person'] }
    );
    
    // Assert: DTO handles soft-deleted relation
    const dto = LeadDto.fromEntity(leadWithPerson!);
    
    // Choose based on your DTO strategy (A/B/C)
    expect(dto.person?.name).toBe('[Deleted Contact]'); // Strategy A
    // OR
    expect(dto.person).toBeUndefined(); // Strategy B  
    // OR expect 4xx thrown during DTO creation // Strategy C
  });
});
```
</CodeGroup>
