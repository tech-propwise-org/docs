<Warning>
**Status:** Project-wide standard — MANDATORY for all entities, services, and DTOs

This document is the authoritative reference for the soft-delete / archive filter pattern. See `QUERY_OPTIMIZATION_PATTERNS.md` §5.5 for MikroORM internals deep-dive.
</Warning>

## TL;DR — what every backend dev needs to know

This project uses **MikroORM v6** with `loadStrategy: JOINED`, `autoJoinRefsForFilters: true`, and `filtersOnRelations: true`. Combined with our `isDeleted` and `active` entity filters, this produces a **silent data-loss bug** if entities are not declared correctly:

<Warning>
A row whose own `is_deleted = false` will silently disappear from every query if any of its non-nullable owner relations points at a soft-deleted target.
</Warning>

The fix is **structural** and lives on the entity declaration. Every new entity MUST follow it.

### Quick reference table

| What | Rule |
|------|------|
| Non-nullable `@ManyToOne` / `@OneToOne(owner)` to an entity that declares `@Filter({ name: 'isDeleted', ... })` | Add `{ filters: { isDeleted: false } }` to the relation options |
| `@Filter({ name: 'active', ... })` on a soft-deletable + archivable entity | `cond` MUST be `{ isArchived: false }` only — NEVER `{ isDeleted: false, isArchived: false }` |
| Non-nullable `@ManyToOne` / `@OneToOne(owner)` to an **ARCHIVABLE** target (one that declares `@Filter({ name: 'active' })`) | Decide per §2.2.1: record / history / financial / document / tracker child → override **BOTH** `{ isDeleted: false, active: false }` (persists across owner archive); active-view junction → `{ isDeleted: false }` only (drops on owner archive by design — callers opt into archived rows explicitly) |
| Bulk re-point inside a merge service (`em.nativeUpdate(...)` that re-points children of an entity that is about to be soft-deleted) | Pass `{ filters: false }` so the re-point hits archived AND soft-deleted children |
| `{ filters: false }` anywhere else | Allowed only in the explicit-business-intent categories in §2.3 (merge / restore / conflict-detection / audit-timeline / notification fan-out / admin / seed). Every call site MUST have a code comment naming the category. |
| User reference on a new entity | Classify per §3.1.1 — `createdBy`/`updatedBy`/`archivedBy`/etc. are Tier 1 (audit ref); `assignedTo`/`owner`/`recipient`/`user` are Tier 2 or 3 |
| DTO with a Tier-3 owner relation (`person`, `organization`) | If `populated.isDeleted === true` → DTO MUST use placeholder / omit / 4xx (Strategies A / B / C in §3.3.1). NEVER serialize the full Tier-3 payload when soft-deleted |
| New entity with non-nullable owners | Update the audit table in `QUERY_OPTIMIZATION_PATTERNS.md` §5.5 + extend `DataIntegrityAuditService` for Tier 2/3 (the daily cron picks it up — do NOT add an HTTP endpoint) + add the two mandatory tests from §4.1 |

<Note>
If you skip any of these, the regression will not show up in unit tests — it surfaces as "this row exists in the DB but the API returns nothing" weeks later.
</Note>

## 1. Cause — why rows silently disappear

MikroORM v6 (`@mikro-orm/core@6.6.7`) has three defaults that interact badly with soft-deletable graphs:

```typescript
// mikro-orm.config.ts (defaults — NOT to be disabled globally)
loadStrategy: LoadStrategy.JOINED; // populated relations are joined into the main SELECT
autoJoinRefsForFilters: true; // auto-add :ref JOIN for any relation whose target has filters
filtersOnRelations: true; // apply target-entity filters via JOIN ON
```

For every M:1 / 1:1-owner relation on the queried entity, MikroORM checks whether the target entity declares any default-enabled filters (e.g. `@Filter({ name: 'isDeleted', default: true, cond: { isDeleted: false } })`). If so, it adds an auto `:ref` JOIN to that relation **even if you did not populate it**, so the filter can be applied as a JOIN `ON` condition.

For a **non-nullable** M:1 the resulting JOIN type is `INNER JOIN` (see `AbstractSqlDriver.getFieldsForJoinedLoad`):

```typescript
const mandatoryToOneProperty =
  [ReferenceKind.MANY_TO_ONE, ReferenceKind.ONE_TO_ONE].includes(prop.kind) && !prop.nullable;
const joinType = (hint.filter && !prop.nullable) || mandatoryToOneProperty ? INNER_JOIN : LEFT_JOIN;
```

### The failure mode

`em.find(Lead, { id: leadId })` becomes (roughly):

```sql
SELECT lead.*
  FROM lead
  INNER JOIN person p ON p.id = lead.person_id AND p.is_deleted = false
  INNER JOIN organization o ON o.id = lead.organization_id AND o.is_deleted = false
 WHERE lead.id = $1
   AND lead.is_deleted = false;
```

<Warning>
If **any** of those non-nullable joined targets fails the filter (e.g. `Person.isDeleted = true` because the lead's person was merged into another person), the **lead row is silently dropped** — even though `lead.is_deleted = false`.
</Warning>

For nullable relations the JOIN is `LEFT JOIN` and the relation hydrates to `null` instead of dropping the row (see MikroORM 6.6 release notes — the v6.5 "Strict relation filters" regression was reverted in 6.6).

### Real production examples

<AccordionGroup>
  <Accordion title="Person merge causing lead disappearance">
    Two persons merged → secondary `Person.isDeleted = true` → an archived Lead still pointing at the secondary Person
    
    **Symptom:** The archived Lead was invisible everywhere even though it existed in the DB
  </Accordion>

  <Accordion title="Deal stage cleanup">
    `DealStage` cleanup → custom stage soft-deleted while a Deal still pointed at it
    
    **Symptom:** The Deal vanished from list endpoints
  </Accordion>

  <Accordion title="Agent departure cascade">
    Agent leaves the org → `User.isDeleted = true` → every Deal / Contact / Company / Listing they `createdBy` would have vanished if we shipped the relation override blindly without the Tier-1 carve-out
  </Accordion>
</AccordionGroup>

## 1.2 Special case — the `excludeAiServiceUser` filter

<Info>
The `excludeAiServiceUser` filter is **OPT-IN** (`default: false`) — this is the inverse of the `isDeleted` strategy.
</Info>

The same `autoJoinRefsForFilters` mechanic bit a **non-soft-delete** filter. The `User` entity declares a second filter that hides **all non-human service accounts** — both the per-org **AI agent service account** (`isAiServiceUser = true`, the row ORGANIZATION-scope AI agents act as) and the per-org **System automation actor** (`isSystemServiceUser = true`, the row distribution / escalation / lead-capture stamp as `createdBy`/`assignedBy`/`performedBy`):

```typescript
@Filter({
  name: 'excludeAiServiceUser',
  default: false,
  cond: { isAiServiceUser: false, isSystemServiceUser: false },
})
```

This filter was originally `default: true` to hide the service accounts from every user listing. But each is a **real `user` row** that stamps non-nullable audit columns (`created_by`, `changed_by`, `performed_by`, …) on anything it touches. With `default: true`, MikroORM auto-joined `... INNER JOIN "user" u ON ... AND u.is_ai_service_user = false` (and now `... AND u.is_system_service_user = false`) for those non-nullable `@ManyToOne(User)` relations and **silently dropped** every row a service account created.

### Resolution — invert the default to opt-in

<Steps>
  <Step title="Filter is OFF by default">
    The filter is **OFF by default**, so audit-relation auto-joins no longer drop service-stamped rows. This is the inverse of the `isDeleted` strategy: for `isDeleted` we keep the global filter ON and override per-relation; for `excludeAiServiceUser` we keep it OFF globally and re-enable per-query where the service accounts must be hidden.
  </Step>

  <Step title="User-facing enumerations must re-exclude">
    | Query shape | How to exclude the service accounts |
    |-------------|-------------------------------------|
    | Direct `User` root query (listings, pickers) | `{ filters: { excludeAiServiceUser: true } }` in the query options (now hides both accounts) |
    | `UserOrgRole` enumeration (member/recipient/agent pools) | `user: { isAiServiceUser: false }` in the `where` (the System actor has **no** `UserOrgRole`, so it is already absent) |
    | Raw SQL over `user_org_roles` with no `team_membership` join (e.g. `getOrgAgentIds`) | `JOIN "user" u ON u.id = uor.user_id AND u.is_ai_service_user = false` (System actor absent — no `UserOrgRole`) |
    | `QueryBuilder` user lists (filters are NOT auto-applied) | `qb.andWhere({ isAiServiceUser: false, isSystemServiceUser: false })` |
    | Auth lookups by email (signup / login / verify / reset) | `{ email, isAiServiceUser: false, isSystemServiceUser: false }` |
  </Step>

  <Step title="Inherently service-free queries">
    **No change needed:** queries scoped to a specific `user: <id>`; queries filtered by `User.organizations` (neither account is added to that M2M); any `TeamMembership`-based query (neither has team membership); and **any `UserOrgRole`-based query** (the System actor has no `UserOrgRole` at all, so only direct global `User` queries need to exclude it).
  </Step>

  <Step title="DTO exposure handling">
    The service accounts can surface as actors (`createdBy` / `changedBy` / `assignedBy` / `performedBy`). Their synthetic `ai-user+<orgId>@propwise.com` / `system-user+<orgId>@propwise.com` emails are **not** masked in user DTOs — they are non-routable internal addresses and the display names ("AI USER" / "System Automation") already identify them. The user-facing enumerations above keep them out of pickers/listings, so the emails are only ever seen in actor attribution where surfacing them is harmless.
  </Step>
</Steps>

<Note>
The actor resolver + provisioning seeder rely on the inverted default — the service accounts resolve by default without passing `{ filters: { excludeAiServiceUser: false } }`. `resolveSystemActorId` finds the System actor via a direct `em.findOne(User, { isSystemServiceUser: true })` which works because the filter is off.
</Note>

## 2. Solution — per-relation filter overrides

### 2.1 Non-nullable M:1 / 1:1 to soft-deletable entities

<Check>
**Mandatory override for every non-nullable owner relation**
</Check>

For every **non-nullable** `@ManyToOne` or `@OneToOne(owner)` relation whose target entity declares `@Filter({ name: 'isDeleted', default: true, ... })`, add:

```typescript
@ManyToOne(() => Person, { nullable: false, filters: { isDeleted: false } })
person: Ref<Person>;
```

This disables the `isDeleted` filter on that specific relation's auto `:ref` JOIN, so the JOIN becomes:

```sql
INNER JOIN person p ON p.id = lead.person_id
-- NOT: ... AND p.is_deleted = false
```

The child row is visible even when its owner is soft-deleted.

### 2.2 Archive filter (`active`) — two-tier override

Entities that support **both** soft-delete (`isDeleted`) and archive (`isArchived`) declare two filters:

```typescript
@Filter({ name: 'isDeleted', default: true, cond: { isDeleted: false } })
@Filter({ name: 'active', default: true, cond: { isArchived: false } })
```

<Warning>
**Critical:** The `active` filter MUST be `{ isArchived: false }` ONLY. 

NEVER `{ isDeleted: false, isArchived: false }` — that creates a second trigger for the silent-drop bug.
</Warning>

#### 2.2.1 Decision tree for archivable-target relations

Non-nullable M:1 to an archivable target must override **both** filters or **only** `isDeleted`, depending on child semantics:

<Tabs>
  <Tab title="Record / history / financial / document / tracker children">
    **Override BOTH filters** — these children persist across owner archive:

    ```typescript
    @ManyToOne(() => Deal, { 
      nullable: false, 
      filters: { isDeleted: false, active: false } 
    })
    deal: Ref<Deal>;
    ```

    Examples:
    - `DealActivity.deal`
    - `DealHistory.deal`
    - `Invoice.deal`
    - `DealDocument.deal`
    - `DealStageChange.deal`

    When the deal is archived, these rows remain visible in queries that load the child directly (e.g., "show me all activities for this contact" includes activities on archived deals).
  </Tab>

  <Tab title="Active-view junctions">
    **Override only `isDeleted`** — these children drop when owner is archived:

    ```typescript
    @ManyToOne(() => Deal, { 
      nullable: false, 
      filters: { isDeleted: false } 
    })
    deal: Ref<Deal>;
    ```

    Examples:
    - `DealStakeholder.deal`
    - `DealTeamMember.deal`

    When the deal is archived, the default `active` filter on `Deal` drops these rows unless callers explicitly pass `{ filters: { active: false } }` to opt into archived-deal stakeholders.
  </Tab>
</Tabs>

### 2.3 When to use `{ filters: false }` (global disable)

<Warning>
Allowed **ONLY** in the following explicit-business-intent categories. Every call site MUST have a code comment naming the category.
</Warning>

<AccordionGroup>
  <Accordion title="Merge services" icon="code-merge">
    Bulk re-point inside merge operations:

    ```typescript
    // Category: merge — re-point children before soft-deleting source
    await em.nativeUpdate(
      Lead,
      { person: sourcePersonId },
      { person: targetPersonId },
      { filters: false }
    );
    ```

    Must hit archived AND soft-deleted children.
  </Accordion>

  <Accordion title="Restore operations" icon="arrow-rotate-left">
    Un-soft-delete / un-archive flows:

    ```typescript
    // Category: restore — fetch soft-deleted entity to restore
    const deleted = await em.findOne(
      Person,
      { id },
      { filters: false }
    );
    ```
  </Accordion>

  <Accordion title="Conflict detection" icon="triangle-exclamation">
    Email / phone uniqueness checks:

    ```typescript
    // Category: conflict-detection — check email across soft-deleted
    const existing = await em.findOne(
      User,
      { email },
      { filters: false }
    );
    ```
  </Accordion>

  <Accordion title="Audit timeline" icon="clock-rotate-left">
    Full history including deleted states:

    ```typescript
    // Category: audit-timeline — show complete history
    const history = await em.find(
      DealHistory,
      { deal: dealId },
      { filters: false, orderBy: { createdAt: 'DESC' } }
    );
    ```
  </Accordion>

  <Accordion title="Notification fan-out" icon="bell">
    Must reach users regardless of deletion state:

    ```typescript
    // Category: notification fan-out — deliver to all stakeholders
    const stakeholders = await em.find(
      DealStakeholder,
      { deal: dealId },
      { filters: false }
    );
    ```
  </Accordion>

  <Accordion title="Admin operations" icon="user-shield">
    Support / data-fix tools:

    ```typescript
    // Category: admin — support data export
    const allRecords = await em.find(
      Entity,
      { organization: orgId },
      { filters: false }
    );
    ```
  </Accordion>

  <Accordion title="Seed / migration" icon="database">
    Data initialization:

    ```typescript
    // Category: seed — idempotent seeding
    const existing = await em.findOne(
      SystemConfig,
      { key },
      { filters: false }
    );
    ```
  </Accordion>
</AccordionGroup>

## 3. User relation tiering

### 3.1 Three-tier classification

Every `@ManyToOne(User)` relation falls into one of three tiers:

<CardGroup cols={3}>
  <Card title="Tier 1: Audit references" icon="file-lines">
    `createdBy`, `updatedBy`, `deletedBy`, `archivedBy`, `assignedBy`, `performedBy`, `changedBy`
    
    **Override:** NONE — these are audit stamps that MUST persist even if the user is soft-deleted
  </Card>

  <Card title="Tier 2: Active business owners" icon="user-tie">
    `assignedTo`, `owner`, `recipient`, `agent`
    
    **Override:** `{ filters: { isDeleted: false } }` — users in this tier must be active for the entity to be visible
  </Card>

  <Card title="Tier 3: Primary entity references" icon="address-card">
    `person`, `user` (when the entity IS about that person/user)
    
    **Override:** `{ filters: { isDeleted: false } }` + special DTO handling (§3.3.1)
  </Card>
</CardGroup>

#### 3.1.1 Why Tier 1 gets no override

<Warning>
If `createdBy` / `updatedBy` had `{ filters: { isDeleted: false } }`, every row an agent created would vanish when the agent leaves the org.
</Warning>

Audit stamps are **immutable attribution** — they record who performed an action at a point in time. They must remain queryable even after the user is soft-deleted.

### 3.2 Tier-2 and Tier-3 User reference override

<CodeGroup>
```typescript Tier 2 — assignedTo example
@ManyToOne(() => User, { nullable: false, filters: { isDeleted: false } })
assignedTo: Ref<User>;
```

```typescript Tier 3 — user example
@ManyToOne(() => User, { nullable: false, filters: { isDeleted: false } })
user: Ref<User>;
```
</CodeGroup>

### 3.3 DTO handling for soft-deleted Tier-3 owners

When a Tier-3 owner (`person`, `organization`) is soft-deleted, the child entity is still visible (thanks to the relation override), but **the DTO MUST NOT serialize the full soft-deleted payload**.

#### 3.3.1 Three strategies

<Tabs>
  <Tab title="Strategy A — Placeholder">
    Replace with a synthetic placeholder:

    ```typescript
    if (entity.person.isDeleted) {
      dto.person = {
        id: entity.person.id,
        firstName: '[Deleted]',
        lastName: 'Person',
        email: null,
        // ... other fields nulled or placeholders
      };
    }
    ```

    Use when the UI needs to show "this record had an owner but that owner is gone."
  </Tab>

  <Tab title="Strategy B — Omit">
    Omit the relation from the DTO:

    ```typescript
    if (!entity.person.isDeleted) {
      dto.person = this.personMapper.toDto(entity.person);
    }
    // else: dto.person remains undefined
    ```

    Use when the relation is optional for the UI (e.g., a contact's secondary organization).
  </Tab>

  <Tab title="Strategy C — 4xx error">
    Fail the request:

    ```typescript
    if (entity.person.isDeleted) {
      throw new NotFoundException('Associated person no longer exists');
    }
    ```

    Use when the entity is **logically invalid** without the owner (rare — most cases should use A or B).
  </Tab>
</Tabs>

<Warning>
**NEVER** serialize the full payload when `isDeleted === true`. This leaks soft-deleted data that may have been merged / purged for GDPR.
</Warning>

## 4. Testing requirements

### 4.1 Two mandatory tests per entity

Every entity with non-nullable soft-deletable owners MUST have:

<Steps>
  <Step title="Test 1 — Child remains visible when owner is soft-deleted">
    ```typescript
    it('should return the child entity even when owner is soft-deleted', async () => {
      const owner = await createOwner();
      const child = await createChild({ owner });
      
      await softDeleteOwner(owner.id);
      
      const fetched = await em.findOne(ChildEntity, { id: child.id });
      expect(fetched).toBeDefined();
      expect(fetched.owner.id).toBe(owner.id);
    });
    ```
  </Step>

  <Step title="Test 2 — DTO handles soft-deleted owner correctly">
    ```typescript
    it('should use placeholder for soft-deleted owner in DTO', async () => {
      const owner = await createOwner();
      const child = await createChild({ owner });
      
      await softDeleteOwner(owner.id);
      
      const dto = await service.findOne(child.id);
      expect(dto.owner.firstName).toBe('[Deleted]');
      expect(dto.owner.email).toBeNull();
    });
    ```
  </Step>
</Steps>

### 4.2 Archive-aware tests

For entities with archivable owners (Tier-2 override decision), add:

```typescript
it('should drop active-view junction when owner is archived', async () => {
  const deal = await createDeal();
  const stakeholder = await createStakeholder({ deal });
  
  await archiveDeal(deal.id);
  
  const fetched = await em.findOne(DealStakeholder, { id: stakeholder.id });
  expect(fetched).toBeNull(); // drops by design — active filter on Deal
});

it('should return active-view junction when explicitly including archived', async () => {
  const deal = await createDeal();
  const stakeholder = await createStakeholder({ deal });
  
  await archiveDeal(deal.id);
  
  const fetched = await em.findOne(
    DealStakeholder,
    { id: stakeholder.id },
    { filters: { active: false } }
  );
  expect(fetched).toBeDefined();
});
```

## 5. Data integrity monitoring

### 5.1 Audit service

<Info>
`DataIntegrityAuditService` runs daily via cron to detect orphaned children (entities whose non-nullable owners are soft-deleted despite relation overrides being declared).
</Info>

When adding a new entity with Tier-2 or Tier-3 owners:

<Steps>
  <Step title="Update audit table">
    Add the entity to the audit table in `QUERY_OPTIMIZATION_PATTERNS.md` §5.5
  </Step>

  <Step title="Extend audit service">
    Add a check method:

    ```typescript
    async checkChildEntityOrphans(): Promise<AuditResult> {
      const orphans = await this.em.find(
        ChildEntity,
        { owner: { isDeleted: true } },
        { filters: false }
      );
      return {
        entity: 'ChildEntity',
        orphanCount: orphans.length,
        orphanIds: orphans.map(o => o.id),
      };
    }
    ```
  </Step>

  <Step title="Register in daily cron">
    The cron job picks up the new check automatically via service method enumeration
  </Step>
</Steps>

<Warning>
Do NOT add an HTTP endpoint for the audit service. It runs via cron only — exposing it over HTTP invites abuse (expensive queries, no auth).
</Warning>

## 6. Migration checklist

When adding a new entity or relation:

<Steps>
  <Step title="Classify the relation">
    - Is it non-nullable?
    - Does the target have `@Filter({ name: 'isDeleted' })`?
    - Is it a `User` reference? → Classify Tier 1 / 2 / 3
    - Does the target have `@Filter({ name: 'active' })`? → Record/history vs active-view?
  </Step>

  <Step title="Add relation override">
    ```typescript
    @ManyToOne(() => Target, { 
      nullable: false, 
      filters: { isDeleted: false, /* active: false if record/history */ }
    })
    target: Ref<Target>;
    ```
  </Step>

  <Step title="Implement DTO handling">
    For Tier-3 owners, choose Strategy A / B / C and implement in mapper
  </Step>

  <Step title="Write tests">
    Add the two mandatory tests from §4.1 (+ archive tests if applicable)
  </Step>

  <Step title="Update audit service">
    Add to `QUERY_OPTIMIZATION_PATTERNS.md` table + extend `DataIntegrityAuditService`
  </Step>

  <Step title="Document `{ filters: false }` usage">
    If you add any `{ filters: false }` calls, add a code comment with the category from §2.3
  </Step>
</Steps>

## 7. Common pitfalls

<AccordionGroup>
  <Accordion title="Forgetting the override on new relations" icon="circle-xmark">
    **Symptom:** Entity works in dev, disappears in prod when first owner is soft-deleted
    
    **Fix:** Add `{ filters: { isDeleted: false } }` immediately when declaring the relation
  </Accordion>

  <Accordion title="Using `{ isDeleted: false, isArchived: false }` in `active` filter" icon="circle-xmark">
    **Symptom:** Archived entities vanish even when queried with `{ filters: { active: false } }`
    
    **Fix:** `active` filter MUST be `{ isArchived: false }` only
  </Accordion>

  <Accordion title="Serializing full soft-deleted Tier-3 payload" icon="circle-xmark">
    **Symptom:** Soft-deleted person/org data leaks to frontend
    
    **Fix:** Use Strategy A / B / C from §3.3.1 in the DTO mapper
  </Accordion>

  <Accordion title="Adding `{ filters: false }` without category comment" icon="circle-xmark">
    **Symptom:** Code review rejection; unclear intent
    
    **Fix:** Add `// Category: merge` (or restore / conflict-detection / etc.) above the call
  </Accordion>

  <Accordion title="Overriding Tier-1 audit relations" icon="circle-xmark">
    **Symptom:** Every row created by a departed agent vanishes
    
    **Fix:** Tier-1 relations (`createdBy`, `updatedBy`, etc.) get NO override
  </Accordion>
</AccordionGroup>
