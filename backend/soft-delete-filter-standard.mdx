title: "Soft Delete Filter Standard"
description: "Project-wide standard for handling soft-delete and archive filters with MikroORM to prevent silent data loss"
---

This project uses **MikroORM v6** with specific configuration that can cause silent data loss if entities are not declared correctly. This document establishes mandatory standards to prevent these issues.

<Warning>
A row whose own `is_deleted = false` will silently disappear from every query if any of its non-nullable owner relations points at a soft-deleted target.
</Warning>

## Quick Reference

<Note>
**Status:** Project-wide standard — MANDATORY for all entities, services, and DTOs
</Note>

Every backend developer must follow these rules:

| Scenario | Rule |
|----------|------|
| Non-nullable `@ManyToOne` / `@OneToOne(owner)` to an entity with `@Filter({ name: 'isDeleted' })` | Add `{ filters: { isDeleted: false } }` to the relation options |
| `@Filter({ name: 'active' })` on soft-deletable + archivable entity | `cond` MUST be `{ isArchived: false }` only — NEVER `{ isDeleted: false, isArchived: false }` |
| Bulk re-point in merge service | Pass `{ filters: false }` so the re-point hits archived AND soft-deleted children |
| `{ filters: false }` elsewhere | Allowed only in explicit business categories (see §2.3). MUST have code comment naming the category |
| User reference on new entity | Classify per §3.1.1 — audit refs are Tier 1, assignedTo/owner are Tier 2 or 3 |
| DTO with Tier-3 owner relation | If `populated.isDeleted === true` → use placeholder/omit/4xx. NEVER serialize full payload |

## Root Cause

MikroORM v6 has three defaults that interact badly with soft-deletable graphs:

```ts
// mikro-orm.config.ts (defaults — NOT to be disabled globally)
loadStrategy: LoadStrategy.JOINED; // populated relations are joined into the main SELECT
autoJoinRefsForFilters: true; // auto-add :ref JOIN for any relation whose target has filters
filtersOnRelations: true; // apply target-entity filters via JOIN ON
```

<Info>
For every M:1 / 1:1-owner relation, MikroORM checks if the target entity declares any default-enabled filters. If so, it adds an auto `:ref` JOIN to apply the filter, even if you didn't populate the relation.
</Info>

### The Failure Mode

For **non-nullable** M:1 relations, the JOIN type is `INNER JOIN`:

```sql
SELECT lead.*
  FROM lead
  INNER JOIN person p ON p.id = lead.person_id AND p.is_deleted = false
  INNER JOIN organization o ON o.id = lead.organization_id AND o.is_deleted = false
 WHERE lead.id = $1
   AND lead.is_deleted = false;
```

If **any** of the joined targets fails the filter (e.g., `Person.isDeleted = true`), the lead row is silently dropped even though `lead.is_deleted = false`.

### Production Examples

<AccordionGroup>
  <Accordion title="Person Merge Scenario">
    Two persons merged → secondary `Person.isDeleted = true` → an archived Lead still pointing at the secondary Person becomes invisible everywhere even though it exists in the DB.
  </Accordion>
  
  <Accordion title="Deal Stage Cleanup">
    `DealStage` cleanup → custom stage soft-deleted while a Deal still pointed at it → the Deal vanished from list endpoints.
  </Accordion>
  
  <Accordion title="Agent Departure">
    Agent leaves the org → `User.isDeleted = true` → every Deal/Contact/Company/Listing they `createdBy` would have vanished if we shipped the relation override blindly.
  </Accordion>
</AccordionGroup>

## The Fix: Three Coordinated Parts

<Warning>
The fix has **three** parts. Skipping any one of them re-introduces the bug.
</Warning>

### Part 1: Per-Relation Filter Override

<Steps>
  <Step title="Identify Non-Nullable Relations">
    Find every non-nullable `@ManyToOne` / `@OneToOne(owner)` relation that points to an entity declaring `@Filter({ name: 'isDeleted' })`.
  </Step>
  
  <Step title="Add Filter Override">
    Add `{ filters: { isDeleted: false } }` to the relation options:
    
    ```typescript
    @ManyToOne(() => Person, { 
      nullable: false,
      filters: { isDeleted: false } // MANDATORY
    })
    person: Person;
    ```
  </Step>
</Steps>

### Part 2: Active Filter Pattern

For entities that are both soft-deletable AND archivable:

```typescript
@Filter({ 
  name: 'active', 
  default: true, 
  cond: { isArchived: false } // ONLY isArchived, NOT isDeleted
})
export class SomeEntity {
  @Property()
  isDeleted: boolean = false;
  
  @Property()
  isArchived: boolean = false;
}
```

<Note>
The `active` filter MUST only check `isArchived: false`, never combine with `isDeleted: false`.
</Note>

### Part 3: Controlled Filter Bypass

`{ filters: false }` is allowed only in these explicit business-intent categories:

<Tabs>
  <Tab title="Merge Operations">
    ```typescript
    // When re-pointing children before soft-deletion
    await em.nativeUpdate(Lead, 
      { person: oldPersonId }, 
      { person: newPersonId },
      { filters: false } // Must hit archived AND soft-deleted children
    );
    ```
  </Tab>
  
  <Tab title="Restore Operations">
    ```typescript
    // When restoring soft-deleted entities
    const deletedEntity = await em.findOne(Entity, 
      { id }, 
      { filters: false } // Need to find soft-deleted entity
    );
    ```
  </Tab>
  
  <Tab title="Admin Functions">
    ```typescript
    // Admin endpoints that need full visibility
    const allEntities = await em.find(Entity, 
      {}, 
      { filters: false } // Admin sees everything
    );
    ```
  </Tab>
</Tabs>

<Warning>
Every `{ filters: false }` call site MUST have a code comment naming the category.
</Warning>

## User Reference Classification

### Tier Classification

<CardGroup cols={3}>
  <Card title="Tier 1: Audit References" icon="clock">
    `createdBy`, `updatedBy`, `archivedBy`, `deletedBy`, `performedBy`
    
    **Rule:** Keep filter override — audit trail must be preserved
  </Card>
  
  <Card title="Tier 2: Operational References" icon="user-gear">
    `assignedTo`, `lastContactedBy`, `approvedBy`
    
    **Rule:** Add filter override — operations continue with soft-deleted users
  </Card>
  
  <Card title="Tier 3: Owner References" icon="crown">
    `person`, `organization`, `owner`, `recipient`
    
    **Rule:** Remove filter override — owners must be active for entity to be visible
  </Card>
</CardGroup>

### DTO Handling for Tier 3

When a Tier-3 relation is populated but `populated.isDeleted === true`:

<Tabs>
  <Tab title="Strategy A: Placeholder">
    ```typescript
    // Use synthetic placeholder
    person: populated.isDeleted ? {
      id: populated.id,
      name: "[Deleted Contact]",
      email: null
    } : fullPersonDto
    ```
  </Tab>
  
  <Tab title="Strategy B: Omit">
    ```typescript
    // Omit the field entirely
    const dto = { ...baseDto };
    if (!populated.isDeleted) {
      dto.person = fullPersonDto;
    }
    return dto;
    ```
  </Tab>
  
  <Tab title="Strategy C: Error">
    ```typescript
    // Return 404 or 410
    if (populated.isDeleted) {
      throw new NotFoundException('Contact no longer available');
    }
    ```
  </Tab>
</Tabs>

<Warning>
NEVER serialize the full Tier-3 payload when `isDeleted === true`.
</Warning>

## Special Case: excludeAiServiceUser Filter

The `excludeAiServiceUser` filter is **opt-in** (`default: false`) to prevent service-stamped rows from disappearing:

```typescript
@Filter({
  name: 'excludeAiServiceUser',
  default: false, // Inverted default
  cond: { isAiServiceUser: false, isSystemServiceUser: false },
})
```

### When to Enable

<Tabs>
  <Tab title="Direct User Queries">
    ```typescript
    // User listings, pickers
    await em.find(User, {}, { 
      filters: { excludeAiServiceUser: true } 
    });
    ```
  </Tab>
  
  <Tab title="UserOrgRole Enumeration">
    ```typescript
    // Member/recipient/agent pools
    await em.find(UserOrgRole, { 
      user: { isAiServiceUser: false } 
    });
    ```
  </Tab>
  
  <Tab title="Auth Lookups">
    ```typescript
    // Login, signup, verify, reset
    await em.findOne(User, { 
      email, 
      isAiServiceUser: false, 
      isSystemServiceUser: false 
    });
    ```
  </Tab>
</Tabs>
