## TL;DR — what every backend dev needs to know

This project uses **MikroORM v6** with `loadStrategy: JOINED`, `autoJoinRefsForFilters: true`, and `filtersOnRelations: true`. Combined with our `isDeleted` and `active` entity filters, this produces a **silent data-loss bug** if entities are not declared correctly:

<Note>
**A row whose own `is_deleted = false` will silently disappear from every query if any of its non-nullable owner relations points at a soft-deleted target.**
</Note>

The fix is **structural** and lives on the entity declaration. Every new entity MUST follow it.

### Quick reference table

| What | Rule |
|------|------|
| Non-nullable `@ManyToOne` / `@OneToOne(owner)` to an entity that declares `@Filter({ name: 'isDeleted', ... })` | Add `{ filters: { isDeleted: false } }` to the relation options |
| `@Filter({ name: 'active', ... })` on a soft-deletable + archivable entity | `cond` MUST be `{ isArchived: false }` only — NEVER `{ isDeleted: false, isArchived: false }` |
| Non-nullable `@ManyToOne` / `@OneToOne(owner)` to an **ARCHIVABLE** target (one that declares `@Filter({ name: 'active' })`) | Decide per §2.2.1: record / history / financial / document / tracker child → override **BOTH** `{ isDeleted: false, active: false }` (persists across owner archive); active-view junction → `{ isDeleted: false }` only (drops on owner archive by design — callers opt into archived rows explicitly) |
| Bulk re-point inside a merge service (`em.nativeUpdate(...)` that re-points children of an entity that is about to be soft-deleted) | Pass `{ filters: false }` so the re-point hits archived AND soft-deleted children |
| `{ filters: false }` anywhere else | Allowed only in the explicit-business-intent categories in §2.3 (merge / restore / conflict-detection / audit-timeline / notification fan-out / admin / seed). Every call site MUST have a code comment naming the category. |
| User reference on a new entity | Classify per §3.1.1 — `createdBy`/`updatedBy`/`archivedBy`/etc. are Tier 1 (audit ref); `assignedTo`/`owner`/`recipient`/`user` are Tier 2 or 3 |
| DTO with a Tier-3 owner relation (`person`, `organization`) | If `populated.isDeleted === true` → DTO MUST use placeholder / omit / 4xx (Strategies A / B / C in §3.3.1). NEVER serialize the full Tier-3 payload when soft-deleted |
| New entity with non-nullable owners | Update the audit table in `QUERY_OPTIMIZATION_PATTERNS.md` §5.5 + extend `DataIntegrityAuditService` for Tier 2/3 (the daily cron picks it up — do NOT add an HTTP endpoint) + add the two mandatory tests from §4.1 |

<Warning>
If you skip any of these, the regression will not show up in unit tests — it surfaces as "this row exists in the DB but the API returns nothing" weeks later.
</Warning>

## 1. Cause — why rows silently disappear

MikroORM v6 (`@mikro-orm/core@6.6.7`) has three defaults that interact badly with soft-deletable graphs:

```typescript
// mikro-orm.config.ts (defaults — NOT to be disabled globally)
loadStrategy: LoadStrategy.JOINED; // populated relations are joined into the main SELECT
autoJoinRefsForFilters: true; // auto-add :ref JOIN for any relation whose target has filters
filtersOnRelations: true; // apply target-entity filters via JOIN ON
```

For every M:1 / 1:1-owner relation on the queried entity, MikroORM checks whether the target entity declares any default-enabled filters (e.g. `@Filter({ name: 'isDeleted', default: true, cond: { isDeleted: false } })`). If so, it adds an auto `:ref` JOIN to that relation **even if you did not populate it**, so the filter can be applied as a JOIN `ON` condition.

For a **non-nullable** M:1 the resulting JOIN type is `INNER JOIN` (see `AbstractSqlDriver.getFieldsForJoinedLoad`):

```typescript
const mandatoryToOneProperty =
  [ReferenceKind.MANY_TO_ONE, ReferenceKind.ONE_TO_ONE].includes(prop.kind) && !prop.nullable;
const joinType = (hint.filter && !prop.nullable) || mandatoryToOneProperty ? INNER_JOIN : LEFT_JOIN;
```

### The failure mode

`em.find(Lead, { id: leadId })` becomes (roughly):

```sql
SELECT lead.*
  FROM lead
  INNER JOIN person p ON p.id = lead.person_id AND p.is_deleted = false
  INNER JOIN organization o ON o.id = lead.organization_id AND o.is_deleted = false
 WHERE lead.id = $1
   AND lead.is_deleted = false;
```

If **any** of those non-nullable joined targets fails the filter (e.g. `Person.isDeleted = true` because the lead's person was merged into another person), the **lead row is silently dropped** — even though `lead.is_deleted = false`.

<Info>
For nullable relations the JOIN is `LEFT JOIN` and the relation hydrates to `null` instead of dropping the row (see MikroORM 6.6 release notes — the v6.5 "Strict relation filters" regression was reverted in 6.6).
</Info>

### 1.1 Real production examples this caused

<AccordionGroup>
  <Accordion title="Two persons merged → archived Lead becomes invisible">
    Secondary `Person.isDeleted = true` → an archived Lead still pointing at the secondary Person
    
    **Symptom:** The archived Lead was invisible everywhere even though it existed in the DB
  </Accordion>
  
  <Accordion title="DealStage cleanup → Deal vanishes">
    Custom stage soft-deleted while a Deal still pointed at it
    
    **Symptom:** The Deal vanished from list endpoints
  </Accordion>
  
  <Accordion title="Agent leaves org → created records disappear">
    `User.isDeleted = true` → every Deal / Contact / Company / Listing they `createdBy` would have vanished if we shipped the relation override blindly without the Tier-1 carve-out
  </Accordion>
</AccordionGroup>

### 1.2 Special case — the `excludeAiServiceUser` filter is OPT-IN (`default: false`)

The same `autoJoinRefsForFilters` mechanic bit a **non-soft-delete** filter. The `User` entity declares a second filter that hides **all non-human service accounts** — both the per-org **AI agent service account** (`isAiServiceUser = true`, the row ORGANIZATION-scope AI agents act as) and the per-org **System automation actor** (`isSystemServiceUser = true`, the row distribution / escalation / lead-capture stamp as `createdBy`/`assignedBy`/`performedBy`):

```typescript
@Filter({
  name: 'excludeAiServiceUser',
  default: false,
  cond: { isAiServiceUser: false, isSystemServiceUser: false },
})
```

This filter was originally `default: true` to hide the service accounts from every user listing. But each is a **real `user` row** that stamps non-nullable audit columns (`created_by`, `changed_by`, `performed_by`, …) on anything it touches. With `default: true`, MikroORM auto-joined `... INNER JOIN "user" u ON ... AND u.is_ai_service_user = false` (and now `... AND u.is_system_service_user = false`) for those non-nullable `@ManyToOne(User)` relations and **silently dropped** every row a service account created (e.g. an AI-created `Contact`, or a system-distributed lead's stakeholder, returning 404 immediately after creation).

#### Resolution — invert the default to opt-in (`default: false`)

<Steps>
  <Step title="Filter is OFF by default">
    The filter is **OFF by default**, so audit-relation auto-joins no longer drop service-stamped rows. This is the inverse of the `isDeleted` strategy: for `isDeleted` we keep the global filter ON and override per-relation; for `excludeAiServiceUser` we keep it OFF globally and re-enable per-query where the service accounts must be hidden.
  </Step>
  
  <Step title="User-facing enumerations must re-exclude the service accounts explicitly">
    
| Query shape | How to exclude the service accounts |
|-------------|-------------------------------------|
| Direct `User` root query (listings, pickers) | `{ filters: { excludeAiServiceUser: true } }` in the query options (now hides both accounts) |
| `UserOrgRole` enumeration (member/recipient/agent pools) | `user: { isAiServiceUser: false }` in the `where` (the System actor has **no** `UserOrgRole`, so it is already absent) |
| Raw SQL over `user_org_roles` with no `team_membership` join (e.g. `getOrgAgentIds`) | `JOIN "user" u ON u.id = uor.user_id AND u.is_ai_service_user = false` (System actor absent — no `UserOrgRole`) |
| `QueryBuilder` user lists (filters are NOT auto-applied) | `qb.andWhere({ isAiServiceUser: false, isSystemServiceUser: false })` |
| Auth lookups by email (signup / login / verify / reset) | `{ email, isAiServiceUser: false, isSystemServiceUser: false }` |

  </Step>
  
  <Step title="Inherently service-free queries (no change needed)">
    Queries scoped to a specific `user: <id>`; queries filtered by `User.organizations` (neither account is added to that M2M); any `TeamMembership`-based query (neither has team membership); and **any `UserOrgRole`-based query** (the System actor has no `UserOrgRole` at all, so only direct global `User` queries need to exclude it).
  </Step>
  
  <Step title="DTO exposure">
    The service accounts can surface as actors (`createdBy` / `changedBy` / `assignedBy` / `performedBy`). Their synthetic `ai-user+<orgId>@propwise.com` / `system-user+<orgId>@propwise.com` emails are **not** masked in user DTOs — they are non-routable internal addresses and the display names ("AI USER" / "System Automation") already identify them. The user-facing enumerations above keep them out of pickers/listings, so the emails are only ever seen in actor attribution where surfacing them is harmless.
  </Step>
</Steps>

<Note>
The actor resolver + provisioning seeder rely on the inverted default — the service accounts resolve by default without passing `{ filters: { excludeAiServiceUser: false } }`. `resolveSystemActorId` finds the System actor via a direct `em.findOne(User, { isSystemServiceUser: true })`.
</Note>

## 2. Solution — filter overrides + `{ filters: false }` governance

### 2.1 Entity-level — relation filter override

Every non-nullable `@ManyToOne` / `@OneToOne(owner)` to a soft-deletable target MUST override the `isDeleted` filter:

```typescript
@ManyToOne(() => Person, {
  nullable: false,
  filters: { isDeleted: false }, // ← MANDATORY override
})
person!: Ref<Person>;

@ManyToOne(() => Organization, {
  nullable: false,
  filters: { isDeleted: false }, // ← MANDATORY override
})
organization!: Ref<Organization>;
```

<Warning>
**Do NOT use `{ filters: false }`** (disables all filters) — use `{ filters: { isDeleted: false } }` (disables only `isDeleted`). Other filters (e.g. `active`, tenant-scoped filters) must remain enabled.
</Warning>

#### Why this works

MikroORM still auto-joins the target (because `autoJoinRefsForFilters: true` sees that the target declares filters), but the per-relation override replaces the filter's JOIN condition with a no-op. The resulting JOIN becomes:

```sql
INNER JOIN person p ON p.id = lead.person_id -- no `AND p.is_deleted = false`
```

The person row loads even if soft-deleted. Your DTO logic (see §3) handles the presentation.

---

### 2.2 The `active` filter — archivable entities

Many entities are both **soft-deletable** (`isDeleted`) and **archivable** (`isArchived`). The `active` filter is a convenience shorthand for "not archived":

<Check>
**Correct definition:**
</Check>

```typescript
@Filter({
  name: 'active',
  default: true,
  cond: { isArchived: false }, // ← ONLY isArchived
})
```

<Warning>
**NEVER do this:**
</Warning>

```typescript
@Filter({
  name: 'active',
  default: true,
  cond: { isDeleted: false, isArchived: false }, // ← WRONG
})
```

**Reason:** If `active` includes `isDeleted: false`, then disabling `active` (`{ filters: { active: false } }`) also disables the soft-delete filter, causing the same silent-drop bug on nested relations. The `isDeleted` filter is project-global and overridden per-relation; `active` is entity-local and toggled per-query.

#### 2.2.1 Archivable owner relations — dual override

For a child entity with a non-nullable archivable owner (e.g. `Deal` → `Person` where `Person` declares both `isDeleted` and `active`), decide:

<Tabs>
  <Tab title="Record / history / financial / document / tracker child">
    **Override BOTH filters:**
    
```typescript
@ManyToOne(() => Person, {
  nullable: false,
  filters: { isDeleted: false, active: false },
})
person!: Ref<Person>;
```

**Behavior:** The child persists across owner archive. An archived person's deal history remains visible to users who explicitly fetch archived persons.
  </Tab>
  
  <Tab title="Active-view junction">
    **Override only `isDeleted`:**
    
```typescript
@ManyToOne(() => Person, {
  nullable: false,
  filters: { isDeleted: false },
})
person!: Ref<Person>;
```

**Behavior:** The junction row drops when the owner is archived (by design — callers opt into archived rows explicitly).
  </Tab>
</Tabs>

### 2.3 Query-level — `{ filters: false }` governance

<Warning>
`{ filters: false }` disables **all** filters (soft-delete, active, tenant-scoped, etc.). It is a dangerous override and allowed only in these categories:
</Warning>

<AccordionGroup>
  <Accordion title="1. Merge operations">
    Re-pointing children from a merge source that is about to be soft-deleted
    
```typescript
// ALLOWED: merge / bulk re-point (targets archived + soft-deleted children)
await em.nativeUpdate(
  Lead,
  { person: sourcePersonId },
  { person: targetPersonId },
  { filters: false } // ← mandatory for merge
);
```

**Code comment required:** `// filters:false — merge bulk re-point (archived + soft-deleted)`
  </Accordion>
  
  <Accordion title="2. Restore operations">
    Un-soft-deleting an entity and its dependents
    
```typescript
// ALLOWED: restore service (needs to find soft-deleted target)
const deletedPerson = await em.findOne(
  Person,
  { id },
  { filters: false }
);
```

**Code comment required:** `// filters:false — restore lookup`
  </Accordion>
  
  <Accordion title="3. Conflict detection">
    Checking for duplicates across archived / soft-deleted rows
    
```typescript
// ALLOWED: conflict detection (must check archived + soft-deleted)
const existingEmail = await em.findOne(
  Person,
  { email, organization },
  { filters: false }
);
```

**Code comment required:** `// filters:false — conflict detection (all statuses)`
  </Accordion>
  
  <Accordion title="4. Audit timeline">
    Admin / audit views that show the full history
    
```typescript
// ALLOWED: audit timeline (show deleted + archived)
const fullHistory = await em.find(
  Activity,
  { deal: dealId },
  { filters: false, orderBy: { createdAt: 'desc' } }
);
```

**Code comment required:** `// filters:false — audit timeline (all statuses)`
  </Accordion>
  
  <Accordion title="5. Notification fan-out">
    Webhook / email fan-out that must reach archived / deactivated recipients
    
```typescript
// ALLOWED: notification fan-out (include archived recipients)
const allStakeholders = await em.find(
  DealStakeholder,
  { deal: dealId },
  { filters: false, populate: ['user'] }
);
```

**Code comment required:** `// filters:false — notification fan-out (archived recipients)`
  </Accordion>
  
  <Accordion title="6. Admin / superuser operations">
    Platform admin endpoints that operate across all statuses
    
```typescript
// ALLOWED: admin endpoint (superuser all-status access)
const allOrgs = await em.find(
  Organization,
  {},
  { filters: false }
);
```

**Code comment required:** `// filters:false — admin endpoint (superuser)`
  </Accordion>
  
  <Accordion title="7. Seed / migration scripts">
    One-off scripts that manipulate test / seed data
    
```typescript
// ALLOWED: seed script (test data setup)
await em.nativeDelete(User, {}, { filters: false });
```

**Code comment required:** `// filters:false — seed script`
  </Accordion>
</AccordionGroup>

<Warning>
**Every `{ filters: false }` call site MUST have a code comment naming the category.** PRs with uncommented `{ filters: false }` will be rejected.
</Warning>

## 3. User references — the three tiers

`User` relations fall into three tiers based on how they handle soft-deleted users:

### 3.1.1 Tier classification

<Tabs>
  <Tab title="Tier 1 — audit references">
    **Definition:** Immutable actor attribution (who / when)
    
**Examples:** `createdBy`, `updatedBy`, `deletedBy`, `archivedBy`, `changedBy`, `performedBy`, `assignedBy`, `claimedBy`

**Filter override:**
```typescript
@ManyToOne(() => User, {
  nullable: false,
  // NO filter override — allow soft-deleted users
})
createdBy!: Ref<User>;
```

**DTO:** Serialize the user normally even if soft-deleted (the display name / email are historical actor context)
  </Tab>
  
  <Tab title="Tier 2 — active role references">
    **Definition:** Current role / responsibility (active assignment)
    
**Examples:** `assignedTo`, `owner`, `manager`, `approver`

**Filter override:**
```typescript
@ManyToOne(() => User, {
  nullable: false,
  filters: { isDeleted: false }, // ← override
})
assignedTo!: Ref<User>;
```

**DTO:** Serialize normally if active; if soft-deleted → use placeholder / omit / return 409 (business rule dependent)
  </Tab>
  
  <Tab title="Tier 3 — structural owner references">
    **Definition:** The entity belongs to this user (ownership)
    
**Examples:** `user` (on `UserProfile`, `UserPreferences`, etc.)

**Filter override:**
```typescript
@ManyToOne(() => User, {
  nullable: false,
  filters: { isDeleted: false }, // ← override
})
user!: Ref<User>;
```

**DTO:** Must use placeholder / omit / return 4xx if soft-deleted (see §3.3.1)
  </Tab>
</Tabs>

### 3.2 Person / Organization references — always Tier 3

<Warning>
`Person` and `Organization` are **always Tier 3** (structural owner). Unlike `User`, they have no audit-ref carve-out.
</Warning>

**Entity:**
```typescript
@ManyToOne(() => Person, {
  nullable: false,
  filters: { isDeleted: false }, // ← MANDATORY
})
person!: Ref<Person>;

@ManyToOne(() => Organization, {
  nullable: false,
  filters: { isDeleted: false }, // ← MANDATORY
})
organization!: Ref<Organization>;
```

**DTO:** See §3.3.1 for the three strategies.

### 3.3 DTO handling — soft-deleted owners

#### 3.3.1 Three strategies

When a Tier-3 owner is soft-deleted (`populated.isDeleted === true`), choose one:

<Tabs>
  <Tab title="Strategy A — placeholder">
    Replace with a placeholder object
    
```typescript
// DTO serialization
if (lead.person.isDeleted) {
  dto.person = {
    id: lead.person.id,
    displayName: '[Deleted Person]',
    isDeleted: true,
  };
} else {
  dto.person = this.personMapper.toDto(lead.person);
}
```

**Use when:** The frontend must render something (e.g. a lead card) but can tolerate a placeholder
  </Tab>
  
  <Tab title="Strategy B — omit">
    Omit the relation from the DTO
    
```typescript
// DTO class
class LeadDto {
  person?: PersonDto; // ← optional
}

// Serialization
if (!lead.person.isDeleted) {
  dto.person = this.personMapper.toDto(lead.person);
}
// omitted if deleted
```

**Use when:** The relation is optional for the frontend use case
  </Tab>
  
  <Tab title="Strategy C — return 4xx">
    Return `409 Conflict` / `410 Gone` and force the frontend to handle it
    
```typescript
// Service method
const lead = await em.findOneOrFail(Lead, { id }, { populate: ['person'] });
if (lead.person.isDeleted) {
  throw new ConflictException('Lead person has been deleted');
}
```

**Use when:** The frontend cannot function without the relation (e.g. a detail view)
  </Tab>
</Tabs>

<Check>
**Project standard:** Use **Strategy A** (placeholder) for list endpoints; **Strategy C** (4xx) for detail / edit endpoints
</Check>

### 3.4 User tier-1 carve-out — why audit refs remain unfiltered

The project originally wanted to add `{ filters: { isDeleted: false } }` to **all** user relations (including `createdBy` / `updatedBy` / etc.). This was blocked by a critical edge case:

<Warning>
**Scenario:** An agent leaves the org → `User.isDeleted = true` → every entity they `createdBy` would vanish if the `createdBy` relation had the filter override
</Warning>

The solution: **Tier 1 relations (audit refs) do NOT override the filter.** They remain governed by the global `isDeleted` filter, which is disabled when the soft-deleted user is populated:

```typescript
// The entity does NOT override isDeleted on createdBy
@ManyToOne(() => User, { nullable: false })
createdBy!: Ref<User>;

// The global isDeleted filter applies...
// but the createdBy relation is NEVER queried standalone.
// It's only ever loaded as a populate from the parent query (e.g. em.find(Deal, ...)),
// and the parent query's filter is on the Deal table, not the User table.
```

**DTO:** Serialize the soft-deleted user normally — the display name / email are immutable historical context (who created this at that timestamp).

## 4. Testing requirements

Every new entity with non-nullable Tier-2 / Tier-3 owners MUST add these two tests:

<Steps>
  <Step title="Test 1 — soft-deleted owner does not drop the child">
    
```typescript
it('should return child entity even when owner is soft-deleted', async () => {
  const owner = await factory.create(Person);
  const child = await factory.create(Lead, { person: owner });
  
  owner.isDeleted = true;
  await em.flush();
  
  const found = await service.findOne(child.id);
  expect(found).toBeDefined();
  expect(found.id).toBe(child.id);
});
```
  </Step>
  
  <Step title="Test 2 — DTO handles soft-deleted owner correctly">
    
```typescript
it('should return placeholder when owner is soft-deleted', async () => {
  const owner = await factory.create(Person);
  const child = await factory.create(Lead, { person: owner });
  
  owner.isDeleted = true;
  await em.flush();
  
  const dto = await service.findOneDto(child.id);
  expect(dto.person.displayName).toBe('[Deleted Person]');
  expect(dto.person.isDeleted).toBe(true);
});
```
  </Step>
</Steps>

<Warning>
These tests will FAIL if the entity declaration is wrong (the child will be `undefined` in Test 1). Do NOT merge the PR until both pass.
</Warning>

## 5. Data integrity audit

The `DataIntegrityAuditService` runs daily and checks for "orphaned" children (children whose owner is soft-deleted but the child is not). Every new entity with Tier-2 / Tier-3 owners MUST:

<Steps>
  <Step title="Update the reference table">
    Add a row to the table in `QUERY_OPTIMIZATION_PATTERNS.md` §5.5
  </Step>
  
  <Step title="Extend the audit service">
    Add a check method to `DataIntegrityAuditService`:
    
```typescript
async auditLeadPersonIntegrity(): Promise<AuditResult> {
  const orphans = await this.em.find(Lead, {
    person: { isDeleted: true },
    isDeleted: false,
  });
  
  return {
    entity: 'Lead',
    relation: 'person',
    orphanCount: orphans.length,
    orphanIds: orphans.map(l => l.id),
  };
}
```
  </Step>
  
  <Step title="Add to the daily cron">
    The cron scheduler picks up the new method automatically (it discovers all `audit*Integrity` methods via reflection)
  </Step>
</Steps>

<Warning>
Do NOT add an HTTP endpoint for the audit — it runs as a daily background cron. Platform admins view results in the audit log table.
</Warning>

## 7. Common mistakes

<AccordionGroup>
  <Accordion title="❌ Using `{ filters: false }` on a relation">
    **Wrong:**
```typescript
@ManyToOne(() => Person, {
  nullable: false,
  filters: false, // ← disables ALL filters (including tenant-scoped)
})
person!: Ref<Person>;
```

**Correct:**
```typescript
@ManyToOne(() => Person, {
  nullable: false,
  filters: { isDeleted: false }, // ← disables only isDeleted
})
person!: Ref<Person>;
```
  </Accordion>
  
  <Accordion title="❌ Including `isDeleted` in the `active` filter">
    **Wrong:**
```typescript
@Filter({
  name: 'active',
  default: true,
  cond: { isDeleted: false, isArchived: false },
})
```

**Correct:**
```typescript
@Filter({
  name: 'active',
  default: true,
  cond: { isArchived: false },
})
```
  </Accordion>
  
  <Accordion title="❌ Forgetting the filter override on nullable relations">
    Even nullable relations should override if the relation is business-critical:
    
```typescript
@ManyToOne(() => DealStage, {
  nullable: true,
  filters: { isDeleted: false }, // ← still needed
})
stage?: Ref<DealStage>;
```
  </Accordion>
  
  <Accordion title="❌ Serializing full Tier-3 owner when soft-deleted">
    **Wrong:**
```typescript
dto.person = this.personMapper.toDto(lead.person); // even if isDeleted
```

**Correct:**
```typescript
if (lead.person.isDeleted) {
  dto.person = { id: lead.person.id, displayName: '[Deleted Person]', isDeleted: true };
} else {
  dto.person = this.personMapper.toDto(lead.person);
}
```
  </Accordion>
  
  <Accordion title="❌ Using `{ filters: false }` without a code comment">
    **Wrong:**
```typescript
await em.find(Lead, {}, { filters: false });
```

**Correct:**
```typescript
// filters:false — admin endpoint (superuser)
await em.find(Lead, {}, { filters: false });
```
  </Accordion>
</AccordionGroup>
