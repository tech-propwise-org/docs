This document is the authoritative reference for the soft-delete / archive filter pattern. See `QUERY_OPTIMIZATION_PATTERNS.md` §5.5 for MikroORM internals.

## Overview

This project uses **MikroORM v6** with `loadStrategy: JOINED`, `autoJoinRefsForFilters: true`, and `filtersOnRelations: true`. Combined with our `isDeleted` and `active` entity filters, this produces a **silent data-loss bug** if entities are not declared correctly.

<Note>
**Critical Issue:** A row whose own `is_deleted = false` will silently disappear from every query if any of its non-nullable owner relations points at a soft-deleted target.
</Note>

## Quick Reference

Every backend developer must follow these mandatory rules:

<AccordionGroup>
<Accordion title="Non-nullable Relations">
**Rule:** Add `{ filters: { isDeleted: false } }` to relation options

```typescript
@ManyToOne(() => Person, { filters: { isDeleted: false } })
person!: Person;
```
</Accordion>

<Accordion title="Active Filter on Soft-deletable Entities">
**Rule:** `cond` MUST be `{ isArchived: false }` only — NEVER `{ isDeleted: false, isArchived: false }`

```typescript
@Filter({ name: 'active', default: true, cond: { isArchived: false } })
```
</Accordion>

<Accordion title="Bulk Operations in Merge Services">
**Rule:** Pass `{ filters: false }` for re-pointing operations

```typescript
em.nativeUpdate(Entity, criteria, { newParentId }, { filters: false })
```
</Accordion>

<Accordion title="User References">
**Rule:** Classify per §3.1.1 — audit refs vs business refs have different handling
</Accordion>
</AccordionGroup>

## Root Cause Analysis

### Why Rows Silently Disappear

MikroORM v6 has three defaults that interact badly with soft-deletable graphs:

```typescript
// mikro-orm.config.ts (defaults — NOT to be disabled globally)
loadStrategy: LoadStrategy.JOINED; // populated relations are joined into the main SELECT
autoJoinRefsForFilters: true; // auto-add :ref JOIN for any relation whose target has filters
filtersOnRelations: true; // apply target-entity filters via JOIN ON
```

<Info>
For every M:1 / 1:1-owner relation on the queried entity, MikroORM checks whether the target entity declares any default-enabled filters. If so, it adds an auto `:ref` JOIN to that relation **even if you did not populate it**.
</Info>

### The Failure Mode

For **non-nullable** M:1 relations, the resulting JOIN type is `INNER JOIN`:

```sql
SELECT lead.*
  FROM lead
  INNER JOIN person p ON p.id = lead.person_id AND p.is_deleted = false
  INNER JOIN organization o ON o.id = lead.organization_id AND o.is_deleted = false
 WHERE lead.id = $1
   AND lead.is_deleted = false;
```

<Warning>
If **any** non-nullable joined target fails the filter (e.g., `Person.isDeleted = true`), the lead row is silently dropped — even though `lead.is_deleted = false`.
</Warning>

## The Three-Part Fix

<Steps>
<Step title="Per-relation Filter Override">
Declare `{ filters: { isDeleted: false } }` on **every non-nullable** relation to soft-deletable entities.

```typescript
// ✅ CORRECT — surgical, structural, zero side effects
@ManyToOne(() => Person, { filters: { isDeleted: false } })
person!: Person;

@ManyToOne(() => Organization, { filters: { isDeleted: false } })
organization!: Organization;

@ManyToOne(() => User, { filters: { isDeleted: false } })
createdBy!: User;
```

<Note>
This disables the target's `isDeleted` filter **only** for the auto-`:ref` JOIN. Standalone queries still apply the filter normally.
</Note>
</Step>

<Step title="Active Filter Cleanup">
Every archivable entity should have **two** entity-level filters:

```typescript
@Entity()
@Filter({ name: 'isDeleted', default: true, cond: { isDeleted: false } })
@Filter({ name: 'active', default: true, cond: { isArchived: false } })  // ← NOT { isDeleted: false, isArchived: false }
export class Lead { ... }
```

<Warning>
The `active` filter MUST omit `isDeleted` because the standalone `isDeleted` filter already handles soft-delete exclusion.
</Warning>
</Step>

<Step title="Controlled Usage of filters: false">
`{ filters: false }` is reserved for explicit business intent only. See permitted categories below.
</Step>
</Steps>

### Filter Usage Invariants

<Check>
**Valid Combinations:**
- Default: Both filters enabled (`isDeleted = false AND isArchived = false`)
- Include archived: `{ filters: { active: false } }`
- Include soft-deleted: `{ filters: { isDeleted: false } }` (rare)
- Include both: `{ filters: false }` (restricted use cases only)
</Check>

<Warning>
**Invalid:** Never disable `isDeleted` while keeping `active` — that would return soft-deleted records.
</Warning>

## Permitted Uses of filters: false

`{ filters: false }` disables BOTH `isDeleted` and `active` filters. Use only for explicit business intent, never as a workaround.

<Tabs>
<Tab title="Legitimate Categories">
Every call site **MUST** have a code comment naming the category:

| Category | When to Use | Example |
|----------|-------------|---------|
| **Merge/Restore Operations** | Bulk re-pointing during merges | `// MERGE: Re-point children before soft-delete` |
| **Conflict Detection** | Checking for existing soft-deleted records | `// CONFLICT: Check for deleted duplicates` |
| **Audit Timeline** | Historical data reconstruction | `// AUDIT: Full timeline including deleted` |
| **Admin Operations** | Administrative data access | `// ADMIN: Super-user data recovery` |
| **Notification Fan-out** | Alerting on deleted entity changes | `// NOTIFICATION: Alert on deleted entity update` |
| **Seed Data** | Database initialization | `// SEED: Bootstrap with archived templates` |
</Tab>

<Tab title="Code Examples">
```typescript
// ✅ CORRECT - with required comment
async mergePersons(primaryId: string, secondaryId: string) {
  // MERGE: Re-point all children before soft-deleting secondary
  await this.em.nativeUpdate(
    Lead, 
    { person: secondaryId }, 
    { person: primaryId },
    { filters: false }
  );
}

// ❌ INCORRECT - missing category comment
async findLead(id: string) {
  return this.em.findOne(Lead, { id }, { filters: false });
}
```
</Tab>
</Tabs>

## User Reference Classification

### Tier Classification System

<AccordionGroup>
<Accordion title="Tier 1: Audit References">
**Examples:** `createdBy`, `updatedBy`, `archivedBy`, `mergedBy`

- **Policy:** Never apply `filters: { isDeleted: false }` override
- **Rationale:** Audit integrity requires preserving historical attributions
- **DTO Handling:** Always serialize, even if user is soft-deleted
</Accordion>

<Accordion title="Tier 2: Business Attribution">
**Examples:** `assignedTo`, `lastContactedBy`, `reviewedBy`

- **Policy:** Apply `filters: { isDeleted: false }` override
- **DTO Handling:** Warn-log when populated user is soft-deleted
- **Fallback:** Use placeholder or omit from response
</Accordion>

<Accordion title="Tier 3: Core Business Entities">
**Examples:** `person`, `organization`, `primaryContact`

- **Policy:** Apply `filters: { isDeleted: false }` override
- **DTO Handling:** NEVER serialize full payload when soft-deleted
- **Strategies:** Placeholder, omit, or return 4xx error
</Accordion>
</AccordionGroup>

### DTO Strategies for Soft-deleted References

<Tabs>
<Tab title="Strategy A: Placeholder">
```typescript
// For Tier 2/3 references
if (populated.isDeleted) {
  dto.assignedTo = { 
    id: populated.id, 
    name: "[Deleted User]", 
    isDeleted: true 
  };
}
```
</Tab>

<Tab title="Strategy B: Omit">
```typescript
// Conditionally include the field
if (!populated.isDeleted) {
  dto.assignedTo = this.mapUser(populated);
}
// Field absent from response when deleted
```
</Tab>

<Tab title="Strategy C: Error Response">
```typescript
// For critical Tier 3 references
if (dto.person?.isDeleted) {
  throw new BadRequestException(
    'Referenced person has been deleted'
  );
}
```
</Tab>
</Tabs>

## Testing Requirements

### Mandatory Test Coverage

Every new entity with non-nullable owners must include these tests:

<CodeGroup>
```typescript Test 1: Silent Drop Prevention
describe('Soft-delete filter protection', () => {
  it('should not drop rows when owner is soft-deleted', async () => {
    // Arrange: Create entity with owner
    const owner = await factory.create(Person);
    const entity = await factory.create(Lead, { person: owner });
    
    // Act: Soft-delete the owner
    await em.nativeUpdate(Person, { id: owner.id }, { isDeleted: true });
    
    // Assert: Entity still findable
    const found = await em.findOne(Lead, { id: entity.id });
    expect(found).toBeDefined();
    expect(found.id).toBe(entity.id);
  });
});
```

```typescript Test 2: Populated Relation Handling
describe('Populated soft-deleted relations', () => {
  it('should handle soft-deleted populated owner gracefully', async () => {
    // Arrange: Entity with soft-deleted owner
    const owner = await factory.create(Person, { isDeleted: true });
    const entity = await factory.create(Lead, { person: owner });
    
    // Act: Find with populate
    const found = await em.findOne(
      Lead, 
      { id: entity.id }, 
      { populate: ['person'] }
    );
    
    // Assert: Entity found, populated relation indicates deletion
    expect(found).toBeDefined();
    expect(found.person.isDeleted).toBe(true);
  });
});
```
</CodeGroup>
