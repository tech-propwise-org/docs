<Warning>
**Status:** Project-wide standard — MANDATORY for all entities, services, and DTOs
This document is the authoritative reference for the soft-delete / archive filter pattern.
</Warning>

## TL;DR — what every backend dev needs to know

This project uses **MikroORM v6** with `loadStrategy: JOINED`, `autoJoinRefsForFilters: true`, and `filtersOnRelations: true`. Combined with our `isDeleted` and `active` entity filters, this produces a **silent data-loss bug** if entities are not declared correctly:

<Note>
**A row whose own `is_deleted = false` will silently disappear from every query if any of its non-nullable owner relations points at a soft-deleted target.**
</Note>

The fix is **structural** and lives on the entity declaration. Every new entity MUST follow it.

| What | Rule |
|------|------|
| Non-nullable `@ManyToOne` / `@OneToOne(owner)` to an entity that declares `@Filter({ name: 'isDeleted', ... })` | Add `{ filters: { isDeleted: false } }` to the relation options |
| `@Filter({ name: 'active', ... })` on a soft-deletable + archivable entity | `cond` MUST be `{ isArchived: false }` only — NEVER `{ isDeleted: false, isArchived: false }` |
| Bulk re-point inside a merge service (`em.nativeUpdate(...)` that re-points children of an entity that is about to be soft-deleted) | Pass `{ filters: false }` so the re-point hits archived AND soft-deleted children |
| `{ filters: false }` anywhere else | Allowed only in the explicit-business-intent categories in §2.3 (merge / restore / conflict-detection / audit-timeline / notification fan-out / admin / seed). Every call site MUST have a code comment naming the category. |
| User reference on a new entity | Classify per §3.1.1 — `createdBy`/`updatedBy`/`archivedBy`/etc. are Tier 1 (audit ref); `assignedTo`/`owner`/`recipient`/`user` are Tier 2 or 3 |
| DTO with a Tier-3 owner relation (`person`, `organization`) | If `populated.isDeleted === true` → DTO MUST use placeholder / omit / 4xx (Strategies A / B / C in §3.3.1). NEVER serialize the full Tier-3 payload when soft-deleted |
| New entity with non-nullable owners | Update the audit table in `QUERY_OPTIMIZATION_PATTERNS.md` §5.5 + extend `DataIntegrityAuditService` for Tier 2/3 (the daily cron picks it up — do NOT add an HTTP endpoint) + add the two mandatory tests from §4.1 |

<Warning>
If you skip any of these, the regression will not show up in unit tests — it surfaces as "this row exists in the DB but the API returns nothing" weeks later.
</Warning>

## 1. Cause — why rows silently disappear

MikroORM v6 (`@mikro-orm/core@6.6.7`) has three defaults that interact badly with soft-deletable graphs:

```typescript
// mikro-orm.config.ts (defaults — NOT to be disabled globally)
loadStrategy: LoadStrategy.JOINED; // populated relations are joined into the main SELECT
autoJoinRefsForFilters: true; // auto-add :ref JOIN for any relation whose target has filters
filtersOnRelations: true; // apply target-entity filters via JOIN ON
```

For every M:1 / 1:1-owner relation on the queried entity, MikroORM checks whether the target entity declares any default-enabled filters (e.g. `@Filter({ name: 'isDeleted', default: true, cond: { isDeleted: false } })`). If so, it adds an auto `:ref` JOIN to that relation **even if you did not populate it**, so the filter can be applied as a JOIN `ON` condition.

For a **non-nullable** M:1 the resulting JOIN type is `INNER JOIN`:

```typescript
const mandatoryToOneProperty =
  [ReferenceKind.MANY_TO_ONE, ReferenceKind.ONE_TO_ONE].includes(prop.kind) && !prop.nullable;
const joinType = (hint.filter && !prop.nullable) || mandatoryToOneProperty ? INNER_JOIN : LEFT_JOIN;
```

**The failure mode**: `em.find(Lead, { id: leadId })` becomes (roughly):

```sql
SELECT lead.*
  FROM lead
  INNER JOIN person p ON p.id = lead.person_id AND p.is_deleted = false
  INNER JOIN organization o ON o.id = lead.organization_id AND o.is_deleted = false
 WHERE lead.id = $1
   AND lead.is_deleted = false;
```

<Warning>
If **any** of those non-nullable joined targets fails the filter (e.g. `Person.isDeleted = true` because the lead's person was merged into another person), the **lead row is silently dropped** — even though `lead.is_deleted = false`.
</Warning>

For nullable relations the JOIN is `LEFT JOIN` and the relation hydrates to `null` instead of dropping the row (see MikroORM 6.6 release notes — the v6.5 "Strict relation filters" regression was reverted in 6.6).

### Real production examples this caused

| Scenario | Symptom |
|----------|---------|
| Two persons merged → secondary `Person.isDeleted = true` → an archived Lead still pointing at the secondary Person | The archived Lead was invisible everywhere even though it existed in the DB |
| `DealStage` cleanup → custom stage soft-deleted while a Deal still pointed at it | The Deal vanished from list endpoints |
| Agent leaves the org → `User.isDeleted = true` → every Deal / Contact / Company / Listing they `createdBy` would have vanished if we shipped the relation override blindly without the Tier-1 carve-out | |

### Special case — the `excludeAiServiceUser` filter is OPT-IN (`default: false`)

The same `autoJoinRefsForFilters` mechanic bit a **non-soft-delete** filter. The `User` entity declares a second filter that hides **all non-human service accounts**:

```typescript
@Filter({
  name: 'excludeAiServiceUser',
  default: false,
  cond: { isAiServiceUser: false, isSystemServiceUser: false },
})
```

This filter was originally `default: true` to hide the service accounts from every user listing. But each is a **real `user` row** that stamps non-nullable audit columns on anything it touches. With `default: true`, MikroORM auto-joined `... INNER JOIN "user" u ON ... AND u.is_ai_service_user = false` and **silently dropped** every row a service account created.

<Note>
**Resolution — invert the default to opt-in (`default: false`):**

The filter is **OFF by default**, so audit-relation auto-joins no longer drop service-stamped rows. This is the inverse of the `isDeleted` strategy: for `isDeleted` we keep the global filter ON and override per-relation; for `excludeAiServiceUser` we keep it OFF globally and re-enable per-query where the service accounts must be hidden.
</Note>

**User-facing enumerations must re-exclude the service accounts explicitly:**

| Query shape | How to exclude the service accounts |
|-------------|-----------------------------------|
| Direct `User` root query (listings, pickers) | `{ filters: { excludeAiServiceUser: true } }` in the query options |
| `UserOrgRole` enumeration (member/recipient/agent pools) | `user: { isAiServiceUser: false }` in the `where` (the System actor has **no** `UserOrgRole`) |
| Raw SQL over `user_org_roles` with no `team_membership` join | `JOIN "user" u ON u.id = uor.user_id AND u.is_ai_service_user = false` |
| `QueryBuilder` user lists (filters are NOT auto-applied) | `qb.andWhere({ isAiServiceUser: false, isSystemServiceUser: false })` |
| Auth lookups by email (signup / login / verify / reset) | `{ email, isAiServiceUser: false, isSystemServiceUser: false }` |

## 2. The fix — three coordinated parts

The fix has **three** parts. Skipping any one of them re-introduces the bug.

### 2.1 Per-relation `{ filters: { isDeleted: false } }` override (MANDATORY structural)

Declare `{ filters: { isDeleted: false } }` on **every non-nullable** `@ManyToOne` / `@OneToOne(owner)` relation whose target entity has the `isDeleted` filter:

<CodeGroup>

```typescript Bad - Will cause silent data loss
@Entity()
export class Lead {
  @ManyToOne({ entity: () => Person })
  person!: Person; // ❌ WRONG — non-nullable M:1 to soft-deletable entity

  @ManyToOne({ entity: () => Organization })
  organization!: Organization; // ❌ WRONG — will drop Lead if Organization.isDeleted = true
}
```

```typescript Good - Prevents silent data loss
@Entity()
export class Lead {
  @ManyToOne({ 
    entity: () => Person,
    filters: { isDeleted: false } // ✅ CORRECT — override the filter
  })
  person!: Person;

  @ManyToOne({ 
    entity: () => Organization,
    filters: { isDeleted: false } // ✅ CORRECT — prevents row from disappearing
  })
  organization!: Organization;
}
```

</CodeGroup>

### 2.2 `active` filter must exclude only `isArchived` (NOT `isDeleted`)

For entities that are both **soft-deletable** and **archivable**, the `active` filter must check **only** `isArchived`:

<CodeGroup>

```typescript Bad - Double-filtering causes issues
@Filter({
  name: 'active',
  default: true,
  cond: { isDeleted: false, isArchived: false }, // ❌ WRONG — double-filters
})
@Entity()
export class Deal {
  // ...
}
```

```typescript Good - Single archive filter
@Filter({
  name: 'active',
  default: true,
  cond: { isArchived: false }, // ✅ CORRECT — only archive filter
})
@Entity()
export class Deal {
  // ...
}
```

</CodeGroup>

### 2.3 Limited `{ filters: false }` usage (explicit business intent only)

`{ filters: false }` bypasses **all** filters and is allowed only in these categories:

<AccordionGroup>
<Accordion title="Merge operations">
```typescript
// Bulk re-point children before soft-deleting parent
await em.nativeUpdate(Lead, 
  { person: oldPerson }, 
  { person: newPerson },
  { filters: false } // ✅ Hits archived AND soft-deleted children
);
```
</Accordion>

<Accordion title="Restore operations">
```typescript
// Restore soft-deleted entity
const deleted = await em.findOne(Person, { id }, { filters: false });
if (deleted) {
  deleted.isDeleted = false;
  await em.flush();
}
```
</Accordion>

<Accordion title="Conflict detection">
```typescript
// Check for email conflicts including soft-deleted
const existing = await em.findOne(User, 
  { email }, 
  { filters: false } // ✅ Conflict detection needs to see all records
);
```
</Accordion>

<Accordion title="Audit timeline">
```typescript
// Full history including deleted records
const history = await em.find(AuditLog, 
  { entityId }, 
  { filters: false }
);
```
</Accordion>

<Accordion title="Notification fan-out">
```typescript
// Send notifications to all stakeholders, even archived
const stakeholders = await em.find(DealStakeholder, 
  { deal: dealId }, 
  { filters: false }
);
```
</Accordion>

<Accordion title="Admin operations">
```typescript
// Admin panel showing all records
if (user.role === 'ADMIN') {
  return await em.find(Entity, {}, { filters: false });
}
```
</Accordion>

<Accordion title="Seed/migration scripts">
```typescript
// Data migration needs to see all records
const allRecords = await em.find(Entity, {}, { filters: false });
```
</Accordion>
</AccordionGroup>

<Warning>
Every `{ filters: false }` call site MUST have a code comment naming which category it belongs to.
</Warning>

## 3. User reference classification

### 3.1 Three-tier classification system

<Steps>
<Step title="Tier 1: Audit references (NEVER override filters)">
**Definition:** System-stamped audit columns that track who/when actions occurred.

**Examples:** `createdBy`, `updatedBy`, `archivedBy`, `restoredBy`, `assignedBy`, `performedBy`

**Behavior:** These references MUST survive user soft-deletion to preserve audit trails.

```typescript
@ManyToOne({ entity: () => User })
createdBy!: User; // ✅ NO filters override — audit trail preserved
```
</Step>

<Step title="Tier 2: Workflow/assignment references (conditional override)">
**Definition:** Business-critical assignments where soft-deletion should reassign, not hide records.

**Examples:** `assignedTo`, `owner`, `primaryAgent`, `accountManager`

**Behavior:** Override filters, but implement reassignment logic.

```typescript
@ManyToOne({ 
  entity: () => User,
  filters: { isDeleted: false } // ✅ Override to prevent hiding
})
assignedTo!: User;
```
</Step>

<Step title="Tier 3: Strong business entity references (override filters)">
**Definition:** References to core business entities that should never cause records to disappear.

**Examples:** `person`, `organization`, `company`, `contact`

**Behavior:** Always override filters.

```typescript
@ManyToOne({ 
  entity: () => Person,
  filters: { isDeleted: false } // ✅ Always override
})
person!: Person;
```
</Step>
</Steps>

### 3.2 DTO handling for soft-deleted references

When a Tier-3 owner relation is soft-deleted, DTOs have three strategies:

<Tabs>
<Tab title="Strategy A: Placeholder">
```typescript
// Replace with placeholder when soft-deleted
if (lead.person.isDeleted) {
  return {
    ...leadDto,
    person: {
      id: lead.person.id,
      name: '[Deleted Contact]',
      email: null
    }
  };
}
```
</Tab>

<Tab title="Strategy B: Omit">
```typescript
// Omit the relation entirely
if (lead.person.isDeleted) {
  const { person, ...leadWithoutPerson } = leadDto;
  return leadWithoutPerson;
}
```
</Tab>

<Tab title="Strategy C: Error Response">
```typescript
// Return 4xx error
if (lead.person.isDeleted) {
  throw new BadRequestException('Referenced contact no longer available');
}
```
</Tab>
</Tabs>

<Warning>
NEVER serialize the full Tier-3 payload when `isDeleted === true`.
</Warning>

## 4. Testing requirements

### 4.1 Mandatory tests for new entities

Every new entity with non-nullable owners requires these two tests:

<CodeGroup>

```typescript Test 1: Visibility after owner soft-deletion
describe('Entity visibility with soft-deleted owners', () => {
  it('should remain visible when owner is soft-deleted', async () => {
    // Create entity with owner
    const owner = await createOwner();
    const entity = await createEntity({ owner });
    
    // Soft-delete the owner
    owner.isDeleted = true;
    await em.flush();
    
    // Entity should still be findable
    const found = await em.findOne(Entity, { id: entity.id });
    expect(found).toBeDefined();
    expect(found.owner.isDeleted).toBe(true);
  });
});
```

```typescript Test 2: DTO handling of soft-deleted references
describe('DTO with soft-deleted references', () => {
  it('should handle soft-deleted owner appropriately', async () => {
    const owner = await createOwner();
    const entity = await createEntity({ owner });
    
    // Soft-delete owner
    owner.isDeleted = true;
    await em.flush();
    
    // DTO should use placeholder/omit/error strategy
    const dto = await entityService.getDto(entity.id);
    
    // Choose one based on your strategy:
    expect(dto.owner.name).toBe('[Deleted Contact]'); // Placeholder
    // OR expect(dto.owner).toBeUndefined(); // Omit
    // OR expect(dto).toThrow(); // Error
  });
});
```

</CodeGroup>

### 4.2 Data integrity audit service

<Note>
New entities with non-nullable owners must extend `DataIntegrityAuditService` for Tier 2/3 references. The daily cron picks it up automatically.
</Note>

<Steps>
<Step title="Add entity to audit table">
Update the audit table in `QUERY_OPTIMIZATION_PATTERNS.md` §5.5 with your new entity.
</Step>

<Step title="Extend audit service">
```typescript
// DataIntegrityAuditService.ts
async auditNewEntity(): Promise<IntegrityIssue[]> {
  const orphaned = await this.em.find(NewEntity, {
    owner: { isDeleted: true } // Find entities with soft-deleted owners
  });
  
  return orphaned.map(entity => ({
    type: 'ORPHANED_REFERENCE',
    entityType: 'NewEntity',
    entityId: entity.id,
    issue: `Owner ${entity.owner.id} is soft-deleted`
  }));
}
```
</Step>

<Step title="Do NOT add HTTP endpoint">
<Warning>
The audit service runs via daily cron only. Do NOT expose it as an HTTP endpoint.
</Warning>
</Step>
</Steps>
