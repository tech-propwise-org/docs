<Warning>
**Status:** Project-wide standard — MANDATORY for all entities, services, and DTOs. This doc is the authoritative reference for the soft-delete / archive filter pattern.
</Warning>

## TL;DR — what every backend dev needs to know

This project uses **MikroORM v6** with `loadStrategy: JOINED`, `autoJoinRefsForFilters: true`, and `filtersOnRelations: true`. Combined with our `isDeleted` and `active` entity filters, this produces a **silent data-loss bug** if entities are not declared correctly:

<Note>
**A row whose own `is_deleted = false` will silently disappear from every query if any of its non-nullable owner relations points at a soft-deleted target.**
</Note>

The fix is **structural** and lives on the entity declaration. Every new entity MUST follow it.

| What | Rule |
|------|------|
| Non-nullable `@ManyToOne` / `@OneToOne(owner)` to an entity that declares `@Filter({ name: 'isDeleted', ... })` | Add `{ filters: { isDeleted: false } }` to the relation options |
| `@Filter({ name: 'active', ... })` on a soft-deletable + archivable entity | `cond` MUST be `{ isArchived: false }` only — NEVER `{ isDeleted: false, isArchived: false }` |
| Bulk re-point inside a merge service (`em.nativeUpdate(...)` that re-points children of an entity that is about to be soft-deleted) | Pass `{ filters: false }` so the re-point hits archived AND soft-deleted children |
| `{ filters: false }` anywhere else | Allowed only in the explicit-business-intent categories in §2.3 (merge / restore / conflict-detection / audit-timeline / notification fan-out / admin / seed). Every call site MUST have a code comment naming the category. |
| User reference on a new entity | Classify per §3.1.1 — `createdBy`/`updatedBy`/`archivedBy`/etc. are Tier 1 (audit ref); `assignedTo`/`owner`/`recipient`/`user` are Tier 2 or 3 |
| DTO with a Tier-3 owner relation (`person`, `organization`) | If `populated.isDeleted === true` → DTO MUST use placeholder / omit / 4xx (Strategies A / B / C in §3.3.1). NEVER serialize the full Tier-3 payload when soft-deleted |
| New entity with non-nullable owners | Update the audit table in `QUERY_OPTIMIZATION_PATTERNS.md` §5.5 + extend `DataIntegrityAuditService` for Tier 2/3 (the daily cron picks it up — do NOT add an HTTP endpoint) + add the two mandatory tests from §4.1 |

<Warning>
If you skip any of these, the regression will not show up in unit tests — it surfaces as "this row exists in the DB but the API returns nothing" weeks later.
</Warning>

## 1. Cause — why rows silently disappear

MikroORM v6 (`@mikro-orm/core@6.6.7`) has three defaults that interact badly with soft-deletable graphs:

```ts
// mikro-orm.config.ts (defaults — NOT to be disabled globally)
loadStrategy: LoadStrategy.JOINED; // populated relations are joined into the main SELECT
autoJoinRefsForFilters: true; // auto-add :ref JOIN for any relation whose target has filters
filtersOnRelations: true; // apply target-entity filters via JOIN ON
```

For every M:1 / 1:1-owner relation on the queried entity, MikroORM checks whether the target entity declares any default-enabled filters (e.g. `@Filter({ name: 'isDeleted', default: true, cond: { isDeleted: false } })`). If so, it adds an auto `:ref` JOIN to that relation **even if you did not populate it**, so the filter can be applied as a JOIN `ON` condition.

For a **non-nullable** M:1 the resulting JOIN type is `INNER JOIN`:

```ts
const mandatoryToOneProperty =
  [ReferenceKind.MANY_TO_ONE, ReferenceKind.ONE_TO_ONE].includes(prop.kind) && !prop.nullable;
const joinType = (hint.filter && !prop.nullable) || mandatoryToOneProperty ? INNER_JOIN : LEFT_JOIN;
```

**The failure mode**: `em.find(Lead, { id: leadId })` becomes (roughly):

```sql
SELECT lead.*
  FROM lead
  INNER JOIN person p ON p.id = lead.person_id AND p.is_deleted = false
  INNER JOIN organization o ON o.id = lead.organization_id AND o.is_deleted = false
 WHERE lead.id = $1
   AND lead.is_deleted = false;
```

<Warning>
If **any** of those non-nullable joined targets fails the filter (e.g. `Person.isDeleted = true` because the lead's person was merged into another person), the **lead row is silently dropped** — even though `lead.is_deleted = false`.
</Warning>

For nullable relations the JOIN is `LEFT JOIN` and the relation hydrates to `null` instead of dropping the row.

### Real production examples this caused

| Scenario | Symptom |
|----------|---------|
| Two persons merged → secondary `Person.isDeleted = true` → an archived Lead still pointing at the secondary Person | The archived Lead was invisible everywhere even though it existed in the DB |
| `DealStage` cleanup → custom stage soft-deleted while a Deal still pointed at it | The Deal vanished from list endpoints |
| Agent leaves the org → `User.isDeleted = true` → every Deal / Contact / Company / Listing they `createdBy` would have vanished if we shipped the relation override blindly without the Tier-1 carve-out |

### 1.2 Special case — the `excludeAiServiceUser` filter is OPT-IN (`default: false`)

The same `autoJoinRefsForFilters` mechanic bit a **non-soft-delete** filter. The `User` entity declares a second filter that hides **all non-human service accounts** — both the per-org **AI agent service account** (`isAiServiceUser = true`) and the per-org **System automation actor** (`isSystemServiceUser = true`):

```typescript
@Filter({
  name: 'excludeAiServiceUser',
  default: false,
  cond: { isAiServiceUser: false, isSystemServiceUser: false },
})
```

This filter was originally `default: true` to hide the service accounts from every user listing. But each is a **real `user` row** that stamps non-nullable audit columns (`created_by`, `changed_by`, `performed_by`, …) on anything it touches.

<Note>
**Resolution — invert the default to opt-in (`default: false`):** The filter is **OFF by default**, so audit-relation auto-joins no longer drop service-stamped rows.
</Note>

<Tabs>
<Tab title="User-facing enumerations">
| Query shape | How to exclude the service accounts |
|-------------|-----------------------------------|
| Direct `User` root query (listings, pickers) | `{ filters: { excludeAiServiceUser: true } }` in the query options |
| `UserOrgRole` enumeration (member/recipient/agent pools) | `user: { isAiServiceUser: false }` in the `where` |
| Raw SQL over `user_org_roles` | `JOIN "user" u ON u.id = uor.user_id AND u.is_ai_service_user = false` |
| `QueryBuilder` user lists | `qb.andWhere({ isAiServiceUser: false, isSystemServiceUser: false })` |
| Auth lookups by email | `{ email, isAiServiceUser: false, isSystemServiceUser: false }` |
</Tab>
<Tab title="DTO exposure">
The service accounts can surface as actors (`createdBy` / `changedBy` / `assignedBy` / `performedBy`). Their synthetic `ai-user+<orgId>@propwise.com` / `system-user+<orgId>@propwise.com` emails are **not** masked in user DTOs — they are non-routable internal addresses and the display names ("AI USER" / "System Automation") already identify them.
</Tab>
</Tabs>

## 2. The fix — three coordinated parts

The fix has **three** parts. Skipping any one of them re-introduces the bug.

<Steps>
<Step title="Per-relation filter override (MANDATORY structural)">

Declare `{ filters: { isDeleted: false } }` on **every non-nullable** `@ManyToOne` / `@OneToOne(owner)` relation to a soft-deletable entity:

```typescript
@Entity()
export class Lead extends BaseEntity {
  @ManyToOne(
    () => Person,
    { nullable: false, filters: { isDeleted: false } } // ← MANDATORY
  )
  person!: Ref<Person>;

  @ManyToOne(
    () => Organization,
    { nullable: false, filters: { isDeleted: false } } // ← MANDATORY
  )
  organization!: Ref<Organization>;
}
```

<Warning>
**For User relations:** Only apply this override to Tier-2 and Tier-3 user relations. Tier-1 audit relations (`createdBy`, `updatedBy`, etc.) must NOT have the override to preserve audit history.
</Warning>

</Step>
<Step title="Active filter declaration">

For entities that are both soft-deletable AND archivable, the `@Filter({ name: 'active', ... })` condition MUST be `{ isArchived: false }` only — NEVER include `isDeleted`:

```typescript
@Filter({
  name: 'active',
  default: true,
  cond: { isArchived: false } // ← CORRECT
})
// NOT: { isDeleted: false, isArchived: false } ← WRONG
```

</Step>
<Step title="Explicit business intent for filters: false">

`{ filters: false }` is allowed only for explicit business-intent categories. Every call site MUST have a code comment naming the category:

```typescript
// Merge service — must hit archived AND soft-deleted children
em.nativeUpdate(Contact, { person: oldPersonId }, { person: newPersonId }, { filters: false });

// Restore service — must find soft-deleted target
const deletedPerson = await em.findOne(Person, { id }, { filters: false });

// Admin panel — show all data for compliance
const allLeads = await em.find(Lead, {}, { filters: false });
```

</Step>
</Steps>

## 3. User relation classification

All `@ManyToOne(User)` relations are classified into three tiers based on their business intent:

### 3.1.1 Tier classification

<AccordionGroup>
<Accordion title="Tier 1: Audit references (NEVER override filters)">
**Purpose:** Historical attribution — who performed an action at what time
**Examples:** `createdBy`, `updatedBy`, `deletedBy`, `archivedBy`, `performedBy`, `assignedBy`, `escalatedBy`
**Filter rule:** NO `{ filters: { isDeleted: false } }` override
**Reason:** Audit integrity — we must preserve who did what even if they've left the org
</Accordion>

<Accordion title="Tier 2: Active workflow references (ALWAYS override filters)">
**Purpose:** Current assignment/ownership for active workflows
**Examples:** `assignedTo`, `currentOwner`, `primaryAgent`, `listingAgent`
**Filter rule:** ALWAYS `{ filters: { isDeleted: false } }` override
**Reason:** Active workflows shouldn't reference departed users
</Accordion>

<Accordion title="Tier 3: Business entity references (ALWAYS override filters + special DTO handling)">
**Purpose:** The user IS the business entity being referenced
**Examples:** `user` on UserProfile, `recipient` on Message, `contact.person.user`
**Filter rule:** ALWAYS `{ filters: { isDeleted: false } }` override + placeholder DTO strategy
**Reason:** Business entities should not reference soft-deleted users, and DTOs need graceful degradation
</Accordion>
</AccordionGroup>

### 3.2 DTO handling for Tier-3 relations

When a Tier-3 relation points to a soft-deleted user, the DTO MUST handle it gracefully:

<Tabs>
<Tab title="Strategy A: Placeholder">
```typescript
// If populated.isDeleted === true, use placeholder
return {
  id: user.id,
  name: "[Deleted User]",
  email: null,
  isDeleted: true
};
```
</Tab>
<Tab title="Strategy B: Omit">
```typescript
// Simply omit the field from the DTO
const dto = { ...otherFields };
if (!user.isDeleted) {
  dto.user = mapUserToDto(user);
}
return dto;
```
</Tab>
<Tab title="Strategy C: 4xx response">
```typescript
// Return an error if the core entity reference is deleted
if (message.recipient?.isDeleted) {
  throw new BadRequestException('Message recipient no longer exists');
}
```
</Tab>
</Tabs>

## 4. Testing requirements

<Steps>
<Step title="Add mandatory regression tests">

Every new entity with non-nullable Tier-2/3 user relations MUST include these two tests:

```typescript
describe('Soft-delete filter regression tests', () => {
  it('should find entity when Tier-2 user relation points to soft-deleted user', async () => {
    const user = await createTestUser();
    const entity = await createTestEntity({ assignedTo: user });
    
    // Soft-delete the user
    await userService.softDelete(user.id);
    
    // Entity should still be found (relation overrides isDeleted filter)
    const found = await entityService.findById(entity.id);
    expect(found).toBeDefined();
    expect(found.assignedTo.isDeleted).toBe(true);
  });

  it('should handle soft-deleted Tier-3 user gracefully in DTO', async () => {
    const user = await createTestUser();
    const entity = await createTestEntity({ user });
    
    // Soft-delete the user
    await userService.softDelete(user.id);
    
    // DTO should use placeholder strategy
    const dto = await entityService.getEntityDto(entity.id);
    expect(dto.user.name).toBe('[Deleted User]');
    expect(dto.user.isDeleted).toBe(true);
  });
});
```

</Step>
<Step title="Update data integrity audit">

Add the new entity to the `DataIntegrityAuditService` for Tier-2/3 relations. The daily cron will pick it up automatically.

<Warning>
Do NOT add an HTTP endpoint for data integrity audits — these run as background jobs only.
</Warning>

</Step>
</Steps>

## 5. Examples and patterns

### 5.1 Correct entity declaration

```typescript
@Entity()
@Filter({ name: 'isDeleted', default: true, cond: { isDeleted: false } })
@Filter({ name: 'active', default: true, cond: { isArchived: false } }) // NOT isDeleted + isArchived
export class Deal extends BaseEntity {
  // Tier 1: Audit reference — NO filter override
  @ManyToOne(() => User, { nullable: false })
  createdBy!: Ref<User>;

  // Tier 2: Active workflow — ALWAYS filter override
  @ManyToOne(() => User, { 
    nullable: false, 
    filters: { isDeleted: false } 
  })
  assignedTo!: Ref<User>;

  // Tier 3: Business entity — filter override + DTO handling
  @ManyToOne(() => Person, { 
    nullable: false, 
    filters: { isDeleted: false } 
  })
  primaryContact!: Ref<Person>;

  // Other soft-deletable entities — ALWAYS filter override for non-nullable
  @ManyToOne(() => DealStage, { 
    nullable: false, 
    filters: { isDeleted: false } 
  })
  stage!: Ref<DealStage>;
}
```

### 5.2 Service layer patterns

<CodeGroup>
```typescript Merge service
export class PersonMergeService {
  async mergePerson(primaryId: string, secondaryId: string) {
    // Bulk re-point — must hit archived AND soft-deleted children
    await this.em.nativeUpdate(
      Contact, 
      { person: secondaryId }, 
      { person: primaryId }, 
      { filters: false } // Explicit business intent: merge service
    );
    
    // Soft-delete the secondary
    await this.personService.softDelete(secondaryId);
  }
}
```

```typescript Restore service  
export class PersonRestoreService {
  async restorePerson(id: string) {
    // Must find soft-deleted target
    const person = await this.em.findOne(
      Person, 
      { id }, 
      { filters: false } // Explicit business intent: restore service
    );
    
    if (!person) {
      throw new NotFoundException('Person not found');
    }
    
    person.isDeleted = false;
    person.deletedAt = null;
    await this.em.persistAndFlush(person);
  }
}
```

```typescript Admin service
export class AdminAuditService {
  async getAllDeals() {
    // Admin panel — show all data for compliance
    return this.em.find(
      Deal, 
      {}, 
      { filters: false } // Explicit business intent: admin panel
    );
  }
}
```
</CodeGroup>

### 5.3 Query patterns that need special handling

<Tabs>
<Tab title="User enumeration">

```typescript
// Direct User queries — explicitly exclude service accounts
const users = await em.find(User, {}, {
  filters: { excludeAiServiceUser: true }

// UserOrgRole-based queries — service accounts already absent

  organization: orgId,
  role: UserRole.AGENT,

});
```
</Tab>
<Tab title="QueryBuilder (filters not auto-applied)">
```typescript

qb.where({
  isDeleted: false,
  isAiServiceUser: false,
  isSystemServiceUser: false
});
```
</Tab>
<Tab title="Auth lookups">

```typescript
const user = await em.findOne(User, {
  email,
  isDeleted: false,
  isAiServiceUser: false,
  isSystemServiceUser: false

```
</Tab>
