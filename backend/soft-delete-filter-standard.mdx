<Warning>
**Status:** Project-wide standard — MANDATORY for all entities, services, and DTOs

This project uses **MikroORM v6** with `loadStrategy: JOINED`, `autoJoinRefsForFilters: true`, and `filtersOnRelations: true`. Combined with our `isDeleted` and `active` entity filters, this produces a **silent data-loss bug** if entities are not declared correctly.
</Warning>

## TL;DR — what every backend dev needs to know

<Note>
A row whose own `is_deleted = false` will silently disappear from every query if any of its non-nullable owner relations points at a soft-deleted target.
</Note>

The fix is **structural** and lives on the entity declaration. Every new entity MUST follow it.

| What | Rule |
|------|------|
| Non-nullable `@ManyToOne` / `@OneToOne(owner)` to an entity that declares `@Filter({ name: 'isDeleted', ... })` | Add `{ filters: { isDeleted: false } }` to the relation options |
| `@Filter({ name: 'active', ... })` on a soft-deletable + archivable entity | `cond` MUST be `{ isArchived: false }` only — NEVER `{ isDeleted: false, isArchived: false }` |
| Bulk re-point inside a merge service (`em.nativeUpdate(...)` that re-points children of an entity that is about to be soft-deleted) | Pass `{ filters: false }` so the re-point hits archived AND soft-deleted children |
| `{ filters: false }` anywhere else | Allowed only in the explicit-business-intent categories in [Filters False Usage](#filters-false-is-reserved-for-explicit-business-intent-mandatory-framing) (merge / restore / conflict-detection / audit-timeline / notification fan-out / admin / seed). Every call site MUST have a code comment naming the category. |
| User reference on a new entity | Classify per User Reference Classification — `createdBy`/`updatedBy`/`archivedBy`/etc. are Tier 1 (audit ref); `assignedTo`/`owner`/`recipient`/`user` are Tier 2 or 3 |
| DTO with a Tier-3 owner relation (`person`, `organization`) | If `populated.isDeleted === true` → DTO MUST use placeholder / omit / 4xx (Strategies A / B / C). NEVER serialize the full Tier-3 payload when soft-deleted |
| New entity with non-nullable owners | Update the audit table in `QUERY_OPTIMIZATION_PATTERNS.md` §5.5 + extend `DataIntegrityAuditService` for Tier 2/3 (the daily cron picks it up — do NOT add an HTTP endpoint) + add the two mandatory tests |

<Warning>
If you skip any of these, the regression will not show up in unit tests — it surfaces as "this row exists in the DB but the API returns nothing" weeks later.
</Warning>

## Cause — why rows silently disappear

MikroORM v6 (`@mikro-orm/core@6.6.7`) has three defaults that interact badly with soft-deletable graphs:

```typescript
// mikro-orm.config.ts (defaults — NOT to be disabled globally)
loadStrategy: LoadStrategy.JOINED; // populated relations are joined into the main SELECT
autoJoinRefsForFilters: true; // auto-add :ref JOIN for any relation whose target has filters
filtersOnRelations: true; // apply target-entity filters via JOIN ON
```

For every M:1 / 1:1-owner relation on the queried entity, MikroORM checks whether the target entity declares any default-enabled filters (e.g. `@Filter({ name: 'isDeleted', default: true, cond: { isDeleted: false } })`). If so, it adds an auto `:ref` JOIN to that relation **even if you did not populate it**, so the filter can be applied as a JOIN `ON` condition.

For a **non-nullable** M:1 the resulting JOIN type is `INNER JOIN`:

```typescript
const mandatoryToOneProperty =
  [ReferenceKind.MANY_TO_ONE, ReferenceKind.ONE_TO_ONE].includes(prop.kind) && !prop.nullable;
const joinType = (hint.filter && !prop.nullable) || mandatoryToOneProperty ? INNER_JOIN : LEFT_JOIN;
```

**The failure mode**: `em.find(Lead, { id: leadId })` becomes (roughly):

```sql
SELECT lead.*
  FROM lead
  INNER JOIN person p ON p.id = lead.person_id AND p.is_deleted = false
  INNER JOIN organization o ON o.id = lead.organization_id AND o.is_deleted = false
 WHERE lead.id = $1
   AND lead.is_deleted = false;
```

<Tip>
If **any** of those non-nullable joined targets fails the filter (e.g. `Person.isDeleted = true` because the lead's person was merged into another person), the **lead row is silently dropped** — even though `lead.is_deleted = false`.
</Tip>

For nullable relations the JOIN is `LEFT JOIN` and the relation hydrates to `null` instead of dropping the row.

### Real production examples this caused

| Scenario | Symptom |
|----------|---------|
| Two persons merged → secondary `Person.isDeleted = true` → an archived Lead still pointing at the secondary Person | The archived Lead was invisible everywhere even though it existed in the DB |
| `DealStage` cleanup → custom stage soft-deleted while a Deal still pointed at it | The Deal vanished from list endpoints |
| Agent leaves the org → `User.isDeleted = true` → every Deal / Contact / Company / Listing they `createdBy` would have vanished if we shipped the relation override blindly without the Tier-1 carve-out |

## The fix — three coordinated parts

The fix has **three** parts. Skipping any one of them re-introduces the bug.

<Steps>
<Step title="Per-relation filter override (MANDATORY structural)">

Declare `{ filters: { isDeleted: false } }` on **every non-nullable** `@ManyToOne` / `@OneToOne(owner)` whose target entity has an `isDeleted` filter:

```typescript
// ✅ CORRECT — surgical, structural, zero side effects on standalone queries
@ManyToOne(() => Person, { filters: { isDeleted: false } })
person!: Person;

@ManyToOne(() => Organization, { filters: { isDeleted: false } })
organization!: Organization;

@ManyToOne(() => User, { filters: { isDeleted: false } })
createdBy!: User;
```

This disables the target's `isDeleted` filter **only** for the auto-`:ref` JOIN MikroORM adds for this relation. Standalone `em.find(Person, ...)` / `em.findOne(Organization, ...)` queries still apply the target's `isDeleted` filter — the override is local to this relation site.

<Info>
**Side effect to be aware of:** when you explicitly populate this relation (`populate: ['person']`), the populate JOIN ALSO loses the target's `isDeleted` filter. A soft-deleted populated row will hydrate fully instead of being filtered out. This is acceptable but each Tier-2/Tier-3 relation MUST add a DTO warn-log.
</Info>

Nullable attribution relations may also need the override when product behavior keeps historical references after the target is soft-deleted and callers populate the relation.

</Step>

<Step title="Active filter cleanup (MANDATORY for archivable entities)">

Every archivable entity should have **two** entity-level filters:

```typescript
@Entity()
@Filter({ name: 'isDeleted', default: true, cond: { isDeleted: false } })
@Filter({ name: 'active', default: true, cond: { isArchived: false } })  // ← NOT { isDeleted: false, isArchived: false }
export class Lead { ... }
```

Both filters default to `true`, so all standard queries apply `isDeleted = false AND isArchived = false`. The `active` filter MUST omit `isDeleted` because the standalone `isDeleted` filter already handles soft-delete exclusion.

<Warning>
**INVARIANTS:**
- Never disable `isDeleted` while keeping `active` — that would return soft-deleted records.
- To include archived rows, use `{ filters: { active: false } }`.
- To include soft-deleted rows, use `{ filters: { isDeleted: false } }` (rare — typically only in repair scripts and merge code).
- To include both, use `{ filters: false }` (only valid in merge / cascade / RLS-bypass paths).
</Warning>

</Step>

<Step title="Filters false is reserved for explicit business intent">

`{ filters: false }` disables BOTH `isDeleted` and `active` filters at once. It is **never** a workaround for the `autoJoinRefsForFilters` symptom — the per-relation override is the structural fix for that. Use `{ filters: false }` only when the code path **explicitly intends** to read soft-deleted or archived rows.

<Note>
Every call site MUST have a code comment naming the category.
</Note>

#### When `{ filters: false }` IS legitimate

A request handler / service / job may use `{ filters: false }` if and only if it falls into one of these categories:

| Category | When to use | Canonical example |
|----------|-------------|-------------------|
| Merge / cascade operations | Before soft-deleting an entity, bulk re-point all its children to a new parent | `PersonMergeService` finding all leads / contacts / deals of a to-be-deleted Person |
| Restore operations | Bring back a soft-deleted entity and its associated data | Admin restore workflow |
| Conflict detection | Check for duplicates including soft-deleted records | Email uniqueness validation |
| Audit timeline | Historical reporting that needs complete data | Compliance reports |
| Notification fan-out | System notifications about archived/deleted entities | Webhook delivery |
| Admin / support tools | Support dashboard showing all data states | Customer service tools |
| Data seeding | Bootstrap / migration scripts | Database initialization |

</Step>
</Steps>

## User reference classification

### Tier classification for user relations

<Tabs>
<Tab title="Tier 1 (Audit refs)">
**Always use `{ filters: { isDeleted: false } }`** — NEVER let audit references drop rows.

- `createdBy`, `updatedBy`, `deletedBy`, `archivedBy`
- `approvedBy`, `rejectedBy`, `assignedBy`, `mergedBy`

These are attribution-only and should never cause business data to disappear when a user account is deactivated.
</Tab>

<Tab title="Tier 2 (Operational refs)">
**Use `{ filters: { isDeleted: false } }`** + add DTO validation.

- `assignedTo`, `owner`, `manager`
- `reviewer`, `delegate`

The entity should remain visible even if the assigned user is soft-deleted, but the DTO should handle the soft-deleted user appropriately.
</Tab>

<Tab title="Tier 3 (Business entities)">
**Use `{ filters: { isDeleted: false } }`** + add DTO validation + business logic for handling soft-deleted references.

- `person`, `organization`, `company`
- `contact`, `client`, `customer`

These represent core business relationships that should persist even when the related entity is soft-deleted.
</Tab>
</Tabs>

### DTO handling for soft-deleted populated relations

When a Tier-2 or Tier-3 relation is populated and `populated.isDeleted === true`, the DTO MUST use one of these strategies:

<Tabs>
<Tab title="Strategy A: Placeholder">
```typescript
// ✅ Show meaningful placeholder
userDto = populated.isDeleted 
  ? { id: user.id, name: '[Deactivated User]', isDeleted: true }
  : UserDto.fromEntity(populated);
```
</Tab>

<Tab title="Strategy B: Omit">
```typescript
// ✅ Omit the relation entirely
if (!populated.isDeleted) {
  dto.assignedUser = UserDto.fromEntity(populated);
}
```
</Tab>

<Tab title="Strategy C: Error response">
```typescript
// ✅ Return 4xx for critical missing references
if (populated.isDeleted) {
  throw new BadRequestException('Referenced user no longer available');
}
```
</Tab>
</Tabs>

<Warning>
**NEVER serialize the full Tier-3 payload when soft-deleted** — this can leak sensitive data about deleted entities.
</Warning>

## Testing requirements

Every entity with non-nullable owner relations MUST have these two tests:

<CodeGroup>
```typescript test-1.spec.ts
describe('Soft-delete filter protection', () => {
  it('should not drop rows when non-nullable relation target is soft-deleted', async () => {
    // Arrange: Create entity with non-nullable relations
    const lead = await createTestLead({ person: testPerson, organization: testOrg });
    
    // Act: Soft-delete one of the relation targets
    await em.nativeUpdate(Person, { id: testPerson.id }, { isDeleted: true });
    
    // Assert: Original entity is still findable
    const found = await em.findOne(Lead, { id: lead.id });
    expect(found).toBeDefined();
    expect(found!.id).toBe(lead.id);
  });
});
```

```typescript test-2.spec.ts
describe('Populated soft-deleted relations', () => {
  it('should populate soft-deleted relations and handle in DTO', async () => {
    // Arrange: Create entity and soft-delete relation target
    const lead = await createTestLead({ person: testPerson });
    await em.nativeUpdate(Person, { id: testPerson.id }, { isDeleted: true });
    
    // Act: Find with populate
    const found = await em.findOne(Lead, { id: lead.id }, { populate: ['person'] });
    
    // Assert: Relation is populated but marked as deleted
    expect(found!.person.isDeleted).toBe(true);
    
    // Assert: DTO handles soft-deleted relation appropriately
    const dto = LeadDto.fromEntity(found!);
    // Verify DTO follows Strategy A, B, or C from above
  });
});
```
</CodeGroup>
