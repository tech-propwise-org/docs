<Warning>
This is a **project-wide standard** — MANDATORY for all entities, services, and DTOs. Skipping any part of this standard will cause silent data loss that won't show up in unit tests.
</Warning>

## Overview

This project uses **MikroORM v6** with `loadStrategy: JOINED`, `autoJoinRefsForFilters: true`, and `filtersOnRelations: true`. Combined with our `isDeleted` and `active` entity filters, this produces a **silent data-loss bug** if entities are not declared correctly.

<Note>
A row whose own `is_deleted = false` will silently disappear from every query if any of its non-nullable owner relations points at a soft-deleted target.
</Note>

## Quick Reference

Every backend developer must follow these rules:

| What | Rule |
|------|------|
| Non-nullable `@ManyToOne` / `@OneToOne(owner)` to an entity with `@Filter({ name: 'isDeleted', ... })` | Add `{ filters: { isDeleted: false } }` to the relation options |
| `@Filter({ name: 'active', ... })` on soft-deletable + archivable entity | `cond` MUST be `{ isArchived: false }` only — NEVER `{ isDeleted: false, isArchived: false }` |
| Bulk re-point inside merge service | Pass `{ filters: false }` so the re-point hits archived AND soft-deleted children |
| `{ filters: false }` anywhere else | Allowed only in explicit-business-intent categories (see below). Every call site MUST have a code comment naming the category |
| User reference on new entity | Classify per user reference tiers — audit refs vs business refs |
| DTO with Tier-3 owner relation | If `populated.isDeleted === true` → use placeholder/omit/4xx. NEVER serialize full payload when soft-deleted |
| New entity with non-nullable owners | Update audit table + extend `DataIntegrityAuditService` + add mandatory tests |

## Root Cause

MikroORM v6 has three defaults that interact badly with soft-deletable graphs:

```typescript
// mikro-orm.config.ts (defaults — NOT to be disabled globally)
loadStrategy: LoadStrategy.JOINED; // populated relations are joined into the main SELECT
autoJoinRefsForFilters: true; // auto-add :ref JOIN for any relation whose target has filters
filtersOnRelations: true; // apply target-entity filters via JOIN ON
```

### The Failure Mode

For every M:1 / 1:1-owner relation on the queried entity, MikroORM checks whether the target entity declares any default-enabled filters. If so, it adds an auto `:ref` JOIN to that relation **even if you did not populate it**.

For a **non-nullable** M:1 the resulting JOIN type is `INNER JOIN`:

```sql
SELECT lead.*
  FROM lead
  INNER JOIN person p ON p.id = lead.person_id AND p.is_deleted = false
  INNER JOIN organization o ON o.id = lead.organization_id AND o.is_deleted = false
 WHERE lead.id = $1
   AND lead.is_deleted = false;
```

<Warning>
If **any** of those non-nullable joined targets fails the filter (e.g. `Person.isDeleted = true`), the **lead row is silently dropped** — even though `lead.is_deleted = false`.
</Warning>

### Real Production Examples

| Scenario | Symptom |
|----------|---------|
| Two persons merged → secondary `Person.isDeleted = true` → archived Lead still pointing at secondary Person | The archived Lead was invisible everywhere even though it existed in the DB |
| `DealStage` cleanup → custom stage soft-deleted while Deal still pointed at it | The Deal vanished from list endpoints |
| Agent leaves org → `User.isDeleted = true` → every Deal/Contact they `createdBy` would vanish | Silent data loss prevented by Tier-1 carve-out |

## The Solution

The fix has **three coordinated parts**. Skipping any one re-introduces the bug.

### 1. Per-Relation Filter Override (MANDATORY)

Declare `{ filters: { isDeleted: false } }` on **every non-nullable** `@ManyToOne` / `@OneToOne(owner)` whose target entity has an `isDeleted` filter:

<CodeGroup>
```typescript Correct Implementation
// ✅ CORRECT — surgical, structural, zero side effects on standalone queries
@ManyToOne(() => Person, { filters: { isDeleted: false } })
person!: Person;

@ManyToOne(() => Organization, { filters: { isDeleted: false } })
organization!: Organization;

@ManyToOne(() => User, { filters: { isDeleted: false } })
createdBy!: User;
```

```typescript Incorrect Implementation
// ❌ WRONG — will cause silent data loss
@ManyToOne(() => Person)
person!: Person;

@ManyToOne(() => Organization)
organization!: Organization;
```
</CodeGroup>

<Info>
This disables the target's `isDeleted` filter **only** for the auto-`:ref` JOIN MikroORM adds for this relation. Standalone queries still apply the target's `isDeleted` filter.
</Info>

**Side Effect to Be Aware Of:** When you explicitly populate this relation (`populate: ['person']`), the populate JOIN ALSO loses the target's `isDeleted` filter. A soft-deleted populated row will hydrate fully instead of being filtered out.

### 2. Active Filter Cleanup (MANDATORY)

Every archivable entity should have **two** entity-level filters:

```typescript
@Entity()
@Filter({ name: 'isDeleted', default: true, cond: { isDeleted: false } })
@Filter({ name: 'active', default: true, cond: { isArchived: false } })  // ← NOT { isDeleted: false, isArchived: false }
export class Lead { ... }
```

<Warning>
The `active` filter MUST omit `isDeleted` because the standalone `isDeleted` filter already handles soft-delete exclusion.
</Warning>

#### Filter Usage Invariants

<AccordionGroup>
<Accordion title="Filter Combination Rules">
- Never disable `isDeleted` while keeping `active` — that would return soft-deleted records
- To include archived rows, use `{ filters: { active: false } }`
- To include soft-deleted rows, use `{ filters: { isDeleted: false } }` (rare — typically only in repair scripts)
- To include both, use `{ filters: false }` (only valid in specific business scenarios)
</Accordion>
</AccordionGroup>

### 3. Controlled Use of `filters: false` (MANDATORY)

`{ filters: false }` disables BOTH `isDeleted` and `active` filters at once. It is **never** a workaround for the `autoJoinRefsForFilters` symptom — the per-relation override is the structural fix.

#### Legitimate Use Cases

Use `{ filters: false }` only when the code path **explicitly intends** to read soft-deleted or archived rows. **Every call site MUST have a code comment naming the category.**

| Category | When to Use | Example |
|----------|-------------|---------|
| **Merge/Cascade Operations** | Re-pointing children before soft-deleting parent | `// Category: merge - re-point children before soft-delete` |
| **Restore Operations** | Checking what can be restored | `// Category: restore - validating restore prerequisites` |
| **Conflict Detection** | Checking for name conflicts including archived | `// Category: conflict - include archived in uniqueness check` |
| **Audit Timeline** | Building complete history view | `// Category: audit - complete timeline including deleted` |
| **Notification Fan-out** | Sending notifications about archived entities | `// Category: notification - include archived in recipient list` |
| **Admin Operations** | Administrative data management | `// Category: admin - data cleanup and management` |
| **Seed/Migration** | Data seeding and schema migrations | `// Category: seed - initial data population` |

## User Reference Classification

### Tier Classification System

<Tabs>
<Tab title="Tier 1: Audit References">
**Never filtered out** - Critical for data integrity and audit trails.

Examples: `createdBy`, `updatedBy`, `archivedBy`, `deletedBy`

```typescript
// Tier 1 - No DTO warnings needed
@ManyToOne(() => User, { filters: { isDeleted: false } })
createdBy!: User;
```
</Tab>

<Tab title="Tier 2: Business References">
**Filtered with warnings** - Important for business logic but can be soft-deleted.

Examples: `assignedTo`, `owner`, `primaryContact`

```typescript
// Tier 2 - DTO must warn if soft-deleted
@ManyToOne(() => User, { filters: { isDeleted: false } })
assignedTo!: User;
```
</Tab>

<Tab title="Tier 3: External References">
**Strict filtering required** - External entities that require careful handling.

Examples: `person`, `organization`, `company`

```typescript
// Tier 3 - DTO must use placeholder/omit/error if soft-deleted
@ManyToOne(() => Person, { filters: { isDeleted: false } })
person!: Person;
```
</Tab>
</Tabs>

### DTO Handling for Soft-Deleted References

For Tier-3 owner relations, if `populated.isDeleted === true`, the DTO MUST use one of these strategies:

<Steps>
<Step title="Strategy A: Placeholder">
Replace with generic placeholder data
</Step>

<Step title="Strategy B: Omit">
Remove the field from the response
</Step>

<Step title="Strategy C: 4xx Error">
Return an error indicating data corruption
</Step>
</Steps>

<Warning>
NEVER serialize the full Tier-3 payload when the target is soft-deleted.
</Warning>

## Testing Requirements

Every new entity with non-nullable owners must include these two mandatory tests:

<CodeGroup>
```typescript Test 1: Relation Override Effectiveness
describe('Soft-delete filter bypass', () => {
  it('should find entity even when non-nullable owner is soft-deleted', async () => {
    // Category: test - validating filter override behavior
    const softDeletedOwner = await em.findOne(Owner, ownerId, { filters: false });
    await em.nativeUpdate(Owner, ownerId, { isDeleted: true });
    
    const entity = await em.findOne(Entity, entityId);
    expect(entity).toBeDefined(); // Should not be silently dropped
  });
});
```

```typescript Test 2: DTO Soft-Delete Handling
describe('DTO soft-delete handling', () => {
  it('should handle soft-deleted Tier-3 reference correctly', async () => {
    await em.nativeUpdate(Person, personId, { isDeleted: true });
    
    const entity = await em.findOne(Entity, entityId, { 
      populate: ['person'] 
    });
    const dto = EntityDto.fromEntity(entity);
    
    // Verify placeholder/omit/error strategy is applied
    expect(dto.person).toBeNull(); // or appropriate placeholder
  });
});
```
</CodeGroup>

<Note>
These regression tests are critical because the soft-delete filter bug does not show up in standard unit tests — it only surfaces in production as "this row exists in the DB but the API returns nothing."
</Note>
