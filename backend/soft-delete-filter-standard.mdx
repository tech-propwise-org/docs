This document is the authoritative reference for the soft-delete / archive filter pattern. See `QUERY_OPTIMIZATION_PATTERNS.md` ¬ß5.5 for MikroORM internals.

## TL;DR ‚Äî what every backend dev needs to know

This project uses **MikroORM v6** with `loadStrategy: JOINED`, `autoJoinRefsForFilters: true`, and `filtersOnRelations: true`. Combined with our `isDeleted` and `active` entity filters, this produces a **silent data-loss bug** if entities are not declared correctly:

<Note>
**Critical Issue:** A row whose own `is_deleted = false` will silently disappear from every query if any of its non-nullable owner relations points at a soft-deleted target.
</Note>

The fix is **structural** and lives on the entity declaration. Every new entity MUST follow it.

| What | Rule |
|------|------|
| Non-nullable `@ManyToOne` / `@OneToOne(owner)` to an entity that declares `@Filter({ name: 'isDeleted', ... })` | Add `{ filters: { isDeleted: false } }` to the relation options |
| `@Filter({ name: 'active', ... })` on a soft-deletable + archivable entity | `cond` MUST be `{ isArchived: false }` only ‚Äî NEVER `{ isDeleted: false, isArchived: false }` |
| Bulk re-point inside a merge service (`em.nativeUpdate(...)` that re-points children of an entity that is about to be soft-deleted) | Pass `{ filters: false }` so the re-point hits archived AND soft-deleted children |
| `{ filters: false }` anywhere else | Allowed only in the explicit-business-intent categories in ¬ß2.3 (merge / restore / conflict-detection / audit-timeline / notification fan-out / admin / seed). Every call site MUST have a code comment naming the category. |
| User reference on a new entity | Classify per ¬ß3.1.1 ‚Äî `createdBy`/`updatedBy`/`archivedBy`/etc. are Tier 1 (audit ref); `assignedTo`/`owner`/`recipient`/`user` are Tier 2 or 3 |
| DTO with a Tier-3 owner relation (`person`, `organization`) | If `populated.isDeleted === true` ‚Üí DTO MUST use placeholder / omit / 4xx (Strategies A / B / C in ¬ß3.3.1). NEVER serialize the full Tier-3 payload when soft-deleted |
| New entity with non-nullable owners | Update the audit table in `QUERY_OPTIMIZATION_PATTERNS.md` ¬ß5.5 + extend `DataIntegrityAuditService` for Tier 2/3 (the daily cron picks it up ‚Äî do NOT add an HTTP endpoint) + add the two mandatory tests from ¬ß4.1 |

<Warning>
If you skip any of these, the regression will not show up in unit tests ‚Äî it surfaces as "this row exists in the DB but the API returns nothing" weeks later.
</Warning>

## Cause ‚Äî why rows silently disappear

MikroORM v6 (`@mikro-orm/core@6.6.7`) has three defaults that interact badly with soft-deletable graphs:

```ts
// mikro-orm.config.ts (defaults ‚Äî NOT to be disabled globally)
loadStrategy: LoadStrategy.JOINED; // populated relations are joined into the main SELECT
autoJoinRefsForFilters: true; // auto-add :ref JOIN for any relation whose target has filters
filtersOnRelations: true; // apply target-entity filters via JOIN ON
```

For every M:1 / 1:1-owner relation on the queried entity, MikroORM checks whether the target entity declares any default-enabled filters (e.g. `@Filter({ name: 'isDeleted', default: true, cond: { isDeleted: false } })`). If so, it adds an auto `:ref` JOIN to that relation **even if you did not populate it**, so the filter can be applied as a JOIN `ON` condition.

For a **non-nullable** M:1 the resulting JOIN type is `INNER JOIN` (see `AbstractSqlDriver.getFieldsForJoinedLoad`):

```ts
const mandatoryToOneProperty =
  [ReferenceKind.MANY_TO_ONE, ReferenceKind.ONE_TO_ONE].includes(prop.kind) && !prop.nullable;
const joinType = (hint.filter && !prop.nullable) || mandatoryToOneProperty ? INNER_JOIN : LEFT_JOIN;
```

**The failure mode**: `em.find(Lead, { id: leadId })` becomes (roughly):

```sql
SELECT lead.*
  FROM lead
  INNER JOIN person p ON p.id = lead.person_id AND p.is_deleted = false
  INNER JOIN organization o ON o.id = lead.organization_id AND o.is_deleted = false
 WHERE lead.id = $1
   AND lead.is_deleted = false;
```

<Info>
If **any** of those non-nullable joined targets fails the filter (e.g. `Person.isDeleted = true` because the lead's person was merged into another person), the **lead row is silently dropped** ‚Äî even though `lead.is_deleted = false`.
</Info>

For nullable relations the JOIN is `LEFT JOIN` and the relation hydrates to `null` instead of dropping the row (see MikroORM 6.6 release notes ‚Äî the v6.5 "Strict relation filters" regression was reverted in 6.6).

### Real production examples this caused

| Scenario | Symptom |
|----------|---------|
| Two persons merged ‚Üí secondary `Person.isDeleted = true` ‚Üí an archived Lead still pointing at the secondary Person | The archived Lead was invisible everywhere even though it existed in the DB |
| `DealStage` cleanup ‚Üí custom stage soft-deleted while a Deal still pointed at it | The Deal vanished from list endpoints |
| Agent leaves the org ‚Üí `User.isDeleted = true` ‚Üí every Deal / Contact / Company / Listing they `createdBy` would have vanished if we shipped the relation override blindly without the Tier-1 carve-out |

### Special case ‚Äî the `excludeAiServiceUser` filter is OPT-IN (`default: false`)

The same `autoJoinRefsForFilters` mechanic bit a **non-soft-delete** filter. The `User` entity declares a second filter that hides **all non-human service accounts** ‚Äî both the per-org **AI agent service account** (`isAiServiceUser = true`, the row ORGANIZATION-scope AI agents act as) and the per-org **System automation actor** (`isSystemServiceUser = true`, the row distribution / escalation / lead-capture stamp as `createdBy`/`assignedBy`/`performedBy`):

```typescript
@Filter({
  name: 'excludeAiServiceUser',
  default: false,
  cond: { isAiServiceUser: false, isSystemServiceUser: false },
})
```

This filter was originally `default: true` to hide the service accounts from every user listing. But each is a **real `user` row** that stamps non-nullable audit columns (`created_by`, `changed_by`, `performed_by`, ‚Ä¶) on anything it touches. With `default: true`, MikroORM auto-joined `... INNER JOIN "user" u ON ... AND u.is_ai_service_user = false` (and now `... AND u.is_system_service_user = false`) for those non-nullable `@ManyToOne(User)` relations and **silently dropped** every row a service account created (e.g. an AI-created `Contact`, or a system-distributed lead's stakeholder, returning 404 immediately after creation).

**Resolution ‚Äî invert the default to opt-in (`default: false`):**

<Steps>
<Step title="Filter is OFF by default">
The filter is **OFF by default**, so audit-relation auto-joins no longer drop service-stamped rows. This is the inverse of the `isDeleted` strategy: for `isDeleted` we keep the global filter ON and override per-relation; for `excludeAiServiceUser` we keep it OFF globally and re-enable per-query where the service accounts must be hidden.
</Step>

<Step title="User-facing enumerations must re-exclude the service accounts explicitly">
| Query shape | How to exclude the service accounts |
|-------------|-----------------------------------|
| Direct `User` root query (listings, pickers) | `{ filters: { excludeAiServiceUser: true } }` in the query options (now hides both accounts) |
| `UserOrgRole` enumeration (member/recipient/agent pools) | `user: { isAiServiceUser: false }` in the `where` (the System actor has **no** `UserOrgRole`, so it is already absent) |
| Raw SQL over `user_org_roles` with no `team_membership` join (e.g. `getOrgAgentIds`) | `JOIN "user" u ON u.id = uor.user_id AND u.is_ai_service_user = false` (System actor absent ‚Äî no `UserOrgRole`) |
| `QueryBuilder` user lists (filters are NOT auto-applied) | `qb.andWhere({ isAiServiceUser: false, isSystemServiceUser: false })` |
| Auth lookups by email (signup / login / verify / reset) | `{ email, isAiServiceUser: false, isSystemServiceUser: false }` |
</Step>

<Step title="DTO exposure">
The service accounts can surface as actors (`createdBy` / `changedBy` / `assignedBy` / `performedBy`). Their synthetic `ai-user+<orgId>@propwise.com` / `system-user+<orgId>@propwise.com` emails are **not** masked in user DTOs ‚Äî they are non-routable internal addresses and the display names ("AI USER" / "System Automation") already identify them.
</Step>
</Steps>

## The fix ‚Äî three coordinated parts

The fix has **three** parts. Skipping any one of them re-introduces the bug.

### Per-relation `{ filters: { isDeleted: false } }` override (MANDATORY structural)

Declare `{ filters: { isDeleted: false } }` on **every non-nullable** `@ManyToOne` / `@OneToOne(owner)` relation to an entity that declares `@Filter({ name: 'isDeleted', ... })`.

<Warning>
This override is **structural** ‚Äî it goes in the entity declaration, not in query code.
</Warning>

<CodeGroup>

```typescript Good ‚Äî Lead entity
@Entity()
export class Lead extends BaseEntity {
  @ManyToOne(() => Person, { nullable: false, filters: { isDeleted: false } })
  person!: Person;

  @ManyToOne(() => Organization, { nullable: false, filters: { isDeleted: false } })
  organization!: Organization;

  @ManyToOne(() => User, { nullable: false, filters: { isDeleted: false } })
  createdBy!: User;

  @ManyToOne(() => DealStage, { nullable: false, filters: { isDeleted: false } })
  stage!: DealStage;
}
```

```typescript Bad ‚Äî Missing overrides
@Entity()
export class Lead extends BaseEntity {
  @ManyToOne(() => Person, { nullable: false })
  person!: Person; // ‚ùå Will drop lead rows if person.isDeleted = true

  @ManyToOne(() => Organization, { nullable: false })
  organization!: Organization; // ‚ùå Will drop lead rows if org.isDeleted = true

  @ManyToOne(() => User, { nullable: false })
  createdBy!: User; // ‚ùå Will drop lead rows if user.isDeleted = true
}
```

</CodeGroup>

### User relation classification system (Tier 1/2/3)

Not all `User` relations get the same override. Classify each `@ManyToOne(User)` relation:

<Tabs>
  <Tab title="Tier 1 ‚Äî Audit Relations">
    **Definition:** Actor attribution for who performed an action
    
    **Examples:** `createdBy`, `updatedBy`, `archivedBy`, `assignedBy`, `performedBy`, `reviewedBy`, `approvedBy`
    
    **Override:** `{ filters: { isDeleted: false } }`
    
    **Rationale:** Audit trails must be stable. If a user is soft-deleted, we still need to see the actions they took.
  </Tab>
  
  <Tab title="Tier 2 ‚Äî Business Relations">
    **Definition:** Current business assignment/ownership that can be reassigned
    
    **Examples:** `assignedTo`, `owner`, `primaryAgent`, `teamLead`
    
    **Override:** `{ filters: { isDeleted: false } }`
    
    **Note:** When the user is soft-deleted, trigger reassignment in the pre-delete hook
  </Tab>
  
  <Tab title="Tier 3 ‚Äî Entity References">
    **Definition:** The user IS the entity being referenced
    
    **Examples:** `user` on Contact, `recipient` on Notification
    
    **Override:** **NO override** ‚Äî let the row drop if the user is soft-deleted
  </Tab>
</Tabs>

### Service-layer `{ filters: false }` (explicit business intent only)

`{ filters: false }` disables ALL entity filters for that query. It's only allowed for specific business purposes:

<AccordionGroup>
  <Accordion title="Allowed Categories">
    - **Merge operations** ‚Äî finding all children to re-point before soft-delete
    - **Restore operations** ‚Äî finding soft-deleted entities to restore
    - **Conflict detection** ‚Äî checking for soft-deleted duplicates before create
    - **Audit timeline** ‚Äî historical views including deleted items
    - **Notification fan-out** ‚Äî ensuring messages reach archived entities
    - **Admin operations** ‚Äî superuser views of all data
    - **Seed/migration** ‚Äî data initialization ignoring current state
  </Accordion>
  
  <Accordion title="Required Documentation">
    Every `{ filters: false }` call site MUST have a code comment naming the category:
    
    ```typescript
    // Merge operation: find all children to re-point before soft-delete
    const children = await em.find(Lead, { person: personId }, { filters: false });
    
    // Admin operation: superuser view of all organizations
    const allOrgs = await em.find(Organization, {}, { filters: false });
    ```
  </Accordion>
</AccordionGroup>

## Entity implementation patterns

### Soft-deletable entities

<CodeGroup>

```typescript Standard Pattern
@Entity()
@Filter({
  name: 'isDeleted',
  default: true,
  cond: { isDeleted: false },
})
export class Person extends BaseEntity {
  @Property({ default: false })
  isDeleted = false;

  @Property({ nullable: true })
  deletedAt?: Date;

  // Non-nullable relations MUST have the override
  @ManyToOne(() => Organization, { 
    nullable: false, 
    filters: { isDeleted: false } 
  })
  organization!: Organization;

  @ManyToOne(() => User, { 
    nullable: false, 
    filters: { isDeleted: false } 
  })
  createdBy!: User;
}
```

```typescript Archivable + Soft-deletable
@Entity()
@Filter({
  name: 'isDeleted',
  default: true,
  cond: { isDeleted: false },
})
@Filter({
  name: 'active',
  default: true,
  cond: { isArchived: false }, // ‚ùå NEVER include isDeleted here
})
export class Deal extends BaseEntity {
  @Property({ default: false })
  isDeleted = false;

  @Property({ default: false })
  isArchived = false;

  @Property({ nullable: true })
  deletedAt?: Date;

  @Property({ nullable: true })
  archivedAt?: Date;
}
```

</CodeGroup>

### DTO handling for soft-deleted relations

When a DTO includes a relation that might be soft-deleted, handle it explicitly:

<Tabs>
  <Tab title="Strategy A ‚Äî Placeholder">
    ```typescript
    // In the DTO transform
    if (entity.person.isDeleted) {
      dto.person = {
        id: entity.person.id,
        name: '[Deleted Contact]',
        email: null,
        isDeleted: true,
      };
    } else {
      dto.person = this.personToDto(entity.person);
    }
    ```
  </Tab>
  
  <Tab title="Strategy B ‚Äî Omit">
    ```typescript
    // In the DTO transform
    if (!entity.person.isDeleted) {
      dto.person = this.personToDto(entity.person);
    }
    // dto.person remains undefined if deleted
    ```
  </Tab>
  
  <Tab title="Strategy C ‚Äî Error">
    ```typescript
    // In the service layer
    if (entity.person.isDeleted) {
      throw new BadRequestException(
        'Cannot access lead - associated contact has been deleted'
      );
    }
    ```
  </Tab>
</Tabs>

## Testing requirements

Every new entity with non-nullable owner relations MUST include these tests:

<Steps>
<Step title="Soft-delete transparency test">
```typescript
describe('Lead entity soft-delete transparency', () => {
  it('should remain findable when owner relations are soft-deleted', async () => {
    // Create lead with person and organization
    const lead = await createLead();
    
    // Soft-delete the person
    await em.nativeUpdate(Person, { id: lead.person.id }, { 
      isDeleted: true,
      deletedAt: new Date() 
    });
    
    // Lead should still be findable
    const foundLead = await em.findOne(Lead, { id: lead.id });
    expect(foundLead).toBeDefined();
    
    // Person relation should be populated as soft-deleted
    expect(foundLead.person.isDeleted).toBe(true);
  });
});
```
</Step>

<Step title="DTO strategy test">
```typescript
it('should handle soft-deleted relations in DTOs correctly', async () => {
  const lead = await createLead();
  
  // Soft-delete the person
  await em.nativeUpdate(Person, { id: lead.person.id }, { 
    isDeleted: true,
    deletedAt: new Date() 
  });
  
  const dto = await leadService.getLeadDto(lead.id);
  
  // Verify the chosen strategy (A, B, or C)
  if (dto.person) {
    expect(dto.person.name).toBe('[Deleted Contact]'); // Strategy A
  } else {
    expect(dto.person).toBeUndefined(); // Strategy B
  }
});
```
</Step>

## Common pitfalls to avoid

<Warning>
**Never combine filters in the `active` condition**

```typescript
// ‚ùå WRONG - will cause double-filtering
@Filter({
  name: 'active',
  default: true,
  cond: { isDeleted: false, isArchived: false }, // Don't do this
})

// ‚úÖ CORRECT - only filter on archive status
@Filter({
  name: 'active',
  default: true,
  cond: { isArchived: false }, // isDeleted filter is separate
})
```
</Warning>

<Info>
**Remember nullable vs non-nullable relations**

- **Nullable relations**: No override needed, they become `LEFT JOIN` and hydrate to `null`
- **Non-nullable relations**: Always need `{ filters: { isDeleted: false } }` override
</Info>

<Tip>
**Use the audit service for monitoring**

The `DataIntegrityAuditService` runs daily and reports entities that would be affected by soft-delete cascades. Extend it for new entities with Tier 2/3 relations.
</Tip>
