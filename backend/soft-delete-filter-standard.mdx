## TL;DR — what every backend dev needs to know

This project uses **MikroORM v6** with `loadStrategy: JOINED`, `autoJoinRefsForFilters: true`, and `filtersOnRelations: true`. Combined with our `isDeleted` and `active` entity filters, this produces a **silent data-loss bug** if entities are not declared correctly:

<Warning>
A row whose own `is_deleted = false` will silently disappear from every query if any of its non-nullable owner relations points at a soft-deleted target.
</Warning>

The fix is **structural** and lives on the entity declaration. Every new entity MUST follow it.

| What | Rule |
| --- | --- |
| Non-nullable `@ManyToOne` / `@OneToOne(owner)` to an entity that declares `@Filter({ name: 'isDeleted', ... })` | Add `{ filters: { isDeleted: false } }` to the relation options |
| `@Filter({ name: 'active', ... })` on a soft-deletable + archivable entity | `cond` MUST be `{ isArchived: false }` only — NEVER `{ isDeleted: false, isArchived: false }` |
| Bulk re-point inside a merge service (`em.nativeUpdate(...)` that re-points children of an entity that is about to be soft-deleted) | Pass `{ filters: false }` so the re-point hits archived AND soft-deleted children |
| `{ filters: false }` anywhere else | Allowed only in the explicit-business-intent categories in §2.3 (merge / restore / conflict-detection / audit-timeline / notification fan-out / admin / seed). Every call site MUST have a code comment naming the category. |
| User reference on a new entity | Classify per §3.1.1 — `createdBy`/`updatedBy`/`archivedBy`/etc. are Tier 1 (audit ref); `assignedTo`/`owner`/`recipient`/`user` are Tier 2 or 3 |
| DTO with a Tier-3 owner relation (`person`, `organization`) | If `populated.isDeleted === true` → DTO MUST use placeholder / omit / 4xx (Strategies A / B / C in §3.3.1). NEVER serialize the full Tier-3 payload when soft-deleted |
| New entity with non-nullable owners | Update the audit table in `QUERY_OPTIMIZATION_PATTERNS.md` §5.5 + extend `DataIntegrityAuditService` for Tier 2/3 (the daily cron picks it up — do NOT add an HTTP endpoint) + add the two mandatory tests from §4.1 |

<Warning>
If you skip any of these, the regression will not show up in unit tests — it surfaces as "this row exists in the DB but the API returns nothing" weeks later.
</Warning>

## 1. Cause — why rows silently disappear

MikroORM v6 (`@mikro-orm/core@6.6.7`) has three defaults that interact badly with soft-deletable graphs:

```typescript
// mikro-orm.config.ts (defaults — NOT to be disabled globally)
loadStrategy: LoadStrategy.JOINED; // populated relations are joined into the main SELECT
autoJoinRefsForFilters: true; // auto-add :ref JOIN for any relation whose target has filters
filtersOnRelations: true; // apply target-entity filters via JOIN ON
```

For every M:1 / 1:1-owner relation on the queried entity, MikroORM checks whether the target entity declares any default-enabled filters (e.g. `@Filter({ name: 'isDeleted', default: true, cond: { isDeleted: false } })`). If so, it adds an auto `:ref` JOIN to that relation **even if you did not populate it**, so the filter can be applied as a JOIN `ON` condition.

For a **non-nullable** M:1 the resulting JOIN type is `INNER JOIN`:

```typescript
const mandatoryToOneProperty =
  [ReferenceKind.MANY_TO_ONE, ReferenceKind.ONE_TO_ONE].includes(prop.kind) && !prop.nullable;
const joinType = (hint.filter && !prop.nullable) || mandatoryToOneProperty ? INNER_JOIN : LEFT_JOIN;
```

**The failure mode**: `em.find(Lead, { id: leadId })` becomes (roughly):

```sql
SELECT lead.*
  FROM lead
  INNER JOIN person p ON p.id = lead.person_id AND p.is_deleted = false
  INNER JOIN organization o ON o.id = lead.organization_id AND o.is_deleted = false
 WHERE lead.id = $1
   AND lead.is_deleted = false;
```

If **any** of those non-nullable joined targets fails the filter (e.g. `Person.isDeleted = true` because the lead's person was merged into another person), the **lead row is silently dropped** — even though `lead.is_deleted = false`.

For nullable relations the JOIN is `LEFT JOIN` and the relation hydrates to `null` instead of dropping the row.

### Real production examples this caused

| Scenario | Symptom |
| --- | --- |
| Two persons merged → secondary `Person.isDeleted = true` → an archived Lead still pointing at the secondary Person | The archived Lead was invisible everywhere even though it existed in the DB |
| `DealStage` cleanup → custom stage soft-deleted while a Deal still pointed at it | The Deal vanished from list endpoints |
| Agent leaves the org → `User.isDeleted = true` → every Deal / Contact / Company / Listing they `createdBy` would have vanished if we shipped the relation override blindly without the Tier-1 carve-out | All records created by that user would disappear |

## 2. The fix — three coordinated parts

The fix has **three** parts. Skipping any one of them re-introduces the bug.

### 2.1 Per-relation `{ filters: { isDeleted: false } }` override (MANDATORY structural)

Declare `{ filters: { isDeleted: false } }` on **every non-nullable** `@ManyToOne` / `@OneToOne(owner)` whose target entity has an `isDeleted` filter:

```typescript
// ✅ CORRECT — surgical, structural, zero side effects on standalone queries
@ManyToOne(() => Person, { filters: { isDeleted: false } })
person!: Person;

@ManyToOne(() => Organization, { filters: { isDeleted: false } })
organization!: Organization;

@ManyToOne(() => User, { filters: { isDeleted: false } })
createdBy!: User;
```

This disables the target's `isDeleted` filter **only** for the auto-`:ref` JOIN MikroORM adds for this relation. Standalone `em.find(Person, ...)` / `em.findOne(Organization, ...)` queries still apply the target's `isDeleted` filter — the override is local to this relation site.

<Info>
**Side effect to be aware of:** when you explicitly populate this relation (`populate: ['person']`), the populate JOIN ALSO loses the target's `isDeleted` filter. A soft-deleted populated row will hydrate fully instead of being filtered out. This is acceptable but each Tier-2/Tier-3 relation MUST add a DTO warn-log (see §3 below).
</Info>

Nullable attribution relations may also need the override when product behavior keeps historical references after the target is soft-deleted and callers populate the relation. Example: `Lead.sourceConversation` is nullable, but it declares `{ filters: { isDeleted: false } }` so historical lead rows are not dropped or distorted when a referenced messaging conversation is soft-deleted.

### 2.2 `active` filter cleanup (MANDATORY for archivable entities)

Every archivable entity should have **two** entity-level filters:

```typescript
@Entity()
@Filter({ name: 'isDeleted', default: true, cond: { isDeleted: false } })
@Filter({ name: 'active', default: true, cond: { isArchived: false } })  // ← NOT { isDeleted: false, isArchived: false }
export class Lead { ... }
```

Both filters default to `true`, so all standard queries apply `isDeleted = false AND isArchived = false`. The `active` filter MUST omit `isDeleted` because the standalone `isDeleted` filter already handles soft-delete exclusion.

<Warning>
The redundant clause is dangerous because it creates the appearance that disabling `active` also restores soft-deleted rows — it doesn't, and reasoning about it correctly requires understanding both filters at once.
</Warning>

**INVARIANTS** (already documented inline on every fixed entity):

- Never disable `isDeleted` while keeping `active` — that would return soft-deleted records.
- To include archived rows, use `{ filters: { active: false } }`.
- To include soft-deleted rows, use `{ filters: { isDeleted: false } }` (rare — typically only in repair scripts and merge code).
- To include both, use `{ filters: false }` (only valid in merge / cascade / RLS-bypass paths — see §2.3).

### 2.3 `filters: false` is reserved for explicit business intent (MANDATORY framing)

`{ filters: false }` disables BOTH `isDeleted` and `active` filters at once. It is **never** a workaround for the `autoJoinRefsForFilters` symptom from §1 — the per-relation override in §2.1 is the structural fix for that. Use `{ filters: false }` only when the code path **explicitly intends** to read soft-deleted or archived rows.

#### When `{ filters: false }` IS legitimate

A request handler / service / job may use `{ filters: false }` if and only if it falls into one of these categories. **Every call site MUST have a code comment naming the category.**

| Category | When to use | Canonical example |
| --- | --- | --- |
| Merge / cascade operations | Bulk re-point children of an entity that is about to be soft-deleted | `PersonMergeService.bulkRepoint()` |
| Restore operations | Un-soft-delete an entity and its dependencies | `RestoreService.restoreEntity()` |
| Conflict detection | Check for existing records regardless of deletion status | Duplicate detection during imports |
| Audit timeline | Show historical state changes including deletions | Admin audit logs, change history |
| Notification fan-out | Send notifications about archived/deleted entities | Email notifications about deleted records |
| Admin operations | Support tools that need full visibility | Admin panels, data export tools |
| Seed / migration | Database initialization and migration scripts | Test fixtures, data seeding |

<Check>
**Every call site MUST have a code comment naming the category** to make the intent explicit and reviewable.
</Check>

## 3. Entity classification and DTO handling

### 3.1 User reference classification

User references fall into three tiers based on business intent:

#### 3.1.1 Tier 1: Audit references (DO NOT override)

These represent **who performed an action** and must maintain historical integrity:

```typescript
// ✅ CORRECT — no filter override for audit refs
@ManyToOne(() => User)
createdBy!: User;

@ManyToOne(() => User)
updatedBy!: User;

@ManyToOne(() => User)
archivedBy!: User;
```

Audit references intentionally break the query when the user is soft-deleted, forcing explicit business decisions about data retention.

#### 3.1.2 Tier 2: Operational references (Override required)

These represent **current business assignments** and should not break queries:

```typescript
// ✅ CORRECT — filter override for operational refs
@ManyToOne(() => User, { filters: { isDeleted: false } })
assignedTo!: User;

@ManyToOne(() => User, { filters: { isDeleted: false } })
owner!: User;
```

#### 3.1.3 Tier 3: Entity ownership (Override required)

These represent **primary ownership** relationships:

```typescript
// ✅ CORRECT — filter override for ownership refs
@ManyToOne(() => Person, { filters: { isDeleted: false } })
person!: Person;

@ManyToOne(() => Organization, { filters: { isDeleted: false } })
organization!: Organization;
```

### 3.2 DTO handling for soft-deleted relations

When a Tier-2 or Tier-3 relation is populated and `populated.isDeleted === true`, the DTO MUST handle this gracefully using one of three strategies:

<Tabs>
  <Tab title="Strategy A: Placeholder">
    ```typescript
    // Use placeholder data for soft-deleted entities
    if (lead.person.isDeleted) {
      return {
        personId: lead.person.id,
        personName: '[Deleted Contact]',
        personEmail: null
      };
    }
    ```
  </Tab>
  
  <Tab title="Strategy B: Omit">
    ```typescript
    // Omit the relation entirely
    if (lead.person.isDeleted) {
      return {
        // personId, personName, personEmail omitted
      };
    }
    ```
  </Tab>
  
  <Tab title="Strategy C: 4xx Error">
    ```typescript
    // Return an error for critical missing data
    if (lead.person.isDeleted) {
      throw new NotFoundException('Contact no longer available');
    }
    ```
  </Tab>
</Tabs>

<Warning>
**NEVER serialize the full Tier-3 payload when soft-deleted** — this would leak deleted entity data to the API response.
</Warning>

## 4. Testing requirements

### 4.1 Mandatory tests for new entities

Every new entity with non-nullable owner relations MUST include these two test scenarios:

<Steps>
  <Step title="Test soft-deleted relation visibility">
    ```typescript
    it('should still be visible when owner is soft-deleted', async () => {
      // Arrange: Create entity with owner
      const owner = await createTestPerson();
      const entity = await createTestEntity({ person: owner });
      
      // Act: Soft-delete the owner
      await personService.softDelete(owner.id);
      
      // Assert: Entity is still findable
      const found = await entityService.findById(entity.id);
      expect(found).toBeDefined();
      expect(found.id).toBe(entity.id);
    });
    ```
  </Step>
  
  <Step title="Test DTO handling of soft-deleted relations">
    ```typescript
    it('should handle soft-deleted owner in DTO serialization', async () => {
      // Arrange: Create entity with owner
      const owner = await createTestPerson();
      const entity = await createTestEntity({ person: owner });
      
      // Act: Soft-delete the owner and fetch DTO
      await personService.softDelete(owner.id);
      const dto = await entityService.getDto(entity.id);
      
      // Assert: DTO handles soft-deleted owner appropriately
      expect(dto.personName).toBe('[Deleted Contact]'); // or omitted, or throws
    });
    ```
  </Step>
</Steps>

### 4.2 Data integrity audit

For Tier 2/3 relations, extend `DataIntegrityAuditService` to detect orphaned references:

```typescript
// Add to DataIntegrityAuditService
async auditEntityReferences(): Promise<IntegrityIssue[]> {
  const orphaned = await this.em.createQueryBuilder(Entity, 'e')
    .leftJoin('e.person', 'p')
    .where('p.id IS NULL OR p.isDeleted = true')
    .andWhere('e.isDeleted = false')
    .getResult();
    
  return orphaned.map(entity => ({
    type: 'orphaned_reference',
    entityType: 'Entity',
    entityId: entity.id,
    message: 'References soft-deleted person'
  }));
}
```

<Note>
The daily cron picks up new audit methods automatically — do NOT add an HTTP endpoint for data integrity checks.
</Note>

## 5. Quick reference

### Common patterns

<AccordionGroup>
  <Accordion title="Entity declaration template">
    ```typescript
    @Entity()
    @Filter({ name: 'isDeleted', default: true, cond: { isDeleted: false } })
    @Filter({ name: 'active', default: true, cond: { isArchived: false } })
    export class MyEntity {
      // Audit refs - no override
      @ManyToOne(() => User)
      createdBy!: User;
      
      // Operational refs - override required
      @ManyToOne(() => User, { filters: { isDeleted: false } })
      assignedTo!: User;
      
      // Owner refs - override required
      @ManyToOne(() => Person, { filters: { isDeleted: false } })
      person!: Person;
    }
    ```
  </Accordion>
  
  <Accordion title="Service query patterns">
    ```typescript
    // Standard query (active only)
    const active = await em.find(MyEntity, { someField: value });
    
    // Include archived
    const withArchived = await em.find(MyEntity, 
      { someField: value }, 
      { filters: { active: false } }
    );
    
    // Merge operation (all records)
    const all = await em.find(MyEntity, 
      { someField: value }, 
      { filters: false } // Category: merge operation
    );
    ```
  </Accordion>
  
  <Accordion title="DTO soft-delete handling">
    ```typescript
    toDto(entity: MyEntity): MyEntityDto {
      if (entity.person.isDeleted) {
        return {
          ...baseFields,
          personName: '[Deleted Contact]',
          personEmail: null
        };
      }
      
      return {
        ...baseFields,
        personName: entity.person.name,
        personEmail: entity.person.email
      };
    }
    ```
  </Accordion>
</AccordionGroup>

### Troubleshooting

<CardGroup cols={2}>
  <Card title="Row exists in DB but API returns empty" icon="database">
    **Cause:** Missing `{ filters: { isDeleted: false } }` on non-nullable relation  
    **Fix:** Add filter override to entity declaration
  </Card>
  
  <Card title="Archived records appearing in results" icon="archive">
    **Cause:** Disabled `active` filter or incorrect filter condition  
    **Fix:** Use `{ filters: { active: false } }` to include archived
  </Card>
  
  <Card title="Soft-deleted data in API response" icon="eye-slash">
    **Cause:** DTO not checking `populated.isDeleted`  
    **Fix:** Add soft-delete handling to DTO serialization
  </Card>
  
  <Card title="Tests pass but production fails" icon="test-tube">
    **Cause:** Missing integration test for soft-deleted relations  
    **Fix:** Add the two mandatory tests from §4.1
