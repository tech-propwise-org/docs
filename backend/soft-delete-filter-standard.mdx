<Warning>
**Status:** Project-wide standard — MANDATORY for all entities, services, and DTOs
</Warning>

This project uses **MikroORM v6** with `loadStrategy: JOINED`, `autoJoinRefsForFilters: true`, and `filtersOnRelations: true`. Combined with our `isDeleted` and `active` entity filters, this produces a **silent data-loss bug** if entities are not declared correctly.

<Note>
A row whose own `is_deleted = false` will silently disappear from every query if any of its non-nullable owner relations points at a soft-deleted target.
</Note>

## Quick Reference

<CardGroup cols={2}>
  <Card title="Entity Relations" icon="code">
    Add `{ filters: { isDeleted: false } }` to non-nullable relations targeting soft-deletable entities
  </Card>
  <Card title="Active Filter" icon="filter">
    Use `{ isArchived: false }` only — NEVER combine with `isDeleted: false`
  </Card>
  <Card title="Bulk Operations" icon="database">
    Pass `{ filters: false }` for merge operations to hit archived AND soft-deleted children
  </Card>
  <Card title="Filters: False" icon="warning">
    Only use with explicit business intent and proper code comments
  </Card>
</CardGroup>

## Essential Rules

| What | Rule |
| --- | --- |
| Non-nullable `@ManyToOne` / `@OneToOne(owner)` to an entity that declares `@Filter({ name: 'isDeleted', ... })` | Add `{ filters: { isDeleted: false } }` to the relation options |
| `@Filter({ name: 'active', ... })` on a soft-deletable + archivable entity | `cond` MUST be `{ isArchived: false }` only — NEVER `{ isDeleted: false, isArchived: false }` |
| Bulk re-point inside a merge service (`em.nativeUpdate(...)` that re-points children of an entity that is about to be soft-deleted) | Pass `{ filters: false }` so the re-point hits archived AND soft-deleted children |
| `{ filters: false }` anywhere else | Allowed only in the explicit-business-intent categories below. Every call site MUST have a code comment naming the category. |

## Why Rows Silently Disappear

MikroORM v6 has three defaults that interact badly with soft-deletable graphs:

```typescript
// mikro-orm.config.ts (defaults — NOT to be disabled globally)
loadStrategy: LoadStrategy.JOINED;       // populated relations are joined into the main SELECT
autoJoinRefsForFilters: true;            // auto-add :ref JOIN for any relation whose target has filters
filtersOnRelations: true;                // apply target-entity filters via JOIN ON
```

For every M:1 / 1:1-owner relation on the queried entity, MikroORM checks whether the target entity declares any default-enabled filters. If so, it adds an auto `:ref` JOIN to that relation **even if you did not populate it**.

<Warning>
For a **non-nullable** M:1 the resulting JOIN type is `INNER JOIN`. This means if any non-nullable joined target fails the filter (e.g., `Person.isDeleted = true`), the **entire row is silently dropped** — even though the main entity has `is_deleted = false`.
</Warning>

### Example Failure Case

```sql
-- What em.find(Lead, { id: leadId }) becomes:
SELECT lead.*
  FROM lead
  INNER JOIN person p ON p.id = lead.person_id AND p.is_deleted = false
  INNER JOIN organization o ON o.id = lead.organization_id AND o.is_deleted = false
 WHERE lead.id = $1
   AND lead.is_deleted = false;
```

If **any** of those non-nullable joined targets fails the filter, the lead row disappears entirely.

### Real Production Examples

| Scenario | Symptom |
| --- | --- |
| Two persons merged → secondary `Person.isDeleted = true` → archived Lead still pointing at secondary Person | The archived Lead was invisible everywhere despite existing in the DB |
| `DealStage` cleanup → custom stage soft-deleted while a Deal still pointed at it | The Deal vanished from list endpoints |
| Agent leaves org → `User.isDeleted = true` → every Deal/Contact/Company/Listing they `createdBy` would vanish | Silent data loss across entire user history |

## The Fix: Three Coordinated Parts

<Steps>
  <Step title="Per-relation filter override (MANDATORY)">
    Declare `{ filters: { isDeleted: false } }` on **every non-nullable** `@ManyToOne` / `@OneToOne(owner)` whose target entity has an `isDeleted` filter.

    ```typescript
    // ✅ CORRECT — surgical, structural, zero side effects on standalone queries
    @ManyToOne(() => Person, { filters: { isDeleted: false } })
    person!: Person;

    @ManyToOne(() => Organization, { filters: { isDeleted: false } })
    organization!: Organization;

    @ManyToOne(() => User, { filters: { isDeleted: false } })
    createdBy!: User;
    ```

    This disables the target's `isDeleted` filter **only** for the auto-`:ref` JOIN MikroORM adds for this relation.
  </Step>

  <Step title="Active filter cleanup (MANDATORY for archivable entities)">
    Every archivable entity should have **two** entity-level filters:

    ```typescript
    @Entity()
    @Filter({ name: 'isDeleted', default: true, cond: { isDeleted: false } })
    @Filter({ name: 'active', default: true, cond: { isArchived: false } })  // ← NOT { isDeleted: false, isArchived: false }
    export class Lead { ... }
    ```

    <Info>
    The `active` filter MUST omit `isDeleted` because the standalone `isDeleted` filter already handles soft-delete exclusion.
    </Info>
  </Step>

  <Step title="Restrict filters: false usage">
    `{ filters: false }` disables BOTH `isDeleted` and `active` filters at once. Use only when the code path **explicitly intends** to read soft-deleted or archived rows.
  </Step>
</Steps>

### Filter Invariants

<Note>
**INVARIANTS** (documented inline on every fixed entity):
- Never disable `isDeleted` while keeping `active` — that would return soft-deleted records
- To include archived rows, use `{ filters: { active: false } }`
- To include soft-deleted rows, use `{ filters: { isDeleted: false } }` (rare)
- To include both, use `{ filters: false }` (only valid in specific business contexts)
</Note>

## When `filters: false` Is Legitimate

<Warning>
Every call site MUST have a code comment naming the category.
</Warning>

### Valid Use Cases

<AccordionGroup>
  <Accordion title="Merge / Repair / Cascade">
    Re-pointing or cascading children of a row about to be soft-deleted, so archived/soft-deleted children get re-pointed too.

    ```typescript
    // ✅ CORRECT — merge: re-point archived AND soft-deleted children
    await em.nativeUpdate(
      'Lead',
      { person: { $in: uniqueSecondaryIds } },
      { person: primaryPersonId },
      { filters: false }, // merge category: hit archived/soft-deleted leads too
    );
    ```
  </Accordion>

  <Accordion title="Restore / Unarchive / Revive">
    Resolver/endpoint that must find a soft-archived row to bring it back.

    ```typescript
    // ✅ CORRECT — restore category: finding archived item to unarchive
    const contact = await em.findOne(Contact, { id }, { filters: false });
    ```
  </Accordion>

  <Accordion title="Conflict Detection on Re-create">
    Looking up a soft-deleted row by unique key to prevent constraint violation, or to revive in place of `INSERT`.

    ```typescript
    // ✅ CORRECT — conflict detection: check for soft-deleted membership
    const existing = await em.findOne(TeamMembership, { user, team }, { filters: false });
    ```
  </Accordion>

  <Accordion title="Audit / Timeline / History Rendering">
    Fetching deleted stages/users/rules so historical entries render correctly.

    ```typescript
    // ✅ CORRECT — audit category: show historical data
    const timeline = await em.find(AuditEvent, {}, { filters: false });
    ```
  </Accordion>

  <Accordion title="Notification Fan-out">
    Loading archived deals/leads so their creators are still notified when a stakeholder changes.

    ```typescript
    // ✅ CORRECT — notification category: notify across archived parents
    const leads = await em.find(Lead, { contact }, { filters: false });
    ```
  </Accordion>

  <Accordion title="Admin / System Maintenance">
    System-admin tools that intentionally surface soft-deleted records.

    ```typescript
    // ✅ CORRECT — admin category: system maintenance view
    const allUsers = await em.find(User, {}, { filters: false });
    ```
  </Accordion>

  <Accordion title="Seed / Migration / Fixture / Repair Scripts">
    Seeders, migrations, repair CLI scripts.

    ```typescript
    // ✅ CORRECT — seed category: data initialization
    const existingData = await em.find(Entity, {}, { filters: false });
    ```
  </Accordion>
</AccordionGroup>

### Invalid Use Cases

<CodeGroup>
```typescript ❌ NEVER
// Using `filters: false` to "fix" a list endpoint
async getLeads(...) {
  return em.find(Lead, { ... }, { filters: false }); // WRONG: use per-relation override instead
}
```

```typescript ❌ NEVER  
// Using `filters: false` on public endpoints
@Get('search')
@PublicEndpoint()
async publicSearch(...) {
  return em.find(Entity, { ... }, { filters: false }); // WRONG: data leak vector
}
```
</CodeGroup>

## Alternative Approaches by Intent

<Tabs>
  <Tab title="Active rows only">
    No options — both filters are on by default
    ```typescript
    const leads = await em.find(Lead, { status: 'open' });
    ```
  </Tab>
  
  <Tab title="Active + archived">
    Use `{ filters: { active: false } }` — keeps `isDeleted = false` enforced
    ```typescript
    const leads = await em.find(Lead, { id }, { filters: { active: false } });
    ```
  </Tab>
  
  <Tab title="Include soft-deleted">
    Use `{ filters: { isDeleted: false } }` on specific lookups (e.g., revive flow)
    ```typescript
    const contact = await em.findOne(Contact, { id }, { filters: { isDeleted: false } });
    ```
  </Tab>
  
  <Tab title="Both archived + soft-deleted">
    Use `{ filters: false }` with proper business justification
    ```typescript
    // merge category: re-point all children before soft-delete
    await em.nativeUpdate(Entity, filter, update, { filters: false });
    ```
  </Tab>
</Tabs>

## User Reference Classification

When adding user references to new entities, classify them properly:

### Tier 1: Audit References
`createdBy`, `updatedBy`, `archivedBy` — these track who performed actions and should never cause row disappearance.

### Tier 2-3: Business References  
`assignedTo`, `owner`, `recipient`, `user` — these represent business relationships and need careful DTO handling.

<Warning>
For DTOs with Tier-3 owner relations (`person`, `organization`): If `populated.isDeleted === true`, the DTO MUST use placeholder/omit/4xx strategies. NEVER serialize the full Tier-3 payload when soft-deleted.
</Warning>
