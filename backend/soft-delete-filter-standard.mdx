<Warning>
**Status:** Project-wide standard — MANDATORY for all entities, services, and DTOs
</Warning>

This project uses **MikroORM v6** with `loadStrategy: JOINED`, `autoJoinRefsForFilters: true`, and `filtersOnRelations: true`. Combined with our `isDeleted` and `active` entity filters, this produces a **silent data-loss bug** if entities are not declared correctly.

<Note>
**Critical Issue:** A row whose own `is_deleted = false` will silently disappear from every query if any of its non-nullable owner relations points at a soft-deleted target.
</Note>

## Quick Reference

Every backend developer MUST follow these rules:

<CardGroup cols={2}>
<Card title="Non-nullable Relations" icon="link">
Add `{ filters: { isDeleted: false } }` to relation options for `@ManyToOne` / `@OneToOne(owner)` targeting entities with `@Filter({ name: 'isDeleted' })`
</Card>

<Card title="Active Filter Pattern" icon="filter">
Use `{ isArchived: false }` only — NEVER `{ isDeleted: false, isArchived: false }`
</Card>

<Card title="Bulk Operations" icon="database">
Pass `{ filters: false }` for re-pointing operations in merge services with explicit business intent comment
</Card>

<Card title="User References" icon="user">
Classify as Tier 1 (audit), Tier 2, or Tier 3 based on relationship type and handle soft-deleted references appropriately
</Card>
</CardGroup>

## Root Cause Analysis

### Why Rows Silently Disappear

MikroORM v6 has three defaults that interact badly with soft-deletable entity graphs:

```typescript
// mikro-orm.config.ts (defaults — NOT to be disabled globally)
loadStrategy: LoadStrategy.JOINED; // populated relations are joined into the main SELECT
autoJoinRefsForFilters: true; // auto-add :ref JOIN for any relation whose target has filters
filtersOnRelations: true; // apply target-entity filters via JOIN ON
```

<Info>
For every M:1 / 1:1-owner relation on the queried entity, MikroORM checks whether the target entity declares any default-enabled filters. If so, it adds an auto `:ref` JOIN to that relation **even if you did not populate it**.
</Info>

### The Failure Mode

For **non-nullable** M:1 relations, the resulting JOIN type is `INNER JOIN`:

```sql
SELECT lead.*
  FROM lead
  INNER JOIN person p ON p.id = lead.person_id AND p.is_deleted = false
  INNER JOIN organization o ON o.id = lead.organization_id AND o.is_deleted = false
 WHERE lead.id = $1
   AND lead.is_deleted = false;
```

<Warning>
If **any** of those non-nullable joined targets fails the filter (e.g. `Person.isDeleted = true`), the **lead row is silently dropped** — even though `lead.is_deleted = false`.
</Warning>

### Real Production Examples

<AccordionGroup>
<Accordion title="Person Merge Scenario">
Two persons merged → secondary `Person.isDeleted = true` → an archived Lead still pointing at the secondary Person resulted in the archived Lead being invisible everywhere even though it existed in the DB.
</Accordion>

<Accordion title="Deal Stage Cleanup">
`DealStage` cleanup → custom stage soft-deleted while a Deal still pointed at it caused the Deal to vanish from list endpoints.
</Accordion>

<Accordion title="Agent Departure">
Agent leaves the org → `User.isDeleted = true` → every Deal / Contact / Company / Listing they `createdBy` would have vanished if we shipped the relation override without the Tier-1 carve-out.
</Accordion>
</AccordionGroup>

### Special Case: excludeAiServiceUser Filter

<Note>
The `excludeAiServiceUser` filter is **OPT-IN** (`default: false`) to prevent the same auto-join issue with service accounts.
</Note>

```typescript
@Filter({
  name: 'excludeAiServiceUser',
  default: false,
  cond: { isAiServiceUser: false, isSystemServiceUser: false },
})
```

This filter was inverted from `default: true` because service accounts stamp non-nullable audit columns, and the auto-join was silently dropping service-created rows.

## The Three-Part Fix

<Warning>
The fix has **three** coordinated parts. Skipping any one re-introduces the bug.
</Warning>

### Part 1: Per-Relation Filter Override (MANDATORY)

<Steps>
<Step title="Identify Target Relations">
Find every non-nullable `@ManyToOne` / `@OneToOne(owner)` relation pointing to an entity that declares `@Filter({ name: 'isDeleted' })`.
</Step>

<Step title="Add Filter Override">
Declare `{ filters: { isDeleted: false } }` in the relation options:

```typescript
@Entity()
export class Lead {
  @ManyToOne(() => Person, { 
    nullable: false,
    filters: { isDeleted: false } // MANDATORY override
  })
  person!: Rel<Person>;
  
  @ManyToOne(() => Organization, { 
    nullable: false,
    filters: { isDeleted: false } // MANDATORY override
  })
  organization!: Rel<Organization>;
}
```
</Step>

<Step title="Verify Implementation">
The override tells MikroORM to use `LEFT JOIN` instead of `INNER JOIN`, making the relation hydrate to `null` when the target is soft-deleted rather than dropping the entire row.
</Step>
</Steps>

### Part 2: Correct Active Filter Pattern

<Warning>
The `active` filter condition MUST be `{ isArchived: false }` only — NEVER combine with `isDeleted`.
</Warning>

```typescript
// ❌ WRONG — creates the same auto-join problem
@Filter({ 
  name: 'active', 
  default: true, 
  cond: { isDeleted: false, isArchived: false } 
})

// ✅ CORRECT — only filters on archive status
@Filter({ 
  name: 'active', 
  default: true, 
  cond: { isArchived: false } 
})
```

### Part 3: Explicit Business Intent for `{ filters: false }`

<Tip>
`{ filters: false }` is allowed only in specific business contexts and MUST include a code comment naming the category.
</Tip>

**Approved categories:**

<Tabs>
<Tab title="Merge Operations">
```typescript
// Category: merge — re-pointing children before soft-delete
await em.nativeUpdate(Lead, 
  { person: oldPersonId }, 
  { person: newPersonId }, 
  { filters: false }
);
```
</Tab>

<Tab title="Restore Operations">
```typescript
// Category: restore — checking soft-deleted dependencies
const deletedDeps = await em.find(Person, 
  { organization: orgId }, 
  { filters: false }
);
```
</Tab>

<Tab title="Admin Operations">
```typescript
// Category: admin — full data access for compliance
const allRecords = await em.find(Lead, 
  { createdAt: { $gte: startDate } }, 
  { filters: false }
);
```
</Tab>
</Tabs>

### DTO Handling for Soft-Deleted References

When a Tier-3 owner relation has `populated.isDeleted === true`:

<Tabs>
<Tab title="Strategy A: Placeholder">
```typescript
// Use placeholder data
return {
  person: {
    id: lead.person.id,
    displayName: '[Deleted Contact]',
    isDeleted: true
  }
}
```
</Tab>

<Tab title="Strategy B: Omit">
```typescript
// Omit the relation entirely
return {
  // person field not included
  organization: lead.organization
}
```
</Tab>

<Tab title="Strategy C: Error Response">
```typescript
// Return 4xx error
if (lead.person.isDeleted) {
  throw new BadRequestException('Referenced contact no longer available');
}
```
</Tab>
</Tabs>

<Warning>
**NEVER serialize the full Tier-3 payload when soft-deleted** — this exposes deleted business data.
</Warning>

### Testing Requirements

<CodeGroup>

```typescript Soft-Delete Regression Test
describe('Entity visibility with soft-deleted relations', () => {
  it('should not hide entity when non-nullable relation target is soft-deleted', async () => {
    // Arrange: Create entity with relation
    const person = await createPerson();
    const lead = await createLead({ person });
    
    // Act: Soft-delete the relation target
    await softDeletePerson(person.id);
    
    // Assert: Entity should still be visible
    const foundLead = await em.findOne(Lead, { id: lead.id });
    expect(foundLead).toBeDefined();
    expect(foundLead.person).toBeNull(); // Relation hydrates to null
  });
});
```

```typescript DTO Handling Test  
describe('DTO with soft-deleted references', () => {
```
