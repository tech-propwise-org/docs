This document outlines critical issues discovered during a deep review of the Activity Monitor and Escalation modules. These issues range from dead code and non-deterministic behavior to potential data consistency problems.

## Dead code and unused methods

### `markAlertIgnored` method is unused

<Warning>
The `ActivityMonitorService.markAlertIgnored` method is defined but never called anywhere in the codebase.
</Warning>

The follow-up processing logic marks alerts as `IGNORED` using raw SQL queries instead of utilizing this dedicated method. This creates inconsistency in how alert status updates are handled.

**Recommendation**: Either consolidate all alert status updates to use the `markAlertIgnored` method or remove it entirely to reduce code complexity.

## Non-deterministic behavior

### Team lead resolution lacks determinism

<Warning>
Both Activity Monitor and Escalation modules use `LIMIT 1` or `DISTINCT ON` without deterministic `ORDER BY` clauses when multiple `TEAM_ADMIN` users exist.
</Warning>

When a team has multiple administrators, the system arbitrarily selects which admin receives alerts. This selection could change between different scans, leading to inconsistent alert routing.

**Impact**: 
- Unpredictable alert distribution
- Potential confusion among team administrators
- Difficulty in tracking alert assignments

**Recommendation**: Implement a deterministic ordering mechanism, such as:
- Ordering by user creation date
- Using a round-robin assignment system
- Implementing a designated primary admin role

## Data consistency issues

### Insert idempotency mismatch in escalation

<Note>
Escalation comments reference `INSERT … ON CONFLICT DO NOTHING`, but the actual implementation uses `INSERT … WHERE NOT EXISTS`.
</Note>

Under concurrent transaction scenarios, unique constraint violations can still occur on partial unique indexes. While unlikely, this edge case is currently unhandled and could cause unexpected errors.

**Recommendation**: Align the implementation with the documented approach using proper conflict resolution.

## Performance and fairness concerns

### Inconsistent batch processing limits

<Info>
Activity Monitor processes 200 leads per rule per tick, while Escalation handles 500 leads per batch.
</Info>

This discrepancy in batch sizes, combined with the lack of fairness mechanisms, means:
- Large backlogs drain slowly
- Rules processed earlier in iteration order receive priority
- No load balancing across different rules

**Recommendation**: 
- Standardize batch sizes across modules
- Implement round-robin or weighted fair queuing for rule processing

## State management problems

### Exhausted escalation tiers create limbo state

<Warning>
When all escalation tiers have fired, trackers remain in `ACTIVE` state with `next_escalation_at = NULL`, creating an indefinite limbo state.
</Warning>

These trackers accumulate over time with:
- No scheduled actions
- No dashboard visibility into "stuck" state
- Dependency on manual intervention or auto-resolution events

**Recommendation**: Implement a final state transition or cleanup mechanism for exhausted escalation chains.

### Stranded activity alerts when notifications disabled

When `notifyTeamLead` is set to `false`, the system:
1. Stamps `follow_up_sent_at` timestamp
2. Never transitions alerts out of `PENDING`/`ESCALATED` status
3. Leaves alerts in an indefinite state

<Warning>
Alerts with disabled notifications can linger forever without reaching a terminal state.
</Warning>

## Semantic inconsistencies

### Divergent resolution status semantics

<Tabs>
  <Tab title="Archived/Deleted Leads">
    - Status: `CANCELLED`
    - Reason: `AUTO_ARCHIVED` or `AUTO_DELETED`
  </Tab>
  <Tab title="Auto-Resolved Cases">
    - Status: `RESOLVED` 
    - Triggers: Activity, stage changes, reassignment
  </Tab>
</Tabs>

This inconsistency affects:
- Analytics that count only `RESOLVED` status
- Dashboard metrics and reporting
- Overall escalation completion tracking

**Impact**: Analytics counting only `RESOLVED` status would undercount total closed escalations.

## Documentation discrepancies

### API route specification mismatch

<Note>
Activity Monitor specification documents reference REST-style paths like `/activity-monitor/rules`, but actual controller routes use hyphenated resource naming.
</Note>

**Documented paths**:
- `/activity-monitor/rules`
- `/activity-monitor/alerts`

**Actual implementation**:
- `/activity-monitor-rules` 
- `/activity-monitor-alerts`

**Recommendation**: Update either the documentation or the route implementation to maintain consistency.

## Next steps
