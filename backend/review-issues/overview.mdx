This document outlines critical code quality issues discovered during deep reviews of the activity monitor and escalation modules. These issues range from dead code to potential data integrity problems.

<Warning>
Several issues identified can lead to data inconsistencies or system instability in production environments.
</Warning>

## Dead Code and Unused Methods

### `markAlertIgnored` Method Never Used

The `ActivityMonitorService.markAlertIgnored` method is defined but never called anywhere in the codebase. Instead, the follow-up processing directly executes raw SQL to mark alerts as `IGNORED`.

<CodeGroup>
```sql SQL - Current Implementation
UPDATE activity_alerts 
SET status = 'IGNORED' 
WHERE id = ?
```

```javascript JavaScript - Unused Method
// This method exists but is never called
markAlertIgnored(alertId) {
  // Implementation exists but unused
}
```
</CodeGroup>

**Recommended Action**: Either consolidate to use the existing method or remove the dead code to reduce maintenance overhead.

## Non-Deterministic Behavior

### Team Lead Resolution Randomness

Both activity monitor and escalation modules suffer from non-deterministic behavior when selecting team administrators.

<Note>
The issue occurs when multiple `TEAM_ADMIN` users exist on a single team. The system uses `LIMIT 1` or `DISTINCT ON` without proper ordering.
</Note>

```sql
-- Problematic query - non-deterministic results
SELECT user_id FROM team_members 
WHERE team_id = ? AND role = 'TEAM_ADMIN' 
LIMIT 1;
```

**Impact**: Which admin receives alerts is arbitrary and could change between scans, leading to inconsistent notification patterns.

## Concurrency and Data Integrity Issues

### Insert Idempotency Mismatch

<Warning>
The escalation module has a mismatch between documented and implemented idempotency patterns.
</Warning>

<Tabs>
<Tab title="Documentation Claims">
```sql
INSERT INTO escalation_comments (...) 
ON CONFLICT DO NOTHING;
```
</Tab>
<Tab title="Actual Implementation">
```sql
INSERT INTO escalation_comments (...) 
WHERE NOT EXISTS (SELECT 1 FROM escalation_comments WHERE ...);
```
</Tab>
</Tabs>

Under concurrent transactions, unique violations are still possible on the partial unique index, though unlikely.

## Performance and Fairness Concerns

### Inconsistent Batch Processing Limits

The scanner modules use different batch size caps without fairness mechanisms:

<CardGroup cols={2}>
<Card title="Activity Monitor" icon="chart-line">
Processes 200 leads per rule per tick
</Card>
<Card title="Escalation Module" icon="clock">
Processes 500 leads per rule per tick
</Card>
</CardGroup>

**Problem**: Large backlogs drain slowly, and earlier rules in iteration order receive priority processing, creating unfair resource allocation.

## State Management Issues

### Exhausted Escalation Tiers

<Info>
When all escalation tiers have fired, trackers enter an undefined state that can persist indefinitely.
</Info>

<Steps>
<Step title="All Tiers Complete">
Escalation tracker completes all configured notification tiers
</Step>
<Step title="Limbo State">
Tracker remains `ACTIVE` with `next_escalation_at = NULL`
</Step>
<Step title="Accumulation">
Trackers accumulate with no scheduled actions and no dashboard visibility
</Step>
</Steps>

### Stranded Activity Alerts

When `notifyTeamLead` is set to `false`, activity alerts can become permanently stranded:

```javascript
if (!notifyTeamLead) {
  // Stamps follow_up_sent_at but never transitions status
  await this.stampFollowUpSent(alertId);
  // Alert remains in PENDING/ESCALATED forever
}
```

## Semantic Inconsistencies

### Resolution Status Terminology

<AccordionGroup>
<Accordion title="RESOLVED vs CANCELLED Semantics">
The system uses inconsistent terminology for closed escalations:

- **CANCELLED + AUTO_ARCHIVED/AUTO_DELETED**: Used for archived/deleted leads
- **RESOLVED**: Used by auto-resolver for activity/stage/reassign events

This divergence can cause analytics undercounting when filtering only on `RESOLVED` status.
</Accordion>
</AccordionGroup>
