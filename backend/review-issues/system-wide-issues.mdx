# System-Wide Review Issues

<Warning>
This document outlines critical issues identified during a deep review of the Activity Monitor and Escalation modules that require immediate attention to ensure system reliability and data integrity.
</Warning>

## Dead Code and Unused Methods

### markAlertIgnored Method

<Info>
**Issue**: The `ActivityMonitorService.markAlertIgnored` method is defined but never called throughout the codebase.
</Info>

The follow-up processing path marks alerts as `IGNORED` using raw SQL statements instead of utilizing this dedicated method. This creates inconsistency in the codebase and potential maintenance issues.

**Recommended Actions:**
- Consolidate alert status updates to use the `markAlertIgnored` method
- Alternatively, remove the unused method if raw SQL approach is preferred
- Ensure consistent alert status handling across all code paths

## Non-Deterministic Behavior

### Team Lead Resolution

<Warning>
Both Activity Monitor and Escalation modules exhibit non-deterministic behavior when selecting team administrators.
</Warning>

The current implementation uses `LIMIT 1` or `DISTINCT ON` clauses without deterministic `ORDER BY` statements when multiple `TEAM_ADMIN` users exist on a team. This results in:

- Arbitrary selection of which admin receives alerts
- Potential inconsistency between different scan cycles
- Unpredictable alert routing behavior

**Impact**: Alert distribution becomes unreliable and may change unexpectedly between processing cycles.

## Data Integrity Issues

### Insert Idempotency Mismatch

<Note>
**Module**: Escalation system
**Issue**: Inconsistency between documented and implemented conflict resolution
</Note>

The escalation module comments reference `INSERT ... ON CONFLICT DO NOTHING` patterns, but the actual code implementation uses `INSERT ... WHERE NOT EXISTS`. Under concurrent transaction scenarios:

- Unique violations remain possible on partial unique indexes
- Error handling for concurrent insertions is incomplete
- System behavior under high concurrency is unpredictable

### Scanner Performance and Fairness

The scanning mechanisms across modules have different configurations that impact performance and fairness:

<Tabs>
  <Tab title="Activity Monitor">
    - **Batch limit**: 200 leads per rule per tick
    - **Processing order**: Sequential by rule iteration order
  </Tab>
  <Tab title="Escalation">
    - **Batch limit**: 500 items per scan
    - **Processing order**: Sequential by rule iteration order
  </Tab>
</Tabs>

**Problems:**
- Large backlogs process slowly due to batch size limitations
- No fairness mechanism exists across different rules
- Earlier rules in iteration order receive processing priority
- Uneven resource allocation across rule sets

## State Management Issues

### Exhausted Escalation Tiers

<Warning>
Critical issue: Escalation trackers can become permanently stranded in an active state.
</Warning>

When all escalation tiers have been triggered:
- Tracker remains in `ACTIVE` status
- `next_escalation_at` field is set to `NULL`
- Tracker persists indefinitely until auto-resolution events fire
- No dashboard visibility into "stuck" tracker states
- Potential accumulation of stale trackers with no scheduled actions

### Stranded Activity Alerts

<Info>
**Trigger condition**: When `notifyTeamLead` configuration is set to `false`
</Info>

The follow-up processing path exhibits problematic behavior:
1. `follow_up_sent_at` timestamp is recorded
2. Alert remains in `PENDING` or `ESCALATED` status
3. No transition to terminal state occurs
4. Alerts can persist indefinitely without resolution

## Semantic Inconsistencies

### Status Resolution Terminology

<AccordionGroup>
  <Accordion title="RESOLVED vs CANCELLED Status Usage">
    **Archived/Deleted Leads:**
    - Tracker status: `CANCELLED`
    - Resolution reason: `AUTO_ARCHIVED` or `AUTO_DELETED`

    **Auto-Resolver Actions:**
    - Tracker status: `RESOLVED`
    - Covers: activity changes, stage transitions, reassignments

    **Analytics Impact:**
    - Dashboards counting only `RESOLVED` status will undercount total closed escalations
    - Inconsistent terminology affects reporting accuracy
  </Accordion>
</AccordionGroup>
