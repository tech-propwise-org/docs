This document outlines critical issues identified during a comprehensive review of the Activity Monitor and Escalation modules. These issues range from dead code to non-deterministic behavior and potential system states that could lead to operational problems.

## Overview of Issues

The review identified 8 distinct categories of issues affecting the reliability, predictability, and maintainability of the Activity Monitor and Escalation systems.

<Warning>
Several of these issues can lead to alerts being stranded in pending states or non-deterministic behavior in production environments.
</Warning>

## Critical Issues

<AccordionGroup>
<Accordion title="1. Dead Code in Alert Management">

### `markAlertIgnored` Method Never Used

The `ActivityMonitorService.markAlertIgnored` method is defined but never called anywhere in the codebase.

**Problem:** The follow-up path marks alerts as `IGNORED` using raw SQL inline instead of utilizing this dedicated method.

**Impact:**
- Code maintenance burden
- Inconsistent alert state management
- Potential for bugs when the inline SQL diverges from the method implementation

**Resolution Options:**
- Consolidate to use the `markAlertIgnored` method consistently
- Remove the unused method if inline SQL is preferred

</Accordion>

<Accordion title="2. Non-Deterministic Team Lead Resolution">

### Arbitrary Admin Selection

Both modules use `LIMIT 1` or `DISTINCT ON` without deterministic `ORDER BY` clauses when multiple `TEAM_ADMIN` users exist.

**Problem:** Which admin receives alerts is arbitrary and could change between scans.

```sql
-- Problematic query pattern
SELECT user_id FROM team_members 
WHERE team_id = ? AND role = 'TEAM_ADMIN' 
LIMIT 1;
```

**Impact:**
- Unpredictable alert routing
- Inconsistent user experience
- Difficulty in debugging escalation paths

<Tip>
Add a deterministic ordering criteria such as `ORDER BY user_id` or `ORDER BY created_at` to ensure consistent behavior.
</Tip>

</Accordion>

<Accordion title="3. Insert Idempotency Race Condition">

### Escalation Comments Concurrency Issue

Documentation references `INSERT ... ON CONFLICT DO NOTHING`, but the actual code uses `INSERT ... WHERE NOT EXISTS`.

**Problem:** Under concurrent transactions, unique violations are still possible on partial unique indexes.

**Impact:**
- Potential application crashes during high concurrency
- Data integrity issues
- Inconsistent comment creation

**Technical Details:**
- Race condition window exists between the `WHERE NOT EXISTS` check and the actual insert
- Partial unique indexes may not prevent all duplicate scenarios

</Accordion>

<Accordion title="4. Unfair Batch Processing">

### Inconsistent Batch Caps and No Fairness

- Activity monitor: 200 leads/rule/tick
- Escalation: 500 leads/rule/tick

**Problem:** Large backlogs drain slowly with no fairness mechanism across rules.

**Impact:**
- Rules processed earlier in iteration order get priority
- Some rules may be starved of processing time
- Inconsistent SLA performance across different rule types

<Note>
Consider implementing a round-robin or weighted fair queuing mechanism to ensure all rules receive processing time.
</Note>

</Accordion>

<Accordion title="5. Stuck Escalation Trackers">

### Limbo State After Tier Exhaustion

When all escalation tiers have fired, trackers remain `ACTIVE` with `next_escalation_at = NULL`.

**Problem:** Trackers sit indefinitely until an auto-resolution event fires.

**Consequences:**
- Accumulation of stale trackers with no scheduled actions
- No dashboard visibility into "stuck" states
- Potential memory/storage leaks over time

**Current State Flow:**
```
ACTIVE (all tiers fired) -> next_escalation_at = NULL -> Indefinite wait
```

**Recommended State Flow:**
```
ACTIVE (all tiers fired) -> EXHAUSTED -> Manual review or auto-timeout
```

</Accordion>
</AccordionGroup>

## Configuration and State Management Issues

<AccordionGroup>
<Accordion title="6. Stranded Activity Alerts">

### `notifyTeamLead = false` Configuration Problem

When `notifyTeamLead` is disabled, the follow-up path stamps `follow_up_sent_at` but never transitions alerts out of `PENDING`/`ESCALATED` status.

**Problem Flow:**
<Steps>
<Step title="Alert Created">
Alert enters `PENDING` state
</Step>
<Step title="Follow-up Processing">
`follow_up_sent_at` timestamp is set
</Step>
<Step title="No State Transition">
Alert remains in `PENDING`/`ESCALATED` indefinitely
</Step>
</Steps>

**Impact:**
- Alerts linger forever without terminal state
- Dashboard metrics become inaccurate
- No clear resolution path for operators

</Accordion>

<Accordion title="7. Inconsistent Resolution Semantics">

### RESOLVED vs CANCELLED Status Divergence

Different resolution paths use different status codes for similar outcomes:

| Trigger | Status | Resolution Type |
|---------|--------|-----------------|
| Archived/deleted leads | `CANCELLED` | `AUTO_ARCHIVED`/`AUTO_DELETED` |
| Auto-resolver actions | `RESOLVED` | Activity/stage/reassign |

**Impact:**
- Analytics counting only `RESOLVED` status undercount total closed escalations
- Inconsistent reporting and metrics
- Confusion in operational dashboards

<Warning>
This semantic inconsistency can lead to significant gaps in reporting and monitoring effectiveness.
</Warning>

</Accordion>

<Accordion title="8. API Documentation Mismatch">

### Specification vs Implementation Divergence

**Documentation References:**
- `/activity-monitor/rules`
- `/activity-monitor/alerts`

**Actual Controller Routes:**
- `/activity-monitor-rules`
- `/activity-monitor-alerts`

**Impact:**
- Developer confusion during integration
- Potential API client failures
- Documentation maintenance burden

<Check>
Ensure all API documentation reflects the actual hyphenated resource-style routes used in the implementation.
</Check>

</Accordion>
</AccordionGroup>
