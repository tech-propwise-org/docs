This document outlines critical issues identified during a comprehensive code review of the Activity Monitor and Escalation modules. These findings highlight areas requiring immediate attention to improve system reliability and maintainability.

## Dead Code Issues

### Unused markAlertIgnored Method

<Warning>
The `ActivityMonitorService.markAlertIgnored` method is defined but never called in the codebase.
</Warning>

**Issue**: The follow-up path marks alerts as `IGNORED` using raw SQL inline instead of utilizing this dedicated method.

**Impact**: Code bloat and potential maintenance confusion

**Recommendation**: Either consolidate alert marking logic to use the existing method or remove the unused code entirely.

## Non-Deterministic Behavior

### Team Lead Resolution Inconsistency

<Warning>
Both Activity Monitor and Escalation modules exhibit non-deterministic behavior when selecting team leads.
</Warning>

**Issue**: The system uses `LIMIT 1` or `DISTINCT ON` without deterministic `ORDER BY` clauses when multiple `TEAM_ADMIN` users exist on a team.

**Impact**: 
- Alert assignment becomes arbitrary
- Results can vary between database scans
- Unpredictable user experience

**Recommendation**: Implement consistent ordering logic (e.g., by user ID, creation date, or last activity) to ensure reproducible team lead selection.

## Database Consistency Issues

### Insert Idempotency Mismatch

<Warning>
Escalation module comments reference `INSERT … ON CONFLICT DO NOTHING` but actual implementation uses `INSERT … WHERE NOT EXISTS`.
</Warning>

**Issue**: Under concurrent transactions, unique violations remain possible on partial unique indexes.

**Impact**: 
- Potential application errors during high concurrency
- Data inconsistency risks
- Unhandled exception scenarios

**Recommendation**: Align implementation with documented approach using proper conflict resolution.

## Performance and Scalability Concerns

### Inconsistent Batch Processing Limits

<Info>
Scanner batch processing caps differ between modules without clear justification.
</Info>

**Current Configuration**:
- Activity Monitor: 200 leads per rule per tick
- Escalation: 500 leads per rule per tick

**Issues**:
- Large backlogs drain slowly
- No fairness mechanism across rules
- Earlier rules in iteration order receive priority

**Recommendation**: 
1. Standardize batch sizes based on performance testing
2. Implement round-robin or weighted fairness algorithms
3. Add backlog monitoring and alerting

## State Management Problems

### Exhausted Escalation Tiers

<Warning>
Escalation trackers become stranded when all tiers have been exhausted.
</Warning>

**Issue**: When all escalation tiers fire, trackers remain `ACTIVE` with `next_escalation_at = NULL`, sitting indefinitely until auto-resolution occurs.

**Impact**:
- Accumulation of stale trackers
- No scheduled actions for affected items  
- Poor dashboard visibility into "stuck" states

**Recommendation**: Implement a final state transition or cleanup mechanism for exhausted escalation paths.

### Activity Alert State Limbo

<Warning>
Activity alerts can become permanently stranded when `notifyTeamLead = false`.
</Warning>

**Issue**: The follow-up path stamps `follow_up_sent_at` but never transitions alerts out of `PENDING`/`ESCALATED` status.

**Impact**: Alerts linger indefinitely with no terminal state resolution

**Recommendation**: Ensure proper state transitions regardless of team lead notification settings.

## Semantic Inconsistencies

### Status Resolution Terminology

<Info>
Different resolution mechanisms use inconsistent status terminology.
</Info>

**Current Behavior**:
- Archived/deleted leads → `CANCELLED` status with `AUTO_ARCHIVED`/`AUTO_DELETED`
- Auto-resolver actions → `RESOLVED` status for activity/stage/reassign events

**Impact**: 
- Analytics queries counting only `RESOLVED` status undercount total closed escalations
- Reporting inconsistencies across different resolution paths

**Recommendation**: Standardize resolution status taxonomy and update existing data accordingly.
