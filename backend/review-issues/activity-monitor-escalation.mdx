This document outlines critical issues identified during a comprehensive review of the Activity Monitor and Escalation modules located in `src/modules/activity-monitor/` and `src/modules/escalation/`.

<Warning>
These issues range from dead code and non-deterministic behavior to potential data consistency problems that could affect system reliability and user experience.
</Warning>

## Dead Code Issues

### `markAlertIgnored` Method Not Used

The `ActivityMonitorService.markAlertIgnored` method is defined but never called in the codebase. Instead, the follow-up path marks alerts as `IGNORED` using raw SQL inline.

<Note>
**Recommendation:** Either consolidate the codebase to use the existing method or remove it to reduce technical debt.
</Note>

```typescript
// Currently unused method
ActivityMonitorService.markAlertIgnored(alertId: string)

// Instead, raw SQL is used directly in follow-up paths
UPDATE activity_alerts SET status = 'IGNORED' WHERE id = ?
```

## Non-Deterministic Behavior

### Team Lead Resolution Issue

Both modules exhibit non-deterministic behavior when selecting team administrators:

<Warning>
Using `LIMIT 1` or `DISTINCT ON` without a deterministic `ORDER BY` clause means which admin receives alerts is arbitrary and could change between scans when multiple `TEAM_ADMIN` users exist.
</Warning>

**Impact:** Inconsistent alert routing that could confuse team members and affect accountability.

### Insert Idempotency Mismatch

## Performance & Scalability Concerns

### Scanner Batch Processing Issues

<AccordionGroup>
  <Accordion title="Different Batch Caps">
    - Activity monitor: 200 leads/rule/tick
    - Escalation: 500 leads/rule/tick
    
    This inconsistency makes capacity planning and performance tuning more difficult.
  </Accordion>
  
  <Accordion title="No Fairness Mechanism">
    Large backlogs drain slowly with no fairness mechanism across rules. Earlier rules in iteration order get priority, potentially starving later rules of processing time.
  </Accordion>
</AccordionGroup>

## State Management Problems

### Exhausted Escalation Tiers

<Info>
When all escalation tiers have fired, the tracker remains in an `ACTIVE` state with `next_escalation_at = NULL`.
</Info>

**Consequences:**
- Trackers sit indefinitely until an auto-resolution event fires
- Accumulation of stale trackers with no scheduled action
- No dashboard visibility into "stuck" state

### Stranded Activity Alerts

When `notifyTeamLead` is set to `false`:

<Steps>
  <Step title="Follow-up Path Executes">
    The system stamps `follow_up_sent_at` timestamp
  </Step>
  <Step title="Alert Never Transitions">
    Alert never transitions out of `PENDING`/`ESCALATED` status
  </Step>
  <Step title="Alert Remains Indefinitely">
    Alerts can linger forever with no terminal state
  </Step>
</Steps>

## Semantic Inconsistencies

### Status Resolution Semantics

<Tabs>
  <Tab title="Archived/Deleted Leads">
    - Tracker status: `CANCELLED`
    - Resolution type: `AUTO_ARCHIVED` or `AUTO_DELETED`
  </Tab>
  <Tab title="Auto-Resolver Cases">
    - Status: `RESOLVED`
    - Used for: activity/stage/reassign scenarios
  </Tab>
</Tabs>

<Warning>
Analytics or dashboards counting only `RESOLVED` status would undercount total closed escalations, leading to inaccurate reporting.
</Warning>
