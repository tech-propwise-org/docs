This document outlines critical issues discovered during a comprehensive code review of the activity monitor and escalation modules in `src/modules/activity-monitor/` and `src/modules/escalation/`.

<Warning>
These issues could impact system reliability, data consistency, and user experience. Prioritize fixes based on your production usage patterns.
</Warning>

## Dead Code Detection

### `markAlertIgnored` is unused

The `ActivityMonitorService.markAlertIgnored` method is defined but never called in the codebase. Instead, the follow-up path marks alerts as `IGNORED` using raw SQL inline.

<Tabs>
  <Tab title="Current Issue">
    ```typescript
    // Method exists but is never called
    async markAlertIgnored(alertId: string) {
      // Implementation exists
    }

    // Raw SQL used instead
    await db.query('UPDATE alerts SET status = $1 WHERE id = $2', ['IGNORED', alertId]);
    ```
  </Tab>
  <Tab title="Recommended Fix">
    ```typescript
    // Either use the existing method
    await activityMonitorService.markAlertIgnored(alertId);

    // Or remove the unused method entirely
    ```
  </Tab>
</Tabs>

**Impact**: Code maintenance burden and potential confusion for developers.

## Non-Deterministic Behavior

### Team lead resolution inconsistency

Both modules use `LIMIT 1` or `DISTINCT ON` without deterministic `ORDER BY` clauses when multiple `TEAM_ADMIN` users exist on a team.

<Note>
Which admin receives alerts is arbitrary and could change between database scans, leading to unpredictable notification patterns.
</Note>

```sql
-- Problematic query
SELECT user_id FROM team_members 
WHERE team_id = $1 AND role = 'TEAM_ADMIN' 
LIMIT 1;

-- Should be
SELECT user_id FROM team_members 
WHERE team_id = $1 AND role = 'TEAM_ADMIN' 
ORDER BY created_at ASC 
LIMIT 1;
```

## Concurrency Issues

### Insert idempotency mismatch in escalation

Escalation module comments reference `INSERT … ON CONFLICT DO NOTHING`, but the actual code uses `INSERT … WHERE NOT EXISTS`.

<Warning>
Under concurrent transactions, unique violations are still possible on partial unique indexes, though unlikely.
</Warning>

<CodeGroup>
```sql Comment Documentation
-- Comments suggest this approach
INSERT INTO escalation_comments (tracker_id, content) 
VALUES ($1, $2) 
ON CONFLICT DO NOTHING;
```

```sql Actual Implementation
-- Actual code uses this
INSERT INTO escalation_comments (tracker_id, content) 
SELECT $1, $2 
WHERE NOT EXISTS (
  SELECT 1 FROM escalation_comments 
  WHERE tracker_id = $1
);
```
</CodeGroup>

## Performance & Fairness Issues

### Scanner batch caps differ

<Info>
Activity monitor processes 200 leads per rule per tick, while escalation processes 500. This inconsistency affects system load balancing.
</Info>

**Current Limits:**
- Activity monitor: 200 leads/rule/tick
- Escalation: 500 items/batch

**Issues:**
- Large backlogs drain slowly
- No fairness mechanism across rules
- Earlier rules in iteration order get priority

### Recommended batch processing strategy

<Steps>
  <Step title="Standardize batch sizes">
    Align both modules to use the same configurable batch size
  </Step>
  <Step title="Implement round-robin processing">
    Ensure fair processing across all active rules
  </Step>
  <Step title="Add backlog monitoring">
    Track and alert on processing delays
  </Step>
</Steps>

## State Management Problems

### Exhausted escalation tiers create limbo state

When all escalation tiers have fired, trackers remain `ACTIVE` with `next_escalation_at = NULL`, sitting indefinitely until an auto-resolution event occurs.

```typescript
// Problematic state
{
  status: 'ACTIVE',
  next_escalation_at: null,
  // No scheduled action, no dashboard visibility
}
```

<Tip>
Consider adding a `EXHAUSTED` status or automatic cleanup for trackers with no future escalation schedule.
</Tip>

### Activity alerts can become stranded

When `notifyTeamLead` is false, the follow-up path stamps `follow_up_sent_at` but never transitions alerts out of `PENDING`/`ESCALATED` status.

<AccordionGroup>
  <Accordion title="Current Flow Issues">
    1. Alert created in `PENDING` status
    2. `notifyTeamLead = false` is processed
    3. `follow_up_sent_at` timestamp is set
    4. Alert status never changes
    5. Alert remains in limbo indefinitely
  </Accordion>
  <Accordion title="Recommended Resolution">
    1. Define terminal states for non-notification scenarios
    2. Implement automatic transitions after follow-up processing
    3. Add cleanup jobs for orphaned alerts
  </Accordion>
</AccordionGroup>

## Semantic Inconsistencies

### RESOLVED vs CANCELLED status divergence

The system uses different status values for similar resolution scenarios:

<CardGroup cols={2}>
  <Card title="Archived/Deleted Leads">
    - Status: `CANCELLED`
    - Reason: `AUTO_ARCHIVED` or `AUTO_DELETED`
  </Card>
  <Card title="Auto-Resolved Items">
    - Status: `RESOLVED`
    - Triggers: Activity/stage/reassignment changes
  </Card>
</CardGroup>

<Warning>
Analytics or dashboards counting only `RESOLVED` status would undercount total closed escalations.
</Warning>
