This document outlines technical debt and issues identified during a comprehensive code review of the Activity Monitor and Escalation modules located in `src/modules/activity-monitor/` and `src/modules/escalation/`.

<Warning>
These issues represent potential reliability, performance, and maintainability concerns that should be addressed in future development cycles.
</Warning>

## Dead Code Issues

### Unused Alert Ignore Method

The `ActivityMonitorService.markAlertIgnored` method is defined but never called anywhere in the codebase. Instead, the follow-up processing path marks alerts as `IGNORED` using raw SQL queries inline.

<Note>
**Recommendation**: Either consolidate alert ignore logic to use the existing method or remove the dead code to reduce maintenance burden.
</Note>

## Non-Deterministic Behavior

### Team Lead Resolution Randomness

Both Activity Monitor and Escalation modules exhibit non-deterministic behavior when selecting team administrators:

- Uses `LIMIT 1` or `DISTINCT ON` clauses without deterministic `ORDER BY`
- When multiple `TEAM_ADMIN` users exist on a team, which admin receives alerts is arbitrary
- Selection could change between different scan executions

<CodeGroup>
```sql Current Problematic Query
SELECT user_id FROM team_members 
WHERE team_id = ? AND role = 'TEAM_ADMIN' 
LIMIT 1;
```

```sql Recommended Fix
SELECT user_id FROM team_members 
WHERE team_id = ? AND role = 'TEAM_ADMIN' 
ORDER BY user_id ASC 
LIMIT 1;
```
</CodeGroup>

## Concurrency Issues

### Insert Idempotency Mismatch

The escalation module has inconsistent approaches to preventing duplicate records:

- Documentation references `INSERT … ON CONFLICT DO NOTHING`
- Actual implementation uses `INSERT … WHERE NOT EXISTS`
- Under concurrent transactions, unique violations on partial unique indexes remain possible

<Warning>
This mismatch could lead to unexpected database errors during high-concurrency scenarios, though the likelihood is low.
</Warning>

## Performance & Fairness Issues

### Inconsistent Batch Processing Limits

The two modules use different batch processing caps without clear justification:

<AccordionGroup>
<Accordion title="Activity Monitor Batching">
- Caps at **200 leads per rule per tick**
- Processes rules in iteration order
- No fairness mechanism across rules
</Accordion>

<Accordion title="Escalation Batching">
- Caps at **500 items per batch**
- Also processes in iteration order
- Earlier rules get priority over later ones
</Accordion>
</AccordionGroup>

<Tip>
Consider implementing a round-robin or weighted fairness algorithm to ensure all rules get processing time, especially during backlog conditions.
</Tip>

## State Management Issues

### Exhausted Escalation Tiers

When all escalation tiers have been triggered, trackers exhibit problematic behavior:

<Steps>
<Step title="All Tiers Fire">
Escalation tracker completes all configured escalation levels
</Step>

<Step title="Tracker Becomes Orphaned">
- Status remains `ACTIVE`
- `next_escalation_at` field set to `NULL`
- No scheduled actions remain
</Step>

<Step title="Indefinite Limbo">
Tracker sits indefinitely until an auto-resolution event fires
</Step>
</Steps>

This can lead to accumulation of stale trackers with no dashboard visibility into the "stuck" state.

### Stranded Activity Alerts

When the `notifyTeamLead` configuration is set to `false`:

1. Follow-up path stamps `follow_up_sent_at` timestamp
2. Alert status never transitions out of `PENDING` or `ESCALATED`
3. Alerts can remain in these states indefinitely with no terminal resolution

<Info>
This creates data inconsistency where alerts appear processed (timestamp exists) but remain in non-terminal states.
</Info>

## Status Semantics Inconsistency

### Resolution Status Divergence

The system uses different status values for similar resolution scenarios:

<CardGroup cols={2}>
<Card title="Archived/Deleted Leads" icon="archive">
- Tracker status: `CANCELLED`
- Reason: `AUTO_ARCHIVED` or `AUTO_DELETED`
</Card>

<Card title="Auto-Resolved Cases" icon="check">
- Tracker status: `RESOLVED`
- Used for activity/stage/reassignment resolutions
</Card>
</CardGroup>

<Warning>
Analytics queries or dashboards that count only `RESOLVED` status would undercount total closed escalations, potentially skewing metrics and reporting.
</Warning>

## Documentation Inconsistencies

### API Route Specification Mismatch

There's a discrepancy between documented and actual API endpoints:

<Tabs>
<Tab title="Specification Docs">
References paths like:
- `/activity-monitor/rules`
- `/activity-monitor/alerts`
</Tab>

<Tab title="Actual Routes">
Controller implements:
- `/activity-monitor-rules`
- `/activity-monitor-alerts`
</Tab>
</Tabs>

The actual implementation uses hyphenated, resource-style routing conventions rather than the nested paths shown in specifications.
