This page documents critical issues identified during a comprehensive review of the Activity Monitor and Escalation modules. These findings highlight potential bugs, performance issues, and inconsistencies that should be addressed.

## Overview

The review covered the following modules:
- `src/modules/activity-monitor/`
- `src/modules/escalation/`

<Warning>
Several issues involve potential data inconsistencies and non-deterministic behavior that could affect system reliability.
</Warning>

## Critical Issues

### Dead Code Detection

<Accordion title="markAlertIgnored is dead code">

**Issue**: `ActivityMonitorService.markAlertIgnored` is defined but never called anywhere in the codebase.

**Details**: The follow-up processing path marks alerts as `IGNORED` using raw SQL inline instead of utilizing this dedicated method.

**Recommendation**: Either consolidate the code to use the existing method or remove the unused `markAlertIgnored` method to reduce code bloat.

</Accordion>

### Non-Deterministic Behavior

<Accordion title="Team lead resolution is non-deterministic">

**Issue**: Both modules use `LIMIT 1` or `DISTINCT ON` without a deterministic `ORDER BY` clause when multiple `TEAM_ADMIN` users exist on a team.

**Impact**: Which admin receives alerts becomes arbitrary and could change between different scans, leading to inconsistent notification behavior.

**Example**:
```sql
SELECT user_id FROM team_members 
WHERE team_id = $1 AND role = 'TEAM_ADMIN' 
LIMIT 1  -- Missing ORDER BY clause
```

<Tip>
Add a deterministic ordering such as `ORDER BY user_id ASC` or `ORDER BY created_at ASC` to ensure consistent team lead selection.
</Tip>

</Accordion>

### Concurrency Issues

<Accordion title="Insert idempotency mismatch (escalation)">

**Issue**: Escalation comments reference `INSERT … ON CONFLICT DO NOTHING` in documentation, but the actual code uses `INSERT … WHERE NOT EXISTS`.

**Risk**: Under concurrent transactions, unique violations are still possible on the partial unique index, though unlikely.

**Current Implementation**:
```sql
INSERT INTO escalation_comments (...)
WHERE NOT EXISTS (SELECT 1 FROM escalation_comments WHERE ...)
```

**Safer Approach**:
```sql
INSERT INTO escalation_comments (...) 
ON CONFLICT DO NOTHING
```

</Accordion>

## Performance & Scalability Issues

### Batch Processing Inconsistencies

<Info>
Scanner batch limits vary between modules and lack fairness mechanisms for processing large backlogs.
</Info>

<CardGroup cols={2}>
  <Card title="Activity Monitor" icon="gauge">
    Caps at 200 leads per rule per tick
  </Card>
  <Card title="Escalation Module" icon="gauge">
    Caps at 500 leads per processing cycle
  </Card>
</CardGroup>

**Problem**: Large backlogs drain slowly with no fairness mechanism across rules. Earlier rules in iteration order receive priority, potentially starving later rules of processing time.

### State Management Issues

<Warning>
Multiple issues can leave records in indefinite limbo states without clear resolution paths.
</Warning>

<Steps>
  <Step title="Exhausted Escalation Tiers">
    When all escalation tiers have fired, trackers remain `ACTIVE` with `next_escalation_at = NULL`, sitting indefinitely until an auto-resolution event occurs.
  </Step>
  
  <Step title="Stranded Activity Alerts">
    When `notifyTeamLead = false`, the follow-up path stamps `follow_up_sent_at` but never transitions alerts out of `PENDING`/`ESCALATED` status.
  </Step>
  
  <Step title="Inconsistent Resolution States">
    Archived/deleted leads get `CANCELLED` + `AUTO_ARCHIVED`/`AUTO_DELETED`, while auto-resolver uses `RESOLVED` for other cases, creating semantic inconsistencies.
  </Step>
</Steps>

## API Documentation Misalignment

<Accordion title="Spec docs vs actual API routes">

**Documented Paths**:
```
/activity-monitor/rules
/activity-monitor/alerts
```

**Actual Controller Routes**:
```
/activity-monitor-rules
/activity-monitor-alerts
```

The API specification documents reference different path formats than what's actually implemented in the controllers. This discrepancy could confuse developers and cause integration issues.

</Accordion>

## Resolution Status Semantics

<Note>
The system uses different resolution statuses that may cause analytics discrepancies.
</Note>

| Trigger | Status | Reason |
|---------|--------|---------|
| Lead archived | `CANCELLED` | `AUTO_ARCHIVED` |
| Lead deleted | `CANCELLED` | `AUTO_DELETED` |
| Activity/stage/reassign | `RESOLVED` | Auto-resolver |

**Impact**: Analytics or dashboards counting only `RESOLVED` escalations would undercount the total number of closed escalations, missing those marked as `CANCELLED`.

## Recommendations

<Steps>
  <Step title="Cleanup Dead Code">
    Remove unused `markAlertIgnored` method or refactor code to use it consistently.
  </Step>
  
  <Step title="Fix Non-Deterministic Queries">
    Add explicit `ORDER BY` clauses to all queries using `LIMIT 1` or `DISTINCT ON`.
  </Step>
  
  <Step title="Implement Proper Idempotency">
    Use `ON CONFLICT DO NOTHING` for all insert operations that should be idempotent.
  </Step>
  
  <Step title="Add Fairness Mechanisms">
    Implement round-robin or weighted processing for handling large backlogs across multiple rules.
  </Step>
  
  <Step title="Handle Limbo States">
    Create cleanup jobs or status transitions for records stuck in indefinite states.
  </Step>
  
  <Step title="Standardize API Documentation">
    Align specification documents with actual implemented routes.
  </Step>
</Steps>
