This document outlines critical issues identified during a comprehensive code review of the activity monitor and escalation modules located in `src/modules/activity-monitor/` and `src/modules/escalation/`.

<Warning>
These issues range from dead code and non-deterministic behavior to potential data consistency problems and API documentation mismatches.
</Warning>

## Dead Code Issues

### `markAlertIgnored` is unused

The `ActivityMonitorService.markAlertIgnored` method is defined but never called anywhere in the codebase. The follow-up processing path marks alerts as `IGNORED` using raw SQL inline instead of utilizing this dedicated method.

<Note>
**Recommendation**: Either consolidate the codebase to use the existing method or remove the dead code to reduce maintenance overhead.
</Note>

## Non-Deterministic Behavior

### Team lead resolution lacks consistency

Both the activity monitor and escalation modules exhibit non-deterministic behavior when selecting team administrators:

- Use `LIMIT 1` or `DISTINCT ON` without deterministic `ORDER BY` clauses
- When multiple `TEAM_ADMIN` users exist on a team, alert assignment is arbitrary
- The selected admin can change between different scans

<Warning>
This unpredictable behavior can lead to inconsistent user experience and makes debugging difficult.
</Warning>

**Example problematic query pattern:**
```sql
SELECT user_id FROM team_members 
WHERE team_id = ? AND role = 'TEAM_ADMIN' 
LIMIT 1;
-- Missing ORDER BY clause makes selection non-deterministic
```

## Data Consistency Issues

### Insert idempotency mismatch in escalation module

The escalation module has inconsistent approaches to preventing duplicate inserts:

- Code comments reference `INSERT ... ON CONFLICT DO NOTHING`
- Actual implementation uses `INSERT ... WHERE NOT EXISTS`
- Under concurrent transactions, unique violations are still possible on partial unique indexes

<Tip>
Consider using proper `ON CONFLICT` clauses with appropriate conflict resolution strategies to handle concurrent scenarios reliably.
</Tip>

## Performance and Fairness Concerns

### Inconsistent batch processing limits

The two modules have different batch processing configurations:

- **Activity Monitor**: Caps at 200 leads per rule per tick
- **Escalation**: Caps at 500 items per batch

<Info>
Large backlogs drain slowly with no fairness mechanism across rules. Earlier rules in iteration order receive priority, potentially starving later rules of processing time.
</Info>

## State Management Problems

### Exhausted escalation tiers create orphaned trackers

When all escalation tiers have been triggered:

- Tracker remains in `ACTIVE` status
- `next_escalation_at` field is set to `NULL`
- Tracker sits indefinitely until an auto-resolution event occurs
- No dashboard visibility into "stuck" state

<Warning>
This can lead to accumulation of stale trackers with no scheduled actions and no clear resolution path.
</Warning>

### Stranded activity alerts with disabled notifications

When `notifyTeamLead` is configured as `false`:

- Follow-up processing stamps `follow_up_sent_at` timestamp
- Alert never transitions out of `PENDING` or `ESCALATED` status
- Alerts can remain in limbo indefinitely

**Problem flow:**
<Steps>
<Step title="Alert created">
Activity alert is generated and enters `PENDING` status
</Step>
<Step title="Follow-up processing">
System processes alert but `notifyTeamLead = false`
</Step>
<Step title="Timestamp updated">
`follow_up_sent_at` is set but status remains unchanged
</Step>
<Step title="Alert stranded">
No terminal state transition occurs, alert persists indefinitely
</Step>
</Steps>

## Status Semantics Inconsistency

### Divergent resolution status handling

The system uses inconsistent status values for different resolution scenarios:

<Tabs>
<Tab title="Archive/Delete Events">
- **Status**: `CANCELLED`
- **Reason**: `AUTO_ARCHIVED` or `AUTO_DELETED`
- Used for leads that are archived or deleted
</Tab>
<Tab title="Auto-Resolution Events">
- **Status**: `RESOLVED`
- Used for activity, stage, or reassignment events
</Tab>
</Tabs>

<Note>
Analytics or dashboards that count only `RESOLVED` status would undercount the total number of closed escalations, missing those marked as `CANCELLED`.
</Note>

## Documentation Mismatches

### API route specification inconsistency

There's a mismatch between specification documents and actual API implementation:

- **Specification docs**: Reference paths like `/activity-monitor/rules`
- **Actual routes**: Use `/activity-monitor-rules` and `/activity-monitor-alerts`

<Check>
**Action Required**: Update either the API routes to match specifications or update documentation to reflect actual implementation.
</Check>

## Recommended Actions

<AccordionGroup>
<Accordion title="High Priority Fixes">

- Implement proper terminal states for disabled notifications
- Standardize resolution status semantics across modules
</Accordion>

<Accordion title="Medium Priority Improvements">
- Implement fairness mechanisms for batch processing
- Add monitoring for "stuck" escalation trackers
- Resolve insert idempotency implementation inconsistencies
- Align API documentation with actual routes
</Accordion>

<Accordion title="Performance Considerations">
- Standardize batch processing limits across modules
- Implement proper conflict resolution for concurrent operations
- Add metrics for tracking processing fairness
</Accordion>
</AccordionGroup>
