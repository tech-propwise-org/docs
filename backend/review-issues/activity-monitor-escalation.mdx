Critical issues identified during deep review of `src/modules/activity-monitor/` and `src/modules/escalation/` modules that require attention for production stability and data consistency.

<Warning>
These issues range from dead code to potential data inconsistencies and stuck states that could impact system reliability.
</Warning>

## Code Quality Issues

### Dead Code: `markAlertIgnored` Method

The `ActivityMonitorService.markAlertIgnored` method is defined but never called anywhere in the codebase.

<Info>
The follow-up path marks alerts as `IGNORED` using raw SQL inline instead of utilizing this service method.
</Info>

**Recommendation**: Either consolidate the codebase to use the existing method or remove the dead code entirely.

### API Documentation Mismatch

There's a discrepancy between specification documentation and actual implemented routes.

<Tabs>
  <Tab title="Documented Paths">
    ```
    /activity-monitor/rules
    /activity-monitor/alerts
    ```
  </Tab>
  <Tab title="Actual Routes">
    ```
    /activity-monitor-rules
    /activity-monitor-alerts
    ```
  </Tab>
</Tabs>

## Data Consistency Problems

### Non-Deterministic Team Lead Resolution

<Warning>
Both modules use `LIMIT 1` or `DISTINCT ON` without deterministic `ORDER BY` clauses when multiple `TEAM_ADMIN` users exist on a team.
</Warning>

This creates arbitrary alert assignment that could change between scans, leading to:
- Inconsistent notification delivery
- Unpredictable escalation paths
- Difficulty in troubleshooting alert routing

### Insert Idempotency Mismatch

<Note>
Escalation comments reference `INSERT … ON CONFLICT DO NOTHING` in documentation, but the actual code uses `INSERT … WHERE NOT EXISTS`.
</Note>

Under concurrent transactions, unique violations are still possible on the partial unique index, though unlikely. This edge case remains unhandled.

## Performance and Fairness Issues

### Inconsistent Scanner Batch Caps

The two modules have different processing limits without fairness mechanisms:

<CardGroup cols={2}>
  <Card title="Activity Monitor" icon="activity">
    200 leads per rule per tick
  </Card>
  <Card title="Escalation" icon="triangle-exclamation">
    500 leads per rule per tick
  </Card>
</CardGroup>

**Impact**: Large backlogs drain slowly with no fairness mechanism across rules. Earlier rules in iteration order receive priority processing.

## State Management Issues

### Exhausted Escalation Tiers

<Warning>
When all escalation tiers have fired, trackers remain `ACTIVE` with `next_escalation_at = NULL`, sitting indefinitely until an auto-resolution event occurs.
</Warning>

This can lead to:
- Accumulation of stale trackers
- No scheduled actions for stuck states  
- No dashboard visibility into "stuck" escalations

### Stranded Activity Alerts

When `notifyTeamLead` is set to `false`:

<Steps>
  <Step title="Follow-up Process Starts">
    The follow-up path stamps `follow_up_sent_at` timestamp
  </Step>
  <Step title="Status Remains Unchanged">
    Alert never transitions out of `PENDING` or `ESCALATED` status
  </Step>
  <Step title="Alert Gets Stranded">
    Alerts can linger forever with no terminal state
  </Step>
</Steps>

### Inconsistent Resolution Semantics

<Tabs>
  <Tab title="Archive/Delete Events">
    ```
    Status: CANCELLED
    Reason: AUTO_ARCHIVED | AUTO_DELETED
    ```
  </Tab>
  <Tab title="Auto-Resolution Events">
    ```
    Status: RESOLVED
    Reason: Activity/Stage/Reassign triggers
    ```
  </Tab>
</Tabs>

<Tip>
Analytics or dashboards counting only `RESOLVED` status would undercount total closed escalations, missing `CANCELLED` cases.
</Tip>

## Recommendations

<AccordionGroup>
  <Accordion title="Immediate Actions">
    - Remove or utilize the `markAlertIgnored` dead code
    - Add deterministic ordering to team lead queries
    - Standardize resolution status semantics across modules
  </Accordion>
  
  <Accordion title="Medium Priority">
    - Implement fairness mechanism for scanner batch processing
    - Add timeout handling for exhausted escalation tiers
    - Fix stranded alert state transitions when `notifyTeamLead = false`
  </Accordion>
  
  <Accordion title="Documentation">
    - Update API documentation to match actual route implementations
    - Clarify resolution vs cancellation semantics in system documentation
  </Accordion>
</AccordionGroup>
