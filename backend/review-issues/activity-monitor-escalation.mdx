This document outlines critical issues discovered during a comprehensive review of the Activity Monitor and Escalation modules located in `src/modules/activity-monitor/` and `src/modules/escalation/`.

<Warning>
These issues range from dead code and non-deterministic behavior to potential data inconsistencies and system reliability problems that should be addressed in order of priority.
</Warning>

## Code Quality Issues

### Dead Code in Activity Monitor

The `ActivityMonitorService.markAlertIgnored` method is defined but never used in the codebase. Instead, alerts are marked as `IGNORED` using raw SQL queries inline.

<Note>
**Resolution Options:**
- Consolidate all alert status updates to use the service method
- Remove the unused method if inline SQL is preferred
</Note>

### API Documentation Mismatch

There's a discrepancy between specification documentation and actual implementation:

- **Docs reference:** `/activity-monitor/rules`
- **Actual routes:** `/activity-monitor-rules` and `/activity-monitor-alerts`

The actual controller uses hyphenated resource-style naming conventions.

## Data Consistency & Reliability Issues

### Non-Deterministic Team Lead Resolution

<Warning>
Both modules use `LIMIT 1` or `DISTINCT ON` without deterministic `ORDER BY` clauses when multiple `TEAM_ADMIN` users exist on a team.
</Warning>

**Impact:** Which admin receives alerts is arbitrary and could change between scans, leading to:
- Inconsistent alert routing
- Unpredictable user experience
- Potential missed notifications

**Recommended Fix:**
```sql
-- Add deterministic ordering
SELECT * FROM users 
WHERE team_id = ? AND role = 'TEAM_ADMIN' 
ORDER BY created_at ASC, id ASC 
LIMIT 1
```

### Insert Idempotency Mismatch

<Warning>
Escalation module comments reference `INSERT … ON CONFLICT DO NOTHING` but the actual code uses `INSERT … WHERE NOT EXISTS`.
</Warning>

Under concurrent transactions, unique violations are still possible on partial unique indexes, though unlikely. This edge case remains unhandled.

### Status Transition Issues

<AccordionGroup>
<Accordion title="Escalation Trackers in Limbo">
When all escalation tiers have fired, trackers remain in `ACTIVE` status with `next_escalation_at = NULL`. These trackers:
- Stay indefinitely until auto-resolution events fire
- Accumulate as "stuck" trackers with no scheduled actions
- Have no dashboard visibility for monitoring
</Accordion>

<Accordion title="Stranded Activity Alerts">
When `notifyTeamLead = false`, the follow-up path:
- Stamps `follow_up_sent_at` timestamp
- Never transitions alerts out of `PENDING`/`ESCALATED` status
- Leaves alerts lingering forever with no terminal state
</Accordion>

<Accordion title="Inconsistent Resolution Semantics">
Different resolution paths use different status codes:
- **Archived/deleted leads:** `CANCELLED` + `AUTO_ARCHIVED`/`AUTO_DELETED`
- **Auto-resolver:** `RESOLVED` for activity/stage/reassign events

This divergence affects analytics and dashboard counts, as systems counting only `RESOLVED` status would undercount total closed escalations.
</Accordion>
</AccordionGroup>

## Performance & Scalability Issues

### Inconsistent Batch Processing

<Info>
Scanner batch limits differ between modules without clear justification:
</Info>

| Module | Batch Size | Processing Pattern |
|--------|------------|-------------------|
| Activity Monitor | 200 leads/rule/tick | No fairness mechanism |
| Escalation | 500 leads/rule/tick | Rules processed in iteration order |

**Issues:**
- Large backlogs drain slowly
- Earlier rules in iteration order get priority
- No fairness mechanism across rules
- Potential for resource starvation

## Recommended Actions

<Steps>
<Step title="Immediate Priority">
Fix non-deterministic team lead resolution to ensure consistent alert routing
</Step>

<Step title="High Priority">
Address status transition issues to prevent alerts and trackers from getting stuck in limbo states
</Step>

<Step title="Medium Priority">
Standardize batch processing limits and implement fairness mechanisms
</Step>

<Step title="Low Priority">
Clean up dead code and align API documentation with actual implementation
</Step>
</Steps>

<Tip>
Consider implementing monitoring dashboards to track alerts and escalation trackers in various states, especially those that may be "stuck" without clear resolution paths.
</Tip>
