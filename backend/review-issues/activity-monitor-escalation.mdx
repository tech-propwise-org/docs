This document outlines critical issues identified during a comprehensive review of the activity monitor (`src/modules/activity-monitor/`) and escalation (`src/modules/escalation/`) modules.

<Warning>
These issues range from dead code and non-deterministic behavior to potential data inconsistencies and infinite loops. Addressing these findings is recommended before production deployment.
</Warning>

## Dead Code Issues

### markAlertIgnored Method Never Used

**Module:** Activity Monitor  
**Impact:** Code maintenance and clarity

The `ActivityMonitorService.markAlertIgnored` method is defined but never called anywhere in the codebase. Instead, the follow-up processing path marks alerts as `IGNORED` using raw SQL queries inline.

<Note>
**Recommendation:** Either consolidate the codebase to use the existing method or remove it to reduce technical debt.
</Note>

## Non-Deterministic Behavior

### Team Lead Resolution Inconsistency

**Modules:** Activity Monitor, Escalation  
**Impact:** Alert routing reliability

Both modules use `LIMIT 1` or `DISTINCT ON` clauses without deterministic `ORDER BY` statements when multiple `TEAM_ADMIN` users exist on a team.

```sql
-- Current problematic pattern
SELECT user_id FROM team_members 
WHERE team_id = ? AND role = 'TEAM_ADMIN' 
LIMIT 1;
```

**Consequences:**
- Alert assignment to team leads is arbitrary
- Assignment could change between scanner runs
- No consistent escalation path

<Tip>
Add a deterministic ordering criterion such as `ORDER BY user_id ASC` or `ORDER BY created_at ASC` to ensure consistent team lead selection.
</Tip>

## Data Consistency Issues

### Insert Idempotency Mismatch

**Module:** Escalation  
**Impact:** Potential unique constraint violations

The escalation module has inconsistent idempotency handling:

- **Documentation:** References `INSERT … ON CONFLICT DO NOTHING`
- **Implementation:** Uses `INSERT … WHERE NOT EXISTS`

Under concurrent transactions, unique violations are still possible on partial unique indexes, though unlikely.

<Warning>
This mismatch could lead to unhandled database exceptions in high-concurrency scenarios.
</Warning>

## Performance and Fairness Issues

### Inconsistent Scanner Batch Limits

**Modules:** Activity Monitor, Escalation  
**Impact:** Processing fairness and performance

Different batch size caps are applied:
- **Activity Monitor:** 200 leads per rule per tick
- **Escalation:** 500 leads per rule per tick

**Problems:**
- Large backlogs drain slowly
- No fairness mechanism across rules
- Earlier rules in iteration order get processing priority

### Processing Fairness Concerns

The current implementation processes rules in iteration order without round-robin or fairness algorithms, potentially causing some rules to be starved of processing time.

## State Management Issues

### Exhausted Escalation Tiers

**Module:** Escalation  
**Impact:** Tracker lifecycle management

When all escalation tiers have been fired, trackers remain in an undefined state:

<Steps>
<Step title="Tier Exhaustion">
All configured escalation tiers have been processed and fired
</Step>

<Step title="Limbo State">
Tracker stays `ACTIVE` with `next_escalation_at = NULL`
</Step>

<Step title="Indefinite Wait">
Tracker sits indefinitely until an auto-resolution event fires
</Step>
</Steps>

**Consequences:**
- Accumulation of stale trackers with no scheduled actions
- No dashboard visibility into "stuck" state
- Potential memory and storage bloat

### Stranded Activity Alerts

**Module:** Activity Monitor  
**Impact:** Alert lifecycle completion

When `notifyTeamLead` is set to `false`, alerts can become stranded:

1. Follow-up path stamps `follow_up_sent_at` timestamp
2. Alert never transitions out of `PENDING`/`ESCALATED` status
3. Alerts linger indefinitely with no terminal state

<Info>
This creates orphaned alerts that may never reach resolution, affecting reporting accuracy and system cleanup.
</Info>

## Semantic Inconsistencies

### Resolution Status Divergence

**Modules:** Activity Monitor, Escalation  
**Impact:** Analytics and reporting accuracy

Different resolution semantics are used for similar outcomes:

<Tabs>
<Tab title="Archived/Deleted Leads">
- Tracker status: `CANCELLED`
- Resolution reason: `AUTO_ARCHIVED` or `AUTO_DELETED`
</Tab>

<Tab title="Auto-Resolved Cases">
- Tracker status: `RESOLVED`
- Resolution reason: Activity/stage/reassign triggers
</Tab>
</Tabs>

**Impact:** Analytics or dashboards counting only `RESOLVED` statuses would undercount total closed escalations.

## API Documentation Misalignment

### Route Path Discrepancies

**Module:** Activity Monitor  
**Impact:** API documentation accuracy

Specification documentation and actual implementation differ:

<CodeGroup>
```text Documented Paths
/activity-monitor/rules
/activity-monitor/alerts
```

```text Actual Controller Routes
/activity-monitor-rules
/activity-monitor-alerts
```
</CodeGroup>

The actual routes use hyphenated resource-style naming instead of the documented nested path structure.

<Check>
**Action Required:** Update API documentation to reflect actual route implementations or modify routes to match documentation.
</Check>

## Recommendations Summary

<AccordionGroup>
<Accordion title="Immediate Actions">
- Remove unused `markAlertIgnored` method or implement its usage
- Add deterministic ordering to team lead queries
- Align API documentation with actual routes
</Accordion>

<Accordion title="Critical Fixes">
- Implement proper state transitions for exhausted escalation tiers
- Fix stranded alert handling when `notifyTeamLead = false`
- Standardize resolution status semantics across modules
</Accordion>

<Accordion title="Performance Improvements">
- Implement fairness algorithms for rule processing
- Consider batch size standardization
- Add monitoring for stuck/stale trackers
</Accordion>

<Accordion title="Data Consistency">
- Resolve insert idempotency implementation discrepancies
- Add proper error handling for concurrent transaction scenarios
</Accordion>
</AccordionGroup>
