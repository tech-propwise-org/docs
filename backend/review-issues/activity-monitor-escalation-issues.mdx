Issues identified during deep review of `src/modules/activity-monitor/` and `src/modules/escalation/` modules.

<Warning>
These issues may affect system reliability, data consistency, and user experience. Review and address based on priority and impact.
</Warning>

## Dead Code & Unused Methods

### `markAlertIgnored` is dead code

The `ActivityMonitorService.markAlertIgnored` method is defined but never called in the codebase. Instead, the follow-up processing marks alerts as `IGNORED` using raw SQL inline.

<Info>
**Recommendation**: Either consolidate to use the existing method or remove it to reduce code complexity.
</Info>

## Non-Deterministic Behavior

### Team lead resolution lacks determinism

Both modules use `LIMIT 1` or `DISTINCT ON` without a deterministic `ORDER BY` clause when multiple `TEAM_ADMIN` users exist on a team.

**Impact**: Which admin receives alerts is arbitrary and could change between scans, leading to inconsistent alert routing.

<CodeGroup>
```sql Current Implementation
SELECT user_id FROM team_users 
WHERE role = 'TEAM_ADMIN' 
LIMIT 1
```

```sql Recommended Fix
SELECT user_id FROM team_users 
WHERE role = 'TEAM_ADMIN' 
ORDER BY created_at ASC, user_id ASC
LIMIT 1
```
</CodeGroup>

## Data Consistency Issues

### Insert idempotency mismatch in escalation

Escalation comments reference `INSERT … ON CONFLICT DO NOTHING`, but the actual code uses `INSERT … WHERE NOT EXISTS`.

<Warning>
Under concurrent transactions, a unique violation is still possible on the partial unique index, though unlikely. This scenario is currently unhandled.
</Warning>

### RESOLVED vs CANCELLED semantics diverge

The system uses inconsistent status semantics:
- Archived/deleted leads: `CANCELLED` status with `AUTO_ARCHIVED`/`AUTO_DELETED` reasons
- Auto-resolved cases: `RESOLVED` status for activity/stage/reassign events

<Note>
Analytics or dashboards counting only `RESOLVED` status would undercount total closed escalations.
</Note>

## Performance & Fairness Concerns

### Scanner batch caps differ without fairness mechanism

<Tabs>
<Tab title="Activity Monitor">
- Caps at **200 leads/rule/tick**
- No fairness across rules
</Tab>
<Tab title="Escalation">
- Caps at **500 leads/rule/tick** 
- Earlier rules in iteration order get priority
</Tab>
</Tabs>

**Issue**: Large backlogs drain slowly with no fairness mechanism across rules.

## State Management Problems

### Exhausted escalation tiers leave trackers in limbo

When all escalation tiers have fired, trackers remain in `ACTIVE` state with `next_escalation_at = NULL`.

<Steps>
<Step title="All tiers fire">
Escalation system processes all configured tiers for a tracker
</Step>
<Step title="Tracker becomes stuck">
Tracker stays `ACTIVE` but has no scheduled next action
</Step>
<Step title="Indefinite wait">
Tracker sits indefinitely until an auto-resolution event fires
</Step>
</Steps>

**Result**: Accumulation of stale trackers with no dashboard visibility into "stuck" state.

### Activity alerts can be stranded

When `notifyTeamLead = false`, the follow-up path stamps `follow_up_sent_at` but never transitions the alert out of `PENDING`/`ESCALATED` status.

<Warning>
Alerts can linger forever with no terminal state, affecting system performance and user experience.
</Warning>
