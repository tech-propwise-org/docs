## Dead Code and Unused Methods

### markAlertIgnored Method

The `ActivityMonitorService.markAlertIgnored` method is defined but never utilized in the codebase. Instead, the follow-up processing path marks alerts as `IGNORED` using raw SQL queries inline.

<Note>
**Recommendation**: Either consolidate alert status updates to use the existing method or remove the unused `markAlertIgnored` method to reduce code maintenance overhead.
</Note>

```sql
-- Current inline approach
UPDATE activity_alerts SET status = 'IGNORED' WHERE id = ?

-- vs using the defined method
ActivityMonitorService.markAlertIgnored(alertId)
```

## Non-Deterministic Behavior

### Team Lead Resolution

Both activity monitor and escalation modules exhibit non-deterministic behavior when selecting team administrators for alert assignment.

<Warning>
The current implementation uses `LIMIT 1` or `DISTINCT ON` clauses without deterministic `ORDER BY` statements when multiple `TEAM_ADMIN` users exist on a team.
</Warning>

**Impact**: 
- Alert assignment becomes arbitrary
- Results may vary between different scan executions
- No consistent escalation path for team notifications

**Affected Queries**:
```sql
-- Non-deterministic team admin selection
SELECT user_id FROM team_members 
WHERE team_id = ? AND role = 'TEAM_ADMIN' 
LIMIT 1;
```

<Tip>
Add deterministic ordering criteria such as user creation date, user ID, or last login time to ensure consistent team lead selection.
</Tip>

## Concurrency and Race Conditions

### Insert Idempotency Mismatch

The escalation module documentation references `INSERT ... ON CONFLICT DO NOTHING` for handling duplicate escalation comments, but the actual implementation uses `INSERT ... WHERE NOT EXISTS`.

<Warning>
Under concurrent transaction scenarios, unique constraint violations are still possible on partial unique indexes, creating unhandled exception conditions.
</Warning>

```sql
-- Documented approach
INSERT INTO escalation_comments (...) 
VALUES (...) ON CONFLICT DO NOTHING;

-- Actual implementation
INSERT INTO escalation_comments (...) 
SELECT ... WHERE NOT EXISTS (...);
```

## State Management Problems

### Exhausted Escalation Tiers

When all escalation tiers have been fired for a tracker, the system exhibits problematic behavior:

<Steps>
<Step title="Final Tier Execution">
The last escalation tier completes successfully
</Step>
<Step title="Limbo State">
Tracker remains `ACTIVE` with `next_escalation_at = NULL`
</Step>
<Step title="Indefinite Wait">
Tracker sits indefinitely until an auto-resolution event triggers
</Step>
</Steps>

**Consequences**:
- Accumulation of stale trackers with no scheduled actions
- No dashboard visibility into "stuck" tracker states
- Potential memory and performance degradation over time

### Stranded Activity Alerts

When `notifyTeamLead` is configured as `false`, the follow-up processing path exhibits incomplete state transitions:

```javascript
// Follow-up processing stamps timestamp
alert.follow_up_sent_at = new Date();

// But never transitions alert status
// Alert remains in PENDING/ESCALATED state indefinitely
```

<Warning>
Alerts can remain stranded in non-terminal states forever when team lead notifications are disabled.
</Warning>

## Semantic Inconsistencies

### Resolution Status Divergence

The system uses different resolution semantics across different scenarios:

<AccordionGroup>
<Accordion title="Archived/Deleted Leads">
- Tracker status: `CANCELLED`
- Resolution reason: `AUTO_ARCHIVED` or `AUTO_DELETED`
</Accordion>
<Accordion title="Auto-Resolution Events">
- Tracker status: `RESOLVED`
- Resolution reason: Activity/stage/reassign triggers
</Accordion>
</AccordionGroup>

**Impact**: Analytics and dashboards counting only `RESOLVED` status will undercount total closed escalations, leading to inaccurate reporting and metrics.
