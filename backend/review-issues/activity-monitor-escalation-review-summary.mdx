This document outlines critical issues identified during a comprehensive code review of the activity monitor and escalation modules in `src/modules/activity-monitor/` and `src/modules/escalation/`.

<Warning>
These issues may impact system reliability, data consistency, and user experience. Priority should be given to addressing deterministic ordering and state management problems.
</Warning>

## Dead Code Issues

### markAlertIgnored Method Unused

The `ActivityMonitorService.markAlertIgnored` method is defined but never called in the codebase. Instead, the follow-up path marks alerts as `IGNORED` using raw SQL inline.

<Note>
**Recommendation**: Either consolidate to use the method consistently or remove the dead code to reduce maintenance overhead.
</Note>

## Non-Deterministic Behavior

### Team Lead Resolution Randomness

Both modules use `LIMIT 1` or `DISTINCT ON` queries without deterministic `ORDER BY` clauses when multiple `TEAM_ADMIN` users exist on a team.

**Impact**: Which admin receives alerts is arbitrary and could change between scans, leading to inconsistent notification patterns.

<CodeGroup>
```sql Problem Example
SELECT user_id FROM team_users 
WHERE role = 'TEAM_ADMIN' 
LIMIT 1; -- Non-deterministic!
```

```sql Recommended Fix
SELECT user_id FROM team_users 
WHERE role = 'TEAM_ADMIN' 
ORDER BY created_at ASC, user_id ASC 
LIMIT 1; -- Deterministic
```
</CodeGroup>

## Data Consistency Issues

### Escalation Insert Idempotency Mismatch

Escalation comments reference `INSERT … ON CONFLICT DO NOTHING` pattern, but the actual code uses `INSERT … WHERE NOT EXISTS`.

<Warning>
Under concurrent transactions, a unique violation is still possible on the partial unique index. While unlikely, this scenario is currently unhandled and could cause application errors.
</Warning>

### Inconsistent Status Semantics

The system uses different status values for similar resolution scenarios:

<Tabs>
<Tab title="Archive/Delete Events">
- Status: `CANCELLED`
- Reason: `AUTO_ARCHIVED` or `AUTO_DELETED`
</Tab>
<Tab title="Auto-Resolution Events">
- Status: `RESOLVED`
- Reason: Activity/stage/reassign triggers
</Tab>
</Tabs>

<Info>
This divergence may cause analytics or dashboards that count only `RESOLVED` status to undercount total closed escalations.
</Info>

## Performance & Scalability Concerns

### Scanner Batch Size Inconsistencies

Different batch caps exist between modules without fairness mechanisms:

<CardGroup cols={2}>
<Card title="Activity Monitor" icon="chart-line">
200 leads per rule per tick
</Card>
<Card title="Escalation Module" icon="exclamation-triangle">
500 leads per rule per tick
</Card>
</CardGroup>

**Problem**: Large backlogs drain slowly with no fairness mechanism across rules. Earlier rules in iteration order receive priority processing.

## State Management Issues

### Exhausted Escalation Tiers

<Steps>
<Step title="All escalation tiers fire">
Tracker processes through all configured escalation levels
</Step>
<Step title="Tracker enters limbo">
Status remains `ACTIVE` with `next_escalation_at = NULL`
</Step>
<Step title="Indefinite waiting">
Tracker sits indefinitely until an auto-resolution event fires
</Step>
</Steps>

<Warning>
This could accumulate stale trackers with no scheduled action and no dashboard visibility into the "stuck" state.
</Warning>

### Stranded Activity Alerts

When `notifyTeamLead` is configured as `false`, the follow-up path stamps `follow_up_sent_at` but never transitions the alert out of `PENDING` or `ESCALATED` status.

<Note>
**Result**: Alerts can linger forever with no terminal state, creating data pollution and potential confusion for users.
</Note>

## Documentation Inconsistencies

### API Route Specification Mismatch

There's a discrepancy between documented and actual API routes:

<AccordionGroup>
<Accordion title="Specification Documentation">
References paths like:
- `/activity-monitor/rules`
- `/activity-monitor/alerts`
</Accordion>
<Accordion title="Actual Controller Routes">
Implements hyphenated resource-style paths:
- `/activity-monitor-rules`
- `/activity-monitor-alerts`
</Accordion>
</AccordionGroup>

<Tip>
Consider updating either the documentation or the actual routes to maintain consistency and avoid integration confusion.
</Tip>
