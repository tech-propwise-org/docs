This document outlines specific issues identified during a deep review of the Activity Monitor and Escalation modules located in `src/modules/activity-monitor/` and `src/modules/escalation/`.

## Dead Code Issues

### `markAlertIgnored` is dead code

<Warning>
The `ActivityMonitorService.markAlertIgnored` method is defined but never called anywhere in the codebase.
</Warning>

The follow-up path marks alerts as `IGNORED` via raw SQL inline instead of using this dedicated method. This creates inconsistency in the codebase.

**Recommended action:** Either consolidate to use the method consistently or remove it entirely to reduce code complexity.

## Non-Deterministic Behavior

### Team lead resolution is non-deterministic

<Warning>
Both modules use `LIMIT 1` or `DISTINCT ON` without a deterministic `ORDER BY` when multiple `TEAM_ADMIN` users exist on a team.
</Warning>

This means which admin receives alerts is arbitrary and could change between scans, leading to inconsistent alert routing.

**Impact:** 
- Unpredictable alert assignment
- Potential confusion among team administrators
- Inconsistent alert handling patterns

## Data Integrity Issues

### Insert idempotency mismatch (escalation)

<Note>
Escalation comments reference `INSERT … ON CONFLICT DO NOTHING`, but the actual code uses `INSERT … WHERE NOT EXISTS`.
</Note>

Under concurrent transactions, a unique violation is still possible on the partial unique index. While unlikely, this scenario is currently unhandled and could cause runtime errors.

## Performance and Fairness Issues

### Scanner batch caps differ and lack fairness

<Info>
Current batch processing limits:
- Activity monitor: 200 leads/rule/tick
- Escalation: 500 leads/rule/tick
</Info>

**Problems:**
- Large backlogs drain slowly
- No fairness mechanism across rules
- Earlier rules in iteration order get priority
- Inconsistent processing capacity between modules

## State Management Issues

### Exhausted escalation tiers leave trackers in limbo

<Warning>
When all escalation tiers have fired, the tracker stays `ACTIVE` with `next_escalation_at = NULL`.
</Warning>

**Consequences:**
- Trackers sit indefinitely until an auto-resolution event fires
- Accumulation of stale trackers with no scheduled action
- No dashboard visibility into "stuck" state
- Potential memory and performance impact over time

### `notifyTeamLead = false` leaves activity alerts stranded

When `notifyTeamLead` is configured as `false`:

<Steps>
<Step title="Follow-up path executes">
The system stamps `follow_up_sent_at` on the alert
</Step>

<Step title="Alert remains in pending state">
The alert never transitions out of `PENDING`/`ESCALATED` status
</Step>

<Step title="Alert becomes stranded">
Alerts can linger forever with no terminal state
</Step>
</Steps>

## Semantic Inconsistencies

### RESOLVED vs CANCELLED semantics diverge

<Info>
Different resolution paths use different status values:
- **Archived/deleted leads:** `CANCELLED` status with `AUTO_ARCHIVED`/`AUTO_DELETED`
- **Auto-resolver:** `RESOLVED` status for activity/stage/reassign events
</Info>

**Impact:** Analytics or dashboards counting only `RESOLVED` status would undercount total closed escalations, leading to inaccurate reporting.

## Documentation Misalignment

### Spec docs vs actual API routes

<Warning>
Documentation and implementation are out of sync for API endpoints.
</Warning>

<Tabs>
<Tab title="Documented Routes">
- `/activity-monitor/rules`
- `/activity-monitor/alerts`
</Tab>

<Tab title="Actual Routes">
- `/activity-monitor-rules`
- `/activity-monitor-alerts`
</Tab>
</Tabs>

The actual controller uses hyphenated resource-style routing, which differs from the specification documentation.

- Align insert idempotency patterns

</Accordion>
