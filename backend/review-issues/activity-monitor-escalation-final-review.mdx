This document outlines critical issues identified during the comprehensive review of the `src/modules/activity-monitor/` and `src/modules/escalation/` modules. These issues range from dead code and non-deterministic behavior to architectural concerns that could impact system reliability.

<Warning>
These issues should be prioritized for resolution as they affect core monitoring and escalation functionality.
</Warning>

## Dead Code Issues

### 1. `markAlertIgnored` is dead code

The `ActivityMonitorService.markAlertIgnored` method is defined but never called anywhere in the codebase. Instead, the follow-up path marks alerts as `IGNORED` via raw SQL inline.

<Note>
**Resolution Options:**
- Consolidate to use the existing method consistently
- Remove the unused method to reduce code complexity
</Note>

```sql
-- Current inline approach
UPDATE activity_alerts SET status = 'IGNORED' WHERE ...
```

## Non-Deterministic Behavior

### 2. Team lead resolution is non-deterministic

Both modules use `LIMIT 1` or `DISTINCT ON` without a deterministic `ORDER BY` clause when multiple `TEAM_ADMIN` users exist on a team.

<Warning>
Which admin receives alerts is arbitrary and could change between database scans, leading to inconsistent escalation patterns.
</Warning>

**Impact:** Alert routing becomes unpredictable when teams have multiple administrators.

**Recommendation:** Add deterministic ordering (e.g., by user ID, creation date, or last activity).

## Concurrency Issues

### 3. Insert idempotency mismatch (escalation)

Escalation comments reference `INSERT … ON CONFLICT DO NOTHING` in documentation, but the actual code uses `INSERT … WHERE NOT EXISTS`.

<Info>
Under concurrent transactions, a unique violation is still possible on the partial unique index, though unlikely.
</Info>

```sql
-- Documented approach
INSERT ... ON CONFLICT DO NOTHING

-- Actual implementation  
INSERT ... WHERE NOT EXISTS (...)
```

## Performance & Fairness Issues

### 4. Scanner batch caps differ and lack fairness

The scanning mechanisms have different batch size limits without fairness considerations:

<Tabs>
  <Tab title="Activity Monitor">
    - Caps at **200 leads/rule/tick**
  </Tab>
  <Tab title="Escalation">
    - Caps at **500 leads/rule/tick**  
  </Tab>
</Tabs>

<Warning>
Large backlogs drain slowly with no fairness mechanism across rules. Earlier rules in iteration order get priority, potentially starving later rules.
</Warning>

## State Management Issues

### 5. Exhausted escalation tiers leave trackers in limbo

When all escalation tiers have fired, the tracker remains `ACTIVE` with `next_escalation_at = NULL`.

**Problem:** Trackers sit indefinitely until an auto-resolution event fires, potentially accumulating stale trackers with:
- No scheduled action
- No dashboard visibility into "stuck" state

<Steps>
  <Step title="Current behavior">
    All escalation tiers fire successfully
  </Step>
  <Step title="Result">
    Tracker status remains `ACTIVE` with `next_escalation_at = NULL`
  </Step>
  <Step title="Issue">
    Tracker sits indefinitely with no further action scheduled
  </Step>
</Steps>

### 6. `notifyTeamLead = false` leaves activity alerts stranded

When the `notifyTeamLead` configuration is set to `false`, the follow-up path:
1. Stamps `follow_up_sent_at` timestamp
2. Never transitions the alert out of `PENDING`/`ESCALATED` status

<Warning>
Alerts can linger forever with no terminal state, creating data pollution and potential memory leaks.
</Warning>

## Semantic Inconsistencies

### 7. RESOLVED vs CANCELLED semantics diverge

Different resolution paths use inconsistent status terminology:

<AccordionGroup>
  <Accordion title="Archived/Deleted Leads">
    - Tracker status: `CANCELLED`
    - Resolution reason: `AUTO_ARCHIVED`/`AUTO_DELETED`
  </Accordion>
  
  <Accordion title="Auto-Resolver Actions">
    - Tracker status: `RESOLVED`  
    - Resolution reason: Activity/stage/reassign based
  </Accordion>
</AccordionGroup>

<Note>
Analytics or dashboards counting only `RESOLVED` status would undercount total closed escalations, skewing metrics and reporting.
</Note>

## Documentation Discrepancies

### 8. Spec docs vs actual API routes

There's a mismatch between specification documentation and actual implementation:

<CodeGroup>
  ```http Specification Documentation

  ```


The actual routes use hyphenated resource-style naming while documentation shows slash-separated paths.
