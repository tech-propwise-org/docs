This document outlines critical issues and potential problems identified during a comprehensive review of the Activity Monitor and Escalation modules located in `src/modules/activity-monitor/` and `src/modules/escalation/`.

<Warning>
These issues may impact system reliability, data consistency, and user experience. Review and address based on your application's criticality requirements.
</Warning>

## Dead Code Issues

<AccordionGroup>
<Accordion title="markAlertIgnored is dead code">

### Issue
`ActivityMonitorService.markAlertIgnored` is defined but never called. The follow-up path marks alerts `IGNORED` via raw SQL inline instead of using this method.

### Impact
- Unused code increases maintenance burden
- Inconsistent alert status handling
- Potential confusion for developers

### Resolution
Either consolidate to use the method or remove it entirely to maintain code clarity.

</Accordion>
</AccordionGroup>

## Non-Deterministic Behavior

<AccordionGroup>
<Accordion title="Team lead resolution is non-deterministic">

### Issue
Both modules use `LIMIT 1` or `DISTINCT ON` without a deterministic `ORDER BY` when multiple `TEAM_ADMIN` users exist on a team.

### Impact
- Which admin receives alerts is arbitrary
- Behavior could change between scans
- Inconsistent alert routing

### Current Behavior
```sql
SELECT user_id FROM team_members 
WHERE role = 'TEAM_ADMIN' 
LIMIT 1  -- Non-deterministic without ORDER BY
```

### Recommended Fix
```sql
SELECT user_id FROM team_members 
WHERE role = 'TEAM_ADMIN' 
ORDER BY created_at ASC, user_id ASC  -- Deterministic ordering
LIMIT 1
```

</Accordion>
</AccordionGroup>

## Concurrency Issues

<AccordionGroup>
<Accordion title="Insert idempotency mismatch (escalation)">

### Issue
Escalation comments reference `INSERT … ON CONFLICT DO NOTHING`, but code uses `INSERT … WHERE NOT EXISTS`. Under concurrent transactions, a unique violation is still possible on the partial unique index.

### Impact
- Unlikely but unhandled database errors
- Potential application crashes during high concurrency
- Data consistency issues

### Technical Details
The partial unique index doesn't prevent all concurrent insert scenarios when using `WHERE NOT EXISTS` pattern.

</Accordion>
</AccordionGroup>

## Performance & Fairness Issues

<AccordionGroup>
<Accordion title="Scanner batch caps differ and lack fairness">

### Issue
- Activity monitor caps at 200 leads/rule/tick
- Escalation caps at 500
- Large backlogs drain slowly with no fairness mechanism across rules

### Impact
- Earlier rules in iteration order get priority
- Some rules may be starved of processing time
- Uneven system performance

### Current Behavior
```javascript
// Activity Monitor
const BATCH_SIZE = 200;

// Escalation  
const BATCH_SIZE = 500;
```

### Recommendation
Implement fairness scheduling or normalize batch sizes across modules.

</Accordion>
</AccordionGroup>

## State Management Issues

<AccordionGroup>
<Accordion title="Exhausted escalation tiers leave trackers in limbo">

### Issue
When all tiers have fired, the tracker stays `ACTIVE` with `next_escalation_at = NULL`. It sits indefinitely until an auto-resolution event fires.

### Impact
- Accumulation of stale trackers with no scheduled action
- No dashboard visibility into "stuck" state
- Resource waste and data pollution

### Current State Flow
<Steps>
<Step title="All escalation tiers fire">
Tracker remains `ACTIVE` but `next_escalation_at` becomes `NULL`
</Step>
<Step title="Tracker enters limbo">
No further automatic processing occurs
</Step>
<Step title="Manual intervention required">
Only auto-resolution events can clear the tracker
</Step>
</Steps>

</Accordion>
</AccordionGroup>

<AccordionGroup>
<Accordion title="notifyTeamLead = false leaves activity alerts stranded">

### Issue
When `notifyTeamLead` is false, the follow-up path stamps `follow_up_sent_at` but never transitions the alert out of `PENDING`/`ESCALATED`.

### Impact
- Alerts can linger forever with no terminal state
- Dashboard confusion
- Data integrity issues

### Problem Flow
<Steps>
<Step title="Alert created">
Alert status set to `PENDING` or `ESCALATED`
</Step>
<Step title="notifyTeamLead = false">
System stamps `follow_up_sent_at` but doesn't change status
</Step>
<Step title="Alert stuck">
No further state transitions occur
</Step>
</Steps>

</Accordion>
</AccordionGroup>

## Semantic Inconsistencies

<AccordionGroup>
<Accordion title="RESOLVED vs CANCELLED semantics diverge">

### Issue
- Archived/deleted leads get tracker status `CANCELLED` + `AUTO_ARCHIVED`/`AUTO_DELETED`
- Auto-resolver uses `RESOLVED` for activity/stage/reassign events

### Impact
Analytics or dashboards counting only `RESOLVED` would undercount total closed escalations.

### Status Usage
<Tabs>
<Tab title="CANCELLED">
Used for:
- Auto-archived leads
- Auto-deleted leads
- Manual cancellations
</Tab>
<Tab title="RESOLVED">
Used for:
- Activity-based resolution
- Stage change resolution  
- Reassignment resolution
</Tab>
</Tabs>

<Tip>
Consider normalizing to a single "closed" semantic or clearly document the distinction for analytics purposes.
</Tip>

</Accordion>
</AccordionGroup>

## Documentation Mismatches

<AccordionGroup>
<Accordion title="Spec docs vs actual API routes">

### Issue
Activity monitor specification docs reference paths like `/activity-monitor/rules`; actual controller routes are `/activity-monitor-rules` and `/activity-monitor-alerts` (hyphenated resource-style).

### Impact
- Developer confusion
- Integration difficulties
- Documentation debt

### Documented vs Actual
<CodeGroup>
```http Documented Routes
GET /activity-monitor/rules
GET /activity-monitor/alerts
```

```http Actual Routes
GET /activity-monitor-rules
GET /activity-monitor-alerts
```
</CodeGroup>
