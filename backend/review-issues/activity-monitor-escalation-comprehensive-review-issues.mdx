This document outlines critical issues identified during a deep review of the Activity Monitor and Escalation modules located in `src/modules/activity-monitor/` and `src/modules/escalation/`.

<Warning>
These issues range from dead code and non-deterministic behavior to potential data inconsistencies and architectural concerns that may impact system reliability.
</Warning>

## Dead Code Issues

### markAlertIgnored Method Unused

The `ActivityMonitorService.markAlertIgnored` method is defined but never called anywhere in the codebase. Instead, the follow-up processing path marks alerts as `IGNORED` using raw SQL inline.

<Note>
**Recommendation**: Either consolidate the codebase to use the existing method or remove it entirely to reduce maintenance overhead.
</Note>

## Non-Deterministic Behavior

### Team Lead Resolution Inconsistency

Both Activity Monitor and Escalation modules exhibit non-deterministic behavior when selecting team administrators:

- Uses `LIMIT 1` or `DISTINCT ON` without deterministic `ORDER BY` clauses
- When multiple `TEAM_ADMIN` users exist on a team, alert assignment is arbitrary
- Results can change between scans, leading to inconsistent escalation patterns

<Warning>
This non-deterministic behavior could result in uneven workload distribution and unpredictable alert routing.
</Warning>

**Example problematic query pattern:**
```sql
SELECT user_id FROM team_users 
WHERE team_id = ? AND role = 'TEAM_ADMIN' 
LIMIT 1;
```

## Data Consistency Issues

### Insert Idempotency Mismatch

The escalation module has a mismatch between documentation and implementation:

- **Documentation**: References `INSERT ... ON CONFLICT DO NOTHING`
- **Implementation**: Uses `INSERT ... WHERE NOT EXISTS`

<Info>
Under concurrent transactions, unique violations are still possible on partial unique indexes, though unlikely. This scenario is currently unhandled.
</Info>

### Status Semantics Divergence

Inconsistent status handling between different resolution types:

<Tabs>
  <Tab title="Archived/Deleted Leads">
    - Tracker status: `CANCELLED`
    - Resolution type: `AUTO_ARCHIVED` or `AUTO_DELETED`
  </Tab>
  <Tab title="Auto-Resolved Cases">
    - Tracker status: `RESOLVED`
    - Resolution type: Activity/stage/reassign based
  </Tab>
</Tabs>

<Warning>
Analytics or dashboards counting only `RESOLVED` status would undercount total closed escalations, leading to inaccurate reporting.
</Warning>

## Performance and Fairness Issues

### Scanner Batch Limitations

The two modules use different batch processing limits without fairness mechanisms:

- **Activity Monitor**: 200 leads per rule per tick
- **Escalation**: 500 items per batch

<Note>
Large backlogs drain slowly with no fairness mechanism across rules. Earlier rules in iteration order receive priority processing.
</Note>

## State Management Issues

### Exhausted Escalation Tiers

When all escalation tiers have fired, trackers remain in an problematic state:

<Steps>
  <Step title="All Tiers Exhausted">
    All configured escalation tiers have been triggered for a lead
  </Step>
  <Step title="Tracker State">
    Tracker remains `ACTIVE` with `next_escalation_at = NULL`
  </Step>
  <Step title="Indefinite Wait">
    Tracker sits indefinitely until an auto-resolution event fires
  </Step>
  <Step title="Accumulation Risk">
    Stale trackers accumulate with no scheduled action and no dashboard visibility
  </Step>
</Steps>

### Stranded Activity Alerts

When `notifyTeamLead` configuration is set to `false`:

- Follow-up path stamps `follow_up_sent_at` timestamp
- Alert never transitions out of `PENDING` or `ESCALATED` status
- Alerts can remain stranded indefinitely with no terminal state

<Check>
**Impact**: Alerts may accumulate in non-terminal states, affecting system performance and data accuracy over time.
</Check>
