Issues identified during deep review of `src/modules/activity-monitor/` and `src/modules/escalation/`.

## Dead Code Issues

### `markAlertIgnored` is unused

<Warning>
`ActivityMonitorService.markAlertIgnored` is defined but never called anywhere in the codebase.
</Warning>

The follow-up path marks alerts as `IGNORED` via raw SQL inline instead of using this method. This creates inconsistency in how alert status updates are handled.

**Recommendation:** Either consolidate to use the method consistently or remove the dead code.

## Non-Deterministic Behavior

### Team lead resolution lacks determinism

<Warning>
Both modules use `LIMIT 1` or `DISTINCT ON` without a deterministic `ORDER BY` when multiple `TEAM_ADMIN` users exist on a team.
</Warning>

When multiple team administrators are available, which admin receives alerts is arbitrary and could change between scans. This makes the system behavior unpredictable.

**Impact:** Alert assignment becomes inconsistent, potentially affecting team workflows and accountability.

## Concurrency Issues

### Insert idempotency mismatch in escalation

<Info>
Escalation comments reference `INSERT … ON CONFLICT DO NOTHING`, but actual code uses `INSERT … WHERE NOT EXISTS`.
</Info>

Under concurrent transactions, a unique violation is still possible on the partial unique index. While unlikely, this scenario remains unhandled and could cause runtime errors.

## Performance & Fairness Problems

### Scanner batch caps differ without fairness mechanism

<Steps>
<Step title="Activity Monitor Limitation">
Activity monitor caps at 200 leads/rule/tick
</Step>
<Step title="Escalation Limitation">
Escalation caps at 500 leads/rule/tick
</Step>
<Step title="Fairness Issue">
Large backlogs drain slowly with no fairness mechanism across rules — earlier rules in iteration order get priority
</Step>
</Steps>

This can lead to some rules being processed while others are starved, especially under high load conditions.

## State Management Issues

### Exhausted escalation tiers create limbo state

<Warning>
When all escalation tiers have fired, the tracker stays `ACTIVE` with `next_escalation_at = NULL`.
</Warning>

These trackers sit indefinitely until an auto-resolution event fires, potentially accumulating stale trackers with:
- No scheduled action
- No dashboard visibility into "stuck" state
- Unclear resolution path

### Activity alerts can become stranded

When `notifyTeamLead = false`, the follow-up path:
1. Stamps `follow_up_sent_at`
2. Never transitions the alert out of `PENDING`/`ESCALATED` status

<Note>
Alerts can linger forever with no terminal state, creating data inconsistencies.
</Note>

## Semantic Inconsistencies

### RESOLVED vs CANCELLED status divergence

<Tabs>
<Tab title="Archived/Deleted Leads">
- Tracker status: `CANCELLED`
- Resolution type: `AUTO_ARCHIVED`/`AUTO_DELETED`
</Tab>
<Tab title="Auto-Resolver Events">
- Tracker status: `RESOLVED`
- Resolution types: activity/stage/reassign events
</Tab>
</Tabs>

**Impact:** Analytics or dashboards counting only `RESOLVED` status would undercount total closed escalations, skewing metrics and reporting.

## Documentation Misalignment

### API route specification mismatch

<Warning>
Activity monitor specification docs reference paths like `/activity-monitor/rules`, but actual controller routes use hyphenated resource-style formatting.
</Warning>

**Actual routes:**
- `/activity-monitor-rules`
- `/activity-monitor-alerts`

This discrepancy between documentation and implementation can confuse API consumers and complicate integration efforts.
