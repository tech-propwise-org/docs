This document provides a comprehensive review of critical issues identified in the Activity Monitor and Escalation modules during deep code analysis. These issues range from dead code and non-deterministic behavior to potential data consistency problems.

## Dead Code Issues

### `markAlertIgnored` Method Never Used

<Warning>
The `ActivityMonitorService.markAlertIgnored` method is defined but never called anywhere in the codebase.
</Warning>

The follow-up processing path marks alerts as `IGNORED` using raw SQL statements instead of utilizing this dedicated method. This creates inconsistency in how alert status updates are handled.

**Recommendation:**
- Either consolidate all `IGNORED` status updates to use the `markAlertIgnored` method
- Or remove the method entirely if the raw SQL approach is preferred

## Non-Deterministic Behavior

### Team Lead Resolution Inconsistency

<Warning>
Both Activity Monitor and Escalation modules use non-deterministic queries when selecting team administrators.
</Warning>

The current implementation uses `LIMIT 1` or `DISTINCT ON` clauses without deterministic `ORDER BY` statements when multiple `TEAM_ADMIN` users exist on a team. This means:

- Which admin receives alerts is arbitrary
- Alert assignment could change between scans
- Results may vary across different database instances

**Impact:**
```sql
-- Current problematic pattern
SELECT user_id FROM team_members 
WHERE team_id = ? AND role = 'TEAM_ADMIN' 
LIMIT 1;  -- Non-deterministic!
```

**Recommended Fix:**
```sql
-- Deterministic approach
SELECT user_id FROM team_members 
WHERE team_id = ? AND role = 'TEAM_ADMIN' 
ORDER BY user_id ASC  -- or created_at, etc.
LIMIT 1;
```

## Concurrency Issues

### Insert Idempotency Mismatch in Escalation

<Info>
Escalation module comments reference `INSERT ... ON CONFLICT DO NOTHING`, but the actual code uses `INSERT ... WHERE NOT EXISTS`.
</Info>

Under high concurrency, this mismatch can lead to:
- Unique constraint violations on partial unique indexes
- Unhandled exceptions during concurrent transaction processing
- Potential data inconsistency

**Current Implementation:**
```sql
INSERT INTO escalation_comments (...)
WHERE NOT EXISTS (SELECT 1 FROM escalation_comments WHERE ...)
```

**Expected Implementation:**
```sql
INSERT INTO escalation_comments (...) 
ON CONFLICT DO NOTHING
```

## Performance & Fairness Issues

### Inconsistent Scanner Batch Limits

<Tip>
Different batch size limits across modules can lead to uneven processing performance.
</Tip>

Current batch size configuration:
- **Activity Monitor:** 200 leads per rule per tick
- **Escalation:** 500 items per batch

**Problems:**
- Large backlogs drain slowly with no fairness mechanism
- Rules processed earlier in iteration order get priority
- No load balancing across different rule types

### Recommendations for Batch Processing

<Steps>
<Step title="Standardize batch sizes">
Align batch sizes across both modules for consistent performance characteristics.
</Step>

<Step title="Implement fairness mechanisms">
Add round-robin or weighted scheduling to ensure fair processing across all rules.
</Step>

<Step title="Add backlog monitoring">
Implement metrics to track and alert on processing backlogs.
</Step>
</Steps>

## State Management Issues

### Exhausted Escalation Tiers Create Limbo State

<Warning>
When all escalation tiers have fired, trackers remain in an indefinite `ACTIVE` state.
</Warning>

**Current Behavior:**
- Tracker status: `ACTIVE`
- `next_escalation_at`: `NULL`
- No scheduled actions
- Invisible on dashboards

**Problems:**
- Accumulation of stale trackers
- No visibility into "stuck" escalations
- Trackers wait indefinitely for auto-resolution events

### Activity Alerts Stranded Without Team Lead Notification

When `notifyTeamLead` is set to `false`, the system exhibits problematic behavior:

1. Follow-up path stamps `follow_up_sent_at`
2. Alert never transitions out of `PENDING`/`ESCALATED` status
3. Alerts can linger indefinitely with no terminal state

<Note>
This creates a scenario where alerts are marked as processed but never reach resolution.
</Note>

## Semantic Inconsistencies

### RESOLVED vs CANCELLED Status Confusion

Different resolution paths use inconsistent status values:

<Tabs>
<Tab title="Archive/Delete Path">
- Tracker status: `CANCELLED`
- Resolution reason: `AUTO_ARCHIVED` or `AUTO_DELETED`
</Tab>

<Tab title="Auto-Resolver Path">
- Tracker status: `RESOLVED`
- Used for activity/stage/reassign events
</Tab>
</Tabs>

**Impact on Analytics:**
- Dashboards counting only `RESOLVED` status will undercount total closed escalations
- Reporting inconsistencies across different closure scenarios
- Potential confusion in business metrics

## Documentation Misalignment

### API Route Specification Mismatch

<Info>
Activity Monitor specification documents don't match actual API implementation.
</Info>

**Documented Routes:**
- `/activity-monitor/rules`
- `/activity-monitor/alerts`

**Actual Controller Routes:**
- `/activity-monitor-rules`
- `/activity-monitor-alerts`

This hyphenated resource-style routing differs from the documented nested approach.
