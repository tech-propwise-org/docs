## Dead Code and Unused Methods

### `markAlertIgnored` Method Not Used

The `ActivityMonitorService.markAlertIgnored` method is defined but never called in the codebase. Instead, the follow-up processing marks alerts as `IGNORED` using raw SQL queries inline.

<Steps>
<Step title="Identify usage patterns">
Audit all locations where alerts are marked as ignored
</Step>
<Step title="Choose consolidation approach">
Either refactor to use the existing method or remove the dead code
</Step>
<Step title="Update tests">
Ensure test coverage reflects the chosen approach
</Step>
</Steps>

<Note>
This inconsistency could lead to maintenance issues and makes the codebase harder to understand.
</Note>

## Non-Deterministic Behavior

### Team Lead Resolution Inconsistency

Both Activity Monitor and Escalation modules exhibit non-deterministic behavior when selecting team administrators.

**Problem**: When multiple `TEAM_ADMIN` users exist on a team, the modules use `LIMIT 1` or `DISTINCT ON` without a deterministic `ORDER BY` clause.

**Impact**: 
- Which admin receives alerts is arbitrary
- Behavior could change between scans
- Inconsistent user experience

<CodeGroup>
```sql Current (Non-deterministic)
SELECT user_id FROM team_users 
WHERE team_id = $1 AND role = 'TEAM_ADMIN' 
LIMIT 1
```

```sql Recommended (Deterministic)
SELECT user_id FROM team_users 
WHERE team_id = $1 AND role = 'TEAM_ADMIN' 
ORDER BY created_at ASC, user_id ASC 
LIMIT 1
```
</CodeGroup>

## Data Consistency Issues

### Insert Idempotency Mismatch

**Location**: Escalation module comment handling

The documentation references `INSERT … ON CONFLICT DO NOTHING` pattern, but the actual implementation uses `INSERT … WHERE NOT EXISTS`. Under concurrent transactions, this can still result in unique violations on partial unique indexes.

<Warning>
While unlikely, this race condition is unhandled and could cause escalation processing failures.
</Warning>

## Performance and Fairness Issues

### Inconsistent Batch Processing Limits

The two modules use different batch size caps without clear justification:

<CardGroup cols={2}>
<Card title="Activity Monitor" icon="gauge">
200 leads per rule per processing tick
</Card>
<Card title="Escalation Module" icon="gauge">
500 items per batch
</Card>
</CardGroup>

**Problems**:
- Large backlogs drain slowly
- No fairness mechanism across rules
- Earlier rules in iteration order get processing priority

### Recommendation

Implement a fair queuing mechanism or standardize batch sizes with clear performance rationale.

## State Management Issues

### Exhausted Escalation Tiers

When all escalation tiers have fired, trackers remain in an ambiguous state:

<Tabs>
<Tab title="Current Behavior">
- Status: `ACTIVE`
- `next_escalation_at`: `NULL`
- No scheduled actions
- Invisible to dashboards
</Tab>
<Tab title="Expected Behavior">
- Clear terminal state
- Dashboard visibility
- Defined cleanup process
</Tab>
</Tabs>

<Info>
This can lead to accumulation of stale trackers with no clear resolution path.
</Info>

### Stranded Activity Alerts

When `notifyTeamLead = false`, alerts can become stranded:

1. Follow-up process stamps `follow_up_sent_at`
2. Alert never transitions out of `PENDING`/`ESCALATED` status
3. Alerts linger indefinitely with no terminal state

## Semantic Inconsistencies

### RESOLVED vs CANCELLED Status Divergence

Different automatic resolution scenarios use inconsistent status values:

<AccordionGroup>
<Accordion title="Archived/Deleted Leads">
- Tracker status: `CANCELLED`
- Resolution reason: `AUTO_ARCHIVED` or `AUTO_DELETED`
</Accordion>
<Accordion title="Auto-Resolver Actions">
- Tracker status: `RESOLVED`
- Covers: activity changes, stage changes, reassignments
</Accordion>
</AccordionGroup>

**Impact**: Analytics counting only `RESOLVED` status will undercount total closed escalations.
