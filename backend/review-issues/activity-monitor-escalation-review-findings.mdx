This document outlines critical issues identified during a comprehensive review of the Activity Monitor and Escalation modules. These findings highlight areas requiring immediate attention to improve system reliability and maintainability.

## Dead Code and Unused Methods

<Warning>
The following method is defined but never used in the codebase, creating maintenance overhead and potential confusion.
</Warning>

### markAlertIgnored Method

The `ActivityMonitorService.markAlertIgnored` method is defined but never called anywhere in the codebase. Instead, the follow-up path marks alerts as `IGNORED` using raw SQL inline rather than utilizing this dedicated method.

**Recommendation:** Either consolidate the code to use the existing method or remove it entirely to reduce technical debt.

## Non-Deterministic Behavior

<Warning>
Critical issue affecting system predictability and debugging capabilities.
</Warning>

### Team Lead Resolution

Both Activity Monitor and Escalation modules exhibit non-deterministic behavior when selecting team leads:

- Uses `LIMIT 1` or `DISTINCT ON` without deterministic `ORDER BY` clauses
- When multiple `TEAM_ADMIN` users exist on a team, alert assignment becomes arbitrary
- Selected admin can change between scans, making behavior unpredictable

**Impact:** This makes debugging difficult and creates inconsistent user experiences.

## Data Integrity Issues

### Insert Idempotency Mismatch

<Info>
Escalation module has a race condition vulnerability under concurrent transactions.
</Info>

The escalation comments reference using `INSERT … ON CONFLICT DO NOTHING` for idempotency, but the actual implementation uses `INSERT … WHERE NOT EXISTS`. This mismatch means:

- Under concurrent transactions, unique violations are still possible
- The partial unique index doesn't prevent all race conditions
- While unlikely, this scenario is currently unhandled

## Performance and Fairness Concerns

<Note>
Different batch processing limits between modules can lead to uneven performance characteristics.
</Note>

### Scanner Batch Caps

The two modules use different batch processing limits:

- **Activity Monitor:** 200 leads/rule/tick
- **Escalation:** 500 leads/rule/tick

**Issues:**
- Large backlogs drain slowly
- No fairness mechanism across rules
- Earlier rules in iteration order receive priority
- Can lead to starvation of later rules

## State Management Problems

### Exhausted Escalation Tiers

<Warning>
Escalation trackers can become permanently stuck in an active state.
</Warning>

When all escalation tiers have fired:
- Tracker remains `ACTIVE` with `next_escalation_at = NULL`
- Sits indefinitely until an auto-resolution event fires
- Can accumulate stale trackers with no scheduled action
- No dashboard visibility into "stuck" state

### Stranded Activity Alerts

When `notifyTeamLead` is set to `false`:
- Follow-up path stamps `follow_up_sent_at`
- Never transitions alert out of `PENDING`/`ESCALATED` status
- Alerts can linger forever with no terminal state

## Semantic Inconsistencies

<Info>
Status semantics vary between different resolution paths, affecting analytics and reporting.
</Info>

### RESOLVED vs CANCELLED Status

The system uses different status values for similar outcomes:

<Tabs>
  <Tab title="Archived/Deleted Leads">
    - Tracker status: `CANCELLED`
    - Reason: `AUTO_ARCHIVED` or `AUTO_DELETED`
  </Tab>
  <Tab title="Auto-Resolver">
    - Tracker status: `RESOLVED`
    - For: activity, stage, and reassign events
  </Tab>
</Tabs>

**Impact:** Analytics or dashboards counting only `RESOLVED` status would undercount total closed escalations.

## API Documentation Inconsistencies

<Check>
Quick fix required for API documentation alignment.
</Check>

### Route Path Discrepancies

The specification documents reference REST-style paths that don't match actual implementation:

| Documentation | Actual Routes |
|---------------|---------------|
| `/activity-monitor/rules` | `/activity-monitor-rules` |
| `/activity-monitor/alerts` | `/activity-monitor-alerts` |

The actual controller routes use hyphenated resource-style naming conventions.

## Recommended Actions

<Steps>
  <Step title="Address Dead Code">
    Remove unused `markAlertIgnored` method or refactor code to use it consistently
  </Step>
  <Step title="Fix Non-Deterministic Behavior">
    Add deterministic `ORDER BY` clauses for team lead selection
  </Step>
  <Step title="Resolve Data Integrity Issues">
    Implement proper `INSERT … ON CONFLICT DO NOTHING` handling
  </Step>
  <Step title="Standardize Batch Processing">
    Align batch caps and implement fairness mechanisms
  </Step>
  <Step title="Improve State Management">
    Add terminal states for exhausted escalations and stranded alerts
  </Step>
  <Step title="Standardize Status Semantics">
    Align status values across different resolution paths
  </Step>
  <Step title="Update API Documentation">
    Correct route paths in specification documents
  </Step>
</Steps>
