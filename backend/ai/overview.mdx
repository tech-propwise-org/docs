The AI module provides artificial intelligence capabilities for the PropWise CRM system, including knowledge management, document processing, embeddings, and system prompt management. It supports both internal authenticated operations and external API access for integrations like n8n workflows.

<Info>
The module is designed around a knowledge base system where documents and web content can be ingested, processed into chunks, embedded for semantic search, and used to provide contextual AI responses through system prompts.

Last Updated: 2024-12-19
</Info>

## Architecture

The AI module follows a service-oriented architecture with clear separation of concerns:

- **Knowledge Management**: Handles ingestion and storage of knowledge sources (documents, URLs)
- **Document Processing**: Converts various document formats (PDF, DOCX, TXT) into processable text
- **Embedding Generation**: Creates vector embeddings using Google Gemini API for semantic search
- **Web Scraping**: Integrates with Firecrawl service for URL content extraction
- **Public API**: Provides external access for integrations with API key authentication

### Design Decisions

<AccordionGroup>
  <Accordion title="Multi-tenant Architecture">
    All entities are organization-scoped with RLS (Row Level Security) enforcement to ensure complete data isolation between organizations.
  </Accordion>
  
  <Accordion title="Chunking Strategy">
    Documents are split into 3000-character chunks with 200-character overlap for optimal embedding performance and semantic search accuracy.
  </Accordion>
  
  <Accordion title="Soft Deletes">
    All entities use `isDeleted` flag with MikroORM filters for data retention while maintaining clean user interfaces.
  </Accordion>
  
  <Accordion title="External Dependencies">
    Google Gemini for embeddings and Firecrawl for web scraping provide best-in-class AI and content extraction capabilities.
  </Accordion>
  
  <Accordion title="Dual Access Pattern">
    Internal authenticated endpoints for CRM users and public API key endpoints for external integrations like n8n workflows.
  </Accordion>
</AccordionGroup>

## Entities

### KnowledgeSource

Represents a source of knowledge that can be processed and chunked.

| Field | Type | Description |
|-------|------|-------------|
| `id` | UUID | Primary key |
| `name` | string | Display name for the source |
| `type` | KnowledgeSourceType | Either 'document' or 'url' |
| `url` | string? | URL if type is 'url' |
| `filename` | string? | Original filename if type is 'document' |
| `mimeType` | string? | MIME type of document |
| `organization` | Organization | Organization owner |
| `isDeleted` | boolean | Soft delete flag (default: false) |
| `createdAt` | Date | Creation timestamp |
| `updatedAt` | Date | Last update timestamp |
| `chunks` | KnowledgeChunk[] | Associated knowledge chunks |

### KnowledgeChunk

Represents a processed chunk of text from a knowledge source with its embedding.

| Field | Type | Description |
|-------|------|-------------|
| `id` | UUID | Primary key |
| `source` | KnowledgeSource | Parent knowledge source |
| `text` | string | The actual text content |
| `embedding` | number[] | Vector embedding for semantic search |
| `organization` | Organization | Organization owner |
| `isDeleted` | boolean | Soft delete flag (default: false) |
| `createdAt` | Date | Creation timestamp |
| `updatedAt` | Date | Last update timestamp |

### SystemPrompt

Stores system prompts used for AI interactions.

| Field | Type | Description |
|-------|------|-------------|
| `id` | UUID | Primary key |
| `prompt` | string | The system prompt text |
| `isActive` | boolean | Whether this prompt is active (default: true) |
| `organization` | Organization | Organization owner |
| `isDeleted` | boolean | Soft delete flag (default: false) |
| `createdAt` | Date | Creation timestamp |
| `updatedAt` | Date | Last update timestamp |

## API Reference

<Tabs>
  <Tab title="Internal API">
    ### Internal API (Authenticated)

    **Base Path:** `/ai/knowledge`  
    **Authentication:** Bearer token required  
    **Authorization:** Role-based access control

    #### Upload Document

    <CodeGroup>
    ```bash cURL
    curl -X POST /ai/knowledge/sources/upload \
      -H "Authorization: Bearer {token}" \
      -F "file=@document.pdf" \
      -F "name=Important Document"
    ```
    </CodeGroup>

    **Parameters:**
    - `file`: File (PDF, DOCX, or TXT)
    - `name`: string (display name)

    **Response:**
    ```json
    {
      "id": "uuid",
      "name": "string",
      "type": "document",
      "filename": "string",
      "mimeType": "string",
      "createdAt": "2024-12-19T10:00:00Z"
    }
    ```

    #### Add URL Source

    <CodeGroup>
    ```bash cURL
    curl -X POST /ai/knowledge/sources/url \
      -H "Authorization: Bearer {token}" \
      -H "Content-Type: application/json" \
      -d '{
        "name": "Company Website",
        "url": "https://example.com"
      }'
    ```
    </CodeGroup>

    #### Get Knowledge Sources

    <CodeGroup>
    ```bash cURL
    curl -X GET /ai/knowledge/sources \
      -H "Authorization: Bearer {token}"
    ```
    </CodeGroup>

    #### Update Knowledge Source

    <CodeGroup>
    ```bash cURL
    curl -X PUT /ai/knowledge/sources/{id} \
      -H "Authorization: Bearer {token}" \
      -H "Content-Type: application/json" \
      -d '{"name": "Updated Name"}'
    ```
    </CodeGroup>

    #### Delete Knowledge Source

    <CodeGroup>
    ```bash cURL
    curl -X DELETE /ai/knowledge/sources/{id} \
      -H "Authorization: Bearer {token}"
    ```
    </CodeGroup>
  </Tab>

  <Tab title="Public API">
    ### Public API (API Key)

    **Base Path:** `/ai/public`  
    **Authentication:** API key header required

    #### Get Active System Prompt

    <CodeGroup>
    ```bash cURL
    curl -X GET /ai/public/system-prompt \
      -H "X-API-Key: {api-key}"
    ```

    ```javascript JavaScript
      headers: {
    ```

    ```json
      "createdAt": "2024-12-19T10:00:00Z"

    ```

    <Note>
    This endpoint is designed for external integrations like n8n workflows that need to retrieve the current active system prompt for AI interactions.
    </Note>

</Tabs>

## Services

### AIKnowledgeService

Primary service for knowledge management operations:

- Upload and process documents
- Add URL-based knowledge sources
- Manage knowledge source lifecycle
- Handle document processing and chunking

### AIPublicService

Handles public API operations with API key authentication:

- Retrieve active system prompts
- Designed for external integrations (n8n, etc.)

### DocumentProcessorService

Processes various document formats into text:

<CardGroup cols={2}>
  <Card title="PDF Processing" icon="file-pdf">
    Uses `pdf-parse` library for reliable PDF text extraction
  </Card>
  <Card title="DOCX Processing" icon="file-word">
    Uses `mammoth` library for Microsoft Word document conversion
  </Card>
  <Card title="TXT Processing" icon="file-text">
    Direct text extraction with encoding detection
  </Card>
  <Card title="Text Chunking" icon="scissors">
    Splits text into 3000-character chunks with 200-character overlap
  </Card>
</CardGroup>

### EmbeddingService

Manages vector embedding generation using Google Gemini API:

- **Batch Processing**: Processes 200 items per batch for optimal performance
- **Rate Limiting**: Handles API rate limits with automatic retry logic
- **Vector Storage**: Stores embeddings for semantic search capabilities

<Note>
The EmbeddingService manages vector embedding generation using Google Gemini API with batch processing (200 items per batch) and handles API rate limiting and retries automatically.
</Note>

### FirecrawlService

Integrates with Firecrawl service for web content extraction:

- **Website Mapping**: Discovers URLs through site crawling
- **Content Scraping**: Extracts and converts web content to markdown
- **API Integration**: Handles external API communication with error handling

## Business Rules

### Knowledge Sources

<Steps>
  <Step title="Organization Isolation">
    Users can only access knowledge sources within their organization through RLS enforcement
  </Step>
  <Step title="File Type Restrictions">
    Only PDF, DOCX, and TXT files are supported for document upload
  </Step>
  <Step title="URL Validation">
    URLs must be accessible and return valid content before processing
  </Step>
  <Step title="Soft Delete">
    Deleted sources are flagged but data is retained for audit purposes
  </Step>
</Steps>

### System Prompts

<Warning>
Only one system prompt can be active per organization at a time. When a new prompt is activated, the previous one is automatically deactivated but retained for version history.
</Warning>

1. **Single Active Prompt**: Only one system prompt can be active per organization at a time
2. **Required Content**: System prompts cannot be empty
3. **Version History**: Previous prompts are retained when deactivated

### Processing Rules

1. **Chunking Strategy**: 3000 characters per chunk with 200-character overlap
2. **Embedding Generation**: All chunks must have embeddings generated
3. **Batch Processing**: Large documents are processed in batches to avoid timeouts

## Patterns and Conventions

### Error Handling

- Service-level exceptions are caught and converted to appropriate HTTP status codes
- File upload errors return `BadRequestException` with descriptive messages
- Missing resources return `NotFoundException`

### Data Access

- All repository operations respect organization boundaries through RLS
- Soft delete pattern implemented consistently across all entities
- Audit timestamps (`createdAt`, `updatedAt`) maintained automatically

### Security

<Check>
The AI module implements comprehensive security measures including API key authentication for public endpoints, Bearer token authentication for internal endpoints, and role-based authorization using `@CheckAccess` decorator.
</Check>

### Performance

- Batch processing for embedding generation
- Chunked document processing to handle large files
- Async operations for external API calls
