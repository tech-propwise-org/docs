The AI module owns both the **AI Agent runtime** (fully implemented with complete conversation management, tool execution, budget controls, eval capabilities, and inbound coalescing) AND the **organization-scoped knowledge base** that powers retrieval-augmented generation across the platform — for both the internal CRM assistant (`AiAssistantModule` consumes it through `AIKnowledgeService` + `AiKnowledgeToolsProvider`) and the AI Agent runtime that drafts customer-facing replies.

<Info>
**Implementation Status:** The AI module is now fully operational and integrated into the main application with complete AI Agent runtime implementation. This module owns BOTH the AI Agent runtime (PCD-64 `AiAgent*` entities/services, `AiAgentInboundListener`, `AiAgentExecuteWorker`, agent inbox `/v1/ai-agents/*` HTTP surface, workflow/messaging-automation integration) AND the shared OpenAI BYOK knowledge base stack consumed by the internal CRM assistant (`KnowledgeBase` + `KnowledgeChunk` entities, `EmbeddingService`, `OrganizationLlmKey` plumbing, `/v1/ai-knowledge-bases` controller).

**Template System:** The module includes 10 pre-configured AI agent templates seeded via `AiAgentTemplateSeeder`: Receptionist, Sales Qualification, Listing Inquiry, Off-Plan Inquiry, Appointment Booking, After-Hours, FAQ & Support, Campaign Lead Capture, Spam Handler, and Human Handoff. These templates provide ready-to-use agent configurations for common real estate use cases.

**Full Implementation:** All core services are implemented including `AiAgentService`, `AiAgentRuntimeService`, `AiAgentResolverService`, `AiAgentContextBuilderService`, `AiAgentTokenBudgetService`, `AiAgentToolRegistryService`, `AiAgentActionService`, `AiAgentMediaProcessorService`, `AiAgentPricingService`, `AiAgentOptimizeService`, `AiAgentTemplateService`, `AiAgentKnowledgeBaseService`, `AiActivityLogService`, and supporting infrastructure. AI-credit affordability/gating is exclusively handled by the subscription module's unified-wallet `CreditMeteringService`. The legacy USD telemetry budget system has been completely removed.

Last Updated: 2026-05-28 | Status: Active — Fully Implemented and Operational
</Info>

Four ingestion shapes are supported through one entity (`KnowledgeBase`): uploaded **Documents** (PDF / DOCX / XLSX / TXT / Markdown), scraped **Pages**, **FAQ** rows, and freeform **Snippets**. Every ingestion path produces one `KnowledgeBase` row plus N `KnowledgeChunk` rows with pgvector embeddings (`text-embedding-3-small`, 768 dims) generated through the requesting organization's own OpenAI API key (BYOK — see `OrganizationLlmKey` below). The HTTP surface is `/v1/ai-knowledge-bases/*` (`AiAgentKnowledgeBaseController`); there is no API-key public surface — only authenticated, org-scoped, tenant-isolated access.

The PCD-64-shared `EmbeddingService` is intentionally stateless and takes the OpenAI key as a per-call argument. This is what makes BYOK clean: there is no global Gemini / OpenAI key on the platform anymore. Every embedding call is billed to the originating org's OpenAI project (provisioned by `OpenAiProjectProvisioner` against the platform admin key).

The `AiKnowledgeToolsProvider` implements the assistant's `searchKnowledge` tool with security-first design: knowledge chunks are kept OUT of the summary and only exposed through the fenced untrusted content channel to prevent prompt injection from document content. CRUD goes through `KnowledgeBaseService` + the controller.

The AI assistant speaks **"assignment" / "assignee" / "assigned to"** to users — never "stakeholder". The `EntityStakeholder` entity, table, service classes, AI tool name (`getLeadStakeholders`), and REST DTO field names (`stakeholders` / `primaryStakeholder`) remain unchanged as an internal contract. To stop the legacy word from leaking via tool-call traces, two boundary surfaces project the rows under assignee-named keys:

<Note>

- **`getLeadStakeholders.execute()`** (in `src/modules/crm/lead/ai/lead-ai-tools.provider.ts`) returns `assigneeCount`, `userAssigneeCount`, `teamAssigneeCount`, `primaryAssignee`, `assignees`, `assignedTeam`
- **Pre-hydrated lead context** (in `src/modules/ai-assistant/services/ai-context-builder.service.ts`) projects `compactContext.lead.primaryAssignee` and `compactContext.lead.assignees` so the LEAD-scope context payload uses the same keys as the tool output
- **Lead schema glossary** (`src/modules/crm/lead/ai/lead-schema-glossary.constants.ts` → `assignment` section) documents `assignees` / `primaryAssignee` / `assignedTeam` as the canonical field names

See **`Docs/STAKEHOLDER_SYSTEM.md` → Vocabulary** for the cross-cutting policy.

## Security Architecture

The AI module implements comprehensive security measures to protect against prompt injection, unauthorized access, and data leakage:

<AccordionGroup>
  <Accordion title="Knowledge Base Security">
    The `searchKnowledge` tool keeps document content out of the main summary and routes it through the `untrustedContent` channel, wrapped in XML `<source author_controlled="true">` tags. This prevents malicious document content from being interpreted as instructions by the AI model. Knowledge chunks are kept OUT of the summary and only exposed through the fenced untrusted content channel to prevent prompt injection from document content.
  </Accordion>
  
  <Accordion title="Permission Context Security">
    User-authored free-text fields are systematically identified and routed through the untrusted content pipeline, ensuring that CRM notes, lead source details, and property interest notes cannot inject prompt instructions. The system maintains a clear separation between structured data and potentially malicious user content.
  </Accordion>
  
  <Accordion title="Live Model-Facing Sanitization">
    The tool bridge applies real-time permission key scrubbing to structured summaries, warnings, and citation labels before they reach the model, preventing internal RBAC permission keys from leaking through tool responses. This maintains proper access control boundaries and prevents system information disclosure through tool outputs, ensuring that internal permission identifiers never appear in conversation contexts where they could be exploited or confuse the model.
  </Accordion>
  
  <Accordion title="Byte-Capped Result Truncation">
    All tool outputs and context snapshots are automatically truncated using `capToByteBudget()` to prevent oversized responses that could exceed model context limits or degrade performance. This shape-aware truncation intelligently preserves identity and structural fields (IDs, names, primary keys) while shrinking large payloads first (e.g., property interest snapshots, assignee lists, deal arrays, nested relationship data). The truncation algorithm understands data importance hierarchies, ensuring critical information remains intact while safely reducing less critical content, maintaining both functionality and performance under token budget constraints.
  </Accordion>
  
  <Accordion title="Enhanced Error Handling">
    The tool bridge converts thrown exceptions into sanitized, typed results instead of letting raw errors reach the AI SDK. `ForbiddenException` becomes `access_denied` with a generic message; other errors become `tool_failed` with scrubbed messages to prevent information leakage through error messages. This ensures that stack traces, internal file paths, database connection strings, environment variable names, and other sensitive system details never reach the model or appear in logs accessible to end users. The sanitization layer acts as a security boundary between internal application state and external AI processing.
  </Accordion>
  
  <Accordion title="Actor Resolution and Audit Attribution">
    AI agents use shared actor resolution utilities from `@/modules/shared/org-member.utils` (`resolveAiServiceUserId` for organization-scope agents) and `@/modules/shared/actor-hydration.util` (`hydrateActorRequestUser` for full RequestUser hydration with roles and team memberships). These utilities provide a single source of truth for per-org AI service account lookup and are shared with the workflow engine to ensure consistent actor resolution patterns across the platform.
    
    Organization-scope agent actions are attributed to the per-org AI service user (renders as "AI USER" in audit logs), while personal-scope actions are attributed to the agent owner. This ensures AI-initiated lead updates, conversation assignments, and person mutations are never "Unknown user" in the audit trail through proper use of `auditContextForActor`. All AI operations maintain full audit trails with proper attribution for compliance, accountability, and security review. The shared utilities eliminate duplication and ensure workflows and AI agents use identical actor resolution logic for consistent audit behavior.
  </Accordion>
  
  <Accordion title="Unified Credit System">
    The AI module exclusively uses the subscription module's unified-wallet `CreditMeteringService` for all affordability checks and credit deductions. The legacy USD telemetry budget system (`AiCreditUsage` entity, `AiAgentBudgetGuardService`) has been completely removed from the codebase.
    
    All AI operations (agent runtime execution, instruction optimization, knowledge base embedding ingestion, and media processing) are gated by and billed to the unified credit wallet with org pool allocation and per-user spending ceilings. The AI module no longer maintains any USD telemetry ledger or separate budget tracking - all spend tracking, affordability gating, pre-execution checks, and billing is handled exclusively by the unified credit system. This ensures consistent credit management across all platform capabilities, simplifies the architecture by removing duplicate budget systems, and provides organizations with unified visibility into all AI-related costs through a single credit accounting system.
  </Accordion>
  
  <Accordion title="Untrusted Content Fencing">
    User-authored free-text from lead source details, disqualification notes, and property interest notes is systematically routed through the `untrustedContext` channel and rendered as fenced XML blocks by the system prompt. This prevents prompt injection while preserving informational content for legitimate AI processing. The fencing mechanism uses clear XML boundaries (`<source author_controlled="true">`) that the system prompt is trained to recognize and handle appropriately, ensuring user content never bleeds into instruction context where it could override system behavior or inject malicious commands. This architectural separation allows the AI to access user-provided information while maintaining strict control boundaries.
  </Accordion>
</AccordionGroup>

## Live Web Search (Primary Market Grounding)

As of `crm-assistant-v0.38-live-web-search`, the Market & Regulations persona's PRIMARY grounding source is **live web search**, not the curated corpus below.

<Info>
**Firecrawl Integration:** `FirecrawlService.search(query, options?)` calls the Firecrawl `/search` endpoint with `scrapeOptions: { formats: ['markdown'], onlyMainContent: true, maxAge: FIRECRAWL_SEARCH_SCRAPE_MAX_AGE_MS }`, returning the top results with their live-scraped markdown. Unlike `scrapeUrl` (one caller-supplied URL → SSRF guard), `search` issues a query to Firecrawl's search engine — there is no user-controlled URL, so no SSRF surface on our side (Firecrawl fetches server-side). Default `5` results, capped at `10`; per-result markdown capped at `FIRECRAWL_SEARCH_CONTENT_MAX_CHARS`.

**API Key Requirement:** Requires `FIRECRAWL_API_KEY` — there is no offline fallback for search; without the key it throws a clear `BadRequestException` so the assistant degrades to answering from general knowledge.

**Latency Controls:** The request sends a server-side body `timeout` to prevent excessive web retry cases that could cause the assistant to appear stuck on "Thinking" status.

**Security Implementation:** The `searchRealEstateWeb` tool maps the hits where scraped page text reaches the model ONLY via `untrustedContent[]` (XML-fenced `<source author_controlled="true">` — same prompt-injection prevention as knowledge base content).
</Info>

**Authentication:** Bearer token required

#### Create AI Agent

<CodeGroup>
```bash cURL
curl -X POST /v1/ai-agents \
  -H "Authorization: Bearer {token}" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Sales Assistant",
    "instructions": "Help qualify leads and schedule appointments",
    "aiMode": "AUTO",
    "templateId": "uuid"
  }'
```
