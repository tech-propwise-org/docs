## Overview

The Context Extraction Service automatically extracts structured CRM data from messaging conversations using LLM analysis. It runs asynchronously outside the message pipeline, analyzing conversation text to identify person details, lead qualification data, property interests, and intent signals. Extracted data is presented to agents for review before being committed to the CRM.

## Extraction Targets

### Person Details

Fields extracted and mapped to the `Person` entity:

| Field | Entity Field | Example Input | Extracted Value |
|-------|--------------|---------------|-----------------|
| First name | `Person.firstName` | "My name is Ahmed" | `Ahmed` |
| Last name | `Person.lastName` | "I'm Ahmed Al-Rashid" | `Al-Rashid` |
| Email | `PersonEmail.email` | "reach me at ahmed@example.com" | `ahmed@example.com` |
| Phone | `PersonPhone.phoneNumber` | "call me on 0501234567" | `+971501234567` |
| Primary language | `Person.languages[]` with `NATIVE` proficiency | (detected from message language) | `{ code: 'ar', proficiency: 'NATIVE' }` |

### Lead Qualification Details

Fields extracted and mapped to the `Lead` entity:

| Field | Entity Field | Example Input | Extracted Value |
|-------|--------------|---------------|-----------------|
| Budget | `Lead.budget` | "budget is around 2 million" | `2000000` |
| Budget currency | `Lead.budgetCurrency` | "2M AED" | `AED` |
| Timeline | `Lead.timeline` | "looking to move in 3 months" | `1-3 months` |
| Intent | `Lead.intent` | "I want to buy an apartment" | `buyer` |
| Temperature | `Lead.temperature` | (inferred from urgency signals) | `hot` / `warm` / `cold` |
| Company | `Lead.companyName` | "I work at Emaar" | `Emaar` |

#### Intent Mapping

Intent signals are mapped to the `LeadIntent` enum:

| Detected Signal | Enum Value |
|----------------|------------|
| buy, purchase, own | `buyer` |
| sell, list, dispose | `seller` |
| rent, lease, tenant | `renter` |
| lease out, find tenants, landlord | `landlord` |
| invest, ROI, yield, portfolio | `investor` |

### Property Interest Details

Fields extracted and mapped to `LeadPropertyInterest` entity:

| Field | Entity Field | Example Input | Extracted Value |
|-------|--------------|---------------|-----------------|
| Bedroom range | `bedroomsMin` / `bedroomsMax` | "3 bedroom apartment" | `3` / `3` |
| Budget range | `budgetMin` / `budgetMax` | "between 1.5M and 2M" | `1500000` / `2000000` |
| Size range | `sizeMin` / `sizeMax` | "at least 1500 sqft" | `1500` / `null` |
| Location refs | `wished*Ids` + snapshots | "in Dubai Marina or JBR" | matched Labs IDs + labels |
| Inventory units | `inventoryUnitIds` | "that unit in Marina" | selected inventory UUIDs |

## Structured Output Schema

### LLM Output Format

The extraction service uses **JSON mode** (OpenAI) or **function calling** to ensure structured output. The LLM returns a single JSON object:

```typescript
interface ExtractionResult {
  confidence: number; // 0.0 - 1.0 overall confidence
  person: {
    firstName?: string;
    lastName?: string;
    email?: string;
    phone?: string; // normalized to E.164 format
    languages?: Array<{ 
      code: string; 
      proficiency: 'BASIC' | 'CONVERSATIONAL' | 'FLUENT' | 'NATIVE' 
    }>;
  } | null;
  lead: {
    budget?: number;
    budgetCurrency?: string;
    timeline?: 'immediate' | '1-3 months' | '3-6 months' | '6+ months';
    intent?: 'buyer' | 'seller' | 'renter' | 'landlord' | 'investor';
    temperature?: 'cold' | 'warm' | 'hot';
    companyName?: string;
  } | null;
  propertyInterest?: {
    inventoryUnitIds?: string[];
    bedroomsMin?: number;
    bedroomsMax?: number;
    budgetMin?: number;
    budgetMax?: number;
    sizeMin?: number;
    sizeMax?: number;
    wishedAreaIds?: number[];
    wishedAreaSnapshots?: Array<{ 
      id: number; 
      name?: string; 
      nameEn?: string; 
      nameAr?: string 
    }>;
    notes?: string;
  } | null;
  intentSignals: Array<{
    type: 'viewing_request' | 'price_negotiation' | 'financing' | 'urgency' | 'comparison';
    detail: string; // brief description of what was detected
    confidence: number; // 0.0 - 1.0
  }>;
}
```

### Extraction Prompt Template

```text
Analyze the following conversation between a customer and a real estate agent.
Extract any structured information about the customer and their property interests.

Rules:
- Only extract information explicitly stated or strongly implied in the conversation.
- Do not infer or guess values not supported by the text.
- Normalize phone numbers to E.164 format (e.g., +971501234567).
- Normalize budget values to numeric (e.g., "2M" → 2000000).
- Set confidence based on how explicit the information is (1.0 = directly stated, 0.5 = implied).
- Return null for person/lead sections if no relevant information found.
- Return `propertyInterest: null` and an empty `intentSignals` array if none are detected.

Conversation:
{conversationMessages}

Respond with a JSON object matching the ExtractionResult schema.
```

<Note>
Confidence scores are used to prioritize which extracted data should be presented to agents and to inform the review UI's visual indicators.
</Note>

## Async Queue Job Design

### Queue Configuration

Extraction runs as an **async pg-boss job** on the `context-extraction` queue, completely decoupled from the message processing pipeline.

<Check>
**Benefits of Async Processing:**
- No latency impact on message persistence or delivery
- Extraction can take longer (10-30s) without affecting UX
- Failed extractions can be retried independently
- Extraction can be triggered on-demand or periodically
</Check>

### Job Trigger Points

| Trigger | When | Payload |
|---------|------|---------|
| Message count threshold | Every N new messages in a conversation (default: 5) | `{ conversationId, organizationId }` |
| Conversation close | When status changes to CLOSED | `{ conversationId, organizationId }` |
| Manual trigger | Agent clicks "Extract info" button | `{ conversationId, organizationId, triggeredBy }` |
| Scheduled | Periodic scan of active conversations (daily) | `{ conversationId, organizationId }` |

### Job Processing Flow

<Steps>
  <Step title="Load conversation data">
    Load conversation + last N messages (default: 50)
  </Step>
  
  <Step title="Load existing CRM data">
    Load existing Person + Lead data for delta detection
  </Step>
  
  <Step title="Compose extraction prompt">
    Build prompt with conversation text and extraction rules
  </Step>
  
  <Step title="Call LLM">
    Execute LLM call with JSON mode / function calling
  </Step>
  
  <Step title="Parse and validate">
    Parse and validate the ExtractionResult response
  </Step>
  
  <Step title="Compute delta">
    Compare new data vs existing CRM data
    <Note>Skip fields that already have values unless new value has higher confidence</Note>
  </Step>
  
  <Step title="Create review record">
    Create ExtractionReview record pending agent approval
  </Step>
  
  <Step title="Notify agent">
    Send notification via WebSocket + optional push notification
  </Step>
</Steps>

### Deduplication

<Info>
Jobs are deduplicated by `conversationId` — if an extraction job is already queued or in progress for a conversation, new triggers are skipped.

pg-boss `singletonKey: conversationId` ensures at most one active job per conversation.
</Info>

### Retry Policy

- **Max retries:** 2
- **Backoff:** Exponential (30s, 120s)
- **Final failure:** Log error, mark extraction as `FAILED`, do not block conversation

## Agent Review and Approval Flow

### ExtractionReview Entity

```typescript
@Entity({ tableName: 'extraction_review' })
class ExtractionReview {
  id: string; // UUID PK
  conversation: Conversation; // FK
  organization: Organization; // FK
  extractedData: ExtractionResult; // JSONB — full extraction output
  delta: Record<string, any>; // JSONB — only new/changed fields
  status: 'PENDING' | 'APPROVED' | 'REJECTED' | 'PARTIAL';
  reviewedBy?: User; // FK — agent who reviewed
  reviewedAt?: Date;
  appliedFields?: string[]; // which fields were accepted (for PARTIAL)
  rejectionReason?: string;
  createdAt: Date;
  updatedAt: Date;
  isDeleted: boolean;
}
```

### Review UI Flow

<Steps>
  <Step title="Receive notification">
    Agent receives a WebSocket event `extraction-ready`:
    
    ```typescript
    {
      conversationId: string;
      extractionReviewId: string;
      summary: string; // "Found: name, email, budget, 2 property interests"
      fieldCount: number;
    }
    ```
  </Step>
  
  <Step title="Open review panel">
    Agent opens the extraction review panel in the conversation view. The panel shows:
    
    - Each extracted field with current CRM value (if any) vs extracted value
    - Confidence score per field (color-coded: green > 0.8, yellow > 0.5, red < 0.5)
    - Checkboxes for selective approval (PARTIAL mode)
  </Step>
  
  <Step title="Review and approve">
    Agent chooses an action:
    
    <Tabs>
      <Tab title="Approve All">
        Apply all extracted fields to CRM entities
      </Tab>
      <Tab title="Approve Selected">
        Apply only checked fields (status = PARTIAL)
      </Tab>
      <Tab title="Reject">
        Dismiss extraction (status = REJECTED)
      </Tab>
      <Tab title="Edit & Approve">
        Agent modifies extracted values before applying
      </Tab>
    </Tabs>
  </Step>
</Steps>

## Entity Update Logic

### Application Flow

When an agent approves an extraction (fully or partially), the service applies changes within a transaction:

```typescript
async applyExtraction(
  extractionReview: ExtractionReview,
  approvedFields: string[],
  em: EntityManager,
): Promise<void>
```

### Update Rules

<AccordionGroup>
  <Accordion title="Person Updates">
    - **firstName / lastName:** Only set if currently empty on Person. Never overwrite existing names.
    - **Email:** Create new `PersonEmail` record if email doesn't exist. Mark as `isPrimary: true` if no other primary email exists.
    - **Phone:** Create new `PersonPhone` record if phone doesn't exist (after normalization). Mark as `isPrimary: true` if no other primary phone exists.
    - **Languages:** Add detected language to `Person.languages[]` if not already present. Set proficiency to `NATIVE` for detected native language.
  </Accordion>
  
  <Accordion title="Lead Updates">
    - **Budget / budgetCurrency:** Only update if current value is empty OR new confidence > 0.8
    - **Timeline:** Update if new value indicates more urgency (e.g., "immediate" overwrites "6+ months")
    - **Intent:** Only set if empty. Do not change existing intent.
    - **Temperature:** Update if new value is "hotter" (hot > warm > cold)
    - **Company:** Only set if empty
  </Accordion>
  
  <Accordion title="Property Interest Updates">
    - **New Interest:** If no active `LeadPropertyInterest` exists, create a new one with extracted fields
    - **Existing Interest:** Merge extracted fields with existing interest:
      - Expand bedroom/budget/size ranges if new values are broader
      - Add new `wishedAreaIds` and `inventoryUnitIds` to existing arrays (deduplicated)
      - Append to `notes` field with timestamp
    - **Multiple Candidates:** If multiple inventory units match, add all to `inventoryUnitIds[]`
  </Accordion>
  
  <Accordion title="Intent Signals">
    Intent signals do **not** directly update entities but trigger actions:
    
    - **viewing_request:** Create a task for the assigned agent
    - **financing:** Add tag `financing-needed` to Lead
    - **urgency:** Update `temperature` and `timeline` on Lead
    - **price_negotiation / comparison:** Log as activity on Lead for agent visibility
  </Accordion>
</AccordionGroup>

### Conflict Resolution

<Warning>
When extracted data conflicts with existing CRM data, the system follows these rules:

1. **Explicit agent input always wins** — never overwrite data that was manually entered or edited by an agent
2. **Higher confidence wins** — if confidence > 0.8, prefer extracted value over existing value (unless manually set)
3. **Additive where possible** — for arrays (emails, phones, areas), add rather than replace
4. **Timestamp-based** — for comparable fields, prefer the most recent value
</Warning>

### Transaction Safety

All entity updates are executed within a single database transaction:

```typescript
await em.transactional(async (em) => {
  // 1. Update Person
  // 2. Update/create PersonEmail, PersonPhone
  // 3. Update Lead
  // 4. Update/create LeadPropertyInterest
  // 5. Create tasks/activities for intent signals
  // 6. Mark ExtractionReview as APPROVED/PARTIAL
});
```

<Check>
If any update fails, the entire transaction rolls back and the extraction remains in `PENDING` status for retry.
</Check>

## Implementation Considerations

### Performance Optimization

<CardGroup cols={2}>
  <Card title="LLM Caching" icon="bolt">
    Cache extraction results by conversation hash to avoid re-processing unchanged conversations
  </Card>
  
  <Card title="Batch Processing" icon="layer-group">
    Process multiple extractions in parallel using pg-boss concurrency settings
  </Card>
  
  <Card title="Context Window" icon="window-maximize">
    Limit message history to last 50 messages (configurable) to stay within token limits
  </Card>
  
  <Card title="Rate Limiting" icon="gauge-high">
    Implement per-organization rate limits to prevent LLM API quota exhaustion
  </Card>
</CardGroup>

### Error Handling

<Tabs>
  <Tab title="LLM Errors">
    - Log full error details with conversation context
    - Retry with exponential backoff
    - After max retries, mark as `FAILED` and notify engineering team
    - Do not block conversation or agent workflow
  </Tab>
  
  <Tab title="Parsing Errors">
    - If LLM returns invalid JSON, log the raw response
    - Attempt partial extraction if some fields are valid
    - Create review record with warning flag
    - Allow agent to manually correct
  </Tab>
  
  <Tab title="Database Errors">
    - Roll back entire transaction on any write failure
    - Log error with extraction ID for debugging
    - Keep extraction in `PENDING` state for manual review
    - Retry transaction up to 2 times
  </Tab>
</Tabs>

### Privacy and Compliance

<Warning>
**Data Handling Requirements:**

- Extraction jobs must respect organization's data retention policies
- PII in extraction results must be encrypted at rest in `extractedData` JSONB field
- Rejected extractions should be hard-deleted after 30 days
- Audit log all extraction approvals/rejections with agent identity and timestamp
</Warning>

<Tip>
Consider implementing extraction opt-out at the organization level for privacy-sensitive clients.
</Tip>

### LLM Evaluation

<Steps>
  <Step title="Create test dataset">
    Build a dataset of 100+ real conversation samples with ground truth labels
  </Step>
  
  <Step title="Run batch extraction">
    Process all test conversations through the extraction service
  </Step>
  
  <Step title="Calculate metrics">
    - **Precision:** % of extracted fields that are correct
    - **Recall:** % of available fields that were extracted
    - **F1 Score:** Harmonic mean of precision and recall
  </Step>
  
  <Step title="Iterate on prompts">
    Refine extraction prompts based on failure analysis
  </Step>
</Steps>

<Info>
Target metrics: Precision > 0.90, Recall > 0.85 for high-confidence extractions
</Info>
