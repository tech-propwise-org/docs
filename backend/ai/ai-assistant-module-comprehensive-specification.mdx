## Recent Major Enhancements

### 1. Centralized Access Scope Contract

<Info>
Token reduction of ~10k characters per turn through centralized scope documentation
</Info>

The access scope contract has been centralized to eliminate redundant documentation across multiple tools:

#### Implementation Details

- **Single Source of Truth**: Access scope rules now defined once in `SYSTEM_PROMPT` "Access scope (`view` / `scope`) rules" block
- **Shared Constants**: `CRM_SCOPE_VIEW_DESCRIBE` and `CRM_SCOPE_TEAMID_DESCRIBE` constants provide consistent descriptions
- **Helper Function**: `crmScopeViewDescribe(myClause)` in `src/modules/crm/shared/ai/scope-describe.constants.ts`

#### Affected Tools

The following CRM tools now reference the centralized contract:
- `searchLeads`, `searchDeals`, `searchContacts`, `searchCompanies`
- `searchCommissionPayments`
- Summary and velocity aggregates (`*Summary`, `*Velocity`)
- Pipeline and duplicate detection tools
- Follow-up and attention monitoring tools
- `getMyTopAreas`, `getMyTopCommunities`

<CodeGroup>
```typescript Before - Redundant Documentation
{
  name: "searchLeads",
  description: "Search leads with comprehensive access control. The `view` parameter controls data scope: 'all' returns organization-wide results (requires team_crm.view permission), 'team' returns team-scoped results (requires teamId parameter and team membership), 'my' returns user's own leads only. When scope='team', teamId parameter is required and must match user's team membership...",
  inputSchema: z.object({
    view: z.enum(['all', 'my', 'team']).describe('Access scope - all/my/team with full permission rules...'),
    teamId: z.string().uuid().optional().describe('Required when view=team, must match user team...')
  })
}
```

```typescript After - Centralized Reference
{
  name: "searchLeads", 
  description: "Search leads with comprehensive filtering. See 'Access scope rules' section for view/teamId parameter usage.",
  inputSchema: z.object({
    view: z.enum(['all', 'my', 'team']).describe(CRM_SCOPE_VIEW_DESCRIBE),
    teamId: z.string().uuid().optional().describe(CRM_SCOPE_TEAMID_DESCRIBE)
  })
}
```
</CodeGroup>

### 2. Intent Router Enhancement

<Warning>
Active entity context is now treated as a signal, not an absolute override
</Warning>

The intent router has been improved to handle conflicts between active entity context and message content:

#### Routing Logic Priority

<Steps>
<Step title="No Message Keywords">
Fall back to active entity group (e.g., "what do you know about this lead?" on lead surface → `crm_records`)
</Step>
<Step title="Single Agreeing Family">
Use the agreeing family group (fast path, no model call required)
</Step>
<Step title="Single Conflicting Family">
Abstain from entity context, allow Tier B classifier to re-route based on message text
</Step>
<Step title="Multiple Mixed Families">
Abstain regardless of entity context, delegate to full classification
</Step>
</Steps>

#### Example Scenarios

<Tabs>
<Tab title="Before (Incorrect)">
User on lead detail page asks: "What's the off-plan market in Arjan?"
- **Result**: Force-routed to `crm_records` group
- **Problem**: Excluded all `real-estate.*` market tools
- **Outcome**: Poor response quality
</Tab>
<Tab title="After (Correct)">
User on lead detail page asks: "What's the off-plan market in Arjan?"
- **Detection**: Conflict between entity context (CRM) and message (real estate)
- **Action**: Abstain from entity context
- **Result**: Tier B classifier routes to appropriate market tools
</Tab>
</Tabs>

### 3. Market & Regulations Performance Enhancement

<Tip>
New phase-split prompting with configurable research depth for improved performance
</Tip>

#### Performance Optimizations

- **Phase-Split Prompting**: Lean routing prompt followed by comprehensive answer prompt
- **Model Tier Selection**: Low reasoning for routing, tier-default for answering
- **Research Depth Control**: User-selectable `fast` vs `deep` research modes

#### Research Depth Options

<AccordionGroup>
<Accordion title="Fast Research Mode">
- Limited source pool with tight timeout
- Single API call for quick responses
- Optimized for immediate insights
</Accordion>
<Accordion title="Deep Research Mode">
- Expanded source pool with extended timeout
- Up to 3 API calls for comparison and synthesis
- Comprehensive analysis with multiple perspectives
</Accordion>
</AccordionGroup>

#### Source Quality Improvements

The `rankRealEstateWebResults` function now prioritizes sources in this order:
1. **UAE Government/Regulatory sources** - Highest priority
2. **Press and news sources** - Secondary priority  
3. **Real estate portals** - Base priority
4. **Recency scoring** - Time-based relevance
5. **Content relevance** - Query matching

#### Mention Hydration Process

<Steps>
<Step title="Mention Resolution">
Top-level `request.mentions` merged with `clientContext.mentions` via `resolveClientContextWithMentions()`
</Step>
<Step title="Access Validation">
Each mention validated through existing entity builders with access control enforcement
</Step>
<Step title="Context Integration">
Accessible mentions hydrated into `trustedContext.mentionedEntities` with full entity projections
</Step>
<Step title="Fallback Handling">
Inaccessible mentions preserved as bare references for potential tool-call resolution
</Step>
</Steps>

### 5. Server-Authoritative Property Listing

<Info>
New `listPropertiesInPlace` tool eliminates model routing errors for location-based property searches
</Info>

#### Problem Solved

Previous implementation allowed models to misroute resolved location IDs, leading to:
- Wrong location level targeting
- Incorrect parameter mapping  
- Failed or irrelevant search results

#### Solution: `listPropertiesInPlace` Tool

<Tabs>
<Tab title="Tool Interface">
```typescript
interface ListPropertiesInPlaceParams {
  placeName: string; // Only parameter - no IDs or technical details
}

interface ListPropertiesInPlaceResult {
  found: boolean;
  place: ResolvedLocation;
  total: number;
  counts: {
    offPlan: number;
    secondary: number;
  };
  properties: PropertySummary[]; // Combined, prepared window
}
```
</Tab>
<Tab title="Processing Flow">
1. **Name Resolution**: Server resolves place name via `PropwiseLabsService.resolveLocationByName`
2. **Dual Search**: Searches both off-plan AND secondary catalogues with resolved scope
3. **Result Preparation**: Combines and formats up to 18 results (off-plan first, then secondary)
4. **Metadata Inclusion**: Provides totals, counts, and resolved location details
</Tab>
</Tabs>

#### Frontend Integration

- **Card Extraction**: `extract-cards.ts` maps results to appropriate card types
- **Progressive Display**: Shows first 6 cards with "Show more" pagination (6 additional per click)
- **No Re-fetching**: All results prepared server-side for instant progressive reveal
