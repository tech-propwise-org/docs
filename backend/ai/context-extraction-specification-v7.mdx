# Context Extraction Service Specification v7

<Info>
Last Updated: 2026-03-04  
Status: Draft
</Info>

The Context Extraction Service automatically extracts structured CRM data from messaging conversations using LLM analysis. It runs asynchronously outside the message pipeline, analyzing conversation text to identify person details, lead qualification data, property interests, and intent signals. Extracted data is presented to agents for review before being committed to the CRM.

## Extraction Targets

### Person Details

Fields extracted and mapped to the `Person` entity:

| Field | Entity Field | Example Input | Extracted Value |
|-------|-------------|---------------|----------------|
| First name | `Person.firstName` | "My name is Ahmed" | `Ahmed` |
| Last name | `Person.lastName` | "I'm Ahmed Al-Rashid" | `Al-Rashid` |
| Email | `PersonEmail.email` | "reach me at ahmed@example.com" | `ahmed@example.com` |
| Phone | `PersonPhone.phoneNumber` | "call me on 0501234567" | `+971501234567` |
| Preferred language | `Person.preferredLanguage` | (detected from message language) | `ar` / `en` |

### Lead Qualification Details

Fields extracted and mapped to the `Lead` entity:

| Field | Entity Field | Example Input | Extracted Value |
|-------|-------------|---------------|----------------|
| Budget | `Lead.budget` | "budget is around 2 million" | `2000000` |
| Budget currency | `Lead.budgetCurrency` | "2M AED" | `AED` |
| Timeline | `Lead.timeline` | "looking to move in 3 months" | `1-3 months` |
| Intent | `Lead.intent` | "I want to buy an apartment" | `buyer` |
| Temperature | `Lead.temperature` | (inferred from urgency signals) | `hot` / `warm` / `cold` |
| Company | `Lead.companyName` | "I work at Emaar" | `Emaar` |

<Accordion title="Intent mapping to LeadIntent enum">
| Detected Signal | Enum Value |
|----------------|-----------|
| buy, purchase, own | `buyer` |
| sell, list, dispose | `seller` |
| rent, lease, tenant | `renter` |
| lease out, find tenants, landlord | `landlord` |
| invest, ROI, yield, portfolio | `investor` |
</Accordion>

### Property Interest Details

Fields extracted and mapped to `LeadPropertyInterest` entity:

| Field | Entity Field | Example Input | Extracted Value |
|-------|-------------|---------------|----------------|
| Property type | `propertyType` | "looking for a villa" | `Villa` |
| Bedrooms | `bedrooms` | "3 bedroom apartment" | `3` |
| Budget range | `budgetMin` / `budgetMax` | "between 1.5M and 2M" | `1500000` / `2000000` |
| Size range | `sizeMin` / `sizeMax` | "at least 1500 sqft" | `1500` / `null` |
| Preferred areas | `preferredAreas` | "in Dubai Marina or JBR" | `["Dubai Marina", "JBR"]` |
| Communities | `preferredCommunities` | "Palm Jumeirah" | `["Palm Jumeirah"]` |
| Layout | `layoutType` | "open plan kitchen" | `OpenPlan` |
| Project reference | `refProjectId` | "Emaar Beachfront Tower 2" | (matched via project lookup) |

### Intent Signals

Higher-level signals that don't map directly to a single field but inform lead scoring and routing:

| Signal | Indicator | Action |
|--------|-----------|--------|
| High urgency | "need to move ASAP", "this week" | Set `temperature = hot`, `timeline = immediate` |
| Viewing request | "can I see the property?", "schedule a visit" | Create task for agent, flag for follow-up |
| Price negotiation | "is there any discount?", "best price?" | Flag as high-intent buyer |
| Comparison shopping | "what else do you have?", "other options" | Suggest similar properties |
| Financing inquiry | "mortgage", "payment plan", "installments" | Tag lead as `financing-needed` |
| Multiple interests | Multiple property types/areas mentioned | Create multiple `LeadPropertyInterest` records |

## Structured Output Schema

### LLM Output Format

The extraction service uses **JSON mode** (OpenAI) or **function calling** to ensure structured output. The LLM returns a single JSON object:

```typescript
interface ExtractionResult {
  confidence: number; // 0.0 - 1.0 overall confidence
  person: {
    firstName?: string;
    lastName?: string;
    email?: string;
    phone?: string; // normalized to E.164 format
    preferredLanguage?: string; // ISO 639-1
  } | null;
  lead: {
    budget?: number;
    budgetCurrency?: string;
    timeline?: 'immediate' | '1-3 months' | '3-6 months' | '6+ months';
    intent?: 'buyer' | 'seller' | 'renter' | 'landlord' | 'investor';
    temperature?: 'cold' | 'warm' | 'hot';
    companyName?: string;
  } | null;
  propertyInterests: Array<{
    propertyType?: string;
    bedrooms?: number;
    budgetMin?: number;
    budgetMax?: number;
    sizeMin?: number;
    sizeMax?: number;
    preferredAreas?: string[];
    preferredCommunities?: string[];
    notes?: string;
  }>;
  intentSignals: Array<{
    type: 'viewing_request' | 'price_negotiation' | 'financing' | 'urgency' | 'comparison';
    detail: string; // brief description of what was detected
    confidence: number; // 0.0 - 1.0
  }>;
}
```

### Extraction Prompt Template

```text
Analyze the following conversation between a customer and a real estate agent.
Extract any structured information about the customer and their property interests.

Rules:
- Only extract information explicitly stated or strongly implied in the conversation.
- Do not infer or guess values not supported by the text.
- Normalize phone numbers to E.164 format (e.g., +971501234567).
- Normalize budget values to numeric (e.g., "2M" → 2000000).
- Set confidence based on how explicit the information is (1.0 = directly stated, 0.5 = implied).
- Return null for person/lead sections if no relevant information found.
- Return empty array for propertyInterests/intentSignals if none detected.

Conversation:
{conversationMessages}

Respond with a JSON object matching the ExtractionResult schema.
```

## Async Queue Job Design

### Queue: `context-extraction`

<Note>
Extraction runs as an **async pg-boss job**, completely decoupled from the message processing pipeline. This ensures no latency impact on message persistence or delivery.
</Note>

Benefits of async processing:
- Extraction can take longer (10-30s) without affecting UX
- Failed extractions can be retried independently
- Extraction can be triggered on-demand or periodically

### Job Trigger Points

| Trigger | When | Payload |
|---------|------|---------|
| Message count threshold | Every N new messages in a conversation (default: 5) | `{ conversationId, organizationId }` |
| Conversation close | When status changes to CLOSED | `{ conversationId, organizationId }` |
| Manual trigger | Agent clicks "Extract info" button | `{ conversationId, organizationId, triggeredBy }` |
| Scheduled | Periodic scan of active conversations (daily) | `{ conversationId, organizationId }` |

### Job Processing Flow

<Steps>
<Step title="Load conversation data">
Load conversation + last N messages (default: 50)
</Step>

<Step title="Load existing CRM data">
Load existing Person + Lead data for delta detection
</Step>

<Step title="Compose extraction prompt">
Compose extraction prompt with conversation text
</Step>

<Step title="Call LLM">
Call LLM with JSON mode / function calling
</Step>

<Step title="Parse and validate">
Parse and validate ExtractionResult
</Step>

<Step title="Compute delta">
Compute delta (new data vs existing CRM data). Skip fields that already have values unless new value has higher confidence
</Step>

<Step title="Create review record">
Create ExtractionReview record (pending agent approval)
</Step>

<Step title="Notify agent">
Notify agent via WebSocket + optional push notification
</Step>
</Steps>

### Deduplication

<Warning>
Jobs are deduplicated by `conversationId` — if an extraction job is already queued or in progress for a conversation, new triggers are skipped. pg-boss `singletonKey: conversationId` ensures at most one active job per conversation.
</Warning>

### Retry Policy

- **Max retries:** 2
- **Backoff:** exponential (30s, 120s)
- **On final failure:** log error, mark extraction as `FAILED`, do not block conversation

<Steps>

Available actions:

</Step>
</Steps>

## Entity Update Logic

### Application Flow (After Approval)

When an agent approves an extraction (fully or partially), the service applies changes within a transaction:

```typescript
async applyExtraction(
  extractionReview: ExtractionReview,
  approvedFields: string[],
  em: EntityManager,
): Promise<void>
```

### Update Rules

<Tip>
The system follows conservative update rules to prevent data loss and maintain data integrity.
</Tip>

**Person Entity Updates:**
- **Create if missing:** If no Person exists for the conversation contact, create new Person
- **Update empty fields:** Only populate fields that are currently null/empty
- **Higher confidence wins:** If field already has value, only update if new value has confidence > 0.8
- **Email/Phone deduplication:** Check for existing Person with same email/phone before creating

**Lead Entity Updates:**
- **Create if missing:** If no Lead exists for the conversation, create new Lead
- **Budget updates:** Only update budget if new value is more specific (e.g., range → exact amount)
- **Timeline updates:** Update if new timeline is more specific (e.g., "soon" → "1-3 months")
- **Temperature escalation:** Can escalate temperature (cold → warm → hot) but not downgrade

**PropertyInterest Updates:**
- **Multiple interests:** Can create multiple PropertyInterest records per Lead
- **Merge similar:** If existing interest has same property type + area, merge details
- **Size/Budget ranges:** Expand existing ranges if new data suggests wider criteria

<Check>
All updates are logged in an audit trail for transparency and compliance.
</Check>

```typescript Error Recovery
// Handle partial extraction failures
if (!isValidExtractionResult(llmResponse)) {
  // Log validation errors
  logger.warn('Invalid LLM response', { conversationId, errors });
  
  // Attempt to salvage valid parts
  const partialResult = sanitizeExtractionResult(llmResponse);
  
  if (partialResult.confidence > 0.3) {
    // Proceed with partial extraction
    return partialResult;
  } else {
    // Mark as failed, retry later

  }
```

### Common Failure Scenarios

<AccordionGroup>
<Accordion title="LLM API failures">
- **Timeout:** Retry with exponential backoff
- **Rate limiting:** Queue for later processing
- **Invalid JSON:** Log error, mark extraction as failed
- **Cost limits:** Pause extractions, alert administrators
</Accordion>

<Accordion title="Data validation failures">
- **Invalid phone format:** Attempt normalization, fallback to raw text
- **Unrecognized property types:** Map to "Other" category
- **Invalid email format:** Skip email field, proceed with other data
- **Budget parsing errors:** Store as text note for manual review
</Accordion>

<Accordion title="Database transaction failures">
- **Constraint violations:** Handle duplicate emails/phones gracefully
- **Connection errors:** Retry transaction up to 3 times
- **Deadlocks:** Randomized retry delay to avoid collision
- **Storage limits:** Alert administrators, pause extractions
</Accordion>
</AccordionGroup>

## Performance Considerations

### Resource Optimization

- **Token usage:** Limit conversation history to last 50 messages (~8K tokens max)
- **Batch processing:** Group multiple conversations in single LLM request when possible
- **Caching:** Cache extraction results for 24 hours to avoid re-processing
- **Rate limiting:** Max 5 concurrent extractions per organization

### Monitoring Metrics

Key metrics to track:
- **Extraction latency:** P95 < 30 seconds
- **Success rate:** > 95% of jobs complete successfully
- **Accuracy rate:** Agent approval rate > 85%
- **Token cost:** Track monthly LLM API costs per organization
- **Queue depth:** Alert if queue length > 100 pending jobs

<Warning>
Monitor LLM costs closely as they can scale rapidly with conversation volume.
</Warning>

## Security & Privacy

### Data Handling

- **PII protection:** All extraction data encrypted at rest
- **Retention limits:** Extraction reviews auto-deleted after 90 days
- **Access controls:** Only conversation participants can view extractions
- **Audit logging:** All extraction approvals/rejections logged

### LLM Data Sharing

<Note>
Configure LLM providers to not use conversation data for model training. Use dedicated API keys with data processing agreements.
</Note>

- **OpenAI:** Enable zero data retention policy
- **Conversation anonymization:** Strip sensitive identifiers before LLM processing
- **Fallback processing:** Local extraction models for highly sensitive conversations
