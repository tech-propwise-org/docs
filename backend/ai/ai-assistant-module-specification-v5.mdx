## Access Scope Contract (Lever 1)

<Warning>
Breaking Change: The access scope contract has been centralized to reduce token overhead while maintaining full functionality.
</Warning>

### Centralized Scope Description

The `all`/`my`/`team` scope contract is now stated once in the `SYSTEM_PROMPT` "Access scope (`view` / `scope`) rules" block, referenced by all CRM tools through shared constants:

```typescript
// Shared constants in scope-describe.constants.ts
export const CRM_SCOPE_VIEW_DESCRIBE = "See system access scope rules";
export const CRM_SCOPE_TEAMID_DESCRIBE = "Team ID for team-scoped access";

export function crmScopeViewDescribe(myClause: string): string {
  return `${CRM_SCOPE_VIEW_DESCRIBE}. ${myClause}`;
}
```

## Intent Routing Improvements

### Signal-Based Routing

The intent router now treats active entity context as a signal rather than an override:

<Steps>
  <Step title="Keyword Analysis">
    Extract keyword families from the user message
  </Step>
  <Step title="Context Evaluation">
    Compare keyword families with active entity context:
    - No keywords → use entity context
    - One family agreeing with context → fast route
    - One family conflicting with context → defer to Tier B classifier
    - Multiple families → defer regardless of context
  </Step>
  <Step title="Tier B Classification">
    Use lite classifier for ambiguous cases requiring model analysis
  </Step>
</Steps>

### Example Scenarios

```typescript
// Previous behavior (incorrect)
// On lead detail page: "what's the off-plan market in Arjan?"
// → Force-routed to crm_records (excluded real-estate tools)

// New behavior (correct)  
// On lead detail page: "what's the off-plan market in Arjan?"
// → Detects real-estate keywords conflicting with CRM context
// → Routes to Tier B classifier → real_estate_market group
```

### Global Pre-Hydration

<Tip>
Mentions are now globally pre-hydrated to eliminate slow tool round-trips during conversations.
</Tip>

The system pre-hydrates up to 4 mentioned entities per turn:

```typescript
// Pre-hydration process
export async function buildGlobalContext(request: AiAssistantRequest) {
  const mentions = resolveClientContextWithMentions(request);
  
  for (const mention of mentions.slice(0, AI_ASSISTANT_MAX_HYDRATED_MENTIONS)) {
    const hydratedEntity = await delegateToEntityBuilder(mention);
    if (hydratedEntity) {
      context.mentionedEntities.push(hydratedEntity);
    }
  }
  
  return context;
}
```

### Mention Resolution

<Steps>
  <Step title="Wiring Fix">
    Top-level `request.mentions` are now properly merged with `clientContext.mentions`
  </Step>
  <Step title="Access Control">
    Each mention goes through the same access checks as detail scopes
  </Step>
  <Step title="Hydration">
    Accessible mentions are fully hydrated using existing entity builders
  </Step>
  <Step title="Fallback">
    Inaccessible mentions are preserved as bare references for tool fallback
  </Step>
</Steps>

## Market & Regulations Module

### Phase-Split Architecture

The market research system uses a two-phase approach for optimal performance:

<Tabs>
  <Tab title="Route Phase">
    **Lean Pre-Processing**
    - `MARKET_REGULATIONS_ROUTE_PROMPT`
    - Low reasoning model tier
    - Quick intent classification
  </Tab>
  <Tab title="Answer Phase">  
    **Full Research & Synthesis**
    - `MARKET_REGULATIONS_ANSWER_PROMPT`
    - Tier-default model
    - Comprehensive analysis
  </Tab>
</Tabs>

### Research Depth Control

Users can select research depth via the "Deep Research" toggle:

```typescript
export type ResearchDepth = 'fast' | 'deep';

// Fast: Limited sources, tight timeout, 1 call
// Deep: Larger source pool, up to 3 calls for synthesis
const maxCalls = resolveMaxCallsPerTurn(researchDepth);
```

### Source Ranking

Sources are ranked for quality and relevance:

<Steps>
  <Step title="Authority Ranking">
    UAE government/regulator sources → Press → Property portals
  </Step>
  <Step title="Recency Factor">
    More recent sources ranked higher within same authority tier
  </Step>
  <Step title="Relevance Scoring">
    Content relevance to query affects final ranking
  </Step>
</Steps>

interface ListPropertiesInPlaceInput {

## Validation & Hardening

### UUID Normalization

Task tools now handle placeholder UUIDs gracefully:

```typescript
// Drops RFC 9562 reserved sentinels
const normalizedId = normalizeOptionalUuid(assigneeId);
// '00000000-0000-0000-0000-000000000000' → undefined
// 'ffffffff-ffff-ffff-ffff-ffffffffffff' → undefined
```

### Unit Number Scope Collapse

Property searches automatically optimize scope hierarchy:

```typescript
// If unitNumber is specified with multiple location IDs
// Collapse to most specific scope (building > project > subcommunity > community > area)
if (input.unitNumber && input.buildingId) {
  // Drop redundant areaId, communityId, subcommunityId
  input = { ...input, areaId: undefined, communityId: undefined, subcommunityId: undefined };
}
```

### Valuation Scope Restriction  

<Warning>
AI unit valuations are available only for secondary (resale/ready) units, not off-plan properties.
</Warning>

The `getUnitEvaluation` tool:
- Explicitly scopes to secondary units in description
- Returns `valuationUnavailable` (503) for off-plan units
- Instructs model to explain limitation rather than fabricate values
