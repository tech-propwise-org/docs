## Overview

The AI Assistant Module provides a comprehensive conversational AI system for real estate CRM operations, market analytics, and property research. This specification covers system prompts, tool registry, context building, intent routing, and UAE real-estate integration.

## Recent Updates (v0.44+)

### Centralized Access Scope Contract

<Info>
**Version:** `crm-assistant-v0.44-centralized-access-scope`, Registry: `phase-1g-scope-describe-slim`
</Info>

The paragraph-length `all`/`my`/`team` scope contract that was previously restated on the `view`/`teamId` field of every `crm_records` list and aggregate tool is now stated **once** in the `SYSTEM_PROMPT` "Access scope (`view` / `scope`) rules" block.

#### Key Changes

<AccordionGroup>
<Accordion title="Scope Description Consolidation">
- **Previously:** Each tool (`searchLeads`, `searchDeals`, `searchContacts`, `searchCompanies`, `searchCommissionPayments`, aggregates, follow-up monitors) embedded the full scope contract in its description
- **Now:** Common contract maintained by `assembleSystemPrompt` for the `crm_records` group
- **Location:** Sits outside the real-estate anchors that `assembleSystemPrompt` strips
- **Per-tool reference:** Shared `CRM_SCOPE_VIEW_DESCRIBE` / `CRM_SCOPE_TEAMID_DESCRIBE` constants
- **Helper function:** `crmScopeViewDescribe(myClause)` in `src/modules/crm/shared/ai/scope-describe.constants.ts`
</Accordion>

<Accordion title="Token Reduction Impact">
- **Model input:** Receives `tool.description` + Zod `inputSchema` (`.describe()` → JSON Schema via `AiToolBridgeService.adaptTool`)
- **Net token cut:** ~10k characters removed across five providers
- **Diagnostic snapshot:** `tool-registry-snapshot.json` shows ~1.3% byte delta (understates actual model-facing reduction)
- **Type:** De-duplication + centralization, NOT truncation
</Accordion>

<Accordion title="Unchanged Behavior">
All functionality remains identical:
- `access_denied_to_view` recovery contract
- `appliedScope`/`scope` echo behavior
- Access checks and guards
- `BadRequestException("... view='team' requires teamId")` validation
- All executor logic byte-for-byte unchanged
- Parity eval-gated via `pnpm eval:differential`
</Accordion>
</AccordionGroup>

<Warning>
Task (`my`/`team`/`all` tier) and event (`my`/`team`/`org` calendar) `scope` parameters are intentionally **NOT** folded into this consolidation—they have distinct semantics, are already concise, and contain no raw permission keys.
</Warning>

### Security: No Permission Leak Gate

The `ai-tool-descriptions-no-permission-leak.spec.ts` AST gate is now **ACTIVE** (flipped from `describe.skip`):

<Steps>
<Step title="RBAC Key Removal">
Last raw RBAC keys in model-facing description strings eliminated:
- `team_crm.view` in lead pipeline/velocity tools → "team-level CRM access"
- Company glossary `system.admin / crm.view` → "CRM or admin access"
</Step>

<Step title="Automated Validation">
Gate scans all tool descriptions at test time to prevent permission key leakage to the model
</Step>
</Steps>

## Intent Router Improvements

### Active Entity as Signal (Not Override)

<Warning>
**Previous Behavior:** `AiIntentRouterService.resolveHeuristicGroup` (Tier A) returned the `clientContext.activeEntity.type` / `pageType` group immediately before inspecting the message, causing off-domain questions on CRM surfaces to be force-routed to `crm_records`.
</Warning>

**Example of previous issue:**
```
Question: "What's the off-plan market in Arjan?"
Context: Lead detail page open
Old routing: crm_records group (excludes all real-estate.* market tools)
```

#### Tier A Reordering

<Steps>
<Step title="No Message Keywords">
Fall back to active-entity group only—this is the single signal
- Example: "What do you know about this lead?" on lead surface → stays `crm_records`
</Step>

<Step title="Single Agreeing Family">
Exactly one keyword family that agrees with (or there is no) entity group → use that group
- Fast hit, no model call required
</Step>

<Step title="Single Conflicting Family">
Exactly one family that conflicts with entity group → abstain (`null`)
- Tier B's lite classifier re-routes based on message text
</Step>

<Step title="Multiple Families">
Two or more families (mixed) → abstain regardless of context
</Step>
</Steps>

<Check>
Tier B + Phase 3/4 narrowing logic unchanged. No prompt/registry version bump required (Tier A is a pre-prompt routing decision).
</Check>

**New unit tests cover:**
- Conflict cases
- Agreement cases
- No-keyword scenarios
- Mixed-on-entity cases

## Mention Hydration System

### Global Mention Pre-Hydration

<Warning>
**Issue Fixed:** `@LEAD-PRSE-018 What do you know about this lead?` took ~51s and answered "I don't have that lead's details"

**Root Cause:** Top-level `request.mentions` array was dropped—orchestrator only forwarded `request.clientContext` to `buildContext`, which only reads `clientContext.mentions`.
</Warning>

#### Two-Part Fix

<Tabs>
  <Tab title="1. Wiring Fix">
    **New Helper:** `resolveClientContextWithMentions(request)`
    
    ```typescript
    // Folds top-level request.mentions onto clientContext.mentions
    // De-duplicates by type:id
    // Runs before buildContext
    // Merged context persisted on metadata.clientContext
    ```
  </Tab>
  
  <Tab title="2. Global Pre-Hydration">
    **Service:** `AiContextBuilderService.buildGlobalContext`
    
    ```typescript
    // Pre-hydrates each accessible @-mention into
    // trustedContext.mentionedEntities
    
    // Delegates to same per-entity builders:
    - buildLeadContext
    - buildDealContext
    - buildContactContext
    - buildCompanyContext
    - buildAgencyCommissionContext
    ```
    
    **Benefits:**
    - Full entity projection inline in first pass
    - No tool round-trips needed
    - One-pass answer delivery
  </Tab>
</Tabs>

### Access Enforcement

<Steps>
<Step title="Delegated Builder Checks">
Access enforced by delegated builders (same logic as detail scopes)
</Step>

<Step title="Inaccessible Handling">
Inaccessible/not-found mentions:
- Omitted from hydrated context
- Counted but never disclosed
- Preserved as bare `{type,id}` reference for tool-call fallback
</Step>

<Step title="Hydration Limits">
Capped at `AI_ASSISTANT_MAX_HYDRATED_MENTIONS` (4) per turn + re-capped to `AI_ASSISTANT_CONTEXT_SNAPSHOT_MAX_BYTES`
</Step>
</Steps>

<Info>
**Scope:** GLOBAL-only today. Entity-detail scopes continue hydrating their active entity + listing extra mentions as bare references.

## Unit Valuation System

### Secondary-Only Scope

<Warning>
**Domain Correction:** AI unit valuations exist ONLY for SECONDARY (resale/ready) units, NOT off-plan/pre-launch.
</Warning>

#### Three-Part Implementation

<Tabs>
  <Tab title="1. Tool Description">
    `getUnitEvaluation` description now:
    - Scopes tool to secondary units explicitly
    - Instructs model for off-plan units: explain valuations only available for secondary
    - Instructs: do NOT call tool or fabricate values
    - Enforcement: "soft" (description + upstream `503`→`valuationUnavailable`)
    - No server-side `marketType` guard
  </Tab>
  
  <Tab title="2. Scope Collapse">
    `searchUnits.validateInput` collapses `unitNumber` POINT lookup to most-specific scope:
    
    **Hierarchy (most to least specific):**
    1. Building
    2. Project
    3. Subcommunity
    4. Community
    5. Area
    
    **Logic:**
    - Unit lives in ONE building
    - Redundant broader/sibling parent IDs dropped
    - Building ID alone = precise scope
    - Prevents over-constraint from guessed parent IDs
  </Tab>
  
  <Tab title="3. Eval Updates">
    **Labs Stub Changes:**
    - Returns `valuationUnavailable` (503) for OFF-PLAN Tiger units (9001–9003)
    - Returns real cached valuation only for SECONDARY fixtures (Samana Green unit 9201)
    
    **Test Cases:**
    - `re-unit-evaluation-worth-001`: Points at secondary unit 9201
    - `re-offplan-unit-valuation-declined-001`: Asserts assistant explains off-plan isn't valuated + surfaces NO fabricated value
    - Coverage intent notes: secondary-only
    
    **Results:** All three valuation eval cases pass 3/3
  </Tab>
</Tabs>

## Access Scope Rules

### CRM Records Group

The centralized access scope contract in `SYSTEM_PROMPT`:

<AccordionGroup>
<Accordion title="Scope: 'my' (Personal)">
**Filter behavior:**
- Returns only records where user is the owner/assignee
- No `teamId` parameter required or accepted
- Most restrictive scope

**Applied to:**
- Leads, Deals, Contacts, Companies
- Commission payments
- Follow-up monitors
- Needs-attention queries
</Accordion>

<Accordion title="Scope: 'team' (Team Level)">
**Filter behavior:**
- Returns records for specified team members
- **Requires** `teamId` parameter
- Throws `BadRequestException` if `teamId` omitted
- Permission check: user must have team-level CRM access

**Applied to:**
- All CRM record types
- Pipeline/velocity aggregates
- Repeated participant detection
- Duplicate analysis
</Accordion>

<Accordion title="Scope: 'all' (Organization Wide)">
**Filter behavior:**
- Returns all organization records user can access
- No team filtering applied
- Requires org-level CRM or admin access

**Applied to:**
- Cross-team analytics
- Organization-wide summaries
- Executive dashboards
</Accordion>
</AccordionGroup>

### Tasks & Events Scope

<Note>
Tasks and events use distinct `scope` semantics—intentionally NOT consolidated with CRM records scope contract.
</Note>

**Task scopes:**
- `my`: Personal tasks
- `team`: Team tasks  
- `all`: Organization tasks

**Event scopes:**
- `my`: Personal calendar
- `team`: Team calendar
- `org`: Organization calendar

## System Architecture

### Tool Registry

<Steps>
<Step title="Tool Adaptation">
`AiToolBridgeService.adaptTool` converts tools to model format:
- Extracts `tool.description`
- Converts Zod `inputSchema` to JSON Schema via `.describe()`
- Applies scope description references
</Step>

<Step title="Provider Organization">
Tools organized by provider:
- `CrmAiToolsProvider`: Lead, deal, contact, company operations
- `TaskAiToolsProvider`: Task management
- `EventAiToolsProvider`: Calendar operations
- `OffPlanAiToolsProvider`: Off-plan property catalogue
- `SecondaryMarketAiToolsProvider`: Secondary market catalogue
- `MarketAnalyticsAiToolsProvider`: Market data and trends
</Step>

<Step title="Group Exclusions">
Phase 3 `resolveExcludeNamesForGroup` filters tools by intent group
</Step>

<Step title="Prompt Splitting">
Phase 4 assembles/strips prompt blocks based on active entity context
</Step>
</Steps>

### Context Building

```typescript
// Context assembly pipeline
AiContextBuilderService.buildGlobalContext
  → resolveClientContextWithMentions  // Merge top-level mentions
  → Pre-hydrate accessible mentions    // Via entity builders
  → Cap at MAX_HYDRATED_MENTIONS       // 4 per turn
  → Re-cap to CONTEXT_SNAPSHOT_MAX_BYTES

// Entity-specific builders
buildLeadContext
buildDealContext  
buildContactContext
buildCompanyContext
buildAgencyCommissionContext
```

### Intent Resolution

<Steps>
<Step title="Tier A: Heuristic Routing">
**Fast-path resolution:**
1. Extract keyword families from message
2. Check active entity context
3. Apply reordered logic (see "Active Entity as Signal" above)
4. Return group or abstain (`null`)
</Step>

<Step title="Tier B: Lite Classifier">
**Fallback when Tier A abstains:**
- Runs lite model classification on message text
- Re-routes based on semantic analysis
- More expensive but handles ambiguous cases
</Step>

<Step title="Phase 3: Tool Filtering">
Apply `resolveExcludeNamesForGroup` to narrow available tools
</Step>

<Step title="Phase 4: Prompt Assembly">
Assemble appropriate prompt blocks, strip irrelevant sections
</Step>
</Steps>

### UUID Normalization

```typescript
export function normalizeOptionalUuid(uuid?: string): string | undefined {
  if (!uuid) return undefined
  
  // Drop RFC 9562 reserved sentinels
  if (uuid === '00000000-0000-0000-0000-000000000000') return undefined
  if (uuid === 'ffffffff-ffff-ffff-ffff-ffffffffffff') return undefined
  
  return uuid
}
```

## Best Practices

<Steps>
<Step title="Prompt Engineering">
- Centralize repeated contracts in system prompt
- Use scope references instead of inline descriptions
- Keep entity-specific semantics close to usage
- Strip irrelevant blocks for focused context
</Step>

<Step title="Tool Design">
- Server-authoritative for complex resolution logic
- Validate and normalize user/placeholder inputs
- Collapse redundant scope parameters
- Provide clear fallback instructions
</Step>

<Step title="Access Control">
- Enforce at delegated builder level
- Never disclose inaccessible entities
- Provide graceful degradation
- Count omissions for debugging
</Step>

<Step title="Performance">
- Use phase-split prompts for multi-step operations
- Implement per-step reasoning tier swaps
- Cache prompt prefixes where possible
- Monitor latency percentiles, not just means
</Step>

<Step title="Testing">
- Regenerate snapshots after tool changes
- Run differential evals for behavior parity
- Gate against permission leaks at AST level
- Cover conflict, agreement, and mixed scenarios
</Step>
</Steps>
