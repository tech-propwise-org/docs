## Overview

The Context Extraction Service automatically extracts structured CRM data from messaging conversations using LLM analysis. It runs asynchronously outside the message pipeline, analyzing conversation text to identify person details, lead qualification data, property interests, and intent signals. Extracted data is presented to agents for review before being committed to the CRM.

<Note>
This service is designed to run completely outside the messaging pipeline to ensure zero impact on message delivery performance.
</Note>

## Extraction targets

### Person details

Fields extracted and mapped to the `Person` entity:

| Field | Entity Field | Example Input | Extracted Value |
| --- | --- | --- | --- |
| First name | `Person.firstName` | "My name is Ahmed" | `Ahmed` |
| Last name | `Person.lastName` | "I'm Ahmed Al-Rashid" | `Al-Rashid` |
| Email | `PersonEmail.email` | "reach me at ahmed@example.com" | `ahmed@example.com` |
| Phone | `PersonPhone.phoneNumber` | "call me on 0501234567" | `+971501234567` |
| Preferred language | `Person.preferredLanguage` | (detected from message language) | `ar` / `en` |

## Structured output schema

### LLM output format

The extraction service uses **JSON mode** (OpenAI) or **function calling** to ensure structured output. The LLM returns a single JSON object:

```typescript
interface ExtractionResult {
  confidence: number; // 0.0 - 1.0 overall confidence
  person: {
    firstName?: string;
    lastName?: string;
    email?: string;
    phone?: string; // normalized to E.164 format
    preferredLanguage?: string; // ISO 639-1
  } | null;
  lead: {
    budget?: number;
    budgetCurrency?: string;
    timeline?: 'immediate' | '1-3 months' | '3-6 months' | '6+ months';
    intent?: 'buyer' | 'seller' | 'renter' | 'landlord' | 'investor';
    temperature?: 'cold' | 'warm' | 'hot';
    companyName?: string;
  } | null;
  propertyInterests: Array<{
    propertyType?: string;
    bedrooms?: number;
    budgetMin?: number;
    budgetMax?: number;
    sizeMin?: number;
    sizeMax?: number;
    preferredAreas?: string[];
    preferredCommunities?: string[];
    notes?: string;
  }>;
  intentSignals: Array<{
    type: 'viewing_request' | 'price_negotiation' | 'financing' | 'urgency' | 'comparison';
    detail: string; // brief description of what was detected
    confidence: number; // 0.0 - 1.0
  }>;
}
```

### Extraction prompt template

```
Analyze the following conversation between a customer and a real estate agent.
Extract any structured information about the customer and their property interests.

Rules:
- Only extract information explicitly stated or strongly implied in the conversation.
- Do not infer or guess values not supported by the text.
- Normalize phone numbers to E.164 format (e.g., +971501234567).
- Normalize budget values to numeric (e.g., "2M" → 2000000).
- Set confidence based on how explicit the information is (1.0 = directly stated, 0.5 = implied).
- Return null for person/lead sections if no relevant information found.
- Return empty array for propertyInterests/intentSignals if none detected.

Conversation:
{conversationMessages}

Respond with a JSON object matching the ExtractionResult schema.
```

<Info>
Extraction runs completely decoupled from the message processing pipeline to ensure no latency impact on message delivery.
</Info>

### Job trigger points

| Trigger | When | Payload |
| --- | --- | --- |
| Message count threshold | Every N new messages in a conversation (default: 5) | `{ conversationId, organizationId }` |
| Conversation close | When status changes to CLOSED | `{ conversationId, organizationId }` |
| Manual trigger | Agent clicks "Extract info" button | `{ conversationId, organizationId, triggeredBy }` |
| Scheduled | Periodic scan of active conversations (daily) | `{ conversationId, organizationId }` |

### Job processing flow

<Steps>
<Step title="Load conversation data">
Load conversation and last N messages (default: 50)
</Step>

<Step title="Load existing CRM data">
Load existing Person and Lead data for delta detection
</Step>

<Step title="Compose extraction prompt">
Create extraction prompt with conversation text
</Step>

<Step title="Call LLM">
Execute LLM request with JSON mode or function calling
</Step>

<Step title="Parse and validate">
Parse and validate the ExtractionResult response
</Step>

<Step title="Compute delta">
Calculate new data vs existing CRM data. Skip fields that already have values unless new value has higher confidence.
</Step>

<Step title="Create review record">
Create ExtractionReview record with status PENDING for agent approval
</Step>

<Step title="Notify agent">
Send notification via WebSocket and optional push notification
</Step>
</Steps>

### Deduplication and retry policy

<Tabs>
<Tab title="Deduplication">
- Jobs are deduplicated by `conversationId`
- If an extraction job is already queued or in progress for a conversation, new triggers are skipped
- pg-boss `singletonKey: conversationId` ensures at most one active job per conversation
</Tab>

<Tab title="Retry Policy">
- **Max retries**: 2
- **Backoff**: Exponential (30s, 120s)
- **Final failure**: Log error, mark extraction as `FAILED`, do not block conversation
</Tab>
</Tabs>

## Agent review and approval flow

### ExtractionReview entity

```typescript
@Entity({ tableName: 'extraction_review' })
class ExtractionReview {
  id: string; // UUID PK
  conversation: Conversation; // FK
  organization: Organization; // FK
  extractedData: ExtractionResult; // JSONB — full extraction output
  delta: Record<string, any>; // JSONB — only new/changed fields
  status: 'PENDING' | 'APPROVED' | 'REJECTED' | 'PARTIAL';
  reviewedBy?: User; // FK — agent who reviewed
  reviewedAt?: Date;
  appliedFields?: string[]; // which fields were accepted (for PARTIAL)
  rejectionReason?: string;
  createdAt: Date;
  updatedAt: Date;
  isDeleted: boolean;
}
```

### Review UI flow

<Steps>
<Step title="Notification">
Agent receives a WebSocket event `extraction-ready`:

```typescript
{
  conversationId: string;
  extractionReviewId: string;
  summary: string; // "Found: name, email, budget, 2 property interests"
  fieldCount: number;
}
```
</Step>

<Step title="Review panel">
Agent opens the extraction review panel in the conversation view showing:
- Each extracted field with current CRM value (if any) vs extracted value
- Confidence score per field (color-coded: green > 0.8, yellow > 0.5, red < 0.5)
- Checkboxes for selective approval (PARTIAL mode)
</Step>

<Step title="Agent actions">
Available actions:
- **Approve All**: Apply all extracted fields to CRM entities
- **Approve Selected**: Apply only checked fields (status = PARTIAL)
- **Reject**: Dismiss extraction (status = REJECTED)
- **Edit & Approve**: Agent modifies extracted values before applying
</Step>
</Steps>

## Entity update logic

### Application flow

When an agent approves an extraction (fully or partially), the service applies changes within a transaction:

```typescript
async applyExtraction(
  extractionReview: ExtractionReview,
  approvedFields: string[],
  em: EntityManager,
): Promise<void>
```

### Update rules

<Warning>
Fields are only updated if they don't already exist or if the new value has higher confidence than the existing value.
</Warning>

**Person Updates**:
- Create new Person if none exists for the conversation
- Add new email/phone records (do not overwrite existing)
- Update name fields only if currently empty
- Set preferred language only if detected with high confidence

**Lead Updates**:
- Create new Lead if none exists for the conversation
- Update budget only if new value is more specific or has higher confidence
- Timeline and temperature can be updated if new signal is stronger
- Company name updated only if currently empty

**Property Interest Updates**:
- Create new LeadPropertyInterest records for each property mentioned
- Merge similar interests (same type + overlapping criteria)
- Preserve agent-modified interests (check `isManuallyEdited` flag)

<Check>
All updates are logged in an audit trail for compliance and debugging purposes.
</Check>

### Confidence thresholds

| Field Type | Minimum Confidence | Auto-Apply Threshold |
| --- | --- | --- |
| Contact info (email, phone) | 0.7 | 0.9 |
| Names | 0.6 | 0.8 |
| Budget/Timeline | 0.5 | 0.7 |
| Property preferences | 0.4 | 0.6 |
| Intent signals | 0.3 | 0.5 |

<Tip>
Fields with confidence below the auto-apply threshold require agent review even in automated workflows.
</Tip>
