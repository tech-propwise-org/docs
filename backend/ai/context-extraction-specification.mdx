## Overview

The Context Extraction Service automatically extracts structured CRM data from messaging conversations using LLM analysis. It runs asynchronously outside the message pipeline, analyzing conversation text to identify person details, lead qualification data, property interests, and intent signals. Extracted data is presented to agents for review before being committed to the CRM.

<Note>
This service operates completely asynchronously to avoid any impact on message processing performance, with extraction times ranging from 10-30 seconds.
</Note>

## Extraction Targets

### Person Details

Fields extracted and mapped to the `Person` entity:

| Field | Entity Field | Example Input | Extracted Value |
|-------|-------------|---------------|-----------------|
| First name | `Person.firstName` | "My name is Ahmed" | `Ahmed` |
| Last name | `Person.lastName` | "I'm Ahmed Al-Rashid" | `Al-Rashid` |
| Email | `PersonEmail.email` | "reach me at ahmed@example.com" | `ahmed@example.com` |
| Phone | `PersonPhone.phoneNumber` | "call me on 0501234567" | `+971501234567` |
| Preferred language | `Person.preferredLanguage` | (detected from message language) | `ar` / `en` |

### Lead Qualification Details

Fields extracted and mapped to the `Lead` entity:

| Field | Entity Field | Example Input | Extracted Value |
|-------|-------------|---------------|-----------------|

| Company | `Lead.companyName` | "I work at Emaar" | `Emaar` |

### Property Interest Details

Fields extracted and mapped to `LeadPropertyInterest` entity:

| Field | Entity Field | Example Input | Extracted Value |
|-------|-------------|---------------|-----------------|
| Property type | `propertyType` | "looking for a villa" | `Villa` |
| Bedrooms | `bedrooms` | "3 bedroom apartment" | `3` |
| Budget range | `budgetMin` / `budgetMax` | "between 1.5M and 2M" | `1500000` / `2000000` |
| Size range | `sizeMin` / `sizeMax` | "at least 1500 sqft" | `1500` / `null` |
| Preferred areas | `preferredAreas` | "in Dubai Marina or JBR" | `["Dubai Marina", "JBR"]` |
| Communities | `preferredCommunities` | "Palm Jumeirah" | `["Palm Jumeirah"]` |
| Layout | `layoutType` | "open plan kitchen" | `OpenPlan` |
| Project reference | `refProjectId` | "Emaar Beachfront Tower 2" | (matched via project lookup) |

### Intent Signals

Higher-level signals that don't map directly to a single field but inform lead scoring and routing:

| Signal | Indicator | Action |
|--------|-----------|---------|
| High urgency | "need to move ASAP", "this week" | Set `temperature = hot`, `timeline = immediate` |
| Viewing request | "can I see the property?", "schedule a visit" | Create task for agent, flag for follow-up |
| Price negotiation | "is there any discount?", "best price?" | Flag as high-intent buyer |
| Comparison shopping | "what else do you have?", "other options" | Suggest similar properties |
| Financing inquiry | "mortgage", "payment plan", "installments" | Tag lead as `financing-needed` |
| Multiple interests | Multiple property types/areas mentioned | Create multiple `LeadPropertyInterest` records |

## Structured Output Schema

### LLM Output Format

The extraction service uses **JSON mode** (OpenAI) or **function calling** to ensure structured output. The LLM returns a single JSON object:

<CodeGroup>
```typescript ExtractionResult Interface
interface ExtractionResult {
  confidence: number; // 0.0 - 1.0 overall confidence
  person: {
    firstName?: string;
    lastName?: string;
    email?: string;
    phone?: string; // normalized to E.164 format
    preferredLanguage?: string; // ISO 639-1
  } | null;
  lead: {
    budget?: number;
    budgetCurrency?: string;
    timeline?: 'immediate' | '1-3 months' | '3-6 months' | '6+ months';
    intent?: 'buyer' | 'seller' | 'renter' | 'landlord' | 'investor';
    temperature?: 'cold' | 'warm' | 'hot';
    companyName?: string;
  } | null;
  propertyInterests: Array<{
    propertyType?: string;
    bedrooms?: number;
    budgetMin?: number;
    budgetMax?: number;
    sizeMin?: number;
    sizeMax?: number;
    preferredAreas?: string[];
    preferredCommunities?: string[];
    notes?: string;
  }>;
  intentSignals: Array<{
    type: 'viewing_request' | 'price_negotiation' | 'financing' | 'urgency' | 'comparison';
    detail: string; // brief description of what was detected
    confidence: number; // 0.0 - 1.0
  }>;
}
```
</CodeGroup>

### Extraction Prompt Template

```
Analyze the following conversation between a customer and a real estate agent.
Extract any structured information about the customer and their property interests.

Rules:
- Only extract information explicitly stated or strongly implied in the conversation.
- Do not infer or guess values not supported by the text.
- Normalize phone numbers to E.164 format (e.g., +971501234567).
- Normalize budget values to numeric (e.g., "2M" â†’ 2000000).
- Set confidence based on how explicit the information is (1.0 = directly stated, 0.5 = implied).
- Return null for person/lead sections if no relevant information found.
- Return empty array for propertyInterests/intentSignals if none detected.

Conversation:
{conversationMessages}

Respond with a JSON object matching the ExtractionResult schema.
```

## Async Queue Job Design

### Queue: `context-extraction`

Extraction runs as an **async pg-boss job**, completely decoupled from the message processing pipeline. This ensures:

<CardGroup cols={2}>
<Card title="No Latency Impact" icon="clock">
Message persistence and delivery remain unaffected by extraction processing
</Card>
<Card title="Extended Processing" icon="timer">
Extraction can take 10-30s without affecting user experience
</Card>
<Card title="Independent Retry" icon="rotate-right">
Failed extractions can be retried without impacting other operations
</Card>
<Card title="Flexible Triggers" icon="bolt">
Can be triggered on-demand or periodically as needed
</Card>
</CardGroup>

### Job Trigger Points

| Trigger | When | Payload |
|---------|------|---------|
| Message count threshold | Every N new messages in a conversation (default: 5) | `{ conversationId, organizationId }` |
| Conversation close | When status changes to CLOSED | `{ conversationId, organizationId }` |
| Manual trigger | Agent clicks "Extract info" button | `{ conversationId, organizationId, triggeredBy }` |
| Scheduled | Periodic scan of active conversations (daily) | `{ conversationId, organizationId }` |

### Job Processing Flow

<Steps>
<Step title="Load conversation data">
Load conversation + last N messages (default: 50)
</Step>
<Step title="Load existing CRM data">
Load existing Person + Lead data for delta detection
</Step>
<Step title="Compose extraction prompt">
Create prompt with conversation text
</Step>
<Step title="Call LLM">
Execute LLM with JSON mode / function calling
</Step>
<Step title="Parse and validate">
Parse and validate ExtractionResult
</Step>
<Step title="Compute delta">
Calculate new data vs existing CRM data. Skip fields that already have values unless new value has higher confidence.
</Step>
<Step title="Create review record">
Create ExtractionReview record pending agent approval
</Step>
<Step title="Notify agent">
Send notification via WebSocket + optional push notification
</Step>
</Steps>

### Deduplication and Retry

<Info>
**Deduplication:** Jobs are deduplicated by `conversationId` using pg-boss `singletonKey: conversationId` to ensure at most one active job per conversation.
</Info>

**Retry Policy:**
- Max retries: 2
- Backoff: exponential (30s, 120s)
- On final failure: log error, mark extraction as `FAILED`, do not block conversation

## Entity Update Logic

### Application Flow

When an agent approves an extraction (fully or partially), the service applies changes within a transaction:

<CodeGroup>
```typescript Application Function
async applyExtraction(
  extractionReview: ExtractionReview,
  approvedFields: string[],
  em: EntityManager,
): Promise<void>
```
</CodeGroup>

### Update Rules

<AccordionGroup>
<Accordion title="Person Updates">
**Person Entity:**
- Create new Person if conversation has no associated person
- Update existing Person fields only if:
  - Current field is null/empty, OR
  - New value has confidence â‰Ą 0.8 AND is different from current value
- Never overwrite manually-entered data (marked with `isManuallyVerified: true`)

**Email/Phone handling:**
- Add new PersonEmail/PersonPhone records (don't replace existing)
- Mark as `isPrimary: false` initially â€” agent can promote later
</Accordion>
<Accordion title="Lead Updates">
**Lead Entity:**
- Create new Lead if person has no active lead
- Update existing Lead fields using same confidence rules as Person
- Special handling for `budget`: only update if new value is within 20% of existing OR has confidence â‰Ą 0.9

**Temperature logic:**
- Only increase temperature (cold â†’ warm â†’ hot), never decrease
- Exception: if extraction confidence â‰Ą 0.9, allow any temperature change
</Accordion>
<Accordion title="Property Interest Updates">
**LeadPropertyInterest:**
- Always create new records (don't update existing interests)
- Deduplicate based on `propertyType` + `bedrooms` + `preferredAreas` combination
- If similar interest exists, merge budgetMin/Max ranges and append to preferredAreas
</Accordion>
</AccordionGroup>

### Transaction Safety

<Warning>
All entity updates are wrapped in a database transaction. If any update fails, the entire extraction application is rolled back and marked as `FAILED`.
</Warning>

<CodeGroup>
```typescript Transaction Wrapper
await em.transactional(async (trxEm) => {
  // Update Person
  await updatePersonFromExtraction(person, extractedPerson, trxEm);
  
  // Update/Create Lead
  await updateLeadFromExtraction(lead, extractedLead, trxEm);
  
  // Create Property Interests
  await createPropertyInterests(lead, extractedInterests, trxEm);
  
  // Mark extraction as applied
  extractionReview.status = 'APPROVED';
  extractionReview.appliedFields = approvedFields;
  await trxEm.persistAndFlush(extractionReview);
});
```
</CodeGroup>

### Optimization Strategies

<Tip>
**Context Window Management:** Limit conversation history to last 50 messages or 4000 tokens to stay within LLM context limits while maintaining extraction quality.
</Tip>

<Note>
**Batch Processing:** For high-volume periods, consider batching multiple conversations in a single LLM call to reduce API costs and latency.
