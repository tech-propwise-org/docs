## Overview

The Context Extraction Service automatically extracts structured CRM data from messaging conversations using LLM analysis. It runs asynchronously outside the message pipeline, analyzing conversation text to identify person details, lead qualification data, property interests, and intent signals. Extracted data is presented to agents for review before being committed to the CRM.

<Note>
  This service operates completely asynchronously to ensure zero latency impact on message delivery and processing.
</Note>

## Structured Output Schema

### LLM Output Format

The extraction service uses **JSON mode** (OpenAI) or **function calling** to ensure structured output. The LLM returns a single JSON object:

```typescript
interface ExtractionResult {
  confidence: number; // 0.0 - 1.0 overall confidence
  person: {
    firstName?: string;
    lastName?: string;
    email?: string;
    phone?: string; // normalized to E.164 format
    languages?: Array<{ 
      code: string; 
      proficiency: 'BASIC' | 'CONVERSATIONAL' | 'FLUENT' | 'NATIVE' 
    }>;
  } | null;
  lead: {
    budget?: number;
    budgetCurrency?: string;
    timeline?: 'immediate' | '1-3 months' | '3-6 months' | '6+ months';
    intent?: 'buyer' | 'seller' | 'renter' | 'landlord' | 'investor';
    temperature?: 'cold' | 'warm' | 'hot';
    companyName?: string;
  } | null;
  propertyInterest?: {
    inventoryUnitIds?: string[];
    bedroomsMin?: number;
    bedroomsMax?: number;
    budgetMin?: number;
    budgetMax?: number;
    sizeMin?: number;
    sizeMax?: number;
    wishedAreaIds?: number[];
    wishedAreaSnapshots?: Array<{ 
      id: number; 
      name?: string; 
      nameEn?: string; 
      nameAr?: string 
    }>;
    notes?: string;
  } | null;
  intentSignals: Array<{
    type: 'viewing_request' | 'price_negotiation' | 'financing' | 'urgency' | 'comparison';
    detail: string; // brief description of what was detected
    confidence: number; // 0.0 - 1.0
  }>;
}
```

### Extraction Prompt Template

<CodeGroup>

```text Prompt Template
Analyze the following conversation between a customer and a real estate agent.
Extract any structured information about the customer and their property interests.

Rules:
- Only extract information explicitly stated or strongly implied in the conversation.
- Do not infer or guess values not supported by the text.
- Normalize phone numbers to E.164 format (e.g., +971501234567).
- Normalize budget values to numeric (e.g., "2M" → 2000000).
- Set confidence based on how explicit the information is (1.0 = directly stated, 0.5 = implied).
- Return null for person/lead sections if no relevant information found.
- Return `propertyInterest: null` and an empty `intentSignals` array if none are detected.

Conversation:
{conversationMessages}

Respond with a JSON object matching the ExtractionResult schema.
```

</CodeGroup>

<Warning>
  The LLM should only extract information that is explicitly stated or strongly implied. Avoid inferring or guessing values not supported by the conversation text.
</Warning>

## Async Queue Job Design

### Queue: `context-extraction`

Extraction runs as an **async pg-boss job**, completely decoupled from the message processing pipeline. This ensures:

<CardGroup cols={2}>
  <Card title="No latency impact" icon="bolt">
    Message persistence and delivery remain fast
  </Card>
  <Card title="Extended processing time" icon="clock">
    Extraction can take 10-30s without affecting UX
  </Card>
  <Card title="Independent retry" icon="rotate">
    Failed extractions retry without blocking conversations
  </Card>
  <Card title="Flexible triggering" icon="wand-magic-sparkles">
    On-demand or periodic extraction available
  </Card>
</CardGroup>

### Job Trigger Points

| Trigger                 | When                                                | Payload                                           |
| ----------------------- | --------------------------------------------------- | ------------------------------------------------- |
| Message count threshold | Every N new messages in a conversation (default: 5) | `{ conversationId, organizationId }`              |
| Conversation close      | When status changes to CLOSED                       | `{ conversationId, organizationId }`              |
| Manual trigger          | Agent clicks "Extract info" button                  | `{ conversationId, organizationId, triggeredBy }` |
| Scheduled               | Periodic scan of active conversations (daily)       | `{ conversationId, organizationId }`              |

### Job Processing Flow

<Steps>
  <Step title="Load conversation data">
    Load conversation and last N messages (default: 50)
  </Step>

  <Step title="Load existing CRM data">
    Load existing Person and Lead data for delta detection
  </Step>

  <Step title="Compose extraction prompt">
    Build the prompt with conversation text
  </Step>

  <Step title="Call LLM">
    Execute LLM call with JSON mode or function calling
  </Step>

  <Step title="Parse and validate">
    Parse and validate the ExtractionResult
  </Step>

  <Step title="Compute delta">
    Calculate new data vs existing CRM data
    
    <Note>
      Skip fields that already have values unless new value has higher confidence
    </Note>
  </Step>

  <Step title="Create review record">
    Create ExtractionReview record pending agent approval
  </Step>

  <Step title="Notify agent">
    Send WebSocket notification and optional push notification
  </Step>
</Steps>

### Deduplication

<Info>
  Jobs are deduplicated by `conversationId` — if an extraction job is already queued or in progress for a conversation, new triggers are skipped.
</Info>

- pg-boss `singletonKey: conversationId` ensures at most one active job per conversation
- Prevents duplicate processing and wasted LLM API calls

### Retry Policy

| Parameter     | Value                   |
| ------------- | ----------------------- |
| Max retries   | 2                       |
| Backoff       | Exponential (30s, 120s) |
| Final failure | Log error, mark as `FAILED`, do not block conversation |

### ExtractionReview Entity

```typescript
@Entity({ tableName: 'extraction_review' })
class ExtractionReview {
  id: string; // UUID PK
  conversation: Conversation; // FK
  organization: Organization; // FK
  extractedData: ExtractionResult; // JSONB — full extraction output

  reviewedAt?: Date;
  appliedFields?: string[]; // which fields were accepted (for PARTIAL)
  rejectionReason?: string;
  createdAt: Date;
  updatedAt: Date;
  isDeleted: boolean;
}
```

## Entity Update Logic

### Application Flow (After Approval)

When an agent approves an extraction (fully or partially), the service applies changes within a transaction:

```typescript
async applyExtraction(
  extractionReview: ExtractionReview,
  approvedFields: string[],
  em: EntityManager,
): Promise<void>
```

### Update Rules

<AccordionGroup>
  <Accordion title="Person updates">
    **firstName / lastName:**
    - Only set if currently empty on Person
    - Never overwrite existing names
    
    **Email:**
    - Create new `PersonEmail` record if email doesn't exist
    - Set as primary if it's the first email
    - Skip if exact email already exists
    
    **Phone:**
    - Normalize to E.164 format before storing
    - Create new `PersonPhone` record if number doesn't exist
    - Set as primary if it's the first phone
    - Skip if exact number already exists
    
    **Languages:**
    - Add to `Person.languages` array if not already present
    - Update proficiency if new confidence is higher
  </Accordion>

  <Accordion title="Lead updates">
    **Budget / budgetCurrency:**
    - Only update if current value is null or new confidence > 0.8
    - Always update currency when updating budget
    
    **Timeline:**
    - Update if current timeline is null or less specific (e.g., "6+ months" → "1-3 months")
    
    **Intent:**
    - Only set if currently null
    - Never change existing intent automatically
    
    **Temperature:**
    - Update if new temperature is "hotter" than current (cold → warm → hot)
    - Preserve "hot" unless explicitly overridden
    
    **Company Name:**
    - Only set if currently empty
  </Accordion>

  <Accordion title="Property interest updates">
    **Inventory Units:**
    - Create or update `LeadPropertyInterest` record
    - Merge `inventoryUnitIds` with existing candidates
    - Remove duplicates
    
    **Search Criteria:**
    - Update `bedroomsMin/Max`, `budgetMin/Max`, `sizeMin/Max` if more specific than existing
    - Example: existing range [2, 4] + extracted value 3 → [3, 3]
    
    **Location Preferences:**
    - Merge `wishedAreaIds` with existing area preferences
    - Store `wishedAreaSnapshots` for display (names may change over time)
    
    **Notes:**
    - Append new notes to existing notes field with timestamp
  </Accordion>

  <Accordion title="Intent signals">
    **Viewing Requests:**
    - Create agent task: "Schedule property viewing"
    - Set task priority based on urgency signal
    
    **Price Negotiation:**
    - Add tag to lead: `high-intent-buyer`
    - Notify assigned agent
    
    **Financing Inquiry:**
    - Add tag to lead: `financing-needed`
    - Create follow-up task if mortgage partner integration exists
    
    **Urgency:**
    - Update lead temperature to `hot`
    - Set timeline to `immediate`
    - Trigger high-priority notification
    
    **Comparison Shopping:**
    - Create task: "Send similar properties"
    - Include extracted criteria in task description
  </Accordion>
</AccordionGroup>

### Transaction Guarantees

<Warning>
  All entity updates are wrapped in a database transaction. If any update fails, all changes are rolled back and the extraction review status is set to `FAILED`.
</Warning>

```typescript
await em.transactional(async (em) => {
  // Update Person
  if (approvedFields.includes('person')) {
    await updatePerson(extractionReview, em);
  }
  
  // Update Lead
  if (approvedFields.includes('lead')) {
    await updateLead(extractionReview, em);
  }
  
  // Update Property Interests
  if (approvedFields.includes('propertyInterest')) {
    await updatePropertyInterest(extractionReview, em);
  }
  
  // Process Intent Signals
  await processIntentSignals(extractionReview, em);
  
  // Mark review as applied
  extractionReview.status = 'APPROVED';
  extractionReview.appliedFields = approvedFields;
  extractionReview.reviewedAt = new Date();
  await em.persistAndFlush(extractionReview);
});
```

## Confidence Scoring

### Field-Level Confidence

Each extracted field has an associated confidence score (0.0 - 1.0):

| Score Range | Interpretation | UI Indicator | Auto-Apply Threshold |
| ----------- | -------------- | ------------ | -------------------- |
| 0.9 - 1.0   | Explicitly stated | Green ✓ | Yes (if enabled) |
| 0.7 - 0.89  | Strongly implied | Yellow ⚠ | Requires review |
| 0.5 - 0.69  | Moderately implied | Orange ⚠ | Requires review |
| 0.0 - 0.49  | Weakly implied | Red ✗ | No |

<Tip>
  Organizations can configure a confidence threshold for auto-approval. Fields above this threshold are automatically applied without agent review.
</Tip>

### Overall Extraction Confidence

The overall `confidence` score in `ExtractionResult` is calculated as:

```
overall_confidence = weighted_average([
  (field_confidence_1, field_importance_1),
  (field_confidence_2, field_importance_2),
  ...
])
```

**Field importance weights:**

- Email, Phone: 1.0 (critical contact info)
- Name, Budget, Intent: 0.8 (high value)
- Timeline, Temperature, Property details: 0.6 (medium value)
- Company, Secondary phones: 0.4 (nice to have)

## Error Handling

### LLM API Failures

<Tabs>
  <Tab title="Timeout">
    - Retry with exponential backoff
    - Max timeout: 30s
    - After 2 retries, mark extraction as `FAILED`
  </Tab>
  
  <Tab title="Rate Limit">
    - Implement token bucket rate limiter
    - Queue job for retry after rate limit reset
    - Log rate limit events for capacity planning
  </Tab>
  
  <Tab title="Invalid Response">
    - Log malformed JSON response
    - Retry once with explicit schema validation prompt
    - If still invalid, mark as `FAILED` with error details
  </Tab>
  
  <Tab title="Content Moderation">
    - If conversation flagged by LLM content filters
    - Skip extraction, log warning
    - Do not retry (preserve conversation)
  </Tab>
</Tabs>

### Schema Validation Failures

```typescript
try {
  const validated = ExtractionResultSchema.parse(llmResponse);
} catch (error) {
  if (error instanceof ZodError) {
    logger.error('Extraction schema validation failed', {
      conversationId,
      errors: error.errors,
      rawResponse: llmResponse,
    });
    
    // Mark extraction as failed with detailed error
    await markExtractionFailed(extractionReview, {
      reason: 'VALIDATION_ERROR',
      details: error.errors,
    });
  }
}
```

### Entity Update Failures

<Warning>
  If entity updates fail during application (after approval), the extraction review remains in `APPROVED` state but updates are not applied.
</Warning>

```typescript
try {
  await applyExtraction(extractionReview, approvedFields, em);
} catch (error) {
  logger.error('Failed to apply extraction', {
    extractionReviewId: extractionReview.id,
    error,
  });
  
  // Create alert for manual intervention
  await createAlert({
    type: 'EXTRACTION_APPLICATION_FAILED',
    extractionReviewId: extractionReview.id,
    error: error.message,
    requiresAction: true,
  });
}
```

## Performance Considerations

### Message Window

<Info>
  The extraction service analyzes the last **50 messages** by default to balance context and token usage.
</Info>

**Tuning considerations:**

- **More messages:** Better context, higher cost, longer processing time
- **Fewer messages:** Faster, cheaper, may miss early context
- Configurable per organization based on conversation patterns

### Token Usage Optimization

<Steps>
  <Step title="Message filtering">
    Exclude system messages and bot responses from extraction context
  </Step>
  
  <Step title="Deduplication">
    If conversation hasn't changed since last extraction, skip job
  </Step>
  
  <Step title="Incremental extraction">
    Store hash of analyzed messages; only include new messages in next extraction
  </Step>
  
  <Step title="Delta-only prompting">
    If existing CRM data is rich, prompt only asks for missing fields
  </Step>
</Steps>

### Caching Strategy

```typescript
// Cache extraction results by conversation content hash
const contentHash = hashConversationMessages(messages);
const cached = await cache.get(`extraction:${conversationId}:${contentHash}`);

if (cached) {
  return cached; // Skip LLM call
}

const result = await callLLM(prompt);
await cache.set(`extraction:${conversationId}:${contentHash}`, result, {
  ttl: 3600, // 1 hour
});
```

## Privacy & Compliance

### Data Retention

<Warning>
  Extraction results contain PII and must follow organization data retention policies.
</Warning>

- `ExtractionReview.extractedData` is automatically redacted after 90 days (configurable)
- Applied data in CRM entities follows normal CRM retention rules
- Rejected extractions are purged after 30 days

### GDPR Compliance

<Tabs>

  <Tab title="Consent">

### Approve Extraction

```http
PUT /messaging/extractions/:id/approve
```

**Request:**

```json
{
  "approveAll": true,
  "selectedFields": null,
  "modifications": null
}
```

**Partial approval:**

```json
{
  "approveAll": false,
  "selectedFields": ["person.firstName", "person.email", "lead.budget"],
  "modifications": {
    "person.email": "ahmed.corrected@example.com"
  }
}
```

**Response:**

```json
{
  "success": true,
  "appliedFields": ["person.firstName", "person.email", "lead.budget"],
  "updatedEntities": {
    "person": "01HQZPPP...",
    "lead": "01HQZLLL..."
  }
}
```

### Reject Extraction

```http
PUT /messaging/extractions/:id/reject
```

**Request:**

```json
{
  "reason": "Extracted information is incorrect"
}
```

**Response:**

```json
{
  "success": true,
  "status": "REJECTED"
}
```
