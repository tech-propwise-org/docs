## Overview

The AI Assistant module provides conversational AI capabilities within the PropWise CRM, enabling users to interact with an intelligent assistant that can access and manipulate CRM data through natural language. The module implements a chat interface with streaming responses, tool execution capabilities, and comprehensive usage tracking.

<Info>
**Current Status:** Active Development  
**System Prompt Version:** `crm-assistant-v0.14-readonly-draft-and-phrasing`  
**Last Updated:** 2026-05-23
</Info>

The assistant operates within conversation scopes (global, lead, deal, contact, etc.) and can execute read-only and draft tools to help users with CRM tasks. It includes rate limiting, cost tracking, and observability features.

The module follows a layered architecture with clear separation of concerns:

### Key Design Decisions

- **Provider-agnostic AI model interface** supporting multiple LLM providers
- **Streaming response architecture** using Server-Sent Events
- **Tool registry pattern** allowing other modules to contribute capabilities
- **Usage tracking and cost estimation** for budget management
- **In-memory rate limiting** for Phase 0 (single-instance deployment)

## API Endpoints

### POST /ai-assistant/chat

Start or continue a conversation with the AI assistant.

<CodeGroup>
```typescript Request
{
  message: string;           // User message (max 8000 chars)
  conversationId?: string;   // Existing conversation ID
  entityRef?: {             // Context entity reference
    type: AiConversationScopeType;
    id: string;
  };
  streamMode: boolean;      // Enable streaming response
}
```

```typescript Response
// Streaming: NDJSON chunks with assistant response
// Non-streaming: Complete response object
```
</CodeGroup>

### GET /ai-assistant/conversations

List user's conversations with pagination.

**Parameters:**
- `page`: number (default: 1)
- `limit`: number (default: 20)
- `scopeType`: AiConversationScopeType (optional)
- `scopeEntityId`: string (optional)

### DELETE /ai-assistant/conversations/:id

Delete a conversation and all associated messages.

### POST /ai-assistant/tools/run

Execute an AI tool directly (for testing/debugging).

<CodeGroup>
```typescript Request
{
  toolName: string;
  toolArgs: Record<string, any>;
  conversationId?: string;
}
```
</CodeGroup>

### GET /ai-assistant/mentions/search

Search for entities that can be mentioned in chat.

**Parameters:**
- `query`: string
- `scopeType`: AiConversationScopeType (optional)
- `limit`: number (default: 10)

## Business Rules

### Conversation Scope Rules

<Note>
- Global conversations have no entity context
- Scoped conversations are tied to specific CRM entities (leads, deals, contacts, etc.)
- Users can only access conversations within their organization
- Entity access is validated through the RBAC system
</Note>

### Message Limits

- User messages are capped at **8,000 characters**
- Conversation titles are truncated to **50 characters**
- Stream chunks are limited to **512 characters**

### Rate Limiting

<Warning>
Users are limited to a configurable number of messages per minute. Default limit prevents abuse during Phase 0 rollout. Implemented as in-memory sliding window for single-instance deployment.
</Warning>

### Tool Execution

- Only `read` and `draft` tools are enabled in Phase 0
- `write` and `sensitiveWrite` tools are reserved for future confirmation flow
- Tool access is validated through RBAC permissions
- Tool execution is time-bounded and logged

### Cost Control

- All AI provider calls are tracked for usage estimation
- Usage events record token counts and estimated costs
- Budget accounts accumulate usage per user and organization
- Phase 0 is observe-only with no enforcement

## Tool System

### Tool Registry Pattern

Tools are registered by implementing the `AiToolProvider` interface:

```typescript
interface AiToolProvider {
  getTools(): AiToolDefinition[];
}
```

<Tip>
Tools are automatically discovered and registered at module initialization. `AiAssistantModule` also registers `AiAssistantDomainToolsProvider`, an explicit aggregator that injects every domain tool provider and returns their combined tool list.
</Tip>

### Tool Definition Structure

```typescript
interface AiToolDefinition<TInput = unknown, TSummary = unknown> {
  name: string;
  description: string;
  // Zod schema; AI SDK uses it for both model-facing JSON Schema derivation
  // and runtime input validation.
  inputSchema: z.ZodTypeAny;
  inputSchemaHint?: Record<string, unknown>;
  // Defense-in-depth runtime validation
  validateInput: (raw: unknown) => TInput;
  accessMode: 'read' | 'draft' | 'write' | 'sensitiveWrite';
  requiresConfirmation: boolean;
  // Per-row resource gate enforced by AiToolExecutorService
  resourceCheck?: {
    type: ResourceType;
    action: ResourceAction;
    inputField: string;
  };
  // OR-mode permission bypass
  permissionBypass?: string[];
  requiredOrgPermissions?: string[];
  requiredTeamPermissions?: string[];
  contextScopes: AiToolContextScope[];
  // Scopes where the tool is hidden from the agentic loop
  excludeFromAgenticScopes?: AiConversationScopeType[];
  maxResults?: number;
  maxOutputBytes: number;
  auditCategory: string;
  execute: (input: TInput, context: AiToolExecutionContext) => Promise<AiToolResult<TSummary>>;
}
```

### Context Scopes

Tools declare `contextScopes: AiToolContextScope[]` — a free-form list of scopes. The registry's `listToolsForScope` then unions the conversation's scope with these:

| Conversation Scope | Tool contextScopes that Match |
|-------------------|-------------------------------|
| `LEAD` | `lead`, `global`, `real-estate`, `knowledge` |
| `DEAL` | `deal`, `global`, `real-estate`, `knowledge` |
| `CONTACT` | `contact`, `global`, `knowledge` |
| `COMPANY` | `company`, `contact`, `global`, `knowledge` |
| `CONVERSATION` | `messaging`, `global`, `knowledge` |
| `PROPERTY`/`PROJECT`/`BUILDING`/`UNIT` | `real-estate`, `global`, `knowledge` |
| `GLOBAL` | `global`, `knowledge`, `real-estate` |

### Pre-hydration & Agentic-Loop Exclusion

<Steps>
  <Step title="Context Building">
    When the user opens the assistant on an entity page (e.g. `/home/leads/:id`), the orchestrator calls `AiContextBuilderService.buildContext()` to pre-hydrate the entity into `compactContext.<scope>` BEFORE the model runs.
  </Step>
  
  <Step title="Lead Context Example">
    For `LEAD` scope, `buildLeadContext` projects the lead's id, publicId, name, stage, intent, budget, full property interest summary, primary stakeholder + up to 6 secondary stakeholders, assignedTeam, won/lost state, disqualification, commission resolution, tagIds, archive state, attribution, stageTiming, score, and `DealContact`-shaped `dealLinks[]`.
  </Step>
  
  <Step title="Tool Exclusion">
    Tools that would just re-fetch this data are excluded from the agentic loop via `excludeFromAgenticScopes`. Currently: `getLeadDetails` is hidden on `LEAD` scope, `getDealDetails` on `DEAL` scope, `getContactDetails` on `CONTACT` scope.
  </Step>
</Steps>

<Check>
This pre-hydration approach reduces redundant API calls and provides immediate context to the AI assistant while maintaining tool availability for cross-scope operations.
</Check>
