## Overview

The Context Extraction Service automatically extracts structured CRM data from messaging conversations using LLM analysis. It runs asynchronously outside the message pipeline, analyzing conversation text to identify person details, lead qualification data, property interests, and intent signals. Extracted data is presented to agents for review before being committed to the CRM.

<Note>
This service operates completely asynchronously to ensure zero latency impact on message processing and delivery.
</Note>

## Extraction Targets

### Person Details

Fields extracted and mapped to the `Person` entity:

| Field | Entity Field | Example Input | Extracted Value |
|-------|--------------|---------------|-----------------|
| First name | `Person.firstName` | "My name is Ahmed" | `Ahmed` |
| Last name | `Person.lastName` | "I'm Ahmed Al-Rashid" | `Al-Rashid` |
| Email | `PersonEmail.email` | "reach me at ahmed@example.com" | `ahmed@example.com` |
| Phone | `PersonPhone.phoneNumber` | "call me on 0501234567" | `+971501234567` |
| Primary language | `Person.languages[]` with `NATIVE` proficiency | (detected from message language) | `{ code: 'ar', proficiency: 'NATIVE' }` |

### Lead Qualification Details

Fields extracted and mapped to the `Lead` entity:

| Field | Entity Field | Example Input | Extracted Value |
|-------|--------------|---------------|-----------------|
| Budget | `Lead.budget` | "budget is around 2 million" | `2000000` |
| Budget currency | `Lead.budgetCurrency` | "2M AED" | `AED` |
| Timeline | `Lead.timeline` | "looking to move in 3 months" | `1-3 months` |
| Intent | `Lead.intent` | "I want to buy an apartment" | `buyer` |
| Temperature | `Lead.temperature` | (inferred from urgency signals) | `hot` / `warm` / `cold` |
| Company | `Lead.companyName` | "I work at Emaar" | `Emaar` |

<Info>
Intent is mapped to the `LeadIntent` enum based on specific keywords and phrases detected in the conversation.
</Info>

    |----------------|------------|
    | buy, purchase, own | `buyer` |
    | sell, list, dispose | `seller` |

    | lease out, find tenants, landlord | `landlord` |

### Property Interest Details

Fields extracted and mapped to `LeadPropertyInterest` entity:

| Field | Entity Field | Example Input | Extracted Value |
|-------|--------------|---------------|-----------------|
| Bedroom range | `bedroomsMin` / `bedroomsMax` | "3 bedroom apartment" | `3` / `3` |
| Budget range | `budgetMin` / `budgetMax` | "between 1.5M and 2M" | `1500000` / `2000000` |
| Size range | `sizeMin` / `sizeMax` | "at least 1500 sqft" | `1500` / `null` |
| Location refs | `wished*Ids` + snapshots | "in Dubai Marina or JBR" | matched Labs IDs + labels |
| Inventory units | `inventoryUnitIds` | "that unit in Marina" | selected inventory UUIDs |

### Intent Signals

Higher-level signals that don't map directly to a single field but inform lead scoring and routing:

| Signal | Indicator | Action |
|--------|-----------|--------|
| High urgency | "need to move ASAP", "this week" | Set `temperature = hot`, `timeline = immediate` |
| Viewing request | "can I see the property?", "schedule a visit" | Create task for agent, flag for follow-up |
| Price negotiation | "is there any discount?", "best price?" | Flag as high-intent buyer |
| Comparison shopping | "what else do you have?", "other options" | Suggest similar properties |
| Financing inquiry | "mortgage", "payment plan", "installments" | Tag lead as `financing-needed` |
| Multiple unit candidates | Multiple units match the same search | Add candidates to the single `LeadPropertyInterest` |

## Structured Output Schema

### LLM Output Format

The extraction service uses **JSON mode** (OpenAI) or **function calling** to ensure structured output. The LLM returns a single JSON object:

```typescript
interface ExtractionResult {
  confidence: number; // 0.0 - 1.0 overall confidence
  person: {
    firstName?: string;
    lastName?: string;
    email?: string;
    phone?: string; // normalized to E.164 format
    languages?: Array<{ 
      code: string; 
      proficiency: 'BASIC' | 'CONVERSATIONAL' | 'FLUENT' | 'NATIVE' 
    }>;
  } | null;
  lead: {
    budget?: number;
    budgetCurrency?: string;
    timeline?: 'immediate' | '1-3 months' | '3-6 months' | '6+ months';
    intent?: 'buyer' | 'seller' | 'renter' | 'landlord' | 'investor';
    temperature?: 'cold' | 'warm' | 'hot';
    companyName?: string;
  } | null;
  propertyInterest?: {
    inventoryUnitIds?: string[];
    bedroomsMin?: number;
    bedroomsMax?: number;
    budgetMin?: number;
    budgetMax?: number;
    sizeMin?: number;
    sizeMax?: number;
    wishedAreaIds?: number[];
    wishedAreaSnapshots?: Array<{ 
      id: number; 
      name?: string; 
      nameEn?: string; 
      nameAr?: string 
    }>;
    notes?: string;
  } | null;
  intentSignals: Array<{
    type: 'viewing_request' | 'price_negotiation' | 'financing' | 'urgency' | 'comparison';
    detail: string; // brief description of what was detected
    confidence: number; // 0.0 - 1.0
  }>;
}
```

### Extraction Prompt Template

```text
Analyze the following conversation between a customer and a real estate agent.
Extract any structured information about the customer and their property interests.

Rules:
- Only extract information explicitly stated or strongly implied in the conversation.
- Do not infer or guess values not supported by the text.
- Normalize phone numbers to E.164 format (e.g., +971501234567).
- Normalize budget values to numeric (e.g., "2M" → 2000000).
- Set confidence based on how explicit the information is (1.0 = directly stated, 0.5 = implied).
- Return null for person/lead sections if no relevant information found.
- Return `propertyInterest: null` and an empty `intentSignals` array if none are detected.

Conversation:
{conversationMessages}

Respond with a JSON object matching the ExtractionResult schema.
```

<Warning>
The LLM should only extract information that is explicitly stated or strongly implied. Avoid inferring or guessing values not supported by the conversation text.
</Warning>

## Async Queue Job Design

### Queue Configuration

**Queue Name:** `context-extraction`

Extraction runs as an **async pg-boss job**, completely decoupled from the message processing pipeline. This ensures:

<CardGroup cols={2}>
  <Card title="No Latency Impact" icon="gauge-high">
    Zero impact on message persistence or delivery performance
  </Card>
  <Card title="Extended Processing Time" icon="clock">
    Extraction can take 10-30s without affecting user experience
  </Card>
  <Card title="Independent Retry" icon="rotate-right">
    Failed extractions can be retried without blocking messages
  </Card>
  <Card title="Flexible Triggering" icon="bolt">
    Can be triggered on-demand, periodically, or automatically
  </Card>
</CardGroup>

### Job Trigger Points

| Trigger | When | Payload |
|---------|------|---------|
| Message count threshold | Every N new messages in a conversation (default: 5) | `{ conversationId, organizationId }` |
| Conversation close | When status changes to CLOSED | `{ conversationId, organizationId }` |
| Manual trigger | Agent clicks "Extract info" button | `{ conversationId, organizationId, triggeredBy }` |
| Scheduled | Periodic scan of active conversations (daily) | `{ conversationId, organizationId }` |

### Job Processing Flow

<Steps>
  <Step title="Load conversation data">
    Load conversation and last N messages (default: 50)
  </Step>
  
  <Step title="Load existing CRM data">
    Load existing Person + Lead data for delta detection
  </Step>
  
  <Step title="Compose extraction prompt">
    Compose extraction prompt with conversation text
  </Step>
  
  <Step title="Call LLM">
    Call LLM with JSON mode / function calling
  </Step>
  
  <Step title="Parse and validate">
    Parse and validate ExtractionResult
  </Step>
  
  <Step title="Compute delta">
    Compute delta (new data vs existing CRM data)
    
    <Note>
    Skip fields that already have values unless new value has higher confidence
    </Note>
  </Step>
  
  <Step title="Create review record">
    Create ExtractionReview record (pending agent approval)
  </Step>
  
  <Step title="Notify agent">
    Notify agent via WebSocket + optional push notification
  </Step>
</Steps>

### Deduplication Strategy

<Tip>
Jobs are deduplicated by `conversationId` using pg-boss `singletonKey`. If an extraction job is already queued or in progress for a conversation, new triggers are automatically skipped.
</Tip>

- **Deduplication key:** `conversationId`
- **Behavior:** At most one active job per conversation
- **Implementation:** pg-boss `singletonKey: conversationId`

### Retry Policy

| Parameter | Value |
|-----------|-------|
| Max retries | 2 |
| Backoff strategy | Exponential (30s, 120s) |
| Final failure action | Log error, mark extraction as `FAILED`, do not block conversation |

<Warning>
Failed extractions do not block conversation flow. The conversation remains fully functional even if extraction fails.
</Warning>

## Agent Review and Approval Flow

### ExtractionReview Entity

The `ExtractionReview` entity stores extraction results pending agent review:

```typescript
@Entity({ tableName: 'extraction_review' })
class ExtractionReview {
  id: string; // UUID PK
  conversation: Conversation; // FK
  organization: Organization; // FK
  extractedData: ExtractionResult; // JSONB — full extraction output
  delta: Record<string, any>; // JSONB — only new/changed fields
  status: 'PENDING' | 'APPROVED' | 'REJECTED' | 'PARTIAL';
  reviewedBy?: User; // FK — agent who reviewed
  reviewedAt?: Date;
  appliedFields?: string[]; // which fields were accepted (for PARTIAL)
  rejectionReason?: string;
  createdAt: Date;
  updatedAt: Date;
  isDeleted: boolean;
}
```

## Entity Update Logic

### Application Flow

When an agent approves an extraction (fully or partially), the service applies changes within a transaction:

```typescript
async applyExtraction(
  extractionReview: ExtractionReview,
  approvedFields: string[],
  em: EntityManager,
): Promise<void>
```

### Update Rules

<AccordionGroup>
  <Accordion title="Person Updates">
    **First Name / Last Name:**
    - Only set if currently empty on Person
    - Never overwrite existing names
    - Rationale: Names are critical identity fields that should not be auto-updated
    
    **Email:**
    - Create new `PersonEmail` record if email doesn't exist
    - Mark as `isPrimary: true` if it's the first email
    - Do not overwrite or delete existing emails
    
    **Phone:**
    - Create new `PersonPhone` record if phone doesn't exist
    - Normalize to E.164 format before storing
    - Mark as `isPrimary: true` if it's the first phone
    - Do not overwrite or delete existing phones
    
    **Languages:**
    - Merge with existing `Person.languages[]` array
    - Do not duplicate language codes
    - If detected language already exists, upgrade proficiency if new value is higher
  </Accordion>
  
  <Accordion title="Lead Updates">
    **Budget / Budget Currency:**
    - Update if currently null or if new confidence > 0.8
    - Always update both fields together to maintain consistency
    
    **Timeline:**
    - Update if currently null or if new value indicates higher urgency
    
    **Intent:**
    - Update if currently null
    - Do not overwrite existing intent (manual override takes precedence)
    
    **Temperature:**
    - Update based on latest intent signals and urgency indicators
    - Can be overwritten as it reflects current engagement level
    
    **Company Name:**
    - Update if currently null
    - Do not overwrite existing company (manual data takes precedence)
  </Accordion>
  
  <Accordion title="Property Interest Updates">
    **Bedroom / Budget / Size Ranges:**
    - Create new `LeadPropertyInterest` if none exists matching the criteria
    - Update existing interest if ranges overlap or are more specific
    
    **Location References:**
    - Add to `wishedAreaIds` and `wishedAreaSnapshots`
    - Merge with existing locations (do not duplicate)
    
    **Inventory Units:**
    - Add to `inventoryUnitIds` array
    - Create association if unit is already in system
    - Flag for agent review if unit reference is ambiguous
    
    **Multiple Candidates:**
    - Add all candidate units to the same `LeadPropertyInterest`
    - Store in priority order based on conversation context
  </Accordion>
  
  <Accordion title="Intent Signals Processing">
    Intent signals do not directly update entities but trigger actions:
    
    **Viewing Request:**
    - Create a task for the assigned agent
    - Set task type to `SCHEDULE_VIEWING`
    - Set priority to `HIGH` if urgency signal detected
    
    **Price Negotiation:**
    - Add tag `price-negotiation` to Lead
    - Increase lead temperature if not already `hot`
    
    **Financing Inquiry:**
    - Add tag `financing-needed` to Lead
    - Create task for agent to discuss financing options
    
    **Comparison Shopping:**
    - Flag conversation for property recommendation
    - Trigger similar property search if criteria available
    
    **High Urgency:**
    - Set lead temperature to `hot`
    - Set timeline to `immediate`
    - Create high-priority follow-up task
  </Accordion>
</AccordionGroup>

<Check>
All entity updates are performed within a database transaction to ensure data consistency. If any update fails, the entire extraction application is rolled back.
</Check>

### Transaction Safety

```typescript
await em.transactional(async (em) => {
  // Update Person
  if (approvedFields.includes('person.*')) {
    await updatePerson(person, extractedData.person, em);
  }
  
  // Update Lead
  if (approvedFields.includes('lead.*')) {
    await updateLead(lead, extractedData.lead, em);
  }
  
  // Update Property Interests
  if (approvedFields.includes('propertyInterest.*')) {
    await updatePropertyInterests(lead, extractedData.propertyInterest, em);
  }
  
  // Process Intent Signals
  await processIntentSignals(conversation, extractedData.intentSignals, em);
  
  // Mark extraction as applied
  extractionReview.status = 'APPROVED';
  extractionReview.reviewedBy = currentUser;
  extractionReview.reviewedAt = new Date();
  extractionReview.appliedFields = approvedFields;
  await em.persistAndFlush(extractionReview);
});
```

## Error Handling

### LLM API Errors

| Error Type | Handling Strategy |
|------------|-------------------|
| Rate limit (429) | Exponential backoff + retry with delay |
| Timeout | Retry up to MAX_RETRIES |
| Invalid response | Log error, mark extraction as FAILED |
| Content filter | Skip extraction, log conversation ID for review |

### Data Validation Errors

<Steps>
  <Step title="Schema validation">
    Validate LLM response against `ExtractionResult` schema using Zod or similar
  </Step>
  
  <Step title="Field normalization">
    Normalize phone numbers, emails, and budget values
    
    If normalization fails, log warning but continue with other fields
  </Step>
  
  <Step title="Confidence filtering">
    Filter out fields with confidence < 0.3 (too uncertain)
  </Step>
  
  <Step title="Partial success">
    If some fields are valid and others fail validation, save the valid fields and mark extraction as PARTIAL
  </Step>
</Steps>

### Transaction Rollback

If any step in the entity update transaction fails:

1. Roll back all changes
2. Mark extraction as `FAILED`
3. Log detailed error information
4. Notify agent with error message
5. Do not retry automatically (requires manual intervention)

## Security and Privacy

<Warning>
Extraction involves processing potentially sensitive customer information. Ensure compliance with data protection regulations.
</Warning>

### Data Handling

- **Conversation data**: Only load messages necessary for context (default: last 50)
- **PII extraction**: All extracted PII (email, phone) is encrypted at rest
- **LLM provider**: Use providers with GDPR compliance and data processing agreements
- **Data retention**: ExtractionReview records follow organization's retention policy

### Access Control

- Extraction jobs run in organization context (org ID in payload)
- Agents can only review extractions for conversations they have access to
- API endpoints enforce RBAC (role-based access control)
- Manual extraction trigger requires `TRIGGER_EXTRACTION` permission
