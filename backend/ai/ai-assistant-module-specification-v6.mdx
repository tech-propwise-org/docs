## Architecture Changes

### Centralized Access-Scope Contract

<Info>
The paragraph-length access scope descriptions previously embedded in every CRM tool have been centralized to reduce token usage by ~10k characters per turn.
</Info>

The `all`/`my`/`team` scope contract is now stated once in the `SYSTEM_PROMPT` **"Access scope (`view` / `scope`) rules"** block. Individual tools reference this shared contract via:

- `CRM_SCOPE_VIEW_DESCRIBE` constant
- `CRM_SCOPE_TEAMID_DESCRIBE` constant  
- `crmScopeViewDescribe(myClause)` helper

<CodeGroup>
```typescript src/modules/crm/shared/ai/scope-describe.constants.ts
export const CRM_SCOPE_VIEW_DESCRIBE = "References centralized access scope rules in system prompt";

export function crmScopeViewDescribe(myClause: string): string {
  return `${CRM_SCOPE_VIEW_DESCRIBE}. ${myClause}`;
}
```

```typescript Tool Implementation
// Before: Long embedded description
description: "Search leads with view parameter: 'all' for organization-wide access (requires admin), 'team' for team-level access (requires teamId), 'my' for personal leads only..."

// After: Centralized reference
description: crmScopeViewDescribe("Personal leads include those assigned to you")
```
</CodeGroup>

### Per-Tool Schema Optimization

Tools now receive optimized schemas through `AiToolBridgeService.adaptTool`:

- `tool.description` + Zod `inputSchema` via `.describe()` → JSON Schema
- Real per-turn token reduction (~10k characters)
- Diagnostic `tool-registry-snapshot.json` understates actual model-facing savings

## Intent Routing Improvements

### Active Entity Signal Enhancement

<Warning>
Previously, active entity context could override contradicting message intent. This has been reordered for better accuracy.
</Warning>

The `AiIntentRouterService.resolveHeuristicGroup` now processes in this order:

<Steps>
  <Step title="No Message Keywords">
    Fall back to active entity group (e.g., "what do you know about this lead?" on lead surface → `crm_records`)
  </Step>
  <Step title="Single Agreeing Family">
    Use that group if it agrees with or there's no entity context
  </Step>
  <Step title="Single Conflicting Family">
    Abstain (`null`) to trigger Tier B classifier re-routing
  </Step>
  <Step title="Mixed Families">
    Abstain regardless of context
  </Step>
</Steps>

## Enhanced Mention System

### Real Estate Entity Tagging

<Tip>
The mention picker now supports the same 8 real-estate entities available in global search.
</Tip>

**Supported Entity Types:**
- `inventory-unit` (indexed)
- `area` / `community` / `subcommunity` (federated)
- `secondary-project` / `secondary-building` (federated)
- `off-plan-project` / `off-plan-building` (federated)

### Mention Wiring & Hydration

<AccordionGroup>
  <Accordion title="Top-Level Mention Processing">
    The `resolveClientContextWithMentions(request)` helper merges top-level `request.mentions` with `clientContext.mentions`, de-duplicated by `type:id`.
  </Accordion>
  
  <Accordion title="Global Pre-Hydration">
    `AiContextBuilderService.buildGlobalContext` pre-hydrates accessible mentions using the same builders as detail scopes:
    - `buildLeadContext`
    - `buildDealContext` 
    - `buildContactContext`
    - `buildCompanyContext`
    - `buildAgencyCommissionContext`

  </Accordion>

## Tool Improvements

### Server-Authoritative Place Listings

<Warning>
The `listPropertiesInPlace` tool replaces client-side routing to prevent model misrouting of location IDs.
</Warning>

**New Tool: `listPropertiesInPlace`**
- Takes only place NAME (no IDs)
- Server resolves via `PropwiseLabsService.resolveLocationByName`
- Searches both off-plan AND secondary catalogues
- Returns combined prepared window (up to 18 items)

### Validation Hardening

<Steps>
  <Step title="UUID Normalization">
    `listMyTasks` now uses `normalizeOptionalUuid` to drop RFC 9562 reserved sentinel UUIDs (`00000000-…`, `ffffffff-…`)
  </Step>
  
  <Step title="Unit Scope Collapse">
    `searchUnits.validateInput` collapses `unitNumber` point lookups to most-specific scope (building > project > subcommunity > community > area)
  </Step>
  
  <Step title="Valuation Scope">
    `getUnitEvaluation` now explicitly scopes to secondary units only, with soft enforcement via description
  </Step>
</Steps>

## Security & Permissions

### Permission Leak Prevention

<Check>
The `ai-tool-descriptions-no-permission-leak.spec.ts` AST gate is now **ACTIVE** and enforces clean descriptions.
</Check>

**Changes Made:**
- Raw RBAC keys (`team_crm.view`) → descriptive text ("team-level CRM access")
- System permissions (`system.admin / crm.view`) → user-friendly descriptions ("CRM or admin access")

### Access Control

Real-estate mentions have no hydration builder and ride as bare `{ type, id, label }` references. The model resolves these via existing market-analytics tools, maintaining proper access controls.
