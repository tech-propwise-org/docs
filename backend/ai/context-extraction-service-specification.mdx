## Overview

The Context Extraction Service automatically extracts structured CRM data from messaging conversations using LLM analysis. It runs asynchronously outside the message pipeline, analyzing conversation text to identify person details, lead qualification data, property interests, and intent signals. Extracted data is presented to agents for review before being committed to the CRM.

<Note>
This service operates completely asynchronously to ensure zero latency impact on message processing and delivery.
</Note>

### Intent Signals

Higher-level signals that don't map directly to a single field but inform lead scoring and routing:

| Signal | Indicator | Action |
|--------|-----------|--------|
| High urgency | "need to move ASAP", "this week" | Set `temperature = hot`, `timeline = immediate` |
| Viewing request | "can I see the property?", "schedule a visit" | Create task for agent, flag for follow-up |
| Price negotiation | "is there any discount?", "best price?" | Flag as high-intent buyer |
| Comparison shopping | "what else do you have?", "other options" | Suggest similar properties |
| Financing inquiry | "mortgage", "payment plan", "installments" | Tag lead as `financing-needed` |
| Multiple unit candidates | Multiple units match the same search | Add candidates to the single `LeadPropertyInterest` |

## Structured Output Schema

### LLM Output Format

The extraction service uses **JSON mode** (OpenAI) or **function calling** to ensure structured output. The LLM returns a single JSON object:

```typescript
interface ExtractionResult {
  confidence: number; // 0.0 - 1.0 overall confidence
  person: {
    firstName?: string;
    lastName?: string;
    email?: string;
    phone?: string; // normalized to E.164 format
    languages?: Array<{ 
      code: string; 
      proficiency: 'BASIC' | 'CONVERSATIONAL' | 'FLUENT' | 'NATIVE' 
    }>;
  } | null;
  lead: {
    budget?: number;
    budgetCurrency?: string;
    timeline?: 'immediate' | '1-3 months' | '3-6 months' | '6+ months';
    intent?: 'buyer' | 'seller' | 'renter' | 'landlord' | 'investor';
    temperature?: 'cold' | 'warm' | 'hot';
    companyName?: string;
  } | null;
  propertyInterest?: {
    inventoryUnitIds?: string[];
    bedroomsMin?: number;
    bedroomsMax?: number;
    budgetMin?: number;
    budgetMax?: number;
    sizeMin?: number;
    sizeMax?: number;
    wishedAreaIds?: number[];
    wishedAreaSnapshots?: Array<{ 
      id: number; 
      name?: string; 
      nameEn?: string; 
      nameAr?: string 
    }>;
    notes?: string;
  } | null;
  intentSignals: Array<{
    type: 'viewing_request' | 'price_negotiation' | 'financing' | 'urgency' | 'comparison';
    detail: string; // brief description of what was detected
    confidence: number; // 0.0 - 1.0
  }>;
}
```

### Extraction Prompt Template

```text
Analyze the following conversation between a customer and a real estate agent.
Extract any structured information about the customer and their property interests.

Rules:
- Only extract information explicitly stated or strongly implied in the conversation.
- Do not infer or guess values not supported by the text.
- Normalize phone numbers to E.164 format (e.g., +971501234567).
- Normalize budget values to numeric (e.g., "2M" → 2000000).
- Set confidence based on how explicit the information is (1.0 = directly stated, 0.5 = implied).
- Return null for person/lead sections if no relevant information found.
- Return `propertyInterest: null` and an empty `intentSignals` array if none are detected.

Conversation:
{conversationMessages}

Respond with a JSON object matching the ExtractionResult schema.
```

## Async Queue Job Design

### Queue: `context-extraction`

Extraction runs as an **async pg-boss job**, completely decoupled from the message processing pipeline. This ensures:

- No latency impact on message persistence or delivery
- Extraction can take longer (10-30s) without affecting UX
- Failed extractions can be retried independently
- Extraction can be triggered on-demand or periodically

### Job Processing Flow

<Steps>
<Step title="Load conversation data">
Load conversation + last N messages (default: 50)
</Step>

<Step title="Load existing CRM data">
Load existing Person + Lead data (for delta detection)
</Step>

<Step title="Compose extraction prompt">
Compose extraction prompt with conversation text
</Step>

<Step title="Call LLM">
Call LLM with JSON mode / function calling
</Step>

<Step title="Parse and validate">
Parse and validate ExtractionResult
</Step>

<Step title="Compute delta">
Compute delta (new data vs existing CRM data)
Skip fields that already have values unless new value has higher confidence
</Step>

<Step title="Create review record">
Create ExtractionReview record (pending agent approval)
</Step>

<Step title="Notify agent">
Notify agent via WebSocket + optional push notification
</Step>
</Steps>

### Deduplication

<Info>
Jobs are deduplicated by `conversationId` — if an extraction job is already queued or in progress for a conversation, new triggers are skipped. pg-boss `singletonKey: conversationId` ensures at most one active job per conversation.
</Info>

### Retry Policy

- **Max retries:** 2
- **Backoff:** exponential (30s, 120s)
- **On final failure:** log error, mark extraction as `FAILED`, do not block conversation

## Agent Review/Approval Flow

### ExtractionReview Entity (Planned)

```typescript
@Entity({ tableName: 'extraction_review' })
class ExtractionReview {
  id: string; // UUID PK
  conversation: Conversation; // FK
  organization: Organization; // FK
  extractedData: ExtractionResult; // JSONB — full extraction output
  delta: Record<string, any>; // JSONB — only new/changed fields
  status: 'PENDING' | 'APPROVED' | 'REJECTED' | 'PARTIAL';
  reviewedBy?: User; // FK — agent who reviewed
  reviewedAt?: Date;
  appliedFields?: string[]; // which fields were accepted (for PARTIAL)
  rejectionReason?: string;
  createdAt: Date;
  updatedAt: Date;
  isDeleted: boolean;
}
```

## Entity Update Logic

### Application Flow (After Approval)

When an agent approves an extraction (fully or partially), the service applies changes within a transaction:

```typescript
async applyExtraction(
  extractionReview: ExtractionReview,
  approvedFields: string[],
  em: EntityManager,
): Promise<void>
```

### Update Rules

<Warning>
The following update rules ensure data integrity and prevent accidental overwrites of existing CRM data.
</Warning>

**Person updates:**

- `firstName` / `lastName`: Only set if currently empty on Person. Never overwrite existing names.
- `email` / `phone`: Add as new `PersonEmail` / `PersonPhone` records if not already present. Mark as `isPrimary: true` if no primary contact exists.
- `languages`: Add detected language if not already in the person's language list.

**Lead updates:**

- All lead fields can be updated if the extracted confidence > 0.7 OR the current field is empty.
- `temperature`: Can always be updated to a "hotter" level (`cold` → `warm` → `hot`) but never cooled down automatically.
- `budget` / `budgetCurrency`: Only update if new value is more specific (range → exact amount) or has higher confidence.

**PropertyInterest updates:**

- Create new `LeadPropertyInterest` record if none exists for this lead.
- If multiple property interests exist, update the most recent one or create a new one if the extracted interest is substantially different.
- `inventoryUnitIds`: Append to existing list, do not replace.
- Location preferences: Merge with existing `wishedAreaIds`, do not replace.

<Tip>
All updates maintain an audit trail through the `ExtractionReview` record, allowing agents to see what was extracted and when it was applied.
</Tip>
