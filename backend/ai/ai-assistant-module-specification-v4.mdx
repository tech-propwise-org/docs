## Recent Updates

### Placeholder-UUID Hardening

<Warning>
The `listMyTasks` function now includes enhanced UUID validation to prevent false access denial errors when models over-fill placeholder UUIDs.
</Warning>

**Key Changes:**
- `validateInput` now processes `assigneeId` and `teamId` through shared `normalizeOptionalUuid`
- RFC 9562 reserved nil (`00000000-…`) and max (`ffffffff-…`) sentinel UUIDs are dropped
- Field/tool descriptions now instruct models to resolve real IDs via `searchOrgUsers`/`searchTeams`

### Real-Estate Mention Tagging

The mention picker now supports the same 8 real-estate entities available in global search:

<AccordionGroup>
  <Accordion title="Supported Entity Types">
    - `inventory-unit` (indexed)
    - `area` (federated Propwise Labs)
    - `community` (federated Propwise Labs)
    - `subcommunity` (federated Propwise Labs)
    - `secondary-project` (federated Propwise Labs)
    - `secondary-building` (federated Propwise Labs)
    - `off-plan-project` (federated Propwise Labs)
    - `off-plan-building` (federated Propwise Labs)
  </Accordion>
</AccordionGroup>

**Implementation Details:**
- New `AI_MENTION_ENTITY_TYPES` drives the `searchMentions` allow-list
- `AiAssistantEntityRefDto.id` relaxed from `@IsUUID()` to non-empty string
- Labs IDs are numeric-as-string with optional `label` field
- Real-estate mentions have NO hydration builder and ride as bare `{ type, id, label }` references

### Mention Wiring & Global Hydration

<Steps>
  <Step title="Wiring Fix">
    New `resolveClientContextWithMentions(request)` helper folds top-level `request.mentions` onto `clientContext.mentions` before `buildContext`
  </Step>
  <Step title="Global Pre-hydration">
    `AiContextBuilderService.buildGlobalContext` now pre-hydrates each accessible @-mention into `trustedContext.mentionedEntities`
  </Step>
  <Step title="Access Control">
    Access enforcement by delegated builders with inaccessible mentions omitted and counted, never disclosed
  </Step>
  <Step title="Capacity Limits">
    Hydration capped at `AI_ASSISTANT_MAX_HYDRATED_MENTIONS` (4) per turn and re-capped to `AI_ASSISTANT_CONTEXT_SNAPSHOT_MAX_BYTES`
  </Step>
</Steps>

## Property Search Tools

### Server-Authoritative Place Listing

The new `listPropertiesInPlace` tool provides server-authoritative property searches to prevent model misrouting:

```typescript
// Tool signature
listPropertiesInPlace(placeName: string): {
  found: boolean;
  place: ResolvedPlace;
  total: number;
  counts: { offPlan: number; secondary: number };
  results: PropertyResult[];
}
```

<Info>
This tool takes ONLY the place NAME, resolves it server-side via `PropwiseLabsService.resolveLocationByName`, and searches both off-plan and secondary catalogues with the resolved ID.
</Info>

**Features:**
- Combined off-plan and secondary results (off-plan first, up to 18 items)
- Server-side place resolution prevents model misrouting
- Prepared pagination with "Show 6 / Show more" frontend cards

### Location Scope Inheritance

Same-turn location-scope inheritance prevents place name confusion:

<Tabs>
  <Tab title="Server Logic">
    - `resolvedLocationScope` memory tracks per-turn resolved places
    - `searchSecondaryListings`/`searchOffPlanDirectory`/`searchProperties` inherit scope when no explicit location ID provided
    - Overrides re-typed place names in query to prevent homonym confusion
  </Tab>
  <Tab title="Prompt Rules">
    - Copy `propertyScope` ID number VERBATIM
    - NEVER re-type/re-spell/re-resolve place NAME on listing calls
    - Includes Arjan-56 vs Ashjar-1 homonym example
  </Tab>
</Tabs>

### Misrouted Scope Override

Deterministic guards prevent explicit wrong IDs from bypassing validation:

1. **Resolved ID Tracking**: Records every ID `searchRealEstateLocations` returns per turn
2. **Override Logic**: When explicit directory location IDs weren't resolved this turn, override with single resolved place
3. **Multi-place Preservation**: Multiple resolved places ("Arjan vs JVC") keep each ID since both are in `resolvedIds`

## Unit Valuations

<Warning>
AI unit valuations are available ONLY for SECONDARY (resale/ready) units, NOT off-plan/pre-launch properties.
</Warning>

### Valuation Scope

The `getUnitEvaluation` tool includes specific scoping:

```typescript
// Tool behavior
if (marketType === 'off-plan') {
  return {
    status: 'valuationUnavailable',
    message: 'AI valuations are only available for secondary market units'
  };
}
```

**Key Features:**
- Secondary-only scope with clear model instructions
- Soft enforcement through description and upstream 503→valuationUnavailable
- Unit number point-lookup scope collapse to most-specific level

### Scope Collapse Logic

`searchUnits.validateInput` now collapses unit number POINT lookups:

<Steps>
  <Step title="Building Priority">
    A unit lives in ONE building, so building ID alone provides precise scope
  </Step>
  <Step title="Redundant ID Removal">
    Drops broader/sibling parent IDs that can over-constrain Labs lookup
  </Step>
  <Step title="Precision Targeting">
    Prevents weak models from padding real `buildingId` with guessed parent IDs
  </Step>
</Steps>

## Placeholder Input Hardening

### Plausible Bound Validation

New bounds validation prevents implausible placeholder values:

```typescript
const MIN_PLAUSIBLE_PRICE_AED = 1000;
const MIN_PLAUSIBLE_SIZE_SQFT = 10;

function plausiblePriceBound(value: number): boolean {
  return value >= MIN_PLAUSIBLE_PRICE_AED;
}

function plausibleSizeBound(value: number): boolean {
  return value >= MIN_PLAUSIBLE_SIZE_SQFT;
}
```

<Tip>
The `hasFabricatedBound` function treats implausible bounds as over-fill signals, helping detect when models generate placeholder values.
</Tip>

### UUID Normalization

The shared `normalizeOptionalUuid` function handles:
- RFC 9562 reserved nil UUID filtering
- Reserved max UUID filtering  
- Consistent optional UUID processing across tools
