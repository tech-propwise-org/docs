## Overview

The Context Extraction Service automatically extracts structured CRM data from messaging conversations using LLM analysis. It runs asynchronously outside the message pipeline, analyzing conversation text to identify person details, lead qualification data, property interests, and intent signals. Extracted data is presented to agents for review before being committed to the CRM.

<Note>
This service operates completely decoupled from the message processing pipeline to ensure zero latency impact on message delivery.
</Note>

## Structured Output Schema

### LLM Output Format

The extraction service uses **JSON mode** (OpenAI) or **function calling** to ensure structured output. The LLM returns a single JSON object:

```typescript
interface ExtractionResult {
  confidence: number; // 0.0 - 1.0 overall confidence
  person: {
    firstName?: string;
    lastName?: string;
    email?: string;
    phone?: string; // normalized to E.164 format
    languages?: Array<{ 
      code: string; 
      proficiency: 'BASIC' | 'CONVERSATIONAL' | 'FLUENT' | 'NATIVE' 
    }>;
  } | null;
  lead: {
    budget?: number;
    budgetCurrency?: string;
    timeline?: 'immediate' | '1-3 months' | '3-6 months' | '6+ months';
    intent?: 'buyer' | 'seller' | 'renter' | 'landlord' | 'investor';
    temperature?: 'cold' | 'warm' | 'hot';
    companyName?: string;
  } | null;
  propertyInterest?: {
    inventoryUnitIds?: string[];
    bedroomsMin?: number;
    bedroomsMax?: number;
    budgetMin?: number;
    budgetMax?: number;
    sizeMin?: number;
    sizeMax?: number;
    wishedAreaIds?: number[];
    wishedAreaSnapshots?: Array<{ 
      id: number; 
      name?: string; 
      nameEn?: string; 
      nameAr?: string 
    }>;
    notes?: string;
  } | null;
  intentSignals: Array<{
    type: 'viewing_request' | 'price_negotiation' | 'financing' | 'urgency' | 'comparison';
    detail: string; // brief description of what was detected
    confidence: number; // 0.0 - 1.0
  }>;
}
```

### Extraction Prompt Template

```text
Analyze the following conversation between a customer and a real estate agent.
Extract any structured information about the customer and their property interests.

Rules:
- Only extract information explicitly stated or strongly implied in the conversation.
- Do not infer or guess values not supported by the text.
- Normalize phone numbers to E.164 format (e.g., +971501234567).
- Normalize budget values to numeric (e.g., "2M" → 2000000).
- Set confidence based on how explicit the information is (1.0 = directly stated, 0.5 = implied).
- Return null for person/lead sections if no relevant information found.
- Return `propertyInterest: null` and an empty `intentSignals` array if none are detected.

Conversation:
{conversationMessages}

Respond with a JSON object matching the ExtractionResult schema.
```

<Warning>
The LLM should only extract information that is explicitly stated or strongly implied. Avoid inferring or guessing values not supported by the conversation text.
</Warning>

## Async Queue Job Design

### Queue: `context-extraction`

Extraction runs as an **async pg-boss job**, completely decoupled from the message processing pipeline. This ensures:

- No latency impact on message persistence or delivery
- Extraction can take longer (10-30s) without affecting UX
- Failed extractions can be retried independently
- Extraction can be triggered on-demand or periodically

### Job Trigger Points

| Trigger | When | Payload |
|---------|------|---------|
| Message count threshold | Every N new messages in a conversation (default: 5) | `{ conversationId, organizationId }` |
| Conversation close | When status changes to CLOSED | `{ conversationId, organizationId }` |
| Manual trigger | Agent clicks "Extract info" button | `{ conversationId, organizationId, triggeredBy }` |
| Scheduled | Periodic scan of active conversations (daily) | `{ conversationId, organizationId }` |

### Job Processing Flow

<Steps>
  <Step title="Load conversation data">
    Load conversation + last N messages (default: 50)
  </Step>
  
  <Step title="Load existing CRM data">
    Load existing Person + Lead data for delta detection
  </Step>
  
  <Step title="Compose extraction prompt">
    Compose extraction prompt with conversation text
  </Step>
  
  <Step title="Call LLM">
    Call LLM with JSON mode / function calling
  </Step>
  
  <Step title="Parse and validate">
    Parse and validate ExtractionResult
  </Step>
  
  <Step title="Compute delta">
    Compute delta (new data vs existing CRM data)
    
    Skip fields that already have values unless new value has higher confidence
  </Step>
  
  <Step title="Create review record">
    Create ExtractionReview record (pending agent approval)
  </Step>
  
  <Step title="Notify agent">
    Notify agent via WebSocket + optional push notification
  </Step>
</Steps>

### Deduplication

<Info>
Jobs are deduplicated by `conversationId` — if an extraction job is already queued or in progress for a conversation, new triggers are skipped.
</Info>

- pg-boss `singletonKey: conversationId` ensures at most one active job per conversation
- This prevents duplicate extractions from overwhelming the system

### Retry Policy

- **Max retries:** 2
- **Backoff:** exponential (30s, 120s)
- **On final failure:** log error, mark extraction as `FAILED`, do not block conversation

## Agent Review and Approval Flow

### ExtractionReview Entity

```typescript
@Entity({ tableName: 'extraction_review' })
class ExtractionReview {
  id: string; // UUID PK
  conversation: Conversation; // FK
  organization: Organization; // FK
  extractedData: ExtractionResult; // JSONB — full extraction output
  delta: Record<string, any>; // JSONB — only new/changed fields
  status: 'PENDING' | 'APPROVED' | 'REJECTED' | 'PARTIAL';
  reviewedBy?: User; // FK — agent who reviewed
  reviewedAt?: Date;
  appliedFields?: string[]; // which fields were accepted (for PARTIAL)
  rejectionReason?: string;
  createdAt: Date;
  updatedAt: Date;
  isDeleted: boolean;
}
```

### Review UI Flow

<Steps>
  <Step title="Notification">
    Agent receives a WebSocket event `extraction-ready`:

    ```typescript
    {
      conversationId: string;
      extractionReviewId: string;
      summary: string; // "Found: name, email, budget, 2 property interests"
      fieldCount: number;
    }
    ```
  </Step>

  <Step title="Review Panel">
    Agent opens the extraction review panel in the conversation view. Shows:
    
    - Each extracted field with current CRM value (if any) vs extracted value
    - Confidence score per field (color-coded: green > 0.8, yellow > 0.5, red < 0.5)
    - Checkboxes for selective approval (PARTIAL mode)
  </Step>

  <Step title="Actions">
    <Tabs>
      <Tab title="Approve All">
        Apply all extracted fields to CRM entities
      </Tab>
      <Tab title="Approve Selected">
        Apply only checked fields (status = PARTIAL)
      </Tab>
      <Tab title="Reject">
        Dismiss extraction (status = REJECTED)
      </Tab>
      <Tab title="Edit & Approve">
        Agent modifies extracted values before applying
      </Tab>
    </Tabs>
  </Step>
</Steps>

### API Endpoints

```typescript
// List extractions for conversation
GET /messaging/conversations/:id/extractions
```

## Entity Update Logic

### Application Flow

When an agent approves an extraction (fully or partially), the service applies changes within a transaction:

```typescript
async applyExtraction(
  extractionReview: ExtractionReview,
  approvedFields: string[],
  em: EntityManager,
): Promise<void>
```

### Update Rules

<AccordionGroup>
  <Accordion title="Person Updates">
    **First Name / Last Name:**
    - Only set if currently empty on Person
    - Never overwrite existing names
    
    **Email:**
    - Add new `PersonEmail` if email doesn't exist
    - Mark as `type: 'personal'`, `isPrimary: true` if no primary email exists
    - Skip if email already associated with Person
    
    **Phone:**
    - Normalize to E.164 format
    - Add new `PersonPhone` if number doesn't exist
    - Mark as `type: 'mobile'`, `isPrimary: true` if no primary phone exists
    - Skip if phone already associated with Person
    
    **Languages:**
    - Add to `Person.languages` array if not already present
    - Only add language with `NATIVE` proficiency if detected from conversation
  </Accordion>

  <Accordion title="Lead Updates">
    **Budget / Currency:**
    - Update `Lead.budget` and `Lead.budgetCurrency` if:
      - Lead currently has no budget, OR
      - Extracted value has confidence > 0.8 and is significantly different (>20% change)
    
    **Timeline:**
    - Update `Lead.timeline` if currently null or if new timeline is more urgent
    
    **Intent:**
    - Update `Lead.intent` if currently null
    - If Lead already has intent, only update if extracted intent has confidence > 0.9
    
    **Temperature:**
    - Update based on intent signals:
      - `hot`: urgent signals detected + viewing request + high budget
      - `warm`: specific requirements + active engagement
      - `cold`: general inquiry, no urgency
    - Never downgrade temperature (e.g., hot → warm)
    
    **Company Name:**
    - Update `Lead.companyName` if currently null
  </Accordion>

  <Accordion title="Property Interest Updates">
    **New Property Interest:**
    - Create new `LeadPropertyInterest` if none exists
    - If multiple interests exist, determine which to update based on:
      - Matching location (wishedAreaIds overlap)
      - Matching bedroom range
      - Most recently updated interest
    
    **Update Existing Interest:**
    - **Bedroom Range:** Update if currently null or if new range is more specific
    - **Budget Range:** Update if currently null or if new range narrows the existing range
    - **Size Range:** Update if currently null or if new range narrows the existing range
    - **Location References:**
      - Merge `wishedAreaIds` (union of existing + new)
      - Update `wishedAreaSnapshots` with latest labels from Labs API
    - **Inventory Units:**
      - Add to `inventoryUnitIds` array (no duplicates)
      - Maximum 10 units per interest
    
    **Notes:**
    - Append extracted notes to existing `notes` field with timestamp
  </Accordion>

  <Accordion title="Intent Signal Handling">
    **Viewing Request:**
    - Create Task for agent: "Schedule property viewing"
    - Set task due date based on urgency (same day if "ASAP", within 2 days otherwise)
    - Link task to conversation and lead
    
    **Price Negotiation:**
    - Add note to Lead: "Customer inquired about pricing/discounts"
    - Flag Lead for follow-up
    
    **Financing Inquiry:**
    - Add tag to Lead: `financing-needed`
    - Create Task: "Discuss financing options"
    
    **Urgency Signal:**
    - Update Lead temperature to `hot` if not already
    - Update Lead timeline to `immediate`
    
    **Comparison Shopping:**
    - Create Task: "Send similar property options"
    - Flag Lead as "comparison shopping"
  </Accordion>
</AccordionGroup>

<Tip>
All updates respect existing data and use confidence scores to determine whether to overwrite. This prevents low-quality extractions from corrupting good data.
</Tip>

## Error Handling and Edge Cases

### LLM Failures

<Steps>
  <Step title="Timeout">
    If LLM request times out (>30s):
    - Retry with exponential backoff
    - After max retries, mark extraction as `FAILED`
    - Log error with conversation context
  </Step>

  <Step title="Invalid JSON">
    If LLM returns invalid JSON:
    - Attempt to parse partial JSON
    - Extract any valid fields
    - Mark extraction as `PARTIAL` with lower confidence
  </Step>

  <Step title="Schema Validation Failure">
    If extracted data doesn't match schema:
    - Log validation errors
    - Accept valid fields, reject invalid ones
    - Mark extraction as `PARTIAL`
  </Step>
</Steps>

### Data Quality Issues

<Warning>
Low confidence extractions (confidence < 0.5) are flagged in the review UI with a warning indicator. Agents should carefully verify these fields before approval.
</Warning>

**Handling conflicts:**

1. **Duplicate Detection:**
   - If extracted email/phone matches different Person, flag for manual review
   - Do not auto-merge persons

2. **Data Inconsistency:**
   - If extracted budget currency conflicts with organization default, flag for review
   - If property location doesn't exist in Labs API, skip location update

3. **Ambiguous References:**
   - If "that property" could refer to multiple units, present all candidates for agent selection

### Performance Considerations

<Info>
The extraction service is designed to handle high volumes of conversations efficiently. However, monitor the following metrics:

- Average extraction time per conversation
- Queue depth for `context-extraction` jobs
- LLM API rate limits and costs
- Agent review completion rate
</Info>

**Optimization strategies:**

- Batch similar conversations for extraction
- Cache Labs API lookups for common locations
- Use smaller context windows (last 50 messages) to reduce token costs
- Implement rate limiting for LLM calls to avoid quota exhaustion

## Configuration and Feature Flags

### Organization-Level Settings

```typescript
{
  contextExtraction: {
    enabled: boolean; // default: true
    autoTriggerThreshold: number; // messages before auto-extraction, default: 5
    includeScheduledScans: boolean; // daily scan of active conversations, default: false
    minConfidenceThreshold: number; // minimum confidence to present for review, default: 0.3
    autoApplyHighConfidence: boolean; // auto-apply extractions with confidence > 0.95, default: false
    notifyAgentOnExtraction: boolean; // WebSocket + push notification, default: true
    llmProvider: 'openai' | 'anthropic'; // default: 'openai'
    llmModel: string; // default: 'gpt-4o-mini'
  }
}
```

### Feature Flags

| Flag | Description | Default |
|------|-------------|---------|
| `context-extraction-v2` | Enable new extraction service | `true` |
| `auto-apply-extractions` | Allow auto-approval for high-confidence extractions | `false` |
| `extraction-intent-signals` | Enable intent signal detection | `true` |
| `extraction-property-matching` | Enable inventory unit matching | `true` |
