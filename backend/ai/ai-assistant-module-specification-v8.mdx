# AI Assistant Module Specification v8

<Note>
**Last Updated:** 2026-06-07  
**Version:** Lever 1 - centralized access-scope contract + per-tool schema slimming
</Note>

This document provides a comprehensive specification for the AI Assistant Module, detailing the latest optimizations, routing improvements, and feature implementations.

## Overview

The AI Assistant Module serves as the core intelligence layer for the platform, providing:

- **Centralized Access Scope Contracts** - Unified permission handling across CRM tools
- **Schema Optimization** - Reduced token usage through intelligent de-duplication
- **Intent Routing** - Smart message classification and tool selection
- **Market Research** - Real estate analytics with configurable research depth
- **Mention System** - Entity tagging and pre-hydration capabilities

## Key Features

### Centralized Access Scope Contracts

<Info>
The access scope contract for CRM tools has been centralized to reduce token usage by approximately 10k characters per turn.
</Info>

#### Implementation Details

The paragraph-length scope contract that was previously restated on every CRM tool is now defined once in the `SYSTEM_PROMPT` **"Access scope (`view` / `scope`) rules"** block:

<CodeGroup>
```typescript filename="src/modules/crm/shared/ai/scope-describe.constants.ts"
export const CRM_SCOPE_VIEW_DESCRIBE = "..." // Centralized description
export const CRM_SCOPE_TEAMID_DESCRIBE = "..." // Centralized team ID description

export function crmScopeViewDescribe(myClause: string): string {
  return `${CRM_SCOPE_VIEW_DESCRIBE} ${myClause}`;
}
```
</CodeGroup>

#### Affected Tools

The following CRM tools now reference the centralized scope contract:

- `searchLeads` / `searchDeals` / `searchContacts` / `searchCompanies`
- `searchCommissionPayments`
- Summary and velocity aggregates (`*Summary` / `*Velocity`)
- Pipeline and duplicate detection tools
- Follow-up and needs-attention monitors
- `getMyTopAreas` / `getMyTopCommunities`

<Check>
All access checks, `BadRequestException` guards, and executor logic remain unchanged - this is purely a documentation de-duplication effort.
</Check>

### Intent Routing Improvements

#### Active Entity Signal Handling

<Warning>
Previous behavior: Active entity context would override message content, leading to incorrect tool routing.
</Warning>

The intent router now properly handles active entity signals:

<Steps>
<Step title="No Message Keywords">
Fall back to active-entity group (e.g., "what do you know about this lead?" on lead surface stays `crm_records`)
</Step>

<Step title="Single Keyword Family - Agreement">
Use the keyword family group when it agrees with or there's no entity group
</Step>

<Step title="Single Keyword Family - Conflict">
Abstain and let Tier B's lite classifier re-route based on message text
</Step>

<Step title="Multiple Keyword Families">
Abstain regardless of context to avoid conflicts
</Step>
</Steps>

#### Router Configuration

<CodeGroup>
```typescript filename="AiIntentRouterService.resolveHeuristicGroup"
// Tier A reordering - entity context never overrides contradicting message
if (noMessageKeywords) {
  return activeEntityGroup; // Fast path
}

if (exactlyOneFamilyAgreesOrNoEntity) {
  return keywordFamilyGroup; // Fast hit
}

if (exactlyOneFamilyConflicts) {
  return null; // Abstain - let Tier B handle
}

return null; // Mixed families - abstain
```
</Typescript>
</CodeGroup>

#### Mention Wiring & Global Pre-hydration

<Steps>
<Step title="Request Processing">
Top-level `request.mentions` array is merged with `clientContext.mentions` using `resolveClientContextWithMentions(request)`
</Step>

<Step title="Access Enforcement">
Each mention is validated through entity-specific builders:
- `buildLeadContext`
- `buildDealContext` 
- `buildContactContext`
- `buildCompanyContext`
- `buildAgencyCommissionContext`
</Step>

<Step title="Pre-hydration">
Accessible mentions are pre-hydrated into `trustedContext.mentionedEntities` for immediate model access
</Step>

<Step title="Fallback Handling">
Inaccessible mentions are preserved as bare `{type, id}` references for tool-call fallback
</Step>
</Steps>

### Task Management Hardening

#### UUID Validation

<Warning>
Previous issue: Model-generated placeholder UUIDs were causing false permission errors.
</Warning>

The `listMyTasks` tool now includes hardened UUID validation:

<CodeGroup>
```typescript filename="validateInput"
// Normalize optional UUIDs, dropping RFC 9562 reserved values
assigneeId = normalizeOptionalUuid(assigneeId);
teamId = normalizeOptionalUuid(teamId);

// Reserved UUIDs that are dropped:
// - nil: 00000000-0000-0000-0000-000000000000
// - max: ffffffff-ffff-ffff-ffff-ffffffffffff
```
</CodeGroup>

### Property Catalogue Tools

#### Server-Authoritative Place Listing

<Info>
The `listPropertiesInPlace` tool eliminates model misrouting by handling location resolution server-side.
</Info>

**Previous Issues:**
- Model misrouting resolved IDs (`developerId` → `subcommunityId`)
- Wrong-level ID conflicts (e.g., `subcommunityId:56` JVC vs correct ID)
- Boundary smearing in dual-search operations

**Solution:**
New `listPropertiesInPlace` tool that:

<Steps>
<Step title="Server Resolution">
Takes only the place NAME, resolves via `PropwiseLabsService.resolveLocationByName`
</Step>

<Step title="Dual Catalogue Search">
Searches both off-plan AND secondary catalogues with resolved ID scope
</Step>

<Step title="Prepared Results">
Returns combined window (off-plan first, then secondary) up to 18 items with metadata
</Step>
</Steps>

### Unit Valuations

#### Secondary-Only Scope

<Warning>
AI unit valuations are only available for SECONDARY (resale/ready) units, NOT off-plan/pre-launch properties.
</Warning>

#### Implementation Changes

<Steps>
<Step title="Tool Description Update">
`getUnitEvaluation` now explicitly scopes to secondary units and instructs models to explain limitations for off-plan requests
</Step>

<Step title="Scope Collapse">
`searchUnits.validateInput` collapses `unitNumber` point lookups to most-specific scope (building > project > subcommunity > community > area)
</Step>

<Step title="Evaluation Service">
Returns `valuationUnavailable` (503) for off-plan units, cached valuations only for secondary fixtures
</Step>
</Steps>
