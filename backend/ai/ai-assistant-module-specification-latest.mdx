## Intent Routing System

### Routing Hierarchy

<Steps>
  <Step title="Tier A - Heuristic Analysis">
    Analyzes message keywords and active entity context to determine tool group
  </Step>
  <Step title="Tier B - Model Classification">
    Uses lightweight model classification when heuristics are inconclusive
  </Step>
  <Step title="Phase 3 - Tool Exclusion">
    Applies group-specific tool filtering based on resolved intent
  </Step>
  <Step title="Phase 4 - Prompt Assembly">
    Constructs context-specific system prompts with relevant tool schemas
  </Step>
</Steps>

### Entity Context Priority

The routing system now treats active entity context as a **signal rather than override**:

<CodeGroup>
```typescript Active Entity Resolution
// Priority order for conflicting signals
if (noMessageKeywords) {
  return activeEntityGroup; // Fall back to context
}
if (singleKeywordFamily && agreesWithContext) {
  return keywordGroup; // Fast path
}
if (singleKeywordFamily && conflictsWithContext) {
  return null; // Defer to Tier B classification
}
if (multipleKeywordFamilies) {
  return null; // Always defer when mixed signals
}
```
</CodeGroup>

## Context Building System

### Mention Hydration

<Warning>
Global mention pre-hydration is now active to eliminate slow tool round-trips
</Warning>

The system pre-hydrates `@`-mentioned entities into the conversation context:

<Steps>
  <Step title="Mention Resolution">
    Merge top-level `request.mentions` with `clientContext.mentions`
  </Step>
  <Step title="Access Validation">
    Validate user permissions for each mentioned entity
  </Step>
  <Step title="Context Hydration">
    Pre-load full entity details using existing context builders
  </Step>
  <Step title="Context Assembly">
    Include hydrated entities in `trustedContext.mentionedEntities`
  </Step>
</Steps>

## Property Catalog Integration

### Server-Authoritative Listing

<Check>
New `listPropertiesInPlace` tool eliminates model routing errors
</Check>

Instead of trusting the model to compose location resolution and searches, the system now provides a server-authoritative tool:

<Steps>
  <Step title="Name Resolution">
    Server resolves place name via `PropwiseLabsService.resolveLocationByName`
  </Step>
  <Step title="Dual Catalog Search">
    Searches both off-plan and secondary catalogs with resolved location
  </Step>
  <Step title="Unified Results">
    Returns combined, prepared result window (up to 18 items)
  </Step>
  <Step title="Frontend Pagination">
    Frontend shows 6 cards initially with "Show more" for additional sets
  </Step>
</Steps>

### Valuation Scope Limitation

<Warning>
AI unit valuations are only available for secondary (resale/ready) units, not off-plan properties
</Warning>

The `getUnitEvaluation` tool now:
- Explicitly scopes to secondary units only
- Returns `valuationUnavailable` for off-plan requests
- Instructs models not to fabricate valuation data

## Input Validation & Hardening

### UUID Normalization

<Tip>
The system now normalizes placeholder UUIDs to prevent false access errors
</Tip>

```typescript
// Drops RFC 9562 reserved sentinel UUIDs
const normalizeOptionalUuid = (uuid: string) => {
  if (uuid === '00000000-0000-0000-0000-000000000000' || 
      uuid === 'ffffffff-ffff-ffff-ffff-ffffffffffff') {
    return undefined;
  }
  return uuid;
};
```

### Unit Search Scope Collapse

For `unitNumber` point lookups, the system collapses to the most specific scope:

- Building ID alone is sufficient (most specific)
- Redundant parent IDs are dropped to prevent over-constraining
- Prevents model padding errors from breaking searches

## Error Handling & Recovery

### Access Denied Recovery

The system maintains robust error recovery for access control violations:

<Steps>
  <Step title="Permission Check">
    Validate user access to requested data scope
  </Step>
  <Step title="Graceful Degradation">
    Return appropriate error messages without exposing restricted data
  </Step>
  <Step title="Alternative Suggestions">
    Guide users toward accessible data scopes when possible
  </Step>
</Steps>

### Validation Gates

<Note>
The `ai-tool-descriptions-no-permission-leak.spec.ts` AST gate is now active
</Note>

This prevents raw RBAC keys from appearing in model-facing descriptions, ensuring security through:
- Automated AST analysis of tool descriptions
- Conversion of technical permission keys to user-friendly terms
- Continuous integration enforcement
