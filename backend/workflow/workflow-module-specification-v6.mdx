## Overview

The Workflow module provides a comprehensive automation engine for PropWise CRM, enabling organizations to create, manage, and execute automated workflows triggered by various events. The module supports complex multi-step processes including messaging, data manipulation, external integrations, conditional logic, and AI agent integration.

### Architecture Model

The workflow system operates on a **graph-based execution model** where workflows are defined as directed graphs of nodes with per-step connections. Workflows can be triggered by messaging/CRM events or webhooks.

<Info>
**Canonical Persisted Shape (2025+):**  
`Workflow.graph = { nodes: WorkflowStepDefinition[], edges: WorkflowEdge[] }`  

With `AutomationScope`: **`ORGANIZATION | PERSONAL`**  
And `WorkflowStatus`: **`DRAFT | PUBLISHED | ARCHIVED`**
</Info>

<Warning>
Legacy documentation referencing `steps[]` + `connections[]` pairings, **`GLOBAL` scope**, `isActive`, or **activate/deactivate** APIs is obsolete.
</Warning>

## Workflow Entities

### Workflow Entity

The core workflow definition entity with graph-based structure.

```typescript
interface Workflow {
  id: string;
  organizationId: string;
  name: string;
  description?: string;
  scope: AutomationScope; // ORGANIZATION | PERSONAL
  status: WorkflowStatus; // DRAFT | PUBLISHED | ARCHIVED
  graph: {
    nodes: WorkflowStepDefinition[];
    edges: WorkflowEdge[];
  };
  createdBy: string;
  createdAt: Date;
  updatedAt: Date;
}
```

<Tip>
Use `ORGANIZATION` scope for team-wide automations and `PERSONAL` scope for user-specific workflows.
</Tip>

### WorkflowStepDefinition

Defines individual workflow steps with configuration and connections.

```typescript
interface WorkflowStepDefinition {
  id: string;
  type: WorkflowStepType;
  label: string;
  config: Record<string, any>;
  position: { x: number; y: number };
  connections: WorkflowConnection[];
}
```

### WorkflowEdge

Defines connections between workflow steps.

```typescript
interface WorkflowEdge {
  id: string;
  source: string;
  target: string;
  label?: string;
  type?: 'default' | 'conditional';
}
```

### WorkflowExecution

Tracks individual workflow execution instances.

```typescript
interface WorkflowExecution {
  id: string;
  workflowId: string;
  organizationId: string;
  status: ExecutionStatus; // PENDING | RUNNING | COMPLETED | FAILED | CANCELLED
  triggeredBy: string;
  triggerEvent: string;
  context: Record<string, any>;
  currentStepId?: string;
  startedAt?: Date;
  completedAt?: Date;
  error?: string;
}
```

### WorkflowStepExecution

Tracks individual step executions within a workflow.

```typescript
interface WorkflowStepExecution {
  id: string;
  executionId: string;
  stepId: string;
  stepType: WorkflowStepType;
  status: ExecutionStatus;
  input?: Record<string, any>;
  output?: Record<string, any>;
  error?: string;
  startedAt?: Date;
  completedAt?: Date;
}
```

## Workflow Validation

### Publish Validation

Server-side validation ensures workflow integrity before publishing.

<Steps>
  <Step title="Entry Point Check">
    Validates exactly one entry point exists (typically TRIGGER or START node).
  </Step>
  
  <Step title="Reference Validation">
    Checks that all step references (connections, branch targets, jump destinations) point to existing steps.
  </Step>
  
  <Step title="Reachability Analysis">
    Ensures all nodes are reachable from the entry point, flagging orphaned steps.
  </Step>
  
  <Step title="Cycle Detection">
    Rejects accidental cycles unless explicitly modeled with JUMP_TO steps.
  </Step>
  
  <Step title="Terminal Node Validation">
    Blocks outgoing connections from END nodes.
  </Step>
</Steps>

<CodeGroup>
```typescript Validation Example
// Valid workflow
{
  "graph": {
    "nodes": [
      { "id": "start", "type": "TRIGGER" },
      { "id": "msg", "type": "SEND_MESSAGE" },
      { "id": "end", "type": "END" }
    ],
    "edges": [
      { "source": "start", "target": "msg" },
      { "source": "msg", "target": "end" }
    ]
  }
}
```

```typescript Validation Errors
// Invalid workflow - unreachable node
{
  "errors": [
    {
      "type": "UNREACHABLE_NODE",
      "nodeId": "orphan-step",
      "message": "Step is not reachable from entry point"
    }
  ]
}

// Invalid workflow - cycle without JUMP_TO
{
  "errors": [
    {
      "type": "INVALID_CYCLE",
      "nodes": ["step1", "step2", "step3"],
      "message": "Cycle detected without explicit JUMP_TO"
    }
  ]
}
```
</CodeGroup>

## Execution Engine

### Workflow Execution Flow

<Steps>
  <Step title="Trigger Event">
    Workflow triggered by event (message received, contact created, webhook, etc.)
  </Step>
  
  <Step title="Execution Creation">
    `WorkflowEngineService` creates `WorkflowExecution` record with initial context
  </Step>
  
  <Step title="Queue Job Creation">
    Execution job added to pg-boss queue for async processing
  </Step>
  
  <Step title="Worker Processing">
    Worker picks up job and begins step-by-step execution
  </Step>
  
  <Step title="Step Execution">
    Each step handler processes step configuration and updates execution state
  </Step>
  
  <Step title="Next Step Resolution">
    Engine determines next step based on connections, conditions, and step output
  </Step>
  
  <Step title="Completion or Error">
    Execution completes successfully or fails with error tracking
  </Step>
</Steps>

### Queue Integration

Workflows execute asynchronously via pg-boss workers.

<Tabs>
  <Tab title="Queue Configuration">
    ```typescript
    {
      name: 'workflow-execution',
      options: {
        retryLimit: 3,
        retryDelay: 60, // seconds
        retryBackoff: true,
        expireInHours: 24
      }
    }
    ```
  </Tab>
  
  <Tab title="Error Handling">
    ```typescript
    interface ExecutionError {
      stepId: string;
      stepType: WorkflowStepType;
      error: string;
      timestamp: Date;
      retryCount: number;
      isFatal: boolean;
    }
    ```
    
    <Note>
    Non-fatal errors trigger automatic retry with exponential backoff. Fatal errors immediately fail the execution.
    </Note>
  </Tab>
  
  <Tab title="Retry Logic">
    - **Transient Errors**: Network failures, rate limits → Retry with backoff
    - **Configuration Errors**: Invalid step config → Immediate failure
    - **Permission Errors**: Insufficient access → Immediate failure
    - **Resource Errors**: Missing contact, workflow → Retry once, then fail
  </Tab>
</Tabs>

### Scope-Aware Routing

`WorkflowTriggerService` routes triggers based on workflow scope.

<Steps>
  <Step title="Event Capture">
    System event captured with full context (organization, user, contact, etc.)
  </Step>
  
  <Step title="Workflow Discovery">
    Find all published workflows matching event type
  </Step>
  
  <Step title="Scope Filtering">
    - **ORGANIZATION** workflows: Available to entire organization
    - **PERSONAL** workflows: Only trigger for creating user's actions
  </Step>
  
  <Step title="Condition Evaluation">
    Evaluate trigger conditions against event context
  </Step>
  
  <Step title="Execution Initiation">
    Create execution for each matching workflow
  </Step>
</Steps>

### Depth Limits and Validation

<Warning>
Workflow chaining is limited to a maximum depth of 5 levels to prevent infinite recursion and resource exhaustion.
</Warning>

<Steps>
  <Step title="Depth Tracking">
    Each workflow execution tracks its chain depth:
    
    ```typescript
    {
      "executionId": "exec-123",
      "workflowId": "parent-workflow",
      "chainDepth": 1,
      "parentExecutionId": null
    }
    ```
  </Step>
  
  <Step title="Depth Validation">
    Before triggering child workflow:
    
    ```typescript
    if (currentDepth >= 5) {
      throw new Error('Maximum workflow chain depth exceeded');
    }
    ```
  </Step>
  
  <Step title="Scope Validation">
    Child workflows must be accessible to the triggering scope:
    
    - ORGANIZATION workflows can trigger ORGANIZATION workflows
    - PERSONAL workflows can only trigger owner's PERSONAL workflows
    - Cross-organization triggering is blocked
  </Step>
  
  <Step title="Circular Reference Detection">
    System detects and prevents circular workflow references:
    
    ```typescript
    const executionChain = [parentId, grandparentId, ...];
    if (executionChain.includes(childWorkflowId)) {
      throw new Error('Circular workflow reference detected');
    }
    ```
  </Step>
</Steps>

## Scope and Permissions

### Workflow Scopes

Workflows operate under two distinct scopes with different visibility and access rules.

<Tabs>
  <Tab title="ORGANIZATION Scope">
    **Visibility:** All organization members  
    **Triggers:** Organization-wide events  
    **Access Control:** Based on organization roles
    
    ```typescript
    {
      "scope": "ORGANIZATION",
      "organizationId": "org-123",
      "createdBy": "user-456",
      "permissions": {
        "view": ["ADMIN", "MANAGER", "AGENT"],
        "edit": ["ADMIN", "MANAGER"],
        "execute": ["ADMIN", "MANAGER", "AGENT"]
      }
    }
    ```
    
    <Info>
    ORGANIZATION workflows are ideal for team-wide automations like lead routing, welcome messages, and escalation procedures.
    </Info>
  </Tab>
  
  <Tab title="PERSONAL Scope">
    **Visibility:** Only the creator  
    **Triggers:** Only creator's events  
    **Access Control:** Restricted to creator
    
    ```typescript
    {
      "scope": "PERSONAL",
      "organizationId": "org-123",
      "createdBy": "user-456",
      "permissions": {
        "view": ["user-456"],
        "edit": ["user-456"],
        "execute": ["user-456"]
      }
    }
    ```
    
    <Tip>
    PERSONAL workflows are perfect for individual productivity automations, personal follow-up sequences, and custom reminders.
    </Tip>
  </Tab>
</Tabs>

### Permission Enforcement

<Steps>
  <Step title="Read Permissions">
    - ORGANIZATION: All org members can view
    - PERSONAL: Only creator can view
    - Admins can view all workflows
  </Step>
  
  <Step title="Write Permissions">
    - ORGANIZATION: Admins and Managers can edit
    - PERSONAL: Only creator can edit
    - Workflow transfers require admin approval
  </Step>
  
  <Step title="Execute Permissions">
    - ORGANIZATION: All org members can manually trigger
    - PERSONAL: Only creator can manually trigger
    - Automatic triggers respect scope boundaries
  </Step>
  
  <Step title="Delete Permissions">
    - ORGANIZATION: Admins can delete
    - PERSONAL: Creator can delete
    - Deletion soft-deletes for audit trail
  </Step>
</Steps>

## Best Practices

### Workflow Design

<Steps>
  <Step title="Single Responsibility">
    Design workflows with a single, clear purpose. Avoid creating overly complex workflows that try to handle multiple unrelated scenarios.
    
    ✅ Good: "Welcome new WhatsApp contacts"  
    ❌ Bad: "Handle all messaging scenarios"
  </Step>
  
  <Step title="Clear Naming">
    Use descriptive names that explain the workflow's purpose and trigger.
    
    ✅ Good: "After-Hours Auto-Response"  
    ❌ Bad: "Workflow 1"
  </Step>
  
  <Step title="Error Handling">
    Always include error handling branches and timeout logic for interactive steps.
    
    ```typescript
    {
      "type": "ASK_QUESTION",
      "config": {
        "timeoutMinutes": 1440,
        "timeoutAction": "SEND_REMINDER"
      }
    }
    ```
  </Step>
  
  <Step title="Testing">
    Test workflows in DRAFT status before publishing. Use manual triggers with test contacts.
  </Step>
  
  <Step title="Documentation">
    Add clear descriptions to workflows and complex steps. Document custom logic in comments.
  </Step>
</Steps>

### Performance Optimization

<Tabs>
  <Tab title="Minimize Steps">
    Keep workflows concise. Combine related operations into single steps where possible.
    
    ✅ Good: Update contact with multiple fields in one UPDATE_CONTACT step  
    ❌ Bad: Separate UPDATE_CONTACT step for each field
  </Tab>
  
  <Tab title="Avoid Deep Nesting">
    Limit workflow chaining to necessary levels. Each additional level adds latency and complexity.
    
    Recommended maximum: 3 levels
  </Tab>
  
  <Tab title="Batch Operations">

### Security Best Practices

<Warning>
Always validate user permissions before workflow operations and sanitize external inputs.
</Warning>

<Steps>
  <Step title="Permission Checks">
    Verify user has appropriate permissions for workflow actions:
    
    - Contact updates require contact:write permission
    - Task creation requires task:create permission
    - User assignment requires organization role checks
  </Step>
  
  <Step title="Input Validation">
    Validate and sanitize all user inputs and external data:
    
    ```typescript
    {
      "type": "ASK_QUESTION",
      "config": {
        "validation": {
          "type": "EMAIL",
          "sanitize": true,
          "maxLength": 255
        }
      }
    }
    ```
  </Step>
  
  <Step title="Rate Limiting">
    Implement rate limits for external integrations and message sending:
    
    - Maximum 100 messages per workflow execution
    - Maximum 10 API calls per minute per workflow
    - Maximum 5 concurrent executions per workflow
  </Step>
  
  <Step title="Sensitive Data">
    Avoid logging sensitive information in execution context:
    
    ```typescript
    // Mask sensitive fields in logs
    {
      "sensitiveFields": ["password", "apiKey", "creditCard"]
    }
    ```
  </Step>
</Steps>

## Migration from Legacy

<Warning>
Legacy `steps[]` + `connections[]` format and `GLOBAL` scope are deprecated. Migrate to `graph` format and explicit scopes.
</Warning>

### Migration Steps

<Steps>
  <Step title="Update Workflow Structure">
    Convert legacy format to graph format:
    
    ```typescript
    // Legacy (deprecated)
    {
      "steps": [...],
      "connections": [...],
      "isActive": true
    }
    
    // Modern (current)
    {
      "graph": {
        "nodes": [...],
        "edges": [...]
      },
      "status": "PUBLISHED"
    }
    ```
  </Step>
  
  <Step title="Update Scope">
    Convert GLOBAL scope to ORGANIZATION or PERSONAL:
    
    ```typescript
    // Legacy
    { "scope": "GLOBAL" }
    
    // Modern
    { "scope": "ORGANIZATION" }
    ```
  </Step>
  
  <Step title="Update Activation">
    Replace activate/deactivate with status:
    
    ```typescript
    // Legacy
    POST /api/workflows/:id/activate
    POST /api/workflows/:id/deactivate
    
    // Modern
    POST /api/workflows/:id/publish
    POST /api/workflows/:id/archive
    ```
  </Step>
  
  <Step title="Test Migration">
    Validate migrated workflows:
    
    ```typescript
    POST /api/workflows/:id/validate
    
    Response:
    {
      "valid": true,
      "migrationComplete": true,
      "warnings": []
    }
    ```
  </Step>
</Steps>

## Troubleshooting

### Common Issues

<AccordionGroup>
  <Accordion title="Workflow Not Triggering">
    **Symptoms:** Workflow doesn't execute when expected event occurs
    
    **Causes:**
    - Workflow in DRAFT or ARCHIVED status
    - Trigger conditions don't match event
    - Scope mismatch (PERSONAL workflow for org-wide event)
    - Permission issues
    
    **Solutions:**
    1. Verify workflow status is PUBLISHED
    2. Check trigger configuration matches event type
    3. Review scope settings
    4. Validate user permissions
  </Accordion>
  
  <Accordion title="Execution Stuck in RUNNING">
    **Symptoms:** Execution stays in RUNNING status indefinitely
    
    **Causes:**
    - ASK_QUESTION step waiting for response
    - WAIT step with long duration
    - Worker processing failure
    - Queue backlog
    
    **Solutions:**
    1. Check for pending ASK_QUESTION responses
    2. Review WAIT step durations
    3. Restart workflow workers
    4. Check pg-boss queue health
  </Accordion>
  
  <Accordion title="Step Execution Failures">
    **Symptoms:** Workflow fails at specific step
    
    **Causes:**
    - Invalid step configuration
    - Missing required fields
    - Permission errors
    - External service failures
    
    **Solutions:**
    1. Review step configuration
    2. Validate required fields present
    3. Check user permissions
    4. Verify external service availability
    5. Review error logs for details
  </Accordion>
  
  <Accordion title="AI Agent Not Responding">
    **Symptoms:** AI_AGENT step fails or times out
    
    **Causes:**
    - Insufficient AI credits
    - Invalid agent configuration
    - OpenAI service issues
    - Knowledge base empty
    
    **Solutions:**
    1. Check organization credit balance
    2. Validate agent
