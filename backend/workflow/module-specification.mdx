## Architecture Overview

The workflow system operates on a graph-based execution model where workflows are defined as directed graphs of **nodes** with per-step **connections**.

### Canonical Data Structure

<CodeGroup>
```typescript Workflow Schema
interface Workflow {
  graph: {
    nodes: WorkflowStepDefinition[];
    edges: WorkflowEdge[];
  };
  scope: AutomationScope; // ORGANIZATION | PERSONAL
  status: WorkflowStatus; // DRAFT | PUBLISHED | ARCHIVED
}
```

```typescript Step Definition
interface WorkflowStepDefinition {
  id: string;
  type: WorkflowStepType;
  position: { x: number; y: number };
  data: Record<string, any>;
  connections?: string[]; // Connected step IDs
}
```
</AccordionGroup>

<Warning>
Legacy documentation that references `steps[]` + `connections[]` pairings, `GLOBAL` scope, `isActive`, or activate/deactivate APIs is obsolete.
</Warning>

### Workflow Validation

Server-side publish validation ensures:

- Single entry point requirement
- Rejection of broken references
- Rejection of unreachable nodes
- Prevention of accidental cycles (unless explicitly modeled with `JUMP_TO`)
- Blocking outgoing connections from `END` nodes

## Trigger System

### Event-driven Activation

Workflows can be triggered by:

- **Messaging Events**: Incoming messages, replies, delivery status
- **CRM Events**: Contact creation, updates, assignments
- **Webhook Events**: External system integrations
- **Manual Triggers**: User-initiated workflow execution
- **AI Agent Actions**: AI agents can trigger workflows via `trigger_workflow` action

### Scope Management

<Info>
Workflows support both **Organization** and **Personal** scopes with proper permission enforcement and routing.
</Info>
