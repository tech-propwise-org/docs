## Overview

The workflow module provides a comprehensive automation engine for PropWise CRM, enabling organizations to create, manage, and execute automated workflows triggered by various events. The module supports complex multi-step processes including messaging, data manipulation, external integrations, conditional logic, and AI agent integration.

### Key Features

- **Visual Workflow Builder**: Graph-based workflow definition with nodes and connections
- **Template System**: 11 pre-configured starter automation templates
- **Execution Engine**: Reliable workflow processing with step tracking and retry logic
- **Queue Integration**: Asynchronous execution via pg-boss workers
- **Trigger System**: Event-driven workflow initiation with scope-aware routing
- **AI Integration**: Bidirectional integration with the comprehensive AI module
- **Interactive Elements**: Question/answer flows with validation and timeout handling
- **Advanced Branching**: Structured condition system with category-based field resolution
- **Workflow Chaining**: Support for triggering child workflows with depth limits

### Data Model

The workflow system operates on a graph-based execution model where workflows are defined as directed graphs of **nodes** with per-step **connections**.

<Info>
**Canonical Persisted Shape (2025+)**

```typescript
Workflow.graph = {
  nodes: WorkflowStepDefinition[],
  edges: WorkflowEdge[]
}
```

With `AutomationScope` **`ORGANIZATION | PERSONAL`** and `WorkflowStatus` **`DRAFT | PUBLISHED | ARCHIVED`**.
</Info>

<Warning>
Legacy documentation that references `steps[]` + `connections[]` pairings, **`GLOBAL` scope**, `isActive`, or **activate/deactivate** APIs is obsolete.
</Warning>

## Workflow Scopes

Workflows can be scoped to either organization-wide use or personal use:

### Organization Scope

- Available to all users within the organization
- Managed by administrators and workflow managers
- Can be triggered by any user's events matching the trigger criteria
- Suitable for team-wide automations

### Personal Scope

- Created and owned by individual users
- Only triggered by the owner's events
- Private to the creator
- Ideal for personal automation preferences

<Tip>
Use organization scope for company-wide processes like lead qualification and welcome messages. Use personal scope for individual agent workflows like follow-up reminders.
</Tip>

## Workflow Status Lifecycle

<Steps>
  <Step title="Draft">
    Initial state when creating a workflow. Can be edited freely without validation constraints.
  </Step>
  
  <Step title="Published">
    Workflow is validated and active. Server-side validation requires:
    - Single entry point
    - No broken references
    - No unreachable nodes
    - No accidental cycles (unless explicitly modeled with `JUMP_TO`)
    - No outgoing connections from `END` nodes
  </Step>
  
  <Step title="Archived">
    Workflow is inactive and hidden from active lists. Can be restored to draft state.
  </Step>
</Steps>

## Step Types

The workflow engine supports a comprehensive set of step types for building complex automations.

### Communication Steps

<Tabs>
  <Tab title="SEND_MESSAGE">
    Sends a message to the contact via the conversation channel.
    
    **Configuration:**
    - `message`: Message content (supports variable interpolation)
    - `attachments`: Optional file attachments
    
    **Use Cases:**
    - Welcome messages
    - Automated responses
    - Notifications
  </Tab>
  
  <Tab title="ASK_QUESTION">
    Prompts the contact with a question and waits for a response.
    
    **Configuration:**
    - `question`: Question text
    - `expectedResponseType`: TEXT, NUMBER, DATE, CHOICE
    - `validationRules`: Optional validation constraints
    - `timeoutMinutes`: Response timeout duration
    - `contactFieldToUpdate`: Optional contact field to store response
    
    **Features:**
    - Type validation
    - Timeout handling
    - Contact field updates
    - Conditional branching based on response
  </Tab>
</Tabs>

### Logic Steps

<Tabs>
  <Tab title="BRANCH">
    Implements conditional logic with multiple evaluation paths.
    
    **Configuration:**
    - `conditions`: Array of condition groups
    - `operator`: AND / OR for combining conditions
    - `connections`: Output paths for true/false branches
    
    **Condition Categories:**
    - Contact fields
    - Conversation properties
    - Date/time comparisons
    - Custom field values
    
    **Operators:**
    - Equality: `EQUALS`, `NOT_EQUALS`
    - Comparison: `GREATER_THAN`, `LESS_THAN`, `GREATER_THAN_OR_EQUAL`, `LESS_THAN_OR_EQUAL`
    - String: `CONTAINS`, `NOT_CONTAINS`, `STARTS_WITH`, `ENDS_WITH`
    - Existence: `IS_EMPTY`, `IS_NOT_EMPTY`
    - Set membership: `IN`, `NOT_IN`
  </Tab>
  
  <Tab title="JUMP_TO">
    Unconditional jump to another step in the workflow, enabling explicit loops.
    
    **Configuration:**
    - `targetStepId`: ID of the step to jump to
    
    **Use Cases:**
    - Retry loops
    - Multi-stage processes
    - Conditional recursion
    
    <Warning>
    Use with caution to avoid infinite loops. The execution engine enforces maximum iteration limits.
    </Warning>
  </Tab>
  
  <Tab title="END">
    Terminates the workflow execution.
    
    **Configuration:**
    - No additional configuration required
    
    <Check>
    Server-side validation blocks outgoing connections from END nodes.
    </Check>
  </Tab>
</Tabs>

<Tabs>
  <Tab title="UPDATE_CONTACT">

    **Use Cases:**
    - Follow-up reminders
    - Action items
    - Escalation tasks
  </Tab>
  
  <Tab title="ASSIGN_TO">

    **Configuration:**
    - `userId`: Target user ID
    - `teamId`: Target team ID
    - `assignmentStrategy`: ROUND_ROBIN, LEAST_ACTIVE, SPECIFIC_USER
    
    **Features:**

</Tabs>

### Integration Steps

<Tabs>
  <Tab title="AI_AGENT">
    Activates an AI agent to handle the conversation.
    
    **Configuration:**
    - `agentId`: ID of the AI agent to activate
    - `contextVariables`: Additional context for the agent
    - `maxTurns`: Maximum conversation turns
    - `fallbackBehavior`: Action if agent fails

    Maximum workflow chain depth is enforced to prevent stack overflow and infinite recursion.
    </Warning>
  </Tab>
  
  <Tab title="WEBHOOK">

    - `url`: Target webhook URL

## Execution Engine

The `WorkflowEngineService` manages workflow execution with robust tracking and error handling.

### Execution Lifecycle

<Steps>
  <Step title="Trigger">
    Workflow is triggered by an event matching the trigger configuration.
  </Step>
  
  <Step title="Initialization">
    Create `WorkflowExecution` record with initial context and state.
  </Step>
  
  <Step title="Step Processing">
    Execute each step sequentially, updating execution state:
    - Load step configuration
    - Execute step handler
    - Create `WorkflowExecutionStep` record
    - Determine next step via connections
  </Step>
  
  <Step title="Branching">
    For `BRANCH` steps, evaluate conditions and follow appropriate connection.
  </Step>
  
  <Step title="Completion">
    Mark execution as completed or failed based on final state.
  </Step>
</Steps>

### Execution Context

Each workflow execution maintains a context object containing:

```typescript
{
  contact: ContactEntity,           // Current contact
  conversation: ConversationEntity, // Current conversation
  trigger: TriggerEvent,            // Triggering event
  variables: Record<string, any>,   // Workflow variables
  executionId: string,              // Execution ID
  organizationId: string,           // Organization context
  userId: string,                   // User context
  stepResults: Record<string, any>  // Previous step outputs
}
```

### Error Handling

<Tabs>
  <Tab title="Retry Logic">
    Failed steps are automatically retried based on configuration:
    - Exponential backoff
    - Maximum retry attempts
    - Retry-specific error types
  </Tab>
  
  <Tab title="Error Recovery">
    Workflows can define error recovery paths:
    - Catch error conditions
    - Route to error handling steps
    - Notify administrators
    - Graceful degradation
  </Tab>
  
  <Tab title="Failure Modes">
    Different failure modes for different scenarios:
    - **STOP**: Halt execution immediately
    - **CONTINUE**: Log error and continue
    - **RETRY**: Retry the failed step
    - **BRANCH_ERROR**: Follow error branch connection
  </Tab>
</Tabs>

### Queue Integration

Workflow execution uses pg-boss for reliable queue processing:

<CodeGroup>
```typescript Enqueueing Execution
await workflowQueueService.enqueueExecution({
  workflowId: workflow.id,
  executionId: execution.id,
  context: executionContext,
  priority: workflow.priority || 0
});
```

```typescript Worker Processing
@Worker('workflow-execute')
async processWorkflowExecution(job: Job<WorkflowExecutionJob>) {
  const { workflowId, executionId, context } = job.data;
  
  try {
    await this.workflowEngineService.executeWorkflow(
      workflowId,
      executionId,
      context
    );
    return { success: true };
  } catch (error) {
    await this.handleExecutionError(executionId, error);
    throw error; // Triggers retry
  }
}
```
</CodeGroup>

<Steps>

    ```json
      "name": "Lead Qualification",
      "scope": "ORGANIZATION",

        "nodes": [...],
        "edges": [...]
      },
      "trigger": {...},

      "updatedAt": "2024-01-15T00:00:00Z"
    }
    ```
  </Accordion>
  
  <Accordion title="POST /workflows">
    Create a new workflow.
    
    **Request Body:**
    ```json
    {
      "name": "New Workflow",
      "scope": "ORGANIZATION",
      "description": "Workflow description",
      "graph": {
        "nodes": [...],
        "edges": [...]
      }
    }
    ```
    
    <Note>
    New workflows are created in DRAFT status by default.
    </Note>
  </Accordion>
  
  <Accordion title="PATCH /workflows/:id">
    Update workflow configuration.
    
    **Request Body:**
    ```json
    {
      "name": "Updated Name",
      "graph": {
        "nodes": [...],
        "edges": [...]
      }
    }
    ```
    
    <Warning>
    Published workflows cannot be edited. Archive and create a new version instead.
    </Warning>
  </Accordion>
  
  <Accordion title="POST /workflows/:id/publish">
    Publish a draft workflow with validation.
    
    **Validation Checks:**
    - Single entry point
    - No broken references
    - No unreachable nodes
    - No accidental cycles
    - No outgoing connections from END nodes
    
    **Response:**
    ```json
    {
      "success": true,
      "workflow": {...}
    }
    ```
  </Accordion>
  
  <Accordion title="POST /workflows/:id/archive">
    Archive a workflow.
    
    **Response:**
    ```json
    {
      "success": true,
      "workflow": {...}
    }
    ```
  </Accordion>
  
  <Accordion title="DELETE /workflows/:id">
    Delete a workflow (soft delete).
    
    <Warning>
    Only archived workflows can be deleted. Published workflows must be archived first.
    </Warning>
  </Accordion>
</AccordionGroup>

```typescript Workflow Entity
  @ManyToOne(() => Organization)
  organization: Organization;

  @ManyToOne(() => User)
  createdBy: User;

  @CreateDateColumn()

  @UpdateDateColumn()
  updatedAt: Date;
}
```

    - Step processing and transitions

    - Error handling and recovery
    - Execution state tracking
    
    **Key Methods:**
    - `executeWorkflow(workflowId, context)`
    - `executeStep(executionId, stepId)`
    - `handleStepCompletion(executionId, stepId, output)`
    - `handleStepFailure(executionId, stepId, error)`
  </Tab>
  
  <Tab title="WorkflowTriggerService">
    Event routing and workflow initiation:
    - Event listener registration
    - Trigger condition evaluation
    - Scope-aware workflow matching
    - Execution enqueueing
    - Priority resolution
    
    **Key Methods:**
    - `registerTrigger(workflow, trigger)`
    - `handleEvent(event, context)`
    - `matchWorkflows(event, scope)`
    - `triggerWorkflow(workflowId, context)`
  </Tab>
  
  <Tab title="WorkflowValidationService">
    Graph validation and integrity checks:
    - Entry point validation
    - Reference validation
    - Reachability analysis
    - Cycle detection

    - `validateWorkflowGraph(graph)`
    - `validatePublish(workflow)`

## Performance Considerations

### Optimization Strategies

<Tabs>
  <Tab title="Queue Management">
    - **Priority Queues**: High-priority workflows processed first
    - **Batch Processing**: Group similar operations for efficiency
    - **Rate Limiting**: Prevent queue overload
    - **Dead Letter Queue**: Handle persistent failures
  </Tab>
  
  <Tab title="Database Performance">
    - **Indexes**: On workflow_id, organization_id, status fields
    - **Partitioning**: Partition execution tables by date
    - **Archival**: Move old executions to archive tables
    - **Caching**: Cache published workflow definitions
  </Tab>
  
  <Tab title="Execution Optimization">
    - **Parallel Execution**: Execute independent branches concurrently
    - **Step Caching**: Cache step results within execution context
    - **Lazy Loading**: Load context data only when needed
    - **Connection Pooling**: Reuse database connections
  </Tab>
  
  <Tab title="Monitoring">
    - **Execution Metrics**: Track duration, failure rates, throughput
    - **Resource Usage**: Monitor CPU, memory, queue depth
    - **Alert Thresholds**: Alert on high failure rates or long durations
    - **Performance Profiling**: Identify bottlenecks in step execution
  </Tab>
</Tabs>

### Scaling Guidelines

<Steps>
  <Step title="Horizontal Scaling">
    Add more worker processes to handle increased load. pg-boss supports multiple workers processing the same queue.
  </Step>
  
  <Step title="Queue Partitioning">
    Separate queues for different priority levels or workflow types.
  </Step>
  
  <Step title="Database Optimization">
    Use read replicas for execution history queries. Implement table partitioning for large execution tables.
  </Step>
  
  <Step title="Caching Layer">
    Implement Redis caching for frequently accessed workflow definitions and contact data.
  </Step>
</Steps>

## Migration Guide

### Legacy to v7 Migration

For systems upgrading from legacy workflow implementations:

<Steps>
  <Step title="Data Migration">
    Run migration scripts to convert:
    - `steps[]` + `connections[]` to `graph.nodes` + `graph.edges`
    - `GLOBAL` scope to `ORGANIZATION` scope
    - `isActive` boolean to `status` enum (PUBLISHED/ARCHIVED)
    - Legacy trigger format to new trigger structure
  </Step>
  
  <Step title="API Updates">
    Update API calls:
    - Replace `activate`/`deactivate` with `publish`/`archive`
    - Update scope filters from `GLOBAL` to `ORGANIZATION`
    - Use new graph structure in create/update requests
  </Step>
  
  <Step title="Template Migration">
    Re-seed templates using `WorkflowTemplateSeederService`:
    ```typescript
    await workflowTemplateSeeder.seedTemplates(entityManager);
    ```
  </Step>
  
  <Step title="Testing">
    Thoroughly test migrated workflows:
    - Verify trigger conditions still match
    - Test execution paths
    - Validate branching logic
    - Check AI agent integrations
  </Step>
</Steps>

## Troubleshooting

### Common Issues

<AccordionGroup>
  <Accordion title="Workflow Not Triggering">
    **Possible Causes:**
    - Workflow status is DRAFT or ARCHIVED (must be PUBLISHED)
    - Trigger conditions don't match event
    - Scope mismatch (personal workflow for wrong user)
    - Event filtering excludes the event
    
    **Solution:**
    - Verify workflow status is PUBLISHED
    - Review trigger configuration and event details
    - Check scope and user context
    - Test trigger conditions with sample events
  </Accordion>
  
  <Accordion title="Execution Stuck in RUNNING State">
    **Possible Causes:**
    - Step timeout not configured
    - Worker process crashed
    - Queue processing halted
    - Infinite loop in workflow
    
    **Solution:**
    - Check worker logs for errors
    - Review step timeout configuration
    - Verify queue health
    - Analyze workflow graph for cycles
    - Manually cancel stuck execution
  </Accordion>
  
  <Accordion title="AI Agent Step Failing">
    **Possible Causes:**
    - Insufficient AI credits
    - Agent not found or inactive
    - OpenAI API issues
    - Knowledge base not configured
    
    **Solution:**
    - Check organization credit balance
    - Verify agent exists and is active
    - Review AI activity logs
    - Test OpenAI connectivity
    - Validate knowledge base setup
  </Accordion>
  
  <Accordion title="Variable Interpolation Not Working">
    **Possible Causes:**
    - Incorrect variable syntax
    - Variable not in execution context
    - Type mismatch
    - Nested property access error
    
    **Solution:**
    - Use correct syntax: `{{variable.path}}`
    - Verify variable exists in context
    - Check execution logs for resolution errors
    - Test with simple variables first
  </Accordion>
  
  <Accordion title="Branch Condition Always False">
    **Possible Causes:**
    - Wrong field category
    - Incorrect operator for field type
    - Field value mismatch
    - Null or undefined field value
    
    **Solution:**
    - Verify field category (CONTACT, CONVERSATION, etc.)
    - Use type-appropriate operator
    - Check actual field values in logs
    - Add IS_NOT_EMPTY condition first
  </Accordion>
</AccordionGroup>

## Best Practices

### Workflow Design

<Check>
**Design Recommendations**

- Keep workflows focused on a single business process
- Use descriptive names for steps and workflows
- Add END nodes to all terminal branches
- Limit workflow depth to 3-4 levels when chaining
- Use templates as starting points for common patterns
- Test workflows thoroughly before publishing
- Document complex branching logic with comments
- Use organization scope for team-wide processes
- Reserve personal scope for individual preferences
</Check>

### Performance Best Practices

<Tip>
**Optimization Tips**

- Minimize external webhook calls
- Use branching to avoid unnecessary steps
- Set appropriate timeouts for wait steps
- Avoid deep workflow chains
- Use batch operations where possible
- Cache frequently accessed contact data
- Monitor execution times and optimize slow steps
- Archive old workflow executions regularly
</Tip>

### Error Handling

<Info>
**Robust Error Handling**

- Always define error branches for critical steps
- Set reasonable retry limits (3-5 attempts)
- Use exponential backoff for retries
- Log detailed error context
- Implement fallback actions
- Alert administrators for critical failures
- Provide user-friendly error messages
- Test failure scenarios explicitly
</Info>
