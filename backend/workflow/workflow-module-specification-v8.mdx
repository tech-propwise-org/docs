## Overview

The workflow module provides a comprehensive automation engine for PropWise CRM, enabling organizations to create, manage, and execute automated workflows triggered by various events. The module supports complex multi-step processes including messaging, data manipulation, external integrations, conditional logic, and AI agent integration.

The workflow system operates on a **graph-based execution model** where workflows are defined as directed graphs of nodes with per-step connections. Workflows can be triggered by messaging/CRM events or webhooks. Server-side publish validation ensures:

- Single entry point requirement
- No broken references
- No unreachable nodes
- Cycle prevention (unless explicitly modeled with `JUMP_TO`)
- Blocked outgoing connections from `END` nodes

<Info>
**Canonical persisted shape (2025+):**
```typescript
Workflow.graph = {
  nodes: WorkflowStepDefinition[],
  edges: WorkflowEdge[]
}
```

With `AutomationScope`: **`ORGANIZATION | PERSONAL`** and `WorkflowStatus`: **`DRAFT | PUBLISHED | ARCHIVED`**

Legacy documentation referencing `steps[]` + `connections[]` pairings, `GLOBAL` scope, `isActive`, or activate/deactivate APIs is obsolete.
</Info>

## Workflow Engine

### Execution Flow

<Steps>
  <Step title="Workflow Trigger">
    Event occurs (message received, contact created, webhook) that matches workflow trigger conditions
  </Step>
  
  <Step title="Validation">
    `WorkflowValidatorService` validates workflow graph structure and permissions
  </Step>
  
  <Step title="Queue Job">
    Execution job enqueued via pg-boss with retry configuration
  </Step>
  
  <Step title="Worker Processing">
    `WorkflowExecuteWorker` picks up job and initializes execution context
  </Step>
  
  <Step title="Step Execution">
    `WorkflowEngineService` executes steps sequentially, handling connections and branching
  </Step>
  
  <Step title="Step Logging">
    Each step execution logged to `WorkflowStepLog` with input, output, and status
  </Step>
  
  <Step title="Completion">
    Execution marked as completed/failed with final status and error details if applicable
  </Step>
</Steps>

## Validation System

### Graph Validation

<Steps>
  <Step title="Entry Point Validation">
    Ensures exactly one entry node exists (no incoming connections)
  </Step>
  
  <Step title="Reference Validation">
    Verifies all step connections reference valid step IDs
  </Step>
  
  <Step title="Reachability Check">
    Ensures all nodes are reachable from the entry point
  </Step>
  
  <Step title="Cycle Detection">
    Prevents accidental infinite loops (unless using explicit `JUMP_TO`)
  </Step>
  
  <Step title="End Node Validation">
    Blocks outgoing connections from `END` nodes
  </Step>
  
  <Step title="Step Configuration Validation">
    Validates each step's configuration against its schema
  </Step>
</Steps>

```typescript
export class WorkflowValidatorService {
  /**
   * Validate complete workflow graph
   */
  async validateWorkflow(workflow: Workflow): Promise<ValidationResult> {
    const errors: string[] = [];

    // Validate graph structure
    errors.push(...this.validateGraphStructure(workflow.graph));

    // Validate each step configuration
    for (const node of workflow.graph.nodes) {
      const handler = this.getStepHandler(node.type);
      const stepValidation = await handler.validate(node);
      if (!stepValidation.isValid) {
        errors.push(...stepValidation.errors);
      }
    }

    errors.push(...this.validateTriggers(workflow));

      isValid: errors.length === 0,
```

## Queue Integration

### pg-boss Configuration

<CodeGroup>
```typescript Worker Registration
// workflow-execute.worker.ts
@Injectable()
export class WorkflowExecuteWorker {
  constructor(
    private readonly workflowEngine: WorkflowEngineService,
    private readonly boss: PgBoss
  ) {}

  async onModuleInit() {
    await this.boss.work(
      'workflow-execute',
      {
        teamSize: 5,
        teamConcurrency: 1,
        retryLimit: 3,
        retryDelay: 60, // 1 minute
        retryBackoff: true
      },
      this.handleJob.bind(this)
    );
  }

  async handleJob(job: PgBoss.Job<WorkflowExecuteJob>) {
    const { workflowId, context } = job.data;

    try {
      await this.workflowEngine.executeWorkflow(
        workflowId,
        context
      );
    } catch (error) {
      // Error logged to WorkflowExecution
      throw error; // Trigger retry
    }
  }
}
```

```typescript Job Enqueue
    depth: 0
  }
}, {
  priority: workflow.priority || 0,
```

    ```typescript
      input: { ... },
      output: null
    }
    ```

    Workflow continues to next step unless configured to halt on error.
  </Tab>

  <Tab title="Execution-Level Errors">
    Critical errors halt workflow execution:

    ```typescript
    {
      status: 'FAILED',
      errorMessage: 'Unhandled exception in step handler',
      currentStepId: 'step-123',
      completedAt: new Date()
    }
    ```

    Job will be retried based on pg-boss configuration.
  </Tab>

  <Tab title="Retry Strategy">
    - **Retry Limit**: 3 attempts
    - **Retry Delay**: 1 minute
    - **Backoff**: Exponential (1m, 2m, 4m)
    - **Dead Letter**: After exhausting retries
  </Tab>
</Tabs>

## Permissions & Scoping

### Automation Scope

<Tabs>
  <Tab title="ORGANIZATION">
    **Organization-scoped workflows:**
    - Visible to all organization members
    - Can be triggered by any organization event
    - Require organization-level permissions to edit
    - Execution uses organization resources

    ```typescript
    {
      scope: AutomationScope.ORGANIZATION,
      organization: { id: 'org-123' },
      createdBy: { id: 'user-456' }
    }
    ```
  </Tab>

  <Tab title="PERSONAL">
    **Personal workflows:**
    - Visible only to creator
    - Triggered by events related to creator's contacts/conversations
    - Creator has full edit permissions
    - Execution uses personal quotas

    ```typescript
    {
      scope: AutomationScope.PERSONAL,
      organization: { id: 'org-123' },
      createdBy: { id: 'user-456' }
    }
    ```
  </Tab>
</Tabs>

### Permission Checks

<CodeGroup>
```typescript View Permission
async canViewWorkflow(user: User, workflow: Workflow): Promise<boolean> {
  if (workflow.scope === AutomationScope.PERSONAL) {
    return workflow.createdBy.id === user.id;
  }
  
  return user.organization.id === workflow.organization.id;
}
```

```typescript Edit Permission
async canEditWorkflow(user: User, workflow: Workflow): Promise<boolean> {
  if (workflow.scope === AutomationScope.PERSONAL) {
    return workflow.createdBy.id === user.id;
  }
  
  // Organization workflows require admin or manager role
  return user.hasRole(['ADMIN', 'MANAGER']) 
    && user.organization.id === workflow.organization.id;
}
```

```typescript Delete Permission
async canDeleteWorkflow(user: User, workflow: Workflow): Promise<boolean> {
  if (workflow.scope === AutomationScope.PERSONAL) {
    return workflow.createdBy.id === user.id;
  }
  
  // Only admins can delete organization workflows
  return user.hasRole('ADMIN') 
    && user.organization.id === workflow.organization.id;
}
```

## Migration Notes

<Warning>
**Legacy API Deprecation**

The following legacy patterns are deprecated and will be removed:
- `steps[]` + `connections[]` structure → Use `graph.nodes` + `graph.edges`
- `GLOBAL` scope → Use `ORGANIZATION` scope
- `isActive` flag → Use `status` field with `DRAFT | PUBLISHED | ARCHIVED`
- `/workflows/:id/activate` endpoint → Use `/workflows/:id/publish`
- `/workflows/:id/deactivate` endpoint → Use `/workflows/:id/archive`
</Warning>

### Migration Steps

<Steps>
  <Step title="Update Data Model">
    Convert legacy workflow structure to new graph-based model
    
    ```typescript
    // Legacy structure
    { steps: [...], connections: [...] }
    
    // New structure
    { graph: { nodes: [...], edges: [...] } }
    ```
  </Step>
  
  <Step title="Update Scope Values">
    Replace `GLOBAL` with `ORGANIZATION`
    
    ```typescript
    // Legacy
    { scope: 'GLOBAL' }
    
    // New
    { scope: AutomationScope.ORGANIZATION }
    ```
  </Step>
  
  <Step title="Update Status Handling">
    Replace `isActive` with `status` enum
    
    ```typescript
    // Legacy
    { isActive: true }
    
    // New
    { status: WorkflowStatus.PUBLISHED }
    ```
