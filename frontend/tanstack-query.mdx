## Implementation Decision Rules

**Read this section FIRST when implementing any data fetching feature.**

### Decision Flowchart: When to Use TanStack Query

```
Is this fetching data from an API?
├── NO  → Don't use TanStack Query (use useState/useReducer)
└── YES → Continue...
    │
    Is this a one-time action (POST/PUT/DELETE)?
    ├── YES → Use `useMutation` inline in component
    └── NO  → Use `useQuery`
        │
        Is this query used in 3+ components?
        ├── YES → Add to `use-query-hooks.ts`
        └── NO  → Use `useQuery` inline in component
```

### Rule 1: ALWAYS Use TanStack Query for API Data Fetching

| Scenario | Use TanStack Query? | Pattern |
|----------|---------------------|---------|
| GET data from API | ✅ YES | `useQuery` |
| POST/PUT/DELETE to API | ✅ YES | `useMutation` |
| Client-side state only | ❌ NO | `useState` |
| Form field values | ❌ NO | `useState` / React Hook Form |
| UI state (modals, tabs) | ❌ NO | `useState` |
| Computed/derived data | ❌ NO | `useMemo` |

### Rule 2: When to Create Custom Hooks in `use-query-hooks.ts`

**✅ CREATE a custom hook when:**

| Condition | Example |
|-----------|---------|
| Used in 3+ components | `useUsers()` for dropdowns |
| Complex `enabled` logic | Auth-dependent queries |
| Data transformation needed | Response normalization |
| Shared lookup data | Teams, Roles, Stages |
| Auth/user context data | `useCurrentUser()` |

**❌ DO NOT create a custom hook when:**

| Condition | Example |
|-----------|---------|
| Page-specific query | Lead details page |
| One-off query | Analytics dashboard |
| Mutations | Create/Update/Delete |
| Component-specific logic | Modal with unique requirements |

### Rule 3: Mutations Stay in Components

```typescript
// ✅ CORRECT - Mutation defined in component
function CreateLeadModal() {
  const queryClient = useQueryClient();
  
  const createMutation = useMutation({
    mutationFn: (data) => LeadApi.create(data),
    onSuccess: () => {
      queryClient.invalidateQueries({ queryKey: queryKeys.leads.all });
      toast.success('Lead created');
    },
  });
  
  // ... component logic
}

// ❌ WRONG - Don't add mutations to use-query-hooks.ts
// export const useCreateLead = () => useMutation({ ... })
```

### Rule 4: Query Keys Are MANDATORY

```typescript
// ❌ NEVER do this
useQuery({
  queryKey: ['leads', filters],  // Inline key
  queryFn: () => LeadApi.findAll(filters),
});

// ✅ ALWAYS do this
useQuery({
  queryKey: queryKeys.leads.list(filters),  // From query-keys.ts
  queryFn: () => LeadApi.findAll(filters),
});
```

### Decision Examples

| Task | Decision | Implementation |
|------|----------|----------------|
| "Fetch leads for leads page" | Inline `useQuery` | `useQuery({ queryKey: queryKeys.leads.list(filters), ... })` |
| "Fetch user data for dropdown in 5 modals" | Custom hook | Add `useUsers()` to `use-query-hooks.ts` |
| "Create a new lead" | Inline `useMutation` | `useMutation({ mutationFn: LeadApi.create, ... })` |
| "Track selected tab" | NOT TanStack Query | `useState('tab1')` |
| "Store form values" | NOT TanStack Query | `useForm()` from react-hook-form |
| "Fetch current user on app load" | Custom hook | Use existing `useCurrentUser()` |
| "Fetch deal details on deal page" | Inline `useQuery` | `useQuery({ queryKey: queryKeys.deals.detail(id), ... })` |

### Quick Reference: What Goes Where

| File | Contains | Don't Put |
|------|----------|-----------|
| `src/lib/query-keys.ts` | All query key definitions | Query logic, API calls |
| `src/hooks/use-query-hooks.ts` | Shared query hooks (3+ usage) | Page-specific queries, mutations |
| `src/services/api/*.api.ts` | API methods only | Query hooks, React code |
| Component files | Page-specific queries, all mutations | Shared query definitions |

### Implementation Checklist

Before writing any data fetching code, verify:

```
[ ] Is this fetching from an API? → Use TanStack Query
[ ] Is this a GET request? → Use useQuery
[ ] Is this POST/PUT/DELETE? → Use useMutation (inline)
[ ] Will this query be reused in 3+ places?
    [ ] YES → Add to use-query-hooks.ts
    [ ] NO  → Write inline in component
[ ] Am I using queryKeys from query-keys.ts? → REQUIRED
[ ] Am I invalidating queries after mutation? → REQUIRED
[ ] Am I showing user feedback (toast)? → REQUIRED for mutations
```

## Overview

### What is TanStack Query?

TanStack Query is a powerful data-fetching and state management library that provides:

- **Automatic caching** — Reduces redundant API calls
- **Background refetching** — Keeps data fresh automatically
- **Optimistic updates** — Instant UI feedback before server response
- **Automatic retry** — Resilient error handling
- **DevTools** — Visual debugging of queries and cache

### Why We Use It

- ✅ Eliminates manual loading state management
- ✅ Prevents race conditions
- ✅ Reduces boilerplate code by 60-70%
- ✅ Provides consistent patterns across the codebase
- ✅ Better TypeScript integration

## Architecture & Setup

### 1. QueryClient Configuration

**Location:** `src/components/providers.tsx`

```typescript
const [queryClient] = useState(
  () =>
    new QueryClient({
      defaultOptions: {
        queries: {
          staleTime: 60 * 1000,        // 1 minute - data considered fresh
          gcTime: 5 * 60 * 1000,        // 5 minutes - data kept in cache
          retry: 1,                     // Retry failed queries once
          refetchOnWindowFocus: false,  // Prevent excessive refetches
        },
      },
    }),
);
```

**Key Configuration Options:**

| Option | Value | Purpose |
|--------|-------|---------|
| `staleTime` | 1-5 minutes | How long data is considered fresh |
| `gcTime` | 5-30 minutes | How long unused data stays in cache |
| `retry` | 1 | Number of retry attempts on failure |
| `refetchOnWindowFocus` | false | Disable auto-refetch when window regains focus |

### 2. Provider Setup

```typescript
<QueryClientProvider client={queryClient}>
  <ThemeProvider>
    <AuthProvider>
      {children}
    </AuthProvider>
  </ThemeProvider>
  {/* DevTools - only shows in development */}
  <ReactQueryDevtools initialIsOpen={false} buttonPosition='bottom-left' />
</QueryClientProvider>
```

### 3. Centralized Query Keys

**Location:** `src/lib/query-keys.ts`

All query keys follow a hierarchical structure:

```typescript
export const queryKeys = {
  user: {
    all: ['users'] as const,
    lists: () => [...queryKeys.user.all, 'list'] as const,
    list: (filters?: Record<string, unknown>) => 
      [...queryKeys.user.lists(), filters] as const,
    details: () => [...queryKeys.user.all, 'detail'] as const,
    detail: (id: string) => 
      [...queryKeys.user.details(), id] as const,
  },
  // ... other domains
} as const;
```

**Hierarchy Benefits:**

- Invalidate all users: `queryKeys.user.all`
- Invalidate all user lists: `queryKeys.user.lists()`
- Invalidate specific user: `queryKeys.user.detail(id)`

### 4. Invalidation Groups

**Location:** `src/lib/query-keys.ts`

Cross-domain cache invalidation for related data:

```typescript
export const invalidationGroups = {
  leads: [
    queryKeys.leads.all,
    queryKeys.analytics.leadStats,
    queryKeys.auditLogs.all,
    queryKeys.entityNames.all,
  ],
  // ... other groups
} as const;
```

## Best Practices & Patterns

### 1. Centralized Query Keys ✅

**Rule:** All query keys MUST come from `@/lib/query-keys`. Never use inline arrays.

```typescript
// ❌ BAD
useQuery({
  queryKey: ['leads'],
  queryFn: () => LeadApi.findAll(),
});

// ✅ GOOD
useQuery({
  queryKey: queryKeys.leads.list({}),
  queryFn: () => LeadApi.findAll(),
});
```

### 2. Reusable Query Hooks ✅

**Location:** `src/hooks/use-query-hooks.ts`

Create custom hooks for common queries:

```typescript
export const useUsers = (filters?: Record<string, unknown>, enabled = true) => {
  return useQuery({
    queryKey: queryKeys.user.list(filters),
    queryFn: () => UserApi.findAll(filters),
    enabled,
    staleTime: 1000 * 60 * 5, // 5 minutes
    select: (data) => {
      // Transform at select level to keep raw data in cache
      return Array.isArray(data) ? data : data.data || [];
    },
  });
};
```

**Benefits:**
- Single source of truth
- Consistent configuration
- Easier testing and maintenance

### 3. Conditional Fetching with `enabled` ✅

Disable queries until dependencies are ready:

```typescript
// Depend on user authentication
useQuery({
  queryKey: queryKeys.leads.list(),
  queryFn: () => LeadApi.findAll(),
  enabled: !!currentUser,
});

// Depend on ID
useQuery({
  queryKey: queryKeys.lead.detail(id),
  queryFn: () => LeadApi.getById(id),
  enabled: !!id,
});

// Multiple dependencies
useQuery({
  queryKey: queryKeys.analytics.dashboard(filters),
  queryFn: () => AnalyticsApi.getStats(filters),
  enabled: !!currentUser && !!selectedOrgId,
});
```

### 4. Response Normalization ✅

**Location:** `src/services/api/api.helpers.ts`

Use helper functions instead of inline normalization:

```typescript
// Helper function (already implemented)
export function normalizeResponse<T>(
  response: T[] | { data: T[] }
): T[] {
  return Array.isArray(response) ? response : response.data || [];
}

// ✅ GOOD - Use in select
useQuery({
  queryKey: queryKeys.users.list({}),
  queryFn: () => UserApi.findAll(),
  select: (data) => normalizeResponse(data),
});
```

### 5. Mutations with Proper Invalidation ✅

Every mutation MUST:
1. Invalidate related queries in `onSuccess`
2. Show user feedback (toast)
3. Handle errors appropriately

```typescript
const createMutation = useMutation({
  mutationFn: (data: CreateDto) => SomeApi.create(data),
  onSuccess: (data) => {
    queryClient.invalidateQueries({ queryKey: queryKeys.someDomain.all });
    toast.success('Created successfully');
    onSuccessCallback?.(data);
  },
  onError: (error: any) => {
    toast.error(error?.response?.data?.message ?? 'Operation failed');
  },
});
```

### 6. Data Transformation with `select` ✅

Transform data without affecting the cache:

```typescript
useQuery({
  queryKey: queryKeys.roles.list({ scope: 'org' }),
  queryFn: () => RoleApi.findAll({ scope: 'org' }),
  select: (data) => {
    // Raw data stays in cache; transformation happens on read
    const roles = Array.isArray(data) ? data : data.data || [];
    return roles.map(role => ({
      id: role.id,
      name: role.name,
      priority: role.isSystem ? 100 : 50,
    }));
  },
});
```

### 7. Prefetching for Better UX ✅

Prefetch data on hover or route transitions:

```typescript
// On hover
<Button
  onMouseEnter={() => {
    queryClient.prefetchQuery({
      queryKey: queryKeys.leads.detail(id),
      queryFn: () => LeadApi.getById(id),
    });
  }}
>
  View Lead
</Button>

// Before navigation
const router = useRouter();
const navigateToLead = async (id: string) => {
  // Prefetch before navigating
  await queryClient.prefetchQuery({
    queryKey: queryKeys.leads.detail(id),
    queryFn: () => LeadApi.getById(id),
  });
  router.push(`/leads/${id}`);
};
```

### 8. Optimistic Updates (Advanced) ⚡

For instant UI feedback before server response:

```typescript
useMutation({
  mutationFn: (id: string) => LeadApi.markAsRead(id),
  onMutate: async (id) => {
    // Cancel outgoing queries
    await queryClient.cancelQueries({ queryKey: queryKeys.leads.all });
    
    // Snapshot current value
    const previous = queryClient.getQueryData(queryKeys.leads.list({}));
    
    // Optimistically update
    queryClient.setQueryData(queryKeys.leads.list({}), (old: any[]) =>
      old.map(lead => lead.id === id ? { ...lead, read: true } : lead)
    );
    
    return { previous };
  },
  onError: (err, id, context) => {
    // Rollback on error
    if (context?.previous) {
      queryClient.setQueryData(queryKeys.leads.list({}), context.previous);
    }
  },
  onSettled: () => {
    // Refetch to ensure consistency
    queryClient.invalidateQueries({ queryKey: queryKeys.leads.all });
  },
});
```

### 9. Pagination with `placeholderData` ✅

For smooth pagination UX (TanStack Query v5):

```typescript
import { keepPreviousData } from '@tanstack/react-query';

const { data, isLoading } = useQuery({
  queryKey: queryKeys.leads.list({ page, limit: 10 }),
  queryFn: async () => {
    const response = await LeadApi.findAll({ page, limit: 10 });
    return {
      items: Array.isArray(response) ? response : response.data || [],
      totalPages: response.totalPages || 0,
    };
  },
  placeholderData: keepPreviousData, // ✅ v5 pattern
  staleTime: 1000 * 60 * 2, // 2 minutes
});
```

**Note:** `keepPreviousData: true` (v4) is deprecated.

## Migration Patterns

### Pattern 1: Simple Data Loading

**BEFORE (Deprecated):**

```typescript
const [users, setUsers] = useState([]);
const [loading, setLoading] = useState(false);

useEffect(() => {
  const loadUsers = async () => {
    try {
      setLoading(true);
      const response = await UserApi.findAll();
      setUsers(Array.isArray(response) ? response : response.data || []);
    } catch (error) {
      toast.error('Failed to load users');
    } finally {
      setLoading(false);
    }
  };
  loadUsers();
}, []);
```

**AFTER (TanStack Query):**

```typescript
const { data: users = [], isLoading: loading } = useQuery({
  queryKey: queryKeys.user.list({}),
  queryFn: () => UserApi.findAll(),
  select: (data) => Array.isArray(data) ? data : data.data || [],
  staleTime: 1000 * 60 * 5,
});
```

### Pattern 2: Modal Data Loading

**BEFORE:**

```typescript
const [teams, setTeams] = useState([]);
const [loading, setLoading] = useState(false);

useEffect(() => {
  if (open) {
    loadInitialData();
  }
}, [open]);

const loadInitialData = async () => {
  try {
    setLoading(true);
    const response = await TeamApi.findAll();
    setTeams(Array.isArray(response) ? response : response.data || []);
  } catch (error) {
    toast.error('Failed to load teams');
  } finally {
    setLoading(false);
  }
};
```

**AFTER:**

```typescript
const { data: teams = [], isLoading: loading } = useQuery({
  queryKey: queryKeys.teams.list({}),
  queryFn: () => TeamApi.findAll(),
  select: (data) => Array.isArray(data) ? data : data.data || [],
  enabled: open && !!selectedUser, // Only fetch when modal is open
  staleTime: 1000 * 60 * 5,
});
```

### Pattern 3: Multiple Data Sources

**BEFORE:**

```typescript
const [users, setUsers] = useState([]);
const [teams, setTeams] = useState([]);
const [loading, setLoading] = useState(false);

useEffect(() => {
  const loadAllData = async () => {
    try {
      setLoading(true);
      const [usersRes, teamsRes] = await Promise.all([
        UserApi.findAll(),
        TeamApi.findAll(),
      ]);
      setUsers(Array.isArray(usersRes) ? usersRes : usersRes.data || []);
      setTeams(Array.isArray(teamsRes) ? teamsRes : teamsRes.data || []);
    } catch (error) {
      toast.error('Failed to load data');
    } finally {
      setLoading(false);
    }
  };
  loadAllData();
}, []);
```

**AFTER:**

```typescript
// Each query is independent and cached separately
const { data: users = [] } = useQuery({
  queryKey: queryKeys.user.list({}),
  queryFn: () => UserApi.findAll(),
  select: (data) => Array.isArray(data) ? data : data.data || [],
  staleTime: 1000 * 60 * 5,
});

const { data: teams = [] } = useQuery({
  queryKey: queryKeys.teams.list({}),
  queryFn: () => TeamApi.findAll(),
  select: (data) => Array.isArray(data) ? data : data.data || [],
  staleTime: 1000 * 60 * 5,
});

// Queries run in parallel automatically
// No manual loading state management needed
```

## Advanced Features

### 1. Infinite Scroll with `useInfiniteQuery`

```typescript
const {
  data,
  fetchNextPage,
  hasNextPage,
  isFetchingNextPage,
} = useInfiniteQuery({
  queryKey: queryKeys.leads.list(filters),
  queryFn: ({ pageParam = 1 }) => 
    LeadApi.findAll({ ...filters, page: pageParam }),
  getNextPageParam: (lastPage, pages) => {
    return lastPage.hasMore ? pages.length + 1 : undefined;
  },
  initialPageParam: 1,
});

// Flatten pages
const leads = data?.pages.flatMap(page => page.data) ?? [];
```

### 2. Query Invalidation Helpers

**Location:** `src/hooks/use-query-hooks.ts`

```typescript
export const useInvalidateQueries = () => {
  const queryClient = useQueryClient();

  return {
    invalidateLeads: () => 
      queryClient.invalidateQueries({ queryKey: invalidationGroups.leads }),
    invalidateContacts: () =>
      queryClient.invalidateQueries({ queryKey: invalidationGroups.contacts }),
    invalidateAll: () => 
      queryClient.invalidateQueries(),
  };
};
```

### 3. Dependent Queries

When one query depends on another:

```typescript
// First query
const { data: user } = useQuery({
  queryKey: queryKeys.user.detail(userId),
  queryFn: () => UserApi.getById(userId),
});

// Second query depends on first
const { data: teams } = useQuery({
  queryKey: queryKeys.teams.list({ userId: user?.id }),
  queryFn: () => TeamApi.findByUser(user!.id),
  enabled: !!user?.id, // Only fetch when user is loaded
});
```

### 4. Parallel Queries

Run multiple queries simultaneously:

```typescript
const results = useQueries({
  queries: [
    {
      queryKey: queryKeys.users.list({}),
      queryFn: () => UserApi.findAll(),
    },
    {
      queryKey: queryKeys.teams.list({}),
      queryFn: () => TeamApi.findAll(),
    },
    {
      queryKey: queryKeys.roles.list({}),
      queryFn: () => RoleApi.findAll(),
    },
  ],
});

const [usersQuery, teamsQuery, rolesQuery] = results;
```

## Troubleshooting

### Common Issues

#### 1. Query Not Refetching

**Problem:** Data doesn't update after mutation.

**Solution:** Ensure proper invalidation:

```typescript
onSuccess: () => {
  queryClient.invalidateQueries({ queryKey: queryKeys.leads.all });
},
```

#### 2. Infinite Refetch Loop

**Problem:** Query keeps refetching continuously.

**Solution:** Stabilize query keys:

```typescript
// ❌ BAD - object reference changes every render
queryKey: queryKeys.leads.list({ filters })

// ✅ GOOD - use stable filter object
const stableFilters = useMemo(() => filters, [
  filters.search,
  filters.stage,
]);
queryKey: queryKeys.leads.list(stableFilters)
```

#### 3. Stale Data After Navigation

**Problem:** Data is stale when returning to a page.

**Solution:** Adjust `staleTime`:

```typescript
useQuery({
  queryKey: queryKeys.leads.list({}),
  queryFn: () => LeadApi.findAll(),
  staleTime: 1000 * 30, // 30 seconds instead of 5 minutes
});
```

## Performance Guidelines

### Stale Time Recommendations

| Data Type | Recommended `staleTime` |
|-----------|------------------------|
| User profile, org data | 5 minutes |
| Lists (leads, contacts) | 2 minutes |
| Real-time data | 30 seconds |
| Static data (config) | 10 minutes |

### When to Use Different Patterns

| Pattern | Use Case |
|---------|----------|
| `useQuery` | Standard data fetching |
| `useInfiniteQuery` | Infinite scroll, "Load More" |
| `useMutation` | Create, update, delete |
| `useQueries` | Multiple parallel queries |
| Optimistic updates | Instant UI feedback (read status, favorites) |
