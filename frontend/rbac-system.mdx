## Overview

The RBAC (Role-Based Access Control) system implements a **three-layer access model**:

| Layer | Scope | Example | Bypass Behavior |
|-------|-------|---------|-----------------|
| **Layer 1: Org RBAC** | Organization-wide | `crm.manage`, `system.admin` | Bypasses Layer 2 & 3 |
| **Layer 2: Team RBAC** | Team-specific | `team_crm.manage` in Team Sales | Bypasses Layer 3 |
| **Layer 3: Resource Access** | Entity-level | WRITE access on Lead #123 | Final check |

**Key Principles:**
- **Frontend checks are UX only** - Security is enforced on the backend
- **Backend is source of truth** - Permissions are computed server-side
- **Fail closed** - When in doubt, deny access
- **Owner bypass** - Organization owners have full access

## Architecture

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                         ACCESS CHECK FLOW                                   │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  User wants to perform ACTION on RESOURCE                                   │
│                                                                             │
│  ┌────────────────────────────────────────────────────────────────────────┐ │
│  │ LAYER 1: Org RBAC (Bypass Check)                                       │ │
│  │ Does user have org-level permission? (e.g., crm.manage, system.admin)  │ │
│  │   → YES: ALLOW                                                         │ │
│  │   → NO: Continue to Layer 2                                            │ │
│  └────────────────────────────────────────────────────────────────────────┘ │
│                               ↓                                             │
│  ┌────────────────────────────────────────────────────────────────────────┐ │
│  │ LAYER 2: Team RBAC                                                     │ │
│  │ Is resource assigned to a team?                                        │ │
│  │   → YES: Does user have required permission IN that specific team?     │ │
│  │       → YES: ALLOW                                                     │ │
│  │       → NO: Continue to Layer 3                                        │ │
│  │   → NO: Continue to Layer 3                                            │ │
│  └────────────────────────────────────────────────────────────────────────┘ │
│                               ↓                                             │
│  ┌────────────────────────────────────────────────────────────────────────┐ │
│  │ LAYER 3: Resource Access                                               │ │
│  │ Does user have direct access? (stakeholder, creator, etc.)             │ │
│  │   → YES: ALLOW                                                         │ │
│  │   → NO: DENY                                                           │ │
│  └────────────────────────────────────────────────────────────────────────┘ │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Hooks API

### usePermissions

The main hook providing access to all permission checking methods.

```typescript
import { usePermissions } from '@/services/rbac';

const {
  // Basic permission checks
  hasPermission,         // Check single permission
  hasAllPermissions,     // Check ALL permissions (AND)
  hasAnyPermission,      // Check ANY permission (OR)
  
  // Role checks
  isOwner,               // Check if user is org Owner
  isAdmin,               // Check if user is org Admin
  hasRole,               // Check specific role
  
  // Team checks (any team)
  isTeamLeader,          // Check if user is team leader in ANY team
  isTeamMember,          // Check if user is member of ANY team
  hasTeamPermission,     // Check permission in ANY team
  
  // Team-specific checks
  hasTeamPermissionIn,   // Check permission in SPECIFIC team
  getTeamPermissions,    // Get all permissions in SPECIFIC team
  isTeamLeaderIn,        // Check if team leader in SPECIFIC team
  hasTeamRole,           // Check if has role in ANY team
  hasTeamRoleIn,         // Check if has role in SPECIFIC team
} = usePermissions();
```

### useAccessCheck

The main combined three-layer access check hook.

```typescript
import { useAccessCheck, OrgPermissionKey, TeamPermissionKey } from '@/services/rbac';

function LeadActions({ lead }: { lead: LeadDto }) {
  const { allowed, reason, grantedBy } = useAccessCheck({
    // Layer 1: Org permissions (bypass)
    orgPermissions: [OrgPermissionKey.CRM_MANAGE],
    
    // Layer 2: Team permissions
    team: {
      teamId: lead.assignedTeam?.id,
      permissions: [TeamPermissionKey.TEAM_CRM_MANAGE],
    },
    
    // Layer 3: Resource permissions (from backend)
    resourcePermissions: lead.permissions,
    
    // Action type
    action: 'edit',
    
    // Mode (default: OR)
    mode: AccessMode.OR,
  });
  
  console.log({ allowed, reason, grantedBy });
  // { allowed: true, grantedBy: 'org' }
  // { allowed: true, grantedBy: 'team' }
  // { allowed: true, grantedBy: 'resource' }
  // { allowed: false, reason: 'No access granted by any layer' }
}
```

## Guard Components

### CombinedAccessGuard (Org-Only Usage)

For simple org-level permission checks (e.g., admin-only pages), use `CombinedAccessGuard` with only `orgPermissions`. This replaces the former `PermissionGuard` component.

```typescript
import { CombinedAccessGuard, OrgPermissionKey } from '@/services/rbac';

// Redirect if no permission (admin-only page)
<CombinedAccessGuard
  orgPermissions={[OrgPermissionKey.SYSTEM_ADMIN]}
  action="view"
  redirect="/home"
>
  <SettingsPage />
</CombinedAccessGuard>

// Show fallback if no permission (multiple permissions = OR by default)
<CombinedAccessGuard
  orgPermissions={[OrgPermissionKey.CRM_VIEW, OrgPermissionKey.CRM_MANAGE]}
  action="view"
  fallback={<AccessDenied />}
>
  <CRMDashboard />
</CombinedAccessGuard>

// Show disabled state
<CombinedAccessGuard
  orgPermissions={[OrgPermissionKey.CRM_MANAGE]}
  action="edit"
  showDisabled
>
  <Button>Create Lead</Button>
</CombinedAccessGuard>
```

### TeamPermissionGuard

Guard for team-specific permission checks.

```typescript
import { TeamPermissionGuard, TeamPermissionKey, OrgPermissionKey } from '@/services/rbac';

interface Props {

  orgBypass?: string[];                 // Org permissions that bypass team check
```

### CombinedAccessGuard

The main guard component combining all three layers.

```typescript
import { CombinedAccessGuard, OrgPermissionKey, TeamPermissionKey, AccessMode } from '@/services/rbac';

interface Props {
  // Layer 1: Org permissions (bypass)
  orgPermissions?: string[];
  
  // Layer 2: Team permissions
  team?: {
    teamId: string | undefined;
    permissions: string[];
  };
  
  // Layer 3: Resource permissions
  resourcePermissions?: ResourcePermissions;
  
  // Action type
  action: 'view' | 'edit' | 'delete';
  
  // Combine mode (default: OR)
  mode?: AccessMode;
  
  // Rendering options
  children: React.ReactNode;
  fallback?: React.ReactNode;
  redirect?: string | boolean;
  showDisabled?: boolean;
  disabledClassName?: string;
  showReason?: boolean;            // Show reason in dev mode
}
```

**Example:**

```typescript
// Full three-layer check for edit button
<CombinedAccessGuard
  orgPermissions={[OrgPermissionKey.CRM_MANAGE]}
  team={{
    teamId: lead.assignedTeam?.id,
    permissions: [TeamPermissionKey.TEAM_CRM_MANAGE],
  }}
  resourcePermissions={lead.permissions}
  action="edit"
  showDisabled
>
  <Button onClick={() => handleEdit(lead)}>
    <Edit className="h-4 w-4 mr-2" />
    Edit Lead
  </Button>
</CombinedAccessGuard>

// Delete with AND mode (must have ALL permissions)
<CombinedAccessGuard
  orgPermissions={[OrgPermissionKey.SYSTEM_ADMIN]}
  resourcePermissions={lead.permissions}
  action="delete"
  mode={AccessMode.AND}
>
  <Button variant="destructive">Delete Forever</Button>
</CombinedAccessGuard>
```

```typescript
'use client';

          )}
        </div>

        <div className="flex gap-2">
```

```typescript
            orgPermissions={[OrgPermissionKey.SALES_MANAGE]}
            team={{
              teamId: deal.assignedTeam?.id,
              permissions: [TeamPermissionKey.TEAM_SALES_MANAGE],
            }}
            resourcePermissions={deal.permissions}
            showDisabled
          >

          </CanEditResource>

          <CanDeleteResource
            orgPermissions={[OrgPermissionKey.SALES_MANAGE]}

              <Trash className="h-4 w-4 mr-2" />

            </Button>

      {/* Deal content */}
      <Card>
        <CardHeader>
          <CardTitle>Deal Information</CardTitle>

            </div>
            <div>
              <dt className="font-medium text-muted-foreground">Stage</dt>
              <dd>{deal.stage?.name}</dd>
            </div>
          </dl>
        </CardContent>
      </Card>

      {/* Conditional sections based on edit access */}
      {canEdit && (
        <Card>
          <CardHeader>
            <CardTitle>Quick Actions</CardTitle>
          </CardHeader>
          <CardContent className="flex gap-2">
            <Button variant="outline">Move to Next Stage</Button>
            <Button variant="outline">Assign to Team</Button>
            <Button variant="outline">Add Stakeholder</Button>
          </CardContent>
        </Card>
      )}
    </div>
  );
}
```

### Example 6: Bulk Actions with Access Filtering

Filter bulk actions based on individual resource permissions.

```typescript
// src/components/pages/leads/lead-bulk-actions.tsx
'use client';

import { useMemo } from 'react';
import { OrgPermissionKey, TeamPermissionKey, usePermissions } from '@/services/rbac';

interface LeadBulkActionsProps {
  selectedLeads: LeadDto[];
  onBulkDelete: (leads: LeadDto[]) => void;
  onBulkArchive: (leads: LeadDto[]) => void;
}

export function LeadBulkActions({
  selectedLeads,
  onBulkDelete,
  onBulkArchive,
}: LeadBulkActionsProps) {
  const { hasAnyPermission, hasTeamPermissionIn, isOwner } = usePermissions();

  // Filter leads user can delete
  const deletableLeads = useMemo(() => {
    return selectedLeads.filter((lead) => {
      // Owner can delete all
      if (isOwner()) return true;
      
      // Check org permission
      if (hasAnyPermission([OrgPermissionKey.CRM_MANAGE]).allowed) return true;
      
      // Check team permission
      if (lead.assignedTeam?.id) {
        if (hasTeamPermissionIn(
          lead.assignedTeam.id, 
          [TeamPermissionKey.TEAM_CRM_MANAGE]
        ).allowed) return true;
      }
      
      // Check resource permission
      return lead.permissions?.canDelete ?? false;
    });
  }, [selectedLeads, hasAnyPermission, hasTeamPermissionIn, isOwner]);

  const restrictedCount = selectedLeads.length - deletableLeads.length;

  return (
    <div className="flex items-center gap-4 p-4 bg-muted rounded-lg">
      <span className="text-sm">
        {selectedLeads.length} selected
      </span>

      <Button
        variant="destructive"
        size="sm"
        onClick={() => onBulkDelete(deletableLeads)}
        disabled={deletableLeads.length === 0}
      >
        Delete ({deletableLeads.length})
      </Button>

      <Button
        variant="outline"
        size="sm"
        onClick={() => onBulkArchive(deletableLeads)}
        disabled={deletableLeads.length === 0}
      >
        Archive ({deletableLeads.length})
      </Button>

      {restrictedCount > 0 && (
        <span className="text-sm text-muted-foreground">
          {restrictedCount} item(s) cannot be modified
        </span>
      )}
    </div>
  );
}
```

### From Manual Checks to New System

### Migration Steps

1. **Identify Permission Checks**
   - Search for `currentUser?.orgRoles`
   - Search for `isOwner`, `isAdmin` patterns
   - Search for `createdBy?.id === currentUser?.id`

2. **Replace with Appropriate Component**

   |-------------|--------------|
   | Admin-only check | `CombinedAccessGuard` with `orgPermissions={[OrgPermissionKey.SYSTEM_ADMIN]}` |
   | Role-based check | `CombinedAccessGuard` with `orgPermissions` (preferred) or `useHasPermission` hook |

3. **Test All Scenarios**

   - Resource creator vs non-creator

## Best Practices

### 1. Always Use Enums

```typescript
// ✅ Good - Type-safe, autocomplete, compile-time checking
import { OrgPermissionKey } from '@/services/rbac';
useHasPermission(OrgPermissionKey.CRM_MANAGE);

// ❌ Bad - Typos won't be caught
useHasPermission('crm.manage');
```

### 2. Use showDisabled for Better UX

```typescript
// ✅ Good - User sees button but understands they can't use it
<CombinedAccessGuard action="edit" showDisabled resourcePermissions={lead.permissions}>
  <Button>Edit</Button>
</CombinedAccessGuard>

// ❌ Bad - Button disappears, user confused
<CombinedAccessGuard action="edit" resourcePermissions={lead.permissions}>
  <Button>Edit</Button>
</CombinedAccessGuard>
```

### 3. Trust Backend Permissions

```typescript
// ✅ Good - Use backend-computed permissions
<CombinedAccessGuard resourcePermissions={lead.permissions} action="edit">
  <Button>Edit</Button>
</CombinedAccessGuard>

// ❌ Bad - Don't duplicate backend logic
const canEdit = lead.createdBy?.id === currentUser?.id ||
                lead.stakeholders?.some(s => s.userId === currentUser?.id);
```

### 4. Include Team Info When Available

```typescript
// ✅ Good - Include team for full three-layer check
<CombinedAccessGuard
  orgPermissions={[OrgPermissionKey.CRM_MANAGE]}
  team={{
    teamId: lead.assignedTeam?.id,  // Can be undefined
    permissions: [TeamPermissionKey.TEAM_CRM_MANAGE],
  }}
  resourcePermissions={lead.permissions}
  action="edit"
>
  <Button>Edit</Button>
</CombinedAccessGuard>

// ⚠️ Incomplete - Missing team layer
<CombinedAccessGuard
  orgPermissions={[OrgPermissionKey.CRM_MANAGE]}
  resourcePermissions={lead.permissions}
  action="edit"
>
  <Button>Edit</Button>
</CombinedAccessGuard>
```

### 5. Use Correct Action Type

```typescript
// ✅ Good - Delete action for archive
<CombinedAccessGuard action="delete" resourcePermissions={lead.permissions}>
  <Button onClick={handleArchive}>Archive</Button>
</CombinedAccessGuard>

// ❌ Bad - Edit action for archive
<CombinedAccessGuard action="edit" resourcePermissions={lead.permissions}>
  <Button onClick={handleArchive}>Archive</Button>
</CombinedAccessGuard>
```

### 6. Use AND Mode for Sensitive Operations

```typescript
// ✅ Good - AND mode requires ALL checks to pass
<CombinedAccessGuard
  mode={AccessMode.AND}
  orgPermissions={[OrgPermissionKey.SYSTEM_ADMIN]}
  resourcePermissions={resource.permissions}
  action="delete"
>
  <Button variant="destructive">Permanently Delete</Button>
</CombinedAccessGuard>
```

### 7. Prefer Guard Components Over Manual Hook Checks

**Use guard components** (`CombinedAccessGuard`, `CanEditResource`, `CanDeleteResource`, `CanViewResource`, `TeamPermissionGuard`, `ResourceAccessGuard`) whenever possible. They provide declarative, consistent access control in JSX with built-in support for `fallback`, `showDisabled`, and `redirect`.

**Only use hooks** (`useHasPermission`, `useCanEdit`, `useAccessCheck`) when:
- You need the permission boolean for **non-rendering logic** (e.g., TanStack Query `enabled` flag)
- You're controlling **navigation items** (sidebar links, tab triggers) where a wrapping guard is impractical
- You need the result for **complex conditional logic** beyond simple show/hide

```typescript
// ✅ Good - Guard for rendering permission-gated UI
<CombinedAccessGuard
  orgPermissions={[OrgPermissionKey.SYSTEM_ADMIN]}
  action="edit"
  showDisabled
>
  <Button>Edit User</Button>
</CombinedAccessGuard>

// ✅ Good - Hook for query enabled flag (non-rendering logic)
const hasSalesView = useHasPermission(OrgPermissionKey.SALES_VIEW);
const { data } = useQuery({
  queryKey: queryKeys.user.commissions(userId),
  queryFn: () => UserApi.getCommissions(userId),
  enabled: !!userId && hasSalesView,
});

// ✅ Good - Hook for conditional tab triggers (guard is impractical here)
const hasCrmPermission = useHasPermission(OrgPermissionKey.CRM_VIEW);
<TabsList>
  <TabsTrigger value="overview">Overview</TabsTrigger>
  {hasCrmPermission && <TabsTrigger value="leads">Leads</TabsTrigger>}
</TabsList>

// ❌ Bad - Manual hook check where a guard would work
const canEdit = useHasPermission(OrgPermissionKey.CRM_MANAGE);
{canEdit && <Button>Edit</Button>}  // Use CombinedAccessGuard instead
```

## Troubleshooting

### Common Issues

#### 1. Permission Check Always Returns False

**Possible causes:**
- User data not loaded yet
- Permission context not wrapped around component
- Wrong permission key (use enums!)

**Solution:**
```typescript
// Check if user is loaded
const { currentUser, isLoading } = usePermissions();

if (isLoading) return <Loading />;
if (!currentUser) return <LoginRequired />;
```

#### 2. Team Permission Not Working

**Possible causes:**
- Team ID is undefined/null
- User not a member of the team
- Wrong team permission key

**Solution:**
```typescript
// Debug team membership
const { getTeamPermissions, hasTeamPermissionIn } = usePermissions();

console.log('Team ID:', lead.assignedTeam?.id);
console.log('Team permissions:', getTeamPermissions(lead.assignedTeam?.id));
console.log('Check result:', hasTeamPermissionIn(
  lead.assignedTeam?.id,
  TeamPermissionKey.TEAM_CRM_MANAGE
));
```

#### 3. Resource Permissions Missing

**Possible causes:**
- Backend not returning permissions
- API response structure changed
- DTO not updated

**Solution:**
```typescript
// Verify API response
const { data: lead } = useQuery({
  queryKey: queryKeys.leads.detail(id),
  queryFn: () => LeadApi.findOne(id),
});

console.log('Lead permissions:', lead?.permissions);
// Should show: { canView: true, canEdit: true, canDelete: false, ... }
```

#### 4. Guard Not Rendering Children

**Possible causes:**
- All permission checks failing
- Fallback not provided
- redirect causing immediate navigation

**Solution:**
```typescript
// Add showReason for debugging
<CombinedAccessGuard
  showReason  // Shows reason in dev mode
  // ... other props
>
  <Button>Edit</Button>
</CombinedAccessGuard>
```

### Debug Mode

Enable debug mode in development to see permission check results:

```typescript
// In permission-context.tsx or your component
if (process.env.NODE_ENV === 'development') {
  console.log('Permission check:', {
    orgPermissions,
    team,
    resourcePermissions,
    action,
    result: { allowed, reason, grantedBy },
  });
}
```
