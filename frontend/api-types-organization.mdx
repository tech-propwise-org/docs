# API Types Organization Guide

This document defines how TypeScript interfaces are organized between `types.ts` and individual `*.api.ts` files in the frontend API layer.

---

## Overview

The frontend API layer follows a clear separation pattern:

| File | Purpose |
|------|---------|
| `src/services/api/types.ts` | Shared interfaces used across multiple entities |
| `src/services/api/*.api.ts` | Module-specific interfaces (request DTOs, module-only types) |

---

## Decision Rules

### When to put interfaces in `types.ts`

Put an interface in `types.ts` when:

1. **Used by multiple entities** - The type is referenced in 2+ different modules
2. **Response DTOs** - Entity response types that appear in relationships (e.g., `UserDto` in `LeadDto.createdBy`)
3. **Shared enums** - Enums used across multiple modules
4. **Core domain types** - Types representing core business entities

**Examples of types that belong in `types.ts`:**

```typescript
// Response DTOs used across entities
UserDto           // Used in leads, deals, contacts, activities, etc.
SimpleUserDto     // Used in selection dropdowns across modules
OrganizationDto   // Used in LoginUserDto, user context
RoleDto           // Used in users, teams, permissions
TeamDto           // Used in users, memberships

// Shared enums
LeadSource        // Used in leads, persons, activities
ActivityType      // Used in activities, timeline views
AccessLevel       // Used in stakeholders, deals, leads

// Nested types that appear in multiple entities
Address           // Used in organizations, persons, companies
FileMetadata      // Used in developments, units, templates
```

### When to put interfaces in `*.api.ts`

Put an interface in the specific `*.api.ts` file when:

1. **Request DTOs** - Create/Update DTOs used only for API calls to that module
2. **Module-specific types** - Types only used within that module's context
3. **Query parameter types** - Filter/pagination params specific to that endpoint
4. **Internal helper types** - Types used only within that API file

**Examples of types that belong in `*.api.ts` files:**

```typescript
// organization.api.ts
CreateOrganizationDto    // Only used for POST /organizations
UpdateOrganizationDto    // Only used for PUT /organizations/:id

// user.api.ts
AssignableUsersParams    // Query params for user selection endpoint

// lead.api.ts
CreateLeadDto            // Only used for POST /leads
UpdateLeadDto            // Only used for PATCH /leads/:id
LeadFilterParams         // Query params for lead listing

// activity.api.ts
CreateCallLogDto         // Only used for creating call logs
CreateMeetingDto         // Only used for creating meetings
CallLogDto               // Only returned from activity endpoints
MeetingDto               // Only returned from activity endpoints
```

## File Structure Pattern

```
src/services/api/
├── types.ts              # Shared types (response DTOs, enums, shared interfaces)
├── api.base.ts           # Base API client
├── api.helpers.ts        # Response normalization helpers
├── index.ts              # Barrel exports
│
├── user.api.ts           # User API + user-specific request types
├── organization.api.ts   # Organization API + org-specific request types
├── lead.api.ts           # Lead API + lead-specific request types
├── deal.api.ts           # Deal API + deal-specific request types
├── contact.api.ts        # Contact API + contact-specific request types
├── company.api.ts        # Company API + company-specific request types
├── activity.api.ts       # Activity API + activity channel types
├── team.api.ts           # Team API + team-specific request types
├── role.api.ts           # Role API + role-specific request types
└── ...
```

## Matching Backend DTOs

Frontend types must **exactly match** their corresponding backend DTOs:

| Backend Location | Frontend Location |
|------------------|-------------------|
| `*.dto.ts` response classes | `types.ts` (if shared) or `*.api.ts` (if module-specific) |
| `*.dto.ts` input classes (Create/Update) | `*.api.ts` |
| Shared enums in `modules/shared/` | `types.ts` |

### Checklist for New Types

When adding new types, ask:

- [ ] Is this type used by other entities as a nested field? → `types.ts`
- [ ] Is this a response DTO that will appear in relationships? → `types.ts`
- [ ] Is this a Create/Update request DTO? → `*.api.ts`
- [ ] Is this only used within one module? → `*.api.ts`
- [ ] Does this match the backend DTO exactly? → Verify all fields match

## Import Patterns

### Importing shared types

```typescript
// In any *.api.ts file
import { UserDto, RoleDto, Address, LeadSource } from './types';
```

### Re-exporting module types

If a type from `*.api.ts` needs to be accessible from the barrel export:

```typescript
// In index.ts
export type { CreateLeadDto, UpdateLeadDto } from './lead.api';
```

### Avoiding circular dependencies

- `types.ts` should NOT import from `*.api.ts` files
- `*.api.ts` files CAN import from `types.ts`
- Use interface merging or separate files if circular deps arise

## Examples by Module

### User Module

**In `types.ts`:**
```typescript
UserDto              // Used everywhere (createdBy, updatedBy, assignedTo, etc.)
SimpleUserDto        // Used in selection dropdowns
UserOrgRoleDto       // Used in UserDto
MyUserDto            // Used in auth context
LoginUserDto         // Used in auth flow
UpdateUserDto        // Used in user management (shared across components)
RoleAssignmentDto    // Used in role management
```

**In `user.api.ts`:**
```typescript
AssignableUsersParams   // Query params for /users?assignable=true
```

### Organization Module

**In `types.ts`:**
```typescript
OrganizationDto         // Used in LoginUserDto.organizations, user context
Address                 // Used in organizations, persons, companies
```

**In `organization.api.ts`:**
```typescript
CreateOrganizationDto   // POST /organizations
UpdateOrganizationDto   // PUT /organizations/:id
```

### Activity Module

**In `types.ts`:**
```typescript
ActivityDto             // Timeline view across leads, deals, contacts
ActivityType            // Enum used in filters, creation
ActivityTargetType      // Enum for entity targeting
BasicUserDto            // Used in ActivityDto.performedBy
```

**In `activity.api.ts`:**
```typescript
// Channel-specific DTOs (only returned from activity endpoints)
CallLogDto, CreateCallLogDto, UpdateCallLogDto
MeetingDto, CreateMeetingDto, UpdateMeetingDto
NoteDto, CreateNoteDto, UpdateNoteDto
EmailLogDto, SmsLogDto, SiteVisitDto
// Enums specific to channels
CallDirection, CallOutcome, MeetingStatus, MeetingType
```

## Migration Guide

When refactoring existing types:

1. **Identify usage** - Search for all imports of the type
2. **Count modules** - If used in 2+ modules, keep in `types.ts`
3. **Check relationships** - If it appears as a nested field in other DTOs, keep in `types.ts`
4. **Move if appropriate** - If only used in one module, move to that `*.api.ts`
5. **Update imports** - Fix all import statements
6. **Verify no errors** - Run TypeScript compiler

## Summary

| Question | Answer |
|----------|--------|
| Is it a response DTO used in relationships? | `types.ts` |
| Is it a shared enum? | `types.ts` |
| Is it a Create/Update request DTO? | `*.api.ts` |
| Is it only used in one module? | `*.api.ts` |
| Does it need to be imported by other `*.api.ts` files? | `types.ts` |

**When in doubt:** If the type is returned from an API and might be nested in other entities, put it in `types.ts`.
